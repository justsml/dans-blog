The easiest open source project to trust is the one everyone already trusts.

React. TypeScript. Postgres. Boring answer, boring risk profile.

The harder question is the one engineers actually face on a Tuesday afternoon: some package solves your problem in 40 lines, the README looks competent, the npm sidebar has numbers, the repo has stars, and your build is waiting.

So is it legit?

That question is usually answered with vibes: stars, downloads, a familiar logo, the last commit date, maybe the issue count if someone is feeling investigative.

Those are not useless signals. They are just too easy to misread.

Downloads can be inflated by CI. Stars can be purchased, farmed, or dragged along by an old Hacker News moment. A "recent commit" can be a typo fix after three years of neglect. A low issue count can mean the project is clean, unused, or aggressively closed.

The better question is:

> Does the project's public behavior look like real software maintained by real people under real constraints?

That is harder to fake.

---

## First: Decide How Much Trust You Need

Not every dependency deserves a courtroom drama.

If it is a dev-only color formatter in a throwaway experiment, skim it and move on. If it runs in production, touches auth, parses untrusted input, executes shell commands, ships browser code to customers, or runs during `postinstall`, slow down.

Use a rough trust ladder:

| Dependency shape | Review depth |
| --- | --- |
| Dev-only, no scripts, no network, easy to remove | Quick scan |
| Runtime utility in a low-risk path | Repo and package review |
| Auth, crypto, payments, parsing, build tooling, CI, CLIs | Deep review |
| Anything with install scripts or native binaries | Deep review, preferably isolated |
| Anything your company standardizes on | Write down the decision |

This is not bureaucracy. It is matching scrutiny to blast radius.

The problem is not that every unknown package is malicious. Most are not. The problem is that a package manager is an execution engine with a nice progress bar.

## The Fast Checklist

Start with the shape:

| Signal | Good shape | Bad smell |
| --- | --- | --- |
| Project age | Exists long enough to have maintenance history | Created yesterday, already weirdly popular |
| Commit rhythm | Periods of work, review, release, and support | One giant dump, then silence |
| Contributor shape | More than one meaningful contributor, or a clear single-maintainer project with boundaries | Many drive-by typo commits pretending to be community |
| PR behavior | Review comments, tests, reverts, small fixes, occasional large features | Huge unreviewed PRs, all from one account, always merged instantly |
| PR size variance | Mix of small fixes, medium changes, and rare big rewrites | Every change is enormous, or every change is performative noise |
| File churn | Mostly updates, some additions, few deletions | Release adds a pile of new opaque files, especially generated blobs |
| Release trail | Tags, changelog, npm versions, GitHub releases line up | npm package has code the repo never had |
| Artifact shape | `npm pack` output is boring and expected | Tarball includes tests? fine. Includes minified mystery blobs? pause |
| Maintainer identity | Maintainers have history across issues, releases, and other projects | Fresh accounts control publish rights |
| Security posture | 2FA/provenance/signatures/advisories are visible where supported | Long-lived publish tokens, no provenance, no explanation |
| Public conversation | Mentions appear in places that match the project and include real opinions | Zero conversation, or giant numbers on only one or two platforms |
| Issue handling | Bugs get triaged, reproduced, fixed, or honestly declined | Maintainer only says "works for me" and closes |
| Missing history | Old issues and PRs remain linkable, even when closed | Changelog, commits, or comments reference issues/PRs that now 404 |
| Docs honesty | README names limits and non-goals | README promises magic and hides tradeoffs |

None of these proves safety.

Together, they tell you whether the project has a pulse or a costume.

## Mini Guide: Reading a GitHub Repo

Start on GitHub because the repo shows behavior. npm shows the package. You need both.

### 1. Look Past Stars

Stars answer a popularity question badly. They do not tell you who maintains the project, whether it still works, whether releases are reviewed, whether npm matches the repo, or whether ownership changed.

A project with 30 stars may be excellent and boring. A project with 30,000 stars may be abandoned, compromised, or solving a problem you do not have.

I care more about the shape of maintenance than the size of applause.

### 2. Open Insights, Not Just the README

GitHub's [Pulse view](https://docs.github.com/en/repositories/viewing-activity-and-data-for-your-repository/using-pulse-to-view-a-summary-of-repository-activity), graphs, and [statistics API](https://docs.github.com/en/rest/metrics/statistics) show pull requests, issues, commits, code frequency, and contributor activity.

Good signs:

- multiple weeks or months with meaningful activity
- merged PRs tied to issues or release notes
- small bug fixes mixed with feature work
- maintenance commits after releases
- real review comments, not only auto-merge ceremony

Bad smells:

- all activity comes from dependency bots
- the only recent commits are README badges
- every PR is merged by the author with no discussion
- no release tags even though npm has many versions
- a long quiet period followed by a sudden publish-heavy burst

A long quiet period followed by a publish-heavy burst deserves extra attention. Compromised accounts, ownership transfers, and rushed rewrites often look different from normal maintenance.

### 3. Count People, But Weight the Work

Contributor count is useful only after you subtract noise. Ten people fixing typos is not the same as three people reviewing code, cutting releases, answering bugs, and touching the core implementation.

Look for:

- who authored the last 20 meaningful commits
- who merged the last 20 PRs
- who publishes releases
- who responds when users report breakage
- whether contributors are independent or all tied to one company

A single-maintainer project can be legit. Plenty of excellent open source is one person with taste and a calendar problem. The question is whether the project is honest about that shape: narrow scope, conservative changes, few dependencies, no grand claims about enterprise destiny.

### 4. Measure PR Frequency, Size, and Variance

PRs tell you how a project changes when nobody is writing marketing copy. I like a mix:

- tiny PRs for bugs, docs, types, and dependency bumps
- medium PRs for features
- rare large PRs for architecture, generated files, migrations, or major releases

That variance looks like normal software. Uniformity is suspicious in both directions.

If every PR changes 2,000 lines, the project may be hard to review. If every PR changes one sentence, the contributor graph may be inflated. If a risky runtime package has a stream of giant, unreviewed changes from new accounts, do not be the brave install.

A quick local check:

```bash
gh pr list -R OWNER/REPO --state merged --limit 50 \
  --json number,title,createdAt,mergedAt,additions,deletions,changedFiles,author
```

The goal is not a perfect score. It is a believable distribution. Real maintenance has texture.

### 5. Watch the File Churn

File churn is one of the best hard-to-fake signals. Healthy projects mostly update existing code. They add files when a feature needs structure. They delete code when removing dead paths, replacing implementations, or cleaning generated output.

Riskier shapes:

- a patch release that adds many new files
- a tiny feature PR that includes bundled/minified artifacts
- new files under `scripts/`, `.github/workflows/`, `bin/`, or `postinstall` paths
- binary files or generated blobs that reviewers cannot inspect
- dependency changes hidden inside a large formatting diff

This matters for npm because malicious behavior often arrives as *new surface area*: a lifecycle script, dependency, binary, network call, or obfuscated file.

Locally, the simple version is:

```bash
git diff --stat v1.2.3..v1.2.4
git diff --name-status v1.2.3..v1.2.4
```

For a PR:

```bash
gh pr diff 123 -R OWNER/REPO --name-only
```

Do not only ask "how many lines changed?" Ask "what kind of surface area appeared?"

### 6. Read the Negative Space

Good projects say no. They close feature requests that do not fit, remove old APIs carefully, document breaking changes, explain why a dependency was not added, and have boring compatibility conversations.

The project that accepts every feature, supports every runtime, claims every use case, and has no visible tradeoffs is either very young or lying to itself.

The README should have limits. The issue tracker should have decisions. The changelog should have consequences.

### 7. Look for Deleted Issues and PRs

Deleted history is not automatically sinister. Maintainers remove spam, abusive threads, leaked secrets, and duplicates. That is normal. But missing project history is still a signal.

GitHub issue and PR numbers are sequential within a repo. If the changelog says "fixes #418" and `#418` is gone, that deserves a look. If release notes reference PRs that return 404, commits mention issue numbers nobody can open, or old discussions quote replies that no longer exist, the public story has holes.

Look for:

- changelog entries that link to deleted issues or PRs
- commit messages like `fix #123` where `#123` is unavailable
- release notes with missing comparison links or missing PR references
- issue threads where replies quote a maintainer comment that no longer exists
- large gaps in issue/PR numbers around a controversial release
- a sudden switch from public review to direct pushes
- old package versions tied to tags or PRs that disappeared

A few missing numbers are boring. A pattern around a risky release is not. The useful question is not "did anyone ever delete anything?" It is "did the project remove the evidence I need to understand this change?"

Quick checks:

```bash
gh issue view 123 -R OWNER/REPO
gh pr view 456 -R OWNER/REPO
git log --oneline --grep '#[0-9]\\+' --all
```

If the project has a polished changelog but the referenced review trail is gone, trust the gap more than the polish.

### 8. Read the Public Discourse Shape

Public conversation is softer than a Git diff, but it is still evidence. Not follower counts. Not launch-day applause. Shape.

Real products leave messy trails. People use them, complain about them, recommend them, misunderstand them, defend them, and occasionally write a 1,400-word post about a settings screen.

Think about products you know are real:

| Product | Normal discourse shape |
| --- | --- |
| Zoom | Support threads, workplace jokes, security debates, school stories, feature complaints, procurement posts |
| Stripe | Developer praise, integration pain, pricing arguments, startup advice, support complaints, API taste debates |
| Cinnabon | Photos, cravings, airport takes, mall nostalgia, nutrition regret, strong frosting opinions |

The point is not that all attention is good. The point is that real things create platform-shaped attention.

A developer library should leave traces on GitHub, npm, Stack Overflow, Hacker News, Reddit, blogs, conference slides, and company engineering posts. A beauty subscription box should leave more traces on Instagram, TikTok, YouTube, Reddit, reviews, and support threads. Nobody is filing a pull request against mascara.

Match the platform to the thing. Then check the ratio:

- Do the platforms match the kind of product?
- Does LinkedIn activity line up with news, customers, hiring, funding, conference talks, or adoption claims?
- Are there zeros where there should obviously be some signal?
- Are one or two platforms inflated while every other surface is empty?
- Do posts include specific nouns: versions, bugs, migrations, invoices, outages, support tickets, code snippets?
- Do complaints cluster around a real event, like a breaking release, pricing change, outage, ownership transfer, or security issue?
- Are people arguing about tradeoffs, or only repeating the tagline?

Spicy posts get more attention than boring posts. Do not panic because negative posts exist. Ask whether they are the whole public record.

Zeros are a red flag. So are huge numbers on one or two platforms paired with silence everywhere else.

Slow down when:

- an infrastructure project has npm downloads, GitHub stars, and no organic developer mentions anywhere else
- a company claims enterprise adoption, but LinkedIn, job posts, conference talks, case studies, and news mentions are empty
- a consumer product claims a huge community, but only has inflated Instagram numbers and no Reddit threads, reviews, support questions, or annoyed customers

That last one is not a community. It is a billboard.

Use social evidence the same way you use downloads: weak alone, useful in context.

The public conversation should not be perfectly positive. Real software has fans, skeptics, support requests, migrations, stale advice, and arguments about whether the new version made things better or merely more modern. That mess is part of the signal.

## Mini Guide: Reading an npm Package

GitHub tells you how the project behaves. npm tells you what you are actually installing. Those are not the same thing.

### 1. Compare the npm Package to the Repo

The npm registry exposes package metadata, commonly called the ["packument"](https://github.com/npm/registry/blob/main/docs/REGISTRY-API.md): versions, timestamps, maintainers, dependencies, scripts, repository links, dist info, and more.

Start with:

```bash
npm view PACKAGE \
  name version time maintainers repository license dist-tags \
  dependencies peerDependencies optionalDependencies scripts \
  --json
```

Check whether:

- the repository link points to the repo you inspected
- the latest version has a matching git tag
- the package was created recently but already has strange download numbers
- maintainers changed recently
- lifecycle scripts exist
- dependencies jumped unexpectedly
- `latest`, `next`, and other dist-tags look reasonable

The package page is not a birth certificate. It is a clue board.

### 2. Inspect the Tarball

The README is not what runs. The tarball is.

Use:

```bash
npm pack PACKAGE@latest --dry-run
```

Ask:

- Are the files expected for this package?
- Is the unpacked size reasonable?
- Are there surprise binaries?
- Are there minified files that do not correspond to source?
- Is `package.json` pointing `main`, `exports`, or `bin` at expected paths?
- Are test fixtures or examples accidentally shipping secrets?

For packages you are about to standardize on, download the tarball in a temporary directory and inspect it like code, because it is code.

### 3. Diff Releases, Not Just Source

For npm, the release artifact matters more than the GitHub diff.

`npm diff` can compare published package versions from the registry:

```bash
npm diff --diff=PACKAGE@1.2.3 --diff=PACKAGE@1.2.4
```

Look for:

- new lifecycle scripts
- new `bin` entries
- new dependencies
- changes to `exports`
- obfuscated JavaScript
- generated bundles with no source change
- large new files in a patch release

A patch release that fixes one TypeScript type should not introduce a downloader, a shell script, and a new dependency graph.

### 4. Treat Downloads as Weak Evidence

npm downloads are useful for rough popularity. They are not a user count, quality score, or safety guarantee.

CI installs packages. Bots install packages. Mirrors and scanners install packages. Monorepos reinstall packages a lot. Tiny transitive dependencies can have enormous downloads because they sit under popular tooling.

Use downloads only in context:

- compare downloads against dependents
- compare downloads against GitHub activity
- compare downloads against issue volume
- compare downloads over time, not only weekly count
- ask whether the number makes sense for the package's job

If a package has huge downloads, no issues, no repo activity, no dependents you recognize, and a maintainer account with no visible history, the number is not comforting. It is just loud.

### 5. Prefer Provenance, Signatures, and Boring Publishing

npm provenance is useful because it connects a published package back to a source commit and build environment. npm's docs say provenance details can show the build environment, workflow run, source commit, build file, and public transparency-log entry for packages published with provenance.

You can also verify registry signatures and provenance attestations with:

```bash
npm audit signatures
```

npm [trusted publishing](https://docs.npmjs.com/trusted-publishers/) uses OIDC from supported CI providers, so packages can be published with short-lived credentials instead of long-lived npm tokens. For public packages from public repos, it can also generate provenance attestations.

This does not make a package safe. It makes one important question less hand-wavy:

> Did this npm artifact come from the public source and build path it claims to come from?

That is stronger than "the README has badges."

### 6. Read Scripts Like They Are Production Code

In `package.json`, scripts are not decoration. Pay special attention to:

- `preinstall`
- `install`
- `postinstall`
- `prepare`
- `prepublishOnly`
- `bin`

Install scripts can run on your machine during dependency installation. Sometimes that is legitimate: native modules, generated binaries, compatibility setup. Sometimes it is the whole attack.

If a small utility package has an install script, it owes you an explanation. If the script reads environment variables, opens the network, shells out, downloads binaries, or behaves differently in CI, isolate it before installing.

### 7. Check the Maintainer Surface

For npm, maintainer shape matters because publish rights are power.

Look for:

- maintainers with long-lived accounts
- multiple maintainers on important packages
- organization ownership for critical packages
- consistent maintainer names across GitHub and npm
- no sudden transfer to unknown accounts
- no abandoned maintainer accounts with broad publish rights

One maintainer is not automatically bad. One maintainer plus a critical runtime package, no provenance, install scripts, and sudden releases after years of quiet is a different conversation.

Security is often about combinations.

The best signals are not impossible to fake, just expensive: multi-year issue history, review texture, contributor continuity, normal file churn, source-to-package alignment, recognizable dependents, platform-shaped discourse, and narrow scope.

Hard-to-fake does not mean safe. It means less likely to be empty theater.

## Red Flags That Deserve a Pause

I would slow down if I saw:

- brand-new package with a name close to a popular package
- mismatched GitHub repo and npm package
- npm package has files not present in source or build output
- new maintainer publishes after a long dormant period
- patch release adds install scripts, binaries, or network behavior
- repository has lots of stars but no issue/PR activity
- changelog, commits, or release notes reference issues/PRs that now 404
- public review history disappears around a sensitive release
- contributor graph is inflated by trivial changes
- zero public conversation where the project's claimed adoption should leave some
- social numbers are huge on one or two platforms and empty everywhere else
- praise repeats the same phrasing across unrelated accounts
- generated bundles are checked in with no reproducible build story
- package requests broad permissions for a narrow job
- README has badges but no operational detail
- changelog skips risky versions
- tests are missing for the core behavior
- license is missing or inconsistent

One red flag is a question.

Several red flags are an answer.

## A Practical Review Flow

For a GitHub repo:

1. Read the README for scope, non-goals, install path, and examples.
2. Check last release, last meaningful commit, and issue triage.
3. Open Insights or use GitHub's stats API for activity shape.
4. Inspect recent merged PRs for size, review behavior, and file churn.
5. Check contributors by meaningful work, not raw count.
6. Compare tags, releases, changelog, and referenced issue/PR links.
7. Search for deleted or missing review history around risky releases.
8. Search for public discussion in the places this kind of project should naturally appear.
9. Search for security issues, ownership transfers, and suspicious workflow changes.

For an npm package:

1. Run `npm view PACKAGE --json` and inspect metadata.
2. Confirm the repository link, maintainers, versions, and dist-tags.
3. Run `npm pack PACKAGE@latest --dry-run`.
4. Run `npm diff` between your current version and the target version.
5. Look closely at scripts, `bin`, `exports`, dependencies, and tarball contents.
6. Prefer packages with provenance and verify with `npm audit signatures` when available.
7. Install risky packages only inside an isolated environment.

The goal is not to create a perfect scorecard. The goal is to build a habit of looking at behavior before trust.

## The Decision Rule

A legitimate project does not have to be famous.

It does not need a corporate sponsor, a huge contributor graph, or a weekly release schedule. Some of the best software on the internet is maintained by one careful person who changes it only when needed.

But it should make sense.

The repo should look like the package. The package should look like the release. The release should look like the changelog. The changelog should look like the PRs. The PRs should look like humans maintaining software, not accounts manufacturing motion.

That is the real signal.

Not stars. Not badges. Not one big number in a sidebar.

Legit projects have a story the evidence can survive.
