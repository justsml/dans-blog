## समस्या: `num_beams` एक झूठ था

`transformers.js` के generation लूप की गहराइयों में, एक कमेंट बहुत लंबे समय से वहाँ मौजूद था:

```js
// TODO: Support beam search
```

और ठीक उसके नीचे, एक `break` स्टेटमेंट था जो चुपचाप पहले टोकन के बाद लूप से बाहर निकाल देता था। हर मॉडल कॉन्फ़िग जो `num_beams > 1` के साथ आता था — T5, BART, Whisper — चुपचाप greedy decoding ही कर रहा था। कोई चेतावनी नहीं। कोई त्रुटि नहीं। बस... गलत आउटपुट।

मुझे यह एक summarization pipeline का परीक्षण करते समय पता चला, और मैं सोच रहा था कि मेरे आउटपुट Python reference की तुलना में इतने खराब क्यों हैं। इसे `modeling_utils.js` तक ट्रेस किया, वह TODO देखा, और गलती से सोचा, "यह कितना मुश्किल हो सकता है?"

जवाब है: काफी मुश्किल, लेकिन दिलचस्प तरीकों से।

## Beam Search वास्तव में क्या है

कौन नहीं जानता

Greedy decoding हर चरण में सबसे अधिक संभाव्यता वाला टोकन चुनता है। सरल, तेज़, अक्सर अनुपयुक्त — आपके मुँह से निकला पहला शब्द हमेशा वाक्य के लिए सबसे अच्छी शुरुआत नहीं होता।

Beam search इसके बजाय `num_beams` उम्मीदवार अनुक्रमों को एक साथ जीवित रखता है, हर चरण में प्रत्येक को पूर्ण शब्दावली से विस्तारित करता है, फिर संचयी log-probability के आधार पर शीर्ष `num_beams` तक छाँटता है। यह टोकन स्पेस में एक सीमित breadth-first सर्च जैसा है।

परिणाम वैश्विक रूप से बेहतर अनुक्रम हैं, `num_beams`× कंप्यूट की कीमत पर।

तीन प्रकार मौजूद हैं:

- **Standard beam search** — निश्चित, argmax उम्मीदवार लेता है, सर्वश्रेष्ठ समग्र अनुक्रम
- **Diverse beam search** — बीम्स को समूहों में बाँटता है, पहले समूहों द्वारा पहले ही चुने गए टोकनों के लिए दंड देता है ताकि आपके आउटपुट उम्मीदवार सब एक ही बात न कहें
- **Beam sampling** — यादृच्छिक, beam फ्रेमवर्क के भीतर top-k + softmax + यादृच्छिक sampling लागू करता है

तीनों अब PR में हैं।

## वह आर्किटेक्चर निर्णय जिससे मैं वास्तव में जूझा

मौजूदा कोडबेस में एक `BeamSearchSampler` क्लास थी। यह प्रासंगिक लग रही थी। लेकिन एक सूक्ष्म जाल था: यह प्रति बीम केवल शीर्ष `num_beams` टोकन ही लौटाता था। यह सही लगता है जब तक आपको एहसास न हो कि वास्तविक beam search के लिए यह पर्याप्त नहीं है।

उचित beam search को प्रति बैच आइटम **सभी `num_beams × vocab_size` उम्मीदवारों** पर विचार करने की आवश्यकता होती है ताकि वैश्विक रूप से सर्वश्रेष्ठ continuations खोजे जा सकें। आप केवल प्रत्येक बीम से शीर्ष कुछ टोकनों को अलग-अलग देखकर नहीं कर सकते — आपको सभी बीम्स को एक साथ रैंक करने की आवश्यकता होती है।

इसलिए मैंने मौजूदा sampler को पूरी तरह bypass कर दिया। प्रोसेस किए गए logits पर सीधे `log_softmax` की गणना की, संचयी beam scores जोड़े, और संयुक्त उम्मीदवार स्पेस में दो-स्तरीय sort किया। साफ़ गणित, सही semantics।

`BeamSearchSampler` क्लास अभी भी वहीं है, अछूता, अभी भी उस काम के लिए उपयोगी जिसके लिए यह मूल रूप से बनाया गया था। यह उन मामलों में से एक है जहाँ "स्पष्ट" पुनः उपयोग का रास्ता आपको गलत जगह ले जाता है।

## सबसे कष्टप्रद बग: KV Cache Reordering

जब beam search अनुक्रमों को छाँटता है, तो यह केवल टोकनों को छोटा नहीं करता — यह *reorder* करता है कि कौन से बीम्स बचते हैं। बीम 3 सबसे अच्छा continuation उत्पन्न कर सकता है और clone हो सकता है; बीम 0 और 2 को छोड़ा जा सकता है।

समस्या यह है कि ट्रांसफॉर्मर के attention तंत्र से key-value cache बैच आयाम के साथ बीम द्वारा अनुक्रमित होती है। यदि आप आउटपुट अनुक्रमों को reorder करते हैं बिना cache को reorder किए, तो आपके पास mismatched state है। मॉडल गलत अतीत पर ध्यान केंद्रित कर रहा है।

सुधार `_reorder_cache()` है — एक विधि जो हर past key-value tensor पर `index_select` को कॉल करती है ताकि उन्हें नए beam क्रम के अनुसार reindex किया जा सके, फिर stale tensors को dispose कर दिया जाए।

CPU के लिए यह सीधा है: typed arrays को पंक्ति द्वारा slice करें। GPU tensors के लिए यह और कष्टप्रद हो जाता है — आपको async-download करना होता है (`ort_tensor.getData(true)`), reorder करना होता है, और re-upload करना होता है। मैंने दोनों पथों को संभालने के लिए `tensor.js` में `index_select` (sync, CPU) और `index_select_async` दोनों जोड़े।

Encoder-decoder मॉडल्स (T5, BART) के पास *दो* caches होती हैं: encoder और decoder। Encoder PKVs डिकोडिंग के दौरान नहीं बदलती हैं, इसलिए वे अपरिवर्तित पारित होती हैं। केवल decoder PKVs को reordering की आवश्यकता होती है। इस भेद को गलत समझना बहुत बुरे आउटपुट उत्पन्न करता है, सूक्ष्म रूप से — वह प्रकार की गलती जो लगभग सही दिखती है जब तक आप इसे reference से तुलना न करें।

## Diverse Beam Search: मज़ेदार वाला

Diverse beam search एक `diversity_penalty` जोड़ता है जो बाद के beam समूहों को पहले समूहों द्वारा पहले ही चुने गए टोकनों को चुनने से हतोत्साहित करता है। अंतर्ज्ञान यह है: यदि आपके सभी बीम एक ही आउटपुट पर अभिसरित हो जाते हैं, तो आपने वास्तव में hypothesis space का अन्वेषण नहीं किया है।

कार्यान्वयन के रूप में, समूहों को प्रत्येक डिकोडिंग चरण के भीतर *क्रमिक* रूप से प्रोसेस किया जाना चाहिए, समानांतर में नहीं, क्योंकि प्रत्येक समूह को अपने scores की गणना करने से पहले यह देखने की आवश्यकता होती है कि पहले समूहों ने क्या चुना है।

वह संरचना जो मैं अंततः लेकर आया:

```
for each step:
  token_counts = {}
  for each group in groups:
    extract this group's beams and logits
    for each token selected by previous groups:
      logits[token] -= diversity_penalty * token_counts[token]
    score candidates, select top 2×group_size
    group_scorer.process(...)
    record newly selected tokens into token_counts
```

यहाँ क्रमिक निर्भरता वास्तविक है। यदि आप इसे समानांतर करते हैं, तो आप diversity गारंटी खो देते हैं। मैं संक्षिप्त रूप से इसे वैसे भी batch करने के लिए प्रलोभित हुआ था और यह एक गलती होती।

## `BeamHypotheses` Priority Queue

जब एक बीम EOS टोकन से पहले `max_length` पर पहुँचता है, तो यह "done" होता है — लेकिन आप इसे तुरंत छोड़ या तुरंत return नहीं कर सकते। आप इसे `BeamHypotheses` नामक एक सीमित priority queue में जोड़ते हैं।

Queue प्रति बैच आइटम अधिकतम `num_beams` पूर्ण अनुक्रम रखती है, जिन्हें इसके द्वारा scored किया जाता है:

```
score = sum_logprobs / (length ^ length_penalty)
```

`length_penalty > 1.0` लंबे आउटपुट को पुरस्कृत करता है; `< 1.0` छोटे आउटपुट को पुरस्कृत करता है। `early_stopping` फ्लैग नियंत्रित करता है कि क्या बीम को तब "done" माना जाता है जब queue भर जाती है (`true`), कभी नहीं जब तक `max_length` न पहुँच जाए (`"never"`), या done जब कोई शेष बीम वर्तमान सबसे खराब hypothesis को संभावित रूप से हरा नहीं सकता (`false`)।

`false` मामला दिलचस्प है — यह ट्रैक करने की आवश्यकता होती है कि क्या कोई live बीम अधिकतम संभावित शेष score को देखते हुए भी वर्तमान सबसे खराब hypothesis को हरा सकता है। यह एक pruning अनुकूलन है जो `max_length` तक चलने से रोकता है जब आपके पास पहले से ही अच्छे hypotheses हैं।

यह `beam_search.js` में रहता है, नई फ़ाइल, कुल ~240 लाइनें। `BeamSearchScorer` भी export करता है, जो बैच में `BeamHypotheses` instances का प्रबंधन करता है और `finalize()` को संभालता है।

## Python Reference के विरुद्ध परीक्षण

यहाँ हर गैर-तुच्छ कार्यान्वयन विवरण का HuggingFace के `transformers` लाइब्रेरी में एक Python समकक्ष है। मैंने इस पर बहुत अधिक निर्भर किया।

मेरे द्वारा जोड़ा गया test suite कवर करता है:

- Encoder-decoder (T5) और decoder-only (LLaMA-ish) पर standard beam search
- `num_beam_groups=2, diversity_penalty=0.5` के साथ diverse beam search
- `do_sample=true, top_k=10` के साथ beam sampling
- `num_return_sequences > 1` — verifying output shape is `[N, seq_len]`
- असंगत combos के लिए सही error throws: CFG + beam search, streaming + beam search, `num_return_sequences > num_beams`

"सही error throws" tests underrated हैं। वे intentional सीमाओं का दस्तावेजीकरण करते हैं और किसी को silently गलत आउटपुट प्राप्त करने से रोकते हैं जब वे ऐसी सुविधाओं को जोड़ने का प्रयास करते हैं जिन्हें जोड़ा नहीं जा सकता। (मुझे पता है क्योंकि मैंने विकास के दौरान CFG और beam search को जोड़ने का प्रयास किया था। गणित काम नहीं करती। यह अब throw करता है।)

## अभी भी क्या गायब है

कुछ चीज़ें जिन्हें मैंने स्पष्ट रूप से छोड़ दिया, `throws` के साथ चिह्नित:

- **Diverse beam sampling** (`num_beam_groups > 1` + `do_sample`): यहाँ गणित वास्तव में जटिल हो जाती है। Standard diverse beam search समूहों में क्रमिक होती है; इसमें sampling जोड़ने के लिए यह सावधानीपूर्वक सोचने की आवश्यकता होती है कि diversity penalty को यादृच्छिक मोड में कैसे लागू किया जाए। यह करने योग्य है, बस नहीं किया गया।
- **Streaming + beam search**: Streaming टोकनों को उत्पन्न होते ही उपज करता है। Beam search परिभाषा के अनुसार कई चरणों तक यह नहीं जानता कि कौन सा अनुक्रम सर्वश्रेष्ठ है। ये मौलिक रूप से तनाव में हैं। आप अब तक का सबसे अच्छा बीम stream कर सकते हैं, लेकिन वह एक अलग सुविधा है जिसके अपने डिज़ाइन प्रश्न हैं।

## वह हिस्सा जिसके बारे में कोई नहीं बात करता: Open Source विलंबता

कोड काम करता है। Tests pass होते हैं। मौजूदा test suite clean है। यह महीनों से review में बैठा है।

यह बस बड़े, लोकप्रिय open source projects के साथ ऐसे ही होता है। Hugging Face team तेज़ी से ship कर रही है, issue queue विशाल है, और core generation loop को छूने वाला ~1,000-लाइन feature PR एक गैर-तुच्छ review commitment है। वे comments में responsive रहे हैं और जब भी उन्होंने इसे देखा तो वास्तव में engaged रहे। मैं शिकायत नहीं कर रहा हूँ — मैं दस्तावेजीकरण कर रहा हूँ।

यदि आप एक major OSS project में योगदान कर रहे हैं और quick merge की अपेक्षा कर रहे हैं: अपनी अपेक्षाओं को समायोजित करें। इस आकार की चीज़ के लिए कुछ महीने सामान्य हैं। कोड अभी भी आपके fork पर पूरे समय काम करता है।

## मुझे वास्तव में इससे क्या मिला

कुछ चीज़ें जो मेरे पहले नहीं थीं:

1. **Beam search की एक वास्तविक मानसिक मॉडल** — पाठ्यपुस्तक संस्करण नहीं, बल्कि edge cases वाली। KV caches कैसे break होती हैं। दो-स्तरीय sort क्यों मायने रखता है। `length_penalty` वास्तव में scores के साथ क्या करता है।

2. **JS में typed array math के लिए और अधिक प्रशंसा** — CPU typed arrays पर `index_select` को लागू करना उस स्तर पर low-level है जिसे आप web code में शायद ही छूते हैं। यह ठीक है, लेकिन यह वह नहीं है जिसके लिए JavaScript को डिज़ाइन किया गया था और आप इसे महसूस करते हैं।

3. **Python reference implementation के लिए नवीनीत सम्मान।** HuggingFace `transformers` लाइब्रेरी बड़ी और कभी-कभी gnarly है, लेकिन beam search logic अच्छी तरह comment किया गया है और design decisions स्पष्ट रूप से intentional हैं। इसे पढ़ना वास्तव में क्या बनाना था यह समझने का सबसे तेज़ तरीका था।

4. **जंगल में एक patch** — भले ही यह merge नहीं हुआ हो, यह मौजूद है, यह काम करता है, और लोग इसे PR branch से उपयोग कर सकते हैं। यह काफी है।

वह TODO comment जिसने इसे शुरू किया था, मेरे fork से गायब है। यह एक शांत, nerd तरीके से संतोषजनक है।
