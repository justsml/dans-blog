## समस्या: `num_beams` एक झूठ था

`transformers.js` के जनरेशन लूप के अंदर, एक कमेंट था जो काफी समय से वहाँ रहा था:

```js
// TODO: Support beam search
```

और ठीक उसके नीचे, एक `break` स्टेटमेंट जो पहले टोकन के बाद लूप को चुपचाप बाहर निकाल देता था। हर मॉडल कॉन्फिग जिसमें `num_beams > 1` शिप हुआ — T5, BART, Whisper — को चुपचाप greedy decoding मिल रहा था इसके बजाय। कोई वारनिंग नहीं। कोई एरर नहीं। सिर्फ... गलत आउटपुट।

मैंने इसे एक summarization पाइपलाइन टेस्ट करते हुए पाया और सोच रहा था कि मेरे आउटपुट Python रेफरेंस के मुकाबले कैसे इतने डिग्रेड हो गए। इसे `modeling_utils.js` में ट्रैक किया, TODO देखा, और यह गलती की कि "कैसे मुश्किल हो सकता है?"

जवाब है: काफी मुश्किल, लेकिन interesting तरीकों में।

## Beam Search वास्तव में क्या है

कौन नहीं

Greedy decoding हर स्टेप पर सबसे ज्यादा-प्रॉबेबिलिटी वाला टोकन पिक करता है। सिंपल, फास्ट, अक्सर सबऑप्टिमल — आपके मुंह से निकलने वाला पहला शब्द हमेशा सेंटेंस का सबसे अच्छा स्टार्ट नहीं होता।

Beam search इसके बजाय `num_beams` candidate sequences को समान रूप से जीवित रखता है, हर स्टेप पर फुल वोकेबुलरी के साथ प्रत्येक को expand करता है, फिर cumulative log-probability के अनुसार टॉप `num_beams` में प्रून बैक करता है। यह टोकन स्पेस में bounded breadth-first search जैसा है।

नतीजा globally बेटर sequences है, `num_beams`× कंप्यूट के खर्च पर।

तीन flavors exist करती हैं:

- **Standard beam search** — deterministic, argmax candidates लेता है, सबसे अच्छा overall sequence
- **Diverse beam search** — beams को groups में split करता है, earlier groups द्वारा already चुने गए टोकन्स को penalize करता है ताकि आपके आउटपुट candidates सब एक जैसा न कहें
- **Beam sampling** — stochastic, beam framework के भीतर top-k + softmax + random sampling apply करता है

तीनों अब PR में हैं।

## वह Architecture Decision जिससे मैंने वास्तव में जूझा

Existing codebase में `BeamSearchSampler` class था। यह relevant दिखता था। लेकिन एक subtle trap था: यह प्रत्येक beam के लिए सिर्फ टॉप `num_beams` tokens return करता था। यह सही सुनता है जब तक आप यह realize न करें कि actual beam search के लिए यह enough नहीं है।

Proper beam search को **all `num_beams × vocab_size` candidates per batch item** consider करने की ज़रूरत है globally सबसे अच्ठे continuations ढूंढने के लिए। आप हर beam के टॉप few tokens को isolation में नहीं देख सकते — आपको सभी beams के साथ combined candidate space में rank करना होगा।

तो मैंने existing sampler को entirely bypass किया। Processed logits पर directly `log_softmax` computed, cumulative beam scores जोड़े, और combined candidate space में two-level sort किया। Cleaner math, correct semantics।

`BeamSearchSampler` class अब भी वहाँ है, untouched, अब भी वह useful है जो वह originally कर रहा था। यह एक ऐसा case है जहाँ "obvious" reuse path आपको somewhere wrong ले जाता है।

## सबसे Annoying Bug: KV Cache Reordering

जब beam search sequences prune करता है, यह सिर्फ tokens truncate नहीं करता — यह *reorder* करता है कि कौन से beams survive करें। Beam 3 सबसे अच्छा continuation produce कर सकता है और clone हो सकता है; beams 0 और 2 discard हो सकते हैं।

समस्या यह है कि transformer's attention mechanism का key-value cache batch dimension के साथ beam द्वारा indexed है। अगर आप output sequences reorder करें cache reorder करने के बजाय, आप mismatched state के साथ हैं। मॉडल wrong past को attend कर रहा है।

Fix `_reorder_cache()` है — एक method जो हर past key-value tensor पर `index_select` call करता है नए beam order के अनुसार उन्हें reindex करने के लिए, फिर stale tensors dispose करता है।

CPU के लिए यह straightforward है: typed arrays को row से slice करें। GPU tensors के लिए यह और annoying हो जाता है — आपको data async-download करना होगा (`ort_tensor.getData(true)`), reorder करना होगा, और re-upload करना होगा। मैंने both `index_select` (sync, CPU) और `index_select_async` `tensor.js` में जोड़ा both paths handle करने के लिए।

Encoder-decoder models (T5, BART) में *two* caches हैं: encoder और decoder। Encoder PKVs decoding के दौरान change नहीं होते, तो वे unchanged pass होते हैं। Only decoder PKVs को reordering ज़रूरत है। यह distinction गलत करने पर very bad outputs produce होते हैं, subtly — ऐसा wrong जो almost right दिखता है जब तक आप reference compare न करें।

## Diverse Beam Search: The Fun One

Diverse beam search एक `diversity_penalty` जोड़ता है जो later beam groups को earlier groups द्वारा already चुने गए tokens select करने से discourage करता है। Intuition यह है: अगर आपके सभी beams same output पर converge करते हैं, आपने really hypothesis space explore नहीं किया।

Implementation-wise, groups को प्रत्येक decoding step के भीतर *sequentially* process होना चाहिए, parallel में नहीं, क्योंकि प्रत्येक group को earlier groups ने क्या select किया देखना होगा अपने own scores compute करने से पहले।

Structure मैं जिसके साथ end up हुआ:

```
for each step:
  token_counts = {}
  for each group in groups:
    extract this group's beams and logits
    for each token selected by previous groups:
      logits[token] -= diversity_penalty * token_counts[token]
    score candidates, select top 2×group_size
    group_scorer.process(...)
    record newly selected tokens into token_counts
```

यहाँ sequential dependency real है। अगर आप इसे parallelize करें, आप diversity guarantee lose करते हैं। मैं briefly इसे वैसे भी batching try करने के लिए tempted था और यह एक mistake होता।

## `BeamHypotheses` Priority Queue

जब एक beam `max_length` से पहले EOS token hit करता है, यह "done" है — लेकिन आप इसे सिर्फ discard नहीं कर सकते या immediately return नहीं कर सकते। आप इसे एक bounded priority queue में जोड़ते हैं जिसे `BeamHypotheses` कहते हैं।

Queue प्रत्येक batch item के लिए `num_beams` completed sequences तक रखता है, scored by:

```
score = sum_logprobs / (length ^ length_penalty)
```

`length_penalty > 1.0` longer outputs reward करता है; `< 1.0` shorter reward करता है। `early_stopping` flag control करता है कि beam done consider होगा जब queue full हो (`true`), never `max_length` तक (`"never"`), या done जब no remaining beam possibly worst completed hypothesis outscore कर सके (`false`)।

`false` case interesting one है — यह require करता है track करना कि क्या any live beam still current worst hypothesis beat कर सकता है maximum possible remaining score देखते हुए। यह एक pruning optimization है जो `max_length` तक run करने prevent करता है जब आप already अच्छे hypotheses रखते हैं।

यह `beam_search.js` में live करता है, new file, ~240 lines total। `BeamSearchScorer` भी export करता है, जो batch के साथ `BeamHypotheses` instances manage करता है और `finalize()` handle करता है।

## Python Reference के खिलाफ Testing

यहाँ हर non-trivial implementation detail का Python counterpart HuggingFace के `transformers` library में है। मैंने इस पर heavily lean किया।

Test suite मैंने जोड़ा जो cover करता है:

- Encoder-decoder (T5) और decoder-only (LLaMA-ish) पर Standard beam search
- `num_beam_groups=2, diversity_penalty=0.5` के साथ Diverse beam search
- `do_sample=true, top_k=10` के साथ Beam sampling
- `num_return_sequences > 1` — output shape `[N, seq_len]` है verify करना
- Incompatible combos के लिए Correct error throws: CFG + beam search, streaming + beam search, `num_return_sequences > num_beams`

"Correct error throws" tests underrated हैं। वे intentional limitations document करते हैं और someone को silently wrong outputs पाने से prevent करते हैं जब वे features combine try करें जो combine नहीं हो सकते। (मैं जानता हूँ क्योंकि मैंने development के दौरान CFG और beam search combine try किया। Math work नहीं करता। अब यह throw करता है।)

## क्या अब भी Missing है

Few things मैंने explicitly छोड़ा, `throws` से mark किया:

- **Diverse beam sampling** (`num_beam_groups > 1` + `do_sample`): Math यहाँ genuinely complicated हो जाता है। Standard diverse beam search groups के साथ sequential है; उसमें sampling जोड़ने के लिए careful thought ज़रूरत है कि stochastic mode में diversity penalty कैसे apply करें। यह doable है, सिर्फ done नहीं है।
- **Streaming + beam search**: Streaming tokens yield करता है जैसे वे generate होते हैं। Beam search by definition नहीं जानता कौन सा sequence सबसे अच्छा है multiple steps बाद तक। ये fundamentally tension में हैं। आप सबसे अच्छे beam तक now stream कर सकते हैं, लेकिन यह अलग feature है अपने design questions के साथ।

## मैंने वास्तव में इससे क्या पाया

Few things मैं पहले नहीं रखता था:

1. **Beam search का एक real mental model** — textbook version नहीं, बल्कि edge cases वाला। KV caches कैसे break होते हैं। Why two-level sort matter करता है। `length_penalty` actually scores के साथ क्या करता है।

2. **JS में typed array math के लिए और appreciation** — CPU typed arrays पर `index_select` implement करना low-level है ऐसे way में जो आप web code में rarely touch करते हैं। यह fine है, लेकिन यह वह नहीं जिसके लिए JavaScript designed था और आप वो feel करते हैं।

3. **Python reference implementation के लिए renewed respect।** HuggingFace `transformers` library बड़ी है और कभी-कभी gnarly, लेकिन beam search logic well-commented है और design decisions clearly intentional हैं। इसे पढ़ना fastest way था understand करने के लिए कि मैं actually क्या build करना supposed था।

4. **Wild में एक patch** — even अगर यह अब भी merged नहीं है, यह exist करता है, यह work करता है, और people इसे PR branch से use कर सकते हैं। यह enough है।
