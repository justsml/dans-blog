## Le problème : `num_beams` était un mensonge

Au fond de la boucle de génération de `transformers.js`, il y avait un commentaire qui traînait depuis longtemps :

```js
// TODO: Support beam search
```

Et juste en dessous, un `break` qui éjectait silencieusement la boucle après le premier token. Chaque config de modèle livrée avec `num_beams > 1` — T5, BART, Whisper — se retrouvait subrepticement en décodage glouton. Pas d'avertissement. Pas d'erreur. Juste... un résultat faux.

Je suis tombé dessus en testant un pipeline de résumé et en me demandant pourquoi mes sorties étaient aussi dégradées par rapport à la référence Python. J'ai remonté jusqu'à `modeling_utils.js`, vu le TODO, et j'ai fait l'erreur de me dire « ça peut pas être si compliqué ? »

Réponse : plutôt compliqué, mais de façon intéressante.

## Ce qu'est vraiment le beam search

Qui n'a jamais

Le décodage glouton choisit le token de plus haute probabilité à chaque étape. Simple, rapide, souvent sous-optimal — le premier mot qui vous sort de la bouche n'est pas toujours le meilleur début de phrase.

Le beam search garde plutôt `num_beams` séquences candidates en vie simultanément, en développant chacune par l'intégralité du vocabulaire à chaque étape, puis en élaguant pour ne garder que les `num_beams` meilleurs selon la log-probabilité cumulée. C'est une recherche en largeur bornée dans l'espace des tokens.

Le résultat : des séquences globalement meilleures, pour un coût multiplié par `num_beams`.

Trois variantes existent :

- **Beam search standard** — déterministe, prend les meilleurs candidats, meilleure séquence globale
- **Beam search diversifié** — divise les beams en groupes, pénalise les tokens déjà choisis par les groupes précédents pour que vos candidats ne disent pas tous la même chose
- **Beam sampling** — stochastique, applique top-k + softmax + échantillonnage aléatoire dans le cadre du beam search

Les trois sont dans la PR.

## La décision d'architecture qui m'a vraiment fait transpirer

La codebase existante avait une classe `BeamSearchSampler`. Elle avait l'air pertinente. Mais il y avait un piège subtil : elle ne renvoyait que les `num_beams` meilleurs tokens par beam. Ça a l'air correct jusqu'à ce qu'on réalise que ce n'est pas suffisant pour un vrai beam search.

Un beam search correct doit considérer **tous les `num_beams × vocab_size` candidats par élément de batch** pour trouver les meilleures continuations globales. On ne peut pas se contenter de regarder les meilleurs tokens de chaque beam isolément — il faut classer sur l'ensemble des beams réunis.

J'ai donc contourné entièrement le sampler existant. J'ai calculé `log_softmax` directement sur les logits traités, ajouté les scores cumulés des beams, et fait un tri à deux niveaux sur l'espace combiné des candidats. Des maths plus propres, une sémantique correcte.

La classe `BeamSearchSampler` est toujours là, intacte, toujours utile pour ce qu'elle faisait à l'origine. C'est un de ces cas où le chemin de réutilisation « évident » vous mène quelque part au mauvais endroit.

## Le bug le plus pénible : réordonner le cache KV

Quand le beam search élaguent des séquences, il ne fait pas que tronquer des tokens — il *réordonne* les beams qui survivent. Le beam 3 peut produire la meilleure continuation et être cloné ; les beams 0 et 2 peuvent être écartés.

Le problème, c'est que le cache clé-valeur du mécanisme d'attention du transformer est indexé le long de la dimension de batch par beam. Si vous réordonnez les séquences de sortie sans réordonner le cache, vous avez un état désynchronisé. Le modèle fait attention au mauvais passé.

La solution : `_reorder_cache()` — une méthode qui appelle `index_select` sur chaque tenseur clé-valeur passé pour les réindexer selon le nouvel ordre des beams, puis libère les tenseurs obsolètes.

Pour le CPU c'est direct : on découpe des typed arrays par ligne. Pour les tenseurs GPU c'est plus pénible — il faut télécharger les données de façon asynchrone (`ort_tensor.getData(true)`), réordonner, et remonter. J'ai ajouté `index_select` (synchrone, CPU) et `index_select_async` à `tensor.js` pour gérer les deux chemins.

Les modèles encodeur-décodeur (T5, BART) ont *deux* caches : encodeur et décodeur. Les PKV de l'encodeur ne changent pas pendant le décodage, ils passent inchangés. Seuls les PKV du décodeur nécessitent un réordonnancement. Se tromper sur cette distinction produit des sorties très mauvaises, subtilement — le genre de faux qui a presque l'air juste tant qu'on ne compare pas à une référence.

## Le beam search diversifié : le fun

Le beam search diversifié ajoute une `diversity_penalty` qui décourage les groupes de beams ultérieurs de sélectionner des tokens déjà choisis par les groupes précédents. L'intuition : si tous vos beams convergent vers la même sortie, vous n'avez pas vraiment exploré l'espace des hypothèses.

En termes d'implémentation, les groupes doivent être traités *séquentiellement* à chaque étape de décodage, pas en parallèle, parce que chaque groupe doit voir ce que les groupes précédents ont sélectionné avant de calculer ses propres scores.

La structure que j'ai obtenue :

```
pour chaque étape :
  token_counts = {}
  pour chaque groupe dans les groupes :
    extraire les beams et logits de ce groupe
    pour chaque token sélectionné par les groupes précédents :
      logits[token] -= diversity_penalty * token_counts[token]
    scorer les candidats, sélectionner les 2 × group_size meilleurs
    group_scorer.process(...)
    enregistrer les tokens nouvellement sélectionnés dans token_counts
```

La dépendance séquentielle ici est réelle. Si vous la parallélisez, vous perdez la garantie de diversité. J'ai été brièvement tenté d'essayer de paralléliser ça quand même et ça aurait été une erreur.

## La file de priorité `BeamHypotheses`

Quand un beam atteint le token EOS avant `max_length`, il est « terminé » — mais on ne peut pas le jeter ou le renvoyer immédiatement. On l'ajoute à une file de priorité bornée appelée `BeamHypotheses`.

La file garde jusqu'à `num_beams` séquences complétées par élément de batch, scorées par :

```
score = sum_logprobs / (longueur ^ length_penalty)
```

`length_penalty > 1.0` récompense les sorties plus longues ; `< 1.0` récompense les plus courtes. Le flag `early_stopping` contrôle si le beam est considéré comme terminé une fois la file pleine (`true`), jamais avant `max_length` (`"never"`), ou terminé quand aucun beam actif ne pourrait surpasser la pire hypothèse complétée (`false`).

Le cas `false` est le plus intéressant — il nécessite de suivre si un beam en vie pourrait encore battre la pire hypothèse courante étant donné le score maximal restant possible. C'est une optimisation d'élagage qui évite de courir jusqu'à `max_length` quand on a déjà de bonnes hypothèses.

Ça vit dans `beam_search.js`, nouveau fichier, ~240 lignes au total. Exporte aussi `BeamSearchScorer`, qui gère les instances de `BeamHypotheses` à travers le batch et gère `finalize()`.

## Tests contre la référence Python

Chaque détail d'implémentation non trivial ici a un équivalent Python dans la bibliothèque `transformers` de HuggingFace. Je m'y suis beaucoup appuyé.

La suite de tests que j'ai ajoutée couvre :

- Beam search standard sur encodeur-décodeur (T5) et décodeur-only (style LLaMA)
- Beam search diversifié avec `num_beam_groups=2, diversity_penalty=0.5`
- Beam sampling avec `do_sample=true, top_k=10`
- `num_return_sequences > 1` — vérification que la forme de sortie est `[N, seq_len]`
- Leves d'erreur correctes pour les combinaisons incompatibles : CFG + beam search, streaming + beam search, `num_return_sequences > num_beams`

Les tests « leves d'erreur correctes » sont sous-estimés. Ils documentent les limitations intentionnelles et empêchent quelqu'un d'obtenir silencieusement de mauvais résultats en combinant des fonctionnalités qui ne peuvent pas l'être. (Je sais de quoi je parle parce que j'ai essayé de combiner CFG et beam search pendant le développement. Les maths ne marchent pas. Ça lève une erreur maintenant.)

## Ce qui manque encore

Quelques choses que j'ai laissées exprès de côté, marquées avec `throws` :

- **Beam sampling diversifié** (`num_beam_groups > 1` + `do_sample`) : les maths deviennent franchement compliquées ici. Le beam search diversifié standard est séquentiel à travers les groupes ; ajouter de l'échantillonnage à ça demande une réflexion soigneuse sur la façon d'appliquer la pénalité de diversité en mode stochastique. C'est faisable, juste pas fait.
- **Streaming + beam search** : le streaming produit des tokens au fur et à mesure de leur génération. Le beam search, par définition, ne sait pas quelle séquence est la meilleure avant plusieurs étapes. C'est fondamentalement en tension. On pourrait streamer le meilleur beam actuel, mais c'est une fonctionnalité différente avec ses propres questions de conception.

## Ce dont personne ne parle : la latence open source

Le code marche. Les tests passent. La suite de tests existante est propre. Ça fait des mois que ça attend en revue.

C'est juste comme ça avec les grands projets open source populaires. L'équipe Hugging Face sort des choses rapidement, la file d'issues est énorme, et une feature PR de ~1 000 lignes qui touche la boucle de génération principale représente un engagement de revue non trivial. Ils ont été réactifs dans les commentaires et sincèrement engagés quand ils s'y penchent. Je ne me plains pas — je documente.

Si vous contribuez à un projet OSS majeur et que vous vous attendez à une fusion rapide : ajustez vos attentes. Quelques mois c'est normal pour quelque chose de cette envergure. Le code continue de marcher sur votre fork pendant tout ce temps.

## Ce que j'en ai vraiment tiré

Quelques choses que je n'avais pas avant :

1. **Un modèle mental réel du beam search** — pas la version manuel, mais celle avec les cas limites. Comment les caches KV cassent. Pourquoi le tri à deux niveaux compte. Ce que `length_penalty` fait réellement aux scores.

2. **Plus d'appréciation pour les mathématiques sur typed arrays en JS** — implémenter `index_select` sur des typed arrays CPU, c'est du bas niveau d'une façon qu'on touche rarement dans le code web. Ça va, mais ce n'est pas pour ça que JavaScript a été conçu, et on le sent.

3. **Un respect renouvelé pour l'implémentation de référence en Python.** La bibliothèque `transformers` de HuggingFace est grosse et parfois brouillonne, mais la logique de beam search est bien commentée et les décisions de conception sont clairement intentionnelles. La lire a été le moyen le plus rapide de comprendre ce que j'étais censé construire.

4. **Un patch dans la nature** — même s'il n'est pas encore fusionné, il existe, il marche, et les gens peuvent l'utiliser depuis la branche de la PR. Ça suffit.

Le commentaire TODO qui a lancé tout ça a disparu de mon fork. C'est satisfaisant, d'une façon tranquille et nerd.
