## 問題: `num_beams` は嘘だった

`transformers.js` の生成ループの奥深くに、長い間存在していたコメントがありました:

```js
// TODO: Support beam search
```

そしてその直下に、最初のトークンの後にループを静かに脱出させる `break` 文が。`num_beams > 1` を持つすべてのモデル設定 — T5、BART、Whisper — は、ビームサーチの代わりに貪欲デコーディングを静かに受け取っていました。警告なし。エラーなし。ただ…間違った出力。

要約パイプラインをテストしていて、なぜ出力が Python のリファレンスと比較してこんなに劣化するのか不思議に思っていた時にこれを見つけました。`modeling_utils.js` に遡って TODO を発見し、「どれだけ難しいわけ？」と思ったのが間違いでした。

答えは: かなり難しい、でも面白い意味で。

## ビームサーチの実際

貪欲デコーディングは、各ステップで最も確率の高いトークンを選びます。シンプル、高速、しかし頻繁に最適ではない — 口から最初に出る言葉が、必ずしも文章の最適な始まりとは限りません。

ビームサーチは代わりに、`num_beams` 個の候補シーケンスを同時に維持し、各ステップで語彙全体によってそれぞれを展開し、累積対数確率によって上位 `num_beams` に刈り込みます。トークン空間を通る制限付き幅優先探索のようなものです。

結果はより大域的に良いシーケンスですが、`num_beams` 倍の計算コストがかかります。

3つのフレーバーが存在します:

- **標準ビームサーチ** — 決定論的、argmax 候補を取得、最適な全体シーケンス
- **多様性ビームサーチ** — ビームをグループに分割、以前のグループによって既に選択されたトークンにペナルティを課し、出力候補がすべて同じことを言わないようにする
- **ビームサンプリング** — 確率的、ビームフレームワーク内で top-k + softmax + ランダムサンプリングを適用

これら3つすべてが PR に含まれています。

## 実際に格闘したアーキテクチャの決定

既存のコードベースには `BeamSearchSampler` クラスがありました。関連しているように見えました。しかし微妙な罠がありました: これはビームごとに上位 `num_beams` トークンのみを返していました。正しいように聞こえますが、実際のビームサーチには十分ではないのです。

適切なビームサーチは、大域的に最適な継続を見つけるために、バッチアイテムごとに**すべての `num_beams × vocab_size` 候補**を考慮する必要があります。各ビームから孤立して上位いくつかのトークンを見るだけでは不十分です — すべてのビーム全体でランク付けする必要があります。

そこで、既存のサンプラーを完全にバイパスしました。処理されたロジット上で直接 `log_softmax` を計算し、累積ビームスコアを加算し、結合された候補空間全体で2レベルのソートを行いました。よりクリーンな数学、正しいセマンティクス。

`BeamSearchSampler` クラスはそのまま残っており、元々やっていたことのために依然として有用です。これは「明らかな」再利用パスが間違った場所に導くケースの1つです。

## 最も厄介なバグ: KV キャッシュの並べ替え

ビームサーチがシーケンスを刈り込むとき、トークンを切り捨てるだけではありません — どのビームが生き残るか*並べ替え*ます。ビーム3が最適な継続を生成してクローンされるかもしれません。ビーム0と2は破棄されるかもしれません。

問題は、トランスフォーマーの注意力メカニズムからのキーバリューキャッシュがバッチ次元に沿ってビームによってインデックスされていることです。出力シーケンスを並べ替えてキャッシュを並べ替えない場合、ミスマッチな状態になります。モデルは間違った過去を参照しています。

修正は `_reorder_cache()` — すべての過去キーバリューテンソルで `index_select` を呼び出し、新しいビーム順序に従って再インデックス化し、古いテンソルを破棄するメソッドです。

CPU の場合、これは簡単です: 行ごとに型付き配列をスライスします。GPU テンソルの場合、より厄介になります — データを非同期でダウンロードし (`ort_tensor.getData(true)`)、並べ替え、再アップロードする必要があります。両方のパスを処理するために、`tensor.js` に `index_select` (同期、CPU) と `index_select_async` の両方を追加しました。

エンコーダー-デコーダーモデル (T5、BART) には*2つの*キャッシュがあります: エンコーダーとデコーダーです。エンコーダー PKV はデコード中に変更されないため、変更なしで通過します。デコーダー PKV のみが並べ替えを必要とします。この区別を間違えると、非常に悪い出力が生成されます — 微妙に間違っていて、リファレンスと比較するまでほとんど正しいように見える種類の間違いです。

## 多様性ビームサーチ: 面白いやつ

多様性ビームサーチは、後のビームグループが以前のグループによって既に選択されたトークンを選択するのを抑制する `diversity_penalty` を追加します。直感は: すべてのビームが同じ出力に収束する場合、仮説空間を実際に探索していないということです。

実装面では、グループは各デコードステップ内で並列ではなく*順次*処理する必要があります。各グループが独自のスコアを計算する前に、以前のグループが何を選択したかを見る必要があるためです。

最終的な構造:

```
for each step:
  token_counts = {}
  for each group in groups:
    extract this group's beams and logits
    for each token selected by previous groups:
      logits[token] -= diversity_penalty * token_counts[token]
    score candidates, select top 2×group_size
    group_scorer.process(...)
    record newly selected tokens into token_counts
```

ここでの順次依存は本物です。並列化すると、多様性の保証が失われます。一時的にこれをバッチ処理しようかとtemptationを感じましたが、間違いだったでしょう。

## `BeamHypotheses` 優先度キュー

ビームが `max_length` より前に EOS トークンに到達すると、「完了」です — しかし、それを破棄したり即座に返したりすることはできません。`BeamHypotheses` と呼ばれる有界優先度キューに追加します。

キューは、バッチアイテムごとに最大 `num_beams` 個の完了シーケンスを保持し、次のようにスコア付けされます:

```
score = sum_logprobs / (length ^ length_penalty)
```

`length_penalty > 1.0` はより長い出力を報酬します; `< 1.0` はより短い出力を報酬します。`early_stopping` フラグは、キューがいっぱいになったときにビームが完了したと見なされるか (`true`)、`max_length` まで決して見なされないか (`"never"`)、または残りのビームが現在の最悪の完了仮説を上回る可能性がなくなったときに完了したと見なされるか (`false`) を制御します。

`false` のケースが興味深いものです — 残りのビームが最大可能な残りのスコアを考慮して現在の最悪の仮説を上回る可能性があるかどうかを追跡する必要があります。これは、良い仮説が既にある場合に `max_length` まで実行するのを防ぐ刈り込み最適化です。

これは `beam_search.js` にあり、新しいファイルで、合計約240行です。また、バッチ全体で `BeamHypotheses` インスタンスを管理し、`finalize()` を処理する `BeamSearchScorer` もエクスポートします。

## Python リファレンスに対するテスト

ここでの些細でない実装の詳細のすべてに、HuggingFace の `transformers` ライブラリに Python の対応物があります。それに大きく依存しました。

追加したテストスイートは次のことをカバーします:

- エンコーダー-デコーダー (T5) およびデコーダーのみ (LLaMA-ish) での標準ビームサーチ
- `num_beam_groups=2, diversity_penalty=0.5` を使用した多様性ビームサーチ
- `do_sample=true, top_k=10` を使用したビームサンプリング
- `num_return_sequences > 1` — 出力形状が `[N, seq_len]` であることを確認
- 互換性のない組み合わせの正しいエラー投げ: CFG + ビームサーチ、ストリーミング + ビームサーチ、`num_return_sequences > num_beams`

「正しいエラー投げ」テストは過小評価されています。これらは意図的な制限を文書化し、組み合わせることができない機能を組み合わせようとしたときに誰かが静かに間違った出力を得るのを防ぎます。(開発中に CFG とビームサーチを組み合わせようとしたことがあります。数学が機能しません。今は投げます。)

## まだ欠けているもの

`throws` でマークした、明示的に除外したことがいくつかあります:

- **多様性ビームサンプリング** (`num_beam_groups > 1` + `do_sample`): ここでの数学は本当に複雑になります。標準の多様性ビームサーチはグループ間で順次です。それにサンプリングを追加するには、確率的モードで多様性ペナルティを適用する方法について慎重な考慮が必要です。可能ですが、まだ完了していません。
- **ストリーミング + ビームサーチ**: ストリーミングは生成されたトークンを生成しながら出力します。ビームサーチは定義上、複数のステップが経過するまでどのシーケンスが最適かを知りません。これらは根本的に緊張関係にあります。これまでの最良のビームをストリーミングすることはできますが、それは独自の設計質問を持つ異なる機能です。

## 誰も話さない部分: オープンソースのレイテンシ

コードは機能します。テストは合格します。既存のテストスイートはクリーンです。数ヶ月間レビューに座っています。

これは大規模で人気のあるオープンソースプロジェクトではよくあることです。Hugging Face チームは迅速に出荷し、問題キューは膨大で、コア生成ループに触れる約1,000行の機能 PR は些細でないレビューコミットメントです。コメントで対応し、実際に見た時に真摯に関わってきました。文句を言っているのではなく、文書化しています。

大規模な OSS プロジェクトに貢献していて、すぐにマージされることを期待している場合: 期待を調整してください。このサイズのものは数ヶ月は普通です。コードはフォーク上で常に機能します。

## 実際に得たもの

以前になかったいくつかのこと:

1. **ビームサーチの本当のメンタルモデル** — テキストブックのバージョンではなく、エッジケースを含むもの。KV キャッシュがどのように壊れるか。2レベルのソートがなぜ重要か。`length_penalty` が実際にスコアに何をするか。

2. **JS での型付き配列数学へのより深い感謝** — CPU 型付き配列で `index_select` を実装することは、Web コードでめったに触れない方法で低レベルです。問題ありませんが、JavaScript が設計されたものではなく、それを感じます。

3. **Python リファレンス実装への新たな敬意。** HuggingFace `transformers` ライブラリは大きく、時々厄介ですが、ビームサーチロジックはよくコメントされており、設計決定は明らかに意図的です。それを読むことが、実際に何を構築すべきかを理解する最速の方法でした。

4. **野外のパッチ** — まだマージされていなくても、存在し、機能し、人々は PR ブランチから使用できます。それで十分です。

これを始めた TODO コメントは、私のフォークから消えました。それは静かで、nerdy な方法で満足です。

今日 JavaScript で seq2seq の作業をしていて、適切なビームサーチが必要な場合、[PR は公開されています](https://github.com/huggingface/transformers.js/pull/1539)。
