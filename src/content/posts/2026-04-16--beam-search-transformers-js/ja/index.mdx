## 問題：`num_beams`は嘘だった

`transformers.js`の生成ループの奥深くに、長らく存在していたコメントがある：

```js
// TODO: Support beam search
```

そしてそのすぐ下に、ループを最初のトークンで静かに抜ける`break`文があった。`num_beams > 1`で shipped されたすべてのモデル設定 — T5、BART、Whisper — は、ひっそりとgreedy decodingを受けていた。警告なし。エラーなし。ただ…間違った出力。

これを見つけたのは、summarization pipelineをテストしていて、Pythonの参照実装と比べて出力がこんなにも劣化しているのはなぜだろうと考えていた時だった。`modeling_utils.js`まで追跡し、TODOを見つけ、「どれくらい難しいんだ？」と考えたのが間違いの始まりだった。

答えは：かなり難しい、ただし面白い意味で。

## Beam Searchとは実際何か

Greedy decodingは毎ステップで最高確率のトークンを選ぶ。単純、高速、頻繁に最適ではない — 口から最初に出た言葉が、いつも文の最良の始まりではない。

Beam searchは代わりに`num_beams`個の候補シーケンスを同時に生存させ、各ステップで全語彙に各候補を展開し、累積log確率で上位`num_beams`に切り詰める。トークン空間を制限付き幅優先探索するようなもの。

結果はグローバルに優れたシーケンス、ただし`num_beams`倍の計算量を犠牲にして。

三つの変種がある：

- **標準beam search** — 確定的、argmax候補を取る、全体で最良のシーケンス
- **Diverse beam search** — beamをグループに分割、以前のグループが既に選んだトークンにペナルティを課し、出力候補がすべて同じ内容にならないようにする
- **Beam sampling** — 確率的、beamフレームワーク内でtop-k + softmax + ランダムサンプリングを適用

これら三つすべてがPRに入っている。

## 実際に悩んだアーキテクチャ決定

既存のコードベースには`BeamSearchSampler`クラスがあった。関係ありそうに見えた。しかし微妙な罠があった：それは各beamから上位`num_beams`トークンだけを返す。正しそうに聞こえるが、実際のbeam searchにはそれだけでは足りないことに気付くまで。

正しいbeam searchは、各バッチ項目で**すべての`num_beams × vocab_size`候補**を考慮して、グローバルに最良の継続を見つける必要がある。各beamから上位数トークンを単独で見るだけではダメ — すべてのbeamを合わせてランキングしなければならない。

だから既存のサンプラーを完全にバイパスした。処理済みlogitsに直接`log_softmax`を計算し、累積beamスコアを追加し、統合候補空間で2レベルのソートを行った。より綺麗な数学、正しいセマンティクス。

`BeamSearchSampler`クラスはまだそこにあり、手つかずで、本来やっていた用途にはまだ有用。これは「明らかな」再利用パスが間違った場所に導くケースの一つ。

## 最も厄介なバグ：KVキャッシュの再順序付け

Beam searchがシーケンスを切り詰める時、トークンを切り詰めるだけではない — どのbeamが生存するかを*再順序付け*する。Beam 3が最良の継続を生み出し克隆されるかもしれない；beam 0と2は淘汰されるかもしれない。

問題は、transformerのattention mechanismからのkey-valueキャッシュが、beamによってバッチ次元に沿ってインデックスされている。出力シーケンスを再順序付けしてキャッシュを再順序付けしないと、不整合な状態になる。モデルが間違った過去にattendしている。

修正は`_reorder_cache()` — 新しいbeam順序に従って過去のすべてのkey-value tensorを`index_select`で再インデックスし、古くなったtensorを dispose するメソッド。

CPUの場合これは単純：typed arraysを行でスライス。GPU tensorsではもっと厄介になる — データを非同期でダウンロード（`ort_tensor.getData(true)`）し、再順序付けし、再アップロードしなければならない。両方のパスを扱うために、`tensor.js`に`index_select`（同期、CPU）と`index_select_async`を追加した。

Encoder-decoderモデル（T5、BART）は*二つの*キャッシュを持つ：encoderとdecoder。encoderのPKVはdecoding中に変化しないので、そのまま渡される。再順序付けが必要なのはdecoderのPKVだけ。この区別を間違えると、非常に悪い出力を微妙に生み出す — 参照と比較するまでほぼ正しく見える類の間違い。

## Diverse Beam Search：楽しい奴

Diverse beam searchは`diversity_penalty`を追加し、後のbeamグループが以前のグループが既に選んだトークンを選ぶのを妨げる。直感はこうだ：すべてのbeamが同じ出力に収束したら、仮説空間を本当に探索していない。

実装上、グループは各decodingステップ内で*逐次的に*処理されなければならない。並列ではなく。各グループが自身のスコアを計算する前に、以前のグループが何を選んだかを見る必要があるから。

最終的にこういう構造になった：

```
for each step:
  token_counts = {}
  for each group in groups:
    extract this group's beams and logits
    for each token selected by previous groups:
      logits[token] -= diversity_penalty * token_counts[token]
    score candidates, select top 2×group_size
    group_scorer.process(...)
    record newly selected tokens into token_counts
```

ここでの逐次依存は本物。並列化すると多様性保証を失う。一時バッチ処理を試そうと思ったが、それは間違いだった。

## `BeamHypotheses`優先度付きキュー

beamが`max_length`の前にEOSトークンに達すると、「完了」になる — しかしすぐに捨てたり返したりはできない。 bounded 優先度付きキュー`BeamHypotheses`に追加する。

キューは各バッチ項目で最大`num_beams`個の完了シーケンスを保持し、次のようにスコアリングする：

```
score = sum_logprobs / (length ^ length_penalty)
```

`length_penalty > 1.0`は長い出力を優遇；`< 1.0`は短い出力を優遇。`early_stopping`フラグは、キューが満杯になった時beamを完了と見なすか（`true`）、`max_length`まで完了と見なさないか（`"never"`）、残りのbeamが現在の最悪の完了仮説をスコアで上回る可能性がない時に完了と見なすか（`false`）を制御する。

`false`のケースが興味深い — 残りの任意の生きているbeamが、可能な残り最大スコアを考慮しても現在の最悪仮説を上回れないかを追跡する必要がある。これは pruning 最適化で、既に良い仮説がある時に`max_length`まで走るのを防ぐ。

これは`beam_search.js`の新ファイルにあり、合計~240行。`BeamSearchScorer`も export し、バッチ全体の`BeamHypotheses`インスタンスを管理し、`finalize()`を処理する。

## Python参照実装とのテスト

ここでの非自明な実装詳細すべては、HuggingFaceの`transformers`ライブラリにPythonの対応物がある。それに大きく頼った。

追加したテストスイートは次をカバーする：

- Encoder-decoder（T5）とdecoder-only（LLaMAっぽい）での標準beam search
- `num_beam_groups=2, diversity_penalty=0.5`でのDiverse beam search
- `do_sample=true, top_k=10`でのBeam sampling
- `num_return_sequences > 1` — 出力形状が`[N, seq_len]`になることの確認
- 互換性のない組み合わせでの正しいエラーthrow：CFG + beam search、streaming + beam search、`num_return_sequences > num_beams`

「正しいエラーthrow」のテストは過小評価されている。意図的な制限を文書化し、組み合わせられない機能を組み合わせようとした時にひっそりと間違った出力が出るのを防ぐ。（開発中にCFGとbeam searchを組み合わせようとしてみたから分かる。数学が成立しない。今はthrowする。）

## まだ欠けているもの

明示的に外したいくつかのもの、`throws`でマーク：

- **Diverse beam sampling**（`num_beam_groups > 1` + `do_sample`）：ここでの数学は genuinely complicatedになる。標準diverse beam searchはグループ間で逐次的；これにサンプリングを追加するには、確率的モードで多様性ペナルティをどう適用するかについて慎重な思考が必要。可能だ、ただまだ未実装。
- **Streaming + beam search**：Streamingはトークンが生成されるたびにyieldする。Beam searchは定義上、複数ステップ進んでからどのシーケンスが最良か分かる。これらは根本的に矛盾している。今の最良beamをstreamすることはできるが、それは別の機能で独自の設計問題がある。

## 誰も語らない部分：オープンソースのラグ

コードは動く。テストは通る。既存のテストスイートは clean。数ヶ月reviewに置かれている。

これは大きく人気のあるオープンソースプロジェクトではこうなる。Hugging Faceチームは速くshippingし、issue queueは巨大で、~1,000行の核心生成ループに触れるfeature PRは自明ではないreview commitmentだ。コメントでは responsive で、実際に見る時は genuinely engaged だ。不満ではない — 記録している。

大規模OSSプロジェクトにcontributeして素早いmergeを期待しているなら：期待を調整しよう。この規模で数ヶ月は普通。コードはfork上でずっと動き続ける。

## 実際に得たもの

以前持っていなかったいくつか：

1. **Beam searchの本当のメンタルモデル** — 教科書版ではなく、エッジケース付きの版。KVキャッシュがどう壊れるか。2レベルのソートがなぜ重要か。`length_penalty`が実際にスコアに何をするか。

2. **JSでのtyped array数学へのより深い理解** — CPU typed arraysに`index_select`を実装するのは、web codeでほっと触れない低レベルな方法。問題ないが、JavaScriptが設計された目的ではないと感じる。

3. **Python参照実装への新たな敬意** — HuggingFaceの`transformers`ライブラリは大きく、時に gnarly だが、beam searchロジックはよくコメントされ、設計決定が明確に意図的である。読むのが自分が実際に何を build すべきかを理解する最速の道だった。

4. **野外のパッチ** — mergeされていなくても、存在し、動き、PR branchから使える。それで十分。

これを始めたTODOコメントはforkから消えた。静かな、オタク的な満足感。

JavaScriptでseq2seq仕事をしていて今日proper beam searchが欲しいなら、[PRは public](https://github.com/huggingface/transformers.js/pull/1539)。
