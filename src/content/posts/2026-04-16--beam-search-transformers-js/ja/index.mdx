## 問題：`num_beams`は嘘だった

`transformers.js`の生成ループの奥深くに、長い間そのままだったコメントがありました：

```js
// TODO: Support beam search
```

その 바로下には、最初のトークンの後でループを静かに退出する`break`文がありました。`num_beams > 1`で出荷されたすべてのモデル設定（T5、BART、Whisper）は、静かにグリーディーデコーディングを受けていました。警告なし。エラーなし。ただ... 잘못した出力。

要約パイプラインをテストしているときに、これがきっかけでした。なぜ出力がPythonリファレンスと比較してこんなに劣化しているのか不思議に思い、`modeling_utils.js`までトレースして、TODOを見つけ、そして「どれほど難しいだろうか」という過ちを犯しました。

答えは：結構難しい、だが興味深い方法で。

## Beam Searchとは何か

誰もが...

グリーディーデコーディングは、各ステップで最高確率のトークンを選びます。シンプルで高速だが、しばしば最適ではない — 口から最初に出る言葉が常に文の最佳の始まりとは限りません。

Beam searchは代わりに`num_beams`個の候補シーケンスを同時に存活させ、各シーケンスを全ボキャブラリで拡張し、累積ログ確率で上位`num_beams`に剪定します。トークン空間に対する境界付き幅優先探索のようなものです。

結果はグローバルに優れたシーケンスですが、`num_beams`×の計算コストがかかります。

3つのフレーバーがあります：

- **標準beam search** — 決定論的、argmax候補を取得、全体として最適なシーケンス
- **多様なbeam search** —  beamをグループに分割し、早期グループがすでに選んだトークンにペナルティを課す sehingga出力候補がすべて同じことを言わないようにする
- **Beam sampling** — 確率的、beamフレームワーク内でtop-k + softmax +ランダムサンプリングを適用

すべて3つ，今PRに含まれています。

## 私が実際に頭を悩ませたアーキテクチャの決定

既存のコードベースには`BeamSearchSampler`クラスがありました。関連するように見えました。しかし微妙な罠がありました：それはbeamごとに上位`num_beams`トークンだけを返していました。それが正しい聞こえますが、実際にbeam searchには十分ではありません。

適切なbeam searchを得るには、**全`num_beams`×ボキャブラリサイズ**の候補をバッチアイテムごとに考慮してグローバルに最適な продолженияを見つける必要があります。各beamのトップトークンを分離して見ることはできません — すべてのbeamを一緒にランク付けする必要があります。

 그래서私は既存のサンプラーを完全にバイパスしました。処理されたlogitsに対して直接`log_softmax`を計算し、累積beamスコアを加え、 combined candidate space全体の2レベルソートを行いました。よりクリーンな数学、より正しいセマンティクス。

`BeamSearchSampler`クラスはまだそこにあり、触れないまま、あなたがたぶん当初使っていた何かを続けています。これは「明白な」再利用パスがあなたを間違った場所に導くそのようなケースの1つです。

## 最も烦わしいバグ：KVキャッシュのリordering

beam searchがシーケンスを剪定するとき、トークンを単にtruncateするだけでなく、**どのbeamが存活するかをリオーダー**します。Beam 3が最も良い продолжениеproducing し、cloneされるかもしれない；beam 0と2はdiscardされるかもしれない。

問題は、トランスフォーマーの注意メカニズムからのキー-valueキャッシュがbatch次元沿ってbeamでインデックスされていることです。出力シーケンスをリ順序付けせずにキャッシュをリ順序付けしないなら、ステートの不整合があります。モデルは間違った過去にattendしています。

修正は`_reorder_cache()` — これは新しいbeam順序に従って再インデックスするためにすべてのpast key-value tensorに対して`index_select`を呼び出し、その後古いtensorをdisposeするメソッドです。

CPUについてはこれは単純です：行ごとにtyped arraysをスライスします。GPU tensorについてはもっと面倒になります — async-downloadする必要があります（`ort_tensor.getData(true)`）、リ順序付け、そしてre-upload。私は両方の`index_select`（同期、CPU）と`index_select_async`を両方のpathを処理するために`tensor.js`に追加しました。

Encoder-decoderモデル（T5、BART）は**2つのキャッシュ**を持ちます：encoderとdecoder。Encoder PKVはdecoding中に変わらないので、を変更せず渡します。_decoder PKVだけがリ順序付けが必要です。 その区別を間違うと очень bad outputs、subtly — 間違っているがAlmost rightに見えるそんな種類。

## Diverse Beam Search：楽しい方

Diverse beam searchは、後続のbeamグループが早期グループによってすでに選択されたトークンを選択しないようにする`diversity_penalty`を追加します。直感は：すべてのbeamが同じ出力に収束するなら、あなたは実際には仮説空間を探索していません。

実装に関しては、 groupsは各decodingステップで*順番に*処理する必要があります 並列ではなく、なぜなら各グループは自分のスコアを計算する前に早期グループが何を選択したかを見る必要があるからです。

私が最終的に落ち着いた構造：

```
for each step:
  token_counts = {}
  for each group in groups:
    this group's beams and logitsを抽出
    for each token selected by previous groups:
      logits[token] -= diversity_penalty * token_counts[token]
    候補をスコア化し、上位2×group_sizeを選択
    group_scorer.process(...)
    newly selected tokensをtoken_countsに記録
```

ここでの逐次依存関係は本物です。あなたはこれをbatchingしようとする誘惑われて稍微、そしてそれは単なる過ちだったでしょう。

## `BeamHypotheses`優先度キュー

beamが`max_length`前にEOSトークンにhitすると、"done"です — しかしそれを単にdiscardしたり即座に戻したりすることはできません。有界の優先度キュー called `BeamHypotheses`に追加します。

キューはバッチアイテムごとに最大`num_beams`個のcompletedシーケンスを保持し、以下の式でスコアリング：

```
score = sum_logprobs / (length ^ length_penalty)
```

`length_penalty > 1.0`はより長い出力を奖励し、`< 1.0`はより短いものを奖励します。`early_stopping`フラグは、queueがいっぱいになったらbeamが完了したとみなすかどうか（`true`）、決して`max_length`まで待たない（`"never"`）、または残っているどのbeamも現在のworst hypothesisをoutscoreする 가능성이なくなったときに完了（`false`）を制御します。

`false`ケースは興味深いものです — これは、与えられた利用可能な最大remaining score、それでも生き残っているどのbeamも現在のworst hypothesisをoutscoreできるかどうかを追跡する必要があります。それは`max_length`まで走ることをpreventsする剪定最適化です你已经 хорошие hypothesesを持っているときに。

これは`beam_search.js`に住んでいます、新しいファイル、約240行。それ такжеエクスポート `BeamSearchScorer`、これはバッチ全体で`BeamHypotheses`インスタンスを管理し、`finalize()`を処理します。

## Pythonリファレンスに対してテストする

ここに每一个非自明な実装の詳細には、HuggingFaceの`transformers`ライブラリに対応するPython counterpartsがあります。私はそれを大いに活用しました。

追加したテストスイートは以下をカバーします：

- Encoder-decoder（T5）とdecoder-only（LLaMA-ish）での標準beam search
- `num_beam_groups=2, diversity_penalty=0.5`での diverse beam search
- `do_sample=true, top_k=10`でのbeam sampling
- `num_return_sequences > 1` — 出力shapeが`[N, seq_len]`であることを確認
- 非互換なcombosの正しいエラーthrow：CFG + beam search、streaming + beam search、`num_return_sequences > num_beams`

「正しいエラーthrow」テストは過小評価されています。それらは意図的な制限を記録し、誰かが組み合わせられない特徴を組み合わせようとするとき、静かに間違った出力を得るのをpreventsします。（私は開発中にCFGとbeam searchを組み合わせようとしたことを知っています。数学は機能しません。今throwします。）

## まだ欠けているもの

あなたがたぶん残しているものをマークするいくつかのもの、`throws`で：

- **Diverse beam sampling**（`num_beam_groups > 1` + `do_sample`）：数学はここで本当に複雑になります。標準のdiverse beam searchはグループ間で逐次です；それにsamplingを追加するには確率的モードでdiversityペナルティを適用する方法について慎重な思考が必要です。可能是 tetapiまだ終わっていません。
- **Streaming + beam search**：Streamingは生成されるにつれてトークンをyieldします。Beam searchは定義上、複数のステップアウトするまではどのシーケンスが最長未知です。これらは本質的に緊張関係にあります。あなたはこれまで中最良のbeamをstreamできますが、それはそれ自体がデザイン質問を持つ別の機能です。

## 私が実際にそれから得たもの

私が以前持っていなかったいくつかのことがあります：

1. **Beam searchの реальныйmental model** — 教科書バージョンではなく、エッジケースのあるもの。KVキャッシュが 어떻게壊れるか。2レベルソートがなぜ重要か。`length_penalty`が実際にスコアに何をするか。

2. **JSでのtyped array mathへの新たな感謝** — CPU typed arraysで`index_select`を実装することは、Webコードでまれにに触れることのないlow-levelです。それはFineですが、JavaScriptのために設計されたものではなく、あなたがその感じます。

3. **Pythonリファレンス実装への新たな尊敬。** HuggingFaceの`transformers`ライブラリは大きくて時に面倒ですが、beam searchロジックは хорошоコメントされ、デザイン決定は意図的に明確です。それを読むことは、私が実際に作るべきものを理解するための最速の方法でした。

4. **Wildのパッチ** — まだマージされていなくても、それは存在し、機能し、人々はPRブランチからそれを使用できます。それが十分です。
