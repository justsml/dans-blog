## 問題：`num_beams` は嘘だった

`transformers.js` の生成ループの奥深くに、長い間放置されていたコメントがありました。

```js
// TODO: Support beam search
```

そしてそのすぐ下には、最初のトークンの後にループを静かに終了させる `break` 文がありました。T5、BART、Whisper など、`num_beams > 1` で出荷されたすべてのモデル設定は、警告もエラーもなく、密かにグリーディデコーディング（貪欲法）を適用されていたのです。ただ……出力が間違っていただけです。

これに気づいたのは、要約パイプラインをテストしていて、なぜ Python のリファレンスと比較して出力の質がこれほど低いのか疑問に思ったときでした。`modeling_utils.js` までトレースしたところ、あの TODO を見つけ、「まあ、どれほど難しいことがあるだろうか？」と勘違いしてしまったのが運の尽きでした。

答えは「かなり難しい、しかし興味深い方法で」でした。

## ビームサーチとは何か

グリーディデコーディング（貪欲デコーディング）は、各ステップで最も確率の高いトークンを選択します。シンプルで高速ですが、最適でないことがよくあります。口から最初に出てくる言葉が、常に文章の最良の始まりとは限らないのと同じです。

一方、ビームサーチは `num_beams` 個の候補シーケンスを同時に維持します。各ステップで全語彙を使って各シーケンスを拡張し、累積対数確率（log-probability）によって上位 `num_beams` 個に絞り込みます。これは、トークン空間を通る、範囲の制限された幅優先探索のようなものです。

その結果、計算量は `num_beams` 倍になりますが、全体としてより優れたシーケンスが得られます。

主に3つのフレーバーがあります：

- **標準ビームサーチ** — 決定論的で、候補の argmax を取り、全体として最適なシーケンスを選びます。
- **多様なビームサーチ（Diverse beam search）** — ビームをグループに分け、先行するグループが選択したトークンにペナルティを課すことで、出力候補がすべて同じ内容にならないようにします。
- **ビームサンプリング** — 確率論的で、ビームサーチの枠組みの中で top-k + softmax + ランダムサンプリングを適用します。

今回の PR では、これら3つすべてを実装しました。

## 格闘したアーキテクチャの決定

既存のコードベースには `BeamSearchSampler` クラスがありました。関連がありそうに見えましたが、微妙な罠が隠されていました。それは、ビームごとに上位 `num_beams` 個のトークンのみを返すという仕様です。一見正しそうですが、実際のビームサーチには不十分です。

適切なビームサーチを行うには、バッチアイテムごとに **`num_beams × vocab_size` 個の全候補**を考慮して、グローバルに最適な継続を見つける必要があります。各ビームから上位数個のトークンを分離して見るだけでは不十分で、すべてのビームをまとめてランク付けする必要があるのです。

そこで、私は既存のサンプラーを完全にバイパスしました。処理されたロジット（logits）に対して直接 `log_softmax` を計算し、累積ビームスコアを加算し、結合された候補空間全体で2段階のソートを行いました。数学的にもすっきりし、セマンティクスも正確になりました。

`BeamSearchSampler` クラスは、元々何に使われていたにせよ、そのまま手付かずで残されています。「明白な」再利用の道が誤った方向へ導く典型的な例でした。

## 最も厄介だったバグ：KVキャッシュの並べ替え

ビームサーチがシーケンスを切り捨てるとき、単にトークンを切り詰めるだけでなく、どのビームが生き残るかの *順序を入れ替えます*。例えばビーム3が最適な継続を生成して複製され、ビーム0と2が破棄されるといったことが起こります。

問題は、トランスフォーマーのアテンション（注意）メカニズムからのキー・バリュー（KV）キャッシュが、バッチ次元に沿ってビームごとにインデックス化されていることです。キャッシュを並べ替えずに出力シーケンスだけを並べ替えると、状態が一致しなくなります。モデルが間違った過去を参照してしまうのです。

修正策は `_reorder_cache()` です。これは、新しいビームの順序に従ってすべての過去のキー・バリューテンソルに対して `index_select` を呼び出し、古いテンソルを破棄する方法です。

CPU の場合、これは単純です。型付き配列（Typed arrays）を行ごとにスライスするだけです。GPU テンソルの場合はもう少し面倒で、データを非同期でダウンロード（`ort_tensor.getData(true)`）し、並べ替えてから再アップロードする必要があります。両方のパスを処理するために、`tensor.js` に `index_select`（同期、CPU）と `index_select_async` の両方を追加しました。

エンコーダ・デコーダモデル（T5, BART）には、エンコーダ用とデコーダ用の *2つ* のキャッシュがあります。エンコーダの PKV（Past Key-Values）はデコーディング中に変化しないため、そのまま渡されます。デコーダの PKV のみを並べ替える必要があります。この区別を間違えると、非常に出力がおかしくなります。それも、リファレンスと比較するまでは一見正しそうに見える、巧妙な間違い方をするのです。

## 多様なビームサーチ：楽しいやつ

多様なビームサーチ（Diverse beam search）は、先行するビームグループがすでに選択したトークンを選択しないように促す `diversity_penalty` を追加します。直感的には、すべてのビームが同じ出力に収束してしまうと、仮説空間を十分に探索できていないことになります。

実装上、各デコーディングステップ内でグループを並列ではなく *順次* 処理する必要があります。各グループは、独自のスコアを計算する前に、先行するグループが何を選択したかを知る必要があるからです。

最終的な構造は以下の通りです：

```
各ステップにおいて:
  token_counts = {}
  各グループにおいて:
    このグループのビームとロジットを抽出
    先行グループが選択した各トークンについて:
      logits[token] -= diversity_penalty * token_counts[token]
    候補をスコアリングし、上位 2×group_size を選択
    group_scorer.process(...)
    新しく選択されたトークンを token_counts に記録
```

ここでの逐次的な依存関係は不可避です。これを並列化してしまうと、多様性の保証が失われます。一瞬、これをバッチ化しようかと誘惑に駆られましたが、それは間違いだったでしょう。

## `BeamHypotheses` 優先度付きキュー

ビームが `max_length` に達する前に EOS トークンにヒットした場合、それは「完了」したことになりますが、すぐに破棄したり返したりすることはできません。それを `BeamHypotheses` と呼ばれる制限付きの優先度付きキューに追加します。

このキューは、バッチアイテムごとに最大 `num_beams` 個の完了したシーケンスを保持し、以下の式でスコアリングします：

```
score = sum_logprobs / (length ^ length_penalty)
```

`length_penalty > 1.0` は長い出力を推奨し、`< 1.0` は短い出力を推奨します。`early_stopping` フラグは、キューがいっぱいになった時点でビームを完了と見なすか（`true`）、`max_length` まで決して完了と見なさないか（`"never"`）、あるいは残りのビームが現在の最悪の候補を上回る可能性がなくなった時点で完了とするか（`false`）を制御します。

`false` のケースが興味深いものでした。これは、残りの最大可能スコアを考慮して、生存しているビームが現在の最悪の候補を打ち負かすことができるかどうかを追跡する必要があります。これは、すでに良い候補が得られている場合に `max_length` まで実行するのを防ぐプルーニング最適化です。

このロジックは、新設された `beam_search.js`（約240行）に収められています。また、バッチ全体の `BeamHypotheses` インスタンスを管理し、`finalize()` を処理する `BeamSearchScorer` もエクスポートしています。

## Python リファレンスとの比較テスト

ここでの些末でない実装の詳細はすべて、HuggingFace の `transformers` ライブラリに Python の対応するものがあります。私はそれを大いに活用しました。

追加したテストスイートは以下をカバーしています：

- エンコーダ・デコーダ（T5）およびデコーダのみ（LLaMA系）での標準ビームサーチ
- `num_beam_groups=2, diversity_penalty=0.5` を指定した多様なビームサーチ
- `do_sample=true, top_k=10` を指定したビームサンプリング
- `num_return_sequences > 1` — 出力形状が `[N, seq_len]` であることの確認
- 互換性のない組み合わせに対する正しいエラーのスロー：CFG + ビームサーチ、ストリーミング + ビームサーチ、`num_return_sequences > num_beams`

「正しいエラーのスロー」テストは過小評価されがちです。これらは意図的な制限を文書化し、組み合わせ不可能な機能を組み合わせようとしたときに、誰かが誤った出力を黙って受け取ってしまうのを防ぎます（開発中に CFG とビームサーチを組み合わせようとしたからこそ分かります。計算が成り立たないのです。今はエラーを投げるようになっています）。

## まだ足りないもの

いくつか、明示的に除外して `throws` とマークしたものがあります：

- **多様なビームサンプリング** (`num_beam_groups > 1` + `do_sample`): ここでの計算は本当に複雑になります。標準的な多様なビームサーチはグループ間で逐次的ですが、そこにサンプリングを追加するには、確率論的モードで多様性ペナルティをどのように適用するか慎重に考える必要があります。不可能ではありませんが、まだ実装していません。
- **ストリーミング + ビームサーチ**: ストリーミングは生成されたトークンを逐次提供します。ビームサーチは定義上、数ステップ先まで進まないとどのシーケンスが最適か分かりません。これらは根本的に対立しています。現時点で最良のビームをストリーミングすることも可能ですが、それはまた別の機能であり、独自の設計上の課題があります。

## 得られたもの

以前は持っていなかった、いくつかのものを得ることができました：

1. **ビームサーチのリアルなメンタルモデル** — 教科書的なバージョンではなく、エッジケースを含んだモデルです。KVキャッシュがどう壊れるか。なぜ2段階ソートが重要なのか。`length_penalty` がスコアに実際に何をするのか。

2. **JS での型付き配列計算への理解** — CPU の型付き配列に対して `index_select` を実装するのは、Web コードでは滅多に触れない低レベルな作業です。悪くはありませんが、JavaScript がそのために設計されたものではないことを実感させられます。

3. **Python リファレンス実装への新たな敬意** — HuggingFace の `transformers` ライブラリは巨大で、時には難解ですが、ビームサーチのロジックはよくコメントされており、設計上の決定は明らかに意図的なものです。それを読むことが、実際に何を構築すべきかを理解する最短の道でした。

4. **世に出したパッチ** — たとえまだマージされていなくても、それは存在し、動作し、人々は PR ブランチからそれを使うことができます。それで十分です。

この作業のきっかけとなった TODO コメントは、私のフォークからは消えました。それは静かでマニアックな満足感を与えてくれます。
