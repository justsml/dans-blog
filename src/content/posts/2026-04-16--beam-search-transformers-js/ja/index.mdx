## 問題: `num_beams` は嘘だった

`transformers.js` の生成ループの奥深くに、長い間放置されていたコメントがあった:

```js
// TODO: Support beam search
```

その真下には、最初のトークンの後にループを静かに終了させる `break` 文があった。`num_beams > 1` を搭載したすべてのモデル設定 — T5、BART、Whisper — は、実際には貪欲デコード（greedy decoding）を実行していた。警告もなければ、エラーもない。ただ... 間違った出力が出ていた。

これを発見したのは、要約パイプラインをテストしていて、Python のリファレンス実装と比べて出力がなぜここまで劣化しているのか疑問に思ったときだ。`modeling_utils.js` を遡って調べて、その TODO を見つけ、「どれくらい難しいんだろう？」と思ってしまったのが間違いだった。

答えは: かなり難しい。だけど、興味深い形で。

## Beam Search とは実際には何か

誰だって

貪欲デコードは、各ステップで最も確率の高いトークンを選ぶ。シンプルで高速、しばしば最適ではない — 口から出てきた最初の言葉が、常に文章の最良の始まりになるわけではない。

Beam search は代わりに、`num_beams` 個の候補シーケンスを同時に保持し、各ステップでそれぞれをボキャブラリ全体で展開し、累積対数確率（cumulative log-probability）でトップ `num_beams` 個に刈り込む。トークン空間を通る制限付き幅優先探索のようなものだ。

その結果、計算量は `num_beams` 倍になる代わりに、大域的により良いシーケンスが得られる。

三つのバリエーションがある:

- **Standard beam search** — 決定論的、argmax 候補を採用、最良の全体的シーケンス
- **Diverse beam search** — ビームをグループに分割し、先行グループが既に選んだトークンにペナルティを与えることで、出力候補が皆同じことを言わないようにする
- **Beam sampling** — 確率的、ビームフレームワーク内で top-k + softmax + ランダムサンプリングを適用する

三つすべてが PR に含まれている。

## 実際に悩んだアーキテクチャ上の決定

既存のコードベースには `BeamSearchSampler` クラスがあった。関連しているように見えた。しかし、微妙な落とし穴があった: そのクラスは各ビームあたりトップ `num_beams` トークンのみを返していた。それは正しいように聞こえるが、実際の beam search には不十分だ。

正しい beam search では、各バッチアイテムあたり **すべての `num_beams × vocab_size` 候補** を考慮して、大域的に最良の継続を見つける必要がある。各ビームを独立に見て上位のトークンだけを見るのではなく、すべてのビームを横断してランク付けする必要がある。

だから既存の sampler を完全にバイパスした。処理済みの logits に直接 `log_softmax` を計算し、累積ビームスコアを加算し、結合された候補空間で二段階ソートを行った。よりクリーンな数学、正しい意味論。

`BeamSearchSampler` クラスはまだそこにあり、変更されていない。元々の用途には依然として有用だ。これは、「明らかな」再利用の道が間違った方向に導くケースの一つだ。

## 最も面倒だったバグ: KV Cache の再配置

Beam search がシーケンスを刈り込むとき、単にトークンを切り詰めるだけではない — 生存するビームを *再配置* する。ビーム 3 が最良の継続を生み出してクローンされるかもしれない。ビーム 0 と 2 は破棄されるかもしれない。

問題は、トランスフォーマーのアテンションメカニズムからの key-value cache が、バッチ次元に沿ってビームでインデックスされていることだ。出力シーケンスを再配置して cache を再配置しなければ、状態が不一致になる。モデルが間違った過去にアテンションを向けていることになる。

修正策は `_reorder_cache()` — 新しいビーム順に従ってすべての過去の key-value テンソルに `index_select` を呼び出し、古いテンソルを破棄するメソッドだ。

CPU の場合、これは単純だ: typed array を行でスライスする。GPU テンソルの場合はもっと面倒 — データを非同期ダウンロード（`ort_tensor.getData(true)`）して再配置し、再アップロードする必要がある。両方のパスを処理するために、`tensor.js` に `index_select`（同期、CPU）と `index_select_async` の両方を追加した。

エンコーダ・デコーダモデル（T5、BART）は *二つ* の cache を持つ: エンコーダとデコーダの cache だ。エンコーダの PKV はデコード中に変化しないので、そのまま通過する。デコーダの PKV のみが再配置が必要だ。この区別を間違えると、出力が非常に悪くなる — リファレンスと比較するまではほぼ正しいように見える、あの種類の微妙に間違った出力だ。

## Diverse Beam Search: 楽しい方

Diverse beam search は `diversity_penalty` を追加し、後続のビームグループが先行グループが既に選んだトークンを選ぶことを妨げる。直感として: すべてのビームが同じ出力に収束したら、仮説空間を実際に探索したことにはならない。

実装上、グループは各デコードステップ内で並列ではなく *順次* 処理されなければならない。なぜなら、各グループは自分のスコアを計算する前に先行グループが何を選んだかを見る必要があるからだ。

最終的に得た構造:

```
for each step:
  token_counts = {}
  for each group in groups:
    extract this group's beams and logits
    for each token selected by previous groups:
      logits[token] -= diversity_penalty * token_counts[token]
    score candidates, select top 2×group_size
    group_scorer.process(...)
    record newly selected tokens into token_counts
```

ここの順次依存関係は本物だ。これを並列化すると、多様性の保証が失われる。一瞬バッチ処理しようと誘惑されたが、それは間違いだっただろう。

## `BeamHypotheses` 優先度キュー

ビームが `max_length` より前に EOS トークンに到達すると、「完了」になる — しかし、すぐに破棄したり即座に返したりはできない。`BeamHypotheses` と呼ばれる制限付き優先度キューに追加する。

そのキューは、バッチアイテムあたり最大 `num_beams` 個の完了シーケンスを保持し、以下でスコア付けする:

```
score = sum_logprobs / (length ^ length_penalty)
```

`length_penalty > 1.0` はより長い出力を報酬し; `< 1.0` はより短い出力を報酬する。`early_stopping` フラグは、キューが満杯になった時点でビームを完了と見なす（`true`）、`max_length` まで決して完了としない（`"never"`）、または残りのビームが現在の最悪仮説を上回る可能性がない時点で完了とする（`false`）かを制御する。

`false` のケースが面白い — 残りのビームが最大可能残りスコアを考慮しても現在の最悪仮説を上回る可能性があるかどうかを追跡する必要がある。これは、`max_length` まで実行する必要がない時に実行を防ぐ剪定最適化だ。

これは `beam_search.js` にある、新しいファイル、合計約 240 行。また、`BeamSearchScorer` もエクスポートし、これはバッチ全体の `BeamHypotheses` インスタンスを管理し、`finalize()` を処理する。

## Python リファレンスに対するテスト

ここのすべての自明でない実装の詳細には、HuggingFace の `transformers` ライブラリにある Python の対応部分がある。それに大いに依存した。

追加したテストスイートは以下をカバーする:

- エンコーダ・デコーダ（T5）とデコーダのみ（LLaMA 風）での標準 beam search
- `num_beam_groups=2, diversity_penalty=0.5` を使った diverse beam search
- `do_sample=true, top_k=10` を使った beam sampling
- `num_return_sequences > 1` — 出力シェイプが `[N, seq_len]` であることを確認
- 互換性のない組み合わせに対する正しいエラー投げ: CFG + beam search、ストリーミング + beam search、`num_return_sequences > num_beams`

「正しいエラー投げ」テストは過小評価されている。これらは意図的な制限を文書化し、誰かが組み合わせられない機能を組み合わせようとした時に、静かに間違った出力を得るのを防ぐ。（開発中に CFG と beam search を組み合わせようとして気づいた。数学的に成立しない。今はエラーを投げる。）

## まだ不足しているもの

明示的に除外したいくつかのもの、`throws` でマークされている:

- **Diverse beam sampling** (`num_beam_groups > 1` + `do_sample`): ここの数学は本当に複雑になる。標準の diverse beam search はグループ間で順次実行される; そこにサンプリングを追加するには、確率的モードで多様性ペナルティをどう適用するかを慎重に考える必要がある。可能だが、まだ実装されていない。
- **ストリーミング + beam search**: ストリーミングはトークンを生成されるとすぐに出力する。Beam search は定義上、複数ステップ先になるまでどのシーケンスが最良か分からない。これらは根本的に矛盾している。これまでで最良のビームをストリーミングすることは可能だが、それは独自の設計上の問いを持つ別の機能だ。

## 誰も話さない部分: オープンソースのレイテンシ

コードは動く。テストは通る。既存のテストスイートはクリーンだ。レビューに数ヶ月かかっている。

これは大規模で人気のあるオープンソースプロジェクトではよくあることだ。Hugging Face チームは高速にリリースしている、issue キューは巨大で、コア生成ループに触れる約 1,000 行の機能 PR は、自明でないレビューコミットメントだ。コメントでは対応してくれ、実際に見てくれる時は本当に参加してくれている。文句を言っているわけではない — 記録しているだけだ。

主要な OSS プロジェクトに貢献して、素早いマージを期待しているなら: 期待値を調整してほしい。これくらいの規模なら数ヶ月は普通だ。コードはその間ずっとフォーク上で動く。

## 実際に得たもの

以前はなかったいくつかのもの:

1. **Beam search の実際のメンタルモデル** — 教科書版ではなく、エッジケース込みのバージョン。KV cache が壊れる仕組み。二段階ソートが重要な理由。`length_penalty` がスコアに実際に何をするか。

2. **JS での typed array 計算へのより深い感謝** — CPU typed array で `index_select` を実装することは、ウェブコードではめったに触れない低レベルの作業だ。悪くはないが、JavaScript が設計された目的ではなく、それを感じる。

3. **Python リファレンス実装への再認識。** HuggingFace の `transformers` ライブラリは大きく、時に複雑だが、beam search のロジックはよくコメントされていて、設計上の決定は明確に意図的だ。それを読むのが、実際に何を構築すべきかを理解する最速の方法だった。

4. **実際に存在するパッチ** — マージされていなくても、それは存在し、動作し、人々は PR ブランチから使用できる。それで十分だ。

このきっかけとなった TODO コメントは、私のフォークから消えた。それは静かで、オタク的な満足感がある。

今日、JavaScript で seq2seq の仕事をしていて適切な beam search が欲しいなら、[PR は公開されている](https://github.com/huggingface/transformers.js/pull/1539)。
