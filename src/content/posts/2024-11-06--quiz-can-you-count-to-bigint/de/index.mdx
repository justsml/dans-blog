    ```tsx
    0.1 + 0.2 === 0.3
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Aufgrund von Präzisionsfehlern bei Gleitkommazahlen ist `0.1 + 0.2` nicht exakt gleich `0.3`. Aufgrund der Art und Weise, wie Gleitkommazahlen im Speicher abgelegt werden, ist das Ergebnis `0.30000000000000004`. Der IEEE-754-Standard für die Gleitkommaarithmetik ist der Übeltäter – er kann manche Zahlen nicht exakt darstellen. Dies ist ein häufiges Problem in allen Programmiersprachen. Irgendwann stößt man auf eine unendlich wiederholende Dezimalzahl, und egal in welcher Sprache – der Computer muss einfach aufhören, unendlich viele Nachkommastellen zu verfolgen.

    Manche Sprachen wie Python und Java haben `Decimal` oder `BigDecimal`, um damit umzugehen, aber in JavaScript ist das nicht eingebaut. Du kannst Bibliotheken wie `big.js` oder `decimal.js` verwenden, um solche Fälle zu handhaben.

    (Hinweis: Manche Sprachen sind darauf ausgelegt, Brüche, imaginäre Zahlen usw. auf einer höheren logischen Ebene zu behandeln, indem sie wörtliche Ausdrücke bewahren. Aber sie müssen sich dennoch mit denselben Gleitkomma-Präzisionsproblemen auf Hardwareebene auseinandersetzen.)
  </div>
  </slot>
</Challenge>

  title="Gleichheitsvergleich zwischen `parseInt` und `parseFloat`"

  ]}
>
  <slot name="question">
  <div className="question">
    ```jsx
    parseInt("42") === parseFloat("42")
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    In JavaScript interpretieren sowohl `parseInt` als auch `parseFloat` den String `"42"` als die Zahl `42`. Daher ergibt der Vergleich `parseInt("42") === parseFloat("42")` den Wert `true`. Während `parseInt` das Parsen beim ersten nicht-ziffrigen Zeichen stoppt, fährt `parseFloat` fort, bis es auf ein Zeichen stößt, das nicht Teil einer Gleitkommazahl ist. Da es jedoch keine Dezimalpunkte oder andere nicht-numerische Zeichen in `"42"` gibt, geben beide Funktionen denselben Wert zurück.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={9}
  group="Number[]"
  title="Verwendung von `.map(parseInt)`"
  options={[
    {text: '[24, NaN, NaN]', isAnswer: true},
    {text: '[24, NaN, 42]'},
    {text: '[24, 42]'},
    {text: 'NaN'},
  ]}
>
  <slot name="question">
  <div className="question">
    ```jsx
    [24, 'One', 42].map(parseInt)
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Das zweite Argument von `parseInt` (der Radix) stimmt mit dem `index`-Argument der Array-Methoden überein. Dies führt zu unerwarteten Ergebnissen, da `parseInt("One", 1)` aufgrund der ungültigen Eingabe `NaN` zurückgibt.

    Das erste Element `24` wird in Basis 0 (automatische Erkennung) als `24` geparst, bleibt also `24`. Das zweite Element `'One'` wird in Basis 1 als `NaN` geparst. Das dritte Element `42` wird mit Basis 2 geparst. In Basis 2 ist `'42'` `NaN`, also ist das Ergebnis `[24, NaN, NaN]`.

  Dies ist eine häufige Falle bei `parseInt` und `map`. Wenn du ein Array von Strings in Zahlen parsen möchtest, ist die einzige sichere "eingebaute" Methode `.map(Number)` oder das Hinzufügen eines Callbacks/Enclosures `.map(x =&gt; parseInt(x, 10))`.
  </div>
  </slot>
</Challenge>
