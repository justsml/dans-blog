RAG 演示总是看起来很棒。

你把一些文档嵌入，启动向量库，将其接入聊天接口，看到模型像在内部知识库工作多年一样引用你的资料。很漂亮。利益相关者被震撼。有人说“我们上线吧”。

六周后，用户开始自信地得到错误答案。工单堆积。系统*还能运行*，只是不符合人们的需求。

这类失败通常不是单一的致命错误，而是五个乏味的错误叠加在一起。

---

## 1. 你的块大小不对

这种错误不会直接崩溃系统，只是让每个答案略有下降，最终让整个功能失去可信度。

向量检索返回的是*块*，而不是完整文档。你把源材料切成什么样，就成了检索器的真相单元。如果块切得不对，模型就会用错误的片段作答。

**块太小**：块只包含答案的一部分。嵌入捕捉到了正确的主题，但检索到的文本缺失上下文。比如你得到 “最大超时时间是 30 秒”，却没有前面的句子 “在使用旧版 API 时”。  

**块太大**：嵌入变成了多个想法的模糊平均。语义搜索会困惑，因为块涉及多项内容， resulting vector 并不能清晰地代表其中任何一个。

合适的块大小完全取决于你的内容。技术文档、法律合同和支持对话的切分方式各不相同，没有通用的答案。

**该怎么做**：测量。先从语料库中构建一个问答对的评估集。尝试 256、512、1024 token 的块大小。衡量检索精度：正确的块是否出现在前 5 名？你会很快发现块大小的影响比你为嵌入模型纠结的程度更大。

同时使用重叠。一个 512 token 的块在两侧各保留 64 token 的重叠，这样跨边界的答案仍能被检索到。大多数向量库都支持此功能，但大多数人会跳过它。

---

## 2. 你的嵌入会陈旧（而且你往往察觉不到）

想象一下，你的公司改了品牌，或更改了产品名称，或更新了定价，亦或废弃了某个 API。

你更新了文档，却没有重新对块进行嵌入。向量索引仍然代表旧的内容。

用户询问新定价时，嵌入把他们引向旧的内容。模型读取旧内容并自信地解释旧的定价，导致支持团队收到工单。

每个严肃的 RAG 系统迟早都会碰到这个问题。解决方案听起来很显而易见——内容变更时重新嵌入——但团队很少在第一次事故发生前就构建好相应的流水线。

你需要使用内容指纹进行增量重新索引：

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // 内容未变，跳过重新嵌入
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

写入时重新索引，指纹基于内容而非时间戳。文档在 CMS 中经常被“更新”，但实际内容并未改变。

## 3. 检索精度 vs. 召回率：你在优化错误的目标

大多数 RAG 教程只演示如何检索 top‑K 块，却不说明两个相互冲突目标之间的权衡。

**高召回**：返回所有可能相关的块。用户总能得到答案，但模型的上下文窗口被大量边缘噪声占满，模型会在碎片之间“填空”，产生幻觉。

**高精度**：只返回最相关的块。模型得到干净、聚焦的上下文。但如果关键块不在 top‑3，模型就缺少信息，却仍会自信地编造答案。

这两种失效在用户眼里表现相同：答案错误。但根因和解决方案恰恰相反。

真正有帮助的两种技术：

**重新排序**：先检索更多候选（top‑20），再使用 cross‑encoder 模型按相关性重新排序，随后再喂给 LLM。Cross‑encoder 比向量相似度慢，但在最终排序阶段准确度显著提升。

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // 现在的 top‑5 真有意义
```

**混合检索**：将向量检索（语义相似）与关键词检索（BM25）结合。两者的失效方式不同。向量检索在处理特定术语、模型名称和 ID 时表现欠佳；关键词检索在同义词和改写上受限。二者合并即可互补盲点。

## 4. 你的上下文窗口形状不对

你已经检索到了正确的块。恭喜。模型仍然会把它们用错。

问题不只在于你检索了什么，而在于你把它们放在哪里。

大语言模型会出现“中间丢失”问题。Liu 等人测量到，长上下文模型在相关信息出现在提示的中部时，检索可靠性会下降，而在开头或结尾时表现更好。

如果你把 20 个块直接塞进一个平铺列表，指望模型自行正确综合，那就是在浪费性能。

真正有帮助的做法：

**评估最相关块的首尾放置位置。** 常用的经验法则是把排名第一的块放在最前，排名第二的块放在最后，其余块放在中间。看似违背直觉，但值得在你的模型和提示形状上进行实验验证。

**显式编号并标记上下文段落。** `[Source 1]` … `[Source 2]` 为模型提供了可供推理的锚点。

**加入检索置信度信号。** 如果相似度得分在 0‑1 之间为 0.65，告诉模型：“以下上下文的检索置信度为中等。如果答案不明确，请注明不确定性。”

**设定上下文预算。** 不要把检索到的所有内容都直接塞进去。统计 token 数量，按相关性得分排序，并在模型上下文窗口的 60‑70% 处硬性截断。为模型留出推理空间，避免频繁换页。

参考文献: [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172)。

## 5. 你根本不知道何时出错

这是最隐蔽的失败：答案返回了，UI 看起来正常，但内容却是错误的。

使用传统 API 时，错误是显而易见的：HTTP 500、超时、模式校验错误。你会立刻知道。RAG 的错误更安静：系统返回了一个看似合理的响应，却是错误的。

你可能在用户指出之前根本不知道 RAG 流程出问题。通常他们不会直接指出，只是停止信任，回到手动 Ctrl+F。

生产环境下 RAG 系统的最小可行可观测性配置：

**记录检索链路。** 每一次查询、检索到的内容（块 ID + 得分）以及模型生成的输出都要日志化。调试任何问题都离不开这些信息。

**跟踪检索指标。** 如果有真实标签，计算平均倒数排名（MRR）和 NDCG。至少要监控相似度得分分布——当你的 P50 检索得分下降时，说明索引质量下降。

**构建反馈回路。** 即使是对响应的点赞/点踩，关联回查询和检索到的块，也能提供训练信号。没有这些，你就是盲目飞行。

**定期运行评估。** 准备一套 50‑100 条已知正确答案的问题作为测试集，每周跑一次，就能在用户发现之前捕获回归。一个电子表格加上一个脚本就足够起步。

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // 当分数低于阈值时发出警报
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## 实际问题

这些失败并非主要源于嵌入模型或向量数据库本身，而是围绕它们的系统设计。

RAG 演示之所以成功，是因为演示环境受控：文档干净、问题格式良好、评估宽容。进入生产后，这些条件全部失效。

上述每一种失败都可以被诊断，但前提是你在度量。可靠的 RAG 团队并未使用任何奇技淫巧，他们把检索质量当作真实子系统来对待，而不是演示产物。

先搭建评估循环。一旦可以度量，其他一切都会变得更容易。
