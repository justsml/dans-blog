RAG डेमो हमेशा शानदार दिखता है।

आप कुछ डॉक्स (docs) को एम्बेड करते हैं, एक वेक्टर स्टोर खड़ा करते हैं, उसे चैट से जोड़ते हैं, और मॉडल को अपने इंटरनल नॉलेज बेस का ऐसे हवाला देते हुए देखते हैं जैसे वह सालों से वहां काम कर रहा हो। खूबसूरत। स्टेकहोल्डर्स प्रभावित होते हैं। कोई कहता है "चलो इसे शिप करते हैं।"

छह हफ्ते बाद, यूज़र्स को आत्मविश्वास से भरे गलत जवाब मिल रहे हैं। सपोर्ट टिकटों का ढेर लग जाता है। सिस्टम *काम* तो करता है, लेकिन उस तरह नहीं जिसकी लोगों को ज़रूरत है।

असफलता आमतौर पर किसी एक बड़ी गलती की वजह से नहीं होती। यह पांच छोटी गलतियों का जोड़ होती है।

## 1. आपके चंक्स (Chunks) का साइज़ गलत है

यह विफलता सिस्टम को क्रैश नहीं करती। यह बस हर जवाब को थोड़ा और खराब कर देती है, जब तक कि पूरा फीचर अविश्वसनीय महसूस न होने लगे।

वेक्टर सर्च *चंक्स* को निकालता है, डॉक्युमेंट्स को नहीं। आप अपनी सोर्स सामग्री को जिस भी टुकड़े में बांटते हैं, वही रिट्रीवर (retriever) के लिए 'यूनिट ऑफ ट्रुथ' बन जाता है। अगर चंक्स गलत हैं, तो मॉडल गलत टुकड़ों के साथ जवाब देता है।

**बहुत छोटा (Too small)**: चंक में अधूरा जवाब होता है। एम्बेडिंग सही विषय को तो पकड़ लेती है, लेकिन निकाले गए टेक्स्ट में कॉन्टेक्स्ट गायब होता है। आप "अधिकतम टाइमआउट 30 सेकंड है" तो निकाल लेते हैं, लेकिन उससे पहले वाला वाक्य "लिगेसी API का उपयोग करते समय" छूट जाता है।

**बहुत बड़ा (Too large)**: एम्बेडिंग कई विचारों का एक धुंधला औसत बन जाती है। सिमेंटिक सर्च (Semantic search) भ्रमित हो जाता है क्योंकि चंक कई चीज़ों के बारे में होता है, और परिणामी वेक्टर उनमें से किसी का भी साफ तौर पर प्रतिनिधित्व नहीं करता।

सही चंक साइज़ पूरी तरह से आपके कंटेंट पर निर्भर करता है। टेक्निकल डॉक्स, कानूनी कॉन्ट्रैक्ट्स और सपोर्ट ट्रांसक्रिप्ट्स सभी के चंक्स अलग तरह से बनते हैं। इसका कोई सार्वभौमिक जवाब नहीं है।

क्या करें: मापें (Measure)। अपने कॉर्पस से प्रश्न/उत्तर के जोड़ों का एक इवल सेट (eval set) बनाएँ। 256, 512 और 1024-टोकन चंक्स आज़माएँ। रिट्रीवल प्रिसिजन (retrieval precision) को मापें: क्या सही चंक टॉप 5 में आता है? आप जल्दी ही जान जाएंगे कि चंक साइज़ उस एम्बेडिंग मॉडल से कहीं ज़्यादा मायने रखता है जिस पर आपने घंटों विचार किया था।

ओवरलैप (overlap) का भी इस्तेमाल करें। 64 टोकन के ओवरलैप के साथ 512-टोकन का चंक होने का मतलब है कि सीमाओं के बीच वाले जवाब भी निकाल लिए जाते हैं। अधिकांश वेक्टर लाइब्रेरी इसका समर्थन करती हैं। ज़्यादातर लोग इसे छोड़ देते हैं।

## 2. आपकी एम्बेडिंग्स पुरानी (Stale) हो जाती हैं (और आपको पता भी नहीं चलेगा)

कल्पना कीजिए कि आपकी कंपनी रीब्रांड करती है। या किसी प्रोडक्ट का नाम बदलती है। या अपनी कीमत अपडेट करती है। या किसी API को हटा देती है।

आप डॉक्युमेंटेशन तो अपडेट कर देते हैं लेकिन चंक्स को फिर से एम्बेड (re-embed) नहीं करते। वेक्टर इंडेक्स अभी भी पुराने कंटेंट को ही दर्शाता है।

यूज़र्स नई कीमत के बारे में पूछते हैं। एम्बेडिंग्स उन्हें पुराने कंटेंट की ओर ले जाती हैं। मॉडल पुराना कंटेंट पढ़ता है और आत्मविश्वास के साथ पुरानी कीमत बता देता है। सपोर्ट को एक टिकट मिलता है।

हर गंभीर RAG सिस्टम कभी न कभी इस स्थिति में पहुँचता है। इसका समाधान सरल लगता है - कंटेंट बदलने पर फिर से एम्बेड करें - लेकिन टीमें पहली घटना होने से पहले शायद ही कभी ऐसी पाइपलाइन बनाती हैं।

आपको कंटेंट फिंगरप्रिंटिंग के साथ इंक्रीमेंटल री-इंडेक्सिंग (incremental re-indexing) की आवश्यकता है:

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // सामग्री अपरिवर्तित है, पुन: एम्बेडिंग छोड़ें
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

राइट (write) पर री-इंडेक्स करें, टाइमस्टैम्प के बजाय कंटेंट पर फिंगरप्रिंट करें। आपके CMS में डॉक्युमेंट अक्सर बिना कंटेंट बदले ही अपडेट हो जाते हैं।

## 3. रिट्रीवल प्रिसिजन बनाम रिकॉल: आप गलत चीज़ को ऑप्टिमाइज़ कर रहे हैं

अधिकांश RAG ट्यूटोरियल दिखाते हैं कि टॉप-के (top-K) चंक्स कैसे निकालें। वे उन दो लक्ष्यों के बीच के समझौते (tradeoff) को नहीं समझाते जो विपरीत दिशाओं में इशारा करते हैं।

**हाई रिकॉल (High recall)**: वह सब कुछ लौटाएँ जो प्रासंगिक हो सकता है। यूज़र्स को हमेशा जवाब मिलता है। लेकिन मॉडल की कॉन्टेक्स्ट विंडो प्रासंगिक शोर से भरी होती है, और वह टुकड़ों के बीच के अंतराल को भरने के लिए मतिभ्रम (hallucinate) करने लगता है।

**हाई प्रिसिजन (High precision)**: केवल सबसे प्रासंगिक चंक्स ही लौटाएँ। मॉडल साफ और केंद्रित कॉन्टेक्स्ट के साथ काम करता है। लेकिन अगर सही चंक टॉप 3 में नहीं है, तो मॉडल के पास जानकारी नहीं होती और वह फिर भी आत्मविश्वास के साथ कुछ भी बना देता है।

यूज़र्स को विफलता के दोनों तरीके एक जैसे ही लगते हैं: गलत जवाब। लेकिन कारण, और उनके समाधान, विपरीत हैं।

दो तकनीकें जो वास्तव में मदद करती हैं:

**री-रैंकिंग (Reranking)**: अधिक उम्मीदवारों (top-20) को निकालें, फिर LLM को पास करने से पहले प्रासंगिकता के आधार पर उन्हें री-रैंक करने के लिए क्रॉस-एनकोडर मॉडल का उपयोग करें। क्रॉस-एनकोडर वेक्टर समानता की तुलना में धीमे होते हैं लेकिन अंतिम रैंकिंग स्टेप पर नाटकीय रूप से अधिक सटीक होते हैं।

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // अब टॉप-5 का वास्तव में कोई अर्थ है
```

**हाइब्रिड सर्च (Hybrid search)**: वेक्टर सर्च (सिमेंटिक समानता) को कीवर्ड सर्च (BM25) के साथ मिलाएँ। वे अलग-अलग तरीकों से फेल होते हैं। वेक्टर सर्च विशिष्ट शब्दों, मॉडल नामों और आईडी के साथ संघर्ष करता है। कीवर्ड सर्च पैराफ्रेज़िंग और पर्यायवाची शब्दों के साथ संघर्ष करता है। साथ मिलकर, वे एक-दूसरे की कमियों को ढंक लेते हैं।

## 4. आपकी कॉन्टेक्स्ट विंडो का आकार गलत है

आपने सही चंक्स निकाल लिए हैं। बधाई हो। फिर भी मॉडल इसे गलत करने वाला है।

मुद्दा सिर्फ यह नहीं है कि आप क्या निकालते हैं। मुद्दा यह है कि आप उसे कहाँ रखते हैं।

LLMs "बीच में खो जाने" (lost in the middle) की समस्या से जूझ सकते हैं। लियू एट अल. ने पाया कि लंबे कॉन्टेक्स्ट वाले मॉडल प्रासंगिक जानकारी का उपयोग कम भरोसेमंद तरीके से करते हैं जब वह शुरुआत या अंत के बजाय प्रॉम्प्ट के बीच में दिखाई देती है।

अगर आप 20 चंक्स को एक सपाट लिस्ट में डाल रहे हैं और उम्मीद कर रहे हैं कि मॉडल उन्हें सही ढंग से संश्लेषित (synthesize) करेगा, तो आप परफॉरमेंस गँवा रहे हैं।

चीजें जो वास्तव में मदद करती हैं:

**अपने सबसे प्रासंगिक चंक्स के लिए स्टार्ट/एंड प्लेसमेंट का मूल्यांकन करें।** एक सामान्य तरीका यह है कि टॉप-रैंक वाला पहले, सेकंड-रैंक वाला आखिरी, और बाकी बीच में। यह सुनने में अजीब लगता है, लेकिन इसे अपने मॉडल और प्रॉम्प्ट के आकार के हिसाब से टेस्ट करना ज़रूरी है।

**अपने कॉन्टेक्स्ट सेक्शन्स को स्पष्ट रूप से नंबर और लेबल दें।** `[Source 1]` ... `[Source 2]` मॉडल को सोचने के लिए एंकर (anchors) प्रदान करते हैं।

**एक रिट्रीवल कॉन्फिडेंस सिग्नल जोड़ें।** यदि 0-1 के पैमाने पर आपका समानता स्कोर 0.65 है, तो मॉडल को बताएं: "निम्नलिखित कॉन्टेक्स्ट मध्यम आत्मविश्वास के साथ निकाला गया था। यदि उत्तर स्पष्ट नहीं है तो अनिश्चितता स्वीकार करें।"

**एक कॉन्टेक्स्ट बजट निर्धारित करें।** निकाली गई हर चीज़ को पास न करें। टोकन गिनें, प्रासंगिकता स्कोर के आधार पर प्राथमिकता दें, और मॉडल की कॉन्टेक्स्ट विंडो के 60-70% पर हार्ड-कट लगा दें। मॉडल को बिना थके सोचने के लिए जगह छोड़ें।

संदर्भ: [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172).

## 5. आपको पता ही नहीं होता कि यह कब गलत है

यह एक शांत विफलता है: जवाब आता है, UI स्वस्थ दिखता है, और कंटेंट गलत होता है।

पारंपरिक API के साथ, विफलताएं दिखाई देती हैं: HTTP 500, टाइमआउट, स्कीमा वैलिडेशन एरर। आपको तुरंत पता चल जाता है। RAG की विफलताएं शांत होती हैं: सिस्टम एक प्रतिक्रिया देता है, वह प्रशंसनीय लगती है, और वह गलत होती है।

जब तक यूज़र्स आपको न बताएं, तब तक आपको पता भी नहीं चलेगा कि आपकी RAG पाइपलाइन फेल हो रही है। अक्सर वे बताते भी नहीं हैं। वे बस इस पर भरोसा करना बंद कर देते हैं और वापस Ctrl+F पर चले जाते हैं।

प्रोडक्शन RAG सिस्टम के लिए न्यूनतम आवश्यक ऑब्जर्वेबिलिटी (observability) सेटअप:

**अपनी रिट्रीवल चेन को लॉग करें।** हर क्वेरी, क्या निकाला गया (चंक आईडी + स्कोर), और मॉडल ने क्या परिणाम दिया। किसी भी चीज़ को डीबग करने के लिए आपको इसकी आवश्यकता है।

**रिट्रीवल मैट्रिक्स को ट्रैक करें।** यदि आपके पास ग्राउंड-ट्रुथ लेबल हैं, तो मीन रेसिप्रोकल रैंक (MRR) और NDCG ट्रैक करें। कम से कम, समानता स्कोर डिस्ट्रीब्यूशन को ट्रैक करें — यदि आपका P50 रिट्रीवल स्कोर गिरता है, तो आपकी इंडेक्स क्वालिटी गिर गई है।

**एक फीडबैक लूप बनाएँ।** जवाबों पर एक थम्स अप/डाउन भी, जो क्वेरी और निकाले गए चंक्स से जुड़ा हो, आपको ट्रेनिंग सिग्नल देता है। इसके बिना, आप अंधेरे में उड़ान भर रहे हैं।

**समय-समय पर इवल (evals) चलाएँ।** ज्ञात सही उत्तरों के साथ 50-100 प्रश्नों का एक टेस्ट सेट, जो साप्ताहिक रूप से चलाया जाए, यूज़र्स से पहले ही गड़बड़ी को पकड़ लेगा। शुरू करने के लिए एक स्प्रेडशीट और एक स्क्रिप्ट काफी है।

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`इवल स्कोर: ${(avgScore * 100).toFixed(1)}%`);
  
  // यदि स्कोर थ्रेशोल्ड से नीचे गिरता है तो टीम को सूचित करें
  if (avgScore < 0.75) {
    await notifyTeam(`RAG इवल स्कोर गिरकर ${(avgScore * 100).toFixed(1)}% हो गया`);
  }

  return results;
}
```

## असल समस्या

ये विफलताएं मुख्य रूप से एम्बेडिंग मॉडल या वेक्टर डेटाबेस के बारे में नहीं हैं। ये उनके आस-पास के सिस्टम के बारे में हैं।

RAG डेमो इसलिए काम करते हैं क्योंकि डेमो की स्थितियां नियंत्रित होती हैं: साफ डॉक्युमेंट्स, अच्छी तरह से तैयार किए गए प्रश्न, सहनशील मूल्यांकनकर्ता। प्रोडक्शन इसलिए फेल होता है क्योंकि वहां इनमें से कोई भी स्थिति नहीं होती।

ऊपर दी गई हर विफलता का निदान संभव है, लेकिन तभी जब आप उसे माप रहे हों। विश्वसनीय RAG टीमें कुछ अलग नहीं कर रही हैं। वे रिट्रीवल क्वालिटी को डेमो आर्टिफैक्ट के बजाय एक वास्तविक सब-सिस्टम की तरह मान रही हैं।

सबसे पहले इवल लूप (eval loop) सेट करें। एक बार जब आप इसे माप सकें, तो बाकी सब कुछ आसान हो जाता है।
