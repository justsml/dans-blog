आरएजी प्रदर्शन हमेशा शानदार लगता है।

आप कुछ दस्तावेज़ों को एम्बेड करते हैं, एक सदिश डेटाबेस चलाते हैं, इसे चैट से जोड़ देते हैं और मॉडल को देखते हैं कि यह अपने आंतरिक ज्ञान डेटाबेस का हवाला दे रहा है जैसे कि यह वहाँ सालों काम कर चुका हो। सुंदर। स्टेकहोल्डर्स प्रभावित होते हैं। कोई कहता है कि "चलो इसे लॉन्च कर दें।"

छह सप्ताह बाद, उपयोगकर्ता सुनिश्चित रूप से गलत उत्तर प्राप्त कर रहे हैं। समर्थन टिकट भर जाते हैं। प्रणाली *काम कर रही है*, बस लोगों की आवश्यकताओं के अनुसार नहीं।

असफलता आमतौर पर एक भव्य गलती नहीं होती। यह पांच बोरिंग गलतियाँ एक साथ ढेर कर देते हैं।

---

## 1. आपके चंक आकार में गलत हैं

यह विफलता क्रैश नहीं करती। यह बस हर उत्तर को थोड़ा खराब बनाती है जब तक कि पूरा फीचर अविश्वसनीय महसूर करे।

वेक्टर सर्च *चंक* रिट्राइव करता है, नहीं डॉक्यूमेंट। जो भी आप अपने स्रोत सामग्री को बांटते हैं, वह रिट्राइवर के यूनिट ऑफ ट्रूथ बन जाता है। अगर चंक गलत हैं, तो मॉडल गलत फ्रैगमेंट के साथ उत्तर देता है।

**बहुत छोटा**: चंक में आंशिक उत्तर होता है। एम्बेडिंग सही विषय को पकड़ता है, लेकिन रिट्राइव किया गया पाठ कंटेक्स्ट से वंचित होता है। आप "अधिकतम टाइमआउट 30 सेकंड है" ले आते हैं लेकिन इसके पहले वाली वाक्यांश के बिना जो "जब लेगेसी API का उपयोग किया जाता है" कहता है।

**बहुत बड़ा**: एम्बेडिंग कई विचारों का धुंधला औसत बन जाता है। सेमेंटिक सर्च भ्रमित हो जाती है क्योंकि चंक कई चीजों के बारे में है, और परिणामी वेक्टर उनमें से किसी को भी स्पष्ट रूप से प्रतिनिधित्व नहीं करता।

सही चंक आकार पूरी तरह आपकी सामग्री पर निर्भर करता है। तकनीकी दस्तावेज, कानूनी समझौते, और समर्थन ट्रांसक्रिप्ट सभी अलग-अलग चंक होते हैं। कोई सार्वभौमिक उत्तर नहीं है।

क्या करें: मापें। अपने कॉर्पस से प्रश्न/उत्तर जोड़ियों का एक एवल सेट बनाएं। 256, 512, और 1024-टोकन चंक की कोशिश करें। रिट्रीवल परिशुद्धता को मापें: शीर्ष 5 में सही चंक दिखाई देता है? आप जल्दी से सीख जाएंगे कि चंक आकार एम्बेडिंग मॉडल जिस पर आप चिंतित रहे, से अधिक महत्वपूर्ण है।

ओवरलैप का उपयोग भी करें। 512-टोकन चंक के साथ प्रत्येक ओर 64 टोकन का ओवरलैप बाउंडरी-स्पैनिंग उत्तरों को अभी भी पुनर्प्राप्त करने की अनुमति देता है। अधिकांश वेक्टर लाइब्रेरीज इसका समर्थन करती हैं। अधिकांश लोग इसे छोड़ देते हैं।

## 2. आपके एम्बेडिंग्स स्टेल हो जाते हैं (और आप इसे ध्यान नहीं देंगे)

कल्पना करें कि आपकी कंपनी की ब्रैंडिंग बदल जाती है। या एक उत्पाद का नाम बदल जाता है। या उसकी कीमत अद्यतन हो जाती है। या एक API अपडेट हो जाता है।

आप दस्तावेज़ अद्यतन करते हैं लेकिन चंक को फिर से एम्बेड नहीं करते हैं। वेक्टर इंडेक्स पुराने सामग्री का प्रतिनिधित्व करता रहता है।

उपयोगकर्ता नई कीमत के बारे में पूछते हैं। एम्बेडिंग्स उन्हें पुराने सामग्री पर भेज देते हैं। मॉडल पुराने सामग्री को पढ़ता है और निश्चित रूप से पुरानी कीमत की व्याख्या करता है। समर्थन को एक टिकट मिल जाता है।

प्रत्येक गंभीर RAG प्रणाली अंततः इसे हिट करती है। ठीक ठाक दिखने वाला ठीक है - सामग्री परिवर्तन पर पुनः-एम्बेड करें - लेकिन टीमें पहले घटना से पहले इस पाइपलाइन का निर्माण दुर्लभ रूप से करती हैं।

आपको सामग्री अंगुलिछाप के साथ अंशतः पुनः-सूचकांकन की आवश्यकता है:

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // सामग्री अपरिवर्तित, पुनः-एम्बेडिंग से बचें
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

लिखित डेटा पर पुनः-सूचकांकन करें, समयच्छेदक नहीं, सामग्री पर अंगुलिछाप बनाएं। आपके CMS में दस्तावेज़ हमेशा अद्यतन होते रहते हैं बिना वास्तविक सामग्री में परिवर्तन के।

## 3. पुनर्प्राप्ति सटीकता बनाम पुनर्प्राप्ति: आप गलत एक को अनुकूलित कर रहे हैं

अधिकांश RAG ट्यूटोरियल शीर्ष-K खंड प्राप्त करने का तरीका दिखाते हैं। वे दो लक्ष्यों के बीच के त्याग की व्याख्या नहीं करते जो विपरीत दिशाओं में इशारा करते हैं।

**उच्च पुनर्प्राप्ति**: हर चीज लौटाएं जो संबंधित हो सकती है। उपयोगकर्ता हमेशा एक उत्तर प्राप्त करते हैं। लेकिन मॉडल के संदर्भ खिड़की में अप्रासंगिक शोर भरा होता है, और यह खंडों के बीच अंतराल को भरने के लिए कल्पना करता है।

**उच्च सटीकता**: केवल सबसे संबंधित खंड लौटाएं। मॉडल स्पष्ट और निर्देशित संदर्भ के साथ काम करता है। लेकिन यदि सही खंड शीर्ष 3 में नहीं है, तो मॉडल के पास जानकारी नहीं होती और फिर भी वह निश्चित रूप से कुछ बना देता है।

उपयोगकर्ताओं के लिए विफलता के मोड एक जैसे दिखते हैं: गलत उत्तर। लेकिन कारण और उपाय विपरीत होते हैं।

दो तकनीकें जो वास्तव में मदद करती हैं:

**पुनर्क्रमण (Reranking)**: अधिक उम्मीदवार प्राप्त करें (शीर्ष-20), फिर एक क्रॉस-एन्कोडर मॉडल का उपयोग करके उन्हें प्रासंगिकता के आधार पर पुनर्क्रमित करें। क्रॉस-एन्कोडर सदिश समानता की तुलना में धीमे होते हैं लेकिन अंतिम क्रमांकन चरण में बेहद सटीक होते हैं।

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // अब शीर्ष-5 वास्तव में कुछ मतलब रखता है
```

**हाइब्रिड सर्च**: सदिश सर्च (सेमांटिक समानता) के साथ शब्द खोज (BM25) को जोड़ें। वे अलग-अलग तरीकों से विफल होते हैं। सदिश सर्च विशिष्ट शब्द, मॉडल नामों और आईडी के साथ संघर्ष करता है। शब्द खोज पुनर्व्याख्या और समानार्थी शब्दों के साथ संघर्ष करता है। एक साथ, वे एक दूसरे के अंधेरे क्षेत्रों को कवर करते हैं।

## 4. आपकी संदर्भ खिड़की गलत आकार की है

आपने सही खंड पुनर्प्राप्त कर लिए हैं। बधाई। मॉडल अभी भी इसे गलत तरीके से प्राप्त करेगा।

मुद्दा केवल आपके द्वारा पुनर्प्राप्त किए गए डेटा से नहीं है। यह आप उसे कहाँ रखते हैं उससे भी है।

एलएलएम (LLM) "मध्य में खो जाने" (lost in the middle) की समस्या से पीड़ित हो सकते हैं। लियू एट अल। लंबे संदर्भ मॉडलों का मूल्यांकन करते हुए पाया कि जब संबंधित जानकारी प्रॉम्प्ट के शुरुआत या अंत के पास न होकर मध्य में हो तो उसकी विश्वसनीयता कम हो जाती है।

अगर आप 20 खंडों को एक समतल सूची में भर रहे हैं और मॉडल के सही तरीके से संश्लेषित करने की उम्मीद कर रहे हैं, तो आप प्रदरशन को तालिका पर छोड़ दे रहे हैं।
---

वे चीजें जो वास्तव में मदद करती हैं:

**अपने सबसे प्रासंगिक खंडों के शुरुआत/अंत की स्थिति का मूल्यांकन करें।** एक सामान्य अनुमान यह है कि शीर्ष रैंकिंग वाले पहले, द्वितीय रैंकिंग वाले अंत में, और शेष मध्य में रखे जाएं। यह अंतर्ज्ञानविरोधी है, लेकिन अपने मॉडल और प्रॉम्प्ट आकार के खिलाफ परीक्षण करने लायक है।

**अपने संदर्भ अनुभागों को स्पष्ट रूप से नंबर दें और लेबल करें।** `[Source 1]` ... `[Source 2]` मॉडल को तर्क करने के लिए आंकड़े देता है।

**एक पुनर्प्राप्ति आत्मविश्वास संकेत जोड़ें।** अगर आपका समानता स्कोर 0-1 पैमाने पर 0.65 है, तो मॉडल को बताएं: "निम्नलिखित संदर्भ को मध्यम आत्मविश्वास के साथ पुनर्प्राप्त किया गया था। यदि उत्तर अस्पष्ट है तो अनिश्चितता को स्वीकार करें।"

**एक संदर्भ बजट निर्धारित करें।** अपने द्वारा पुनर्प्राप्त सभी चीजों को न भेजें। टोकन गिनें, प्रासंगिकता स्कोर द्वारा प्राथमिकता दें, और मॉडल के संदर्भ खिड़की के 60-70% तक काट दें। मॉडल को तर्क करने के लिए जगह छोड़ें ताकि यह अव्यवस्था में न फंसे।

संदर्भ: [मध्य में खो जाना: भाषा मॉडल कैसे लंबे संदर्भों का उपयोग करते हैं](https://arxiv.org/abs/2307.03172).

## 5. आपको इसके गलत होने का कोई अंदाजा नहीं है

यह शांत विफलता है: उत्तर लौट आता है, UI स्वस्थ लगता है, और सामग्री गलत होती है।

एक पारंपरिक API के साथ, विफलताएं दृश्यमान होती हैं: HTTP 500, समय सीमा, स्कीमा वैधता त्रुटि। आप तुरंत पता करते हैं। RAG विफलताएं शांत होती हैं: प्रणाली एक प्रतिक्रिया लौटाती है, यह संभावना से भरा लगता है, और यह गलत होता है।

आपको यह पता नहीं चल सकता कि आपका RAG पाइपलाइन विफल हो रहा है जब तक कि उपयोगकर्ता आपको नहीं बताते हैं। अक्सर वे नहीं बताते हैं। वे इस पर भरोसा करना बंद कर देते हैं और Ctrl+F पर वापस आ जाते हैं।

एक उत्पादन RAG प्रणाली के लिए न्यूनतम अवलोकनीय स्थापना:

**अपनी पुनर्प्राप्ति श्रृंखला का लॉग करें।** हर प्रश्न के लिए, क्या पुनर्प्राप्त किया गया (चूक संख्याएं + स्कोर) और मॉडल ने क्या उत्पन्न किया। आपको इसकी आवश्यकता है किसी भी चीज के डीबग करने के लिए।

**पुनर्प्राप्ति में आंकड़े ट्रैक करें।** अगर आपके पास जमीनी सत्य लेबल हैं, तो माध्य विपरीत रैंक (MRR) और NDCG का ट्रैक करें। न्यूनतम, समानता स्कोर वितरण का ट्रैक करें — अगर आपका P50 पुनर्प्राप्ति स्कोर गिर जाता है, तो आपकी सूचकांक गुणवत्ता गिर गई है।

**एक प्रतिक्रिया लूप बनाएं।** यहां तक कि प्रतिक्रियाओं पर अंगूठा ऊपर/नीचे भी, जो प्रश्न और पुनर्प्राप्त चूक से जुड़ा हो, आपको प्रशिक्षण संकेत देता है। इसके बिना, आप अंधेरे में उड़ रहे हैं।

**समय-समय पर मूल्यांकन चलाएं।** 50-100 प्रश्नों का एक परीक्षण सेट, जिसमें ज्ञात सही उत्तर हों, जिसे सप्ताह में एक बार चलाना, उपयोगकर्ताओं से पहले प्रतिक्रिया को पकड़ेगा। एक स्प्रेडशीट और एक स्क्रिप्ट शुरू करने के लिए पर्याप्त है।

```typescript
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // अगर स्कोर निर्धारित नीचे गिर जाता है तो चेतावनी दें
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## वास्तविक समस्या

ये विफलताएँ मुख्य रूप से एम्बेडिंग मॉडल या वेक्टर डेटाबेस के बारे में नहीं हैं। ये उनके चारों ओर के सिस्टम के बारे में हैं।  

RAG डेमो काम करते हैं क्योंकि डेमो की स्थितियाँ नियंत्रित होती हैं: स्वच्छ दस्तावेज, अच्छी तरह से बनाए गए प्रश्न, सहिष्णु मूल्यांकनकर्ता। उत्पादन विफल हो जाता है क्योंकि इनमें से कोई भी स्थिति नहीं बरकरार रहती।  

ऊपर की हर विफलता निर्धारण योग्य है, लेकिन केवल तभी यदि आप माप रहे हैं। विश्वसनीय RAG टीमें कुछ अद्वितीय नहीं कर रही होती हैं। वे पुनर्प्राप्ति गुणवत्ता को एक वास्तविक उप-सिस्टम के रूप में लेती हैं, न कि एक डेमो आर्टिफैक्ट के रूप में।  

पहले मूल्यांकन लूप स्थापित करें। एक बार जब आप माप सकेंगे, तो अन्य सब कुछ आसान हो जाएगा।
