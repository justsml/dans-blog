العرض التجريبي لـ RAG يبدو دائمًا رائعًا.

أنت تقوم بدمج بعض المستندات، وتُنشئ مخزنًا متجهًا، وتربطه بالدردشة، وتراقب النموذج وهو يقتبس من قاعدة المعرفة الداخلية كأنه يعمل هناك منذ سنوات. جميل. ينعجب أصحاب المصلحة. أحدهم يقول "لننشر هذا."

بعد ستة أسابيع، يبدأ المستخدمون في الحصول على إجابات خاطئة بثقة. تتراكم تذاكر الدعم. النظام *يعمل*، فقط ليس بالطريقة التي يحتاجها الناس.

الفشل عادةً ليس خطأً درامياً واحدًا. إنه خمسة أخطاء مملة مكدسة معًا.

---

## 1. أحجام القطع غير صحيحة

هذا الفشل لا يتسبب في تعطل. إنه يجعل كل إجابة أسوأ قليلًا حتى يصبح الميزة بأكملها غير موثوقة.

بحث المتجهات يسترجع *القطع*، وليس المستندات. ما تقسمه من مادة المصدر يصبح وحدة الحقيقة للمسترجع. إذا كانت القطع غير صحيحة، يجيب النموذج بقطاعات خاطئة.

**صغيرة جدًا**: القطعة تحتوي على إجابة جزئية. التضمين يلتقط الموضوع الصحيح، لكن النص المسترجع يفتقر إلى السياق. تسترجع "الحد الأقصى للمهلة هو 30 ثانية" دون الجملة التي تسبقها وتقول "عند استخدام واجهة برمجة التطبيقات القديمة."

**كبيرة جدًا**: يصبح التضمين متوسطًا ضبابيًا للعديد من الأفكار. البحث الدلالي يختلط لأنه القطعة تتناول عدة أمور، وبالتالي المتجه الناتج لا يمثل أيًا منها بوضوح.

يعتمد حجم القطعة المناسب بالكامل على محتوىك. الوثائق التقنية، العقود القانونية، ونصوص الدعم تُقسم إلى قطع بأحجام مختلفة. لا توجد إجابة عامة.

ما يجب فعله: القياس. أنشئ مجموعة تقييم من أزواج سؤال/إجابة مأخوذة من مجموعة البيانات الخاصة بك. جرّب قطعًا بطول 256، 512، و1024 رمزًا. قس دقة الاسترجاع: هل تظهر القطعة الصحيحة ضمن أعلى 5 نتائج؟ ستدرك سريعًا أن حجم القطعة يؤثر أكثر من نموذج التضمين الذي عانيت في اختياره.

استخدم أيضًا التداخل. قطعة بطول 512 رمزًا مع تداخل 64 رمزًا على كل جانب تعني أن الإجابات التي تمتد عبر الحدود لا تزال تُسترجع. معظم مكتبات المتجهات تدعم ذلك. معظم الأشخاص يتجاهلون ذلك.

## 2. التضمينات تصبح قديمة (ولن تلاحظ ذلك)

تخيل أن شركتك تغير علامتها التجارية. أو تعيد تسمية منتج. أو تُحدّث أسعارها. أو تُوقف واجهة برمجة تطبيقات.

تُحدّث الوثائق لكن لا تُعيد تضمين القطع. لا يزال فهرس المتجهات يمثل المحتوى القديم.

يسأل المستخدمون عن الأسعار الجديدة. توجّههم التضمينات إلى المحتوى القديم. يقرأ النموذج المحتوى القديم ويشرح الأسعار القديمة بثقة. يتلقى فريق الدعم تذكرة.

كل نظام RAG جاد يواجه هذا في النهاية. الحل يبدو بديهيًا – أعد التضمين عند تغيير المحتوى – لكن الفرق نادراً ما يبني هذه العملية قبل وقوع الحادثة الأولى.

تحتاج إلى فهرسة متزايدة مع بصمة محتوى:

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Content unchanged, skip re-embedding
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

إعادة الفهرسة عند الكتابة، والبصمة على المحتوى وليس على الطوابع الزمنية. المستندات تُحدَّث في نظام إدارة المحتوى باستمرار دون أن يتغير المحتوى الفعلي.

## 3. دقة الاسترجاع مقابل الاستدعاء: أنت تُحسّن العنصر الخطأ

معظم دروس RAG تُظهر كيفية استرجاع أعلى K قطع. لا يوضحون الموازنة بين هدفين يتجهان في اتجاهين متعاكسين.

**استدعاء عالي**: إرجاع كل ما قد يكون ذا صلة. يحصل المستخدم دائمًا على إجابة. لكن نافذة سياق النموذج تُملأ بالضوضاء المتفرقة، فيُضيف النموذج معلومات وهمية لسد الفجوات بين القطع.

**دقة عالية**: إرجاع القطع الأكثر صلة فقط. يعمل النموذج بسياق نظيف ومركز. لكن إذا لم تكن القطعة الصحيحة ضمن أعلى 3، لا يمتلك النموذج المعلومات ويختلق إجابة بثقة.

أنماط الفشل تبدو للمستخدمين متطابقة: إجابات خاطئة. لكن الأسباب والحلول معكوسة.

تقنيتان فعليًا تساعدان:

**إعادة الترتيب**: استرجع عددًا أكبر من المرشحين (أعلى 20)، ثم استخدم نموذج cross‑encoder لإعادة ترتيبهم حسب الصلة قبل تمريرهم إلى النموذج اللغوي. الـ cross‑encoders أبطأ من تشابه المتجهات لكن دقتها أعلى بكثير في خطوة الترتيب النهائية.

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // الآن أعلى 5 يعني شيئًا فعليًا
```

**البحث الهجين**: دمج البحث المتجه (التشابه الدلالي) مع البحث بالكلمة المفتاحية (BM25). كل منهما يفشل بطرق مختلفة. البحث المتجه يواجه صعوبة مع المصطلحات المحددة، أسماء النماذج، والمعرفات. البحث بالكلمة المفتاحية يواجه صعوبة مع الصياغة المتنوعة والمرادفات. معًا يغطيان نقاط الضعف لدى بعضهما.

## 4. نافذة السياق لديك ذات الشكل الخاطئ

لقد استرجعت القطع الصحيحة. مبروك. النموذج لا يزال سيخطئ.

المشكلة ليست فقط ما تسترجعه. بل أين تضعه.

يمكن أن تعاني نماذج اللغة الكبيرة من مشكلة “الضياع في الوسط”. قيّمت دراسة Liu وآخرون نماذج السياق الطويل ووجدت أنها تستخدم المعلومات ذات الصلة بشكل أقل موثوقية عندما تظهر في وسط المطالبة بدلاً من قرب البداية أو النهاية.

إذا كنت تدفع 20 قطعة إلى قائمة مسطحة وتأمل أن يجمع النموذج بينها بشكل صحيح، فأنت تُهدر الأداء.

الأشياء التي تساعد فعليًا:

**قَيِّم وضع البداية/النهاية لأكثر القطع صلة.** قاعدة إرشادية شائعة هي وضع الأعلى تصنيفًا أولاً، والثاني تصنيفًا أخيرًا، والبقية في الوسط. قد يبدو ذلك غير بديهي، لكنه يستحق الاختبار مع نموذجك وشكل المطالبة.

**رقِّم ووسم أقسام السياق صراحة.** `[Source 1]` … `[Source 2]` يمنح النموذج نقاط ارتكاز للتفكير.

**أضف إشارة ثقة الاسترجاع.** إذا كان معدل التشابه لديك 0.65 على مقياس من 0 إلى 1، أخبر النموذج: "تم استرجاع السياق التالي بثقة متوسطة. اعترف بالعدم اليقين إذا كان الجواب غير واضح."

**حدد ميزانية السياق.** لا تمرّر كل ما استرجعته. احسب عدد الرموز، أعطِ الأولوية حسب درجة الصلة، واقطع بحزم عند 60‑70 % من نافذة سياق النموذج. اترك مساحة للنموذج للتفكير دون أن ينهك الذاكرة.

المرجع: [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172).

## 5. لا تعرف متى يكون الخطأ

هذا هو الفشل الصامت: يُعيد الجواب، تبدو الواجهة صحية، لكن المحتوى خاطئ.

مع واجهة برمجة تطبيقات تقليدية، تكون الأخطاء مرئية: HTTP 500، مهلة، خطأ في التحقق من المخطط. تعرف ذلك فورًا. أخطاء RAG أهدأ: النظام يُعيد استجابة، تبدو معقولة، لكنها خاطئة.

قد لا تعرف أن خط أنابيب RAG الخاص بك يفشل إلا عندما يخبرك المستخدمون. غالبًا لا يفعلون ذلك. فقط يتوقفون عن الثقة به ويعودون إلى Ctrl+F.

إعداد الحد الأدنى للملاحظة القابلة للتطبيق في نظام RAG إنتاجي:

**سجّل سلسلة الاسترجاع.** كل استعلام، ما تم استرجاعه (معرفات القطع + الدرجات)، وما أنتجه النموذج. تحتاج هذا لتصحيح أي مشكلة.

**تتبّع مقاييس الاسترجاع.** متوسط رتبة الاسترجاع العكسي (MRR) وNDCG إذا كان لديك تسميات صحيحة. على الأقل، راقب توزيعات درجات التشابه — إذا انخفضت درجة الاسترجاع P50، فإن جودة الفهرس قد تدهورت.

**أنشئ حلقة تغذية راجعة.** حتى إشارة إبهام لأعلى/لأسفل على الردود، المرتبطة بالاستعلام والقطع المسترجعة، توفر إشارة تدريب. بدونها، تعمل عميًا.

**قم بتشغيل تقييمات دورية.** مجموعة اختبار من 50‑100 سؤال مع إجابات صحيحة معروفة، تُجرى أسبوعيًا، ستكشف الانحدار قبل أن يلاحظه المستخدمون. جدول بيانات وسكربت يكفيان للبدء.

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // Alert if score drops below threshold
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## المشكلة الفعلية

هذه الإخفاقات لا تتعلق أساسًا بنموذج التضمين أو قاعدة البيانات المتجهية. إنها تتعلق بالنظام المحيط بهما.

تعمل عروض RAG لأن ظروف العرض مُتحكم فيها: مستندات نظيفة، أسئلة مُصاغة جيدًا، مقيمين متسامحين. تفشل الإنتاجية لأن لا أحد من هذه الشروط يتحقق.

كل فشل مذكور أعلاه قابل للتشخيص، ولكن فقط إذا كنت تقيس. الفرق الموثوقة في RAG لا تقوم بأي شيء غريب. إنها تتعامل مع جودة الاسترجاع كعنصر فرعي حقيقي بدلاً من مجرد أثر عرض تجريبي.

قم بإعداد حلقة التقييم أولًا. كل شيء آخر يصبح أسهل بمجرد أن تتمكن من قياسه.
