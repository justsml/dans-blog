La démo RAG est toujours belle.

Vous intégrez quelques documents, vous lancez un vecteur de recherche, vous le branchez à un chat, et vous voyez le modèle citer votre base de connaissances internes comme s'il y avait travaillé des années. Magnifique. Les parties prenantes sont impressionnées. Quelqu'un dit : "mettons ça en production."

Six semaines plus tard, les utilisateurs obtiennent des réponses fausses avec assurance. Les tickets de support s'accumulent. Le système *fonctionne*, juste pas de la façon dont les gens en ont besoin.

L'échec est rarement une erreur dramatique. Ce sont cinq erreurs banales empilées ensemble.

---

## 1. Vos morceaux sont de la mauvaise taille

Cet échec ne fait pas planter. Il rend juste chaque réponse légèrement moins bonne jusqu'à ce que toute la fonctionnalité semble peu fiable.

La recherche vectorielle récupère des *morceaux*, pas des documents. Peu importe comment vous divisez votre matière source, cela devient l'unité de vérité du récupérateur. Si les morceaux sont mauvais, le modèle répond avec les mauvais fragments.

**Trop petits** : Le morceau contient une réponse partielle. L'intégration capture le bon sujet, mais le texte récupéré manque de contexte. Vous récupérez "Le délai maximum est de 30 secondes" sans la phrase avant qui dit "lors de l'utilisation de l'API héritée."

**Trop grands** : L'intégration devient une moyenne floue de nombreuses idées. La recherche sémantique est confuse car le morceau concerne plusieurs choses, et le vecteur résultant ne représente proprement aucune d'entre elles.

La bonne taille de morceau dépend entièrement de votre contenu. Les documents techniques, les contrats juridiques et les transcriptions de support se découpent différemment. Il n'y a pas de réponse universelle.

Que faire : mesurez. Créez un ensemble d'évaluation de paires question/réponse à partir de votre corpus. Essayez des morceaux de 256, 512 et 1024 jetons. Mesurez la précision de récupération : le bon morceau apparaît-il dans les 5 premiers ? Vous apprendrez rapidement que la taille du morceau compte plus que le modèle d'intégration sur lequel vous vous êtes torturé.

Utilisez aussi un chevauchement. Un morceau de 512 jetons avec 64 jetons de chevauchement de chaque côté signifie que les réponses qui traversent les limites sont quand même récupérées. La plupart des bibliothèques vectorielles le supportent. La plupart des gens sautent cette étape.

## 2. Vos intégrations deviennent périmées (et vous ne le remarquerez pas)

Imaginez que votre entreprise change de marque. Ou renomme un produit. Ou met à jour ses prix. Ou déprécie une API.

Vous mettez à jour la documentation mais vous ne réintégrez pas les morceaux. L'index vectoriel représente toujours l'ancien contenu.

Les utilisateurs demandent les nouveaux prix. Les intégrations les dirigent vers l'ancien contenu. Le modèle lit l'ancien contenu et explique avec assurance les anciens prix. Le support reçoit un ticket.

Chaque système RAG sérieux rencontre cela finement. La solution semble évidente : réintégrez quand le contenu change, mais les équipes construisent rarement ce pipeline avant le premier incident.

Vous avez besoin de ré-indexation incrémentielle avec empreinte de contenu :

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Contenu inchangé, sautez la réintégration
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

Ré-indexez à l'écriture, empreinte sur le contenu, pas sur les horodatages. Les documents sont mis à jour dans votre CMS tout le temps sans que le contenu réel ne change.

## 3. Précision de récupération vs rappel : vous optimisez le mauvais

La plupart des tutoriels RAG montrent comment récupérer les K morceaux supérieurs. Ils n'expliquent pas le compromis entre deux objectifs qui pointent dans des directions opposées.

**Rappel élevé** : Retournez tout ce qui pourrait être pertinent. Les utilisateurs obtiennent toujours une réponse. Mais le contexte du modèle est rempli de bruit tangentiellement lié, et il hallucine pour combler les lacunes entre les fragments.

**Précision élevée** : Retournez seulement les morceaux les plus pertinents. Le modèle travaille avec un contexte propre et focalisé. Mais si le bon morceau n'est pas dans les 3 premiers, le modèle n'a pas l'information et invente quelque chose avec assurance de toute façon.

Les modes d'échec ressemblent aux utilisateurs : des réponses fausses. Mais les causes, et les correctifs, sont opposés.

Deux techniques qui aident vraiment :

**Reclassement** : Récupérez plus de candidats (top-20), puis utilisez un modèle de croisé pour les reclasse par pertinence avant de passer au LLM. Les croisés sont plus lents que la similarité vectorielle mais considérablement plus précis à l'étape de classement final.

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // Maintenant top-5 signifie vraiment quelque chose
```

**Recherche hybride** : Combine la recherche vectorielle (similarité sémantique) avec la recherche par mots-clés (BM25). Elles échouent de façons différentes. La recherche vectorielle struggle avec les termes spécifiques, les noms de modèles et les IDs. La recherche par mots-clés struggle avec les paraphrases et les synonymes. Ensemble, elles couvrent les angles morts de l'autre.

## 4. Votre fenêtre de contexte est de la mauvaise forme

Vous avez récupéré les bons morceaux. Félicitations. Le modèle va quand même se tromper.

Le problème n'est pas seulement ce que vous récupérez. C'est où vous le placez.

Les LLM peuvent souffrir du problème "perdu au milieu". Liu et al. ont mesuré que les modèles à contexte long utilisaient moins fiablement les informations pertinentes quand elles apparaissaient au milieu du prompt au lieu d'être près du début ou de la fin.

Si vous enfournez 20 morceaux dans une liste plate et espérez que le modèle les synthétise correctement, vous laissez de la performance sur la table.

Les choses qui aident vraiment :

**Évaluez le placement début/fin pour vos morceaux les plus pertinents.** Une heuristique commune est premier rangé en premier, deuxième rangé en dernier, et le reste au milieu. Contre-intuitif, mais vaut la peine d'être testé contre votre modèle et la forme du prompt.

**Numérotez et étiquetez explicitement vos sections de contexte.** `[Source 1]` ... `[Source 2]` donne au modèle des ancres pour raisonner.

**Ajoutez un signal de confiance de récupération.** Si votre score de similarité est 0.65 sur une échelle 0-1, dites-le au modèle : "Le contexte suivant a été récupéré avec une confiance modérée. Reconnaissez l'incertitude si la réponse n'est pas claire."

**Définissez un budget de contexte.** Ne passez pas tout ce que vous avez récupéré. Comptez les jetons, priorisez par score de pertinence, et coupez dur à 60-70% de la fenêtre de contexte du modèle. Laissez de la place pour que le modèle raisonne sans vaciller.

Référence : [Perdu au milieu : comment les modèles de langage utilisent les longs contextes](https://arxiv.org/abs/2307.03172).

## 5. Vous n'avez aucune idée quand c'est faux

C'est l'échec silencieux : la réponse revient, l'UI semble saine, et le contenu est faux.

Avec une API traditionnelle, les échecs sont visibles : HTTP 500, timeout, erreur de validation de schéma. Vous le savez immédiatement. Les échecs RAG sont plus silencieux : le système retourne une réponse, elle semble plausible, et elle est fausse.

Vous ne savez peut-être pas que votre pipeline RAG échoue jusqu'à ce que les utilisateurs vous le disent. Souvent ils ne le font pas. Ils arrêtent juste d'y faire confiance et reviennent à Ctrl+F.

La configuration d'observabilité viable minimale pour un système RAG en production :

**Journalisez votre chaîne de récupération.** Chaque requête, ce qui a été récupéré (IDs de morceaux + scores), et ce que le modèle a produit. Vous avez besoin de cela pour déboguer quoi que ce soit.

**Suivez les métriques de récupération.** MRR et NDCG si vous avez des étiquettes de vérité terrain. Au minimum, suivez les distributions de scores de similarité - si votre score de récupération P50 baisse, la qualité de votre index a baissé.

**Construisez une boucle de rétroaction.** Même un pouce haut/bas sur les réponses, connecté back à la requête et aux morceaux récupérés, vous donne un signal d'entraînement. Sans cela, vous volez à l'aveugle.

**Exécutez des évaluations périodiques.** Un ensemble de test de 50-100 questions avec des réponses correctes connues, exécutées hebdomadairement, capturera la régression avant les utilisateurs. Une feuille de calcul et un script suffisent pour commencer.

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Score d'évaluation : ${(avgScore * 100).toFixed(1)}%`);
  
  // Alerte si le score tombe en dessous du seuil
  if (avgScore < 0.75) {
    await notifyTeam(`Score d'évaluation RAG tombé à ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## Le vrai problème

Ces échecs ne sont pas principalement sur le modèle d'intégration ou la base de données vectorielle. Ils sont sur le système autour d'eux.

Les démos RAG fonctionnent parce que les conditions de démo sont contrôlées : documents propres, questions bien formées, évaluateurs tolérants. La production échoue parce qu'aucune de ces conditions ne tient.

Chaque échec ci-dessus est diagnostiquable, mais seulement si vous mesurez. Les équipes RAG fiables ne font rien d'exotique. Elles traitent la qualité de récupération comme un vrai sous-système au lieu d'un artefact de démo.

Mettez en place la boucle d'évaluation en premier. Tout le reste est plus facile une fois que vous pouvez la mesurer.
