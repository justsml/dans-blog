RAGのデモは、常に素晴らしく見えます。

ドキュメントを埋め込み（Embed）、ベクトルストアを立ち上げ、チャットに繋ぎます。すると、モデルがまるで長年そこで働いているかのように社内のナレッジベースを引用し始めます。美しい光景です。ステークホルダーは感銘を受け、誰かが「これをリリースしよう」と言い出します。

6週間後、ユーザーは「自信満々に間違った回答」を受け取ることになります。サポートチケットが積み上がります。システムは「動いて」はいますが、人々が必要としている形では動いていないのです。

この失敗は、通常、一つの劇的なミスではありません。5つの「退屈な」ミスが積み重なった結果なのです。

---

## 1. チャンクサイズが不適切

この失敗は、システムをクラッシュさせるわけではありません。ただ、機能全体が信頼できなくなるまで、すべての回答を少しずつ悪化させます。

ベクトル検索が取得するのは「ドキュメント」ではなく「チャンク（断片）」です。ソース資料をどのように分割したかが、リトリーバー（検索器）にとっての「真実の単位」になります。チャンクが不適切であれば、モデルは間違った断片を使って回答してしまいます。

**小さすぎる場合**: チャンクに回答の一部しか含まれません。埋め込みは正しいトピックを捉えていますが、取得されたテキストにコンテキストが欠けています。「最大タイムアウトは30秒です」という一文は取得できても、その前の「レガシーAPIを使用する場合」という重要な前提条件が抜け落ちてしまいます。

**大きすぎる場合**: 埋め込みが多くのアイデアの「ぼやけた平均値」になってしまいます。チャンクが複数の事柄について言及しているため、セマンティック検索が混乱し、結果として得られるベクトルがどのトピックも正確に表現できなくなります。

適切なチャンクサイズは、コンテンツに完全に依存します。技術ドキュメント、法的契約書、サポートのトランスクリプトでは、それぞれ最適なチャンキングが異なります。万能な答えはありません。

**対策**: 測定することです。コーパスから質問と回答のペアを作成し、評価セットを構築します。256、512、1024トークンのチャンクを試してみてください。検索精度（Retrieval Precision）を測定します。「正しいチャンクが上位5位以内に入っているか？」を検証するのです。悩んでいた埋め込みモデルの選択よりも、チャンクサイズの方が重要であることにすぐに気づくでしょう。

また、オーバーラップ（重なり）を使用してください。512トークンのチャンクで、前後に64トークンのオーバーラップを持たせれば、境界をまたぐ回答も取得できるようになります。ほとんどのベクトルライブラリがこれをサポートしていますが、多くの人がこれを見落としています。

## 2. 埋め込みが古くなる（そして気づかない）

会社のリブランディング、製品名の変更、価格の更新、APIの廃止などを想像してみてください。

ドキュメントは更新しても、チャンクの再埋め込みを忘れてしまうことがあります。ベクトルインデックスは依然として古いコンテンツを代表したままです。

ユーザーが新しい価格について質問すると、埋め込みは古いコンテンツへと誘導します。モデルは古いコンテンツを読み、自信満々に古い価格を説明します。そしてサポートにチケットが届きます。

真剣にRAGシステムを構築していれば、いつか必ずこの問題に直面します。解決策は明白（コンテンツが変わったら再埋め込みする）ですが、最初の事故が起きる前にこのパイプラインを構築しているチームは稀です。

コンテンツのフィンガープリント（指紋）を用いた増分インデックス作成が必要です。

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // コンテンツに変更がないため、再埋め込みをスキップ
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

書き込み時に再インデックスを行い、タイムスタンプではなくコンテンツでフィンガープリントを取るようにしましょう。CMSのドキュメントは、実際の内容が変わらなくても頻繁に更新されるからです。

## 3. 検索の適合率（Precision）と再現率（Recall）の履き違え

ほとんどのRAGチュートリアルは、上位K個のチャンクを取得する方法を教えるだけです。しかし、相反する2つの目標のトレードオフについては説明しません。

**高い再現率（High Recall）**: 関連する可能性のあるものをすべて返します。ユーザーは常に回答を得られますが、モデルのコンテキストウィンドウは付随的なノイズで埋め尽くされ、モデルは断片間の隙間を埋めるためにハルシネーション（幻覚）を起こします。

**高い適合率（High Precision）**: 最も関連性の高いチャンクのみを返します。モデルはクリーンで焦点の絞られたコンテキストで動作します。しかし、正しいチャンクが上位3位に入っていなければ、モデルは情報を持っていないにもかかわらず、やはり自信満々に何かを捏造します。

ユーザーから見れば失敗モードは同じ「間違った回答」です。しかし、原因と修正方法は正反対です。

実際に役立つ2つのテクニックを紹介します：

**リランキング（Reranking）**: まず多めの候補（上位20個など）を取得し、LLMに渡す前にクロスエンコーダーモデルを使用して関連性でランク付けし直します。クロスエンコーダーはベクトルの類似度計算よりも低速ですが、最終的なランキング精度は劇的に向上します。

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // これで「上位5つ」が真に意味を持ちます
```

**ハイブリッド検索**: ベクトル検索（意味的類似性）とキーワード検索（BM25）を組み合わせます。これらは失敗の仕方が異なります。ベクトル検索は特定の用語やモデル名、IDに弱く、キーワード検索は言い換えや類義語に弱いです。組み合わせることで、お互いの死角を補い合うことができます。

## 4. コンテキストウィンドウの「形状」が不適切

正しいチャンクを取得できたとしましょう。おめでとうございます。それでもモデルは間違える可能性があります。

問題は「何を」取得するかだけでなく、「どこに」置くかにもあります。

LLMは「Lost in the Middle（中央での喪失）」問題に悩まされることがあります。Liuらによる研究では、関連情報がプロンプトの最初や最後ではなく中央にある場合、ロングコンテキストモデルであってもその情報を正確に利用する確率が下がることが示されました。

20個のチャンクをただのリストとして詰め込み、モデルがうまく合成してくれることを期待しているなら、パフォーマンスを無駄にしています。

実際に役立つヒント：

**最も関連性の高いチャンクを最初と最後に配置することを検討する。** 一般的なヒューリスティックとして、最も重要なものを最初に、2番目に重要なものを最後に、残りを中央に配置するという方法があります。直感に反するかもしれませんが、モデルとプロンプトの形状に合わせてテストする価値はあります。

**コンテキストセクションに明示的に番号とラベルを付ける。** `[ソース 1]` ... `[ソース 2]` のように記述することで、モデルが推論するためのアンカー（手がかり）を提供します。

**検索の信頼度信号を追加する。** 類似度スコアが0から1のスケールで0.65程度であれば、モデルにこう伝えます。「以下のコンテキストは中程度の信頼度で取得されました。答えが不明確な場合は不確実性を認めてください。」

**コンテキストの予算を設定する。** 取得したものをすべて渡すのではなく、トークンをカウントし、関連性スコアで優先順位を付け、モデルのコンテキストウィンドウの60〜70%で強制的にカットします。モデルが混乱せずに推論できる余白を残してください。

参考文献: [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172).

## 5. 間違っていることに気づけない

これは「静かな失敗」です。回答は返ってき、UIも正常に見えますが、内容が間違っています。

従来のAPIであれば、HTTP 500、タイムアウト、スキーマ検証エラーなど、失敗は可視化されます。すぐにわかります。しかしRAGの失敗はもっと静かです。システムはレスポンスを返し、それはもっともらしく見え、そして間違っています。

ユーザーに指摘されるまで、RAGパイプラインが失敗していることに気づかないかもしれません。そして多くの場合、ユーザーは指摘さえしてくれません。ただ信頼するのをやめ、Ctrl+F（検索）に戻るだけです。

本番環境のRAGシステムに必要な、最小限のオブザーバビリティ（観測性）セットアップ：

**検索チェーンをログに記録する。** すべてのクエリ、何が取得されたか（チャンクIDとスコア）、そしてモデルが何を生成したかを記録します。デバッグにはこれが必要です。

**検索メトリクスを追跡する。** 正解ラベルがある場合はMRRやNDCGを追跡します。少なくとも、類似度スコアの分布は追跡してください。P50（中央値）の検索スコアが下がれば、インデックスの品質が低下したことを意味します。

**フィードバックループを構築する。** 回答に対する「グッド/バッド」ボタンだけでも、クエリや取得されたチャンクと紐付けることでトレーニング信号になります。これがないと、目隠しをして飛行機を操縦しているようなものです。

**定期的な評価（Eval）を実行する。** 正解がわかっている50〜100問のテストセットを作成し、毎週実行することで、ユーザーが気づく前に退化（リグレッション）を察知できます。最初はスプレッドシートとスクリプトだけでも十分です。

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // スコアがしきい値を下回った場合にアラートを出す
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## 真の問題

これらの失敗は、主に埋め込みモデルやベクトルデータベースの問題ではありません。それらを取り巻く「システム」の問題です。

RAGのデモがうまくいくのは、条件がコントロールされているからです（クリーンなドキュメント、整った質問、寛容な評価者）。本番環境が失敗するのは、それらの条件が一つも成立しないからです。

上記の失敗はすべて診断可能ですが、それは「測定」している場合に限られます。信頼できるRAGを構築しているチームは、エキゾチックなことをしているわけではありません。彼らは検索の品質を「デモの副産物」ではなく、一つの「本物のサブシステム」として扱っているのです。

まずは評価ループ（Eval loop）を構築しましょう。測定さえできれば、他のすべてのことは容易になります。
