RAGのデモは、いつだって素晴らしく見えます。

ドキュメントを埋め込み、ベクトルストアを立ち上げ、チャットにつなぐ。するとモデルは、まるで何年もそこにいた社員みたいに社内ナレッジベースを引用し始めます。見た目はいい。ステークホルダーは感心する。誰かが「これ、出そう」と言う。

6週間後、ユーザーは自信満々に間違った答えを受け取ります。サポートチケットが積み上がる。システムは「動いて」はいますが、人が必要としている形では動いていません。

失敗はたいてい、一発で致命傷になるミスではありません。5つの退屈なミスが積み重なった結果です。

---

## 1. チャンクサイズが間違っている

この失敗はクラッシュを起こしません。ただ、すべての回答を少しずつ悪くしていき、最終的に機能全体を信用できなくします。

ベクトル検索が取り出すのは「ドキュメント」ではなく「チャンク」です。ソース資料をどう分割したかが、リトリーバーにとっての真実の単位になります。チャンクが間違っていれば、モデルは間違った断片で答えます。

**小さすぎる場合**: チャンクには答えの一部しか入っていません。埋め込みは正しい話題を捉えていても、取得したテキストには文脈が足りません。「最大タイムアウトは30秒です」という一文は取れても、その前にある「レガシーAPIを使う場合」という重要な前提が抜け落ちます。

**大きすぎる場合**: 埋め込みは複数の話題が混ざったぼやけた平均になります。チャンクがいろいろなことを同時に語っているせいで、セマンティック検索が混乱し、結果のベクトルはどの話題もきれいに表せません。

適切なチャンクサイズは、コンテンツ次第です。技術文書、契約書、サポートのトランスクリプトでは、それぞれ最適な分け方が違います。万能の答えはありません。

**やること**: 測ることです。コーパスから質問と回答のペアを作り、評価セットを作成してください。256、512、1024トークンのチャンクを試す。検索精度を測る。「正しいチャンクが上位5件に入るか？」を見る。悩んでいた埋め込みモデル選びより、チャンクサイズのほうが効くとすぐ分かります。

それと、オーバーラップも使ってください。512トークンのチャンクに前後64トークンのオーバーラップを入れれば、境界をまたぐ答えも拾えます。ほとんどのベクトルライブラリは対応していますが、多くの人はここを飛ばします。

## 2. 埋め込みが古くなる。そして誰も気づかない

会社のリブランディング、製品名の変更、価格改定、APIの廃止を想像してください。

ドキュメントは更新したのに、チャンクの再埋め込みを忘れる。ベクトルインデックスは、古い内容のままです。

ユーザーが新しい価格を聞く。埋め込みは古い内容へ誘導する。モデルは古い内容を読み、古い価格を自信満々に説明する。そしてサポートにチケットが届きます。

本気でRAGをやるなら、いつか必ずここでつまずきます。解決策は明白です。内容が変わったら再埋め込みする。ただ、最初の事故が起きる前にこのパイプラインを作っているチームは多くありません。

必要なのは、コンテンツのフィンガープリントを使った増分インデックスです。

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // 内容が変わっていないので再埋め込みは不要
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

書き込み時に再インデックスし、タイムスタンプではなく内容でフィンガープリントを取りましょう。CMSの文書は、実際の中身が変わっていなくても頻繁に更新されるからです。

## 3. Precision と Recall を取り違えている

ほとんどのRAGチュートリアルは、上位K件のチャンクをどう取得するかしか教えません。相反する2つの目標のトレードオフは説明しません。

**高い再現率（High Recall）**: 関連しそうなものを全部返す。ユーザーは常に答えを得られますが、モデルのコンテキストウィンドウは関係ないノイズで埋まり、断片の隙間を埋めるためにハルシネーションします。

**高い適合率（High Precision）**: 最も関連性の高いチャンクだけを返す。モデルはきれいで焦点の定まったコンテキストで動きます。しかし、正しいチャンクが上位3位に入っていなければ、情報がないまま、それでも自信満々に何かを作ります。

ユーザーから見れば、失敗モードはどちらも同じ「間違った答え」です。ただし、原因と直し方は真逆です。

実際に効く手法は2つあります。

**リランキング**: まず多めに候補を取り（上位20件など）、LLMに渡す前にクロスエンコーダーモデルで関連度順に並べ直します。クロスエンコーダーはベクトル類似度より遅いですが、最終ランキングの精度はかなり上がります。

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // これで「上位5件」に意味が出る
```

**ハイブリッド検索**: ベクトル検索（意味的類似）とキーワード検索（BM25）を組み合わせます。失敗の仕方が違うからです。ベクトル検索は特定の用語、モデル名、IDに弱い。キーワード検索は言い換えと類義語に弱い。両方を使えば、互いの死角を埋められます。

## 4. コンテキストウィンドウの形が悪い

正しいチャンクを取れたとしましょう。おめでとうございます。それでもモデルは間違えます。

問題は、何を取るかだけではありません。どこに置くかです。

LLMには「Lost in the Middle（中央での迷子）」問題があります。Liuらの研究では、関連情報がプロンプトの中央にあると、最初や最後にある場合よりも、長文コンテキストモデルがそれをうまく使えないことが示されました。

20個のチャンクをそのまま平たいリストに詰め込んで、モデルがうまく統合してくれると期待するのは、性能を無駄にしています。

実際に効く工夫:

**最も関連性の高いチャンクを最初と最後に置く。** ありがちなヒューリスティックとして、1位を先頭、2位を末尾、それ以外を中央に置きます。直感に反しますが、モデルとプロンプトの形に合わせて試す価値はあります。

**コンテキストセクションに明示的な番号とラベルを付ける。** `[Source 1]` ... `[Source 2]` のように書くと、モデルの推論用アンカーになります。

**検索の信頼度を伝える。** 類似度スコアが0〜1で0.65なら、モデルにこう言います。「以下のコンテキストは中程度の信頼度で取得されました。答えが曖昧なら、不確実性を認めてください。」

**コンテキスト予算を決める。** 取得したものを全部渡さない。トークンを数え、関連度で優先順位を付け、モデルのコンテキストウィンドウの60〜70%で強制的に切る。モデルが落ち着いて推論できる余白を残してください。

参考文献: [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172).

## 5. 間違っていると分からない

これは静かな失敗です。答えは返る。UIは正常に見える。中身は間違っている。

従来のAPIなら、失敗は見えます。HTTP 500、タイムアウト、スキーマ検証エラー。すぐ分かる。RAGの失敗はもっと静かです。システムは返答し、それはもっともらしく見え、そして間違っています。

ユーザーに言われるまで、RAGパイプラインが壊れていると気づかないかもしれません。たいてい、ユーザーは言いません。ただ信頼をやめて、Ctrl+Fに戻るだけです。

本番RAGに必要な最低限のオブザーバビリティ:

**検索チェーンをログに残す。** すべてのクエリ、何を取得したか（チャンクIDとスコア）、そしてモデルが何を出したかを記録する。デバッグにはこれが必要です。

**検索メトリクスを追う。** 正解ラベルがあるならMRRやNDCGを追う。最低でも類似度スコアの分布は見る。P50の検索スコアが落ちたら、インデックスの品質が落ちたということです。

**フィードバックループを作る。** 回答への「いいね/悪いね」だけでも、クエリと取得チャンクに紐づければ学習信号になります。これがなければ、目隠しで飛行機を操縦しているのと同じです。

**定期的にevalを回す。** 正解が分かっている50〜100問のテストセットを作り、毎週回せば、ユーザーが気づく前に劣化を捕まえられます。最初はスプレッドシートとスクリプトで十分です。

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // スコアがしきい値を下回ったら通知する
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## 本当の問題

これらの失敗は、主に埋め込みモデルやベクトルデータベースの問題ではありません。周辺のシステムの問題です。

RAGのデモがうまくいくのは、条件が管理されているからです。きれいな文書、整った質問、寛容な評価者。本番が壊れるのは、そのどれも揃っていないからです。

上の失敗は全部、測っていれば診断できます。信頼できるRAGを作っているチームは、変わったことをしているわけではありません。検索品質をデモの副産物ではなく、ちゃんとしたサブシステムとして扱っているだけです。

まずはevalループを作ってください。測れれば、あとは全部やりやすくなります。
