RAG-демо всегда выглядит отлично.

Вы встраиваете документы, поднимаете векторное хранилище, соединяете его с чатом и наблюдаете, как модель цитирует вашу внутреннюю базу знаний, будто проработала там годами. Красота. Стейкхолдеры впечатлены. Кто-то говорит: «Давайте это отгрузим».

Через шесть недель пользователи получают уверенно неправильные ответы. Копятся тикеты поддержки. Система *работает*, просто не так, как людям нужно.

Проблема обычно не в одной драматичной ошибке. А в пяти скучных, сложенных вместе.

---

## 1. Ваши чанки неправильного размера

Эта ошибка не роняет сервис. Она просто делает каждый ответ чуть хуже, пока вся фича не перестанет вызывать доверие.

Векторный поиск извлекает *чанки*, а не документы. То, на что вы разбили исходный материал, становится единицей истины для ретривера. Если чанки неправильные, модель отвечает неправильными фрагментами.

**Слишком мелкие**: чанк содержит частичный ответ. Эмбеддинг захватывает нужную тему, но извлечённый текст лишён контекста. Вы получаете «Максимальный таймаут — 30 секунд» без предыдущего предложения: «при использовании легаси-API».

**Слишком крупные**: эмбеддинг становится размытым средним множества идей. Семантический поиск путается, потому что чанк о многом сразу, и итоговый вектор не представляет ни одну из идей чисто.

Правильный размер чанка целиком зависит от вашего контента. Техническая документация, юридические контракты и стенограммы поддержки чанкуются по-разному. Универсального ответа нет.

Что делать: измерять. Соберите eval-набор пар вопрос/ответ из вашего корпуса. Попробуйте чанки по 256, 512 и 1024 токенов. Измерьте точность ретривала: появляется ли нужный чанк в топ-5? Вы быстро поймёте, что размер чанка важнее модели эмбеддингов, над которой вы корпели.

Используйте перекрытие. Чанк в 512 токенов с перекрытием по 64 токена с каждой стороны означает, что ответы на границах всё равно будут извлечены. Большинство векторных библиотек это поддерживают. Большинство людей это пропускают.

## 2. Ваши эмбеддинги протухают (и вы этого не заметите)

Представьте, что компания делает ребрендинг. Или переименовывает продукт. Или обновляет цены. Или депрекейтит API.

Вы обновляете документацию, но не пере-эмбеддите чанки. Векторный индекс по-прежнему отражает старый контент.

Пользователи спрашивают про новые цены. Эмбеддинги ведут их к старому контенту. Модель читает старый контент и уверенно объясняет старые цены. Поддержка получает тикет.

Каждая серьёзная RAG-система рано или поздно с этим сталкивается. Решение звучит очевидно — пере-эмбеддить при изменении контента — но команды редко строят этот пайплайн до первого инцидента.

Вам нужно инкрементальное переиндексирование с фингерпринтингом контента:

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Content unchanged, skip re-embedding
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

Переиндексируйте при записи, фингерпринтуйте по контенту, а не по таймстампам. Документы обновляются в CMS постоянно, при этом фактический контент может не меняться.

## 3. Точность против полноты ретривала: вы оптимизируете не то

Большинство туториалов по RAG показывают, как извлекать топ-K чанков. Они не объясняют.tradeoff между двумя целями, которые тянут в разные стороны.

**Высокая полнота (recall)**: вернуть всё, что может быть релевантным. Пользователи всегда получают ответ. Но контекстное окно модели забито шумом, лишь косвенно связанным с запросом, и модель галлюцинирует, заполняя пробелы между фрагментами.

**Высокая точность (precision)**: возвращать только самые релевантные чанки. Модель работает с чистым, сфокусированным контекстом. Но если нужный чанк не попал в топ-3, модель не имеет информации и всё равно уверенно что-то выдумывает.

Режимы отказа выглядят для пользователей одинаково: неправильные ответы. Но причины и исправления — противоположные.

Две техники, которые действительно помогают:

**Реранкинг**: извлеките больше кандидатов (топ-20), затем используйте модель кросс-энкодера для переранжирования по релевантности перед передачей в LLM. Кросс-энкодеры медленнее векторного сходства, но значительно точнее на финальном этапе ранжирования.

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // Now top-5 actually means something
```

**Гибридный поиск**: объедините векторный поиск (семантическое сходство) с ключевым поиском (BM25). Они ошибаются по-разному. Векторный поиск спотыкается о конкретные термины, названия моделей и идентификаторы. Ключевой поиск спотыкается о перефразирование и синонимы. Вместе они закрывают слепые зоны друг друга.

## 4. Ваше контекстное окно неправильной формы

Вы извлекли нужные чанки. Поздравляю. Модель всё равно ответит неправильно.

Проблема не только в том, *что* вы извлекли. А в том, *куда* вы это положили.

LLM страдают от проблемы «потерянного в середине». Liu et al. измерили, что модели с длинным контекстом используют релевантную информацию менее надёжно, когда она находится в середине промпта, а не ближе к началу или концу.

Если вы сваливаете 20 чанков в плоский список и надеетесь, что модель их правильно синтезирует, вы оставляете производительность на столе.

Что действительно помогает:

**Оцените размещение в начале/конце для самых релевантных чанков.** Распространённая эвристика: самый релевантный — первым, второй — последним, остальные — в середине. Контринтуитивно, но стоит протестировать на вашей модели и форме промпта.

**Явно нумеруйте и маркируйте секции контекста.** `[Source 1]` ... `[Source 2]` даёт модели якоря для рассуждений.

**Добавьте сигнал уверенности ретривала.** Если ваш score сходства — 0,65 по шкале 0–1, скажите модели: «Ниже приведён контекст, извлечённый с умеренной уверенностью. Признавайте неопределённость, если ответ неясен».

**Установите бюджет контекста.** Не передавайте всё, что извлекли. Считайте токены, приоритизируйте по score релевантности и жёстко обрезайте на 60–70% контекстного окна модели. Оставьте модели пространство для рассуждений без троттлинга.

Ссылка: [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172).

## 5. Вы не знаете, когда система ошибается

Это тихая ошибка: ответ возвращается, UI выглядит здоровым, а контент — неправильный.

В традиционном API отказы видимы: HTTP 500, таймаут, ошибка валидации схемы. Вы знаете немедленно. Отказы RAG тише: система возвращает ответ, он выглядит правдоподобно, и он неправилен.

Вы можете не знать, что ваш RAG-пайплайн ломается, пока пользователи не скажут вам. Часто они не говорят. Они просто перестают доверять системе и возвращаются к Ctrl+F.

Минимально жизнеспособная наблюдаемость для продакшен-RAG:

**Логируйте цепочку ретривала.** Каждый запрос, что было извлечено (ID чанков + scores) и что произвела модель. Без этого вы ничего не отладите.

**Отслеживайте метрики ретривала.** Mean Reciprocal Rank (MRR) и NDCG, если у вас есть ground-truth метки. Как минимум — отслеживайте распределение scores сходства — если ваш P50 score ретривала упал, качество индекса упало.

**Постройте петлю обратной связи.** Даже палец вверх/вниз на ответах, привязанный к запросу и извлечённым чанкам, даёт обучающий сигнал. Без него вы летите вслепую.

**Запускайте периодические эвалы.** Тестовый набор из 50–100 вопросов с известными правильными ответами, запускаемый еженедельно, поймает регрессии до пользователей. Таблицы и скрипта достаточно для старта.

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // Alert if score drops below threshold
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## Настоящая проблема

Эти отказы — не столько про модель эмбеддингов или векторную базу данных. Сколько про систему вокруг них.

RAG-демо работают, потому что условия демо контролируемые: чистые документы, хорошо сформулированные вопросы, толерантные оценщики. Продакшен ломается, потому что ни одно из этих условий не выполняется.

Каждый отказ выше диагностируем, но только если вы измеряете. Надёжные RAG-команды не делают ничего экзотического. Они относятся к качеству ретривала как к реальной подсистеме, а не к демо-артефакту.

Настройте eval-цикл первым. Всё остальное проще, когда вы можете это измерить.
