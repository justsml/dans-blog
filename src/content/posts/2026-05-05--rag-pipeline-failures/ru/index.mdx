Демо RAG всегда выглядит отлично.

Вы эмбедите документы, поднимаете векторное хранилище, подключаете его к чату и смотрите, как модель цитирует вашу внутреннюю базу знаний, будто проработала там годы. Красота. Заинтересованные стороны впечатлены. Кто-то говорит: «Давайте выпустим это».

Шесть недель спустя пользователи получают уверенно неверные ответы. Тикеты в поддержку растут. Система *работает* — просто не так, как нужно людям.

Проблема обычно не в одной драматической ошибке. Это пять скучных ошибок, сложенных вместе.

---

## 1. Ваши чанки неправильного размера

Этот сбой не вызывает краш. Он просто делает каждый ответ чуть хуже, пока вся фича не начинает выглядеть ненадёжной.

Векторный поиск извлекает *чанки*, а не документы. Всё, на что вы разбили исходный материал, становится единицей истины для ретривера. Если чанки неправильные — модель отвечает не теми фрагментами.

**Слишком маленький**: Чанк содержит частичный ответ. Эмбеддинг захватывает нужную тему, но в извлечённом тексте не хватает контекста. Вы получаете «Максимальный таймаут — 30 секунд» без предшествующего предложения «при использовании старого API».

**Слишком большой**: Эмбеддинг превращается в размытое среднее множества идей. Семантический поиск путается, потому что чанк охватывает несколько тем, и результирующий вектор не представляет ни одну из них чисто.

Правильный размер чанка полностью зависит от вашего контента. Техническая документация, юридические контракты и записки поддержки требуют разного дробления. Универсального ответа нет.

Что делать: измерять. Соберите набор пар вопрос/ответ из вашего корпуса. Попробуйте чанки по 256, 512 и 1024 токена. Измерьте точность извлечения: появляется ли правильный чанк в топ-5? Вы быстро поймёте, что размер чанка важнее, чем модель эмбеддингов, над которой вы бились.

Также используйте перекрытие. Чанк в 512 токенов с перекрытием в 64 токена с каждой стороны гарантирует, что ответы на границах чанков всё равно будут найдены. Большинство векторных библиотек это поддерживают. Большинство людей этим пренебрегают.

## 2. Ваши эмбеддинги устаревают (и вы этого не заметите)

Представьте, что ваша компания ребрендируется. Или переименовывает продукт. Или меняет цены. Или прекращает поддержку API.

Вы обновляете документацию, но не пересоздаёте эмбеддинги. Векторный индекс всё ещё представляет старый контент.

Пользователи спрашивают о новых ценах. Эмбеддинги направляют их к старому контенту. Модель читает старый контент и уверенно объясняет старые цены. В поддержке — тикет.

Каждая серьёзная RAG-система рано или поздно с этим сталкивается. Исправление звучит очевидно — пересоздавать эмбеддинги при изменении контента — но команды редко строят такой пайплайн до первого инцидента.

Вам нужно инкрементальное переиндексирование с fingerprinting'ом контента:

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Контент не изменился, пропускаем переэмбеддинг
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

Переиндексируйте при записи, используйте fingerprint контента, а не временных меток. Документы в CMS обновляются постоянно, но реальное содержимое может и не меняться.

## 3. Точность vs. Полнота извлечения: вы оптимизируете не то

Большинство туториалов по RAG показывают, как извлечь top-K чанков. Они не объясняют компромисс между двумя целями, которые направлены в противоположные стороны.

**Высокая полнота (recall)**: Возвращайте всё, что может быть релевантно. Пользователи всегда получают ответ. Но контекстное окно модели забито посторонним шумом, и она галлюцинирует, чтобы заполнить пробелы между фрагментами.

**Высокая точность (precision)**: Возвращайте только самые релевантные чанки. Модель работает с чистым, сфокусированным контекстом. Но если правильного чанка нет в топ-3, модель не имеет информации и уверенно выдумывает ответ всё равно.

Пользователи видят одно и то же: неверные ответы. Но причины и способы исправления — противоположны.

Две техники, которые реально помогают:

**Переранжирование (reranking)**: Извлеките больше кандидатов (top-20), затем используйте cross-encoder модель, чтобы переранжировать их по релевантности перед передачей LLM. Cross-энкодеры медленнее векторного поиска, но значительно точнее на финальном этапе ранжирования.

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // Теперь топ-5 действительно что-то значит
```

**Гибридный поиск**: Объедините векторный поиск (семантическое сходство) с ключевым поиском (BM25). Они терпят неудачу по-разному. Векторный поиск плохо справляется с конкретными терминами, названиями моделей и ID. Ключевой поиск плох с перефразированиями и синонимами. Вместе они покрывают слепые зоны друг друга.

## 4. Ваше контекстное окно имеет неправильную форму

Вы извлекли правильные чанки. Поздравляю. Модель всё равно ошибётся.

Проблема не только в том, что вы извлекли. Но и в том, куда вы это поместили.

LLM страдают от проблемы «потеряно в середине». Лю и соавторы показали, что модели с длинным контекстом используют релевантную информацию менее надёжно, когда она находится в середине промпта, а не ближе к началу или концу.

Если вы засовываете 20 чанков в плоский список в надежде, что модель синтезирует их правильно — вы оставляете производительность на столе.

Что реально помогает:

**Оцените размещение самых релевантных чанков в начале и конце.** Распространённая эвристика: самый релевантный — первым, второй по релевантности — последним, остальные — в середине. Это контринтуитивно, но стоит протестировать на вашей модели и форме промпта.

**Явно нумеруйте и маркируйте секции контекста.** `[Источник 1]` ... `[Источник 2]` даёт модели якоря для рассуждения.

**Добавьте сигнал уверенности в результатах поиска.** Если ваш показатель сходства — 0.65 по шкале 0–1, скажите модели: «Следующий контекст был извлечён с умеренной уверенностью. Укажи на неопределённость, если ответ неясен.»

**Установите бюджет контекста.** Не передавайте всё, что извлекли. Считайте токены, приоритизируйте по оценке релевантности и жёстко обрезайте на 60–70% контекстного окна модели. Оставьте место для рассуждения без захлёбывания.

Источник: [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172).

## 5. Вы понятия не имеете, когда он ошибается

Это тихий сбой: ответ возвращается, UI выглядит здоровым, а ответ — неверный.

С традиционным API ошибки видны: HTTP 500, таймаут, ошибка валидации схемы. Вы знаете сразу. Сбои RAG тише: система возвращает ответ, он выглядит правдоподобно и является неверным.

Вы можете не знать, что ваш RAG-пайплайн даёт сбой, пока пользователи не скажут вам. Часто они не говорят. Они просто перестают ему доверять и возвращаются к Ctrl+F.

Минимально жизнеспособная наблюдаемость для продакшн RAG-системы:

**Логируйте цепочку извлечения.** Каждый запрос, что было извлечено (ID чанков + оценки) и что модель сгенерировала. Это нужно для отладки чего угодно.

**Отслеживайте метрики извлечения.** Средний обратный ранг (MRR) и NDCG, если у вас есть эталонные метки. Как минимум — отслеживайте распределение оценок сходства: если ваша медианная оценка упала — качество индекса упало.

**Постройте контур обратной связи.** Даже «палец вверх/вниз» на ответах, привязанный к запросу и извлечённым чанкам, даёт обучающий сигнал. Без него вы летите вслепую.

**Запускайте периодические eval'ы.** Тестовый набор из 50–100 вопросов с известными правильными ответами, запускаемый еженедельно, выявит регрессию раньше пользователей. Таблицы и скрипта достаточно для начала.

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Оценка eval: ${(avgScore * 100).toFixed(1)}%`);

  // Оповещение, если оценка упала ниже порога
  if (avgScore < 0.75) {
    await notifyTeam(`Оценка RAG eval упала до ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## Настоящая проблема

Эти сбои связаны не столько с моделью эмбеддингов или векторной базой данных. Они связаны с системой вокруг них.

Демо RAG работают, потому что условия демо контролируемы: чистые документы, хорошо сформулированные вопросы, терпимые оценщики. Продакшн падает, потому что ни одно из этих условий не выполняется.

Каждый из описанных сбоев диагностируем, но только если вы измеряете. Надёжные RAG-команды не делают ничего экзотического. Они относятся к качеству извлечения как к настоящей подсистеме, а не как к демо-артефакту.

Сначала настройте eval-цикл. Всё остальное становится проще, когда вы можете это измерить.
