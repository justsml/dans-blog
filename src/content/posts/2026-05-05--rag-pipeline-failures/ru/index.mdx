RAG-демо всегда выглядят отлично.

Вы встраиваете документацию, поднимаете векторный магазин, подключаете к чату и смотрите, как модель цитирует вашу внутреннюю базу знаний так, будто работает там годами. Красиво. Стейкхолдеры впечатлены. Кто-то говорит «давайте это_ship_».

Через шесть недель пользователи получают уверенно неправильные ответы. Билеты в поддержку накапливаются. Система *работает*, просто не так, как людям нужно.

Отказ обычно не один драматичный провал. Это пять скучных, наложенных друг на друга.

---

## 1. Ваши чанки не того размера

Этот отказ не падает. Он просто делает каждый ответ чуть хуже, пока вся фича не станет ненадёжной.

Векторный поиск извлекает *чанки*, а не документы. whatever вы разделили свой исходный материал на, становится единицей правды ретривера. Если чанки неправильные, модель отвечает неправильными фрагментами.

**Слишком маленький**: чанк содержит частичный ответ. Встраивание захватывает правильную тему, но извлечённый текст не хватает контекста. Вы берёте «Максимальный таймаут — 30 секунд» без предложения перед ним, что говорит «при использовании легаси API».

**Слишком большой**: встраивание становится размытым усреднением многих идей. Семантический поиск путается, потому что чанк о многих вещах, и результирующий вектор не представляет ни одну из них чистым образом.

Правильный размер чанка полностью зависит от вашего контента. Техническая документация, юридические контракты и транскрипты поддержки все чанкуются по-разному. Универсального ответа нет.

Что делать: измеряйте. Постройте eval-набор пар вопрос/ответ из вашего корпуса. Попробуйте 256, 512 и 1024-токенные чанки. Измерьте точность ретрива: появляется ли правильный чанк в топ-5? Вы быстро научитесь, что размер чанка важнее модели встраивания, над которой вы мучитесь.

Также используйте оверлэп. 512-токенный чанк с 64 токенами оверлэпа с каждой стороны означает, что чанки, пересекающие границы, всё равно извлекаются. Большинство векторных библиотек поддерживают это. Большинство людей пропускают.

## 2. Ваши встраивания устаревают (и вы этого не заметите)

Представьте, что ваша компания меняет бренд. Или переименовывает продукт. Или обновляет ценообразование. Или депрекейтит API.

Вы обновляете документацию, но не пере-встраиваете чанки. Векторный индекс всё ещё представляет старый контент.

Пользователи спрашивают о новом ценообразовании. Встраивания направляют их к старому контенту. Модель читает старый контент и уверенно объясняет старое ценообразование. Поддержка получает тикет.

Каждая серьёзная RAG-система сталкивается с этим в какой-то момент. Исправление звучит очевидно — пере-встраивайте при изменении контента — но команды редко строят этот пайплайн до первого инцидента.

Вам нужен инкрементальный ре-индекс с фипринтированием контента:

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Контент не изменился, пропустите пере-встраивание
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

Ре-индекс на записи, фипринт на контенте, а не на временных метках. Документы обновляются в вашем CMS всё время без изменения самого контента.

## 3. Точность ретрива против полноты: вы оптимизируете не то

Большинство RAG-туториалов показывают, как извлечь топ-K чанков. Они не объясняют компромисс между двумя целями, указывающими в противоположных направлениях.

**Высокая полнота**: верните всё, что может быть релевантным. Пользователи всегда получают ответ. Но контекстное окно модели полно косвенно связанных шумов, и она галлюцинирует, чтобы заполнить пробелы между фрагментами.

**Высокая точность**: верните только самые релевантные чанки. Модель работает с чистым, сфокусированным контекстом. Но если правильный чанк не в топ-3, у модели нет информации, и она уверенно придумывает что-то anyway.

Режимы отказа выглядят идентично для пользователей: неправильные ответы. Но причины и исправления противоположны.

Две техники, которые действительно помогают:

**Реранкинг**: извлеките больше кандидатов (топ-20), затем используйте кросс-энкодер модель для реранкинга их по релевантности перед передачей LLM. Кросс-энкодеры медленнее векторного сходства, но драматичнее точные на финальном шаге ранжирования.

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // Теперь топ-5 действительно что-то значит
```

**Гибридный поиск**: объедините векторный поиск (семантическое сходство) с поиском по ключевым словам (BM25). Они отказывают по-разному. Векторный поиск борется с конкретными терминами, именами моделей и ID. Поиск по ключевым словам борется с парафразированием и синонимами. Вместе они покрывают слепые зоны друг друга.

## 4. Ваше контекстное окно неправильной формы

Вы извлекли правильные чанки. Поздравляю. Модель всё равно сделает это неправильно.

Проблема не только в том, что вы извлекаете. Это в том, куда вы это помещаете.

LLM могут страдать от проблемы «потеряны посередине». Liu et al. измерили, что длинно-контекстные модели используют релевантную информацию менее надёжно, когда она появлялась в середине промпта вместо начала или конца.

Если вы заталкиваете 20 чанков в плоский список и надеетесь, что модель синтезирует их правильно, вы оставляете производительность на столе.

Вещи, которые действительно помогают:

**Оцените размещение в начале/конце для ваших самых релевантных чанков**. Общая эвристика — топ-ранжированный первый, второй-ранжированный последний, остальные посередине. Контр-интуитивно, но стоит тестировать против вашей модели и формы промпта.

**Явно пронумеруйте и пометьте свои секции контекста**. `[Источник 1]` ... `[Источник 2]` даёт модели якоря для рассуждения.

**Добавьте сигнал уверенности ретрива**. Если ваш балл сходства 0.65 на шкале 0-1, скажите модели: «Следующий контекст извлечён с умеренной уверенностью. Признайте неопределённость, если ответ неясен».

**Установите бюджет контекста**. Не просто передавайте всё, что вы извлекли. Считайте токены, приоритизируйте по баллу релевантности и жёстко режьте на 60-70% контекстного окна модели. Оставьте место для модели, чтобы она рассуждала без тряски.

Ссылка: [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172).

## 5. Вы не знаете, когда это неправильно

Это тихий отказ: ответ возвращается, UI выглядит здоровым, и контент неверен.

С традиционным API отказы видны: HTTP 500, таймаут, ошибка валидации схемы. Вы знаете немедленно. RAG-отказы тише: система возвращает ответ, он выглядит правдоподобно, и он неверен.

Вы можете не знать, что ваш RAG-пайплайн отказывает, пока пользователи не скажут вам. Часто они не говорят. Они просто перестают доверять ему и идут обратно к Ctrl+F.

Минимально жизнеспособная настройка наблюдаемости для продакшен RAG-системы:

**Логируйте цепочку ретрива**. Каждый запрос, что было извлечено (ID чанков + баллы) и что модель произвела. Вам это нужно для отладки чего угодно.

**Отслеживайте метрики ретрива**. Средний обратный ранг (MRR) и NDCG, если у вас есть ground-truth метки. Минимум, отслеживайте распределения баллов сходства — если ваш P50 балл ретрива падает, качество индекса упало.

**Постройте цикл обратной связи**. Даже большой палец вверх/вниз на ответах, подключенный обратно к запросу и извлечённым чанкам, даёт вам сигнал обучения. Без этого вы летите вслепую.

**Запускайте периодические eval-ы**. Тестовый набор из 50-100 вопросов с известными правильными ответами, запускаемый еженедельно, поймаёт регрессию до того, как это сделают пользователи. Таблицы и скрипта достаточно для начала.

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // Уведомите команду, если балл падает ниже порога
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score упал до ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## Фактическая проблема

Эти отказы не в основном о модели встраивания или векторной базе данных. Это о системе вокруг них.

RAG-демо работают, потому что условия демо контролируемые: чистые документы, хорошо-сформированные вопросы, толерантные оценщики. Продакшен отказывает, потому что ни одно из тех условий не держится.

Каждый отказ выше диагностируем, но только если вы измеряете. Надёжные RAG-команды не делают ничего экзотического. Они относятся к качеству ретрива как к реальному подсистеме, а не артефакту демо.

Настройте eval-цикл первым. Всё остальное легче,once вы можете это измерить.
