Die RAG-Demo sieht immer großartig aus.

Sie einbetten einige Docs, drehen einen Vektorspeicher auf, verdrahten ihn mit Chat, und beobachten, wie das Modell Ihre interne Wissensbasis zitiert, als hätte es dort jahrelang gearbeitet. Schön. Stakeholder sind beeindruckt. Jemand sagt „lassen Sie uns das ausliefern."

Sechs Wochen später erhalten Nutzer zuversichtlich falsche Antworten. Support-Tickets häufen sich. Das System *funktioniert*, nur nicht so, wie die Leute es brauchen.

Das Scheitern ist meist kein dramatischer Fehler. Es sind fünf langweilige, die sich stapeln.

---

## 1. Ihre Chunks sind falsch dimensioniert

Dieser Fehler stürzt nicht ab. Er macht nur jede Antwort leicht schlechter, bis das ganze Feature unzuverlässig wirkt.

Vektorsuche ruft *Chunks* ab, nicht Dokumente. Was auch immer Sie als Einheit Ihrer Quelldaten aufteilen, wird zur Wahrheit des Retrievers. Wenn die Chunks falsch sind, antwortet das Modell mit den falschen Fragmenten.

**Zu klein**: Der Chunk enthält eine teilweise Antwort. Die Einbettung erfasst das richtige Thema, aber der abgerufene Text fehlt Kontext. Sie holen sich „Die maximale Timeout-Zeit beträgt 30 Sekunden" ohne den Satz davor, der sagt „bei Verwendung der Legacy-API."

**Zu groß**: Die Einbettung wird zu einem verschwommenen Durchschnitt vieler Ideen. Semantische Suche wird verwirrt, weil der Chunk über mehrere Dinge handelt, und der resultierende Vektor repräsentiert keines davon sauber.

Die richtige Chunk-Größe hängt vollständig von Ihrem Inhalt ab. Technische Docs, Verträge und Support-Transkripte chunken unterschiedlich. Es gibt keine universelle Antwort.

Was zu tun ist: messen. Bauen Sie eine Eval-Set aus Frage/Antwort-Paaren aus Ihrem Korpus. Probieren Sie 256, 512 und 1024-Token-Chunks. Messen Sie Retrieval-Präzision: erscheint der richtige Chunk unter den Top-5? Sie lernen schnell, dass Chunk-Größe wichtiger ist als das Einbettungsmodell, über das Sie sich gequält haben.

Verwenden Sie auch Overlap. Ein 512-Token-Chunk mit 64 Token Overlap auf jeder Seite bedeutet, dass grenzüberspannende Antworten trotzdem abgerufen werden. Die meisten Vektorbibliotheken unterstützen das. Die meisten Menschen überspringen es.

---

## 2. Ihre Embeddings werden veraltet (und Sie werden es nicht merken)

Stellen Sie sich vor, Ihr Unternehmen rebrandet. Oder benennt ein Produkt um. Oder aktualisiert seine Preise. Oder stellt eine API außer Betrieb.

Sie aktualisieren die Dokumentation, aber re-embedden die Chunks nicht. Der Vektorindex repräsentiert immer noch den alten Inhalt.

Nutzer fragen nach den neuen Preisen. Die Embeddings leiten sie zu den alten Inhalten. Das Modell liest die alten Inhalte und erklärt zuversichtlich die alten Preise. Support bekommt ein Ticket.

Jedes ernsthafte RAG-System trifft das irgendwann. Die Lösung klingt offensichtlich – re-embedden, wenn sich Inhalte ändern – aber Teams bauen selten diese Pipeline vor dem ersten Vorfall.

Sie benötigen inkrementelles Re-Indexing mit Content-Fingerprinting:

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Inhalt unverändert, re-embedding überspringen
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

Re-index beim Schreiben, Fingerprint auf Inhalt, nicht auf Zeitstempel. Dokumente werden in Ihrem CMS oft aktualisiert, ohne dass sich der tatsächliche Inhalt ändert.

## 3. Retrieval-Präzision vs. Recall: Sie optimieren den falschen Wert

Die meisten RAG-Tutorials zeigen, wie man die Top-K-Chunks abruft. Sie erklären nicht den Zielkonflikt zwischen zwei Zielen, die in entgegengesetzte Richtungen zeigen.

**Hoher Recall**: Geben Sie alles zurück, was relevant sein könnte. Nutzer erhalten immer eine Antwort. Aber der Kontext-Window des Modells ist mit randständigem Rauschen voll, und es halluziniert, um die Lücken zwischen Fragmenten zu füllen.

**Hohe Präzision**: Geben Sie nur die relevantesten Chunks zurück. Das Modell arbeitet mit sauberem, fokussiertem Kontext. Aber wenn der richtige Chunk nicht unter den Top-3 ist, hat das Modell die Information nicht und erfindet trotzdem zuversichtlich etwas.

Die Ausfallmuster sehen für Nutzer identisch aus: falsche Antworten. Aber die Ursachen und die Fixes sind entgegengesetzt.

Zwei Techniken, die wirklich helfen:

**Reranking**: Rufen Sie mehr Kandidaten ab (Top-20), dann verwenden Sie ein Cross-Encoder-Modell, um sie nach Relevanz neu zu sortieren, bevor sie an das LLM übergeben werden. Cross-Encoder sind langsamer als Vektorsimilität, aber deutlich genauer beim finalen Ranking.

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // Jetzt bedeutet Top-5 wirklich etwas
```

**Hybrid-Suche**: Kombinieren Sie Vektorsuche (semantische Ähnlichkeit) mit Keywordsuche (BM25). Sie scheitern auf unterschiedliche Weise. Vektorsuche hat Schwierigkeiten mit spezifischen Begriffen, Modellnamen und IDs. Keywordsuche hat Schwierigkeiten mit Paraphrasierung und Synonymen. Zusammen decken sie die Blindstellen des anderen ab.

## 4. Ihr Kontext-Window ist die falsche Form

Sie haben die richtigen Chunks abgerufen. Glückwunsch. Das Modell wird es trotzdem falsch machen.

Das Problem ist nicht nur, was Sie abrufen. Es ist, wo Sie es platzieren.

LLMs können das „Lost-in-the-Middle"-Problem haben. Liu et al. maßen, dass Long-Context-Modelle relevante Informationen weniger zuverlässig verwenden, wenn sie in der Mitte des Prompts erschienen, statt nahe dem Anfang oder Ende.

Wenn Sie 20 Chunks in eine flache Liste schubsen und hoffen, dass das Modell sie korrekt synthetisiert, lassen Sie Leistung auf dem Tisch liegen.

Dinge, die wirklich helfen:

**Bewerten Sie Start/End-Platzierung für Ihre relevantesten Chunks.** Eine gängige Heuristik ist: am höchsten sortiert zuerst, zweit-höchster sortiert zuletzt, der Rest in der Mitte. Kontraintuitiv, aber wert, gegen Ihr Modell und Prompt-Form zu testen.

**Nummerieren und kennzeichnen Sie Ihre Kontextabschnitte explizit.** `[Quelle 1]` ... `[Quelle 2]` gibt dem Modell Anker zum Reasoning.

**Fügen Sie ein Retrieval-Vertrauenssignal hinzu.** Wenn Ihre Ähnlichkeitsbewertung 0.65 auf einer 0-1-Skala ist, sagen Sie dem Modell: „Der folgende Kontext wurde mit moderatem Vertrauen abgerufen. Anerkennen Sie Unsicherheit, wenn die Antwort unklar ist."

**Setzen Sie ein Kontext-Budget.** Geben Sie nicht einfach alles zurück, was Sie abgerufen haben. Zählen Sie Tokens, priorisieren Sie nach Relevanzbewertung, und schneiden Sie hart bei 60-70% des Kontext-Windows des Modells ab. Lassen Sie Raum für das Modell, um zu reasoning, ohne zu thrashen.

Referenz: [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172).

## 5. Sie haben keine Ahnung, wann es falsch ist

Das ist das stille Scheitern: die Antwort kommt zurück, das UI sieht gesund aus, und der Inhalt ist falsch.

Bei einer traditionellen API sind Fehler sichtbar: HTTP 500, Timeout, Schema-Validierungsfehler. Sie wissen es sofort. RAG-Fehler sind leiser: das System gibt eine Antwort zurück, sie wirkt plausibel, und sie ist falsch.

Sie wissen vielleicht nicht, dass Ihre RAG-Pipeline fehlschlägt, bis Nutzer es Ihnen sagen. Oft tun sie das nicht. Sie hören einfach auf, ihr zu vertrauen, und gehen zurück zu Strg+F.

Die minimal brauchbare Observability-Setup für ein Produktions-RAG-System:

**Protokollieren Sie Ihre Retrieval-Kette.** Jede Anfrage, was abgerufen wurde (Chunk-IDs + Scores), und was das Modell produziert hat. Sie brauchen das, um etwas zu debuggen.

**Verfolgen Sie Retrieval-Metriken.** Mean Reciprocal Rank (MRR) und NDCG, wenn Sie Ground-Truth-Labels haben. Mindestens, verfolgen Sie Ähnlichkeits-Score-Verteilungen – wenn Ihr P50-Retrieval-Score sinkt, sinkt Ihre Indexqualität.

**Bauen Sie eine Feedback-Schleife auf.** Selbst ein Daumen hoch/runter auf Antworten, zurückverbunden mit der Anfrage und den abgerufenen Chunks, gibt Ihnen Trainingssignal. Ohne fliegen Sie blind.

**Führen Sie periodische Evals durch.** Eine Testmenge von 50-100 Fragen mit bekannten korrekten Antworten, wöchentlich ausgeführt, fängt Regression ab, bevor Nutzer es tun. Ein Spreadsheet und ein Script reichen zum Starten.

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval-Score: ${(avgScore * 100).toFixed(1)}%`);
  
  // Alarm, wenn Score unter Threshold fällt
  if (avgScore < 0.75) {
    await notifyTeam(`RAG-Eval-Score sank auf ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## Das eigentliche Problem

Diese Fehler sind nicht hauptsächlich über das Einbettungsmodell oder Vektordatenbank. Sie sind über das System darum herum.

RAG-Demos funktionieren, weil die Demo-Bedingungen kontrolliert sind: saubere Dokumente, wohlgeformte Fragen, tolerante Evaluatoren. Produktion scheitert, weil keine dieser Bedingungen gilt.

Jeder Fehler oben ist diagnostizierbar, aber nur, wenn Sie messen. Zuverlässige RAG-Teams machen nichts Exotisches. Sie behandeln Retrieval-Qualität wie ein echtes Subsystem, statt wie ein Demo-Artefakt.

Richten Sie die Eval-Schleife zuerst ein. Alles andere ist einfacher, sobald Sie es messen können.
