---
title: "Fünf Gründe, warum RAG in der Produktion scheitert"
subTitle: "Die Demo hat funktioniert. Bei echten Nutzern bricht die Pipeline."
---

Die RAG-Demo sieht immer großartig aus.

Man bettet ein paar Dokumente ein, setzt einen Vector Store auf, verbindet ihn mit einem Chat und sieht zu, wie das Modell die interne Wissensdatenbank zitiert, als würde es dort seit Jahren arbeiten. Wunderschön. Die Stakeholder sind beeindruckt. Jemand sagt: "Lass uns das releasen."

Sechs Wochen später erhalten die Nutzer souverän vorgetragene falsche Antworten. Die Support-Tickets stapeln sich. Das System *funktioniert* zwar, aber nicht so, wie die Leute es brauchen.

Das Scheitern ist meist kein einzelner, dramatischer Fehler. Es sind fünf banale Fehler, die sich aufeinandertürmen.

---

## 1. Ihre Chunks haben die falsche Größe

Dieser Fehler führt nicht zum Absturz. Er sorgt nur dafür, dass jede Antwort schlechter wird, bis das gesamte Feature unglaubwürdig wirkt.

Die Vektorsuche ruft *Chunks* ab, keine Dokumente. In was auch immer Sie Ihr Quellmaterial aufteilen, wird zur Wahrheitseinheit des Retrievers. Wenn die Chunks falsch sind, antwortet das Modell mit den falschen Fragmenten.

**Zu klein**: Der Chunk enthält eine unvollständige Antwort. Das Embedding erfasst zwar das richtige Thema, aber dem abgerufenen Text fehlt der Kontext. Sie erhalten "Das maximale Timeout beträgt 30 Sekunden", ohne den Satz davor, der besagt: "bei Verwendung der Legacy-API."

**Zu groß**: Das Embedding wird zu einem unscharfen Durchschnitt vieler Ideen. Die semantische Suche wird verwirrt, weil der Chunk von mehreren Dingen handelt und der resultierende Vektor keines davon sauber repräsentiert.

Die richtige Chunk-Größe hängt vollständig von Ihrem Inhalt ab. Technische Dokumentationen, Verträge und Support-Transkripte müssen alle unterschiedlich gechunked werden. Es gibt keine universelle Antwort.

Was zu tun ist: Messen. Erstellen Sie ein Evaluierungs-Set aus Frage/Antwort-Paaren Ihres Korpus. Testen Sie Chunks mit 256, 512 und 1024 Token. Messen Sie die Retrieval-Präzision: Erscheint der richtige Chunk in den Top 5? Sie werden schnell lernen, dass die Chunk-Größe wichtiger ist als das Embedding-Modell, über das Sie sich den Kopf zerbrochen haben.

Nutzen Sie außerdem Overlap (Überlappung). Ein 512-Token-Chunk mit 64 Token Überlappung an jeder Seite bedeutet, dass Antworten, die über Grenzen hinweggehen, trotzdem gefunden werden. Die meisten Vektor-Bibliotheken unterstützen dies. Die meisten Leute lassen es weg.

---

## 2. Ihre Embeddings veralten (und Sie merken es nicht)

Stellen Sie sich vor, Ihr Unternehmen rebrandet. Oder benennt ein Produkt um. Oder aktualisiert die Preise. Oder stellt eine API ein.

Sie aktualisieren die Dokumentation, betten die Chunks aber nicht neu ein. Der Vektor-Index repräsentiert immer noch den alten Inhalt.

Nutzer fragen nach den neuen Preisen. Die Embeddings leiten sie zum alten Inhalt. Das Modell liest den alten Inhalt und erklärt selbstbewusst die alten Preise. Der Support erhält ein Ticket.

Jedes ernsthafte RAG-System stößt irgendwann auf dieses Problem. Die Lösung klingt offensichtlich – neu einbetten, wenn sich der Inhalt ändert –, aber Teams bauen diese Pipeline selten vor dem ersten Vorfall.

Sie benötigen ein inkrementelles Re-Indexing mit Content-Fingerprinting:

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Inhalt unverändert, Re-Embedding überspringen
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

Indizieren Sie beim Schreiben neu, erstellen Sie Fingerprints basierend auf dem Inhalt, nicht auf Zeitstempeln. Dokumente werden in Ihrem CMS ständig aktualisiert, ohne dass sich der eigentliche Inhalt ändert.

## 3. Retrieval Precision vs. Recall: Sie optimieren das Falsche

Die meisten RAG-Tutorials zeigen, wie man die Top-K Chunks abruft. Sie erklären nicht den Kompromiss zwischen zwei Zielen, die in entgegengesetzte Richtungen weisen.

**Hoher Recall**: Alles zurückgeben, was relevant sein könnte. Nutzer erhalten immer eine Antwort. Aber das Kontextfenster des Modells ist voll mit tangential relevantem Rauschen, und es halluziniert, um die Lücken zwischen den Fragmenten zu füllen.

**Hohe Präzision**: Nur die relevantesten Chunks zurückgeben. Das Modell arbeitet mit sauberem, fokussiertem Kontext. Aber wenn der richtige Chunk nicht in den Top 3 ist, fehlen dem Modell die Informationen und es denkt sich trotzdem souverän etwas aus.

Die Fehlermodi sehen für den Nutzer identisch aus: falsche Antworten. Aber die Ursachen und die Lösungen sind gegensätzlich.

Zwei Techniken, die tatsächlich helfen:

**Reranking**: Rufen Sie mehr Kandidaten ab (Top-20) und verwenden Sie dann ein Cross-Encoder-Modell, um sie vor der Übergabe an das LLM nach Relevanz neu zu ordnen. Cross-Encoder sind langsamer als die Vektor-Ähnlichkeit, aber am finalen Ranking-Schritt dramatisch genauer.

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // Jetzt bedeuten die Top-5 wirklich etwas
```

**Hybrid Search**: Kombinieren Sie Vektorsuche (semantische Ähnlichkeit) mit Stichwortsuche (BM25). Beide scheitern auf unterschiedliche Weise. Die Vektorsuche hat Schwierigkeiten mit spezifischen Begriffen, Modellnamen und IDs. Die Stichwortsuche scheitert an Paraphrasierungen und Synonymen. Zusammen decken sie die blinden Flecken des jeweils anderen ab.

## 4. Ihr Kontextfenster hat die falsche Form

Sie haben die richtigen Chunks abgerufen. Herzlichen Glückwunsch. Das Modell wird es trotzdem falsch verstehen.

Das Problem ist nicht nur, *was* Sie abrufen. Es ist, *wo* Sie es platzieren.

LLMs können unter dem "Lost in the Middle"-Problem leiden. Liu et al. stellten fest, dass Long-Context-Modelle relevante Informationen weniger zuverlässig nutzen, wenn diese in der Mitte des Prompts erscheinen, statt am Anfang oder Ende.

Wenn Sie 20 Chunks in eine flache Liste schieben und hoffen, dass das Modell sie korrekt synthetisiert, verschenken Sie Performance.

Dinge, die tatsächlich helfen:

**Evaluieren Sie die Platzierung am Anfang/Ende für Ihre relevantesten Chunks.** Eine gängige Heuristik ist: Bestbewertetes zuerst, zweitbestes zuletzt und der Rest in die Mitte. Kontraintuitiv, aber einen Test gegen Ihr Modell und Ihre Prompt-Struktur wert.

**Nummerieren und kennzeichnen Sie Ihre Kontextabschnitte explizit.** `[Quelle 1]` ... `[Quelle 2]` gibt dem Modell Ankerpunkte für logische Schlussfolgerungen.

**Fügen Sie ein Signal für das Retrieval-Vertrauen hinzu.** Wenn Ihr Ähnlichkeitswert bei 0,65 auf einer Skala von 0-1 liegt, sagen Sie dem Modell: "Der folgende Kontext wurde mit moderater Zuversicht abgerufen. Weisen Sie auf Unsicherheiten hin, falls die Antwort unklar ist."

**Setzen Sie ein Kontext-Budget.** Geben Sie nicht einfach alles weiter, was Sie abgerufen haben. Zählen Sie Token, priorisieren Sie nach Relevanz-Score und schneiden Sie hart bei 60-70% des Kontextfensters des Modells ab. Lassen Sie dem Modell Raum zum "Nachdenken", ohne es zu überfordern.

Referenz: [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172).

## 5. Sie haben keine Ahnung, wann es falsch ist

Dies ist das stille Scheitern: Die Antwort kommt zurück, die UI sieht gesund aus, aber der Inhalt ist falsch.

Bei einer traditionellen API sind Fehler sichtbar: HTTP 500, Timeout, Schema-Validierungsfehler. Man weiß es sofort. RAG-Fehler sind leiser: Das System liefert eine Antwort, sie sieht plausibel aus, aber sie ist falsch.

Sie wissen vielleicht gar nicht, dass Ihre RAG-Pipeline versagt, bis die Nutzer es Ihnen sagen. Oft tun sie das nicht. Sie verlieren einfach das Vertrauen und kehren zu Strg+F zurück.

Das minimale Setup für die Observability eines RAG-Systems in Produktion:

**Loggen Sie Ihre Retrieval-Kette.** Jede Anfrage, was abgerufen wurde (Chunk-IDs + Scores) und was das Modell produziert hat. Das brauchen Sie, um irgendetwas zu debuggen.

**Verfolgen Sie Retrieval-Metriken.** Mean Reciprocal Rank (MRR) und NDCG, falls Sie Ground-Truth-Labels haben. Verfolgen Sie mindestens die Verteilung der Ähnlichkeitswerte – wenn Ihr P50-Retrieval-Score sinkt, hat die Indexqualität abgenommen.

**Bauen Sie eine Feedback-Schleife auf.** Selbst ein Daumen hoch/runter für Antworten, verknüpft mit der Anfrage und den abgerufenen Chunks, gibt Ihnen ein Trainingssignal. Ohne das fliegen Sie blind.

**Führen Sie regelmäßige Evals durch.** Ein Testset von 50-100 Fragen mit bekannten korrekten Antworten, das wöchentlich ausgeführt wird, erkennt Regressionen, bevor es die Nutzer tun. Eine Tabellenkalkulation und ein Skript reichen für den Anfang.

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // Benachrichtigung, wenn der Score unter den Schwellenwert fällt
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## Das eigentliche Problem

Diese Fehler haben nicht primär mit dem Embedding-Modell oder der Vektordatenbank zu tun. Es geht um das System drumherum.

RAG-Demos funktionieren, weil die Bedingungen kontrolliert sind: saubere Dokumente, wohlgeformte Fragen, tolerante Prüfer. Die Produktion scheitert, weil keine dieser Bedingungen zutrifft.

Jeder oben genannte Fehler ist diagnostizierbar, aber nur, wenn man misst. Erfolgreiche RAG-Teams machen nichts Exotisches. Sie behandeln die Retrieval-Qualität wie ein echtes Subsystem statt wie ein Demo-Artefakt.

Richten Sie zuerst den Eval-Loop ein. Alles andere wird einfacher, sobald Sie es messen können.
