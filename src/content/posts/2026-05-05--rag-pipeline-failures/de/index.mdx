Die RAG-Demo sieht immer großartig aus.

Man bettet ein paar Dokumente ein, startet einen Vector Store, verdrahtet alles mit Chat und beobachtet, wie das Modell die interne Wissensdatenbank zitiert, als hätte es dort jahrelang gearbeitet. Wunderschön. Stakeholder sind beeindruckt. Jemand sagt: „Das bringen wir raus."

Sechs Wochen später bekommen Nutzer zuversichtlich falsche Antworten. Support-Tickets häufen sich. Das System *funktioniert*, nur nicht so, wie die Leute es brauchen.

Der Fehler ist normalerweise kein dramatischer Einzelfehler. Es sind fünf langweilige, die sich übereinanderstapeln.

---

## 1. Deine Chunks haben die falsche Größe

Dieser Fehler stürzt nicht ab. Er macht nur jede Antwort etwas schlechter, bis das ganze Feature unzuverlässig wirkt.

Die Vektorsuche findet *Chunks*, nicht Dokumente. Was auch immer du aus deinem Quellmaterial machst, wird zur Wahrheitseinheit des Retrievers. Wenn die Chunks falsch sind, antwortet das Modell mit den falschen Fragmenten.

**Zu klein**: Der Chunk enthält eine Teilantwort. Der Embedding erfasst das richtige Thema, aber der abgerufene Text fehlt der Kontext. Man erhält „Die maximale Zeitüberschreitung beträgt 30 Sekunden" ohne den Satz davor: „bei Verwendung der Legacy-API."

**Zu groß**: Der Embedding wird zu einem unscharfen Durchschnitt vieler Ideen. Die semantische Suche wird verwirrt, weil der Chunk über mehrere Dinge handelt, und der resultierende Vektor repräsentiert keines davon sauber.

Die richtige Chunk-Größe hängt vollständig von deinem Inhalt ab. Technische Dokumente, juristische Verträge und Support-Transkripte werden alle unterschiedlich gechunkt. Es gibt keine universelle Antwort.

Was zu tun ist: messen. Erstelle ein Eval-Set aus Frage-Antwort-Paaren aus deinem Korpus. Probiere 256-, 512- und 1024-Token-Chunks. Miss die Retrieval-Präzision: Erscheint der richtige Chunk in den Top 5? Du lernst schnell, dass die Chunk-Größe wichtiger ist als das Embedding-Modell, über das du gegrübelt hast.

Verwende außerdem Überlappung. Ein 512-Token-Chunk mit 64 Token Überlappung auf jeder Seite bedeutet, dass grenzüberschreitende Antworten trotzdem abgerufen werden. Die meisten Vector-Bibliotheken unterstützen das. Die meisten Leute lassen es weg.

---

## 2. Deine Embeddings veralten (und du bemerkst es nicht)

Stell dir vor, deine Firma macht ein Rebranding. Oder benennt ein Produkt um. Oder aktualisiert die Preise. Oder stellt eine API ein.

Du aktualisierst die Dokumentation, bettest die Chunks aber nicht neu ein. Der Vector-Index repräsentiert noch den alten Inhalt.

Nutzer fragen nach den neuen Preisen. Die Embeddings leiten sie zum alten Inhalt. Das Modell liest den alten Inhalt und erklärt zuversichtlich die alten Preise. Der Support bekommt ein Ticket.

Jedes ernsthafte RAG-System trifft das irgendwann. Die Lösung klingt offensichtlich – neu einbetten, wenn sich Inhalte ändern – aber Teams bauen diese Pipeline selten vor dem ersten Vorfall.

Du brauchst inkrementelles Neu-Indizieren mit Content-Fingerprinting:

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Content unchanged, skip re-embedding
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

Neu indizieren beim Schreiben, Fingerprinting auf Inhalt, nicht auf Zeitstempeln. Dokumente werden in deinem CMS ständig aktualisiert, ohne dass sich der eigentliche Inhalt ändert.

## 3. Retrieval-Präzision vs. Recall: Du optimierst das Falsche

Die meisten RAG-Tutorials zeigen, wie man die Top-K-Chunks abruft. Sie erklären nicht den Kompromiss zwischen zwei Zielen, die in entgegengesetzte Richtungen zeigen.

**Hoher Recall**: Gib alles zurück, was relevant sein könnte. Nutzer bekommen immer eine Antwort. Aber das Kontextfenster des Modells ist voll von tangential verwandtem Rauschen, und es halluziniert, um die Lücken zwischen Fragmenten zu füllen.

**Hohe Präzision**: Gib nur die relevantesten Chunks zurück. Das Modell arbeitet mit sauberem, fokussiertem Kontext. Aber wenn der richtige Chunk nicht in den Top 3 ist, hat das Modell die Information nicht und erfindet trotzdem zuversichtlich etwas.

Die Fehlermodi sehen für Nutzer identisch aus: falsche Antworten. Aber die Ursachen und die Lösungen sind gegensätzlich.

Zwei Techniken, die tatsächlich helfen:

**Reranking**: Rufe mehr Kandidaten ab (Top-20), dann verwende ein Cross-Encoder-Modell, um sie nach Relevanz neu zu ranken, bevor sie an das LLM übergeben werden. Cross-Encoder sind langsamer als Vektorähnlichkeit, aber dramatisch genauer beim finalen Ranking-Schritt.

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // Now top-5 actually means something
```

**Hybride Suche**: Kombiniere Vektorsuche (semantische Ähnlichkeit) mit Keywordsuche (BM25). Sie scheitern auf unterschiedliche Weise. Die Vektorsuche hat Probleme mit spezifischen Begriffen, Modellnamen und IDs. Die Keywordsuche hat Probleme mit Paraphrasierung und Synonymen. Zusammen decken sie die blinden Flecken des jeweils anderen ab.

## 4. Dein Kontextfenster hat die falsche Form

Du hast die richtigen Chunks abgerufen. Glückwunsch. Das Modell wird es trotzdem falsch machen.

Das Problem ist nicht nur, was du abrufst. Es ist, wo du es platzierst.

LLMs können unter dem „Lost in the Middle"-Problem leiden. Liu et al. haben gemessen, dass Long-Context-Modelle relevante Informationen weniger zuverlässig nutzen, wenn sie in der Mitte des Prompts erscheinen statt nahe dem Anfang oder Ende.

Wenn du 20 Chunks in eine flache Liste stopfst und hoffst, dass das Modell sie korrekt synthetisiert, lässt du Performance auf dem Tisch liegen.

Dinge, die tatsächlich helfen:

**Evaluiere die Platzierung am Anfang/Ende für deine relevantesten Chunks.** Eine gängige Heuristik ist: Top-Ranking zuerst, Zweitplatzierte am Ende, der Rest in der Mitte. Kontraintuitiv, aber es lohnt sich, das gegen dein Modell und die Prompt-Form zu testen.

**Nummeriere und beschrifte deine Kontextabschnitte explizit.** `[Source 1]` … `[Source 2]` gibt dem Modell Anker, um damit zu argumentieren.

**Füge ein Retrieval-Confidence-Signal hinzu.** Wenn dein Ähnlichkeitswert 0,65 auf einer 0-1-Skala beträgt, sag dem Modell: „Der folgende Kontext wurde mit moderater Zuverlässigkeit abgerufen. Erwähne Unsicherheit, wenn die Antwort unklar ist."

**Setze ein Kontextbudget.** Übergib nicht einfach alles, was du abgerufen hast. Zähle Tokens, priorisiere nach Relevanzwert und schneide hart bei 60-70 % des Kontextfensters des Modells ab. Lass dem Modell Raum zum Nachdenken, ohne zu thrashen.

Referenz: [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172).

## 5. Du hast keine Ahnung, wann es falsch liegt

Das ist der stille Fehler: Die Antwort kommt zurück, die UI sieht gesund aus, und der Inhalt ist falsch.

Bei einer traditionellen API sind Fehler sichtbar: HTTP 500, Timeout, Schema-Validierungsfehler. Du weißt es sofort. RAG-Fehler sind leiser: Das System liefert eine Antwort, sie sieht plausibel aus, und sie ist falsch.

Du erfährst vielleicht erst, dass deine RAG-Pipeline scheitert, wenn Nutzer es dir sagen. Oft tun sie das nicht. Sie hören einfach auf, ihr zu vertrauen, und gehen zurück zu Strg+F.

Das minimal überlebensfähige Observability-Setup für ein Produktions-RAG-System:

**Logge deine Retrieval-Kette.** Jede Abfrage, was abgerufen wurde (Chunk-IDs + Scores), und was das Modell produziert hat. Du brauchst das, um irgendetwas zu debuggen.

**Tracke Retrieval-Metriken.** Mean Reciprocal Rank (MRR) und NDCG, wenn du Ground-Truth-Labels hast. Zumindest tracke die Verteilung der Ähnlichkeitswerte – wenn dein P50-Retrieval-Score fällt, ist deine Index-Qualität gefallen.

**Baue eine Feedback-Schleife.** Selbst ein Thumbs-Up/Down auf Antworten, zurückgeführt auf die Abfrage und die abgerufenen Chunks, gibt dir Trainingssignal. Ohne es fliegst du blind.

**Führe periodische Evals durch.** Ein Testset von 50-100 Fragen mit bekannten korrekten Antworten, wöchentlich ausgeführt, fängt Regressionen ab, bevor Nutzer es tun. Eine Tabelle und ein Skript reichen für den Anfang.

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // Alert if score drops below threshold
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## Das eigentliche Problem

Diese Fehler sind nicht hauptsächlich über das Embedding-Modell oder die Vector-Datenbank. Sie sind über das System um sie herum.

RAG-Demos funktionieren, weil die Demo-Bedingungen kontrolliert sind: saubere Dokumente, wohlgeformte Fragen, tolerante Evaluatoren. Die Produktion scheitert, weil keine dieser Bedingungen gilt.

Jeder der oben genannten Fehler ist diagnostizierbar, aber nur, wenn du misst. Zuverlässige RAG-Teams tun nichts Exotisches. Sie behandeln Retrieval-Qualität wie ein echtes Subsystem statt wie ein Demo-Artefakt.

Richte zuerst den Eval-Loop ein. Alles andere ist einfacher, sobald du es messen kannst.
