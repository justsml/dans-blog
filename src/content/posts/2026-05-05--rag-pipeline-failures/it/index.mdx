La demo del RAG sembra sempre fantastica.

Incorpori alcuni documenti, avvii un vettore store, lo colleghi alla chat e guardi il modello citare la tua base di conoscenza interna come se ci avesse lavorato per anni. Bellissimo. Gli stakeholder sono impressionati. Qualcuno dice "facciamo questo."

Sei settimane dopo, gli utenti ricevono risposte sbagliate con sicurezza. I ticket di supporto si accumulano. Il sistema *funziona*, solo non nel modo in cui le persone hanno bisogno che funzioni.

Il guasto di solito non è un errore drammatico. È cinque errori noiosi impilati insieme.

---

## 1. I Tuoi Chunk Sono delle Dimensioni Sbagliate

Questo guasto non fa crash. Rende solo ogni risposta leggermente peggiore finché tutta la funzionalità sembra inaffidabile.

La ricerca vettoriale recupera *chunk*, non documenti. Qualsiasi tu divida il tuo materiale di origine diventa l'unità di verità del recuperatore. Se gli chunk sono sbagliati, il modello risponde con i frammenti sbagliati.

**Troppo piccoli**: Lo chunk contiene una risposta parziale. L'embedding cattura l'argomento giusto, ma il testo recuperato manca di contesto. Recuperi "Il timeout massimo è 30 secondi" senza la frase prima che dice "quando si usa l'API legacy."

**Troppo grandi**: L'embedding diventa una sfocata media di molte idee. La ricerca semantica si confonde perché lo chunk riguarda molte cose, e il vettore risultante non rappresenta nessuna di esse pulitamente.

La dimensione corretta dello chunk dipende interamente dal tuo contenuto. Documenti tecnici, contratti legali e trascrizioni del supporto si dividono tutti in modo diverso. Non c'è una risposta universale.

Cosa fare: misura. Crea un set di valutazione di coppie domanda/risposta dal tuo corpus. Prova chunk da 256, 512 e 1024 token. Misura la precisione di recupero: appare lo chunk giusto nelle top 5? Imparerai velocemente che la dimensione dello chunk conta più del modello di embedding che hai agonizzato su.

Usa anche l'overlap. Uno chunk da 512 token con 64 token di overlap su ogni lato significa che le risposte che attraversano i confini vengono comunque recuperate. La maggior parte delle librerie vettoriali supporta questo. La maggior parte delle persone lo salta.

## 2. I Tuoi Embedding Diventano Obsoleti (E Non te ne Accorgerai)

Immagina che la tua azienda si rebranding. O rinomina un prodotto. O aggiorna i suoi prezzi. O deprecate un'API.

Aggiorni la documentazione ma non ri-embeddi gli chunk. L'indice vettoriale rappresenta ancora il vecchio contenuto.

Gli utenti chiedono sui nuovi prezzi. Gli embedding li indirizzano al vecchio contenuto. Il modello legge il vecchio contenuto e spiega con sicurezza i vecchi prezzi. Il supporto riceve un ticket.

Ogni serio sistema RAG incontra questo alla fine. La soluzione sembra ovvia - ri-embeddi quando il contenuto cambia - ma i team raramente costruiscono quella pipeline prima del primo incidente.

Hai bisogno di re-indicizzazione incrementale con fingerprinting del contenuto:

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Contenuto invariato, salta il ri-embedding
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

Re-indicizza su scrittura, fingerprint sul contenuto, non sui timestamp. I documenti vengono aggiornati nel tuo CMS tutto il tempo senza che il contenuto cambi effettivamente.

## 3. Precisione di Recupero vs. Recall: Stai Ottimizzando la Cosa Sbagliata

La maggior parte dei tutorial RAG mostra come recuperare gli chunk top-K. Non spiegano il compromesso tra due obiettivi che puntano in direzioni opposte.

**Alta recall**: Restituisci tutto ciò che potrebbe essere rilevante. Gli utenti ricevono sempre una risposta. Ma la finestra di contesto del modello è piena di rumore tangenzialmente correlato, e allucina per riempire i gap tra i frammenti.

**Alta precisione**: Restituisci solo gli chunk più rilevanti. Il modello lavora con contesto pulito e focalizzato. Ma se lo chunk giusto non è nelle top 3, il modello non ha le informazioni e con sicurezza inventa comunque qualcosa.

I modi di fallimento sembrano identici agli utenti: risposte sbagliate. Ma le cause, e le soluzioni, sono opposte.

Due tecniche che aiutano davvero:

**Reranking**: Recupera più candidati (top-20), poi usa un modello cross-encoder per riordinarli per rilevanza prima di passare all'LLM. I cross-encoder sono più lenti della similarità vettoriale ma drammaticamente più accurati nel passo di ordinamento finale.

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // Ora top-5 significa davvero qualcosa
```

**Ricerca ibrida**: Combina la ricerca vettoriale (similarità semantica) con la ricerca per parole chiave (BM25). Falliscono in modi diversi. La ricerca vettoriale lotta con termini specifici, nomi di modelli e ID. La ricerca per parole chiave lotta con parafrasi e sinonimi. Insieme, coprono i punti ciechi l'uno dell'altro.

## 4. La Tua Finestra di Contesto Ha la Forma Sbagliata

Hai recuperato gli chunk giusti. Congratulazioni. Il modello sta ancora sbagliando questo.

Il problema non è solo cosa recuperi. È dove lo metti.

Gli LLM possono soffrire del problema "perso nel mezzo". Liu et al. hanno misurato modelli a lungo contesto che usano informazioni rilevanti meno affidabilmente quando appaiono nel mezzo della richiesta invece che vicino all'inizio o alla fine.

Se stai spingendo 20 chunk in una lista piatta e sperando che il modello li sintetizzi correttamente, stai lasciando prestazioni sul tavolo.

Cose che aiutano davvero:

**Valuta il posizionamento start/end per i tuoi chunk più rilevanti.** Una euristiche comune è primo ranking all'inizio, secondo ranking alla fine, e il resto nel mezzo. Contro-intuitivo, ma vale la pena testare contro il tuo modello e la forma della richiesta.

**Numera ed etichetta esplicitamente le tue sezioni di contesto.** `[Fonte 1]` ... `[Fonte 2]` dà al modello ancore per ragionare.

**Aggiungi un segnale di confidenza di recupero.** Se la tua similarità score è 0.65 su una scala 0-1, dì al modello: "Il seguente contesto è stato recuperato con confidenza moderata. Riconosci l'incertezza se la risposta non è chiara."

**Imposta un budget di contesto.** Non passare tutto ciò che hai recuperato. Conta token, priorizza per score di rilevanza, e hard-cut al 60-70% della finestra di contesto del modello. Lascia spazio al modello per ragionare senza thrashare.

Riferimento: [Perso nel Mezzo: Come i Modelli Linguistici Usano i Lunghi Contesti](https://arxiv.org/abs/2307.03172).

## 5. Non Hai Idea Quando È Sbagliato

Questo è il guasto silenzioso: la risposta torna, l'UI sembra sana, e il contenuto è sbagliato.

Con un'API tradizionale, i guasti sono visibili: HTTP 500, timeout, errore di validazione schema. Lo sai immediatamente. I guasti RAG sono più silenziosi: il sistema restituisce una risposta, sembra plausibile, ed è sbagliato.

Potresti non sapere che la tua pipeline RAG sta fallendo finché gli utenti non te lo dicono. Spesso non lo fanno. Si limitano a smettere di fidarsene e tornano a Ctrl+F.

L'insieme minimo di osservabilità per un sistema RAG in produzione:

**Registra la tua catena di recupero.** Ogni query, cosa è stato recuperato (chunk IDs + scores), e cosa ha prodotto il modello. Hai bisogno di questo per debuggare qualsiasi cosa.

**Traccia le metriche di recupero.** Mean reciprocal rank (MRR) e NDCG se hai etichette ground-truth. Al minimo, traccia le distribuzioni di score di similarità - se il tuo P50 score di recupero scende, la qualità del tuo indice è scesa.

**Costruisci un loop di feedback.** Anche un pollice su/giù sulle risposte, collegato back alla query e agli chunk recuperati, ti dà un segnale di training. Senza quello, voli alla cieca.

**Esegui valutazioni periodiche.** Un set di test di 50-100 domande con risposte corrette note, eseguite settimanalmente, catturerà regressioni prima che lo facciano gli utenti. Un foglio di calcolo e uno script sono sufficienti per iniziare.

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // Alert se il score scende sotto la soglia
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score sceso a ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## Il Problema Reale

Questi guasti non sono principalmente sul modello di embedding o database vettoriale. Sono sul sistema intorno ad essi.

Le demo RAG funzionano perché le condizioni della demo sono controllate: documenti puliti, domande ben formate, valutatori tolleranti. La produzione fallisce perché nessuna di quelle condizioni vale.

Ogni guasto sopra è diagnosabile, ma solo se stai misurando. I team RAG affidabili non stanno facendo nulla di esotico. Stanno trattando la qualità di recupero come un vero sottosistema invece che un artefatto demo.

Imposta il loop di eval prima. Tutto il resto è più facile una volta che puoi misurarlo.
