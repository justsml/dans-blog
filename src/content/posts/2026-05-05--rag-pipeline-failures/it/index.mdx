La demo RAG è sempre fantastica.

Incorpori dei documenti, monti un vector store, lo colleghi a una chat e guardi il modello citare la tua knowledge base interna come se ci lavorasse da anni. Bellissimo. I dirigenti sono colpiti. Qualcuno dice: "mettiamolo in produzione."

Sei settimane dopo, gli utenti ricevono risposte sicure ma sbagliate. I ticket di supporto si accumulano. Il sistema *funziona*, ma non nel modo in cui la gente ne ha bisogno.

Il fallimento di solito non è un singolo errore drammatico. Sono cinque errori noiosi messi insieme.

---

## 1. I Tuoi Chunk Hanno la Dimensione Sbagliata

Questo fallimento non causa crash. Rende ogni risposta un po' peggiore fino a quando l'intera funzionalità sembra inaffidabile.

La ricerca vettoriale recupera *chunk*, non documenti. Qualunque sia la suddivisione del materiale originale, diventa l'unità di verità del retriever. Se i chunk sono sbagliati, il modello risponde con i frammenti sbagliati.

**Troppo piccoli**: Il chunk contiene una risposta parziale. L'embedding cattura l'argomento giusto, ma al testo recuperato manca il contesto. Recuperi "Il timeout massimo è di 30 secondi" senza la frase precedente che dice "quando si usa l'API legacy."

**Troppo grandi**: L'embedding diventa una media sfocata di molte idee. La ricerca semantica si confonde perché il chunk parla di più cose, e il vettore risultante non rappresenta nessuna di esse in modo nitido.

La dimensione giusta del chunk dipende interamente dal tuo contenuto. Documentazione tecnica, contratti legali e trascrizioni del supporto si suddividono tutti in modo diverso. Non esiste una risposta universale.

Cosa fare: misurare. Costruisci un set di valutazione con coppie domanda/risposta tratte dal tuo corpus. Prova chunk da 256, 512 e 1024 token. Misura la precisione del recupero: il chunk giusto appare nei primi 5 risultati? Scoprirai subito che la dimensione dei chunk è più importante del modello di embedding su cui ti sei angosciato.

Usa anche la sovrapposizione. Un chunk da 512 token con 64 token di sovrapposizione su ciascun lato fa sì che le risposte a cavallo tra due chunk vengano comunque recuperate. La maggior parte delle librerie vettoriali lo supporta. La maggior parte delle persone lo salta.

## 2. I Tuoi Embedding Diventano Obsoleti (E Non Te Ne Accorgerai)

Immagina che la tua azienda cambi il brand. O rinominizzi un prodotto. O aggiorni i prezzi. O deprechi un'API.

Aggiorni la documentazione ma non reincorpori i chunk. L'indice vettoriale rappresenta ancora il vecchio contenuto.

Gli utenti chiedono informazioni sui nuovi prezzi. Gli embedding li indirizzano al vecchio contenuto. Il modello legge il vecchio contenuto e spiega con sicurezza i vecchi prezzi. Il supporto riceve un ticket.

Ogni sistema RAG serio prima o poi incontra questo problema. La soluzione sembra ovvia — reincorporare quando il contenuto cambia — ma raramente i team costruiscono quella pipeline prima del primo incidente.

Serve una re-indicizzazione incrementale con fingerprinting del contenuto:

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Contenuto invariato, salta la re-incorporazione
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

Re-indicizza in scrittura, usa il fingerprint sul contenuto, non sui timestamp. I documenti vengono aggiornati nel CMS continuamente senza che il contenuto effettivo cambi.

## 3. Precisione vs. Recall nel Recupero: Stai Ottimizzando Quello Sbagliato

La maggior parte dei tutorial sulla RAG mostra come recuperare i primi K chunk. Non spiegano il compromesso tra due obiettivi che puntano in direzioni opposte.

**Recall elevato**: Restituisci tutto ciò che potrebbe essere rilevante. Gli utenti ricevono sempre una risposta. Ma la finestra di contesto del modello è piena di rumore solo tangenzialmente correlato, e il modello allucina per riempire i vuoti tra i frammenti.

**Precisione elevata**: Restituisci solo i chunk più rilevanti. Il modello lavora con un contesto pulito e focalizzato. Ma se il chunk giusto non è tra i primi 3, il modello non ha l'informazione e inventa con sicurezza qualcosa comunque.

Le modalità di fallimento appaiono identiche agli utenti: risposte sbagliate. Ma le cause, e le soluzioni, sono opposte.

Due tecniche che aiutano davvero:

**Reranking**: Recupera più candidati (top-20), poi usa un modello cross-encoder per riordinarli per rilevanza prima di passarli all'LLM. I cross-encoder sono più lenti della similarità vettoriale ma drasticamente più accurati nella fase finale di classificazione.

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // Ora i primi 5 significano davvero qualcosa
```

**Ricerca ibrida**: Combina la ricerca vettoriale (similarità semantica) con la ricerca per parole chiave (BM25). Falliscono in modi diversi. La ricerca vettoriale fa fatica con termini specifici, nomi di modelli e ID. La ricerca per parole chiave fa fatica con parafrasi e sinonimi. Insieme, coprono i punti ciechi reciproci.

## 4. La Tua Finestra di Contesto Ha la Forma Sbagliata

Hai recuperato i chunk giusti. Congratulazioni. Il modello sbaglierà comunque.

Il problema non è solo ciò che recuperi. È dove lo metti.

Gli LLM possono soffrire del problema "perso nel mezzo" (lost in the middle). Liu et al. hanno misurato come i modelli a contesto lungo usino le informazioni rilevanti in modo meno affidabile quando appaiono nel mezzo del prompt invece che vicino all'inizio o alla fine.

Se stai infilando 20 chunk in una lista piatta e speri che il modello li sintetizzi correttamente, stai lasciando prestazioni sul tavolo.

Cose che aiutano davvero:

**Valuta il posizionamento all'inizio/fine per i chunk più rilevanti.** Un'euristica comune è il primo classificato all'inizio, il secondo alla fine, e il resto in mezzo. Controintuitivo, ma vale la pena testarlo con il tuo modello e la forma del prompt.

**Numera ed etichetta esplicitamente le sezioni di contesto.** `[Fonte 1]` ... `[Fonte 2]` dà al modello dei punti di ancoraggio per ragionare.

**Aggiungi un segnale di confidenza del recupero.** Se il tuo punteggio di similarità è 0,65 su una scala 0-1, dì al modello: "Il seguente contesto è stato recuperato con confidenza moderata. Riconosci l'incertezza se la risposta non è chiara."

**Imposta un budget di contesto.** Non passare tutto ciò che hai recuperato. Conta i token, dai priorità in base al punteggio di rilevanza, e taglia duro al 60-70% della finestra di contesto del modello. Lascia spazio al modello per ragionare senza fare thrashing.

Riferimento: [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172).

## 5. Non Hai Idea di Quando Sbaglia

Questo è il fallimento silenzioso: la risposta arriva, l'interfaccia è sana, e il contenuto è sbagliato.

Con un'API tradizionale, i fallimenti sono visibili: HTTP 500, timeout, errore di validazione dello schema. Lo sai immediatamente. I fallimenti della RAG sono più silenziosi: il sistema restituisce una risposta, sembra plausibile, ed è sbagliata.

Potresti non sapere che la tua pipeline RAG sta fallendo finché gli utenti non te lo dicono. Spesso non lo fanno. Semplicemente smettono di fidarsi e tornano a Ctrl+F.

La configurazione di osservabilità minima per un sistema RAG in produzione:

**Registra la tua catena di recupero.** Ogni query, cosa è stato recuperato (ID chunk + punteggi), e cosa ha prodotto il modello. Ti serve per fare debug di qualsiasi cosa.

**Traccia le metriche di recupero.** Mean Reciprocal Rank (MRR) e NDCG se hai etichette di verità. Come minimo, traccia le distribuzioni dei punteggi di similarità — se il tuo P50 del punteggio di recupero cala, la qualità del tuo indice è calata.

**Costruisci un ciclo di feedback.** Anche un pollice su/giù sulle risposte, collegato alla query e ai chunk recuperati, ti dà segnale di training. Senza, voli alla cieca.

**Esegui valutazioni periodiche.** Un test set di 50-100 domande con risposte corrette note, eseguito settimanalmente, individuerà le regressioni prima degli utenti. Un foglio di calcolo e uno script sono sufficienti per iniziare.

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Punteggio valutazione: ${(avgScore * 100).toFixed(1)}%`);
  
  // Avvisa se il punteggio scende sotto la soglia
  if (avgScore < 0.75) {
    await notifyTeam(`Punteggio valutazione RAG sceso al ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## Il Problema Reale

Questi fallimenti non riguardano principalmente il modello di embedding o il database vettoriale. Riguardano il sistema che li circonda.

Le demo RAG funzionano perché le condizioni della demo sono controllate: documenti puliti, domande ben formulate, valutatori tolleranti. La produzione fallisce perché nessuna di queste condizioni vale.

Ogni fallimento sopra è diagnosticabile, ma solo se stai misurando. I team RAG affidabili non fanno nulla di esotico. Trattano la qualità del recupero come un vero sottosistema invece che come un artefatto da demo.

Imposta prima il ciclo di valutazione. Tutto il resto è più facile una volta che puoi misurarlo.
