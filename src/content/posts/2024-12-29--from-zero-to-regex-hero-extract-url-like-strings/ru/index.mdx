## 🚀 Введение

Извлечение URL из сырого текста иногда похоже на игру в утомительный "крестики-нолики". Знаки препинания, скобочные обёртки и неоднозначное форматирование создают проблемы. Независимо от того, разрабатываете ли вы веб-скрейпер, аналитический инструмент или чат-приложение, точное извлечение URL является критически важным.

В этой статье мы решим проблему напрямую, используя гибкий двухэтапный подход. Наша цель — **сначала захватить все _потенциальные_ URL-подобные строки**, а затем обрабатывать валидацию в отдельном процессе.

> 💡 **Примечание:** Этот шаблон не предназначен для **_валидации_** URL! Он специально допускает гибкость в использовании знаков препинания и ошибок написания.

## 🔍 Цель: Извлечь URL из текста

При извлечении URL из сырого текста двухэтапный подход оказывается эффективным:

1. **Захватить всё URL-подобное**: Использовать широкую сеть для захвата всех строк, которые *могут* быть URL. Именно здесь наш "регулярное выражение из 120+ байт" демонстрирует свои сильные стороны.
2. **Провести валидацию**: После захвата этих кандидатов используйте вторичные проверки (например, разрешение DNS, сравнение с известными доменами), чтобы исключить недействительные записи.

### Визуализация задачи

Термины вроде `извлечение` и `разбор` часто используются как синонимы, но они описывают разные процессы. Извлечение URL включает в себя идентификацию и захват потенциальных URL из более крупного текста. Разбор, с другой стороны, подразумевает разложение этих URL на их составные части.

## Регулярное выражение из 120+ байт

Ниже приведено компактное регулярное выражение, предназначенное для однократного извлечения и разбора URL. Оно поддерживает различные протоколы, домены, пути и необязательные секции запроса/фрагмента.

Не волнуйтесь — мы разберём его пошагово!

```js title="120+ Byte URL Regex" frame="code"
const urlRegex = /([-.a-z0-9]+:\/{1,3})([^-\/\.[\](|)\s?][^`\/\s\]?]+)([-_a-z0-9!@$%^&*()=+;/~\.]*)[?]?([^#\s`?]*)[#]?([^#\s'"`\.,!]*)/gi;
// Compatibility: ES5+

// Same pattern, split on newlines for readability:
([-.a-z0-9]+:\/{1,3})
([^-\/\.[\](|)\s?][^`\/\s\]?]+)
([-_a-z0-9!@$%^&*()=+;/~\.]*)
[?]?([^#\s`?]*)
[#]?([^#\s'"`\.,!]*) 
```

## 🧩 Постепенный разбор по частям

Разберём регулярное выражение по частям, чтобы понять, как оно работает:

<h3>1. Протокол (Группа 1): <code>{`([-.a-z0-9]+:\/{1,3})`}</code></h3>

<ul>
  <li>**Назначение:** Соответствует протоколу URL (например, `http://`, `ftp://`, `custom-scheme://`).</li>
  <li>
    **Пояснение:**
    <ul>
      <li><code>[-.a-z0-9]+</code>: Соответствует одному или нескольким строчным буквам, цифрам, дефисам или точкам (часто встречаются в схемах протоколов).</li>
      <li><code>{`:\/{1,3}`}</code>: Соответствует двоеточию, за которым следуют один-три слэша (<code>:/</code>, <code>://</code> или <code>:///</code>).</li>
    </ul>
  </li>
</ul>

<h3>2. Домен (Группа 2): <code>{`([^-\/\.[\](|)\s?][^\`\/\s\]?]+)`}</code></h3>

<ul>
  <li>**Назначение:** Извлекает домен или хост URL.</li>
  <li>
    **Пояснение:**
    <ul>
      <li><code>[^-\/\.[\](|)\s?]</code>: Соответствует любому символу, кроме указанных специальных символов и пробелов.</li>
      <li><code>[^`\/\s\]?]+</code>: Соответствует одному или нескольким символам, кроме обратных кавычек, слэшей, пробелов или закрывающих квадратных скобок.</li>
    </ul>
  </li>
</ul>

<h3>3. Путь (Группа 3): <code>{`([-_a-z0-9!@$%^&*()=+;/~\\.]*)`}</code></h3>

<ul>
  <li>**Цель:** Соответствует компоненту пути URL.</li>
  <li>
    **Объяснение:**
    <ul>
      <li><code>[-_a-z0-9!@$%^&*()=+;/~\.]*</code>: Соответствует нулю или более URL-совместимым символам, часто встречающимся в путях.</li>
    </ul>
  </li>
</ul>

<h3>4. Запрос (Группа 4): <code>[?]?([^#\s`?]*)</code></h3>

<ul>
  <li>**Цель:** Опционально соответствует строке запроса, начинающейся с любого символа <code>?</code>.</li>
  <li>
    **Объяснение:**
    <ul>
      <li><code>[?]?</code>: Опционально соответствует символу <code>?</code>. (Квадратные скобки не обязательны, но обеспечивают большую ясность по сравнению с сокращенной записью <code>??</code>. Также создают визуальное сходство с аналогичной следующей группой <code>[#]?</code>.)</li>
      <li><code>([^#\s`?]*)</code>: Соответствует нулю или более символам, не являющимся решеткой, пробелом, обратной гравис или вопросительным знаком.</li>
    </ul>
  </li>
</ul>

<h3>5. Фрагмент (Группа 5): <code>[#]?([^#\s'"`\.,!]*)</code></h3>

<ul>
  <li>**Цель:** Опционально соответствует идентификатору фрагмента, начинающемуся с <code>#</code>.</li>
  <li>
    **Объяснение:**
    <ul>
      <li><code>[#]?</code>: Опционально соответствует символу <code>#</code>.</li>
      <li><code>([^#\s'"`\.,!]*)</code>: Соответствует нулю или более символам, не являющимся запрещенной пунктуацией или пробелом.</li>
    </ul>
  </li>
</ul>

## 🛠️ Пример парсинга

Вот как можно использовать этот гигантский регулярное выражение с помощью немного JavaScript:

 tabs={[

```js title="extract-urls.js" frame="code"
const text = `
Посмотрите сюда: https://example.com/path?query=123#section
И также (ftp://files.server.org/index).
Еще странный: custom-scheme://host/param;weird^stuff
`;

const urlRegex =
  /([-.a-z0-9]+:\/{1,3})([^-\/\.[\](|)\s?][^`\/\s\]?]+)([-_a-z0-9!@$%^&*()=+;/~\.]*)[?]?([^#\s`?]*)[#]?([^#\s'"`\.,!]*)/gi;

const matches = [
  ...text.matchAll(urlRegex),
].map((match) => match[0]);
console.log("Извлеченные URL:", matches);

const parts = [
  ...text.matchAll(urlRegex),
].map((match) => match.slice(1));
console.log("Извлеченные Части:", parts);
```

## ☑️ Следующие шаги

В зависимости от вашей задачи вам может понадобиться уточнить это регулярное выражение или добавить дополнительную проверку и постобработку.

### Разные проекты, разные потребности

Проекты имеют разные требования и проблемы безопасности:

1. **Веб-скрейпинг**: Проверяйте URL-адреса, чтобы убедиться, что они доступны и надежны.  
2. **Обработка данных**: Извлекайте URL-адреса из пользовательского контента с учетом безопасности.  
3. **Анализ данных**: Фильтруйте дубликаты или нерелевантные ссылки для исследований или маркетинга.  
4. **Пользовательские приложения**: Автоматически создавайте гиперссылки в чат-приложениях или форумах.  

### Послепроцессинг и проверка  

После сбора потенциальных URL-адресов примените дополнительные проверки:  

- **Проверка DNS**: Убедитесь, что домены разрешаются.  
- **Проверки безопасности**: Используйте сервисы для обнаружения вредоносных или мошеннических сайтов.  
- **Пользовательские правила**: Применяйте фильтры, специфичные для проекта (например, разрешенные TLD, максимальная длина URL).  

## 📝 Вывод  

Извлечение полуструктурированных строковых данных может стать наиболее удовлетворительной частью освоения регулярных выражений.

Вот краткое изложение ключевых выводов:

- **Используйте визуальный инструмент для написания, тестирования и понимания ваших [регулярных выражений.](https://regex101.com/r/jO8bC4/69)**
- **Разбейте задачу на части и решайте каждую часть отдельно.** В некотором смысле, группы захвата предоставляют нам метафорические «маркировки» для наших регулярных выражений.
- **Используйте «свободные» выражения для сопоставления, избегайте строгого соблюдения спецификаций при инжекции данных.**
- **Применение шагов проверки после начального извлечения обязательно** — всегда учитывайте безопасность и специфические потребности вашего проекта.

Следуя этим шагам, вы сможете эффективно извлекать любые полуструктурированные строковые данные, создавая основу для дальнейшей обработки и проверки.
