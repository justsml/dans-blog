    Quale di queste conversioni di tipo è **non valida** ❌?
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    PostgreSQL supporta tre sintassi di cast:

    1. ANSI SQL: `CAST(expression AS type)`.
    2. PostgreSQL: `expression::type`.
    3. Funzione di tipo: `type 'literal'`.

    Tutte sono funzionalmente equivalenti, ma:
    - `CAST()` è la più portabile.
    - `::` è specifico di PostgreSQL ma comunemente usato.
    - Lo stile infisso `type 'literal'` è meno comune ma comunque valido.

    ```sql
        );
    ```
  </div>
  </slot>
  <slot name='explanation'>

    - Consentono più valori NULL.
    - `NULL` ≠ `NULL` nei controlli di unicità.

    ```sql
        CREATE TABLE student_emails (
          student_id INTEGER,

          UNIQUE(email)
        );
    ```

  group="Timestamp"
  title="timestamptz vs timestamp"
  options={[
    {text: 'Entrambi occupano 8 byte, ma rappresentano semantiche di timestamp diverse', isAnswer: true},
    {text: 'They"'},
    {text: 'timestamptz conserva qualsiasi fuso orario di input'},
    {text: 'timestamptz memorizza il nome o l\'offset del fuso orario originale'},
    {text: 'timestamptz memorizza un valore di 2 byte per il fuso orario'},
    {text: 'timestamptz è il successore di timestamp'},
  ]}
>
  <slot name="question">
  <div className="question">
    Qual è l'affermazione **più accurata** su `timestamptz` e `timestamp`?
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Entrambi sono 8 byte, ma non memorizzano lo stesso tipo di valore.

    Qual è la differenza? Sta nell'analisi dell'input.

    **`timestamptz`**
    - Normalizza l'input a un punto assoluto nel tempo.
    - Tiene conto dell'impostazione `TimeZone` del server/connessione quando analizza input senza offset esplicito e quando visualizza l'output.

    **`timestamp`**
    - Memorizza data e ora senza conversione di fuso orario.
    - Non conserva né normalizza le informazioni sul fuso orario.


    **`timestamp`**

    - Memorizza data e ora senza informazioni sul fuso orario.
    - Utile per archiviare esplicitamente date standardizzate, sia in UTC che in un fuso orario specifico.
  </div>
  </slot>
</Challenge>

  <slot name="question">
  <div className="question">
    Cosa succede quando si calcola il totale possibile di ID studente?
    ```sql
        SELECT 256 * 256 * 256 * 256;
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Il tipo `integer` di PostgreSQL è a 32 bit con segno, con intervallo da `-2,147,483,648` a `2,147,483,647`.

    Il calcolo `256^4` = `4,294,967,296` supera questo intervallo.

    Per gestire numeri più grandi:
    ```sql
        -- Use BIGINT
        SELECT 256::bigint * 256 * 256 * 256;

        -- Or numeric for arbitrary precision
        SELECT 256::numeric * 256 * 256 * 256;
    ```
  </div>
  </slot>

    ```sql
        );
    ```

    - I fusi orari sono accettati per `timestamptz`, ma non obbligatori

    **Curiosità poco comune:** Alcuni linguaggi/framework inviano precisione a nanosecondi, ma PostgreSQL memorizza i timestamp con precisione microsecondi.

  title="Tempistica del vincolo CHECK"
  options={[
    {text: 'Immediatamente per le righe nuove o modificate', isAnswer: true},
    {text: 'Al commit della transazione'},
    {text: 'Alla prossima query'},
    {text: 'Mai - i vincoli vengono controllati solo su INSERT'},
  ]}
>
  <slot name="question">
  <div className="question">
    Quando viene verificato questo vincolo?
    ```sql
        ALTER TABLE students 
        ADD CONSTRAINT valid_grade 
        CHECK (
          (grade >= 0 AND grade <= 100) OR 
          grade IS NULL
        ) NOT VALID;
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `NOT VALID` constraints:
    - Vengono controllati immediatamente per nuovi inserimenti e aggiornamenti
    - Non convalidano le righe esistenti
    - È possibile convalidare le righe esistenti in seguito con `VALIDATE CONSTRAINT`
    - Sono utili per tabelle grandi

    Without `NOT VALID`:
    - Il vincolo viene controllato immediatamente
    - Tutte le righe esistenti sono convalidate
    - Può essere lento su tabelle grandi
  </div>
  </slot>
