  </div>

  <slot name='explanation'>
  <div className="explanation">
    PostgreSQL prend en charge trois syntaxes de transtypage :

    1. SQL ANSI : `CAST(expression AS type)`.
    2. PostgreSQL : `expression::type`.
    3. Fonction de type : `type 'literal'`.

    Toutes sont fonctionnellement équivalentes, mais :
    - `CAST()` est la plus portable.
    - `::` est spécifique à PostgreSQL mais couramment utilisée.
    - La syntaxe infixe `type 'literal'` est moins courante mais toujours valide.

  options={[
    {text: 'Ils occupent tous deux 8 octets, mais représentent des sémantiques de timestamp différentes.', isAnswer: true},
    {text: 'Ils'},
    {text: 'timestamptz conserve tout fuseau horaire saisi'},
    {text: 'timestamptz stocke le nom ou le décalage du fuseau horaire d\'origine'},
    {text: 'timestamptz stocke une valeur de 2 octets pour le fuseau horaire'},
    {text: 'timestamptz est le successeur de timestamp'},
  ]}
>
  <slot name="question">
  <div className="question">
    Quelle est l'affirmation **la plus précise** concernant `timestamptz` et `timestamp` ?
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Ils occupent tous deux 8 octets, mais ils ne stockent pas le même type de valeur.

    Alors quelle est la différence ? Elle réside dans l'analyse de l'entrée.

    **`timestamptz`**
    - Normalise l'entrée en un point absolu dans le temps.
    - Prend en compte le paramètre `TimeZone` du serveur/de la connexion lors de l'analyse d'une entrée sans décalage explicite et lors de l'affichage.

    **`timestamp`**
    - Stocke une date et une heure sans conversion de fuseau horaire.
    - Ne conserve ni ne normalise les informations de fuseau horaire.


    **`timestamp`**

    - Stocke la date et l'heure sans informations de fuseau horaire.
    - Utile pour stocker explicitement des dates standardisées, soit en UTC, soit dans un fuseau horaire spécifique.
  </div>
  </slot>
</Challenge>

  </div>
  </slot>
  <slot name='explanation'>

    Le type `integer` de PostgreSQL est un entier signé 32 bits, allant de `-2,147,483,648` à `2,147,483,647`.

    Pour gérer des nombres plus grands :
    ```sql
    -- Use BIGINT
    SELECT 256::bigint * 256 * 256 * 256;

    -- Or numeric for arbitrary precision
    SELECT 256::numeric * 256 * 256 * 256;
    ```
  </div>

    Quand cette contrainte de note est-elle vérifiée ?
    ```sql
    ALTER TABLE students 
    ADD CONSTRAINT valid_grade 
    CHECK (
      (grade >= 0 AND grade <= 100) OR 
      grade IS NULL
    ) NOT VALID;
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Contraintes `NOT VALID` :
    - Sont vérifiées immédiatement pour les nouvelles insertions et mises à jour
    - Ne valident pas les lignes existantes
    - Peuvent valider les lignes existantes plus tard avec `VALIDATE CONSTRAINT`
    - Sont utiles pour les grandes tables

    Sans `NOT VALID` :
    - La contrainte est vérifiée immédiatement
    - Toutes les lignes existantes sont validées
    - Peut être lent sur les grandes tables
