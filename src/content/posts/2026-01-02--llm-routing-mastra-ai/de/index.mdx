Die meisten Ingenieurteams wählen ein Sprachmodell und bleiben dabei. Ein Anbieter, ein Modell, alle Aufgaben. Es ist so, als würdest du eine einzige Person einstellen, die für dein Programmieren, dein Copywriting und deine Steuererklärungen zuständig ist, nur weil sie beim ersten Interview gut abgeschnitten hat.

Zu jedem beliebigen Zeitpunkt ist ein Modell besser beim Programmieren, ein anderes eignet sich besser für lange, chaotische Kontexte, und ein drittes ist das günstigste, langweilige Arbeitspferd für die Klassifizierung. Die Namen ändern sich. Die Art des Problems bleibt gleich. Wenn du ein Modell so behandelst, als wäre es für alles hervorragend, zahlst du entweder für einfache Aufgaben zu viel oder bekommst bei spezialisierten Aufgaben schlechte Ergebnisse.

Ich sah ein Team, das durchläuft Tausende von Dollar, indem sie eine Sentiment-Analyse mit einem 30-Dollar-pro-Million-Token-Modell laufen lässt, obwohl ein 0,50-Dollar-Modell genausogut auskommen würde. Einfache JSON-Formatierung, grundlegende Klassifizierungsaufgaben, alles läuft über ihren Premium-Anbieter. Die einzige Stelle, die auf Hitzefliegen lief, war ihre AWS-Rechnung.

Es gibt einen besseren Weg, und er ist nicht besonders kompliziert.

## Delegation Über Hingabe

Wie wäre es, wenn du Anfragen an das Modell umleiten könntest, das tatsächlich am besten für diese spezifische Aufgabe geeignet ist? Nutze dein teures Power-Modell für die schwierigen Aufgaben, aber schick die einfachen Parsing- und Formatierungsaufgaben an etwas Günstigeres. Du erhältst die Vorteile mehrerer Anbieter, ohne sie in deinem Codebase manuell hin und her schieben zu müssen.

Mastra ermöglicht dir, genau dieses System zu bauen. Du richtest Spezialisten für verschiedene Arten von Arbeit ein und erstellst dann einen Router-Agenten, der herausfindet, welchen Spezialisten jede Anfrage bearbeiten soll. Die unten stehenden Modell-IDs sind Beispiele, kein Scoreboard. Tausche sie gegen die aktuellen Modelle, die deine Tests gewinnen und zu deinem Budget passen.

Stell dir vor, du hast drei Spezialisten in deinem Team.

```typescript
// ./src/mastra/index.ts
import { Mastra } from '@mastra/core';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { anthropic } from '@ai-sdk/anthropic';
import { google } from '@ai-sdk/google';

export const claudeAgent = new Agent({
  id: 'claude-agent',
  instructions: 'Du bist ein Experte im Programmieren. Bugs schreiben? Du wirst gefeuert.',
  model: anthropic(process.env.CODE_MODEL ?? 'claude-sonnet-4-5'),
});

export const geminiAgent = new Agent({
  id: 'gemini-agent',
  instructions: 'Du bist ein kreativer Schriftsteller. Sei merkwürdig.',
  model: google(process.env.LONG_CONTEXT_MODEL ?? 'gemini-3-pro-preview'),
});

export const gptAgent = new Agent({
  id: 'gpt-agent',
  instructions: 'Du bist ein hilfreicher Assistent. Sei langweilig.',
  model: openai(process.env.GENERAL_MODEL ?? 'gpt-5.2'),
});
```

Jeder hat eine Aufgabe. Dein Code-Agent sollte das Modell sein, das deine repository-spezifischen Code-Tests besteht. Dein Lang-Kontext-Agent sollte das sein, das deine tatsächlichen Dokumente übersteht, ohne die Mitte zu einer Suppe zu verarbeiten. Dein allgemeiner Agent sollte günstig, zuverlässig und auf die beste Art langweilig sein.

Hier wird es interessanter. Du fügst einen Router hinzu, der als intelligenter Proxy fungiert:

```typescript
export const routerAgent = new Agent({
  id: 'router-agent',
  name: 'Der Chef',
  instructions: `Du bist ein intelligenter Router.
  - Programmieren -> Claude
  - Poesie -> Gemini
  - Fakten -> GPT

  Mache die Arbeit nicht selbst. Du delegierst.`,
  model: openai(process.env.ROUTER_MODEL ?? 'gpt-5-mini'), // Nutze ein günstiges Modell für das Routing!
  agents: {
    claudeAgent,
    geminiAgent,
    gptAgent,
  },
});

export const mastra = new Mastra({
  agents: { routerAgent, claudeAgent, geminiAgent, gptAgent },
});
```

Der Router selbst läuft auf einem leichten Modell, da er nur Entscheidungen darüber trifft, an welche Route er den Verkehr senden soll. Du zahlst keine Premium-Raten, um herauszufinden, welches andere Premium-Modell verwendet werden soll. Messen Sie auch das; ein schlechter Router verwandelt Ersparnisse stillschweigend in umgeleitete Anfragen.

Wenn jemand eine Bubble-Sort-Implementierung anfordert, erkennt der Router es als Codetätigkeit und übergibt sie deinem Code-Spezialisten. Kreatives Schreib-Prompt? Das geht an das Modell, das du für Stimme und Ausdruck gewählt hast. Faktische Frage zu historischen Ereignissen? Leite sie an den allgemeinen Agenten weiter, idealerweise mit Abruf, wenn Aktualität oder Zitierung wichtig ist.

## Die Praktischen Vorteile

**Kosteneffizienz ist wichtiger, als du denkst.** Ein kleines Routing-Modell, das Delegierungsentscheidungen trifft, kostet nur einen Bruchteil dessen, jeden einzelnen Anfrage durch deinen teuersten Anbieter laufen zu lassen. Über die Zeit, insbesondere im großen Maßstab, summieren sich dies zu echtem Geld. Du zahlst nur für schwere Intelligenz, wenn du sie wirklich brauchst.

**Die Qualität verbessert sich, wenn du Modelle an Aufgaben zuordnest.** Der Gewinner wechselt monatsweise, Aufgabe und Prompt-Form. Deshalb sollte die Routing-Schicht von deinen Tests abhängen, nicht von welchem Modell gerade im Twitter-Feed die meisten Follower hatte, als du die Integration geschrieben hast.

**Resilienz wird ein Nebeneffekt.** Wenn OpenAI eine ihrer periodischen Ausfälle hat (und die hat sie), kann dein Router den Verkehr an andere Anbieter umleiten. Du bist nicht ohne Wasser am Land, wartest darauf, dass eine bestimmte API wieder online geht.

Es geht nicht darum, clever zu sein, nur um clever zu sein. Es geht darum, Systeme zu bauen, die sowohl finanziell als auch technisch Sinn ergeben. Du würdest nicht denselben Hammer für jede Baumaßnahme verwenden, und du solltest wahrscheinlich auch nicht dasselbe Sprachmodell für jede KI-Aufgabe verwenden.

Die Schönheit dieses Ansatzes ist, dass sich dein Anwendungscode nicht ändert. Du rufst immer noch einfach deinen Router-Agent auf. Die Komplexität der Entscheidung, welches Modell für welche Aufgabe verwendet werden soll, lebt an einem Ort, einmal konfiguriert, statt verteilt über deinen Codebase in einer Reihe von bedingter Logik.
