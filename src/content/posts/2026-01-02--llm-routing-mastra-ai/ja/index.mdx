多くのエンジニアリングチームは、ひとつの言語モデルを選んでそれに固執する。ひとつのプロバイダー、ひとつのモデル、すべてのタスク。面接で一番最初に良さそうだった人に、コーディング、コピーライティング、税務処理まで全部任せるようなものだ。

どの時点でも、あるモデルはコードに強く、別のモデルは長く混沌としたコンテキストに強く、さらに別のモデルは分類タスク向けの安くて堅実な働き手だ。名前は変わる。問題の形は変わらない。ひとつのモデルがすべてに秀逸だと思い込むなら、単純なタスクで過剰に払っているか、専門性の高いタスクで劣った結果を得ているかのどちらかだ。

あるチームが感情分析を1トークン300ドルのモデルに流して何千ドルも燃やしているのを見た。1トークン0.50ドルのモデルで同じ仕事が同じ精度でできるのに。単純なJSONフォーマット、基本的な分類タスク、すべてが高級プロバイダー経由。熱くなっていたのはAWSの請求書だけだった。

もっといいやり方がある。しかも、特に複雑ではない。

## 没頭より委任

リクエストをそのタスクに本当に適したモデルにルーティングできたらどうだろう？ 難しい仕事には高価な主力モデルを使い、単純なパースやフォーマットは安いものに落とす。複数のプロバイダーのメリットを、コードベースで手作業で振り回すことなく得られる。

Mastraを使えば、こういうシステムをそのまま構築できる。異なる種類の仕事向けに専門エージェントを設定し、それから各リクエストをどの専門家が処理すべきか判断するルーターエージェントを作る。下のモデルIDは例であり、ランキングではない。自分の評価と予算に合う現在のモデルに差し替えてほしい。

こう考えてほしい：チームに3人の専門家がいる。

```typescript
// ./src/mastra/index.ts
import { Mastra } from '@mastra/core';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { anthropic } from '@ai-sdk/anthropic';
import { google } from '@ai-sdk/google';

export const claudeAgent = new Agent({
  id: 'claude-agent',
  instructions: 'You are an expert engineer. Write bugs? You are fired.',
  model: anthropic(process.env.CODE_MODEL ?? 'claude-sonnet-4-5'),
});

export const geminiAgent = new Agent({
  id: 'gemini-agent',
  instructions: 'You are a creative writer. Be weird.',
  model: google(process.env.LONG_CONTEXT_MODEL ?? 'gemini-3-pro-preview'),
});

export const gptAgent = new Agent({
  id: 'gpt-agent',
  instructions: 'You are a helpful assistant. Be boring.',
  model: openai(process.env.GENERAL_MODEL ?? 'gpt-5.2'),
});
```

それぞれが仕事を持つ。コードエージェントは、あなたのリポジトリ固有のコード評価をパスするモデルにする。長コンテキストエージェントは、実際のドキュメントで中間部分がスープにならずに耐えられるモデルにする。汎用エージェントは、安く、信頼でき、最高の意味で無味乾燥であるべきだ。

ここが面白くなる。知能プロキシとして働くルーターを追加する：

```typescript
export const routerAgent = new Agent({
  id: 'router-agent',
  name: 'The Boss',
  instructions: `You are an intelligent router.
  - Coding -> Claude
  - Poetry -> Gemini
  - Facts -> GPT

  Do not do the work yourself. Delegate.`,
  model: openai(process.env.ROUTER_MODEL ?? 'gpt-5-mini'), // Use a cheap model for routing!
  agents: {
    claudeAgent,
    geminiAgent,
    gptAgent,
  },
});

export const mastra = new Mastra({
  agents: { routerAgent, claudeAgent, geminiAgent, gptAgent },
});
```

ルーター自身は軽量モデルで動く。送り先を決めるだけだから。どの高級モデルを使うかを決めるために高級レートを払う必要はない。これも測ること。悪いルーターは節約を静かに誤ルートに変える。

バブルソートの実装を頼むと、ルーターはそれをコード仕事と認識してコード専門家に渡す。創作プロンプト？ 声と幅のために選んだモデルに行く。歴史的出来事に関する事実質問？ 汎用エージェントにルーティング。鮮度や引用が重要なら、理想は取得（ retrieval ）付きで。

## 実践的なメリット

**コスト効率は思ったより重要だ。** 委任決定を行う小さなルーティングモデルは、すべてのリクエントを最も高価なプロバイダーに流すコストの端切れにしかならない。特に大規模では、時間とともにこれが本当の金額になる。重い知力には実際に必要な時だけ払う。

**モデルをタスクに合わせると品質が上がる。** 優者は月、タスク、プロンプトの形で変わる。だからルーティング層はあなたの評価に依存すべきで、統合を書いた週にTwitterで勝っていたモデルにではない。

**レジリエンスが副産物になる。** OpenAIが周期的な障害を起こす時（そしてそうなる）、ルーターはトラフィックを他のプロバイダーに再ルーティングできる。特定のAPIがオンラインに戻るのを待って手が止まることはない。

これは賢さのための賢さではない。財政的にも技術的にも筋が通るシステムを構築することだ。すべての建築タスクに同じハンマーを使わないし、すべてのAIタスクに同じ言語モデルを使うべきでもない。

このやり方の美点は、アプリケーションコードが変わらないこと。依然としてルーターエージェントを呼ぶだけ。どのモデルをどのタスクに使うかの複雑性は一箇所に住み、一度設定され、コードベースに散らばる条件分岐の山にはならない。
