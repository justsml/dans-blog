ほとんどのエンジニアリングチームは、一つの言語モデルを選んだら、ずっとそれを使い続けます。一つのプロバイダー、一つのモデル、すべてのタスク。それはまるで、最初の面接が良かったからという理由だけで、プログラミングもコピーライティングも確定申告もすべて一人の人間に任せるようなものです。

どんな時でも、あるモデルはコードに長け、別のモデルは複雑で長いコンテキストの扱いに優れ、また別のモデルは分類タスクにおいて最も安価で堅実な働きをします。モデルの名前は変わりますが、問題の本質は変わりません。一つのモデルがすべてにおいて優れているかのように扱うことは、単純なタスクに過剰な料金を支払っているか、専門的なタスクで期待以下の結果しか得られていないかのどちらかを意味します。

私は、0.50ドルのモデルで十分なセンチメント分析を、100万トークンあたり30ドルのモデルで実行し、何千ドルも浪費しているチームを見たことがあります。単純なJSONフォーマットや基本的な分類タスクまで、すべてをプレミアムプロバイダーに流していました。熱くなっていたのは、彼らのAWSの請求書だけでした。

もっと良い方法があります。しかも、それほど複雑ではありません。

## 執着よりも委譲

特定のタスクに最適なモデルへリクエストをルーティングできたらどうでしょうか？難しい処理には高価で強力なモデルを使い、単純なパースやフォーマットはより安価なモデルに任せるのです。コードベースで手動でやりくりすることなく、複数のプロバイダーの恩恵を享受できます。

Mastraを使えば、まさにこのようなシステムを構築できます。異なる種類の作業に合わせてスペシャリストエージェントをセットアップし、各リクエストをどのスペシャリストが処理すべきかを判断するルーターエージェントを作成します。以下のモデルIDはあくまで例であり、ランキングではありません。ご自身の評価（eval）で勝利し、予算に合う最新のモデルに入れ替えてください。

あなたのチームに3人のスペシャリストがいると考えてみてください。

```typescript
// ./src/mastra/index.ts
import { Mastra } from '@mastra/core';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { anthropic } from '@ai-sdk/anthropic';
import { google } from '@ai-sdk/google';

export const claudeAgent = new Agent({
  id: 'claude-agent',
  instructions: 'You are an expert engineer. Write bugs? You are fired.',
  model: anthropic(process.env.CODE_MODEL ?? 'claude-sonnet-4-5'),
});

export const geminiAgent = new Agent({
  id: 'gemini-agent',
  instructions: 'You are a creative writer. Be weird.',
  model: google(process.env.LONG_CONTEXT_MODEL ?? 'gemini-3-pro-preview'),
});

export const gptAgent = new Agent({
  id: 'gpt-agent',
  instructions: 'You are a helpful assistant. Be boring.',
  model: openai(process.env.GENERAL_MODEL ?? 'gpt-5.2'),
});
```

それぞれに役割があります。コードエージェントは、リポジトリ固有のコーディング評価をパスするモデルであるべきです。ロングコンテキストエージェントは、実際のドキュメントを入力しても内容が支離滅裂にならないモデルであるべきです。そして汎用エージェントは、最高に安価で信頼性が高く、堅実であるべきです。

ここからが面白いところです。インテリジェントなプロキシとして機能するルーターを追加します。

```typescript
export const routerAgent = new Agent({
  id: 'router-agent',
  name: 'The Boss',
  instructions: `You are an intelligent router.
  - Coding -> Claude
  - Poetry -> Gemini
  - Facts -> GPT

  Do not do the work yourself. Delegate.`,
  model: openai(process.env.ROUTER_MODEL ?? 'gpt-5-mini'), // Use a cheap model for routing!
  agents: {
    claudeAgent,
    geminiAgent,
    gptAgent,
  },
});

export const mastra = new Mastra({
  agents: { routerAgent, claudeAgent, geminiAgent, gptAgent },
});
```

ルーター自体は、トラフィックをどこに送るかを決定するだけなので、軽量なモデルで動作します。どのプレミアムモデルを使うかを決めるためにプレミアムな料金を支払う必要はありません。これも測定が必要です。精度の低いルーターは、せっかくの節約チャンスを誤ルーティングに変えてしまいます。

誰かがバブルソートの実装を求めた場合、ルーターはそれをコード作成の作業と認識し、コードスペシャリストに渡します。クリエイティブライティングのプロンプトであれば、表現力や幅広さで選んだモデルに送られます。歴史的事実に関する質問なら汎用エージェントにルーティングし、鮮度や出典が重要な場合は検索（retrieval）を組み合わせるのが理想的です。

## 実用的なメリット

**コスト効率は想像以上に重要です。** 委譲の判断を下す小さなルーティングモデルのコストは、すべてのリクエストを最も高価なプロバイダーで実行する場合の数分の一です。時間が経つにつれ、特に大規模な運用では、これが大きな金額の差となります。本当に高度な知能が必要な時だけ、その対価を支払えばよいのです。

**タスクにモデルを適合させることで品質が向上します。** どのモデルが最適かは、月ごと、タスクごと、そしてプロンプトの形状によって変わります。だからこそ、ルーティング層は、あなたが統合コードを書いた週にTwitterで話題になっていたモデルではなく、自身の評価結果（evals）に基づいて決定されるべきです。

**レジリエンス（回復力）が副次的なメリットになります。** OpenAIが定期的な障害を起こしても（実際に起こります）、ルーターはトラフィックを他のプロバイダーにリダイレクトできます。特定のAPIが復旧するのをただ待つだけで、サービスが停止してしまうことはありません。

これは単に巧妙さを競うためのものではありません。経済的にも技術的にも理にかなったシステムを構築するためのものです。建設現場ですべての作業に同じハンマーを使わないように、すべてのAIタスクに同じ言語モデルを使うべきでもありません。

このアプローチの素晴らしい点は、アプリケーションコードを変更する必要がないことです。あなたは引き続き、ルーターエージェントを呼び出すだけです。どのタスクにどのモデルを使うかを決定する複雑なロジックは、コードベースのあちこちに条件分岐として散らばるのではなく、一箇所に集約され、一度設定するだけで済みます。
