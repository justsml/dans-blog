La plupart des équipes d'ingénierie choisissent un modèle de langage et s'y tiennent. Un fournisseur, un modèle, toutes les tâches. C'est comme embaucher une seule personne pour faire votre codage, votre rédaction de contenu et vos impôts parce qu'elle était bonne à l'entretien d'embauche.

À n'importe quel moment donné, un modèle est meilleur pour le code, un autre est meilleur avec un contexte long et brouillon, et un autre est le moins cher et le meilleur pour la classification. Les noms changent. La forme du problème ne change pas. Traiter un modèle comme s'il excellait à tout signifie soit payer trop pour des tâches simples, soit obtenir des résultats médiocres sur des tâches spécialisées.

J'ai regardé une équipe brûler des milliers de dollars en exécutant une analyse de sentiment via un modèle à 30 dollars par million de jetons quand un modèle à 0,50 dollars aurait fait le travail aussi bien. Un simple formatage JSON, des tâches de classification basiques, tous passant via leur fournisseur premium. Seule chose qui chauffait, leur facture AWS.

Il existe une meilleure façon, et elle n'est pas particulièrement compliquée.

## La délégation plutôt que la dévotion

Et si vous pouviez router les requêtes vers le modèle qui est réellement le mieux adapté à cette tâche spécifique ? Utilisez votre puissant modèle coûteux pour les choses difficiles, mais déposez le parsing et le formatage simples vers quelque chose de moins cher. Obtenez les avantages de plusieurs fournisseurs sans avoir à jongler manuellement avec eux dans votre codebase.

Mastra vous permet de construire exactement ce type de système. Vous mettez en place des agents spécialisés pour différents types de travail, puis vous créez un agent routeur qui figure quel spécialiste devrait gérer chaque requête. Les IDs de modèle ci-dessous sont des exemples, pas un classement. Remplacez-les par les modèles actuels qui gagnent vos évaluations et correspondent à votre budget.

Pensez à ça comme ceci : vous avez trois spécialistes sur votre équipe.

```typescript
// ./src/mastra/index.ts
import { Mastra } from '@mastra/core';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { anthropic } from '@ai-sdk/anthropic';
import { google } from '@ai-sdk/google';

export const claudeAgent = new Agent({
  id: 'claude-agent',
  instructions: 'Vous êtes un expert ingénieur. Écrivez des bugs ? Vous êtes licencié.',
  model: anthropic(process.env.CODE_MODEL ?? 'claude-sonnet-4-5'),
});

export const geminiAgent = new Agent({
  id: 'gemini-agent',
  instructions: 'Vous êtes un écrivain créatif. Soyez bizarre.',
  model: google(process.env.LONG_CONTEXT_MODEL ?? 'gemini-3-pro-preview'),
});

export const gptAgent = new Agent({
  id: 'gpt-agent',
  instructions: 'Vous êtes un assistant utile. Soyez ennuyeux.',
  model: openai(process.env.GENERAL_MODEL ?? 'gpt-5.2'),
});
```

Chacun a un travail. Votre agent de code devrait être le modèle qui passe vos évaluations spécifiques au dépôt. Votre agent de contexte long devrait être celui qui survit à vos documents réels sans transformer le milieu en bouillie. Votre agent général devrait être peu coûteux, fiable et ennuyeux de la meilleure façon possible.

Voici où ça devient intéressant. Vous ajoutez un routeur qui agit comme un proxy intelligent :

```typescript
export const routerAgent = new Agent({
  id: 'router-agent',
  name: 'Le Chef',
  instructions: `Vous êtes un routeur intelligent.
  - Codage -> Claude
  - Poésie -> Gemini
  - Faits -> GPT

  Ne faites pas le travail vous-même. Déléguez.`,
  model: openai(process.env.ROUTER_MODEL ?? 'gpt-5-mini'), // Utilisez un modèle peu coûteux pour le routage !
  agents: {
    claudeAgent,
    geminiAgent,
    gptAgent,
  },
});

export const mastra = new Mastra({
  agents: { routerAgent, claudeAgent, geminiAgent, gptAgent },
});
```

Le routeur lui-même tourne sur un modèle léger car il prend simplement des décisions sur où envoyer le trafic. Vous ne payez pas de taux premium pour déterminer quel autre modèle premium utiliser. Mesurez ceci aussi ; un mauvais routeur secrètement transforme les économies en erreurs de routage.

Quand quelqu'un demande une implémentation d'un tri à bulles, le routeur reconnaît que c'est un travail de code et le transmet à votre spécialiste de code. Prompt d'écriture créative ? Celui-ci va vers le modèle que vous avez choisi pour la voix et l'étendue. Question factuelle sur des événements historiques ? Routez-le vers l'agent général, idéalement avec une récupération lorsque la fraîcheur ou la citation compte.

## Les bénéfices pratiques

**L'efficacité des coûts compte plus que vous ne le pensez.** Un petit modèle de routage faisant des décisions de délégation coûte une fraction de l'exécution de chaque requête via votre fournisseur le plus cher. Dans le temps, surtout à grande échelle, cela s'ajoute à de vrais économies. Vous ne payez pour l'intelligence lourde que lorsque vous en avez vraiment besoin.

**La qualité s'améliore quand vous accordez les modèles aux tâches.** Le gagnant change par mois, par tâche et par la forme de la requête. C'est pourquoi la couche de routage devrait dépendre de vos évaluations, pas du modèle qui gagnait sur Twitter la semaine où vous avez écrit l'intégration.

**La résilience devient un bénéfice secondaire.** Quand OpenAI a l'une de ses pannes périodiques (et il en a), votre routeur peut rediriger le trafic vers d'autres fournisseurs. Vous n'êtes pas à sec en attendant qu'une API spécifique revienne en ligne.

Ce n'est pas juste pour être astucieux pour le plaisir. C'est construire des systèmes qui ont du sens à la fois financièrement et techniquement. Vous n'utiliseriez pas le même marteau pour chaque tâche de construction, et vous ne devriez probablement pas utiliser le même modèle de langage pour chaque tâche d'IA non plus.

La beauté de cette approche est que votre code d'application ne change pas. Vous continuez simplement à appeler votre agent routeur. La complexité de décider quel modèle utiliser pour quelle tâche vit en un seul endroit, configuré une fois, plutôt qu'étalé dans votre codebase dans une logique conditionnelle disparate.
