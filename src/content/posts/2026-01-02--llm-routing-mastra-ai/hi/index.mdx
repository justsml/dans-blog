subTitle: "LLM रूटिंग, अभी बहुत ट्रेंड में है"
---

ज़्यादातर इंजीनियरिंग टीमें एक लैंग्वेज मॉडल चुनती हैं और उसी पर टिकी रहती हैं। एक प्रोवाइडर, एक मॉडल, सारे काम। यह ठीक उसी तरह है जैसे किसी एक शख्स को अपना कोडिंग, कॉपीराइटिंग और टैक्स का काम सौंप दो सिर्फ इसलिए कि वह पहले इंटरव्यू में अच्छा था।

किसी भी पल, एक मॉडल कोडिंग में बेहतर होता है, दूसरा लंबे और उलझे हुए कॉन्टेक्स्ट में बेहतर होता है, और तीसरा क्लासिफिकेशन के लिए सबसे सस्ता और भरोसेमंद विकल्प होता है। नाम बदलते रहते हैं। समस्या का स्वरूप नहीं। किसी एक मॉडल को हर चीज़ में माहिर मानना मतलब है कि आप साधारण कामों के लिए ज़्यादा पैसे दे रहे हैं या फिर विशेषज्ञता वाले कामों पर औसत नतीजे पा रहे हैं।

मैंने एक टीम को हज़ारों डॉलर बर्बाद करते देखा है, जो सेंटिमेंट एनालिसिस के लिए $30-प्रति-मिलियन-टोकन वाले मॉडल का इस्तेमाल कर रही थी, जबकि $0.50 वाला मॉडल भी उतना ही अच्छा काम कर देता। साधारण JSON फ़ॉर्मेटिंग, बेसिक क्लासिफिकेशन टास्क, सब कुछ उनके प्रीमियम प्रोवाइडर से जा रहा था। सिर्फ उनका AWS बिल ही गर्म हो रहा था।

इससे बेहतर तरीका है, और वह खास मुश्किल भी नहीं।

## समर्पण से बेहतर, सौंपना

क्या हो अगर आप हर रिक्वेस्ट को उस मॉडल पर भेज सकें जो उस खास काम के लिए सबसे बेहतर हो? मुश्किल कामों के लिए अपना महंगा पावरहाउस इस्तेमाल करो, लेकिन साधारण पार्सिंग और फ़ॉर्मेटिंग को किसी सस्ते मॉडल पर डाल दो। कई प्रोवाइडर्स के फ़ायदे उठाओ, बिना अपने कोडबेस में उन्हें हाथ से जोड़ने-घटाने के।

Mastra ठीक इसी तरह का सिस्टम बनाने देता है। आप अलग-अलग कामों के लिए विशेषज्ञ एजेंट बनाते हो, फिर एक राउटर एजेंट बनाते हो जो तय करता है कि हर रिक्वेस्ट किस विशेषज्ञ को मिलनी चाहिए। नीचे दिए मॉडल ID उदाहरण हैं, कोई लीडरबोर्ड नहीं। उन्हें उन मॉडल्स से बदलो जो तुम्हारे evals में जीतते हैं और तुम्हारे बजट में फिट बैठते हैं।

इसे ऐसे समझो: आपकी टीम में तीन विशेषज्ञ हैं।

```typescript
// ./src/mastra/index.ts
import { Mastra } from '@mastra/core';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { anthropic } from '@ai-sdk/anthropic';
import { google } from '@ai-sdk/google';

export const claudeAgent = new Agent({
  id: 'claude-agent',
  instructions: 'You are an expert engineer. Write bugs? You are fired.',
  model: anthropic(process.env.CODE_MODEL ?? 'claude-sonnet-4-5'),
});

export const geminiAgent = new Agent({
  id: 'gemini-agent',
  instructions: 'You are a creative writer. Be weird.',
  model: google(process.env.LONG_CONTEXT_MODEL ?? 'gemini-3-pro-preview'),
});

export const gptAgent = new Agent({
  id: 'gpt-agent',
  instructions: 'You are a helpful assistant. Be boring.',
  model: openai(process.env.GENERAL_MODEL ?? 'gpt-5.2'),
});
```

हर एक का एक काम है। आपका कोड एजेंट वह मॉडल होना चाहिए जो आपके रेपो-स्पेसिफिक कोडिंग evals पास करे। आपका लॉन्ग-कॉन्टेक्स्ट एजेंट वह होना चाहिए जो आपके असली दस्तावेज़ों को बीच में अस्त-व्यस्त किए बिना संभाल ले। आपका जनरल एजेंट सस्ता, भरोसेमंद और सबसे अच्छे अंदाज़ में बोरिंग होना चाहिए।

यहाँ मज़ेदार हिस्सा शुरू होता है। आप एक राउटर जोड़ते हो जो एक इंटेलिजेंट प्रॉक्सी की तरह काम करता है:

```typescript
export const routerAgent = new Agent({
  id: 'router-agent',
  name: 'The Boss',
  instructions: `You are an intelligent router.
  - Coding -> Claude
  - Poetry -> Gemini
  - Facts -> GPT

  Do not do the work yourself. Delegate.`,
  model: openai(process.env.ROUTER_MODEL ?? 'gpt-5-mini'), // Use a cheap model for routing!
  agents: {
    claudeAgent,
    geminiAgent,
    gptAgent,
  },
});

export const mastra = new Mastra({
  agents: { routerAgent, claudeAgent, geminiAgent, gptAgent },
});
```

राउटर खुद एक हल्के मॉडल पर चलता है क्योंकि वह सिर्फ यह तय कर रहा है कि ट्रैफिक कहाँ भेजना है। आप यह जानने के लिए प्रीमियम दर नहीं दे रहे कि कौन से दूसरे प्रीमियम मॉडल का इस्तेमाल करना है। इसे भी मापो; एक खराब राउटर चुपचाप बचत को गलत रास्तों में बदल देता है।

जब कोई बबल सॉर्ट का इम्प्लीमेंटेशन मांगता है, राउटर उसे कोड का काम पहचानकर आपके कोड विशेषज्ञ को सौंप देता है। क्रिएटिव राइटिंग का प्रॉम्प्ट? वह उस मॉडल को जाता है जिसे आपने आवाज़ और रेंज के लिए चुना है। ऐतिहासिक घटनाओं पर तथ्यात्मक सवाल? उसे जनरल एजेंट को भेजो, खास तौर पर तब जब ताज़गी या साइटेशन मायने रखती हो।

## व्यावहारिक फ़ायदे

**कॉस्ट एफिशिएंसी उतनी ही अहम है जितनी तुम सोचते हो।** एक छोटा राउटिंग मॉडल जो प्रतिनिधित्व के फ़ैसले करता है, वह आपके सबसे महंगे प्रोवाइडर से हर रिक्वेस्ट चलाने के खर्चे का एक छोटा सा हिस्सा है। समय के साथ, खास तौर पर स्केल पर, यह असली पैसे बचाती है। आप भारी-भरकम इंटेलिजेंस के लिए तभी पैसे देते हो जब आपको वास्तव में उसकी ज़रूरत होती है।

**क्वालिटी बेहतर होती है जब आप मॉडल को काम से मिलाते हो।** जीतने वाला हर महीने, हर काम और हर प्रॉम्प्ट के आकार से बदलता है। इसीलिए राउटिंग लेयर को आपके evals पर निर्भर होना चाहिए, न कि उस मॉडल पर जो उस हफ्ते Twitter पर छाया हुआ था जब आपने इंटीग्रेशन लिखा था।

**लचीलापन एक अतिरिक्त फ़ायदा बन जाता है।** जब OpenAI का एक उसके आवर्ती आउटेज होता है (और होते हैं), आपका राउटर ट्रैफिक को दूसरे प्रोवाइडर्स पर भेज सकता है। आप एक खास API के ऑनलाइन आने का इंतज़ार करते हुए ठप नहीं पड़े रहते।

यह सिर्फ होशियारी दिखाने के लिए नहीं है। यह ऐसे सिस्टम बनाने के बारे में है जो वित्तीय और तकनीकी दोनों तौर पर समझदारी हों। आप हर निर्माण काम के लिए एक ही हथौड़ा नहीं इस्तेमाल करेंगे, और शायद हर AI काम के लिए एक ही लैंग्वेज मॉडल भी नहीं इस्तेमाल करना चाहिए।

इस तरीके की खूबसूरती यह है कि आपका एप्लिकेशन कोड नहीं बदलता। आप अब भी सिर्फ अपने राउटर एजेंट को कॉल करते हो। यह तय करने की जटिलता कि किस काम के लिए कौन सा मॉडल इस्तेमाल करना है, एक ही जगह पर रहती है, एक बार कॉन्फ़िगर की जाती है, बजाय इसके कि आपके पूरे कोडबेस में ढेर सारी कंडीशनल लॉजिक में बिखरी हो।
