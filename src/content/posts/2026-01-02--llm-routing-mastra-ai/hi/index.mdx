ज्यादातर इंजीनियरिंग टीमें एक लैंग्वेज मॉडल चुनती हैं और उसी पर टिकी रहती हैं। एक ही प्रोवाइडर, एक ही मॉडल, सारे काम। यह वैसा ही है जैसे आप कोडिंग, कॉपीराइटिंग और टैक्स भरने के लिए सिर्फ एक ही इंसान को काम पर रख लें क्योंकि वह पहले इंटरव्यू में अच्छा था।

किसी भी समय, एक मॉडल कोड में बेहतर होता है, दूसरा उलझे हुए लंबे कॉन्टेक्स्ट (context) को संभालने में, और तीसरा वर्गीकरण (classification) जैसे छोटे कामों के लिए सबसे सस्ता और भरोसेमंद होता है। नाम बदलते रहते हैं, लेकिन समस्या का रूप नहीं बदलता। यह मानना कि एक ही मॉडल हर चीज़ में माहिर है, इसका मतलब है कि या तो आप साधारण कामों के लिए ज़रूरत से ज़्यादा पैसे दे रहे हैं या जटिल कामों के लिए औसत परिणाम पा रहे हैं।

मैंने एक टीम को एक ऐसे मॉडल पर हज़ारों डॉलर फूँकते देखा जो $30 प्रति मिलियन टोकन चार्ज करता था, जबकि वही सेंटीमेंट एनालिसिस (sentiment analysis) का काम $0.50 वाला मॉडल भी उतनी ही बखूबी कर सकता था। साधारण JSON फ़ॉर्मेटिंग और बुनियादी क्लासिफिकेशन जैसे काम भी उनके प्रीमियम प्रोवाइडर के पास जा रहे थे। तपिश सिर्फ उनके AWS बिल में महसूस हो रही थी।

इसका एक बेहतर तरीका है, और वह बहुत जटिल नहीं है।

## निष्ठा के बजाय प्रतिनिधि मंडल (Delegation Over Devotion)

क्या हो अगर आप रिक्वेस्ट को उस मॉडल पर भेज सकें जो उस खास काम के लिए सबसे उपयुक्त है? मुश्किल कामों के लिए अपने महंगे पावरहाउस का इस्तेमाल करें, लेकिन साधारण पार्सिंग और फ़ॉर्मेटिंग को किसी सस्ते मॉडल पर डाल दें। कई प्रोवाइडर्स का फायदा उठाएं, वह भी अपने कोडबेस में मैन्युअल रूप से उनके साथ उलझे बिना।

Mastra आपको बिल्कुल ऐसा ही सिस्टम बनाने की सुविधा देता है। आप अलग-अलग तरह के कामों के लिए विशेषज्ञ एजेंट सेटअप करते हैं, और फिर एक राउटर एजेंट बनाते हैं जो यह तय करता है कि प्रत्येक रिक्वेस्ट को किस विशेषज्ञ के पास जाना चाहिए। नीचे दिए गए मॉडल ID सिर्फ उदाहरण हैं, कोई रैंकिंग नहीं। इन्हें उन मॉडल्स से बदलें जो आपके इवैल्यूएशन (evals) में खरे उतरें और आपके बजट में फिट बैठें।

इसे इस तरह सोचें: आपकी टीम में तीन विशेषज्ञ हैं।

```typescript
// ./src/mastra/index.ts
import { Mastra } from '@mastra/core';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { anthropic } from '@ai-sdk/anthropic';
import { google } from '@ai-sdk/google';

export const claudeAgent = new Agent({
  id: 'claude-agent',
  instructions: 'You are an expert engineer. Write bugs? You are fired.',
  model: anthropic(process.env.CODE_MODEL ?? 'claude-sonnet-4-5'),
});

export const geminiAgent = new Agent({
  id: 'gemini-agent',
  instructions: 'You are a creative writer. Be weird.',
  model: google(process.env.LONG_CONTEXT_MODEL ?? 'gemini-3-pro-preview'),
});

export const gptAgent = new Agent({
  id: 'gpt-agent',
  instructions: 'You are a helpful assistant. Be boring.',
  model: openai(process.env.GENERAL_MODEL ?? 'gpt-5.2'),
});
```

हर एक का अपना काम है। आपका कोड एजेंट वह मॉडल होना चाहिए जो आपके रेपो-विशिष्ट (repo-specific) कोडिंग इवैल्यूएशन को पास करे। आपका लॉन्ग-कॉन्टेक्स्ट एजेंट वह होना चाहिए जो आपके वास्तविक दस्तावेज़ों को बिना गड़बड़ी के समझ सके। आपका जनरल एजेंट सस्ता, भरोसेमंद और अपने काम में माहिर होना चाहिए।

यहाँ चीज़ें दिलचस्प हो जाती हैं। आप एक राउटर जोड़ते हैं जो एक बुद्धिमान प्रॉक्सी के रूप में काम करता है:

```typescript
export const routerAgent = new Agent({
  id: 'router-agent',
  name: 'The Boss',
  instructions: `You are an intelligent router.
  - Coding -> Claude
  - Poetry -> Gemini
  - Facts -> GPT

  Do not do the work yourself. Delegate.`,
  model: openai(process.env.ROUTER_MODEL ?? 'gpt-5-mini'), // Use a cheap model for routing!
  agents: {
    claudeAgent,
    geminiAgent,
    gptAgent,
  },
});

export const mastra = new Mastra({
  agents: { routerAgent, claudeAgent, geminiAgent, gptAgent },
});
```

राउटर खुद एक हल्के (lightweight) मॉडल पर चलता है क्योंकि यह केवल यह निर्णय ले रहा है कि ट्रैफिक को कहाँ भेजना है। आप यह तय करने के लिए प्रीमियम दरों का भुगतान नहीं कर रहे हैं कि किस अन्य प्रीमियम मॉडल का उपयोग करना है। इसे भी मापें; एक खराब राउटर चुपचाप आपकी बचत को गलत फैसलों में बदल सकता है।

जब कोई बबल सॉर्ट (bubble sort) इम्प्लीमेंटेशन मांगता है, तो राउटर इसे कोड के काम के रूप में पहचानता है और इसे आपके कोड विशेषज्ञ को सौंप देता है। कोई रचनात्मक लेखन (creative writing) का प्रॉम्प्ट? वह उस मॉडल के पास जाता है जिसे आपने आवाज़ और रेंज के लिए चुना है। ऐतिहासिक घटनाओं के बारे में तथ्यात्मक प्रश्न? इसे जनरल एजेंट के पास भेजें, और ताज़गी या संदर्भ के लिए रिट्रीवल (retrieval) का उपयोग करें।

## व्यावहारिक लाभ

**लागत दक्षता (Cost efficiency) आपकी सोच से कहीं ज़्यादा मायने रखती है।** डेलीगेशन के निर्णय लेने वाला एक छोटा राउटिंग मॉडल, हर रिक्वेस्ट को अपने सबसे महंगे प्रोवाइडर के माध्यम से चलाने की तुलना में बहुत कम खर्चीला होता है। समय के साथ, विशेष रूप से बड़े पैमाने पर, यह काफी बचत करता है। आप भारी-भरकम बुद्धिमत्ता के लिए तभी भुगतान करते हैं जब आपको वास्तव में इसकी आवश्यकता होती है।

**जब आप काम के हिसाब से मॉडल चुनते हैं, तो गुणवत्ता में सुधार होता है।** विजेता हर महीने, हर काम और हर प्रॉम्प्ट के साथ बदलता रहता है। यही कारण है कि राउटिंग लेयर आपके इवैल्यूएशन पर निर्भर होनी चाहिए, न कि उस मॉडल पर जो उस हफ्ते चर्चा में था जब आपने इसे बनाया था।

**लचीलापन (Resilience) एक अतिरिक्त लाभ बन जाता है।** जब OpenAI जैसे प्रोवाइडर्स की सेवाओं में रुकावट आती है (जो कि होती रहती है), तो आपका राउटर ट्रैफिक को अन्य प्रोवाइडर्स की ओर मोड़ सकता है। आप एक विशिष्ट API के वापस ऑनलाइन आने के इंतज़ार में हाथ पर हाथ धरे नहीं बैठे रहेंगे।

यह सिर्फ चतुर दिखने के लिए नहीं है। यह उन सिस्टम्स के निर्माण के बारे में है जो वित्तीय और तकनीकी दोनों रूप से समझदारी भरे हों। आप निर्माण के हर काम के लिए एक ही हथौड़े का उपयोग नहीं करेंगे, और आपको हर AI काम के लिए एक ही लैंग्वेज मॉडल का उपयोग भी नहीं करना चाहिए।

इस दृष्टिकोण की खूबी यह है कि आपका एप्लिकेशन कोड नहीं बदलता है। आप अभी भी केवल अपने राउटर एजेंट को कॉल करते हैं। किस काम के लिए कौन सा मॉडल उपयोग करना है, यह तय करने की जटिलता एक ही स्थान पर रहती है, बजाय इसके कि आपके पूरे कोडबेस में ढेर सारी कंडीशनल लॉजिक (conditional logic) फैली हो।
