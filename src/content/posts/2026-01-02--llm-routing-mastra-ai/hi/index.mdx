ज़्यादातर इंजीनियरिंग टीमें एक भाषा मॉडल चुनती हैं और उसी पर टिकी रहती हैं। एक प्रोवाइडर, एक मॉडल, सारे काम। ये वैसा ही है जैसे एक ही व्यक्ति को अपना कोडिंग, कॉपीराइटिंग और टैक्स करने के लिए रख लो क्योंकि वो पहले इंटरव्यू में अच्छा लगा था।

किसी भी समय, एक मॉडल कोड में बेहतर होता है, दूसरा लंबे अव्यवस्थित कॉन्टेक्स्ट में, और तीसरा क्लासिफिकेशन जैसे सस्ते बोरिंग काम के लिए सबसे अच्छा होता है। नाम बदलते रहते हैं। समस्या का आकार नहीं बदलता। एक मॉडल को हर चीज में माहिर मानना या तो मतलब है कि सिम्पल टास्क के लिए ज़्यादा पैसे खर्च कर रहे हो या स्पेशलाइज़्ड काम पर खराब नतीजे पा रहे हो।

मैंने एक टीम को हज़ारों डॉलर जलाते देखा जो सेंटिमेंट एनालिसिस को $30 प्रति मिलियन-टोकन मॉडल से करवा रही थी जबकि $0.50 वाला मॉडल उतना ही अच्छा काम कर सकता था। सिम्पल JSON फॉर्मेटिंग, बेसिक क्लासिफिकेशन टास्क, सब कुछ उनके प्रीमियम प्रोवाइडर से हो रहा था। जो चीज़ गर्म हो रही थी वो सिर्फ उनका AWS बिल था।

एक बेहतर तरीका है, और वो खास मुश्किल नहीं है।

## Delegation Over Devotion

क्या हो अगर तुम रिक्वेस्ट उस मॉडल पर रूट कर सको जो उस खास टास्क के लिए सबसे उपयुक्त हो? अपने महंगे पावरहाउस का इस्तेमाल कठिन कामों के लिए करो, लेकिन सिम्पल पार्सिंग और फॉर्मेटिंग को कुछ सस्ते विकल्प पर कर दो। एकाधिक प्रोवाइडर के फायदे पाओ बिना उन्हें अपने कोडबेस में मैन्युअली जबल किए।

Mastra तुम्हें ऐसा सिस्टम बनाने देता है। तुम अलग-अलग तरह के काम के लिए स्पेशलिस्ट एजेंट सेट करते हो, फिर एक राउटर एजेंट बनाते हो जो तय करता है कि किस स्पेशलिस्ट को हर रिक्वेस्ट मिलनी चाहिए। नीचे दिए मॉडल आईडी उदाहरण हैं, लीडरबोर्ड नहीं। इन्हें अपने वर्तमान मॉडलों से बदल दो जो तुम्हारे evals पास करें और बजट में फिट आएं।

इस तरह सोचो: तुम्हारी टीम में तीन स्पेशलिस्ट हैं।

```typescript
// ./src/mastra/index.ts
import { Mastra } from '@mastra/core';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { anthropic } from '@ai-sdk/anthropic';
import { google } from '@ai-sdk/google';

export const claudeAgent = new Agent({
  id: 'claude-agent',
  instructions: 'You are an expert engineer. Write bugs? You are fired.',
  model: anthropic(process.env.CODE_MODEL ?? 'claude-sonnet-4-5'),
});

export const geminiAgent = new Agent({
  id: 'gemini-agent',
  instructions: 'You are a creative writer. Be weird.',
  model: google(process.env.LONG_CONTEXT_MODEL ?? 'gemini-3-pro-preview'),
});

export const gptAgent = new Agent({
  id: 'gpt-agent',
  instructions: 'You are a helpful assistant. Be boring.',
  model: openai(process.env.GENERAL_MODEL ?? 'gpt-5.2'),
});
```

हर एक का अपना काम है। तुम्हारा कोड एजेंट वो मॉडल होना चाहिए जो तुम्हारी repo-स्पेसिफिक कोडिंग evals पास करे। तुम्हारा लॉन्ग-कॉन्टेक्स्ट एजेंट वो होना चाहिए जो तुम्हारे असली डॉक्यूमेंट्स में सर्वाइव करे बिना बीच वाले हिस्से को सूप बनाए। तुम्हारा जeneral एजेंट सस्ता, भरोसेमंद, और अच्छे मतलब में बोरिंग होना चाहिए।

यहीं पर दिलचस्पी आती है। तुम एक राउटर जोड़ते हो जो इंटेलिजेंट प्रॉक्सी की तरह काम करे:

```typescript
export const routerAgent = new Agent({
  id: 'router-agent',
  name: 'The Boss',
  instructions: `You are an intelligent router.
  - Coding -> Claude
  - Poetry -> Gemini
  - Facts -> GPT

  Do not do the work yourself. Delegate.`,
  model: openai(process.env.ROUTER_MODEL ?? 'gpt-5-mini'), // Use a cheap model for routing!
  agents: {
    claudeAgent,
    geminiAgent,
    gptAgent,
  },
});

export const mastra = new Mastra({
  agents: { routerAgent, claudeAgent, geminiAgent, gptAgent },
});
```

राउटर खुद एक लाइटवेट मॉडल पर चलता है क्योंकि सिर्फ तय कर रहा है कि ट्रैफिक कहां भेजना है। तुम उस प्रीमियम रेट नहीं दे रहे हो जिस प्रीमियम मॉडल का इस्तेमाल करना है। इसे भी मेज़ो; एक बुरा राउटर चुपचाप सेविंग्स को गलत रूट में बदल देता है।

जब कोई बबल सॉर्ट इम्प्लीमेंटेशन मांगता है, राउटर इसे कोड का काम पहचानता है और तुम्हारे कोड स्पेशलिस्ट को सौंपता है। क्रिएटिव राइटिंग प्रॉम्प्ट? वो उस मॉडल पर जाता है जिसे तुमने वॉइस और रेंज के लिए चुना है। ऐतिहासिक घटनाओं के बारे में फैक्चुअल सवाल? जeneral एजेंट पर रूट करो, आइडियली रिट्रीवल के साथ जब फ्रेशनेस या साइटेशन मायने रखती है।

## व्यावहारिक फायदे

**कॉस्ट एफिशिएंसी इसलिए मायने रखती है क्योंकि तुम सोचते हो।** एक छोटा राउटिंग मॉडल जो डेलिगेशन के फैसले करता है उसकी लागत तुम्हारे सबसे महंगे प्रोवाइडर से हर रिक्वेस्ट चलाने की तुलना में एक अंश है। समय के साथ, खासकर स्केल पर, ये असली पैसों में जुड़ जाता है। तुम सिर्फ तभी भारी-ड्यूटी इंटेलिजेंस के लिए पैसे देते हो जब तुम्हें वाकई चाहिए।

**क्वालिटी बेहतर होती है जब मॉडल टास्क से मैच करते हैं।** विजेता महीने, टास्क और प्रॉम्प्ट शेप से बदलता है। इसीलिए राउटिंग लेयर तुम्हारे evals पर निर्भर होनी चाहिए, उस मॉडल पर नहीं जो उस हफ्ते ट्विटर जीत रहा था जब तुमने इंटीग्रेशन लिखी थी।

**रिज़ीलिएंस एक साइड बेनिफिट बन जाती है।** जब OpenAI के पीरियodic आउटेज होते हैं (और होते हैं), तुम्हारा राउटर दूसरे प्रोवाइडर पर ट्रैफिक भेज सकता है। तुम एक खास API के वापस आने का इंतज़ार में नहीं मर रहे।

ये सिर्फ क्लेवरनेस के लिए नहीं है। ये उन सिस्टम्स को बनाने के बारे में है जो फाइनेंशियली और टेक्निकली सेंस बनाते हैं। तुम हर कंस्ट्रक्शन टास्क के लिए एक ही हथौड़ा नहीं इस्तेमाल करते, और शायद तुम्हें हर AI टास्क के लिए एक ही भाषा मॉडल नहीं इस्तेमाल करना चाहिए।

इस एप्रोच की खूबसूरती ये है कि तुम्हारा ऐप्लिकेशन कोड नहीं बदलता। तुम अभी भी सिर्फ अपने राउटर एजेंट को कॉल करते हो। किस मॉडल को किस टास्क के लिए इस्तेमाल करने का फैसला एक जगह है, एक बार कॉन्फ़िगर किया गया, कोडबेस में कंडीशनल लॉजिक में नहीं बिखरा हुआ।
