subTitle: "LLM राउटिंग, अभी बहुत चर्चा में है"
---

ज्यादातर इंजीनियरिंग टीमें एक लैंग्वेज मॉडल चुनती हैं और उसी पर टिकी रहती हैं। एक प्रोवाइडर, एक मॉडल, सभी टास्क। यह ऐसे ही है जैसे आपने किसी एक व्यक्ति को कोडिंग, कॉपीराइटिंग और टैक्स भरने के लिए रख लिया क्योंकि वह पहले इंटरव्यू में अच्छा था।

किसी भी समय, एक मॉडल कोड के लिए बेहतर होता है, दूसरा लंबे और अस्त-व्यस्त कॉन्टेक्स्ट के लिए, और तीसरा क्लासिफिकेशन जैसे बोरिंग काम के लिए सबसे सस्ता वर्कहॉर्स। नाम बदलते रहते हैं। प्रॉब्लम का स्वरूप नहीं बदलता। एक मॉडल को ऐसा मानना कि वह हर चीज़ में माहिर है, मतलब या तो आप सिंपल टास्क के लिए ज़्यादा पैसे दे रहे हैं या स्पेशलाइज़्ड काम में खराब रिजल्ट पा रहे हैं।

मैंने एक टीम को हज़ारों डॉलर जलाते देखा — सेंटीमेंट एनालिसिस को $30 प्रति मिलियन टोकन वाले मॉडल पर चला रहे थे, जबकि $0.50 वाला मॉडल वही काम उतनी ही अच्छी तरह कर सकता था। सिंपल JSON फॉर्मेटिंग, बेसिक क्लासिफिकेशन टास्क, सब उनके प्रीमियम प्रोवाइडर से होकर जा रहा था। गर्म सिर्फ़ उनका AWS बिल हो रहा था।

इसका एक बेहतर तरीक़ा है, और यह ख़ास मुश्किल नहीं है।

## भक्ति से ज़्यादा डेलिगेशन

क्या हो अगर आप रिक्वेस्ट्स को उस मॉडल पर राउट कर सकें जो उस खास टास्क के लिए वाकई सबसे उपयुक्त हो? महंगे पावरहाउस को मुश्किल काम के लिए रखें, लेकिन सिंपल पार्सिंग और फॉर्मेटिंग को किसी सस्ते मॉडल पर भेज दें। कई प्रोवाइडर्स के फ़ायदे मिलें, बिना कोडबेस में उन्हें मैन्युअली जuggling किए।

Mastra आपको इस तरह की सिस्टम बनाने देता है। आप अलग-अलग काम के लिए स्पेशलिस्ट एजेंट सेट करते हैं, फिर एक राउटर एजेंट बनाते हैं जो तय करता है कि हर रिक्वेस्ट को किस स्पेशलिस्ट को सौंपना है। नीचे दिए गए मॉडल ID उदाहरण हैं, लीडरबोर्ड नहीं। उन्हें उन मॉडल्स से बदलें जो आपके evals में जीतते हों और बजट में फिट बैठते हों।

इसे ऐसे समझें: आपके टीम में तीन स्पेशलिस्ट हैं।

```typescript
// ./src/mastra/index.ts
import { Mastra } from '@mastra/core';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { anthropic } from '@ai-sdk/anthropic';
import { google } from '@ai-sdk/google';

export const claudeAgent = new Agent({
  id: 'claude-agent',
  instructions: 'You are an expert engineer. Write bugs? You are fired.',
  model: anthropic(process.env.CODE_MODEL ?? 'claude-sonnet-4-5'),
});

export const geminiAgent = new Agent({
  id: 'gemini-agent',
  instructions: 'You are a creative writer. Be weird.',
  model: google(process.env.LONG_CONTEXT_MODEL ?? 'gemini-3-pro-preview'),
});

export const gptAgent = new Agent({
  id: 'gpt-agent',
  instructions: 'You are a helpful assistant. Be boring.',
  model: openai(process.env.GENERAL_MODEL ?? 'gpt-5.2'),
});
```

हर एक का एक काम है। आपका कोड एजेंट वह मॉडल होना चाहिए जो आपके repo-specific coding evals पास करता है। आपका लॉन्ग-कॉन्टेक्स्ट एजेंट वह होना चाहिए जो आपके असली डॉक्युमेंट्स को संभाल पाए, बीच में soup न बना दे। आपका जनरल एजेंट सस्ता, भरोसेमंद और सबसे अच्छे मायने में बोरिंग होना चाहिए।

यहाँ बात दिलचस्प होती है। आप एक राउटर जोड़ते हैं जो इंटेलिजेंट प्रॉक्सी की तरह काम करता है:

```typescript
export const routerAgent = new Agent({
  id: 'router-agent',
  name: 'The Boss',
  instructions: `You are an intelligent router.
  - Coding -> Claude
  - Poetry -> Gemini
  - Facts -> GPT

  Do not do the work yourself. Delegate.`,
  model: openai(process.env.ROUTER_MODEL ?? 'gpt-5-mini'), // Use a cheap model for routing!
  agents: {
    claudeAgent,
    geminiAgent,
    gptAgent,
  },
});

export const mastra = new Mastra({
  agents: { routerAgent, claudeAgent, geminiAgent, gptAgent },
});
```

राउटर खुद एक हल्के मॉडल पर चलता है क्योंकि वह सिर्फ़ ट्रैफ़िक कहाँ भेजना है, यह तय करता है। आप प्रीमियम रेट्स इसलिए नहीं दे रहे ताकि पता चले कि किस दूसरे प्रीमियम मॉडल का उपयोग करना है। इसे मापें भी; एक खराब राउटर बचत को चुपचाप misroutes में बदल देता है।

जब कोई bubble sort इम्प्लीमेंटेशन मांगता है, तो राउटर उसे कोड का काम पहचानकर कोड स्पेशलिस्ट को भेज देता है। क्रिएटिव राइटिंग प्रॉम्प्ट? वह उस मॉडल को जाता है जिसे आपने voice और range के लिए चुना है। ऐतिहासिक घटनाओं के बारे में तथ्यात्मक सवाल? उसे जनरल एजेंट को भेजें, ideally retrieval के साथ जब freshness या citation मायने रखता हो।

## व्यावहारिक फ़ायदे

**लागत दक्षता इससे ज़्यादा मायने रखती है जितना आप सोचते हैं।** एक छोटा राउटिंग मॉडल जो डेलिगेशन के फैसले लेता है, हर सिंगल रिक्वेस्ट को सबसे महंगे प्रोवाइडर पर चलाने का एक अंश भर खर्च करता है। समय के साथ, खासकर स्केल पर, यह असली पैसे बन जाता है। आप हेवी-ड्यूटी इंटेलिजेंस के लिए तभी पैसे देते हैं जब वाकई ज़रूरत हो।

**मॉडल्स को टास्क से मैच करने पर क्वालिटी बेहतर होती है।** विजेता महीने, टास्क और प्रॉम्प्ट के स्वरूप के हिसाब से बदलता रहता है। इसलिए राउटिंग लेयर को आपके evals पर निर्भर होना चाहिए, न कि उस मॉडल पर जो उस हफ्ते Twitter पर जीत रहा था जब आपने इंटीग्रेशन लिखा था।

**रेज़िलियंस एक साइड बेनिफिट बन जाता है।** जब OpenAI के पीरियडिक आउटेज होते हैं (और वे होते हैं), तो आपका राउटर ट्रैफ़िक को दूसरे प्रोवाइडर्स पर भेज सकता है। आप किसी एक खास API के वापस आने का इंतज़ार करते हुए dead in the water नहीं हैं।

यह सिर्फ़ clever बनने के बारे में नहीं है। यह ऐसे सिस्टम बनाने के बारे है जो financially और technically दोनों तरह से समझदारी के हों। आप हर construction टास्क के लिए एक ही हथौड़े का उपयोग नहीं करेंगे, और शायद हर AI टास्क के लिए एक ही लैंग्वेज मॉडल का उपयोग भी नहीं करना चाहिए।

इस अप्रोच की खूबसूरती यह है कि आपका एप्लिकेशन कोड नहीं बदलता। आप अभी भी बस अपने राउटर एजेंट को कॉल करते हैं। यह तय करने की कॉम्प्लेक्सिटी कि किस टास्क के लिए कौन सा मॉडल उपयोग करना है, एक जगह रहती है, एक बार कॉन्फ़िगर होती है, बजाय इसके कि आपकी कोडबेस में conditional logic के ढेर में बिखरी हो।
