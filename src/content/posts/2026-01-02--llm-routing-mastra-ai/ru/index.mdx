Большинство инженерных команд выбирают одну языковую модель и работают только с ней. Один провайдер, одна модель, все задачи. Это как нанять одного человека для программирования, копирайтинга и бухгалтерии только потому, что он хорошо показал себя на первом собеседовании.

В любой момент времени одна модель лучше справляется с кодом, другая — с длинным запутанным контекстом, а третья — самый дешёвый надёжный работник для классификации. Имена меняются. Форма проблемы — нет. Использование одной модели для всего означает, что вы либо переплачиваете за простые задачи, либо получаете посредственные результаты на специализированных.

Я видел, как команда потратила тысячи долларов на анализ тональности через модель стоимостью $30 за миллион токенов, когда модель за $0.50 справилась бы не хуже. Простое форматирование JSON, базовая классификация — всё шло через премиум-провайдера. Единственное, что нагревалось — их счёт за AWS.

Есть способ лучше, и он не особо сложный.

## Делегирование вместо преданности

Что если вы можете маршрутизировать запросы к модели, которая действительно лучше всего подходит для этой конкретной задачи? Используйте вашу дорогую рабочую лошадку для сложных вещей, но переложите простое парсирование и форматирование на что-то дешёвое. Получите преимущества нескольких провайдеров без необходимости вручную переключать их в коде.

Mastra позволяет построить именно такую систему. Вы настраиваете специализированных агентов для разных типов работ, затем создаёте агента-маршрутизатора, который решает, какой специалист должен обработать каждый запрос. ID моделей ниже — это примеры, а не рейтинг. Замените их на текущие модели, которые проходят ваши эвалы и укладываются в бюджет.

Думайте об этом так: у вас три специалиста в команде.

```typescript
// ./src/mastra/index.ts
import { Mastra } from '@mastra/core';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { anthropic } from '@ai-sdk/anthropic';
import { google } from '@ai-sdk/google';

export const claudeAgent = new Agent({
  id: 'claude-agent',
  instructions: 'You are an expert engineer. Write bugs? You are fired.',
  model: anthropic(process.env.CODE_MODEL ?? 'claude-sonnet-4-5'),
});

export const geminiAgent = new Agent({
  id: 'gemini-agent',
  instructions: 'You are a creative writer. Be weird.',
  model: google(process.env.LONG_CONTEXT_MODEL ?? 'gemini-3-pro-preview'),
});

export const gptAgent = new Agent({
  id: 'gpt-agent',
  instructions: 'You are a helpful assistant. Be boring.',
  model: openai(process.env.GENERAL_MODEL ?? 'gpt-5.2'),
});
```

У каждого своя работа. Ваш кодовый агент должен быть моделью, которая проходит ваши репозиторные кодовые эвалы. Агент для длинного контекста должен выживать среди ваших реальных документов, не превращая середину в суп. Общий агент должен быть дешёвым, надёжным и скучным в лучшем смысле слова.

Вот тут становится интересно. Вы добавляете маршрутизатор, который действует как интеллектуальный прокси:

```typescript
export const routerAgent = new Agent({
  id: 'router-agent',
  name: 'The Boss',
  instructions: `You are an intelligent router.
  - Coding -> Claude
  - Poetry -> Gemini
  - Facts -> GPT

  Do not do the work yourself. Delegate.`,
  model: openai(process.env.ROUTER_MODEL ?? 'gpt-5-mini'), // Use a cheap model for routing!
  agents: {
    claudeAgent,
    geminiAgent,
    gptAgent,
  },
});

export const mastra = new Mastra({
  agents: { routerAgent, claudeAgent, geminiAgent, gptAgent },
});
```

Сам маршрутизатор работает на лёгкой модели, потому что он просто принимает решения о перенаправлении трафика. Вы не платите премиальные ставки за то, чтобы решить, какую другую премиальную модель использовать. Тоже измеряйте; плохой маршрутизатор тихо превращает экономию в ошибочные маршруты.

Когда кто-то просит реализацию пузырьковой сортировки, маршрутизатор распознаёт это как кодовую работу и передаёт вашему специалисту по коду. Креативное письмо? Оно идёт к модели, которую вы выбрали для стиля и разнообразия. Фактический вопрос об исторических событиях? Маршрутизируйте к общему агенту, желательно с поиском, когда важна актуальность или цитирование.

## Практические преимущества

**Экономическая эффективность важнее, чем вы думаете.** Небольшая модель маршрутизации, принимающая решения о делегировании, стоит лишь малую часть стоимости запуска каждого запроса через вашего самого дорогого провайдера. Со временем, особенно в масштабе, это складывается в реальные деньги. Вы платите за тяжёлую интеллектуальную работу только тогда, когда она действительно нужна.

**Качество улучшается, когда модели соответствуют задачам.** Победитель меняется в зависимости от месяца, задачи и формы промпта. Поэтому уровень маршрутизации должен зависеть от ваших эвалов, а не от того, какая модель побеждала в Твиттере на прошлой неделе.

**Устойчивость становится приятным бонусом.** Когда у OpenAI случается один из её периодических сбоев (а они случаются), ваш маршрутизатор может перенаправить трафик к другим провайдерам. Вы не остаётесь с мёртвым кораблём в ожидании возвращения одного конкретного API.

Речь не о том, чтобы быть умным ради умного. Речь о построении систем, которые имеют смысл как финансово, так и технически. Вы бы не использовали один и тот же молот для всех строительных задач, и, вероятно, не должны использовать одну и ту же языковую модель для всех задач ИИ.

Красота этого подхода в том, что код вашего приложения не меняется. Вы по-прежнему просто вызываете агента-маршрутизатора. Сложность решения, какую модель использовать для какой задачи, живёт в одном месте, настроенная один раз, а не разбросана по всему вашему коду в куче условной логики.
