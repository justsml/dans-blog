Большинство инженерных команд выбирают одну языковую модель и придерживаются её. Один провайдер, одна модель, все задачи. Это всё равно что нанять одного человека для написания кода, копирайтинга и налогов — просто потому, что он хорошо прошёл первое собеседование.

В любой момент времени одна модель лучше пишет код, другая лучше работает с длинным запутанным контекстом, а третья — самый дешёвый работяга для классификации. Названия меняются. Форма проблемы — нет. Относиться к одной модели так, будто она превосходна во всём, означает либо переплачивать за простые задачи, либо получать посредственные результаты на специализированных.

Я наблюдал, как команда сжигала тысячи долларов, прогоняя сентимент-анализ через модель за $30 за миллион токенов, когда модель за $0,50 справилась бы не хуже. Простое JSON-форматирование, базовые задачи классификации — всё уходило через их премиум-провайдера. Единственное, что нагревалось, — это их счёт от AWS.

Есть способ лучше, и он не особенно сложен.

## Делегирование вместо преданности

Что, если бы вы могли направлять запросы к модели, которая действительно лучше всего подходит для конкретной задачи? Использовать дорогого тяжеловеса для сложных задач, а простой разбор и форматирование спускать на что-то подешевле. Получать преимущества нескольких провайдеров, не жонглируя ими вручную в своей кодовой базе.

Mastra позволяет построить именно такую систему. Вы создаёте агентов-специалистов для разных типов работы, а затем создаёте агента-роутер, который определяет, какой специалист должен обработать каждый запрос. Указанные ниже идентификаторы моделей — это примеры, а не рейтинг. Замените их на текущие модели, которые побеждают в ваших eval-тестах и вписываются в ваш бюджет.

Думайте об этом так: у вас в команде три специалиста.

```typescript
// ./src/mastra/index.ts
import { Mastra } from '@mastra/core';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { anthropic } from '@ai-sdk/anthropic';
import { google } from '@ai-sdk/google';

export const claudeAgent = new Agent({
  id: 'claude-agent',
  instructions: 'You are an expert engineer. Write bugs? You are fired.',
  model: anthropic(process.env.CODE_MODEL ?? 'claude-sonnet-4-5'),
});

export const geminiAgent = new Agent({
  id: 'gemini-agent',
  instructions: 'You are a creative writer. Be weird.',
  model: google(process.env.LONG_CONTEXT_MODEL ?? 'gemini-3-pro-preview'),
});

export const gptAgent = new Agent({
  id: 'gpt-agent',
  instructions: 'You are a helpful assistant. Be boring.',
  model: openai(process.env.GENERAL_MODEL ?? 'gpt-5.2'),
});
```

У каждого своя работа. Ваш код-агент должен использовать модель, которая проходит ваши специфические eval-тесты для репозитория. Ваш агент для длинного контекста — ту, что выживает в ваших реальных документах, не превращая середину в кашу. Ваш общий агент должен быть дешёвым, надёжным и скучным в самом лучшем смысле.

А вот где становится интересно. Вы добавляете роутер, который работает как интеллектуальный прокси:

```typescript
export const routerAgent = new Agent({
  id: 'router-agent',
  name: 'The Boss',
  instructions: `You are an intelligent router.
  - Coding -> Claude
  - Poetry -> Gemini
  - Facts -> GPT

  Do not do the work yourself. Delegate.`,
  model: openai(process.env.ROUTER_MODEL ?? 'gpt-5-mini'), // Используйте дешёвую модель для роутинга!
  agents: {
    claudeAgent,
    geminiAgent,
    gptAgent,
  },
});

export const mastra = new Mastra({
  agents: { routerAgent, claudeAgent, geminiAgent, gptAgent },
});
```

Сам роутер работает на лёгкой модели, потому что он просто принимает решения о том, куда направить трафик. Вы не платите премиум-ставки, чтобы выяснить, какую другую премиум-модель использовать. Измеряйте и это тоже; плохой роутер незаметно превращает экономию в неверные направления.

Когда кто-то просит реализовать сортировку пузырьком, роутер распознаёт это как код-работу и передаёт её вашему специалисту по коду. Креативное писательское задание? Оно уходит к модели, которую вы выбрали за голос и диапазон. Фактический вопрос об исторических событиях? Направляйте его общему агенту, желательно с retrieval, когда важна свежесть или цитирование.

## Практические преимущества

**Экономическая эффективность важнее, чем вы думаете.** Маленькая модель роутинга, принимающая решения о делегировании, стоит копейки по сравнению с прогоном каждого запроса через самого дорогого провайдера. Со временем, особенно в масштабе, это выливается в реальные деньги. Вы платите за тяжёлую интеллектуальную работу только тогда, когда она действительно нужна.

**Качество повышается, когда вы подбираете модель под задачу.** Победитель меняется от месяца к месяцу, от задачи к задаче и от формулировки промпта к формулировке. Вот почему слой роутинга должен опираться на ваши eval-тесты, а не на то, какая модель побеждала в твиттере на неделе, когда вы писали интеграцию.

**Отказоустойчивость становится приятным побочным эффектом.** Когда у OpenAI случается очередной сбой (а они случаются), ваш роутер может перенаправить трафик другим провайдерам. Вы не остаётесь с пустыми руками в ожидании, пока один конкретный API снова заработает.

Это не про то, чтобы быть умным ради самого умничанья. Это про создание систем, которые имеют смысл и финансово, и технически. Вы же не используете один и тот же молоток для всех строительных задач — и не стоит использовать одну и ту же языковую модель для всех AI-задач.

Прелесть этого подхода в том, что ваш прикладной код не меняется. Вы по-прежнему просто вызываете своего агента-роутера. Сложность решения, какую модель использовать для какой задачи, живёт в одном месте, настраивается один раз, а не разбросана по всей кодовой базе в куче условной логики.
