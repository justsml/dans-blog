**अपडेट:** इस लेख के परिणामस्वरूप [`llm://` URI स्कीम के लिए एक Internet-Draft](https://datatracker.ietf.org/doc/draft-levy-llm-uri-scheme/) तैयार की गई।
</blockquote>


क्या आपको वो पुराने बुरे दिन याद हैं जब डेटाबेस से कनेक्ट करने का मतलब था अनगिनत एनवायरनमेंट वेरिएबल्स के साथ जूझना?

यह एक नाज़ुक संरचना थी। `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`... या फिर क्या यह `DB_USERNAME` था? क्या यह `DB_PASS` था या `DB_PWD`? क्या मुझे इस बार `PG_*` प्रीफिक्स चाहिए? और टाइमआउट सेटिंग कहाँ जाती है?

यह एक खतरनाक जोखरी थी, जो आपके प्रोडक्शन बिल्ड को गिरा सकती थी क्योंकि आपने `HOST` को कैपिटलाइज़ करना भूल गए।

फिर, किसी को यह शानदार विचार आया कि बस एक URL¹ का उपयोग कर लिया जाए:

```bash
postgres://user:pass@host:5432/dbname
```

एक स्ट्रिंग। जो कुछ भी आपको चाहिए। सार्वभौमिक रूप से पार्स करने योग्य। पोर्टेबल। कहें तो... खूबसूरत?

तो फिर हम LLMs के साथ 1999 की तरह क्यों पेश आ रहे हैं?

## Env Var विस्फोट

अभी, मेरी `.env` फाइल abandoned API keys का कब्रगाह जैसी दिखती है। `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `MISTRAL_API_KEY`, `GROQ_API_KEY`। और Azure की बात छोड़ दो—आपको एक endpoint, एक deployment name, एक API version, और बस "नमस्ते" कहने के लिए एक key चाहिए।

यह सिर्फ बदसूरत नहीं है; यह घर्षण है। हर बार जब मैं एक मॉडल बदलना चाहता हूँ या एक नया प्रोवाइडर टेस्ट करना चाहता हूँ, मुझे initialization code फिर से लिखना पड़ता है, specific parameter names के लिए दस्तावेज़ीकरण ढूंढना पड़ता है, और मेरे एनवायरनमेंट कॉन्फ़िगरेशन में तीन और लाइनें जोड़नी पड़ती हैं।

क्या हो अगर हम बस... ~~चुरा लें~~ उधार ले लें DB URL विचार?

## LLM कनेक्शन स्ट्रिंग्स का परिचय

कल्पना करें कि आप एक ही लाइन से अपना पूरा मॉडल इंटरफ़ेस कॉन्फ़िगर करते हैं:

```bash
llm://api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7&max_tokens=1500
llm://api.z.ai/glm-4.7?top_p=0.9&cache=true
```

### LLM कनेक्शन स्ट्रिंग की एनाटॉमी

![LLM कनेक्शन स्ट्रिंग के भाग](./inline-url-diagram-dark.svg)

स्कीम है `llm://`। Host है प्रोवाइडर का API base URL। path है मॉडल का नाम। और query parameters सभी runtime options को संभालती हैं जो आमतौर पर आपके code को गंदा करते हैं।

## Auth चाहिए? बहुत अच्छा, जोड़ लो।

बिल्कुल `postgres://` की तरह, हम authentication को सीधे बेक कर सकते हैं:

```bash
llm://app-name:sk-proj-123456@api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7
```

*नोट: हाँ, URLs में credentials डालना सुरक्षा जोखिम हो सकता है अगर आप उन्हें public logs में पेस्ट कर रहे हैं। लेकिन आधुनिक logging services इन पैटर्न को scrub करने में काफी अच्छे हैं, और honestly, क्या आप अपनी `.env` फाइल से बेहतर सलूक रहे हैं? वेरिफाई करें, sanitize करें, और सावधानी से उपयोग करें।*

## Resiliency? क्यों नहीं।

कई डेटाबेस लाइब्रेरीज़ कई hosts specify करके round-robin failover सपोर्ट करती हैं। हमारे AI agents को भी वही reliability क्यों नहीं मिलनी चाहिए?

```bash
llms://primary.gpt,backup.gpt/gpt-6?temp=0.9
```

वो `llms://` में `s` टाइपो नहीं है। यह plural है। अगर `primary.gpt` hang हो जाता है, तो client automatic retry करता है `backup.gpt`। कोई complex router logic ज़रूरी नहीं।

<blockquote class="inset">एक स्ट्रिंग में आपका **auth**, आपका **endpoint**, और आपके **hyperparameters** — सब कुछ।</blockquote>

## वैकल्पिक फॉर्मेट

मैं `llm://` से शादीशुदा नहीं हूँ। specific scheme कम महत्व रखता है, standard itself ज़्यादा महत्वपूर्ण है।

मैं एक ऐसी दुनिया कल्पना कर सकता हूँ जहाँ brevity के लिए provider-specific schemes का उपयोग करते हैं, जबकि standard structure बनाए रखते हैं:

```bash
ollama://localhost:11434/llama3
vercel://anthropic/sonnet-4.5?temp=0.8&web_search={"maxUses":3}
bedrock://us-west-2.aws/anthropic/sonnet-4.5?temp=0.8&cacheControl=ephemeral
```

exact syntax जो भी हो, core benefits निर्विवाद हैं:

1.  **पोर्टेबिलिटी:** अपना पूरा कॉन्फ़िगरेशन local script से cloud worker में copy & paste करें।
2.  **CLI Friendly:** अपने scripts में एक ही argument पास करें। `my-agent --model "llm://..."` बेहतर है `my-agent --model gpt-4 --temp 0.7 --key $KEY --host ...` से।
3.  **Language Agnostic:** हर प्रोग्रामिंग भाषा के पास एक robust URL parser है। हमें मुफ्त में validation, parsing, और sanitization मिलता है।

<blockquote class="ai-response inset">डेटाबेस की दुनिया ने इसे समझने में दशक लगाए।<br /><b>अच्छी खबर, AI timelines में, यह बस लगभग आधा vibe-year पहले की बात है।</b></blockquote>

## फैसला

हमें एक और complex कॉन्फ़िगरेशन standard या एक नया YAML-based manifest file नहीं चाहिए। हमें बस उस एक टूल का उपयोग करना चाहिए जो पिछले 30 सालों से बाकी इंटरनेट के लिए काम कर रहा है।

आइए पहिया को फिर से invented करना बंद करें और अपने LLM कनेक्शनों के साथ वही सम्मान करना शुरू करें जो हम अपने डेटाबेस देते हैं। आपकी `.env` file (और आपकी sanity) आपको धन्यवाद देगी।
