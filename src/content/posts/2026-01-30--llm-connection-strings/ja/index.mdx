データベースに接続するために、色々な環境変数を取り混ぜて管理していた暗黒時代を覚えているだろうか？

それは繊細な設定の塔だった。`DB_HOST`、`DB_PORT`、`DB_USER`、`DB_PASSWORD`、`DB_NAME`... いや待て、今回は`DB_USERNAME`だったっけ？`DB_PASS`と`DB_PWD`のどっちだ？今回は`PG_*`プレフィックスが必要なのか？そしてタイムアウト設定はどこに書くんだ？

それはカードの脆弱な家で、`HOST`の大文字小文字を間違えただけで本番ビルドが崩壊しかねないものだった。

そして、誰かが素晴らしいアイデアを思いついた。URLを使おうと¹：

```bash
postgres://user:pass@host:5432/dbname
```

一つの文字列。必要なものがすべて。普遍的にパース可能。ポータブル。敢えて言おう...美しい？

なのに、なぜ我々はLLMを1999年のように扱っているのか？

## 環境変数の爆発

今、私の`.env`ファイルは放棄されたAPIキーの墓場のように見える。`OPENAI_API_KEY`、`ANTHROPIC_API_KEY`、`MISTRAL_API_KEY`、`GROQ_API_KEY`。そしてAzureの話はもういいだろう—エンドポイント、デプロイ名、APIバージョン、キーが必要で、「こんにちは」と言うだけなのに。

見た目が悪いだけじゃない。摩擦なのだ。モデルを入れ替えたり、新しいプロバイダーを試したりするたびに、初期化コードを書き直し、特定のパラメータ名のドキュメントを探し、環境設定にさらに3行追加している。

もし、データベースURLのアイデアを... ~~盗んで~~ 借りたらどうだろう？

## LLM接続文字列の登場

モデルインターフェース全体を一行で設定できると想像してみてほしい：

```bash
llm://api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7&max_tokens=1500
llm://api.z.ai/glm-4.7?top_p=0.9&cache=true
```

### LLM接続文字列の構造

![LLM接続文字列の各部分](./inline-url-diagram-dark.svg)

スキームは`llm://`だ。ホストはプロバイダーのAPIベースURL。パスはモデル名。そしてクエリパラメータは、通常コードを散らかすすべてのランタイムオプションを処理する。

## 認証が必要？いいね、追加しよう

`postgres://`と同じように、認証情報をそのまま組み込める：

```bash
llm://app-name:sk-proj-123456@api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7
```

*注：はい、URLに認証情報を入れることは、公開ログに貼り付ける場合セキュリティリスクになりうる。しかし、現代のログサービスはこれらのパターンをかなり上手くスクラブしており、正直なところ、君の`.env`ファイルをそれほど良く扱っているか？検証し、サニタイズし、注意して使おう。*

## 耐障害性？なぜダメなんだ

多くのデータベースライブラリは、複数のホストを指定することでラウンドロビンフェイルオーバーをサポートしている。なぜ我々のAIエージェントにも同じ信頼性を持たせないのか？

```bash
llms://primary.gpt,backup.gpt/gpt-6?temp=0.9
```

`llms://`の`s`はタイポじゃない。複数形だ。`primary.gpt`が応答しない場合、クライアントは自動的に`backup.gpt`を再試行する。複雑なルーターロジックは不要だ。

<blockquote class="inset">**認証**から**エンドポイント**、**ハイパーパラメータ**まで、すべてが一つの文字列に。</blockquote>

## 代替フォーマット

私は`llm://`にこだわっているわけじゃない。特定のスキームより、標準そのものが重要だ。

プロバイダー固有のスキームを簡潔さのために使いつつ、標準的な構造を保つ世界が想像できる：

```bash
ollama://localhost:11434/llama3
vercel://anthropic/sonnet-4.5?temp=0.8&web_search={"maxUses":3}
bedrock://us-west-2.aws/anthropic/sonnet-4.5?temp=0.8&cacheControl=ephemeral
```

正確な構文はどうあれ、核心的な利点は明らかだ：

1.  **ポータブル：** ローカルスクリプトからクラウドワーカーまで、設定全体をコピー＆ペーストできる。
2.  **CLIフレンドリー：** スクリプトに単一の引数を渡すだけ。`my-agent --model "llm://..."`は`my-agent --model gpt-4 --temp 0.7 --key $KEY --host ...`よりずっといい。
3.  **言語に依存しない：** すべてのプログラミング言語には堅牢なURLパーサーがある。検証、パース、サニタイズを無料で手に入れられる。

<blockquote class="ai-response inset">データベースの世界がこれを理解するのに数十年かかった。<br /><b>良い知らせだ。AIのタイムラインでは、それはほんの半年前のバイブ年程度のことだ。</b></blockquote>

## 結論

我々は、もう一つの複雑な設定標準や、新しいYAMLベースのマニフェストファイルを必要としていない。我々が必要なのは、過去30年間、インターネットの他の部分で機能してきたあのツールを使うことだけだ。

車輪の再発明はやめて、LLM接続をデータベースと同じ敬意を持って扱い始めよう。君の`.env`ファイル（と君の正気）が感謝するだろう。
