データベースに接続するために、多種多様な環境変数をやりくりしていた古き悪き時代を覚えていますか？

それは繊細な設定の塔でした。`DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`... おっと、`DB_USERNAME`だったかな？ `DB_PASS`か`DB_PWD`か？ 今回は`PG_*`プレフィックスが必要だろうか？ そして、タイムアウト設定は一体どこに書けばいいんだ？

それは、`HOST`を大文字にするのを忘れただけで本番環境のビルドが崩れ去るような、脆いカードの家でした。

その後、誰かが「URLを使えばいいじゃないか¹」という素晴らしいアイデアを思いつきました。

```bash
postgres://user:pass@host:5432/dbname
```

ひとつの文字列。必要なすべて。普遍的にパース可能。ポータブル。あえて言わせてもらえば... 美しい？

では、なぜLLMを1999年のような扱いにしているのでしょうか？

## 環境変数の爆発

現在、私の`.env`ファイルは、放置されたAPIキーの墓場のようになっています。`OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `MISTRAL_API_KEY`, `GROQ_API_KEY`。そしてAzureについては言わずもがなです。「こんにちは」と言うだけで、エンドポイント、デプロイ名、APIバージョン、そしてキーが必要になります。

これは単に見た目が悪いだけではありません。摩擦（フリクション）なのです。モデルを入れ替えたり、新しいプロバイダーをテストしたりするたびに、初期化コードを書き換え、特定のパラメータ名のためにドキュメントを漁り、環境設定にさらに3行追加することになります。

もし、データベースのURLのアイデアを... ~~盗んで~~ 拝借したらどうなるでしょうか？

## LLM接続文字列の導入

モデルインターフェース全体をたった一行で設定できる様子を想像してみてください。

```bash
llm://api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7&max_tokens=1500
llm://api.z.ai/glm-4.7?top_p=0.9&cache=true
```

### LLM接続文字列の解剖学

![LLM接続文字列の各構成要素](./inline-url-diagram-dark.svg)

スキームは `llm://` です。ホストはプロバイダーのAPIベースURL。パスはモデル名。そしてクエリパラメータが、普段コードを散らかしているすべての実行時オプションを処理します。

## 認証が必要？ もちろんです。

`postgres://` と同じように、認証情報を直接組み込むことができます。

```bash
llm://app-name:sk-proj-123456@api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7
```

*注：URLに認証情報を入れることは、公開ログに貼り付けてしまうとセキュリティリスクになる可能性があります。しかし、現代のロギングサービスはこれらのパターンをスクラブ（除去）するのに非常に優れていますし、正直なところ、あなたの .env ファイルの扱いはそれよりずっとマシですか？ 検証し、サニタイズし、注意して使用してください。*

## 回復性？ もちろん、それも。

多くのデータベースライブラリは、複数のホストを指定することでラウンドロビン方式のフェイルオーバーをサポートしています。AIエージェントにも同じ信頼性があっていいはずではないでしょうか？

```bash
llms://primary.gpt,backup.gpt/gpt-6?temp=0.9
```

その `s` は誤植ではありません。複数形（plural）です。もし `primary.gpt` がハングアップした場合、クライアントは自動的に `backup.gpt` をリトライします。複雑なルーターロジックは必要ありません。

<blockquote class="inset">認証からエンドポイント、ハイパーパラメータまで、すべてが含まれたひとつの文字列。</blockquote>

## 代替フォーマット

私は `llm://` というスキームに固執しているわけではありません。特定のスキームよりも、標準そのものの方が重要です。

標準的な構造を維持しつつ、簡潔さのためにプロバイダー固有のスキームを使用する世界も想像できます。

```bash
ollama://localhost:11434/llama3
vercel://anthropic/sonnet-4.5?temp=0.8&web_search={"maxUses":3}
bedrock://us-west-2.aws/anthropic/sonnet-4.5?temp=0.8&cacheControl=ephemeral
```

正確な構文がどうであれ、その核心的なメリットは否定できません。

1.  **ポータビリティ:** ローカルスクリプトからクラウドワーカーへ、設定全体をコピー＆ペーストできます。
2.  **CLIフレンドリー:** スクリプトに単一の引数を渡すだけです。`my-agent --model "llm://..."` は、`my-agent --model gpt-4 --temp 0.7 --key $KEY --host ...` よりもはるかに優れています。
3.  **言語に依存しない:** すべてのプログラミング言語に堅牢なURLパーサーが存在します。バリデーション、パース、サニタイズを無料で手に入れることができます。

## 結論

新たな複雑な設定標準や、新しいYAMLベースのマニフェストファイルは必要ありません。過去30年間、インターネットの他の部分で機能し続けてきた唯一のツールを使えばいいのです。

車輪の再発明はやめて、データベースと同じようにLLM接続を扱い始めましょう。あなたの `.env` ファイル（そしてあなたの正気）が感謝することでしょう。
