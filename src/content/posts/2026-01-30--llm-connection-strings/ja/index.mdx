データベースに接続する際、雑多な環境変数をやり繰りしなければならなかったあの悪しき時代を覚えているか？

崩れやすい設定の塔だった。`DB_HOST`、`DB_PORT`、`DB_USER`、`DB_PASSWORD`、`DB_NAME`……いや、`DB_USERNAME`だったか？ `DB_PASS`か`DB_PWD`か？ 今回は`PG_*`プレフィックスが必要？ そしてタイムアウト設定はどこに入れるんだ？

`HOST`の大文字小文字を間違えれば本番ビルドが崩落する、危ういカードの家だった。

その後、誰かがURLを使えばいいという素晴らしい考えを出した¹：

```bash
postgres://user:pass@host:5432/dbname
```

一つの文字列。必要なものすべて。普遍的に解析可能。移植可能。あえて言おう……美しい？

ならば、なぜLLMを1999年の時のように扱っているのか？

## 環境変数の爆発

今、私の`.env`ファイルは放置されたAPIキーの墓地だ。`OPENAI_API_KEY`、`ANTHROPIC_API_KEY`、`MISTRAL_API_KEY`、`GROQ_API_KEY`。Azureに至っては—「hello」と言うためだけにエンドポイント、デプロイメント名、APIバージョン、キーが必要だ。

見苦しいだけではない。摩擦だ。モデルを替えたり新プロバイダーを試すたびに、初期化コードを書き直し、パラメータ名のドキュメントを探し回し、環境設定に三行追加している。

DB URLの考えを……~~盗む~~借用したらどうだろう？

## LLM接続文字列の紹介

モデルインターフェース全体を一行で設定できると想像してほしい：

```bash
llm://api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7&max_tokens=1500
llm://api.z.ai/glm-4.7?top_p=0.9&cache=true
```

### LLM接続文字列の解剖

![LLM接続文字列の各構成要素](./inline-url-diagram-dark.svg)

スキームは`llm://`。ホストはプロバイダーのAPIベースURL。パスはモデル名。クエリパラメータが、普段コードを散らかす実行時オプションを処理する。

## 認証が必要？ いいね、追加しよう。

`postgres://`と同様に、認証を直接組み込める：

```bash
llm://app-name:sk-proj-123456@api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7
```

*注：URLに資格情報を入れるのは、公開ログに貼り付けるとセキュリティリスクになり得る。しかし現代のログサービスはこれらのパターンのスクラビングがかなり得意だし、正直、`.env`ファイルの扱いはそんなにマシか？ 検証し、サニタイズし、慎重に使うこと。*

## レジリエncy？ 当然だ。

多くのデータベースライブラリは複数ホストの指定でラウンドロビンフェイルオーバーをサポートしている。AIエージェントにも同じ信頼性があるべきではないか？

```bash
llms://primary.gpt,backup.gpt/gpt-6?temp=0.9
```

`llms://`の`s`はタイプミスではない。複数形だ。`primary.gpt`がハングしたら、クライアントは自動的に`backup.gpt`にリトライする。複雑なルーターロジックは不要。

<blockquote class="inset">一つの文字列に**認証**から**エンドポイント**から**ハイパーパラメータ**まですべて。</blockquote>

## 代替形式

`llm://`に固執しているわけではない。具体的なスキームより標準そのものが重要だ。

簡潔さのためプロバイダー固有のスキームを使う一方、標準構造を維持する世界も想像できる：

```bash
ollama://localhost:11434/llama3
vercel://anthropic/sonnet-4.5?temp=0.8&web_search={"maxUses":3}
bedrock://us-west-2.aws/anthropic/sonnet-4.5?temp=0.8&cacheControl=ephemeral
```

正確な構文はともかく、核心の利点は否定できない：

1.  **移植性：** 全設定をローカルスクリプトからクラウドワーカーにコピー＆ペースト。
2.  **CLI対応：** スクリプトに一つの引数を渡す。`my-agent --model "llm://..."`は`my-agent --model gpt-4 --temp 0.7 --key $KEY --host ...`より優れている。
3.  **言語非依存：** すべてのプログラミング言語に堅牢なURLパーサーがある。検証、解析、サニタイズを無料で得られる。

<blockquote class="ai-response inset">データベース界はこれを見つけるのに何十年もかかった。<br /><b>良いことに、AIの時系列では約半vibe年前だ。</b></blockquote>

## 結論

別の複雑な設定標準や新しいYAMLベースのマニフェストファイルは必要ない。必要なのは、過去30年間インターネットの残りの部分で機能してきた唯一のツールを使うことだ。

車輪の再発明を止めて、LLM接続をデータベースと同じ敬意で扱い始めよう。`.env`ファイル（そしてあなたの理性）が感謝するだろう。
