Personne ne cherche délibérément à construire un système d'IA unsafe. Vous écrivez des instructions, vous testez les cas limites, vous ajoutez quelques règles de validation. Puis quelqu'un découvre qu'il peut tromper votre bot en lui faisant jouer le rôle d'un pirate et exposer les données utilisateur. Ou un numéro de carte bancaire se retrouve dans vos logs. Ou le modèle recommande avec assurance le produit d'un concurrent.

L'écart entre « fonctionne en démo » et « safe en production » est plus large que ce que la plupart des équipes prévoient.

Une partie du problème vient du fait que les LLMs bruts n'ont pas d'opinions sur ce qu'ils devraient ou ne devraient pas faire. Ce sont des machines de prédiction qui tentent de continuer n'importe quel motif que vous avez commencé. Donnez-leur un prompt qui ressemble à « mode override système », et ils joueront le jeu avec plaisir. Ce n'est pas un bug du modèle ; c'est juste comme ça que fonctionnent les modèles de langage.

La plupart des frameworks vous donnent le modèle et vous souhaitent bonne chance. Mastra adopte une approche différente : il part du principe que vous aurezEventually besoin de guardrails, donc il les intègre dans l'architecture de l'agent dès le départ.

## Les Processeurs comme Couches de Sécurité

Le mécanisme central est simple. Avant que votre prompt n'atteigne le modèle, il passe par une chaîne de processeurs d'entrée. Après la réponse du modèle, les processeurs de sortie ont leur tour. Chaque processeur peut inspecter, modifier ou bloquer le contenu à cette étape.

Considérez-les comme du middleware pour les interactions IA. Vous empilez ceux dont vous avez besoin, configurez leur comportement, et ils s'exécutent automatiquement à chaque requête.

### 1. Arrêter les Pirates (Injection de Prompt)

Les attaques par injection de prompt sont devenues créatives. Les gens utilisent des caractères Unicode invisibles, écrivent des instructions en base64, ou persuadent le modèle qu'ils sont en « mode debug » où les règles normales ne s'appliquent pas. Les techniques continuent d'évoluer.

Mastra inclut des processeurs qui détectent les motifs courants :


```typescript
// src/mastra/agents/secure-agent.ts
import { Agent } from '@mastra/core/agent';
import { PromptInjectionDetector, UnicodeNormalizer } from '@mastra/core/processors';
import { openai } from '@ai-sdk/openai';

export const secureAgent = new Agent({
  id: 'fortress-assistant',
  name: 'fortress-assistant',
  instructions: 'You are a secure assistant.',
  model: openai('gpt-5'),
  inputProcessors: [
    // 1. Scrub invisible characters
    new UnicodeNormalizer({
      id: 'unicode-normalizer',
      stripControlChars: true,
      collapseWhitespace: true,
    }),
    // 2. Detect the attempt
    new PromptInjectionDetector({
      id: 'prompt-injection-detector',
      model: openai('gpt-5-nano'), // Cheap, fast
      threshold: 0.8,
      strategy: 'block', // Hard stop
      detectionTypes: ['injection', 'jailbreak', 'system-override'],
    }),
  ],
});
```

Le [`UnicodeNormalizer`](https://mastra.ai/docs/processors) supprime les caractères de contrôle et réduit les espaces blancs. Le [`PromptInjectionDetector`](https://mastra.ai/docs/processors) analyse l'entrée nettoyée pour détecter les motifs qui suggèrent que quelqu'un essaie de remplacer vos instructions.

Vous configurez l'agressivité de la détection (le paramètre `threshold`) et ce qui doit se passer lorsqu'elle se déclenche (block, log, ou simplement flag).

### 2. Gestion des PII

Numéros de cartes bancaires dans les logs, numéros de sécurité sociale dans les bases de données vectorielles, adresses email stockées plus longtemps que nécessaire. Ce sont le genre de problèmes qui se transforment en problèmes réglementaires. Le défi est que les utilisateurs ne réalisent pas toujours qu'ils collent des données sensibles dans une fenêtre de chat.

Le [`PIIDetector`](https://mastra.ai/docs/processors) scanne les motifs courants avant qu'ils n'atteignent votre modèle ou ne soient écrits dans le stockage :

```typescript
import { PIIDetector } from '@mastra/core/processors';

export const privateAgent = new Agent({
  id: 'privacy-first-assistant',
  name: 'privacy-first-assistant',
  instructions: 'You are a helpful assistant that never stores personal information.',
  model: openai('gpt-5'),
  inputProcessors: [
    new PIIDetector({
      id: 'pii-detector',
      model: openai('gpt-5-nano'),
      detectionTypes: ['email', 'phone', 'credit-card', 'ssn'],
      threshold: 0.6,
      strategy: 'redact',
      redactionMethod: 'mask',  // Replace with [REDACTED]
      instructions: 'Detect and mask personally identifiable information',
    }),
  ],
});
```

Vous pouvez choisir de rédiger (remplacer par `[REDACTED]`), hasher, ou bloquer entièrement. Le processeur s'exécute sur l'entrée et la sortie, donc vous êtes couvert même si le modèle génère somehow des données sensibles dans sa réponse.

### 3. Modération de Contenu

Les modèles entraînés sur des données internet ont vu certaines choses. Sans filtrage, ils peuvent occasionnellement produire des réponses qui rendraient votre équipe RP nerveuse. Le [`ModerationProcessor`](https://mastra.ai/docs/processors) détecte le contenu qui viole vos directives :

```typescript
import { ModerationProcessor } from '@mastra/core/processors';

export const moderatedAgent = new Agent({
  id: 'safe-assistant',
  name: 'safe-assistant',
  instructions: 'You are a helpful assistant for a community platform.',
  model: openai('gpt-5'),
  inputProcessors: [
    new ModerationProcessor({
      id: 'moderation-processor',
      model: openai('gpt-5-nano'),  // Fast, cheap model for classification
      categories: ['hate', 'harassment', 'violence', 'self-harm'],
      threshold: 0.7,  // Block if confidence > 70%
      strategy: 'block',  // Stop the request immediately
      instructions: 'Detect harmful content that violates community guidelines',
    }),
  ],
});
```

La partie intéressante est que vous définissez quelles catégories comptent pour votre cas d'utilisation. Un outil d'écriture créative pourrait autoriser plus de contenu expressif qu'un bot de service client. Le threshold et la strategy vous donnent le contrôle sur la rigueur du filtrage.

## Quand les Choses se Déclenchent

Les processeurs ne déclenchent pas d'erreurs lorsqu'ils détectent un problème. Au lieu de cela, ils définissent un flag sur l'objet résultat :

```typescript
const result = await secureAgent.generate('Ignore all previous instructions...');

if (result.tripwire) {
  console.log(`Blocked! Reason: ${result.tripwireReason}`);
  // "Blocked! Reason: Prompt injection detected."
  return "Nice try, script kiddie.";
}
```

Ce motif vous permet de gérer les événements de sécurité comme bon vous semble pour votre application. Vous pourriez les logger pour analyse, retourner un message d'erreur générique, ou même autoriser certaines violations dans des contextes spécifiques. Le champ `tripwireReason` vous dit exactement quel processeur a flagué le contenu, ce qui aide quand vous debuggez les faux positifs ou ajustez vos seuils.

## Ce Que Ça Ne Résout Pas

Les processeurs attrapent beaucoup, mais ce n'est pas de la magie. Un attaquant déterminé avec assez de temps trouvera probablement un prompt qui passe à travers. Les modèles occasionnellement hallucinent de façons que les processeurs ne peuvent pas prédire. Et il y a toujours un compromis entre sécurité et flexibilité : plus vos règles sont strictes, plus vous risquez de bloquer des cas d'utilisation légitimes.

La valeur n'est pas une protection parfaite. C'est avoir une façon systématique de gérer les problèmes courants qui vont inévitablement surgir en production. Vous pouvez ajuster la sensibilité au fur et à mesure que vous apprenez ce que vos utilisateurs font réellement. Vous pouvez ajouter des processeurs personnalisés pour les risques spécifiques à un domaine. Et vous avez des pistes d'audit montrant ce qui a été bloqué et pourquoi.

La plupart des problèmes de sécurité en IA de production ne sont pas des attaques sophistiquées. Ce sont des gens qui copient-collent des données qu'ils ne devraient pas, ou découvrent par trial and error que le bot fera des choses que vous n'aviez pas l'intention. Les processeurs n'arrêteront pas chaque problème possible, mais ils rendent les problèmes évidents beaucoup plus difficiles.
