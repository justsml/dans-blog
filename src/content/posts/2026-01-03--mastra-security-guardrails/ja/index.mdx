安全なAIシステムを構築しようとして不安全になる人はいません。指示を書き、エッジケースをテストし、いくつかのバリデーションルールを追加します。それでも、誰かが海盗船乗りのロールプレイさせてユーザーデータを漏えいさせる方法を考え出すかもしれません。あるいは、ログにクレジットカード番号が記録されるかもしれません。あるいは、モデルが自信満々に競合製品の推介をしているかもしれません。

「デモでは動作する」と「本番では安全だ」の間のギャップは、チームが思っているよりも広いです。

問題の多くは、生のLLMにはすべきこと・すべきでないことに対する意見がないということです。それは単に継続しようとする予測マシンです。「システムオーバーライドモード」のように見えるプロンプトを与えれば、喜んで付き合ってくれます。これはモデルのバグではなく、语言モデルの仕組み就是这样です。

ほとんどのフレームワークはモデルを渡して「おluck」と言いいます。Mastraは違うアプローチを取っています：最終的にガードレールが必要になることを前提として、エージェントアーキテクチャ最初からガードレールを組み込んでいるのです。

## 安全レイヤーとしてのプロセッサ

コアメカニズムは単純です。プロンプトがモデルに届く前に、入力プロセッサのチェーンを通過します。モデルが応答した後、出力プロセッサの出番です。各プロセッサは、その段階でコンテンツ进行检查、修正、またはブロックできます。

AIインタラクションのミドルウェア，就像是考えてください。必要なものをスタックして、動作を設定すれば、すべてのリクエストで自動的に実行されます。

### 1. 海贼を止める（プロンプトインジェクション）

プロンプトインジェクション攻撃は巧妙になっています。人々は不可視のUnicode文字を使い、base64で指示を書いたり、デバッグモードでは通常のルールが適用されないとモデルに信じ込ませたりします。技術は 계속 발전しています.

Mastraには、一般的なパターンを検出するプロセッサが含まれています：


```typescript
// src/mastra/agents/secure-agent.ts
import { Agent } from '@mastra/core/agent';
import { PromptInjectionDetector, UnicodeNormalizer } from '@mastra/core/processors';
import { openai } from '@ai-sdk/openai';

export const secureAgent = new Agent({
  id: 'fortress-assistant',
  name: 'fortress-assistant',
  instructions: 'You are a secure assistant.',
  model: openai('gpt-5'),
  inputProcessors: [
    // 1. Scrub invisible characters
    new UnicodeNormalizer({
      id: 'unicode-normalizer',
      stripControlChars: true,
      collapseWhitespace: true,
    }),
    // 2. Detect the attempt
    new PromptInjectionDetector({
      id: 'prompt-injection-detector',
      model: openai('gpt-5-nano'), // Cheap, fast
      threshold: 0.8,
      strategy: 'block', // Hard stop
      detectionTypes: ['injection', 'jailbreak', 'system-override'],
    }),
  ],
});
```

[`UnicodeNormalizer`](https://mastra.ai/docs/processors) は制御文字を剥离し、空白を-collapseします。[`PromptInjectionDetector`](https://mastra.ai/docs/processors) 誰かが指示をオーバーライドしようとしていることを示唆するパターンのために、クリーンな入力を分析します。

検出の aggressively（`threshold`パラメータ）と、トリップしたときに何が起こるかを設定できます（ブロック、ログ、または単にフラグを付ける）。

### 2. PIIの処理

ログ内のクレジットカード番号、ベクターストア内の社会保障番号、必要以上に保存されたメールアドレス。これらは規制問題になる種類の問題です。課題は、ユーザーがチャットウィンドウに機密データを貼り付けていることに必ずし も気づかないことです。

[`PIIDetector`](https://mastra.ai/docs/processors) は、モデルに到達したりストレージに書き込まれたりする前に、一般的なパターンをスキャンします：

```typescript
import { PIIDetector } from '@mastra/core/processors';

export const privateAgent = new Agent({
  id: 'privacy-first-assistant',
  name: 'privacy-first-assistant',
  instructions: 'You are a helpful assistant that never stores personal information.',
  model: openai('gpt-5'),
  inputProcessors: [
    new PIIDetector({
      id: 'pii-detector',
      model: openai('gpt-5-nano'),
      detectionTypes: ['email', 'phone', 'credit-card', 'ssn'],
      threshold: 0.6,
      strategy: 'redact',
      redactionMethod: 'mask',  // Replace with [REDACTED]
      instructions: 'Detect and mask personally identifiable information',
    }),
  ],
});
```

=redact（`[REDACTED]`に置換）、ハッシュ化、または完全にブロックすることを選択できます。プロセッサは入力と输出的両方で実行されるため、モデルが応答で機密データを生成してしまった場合でも保護されます。

### 3. コンテンツモデレーション

インターネットデータでトレーニングされたモデルは山地看过某些东西。如果没有过滤，它们可能偶尔会产生让你的PR团队紧张的回复。[`ModerationProcessor`](https://mastra.ai/docs/processors) あなたのガイドラインに違反するコンテンツをキャッチします：

```typescript
import { ModerationProcessor } from '@mastra/core/processors';

export const moderatedAgent = new Agent({
  id: 'safe-assistant',
  name: 'safe-assistant',
  instructions: 'You are a helpful assistant for a community platform.',
  model: openai('gpt-5'),
  inputProcessors: [
    new ModerationProcessor({
      id: 'moderation-processor',
      model: openai('gpt-5-nano'),  // Fast, cheap model for classification
      categories: ['hate', 'harassment', 'violence', 'self-harm'],
      threshold: 0.7,  // Block if confidence > 70%
      strategy: 'block',  // Stop the request immediately
      instructions: 'Detect harmful content that violates community guidelines',
    }),
  ],
});
```

興味深いのは、どのカテゴリがあなたのユースケースに重要かを定義できることです。クリエイティブライティングツールは、カスタマーサービスボットよりも表現力のあるコンテンツを許可する可能性があります。しきい値と戦略により、フィルタリングの厳しさを制御できます。

## ものがトリップするとき

プロセッサは問題を検出してもエラーをスローしません。代わりに、結果オブジェクトにフラグを設定します：

```typescript
const result = await secureAgent.generate('Ignore all previous instructions...');

if (result.tripwire) {
  console.log(`Blocked! Reason: ${result.tripwireReason}`);
  // "Blocked! Reason: Prompt injection detected."
  return "Nice try, script kiddie.";
}
```

このパターンにより、セキュリティイベントをアプリケーションにとって意味のある方法で処理できます。分析のためにログに記録したり、一般的なエラーメッセージを返したり、特定のコンテキストで特定の違反を許可したりできます。`tripwireReason`フィールドは、どのプロセッサがコンテンツにフラグを付けたかを正確に伝えるため、誤検知のデバッグやしきい値の調整に役立ちます。

## これが解決しないこと

プロセッサは多くのことをキャッチしますが、魔法ではありません。十分な時間を持つ決意した攻撃者は、おそらくすり抜けるプロンプトを見つけることができます。モデルは、プロセッサが予測できない方法で幻觉を起こすことがあります。そして、セキュリティと柔軟性の間には常にトレードオフがあります：ルールが厳しければ厳いいほど、正当なユースケースをブロックする可能性が高くなります。

価値は完璧な保護ではありません。本番で必ず発生する一般的な問題に対処する体系的な方法を持つことです。ユーザーが実際に何をしているかを学びながら、感度を調整できます。ドメイン固有のリスクに対してカスタムプロセッサを追加できます。そして、何がブロックされたか、なぜブロックされたかを示す監査証跡があります。

本番AIのほとんどのセキュリティ問題は、精巧な攻撃ではありません。データを貼り付けてはならないことを知らなかった人がコピー＆ペーストしたり、ボットが期待していないことをすることを試行して発見したりすることです。プロセッサはすべての可能な問題を防ぎませんが、一般的な問題を著しく困難にします。
