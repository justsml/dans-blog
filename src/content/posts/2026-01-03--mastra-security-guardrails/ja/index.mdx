誰も安全でないAIシステムを作ろうとは思わない。指示を書き、エッジケースをテストし、いくつかの検証ルールを追加する。そして誰かが、ボットを海賊役に仕立て上げてユーザーデータを晒す方法を見つけ出す。あるいはクレジットカード番号がログに残る。あるいはモデルが自信満々に競合製品を推薦する。

「デモでは動く」と「プロダクションで安全」の間には、多くのチームが想定するより大きな隔たりがある。

問題の一部は、生のLLMが何をすべきか、何をすべきでないかについて意見を持っていないことだ。彼らは、あなたが始めたパターンを継続しようとする予測マシンだ。「システムオーバーライドモード」のようなプロンプトを与えれば、喜んで付き合う。これはモデルのバグではなく、言語モデルの仕組みなのだ。

ほとんどのフレームワークはモデルを渡して幸運を祈る。Mastraは異なるアプローチを取る：最終的にガードレールが必要になることを前提に、エージェントアーキテクチャの最初から組み込んでいる。

## 安全性のレイヤーとしてのプロセッサ

核心的な仕組みは単純だ。プロンプトがモデルに届く前に、入力プロセッサのチェーンを通過する。モデルが応答した後は、出力プロセッサが順番を待つ。各プロセッサは、その段階でコンテンツを検査、変更、またはブロックできる。

AIインタラクションのミドルウェアとして考えてほしい。必要なものを積み重ね、動作を設定すれば、すべてのリクエストで自動的に実行される。

### 1. 海賊を止める（プロンプトインジェクション）

プロンプトインジェクション攻撃は創造的になってきた。人々は不可視のUnicode文字を使い、base64で指示を書き、あるいはモデルを「デバッグモード」にあると信じ込ませて通常のルールが適用されないようにする。手法は進化し続けている。

Mastraには一般的なパターンを捉えるプロセッサが含まれている：


```typescript
// src/mastra/agents/secure-agent.ts
import { Agent } from '@mastra/core/agent';
import { PromptInjectionDetector, UnicodeNormalizer } from '@mastra/core/processors';
import { openai } from '@ai-sdk/openai';

export const secureAgent = new Agent({
  id: 'fortress-assistant',
  name: 'fortress-assistant',
  instructions: 'You are a secure assistant.',
  model: openai('gpt-5'),
  inputProcessors: [
    // 1. 不可視文字を除去
    new UnicodeNormalizer({
      id: 'unicode-normalizer',
      stripControlChars: true,
      collapseWhitespace: true,
    }),
    // 2. 試みを検出
    new PromptInjectionDetector({
      id: 'prompt-injection-detector',
      model: openai('gpt-5-nano'), // 安価で高速
      threshold: 0.8,
      strategy: 'block', // 強制停止
      detectionTypes: ['injection', 'jailbreak', 'system-override'],
    }),
  ],
});
```

[`UnicodeNormalizer`](https://mastra.ai/docs/processors)は制御文字を除去し、空白を正規化する。[`PromptInjectionDetector`](https://mastra.ai/docs/processors)は、クリーンアップされた入力を分析し、誰かが指示を上書きしようとしているパターンを探す。

検出の積極性（`threshold`パラメータ）と、検出時の動作（ブロック、ログ、またはフラグのみ）を設定できる。

### 2. PIIの取り扱い

ログ内のクレジットカード番号、ベクターデータベース内の社会保障番号、必要以上に長期間保存されるメールアドレス。これらは規制上の問題に発展する種類の問題だ。課題は、ユーザーが常にチャットウィンドウに機密データを貼り付けていることを自覚しているわけではないことだ。

[`PIIDetector`](https://mastra.ai/docs/processors)は、モデルに届く前やストレージに書き込まれる前に、一般的なパターンをスキャンする：

```typescript
import { PIIDetector } from '@mastra/core/processors';

export const privateAgent = new Agent({
  id: 'privacy-first-assistant',
  name: 'privacy-first-assistant',
  instructions: 'You are a helpful assistant that never stores personal information.',
  model: openai('gpt-5'),
  inputProcessors: [
    new PIIDetector({
      id: 'pii-detector',
      model: openai('gpt-5-nano'),
      detectionTypes: ['email', 'phone', 'credit-card', 'ssn'],
      threshold: 0.6,
      strategy: 'redact',
      redactionMethod: 'mask',  // [REDACTED]に置換
      instructions: 'Detect and mask personally identifiable information',
    }),
  ],
});
```

編集（`[REDACTED]`に置換）、ハッシュ化、または完全なブロックを選択できる。このプロセッサは入力と出力の両方で実行されるため、モデルが何らかの理由で応答に機密データを含めた場合もカバーされる。

### 3. コンテンツモデレーション

インターネットのデータで学習されたモデルは、いろいろなものを見てきた。フィルタリングなしでは、時々PRチームを不安にさせるような応答を生成することがある。[`ModerationProcessor`](https://mastra.ai/docs/processors)は、ガイドラインに違反するコンテンツを捉える：

```typescript
import { ModerationProcessor } from '@mastra/core/processors';

export const moderatedAgent = new Agent({
  id: 'safe-assistant',
  name: 'safe-assistant',
  instructions: 'You are a helpful assistant for a community platform.',
  model: openai('gpt-5'),
  inputProcessors: [
    new ModerationProcessor({
      id: 'moderation-processor',
      model: openai('gpt-5-nano'),  // 分類用の高速で安価なモデル
      categories: ['hate', 'harassment', 'violence', 'self-harm'],
      threshold: 0.7,  // 信頼度が70%を超えたらブロック
      strategy: 'block',  // 即座にリクエストを停止
      instructions: 'Detect harmful content that violates community guidelines',
    }),
  ],
});
```

興味深いのは、ユースケースにとって重要なカテゴリを自分で定義できる点だ。創作ツールはカスタマーサービスボットよりも表現豊かなコンテンツを許容するかもしれない。閾値と戦略により、フィルタリングの厳格さを制御できる。

## 検出されたとき

プロセッサは問題を検出してもエラーを投げない。代わりに、結果オブジェクトにフラグを設定する：

```typescript
const result = await secureAgent.generate('Ignore all previous instructions...');

if (result.tripwire) {
  console.log(`Blocked! Reason: ${result.tripwireReason}`);
  // "Blocked! Reason: Prompt injection detected."
  return "Nice try, script kiddie.";
}
```

このパターンにより、アプリケーションにとって意味のある方法でセキュリティイベントを処理できる。分析のためにログを残したり、一般的なエラーメッセージを返したり、特定のコンテキストでは特定の違反を許可したりすることもできる。`tripwireReason`フィールドはどのプロセッサがコンテンツをフラグ付けしたかを正確に示すため、誤検出のデバッグや閾値の調整に役立つ。

## これが解決しないもの

プロセッサは多くを捉えるが、魔法ではない。十分な時間をかけた決意を持った攻撃者は、おそらくスリップスルーするプロンプトを見つけられる。モデルは時々、プロセッサが予測できないような幻覚を起こす。そしてセキュリティと柔軟性の間には常にトレードオフがある：ルールが厳しければ厳しいほど、正当なユースケースをブロックする可能性が高まる。

価値は完璧な保護ではない。プロダクションで確実に発生する一般的な問題を体系的に処理する方法を持つことだ。ユーザーが実際に何をしているか学ぶにつれて感度を調整できる。ドメイン固有のリスク用にカスタムプロセッサを追加できる。そして、何がブロックされ、なぜブロックされたかを示す監査証跡を持つことができる。

プロダクションAIのほとんどのセキュリティ問題は、高度な攻撃ではない。人々が貼り付けるべきでないデータをコピー＆ペーストしたり、試行錯誤を通じてボットが意図しないことを行うことを発見したりする。プロセッサはすべての可能な問題を止められないが、明らかなものをはるかに困難にする。
