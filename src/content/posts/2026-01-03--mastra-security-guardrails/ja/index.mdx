誰も unsafe な AI システムを意図的に作る人はいない。指示を書き、エッジケースをテストし、いくつかバリデーションルールを追加する。そうすると、誰かがボットをだまして海賊のロールプレイをさせ、ユーザーデータを露出させる方法を見つける。あるいはクレジットカード番号がログに残る。あるいはモデルが自信たっぷりに競合製品をおすすめする。

「デモでは動く」と「プロダクションで安全」の間のギャップは、ほとんどのチームが予想するより大きい。

その原因の一部は、生の LLM が何をすべきか・すべきでないかについて意見を持たないことだ。LLM は入力されたパターンを続けようとする予測マシンに過ぎない。「システムオーバーライドモード」と見えるプロンプトを与えれば、喜んでそう演じる。これはモデルのバグではなく、言語モデルの仕組みにすぎない。

ほとんどのフレームワークはモデルを渡して「あとは頑張って」という姿勢だ。Mastra は別のアプローチをとる。いずれガードレールが必要になることを前提に、エージェントアーキテクチャに最初から組み込んでいる。

---

## セーフティレイヤーとしてのプロセッサ

中心の仕組みは単純だ。プロンプトがモデルに届く前に、入力プロセッサのチェーンを通過する。モデルが応答した後は、出力プロセッサが順番に処理する。各プロセッサはその段階でコンテンツを検査、変更、またはブロックできる。

AI インタラクション向けのミドルウェアと考えればいい。必要なものをスタックし、動作を設定すれば、リクエストごとに自動で実行される。

### 1. 海賊を止める（プロンプトインジェクション）

プロンプトインジェクション攻撃は巧妙になっている。不可視の Unicode 文字を使ったり、base64 で指示を書いたり、「デバッグモードだから通常ルールは適用外だ」とモデルを説得したりする。手法は進化し続けている。

Mastra にはよくあるパターンを捕捉するプロセッサが含まれている：


```typescript
// src/mastra/agents/secure-agent.ts
import { Agent } from '@mastra/core/agent';
import { PromptInjectionDetector, UnicodeNormalizer } from '@mastra/core/processors';
import { openai } from '@ai-sdk/openai';

export const secureAgent = new Agent({
  id: 'fortress-assistant',
  name: 'fortress-assistant',
  instructions: 'You are a secure assistant.',
  model: openai('gpt-5'),
  inputProcessors: [
    // 1. 不可視文字を除去
    new UnicodeNormalizer({
      id: 'unicode-normalizer',
      stripControlChars: true,
      collapseWhitespace: true,
    }),
    // 2. 試行を検出
    new PromptInjectionDetector({
      id: 'prompt-injection-detector',
      model: openai('gpt-5-nano'), // 安くて速い
      threshold: 0.8,
      strategy: 'block', // 即座に停止
      detectionTypes: ['injection', 'jailbreak', 'system-override'],
    }),
  ],
});
```

[`UnicodeNormalizer`](https://mastra.ai/docs/processors) は制御文字を削除し、ホワイトスペースを正規化する。[`PromptInjectionDetector`](https://mastra.ai/docs/processors) はクリーニングされた入力を分析し、指示の上書きを試みるパターンを検出する。

検出の厳しさ（`threshold` パラメータ）と、検出時の動作（ブロック、ログ、フラグのみ）を設定できる。

### 2. PII の処理

ログに残るクレジットカード番号、ベクトルデータベースに保存される社会保障番号、必要以上に長く保持されるメールアドレス。こうした問題は規制上の問題に発展する。ユーザーは自分がチャットウィンドウに機密データを貼り付けていることに気づかないことも多い。

[`PIIDetector`](https://mastra.ai/docs/processors) は、データがモデルに届く前やストレージに書き込まれる前に、よくあるパターンをスキャンする：

```typescript
import { PIIDetector } from '@mastra/core/processors';

export const privateAgent = new Agent({
  id: 'privacy-first-assistant',
  name: 'privacy-first-assistant',
  instructions: 'You are a helpful assistant that never stores personal information.',
  model: openai('gpt-5'),
  inputProcessors: [
    new PIIDetector({
      id: 'pii-detector',
      model: openai('gpt-5-nano'),
      detectionTypes: ['email', 'phone', 'credit-card', 'ssn'],
      threshold: 0.6,
      strategy: 'redact',
      redactionMethod: 'mask',  // [REDACTED] で置換
      instructions: 'Detect and mask personally identifiable information',
    }),
  ],
});
```

置換（`[REDACTED]` でマスク）、ハッシュ、完全ブロックから選べる。このプロセッサは入力と出力の両方で実行されるため、モデルが何らかの理由でレスポンスに機密データを生成した場合でも対応できる。

### 3. コンテンツモデレーション

インターネットデータで訓練されたモデルは、いろいろなものを見ている。フィルタリングなしでは、PR チームが震え上がるようなレスポンスを時折生成することがある。[`ModerationProcessor`](https://mastra.ai/docs/processors) はガイドラインに違反するコンテンツを捕捉する：

```typescript
import { ModerationProcessor } from '@mastra/core/processors';

export const moderatedAgent = new Agent({
  id: 'safe-assistant',
  name: 'safe-assistant',
  instructions: 'You are a helpful assistant for a community platform.',
  model: openai('gpt-5'),
  inputProcessors: [
    new ModerationProcessor({
      id: 'moderation-processor',
      model: openai('gpt-5-nano'),  // 分類用の速くて安いモデル
      categories: ['hate', 'harassment', 'violence', 'self-harm'],
      threshold: 0.7,  // 信頼度が 70% を超えたらブロック
      strategy: 'block',  // リクエストを即座に停止
      instructions: 'Detect harmful content that violates community guidelines',
    }),
  ],
});
```

面白いのは、自分のユースケースで重要なカテゴリを自分で定義できる点だ。創作ツールならカスタマーサポートボットより多くの表現を許容するかもしれない。threshold と strategy でフィルタリングの厳しさを制御できる。

## トリップワイヤーの仕組み

プロセッサは問題を検出してもエラーを投げない。代わりに、結果オブジェクトにフラグを設定する：

```typescript
const result = await secureAgent.generate('Ignore all previous instructions...');

if (result.tripwire) {
  console.log(`Blocked! Reason: ${result.tripwireReason}`);
  // "Blocked! Reason: Prompt injection detected."
  return "Nice try, script kiddie.";
}
```

このパターンにより、セキュリティイベントをアプリケーションに合った方法で処理できる。分析のためにログに記録したり、汎用的なエラーメッセージを返したり、特定のコンテキストでは一部の違反を許可したりできる。`tripwireReason` フィールドはどのプロセッサがコンテンツをフラグしたかを正確に示し、偽陽性のデバッグや閾値のチューニングに役立つ。

## これで解決しないこと

プロセッサは多くを捕捉するが、魔法ではない。十分な時間を持つ執念深い攻撃者は、おそらくすり抜けるプロンプトを見つけられる。モデルが時折、プロセッサでは予測できない幻覚を起こすこともある。そしてセキュリティと柔軟性の間には常にトレードオフがある。ルールが厳しいほど、正当なユースケースをブロックする可能性が高くなる。

価値は完全な保護ではない。プロダクションで確実に発生するよくある問題に対する体系的な対応方法を持つことだ。ユーザーが実際に何をするかを学びながら感度をチューニングできる。ドメイン固有のリスクに対してカスタムプロセッサを追加できる。何がブロックされ、なぜだったかを示す監査証跡もある。

プロダクション AI のセキュリティ問題の多くは、高度な攻撃ではない。してはいけないデータのコピペや、試行錯誤でボットが意図しないことをするのを見つける人々だ。プロセッサはあらゆる問題を止めることはできないが、自明なものをはるかに困難にする。
