Niemand nimmt sich vor, ein unsicheres AI-System zu bauen. Man schreibt Anweisungen, testet Randfälle, fügt ein paar Validierungsregeln hinzu. Dann findet jemand heraus, dass er deinen Bot dazu bringen kann, als Pirat zu rollenspielen und Benutzerdaten preiszugeben. Oder eine Kreditkartennummer landet in deinen Logs. Oder das Modell empfiehlt selbstbewusst ein Konkurrenzprodukt.

Die Lücke zwischen „funktioniert in der Demo" und „sicher in der Produktion" ist größer, als die meisten Teams erwarten.

Ein Teil des Problems ist, dass rohe LLMs keine Meinung dazu haben, was sie tun oder lassen sollten. Sie sind Vorhersagemaschinen, die versuchen, whatever Muster fortzusetzen, das du gestartet hast. Gib ihnen einen Prompt, der wie „System-Override-Modus" aussieht, und sie spielen bereitwillig mit. Das ist kein Bug im Modell; so funktionieren Sprachmodelle nun mal.

Die meisten Frameworks geben dir das Modell und wünschen dir Glück. Mastra geht einen anderen Weg: Es nimmt an, dass du irgendwann Guardrails brauchst, und baut sie von Anfang an in die Agent-Architektur ein.

---

## Prozessoren als Sicherheitsschichten

Der Kernmechanismus ist einfach. Bevor dein Prompt das Modell erreicht, durchläuft er eine Kette von Eingabeverarbeitern. Nachdem das Modell antwortet, kommen die Ausgabeverarbeiter an die Reihe. Jeder Prozessor kann den Inhalt zu diesem Zeitpunkt inspizieren, ändern oder blockieren.

Betrachte sie als Middleware für AI-Interaktionen. Du stapelst die, die du brauchst, konfigurierst ihr Verhalten und sie laufen automatisch bei jeder Anfrage.

### 1. Die Piraten stoppen (Prompt-Injection)

Prompt-Injection-Angriffe sind kreativ geworden. Leute verwenden unsichtbare Unicode-Zeichen, schreiben Anweisungen in Base64 oder überreden das Modell, es befinde sich im „Debug-Modus", in dem normale Regeln nicht gelten. Die Techniken entwickeln sich ständig weiter.

Mastra enthält Prozessoren, die häufige Muster erkennen:


```typescript
// src/mastra/agents/secure-agent.ts
import { Agent } from '@mastra/core/agent';
import { PromptInjectionDetector, UnicodeNormalizer } from '@mastra/core/processors';
import { openai } from '@ai-sdk/openai';

export const secureAgent = new Agent({
  id: 'fortress-assistant',
  name: 'fortress-assistant',
  instructions: 'You are a secure assistant.',
  model: openai('gpt-5'),
  inputProcessors: [
    // 1. Scrub invisible characters
    new UnicodeNormalizer({
      id: 'unicode-normalizer',
      stripControlChars: true,
      collapseWhitespace: true,
    }),
    // 2. Detect the attempt
    new PromptInjectionDetector({
      id: 'prompt-injection-detector',
      model: openai('gpt-5-nano'), // Cheap, fast
      threshold: 0.8,
      strategy: 'block', // Hard stop
      detectionTypes: ['injection', 'jailbreak', 'system-override'],
    }),
  ],
});
```

Der [`UnicodeNormalizer`](https://mastra.ai/docs/processors) entfernt Steuerzeichen und reduziert Whitespace. Der [`PromptInjectionDetector`](https://mastra.ai/docs/processors) analysiert die bereinigte Eingabe auf Muster, die darauf hindeuten, dass jemand versucht, deine Anweisungen zu überschreiben.

Du kannst konfigurieren, wie aggressiv die Erkennung sein soll (der `threshold`-Parameter) und was passieren soll, wenn er auslöst (blockieren, protokollieren oder nur markieren).

### 2. PII-Handling

Kreditkartennummern in Logs, Sozialversicherungsnummern in Vektordatenbanken, E-Mail-Adressen, die länger als nötig gespeichert werden. Das sind die Arten von Problemen, die zu regulatorischen Schwierigkeiten führen. Die Herausforderung ist, dass Benutzer nicht immer merken, dass sie sensible Daten in ein Chat-Fenster einfügen.

Der [`PIIDetector`](https://mastra.ai/docs/processors) scannt nach häufigen Mustern, bevor sie dein Modell erreichen oder in Speicher geschrieben werden:

```typescript
import { PIIDetector } from '@mastra/core/processors';

export const privateAgent = new Agent({
  id: 'privacy-first-assistant',
  name: 'privacy-first-assistant',
  instructions: 'You are a helpful assistant that never stores personal information.',
  model: openai('gpt-5'),
  inputProcessors: [
    new PIIDetector({
      id: 'pii-detector',
      model: openai('gpt-5-nano'),
      detectionTypes: ['email', 'phone', 'credit-card', 'ssn'],
      threshold: 0.6,
      strategy: 'redact',
      redactionMethod: 'mask',  // Replace with [REDACTED]
      instructions: 'Detect and mask personally identifiable information',
    }),
  ],
});
```

Du kannst wählen, ob du redigieren (durch `[REDACTED]` ersetzen), hashen oder komplett blockieren möchtest. Der Prozessor läuft sowohl bei der Eingabe als auch bei der Ausgabe, sodass du auch dann abgesichert bist, wenn das Modell somehow sensible Daten in seiner Antwort generiert.

### 3. Inhaltsmoderation

Modelle, die mit Internetdaten trainiert wurden, haben einiges gesehen. Ohne Filterung können sie gelegentlich Antworten produzieren, die dein PR-Team nervös machen würden. Der [`ModerationProcessor`](https://mastra.ai/docs/processors) erkennt Inhalte, die gegen deine Richtlinien verstoßen:

```typescript
import { ModerationProcessor } from '@mastra/core/processors';

export const moderatedAgent = new Agent({
  id: 'safe-assistant',
  name: 'safe-assistant',
  instructions: 'You are a helpful assistant for a community platform.',
  model: openai('gpt-5'),
  inputProcessors: [
    new ModerationProcessor({
      id: 'moderation-processor',
      model: openai('gpt-5-nano'),  // Fast, cheap model for classification
      categories: ['hate', 'harassment', 'violence', 'self-harm'],
      threshold: 0.7,  // Block if confidence > 70%
      strategy: 'block',  // Stop the request immediately
      instructions: 'Detect harmful content that violates community guidelines',
    }),
  ],
});
```

Der interessante Teil ist, dass du definierst, welche Kategorien für deinen Anwendungsfall relevant sind. Ein Tool zum kreativen Schreiben darf möglicherweise ausdrucksstärkere Inhalte zulassen als ein Kundenservice-Bot. Der Schwellenwert und die Strategie geben dir die Kontrolle darüber, wie streng die Filterung sein soll.

## Wenn etwas auslöst

Prozessoren werfen keine Fehler, wenn sie ein Problem erkennen. Stattdessen setzen sie ein Flag im Ergebnisobjekt:

```typescript
const result = await secureAgent.generate('Ignore all previous instructions...');

if (result.tripwire) {
  console.log(`Blocked! Reason: ${result.tripwireReason}`);
  // "Blocked! Reason: Prompt injection detected."
  return "Nice try, script kiddie.";
}
```

Dieses Muster lässt dich Sicherheitsereignisse so behandeln, wie es für deine Anwendung sinnvoll ist. Du könntest sie zur Analyse protokollieren, eine generische Fehlermeldung zurückgeben oder sogar bestimmte Verstöße in bestimmten Kontexten zulassen. Das `tripwireReason`-Feld sagt dir genau, welcher Prozessor den Inhalt markiert hat, was hilfreich ist, wenn du False Positives debuggst oder deine Schwellenwerte anpasst.

## Was das nicht löst

Prozessoren fangen viel ab, aber sie sind keine Magie. Ein entschlossener Angreifer mit genug Zeit kann wahrscheinlich einen Prompt finden, der durchrutscht. Modelle halluzinieren gelegentlich auf Weise, die Prozessoren nicht vorhersagen können. Und es gibt immer einen Kompromiss zwischen Sicherheit und Flexibilität: Je strenger deine Regeln, desto wahrscheinlicher blockierst du legitime Anwendungsfälle.

Der Wert liegt nicht in perfektem Schutz. Es ist eine systematische Methode, um die häufigen Probleme zu behandeln, die in der Produktion definitiv auftreten werden. Du kannst die Empfindlichkeit anpassen, während du lernst, was deine Benutzer tatsächlich tun. Du kannst benutzerdefinierte Prozessoren für domänenspezifische Risiken hinzufügen. Und du hast Audit-Trails, die zeigen, was blockiert wurde und warum.

Die meisten Sicherheitsprobleme in produktiver AI sind keine ausgeklügelten Angriffe. Es sind Leute, die Daten kopieren und einfügen, die sie nicht sollten, oder die durch Versuch und Irrtum entdecken, dass der Bot Dinge tut, die du nicht beabsichtigt hast. Prozessoren werden nicht jedes mögliche Problem stoppen, aber sie machen die offensichtlichen viel schwerer.
