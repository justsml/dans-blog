subTitle: "अगर आपके एजेंट के पास गार्डरेल्स नहीं हैं, तो आप प्रोडक्शन के लिए तैयार नहीं हैं।"
---

कोई भी असुरक्षित एआई सिस्टम बनाने का इरादा नहीं रखता है। आप निर्देश लिखते हैं, आप किनारे के मामलों (edge cases) का परीक्षण करते हैं, आप कुछ सत्यापन नियम जोड़ते हैं। फिर कोई यह पता लगा लेता है कि वे आपके बॉट को समुद्री डाकू (pirate) के रूप में भूमिका निभाने और उपयोगकर्ता डेटा को उजागर करने के लिए धोखा दे सकते हैं। या आपके लॉग में एक क्रेडिट कार्ड नंबर दिखाई देने लगता है। या मॉडल आत्मविश्वास से किसी प्रतियोगी के उत्पाद की सिफारिश कर देता है।

"डेमो में काम करता है" और "प्रोडक्शन में सुरक्षित" के बीच का अंतर अधिकांश टीमों की अपेक्षा से कहीं अधिक बड़ा है।

समस्या का एक हिस्सा यह है कि कच्चे एलएलएम (raw LLMs) की इस बारे में कोई राय नहीं होती है कि उन्हें क्या करना चाहिए या क्या नहीं। वे भविष्यवाणी करने वाली मशीनें (prediction machines) हैं जो आपके द्वारा शुरू किए गए किसी भी पैटर्न को जारी रखने की कोशिश करती हैं। उन्हें एक प्रॉम्प्ट दें जो "सिस्टम ओवरराइड मोड" जैसा दिखता हो, और वे खुशी-खुशी उसका साथ देंगे। यह मॉडल में कोई बग नहीं है; यह बस भाषा मॉडल के काम करने का तरीका है।

अधिकांश फ्रेमवर्क आपको मॉडल सौंप देते हैं और आपकी सफलता की कामना करते हैं। Mastra एक अलग दृष्टिकोण अपनाता है: यह मानता है कि आपको अंततः गार्डरेल्स की आवश्यकता होगी, इसलिए यह उन्हें शुरुआत से ही एजेंट आर्किटेक्चर में बनाता है।

## सुरक्षा परतों के रूप में प्रोसेसर्स (Processors as Safety Layers)

मूल तंत्र सीधा है। इससे पहले कि आपका प्रॉम्प्ट मॉडल तक पहुँचे, वह इनपुट प्रोसेसर्स की एक श्रृंखला से होकर गुजरता है। मॉडल के जवाब देने के बाद, आउटपुट प्रोसेसर्स की बारी आती है। प्रत्येक प्रोसेसर उस स्तर पर सामग्री का निरीक्षण, संशोधन या उसे ब्लॉक कर सकता है।

उन्हें एआई इंटरैक्शन के लिए मिडलवेयर (middleware) के रूप में सोचें। आप अपनी ज़रूरत के प्रोसेसर्स को स्टैक करते हैं, उनके व्यवहार को कॉन्फ़िगर करते हैं, और वे हर अनुरोध पर स्वचालित रूप से चलते हैं।

### 1. पायरेट्स को रोकना (प्रॉम्प्ट इंजेक्शन)

प्रॉम्प्ट इंजेक्शन हमले रचनात्मक हो गए हैं। लोग अदृश्य यूनिकोड वर्णों का उपयोग करते हैं, बेस64 में निर्देश लिखते हैं, या मॉडल को विश्वास दिलाते हैं कि वे "डीबग मोड" में हैं जहाँ सामान्य नियम लागू नहीं होते हैं। तकनीकें विकसित होती रहती हैं।

Mastra में ऐसे प्रोसेसर्स शामिल हैं जो सामान्य पैटर्न को पकड़ते हैं:

```typescript
// src/mastra/agents/secure-agent.ts
import { Agent } from '@mastra/core/agent';
import { PromptInjectionDetector, UnicodeNormalizer } from '@mastra/core/processors';
import { openai } from '@ai-sdk/openai';

export const secureAgent = new Agent({
  id: 'fortress-assistant',
  name: 'fortress-assistant',
  instructions: 'You are a secure assistant.',
  model: openai('gpt-5'),
  inputProcessors: [
    // 1. अदृश्य वर्णों को साफ करें
    new UnicodeNormalizer({
      id: 'unicode-normalizer',
      stripControlChars: true,
      collapseWhitespace: true,
    }),
    // 2. प्रयास का पता लगाएं
    new PromptInjectionDetector({
      id: 'prompt-injection-detector',
      model: openai('gpt-5-nano'), // सस्ता और तेज़
      threshold: 0.8,
      strategy: 'block', // पूरी तरह से रोकना
      detectionTypes: ['injection', 'jailbreak', 'system-override'],
    }),
  ],
});
```

[`UnicodeNormalizer`](https://mastra.ai/docs/processors) अदृश्य नियंत्रण वर्णों (control characters) को हटा देता है और अतिरिक्त रिक्त स्थान को कम कर देता है। [`PromptInjectionDetector`](https://mastra.ai/docs/processors) साफ किए गए इनपुट का उन पैटर्नों के लिए विश्लेषण करता है जो बताते हैं कि कोई आपके निर्देशों को ओवरराइड करने की कोशिश कर रहा है।

आप कॉन्फ़िगर करते हैं कि आप डिटेक्शन को कितना आक्रामक बनाना चाहते हैं (`threshold` पैरामीटर) और उल्लंघन होने पर क्या होना चाहिए (ब्लॉक करना, लॉग करना, या सिर्फ उसे फ्लैग करना)।

### 2. PII को संभालना

लॉग में क्रेडिट कार्ड नंबर, वेक्टर डेटाबेस में सोशल सिक्योरिटी नंबर, आवश्यकता से अधिक समय तक संग्रहीत ईमेल पते। ये उस प्रकार की समस्याएं हैं जो नियामक समस्याओं (regulatory problems) में बदल जाती हैं। चुनौती यह है कि उपयोगकर्ताओं को हमेशा यह एहसास नहीं होता है कि वे चैट विंडो में संवेदनशील डेटा पेस्ट कर रहे हैं।

[`PIIDetector`](https://mastra.ai/docs/processors) सामान्य पैटर्नों के लिए स्कैन करता है इससे पहले कि वे आपके मॉडल तक पहुँचें या स्टोरेज में लिखे जाएँ:

```typescript
import { PIIDetector } from '@mastra/core/processors';

export const privateAgent = new Agent({
  id: 'privacy-first-assistant',
  name: 'privacy-first-assistant',
  instructions: 'You are a helpful assistant that never stores personal information.',
  model: openai('gpt-5'),
  inputProcessors: [
    new PIIDetector({
      id: 'pii-detector',
      model: openai('gpt-5-nano'),
      detectionTypes: ['email', 'phone', 'credit-card', 'ssn'],
      threshold: 0.6,
      strategy: 'redact',
      redactionMethod: 'mask',  // [REDACTED] से बदलें
      instructions: 'व्यक्तिगत रूप से पहचान योग्य जानकारी का पता लगाएं और उसे मास्क करें',
    }),
  ],
});
```

आप रीडैक्ट (redact) करने (`[REDACTED]` के साथ बदलना), हैश (hash) करने या पूरी तरह से ब्लॉक करने का विकल्प चुन सकते हैं। प्रोसेसर इनपुट और आउटपुट दोनों पर चलता है, इसलिए आप तब भी सुरक्षित रहते हैं जब मॉडल किसी तरह अपनी प्रतिक्रिया में संवेदनशील डेटा उत्पन्न कर देता है।

### 3. कंटेंट मॉडरेशन (Content Moderation)

इंटरनेट डेटा पर प्रशिक्षित मॉडल ने बहुत कुछ देखा है। फ़िल्टरिंग के बिना, वे कभी-कभी ऐसी प्रतिक्रियाएँ दे सकते हैं जो आपकी पीआर टीम को परेशान कर सकती हैं। [`ModerationProcessor`](https://mastra.ai/docs/processors) उस सामग्री को पकड़ता है जो आपके दिशानिर्देशों का उल्लंघन करती है:

```typescript
import { ModerationProcessor } from '@mastra/core/processors';

export const moderatedAgent = new Agent({
  id: 'safe-assistant',
  name: 'safe-assistant',
  instructions: 'You are a helpful assistant for a community platform.',
  model: openai('gpt-5'),
  inputProcessors: [
    new ModerationProcessor({
      id: 'moderation-processor',
      model: openai('gpt-5-nano'),  // वर्गीकरण के लिए तेज़, सस्ता मॉडल
      categories: ['hate', 'harassment', 'violence', 'self-harm'],
      threshold: 0.7,  // यदि कॉन्फिडेंस > 70% है तो ब्लॉक करें
      strategy: 'block',  // अनुरोध को तुरंत रोकें
      instructions: 'हानिकारक सामग्री का पता लगाएं जो सामुदायिक दिशानिर्देशों का उल्लंघन करती है',
    }),
  ],
});
```

दिलचस्प बात यह है कि आप यह परिभाषित करते हैं कि आपके उपयोग के मामले के लिए कौन सी श्रेणियां मायने रखती हैं। एक रचनात्मक लेखन उपकरण ग्राहक सेवा बॉट की तुलना में अधिक अभिव्यंजक सामग्री की अनुमति दे सकता है। थ्रेशोल्ड और रणनीति आपको इस पर नियंत्रण देते हैं कि फ़िल्टरिंग कितनी सख्त होनी चाहिए।

## जब चीजें ट्रिप होती हैं (When Things Trip)

प्रोसेसर किसी समस्या का पता चलने पर त्रुटियां (errors) नहीं फेंकते हैं। इसके बजाय, वे परिणाम ऑब्जेक्ट पर एक फ्लैग सेट करते हैं:

```typescript
const result = await secureAgent.generate('Ignore all previous instructions...');

if (result.tripwire) {
  console.log(`Blocked! Reason: ${result.tripwireReason}`);
  // "Blocked! Reason: Prompt injection detected."
  return "अच्छा प्रयास है, स्क्रिप्ट किड्डी (script kiddie)।";
}
```

यह पैटर्न आपको सुरक्षा घटनाओं को आपके एप्लिकेशन के लिए सही तरीके से संभालने की अनुमति देता है। आप उन्हें विश्लेषण के लिए लॉग कर सकते हैं, एक सामान्य त्रुटि संदेश लौटा सकते हैं, या विशिष्ट संदर्भों में कुछ उल्लंघनों की अनुमति भी दे सकते हैं। `tripwireReason` फ़ील्ड आपको बिल्कुल बताती है कि किस प्रोसेसर ने सामग्री को फ्लैग किया है, जो झूठी सकारात्मकता (false positives) को डीबग करने या आपके थ्रेशोल्ड को ट्यून करने में मदद करता है।

## यह क्या हल नहीं करता है (What This Doesn't Solve)

प्रोसेसर्स बहुत कुछ पकड़ लेते हैं, लेकिन वे जादू नहीं हैं। पर्याप्त समय वाला एक दृढ़ हमलावर शायद ऐसा प्रॉम्प्ट ढूंढ सकता है जो सुरक्षा घेरे से निकल जाए। मॉडल कभी-कभी ऐसे मतिभ्रम (hallucinations) करते हैं जिनकी भविष्यवाणी प्रोसेसर्स नहीं कर सकते। और सुरक्षा और लचीलेपन के बीच हमेशा एक समझौता होता है: आपके नियम जितने सख्त होंगे, वैध उपयोग के मामलों को ब्लॉक करने की संभावना उतनी ही अधिक होगी।

मूल्य पूर्ण सुरक्षा नहीं है। यह उन सामान्य मुद्दों को संभालने का एक व्यवस्थित तरीका है जो निश्चित रूप से प्रोडक्शन में आएंगे। जैसे-जैसे आप सीखते हैं कि आपके उपयोगकर्ता वास्तव में क्या करते हैं, आप संवेदनशीलता को ट्यून कर सकते हैं। आप डोमेन-विशिष्ट जोखिमों के लिए कस्टम प्रोसेसर्स जोड़ सकते हैं। और आपके पास ऑडिट ट्रेल्स हैं जो दिखाते हैं कि क्या ब्लॉक किया गया और क्यों।

प्रोडक्शन एआई में अधिकांश सुरक्षा समस्याएं परिष्कृत हमले नहीं होती हैं। वे लोग होते हैं जो डेटा को कॉपी और पेस्ट करते हैं जो उन्हें नहीं करना चाहिए, या परीक्षण और त्रुटि के माध्यम से पता लगाते हैं कि बॉट वह काम करेगा जो आपका इरादा नहीं था। प्रोसेसर्स हर संभव समस्या को नहीं रोकेंगे, लेकिन वे स्पष्ट समस्याओं को बहुत कठिन बना देते हैं।
