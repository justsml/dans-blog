कोई भी असुरक्षित AI सिस्टम बनाने की कोशिश नहीं करता। आप इंस्ट्रक्शंस लिखते हैं, एज केस टेस्ट करते हैं, कुछ वैलिडेशन रूल्स जोड़ते हैं। फिर किसी को पता चलता है कि वे आपके बॉट को पाइरेट रोलप्ले करने और यूज़र डेटा एक्सपोज़ करने के लिए धोखा दे सकते हैं। या आपके लॉग्स में क्रेडिट कार्ड नंबर आ जाता है। या मॉडल आत्मविश्वास से किसी प्रतिस्पर्धी के प्रोडक्ट की सिफारिश करता है।

"डेमो में काम करता है" और "प्रोडक्शन में सुरक्षित है" के बीच का अंतर ज्यादातर टीम्स की उम्मीद से ज़्यादा होता है।

समस्या का एक हिस्सा यह है कि रॉ LLMs की कोई राय नहीं होती कि उन्हें क्या करना चाहिए और क्या नहीं। वे प्रेडिक्शन मशीन्स हैं जो जो भी पैटर्न आपने शुरू किया है उसे जारी रखने की कोशिश करती हैं। उन्हें ऐसा प्रॉम्प्ट दें जो "सिस्टम ओवरराइड मोड" जैसा दिखता है, और वे खुशी-खुशी साथ खेलेंगे। यह मॉडल में कोई बग नहीं है; यह सिर्फ लैंग्वेज मॉडल्स का काम करने का तरीका है।

ज्यादातर फ्रेमवर्क्स आपको मॉडल देते हैं और शुभकामना देते हैं। Mastra एक अलग दृष्टिकोण अपनाता है: यह मान लेता है कि आपको कभी-न-कभी गार्डरेल्स की ज़रूरत होगी, इसलिए यह उन्हें शुरू से ही एजेंट आर्किटेक्चर में बना देता है।

## सुरक्षा परतों के रूप में प्रोसेसर्स

मुख्य मैकेनिज्म सीधा है। आपके प्रॉम्प्ट के मॉडल तक पहुँचने से पहले, यह इनपुट प्रोसेसर्स की एक चेन से गुज़रता है। मॉडल के जवाब देने के बाद, आउटपुट प्रोसेसर्स की बारी आती है। हर प्रोसेसर उस स्टेज पर कंटेंट को इंस्पेक्ट, मॉडिफाई या ब्लॉक कर सकता है।

इन्हें AI इंटरैक्शंस के लिए मिडलवेयर समझें। आप अपनी ज़रूरत के प्रोसेसर्स स्टैक करें, उनका व्यवहार कॉन्फ़िगर करें, और वे हर रिक्वेस्ट पर ऑटोमैटिक चलेंगे।

### 1. पाइरेट्स को रोकना (प्रॉम्प्ट इंजेक्शन)

प्रॉम्प्ट इंजेक्शन अटैक्स रचनात्मक हो गए हैं। लोग इनविज़िबल यूनिकोड कैरेक्टर्स इस्तेमाल करते हैं, इंस्ट्रक्शंस base64 में लिखते हैं, या मॉडल को यकीन दिलाते हैं कि वह "डीबग मोड" में है जहाँ सामान्य नियम लागू नहीं होते। तकनीकें लगातार विकसित हो रही हैं।

Mastra में ऐसे प्रोसेसर्स शामिल हैं जो सामान्य पैटर्न्स को पकड़ते हैं:


```typescript
// src/mastra/agents/secure-agent.ts
import { Agent } from '@mastra/core/agent';
import { PromptInjectionDetector, UnicodeNormalizer } from '@mastra/core/processors';
import { openai } from '@ai-sdk/openai';

export const secureAgent = new Agent({
  id: 'fortress-assistant',
  name: 'fortress-assistant',
  instructions: 'You are a secure assistant.',
  model: openai('gpt-5'),
  inputProcessors: [
    // 1. Scrub invisible characters
    new UnicodeNormalizer({
      id: 'unicode-normalizer',
      stripControlChars: true,
      collapseWhitespace: true,
    }),
    // 2. Detect the attempt
    new PromptInjectionDetector({
      id: 'prompt-injection-detector',
      model: openai('gpt-5-nano'), // Cheap, fast
      threshold: 0.8,
      strategy: 'block', // Hard stop
      detectionTypes: ['injection', 'jailbreak', 'system-override'],
    }),
  ],
});
```

[`UnicodeNormalizer`](https://mastra.ai/docs/processors) कंट्रोल कैरेक्टर्स हटा देता है और व्हाइटस्पेस कोलैप्स करता है। [`PromptInjectionDetector`](https://mastra.ai/docs/processors) साफ़ किए गए इनपुट में ऐसे पैटर्न्स की जाँच करता है जो बताते हैं कि कोई आपके इंस्ट्रक्शंस को ओवरराइड करने की कोशिश कर रहा है।

आप कॉन्फ़िगर करते हैं कि डिटेक्शन कितना एग्रेसिव होना चाहिए (`threshold` पैरामीटर) और जब यह ट्रिगर हो तो क्या होना चाहिए (ब्लॉक, लॉग, या सिर्फ फ्लैग करें)।

### 2. PII संभालना

लॉग्स में क्रेडिट कार्ड नंबर, वेक्टर डेटाब्बेस में सोशल सिक्योरिटी नंबर, ज़रूरत से ज़्यादा समय तक स्टोर किए गए ईमेल पते। ये वे मुद्दे हैं जो रेगुलेटरी प्रॉब्लम्स बन जाते हैं। चुनौती यह है कि यूज़र्स को हमेशा पता नहीं चलता कि वे सेंसिटिव डेटा चैट विंडो में पेस्ट कर रहे हैं।

[`PIIDetector`](https://mastra.ai/docs/processors) सामान्य पैटर्न्स को स्कैन करता है इससे पहले कि वे आपके मॉडल तक पहुँचें या स्टोरेज में लिखे जाएँ:

```typescript
import { PIIDetector } from '@mastra/core/processors';

export const privateAgent = new Agent({
  id: 'privacy-first-assistant',
  name: 'privacy-first-assistant',
  instructions: 'You are a helpful assistant that never stores personal information.',
  model: openai('gpt-5'),
  inputProcessors: [
    new PIIDetector({
      id: 'pii-detector',
      model: openai('gpt-5-nano'),
      detectionTypes: ['email', 'phone', 'credit-card', 'ssn'],
      threshold: 0.6,
      strategy: 'redact',
      redactionMethod: 'mask',  // Replace with [REDACTED]
      instructions: 'Detect and mask personally identifiable information',
    }),
  ],
});
```

आप रिडैक्ट (`[REDACTED]` से बदलना), हैश, या पूरी तरह ब्लॉक करने में से चुन सकते हैं। प्रोसेसर इनपुट और आउटपुट दोनों पर चलता है, इसलिए अगर मॉडल किसी तरह अपने जवाब में सेंसिटिव डेटा जनरेट भी कर दे तो आप सुरक्षित हैं।

### 3. कंटेंट मॉडरेशन

इंटरनेट डेटा पर ट्रेंड मॉडल्स ने कुछ चीज़ें देखी हैं। बिना फ़िल्टरिंग के, वे कभी-कभी ऐसे जवाब दे सकते हैं जो आपकी PR टीम को घबरा दें। [`ModerationProcessor`](https://mastra.ai/docs/processors) ऐसा कंटेंट पकड़ता है जो आपके गाइडलाइंस का उल्लंघन करता है:

```typescript
import { ModerationProcessor } from '@mastra/core/processors';

export const moderatedAgent = new Agent({
  id: 'safe-assistant',
  name: 'safe-assistant',
  instructions: 'You are a helpful assistant for a community platform.',
  model: openai('gpt-5'),
  inputProcessors: [
    new ModerationProcessor({
      id: 'moderation-processor',
      model: openai('gpt-5-nano'),  // Fast, cheap model for classification
      categories: ['hate', 'harassment', 'violence', 'self-harm'],
      threshold: 0.7,  // Block if confidence > 70%
      strategy: 'block',  // Stop the request immediately
      instructions: 'Detect harmful content that violates community guidelines',
    }),
  ],
});
```

दिलचस्प बात यह है कि आप परिभाषित करते हैं कि आपके यूज़ केस के लिए कौन सी कैटेगरी मायने रखती है। एक क्रिएटिव राइटिंग टूल कस्टमर सर्विस बॉट से ज़्यादा एक्सप्रेसिव कंटेंट की अनुमति दे सकता है। थ्रेशहोल्ड और स्ट्रैटेजी आपको कंट्रोल देते हैं कि फ़िल्टरिंग कितनी सख्त होनी चाहिए।

## जब चीज़ें ट्रिगर होती हैं

प्रोसेसर्स किसी समस्या का पता लगाने पर एरर नहीं थ्रो करते। इसके बजाय, वे रिज़ल्ट ऑब्जेक्ट पर एक फ्लैग सेट करते हैं:

```typescript
const result = await secureAgent.generate('Ignore all previous instructions...');

if (result.tripwire) {
  console.log(`Blocked! Reason: ${result.tripwireReason}`);
  // "Blocked! Reason: Prompt injection detected."
  return "Nice try, script kiddie.";
}
```

यह पैटर्न आपको सिक्योरिटी इवेंट्स को अपने एप्लिकेशन के लिए जो तरीका सही हो उससे हैंडल करने देता है। आप उन्हें विश्लेषण के लिए लॉग कर सकते हैं, एक सामान्य एरर मैसेज रिटर्न कर सकते हैं, या विशिष्ट संदर्भों में कुछ उल्लंघनों की अनुमति भी दे सकते हैं। `tripwireReason` फ़ील्ड आपको बताता है कि किसने कंटेंट फ्लैग किया, जो फॉल्स पॉज़िटिव को डीबग करते या थ्रेशहोल्ड ट्यून करते समय मदद करता है।

## यह क्या हल नहीं करता

प्रोसेसर्स बहुत कुछ पकड़ते हैं, लेकिन वे जादू नहीं हैं। एक दृढ़ अटैकर पर्याप्त समय के साथ शायद ऐसा प्रॉम्प्ट ढूंढ सकता है जो निकल जाए। मॉडल कभी-कभी ऐसे तरीकों से हैल्यूसिनेट करते हैं जो प्रोसेसर्स प्रेडिक्ट नहीं कर सकते। और सिक्योरिटी और फ्लेक्सिबिलिटी के बीच हमेशा एक ट्रेडऑफ़ होता है: आपके नियम जितने सख्त होंगे, लीगिट यूज़ केस ब्लॉक होने की संभावना उतनी ज़्यादा होगी।

इसकी वैल्यू परफेक्ट प्रोटेक्शन नहीं है। यह सामान्य मुद्दों को सिस्टमैटिक तरीके से हैंडल करना है जो प्रोडक्शन में ज़रूर आएंगे। आप सेंसिटिविटी को ट्यून कर सकते हैं जैसे-जैसे आप सीखते हैं कि आपके यूज़र्स वास्तव में क्या करते हैं। आप डोमेन-विशिष्ट जोखिमों के लिए कस्टम प्रोसेसर्स जोड़ सकते हैं। और आपके पास ऑडिट ट्रेल्स हैं जो दिखाते हैं कि क्या ब्लॉक हुआ और क्यों।

प्रोडक्शन AI में ज्यादातर सिक्योरिटी प्रॉब्लम्स सोफिस्टिकेटेड अटैक्स नहीं हैं। वे ऐसे लोग हैं जो ऐसा डेटा कॉपी-पेस्ट कर रहे हैं जो नहीं करना चाहिए, या ट्रायल एंड एरर से खोज रहे हैं कि बॉट ऐसी चीज़ें कर देगा जो आपने नहीं चाही थीं। प्रोसेसर्स हर संभव समस्या को नहीं रोकेंगे, लेकिन वे सामान्य वालों को बहुत मुश्किल बना देंगे।
