Chat was the training wheels.

The first generation of LLM apps mostly looked like a text box stapled to a product. The model returned prose. The frontend rendered markdown. If the user needed to take action, the assistant described the button the user should go click somewhere else.

That was fine for demos. It is not where this is going.

The next useful step is **generative UI**: the model does not merely answer with text; it helps decide which interface the user needs right now. Sometimes that means calling a tool and rendering a prebuilt card. Sometimes it means filling a known workflow component with fresh data. Sometimes it means composing a temporary dashboard, form, comparison table, chart, or interactive widget.

Unfortunately, "generative UI" has become one of those phrases that means five different things before breakfast.

People use it to describe:

- a model choosing from developer-defined React components
- a JSON spec that the frontend renders into native components
- an iframe app returned from an MCP tool
- a chat UI library that supports tool calls
- an agent protocol that streams state between backend and frontend
- a design-time code generator like v0, Lovable, Bolt, or Cursor
- a model literally writing HTML, SVG, Canvas, or React at runtime

Those are related, but they are not the same layer. If you blur them together, every architecture conversation turns into soup.

This is the map I wish I had when I first started comparing the current stack.

## The Core Misunderstanding

The biggest mistake is treating "generative UI" as one technology choice.

It is better to separate the problem into four layers:

1.  **Product shell**: the thing users touch. This might be a chat, sidebar copilot, dashboard, workflow builder, IDE panel, ChatGPT app, mobile screen, or support console.
2.  **UI composition model**: the grammar the model is allowed to speak. It might be tool calls, JSON, A2UI, json-render, OpenUI Lang, Hashbrown component selection, or sandboxed HTML.
3.  **Runtime and transport**: how messages, tool calls, state deltas, user actions, and UI artifacts move between the agent and the frontend. AG-UI, MCP Apps, Apps SDK, A2A, SSE, WebSockets, and plain old HTTP all live around here.
4.  **Agent and tool backend**: LangGraph, Google ADK, CrewAI, Mastra, LlamaIndex, Pydantic AI, Agno, OpenAI Agents SDK, custom functions, databases, retrieval, and all the boring business logic that still has to be correct.

Once you split the layers, the ecosystem gets much less mystical.

[AG-UI](https://github.com/ag-ui-protocol/ag-ui) is not really a competitor to [A2UI](https://github.com/google/A2UI). AG-UI is an event protocol for agent-to-application interaction. A2UI is a declarative UI format the agent can send. You can put A2UI over AG-UI. You can also put custom tool-rendered components over AG-UI.

[json-render](https://github.com/vercel-labs/json-render) is not a chat product. It is a component catalog and renderer architecture: define the components the model may use, have the model emit a valid JSON tree, and render that tree safely.

[CopilotKit](https://github.com/CopilotKit/CopilotKit) is not just a chat bubble. It is a frontend stack for agent-native apps: chat UI, generative UI, shared state, frontend tools, and human-in-the-loop flows.

[OpenAI Apps SDK](https://developers.openai.com/apps-sdk) and [MCP Apps](https://github.com/MCP-UI-Org/mcp-ui) are not "make my React app dynamic" tools. They are host integration models for rendering widgets inside ChatGPT or other MCP-compatible hosts.

The names are confusing because the space is young. The layers are the part that stays useful.

## The Control Spectrum

Generative UI is a trade-off between **developer control** and **agent freedom**.

Too much control and the assistant feels like a command palette wearing a costume. Too much freedom and the model starts inventing weird layout, vague buttons, broken visual hierarchy, impossible states, and security problems with a confident little grin.

The trick is to choose the smallest amount of freedom that solves the user problem.

![A spectrum from tool-rendered components to open-ended generated HTML](./control-spectrum.webp)

I think about the spectrum like this:

**Tool-to-component rendering** is the safest default. The model calls `get_weather`, `search_products`, `compare_plans`, or `draft_invoice`. The app maps that tool result to a component you already own: `WeatherCard`, `ProductGrid`, `PlanComparison`, `InvoiceReview`. The model decides *when* the UI is useful. Developers still own layout, styling, accessibility, loading states, empty states, and dangerous actions.

This is the pattern documented in [Vercel AI SDK's generative UI guide](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces): the model calls a tool, the tool returns data, and the UI renders a component from the result. It is also the mental model behind many CopilotKit and assistant-ui implementations.

**Declarative component catalogs** give the model more room. Instead of choosing one component, the model composes a tree from allowed parts. A catalog might include `Metric`, `Table`, `Chart`, `FilterBar`, `ApprovalPanel`, and `Timeline`. The model can assemble a dashboard or workflow step, but it cannot execute arbitrary code. This is where [A2UI](https://github.com/google/A2UI), [json-render](https://github.com/vercel-labs/json-render), [Hashbrown](https://github.com/liveloveapp/hashbrown), and [OpenUI](https://github.com/thesysdev/openui) sit.

**Iframe mini-apps** make sense when the UI needs to be richer than a component tree, or when a remote tool provider owns the experience. MCP Apps and OpenAI Apps SDK let a tool return structured data plus a widget resource that the host renders in an iframe. This is powerful for maps, shopping carts, booking flows, charts, and external product surfaces. It also creates a harder boundary between the host app and the widget.

**Open-ended generation** is the far end: the agent emits HTML, SVG, Canvas, WebGL, or other code-like artifacts into a sandbox. [OpenGenerativeUI](https://github.com/CopilotKit/OpenGenerativeUI) is the best current example: the agent can generate algorithm visualizations, 3D scenes, diagrams, and simulations inside sandboxed iframes. This is great for one-off visual explanations. It is not where I would start for an enterprise approval flow.

It helps to name the key distinction here: **iframe HTML** (the model writes code into a sandbox) vs. **JSON catalog** (the model emits a structured spec and your renderer maps it to pre-built components). These sound related but carry very different risk and complexity profiles. Iframe HTML is maximally expressive; the iframe boundary does the security work. JSON catalog gives the model no executable freedom — it can only reference component types you defined in advance. Most of the frameworks in this space fall clearly into one or the other camp.

**Beyond the sandbox**: very recent demos suggest a fourth mode is forming — LLMs driving game-like or immersive experiences by controlling visual output more directly than any component catalog allows. Projects generating explorable 3D worlds from prompts, LLM-directed NPC behavior at runtime, and in-browser model inference via WebGPU ([WebLLM](https://mlc.ai/web-llm/)) are early markers. There are no stable frameworks to build production work with here yet. I'll cover this direction in a dedicated article once that changes.

## High-Level Components vs Granular Components

This is the most important design decision.

If your catalog is too granular, the model has to become a frontend engineer:

```tsx
Container
Row
Column
Text
Button
Icon
Spacer
Divider
```

That looks flexible, but now the model has to decide spacing, hierarchy, grouping, empty states, button labels, error treatment, and responsive behavior. You have also made the prompt bigger and the output easier to break.

If your catalog is too high-level, the model is trapped:

```tsx
WeatherCard
StockCard
HotelCard
```

That is safe, but it only works for known scenarios. The model cannot make a comparison matrix, ask for missing inputs, or adapt the information architecture when the user's question changes.

The useful middle is **domain-level components with constrained slots**:

```tsx
SearchResults
ComparisonTable
MetricGroup
EditablePlan
ApprovalRequest
Timeline
DataCollectionForm
CheckoutReview
```

These components encode product taste and business constraints. The model gets to decide *what should be shown*, but not every CSS decision.

For example, a travel agent does not need `div`, `span`, and `button`. It needs:

- `TripSummary`
- `FlightOptionList`
- `HotelComparison`
- `TravelerForm`
- `PolicyNotice`
- `BookingConfirmation`

A finance agent does not need a generic charting playground. It needs:

- `PortfolioSnapshot`
- `TransactionTable`
- `RiskBreakdown`
- `ScenarioComparison`
- `ApprovalGate`

The catalog should sound like your product, not like HTML.

## What To Use For Which Product

Here is the recommendation matrix I would actually use with a team.

**You are adding an assistant to an existing SaaS app.**

Start with tool-to-component rendering. Use [Vercel AI SDK UI](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces), [assistant-ui](https://github.com/assistant-ui/assistant-ui), or [CopilotKit](https://github.com/CopilotKit/CopilotKit), depending on how much agent state and frontend tool integration you need. Keep the catalog tiny at first. Render product components you already trust.

**You are building a serious in-app copilot that needs shared state.**

Look hard at CopilotKit plus AG-UI. The important feature is not "chat." It is shared state and bidirectional interaction: the agent can ask for input, render UI, update state, and pause for approval.

**You have remote agents that need to send UI across a boundary.**

Use A2UI or an A2UI-like declarative protocol. The whole point is that a remote agent can describe UI as data while the host keeps control of native rendering, security, and style. If you also need live agent/app interaction, run it over AG-UI or whatever transport your environment standardizes on.

**You are building inside ChatGPT or an MCP-compatible host.**

Use MCP Apps and the Apps SDK path. OpenAI's current docs recommend the MCP Apps `ui/*` bridge for new work while keeping `window.openai` as a compatibility layer and optional extension surface. Also copy their split between data tools and render tools: let the model fetch and reason over data before it chooses to render a widget.

**You want natural-language dashboards, reports, or forms in your own app.**

Try json-render, Hashbrown, or OpenUI. The key is the catalog. If you expose `LineChart`, `DataTable`, `MetricGroup`, `FilterControl`, and `InsightCallout`, the model can assemble useful reporting surfaces without getting near arbitrary code.

**You want educational, visual, or highly bespoke artifacts.**

Use an open-ended sandbox like OpenGenerativeUI. Let the model write SVG, Canvas, WebGL, or self-contained HTML, but treat the output like untrusted user content. Sandbox it, size it, strip permissions, and keep it away from privileged app state.

**You mainly need prettier streaming markdown with a few inline affordances.**

Do not overbuild. llm-ui or assistant-ui tool rendering might be enough.

## The Mistakes I Would Avoid

**Mistake 1: Letting the model write production React at runtime.**

There are exceptions, but for product UI this is usually the wrong default. Runtime code generation is hard to secure, hard to test, hard to theme, and hard to keep accessible. If the model can get the job done by choosing from trusted components, do that.

**Mistake 2: Exposing design primitives instead of product primitives.**

When you give the model `Row`, `Column`, `Text`, and `Button`, you are asking it to become your design system. It will become a mediocre one. Give it higher-level product nouns.

**Mistake 3: Thinking valid JSON means safe UI.**

A payload can pass schema validation and still be manipulative or dangerous. The label can say "View invoice" while the action archives the account. Treat UI specs as behavior, not decoration. They need policy tests, semantic checks, and human confirmation for consequential actions.

**Mistake 4: Putting business logic in render tools.**

Render tools should render. Data tools should fetch, compute, mutate, and validate. OpenAI's Apps SDK docs call out this split for a reason: if every data tool drags a widget along with it, the model loses room to reason before presenting.

**Mistake 5: Optimizing for novelty instead of task completion.**

The point is not to make every answer a snowflake interface. The point is to reduce friction. A stable, boring approval panel that saves the user four minutes is better than a dazzling generated dashboard that cannot be trusted twice.

## A Practical Architecture

If I were starting a new product today, I would use a staged approach:

1.  **Ship controlled tool UI first.** Map known tools to known components. Log every tool call, UI render, and user action.
2.  **Add a domain catalog.** Once patterns repeat, expose `ComparisonTable`, `DecisionPanel`, `DataCollectionForm`, `Timeline`, and other product-specific components.
3.  **Add transport standardization only when needed.** If you own both frontend and backend, plain streaming may be fine. If you have multiple agent frameworks, use AG-UI. If tools cross product boundaries, use MCP. If agents cross organizational boundaries, watch A2A and A2UI.
4.  **Use iframe widgets for foreign or complex surfaces.** Maps, carts, booking flows, and third-party mini-apps belong behind a boundary.
5.  **Reserve open-ended generation for artifacts.** Diagrams, simulations, temporary explainers, and visual scratchpads are a great fit. Core workflows are not.

The architecture ends up looking like this:

```txt
User intent
  -> agent runtime
  -> tool/data calls
  -> structured result
  -> UI decision
  -> trusted component, declarative spec, or sandboxed widget
  -> user action
  -> state/event stream back to the agent
```

That loop is the real product. The chat box is just one possible input device.

## Evaluation Should Include The UI

LLM teams are slowly learning to eval prompts and model outputs. Generative UI adds another surface area: the interface itself can be wrong.

At minimum, save these artifacts for every generated UI:

- prompt and tool context
- tool calls and tool results
- generated UI spec or component selection
- rendered component name and props
- user-visible labels
- actions attached to buttons/forms
- model-visible state updates from the UI
- user action history

Then write checks like:

- every destructive action must have a confirmation component
- button labels must match action semantics
- render specs may only reference allowed components
- user-visible totals must match tool-result totals
- forms may not request fields outside the task scope
- widgets may not receive secrets that only the model needed
- hidden metadata must not contradict visible labels

This sounds tedious. It is also where production trust comes from.

## A Note on Project Stability

Every major protocol in this space is pre-1.0. Last verified May 8, 2026; plan for change and check the current docs before making a platform bet.

**Vercel AI SDK RSC** — the original "Generative UI" headline feature — had development paused in October 2024 ([Discussion #3251](https://github.com/vercel/ai/discussions/3251)) due to architectural limitations that had no near-term fix. **json-render** (Vercel Labs) emerged as the replacement direction: catalog-based, framework-agnostic, no RSC coupling. It appears to have picked up web-developer attention quickly since its early-2026 launch. The likely reason is DX: json-render works immediately in a standard React project; A2UI's cross-platform scope adds setup friction.

**A2UI** (Google) is pre-1.0 with breaking changes between minor versions and inconsistent roadmap communications. Its advantage is genuine cross-platform reach (web, Flutter, SwiftUI) that json-render doesn't address. For pure-web use cases today, json-render appears to have better tooling coverage; for cross-platform or remote-agent scenarios, A2UI's design is more appropriate. Convergence between the two specs is possible — Vercel has experimented with A2UI-compatible output from json-render.

**AG-UI** (CopilotKit) is also pre-1.0. The most common confusion is the name: AG-UI is a transport protocol, not a UI framework. It defines *how* events flow between agent and frontend; what you render is still your call. The concept is solid and broadly adopted. The spec is still evolving.

## My Take

Generative UI will not replace carefully designed product interfaces. It will replace the lazy assumption that a chat transcript is the universal interface for AI.

The best systems will not let the model freestyle everything. They will give it a small, sharp set of product-native building blocks; a reliable runtime connection; clear security boundaries; and enough freedom to adapt the interface to the task.

The future is not "the model writes your frontend."

The future is closer to: **your frontend becomes an instrument the agent can play, but you still decide what the instrument is allowed to sound like.**
