La chat è stata solo la fase di rodaggio.

La prima generazione di app basate su LLM assomigliava per lo più a una casella di testo appiccicata a un prodotto. Il modello restituiva prosa. Il frontend renderizzava markdown. Se l'utente doveva compiere un'azione, l'assistente descriveva il pulsante che l'utente avrebbe dovuto cliccare da qualche altra parte.

Andava bene per le demo. Non è lì che stiamo andando.

Il prossimo passo utile è la **generative UI**: il modello non si limita a rispondere con del testo; aiuta a decidere di quale interfaccia l'utente ha bisogno in quel preciso momento. A volte ciò significa chiamare un tool e renderizzare una scheda predefinita. A volte significa riempire un componente di workflow noto con dati freschi. A volte significa comporre una dashboard temporanea, un modulo, una tabella di confronto, un grafico o un widget interattivo.

Sfortunatamente, "generative UI" è diventata una di quelle espressioni che significano cinque cose diverse prima di colazione.

Le persone la usano per descrivere:

- un modello che sceglie tra componenti React definiti dallo sviluppatore
- una specifica JSON che il frontend renderizza in componenti nativi
- un'app iframe restituita da un tool MCP
- una libreria di chat UI che supporta le chiamate ai tool
- un protocollo agente che trasmette lo stato tra backend e frontend
- un generatore di codice in fase di progettazione come v0, Lovable, Bolt o Cursor
- un modello che scrive letteralmente HTML, SVG, Canvas o React a runtime

Sono ambiti correlati, ma non sono lo stesso livello. Se li si confonde, ogni conversazione sull'architettura si trasforma in una zuppa.

Questa è la mappa che avrei voluto avere quando ho iniziato a confrontare lo stack attuale.

## Il Malinteso Principale

L'errore più grande è trattare la "generative UI" come una singola scelta tecnologica.

È meglio separare il problema in quattro livelli:

1.  **Guscio del prodotto (Product shell)**: ciò che l'utente tocca. Può essere una chat, un copilot laterale, una dashboard, un costruttore di workflow, un pannello IDE, un'app ChatGPT, uno schermo mobile o una console di supporto.
2.  **Modello di composizione UI**: la grammatica che il modello è autorizzato a parlare. Potrebbero essere chiamate a tool, JSON, A2UI, json-render, OpenUI Lang, la selezione dei componenti di Hashbrown o HTML in sandbox.
3.  **Runtime e trasporto**: come i messaggi, le chiamate ai tool, i delta di stato, le azioni dell'utente e gli artefatti UI si muovono tra l'agente e il frontend. AG-UI, MCP Apps, Apps SDK, A2A, SSE, WebSocket e il vecchio caro HTTP vivono tutti qui intorno.
4.  **Backend dell'agente e dei tool**: LangGraph, Google ADK, CrewAI, Mastra, LlamaIndex, Pydantic AI, Agno, OpenAI Agents SDK, funzioni personalizzate, database, retrieval e tutta la noiosa logica di business che deve comunque essere corretta.

Una volta divisi i livelli, l'ecosistema diventa molto meno mistico.

[AG-UI](https://github.com/ag-ui-protocol/ag-ui) non è propriamente un concorrente di [A2UI](https://github.com/google/A2UI). AG-UI è un protocollo di eventi per l'interazione agente-applicazione. A2UI è un formato UI dichiarativo che l'agente può inviare. Puoi usare A2UI sopra AG-UI. Puoi anche usare componenti personalizzati renderizzati dai tool sopra AG-UI.

[json-render](https://github.com/vercel-labs/json-render) non è un prodotto di chat. È un'architettura di catalogo componenti e renderer: definisci i componenti che il modello può usare, fai in modo che il modello emetta un albero JSON valido e renderizza quell'albero in modo sicuro.

[CopilotKit](https://github.com/CopilotKit/CopilotKit) non è solo una bolla di chat. È uno stack frontend per app native per agenti: chat UI, generative UI, stato condiviso, tool frontend e flussi human-in-the-loop.

[OpenAI Apps SDK](https://developers.openai.com/apps-sdk) e [MCP Apps](https://github.com/MCP-UI-Org/mcp-ui) non sono strumenti per "rendere dinamica la mia app React". Sono modelli di integrazione host per renderizzare widget all'interno di ChatGPT o altri host compatibili con MCP.

I nomi confondono perché lo spazio è giovane. I livelli sono la parte che rimane utile.

## Lo Spettro del Controllo

La Generative UI è un compromesso tra **controllo dello sviluppatore** e **libertà dell'agente**.

Troppo controllo e l'assistente sembra una tavolozza di comandi travestita. Troppa libertà e il modello inizia a inventare layout bizzarri, pulsanti vaghi, gerarchie visive interrotte, stati impossibili e problemi di sicurezza con un sorrisetto sicuro di sé.

Il trucco è scegliere la minima quantità di libertà che risolve il problema dell'utente.

![Uno spettro che va dai componenti renderizzati dai tool all'HTML generato aperto](../control-spectrum.webp)

Penso allo spettro in questo modo:

**Il rendering tool-to-component** è il default più sicuro. Il modello chiama `get_weather`, `search_products`, `compare_plans` o `draft_invoice`. L'app mappa il risultato di quel tool su un componente che già possiedi: `WeatherCard`, `ProductGrid`, `PlanComparison`, `InvoiceReview`. Il modello decide *quando* l'interfaccia è utile. Gli sviluppatori mantengono la proprietà di layout, stile, accessibilità, stati di caricamento, stati vuoti e azioni pericolose.

Questo è il pattern documentato nella [guida alla generative UI di Vercel AI SDK](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces): il modello chiama un tool, il tool restituisce dati e la UI renderizza un componente dal risultato. È anche il modello mentale dietro molte implementazioni di CopilotKit e assistant-ui.

**I cataloghi di componenti dichiarativi** danno più spazio al modello. Invece di scegliere un singolo componente, il modello compone un albero da parti autorizzate. Un catalogo potrebbe includere `Metric`, `Table`, `Chart`, `FilterBar`, `ApprovalPanel` e `Timeline`. Il modello può assemblare una dashboard o un passaggio di un workflow, ma non può eseguire codice arbitrario. È qui che si collocano [A2UI](https://github.com/google/A2UI), [json-render](https://github.com/vercel-labs/json-render), [Hashbrown](https://github.com/liveloveapp/hashbrown) e [OpenUI](https://github.com/thesysdev/openui).

**Le mini-app in iframe** hanno senso quando la UI deve essere più ricca di un albero di componenti, o quando un fornitore di tool remoto possiede l'esperienza. MCP Apps e OpenAI Apps SDK permettono a un tool di restituire dati strutturati più una risorsa widget che l'host renderizza in un iframe. Questo è potente per mappe, carrelli della spesa, flussi di prenotazione, grafici e superfici di prodotti esterni. Crea anche un confine più netto tra l'app host e il widget.

**La generazione aperta** è l'estremo opposto: l'agente emette HTML, SVG, Canvas, WebGL o altri artefatti simili a codice in una sandbox. [OpenGenerativeUI](https://github.com/CopilotKit/OpenGenerativeUI) è il miglior esempio attuale: l'agente può generare visualizzazioni di algoritmi, scene 3D, diagrammi e simulazioni all'interno di iframe in sandbox. È ottimo per spiegazioni visive una tantum. Non è il punto da cui partirei per un flusso di approvazione aziendale.

È utile dare un nome alla distinzione chiave qui: **iframe HTML** (il modello scrive codice in una sandbox) vs. **catalogo JSON** (il modello emette una specifica strutturata e il tuo renderer la mappa su componenti precostruiti). Sembrano correlati ma portano profili di rischio e complessità molto diversi. L'iframe HTML è massimamente espressivo; il confine dell'iframe fa il lavoro di sicurezza. Il catalogo JSON non dà al modello alcuna libertà di esecuzione: può solo fare riferimento a tipi di componenti definiti in precedenza. La maggior parte dei framework in questo spazio rientra chiaramente nell'uno o nell'altro campo.

**Oltre la sandbox**: demo molto recenti suggeriscono che si sta formando una quarta modalità: LLM che guidano esperienze simili a giochi o immersive controllando l'output visivo in modo più diretto di quanto consenta qualsiasi catalogo di componenti. Progetti che generano mondi 3D esplorabili da prompt, comportamenti di NPC diretti da LLM a runtime e inferenza del modello nel browser via WebGPU ([WebLLM](https://mlc.ai/web-llm/)) sono i primi segnali. Non ci sono ancora framework stabili per costruire lavori di produzione qui. Tratterò questa direzione in un articolo dedicato una volta che le cose cambieranno.

## Componenti ad Alto Livello vs Componenti Granulari

Questa è la decisione di progettazione più importante.

Se il tuo catalogo è troppo granulare, il modello deve diventare un ingegnere frontend:

```tsx
Container
Row
Column
Text
Button
Icon
Spacer
Divider
```

Sembra flessibile, ma ora il modello deve decidere spaziatura, gerarchia, raggruppamento, stati vuoti, etichette dei pulsanti, trattamento degli errori e comportamento responsive. Hai anche reso il prompt più grande e l'output più facile da rompere.

Se il tuo catalogo è ad un livello troppo alto, il modello è intrappolato:

```tsx
CustomerSalesChart
OrderTable
InventoryStatus
```

Questo è robusto ma rigido. Se l'utente chiede qualcosa di leggermente diverso, l'agente non può aiutarlo.

La via di mezzo è il **componente orientato al dominio**. Invece di `Row` e `Column`, dai al modello `MetricCard`, `ComparisonTable`, `StatusTimeline`, `ActionPrompt`. Questi portano con sé l'opinione del designer sulla gerarchia, lasciando che il modello possieda il raggruppamento dei dati e l'enfasi.

## Conclusione

La generative UI non serve a rendere la tua app "più intelligente". Serve a rimuovere l'attrito tra l'intento dell'utente e l'interfaccia.

Se stai costruendo un software per cui un set statico di dashboard e tabelle è sufficiente, non forzare la generative UI. È una scommessa sulla complessità che deve pagare i suoi dividendi.

Ma se stai costruendo per casi d'uso dove l'interfaccia ottimale è imprevedibile — strumenti di ricerca, editor creativi, analisi ad-hoc, assistenti operativi complessi — allora mappare questi livelli è l'unico modo per non perdersi.

Scegli il tuo livello di controllo. Definisci il tuo catalogo. E ricorda che l'iframe è il tuo migliore amico quando le cose si fanno selvagge.
