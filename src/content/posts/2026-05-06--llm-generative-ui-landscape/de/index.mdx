Chat war nur das Stützrad.

Die erste Generation von LLM-Apps sah meistens aus wie ein Textfeld, das an ein Produkt geheftet wurde. Das Modell lieferte Prosa. Das Frontend renderte Markdown. Wenn der Benutzer eine Aktion ausführen musste, beschrieb der Assistent den Button, den der Benutzer irgendwo anders anklicken sollte.

Das war okay für Demos. Aber das ist nicht das Ziel.

Der nächste nützliche Schritt ist **Generative UI**: Das Modell antwortet nicht nur mit Text; es hilft dabei zu entscheiden, welche Schnittstelle der Benutzer gerade jetzt braucht. Manchmal bedeutet das, ein Tool aufzurufen und eine vordefinierte Karte zu rendern. Manchmal bedeutet es, eine bekannte Workflow-Komponente mit frischen Daten zu füllen. Manchmal bedeutet es, ein temporäres Dashboard, ein Formular, eine Vergleichstabelle, ein Diagramm oder ein interaktives Widget zusammenzustellen.

Leider ist „Generative UI“ zu einer dieser Phrasen geworden, die schon vor dem Frühstück fünf verschiedene Dinge bedeuten.

Leute benutzen den Begriff, um Folgendes zu beschreiben:

- ein Modell, das aus entwicklerdefinierten React-Komponenten wählt
- eine JSON-Spezifikation, die das Frontend in native Komponenten rendert
- eine Iframe-App, die von einem MCP-Tool zurückgegeben wird
- eine Chat-UI-Bibliothek, die Tool-Aufrufe unterstützt
- ein Agenten-Protokoll, das den Status zwischen Backend und Frontend streamt
- einen Design-Time-Code-Generator wie v0, Lovable, Bolt oder Cursor
- ein Modell, das zur Laufzeit buchstäblich HTML, SVG, Canvas oder React schreibt

Diese Dinge sind verwandt, aber sie befinden sich nicht auf derselben Ebene. Wenn man sie vermischt, wird jede Architektur-Diskussion zu einem Brei.

Dies ist die Karte, die ich mir gewünscht hätte, als ich anfing, den aktuellen Stack zu vergleichen.

## Das grundlegende Missverständnis

Der größte Fehler ist es, „Generative UI“ als eine einzige Technologieentscheidung zu betrachten.

Es ist besser, das Problem in vier Ebenen zu unterteilen:

1.  **Product Shell**: Das, was der Benutzer berührt. Das kann ein Chat, ein Sidebar-Copilot, ein Dashboard, ein Workflow-Builder, ein IDE-Panel, eine ChatGPT-App, ein mobiler Bildschirm oder eine Support-Konsole sein.
2.  **UI-Kompositionsmodell**: Die Grammatik, die das Modell sprechen darf. Das können Tool-Aufrufe, JSON, A2UI, json-render, OpenUI Lang, Hashbrown-Komponentenauswahl oder sandboxed HTML sein.
3.  **Laufzeit und Transport**: Wie Nachrichten, Tool-Aufrufe, Status-Deltas, Benutzeraktionen und UI-Artefakte zwischen dem Agenten und dem Frontend bewegt werden. AG-UI, MCP-Apps, Apps SDK, A2A, SSE, WebSockets und ganz normales HTTP leben hier.
4.  **Agenten- und Tool-Backend**: LangGraph, Google ADK, CrewAI, Mastra, LlamaIndex, Pydantic AI, Agno, OpenAI Agents SDK, benutzerdefinierte Funktionen, Datenbanken, Retrieval und all die langweilige Business-Logik, die immer noch korrekt sein muss.

Sobald man die Ebenen trennt, wird das Ökosystem viel weniger mysteriös.

[AG-UI](https://github.com/ag-ui-protocol/ag-ui) ist kein wirklicher Konkurrent zu [A2UI](https://github.com/google/A2UI). AG-UI ist ein Event-Protokoll für die Interaktion zwischen Agent und Applikation. A2UI ist ein deklaratives UI-Format, das der Agent senden kann. Man kann A2UI über AG-UI nutzen. Man kann auch benutzerdefinierte, durch Tools gerenderte Komponenten über AG-UI nutzen.

[json-render](https://github.com/vercel-labs/json-render) ist kein Chat-Produkt. Es ist eine Komponenten-Katalog- und Renderer-Architektur: Definiere die Komponenten, die das Modell verwenden darf, lass das Modell einen validen JSON-Baum ausgeben und rendere diesen Baum sicher.

[CopilotKit](https://github.com/CopilotKit/CopilotKit) ist nicht nur eine Chat-Blase. Es ist ein Frontend-Stack für agenten-native Apps: Chat-UI, Generative UI, gemeinsamer Status, Frontend-Tools und Human-in-the-Loop-Flows.

[OpenAI Apps SDK](https://developers.openai.com/apps-sdk) und [MCP Apps](https://github.com/MCP-UI-Org/mcp-ui) sind keine Werkzeuge, um „meine React-App dynamisch zu machen“. Sie sind Host-Integrationsmodelle zum Rendern von Widgets innerhalb von ChatGPT oder anderen MCP-kompatiblen Hosts.

Die Namen sind verwirrend, weil der Bereich noch jung ist. Die Ebenen sind der Teil, der nützlich bleibt.

## Das Kontroll-Spektrum

Generative UI ist ein Kompromiss zwischen **Entwickler-Kontrolle** und **Agenten-Freiheit**.

Zu viel Kontrolle und der Assistent fühlt sich an wie eine Befehlspalette in einem Kostüm. Zu viel Freiheit und das Modell fängt an, seltsame Layouts, vage Buttons, kaputte visuelle Hierarchien, unmögliche Zustände und Sicherheitsprobleme mit einem selbstbewussten kleinen Grinsen zu erfinden.

Der Trick besteht darin, das kleinste Maß an Freiheit zu wählen, das das Benutzerproblem löst.

![Ein Spektrum von Tool-gerenderten Komponenten bis hin zu offen generiertem HTML](../control-spectrum.webp)

Ich stelle mir das Spektrum so vor:

**Tool-to-Component-Rendering** ist der sicherste Standard. Das Modell ruft `get_weather`, `search_products`, `compare_plans` oder `draft_invoice` auf. Die App ordnet dieses Tool-Ergebnis einer Komponente zu, die Sie bereits besitzen: `WeatherCard`, `ProductGrid`, `PlanComparison`, `InvoiceReview`. Das Modell entscheidet, *wann* die UI nützlich ist. Die Entwickler besitzen weiterhin Layout, Styling, Barrierefreiheit, Ladezustände, Leerzustände und gefährliche Aktionen.

Dies ist das Muster, das im [Generative UI Guide des Vercel AI SDK](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces) dokumentiert ist: Das Modell ruft ein Tool auf, das Tool gibt Daten zurück und die UI rendert eine Komponente aus dem Ergebnis. Es ist auch das mentale Modell hinter vielen CopilotKit- und assistant-ui-Implementierungen.

**Deklarative Komponenten-Kataloge** geben dem Modell mehr Raum. Anstatt eine Komponente zu wählen, stellt das Modell einen Baum aus erlaubten Teilen zusammen. Ein Katalog könnte `Metric`, `Table`, `Chart`, `FilterBar`, `ApprovalPanel` und `Timeline` enthalten. Das Modell kann ein Dashboard oder einen Workflow-Schritt zusammenstellen, aber es kann keinen beliebigen Code ausführen. Hier sitzen [A2UI](https://github.com/google/A2UI), [json-render](https://github.com/vercel-labs/json-render), [Hashbrown](https://github.com/liveloveapp/hashbrown) und [OpenUI](https://github.com/thesysdev/openui).

**Iframe-Mini-Apps** sind sinnvoll, wenn die UI komplexer sein muss als ein Komponentenbaum oder wenn ein entfernter Tool-Anbieter die Experience besitzt. MCP Apps und das OpenAI Apps SDK erlauben es einem Tool, strukturierte Daten plus eine Widget-Ressource zurückzugeben, die der Host in einem Iframe rendert. Dies ist mächtig für Karten, Warenkörbe, Buchungsabläufe, Diagramme und externe Produktoberflächen. Es schafft auch eine härtere Grenze zwischen der Host-App und dem Widget.

**Offene Generierung** ist das andere Ende: Der Agent gibt HTML, SVG, Canvas, WebGL oder andere codeähnliche Artefakte in eine Sandbox aus. [OpenGenerativeUI](https://github.com/CopilotKit/OpenGenerativeUI) ist das beste aktuelle Beispiel: Der Agent kann Algorithmus-Visualisierungen, 3D-Szenen, Diagramme und Simulationen in Sandboxed Iframes generieren. Dies ist großartig für einmalige visuelle Erklärungen. Es ist nichts, womit ich einen Enterprise-Genehmigungsprozess starten würde.

Es hilft, den entscheidenden Unterschied hier zu benennen: **Iframe HTML** (das Modell schreibt Code in eine Sandbox) vs. **JSON-Katalog** (das Modell gibt eine strukturierte Spezifikation aus und Ihr Renderer ordnet diese vorab erstellten Komponenten zu). Diese klingen verwandt, bringen aber sehr unterschiedliche Risiko- und Komplexitätsprofile mit sich. Iframe HTML ist maximal ausdrucksstark; die Iframe-Grenze erledigt die Sicherheitsarbeit. Der JSON-Katalog gibt dem Modell keine Freiheit zur Code-Ausführung – es kann nur auf Komponententypen verweisen, die Sie im Voraus definiert haben. Die meisten Frameworks in diesem Bereich fallen eindeutig in das eine oder das andere Lager.

**Jenseits der Sandbox**: Sehr junge Demos deuten darauf hin, dass sich ein vierter Modus bildet – LLMs, die spielähnliche oder immersive Erlebnisse steuern, indem sie den visuellen Output direkter kontrollieren, als es jeder Komponenten-Katalog erlaubt. Projekte, die explorierbare 3D-Welten aus Prompts generieren, LLM-gesteuertes NPC-Verhalten zur Laufzeit und In-Browser-Modell-Inferenz via WebGPU ([WebLLM](https://mlc.ai/web-llm/)) sind frühe Wegweiser. Es gibt hier noch keine stabilen Frameworks, um Produktionsarbeit zu leisten. Ich werde diese Richtung in einem eigenen Artikel behandeln, sobald sich das ändert.

## High-Level-Komponenten vs. granulare Komponenten

Dies ist die wichtigste Design-Entscheidung.

Wenn Ihr Katalog zu granular ist, muss das Modell zum Frontend-Entwickler werden:

```tsx
Container
Row
Column
Text
Button
Icon
Spacer
Divider
```

Das sieht flexibel aus, aber jetzt muss das Modell über Abstände, Hierarchien, Gruppierungen, Leerzustände, Button-Beschriftungen, Fehlerbehandlung und responsives Verhalten entscheiden. Sie haben außerdem den Prompt vergrößert und den Output fehleranfälliger gemacht.

Wenn Ihr Katalog zu abstrakt (High-Level) ist, ist das Modell gefangen:

```tsx
WeatherCard
StockCard
HotelCard
```

Das ist sicher, aber es funktioniert nur für bekannte Szenarien. Das Modell kann keine Vergleichsmatrix erstellen, nicht nach fehlenden Eingaben fragen oder die Informationsarchitektur anpassen, wenn sich die Frage des Benutzers ändert.

Die nützliche Mitte sind **Domain-Level-Komponenten mit eingeschränkten Slots**:

```tsx
SearchResults
ComparisonTable
MetricGroup
EditablePlan
ApprovalRequest
Timeline
DataCollectionForm
CheckoutReview
```

Diese Komponenten kodieren den Produktgeschmack und geschäftliche Einschränkungen. Das Modell darf entscheiden, *was gezeigt werden soll*, aber nicht über jede CSS-Entscheidung.

Ein Reisebüro-Assistent braucht zum Beispiel kein `div`, `span` und `button`. Er braucht:

- `TripSummary`
- `FlightOptionList`
- `HotelComparison`
- `TravelerForm`
- `PolicyNotice`
- `BookingConfirmation`

Ein Finanz-Assistent braucht keinen generischen Diagramm-Spielplatz. Er braucht:

- `PortfolioSnapshot`
- `TransactionTable`
- `RiskBreakdown`
- `ScenarioComparison`
- `ApprovalGate`

Der Katalog sollte nach Ihrem Produkt klingen, nicht nach HTML.

## Was man für welches Produkt verwenden sollte

Hier ist die Empfehlungsmatrix, die ich tatsächlich mit einem Team verwenden würde.

**Sie fügen einer bestehenden SaaS-App einen Assistenten hinzu.**

Beginnen Sie mit Tool-to-Component-Rendering. Verwenden Sie das [Vercel AI SDK UI](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces), [assistant-ui](https://github.com/assistant-ui/assistant-ui) oder [CopilotKit](https://github.com/CopilotKit/CopilotKit), je nachdem, wie viel Agenten-Status und Frontend-Tool-Integration Sie benötigen. Halten Sie den Katalog anfangs winzig. Rendern Sie Produktkomponenten, denen Sie bereits vertrauen.

**Sie bauen einen ernsthaften In-App-Copiloten, der gemeinsamen Status benötigt.**

Schauen Sie sich CopilotKit plus AG-UI genau an. Das wichtigste Feature ist nicht der „Chat“. Es ist der gemeinsame Status und die bidirektionale Interaktion: Der Agent kann nach Input fragen, UI rendern, den Status aktualisieren und auf Genehmigung warten.

**Sie haben Remote-Agenten, die UI über eine Grenze hinweg senden müssen.**

Verwenden Sie A2UI oder ein A2UI-ähnliches deklaratives Protokoll. Der ganze Punkt ist, dass ein entfernter Agent UI als Daten beschreiben kann, während der Host die Kontrolle über natives Rendering, Sicherheit und Stil behält. Wenn Sie auch eine Live-Interaktion zwischen Agent und App benötigen, lassen Sie diese über AG-UI oder den Transport laufen, der in Ihrer Umgebung Standard ist.

**Sie bauen innerhalb von ChatGPT oder einem MCP-kompatiblen Host.**

Nutzen Sie MCP Apps und den Apps SDK Pfad. Die aktuellen Dokumente von OpenAI empfehlen die MCP Apps `ui/*`-Bridge für neue Arbeiten, während `window.openai` als Kompatibilitätsschicht und optionale Erweiterungsfläche beibehalten wird. Kopieren Sie auch deren Trennung zwischen Daten-Tools und Rendering-Tools: Lassen Sie das Modell Daten abrufen und analysieren, bevor es sich entscheidet, ein Widget zu rendern.

**Sie möchten Dashboards, Berichte oder Formulare in natürlicher Sprache in Ihrer eigenen App.**

Probieren Sie json-render, Hashbrown oder OpenUI aus. Der Schlüssel ist der Katalog. Wenn Sie `LineChart`, `DataTable`, `MetricGroup`, `FilterControl` und `InsightCallout` exponieren, kann das Modell nützliche Berichterstellungs-Oberflächen zusammenstellen, ohne in die Nähe von beliebigem Code zu kommen.

**Sie möchten pädagogische, visuelle oder hochgradig maßgeschneiderte Artefakte.**

Verwenden Sie eine offene Sandbox wie OpenGenerativeUI. Lassen Sie das Modell SVG, Canvas, WebGL oder eigenständiges HTML schreiben, aber behandeln Sie den Output wie nicht vertrauenswürdigen User-Content. Setzen Sie ihn in eine Sandbox, begrenzen Sie die Größe, entziehen Sie Berechtigungen und halten Sie ihn von privilegiertem App-Status fern.

**Sie brauchen hauptsächlich hübscheres Streaming-Markdown mit ein paar Inline-Funktionen.**

Übertreiben Sie es nicht. llm-ui oder das Tool-Rendering von assistant-ui könnte ausreichen.

## Fehler, die ich vermeiden würde

**Fehler 1: Das Modell zur Laufzeit produktives React schreiben lassen.**

Es gibt Ausnahmen, aber für Produkt-UI ist dies meistens der falsche Standard. Code-Generierung zur Laufzeit ist schwer zu sichern, schwer zu testen, schwer zu thematisieren und schwer barrierefrei zu halten. Wenn das Modell den Job erledigen kann, indem es aus vertrauenswürdigen Komponenten wählt, tun Sie das.

**Fehler 2: Design-Primitive statt Produkt-Primitive exponieren.**

Wenn Sie dem Modell `Row`, `Column`, `Text` und `Button` geben, bitten Sie es, zu Ihrem Design-System zu werden. Es wird ein mittelmäßiges sein. Geben Sie ihm High-Level-Produktbegriffe.

**Fehler 3: Denken, dass valides JSON sichere UI bedeutet.**

Eine Payload kann die Schema-Validierung bestehen und trotzdem manipulativ oder gefährlich sein. Das Label kann „Rechnung ansehen“ sagen, während die Aktion das Konto archiviert. Behandeln Sie UI-Spezifikationen als Verhalten, nicht als Dekoration. Sie benötigen Policy-Tests, semantische Prüfungen und menschliche Bestätigung für folgenreiche Aktionen.

**Fehler 4: Business-Logik in Rendering-Tools packen.**

Rendering-Tools sollten rendern. Daten-Tools sollten abrufen, berechnen, mutieren und validieren. Die Dokumentation des Apps SDK von OpenAI weist aus gutem Grund auf diese Trennung hin: Wenn jedes Daten-Tool ein Widget mitschleppt, verliert das Modell den Raum zum Nachdenken vor der Präsentation.

**Fehler 5: Für Neuheit statt für Aufgabenerfüllung optimieren.**

Der Punkt ist nicht, aus jeder Antwort eine ausgefallene Schnittstelle zu machen. Der Punkt ist, Reibung zu reduzieren. Ein stabiles, langweiliges Genehmigungs-Panel, das dem Benutzer vier Minuten spart, ist besser als ein schillerndes generiertes Dashboard, dem man kein zweites Mal vertrauen kann.

## Eine praktische Architektur

Wenn ich heute ein neues Produkt starten würde, würde ich einen stufenweisen Ansatz wählen:

1.  **Zuerst kontrollierte Tool-UI ausliefern.** Bekannte Tools bekannten Komponenten zuordnen. Jeden Tool-Aufruf, jedes UI-Rendering und jede Benutzeraktion loggen.
2.  **Einen Domain-Katalog hinzufügen.** Sobald sich Muster wiederholen, `ComparisonTable`, `DecisionPanel`, `DataCollectionForm`, `Timeline` und andere produktspezifische Komponenten exponieren.
3.  **Transport-Standardisierung erst bei Bedarf hinzufügen.** Wenn Sie sowohl Frontend als auch Backend besitzen, kann einfaches Streaming ausreichen. Wenn Sie mehrere Agent-Frameworks haben, verwenden Sie AG-UI. Wenn Tools Produktgrenzen überschreiten, verwenden Sie MCP. Wenn Agenten Organisationsgrenzen überschreiten, beobachten Sie A2A und A2UI.
4.  **Iframe-Widgets für fremde oder komplexe Oberflächen verwenden.** Karten, Warenkörbe, Buchungsabläufe und Mini-Apps von Drittanbietern gehören hinter eine Grenze.
5.  **Offene Generierung für Artefakte reservieren.** Diagramme, Simulationen, temporäre Erklärungen und visuelle Notizblöcke passen hervorragend. Kern-Workflows nicht.

Die Architektur sieht am Ende so aus:

```txt
Benutzerabsicht
  -> Agent-Laufzeit
  -> Tool/Daten-Aufrufe
  -> strukturiertes Ergebnis
  -> UI-Entscheidung
  -> vertrauenswürdige Komponente, deklarative Spezifikation oder sandboxed Widget
  -> Benutzeraktion
  -> Status/Event-Stream zurück zum Agenten
```

Dieser Loop ist das eigentliche Produkt. Die Chatbox ist nur ein mögliches Eingabegerät.

## Evaluierung sollte die UI einschließen

LLM-Teams lernen langsam, Prompts und Modell-Outputs zu evaluieren. Generative UI fügt eine weitere Angriffsfläche hinzu: Das Interface selbst kann falsch sein.

Speichern Sie für jede generierte UI mindestens diese Artefakte:

- Prompt und Tool-Kontext
- Tool-Aufrufe und Tool-Ergebnisse
- generierte UI-Spezifikation oder Komponentenauswahl
- gerenderter Komponentenname und Props
- für den Benutzer sichtbare Labels
- Aktionen, die an Buttons/Formularen hängen
- für das Modell sichtbare Status-Updates aus der UI
- Verlauf der Benutzeraktionen

Schreiben Sie dann Prüfungen wie:

- jede destruktive Aktion muss eine Bestätigungskomponente haben
- Button-Labels müssen zur Semantik der Aktion passen
- Rendering-Spezifikationen dürfen nur erlaubte Komponenten referenzieren
- für den Benutzer sichtbare Summen müssen mit den Tool-Ergebnis-Summen übereinstimmen
- Formulare dürfen keine Felder außerhalb des Aufgabenbereichs anfordern
- Widgets dürfen keine Geheimnisse erhalten, die nur das Modell benötigt
- versteckte Metadaten dürfen sichtbaren Labels nicht widersprechen

Das klingt mühsam. Aber genau daraus entsteht Vertrauen in der Produktion.

## Ein Hinweis zur Projektstabilität

Jedes wichtige Protokoll in diesem Bereich ist pre-1.0. Zuletzt verifiziert am 8. Mai 2026; planen Sie Änderungen ein und prüfen Sie die aktuellen Dokumente, bevor Sie eine Plattform-Wette eingehen.

**Vercel AI SDK RSC** – das ursprüngliche „Generative UI“ Schlagzeilen-Feature – wurde im Oktober 2024 aufgrund architektonischer Einschränkungen pausiert ([Diskussion #3251](https://github.com/vercel/ai/discussions/3251)), für die es keine kurzfristige Lösung gab. **json-render** (Vercel Labs) entwickelte sich als Ersatzrichtung: katalogbasiert, framework-agnostisch, keine RSC-Kopplung. Es scheint seit seinem Start Anfang 2026 schnell die Aufmerksamkeit der Web-Entwickler gewonnen zu haben. Der wahrscheinliche Grund ist die DX: json-render funktioniert sofort in einem Standard-React-Projekt; der plattformübergreifende Umfang von A2UI sorgt für mehr Setup-Reibung.

**A2UI** (Google) ist pre-1.0 mit Breaking Changes zwischen Minor-Versionen und inkonsistenter Roadmap-Kommunikation. Sein Vorteil ist die echte plattformübergreifende Reichweite (Web, Flutter, SwiftUI), die json-render nicht adressiert. Für reine Web-Anwendungsfälle scheint json-render heute die bessere Tooling-Abdeckung zu haben; für plattformübergreifende oder Remote-Agenten-Szenarien ist das Design von A2UI angemessener. Eine Konvergenz zwischen den beiden Spezifikationen ist möglich – Vercel hat mit A2UI-kompatiblem Output aus json-render experimentiert.

**AG-UI** (CopilotKit) ist ebenfalls pre-1.0. Die häufigste Verwirrung ist der Name: AG-UI ist ein Transportprotokoll, kein UI-Framework. Es definiert, *wie* Events zwischen Agent und Frontend fließen; was Sie rendern, ist immer noch Ihre Entscheidung. Das Konzept ist solide und wird breit angenommen. Die Spezifikation entwickelt sich noch.

## Mein Fazit

Generative UI wird sorgfältig gestaltete Produktschnittstellen nicht ersetzen. Es wird die faule Annahme ersetzen, dass ein Chat-Transkript das universelle Interface für KI ist.

Die besten Systeme werden dem Modell nicht erlauben, alles frei zu gestalten. Sie werden ihm einen kleinen, scharfen Satz produktnativer Bausteine, eine zuverlässige Laufzeitverbindung, klare Sicherheitsgrenzen und genug Freiheit geben, um das Interface an die Aufgabe anzupassen.

Die Zukunft ist nicht: „Das Modell schreibt dein Frontend.“

Die Zukunft liegt näher an: **Dein Frontend wird zu einem Instrument, das der Agent spielen kann, aber du entscheidest immer noch, wie das Instrument klingen darf.**
