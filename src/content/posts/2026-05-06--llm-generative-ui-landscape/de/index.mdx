Chat war die Trainingshilfe.

Die erste Generation von LLM-Apps sah meist aus wie eine Textbox, die an ein Produkt geklebt wurde. Das Modell lieferte Prosa. Das Frontend renderete Markdown. Wenn der Nutzer eine Aktion ausführen musste, beschrieb der Assistent den Button, den der Nutzer irgendwo anders anklicken sollte.

Das war fine für Demos. Es ist nicht dorthin, wo das geht.

Der nächste nützliche Schritt ist **generative UI**: Das Modell antwortet nicht nur mit Text; es hilft dabei zu entscheiden, welche Oberfläche der Nutzer gerade braucht. Manchmal bedeutet das, ein Tool aufzurufen und eine vorgefertigte Karte zu rendern. Manchmal bedeutet das, eine bekannte Workflow-Komponente mit frischen Daten zu füllen. Manchmal bedeutet das, ein temporäres Dashboard, Formular, eine Vergleichstabelle, ein Diagramm oder ein interaktives Widget zusammenzustellen.

Leider ist „generative UI" zu einem dieser Begriffe geworden, die vor dem Frühstück fünf verschiedene Dinge bedeuten.

Leute verwenden ihn, um Folgendes zu beschreiben:

- ein Modell, das aus entwicklerdefinierten React-Komponenten wählt
- eine JSON-Spezifikation, die das Frontend in native Komponenten renderiert
- eine Iframe-App, die von einem MCP-Tool zurückgegeben wird
- eine Chat-UI-Bibliothek, die Tool-Aufrufe unterstützt
- ein Agentenprotokoll, das Zustände zwischen Backend und Frontend streamt
- ein Code-Generator zur Designzeit wie v0, Lovable, Bolt oder Cursor
- ein Modell, das buchstäblich HTML, SVG, Canvas oder React zur Laufzeit schreibt

Diese sind verwandt, aber sie sind nicht dieselbe Schicht. Wenn man sie verschwimmen lässt, wird jede Architekturkonversation zur Suppe.

Das ist die Karte, die ich mir gewünscht hätte, als ich anfing, den aktuellen Stack zu vergleichen.

## Das grundlegende Missverständnis

Der größte Fehler ist, „generative UI" als eine Technologieentscheidung zu behandeln.

Es ist besser, das Problem in vier Schichten zu trennen:

1.  **Produkt-Shell**: Das Ding, das die Nutzer berühren. Das könnte ein Chat, eine Sidebar-Copilot, ein Dashboard, ein Workflow-Builder, ein IDE-Panel, eine ChatGPT-App, ein mobiler Bildschirm oder eine Support-Konsole sein.
2.  **UI-Kompositionsmodell**: Die Grammatik, die das Modell sprechen darf. Es könnte Tool-Aufrufe, JSON, A2UI, json-render, OpenUI Lang, Hashbrown-Komponentenauswahl oder gesandboxtes HTML sein.
3.  **Laufzeit und Transport**: Wie Nachrichten, Tool-Aufrufe, Zustandsdeltas, Nutzeraktionen und UI-Artefakte zwischen dem Agenten und dem Frontend bewegt werden. AG-UI, MCP Apps, Apps SDK, A2A, SSE, WebSockets und plain old HTTP leben alle hierherum.
4.  **Agenten- und Tool-Backend**: LangGraph, Google ADK, CrewAI, Mastra, LlamaIndex, Pydantic AI, Agno, OpenAI Agents SDK, benutzerdefinierte Funktionen, Datenbanken, Retrieval und die ganze langweilige Geschäftslogik, die trotzdem korrekt sein muss.

Sobald man die Schichten auftrennt, wird das Ökosystem viel weniger mystisch.

[AG-UI](https://github.com/ag-ui-protocol/ag-ui) ist nicht wirklich ein Konkurrent von [A2UI](https://github.com/google/A2UI). AG-UI ist ein Ereignisprotokoll für die Agent-zu-Anwendung-Interaktion. A2UI ist ein deklaratives UI-Format, das der Agent senden kann. Man kann A2UI über AG-UI legen. Man kann auch benutzerdefinierte Tool-gerenderte Komponenten über AG-UI legen.

[json-render](https://github.com/vercel-labs/json-render) ist kein Chat-Produkt. Es ist eine Komponentenkatalog- und Renderer-Architektur: Definiere die Komponenten, die das Modell verwenden darf, lass das Modell einen gültigen JSON-Tree ausgeben und render diesen Tree sicher.

[CopilotKit](https://github.com/CopilotKit/CopilotKit) ist nicht nur eine Chat-Blase. Es ist ein Frontend-Stack für agentennative Apps: Chat-UI, generative UI, gemeinsamer Zustand, Frontend-Tools und Human-in-the-Loop-Flows.

[OpenAI Apps SDK](https://developers.openai.com/apps-sdk) und [MCP Apps](https://github.com/MCP-UI-Org/mcp-ui) sind keine „Mach meine React-App dynamisch"-Tools. Es sind Host-Integrationsmodelle für das Rendering von Widgets innerhalb von ChatGPT oder anderen MCP-kompatiblen Hosts.

Die Namen sind verwirrend, weil der Raum jung ist. Die Schichten sind der Teil, der nützlich bleibt.

## Das Kontroll-Spektrum

Generative UI ist ein Kompromiss zwischen **Entwicklerkontrolle** und **Agentenfreiheit**.

Zu viel Kontrolle und der Assistent fühlt sich an wie eine Befehlspalette, die ein Kostüm trägt. Zu viel Freiheit und das Modell fängt an, seltsames Layout, vage Buttons, kaputte visuelle Hierarchie, unmögliche Zustände und Sicherheitsprobleme mit einem selbstgefälligen kleinen Grinsen zu erfinden.

Der Trick ist, das kleinste Maß an Freiheit zu wählen, das das Nutzerproblem löst.

![Ein Spektrum von Tool-gerenderten Komponenten bis hin zu offen generiertem HTML](../control-spectrum.webp)

Ich denke über das Spektrum so nach:

**Tool-zu-Komponenten-Rendering** ist die sicherste Standardeinstellung. Das Modell ruft `get_weather`, `search_products`, `compare_plans` oder `draft_invoice` auf. Die App mappt dieses Tool-Ergebnis auf eine Komponente, die man bereits besitzt: `WeatherCard`, `ProductGrid`, `PlanComparison`, `InvoiceReview`. Das Modell entscheidet, *wann* die UI nützlich ist. Entwickler besitzen weiterhin Layout, Styling, Barrierefreiheit, Ladezustände, Leerzustände und gefährliche Aktionen.

Das ist das Muster, das im [Vercel AI SDK generative UI guide](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces) dokumentiert ist: Das Modell ruft ein Tool auf, das Tool gibt Daten zurück und die UI renderiert eine Komponente aus dem Ergebnis. Es ist auch das mentale Modell hinter vielen CopilotKit- und assistant-ui-Implementierungen.

**Deklarative Komponentenkataloge** geben dem Modell mehr Raum. Anstatt eine Komponente zu wählen, komponiert das Modell einen Tree aus erlaubten Teilen. Ein Katalog könnte `Metric`, `Table`, `Chart`, `FilterBar`, `ApprovalPanel` und `Timeline` enthalten. Das Modell kann ein Dashboard oder einen Workflow-Schritt zusammenstellen, aber es kann keinen beliebigen Code ausführen. Hier sitzen [A2UI](https://github.com/google/A2UI), [json-render](https://github.com/vercel-labs/json-render), [Hashbrown](https://github.com/liveloveapp/hashbrown) und [OpenUI](https://github.com/thesysdev/openui).

**Iframe-Mini-Apps** sind sinnvoll, wenn die UI reicher sein muss als ein Komponenten-Tree oder wenn ein entfernter Tool-Anbieter die Erfahrung besitzt. MCP Apps und OpenAI Apps SDK lassen ein Tool strukturierte Daten plus eine Widget-Ressource zurückgeben, die der Host in einem Iframe renderiert. Das ist mächtig für Karten, Warenkörbe, Buchungsflüsse, Diagramme und externe Produktoberflächen. Es schafft auch eine härtere Grenze zwischen der Host-App und dem Widget.

**Offene Generierung** ist das ferne Ende: Der Agent gibt HTML, SVG, Canvas, WebGL oder anderen codeähnlichen Artefakten in eine Sandbox aus. [OpenGenerativeUI](https://github.com/CopilotKit/OpenGenerativeUI) ist das beste aktuelle Beispiel: Der Agent kann Algorithmus-Visualisierungen, 3D-Szenen, Diagramme und Simulationen in gesandboxten Iframes generieren. Das ist großartig für einmalige visuelle Erklärungen. Es ist nicht dort, wo ich für einen Enterprise-Genehmigungsflow anfangen würde.

Es hilft, den wichtigsten Unterschied hier zu benennen: **Iframe-HTML** (das Modell schreibt Code in eine Sandbox) vs. **JSON-Katalog** (das Modell gibt eine strukturierte Spezifikation aus und dein Renderer mappt sie auf vorgebaute Komponenten). Diese klingen verwandt, tragen aber sehr unterschiedliche Risiko- und Komplexitätsprofile. Iframe-HTML ist maximal ausdrucksstark; die Iframe-Grenze leistet die Sicherheitsarbeit. JSON-Katalog gibt dem Modell keine ausführbare Freiheit — es kann nur Komponententypen referenzieren, die man im Voraus definiert hat. Die meisten Frameworks in diesem Raum fallen klar in das eine oder andere Lager.

**Jenseits der Sandbox**: Sehr aktuelle Demos deuten darauf hin, dass sich ein vierter Modus bildet — LLMs, die spielähnliche oder immersive Erfahrungen steuern, indem sie visuelle Ausgaben direkter kontrollieren, als es jeder Komponentenkatalog erlaubt. Projekte, die erforschbare 3D-Welten aus Prompts generieren, LLM-gesteuertes NPC-Verhalten zur Laufzeit und Inferenz von Modellen im Browser über WebGPU ([WebLLM](https://mlc.ai/web-llm/)) sind frühe Marker. Es gibt noch keine stabilen Frameworks, um hier Produktionsarbeit zu leisten. Ich werde diese Richtung in einem eigenen Artikel behandeln, sobald sich das ändert.

## Hochlevel-Komponenten vs. granulare Komponenten

Das ist die wichtigste Designentscheidung.

Wenn dein Katalog zu granular ist, muss das Modell zum Frontend-Ingenieur werden:

```tsx
Container
Row
Column
Text
Button
Icon
Spacer
Divider
```

Das sieht flexibel aus, aber jetzt muss das Modell Abstand, Hierarchie, Gruppierung, Leerzustände, Button-Labels, Fehlerbehandlung und responsives Verhalten entscheiden. Man hat auch den Prompt größer und die Ausgabe leichter kaputtzumachen gemacht.

Wenn dein Katalog zu hochlevelig ist, ist das Modell gefangen:

```tsx
WeatherCard
StockCard
HotelCard
```

Das ist sicher, aber es funktioniert nur für bekannte Szenarien. Das Modell kann keine Vergleichsmatrix erstellen, nach fehlenden Eingaben fragen oder die Informationsarchitektur anpassen, wenn sich die Frage des Nutzers ändert.

Die nützliche Mitte ist **domänenspezifische Komponenten mit eingeschränkten Slots**:

```tsx
SearchResults
ComparisonTable
MetricGroup
EditablePlan
ApprovalRequest
Timeline
DataCollectionForm
CheckoutReview
```

Diese Komponenten kodieren Produktgeschmack und Geschäftseinschränkungen. Das Modell darf entscheiden, *was gezeigt werden soll*, aber nicht jede CSS-Entscheidung.

Zum Beispiel braucht ein Reiseassistent kein `div`, `span` und `button`. Er braucht:

- `TripSummary`
- `FlightOptionList`
- `HotelComparison`
- `TravelerForm`
- `PolicyNotice`
- `BookingConfirmation`

Ein Finanzassistent braucht keinen generischen Charting-Spielplatz. Er braucht:

- `PortfolioSnapshot`
- `TransactionTable`
- `RiskBreakdown`
- `ScenarioComparison`
- `ApprovalGate`

Der Katalog sollte nach deinem Produkt klingen, nicht nach HTML.

## Was für welches Produkt verwenden

Hier ist die Empfehlungsmatrix, die ich tatsächlich mit einem Team verwenden würde.

**Du fügst einem bestehenden SaaS-App einen Assistenten hinzu.**

Beginne mit Tool-zu-Komponenten-Rendering. Verwende [Vercel AI SDK UI](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces), [assistant-ui](https://github.com/assistant-ui/assistant-ui) oder [CopilotKit](https://github.com/CopilotKit/CopilotKit), je nachdem, wie viel Agentenzustand und Frontend-Tool-Integration du brauchst. Halte den Katalog am Anfang winzig. Rendere Produktkomponenten, denen du bereits vertraust.

**Du baust einen ernsthaften In-App-Copilot, der gemeinsamen Zustand braucht.**

Schau dir CopilotKit plus AG-UI genau an. Das wichtige Feature ist nicht „Chat". Es ist gemeinsamer Zustand und bidirektionale Interaktion: Der Agent kann nach Eingabe fragen, UI rendern, Zustand aktualisieren und auf Genehmigung pausieren.

**Du hast Remote-Agenten, die UI über eine Grenze senden müssen.**

Verwende A2UI oder ein A2UI-ähnliches deklaratives Protokoll. Der ganze Punkt ist, dass ein Remote-Agent UI als Daten beschreiben kann, während der Host die Kontrolle über natives Rendering, Sicherheit und Stil behält. Wenn du auch live Agent/App-Interaktion brauchst, führe es über AG-UI oder whatever Transport deine Umgebung standardisiert.

**Du baust innerhalb von ChatGPT oder einem MCP-kompatiblen Host.**

Verwende den MCP Apps- und Apps SDK-Pfad. OpenAIs aktuelle Docs empfehlen die MCP Apps `ui/*`-Bridge für neue Arbeit, während `window.openai` als Kompatibilitätsschicht und optionale Erweiterungsoberfläche erhalten bleibt. Kopiere auch deren Aufteilung zwischen Daten-Tools und Render-Tools: Lass das Modell Daten holen und darüber nachdenken, bevor es sich entscheidet, ein Widget zu rendern.

**Du willst Natural-Language-Dashboards, Reports oder Formulare in deiner eigenen App.**

Probiere json-render, Hashbrown oder OpenUI. Der Schlüssel ist der Katalog. Wenn du `LineChart`, `DataTable`, `MetricGroup`, `FilterControl` und `InsightCallout` bereitstellst, kann das Modell nützliche Reporting-Oberflächen zusammenstellen, ohne in die Nähe von beliebigem Code zu kommen.

**Du willst edukative, visuelle oder hochgradig maßgeschneiderte Artefakte.**

Verwende eine offene Sandbox wie OpenGenerativeUI. Lass das Modell SVG, Canvas, WebGL oder in sich geschlossenes HTML schreiben, aber behandle die Ausgabe wie nicht vertrauten Nutzerinhalt. Sandbox sie, dimensioniere sie, streife Berechtigungen ab und halte sie von privilegiertem App-Zustand fern.

**Du brauchst hauptsächlich schöneres Streaming-Markdown mit ein paar Inline-Affordanzen.**

Baue nicht über. llm-ui oder assistant-ui Tool-Rendering könnten genug sein.

## Die Fehler, die ich vermeiden würde

**Fehler 1: Das Modell zur Laufzeit Produktions-React schreiben lassen.**

Es gibt Ausnahmen, aber für Produkt-UI ist das normalerweise die falsche Standardeinstellung. Laufzeit-Code-Generierung ist schwer zu sichern, schwer zu testen, schwer zu thematisieren und schwer barrierefrei zu halten. Wenn das Modell die Arbeit erledigen kann, indem es aus vertrauten Komponenten wählt, tu das.

**Fehler 2: Design-Primitive statt Produkt-Primitive bereitstellen.**

Wenn man dem Modell `Row`, `Column`, `Text` und `Button` gibt, bittet man es, dein Design-System zu werden. Es wird ein mittelmäßiges. Gib ihm höherstufige Produkt-Nomen.

**Fehler 3: Denken, dass gültiges JSON sichere UI bedeutet.**

Eine Payload kann Schema-Validierung passieren und trotzdem manipulativ oder gefährlich sein. Das Label kann „Rechnung anzeigen" sagen, während die Aktion das Konto archiviert. Behandle UI-Spezifikationen als Verhalten, nicht als Dekoration. Sie brauchen Policy-Tests, semantische Checks und menschliche Bestätigung für folgenreiche Aktionen.

**Fehler 4: Geschäftslogik in Render-Tools stecken.**

Render-Tools sollten rendern. Daten-Tools sollten holen, berechnen, mutieren und validieren. OpenAIs Apps SDK-Docs rufen diese Aufteilung aus einem Grund hervor: Wenn jedes Daten-Tool ein Widget mitschleppt, verliert das Modell Raum, um vor dem Präsentieren zu überlegen.

**Fehler 5: Optimierung für Neuheit statt Aufgabenerfüllung.**

Der Punkt ist nicht, jede Antwort zu einer Schneeflocken-UI zu machen. Der Punkt ist, Reibung zu reduzieren. Ein stabiles, langweiliges Genehmigungspanel, das dem Nutzer vier Minuten spart, ist besser als ein blendendes generiertes Dashboard, dem man kein zweites Mal vertrauen kann.

## Eine praktische Architektur

Wenn ich heute ein neues Produkt starten würde, würde ich einen gestuften Ansatz verwenden:

1.  **Controlled Tool UI zuerst ausliefern.** Bekannte Tools auf bekannte Komponenten mappen. Jeden Tool-Aufruf, UI-Render und Nutzeraktion protokollieren.
2.  **Einen Domänenkatalog hinzufügen.** Sobald sich Muster wiederholen, `ComparisonTable`, `DecisionPanel`, `DataCollectionForm`, `Timeline` und andere produktspezifische Komponenten bereitstellen.
3.  **Transportstandardisierung nur bei Bedarf hinzufügen.** Wenn man Frontend und Backend besitzt, kann plain Streaming genügen. Wenn man mehrere Agenten-Frameworks hat, verwende AG-UI. Wenn Tools Produktgrenzen überschreiten, verwende MCP. Wenn Agenten organisatorische Grenzen überschreiten, beobachte A2A und A2UI.
4.  **Iframe-Widgets für fremde oder komplexe Oberflächen verwenden.** Karten, Warenkörbe, Buchungsflüsse und Drittanbieter-Mini-Apps gehören hinter eine Grenze.
5.  **Offene Generierung für Artefakte reservieren.** Diagramme, Simulationen, temporäre Erklärungen und visuelle Scratchpads sind eine großartige Passung. Kern-Workflows sind es nicht.

Die Architektur endet so:

```txt
Nutzerabsicht
  -> Agenten-Laufzeit
  -> Tool/Daten-Aufrufe
  -> strukturiertes Ergebnis
  -> UI-Entscheidung
  -> vertraute Komponente, deklarative Spezifikation oder gesandboxtes Widget
  -> Nutzeraktion
  -> Zustand/Ereignis-Stream zurück zum Agenten
```

Diese Schleife ist das eigentliche Produkt. Die Chat-Box ist nur ein mögliches Eingabegerät.

## Evaluation sollte die UI einschließen

LLM-Teams lernen langsam, Prompts und Modellausgaben zu eval-uieren. Generative UI fügt eine weitere Oberfläche hinzu: Die Schnittstelle selbst kann falsch sein.

Speichere mindestens diese Artefakte für jede generierte UI:

- Prompt und Tool-Kontext
- Tool-Aufrufe und Tool-Ergebnisse
- Generierte UI-Spezifikation oder Komponentenauswahl
- Gerenderter Komponentenname und Props
- Nutzer-sichtbare Labels
- Aktionen, die an Buttons/Formulare gebunden sind
- Modell-sichtbare Zustandsaktualisierungen aus der UI
- Nutzer-Aktionsverlauf

Dann schreibe Checks wie:

- jede destruktive Aktion muss eine Bestätigungskomponente haben
- Button-Labels müssen zur Aktionssemantik passen
- Render-Spezifikationen dürfen nur erlaubte Komponenten referenzieren
- nutzer-sichtbare Summen müssen mit Tool-Ergebnis-Summen übereinstimmen
- Formulare dürfen keine Felder außerhalb des Aufgabenbereichs anfordern
- Widgets dürfen keine Secrets erhalten, die nur das Modell brauchte
- versteckte Metadaten dürfen sichtbaren Labels nicht widersprechen

Das klingt mühsam. Es ist auch dort, wo Produktionsvertrauen herkommt.

## Ein Hinweis zur Projektstabilität

Jedes große Protokoll in diesem Raum ist pre-1.0. Zuletzt geprüft am 8. Mai 2026; plane mit Veränderung und prüfe die aktuellen Docs, bevor du eine Plattform-Wette eingehst.

**Vercel AI SDK RSC** — das ursprüngliche „Generative UI"-Schlagzeilen-Feature — hatte die Entwicklung im Oktober 2024 pausiert ([Discussion #3251](https://github.com/vercel/ai/discussions/3251)) aufgrund von architektonischen Limitierungen, die kurzfristig keine Lösung hatten. **json-render** (Vercel Labs) emerged als Ersatzrichtung: Katalogbasiert, Framework-agnostisch, keine RSC-Kopplung. Es scheint schnell Web-Developer-Aufmerksamkeit seit seinem Early-2026-Start aufgenommen zu haben. Der wahrscheinliche Grund ist DX: json-render funktioniert sofort in einem Standard-React-Projekt; A2UIs plattformübergreifender Scope adds Setup-Reibung.

**A2UI** (Google) ist pre-1.0 mit Breaking Changes zwischen Minor-Versionen und inkonsistenter Roadmap-Kommunikation. Sein Vorteil ist echte plattformübergreifende Reichweite (Web, Flutter, SwiftUI), die json-render nicht adressiert. Für reine Web-Anwendungsfälle heute scheint json-render bessere Tooling-Abdeckung zu haben; für plattformübergreifende oder Remote-Agenten-Szenarien ist A2UIs Design angemessener. Eine Konvergenz zwischen den beiden Specs ist möglich — Vercel hat mit A2UI-kompatiblem Output von json-render experimentiert.

**AG-UI** (CopilotKit) ist ebenfalls pre-1.0. Die häufigste Verwirrung ist der Name: AG-UI ist ein Transportprotokoll, kein UI-Framework. Es definiert, *wie* Ereignisse zwischen Agent und Frontend fließen; was du renderst, ist immer noch deine Entscheidung. Das Konzept ist solide und breit adoptiert. Die Spec entwickelt sich noch.

## Meine Einschätzung

Generative UI wird sorgfältig designte Produktoberflächen nicht ersetzen. Sie wird die faule Annahme ersetzen, dass ein Chat-Transkript die universelle Schnittstelle für KI ist.

Die besten Systeme werden das Modell nicht alles freistellen lassen. Sie werden ihm einen kleinen, scharfen Satz produkt-eigener Bausteine geben; eine zuverlässige Laufzeitverbindung; klare Sicherheitsgrenzen; und genug Freiheit, die Oberfläche an die Aufgabe anzupassen.

Die Zukunft ist nicht „das Modell schreibt dein Frontend."

Die Zukunft ist näher an: **dein Frontend wird zu einem Instrument, das der Agent spielen kann, aber du entscheidest immer noch, wie das Instrument klingen darf.**
