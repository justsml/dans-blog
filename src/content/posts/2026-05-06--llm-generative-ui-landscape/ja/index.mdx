チャットは補助輪だった。

初期のLLMアプリは、テキストボックスをプロダクトに無理やり貼り付けたような見た目だった。モデルは文章を返す。フロントエンドはMarkdownをレンダリングする。ユーザーが何か操作する必要があれば、アシスタントは「どこかのボタンをクリックしてください」と説明するだけだった。

デモとしてはそれで十分だった。だが、これが向かう先ではない。

次の有用なステップは**生成UI**だ。モデルは単にテキストで答えるだけでなく、ユーザーが今どのインターフェースを必要としているかの判断を助ける。ツールを呼び出してビルド済みのカードをレンダする場合もあれば、既知のワークフローコンポーネントに新しいデータを流し込む場合もある。一時的なダッシュボード、フォーム、比較表、チャート、インタラクティブなウィジェットを構成することもある。

残念なことに、「生成UI」という言葉は、朝食前に5つの異なる意味に膨れ上がる単語のひとつになってしまった。

人々はこれを次のような意味で使っている。

- モデルが開発者定義のReactコンポーネントから選択する
- フロントエンドがネイティブコンポーネントにレンダリングするJSON仕様
- MCPツールから返されるiframeアプリ
- ツール呼び出しをサポートするチャットUIライブラリ
- バックエンドとフロントエンド間で状態をストリーミングするエージェントプロトコル
- v0、Lovable、Bolt、Cursorのようなデザインタイムのコードジェネレーター
- モデルが実行時に文字どおりHTML、SVG、Canvas、Reactを記述する

これらは関連しているが、同じレイヤーではない。これらを混同すれば、あらゆるアーキテクチャの議論がドロドロのスープになってしまう。

これが、現在のスタックを比較し始めた最初に持っていればよかった地図だ。

## 根本的な誤解

最大の誤りは、「生成UI」を単一の技術選択として扱うことだ。

問題を4つのレイヤーに分割する方がよい。

1.  **プロダクトシェル**: ユーザーが触れるもの。チャット、サイドバーのコパイロット、ダッシュボード、ワークフロービルダー、IDEパネル、ChatGPTアプリ、モバイル画面、サポートコンソールなど。
2.  **UI構成モデル**: モデルが話すことを許される文法。ツール呼び出し、JSON、A2UI、json-render、OpenUI Lang、Hashbrownのコンポーネント選択、サンドボックス化されたHTMLなど。
3.  **ランタイムとトランスポート**: メッセージ、ツール呼び出し、状態の差分、ユーザーアクション、UI成果物がエージェントとフロントエンド間でどのように移動するか。AG-UI、MCP Apps、Apps SDK、A2A、SSE、WebSockets、そして昔ながらのHTTPがここに属する。
4.  **エージェントとツールのバックエンド**: LangGraph、Google ADK、CrewAI、Mastra、LlamaIndex、Pydantic AI、Agno、OpenAI Agents SDK、カスタム関数、データベース、検索、そして正しさを保ち続けなければならない退屈なビジネスロジックのすべて。

レイヤーを分けさえすれば、エコシステムはずっと神秘的でなくなる。

[AG-UI](https://github.com/ag-ui-protocol/ag-ui)は[A2UI](https://github.com/google/A2UI)の競合ではない。AG-UIはエージェントからアプリケーションへのインタラクションのためのイベントプロトコルだ。A2UIはエージェントが送信できる宣言的UIフォーマットである。A2UIをAG-UIの上に乗せることもできる。AG-UIの上にカスタムのツールレンダリング済みコンポーネントを乗せることもできる。

[json-render](https://github.com/vercel-labs/json-render)はチャットプロダクトではない。コンポーネントカタログとレンダラーアーキテクチャだ。モデルが使用できるコンポーネントを定義し、モデルに有効なJSONツリーを出力させ、そのツリーを安全にレンダリングする。

[CopilotKit](https://github.com/CopilotKit/CopilotKit)は単なるチャットバブルではない。エージェントネイティブアプリのためのフロントエンドスタックだ。チャットUI、生成UI、共有状態、フロントエンドツール、ヒューマンインザループフローを備える。

[OpenAI Apps SDK](https://developers.openai.com/apps-sdk)と[MCP Apps](https://github.com/MCP-UI-Org/mcp-ui)は「Reactアプリを動的にする」ツールではない。ChatGPTやその他のMCP互換ホスト内でウィジェットをレンダリングするためのホスト統合モデルだ。

名前が混乱するのは、この領域が若いからだ。役に立ち続けるのはレイヤーの方である。

## 制御のスペクトラム

生成UIは**開発者の制御**と**エージェントの自由度**のトレードオフだ。

制御が強すぎれば、アシスタントは仮装したコマンドパレットのように感じる。自由を与えすぎれば、モデルは自信満々な笑みを浮かべながら、奇妙なレイアウト、曖昧なボタン、崩れた視覚的階層、あり得ない状態、そしてセキュリティ問題を生み出し始める。

秘訣は、ユーザーの問題を解決するのに十分な最小限の自由度を選ぶことだ。

![ツールレンダリング済みコンポーネントからオープンエンドに生成されたHTMLまでのスペクトラム](./control-spectrum.webp)

私はこのスペクトラムを次のように考えている。

**ツールからコンポーネントへのレンダリング**が最も安全なデフォルトだ。モデルは `get_weather`、`search_products`、`compare_plans`、`draft_invoice` を呼び出す。アプリはそのツール結果を、すでに所有しているコンポーネント（`WeatherCard`、`ProductGrid`、`PlanComparison`、`InvoiceReview`）にマッピングする。モデルはUIがいつ役立つかを*決定*する。開発者は引き続き、レイアウト、スタイリング、アクセシビリティ、ローディング状態、空状態、危険なアクションを所有する。

これは[Vercel AI SDKの生成UIガイド](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces)に記載されているパターンだ。モデルがツールを呼び出し、ツールがデータを返し、UIが結果からコンポーネントをレンダリングする。これは多くのCopilotKitやassistant-uiの実装の背後にあるメンタルモデルでもある。

**宣言的コンポーネントカタログ**はモデルにより多くの余地を与える。1つのコンポーネントを選ぶ代わりに、モデルは許可されたパーツからツリーを構成する。カタログには `Metric`、`Table`、`Chart`、`FilterBar`、`ApprovalPanel`、`Timeline` などが含まれるかもしれない。モデルはダッシュボードやワークフローのステップを組み立てられるが、任意のコードを実行することはできない。[A2UI](https://github.com/google/A2UI)、[json-render](https://github.com/vercel-labs/json-render)、[Hashbrown](https://github.com/liveloveapp/hashbrown)、[OpenUI](https://github.com/thesysdev/openui) がここに位置する。

**iframeミニアプリ**は、UIがコンポーネントツリーよりもリッチである必要がある場合や、リモートのツールプロバイダーが体験を所有する場合に意味をなす。MCP AppsとOpenAI Apps SDKは、ツールが構造化データと、ホストがiframeでレンダリングするウィジェットリソースを返すことを可能にする。これは地図、ショッピングカート、予約フロー、チャート、外部プロダクトのサーフェスにとって強力だ。また、ホストアプリとウィジェットの間に明確な境界も作る。

**オープンエンドな生成**は最も極端な端にある。エージェントはHTML、SVG、Canvas、WebGL、またはその他のコードのような成果物をサンドボックス内に出力する。[OpenGenerativeUI](https://github.com/CopilotKit/OpenGenerativeUI)が現在の最良の例だ。エージェントはサンドボックス化されたiframe内でアルゴリズムの可視化、3Dシーン、図、シミュレーションを生成できる。これは一度限りの視覚的説明には最適だ。だが、エンタープライズの承認フローで最初に選ぶ場所ではない。

ここで重要な区別を名指しすると役立つ。**iframe HTML**（モデルがサンドボックス内にコードを記述する）対**JSONカタログ**（モデルが構造化仕様を出力し、レンダラーがそれをビルド済みのコンポーネントにマッピングする）だ。これらは関連しているように聞こえるが、リスクと複雑さのプロファイルは大きく異なる。iframe HTMLは最大限の表現力を持つ。iframeの境界がセキュリティの役割を果たす。JSONカタログはモデルに実行可能な自由度を与えない。あらかじめ定義したコンポーネントタイプを参照できるだけだ。この領域のほとんどのフレームワークは、明確にどちらかの陣営に分類される。

**サンドボックスの先へ**: ごく最近のデモは、第4のモードが形成されつつあることを示唆している。LLMが、コンポーネントカタログが許すよりも直接的に視覚出力を制御することで、ゲームのような没入型体験を駆動するものだ。プロンプトから探索可能な3Dワールドを生成するプロジェクト、実行時にLLMが指示するNPCの行動、WebGPUによるブラウザ内モデル推論（[WebLLM](https://mlc.ai/web-llm/)）などが初期の兆候だ。まだ本番作業を構築できる安定したフレームワークはない。状況が変わり次第、専用の記事でこの方向性を取り上げる。

## 高レベルコンポーネント対粒度の細かいコンポーネント

これが最も重要な設計判断だ。

カタログの粒度が細かすぎると、モデルはフロントエンドエンジニアにならざるを得なくなる。

```tsx
Container
Row
Column
Text
Button
Icon
Spacer
Divider
```

柔軟に見えるが、モデルは間隔、階層、グルーピング、空状態、ボタンラベル、エラー処理、レスポンシブ動作を決定しなければならなくなる。プロンプトも肥大化し、出力も壊れやすくなる。

カタログが高レベルすぎると、モデルは行き詰まる。

```tsx
WeatherCard
StockCard
HotelCard
```

安全ではあるが、既知のシナリオでしか機能しない。モデルは比較マトリックスを作ったり、不足している入力を求めたり、ユーザーの質問が変わったときに情報アーキテクチャを適応させたりできない。

有用な中間地点は、**制約されたスロットを持つドメインレベルのコンポーネント**だ。

```tsx
SearchResults
ComparisonTable
MetricGroup
EditablePlan
ApprovalRequest
Timeline
DataCollectionForm
CheckoutReview
```

これらのコンポーネントは、プロダクトのテイストとビジネスの制約をエンコードする。モデルは*何を表示すべきか*を決定できるが、すべてのCSSの判断を下すわけではない。

例えば、旅行エージェントには `div`、`span`、`button` は必要ない。必要なのは次の通りだ。

- `TripSummary`
- `FlightOptionList`
- `HotelComparison`
- `TravelerForm`
- `PolicyNotice`
- `BookingConfirmation`

金融エージェントに汎用的なチャート遊び場は必要ない。必要なのは次の通りだ。

- `PortfolioSnapshot`
- `TransactionTable`
- `RiskBreakdown`
- `ScenarioComparison`
- `ApprovalGate`

カタログはHTMLのように聞こえるのではなく、あなたのプロダクトのように聞こえるべきだ。

## どのプロダクトに何を使うか

これが、実際にチームと使う推薦マトリックスだ。

**既存のSaaSアプリにアシスタントを追加する場合。**

ツールからコンポーネントへのレンダリングから始める。必要なエージェント状態とフロントエンドツール統合の量に応じて、[Vercel AI SDK UI](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces)、[assistant-ui](https://github.com/assistant-ui/assistant-ui)、または [CopilotKit](https://github.com/CopilotKit/CopilotKit) を使用する。最初はカタログを小さく保つ。すでに信頼しているプロダクトコンポーネントをレンダリングする。

**共有状態を必要とする本格的なアプリ内コパイロットを構築する場合。**

CopilotKitとAG-UIを真剣に検討する。重要な機能は「チャット」ではない。共有状態と双方向インタラクションだ。エージェントは入力を求め、UIをレンダリングし、状態を更新し、承認のために一時停止できる。

**境界を越えてUIを送信する必要があるリモートエージェントを持つ場合。**

A2UI、またはA2UIに似た宣言的プロトコルを使用する。要点は、リモートエージェントがUIをデータとして記述できる一方で、ホストがネイティブレンダリング、セキュリティ、スタイルの制御を維持できることだ。ライブなエージェント/アプリのインタラクションも必要な場合は、AG-UI、または環境で標準化されているトランスポート上で実行する。

**ChatGPTまたはMCP互換ホスト内で構築する場合。**

MCP AppsとApps SDKのパスを使用する。OpenAIの現在のドキュメントは、`window.openai` を互換性レイヤーおよびオプションの拡張サーフェスとして維持しつつ、新しい作業にはMCP Appsの `ui/*` ブリッジを推奨している。また、データツールとレンダーツールの分離も踏襲する。モデルがウィジェットのレンダリングを選択する前に、データのフェッチと推論を行えるようにする。

**自社アプリに自然言語のダッシュボード、レポート、フォームを導入したい場合。**

json-render、Hashbrown、またはOpenUIを試す。鍵はカタログだ。`LineChart`、`DataTable`、`MetricGroup`、`FilterControl`、`InsightCallout` を公開すれば、モデルは任意のコードに近づけることなく有用なレポートサーフェスを組み立てられる。

**教育的、視覚的、または極めて特注の成果物を求める場合。**

OpenGenerativeUIのようなオープンエンドなサンドボックスを使用する。モデルにSVG、Canvas、WebGL、または自己完結型HTMLを記述させるが、出力は信頼できないユーザーコンテンツとして扱う。サンドボックス化し、サイズを制限し、権限を剥ぎ取り、特権的なアプリ状態から遠ざける。

**主に、いくつかのインラインな手がかり付きで、より美しいストリーミングMarkdownが必要な場合。**

過剰に構築しない。llm-uiやassistant-uiのツールレンダリングで十分かもしれない。

## 私が避けるべき間違い

**間違い1: モデルに実行時の本番Reactを書かせること。**

例外はあるが、プロダクトUIにとっては通常、間違ったデフォルトだ。実行時のコード生成は、セキュリティ確保が難しく、テストが難しく、テーマ適用が難しく、アクセシビリティの維持も難しい。モデルが信頼できるコンポーネントからの選択で仕事を済ませられるなら、そうするべきだ。

**間違い2: プロダクトプリミティブではなく、デザインプリミティブを公開すること。**

モデルに `Row`、`Column`、`Text`、`Button` を渡すとき、あなたはそれがあなたのデザインシステムになることを求めている。そしてそれは平凡なものになるだろう。より高レベルのプロダクト名詞を与えるべきだ。

**間違い3: 有効なJSON＝安全なUIと思い込むこと。**

ペイロードはスキーマ検証を通過しつつも、操作的または危険なものである可能性がある。ラベルは「請求書を表示」と言いながら、アクションはアカウントをアーカイブすることがある。UI仕様は装飾ではなく動作として扱う。結果を伴うアクションには、ポリシーテスト、セマンティックチェック、人間の確認が必要だ。

**間違い4: ビジネスロジックをレンダーツールに置くこと。**

レンダーツールはレンダリングすべきだ。データツールはフェッチ、計算、変更、検証を行うべきだ。OpenAIのApps SDKドキュメントがこの分離を指摘しているのには理由がある。すべてのデータツールがウィジェットを一緒に引きずり回せば、モデルは提示する前に推論する余地を失う。

**間違い5: タスク完了ではなく、新規性を最適化すること。**

重要なのは、すべての回答を雪の結晶のようなユニークなインターフェースにすることではない。重要なのは摩擦を減らすことだ。ユーザーの4分を節約する安定した退屈な承認パネルの方が、二度と信頼できないまばゆい生成ダッシュボードより優れている。

## 実用的なアーキテクチャ

今日新しいプロダクトを始めるなら、段階的なアプローチを取るだろう。

1.  **制御されたツールUIをまず出荷する。** 既知のツールを既知のコンポーネントにマッピングする。すべてのツール呼び出し、UIレンダリング、ユーザーアクションをログに記録する。
2.  **ドメインカタログを追加する。** パターンが繰り返され始めたら、`ComparisonTable`、`DecisionPanel`、`DataCollectionForm`、`Timeline`、その他のプロダクト固有のコンポーネントを公開する。
3.  **必要な場合にのみトランスポートの標準化を追加する。** フロントエンドとバックエンドの両方を所有しているなら、プレーンなストリーミングで十分かもしれない。複数のエージェントフレームワークがあるならAG-UIを使用する。ツールがプロダクトの境界を越えるならMCPを使用する。エージェントが組織の境界を越えるなら、A2AとA2UIに注目する。
4.  **外部または複雑なサーフェスにはiframeウィジェットを使用する。** 地図、カート、予約フロー、サードパーティのミニアプリは境界の後ろに置くべきだ。
5.  **オープンエンドな生成は成果物用に予約する。** 図、シミュレーション、一時的な説明、視覚的なスクラッチパッドには非常适合。コアワークフローには不适合。

アーキテクチャは最終的にこのようになる。

```txt
ユーザーの意図
  -> エージェントランタイム
  -> ツール/データ呼び出し
  -> 構造化された結果
  -> UIの判断
  -> 信頼されたコンポーネント、宣言的仕様、またはサンドボックス化されたウィジェット
  -> ユーザーアクション
  -> エージェントへの状態/イベントストリーム
```

そのループこそが本物のプロダクトだ。チャットボックスは単なる入力デバイスのひとつに過ぎない。

## 評価にはUIを含めるべきだ

LLMチームはプロンプトとモデル出力を評価することをゆっくりと学んでいる。生成UIは別の評価対象面を追加する。インターフェース自体が間違っている可能性があるのだ。

最低限、生成されたUIごとにこれらの成果物を保存する。

- プロンプトとツールのコンテキスト
- ツール呼び出しとツール結果
- 生成されたUI仕様またはコンポーネント選択
- レンダリングされたコンポーネント名とprops
- ユーザーに表示されるラベル
- ボタン/フォームに紐づくアクション
- UIからのモデル可視状態更新
- ユーザーアクションの履歴

そして次のようなチェックを書く。

- すべての破壊的アクションには確認コンポーネントが必要
- ボタンラベルはアクションのセマンティクスと一致しなければならない
- レンダ仕様は許可されたコンポーネントのみを参照できる
- ユーザーに表示される合計はツール結果の合計と一致しなければならない
- フォームはタスクのスコープ外のフィールドを要求してはならない
- ウィジェットはモデルのみが必要だったシークレットを受け取ってはいけない
- 隠されたメタデータは可視ラベルと矛盾してはならない

面倒に聞こえるだろう。だが、これこそが本番環境での信頼の源なのだ。

## プロジェクトの安定性に関する注意

この領域の主要なプロトコルはすべて1.0未満だ。最終確認は2026年5月8日。変化を前提に計画し、プラットフォームに賭ける前に現在のドキュメントを確認すること。

**Vercel AI SDK RSC** — 元の「生成UI」の目玉機能 — は、短期的な修正が不可能なアーキテクチャ上の制限により、2024年10月に開発が一時停止した（[Discussion #3251](https://github.com/vercel/ai/discussions/3251)）。代わりに登場したのが **json-render**（Vercel Labs）だ。カタログベース、フレームワーク非依存、RSCとの結合なし。2026年初頭のローンチ以来、Web開発者の注目を急速に集めているようだ。その理由はおそらくDXにある。json-renderは標準的なReactプロジェクトですぐに動作する。一方、A2UIのクロスプラットフォームな範囲はセットアップの摩擦を増やす。

**A2UI**（Google）は1.0未満で、マイナーバージョン間で破壊的変更があり、ロードマップのコミュニケーションに一貫性がない。その利点は、json-renderが扱わない真のクロスプラットフォーム対応（Web、Flutter、SwiftUI）だ。今日の純粋なWebユースケースでは、json-renderの方がツールカバレッジに優れているように見える。クロスプラットフォームやリモートエージェントのシナリオでは、A2UIの設計の方が適切だ。2つの仕様間の収束はあり得る。Vercelはjson-renderからA2UI互換の出力を実験している。

**AG-UI**（CopilotKit）も1.0未満だ。最もよくある混乱は名前に関するものだ。AG-UIはUIフレームワークではなく、トランスポートプロトコルである。エージェントとフロントエンド間でイベントが*どのように*流れるかを定義する。何をレンダリングするかは依然としてあなたの判断だ。コンセプトは堅実で広く採用されている。仕様はまだ進化中だ。

## 私の見解

生成UIが、慎重に設計されたプロダクトインターフェースを置き換えることはない。生成UIが置き換えるのは、「チャットのやり取りがAIのための万能インターフェースである」という怠惰な思い込みだ。

最良のシステムは、モデルにすべてをフリースタイルでやらせない。プロダクト固有の小さく鋭いビルディングブロックのセット、信頼できるランタイム接続、明確なセキュリティ境界、そしてタスクに合わせてインターフェースを適応させるのに十分な自由度を与えるだろう。

未来は「モデルがあなたのフロントエンドを書く」ことではない。

未来はこれに近い。**あなたのフロントエンドはエージェントが演奏できる楽器になるが、その楽器がどのような音を許されるかは、依然としてあなたが決めるのだ。**
