チャットは補助輪だった。

LLMアプリの第一世代は、だいたいテキストボックスをプロダクトに貼り付けたような見た目だった。モデルは文章を返し、フロントエンドはmarkdownをレンダリングした。ユーザーがアクションを起こす必要があるなら、アシスタントは「あそこのボタンを押してください」と説明するだけだった。

デモとしてはそれで良かった。しかし、それが行き着く先ではない。

次に有用なのは**ジェネレーティブUI**だ。モデルは単にテキストで答えるだけでなく、ユーザーが今必要としているインターフェースを決めるのを助ける。時にはツールを呼び出して事前定義のカードを描画する。時には既知のワークフローコンポーネントに新鮮なデータを流し込む。時には一時的なダッシュボード、フォーム、比較テーブル、チャート、インタラクティブウィジェットを組み立てる。

残念ながら、「ジェネレーティブUI」という言葉は、朝食前に5つの違う意味で使われている。

人々はこれを次のような意味で使う：

- 開発者が定義したReactコンポーネントからモデルが選ぶ仕組み
- フロントエンドがネイティブコンポーネントに描画するJSON仕様
- MCPツールから返されるiframeアプリ
- ツールコールをサポートするチャットUIライブラリ
- バックエンドとフロントエンドの間で状態をストリーミングするエージェントプロトコル
- v0、Lovable、Bolt、Cursorのようなデザイン時コードジェネレーター
- 実行時にモデルがHTML、SVG、Canvas、Reactを文字通り書く仕組み

これらは関連しているが、同じレイヤーではない。混同すると、あらゆるアーキテクチャの会話がスープになる。

これは私がこのスタックを最初に比較し始めたときに持っていたかった地図だ。

## 核心の誤解

最大の間違いは、「ジェネレーティブUI」を一つの技術選択として扱うことだ。

この問題を4つのレイヤーに分けると良い：

1. **プロダクトシェル**：ユーザーが触るもの。チャット、サイドバーコパイロット、ダッシュボード、ワークフロービルダー、IDEパネル、ChatGPTアプリ、モバイル画面、サポートコンソールのいずれか。
2. **UI構成モデル**：モデルが話すことを許される文法。ツールコール、JSON、A2UI、json-render、OpenUI Lang、Hashbrownコンポーネント選択、サンドボックス化されたHTMLのいずれか。
3. **ランタイムとトランスポート**：メッセージ、ツールコール、状態差分、ユーザーアクション、UIアーティファクトがエージェントとフロントエンドの間でどう移動するか。AG-UI、MCP Apps、Apps SDK、A2A、SSE、WebSocket、普通のHTTPがここに属する。
4. **エージェントとツールバックエンド**：LangGraph、Google ADK、CrewAI、Mastra、LlamaIndex、Pydantic AI、Agno、OpenAI Agents SDK、カスタム関数、データベース、検索、そしてまだ正しく動かなければならない退屈なビジネスロジック全部。

レイヤーを分ければ、エコシステムはずっと神秘性が減る。

[AG-UI](https://github.com/ag-ui-protocol/ag-ui)は[A2UI](https://github.com/google/A2UI)の競合ではない。AG-UIはエージェント↔アプリケーション間相互作用のイベントプロトコルだ。A2UIはエージェントが送れる宣言的UIフォーマットだ。A2UIをAG-UIの上に乗せることもできる。カスタムツール描画コンポーネントをAG-UIの上に乗せることもできる。

[json-render](https://github.com/vercel-labs/json-render)はチャットプロダクトではない。コンポーネントカタログとレンダラーアーキテクチャだ。モデルが使用できるコンポーネントを定義し、有効なJSONツリーを出力させ、そのツリーを安全に描画する。

[CopilotKit](https://github.com/CopilotKit/CopilotKit)は単なるチャットバブルではない。エージェントネイティブアプリのためのフロントエンドスタックだ。チャットUI、ジェネレーティブUI、共有状態、フロントエンドツール、ヒューマンインザループフローを備えている。

[OpenAI Apps SDK](https://developers.openai.com/apps-sdk)と[MCP Apps](https://github.com/MCP-UI-Org/mcp-ui)は「Reactアプリを動的にする」ツールではない。ChatGPTや他のMCP互換ホスト内にウィジェットを描画するためのホスト統合モデルだ。

名前が紛らわしいのは、この分野が若いからだ。レイヤーの理解こそがずっと役に立つ。

## コントロールのスペクトラム

ジェネレーティブUIは**開発者のコントロール**と**エージェントの自由度**のトレードオフだ。

コントロールが強すぎると、アシスタントはコマンドパレットの着ぐるみに見える。自由度が高すぎると、モデルは変なレイアウト、曖昧なボタン、壊れたビジュアルヒエラルキー、あり得ない状態、セキュリティ問題を自信満々に発明し始める。

コツは、ユーザーの問題を解決する最小限の自由度を選ぶことだ。

![ツール描画コンポーネントから自由生成のHTMLまでのコントロールスペクトラム](./control-spectrum.webp)

私はこのスペクトラムを次のように考える：

**ツール→コンポーネント描画**が最も安全なデフォルトだ。モデルは`get_weather`、`search_products`、`compare_plans`、`draft_invoice`を呼び出す。アプリはそのツール結果を既に所有しているコンポーネントにマッピングする：`WeatherCard`、`ProductGrid`、`PlanComparison`、`InvoiceReview`。モデルが*いつ*UIが有用かを決める。開発者はレイアウト、スタイリング、アクセシビリティ、ローディング状態、空状態、危険なアクションを所有し続ける。

これは[Vercel AI SDKのジェネレーティブUIガイド](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces)で文書化されているパターンだ：モデルがツールを呼び出し、ツールがデータを返し、UIがその結果からコンポーネントを描画する。CopilotKitやassistant-uiの多くの実装でも同じメンタルモデルが使われている。

**宣言的コンポーネントカタログ**はモデルに多くの余地を与える。1つのコンポーネントを選ぶ代わりに、モデルは許可されたパーツからツリーを組み立てる。カタログには`Metric`、`Table`、`Chart`、`FilterBar`、`ApprovalPanel`、`Timeline`が含まれるかもしれない。モデルはダッシュボードやワークフローステップを組み立てられるが、任意のコードは実行できない。[A2UI](https://github.com/google/A2UI)、[json-render](https://github.com/vercel-labs/json-render)、[Hashbrown](https://github.com/liveloveapp/hashbrown)、[OpenUI](https://github.com/thesysdev/openui)がここに位置する。

**iframeミニアプリ**は、UIがコンポーネントツリーより豊かである必要がある場合や、リモートツールプロバイダーがエクスペリエンスを所有する場合に意味がある。MCP AppsとOpenAI Apps SDKは、ツールが構造化データとウィジェットリソースを返し、ホストがそれをiframe内に描画する。これは地図、ショッピングカート、予約フロー、チャート、外部プロダクト画面に強力だ。また、ホストアプリとウィジェットの間により硬い境界を作る。

**自由生成**は遠い端だ：エージェントがHTML、SVG、Canvas、WebGL、または他のコード的なアーティファクトをサンドボックスに出力する。[OpenGenerativeUI](https://github.com/CopilotKit/OpenGenerativeUI)が現在の最良の例だ：エージェントはサンドボックス化されたiframe内でアルゴリズム可視化、3Dシーン、図、シミュレーションを生成できる。これは一回限りの視覚説明に最適だ。エンタープライズの承認フローには、ここから始めないだろう。

ここで重要な区別に名前を付けておくと便利だ：**iframe HTML**（モデルがサンドボックスにコードを書く）と**JSONカタログ**（モデルが構造化仕様を出力し、レンダラーが事前構築コンポーネントにマッピングする）。これらは関連しているように聞こえるが、リスクと複雑さのプロファイルは全く異なる。iframe HTMLは最大限の表現力を持つ；iframeの境界がセキュリティの仕事をする。JSONカタログはモデルに実行可能な自由を与えない — 事前に定義したコンポーネント型のみを参照できる。この分野のフレームワークの大部分は、明確にどちらかの陣営に属する。

**サンドボックスの先**: 非常に最近のデモは、4つ目のモードが形成されていることを示唆している — LLMがコンポーネントカタログが許す以上に直接的に視覚出力を制御し、ゲーム的または没入型エクスペリエンスを駆動する。プロンプトから探索可能な3Dワールドを生成するプロジェクト、実行時にLLMが指示するNPC行動、WebGPU経由のブラウザ内モデル推論（[WebLLM](https://mlc.ai/web-llm/)）は初期の兆候だ。ここにはまだプロダクションワークを構築できる安定したフレームワークはない。この方向性は、状況が変わったら別記事で取り上げる。

## 高レベルコンポーネント vs 細粒度コンポーネント

これが最も重要な設計決定だ。

カタログが細かすぎると、モデルはフロントエンドエンジニアにならなければならない：

```tsx
Container
Row
Column
Text
Button
Icon
Spacer
Divider
```

これは柔軟に見えるが、今やモデルはスペーシング、階層、グループ化、空状態、ボタンラベル、エラー処理、レスポンシブ動作を決めなければならない。プロンプトも大きくなり、出力も壊れやすくなる。

カタログが高レベルすぎると、モデルは閉じ込められる：

```tsx
WeatherCard
StockCard
HotelCard
```

これは安全だが、既知のシナリオでしか機能しない。モデルは比較マトリックスを作ったり、欠落入力を求めたり、ユーザーの質問が変わったときに情報アーキテクチャを適応させたりできない。

有用な中間は**制約付きスロットを持つドメインレベルコンポーネント**だ：

```tsx
SearchResults
ComparisonTable
MetricGroup
EditablePlan
ApprovalRequest
Timeline
DataCollectionForm
CheckoutReview
```

これらのコンポーネントはプロダクトの味付けとビジネス制約をエンコードする。モデルが決めるのは*何を表示すべきか*で、すべてのCSSの決定ではない。

例えば、旅行エージェントは`div`、`span`、`button`を必要としない。必要なのは：

- `TripSummary`
- `FlightOptionList`
- `HotelComparison`
- `TravelerForm`
- `PolicyNotice`
- `BookingConfirmation`

金融エージェントは汎用チャートプレイグラウンドを必要としない。必要なのは：

- `PortfolioSnapshot`
- `TransactionTable`
- `RiskBreakdown`
- `ScenarioComparison`
- `ApprovalGate`

カタログはHTMLではなく、プロダクトの言葉で書かれているべきだ。

## どのプロダクトに何を使うか

これが実際にチームで使うおすすめのマトリックスだ。

**既存のSaaSアプリにアシスタントを追加する場合。**

ツール→コンポーネント描画から始める。エージェント状態とフロントエンドツール統合の必要度に応じて、[Vercel AI SDK UI](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces)、[assistant-ui](https://github.com/assistant-ui/assistant-ui)、または[CopilotKit](https://github.com/CopilotKit/CopilotKit)を使用する。最初はカタログを小さく保つ。既に信頼しているプロダクトコンポーネントを描画する。

**共有状態が必要な本格的なアプリ内コパイロットを構築する場合。**

CopilotKit + AG-UIを真剣に検討する。重要な機能は「チャット」ではない。共有状態と双方向インタラクションだ：エージェントは入力を求め、UIを描画し、状態を更新し、承認のために一時停止できる。

**リモートエージェントが境界を越えてUIを送信する必要がある場合。**

A2UIまたはA2UI的な宣言的プロトコルを使用する。要点は、リモートエージェントがUIをデータとして記述でき、ホストがネイティブ描画、セキュリティ、スタイルの制御を維持することだ。ライブなエージェント/アプリインタラクションも必要な場合は、AG-UIまたは環境が標準化しているトランスポート上で実行する。

**ChatGPTやMCP互換ホスト内で構築する場合。**

MCP AppsとApps SDKパスを使用する。OpenAIの現在のドキュメントは新しい作業にMCP Apps `ui/*`ブリッジを推奨し、`window.openai`を互換レイヤーとオプションの拡張サーフェスとしている。また、データツールと描画ツールの分離をコピーする：モデルにデータを取得・推論させてからウィジェットの描画を選択させる。

**自社アプリで自然言語ダッシュボード、レポート、フォームが欲しい場合。**

json-render、Hashbrown、またはOpenUIを試す。カタログが鍵だ。`LineChart`、`DataTable`、`MetricGroup`、`FilterControl`、`InsightCallout`を公開すれば、モデルは任意のコードに近づくことなく有用なレポート画面を組み立てられる。

**教育的、視覚的、または非常に特殊なアーティファクトが欲しい場合。**

OpenGenerativeUIのような自由サンドボックスを使用する。モデルにSVG、Canvas、WebGL、または自己完結HTMLを書かせるが、出力を信頼できないユーザーコンテンツとして扱う。サンドボックス化し、サイズ制限し、権限を削除し、特権アプリ状態から遠ざける。

**主に、いくつかのインライン機能を備えたきれいなストリーミングmarkdownが欲しい場合。**

作りすぎないこと。llm-uiやassistant-uiのツール描画で十分かもしれない。

## 避けるべき間違い

**間違い1: モデルに実行時にプロダクションReactを書かせる。**

例外はあるが、プロダクトUIでは通常これが間違ったデフォルトだ。実行時コード生成はセキュアにするのが難しく、テストが難しく、テーマ適用が難しく、アクセシビリティを保つのが難しい。モデルが信頼できるコンポーネントから選択することで仕事を果たせるなら、そうすべきだ。

**間違い2: プロダクトプリミティブではなくデザインプリミティブを公開する。**

モデルに`Row`、`Column`、`Text`、`Button`を与えると、モデルにデザインシステムになることを求めている。それは平凡なデザインシステムになる。より高レベルなプロダクト名詞を与えるべきだ。

**間違い3: 有効なJSON = 安全なUIだと思い込む。**

ペイロードはスキーマバリデーションを通っても、操作的または危険であり得る。ラベルは「請求書を見る」と言いつつ、アクションはアカウントをアーカイブするかもしれない。UI仕様を装飾ではなく振る舞いとして扱う必要がある。ポリシーテスト、意味的チェック、結果的なアクションには人間の確認が必要だ。

**間違い4: 描画ツールにビジネスロジックを入れる。**

描画ツールは描画すべきだ。データツールは取得、計算、変更、検証を行うべきだ。OpenAIのApps SDKドキュメントがこの分離を呼びかけているのには理由がある：すべてのデータツールがウィジェットを伴うと、モデルは提示の前に推論する余地を失う。

**間違い5: タスク完了より新しさを最適化する。**

ポイントはすべての回答を雪flakeインターフェースにすることではない。ポイントは摩擦を減らすことだ。ユーザーの4分を節約する安定した退屈な承認パネルは、2回信頼できない派手な生成ダッシュボードより優れている。

## 実践的アーキテクチャ

今日新しいプロダクトを始めるなら、段階的アプローチを使う：

1. **最初は制御されたツールUIを出荷する。** 既知のツールを既知のコンポーネントにマッピングする。すべてのツールコール、UI描画、ユーザーアクションをログに記録する。
2. **ドメインカタログを追加する。** パターンが繰り返されたら、`ComparisonTable`、`DecisionPanel`、`DataCollectionForm`、`Timeline`などプロダクト固有のコンポーネントを公開する。
3. **必要な場合のみトランスポート標準化を追加する。** フロントエンドとバックエンドの両方を所有しているなら、普通のストリーミングで十分かもしれない。複数のエージェントフレームワークがあるなら、AG-UIを使う。ツールがプロダクト境界を越えるなら、MCPを使う。エージェントが組織境界を越えるなら、A2AとA2UIを注視する。
4. **外国または複雑な画面にはiframeウィジェットを使用する。** 地図、カート、予約フロー、サードパーティミニアプリは境界の後ろに属する。
5. **自由生成はアーティファクトのために予約する。** 図、シミュレーション、一時的な説明、視覚スクラッチパッドは良い適合だ。コアワークフローはそうではない。

アーキテクチャは最終的に次のようになる：

```txt
ユーザーの意図
  -> エージェントランタイム
  -> ツール/データ呼び出し
  -> 構造化結果
  -> UI決定
  -> 信頼できるコンポーネント、宣言的仕様、またはサンドボックス化ウィジェット
  -> ユーザーアクション
  -> 状態/イベントストリームがエージェントに戻る
```

このループが本当のプロダクトだ。チャットボックスは可能な入力デバイスの一つに過ぎない。

## 評価にはUIを含めるべき

LLMチームはプロンプトとモデル出力の評価を徐々に学んでいる。ジェネレーティブUIは別の表面積を追加する：インターフェース自体が間違っている可能性がある。

最低でも、生成されたすべてのUIについて次のアーティファクトを保存する：

- プロンプトとツールコンテキスト
- ツールコールとツール結果
- 生成されたUI仕様またはコンポーネント選択
- 描画されたコンポーネント名とprops
- ユーザーに見えるラベル
- ボタン/フォームに付随するアクション
- UIからのモデル可視状態更新
- ユーザーアクション履歴

そして次のようなチェックを書く：

- すべての破壊的アクションには確認コンポーネントが必要
- ボタンラベルはアクションの意味論と一致しなければならない
- 描画仕様は許可されたコンポーネントのみを参照してよい
- ユーザーに見える合計はツール結果の合計と一致しなければならない
- フォームはタスクスコープ外のフィールドを要求してはならない
- ウィジェットはモデルのみが必要だった秘密を受信してはならない
- 隠しメタデータは可視ラベルと矛盾してはならない

これは退屈に聞こえる。しかし、プロダクションの信頼はここから来る。

## プロジェクトの安定性についての注記

この分野の主要なプロトコルはすべてプレ1.0だ。2026年5月8日確認；変更を計画し、プラットフォームの決定をする前に現在のドキュメントを確認すること。

**Vercel AI SDK RSC** — 元の「ジェネレーティブUI」のヘッドライン機能 — は、近期的な修正がないアーキテクチャ上の制限により、2024年10月に開発が一時停止した（[Discussion #3251](https://github.com/vercel/ai/discussions/3251)）。**json-render**（Vercel Labs）が代替方向として登場した：カタログベース、フレームワーク非依存、RSC結合なし。2026年初頭のローンチ以来、Web開発者の注目を急速に集めているように見える。おそらんその理由はDXだ：json-renderは標準的なReactプロジェクトですぐに動く；A2UIのクロスプラットフォームスコープはセットアップの摩擦を増やす。

**A2UI**（Google）はプレ1.0で、マイナーバージョン間に破壊的変更があり、ロードマップのコミュニケーションが一貫していない。利点はjson-renderが対応していない真のクロスプラットフォーム対応（Web、Flutter、SwiftUI）だ。今日の純粋なWebユースケースでは、json-renderの方がツーリングカバレッジが良いように見える；クロスプラットフォームやリモートエージェントのシナリオでは、A2UIの設計がより適切だ。2つの仕様の収束は可能だ — Vercelはjson-renderからA2UI互換の出力を試験している。

**AG-UI**（CopilotKit）もプレ1.0だ。最も一般的な混乱は名前だ：AG-UIはUIフレームワークではなく、トランスポートプロトコルだ。エージェントとフロントエンドの間でイベントがどう流れるかを定義する；何を描画するかはまだあなたの決定だ。コンセプトは確固で広く採用されている。仕様はまだ進化中だ。

## 私の見解

ジェネレーティブUIは慎重に設計されたプロダクトインターフェースを置き換えない。チャットの書き起こしがAIの普遍的インターフェースであるという怠惰な仮定を置き換えるだろう。

最良のシステムはモデルにすべてを即興で作らせない。プロダクトネイティブな構成ブロックの小さく鋭いセット、信頼できるランタイム接続、明確なセキュリティ境界、タスクに合わせてインターフェースを適応させる十分な自由を与える。

未来は「モデルがフロントエンドを書く」ではない。

未来はこれに近い：**フロントエンドがエージェントが演奏できる楽器になるが、その楽器がどんな音を出してよいかはまだあなたが決める。**
