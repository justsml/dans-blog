チャットはتريainingホイールだった。

最初のLLMアプリケーションの世代は 대부분、製品にテキストボックスを縫い合わせたような見た目だった。モデルはテキストで返答し、フロントエンドはマークダウンをレンダリングする。ユーザーがアクションを起こ必要がある場合、アシスタントはユーザーが別の場所でクリックすべきボタンを説明していた。

デモにはそれで十分だった。だが、ここに向かう道ではない。

次の有用的ステップは**生成UI**：モデルは単にテキストで答えるだけでなく、今ユーザーが必要としているインターフェースを一緒に決める。有时候意味着调用工具并渲染预建的卡片，有时候意味着用新鲜数据填充已知的工作流组件，有时候意味着组合一个临时的ダッシュボード、フォーム、比較テーブル、グラフ、インタラクティブウィジェット。

不幸なことに、「生成UI」は朝食前に5つの異なることを意味するフレーズになってしまっている。

人々はこれを使って以下を説明する：

- 開発者が定義したReactコンポーネントからモデルが選択する
- フロントエンドがネイティブコンポーネントにレンダリングするJSON仕様
- MCPツールから返されるiframeアプリ
- ツール呼び出しをサポートするチャットUIライブラリ
- バックエンドとフロントエンド間の状態をストリーミングするエージェントプロトコル
- v0、Lovable、Bolt、Cursorのような設計時コードジェネレーター
- 実行時にモデルが文字通りHTML、SVG、Canvas、Reactを書く

これらは関連しているが、同じレイヤーではない。これらを混在させると、すべてのアーキテクチャconversationがスープになる。

これが、私が初めて現在のスタックを比較し始めた時にあればよかったと思う地図だ。

## コアな誤解

最大の間違いは「生成UI」を 하나의技術選択として扱うことだ。

問題を4つのレイヤーに分離する方がよい：

1.  **プロダクトシェル**：ユーザーが触るもの。チャット、サイドバーcopilot、ダッシュボード、ワークフロービルダー、IDEパネル、ChatGPTアプリ、モバイルスクリーン、サポートコンソールなど。
2.  **UI構成モデル**：モデルが話ことが許可されている文法。ツール呼び出し、JSON、A2UI、json-render、OpenUI Lang、Hashbrownコンポーネント選択、沙箱HTMLなど。
3.  **ランタイムとトランスポート**：メッセージ、ツール呼び出し、状態差分、ユーザーアクション、UI成果物がエージェントとフロントエンドの間を移動する方法。AG-UI、MCP Apps、Apps SDK、A2A、SSE、WebSocket、そしてありふれたHTTPがここに属する。
4.  **エージェントとツールバックエンド**：LangGraph、Google ADK、CrewAI、Mastra、LlamaIndex、Pydantic AI、Agno、OpenAI Agents SDK、カスタム関数、データベース、検索取得、そしてまだ正しい必要のある全部の退屈なビジネスロジック。

レイヤーを分離すると、エコシステムは的神秘的でなくなる。

[AG-UI](https://github.com/ag-ui-protocol/ag-ui)は[A2UI](https://github.com/google/A2UI)の真の競合ではない。AG-UIはエージェントからアプリケーションへのイベントプロトコル。A2UIはエージェントが送信できる宣言的UIフォーマット。A2UIをAG-UIの上に置くこともできる。カスタムツールレンダリングコンポーネントをAG-UIの上に置くこともできる。

[json-render](https://github.com/vercel-labs/json-render)はチャット製品ではない。それはコンポーネントカタログとレンダラーアーキテクチャだ：モデルが使用できるコンポーネントを定義し、モデルに有効なJSONツリーをemitさせ、そのツリーを安全にレンダリングする。

[CopilotKit](https://github.com/CopilotKit/CopilotKit)は単なるチャットバブルではない。それはエージェントネイティブアプリのフロントエンドスタックだ：チャットUI、生成UI、共有状態、フロントエンドツール、ヒューマン・イン・ザ・ループフロー。

[OpenAI Apps SDK](https://developers.openai.com/apps-sdk)と[MCP Apps](https://github.com/MCP-UI-Org/mcp-ui)は「私のReactアプリを動的にする」ツールではない。それらはChatGPTや他のMCP互換ホスト内にウィジェットをレンダリングするためのホスト統合モデル。

名前が混乱しているのは、スペースが若いからだ。レイヤーは有用なまま的部分だ。

## コントロールスペクトラム

生成UIは**開発者コントロール**と**エージェントの自由**のトレードオフだ。

コントロール太多だと、アシスタントはコマンドパレットがコスプレしたように感じる。自由太多だと、モデルは奇妙なレイアウト、自信犯げな微笑，带着不安全元件を作成する。

のコツは、ユーザーの問題を解決する最小限の自由を選択することだ。

![ツールレンダリングコンポーネントからオープンな生成HTMLまでのスペクトラム](./control-spectrum.webp)

私はスペクトラムをこのように考える：

**ツールからコンポーネントへのレンダリング**は最も安全なデフォルト。モデルは`get_weather`、`search_products`、`compare_plans`、`draft_invoice`などを呼び出す。アプリはそれを既に持っているコンポーネントにマッピングする：`WeatherCard`、`ProductGrid`、`PlanComparison`、`InvoiceReview`。モデルはUIがいつ有用かを決定する。開発者は依然としてレイアウト、スタイリング、アクセシビリティ、負荷状態、空状態、危険なアクション的所有する。

これは[Vercel AI SDKの生成UIガイド](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces)で文書化されたパターンだ：モデルはツールを呼び出し、ツールはデータを返し、UIは結果からコンポーネントをレンダリングする。それは多くのCopilotKitとassistant-ui実装のメンタモデルでもある。

**宣言的コンポーネントカタログ**はモデルにより多くの部屋を与える。1つのコンポーネントを選択する代わりに、モデルは許可されたパーツからツリーを構成する。カタログには`Metric`、`Table`、`Chart`、`FilterBar`、`ApprovalPanel`、`Timeline`が含まれるかもしれない。モデルはダッシュボードやワークフローステップをアセンブリできるが、任意のコードを実行することはできない。ここには[A2UI](https://github.com/google/A2UI)、[json-render](https://github.com/vercel-labs/json-render)、[Hashbrown](https://github.com/liveloveapp/hashbrown)、[OpenUI](https://github.com/thesysdev/openui)がある。

**iframeミニアプリ**は、UIがコンポーネントツリーよりも豊か必要がある場合、またはリモートツールプロバイダーが体験所有する場合に理にかなっている。MCP AppsとOpenAI Apps SDKは、ツールが構造化データプラスホストがiframe内にレンダリングするウィジェットリソースを返すことを許可する。これはマップ、ショッピングカート、予約フロー、グラフ、外部製品 поверхностиに役に立つ。それはまたホストアプリとウィジェットの間に より難しい境界を作成する。

**オープンな生成**は遠い端だ：エージェントはHTML、SVG、Canvas、WebGL、または他のコードのような成果物をサンドボックスにemitする。[OpenGenerativeUI](https://github.com/CopilotKit/OpenGenerativeUI)は現在の最良の例だ：エージェントはサンドボックス化されたiframe内でアルゴリズム可視化、3Dシーン、ダイアグラム、シミュレーションを生成できる。これは一回限りの視覚的説明に最適だ。エンタープライズ承認フロー从哪里始める場所ではない。

ここに重要な区別を名前することは役立つ：**iframe HTML**（モデルはコードをサンドボックスに書く）vs **JSONカタログ**（モデルは構造化された仕様をemitし、レンダラーはそれを事前構築されたコンポーネントにマッピングする）。これらは似ているように聞こえるが、非常に異なるリスクと複雑性のプロファイルを持つ。iframe HTMLは最大限に表現力があり、iframe境界がセキュリティ作業をこなす。JSONカタログはモデルに実行可能な自由を与えない——モデルは事前に定義したコンポーネントタイプのみ参照できる。このスペースのフレームワークのほとんどは明確に一方の阵营に落ちる。

**サンドボックスの先に**：非常に最近のデモは、第四のモードが形成されつつあることを示唆している——LLMがコンポーネントカタログよりも直接的に視覚出力を制御しながら、ゲームのような或者没入型体験させているプロンプトから生成可能な3Dワールド、実行時にLLMが指示するNPC動作、WebGPUによるブラウザ内モデル推論（[WebLLM](https://mlc.ai/web-llm/))など。今のところ生産的な仕事を構築するための安定したフレームワークはない。これが変化したら専用の記事でこの方向性を取り上げる。

## 高レベルコンポーネントvs粒状コンポーネント

これは最も重要な設計決定だ。

カタログが細かすぎる場合、モデルはフロントエンドエンジニアにならなければならない：

```tsx
Container
Row
Column
Text
Button
Icon
Spacer
Divider
```

これは柔軟そうに見えるが、モデルはスペーシング、階層、グループ化、空状態、ボタンラベル、エラー処理、応答動作を決定する必要がある。プロンプトが大きくもなり、出力が壊れやすくなる。

カタログが高すぎる場合、モデルは閉じ込められる：

```tsx
WeatherCard
StockCard
HotelCard
```

これは安全だが既知のシナリオでのみ動作する。モデルは比較マトリックスを作成したり不足している入力を要求したりユーザーの質問が変わったときに情報アーキテクチャ適応したりすることはできない。

有用な中間は**制約されたスロットを持つドメインレベルのコンポーネント**だ：

```tsx
SearchResults
ComparisonTable
MetricGroup
EditablePlan
ApprovalRequest
Timeline
DataCollectionForm
CheckoutReview
```

これらのコンポーネントは製品 taste とビジネス制約をエンコードする。モデルは*何を見せるか*を決定できるが、すべてのCSS決定をする必要はない。

例えば、旅行エージェントは`div`、`span`、`button`を必要としない。以下が必要だ：

- `TripSummary`
- `FlightOptionList`
- `HotelComparison`
- `TravelerForm`
- `PolicyNotice`
- `BookingConfirmation`

財務エージェントは汎用チャートプレイグラウンドを必要としない。以下が必要だ：

- `PortfolioSnapshot`
- `TransactionTable`
- `RiskBreakdown`
- `ScenarioComparison`
- `ApprovalGate`

カタログはHTMLのようにではなく、製品のように聞こえるはずだ。

## どの製品に何を使用するか

チームで使用するRecommendationマトリックスはここだ。

**既存のSaaSアプリにアシスタントを追加している。**

ツールからコンポーネントへのレンダリングから始める。[Vercel AI SDK UI](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces)、[assistant-ui](https://github.com/assistant-ui/assistant-ui)、[CopilotKit](https://github.com/CopilotKit/CopilotKit)のどれを使用するかは、どれだけのエージェント状態とフロントエンドツール統合必要があるかに応じる。最初はカタログを小さく保つ。既にある信頼できる製品コンポーネントをレンダリングする。

**深刻なアプリ内copilotを構築していて共有状態が必要。**

CopilotKit plus AG-UIを真剣に見る。重要な機能は「チャット」ではない。それは共有状態と双方向対話だ：エージェントは入力を要求し、UIをレンダリングし、状態を更新し、承認のために一時停止できる。

**UIを境界を越えて送信する必要があるリモートエージェントがいる。**

A2UIまたはA2UIのような宣言的プロトコルを使用する。リモートエージェントがデータをとしてUIを記述できる一方、ホストがネイティブレンダリング、セキュリティ、スタイルを制御保持できる。これが要点だ。ライブエージェント/アプリ対話も必要なら、AG-UIまたは環境標準化しているトランスポート上で実行する。

**ChatGPTまたはMCP互換ホスト内で構築している。**

MCP AppsとApps SDKパスを使用する。OpenAIの現在の docsは新しい作業にMCP Apps `ui/*`ブリッジを優先し、`window.openai`を互換性レイヤーとオプションの拡張表面として保持することを推奨している。またデータツールとrenderツールの分割もコピーする：モデルがウィジェットをレンダリングすることを選択する前にデータをfetchしてreasoningさせる。

**自然な言語ダッシュボード、レポート、フォームを自有のアプリにしたい。**

json-render、Hashbrown、またはOpenUIを試す。关键是カタログだ。`LineChart`、`DataTable`、`MetricGroup`、`FilterControl`、`InsightCallout`を公開すれば、モデルは任意のコードに近づくことなく有用なレポート поверхностиをアセンブリできる。

**教育的、視覚的、または高度にカスタムな成果物が欲しい。**

OpenGenerativeUIのようなオープンサンドボックスを使用する。モデルにSVG、Canvas、WebGL、または自己完結型HTMLを書かせ、出力を信頼されていないユーザーコンテンツとして扱う。サンドボックス化し、サイズを設定し、権限を剥离し、特権アプリ状態から遠ざける。

**主にカスタムインライン affordance を伴うより美しいストリーミングマークダウンが必要。**

過構築しない。llm-uiまたはassistant-uiツールレンダリングで十分な場合がある。

## 私が避けるべき間違い

**間違い1：モデルに実行時に本番のReactを書かせる。**

例外はあるが、製品UIにとってこれは通常間違ったデフォルトだ。実行時コード生成は安全に難しく、テスト難しく、テーマ適用難しく、アクセシビリティ維持難しい。モデルが信頼できるコンポーネントから選択して仕事を完了できるなら、そうする。

**間違い2：設計プリミティブ而不是产品プリミティブを露出する。**

モデルに`Row`、`Column`、`Text`、`Button`を与えると、モデルにデザインシステムになってもらっている。モデルは平庸なそれを作成する。より高レベルの製品名詞を与える。

**間違い3：有効なJSONは安全なUIを意味すると思う。**

ペイロードはスキーマ検証を通過しても操 inúmerや危険である可能性がある。ラベルは「請求書を表示」と表示されるが、アクションはアカウントをアーカイブする。UI仕様を動作として扱い、装飾として扱わない。的政策テスト、セマンティックチェック、重要なアクションのための人間の確認が必要だ。

**間違い4：ビジネスロジックをレンダラーツールに入れる。**

レンダラーツールはレンダリングすべき。データツールはfetch、compute、mutate、validateすべき。OpenAIのApps SDK docsはこの分割を理由に呼び出す：すべてのデータツールがウィジェットを拖いていた場合、モデルは提示する前にreasoningする余地を失う。

**間違い5：新奇instead of タスク完了を最適化する。**

ポイントはすべての回答をスノーフレークインターフェースにすることではない。ポイントは摩擦を減らすことだ。ユーザーが4分を節約する安定した Појединаな承認パネルは、2回信頼できない dazzling 生成ダッシュボードよりも良い。

## 実際のアーキテクチャ

今日新产品を開始するなら、段階的アプローチを使用する：

1.  **まず制御されたツールUIを出荷する。** 既知のツールを既知のコンポーネントにマッピングするすべてのツール呼び出し、UIレンダリング、ユーザーアクションをログする。
2.  **ドメインニcatalogを追加する。** パターンが繰り返になったら、`ComparisonTable`、`DecisionPanel`、`DataCollectionForm`、`Timeline`、以及其他製品固有コンポーネントを露出する。
3.  **必要な時のみトランスポート標準化を追加する。** フロントエンドとバックエンドの両方を所有しているなら、plain streamingで十分な場合がある。複数のエージェントフレームワークがあるなら、AG-UIを使用する。ツールが製品境界を越えるなら、MCPを使用する。エージェントが組織境界を越えるなら、A2AとA2UIをwatchする。
4.  **外部または複雑な поверхностиにはiframeウィジェットを使用する。** マップ、カート予約フロー、サードパーティミニアプリは境界の向こうにあるべき。
5.  **オープンな生成は成果物に予約する。** ダイアグラム、シミュレーション、一時的な説明者、視覚的なスクラッチパッドは良いフィット。コアワークフローはそうではない。

アーキテクチャは最終的に次のようになります：

```txt
User intent
  -> agent runtime
  -> tool/data calls
  -> structured result
  -> UI decision
  -> trusted component, declarative spec, or sandboxed widget
  -> user action
  -> state/event stream back to the agent
```

そのループが実際の製品だ。チャットボックスは単なる可能な入力デバイスだ。

## 評価にはUIも含まれるべき

LLMチームはゆっくりとプロンプトとモデル出力を評価することを学んでいる。生成UIは別の поверхностиを追加する：インターフェース自体が間違っている可能性がある。

少なくとも、すべての生成UIのこれらの成果物を保存する：

- プロンプトとツールコンテキスト
- ツール呼び出しとツール結果
- 生成されたUI仕様またはコンポーネント選択
- レンダリングされたコンポーネント名とprops
- ユーザー可见ラベル
- ボタン/フォームに添付されたアクション
- UIからのモデル可见状態更新
- ユーザーアクション履歴

そして次のようなチェックを書く：

- すべての破壊的アクションには確認コンポーネントが必要
- ボタンのラベルはアクションセマンティクスと一致する必要がある
- レンダリング仕様は許可されたコンポーネントのみ参照できる
- ユーザー可见の合計はツール结果の合計と一致する必要がある
- フォームはタスク範囲外のフィールドをリクエストできない
- ウィジェットはモデルのみが使用した秘密を受け取れない
- 非表示メタデータはユーザー可见ラベルと矛盾してはならない

これは面倒そうに聞こえる。それはまた生産的な信頼が生まれる場所だ。

## プロジェクト安定性に関する注意

このスペースのすべての主要なプロトコルは1.0以前だ。最終確認：2026年5月8日；変更を計画し、プラットフォーム賭けの前に現在のdocsを確認する。

**Vercel AI SDK RSC** — 元の「生成UI」ヘッドライン機能は、2024年10月に（RSCoupledなアーキテクチャ上の制限により近時の修正がないため）開発が一時停止した（[Discussion #3251](https://github.com/vercel/ai/discussions/3251)）。**json-render**（Vercel Labs）は代わりにの方向性として出現した：カタログベース、フレームワーク非依存、RSC結合なし。2026年初頭の発売以来、Web開発者の関心を集めたように見える。的可能性のある理由はDX：json-renderは標準的なReactプロジェクトですぐに動作する；A2UIのクロプラットフォームスコープはセットアップの摩擦を追加する。

**A2UI**（Google）は1.0前でマイナーバージョン間の破壊的変更と矛盾したロードマップコミュニケーションを持つ。その利点はjson-renderが_addressしない真のクロプラットフォームリーチ（Web、Flutter、SwiftUI）だ。今日の純粋Webユースケースには、json-renderの方がツールリングカバレッジが良いように見える；クロプラットフォームまたはリモートエージェントシナリオには、A2UIの設計がより適切だ。2つの spec間の収束可能性がある——Vercelはjson-renderからのA2UI互換出力 экспериментしている。

**AG-UI**（CopilotKit）も1.0前だ。最も一般的な混乱は名前だ：AG-UIはトランスポートプロトコルであり、UIフレームワークではない。それはイベントがエージェントとフロントエンドの間で流れる*方法*を定義する；何をレンダリングするかはあなたの選択だ。概念は堅実で広く採用されている。specは 여전히進化している。

## 私の所感

生成UIは慎重に設計された製品インターフェースを置き換えない。それはチャット成績증이AIの普遍的なインターフェースであるという懒い仮定を置き換える。

最高のシステムはモデルにすべてを свободにさせない。それらは小さな、sharpな製品ネイティブのビルドブロック、信頼性の高い 런타임接続、明確なセキュリティ境界線、そしてタスクにインターフェースを適応させるための十分な自由を与える。

未来は「モデルがあなたのフロントエンドを書く」ことではない。

未来は次により近い：**あなたのフロントエンドはエージェントがプレイできる楽器になるが、あなたが楽器に 허용されている音をまだ決定する。**
