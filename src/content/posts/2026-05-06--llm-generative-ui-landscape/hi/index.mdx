Chat तो बस सहारा था।

LLM ऐप्स की पहली पीढ़ी ज़्यादातर एक टेक्स्ट बॉक्स जैसी दिखती थी, जिसे किसी प्रोडक्ट पर अस्थायी रूप से चिपका दिया गया हो। मॉडल गद्य लौटाता था। फ्रंटएंड markdown रेंडर करता था। अगर यूज़र को कोई एक्शन लेना होता, तो असिस्टेंट बताता कि यूज़र को किस बटन पर कहीं और जाकर क्लिक करना चाहिए।

डेमो के लिए यह ठीक था। लेकिन यहीं नहीं रुकना है।

अगला कदम **जनरेटिव UI** है: मॉडल सिर्फ़ टेक्स्ट से जवाब नहीं देता, बल्कि यह भी तय करता है कि यूज़र को इस समय कौन सा इंटरफ़ेस चाहिए। कभी इसका मतलब होता है कि टूल कॉल करके पहले से बने कार्ड को रेंडर किया जाए। कभी किसी ज्ञात वर्कफ़्लो कंपोनेंट में ताज़ा डेटा भरा जाए। कभी एक अस्थायी डैशबोर्ड, फ़ॉर्म, तुलना तालिका, चार्ट, या इंटरैक्टिव विजेट बनाया जाए।

दुर्भाग्य से, "जनरेटिव UI" वो शब्द बन गया है जो नाश्ते से पहले ही पाँच अलग-अलग चीज़ें बताने लगता है।

लोग इसे इन सबका वर्णन करने में इस्तेमाल करते हैं:

- डेवलपर-परिभाषित React कंपोनेंट्स में से मॉडल का चुनाव
- JSON spec जिसे फ्रंटएंड नेटिव कंपोनेंट्स में रेंडर करता है
- MCP टूल से लौटाया गया iframe ऐप
- टूल कॉल्स को सपोर्ट करने वाली chat UI लाइब्रेरी
- बैकएंड और फ्रंटएंड के बीच state स्ट्रीम करने वाला agent protocol
- v0, Lovable, Bolt, या Cursor जैसा design-time code generator
- रनटाइम पर सीधे HTML, SVG, Canvas, या React लिखता हुआ मॉडल

ये सब जुड़े हुए हैं, लेकिन एक ही परत नहीं हैं। अगर इन्हें एक साथ घुमा दोगे, तो हर आर्किटेक्चर बातचीत अस्त-व्यस्त हो जाएगी।

यह वह मानचित्र है जो मैं चाहता था कि मुझे तब मिलता जब मैंने पहली बार वर्तमान स्टैक की तुलना शुरू की थी।

## मुख्य ग़लतफ़हमी

सबसे बड़ी ग़लती "जनरेटिव UI" को एक ही टेक्नोलॉजी चुनाव मान लेना है।

इसे बेहतर है कि समस्या को चार परतों में बाँटा जाए:

1.  **प्रोडक्ट शेल**: वह चीज़ जिसे यूज़र छूता है। यह chat, sidebar copilot, dashboard, workflow builder, IDE panel, ChatGPT ऐप, mobile screen, या support console हो सकता है।
2.  **UI कम्पोज़िशन मॉडल**: वह व्याकरण जिसे मॉडल बोलने की अनुमति है। यह tool calls, JSON, A2UI, json-render, OpenUI Lang, Hashbrown component selection, या sandboxed HTML हो सकता है।
3.  **रनटाइम और ट्रांसपोर्ट**: संदेश, टूल कॉल्स, state deltas, यूज़र एक्शन, और UI artifacts एजेंट और फ्रंटएंड के बीच कैसे जाते हैं। AG-UI, MCP Apps, Apps SDK, A2A, SSE, WebSockets, और सादा HTTP सब यहीं रहते हैं।
4.  **एजेंट और टूल बैकएंड**: LangGraph, Google ADK, CrewAI, Mastra, LlamaIndex, Pydantic AI, Agno, OpenAI Agents SDK, custom functions, databases, retrieval, और सारा वह उबाऊ बिज़नेस लॉजिक जो अब भी सही होना ज़रूरी है।

एक बार जब आप परतों को अलग कर देते हैं, तो इकोसिस्टम बहुत कम रहस्यमय लगने लगता है।

[AG-UI](https://github.com/ag-ui-protocol/ag-ui) वास्तव में [A2UI](https://github.com/google/A2UI) का प्रतिस्पर्धी नहीं है। AG-UI agent-to-application interaction के लिए एक event protocol है। A2UI एक declarative UI format है जो एजेंट भेज सकता है। आप AG-UI के ऊपर A2UI चला सकते हैं। आप custom tool-rendered कंपोनेंट्स भी AG-UI के ऊपर चला सकते हैं।

[json-render](https://github.com/vercel-labs/json-render) कोई chat प्रोडक्ट नहीं है। यह एक component catalog और renderer architecture है: वे कंपोनेंट्स परिभाषित करें जिनका मॉडल उपयोग कर सकता है, मॉडल को एक वैध JSON tree emit करवाएं, और उस tree को सुरक्षित रूप से रेंडर करें।

[CopilotKit](https://github.com/CopilotKit/CopilotKit) सिर्फ़ एक chat bubble नहीं है। यह agent-native ऐप्स के लिए एक frontend stack है: chat UI, generative UI, shared state, frontend tools, और human-in-the-loop flows।

[OpenAI Apps SDK](https://developers.openai.com/apps-sdk) और [MCP Apps](https://github.com/MCP-UI-Org/mcp-ui) "मेरा React ऐप dynamic बनाओ" वाले टूल नहीं हैं। ये ChatGPT या अन्य MCP-compatible hosts के अंदर widgets रेंडर करने के लिए host integration models हैं।

नाम भ्रमित करने वाले हैं क्योंकि यह क्षेत्र नया है। परतें वह हिस्सा हैं जो काम का रहता है।

## कंट्रोल स्पेक्ट्रम

जनरेटिव UI **डेवलपर कंट्रोल** और **एजेंट फ़्रीडम** के बीच एक ट्रेड-ऑफ़ है।

बहुत ज़्यादा कंट्रोल और असिस्टेंट एक कॉस्ट्यूम पहने हुए command palette जैसा लगने लगता है। बहुत ज़्यादा फ़्रीडम और मॉडल अजीब layout, अस्पष्ट बटन, टूटी हुई visual hierarchy, impossible states, और सुरक्षा समस्याएँ एक आत्मविश्वासी मुस्कान के साथ बनाने लगता है।

ट्रिक यह है कि सबसे कम फ़्रीडम चुनी जाए जो यूज़र की समस्या हल कर सके।

![टूल-रेंडर्ड कंपोनेंट्स से लेकर ओपन-एंडेड जनरेटेड HTML तक का स्पेक्ट्रम](./control-spectrum.webp)

मैं इस स्पेक्ट्रम को इस तरह सोचता हूँ:

**टूल-टू-कंपोनेंट रेंडरिंग** सबसे सुरक्षित default है। मॉडल `get_weather`, `search_products`, `compare_plans`, या `draft_invoice` कॉल करता है। ऐप उस टूल result को आपके पहले से मौजूद कंपोनेंट से मैप करता है: `WeatherCard`, `ProductGrid`, `PlanComparison`, `InvoiceReview`। मॉडल तय करता है कि UI *कब* उपयोगी है। डेवलपर अब भी layout, styling, accessibility, loading states, empty states, और खतरनाक एक्शन को कंट्रोल करते हैं।

यह वही पैटर्न है जो [Vercel AI SDK के generative UI guide](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces) में documented है: मॉडल टूल कॉल करता है, टूल डेटा लौटाता है, और UI result से एक कंपोनेंट रेंडर करता है। यह कई CopilotKit और assistant-ui implementations के पीछे का mental model भी है।

**डिक्लेरेटिव कंपोनेंट कैटलॉग** मॉडल को ज़्यादा जगह देते हैं। एक कंपोनेंट चुनने के बजाय, मॉडल अनुमतित भागों से एक tree compose करता है। एक कैटलॉग में `Metric`, `Table`, `Chart`, `FilterBar`, `ApprovalPanel`, और `Timeline` शामिल हो सकते हैं। मॉडल एक dashboard या workflow step assemble कर सकता है, लेकिन यह arbitrary code execute नहीं कर सकता। यहाँ [A2UI](https://github.com/google/A2UI), [json-render](https://github.com/vercel-labs/json-render), [Hashbrown](https://github.com/liveloveapp/hashbrown), और [OpenUI](https://github.com/thesysdev/openui) रहते हैं।

**Iframe mini-apps** तब समझ में आते हैं जब UI को कंपोनेंट tree से समृद्ध होने की ज़रूरत हो, या जब किसी remote tool provider के पास अनुभव का स्वामित्व हो। MCP Apps और OpenAI Apps SDK एक टूल को structured data plus एक widget resource लौटाने देते हैं जिसे host iframe में रेंडर करता है। यह maps, shopping carts, booking flows, charts, और external product surfaces के लिए शक्तिशाली है। यह host ऐप और widget के बीच एक कठिन boundary भी बनाता है।

**ओपन-एंडेड जनरेशन** दूर का छोर है: एजेंट HTML, SVG, Canvas, WebGL, या अन्य code-like artifacts को sandbox में emit करता है। [OpenGenerativeUI](https://github.com/CopilotKit/OpenGenerativeUI) सबसे बेहतर वर्तमान उदाहरण है: एजेंट sandboxed iframes के अंदर algorithm visualizations, 3D scenes, diagrams, और simulations जनरेट कर सकता है। यह one-off visual explanations के लिए बढ़िया है। यह वह जगह नहीं है जहाँ से मैं एक enterprise approval flow शुरू करूँगा।

यहाँ मुख्य अंतर को नाम देना मददगार है: **iframe HTML** (मॉडल sandbox में code लिखता है) बनाम **JSON catalog** (मॉडल एक structured spec emit करता है और आपका renderer इसे pre-built कंपोनेंट्स में मैप करता है)। ये सुनने में related लगते हैं लेकिन बहुत अलग risk और complexity profiles लेकर आते हैं। Iframe HTML अधिकतम expressive है; iframe boundary सुरक्षा का काम करती है। JSON catalog मॉडल को executable freedom नहीं देता — यह सिर्फ़ आपके पहले से परिभाषित component types reference कर सकता है। इस क्षेत्र में अधिकांश frameworks स्पष्ट रूप से एक या दूसरे camp में गिरते हैं।

**Sandbox से परे**: बहुत हालिया demos से पता चलता है कि एक चौथा mode बन रहा है — LLMs game-like या immersive experiences को सीधे control कर रहे हैं, जो किसी भी component catalog की अनुमति से कहीं ज़्यादा visual output control करते हैं। Prompts से explorable 3D worlds जनरेट करने वाले projects, रनटाइम पर LLM-directed NPC behavior, और WebGPU के माध्यम से in-browser model inference ([WebLLM](https://mlc.ai/web-llm/)) शुरुआती markers हैं। यहाँ अब तक production work के लिए कोई stable frameworks नहीं हैं। मैं इस direction को एक dedicated article में cover करूँगा जब स्थिति बदलेगी।

## हाई-लेवल कंपोनेंट्स बनाम ग्रेन्युलर कंपोनेंट्स

यह सबसे महत्वपूर्ण design decision है।

अगर आपका catalog बहुत ग्रेन्युलर है, तो मॉडल को frontend engineer बनना पड़ेगा:

```tsx
Container
Row
Column
Text
Button
Icon
Spacer
Divider
```

यह लचीला लगता है, लेकिन अब मॉडल को spacing, hierarchy, grouping, empty states, button labels, error treatment, और responsive behavior तय करना होगा। आपने prompt को भी बड़ा और output को टूटने के लिए आसान बना दिया है।

अगर आपका catalog बहुत हाई-लेवल है, तो मॉडल फँसा रह जाता है:

```tsx
WeatherCard
StockCard
HotelCard
```

यह सुरक्षित है, लेकिन यह केवल ज्ञात scenarios के लिए काम करता है। मॉडल comparison matrix नहीं बना सकता, missing inputs के लिए नहीं पूछ सकता, या यूज़र के सवाल बदलने पर information architecture adapt नहीं कर सकता।

उपयोगी बीच का रास्ता **constrained slots वाले domain-level कंपोनेंट्स** हैं:

```tsx
SearchResults
ComparisonTable
MetricGroup
EditablePlan
ApprovalRequest
Timeline
DataCollectionForm
CheckoutReview
```

ये कंपोनेंट्स product taste और business constraints encode करते हैं। मॉडल तय करता है कि *क्या दिखाया जाए*, लेकिन हर CSS decision नहीं।

उदाहरण के लिए, एक travel agent को `div`, `span`, और `button` की ज़रूरत नहीं है। उसे चाहिए:

- `TripSummary`
- `FlightOptionList`
- `HotelComparison`
- `TravelerForm`
- `PolicyNotice`
- `BookingConfirmation`

एक finance agent को generic charting playground की ज़रूरत नहीं है। उसे चाहिए:

- `PortfolioSnapshot`
- `TransactionTable`
- `RiskBreakdown`
- `ScenarioComparison`
- `ApprovalGate`

Catalog को आपके प्रोडक्ट जैसा सुनना चाहिए, HTML जैसा नहीं।

## फ़ीचर तालिका

यह तालिका जानबूझकर opinionated है। यह हर project को stack में एक tool के रूप में देखती है, winner-take-all platform के रूप में नहीं।

| Technology | Layer | Best fit | UI model | Streaming / state | Notes and examples |
| --- | --- | --- | --- | --- | --- |
| [AG-UI](https://github.com/ag-ui-protocol/ag-ui) | Runtime protocol | एजेंट बैकएंड को फ्रंटएंड ऐप्स से जोड़ना | Messages, tools, state, activity, interrupts के लिए events | Yes; event stream plus state snapshots/deltas | तब उपयोग करें जब आपको एक standard agent-to-app pipe चाहिए। यह MCP और A2A को replace नहीं, बल्कि complement करता है। |
| [A2UI](https://github.com/google/A2UI) | Declarative UI protocol | क्रॉस-प्लेटफ़ॉर्म, agent-generated native UI | Components, data model, और updates का वर्णन करता हुआ JSON payload | Incremental updates के लिए designed | Remote agents और trust boundaries के लिए मज़बूत choice। Early public preview, लेकिन conceptually clean। |
| [json-render](https://github.com/vercel-labs/json-render) | Component catalog and renderer | मॉडल को approved कंपोनेंट्स compose करने देना | Typed catalog द्वारा constrained JSON tree | Progressive rendering सपोर्ट करता है | React, Vue, Svelte, Solid, React Native, email, PDF, Remotion, terminal, और अन्य के लिए अच्छा। |
| [CopilotKit](https://github.com/CopilotKit/CopilotKit) | Product shell and agent UI framework | In-app copilots, shared state, frontend tools, HITL | Tool rendering, AG-UI, A2UI, MCP Apps patterns | Yes | "Build agent-native apps" stacks में से one of the broadest। [generative-ui examples](https://github.com/CopilotKit/generative-ui) देखें। |
| [OpenGenerativeUI](https://github.com/CopilotKit/OpenGenerativeUI) | Open-ended UI generation showcase | Visual explanations, diagrams, simulations, charts | Agent HTML / SVG / Canvas को sandboxed iframes में emit करता है | Progressive visual rendering | तब उपयोग करें जब dynamic artifacts के लिए fixed component catalog बहुत सीमित हो। |
| [MCP Apps / mcp-ui](https://github.com/MCP-UI-Org/mcp-ui) | Host/widget standard | MCP के माध्यम से interactive UI लौटाने वाले tool providers | Tool metadata से linked HTML resource | Host bridge और widget actions | तब best जब UI किसी tool provider का हो या iframe isolation की ज़रूरत हो। |
| [OpenAI Apps SDK](https://developers.openai.com/apps-sdk) | ChatGPT app host integration | Custom ChatGPT app widgets बनाना | MCP server tools plus iframe UI components | Tool input/result, widget state, follow-up messages | नए ChatGPT apps को MCP Apps fields और `ui/*` bridge को preference देनी चाहिए, compatibility/extensions के लिए `window.openai` रखते हुए। |
| [Vercel AI SDK UI](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces) | App SDK and chat state | Custom app chat, tool calls, streaming message parts | Tool results को React components के रूप में render करना | Yes, `useChat` और UI message streams के माध्यम से | अच्छा baseline अगर आप पहले से ऐप own करते हैं और lower-level control चाहते हैं। UI primitives के लिए [AI Elements](https://elements.ai-sdk.dev/) के साथ pair करें। |
| [assistant-ui](https://github.com/assistant-ui/assistant-ui) | React chat primitives | Production chat UX custom rendering के साथ | Composable chat primitives, tool call rendering, JSON as components | Yes | तब मज़बूत जब आपको polished chat ergonomics चाहिए लेकिन अपना backend लाना है। |
| [LangGraph Generative UI](https://docs.langchain.com/langgraph-platform/generative-ui-react) | Agent platform integration | Graph code के साथ UI components co-locate करना | Graph named UI messages emit करता है जिन्हें React components render करते हैं | Yes, custom stream events सहित | LangGraph deployments और graph-owned UI components के लिए natural fit। |
| [Hashbrown](https://github.com/liveloveapp/hashbrown) | Frontend GenUI framework | Components और client-side tools expose करने वाले React/Angular apps | LLM allowed app components चुनता और render करता है | Streaming patterns सपोर्ट करता है | सीधे product surfaces में intelligence embed करने के लिए अच्छा, केवल chat नहीं। |
| [OpenUI](https://github.com/thesysdev/openui) | Compact UI language and runtime | JSON से कम tokens में streamable model-generated UI | OpenUI Lang plus React runtime और component libraries | Token streaming के लिए designed | तब interesting जब JSON verbosity bottleneck बन जाए। अभी young, लेकिन देखने लायक। |
| [Tambo](https://github.com/tambo-ai/tambo) | React generative UI SDK | Component selection, stateful components, client-side tool execution | AI components चुनता है और client tools के साथ interact करता है | State-oriented | Automatic component orchestration पर focused लोकप्रिय OSS React option। |
| [llm-ui](https://llm-ui.com/) | Output renderer | Custom inline components के साथ smoother LLM text output | Model output strings को React rendering में parse करता है | Smooth token rendering | Text streams के अंदर हल्के custom components के लिए उपयोगी; पूरा agent UI protocol नहीं। |
| AI SDK RSC / React Server Components | Older pattern / framework feature | Next.js में server-rendered component streams | Model/tool flow server-rendered UI लौटाता है | Yes, लेकिन framework-specific | अक्टूबर 2024 में development रोक दी गई ([Discussion #3251](https://github.com/vercel/ai/discussions/3251)); recommended path नहीं। `useObject` या json-render पर migrate करें। |

## किस प्रोडक्ट के लिए क्या उपयोग करें

यह वह recommendation matrix है जो मैं वास्तव में एक team के साथ उपयोग करूँगा।

**आप एक मौजूदा SaaS ऐप में assistant जोड़ रहे हैं।**

Tool-to-component rendering से शुरू करें। [Vercel AI SDK UI](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces), [assistant-ui](https://github.com/assistant-ui/assistant-ui), या [CopilotKit](https://github.com/CopilotKit/CopilotKit) का उपयोग करें, इस पर निर्भर करते हुए कि आपको कितना agent state और frontend tool integration चाहिए। शुरुआत में catalog छोटा रखें। वे product components render करें जिन पर आप पहले से भरोसा करते हैं।

**आप एक गंभीर in-app copilot बना रहे हैं जिसे shared state की ज़रूरत है।**

CopilotKit plus AG-UI पर ध्यान से देखें। महत्वपूर्ण feature "chat" नहीं है। यह shared state और bidirectional interaction है: agent input माँग सकता है, UI render कर सकता है, state update कर सकता है, और approval के लिए रुक सकता है।

**आपके पास remote agents हैं जिन्हें UI boundary के पार भेजना होता है।**

A2UI या A2UI-like declarative protocol का उपयोग करें। पूरा मुद्दा यह है कि एक remote agent UI को data के रूप में describe कर सकता है जबकि host native rendering, security, और style का नियंत्रण रखता है। अगर आपको live agent/app interaction भी चाहिए, तो इसे AG-UI या जो भी transport आपका environment standardize करता है उसके ऊपर चलाएं।

**आप ChatGPT या MCP-compatible host के अंदर बना रहे हैं।**

MCP Apps और Apps SDK path का उपयोग करें। OpenAI की current docs नए काम के लिए MCP Apps `ui/*` bridge को recommend करती हैं, compatibility layer और optional extension surface के रूप में `window.openai` रखते हुए। उनके data tools और render tools के बीच के split को भी copy करें: मॉडल को डेटा fetch और reason करने दें, इससे पहले कि वह widget render करने का चुनाव करे।

**आप अपनी खुद की ऐप में natural-language dashboards, reports, या forms चाहते हैं।**

json-render, Hashbrown, या OpenUI आजमाएं। कुंजी catalog है। अगर आप `LineChart`, `DataTable`, `MetricGroup`, `FilterControl`, और `InsightCallout` expose करते हैं, तो मॉडल useful reporting surfaces assemble कर सकता है बिना arbitrary code के पास जाए।

**आपको educational, visual, या highly bespoke artifacts चाहिए।**

OpenGenerativeUI जैसा open-ended sandbox उपयोग करें। मॉडल को SVG, Canvas, WebGL, या self-contained HTML लिखने दें, लेकिन output को untrusted user content की तरह treat करें। उसे sandbox करें, size करें, permissions strip करें, और उसे privileged app state से दूर रखें।

**आपको मुख्य रूप से prettier streaming markdown चाहिए कुछ inline affordances के साथ।**

Overbuild न करें। llm-ui या assistant-ui tool rendering शायद काफ़ी है।

## गलतियाँ जिनसे मैं बचूँगा

**गलती 1: मॉडल को रनटाइम पर production React लिखने देना।**

अपवाद हैं, लेकिन product UI के लिए यह आमतौर पर गलत default है। रनटाइम code generation secure करना कठिन है, test करना कठिन है, theme करना कठिन है, और accessible रखना कठिन है। अगर मॉडल trusted components चुनकर काम कर सकता है, तो वही करें।

**गलती 2: Design primitives expose करना product primitives के बजाय।**

जब आप मॉडल को `Row`, `Column`, `Text`, और `Button` देते हैं, तो आप उसे अपना design system बनने के लिए कह रहे हैं। यह एक mediocre design system बनेगा। इसे higher-level product nouns दें।

**गलती 3: यह सोचना कि valid JSON का मतलब safe UI है।**

एक payload schema validation pass कर सकता है और फिर भी manipulative या खतरनाक हो सकता है। Label कह सकता है "View invoice" जबकि action account archive कर दे। UI specs को behavior की तरह treat करें, decoration की तरह नहीं। उन्हें policy tests, semantic checks, और consequential actions के लिए human confirmation की ज़रूरत है।

**गलती 4: Render tools में business logic डालना।**

Render tools को render करना चाहिए। Data tools को fetch, compute, mutate, और validate करना चाहिए। OpenAI के Apps SDK docs में इस split का उल्लेख किसी वजह से किया गया है: अगर हर data tool अपने साथ एक widget घसीटता है, तो मॉडल present करने से पहले reason करने की जगह खो देता है।

**गलती 5: Novelty के लिए optimize करना task completion के बजाय।**

मुद्दा यह नहीं है कि हर जवाब एक snowflake interface बने। मुद्दा यह है कि friction कम हो। एक stable, boring approval panel जो यूज़र के चार मिनट बचाता है, एक dazzling generated dashboard से बेहतर है जिस पर दोबारा भरोसा नहीं किया जा सकता।

## एक व्यावहारिक आर्किटेक्चर

अगर मैं आज एक नया प्रोडक्ट शुरू कर रहा होता, तो मैं एक staged approach अपनाता:

1.  **पहले controlled tool UI ship करें।** ज्ञात tools को ज्ञात components से मैप करें। हर tool call, UI render, और user action log करें।
2.  **एक domain catalog जोड़ें।** एक बार patterns दोहराने लगें, तो `ComparisonTable`, `DecisionPanel`, `DataCollectionForm`, `Timeline`, और अन्य product-specific components expose करें।
3.  **Transport standardization तब ही जोड़ें जब ज़रूरत हो।** अगर आप frontend और backend दोनों own करते हैं, तो सादा streaming शायद ठीक है। अगर आपके पास multiple agent frameworks हैं, तो AG-UI का उपयोग करें। अगर tools product boundaries पार करते हैं, तो MCP का उपयोग करें। अगर agents organizational boundaries पार करते हैं, तो A2A और A2UI पर नज़र रखें।
4.  **Iframe widgets का उपयोग करें विदेशी या complex surfaces के लिए।** Maps, carts, booking flows, और third-party mini-apps एक boundary के पीछे belong करते हैं।
5.  **ओपन-एंडेड जनरेशन artifacts के लिए reserve रखें।** Diagrams, simulations, temporary explainers, और visual scratchpads के लिए बढ़िया fit हैं। Core workflows के लिए नहीं।

आर्किटेक्चर अंत में कुछ इस तरह दिखता है:

```txt
User intent
  -> agent runtime
  -> tool/data calls
  -> structured result
  -> UI decision
  -> trusted component, declarative spec, or sandboxed widget
  -> user action
  -> state/event stream back to the agent
```

यह loop असली प्रोडक्ट है। Chat box सिर्फ़ एक possible input device है।

## इवैलुएशन में UI शामिल होना चाहिए

LLM teams धीरे-धीरे prompts और model outputs को eval करना सीख रहे हैं। जनरेटिव UI एक और surface area जोड़ता है: interface itself गलत हो सकता है।

कम से कम, हर generated UI के लिए ये artifacts save करें:

- prompt और tool context
- tool calls और tool results
- generated UI spec या component selection
- rendered component name और props
- user-visible labels
- buttons/forms से जुड़े actions
- UI से model-visible state updates
- user action history

फिर इस तरह के checks लिखें:

- हर destructive action के पास confirmation component होना चाहिए
- button labels को action semantics से मेल खाना चाहिए
- render specs सिर्फ़ allowed components reference कर सकते हैं
- user-visible totals को tool-result totals से मेल खाना चाहिए
- forms task scope के बाहर fields नहीं माँग सकते
- widgets वे secrets नहीं प्राप्त कर सकते जिनकी ज़रूरत सिर्फ़ मॉडल को थी
- hidden metadata visible labels का विरोधाभास नहीं कर सकता

यह tedious लग सकता है। यह production trust का स्रोत भी है।

## प्रोजेक्ट स्थिरता पर एक नोट

इस क्षेत्र में हर major protocol pre-1.0 है। Last verified May 8, 2026; बदलाव के लिए तैयार रहें और platform bet लगाने से पहले current docs check करें।

**Vercel AI SDK RSC** — मूल "Generative UI" headline feature — का development अक्टूबर 2024 में रोक दिया गया ([Discussion #3251](https://github.com/vercel/ai/discussions/3251)) क्योंकि architectural limitations का कोई near-term fix नहीं था। **json-render** (Vercel Labs) replacement direction के रूप में उभरा: catalog-based, framework-agnostic, कोई RSC coupling नहीं। इसे early-2026 launch के बाद से web-developer attention तेज़ी से मिला है। संभावित कारण DX है: json-render तुरंत एक standard React project में काम करता है; A2UI का cross-platform scope setup friction जोड़ता है।

**A2UI** (Google) pre-1.0 है, minor versions के बीच breaking changes हैं, और inconsistent roadmap communications हैं। इसका फ़ायदा genuine cross-platform reach (web, Flutter, SwiftUI) है जिसे json-render address नहीं करता। आज के pure-web use cases के लिए, json-render में बेहतर tooling coverage दिखती है; cross-platform या remote-agent scenarios के लिए, A2UI का design ज़्यादा उपयुक्त है। दोनों specs के बीच convergence संभव है — Vercel ने json-render से A2UI-compatible output के साथ experiment किया है।

**AG-UI** (CopilotKit) भी pre-1.0 है। सबसे common confusion नाम है: AG-UI एक transport protocol है, UI framework नहीं। यह define करता है कि events agent और frontend के बीच *कैसे* flow करते हैं; आप क्या render करते हैं यह अब भी आपका decision है। Concept ठोस है और broadly adopted। Spec अब भी evolve हो रही है।

## मेरा नज़रिया

जनरेटिव UI सावधानी से design किए गए product interfaces को replace नहीं करेगा। यह उस आलसी धारणा को replace करेगा कि chat transcript AI के लिए universal interface है।

सबसे अच्छे systems मॉडल को सब कुछ freestyle नहीं करने देंगे। वे उसे product-native building blocks का एक छोटा, तेज़ सेट; एक reliable runtime connection; स्पष्ट security boundaries; और interface को task के अनुसार adapt करने की enough freedom देंगे।

भविष्य यह नहीं है कि "मॉडल आपका frontend लिखता है।"

भविष्य इसके करीब है: **आपका frontend एक वाद्ययंत्र बन जाता है जिसे agent बजा सकता है, लेकिन आप अब भी तय करते हैं कि वाद्ययंत्र को कैसी ध्वनि निकालने की अनुमति है।**
