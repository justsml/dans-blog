Le chat, c'était les roulettes d'entraînement.

La première génération d'applications LLM ressemblait surtout à une boîte de texte agrafée à un produit. Le modèle renvoyait de la prose. Le frontend rendait du markdown. Si l'utilisateur devait agir, l'assistant décrivait le bouton sur lequel il fallait aller cliquer ailleurs.

C'était bien pour des démos. Ce n'est pas là que cela va.

La prochaine étape utile, c'est **l'interface utilisateur générative** : le modèle ne se contente pas de répondre par du texte ; il aide à décider quelle interface l'utilisateur doit avoir sous les yeux maintenant. Parfois, cela signifie appeler un outil et rendre une carte préconçue. Parfois, cela signifie remplir un composant de workflow connu avec des données fraîches. Parfois, cela signifie composer un tableau de bord temporaire, un formulaire, un tableau comparatif, un graphique ou un widget interactif.

Malheureusement, « interface utilisateur générative » est devenu l'un de ces termes qui signifient cinq choses différentes avant le petit-déjeuner.

Les gens l'utilisent pour décrire :

- un modèle choisissant parmi des composants React définis par le développeur
- une spécification JSON que le frontend transforme en composants natifs
- une application iframe renvoyée par un outil MCP
- une bibliothèque d'interface de chat qui prend en charge les appels d'outils
- un protocole d'agent qui diffuse l'état entre le backend et le frontend
- un générateur de code au moment du design comme v0, Lovable, Bolt ou Cursor
- un modèle écrivant littéralement du HTML, du SVG, du Canvas ou du React à l'exécution

Ce sont des sujets liés, mais ce n'est pas la même couche. Si on les mélange, chaque conversation d'architecture tourne à la soupe.

Voici la carte que j'aurais aimé avoir quand j'ai commencé à comparer la stack actuelle.

## Le malentendu fondamental

La plus grande erreur est de traiter « l'interface utilisateur générative » comme un seul choix technologique.

Il vaut mieux séparer le problème en quatre couches :

1.  **Coquille produit** : ce que l'utilisateur touche. Cela peut être un chat, un copilote en barre latérale, un tableau de bord, un constructeur de workflow, un panneau IDE, une application ChatGPT, un écran mobile ou une console de support.
2.  **Modèle de composition d'interface** : la grammaire que le modèle est autorisé à parler. Cela peut être des appels d'outils, du JSON, A2UI, json-render, OpenUI Lang, la sélection de composants Hashbrown, ou du HTML sandboxé.
3.  **Exécution et transport** : comment les messages, les appels d'outils, les deltas d'état, les actions utilisateur et les artefacts d'interface circulent entre l'agent et le frontend. AG-UI, les applications MCP, Apps SDK, A2A, SSE, WebSockets et le bon vieux HTTP vivent tous autour d'ici.
4.  **Agent et backend d'outils** : LangGraph, Google ADK, CrewAI, Mastra, LlamaIndex, Pydantic AI, Agno, OpenAI Agents SDK, fonctions personnalisées, bases de données, retrieval, et toute la logique métier ennuyeuse qui doit quand même être correcte.

Une fois que vous séparez les couches, l'écosystème devient beaucoup moins mystique.

[AG-UI](https://github.com/ag-ui-protocol/ag-ui) n'est pas vraiment un concurrent d'[A2UI](https://github.com/google/A2UI). AG-UI est un protocole d'événements pour l'interaction agent-application. A2UI est un format d'interface déclaratif que l'agent peut envoyer. Vous pouvez mettre A2UI sur AG-UI. Vous pouvez aussi mettre des composants rendus par des outils personnalisés sur AG-UI.

[json-render](https://github.com/vercel-labs/json-render) n'est pas un produit de chat. C'est une architecture de catalogue de composants et de rendu : définissez les composants que le modèle peut utiliser, faites-lui émettre un arbre JSON valide, et rendez cet arbre en toute sécurité.

[CopilotKit](https://github.com/CopilotKit/CopilotKit) n'est pas juste une bulle de chat. C'est une stack frontend pour les applications natives agent : interface de chat, interface générative, état partagé, outils frontend et flux avec humain dans le circuit.

[OpenAI Apps SDK](https://developers.openai.com/apps-sdk) et [MCP Apps](https://github.com/MCP-UI-Org/mcp-ui) ne sont pas des outils « rendez mon application React dynamique ». Ce sont des modèles d'intégration hôte pour le rendu de widgets dans ChatGPT ou d'autres hôtes compatibles MCP.

Les noms sont confus parce que l'espace est jeune. Les couches sont la partie qui reste utile.

## Le spectre de contrôle

L'interface utilisateur générative est un compromis entre **contrôle du développeur** et **liberté de l'agent**.

Trop de contrôle et l'assistant ressemble à une palette de commandes portant un costume. Trop de liberté et le modèle commence à inventer des mises en page bizarres, des boutons vagues, une hiérarchie visuelle cassée, des états impossibles et des problèmes de sécurité avec un petit sourire confiant.

L'astuce est de choisir la plus petite quantité de liberté qui résout le problème de l'utilisateur.

![Un spectre allant des composants rendus par des outils au HTML généré ouvert](../control-spectrum.webp)

Je pense au spectre comme ceci :

**Le rendu d'outil vers composant** est le défaut le plus sûr. Le modèle appelle `get_weather`, `search_products`, `compare_plans` ou `draft_invoice`. L'application mappe ce résultat d'outil à un composant que vous possédez déjà : `WeatherCard`, `ProductGrid`, `PlanComparison`, `InvoiceReview`. Le modèle décide *quand* l'interface est utile. Les développeurs possèdent toujours la mise en page, le style, l'accessibilité, les états de chargement, les états vides et les actions dangereuses.

C'est le modèle documenté dans le [guide d'interface utilisateur générative du Vercel AI SDK](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces) : le modèle appelle un outil, l'outil renvoie des données, et l'interface rend un composant à partir du résultat. C'est aussi le modèle mental derrière de nombreuses implémentations CopilotKit et assistant-ui.

**Les catalogues de composants déclaratifs** donnent plus de marge au modèle. Au lieu de choisir un composant, le modèle compose un arbre à partir de parties autorisées. Un catalogue peut inclure `Metric`, `Table`, `Chart`, `FilterBar`, `ApprovalPanel` et `Timeline`. Le modèle peut assembler un tableau de bord ou une étape de workflow, mais il ne peut pas exécuter de code arbitraire. C'est là que se situent [A2UI](https://github.com/google/A2UI), [json-render](https://github.com/vercel-labs/json-render), [Hashbrown](https://github.com/liveloveapp/hashbrown) et [OpenUI](https://github.com/thesysdev/openui).

**Les mini-applications iframe** ont du sens quand l'interface doit être plus riche qu'un arbre de composants, ou quand un fournisseur d'outil distant possède l'expérience. Les applications MCP et OpenAI Apps SDK permettent à un outil de renvoyer des données structurées plus une ressource widget que l'hôte rend dans une iframe. C'est puissant pour les cartes, les paniers d'achat, les flux de réservation, les graphiques et les surfaces de produits externes. Cela crée aussi une frontière plus dure entre l'application hôte et le widget.

**La génération ouverte** est l'extrémité opposée : l'agent émet du HTML, du SVG, du Canvas, du WebGL ou d'autres artefacts de type code dans un sandbox. [OpenGenerativeUI](https://github.com/CopilotKit/OpenGenerativeUI) est le meilleur exemple actuel : l'agent peut générer des visualisations d'algorithmes, des scènes 3D, des diagrammes et des simulations dans des iframes sandboxées. C'est génial pour les explications visuelles ponctuelles. Ce n'est pas là où je commencerais pour un flux d'approbation en entreprise.

Il est utile de nommer la distinction clé ici : **HTML en iframe** (le modèle écrit du code dans un sandbox) contre **catalogue JSON** (le modèle émet une spécification structurée et votre rendu la mappe à des composants préconstruits). Cela semble lié, mais cela porte des profils de risque et de complexité très différents. Le HTML en iframe est expressif au maximum ; la frontière de l'iframe fait le travail de sécurité. Le catalogue JSON ne donne aucune liberté exécutable au modèle — il ne peut que référencer des types de composants que vous avez définis à l'avance. La plupart des frameworks dans cet espace tombent clairement dans l'un ou l'autre camp.

**Au-delà du sandbox** : des démos très récentes suggèrent qu'un quatrième mode est en train de se former — les LLM pilotant des expériences de type jeu ou immersives en contrôlant la sortie visuelle plus directement que ne le permet tout catalogue de composants. Des projets générant des mondes 3D explorables à partir de prompts, un comportement de PNJ dirigé par LLM en temps réel et une inférence de modèle dans le navigateur via WebGPU ([WebLLM](https://mlc.ai/web-llm/)) sont des marqueurs précoces. Il n'y a pas encore de frameworks stables pour construire des charges de travail de production ici. Je couvrirai cette direction dans un article dédié une fois que cela changera.

## Composants de haut niveau contre composants granulaires

C'est la décision de conception la plus importante.

Si votre catalogue est trop granulaire, le modèle doit devenir un ingénieur frontend :

```tsx
Container
Row
Column
Text
Button
Icon
Spacer
Divider
```

Ça a l'air flexible, mais maintenant le modèle doit décider de l'espacement, de la hiérarchie, du regroupement, des états vides, des libellés de boutons, du traitement des erreurs et du comportement responsive. Vous avez aussi rendu le prompt plus gros et la sortie plus facile à casser.

Si votre catalogue est trop de haut niveau, le modèle est piégé :

```tsx
WeatherCard
StockCard
HotelCard
```

C'est sûr, mais cela ne fonctionne que pour des scénarios connus. Le modèle ne peut pas faire une matrice de comparaison, demander des entrées manquantes ou adapter l'architecture de l'information quand la question de l'utilisateur change.

Le milieu utile est **les composants de domaine avec des slots contraints** :

```tsx
SearchResults
ComparisonTable
MetricGroup
EditablePlan
ApprovalRequest
Timeline
DataCollectionForm
CheckoutReview
```

Ces composants encodent le goût produit et les contraintes métier. Le modèle peut décider *de ce qui doit être montré*, mais pas de chaque décision CSS.

Par exemple, un agent de voyage n'a pas besoin de `div`, `span` et `button`. Il a besoin de :

- `TripSummary`
- `FlightOptionList`
- `HotelComparison`
- `TravelerForm`
- `PolicyNotice`
- `BookingConfirmation`

Un agent financier n'a pas besoin d'un terrain de jeu graphique générique. Il a besoin de :

- `PortfolioSnapshot`
- `TransactionTable`
- `RiskBreakdown`
- `ScenarioComparison`
- `ApprovalGate`

Le catalogue doit ressembler à votre produit, pas à du HTML.

## Que utiliser pour quel produit

Voici la matrice de recommandation que j'utiliserais réellement avec une équipe.

**Vous ajoutez un assistant à une application SaaS existante.**

Commencez par le rendu d'outil vers composant. Utilisez [Vercel AI SDK UI](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces), [assistant-ui](https://github.com/assistant-ui/assistant-ui) ou [CopilotKit](https://github.com/CopilotKit/CopilotKit), selon la quantité d'état d'agent et d'intégration d'outils frontend dont vous avez besoin. Gardez le catalogue minuscule au début. Rendez les composants produit que vous connaissez déjà.

**Vous construisez un copilote en application sérieux qui a besoin d'état partagé.**

Regardez de près CopilotKit plus AG-UI. La fonctionnalité importante n'est pas le « chat ». C'est l'état partagé et l'interaction bidirectionnelle : l'agent peut demander une entrée, rendre une interface, mettre à jour l'état et faire une pause pour approbation.

**Vous avez des agents distants qui doivent envoyer une interface à travers une frontière.**

Utilisez A2UI ou un protocole déclaratif de type A2UI. L'intérêt principal est qu'un agent distant peut décrire une interface sous forme de données tandis que l'hôte garde le contrôle du rendu natif, de la sécurité et du style. Si vous avez aussi besoin d'une interaction agent/application en direct, faites-le passer par AG-UI ou quel que soit le transport que votre environnement standardise.

**Vous construisez à l'intérieur de ChatGPT ou d'un hôte compatible MCP.**

Utilisez le chemin MCP Apps et Apps SDK. La documentation actuelle d'OpenAI recommande le pont MCP Apps `ui/*` pour les nouveaux travaux tout en gardant `window.openai` comme couche de compatibilité et surface d'extension optionnelle. Copiez aussi leur séparation entre outils de données et outils de rendu : laissez le modèle récupérer et raisonner sur les données avant de choisir de rendre un widget.

**Vous voulez des tableaux de bord, des rapports ou des formulaires en langage naturel dans votre propre application.**

Essayez json-render, Hashbrown ou OpenUI. La clé est le catalogue. Si vous exposez `LineChart`, `DataTable`, `MetricGroup`, `FilterControl` et `InsightCallout`, le modèle peut assembler des surfaces de reporting utiles sans s'approcher de code arbitraire.

**Vous voulez des artefacts éducatifs, visuels ou hautement sur mesure.**

Utilisez un sandbox ouvert comme OpenGenerativeUI. Laissez le modèle écrire du SVG, du Canvas, du WebGL ou du HTML autonome, mais traitez la sortie comme du contenu utilisateur non fiable. Sandboxez-le, dimensionnez-le, supprimez les permissions et gardez-le loin de l'état privilégié de l'application.

**Vous avez surtout besoin d'un streaming markdown plus joli avec quelques affords en ligne.**

Ne surconstruisez pas. llm-ui ou le rendu d'outils assistant-ui pourrait suffire.

## Les erreurs que j'éviterais

**Erreur 1 : Laisser le modèle écrire du React de production à l'exécution.**

Il y a des exceptions, mais pour l'interface produit, c'est généralement le mauvais défaut. La génération de code à l'exécution est difficile à sécuriser, difficile à tester, difficile à thématiser et difficile à garder accessible. Si le modèle peut faire le travail en choisissant parmi des composants de confiance, faites cela.

**Erreur 2 : Exposer des primitives de design au lieu de primitives de produit.**

Quand vous donnez au modèle `Row`, `Column`, `Text` et `Button`, vous lui demandez de devenir votre design system. Il deviendra un médiocre. Donnez-lui des noms de produit de plus haut niveau.

**Erreur 3 : Penser que du JSON valide signifie une interface sûre.**

Une charge utile peut passer la validation de schéma et rester manipulatrice ou dangereuse. Le libellé peut dire « Voir la facture » tandis que l'action archive le compte. Traitez les spécifications d'interface comme du comportement, pas comme de la décoration. Elles ont besoin de tests de politique, de vérifications sémantiques et de confirmation humaine pour les actions conséquentes.

**Erreur 4 : Mettre la logique métier dans les outils de rendu.**

Les outils de rendu doivent rendre. Les outils de données doivent récupérer, calculer, muter et valider. La documentation d'OpenAI Apps SDK souligne cette séparation pour une raison : si chaque outil de données traîne un widget avec lui, le modèle perd de la marge pour raisonner avant de présenter.

**Erreur 5 : Optimiser pour la nouveauté au lieu de l'achèvement de tâche.**

Le but n'est pas de faire de chaque réponse une interface flocon de neige. Le but est de réduire la friction. Un panneau d'approbation stable et ennuyeux qui fait économiser quatre minutes à l'utilisateur vaut mieux qu'un tableau de bord généré éblouissant auquel on ne peut pas faire confiance deux fois.

## Une architecture pratique

Si je devais commencer un nouveau produit aujourd'hui, j'utiliserais une approche par étapes :

1.  **Livrez d'abord l'interface d'outil contrôlée.** Mappez les outils connus aux composants connus. Journalisez chaque appel d'outil, rendu d'interface et action utilisateur.
2.  **Ajoutez un catalogue de domaine.** Une fois que les motifs se répètent, exposez `ComparisonTable`, `DecisionPanel`, `DataCollectionForm`, `Timeline` et d'autres composants spécifiques au produit.
3.  **Ajoutez la standardisation du transport seulement quand c'est nécessaire.** Si vous possédez le frontend et le backend, le streaming simple peut suffire. Si vous avez plusieurs frameworks d'agent, utilisez AG-UI. Si les outils traversent les frontières de produit, utilisez MCP. Si les agents traversent les frontières organisationnelles, surveillez A2A et A2UI.
4.  **Utilisez des widgets iframe pour les surfaces étrangères ou complexes.** Les cartes, les paniers, les flux de réservation et les mini-applications tierces appartiennent derrière une frontière.
5.  **Réservez la génération ouverte pour les artefacts.** Les diagrammes, les simulations, les explications temporaires et les brouillons visuels sont un excellent choix. Les flux de travail principaux ne le sont pas.

L'architecture finit par ressembler à ceci :

```txt
Intention utilisateur
  -> exécution de l'agent
  -> appels d'outils/données
  -> résultat structuré
  -> décision d'interface
  -> composant de confiance, spécification déclarative ou widget sandboxé
  -> action utilisateur
  -> flux d'événements/état vers l'agent
```

Ce cycle est le vrai produit. La boîte de chat n'est qu'un périphérique d'entrée possible.

## L'évaluation doit inclure l'interface

Les équipes LLM apprennent lentement à évaluer les prompts et les sorties de modèle. L'interface utilisateur générative ajoute une autre surface : l'interface elle-même peut être fausse.

Au minimum, sauvez ces artefacts pour chaque interface générée :

- prompt et contexte d'outil
- appels d'outils et résultats d'outils
- spécification d'interface générée ou sélection de composant
- nom et props du composant rendu
- libellés visibles par l'utilisateur
- actions attachées aux boutons/formulaires
- mises à jour d'état visibles par le modèle depuis l'interface
- historique des actions utilisateur

Ensuite, écrivez des vérifications comme :

- chaque action destructive doit avoir un composant de confirmation
- les libellés de boutons doivent correspondre à la sémantique de l'action
- les spécifications de rendu ne peuvent référencer que des composants autorisés
- les totaux visibles par l'utilisateur doivent correspondre aux totaux des résultats d'outils
- les formulaires ne peuvent pas demander de champs hors du scope de la tâche
- les widgets ne peuvent pas recevoir de secrets dont seul le modèle avait besoin
- les métadonnées cachées ne doivent pas contredire les libellés visibles

Cela semble fastidieux. C'est aussi là que vient la confiance en production.

## Une note sur la stabilité des projets

Chaque protocole majeur dans cet espace est pré-1.0. Dernière vérification le 8 mai 2026 ; prévoyez du changement et consultez la documentation actuelle avant de faire un pari de plateforme.

**Vercel AI SDK RSC** — la fonctionnalité « Generative UI » originale — a vu son développement mis en pause en octobre 2024 ([Discussion #3251](https://github.com/vercel/ai/discussions/3251)) en raison de limitations architecturales qui n'avaient pas de correction à court terme. **json-render** (Vercel Labs) a émergé comme la direction de remplacement : basé sur un catalogue, agnostique au framework, pas de couplage RSC. Il semble avoir rapidement gagné l'attention des développeurs web depuis son lancement début 2026. La raison probable est le DX : json-render fonctionne immédiatement dans un projet React standard ; l'étendue multiplateforme d'A2UI ajoute des frictions de configuration.

**A2UI** (Google) est pré-1.0 avec des changements cassants entre versions mineures et des communications de roadmap incohérentes. Son avantage est une véritable portée multiplateforme (web, Flutter, SwiftUI) que json-render n'aborde pas. Pour les cas d'usage purement web aujourd'hui, json-render semble avoir une meilleure couverture d'outillage ; pour les scénarios multiplateformes ou d'agents distants, la conception d'A2UI est plus appropriée. Une convergence entre les deux spécifications est possible — Vercel a expérimenté une sortie compatible A2UI depuis json-render.

**AG-UI** (CopilotKit) est aussi pré-1.0. La confusion la plus courante est le nom : AG-UI est un protocole de transport, pas un framework d'interface. Il définit *comment* les événements circulent entre l'agent et le frontend ; ce que vous rendez reste votre choix. Le concept est solide et largement adopté. La spécification est encore en évolution.

## Mon avis

L'interface utilisateur générative ne remplacera pas les interfaces produit soigneusement conçues. Elle remplacera l'hypothèse paresseuse qu'une transcription de chat est l'interface universelle pour l'IA.

Les meilleurs systèmes ne laisseront pas le modèle improviser tout. Ils lui donneront un petit ensemble pointu de blocs de construction natifs du produit ; une connexion d'exécution fiable ; des frontières de sécurité claires ; et assez de liberté pour adapter l'interface à la tâche.

Le futur n'est pas « le modèle écrit votre frontend. »

Le futur est plus proche de : **votre frontend devient un instrument que l'agent peut jouer, mais vous décidez toujours de ce que l'instrument est autorisé à sonner.**
