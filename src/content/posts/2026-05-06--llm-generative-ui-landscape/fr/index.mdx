Le chat était les roulettes d'entraînement.

La première génération d'applications LLM ressemblait majoritairement à une boîte de texte collée à un produit. Le modèle renvoyait de la prose. Le frontend rendait du markdown. Si l'utilisateur devait agir, l'assistant décrivait le bouton que l'utilisateur devrait aller cliquer ailleurs.

C'était bien pour des démos. Ce n'est pas là que cela va.

La prochaine étape utile est **l'interface générative** : le modèle ne répond pas seulement par du texte ; il aide à décider de quelle interface l'utilisateur a besoin maintenant. Parfois cela signifie appeler un outil et rendre une carte préconstruite. Parfois cela signifie remplir un composant de flux connu avec des données fraîches. Parfois cela signifie composer un tableau de bord temporaire, un formulaire, un tableau comparatif, un graphique ou un widget interactif.

Malheureusement, « interface générative » est devenu l'une de ces phrases qui signifie cinq choses différentes avant le petit-déjeuner.

Les gens l'utilisent pour décrire :

- un modèle choisissant parmi des composants React définis par les développeurs
- un spec JSON que le frontend rend en composants natifs
- une application iframe renvoyée depuis un outil MCP
- une interface de chat qui supporte les appels d'outils
- un protocole d'agent qui diffuse l'état entre backend et frontend
- un générateur de code au moment de la conception comme v0, Lovable, Bolt, ou Cursor
- un modèle écrivant littéralement du HTML, SVG, Canvas, ou React à l'exécution

Ce sont des choses liées, mais ce ne sont pas la même couche. Si vous les confondez, chaque conversation d'architecture devient une soupe.

Voici la carte que j'aurais voulu avoir quand j'ai commencé à comparer la pile actuelle.

## Le Malentendu Fondamental

La plus grande erreur est de traiter « l'interface générative » comme un choix technologique unique.

Il vaut mieux séparer le problème en quatre couches :

1.  **Coquille produit** : la chose que les utilisateurs touchent. Cela peut être un chat, un copilote latéral, un tableau de bord, un constructeur de flux, un panneau IDE, une application ChatGPT, un écran mobile, ou une console de support.
2.  **Modèle de composition UI** : la grammaire que le modèle est autorisé à parler. Cela peut être des appels d'outils, du JSON, A2UI, json-render, OpenUI Lang, la sélection de composants Hashbrown, ou du HTML sandboxé.
3.  **Runtime et transport** : comment les messages, appels d'outils, deltas d'état, actions utilisateur, et artefacts UI circulent entre l'agent et le frontend. AG-UI, MCP Apps, Apps SDK, A2A, SSE, WebSockets, et le HTTP simple vivent tous par ici.
4.  **Backend agent et outils** : LangGraph, Google ADK, CrewAI, Mastra, LlamaIndex, Pydantic AI, Agno, OpenAI Agents SDK, fonctions personnalisées, bases de données, retrieval, et toute la logique métier ennuyeuse qui doit quand même être correcte.

Une fois que vous séparez les couches, l'écosystème devient beaucoup moins mystique.

[AG-UI](https://github.com/ag-ui-protocol/ag-ui) n'est pas vraiment un concurrent de [A2UI](https://github.com/google/A2UI). AG-UI est un protocole d'événements pour l'interaction agent-application. A2UI est un format UI déclaratif que l'agent peut envoyer. Vous pouvez mettre A2UI sur AG-UI. Vous pouvez aussi mettre des composants rendus par outils personnalisés sur AG-UI.

[json-render](https://github.com/vercel-labs/json-render) n'est pas un produit de chat. C'est une architecture de catalogue de composants et de renderer : définissez les composants que le modèle peut utiliser, faites émettre au modèle un arbre JSON valide, et rendez cet arbre en toute sécurité.

[CopilotKit](https://github.com/CopilotKit/CopilotKit) n'est pas qu'une bulle de chat. C'est une pile frontend pour les applications natives d'agent : interface de chat, interface générative, état partagé, outils frontend, et flux humain-dans-la-boucle.

[OpenAI Apps SDK](https://developers.openai.com/apps-sdk) et [MCP Apps](https://github.com/MCP-UI-Org/mcp-ui) ne sont pas des outils « rendez votre application React dynamique ». Ce sont des modèles d'intégration hôte pour rendre des widgets dans ChatGPT ou d'autres hôtes compatibles MCP.

Les noms sont confus parce que l'espace est jeune. Les couches sont la partie qui reste utile.

## Le Spectre du Contrôle

L'interface générative est un compromis entre **contrôle développeur** et **liberté de l'agent**.

Trop de contrôle et l'assistant ressemble à une palette de commandes portant un déguisement. Trop de liberté et le modèle commence à inventer une mise en page bizarre, des boutons vagues, une hiérarchie visuelle cassée, des états impossibles, et des problèmes de sécurité avec un petit sourire confiant.

L'astuce est de choisir la plus petite quantité de liberté qui résout le problème utilisateur.

![Un spectre du rendu de composants par outils à l'HTML généré de manière ouverte](../control-spectrum.webp)

Je pense au spectre comme ceci :

**Le rendu de composants par outil** est le défaut le plus sûr. Le modèle appelle `get_weather`, `search_products`, `compare_plans`, ou `draft_invoice`. L'application mappe ce résultat d'outil vers un composant que vous possédez déjà : `WeatherCard`, `ProductGrid`, `PlanComparison`, `InvoiceReview`. Le modèle décide *quand* l'UI est utile. Les développeurs possèdent toujours la mise en page, le style, l'accessibilité, les états de chargement, les états vides, et les actions dangereuses.

C'est le modèle documenté dans [le guide d'interface générative du Vercel AI SDK](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces) : le modèle appelle un outil, l'outil renvoie des données, et l'UI rend un composant à partir du résultat. C'est aussi le modèle mental derrière beaucoup d'implémentations CopilotKit et assistant-ui.

**Les catalogues de composants déclaratifs** donnent plus de marge au modèle. Au lieu de choisir un seul composant, le modèle compose un arbre à partir de parties autorisées. Un catalogue peut inclure `Metric`, `Table`, `Chart`, `FilterBar`, `ApprovalPanel`, et `Timeline`. Le modèle peut assembler un tableau de bord ou une étape de flux, mais il ne peut pas exécuter de code arbitraire. C'est là que [A2UI](https://github.com/google/A2UI), [json-render](https://github.com/vercel-labs/json-render), [Hashbrown](https://github.com/liveloveapp/hashbrown), et [OpenUI](https://github.com/thesysdev/openui) se situent.

**Les mini-applications iframe** ont du sens quand l'UI a besoin d'être plus riche qu'un arbre de composants, ou quand un fournisseur d'outil distant possède l'expérience. MCP Apps et OpenAI Apps SDK permettent à un outil de renvoyer des données structurées plus une ressource widget que l'hôte rend dans une iframe. C'est puissant pour les cartes, les paniers d'achat, les flux de réservation, les graphiques, et les surfaces produit externes. Cela crée aussi une frontière plus dure entre l'application hôte et le widget.

**La génération ouverte** est l'extrémité lointaine : l'agent émet du HTML, SVG, Canvas, WebGL, ou d'autres artefacts de type code dans un sandbox. [OpenGenerativeUI](https://github.com/CopilotKit/OpenGenerativeUI) est le meilleur exemple actuel : l'agent peut générer des visualisations d'algorithmes, des scènes 3D, des diagrammes, et des simulations dans des iframes sandboxées. C'est bien pour des explications visuelles ponctuelles. Ce n'est pas là que je commencerais pour un flux d'approbation d'entreprise.

Il aide de nommer la distinction clé ici : **HTML iframe** (le modèle écrit du code dans un sandbox) vs **catalogue JSON** (l'agent émet un spec structuré et votre renderer le mappe vers des composants pré-construits). Ces choses sonnent liées mais portent des profils de risque et de complexité très différents. Le HTML iframe est maximalement expressif ; la frontière iframe fait le travail de sécurité. Le catalogue JSON ne donne aucune liberté exécutable au modèle — il ne peut que référencer des types de composants que vous avez définis à l'avance. La plupart des frameworks dans cet espace tombent clairement dans l'un ou l'autre camp.

**Au-delà du sandbox** : des démos très récentes suggèrent qu'un quatrième mode se forme — des LLM pilotant des expériences de type jeu ou immersives en contrôlant la sortie visuelle plus directement que ne le permet aucun catalogue de composants. Des projets générant des mondes 3D explorables à partir de prompts, un comportement NPC dirigé par LLM à l'exécution, et une inférence modèle dans le navigateur via WebGPU ([WebLLM](https://mlc.ai/web-llm/)) sont des marqueurs précoces. Il n'y a pas encore de frameworks stables pour construire des travaux de production ici. Je couvrirai cette direction dans un article dédié une fois que cela changera.

## Composants de Haut Niveau vs Composants Granulaires

Voilà la décision de conception la plus importante.

Si votre catalogue est trop granulaire, le modèle doit devenir un ingénieur frontend :

```tsx
Container
Row
Column
Text
Button
Icon
Spacer
Divider
```

Cela semble flexible, mais maintenant le modèle doit décider de l'espacement, de la hiérarchie, du regroupement, des états vides, des libellés de boutons, du traitement des erreurs, et du comportement responsive. Vous avez aussi rendu le prompt plus grand et la sortie plus facile à casser.

Si votre catalogue est trop haut niveau, le modèle est piégé :

```tsx
WeatherCard
StockCard
HotelCard
```

C'est sûr, mais cela ne fonctionne que pour des scénarios connus. Le modèle ne peut pas faire une matrice de comparaison, demander des entrées manquantes, ou adapter l'architecture de l'information quand la question de l'utilisateur change.

Le milieu utile est **les composants de niveau domaine avec des créneaux contraints** :

```tsx
SearchResults
ComparisonTable
MetricGroup
EditablePlan
ApprovalRequest
Timeline
DataCollectionForm
CheckoutReview
```

Ces composants intègrent le goût produit et les contraintes métier. Le modèle décide *quoi afficher*, mais pas chaque décision CSS.

Par exemple, un agent de voyage n'a pas besoin de `div`, `span`, et `button`. Il a besoin de :

- `TripSummary`
- `FlightOptionList`
- `HotelComparison`
- `TravelerForm`
- `PolicyNotice`
- `BookingConfirmation`

Un agent de finance n'a pas besoin d'un terrain de jeu de graphiques générique. Il a besoin de :

- `PortfolioSnapshot`
- `TransactionTable`
- `RiskBreakdown`
- `ScenarioComparison`
- `ApprovalGate`

Le catalogue devrait ressembler à votre produit, pas à du HTML.

## Quoi Utiliser Pour Quel Produit

Voilà la matrice de recommandation que j'utiliserais réellement avec une équipe.

**Vous ajoutez un assistant à une application SaaS existante.**

Commencez par le rendu de composants par outil. Utilisez [Vercel AI SDK UI](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces), [assistant-ui](https://github.com/assistant-ui/assistant-ui), ou [CopilotKit](https://github.com/CopilotKit/CopilotKit), selon la quantité d'état d'agent et d'intégration d'outils frontend dont vous avez besoin. Gardez le catalogue minuscule au début. Rendez des composants produit auxquels vous faites déjà confiance.

**Vous construisez un copilote in-app sérieux qui a besoin d'état partagé.**

Regardez attentivement CopilotKit plus AG-UI. La fonctionnalité importante n'est pas le « chat ». C'est l'état partagé et l'interaction bidirectionnelle : l'agent peut demander une saisie, rendre de l'UI, mettre à jour l'état, et marquer une pause pour approbation.

**Vous avez des agents distants qui doivent envoyer de l'UI à travers une frontière.**

Utilisez A2UI ou un protocole déclaratif de type A2UI. Tout l'intérêt est qu'un agent distant peut décrire l'UI comme des données tandis que l'hôte garde le contrôle du rendu natif, de la sécurité et du style. Si vous avez aussi besoin d'interaction agent/application en direct, faites-la tourner sur AG-UI ou sur le transport que votre environnement standardise.

**Vous construisez dans ChatGPT ou un hôte compatible MCP.**

Utilisez MCP Apps et la voie Apps SDK. Les docs OpenAI actuelles recommandent le pont MCP Apps `ui/*` pour le nouveau travail, tout en gardant `window.openai` comme couche de compatibilité et surface d'extension optionnelle. Copiez aussi leur séparation entre outils de données et outils de rendu : laissez le modèle récupérer et raisonner sur les données avant de choisir de rendre un widget.

**Vous voulez des tableaux de bord, rapports ou formulaires en langage naturel dans votre propre application.**

Essayez json-render, Hashbrown, ou OpenUI. La clé est le catalogue. Si vous exposez `LineChart`, `DataTable`, `MetricGroup`, `FilterControl`, et `InsightCallout`, le modèle peut assembler des surfaces de reporting utiles sans s'approcher du code arbitraire.

**Vous voulez des artefacts éducatifs, visuels ou hautement personnalisés.**

Utilisez un sandbox ouvert comme OpenGenerativeUI. Laissez le modèle écrire du SVG, Canvas, WebGL ou HTML auto-contenu, mais traitez la sortie comme du contenu utilisateur non fiable. Sandboxez-la, dimensionnez-la, retirez les permissions, et gardez-la loin de l'état d'application privilégié.

**Vous avez surtout besoin d'un markdown de streaming plus joli avec quelques affordances inline.**

Ne sur-construisez pas. llm-ui ou le rendu d'outils assistant-ui pourrait suffire.

## Les Erreurs Que J'Éviterais

**Erreur 1 : Laisser le modèle écrire du React de production à l'exécution.**

Il y a des exceptions, mais pour l'UI produit c'est généralement le mauvais choix par défaut. La génération de code à l'exécution est difficile à sécuriser, difficile à tester, difficile à thématiser, et difficile à garder accessible. Si le modèle peut faire le travail en choisissant parmi des composants de confiance, faites cela.

**Erreur 2 : Exposer des primitives de conception au lieu de primitives produit.**

Quand vous donnez au modèle `Row`, `Column`, `Text`, et `Button`, vous lui demandez de devenir votre système de conception. Il en sera un médiocre. Donnez-lui des noms de produit de plus haut niveau.

**Erreur 3 : Penser qu'un JSON valide signifie une UI sûre.**

Un payload peut passer la validation de schéma et quand même être manipulateur ou dangereux. Le libellé peut dire « Voir la facture » tandis que l'action archive le compte. Traitez les specs UI comme du comportement, pas de la décoration. Elles ont besoin de tests de politique, de vérifications sémantiques, et de confirmation humaine pour les actions conséquentes.

**Erreur 4 : Mettre la logique métier dans les outils de rendu.**

Les outils de rendu doivent rendre. Les outils de données doivent récupérer, calculer, muter et valider. Les docs OpenAI Apps SDK soulignent cette séparation pour une raison : si chaque outil de données traîne un widget avec lui, le modèle perd de la place pour raisonner avant de présenter.

**Erreur 5 : Optimiser pour la nouveauté au lieu de l'achèvement de la tâche.**

Le but n'est pas de faire de chaque réponse une interface unique et éblouissante. Le but est de réduire la friction. Un panneau d'approbation stable et ennuyeux qui fait gagner quatre minutes à l'utilisateur est mieux qu'un tableau de bord généré éblouissant auquel on ne peut pas faire confiance deux fois.

## Une Architecture Pratique

Si je commençais un nouveau produit aujourd'hui, j'utiliserais une approche par étapes :

1.  **Expédiez d'abord l'UI outil contrôlée.** Mappez des outils connus vers des composants connus. Enregistrez chaque appel d'outil, rendu UI, et action utilisateur.
2.  **Ajoutez un catalogue domaine.** Une fois que les motifs se répètent, exposez `ComparisonTable`, `DecisionPanel`, `DataCollectionForm`, `Timeline`, et d'autres composants spécifiques au produit.
3.  **Ajoutez la standardisation du transport seulement quand nécessaire.** Si vous possédez à la fois le frontend et le backend, le streaming simple peut suffire. Si vous avez plusieurs frameworks d'agent, utilisez AG-UI. Si les outils traversent les frontières produit, utilisez MCP. Si les agents traversent les frontières organisationnelles, surveillez A2A et A2UI.
4.  **Utilisez des widgets iframe pour les surfaces étrangères ou complexes.** Les cartes, paniers, flux de réservation et mini-applications tierces appartiennent derrière une frontière.
5.  **Réservez la génération ouverte pour les artefacts.** Les diagrammes, simulations, explications temporaires et espaces de travail visuels sont parfaitement adaptés. Les flux de travail principaux ne le sont pas.

L'architecture ressemble à ceci :

```txt
Intention utilisateur
  -> runtime agent
  -> appels d'outils/données
  -> résultat structuré
  -> décision UI
  -> composant de confiance, spec déclaratif, ou widget sandboxé
  -> action utilisateur
  -> flux d'état/événement vers l'agent
```

Cette boucle est le vrai produit. La boîte de chat n'est qu'un dispositif d'entrée possible.

## L'Évaluation Doit Inclure L'UI

Les équipes LLM apprennent lentement à évaluer les prompts et les sorties de modèle. L'interface générative ajoute une autre surface : l'interface elle-même peut être erronée.

Au minimum, sauvegardez ces artefacts pour chaque interface générée :

- prompt et contexte d'outil
- appels d'outils et résultats d'outils
- spec UI généré ou sélection de composant
- nom du composant rendu et props
- libellés visibles par l'utilisateur
- actions attachées aux boutons/formulaires
- mises à jour d'état visibles par le modèle depuis l'UI
- historique des actions utilisateur

Ensuite, écrivez des vérifications comme :

- chaque action destructive doit avoir un composant de confirmation
- les libellés des boutons doivent correspondre à la sémantique de l'action
- les specs de rendu ne peuvent référencer que des composants autorisés
- les totaux visibles par l'utilisateur doivent correspondre aux totaux des résultats d'outil
- les formulaires ne peuvent pas demander des champs en dehors du périmètre de la tâche
- les widgets ne peuvent pas recevoir de secrets que seul le modèle nécessitait
- les métadonnées cachées ne doivent pas contredire les libellés visibles

Cela semble fastidieux. C'est aussi là que vient la confiance en production.

## Une Note sur la Stabilité des Projets

Chaque protocole majeur dans cet espace est en pré-1.0. Dernière vérification le 8 mai 2026 ; prévoyez du changement et vérifiez les docs actuelles avant de faire un pari sur une plateforme.

**Vercel AI SDK RSC** — la fonctionnalité vedette « Generative UI » originale — a vu son développement mis en pause en octobre 2024 ([Discussion #3251](https://github.com/vercel/ai/discussions/3251)) à cause de limitations architecturales sans correctif à court terme. **json-render** (Vercel Labs) a émergé comme la direction de remplacement : basé sur catalogue, agnostique en termes de framework, sans couplage RSC. Il semble avoir rapidement attiré l'attention des développeurs web depuis son lancement début 2026. La raison probable est la DX : json-render fonctionne immédiatement dans un projet React standard ; la portée multiplateforme d'A2UI ajoute une friction de configuration.

**A2UI** (Google) est en pré-1.0 avec des changements cassants entre les versions mineures et des communications de feuille de route incohérentes. Son avantage est la véritable portée multiplateforme (web, Flutter, SwiftUI) que json-render n'aborde pas. Pour les cas d'usage purement web aujourd'hui, json-render semble avoir une meilleure couverture d'outillage ; pour les scénarios d'agent distant ou multiplateforme, la conception d'A2UI est plus appropriée. Une convergence entre les deux specs est possible — Vercel a expérimenté une sortie compatible A2UI depuis json-render.

**AG-UI** (CopilotKit) est aussi en pré-1.0. La confusion la plus courante concerne le nom : AG-UI est un protocole de transport, pas un framework UI. Il définit *comment* les événements circulent entre l'agent et le frontend ; ce que vous rendez reste votre décision. Le concept est solide et largement adopté. Le spec est encore en évolution.

## Mon Avis

L'interface générative ne remplacera pas les interfaces produit soigneusement conçues. Elle remplacera l'hypothèse paresseuse qu'un transcript de chat est l'interface universelle pour l'IA.

Les meilleurs systèmes ne laisseront pas le modèle improviser librement. Ils lui donneront un petit ensemble de blocs de construction natifs au produit ; une connexion de transport fiable ; des limites de sécurité claires ; et assez de liberté pour adapter l'interface à la tâche.

Le futur n'est pas « le modèle écrit votre frontend. »

Le futur est plus proche de : **votre frontend devient un instrument que l'agent peut jouer, mais vous décidez toujours de ce à quoi l'instrument est autorisé à ressembler.**
