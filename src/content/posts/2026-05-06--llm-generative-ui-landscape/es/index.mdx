El chat fue las ruedas de entrenamiento.

La primera generación de apps con LLMs parecía principalmente una caja de texto pegada a un producto. El modelo devolvía prosa. El frontend renderizaba markdown. Si el usuario necesitaba tomar una acción, el asistente describía el botón que el usuario debería ir a hacer clic en algún otro lugar.

Eso estaba bien para demos. No es ahí donde esto va.

El siguiente paso útil es **UI generativo**: el modelo no simplemente responde con texto; ayuda a decidir qué interfaz necesita el usuario en este momento. A veces eso significa llamar a una herramienta y renderizar una tarjeta preconstruida. A veces significa llenar un componente de flujo de trabajo conocido con datos frescos. A veces significa componer un dashboard temporal, formulario, tabla comparativa, gráfico, o widget interactivo.

Desafortunadamente, "UI generativo" se ha convertido en una de esas frases que significa cinco cosas diferentes antes del desayuno.

La gente la usa para describir:

- un modelo eligiendo entre componentes React definidos por el desarrollador
- un spec JSON que el frontend renderiza en componentes nativos
- una app iframe retornada desde una herramienta MCP
- una librería de UI de chat que soporta llamadas a herramientas
- un protocolo de agente que transmite estado entre backend y frontend
- un generador de código en tiempo de diseño como v0, Lovable, Bolt, o Cursor
- un modelo escribiendo literalmente HTML, SVG, Canvas, o React en tiempo de ejecución

Estas cosas están relacionadas, pero no son la misma capa. Si las mezclas, cada conversación de arquitectura se convierte en sopa.

Este es el mapa que desearía haber tenido cuando empecé a comparar el stack actual.

![Un mapa por capas del panorama del UI generativo con LLMs](./landscape-map.webp)

## El Malentendido Central

El error más grande es tratar "UI generativo" como una única elección tecnológica.

Es mejor separar el problema en cuatro capas:

1.  **Carcasa del producto**: la cosa que los usuarios tocan. Puede ser un chat, un copiloto lateral, un dashboard, un constructor de flujos, un panel de IDE, una app de ChatGPT, una pantalla móvil, o una consola de soporte.
2.  **Modelo de composición de UI**: la gramática que se permite hablar al modelo. Puede ser llamadas a herramientas, JSON, A2UI, json-render, OpenUI Lang, selección de componentes Hashbrown, o HTML sandbox.
3.  **Tiempo de ejecución y transporte**: cómo los mensajes, llamadas a herramientas, deltas de estado, acciones del usuario, y artefactos de UI se mueven entre el agente y el frontend. AG-UI, MCP Apps, Apps SDK, A2A, SSE, WebSockets, y el viejo y simple HTTP viven por aquí.
4.  **Agente y backend de herramientas**: LangGraph, Google ADK, CrewAI, Mastra, LlamaIndex, Pydantic AI, Agno, OpenAI Agents SDK, funciones personalizadas, bases de datos, recuperación, y toda la lógica de negocio aburrida que todavía tiene que ser correcta.

Una vez que separas las capas, el ecosistema se vuelve mucho menos místico.

[AG-UI](https://github.com/ag-ui-protocol/ag-ui) no es realmente un competidor de [A2UI](https://github.com/google/A2UI). AG-UI es un protocolo de eventos para la interacción agente-aplicación. A2UI es un formato de UI declarativo que el agente puede enviar. Puedes poner A2UI sobre AG-UI. También puedes poner componentes renderizados por herramientas personalizadas sobre AG-UI.

[json-render](https://github.com/vercel-labs/json-render) no es un producto de chat. Es una arquitectura de catálogo de componentes y renderizador: define los componentes que el modelo puede usar, haz que el modelo emita un árbol JSON válido, y renderiza ese árbol de forma segura.

[CopilotKit](https://github.com/CopilotKit/CopilotKit) no es solo una burbuja de chat. Es un stack de frontend para apps nativas de agentes: UI de chat, UI generativo, estado compartido, herramientas de frontend, y flujos de humano-en-el-circuito.

[OpenAI Apps SDK](https://developers.openai.com/apps-sdk) y [MCP Apps](https://github.com/MCP-UI-Org/mcp-ui) no son herramientas de "haz mi React app dinámica". Son modelos de integración de host para renderizar widgets dentro de ChatGPT u otros hosts compatibles con MCP.

Los nombres son confusos porque el espacio es joven. Las capas son la parte que permanece útil.

## El Espectro de Control

El UI generativo es una disyuntiva entre **control del desarrollador** y **libertad del agente**.

Demasiado control y el asistente se siente como un menú de comandos disfrazado. Demasiada libertad y el modelo empieza a inventar diseños extraños, botones vagos, jerarquía visual rota, estados imposibles, y problemas de seguridad con una sonrisita confiada.

El truco es elegir la menor cantidad de libertad que resuelva el problema del usuario.

![Un espectro desde componentes renderizados por herramientas hasta HTML generado abierto](./control-spectrum.webp)

Lo pienso así:

**Renderización de herramientas a componentes** es el default más seguro. El modelo llama `get_weather`, `search_products`, `compare_plans`, o `draft_invoice`. La app mapea ese resultado de herramienta a un componente que ya tienes: `WeatherCard`, `ProductGrid`, `PlanComparison`, `InvoiceReview`. El modelo decide *cuándo* la UI es útil. Los desarrolladores todavía poseen layout, estilos, accesibilidad, estados de carga, estados vacíos, y acciones peligrosas.

Este es el patrón documentado en [la guía de UI generativo de Vercel AI SDK](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces): el modelo llama a una herramienta, la herramienta retorna datos, y la UI renderiza un componente del resultado. También es el modelo mental detrás de muchas implementaciones de CopilotKit y assistant-ui.

**Catálogos de componentes declarativos** le dan más espacio al modelo. En lugar de elegir un componente, el modelo compone un árbol de partes permitidas. Un catálogo podría incluir `Metric`, `Table`, `Chart`, `FilterBar`, `ApprovalPanel`, y `Timeline`. El modelo puede ensamblar un dashboard o paso de flujo de trabajo, pero no puede ejecutar código arbitrario. Aquí es donde están [A2UI](https://github.com/google/A2UI), [json-render](https://github.com/vercel-labs/json-render), [Hashbrown](https://github.com/liveloveapp/hashbrown), y [OpenUI](https://github.com/thesysdev/openui).

**Mini-apps de iframe** tienen sentido cuando la UI necesita ser más rica que un árbol de componentes, o cuando un proveedor de herramientas remoto posee la experiencia. MCP Apps y OpenAI Apps SDK permiten que una herramienta retorne datos estructurados más un recurso de widget que el host renderiza en un iframe. Esto es poderoso para mapas, carritos de compra, flujos de reservas, gráficos, y superficies externas de productos. También crea una frontera más dura entre la app host y el widget.

**Generación abierta** es el extremo: el agente emite HTML, SVG, Canvas, WebGL, u otros artefactos similares a código en un sandbox. [OpenGenerativeUI](https://github.com/CopilotKit/OpenGenerativeUI) es el mejor ejemplo actual: el agente puede generar visualizaciones de algoritmos, escenas 3D, diagramas, y simulaciones dentro de iframes sandbox. Esto es genial para explicaciones visuales únicas. No es por donde empezaría para un flujo de aprobación empresarial.

Ayuda nombrar la distinción clave aquí: **HTML en iframe** (el modelo escribe código en un sandbox) vs. **catálogo JSON** (el modelo emite un spec estructurado y tu renderizador lo mapea a componentes pre-construidos). Suenan relacionados pero tienen perfiles de riesgo y complejidad muy diferentes. HTML en iframe es máxima mente expresivo; la frontera del iframe hace el trabajo de seguridad. El catálogo JSON no le da al modelo libertad ejecutable — solo puede referenciar tipos de componentes que definiste de antemano. La mayoría de los frameworks en este espacio caen claramente en uno u otro bando.

**Más allá del sandbox**: demos muy recientes sugieren que se está formando un cuarto modo — LLMs conduciendo experiencias tipo juego o inmersivas controlando la salida visual más directamente de lo que cualquier catálogo de componentes permite. Proyectos que generan mundos 3D explorables desde prompts, comportamiento de NPCs dirigido por LLMs en tiempo de ejecución, e inferencia de modelos en navegador vía WebGPU ([WebLLM](https://mlc.ai/web-llm/)) son marcadores tempranos. No hay frameworks estables para construir trabajo de producción aquí aún. Cubriré esta dirección en un artículo dedicado una vez que esto cambie.

## Componentes de Alto Nivel vs Componentes Granulares

Esta es la decisión de diseño más importante.

Si tu catálogo es demasiado granular, el modelo tiene que convertirse en un ingeniero de frontend:

```tsx
Container
Row
Column
Text
Button
Icon
Spacer
Divider
```

Eso se ve flexible, pero ahora el modelo tiene que decidir espaciado, jerarquía, agrupación, estados vacíos, etiquetas de botones, tratamiento de errores, y comportamiento responsivo. También has hecho el prompt más grande y la salida más fácil de romper.

Si tu catálogo es demasiado de alto nivel, el modelo queda atrapado:

```tsx
WeatherCard
StockCard
HotelCard
```

Eso es seguro, pero solo funciona para escenarios conocidos. El modelo no puede hacer una matriz comparativa, pedir inputs faltantes, o adaptar la arquitectura de información cuando la pregunta del usuario cambia.

El medio útil es **componentes de nivel de dominio con ranuras restringidas**:

```tsx
SearchResults
ComparisonTable
MetricGroup
EditablePlan
ApprovalRequest
Timeline
DataCollectionForm
CheckoutReview
```

Estos componentes codifican el gusto del producto y las restricciones de negocio. El modelo decide *qué debe mostrarse*, pero no cada decisión de CSS.

Por ejemplo, un agente de viajes no necesita `div`, `span`, y `button`. Necesita:

- `TripSummary`
- `FlightOptionList`
- `HotelComparison`
- `TravelerForm`
- `PolicyNotice`
- `BookingConfirmation`

Un agente de finanzas no necesita un playground de gráficos genérico. Necesita:

- `PortfolioSnapshot`
- `TransactionTable`
- `RiskBreakdown`
- `ScenarioComparison`
- `ApprovalGate`

El catálogo debería sonar como tu producto, no como HTML.

## Qué Usar Para Qué Producto

Aquí está la matriz de recomendaciones que realmente usaría con un equipo.

**Estás agregando un asistente a una SaaS app existente.**

Empieza con renderización de herramientas a componentes. Usa [Vercel AI SDK UI](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces), [assistant-ui](https://github.com/assistant-ui/assistant-ui), o [CopilotKit](https://github.com/CopilotKit/CopilotKit), dependiendo de cuánta integración de estado de agente y herramientas de frontend necesites. Mantén el catálogo tiny al principio. Renderiza componentes de producto que ya confías.

**Estás construyendo un copiloto serio dentro de la app que necesita estado compartido.**

Mira fuerte a CopilotKit más AG-UI. La característica importante no es "chat". Es estado compartido e interacción bidireccional: el agente puede pedir input, renderizar UI, actualizar estado, y pausar para aprobación.

**Tienes agentes remotos que necesitan enviar UI a través de un límite.**

Usa A2UI o un protocolo declarativo similar a A2UI. El punto completo es que un agente remoto puede describir UI como datos mientras el host mantiene control de renderizado nativo, seguridad, y estilo. Si también necesitas interacción agente/app en vivo, ejecútalo sobre AG-UI o el transporte que tu entorno estandariza.

**Estás construyendo dentro de ChatGPT o un host compatible con MCP.**

Usa MCP Apps y el camino de Apps SDK. Los docs actuales de OpenAI recomiendan el puente `ui/*` de MCP Apps para trabajo nuevo mientras mantienen `window.openai` como una capa de compatibilidad y superficie de extensión opcional. También copia su división entre herramientas de datos y herramientas de renderizado: deja que el modelo obtenga y razone sobre datos antes de elegir renderizar un widget.

**Quieres dashboards, reportes, o formularios en lenguaje natural en tu propia app.**

Prueba json-render, Hashbrown, u OpenUI. La clave es el catálogo. Si expones `LineChart`, `DataTable`, `MetricGroup`, `FilterControl`, e `InsightCallout`, el modelo puede ensamblar superficies de reportes útiles sin acercarse a código arbitrario.

**Quieres artefactos educativos, visuales, o altamente personalizados.**

Usa un sandbox de generación abierta como OpenGenerativeUI. Deja que el modelo escriba SVG, Canvas, WebGL, o HTML autocontenido, pero trata la salida como contenido de usuario no confiable. Sandbox-lo, dimensionarlo, despojar permisos, y mantenerlo lejos de estado de app privilegiado.

**Principalmente necesitas markdown en streaming más bonito con algunas funcionalidades integradas.**

No sobre construyas. llm-ui o renderizado de herramientas de assistant-ui podrían ser suficientes.

## Los Errores Que Evitaría

**Error 1: Dejar que el modelo escriba React de producción en tiempo de ejecución.**

Hay excepciones, pero para UI de producto esto es usualmente el default equivocado. La generación de código en tiempo de ejecución es difícil de asegurar, difícil de testear, difícil de theme, y difícil de mantener accesible. Si el modelo puede hacer el trabajo eligiendo de componentes confiables, haz eso.

**Error 2: Exponer primitivas de diseño en lugar de primitivas de producto.**

Cuando le das al modelo `Row`, `Column`, `Text`, y `Button`, le estás pidiendo que se convierta en tu sistema de diseño. Se convertirá en uno mediocre. Dale sustantivos de producto de más alto nivel.

**Error 3: Pensar que JSON válido significa UI segura.**

Un payload puede pasar validación de esquema y aún ser manipulativo o peligroso. La etiqueta puede decir "Ver factura" mientras la acción archiva la cuenta. Trata los specs de UI como comportamiento, no decoración. Necesitan tests de política, revisiones semánticas, y confirmación humana para acciones consequenciales.

**Error 4: Poner lógica de negocio en herramientas de renderizado.**

Las herramientas de renderizado deberían renderizar. Las herramientas de datos deberían obtener, calcular, mutar, y validar. Los docs de Apps SDK de OpenAI llaman la atención sobre esta división por una razón: si cada herramienta de datos arrastra un widget con ella, el modelo pierde espacio para razonar antes de presentar.

**Error 5: Optimizar por novedad en lugar de completion de tarea.**

El punto no es hacer de cada respuesta una interfaz copo de nieve. El punto es reducir fricción. Un panel de aprobación estable y aburrido que le ahorra al usuario cuatro minutos es mejor que un dashboard generado impresionante que no puede ser confiado dos veces.

## Una Arquitectura Práctica

Si estuviera comenzando un nuevo producto hoy, usaría un enfoque por etapas:

1.  **Envía primero UI de herramientas controlada.** Mapea herramientas conocidas a componentes conocidos. Registra cada llamada a herramienta, renderizado de UI, y acción de usuario.
2.  **Añade un catálogo de dominio.** Una vez que los patrones se repiten, expone `ComparisonTable`, `DecisionPanel`, `DataCollectionForm`, `Timeline`, y otros componentes específicos de producto.
3.  **Añade estandarización de transporte solo cuando sea necesario.** Si posees tanto frontend como backend, el streaming plano puede ser suficiente. Si tienes múltiples frameworks de agentes, usa AG-UI. Si las herramientas cruzan límites de producto, usa MCP. Si los agentes cruzan límites organizacionales, observa A2A y A2UI.
4.  **Usa widgets de iframe para superficies extrañas o complejas.** Mapas, carritos, flujos de reservas, y mini-apps de terceros pertenecen detrás de una frontera.
5.  **Reserva generación abierta para artefactos.** Diagramas, simulaciones, explicadores temporales, y blocs de notas visuales son un gran ajuste. Los flujos de trabajo principales no lo son.

La arquitectura termina luciendo así:

```txt
Intención del usuario
  -> tiempo de ejecución del agente
  -> llamadas a herramientas/datos
  -> resultado estructurado
  -> decisión de UI
  -> componente confiable, spec declarativo, o widget sandbox
  -> acción del usuario
  -> flujo de estado/evento de vuelta al agente
```

Ese loop es el producto real. La caja de chat es solo un dispositivo de input posible.

## La Evaluación Debería Incluir La UI

Los equipos de LLM están aprendiendo lentamente a evaluar prompts y salidas de modelos. El UI generativo añade otra área de superficie: la interfaz misma puede estar equivocada.

Como mínimo, guarda estos artefactos para cada UI generada:

- prompt y contexto de herramienta
- llamadas a herramientas y resultados de herramientas
- spec de UI generada o selección de componente
- nombre de componente renderizado y props
- etiquetas visibles al usuario
- acciones adjuntas a botones/formularios
- actualizaciones de estado visibles al modelo desde la UI
- historial de acciones del usuario

Luego escribe verificaciones como:

- cada acción destructiva debe tener un componente de confirmación
- las etiquetas de botones deben coincidir con la semántica de acción
- los specs de renderizado solo pueden referenciar componentes permitidos
- los totales visibles al usuario deben coincidir con los totales de resultados de herramientas
- los formularios no pueden pedir campos fuera del alcance de la tarea
- los widgets no pueden recibir secretos que solo el modelo necesitaba
- los metadatos ocultos no deben contradecir etiquetas visibles

Esto suena tedioso. También es de donde viene la confianza de producción.

## Una Nota sobre Estabilidad de Proyectos

Cada protocolo mayor en este espacio es pre-1.0. Última verificación 8 de mayo de 2026; planifica para cambios y consulta los docs actuales antes de hacer una apuesta de plataforma.

**Vercel AI SDK RSC** — la característica original de titular de "UI Generativo" — tuvo desarrollo pausado en octubre de 2024 ([Discusión #3251](https://github.com/vercel/ai/discussions/3251)) debido a limitaciones arquitectónicas que no tenían corrección en el corto plazo. **json-render** (Vercel Labs) emergió como la dirección de reemplazo: basado en catálogo, agnóstico de framework, sin acoplamiento RSC. Parece haber capturado atención de desarrolladores web rápidamente desde su lanzamiento a principios de 2026. La razón probable es DX: json-render funciona inmediatamente en un proyecto React estándar; el alcance multiplataforma de A2UI añade fricción de configuración.

**A2UI** (Google) es pre-1.0 con cambios breakage entre versiones menores y comunicaciones de roadmap inconsistentes. Su ventaja es alcance genuino multiplataforma (web, Flutter, SwiftUI) que json-render no aborda. Para casos de uso puramente web hoy, json-render parece tener mejor cobertura de herramientas; para escenarios multiplataforma o de agente remoto, el diseño de A2UI es más apropiado. La convergencia entre los dos specs es posible — Vercel ha experimentado con salida compatible con A2UI desde json-render.

**AG-UI** (CopilotKit) también es pre-1.0. La confusión más común es el nombre: AG-UI es un protocolo de transporte, no un framework de UI. Define *cómo* fluyen los eventos entre agente y frontend; lo que renderizas sigue siendo tu decisión. El concepto es sólido y ampliamente adoptado. El spec todavía está evolucionando.

## Mi Opinión

El UI generativo no reemplazará interfaces de producto cuidadosamente diseñadas. Reemplazará la suposición perezosa de que una transcripción de chat es la interfaz universal para IA.

Los mejores sistemas no dejarán que el modelo improvisé todo. Le darán un conjunto pequeño y afilado de bloques de construcción nativos de producto; una conexión de tiempo de ejecución confiable; fronteras de seguridad claras; y suficiente libertad para adaptar la interfaz a la tarea.

El futuro no es "el modelo escribe tu frontend."

El futuro es más cercano a: **tu frontend se convierte en un instrumento que el agente puede tocar, pero tú todavía decides cómo se permite que suene el instrumento.**
