El chat fue apenas el comienzo.

La primera generación de aplicaciones LLM se parecía principalmente a un cuadro de texto pegado a un producto. El modelo devolvía prosa. El frontend renderizaba markdown. Si el usuario necesitaba realizar una acción, el asistente describía el botón que el usuario debía ir a pulsar a otra parte.

Eso estaba bien para las demos. No es hacia donde se dirige esto.

El siguiente paso útil es la **UI generativa**: el modelo no se limita a responder con texto; ayuda a decidir qué interfaz necesita el usuario en este momento. A veces eso significa llamar a una herramienta y renderizar una tarjeta preconstruida. A veces significa rellenar un componente de flujo de trabajo conocido con datos frescos. A veces significa componer un panel temporal, un formulario, una tabla comparativa, un gráfico o un widget interactivo.

Desafortunadamente, "UI generativa" se ha convertido en una de esas frases que significan cinco cosas distintas antes del desayuno.

La gente la usa para describir:

- un modelo eligiendo entre componentes de React definidos por el desarrollador
- una especificación JSON que el frontend renderiza en componentes nativos
- una aplicación en iframe devuelta desde una herramienta MCP
- una biblioteca de UI de chat que soporta llamadas a herramientas
- un protocolo de agente que transmite el estado entre el backend y el frontend
- un generador de código en tiempo de diseño como v0, Lovable, Bolt o Cursor
- un modelo escribiendo literalmente HTML, SVG, Canvas o React en tiempo de ejecución

Esos están relacionados, pero no son la misma capa. Si los mezclas todos, cada conversación sobre arquitectura se convierte en una sopa de conceptos.

Este es el mapa que desearía haber tenido cuando empecé a comparar el stack actual.

## El Error Fundamental

El mayor error es tratar la "UI generativa" como una sola elección tecnológica.

Es mejor separar el problema en cuatro capas:

1.  **Carcasa del producto (Product shell)**: lo que los usuarios tocan. Puede ser un chat, un copiloto en la barra lateral, un panel de control, un constructor de flujos de trabajo, un panel de IDE, una aplicación de ChatGPT, una pantalla móvil o una consola de soporte.
2.  **Modelo de composición de UI**: la gramática que el modelo tiene permitido hablar. Pueden ser llamadas a herramientas, JSON, A2UI, json-render, OpenUI Lang, selección de componentes Hashbrown o HTML en un sandbox.
3.  **Entorno de ejecución y transporte (Runtime and transport)**: cómo se mueven los mensajes, las llamadas a herramientas, los deltas de estado, las acciones del usuario y los artefactos de UI entre el agente y el frontend. AG-UI, MCP Apps, Apps SDK, A2A, SSE, WebSockets y el viejo y confiable HTTP viven por aquí.
4.  **Backend de agentes y herramientas**: LangGraph, Google ADK, CrewAI, Mastra, LlamaIndex, Pydantic AI, Agno, OpenAI Agents SDK, funciones personalizadas, bases de datos, recuperación y toda la aburrida lógica de negocio que todavía tiene que ser correcta.

Una vez que divides las capas, el ecosistema se vuelve mucho menos místico.

[AG-UI](https://github.com/ag-ui-protocol/ag-ui) no es realmente un competidor de [A2UI](https://github.com/google/A2UI). AG-UI es un protocolo de eventos para la interacción agente-aplicación. A2UI es un formato de UI declarativo que el agente puede enviar. Puedes poner A2UI sobre AG-UI. También puedes poner componentes personalizados renderizados por herramientas sobre AG-UI.

[json-render](https://github.com/vercel-labs/json-render) no es un producto de chat. Es una arquitectura de catálogo de componentes y renderizado: define los componentes que el modelo puede usar, haz que el modelo emita un árbol JSON válido y renderiza ese árbol de forma segura.

[CopilotKit](https://github.com/CopilotKit/CopilotKit) no es solo una burbuja de chat. Es un stack de frontend para aplicaciones nativas de agentes: UI de chat, UI generativa, estado compartido, herramientas de frontend y flujos con humanos en el bucle (human-in-the-loop).

[OpenAI Apps SDK](https://developers.openai.com/apps-sdk) y [MCP Apps](https://github.com/MCP-UI-Org/mcp-ui) no son herramientas para "hacer que mi aplicación de React sea dinámica". Son modelos de integración de host para renderizar widgets dentro de ChatGPT u otros hosts compatibles con MCP.

Los nombres son confusos porque el espacio es joven. Las capas son la parte que sigue siendo útil.

## El Espectro de Control

La UI generativa es un equilibrio entre el **control del desarrollador** y la **libertad del agente**.

Demasiado control y el asistente se siente como una paleta de comandos disfrazada. Demasiada libertad y el modelo empieza a inventar diseños extraños, botones vagos, jerarquías visuales rotas, estados imposibles y problemas de seguridad con una sonrisita confiada.

El truco es elegir la menor cantidad de libertad que resuelva el problema del usuario.

![Un espectro desde componentes renderizados por herramientas hasta HTML generado de forma abierta](../control-spectrum.webp)

Pienso en el espectro de esta manera:

El **renderizado de herramientas a componentes** es el valor predeterminado más seguro. El modelo llama a `get_weather`, `search_products`, `compare_plans` o `draft_invoice`. La aplicación mapea ese resultado de la herramienta a un componente que ya posees: `WeatherCard`, `ProductGrid`, `PlanComparison`, `InvoiceReview`. El modelo decide *cuándo* es útil la UI. Los desarrolladores siguen siendo dueños del diseño, el estilo, la accesibilidad, los estados de carga, los estados vacíos y las acciones peligrosas.

Este es el patrón documentado en la [guía de interfaces de usuario generativas de Vercel AI SDK](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces): el modelo llama a una herramienta, la herramienta devuelve datos y la UI renderiza un componente a partir del resultado. También es el modelo mental detrás de muchas implementaciones de CopilotKit y assistant-ui.

Los **catálogos de componentes declarativos** le dan más espacio al modelo. En lugar de elegir un componente, el modelo compone un árbol a partir de partes permitidas. Un catálogo puede incluir `Metric`, `Table`, `Chart`, `FilterBar`, `ApprovalPanel` y `Timeline`. El modelo puede ensamblar un panel o un paso de flujo de trabajo, pero no puede ejecutar código arbitrario. Aquí es donde se sitúan [A2UI](https://github.com/google/A2UI), [json-render](https://github.com/vercel-labs/json-render), [Hashbrown](https://github.com/liveloveapp/hashbrown) y [OpenUI](https://github.com/thesysdev/openui).

Las **mini-aplicaciones en iframe** tienen sentido cuando la UI necesita ser más rica que un árbol de componentes, o cuando un proveedor de herramientas remoto es dueño de la experiencia. MCP Apps y OpenAI Apps SDK permiten que una herramienta devuelva datos estructurados más un recurso de widget que el host renderiza en un iframe. Esto es potente para mapas, carritos de compra, flujos de reserva, gráficos y superficies de productos externos. También crea un límite más sólido entre la aplicación host y el widget.

La **generación abierta** es el extremo opuesto: el agente emite HTML, SVG, Canvas, WebGL u otros artefactos de código dentro de un sandbox. [OpenGenerativeUI](https://github.com/CopilotKit/OpenGenerativeUI) es el mejor ejemplo actual: el agente puede generar visualizaciones de algoritmos, escenas 3D, diagramas y simulaciones dentro de iframes en un sandbox. Esto es genial para explicaciones visuales puntuales. No es donde yo empezaría para un flujo de aprobación empresarial.

Ayuda a nombrar la distinción clave aquí: **HTML en iframe** (el modelo escribe código en un sandbox) frente a **catálogo JSON** (el modelo emite una especificación estructurada y tu renderizador la mapea a componentes preconstruidos). Estos suenan relacionados pero conllevan perfiles de riesgo y complejidad muy diferentes. El HTML en iframe es máximamente expresivo; el límite del iframe hace el trabajo de seguridad. El catálogo JSON no le da al modelo libertad de ejecución; solo puede hacer referencia a tipos de componentes que definiste de antemano. La mayoría de los frameworks en este espacio caen claramente en un campo o en el otro.

**Más allá del sandbox**: demostraciones muy recientes sugieren que se está formando un cuarto modo — LLMs impulsando experiencias inmersivas o similares a juegos controlando la salida visual de forma más directa de lo que permite cualquier catálogo de componentes. Proyectos que generan mundos 3D explorables a partir de prompts, comportamiento de NPCs dirigido por LLMs en tiempo de ejecución e inferencia de modelos en el navegador a través de WebGPU ([WebLLM](https://mlc.ai/web-llm/)) son marcadores tempranos. Todavía no hay frameworks estables para construir trabajo de producción aquí. Cubriré esta dirección en un artículo dedicado una vez que eso cambie.

## Componentes de Alto Nivel vs. Componentes Granulares

Esta es la decisión de diseño más importante.

Si tu catálogo es demasiado granular, el modelo tiene que convertirse en un ingeniero frontend:

```tsx
Container
Row
Column
Text
Button
Icon
Spacer
Divider
```

Eso parece flexible, pero ahora el modelo tiene que decidir el espaciado, la jerarquía, la agrupación, los estados vacíos, las etiquetas de los botones, el tratamiento de errores y el comportamiento responsivo. También has hecho que el prompt sea más grande y la salida sea más fácil de romper.

Si tu catálogo es de un nivel demasiado alto, el modelo está atrapado:

```tsx
WeatherCard
StockCard
HotelCard
```

Eso es seguro, pero solo funciona para escenarios conocidos. El modelo no puede crear una matriz de comparación, pedir entradas faltantes o adaptar la arquitectura de la información cuando cambia la pregunta del usuario.

El punto medio útil son los **componentes a nivel de dominio con espacios (slots) restringidos**:

```tsx
SearchResults
ComparisonTable
MetricGroup
EditablePlan
ApprovalRequest
Timeline
DataCollectionForm
CheckoutReview
```

Estos componentes codifican el gusto del producto y las restricciones del negocio. El modelo llega a decidir *qué debe mostrarse*, pero no cada decisión de CSS.

Por ejemplo, un agente de viajes no necesita `div`, `span` y `button`. Necesita:

- `TripSummary`
- `FlightOptionList`
- `HotelComparison`
- `TravelerForm`
- `PolicyNotice`
- `BookingConfirmation`

Un agente financiero no necesita un patio de juegos de gráficos genéricos. Necesita:

- `PortfolioSnapshot`
- `TransactionTable`
- `RiskBreakdown`
- `ScenarioComparison`
- `ApprovalGate`

El catálogo debería sonar como tu producto, no como HTML.

## Qué Usar Para Cada Producto

Aquí está la matriz de recomendaciones que usaría realmente con un equipo.

**Estás añadiendo un asistente a una aplicación SaaS existente.**

Empieza con el renderizado de herramientas a componentes. Usa [Vercel AI SDK UI](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces), [assistant-ui](https://github.com/assistant-ui/assistant-ui) o [CopilotKit](https://github.com/CopilotKit/CopilotKit), dependiendo de cuánta integración de estado de agente y herramientas de frontend necesites. Mantén el catálogo pequeño al principio. Renderiza componentes de producto en los que ya confíes.

**Estás construyendo un copiloto serio en la aplicación que necesita estado compartido.**

Analiza seriamente CopilotKit más AG-UI. La característica importante no es el "chat". Es el estado compartido y la interacción bidireccional: el agente puede pedir información, renderizar la UI, actualizar el estado y pausar para aprobación.

**Tienes agentes remotos que necesitan enviar UI a través de un límite.**

Usa A2UI o un protocolo declarativo similar a A2UI. El punto central es que un agente remoto pueda describir la UI como datos mientras el host mantiene el control del renderizado nativo, la seguridad y el estilo. Si también necesitas interacción en vivo entre el agente y la aplicación, ejecútalo sobre AG-UI o cualquier transporte que tu entorno estandarice.

**Estás construyendo dentro de ChatGPT o un host compatible con MCP.**

Usa MCP Apps y la ruta del Apps SDK. Las recomendaciones actuales de OpenAI sugieren el puente `ui/*` de MCP Apps para nuevos trabajos, manteniendo `window.openai` como una capa de compatibilidad y superficie de extensión opcional. También copia su división entre herramientas de datos y herramientas de renderizado: deja que el modelo obtenga y razone sobre los datos antes de que elija renderizar un widget.

**Quieres paneles, informes o formularios en lenguaje natural en tu propia aplicación.**

Prueba json-render, Hashbrown o OpenUI. La clave es el catálogo. Si expones `LineChart`, `DataTable`, `MetricGroup`, `FilterControl` e `InsightCallout`, el modelo puede ensamblar superficies de informes útiles sin acercarse al código arbitrario.

**Quieres artefactos educativos, visuales o altamente personalizados.**

Usa un sandbox abierto como OpenGenerativeUI. Deja que el modelo escriba SVG, Canvas, WebGL o HTML autocontenido, pero trata la salida como contenido de usuario no confiable. Ponlo en un sandbox, dimensiona su tamaño, quita permisos y mantenlo alejado del estado privilegiado de la aplicación.

**Principalmente necesitas un markdown en streaming más bonito con algunas funcionalidades integradas.**

No construyas de más. llm-ui o el renderizado de herramientas de assistant-ui podrían ser suficientes.

## Los Errores Que Yo Evitaría

**Error 1: Dejar que el modelo escriba React de producción en tiempo de ejecución.**

Hay excepciones, pero para la UI de un producto esto suele ser el punto de partida equivocado. La generación de código en tiempo de ejecución es difícil de asegurar, difícil de probar, difícil de tematizar y difícil de mantener accesible. Si el modelo puede hacer el trabajo eligiendo entre componentes de confianza, haz eso.

**Error 2: Exponer primitivas de diseño en lugar de primitivas de producto.**

Cuando le das al modelo `Row`, `Column`, `Text` y `Button`, le estás pidiendo que se convierta en tu sistema de diseño. Se convertirá en uno mediocre. Dale sustantivos de producto de mayor nivel.

**Error 3: Pensar que un JSON válido significa una UI segura.**

Un payload puede pasar la validación del esquema y seguir siendo manipulador o peligroso. La etiqueta puede decir "Ver factura" mientras la acción archiva la cuenta. Trata las especificaciones de UI como comportamiento, no como decoración. Necesitan pruebas de política, verificaciones semánticas y confirmación humana para acciones trascendentales.

**Error 4: Poner lógica de negocio en las herramientas de renderizado.**

Las herramientas de renderizado deberían renderizar. Las herramientas de datos deberían obtener, computar, mutar y validar. Los documentos del Apps SDK de OpenAI mencionan esta división por una razón: si cada herramienta de datos arrastra un widget con ella, el modelo pierde espacio para razonar antes de presentar.

**Error 5: Optimizar para la novedad en lugar de para la finalización de la tarea.**

El punto no es hacer que cada respuesta sea una interfaz única. El punto es reducir la fricción. Un panel de aprobación estable y aburrido que le ahorre al usuario cuatro minutos es mejor que un panel generado deslumbrante en el que no se puede confiar dos veces.

## Una Arquitectura Práctica

Si estuviera empezando un producto nuevo hoy, usaría un enfoque por etapas:

1.  **Lanzar primero una UI de herramientas controlada.** Mapea herramientas conocidas a componentes conocidos. Registra cada llamada a herramienta, renderizado de UI y acción del usuario.
2.  **Añadir un catálogo de dominio.** Una vez que los patrones se repitan, expón `ComparisonTable`, `DecisionPanel`, `DataCollectionForm`, `Timeline` y otros componentes específicos del producto.
3.  **Añadir estandarización de transporte solo cuando sea necesario.** Si eres dueño de tanto el frontend como del backend, el streaming simple puede estar bien. Si tienes múltiples frameworks de agentes, usa AG-UI. Si las herramientas cruzan los límites del producto, usa MCP. Si los agentes cruzan límites organizativos, vigila A2A y A2UI.
4.  **Usar widgets en iframe para superficies externas o complejas.** Los mapas, carritos, flujos de reserva y mini-apps de terceros pertenecen a un límite.
5.  **Reservar la generación abierta para artefactos.** Los diagramas, las simulaciones, los explicadores temporales y los borradores visuales son un gran ajuste. Los flujos de trabajo principales no lo son.

La arquitectura termina viéndose así:

```txt
Intención del usuario
  -> runtime del agente
  -> llamadas a herramientas/datos
  -> resultado estructurado
  -> decisión de UI
  -> componente de confianza, especificación declarativa o widget en sandbox
  -> acción del usuario
  -> flujo de estado/eventos de vuelta al agente
```

Ese bucle es el producto real. El cuadro de chat es solo un posible dispositivo de entrada.

## La Evaluación Debería Incluir la UI

Los equipos de LLM están aprendiendo poco a poco a evaluar los prompts y las salidas de los modelos. La UI generativa añade otra superficie de ataque: la interfaz en sí misma puede estar mal.

Como mínimo, guarda estos artefactos para cada UI generada:

- prompt y contexto de la herramienta
- llamadas a herramientas y resultados de herramientas
- especificación de UI generada o selección de componentes
- nombre del componente renderizado y sus props
- etiquetas visibles para el usuario
- acciones adjuntas a botones/formularios
- actualizaciones de estado visibles para el modelo desde la UI
- historial de acciones del usuario

Luego escribe verificaciones como:

- cada acción destructiva debe tener un componente de confirmación
- las etiquetas de los botones deben coincidir con la semántica de la acción
- las especificaciones de renderizado solo pueden hacer referencia a componentes permitidos
- los totales visibles para el usuario deben coincidir con los totales de los resultados de la herramienta
- los formularios no pueden solicitar campos fuera del alcance de la tarea
- los widgets no pueden recibir secretos que solo el modelo necesitaba
- los metadatos ocultos no deben contradecir las etiquetas visibles

Esto suena tedioso. También es de donde proviene la confianza en la producción.

## Una Nota sobre la Estabilidad del Proyecto

Cada protocolo importante en este espacio es pre-1.0. Verificado por última vez el 8 de mayo de 2026; planifica para el cambio y consulta la documentación actual antes de hacer una apuesta de plataforma.

**Vercel AI SDK RSC** —la característica original que encabezó la "UI Generativa"— tuvo su desarrollo pausado en octubre de 2024 ([Discusión #3251](https://github.com/vercel/ai/discussions/3251)) debido a limitaciones arquitectónicas que no tenían una solución a corto plazo. **json-render** (Vercel Labs) surgió como la dirección de reemplazo: basada en catálogo, agnóstica al framework, sin acoplamiento a RSC. Parece haber captado la atención de los desarrolladores web rápidamente desde su lanzamiento a principios de 2026. La razón probable es la DX: json-render funciona inmediatamente en un proyecto estándar de React; el alcance multiplataforma de A2UI añade fricción en la configuración.

**A2UI** (Google) es pre-1.0 con cambios disruptivos entre versiones menores y comunicaciones de hoja de ruta inconsistentes. Su ventaja es el alcance genuinamente multiplataforma (web, Flutter, SwiftUI) que json-render no aborda. Para casos de uso puramente web hoy en día, json-render parece tener mejor cobertura de herramientas; para escenarios multiplataforma o de agentes remotos, el diseño de A2UI es más apropiado. La convergencia entre las dos especificaciones es posible —Vercel ha experimentado con salidas compatibles con A2UI desde json-render.

**AG-UI** (CopilotKit) también es pre-1.0. La confusión más común es el nombre: AG-UI es un protocolo de transporte, no un framework de UI. Define *cómo* fluyen los eventos entre el agente y el frontend; lo que renderizas sigue siendo tu elección. El concepto es sólido y ampliamente adoptado. La especificación sigue evolucionando.

## Mi Opinión

La UI generativa no reemplazará las interfaces de producto cuidadosamente diseñadas. Reemplazará la suposición perezosa de que una transcripción de chat es la interfaz universal para la IA.

Los mejores sistemas no dejarán que el modelo improvise todo. Le darán un conjunto pequeño y afilado de bloques de construcción nativos del producto; una conexión de tiempo de ejecución confiable; límites de seguridad claros; y suficiente libertad para adaptar la interfaz a la tarea.

El futuro no es "el modelo escribe tu frontend".

El futuro es más parecido a: **tu frontend se convierte en un instrumento que el agente puede tocar, pero tú sigues decidiendo cómo se permite que suene el instrumento.**
