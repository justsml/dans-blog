AG-UI, MCP Apps, Apps SDK, A2A, SSE, WebSockets, y HTTP plano todos viven por aquí. **Backend de agente y herramientas**: LangGraph, Google ADK, CrewAI, Mastra, LlamaIndex, Pydantic AI, Agno, OpenAI Agents SDK, funciones custom, bases de datos, retrieval, y toda la lógica de negocio aburrida que aún tiene que ser correcta. A2UI es un formato de UI declarativo que el agente puede enviar. Puedes poner A2UI sobre AG-UI. También puedes poner componentes custom renderizados por herramientas sobre AG-UI. json-render no es un producto de chat. Es una arquitectura de catálogo de componentes y renderer: define los componentes que el modelo puede usar, haz que el modelo emita un árbol JSON válido, y renderiza ese árbol de forma segura. CopilotKit no es solo una burbuja de chat. Es un stack frontend para apps agent-native: UI de chat, UI generativa, estado compartido, herramientas frontend, y flujos human-in-the-loop. OpenAI Apps SDK y MCP Apps no son herramientas de "haz mi app React dinámica". Son modelos de integración de host para renderizar widgets dentro de ChatGPT u otros hosts compatibles con MCP. Los nombres son confusos porque el espacio es joven. Las capas son la parte que se mantiene útil. La UI generativa es un trade-off entre **control del desarrollador** y **libertad del agente**. Demasiado control y el asistente se siente como una paleta de comandos disfrazada. Demasiada libertad y el modelo empieza a inventar layouts extraños, botones vagos, jerarquía visual rota, estados imposibles, y problemas de seguridad con una sonrisita confiada. El truco es elegir la menor cantidad de libertad que resuelve el problema del usuario. Pienso en el espectro así: **Renderización de herramienta-a-componente** es el default más seguro. El modelo llama a `get_weather`, `search_products`, `compare_plans`, o `draft_invoice`. La app mapea ese resultado de herramienta a un componente que ya posees: `WeatherCard`, `ProductGrid`, `PlanComparison`, `InvoiceReview`. El modelo decide *cuándo* la UI es útil. Los desarrolladores aún controlan layout, styling, accesibilidad, estados de carga, estados vacíos, y acciones peligrosas. Este es el patrón documentado en la guía de UI generativa de Vercel AI SDK: el modelo llama a una herramienta, la herramienta devuelve datos, y la UI renderiza un componente con el resultado. También es el modelo mental detrás de muchas implementaciones de CopilotKit y assistant-ui. **Catálogos declarativos de componentes** dan más espacio al modelo. En vez de elegir un componente, el modelo compone un árbol de partes permitidas. Un catálogo podría incluir `Metric`, `Table`, `Chart`, `FilterBar`, `ApprovalPanel`, y `Timeline`. El modelo puede ensamblar un dashboard o paso de workflow, pero no puede ejecutar código arbitrario. Aquí se sitúan A2UI, json-render, Hashbrown, y OpenUI. **Mini-apps iframe** tienen sentido cuando la UI necesita ser más rica que un árbol de componentes, o cuando un proveedor de herramientas remoto posee la experiencia. MCP Apps y OpenAI Apps SDK permiten que una herramienta devuelva datos estructurados plus un recurso de widget que el host renderiza en un iframe. Esto es potente para mapas, carritos de compra, flujos de reserva, gráficos, y surfaces de producto externas. También crea una frontera más dura entre la app host y el widget. **Generación abierta** es el extremo: el agente emite HTML, SVG, Canvas, WebGL, u otros artefactos tipo código en un sandbox. OpenGenerativeUI es el mejor ejemplo actual: el agente puede generar visualizaciones de algoritmos, escenas 3D, diagramas, y simulaciones dentro de iframes sandboxeados. Esto es genial para explicaciones visuales de un solo uso. No es donde empezaría para un flujo de aprobación empresarial. Ayuda nombrar la distinción clave: **iframe HTML** (el modelo escribie código en un sandbox) vs. **catálogo JSON** (el modelo emite una especificación estructurada y tu renderer la mapea a componentes preconstruidos). Suenan relacionados pero tienen profiles de riesgo y complejidad muy distintos. iframe HTML es maximalmente expresivo; la frontera del iframe hace el trabajo de seguridad. catálogo JSON da al modelo ninguna libertad ejecutable — solo puede referenciar tipos de componentes que definiste previamente. La mayoría de los frameworks en este espacio caen claramente en uno u otro campo. **Más allá del sandbox**: demos muy recientes sugieren que un cuarto modo se está formando — LLMs dirigiendo experiencias tipo juego o inmersivas controlando output visual más directamente que cualquier catálogo de componentes permite. Proyectos generando mundos 3D explorables desde prompts, comportamiento NPC dirigido por LLM en runtime, e inferencia de modelo in-browser via WebGPU (WebLLM) son marcadores tempranos. No hay frameworks estables para construir trabajo de producción aquí aún. Cubriré esta dirección en un artículo dedicado cuando eso cambie. Esta es la decisión de diseño más importante. Si tu catálogo es demasiado granular, el modelo tiene que convertirse en ingeniero frontend: Eso parece flexible, pero ahora el modelo tiene que decidir spacing, jerarquía, agrupación, estados vacíos, labels de botones, tratamiento de errores, y comportamiento responsive. También has hecho el prompt más grande y el output más fácil de romper. Si tu catálogo es demasiado alto-nivel, el modelo está atrapado: Eso es seguro, pero solo funciona para escenarios conocidos. El modelo no puede hacer una matriz de comparación, pedir inputs faltantes, o adaptar la arquitectura de información cuando la pregunta del usuario cambia. El medio útil es **componentes de nivel de dominio con slots restringidos**: Estos componentes codifican gusto de producto y restricciones de negocio. El modelo decide *qué debería mostrarse*, pero no cada decisión de CSS. Por ejemplo, un agente de viajes no necesita `div`, `span`, y `button`. Necesita: `TripSummary`, `FlightOptionList`, `HotelComparison`, `TravelerForm`, `PolicyNotice`, `BookingConfirmation`. Un agente financiero no necesita un playground de charts genérico. Necesita: `PortfolioSnapshot`, `TransactionTable`, `RiskBreakdown`, `ScenarioComparison`, `ApprovalGate`. El catálogo debería sonar como tu producto, no como HTML. Esta tabla es intencionalmente opinada. Trata cada proyecto como una herramienta en un stack, no como una plataforma winner-take-all. Aquí está la matriz de recomendaciones que realmente usaría con un equipo. **Estás añadiendo un asistente a una app SaaS existente.** Empieza con renderización de herramienta-a-componente. Usa Vercel AI SDK UI, assistant-ui, o CopilotKit, dependiendo de cuánto estado de agente e integración de herramientas frontend necesites. Mantén el catálogo pequeño al principio. Renderiza componentes de producto que ya confías. **Estás construyendo un copilot serio in-app que necesita estado compartido.** Mira seriamente CopilotKit plus AG-UI. La característica importante no es "chat." Es estado compartido e interacción bidireccional: el agente puede pedir input, renderizar UI, actualizar estado, y pausar para aprobación. **Tienes agentes remotos que necesitan enviar UI cruzando una frontera.** Usa A2UI o un protocolo declarativo tipo A2UI. Todo el punto es que un agente remoto puede describir UI como datos mientras el host mantiene control de renderización nativa, seguridad, y estilo. Si también necesitas interacción agente/app en vivo, ejecútalo sobre AG-UI o cualquier transporte que tu entorno estandarice. **Estás construyendo dentro de ChatGPT o un host compatible con MCP.** Usa MCP Apps y el camino del Apps SDK. Los docs actuales de OpenAI recomiendan el bridge MCP Apps `ui/*` para trabajo nuevo mientras mantienen `window.openai` como capa de compatibilidad y surface de extensión opcional. También copia su split entre herramientas de datos y herramientas de render: deja que el modelo fetch y razone sobre datos antes de elegir renderizar un widget. **Quieres dashboards, reportes, o formularios por lenguaje natural en tu propia app.** Prueba json-render, Hashbrown, o OpenUI. La clave es el catálogo. Si expones `LineChart`, `DataTable`, `MetricGroup`, `FilterControl`, e `InsightCallout`, el modelo puede ensamblar surfaces de reporte útil sin acercarse a código arbitrario. **Quieres artefactos educativos, visuales, o altamente bespoke.** Usa un sandbox abierto como OpenGenerativeUI. Deja que el modelo escriba SVG, Canvas, WebGL, o HTML autocontenido, pero trata el output como contenido de usuario no confiable. Sandboxea, dimensiona, quita permisos, y manténlo lejos de estado de app privilegiado. **Principalmente necesitas markdown streaming más bonito con unos pocos affordances inline.** No overbuild. llm-ui o renderización de herramientas de assistant-ui podría ser suficiente. **Error 1: Dejar que el modelo escriba React de producción en runtime.** Hay excepciones, pero para UI de producto este es usualmente el default equivocado. Generación de código en runtime es difícil de asegurar, difícil de testear, difícil de temear, y difícil de mantener accesible. Si el modelo puede hacer el trabajo eligiendo componentes confiables, haz eso. **Error 2: Exponer primitivas de diseño en vez de primitivas de producto.** Cuando das al modelo `Row`, `Column`, `Text`, y `Button`, le estás pidiendo que se convierta en tu design system. Se convertirá en uno mediocre. Dale nouns de producto de alto nivel. **Error 3: Pensar que JSON válido significa UI segura.** Un payload puede pasar validación de schema y aún ser manipulativo o peligroso. El label puede decir "Ver factura" mientras la acción archiva la cuenta. Trata las specs de UI como comportamiento, no decoración. Necesitan tests de policy, checks semánticos, y confirmación humana para acciones consequentes. **Error 4: Poner lógica de negocio en herramientas de render.** Las herramientas de render deberían renderizar. Las herramientas de datos deberían fetch, computar, mutar, y validar. Los docs del Apps SDK de OpenAI destacan este split por una razón: si cada herramienta de datos arrastra un widget con ella, el modelo pierde espacio para razonear antes de presentar. **Error 5: Optimizar por novedad en vez de completar tareas.** El punto no es hacer cada respuesta una interface snowflake. El punto es reducir fricción. Un panel de aprobación estable y aburrido que salva al usuario cuatro minutos es mejor que un dashboard generado deslumbrante que no se puede confiar dos veces. Si estuviera empezando un producto nuevo hoy, usaría un enfoque escalonado: Ship UI de herramienta controlada primero. Mapea herramientas conocidas a componentes conocidos. Loguea cada llamada a herramienta, render de UI, y acción de usuario. Añade un catálogo de dominio. Una vez que los patrones se repiten, expone `ComparisonTable`, `DecisionPanel`, `DataCollectionForm`, `Timeline`, y otros componentes producto-specificos. Añade estandarización de transporte solo cuando sea necesario. Si posees tanto frontend como backend, streaming plano puede ser suficiente. Si tienes múltiples frameworks de agente, usa AG-UI. Si herramientas cruzan fronteras de producto, usa MCP. Si agentes cruzan fronteras organizacionales, mira A2A y A2UI. Usa widgets iframe para surfaces foreign o complejas. Mapas, carritos, flujos de reserva, y mini-apps de terceros pertenecen detrás de una frontera. Reserva generación abierta para artefactos. Diagramas, simulaciones, explicadores temporales, y scratchpads visuales son un buen fit. Workflows core no lo son. La arquitectura termina pareciendo así: Ese loop es el producto real. La caja de chat es solo un posible dispositivo de entrada. Los equipos de LLM están lentamente aprendiendo a evaluar prompts y outputs de modelo. UI generativa añade otra surface area: la interface misma puede estar equivocada. Al mínimo, salva estos artefactos para cada UI generada: prompt y contexto de herramienta, llamadas a herramientas y resultados de herramientas, spec de UI generada o selección de componente, nombre de componente renderizado y props, labels visible para el usuario, acciones attachadas a botones/formularios, actualizaciones de estado model-visible desde la UI, historia de acciones del usuario. Luego escribe checks como: cada acción destructiva debe tener un componente de confirmación, labels de botones deben coincidir con semántica de acción, specs de render solo pueden referenciar componentes permitidos, totales visible para el usuario deben coincidir con totales de resultado de herramienta, formularios no pueden pedir campos fuera del scope de tarea, widgets no pueden recibir secrets que solo el modelo necesitaba, metadata oculta no debe contradecir labels visibles. Esto suena tedioso. También es de donde viene la confianza de producción. Si quieres ir de artículo a código, estos son los mejores puntos de partida que encontré: Cada protocolo major en este espacio es pre-1.0. Verificado el 8 de mayo, 2026; planifica para cambio y revisa los docs actuales antes de hacer un bet de plataforma. Vercel AI SDK RSC — el feature headline original de "Generative UI" — tuvo su desarrollo pausado en octubre de 2024 (Discussion #3251) debido a limitaciones arquitecturales sin fix cercano. json-render (Vercel Labs) emergió como la dirección de reemplazo: catalog-based, framework-agnostic, sin coupling de RSC. Parece haber ganado atención de web-developers rápidamente desde su launch de early-2026. La razón probable es DX: json-render funciona inmediatamente en un proyecto React estándar; el scope cross-platform de A2UI añade fricción de setup. A2UI (Google) es pre-1.0 con breaking changes entre versiones minor y comunicaciones de roadmap inconsistentes. Su ventaja es reach cross-platform genuino (web, Flutter, SwiftUI) que json-render no address. Para casos de uso pure-web hoy, json-render parece tener mejor coverage de tooling; para escenarios cross-platform o remote-agent, el diseño de A2UI es más apropiado. Convergencia entre los dos specs es posible — Vercel ha experimentado con output A2UI-compatible desde json-render. AG-UI (CopilotKit) también es pre-1.0. La confusión más común es el nombre: AG-UI es un protocolo de transporte, no un framework de UI. Define *cómo* los eventos fluyen entre agente y frontend; lo que renderizas sigue siendo tu decisión. El concepto es sólido y broadly adoptado. El spec sigue evolucionando. La UI generativa no reemplazará interfaces de producto cuidadosamente diseñadas. Reemplazará la suposición lazy que un transcript de chat es la interface universal para AI. Los mejores sistemas no dejarán al modelo freestyle todo. Le darán un set pequeño y sharp de building blocks producto-nativos; una conexión runtime confiable; fronteras de seguridad claras; y suficiente libertad para adaptar la interface a la tarea. El futuro no es "el modelo escribie tu frontend." El futuro es más cercano a: **tu frontend se convierte en un instrumento que el agente puede tocar, pero tú aún decides qué se le permite sonar al instrumento.**

## El error central

El error más grande es tratar "UI generativa" como una única decisión tecnológica.

Es mejor separar el problema en cuatro capas:

1.  **Shell del producto**: lo que los usuarios tocan. Puede ser un chat, copilot lateral, dashboard, constructor de workflows, panel de IDE, app ChatGPT, pantalla móvil, o consola de soporte.
2.  **Modelo de composición de UI**: la gramática que el modelo puede usar. Puede ser llamadas a herramientas, JSON, A2UI, json-render, OpenUI Lang, selección de componentes Hashbrown, o HTML sandboxeado.
3.  **Runtime y transporte**: cómo se mueven mensajes, llamadas a herramientas, deltas de estado, acciones del usuario, y artefactos de UI entre el agente y el frontend. AG-UI, MCP Apps, Apps SDK, A2A, SSE, WebSockets, y HTTP plano todos viven por aquí.
4.  **Backend de agente y herramientas**: LangGraph, Google ADK, CrewAI, Mastra, LlamaIndex, Pydantic AI, Agno, OpenAI Agents SDK, funciones custom, bases de datos, retrieval, y toda la lógica de negocio aburrida que aún tiene que ser correcta.

Una vez que separas las capas, el ecosistema se vuelve mucho menos místico.

[AG-UI](https://github.com/ag-ui-protocol/ag-ui) no es realmente un competidor de [A2UI](https://github.com/google/A2UI). AG-UI es un protocolo de eventos para interacción agente-a-aplicación. A2UI es un formato de UI declarativo que el agente puede enviar. Puedes poner A2UI sobre AG-UI. También puedes poner componentes custom renderizados por herramientas sobre AG-UI.

[json-render](https://github.com/vercel-labs/json-render) no es un producto de chat. Es una arquitectura de catálogo de componentes y renderer: define los componentes que el modelo puede usar, haz que el modelo emita un árbol JSON válido, y renderiza ese árbol de forma segura.

[CopilotKit](https://github.com/CopilotKit/CopilotKit) no es solo una burbuja de chat. Es un stack frontend para apps agent-native: UI de chat, UI generativa, estado compartido, herramientas frontend, y flujos human-in-the-loop.

[OpenAI Apps SDK](https://developers.openai.com/apps-sdk) y [MCP Apps](https://github.com/MCP-UI-Org/mcp-ui) no son herramientas de "haz mi app React dinámica". Son modelos de integración de host para renderizar widgets dentro de ChatGPT u otros hosts compatibles con MCP.

Los nombres son confusos porque el espacio es joven. Las capas son la parte que se mantiene útil.

## El espectro de control

La UI generativa es un trade-off entre **control del desarrollador** y **libertad del agente**.

Demasiado control y el asistente se siente como una paleta de comandos disfrazada. Demasiada libertad y el modelo empieza a inventar layouts extraños, botones vagos, jerarquía visual rota, estados imposibles, y problemas de seguridad con una sonrisita confiada.

El truco es elegir la menor cantidad de libertad que resuelve el problema del usuario.

![Un espectro desde componentes renderizados por herramientas hasta HTML generado abiertamente](./control-spectrum.webp)

Este es el patrón documentado en [la guía de UI generativa de Vercel AI SDK](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces): el modelo llama a una herramienta, la herramienta devuelve datos, y la UI renderiza un componente con el resultado. ## Componentes de alto nivel vs componentes granulares

Esta es la decisión de diseño más importante.

Si tu catálogo es demasiado granular, el modelo tiene que convertirse en ingeniero frontend:

```tsx
Container
Row
Column
Text
Button
Icon
Spacer
Divider
```

Eso parece flexible, pero ahora el modelo tiene que decidir spacing, jerarquía, agrupación, estados vacíos, labels de botones, tratamiento de errores, y comportamiento responsive. También has hecho el prompt más grande y el output más fácil de romper.

Si tu catálogo es demasiado alto-nivel, el modelo está atrapado:

```tsx
WeatherCard
StockCard
HotelCard
```

Eso es seguro, pero solo funciona para escenarios conocidos. El modelo no puede hacer una matriz de comparación, pedir inputs faltantes, o adaptar la arquarquitectura de información cuando la pregunta del usuario cambia.

El medio útil es **componentes de nivel de dominio con slots restringidos**:

```tsx
SearchResults
ComparisonTable
MetricGroup
EditablePlan
ApprovalRequest
Timeline
DataCollectionForm
CheckoutReview
```

Estos componentes codifican gusto de producto y restricciones de negocio. El modelo decide *qué debería mostrarse*, pero no cada decisión de CSS.

Por ejemplo, un agente de viajes no necesita `div`, `span`, y `button`. Necesita:

- `TripSummary`
- `FlightOptionList`
- `HotelComparison`
- `TravelerForm`
- `PolicyNotice`
- `BookingConfirmation`

Un agente financiero no necesita un playground de charts genérico. Necesita:

- `PortfolioSnapshot`
- `TransactionTable`
- `RiskBreakdown`
- `ScenarioComparison`
- `ApprovalGate`

El catálogo debería sonar como tu producto, no como HTML.

## Qué usar para cada producto

Aquí está la matriz de recomendaciones que realmente usaría con un equipo.

**Estás añadiendo un asistente a una app SaaS existente.**

Empieza con renderización de herramienta-a-componente. Usa [Vercel AI SDK UI](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces), [assistant-ui](https://github.com/assistant-ui/assistant-ui), o [CopilotKit](https://github.com/CopilotKit/CopilotKit), dependiendo de cuánto estado de agente e integración de herramientas frontend necesites. Mantén el catálogo pequeño al principio. Renderiza componentes de producto que ya confías.

**Estás construyendo un copilot serio in-app que necesita estado compartido.**

Mira seriamente CopilotKit plus AG-UI. La característica importante no es "chat." Es estado compartido e interacción bidireccional: el agente puede pedir input, renderizar UI, actualizar estado, y pausar para aprobación.

**Tienes agentes remotos que necesitan enviar UI cruzando una frontera.**

Usa A2UI o un protocolo declarativo tipo A2UI. Todo el punto es que un agente remoto puede describir UI como datos mientras el host mantiene control de renderización nativa, seguridad, y estilo. Si también necesitas interacción agente/app en vivo, ejecútalo sobre AG-UI o cualquier transporte que tu entorno estandarize.

**Estás construyendo dentro de ChatGPT o un host compatible con MCP.**

Usa MCP Apps y el camino del Apps SDK. Los docs actuales de OpenAI recomiendan el bridge MCP Apps `ui/*` para trabajo nuevo mientras mantienen `window.openai` como capa de compatibilidad y surface de extensión opcional. También copia su split entre herramientas de datos y herramientas de render: deja que el modelo fetch y razone sobre datos antes de elegir renderizar un widget.

**Quieres dashboards, reportes, o formularios por lenguaje natural en tu propia app.**

Prueba json-render, Hashbrown, o OpenUI. La clave es el catálogo. Si expones `LineChart`, `DataTable`, `MetricGroup`, `FilterControl`, e `InsightCallout`, el modelo puede ensamblar surfaces de reporte útil sin acercarse a código arbitrario.

**Quieres artefactos educativos, visuales, o altamente bespoke.**

Usa un sandbox abierto como OpenGenerativeUI. Deja que el modelo escriba SVG, Canvas, WebGL, o HTML autocontenido, pero trata el output como contenido de usuario no confiable. Sandboxea, dimensiona, quita permisos, y manténlo lejos de estado de app privilegiado.

**Principalmente necesitas markdown streaming más bonito con unos pocos affordances inline.**

No overbuild. llm-ui o renderización de herramientas de assistant-ui podría ser suficiente.

## Los errores que evitaría

**Error 1: Dejar que el modelo escriba React de producción en runtime.**

Hay excepciones, pero para UI de producto este es usualmente el default equivocado. Generación de código en runtime es difícil de asegurar, difícil de testear, difícil de temear, y difícil de mantener accesible. Si el modelo puede hacer el trabajo eligiendo componentes confiables, haz eso.

**Error 2: Exponer primitivas de diseño en vez de primitivas de producto.**

Cuando das al modelo `Row`, `Column`, `Text`, y `Button`, le estás pidiendo que se convierta en tu design system. Se convertirá en uno mediocre. Dale nouns de producto de alto nivel.

**Error 3: Pensar que JSON válido significa UI segura.**

Un payload puede pasar validación de schema y aún ser manipulativo o peligroso. El label puede decir "Ver factura" mientras la acción archiva la cuenta. Trata las specs de UI como comportamiento, no decoración. Necesitan tests de policy, checks semánticos, y confirmación humana para acciones consequentes.

**Error 4: Poner lógica de negocio en herramientas de render.**

Las herramientas de render deberían renderizar. Las herramientas de datos deberían fetch, computar, mutar, y validar. Los docs del Apps SDK de OpenAI destacan este split por una razón: si cada herramienta de datos arrastra un widget con ella, el modelo pierde espacio para razonear antes de presentar.

**Error 5: Optimizar por novedad en vez de completar tareas.**

El punto no es hacer cada respuesta una interface snowflake. El punto es reducir fricción. Un panel de aprobación estable y aburrido que salva al usuario cuatro minutos es mejor que un dashboard generado deslumbrante que no se puede confiar dos veces.

## Una arquitectura práctica

Si estuviera empezando un producto nuevo hoy, usaría un enfoque escalonado:

1.  **Ship UI de herramienta controlada primero.** Mapea herramientas conocidas a componentes conocidos. Loguea cada llamada a herramienta, render de UI, y acción de usuario.
2.  **Añade un catálogo de dominio.** Una vez que los patrones se repiten, expone `ComparisonTable`, `DecisionPanel`, `DataCollectionForm`, `Timeline`, y otros componentes producto-specificos.
3.  **Añade estandarización de transporte solo cuando sea necesario.** Si posees tanto frontend como backend, streaming plano puede ser suficiente. Si tienes múltiples frameworks de agente, usa AG-UI. Si herramientas cruzan fronteras de producto, usa MCP. Si agentes cruzan fronteras organizacionales, mira A2A y A2UI.
4.  **Usa widgets iframe para surfaces foreign o complejas.** Mapas, carritos, flujos de reserva, y mini-apps de terceros pertenecen detrás de una frontera.
5.  **Reserva generación abierta para artefactos.** Diagramas, simulaciones, explicadores temporales, y scratchpads visuales son un buen fit. Workflows core no lo son.

La arquitectura termina pareciendo así:

```txt
Intent del usuario
  -> runtime de agente
  -> llamadas de herramienta/datos
  -> resultado estructurado
  -> decisión de UI
  -> componente confiable, spec declarativa, o widget sandboxeado
  -> acción del usuario
  -> stream de estado/eventos de vuelta al agente
```

Ese loop es el producto real. La caja de chat es solo un posible dispositivo de entrada.

## La evaluación debería incluir la UI

Los equipos de LLM están lentamente aprendiendo a evaluar prompts y outputs de modelo. UI generativa añade otra surface area: la interface misma puede estar equivocada.

Al mínimo, salva estos artefactos para cada UI generada:

- prompt y contexto de herramienta
- llamadas a herramientas y resultados de herramientas
- spec de UI generada o selección de componente
- nombre de componente renderizado y props
- labels visible para el usuario
- acciones attachadas a botones/formularios
- actualizaciones de estado model-visible desde la UI
- historia de acciones del usuario

Luego escribe checks como:

- cada acción destructiva debe tener un componente de confirmación
- labels de botones deben coincidir con semántica de acción
- specs de render solo pueden referenciar componentes permitidos
- totales visible para el usuario deben coincidir con totales de resultado de herramienta
- formularios no pueden pedir campos fuera del scope de tarea
- widgets no pueden recibir secrets que solo el modelo necesitaba
- metadata oculta no debe contradecir labels visibles

Esto suena tedioso. También es de donde viene la confianza de producción.

## Una nota sobre estabilidad de proyectos

Cada protocolo major en este espacio es pre-1.0. Verificado el 8 de mayo, 2026; planifica para cambio y revisa los docs actuales antes de hacer un bet de plataforma.

**Vercel AI SDK RSC** — el feature headline original de "Generative UI" — tuvo su desarrollo pausado en octubre de 2024 ([Discussion #3251](https://github.com/vercel/ai/discussions/3251)) debido a limitaciones arquitecturales sin fix cercano. **json-render** (Vercel Labs) emergió como la dirección de reemplazo: catalog-based, framework-agnostic, sin coupling de RSC. Parece haber ganado atención de web-developers rápidamente desde su launch de early-2026. La razón probable es DX: json-render funciona inmediatamente en un proyecto React estándar; el scope cross-platform de A2UI añade fricción de setup.

**A2UI** (Google) es pre-1.0 con breaking changes entre versiones minor y comunicaciones de roadmap inconsistentes. Su ventaja es reach cross-platform genuino (web, Flutter, SwiftUI) que json-render no address. Para casos de uso pure-web hoy, json-render parece tener mejor coverage de tooling; para escenarios cross-platform o remote-agent, el diseño de A2UI es más apropiado. Convergencia entre los dos specs es posible — Vercel ha experimentado con output A2UI-compatible desde json-render.

**AG-UI** (CopilotKit) también es pre-1.0. La confusión más común es el nombre: AG-UI es un protocolo de transporte, no un framework de UI. Define *cómo* los eventos fluyen entre agente y frontend; lo que renderizas sigue siendo tu decisión. El concepto es sólido y broadly adoptado. El spec sigue evolucionando.

## Mi take

La UI generativa no reemplazará interfaces de producto cuidadosamente diseñadas. Reemplazará la suposición lazy que un transcript de chat es la interface universal para AI.

Los mejores sistemas no dejarán al modelo freestyle todo. Le darán un set pequeño y sharp de building blocks producto-nativos; una conexión runtime confiable; fronteras de seguridad claras; y suficiente libertad para adaptar la interface a la tarea.

El futuro no es "el modelo escribie tu frontend."

El futuro es más cercano a: **tu frontend se convierte en un instrumento que el agente puede tocar, pero tú aún decides qué se le permite sonar al instrumento.**
