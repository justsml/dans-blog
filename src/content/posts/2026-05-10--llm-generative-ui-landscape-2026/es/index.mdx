"IU Generativa" significa al menos cinco cosas distintas dependiendo de quién lo diga.

- Interfaces de chat que incrustan tarjetas de producto a partir de llamadas a herramientas del modelo
- Especificaciones JSON en tiempo de ejecución que el frontend renderiza como árboles de componentes
- Iframes sandbox devueltos por herramientas MCP a aplicaciones anfitrionas (desde pedidos de tickets, reserva de hoteles hasta renderización de mapas, widgets de checkout)
- Protocolos de eventos que transmiten el estado del agente al frontend
- v0, Lovable y Bolt: herramientas de IA que escriben React en tiempo de diseño

Son conceptos relacionados, pero viven en distintas capas de la pila y conllevan distintos perfiles de riesgo, distintos costes de implementación y distintos casos de uso apropiados. Confluirlos convierte cada discusión de arquitectura en un caos.

Este es el mapa que quiero cuando decido a qué parte de la pila recurrir.

---

## Lo que la IU Generativa No Es

Antes de definir lo que es, tres cosas a dejar aparte:

**Generación de código en tiempo de diseño** — v0, Lovable, Bolt, Cursor componiendo componentes React. Estas herramientas generan código que los desarrolladores revisan y commitean. La IA se ejecuta en tiempo de desarrollo. Lo que llega al usuario es estático desde su perspectiva. Es una categoría excelente de herramientas. No es lo que significa "IU generativa en tiempo de ejecución".

**Autocompletado de formularios asistido por IA** — el modelo rellena valores de campos a partir del contexto. La estructura de la interfaz sigue fija; solo se mueve el contenido. Es un patrón útil. No es IU generativa.

**IA escribiendo HTML crudo en una página** — el modelo produce cadenas `<div>` y `<button>` que se inyectan vía `innerHTML` o `dangerouslySetInnerHTML`. Esto *es* IU generativa en tiempo de ejecución en el sentido más técnico. Es también la versión más peligrosa, y la que todo framework maduro en este espacio existe para evitar. HTML crudo generado por IA significa riesgo de XSS, atributos inaccesibles, estilos inconsistentes y estructura alucinada. El resto de este artículo trata de hacer algo mejor.

---

## Una Definición Operativa

IU generativa en tiempo de ejecución significa: **el modelo determina qué componente de interfaz o composición de componentes ve el usuario, según el estado de la conversación o tarea.**

No las palabras. La interfaz.

El caso más simple: tu asistente de reserva de vuelos llama a una herramienta `search_flights`. En lugar de devolver texto plano ("Aquí tienes tres opciones..."), renderiza un componente `<FlightResultsCard>` con vuelos seleccionables, toggles de clase de asiento y un botón "Book". El modelo decidió que una tarjeta estructurada era la respuesta adecuada. El desarrollador decidió cómo se ve esa tarjeta y qué hace "Book".

El caso más complejo: un agente de análisis financiero recibe una pregunta sobre una cartera y decide componer una respuesta con un `MetricGroup` mostrando cifras clave, un gráfico `RiskBreakdown`, una tabla `ScenarioComparison` y un `PolicyNotice`. El modelo ensambló ese layout a partir de un catálogo de componentes pre-aprobados. El desarrollador definió cada componente. El modelo eligió cuáles usar y qué datos poner en ellos.

Ambos casos son IU generativa. Difieren en cuánta libertad de composición tiene el modelo, lo cual determina tanto la riqueza de las posibles salidas como la complejidad de lo que puede salir mal.

## Los Tres Patrones

Todo el espacio se colapsa en tres patrones, cada uno con una gramática de salida diferente.

![Un diagrama de espectro mostrando tres patrones: solo llamadas a herramientas a la izquierda (más seguro), catálogo de componentes en el centro y generación abierta a la derecha (más expresivo).](./output-grammar-spectrum.svg)

_Cada decisión de IU generativa es un punto en este espectro. Empieza por la izquierda._

### Patrón 1: Renderización de herramienta a componente

El modelo llama a una herramienta nombrada. Tu aplicación tiene un mapa de nombres de herramientas a componentes. La llamada a la herramienta dispara la renderización de un componente.

```tsx
// El modelo llama: { name: "show_flight_results", args: { flights: [...] } }

useCopilotAction({
  name: "show_flight_results",
  render: ({ args }) => <FlightResultsCard flights={args.flights} />,
});
```

Es el patrón más seguro porque el layout nunca viene del modelo. El modelo decide *cuándo* mostrar un componente y *qué datos* poblarlo. Tus desarrolladores siguen siendo propietarios del código del componente, el diseño visual, la implementación de accesibilidad y cada caso extremo en la lógica de renderización.

`useChat` del Vercel AI SDK con handlers de herramientas hace esto. La renderización de herramientas de assistant-ui hace esto. La "IU Generativa Estática" de CopilotKit es este patrón. La mayoría de los copilot UIs en producción que funcionan de forma fiable están haciendo esto.

**Apropiado cuando**: el conjunto de cosas que podrías querer mostrar es enumerable en tiempo de desarrollo. Confirmaciones de reserva, resultados de búsqueda, resúmenes de cuenta, widgets de aprobación. Si puedes enumerar los escenarios, este patrón los cubre.

### Patrón 2: Composición de catálogo de componentes

El modelo emite un árbol JSON tipado que referencia componentes de un catálogo definido por el desarrollador. Tu frontend tiene un renderer que recorre el árbol e instancia cada componente.

```json
[
  { "type": "metric_group", "metrics": [
    { "label": "MRR", "value": "$82,400", "delta": "+12%" },
    { "label": "Churn", "value": "2.1%", "delta": "-0.4%" }
  ]},
  { "type": "line_chart", "title": "30-day growth", "data_ref": "mrr_series" },
  { "type": "insight_callout", "text": "Expansion revenue driving the delta — avg seat count up 18%." }
]
```

El modelo compuso ese layout. Un `MetricGroup`, un `LineChart`, un `InsightCallout`. Pero tú definiste qué significa cada tipo de componente, qué props acepta y cómo se renderiza. Si el modelo intenta emitir `{ "type": "custom_untested_thing" }`, tu validación de schema lo detecta y el renderer lo ignora o rechaza.

Este es el patrón detrás de `json-render`, `A2UI`, `Hashbrown`, `OpenUI` y `Tambo`. El trabajo de ingeniería clave es **el diseño del catálogo** — decidir qué tipos de componentes existen, cómo son sus schemas y qué se le permite y no se le permite componer al modelo.

**Apropiado cuando**: la estructura de lo que quieres mostrar varía legítimamente según los datos o la solicitud del usuario. Dashboards que se adaptan a lo que es notable en las cifras. Informes que muestran distintas secciones según el contexto. Paneles de workflow que cambian según el paso en que esté un agente.

### Patrón 3: Generación abierta

El modelo escribe HTML, SVG, Canvas o WebGL que se renderiza dentro de un iframe sandbox con una Content Security Policy estricta.

Es apropiado para cosas donde ningún catálogo fijo de componentes servirá: visualizaciones de algoritmos, diagramas arquitectónicos, gráficos ad hoc, arte generativo, simulaciones educativas. El límite del iframe hace el trabajo de seguridad aquí; quítalo y vuelves al problema de inyección de HTML crudo del inicio de este artículo.

`CopilotKit/OpenGenerativeUI` es la mejor implementación de referencia actual de este patrón. El sandbox elimina scripts, limita el paso de mensajes y mantiene el artefacto generado lejos del estado privilegiado de tu aplicación.

**Apropiado cuando**: genuinamente necesitas salida visual arbitraria — diagramas explicativos únicos, simulaciones dinámicas, artefactos creativos. No uses esto para UI transaccional. Una confirmación de checkout no necesita un iframe sandbox.

### Más allá de los tres patrones: LLMs controlando píxeles directamente

Hay una cuarta dirección emergente que no encaja limpiamente en ninguno de estos patrones: los LLMs impulsando **experiencias inmersivas tipo juego** controlando la salida visual de forma más directa que un iframe sandbox.

La distinción canónica dentro de la IU generativa es **iframe HTML vs. catálogo JSON**:

- **Iframe HTML** — el modelo escribe HTML, SVG, Canvas o WebGL que se renderiza en un sandbox isolado. Máxima libertad expresiva; la seguridad depende totalmente del límite del iframe. Ejemplos: Anthropic Artifacts, OpenGenerativeUI.
- **Catálogo JSON** — el modelo emite un payload estructurado limitado a un catálogo de componentes definido por el desarrollador; tu renderer instancia componentes de confianza pre-construidos a partir de esa spec. El modelo decide *qué* mostrar; tú decides *cómo* se renderiza. Ejemplos: json-render, A2UI.

Más allá de estos, demos muy recientes apuntan a un tercer modo donde el modelo no está eligiendo componentes ni escribiendo HTML sandbox — está controlando el canvas de forma más directa. Proyectos como [HunyuanWorld de Tencent](https://arxiv.org/abs/2502.01999), que genera entornos 3D explorables a partir de una sola imagen, y arquitecturas de juegos donde los LLMs generan mapas, NPCs y misiones en tiempo de ejecución en lugar de llamar a un catálogo de componentes, sugieren un futuro donde el modelo es más un director de juego que un renderer de formularios. La inferencia de LLM en navegador vía WebGPU ([WebLLM](https://mlc.ai/web-llm/)) está empujando la misma frontera localmente.

Este territorio es genuinamente excitante y genuinamente incipiente. No hay frameworks estables para construir productos de producción aún. Cubriré este enfoque en un artículo dedicado cuando eso cambie.

## Los Protocolos: AG-UI y A2UI

AG-UI y A2UI son los dos estándares principales en la capa de protocolo. Resuelven problemas distintos y no son competidores.

### AG-UI

**GitHub**: [ag-ui-protocol/ag-ui](https://github.com/ag-ui-protocol/ag-ui)

AG-UI es un protocolo basado en eventos para la comunicación entre agentes de IA y aplicaciones frontend. Define unos 16 tipos de eventos: `TEXT_MESSAGE_START`, `TEXT_MESSAGE_CONTENT`, `TOOL_CALL_START`, `TOOL_CALL_END`, `STATE_SNAPSHOT`, `STATE_DELTA`, y demás. El transporte depende de ti — SSE, WebSockets, webhooks, todos funcionan. El formato es deliberadamente laxo para permitir adopción amplia.

AG-UI no define cómo se ve tu UI. Define cómo se comunica el agente *con* tu frontend. Piensa en él como la capa de protocolo en cable que permite a tu app React suscribirse a un agente LangGraph de la misma forma que a un agente CrewAI, sin cambiar el código frontend.

CopilotKit creó AG-UI a partir de su trabajo con LangGraph y CrewAI. Ha sido adoptado por LangChain, Mastra, PydanticAI y otros. Microsoft ha publicado una guía de integración AG-UI. Si estás construyendo un frontend multi-agente y necesitas desacoplar frameworks de backend del código frontend, AG-UI es la respuesta.

**Una aclaración que confunde a la gente**: AG-UI no es un framework de UI. No te dice qué renderizar. Te dice *que* el agente dijo algo, llamó a una herramienta o actualizó estado compartido. Qué renderizas en respuesta sigue siendo tu decisión.

### A2UI

**GitHub**: [google/A2UI](https://github.com/google/A2UI) · Spec: [a2ui.org](https://a2ui.org/)

A2UI es la especificación declarativa de Google para lo que envían los agentes cuando quieren mostrar UI. Donde AG-UI responde "¿cómo se comunica el agente?", A2UI responde "¿qué formato usa el agente para describir un layout de componentes?".

A2UI usa un formato JSONL plano: un descriptor de componente por línea, cada uno con un ID, un tipo y datos. Plano es intencional. Árboles anidados requieren que el modelo conozca la estructura completa antes de poder empezar a streamear. Una lista plana permite al modelo emitir cada componente mientras "piensa" en él, lo que significa que tu frontend puede empezar a renderizar la primera tarjeta de KPI mientras el modelo aún está decidiendo si añadir un gráfico.

```jsonl
{"id":"h1","type":"kpi_card","title":"MRR","value":"$82,400","delta":"+12%"}
{"id":"h2","type":"kpi_card","title":"Churn","value":"2.1%","delta":"-0.4%"}
{"id":"c1","type":"line_chart","title":"30-day MRR","data_ref":"mrr_series"}
{"id":"t1","type":"data_table","cols":["Month","MRR","Net New"],"data_ref":"monthly"}
```

A2UI es consciente de la seguridad: la spec es un formato de datos, no código ejecutable. El catálogo de componentes es pre-definido por el desarrollador; el agente solo puede referenciar tipos en ese catálogo. Un renderer A2UI que recibe un nombre de tipo desconocido lo ignora.

El formato "Open-JSON-UI" de CopilotKit es compatible con A2UI. Si estás eligiendo un formato de spec para un catálogo de componentes hoy, A2UI es el que tiene el respaldo multiplataforma más amplio.

**Nota sobre estabilidad**: A2UI es pre-1.0 — v0.9 cuando se comprobó por última vez el 8 de mayo de 2026 — y ha shippeado breaking spec changes entre versiones minor. Las comunicaciones de Google sobre la roadmap han sido esporádicas y algunos renderers (Lit, Flutter) han quedado rezagados respecto a updates de la spec. Reserva tiempo para spec drift si estás construyendo sobre ella hoy. Para casos puramente web, json-render parece tener tooling más completo actualmente. La ventaja a largo plazo de A2UI es alcance multiplataforma (web, Flutter, SwiftUI, Android) que json-render no tiene.

### MCP Apps

**GitHub**: [modelcontextprotocol](https://github.com/modelcontextprotocol) · Relacionado: [mcp-ui](https://github.com/MCP-UI-Org/mcp-ui)

MCP comenzó como un protocolo para conectar LLMs a herramientas y datos. La extensión Apps permite que las herramientas MCP devuelvan no solo datos, sino artefactos de UI interactivos: componentes React, formularios, dashboards, mapas.

El modelo de seguridad es estricto y eso es precisamente el punto: todo se renderiza en un iframe sandbox con permisos limitados, los templates se pre-declaran para que la app anfitriona pueda revisarlos, y toda comunicación es JSON-RPC auditable. Es el modelo correcto para tool providers — un server MCP de Shopify puede devolver un widget de checkout; un servicio de mapas puede devolver un mapa embebible. La app anfitriona no es propietaria ni confía en el código de ese widget.

MCP Apps es la elección correcta cuando la UI *pertenece al tool provider*, no a tu aplicación. Para UI que vive en el dominio de tu aplicación, quédate con el Patrón 1 o 2.

## Los Frameworks de JavaScript/TypeScript

### CopilotKit

**GitHub**: [CopilotKit/CopilotKit](https://github.com/CopilotKit/CopilotKit) · Ejemplos: [CopilotKit/generative-ui](https://github.com/CopilotKit/generative-ui)

CopilotKit es el framework más completo para aplicaciones frontend nativas de agentes. Gestiona el ciclo de vida completo: conexión con backends de agentes vía AG-UI, gestión bidireccional del estado de conversación, renderización de componentes de IU generativa y la infraestructura de estado compartido que permite a agentes y usuarios modificar los mismos datos.

El modelo de tres patrones se mapea limpiamente a las APIs de CopilotKit:
- `useCopilotAction` con callback `render` → Patrón 1
- Renderización A2UI/Open-JSON-UI → Patrón 2
- Artefactos sandbox `OpenGenerativeUI` → Patrón 3

La característica importante de CopilotKit que se discute menos es **estado compartido y humano-en-el-bucle**: el agente puede leer y escribir estado de la aplicación, el usuario puede leerlo y escribirlo, y los cambios fluyen bidireccionalmente. Esto es lo que hace que los copilot UIs se sientan como una colaboración real en vez de una caja de chat grapada a un producto.

### Vercel AI SDK

**GitHub**: [vercel/ai](https://github.com/vercel/ai) · Docs: [ai-sdk.dev](https://ai-sdk.dev/)

El Vercel AI SDK es el baseline de facto en TypeScript para apps de IA. Para IU generativa en concreto:

**`useObject`** streamea un objeto JSON estructurado desde el servidor mientras se genera. Defines un schema Zod; el SDK parsea el JSON parcial y dispara re-renderizaciones conforme llegan los campos. Es el camino más fluido al Patrón 2 en una app Next.js.

```tsx
const { object: dashboard } = useObject({
  api: "/api/generate-dashboard",
  schema: z.object({
    title: z.string(),
    metrics: z.array(z.object({ label: z.string(), value: z.number() })),
    insights: z.array(z.string()),
  }),
});
```

**`useChat` con tool handlers** → Patrón 1. El modelo llama a herramientas; tú mapeas nombres de herramientas a componentes.

**AI Elements** ([elements.ai-sdk.dev](https://elements.ai-sdk.dev/)) proporciona primitivas de UI ready-made para pairing con el SDK.

**Nota sobre la trayectoria confusa aquí**: En octubre de 2024, Vercel anunció en la [GitHub Discussion #3251](https://github.com/vercel/ai/discussions/3251) que AI SDK RSC — el patrón de streaming de React Server Components promovido como feature headline de "Generative UI" en SDK 3.0 — se pausaba indefinidamente debido a "varias limitaciones persistentes" sin buenas soluciones a corto plazo. Los equipos que habían construido estrategias de producto sobre RSC streaming se vieron pillados off-guard. Las APIs `generateObject`/`streamObject` también fueron deprecated posteriormente en SDK 6.0. La migración recomendada desde AI SDK RSC es el patrón `useObject` de arriba, o json-render para generación basada en catálogo.

### assistant-ui

**GitHub**: [assistant-ui/assistant-ui](https://github.com/assistant-ui/assistant-ui)

assistant-ui es un conjunto de primitivas React componibles para construir interfaces de chat de calidad de producción. Es la respuesta correcta cuando necesitas chat UX pulido — burbujas de mensaje, streaming de tokens, acciones de copiar/editar/regenerar, estados de pensamiento — y quieres traer tu propio backend y tu propia renderización de herramientas.

Funciona bien emparejado con cualquier backend (OpenAI, Anthropic, modelos locales, endpoints custom) y gestiona la renderización de tool calls vía un modelo familiar de slot/render prop.

### json-render

**GitHub**: [vercel-labs/json-render](https://github.com/vercel-labs/json-render) · Docs: [json-render.dev](https://json-render.dev/)

json-render operacionaliza el Patrón 2 con un enfoque opinado y batteries-included. Obtienes un catálogo de componentes pre-construido (componentes shadcn/ui con schemas Zod), un renderer y un loop de generación cerrado donde el modelo está limitado al catálogo por schema.

Las features distintivas:
- **Renderización multi-target**: la misma spec JSON puede renderizar a una app web React, una app móvil React Native, un PDF, un email HTML o un vídeo Remotion. Es genuinamente útil para informes.
- **Renderización progresiva**: los componentes aparecen conforme el modelo los streamea, no después de que llegue la spec completa.
- **Constraints de schema cerrados**: el catálogo está diseñado para que el modelo no pueda alucinar tipos de componentes válidos-pero-desconocidos.

Si estás construyendo generación de dashboards o informes y quieres saltar el trabajo de infraestructura de diseñar tu propio catálogo, json-render es el camino más rápido para apps web.

**Sobre momentum**: json-render se lanzó desde Vercel Labs a principios de 2026 y parece haber captado atención de desarrolladores web rápidamente porque es inmediatamente útil en proyectos React/Next.js estándar. Eso dicho, json-render sigue en pre-1.0 y la relación entre json-render y A2UI sigue sin resolverse — Vercel ha experimentado con output compatible con A2UI, así que convergencia es posible. Para cross-platform (mobile nativo, múltiples frameworks), A2UI es la mejor apuesta a largo plazo.

### Hashbrown

**GitHub**: [liveloveapp/hashbrown](https://github.com/liveloveapp/hashbrown)

Hashbrown toma un enfoque distintivo: en vez de construir una capa de UI de IA separada, embede la selección de componentes de IA directamente en tu app React o Angular existente. Expones los componentes de tu app al LLM; el LLM selecciona cuáles renderizar y puede invocar tools client-side.

Es la herramienta correcta cuando quieres infundir inteligencia en superficies de producto que no son "chat" — una página de producto que adapta su layout, un panel de settings que surfacea las opciones correctas, un editor de workflow que sugiere el próximo paso.

### OpenUI

**GitHub**: [thesysdev/openui](https://github.com/thesysdev/openui) · Docs: [openui.com](https://www.openui.com/)

OpenUI reemplaza JSON con un formato line-oriented code-like ("OpenUI Lang") diseñado para renderización progresiva y eficiencia de tokens. La claim es ~67% menos tokens que JSON equivalente para layouts complejos.

El tradeoff es madurez del ecosistema — OpenUI es más nuevo y el tooling es más delgado que los enfoques basados en JSON. Pero si el coste de tokens es una constraint significativa y estás generando layouts complejos a alta frecuencia, la eficiencia del formato es real.

### Tambo

**GitHub**: [tambo-ai/tambo](https://github.com/tambo-ai/tambo)

Tambo se centra en selección de componentes stateful: la IA selecciona componentes y puede interactuar con ellos vía tools client-side, manteniendo el estado del componente a lo largo de la conversación. Bueno para casos de uso donde elementos de UI persisten entre turnos — un componente de filtro que el usuario ajusta mientras la IA continúa razonando sobre los datos filtrados.

## La Capa Python

El ecosistema Python aborda las interfaces de IA de forma diferente. Estas herramientas están optimizadas para demos de modelos de ML, aplicaciones de datos y tooling interno — no para apps de consumo de producción con composición de layout impulsada por agentes.

No es una crítica. Para los casos de uso correctos, Gradio y Streamlit son las únicas herramientas que necesitas.

### Gradio

**GitHub**: [gradio-app/gradio](https://github.com/gradio-app/gradio) · PyPI: `gradio`

El valor core de Gradio: escribes una función Python; Gradio la envuelve en una UI web. La clase `Interface` son 3 líneas para un clasificador de imágenes. `ChatInterface` son 10 líneas para un chatbot. `Blocks` te da control de layout fino cuando lo necesitas.

La "IU generativa" en Gradio la define el desarrollador Python, no el modelo. La visibilidad y configuración de componentes pueden cambiar dinámicamente según outputs del modelo, pero el catálogo de componentes es estático — no le estás pidiendo al modelo que componga layouts.

Gradio es el estándar para HuggingFace Spaces y el ecosistema de demos de ML. Tiene millones de descargas mensuales y alimenta una gran fracción del landscape de demos de IA.

**Recurre a Gradio cuando**: eres un desarrollador Python construyendo una demo de modelo de ML, un prototipo de investigación o una tool interna, y no quieres tocar JavaScript.

### Streamlit

**GitHub**: [streamlit/streamlit](https://github.com/streamlit/streamlit)

El modelo de Streamlit es más opinado: un script Python se ejecuta end-to-end en cada interacción. Llamas `st.chat_message()`, `st.dataframe()`, `st.plotly_chart()`. El framework gestiona el layout.

El modelo de full-script-rerun parece ineficiente pero es sorprendentemente ergonómico para chatbots de IA que acumulan historial de conversación — el script completo se re-ejecuta, el historial del chat está en session state y el output es determinista. Streamlit ahora tiene soporte first-party para los principales proveedores de LLM y se integra nativamente con Snowflake Cortex.

**Recurre a Streamlit cuando**: estás construyendo una app de datos powered-by-IA, una tool de reporting interna o un dashboard backed-by-ML en Python y quieres el deployment path más simple posible.

### LangChain y Haystack

Son frameworks de orquestación de backend, no frameworks de UI. Aparecen en cualquier mapa honesto de la pila de IU generativa porque típicamente son la capa donde se generan outputs estructurados antes de enviarse al frontend.

**LangChain** ([langchain-ai/langchain](https://github.com/langchain-ai/langchain)): `.with_structured_output()` en cualquier LLM te da generación de JSON constrained-by-Pydantic. El decorador `@tool` con generación automática de schemas es la forma más limpia de definir qué tools puede llamar el modelo. LangChain alimenta resultados estructurados a cualquier capa frontend que estés usando.

**Haystack** ([deepset-ai/haystack](https://github.com/deepset-ai/haystack)): arquitectura de pipeline modular con fuerte soporte RAG. Hayhooks envuelve pipelines de Haystack como endpoints HTTP — incluyendo endpoints MCP-compatible. Si tu IU generativa necesita un backbone de retrieval, la arquitectura de pipeline de Haystack lo gestiona limpiamente.

Ninguno de los dos frameworks owns la capa de UI. Generan los datos que tu frontend (Patrón 1, 2 o 3) renderiza.

## Referencia de Features

Usa el catálogo de arriba como orientación, no como shopping list. La pila suele colapsar a una elección en cada capa:

| Necesidad | Empieza aquí |
|-----------|--------------|

## Velocidad del Ecosistema y Terreno Inestable

Este espacio se mueve rápido y varios proyectos han shippeado comunicaciones confusas junto con su código. Verificado por última vez el 8 de mayo de 2026; trata las notas sobre el estado de los proyectos aquí como una lectura timestamped, no como un verdict permanente.

**Vercel AI SDK RSC** era la feature flagship de IU Generativa cuando SDK 3.0 se lanzó. Vercel pausó su desarrollo en octubre de 2024 ([Discussion #3251](https://github.com/vercel/ai/discussions/3251)) citando limitaciones arquitectónicas con React Server Components sin solución a corto plazo. Los equipos que habían construido sobre él estaban comprensiblemente frustrados. Sigue en la docs pero no es el path recomendado; `useObject` lo es.

**json-render** (Vercel Labs) es la nueva dirección — una alternativa basada en catálogo y agnóstica de framework que evita los problemas de coupling con RSC. Está en pre-1.0 y parece haber despertado interés fuerte temprano entre desarrolladores React/web. La razón probable de DX: json-render es inmediatamente útil en un proyecto React/Next.js estándar, mientras que el scope cross-platform de A2UI añade setup friction. Cómo se resuelve esto conforme ambas specs maduren es genuinamente incierto. Vercel ha explorado compatibilidad A2UI en json-render, sugiriendo que convergencia es posible.

**A2UI** (Google) está en pre-1.0 (v0.9 cuando se comprobó por última vez) con breaking changes entre versiones minor y comunicaciones inconsistentes de Google sobre su roadmap. Es la elección correcta para alcance cross-platform (web + Flutter + SwiftUI) que json-render no aborda, y tiene backing enterprise significativo. Para proyectos puramente web ahora mismo, la DX es más rough.

**AG-UI** (CopilotKit) también está en pre-1.0. La confusión más común: el nombre hace que suene como un framework de UI. No lo es — es un protocolo de transporte. AG-UI define cómo fluyen los eventos entre backends de agentes y tu frontend; qué renderizas en respuesta sigue siendo tu decisión. Ese modelo mental es sólido y ampliamente adoptado, pero la spec pre-1.0 significa que edge cases siguen sin resolverse.

El upshot práctico: **cada major player aquí está en pre-1.0**. Planifica para API changes. Los patrones — tool-to-component, catalog composition, sandboxed generation — son suficientemente estables para construir sobre ellos. Las elecciones específicas de protocolo no.

## Diseño del Catálogo de Componentes: El Verdadero Trabajo de Ingeniería

La mayor parte de la complejidad interesante en el Patrón 2 no está en el renderer — está en el catálogo.

El catálogo es una **decisión de producto codificada como schema**. Responde: ¿cuáles son los objetos de UI significativos en este dominio? No "¿qué componentes React existen?" sino "¿qué necesita realmente ver e interactuar un usuario en este contexto?"

**El modo de fallo too-granular**: expones `Row`, `Column`, `Text`, `Button`, `Icon`. Ahora el modelo tiene que ser un frontend engineer. Generará layout mediocre que no matcheará tu design system, omitirá empty states, producirá markup inaccesible y cambiará su enfoque de respuesta a respuesta porque nada en el catálogo limita el output al lenguaje visual de tu producto.

**El modo de fallo too-coarse**: expones `WeatherCard`, `FlightCard`, `HotelCard`. El modelo no puede adaptarse cuando el usuario pide algo que no mapea a una card pre-baked. Recurre a texto.

**El middle útil**: componentes de nivel de dominio con slots constrained.

Un catálogo de app de viajes podría verse así:

```
TripSummary         — itinerario de un vistazo
FlightOptionList    — opciones de vuelo seleccionables con precios
HotelComparison     — cards de hoteles side-by-side
TravelerForm        — recogida de detalles del viajero
PolicyNotice        — callout de regulación/reglas de tarifa
BookingConfirmation — confirmación final con botón de acción
```

Un catálogo de app financiera podría verse así:

```
PortfolioSnapshot   — posiciones clave y P&L
TransactionTable    — transacciones filtrables y paginadas
RiskBreakdown       — métricas de allocation y volatilidad
ScenarioComparison  — modelado de escenarios side-by-side
ApprovalGate        — acción que requiere confirmación humana
```

El catálogo suena como el vocabulario de tu producto. Codifica tus decisiones de UX, tus requisitos de accesibilidad, tu handling de empty states y tus patterns de acciones peligrosas en el código de componentes. El modelo se encarga de organizar esas piezas. Tú sigues decidiendo cómo se ve cada pieza y qué se le permite hacer.

**Reglas de design de schema que reducen alucinaciones**:

1. Mantén los valores de enum cortos y obvios. `"type": "bar_chart"` no `"type": "data-visualization-bar-type-vertical"`.
2. Haz imposible la composición inválida. Si un `PolicyNotice` solo puede aparecer al final de un layout, no lo pongas en el mismo nivel de schema que cosas que pueden aparecer en cualquier lugar.
3. Usa required fields generosamente. Un campo opcional es un campo que el modelo podría omitir y tu renderer tiene que handle como null.
4. Prueba el catálogo contra prompts reales antes de shippear. Guarda las specs generadas; revisa para violations de schema, valores de campo alucinados y composiciones que son técnicamente válidas pero semánticamente incorrectas.

## Trampas Comunes

**Trampa: tratar JSON válido como comportamiento seguro.** Schema validation confirma la estructura. No dice nada sobre si la acción attached a un botón matchea su label, si un total matchea los datos de los que deriva, o si un componente de UI está haciendo algo que el usuario no esperaba. Specs de UI generada necesitan review semántica, no solo schema validation. Como mínimo, acciones destructivas deberían requerir un componente de confirmación, y los labels de esos componentes deberían ser testados contra las acciones que triggeran.

**Trampa: exponer design primitives en vez de product primitives.** Si el modelo tiene que decidir si usar padding de 16px o 20px, le has dado el abstraction level equivocado. Los componentes de dominio deben encode product taste. El modelo debería estar componiendo behaviour, no managing presentation details.

**Trampa: usar IU generativa donde UI static serviría.** Si la estructura de lo que quieres mostrar es enumerable en tiempo de desarrollo — y normalmente lo es — el Patrón 1 con componentes pre-construidos es más rápido, más seguro y más consistente. La IU generativa justifica su complejidad cuando la estructura varía genuinamente según los datos o el contexto de la tarea.

**Trampa: skippear accesibilidad.** Los LLMs alucinan violaciones de WCAG. Emitirán `role="region"` en elementos interactivos, generarán forms con labels missing y producirán contrast ratios que fallan WCAG AA. Tu librería de componentes puede ser fully accessible; composiciones AI-generated de esos componentes no son automáticamente accesibles. Testea el full render path, no solo los componentes en isolation.

**Trampa: confluir el protocolo y el framework.** AG-UI no es un framework frontend. A2UI no es una librería React. Son wire formats y protocols de eventos. Todavía necesitas un framework frontend para implementarlos. CopilotKit implementa AG-UI y A2UI. json-render implementa el pattern de catálogo A2UI/Open-JSON-UI. Son capas diferentes.

## Recomendaciones por Caso de Uso

**Añadir un copilot a una app SaaS existente**: Empieza con el Patrón 1 (tool-to-component). Usa Vercel AI SDK `useChat` o CopilotKit. Mapea tus top 5–10 acciones de agente a componentes pre-construidos. Shippea eso, mide, y luego expande el catálogo solo si los users demuestran necesidad de composición más rica.

**Generación de dashboards desde lenguaje natural**: Usa el Patrón 2 con json-render o un catálogo A2UI custom. Define un catálogo de 8–15 tipos de componentes cubriendo tus chart types, metric cards y variants de table. Feed el schema al modelo; deja que componga el layout. Construye validation que capture tipos desconocidos antes de que lleguen al renderer.

**Frontend multi-agente**: Usa CopilotKit con AG-UI. El stream de eventos gestiona streaming real-time a través de agent backends; el estado compartido gestiona handoff entre agentes; el pattern HITL gestiona approval gates.

**Construir dentro de ChatGPT u otro host MCP**: Usa MCP Apps. Define tu tool como una data tool que fetcha y razona, y una render tool separada que requesta un widget. Mantén business logic fuera del widget template.

**Demos de modelos de ML y apps de datos (team Python)**: Gradio para demos y HuggingFace Spaces. Streamlit para apps de datos con interacción más compleja. Ninguno requiere JavaScript.

**Artefactos visuales, simulaciones, diagramas**: Usa el Patrón 3 (OpenGenerativeUI o equivalente). Establece iframe CSP estricto. Trata el output como untrusted user content desde el punto de vista de seguridad.

---

Los frameworks están madurando rápido. La convergencia de protocolos (AG-UI para streaming, A2UI/Open-JSON-UI para specs de catálogo) sigue in progress, pero la forma se está volviendo suficientemente clara para construir sobre ella.

Los challenges de ingeniería que más importan ahora no son framework selection. Son diseño de catálogo — decidir qué se le permite decir al modelo, lo que requiere claridad de producto más que skill técnica. Son validation semántica — testear que la UI generada hace lo que claima, no solo que pasa schema validation. Y son la gap de accesibilidad — construir catálogos donde cada componente, y cada composición de componentes, cumpla el bar de accesibilidad que exigirías a UI escrita a mano.

El modelo hará lo que le digas que haga dentro de la gramática que le dé. Haz que la gramática sea deliberada.
