"IU Generativa" significa al menos cinco cosas distintas dependiendo de quién lo diga.

- Interfaces de chat que incrustan tarjetas de producto a partir de llamadas a herramientas del modelo
- Especificaciones JSON en tiempo de ejecución que el frontend renderiza como árboles de componentes
- Iframes aislados (sandboxed) devueltos por herramientas MCP a aplicaciones anfitrionas (desde pedidos de tickets, reserva de hoteles hasta renderizado de mapas, widgets de pago)
- Protocolos de eventos que transmiten el estado del agente al frontend
- v0, Lovable y Bolt: herramientas de IA que escriben React en tiempo de diseño

Son conceptos relacionados, pero viven en diferentes capas de la pila y conllevan diferentes perfiles de riesgo, diferentes costes de implementación y diferentes casos de uso apropiados. Confundirlos convierte cada discusión de arquitectura en un caos.

Este es el mapa que quiero cuando decido a qué parte de la pila recurrir.

---

## Lo Que la IU Generativa No Es

Antes de definir lo que es, tres cosas que debemos dejar de lado:

**Generación de código en tiempo de diseño** — v0, Lovable, Bolt, Cursor componiendo componentes React. Estas herramientas generan código que los desarrolladores revisan y confirman (commit). La IA se ejecuta en tiempo de desarrollo. Lo que se envía es estático desde la perspectiva del usuario. Esta es una excelente categoría de herramientas. No es lo que significa "IU generativa en tiempo de ejecución".

**Autocompletado de formularios asistido por IA** — el modelo rellena los valores de los campos basándose en el contexto. La estructura de la interfaz sigue siendo fija; solo cambia el contenido. Este es un patrón útil. No es IU generativa.

**IA escribiendo HTML puro en una página** — el modelo genera cadenas `<div>` y `<button>` que se inyectan mediante `innerHTML` o `dangerouslySetInnerHTML`. Esto *es* IU generativa en tiempo de ejecución en el sentido más técnico. También es la versión más peligrosa, y la que todos los frameworks maduros en este espacio existen para evitar. El marcado puro generado por IA significa riesgo de XSS, atributos inaccesibles, estilo inconsistente y estructura alucinada. El resto de este artículo trata sobre cómo hacerlo mejor.

## Una Definición de Trabajo

IU generativa en tiempo de ejecución significa: **el modelo determina qué componente de interfaz o composición de componentes ve el usuario, basándose en el estado de la conversación o tarea.**

No las palabras. La interfaz.

El caso más simple: tu asistente de reserva de vuelos llama a una herramienta `search_flights`. En lugar de devolver texto plano ("Aquí tienes tres opciones..."), renderiza un componente `<FlightResultsCard>` con vuelos seleccionables, selectores de clase de asiento y un botón de "Reservar". El modelo decidió que una tarjeta estructurada era la respuesta correcta aquí. El desarrollador decidió cómo se ve esa tarjeta y qué hace "Reservar".

El caso más complejo: un agente de análisis financiero recibe una pregunta sobre una cartera y decide componer una respuesta con un `MetricGroup` que muestra números clave, un gráfico de `RiskBreakdown`, una tabla de `ScenarioComparison` y un `PolicyNotice`. El modelo ensambló ese diseño a partir de un catálogo de componentes pre-aprobados. El desarrollador definió cada componente. El modelo eligió cuáles usar y qué datos poner en ellos.

Ambos casos son IU generativa. Difieren en cuánta libertad de composición tiene el modelo, lo que determina tanto la riqueza de las posibles salidas como la complejidad de lo que puede salir mal.

## Los Tres Patrones

Todo el espacio se colapsa en tres patrones, cada uno con una gramática de salida diferente.

![Un diagrama de espectro que muestra tres patrones: solo llamadas a herramientas a la izquierda (más seguro), catálogo de componentes en el medio y generación abierta a la derecha (más expresivo).](./output-grammar-spectrum.svg)

_Cada decisión de IU generativa es un punto en este espectro. Empieza por la izquierda._

### Patrón 1: Renderizado de herramienta a componente

El modelo llama a una herramienta con nombre. Tu aplicación tiene un mapeo de nombres de herramientas a componentes. La llamada a la herramienta desencadena el renderizado de un componente.

```tsx
// El modelo llama a: { name: "show_flight_results", args: { flights: [...] } }

useCopilotAction({
  name: "show_flight_results",
  render: ({ args }) => <FlightResultsCard flights={args.flights} />,
});
```

Este es el patrón más seguro porque el diseño nunca proviene del modelo. El modelo decide *cuándo* mostrar un componente y *qué datos* usar para poblarlo. Tus desarrolladores siguen siendo los propietarios del código del componente, el diseño visual, la implementación de la accesibilidad y cada caso límite en la lógica de renderizado.

El `useChat` del Vercel AI SDK con manejadores de herramientas (`tool`) hace esto. El renderizado de herramientas de assistant-ui hace esto. La "IU Generativa Estática" de CopilotKit es este patrón. La mayoría de las interfaces de copiloto en producción que funcionan de manera fiable están haciendo esto.

**Apropiado cuando**: el conjunto de cosas que podrías querer mostrar es conocible en tiempo de desarrollo. Confirmaciones de reserva, resultados de búsqueda, resúmenes de cuenta, widgets de aprobación. Si puedes enumerar los escenarios, este patrón los cubre.

### Patrón 2: Composición de catálogo de componentes

El modelo emite un árbol JSON tipado que hace referencia a componentes de un catálogo definido por el desarrollador. Tu frontend tiene un renderizador que recorre el árbol e instancia cada componente.

```json
[
  { "type": "metric_group", "metrics": [
    { "label": "MRR", "value": "$82,400", "delta": "+12%" },
    { "label": "Churn", "value": "2.1%", "delta": "-0.4%" }
  ]},
  { "type": "line_chart", "title": "30-day growth", "data_ref": "mrr_series" },
  { "type": "insight_callout", "text": "Expansion revenue driving the delta — avg seat count up 18%." }
]
```

El modelo compuso ese diseño. Un `MetricGroup`, un `LineChart`, un `InsightCallout`. Pero tú definiste qué significa cada tipo de componente, qué propiedades (props) acepta y cómo se renderiza. Si el modelo intenta emitir `{ "type": "custom_untested_thing" }`, tu validación de esquema lo detecta y el renderizador lo ignora o rechaza.

Este es el patrón detrás de `json-render`, `A2UI`, `Hashbrown`, `OpenUI` y `Tambo`. El trabajo de ingeniería clave es el **diseño del catálogo** — decidir qué tipos de componentes existen, cómo se ven sus esquemas y qué se le permite y no se le permite componer al modelo.

**Apropiado cuando**: la estructura de lo que quieres mostrar varía legítimamente según los datos o la solicitud del usuario. Dashboards que se adaptan a lo que es notable en los números. Informes que muestran diferentes secciones según el contexto. Paneles de flujo de trabajo que cambian según el paso en el que se encuentra un agente.

### Patrón 3: Generación abierta

El modelo escribe HTML, SVG, Canvas o WebGL que se renderiza dentro de un iframe aislado (sandboxed) con una Política de Seguridad de Contenido (CSP) estricta.

Esto es apropiado para cosas donde ningún catálogo de componentes fijo servirá: visualizaciones de algoritmos, diagramas arquitectónicos, gráficos ad hoc, arte generativo, simulaciones educativas. El límite del iframe realiza el trabajo de seguridad aquí; si lo eliminas, vuelves al problema de la inyección de HTML puro mencionado al principio de este artículo.

`CopilotKit/OpenGenerativeUI` es la mejor implementación de referencia actual de este patrón. El sandbox elimina scripts, limita el paso de mensajes y mantiene el artefacto generado alejado del estado privilegiado de tu aplicación.

**Apropiado cuando**: realmente necesitas una salida visual arbitraria — diagramas explicativos únicos, simulaciones dinámicas, artefactos creativos. No uses esto para interfaces transaccionales. Una confirmación de pago no necesita un iframe aislado.

### Más allá de los tres patrones: LLMs controlando píxeles directamente

Existe una cuarta dirección emergente que no encaja limpiamente en ninguno de estos patrones: los LLMs impulsando **experiencias inmersivas tipo juego** al controlar la salida visual de forma más directa que un iframe aislado.

La distinción canónica dentro de la IU generativa es **HTML en iframe vs. catálogo JSON**:

- **HTML en iframe** — el modelo escribe HTML, SVG, Canvas o WebGL que se renderiza en un sandbox aislado. Máxima libertad expresiva; la seguridad depende totalmente del límite del iframe. Ejemplos: Anthropic Artifacts, OpenGenerativeUI.
- **Catálogo JSON** — el modelo emite un payload estructurado limitado a un catálogo de componentes definido por el desarrollador; tu renderizador instancia componentes de confianza pre-construidos a partir de esa especificación. El modelo decide *qué* mostrar; tú decides *cómo* se renderiza. Ejemplos: json-render, A2UI.

Más allá de estos, demostraciones muy recientes apuntan a un tercer modo donde el modelo no elige componentes ni escribe HTML aislado — está controlando el lienzo (canvas) de forma más directa. Proyectos como [HunyuanWorld de Tencent](https://arxiv.org/abs/2502.01999), que genera entornos 3D explorables a partir de una sola imagen, y arquitecturas de juegos donde los LLMs generan mapas, NPCs y misiones en tiempo de ejecución en lugar de llamar a un catálogo de componentes, sugieren un futuro donde el modelo es más un director de juego que un renderizador de formularios. La inferencia de LLM en el navegador mediante WebGPU ([WebLLM](https://mlc.ai/web-llm/)) está impulsando la misma frontera localmente.

Este territorio es genuinamente emocionante y genuinamente incipiente. Aún no hay frameworks estables para construir productos de producción. Cubriré este enfoque en un artículo dedicado una vez que eso cambie.

## Los Protocolos: AG-UI y A2UI

AG-UI y A2UI son los dos estándares principales en la capa de protocolo. Resuelven problemas diferentes y no son competidores.

### AG-UI

**GitHub**: [ag-ui-protocol/ag-ui](https://github.com/ag-ui-protocol/ag-ui)

AG-UI es un protocolo basado en eventos para la comunicación entre agentes de IA y aplicaciones frontend. Define unos 16 tipos de eventos: `TEXT_MESSAGE_START`, `TEXT_MESSAGE_CONTENT`, `TOOL_CALL_START`, `TOOL_CALL_END`, `STATE_SNAPSHOT`, `STATE_DELTA`, y así sucesivamente. El transporte depende de ti — SSE, WebSockets, webhooks, todos funcionan. El formato es deliberadamente flexible para permitir una adopción amplia.

AG-UI no define cómo se ve tu interfaz. Define cómo se comunica el agente *con* tu frontend. Piensa en él como la capa de protocolo de cable que permite a tu aplicación React suscribirse a un agente de LangGraph de la misma manera que se suscribe a un agente de CrewAI, sin cambiar el código del frontend.

CopilotKit creó AG-UI a partir de su trabajo con LangGraph y CrewAI. Ha sido adoptado por LangChain, Mastra, PydanticAI y otros. Microsoft ha publicado una guía de integración de AG-UI. Si estás construyendo un frontend multi-agente y necesitas desacoplar los frameworks de backend del código del frontend, AG-UI es la respuesta.

**Una aclaración que suele confundir a la gente**: AG-UI no es un framework de interfaz de usuario. No te dice qué renderizar. Te dice *que* el agente dijo algo, llamó a una herramienta o actualizó el estado compartido. Lo que renderices en respuesta sigue siendo tu decisión.

### A2UI

**GitHub**: [google/A2UI](https://github.com/google/A2UI) · Especificación: [a2ui.org](https://a2ui.org/)

A2UI es la especificación declarativa de Google para lo que envían los agentes cuando quieren mostrar una interfaz. Mientras que AG-UI responde "¿cómo se comunica el agente?", A2UI responde "¿qué formato usa el agente para describir el diseño de un componente?".

A2UI utiliza un formato JSONL plano: un descriptor de componente por línea, cada uno con un ID, un tipo y datos. El formato plano es intencional. Los árboles anidados requieren que el modelo conozca la estructura completa antes de poder empezar a transmitir. Una lista plana permite al modelo emitir cada componente mientras "piensa" en él, lo que significa que tu frontend puede empezar a renderizar la primera tarjeta de métricas mientras el modelo todavía está decidiendo si añadir un gráfico.

```jsonl
{"id":"h1","type":"kpi_card","title":"MRR","value":"$82,400","delta":"+12%"}
{"id":"h2","type":"kpi_card","title":"Churn","value":"2.1%","delta":"-0.4%"}
{"id":"c1","type":"line_chart","title":"30-day MRR","data_ref":"mrr_series"}
{"id":"t1","type":"data_table","cols":["Month","MRR","Net New"],"data_ref":"monthly"}
```

A2UI es consciente de la seguridad: la especificación es un formato de datos, no código ejecutable. El catálogo de componentes es pre-definido por el desarrollador; el agente solo puede hacer referencia a tipos en ese catálogo. Un renderizador A2UI que recibe un nombre de tipo desconocido lo ignora.

El formato "Open-JSON-UI" de CopilotKit es compatible con A2UI. Si estás eligiendo un formato de especificación para un catálogo de componentes hoy en día, A2UI es el que tiene el respaldo multiplataforma más amplio.

**Una nota sobre la estabilidad**: A2UI es pre-1.0 — v0.9 la última vez que se comprobó el 8 de mayo de 2026 — y ha enviado cambios de especificación que rompen la compatibilidad entre versiones menores. Las comunicaciones de Google sobre la hoja de ruta han sido esporádicas y algunos renderizadores (Lit, Flutter) se han quedado atrás respecto a las actualizaciones de la especificación. Reserva tiempo para posibles cambios en la especificación si estás construyendo sobre ella hoy. Para casos de uso puramente web, json-render parece tener herramientas más completas actualmente. La ventaja a largo plazo de A2UI es el alcance multiplataforma (web, Flutter, SwiftUI, Android) que json-render no tiene.

### MCP Apps

**GitHub**: [modelcontextprotocol](https://github.com/modelcontextprotocol) · Relacionado: [mcp-ui](https://github.com/MCP-UI-Org/mcp-ui)

MCP comenzó como un protocolo para conectar LLMs con herramientas y datos. La extensión Apps permite que las herramientas MCP devuelvan no solo datos, sino también artefactos de interfaz interactivos: componentes React, formularios, dashboards, mapas.

El modelo de seguridad es estricto y ese es el punto: todo se renderiza en un iframe aislado con permisos limitados, las plantillas se declaran previamente para que la aplicación anfitriona pueda revisarlas y toda la comunicación es JSON-RPC auditable. Este es el modelo correcto para proveedores de herramientas — un servidor MCP de Shopify puede devolver un widget de pago; un servicio de mapas puede devolver un mapa incrustable. La aplicación anfitriona no es la propietaria ni confía en el código de ese widget.

MCP Apps es la elección correcta cuando la interfaz *pertenece al proveedor de la herramienta*, no a tu aplicación. Para interfaces que viven en el dominio de tu aplicación, quédate con el Patrón 1 o 2.

## Los Frameworks de JavaScript/TypeScript

### CopilotKit

**GitHub**: [CopilotKit/CopilotKit](https://github.com/CopilotKit/CopilotKit) · Ejemplos: [CopilotKit/generative-ui](https://github.com/CopilotKit/generative-ui)

CopilotKit es el framework más completo para aplicaciones frontend nativas de agentes. Gestiona todo el ciclo de vida: conexión con backends de agentes sobre AG-UI, gestión del estado de conversación bidireccional, renderizado de componentes de IU generativa y provisión de la fontanería de estado compartido que permite a agentes y usuarios modificar los mismos datos.

El modelo de tres patrones se mapea limpiamente a las APIs de CopilotKit:
- `useCopilotAction` con un callback `render` → Patrón 1
- Renderizado de A2UI/Open-JSON-UI → Patrón 2
- Artefactos aislados `OpenGenerativeUI` → Patrón 3

La característica importante de CopilotKit que menos se discute es el **estado compartido y el humano en el bucle**: el agente puede leer y escribir el estado de la aplicación, el usuario puede leerlo y escribirlo, y los cambios fluyen bidireccionalmente. Esto es lo que hace que las interfaces tipo copiloto se sientan como una colaboración real en lugar de una caja de chat pegada a un producto.

### Vercel AI SDK

**GitHub**: [vercel/ai](https://github.com/vercel/ai) · Docs: [ai-sdk.dev](https://ai-sdk.dev/)

El Vercel AI SDK es la base de referencia de facto en TypeScript para aplicaciones de IA. Para IU generativa específicamente:

**`useObject`** transmite un objeto JSON estructurado desde el servidor a medida que se genera. Defines un esquema Zod; el SDK analiza el JSON parcial y desencadena re-renderizados a medida que llegan los campos. Este es el camino más fluido hacia el Patrón 2 en una aplicación Next.js.

```tsx
const { object: dashboard } = useObject({
  api: "/api/generate-dashboard",
  schema: z.object({
    title: z.string(),
    metrics: z.array(z.object({ label: z.string(), value: z.number() })),
    insights: z.array(z.string()),
  }),
});
```

**`useChat` con manejadores de herramientas** → Patrón 1. El modelo llama a herramientas; tú mapeas nombres de herramientas a componentes.

**AI Elements** ([elements.ai-sdk.dev](https://elements.ai-sdk.dev/)) proporciona primitivas de IU listas para usar con el SDK.

**Una nota sobre la confusa trayectoria aquí**: En octubre de 2024, Vercel anunció en la [Discusión de GitHub #3251](https://github.com/vercel/ai/discussions/3251) que AI SDK RSC — el patrón de transmisión de React Server Components promocionado como la característica estrella de "IU Generativa" en el SDK 3.0 — se pausaba indefinidamente debido a "varias limitaciones persistentes" sin buenas soluciones a corto plazo. Los equipos que habían construido estrategias de producto basadas en la transmisión RSC se vieron sorprendidos. Las APIs `generateObject`/`streamObject` también fueron deprecadas posteriormente en el SDK 6.0. La migración recomendada desde AI SDK RSC es el patrón `useObject` mencionado anteriormente, o json-render para la generación basada en catálogos.

### assistant-ui

**GitHub**: [assistant-ui/assistant-ui](https://github.com/assistant-ui/assistant-ui)

assistant-ui es un conjunto de primitivas React componibles para construir interfaces de chat de calidad de producción. Es la respuesta correcta cuando necesitas una experiencia de usuario de chat pulida — burbujas de mensaje, transmisión de tokens, acciones de copiar/editar/regenerar, estados de pensamiento — y quieres traer tu propio backend y tu propio renderizado de herramientas.

Funciona bien emparejado con cualquier backend (OpenAI, Anthropic, modelos locales, endpoints personalizados) y gestiona el renderizado de llamadas a herramientas mediante un modelo familiar de slot/render prop.

### json-render

**GitHub**: [vercel-labs/json-render](https://github.com/vercel-labs/json-render) · Docs: [json-render.dev](https://json-render.dev/)

json-render operacionaliza el Patrón 2 con un enfoque opinado y listo para usar. Obtienes un catálogo de componentes pre-construido (componentes de shadcn/ui con esquemas Zod), un renderizador y un bucle de generación ajustado donde el modelo está limitado al catálogo por el esquema.

Las características distintivas:
- **Renderizado multiobjetivo**: el mismo esquema JSON puede renderizarse en una aplicación web React, una aplicación móvil React Native, un PDF, un correo electrónico HTML o un vídeo de Remotion. Esto es genuinamente útil para informes.
- **Renderizado progresivo**: los componentes aparecen a medida que el modelo los transmite, no después de que llegue la especificación completa.
- **Limitaciones estrictas de esquema**: el catálogo está diseñado para que el modelo no pueda alucinar tipos de componentes válidos pero desconocidos.

Si estás construyendo una funcionalidad de generación de dashboards o informes y quieres saltarte el trabajo de infraestructura de diseñar tu propio catálogo, json-render es el camino más rápido para aplicaciones web.

**Sobre el impulso**: json-render se lanzó desde Vercel Labs a principios de 2026 y parece haber captado rápidamente la atención de los desarrolladores web porque es inmediatamente útil en proyectos estándar de React/Next.js. Dicho esto, json-render todavía está en fase pre-1.0 y la relación entre json-render y A2UI todavía se está definiendo — Vercel ha experimentado con salidas compatibles con A2UI, por lo que la convergencia es posible. Para proyectos multiplataforma (móvil nativo, múltiples frameworks), A2UI es la mejor apuesta a largo plazo.

### Hashbrown

**GitHub**: [liveloveapp/hashbrown](https://github.com/liveloveapp/hashbrown)

Hashbrown adopta un enfoque distintivo: en lugar de construir una capa de interfaz de IA separada, incrusta la selección de componentes de IA directamente en tu aplicación React o Angular existente. Expones los componentes de tu aplicación al LLM; el LLM selecciona cuáles renderizar y puede invocar herramientas del lado del cliente.

Esta es la herramienta adecuada cuando quieres infundir inteligencia en superficies de producto que no son "chat" — una página de producto que adapta su diseño, un panel de configuración que muestra las opciones correctas, un editor de flujo de trabajo que sugiere el siguiente paso.

### OpenUI

**GitHub**: [thesysdev/openui](https://github.com/thesysdev/openui) · Docs: [openui.com](https://www.openui.com/)

OpenUI reemplaza JSON con un formato orientado a líneas tipo código ("OpenUI Lang") diseñado para el renderizado progresivo y la eficiencia de tokens. La promesa es aproximadamente un 67% menos de tokens que el JSON equivalente para diseños complejos.

La contrapartida es la madurez del ecosistema — OpenUI es más nuevo y las herramientas son más escasas que en los enfoques basados en JSON. Pero si el coste de los tokens es una restricción significativa y estás generando diseños complejos a alta frecuencia, la eficiencia del formato es real.

### Tambo

**GitHub**: [tambo-ai/tambo](https://github.com/tambo-ai/tambo)

Tambo se centra en la selección de componentes con estado: la IA selecciona componentes y puede interactuar con ellos a través de herramientas del lado del cliente, manteniendo el estado del componente a lo largo de la conversación. Ideal para casos de uso donde los elementos de la interfaz persisten entre turnos — un componente de filtro que el usuario ajusta mientras la IA continúa razonando sobre los datos filtrados.

## La Capa Python

El ecosistema Python aborda las interfaces de IA de manera diferente. Estas herramientas están optimizadas para demostraciones de modelos de ML, aplicaciones de datos y herramientas internas — no para aplicaciones de consumo de producción con composición de diseño impulsada por agentes.

Eso no es una crítica. Para los casos de uso adecuados, Gradio y Streamlit son las únicas herramientas que necesitas.

### Gradio

**GitHub**: [gradio-app/gradio](https://github.com/gradio-app/gradio) · PyPI: `gradio`

El valor central de Gradio: escribes una función en Python; Gradio la envuelve en una interfaz web. La clase `Interface` son 3 líneas para un clasificador de imágenes. `ChatInterface` son 10 líneas para un chatbot. `Blocks` te da control de diseño detallado cuando lo necesitas.

La "IU generativa" en Gradio la define el desarrollador de Python, no el modelo. La visibilidad y configuración de los componentes pueden cambiar dinámicamente basándose en las salidas del modelo, pero el catálogo de componentes es estático — no le pides al modelo que componga diseños.

Gradio es el estándar para HuggingFace Spaces y el ecosistema de demos de ML. Tiene millones de descargas mensuales y potencia una gran fracción del panorama de demostraciones de IA.

**Recurre a Gradio cuando**: seas un desarrollador de Python construyendo una demo de un modelo de ML, un prototipo de investigación o una herramienta interna, y no quieras tocar JavaScript.

### Streamlit

**GitHub**: [streamlit/streamlit](https://github.com/streamlit/streamlit)

El modelo de Streamlit es más opinado: un script de Python se ejecuta de principio a fin en cada interacción. Llamas a `st.chat_message()`, `st.dataframe()`, `st.plotly_chart()`. El framework se encarga del diseño.

El modelo de re-ejecución completa del script parece ineficiente pero es sorprendentemente ergonómico para chatbots de IA que acumulan historial de conversación — todo el script se vuelve a ejecutar, el historial del chat está en el estado de la sesión y la salida es determinista. Streamlit ahora tiene soporte nativo para la mayoría de los principales proveedores de LLM e se integra de forma nativa con Snowflake Cortex.

**Recurre a Streamlit cuando**: estés construyendo una aplicación de datos impulsada por IA, una herramienta de informes interna o un dashboard respaldado por ML en Python y quieras el camino de despliegue más simple posible.

### LangChain y Haystack

Estos son frameworks de orquestación de backend, no frameworks de interfaz de usuario. Aparecen en cualquier mapa honesto de la pila de IU generativa porque suelen ser la capa donde se generan las salidas estructuradas antes de enviarse al frontend.

**LangChain** ([langchain-ai/langchain](https://github.com/langchain-ai/langchain)): `.with_structured_output()` en cualquier LLM te da generación de JSON limitada por Pydantic. El decorador `@tool` con generación automática de esquemas es la forma más limpia de definir qué herramientas puede llamar el modelo. LangChain envía los resultados estructurados a cualquier capa de frontend que estés usando.

**Haystack** ([deepset-ai/haystack](https://github.com/deepset-ai/haystack)): arquitectura de pipeline modular con fuerte soporte para RAG. Hayhooks envuelve los pipelines de Haystack como endpoints HTTP — incluyendo endpoints compatibles con MCP. Si tu IU generativa necesita una base de recuperación, la arquitectura de pipeline de Haystack gestiona eso limpiamente.

Ninguno de los dos frameworks es propietario de la capa de interfaz. Generan los datos que tu frontend (Patrón 1, 2 o 3) renderiza.

## Velocidad del Ecosistema y Terreno Inestable

Este espacio se mueve rápido y varios proyectos han enviado comunicaciones confusas junto con su código. Verificado por última vez el 8 de mayo de 2026; trata las notas sobre el estado de los proyectos aquí como una lectura con marca de tiempo, no como un veredicto permanente.

**Vercel AI SDK RSC** era la característica estrella de la IU Generativa cuando se lanzó el SDK 3.0. Vercel detuvo su desarrollo en octubre de 2024 ([Discusión #3251](https://github.com/vercel/ai/discussions/3251)) citando limitaciones arquitectónicas con React Server Components que no tenían solución a corto plazo. Los equipos que habían construido sobre él estaban comprensiblemente frustrados. Sigue en los documentos pero no es el camino recomendado; `useObject` lo es.

**json-render** (Vercel Labs) es la nueva dirección — una alternativa basada en catálogos e independiente del framework que evita los problemas de acoplamiento con RSC. Está en fase pre-1.0 y parece haber despertado un fuerte interés temprano entre los desarrolladores web/React. La probable razón de la DX (experiencia del desarrollador): json-render es inmediatamente útil en un proyecto estándar de React/Next.js, mientras que el alcance multiplataforma de A2UI añade fricción en la configuración. Cómo acabará esto a medida que ambas especificaciones maduren es genuinamente incierto. Vercel ha explorado la compatibilidad con A2UI en json-render, lo que sugiere que la convergencia es posible.

**A2UI** (Google) está en fase pre-1.0 (v0.9 la última vez que se comprobó) con cambios que rompen la compatibilidad entre versiones menores y comunicaciones inconsistentes de Google sobre su hoja de ruta. Es la elección correcta para el alcance multiplataforma (web + Flutter + SwiftUI) que json-render no aborda, y tiene un respaldo empresarial significativo. Para proyectos puramente web ahora mismo, la DX es más tosca.

**AG-UI** (CopilotKit) también está en fase pre-1.0. La confusión más común: el nombre hace que parezca un framework de interfaz de usuario. No lo es — es un protocolo de transporte. AG-UI define cómo fluyen los eventos entre los backends de los agentes y tu frontend; lo que renderices en respuesta sigue siendo tu decisión. Ese modelo mental es sólido y ampliamente adoptado, pero la especificación pre-1.0 significa que todavía se están resolviendo casos límite.

La conclusión práctica: **cada actor principal aquí está en fase pre-1.0**. Planifica para cambios en la API. Los patrones — renderizado de herramienta a componente, composición de catálogo, generación aislada — son lo suficientemente estables como para construir sobre ellos. Las opciones específicas de protocolo no lo son.

## Diseño del Catálogo de Componentes: El Verdadero Trabajo de Ingeniería

La mayor parte de la complejidad interesante en el Patrón 2 no está en el renderizador — está en el catálogo.

El catálogo es una **decisión de producto codificada como un esquema**. Responde: ¿cuáles son los objetos de IU significativos en este dominio? No "¿qué componentes de React existen?", sino "¿qué es lo que un usuario en este contexto realmente necesita ver e interactuar?".

**El modo de fallo por exceso de granularidad**: expones `Row`, `Column`, `Text`, `Button`, `Icon`. Ahora el modelo tiene que ser un ingeniero frontend. Generará diseños mediocres que no coincidirán con tu sistema de diseño, olvidará los estados vacíos, producirá un marcado inaccesible y cambiará su enfoque de una respuesta a otra porque nada en el catálogo limita la salida al lenguaje visual de tu producto.

**El modo de fallo por exceso de generalidad**: expones `WeatherCard`, `FlightCard`, `HotelCard`. El modelo no puede adaptarse cuando el usuario pide algo que no se mapea a una tarjeta pre-construida. Recurre al texto.

**El punto medio útil**: componentes de nivel de dominio con espacios (slots) limitados.

Un catálogo de una aplicación de viajes podría verse así:

```
TripSummary         — itinerario de un vistazo
FlightOptionList    — opciones de vuelo seleccionables con precios
HotelComparison     — tarjetas de hoteles comparadas lado a lado
TravelerForm        — recogida de detalles del viajero
PolicyNotice        — aviso de política o reglas de tarifa
BookingConfirmation — confirmación final con botón de acción
```

Un catálogo de una aplicación financiera podría verse así:

```
PortfolioSnapshot   — posiciones clave y pérdidas y ganancias (P&L)
TransactionTable    — transacciones filtrables y paginadas
RiskBreakdown       — métricas de asignación y volatilidad
ScenarioComparison  — modelado de escenarios lado a lado
ApprovalGate        — acción que requiere confirmación humana
```

El catálogo suena al vocabulario de tu producto. Codifica tus decisiones de UX, tus requisitos de accesibilidad, tu gestión de estados vacíos y tus patrones de acciones peligrosas en el código del componente. El modelo se encarga de organizar esas piezas. Tú sigues decidiendo cómo se ve cada pieza y qué se le permite hacer.

**Reglas de diseño de esquemas que reducen las alucinaciones**:

1. Mantén los valores de los enums cortos y obvios. `"type": "bar_chart"` en lugar de `"type": "data-visualization-bar-type-vertical"`.
2. Haz imposible la composición inválida. Si un `PolicyNotice` solo puede aparecer al final de un diseño, no lo pongas en el mismo nivel de esquema que las cosas que pueden aparecer en cualquier lugar.
3. Usa campos obligatorios generosamente. Un campo opcional es un campo que el modelo podría omitir y tu renderizador tiene que gestionar como nulo.
4. Prueba el catálogo con prompts reales antes de enviarlo. Guarda los esquemas generados; compruébalos en busca de violaciones de esquema, valores de campo alucinados y composiciones que son técnicamente válidas pero semánticamente incorrectas.

## Trampas Comunes

**Trampa: tratar el JSON válido como un comportamiento seguro.** La validación del esquema confirma la estructura. No dice nada sobre si la acción adjunta a un botón coincide con su etiqueta, si un total coincide con los datos de los que deriva o si un componente de interfaz está haciendo algo que el usuario no esperaba. Las especificaciones de IU generada necesitan una revisión semántica, no solo una validación de esquema. Como mínimo, las acciones destructivas deberían requerir un componente de confirmación, y las etiquetas de esos componentes deberían probarse contra las acciones que activan.

**Trampa: exponer primitivas de diseño en lugar de primitivas de producto.** Si el modelo tiene que decidir si usar un padding de 16px o 20px, le has dado el nivel de abstracción equivocado. Los componentes de dominio deben codificar el gusto por el producto. El modelo debería estar componiendo comportamiento, no gestionando detalles de presentación.

**Trampa: usar IU generativa donde bastaría con IU estática.** Si la estructura de lo que quieres mostrar es conocible en tiempo de desarrollo — y normalmente lo es — el Patrón 1 con componentes pre-construidos es más rápido, más seguro y más consistente. La IU generativa justifica su complejidad cuando la estructura varía genuinamente según los datos o el contexto de la tarea.

**Trampa: saltarse la accesibilidad.** Los LLMs alucinan violaciones de WCAG. Emitirán `role="region"` en elementos interactivos, generarán formularios sin etiquetas y producirán ratios de contraste que no superan el nivel WCAG AA. Tu librería de componentes puede ser totalmente accesible; las composiciones de esos componentes generadas por IA no son automáticamente accesibles. Prueba toda la ruta de renderizado, no solo los componentes de forma aislada.

**Trampa: confundir el protocolo y el framework.** AG-UI no es un framework frontend. A2UI no es una librería de React. Son formatos de cable y protocolos de eventos. Todavía necesitas un framework frontend para implementarlos. CopilotKit implementa AG-UI y A2UI. json-render implementa el patrón de catálogo A2UI/Open-JSON-UI. Son capas diferentes.

## Recomendaciones por Caso de Uso

**Añadir un copiloto a una aplicación SaaS existente**: Empieza con el Patrón 1 (herramienta a componente). Usa Vercel AI SDK `useChat` o CopilotKit. Mapea tus 5–10 acciones principales del agente a componentes pre-construidos. Envía eso, mídelo y luego expande el catálogo solo si los usuarios demuestran que necesitan una composición más rica.

**Generación de dashboards a partir de lenguaje natural**: Usa el Patrón 2 con json-render o un catálogo A2UI personalizado. Define un catálogo de 8–15 tipos de componentes que cubran tus tipos de gráficos, tarjetas de métricas y variantes de tablas. Entrega el esquema al modelo; deja que componga el diseño. Construye una validación que detecte tipos desconocidos antes de que lleguen al renderizador.

**Frontend multi-agente**: Usa CopilotKit con AG-UI. El flujo de eventos gestiona la transmisión en tiempo real a través de los backends de los agentes; el estado compartido gestiona el traspaso entre agentes; el patrón HITL gestiona las puertas de aprobación.

**Construir dentro de ChatGPT u otro anfitrión MCP**: Usa MCP Apps. Define tu herramienta como una herramienta de datos que recupera y razona, y una herramienta de renderizado separada que solicita un widget. Mantén la lógica de negocio fuera de la plantilla del widget.

**Demos de modelos de ML y aplicaciones de datos (equipo de Python)**: Gradio para demos y HuggingFace Spaces. Streamlit para aplicaciones de datos con interacciones más complejas. Ninguno requiere JavaScript.

**Artefactos visuales, simulaciones, diagramas**: Usa el Patrón 3 (OpenGenerativeUI o equivalente). Establece una CSP de iframe estricta. Trata la salida como contenido de usuario no confiable desde el punto de vista de la seguridad.

---

Los frameworks están madurando rápido. La convergencia de protocolos (AG-UI para la transmisión, A2UI/Open-JSON-UI para las especificaciones del catálogo) todavía está en progreso, pero la forma se está volviendo lo suficientemente clara como para construir sobre ella.

Los desafíos de ingeniería que más importan ahora mismo no son la selección del framework. Son el diseño del catálogo — decidir qué se le permite decir al modelo, lo que requiere claridad de producto más que habilidad técnica. Son la validación semántica — probar que la interfaz generada hace lo que dice, no solo que pasa la validación del esquema. Y son la brecha de accesibilidad — construir catálogos donde cada componente, y cada composición de componentes, cumpla con el listón de accesibilidad que exigirías a una interfaz escrita a mano.

El modelo hará lo que le digas que haga dentro de la gramática que le des. Haz que la gramática sea deliberada.
