"Generative UI" משמעותו לפחות חמש דברים שונים תלוי במי שמגדיר זאת.

- ממשקי צ'אט שמשלבים כרטיסי מוצר משיחות כלי מודל  
- מפרטי JSON בזמן ריצה שהפרונטאנד מציג כעצים של רכיבים  
- iframes מבודדים שמוחזרים על‑ידי כלי MCP לתוך אפליקציות מארחות (מהזמנת כרטיסים, הזמנת מלון ועד רינדור מפה, וידג'טים של תשלום)  
- פרוטוקולי אירועים שמזרים מצב של סוכן אל הפרונטאנד  
- v0, Lovable, ו‑Bolt: כלי AI שכותבים React בזמן תכנון  

אלו מושגים קשורים, אך הם פועלים בשכבות שונות של הסטרק ונושאים פרופילים שונים של סיכון, עלויות יישום שונות, ושימושים מתאימים שונים. ערבוב ביניהם הופך כל דיון ארכיטקטוני למבולגן.

זה המפה שאני רוצה כשאני מחליט באיזו שכבה של הסטרק להגיע.

---

## מה Generative UI **אינו**

לפני שמגדירים מה זה, שלוש נקודות לשים בצד:

**הפקת קוד בזמן תכנון** — v0, Lovable, Bolt, Cursor שמרכיבים רכיבי React. כלים אלה מייצרים קוד שהמפתחים בודקים ומתחייבים אליו. ה‑AI פועל בזמן פיתוח. מה שמופץ הוא סטטי מנקודת מבט של המשתמש. זו קטגוריה מצוינת של כלים. זה לא מה שמובן תחת "generative UI בזמן ריצה".

**השלמת טפסים בעזרת AI** — המודל ממלא ערכי שדות מהקשר. מבנה הממשק נשאר קבוע; רק התוכן זז. זהו תבנית שימושית. זה לא Generative UI.

**כתיבת HTML גולמי על‑ידי AI לתוך דף** — המודל מפיק מחרוזות `<div>` ו‑`<button>` שמוזרות דרך `innerHTML` או `dangerouslySetInnerHTML`. זה *הוא* Generative UI בזמן ריצה במובן הטכני ביותר. זה גם הגרסה המסוכנת ביותר, והסיבה שכל מסגרת成熟 בתחום הזה נבנתה כדי להימנע ממנה. קוד markup גולמי שנוצר על‑ידי AI גורם לסיכון XSS, לתכונות נגישות חסרות, לעיצוב לא עקבי, ולמבנה הולוגינציה. שאר המאמר עוסק בדרכים לעשות זאת בצורה בטוחה יותר.

## הגדרה עובדת

Generative UI בזמן ריצה משמעותו: **המודל קובע איזה רכיב ממשק או קומפוזיציית רכיבים המשתמש רואה, על‑בסס מצב השיחה או המשימה.**

לא המילים. הממשק.

המקרה הפשוט ביותר: העוזר שלך להזמנת טיסות קורא לכלי `search_flights`. במקום להחזיר טקסט פשוט („הנה שלוש אפשרויות…”) הוא מציג רכיב `<FlightResultsCard>` עם טיסות שניתן לבחור, מתגים למחלקת מושב, וכפתור „Book”. המודל החליט שכרטיס מובנה הוא התגובה המתאימה כאן. המפתח קבע איך נראה הכרטיס ומה עושה הכפתור „Book”.

המקרה המורכב יותר: סוכן ניתוח פיננסי מקבל שאלה על תיק השקעות ומחליט להרכיב תגובה עם `MetricGroup` המציג מספרים מרכזיים, גרף `RiskBreakdown`, טבלת `ScenarioComparison`, והודעת `PolicyNotice`. המודל אסף את הפריסה הזאת מקטלוג של רכיבים מאושרים מראש. המפתח הגדיר כל רכיב. המודל בחר אילו להשתמש ובאילו נתונים למלא אותם.

שני המקרים הם Generative UI. ההבדל הוא כמה חופש הרכבה יש למודל, מה שמכתיב הן את העושר של הפלטים האפשריים והן את המורכבות של מה שעלול להשתבש.

## שלושת הדפוסים

המרחב כולו מצטמצם לשלושה דפוסים, שלכל אחד מהם תחביר פלט שונה.

![דיאגרמת ספקטרום המציגה שלושה דפוסים: קריאות לכלי רק בצד שמאל (הבטוח ביותר), קטלוג רכיבים במרכז, ויצירת קוד פתוחה בצד ימין (הביטוי ביותר).](../output-grammar-spectrum.svg)

_כל החלטת UI גנרטיבית היא נקודה על הספקטרום הזה. התחילו משמאל._

### דפוס 1: רינדור כלי‑לרכיב

המודל קורא לכלי בעל שם. ליישום שלכם יש מפת שמות כלים לרכיבים. קריאת הכלי מפעילה רינדור של רכיב.

```tsx
// The model calls: { name: "show_flight_results", args: { flights: [...] } }

useCopilotAction({
  name: "show_flight_results",
  render: ({ args }) => <FlightResultsCard flights={args.flights} />,
});
```

זהו הדפוס הבטוח ביותר מכיוון שהפריסה לעולם לא מגיעה מהמודל. המודל מחליט *מתי* להציג רכיב ו*אילו נתונים* למלא בו. המפתחים שלכם עדיין אחראים על קוד הרכיב, העיצוב הוויזואלי, יישום הנגישות, וכל מקרה קצה בלוגיקת הרינדור.

SDK של Vercel AI `useChat` עם מטפלים `tool` עושה זאת. רינדור הכלים של assistant‑ui עושה זאת. "Static Generative UI" של CopilotKit הוא דפוס זה. רוב ממשקי ה‑copilot בתפוקה שמספקים אמינות עושים זאת.

**מתאים כאשר**: קבוצת הדברים שתרצו להציג ניתנת להכרה בזמן הפיתוח. אישורי הזמנות, תוצאות חיפוש, סיכומי חשבון, וידג׳טים לאישור. אם ניתן למנות את הסצנריואים, דפוס זה מכסה אותם.

### דפוס 2: הרכבת קטלוג רכיבים

המודל מפיק עץ JSON Typed שמפנה לרכיבים מקטלוג שהוגדר על‑ידי המפתח. לחזית שלכם יש רנדרר שמטייל בעץ ומאתחל כל רכיב.

```json
[
  { "type": "metric_group", "metrics": [
    { "label": "MRR", "value": "$82,400", "delta": "+12%" },
    { "label": "Churn", "value": "2.1%", "delta": "-0.4%" }
  ]},
  { "type": "line_chart", "title": "30-day growth", "data_ref": "mrr_series" },
  { "type": "insight_callout", "text": "Expansion revenue driving the delta — avg seat count up 18%." }
]
```

המודל הרכב את הפריסה: `MetricGroup`, `LineChart`, `InsightCallout`. אבל אתם מגדירים מה משמעות כל סוג רכיב, אילו פרופס הוא מקבל, וכיצד הוא מוצג. אם המודל מנסה לפלוט `{ "type": "custom_untested_thing" }`, אימות הסכמה שלכם תופס זאת והרנדרר מתעלם או דוחה אותו.

זהו הדפוס שמאחורי `json-render`, `A2UI`, `Hashbrown`, `OpenUI` ו‑`Tambo`. העבודה ההנדסית המרכזית היא **תכנון הקטלוג** — החלטה אילו סוגי רכיבים קיימים, איך נראות הסכמות שלהם, ומה המודל מותר ולא מותר להרכיב.

**מתאים כאשר**: המבנה של מה שאתם רוצים להציג משתנה באופן לגיטימי בהתאם לנתונים או לבקשת המשתמש. לוחות מחוונים שמסתגלים למה שבולט במספרים. דוחות שמציגים קטעים שונים בהתאם להקשר. פאנלים של זרימת עבודה שמשתנים לפי השלב שבו נמצא הסוכן.

### דפוס 3: יצירה פתוחה‑קצה

המודל כותב HTML, SVG, Canvas או WebGL שמוצגים בתוך iframe מבודד עם מדיניות אבטחה קפדנית (Content Security Policy).

זה מתאים לדברים שבהם קטלוג רכיבים קבוע לא יספיק: ויזואליזציות של אלגוריתמים, דיאגרמות ארכיטקטוניות, גרפים אד‑הוק, אומנות גנרטיבית, סימולציות חינוכיות. כאן גבול ה‑iframe הוא שמבצע את העבודה האבטחתית; אם תסירו אותו אתם חוזרים לבעיה של הזרקת HTML גולמי מהחלק העליון של מאמר זה.

`CopilotKit/OpenGenerativeUI` הוא יישום ההפניה הטוב ביותר כיום לדפוס זה. ה‑sandbox מסנן סקריפטים, מגביל העברת הודעות, ומשאיר את הארטיפקט שנוצר רחוק ממצב הפריבילגיות של האפליקציה שלכם.

**מתאים כאשר**: אתם באמת זקוקים לפלט חזותי שרירותי — דיאגרמות הסבריות חד‑פעמיות, סימולציות דינמיות, יצירות יצירתיות. אל תשתמשו בזה לממשק משתמש טרנזקציוני. אישור רכישה אינו דורש iframe מבודד.

### מעבר לשלושה הדפוסים: LLMs שמנהלים פיקסלים ישירות

מתגמת כיוון רביעי מתהווה שאינו מתאים באופן נקי לאף אחד מהדפוסים הקיימים: LLMs שמניעים **חוויות משקיעות, דמויות‑משחק** על‑ידי שליטה בפלט החזותי בצורה ישירה יותר מאשר iframe מבודד.

ההבחנה הקנונית בתוך UI גנרטיבי היא **iframe HTML vs. JSON catalog**:

- **Iframe HTML** — המודל כותב HTML, SVG, Canvas או WebGL שמוצג בחוליית sandbox מבודדת. חופש ביטוי מרבי; האבטחה תלויה לחלוטין בגבול ה‑iframe. דוגמאות: Anthropic Artifacts, OpenGenerativeUI.  
- **JSON catalog** — המודל מפיק מטען מובנה המוגבל לקטלוג רכיבים שהמפתח מגדיר; המנגנון שלך יוצר מופעים של רכיבים מהימנים וקודמים מההגדרה. המודל מחליט *מה* להציג; אתה מחליט *איך* זה מוצג. דוגמאות: json-render, A2UI.  

מעבר לכך, הדגמות חדשות מרמזות על מצב שלישי שבו המודל אינו בוחר רכיבים ולא כותב HTML מבודד — הוא מניע את הקנבס בצורה ישירה יותר. פרויקטים כמו [Tencent's HunyuanWorld](https://arxiv.org/abs/2502.01999), שמייצרים סביבות 3D ניתנות לחקירה מתמונה אחת, וארכיטקטורות משחק שבהן LLMs מייצרים מפות, דמויות NPC ומשימות בזמן ריצה במקום לקרוא לקטלוג רכיבים, מרמזים על עתיד שבו המודל דומה יותר למנהל משחק מאשר למעבד טפסים. רינדור LLM בדפדפן דרך WebGPU ([WebLLM](https://mlc.ai/web-llm/)) דוחף את הגבול הזה מקומית.  

התחום הזה מרגש באמת והוא עדיין בשל שלב מוקדם. אין עדיין מסגרות יציבות לבניית מוצרי ייצור. את הגישה הזו אסביר במאמר ייעודי ברגע שהדבר ישתנה.

## הפרוטוקולים: AG-UI ו‑A2UI  

AG-UI ו‑A2UI הם שני הסטנדרטים המרכזיים בשכבת הפרוטוקול. הם פותרים בעיות שונות ואינם מתחרים זה עם זה.

### AG-UI

**GitHub**: [ag-ui-protocol/ag-ui](https://github.com/ag-ui-protocol/ag-ui)

AG-UI הוא פרוטוקול מבוסס אירועים לתקשורת בין סוכני AI ליישומי קצה. הוא מגדיר כ‑16 סוגי אירועים: `TEXT_MESSAGE_START`, `TEXT_MESSAGE_CONTENT`, `TOOL_CALL_START`, `TOOL_CALL_END`, `STATE_SNAPSHOT`, `STATE_DELTA`, וכן הלאה. תשתית ההעברה תלויה בכם — SSE, WebSockets, webhooks כולם עובדים. הפורמט נשאר רפוי במכוון כדי לאפשר אימוץ רחב.

AG-UI אינו מגדיר איך הממשק שלכם יראה. הוא מגדיר איך הסוכן מתקשר *עם* הקצה שלכם. חשבו על זה כשכבת פרוטוקול החוט שמאפשרת לאפליקציית React שלכם להירשם לסוכן LangGraph באותה צורה שהיא נרשמת לסוכן CrewAI, בלי לשנות קוד קצה.

CopilotKit יצר את AG-UI מתוך עבודתם עם LangGraph ו‑CrewAI. הוא אומץ על‑ידי LangChain, Mastra, PydanticAI ועוד. מיקרוסופט פרסמה מדריך אינטגרציה ל‑AG-UI. אם אתם בונים קצה מרובה‑סוכנים וצריכים להפריד בין מסגרות backend לקוד קצה, AG-UI הוא הפתרון.

**הבהרה אחת שמבלבלת רבים**: AG-UI אינו מסגרת UI. הוא לא אומר לכם מה לרנדר. הוא מודיע לכם *שהסוכן אמר משהו*, קרא לכלי, או עדכן מצב משותף. מה שתציגו בתגובה הוא עדיין החלטתכם.

### A2UI

**GitHub**: [google/A2UI](https://github.com/google/A2UI) · Spec: [a2ui.org](https://a2ui.org/)

A2UI היא המפרט הדקלרטיבי של גוגל למה שסוכנים שולחים כשברצונם להציג UI. כאשר AG-UI עונה על השאלה "איך הסוכן מתקשר?", A2UI עונה על השאלה "באיזה פורמט הסוכן משתמש לתיאור פריסת רכיבים?".

A2UI משתמשת בפורמט JSONL שטוח: תיאור רכיב אחד לשורה, עם מזהה, סוג ונתונים. השטחה היא בכוונה. עצים מקוננים דורשים מהמודל לדעת את המבנה המלא לפני שהוא יכול להתחיל להזרים. רשימה שטוחה מאפשרת למודל לשחרר כל רכיב כש"מתחיל לחשוב" עליו, מה שמאפשר לקצה שלכם להתחיל לרנדר את כרטיס המדד הראשון בעוד המודל עדיין מחליט אם להוסיף גרף.

```jsonl
{"id":"h1","type":"kpi_card","title":"MRR","value":"$82,400","delta":"+12%"}
{"id":"h2","type":"kpi_card","title":"Churn","value":"2.1%","delta":"-0.4%"}
{"id":"c1","type":"line_chart","title":"30-day MRR","data_ref":"mrr_series"}
{"id":"t1","type":"data_table","cols":["Month","MRR","Net New"],"data_ref":"monthly"}
```

A2UI מודעת לאבטחה: המפרט הוא פורמט נתונים, ולא קוד שניתן לבצע. קטלוג הרכיבים מוגדר מראש על‑ידי המפתחים; הסוכן יכול להתייחס רק לסוגים הקיימים בקטלוג. רנדרר של A2UI שמקבל שם סוג בלתי מוכר פשוט מתעלם ממנו.

הפורמט “Open-JSON-UI” של CopilotKit תואם ל‑A2UI. אם אתם בוחרים פורמט מפרט עבור קטלוג רכיבים היום, A2UI הוא זה עם התמיכה הרחבה ביותר חוצה פלטפורמות.

**הערה על יציבות**: A2UI עדיין לפני גרסה 1.0 — v0.9 במועד הבדיקה האחרון ב‑8 מאי 2026 — והייתה שינויי מפרט שבורים בין גרסאות מינוריות. העדכונים של גוגל לגבי מפת הדרכים היו מפוזרים וחלק מהרנדררים (Lit, Flutter) התעדכנו באיחור. יש לכלול זמן לתקופת סטייה של המפרט אם אתם בונים עליו היום. למקרים של שימוש טהור ברשת, json-render נראה כיום בעל כלי פיתוח שלמים יותר. היתרון ארוך‑טווח של A2UI הוא הגעה חוצה פלטפורמות (web, Flutter, SwiftUI, Android) ש‑json-render אינו מציע.

### אפליקציות MCP

**GitHub**: [modelcontextprotocol](https://github.com/modelcontextprotocol) · קשור: [mcp-ui](https://github.com/MCP-UI-Org/mcp-ui)

MCP החל כפרוטוקול לחיבור מודלים גדולים (LLM) לכלים ולנתונים. הרחבת ה‑Apps מאפשרת לכלי MCP להחזיר לא רק נתונים, אלא גם חפצים אינטראקטיביים של UI: רכיבי React, טפסים, לוחות מחוונים, מפות.

מודל האבטחה הוא קפדני וזה הכוונה: כל הרינדור מתבצע ב‑iframe מבודד עם הרשאות מצומצמות, תבניות מוגדרות מראש כך שהאפליקציה המארחת יכולה לסקור אותן, וכל התקשורת היא JSON‑RPC שניתן לעקוב אחריו. זהו המודל הנכון עבור ספקי כלים — שרת Shopify MCP יכול להחזיר וידג׳ט קופה; שירות מיפוי יכול להחזיר מפה שניתנת להטמעה. האפליקציה המארחת איננה בעלת הקוד של הווידג׳ט ולא סומכת עליו.

MCP Apps הוא הבחירה הנכונה כאשר ה‑UI *שייך לספק הכלי*, ולא לאפליקציה שלכם. עבור UI שממוקם בתחום האפליקציה שלכם, השתמשו בתבנית 1 או 2.

## מסגרות JavaScript/TypeScript


### CopilotKit

**GitHub**: [CopilotKit/CopilotKit](https://github.com/CopilotKit/CopilotKit) · דוגמאות: [CopilotKit/generative-ui](https://github.com/CopilotKit/generative-ui)

CopilotKit הוא המסגרת המלאה ביותר ליישומי חזית‑מול‑סוכן. היא מנהלת את כל מחזור החיים: חיבור לגיבוי של סוכנים דרך AG‑UI, ניהול מצב שיחה דו‑כיווני, רינדור רכיבי UI גנרטיביים, והצעת צינור מצב משותף שמאפשר לסוכנים ולמשתמשים לשנות את אותם הנתונים.

מודל שלושת‑התבניות מתמפה בצורה נקייה ל‑APIs של CopilotKit:
- `useCopilotAction` עם קריאת `render` → תבנית 1
- רינדור A2UI/Open-JSON-UI → תבנית 2
- ארטיפקטים מבודדים `OpenGenerativeUI` → תבנית 3

התכונה החשובה של CopilotKit שמקבלת מעט מדי דיון היא **מצב משותף והשתתפות אנושית**: הסוכן יכול לקרוא ולכתוב למצב היישום, המשתמש יכול לקרוא ולכתוב גם הוא, והשינויים זורמים דו‑כיוונית. זה מה שמקנה ל‑UI בסגנון copilot תחושה של שיתוף פעולה אמיתי ולא רק תיבת צ׳אט מחוברת למוצר.

### Vercel AI SDK

**GitHub**: [vercel/ai](https://github.com/vercel/ai) · תיעוד: [ai-sdk.dev](https://ai-sdk.dev/)

Vercel AI SDK הוא הבסיס ה‑TypeScript ה‑de facto ליישומי AI. עבור UI גנרטיבי באופן ספציפי:

**`useObject`** משדר אובייקט JSON מובנה מהשרת בזמן שהוא נוצר. אתה מגדיר סכמת Zod; ה‑SDK מנתח את ה‑JSON החלקי ומפעיל רינדורים מחדש כאשר שדות מגיעים. זו הדרך החלקה ביותר לתבנית 2 ביישום Next.js.

```tsx
const { object: dashboard } = useObject({
  api: "/api/generate-dashboard",
  schema: z.object({
    title: z.string(),
    metrics: z.array(z.object({ label: z.string(), value: z.number() })),
    insights: z.array(z.string()),
  }),
});
```

**`useChat` עם מטפלי כלים** → תבנית 1. המודל קורא לכלים; אתה ממפה שמות כלים לרכיבים.

**אלמנטים AI** ([elements.ai-sdk.dev](https://elements.ai-sdk.dev/)) מספקים פרימיטיבים UI מוכנים לשילוב עם ה‑SDK.

**הערה על המסלול המבלבל כאן**: באוקטובר 2024, Vercel הודיעה ב‑[GitHub Discussion #3251](https://github.com/vercel/ai/discussions/3251) שה‑AI SDK RSC — תבנית ה‑React Server Components שמוזכרת כמאפיין הכותרת “Generative UI” ב‑SDK 3.0 — הושהתה ללא תאריך סיום בגלל “מספר מגבלות ארוכות‑טווח” שאין להן פתרונות קרובים. צוותים שבנו אסטרטגיות מוצר סביב זרימת RSC נתקלו בהפתעה. ה‑APIs `generateObject`/`streamObject` הוסרו גם הם ב‑SDK 6.0. ההגירה המומלצת מ‑AI SDK RSC היא תבנית `useObject` שהוצגה למעלה, או json-render לייצור מבוסס‑קטלוג.

### assistant-ui

**GitHub**: [assistant-ui/assistant-ui](https://github.com/assistant-ui/assistant-ui)

assistant-ui הוא אוסף של פרימיטיבים React קומפוזיביים לבניית ממשקי צ׳אט ברמת ייצור. זה הפתרון הנכון כשצריך חוויית צ׳אט מלוטשת — בועות הודעה, אסימון זרימה, פעולות העתקה/עריכה/הפקה מחדש, מצבי חשיבה — ועדיין רוצים להשתמש בבקאנד שלכם ובכלי הרינדור שלכם.

הספרייה עובדת היטב עם כל בקאנד (OpenAI, Anthropic, מודלים מקומיים, נקודות קצה מותאמות) ומטפלת ברינדור קריאות כלי דרך מודל slot/render prop מוכר.

### json-render

**GitHub**: [vercel-labs/json-render](https://github.com/vercel-labs/json-render) · Docs: [json-render.dev](https://json-render.dev/)

json-render מממשת את תבנית 2 בגישה opinionated, batteries‑included. אתם מקבלים קטלוג רכיבים מוכן (רכיבי shadcn/ui עם סכמות Zod), מנגנון רינדור, ולולאת יצור צמודה שבה המודל מוגבל לקטלוג באמצעות הסכמה.

המאפיינים המבדילים:
- **רינדור מרובה‑יעדים**: אותה ספציפיקציית JSON יכולה לרנדר אפליקציית React web, אפליקציית React Native למובייל, PDF, אימייל HTML, או וידאו Remotion. זה באמת שימושי לדוחות.
- **רינדור פרוגרסיבי**: רכיבים מופיעים בזמן שהמודל משדר אותם, ולא רק אחרי שה‑spec המלא מגיע.
- **הגבלות סכמתיות צמודות**: הקטלוג נבנה כך שהמודל לא יוכל להמציא סוגי רכיבים תקינים‑אבל‑לא‑ידועים.

אם אתה בונה תכונה של לוח מחוונים או יצירת דוחות ורוצה לדלג על העבודה התשתיתית של תכנון הקטלוג שלך, json-render הוא המסלול המהיר ביותר לאפליקציות ווב.

**על המומנטום**: json-render הושק על‑ידי Vercel Labs בתחילת 2026 ונראה כי קיבל תשומת לב מהמתכנתים במהירות מכיוון שהוא מיידי לשימוש בפרויקטים סטנדרטיים של React/Next.js. עם זאת, json-render עדיין לפני גרסה 1.0 והקשר בין json-render ל‑A2UI עדיין בתהליך הגדרה — Vercel ניסתה פלט תואם ל‑A2UI, ולכן קונברג׳נס אפשרי. עבור פלטפורמות מרובות (מובייל נייטיב, מסגרות שונות), A2UI הוא ההימור הטוב יותר בטווח הארוך.

### Hashbrown

**GitHub**: [liveloveapp/hashbrown](https://github.com/liveloveapp/hashbrown)

Hashbrown נוקט בגישה ייחודית: במקום לבנות שכבת ממשק AI נפרדת, הוא משלב בחירת רכיבי AI ישירות לתוך אפליקציית React או Angular הקיימת שלך. אתה חושף את רכיבי האפליקציה למודל השפה; המודל בוחר אילו רכיבים לרנדר ויכול להפעיל כלים בצד הלקוח.

זהו הכלי המתאים כאשר אתה רוצה לשלב אינטליגנציה בממשקי מוצר שאינם “צ׳אט” — לדוגמה, דף מוצר שמסתגל את הפריסה שלו, פאנל הגדרות שמציג את האפשרויות הרלוונטיות, או עורך זרימת עבודה שמציע את הצעד הבא.

### OpenUI

**GitHub**: [thesysdev/openui](https://github.com/thesysdev/openui) · Docs: [openui.com](https://www.openui.com/)

OpenUI מחליף את JSON בפורמט קו‑מכוון דומה לקוד (“OpenUI Lang”) שמיועד לרינדור פרוגרסיבי ויעילות בטוקנים. הטענה היא חיסכון של כ‑67 % בטוקנים לעומת JSON שווה ערך עבור פריסות מורכבות.

החיסרון הוא בגרסת המערכת — OpenUI חדש יותר והכלים סביבו פחות מפותחים מאשר גישות מבוססות JSON. אך אם עלות הטוקנים היא מגבלה משמעותית ואתה מייצר פריסות מורכבות בתדירות גבוהה, היעילות של הפורמט היא ממשית.

### Tambo

**GitHub**: [tambo-ai/tambo](https://github.com/tambo-ai/tambo)

Tambo מתמקדת בבחירת רכיבים עם מצב: ה‑AI בוחר רכיבים ויכול לתקשר איתם דרך כלים בצד הלקוח, תוך שמירת מצב הרכיב לאורך השיחה. מתאים למקרים שבהם אלמנטים בממשק ממשיכים להתקיים בין סבבים — רכיב סינון שהמשתמש מכוונן בעוד ה‑AI ממשיך לנתח את הנתונים המסוננים.

## שכבת הפייתון

האקוסיסטם של פייתון ניגש לממשקי AI בצורה שונה. כלים אלה מותאמים לדמואים של מודלי למידת מכונה, יישומי נתונים, וכלי פנימיים — ולא לאפליקציות צרכניות במוצר סופי עם הרכבת פריסות מונעת סוכן.

זה לא ביקורת. למקרים המתאימים, Gradio ו‑Streamlit הם כלים היחידים שאתה צריך.

### Gradio

**GitHub**: [gradio-app/gradio](https://github.com/gradio-app/gradio) · PyPI: `gradio`

הערך המרכזי של Gradio: אתה כותב פונקציית פייתון; Gradio עוטפת אותה בממשק אינטרנטי. מחלקת `Interface` היא 3 שורות למסווג תמונות. `ChatInterface` היא 10 שורות לצ'אטבוט. `Blocks` נותן שליטה מדויקת על הפריסה כשזה נדרש.

ה‑"UI גנרטיבי" ב‑Gradio מוגדר על‑ידי המפתח בפייתון, ולא על‑ידי המודל. נראות הרכיבים והקונפיגורציה שלהם יכולים להשתנות דינמית בהתבסס על פלטי המודל, אך קטלוג הרכיבים הוא סטטי — אינך מבקש מהמודל להרכיב פריסות.

Gradio הוא ברירת המחדל עבור HuggingFace Spaces והאקו‑סיסטם של הדגמות ML. יש לו מיליוני הורדות חודשיות והוא מניע חלק ניכר מנוף ההדגמות של AI.

**מתי לבחור ב‑Gradio**: אתה מפתח פייתון הבונה הדגמת מודל ML, אבטיפוס מחקרי, או כלי פנימי, ולא רוצה להתעסק ב‑JavaScript.

### Streamlit

**GitHub**: [streamlit/streamlit](https://github.com/streamlit/streamlit)

המודל של Streamlit יותר דעה‑מובנת: סקריפט פייתון רץ מקצה לקצה בכל אינטראקציה. אתה קורא ל‑`st.chat_message()`, `st.dataframe()`, `st.plotly_chart()`. המסגרת מטפלת בפריסה.

המודל של ריצה מלאה של הסקריפט נשמע לא יעיל, אך הוא נוח באופן מפתיע עבור צ'אטבוטים מבוססי AI שמצברים היסטוריית שיחה — כל הסקריפט מתבצע מחדש, היסטוריית הצ'אט נשמרת במצב הסשן, והתוצאה דטרמיניסטית. ל‑Streamlit יש כעת תמיכה רשמית ברוב ספקי ה‑LLM המרכזיים ומשתלב באופן טבעי עם Snowflake Cortex.

**מתי לבחור ב‑Streamlit**: אתה בונה אפליקציית נתונים מונעת AI, כלי דיווח פנימי, או לוח מחוונים מגובה ML בפייתון ורוצה את נתיב הפריסה הפשוט ביותר.

### LangChain ו‑Haystack

אלו מסגרות תזמור צד‑שרת, ולא מסגרות UI. הן מופיעות במפת ערימת UI גנרטיבי כנה כי הן בדרך‑כלל השכבה שבה נוצרים פלטים מובנים לפני שנשלחים ל‑frontend.

**LangChain** ([langchain-ai/langchain](https://github.com/langchain-ai/langchain)): `.with_structured_output()` על כל LLM מספק יצירת JSON עם מגבלות Pydantic. הדקורטור `@tool` עם יצירת סכימה אוטומטית הוא הדרך הנקייה ביותר להגדיר אילו כלים המודל יכול לקרוא. LangChain מזרים תוצאות מובנות לשכבת ה‑frontend שבה אתה משתמש.

**Haystack** ([deepset-ai/haystack](https://github.com/deepset-ai/haystack)): ארכיטקטורת צינור מודולרי עם תמיכה חזקה ב‑RAG. Hayhooks עוטף צינורות Haystack כנקודות קצה HTTP — כולל נקודות קצה תואמות MCP. אם ה‑UI הגנרטיבי שלך דורש תשתית אחזור, ארכיטקטורת הצינורות של Haystack מטפלת בזה בצורה נקייה.

אף אחד מהמסגרות האלו אינו שולט בשכבת ה‑UI. הן מייצרות את הנתונים שה‑frontend שלך (תבנית 1, 2 או 3) מציג.

## מהירות האקוסיסטם וקרקע לא יציבה

התחום הזה מתעדכן במהירות ויש כמה פרויקטים שהפיצו הודעות מבלבלות לצד הקוד שלהם. האימות האחרון היה ב‑8 במאי 2026; התייחס להערות מצב הפרויקט כאן כקריאה מתוזמנת, לא כהחלטה סופית.

**Vercel AI SDK RSC** היה תכונת הדגל של Generative UI כאשר ה‑SDK 3.0 יצא. Vercel עצרה את הפיתוח באוקטובר 2024 ([Discussion #3251](https://github.com/vercel/ai/discussions/3251)) בטענה למגבלות ארכיטקטוניות עם React Server Components שאין להן פתרון בטווח הקצר. צוותים שבנו עליו היו מובנים מתוסכלים. הוא עדיין בתיעוד אך אינו הנתיב המומלץ; `useObject` הוא.

**json-render** (Vercel Labs) הוא הכיוון החדש — חלופה מבוססת‑קטלוג, בלתי תלויה במסגרת, שמתחמקת מבעיות הקשירה של RSC. היא עדיין לפני גרסה 1.0 ונראה שיש עניין מוקדם חזק בקרב מפתחי React/​web. הסיבה המשוערת לחוויית המפתחים (DX): json-render שימושית מיידית בפרויקט React/Next.js סטנדרטי, בעוד שהיקף הפלטפורמות של A2UI מוסיף חיכוך בהגדרה. איך זה יתפתח ככל שהמפרטים יתבגרו עדיין לא ברור. Vercel בחנה תאימות של A2UI ב‑json-render, מה שמרמז על אפשרות להתכנסות.

**A2UI** (Google) היא לפני גרסה 1.0 (v0.9 במועד הבדיקה האחרון) עם שינויים שבורים בין גרסאות מינוריות ותקשורת בלתי עקבית של Google לגבי מפת הדרכים שלה. היא הבחירה הנכונה להגעה חוצת‑פלטפורמות (web + Flutter + SwiftUI) ש‑json-render לא מכסה, ויש לה תמיכה משמעותית ברמת הארגון. לפרויקטים טהורים של web כרגע, חוויית המפתחים (DX) יותר קשה.

**AG-UI** (CopilotKit) גם היא לפני גרסה 1.0. הבלבול הנפוץ ביותר: השם מרמז על מסגרת UI. זה לא כך — זה פרוטוקול תחבורה. AG-UI מגדיר איך אירועים זורמים בין backend של הסוכן ובין ה‑frontend שלך; מה שאתה מציג בתגובה הוא עדיין החלטתך. מודל מחשבה זה יציב ומאומץ ברבים, אך המפרט לפני גרסה 1.0 אומר שמקרים קיצוניים עדיין נפתרים.

המסקנה המעשית: **כל השחקנים המרכזיים כאן הם לפני גרסה 1.0**. תכננו שינויי API. הדפוסים — כלי‑ל‑רכיב, הרכבת קטלוג, יצירה מבודדת — מספיקים לבנייה. בחירות הפרוטוקול הספציפיות אינן.

## תכנון קטלוג רכיבים: העבודה ההנדסית האמיתית

הרוב המורכב של דפוס 2 אינו ברנדר — הוא בקטלוג.

הקטלוג הוא **החלטת מוצר מקודדת כסכמה**. הוא עונה: מהם האובייקטים UI המשמעותיים בתחום הזה? לא "אילו רכיבי React קיימים?" אלא "מה המשתמש בהקשר הזה באמת צריך לראות ולפעול איתו?"

**מצב כשל יתר‑גסול**: אתה חושף `Row`, `Column`, `Text`, `Button`, `Icon`. עכשיו המודל צריך להיות מהנדס Front‑end. הוא ייצור פריסות ממוצעות שלא תואמות למערכת העיצוב שלך, יפספס מצבי ריק, יפיק markup שאינו נגיש, וישנה את הגישה שלו מתגובה לתגובה מכיוון שאין שום דבר בקטלוג שמגביל את הפלט לשפה הוויזואלית של המוצר שלך.

**מצב כשל יתר‑גסול**: אתה חושף `WeatherCard`, `FlightCard`, `HotelCard`. המודל לא יכול להתאים כשמשתמש מבקש משהו שלא מתורגם לכרטיס מוכן מראש. הוא חוזר לטקסט.

**האמצע השימושי**: רכיבי רמה תחומית עם משבצות מוגבלות.

קטלוג של אפליקציית נסיעות עשוי להיראות כך:

```
TripSummary         — מסלול בטווח מבט
FlightOptionList    — אפשרויות טיסה ניתנות לבחירה עם תמחור
HotelComparison     — כרטיסי מלון זה לצד זה
TravelerForm        — איסוף פרטי הנוסע
PolicyNotice        — קריאה לתקנות/כללי תעריף
BookingConfirmation — אישור סופי עם כפתור פעולה
```

קטלוג של אפליקציית פיננסים עשוי להיראות כך:

```
PortfolioSnapshot   — מיקומים מרכזיים ו‑P&L
TransactionTable    — עסקאות ניתנות לסינון, עם דפדוף
RiskBreakdown       — מדדי הקצאה ותנודתיות
ScenarioComparison  — מודלינג של תרחישים זה לצד זה
ApprovalGate        — פעולה הדורשת אישור אנושי
```

הקטלוג נשמע כמו המילון של המוצר שלך. הוא מקודד את החלטות ה‑UX, דרישות הנגישות, טיפול במצבי ריק, ותבניות הפעולות המסוכנות בקוד הרכיב. המודל מקבל לסדר את החלקים האלה. אתה עדיין מחליט איך כל חלק נראה ומה מותר לו לעשות.

**כללי תכנון סכימה שמפחיתים הזיות**:

1. שמור ערכי enum קצרים וברורים. `"type": "bar_chart"` ולא `"type": "data-visualization-bar-type-vertical"`.
2. הפוך הרכבה לא חוקית לבלתי אפשרית. אם `PolicyNotice` יכול להופיע רק בסוף הפריסה, אל תשים אותו באותו רמת סכימה עם דברים שיכולים להופיע בכל מקום.
3. השתמש בשדות חובה בנדיבות. שדה אופציונלי הוא שדה שהמודל עשוי להשמיט וה‑renderer שלך צריך להתמודד איתו כ‑null.
4. בדוק את הקטלוג מול פקודות אמת לפני השחרור. שמור את המפרטים שנוצרו; בדוק אותם עבור הפרות סכימה, ערכי שדה שהומצאו, והרכבות שהן תקניות טכנית אך שגויות סמנטית.

## מלכודות נפוצות

**מלכודה: טיפול ב‑JSON תקין כהתנהגות בטוחה.** אימות סכימה מאשר רק את המבנה. הוא לא אומר דבר לגבי האם הפעולה שמקושרת לכפתור תואמת את התווית שלו, האם סך הכול תואם לנתונים שממנו נגזר, או האם רכיב UI עושה משהו שהמשתמש לא ציפה לו. מפרטים של UI שנוצרו דורשים ביקורת סמנטית, לא רק אימות סכימה. לפחות, פעולות הרסניות צריכות לדרוש רכיב אישור, והתוויות על רכיבים אלה צריכות להיבדק מול הפעולות שהן מפעילות.

**מלכודה: חשיפת פרימיטיבים של עיצוב במקום פרימיטיבים של מוצר.** אם המודל צריך להחליט האם להשתמש בריפוד של 16 px או 20 px, אתה נותן לו רמת הפשטה שגויה. רכיבי תחום צריכים לקודד את טעם המוצר. המודל צריך להרכיב התנהגות, לא לנהל פרטי הצגה.

**מלכודה: שימוש ב‑UI גנרטיבי במקום UI סטטי.** אם המבנה של מה שאתה רוצה להציג ניתן לדעת בזמן הפיתוח — וזה בדרך כלל המצב — דפוס 1 עם רכיבים מוכנים מראש הוא מהיר יותר, בטוח יותר ועקבי יותר. UI גנרטיבי מרוויח את המורכבות שלו רק כאשר המבנה באמת משתנה בהתאם לנתונים או להקשר המשימה.

**מלכודה: דילוג על נגישות.** מודלים גדולים עלולים להמציא הפרות WCAG. הם יוציאו `role="region"` על אלמנטים אינטראקטיביים, ייצרו טפסים ללא תוויות, וייצרו יחס ניגוד שלא עומד ב‑WCAG AA. ספריית הרכיבים שלך עשויה להיות נגישת במלואה; הרכבות שנוצרות על‑ידי AI אינן נגישות באופן אוטומטי. יש לבדוק את מסלול הרינדור המלא, לא רק את הרכיבים בנפרד.

**מלכודה: בלבול בין הפרוטוקול למסגרת.** AG‑UI איננה מסגרת קידמת‑קצה. A2UI איננה ספריית React. הם פורמטים של חוטים ופרוטוקולי אירועים. עדיין נדרש להשתמש במסגרת קידמת‑קצה כדי לממש אותם. CopilotKit מממשת AG‑UI ו‑A2UI. `json-render` מממשת את דפוס הקטלוג A2UI/Open‑JSON‑UI. אלו שכבות שונות.

## המלצות לפי מקרה שימוש

**הוספת קופיילוט לאפליקציית SaaS קיימת**: להתחיל עם דפוס 1 (כלי‑לרכיב). להשתמש ב‑`useChat` של Vercel AI SDK או ב‑CopilotKit. למפות את חמש‑עשר הפעולות המרכזיות של הסוכן לרכיבים מוכנים מראש. לשחרר זאת, למדוד, ואז להרחיב את הקטלוג רק אם המשתמשים מצביעים בפועל על צורך בהרכבות מורכבות יותר.

**יצירת לוח מחוונים משפה טבעית**: להשתמש בדפוס 2 עם `json-render` או קטלוג A2UI מותאם. להגדיר קטלוג של 8‑15 סוגי רכיבים המכסים את סוגי הגרפים, כרטיסי המדדים והגרסאות של טבלאות. להעביר את הסכמה למודל; לאפשר לו להרכיב את הפריסה. לבנות אימות שתופס סוגים לא מוכרים לפני שהם מגיעים לרנדרר.

**קידמת‑קצה מרובת‑סוכנים**: להשתמש ב‑CopilotKit עם AG‑UI. זרם האירועים מטפל ב‑streaming בזמן אמת בין backend‑ים של סוכנים; מצב משותף מנהל את המעבר בין סוכנים; דפוס HITL מטפל בשערי אישור.

**פיתוח בתוך ChatGPT או מארח MCP אחר**: להשתמש ב‑MCP Apps. להגדיר את הכלי שלך ככלי נתונים שמבצע שליפה והיסק, וכלי רינדור נפרד שמבקש וידג׳ט. לשמור את הלוגיקה העסקית מחוץ לתבנית הווידג׳ט.

**הדגמות מודלי ML ויישומי נתונים (צוות פייתון)**: Gradio להדגמות ו‑HuggingFace Spaces. Streamlit ליישומי נתונים עם אינטראקציה מורכבת יותר. אף אחד מהם אינו דורש JavaScript.

**אומנות חזותית, סימולציות, דיאגרמות**: השתמשו בדפוס 3 (OpenGenerativeUI או מקביל). הקימו CSP קפדני ל‑iframe. התייחסו לפלט כאל תוכן משתמש לא מהימן מנקודת מבט אבטחתית.

המסגרות מתבגרות בקצב מהיר. ההתכנסות של הפרוטוקול (AG‑UI ל‑streaming, A2UI/Open‑JSON‑UI למפרטי הקטלוג) עדיין בתהליך, אך הצורה מתבהרת במידה מספקת כדי להתחיל לבנות עליה.

האתגרים ההנדסיים החשובים ביותר כרגע אינם בחירת המסגרת. הם תכנון הקטלוג — החלטה מה המודל מורשה לומר, דבר הדורש בהירות מוצרית יותר מאשר מיומנות טכנית. הם אימות סמנטי — בדיקה שה‑UI שנוצר עושה מה שהוא טוען, ולא רק שעובר אימות סכימה. והם פער הנגישות — בניית קטלוגים שבהם כל רכיב, וכל הרכבה של רכיבים, עומדים ברף הנגישות שהייתם מציבים ל‑UI שנכתב ידנית.

המודל יעשה בדיוק את מה שאתם מגדירים לו בתוך הדקדוק שאתם מספקים. הפכו את הדקדוק למכוון.
