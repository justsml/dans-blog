« Generative UI » signifie au moins cinq choses distinctes selon l'interlocuteur.

- Les interfaces de chat qui intègrent des cartes produits issues des appels d'outils du modèle
- Des spécifications JSON à runtime que le frontend rend sous forme d'arbres de composants
- Des iframes sandboxées retournées par les outils MCP dans des applications hôtes (de la commande de billets, la réservation d'hôtel au rendu de cartes, les widgets de paiement)
- Des protocoles événementiels qui transmettent l'état de l'agent au frontend
- v0, Lovable et Bolt : des outils IA qui écrivent du React au moment du design

Ces concepts sont liés, mais ils opèrent à différentes couches de la pile et présentent des profils de risque, des coûts de mise en œuvre et des cas d'usage appropriés différents. Les confondre transforme chaque discussion d'architecture en gâchis.

Ceci est la carte que je veux quand je décide où agir dans la pile.

---

## Ce que Generative UI n'est pas

Avant de définir ce que c'est, trois choses à écarter :

**La génération de code au moment du design** — v0, Lovable, Bolt, Cursor composant des composants React. Ces outils génèrent du code que les développeurs examinent et valident. L'IA s'exécute au moment du développement. Quoi que soit livré est statique du point de vue de l'utilisateur. C'est une excellente catégorie d'outils. Ce n'est pas ce que signifie « generative UI à runtime ».

**Le pré-remplissage de formulaires par IA** — le modèle remplit les valeurs des champs à partir du contexte. La structure de l'interface reste fixe ; seul le contenu varie. C'est un schéma utile. Ce n'est pas du generative UI.

**L'IA écrivant du HTML brut dans une page** — le modèle produit des chaînes `<div>` et `<button>` injectées via `innerHTML` ou `dangerouslySetInnerHTML`. C'est *techniquement* du generative UI à runtime dans son sens le plus large. C'est aussi la version la plus dangereuse, et celle pour laquelle chaque framework mature de cet espace existe pour l'éviter. Le balisage brut généré par IA signifie risque XSS, attributs d'accessibilité manquants, styles incohérents et structure hallucinée. Le reste de cet article explique comment faire mieux.

## Une définition de travail

Le generative UI à runtime signifie : **le modèle détermine quel composant d'interface ou quelle composition de composants l'utilisateur voit, en fonction de l'état de la conversation ou de la tâche.**

Pas les mots. L'interface.

Le cas le plus simple : votre assistant de réservation de vols appelle un outil `search_flights`. Au lieu de retourner du texte brut (« Voici trois options... »), il rend un composant `<FlightResultsCard>` avec des vols sélectionnables, des interrupteurs de classe de siège et un bouton « Book ». Le modèle a décidé qu'une carte structurée était la bonne réponse ici. Le développeur a décidé de l'apparence de cette carte et de ce que fait « Book ».

Le cas plus complexe : un agent d'analyse financière reçoit une question sur un portefeuille et décide de composer une réponse avec un `MetricGroup` montrant les chiffres clés, un `RiskBreakdown` chart, un `ScenarioComparison` table et un `PolicyNotice`. Le modèle a assemblé cette mise en page à partir d'un catalogue de composants pré-approuvés. Le développeur a défini chaque composant. Le modèle a choisi lesquels utiliser et quelles données y mettre.

Les deux cas sont du generative UI. Ils diffèrent par le degré de liberté de composition accordé au modèle, ce qui détermine à la fois la richesse des sorties possibles et la complexité de ce qui peut mal tourner.

## Les trois schémas

L'ensemble de l'espace se réduit à trois schémas, chacun avec une grammaire de sortie différente.

![Un diagramme de spectre montrant trois schémas : appels d'outils seuls à gauche (le plus sûr), catalogue de composants au milieu, et génération ouverte à droite (la plus expressive).](../output-grammar-spectrum.svg)

_Toute décision de generative UI est un point sur ce spectre. Commencez à gauche._

### Schéma 1 : Rendu outil-composant

Le modèle appelle un outil nommé. Votre application a une correspondance des noms d'outils vers les composants. L'appel d'outil déclenche un rendu de composant.

```tsx
// Le modèle appelle : { name: "show_flight_results", args: { flights: [...] } }

useCopilotAction({
  name: "show_flight_results",
  render: ({ args }) => <FlightResultsCard flights={args.flights} />,
});
```

C'est le schéma le plus sûr parce que la mise en page ne vient jamais du modèle. Le modèle décide *quand* afficher un composant et *quelles données* y populer. Vos développeurs possèdent toujours le code du composant, le design visuel, l'implémentation de l'accessibilité et chaque cas limite dans la logique de rendu.

Le `useChat` de Vercel AI SDK avec les gestionnaires d'outils fait cela. Le rendu d'outils de assistant-ui fait cela. Le « Static Generative UI » de CopilotKit est ce schéma. La plupart des copilots UI de production qui fonctionnent de manière fiable font cela.

**Approprié quand** : l'ensemble des choses que vous pourriez vouloir afficher est connaissable au moment du développement. Confirmations de réservation, résultats de recherche, synthèses de compte, widgets d'approbation. Si vous pouvez énumérer les scénarios, ce schéma les couvre.

### Schéma 2 : Composition de catalogue de composants

Le modèle émet un arbre JSON typé qui référence des composants d'un catalogue défini par le développeur. Votre frontend a un renderer qui parcourt l'arbre et instancie chaque composant.

```json
[
  { "type": "metric_group", "metrics": [
    { "label": "MRR", "value": "$82,400", "delta": "+12%" },
    { "label": "Churn", "value": "2.1%", "delta": "-0.4%" }
  ]},
  { "type": "line_chart", "title": "30-day growth", "data_ref": "mrr_series" },
  { "type": "insight_callout", "text": "Expansion revenue driving the delta — avg seat count up 18%." }
]
```

Le modèle a composé cette mise en page. Un `MetricGroup`, un `LineChart`, un `InsightCallout`. Mais vous avez défini ce que chaque type de composant signifie, quelles props il accepte et comment il rend. Si le modèle tente d'émettre `{ "type": "custom_untested_thing" }`, votre validation de schéma le détecte et le renderer ignore ou rejette.

C'est le schéma derrière `json-render`, `A2UI`, `Hashbrown`, `OpenUI` et `Tambo`. Le travail d'ingénierie clé est la **conception du catalogue** — décider quels types de composants existent, à quoi ressemblent leurs schémas et ce que le modèle a le droit ou non de composer.

**Approprié quand** : la structure de ce que vous voulez afficher varie légitimement selon les données ou la demande de l'utilisateur. Des tableaux de bord qui s'adaptent à ce qui est notable dans les chiffres. Des rapports qui montrent différentes sections selon le contexte. Des panneaux de workflow qui changent selon l'étape où se trouve l'agent.

### Schéma 3 : Génération ouverte

Le modèle écrit du HTML, SVG, Canvas ou WebGL qui est rendu dans une iframe sandboxée avec une politique de sécurité de contenu stricte.

C'est approprié pour les choses où aucun catalogue de composants fixes ne convient : visualisations d'algorithmes, diagrammes d'architecture, graphiques ad hoc, art génératif, simulations éducatives. La limite de l'iframe fait le travail de sécurité ici ; supprimez-la et vous revenez au problème d'injection HTML brut du début de cet article.

`CopilotKit/OpenGenerativeUI` est la meilleure implémentation de référence actuelle de ce schéma. Le sandbox supprime les scripts, limite la transmission de messages et garde l'artefact généré éloigné de l'état privilégié de votre application.

**Approprié quand** : vous avez réellement besoin d'une sortie visuelle arbitraire — diagrammes explicatifs uniques, simulations dynamiques, artefacts créatifs. Ne l'utilisez pas pour des UI transactionnelles. Une confirmation de paiement n'a pas besoin d'une iframe sandboxée.

### Au-delà des trois schémas : les LLM pilotant les pixels directement

Il y a une quatrième direction émergente qui ne s'intègre proprement dans aucun de ces schémas : les LLM pilotant des expériences **immersives, type jeu vidéo** en contrôlant la sortie visuelle plus directement qu'une iframe sandboxée.

La distinction canonique au sein du generative UI est **iframe HTML vs. JSON catalog** :

- **Iframe HTML** — le modèle écrit du HTML, SVG, Canvas ou WebGL rendu dans un sandbox isolé. Liberté d'expression maximale ; la sécurité dépend entièrement de la limite de l'iframe. Exemples : Anthropic Artifacts, OpenGenerativeUI.
- **JSON catalog** — le modèle émet une payload structurée contrainte à un catalogue de composants défini par le développeur ; votre renderer instancie des composants pré-construits et fiables à partir de cette spec. Le modèle décide *quoi* afficher ; vous décidez *comment* le rendre. Exemples : json-render, A2UI.

Au-delà de ces deux, des démos très récentes suggèrent un troisième mode où le modèle ne choisit pas des composants ou n'écrit pas du HTML sandboxé — il pilote le canvas plus directement. Des projets comme [Tencent's HunyuanWorld](https://arxiv.org/abs/2502.01999), qui génèrent des environnements 3D explorables à partir d'une seule image, et des architectures de jeu où les LLM génèrent des cartes, des PNJ et des quêtes à runtime plutôt que d'appeler un catalogue de composants, suggèrent un futur où le modèle ressemble plus à un directeur de jeu qu'à un moteur de formulaires. L'inférence LLM dans le navigateur via WebGPU ([WebLLM](https://mlc.ai/web-llm/)) pousse la même frontière localement.

Ce territoire est véritablement excitant et véritablement précoce. Il n'y a pas encore de frameworks stables pour construire des produits de production. Je couvrirai cette approche dans un article dédié quand cela changera.

## Les protocoles : AG-UI et A2UI

AG-UI et A2UI sont les deux standards principaux de la couche protocole. Ils résolvent des problèmes différents et ne sont pas concurrents.

### AG-UI

**GitHub** : [ag-ui-protocol/ag-ui](https://github.com/ag-ui-protocol/ag-ui)

AG-UI est un protocole basé sur les événements pour la communication entre les agents IA et les applications frontend. Il définit environ 16 types d'événements : `TEXT_MESSAGE_START`, `TEXT_MESSAGE_CONTENT`, `TOOL_CALL_START`, `TOOL_CALL_END`, `STATE_SNAPSHOT`, `STATE_DELTA`, etc. Le transport vous appartient — SSE, WebSockets, webhooks fonctionnent tous. Le format est délibérément flexible pour permettre une adoption large.

AG-UI ne définit pas à quoi ressemble votre UI. Il définit comment l'agent communique *avec* votre frontend. Considérez-le comme la couche protocole qui permet à votre application React de s'abonner à un agent LangGraph de la même manière qu'elle s'abonne à un agent CrewAI, sans changer de code frontend.

CopilotKit a créé AG-UI à partir de leur travail avec LangGraph et CrewAI. Il a été adopté par LangChain, Mastra, PydanticAI et d'autres. Microsoft a publié un guide d'intégration AG-UI. Si vous construisez un frontend multi-agents et avez besoin de découpler les frameworks backend du code frontend, AG-UI est la réponse.

**Une clarification qui déroute** : AG-UI n'est pas un framework UI. Il ne vous dit pas quoi rendre. Il vous dit *que* l'agent a dit quelque chose, a appelé un outil ou a mis à jour un état partagé. Ce que vous rendez en réponse est toujours votre décision.

### A2UI

**GitHub** : [google/A2UI](https://github.com/google/A2UI) · Spec : [a2ui.org](https://a2ui.org/)

A2UI est la spécification déclarative de Google pour ce que les agents envoient quand ils veulent afficher une UI. Là où AG-UI répond « comment l'agent communique-t-il ? », A2UI répond « quel format l'agent utilise-t-il pour décrire une disposition de composants ? ».

A2UI utilise un format JSONL plat : un descripteur de composant par ligne, chacun avec un ID, un type et des données. Plat est intentionnel. Les arbres nichés nécessitent que le modèle connaisse la structure complète avant de pouvoir commencer à streamer. Une liste plate permet au modèle d'émettre chaque composant quand il « réfléchit » à celui-ci, ce qui signifie que votre frontend peut commencer à rendre la première carte métrique pendant que le modèle décide encore s'il veut ajouter un graphique.

```jsonl
{"id":"h1","type":"kpi_card","title":"MRR","value":"$82,400","delta":"+12%"}
{"id":"h2","type":"kpi_card","title":"Churn","value":"2.1%","delta":"-0.4%"}
{"id":"c1","type":"line_chart","title":"30-day MRR","data_ref":"mrr_series"}
{"id":"t1","type":"data_table","cols":["Month","MRR","Net New"],"data_ref":"monthly"}
```

A2UI est soucieux de la sécurité : la spec est un format de données, pas du code exécutable. Le catalogue de composants est pré-défini par le développeur ; l'agent ne peut référencer que les types de ce catalogue. Un renderer A2UI qui reçoit un nom de type inconnu l'ignore.

Le format « Open-JSON-UI » de CopilotKit est compatible avec A2UI. Si vous choisissez un format de spec pour un catalogue de composants aujourd'hui, A2UI est celui avec le soutien multiplateforme le plus large.

**Une note sur la stabilité** : A2UI est pré-1.0 — v0.9 lors de la dernière vérification le 8 mai 2026 — et a connu des changements cassants entre versions mineures. Les communications de Google sur la roadmap ont été sporadiques et certains renderers (Lit, Flutter) ont traîné derrière les mises à jour de spec. Prévoyez du temps pour la dérive de spec si vous construisez dessus aujourd'hui. Pour les cas d'usage web pur, json-render semble avoir des outils plus complets actuellement. L'avantage à long terme de A2UI est la portée multiplateforme (web, Flutter, SwiftUI, Android) que json-render n'a pas.

### MCP Apps

**GitHub** : [modelcontextprotocol](https://github.com/modelcontextprotocol) · Relatif : [mcp-ui](https://github.com/MCP-UI-Org/mcp-ui)

MCP a commencé comme un protocole pour connecter les LLMs aux outils et données. L'extension Apps permet aux outils MCP de retourner non seulement des données, mais des artefacts UI interactifs : composants React, formulaires, tableaux de bord, cartes.

Le modèle de sécurité est strict et c'est le but : tout rend dans une iframe sandboxée avec des permissions剥离, les templates sont pré-déclarés pour que l'application hôte puisse les examiner, et toute communication est de la JSON-RPC auditable. C'est le bon modèle pour les fournisseurs d'outils — un serveur MCP Shopify peut retourner un widget de paiement ; un service de cartographie peut retourner une carte intégrable. L'application hôte ne possède pas ou ne fait pas confiance au code de ce widget.

MCP Apps est le bon choix quand l'UI *appartient au fournisseur d'outils*, pas à votre application. Pour l'UI qui réside dans le domaine de votre application, restez avec le Schéma 1 ou 2.

## Les frameworks JavaScript/TypeScript

### CopilotKit

**GitHub** : [CopilotKit/CopilotKit](https://github.com/CopilotKit/CopilotKit) · Exemples : [CopilotKit/generative-ui](https://github.com/CopilotKit/generative-ui)

CopilotKit est le framework le plus complet pour les applications frontend natives aux agents. Il gère le cycle de vie complet : connexion aux backends d'agents via AG-UI, gestion d'état conversationnel bidirectionnel, rendu de composants generative UI, et shared state plumbing qui permet aux agents et aux utilisateurs de modifier les mêmes données.

Le modèle à trois schémas se map cleanly aux API CopilotKit :
- `useCopilotAction` avec un callback `render` → Schéma 1
- Rendu A2UI/Open-JSON-UI → Schéma 2
- Artefacts sandboxés `OpenGenerativeUI` → Schéma 3

La fonctionnalité CopilotKit importante qui est sous-discutée est **l'état partagé et l'humain dans la boucle** : l'agent peut lire et écrire l'état de l'application, l'utilisateur peut le lire et l'écrire, et les changements circulent bidirectionnellement. C'est ce qui rend les UI de style copilote feel like a real collaboration plutôt qu'un chat collé à un produit.

### Vercel AI SDK

**GitHub** : [vercel/ai](https://github.com/vercel/ai) · Docs : [ai-sdk.dev](https://ai-sdk.dev/)

Le Vercel AI SDK est la référence de facto TypeScript pour les apps IA. Pour le generative UI spécifiquement :

**`useObject`** stream un objet JSON structuré depuis le serveur au fur et à mesure de sa génération. Vous définissez un schéma Zod ; le SDK parse le JSON partiel et déclenche des re-rendus au fur et à mesure de l'arrivée des champs. C'est le chemin le plus fluide vers le Schéma 2 dans une app Next.js.

```tsx
const { object: dashboard } = useObject({
  api: "/api/generate-dashboard",
  schema: z.object({
    title: z.string(),
    metrics: z.array(z.object({ label: z.string(), value: z.number() })),
    insights: z.array(z.string()),
  }),
});
```

**`useChat` avec gestionnaires d'outils** → Schéma 1. Le modèle appelle des outils ; vous mappez les noms d'outils vers des composants.

**AI Elements** ([elements.ai-sdk.dev](https://elements.ai-sdk.dev/)) fournit des primitives UI prêtes à l'emploi à coupler avec le SDK.

**Une note sur la trajectoire confuse** : En octobre 2024, Vercel a annoncé dans [GitHub Discussion #3251](https://github.com/vercel/ai/discussions/3251) que AI SDK RSC — le pattern de streaming React Server Components promu comme la fonctionnalité « Generative UI » de référence dans SDK 3.0 — était en pause indéfiniment en raison de « plusieurs limitations de longue date » sans de bonnes solutions à court terme. Les équipes qui avaient bâti une stratégie produit autour du streaming RSC ont été prises au dépourvu. Les API `generateObject`/`streamObject` ont également été dépréciées plus tard dans SDK 6.0. La migration recommandée depuis AI SDK RSC est le pattern `useObject` ci-dessus, ou json-render pour la génération basée sur un catalogue.

### assistant-ui

**GitHub** : [assistant-ui/assistant-ui](https://github.com/assistant-ui/assistant-ui)

assistant-ui est un ensemble de primitives React composables pour construire des interfaces de chat de qualité production. C'est la bonne réponse quand vous avez besoin d'un chat UX poli — bulles de message, tokens en streaming, actions copier/éditer/régénérer, états de réflexion — et voulez apporter votre propre backend et votre propre rendu d'outils.

Il fonctionne bien pairé avec n'importe quel backend (OpenAI, Anthropic, modèles locaux, endpoints personnalisés) et gère le rendu d'appels d'outils via un modèle familier de slot/render prop.

### json-render

**GitHub** : [vercel-labs/json-render](https://github.com/vercel-labs/json-render) · Docs : [json-render.dev](https://json-render.dev/)

json-render opérationnalise le Schéma 2 avec une approche opinionnée et complète. Vous obtenez un catalogue de composants pré-construits (composants shadcn/ui avec schémas Zod), un renderer, et une boucle de génération serrée où le modèle est contraint au catalogue par le schéma.

Les fonctionnalités distinctives :
- **Rendu multi-cibles** : la même spec JSON peut rendre dans une app web React, une app mobile React Native, un PDF, un email HTML ou une vidéo Remotion. C'est véritablement utile pour les rapports.
- **Rendu progressif** : les composants apparaissent au fur et à mesure que le modèle les stream, pas après l'arrivée de la spec complète.
- **Contraintes de schéma serrées** : le catalogue est conçu pour que le modèle ne puisse pas halluciner de types de composants valides mais inconnus.

Si vous construisez une fonctionnalité de génération de tableaux de bord ou de rapports et voulez sauter le travail d'infrastructure de conception de votre propre catalogue, json-render est le chemin le plus rapide pour les apps web.

**Sur l'élan** : json-render a été lancé par Vercel Labs début 2026 et semble avoir rapidement attiré l'attention des développeurs web parce qu'il est immédiatement utile dans des projets React/Next.js standards. Cela dit, json-render est encore pré-1.0 et la relation entre json-render et A2UI est encore en cours de travail — Vercel a expérimenté avec une sortie compatible A2UI, donc une convergence est possible. Pour le multiplateforme (mobile natif, plusieurs frameworks), A2UI est le meilleur pari à long terme.

### Hashbrown

**GitHub** : [liveloveapp/hashbrown](https://github.com/liveloveapp/hashbrown)

Hashbrown adopte une approche distinctive : au lieu de construire une couche d'interface IA séparée, il intègre la sélection de composants IA directement dans votre application React ou Angular existante. Vous exposez les composants de votre app au LLM ; le LLM sélectionne lesquels rendre et peut invoquer des outils côté client.

C'est le bon outil quand vous voulez infuser de l'intelligence dans des surfaces produit qui ne sont pas du « chat » — une page produit qui adapte sa mise en page, un panneau de paramètres qui fait apparaître les bonnes options, un éditeur de workflow qui suggère l'étape suivante.

### OpenUI

**GitHub** : [thesysdev/openui](https://github.com/thesysdev/openui) · Docs : [openui.com](https://www.openui.com/)

OpenUI remplace JSON par un format code-like orienté ligne (« OpenUI Lang ») conçu pour le rendu progressif et l'efficacité token. La claim est d'environ 67% de tokens en moins que du JSON équivalent pour des mises en page complexes.

Le compromis est la maturité de l'écosystème — OpenUI est plus récent et les outils sont plus minces que les approches basées sur JSON. Mais si le coût en tokens est une contrainte significative et que vous générez des mises en page complexes à haute fréquence, l'efficacité du format est réelle.

### Tambo

**GitHub** : [tambo-ai/tambo](https://github.com/tambo-ai/tambo)

Tambo se concentre sur la sélection de composants avec état : l'IA sélectionne des composants et peut interagir avec eux via des outils côté client, maintenant l'état des composants à travers la conversation. Bon pour les cas d'usage où les éléments UI persistent à travers les tours — un composant filtre que l'utilisateur ajuste pendant que l'IA continue de raisonner sur les données filtrées.

## La couche Python

L'écosystème Python approche les interfaces IA différemment. Ces outils sont optimisés pour les démos de modèles ML, les applications de données et les outils internes — pas les apps consommateurs de production avec composition de mise en page pilotée par agent.

Ce n'est pas une critique. Pour les bons cas d'usage, Gradio et Streamlit sont les seuls outils dont vous avez besoin.

### Gradio

**GitHub** : [gradio-app/gradio](https://github.com/gradio-app/gradio) · PyPI : `gradio`

La valeur centrale de Gradio : vous écrivez une fonction Python ; Gradio l'enveloppe dans une UI web. La classe `Interface` c'est 3 lignes pour un classificateur d'images. `ChatInterface` c'est 10 lignes pour un chatbot. `Blocks` vous donne un contrôle de mise en page fin quand vous en avez besoin.

Le « generative UI » dans Gradio est défini par le développeur Python, pas par le modèle. La visibilité et la configuration des composants peuvent changer dynamiquement selon les sorties du modèle, mais le catalogue de composants est statique — vous ne demandez pas au modèle de composer des mises en page.

Gradio est la valeur par défaut pour les HuggingFace Spaces et l'écosystème de démos ML. Il a des millions de téléchargements mensuels et alimente une grande partie du paysage de démos IA.

**Utilisez Gradio quand** : vous êtes un développeur Python construisant une démo de modèle ML, un prototype de recherche ou un outil interne, et ne voulez pas toucher au JavaScript.

### Streamlit

**GitHub** : [streamlit/streamlit](https://github.com/streamlit/streamlit)

Le modèle de Streamlit est plus opinionné : un script Python s'exécute de bout en bout à chaque interaction. Vous appelez `st.chat_message()`, `st.dataframe()`, `st.plotly_chart()`. Le framework gère la mise en page.

Le modèle de re-run complet du script ressemble à une inefficacité mais est surprenamment ergonomique pour les chatbots IA qui accumulent l'historique de conversation — le script entier se ré-exécute, l'historique de chat est dans l'état de session, et la sortie est déterministe. Streamlit a maintenant le support de première partie pour la plupart des fournisseurs de LLM majeurs et s'intègre nativement avec Snowflake Cortex.

**Utilisez Streamlit quand** : vous construisez une app de données alimentée par IA, un outil de reporting interne ou un tableau de bord backing par ML en Python et voulez le chemin de déploiement le plus simple possible.

### LangChain et Haystack

Ce sont des frameworks d'orchestration backend, pas des frameworks UI. Ils apparaissent dans toute carte honnête de stack generative UI parce qu'ils sont typiquement la couche où les sorties structurées sont générées avant d'être envoyées à un frontend.

**LangChain** ([langchain-ai/langchain](https://github.com/langchain-ai/langchain)) : `.with_structured_output()` sur n'importe quel LLM vous donne de la génération JSON contrainte par Pydantic. Le décorateur `@tool` avec génération automatique de schéma est la façon la plus propre de définir quels outils le modèle peut appeler. LangChain envoie les résultats structurés vers la couche frontend que vous utilisez.

**Haystack** ([deepset-ai/haystack](https://github.com/deepset-ai/haystack)) : architecture de pipeline modulaire avec un support RAG fort. Hayhooks enveloppe les pipelines Haystack comme endpoints HTTP — y compris les endpoints compatibles MCP. Si votre generative UI a besoin d'une base de retrieval, l'architecture de pipeline de Haystack gère cela proprement.

Aucun framework ne possède la couche UI. Ils génèrent les données que votre frontend (Schéma 1, 2 ou 3) rend.

## Vélocité de l'écosystème et terrain instable

Cet espace bouge vite et plusieurs projets ont envoyé des communications confuses aux côtés de leur code. Dernière vérification le 8 mai 2026 ; treat the project-status notes here as a timestamped read, pas un verdict permanent.

**Vercel AI SDK RSC** était la fonctionnalité Generative UI flagship quand SDK 3.0 a été lancé. Vercel a mis son développement en pause en octobre 2024 ([Discussion #3251](https://github.com/vercel/ai/discussions/3251)) citant des limitations architecturales avec React Server Components qui n'avaient pas de solution à court terme. Les équipes qui avaient bâti dessus étaient compréhensiblement frustrées. C'est toujours dans les docs mais ce n'est pas le chemin recommandé ; `useObject` l'est.

**json-render** (Vercel Labs) est la nouvelle direction — une alternative basée sur un catalogue, agnostique du framework qui évite les problèmes de couplage RSC. C'est pré-1.0 et semble avoir un intérêt précoce fort parmi les développeurs React/web. La raison DX probable : json-render est immédiatement utile dans un projet React/Next.js standard, tandis que la portée multiplateforme de A2UI ajoute de la friction de setup. Comment cela évoluera au fur et à mesure que les deux specs murissent est véritablement incertain. Vercel a sondé la compatibilité A2UI dans json-render, suggérant qu'une convergence est possible.

**A2UI** (Google) est pré-1.0 (v0.9 lors de la dernière vérification) avec des changements cassants entre versions mineures et des communications de Google sur sa roadmap inconsistantes. C'est le bon choix pour la portée multiplateforme (web + Flutter + SwiftUI) que json-render n'adresse pas, et il a un soutien entreprise significatif. Pour les projets web purs maintenant, le DX est plus rugueux.

**AG-UI** (CopilotKit) est aussi pré-1.0. La confusion la plus commune : le nom donne l'impression que c'est un framework UI. Ce n'est pas — c'est un protocole de transport. AG-UI définit comment les événements circulent entre les backends d'agents et votre frontend ; ce que vous rendez en réponse est toujours votre décision. Ce modèle mental est solide et largement adopté, mais la spec pré-1.0 signifie que les cas limites sont encore en cours de résolution.

Le résultat pratique : **chaque joueur majeur ici est pré-1.0**. Prévoyez des changements d'API. Les schémas — outil-vers-composant, composition de catalogue, génération sandboxée — sont assez stables pour construire dessus. Les choix de protocole spécifiques ne le sont pas.

## Conception du catalogue de composants : le vrai travail d'ingénierie

La plupart de la complexité intéressante dans le Schéma 2 n'est pas dans le renderer — c'est dans le catalogue.

Le catalogue est **une décision produit encodée en schéma**. Il répond : quels sont les objets UI significatifs dans ce domaine ? Pas « quels composants React existent ? » mais « de quoi un utilisateur dans ce contexte a-t-il réellement besoin de voir et d'interagir ? »

**Le mode d'échec trop granulaire** : vous exposez `Row`, `Column`, `Text`, `Button`, `Icon`. Maintenant le modèle doit être un ingénieur frontend. Il générera une mise en page médiocre qui ne correspond pas à votre design system, manquera les états vides, produira un balisage inaccessible et changera son approche de réponse en réponse parce que rien dans le catalogue ne contraint la sortie au langage visuel de votre produit.

**Le mode d'échec trop grossier** : vous exposez `WeatherCard`, `FlightCard`, `HotelCard`. Le modèle ne peut pas s'adapter quand l'utilisateur demande quelque chose qui ne correspond pas à une carte pré-cuite. Il revient au texte.

**Le milieu utile** : composants de niveau domaine avec slots contraints.

Un catalogue d'app de voyage pourrait ressembler à :

```
TripSummary         — aperçu de l'itinéraire
FlightOptionList    — options de vol sélectionnables avec prix
HotelComparison     — cartes hôtelières comparatives
TravelerForm        — collecter les détails du voyageur
PolicyNotice        — encadré règles tarifaires/réglementaires
BookingConfirmation — confirmation finale avec bouton d'action
```

Un catalogue d'app finance pourrait ressembler à :

```
PortfolioSnapshot   — positions clés et P&L
TransactionTable   — transactions filtrables, paginées
RiskBreakdown      — allocation et métriques de volatilité
ScenarioComparison  — modélisation de scénarios comparatifs
ApprovalGate       — action nécessitant confirmation humaine
```

Le catalogue ressemble au vocabulaire de votre produit. Il encode vos décisions UX, vos exigences d'accessibilité, votre gestion des états vides et vos schémas d'actions dangereuses dans du code composant. Le modèle peut arranger ces pièces. Vous décidez toujours ce que chaque pièce ressemble et ce qu'elle a le droit de faire.

**Règles de conception de schéma qui réduisent les hallucinations** :

1. Gardez les valeurs enum courtes et évidentes. `"type": "bar_chart"` pas `"type": "data-visualization-bar-type-vertical"`.
2. Rendez la composition invalide impossible. Si un `PolicyNotice` ne peut apparaître qu'à la fin d'une mise en page, ne le mettez pas au même niveau de schéma que les choses qui peuvent apparaître n'importe où.
3. Utilisez les champs requis généreusement. Un champ optionnel est un champ que le modèle pourrait omettre et que votre renderer doit gérer comme null.
4. Testez le catalogue contre de vraies invites avant de shipper. Sauvegardez les specs générées ; vérifiez-les pour les violations de schéma, les valeurs de champs hallucitées et les compositions techniquement valides mais sémantiquement fausses.

## Pièges courants

**Piège : traiter du JSON valide comme un comportement sûr.** La validation de schéma confirme la structure. Elle ne dit rien sur si l'action attachée à un bouton correspond à son label, si un total correspond aux données dont il est dérivé, ou si un composant UI fait quelque chose que l'utilisateur n'attendait pas. Les specs UI générées ont besoin d'une revue sémantique, pas juste d'une validation de schéma. Au minimum, les actions destructives devraient nécessiter un composant de confirmation, et les labels sur ces composants devraient être testés contre les actions qu'ils déclenchent.

**Piège : exposer des primitives de design au lieu de primitives produit.** Si le modèle doit décider s'il utilise 16px ou 20px de padding, vous lui avez donné le mauvais niveau d'abstraction. Les composants de domaine devraient encoder le goût produit. Le modèle devrait composer des comportements, pas gérer des détails de présentation.

**Piège : utiliser du generative UI là où du static UI fera l'affaire.** Si la structure de ce que vous voulez afficher est connaissable au moment du développement — et c'est habituellement le cas — le Schéma 1 avec des composants pré-construits est plus rapide, plus sûr et plus cohérent. Le generative UI mérite sa complexité quand la structure varie véritablement selon les données ou le contexte de la tâche.

**Piège : zapper l'accessibilité.** Les LLMs hallucinent des violations WCAG. Ils émettront `role="region"` sur des éléments interactifs, généreront des formulaires avec des labels manquants et produiront des ratios de contraste qui échouent au WCAG AA. Votre bibliothèque de composants peut être pleinement accessible ; les compositions de ces composants générées par IA ne sont pas automatiquement accessibles. Testez le chemin de rendu complet, pas juste les composants isolément.

**Piège : confondre le protocole et le framework.** AG-UI n'est pas un framework frontend. A2UI n'est pas une bibliothèque React. Ce sont des formats de transmission et des protocoles événementiels. Vous avez toujours besoin d'un framework frontend pour les implémenter. CopilotKit implémente AG-UI et A2UI. json-render implémente le schéma de catalogue A2UI/Open-JSON-UI. Ce sont des couches différentes.

## Recommandations par cas d'usage

**Ajouter un copilote à une app SaaS existante** : Commencez avec le Schéma 1 (outil-vers-composant). Utilisez `useChat` de Vercel AI SDK ou CopilotKit. Mappez vos 5–10 actions d'agent principales vers des composants pré-construits. Livrez cela, mesurez, puis élargissez le catalogue seulement si les utilisateurs ont manifestement besoin d'une composition plus riche.

**Génération de tableaux de bord depuis le langage naturel** : Utilisez le Schéma 2 avec json-render ou un catalogue A2UI personnalisé. Définissez un catalogue de 8–15 types de composants couvrant vos types de graphiques, cartes métriques et variantes de tableaux. Fournissez le schéma au modèle ; laissez-le composer la mise en page. Construisez une validation qui détecte les types inconnus avant qu'ils n'atteignent le renderer.

**Frontend multi-agents** : Utilisez CopilotKit avec AG-UI. Le flux d'événements gère le streaming en temps réel à travers les backends d'agents ; l'état partagé gère le transfert entre agents ; le pattern HITL gère les portes d'approbation.

**Construire dans ChatGPT ou un autre hôte MCP** : Utilisez MCP Apps. Définissez votre outil comme un outil de données qui récupère et raisonne, et un outil de rendu séparé qui demande un widget. Gardez la logique métier hors du template du widget.

**Demos de modèles ML et apps de données (équipe Python)** : Gradio pour les démos et HuggingFace Spaces. Streamlit pour les apps de données avec une interaction plus complexe. Aucun des deux n'exige de JavaScript.

**Artefacts visuels, simulations, diagrammes** : Utilisez le Schéma 3 (OpenGenerativeUI ou équivalent). Établissez une CSP iframe stricte. Traitez la sortie comme du contenu utilisateur non fiable d'un point de vue sécurité.

---

Les frameworks maturissent vite. La convergence de protocole (AG-UI pour le streaming, A2UI/Open-JSON-UI pour les specs de catalogue) est encore en cours, mais la forme devient assez claire pour construire dessus.

Les défis d'ingénierie qui comptent le plus en ce moment ne sont pas le choix de framework. Ils sont la conception du catalogue — décider ce que le modèle a le droit de dire, ce qui exige davantage de clarté produit que de compétence technique. Ils sont la validation sémantique — tester que l'UI générée fait ce qu'elle prétend, pas juste qu'elle passe la validation de schéma. Et ils sont le fossé d'accessibilité — construire des catalogues où chaque composant, et chaque composition de composants, répond au standard d'accessibilité auquel vous soumettriez l'UI écrite à la main.

Le modèle fera ce que vous lui dites de faire dans la grammaire que vous lui donnez. Rendez la grammaire délibérée.
