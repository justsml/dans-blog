« Generative UI » signifie au moins cinq choses distinctes selon l'interlocuteur.

- Les interfaces de chat qui intègrent des cartes produit issues des appels d'outils du modèle
- Des spécifications JSON exécutées côté runtime que le frontend render en arbres de composants
- Des iframes bac à sable retournées par des outils MCP dans des applications hôtes (de la commande de tickets, réservation d'hôtels au rendu de cartes, widgets de paiement)
- Des protocoles d'événements qui transmettent l'état de l'agent au frontend
- v0, Lovable et Bolt : des outils IA qui génèrent du React au moment de la conception

Ces concepts sont liés, mais ils résident à des couches différentes de la pile et présentent des profils de risque, des coûts d'implémentation et des cas d'usage appropriés distincts. Les mêler transforme chaque discussion d'architecture en gâchis.

Voici la carte que je veux quand je décide où intervene dans la pile.

---

## Ce que Generative UI n'est pas

Avant de définir ce que c'est, trois choses à écarter :

**La génération de code au moment de la conception** — v0, Lovable, Bolt, Cursor composant des composants React. Ces outils génèrent du code que les développeurs examinent et mergent. L'IA s'exécute au moment du développement. Quoi que soit livré est statique du point de vue de l'utilisateur. C'est une excellente catégorie d'outils. Ce n'est pas ce que signifie « generative UI à l'exécution ».

**Le remplissage automatique de formulaires assisté par IA** — le modèle remplissant des valeurs de champ à partir du contexte. La structure de l'interface reste fixe ; seul le contenu change. C'est un pattern utile. Ce n'est pas du generative UI.

**L'IA écrivant du HTML brut dans une page** — le modèle produit des chaînes `<div>` et `<button>` injectées via `innerHTML` ou `dangerouslySetInnerHTML`. C'est *techniquement* du generative UI à l'exécution dans son sens le plus strict. C'est aussi la version la plus dangereuse, et celle pour laquelle chaque framework mature dans ce domaine existe pour éviter. Le balisage brut généré par IA signifie risque XSS, attributs inaccessibles, styles incohérents et structurehallucinée. Le reste de cet article concerne faire mieux que cela.

## Une définition opérationnelle

Le generative UI à l'exécution signifie : **le modèle détermine quel composant d'interface ou quelle composition de composants l'utilisateur voit, en fonction de l'état de la conversation ou de la tâche.**

Pas les mots. L'interface.

Le cas le plus simple : votre assistant de réservation de vols appelle un outil `search_flights`. Au lieu de retourner du texte brut (« Voici trois options... »), il rend un composant `<FlightResultsCard>` avec des vols sélectionnables, des bascules de classe de siège et un bouton « Réserver ». Le modèle a décidé qu'une carte structurée était la bonne réponse ici. Le développeur a décidé de l'apparence de cette carte et de ce que fait « Réserver ».

Le cas plus complexe : un agent d'analyse financière reçoit une question sur un portefeuille et décide de composer une réponse avec un `MetricGroup` affichant les chiffres clés, un `RiskBreakdown` chart, un `ScenarioComparison` tableau et un `PolicyNotice`. L'agent a assemblé cette mise en page à partir d'un catalogue de composants pré-approuvés. Le développeur a défini chaque composant. Le modèle a choisi lesquels utiliser et quelles données y insérer.

Les deux cas sont du generative UI. Ils diffèrent par le degré de liberté de composition laissé au modèle, ce qui détermine à la fois la richesse des sorties possibles et la complexité de ce qui peut mal tourner.

## Les trois patterns

L'ensemble du domaine se réduit à trois patterns, chacun avec une grammaire de sortie différente.

![Diagramme de spectre montrant trois patterns : appels d'outils uniquement à gauche (le plus sûr), catalogue de composants au milieu, et génération ouverte à droite (la plus expressive).](../output-grammar-spectrum.svg)

_Chaque décision de generative UI est un point sur ce spectre. Commencez à gauche._

### Pattern 1 : Rendu outil-vers-composant

Le modèle appelle un outil nommé. Votre application a une correspondance entre les noms d'outils et les composants. L'appel d'outil déclenche un rendu de composant.

```tsx
// Le modèle appelle : { name: "show_flight_results", args: { flights: [...] } }

useCopilotAction({
  name: "show_flight_results",
  render: ({ args }) => <FlightResultsCard flights={args.flights} />,
});
```

C'est le pattern le plus sûr car la mise en page ne vient jamais du modèle. Le modèle décide *quand* afficher un composant et *quelles données* y insérer. Vos développeurs possèdent toujours le code du composant, le design visuel, l'implémentation de l'accessibilité et chaque cas limite dans la logique de rendu.

Le `useChat` de Vercel AI SDK avec les gestionnaires `tool` fait cela. Le rendu d'outils de assistant-ui fait cela. Le « Static Generative UI » de CopilotKit est ce pattern. La plupart des copilot UIs de production qui fonctionnent de manière fiable font cela.

**Approprié quand** : l'ensemble des choses que vous pourriez vouloir afficher est connaissable au moment du développement. Confirmations de réservation, résultats de recherche, synthèses de compte, widgets d'approbation. Si vous pouvez énumérer les scénarios, ce pattern les couvre.

### Pattern 2 : Composition de catalogue de composants

Le modèle émet un arbre JSON typé qui référence des composants d'un catalogue défini par le développeur. Votre frontend a un renderer qui parcourt l'arbre et instancies chaque composant.

```json
[
  { "type": "metric_group", "metrics": [
    { "label": "MRR", "value": "$82,400", "delta": "+12%" },
    { "label": "Churn", "value": "2.1%", "delta": "-0.4%" }
  ]},
  { "type": "line_chart", "title": "30-day growth", "data_ref": "mrr_series" },
  { "type": "insight_callout", "text": "Expansion revenue driving the delta — avg seat count up 18%." }
]
```

Le modèle a composé cette mise en page. Un `MetricGroup`, un `LineChart`, un `InsightCallout`. Mais vous avez défini ce que chaque type de composant signifie, quelles props il accepte et comment il rend. Si le modèle essaie d'émettre `{ "type": "custom_untested_thing" }`, votre validation de schéma le détecte et le renderer ignore ou rejette.

C'est le pattern derrière `json-render`, `A2UI`, `Hashbrown`, `OpenUI` et `Tambo`. Le travail d'ingénierie clé est la **conception du catalogue** — décider quels types de composants existent, à quoi ressemblent leurs schémas et ce que le modèle a et n'a pas le droit de composer.

**Approprié quand** : la structure de ce que vous voulez afficher varie légitimement selon les données ou la demande de l'utilisateur. Tableaux de bord qui s'adaptent à ce qui est notable dans les chiffres. Rapports qui affichent différentes sections selon le contexte. Panneaux de workflow qui changent selon l'étape où se trouve l'agent.

### Pattern 3 : Génération ouverte

Le modèle écrit du HTML, SVG, Canvas ou WebGL qui est rendu dans une iframe bac à sable avec une politique de sécurité de contenu stricte.

C'est approprié pour les choses où aucun catalogue de composants fixe ne convient : visualisations d'algorithmes, diagrammes architecturaux, graphiques ad hoc, art génératif, simulations éducatives. La frontière de l'iframe fait le travail de sécurité ici ; retirez-la et vous revenez au problème d'injection de HTML brut du début de cet article.

`CopilotKit/OpenGenerativeUI` est la meilleure implémentation de référence actuelle de ce pattern. Le bac à sable supprime les scripts, limite la transmission de messages et garde l'artefact généré éloigné de l'état privilégié de votre application.

**Approprié quand** : vous avez réellement besoin d'une sortie visuelle arbitraire — diagrammes explicatifs uniques, simulations dynamiques, artefacts créatifs. Ne l'utilisez pas pour de l'UI transactionnelle. Une confirmation de paiement n'a pas besoin d'une iframe bac à sable.

### Au-delà des trois patterns : les LLMs pilotant directement les pixels

Il existe une quatrième direction émergente qui ne s'intègre proprement dans aucun de ces patterns : les LLMs pilotant des **expériences immersives de type jeu** en contrôlant la sortie visuelle plus directement qu'une iframe bac à sable.

La distinction canonique au sein du generative UI est **iframe HTML vs. catalogue JSON** :

- **Iframe HTML** — le modèle écrit du HTML, SVG, Canvas ou WebGL rendu dans un bac à sable isolé. Liberté d'expression maximale ; la sécurité dépend entièrement de la frontière iframe. Exemples : Anthropic Artifacts, OpenGenerativeUI.
- **Catalogue JSON** — le modèle émet une charge utile structurée contrainte à un catalogue de composants défini par le développeur ; votre renderer instancie des composants pré-construits et dignes de confiance à partir de cette spec. Le modèle décide *quoi* afficher ; vous décidez *comment* le rendre. Exemples : json-render, A2UI.

Au-delà de ces deux-là, des démos très récentes suggèrent un troisième mode où le modèle ne choisit pas des composants ni n'écrit du HTML bac à sable — il pilote le canvas plus directement. Des projets comme [Tencent's HunyuanWorld](https://arxiv.org/abs/2502.01999), qui génère des environnements 3D explorables à partir d'une seule image, et des architectures de jeu où les LLMs génèrent des cartes, des PNJ et des quêtes à l'exécution plutôt que d'appeler un catalogue de composants, suggèrent un avenir où le modèle ressemble davantage à un directeur de jeu qu'à un moteur de formulaires. L'inférence LLM dans le navigateur via WebGPU ([WebLLM](https://mlc.ai/web-llm/)) pousse la même frontière localement.

Ce territoire est véritablement excitant et véritablement précoce. Il n'existe pas encore de frameworks stables pour construire des produits de production. Je couvrirai cette approche dans un article dédié une fois que cela aura changé.

## Les protocoles : AG-UI et A2UI

AG-UI et A2UI sont les deux principales normes dans la couche protocole. Ils résolvent des problèmes différents et ne sont pas concurrents.

### AG-UI

**GitHub** : [ag-ui-protocol/ag-ui](https://github.com/ag-ui-protocol/ag-ui)

AG-UI est un protocole événementiel pour la communication entre agents IA et applications frontend. Il définit environ 16 types d'événements : `TEXT_MESSAGE_START`, `TEXT_MESSAGE_CONTENT`, `TOOL_CALL_START`, `TOOL_CALL_END`, `STATE_SNAPSHOT`, `STATE_DELTA`, etc. Le transport vous appartient — SSE, WebSockets, webhooks fonctionnent tous. Le format est délibérément souple pour permettre une large adoption.

AG-UI ne définit pas à quoi ressemble votre UI. Il définit comment l'agent communique *avec* votre frontend. Considérez-le comme la couche protocole qui permet à votre application React de s'abonner à un agent LangGraph de la même manière qu'elle s'abonne à un agent CrewAI, sans changer de code frontend.

CopilotKit a créé AG-UI à partir de leur travail avec LangGraph et CrewAI. Il a été adopté par LangChain, Mastra, PydanticAI et d'autres. Microsoft a publié un guide d'intégration AG-UI. Si vous construisez un frontend multi-agents et devez découpler les frameworks backend du code frontend, AG-UI est la réponse.

**Une clarification qui déroute les gens** : AG-UI n'est pas un framework UI. Il ne vous dit pas quoi rendre. Il vous dit *que* l'agent a dit quelque chose, a appelé un outil ou a mis à jour un état partagé. Ce que vous rendez en réponse est toujours votre décision.

### A2UI

**GitHub** : [google/A2UI](https://github.com/google/A2UI) · Spec : [a2ui.org](https://a2ui.org/)

A2UI est la spécification déclarative de Google pour ce que les agents envoient quand ils veulent afficher une UI. Là où AG-UI répond « comment l'agent communique-t-il ? », A2UI répond « quel format l'agent utilise-t-il pour décrire une mise en page de composants ? ».

A2UI utilise un format JSONL plat : un descripteur de composant par ligne, chacun avec un ID, un type et des données. Plat est intentionnel. Les arbres imbriqués nécessitent que le modèle connaisse la structure complète avant de pouvoir commencer à streamer. Une liste plate permet au modèle d'émettre chaque composant au fur et à mesure qu'il « réfléchit », ce qui signifie que votre frontend peut commencer à rendre la première carte métrique pendant que le modèle décide encore s'il doit ajouter un graphique.

```jsonl
{"id":"h1","type":"kpi_card","title":"MRR","value":"$82,400","delta":"+12%"}
{"id":"h2","type":"kpi_card","title":"Churn","value":"2.1%","delta":"-0.4%"}
{"id":"c1","type":"line_chart","title":"30-day MRR","data_ref":"mrr_series"}
{"id":"t1","type":"data_table","cols":["Month","MRR","Net New"],"data_ref":"monthly"}
```

A2UI est soucieux de la sécurité : la spec est un format de données, pas du code exécutable. Le catalogue de composants est prédéfini par le développeur ; l'agent ne peut référencer que les types de ce catalogue. Un renderer A2UI qui reçoit un nom de type inconnu l'ignore.

Le format « Open-JSON-UI » de CopilotKit est compatible avec A2UI. Si vous choisissez un format de spec pour un catalogue de composants aujourd'hui, A2UI est celui avec le soutien cross-platform le plus large.

**Une note sur la stabilité** : A2UI est pré-1.0 — v0.9 lors de la dernière vérification le 8 mai 2026 — et a subi des changements cassants entre versions mineures. Les communications de Google sur la roadmap ont été sporadiques et certains renderers (Lit, Flutter) ont pris du retard sur les mises à jour de spec. Prévoyez du temps pour la dérive de spec si vous construisez dessus aujourd'hui. Pour les cas d'usage web purs, json-render semble actuellement avoir des outils plus complets. L'avantage à long terme de A2UI est la portée cross-platform (web, Flutter, SwiftUI, Android) que json-render n'a pas.

### MCP Apps

**GitHub** : [modelcontextprotocol](https://github.com/modelcontextprotocol) · Lié : [mcp-ui](https://github.com/MCP-UI-Org/mcp-ui)

MCP a commencé comme protocole pour connecter les LLMs aux outils et données. L'extension Apps permet aux outils MCP de retourner non seulement des données, mais des artefacts UI interactifs : composants React, formulaires, tableaux de bord, cartes.

Le modèle de sécurité est strict et c'est bien là le but : tout rend dans une iframe bac à sable avec des permissions retirées, les templates sont déclarés à l'avance pour que l'application hôte puisse les examiner, et toute communication est de la JSON-RPC auditable. C'est le bon modèle pour les fournisseurs d'outils — un serveur MCP Shopify peut retourner un widget de paiement ; un service de cartographie peut retourner une carte intégrable. L'application hôte ne possède ni ne fait confiance au code de ce widget.

MCP Apps est le bon choix quand l'UI *appartient au fournisseur d'outils*, pas à votre application. Pour l'UI qui vit dans le domaine de votre application, restez au Pattern 1 ou 2.

## Les frameworks JavaScript/TypeScript

### CopilotKit

**GitHub** : [CopilotKit/CopilotKit](https://github.com/CopilotKit/CopilotKit) · Exemples : [CopilotKit/generative-ui](https://github.com/CopilotKit/generative-ui)

CopilotKit est le framework le plus complet pour les applications frontend natives aux agents. Il gère le cycle de vie complet : connexion aux backends d'agents via AG-UI, gestion de l'état de conversation bidirectionnel, rendu de composants generative UI, et 提供 shared state plumbing qui permet aux agents et aux utilisateurs de modifier les mêmes données.

Le modèle à trois patterns se mappe proprement aux APIs de CopilotKit :
- `useCopilotAction` avec un callback `render` → Pattern 1
- Rendu A2UI/Open-JSON-UI → Pattern 2
- Artefacts bac à sable `OpenGenerativeUI` → Pattern 3

La fonctionnalité importante de CopilotKit qui est sous-discutée est **l'état partagé et l'humain dans la boucle** : l'agent peut lire et écrire l'état de l'application, l'utilisateur peut lire et écrire dedans, et les changements circulent bidirectionnellement. C'est ce qui rend les copilot UIs véritablement collaboratifs plutôt qu'une boîte de chat agrafée à un produit.

### Vercel AI SDK

**GitHub** : [vercel/ai](https://github.com/vercel/ai) · Docs : [ai-sdk.dev](https://ai-sdk.dev/)

Le Vercel AI SDK est la référence TypeScript de facto pour les apps IA. Pour le generative UI spécifiquement :

**`useObject`** stream un objet JSON structuré depuis le serveur au fur et à mesure de sa génération. Vous définissez un schéma Zod ; le SDK parse le JSON partiel et déclenche des re-rendus au fur et à mesure de l'arrivée des champs. C'est le chemin le plus fluide vers le Pattern 2 dans une app Next.js.

```tsx
const { object: dashboard } = useObject({
  api: "/api/generate-dashboard",
  schema: z.object({
    title: z.string(),
    metrics: z.array(z.object({ label: z.string(), value: z.number() })),
    insights: z.array(z.string()),
  }),
});
```

**`useChat` avec gestionnaires d'outils** → Pattern 1. Le modèle appelle des outils ; vous mappez les noms d'outils aux composants.

**AI Elements** ([elements.ai-sdk.dev](https://elements.ai-sdk.dev/)) fournit des primitives UI prêtes à l'emploi à coupler avec le SDK.

**Une note sur la trajectoire confuse ici** : En octobre 2024, Vercel a annoncé dans la [Discussion GitHub #3251](https://github.com/vercel/ai/discussions/3251) que AI SDK RSC — le pattern React Server Components streaming promu comme la fonctionnalité headline « Generative UI » dans SDK 3.0 — était en pause indéfiniment en raison de « plusieurs limitations de longue date » avec les React Server Components sans bonne solution à court terme. Les équipes qui avaient construit une stratégie produit autour du streaming RSC ont été prises au dépourvu. Les APIs `generateObject`/`streamObject` ont également été dépréciées plus tard dans SDK 6.0. La migration recommandée depuis AI SDK RSC est le pattern `useObject` ci-dessus, ou json-render pour la génération basée sur un catalogue.

### assistant-ui

**GitHub** : [assistant-ui/assistant-ui](https://github.com/assistant-ui/assistant-ui)

assistant-ui est un ensemble de primitives React composables pour construire des interfaces de chat de qualité production. C'est la bonne réponse quand vous avez besoin d'une UX de chat soignée — bulles de message, tokens en streaming, actions copier/éditer/régénérer, états de réflexion — et voulez apporter votre propre backend et votre propre rendu d'outils.

Il fonctionne bien avec n'importe quel backend (OpenAI, Anthropic, modèles locaux, endpoints personnalisés) et gère le rendu d'appels d'outils via un modèle familier de slot/render prop.

### json-render

**GitHub** : [vercel-labs/json-render](https://github.com/vercel-labs/json-render) · Docs : [json-render.dev](https://json-render.dev/)

json-render opérationnalise le Pattern 2 avec une approche opiniâtre et complète. Vous obtenez un catalogue de composants pré-construit (composants shadcn/ui avec schémas Zod), un renderer et une boucle de génération serrée où le modèle est contraint au catalogue par le schéma.

Les caractéristiques distinctives :
- **Rendu multi-cibles** : la même spec JSON peut rendre vers une app web React, une app mobile React Native, un PDF, un email HTML ou une vidéo Remotion. C'est véritablement utile pour les rapports.
- **Rendu progressif** : les composants apparaissent au fur et à mesure que le modèle les stream, pas après l'arrivée de la spec complète.
- **Contraintes de schéma serrées** : le catalogue est conçu pour que le modèle ne puisse pas halluciner des types de composants valides mais inconnus.

Si vous construisez une fonctionnalité de génération de tableaux de bord ou de rapports et voulez éviter le travail d'infrastructure de conception de votre propre catalogue, json-render est le chemin le plus rapide pour les apps web.

**Sur l'élan** : json-render a été lancé par Vercel Labs début 2026 et semble avoir rapidement attiré l'attention des développeurs web parce qu'il est immédiatement utile dans des projets React/Next.js standard. Cela dit, json-render est encore pré-1.0 et la relation entre json-render et A2UI est encore en cours de clarification — Vercel a expérimenté avec une sortie compatible A2UI, donc une convergence est possible. Pour le cross-platform (mobile natif, multiples frameworks), A2UI est le meilleur pari à long terme.

### Hashbrown

**GitHub** : [liveloveapp/hashbrown](https://github.com/liveloveapp/hashbrown)

Hashbrown adopte une approche distinctive : au lieu de construire une couche d'interface IA séparée, il intègre la sélection de composants IA directement dans votre application React ou Angular existante. Vous exposez les composants de votre app au LLM ; le LLM sélectionne lesquels rendre et peut invoquer des outils côté client.

C'est le bon outil quand vous voulez infuser de l'intelligence dans des surfaces produit qui ne sont pas du « chat » — une page produit qui adapte sa mise en page, un panneau de paramètres qui expose les bonnes options, un éditeur de workflow qui suggère l'étape suivante.

### OpenUI

**GitHub** : [thesysdev/openui](https://github.com/thesysdev/openui) · Docs : [openui.com](https://www.openui.com/)

OpenUI remplace JSON par un format orienté ligne similaire à du code (« OpenUI Lang ») conçu pour le rendu progressif et l'efficacité en tokens. La revendication est environ 67% moins de tokens que du JSON équivalent pour des mises en page complexes.

Le compromis est la maturité de l'écosystème — OpenUI est plus récent et les outils sont plus minces que les approches basées sur JSON. Mais si le coût en tokens est une contrainte significative et que vous générez des mises en page complexes à haute fréquence, l'efficacité du format est réelle.

### Tambo

**GitHub** : [tambo-ai/tambo](https://github.com/tambo-ai/tambo)

Tambo se concentre sur la sélection de composants avec état : l'IA sélectionne des composants et peut interagir avec eux via des outils côté client, maintenant l'état des composants à travers la conversation. Bon pour les cas d'usage où les éléments UI persistent à travers les tours — un composant filtre que l'utilisateur ajuste pendant que l'IA continue de raisonner sur les données filtrées.

## La couche Python

L'écosystème Python aborde les interfaces IA différemment. Ces outils sont optimisés pour les démos de modèles ML, les applications de données et les outils internes — pas les apps consumer de production avec composition de mise en page pilotée par agent.

Ce n'est pas une critique. Pour les bons cas d'usage, Gradio et Streamlit sont les seuls outils dont vous avez besoin.

### Gradio

**GitHub** : [gradio-app/gradio](https://github.com/gradio-app/gradio) · PyPI : `gradio`

La valeur principale de Gradio : vous écrivez une fonction Python ; Gradio l'enveloppe dans une UI web. La classe `Interface` c'est 3 lignes pour un classifier d'images. `ChatInterface` c'est 10 lignes pour un chatbot. `Blocks` vous donne un contrôle de mise en page fin quand vous en avez besoin.

Le « generative UI » dans Gradio est défini par le développeur Python, pas par le modèle. La visibilité et la configuration des composants peuvent changer dynamiquement selon les sorties du modèle, mais le catalogue de composants est statique — vous ne demandez pas au modèle de composer des mises en page.

Gradio est la valeur par défaut pour les HuggingFace Spaces et l'écosystème de démos ML. Il a des millions de téléchargements mensuels et alimente une grande fraction du paysage des démos IA.

**Choisissez Gradio quand** : vous êtes un développeur Python construisant une démo de modèle ML, un prototype de recherche ou un outil interne, et ne voulez pas toucher au JavaScript.

### Streamlit

**GitHub** : [streamlit/streamlit](https://github.com/streamlit/streamlit)

Le modèle de Streamlit est plus opinâtre : un script Python s'exécute de bout en bout à chaque interaction. Vous appelez `st.chat_message()`, `st.dataframe()`, `st.plotly_chart()`. Le framework gère la mise en page.

Le modèle de réexécution de script complet semble inefficace mais est étonnamment ergonomique pour les chatbots IA qui accumulent l'historique de conversation — le script entier réexécute, l'historique de chat est dans l'état de session et la sortie est déterministe. Streamlit a maintenant un support de première partie pour la plupart des fournisseurs LLM majeurs et s'intègre nativement avec Snowflake Cortex.

**Choisissez Streamlit quand** : vous construisez une app de données alimentée par IA, un outil de rapport interne ou un tableau de bord ML-backed en Python et voulez le chemin de déploiement le plus simple possible.

### LangChain et Haystack

Ce sont des frameworks d'orchestration backend, pas des frameworks UI. Ils apparaissent dans toute carte honnête de pile generative UI parce qu'ils sont typiquement la couche où les sorties structurées sont générées avant d'être envoyées à un frontend.

**LangChain** ([langchain-ai/langchain](https://github.com/langchain-ai/langchain)) : `.with_structured_output()` sur n'importe quel LLM vous donne une génération JSON contrainte par Pydantic. Le décorateur `@tool` avec génération automatique de schéma est la façon la plus propre de définir quels outils le modèle peut appeler. LangChain transmet les résultats structurés au layer frontend que vous utilisez.

**Haystack** ([deepset-ai/haystack](https://github.com/deepset-ai/haystack)) : architecture de pipeline modulaire avec un support RAG solide. Hayhooks enveloppe les pipelines Haystack comme endpoints HTTP — y compris les endpoints compatibles MCP. Si votre generative UI a besoin d'une base de retrieval, l'architecture de pipeline de Haystack gère cela proprement.

Aucun framework ne possède la couche UI. Ils génèrent les données que votre frontend (Pattern 1, 2 ou 3) rend.

## Vélocité de l'écosystème et terrain instable

Cet espace bouge vite et plusieurs projets ont livré des communications confuses avec leur code. Vérifié pour la dernière fois le 8 mai 2026 ; traitez les notes sur le statut des projets ici comme une lecture horodatée, pas un verdict permanent.

**Vercel AI SDK RSC** était la fonctionnalité GenUI flagship quand SDK 3.0 a été lancé. Vercel a mis son développement en pause en octobre 2024 ([Discussion #3251](https://github.com/vercel/ai/discussions/3251)) citant des limitations architecturales avec les React Server Components sans solution à court terme. Les équipes qui avaient bâti dessus étaient compréhensiblement frustrées. C'est encore dans les docs mais ce n'est pas le chemin recommandé ; c'est `useObject`.

**json-render** (Vercel Labs) est la nouvelle direction — une alternative basée sur un catalogue, agnostique du framework qui évite les problèmes de couplage RSC. C'est pré-1.0 et semble avoir un fort intérêt précoce parmi les développeurs React/web. La raison DX probable : json-render est immédiatement utile dans un projet React/Next.js standard, tandis que la portée cross-platform de A2UI ajoute de la friction de setup. Comment cela se passera à mesure que les deux specs mûrissent est véritablement obscur. Vercel a sondé la compatibilité A2UI dans json-render, suggérant qu'une convergence est possible.

**A2UI** (Google) est pré-1.0 (v0.9 lors de la dernière vérification) avec des changements cassants entre versions mineures et des communications Google inconsistantes sur sa roadmap. C'est le bon choix pour la portée cross-platform (web + Flutter + SwiftUI) que json-render n'adresse pas, et il a un soutien d'entreprise significatif. Pour les projets web purs maintenant, la DX est plus rugueuse.

**AG-UI** (CopilotKit) est aussi pré-1.0. La confusion la plus courante : le nom le fait paraître comme un framework UI. Ce n'est pas — c'est un protocole de transport. AG-UI définit comment les événements circulent entre les backends d'agents et votre frontend ; ce que vous rendez en réponse est toujours votre décision. Ce modèle mental est solide et largement adopté, mais la spec pré-1.0 signifie que les cas limites sont encore en cours de résolution.

Le résultat pratique : **chaque acteur majeur ici est pré-1.0**. Prévoyez des changements d'API. Les patterns — outil-vers-composant, composition de catalogue, génération bac à sable — sont assez stables pour construire dessus. Les choix protocolaires spécifiques ne le sont pas.

## Conception du catalogue de composants : le vrai travail d'ingénierie

La majeure partie de la complexité intéressante dans le Pattern 2 n'est pas dans le renderer — c'est dans le catalogue.

Le catalogue est **une décision produit encodée en schéma**. Il répond : quels sont les objets UI significatifs dans ce domaine ? Pas « quels composants React existent ? » mais « qu'est-ce qu'un utilisateur dans ce contexte a réellement besoin de voir et avec quoi interagir ? »

**Le mode d'échec trop granulaire** : vous exposez `Row`, `Column`, `Text`, `Button`, `Icon`. Maintenant le modèle doit être un ingénieur frontend. Il générera une mise en page médiocre qui ne correspond pas à votre système de design, manquera les états vides, produira un balisage inaccessible et changera d'approche de réponse en réponse parce que rien dans le catalogue ne contraint la sortie au langage visuel de votre produit.

**Le mode d'échec trop grossier** : vous exposez `WeatherCard`, `FlightCard`, `HotelCard`. Le modèle ne peut pas s'adapter quand l'utilisateur demande quelque chose qui ne correspond pas à une carte pré-cuite. Il revient au texte.

**Le milieu utile** : composants de niveau domaine avec slots contraints.

Un catalogue d'app de voyage pourrait ressembler à :

```
TripSummary         — itinerary en un coup d'œil
FlightOptionList    — options de vol sélectionnables avec tarification
HotelComparison     — cartes d'hôtels en comparaison côte à côte
TravelerForm        — collecter les détails du voyageur
PolicyNotice        — encadré règles réglementaires/prix
BookingConfirmation — confirmation finale avec bouton d'action
```

Un catalogue d'app de finance pourrait ressembler à :

```
PortfolioSnapshot   — positions clés et P&L
TransactionTable   — transactions filtrables, paginées
RiskBreakdown       — allocation et métriques de volatilité
ScenarioComparison  — modélisation de scénarios côte à côte
ApprovalGate        — action nécessitant confirmation humaine
```

Le catalogue ressemble au vocabulaire de votre produit. Il encode vos décisions UX, vos exigences d'accessibilité, votre gestion des états vides et vos patterns d'action dangereuse dans du code de composant. Le modèle arrange ces pièces. Vous décidez toujours à quoi ressemble chaque pièce et ce qu'elle est autorisée à faire.

**Règles de conception de schéma qui réduisent les hallucinations** :

1. Gardez les valeurs enum courtes et évidentes. `"type": "bar_chart"` pas `"type": "data-visualization-bar-type-vertical"`.
2. Rendez la composition invalide impossible. Si un `PolicyNotice` ne peut apparaître qu'à la fin d'une mise en page, ne le mettez pas au même niveau de schéma que les choses qui peuvent apparaître n'importe où.
3. Utilisez les champs requis généreusement. Un champ optionnel est un champ que le modèle pourrait omettre et que votre renderer doit gérer comme null.
4. Testez le catalogue contre de vraies invites avant de shipper. Sauvegardez les specs générées ; vérifiez-les pour les violations de schéma, les valeurs de champ hallucinée et les compositions techniquement valides mais sémantiquement incorrectes.

## Pièges courants

**Piège : traiter du JSON valide comme un comportement sûr.** La validation de schéma confirme la structure. Elle ne dit rien sur le fait que l'action attachée à un bouton corresponde à son label, qu'un total corresponde aux données dont il est dérivé, ou qu'un composant UI fasse quelque chose que l'utilisateur n'attendait pas. Les specs UI générées ont besoin d'une revue sémantique, pas juste de validation de schéma. Au minimum, les actions destructives devraient nécessiter un composant de confirmation, et les labels sur ces composants devraient être testés contre les actions qu'ils déclenchent.

**Piège : exposer des primitives de design au lieu de primitives de produit.** Si le modèle doit décider s'il faut utiliser un padding de 16px ou 20px, vous lui avez donné le mauvais niveau d'abstraction. Les composants de domaine devraient encoder le goût produit. Le modèle devrait composer du comportement, pas gérer des détails de présentation.

**Piège : utiliser du generative UI là où du static UI suffira.** Si la structure de ce que vous voulez afficher est connaissable au moment du développement — et c'est généralement le cas — le Pattern 1 avec des composants pré-construits est plus rapide, plus sûr et plus cohérent. Le generative UI justifie sa complexité quand la structure varie réellement selon les données ou le contexte de la tâche.

**Piège : négliger l'accessibilité.** Les LLMs hallucinent des violations WCAG. Ils émettront `role="region"` sur des éléments interactifs, généreront des formulaires avec des labels manquants et produiront des ratios de contraste qui échouent WCAG AA. Votre bibliothèque de composants peut être entièrement accessible ; les compositions générées par IA de ces composants ne sont pas automatiquement accessibles. Testez le chemin de rendu complet, pas juste les composants isolément.

**Piège : confondre le protocole et le framework.** AG-UI n'est pas un framework frontend. A2UI n'est pas une bibliothèque React. Ce sont des formats de transport et des protocoles d'événements. Vous avez toujours besoin d'un framework frontend pour les implémenter. CopilotKit implémente AG-UI et A2UI. json-render implémente le pattern catalogue A2UI/Open-JSON-UI. Ce sont des couches différentes.

## Recommandations par cas d'usage

**Ajouter un copilot à une app SaaS existante** : Commencez par le Pattern 1 (outil-vers-composant). Utilisez Vercel AI SDK `useChat` ou CopilotKit. Mappez vos 5 à 10 principales actions d'agent à des composants pré-construits. Shippez cela, mesurez-le, puis élargissez le catalogue seulement si les utilisateurs démontrent avoir besoin d'une composition plus riche.

**Génération de tableaux de bord en langage naturel** : Utilisez le Pattern 2 avec json-render ou un catalogue A2UI personnalisé. Définissez un catalogue de 8 à 15 types de composants couvrant vos types de graphiques, cartes métriques et variantes de tableaux. Fournissez le schéma au modèle ; laissez-le composer la mise en page. Construisez une validation qui détecte les types inconnus avant qu'ils n'atteignent le renderer.

**Frontend multi-agents** : Utilisez CopilotKit avec AG-UI. Le flux d'événements gère le streaming en temps réel à travers les backends d'agents ; l'état partagé gère le transfert entre agents ; le pattern HITL gère les portails d'approbation.

**Construire dans ChatGPT ou un autre hôte MCP** : Utilisez MCP Apps. Définissez votre outil comme un outil de données qui récupère et raisonne, et un outil de rendu séparé qui demande un widget. Gardez la logique métier hors du template de widget.

**Démos de modèles ML et apps de données (équipe Python)** : Gradio pour les démos et HuggingFace Spaces. Streamlit pour les apps de données avec une interaction plus complexe. Aucun des deux ne requiert JavaScript.

**Artefacts visuels, simulations, diagrammes** : Utilisez le Pattern 3 (OpenGenerativeUI ou équivalent). Établissez une CSP iframe stricte. Traitez la sortie comme du contenu utilisateur non fiable d'un point de vue sécurité.

---

Les frameworks murissent vite. La convergence de protocole (AG-UI pour le streaming, A2UI/Open-JSON-UI pour les specs de catalogue) est encore en cours, mais la forme devient assez claire pour construire dessus.

Les défis d'ingénierie qui comptent le plus maintenant ne sont pas le choix de framework. C'est la conception du catalogue — décider ce que le modèle est autorisé à dire, ce qui requiert de la clarté produit plus que des compétences techniques. C'est la validation sémantique — tester que le UI généré fait ce qu'il prétend, pas juste qu'il passe la validation de schéma. Et c'est l'écart d'accessibilité — construire des catalogues où chaque composant, et chaque composition de composants, respecte la barre d'accessibilité que vous appliqueriez à de l'UI écrite à la main.

Le modèle fera ce que vous lui demandez dans la grammaire que vous lui donnez. Rendez la grammaire délibérée.
