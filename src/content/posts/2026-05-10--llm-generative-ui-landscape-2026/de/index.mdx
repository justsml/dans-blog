„Generative UI“ bedeutet mindestens fünf verschiedene Dinge, je nachdem, wer darüber spricht.

- Chat-Schnittstellen, die Produktkarten aus Tool-Aufrufen des Modells einbetten.
- Runtime-JSON-Spezifikationen, die das Frontend als Komponentenbaum rendert.
- Sandboxed-iFrames, die von MCP-Tools an Host-Apps zurückgegeben werden (von Ticketbestellung und Hotelbuchung bis hin zu Karten-Rendering und Checkout-Widgets).
- Ereignisprotokolle, die den Agentenstatus an das Frontend streamen.
- v0, Lovable und Bolt: KI-Tools, die React zur Designzeit schreiben.

Dies sind verwandte Konzepte, aber sie existieren auf unterschiedlichen Ebenen des Stacks und tragen unterschiedliche Risikoprofile, Implementierungskosten und geeignete Anwendungsfälle. Sie zu vermischen, macht jede Architekturdiskussion zu einem Durcheinander.

Dies ist die Karte, die ich haben möchte, wenn ich entscheide, wo ich im Stack ansetzen soll.

---

## Was Generative UI nicht ist

Bevor wir definieren, was es ist, sollten drei Dinge beiseite gelegt werden:

**Design-time Code-Generierung** — v0, Lovable, Bolt, Cursor, die React-Komponenten erstellen. Diese Tools generieren Code, den Entwickler überprüfen und committen. Die KI läuft zur Entwicklungszeit. Was ausgeliefert wird, ist aus Sicht des Benutzers statisch. Dies ist eine großartige Kategorie von Tools. Es ist nicht das, was „generative UI zur Laufzeit“ bedeutet.

**KI-gestütztes Formular-Autofill** — das Modell füllt Feldwerte aus dem Kontext aus. Die Struktur der Schnittstelle ist immer noch festgelegt; nur der Inhalt ändert sich. Dies ist ein nützliches Muster. Es ist keine generative UI.

**KI schreibt rohes HTML in eine Seite** — das Modell gibt `<div>`- und `<button>`-Strings aus, die über `innerHTML` oder `dangerouslySetInnerHTML` eingefügt werden. Dies *ist* generative UI zur Laufzeit im technischsten Sinne. Es ist auch die gefährlichste Version und diejenige, die jedes ausgereifte Framework in diesem Bereich zu vermeiden versucht. Roh-KI-generiertes Markup bedeutet XSS-Risiko, unzugängliche Attribute, inkonsistentes Styling und halluzinierte Struktur. Der Rest dieses Artikels handelt davon, wie man es besser macht.

## Eine Arbeitsdefinition

Generative UI zur Laufzeit bedeutet: **Das Modell bestimmt, welche Schnittstellenkomponente oder Komponentenzusammensetzung der Benutzer sieht, basierend auf dem Zustand der Konversation oder der Aufgabe.**

Nicht die Worte. Die Schnittstelle.

Der einfachste Fall: Ihr Flugbuchungsassistent ruft ein `search_flights`-Tool auf. Anstatt reinen Text zurückzugeben („Hier sind drei Optionen...“), rendert es eine `<FlightResultsCard>`-Komponente mit auswählbaren Flügen, Sitzklassen-Umschaltern und einem „Buchen“-Button. Das Modell entschied, dass eine strukturierte Karte die richtige Antwort hierfür ist. Der Entwickler entschied, wie diese Karte aussieht und was „Buchen“ bewirkt.

Der komplexere Fall: Ein Finanzanalyse-Agent erhält eine Frage zu einem Portfolio und entscheidet sich, eine Antwort mit einer `MetricGroup` für wichtige Zahlen, einem `RiskBreakdown`-Diagramm, einer `ScenarioComparison`-Tabelle und einem `PolicyNotice` zusammenzustellen. Das Modell hat dieses Layout aus einem Katalog vordefinierter Komponenten zusammengestellt. Der Entwickler hat jede Komponente definiert. Das Modell hat entschieden, welche verwendet werden und welche Daten hinein sollen.

Beide Fälle sind generative UI. Sie unterscheiden sich darin, wie viel Freiheit das Modell bei der Zusammensetzung hat, was sowohl die Reichhaltigkeit der möglichen Ausgaben als auch die Komplexität dessen bestimmt, was schiefgehen kann.

## Die drei Muster

Der gesamte Bereich lässt sich in drei Muster zusammenfassen, jedes mit einer anderen Ausgabegrammatik.

![Ein Spektrumdiagramm mit drei Mustern: Nur Tool-Aufrufe links (am sichersten), Komponenten-Katalog in der Mitte und offene Generierung rechts (am ausdrucksstärksten).](../output-grammar-spectrum.svg)

_Jede generative UI-Entscheidung ist ein Punkt auf diesem Spektrum. Beginnen Sie links._

### Muster 1: Tool-zu-Komponenten-Rendering

Das Modell ruft ein benanntes Tool auf. Ihre Anwendung verfügt über eine Zuordnung von Tool-Namen zu Komponenten. Der Tool-Aufruf löst ein Komponenten-Rendering aus.

```tsx
// Das Modell ruft auf: { name: "show_flight_results", args: { flights: [...] } }

useCopilotAction({
  name: "show_flight_results",
  render: ({ args }) => <FlightResultsCard flights={args.flights} />,
});
```

Dies ist das sicherste Muster, da das Layout niemals vom Modell stammt. Das Modell entscheidet, *wann* eine Komponente angezeigt werden soll und *mit welchen Daten* sie gefüllt werden soll. Ihre Entwickler besitzen weiterhin den Komponenten-Code, das visuelle Design, die Barrierefreiheits-Implementierung und jeden Grenzfall in der Rendering-Logik.

Vercel AI SDKs `useChat` mit `tool`-Handlern macht dies. assistant-uis Tool-Rendering macht dies. CopilotKits „Static Generative UI“ ist dieses Muster. Die meisten Copilot-UIs in der Produktion, die zuverlässig funktionieren, verwenden dieses Muster.

**Geeignet, wenn**: die Menge der Dinge, die Sie anzeigen möchten, zum Zeitpunkt der Entwicklung bekannt ist. Buchungsbestätigungen, Suchergebnisse, Kontozusammenfassungen, Genehmigungs-Widgets. Wenn Sie die Szenarien auflisten können, deckt dieses Muster sie ab.

### Muster 2: Komponenten-Katalog-Zusammensetzung

Das Modell emittiert einen typisierten JSON-Baum, der auf Komponenten aus einem vom Entwickler definierten Katalog verweist. Ihr Frontend verfügt über einen Renderer, der den Baum durchläuft und jede Komponente instanziiert.

```json
[
  { "type": "metric_group", "metrics": [
    { "label": "MRR", "value": "$82.400", "delta": "+12%" },
    { "label": "Churn", "value": "2,1%", "delta": "-0,4%" }
  ]},
  { "type": "line_chart", "title": "30-Tage-Wachstum", "data_ref": "mrr_series" },
  { "type": "insight_callout", "text": "Expansionsumsatz treibt das Delta – durchschnittliche Sitzplatzanzahl um 18% gestiegen." }
]
```

Das Modell hat dieses Layout zusammengestellt. Eine `MetricGroup`, ein `LineChart`, ein `InsightCallout`. Aber Sie haben definiert, was jeder Komponententyp bedeutet, welche Props er akzeptiert und wie er rendert. Wenn das Modell versucht, `{ "type": "custom_untested_thing" }` zu emittieren, fängt Ihre Schema-Validierung dies ab und der Renderer ignoriert oder lehnt es ab.

Dies ist das Muster hinter `json-render`, `A2UI`, `Hashbrown`, `OpenUI` und `Tambo`. Die zentrale Ingenieursarbeit ist das **Katalog-Design** – zu entscheiden, welche Komponententypen existieren, wie deren Schemas aussehen und was das Modell komponieren darf und was nicht.

**Geeignet, wenn**: die Struktur dessen, was Sie zeigen möchten, legitimerweise auf der Grundlage der Daten oder der Anfrage des Benutzers variiert. Dashboards, die sich an das anpassen, was in den Zahlen bemerkenswert ist. Berichte, die je nach Kontext unterschiedliche Abschnitte anzeigen. Workflow-Panels, die sich je nach Schritt ändern, in dem sich ein Agent befindet.

### Muster 3: Offene Generierung

Das Modell schreibt HTML, SVG, Canvas oder WebGL, das in einem Sandboxed-iFrame mit einer strikten Content Security Policy gerendert wird.

Dies ist angemessen für Dinge, für die kein fester Komponentenkatalog ausreicht: Algorithmus-Visualisierungen, Architekturdiagramme, Ad-hoc-Charts, generative Kunst, pädagogische Simulationen. Die iFrame-Grenze übernimmt hier die Sicherheitsarbeit; entfernen Sie sie, und Sie sind zurück beim Roh-HTML-Injection-Problem vom Anfang dieses Artikels.

`CopilotKit/OpenGenerativeUI` ist die beste aktuelle Referenzimplementierung dieses Musters. Die Sandbox entfernt Skripte, begrenzt die Nachrichtenübermittlung und hält das generierte Artefakt von dem privilegierten Status Ihrer Anwendung fern.

**Geeignet, wenn**: Sie wirklich beliebige visuelle Ausgaben benötigen — einmalige erklärende Diagramme, dynamische Simulationen, kreative Artefakte. Verwenden Sie dies nicht für transaktionale UI. Eine Checkout-Bestätigung benötigt keinen Sandboxed-iFrame.

### Jenseits der drei Muster: LLMs steuern Pixel direkt

Es gibt eine aufkommende vierte Richtung, die nicht sauber in eines dieser Muster passt: LLMs steuern **immersive, spielartige Erlebnisse**, indem sie die visuelle Ausgabe direkter steuern als ein Sandboxed-iFrame.

Die kanonische Unterscheidung innerhalb der generativen UI ist **iFrame-HTML vs. JSON-Katalog**:

- **iFrame-HTML** — das Modell schreibt HTML, SVG, Canvas oder WebGL, das in einer isolierten Sandbox rendert. Maximale expressive Freiheit; Sicherheit hängt vollständig von der iFrame-Grenze ab. Beispiele: Anthropic Artifacts, OpenGenerativeUI.
- **JSON-Katalog** — das Modell emittiert ein strukturiertes Payload, das auf einen vom Entwickler definierten Komponentenkatalog beschränkt ist; Ihr Renderer instanziiert vertrauenswürdige, vorgefertigte Komponenten aus dieser Spezifikation. Das Modell entscheidet, *was* angezeigt werden soll; Sie entscheiden, *wie* es rendert. Beispiele: json-render, A2UI.

Darüber hinaus deuten sehr aktuelle Demos auf einen dritten Modus hin, bei dem das Modell nicht Komponenten auswählt oder Sandboxed-HTML schreibt — es steuert die Canvas direkter. Projekte wie [Tencents HunyuanWorld](https://arxiv.org/abs/2502.01999), das explorable 3D-Umgebungen aus einem einzigen Bild generiert, und Spielarchitekturen, bei denen LLMs Karten, NPCs und Quests zur Laufzeit generieren, statt einen Komponentenkatalog aufzurufen, deuten auf eine Zukunft hin, in der das Modell eher ein Spielregisseur als ein Formular-Renderer ist. In-Browser LLM-Inferenz via WebGPU ([WebLLM](https://mlc.ai/web-llm/)) treibt dieselbe Grenze lokal voran.

Dieses Gebiet ist wirklich aufregend und wirklich früh. Es gibt noch keine stabilen Frameworks, um damit Produktionsprodukte zu bauen. Ich werde diesen Ansatz in einem dedizierten Artikel behandeln, sobald sich das ändert.

## Die Protokolle: AG-UI und A2UI

AG-UI und A2UI sind die beiden Hauptstandards auf der Protokollebene. Sie lösen unterschiedliche Probleme und sind keine Konkurrenten.

### AG-UI

**GitHub**: [ag-ui-protocol/ag-ui](https://github.com/ag-ui-protocol/ag-ui)

AG-UI ist ein ereignisbasiertes Protokoll für die Kommunikation zwischen KI-Agenten und Frontend-Anwendungen. Es definiert etwa 16 Ereignistypen: `TEXT_MESSAGE_START`, `TEXT_MESSAGE_CONTENT`, `TOOL_CALL_START`, `TOOL_CALL_END`, `STATE_SNAPSHOT`, `STATE_DELTA` und so weiter. Der Transport bleibt Ihnen überlassen — SSE, WebSockets, Webhooks funktionieren alle. Das Format ist absichtlich locker, um eine breite Akzeptanz zu ermöglichen.

AG-UI definiert nicht, wie Ihre UI aussieht. Es definiert, wie der Agent *mit* Ihrem Frontend kommuniziert. Betrachten Sie es als die Wire-Protokoll-Ebene, die es Ihrer React-App ermöglicht, einen LangGraph-Agenten genauso zu abonnieren wie einen CrewAI-Agenten, ohne den Frontend-Code zu ändern.

CopilotKit hat AG-UI aus ihrer Arbeit mit LangGraph und CrewAI entwickelt. Es wurde von LangChain, Mastra, PydanticAI und anderen übernommen. Microsoft hat einen AG-UI-Integrationsleitfaden veröffentlicht. Wenn Sie ein Frontend mit mehreren Agenten aufbauen und Backend-Frameworks vom Frontend-Code entkoppeln müssen, ist AG-UI die Antwort.

**Eine Klarstellung, über die Leute stolpern**: AG-UI ist kein UI-Framework. Es sagt Ihnen nicht, was zu rendern ist. Es sagt Ihnen, *dass* der Agent etwas gesagt, ein Tool aufgerufen oder den gemeinsamen Status aktualisiert hat. Was Sie als Antwort rendern, bleibt Ihre Entscheidung.

### A2UI

**GitHub**: [google/A2UI](https://github.com/google/A2UI) · Spezifikation: [a2ui.org](https://a2ui.org/)

A2UI ist Googles deklarative Spezifikation dafür, was Agenten senden, wenn sie UI anzeigen möchten. Während AG-UI die Frage „Wie kommuniziert der Agent?“ beantwortet, beantwortet A2UI die Frage „Welches Format verwendet der Agent, um ein Komponentenlayout zu beschreiben?“.

A2UI verwendet ein flaches JSONL-Format: ein Komponentendeskriptor pro Zeile, jeweils mit ID, Typ und Daten. Flach ist beabsichtigt. Verschachtelte Bäume erfordern, dass das Modell die vollständige Struktur kennt, bevor es mit dem Streamen beginnen kann. Eine flache Liste ermöglicht es dem Modell, jede Komponente zu emittieren, während es darüber „nachdenkt“, was bedeutet, dass Ihr Frontend mit dem Rendern der ersten KPI-Karte beginnen kann, während das Modell noch entscheidet, ob es ein Diagramm hinzufügt.

```jsonl
{"id":"h1","type":"kpi_card","title":"MRR","value":"$82.400","delta":"+12%"}
{"id":"h2","type":"kpi_card","title":"Churn","value":"2,1%","delta":"-0,4%"}
{"id":"c1","type":"line_chart","title":"30-Tage-MRR","data_ref":"mrr_series"}
{"id":"t1","type":"data_table","cols":["Monat","MRR","Netto-Neu"],"data_ref":"monthly"}
```

A2UI ist sicherheitsbewusst: Die Spezifikation ist ein Datenformat, kein ausführbarer Code. Der Komponentenkatalog wird vom Entwickler vordefiniert; der Agent kann nur auf Typen in diesem Katalog verweisen. Ein A2UI-Renderer, der einen unbekannten Typnamen erhält, ignoriert ihn.

CopilotKits „Open-JSON-UI“-Format ist kompatibel mit A2UI. Wenn Sie heute ein Spezifikationsformat für einen Komponentenkatalog auswählen, ist A2UI dasjenige mit der breitesten plattformübergreifenden Unterstützung.

**Ein Hinweis zur Stabilität**: A2UI ist vor-1.0 — v0.9 beim letzten Check am 8. Mai 2026 — und hat seitdem Änderungen an der Spezifikation vorgenommen, die nicht abwärtskompatibel waren. Die Kommunikation von Google bezüglich der Roadmap war sporadisch und einige Renderer (Lit, Flutter) hinken hinter den Spezifikationsaktualisierungen her. Planen Sie Zeit für Spezifikationsdrifts ein, wenn Sie heute darauf aufbauen. Für reine Web-Anwendungsfälle scheint json-render derzeit über vollständigere Toolings zu verfügen. A2UIs langfristiger Vorteil ist die plattformübergreifende Reichweite (Web, Flutter, SwiftUI, Android), die json-render nicht hat.

### MCP Apps

**GitHub**: [modelcontextprotocol](https://github.com/modelcontextprotocol) · Verwandt: [mcp-ui](https://github.com/MCP-UI-Org/mcp-ui)

MCP begann als Protokoll zur Verbindung von LLMs mit Tools und Daten. Die Apps-Erweiterung ermöglicht es MCP-Tools, nicht nur Daten, sondern auch interaktive UI-Artefakte zurückzugeben: React-Komponenten, Formulare, Dashboards, Karten.

Das Sicherheitsmodell ist streng und das ist der Punkt: Alles rendert in einer Sandboxed-iFrame mit eingeschränkten Berechtigungen, Vorlagen sind vorab deklariert, damit die Host-App sie überprüfen kann, und die gesamte Kommunikation erfolgt über auditable JSON-RPC. Dies ist das richtige Modell für Tool-Anbieter — ein Shopify-MCP-Server kann ein Checkout-Widget zurückgeben; ein Kartendienst kann eine einbettbare Karte zurückgeben. Die Host-App besitzt oder vertraut dem Code dieses Widgets nicht.

MCP Apps ist die richtige Wahl, wenn die UI *dem Tool-Anbieter* gehört und nicht Ihrer Anwendung. Für UI, die in der Domäne Ihrer Anwendung lebt, bleiben Sie bei Muster 1 oder 2.

## Die JavaScript/TypeScript Frameworks

### CopilotKit

**GitHub**: [CopilotKit/CopilotKit](https://github.com/CopilotKit/CopilotKit) · Beispiele: [CopilotKit/generative-ui](https://github.com/CopilotKit/generative-ui)

CopilotKit ist das vollständigste Framework für Agent-native Frontend-Anwendungen. Es übernimmt den gesamten Lebenszyklus: Verbindung zu Agenten-Backends über AG-UI, Verwaltung des bidirektionalen Konversationsstatus, Rendering generativer UI-Komponenten und Bereitstellung der gemeinsamen Status-Plumbing, die es Agenten und Benutzern ermöglicht, dieselben Daten zu bearbeiten.

Das Drei-Muster-Modell lässt sich sauber auf die APIs von CopilotKit abbilden:
- `useCopilotAction` mit einem `render`-Callback → Muster 1
- A2UI/Open-JSON-UI-Rendering → Muster 2
- `OpenGenerativeUI` Sandboxed-Artefakte → Muster 3

Das wichtige CopilotKit-Feature, das zu wenig diskutiert wird, ist **Shared State und Human-in-the-Loop**: Der Agent kann den Anwendungsstatus lesen und schreiben, der Benutzer kann ihn lesen und schreiben, und Änderungen fließen bidirektional. Das ist es, was Copilot-artige UIs wie eine echte Zusammenarbeit anfühlen lässt, anstatt wie eine Chatbox, die an ein Produkt geheftet wurde.

### Vercel AI SDK

**GitHub**: [vercel/ai](https://github.com/vercel/ai) · Dokumentation: [ai-sdk.dev](https://ai-sdk.dev/)

Das Vercel AI SDK ist die De-facto-TypeScript-Basis für KI-Apps. Speziell für generative UI:

**`useObject`** streamt ein strukturiertes JSON-Objekt vom Server, während es generiert wird. Sie definieren ein Zod-Schema; das SDK analysiert das teilweise JSON und löst Re-Renders aus, während Felder eintreffen. Dies ist der reibungsloseste Pfad zu Muster 2 in einer Next.js-App.

```tsx
const { object: dashboard } = useObject({
  api: "/api/generate-dashboard",
  schema: z.object({
    title: z.string(),
    metrics: z.array(z.object({ label: z.string(), value: z.number() })),
    insights: z.array(z.string()),
  }),
});
```

**`useChat` mit Tool-Handlern** → Muster 1. Das Modell ruft Tools auf; Sie ordnen Tool-Namen Komponenten zu.

**AI Elements** ([elements.ai-sdk.dev](https://elements.ai-sdk.dev/)) bietet vorgefertigte UI-Primitive zur Kopplung mit dem SDK.

**Ein Hinweis zur verwirrenden Entwicklung hier**: Im Oktober 2024 kündigte Vercel in [GitHub Discussion #3251](https://github.com/vercel/ai/discussions/3251) an, dass AI SDK RSC — das React Server Components Streaming-Muster, das als das generative UI-Feature in SDK 3.0 beworben wurde — auf unbestimmte Zeit pausiert wurde, aufgrund „mehrerer langjähriger Einschränkungen“ ohne gute kurzfristige Lösungen. Teams, die Produktstrategien rund um RSC-Streaming aufgebaut hatten, wurden kalt erwischt. Die `generateObject`/`streamObject`-APIs wurden später in SDK 6.0 ebenfalls als veraltet markiert. Die empfohlene Migration vom AI SDK RSC ist das `useObject`-Muster oben oder json-render für katalogbasierte Generierung.

### assistant-ui

**GitHub**: [assistant-ui/assistant-ui](https://github.com/assistant-ui/assistant-ui)

assistant-ui ist eine Reihe von zusammensetzbaren React-Primitiven für den Aufbau von Chat-Schnittstellen in Produktionsqualität. Es ist die richtige Antwort, wenn Sie eine polierte Chat-UX benötigen — Nachrichtenblasen, Streaming-Token, Kopieren/Bearbeiten/Regenerieren-Aktionen, Denkzustände — und Ihr eigenes Backend und Ihr eigenes Tool-Rendering mitbringen möchten.

Es funktioniert gut gepaart mit jedem Backend (OpenAI, Anthropic, lokale Modelle, benutzerdefinierte Endpunkte) und handhabt das Tool-Call-Rendering über ein vertrautes Slot/Render-Prop-Modell.

### json-render

**GitHub**: [vercel-labs/json-render](https://github.com/vercel-labs/json-render) · Dokumentation: [json-render.dev](https://json-render.dev/)

json-render operationalisiert Muster 2 mit einem meinungsstarken, batteriebetriebenen Ansatz. Sie erhalten einen vorgefertigten Komponentenkatalog (shadcn/ui-Komponenten mit Zod-Schemas), einen Renderer und eine enge Generierungsschleife, in der das Modell per Schema auf den Katalog beschränkt ist.

Die herausragenden Merkmale:
- **Multi-Target-Rendering**: Die gleiche JSON-Spezifikation kann in eine React-Web-App, eine React Native-Mobile-App, ein PDF, eine HTML-E-Mail oder ein Remotion-Video rendern. Dies ist für Berichte wirklich nützlich.
- **Progressives Rendering**: Komponenten erscheinen, während das Modell sie streamt, nicht erst, nachdem die vollständige Spezifikation eingetroffen ist.
- **Strenge Schema-Beschränkungen**: Der Katalog ist so konzipiert, dass das Modell keine gültigen, aber unbekannten Komponententypen halluzinieren kann.

Wenn Sie eine Dashboard- oder Bericht-Generierungsfunktion aufbauen und die Infrastrukturarbeit für das Design Ihres eigenen Katalogs überspringen möchten, json-render ist der schnellste Weg für Web-Apps.

**Zur Dynamik**: json-render wurde Anfang 2026 von Vercel Labs gestartet und scheint schnell Aufmerksamkeit bei Webentwicklern gewonnen zu haben, da es sofort in Standard-React/Next.js-Projekten nützlich ist. Allerdings ist json-render noch vor-1.0 und die Beziehung zwischen json-render und A2UI ist noch in Arbeit — Vercel hat mit A2UI-kompatibler Ausgabe experimentiert, also ist eine Konvergenz möglich. Für plattformübergreifende Zwecke (natives Mobile, mehrere Frameworks) ist A2UI die bessere langfristige Wette.

### Hashbrown

**GitHub**: [liveloveapp/hashbrown](https://github.com/liveloveapp/hashbrown)

Hashbrown verfolgt einen unverwechselbaren Ansatz: Anstatt eine separate KI-Schnittstellenebene aufzubauen, bettet es KI-Komponentenauswahl direkt in Ihre bestehende React- oder Angular-App ein. Sie machen Ihre App-Komponenten für das LLM zugänglich; das LLM wählt aus, welche gerendert werden sollen, und kann clientseitige Tools aufrufen.

Dies ist das richtige Tool, wenn Sie Intelligenz in Produktoberflächen einfließen lassen möchten, die keine „Chats“ sind — eine Produktseite, die ihr Layout anpasst, ein Einstellungsfeld, das die richtigen Optionen anzeigt, ein Workflow-Editor, der den nächsten Schritt vorschlägt.

### OpenUI

**GitHub**: [thesysdev/openui](https://github.com/thesysdev/openui) · Dokumentation: [openui.com](https://www.openui.com/)

OpenUI ersetzt JSON durch ein zeilenorientiertes, code-ähnliches Format („OpenUI Lang“), das für progressives Rendering und Token-Effizienz konzipiert ist. Die Behauptung ist etwa 67 % weniger Token als bei äquivalentem JSON für komplexe Layouts.

Der Kompromiss ist die Ökosystemreife — OpenUI ist neuer und das Tooling ist dünner als bei JSON-basierten Ansätzen. Aber wenn Token-Kosten eine sinnvolle Einschränkung sind und Sie komplexe Layouts mit hoher Frequenz generieren, ist die Formateffizienz real.

### Tambo

**GitHub**: [tambo-ai/tambo](https://github.com/tambo-ai/tambo)

Tambo konzentriert sich auf die zustandsbasierte Komponentenauswahl: Die KI wählt Komponenten aus und kann über clientseitige Tools mit ihnen interagieren, wobei der Komponentenstatus über die gesamte Konversation hinweg erhalten bleibt. Gut für Anwendungsfälle, in denen UI-Elemente über Turns hinweg bestehen bleiben — eine Filterkomponente, die der Benutzer anpasst, während die KI weiterhin über die gefilterten Daten nachdenkt.

## Die Python-Ebene

Das Python-Ökosystem nähert sich KI-Schnittstellen anders. Diese Tools sind für ML-Modell-Demos, Datenanwendungen und interne Tools optimiert — nicht für Produktions-Consumer-Apps mit Agent-gesteuerter Layout-Zusammensetzung.

Das ist keine Kritik. Für die richtigen Anwendungsfälle sind Gradio und Streamlit die einzigen Tools, die Sie benötigen.

### Gradio

**GitHub**: [gradio-app/gradio](https://github.com/gradio-app/gradio) · PyPI: `gradio`

Gradios Kernwert: Sie schreiben eine Python-Funktion; Gradio packt sie in eine Web-UI. Die `Interface`-Klasse umfasst 3 Zeilen für einen Bildklassifikator. `ChatInterface` benötigt 10 Zeilen für einen Chatbot. `Blocks` gibt Ihnen bei Bedarf eine fein abgestimmte Layout-Kontrolle.

Die „generative UI“ in Gradio wird vom Python-Entwickler definiert, nicht vom Modell. Die Sichtbarkeit und Konfiguration von Komponenten können sich dynamisch auf der Grundlage von Modellausgaben ändern, aber der Komponentenkatalog ist statisch — Sie fordern das Modell nicht auf, Layouts zusammenzustellen.

Gradio ist der Standard für HuggingFace Spaces und das ML-Demo-Ökosystem. Es hat monatlich Millionen von Downloads und betreibt einen großen Teil der KI-Demo-Landschaft.

**Greifen Sie zu Gradio, wenn**: Sie ein Python-Entwickler sind, der ein ML-Modell-Demo, einen Forschungsprototyp oder ein internes Tool baut und kein JavaScript anfassen möchten.

### Streamlit

**GitHub**: [streamlit/streamlit](https://github.com/streamlit/streamlit)

Streamlits Modell ist meinungsstärker: Ein Python-Skript läuft bei jeder Interaktion von Anfang bis Ende durch. Sie rufen `st.chat_message()`, `st.dataframe()`, `st.plotly_chart()` auf. Das Framework kümmert sich um das Layout.

Das Modell des vollständigen Skript-Reruns klingt ineffizient, ist aber für KI-Chatbots, die Konversationsverlauf akkumulieren, überraschend ergonomisch — das gesamte Skript läuft erneut ab, der Chat-Verlauf befindet sich im Sitzungsstatus, und die Ausgabe ist deterministisch. Streamlit bietet jetzt erstklassige Unterstützung für die meisten großen LLM-Anbieter und lässt sich nativ in Snowflake Cortex integrieren.

**Greifen Sie zu Streamlit, wenn**: Sie eine KI-gestützte Daten-App, ein internes Bericht-Tool oder ein ML-gestütztes Dashboard in Python bauen und den einfachstmöglichen Bereitstellungspfad wünschen.

### LangChain und Haystack

Dies sind Backend-Orchestrierungs-Frameworks, keine UI-Frameworks. Sie erscheinen in jeder ehrlichen Karte des generativen UI-Stacks, weil sie typischerweise die Schicht sind, in der strukturierte Ausgaben generiert werden, bevor sie an ein Frontend gesendet werden.

**LangChain** ([langchain-ai/langchain](https://github.com/langchain-ai/langchain)): `.with_structured_output()` für jedes LLM liefert Ihnen Pydantic-eingeschränkte JSON-Generierung. Der `@tool`-Decorator mit automatischer Schema-Generierung ist der sauberste Weg, um zu definieren, welche Tools das Modell aufrufen kann. LangChain speist strukturierte Ergebnisse in die von Ihnen verwendete Frontend-Schicht ein.

**Haystack** ([deepset-ai/haystack](https://github.com/deepset-ai/haystack)): Modulare Pipeline-Architektur mit starker RAG-Unterstützung. Hayhooks verpackt Haystack-Pipelines als HTTP-Endpunkte — einschließlich MCP-kompatibler Endpunkte. Wenn Ihre generative UI ein Retrieval-Rückgrat benötigt, handhabt Haystacks Pipeline-Architektur dies sauber.

Keines der Frameworks besitzt die UI-Ebene. Sie generieren die Daten, die Ihr Frontend (Muster 1, 2 oder 3) rendert.

## Ökosystem-Geschwindigkeit und instabiler Boden

Dieser Bereich bewegt sich schnell und mehrere Projekte haben neben ihrem Code verwirrende Kommunikation versendet. Zuletzt überprüft am 8. Mai 2026; behandeln Sie die Projektstatus-Hinweise hier als zeitlich markierte Lektüre, nicht als permanentes Urteil.

**Vercel AI SDK RSC** war das Flaggschiff-Feature für generative UI, als SDK 3.0 startete. Vercel pausierte seine Entwicklung im Oktober 2024 ([Discussion #3251](https://github.com/vercel/ai/discussions/3251)) unter Berufung auf architektonische Einschränkungen bei React Server Components, für die es keine kurzfristige Lösung gab. Teams, die darauf aufgebaut hatten, waren verständlicherweise frustriert. Es ist immer noch in der Dokumentation, ist aber nicht der empfohlene Pfad; `useObject` ist es.

**json-render** (Vercel Labs) ist die neue Richtung — eine katalogbasierte, framework-agnostische Alternative, die die Probleme der RSC-Kopplung vermeidet. Es ist vor-1.0 und scheint unter React-/Web-Entwicklern starkes frühes Interesse geweckt zu haben. Der wahrscheinliche DX-Grund: json-render ist in einem Standard-React/Next.js-Projekt sofort nützlich, während A2UIs plattformübergreifender Umfang Setup-Reibung hinzufügt. Wie sich das entwickelt, während beide Spezifikationen reifen, ist wirklich unklar. Vercel hat A2UI-Kompatibilität in json-render untersucht, was auf eine mögliche Konvergenz hindeutet.

**A2UI** (Google) ist vor-1.0 (v0.9 beim letzten Check) mit Änderungen an der Spezifikation, die nicht abwärtskompatibel waren, und inkonsistenter Kommunikation von Google über ihre Roadmap. Es ist die richtige Wahl für plattformübergreifende Reichweite (Web + Flutter + SwiftUI), die json-render nicht adressiert, und es hat eine bedeutende Unternehmensunterstützung. Für reine Web-Projekte ist die DX derzeit rauer.

**AG-UI** (CopilotKit) ist ebenfalls vor-1.0. Die häufigste Verwirrung: Der Name lässt es wie ein UI-Framework klingen. Das ist es nicht — es ist ein Transportprotokoll. AG-UI definiert, wie Ereignisse zwischen Agenten-Backends und Ihrem Frontend fließen; was Sie als Antwort rendern, bleibt Ihre Entscheidung. Dieses mentale Modell ist solide und breit übernommen, aber die vor-1.0 Spezifikation bedeutet, dass Randfälle noch ausgearbeitet werden müssen.

Das praktische Ergebnis: **Jeder wichtige Akteur hier ist vor-1.0**. Planen Sie API-Änderungen ein. Die Muster — Tool-zu-Komponente, Katalog-Zusammensetzung, Sandboxed-Generierung — sind stabil genug, um darauf aufzubauen. Die spezifischen Protokollwahlen nicht.

## Komponenten-Katalog-Design: Die echte Ingenieursarbeit

Der Großteil der interessanten Komplexität bei Muster 2 liegt nicht im Renderer — er liegt im Katalog.

Der Katalog ist eine **Produktentscheidung, die als Schema codiert ist**. Er beantwortet: Was sind die bedeutungsvollen UI-Objekte in dieser Domäne? Nicht „welche React-Komponenten existieren?“, sondern „was muss ein Benutzer in diesem Kontext tatsächlich sehen und womit muss er interagieren?“

**Der Fehlermodus „zu granular“**: Sie legen `Row`, `Column`, `Text`, `Button`, `Icon` offen. Jetzt muss das Modell Frontend-Ingenieur spielen. Es wird mittelmäßiges Layout generieren, das nicht zu Ihrem Designsystem passt, Leerzustände übersehen, unzugängliches Markup produzieren und seinen Ansatz von Antwort zu Antwort ändern, weil nichts im Katalog die Ausgabe auf die visuelle Sprache Ihres Produkts beschränkt.

**Der Fehlermodus „zu grob“**: Sie legen `WeatherCard`, `FlightCard`, `HotelCard` offen. Das Modell kann sich nicht anpassen, wenn der Benutzer nach etwas fragt, das nicht zu einer vorgefertigten Karte passt. Es fällt auf Text zurück.

**Die nützliche Mitte**: Domänen-Ebenen-Komponenten mit eingeschränkten Slots.

Ein Katalog für eine Reise-App könnte so aussehen:

```
TripSummary         — Reiseroute auf einen Blick
FlightOptionList    — auswählbare Flugoptionen mit Preisangaben
HotelComparison     — Hotelkarten nebeneinander
TravelerForm        — Reise-Details erfassen
PolicyNotice        — behördlicher/tariflicher Hinweiskasten
BookingConfirmation — finale Bestätigung mit Aktionsbutton
```

Ein Katalog für eine Finanz-App könnte so aussehen:

```
PortfolioSnapshot   — Schlüsselpositionen und GuV
TransactionTable    — filterbare, paginierte Transaktionen
RiskBreakdown       — Allokations- und Volatilitätskennzahlen
ScenarioComparison  — Szenario-Modellierung nebeneinander
ApprovalGate        — Aktion, die menschliche Bestätigung erfordert
```

Der Katalog klingt wie der Wortschatz Ihres Produkts. Er codiert Ihre UX-Entscheidungen, Ihre Anforderungen an Barrierefreiheit, Ihre Behandlung von Leerzuständen und Ihre Muster für gefährliche Aktionen im Komponenten-Code. Das Modell darf diese Stücke arrangieren. Sie entscheiden immer noch, wie jedes Stück aussieht und was es tun darf.

**Regeln für Schema-Designs, die Halluzinationen reduzieren**:

1. Halten Sie Enum-Werte kurz und offensichtlich. `"type": "bar_chart"` nicht `"type": "data-visualization-bar-type-vertical"`.
2. Machen Sie ungültige Zusammensetzungen unmöglich. Wenn eine `PolicyNotice` nur am Ende eines Layouts erscheinen kann, legen Sie sie nicht auf dieselbe Schema-Ebene wie Dinge, die überall erscheinen können.
3. Verwenden Sie erforderliche Felder großzügig. Ein optionales Feld ist ein Feld, das das Modell weglassen könnte und das Ihr Renderer als null behandeln muss.
4. Testen Sie den Katalog mit echten Prompts, bevor Sie ihn veröffentlichen. Speichern Sie die generierten Spezifikationen; überprüfen Sie sie auf Schema-Verletzungen, halluzinierte Feldwerte und Zusammensetzungen, die technisch gültig, aber semantisch falsch sind.

## Häufige Fallen

**Falle: gültiges JSON als sicheres Verhalten behandeln.** Die Schema-Validierung bestätigt die Struktur. Sie sagt nichts darüber aus, ob die Aktion, die an einen Button angehängt ist, mit dessen Label übereinstimmt, ob eine Summe mit den Daten übereinstimmt, von denen sie abgeleitet ist, oder ob eine UI-Komponente etwas tut, das der Benutzer nicht erwartet hat. Generierte UI-Spezifikationen benötigen semantische Überprüfung, nicht nur Schema-Validierung. Zumindest sollten zerstörerische Aktionen eine Bestätigungskomponente erfordern, und die Labels auf diesen Komponenten sollten gegen die Aktionen getestet werden, die sie auslösen.

**Falle: Design-Primitive statt Produkt-Primitive offenlegen.** Wenn das Modell entscheiden muss, ob 16px oder 20px Padding verwendet werden sollen, haben Sie ihm das falsche Abstraktionsniveau gegeben. Domänenkomponenten sollten Produktgeschmack codieren. Das Modell sollte Verhalten zusammenstellen, nicht Präsentationsdetails verwalten.

**Falle: generative UI verwenden, wo statische UI ausreicht.** Wenn die Struktur dessen, was Sie zeigen möchten, zur Entwicklungszeit bekannt ist — und das ist sie meistens —, ist Muster 1 mit vorgefertigten Komponenten schneller, sicherer und konsistenter. Generative UI verdient ihre Komplexität, wenn die Struktur tatsächlich auf Basis der Daten oder des Aufgabenkontexts variiert.

**Falle: Barrierefreiheit überspringen.** LLMs halluzinieren WCAG-Verletzungen. Sie emittieren `role="region"` für interaktive Elemente, generieren Formulare mit fehlenden Labels und erzeugen Kontrastverhältnisse, die WCAG AA nicht bestehen. Ihre Komponentenbibliothek ist möglicherweise vollständig zugänglich; KI-generierte Zusammensetzungen dieser Komponenten sind nicht automatisch zugänglich. Testen Sie den vollständigen Render-Pfad, nicht nur die Komponenten isoliert.

**Falle: Protokoll und Framework vermischen.** AG-UI ist kein Frontend-Framework. A2UI ist keine React-Bibliothek. Sie sind Wire-Formate und Ereignisprotokolle. Sie benötigen immer noch ein Frontend-Framework, um sie zu implementieren. CopilotKit implementiert AG-UI und A2UI. json-render implementiert das A2UI/Open-JSON-UI-Katalogmuster. Dies sind unterschiedliche Ebenen.

## Empfehlungen nach Anwendungsfall

**Hinzufügen eines Copiloten zu einer bestehenden SaaS-App**: Starten Sie mit Muster 1 (Tool-zu-Komponente). Verwenden Sie Vercel AI SDK `useChat` oder CopilotKit. Ordnen Sie Ihre Top-5–10-Agentenaktionen vorgefertigten Komponenten zu. Liefern Sie das aus, messen Sie es und erweitern Sie den Katalog nur dann, wenn Benutzer nachweislich eine reichhaltigere Zusammensetzung benötigen.

**Dashboard-Generierung aus natürlicher Sprache**: Verwenden Sie Muster 2 mit json-render oder einem benutzerdefinierten A2UI-Katalog. Definieren Sie einen Katalog von 8–15 Komponententypen, die Ihre Diagrammtypen, Metrik-Karten und Tabellenvarianten abdecken. Füttern Sie das Schema an das Modell; lassen Sie es das Layout zusammenstellen. Bauen Sie Validierungen, die unbekannte Typen abfangen, bevor sie den Renderer erreichen.

**Frontend mit mehreren Agenten**: Verwenden Sie CopilotKit mit AG-UI. Der Ereignisstream handhabt Echtzeit-Streaming über Agenten-Backends; Shared State handhabt die Übergabe zwischen Agenten; das HITL-Muster handhabt Genehmigungsschleusen.

**Bauen innerhalb von ChatGPT oder einem anderen MCP-Host**: Verwenden Sie MCP Apps. Definieren Sie Ihr Tool als Daten-Tool, das abruft und schlussfolgert, und ein separates Render-Tool, das ein Widget anfordert. Halten Sie Geschäftslogik aus der Widget-Vorlage heraus.

**ML-Modell-Demos und Daten-Apps (Python-Team)**: Gradio für Demos und HuggingFace Spaces. Streamlit für Daten-Apps mit komplexerer Interaktion. Keines erfordert JavaScript.

**Visuelle Artefakte, Simulationen, Diagramme**: Verwenden Sie Muster 3 (OpenGenerativeUI oder Äquivalent). Etablieren Sie eine strikte iFrame-CSP. Behandeln Sie die Ausgabe wie nicht vertrauenswürdigen Benutzerinhalt aus einer Sicherheitsperspektive.

---

Die Frameworks reifen schnell. Die Protokollkonvergenz (AG-UI für Streaming, A2UI/Open-JSON-UI für Katalog-Spezifikationen) ist noch im Gange, aber die Form wird klar genug, um darauf aufzubauen.

Die wichtigsten technischen Herausforderungen sind derzeit nicht die Framework-Auswahl. Es ist Katalog-Design — zu entscheiden, was das Modell sagen darf, was eher Produktklarheit als technisches Geschick erfordert. Es ist semantische Validierung — zu testen, dass generierte UI tut, was sie behauptet, nicht nur, dass sie die Schema-Validierung besteht. Und es ist die Barrierefreiheitslücke — Kataloge zu bauen, in denen jede Komponente und jede Zusammensetzung von Komponenten den Barrierefreiheitsstandard erfüllt, den Sie für handgeschriebene UI ansetzen würden.

Das Modell wird tun, was Sie ihm innerhalb der Grammatik sagen, die Sie ihm geben. Machen Sie die Grammatik bewusst.
