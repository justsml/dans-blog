«Генеративный UI» означает как минимум пять разных вещей в зависимости от того, кто это говорит.

- Чат-интерфейсы, в которые встраиваются карточки товаров из вызовов инструментов модели
- JSON-спецификации рантайма, которые фронтенд рендерит как деревья компонентов
- Песочницы iframe, возвращаемые MCP-инструментами в хост-приложения (от заказа билетов и бронирования отелей до карт и виджетов оформления заказа)
- Протоколы событий, стримящие состояние агента во фронтенд
- v0, Lovable и Bolt: ИИ-инструменты, генерирующие React на этапе дизайна

Это связанные концепции, но они находятся на разных уровнях стека и несут разные риски, разные затраты на реализацию и разные подходящие сценарии использования. Смешивание их превращает любой архитектурный разговор в кашу.

Вот карта, которой мне хочется пользоваться, когда я решаю, на каком уровне стека действовать.

## Чем Generative UI не является

Прежде чем определить, что это такое, отложим в сторону три вещи:

**Генерация кода на этапе дизайна** — v0, Lovable, Bolt, Cursor генерирующие React-компоненты. Эти инструменты создают код, который разработчики проверяют и коммитят. ИИ работает на этапе разработки. Для пользователя то, что отгружается — статично. Это отличная категория инструментов. Это не то, что значит «генеративный UI в рантайме».

**ИИ-автозаполнение форм** — модель заполняет значения полей из контекста. Структура интерфейса фиксирована; меняется только содержимое. Полезный паттерн. Но это не генеративный UI.

**ИИ, пишущий сырой HTML прямо в страницу** — модель выводит строки `<div>` и `<button>`, которые попадают на страницу через `innerHTML` или `dangerouslySetInnerHTML`. Технически это *есть* генеративный UI в рантайме. Также это самая опасная версия, и каждый зрелый фреймворк в этой области существует именно для того, чтобы от неё уйти. Сырая ИИ-разметка — это XSS-риски, недоступные атрибуты, нестабильный стиль и галлюцинирующая структура. Вся остальная статья — о том, как сделать лучше.

## Рабочее определение

Генеративный UI в рантайме — это: **модель определяет, какой компонент или композицию компонентов видит пользователь, исходя из состояния диалога или задачи.**

Не текст. Интерфейс.

Простейший случай: ваш помощник по бронированию вызывает инструмент `search_flights`. Вместо простого текста («Вот три варианта…») он рендерит `<FlightResultsCard>` с выбираемыми рейсами, переключателями класса обслуживания и кнопкой «Забронировать». Модель решила, что структурированная карточка — подходящий ответ. Разработчик определил внешний вид карточки и поведение кнопки.

Более сложный случай: агент финансового анализа получает вопрос о портфеле и решает собрать ответ из `MetricGroup` с ключевыми показателями, `RiskBreakdown` с диаграммой рисков, `ScenarioComparison` с таблицей и `PolicyNotice`. Модель собрала эту раскладку из каталога предварительно одобренных компонентов. Разработчик определил каждый компонент. Модель выбрала, какие использовать и какие данные в них поместить.

Оба случая — генеративный UI. Различие — в степени свободы композиции, которая определяет как богатство возможных результатов, так и масштаб того, что может пойти не так.

## Три паттерна

Всё пространство сводится к трём паттернам, каждый со своей грамматикой вывода.

![Диаграмма-спектр с тремя паттернами: вызовы инструментов слева (самый безопасный), каталог компонентов в центре, свободная генерация справа (наиболее выразительная).](../output-grammar-spectrum.svg)

_Любое решение в генеративном UI — точка на этом спектре. Начинайте слева._

### Паттерн 1: Рендеринг по вызову инструмента

Модель вызывает именованный инструмент. Ваше приложение содержит маппинг имён инструментов на компоненты. Вызов инструмента триггерит рендер.

```tsx
// Модель вызывает: { name: "show_flight_results", args: { flights: [...] } }

useCopilotAction({
  name: "show_flight_results",
  render: ({ args }) => <FlightResultsCard flights={args.flights} />,
});
```

Это самый безопасный паттерн — раскладка никогда не приходит от модели. Модель решает *когда* показать компонент и *какими данными* его наполнить. Разработчики по-прежнему владеют кодом компонента, визуальным дизайном, реализацией доступности и всеми граничными кейсами рендеринга.

Vercel AI SDK `useChat` с обработчиками `tool` делает именно это. Рендеринг инструментов в assistant-ui делает это. «Static Generative UI» от CopilotKit — тот же паттерн. Большинство продакшен-копайлотов, которые работают надёжно, используют именно его.

**Подходит, когда**: набор вещей, которые может понадобиться показать, известен на этапе разработки. Подтверждения бронирования, результаты поиска, сводки аккаунтов, виджеты согласования. Если можно перечислить сценарии — этот паттерн покроет их.

### Паттерн 2: Композиция из каталога компонентов

Модель выдаёт типизированное JSON-дерево, ссылающееся на компоненты из определённого разработчиком каталога. Фронтенд имеет рендерер, обходящий дерево и создающий экземпляры компонентов.

```json
[
  { "type": "metric_group", "metrics": [
    { "label": "MRR", "value": "$82,400", "delta": "+12%" },
    { "label": "Отток", "value": "2.1%", "delta": "-0.4%" }
  ]},
  { "type": "line_chart", "title": "Рост за 30 дней", "data_ref": "mrr_series" },
  { "type": "insight_callout", "text": "Дельту драйвит расширение выручки — среднее число мест выросло на 18%." }
]
```

Модель собрала эту раскладку. `MetricGroup`, `LineChart`, `InsightCallout`. Но вы определили, что значит каждый тип компонента, какие пропсы он принимает и как рендерится. Если модель попытается выдать `{ "type": "custom_untested_thing" }`, валидация схемы это перехватит, и рендерер проигнорирует или отклонит его.

За этим паттерном стоят `json-render`, `A2UI`, `Hashbrown`, `OpenUI` и `Tambo`. Главная инженерная задача — **проектирование каталога**: какие типы компонентов существуют, как выглядят их схемы и что модели разрешено, а что нет.

**Подходит, когда**: структура того, что нужно показать, действительно меняется в зависимости от данных или запроса пользователя. Дашборды, адаптирующиеся к примечательным изменениям в цифрах. Отчёты с разными разделами в зависимости от контекста. Рабочие панели, меняющиеся в зависимости от этапа агента.

### Паттерн 3: Свободная генерация

Модель пишет HTML, SVG, Canvas или WebGL, который рендерится в песочнице iframe со строгой Content Security Policy.

Подходит для задач, где никакой фиксированный каталог не поможет: алгоритмические визуализации, архитектурные диаграммы, ad hoc графики, генеративное искусство, обучающие симуляции. Граница iframe здесь выполняет работу по безопасности; снимите её — и вы вернётесь к проблеме инъекции сырого HTML из начала статьи.

`CopilotKit/OpenGenerativeUI` — лучшая на сегодняшний день референсная реализация этого паттерна. Песочница вырезает скрипты, ограничивает передачу сообщений и изолирует сгенерированный артефакт от привилегированного состояния приложения.

**Подходит, когда**: вам действительно нужен произвольный визуальный вывод — разовые пояснительные диаграммы, динамические симуляции, творческие артефакты. Не используйте это для транзакционного UI. Подтверждению заказа не нужна песочница iframe.

### За пределами трёх паттернов: LLM, управляющие пикселями напрямую

Существует зарождающееся четвёртое направление, не вписывающееся ни в один из паттернов: LLM, управляющие **иммерсивными, игровыми впечатлениями** путём более прямого контроля визуального вывода, чем песочница iframe.

Каноническое разделение внутри генеративного UI — **iframe HTML против JSON-каталога**:

- **Iframe HTML** — модель пишет HTML, SVG, Canvas или WebGL, рендерящийся в изолированной песочнице. Максимальная свобода выражения; безопасность целиком зависит от границы iframe. Примеры: Anthropic Artifacts, OpenGenerativeUI.
- **JSON-каталог** — модель выдаёт структурированный payload, ограниченный определённым разработчиком каталогом компонентов; ваш рендерер инстанцирует доверенные, заранее собранные компоненты по спецификации. Модель решает *что* показать; вы решаете *как* это рендерится. Примеры: json-render, A2UI.

Помимо этого, совсем свежие демо намекают на третий режим, где модель не выбирает компоненты и не пишет HTML в песочнице — она управляет canvas более напрямую. Проекты вроде [HunyuanWorld от Tencent](https://arxiv.org/abs/2502.01999), генерирующего исследуемые 3D-среды из одного изображения, и игровые архитектуры, где LLM в рантайме генерируют карты, NPC и квесты вместо вызовов каталога компонентов, намекают на будущее, в котором модель больше похожа на игрового режиссёра, чем на рендерер форм. Inference LLM в браузере через WebGPU ([WebLLM](https://mlc.ai/web-llm/)) продвигает тот же фронт локально.

Территория по-настоящему интересная и по-настоящему ранняя. Стабильных фреймворков для продакшен-продуктов пока нет. Освещу этот подход в отдельной статье, когда ситуация изменится.

## Протоколы: AG-UI и A2UI

AG-UI и A2UI — два основных стандарта на уровне протоколов. Они решают разные задачи и не конкурируют.

### AG-UI

**GitHub**: [ag-ui-protocol/ag-ui](https://github.com/ag-ui-protocol/ag-ui)

AG-UI — событийный протокол для коммуникации между ИИ-агентами и фронтенд-приложениями. Он определяет около 16 типов событий: `TEXT_MESSAGE_START`, `TEXT_MESSAGE_CONTENT`, `TOOL_CALL_START`, `TOOL_CALL_END`, `STATE_SNAPSHOT`, `STATE_DELTA` и так далее. Транспорт — на ваш выбор: SSE, WebSockets, вебхуки — всё работает. Формат намеренно свободный для широкого распространения.

AG-UI не определяет, как выглядит ваш UI. Он определяет, как агент общается *с* вашим фронтендом. Думайте об этом как о уровне транспортного протокола, который позволяет вашему React-приложению подписываться на LangGraph-агента так же, как на CrewAI-агента, без изменения кода фронтенда.

CopilotKit создал AG-UI на основе работы с LangGraph и CrewAI. Протокол приняли LangChain, Mastra, PydanticAI и другие. Microsoft опубликовала руководство по интеграции AG-UI. Если вы строите мультиагентный фронтенд и нужно развязать бэкенд-фреймворки от кода фронтенда — AG-UI ответ.

**Уточнение, которое часто путает**: AG-UI — не UI-фреймворк. Он не говорит, что рендерить. Он сообщает, *что* агент что-то сказал, вызвал инструмент или обновил разделяемое состояние. Что рендерить в ответ — по-прежнему ваше решение.

### A2UI

**GitHub**: [google/A2UI](https://github.com/google/A2UI) · Спецификация: [a2ui.org](https://a2ui.org/)

A2UI — декларативная спецификация Google для того, что агенты отправляют, когда хотят показать UI. Если AG-UI отвечает на «как агент общается?», то A2UI отвечает на «в каком формате агент описывает раскладку компонентов?».

A2UI использует плоский формат JSONL: один дескриптор компонента на строку, каждый с ID, типом и данными. Плоскость — осознанный выбор. Вложенные деревья требуют, чтобы модель знала полную структуру до начала стриминга. Плоский список позволяет модели выдавать каждый компонент по мере того, как она его «обдумывает» — фронтенд может начать рендерить первую карточку метрики, пока модель ещё решает, нужен ли график.

```jsonl
{"id":"h1","type":"kpi_card","title":"MRR","value":"$82,400","delta":"+12%"}
{"id":"h2","type":"kpi_card","title":"Отток","value":"2.1%","delta":"-0.4%"}
{"id":"c1","type":"line_chart","title":"MRR за 30 дней","data_ref":"mrr_series"}
{"id":"t1","type":"data_table","cols":["Месяц","MRR","Чистый прирост"],"data_ref":"monthly"}
```

A2UI уделяет внимание безопасности: спецификация — формат данных, не исполняемый код. Каталог компонентов предварительно определён разработчиком; агент может ссылаться только на типы из каталога. Рендерер A2UI, получивший неизвестное имя типа, его проигнорирует.

Формат «Open-JSON-UI» от CopilotKit совместим с A2UI. Если вы выбираете формат спецификации для каталога сегодня — A2UI имеет самую широкую кроссплатформенную поддержку.

**Замечание о стабильности**: A2UI пред-1.0 — v0.9 на момент последней проверки 8 мая 2026 года — и получал breaking changes между минорными версиями. Коммуникации Google о roadmap были нерегулярными, а некоторые рендереры (Lit, Flutter) отставали от обновлений спецификации. Закладывайте время на дрейф спецификации, если строите на ней сейчас. Для чисто веб-сценариев json-render сейчас имеет более полный инструментарий. Долгосрочное преимущество A2UI — кроссплатформенный охват (веб, Flutter, SwiftUI, Android), которого у json-render нет.

### MCP Apps

**GitHub**: [modelcontextprotocol](https://github.com/modelcontextprotocol) · См. также: [mcp-ui](https://github.com/MCP-UI-Org/mcp-ui)

MCP начинался как протокол подключения LLM к инструментам и данным. Расширение Apps позволяет MCP-инструментам возвращать не только данные, но и интерактивные UI-артефакты: React-компоненты, формы, дашборды, карты.

Модель безопасности строгая — и в этом суть: всё рендерится в песочнице iframe с вырезанными разрешениями, шаблоны предварительно объявлены для ревью хост-приложением, вся коммуникация — аудируемый JSON-RPC. Это правильная модель для провайдеров инструментов — MCP-сервер Shopify может вернуть виджет оформления заказа; картографический сервис — встраиваемую карту. Хост-приложение не владеет и не доверяет коду этого виджета.

MCP Apps — правильный выбор, когда UI *принадлежит провайдеру инструмента*, а не вашему приложению. Для UI в домене вашего приложения используйте Паттерн 1 или 2.

## JavaScript/TypeScript-фреймворки

### CopilotKit

**GitHub**: [CopilotKit/CopilotKit](https://github.com/CopilotKit/CopilotKit) · Примеры: [CopilotKit/generative-ui](https://github.com/CopilotKit/generative-ui)

CopilotKit — самый полный фреймворк для агентно-ориентированных фронтенд-приложений. Он покрывает весь жизненный цикл: подключение к агент-бэкендам через AG-UI, управление двунаправленным состоянием диалога, рендеринг генеративных UI-компонентов и разделение состояния, позволяющее агентам и пользователям изменять одни и те же данные.

Модель трёх паттернов чисто ложится на API CopilotKit:
- `useCopilotAction` с `render`-колбэком → Паттерн 1
- Рендеринг A2UI/Open-JSON-UI → Паттерн 2
- Песочница артефактов `OpenGenerativeUI` → Паттерн 3

Важная недооценённая фича CopilotKit — **разделяемое состояние и human-in-the-loop**: агент может читать и писать состояние приложения, пользователь может читать и писать его, и изменения текут в обе стороны. Именно это делает copilot-интерфейсы похожими на реальное сотрудничество, а не на чат-бокс, приклеенный к продукту.

### Vercel AI SDK

**GitHub**: [vercel/ai](https://github.com/vercel/ai) · Документация: [ai-sdk.dev](https://ai-sdk.dev/)

Vercel AI SDK — де-факто TypeScript-базлайн для ИИ-приложений. В части генеративного UI:

**`useObject`** стримит структурированный JSON-объект с сервера по мере его генерации. Вы задаёте Zod-схему; SDK парсит частичный JSON и триггерит ре-рендер по мере поступления полей. Это самый гладкий путь к Паттерну 2 в Next.js-приложении.

```tsx
const { object: dashboard } = useObject({
  api: "/api/generate-dashboard",
  schema: z.object({
    title: z.string(),
    metrics: z.array(z.object({ label: z.string(), value: z.number() })),
    insights: z.array(z.string()),
  }),
});
```

**`useChat` с обработчиками инструментов** → Паттерн 1. Модель вызывает инструменты; вы маппите имена инструментов на компоненты.

**AI Elements** ([elements.ai-sdk.dev](https://elements.ai-sdk.dev/)) предоставляет готовые UI-примитивы для пары с SDK.

**Замечание о запутанной траектории**: В октябре 2024 Vercel объявили в [GitHub Discussion #3251](https://github.com/vercel/ai/discussions/3251), что AI SDK RSC — паттерн стриминга React Server Components, продвигавшийся как заголовочная «Generative UI»-фича в SDK 3.0 — заморожен на неопределённый срок из-за «длительно существующих ограничений» без решений в обозримом будущем. Команды, построившие продукт-стратегии на RSC, были застигнуты врасплох. API `generateObject`/`streamObject` позже были также депрекейтнуты в SDK 6.0. Рекомендуемая миграция с AI SDK RSC — паттерн `useObject` выше или json-render для каталогной генерации.

### assistant-ui

**GitHub**: [assistant-ui/assistant-ui](https://github.com/assistant-ui/assistant-ui)

assistant-ui — набор компонуемых React-примитивов для продакшен-качественных чат-интерфейсов. Правильный выбор, когда нужен полированный чат-UX — пузыри сообщений, стриминг токенов, действия копирования/редактирования/перегенерации, состояния обдумывания — при этом со своим бэкендом и своим рендерингом инструментов.

Хорошо работает с любым бэкендом (OpenAI, Anthropic, локальные модели, кастомные эндпоинты) и обрабатывает рендеринг вызовов инструментов через знакомую модель слотов/render props.

### json-render

**GitHub**: [vercel-labs/json-render](https://github.com/vercel-labs/json-render) · Документация: [json-render.dev](https://json-render.dev/)

json-render реализует Паттерн 2 с opinionated подходом «всё включено». Вы получаете готовый каталог компонентов (shadcn/ui с Zod-схемами), рендерер и плотный цикл генерации, где модель ограничена каталогом через схему.

Отличительные особенности:
- **Мульти-таргетный рендеринг**: одна и та же JSON-спецификация рендерится в React-веб-приложение, React Native-мобильное приложение, PDF, HTML-письмо или Remotion-видео. По-настоящему полезно для отчётов.
- **Прогрессивный рендеринг**: компоненты появляются по мере стриминга моделью, а не после получения полной спецификации.
- **Строгие ограничения схемы**: каталог спроектирован так, чтобы модель не могла галлюцинировать валидные, но неизвестные типы компонентов.

Если вы строите фичу генерации дашбордов или отчётов и хотите обойтись без инфраструктурной работы по проектированию собственного каталога — json-render самый быстрый путь для веб-приложений.

**О динамике развития**: json-render вышел из Vercel Labs в начале 2026 и, похоже, быстро набрал внимание веб-разработчиков, потому что сразу полезен в стандартных React/Next.js-проектах. При этом json-render всё ещё pre-1.0, и отношения между json-render и A2UI всё ещё вырисовываются — Vercel экспериментировал с A2UI-совместимым выводом, так что конвергенция возможна. Для кроссплатформенных сценариев (нативная мобильная разработка, несколько фреймворков) A2UI — лучшая долгосрочная ставка.

### Hashbrown

**GitHub**: [liveloveapp/hashbrown](https://github.com/liveloveapp/hashbrown)

Hashbrown берёт отличительный подход: вместо отдельного AI-интерфейсного слоя он встраивает AI-выбор компонентов прямо в ваше существующее React- или Angular-приложение. Вы открываете компоненты приложения для LLM; LLM выбирает, какие рендерить, и может вызывать клиентские инструменты.

Правильный инструмент, когда нужно добавить интеллект в продуктовые поверхности, не являющиеся «чатом» — продуктовая страница, адаптирующая раскладку; панель настроек, показывающая нужные опции; редактор workflows, предлагающий следующий шаг.

### OpenUI

**GitHub**: [thesysdev/openui](https://github.com/thesysdev/openui) · Документация: [openui.com](https://www.openui.com/)

OpenUI заменяет JSON на строчно-ориентированный кодоподобный формат («OpenUI Lang»), оптимизированный для прогрессивного рендеринга и экономии токенов. Заявлено примерно на 67% меньше токенов, чем эквивалентный JSON при сложных раскладках.

Компромисс — зрелость экосистемы: OpenUI новее, и инструментарий беднее, чем у JSON-подходов. Но если стоимость токенов — значимое ограничение, а вы генерируете сложные раскладки с высокой частотой, эффективность формата реальна.

### Tambo

**GitHub**: [tambo-ai/tambo](https://github.com/tambo-ai/tambo)

Tambo сфокусирован на stateful-выборе компонентов: ИИ выбирает компоненты и может взаимодействовать с ними через клиентские инструменты, сохраняя состояние компонентов на протяжении диалога. Хорош для сценариев, где UI-элементы живут несколько ходов — фильтр, который пользователь подстраивает, пока ИИ продолжает рассуждать об отфильтрованных данных.

## Python-слой

Python-экосистема подходит к ИИ-интерфейсам иначе. Эти инструменты оптимизированы для демо ML-моделей, data-приложений и внутреннего инструментария — не для продакшен-пользовательских приложений с агентно-управляемой композицией раскладки.

Это не критика. Для подходящих задач Gradio и Streamlit — единственные нужные инструменты.

### Gradio

**GitHub**: [gradio-app/gradio](https://github.com/gradio-app/gradio) · PyPI: `gradio`

Главная ценность Gradio: вы пишете Python-функцию; Gradio оборачивает её в веб-интерфейс. Класс `Interface` — 3 строки для классификатора изображений. `ChatInterface` — 10 строк для чат-бота. `Blocks` даёт fine-grained контроль раскладки, когда нужен.

«Генеративный UI» в Gradio определяется Python-разработчиком, а не моделью. Видимость и конфигурация компонентов могут меняться динамически по результатам модели, но каталог компонентов статичен — вы не просите модель компоновать раскладки.

Gradio — стандарт для HuggingFace Spaces и ML-демо-экосистемы. Миллионы загрузок в месяц, большая часть AI-демо-ландшафта работает на нём.

**Gradio, когда**: вы Python-разработчик, делаете демо ML-модели, исследовательский прототип или внутренний инструмент и не хотите трогать JavaScript.

### Streamlit

**GitHub**: [streamlit/streamlit](https://github.com/streamlit/streamlit)

Модель Streamlit более opinionated: Python-скрипт выполняется целиком при каждом взаимодействии. Вы вызываете `st.chat_message()`, `st.dataframe()`, `st.plotly_chart()`. Фреймворк управляет раскладкой.

Модель полного перезапуска скрипта звучит неэффективно, но удивительно эргономична для AI-чат-ботов, накапливающих историю диалогов — скрипт перезапускается целиком, история чата в session state, вывод детерминирован. Streamlit теперь имеет first-party поддержку большинства основных LLM-провайдеров и нативно интегрируется с Snowflake Cortex.

**Streamlit, когда**: вы строите data-приложение на базе ИИ, внутренний инструмент отчётности или ML-дашборд на Python и хотите максимально простой путь деплоя.

### LangChain и Haystack

Это бэкенд-фреймворки оркестрации, не UI-фреймворки. Они появляются в любой честной карте генеративного UI-стека, потому что обычно это тот самый слой, где генерируются структурированные выводы перед отправкой во фронтенд.

**LangChain** ([langchain-ai/langchain](https://github.com/langchain-ai/langchain)): `.with_structured_output()` на любой LLM даёт Pydantic-ограниченную JSON-генерацию. Декоратор `@tool` с автоматической генерацией схемы — самый чистый способ определить инструменты, доступные модели. LangChain передаёт структурированные результаты наверх, к вашему фронтенду (Паттерн 1, 2 или 3).

**Haystack** ([deepset-ai/haystack](https://github.com/deepset-ai/haystack)): модульная pipeline-архитектура с сильной поддержкой RAG. Hayhooks оборачивает Haystack-пайплайны в HTTP-эндпоинты — включая MCP-совместимые. Если вашему генеративному UI нужна retrieval-основа, pipeline-архитектура Haystack обрабатывает это чисто.

Ни один фреймворк не владеет UI-уровнем. Они генерируют данные, которые рендерит ваш фронтенд.

## Скорость экосистемы и шаткая почва

Пространство движется быстро, и несколько проектов выпустили запутанные сообщения вместе с кодом. Последняя проверка 8 мая 2026; рассматривайте заметки о статусе как снимок с временной меткой, а не как окончательный вердикт.

**Vercel AI SDK RSC** был заголовочной Generative UI-фичей при запуске SDK 3.0. Vercel заморозил его развитие в октябре 2024 ([Discussion #3251](https://github.com/vercel/ai/discussions/3251)), сославшись на архитектурные ограничения React Server Components без решений в обозримом будущем. Команды, построившие на нём, были обоснованно разочарованы. RSC всё ещё в документации, но не рекомендуемый путь; `useObject` — да.

**json-render** (Vercel Labs) — новое направление: каталогный, фреймворк-агностичный, без проблем связывания с RSC. Pre-1.0, но, судя по всему, с сильным ранним интересом у React/веб-разработчиков. Вероятная причина DX: json-render сразу полезен в стандартном React/Next.js-проекте, а кроссплатформенный охват A2UI добавляет трения при настройке. Как всё уляжется по мере созревания обеих спецификаций — по-настоящему неясно. Vercel экспериментировала с A2UI-совместимостью в json-render, что намекает на возможную конвергенцию.

**A2UI** (Google) — pre-1.0 (v0.9 на момент проверки) с breaking changes между минорными версиями и непоследовательными коммуникациями Google о roadmap. Правильный выбор для кроссплатформенного охвата (веб + Flutter + SwiftUI), который json-render не покрывает, при значительной enterprise-поддержке. Для чисто веб-проектов прямо сейчас DX грубее.

**AG-UI** (CopilotKit) тоже pre-1.0. Самый частый конфуз: название звучит как UI-фреймворк. Это не так — транспортный протокол. AG-UI определяет, как события текут между агент-бэкендами и вашим фронтендом; что рендерить в ответ — ваше решение. Ментальная модель надёжна и широко принята, но pre-1.0-спецификация означает, что граничные случаи всё ещё дорабатываются.

Практический вывод: **все ключевые игроки здесь — pre-1.0**. Планируйте изменения API. Паттерны — вызов-компонент, каталогная композиция, песочница — достаточно стабильны для построения. Конкретные протоколы — нет.

## Проектирование каталога компонентов: настоящая инженерная работа

Большая часть интересной сложности Паттерна 2 — не в рендерере. Она в каталоге.

Каталог — это **продуктовое решение, закодированное как схема**. Он отвечает: какие значимые UI-объекты существуют в этом домене? Не «какие React-компоненты есть?», а «что пользователю в этом контексте действительно нужно видеть и с чем взаимодействовать?».

**Режим провала: слишком мелкая детализация** — вы открываете `Row`, `Column`, `Text`, `Button`, `Icon`. Теперь модель должна быть фронтенд-разработчиком. Она будет генерировать посредственную раскладку, не соответствующую дизайн-системе, пропустит пустые состояния, создаст недоступную разметку и будет менять подход от ответа к ответу, потому что ничто в каталоге не ограничивает вывод визуальным языком продукта.

**Режим провала: слишком крупная детализация** — вы открываете `WeatherCard`, `FlightCard`, `HotelCard`. Модель не может адаптироваться, когда запрос не маппится на готовую карточку. Она откатывается к тексту.

**Полезная середина**: доменные компоненты с ограниченными слотами.

Каталог travel-приложения мог бы выглядеть так:

```
TripSummary         — маршрут на одном экране
FlightOptionList    — выбираемые рейсы с ценами
HotelComparison     — отели рядом
TravelerForm        — сбор данных путешественника
PolicyNotice        — регуляторные/тарифные правила
BookingConfirmation — финальное подтверждение с кнопкой действия
```

Каталог finance-приложения мог бы выглядеть так:

```
PortfolioSnapshot   — ключевые позиции и P&L
TransactionTable    — фильтруемые, пагинированные транзакции
RiskBreakdown       — распределение и волатильность
ScenarioComparison  — сценарное моделирование рядом
ApprovalGate        — действие, требующее подтверждения человеком
```

Каталог звучит как словарь вашего продукта. Он кодирует UX-решения, требования доступности, обработку пустых состояний и паттерны опасных действий в коде компонентов. Модель расставляет эти элементы. Вы по-прежнему решаете, как выглядит каждый и что ему разрешено делать.

**Правила проектирования схемы, снижающие галлюцинации**:

1. Держите enum-значения короткими и очевидными. `"type": "bar_chart"`, а не `"type": "data-visualization-bar-type-vertical"`.
2. Сделайте невалидную композицию невозможной. Если `PolicyNotice` может появляться только в конце раскладки, не помещайте его на тот же уровень схемы, что и элементы для произвольных позиций.
3. Используйте обязательные поля щедро. Необязательное поле — это поле, которое модель может пропустить, а рендереру придётся обрабатывать как null.
4. Тестируйте каталог на реальных промптах до отгрузки. Сохраняйте сгенерированные спецификации; проверяйте на нарушения схемы, галлюцинированные значения полей и композиции, технически валидные, но семантически неверные.

## Распространённые ловушки

**Ловушка: валидный JSON ≠ безопасное поведение.** Валидация схемы подтверждает структуру. Она ничего не говорит о том, совпадает ли действие кнопки с её меткой, совпадает ли итог с порождающими его данными, или делает ли UI-компонент что-то, чего пользователь не ожидал. Сгенерированные UI-спецификации нуждаются в семантической проверке, а не только в валидации схемы. Как минимум, деструктивные действия должны требовать компонента подтверждения, а метки этих компонентов нужно тестировать на соответствие запускамым ими действиям.

**Ловушка: открытие примитивов дизайна вместо продуктовых.** Если модель решает, использовать ли padding 16px или 20px — вы дали ей неверный уровень абстракции. Доменные компоненты должны кодировать продуктовый вкус. Модель должна компоновать поведение, а не управлять деталями представления.

**Ловушка: генеративный UI там, где достаточно статического.** Если структура того, что нужно показать, известна на этапе разработки — а обычно так и есть — Паттерн 1 с готовыми компонентами быстрее, безопаснее и стабильнее. Генеративный UI оправдывает свою сложность, когда структура действительно варьируется в зависимости от данных или контекста задачи.

**Ловушка: игнорирование доступности.** LLM галлюцинируют нарушения WCAG. Они будут ставить `role="region"` на интерактивных элементах, генерировать формы без меток и выдавать контрастные соотношения, не проходящие WCAG AA. Ваша библиотека компонентов может быть полностью доступной; AI-композиции этих компонентов автоматически доступными не являются. Тестируйте полный путь рендеринга, а не только компоненты в изоляции.

**Ловушка: смешивание протокола и фреймворка.** AG-UI — не фронтенд-фреймворк. A2UI — не React-библиотека. Это форматы передачи и протоколы событий. Для их реализации вам всё ещё нужен фронтенд-фреймворк. CopilotKit реализует AG-UI и A2UI. json-render реализует паттерн каталога A2UI/Open-JSON-UI. Это разные уровни.

## Рекомендации по сценариям

**Добавление копайлота в существующее SaaS-приложение**: Начните с Паттерна 1 (вызов-компонент). Vercel AI SDK `useChat` или CopilotKit. Замапьте 5–10 ключевых действий агента на готовые компоненты. Отгружайте, измеряйте, расширяйте каталог только если пользователи демонстративно нуждаются в более богатой композиции.

**Генерация дашбордов по естественному языку**: Паттерн 2 с json-render или кастомным A2UI-каталогом. Определите каталог из 8–15 типов компонентов: типы графиков, метрические карточки, варианты таблиц. Скормите схему модели; позвольте ей скомпоновать раскладку. Создайте валидацию, перехватывающую неизвестные типы до рендерера.

**Мультиагентный фронтенд**: CopilotKit с AG-UI. Поток событий обрабатывает реалтайм-стриминг между агент-бэкендами; разделяемое состояние управляет передачей между агентами; HITL-паттерн обрабатывает ворота согласования.

**Встраивание в ChatGPT или другой MCP-хост**: MCP Apps. Определите свой инструмент как data-инструмент (выборка и рассуждение) и отдельный render-инструмент (запрос виджета). Держите бизнес-логику вне шаблона виджета.

**ML-демо и data-приложения (Python-команда)**: Gradio для демо и HuggingFace Spaces. Streamlit для data-приложений с более сложным взаимодействием. Ни то, ни другое не требует JavaScript.

**Визуальные артефакты, симуляции, диаграммы**: Паттерн 3 (OpenGenerativeUI или аналог). Жёсткий iframe CSP. Относитесь к выводу как к ненадёжному пользовательскому контенту с точки зрения безопасности.

Фреймворки быстро взрослеют. Конвергенция протоколов (AG-UI для стриминга, A2UI/Open-JSON-UI для каталогных спецификаций) всё ещё в процессе, но контуры уже достаточно ясны для построения.

Самые важные инженерные задачи сейчас — не выбор фреймворка. Это проектирование каталога — решение, что модели разрешено говорить, требующее продуктового чутья больше, чем технических навыков. Семантическая валидация — проверка того, что сгенерированный UI делает то, что заявляет, а не просто проходит схему. И разрыв в доступности — создание каталогов, где каждый компонент и каждая композиция компонентов соответствует стандарту доступности, которого вы требовали бы от hand-written UI.

Модель сделает то, что вы ей скажете, в рамках грамматики, которую вы ей зададите. Сделайте грамматику осознанной.
