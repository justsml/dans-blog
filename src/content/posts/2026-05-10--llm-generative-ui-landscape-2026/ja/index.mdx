「Generative UI（生成型UI）」とは、誰が言うかによって少なくとも5つの異なる意味を持つ。

- モデルのツール呼び出しから商品カードを埋め込むチャットインターフェース
- フロントエンドがコンポーネントツリーとしてレンダリングする実行時JSON仕様
- MCPツールからホストアプリに返されるサンドボックス化されたiframe（チケット予約、ホテル予約から地図レンダリング、決済ウィジェットまで）
- エージェントの状態をフロントエンドにストリーミングするイベントプロトコル
- v0、Lovable、Bolt：設計時にReactを書くAIツール

これらは関連する概念だが、スタックの異なるレイヤーに存在し、異なるリスクプロファイル、実装コスト、適切なユースケースを持つ。これらを混同すると、あらゆるアーキテクチャの議論が混乱に陥る。

これは、スタックのどこに手を伸ばすべきかを判断するときに私が欲しい地図だ。

---

## Generative UI でないもの

定義する前に、3つ脇に置いておくものがある：

**設計時のコード生成**——v0、Lovable、Bolt、CursorによるReactコンポーネントの作成。これらのツールは、開発者がレビューしてコミットするコードを生成する。AIは開発時に実行される。出荷されるものは、ユーザーの視点からは静的だ。これは素晴らしいツールのカテゴリーだ。しかし、これは「実行時のgenerative UI」とは異なる。

**AI支援フォーム自動入力**——モデルがコンテキストからフィールド値を入力する。インターフェースの構造は依然として固定されており、動くのはコンテンツだけだ。これは有用なパターンだ。しかし、generative UI ではない。

**AIが生のHTMLをページに書き込む**——モデルが `<div>` や `<button>` の文字列を出力し、`innerHTML` や `dangerouslySetInnerHTML` で注入される。これは技術的な意味では最も実行時generative UI である。同時に最も危険なバージョンであり、この分野のあらゆる成熟したフレームワークが存在する理由でもある。生のAI生成マークアップは、XSSリスク、アクセシビリティ属性の欠如、一貫性のないスタイリング、幻覚による構造をもたらす。この記事の残りは、これよりも上手くやる方法についてだ。

## 実用的な定義

実行時のgenerative UI とは：**モデルが、会話やタスクの状態に基づいて、ユーザーが見るインターフェースコンポーネントやコンポーネントの構成を決定する**ことだ。

言葉ではない。インターフェースだ。

最も単純なケース：フライト予約アシスタントが `search_flights` ツールを呼び出す。プレーンテキスト（「ここに3つのオプションがあります...」）を返す代わりに、選択可能なフライト、座席クラス切り替え、「予約」ボタンを持つ `<FlightResultsCard>` コンポーネントをレンダリングする。モデルは、構造化カードがここで適切な応答であると判断した。開発者は、そのカードの見た目と「予約」の動作を決定した。

より複雑なケース：金融分析エージェントがポートフォリオに関する質問を受け取り、キー数値を示す `MetricGroup`、リスク内訳チャートの `RiskBreakdown`、シナリオ比較テーブルの `ScenarioComparison`、ポリシー通知の `PolicyNotice` を使って応答を構成すると決定する。モデルは、事前承認されたコンポーネントのカタログからそのレイアウトを組み立てた。開発者はあらゆるコンポーネントを定義した。モデルは、どのコンポーネントを使用し、どのデータを入れるかを選択した。

両方のケースがgenerative UI だ。モデルが持つ構成の自由度の量が異なり、それが可能な出力の豊かさと、問題が起こりうる複雑さの両方を決定する。

## 3つのパターン

この分野全体は、異なる出力文法を持つ3つのパターンに集約される。

![左から右へ：ツール呼び出しのみ（最も安全）、コンポーネントカタログ（中間）、オープンエンド生成（最も表現力豊か）を示すスペクトラム図。](./output-grammar-spectrum.svg)

_あらゆるgenerative UI の決定は、このスペクトラム上の1点だ。左から始めよ。_

### パターン1：ツールからコンポーネントへのレンダリング

モデルが名前付きツールを呼び出す。アプリケーションには、ツール名からコンポーネントへのマップがある。ツール呼び出しがコンポーネントのレンダリングをトリガーする。

```tsx
// モデルが呼び出す: { name: "show_flight_results", args: { flights: [...] } }

useCopilotAction({
  name: "show_flight_results",
  render: ({ args }) => <FlightResultsCard flights={args.flights} />,
});
```

これは最も安全なパターンだ。なぜなら、レイアウトがモデルから来ることは決してないからだ。モデルは、コンポーネントを*いつ*表示するかと、どのデータで埋めるかを決定する。開発者は依然としてコンポーネントコード、ビジュアルデザイン、アクセシビリティ実装、レンダリングロジックのあらゆるエッジケースを所有している。

Vercel AI SDKの `useChat` と `tool` ハンドラはこれを行う。assistant-uiのツールレンダリングはこれを行う。CopilotKitの「Static Generative UI」はこのパターンだ。信頼性が高く動作するほとんどの本番コパイロットUIはこれを行っている。

**適切な場面**：開発時に表示したいもののセットが把握できる場合。予約確認、検索結果、アカウントサマリー、承認ウィジェット。シナリオを列挙できるなら、このパターンでカバーできる。

### パターン2：コンポーネントカタログの構成

モデルは、開発者が定義したカタログからコンポーネントを参照する型付きJSONツリーを出力する。フロントエンドには、そのツリーを走査して各コンポーネントをインスタンス化するレンダラーがある。

```json
[
  { "type": "metric_group", "metrics": [
    { "label": "MRR", "value": "$82,400", "delta": "+12%" },
    { "label": "Churn", "value": "2.1%", "delta": "-0.4%" }
  ]},
  { "type": "line_chart", "title": "30-day growth", "data_ref": "mrr_series" },
  { "type": "insight_callout", "text": "Expansion revenue driving the delta — avg seat count up 18%." }
]
```

モデルはそのレイアウトを構成した。`MetricGroup`、`LineChart`、`InsightCallout`。しかし、各コンポーネントタイプが何を意味し、どのpropsを受け入れ、どのようにレンダリングするかは、あなたが定義したものだ。モデルが `{ "type": "custom_untested_thing" }` を出力しようとした場合、スキーマ検証がそれを捕捉し、レンダラーは無視するか拒否する。

このパターンが `json-render`、`A2UI`、`Hashbrown`、`OpenUI`、`Tambo` の背後にある。重要なエンジニアリング作業は**カタログ設計**——どのコンポーネントタイプが存在し、それらのスキーマがどのように見え、モデルが何を構成してよくて何を構成してはいけないかを決定することだ。

**適切な場面**：表示したいものの構造が、データやユーザーのリクエストに基づいて正当に変化する場合。注目すべき数値に適応するダッシュボード。コンテキストに応じて異なるセクションを表示するレポート。エージェントがどのステップにいるかに基づいて変化するワークフローパネル。

### パターン3：オープンエンド生成

モデルがHTML、SVG、Canvas、またはWebGLを書き、厳格なコンテンツセキュリティポリシーでサンドボックス化されたiframe内にレンダリングされる。

これは、固定されたコンポーネントカタログでは対応できないものに適している：アルゴリズムの可視化、アーキテクチャ図、アドホックなチャート、ジェネレーティブアート、教育シミュレーション。ここではiframe境界がセキュリティ作業を行っている；これを取り除くと、この記事の冒頭の生HTML注入問題に戻ることになる。

`CopilotKit/OpenGenerativeUI` は、このパターンの最良の現在の参照実装だ。サンドボックスはスクリプトを除去し、メッセージ passing を制限し、生成された成果物をアプリケーションの特権状態から遠ざける。

**適切な場面**：本当に任意のビジュアル出力が必要な場合——一度きりの説明図、動的シミュレーション、クリエイティブな成果物。これをトランザクションUIに使用するな。決済確認にサンドボックス化されたiframeは必要ない。

### 3つのパターンを超えて：LLMが直接ピクセルを駆動する

これらのパターンのいずれにもきれいに収まらない、新たな第4の方向性が出現している：LLMが**没入型、ゲームのような体験**を、サンドボックス化されたiframeよりも直接的にビジュアル出力を制御することで駆動する。

Generative UI 内の決定的な区別は**iframe HTML vs JSONカタログ**だ：

- **Iframe HTML**——モデルがHTML、SVG、Canvas、またはWebGLを書き、分離されたサンドボックスでレンダリングする。最大の表現的自由；セキュリティは完全にiframe境界に依存する。例：Anthropic Artifacts、OpenGenerativeUI。
- **JSONカタログ**——モデルは、開発者が定義したコンポーネントカタログに制約された構造化ペイロードを出力する；レンダラーはその仕様から信頼された、事前構築されたコンポーネントをインスタンス化する。モデルは*何を*表示するかを決定する；あなたは*どのように*レンダリングするかを決定する。例：json-render、A2UI。

これらを超えて、非常に最近のデモは、モデルがコンポーネントを選択したりサンドボックス化されたHTMLを書いたりするのではなく——キャンバスをより直接的に駆動する——第3のモードを示唆している。単一の画像から探索可能な3D環境を生成する[TencentのHunyuanWorld](https://arxiv.org/abs/2502.01999)のようなプロジェクトや、LLMがコンポーネントカタログを呼び出すのではなく実行時にマップ、NPC、クエストを生成するゲームアーキテクチャは、モデルがフォームレンダラーというよりゲームディレクターのような未来を示唆している。WebGPUを介したブラウザ内LLM推論（[WebLLM](https://mlc.ai/web-llm/)）も、同じフロンティアをローカルに押し進めている。

この領域は本当にエキサイティングであり、本当に初期段階だ。本番製品を構築できる安定したフレームワークはまだない。それが変わったら、専用の記事でこのアプローチをカバーする。

## プロトコル：AG-UIとA2UI

AG-UIとA2UIは、プロトコルレイヤーの2つの主要な標準だ。異なる問題を解決し、競合関係にはない。

### AG-UI

**GitHub**: [ag-ui-protocol/ag-ui](https://github.com/ag-ui-protocol/ag-ui)

AG-UIは、AIエージェントとフロントエンドアプリケーション間の通信のためのイベントベースのプロトコルだ。約16のイベントタイプを定義する：`TEXT_MESSAGE_START`、`TEXT_MESSAGE_CONTENT`、`TOOL_CALL_START`、`TOOL_CALL_END`、`STATE_SNAPSHOT`、`STATE_DELTA`など。トランスポートはあなた次第——SSE、WebSockets、webhooksがすべて動作する。フォーマットは広範な採用を可能にするために意図的に緩やかだ。

AG-UIは、あなたのUIがどのように見えるかを定義しない。エージェントがフロントエンドと*どのように*通信するかを定義する。ReactアプリがLangGraphエージェントを購読する方法が、CrewAIエージェントを購読する方法と同じであり、フロントエンドコードを変更する必要がない——そう考えてほしい。

CopilotKitは、LangGraphやCrewAIとの作業からAG-UIを作成した。LangChain、Mastra、PydanticAIなどに採用されている。MicrosoftはAG-UI統合ガイドを公開している。マルチエージェントフロントエンドを構築し、バックエンドフレームワークをフロントエンドコードから分離する必要がある場合、AG-UIが答えだ。

**人々を混乱させる1つの明確化**：AG-UIはUIフレームワークではない。何をレンダリングすべきかを教えてくれない。エージェントが何かを言った、ツールを呼び出した、共有状態を更新した——*それ*を教えてくれる。応答して何をレンダリングするかは、依然としてあなたの決定だ。

### A2UI

**GitHub**: [google/A2UI](https://github.com/google/A2UI) · 仕様: [a2ui.org](https://a2ui.org/)

A2UIは、エージェントがUIを表示したいときに何を送信するかについてのGoogleの宣言型仕様だ。AG-UIが「エージェントはどのように通信するか？」に答えるのに対し、A2UIは「エージェントがコンポーネントレイアウトを記述するためにどのフォーマットを使用するか？」に答える。

A2UIはフラットなJSONLフォーマットを使用する：1行につき1つのコンポーネント記述子で、それぞれにID、タイプ、データがある。フラットであることは意図的だ。ネストされたツリーは、モデルがレンダリングを開始する前に完全な構造を知る必要がある。フラットなリストは、モデルがそれを「考える」につれて各コンポーネントを出力できることを意味し、これはフロントエンドが、モデルがチャートを追加するかどうかまだ決定している間に、最初のメトリックカードのレンダリングを開始できることを意味する。

```jsonl
{"id":"h1","type":"kpi_card","title":"MRR","value":"$82,400","delta":"+12%"}
{"id":"h2","type":"kpi_card","title":"Churn","value":"2.1%","delta":"-0.4%"}
{"id":"c1","type":"line_chart","title":"30-day MRR","data_ref":"mrr_series"}
{"id":"t1","type":"data_table","cols":["Month","MRR","Net New"],"data_ref":"monthly"}
```

A2UIはセキュリティを意識している：仕様はデータフォーマットであり、実行可能コードではない。コンポーネントカタログは開発者によって事前定義される；エージェントはそのカタログ内のタイプのみを参照できる。未知のタイプ名を受け取ったA2UIレンダラーは、それを無視する。

CopilotKitの「Open-JSON-UI」フォーマットはA2UIと互換性がある。今日コンポーネントカタログの仕様フォーマットを選ぶなら、A2UIが最も広範なクロスプラットフォームの支持を持つものだ。

**安定性に関する注記**：A2UIはプレ1.0——2026年5月8日時点でv0.9——であり、マイナーバージョン間で破壊的な仕様変更が行われている。Googleのロードマップに関するコミュニケーションは断続的であり、一部のレンダラー（Lit、Flutter）は仕様更新に遅れを取っている。今日これに基づいて構築する場合、仕様の漂流に時間を割くことを見込んでおけ。純粋なWebユースケースでは、json-renderが現在より完全なツールを持っているように見える。A2UIの長期的な利点は、json-renderが対応していないクロスプラットフォームリーチ（web、Flutter、SwiftUI、Android）だ。

### MCP Apps

**GitHub**: [modelcontextprotocol](https://github.com/modelcontextprotocol) · 関連: [mcp-ui](https://github.com/MCP-UI-Org/mcp-ui)

MCPは、LLMをツールとデータに接続するプロトコルとして始まった。Apps拡張機能により、MCPツールはデータだけでなく、インタラクティブなUI成果物——Reactコンポーネント、フォーム、ダッシュボード、地図——を返せるようになった。

セキュリティモデルは厳格であり、それがポイントだ：すべては権限を剥奪されたサンドボックス化されたiframeでレンダリングされ、テンプレートは事前宣言されるのでホストアプリがレビューでき、すべての通信は監査可能なJSON-RPCだ。これはツールプロバイダーにとって正しいモデルだ——Shopify MCPサーバーは決済ウィジェットを返せる；マッピングサービスは埋め込み可能な地図を返せる。ホストアプリは、そのウィジェットのコードを所有も信頼もしない。

MCP Appsは、UIが*ツールプロバイダー*に属し、あなたのアプリケーションに属さない場合に正しい選択だ。あなたのアプリケーションのドメインに存在するUIには、パターン1または2を使用し続けよ。

## JavaScript/TypeScriptフレームワーク

### CopilotKit

**GitHub**: [CopilotKit/CopilotKit](https://github.com/CopilotKit/CopilotKit) · 例: [CopilotKit/generative-ui](https://github.com/CopilotKit/generative-ui)

CopilotKitは、エージェントネイティブのフロントエンドアプリケーションのための最も完全なフレームワークだ。フルライフサイクルを処理する：AG-UIを介したエージェントバックエンドへの接続、双方向の会話状態の管理、generative UIコンポーネントのレンダリング、エージェントとユーザーが同じデータを変更できるようにする共有状態の配管の提供。

3パターンのモデルは、CopilotKitのAPIにきれいにマッピングされる：
- `render` コールバック付き `useCopilotAction` → パターン1
- A2UI/Open-JSON-UIレンダリング → パターン2
- `OpenGenerativeUI` サンドボックス化成果物 → パターン3

議論が不足している重要なCopilotKit機能は**共有状態とヒューマンインザループ**だ：エージェントはアプリケーション状態を読み書きでき、ユーザーはそれを読み書きでき、変更は双方向に流れる。これが、コパイロットスタイルのUIを、製品に付け足されたチャットボックスではなく、本当の協働のように感じさせるものだ。

### Vercel AI SDK

**GitHub**: [vercel/ai](https://github.com/vercel/ai) · ドキュメント: [ai-sdk.dev](https://ai-sdk.dev/)

Vercel AI SDKは、AIアプリの事実上のTypeScriptベースラインだ。Generative UI 特有には：

**`useObject`** は、構造化JSONオブジェクトをサーバーから生成されるにつれてストリーミングする。Zodスキーマを定義する；SDKは部分JSONを解析し、フィールドが到着するにつれて再レンダリングをトリガーする。これは、Next.jsアプリでパターン2に至る最もスムーズなパスだ。

```tsx
const { object: dashboard } = useObject({
  api: "/api/generate-dashboard",
  schema: z.object({
    title: z.string(),
    metrics: z.array(z.object({ label: z.string(), value: z.number() })),
    insights: z.array(z.string()),
  }),
});
```

**ツールハンドラ付き `useChat`** → パターン1。モデルがツールを呼び出す；あなたはツール名をコンポーネントにマッピングする。

**AI Elements** ([elements.ai-sdk.dev](https://elements.ai-sdk.dev/)) は、SDKと組み合わせるための既製のUIプリミティブを提供する。

**ここでの混乱の軌跡に関する注記**：2024年10月、Vercelは[GitHub Discussion #3251](https://github.com/vercel/ai/discussions/3251)で、AI SDK RSC——SDK 3.0の目玉「Generative UI」機能として推進されたReact Server Componentsストリーミングパターン——が「いくつかの長年の制限」により無期限に一時停止されたことを発表した。RSCストリーミングを中心に製品戦略を構築していたチームは、不意を突かれた。`generateObject`/`streamObject` APIも後にSDK 6.0で非推奨となった。AI SDK RSCからの推奨移行先は、上記の `useObject` パターン、またはカタログベース生成のjson-renderだ。

### assistant-ui

**GitHub**: [assistant-ui/assistant-ui](https://github.com/assistant-ui/assistant-ui)

assistant-uiは、本番品質のチャットインターフェースを構築するための、構成可能なReactプリミティブのセットだ。洗練されたチャットUX——メッセージバブル、ストリーミングトークン、コピー/編集/再生成アクション、思考状態——が必要で、独自のバックエンドと独自のツールレンダリングを持ち込みたい場合に正しい答えだ。

どのバックエンド（OpenAI、Anthropic、ローカルモデル、カスタムエンドポイント）ともうまく連携し、おなじみのスロット/レンダープロップモデルを介してツール呼び出しレンダリングを処理する。

### json-render

**GitHub**: [vercel-labs/json-render](https://github.com/vercel-labs/json-render) · ドキュメント: [json-render.dev](https://json-render.dev/)

json-renderは、意見が強く、バッテリー内蔵のアプローチでパターン2を実用化する。事前構築されたコンポーネントカタログ（Zodスキーマを持つshadcn/uiコンポーネント）、レンダラー、およびモデルがスキーマによってカタログに制約されるタイトな生成ループを得られる。

差別化機能：
- **マルチターゲットレンダリング**：同じJSON仕様は、React Webアプリ、React Nativeモバイルアプリ、PDF、HTMLメール、またはRemotionビデオにレンダリングできる。これはレポートにとって本当に有用だ。
- **プログレッシブレンダリング**：コンポーネントは、モデルがそれらをストリーミングするにつれて表示され、完全な仕様が到着してからではない。
- **厳格なスキーマ制約**：カタログは、モデルが有効だが未知のコンポーネントタイプを幻覚で生み出せないように設計されている。

ダッシュボードやレポート生成機能を構築し、独自のカタログを設計するインフラ作業をスキップしたい場合、json-renderはWebアプリへの最速のパスだ。

**勢いについて**：json-renderは2026年初頭にVercel Labsからローンチし、標準的なReact/Next.jsプロジェクトですぐに有用であるため、Web開発者の注目を急速に集めているように見える。とはいえ、json-renderはまだプレ1.0であり、json-renderとA2UIの関係はまだ検討中だ——VercelはA2UI互換出力を試験しており、収束は可能だ。クロスプラットフォーム（ネイティブモバイル、複数のフレームワーク）では、A2UIが長期的により良い賭けだ。

### Hashbrown

**GitHub**: [liveloveapp/hashbrown](https://github.com/liveloveapp/hashbrown)

Hashbrownは、独自のアプローチを取る：別のAIインターフェースレイヤーを構築するのではなく、既存のReactまたはAngularアプリにAIコンポーネント選択を直接埋め込む。アプリのコンポーネントをLLMに公開する；LLMはどれをレンダリングするかを選択し、クライアントサイドツールを呼び出せる。

これは、「チャット」ではない製品画面にインテリジェンスを注入したい場合——レイアウトに適応する製品ページ、適切なオプションを表示する設定パネル、次のステップを提案するワークフローエディタ——に適したツールだ。

### OpenUI

**GitHub**: [thesysdev/openui](https://github.com/thesysdev/openui) · ドキュメント: [openui.com](https://www.openui.com/)

OpenUIは、JSONを、プログレッシブレンダリングとトークン効率のために設計された行指向のコードのようなフォーマット（「OpenUI Lang」）に置き換える。主張は、複雑なレイアウトで同等のJSONより約67%少ないトークンというものだ。

トレードオフはエコシステムの成熟度だ——OpenUIはより新しく、ツールはJSONベースのアプローチより薄い。しかし、トークンコストが意味のある制約であり、高頻度で複雑なレイアウトを生成する場合、フォーマット効率は本物だ。

### Tambo

**GitHub**: [tambo-ai/tambo](https://github.com/tambo-ai/tambo)

Tamboは、ステートフルなコンポーネント選択に焦点を当てる：AIはコンポーネントを選択し、クライアントサイドツールを介してそれらと対話でき、会話を通じてコンポーネント状態を維持する。UI要素がターンを超えて持続するユースケースに適している——ユーザーが調整しながらAIがフィルタリングされたデータについて推論し続けるフィルタコンポーネントなど。

## Pythonレイヤー

Pythonエコシステムは、AIインターフェースに異なるアプローチを取る。これらのツールは、MLモデルのデモ、データアプリケーション、内部ツール——エージェント駆動のレイアウト構成を持つ本番消費者アプリではなく——に最適化されている。

それは批判ではない。適切なユースケースでは、GradioとStreamlitが必要な唯一のツールだ。

### Gradio

**GitHub**: [gradio-app/gradio](https://github.com/gradio-app/gradio) · PyPI: `gradio`

Gradioの中核的な価値：Python関数を書く；GradioはそれをWeb UIにラップする。`Interface` クラスは画像分類器で3行だ。`ChatInterface` はチャットボットで10行だ。`Blocks` は必要なときにきめ細かいレイアウト制御を提供する。

Gradioにおける「generative UI」は、モデルではなくPython開発者によって定義される。コンポーネントの可視性と設定はモデル出力に基づいて動的に変更できるが、コンポーネントカタログは静的だ——モデルにレイアウトを構成させるわけではない。

Gradioは、HuggingFace SpacesとMLデモエコシステムのデフォルトだ。月間数百万のダウンロードがあり、AIデモ環境の大部分を支えている。

**Gradioを選ぶ場面**：MLモデルのデモ、研究プロトタイプ、または内部ツールを構築するPython開発者で、JavaScriptに触れたくない場合。

### Streamlit

**GitHub**: [streamlit/streamlit](https://github.com/streamlit/streamlit)

Streamlitのモデルは、より意見が強い：Pythonスクリプトが各インタラクションでエンドツーエンド実行される。`st.chat_message()`、`st.dataframe()`、`st.plotly_chart()` を呼び出す。フレームワークがレイアウトを処理する。

フルスクリプトリランのモデルは非効率的に聞こえるが、会話履歴を蓄積するAIチャットボットには驚くほど快適だ——スクリプト全体が再実行され、チャット履歴はセッション状態にあり、出力は決定論的だ。Streamlitは現在、ほとんどの主要なLLMプロバイダーをファーストパーティでサポートし、Snowflake Cortexとネイティブに統合している。

**Streamlitを選ぶ場面**：PythonでAI駆動のデータアプリ、内部レポートツール、またはMLバックエンドのダッシュボードを構築し、可能な限りシンプルなデプロイパスが欲しい場合。

### LangChainとHaystack

これらはバックエンドオーケストレーションフレームワークであり、UIフレームワークではない。構造化出力がフロントエンドに送信される前に生成されるレイヤーが通常これらであるため、正直なgenerative UI スタックマップに登場する。

**LangChain** ([langchain-ai/langchain](https://github.com/langchain-ai/langchain))：どのLLMでも `.with_structured_output()` を使うと、Pydanticで制約されたJSON生成が得られる。自動スキーマ生成付きの `@tool` デコレーターは、モデルが呼び出せるツールを定義する最もクリーンな方法だ。LangChainは、使用しているどのフロントエンドレイヤーにも構造化結果を供給する。

**Haystack** ([deepset-ai/haystack](https://github.com/deepset-ai/haystack))：強力なRAGサポートを持つモジュール型パイプラインアーキテクチャ。Hayhooksは、HaystackパイプラインをHTTPエンドポイント——MCP互換エンドポイントを含む——としてラップする。Generative UI に検索バックボーンが必要な場合、Haystackのパイプラインアーキテクチャがそれをきれいに処理する。

どちらのフレームワークもUI層を所有していない。フロントエンド（パターン1、2、または3）がレンダリングするデータを生成する。

## エコシステムの速度と不安定な地盤

この分野は急速に動いており、いくつかのプロジェクトはコードと並行して混乱したコミュニケーションを発信している。2026年5月8日時点で最終確認；ここのプロジェクトステータス注記は、タイムスタンプ付きの読み物として扱い、永続的な評決ではない。

**Vercel AI SDK RSC** は、SDK 3.0がローンチしたときの目玉Generative UI機能だった。Vercelは2024年10月（[Discussion #3251](https://github.com/vercel/ai/discussions/3251)）に、React Server Componentsの「いくつかの長年の制限」により近い将来の解決策がないことを理由に、その開発を一時停止した。それに基づいて製品戦略を構築していたチームは、当然ながら失望した。それはまだドキュメントにあるが、推奨パスではない；`useObject` がそうだ。

**json-render**（Vercel Labs）は新しい方向性——RSCの結合問題を回避する、カタログベースでフレームワークに依存しない代替案だ。プレ1.0であり、React/Web開発者の間で強い初期関心を集めているように見える。可能性の高いDX的理由：json-renderは標準的なReact/Next.jsプロジェクトですぐに有用であり、A2UIのクロスプラットフォームスコープはセットアップの摩擦を増やす。両方の仕様が成熟するにつれてこれがどうなるかは、本当に不明だ。Vercelはjson-renderでA2UI互換性を試験しており、収束は可能だと示唆している。

**A2UI**（Google）はプレ1.0（最終確認時v0.9）で、マイナーバージョン間に破壊的変更があり、Googleのロードマップに関するコミュニケーションは一貫していない。クロスプラットフォームリーチ（web + Flutter + SwiftUI）というjson-renderが対応しない領域では正しい選択であり、意味のあるエンタープライズバックがある。純粋なWebプロジェクトでは、現在DXはより厳しい。

**AG-UI**（CopilotKit）もプレ1.0だ。最も一般的な混乱：名前からUIフレームワークのように聞こえる。そうではない——それはトランスポートプロトコルだ。AG-UIは、エージェントバックエンドとフロントエンド間でイベントがどのように流れるかを定義する；応答して何をレンダリングするかは依然としてあなたの決定だ。そのメンタルモデルは堅実で広く採用されているが、プレ1.0の仕様は、エッジケースがまだ解決中であることを意味する。

実用的な結論：**ここの主要なプレーヤーはすべてプレ1.0だ**。API変更を計画せよ。パターン——ツールからコンポーネント、カタログ構成、サンドボックス化生成——は、構築に十分安定している。特定のプロトコル選択はそうではない。

## コンポーネントカタログ設計：本当のエンジニアリング作業

パターン2の興味深い複雑性のほとんどは、レンダラーではなくカタログにある。

カタログは**スキーマとしてエンコードされたプロダクト決定**だ。それは答える：このドメインで意味のあるUIオブジェクトは何か？「どのReactコンポーネントが存在するか？」ではなく「このコンテキストのユーザーが実際に見て対話する必要があるものは何か？」だ。

**細かすぎる失敗モード**：`Row`、`Column`、`Text`、`Button`、`Icon` を公開する。これでモデルはフロントエンドエンジニアでなければならなくなる。それはあなたのデザインシステムに合わない平凡なレイアウトを生成し、空の状態を見逃し、アクセシブルでないマークアップを生成し、カタログに何もその製品のビジュアル言語に出力を制約しないため、応答ごとにアプローチを変えるだろう。

**粗すぎる失敗モード**：`WeatherCard`、`FlightCard`、`HotelCard` を公開する。モデルは、事前作成されたカードにマッピングされないものをユーザーが求めたときに適応できない。テキストにフォールバックする。

**有用な中間**：制約されたスロットを持つドメインレベルのコンポーネント。

旅行アプリのカタログは次のようになるかもしれない：

```
TripSummary         — 一目でわかる旅程
FlightOptionList    — 価格付きの選択可能なフライトオプション
HotelComparison     — 並べたホテルカード
TravelerForm        — 旅行者詳細の収集
PolicyNotice        — 規制/運賃ルールのコールアウト
BookingConfirmation — アクションボタン付き最終確認
```

金融アプリのカタログは次のようになるかもしれない：

```
PortfolioSnapshot   — 主要ポジションと損益
TransactionTable    — フィルタリング可能、ページネーション付き取引
RiskBreakdown       — アロケーションとボラティリティメトリクス
ScenarioComparison  — 並べたシナリオモデリング
ApprovalGate        — 人間の確認を必要とするアクション
```

カタログは、あなたの製品の語彙のように聞こえる。それは、コンポーネントコードにあなたのUX決定、アクセシビリティ要件、空の状態の処理、危険なアクションパターンをエンコードする。モデルはそれらのピースを配置できる。あなたはまだ、各ピースの見た目と、それが許可されていることを決定する。

**幻覚を減らすスキーマ設計ルール**：

1. enum値は短く明確に保つ。`"type": "bar_chart"` であり、`"type": "data-visualization-bar-type-vertical"` ではない。
2. 無効な構成を不可能にする。`PolicyNotice` がレイアウトの最後にしか現れないなら、どこにでも現れるものと同じスキーマレベルに置かない。
3. 必須フィールドを惜しみなく使用する。オプションフィールドは、モデルが省略するかもしれず、レンダラーがnullとして処理しなければならないフィールドだ。
4. 出荷前に実際のプロンプトに対してカタログをテストする。生成された仕様を保存する；スキーマ違反、幻覚によるフィールド値、技術的には有効だが意味的に間違った構成をチェックする。

## よくある落とし穴

**落とし穴：有効なJSONを安全な動作として扱う。** スキーマ検証は構造を確認する。ボタンに付随するアクションがそのラベルと一致するか、合計が導出元のデータと一致するか、UIコンポーネントがユーザーが期待しないことをしているか——については何も言わない。生成されたUI仕様には、スキーマ検証だけでなく意味的レビューが必要だ。最低限、破壊的アクションは確認コンポーネントを必要とし、それらのコンポーネントのラベルは、それらがトリガーするアクションに対してテストされるべきだ。

**落とし穴：デザインプリミティブではなくプロダクトプリミティブを公開する。** モデルが16pxか20pxのパディングを使うかを決めなければならないなら、間違った抽象化レベルを与えたことになる。ドメインコンポーネントはプロダクトのセンスをエンコードすべきだ。モデルは振る舞いを構成すべきであり、プレゼンテーションの詳細を管理すべきではない。

**落とし穴：静的UIで十分な場所にgenerative UIを使用する。** 表示したいものの構造が開発時に把握できる——そして通常はそうだ——場合、パターン1の事前構築コンポーネントの方が、より速く、安全で、一貫性がある。Generative UI は、構造がデータやタスクコンテキストに基づいて本当に変化するときに、その複雑性を正当化する。

**落とし穴：アクセシビリティをスキップする。** LLMはWCAG違反を幻覚で生み出す。インタラクティブ要素に `role="region"` を付与し、ラベルの欠けたフォームを生成し、WCAG AAに失敗するコントラスト比を生み出す。あなたのコンポーネントライブラリは完全にアクセシブルかもしれない；それらのコンポーネントのAI生成構成は、自動的にアクセシブルではない。コンポーネントを単独でではなく、完全なレンダリングパスをテストせよ。

**落とし穴：プロトコルとフレームワークを混同する。** AG-UIはフロントエンドフレームワークではない。A2UIはReactライブラリではない。それらはワイヤーフォーマットとイベントプロトコルだ。それらを実装するには、依然としてフロントエンドフレームワークが必要だ。CopilotKitはAG-UIとA2UIを実装する。json-renderはA2UI/Open-JSON-UIカタログパターンを実装する。これらは異なるレイヤーだ。

## ユースケース別の推奨事項

**既存のSaaSアプリにコパイロットを追加する**：パターン1（ツールからコンポーネント）から始める。Vercel AI SDK `useChat` またはCopilotKitを使用する。トップ5〜10のエージェントアクションを事前構築コンポーネントにマッピングする。それを出荷し、測定し、ユーザーが実証的により豊かな構成を必要とする場合にのみカタログを拡張する。

**自然言語からのダッシュボード生成**：json-renderまたはカスタムA2UIカタログでパターン2を使用する。チャートタイプ、メトリックカード、テーブルバリエーションをカバーする8〜15のコンポーネントタイプのカタログを定義する。スキーマをモデルに渡す；レイアウトを構成させる。レンダラーに到達する前に未知のタイプを捕捉する検証を構築する。

**マルチエージェントフロントエンド**：AG-UIを使用したCopilotKitを使用する。イベントストリームはエージェントバックエンド間でリアルタイムストリーミングを処理する；共有状態はエージェント間のハンドオフを処理する；HITLパターンは承認ゲートを処理する。

**ChatGPTまたは別のMCPホスト内で構築する**：MCP Appsを使用する。ツールを、フェッチと推論を行うデータツールとして、およびウィジェットを要求する別のレンダリングツールとして定義する。ビジネスロジックをウィジェットテンプレートから出す。

**MLモデルのデモとデータアプリ（Pythonチーム）**：デモとHuggingFace SpacesにはGradio。より複雑な対話を持つデータアプリにはStreamlit。どちらもJavaScriptは必要ない。

**ビジュアル成果物、シミュレーション、図表**：パターン3（OpenGenerativeUIまたは同等物）を使用する。厳格なiframe CSPを確立する。セキュリティの観点から、出力を信頼できないユーザーコンテンツのように扱う。

---

フレームワークは急速に成熟している。プロトコルの収束（ストリーミングにはAG-UI、カタログ仕様にはA2UI/Open-JSON-UI）はまだ進行中だが、構築に十分な形になってきている。

現在最も重要なエンジニアリング課題は、フレームワーク選択ではない。それはカタログ設計——モデルに何を言わせるかを決定すること、これは技術スキルよりもプロダクトの明確さを必要とする——だ。それは意味的検証——生成されたUIがスキーマ検証を通過するだけでなく、主張するとおりのことをするかをテストすること——だ。そしてそれはアクセシビリティのギャップ——すべてのコンポーネントと、それらのすべての構成が、手書きのUIと同じアクセシビリティ基準を満たすカタログを構築すること——だ。

モデルは、あなたが与える文法内であなたが言うことをする。文法を意図的に設計せよ。
