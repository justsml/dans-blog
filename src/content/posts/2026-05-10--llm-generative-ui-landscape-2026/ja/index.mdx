「ジェネラティブUI」という言葉は、誰が言うかによって少なくとも5つの異なる意味を持つ。

- チャットインターフェースにモデルのツール呼び出しから商品カードを埋め込むもの
- フロントエンドがコンポーネントツリーとしてレンダリングするランタイムJSON仕様
- MCPツールからホストアプリに返されるサンドボックス化されたiframe（チケットの順序付け、ホテル予約から地図レンダリング、チェックアウトウィジェットまで）
- エージェントのステートをフロントエンドにストリーミングするイベントプロトコル
- v0、Lovable、Bolt：デザイン時にReactを書くAIツール

これらは関連する概念だが、スタックの異なるレイヤーに存在し、それぞれ異なるリスクプロファイル、異なる実装コスト、異なる適切なユースケースを持っている。これらを混同すると、あらゆるアーキテクチャの議論が混沌としてしまう。

これは、スタックのどこに手を伸ばすべきかを決めるときに私が欲しいと思う地図だ。

## ジェネラティブUIではないもの

定義の前に、脇に置いておくべき3つのこと：

**デザイン時のコード生成** — v0、Lovable、Bolt、CursorによるReactコンポーネントの構成。これらのツールは開発者がレビューしてコミットするコードを生成する。AIは開発時に実行される。 shippingされるものは、ユーザーの観点からは静的だ。これは素晴らしいカテゴリのツールだ。しかし、「実行時のジェネラティブUI」が意味するものではない。

**AI支援フォーム自動入力** — モデルがコンテキストからフィールド値を埋めること。インターフェースの構造は依然として固定されている。移動するのはコンテンツだけだ。これは有用なパターンだ。ジェネラティブUIではない。

**AIがページに生のHTMLを書き込む** — モデルが`<div>`や`<button>`の文字列を出力し、`innerHTML`や`dangerouslySetInnerHTML`で注入される。これは最も技術的な意味での実行時ジェネラティブUIだ。そして最も危険なバージョンでもあり、このスペースのあらゆる成熟したフレームワークが回避しようとしているものだ。AI生成の生マークアップは、XSSリスク、アクセシビリティ属性の欠落、一貫性のないスタイリング、幻覚的な構造を意味する。この記事の残りは、これより良い方法について書いている。

## 作業定義

実行時のジェネラティブUIとは：**モデルが、会話やタスクのステートに基づいて、ユーザーが見るインターフェースコンポーネントまたはコンポーネントの構成を決定すること。**

言葉ではない。インターフェースだ。

最も簡単なケース：フライト予約アシスタントが`search_flights`ツールを呼び出す。プレーンテキストを返す代わりに（「3つのオプションがあります...」）、選択可能なフライト、シートクラストグル、「予約」ボタンを備えた`<FlightResultsCard>`コンポーネントをレンダリングする。モデルは、ここで構造化されたカードが適切な応答だと判断した。開発者はそのカードの外観と「予約」の動作を決定した。

より複雑なケース：金融分析エージェントがポートフォリオに関する質問を受け取り、`MetricGroup`（主要数値を表示）、`RiskBreakdown`チャート、`ScenarioComparison`テーブル、`PolicyNotice`で応答を構成することを決定する。モデルは承認済みコンポーネントのカタログからそのレイアウトを組み立てた。開発者はすべてのコンポーネントを定義した。モデルはどのコンポーネントを使い、どのデータを配置するかを選んだ。

どちらのケースもジェネラティブUIだ。モデルが持つ構成の自由度が異なり、それが可能な出力の豊かさと、失敗しうる複雑さの両方を決定する。

## 3つのパターン

この領域全体は3つのパターンに収束する。それぞれが異なる出力文法を持つ。

![3つのパターンを示すスペクトル図：左にツール呼び出しのみ（最も安全）、中央にコンポーネントカタログ、右にオープンエンド生成（最も表現力豊か）。](./output-grammar-spectrum.svg)

_すべてのジェネラティブUIの決定はこのスペクトル上の一点だ。左から始めよう。_

### パターン1：ツールtoコンポーネントレンダリング

モデルが名前付きツールを呼び出す。アプリケーションにはツール名からコンポーネントへのマップがある。ツール呼び出しがコンポーネントのレンダリングをトリガーする。

```tsx
// モデルが呼び出す: { name: "show_flight_results", args: { flights: [...] } }

useCopilotAction({
  name: "show_flight_results",
  render: ({ args }) => <FlightResultsCard flights={args.flights} />,
});
```

これは最も安全なパターンだ。なぜなら、レイアウトはモデルから来ないからだ。モデルはコンポーネントをいつ表示するか、どのデータで埋めるかを決定する。開発者は引き続き、コンポーネントコード、ビジュアルデザイン、アクセシビリティの実装、レンダリングロジックのすべてのエッジケースを所有する。

Vercel AI SDKの`useChat`と`tool`ハンドラーはこれを行う。assistant-uiのツールレンダリングもこれを行う。CopilotKitの「Static Generative UI」はこのパターンだ。信頼性の高い本番copilot UIのほとんどはこれを行っている。

**適切な場合**：表示したいものの集合が開発時に列挙できる場合。予約確認、検索結果、アカウントサマリー、承認ウィジェット。シナリオを列挙できるなら、このパターンでカバーできる。

### パターン2：コンポーネントカタログ構成

モデルが、開発者が定義したカタログのコンポーネントを参照する型付きJSONツリーを出力する。フロントエンドには、ツリーを歩いて各コンポーネントをインスタンス化するレンダラーがある。

```json
[
  { "type": "metric_group", "metrics": [
    { "label": "MRR", "value": "$82,400", "delta": "+12%" },
    { "label": "Churn", "value": "2.1%", "delta": "-0.4%" }
  ]},
  { "type": "line_chart", "title": "30日間の成長", "data_ref": "mrr_series" },
  { "type": "insight_callout", "text": "拡大収益が差分を牽引 — 平均席位数は18%増。" }
]
```

モデルはそのレイアウトを構成した。`MetricGroup`、`LineChart`、`InsightCallout`。しかし、各コンポーネントタイプが何を意味するか、どのpropsを受け入れるか、どのようにレンダリングするかはあなたが定義した。モデルが`{ "type": "custom_untested_thing" }`を出力しようとしても、スキーマバリデーションがキャッチし、レンダラーは無視または拒否する。

これは`json-render`、`A2UI`、`Hashbrown`、`OpenUI`、`Tambo`の背後にあるパターンだ。重要なエンジニアリング作業は**カタログ設計** — どのコンポーネントタイプが存在するか、そのスキーマがどのようなものか、モデルが何を構成できるか/できないかを決定することだ。

**適切な場合**：表示したい構造がデータやユーザーのリクエストに基づいて legitimately に変化する場合。数値の中で注目すべきものに応じて適応するダッシュボード。コンテキストに応じて異なるセクションを表示するレポート。エージェントがどのステップにいるかに基づいて変化するワークフローパネル。

### パターン3：オープンエンド生成

モデルがHTML、SVG、Canvas、またはWebGLを書き、厳格なContent Security Policyを持つサンドボックス化されたiframe内でレンダリングされる。

これは、固定コンポーネントカタログでは対応できないものに適している：アルゴリズムの可視化、建築図面、その場のチャート、ジェネラティブアート、教育用シミュレーション。iframeの境界線がここでセキュリティの作業を行っている。それを取り除くと、この記事の冒頭で述べた生のHTML注入問題に戻ってしまう。

`CopilotKit/OpenGenerativeUI`はこのパターンの最良の現在の実装例だ。サンドボックスはスクリプトを削除し、メッセージのやり取りを制限し、生成されたアーティファクトをアプリケーションの特権ステートから遠ざける。

**適切な場合**：任意のビジュアル出力が本当に必要な場合 — 1回限りの説明図、動的シミュレーション、クリエイティブなアーティファクト。トランザクションUIにはこれを使用しない。チェックアウト確認にサンドボックス化されたiframeは必要ない。

### 3つのパターンを超えて：LLMが直接ピクセルを駆動する

これらのパターンのいずれにもきれいに収まらない、新たな4つ目の方向性がある：LLMが**没入型のゲームのような体験**を駆動し、サンドボックス化されたiframeよりも直接的にビジュアル出力を制御するものだ。

ジェネラティブUI内の正統な区別は**iframe HTML vs JSONカタログ**：

- **iframe HTML** — モデルがHTML、SVG、Canvas、またはWebGLを書き、隔離されたサンドボックスでレンダリングする。最大の表現の自由度。セキュリティは完全にiframeの境界に依存する。例：Anthropic Artifacts、OpenGenerativeUI。
- **JSONカタログ** — モデルが開発者が定義したコンポーネントカタログに制約された構造化ペイロードを出力。レンダラーはその仕様から信頼された事前構築コンポーネントをインスタンス化する。モデルは*何を*表示するかを決定。あなたは*どのように*レンダリングするかを決定する。例：json-render、A2UI。

これらを超えて、非常に最近のデモは、モデルがコンポーネントを選択したりサンドボックス化されたHTMLを書いたりするのではなく、より直接的にキャンバスを駆動する第3のモードを示唆している。[TencentのHunyuanWorld](https://arxiv.org/abs/2502.01999)（単一の画像から探索可能な3D環境を生成）や、LLMが実行時にコンポーネントカタログを呼び出すのではなく、マップ、NPC、クエストを生成するゲームアーキテクチャなどのプロジェクトは、モデルがフォームレンダラーというよりもゲームディレクターのような存在になる未来を示唆している。WebGPUを介したブラウザ内LLM推論（[WebLLM](https://mlc.ai/web-llm/)）も同じフロンティアをローカルに押し進めている。

この領域は本当にエキサイティングで、本当に初期段階だ。まだ本番製品を構築するための安定したフレームワークはない。変化があれば、 dedicated な記事でこのアプローチをカバーする。

## プロトコル：AG-UIとA2UI

AG-UIとA2UIはプロトコルレイヤーの2つの主要な標準だ。異なる問題を解決しており、競合他社ではない。

### AG-UI

**GitHub**: [ag-ui-protocol/ag-ui](https://github.com/ag-ui-protocol/ag-ui)

AG-UIは、AIエージェントとフロントエンドアプリケーション間の通信のためのイベントベースのプロトコルだ。約16のイベントタイプを定義している：`TEXT_MESSAGE_START`、`TEXT_MESSAGE_CONTENT`、`TOOL_CALL_START`、`TOOL_CALL_END`、`STATE_SNAPSHOT`、`STATE_DELTA`など。トランスポートはあなた次第 — SSE、WebSockets、Webhooksすべてが機能する。フォーマットは広範な採用を可能にするために意図的に緩い。

AG-UIはUIがどのように見えるかを定義しない。エージェントがフロントエンドと*どのように*通信するかを定義する。Reactアプリがフロントエンドコードを変更せずにLangGraphエージェントをサブスクライブするのと同じ方法でCrewAIエージェントをサブスクライブできるようにするワイヤプロトコルレイヤーだと考えてほしい。

CopilotKitはLangGraphとCrewAIでの作業からAG-UIを作成した。LangChain、Mastra、PydanticAIなどによって採用されている。MicrosoftはAG-UI統合ガイドを公開している。マルチエージェントのフロントエンドを構築し、バックエンドフレームワークをフロントエンドコードから分離する必要がある場合、AG-UIが答えだ。

**混乱を招く1つの明確化**：AG-UIはUIフレームワークではない。何をレンダリングするかは指示しない。エージェントが何かを言った、ツールを呼び出した、共有ステートを更新した、ということを*伝える*だけだ。それに応じて何をレンダリングするかは依然としてあなたの決定だ。

### A2UI

**GitHub**: [google/A2UI](https://github.com/google/A2UI) · 仕様: [a2ui.org](https://a2ui.org/)

A2UIは、エージェントがUIを表示したいときに送信するもののためのGoogleの宣言的仕様だ。AG-UIが「エージェントはどのように通信するか？」に答えるのに対し、A2UIは「エージェントがコンポーネントレイアウトを記述するために使用するフォーマットは何か？」に答える。

A2UIはフラットなJSONLフォーマットを使用する：1行に1つのコンポーネント記述子、それぞれにID、タイプ、データがある。フラットなのは意図的だ。ネストされたツリーは、モデルがストリーミングを開始する前に完全な構造を知る必要がある。フラットなリストなら、モデルが「考える」たびに各コンポーネントを出力できる。つまり、モデルがチャートを追加するかどうかをまだ決定している間でも、フロントエンドは最初のメトリックカードのレンダリングを開始できる。

```jsonl
{"id":"h1","type":"kpi_card","title":"MRR","value":"$82,400","delta":"+12%"}
{"id":"h2","type":"kpi_card","title":"Churn","value":"2.1%","delta":"-0.4%"}
{"id":"c1","type":"line_chart","title":"30日間のMRR","data_ref":"mrr_series"}
{"id":"t1","type":"data_table","cols":["月","MRR","純増"],"data_ref":"monthly"}
```

A2UIはセキュリティを重視している：仕様はデータフォーマットであり、実行可能コードではない。コンポーネントカタログは開発者によって事前定義されている。エージェントはそのカタログ内のタイプのみを参照できる。未知のタイプ名を受信したA2UIレンダラーは無視する。

CopilotKitの「Open-JSON-UI」フォーマットはA2UIと互換性がある。今日コンポーネントカタログの仕様フォーマットを選ぶなら、A2UIが最も広範なクロスプラットフォームのサポートを持つものだ。

**安定性に関する注意**：A2UIは1.0未満 — 2026年5月8日に最後に確認した時点でv0.9 — マイナーバージョン間で仕様の変更が破壊的に発生している。Googleのロードマップに関するコミュニケーションは散発的であり、一部のレンダラー（Lit、Flutter）は仕様の更新に遅れをとっている。今日これの上に構築する場合、仕様の漂流に時間を予算化する必要がある。純粋なWebユースケースでは、json-renderが現在より完全なツールを持っているように見える。A2UIの長期的な利点は、json-renderが持たないクロスプラットフォームの到達範囲（Web、Flutter、SwiftUI、Android）だ。

### MCP Apps

**GitHub**: [modelcontextprotocol](https://github.com/modelcontextprotocol) · 関連: [mcp-ui](https://github.com/MCP-UI-Org/mcp-ui)

MCPは、LLMをツールやデータに接続するためのプロトコルとして始まった。Apps拡張機能により、MCPツールはデータだけでなく、インタラクティブなUIアーティファクト（Reactコンポーネント、フォーム、ダッシュボード、マップ）も返すことができる。

セキュリティモデルは厳格で、それがポイントだ：すべてが権限を削除したサンドボックス化されたiframeでレンダリングされ、テンプレートはホストアプリがレビューできるように事前宣言され、すべての通信は監査可能なJSON-RPCだ。これはツールプロバイダーにとって正しいモデルだ — Shopify MCPサーバーはチェックアウトウィジェットを返すことができる。マッピングサービスは埋め込み可能なマップを返すことができる。ホストアプリはそのウィジェットのコードを所有したり信頼したりしない。

MCP Appsは、UIが*あなたのアプリケーションではなく、ツールプロバイダーに属する場合*に正しい選択だ。あなたのアプリケーションのドメインにあるUIの場合は、パターン1または2に固執する。

## JavaScript/TypeScriptフレームワーク

### CopilotKit

**GitHub**: [CopilotKit/CopilotKit](https://github.com/CopilotKit/CopilotKit) · 例: [CopilotKit/generative-ui](https://github.com/CopilotKit/generative-ui)

CopilotKitは、エージェントネイティブなフロントエンドアプリケーションのための最も完全なフレームワークだ。フルライフサイクルを処理する：AG-UIを介してエージェントバックエンドに接続する、双方向の会話ステートを管理する、ジェネラティブUIコンポーネントをレンダリングする、エージェントとユーザーが同じデータを修正できるようにする共有ステートプランビングを提供する。

3パターンのモデルはCopilotKitのAPIにきれいにマッピングする：
- `render`コールバックを持つ`useCopilotAction` → パターン1
- A2UI/Open-JSON-UIレンダリング → パターン2
- `OpenGenerativeUI`サンドボックス化されたアーティファクト → パターン3

過小評価されている重要なCopilotKitの機能は**共有ステートとヒューマンインザループ**：エージェントはアプリケーションステートの読み書きができ、ユーザーも読み書きでき、変更は双方向に流れる。これが、copilotスタイルのUIを、製品に貼り付けられたチャットボックスではなく、本当のコラボレーションのように感じさせるものだ。

### Vercel AI SDK

**GitHub**: [vercel/ai](https://github.com/vercel/ai) · ドキュメント: [ai-sdk.dev](https://ai-sdk.dev/)

Vercel AI SDKは、AIアプリのための事実上のTypeScriptベースラインだ。ジェネラティブUIについては特に：

**`useObject`** は構造化されたJSONオブジェクトをサーバーから生成されながらストリーミングする。Zodスキーマを定義する。SDKは部分的なJSONを解析し、フィールドが到着するたびに再レンダリングをトリガーする。これはNext.jsアプリでパターン2への最もスムーズなパスだ。

```tsx
const { object: dashboard } = useObject({
  api: "/api/generate-dashboard",
  schema: z.object({
    title: z.string(),
    metrics: z.array(z.object({ label: z.string(), value: z.number() })),
    insights: z.array(z.string()),
  }),
});
```

**`useChat`とツールハンドラー** → パターン1。モデルがツールを呼び出す。ツール名をコンポーネントにマッピングする。

**AI Elements**（[elements.ai-sdk.dev](https://elements.ai-sdk.dev/)）は、SDKと組み合わせて使用する準備ができているUIプリミティブを提供する。

**混乱した軌道に関する注意**：2024年10月、Vercelは[GitHub Discussion #3251](https://github.com/vercel/ai/discussions/3251)で、AI SDK RSC（SDK 3.0でヘッドラインの「ジェネラティブUI」機能として宣伝されていたReact Server Componentsストリーミングパターン）が「いくつかの長年の制限」のため無期限に一時停止されたと発表した。RSCストリーミングを中心に製品戦略を構築していたチームは不意を突かれた。`generateObject`/`streamObject` APIも後にSDK 6.0で非推奨となった。AI SDK RSCからの推奨移行は、上記の`useObject`パターン、またはカタログベースの生成のためのjson-renderだ。

### assistant-ui

**GitHub**: [assistant-ui/assistant-ui](https://github.com/assistant-ui/assistant-ui)

assistant-uiは、本番品質のチャットインターフェースを構築するためのコンポーザブルなReactプリミティブのセットだ。ポリッシュされたチャットUX（メッセージバブル、ストリーミングトークン、コピー/編集/再生成アクション、思考ステート）が必要で、独自のバックエンドと独自のツールレンダリングを持ち込みたい場合の正しい答えだ。

任意のバックエンド（OpenAI、Anthropic、ローカルモデル、カスタムエンドポイント）と組み合わせて機能し、 familiar なスロット/レンダリングプロップモデルを介してツール呼び出しレンダリングを処理する。

### json-render

**GitHub**: [vercel-labs/json-render](https://github.com/vercel-labs/json-render) · ドキュメント: [json-render.dev](https://json-render.dev/)

json-renderは、独断的でバッテリー同梱のアプローチでパターン2を運用化する。事前構築されたコンポーネントカタログ（Zodスキーマを持つshadcn/uiコンポーネント）、レンダラー、モデルがスキーマによってカタログに制約されるタイトな生成ループが得られる。

特徴的な機能：
- **マルチターゲットレンダリング**：同じJSON仕様をReact Webアプリ、React Nativeモバイルアプリ、PDF、HTMLメール、またはRemotionビデオにレンダリングできる。これはレポートにとって本当に有用だ。
- **プログレッシブレンダリング**：コンポーネントは、完全な仕様が到着した後ではなく、モデルがストリーミングするたびに表示される。
- **タイトなスキーマ制約**：カタログは、モデルが有効だが未知のコンポーネントタイプを幻覚できないように設計されている。

ダッシュボードやレポート生成機能を構築し、独自のカタログを設計するインフラ作業をスキップしたい場合、json-renderがWebアプリへの最速パスだ。

**モメンタムについて**：json-renderは2026年初頭にVercel Labsからローンチし、標準的なReact/Next.jsプロジェクトで即座に有用であるため、Web開発者の注目を急速に集めているように見える。とは言え、json-renderはまだ1.0未満であり、json-renderとA2UIの関係はまだ調整中だ — VercelはA2UI互換の出力を実験しているため、収束は可能だ。クロスプラットフォーム（ネイティブモバイル、複数フレームワーク）の場合、A2UIがより良い長期の賭けだ。

### Hashbrown

**GitHub**: [liveloveapp/hashbrown](https://github.com/liveloveapp/hashbrown)

Hashbrownは独特のアプローチを取る：別のAIインターフェースレイヤーを構築するのではなく、既存のReactまたはAngularアプリにAIコンポーネント選択を直接埋め込む。アプリのコンポーネントをLLMに公開する。LLMはどれをレンダリングするかを選択し、クライアントサイドツールを呼び出すことができる。

「チャット」ではない製品表面にインテリジェンスを注入したい場合に正しいツールだ — レイアウトを適応させる製品ページ、適切なオプションを浮き彫りにする設定パネル、次のステップを提案するワークフローエディター。

### OpenUI

**GitHub**: [thesysdev/openui](https://github.com/thesysdev/openui) · ドキュメント: [openui.com](https://www.openui.com/)

OpenUIは、JSONをプログレッシブレンダリングとトークン効率のために設計された行指向のコードのようなフォーマット（「OpenUI Lang」）に置き換える。主張はおおよそ、複雑なレイアウトの場合、同等のJSONより約67%少ないトークンだ。

トレードオフはエコシステムの成熟度 — OpenUIはより新しく、ツールはJSONベースのアプローチより薄い。しかし、トークンコストが意味のある制約であり、高頻度で複雑なレイアウトを生成している場合、フォーマット効率は本物だ。

### Tambo

**GitHub**: [tambo-ai/tambo](https://github.com/tambo-ai/tambo)

Tamboはステートフルなコンポーネント選択に焦点を当てる：AIはコンポーネントを選択し、クライアントサイドツールを介してそれらと対話でき、会話全体でコンポーネントステートを維持する。UI要素がターン間で持続するユースケースに適している — ユーザーがフィルターコンポーネントをいじっている間にAIがフィルターされたデータについて推論を続ける場合など。

## Pythonレイヤー

Pythonエコシステムは、AIインターフェースに異なるアプローチを取る。これらのツールは、MLモデルのデモ、データアプリケーション、内部ツール向けに最適化されている — エージェント駆動のレイアウト構成を持つ本番コンシューマーアプリ向けではない。

それは否定ではない。正しいユースケースの場合、GradioとStreamlitは必要な唯一のツールだ。

### Gradio

**GitHub**: [gradio-app/gradio](https://github.com/gradio-app/gradio) · PyPI: `gradio`

Gradioのコアバリュー：Python関数を書く。GradioがそれをWeb UIでラップする。`Interface`クラスは画像分類器のために3行。`ChatInterface`はチャットボットのために10行。`Blocks`は必要なときに細粒度のレイアウト制御を提供する。

Gradioの「ジェネラティブUI」はPython開発者によって定義されており、モデルによって定義されているわけではない。コンポーネントの表示と設定はモデルの出力に基づいて動的に変化できるが、コンポーネントカタログは静的だ — モデルにレイアウトを構成するように依頼しているわけではない。

GradioはHuggingFace SpacesとMLデモエコシステムのデフォルトだ。月間数百万回のダウンロードがあり、AIデモランドスケープの大部分を占めている。

**Gradioに手を伸ばす場合**：Python開発者としてMLモデルのデモ、研究プロトタイプ、または内部ツールを構築しており、JavaScriptに触れたくない場合。

### Streamlit

**GitHub**: [streamlit/streamlit](https://github.com/streamlit/streamlit)

Streamlitのモデルはより独断的だ：Pythonスクリプトが各インタラクションでエンドツーエンドで実行される。`st.chat_message()`、`st.dataframe()`、`st.plotly_chart()`を呼び出す。フレームワークがレイアウトを処理する。

フルスクリプト再実行モデルは非効率的に聞こえるが、会話履歴を蓄積するAIチャットボットにとって驚くほどエルゴノミックだ — スクリプト全体が再実行され、会話履歴はセッションステートにあり、出力は決定的だ。Streamlitは今、主要なLLMプロバイダーのほとんどに対するファーストパーティのサポートを持ち、Snowflake Cortexとネイティブに統合している。

**Streamlitに手を伸ばす場合**：PythonでAI搭載のデータアプリ、内部レポートツール、またはMLバックのダッシュボードを構築し、可能な限り最もシンプルなデプロイパスを望む場合。

### LangChainとHaystack

これらはバックエンドオーケストレーションフレームワークであり、UIフレームワークではない。正直なジェネラティブUIスタックマップに登場するのは、通常、フロントエンドに送信される前に構造化出力が生成されるレイヤーだからだ。

**LangChain**（[langchain-ai/langchain](https://github.com/langchain-ai/langchain)）：任意のLLMの`.with_structured_output()`はPydantic制約のJSON生成を提供する。自動スキーマ生成を持つ`@tool`デコレーターは、モデルが呼び出すことができるツールを定義する最もクリーンな方法だ。LangChainは構造化された結果を、使用しているフロントエンドレイヤーにフィードする。

**Haystack**（[deepset-ai/haystack](https://github.com/deepset-ai/haystack)）：強いRAGサポートを備えたモジュラーパイプラインアーキテクチャ。HayhooksはHaystackパイプラインをHTTPエンドポイントとしてラップする — MCP互換のエンドポイントを含む。ジェネラティブUIに検索のバックボーンが必要な場合、Haystackのパイプラインアーキテクチャがそれをきれいに処理する。

どちらのフレームワークもUIティアを所有していない。それらは、フロントエンド（パターン1、2、または3）がレンダリングするデータを生成する。

## エコシステムの速度と不安定な地面

このスペースは急速に動いており、いくつかのプロジェクトはコードとともに混乱したコミュニケーションを出荷している。最終確認2026年5月8日。ここでのプロジェクトステートの注意は、恒久的な判決ではなく、タイムスタンプ付きの読み取りとして扱う。

**Vercel AI SDK RSC** は、SDK 3.0がローンチしたときの目玉ジェネラティブUI機能だった。Vercelは2024年10月にその開発を一時停止した（[Discussion #3251](https://github.com/vercel/ai/discussions/3251)）、React Server Componentsのアーキテクチャ上の制限を理由に、近い将来の解決策がない。これの上に構築していたチームは当然フラストレーションを感じた。まだドキュメントにあるが、推奨パスではない。`useObject`がそれだ。

**json-render**（Vercel Labs）は新しい方向性だ — RSCのカップリング問題を回避するカタログベースのフレームワーク非依存の代替。1.0未満であり、React/Web開発者の間で強い初期の関心を集めているように見える。おそらくDXの理由：json-renderは標準的なReact/Next.jsプロジェクトで即座に有用だが、A2UIのクロスプラットフォームの範囲はセットアップの摩擦を追加する。両方の仕様が成熟するにつれて、これがどのように落ち着くかは本当に不明だ。Vercelはjson-renderでA2UI互換性を調査しており、収束が可能であることを示唆している。

**A2UI**（Google）は1.0未満（最後に確認した時点でv0.9）で、マイナーバージョン間で破壊的変更があり、Googleのロードマップに関するコミュニケーションに一貫性がない。json-renderが対処しないクロスプラットフォームの到達範囲（Web + Flutter + SwiftUI）にとって正しい選択であり、意味のあるエンタープライズのサポートがある。純粋なWebプロジェクトの場合、今のところDXは粗い。

**AG-UI**（CopilotKit）も1.0未満だ。最も一般的な混乱：名前がUIフレームワークのように聞こえること。そうではない — トランスポートプロトコルだ。AG-UIは、エージェントバックエンドとフロントエンドの間でイベントがどのように流れるかを定義する。それに応じて何をレンダリングするかは依然としてあなたの決定だ。そのメンタルモデルは堅実で広く採用されているが、1.0未満の仕様は、エッジケースがまだ解決されていることを意味する。

実際的な意味：**ここでの主要なプレイヤーはすべて1.0未満だ**。APIの変更を計画する。パターン — ツールtoコンポーネント、カタログ構成、サンドボックス生成 — は構築するのに十分に安定している。特定のプロトコルの選択はそうではない。

## コンポーネントカタログ設計：本当のエンジニアリング作業

パターン2の興味深い複雑さのほとんどはレンダラーではなく、カタログにある。

カタログは**スキーマとしてエンコードされた製品決定**だ。これに答える：このドメインで意味のあるUIオブジェクトは何か？「どのReactコンポーネントが存在するか？」ではなく、「このコンテキストのユーザーが実際に見て対話する必要があるものは何か？」

**細かすぎる失敗モード**：`Row`、`Column`、`Text`、`Button`、`Icon`を公開する。これで、モデルはフロントエンドエンジニアになる必要がある。デザインシステムに一致しない mediocre なレイアウトを生成し、空のステートを見逃し、アクセシビリティのないマークアップを生成し、カタログの出力を製品のビジュアル言語に制約するものがないため、応答ごとに変化する。

**粗すぎる失敗モード**：`WeatherCard`、`FlightCard`、`HotelCard`を公開する。モデルが事前焼き付けカードにマッピングされないものをユーザーが要求した場合に適応できない。テキストにフォールバックする。

**有用な中間**：制約されたスロットを持つドメインレベルのコンポーネント。

旅行アプリのカタログは次のようになる可能性がある：

```
TripSummary         — 旅程の概要
FlightOptionList    — 価格付きの選択可能なフライトオプション
HotelComparison     — 並列ホテルカード
TravelerForm        — 旅行者情報の収集
PolicyNotice        — 規制/運賃規則の呼び出し
BookingConfirmation — アクションボタン付きの最終確認
```

金融アプリのカタログは次のようになる可能性がある：

```
PortfolioSnapshot   — 主要ポジションとP&L
TransactionTable    — フィルター可能、ページネーションされたトランザクション
RiskBreakdown       — 配分とボラティリティメトリクス
ScenarioComparison  — 並列シナリオモデリング
ApprovalGate        — 人間の確認が必要なアクション
```

カタログは製品の語彙のように聞こえる。UXの決定、アクセシビリティの要件、空のステートの処理、危険なアクションのパターンをコンポーネントコードにエンコードする。モデルはそれらのピースを配置できる。あなたは依然として、各ピースがどのように見え、何をすることを許可されているかを決定する。

**幻覚を減らすスキーマ設計ルール**：

1. 列挙値を短く、明らかに保つ。`"type": "bar_chart"`。`"type": "data-visualization-bar-type-vertical"`ではない。
2. 無効な構成を不可能にする。`PolicyNotice`がレイアウトの最後にのみ表示できる場合、どこにでも表示できるものと同じスキーマレベルに配置しない。
3. 必須フィールドを寛大に使用する。オプションのフィールドは、モデルが省略し、レンダラーがnullとして処理する必要があるフィールドだ。
4. 出荷前に実際のプロンプトに対してカタログをテストする。生成された仕様を保存する。スキーマ違反、幻覚されたフィールド値、技術的には有効だが意味的に間違っている構成をチェックする。

## 一般的な罠

**罠：有効なJSONを安全な行動として扱う。** スキーマバリデーションは構造を確認する。ボタンのアクションがラベルと一致するかどうか、合計が派生元のデータと一致するかどうか、UIコンポーネントがユーザーが予期しないことをしていないかどうかについては何も言わない。生成されたUI仕様は、セマンティックレビューが必要だ。少なくとも、破壊的なアクションは確認コンポーネントを必要とし、それらのコンポーネントのラベルはトリガーされるアクションに対してテストされるべきだ。

**罠：設計プリミティブではなく製品プリミティブを公開する。** モデルが16pxまたは20pxのパディングを使用するかどうかを決定する必要がある場合、間違った抽象化レベルを与えている。ドメインコンポーネントは製品の味をエンコードする必要がある。モデルはプレゼンテーションの詳細を管理するのではなく、振る舞いを構成するべきだ。

**罠：静的UIで済む場所でジェネラティブUIを使用する。** 表示したいものの構造が開発時に列挙できる場合 — そして通常は列挙できる — 事前構築されたコンポーネントを持つパターン1の方が速く、安全で、一貫性がある。ジェネラティブUIは、構造がデータやタスクのコンテキストに基づいて legitimately に変化する場合にのみ、その複雑さを正当化する。

**罠：アクセシビリティをスキップする。** LLMはWCAG違反を幻覚する。インタラクティブな要素に`role="region"`を出力し、ラベルのないフォームを生成し、WCAG AAに失敗するコントラスト比を生成する。コンポーネントライブラリは完全にアクセシビリティがあるかもしれない。それらのコンポーネントのAI生成の構成は自動的にアクセシビリティがあるわけではない。コンポーネントを分離してテストするのではなく、レンダリングパス全体をテストする。

**罠：プロトコルとフレームワークを混同する。** AG-UIはフロントエンドフレームワークではない。A2UIはReactライブラリではない。ワイヤフォーマットとイベントプロトコルだ。それらを実装するには、まだフロントエンドフレームワークが必要だ。CopilotKitはAG-UIとA2UIを実装する。json-renderはA2UI/Open-JSON-UIカタログパターンを実装する。これらは異なるレイヤーだ。

## ユースケース別の推奨事項

**既存のSaaSアプリにcopilotを追加する**：パターン1（ツールtoコンポーネント）から始める。Vercel AI SDK `useChat`またはCopilotKitを使用する。上位5〜10のエージェントアクションを事前構築されたコンポーネントにマッピングする。それを shipped し、測定し、ユーザーがより豊かな構成を本当に必要とする場合にのみカタログを拡張する。

**自然言語からのダッシュボード生成**：json-renderまたはカスタムA2UIカタログでパターン2を使用する。チャートタイプ、メトリックカード、テーブルバリアントをカバーする8〜15のコンポーネントタイプのカタログを定義する。スキーマをモデルにフィードする。レイアウトを構成させる。レンダラーに到達する前に未知のタイプをキャッチするバリデーションを構築する。

**マルチエージェントフロントエンド**：AG-UIでCopilotKitを使用する。イベントストリームはエージェントバックエンド間でリアルタイムストリーミングを処理する。共有ステートはエージェント間のハンドオフを処理する。HITLパターンは承認ゲートを処理する。

**ChatGPTまたは他のMCPホスト内で構築する**：MCP Appsを使用する。ツールをフェッチと推論を行うデータツールとして定義し、ウィジェットを要求する別個のレンダリングツールとして定義する。ビジネロジックをウィジェットテンプレートから除外する。

**MLモデルのデモとデータアプリ（Pythonチーム）**：デモとHuggingFace SpacesにはGradio。より複雑なインタラクションを持つデータアプリにはStreamlit。どちらもJavaScriptを必要としない。

**ビジュアルアーティファクト、シミュレーション、図**：パターン3（OpenGenerativeUIまたは同等のもの）を使用する。厳格なiframe CSPを確立する。セキュリティの観点から、出力を信頼されていないユーザーコンテンツとして扱う。

---

フレームワークは急速に成熟している。プロトコルの収束（ストリーミングのためのAG-UI、カタログ仕様のためのA2UI/Open-JSON-UI）はまだ進行中だが、形状は構築するのに十分に明確になりつつある。

今最も重要なエンジニアリングの課題は、フレームワークの選択ではない。カタログ設計 — モデルが何を言うことを許可されるかを決定すること。これは技術的なスキルよりも製品の明確さを必要とする。セマンティックバリデーション — 生成されたUIが主張することを行うことをテストする。スキーマバリデーションに合格するだけでなく。アクセシビリティのギャップ — すべてのコンポーネント、およびコンポーネントのすべての構成が、手書きのUIに課すアクセシビリティの基準を満たすカタログを構築すること。

モデルは、与えた文法内で、指示されたことを行う。文法を意図的にする。
