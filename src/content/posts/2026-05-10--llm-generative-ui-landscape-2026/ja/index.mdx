「ジェネレーティブUI」という言葉は、誰が言うかで少なくとも5つの異なる意味を持つ。

- モデルのツール呼び出しからプロダクトカードを埋め込むチャットインターフェース
- フロントエンドがコンポーネントツリーとしてレンダリングするランタイムJSON仕様
- MCPツールからホストアプリに返されるサンドボックス化されたiframe（チケット注文、ホテル予約から地図レンダリング、チェックアウトウィジェットまで）
- エージェントの状態をフロントエンドにストリーミングするイベントプロトコル
- v0、Lovable、Bolt：デザイン時にReactを記述するAIツール

これらは関連する概念だが、スタックの異なる層に存在し、異なるリスクプロファイル、異なる実装コスト、異なる適切なユースケースを持つ。これらを混同すると、すべてのアーキテクチャ議論が混乱する。

これは、スタックのどこに手を伸ばすべきかを決める際に私が欲しい地図だ。

## ジェネレーティブUIではないもの

定義の前に、3つの除外項目がある。

**デザイン時のコード生成** — v0、Lovable、Bolt、CursorがReactコンポーネントを構成する。これらのツールは開発者が確認してコミットするコードを生成する。AIは開発時に実行される。出荷されるものはユーザーの視点からは静的だ。これは素晴らしいツールカテゴリだ。しかし「ランタイムでのジェネレーティブUI」が意味するものではない。

**AIアシストによるフォーム自動入力** — モデルがコンテキストからフィールド値を埋める。インターフェースの構造はまだ固定で、動くのはコンテンツだけだ。これは便利なパターンだ。ジェネレーティブUIではない。

**AIが生のHTMLをページに書き込む** — モデルが`<div>`や`<button>`文字列を出力し、`innerHTML`や`dangerouslySetInnerHTML`で注入される。これは最も技術的な意味でランタイムジェネレーティブUIである。同時に最も危険なバージョンであり、このスペースの成熟したフレームワークがすべて回避しようとしているものでもある。AI生成の生のマークアップはXSSリスク、アクセシビリティのない属性、一貫性のないスタイル、幻覚による構造を意味する。この記事の残りは、これより良くやる方法についてだ。

## 実用的な定義

ランタイムでのジェネレーティブUIとは：**モデルが会話やタスクの状態に基づいて、ユーザーが目にするインターフェースコンポーネントまたはコンポーネント構成を決定する**こと。

言葉ではなく、インターフェースを。

最もシンプルなケース：航空券予約アシスタントが`search_flights`ツールを呼び出す。「3つのオプションがあります…」というプレーンテキストの代わりに、選択可能なフライト、座席クラストグル、そして「予約」ボタンを持つ`<FlightResultsCard>`コンポーネントをレンダリングする。モデルがここで構造化カードが適切な応答であると決定した。開発者はそのカードがどのように見え、「予約」が何をするかを決定した。

より複雑なケース：金融分析エージェントがポートフォリオに関する質問を受け、主要な数字を示す`MetricGroup`、リスク内訳チャート`RiskBreakdown`、シナリオ比較テーブル`ScenarioComparison`、ポリシー通知`PolicyNotice`で構成された応答を組み立てる。モデルが事前承認されたコンポーネントのカタログからそのレイアウトを構成した。開発者は各コンポーネントタイプを定義した。モデルはどれを使い、そこに何のデータを入れるかを選択した。

どちらのケースもジェネレーティブUIだ。モデルの構成自由度が異なり、それが取り得る出力の豊かさと、問題が起きうる複雑さの両方を決定する。

## 3つのパターン

このスペース全体は3つのパターンに集約される。それぞれに異なる出力文法がある。

![左から右へ3つのパターンを示すスペクトラム図：ツール呼び出しのみ（最も安全）、コンポーネントカタログ（中間）、自由な生成（最も表現力が高い）。](./output-grammar-spectrum.svg)

_ジェネレーティブUIのすべての決定はこのスペクトラム上の点だ。左から始めよう。_

### パターン1：ツール→コンポーネントレンダリング

モデルが名前付きツールを呼び出す。アプリケーションにはツール名からコンポーネントへのマッピングがある。ツール呼び出しがコンポーネントレンダリングをトリガーする。

```tsx
// モデルが呼び出し: { name: "show_flight_results", args: { flights: [...] } }

useCopilotAction({
  name: "show_flight_results",
  render: ({ args }) => <FlightResultsCard flights={args.flights} />,
});
```

これが最も安全なパターンである理由は、レイアウトがモデルから決して来ないからだ。モデルが決定するのはコンポーネントを*いつ*表示するかと*何のデータ*を投入するかだ。コンポーネントのコード、ビジュアルデザイン、アクセシビリティ実装、レンダリングロジックのエッジケースはすべて開発者が所有する。

Vercel AI SDKの`useChat`とツールハンドラーがこれを行う。assistant-uiのツールレンダリングがこれを行う。CopilotKitの「Static Generative UI」がこのパターンだ。確実に動作する本番コパイロットUIのほとんどはこれをやっている。

**適しているケース**：表示したいもののセットが開発時に知り得る場合。予約確認、検索結果、アカウントサマリー、承認ウィジェット。シナリオを列挙できるなら、このパターンでカバーできる。

### パターン2：コンポーネントカタログ構成

モデルが開発者が定義したカタログのコンポーネントを参照する型付きJSONツリーを出力する。フロントエンドにはツリーを走査して各コンポーネントをインスタンス化するレンダラーがある。

```json
[
  { "type": "metric_group", "metrics": [
    { "label": "MRR", "value": "$82,400", "delta": "+12%" },
    { "label": "Churn", "value": "2.1%", "delta": "-0.4%" }
  ]},
  { "type": "line_chart", "title": "30-day growth", "data_ref": "mrr_series" },
  { "type": "insight_callout", "text": "Expansion revenue driving the delta — avg seat count up 18%." }
]
```

モデルがそのレイアウトを構成した。`MetricGroup`、`LineChart`、`InsightCallout`。しかし各コンポーネントタイプが何を意味し、どんなpropsを受け取り、どうレンダリングするかはあなたが定義した。モデルが`{ "type": "custom_untested_thing" }`を出力しようとしたら、スキーマ検証がそれを捕捉し、レンダラーは無視または拒否する。

これが`json-render`、`A2UI`、`Hashbrown`、`OpenUI`、`Tambo`の背後にあるパターンだ。重要なエンジニアリング作業は**カタログ設計**だ — どのコンポーネントタイプが存在し、そのスキーマがどう見え、モデルに構成を許可する・しないを決定すること。

**適しているケース**：表示したいものの構造がデータやユーザーのリクエストに基づいて正当に変動する場合。数字の中で重要なものに適応するダッシュボード。コンテキストに応じて異なるセクションを表示するレポート。エージェントがいるステップに基づいて変わるワークフローパネル。

### パターン3：自由な生成

モデルがHTML、SVG、Canvas、またはWebGLを記述し、厳格なContent Security Policyを持つサンドボックス化されたiframe内でレンダリングされる。

これは固定のコンポーネントカタログでは対応できないものに適している：アルゴリズムの可視化、アーキテクチャ図、アドホックなチャート、ジェネレーティブアート、教育用シミュレーション。ここでセキュリティの役割を果たしているのはiframeの境界だ。それを取り除けば、この記事の冒頭の生のHTML注入問題に戻る。

`CopilotKit/OpenGenerativeUI`がこのパターンの現在の最良のリファレンス実装だ。サンドボックスはスクリプトを除去し、メッセージパッシングを制限し、生成されたアーティファクトをアプリケーションの特権状態から遠ざける。

**適しているケース**：本当に任意のビジュアル出力が必要な場合 — 一回限りの説明図、動的シミュレーション、クリエイティブなアーティファクト。これをトランザクショナルUIに使わないこと。チェックアウト確認にサンドボックス化されたiframeは不要だ。

### 3つのパターンを超えて：LLMがピクセルを直接駆動する

これらのパターンのどれにもきれいに当てはまらない第4の方向が現れつつある：LLMがサンドボックス化されたiframeよりも直接的に**没入型のゲーム的な体験**を駆動する。

ジェネレーティブUIにおけるカノニカルな区別は**iframe HTML vs JSONカタログ**だ：

- **iframe HTML** — モデルがHTML、SVG、Canvas、またはWebGLを記述し、分離されたサンドボックスでレンダリングする。最大の表現の自由度。セキュリティは完全にiframeの境界に依存する。例：Anthropic Artifacts、OpenGenerativeUI。
- **JSONカタログ** — モデルが開発者定義のコンポーネントカタログに制約された構造化ペイロードを出力する。レンダラーがその仕様から信頼できる事前構築コンポーネントをインスタンス化する。モデルが*何を*表示するかを決定し、*どう*レンダリングするかはあなたが決定する。例：json-render、A2UI。

これらを超えて、非常に最近のデモは、モデルがコンポーネントを選択したりサンドボックス化されたHTMLを書いたりするのではなく — より直接的にキャンバスを駆動する第3のモードを示唆している。[TencentのHunyuanWorld](https://arxiv.org/abs/2502.01999)（1枚の画像から探索可能な3D環境を生成）や、コンポーネントカタログを呼び出すのではなくLLMが実行時にマップ、NPC、クエストを生成するゲームアーキテクチャなどのプロジェクトは、モデルがフォームレンダラーよりもゲームディレクターに近い未来を示唆している。WebGPU経由のブラウザ内LLM推論（[WebLLM](https://mlc.ai/web-llm/)）も同じフロンティアをローカルに押し進めている。

この領域は真にエキサイティングであり、真に初期段階だ。本番プロダクトで構築できる安定したフレームワークはまだない。これが変わったら、別途の記事で取り上げる。

## プロトコル：AG-UIとA2UI

AG-UIとA2UIはプロトコル層の2つの主要な標準だ。異なる問題を解決し、競合ではない。

### AG-UI

**GitHub**: [ag-ui-protocol/ag-ui](https://github.com/ag-ui-protocol/ag-ui)

AG-UIはAIエージェントとフロントエンドアプリケーション間の通信のためのイベントベースプロトコルだ。約16種のイベントタイプを定義する：`TEXT_MESSAGE_START`、`TEXT_MESSAGE_CONTENT`、`TOOL_CALL_START`、`TOOL_CALL_END`、`STATE_SNAPSHOT`、`STATE_DELTA`など。トランスポートは自由に選べる — SSE、WebSocket、Webhookすべて動作する。広範な採用を可能にするために意図的に緩いフォーマットだ。

AG-UIはUIがどう見えるかを定義しない。エージェントがフロントエンド*と*どう通信するかを定義する。ReactアプリがLangGraphエージェントとCrewAIエージェントを同じ方法で購読できるようにするワイヤープロトコル層と考えればよい。

CopilotKitがLangGraphやCrewAIとの作業からAG-UIを作り出した。LangChain、Mastra、PydanticAIなどが採用している。MicrosoftはAG-UI統合ガイドを公開している。マルチエージェントフロントエンドを構築し、バックエンドフレームワークをフロントエンドコードから切り離す必要があるなら、AG-UIが答えだ。

**よくある誤解**：AG-UIはUIフレームワークではない。何をレンダリングするかは指示しない。エージェントが何かを言ったこと、ツールを呼び出したこと、共有状態を更新したことだけを通知する。それに応じて何をレンダリングするかは、依然としてあなたの決定だ。

### A2UI

**GitHub**: [google/A2UI](https://github.com/google/A2UI) · 仕様: [a2ui.org](https://a2ui.org/)

A2UIはGoogleの、エージェントがUIを表示したい際に送信する内容の宣言型仕様だ。AG-UIが「エージェントはどう通信するか？」に答えるのに対し、A2UIは「エージェントはコンポーネントレイアウトを記述するためにどんなフォーマットを使うか？」に答える。

A2UIはフラットなJSONLフォーマットを使う：1行に1つのコンポーネント記述子、それぞれがID、タイプ、データを持つ。フラットなのは意図的だ。ネストされたツリーは、ストリーミングを開始する前にモデルが完全な構造を知る必要がある。フラットなリストは、モデルが各コンポーネントを「考え」ながら出力でき、モデルがチャートを追加するかどうかをまだ決定している間にフロントエンドが最初のメトリックカードをレンダリングし始められる。

```jsonl
{"id":"h1","type":"kpi_card","title":"MRR","value":"$82,400","delta":"+12%"}
{"id":"h2","type":"kpi_card","title":"Churn","value":"2.1%","delta":"-0.4%"}
{"id":"c1","type":"line_chart","title":"30-day MRR","data_ref":"mrr_series"}
{"id":"t1","type":"data_table","cols":["Month","MRR","Net New"],"data_ref":"monthly"}
```

A2UIはセキュリティを重視している：仕様はデータフォーマットであり、実行可能コードではない。コンポーネントカタログは開発者が事前定義する。エージェントはそのカタログ内のタイプのみを参照できる。A2UIレンダラーが未知のタイプ名を受け取った場合、それを無視する。

CopilotKitの「Open-JSON-UI」フォーマットはA2UIと互換性がある。今日コンポーネントカタログの仕様フォーマットを選ぶなら、A2UIが最も幅広いクロスプラットフォームの支持を得ている。

**安定性に関する注記**：A2UIはプレ1.0（2026年5月8日時点でv0.9）であり、マイナーバージョン間で破壊的仕様変更を出荷している。Googleからのロードマップに関するコミュニケーションは不規模で、一部のレンダラー（Lit、Flutter）は仕様更新に遅れている。今日これに基づいて構築するなら、仕様ドリフトのための時間を予算に組み込むこと。純粋なWebユースケースでは、json-renderの方がツールが完成しているように見える。A2UIの長期的な利点は、json-renderが持たないクロスプラットフォームの到達範囲（Web、Flutter、SwiftUI、Android）だ。

### MCP Apps

**GitHub**: [modelcontextprotocol](https://github.com/modelcontextprotocol) · 関連: [mcp-ui](https://github.com/MCP-UI-Org/mcp-ui)

MCPは元々、LLMをツールとデータに接続するためのプロトコルとして始まった。Apps拡張により、MCPツールはデータだけでなく、インタラクティブなUIアーティファクト（Reactコンポーネント、フォーム、ダッシュボード、地図）も返せるようになった。

セキュリティモデルは厳格であり、それがポイントだ：すべてサンドボックス化されたiframeでレンダリングされ権限は剥奪される、テンプレートは事前宣言されるためホストアプリがレビューできる、すべての通信は監査可能なJSON-RPCだ。これはツールプロバイダーにとって正しいモデルだ — ShopifyのMCPサーバーがチェックアウトウィジェットを返せるし、マッピングサービスが埋め込み可能な地図を返せる。ホストアプリはそのウィジェットのコードを所有も信頼もしない。

UIが*ツールプロバイダーに属する*場合、MCP Appsが正しい選択だ。あなたのアプリケーションのドメインに属するUIには、パターン1または2を使う。

## JavaScript/TypeScriptフレームワーク

### CopilotKit

**GitHub**: [CopilotKit/CopilotKit](https://github.com/CopilotKit/CopilotKit) · 例: [CopilotKit/generative-ui](https://github.com/CopilotKit/generative-ui)

CopilotKitはエージェントネイティブなフロントエンドアプリケーション向けの最も完全なフレームワークだ。エージェントバックエンドへのAG-UI接続、双方向会話状態の管理、ジェネレーティブUIコンポーネントのレンダリング、エージェントとユーザーが同じデータを変更できるようにする共有状態の配管を含む完全なライフサイクルを扱う。

3つのパターンモデルはCopilotKitのAPIにきれいにマッピングされる：
- `render`コールバック付きの`useCopilotAction` → パターン1
- A2UI/Open-JSON-UIレンダリング → パターン2
- `OpenGenerativeUI`サンドボックス化されたアーティファクト → パターン3

過小評価されているCopilotKitの重要な機能は**共有状態とヒューマン・イン・ザ・ループ**だ：エージェントはアプリケーションの状態を読み書きでき、ユーザーも読み書きでき、変更は双方向に流れる。これがコパイロットスタイルのUIを、プロダクトにステープルされたチャットボックスではなく真のコラボレーションのように感じさせるものだ。

### Vercel AI SDK

**GitHub**: [vercel/ai](https://github.com/vercel/ai) · ドキュメント: [ai-sdk.dev](https://ai-sdk.dev/)

Vercel AI SDKはAIアプリのための事実上のTypeScriptベースラインだ。ジェネレーティブUIに関して具体的には：

**`useObject`** は、生成されるにつれてサーバーから構造化JSONオブジェクトをストリーミングする。Zodスキーマを定義し、SDKが部分JSONをパースし、フィールドが到着するたびに再レンダリングをトリガーする。これがNext.jsアプリでのパターン2への最もスムーズなパスだ。

```tsx
const { object: dashboard } = useObject({
  api: "/api/generate-dashboard",
  schema: z.object({
    title: z.string(),
    metrics: z.array(z.object({ label: z.string(), value: z.number() })),
    insights: z.array(z.string()),
  }),
});
```

**`useChat`とツールハンドラー** → パターン1。モデルがツールを呼び出し、ツール名をコンポーネントにマッピングする。

**AI Elements**（[elements.ai-sdk.dev](https://elements.ai-sdk.dev/)）はSDKとペアリングするための完成済みUIプリミティブを提供する。

**ここでのややこしい軌跡に関する注記**：2024年10月、Vercelは[GitHub Discussion #3251](https://github.com/vercel/ai/discussions/3251)で、AI SDK RSC — SDK 3.0のヘッドライン「ジェネレーティブUI」機能として推進されていたReact Server Componentsストリーミングパターン — を「いくつかの長年の制限」があり近い解決策がないとして無期限に一時停止すると発表した。これに基づいてプロダクト戦略を構築していたチームは困惑した。`generateObject`/`streamObject` APIもSDK 6.0で非推奨となった。AI SDK RSCからの推奨移行先は上記の`useObject`パターン、またはカタログベース生成のためのjson-renderだ。

### assistant-ui

**GitHub**: [assistant-ui/assistant-ui](https://github.com/assistant-ui/assistant-ui)

assistant-uiは本番品質のチャットインターフェースを構築するための構成可能なReactプリミティブのセットだ。洗練されたチャットUX — メッセージバブル、ストリーミングトークン、コピー/編集/再生成アクション、思考状態 — が必要で、独自のバックエンドとツールレンダリングを持ち込みたい場合の正しい選択だ。

任意のバックエンド（OpenAI、Anthropic、ローカルモデル、カスタムエンドポイント）と連携し、使い慣れたスロット/レンダープロップモデルでツール呼び出しレンダリングを扱う。

### json-render

**GitHub**: [vercel-labs/json-render](https://github.com/vercel-labs/json-render) · ドキュメント: [json-render.dev](https://json-render.dev/)

json-renderはパターン2を、オピニオン設定済みのバッテリー付きアプローチで実用化する。事前構築済みのコンポーネントカタログ（Zodスキーマ付きのshadcn/uiコンポーネント）、レンダラー、スキーマによってモデルがカタログに制約されるタイトな生成ループが得られる。

特徴的な機能：
- **マルチターゲットレンダリング**：同じJSON仕様をReact Webアプリ、React Nativeモバイルアプリ、PDF、HTMLメール、Remotionビデオにレンダリングできる。これはレポートにおいて真に有用だ。
- **プログレッシブレンダリング**：コンポーネントはモデルがストリーミングするたびに現れ、完全な仕様が到着した後にではない。
- **タイトなスキーマ制約**：カタログはモデルが有効だが未知のコンポーネントタイプを幻覚できないように設計されている。

ダッシュボードやレポート生成機能を構築していて、独自のカタログ設計のインフラ作業をスキップしたいなら、json-renderがWebアプリにとって最速のパスだ。

**勢いについて**：json-renderは2026年初頭にVercel Labsからローンチし、標準的なReact/Next.jsプロジェクトですぐに使えるため、Web開発者の注目を急速に集めているように見える。とはいえ、json-renderはまだプレ1.0であり、json-renderとA2UIの関係はまだ整理中だ — VercelはA2UI互換出力を実験しており、収束は可能だ。クロスプラットフォーム（ネイティブモバイル、複数フレームワーク）には、A2UIが長期的により良い選択だ。

### Hashbrown

**GitHub**: [liveloveapp/hashbrown](https://github.com/liveloveapp/hashbrown)

Hashbrownは独自のアプローチを取る：別のAIインターフェース層を構築するのではなく、AIコンポーネント選択を既存のReactまたはAngularアプリに直接埋め込む。アプリのコンポーネントをLLMに公開し、LLMがどれをレンダリングするかを選択し、クライアントサイドツールを呼び出せるようにする。

これは「チャット」ではないプロダクトサーフェスに知性を注ぎ込みたい場合に適したツールだ — レイアウトを適応させるプロダクトページ、正しいオプションを浮かび上がらせる設定パネル、次のステップを提案するワークフローエディタ。

### OpenUI

**GitHub**: [thesysdev/openui](https://github.com/thesysdev/openui) · ドキュメント: [openui.com](https://www.openui.com/)

OpenUIはJSONをプログレッシブレンダリングとトークン効率のために設計された行指向のコード風フォーマット（「OpenUI Lang」）に置き換える。複雑なレイアウトで同等のJSONより約67%少ないトークンで済むという主張だ。

トレードオフはエコシステムの成熟度だ — OpenUIは新しく、JSONベースのアプローチよりツールが薄い。しかしトークンコストが意味のある制約で、高頻度で複雑なレイアウトを生成しているなら、フォーマット効率は本物だ。

### Tambo

**GitHub**: [tambo-ai/tambo](https://github.com/tambo-ai/tambo)

Tamboはステートフルなコンポーネント選択に注力する：AIがコンポーネントを選択し、クライアントサイドツールを通じてそれらとやり取りし、会話を通じてコンポーネントの状態を維持する。UI要素がターンをまたいで持続するユースケース — ユーザーが微調整している間にAIがフィルタリングされたデータについて推論を続けるフィルターコンポーネントなど — に適している。

## Pythonレイヤー

PythonエコシステムはAIインターフェースに異なるアプローチを取る。これらのツールはMLモデルデモ、データアプリケーション、内部ツールリング向けに最適化されており — エージェント駆動のレイアウト構成を持つ本番コンシューマーアプリ向けではない。

それは批判ではない。正しいユースケースには、GradioとStreamlitが必要な唯一のツールだ。

### Gradio

**GitHub**: [gradio-app/gradio](https://github.com/gradio-app/gradio) · PyPI: `gradio`

Gradioの核心の価値：Python関数を書くと、GradioがそれをWeb UIでラップする。`Interface`クラスは画像分類器なら3行。`ChatInterface`はチャットボットなら10行。`Blocks`は必要な時にきめ細かいレイアウト制御を提供する。

Gradioにおける「ジェネレーティブUI」は、モデルではなくPython開発者によって定義される。コンポーネントの可視性と設定はモデル出力に基づいて動的に変更できるが、コンポーネントカタログは静的だ — モデルにレイアウト構成を依頼しているわけではない。

GradioはHuggingFace SpacesとMLデモエコシステムのデフォルトだ。月間数百万のダウンロードがあり、AIデモ領域の大部分を支えている。

**Gradioに手を伸ばすケース**：MLモデルデモ、研究プロトタイプ、内部ツールを構築するPython開発者で、JavaScriptに触りたくない場合。

### Streamlit

**GitHub**: [streamlit/streamlit](https://github.com/streamlit/streamlit)

Streamlitのモデルはよりオピニオン設定されている：Pythonスクリプトが各インタラクションでエンドツーエンドで実行される。`st.chat_message()`、`st.dataframe()`、`st.plotly_chart()`を呼び出す。フレームワークがレイアウトを処理する。

完全スクリプトリランのモデルは非効率に聞こえるが、会話履歴を蓄積するAIチャットボットには驚くほど使いやすい — スクリプト全体が再実行され、チャット履歴はセッション状態にあり、出力は決定的だ。Streamlitは主要なLLMプロバイダーのほとんどにファーストパーティサポートを持ち、Snowflake Cortexとネイティブに統合している。

**Streamlitに手を伸ばすケース**：PythonでAI搭載データアプリ、内部レポーティングツール、MLバックダッシュボードを構築し、可能な限りシンプルなデプロイパスを求める場合。

### LangChainとHaystack

これらはバックエンドオーケストレーションフレームワークであり、UIフレームワークではない。構造化出力がフロントエンドに送られる前に生成されるレイヤーであるため、誠実なジェネレーティブUIスタックマップに登場する。

**LangChain**（[langchain-ai/langchain](https://github.com/langchain-ai/langchain)）：任意のLLMで`.with_structured_output()`を使うとPydantic制約付きJSON生成が得られる。自動スキーマ生成付きの`@tool`デコレータは、モデルが呼び出し可能なツールを定義する最もクリーンな方法だ。LangChainは構造化結果を上流の任意のフロントエンド層に供給する。

**Haystack**（[deepset-ai/haystack](https://github.com/deepset-ai/haystack)）：強力なRAGサポートを持つモジュラーパイプラインアーキテクチャ。HayhooksはHaystackパイプラインをHTTPエンドポイントとしてラップする — MCP互換エンドポイントを含む。ジェネレーティブUIが検索バックボーンを必要とする場合、Haystackのパイプラインアーキテクチャがそれをきれいに処理する。

どちらのフレームワークもUI層を所有しない。フロントエンド（パターン1、2、または3）がレンダリングするデータを生成する。

## エコシステムの速度と不安定な地盤

このスペースは急速に動いており、いくつかのプロジェクトはコードとともにややこしいコミュニケーションを出荷している。2026年5月8日に最終確認。プロジェクトステータスのメモはタイムスタンプ付きの読み取りとして扱い、永続的な評価ではない。

**Vercel AI SDK RSC** はSDK 3.0ローンチ時のフラッグシップジェネレーティブUI機能だった。Vercelは2024年10月（[Discussion #3251](https://github.com/vercel/ai/discussions/3251)）にReact Server Componentsの「いくつかの長年の制限」に近い解決策がないとして開発を無期限一時停止した。これに基づいて構築したチームは当然不満だった。ドキュメントにはまだあるが推奨パスではない。`useObject`が推奨パスだ。

**json-render**（Vercel Labs）は新しい方向だ — RSC結合の問題を回避するカタログベースのフレームワーク非依存の代替。プレ1.0で、React/Web開発者の間で強い初期関心を持っているように見える。おそらくDXの理由：json-renderは標準的なReact/Next.jsプロジェクトですぐに使えるが、A2UIのクロスプラットフォームスコープはセットアップの摩擦を増やす。両方の仕様が成熟するにつれてどうなるかは本当に不明だ。Vercelはjson-renderでA2UI互換性を試しており、収束の可能性を示唆している。

**A2UI**（Google）はプレ1.0（最終確認時v0.9）で、マイナーバージョン間の破壊的変更と、ロードマップに関するGoogleの不規則なコミュニケーションがある。json-renderが対応しないクロスプラットフォームの到達範囲（Web + Flutter + SwiftUI）には正しい選択であり、意味のあるエンタープライズ支援がある。純粋なWebプロジェクトの場合、今のところDXは粗い。

**AG-UI**（CopilotKit）もプレ1.0だ。最も一般的な混乱：名前がUIフレームワークのように聞こえるが、そうではない — トランスポートプロトコルだ。AG-UIはイベントがエージェントバックエンドとフロントエンド間をどう流れるかを定義する。それに応じて何をレンダリングするかは依然としてあなたの決定だ。そのメンタルモデルは堅実で広く採用されているが、プレ1.0の仕様はエッジケースがまだ解決中であることを意味する。

実用的な結論：**ここでの主要なプレーヤーはすべてプレ1.0だ**。API変更を計画すること。パターン — ツール→コンポーネント、カタログ構成、サンドボックス化された生成 — は構築するのに十分安定している。特定のプロトコルの選択はそうではない。

## コンポーネントカタログ設計：本当のエンジニアリング作業

パターン2の面白い複雑さのほとんどはレンダラーではなく、カタログにある。

カタログは**スキーマとしてエンコードされたプロダクト決定**だ。それは次の問いに答える：このドメインにおける意味のあるUIオブジェクトは何か？「どんなReactコンポーネントが存在するか」ではなく、「このコンテキストのユーザーが実際に見て相互作用する必要があるものは何か？」

**細かすぎる失敗モード**：`Row`、`Column`、`Text`、`Button`、`Icon`を公開する。これでモデルはフロントエンドエンジニアにならなければならない。デザインシステムに合わない平凡なレイアウトを生成し、空の状態を見逃し、アクセシビリティのないマークアップを生成し、カタログ内の何もプロダクトのビジュアル言語に出力を制約しないため、応答ごとにアプローチを変える。

**粗すぎる失敗モード**：`WeatherCard`、`FlightCard`、`HotelCard`を公開する。事前焼き入れカードにマッピングされないものをユーザーが求めた場合、モデルは適応できない。テキストにフォールバックする。

**有用な中間**：制約付きスロットを持つドメインレベルコンポーネント。

旅行アプリのカタログはこうなるかもしれない：

```
TripSummary         — 一目でわかる旅程
FlightOptionList    — 価格付きの選択可能なフライトオプション
HotelComparison     — サイドバイサイドのホテルカード
TravelerForm        — 旅行者詳細の収集
PolicyNotice        — 規制/運賃ルールのコールアウト
BookingConfirmation — アクションボタン付きの最終確認
```

金融アプリのカタログはこうなるかもしれない：

```
PortfolioSnapshot   — 主要ポジションと損益
TransactionTable    — フィルタリング可能なページネーション付き取引
RiskBreakdown       — 配分とボラティリティメトリクス
ScenarioComparison  — サイドバイサイドのシナリオモデリング
ApprovalGate        — 人間の確認を必要とするアクション
```

カタログはプロダクトの語彙のように聞こえる。UXの決定、アクセシビリティ要件、空の状態の処理、危険なアクションパターンをコンポーネントコードにエンコードする。モデルはそれらのピースを配置できる。各ピースがどう見え、何を許可されるかは依然としてあなたが決定する。

**幻覚を減らすスキーマ設計ルール**：

1. 列挙値は短く明確に。`"type": "bar_chart"`であって`"type": "data-visualization-bar-type-vertical"`ではない。
2. 無効な構成を不可能に。`PolicyNotice`がレイアウトの最後にしか現れないなら、どこにでも現れるものと同じスキーマレベルに置かない。
3. 必須フィールドを惜しみなく使う。オプショナルフィールドは、モデルが省略する可能性があり、レンダラーがnullとして処理しなければならないフィールドだ。
4. 出荷前に実際のプロンプトでカタログをテストする。生成された仕様を保存し、スキーマ違反、幻覚されたフィールド値、技術的には有効だが意味的に間違っている構成をチェックする。

## よくある罠

**罠：有効なJSONを安全な動作と同等視する。** スキーマ検証は構造を確認する。ボタンに添付されたアクションがラベルと一致するか、合計が導出元のデータと一致するか、UIコンポーネントがユーザーの期待しないことをしているかについては何も言わない。生成されたUI仕様には、スキーマ検証だけでなくセマンティックレビューが必要だ。少なくとも、破壊的アクションには確認コンポーネントを必須にし、それらのコンポーネントのラベルがトリガーするアクションに対してテストされているべきだ。

**罠：プロダクトプリミティブではなくデザインプリミティブを公開する。** モデルが16pxか20pxのパディングを使うか決定しなければならないなら、間違った抽象レベルを与えている。ドメインコンポーネントはプロダクトの嗜好をエンコードするべきだ。モデルはプレゼンテーションの詳細を管理するのではなく、振る舞いを構成するべきだ。

**罠：静的UIで十分な場所でジェネレーティブUIを使う。** 表示したいものの構造が開発時に知り得る場合 — そして通常はそうだ — 事前構築コンポーネントを使ったパターン1はより速く、より安全で、より一貫している。ジェネレーティブUIは構造がデータやタスクコンテキストに基づいて正当に変動する場合にのみ、その複雑さに見合う。

**罠：アクセシビリティをスキップする。** LLMはWCAG違反を幻覚する。インタラクティブ要素に`role="region"`を出力し、ラベルの欠落したフォームを生成し、WCAG AAに失敗するコントラスト比を生成する。コンポーネントライブラリは完全にアクセシブルかもしれないが、AI生成のコンポーネント構成が自動的にアクセシブルであるわけではない。コンポーネントを孤立してではなく、完全なレンダリングパスでテストすること。

**罠：プロトコルとフレームワークを混同する。** AG-UIはフロントエンドフレームワークではない。A2UIはReactライブラリではない。これらはワイヤーフォーマットとイベントプロトコルだ。それらを実装するにはフロントエンドフレームワークが依然として必要だ。CopilotKitはAG-UIとA2UIを実装する。json-renderはA2UI/Open-JSON-UIカタログパターンを実装する。これらは異なる層だ。

## ユースケース別推奨

**既存のSaaSアプリにコパイロットを追加**：パターン1（ツール→コンポーネント）から始める。Vercel AI SDKの`useChat`またはCopilotKitを使用。トップ5〜10のエージェントアクションを事前構築コンポーネントにマッピングする。出荷して計測し、ユーザーがより豊かな構成を明確に必要とする場合のみカタログを拡大する。

**自然言語からのダッシュボード生成**：json-renderまたはカスタムA2UIカタログでパターン2を使用。チャートタイプ、メトリックカード、テーブルバリアントをカバーする8〜15のコンポーネントタイプのカタログを定義する。スキーマをモデルに供給し、レイアウトの構成を任せる。未知のタイプがレンダラーに到達する前に捕捉する検証を構築する。

**マルチエージェントフロントエンド**：CopilotKitとAG-UIを使用。イベントストリームはエージェントバックエンド間のリアルタイムストリーミングを処理し、共有状態はエージェント間の引き継ぎを処理し、HITLパターンは承認ゲートを処理する。

**ChatGPTまたは他のMCPホスト内で構築**：MCP Appsを使用。ツールをデータをフェッチして推論するデータツールとして定義し、ウィジェットをリクエストする別のレンダーツールとして定義する。ビジネスロジックをウィジェットテンプレートから除外する。

**MLモデルデモとデータアプリ（Pythonチーム）**：デモとHuggingFace SpacesにはGradio。より複雑なインタラクションを持つデータアプリにはStreamlit。どちらもJavaScriptを必要としない。

**ビジュアルアーティファクト、シミュレーション、図**：パターン3（OpenGenerativeUIまたは同等のもの）を使用。厳格なiframe CSPを確立する。セキュリティの観点から出力を信頼できないユーザーコンテンツとして扱う。

フレームワークは急速に成熟している。プロトコルの収束（ストリーミングにはAG-UI、カタログ仕様にはA2UI/Open-JSON-UI）はまだ進行中だが、構築するのに十分明確な形になりつつある。

今最も重要なエンジニアリングチャレンジはフレームワークの選択ではない。それはカタログ設計だ — モデルに何を言うことを許可するかを決定することであり、それは技術的スキルよりもプロダクトの明確さを必要とする。それはセマンティック検証だ — 生成されたUIがスキーマ検証を通るだけでなく、それが主張することを実際に行うことをテストすること。そしてそれはアクセシビリティギャップだ — すべてのコンポーネント、そしてコンポーネントのすべての構成が、手書きUIに求めるアクセシビリティの水準を満たすカタログを構築すること。

モデルはあなたが与えた文法の中で、あなたが指示したことをする。文法を意図的なものにせよ。
