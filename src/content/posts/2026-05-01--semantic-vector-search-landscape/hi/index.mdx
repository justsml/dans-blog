Search एक चीज़ नहीं है, और semantic search बाकी सबका replacement नहीं है।

"Find user with email `dan@example.com`" और "find me articles about debugging as a new engineer" — दोनों को search कहा जाता है, लेकिन engineering problems के रूप में इनका आपस में लगभग कुछ भी common नहीं है। पहले का एक सही उत्तर है और `O(log n)` index lookup। दूसरे का कोई सही उत्तर नहीं है — केवल relevance है — और इसमें भाषा, intent और meaning को समझने की ज़रूरत होती है।

Search decisions के बारे में सबसे ज़्यादा persuasive engineers — जो arguments जीतते हैं और सही system ship करते हैं — वे पूरे landscape को समझते हैं। उन्हें पता होता है कि किस tool को कब और क्यों reach करना है, और वे इसे स्पष्ट रूप से समझा भी सकते हैं।

यह article semantic layer को कवर करता है: vector search वास्तव में क्या करता है, यह कब जीतता है, और कहाँ इसे रास्ते से हटकर रहना चाहिए। उपयोगी version "embed everything" नहीं है। यह जानना है कि vectors lexical, fuzzy और exact-match search के बगल में hybrid architecture में कब belong करते हैं।

Lexical और fuzzy half की तस्वीर — `tsvector`, `pg_trgm`, `pg_search` — [Postgres Text Searching Guide 2026](/postgres-text-search-guide) में है।

## Terms at a Glance

**Embedding** — एक model द्वारा produced floating-point numbers की dense list, जो text (या image, audio, आदि) के एक piece को high-dimensional space में एक point के रूप में represent करती है। Semantically related content पास-पास land करता है; unrelated content दूर land करता है।

**Lexical search** — exact word और token matching पर आधारित search। तेज़, deterministic, और known terms के लिए सही। Synonyms, paraphrases, या cross-language equivalents को समझ नहीं सकता।

**Semantic search** — meaning पर आधारित search, tokens पर नहीं। "how do I handle timeouts" query "configuring retry policies" शीर्षक वाले document से match कर सकती है — कोई shared words नहीं, लेकिन geometrically close embeddings।

**Vector** — numbers की एक list। search contexts में, embedding model का output। "Vector search" query vector के सबसे नज़दीक vectors को geometric distance से खोजता है।

**FTS (Full-Text Search)** — Postgres का built-in lexical search, `tsvector` / `tsquery` द्वारा संचालित। Text को tokenize, stem और index करता है keyword queries के लिए। Prose और exact-term lookup के लिए मज़बूत; meaning के लिए blind।

**BM25** — Lexical search के लिए ranking algorithm (Elasticsearch, Qdrant और अन्य द्वारा उपयोग किया जाता है)। Term frequency को score करता है, यह weighted against करता है कि term corpus में कितना rare है। Raw keyword matching से बेहतर; अभी भी lexical।

**HNSW (Hierarchical Navigable Small World)** — Vector search के लिए standard approximate nearest-neighbor index। तेज़, high-recall similarity queries के लिए layered proximity graph बनाता है। pgvector, Qdrant, Weaviate और ज़्यादातर अन्य इसे use करते हैं।

**RRF (Reciprocal Rank Fusion)** — Multiple retrieval systems से ranked result lists को merge करने का algorithm। केवल rank position का उपयोग करता है — score normalization की ज़रूरत नहीं। जो result दोनों FTS और vector lists में ऊँचा rank करता है, उसे stronger combined score मिलता है।

## Semantic Search वास्तव में क्या करता है

Vector embeddings text (या images, audio, आदि) को numbers की एक list में convert करते हैं — high-dimensional space में एक point। एक embedding model को train किया जाता है ताकि semantically related text उस space में पास-पास land करे। "Dog" और "canine" पास आते हैं। "Running a marathon" और "running a Python script" एक word share करने के बावजूद दूर आते हैं।

उस space में similarity search उन documents को खोजता है जिनका *meaning* query के meaning के सबसे नज़दीक है, चाहे exact word overlap हो या न हो।

इसका मतलब:
- "How do I configure request timeouts?" एक article titled "Setting connection limits and retry policies" से match कर सकता है — कोई overlapping keywords नहीं, high conceptual relevance
- "Something light for a summer evening" product description में कोई keywords appear किए बिना wine recommendation से match कर सकता है
- अंग्रेज़ी में query French, Spanish या Japanese documents से match कर सकती है यदि embedding model multilingually trained था

Lexical search (`tsvector`, `pg_trgm`) यह कुछ भी नहीं कर सकता। यह words और characters पर operate करता है, meaning पर नहीं। Tools interchangeable नहीं हैं — वे अलग problems solve करते हैं।

## pgvector कब जीतता है

**RAG बनाना।** Retrieval-Augmented Generation user के question के सबसे नज़दीक meaning वाले document chunks को retrieve करता है, फिर उन्हें language model को context के रूप में pass करता है। यह retrieval step एक vector operation है। FTS paraphrases, synonyms और conceptual matches को miss कर देगा जो relevant chunk अलग तरह से express कर सकता है। pgvector का standalone vector store पर advantage: यह आपके existing Postgres instance के अंदर चलता है — कोई separate service deploy, operate या sync करने की ज़रूरत नहीं।

**Users बताते हैं कि उन्हें क्या चाहिए, search क्या करना है यह नहीं।** "Articles about building confidence as a new manager" — relevant posts में reliably appear करने वाले keywords नहीं हैं। "A lightweight framework for handling side effects" — documentation में वे exact words शायद न हों। Vector search spelling को नहीं, intent को match करता है।

**Similar items खोजना।** Related products, similar support tickets, duplicate bug reports, articles you might also like। "Find issues similar to this one" एक nearest-neighbor search है — item को embed करो, geometric neighbors खोजो। एक important caveat: vector search हमेशा results return करता है, भले ही कुछ genuinely similar न हो। Dedup और recommendation use cases के लिए, minimum similarity threshold से filter करो (e.g., cosine similarity ≥ 0.80) ताकि low-confidence matches को meaningful के रूप में surface न किया जाए।

**Semantic deduplication।** RAG या search के लिए content index करने से पहले, अक्सर corpus में near-duplicates identify करने होते हैं — articles revised multiple times, support tickets filed twice, knowledge base entries जो significantly overlap करती हैं। Documents को embed करो और cosine similarity से threshold-filter करो near-duplicates को flag या merge करने के लिए, ताकि वे index को pollute न करें। यह retrieval को multiple near-identical chunks return करने और context window को dilute करने से रोकता है।

**Multilingual search।** Multilingual embedding models semantically equivalent content को languages के across nearby vectors में map करते हैं। "perder peso" के लिए Spanish query "sustainable weight loss habits" पर English article से match कर सकती है — कोई shared tokens नहीं, same underlying meaning। FTS को per-language dictionary configuration चाहिए और cross-language queries को poorly handle करता है। `pg_trgm` language-agnostic है लेकिन orthographic है, semantic नहीं।

### Setting Up pgvector

Extension install से similarity query तक, setup कुछ SQL statements है:

```sql
CREATE EXTENSION IF NOT EXISTS vector;

ALTER TABLE documents ADD COLUMN embedding vector(1536);

-- HNSW is usually the first index to try for moderate-size datasets
CREATE INDEX documents_embedding_idx
  ON documents USING hnsw (embedding vector_cosine_ops);

-- Semantic search query
SELECT id, title, 1 - (embedding <=> $1::vector) AS similarity
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 10;
```

`<=>` cosine distance है। `1 - cosine_distance` cosine similarity देता है (1.0 = identical, 0.0 = orthogonal)। `ivfflat` (पुराना, faster-to-build alternative) के लिए, `lists = sqrt(row_count)` को starting point के रूप में use करो।

### pgvector क्या अच्छे से handle नहीं करता

- Exact token matching — product SKUs, error codes, function names। `ORD-12345` किसी भी चीज़ से semantically similar नहीं है। Embedding-based search `ORD-12344` या nothing relevant return कर सकता है। FTS या B-tree index का उपयोग करो।
- Names और proper nouns। Embedding space meaning से organize होता है, spelling से नहीं। "Micheal Jordan" user record vector space में "Michael Jordan" के ज़रूरी पास नहीं land करेगा।
- Short strings जहाँ character-level similarity meaning से ज़्यादा मायने रखती है। `pg_trgm` इसे handle करता है।
- Queries जहाँ exact term appear होना चाहिए। BM25 और FTS known-term matching के लिए ज़्यादा reliable हैं।

## Hybrid Search: दोनों का case

Technical documentation सबसे clear example है जहाँ कोई एक tool अकेले पर्याप्त नहीं है।

"how to configure timeouts" search करने वाले users को conceptual matching चाहिए: "Setting retry policies and connection limits" शीर्षक article में कोई overlapping keywords नहीं हैं लेकिन यह exactly वही है जो उन्हें चाहिए।

वे same users `withRetry()`, `ECONNRESET` और `ERR_SOCKET_TIMEOUT` भी search करते हैं। ये exact strings appear होने चाहिए — semantic matching उन्हें reliably नहीं खोज सकता, और false positive (conceptually similar लेकिन सही API नहीं) actively misleading है।

Vector search conceptual queries को handle करता है। FTS exact terms को handle करता है। कोई भी अकेले दोनों को अच्छे से handle नहीं करता।

Solution hybrid search है: दोनों run करो और results को fuse करो।

### Reciprocal Rank Fusion

**Reciprocal Rank Fusion (RRF)** different retrieval systems से ranked lists को combine करने का standard algorithm है। इसे systems के across scores को normalize करने की ज़रूरत नहीं — यह केवल rank positions का उपयोग करता है। जो result *दोनों* lists में ऊँचा appear होता है, उसे उस result से stronger combined score मिलता है जो केवल एक में dominate करता है।

```sql
WITH fts_results AS (
  SELECT id,
    ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM documents, to_tsquery('english', $1) query
  WHERE search_vector @@ query
  LIMIT 50
),
vector_results AS (
  SELECT id,
    ROW_NUMBER() OVER (ORDER BY embedding <=> $2::vector) AS rank
  FROM documents
  ORDER BY embedding <=> $2::vector
  LIMIT 50
),
rrf AS (
  SELECT
    COALESCE(f.id, v.id) AS id,
    COALESCE(1.0 / (60 + f.rank), 0) +
    COALESCE(1.0 / (60 + v.rank), 0) AS rrf_score
  FROM fts_results f
  FULL OUTER JOIN vector_results v ON f.id = v.id
)
SELECT d.id, d.title, rrf.rrf_score
FROM rrf
JOIN documents d ON d.id = rrf.id
ORDER BY rrf_score DESC
LIMIT 10;
```

Denominator में `60` RRF constant है। Higher values rank-position differences को dampen करते हैं; lower values amplify करते हैं। Default 60 ज़्यादातर content types के लिए अच्छा काम करता है।

RRF `ts_rank` (log-frequency score) को cosine distance (geometric measure) के against normalize करने की harder problem से बचता है। वे comparable नहीं हैं। RRF केवल पूछता है: "यह result हर list में कितना ऊँचा appear हुआ?"

### Trigrams के साथ भी Hybrid Search

User-facing search over mixed content के लिए — जहाँ users एक ही session में person name, concept या exact term search कर सकते हैं — three-way fusion सबको handle करता है:

```sql
WITH trgm_results AS (
  SELECT id,
    ROW_NUMBER() OVER (ORDER BY similarity(title, $1) DESC) AS rank
  FROM documents
  WHERE title % $1
  LIMIT 50
),
fts_results AS (
  SELECT id,
    ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, to_tsquery('english', $1)) DESC) AS rank
  FROM documents
  WHERE search_vector @@ to_tsquery('english', $1)
  LIMIT 50
),
vector_results AS (
  SELECT id,
    ROW_NUMBER() OVER (ORDER BY embedding <=> $2::vector) AS rank
  FROM documents
  ORDER BY embedding <=> $2::vector
  LIMIT 50
),
rrf AS (
  SELECT
    COALESCE(t.id, f.id, v.id) AS id,
    COALESCE(1.0 / (60 + t.rank), 0) +
    COALESCE(1.0 / (60 + f.rank), 0) +
    COALESCE(1.0 / (60 + v.rank), 0) AS rrf_score
  FROM trgm_results t
  FULL OUTER JOIN fts_results f ON t.id = f.id
  FULL OUTER JOIN vector_results v ON COALESCE(t.id, f.id) = v.id
)
SELECT d.id, d.title, rrf.rrf_score
FROM rrf
JOIN documents d ON d.id = rrf.id
ORDER BY rrf_score DESC
LIMIT 10;
```

यह handle करता है: fuzzy name matches (trigrams), exact keyword matches (FTS), और conceptual queries (vector)। एक ही search box तीनों user intents को serve कर सकता है।

## Multi-Layer Hybrid Architectures

Real applications में शायद ही कभी single search surface होता है। उनके पास multiple होते हैं, हर एक की अलग ज़रूरत:

| Surface | Users क्या query करते हैं | Recommended layers |
|---|---|---|
| Blog / documentation search | Keywords + concepts | FTS + pgvector (RRF) |
| User/customer name lookup | Typos वाले names | `pg_trgm` |
| Product search | Names, descriptions, "similar to" | `pg_trgm` + FTS + pgvector |
| Support ticket dedup | "Issues similar to this one" | pgvector only |
| Internal SKU/order search | Exact identifiers | B-tree index |
| RAG over large knowledge base | Natural language questions | pgvector (chunked docs) |
| E-commerce "you might also like" | Behavioral + semantic similarity | pgvector |
| Autocomplete | Prefix, spelling-tolerant | `pg_trgm` |

ये hypothetical नहीं हैं। ज़्यादातर content-heavy applications को कम से कम दो distinct search surfaces चाहिए अलग query shapes के साथ। Temptation एक approach pick करके उसे हर जगह use करने की होती है — अब usually vector search, क्योंकि यह fashionable choice है। इससे expensive embeddings उन problems के लिए बनते हैं जहाँ trigram index faster, cheaper और ज़्यादा correct होता।

### Rule of Thumb

Layer तब add करो जब कोई failure mode appear हो जिसे current layer fix नहीं कर सकता:

- Users typos match न होने की complaint करते हैं → `pg_trgm` add करो
- Users concept से search करते हैं और relevant results miss करते हैं → pgvector add करो
- Users exact symbols या codes search करते हैं और conceptual results मिलते हैं → FTS add करो या check करो कि vector search पर over-rely तो नहीं कर रहे
- Latency problem बनती है → pre-filtering, approximate indexes, या dedicated store evaluate करो

## यदि Dedicated Vector Store चाहिए

pgvector काफी application search को handle कर लेता है इससे पहले कि आपको दूसरे database की ज़रूरत हो। Rough cutoff vector count, index settings, write rate, filters, hardware और concurrency पर depend करता है, इसलिए "under 10M vectors" rule को product limit नहीं, benchmark के लिए starting assumption के रूप में treat करो। जब आप genuinely outgrow करो — very high concurrency, very low p99 latency requirements, billions of vectors, या serious multi-tenant isolation needs — dedicated vector database landscape wide है और समझने योग्य है।

### Matrix Columns वास्तव में क्या mean करते हैं

**Hybrid search** का मतलब BM25 keyword search और vector similarity एक query में run होते हैं, RRF के माध्यम से merged। इसके बिना, या तो एक search mode pick करो या दो queries को yourself fuse करो।

**Sparse vectors** BM25 से आगे जाते हैं। SPLADE sparse vector में ~30,000 dimensions होते हैं (vocabulary term के लिए एक), ~98% zeros। Non-zero positions बताते हैं कि कौन से terms matter हैं और कितना। "dogs" query "canine" और "pet" को भी weight करती है — BM25-level precision plus term expansion vector index के अंदर। यदि यह column false है, तो exact-term queries के लिए separate FTS layer चाहिए।

```python
# SPLADE: ~30,000 dims, ~60 non-zero — केवल relevant vocabulary positions fire
def encode_splade(text: str) -> dict:
    tokens = tokenizer(text, return_tensors="pt", truncation=True, max_length=512)
    with torch.no_grad():
        output = model(**tokens)
    vec = torch.log1p(torch.relu(output.logits)).max(dim=1).values.squeeze()
    return {"indices": vec.nonzero().squeeze().tolist(), "values": vec[vec != 0].tolist()}
```

**SQL / SQL-like** वास्तव में filtering के बारे में है। Filtering के बिना vector search एक demo है। अभी भी tenant scope, date ranges, permissions और category filters चाहिए। Full SQL (pgvector, LanceDB) इसे existing joins के बगल में express करता है। Purpose-built databases JSON filter objects (Qdrant, Pinecone), query DSL (Elasticsearch, Milvus), या GraphQL (Weaviate) use करते हैं। वे काम करते हैं; filter logic complex होता जाता है तो SQL ज़्यादा attractive बनता है।

```sql
-- pgvector: vector similarity बस एक और expression है
SELECT id, title, 1 - (embedding <=> $1) AS score
FROM documents
WHERE tenant_id = $2
  AND category = ANY($3::text[])
  AND created_at > NOW() - INTERVAL '90 days'
ORDER BY embedding <=> $1
LIMIT 10;
```

```python
# Qdrant: Python object के रूप में equivalent filter — same result, more ceremony
results = client.query_points(
    collection_name="documents", query=query_embedding,

        models.FieldCondition(key="created_at", range=models.DatetimeRange(gte=cutoff)),
    ]),

```

**Multimodal native** का मतलब database non-text content के लिए embedding models ship करता है। आप raw image URL देते हैं; vectorization को handle करता है। ज़्यादातर databases embedding-agnostic हैं — embedding pipeline आपकी ज़िम्मेदारी। Marqo और Weaviate (CLIP/ImageBind modules के माध्यम से) इस loop को close करते हैं।

```python
# Marqo: raw images POST करो, text से query करो — कोई external embedding step नहीं
mq.index("products").add_documents(
    [{"id": "shoe-001", "image": "https://cdn.example.com/shoes/001.jpg"}],
    tensor_fields=["image"]
)
results = mq.index("products").search(q="lightweight shoes for summer")
# zero keyword overlap के बावजूद shoe-001 return करता है — CLIP cross-modal match को handle करता है
```

**Disk-based index** cost lever है। RAM-resident HNSW indexes को प्रति million 1536-dimension vectors कई GB RAM लग सकते हैं एक बार raw vectors, graph overhead और metadata count किए जाएँ। Disk-native alternatives (Milvus DiskANN, Elasticsearch DiskBBQ, LanceDB का Lance format, Turbopuffer का object storage tier) अक्सर query latency trade करते हैं lower infrastructure cost के लिए। RAG workloads के जहाँ model latency already dominate करती है, वह tradeoff अक्सर benchmark करने योग्य होता है।

**Max dimensions** आपके architecture में छिपा migration है। `text-embedding-3-large` 3072 dims use करता है, Jina v3 बड़े embeddings emit कर सकता है, और research models higher push करते रहते हैं। कुछ managed services hard dimension caps publish करते हैं; अन्य high caps या typical embedding models के लिए no practical cap document करते हैं। Commit करने से पहले current docs check करो। Headroom के साथ कुछ pick करो; dimension ceiling hit करने के बाद vector index migrate करना painful sprint है।

### Table में Fit न होने वाली कुछ चीज़ें

**Turbopuffer का multi-tenancy** very high namespace counts के around built है। इसकी public positioning और customer stories Notion के large, namespace-heavy corpus जैसे workloads पर emphasize करती हैं। यदि हर user या organization को isolated vector search चाहिए, वह architecture economics को change कर सकता है, लेकिन अपनी tenant shape को अभी भी benchmark करो।

**LanceDB embedded mode** "SQLite for vector search" के सबसे नज़दीक चीज़ है। यह in-process चलता है, कोई server require नहीं करता, और Lambda, Cloudflare Workers और edge environments में काम करता है। Lance columnar format embedded operation को real scale पर practical बनाता है।

**Chroma dev/test और small app deployments में सबसे मज़बूत है।** यदि आप very large corpora, HA, disk-heavy operation, या first-class hybrid search को target कर रहे हैं, prototype को infrastructure में promote करने से पहले production-oriented store evaluate करो।

**Vespa वहाँ reach करते हैं जब retrieval केवल आधा product है।** यह lexical retrieval, nearest-neighbor search, tensors, ranking expressions, grouping और online serving को combine करता है। वह power real है, लेकिन operational और modeling complexity भी real है। यह search/recommendation teams के लिए fit होता है "add semantic search to my CRUD app" से ज़्यादा।

**ClickHouse conversation में belong करता है जब search analytics से attached है।** यदि आपका source of truth events, logs, traces या metrics है, ClickHouse vector distance, filtering, aggregation और serious full-text indexing को एक SQL engine में रखता है। Purpose-built vector database नहीं, लेकिन अक्सर analytical retrieval के लिए boring-right answer।

**Sparse vectors जिस तरीके से BM25-quality keyword matching vector index के अंदर मिलता है** — separate full-text engine run किए बिना। Qdrant और Elasticsearch में यहाँ especially mature implementations हैं। यदि hybrid search critical है और two-system architecture deal-breaker है, sparse vector support को look करना चाहिए।

### Choosing जब pgvector से outgrow हो चुके हैं

- **SaaS product with per-tenant isolation** → Turbopuffer
- **Complex metadata filtering at scale** → Qdrant
- **Already on Elastic/ELK stack** → Elasticsearch with DiskBBQ
- **AWS shop that wants open-source** → OpenSearch
- **Search/recommendation platform with serious ranking needs** → Vespa
- **Analytics, observability, log/event search** → ClickHouse
- **Billion-scale on-prem / self-hosted** → Milvus
- **Edge / serverless / multimodal** → LanceDB
- **Small JS app, docs site, or edge-native search UX** → Orama
- **Zero ops, cost is secondary** → Pinecone
- **Multimodal-first (images, video, audio)** → Marqo
- **Already on MongoDB** → Atlas Vector Search
- **Already on Postgres, need more headroom** → Supabase Vector or Neon (both pgvector managed, with better tooling)

## One Thing जो नहीं करना है

Vector search को fuzzy text search के रूप में use मत करो उन चीज़ों के लिए जिनके correct answers हैं।

"Find me the user with email `dan@example.com`" vector search problem नहीं है। "Find the order with ID `ORD-12345`" भी नहीं। `ORD-12345` को embed करना और cosine similarity से search करना *something* return करेगा — लेकिन यह ग़लत हो सकता है। Identifier का एक सही उत्तर है। Identifier पर approximate match एक bug है।

Vector search आपके dataset में *most similar* चीज़ return करता है, भले ही कुछ actually relevant न हो। यह नहीं जानता कि जब कोई good answer exist नहीं करता। यह related documents के लिए ठीक है। यह exact record lookup के लिए serious problem है, जहाँ confident wrong answer empty result से बदतर है।

Same दूसरी direction में लागू होता है: FTS को उन queries के लिए use मत करो जहाँ user concept describe कर रहा है। "articles about making hard decisions under uncertainty" में कोई reliable keywords नहीं हैं। FTS या तो noise return करेगा या nothing। Query shape के लिए सही tool का उपयोग करो।

## Full Picture

ज़्यादातर production search systems को एक से ज़्यादा layer चाहिए:

- **`pg_trgm`** names, typos, autocomplete के लिए
- **FTS / `pg_search`** keyword-based prose search के लिए
- **pgvector** semantic और conceptual queries के लिए
- **RRF fusion** उन surfaces के लिए जहाँ users query types mix करते हैं
- **Regular indexes** exact identifiers, filters और sorted lists के लिए

ये competing tools नहीं हैं। ये complementary हैं। Well-built search system हर query shape के लिए सही layer pick करता है — और जब query shapes overlap करते हैं, यह multiple layers run करता है और results को fuse करता है।

जो teams good search features ship करती हैं वे पूरे stack को समझती हैं। जो नहीं समझतीं वे vector database reach करती हैं, everything embed करती हैं, और wonder करती हैं कि exact lookups कभी-कभी ग़लत record क्यों return करते हैं।
