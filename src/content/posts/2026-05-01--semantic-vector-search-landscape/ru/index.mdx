Поиск — это не одна вещь, и семантический поиск не заменяет остальные его виды.

«Найди пользователя с email `dan@example.com`» и «найди статьи об отладке для начинающего инженера» оба называются поиском, но как инженерные задачи они почти не имеют ничего общего. У первого есть правильный ответ и поиск по индексу за `O(log n)`. У второго правильного ответа нет — есть только релевантность — и требуется понимание языка, намерения и смысла.

Инженеры, которые наиболее убедительно аргументируют решения по поиску — те, кто выигрывает споры и собирает правильную систему, — понимают картину целиком. Они знают, какой инструмент выбрать и почему, и могут объяснить это чётко.

Эта статья посвящена семантическому слою: что на самом деле делает векторный поиск, когда он побеждает и где ему лучше не вмешиваться. Полезная версия — не «векторизуйте всё». Она заключается в том, чтобы понимать, когда векторы должны стоять рядом с лексическим, нечётким и точным поиском в гибридной архитектуре.

Лексическая и нечёткая половина картины — `tsvector`, `pg_trgm`, `pg_search` — описана в [Руководстве по полнотекстовому поиску в Postgres 2026](/postgres-text-search-guide).

## Краткий словарь терминов

**Векторное представление (embedding)** — плотный список чисел с плавающей точкой, созданный моделью, представляющий фрагмент текста (или изображения, аудио и т.д.) как точку в многомерном пространстве. Семантически связанное содержимое оказывается поблизости; несвязанное — далеко.

**Лексический поиск** — поиск на основе точного совпадения слов и токенов. Быстрый, детерминированный и точный для известных терминов. Не понимает синонимов, перефразировок или межъязыковых эквивалентов.

**Семантический поиск** — поиск по смыслу, а не по токенам. Запрос «как мне справляться с тайм-аутами» может найти документ под заголовком «настройка политик повторных попыток» без единого общего слова, потому что их векторные представления геометрически близки.

**Вектор** — список чисел. В контексте поиска — результат модели векторизации. «Векторный поиск» находит векторы, наиболее близкие к вектору запроса по геометрическому расстоянию.

**FTS (Full-Text Search)** — встроенный в Postgres лексический поиск на базе `tsvector` / `tsquery`. Токенизирует, приводит к начальной форме и индексирует текст для поиска по ключевым словам. Хорош для прозаического текста и точного поиска по терминам; слеп к смыслу.

**BM25** — алгоритм ранжирования для лексического поиска (используется в Elasticsearch, Qdrant и др.). Оценивает результаты по частоте терминов с учётом их редкости по всему корпусу. Лучше обычного поиска по ключевым словам, но всё ещё лексический.

**HNSW (Hierarchical Navigable Small World)** — стандартный индекс приближённого поиска ближайших соседей для векторного поиска. Строит многослойный граф близости для быстрых запросов по сходству с высоким recall. Его используют pgvector, Qdrant, Weaviate и большинство других решений.

**RRF (Reciprocal Rank Fusion)** — алгоритм объединения ранжированных списков результатов от нескольких систем поиска. Использует только позицию в рейтинге — нормализация оценок не требуется. Результат, занимающий высокие позиции в обоих списках (FTS и векторном), получает более сильную комбинированную оценку, чем тот, который доминирует только в одном.

## Что на самом деле делает семантический поиск

Векторные представления превращают текст (или изображения, аудио и т.д.) в список чисел — точку в многомерном пространстве. Модель векторизации обучена так, чтобы семантически связанные тексты оказывались рядом в этом пространстве. «Собака» и «пёс» оказываются близко. «Бегать марафон» и «запускать Python-скрипт» оказываются далеко, несмотря на общее слово.

Поиск по сходству в этом пространстве находит документы, чей *смысл* ближе всего к смыслу запроса, независимо от совпадения точных слов.

Это значит:
- «Как настроить тайм-ауты запросов?» может найти статью «Установка лимитов подключений и политик повторных попыток» — нет общих ключевых слов, высокая смысловая релевантность
- «Что-нибудь лёгкое для летнего вечера» может найти рекомендацию вина без единого ключевого слова в описании товара
- Запрос на английском может найти релевантные документы на французском, испанском или японском, если модель векторизации обучалась на нескольких языках

Лексический поиск (`tsvector`, `pg_trgm`) не способен ни на одно из этого. Он оперирует словами и символами, а не смыслом. Эти инструменты не взаимозаменяемы — они решают разные задачи.

## Когда побеждает pgvector

**Построение RAG.** Retrieval-Augmented Generation извлекает фрагменты документов, чей смысл ближе всего к вопросу пользователя, а затем передаёт их языковой модели как контекст. Этот шаг извлечения — векторная операция. FTS пропустит перефразировки, синонимы и смысловые совпадения, которые релевантный фрагмент может выразить иначе. Преимущество pgvector перед отдельным векторным хранилищем: он работает внутри существующего экземпляра Postgres — не нужно разворачивать, обслуживать и синхронизировать отдельный сервис.

**Пользователи описывают, что хотят, а не что искать.** «Статьи о том, как обрести уверенность новому руководителю» не содержит ключевых слов, которые надёжно появятся в релевантных постах. «Лёгковесный фреймворк для обработки побочных эффектов» может не использовать эти точные слова в документации. Векторный поиск совпадает с намерением, а не с написанием.

**Поиск похожих элементов.** Похожие товары, похожие тикеты поддержки, дубликаты баг-репортов, статьи, которые вам тоже могут понравиться. «Найди задачи, похожие на эту» — это поиск ближайших соседей: векторизуйте элемент, найдите его геометрических соседей. Важная оговорка: векторный поиск всегда возвращает результаты, даже когда ничего по-настоящему похожего нет. Для дедупликации и рекомендаций фильтруйте по минимальному порогу сходства (например, косинусное сходство ≥ 0.80), чтобы не выдавать малодостоверные совпадения за значимые.

**Семантическая дедупликация.** Перед индексацией контента для RAG или поиска часто нужно выявить почти-дубликаты в корпусе — статьи с несколькими редакциями, тикеты поддержки, поданные дважды, записи базы знаний с существенным пересечением. Векторизуйте документы и отфильтруйте по косинусному сходству, чтобы пометить или объединить почти-дубликаты до того, как они засорят индекс. Это предотвращает возврат нескольких почти идентичных фрагментов и разбавление контекстного окна.

**Мультиязычный поиск.** Мультиязычные модели векторизации помещают семантически эквивалентное содержимое на разных языках в близкие векторы. Запрос на испанском «perder peso» может найти английскую статью о «sustainable weight loss habits» — нет общих токенов, один и тот же смысл. FTS требует настройки словаря для каждого языка и плохо справляется с межъязыковыми запросами. `pg_trgm` не зависит от языка, но работает на уровне орфографии, а не семантики.

### Настройка pgvector

От установки расширения до запроса по сходству — всё это несколько SQL-выражений:

```sql
CREATE EXTENSION IF NOT EXISTS vector;

ALTER TABLE documents ADD COLUMN embedding vector(1536);

-- HNSW — обычно первый индекс, который стоит попробовать для датасетов среднего размера
CREATE INDEX documents_embedding_idx
  ON documents USING hnsw (embedding vector_cosine_ops);

-- Семантический поисковый запрос
SELECT id, title, 1 - (embedding <=> $1::vector) AS similarity
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 10;
```

`<=>` — косинусное расстояние. `1 - косинусное_расстояние` даёт косинусное сходство (1.0 = идентично, 0.0 = ортогонально). Для `ivfflat` (старый вариант, строится быстрее) используйте `lists = sqrt(row_count)` как начальную точку.

### Что pgvector обрабатывает плохо

- Точное совпадение токенов — артикулы товаров, коды ошибок, имена функций. `ORD-12345` семантически не похож ни на что. Поиск на основе векторов может вернуть `ORD-12344` или вообще ничего релевантного. Используйте FTS или B-tree индекс.
- Имена и имена собственные. Векторное пространство организовано по смыслу, а не по написанию. Запись пользователя «Micheal Jordan» не обязательно окажется рядом с «Michael Jordan» в векторном пространстве.
- Короткие строки, где важнее посимвольное сходство, чем смысл. С этим справится `pg_trgm`.
- Запросы, где должен присутствовать точный термин. BM25 и FTS надёжнее для поиска по известным терминам.

## Гибридный поиск: аргумент за оба подхода

Техническая документация — самый очевидный пример, где ни один инструмент в одиночку не достаточен.

Пользователи, ищущие «как настроить тайм-ауты», нуждаются в смысловом сопоставлении: статья «Настройка политик повторных попыток и лимитов подключений» не содержит общих ключевых слов, но именно то, что им нужно.

Те же пользователи ищут `withRetry()`, `ECONNRESET` и `ERR_SOCKET_TIMEOUT`. Эти точные строки должны присутствовать — семантическое сопоставление может не найти их надёжно, а ложноположительный результат (похоже по смыслу, но не тот API) будет активно вредить.

Векторный поиск справляется со смысловыми запросами. FTS — с точными терминами. По отдельности ни один не справляется с обоими.

Решение — гибридный поиск: запустите оба и объедините результаты.

### Reciprocal Rank Fusion

**Reciprocal Rank Fusion (RRF)** — стандартный алгоритм для объединения ранжированных списков от разных систем поиска. Он не требует нормализации оценок между системами — использует только позиции в рейтинге. Результат, который оказывается высоко в *обоих* списках, получает более сильную комбинированную оценку, чем тот, который доминирует только в одном.

```sql
WITH fts_results AS (
  SELECT id,
    ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM documents, to_tsquery('english', $1) query
  WHERE search_vector @@ query
  LIMIT 50
),
vector_results AS (
  SELECT id,
    ROW_NUMBER() OVER (ORDER BY embedding <=> $2::vector) AS rank
  FROM documents
  ORDER BY embedding <=> $2::vector
  LIMIT 50
),
rrf AS (
  SELECT
    COALESCE(f.id, v.id) AS id,
    COALESCE(1.0 / (60 + f.rank), 0) +
    COALESCE(1.0 / (60 + v.rank), 0) AS rrf_score
  FROM fts_results f
  FULL OUTER JOIN vector_results v ON f.id = v.id
)
SELECT d.id, d.title, rrf.rrf_score
FROM rrf
JOIN documents d ON d.id = rrf.id
ORDER BY rrf_score DESC
LIMIT 10;
```

Число `60` в знаменателе — константа RRF. Более высокие значения сглаживают разницу позиций; более низкие — усиливают её. Значение по умолчанию 60 хорошо работает для большинства типов контента.

RRF позволяет обойти более сложную задачу нормализации `ts_rank` (логарифмическая частотная оценка) относительно косинусного расстояния (геометрическая мера). Они несопоставимы. RRF задаёт лишь один вопрос: «насколько высоко этот результат оказался в каждом из списков?»

### Гибридный поиск с триграммами

Для пользовательского поиска по смешанному содержимому — где в одном сеансе могут искать имя человека, понятие или точный термин — тройное слияние покрывает все три варианта:

```sql
WITH trgm_results AS (
  SELECT id,
    ROW_NUMBER() OVER (ORDER BY similarity(title, $1) DESC) AS rank
  FROM documents
  WHERE title % $1
  LIMIT 50
),
fts_results AS (
  SELECT id,
    ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, to_tsquery('english', $1)) DESC) AS rank
  FROM documents
  WHERE search_vector @@ to_tsquery('english', $1)
  LIMIT 50
),
vector_results AS (
  SELECT id,
    ROW_NUMBER() OVER (ORDER BY embedding <=> $2::vector) AS rank
  FROM documents
  ORDER BY embedding <=> $2::vector
  LIMIT 50
),
rrf AS (
  SELECT
    COALESCE(t.id, f.id, v.id) AS id,
    COALESCE(1.0 / (60 + t.rank), 0) +
    COALESCE(1.0 / (60 + f.rank), 0) +
    COALESCE(1.0 / (60 + v.rank), 0) AS rrf_score
  FROM trgm_results t
  FULL OUTER JOIN fts_results f ON t.id = f.id
  FULL OUTER JOIN vector_results v ON COALESCE(t.id, f.id) = v.id
)
SELECT d.id, d.title, rrf.rrf_score
FROM rrf
JOIN documents d ON d.id = rrf.id
ORDER BY rrf_score DESC
LIMIT 10;
```

Это покрывает: нечёткое сопоставление имён (триграммы), точное совпадение ключевых слов (FTS) и смысловые запросы (векторы). Одно поисковое поле может обслуживать все три типа пользовательских намерений.

## Многослойные гибридные архитектуры

Реальные приложения редко имеют одну поисковую поверхность. У их обычно несколько, и у каждой свои потребности:

| Поверхность | Что ищут пользователи | Рекомендуемые слои |
|---|---|---|
| Блог / документация | Ключевые слова + понятия | FTS + pgvector (RRF) |
| Поиск пользователей / клиентов | Имена с опечатками | `pg_trgm` |
| Поиск товаров | Названия, описания, «похожие на» | `pg_trgm` + FTS + pgvector |
| Дедупликация тикетов поддержки | «Задачи, похожие на эту» | Только pgvector |
| Внутренний поиск по артикулам / заказам | Точные идентификаторы | B-tree индекс |
| RAG по большой базе знаний | Вопросы на естественном языке | pgvector (фрагментированные документы) |
| E-commerce «вам может понравиться» | Поведенческое + семантическое сходство | pgvector |
| Автодополнение | По префиксу, с допуском опечаток | `pg_trgm` |

Это не теоретические примеры. Большинству приложений с большим объёмом контента нужно как минимум две разные поисковые поверхности с разными паттернами запросов. Соблазн — выбрать один подход и применить его везде — обычно сейчас это векторный поиск, потому что он в моде. Это приводит к дорогим векторизациям для задач, где триграммный индекс был бы быстрее, дешевле и точнее.

### Практическое правило

Добавляйте слой, когда появляется тип сбоев, который текущий слой не может исправить:

- Пользователи жалуются на опечатки, не дающие результатов → добавьте `pg_trgm`
- Пользователи ищут по смыслу и пропускают релевантные результаты → добавьте pgvector
- Пользователи ищут точные символы или коды и получают смысловые результаты вместо этого → добавьте FTS или проверьте, не слишком ли вы полагаетесь на векторный поиск
- Задержки становятся проблемой → оцените префильтрацию, приближённые индексы или выделенное хранилище

## Если вам действительно нужно выделенное векторное хранилище

pgvector покрывает значительную часть поисковых задач в приложении до того, как понадобится другая база данных. Примерная граница зависит от количества векторов, настроек индекса, интенсивности записи, фильтров, железа и параллелизма, поэтому относитесь к любому правилу «до 10M векторов» как к отправной точке для бенчмаркинга, а не к продуктовому ограничению. Когда вы действительно перерастёте его — очень высокий параллелизм, жёсткие требования к p99-задержке, миллиарды векторов или серьёзные потребности в изоляции арендаторов — ландшафт специализированных векторных баз данных широк и стоит того, чтобы в нём разобраться.

### Что на самом деле значат колонки в таблице

**Гибридный поиск** означает, что поиск по ключевым словам BM25 и векторное сходство выполняются в одном запросе и объединяются через RRF. Без него вы либо выбираете один режим поиска, либо объединяете два запроса вручную.

**Разреженные векторы (sparse vectors)** идут дальше BM25. Разреженный вектор SPLADE имеет ~30 000 измерений (по одному на термин словаря), ~98% из которых — нули. Ненулевые позиции указывают, какие термины важны и насколько. Запрос «dogs» также взвешивает «canine» и «pet» — точность на уровне BM25 плюс расширение терминов внутри векторного индекса. Если в этой колонке указано ❌, вам понадобится отдельный FTS-слой для запросов с точными терминами.

```python
# SPLADE: ~30 000 измерений, ~60 ненулевых — срабатывают только релевантные позиции словаря
def encode_splade(text: str) -> dict:
    tokens = tokenizer(text, return_tensors="pt", truncation=True, max_length=512)
    with torch.no_grad():
        output = model(**tokens)
    vec = torch.log1p(torch.relu(output.logits)).max(dim=1).values.squeeze()
    return {"indices": vec.nonzero().squeeze().tolist(), "values": vec[vec != 0].tolist()}
```

**SQL / SQL-подобный** — на самом деле речь о фильтрации. Векторный поиск без фильтрации — это демо. Вам по-прежнему нужны ограничения по арендатору, диапазоны дат, права доступа и фильтры по категориям. Полноценный SQL (pgvector, LanceDB) выражает всё это рядом с вашими существующими JOIN'ами. Специализированные базы данных используют JSON-объекты для фильтров (Qdrant, Pinecone), query DSL (Elasticsearch, Milvus) или GraphQL (Weaviate). Всё это работает; SQL становится привлекательнее по мере усложнения логики фильтрации.

```sql
-- pgvector: векторное сходство — просто ещё одно выражение
SELECT id, title, 1 - (embedding <=> $1) AS score
FROM documents
WHERE tenant_id = $2
  AND category = ANY($3::text[])
  AND created_at > NOW() - INTERVAL '90 days'
ORDER BY embedding <=> $1
LIMIT 10;
```

```python
# Qdrant: эквивалентный фильтр в виде Python-объекта — тот же результат, больше ceremony
results = client.query_points(
    collection_name="documents", query=query_embedding,
    query_filter=models.Filter(must=[
        models.FieldCondition(key="tenant_id", match=models.MatchValue(value=tenant_id)),
        models.FieldCondition(key="category",  match=models.MatchAny(any=categories)),
        models.FieldCondition(key="created_at", range=models.DatetimeRange(gte=cutoff)),
    ]),
    limit=10,
)
```

**Нативная мультимодальность** означает, что база данных поставляет модели векторизации для нетекстового содержимого. Вы передаёте ей URL сырого изображения, а она сама векторизует. Большинство баз данных агностичны к векторизации — вы владеете пайплайном. Marqo и Weaviate (через модули CLIP/ImageBind) замыкают этот цикл.

```python
# Marqo: POST сырых изображений, текстовый запрос — без внешнего этапа векторизации
mq.index("products").add_documents(
    [{"id": "shoe-001", "image": "https://cdn.example.com/shoes/001.jpg"}],
    tensor_fields=["image"]
)
results = mq.index("products").search(q="lightweight shoes for summer")
# Возвращает shoe-001 при нулевом пересечении ключевых слов — CLIP обрабатывает кросс-модальное сопоставление
```

**Дисковый индекс** — это рычаг экономии. HNSW-индексы, полностью находящиеся в оперативной памяти, могут требовать несколько ГБ RAM на миллион 1536-мерных векторов с учётом сырых векторов, накладных расходов графа и метаданных. Альтернативы, нативно работающие с диском (Milvus DiskANN, Elasticsearch DiskBBQ, Lance-формат LanceDB, объектное хранилище Turbopuffer), часто обменивают некоторую задержку запросов на снижение стоимости инфраструктуры. Для RAG-нагрузок, где задержка модели уже доминирует, этот компромисс часто стоит проверить на бенчмарках.

**Максимальное количество измерений** — это миграция, затаившаяся в вашей архитектуре. `text-embedding-3-large` использует 3072 измерения, Jina v3 может выдавать ещё большие векторы, а исследовательские модели продолжают раздвигать границы. Некоторые управляемые сервисы публикуют жёсткие ограничения; другие документируют высокие лимиты или их отсутствие для типичных моделей векторизации. Проверяйте актуальную документацию перед принятием решения. Выбирайте с запасом; миграция векторного индекса из-за достижения потолка по измерениям — болезненный спринт.

### Несколько вещей, не помещающихся в таблицу

**Мультиарендаторность Turbopuffer** построена вокруг очень большого числа пространств имён. Публичное позиционирование и истории клиентов подчёркивают нагрузки вроде корпуса Notion с большим количеством пространств имён. Если каждому пользователю или организации нужна изолированная область векторного поиска, такая архитектура может изменить экономику, но обязательно проверяйте на собственном профиле арендаторов.

**Режим embedded в LanceDB** — ближайший аналог «SQLite для векторного поиска». Работает in-process, не требует сервера, функционирует в Lambda, Cloudflare Workers и edge-окружениях. Колоночный формат Lance делает работу в embedded-режиме практичной на реальных масштабах.

**Chroma наиболее сильна в dev/test и небольших развёртываниях.** Если вы ориентируетесь на очень большие корпуса, высокую доступность, интенсивную работу с диском или первоклассный гибридный поиск, оцените ориентированное на production хранилище до того, как продвигать прототип в инфраструктуру.

**Vespa — это то, к чему приходят, когда поиск — лишь половина продукта.** Она объединяет лексическое извлечение, поиск ближайших соседей, тензоры, выражения ранжирования, группировку и онлайн-сервис. Эта мощь реальна, но реальна и операционная и модельная сложность. Она подходит командам, занимающимся поиском и рекомендациями, а не тем, кто хочет «добавить семантический поиск в CRUD-приложение».

**ClickHouse заслуживает внимания, когда поиск привязан к аналитике.** Если ваш источник истины — события, логи, трейсы или метрики, ClickHouse держит векторное расстояние, фильтрацию, агрегацию и серьёзное полнотекстовое индексирование в одном SQL-движке. Не специализированная векторная база, но часто скучно-правильный выбор для аналитического поиска.

**Разреженные векторы — это способ получить качество поиска по ключевым словам на уровне BM25 внутри векторного индекса** — без запуска отдельного полнотекстового движка. У Qdrant и Elasticsearch особенно зрелые реализации. Если гибридный поиск критичен, а двухсистемная архитектура неприемлема, поддержка разреженных векторов — вот что стоит искать.

### Как выбрать, когда вы переросли pgvector

- **SaaS-продукт с изоляцией по арендаторам** → Turbopuffer
- **Сложная фильтрация метаданных в большом масштабе** → Qdrant
- **Уже на Elastic/ELK-стеке** → Elasticsearch с DiskBBQ
- **AWS-инфраструктура, нужен open-source** → OpenSearch
- **Поисково-рекомендательная платформа с серьёзными требованиями к ранжированию** → Vespa
- **Аналитика, наблюдаемость, поиск по логам/событиям** → ClickHouse
- **Масштаб миллиардов, on-prem / self-hosted** → Milvus
- **Edge / serverless / мультимодальность** → LanceDB
- **Небольшое JS-приложение, документация или edge-нативный поиск** → Orama
- **Zero ops, стоимость вторична** → Pinecone
- **Мультимодальность в первую очередь (изображения, видео, аудио)** → Marqo
- **Уже на MongoDB** → Atlas Vector Search
- **Уже на Postgres, нужно больше возможностей** → Supabase Vector или Neon (оба — управляемый pgvector с лучшим инструментарием)

## Единственное, чего не стоит делать

Не используйте векторный поиск как нечёткий текстовый поиск для вещей, у которых есть правильные ответы.

«Найди пользователя с email `dan@example.com`» — это не задача векторного поиска. «Найди заказ с ID `ORD-12345`» — тоже. Векторизация `ORD-12345` и поиск по косинусному сходству вернёт *что-то* — но это может быть неверно. У идентификатора есть правильный ответ. Приблизительное совпадение идентификатора — это баг.

Векторный поиск всегда возвращает *наиболее похожий* элемент в вашем наборе данных, даже когда ничего релевантного на самом деле нет. Он не умеет определять, что хорошего ответа не существует. Для связанных документов это нормально. Для точного поиска записи, где уверенный неправильный ответ хуже пустого результата, это серьёзная проблема.

То же верно и в обратную сторону: не используйте FTS для запросов, где пользователь описывает понятие. «статьи о принятии трудных решений в условиях неопределённости» не содержит надёжных ключевых слов. FTS вернёт либо шум, либо ничего. Используйте правильный инструмент для формы запроса.

## Полная картина

Большинство production-систем поиска нуждаются более чем в одном слое:

- **`pg_trgm`** для имён, опечаток, автодополнения
- **FTS / `pg_search`** для поиска по ключевым словам в прозаическом тексте
- **pgvector** для семантических и смысловых запросов
- **RRF-слияние** для поверхностей, где пользователи комбинируют типы запросов
- **Обычные индексы** для точных идентификаторов, фильтров и отсортированных списков

Это не конкурирующие инструменты. Они взаимодополняющие. Хорошо построенная поисковая система выбирает правильный слой для каждой формы запроса — а когда формы запросов пересекаются, запускает несколько слоёв и объединяет результаты.

Команды, которые создают хорошие поисковые функции, понимают весь стек. Те, которые этого не делают, хватаются за векторную базу, векторизуют всё подряд и удивляются, почему точный поиск иногда возвращает неправильную запись.
