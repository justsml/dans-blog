>
  <slot name="question">
  <div className="question">
    Zeit für etwas verschachteltes Destructuring?
    ```js
    'use strict';
    const person = {
      name: { first: 'Dan' },
      address: { city: 'Denver' },
    };
    const {
      name: { first },
      address: { city },
      birth: { place },
    } = person;
    console.log(
      `First: ${first}, City: ${place}`,
    );
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Die Eigenschaft `birth: { place }` existiert nicht auf `person`, weshalb ein Fehler geworfen wird.
    Eine Lösung besteht darin, Standardwerte für verschachtelte Eigenschaften zu definieren.

    Beim Zugriff auf verschachtelte Eigenschaften – aufgepasst – können Fehler leicht übersehen werden. Fehlermeldungen unterscheiden sich zudem je nach Browser und Plattform, was das Debugging etwas kniffliger macht.

    In modernem Chrome: `TypeError: Cannot read properties of undefined (reading 'place')`

    In Node handelt es sich ebenfalls um einen `TypeError`, da JavaScript versucht, `place` aus `undefined` zu destrukturieren, bevor `place` überhaupt gelesen wird.

    Die exakte Formulierung variiert je nach Browser und Laufzeitumgebung.
  </div>
  </slot>
</Challenge>

    Die Eigenschaft `birth` existiert nicht auf `person`, daher benötigt das gesamte Objekt einen Standardwert, nicht nur die verschachtelte Eigenschaft. Im Grunde fehlt dort ein ` = {}` als Default.

    So wie der Code geschrieben ist, bedeutet er: „Wenn `person.birth` `undefined` ist, dann ist `place` `Unknown`“. Da `person.birth` aber `undefined` ist, versucht JavaScript, `undefined` zu destrukturieren – was zu einem Fehler führt.
    ```plaintext
    In modern Chrome: `TypeError: Cannot read properties of undefined (reading 'place')`

    In Node, this is also a `TypeError` because JavaScript tries to destructure `place` from `undefined`.

    Exact wording varies between browsers and runtimes.
    ```
  </div>
  </slot>
</Challenge>

    {text: 'Unknown, N/A, Joburg'},
    {text: 'Fehler'},
  ]}
>
  <slot name="question">
  <div className="question">
    Wie werden `undefined`-Werte behandelt?
    ```js
    'use strict';
    function displayPlace({
      name = "N/A",
      place = "N/A",
      age = -1,
    } = { place: "Unknown" }) {
      console.log(`${place}`);
    }
    displayPlace({ name: "Dan" });
    displayPlace({ name: "Dan", place: undefined });
    displayPlace({ name: "Dan", place: "Joburg" });
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Die Funktion `displayPlace` verwendet NUR dann ein Standardobjekt, wenn gar kein Objekt übergeben wird. Der einzige Weg, den Standardwert `{ place: "Unknown" }` zu erhalten, ist also der Aufruf ohne Argumente: `displayPlace()`.

    Ein weiteres bemerkenswertes Verhalten ist hier, dass die Übergabe von `undefined` für `place` dazu führt, dass der Standardwert genutzt wird – ähnlich wie beim Verhalten von `JSON.stringify` (ignoriert `undefined`, erkennt `null`).

    Das führt zu:
    ```js
    displayPlace() // Unknown
    displayPlace({ name: "Dan" }) // N/A
    displayPlace({ name: "Dan", place: undefined }) // N/A
    ```
  </div>
  </slot>
</Challenge>

    ```ts
        birth: {place: p = "N/A"} = {},
        age = -1,
      }: {

        birth?: {place?: string};
        age?: number;
      }
    ) {
      console.log(`Hi ${f} ${l} from ${place}`);
      // What will 👆 do?

      birth: {place: 'Cape Town'},
    });
    ```

  </slot>

  <div className="explanation">
    Noch ein Fehler! Du fängst an zu raten, nicht wahr?!

    Verschachtelte Destructuring-Ebenen mit Standardwerten, Zuweisungen und Typen sind schwer zu lesen!

    Sobald `place` der Variablen `p` neu zugewiesen wird, ist sie im Geltungsbereich der `console.log`-Anweisung nicht mehr definiert.
    ```ts
    console.log(`Hi ${f} ${l} from ${place}`); // ❌
    // to:
    console.log(`Hi ${f} ${l} from ${p}`); // ✅
    ```
