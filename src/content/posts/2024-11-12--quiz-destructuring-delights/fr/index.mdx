  <div className="question">
    Et si on essayait la destructuration imbriquée ?
    ```js
    'use strict';
    const person = {
      name: { first: 'Dan' },
      address: { city: 'Denver' },
    };
    const {
      name: { first },
      address: { city },
      birth: { place },
    } = person;

      `First: ${first}, City: ${place}`,
    );
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La propriété `birth: { place }` n'existe pas sur `person`, donc cela générera une erreur.
    Une solution consiste à fournir des valeurs par défaut pour les propriétés imbriquées.

    Lors de l'accès à des propriétés imbriquées, soyez prudent - car les erreurs peuvent être difficiles à repérer. Et les messages d'erreur varient entre les navigateurs et autres plateformes, ce qui rend le débogage un peu plus difficile.

    Dans Chrome moderne : `TypeError: Cannot read properties of undefined (reading 'place')`

    Dans Node, c'est aussi une `TypeError` car JavaScript essaie de déstructurer `place` à partir de `undefined` avant même que `place` ne soit lue.

    Le libellé exact varie selon les navigateurs et les environnements d'exécution.
  </div>
  </slot>
</Challenge>

    ```js
    'use strict';
    function displayPlace({

      age = -1,
    } = { place: "Unknown" }) {
    ```

    La fonction `displayPlace` n'utilisera un objet par défaut QUE si aucun objet n'est passé. Donc, la seule façon d'obtenir la valeur par défaut `{ place: "Unknown" }` est avec zéro argument `displayPlace()`.

    Un autre comportement notable ici est que passer `undefined` pour `place` entraînera l'utilisation de la valeur par défaut, un peu comme le comportement de `JSON.stringify` (ignorer `undefined`, reconnaître `null`).

    Cela donne :
    ```js
      displayPlace() // Unknown
      displayPlace({ name: "Dan" }) // N/A
      displayPlace({ name: "Dan", place: undefined }) // N/A
    ```

  ]}
>
  <slot name="question">
  <div className="question">
    Repérez l'erreur de type :
    ```ts
    function greet({
      name: {first = "N/A", last = "N/A"},
      birth: {place = "N/A"} = {},
      age = -1,
    }: {
      name: {first?: string, last?: string};
      birth: {place?: string};
      age: number;
    }) {
      console.log(`Hi ${first} ${last} from ${place}`);
    }
    greet({ name: {first: 'Dan'} });
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    L'erreur se trouve dans la signature de la fonction `greet`. Les propriétés `age` et `birth` sont manquantes dans l'objet passé, elles devraient donc être optionnelles dans la définition de type.

    Même si la propriété `birth` est déstructurée avec une valeur par défaut, la définition de type exige qu'elle soit présente. Pour marquer une propriété comme optionnelle en TypeScript, il faut utiliser l'opérateur `?`.

    Notez que `birth?: { place?: string }` n'est pas la même chose que `birth: { place?: string } | undefined`.
  </div>
  </slot>
</Challenge>

    ```ts
    'use strict';
    function greet(
      {
        name: {first: f = "N/A", last: l = "N/A"},
        birth: {place: p = "N/A"} = {},
        age = -1,
      }: {
        name: {first?: string, last?: string};
        birth?: {place?: string};
        age?: number;
      }
    ) {
      console.log(`Hi ${f} ${l} from ${place}`);
      // What will 👆 do?
    }
    greet({
      name: {first: 'Dan', last: 'Levy'},
      birth: {place: 'Cape Town'},
    });
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Encore une erreur ! Vous commencez à deviner, n'est-ce pas ?!

    Il est difficile de lire les couches de destructuration, avec les valeurs par défaut, l'affectation et les types !

    Dès que `place` est réaffecté à la variable `p`, il n'est plus défini dans la portée de l'instruction `console.log`.
    ```ts
      console.log(`Hi ${f} ${l} from ${place}`); // ❌
      // to:
      console.log(`Hi ${f} ${l} from ${p}`); // ✅
    ```
  </div>
