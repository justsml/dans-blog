開発者のノートPC向けのセキュリティ助言は、だいたい次の2つのどちらかで失敗します。

1つ目は、エンタープライズ向けの壁紙みたいな言い方です。

> エンドポイント保護を使い、定期的にパッチを当て、不審なリンクを避け、インシデントは速やかに報告してください。

全部正しいです。ですが、それだけでは足りません。

2つ目は、生存者のためのナンセンスで、ブラウザ、JavaScript、Wi-Fi、パッケージマネージャ、ベンダー、PDF、チャット、コードエディタ、スマホ、そして喜びまで使うのをやめろ、という方向に行くやつです。

それもまた真面目に無意味です。

現実的な目標は、もっと小さくすることです。

> 自分の権限で動くものは、自分が信頼している“すべて”を自動で継承してはいけない。

これがワークステーション爆発半径（blast-radius）の問題です。

開発が「湿ったセメントをタイプするみたいに」感じられないまま、これを減らすためのガイドです。

最終確認日: 2026年5月9日。ツールの挙動、価格、対応プラットフォームは変わります。チームで標準化する前に最新ドキュメントを確認してください。

---

## 防御の形

必要なのは4つの層です。

| 層 | 役割 |
| --- | --- |
| 隔離 | プロジェクトのツールや危険なコマンドを、残りのマシンから遠ざける。 |
| シークレット管理 | 平文の資格情報を減らし、機密値がうっかり漏れにくくする。 |
| 検知 | 攻撃者や不正な自動化が自然に触れそうな“仕掛け”を置く。 |
| エグレス制御 | 予期しないアウトバウンド接続を検知してブロックする。 |

最初から、ノートPCのあらゆる脅威を全部解こうとしないでください。

攻撃者が実際にやりたがるルートから始めます。
つまり「何かを動かして、シークレットを読み取り、送信して、誰かが気づく前に使う」です。

## 1. プロジェクトを開発コンテナに入れる

[Dev Containers](https://github.com/devcontainers/spec) を使うと、コンテナを“完全な開発環境”として使えます。これは開発者体験（DX）のインフラっぽく聞こえますが、規律を持って使うなら、それもまたセキュリティ境界になります。

雑な設定だと、マウントしすぎてしまいます。

```jsonc
// Too convenient. Too much blast radius.
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "mounts": [
    "source=${localEnv:HOME},target=/host-home,type=bind"
  ]
}
```

それは、コンテナを「妙に形の歪んだホストのアカウント」にしてしまいます。

マウントはもっと狭くしましょう。

```jsonc
// .devcontainer/devcontainer.json
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "workspaceFolder": "/workspaces/app",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ],
  "containerEnv": {
    "NODE_ENV": "development"
  },
  "postCreateCommand": "bun install"
}
```

完璧なサンドボックスではありません。コンテナはカーネルを共有します。Dockerには鋭い刃があります。マウントは“モデル”をまっすぐ貫通する穴を開けます。

しかし、多くの開発ワークフローでは得るものが即座にあります。プロジェクトのコマンドは「あなたのデジタルな屋根裏全部」を見るのではなく、「プロジェクト」を見るようになります。

### 何をマウントするか

リポジトリをマウントします。

必要なら、プロジェクト専用のキャッシュもマウントします。

基本はマウントしないでください。

- `~/.ssh`
- `~/.aws`
- `~/.config/gcloud`
- `~/.azure`
- `~/Downloads`
- `~/Documents`
- パスワードマネージャのエクスポート
- データベースのダンプ
- バックアップフォルダ
- 2021年からずっと存在している“適当なtemp”フォルダ

もしプロジェクトがクラウドアクセスを必要とするなら、そのプロジェクト用の資格情報を注入してください。短命（短い有効期限）が良い。読み取り専用が良い。開発用アカウントだけにアクセスできるトークンは、自分の個人の管理者アイデンティティを小さなスーツケースみたいに連れてコンテナへ入れるよりずっと良い。

### AIコーディングツールもここに置くべき

AIコーディングツールは、開発コンテナを重要にする方向に働きます。重要性は下がりません。

Anthropicの[Claude Codeの権限ドキュメント](https://code.claude.com/docs/en/permissions)では、世界を「権限」と「サンドボックス」に分けています。権限はツール、ファイル、ドメインを制御します。一方サンドボックスは、Bashのファイルシステムとネットワークアクセスに対してOSレベルの強制を提供します。

この区別が、この勝負の全てです。

エージェントがシェルコマンドを実行でき、パッケージをインストールでき、ファイルを調べられ、指示に従って動けるなら、その“シェル作業”は制約されたプロジェクト環境の中に置きましょう。ホストは退屈にしておく。

良いデフォルトです。

- エージェントはホームディレクトリではなく、リポジトリで起動する
- 機微なパスは明示的に拒否する
- インストール/ビルド/テストのコマンドはDev Containerで実行する
- 資格情報や認証設定、パッケージのパブリッシュ、クラウドリソースに触れる“広いextraディレクトリ”を追加しない
- 資格情報、認証設定、パッケージ公開、クラウドリソースに影響する生成コマンドは必ずレビューする

モデルは`~/Documents`フォルダを必要としていません。TypeScriptのエラーを直すのに、そこは要りません。

## 2. 平文の `.env` が増殖しすぎる状態（スプロール）を置き換える

`.env` ファイルは悪ではありません。

ファイルはファイルです。それが問題です。

ファイルはコピーされます。ファイルはインデックスされます。ファイルはマウントされます。見た目はCSSのリントだけをするはずのスクリプトが、それを読みます。デバッグ用のzipに含まれます。チャットに貼られます。誰かが「最後の12行」を忘れただけで。

退屈な階層を使いましょう。

1. 秘密が不要なら、`.env.example` に値を置く
2. ローカル専用の秘密なら、保管時に暗号化する
3. 共有開発の秘密なら、本当のシークレットマネージャやパスワードマネージャに置く
4. 本番の秘密なら、開発者のノートPCに入れない（どうしても必要な具体的理由がある場合を除く）

[VarLock](https://varlock.dev/guides/secrets/) は魅力的です。機微さを明示できるからです。ドキュメントには `@sensitive` で値をマークすること、ローカル値を `varlock()` で暗号化すること、機微な値をコンソール出力から秘匿（redact）すること、既知の機微値の平文出現をプロジェクトファイルでスキャンすることなどが書かれています。

“秘密っぽい形に見えることを期待してリポジトリに対して正規表現を当てる”より、仕組みの形が良いです。

例：

```dotenv
# .env.schema
# @defaultSensitive=false

PUBLIC_APP_NAME=

# @sensitive
STRIPE_SECRET_KEY=

# @sensitive
DATABASE_URL=
```

ローカル上書き：

```dotenv
# .env.local
PUBLIC_APP_NAME=demo
STRIPE_SECRET_KEY=varlock(local:...)
DATABASE_URL=varlock(local:...)
```

これは「読み込んだあとに、安全」と言っているわけではありません。侵害されたプロセスの中に入ったら、何でも安全ではありません。ですが、それでも「ファイルシステムに平文の“景品”が増える」ことを減らせます。

それが効くのは、インフォスティーラー（情報窃取系のマルウェア）、悪意のある依存、AIコンテキストの過剰共有、うっかりコミット、そしておなじみの`console.log(process.env)`の瞬間に対してです。

## 3. 泥棒が見に来る場所にカナリートークンを置く

ほとんどの監視は「既知の悪いこと」が起きたときに教えてくれます。

カナリートークンは、「変なこと」が起きたときに教えてくれます。つまり“知らないはずのもの”に触れたときに。

[Thinkst Canarytokens](https://help.canary.tools/hc/en-gb/articles/10905485310109-Canarytoken-Overview-and-Use-Cases) は、これらをデジタルな踏み台（tripwires）として説明しています。文書、URL、APIキー、VPNプロフィール、QRコード、そしてその他の“存在しているふりをした資産”で、アクセスされるとアラートが上がります。

置き場所が腕の見せ所です。

勝ちを狙って“ばら撒き”をしてから「ほら見たことか」と言わないでください。狙いは「資格情報の窃取」「バックアップの窃取」「偵察（recon）」が自然に起きそうな場所に、カナリーを置くことです。

### ローカルのカナリー

偽のバックアップを作ります。

```text
~/backups/customer-prod-export-2024.sql
```

その中にカナリーのURLやトークンを入れます。

```sql
-- legacy analytics webhook
-- https://canarytokens.example.invalid/static/abc123
```

偽の認証情報ファイルも作れます。

```text
~/Documents/passwords-old.csv
```

あるいは、偽のAWSプロフィール。

```ini
# ~/.aws/credentials
[billing-prod-legacy]
aws_access_key_id = AKIA...
aws_secret_access_key = ...
```

可能なら、実在するAWSのカナリートークン種別を使ってください。そうすれば“ファイルが開かれた”だけではなく、“実際に使われた”ときにアラートが出ます。

### リポジトリのカナリー

ソースアクセス後に攻撃者が調べに来る場所の近くに置きます。

- 内部の実行手順（runbooks）
- 廃止されたデプロイメントドキュメント
- 古い移行ノート
- 明らかに本番ではない `.env.canary` に入れた偽のサービス認証情報
- 偽のバックアップ復元手順

これもまた、セキュリティ・バイ・オブスキュリティ（隠して守る）ではありません。廊下にアラームを鳴らすだけです。

### CIとクラウドのカナリー

クラウドの踏み台として良い場所です。

- 偽のCIシークレット
- 偽のデプロイトークン
- 権限のない偽のDBユーザー
- 使われていないオブジェクトストレージのパス
- 偽のkubeconfig
- 実行手順（runbook）に記載された偽のAPIキー

アラートは“実行可能”にしましょう。メールを無人の受信箱へ送るだけのカナリーは、飾り文字列です。

最低限、アラートは次を伝えるべきです。

- どのトークンが発火したか
- どこに仕掛けたか
- どのシステムが触ったか
- 何をローテーション（入れ替え）する必要があるか
- 誰が対応するべきか

## 4. アウトバウンド通信にゲートをかける

ローカルで何か悪意あるものが動いた場合、情報流出（exfiltration）にはネットワークの通り道が必要です。

ほとんどの開発者用ノートPCは、アウトバウンド通信をデフォルトで許可しています。便利だからです。ですが、だからこそ未知のプロセスが“ローカルの判断なし”に未知の宛先へデータを送ってしまうことも、現実には起きやすいです。

アウトバウンドファイアウォールはシートベルト層です。

すべてのクラッシュを止められるわけではありません。ですが、一部のクラッシュを“生存可能”にします。さらに、普通の状態を覚えさせないと、都合の悪いタイミングで文句も言います。

### macOS

[LuLu](https://objective-see.org/products/lulu.html) は無料でオープンソースです。Objective-Seeの説明では、未知の送信（outgoing）の接続をブロックします。そしてドキュメントには、LuLuはアウトバウンド通信のみを監視すると書かれています。

単純なアウトバウンドのプロンプトが欲しくて、セットアップの手間を多少許容できるなら、まず最初の選択肢として良いです。

[Little Snitch](https://obdev.at/products/littlesnitch/) は商用で、より洗練されています。接続アラートを出し、アプリごとに許可/拒否でき、ネットワークモニタではアプリ、ドメイン、国、ポート、プロトコル、トラフィックの可視性を提供します。

2週間目以降もUIを使い続けられる“強い選択”です。

### Windows

Windows Defender Firewallは、アウトバウンドルールとルールの優先順位（precedence）をサポートします。Microsoftのガイダンスは慎重です。アウトバウンドルールをブロックに変更することは高セキュリティ環境では検討できるが、その場合はアプリを棚卸しして、ネットワーク接続が必要なものに対してルールを作る必要がある、としています。

翻訳：可能で強力だが、面倒になりやすい。

[Portmaster](https://safing.io/) もWindowsで評価する価値があります。Safingの説明では、これはオープンソースのアプリケーションファイアウォールで、ネットワーク接続を監視し、アプリごとのブロックルールを設定します。

### Linux

Portmasterは一般的なLinuxパッケージをサポートしています。OpenSnitchも、評価する価値がある別のLinuxアプリケーションファイアウォールですが、プロジェクトの状態とディストリビューションのパッケージング状況は標準化する前に確認してください。

サーバなら通常のサーバ制御を使います。開発者用ノートPCで重要なのは、アプリケーションレベルの可視性です。「アウトバウンドを全部ブロックして443だけ許可」では足りません。なぜなら興味深い情報流出の経路ほど、だいたい443で会話するからです。

## 5. バックアップに大人の監督をつける

バックアップは冷静じゃありません。ポータブルな形で持ち運べる“機微なデータ”です。

開発者のマシンが、バックアップアーカイブになる必要はありません。バックアップがその役目なら、その仕事をするべきです。

実際に強制したいルールです。

- 本番エクスポートには、オーナーと期限（expiration date）を付ける
- ローカルのDBダンプは暗号化する
- 認証情報を含むエクスポートは、認証情報のローテーションまたはスクラブ（除去）をトリガーにする
- バックアップフォルダはデフォルトでDev Containerにマウントしない
- バックアップフォルダはデフォルトでAIコーディングツールに対して拒否する
- バックアップっぽい保管領域には少なくとも1つカナリーを置く
- 古いエクスポートは“気分”ではなく自動化で削除する

簡単なローカルの慣習：

```bash
mkdir -p ~/sensitive-exports
chmod 700 ~/sensitive-exports
```

より良い慣習：

- 暗号化ボリュームまたは暗号化アーカイブ
- 期限が分かる明確な命名
- 削除の手順が文書化されていること
- 承認なしでコンシューマ向けクラウドドライブへ同期しないこと

例：

```bash
age -r age1yourpublickeyhere -o customer-export-2026-05-09.sql.gz.age customer-export.sql.gz
shred -u customer-export.sql.gz
```

儀式にしないでください。最も良いバックアップポリシーは、そもそも開発者が本番エクスポートを必要とする頻度が低い状態を作れるものです。

## 6. ワークステーションのデフォルトを作る

個人の開発者向けに、まともなベースラインです。

| 領域 | ベースライン |
| --- | --- |
| ブラウザ | 本番のパスワードは保存しない。重要なアカウントはパスワードマネージャとハードウェア対応のMFAを使う。 |
| プロジェクト | パッケージのインストールが必要、信用できないコード、またはAI駆動のシェル作業があるプロジェクトはDev Containerを使う。 |
| シークレット | ディスク上に平文の本番シークレットを置かない。可能ならローカル開発シークレットは暗号化する。 |
| クラウド | 短命の資格情報。開発用と本番用のアイデンティティを分離する。個人の管理者トークンは原則として持たない。 |
| GitHub | 権限の細かいトークン。パッケージ公開トークンはレビューする。組織SSOとハードウェアキーを使う。 |
| AIツール | プロジェクトスコープのアクセス。機微なパスは拒否する。実行コマンドは可能な限りコンテナで動かす。 |
| バックアップ | 暗号化し、期限を設定し、隔離し、監視する。幅広いマウントやAIコンテキストに入れない。 |
| ネットワーク | アウトバウンドファイアウォールは“アラート/監視モード”から始め、次にリスクの高いツールにルールを当てる。 |
| 検知 | バックアップ、認証情報、CI、クラウド、ドキュメントの置き場所にカナリートークンを設置する。 |

チームなら、これに加えて。

- 標準の`.devcontainer`テンプレート
- ローカル/共有開発/ステージング/本番を区別するシークレットポリシー
- カナリートークンの設置規約
- 文書化されたアウトバウンドファイアウォールのプロファイル
- 迅速な認証情報ローテーション手順（プレイブック）
- シアトー（演劇）じゃなくて、脅威モデルを説明するオンボーディング

目的は、全開発者をセキュリティエンジニアにすることではありません。

目的は、より安全な道が“普通”になることです。

## 今週やること

大きく感じたなら、次の5つだけで十分です。

1. 高リスクなリポジトリを1つ選び、狭いマウントのDev Containerを追加する
2. 平文の `.env.local` 秘密を1つ、暗号化されたローカルストレージかパスワードマネージャへ移す
3. 偽のバックアップファイルに1つカナリーを仕込み、アラートが見える場所へルーティングする
4. LuLu、Little Snitch、Portmaster、または同等のものを“モニタモード”でインストールし、実際に何が通信しているかを見る
5. ローカルの本番エクスポートを見つけて、削除するか暗号化するか期限を切る

それでスタートできます。

セキュリティ作業が失敗するのは「到着の仕方が大聖堂」になってしまうからです。まずドアを作りましょう。次に鍵。その次にアラーム。最後に習慣。

ワークステーションは、完璧に信頼されていなくてもいい。

必要なのは、うっかり無限に“信頼され続ける状態”を止めることです。

## 画像プラン

表紙の方向性の案です。

- 図解的なマップ：中央にノートPCを置き、外側に4つの制約されたリングを描いて、隔離、シークレット、検知、エグレス（送信）とラベルを付ける。実務寄りのガイドに向いています。
- 編集的な比喩：鍵や書類、ネットワークケーブルがガラスのドームの下にある作業机。警告灯へ繋がるケーブルを1本だけ作る。シリーズのビジュアルアイデンティティに向いています。
- フェイルモードの場面：ローカルのバックアップフォルダが、本番インフラのように発光し、その周りに小さなトリップワイヤが取り囲む。バックアップのリスクを強めに推すならこちら。

方向性が決まったら、使うアセット一式を次のようにします。

- `desktop-social.webp` at 1200x630
- `wide.webp` at 1600x900
- `square.webp` at 800x800
