開発者向けラップトップのセキュリティアドバイスは、大抵二つのどちらかで失敗する。

エンタープライズ壁紙だ。

> エンドポイント保護を使い、定期的にパッチを当て、不審なリンクを避け、インシデントを速やかに報告しましょう。

全部正しい。でも足りない。

あるいはサバイバリストの無意味さで、ブラウザ、JavaScript、Wi-Fi、パッケージマネージャー、ベンダー、PDF、チャット、コードエディタ、スマホ、そして喜びを使わないことが答えだと言う。

これも役に立たない。

現実的な目標はもっと小さい。

> あなたとして実行される何かが、あなたに信頼されているすべてのことを自動的に引き継ぐべきではない。

それがワークステーションの爆発半径問題だ。

開発が濡れたセメントを通してタイピングしているような気分にならないように、これを縮めるためのガイドだ。

最終確認: 2026年5月9日。ツールの動作、料金、プラットフォーム対応は変わるので、チーム全体で標準化する前に最新のドキュメントを確認すること。

## 防御の形

4つのレイヤーが必要だ。

| レイヤー | 役割 |
| --- | --- |
| 分離 | プロジェクトツールとリスクの高いコマンドをマシンの残りの部分から遠ざける。 |
| シークレット管理 | 平文の認証情報を減らし、機密値が意図せず漏洩しにくくする。 |
| 検出 | 攻撃者や不適切な自動化が自然に探す場所に罠を仕掛ける。 |
| 送信制御 | 予期しない外部接続に気づき、ブロックする。 |

すべてのラップトップ脅威を解決しようとするところから始めるな。

攻撃者が実際に楽しむパスから始めろ：何かを実行し、シークレットを読み取り、外部に送信し、誰も気づく前に利用する。

## 1. プロジェクトを Dev Container に入れる

[Dev Containers](https://github.com/devcontainers/spec)を使うと、コンテナをフル機能の開発環境として使える。開発者体験インフラとして聞こえるだろうが、実際そうだ。だが、規律を持って使えばセキュリティ境界にもなる。

怠惰な設定はマウントしすぎる：

```jsonc
// 便利すぎる。爆発半径が大きすぎる。
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "mounts": [
    "source=${localEnv:HOME},target=/host-home,type=bind"
  ]
}
```

これはコンテナをホストアカウントの歪な形に変えてしまう。

狭いマウントを使え：

```jsonc
// .devcontainer/devcontainer.json
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "workspaceFolder": "/workspaces/app",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ],
  "containerEnv": {
    "NODE_ENV": "development"
  },
  "postCreateCommand": "bun install"
}
```

これは完璧なサンドボックスではない。コンテナはカーネルを共有する。Dockerには鋭いエッジがある。マウントはモデルを素通りできる穴を開けられる。

しかし、ほとんどの開発ワークフローでは、メリットは即座に得られる：プロジェクトコマンドはプロジェクトを見るだけで、あなたのデジタル屋根裏全部を見るわけではない。

### マウントすべきもの

リポジトリをマウントする。

プロジェクト固有のキャッシュをマウントしてもいい。

デフォルトでは以下をマウントするな：

- `~/.ssh`
- `~/.aws`
- `~/.config/gcloud`
- `~/.azure`
- `~/Downloads`
- `~/Documents`
- パスワードマネージャーのエクスポート
- データベースダンプ
- バックアップフォルダ
- 2021年から存在している謎の「temp」フォルダ

プロジェクトがクラウドアクセスを必要とするなら、そのプロジェクト用に作った認証情報を注入しろ。短期がいい。読み取り専用がいい。dev アカウントだけにアクセスできるトークンは、小さなスーツケースを持ってコンテナに入り込む個人管理者 ID よりもいい。

### AI コーディングツールもここに入れるべきだ

AI コーディングツールは Dev Containers をより重要にする。そうでなくするわけではない。

Anthropic の [Claude Code 権限ドキュメント](https://code.claude.com/docs/en/permissions)は、世界を権限とサンドボックスに分割している：権限はツール、ファイル、ドメインを制御し、サンドボックスは Bash ファイルシステムとネットワークアクセスに OS レベルの強制力を提供する。

その区別がすべてだ。

エージェントがシェルコマンドを実行でき、パッケージをインストールでき、ファイルを検査でき、指示に従えるなら、シェル作業を制約されたプロジェクト環境に入れろ。ホストは退屈なままにしておけ。

良いデフォルト：

- ホームディレクトリではなくリポジトリでエージェントを起動する
- 機密パスを明示的に拒否する
- install/build/test コマンドに Dev Container を使う
- 広範な「追加ディレクトリ」をコンテキストに追加しない
- 認証情報、認証設定、パッケージ公開、クラウドリソースに触れる生成コマンドは確認する

モデルは TypeScript のエラーを直すために `~/Documents` フォルダを必要としない。

## 2. 平文 `.env` の蔓延を置き換える

`.env` ファイルは邪悪ではない。

ただのファイルだ。それが問題だ。

ファイルはコピーされる。ファイルはインデックスされる。ファイルはマウントされる。ファイルは CSS の lint だけのはずのスクリプトに読まれる。ファイルはデバッグ用 zip に含まれる。ファイルはチャットに貼り付けられる。誰かが助けを求めて、最後の12行を忘れたからだ。

退屈な階層を使え：

1. シークレット不要：`.env.example` に値を入れる。
2. ローカル限定シークレット：保存時に暗号化する。
3. 共有開発シークレット：本物のシークレットマネージャーかパスワードマネージャーに入れる。
4. 本番シークレット：非常に具体的な理由がない限り、開発者のラップトップに置かない。

[VarLock](https://varlock.dev/guides/secrets/)は機密性を明示的にする点で魅力的だ。そのドキュメントは `@sensitive` で値をマークし、`varlock()` でローカル値を暗号化し、コンソール出力から機密値を編集し、プロジェクトファイル内の既知機密値の平文出現をスキャンすることを説明している。

その形は「リポジトリに正規表現を実行して、シークレットがシークレットっぽく見えることを祈る」よりずっといい。

例示的な方向：

```dotenv
# .env.schema
# @defaultSensitive=false

PUBLIC_APP_NAME=

# @sensitive
STRIPE_SECRET_KEY=

# @sensitive
DATABASE_URL=
```

ローカルオーバーライド：

```dotenv
# .env.local
PUBLIC_APP_NAME=demo
STRIPE_SECRET_KEY=varlock(local:...)
DATABASE_URL=varlock(local:...)
```

これは侵害されたプロセスに読み込まれたシークレットが安全だという意味ではない。何もそうはならない。しかし、ファイルシステムに平文の賞品が減るという意味はある。

それはインフォスティーラー、悪意のある依存関係、広範すぎる AI コンテキスト、うっかりコミット、そしてささやかな `console.log(process.env)` の瞬間に対して意味がある。

## 3. 泥棒が探す場所に Canary トークンを仕掛ける

ほとんどの監視は、既知の悪いことが起きたときに教えてくれる。

Canary トークンは、奇妙な何かが存在を知るはずのない何かに触れたときに教えてくれる。

[Thinkst Canarytokens](https://help.canary.tools/hc/en-gb/articles/10905485310109-Canarytoken-Overview-and-Use-Cases)はこれらをデジタルの罠と説明している。ドキュメント、URL、API キー、VPN プロファイル、QR コード、その他の偽アセットで、アクセス時にアラートを発する。

配置が芸術だ。

ランダムな餌をばらまいて勝利を宣言するな。認証情報の盗難、バックアップの盗難、偵察が自然に行く場所に Canary を置け。

### ローカル Canary

偽のバックアップを作る：

```text
~/backups/customer-prod-export-2024.sql
```

中に Canary URL かトークンを入れる：

```sql
-- legacy analytics webhook
-- https://canarytokens.example.invalid/static/abc123
```

偽の認証情報ファイルを作る：

```text
~/Documents/passwords-old.csv
```

または偽の AWS プロファイル：

```ini
# ~/.aws/credentials
[billing-prod-legacy]
aws_access_key_id = AKIA...
aws_secret_access_key = ...
```

可能なときは本物の AWS Canary トークンタイプを使え。そうすればアラートはファイルオープンだけでなく、使用試行時に発火する。

### リポジトリ Canary

ソースアクセス後に攻撃者が検査する場所の近くに Canary を置く：

- 内部ランブック
- 非推奨のデプロイドキュメント
- 古いマイグレーションノート
- 明らかに非本番の `.env.canary` 内の偽サービス認証情報
- 偽のバックアップ復旧手順

これはセキュリティ through オブスキュリティではない。廊下に置かれた警報だ。

### CI とクラウド Canary

良いクラウド罠の配置場所：

- 偽の CI シークレット
- 偽のデプロイトークン
- 権限のない偽データベースユーザー
- 未使用のオブジェクトストレージパス
- 偽の kubeconfig
- ランブックに記載された偽 API キー

アラートを実行可能にしろ。誰も見ていない受信箱にメールする Canary は、装飾用の文字列だ。

最低限、アラートは以下を伝えるべきだ：

- どのトークンが発火したか
- どこに仕掛けたか
- 何がそれに触れたか
- 何をローテーションすべきか
- 誰が対応を担当するか

## 4. 送信トラフィックにゲートを置く

悪意のある何かがローカルで実行されたら、データ持ち出しにはネットワークパスが必要だ。

ほとんどの開発者ラップトップはデフォルトで送信トラフィックを許可する。便利だ。未知のプロセスがローカルの判断ポイントなしに未知の場所にデータを送れることも意味する。

送信ファイアウォールはシートベルトレイヤーだ。

すべてのクラッシュは防げない。一部のクラッシュは生存可能にする。また、正常な姿を教えるまでは不便な時に文句を言うだろう。

### macOS

[LuLu](https://objective-see.org/products/lulu.html)は無料でオープンソースだ。Objective-See は未知の送信接続をブロックすると説明し、ドキュメントでは LuLu が送信トラフィックのみを監視すると記している。

シンプルな送信プロンプトが欲しく、いくらかの設定摩擦に耐えられるなら良い第一選択だ。

[Little Snitch](https://obdev.at/products/littlesnitch/)は商用で洗練されている。接続アラートを表示し、アプリ接続の許可・拒否を可能にし、アプリ、ドメイン、国、ポート、プロトコル、トラフィックの可視性を持つネットワークモニターを提供する。

プロファイル、ルール管理、そして2週間後も実際に使い続けるような UI が欲しいなら強い選択だ。

### Windows

Windows Defender ファイアウォールは送信ルールと、送受信トラフィックのルール優先度をサポートする。Microsoft のガイダンスは穏やかだ：送信ルールをブロックに変更することは高セキュリティ環境で検討できるが、アプリのインベントリと、ネットワーク接続が必要なもののルール作成が必要だ。

翻訳：可能で強力だが、面倒にするのは簡単だ。

[Portmaster](https://safing.io/)も Windows で評価する価値がある。Safing はネットワーク接続を監視し、アプリケーションごとのブロックルールを設定するオープンソースのアプリケーションファイアウォールだと説明している。

### Linux

Portmaster は一般的な Linux パッケージに対応している。OpenSnitch も評価する価値のある Linux アプリケーションファイアウォールだが、標準化前にプロジェクトの状態とディストリビューションパッケージングを確認すべきだ。

サーバーには通常のサーバー制御を使え。開発者ラップトップでは、重要な機能はアプリケーションレベルの可視性だ。「443以外の送信をすべてブロック」では、興味深いデータ持ち出しパスのすべてが443も話すなら不十分だ。

## 5. バックアップに大人の監督をつける

バックアップは冷えていない。持ち運び可能な形式の機密データだ。

開発マシンはそれが仕事でない限りバックアップアーカイブになるべきではない。

実際に強制するルール：

- 本番エクスポートには責任者と有効期限が必要。
- ローカルデータベースダンプは暗号化必須。
- 認証情報を含むエクスポートは認証情報のローテーションまたは除去をトリガーする。
- バックアップフォルダはデフォルトで Dev Container にマウントしない。
- バックアップフォルダはデフォルトで AI コーディングツールに拒否する。
- バックアップに似たストレージには少なくとも1つの Canary を置く。
- 古いエクスポートは雰囲気ではなく自動化で削除する。

シンプルなローカル規約：

```bash
mkdir -p ~/sensitive-exports
chmod 700 ~/sensitive-exports
```

より良い規約：

- 暗号化ボリュームまたは暗号化アーカイブ
- 有効期限を明記した命名
- ドキュメント化された削除
- 承認なしのコンシューマークラウドドライブへの同期禁止

例：

```bash
age -r age1yourpublickeyhere -o customer-export-2026-05-09.sql.gz.age customer-export.sql.gz
shred -u customer-export.sql.gz
```

これを儀式にするな。最良のバックアップポリシーは、開発者がそもそも本番エクスポートを必要とすることが少ないものだ。

## 6. ワークステーションのデフォルトを構築する

個人開発者向けの健全なベースライン：

| 領域 | ベースライン |
| --- | --- |
| ブラウザ | 本番パスワードの保存なし。重要なアカウントにはパスワードマネージャーとハードウェアバックド MFA を使う。 |
| プロジェクト | パッケージインストール、信頼できないコード、AI によるシェル作業があるプロジェクトに Dev Container を使う。 |
| シークレット | ディスク上に平文本番シークレットを置かない。実用的な範囲でローカル開発シークレットを暗号化する。 |
| クラウド | 短期認証情報。dev と prod の ID を分離。デフォルトで個人管理者トークンを使わない。 |
| GitHub | 細粒度トークン。パッケージ公開トークンを定期的に確認。org SSO とハードウェアキーを使う。 |
| AI ツール | プロジェクトスコープのアクセス、機密パスの拒否、実用的な範囲でコンテナ内でコマンド実行。 |
| バックアップ | 暗号化、有効期限、分離、監視。広いマウントと AI コンテキストから遠ざける。 |
| ネットワーク | 送信ファイアウォールをまずアラートまたはモニターモードで、その後リスクの高いツールにルールを設定。 |
| 検出 | バックアップ、認証情報、CI、クラウド、ドキュメントの場所に Canary トークン。 |

チームには以下を追加：

- 標準 `.devcontainer` テンプレート
- ローカル、共有 dev、staging、prod を区別するシークレットポリシー
- Canary トークンの配置規約
- ドキュメント化された送信ファイアウォールプロファイル
- 高速な認証情報ローテーションプレイブック
- 脅威モデルを劇なしに説明するオンボーディング

目標はすべての開発者をセキュリティエンジニアにすることではない。

目標は安全な道を普通の道にすることだ。

## 今週やるべきこと

大きすぎると感じるなら、5つだけやれ：

1. リスクの高いリポジトリを一つ選び、狭いマウントで Dev Container を追加する。
2. 平文 `.env.local` シークレットを一つ暗号化ローカルストレージまたはパスワードマネージャーに移動する。
3. 偽バックアップファイルに Canary トークンを一つ仕掛け、アラートを見える場所にルーティングする。
4. LuLu、Little Snitch、Portmaster、または同等のものをモニターモードでインストールし、何が実際に通信しているか観察する。
5. ローカル本番エクスポートを見つけ、削除、暗号化、または有効期限を設定する。

始めるにはそれで十分だ。

セキュリティ対策は大聖堂として到達しようとすることが多いので失敗する。まずドアを持ってこい。次に錠。次に警報。次に習慣。

ワークステーションは完全に信頼される必要はない。

偶然に無限に信頼されるのをやめればいい。

## 画像プラン

潜在的なカバーの方向性：

- 図解マップ：中央にラップコンピュータがあり、分離、シークレット、検出、送信の4つの制約されたリングがラベルされている。実践的なガイドに最適。
- 編集的メタファー：鍵、書類、ネットワークケーブルがガラスドームの下にある作業台で、一つのケーブルが警告灯に繋がっている。シリーズのビジュアルアイデンティティに最適。
- 失敗モードシーン：ローカルバックアップフォルダが本番インフラのように輝き、その周りに小さなアラートの罠が取り囲まれている。投稿がバックアップリスクに寄せるなら最適。

方向性が決まった後の推奨アセットセット：

- `desktop-social.webp` at 1200x630
- `wide.webp` at 1600x900
- `square.webp` at 800x800
