開発者ラップトップ向けのセキュリティアドバイスは、たいてい次のどちらかの形で破綻する。

一つはエンタープライズ向けのお飾り壁紙：

> エンドポイント保護を使え、定期的にパッチを当て、不審なリンクを避け、インシデントは即座に報告しろ。

すべて正しい。だが十分ではない。

もう一つはサバイバリストのナンセンス。ブラウザ、JavaScript、Wi-Fi、パッケージマネージャー、ベンダー、PDF、チャット、コードエディタ、電話、そして喜びまで全部使うなと言うやつ。

これも役に立たない。

現実的な目標はもっと小さい：

> 自分として実行されるものは、自分が信頼されていることすべてを自動的に継承すべきではない。

これがワークステントンの爆風半径の問題だ。

開発を濡れたセメントを通してタイピングしているような気分にさせずに、これを減らすためのガイドがこれだ。

最終確認日：2026年5月9日。ツールの動作、価格、プラットフォームサポートは変更されるため、チームで標準化する前に最新のドキュメントを確認すること。

## 防御の形

四つの層が必要だ：

| 層 | 役割 |
| --- | --- |
| 分離 | プロジェクトツールや危険なコマンドをマシンの他の部分から遠ざける。 |
| シークレット管理 | プレーンテキストの認証情報を減らし、機密値がうっかり漏洩しにくくする。 |
| 検知 | 攻撃者や不正な自動化が自然に見に行く場所にトリップワイヤーを仕掛ける。 |
| 出口制御 | 予期しないアウトバウンド接続を検知し、ブロックする。 |

ラップトップの脅威すべてを一度に解決しようとするな。

攻撃者が実際に楽しむパスから始めろ：何かを実行し、シークレットを読み取り、外に送り、誰かが気づく前に使う。

## 1. プロジェクトを Dev Containers に入れろ

[Dev Containers](https://github.com/devcontainers/spec) を使えば、コンテナをフル機能の開発環境として利用できる。開発者体験インフラのように聞こえるし、実際そうだ。だが、規律を持って使えばセキュリティ境界にもなる。

怠惰な設定はマウントしすぎる：

```jsonc
// 便利すぎる。爆風半径が大きすぎる。
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "mounts": [
    "source=${localEnv:HOME},target=/host-home,type=bind"
  ]
}
```

これではコンテナがホストアカウントの奇妙な形のバージョンになってしまう。

代わりに狭いマウントを使え：

```jsonc
// .devcontainer/devcontainer.json
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "workspaceFolder": "/workspaces/app",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ],
  "containerEnv": {
    "NODE_ENV": "development"
  },
  "postCreateCommand": "bun install"
}
```

これは完璧なサンドボックスではない。コンテナはカーネルを共有する。Docker には鋭いエッジがある。マウントはモデルを貫通する穴を開けられる。

だが、ほとんどの開発者ワークフローにとって、効果は即座に現れる：プロジェクトコマンドはプロジェクト全体ではなく、プロジェクトを見る。デジタル屋根裏部屋全体ではない。

### 何をマウントすべきか

リポジトリをマウントしろ。

プロジェクト固有のキャッシュをマウントしてもいいかもしれない。

これらをデフォルトでマウントするな：

- `~/.ssh`
- `~/.aws`
- `~/.config/gcloud`
- `~/.azure`
- `~/Downloads`
- `~/Documents`
- パスワードマネージャーのエクスポート
- データベースダンプ
- バックアップフォルダー
- 2021年から存在するランダムな「temp」フォルダー

プロジェクトがクラウドアクセスを必要とするなら、そのプロジェクト専用の認証情報を注入しろ。短期間の方がよい。読み取り専用の方がよい。開発アカウントにしかアクセスできないトークンの方が、個人の管理者アイデンティティが小さなスーツケースを持ってコンテナに入り込むよりましだ。

### AI コーディングツールもここに含まれる

AI コーディングツールは Dev Containers をより重要にする。重要でなくするのではない。

Anthropic の [Claude Code パーミッションドキュメント](https://code.claude.com/docs/en/permissions) は世界をパーミッションとサンドボックスに分割している：パーミッションはツール、ファイル、ドメインを制御する。サンドボックスは Bash ファイルシステムとネットワークアクセスに対する OS レベルの強制を提供する。

この区別がすべてだ。

エージェントがシェルコマンドの実行、パッケージのインストール、ファイルの検査、指示のフォローができるなら、シェル作業を制約されたプロジェクト環境内に入れろ。ホストは退屈のままにしておけ。

良いデフォルト：

- エージェントをホームディレクトリではなくリポジトリ内で開始する
- 機密パスを明示的に拒否する
- インストール/ビルド/テストコマンドには Dev Container を使用する
- 広範な「追加ディレクトリ」をコンテキストとして追加しない
- 認証情報、認証設定、パッケージ公開、クラウドリソースに触れる生成コマンドはすべてレビューする

モデルが TypeScript のエラーを修正するのに `~/Documents` フォルダーは必要ない。

## 2. プレーンテキストの `.env` 蔓延を置き換えろ

`.env` ファイルは悪ではない。

ただのファイルだ。それが問題なのだ。

ファイルはコピーされる。ファイルはインデックスされる。ファイルはマウントされる。ファイルは CSS をリントするはずだったスクリプトに読み取られる。ファイルはデバッグ用 zip に含まれる。ファイルは誰かが助けを求めて最後の12行を忘れたせいでチャットに貼り付けられる。

退屈な階層を使え：

1. シークレットが不要：値を `.env.example` に入れろ。
2. ローカル専用のシークレット：保管時に暗号化しろ。
3. 共有開発シークレット：本物のシークレットマネージャーまたはパスワードマネージャーに入れろ。
4. 本番シークレット：非常に具体的な理由がない限り、開発者ラップトップに置くな。

[VarLock](https://varlock.dev/guides/secrets/) は機密性を明示的にするため魅力的だ。ドキュメントでは、値に `@sensitive` をマークし、ローカル値を `varlock()` で暗号化し、コンソール出力から機密値を編集し、既知の機密値のプレーンテキスト出現をプロジェクトファイルからスキャンすると説明されている。

この形は「リポジトリに対して正規表現を実行し、シークレットがシークレットらしく見えることを祈る」よりましだ。

例の方向性：

```dotenv
# .env.schema
# @defaultSensitive=false

PUBLIC_APP_NAME=

# @sensitive
STRIPE_SECRET_KEY=

# @sensitive
DATABASE_URL=
```

ローカルオーバーライド：

```dotenv
# .env.local
PUBLIC_APP_NAME=demo
STRIPE_SECRET_KEY=varlock(local:...)
DATABASE_URL=varlock(local:...)
```

これはシークレットが侵害されたプロセスにロードされた時点で安全になるという意味ではない。何もない。だが、ファイルシステム上のプレーンテキストの賞品が減るという意味だ。

これはインフォスティーラー、悪意のある依存関係、広すぎる AI コンテキスト、誤ったコミット、そして謙虚な `console.log(process.env)` の瞬間に対して重要になる。

## 3. 泥棒が見に行く場所にカナリアトークンを仕掛けろ

ほとんどの監視は、既知の悪いことがいつ起きたかを教えてくれる。

カナリアトークンは、存在を知るべきでない何かが何かに触れたときに教えてくれる。

[Thinkst Canarytokens](https://help.canary.tools/hc/en-gb/articles/10905485310109-Canarytoken-Overview-and-Use-Cases) はデジタルトリップワイヤーだと説明している。ドキュメント、URL、API キー、VPN プロファイル、QR コード、その他の偽資産にでき、アクセスされるとアラートを出す。

配置が芸術だ。

ランダムな餌をばらまいて勝利宣言するな。認証情報窃盗、バックアップ窃盗、または偵察が自然に向かう場所にカナリアを置け。

### ローカルカナリア

偽のバックアップを作成しろ：

```text
~/backups/customer-prod-export-2024.sql
```

中にカナリア URL またはトークンを置け：

```sql
-- レガシーアナリティクス webhook
-- https://canarytokens.example.invalid/static/abc123
```

偽の認証情報ファイルを作成しろ：

```text
~/Documents/passwords-old.csv
```

または偽の AWS プロファイル：

```ini
# ~/.aws/credentials
[billing-prod-legacy]
aws_access_key_id = AKIA...
aws_secret_access_key = ...
```

利用可能な場合は本物の AWS カナリアトークンタイプを使用し、ファイルオープンだけでなく使用試行でアラートが発火するようにする。

### リポジトリカナリア

攻撃者がソースアクセス後に検査する場所の近くにカナリアを置け：

- 内部ランブック
- 非推奨のデプロイメントドキュメント
- 古いマイグレーションノート
- 明らかに非本番環境の `.env.canary` 内の偽サービス認証情報
- 偽のバックアップ復元手順

これは隠蔽によるセキュリティではない。廊下のアラームだ。

### CI とクラウドカナリア

良いクラウドトリップワイヤーの場所：

- 偽の CI シークレット
- 偽のデプロイトークン
- 権限のない偽のデータベースユーザー
- 未使用のオブジェクトストレージパス
- 偽の kubeconfig
- ランブックにドキュメント化された偽の API キー

アラートをアクション可能にしろ。無人のメールボックスにメールを送るカナリアは装飾的な文字列だ。

最低限、アラートは次のことを伝えるべきだ：

- どのトークンが発火したか
- どこに仕掛けられたか
- どのシステムが触れたか
- 何をローテーションすべきか
- 誰がレスポンスを担当するか

## 4. アウトバウンドトラフィックに門を設けろ

ローカルで悪意のある何かが実行された場合、窃取にはネットワークパスが必要だ。

ほとんどの開発者ラップトップはデフォルトでアウトバウンドトラフィックを許可している。便利だ。しかし、未知のプロセスがローカルの判断ポイントなしに未知の場所にデータを送信できることも意味する。

アウトバウンドファイアウォールはシートベルトの層だ。

すべてのクラッシュを防ぐわけではない。一部のクラッシュを survivable にする。そして、何が正常かを教えるまで、都合の悪い時に文句を言う。

### macOS

[LuLu](https://objective-see.org/products/lulu.html) は無料でオープンソースだ。Objective-See は不明な送信接続をブロックすると説明しており、ドキュメントでは LuLu は送信トラフィックのみを監視すると記載されている。

シンプルなアウトバウンドプロンプトが必要で、セットアップの摩擦を許容できるなら、良い最初の選択肢だ。

[Little Snitch](https://obdev.at/products/littlesnitch/) は商用でより洗練されている。接続アラートを表示し、アプリ接続の許可または拒否を可能にし、アプリ、ドメイン、国、ポート、プロトコル、トラフィックの可視性を持つネットワークモニターを提供する。

プロファイル、ルール管理、2週目以降も実際に使い続けられる UI を求めるなら、より強力な選択肢だ。

### Windows

Windows Defender Firewall はインバウンドおよびアウトバウンドトラフィックのアウトバウンドルールとルールの優先順位をサポートしている。Microsoft のガイダンスは冷静だ：アウトバウンドルールをブロックに変更することは高セキュリティ環境で検討できるが、アプリのインベントリとネットワーク接続が必要なもののルールの作成が必要だ。

翻訳：可能で、強力で、邪魔にしやすくする。

[Portmaster](https://safing.io/) も Windows で評価する価値がある。Safing はネットワーク接続を監視し、アプリごとのブロックルールを設定するオープンソースのアプリケーションファイアウォールだと説明している。

### Linux

Portmaster は一般的な Linux パッケージをサポートしている。OpenSnitch も評価に値する Linux アプリケーションファイアウォールだが、標準化する前にプロジェクトの状態とディストリビューションのパッケージングを確認すべきだ。

サーバーには通常のサーバーコントロールを使え。開発者ラップトップの場合、重要な機能はアプリケーションレベルの可視性だ。「443 以外のすべてのアウトバウンドをブロック」では十分ではない。興味のあるすべての窃取パスが 443 を話すのだから。

## 5. バックアップに大人の監督を与えろ

バックアップはコールドではない。ポータブル形式の機密データだ。

開発者マシンがバックアップアーカイブになってはならない。それが仕事でない限り。

実際に強制すべきルール：

- 本番エクスポートには所有者と有効期限が必要。
- ローカルデータベースダンプは暗号化しなければならない。
- 認証情報を含むエクスポートは認証情報のローテーションまたはスクラビングをトリガーする。
- バックアップフォルダーはデフォルトで Dev Containers にマウントされない。
- バックアップフォルダーはデフォルトで AI コーディングツールに拒否される。
- 少なくとも1つのカナリアがバックアップのようなストレージに存在する。
- 古いエクスポートはバイブスではなく自動化によって削除される。

シンプルなローカル規約：

```bash
mkdir -p ~/sensitive-exports
chmod 700 ~/sensitive-exports
```

より良い規約：

- 暗号化ボリュームまたは暗号化アーカイブ
- 有効期限付きの明確な命名
- ドキュメント化された削除
- 承認されていない限り、コンシューマークラウドドライブへの同期はなし

例：

```bash
age -r age1yourpublickeyhere -o customer-export-2026-05-09.sql.gz.age customer-export.sql.gz
shred -u customer-export.sql.gz
```

これを儀式にするな。最良のバックアップポリシーは、開発者がそもそも本番エクスポートをほとんど必要としないものだ。

## 6. ワークステントンのデフォルトを構築しろ

個人開発者向けのまともなベースラインはこれだ：

| 領域 | ベースライン |
| --- | --- |
| ブラウザ | 本番パスワードを保存しない。パスワードマネージャーとハードウェアベースの MFA を重要なアカウントに使用する。 |
| プロジェクト | パッケージインストール、信頼できないコード、AI 駆動のシェル作業を含むプロジェクトには Dev Containers を使用する。 |
| シークレット | ディスク上にプレーンテキストの本番シークレットを置かない。ローカル開発シークレットは実用的な範囲で暗号化する。 |
| クラウド | 短期間の認証情報。開発と本番のアイデンティティを分離する。デフォルトで個人の管理者トークンを使用しない。 |
| GitHub | きめ細かいトークン。パッケージ公開トークンをレビューする。組織 SSO とハードウェアキーを使用する。 |
| AI ツール | プロジェクトスコープのアクセス、機密パスを拒否、実用的な場合はコンテナ内でコマンドを実行。 |
| バックアップ | 暗号化、有効期限、分離、監視。広範なマウントと AI コンテキストから遠ざける。 |
| ネットワーク | アラートまたはモニターモードのアウトバウンドファイアウォールから始め、その後リスクのあるツールのルールを設定する。 |
| 検知 | バックアップ、認証情報、CI、クラウド、ドキュメントの場所にカナリアトークン。 |

チームの場合は次を追加しろ：

- 標準の `.devcontainer` テンプレート
- ローカル、共有開発、ステージング、本番を区別するシークレットポリシー
- カナリアトークン配置規約
- ドキュメント化されたアウトバウンドファイアウォールプロファイル
- 高速認証情報ローテーションのプレイブック
- 劇場なしで脅威モデルを説明するオンボーディング

目標はすべての開発者をセキュリティエンジニアにすることではない。

目標は安全なパスを通常のパスにすることだ。

## 今週やること

これが大きすぎるなら、5つやれ：

1. リスクの高いリポジトリを1つ選び、狭いマウントの Dev Container を追加する。
2. プレーンテキストの `.env.local` シークレットを1つ、暗号化されたローカルストレージまたはパスワードマネージャーに移動する。
3. 偽のバックアップファイルにカナリアトークンを1つ仕込み、アラートを見える場所にルーティングする。
4. LuLu、Little Snitch、Portmaster、または同等品をモニターモードでインストールし、実際に何が通信するかを観察する。
5. ローカルの本番エクスポートを見つけ、削除、暗号化、または有効期限を設定する。

始めるにはこれで十分だ。

セキュリティ作業はしばしば大聖堂として到着しようとして失敗する。まずはドアを持っていけ。然后是錠前。然后是警報。然后是習慣。

ワークステントンは完全に信頼される必要はない。

偶然に無限に信頼されるのをやめる必要がある。

## 画像計画

潜在的なカバーの方向性：

- 図解マップ：中央にラップトップ、分離、シークレット、検知、出口の4つの制約リング。実用的なガイドに最適。
- 編集メタファー：ガラスドームの下の鍵、ドキュメント、ネットワークケーブルのある作業台、警告灯につながる1本のケーブル。シリーズのビジュアルアイデンティティに最適。
- 障害モードシーン：本番インフラのように光るローカルバックアップフォルダーを小さなアラートトリップワイヤーが囲む。バックアップリスクにより重点を置く場合に最適。

方向性が決まったら推奨されるアセットセット：

- `desktop-social.webp` 1200x630
- `wide.webp` 1600x900
- `square.webp` 800x800
