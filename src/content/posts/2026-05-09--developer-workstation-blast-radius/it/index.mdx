I consigli di sicurezza per i laptop da sviluppo falliscono quasi sempre in uno di due modi.

O sono wallpaper aziendale:

> Usa protezione endpoint, applica le patch con regolarità, evita i link sospetti, segnala subito gli incidenti.

Tutto vero. Anche tutto insufficiente.

Oppure sono nonsense da survivalista, dove la soluzione è smettere di usare browser, JavaScript, Wi-Fi, package manager, vendor, PDF, chat, editor di codice, telefoni e gioia.

Anche questo non serve.

L'obiettivo pratico è più piccolo:

> Se qualcosa gira come te, non dovrebbe ereditare automaticamente tutto ciò che ti è stato affidato.

Questo è il problema del raggio di esplosione della workstation.

Questa guida spiega come ridurlo senza trasformare lo sviluppo in digitare attraverso cemento bagnato.

Ultima verifica: 9 maggio 2026. Il comportamento degli strumenti, i prezzi e il supporto delle piattaforme cambiano, quindi controlla la documentazione corrente prima di standardizzare il tutto in un team.

## La Forma della Difesa

Ti servono quattro livelli:

| Livello | Compito |
| --- | --- |
| Isolamento | Tenere gli strumenti del progetto e i comandi rischiosi lontani dal resto della macchina. |
| Gestione dei segreti | Ridurre le credenziali in chiaro e rendere più difficile la fuga accidentale di valori sensibili. |
| Rilevazione | Piazzare trappole dove attaccanti o automazione difettosa guarderebbero spontaneamente. |
| Controllo dell'uscita | Notare e bloccare connessioni in uscita inattese. |

Non iniziare cercando di risolvere tutte le minacce del laptop.

Inizia dal percorso che gli attaccanti apprezzano davvero: eseguire qualcosa, leggere i segreti, mandarli fuori, usarli prima che qualcuno se ne accorga.

## 1. Metti i Progetti nei Dev Container

[Dev Containers](https://github.com/devcontainers/spec) ti permettono di usare un container come ambiente di sviluppo completo. Sembra infrastruttura per la developer experience, e lo è. Ma è anche un confine di sicurezza quando lo usi con disciplina.

La configurazione pigra monta troppo:

```jsonc
// Troppo comodo. Troppo raggio di esplosione.
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "mounts": [
    "source=${localEnv:HOME},target=/host-home,type=bind"
  ]
}
```

Questo trasforma il container in una versione dalla forma strana del tuo account host.

Usa invece mount mirati:

```jsonc
// .devcontainer/devcontainer.json
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "workspaceFolder": "/workspaces/app",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ],
  "containerEnv": {
    "NODE_ENV": "development"
  },
  "postCreateCommand": "bun install"
}
```

Questo non è un sandbox perfetto. I container condividono un kernel. Docker ha bordi taglienti. I mount possono bucare il modello da parte a parte.

Ma per la maggior parte dei flussi di lavoro da sviluppatore, il vantaggio è immediato: i comandi del progetto vedono il progetto, non tutto il tuo solaio digitale.

### Cosa Montare

Monta il repo.

Forse monta una cache specifica del progetto.

Non montare di default questi elementi:

- `~/.ssh`
- `~/.aws`
- `~/.config/gcloud`
- `~/.azure`
- `~/Downloads`
- `~/Documents`
- export di password manager
- dump di database
- cartelle di backup
- cartelle “temp” casuali che esistono dal 2021

Se un progetto ha bisogno di accesso al cloud, inietta una credenziale creata per quel progetto. Meglio se a breve durata. Meglio se sola lettura. Un token che può accedere solo a un account dev è meglio della tua identità admin personale che vaga nel container con una valigetta minuscola.

### Anche gli Strumenti AI per il Codice Devono Stare Qui

Gli strumenti AI per il codice rendono i Dev Container più importanti, non meno.

La [documentazione dei permessi di Claude Code](https://code.claude.com/docs/en/permissions) di Anthropic divide il mondo in permessi e sandboxing: i permessi controllano strumenti, file e domini; il sandboxing fornisce enforcement a livello OS per l'accesso Bash a filesystem e rete.

Questa distinzione è tutta la partita.

Se un agente può eseguire comandi shell, installare pacchetti, ispezionare file e seguire istruzioni, fai girare il lavoro shell dentro un ambiente di progetto limitato. Mantieni noioso l'host.

Buon default:

- avvia l'agente nella repo, non nella tua home directory
- nega esplicitamente i percorsi sensibili
- usa un Dev Container per i comandi di installazione, build e test
- evita di aggiungere ampie “directory extra” come contesto
- rivedi qualsiasi comando generato che tocchi credenziali, configurazione auth, pubblicazione di pacchetti o risorse cloud

Il modello non ha bisogno della tua cartella `~/Documents` per correggere un errore TypeScript.

## 2. Sostituisci la Dispersione dei `.env` in Chiaro

I file `.env` non sono malvagi.

Sono solo file. Ed è questo il problema.

I file vengono copiati. I file vengono indicizzati. I file vengono montati. I file vengono letti da script che dovevano solo fare lint del CSS. I file finiscono in zip di debug. I file vengono incollati in chat perché qualcuno voleva aiuto e si è dimenticato le ultime dodici righe.

Usa la gerarchia noiosa:

1. Nessun segreto necessario: metti il valore in `.env.example`.
2. Segreto solo locale: cifralo a riposo.
3. Segreto condiviso per sviluppo: mettilo in un vero secrets manager o password manager.
4. Segreto di produzione: non metterlo sui laptop degli sviluppatori, a meno che non ci sia un motivo molto specifico.

[VarLock](https://varlock.dev/guides/secrets/) è interessante perché rende esplicita la sensibilità. La sua documentazione descrive il marcaggio dei valori con `@sensitive`, la cifratura dei valori locali con `varlock()`, la redazione dei valori sensibili dall'output della console e la scansione dei file di progetto per trovare occorrenze in chiaro di valori già noti come sensibili.

La forma è migliore di “lancia una regex sul repo e spera che il segreto sembri segreto”.

Direzione d'esempio:

```dotenv
# .env.schema
# @defaultSensitive=false

PUBLIC_APP_NAME=

# @sensitive
STRIPE_SECRET_KEY=

# @sensitive
DATABASE_URL=
```

Override locale:

```dotenv
# .env.local
PUBLIC_APP_NAME=demo
STRIPE_SECRET_KEY=varlock(local:...)
DATABASE_URL=varlock(local:...)
```

Questo non significa che i segreti siano al sicuro una volta caricati in un processo compromesso. Nulla lo è. Ma significa che il filesystem ha meno premi in chiaro.

Questo conta contro infostealer, dipendenze malevole, contesto AI troppo ampio, commit accidentali e il semplice momento `console.log(process.env)`.

## 3. Metti Canary Token Dove Li Cercherebbe un Ladro

La maggior parte del monitoraggio ti dice quando è successo qualcosa di noto e cattivo.

I canary token ti dicono quando qualcosa di strano ha toccato qualcosa che non dovrebbe nemmeno sapere esistere.

[Thinkst Canarytokens](https://help.canary.tools/hc/en-gb/articles/10905485310109-Canarytoken-Overview-and-Use-Cases) li descrive come tripwire digitali. Possono essere documenti, URL, chiavi API, profili VPN, codici QR e altri asset finti che avvisano quando vengono accessi.

La collocazione è l'arte.

Non spargere esche casuali e dichiarare vittoria. Metti i canary dove il furto di credenziali, il furto di backup o la ricognizione andrebbero naturalmente a guardare.

### Canary Locali

Crea un backup finto:

```text
~/backups/customer-prod-export-2024.sql
```

Metti dentro un canary URL o token:

```sql
-- legacy analytics webhook
-- https://canarytokens.example.invalid/static/abc123
```

Crea un file credenziali finto:

```text
~/Documents/passwords-old.csv
```

Oppure un profilo AWS finto:

```ini
# ~/.aws/credentials
[billing-prod-legacy]
aws_access_key_id = AKIA...
aws_secret_access_key = ...
```

Usa un vero tipo di canary token AWS quando disponibile, così l'allarme scatta al tentativo di uso, non solo all'apertura del file.

### Canary nel Repo

Metti i canary vicino ai posti che gli attaccanti ispezionano dopo aver ottenuto accesso al codice sorgente:

- runbook interni
- vecchi documenti di deploy
- vecchie note di migrazione
- credenziali finte di servizi in un `.env.canary` chiaramente non di produzione
- istruzioni finte per il restore di backup

Questo non è security by obscurity. È un allarme in un corridoio.

### Canary in CI e nel Cloud

Buoni posti per i tripwire cloud:

- un segreto CI finto
- un token di deploy finto
- un utente database finto senza privilegi
- un percorso di object storage inutilizzato
- un kubeconfig finto
- una chiave API finta documentata in un runbook

Rendi l'allarme azionabile. Un canary che manda mail a una casella non sorvegliata è una stringa decorativa.

Al minimo, l'allarme dovrebbe dirti:

- quale token è scattato
- dove era stato piazzato
- quale sistema lo ha toccato
- cosa ruotare
- chi possiede la risposta

## 4. Metti un Cancello al Traffico in Uscita

Se qualcosa di malevolo gira in locale, l'esfiltrazione ha bisogno di una strada di rete.

La maggior parte dei laptop da sviluppo permette il traffico in uscita di default. È comodo. Significa anche che un processo sconosciuto può spesso inviare dati a un posto sconosciuto senza un punto decisionale locale.

I firewall in uscita sono il livello cintura di sicurezza.

Non fermeranno ogni incidente. Renderanno alcuni incidenti sopravvivibili. E si lamenteranno anche in momenti scomodi finché non insegni loro che cosa è normale.

### macOS

[LuLu](https://objective-see.org/products/lulu.html) è gratuito e open source. Objective-See lo descrive come un blocco delle connessioni in uscita sconosciute, e la documentazione nota che LuLu monitora solo il traffico in uscita.

È una buona prima scelta se vuoi prompt semplici per l'uscita e puoi tollerare un po' di attrito iniziale.

[Little Snitch](https://obdev.at/products/littlesnitch/) è commerciale e più rifinito. Mostra alert di connessione, ti lascia consentire o negare le connessioni delle app e offre un network monitor con visibilità su app, dominio, paese, porta, protocollo e traffico.

È la scelta più forte se vuoi profili, gestione delle regole e una UI che la gente continui a usare anche dopo la seconda settimana.

### Windows

Windows Defender Firewall supporta regole in uscita e precedenza delle regole per traffico in ingresso e in uscita. La guida Microsoft è sobria: cambiare le regole in uscita su bloccato può essere preso in considerazione in ambienti ad alta sicurezza, ma richiede inventariare le app e creare regole per quelle che hanno bisogno di connettività di rete.

Traduzione: possibile, potente e facile da rendere fastidioso.

[Portmaster](https://safing.io/) merita comunque una valutazione anche su Windows. Safing lo descrive come un application firewall open source che monitora le connessioni di rete e imposta regole di blocco per singola applicazione.

### Linux

Portmaster supporta i pacchetti Linux più comuni. OpenSnitch è un altro application firewall Linux da valutare, anche se è bene verificare lo stato del progetto e il packaging delle distro prima di standardizzarlo.

Per i server, usa i normali controlli server. Per i laptop da sviluppo, la caratteristica chiave è la visibilità a livello applicazione. “Blocca tutto l'outbound tranne 443” non basta quando ogni percorso interessante di esfiltrazione parla anche 443.

## 5. Dai alle Backup una Supervisione da Adulti

I backup non sono freddi. Sono dati sensibili in forma portabile.

Le macchine degli sviluppatori non dovrebbero diventare archivi di backup, a meno che non sia quello il loro lavoro.

Regole che farei davvero rispettare:

- le esportazioni di produzione richiedono un owner e una data di scadenza
- i dump dei database locali devono essere cifrati
- qualsiasi export che contiene credenziali attiva rotazione o scrubbing delle credenziali
- le cartelle di backup non vengono montate nei Dev Container di default
- le cartelle di backup vengono negate agli strumenti AI per il codice di default
- almeno un canary vive in uno storage simile a un backup
- i vecchi export vengono cancellati dall'automazione, non dalle vibrazioni

Convenzione locale semplice:

```bash
mkdir -p ~/sensitive-exports
chmod 700 ~/sensitive-exports
```

Convenzione migliore:

- volume cifrato o archivio cifrato
- naming chiaro con scadenza
- cancellazione documentata
- niente sync verso drive cloud consumer, a meno che non sia approvato

Esempio:

```bash
age -r age1yourpublickeyhere -o customer-export-2026-05-09.sql.gz.age customer-export.sql.gz
shred -u customer-export.sql.gz
```

Non trasformarlo in un rituale. La migliore policy di backup è quella in cui gli sviluppatori hanno raramente bisogno di export di produzione.

## 6. Costruisci un Default da Workstation

Ecco una baseline sensata per uno sviluppatore individuale:

| Area | Baseline |
| --- | --- |
| Browser | Nessuna password di produzione salvata. Usa un password manager e MFA con hardware per gli account importanti. |
| Progetti | Usa Dev Container per progetti con installazioni di pacchetti, codice non fidato o lavoro shell guidato da AI. |
| Segreti | Nessun segreto di produzione in chiaro su disco. Cifra i segreti locali di sviluppo quando è pratico. |
| Cloud | Credenziali a breve durata. Identità dev e prod separate. Nessun token admin personale di default. |
| GitHub | Token con privilegi granulari. Rivedi i token di pubblicazione pacchetti. Usa SSO dell'org e chiavi hardware. |
| Strumenti AI | Accesso limitato al progetto, negazione dei percorsi sensibili, comandi eseguiti in container quando pratico. |
| Backup | Cifra, fai scadere, isola e monitora. Tienili fuori dai mount ampi e dal contesto AI. |
| Rete | Firewall in uscita in modalità alert o monitor prima, poi regole per gli strumenti rischiosi. |
| Rilevazione | Canary token nei punti di backup, credenziali, CI, cloud e documentazione. |

Per un team, aggiungi:

- un template standard `.devcontainer`
- una policy dei segreti che distingua locale, dev condiviso, staging e produzione
- convenzioni di posizionamento dei canary token
- profili documentati del firewall in uscita
- playbook rapidi di rotazione credenziali
- onboarding che spieghi il modello di minaccia senza teatro

L'obiettivo non è trasformare ogni sviluppatore in un ingegnere della sicurezza.

L'obiettivo è rendere normale il percorso più sicuro.

## Cosa Fare Questa Settimana

Se tutto questo sembra troppo, fai cinque cose:

1. Scegli una repo ad alto rischio e aggiungi un Dev Container con mount stretti.
2. Sposta un segreto `.env.local` in chiaro verso storage locale cifrato o un password manager.
3. Pianta un canary token in un file di backup finto e manda gli alert in un posto visibile.
4. Installa LuLu, Little Snitch, Portmaster o un equivalente in modalità monitor e guarda davvero cosa parla.
5. Trova gli export di produzione locali e cancellali, cifrali o falli scadere.

Questo basta per iniziare.

Il lavoro di sicurezza spesso fallisce perché cerca di presentarsi come una cattedrale. Porta prima una porta. Poi una serratura. Poi un allarme. Poi un'abitudine.

La workstation non deve essere perfettamente fidata.

Deve smettere di essere fidata all'infinito per errore.
