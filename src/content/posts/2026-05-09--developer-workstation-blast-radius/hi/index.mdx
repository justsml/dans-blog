डेवलपर लैपटॉप के लिए सुरक्षा सलाह आमतौर पर दो में से एक तरीके से फेल होती है।

या तो वह एंटरप्राइज़ वॉलपेपर होती है:

> endpoint protection का उपयोग करें, नियमित रूप से patch करें, संदिग्ध links से बचें, घटनाओं की promptly रिपोर्ट करें।

सब सही है। फिर भी काफी नहीं।

या फिर वह survivalist किस्म की बकवास होती है, जहां जवाब यह होता है कि browsers, JavaScript, Wi-Fi, package managers, vendors, PDFs, chat, code editors, phones, और joy सब बंद कर दो।

वह भी उपयोगी नहीं।

व्यावहारिक लक्ष्य छोटा है:

> अगर कुछ आपके रूप में चल रहा है, तो उसे अपने-आप वह सब inherit नहीं करना चाहिए जो करने पर आपको भरोसा है।

यही workstation blast-radius समस्या है।

यह गाइड उस radius को कम करने के बारे में है, बिना development को गीले cement में टाइप करने जैसा बनाए।

अंतिम सत्यापन: 9 मई, 2026. टूल व्यवहार, pricing, और platform support बदलते रहते हैं, इसलिए टीम-स्तर पर standardize करने से पहले current docs देख लें।

## रक्षा का ढांचा

आपको चार परतें चाहिए:

| परत | काम |
| --- | --- |
| Isolation | Project tools और risky commands को बाकी मशीन से दूर रखें। |
| Secret handling | Plaintext credentials कम करें और sensitive values को गलती से leak करना कठिन बनाएं। |
| Detection | ऐसे tripwires लगाएं जहां attackers या खराब automation स्वाभाविक रूप से देखेंगे। |
| Egress control | Unexpected outbound connections को नोटिस करें और रोकें। |

हर laptop threat को एक साथ हल करने की कोशिश से शुरुआत मत करें।

उस रास्ते से शुरू करें जिसे attackers पसंद करते हैं: कुछ चलाओ, secrets पढ़ो, बाहर भेजो, और किसी के notice करने से पहले उनका उपयोग कर लो।

## 1. Projects को Dev Containers में रखें

[Dev Containers](https://github.com/devcontainers/spec) आपको container को एक full-featured development environment के रूप में उपयोग करने देते हैं। यह developer-experience infrastructure जैसा लगता है, और है भी। लेकिन जब इसे discipline के साथ इस्तेमाल किया जाए, तो यह security boundary भी है।

ढीली configuration बहुत ज़्यादा माउंट करती है:

```jsonc
// Too convenient. Too much blast radius.
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "mounts": [
    "source=${localEnv:HOME},target=/host-home,type=bind"
  ]
}
```

यह container को आपके host account के अजीब-से आकार वाले संस्करण में बदल देता है।

इसके बजाय narrow mounts इस्तेमाल करें:

```jsonc
// .devcontainer/devcontainer.json
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "workspaceFolder": "/workspaces/app",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ],
  "containerEnv": {
    "NODE_ENV": "development"
  },
  "postCreateCommand": "bun install"
}
```

यह perfect sandbox नहीं है। Containers kernel साझा करते हैं। Docker के sharp edges हैं। Mounts model में सीधे छेद कर सकते हैं।

लेकिन ज़्यादातर developer workflows के लिए फायदा तुरंत मिलता है: project commands project देखते हैं, आपका पूरा digital attic नहीं।

### क्या माउंट करें

Repo माउंट करें।

शायद project-specific cache भी माउंट करें।

डिफ़ॉल्ट रूप से ये चीज़ें माउंट न करें:

- `~/.ssh`
- `~/.aws`
- `~/.config/gcloud`
- `~/.azure`
- `~/Downloads`
- `~/Documents`
- password manager exports
- database dumps
- backup folders
- random “temp” folders जो 2021 से मौजूद हैं

अगर किसी project को cloud access चाहिए, तो उस project के लिए बना credential inject करें। Short-lived बेहतर है। Read-only बेहतर है। ऐसा token जो केवल dev account access कर सके, आपके personal admin identity के container में छोटे suitcase के साथ घुसने से बेहतर है।

### AI Coding Tools भी यहीं आते हैं

AI coding tools Dev Containers को कम नहीं, और ज़्यादा महत्वपूर्ण बनाते हैं।

Anthropic की [Claude Code permission docs](https://code.claude.com/docs/en/permissions) दुनिया को permissions और sandboxing में बांटती हैं: permissions tools, files, और domains नियंत्रित करती हैं; sandboxing Bash filesystem और network access के लिए OS-level enforcement देती है।

यही असली फर्क है।

अगर कोई agent shell commands चला सकता है, packages install कर सकता है, files inspect कर सकता है, और instructions follow कर सकता है, तो shell work को constrained project environment के अंदर रखें। Host को boring रखें।

अच्छा default:

- agent को repo में शुरू करें, home directory में नहीं
- sensitive paths explicitly deny करें
- install/build/test commands के लिए Dev Container का उपयोग करें
- broad “extra directories” को context में जोड़ने से बचें
- credentials, auth config, package publishing, या cloud resources को छूने वाले किसी भी generated command की समीक्षा करें

TypeScript error ठीक करने के लिए model को आपके `~/Documents` folder की ज़रूरत नहीं है।

## 2. Plaintext `.env` Sprawl को बदलें

`.env` files बुरी नहीं हैं।

वे सिर्फ files हैं। वही समस्या है।

Files copy हो जाते हैं। Files index हो जाते हैं। Files mount हो जाते हैं। Files ऐसे scripts पढ़ लेते हैं जिन्हें सिर्फ CSS lint करना था। Files debugging zips में शामिल हो जाते हैं। Files chat में paste हो जाते हैं क्योंकि किसी को मदद चाहिए थी और आख़िरी बारह lines भूल गईं।

निम्न boring hierarchy अपनाएँ:

1. Secret की ज़रूरत नहीं: value को `.env.example` में रखें।
2. Local-only secret: उसे rest में encrypt करें।
3. Shared development secret: उसे real secrets manager या password manager में रखें।
4. Production secret: जब तक कोई बहुत खास वजह न हो, उसे developer laptops पर मत रखें।

[VarLock](https://varlock.dev/guides/secrets/) इसलिए आकर्षक है क्योंकि यह sensitivity को स्पष्ट बनाता है। इसके docs `@sensitive` से values mark करने, `varlock()` से local values encrypt करने, console output से sensitive values redact करने, और known sensitive values की plaintext occurrences के लिए project files scan करने का वर्णन करते हैं।

यह shape “repo पर regex चलाओ और hope करो कि secret secret-shaped दिखे” से बेहतर है।

उदाहरण दिशा:

```dotenv
# .env.schema
# @defaultSensitive=false

PUBLIC_APP_NAME=

# @sensitive
STRIPE_SECRET_KEY=

# @sensitive
DATABASE_URL=
```

Local override:

```dotenv
# .env.local
PUBLIC_APP_NAME=demo
STRIPE_SECRET_KEY=varlock(local:...)
DATABASE_URL=varlock(local:...)
```

इसका मतलब यह नहीं कि secrets compromised process में लोड हो जाने के बाद सुरक्षित हैं। कुछ भी नहीं है। लेकिन इसका मतलब यह है कि filesystem पर plaintext prizes कम हैं।

यह infostealers, malicious dependencies, over-broad AI context, accidental commits, और उस साधारण `console.log(process.env)` पल के खिलाफ मायने रखता है।

## 3. Canary Tokens उन जगहों पर रखें जहां चोर देखेगा

अधिकांश monitoring आपको तब बताती है जब कुछ known-bad हुआ होता है।

Canary tokens आपको तब बताते हैं जब किसी अजीब चीज़ ने ऐसी चीज़ छुई जिसे उसे पता ही नहीं होना चाहिए था।

[Thinkst Canarytokens](https://help.canary.tools/hc/en-gb/articles/10905485310109-Canarytoken-Overview-and-Use-Cases) इन्हें digital tripwires के रूप में वर्णित करता है। ये documents, URLs, API keys, VPN profiles, QR codes, और अन्य fake assets हो सकते हैं जो access होने पर alert करते हैं।

Placement ही कला है।

Random bait न फैलाएँ और victory घोषित न करें। Canary उन्हीं जगहों पर रखें जहां credential theft, backup theft, या reconnaissance स्वाभाविक रूप से जाएगा।

### स्थानीय Canaries

एक fake backup बनाएं:

```text
~/backups/customer-prod-export-2024.sql
```

उसमें canary URL या token डालें:

```sql
-- legacy analytics webhook
-- https://canarytokens.example.invalid/static/abc123
```

एक fake credentials file बनाएं:

```text
~/Documents/passwords-old.csv
```

या fake AWS profile:

```ini
# ~/.aws/credentials
[billing-prod-legacy]
aws_access_key_id = AKIA...
aws_secret_access_key = ...
```

जब उपलब्ध हो, real AWS canary token type इस्तेमाल करें ताकि alert attempted use पर बजे, सिर्फ file open पर नहीं।

### Repo Canaries

Source access के बाद attackers जिन जगहों की जांच करते हैं, उनके पास canaries रखें:

- internal runbooks
- deprecated deployment docs
- पुराने migration notes
- fake service credentials एक साफ़ तौर पर non-production `.env.canary` में
- fake backup restore instructions

यह security by obscurity नहीं है। यह hallway में लगा alarm है।

### CI और Cloud Canaries

अच्छी cloud tripwire locations:

- fake CI secret
- fake deploy token
- fake database user जिसके पास कोई privileges न हों
- unused object storage path
- fake kubeconfig
- runbook में documented fake API key

Alert को actionable बनाएं। ऐसा canary जो unattended inbox पर email भेजता है, decorative string है।

कम से कम alert आपको यह बताए:

- कौन-सा token fire हुआ
- वह कहाँ रखा गया था
- किस system ने उसे touch किया
- क्या rotate करना है
- response का owner कौन है

## 4. Outbound Traffic पर Gate लगाएँ

अगर कोई malicious चीज़ locally चलती है, तो exfiltration के लिए network path चाहिए।

ज़्यादातर developer laptops outbound traffic को डिफ़ॉल्ट रूप से allow करते हैं। यह सुविधाजनक है। इसका मतलब यह भी है कि कोई unknown process अक्सर बिना local decision point के unknown जगह डेटा भेज सकता है।

Outbound firewalls seatbelt layer हैं।

वे हर crash नहीं रोकेंगे। वे कुछ crashes को survivable बनाएंगे। वे तब तक inconvenient times पर शिकायत भी करेंगे जब तक आप उन्हें normal व्यवहार सिखा नहीं देते।

### macOS

[LuLu](https://objective-see.org/products/lulu.html) free और open source है। Objective-See इसे unknown outgoing connections block करने वाला बताता है, और इसकी docs note करती हैं कि LuLu सिर्फ outgoing traffic monitor करता है।

अगर आप simple outbound prompts चाहते हैं और setup friction सह सकते हैं, तो यह अच्छा first choice है।

[Little Snitch](https://obdev.at/products/littlesnitch/) commercial और ज़्यादा polished है। यह connection alerts दिखाता है, आपको app connections allow या deny करने देता है, और app, domain, country, port, protocol, और traffic visibility के साथ network monitor देता है।

अगर आप profiles, rule management, और ऐसी UI चाहते हैं जिसे लोग week two के बाद भी इस्तेमाल करते रहें, तो यह मजबूत विकल्प है।

### Windows

Windows Defender Firewall outbound rules और inbound तथा outbound traffic के लिए rule precedence support करता है। Microsoft की guidance सीधी है: outbound rules को blocked करना high-security environments में विचारणीय हो सकता है, लेकिन इसके लिए उन apps की inventory और rules बनाना ज़रूरी है जिन्हें network connectivity चाहिए।

अनुवाद: संभव है, शक्तिशाली है, और इसे झंझटपूर्ण बनाना आसान है।

[Portmaster](https://safing.io/) Windows पर भी देखने लायक है। Safing इसे एक open-source application firewall बताता है जो network connections monitor करता है और per-application blocking rules सेट करता है।

### Linux

Portmaster सामान्य Linux packages को support करता है। OpenSnitch एक और Linux application firewall है जिसे देखना चाहिए, हालांकि standardize करने से पहले project state और distro packaging जांचनी चाहिए।

Servers के लिए normal server controls इस्तेमाल करें। Developer laptops के लिए मुख्य feature application-level visibility है। “सिर्फ 443 छोड़कर सब outbound block कर दो” पर्याप्त नहीं है, क्योंकि दिलचस्प exfiltration paths में से हर एक भी 443 बोलता है।

## 5. Backups को वयस्क निगरानी दें

Backups ठंडे नहीं होते। वे portable form में sensitive data होते हैं।

Developer machines को backup archives नहीं बनना चाहिए, जब तक वही काम न हो।

जो नियम मैं वास्तव में लागू करूँगा:

- Production exports के लिए owner और expiration date होना चाहिए।
- Local database dumps encrypted होने चाहिए।
- किसी भी export में credentials हों तो credential rotation या scrubbing trigger होना चाहिए।
- Backup folders Dev Containers में डिफ़ॉल्ट रूप से mount नहीं होने चाहिए।
- Backup folders AI coding tools से डिफ़ॉल्ट रूप से deny होने चाहिए।
- Backup जैसी storage में कम से कम एक canary होना चाहिए।
- पुराने exports automation से delete हों, vibes से नहीं।

साधारण local convention:

```bash
mkdir -p ~/sensitive-exports
chmod 700 ~/sensitive-exports
```

बेहतर convention:

- encrypted volume या encrypted archive
- expiration के साथ clear naming
- documented deletion
- approved न हो तो consumer cloud drives में sync नहीं

उदाहरण:

```bash
age -r age1yourpublickeyhere -o customer-export-2026-05-09.sql.gz.age customer-export.sql.gz
shred -u customer-export.sql.gz
```

इसे ritual मत बनाइए। सबसे अच्छी backup policy वही है जिसमें developers को production exports की ज़रूरत बहुत कम पड़े।

## 6. एक Workstation Default बनाएं

एक individual developer के लिए एक समझदार baseline यह है:

| क्षेत्र | बेसलाइन |
| --- | --- |
| Browser | No saved production passwords. महत्वपूर्ण खातों के लिए password manager और hardware-backed MFA उपयोग करें। |
| Projects | package installs, untrusted code, या AI-driven shell work वाले projects के लिए Dev Containers का उपयोग करें। |
| Secrets | disk पर plaintext production secrets न रखें. जहाँ व्यावहारिक हो local dev secrets encrypt करें। |
| Cloud | Short-lived credentials. dev और prod identities अलग रखें. डिफ़ॉल्ट रूप से personal admin token न रखें। |
| GitHub | Fine-grained tokens. package publishing tokens की समीक्षा करें. org SSO और hardware keys उपयोग करें। |
| AI tools | Project-scoped access, sensitive paths deny करें, और जब संभव हो commands को containers में चलाएँ। |
| Backups | Encrypt करें, expire करें, isolate करें, और monitor करें. Broad mounts और AI context से बाहर रखें। |
| Network | Outbound firewall पहले alert या monitor mode में, फिर risky tools के लिए rules। |
| Detection | Backup, credential, CI, cloud, और documentation locations में canary tokens। |

टीम के लिए, यह जोड़ें:

- standard `.devcontainer` template
- एक secrets policy जो local, shared dev, staging, और prod में फर्क करे
- canary-token placement conventions
- documented outbound firewall profiles
- तेज़ credential rotation playbooks
- onboarding जो threat model समझाए, theater नहीं

लक्ष्य हर developer को security engineer बनाना नहीं है।

लक्ष्य safer path को normal path बनाना है।

## इस हफ्ते क्या करें

अगर यह बहुत बड़ा लग रहा है, तो ये पाँच चीज़ें करें:

1. एक high-risk repo चुनें और narrow mounts के साथ Dev Container जोड़ें।
2. एक plaintext `.env.local` secret को encrypted local storage या password manager में move करें।
3. एक fake backup file में एक canary token लगाएँ और alerts किसी visible जगह भेजें।
4. LuLu, Little Snitch, Portmaster, या equivalent को monitor mode में install करें और देखें कि वास्तव में कौन बात कर रहा है।
5. local production exports ढूँढें और उन्हें delete, encrypt, या expire करें।

शुरुआत के लिए इतना काफी है।

Security work अक्सर इसलिए fail करता है क्योंकि वह cathedral बनकर आने की कोशिश करता है। पहले एक door लाओ। फिर lock। फिर alarm। फिर habit।

वर्कस्टेशन को perfectly trusted होने की ज़रूरत नहीं है।

उसे बस accident से infinitely trusted होना बंद करना है।
