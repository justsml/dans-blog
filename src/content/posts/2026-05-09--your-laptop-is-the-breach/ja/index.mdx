PDFを開くことがプロダクションインシデントになってはならない。

SMSのリンクをクリックすることがバックアップの漏洩になってはならない。

検索結果から「便利なツール」をインストールすることが、クラウドコンソール、ソースコード、CIトークン、データベースエクスポート、そして`~/Downloads`に置いたまま忘れていたプロダクションのコピーを丸ごと渡してしまうことになってはならない。

それでも現実がこうなっている。現代の 개발자 노트북은 더 이상 노트북이 아니기 때문이다. 그것은 키보드가 딸린 인증 정보 창고다.

ブラウザセッション、SSH鍵、`.env`ファイル、GitHubトークン、パッケージマネージャーの認証、クラウドCLI、パスワードマネージャーのブラウザ拡張、AIコーディングツールのシェルアクセス、ローカルデータベース、旧バックアップ、一回限りのエクスポート、ベンダーのPDFランダムダウンロード、時には暗号通貨ウォレットまで——宇宙がコメディを選んだなら。

旧来のメンタルモデルはこうだった：プロダクションは危険、ローカルは便利。

そのモデルは終わった。

<p class="inset">
問題はブログに表示すべての悪いクリックを避けることができるかではない。問題は、1回の悪いクリックがすべてを読み込み、すべてを使用し、そしてあなたに気づかれる前に退出できるかどうかだ。
</p>

最終確認日：2026年5月9日。以下の脅威例とツールの動作は急速に変化するため、製品詳細は現在のメモとして扱い、聖典としては扱わないこと。

## 脅威レベルを設定する

ほとんどの人はドラマチックな攻撃を想像するため、脅威レベルを低く設定しすぎる。

PDFパーサーのゼロデイを想像する。国家ぐるみ、カレンダー inviteと単眼鏡ついた者。ordinary engineering discipline が無関係に見えるほど exotic な何かを想像する。

退屈なバージョンの方が役立つ。

開発者が十分にに見えたメッセージを受け取る：

- 契約元からのPDF請求書
- 配送またはアカウント警告についてのSMS
- コマンドの貼り付けを求める偽のCAPTCHA
- インストールしようとしていたツールの有毒な検索広告
- 静かに少し多すぎる権限を要求するブラウザ拡張
- postinstallスクリプトを含む開発依存関係を追加するプルリクエスト
- タスクに必要な以上のファイルシステムを読み込むAIコーディングセッション

これらの経路の一部はマルウェアをインストールする。一部はフィッシング直接把認証情報を盗む。一部はローカルエクスプロイトを必要としない。なぜならユーザーは attacker のコマンドを実行するよう騙されるからだ。

Microsoft の Lumma Stealer レポートは現在の脅威の形状を把握するのに役立つ。以下は、一つの prolific  infostealerファミリーへの配送ルートとしてフィッシングメール、マルウェア広告、侵害されたサイトでの drive-by ダウンロード、トロイ化されたアプリケーション、 legitimate サービスの悪用、偽のCAPTCHAフロー、マルウェアローダーを挙げている。興味深い部分は Lumma というブランド名ではない。興味深い部分是配布戦略だ：攻撃者はユーザーが1日中 halfway-trusted ドアの街を通り抜けるとき、完璧なドアを1つ必要としない。

CISA のフィッシングガイダンスは人間のレイヤーで同じポイントを主張している：フィッシングはもうメールだけではない。テキストメッセージ、ダイレクトメッセージ、電話、合作ツール、そして「これはもっともらしく見える」が多すぎる仕事をするような他の場所に姿を現す。

だから脅威レベルはこう設定しろ：

> あるプロセスが数分間あなたとして実行できる可能性があると仮定しろ。

rootとしてではない。永久ではない。cinematic 永続性としてではない。

ただ、あなたとして。

それで十分だ。

## ハードディスクが獎品だ

InfostealerはCPUを褒めたくない。

ディスクが欲しい。より正確に言えば、有用な trust が蓄積されるディスクの部分欲しい。

Microsoft は Lumma がブラウザ、アプリケーション、暗号通貨ウォレット、その他のローカルストアからデータを盗めると言っている。その disruption 後の投稿では、Lumma がパスワード、クレジットカード、銀行口座、ウォレットを盗むために使用され、Microsoft が2025年3月16日から5月16日の間に世界中で394,000台以上の感染したWindowsコンピュータを特定したと述べている。

Mandiant の Snowflake 調査はもっと怖いビジネスレッスンだ。UNC5537 キャンペーンで、Mandiant は対応したすべてのインシデントが Snowflake のエンタープライズ環境の侵害ではなく、顧客の認証情報の漏洩に遡ると報告した。認証情報は主に Infostealer 感染を受けた非 Snowflake システムから取得された。一部の認証情報は2020年にさかのぼる。キャンペーンで使用されたアカウントの少なくとも79.7%が過去の認証情報露出を持っていた。

この部分でChair が uncomfortable になるはずだ。

攻撃者は倉庫を breakout する必要がなかった。デスクの引き出しで古い鍵を見つけ、ロックが決して変更されていないことを発見した。

開発者にとって、デスクの引き出しは通常1つの引き出しではない。、それはジャンクルームだ：

| ローカルアーティファクト | 攻撃者が気にする理由 |
| --- | --- |
| ブラウザのCookieと保存されたセッション | ログインセレモニーをバイパスし、時として MFA 摩擦を減少できる。 |
| `.env`ファイル | APIキー、データベースURL、JWT秘密鍵、サードパーティトークンを含むことが多い。 |
| クラウドCLI設定 | ラップトップの侵害をインフラアクセスに変換できる。 |
| Git認証情報 | ソースコードはシステム、的秘密、デプロイパスのマップになる。 |
| SSH鍵 | まだ везде、まだ powerful、まだマシン間でコピーされている。 |
| データベースダンプ | バックアップはプロダクションより保護されていないことが多く、ログより完全であることが多い。 |
| AIコーディングコンテキスト | アシスタントは機密ファイル、コマンド履歴、または追加ディレクトリを渡されていた可能性がある。 |
| パッケージマネージャートークン | 公開トークンがローカルにある場合、サプライチェーンアクセスは hypothetical ではない。 |

バックアップはここで特別な disrespectful を受ける価値がある。

チームはIAM、ネットワークコントロール、監査ログ、そして adult supervision の小さな ceremony でプロダクションサーバーを保護する。次に誰かが同じデータを`customer-backup-final-2.sql.gz`にエクスポートし、ワークステーションにドロップし、それを忘れる。

そのファイルはプロダクションより多くの機密データを含む可能性がある、なぜならそれはコピーしやすく、検索しやすく、監視されにくいからだ。

バックアップは非アクティブだから安全なのではない。

ただ production にアラームシステムがないだけだ。

## 完全乗っ取りパターン

「データ漏洩」というフレーズは、しばしば続くものするには小さすぎる。

醜い道はこう見える：

1. 初期タッチ：ユーザーがファイルを開く、リンクをクリックする、ツールをインストールする、コピーしたコマンドを実行する、または侵害されたページに着く。
2. ローカルスクレイプ：マルウェアまたは悪意のあるプロセスがブラウザストア、ローカル設定、`.env`ファイル、トークン、SSH鍵、履歴、プロジェクトディレクトリを読み取る。
3. クラウドピボット：有効な認証情報がSaaS、クラウド、GitHub、CI、チャット、データウェアハウスへのアクセスを開く。
4. バックアップスイープ：ローカルエクスポート、クラウドバケット、CIアーティファクト、データベーススナップショットが取得される；プロダクションより soft だから。
5. 正当性による永続化：攻撃者が新しい鍵、OAuthアプリ、デプロイトークン、パーソナルアクセストークン、サービスアカウントを作成する。
6. 恐喝または静密な再販：データが直接収益化されるアクセスとして販売されるか、後続のキャンペーンに保存される。

ステップ2からステップ3への移動が、これがワークステーション問題だけではない理由だ。

あなたのラップトップはアイデンティティ broker だ。それはすべてのシステムに対してあなたであることを証明する。攻撃者がその証明を 충분히盗むことができれば、彼らはあなたに見えて現れることができる。

MFAは役立つ。ハードウェアキーはもっと役立つ。デバイスポスチャチェック、セッション束縛、IP許可リスト、条件付きアクセスはすべて役立つ。しかしあなたのローカルマシンが long-lived トークン、キャッシュされたセッション、プレーンテキスト的秘密、未監視のバックアップを含んでいるなら、あなたはまだ多くの制度的 trust を運ぶように1つのエンドポイントに頼っている。

目標は完璧な安全ではない。

目標は簡単な道を機能させなくすることだ。

## 開発者ツールは炸裂半径を拡大した

不快な部分は最高の開発者ツールも stakes を高めたということだ。

コンテナはローカル環境を再現可能にした。パッケージマネージャーは依存関係のインストールを frictionless にした。クラウドCLIはインフラをプログラム可能にした。AIコーディングツールはターミナルを会話的にした。

すべて良い。

 также：すべて dangerous  Secrets でいっぱいのワークステーションに向けた場合。

開発依存関係におけるサプライチェーン侵害は、プロダクションに配送される必要がないかもしれない。開発者マシンで実行される悪意のある postinstall スクリプトはローカルファイルを読み取り、環境変数を検査し、phone home できる。侵害されたCLIプラグインは同じことができる。広いファイルシステムとシェル権限を持つ親切なAIエージェントが悪い命令、悪い依存関係、または悪い assumption を増幅できる。

これが「小心たれ」が弱いアドバイスである理由だ。それは人間の境界であることを要求する。

人間は境界ではない。人間は交通だ。

境界は退屈なものであり、ファイルシステム分離、保存時の暗号化、default-deny 出力、short-lived 認証情報、ハードウェアバックド認証、偽の秘密が触れられたときに発火するアラートなどだ。

そこがソリューションが面白くなる場所だ。

## より良いフレーム：読み込み、使用、漏洩

すべてのワークステーション防御は3つの質問に答えるはずだ：

1. このプロセスは何を読めるか？
2. どの認証情報可以使用できるか？
3. データをどこ_send できるか？

ほとんどのワークステーションセキュリティ advice は最初の質問に over-focus している。ソフトウェアを最新の状態に保て。怪しい添付ファイルを開くな。アンウイルスを使用しろ。良い、はいObviously。

しかし悪意のあるプロセスが実行された場合、2番目と3番目の質問がbad afternoon と会社全体のインシデントのどちらになるかを決定する。

`~/.aws/credentials`を読めるか？

GitHubトークンを使用できるか？

パスワードマネージャー拡張を開けるか？

3 GB を誰にも気づかれずにランダムなホストにアップロードできるか？

バックアップフォルダを読めるか？

そのディレクトリが3ヶ月前に「追加コンテキスト」として含められたために、別のディレクトリの秘密を要約するようAIエージェントにリクエストできるか？

このフレームは仕事を practical に保つ。它は脅威を fog machine から teeth 付きのチェックリストに変換する。

## 私が最初にやるつもりだったこと

公司を sad airport にしないしながら 개발자 워크스테이션 프로그램을強化するなら、私はこれらのレイヤから始める。

### 1. 危険な作業をDev Containerに移動する

依存関係、ビルドツール、パッケージインストール、AI-assisted シェルコマンドを必要とするプロジェクト作業に[Development Containers](https://github.com/devcontainers/spec)を使用しろ。spec の約束は simple だ：コードベースのツールとランタイムを保持できる完全な機能の開発環境としてコンテナを使用しろ。

それは有用な境界を与えてくれる。魔法の境界ではない。有用な一つ。

勝利は`npm install`、`pip install`、`go generate`、`cargo build`、およびモデルが実行したいものはすべて、あなたのwhole ホームディレクトリを自動的に所有する workspace ではなく、コンテナ内で発生できることだ。

リポジトリをマウントしろ。そのプロジェクトに必要な secrets だけをマウントしろ。便宜上`~/.ssh`、`~/.aws`、`~/Downloads`、`~/Documents`、および entire ホームフォルダのマウントを避けてしろ。

プロジェクトが認証情報を必要とするなら、scoped 認証情報を注入しろ。short-lived トークンを優先しろ。可能な場合は読み取り専用アクセスを優先しろ。

コンテナはDockerを sophisticated に感じさせるためではなく、「このプロセスは私として実行できる」を-less catastrophic にするため存在する。

### 2. `.env`崇拝ではなくローカルSecrets を暗号化しろ

プレーンテキストの`.env`ファイルはファイルが便利だから便利だ。

攻撃者もファイルを好む。

[VarLock](https://varlock.dev/guides/secrets/)は興味深いなぜならそれは正規表現当てゲームではなく構造化されたメタデータとして機密性を扱うからだ。その docs は機密値を明示的にマークし、`varlock()`でローカル secrets を暗号化し、コンソール出力から機密値を redacted し、既知の機密値のプレーンテキスト出現をスキャンすることを描述している。

それが正しい方向だ：secrets は secrets であることを知るべきだ。

それはすべての認証情報問題を解決しない。すでに侵害されたプロセスにロードされた秘密を保護しない。しかしそれはbecome someone else's inventory を待っている価値のあるプレーンテキストファイルの数を減少させる。

### 3. 盗みが痛い場所に Canary Tokens を配置しろ

[Canarytokens](https://help.canary.tools/hc/en-gb/articles/10905485310109-Canarytoken-Overview-and-Use-Cases)はデジタル tripwires だ。あなたは攻撃者が調べるかもしれない場所に偽だがもっとものらしい秘密、ドキュメント、APIキー、URL、または認証情報を配置する。それが触れられたら、通知を受け取る。

のコツは-real compromise が自然浏览む場所にそれらを置くことだ：

- `.env`ファイルの隣に
- 偽の`~/backups/customer-export.sql` 내에
- 偽のAWSプロファイル内に
- 古く見えるパスワードスプレッドシート内に
- 非推奨認証情報のような名前のCI変数内に
- 侵入者が偵察中に開く documentation 内に

Canary は盗みを防がない。それらは検出時間を圧縮する。

それは重要だ、なぜなら多くの侵害は最初の1分で勝たないからだ。それらは最初の認証情報が機能した後の静かな時間に勝てる。

### 4. アウトバウンドファイアウォールを追加しろ

ほとんどの人は「ファイアウォール」と考え、インバウンド接続を想像する。それはワークステーション問題を見落とす。

マルウェアがローカルの secrets を読めるなら、次の質問はそれを送り出せるかどうかだ。

macOSでは、[LuLu](https://objective-see.org/products/lulu.html)가 бесплатный, open-source 옵션であり 알려지지 않은 나가는 연결에 대해 경고하고 나가는 트래픽만 모니터링한다. [Little Snitch](https://obdev.at/products/littlesnitch/)は polished commercial option であり、接続アラート、ネットワーク監視、プロファイル、per-app/domain 可視性がある。

Windows と Linux では、[Portmaster](https://safing.io/)が評価に値するなぜならそれは per-application ルールの open-source アプリケーションファイアウォールだからだ。Windows Defender Firewall は also outbound rules をサポートしているが、Microsoft の own ガイダンスは default-deny outbound が generally high-security environments 用であることに注意している；なぜならそれは carefully application inventory と rule management を必要とするからだ。

このレイヤーは最初は面倒だ。

それはスキップする理由ではない。それはプロファイル、許可リスト、期待値で展開するの理由だ。目標は1日中「拒否」をクリックすることではない。目标是当`invoice-viewer`、`postinstall`、または`python`があなたの火曜日に business がないドメインと話したいと思ったときに notice することだ。

### 5. AIコーディングツールを=== 記憶喪失のあるジュニア管理者===のように扱え

AIコーディングツールは悪くない。私はそれらを使う。私はそれらが好きだ。

しかしそれは読み取りアクセス、書き込みアクセス、シェルアクセス、ネットワークアクセス、そして confident momentum の才能を持つツールだ。

Anthropic の Claude Code docs はツール、ファイル、ドメイン、管理ポリシーの権限を描述し、権限とサンドボックスの区別をしている。権限はエージェントが使用できるものを決定する。サンドボックスは Bash ファイルシステムとネットワークアクセスに対して OS レベルの enforcement を提供する。

その区別が重要な部分だ。

ポリシーテキストはサンドボックスではない。許可プロンプトはサンドボックスではない。良いモデルはサンドボックスではない。

プロジェクトレベルの許可と拒否ルールを使用しろ。機密ファイルをワークディレクトリから除外しろ。危険なコマンドをコンテナ内で実行しろ。エージェントに「コンテキスト」として必要になるかもしれないという理由で entire ホームディレクトリを渡さないしろ。コンテキストは free ではない。時にはコンテキストはあなたのインシデントレポート already written だ。

## すべてのチーム Wiki に欲しいテーブル

これは別の年間トレーニングスライドより見たいワークステーションセキュリティマップだ。

| Layer | Bad default | Better default |
| --- | --- | --- |
| ファイルシステム | プロジェクト、secrets、ダウンロード、バックアップ、ツールがすべて1つのユーザーコンテキストを共有する。 | Dev Container で narrow mount を使用してプロジェクト作業を実行しろ。 |
| Secrets | プレーンテキスト`.env`ファイルと long-lived トークン。 | 暗号化されたローカル secrets、scoped トークン、短い有効期限、ハードウェアバックド認証。 |
| 検出 | EDRが漏洩が完了する前に notice することを期待する。 | 高価値のローカルとクラウドロケーションに Canary tokens。 |
| ネットワーク | any プロセスは reputation によってブロックされない限り呼び出し可能。 | 危険なツール用のルールを持つアウトバウンドアプリケーションファイアウォール。 |
| AIエージェント | メインのワークステーションコンテキストでの broad 読み取り/書き込み/シェル許可。 | プロジェクトスコープの許可 plus サンドボックス化されたコマンド。 |
| バックアップ | ローカルダンプとエクスポートが dead files のように扱われる。 | バックアップアーティファクトへのアクセスの暗号化、有効期限、隔離、監視。 |

ポイントは5つのツールを購入することではない。

ポイントはワークステーションを trusted blob として扱うことをやめることだ。

## バックアップに関する注記

バックアップはセキュリティプログラムが自分自身に嘘をつく場所です。

それらは necessary だ。それらは also dangerous だ。バックアップは often あなたが least want portable なものの most portable な形式だ。

開発者マシンの場合：

-  реаль적인 필요がない限りローカルにプロダクションエクスポートを保存하지 마라.
- ローカルバックアップとデータベースダンプを暗号化しろ。
- エクスポートに有効期限を追加しろ。
- バックアップのような場所に Canary 行またはドキュメントを追加しろ。
- それらを broad Dev Container mount から除外しろ。
- AIツールのコンテキストから除外しろ。
- バックアップ内に現れる認証情報をrotate しろ。

バックアップが認証情報を含むなら、それは単なるバックアップではない。それは delayed 乗っ取りキットだ。

## 実践的な標準

標準は「決して怪しい何かをクリックしない」であるべきではない。

それは poster の advice だ、not a system の。

実践的な標準は：

- 悪いPDFはすべてのプロジェクト secrets を読むべきではない
- 悪意のある依存関係は他のプロジェクトのクラウド認証情報を見るべきではない
- AIツールのミスは whole ホームフォルダを横断すべきではない
- infostealerはプレーンテキストバックアップと long-lived トークンを見つけるべきではない
- 未知のプロセスはローカルアラームなしで機密データをアップロードできるべきではない
- 盗まれた認証情報は完全な乗っ取りになる前に期限切れになるか、MFAに失敗するか、デバイスチェックに失敗するか、canaryに触れるべきではない

セキュリティは人間に完璧であることを求めることをやめ、compromise を profit less profitable にし始めたときに改善する。

あなたのラップトップはもうプロダクションの一部だ。

それに production-shaped 境界を与えろ。
