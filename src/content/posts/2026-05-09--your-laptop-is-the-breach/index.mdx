Opening a PDF should not be a production incident.

Clicking a link in an SMS should not become a backup compromise.

Installing a "quick utility" from a search result should not hand someone your cloud console, your source code, your CI tokens, your database exports, and the copy of production you forgot was sitting in `~/Downloads`.

And yet here we are.

The modern developer laptop is not a laptop anymore. It is a credential warehouse with a keyboard.

It has browser sessions. SSH keys. `.env` files. GitHub tokens. Package manager auth. Cloud CLIs. Password manager browser extensions. AI coding tools with shell access. Local databases. Old backups. One-off exports. Random PDFs from vendors. Maybe a crypto wallet, if the universe has chosen comedy.

The old mental model was: production is dangerous, local is convenient.

That model is done.

<p class="inset">
The question is not whether you can avoid every bad click. The question is whether one bad click can read everything, use everything, and leave before you notice.
</p>

But there is a harder version of this problem.

The attacker is not always a stranger.

Sometimes the attacker is a prompt you approved, a workflow you triggered, a dependency you installed, or a CI job you wrote. The breach is not always something that happened to you. Sometimes you are the one who ran the command.

That reframe matters. It changes what you defend against.

## Set the Threat Level

Most people set the threat level too low because they imagine a dramatic attack.

They imagine a zero-day in a PDF parser. They imagine a nation-state with a calendar invite and a monocle. They imagine something exotic enough that ordinary engineering discipline feels irrelevant.

The boring version is more useful.

A developer gets a message that looks normal enough:

- a PDF invoice from a contractor
- an SMS about a delivery or account warning
- a fake CAPTCHA that asks them to paste a command into their terminal
- a poisoned search ad for a tool they meant to install anyway
- a browser extension that quietly asks for a little too much
- a pull request that adds a dev dependency with a postinstall script
- an AI coding session that reads more of the filesystem than the task required
- a GitHub Actions workflow that exfiltrates secrets through an environment variable it was never supposed to see
- a prompt injected into a document, a web page, or a repository that redirects an AI agent's next action

Some of those paths install malware. Some steal credentials directly through phishing. Some do not need a local exploit at all because the user is tricked into running the attacker's command by hand.

Microsoft's Lumma Stealer writeup is a useful snapshot of the current shape. It lists phishing emails, malvertising, drive-by downloads on compromised sites, trojanized applications, abuse of legitimate services, fake CAPTCHA flows, and malware loaders as delivery routes for one prolific infostealer family. The interesting part is not Lumma as a brand name. The interesting part is the distribution strategy: attackers do not need one perfect door when users move through a city of half-trusted doors all day.

CISA's phishing guidance makes the same point at the human layer: phishing is not just email anymore. It shows up as text messages, direct messages, phone calls, collaboration tools, and other places where "this looks plausible" does too much work.

So set the threat level like this:

> Assume a process can run as you for a few minutes.

Not as root. Not forever. Not with cinematic persistence.

Just as you.

That is already enough.

## You Are the Breach

The phrase "my laptop was compromised" carries a passive voice that does not always fit.

Sometimes the story is: I cloned the repo, ran the install command, and the postinstall script phoned home before the test suite started. I opened the file someone sent. I approved the workflow trigger. I pasted the thing. I added the browser extension. I gave the agent "full context" because that was easier than specifying which files it needed.

The modern attack surface includes the places where you are the actor.

### Prompt Injection

AI coding tools are new enough that their attack surface is still being catalogued. One pattern worth knowing: prompt injection.

A malicious instruction hidden in a file, a web page, a repository README, a PR description, or a comment can redirect an agent's behavior. The agent reads the document as content. The hidden instruction is also content. If the model treats the injected text as a command — which depends on context, framing, and model behavior — the agent may take actions the user never intended: reading files, writing files, calling tools, or following a chain of instructions that was never the user's.

This does not require a compromised model. It requires a document the agent was asked to process.

Practical implications:

- Do not give agents unlimited filesystem access "for context." Context is not free.
- Review what an agent proposes to do before it does it, especially on files or commands it reached for without an explicit request.
- Be skeptical if an agent suddenly wants to read credentials, send network requests, or act on something "it found while looking at the project."
- Keep AI shell sessions inside Dev Containers with narrow mounts. The injected instruction can only act on what the agent can reach.

### GitHub CI/CD

GitHub Actions is a powerful, trusted, and frequently misconfigured attack surface.

The shape of the risk is different than a local laptop compromise, but the consequences often land in the same place: credentials, source code, and deployment access.

Common CI/CD attack patterns right now:

**Poisoned third-party actions.** Your workflow pulls `uses: some-org/some-action@v2`. If the upstream repository is compromised, or if that version tag was moved to a malicious commit, your workflow runs attacker code with your repository's secrets. Pin actions to commit SHAs instead of mutable tags.

**Pull request trigger abuse.** Workflows triggered by `pull_request_target` run in the context of the base repository and have access to secrets. An external contributor opens a PR. If your workflow is written carelessly, it may expose secrets to the PR's code. This is a known footgun and a documented GitHub advisory.

**Workflow injection via untrusted input.** If a workflow interpolates `${{ github.event.pull_request.title }}` or other user-controlled strings directly into a `run:` step, an attacker can craft a PR title that includes shell commands. GitHub's own security hardening docs call this out explicitly.

**Secret exfiltration from forks.** Workflows triggered by forked PRs do not receive repository secrets by default, but misconfigurations around environment protection rules or `pull_request_target` can change that.

The practical floor:

- Pin third-party actions to full commit SHAs.
- Never interpolate `github.event` fields directly into `run:` steps. Use intermediate environment variables.
- Keep production deployment secrets in environments with protection rules and required reviewers.
- Audit who can trigger workflows that have access to sensitive secrets.
- Use OIDC token exchange for cloud access instead of long-lived stored credentials wherever possible.

CI secrets live in a different threat model than laptop secrets, but the exfiltration path is the same: a process with credentials runs, reads them, and sends them somewhere.

## The Hard Disk Is the Prize

Infostealers are not trying to admire your CPU.

They want the disk. More specifically, they want the parts of the disk where useful trust accumulates.

Microsoft says Lumma can steal data from browsers, applications, cryptocurrency wallets, and other local stores. Its disruption post says Lumma was used to steal passwords, credit cards, bank accounts, and wallets, and that Microsoft identified more than 394,000 infected Windows computers globally between March 16 and May 16, 2025.

Mandiant's Snowflake investigation is the scarier business lesson. In the UNC5537 campaign, Mandiant reported that every incident it responded to traced back to compromised customer credentials, not a breach of Snowflake's enterprise environment. The credentials were primarily obtained from infostealer infections on non-Snowflake systems. Some credentials dated back to 2020. At least 79.7% of the accounts used in the campaign had prior credential exposure.

That is the part that should make your chair uncomfortable.

The attacker did not need to break the warehouse. They found old keys in a desk drawer and discovered the locks had never been changed.

For developers, the desk drawer is usually not one drawer. It is a junk room:

| Local artifact | Why attackers care |
| --- | --- |
| Browser cookies and saved sessions | They can bypass the login ceremony and sometimes reduce MFA friction. |
| `.env` files | They often contain API keys, database URLs, JWT secrets, and third-party tokens. |
| Cloud CLI config | It can turn a laptop compromise into infrastructure access. |
| Git credentials | Source code becomes a map of systems, secrets, and deployment paths. |
| SSH keys | Still everywhere, still powerful, still copied between machines. |
| Database dumps | Backups are often less protected than production and more complete than logs. |
| AI coding context | The assistant may have been handed sensitive files, command history, or extra directories. |
| Package manager tokens | Supply chain access is not hypothetical if your publishing token is local. |
| GitHub tokens | Fine-grained or classic PATs can read repos, trigger workflows, and publish packages. |

Backups deserve special disrespect here.

Teams protect production databases with IAM, network controls, audit logs, and a small ceremony of adult supervision. Then someone exports the same data to `customer-backup-final-2.sql.gz`, drops it on a workstation, and forgets it exists.

That file may contain more sensitive data than production because it is easier to copy, easier to search, and less likely to be monitored.

Backups are not safer because they are inert.

They are just production without an alarm system.

## The Complete Takeover Pattern

The phrase "data leak" is too small for what often follows.

The ugly path looks like this:

1. **Initial touch**: the user opens a file, clicks a link, installs a tool, runs a copied command, or lands on a compromised page.
2. **Inventory**: the process surveys the machine — directories, config files, browser stores, environment variables, running processes. Attackers want to know what they have before they take anything.
3. **Local scrape**: browser stores, local config, `.env` files, tokens, SSH keys, history, and project directories.
4. **Cloud pivot**: valid credentials open SaaS, cloud, GitHub, CI, chat, or data warehouse access.
5. **Backup sweep**: local exports, cloud buckets, CI artifacts, and database snapshots get pulled because they are softer than production.
6. **Persistence by legitimacy**: the attacker creates new keys, OAuth apps, deploy tokens, personal access tokens, or service accounts before the window closes.
7. **Extortion or quiet resale**: data gets monetized directly, sold as access, or saved for a later campaign.

The move from step two to step four is why this is not only a workstation problem.

Your laptop is an identity broker. It proves who you are to every system you use. If an attacker can steal enough of that proof, they can show up looking like you.

Notice step two: **inventory first**. Most attackers do a reconnaissance pass before exfiltrating. They look around. They open directories. They check what credentials are present. They try obvious paths.

This is the window canary tokens are designed to exploit.

## Developer Tools Made the Blast Radius Bigger

The uncomfortable part is that the best developer tools also increased the stakes.

Containers made local environments reproducible. Package managers made dependency installation frictionless. Cloud CLIs made infrastructure programmable. AI coding tools made the terminal conversational.

All good.

Also: all dangerous when pointed at a workstation full of secrets.

A supply chain compromise in a dev dependency may not need to ship to production to matter. A malicious postinstall script running on a developer machine can read local files, inspect environment variables, and phone home. A compromised CLI plugin can do the same. A helpful AI agent with broad filesystem and shell permissions can amplify a bad instruction, a bad dependency, or a bad assumption.

This is why "be careful" is such weak advice. It asks the human to be the boundary.

Humans are not boundaries. Humans are traffic.

Boundaries are boring things like filesystem isolation, encrypted-at-rest secrets, default-deny egress, short-lived credentials, hardware-backed auth, and alerts that fire when a fake secret gets touched.

## The Better Frame: Read, Use, Exfiltrate

Every workstation defense should answer three questions:

1. What can this process read?
2. What credentials can it use?
3. Where can it send data?

Most workstation security advice over-focuses on the first question. Keep software updated. Do not open suspicious attachments. Use antivirus. Good, yes, obviously.

But if a malicious process does run, the second and third questions decide whether you have a bad afternoon or a company-wide incident.

Can it read `~/.aws/credentials`?

Can it use a GitHub token?

Can it open your password manager extension?

Can it upload 3 GB to a random host without anyone noticing?

Can it read the backup folder?

Can it ask your AI agent to summarize secrets from another directory because that directory was included as "additional context" three months ago?

This frame keeps the work practical. It turns the threat from a fog machine into a checklist with teeth.

## What I Would Do First

If I were tightening a developer workstation program without turning the company into a sad airport, I would start with these layers.

### 1. Move Risky Work Into Dev Containers

Use [Development Containers](https://github.com/devcontainers/spec) for project work that needs dependencies, build tools, package installation, or AI-assisted shell commands.

That gives you a useful boundary. Not a magic boundary. A useful one.

The win is that `npm install`, `pip install`, `go generate`, `cargo build`, and whatever the model wants to run can happen in a workspace that does not automatically own your whole home directory.

Mount the repo. Mount only the secrets needed for that project. Avoid mounting `~/.ssh`, `~/.aws`, `~/Downloads`, `~/Documents`, and the entire home folder out of convenience.

```jsonc
// .devcontainer/devcontainer.json — narrow mounts only
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "workspaceFolder": "/workspaces/app",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ],
  "containerEnv": {
    "NODE_ENV": "development"
  },
  "postCreateCommand": "bun install"
}
```

If the project needs credentials, inject scoped credentials. Prefer short-lived tokens. Prefer read-only access where possible. A prompt-injected instruction can only reach what the agent can reach. Make that boring.

### 2. Encrypt Local Secrets Instead of Worshipping `.env`

Plaintext `.env` files are convenient because files are convenient.

Attackers also enjoy files.

[VarLock](https://varlock.dev/guides/secrets/) is interesting because it treats sensitivity as structured metadata instead of a regex guessing game. Its docs describe marking sensitive values explicitly, encrypting local secrets with `varlock()`, redacting sensitive values from console output, and scanning for plaintext occurrences of known sensitive values.

```dotenv
# .env.schema
# @sensitive
STRIPE_SECRET_KEY=

# @sensitive
DATABASE_URL=
```

That is the right direction: secrets should know they are secrets. It will not protect a secret that has already been loaded into a compromised process. But it reduces the number of valuable plaintext files sitting around waiting to become someone else's inventory.

### 3. Plant Canary Tokens Everywhere a Thief Would Look

This is the layer most teams skip, and it is arguably the most immediately useful.

[Canarytokens](https://help.canary.tools/hc/en-gb/articles/10905485310109-Canarytoken-Overview-and-Use-Cases) are digital tripwires. You place a fake-but-plausible secret, document, API key, URL, or credential somewhere an attacker might look. If it gets touched, you get an alert — often within seconds of the access.

Recall step two of the takeover pattern: **inventory first**. Attackers browse before they steal. They open config directories. They check for AWS credentials. They look for backup files. They read `.env` directories. That reconnaissance pass is your window.

A canary in the right place fires before the exfiltration starts.

The trick is placement. Random bait is trivia. Strategic bait is an alarm system.

**On the local machine:**

```text
~/backups/customer-prod-export-2024.sql
```

Put a canary URL or token inside it. If anything opens that file and follows the link, you know. Similarly:

```text
~/Documents/passwords-old.csv
~/.aws/credentials   ← add a fake [billing-prod-legacy] profile with a real canary AWS key
~/.ssh/config        ← add a fake host entry pointing to a canary
```

**In repositories:**

- a `.env.canary` file with fake credentials
- old deployment runbooks with fake service tokens
- deprecated config files an attacker would inspect during source reconnaissance

**In CI/CD:**

- a fake CI secret named like a deploy token
- a fake kubeconfig in a GitHub environment
- a fake database user in a workflow comment

**In cloud accounts:**

- a fake IAM user with no privileges but a real canary API key
- an unused S3 bucket path with a canary object
- a fake service account documented in a runbook

The alert should be actionable. A canary that emails an unattended inbox is decoration. Route it somewhere that wakes someone up: PagerDuty, Slack with a ping, SMS. Include which token fired, where it was planted, and the rotation checklist.

#### The Blind Spot Worth Knowing

Canary placement has one real limitation: attacker specialization.

A crypto-wallet infostealer may grab wallet files and browser stores and never touch your fake AWS credentials. A ransomware operator may encrypt the disk before any canary fires. A targeted threat actor who already knows your credential layout may skip the reconnaissance pass entirely.

That is fine. The canary strategy is not designed for every threat.

It is designed for the most common threat: an opportunistic attacker who runs a credential sweep, browses for interesting-looking files, and tries to inventory your access before deciding what to steal.

Most attackers who reach your disk want free infrastructure. Cloud credentials are currency. A fake AWS key that triggers an alert when an attacker tries to use it is exactly the kind of tripwire that catches that behavior — and gives you the window to rotate before they find the real one.

The goal is not omniscience. The goal is to make the reconnaissance pass expensive.

### 4. Add an Outbound Firewall

Most people think "firewall" and picture inbound connections. That misses the workstation problem.

If malware can read local secrets, the next question is whether it can send them out.

On macOS, [LuLu](https://objective-see.org/products/lulu.html) is the free, open-source option that alerts on unknown outgoing connections. [Little Snitch](https://obdev.at/products/littlesnitch/) is the polished commercial option with connection alerts, network monitoring, profiles, and per-app/domain visibility.

On Windows and Linux, [Portmaster](https://safing.io/) is worth evaluating — open-source application firewall with per-application rules.

This layer is annoying at first. That is not a reason to skip it. The goal is not to click "deny" heroically all day. The goal is to notice when `postinstall`, `python`, or `invoice-viewer` wants to talk to a domain that has no business being in your Tuesday.

### 5. Treat AI Coding Tools Like Junior Admins With Amnesia

AI coding tools are not bad. I use them. I like them.

But they are tools with read access, write access, shell access, network access, and a talent for confident momentum. They are also prompt-injectable if they read documents you did not fully review.

Anthropic's Claude Code docs distinguish permissions from sandboxing. Permissions decide what the agent may use. Sandboxing provides OS-level enforcement.

Policy text is not a sandbox. A permission prompt is not a sandbox. A nice model is not a sandbox.

Use project-level allow and deny rules. Keep sensitive files out of working directories. Run risky commands inside containers. Do not hand an agent your entire home directory because it might need "context."

## You Have Minutes, Maybe Hours

This is the part the threat-model diagrams omit.

When a canary fires — or when a vendor emails you about a suspicious login, or when GitHub alerts you that a token was used from an unexpected IP — the next step is not optional reading.

You have a window. It might be minutes. It might be a few hours, if the attacker is being patient. It is not a week.

That window is what canary tokens buy you. An attacker inventories your machine, touches your fake AWS key, triggers an alert, and you now have a lead time that did not exist before.

What to do with it:

- **Rotate first, investigate later.** Revoke tokens before you understand what happened. Understanding comes second. Damage limitation comes first.
- **Check GitHub PATs, OAuth apps, and deploy keys.** An attacker who had your laptop may have created new credentials before they left.
- **Review recent cloud activity.** New IAM users, new service accounts, new API keys, new S3 bucket policies. Persistence by legitimacy is real.
- **Audit CI secrets.** Check whether any workflows ran unexpectedly, especially in repositories you did not touch recently.
- **Kill active browser sessions.** Force logout on anything you care about.
- **Tell someone.** Security incidents improve with witnesses and timestamps.

The security community talks a lot about detection. It talks less about what happens in the twenty minutes after detection when you are alone at your desk trying to remember which services you have tokens for.

That list should exist before the alert fires.

## The Table I Want in Every Team Wiki

| Layer | Bad default | Better default |
| --- | --- | --- |
| Filesystem | Projects, secrets, downloads, backups, and tools all share one user context. | Run project work in Dev Containers with narrow mounts. |
| Secrets | Plaintext `.env` files and long-lived tokens. | Encrypted local secrets, scoped tokens, short lifetimes, hardware-backed auth. |
| Detection | Hope EDR notices before exfiltration finishes. | Canary tokens in high-value local, CI, cloud, and documentation locations. |
| Network | Any process can call out unless blocked by reputation. | Outbound application firewall with rules for risky tools. |
| AI agents | Broad read/write/shell permissions in the main workstation context. | Project-scoped permissions, prompt-injection awareness, sandboxed commands. |
| Backups | Local dumps and exports treated like dead files. | Encrypt, expire, isolate, and monitor access to backup artifacts. |
| CI/CD | Mutable action tags, broad secret access, unsafe input interpolation. | Pinned SHAs, scoped environments, OIDC token exchange, no interpolation of untrusted input. |

## A Note on Backups

Backups are where security programs go to lie to themselves.

They are necessary. They are also dangerous. A backup is often the most portable form of the thing you least want portable.

For developer machines:

- Do not store production exports locally unless there is a real need.
- Encrypt local backups and database dumps.
- Add expiration dates to exports.
- Put canary rows or documents inside backup-like files.
- Keep backups out of broad Dev Container mounts.
- Keep them out of AI tool context.
- Rotate any credential that appears inside a backup.

If the backup contains credentials, it is not just a backup. It is a delayed takeover kit.

## The Practical Standard

The standard should not be "never click anything weird."

That is advice for a poster, not a system.

The practical standard is:

- a bad PDF should not read all project secrets
- a malicious dependency should not see cloud credentials from other projects
- a prompt-injected document should not redirect an agent into your home directory
- a poisoned GitHub Action should not exfiltrate your deploy token
- an AI tool mistake should not traverse the whole home folder
- an infostealer should not find plaintext backups and long-lived tokens without triggering an alarm
- an unknown process should not upload sensitive data without a local alert
- a stolen credential should expire, fail MFA, fail device checks, or hit a canary before it becomes a complete takeover

Security gets better when we stop asking humans to be perfect and start making compromise less profitable.

Your laptop is part of production now.

The attacker does not always break in. Sometimes you let them in without knowing.

Give your systems the kind of boundaries that catch both.
