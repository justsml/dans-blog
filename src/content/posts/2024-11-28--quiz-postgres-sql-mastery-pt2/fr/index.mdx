  <div className="explanation">
    Bien que `SERIAL` soit encore couramment utilisé, la méthode **SQL Standard** est :
    ```sql

    ```
    - `id INT IDENTITY(1,1)` est la syntaxe SQL Server.
    - `id INT AUTO_INCREMENT` est la syntaxe MySQL.
    - `id INTEGER AUTO_INCREMENT PRIMARY KEY` est la syntaxe MySQL.

    - Conforme à la norme SQL
    - Meilleure gestion de la propriété des séquences
    - Sémantique plus claire pour les insertions manuelles
    - Fonctionne mieux avec la réplication

    `SERIAL` n'est en fait qu'un raccourci pour créer une séquence et définir une valeur par défaut, ce qui peut entraîner des problèmes avec les vidages et restaurations.

  </div>

  <slot name='explanation'>
  <div className="explanation">
    Pour moi, ce qui est surprenant ici, c'est qu'un index multi-colonnes peut être utilisé même lorsque l'ordre des colonnes dans la requête ne correspond pas à la définition de l'index.

    **Remarque :** Avec les index B-tree, la colonne de tête est importante. Comme cette requête filtre par `grade_level` et `last_name`, un index commençant par `grade_level` puis `last_name` est un bon choix.

    Plus le SQL peut sélectionner rapidement le moins de lignes, meilleure est la performance.

    Par exemple, si vous avez une table `task` et que vous savez que `status` est plus sélectif que `created_at`, vous pourriez choisir un index commençant par `status` pour une requête comme `status = 'done' AND created_at > '2024-01-01'`.     Un index B+Tree peut être utilisé pour :
    - La colonne de tête, ou un préfixe gauche de colonnes.
    - Les opérateurs tels que `=`, `>`, `<`, `BETWEEN` et `IN`, selon la forme de la requête.
    - La correspondance de préfixe, `LIKE 'prefix%'`.

    Merci à [u/mwdb2](https://www.reddit.com/r/PostgreSQL/comments/1h6kw87/comment/m0fovn0/?utm_source=share&utm_medium=web3x&utm_name=web3xcss&utm_term=1&utm_content=share_button) d'avoir signalé une erreur précédente dans cette explication !

    Comment spécifier un **nom de colonne contenant un point ?**
    ```sql
    CREATE TABLE teachers (
      id INT GENERATED BY DEFAULT AS IDENTITY,

      -- How do you include the column: first.name?
      salary NUMERIC
    );
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    PostgreSQL utilise les guillemets doubles pour citer les identifiants :
    - Les guillemets doubles `"..."` pour les identifiants (noms de colonnes, noms de tables)
    - Les guillemets simples `'...'` pour les chaînes littérales
    - La forme `first.name` est interprétée comme `table_name.column_name`

    Sans guillemets, un point dans un nom de colonne serait interprété comme un séparateur de schéma/table ! De plus, tous les identifiants non cités sont automatiquement mis en minuscules.

    Pour réduire les surprises, il est préférable d'utiliser `snake_case` et d'éviter les caractères spéciaux.

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={8}
  group="Échantillonnage de données"
  title="Syntaxe TABLESAMPLE"
  options={[
  ]}
>
  <slot name="question">
  <div className="question">
    Quelle est la syntaxe valide pour **échantillonner aléatoirement 10%** des étudiants ?
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Celui-ci est piégeux ! Il comporte environ 6 pièges, dont :

    1. Description de haut niveau + fragments de syntaxe. Teste comment vous adaptez et assemblez des informations incomplètes.
    2. Nécessite une connaissance familière de la syntaxe `TABLESAMPLE`, car aucune des options `TABLESAMPLE` n'utilise la syntaxe correcte ! (Diabolique ! Je sais ! Continuez, je crois en vous !!! ❤️)
    3. Les `ROW_NUMBER() OVER (ORDER BY RAND())` et `ORDER BY RANDOM() > 10` sont des leurres supplémentaires. Ils ont l'air corrects, mais ils ne le sont pas. `RAND()` n'existe pas. Et `RANDOM() > 10` est une expression booléenne qui sera toujours fausse car `RANDOM()` est toujours dans l'intervalle `0.0-1.0`.
    Un autre problème : `ORDER BY RANDOM() > 10` trierait aussi sur une valeur booléenne, pas sur un nombre aléatoire. Mais ce `> 10` donnait l'impression de pouvoir être un pourcentage, n'est-ce pas ?
    4. `BERNOULLI` est valide, mais `SAMPLETABLE` ne l'est pas.
    5. `WHERE RANDOM() >= 0.1` est un piège ! `RANDOM()` renvoie une valeur dans l'intervalle `0.0 <= x < 1.0`, donc `>= 0.1` renverrait environ 90% des lignes, pas 10%.

    En ce qui concerne la méthode `TABLESAMPLE`, PostgreSQL prend en charge deux méthodes d'échantillonnage :
    - `BERNOULLI` : chaque ligne a une probabilité égale
    - `SYSTEM` : échantillonnage au niveau du bloc (plus rapide mais moins aléatoire)

    Pour en savoir plus sur l'échantillonnage, consultez cet article de [Render](https://render.com/blog/postgresql-random-samples-big-tables).

    Exemple :
    ```sql
    -- Sample 10% of rows
    SELECT * FROM students TABLESAMPLE BERNOULLI (10);

    -- For repeatable results, use REPEATABLE:
    SELECT * FROM students
    TABLESAMPLE BERNOULLI (10) REPEATABLE (42);
    ```
    Il convient également de mentionner que `TABLESAMPLE` [a été signalé](https://stackoverflow.com/questions/8674718/best-way-to-select-random-rows-postgresql) comme n'étant pas très aléatoire. Si vous avez besoin d'un échantillon de taille fixe _plus_ aléatoire, envisagez d'utiliser `ORDER BY RANDOM() LIMIT 10`. Mais sachez que cela renvoie 10 lignes, pas 10%, et peut être lent sur les grandes tables.

    Il existe également quelques extensions tierces qui fournissent de meilleures méthodes d'échantillonnage aléatoire, mais cela dépasse le cadre de cette question ! 😅
  </div>
  </slot>
</Challenge>

  group="Comportement de NULL"
  title="COUNT et NULL"
  options={[
    {text: '100'},
    {text: '90'},
    {text: '10'},
    {text: '1', isAnswer: true},
    {text: '0'},
  ]}
>
  <slot name="question">
  <div className="question">
    Combien de *lignes* cela va-t-il retourner ?
    ```sql
    SELECT COUNT(*)
    FROM students
    WHERE grade = NULL;
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Ok, c'était un peu piégeux. Le piège est que `x = NULL` ne correspondra jamais à aucune ligne, et cela retournera TOUJOURS une ligne avec un compte de `0`.

    Puisque `NULL` est une valeur inconnue, elle ne peut être comparée à rien, pas même à elle-même.

    La bonne façon de vérifier les valeurs `NULL` est avec `IS NULL`.

    Exemple :
    ```sql
    postgres=# SELECT count(*)
      FROM students
      WHERE age = null;
    count 
    -------
        0
    (1 row)

    postgres=# 
    postgres=# SELECT count(*)
      FROM students
      WHERE age is null;
    count 
    -------
        3
    (1 row)
    ```
  </div>
  </slot>
