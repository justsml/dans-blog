---

AIエージェントを構築しましたね。おそらく、それは素晴らしいものでしょう。プロンプトは洗練され、モデルは高速で、レスポンスは自然に感じられます。

しかし、誰かが Salesforce で顧客レコードを確認するように頼んだとしたらどうでしょう？ あるいは、最新の Jira チケットを取得したり、社内ドキュメントを検索したりするように頼んだとしたら？

すると、あなたの美しいエージェントは... 何もできなくなります。

これは、すべてのAIプラットフォームがいずれ直面する統合の問題です。エージェントには「手」が必要です。実際のビジネスシステムを覗き込む「目」が必要です。それらがなければ、単なる高価なチャットボットを動かしているに過ぎません。

従来の解決策は？ 接続したいすべてのサービスに対して、カスタムAPIラッパーを個別に作成することでした。ドキュメントを読み、認証を処理し、レート制限に対処し、来月エンドポイントが変更されないことを祈る。そして、次のサービスでも同じことを繰り返す。その次も。

Model Context Protocol (MCP) は、この計算を根本から変えます。

---

## MCPが実際に解決すること

USB-Cが登場する前のUSBについて考えてみてください。Mini-USB、Micro-USB、Apple独自のコネクタがあり、特定のデバイスでしか使えないケーブルが引き出しに溢れていました。USB-Cは単に新しいコネクタを追加しただけではありません。どんなケーブルでもどんなデバイスでも動作することを意味する「標準」を確立したのです。

MCPは、AIツールの統合において同じことを行っています。

Salesforce、HubSpot、GitHub、あるいはその他のサービスに接続するためにカスタムコードを書く代わりに、プロトコルを一度実装する（あるいはビルド済みのサーバーをダウンロードする）だけで、MCP互換のエージェントならどれでも即座にそれと対話できるようになります。

プロトコルが通信レイヤーを処理します。あなたは、ツールが何を目的とし、どのようなデータを必要とするかを定義するだけです。

## 複数の統合のセットアップ

Mastraは、[`MCPClient`](https://mastra.ai/docs/mcp/overview) を通じてネイティブにMCPをサポートしています。ローカルツール（子プロセスとして実行）とリモートサービス（独自のインフラで実行）の両方に接続できます。

以下は、ルート検索用のGoogle Maps、気象サービス、およびローカルのWikipedia検索を接続する、現実的なプロダクション環境のセットアップです。

```typescript
// src/mastra/mcp/index.ts
import { MCPClient } from '@mastra/mcp';

export const mcpClient = new MCPClient({
  servers: {
    // ローカルツール (Stdio)
    wikipedia: {
      command: 'npx',
      args: ['-y', 'wikipedia-mcp'],
    },
    // 地図とナビゲーション (Remote/HTTP)
    googleMaps: {
      url: new URL(process.env.GOOGLE_MAPS_MCP_URL!),
      requestInit: {
        headers: {
          Authorization: `Bearer ${process.env.GOOGLE_MAPS_API_KEY}`,
        },
      },
    },
    // 気象サービス統合
    weather: {
      url: new URL('https://mcp.weatherapi.dev/v1'),
      requestInit: {
        headers: {
          'X-API-Key': process.env.WEATHER_API_KEY!,
        },
      },
    },
  },
});
```

クライアントは接続のライフサイクルを管理し、ローカルツールのプロセス生成を処理し、リモートサーバーへのHTTP接続を維持します。ソケットや標準入出力を直接操作する必要はありません。

## ツールをエージェントに接続する

MCPクライアントを設定したら、それらのツールをエージェントに提供するのは簡単です。

```typescript
// src/mastra/agents/navigation-agent.ts
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { mcpClient } from '../mcp';

export const navigationDirectionsAgent = new Agent({
  id: 'navigation-directions-agent',
  name: 'ナビゲーション＆ルート案内アシスタント',
  instructions: `あなたは、ルート計画と旅行のアドバイスを提供する役立つナビゲーションアシスタントです。
    - 必ず出発地と目的地を確認してください
    - 最適なルートを見つけるためにGoogle Mapsツールを使用してください
    - ルート沿いの気象条件を確認してください
    - 推定旅行時間を提供し、天候が悪い場合は代替案を提案してください
    - 交通状況、道路状況、立ち寄りスポットなどの関連する詳細を含めてください
    - 回答は明確で実行可能なものにしてください`,
  model: openai('gpt-5'),
  tools: await mcpClient.getTools(), // <--- これが魔法の一行です
});
```

ユーザーが次のように尋ねたとき： *"サンフランシスコからレイクタホへの最適なルートは何ですか？ また、天候を心配する必要がありますか？"*

エージェントは利用可能なツールの定義を読み、Google Mapsのルート検索と天気予報ツールへのアクセス権があることを認識し、適切なパラメータでそれらを実行し、最適なルートに道中の現在の気象条件を添えて回答します。

あなたは、Google Maps APIのコードも気象サービスの統合コードも、一行も書いていないのです。

## ユーザーごとの認証

ここで陥りやすいセキュリティ上のミスがあります。それは、認証情報のハードコーディングです。

環境変数に1つのGoogle Maps APIキーを入れて済ませてしまうと、すべてのユーザーが同じクォータとレート制限を共有することになります。さらに重要なことに、保存された場所や好みのルートなどのユーザー設定を保存するサービスを使用している場合、全員が同じデータを見ることになってしまいます。これはデモでは問題ありませんが、本番環境ではリスクになります。

Mastraは、ユーザー固有の認証情報を使用してMCPクライアントを動的に作成できるようにすることで、この問題に対処します。

```typescript
async function handleUserRequest(userPrompt: string, userCredentials: UserCreds) {
  // この特定のユーザーのためのクライアントを作成
  const userMcp = new MCPClient({
    servers: {
      googleMaps: {
        url: new URL(process.env.GOOGLE_MAPS_MCP_URL!),
        requestInit: {
          headers: {
            // ユーザー固有のAPIキーまたはトークン
            Authorization: `Bearer ${userCredentials.mapsApiKey}`,
            'X-User-ID': userCredentials.userId,
          },
        },
      },
    },
  });

  const agent = mastra.getAgent('navigationDirectionsAgent');
  
  // 実行時にツールを注入
  const response = await agent.generate(userPrompt, {
    toolsets: await userMcp.getToolsets(),
  });

  return response;
}
```

各ユーザーは、独自のAPIクォータと設定を持つ、分離されたツールセットを取得します。ユーザーAの保存された場所はプライベートなまま維持され、ユーザーBのルート履歴は別に管理されます。これが、マルチテナントSaaSエージェントが実際に機能する仕組みです。

## 複合ツールの構築

複数のMCPツールを1つの操作に組み合わせる必要がある場合もあります。例えば、リアルタイムの交通状況と道中の気象条件の両方を考慮したルートを計画したい場合などです。

MCPツールをカスタムツールの定義の中にラップすることができます。

```typescript
export const smartRouteTool = createTool({
  id: 'smart-route-planner',
  description: '交通状況と気象条件を考慮して最適なルートを計画します',
  execute: async ({ context, mastra }) => {
    // 生のツールを取得
    const tools = await mcpClient.getTools();
    
    // 1. Google Mapsから基本ルートを取得
    const routeData = await tools.googleMaps_getDirections.execute({ 
      context: { 
        origin: context.origin,
        destination: context.destination 
      } 
    });
    
    // 2. ルート沿いの天気をチェック
    const weatherData = await tools.weather_getForecast.execute({
      context: { coordinates: routeData.waypoints }
    });
    
    // 3. 気象警告を追加した強化ルートを返す
    return { 
      ...routeData, 
      weatherAlerts: weatherData.alerts,
      recommendation: weatherData.severe ? '出発を遅らせることを検討してください' : '旅行は安全です'
    };
  },
});
```

これにより、重い処理にはMCPプロトコルを活用しつつ、ツールがどのように相互作用するかをきめ細かく制御できます。

## その先にあるもの

AIエージェントが対話する必要のあるすべてのサービスに対してカスタムAPIクライアントを書くことは、決して持続可能ではありませんでした。スケーラビリティに欠け、頻繁に壊れ、プラットフォームを特定の実装に縛り付けてしまいます。

MCPは、すべての統合の課題を解決するわけではありません。認証は依然として複雑で、レート制限も重要であり、すべてのサービスがまだMCPサーバーを持っているわけではありません。しかし、MCPはエージェントプラットフォームの構築を大幅に容易にする「基盤」を確立します。

外部サービスと対話する必要があるAIシステムを設計しているなら、MCPを理解することは、おそらく時間をかける価値があるでしょう。
