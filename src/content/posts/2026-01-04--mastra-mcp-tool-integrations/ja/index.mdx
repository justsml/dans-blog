title: "あなたのAIエージェントは、これなしでは使えない"
subTitle: "なぜMCPは人工知能のUSB-Cなのか。"
---

AIエージェントを構築した。しかも、それは優れたものかもしれない。プロンプトは緻密で、モデルは高速で、応答は自然に感じられる。

しかし、誰かがSalesforceで顧客レコードを確認してほしいと頼んだり、最新のJiraチケットを取得してほしいと言ったり、社内ドキュメントを検索してほしいと頼んだりする。

そして、あなたの美しいエージェントは、ただ……できない。

これは、あらゆるAIプラットフォームが最終的に直面する統合の問題だ。エージェントには「手」が必要だ。実際のビジネスシステムを覗き込む「目」が必要だ。それがなければ、あなたが運営しているのは高価なチャットボットに過ぎない。

従来の解決策は？ 接続したいサービスごとにカスタムAPIラッパーを書くことだ。ドキュメントを読み、認証を処理し、レート制限に対処し、来月にエンドポイントが変更されないことを祈る。そして次のサービスでも同じことを繰り返す。また次のサービスでも。

Model Context Protocol（MCP）は、この方程式を完全に書き換える。

---

## MCPが実際に解決すること

USB-C以前のUSBを考えてみよう。Mini-USB、Micro-USB、Apple独自のコネクタ、そして特定のデバイスでしか動作しないケーブルでいっぱいの引き出しがあった。USB-Cは単に新しいコネクタを追加したのではない——あらゆるケーブルがあらゆるデバイスで動作するという標準を確立したのだ。

MCPは、AIツールの統合に対して同じことを行っている。

エージェントをSalesforce、HubSpot、GitHub、その他のサービスに接続するためにカスタムコードを書く代わりに、プロトコルを一度実装する（または事前に構築されたサーバーをダウンロードする）だけで、MCP対応のエージェントなら即座にそれと通信できる。

プロトコルが通信層を処理する。あなたが行うべきは、ツールが何をし、どのデータが必要かを定義するだけだ。

---

## 複数の統合をセットアップする

Mastraは、[`MCPClient`](https://mastra.ai/docs/mcp/overview)を通じてネイティブなMCPサポートを提供している。ローカルツール（子プロセスとして実行されるもの）とリモートサービス（独自のインフラストラクチャ上で実行されるもの）の両方を接続できる。

以下は、ルーティング用のGoogle Maps、天気サービス、およびローカルのWikipedia検索を接続する現実的な本番環境の設定例だ：

```typescript
// src/mastra/mcp/index.ts
import { MCPClient } from '@mastra/mcp';

export const mcpClient = new MCPClient({
  servers: {
    // Local tool (Stdio)
    wikipedia: {
      command: 'npx',
      args: ['-y', 'wikipedia-mcp'],
    },
    // Maps & Navigation (Remote/HTTP)
    googleMaps: {
      url: new URL(process.env.GOOGLE_MAPS_MCP_URL!),
      requestInit: {
        headers: {
          Authorization: `Bearer ${process.env.GOOGLE_MAPS_API_KEY}`,
        },
      },
    },
    // Weather Service Integration
    weather: {
      url: new URL('https://mcp.weatherapi.dev/v1'),
      requestInit: {
        headers: {
          'X-API-Key': process.env.WEATHER_API_KEY!,
        },
      },
    },
  },
});
```

クライアントは接続のライフサイクルを管理し、ローカルツールのプロセス生成を処理し、リモートサーバーとのHTTP接続を維持する。ソケットやstdioを直接操作する必要はない。

## ツールをエージェントに接続する

MCPクライアントの設定が完了すれば、それらのツールをエージェントに渡すのは簡単だ：

```typescript
// src/mastra/agents/navigation-agent.ts
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { mcpClient } from '../mcp';

export const navigationDirectionsAgent = new Agent({
  id: 'navigation-directions-agent',
  name: 'Navigation & Directions Assistant',
  instructions: `You are a helpful navigation assistant that provides route planning and travel advice.
    - Always confirm the start and destination locations
    - Use Google Maps tools to find optimal routes
    - Check weather conditions along the route
    - Provide estimated travel times and suggest alternatives if weather is poor
    - Include relevant details like traffic, road conditions, and points of interest
    - Keep responses clear and actionable`,
  model: openai('gpt-5'),
  tools: await mcpClient.getTools(), // <--- This is the magic line
});
```

ユーザーが次のように尋ねた場合：*「サンフランシスコからレイク・タホへの最適なルートは何か、また天気を心配する必要があるか？」*

エージェントは利用可能なツール定義を読み取り、Google Mapsのルーティングツールと天気予報ツールにアクセスできることを認識し、適切なパラメータでそれらを実行し、最適なルートと道中の現在の天気状況を含む回答を返す。

Google Maps APIのコードや天気サービスの統合コードを1行も書いていない。

## ユーザーごとの認証

ここで犯しやすいセキュリティ上のミスがある：認証情報をハードコーディングすることだ。

環境変数に1つのGoogle Maps APIキーを置いてそれで終わりにすれば、すべてのユーザーが同じクォータとレート制限を共有することになる。さらに重要なことに、保存された場所やお気に入りのルートなどのユーザープレファレンスを保存するサービスを使用している場合、誰もが同じデータを見ることになる。これはデモでは十分機能する。しかし、本番環境ではリスクだ。

Mastraは、ユーザー固有の認証情報でMCPクライアントを動的に作成できるようにすることで、これを処理する：

```typescript
async function handleUserRequest(userPrompt: string, userCredentials: UserCreds) {
  // Create a client for THIS specific user
  const userMcp = new MCPClient({
    servers: {
      googleMaps: {
        url: new URL(process.env.GOOGLE_MAPS_MCP_URL!),
        requestInit: {
          headers: {
            // User's specific API key or token
            Authorization: `Bearer ${userCredentials.mapsApiKey}`,
            'X-User-ID': userCredentials.userId,
          },
        },
      },
    },
  });

  const agent = mastra.getAgent('navigationDirectionsAgent');
  
  // Inject tools at runtime
  const response = await agent.generate(userPrompt, {
    toolsets: await userMcp.getToolsets(),
  });

  return response;
}
```

各ユーザーは、独自のAPIクォータとプリファレンスを持つ独立したツールセットを得る。ユーザーAの保存した場所は非公開のままで、ユーザーBのルート履歴は別々に保たれる。これが実際のマルチテナントSaaSエージェントの動作方式だ。

## 複合ツールの構築

複数のMCPツールを単一の操作に結合する必要がある場合がある。たとえば、リアルタイムの交通状況と道中の天気条件の両方を考慮したルートを計画したい場合などだ。

MCPツールをカスタムツール定義でラップできる：

```typescript
export const smartRouteTool = createTool({
  id: 'smart-route-planner',
  description: 'Plans optimal route considering traffic and weather conditions',
  execute: async ({ context, mastra }) => {
    // Get the raw tools
    const tools = await mcpClient.getTools();
    
    // 1. Get base route from Google Maps
    const routeData = await tools.googleMaps_getDirections.execute({ 
      context: { 
        origin: context.origin,
        destination: context.destination 
      } 
    });
    
    // 2. Check weather along the route
    const weatherData = await tools.weather_getForecast.execute({
      context: { coordinates: routeData.waypoints }
    });
    
    // 3. Return enhanced route with weather warnings
    return { 
      ...routeData, 
      weatherAlerts: weatherData.alerts,
      recommendation: weatherData.severe ? 'Consider delaying trip' : 'Safe to travel'
    };
  },
});
```

これにより、重い処理はMCPプロトコルに任せつつ、ツールがどのように連携するかを細かく制御できる。

---

## これが導く先

AIエージェントが通信する必要のあるすべてのサービスに対してカスタムAPIクライアントを書くことは、決して持続可能ではなかった。スケールが悪く、頻繁に壊れ、プラットフォームを特定の実装に縛り付ける。

MCPはすべての統合課題を解決するわけではない——認証は依然として複雑で、レート制限は依然として重要であり、まだすべてのサービスにMCPサーバーがあるわけではない。しかし、エージェントプラットフォームの構築を大幅に楽にする基盤を確立する。

外部サービスと連携する必要のあるAIシステムを設計しているなら、MCPを理解することはおそらく価値のある時間の使い方だ。
