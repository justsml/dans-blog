आपने एक AI एजेंट बनाया है। शायद यह बहुत अच्छा भी हो। इसके प्रॉम्प्ट्स सटीक हैं, मॉडल तेज़ है, और इसके उत्तर स्वाभाविक लगते हैं।

लेकिन फिर कोई इसे सेल्सफोर्स (Salesforce) में ग्राहक रिकॉर्ड चेक करने के लिए कहता है। या नवीनतम जीरा (Jira) टिकट निकालने के लिए। या आपके आंतरिक दस्तावेज़ों को खोजने के लिए।

और आपका शानदार एजेंट बस... यह नहीं कर पाता।

यह एकीकरण (integration) की समस्या है जिससे हर AI प्लेटफ़ॉर्म अंततः टकराता है। आपके एजेंट को हाथों की ज़रूरत है। उसे आपके वास्तविक व्यावसायिक सिस्टम में देखने के लिए आँखों की ज़रूरत है। उनके बिना, आप बस एक महंगा चैटबॉट चला रहे हैं।

पारंपरिक समाधान? हर उस सर्विस के लिए एक कस्टम API रैपर लिखें जिसे आप कनेक्ट करना चाहते हैं। उनके दस्तावेज़ पढ़ें, उनके प्रमाणीकरण (auth) को संभालें, उनकी रेट लिमिट्स (rate limits) से निपटें, और प्रार्थना करें कि वे अगले महीने अपने एंडपॉइंट्स न बदलें। फिर अगली सर्विस के लिए भी यही करें। और फिर अगली।

मॉडल कॉन्टेक्स्ट प्रोटोकॉल (Model Context Protocol - MCP) इस पूरी गणना को बदल देता है।

---

## MCP असल में क्या हल करता है

USB-C से पहले USB के बारे में सोचें। आपके पास Mini-USB, Micro-USB, ऐपल के प्रोप्रायटरी कनेक्टर्स थे, और केबलों से भरी एक दराज थी जो केवल विशिष्ट डिवाइस के साथ काम करती थी। USB-C ने केवल एक नया कनेक्टर नहीं जोड़ा—इसने एक मानक (standard) स्थापित किया जिसका मतलब था कि कोई भी केबल किसी भी डिवाइस के साथ काम कर सकती है।

MCP एआई टूल एकीकरण (integrations) के लिए वही कर रहा है।

अपने एजेंट को सेल्सफोर्स, हबस्पॉट, गिटहब या किसी अन्य सर्विस से जोड़ने के लिए कस्टम कोड लिखने के बजाय, आप एक बार प्रोटोकॉल लागू करते हैं (या प्री-बिल्ट सर्वर डाउनलोड करते हैं), और कोई भी MCP-संगत एजेंट उससे तुरंत बात कर सकता है।

प्रोटोकॉल संचार की परत (communication layer) को संभालता है। आप बस यह परिभाषित करते हैं कि आपके टूल्स क्या करते हैं और उन्हें किस डेटा की आवश्यकता है।

---

## एकाधिक एकीकरण (Integrations) सेटअप करना

Mastra में इसके [`MCPClient`](https://mastra.ai/docs/mcp/overview) के माध्यम से नेटिव MCP सपोर्ट है। आप स्थानीय टूल्स (चाइल्ड प्रोसेस के रूप में चल रहे) और रिमोट सर्विसेस (अपने स्वयं के इंफ्रास्ट्रक्चर पर चल रहे) दोनों को जोड़ सकते हैं।

यहाँ गूगल मैप्स (रूटिंग के लिए), एक वेदर सर्विस और लोकल विकिपीडिया सर्च को जोड़ने वाला एक वास्तविक प्रोडक्शन सेटअप है:

```typescript
// src/mastra/mcp/index.ts
import { MCPClient } from '@mastra/mcp';

export const mcpClient = new MCPClient({
  servers: {
    // Local tool (Stdio)
    wikipedia: {
      command: 'npx',
      args: ['-y', 'wikipedia-mcp'],
    },
    // Maps & Navigation (Remote/HTTP)
    googleMaps: {
      url: new URL(process.env.GOOGLE_MAPS_MCP_URL!),
      requestInit: {
        headers: {
          Authorization: `Bearer ${process.env.GOOGLE_MAPS_API_KEY}`,
        },
      },
    },
    // Weather Service Integration
    weather: {
      url: new URL('https://mcp.weatherapi.dev/v1'),
      requestInit: {
        headers: {
          'X-API-Key': process.env.WEATHER_API_KEY!,
        },
      },
    },
  },
});
```

क्लाइंट कनेक्शन के जीवनचक्र को प्रबंधित करता है, स्थानीय टूल्स के लिए प्रोसेस स्पॉनिंग को संभालता है, और रिमोट सर्वर के लिए HTTP कनेक्शन बनाए रखता है। आपको सीधे सॉकेट या stdio को छूने की ज़रूरत नहीं है।

## टूल्स को एजेंट से जोड़ना

एक बार जब आप अपना MCP क्लाइंट कॉन्फ़िगर कर लेते हैं, तो उन टूल्स को एजेंट को देना सीधा है:

```typescript
// src/mastra/agents/navigation-agent.ts
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { mcpClient } from '../mcp';

export const navigationDirectionsAgent = new Agent({
  id: 'navigation-directions-agent',
  name: 'Navigation & Directions Assistant',
  instructions: `You are a helpful navigation assistant that provides route planning and travel advice.
    - Always confirm the start and destination locations
    - Use Google Maps tools to find optimal routes
    - Check weather conditions along the route
    - Provide estimated travel times and suggest alternatives if weather is poor
    - Include relevant details like traffic, road conditions, and points of interest
    - Keep responses clear and actionable`,
  model: openai('gpt-5'),
  tools: await mcpClient.getTools(), // <--- यह जादुई लाइन है
});
```

जब कोई उपयोगकर्ता पूछता है: *"सैन फ्रांसिस्को से लेक ताहो तक जाने का सबसे अच्छा रास्ता क्या है, और क्या मुझे मौसम के बारे में चिंतित होना चाहिए?"*

एजेंट उपलब्ध टूल परिभाषाओं को पढ़ता है, उसे एहसास होता है कि उसके पास गूगल मैप्स रूटिंग और मौसम पूर्वानुमान टूल्स तक पहुंच है, वह उन्हें सही पैरामीटर्स के साथ निष्पादित करता है, और रास्ते में वर्तमान मौसम की स्थिति के साथ एक इष्टतम मार्ग का उत्तर देता है।

आपने गूगल मैप्स एपीआई कोड या वेदर सर्विस एकीकरण की एक भी लाइन नहीं लिखी।

## प्रति-उपयोगकर्ता प्रमाणीकरण (Authentication)

यहाँ एक सुरक्षा गलती है जिसे करना आसान है: क्रेडेंशियल्स को हार्डकोड करना।

यदि आप अपने एनवायरनमेंट वेरिएबल्स में एक गूगल मैप्स एपीआई की (API key) डालते हैं और काम खत्म मान लेते हैं, तो हर उपयोगकर्ता एक ही कोटा और रेट लिमिट शेयर करता है। इससे भी महत्वपूर्ण बात यह है कि यदि आप ऐसी सर्विसेस का उपयोग कर रहे हैं जो उपयोगकर्ता प्राथमिकताओं (जैसे सहेजे गए स्थान या पसंदीदा मार्ग) को स्टोर करती हैं, तो हर कोई एक ही डेटा देखेगा। यह डेमो के लिए ठीक काम करता है। प्रोडक्शन में यह एक जोखिम (liability) है।

Mastra इसे आपको उपयोगकर्ता-विशिष्ट क्रेडेंशियल्स के साथ गतिशील रूप से MCP क्लाइंट बनाने की अनुमति देकर संभालता है:

```typescript
async function handleUserRequest(userPrompt: string, userCredentials: UserCreds) {
  // इस विशिष्ट उपयोगकर्ता के लिए एक क्लाइंट बनाएं
  const userMcp = new MCPClient({
    servers: {
      googleMaps: {
        url: new URL(process.env.GOOGLE_MAPS_MCP_URL!),
        requestInit: {
          headers: {
            // उपयोगकर्ता की विशिष्ट API key या टोकन
            Authorization: `Bearer ${userCredentials.mapsApiKey}`,
            'X-User-ID': userCredentials.userId,
          },
        },
      },
    },
  });

  const agent = mastra.getAgent('navigationDirectionsAgent');
  
  // रनटाइम पर टूल्स इंजेक्ट करें
  const response = await agent.generate(userPrompt, {
    toolsets: await userMcp.getToolsets(),
  });

  return response;
}
```

प्रत्येक उपयोगकर्ता को अपने स्वयं के एपीआई कोटा और प्राथमिकताओं के साथ अपना स्वयं का आइसोलेटेड टूलसेट मिलता है। उपयोगकर्ता A के सहेजे गए स्थान निजी रहते हैं, उपयोगकर्ता B का मार्ग इतिहास अलग रहता है। व्यवहार में मल्टी-टेनेंट SaaS एजेंट इसी तरह काम करते हैं।

## मिश्रित टूल्स (Composite Tools) बनाना

कभी-कभी आपको कई MCP टूल्स को एक ही ऑपरेशन में संयोजित करने की आवश्यकता होती है। हो सकता है कि आप एक ऐसे मार्ग की योजना बनाना चाहते हों जो वास्तविक समय के ट्रैफ़िक और रास्ते के मौसम दोनों को ध्यान में रखे।

आप कस्टम टूल परिभाषाओं में MCP टूल्स को रैप (wrap) कर सकते हैं:

```typescript
export const smartRouteTool = createTool({
  id: 'smart-route-planner',
  description: 'Plans optimal route considering traffic and weather conditions',
  execute: async ({ context, mastra }) => {
    // रॉ टूल्स प्राप्त करें
    const tools = await mcpClient.getTools();
    
    // 1. गूगल मैप्स से बेस रूट प्राप्त करें
    const routeData = await tools.googleMaps_getDirections.execute({ 
      context: { 
        origin: context.origin,
        destination: context.destination 
      } 
    });
    
    // 2. मार्ग पर मौसम की जाँच करें
    const weatherData = await tools.weather_getForecast.execute({
      context: { coordinates: routeData.waypoints }
    });
    
    // 3. मौसम की चेतावनियों के साथ संवर्धित (enhanced) रूट वापस करें
    return { 
      ...routeData, 
      weatherAlerts: weatherData.alerts,
      recommendation: weatherData.severe ? 'यात्रा टालने पर विचार करें' : 'यात्रा करना सुरक्षित है'
    };
  },
});
```

यह आपको इस पर बारीक नियंत्रण देता है कि टूल्स वास्तव में कैसे इंटरैक्ट करते हैं, जबकि अभी भी भारी काम के लिए MCP प्रोटोकॉल का लाभ उठाया जाता है।

---

## यह हमें कहाँ ले जाता है

आपके एआई एजेंट को जिन भी सर्विसेस से बात करने की आवश्यकता है, उनके लिए कस्टम एपीआई क्लाइंट लिखना कभी भी टिकाऊ नहीं था। यह खराब तरीके से स्केल करता है, अक्सर टूटता है, और आपके प्लेटफ़ॉर्म को विशिष्ट कार्यान्वयन (implementations) से बांध देता है।

MCP हर एकीकरण चुनौती को हल नहीं करता है—प्रमाणीकरण (auth) अभी भी जटिल है, रेट लिमिटिंग अभी भी मायने रखती है, और हर सर्विस के पास अभी तक MCP सर्वर नहीं है। लेकिन यह एक ऐसी नींव स्थापित करता है जो एजेंट प्लेटफ़ॉर्म बनाना काफी आसान बना देती है।

यदि आप एक ऐसे एआई सिस्टम को आर्किटेक्ट कर रहे हैं जिसे बाहरी सर्विसेस के साथ इंटरैक्ट करने की आवश्यकता है, तो MCP को समझना शायद आपके समय के लायक है।
