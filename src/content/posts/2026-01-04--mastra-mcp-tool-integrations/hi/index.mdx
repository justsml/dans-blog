आपने एक AI एजेंट बनाया है। शायद यह एक अच्छा भी है। प्रॉम्प्ट्स सटीक हैं, मॉडल तेज़ है, और प्रतिक्रियाएँ स्वाभाविक लगती हैं।

लेकिन फिर कोई इसे Salesforce में किसी ग्राहक रिकॉर्ड को खोजने के लिए कहता है। या नवीनतम Jira टिकट्स पुल करने के लिए। या आपके आंतरिक दस्तावेज़ीकरण में खोजने के लिए।

और आपका सुंदर एजेंट बस... नहीं कर पाता।

यह वह एकीकरण समस्या है जिससे हर AI प्लेटफ़ॉर्म अंततः टकराता है। आपके एजेंट को हाथ चाहिए। इसे आपके वास्तविक व्यापार प्रणालियों में देखने की क्षमता चाहिए। उनके बिना, आप बस एक महंगा चैटबॉट चला रहे हैं।

पारंपरिक समाधान क्या है? हर उस सेवा के लिए एक कस्टम API रैपर लिखें जिसे आप जोड़ना चाहते हैं। उनके दस्तावेज़ पढ़ें, उनके प्रमाणीकरण को संभालें, उनकी दर सीमाओं से निपटें, प्रार्थना करें कि वे अगले महीने अपने एंडपॉइंट्स न बदलें। फिर अगली सेवा के लिए यह सब दोहराएं। और अगली के लिए।

मॉडल कॉन्टेक्स्ट प्रोटोकॉल इस गणना को पूरी तरह बदल देता है।

---

## MCP वास्तव में क्या हल करता है

USB-C से पहले USB के बारे में सोचें। आपके पास मिनी-USB, माइक्रो-USB, मालिकाना Apple कनेक्टर्स, और केबलों से भरा एक दराज़ा था जो केवल विशिष्ट उपकरणों के साथ काम करते थे। USB-C ने बस एक नया कनेक्टर नहीं जोड़ा—इसने एक मानक स्थापित किया जिसका मतलब था कि कोई भी केबल किसी भी उपकरण के साथ काम कर सकता है।

MCP AI टूल एकीकरणों के लिए वही काम कर रहा है।

अपने एजेंट को Salesforce, HubSpot, GitHub, या किसी अन्य सेवा से जोड़ने के लिए कस्टम कोड लिखने के बजाय, आप प्रोटोकॉल को एक बार लागू करते हैं (या एक पूर्व-निर्मित सर्वर डाउनलोड करते हैं), और कोई भी MCP-संगत एजेंट तुरंत इससे बात कर सकता है।

प्रोटोकॉल संचार परत को संभालता है। आप बस परिभाषित करते हैं कि आपके टूल क्या करते हैं और उन्हें किस डेटा की आवश्यकता है।

---

## कई एकीकरणों की स्थापना

Mastra में अपने [`MCPClient`](https://mastra.ai/docs/mcp/overview) के माध्यम से नेटिव MCP समर्थन है। आप स्थानीय टूल्स (चाइल्ड प्रोसेस के रूप में चलने वाले) और रिमोट सेवाओं (अपने स्वयं के बुनियादी ढांचे पर चलने वाले) दोनों को जोड़ सकते हैं।

यहाँ Google Maps को रूटिंग के लिए, एक मौसम सेवा के लिए, और स्थानीय Wikipedia खोज के लिए जोड़ने वाला एक वास्तविक उत्पादन सेटअप है:

```typescript
// src/mastra/mcp/index.ts
import { MCPClient } from '@mastra/mcp';

export const mcpClient = new MCPClient({
  servers: {
    // स्थानीय टूल (Stdio)
    wikipedia: {
      command: 'npx',
      args: ['-y', 'wikipedia-mcp'],
    },
    // मैप्स और नेविगेशन (रिमोट/HTTP)
    googleMaps: {
      url: new URL(process.env.GOOGLE_MAPS_MCP_URL!),
      requestInit: {
        headers: {
          Authorization: `Bearer ${process.env.GOOGLE_MAPS_API_KEY}`,
        },
      },
    },
    // मौसम सेवा एकीकरण
    weather: {
      url: new URL('https://mcp.weatherapi.dev/v1'),
      requestInit: {
        headers: {
          'X-API-Key': process.env.WEATHER_API_KEY!,
        },
      },
    },
  },
});
```

क्लाइंट कनेक्शन जीवनचक्र का प्रबंधन करता है, स्थानीय टूल्स के लिए प्रोसेस स्पॉनिंग को संभालता है, और रिमोट सर्वरों के लिए HTTP कनेक्शन बनाए रखता है। आप सीधे सॉकेट्स या stdio को नहीं छूते।

## टूल्स को एजेंट्स से जोड़ना

एक बार जब आपका MCP क्लाइंट कॉन्फ़िगर हो जाता है, तो उन टूल्स को एक एजेंट को देना सरल है:

```typescript
// src/mastra/agents/navigation-agent.ts
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { mcpClient } from '../mcp';

export const navigationDirectionsAgent = new Agent({
  id: 'navigation-directions-agent',
  name: 'Navigation & Directions Assistant',
  instructions: `You are a helpful navigation assistant that provides route planning and travel advice.
    - Always confirm the start and destination locations
    - Use Google Maps tools to find optimal routes
    - Check weather conditions along the route
    - Provide estimated travel times and suggest alternatives if weather is poor
    - Include relevant details like traffic, road conditions, and points of interest
    - Keep responses clear and actionable`,
  model: openai('gpt-5'),
  tools: await mcpClient.getTools(), // <--- यह वह जादुई लाइन है
});
```

जब कोई उपयोगकर्ता पूछता है: *"सैन फ्रांसिस्को से लेक ताहो का सबसे अच्छा रास्ता क्या है, और क्या मुझे मौसम की चिंता करनी चाहिए?"*

एजेंट उपलब्ध टूल परिभाषाओं को पढ़ता है, समझता है कि उसके पास Google Maps रूटिंग और मौसम पूर्वानुमान टूल्स तक पहुंच है, सही पैरामीटर्स के साथ उन्हें निष्पादित करता है, और एक इष्टतम मार्ग और वर्तमान मौसम की स्थितियों के साथ उत्तर देता है।

आपने Google Maps API कोड या मौसम सेवा एकीकरण की एक पंक्ति भी नहीं लिखी।

## प्रति-उपयोगकर्ता प्रमाणीकरण

यहाँ एक सुरक्षा गलती है जो करना आसान है: क्रेडेंशियल्स को हार्डकोड करना।

यदि आप अपने एनवायरनमेंट वेरिएबल्स में एक Google Maps API कुंजी डालते हैं और इसे इस तरह छोड़ देते हैं, तो हर उपयोगकर्ता एक ही कोटा और दर सीमाओं को साझा करता है। और भी महत्वपूर्ण, यदि आप ऐसी सेवाओं का उपयोग कर रहे हैं जो उपयोगकर्ता प्राथमिकताएँ संग्रहीत करती हैं (जैसे सहेजे गए स्थान या पसंदीदा मार्ग), तो सबको एक ही डेटा दिखाई देगा। यह डेमो के लिए ठीक काम करता है। उत्पादन में यह एक देयता है।

Mastra इसे उपयोगकर्ता-विशिष्ट क्रेडेंशियल्स के साथ गतिशील रूप से MCP क्लाइंट बनाने की अनुमति देकर संभालता है:

```typescript
async function handleUserRequest(userPrompt: string, userCredentials: UserCreds) {
  // इस विशिष्ट उपयोगकर्ता के लिए एक क्लाइंट बनाएं
  const userMcp = new MCPClient({
    servers: {
      googleMaps: {
        url: new URL(process.env.GOOGLE_MAPS_MCP_URL!),
        requestInit: {
          headers: {
            // उपयोगकर्ता की विशिष्ट API कुंजी या टोकन
            Authorization: `Bearer ${userCredentials.mapsApiKey}`,
            'X-User-ID': userCredentials.userId,
          },
        },
      },
    },
  });

  const agent = mastra.getAgent('navigationDirectionsAgent');
  
  // रनटाइम पर टूल्स इंजेक्ट करें
  const response = await agent.generate(userPrompt, {
    toolsets: await userMcp.getToolsets(),
  });

  return response;
}
```

प्रत्येक उपयोगकर्ता को अपने स्वयं के अलग टूलसेट मिलते हैं जिनमें उनकी अपनी API कोटा और प्राथमिकताएँ होती हैं। उपयोगकर्ता A के सहेजे गए स्थान निजी रहते हैं, उपयोगकर्ता B का मार्ग इतिहास अलग है। यही बहु-किरायी SaaS एजेंट्स व्यवहार में काम करते हैं।

## कंपोजिट टूल्स का निर्माण

कभी-कभी आपको कई MCP टूल्स को एक ही ऑपरेशन में संयोजित करने की आवश्यकता होती है। शायद आप एक ऐसा मार्ग योजना बनाना चाहते हैं जो वास्तविक समय के ट्रैफ़िक और मौसम की स्थितियों दोनों को ध्यान में रखता हो।

आप MCP टूल्स को कस्टम टूल परिभाषाओं में लपेट सकते हैं:

```typescript
export const smartRouteTool = createTool({
  id: 'smart-route-planner',
  description: 'Plans optimal route considering traffic and weather conditions',
  execute: async ({ context, mastra }) => {
    // मूल टूल्स प्राप्त करें
    const tools = await mcpClient.getTools();
    
    // 1. Google Maps से बेस मार्ग प्राप्त करें
    const routeData = await tools.googleMaps_getDirections.execute({ 
      context: { 
        origin: context.origin,
        destination: context.destination 
      } 
    });
    
    // 2. मार्ग के अनुसार मौसम की जाँच करें
    const weatherData = await tools.weather_getForecast.execute({
      context: { coordinates: routeData.waypoints }
    });
    
    // 3. मौसम चेतावनियों के साथ उन्नत मार्ग लौटाएं
    return { 
      ...routeData, 
      weatherAlerts: weatherData.alerts,
      recommendation: weatherData.severe ? 'यात्रा में देरी करने पर विचार करें' : 'यात्रा के लिए सुरक्षित'
    };
  },
});
```

इससे आपको यह सटीक नियंत्रण मिलता है कि टूल्स कैसे परस्पर क्रिया करते हैं, जबकि अभी भी भारी काम के लिए MCP प्रोटोकॉल का लाभ उठाया जाता है।

## यह कहाँ ले जाता है

हर सेवा के लिए कस्टम API क्लाइंट लिखना जिससे आपके AI एजेंट को बात करने की आवश्यकता है, कभी भी टिकाऊ नहीं था। यह खराब तरह से स्केल करता है, अक्सर टूटता है, और आपके प्लेटफ़ॉर्म को विशिष्ट कार्यान्वयनों से बाँधता है।

MCP हर एकीकरण चुनौती को हल नहीं करता—प्रमाणीकरण अभी भी जटिल है, दर सीमित करना अभी भी मायने रखता है, और हर सेवा का अभी तक MCP सर्वर नहीं है। लेकिन यह एक ऐसी नींव स्थापित करता है जो एजेंट प्लेटफ़ॉर्म बनाने को काफी कम दर्दनाक बनाता है।

यदि आप एक AI सिस्टम की आर्किटेक्चर कर रहे हैं जिसे बाहरी सेवाओं के साथ परस्पर क्रिया करने की आवश्यकता है, तो MCP को समझना संभवतः आपके समय के लायक है।
