Du hast einen KI-Agenten gebaut. Vielleicht sogar einen guten. Die Prompts sind präzise, das Modell ist schnell, und die Antworten fühlen sich natürlich an.

Aber dann fragt jemand, ob er Salesforce nach einem Kundendatensatz durchsuchen soll. Oder die neuesten Jira-Tickets abrufen. Oder deine internen Dokumentationen durchsuchen.

Und dein schöner Agent kann einfach... nichts.

Das ist das Integrationsproblem, das jede KI-Plattform irgendwann trifft. Dein Agent braucht Hände. Er braucht Einblick in deine tatsächlichen Geschäftssysteme. Ohne sie betreibst du nur einen teuren Chatbot.

Die traditionelle Lösung? Schreib einen benutzerdefinierten API-Wrapper für jeden einzelnen Dienst, den du verbinden möchtest. Lies ihre Docs, handhabe ihre Authentifizierung, deal mit ihren Rate-Limits, bete, dass sie nächsten Monat nicht ihre Endpoints ändern. Dann mach es wieder für den nächsten Dienst. Und den nächsten.

Das Model Context Protocol ändert diese Rechnung grundlegend.

---

## Was MCP tatsächlich löst

Denk an USB vor USB-C. Du hattest Mini-USB, Micro-USB, proprietäre Apple-Anschlüsse und eine Schublade voller Kabel, die nur mit spezifischen Geräten funktionierten. USB-C hat nicht nur einen neuen Anschluss hinzugefügt—es hat einen Standard etabliert, der bedeutete, dass jedes Kabel mit jedem Gerät funktionieren konnte.

MCP macht dasselbe für KI-Tool-Integrationen.

Anstatt benutzerdefinierten Code zu schreiben, um deinen Agenten mit Salesforce, HubSpot, GitHub oder einem anderen Dienst zu verbinden, implementierst du das Protokoll einmal (oder lädst einen vorgefertigten Server herunter), und jeder MCP-kompatible Agent kann sofort damit kommunizieren.

Das Protokoll übernimmt die Kommunikationsschicht. Du definierst nur, was deine Tools tun und welche Daten sie brauchen.

## Mehrere Integrationen einrichten

Mastra hat native MCP-Unterstützung durch seinen [`MCPClient`](https://mastra.ai/docs/mcp/overview). Du kannst sowohl lokale Tools (die als Child-Prozesse laufen) als auch Remote-Dienste (die auf ihrer eigenen Infrastruktur laufen) verbinden.

Hier ist ein realistisches Produktions-Setup, das Google Maps für Routing, einen Wetterdienst und eine lokale Wikipedia-Suche verbindet:

```typescript
// src/mastra/mcp/index.ts
import { MCPClient } from '@mastra/mcp';

export const mcpClient = new MCPClient({
  servers: {
    // Lokales Tool (Stdio)
    wikipedia: {
      command: 'npx',
      args: ['-y', 'wikipedia-mcp'],
    },
    // Maps & Navigation (Remote/HTTP)
    googleMaps: {
      url: new URL(process.env.GOOGLE_MAPS_MCP_URL!),
      requestInit: {
        headers: {
          Authorization: `Bearer ${process.env.GOOGLE_MAPS_API_KEY}`,
        },
      },
    },
    // Wetterdienst-Integration
    weather: {
      url: new URL('https://mcp.weatherapi.dev/v1'),
      requestInit: {
        headers: {
          'X-API-Key': process.env.WEATHER_API_KEY!,
        },
      },
    },
  },
});
```

Der Client verwaltet den Verbindungslebenszyklus, handhabt das Prozess-Spawning für lokale Tools und pflegt HTTP-Verbindungen zu Remote-Servern. Du musst dich nicht direkt um Sockets oder Stdio kümmern.

## Tools mit Agenten verbinden

Sobald du deinen MCP-Client konfiguriert hast, ist es straightforward, diese Tools einem Agenten zur Verfügung zu stellen:

```typescript
// src/mastra/agents/navigation-agent.ts
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { mcpClient } from '../mcp';

export const navigationDirectionsAgent = new Agent({
  id: 'navigation-directions-agent',
  name: 'Navigation & Directions Assistant',
  instructions: `Du bist ein hilfreicher Navigationsassistent, der Routenplanung und Reiseberatung bietet.
    - Bestätige immer Start- und Zielorte
    - Nutze Google Maps-Tools, um optimale Routen zu finden
    - Prüfe Wetterbedingungen entlang der Route
    - Biete geschätzte Reisezeiten und schlage Alternativen bei schlechtem Wetter vor
    - Berücksichtige relevante Details wie Verkehr, Straßenzustand und POIs
    - Halte Antworten klar und umsetzbar`,
  model: openai('gpt-5'),
  tools: await mcpClient.getTools(), // <--- Das ist die magische Zeile
});
```

Wenn ein Benutzer fragt: *"Was ist die beste Route von San Francisco nach Lake Tahoe, und sollte ich mir Sorgen um das Wetter machen?"*

Liest der Agent die verfügbaren Tool-Definitionen, erkennt, dass er Zugriff auf Google Maps-Routing und Wettervorhersage-Tools hat, führt sie mit den richtigen Parametern aus und antwortet mit einer optimalen Route plus aktuellen Wetterbedingungen entlang des Wegs.

Du hast keine einzige Zeile Google Maps-API-Code oder Wetterdienst-Integration geschrieben.

## Per-User-Authentifizierung

Es gibt einen Sicherheitsfehler, der hier leicht passieren kann: Credentials hardcoden.

Wenn du einen Google Maps-API-Key in deine Umgebungsvariablen packst und es dabei belässt, teilen sich alle Benutzer dasselbe Kontingent und dieselben Rate-Limits. Noch wichtiger: Wenn du Dienste nutzt, die Benutzerpräferenzen speichern (wie gespeicherte Orte oder favorisierte Routen), würde jeder dieselben Daten sehen. Das funktioniert für Demos. In Produktion ist es eine Haftung.

Mastra handhabt dies, indem es dir erlaubt, MCP-Clients dynamisch mit benutzerspezifischen Credentials zu erstellen:

```typescript
async function handleUserRequest(userPrompt: string, userCredentials: UserCreds) {
  // Erstelle einen Client für DIESEN spezifischen Benutzer
  const userMcp = new MCPClient({
    servers: {
      googleMaps: {
        url: new URL(process.env.GOOGLE_MAPS_MCP_URL!),
        requestInit: {
          headers: {
            // Benutzerspezifischer API-Key oder Token
            Authorization: `Bearer ${userCredentials.mapsApiKey}`,
            'X-User-ID': userCredentials.userId,
          },
        },
      },
    },
  });

  const agent = mastra.getAgent('navigationDirectionsAgent');

  // Tools zur Laufzeit injizieren
  const response = await agent.generate(userPrompt, {
    toolsets: await userMcp.getToolsets(),
  });

  return response;
}
```

Jeder Benutzer bekommt sein eigenes isoliertes Toolset mit eigenen API-Kontingenten und Präferenzen. Benutzer A's gespeicherte Orte bleiben privat, Benutzer B's Routenverlauf ist separat. So funktionieren Multi-Tenant-SaaS-Agenten in der Praxis.

## Composite Tools bauen

Manchmal musst du mehrere MCP-Tools zu einer einzigen Operation kombinieren. Vielleicht möchtest du eine Route planen, die sowohl Echtzeit-Verkehr als auch Wetterbedingungen entlang des Weges berücksichtigt.

Du kannst MCP-Tools in benutzerdefinierten Tool-Definitionen wrappen:

```typescript
export const smartRouteTool = createTool({
  id: 'smart-route-planner',
  description: 'Plant optimale Route unter Berücksichtigung von Verkehr und Wetterbedingungen',
  execute: async ({ context, mastra }) => {
    // Hole die rohen Tools
    const tools = await mcpClient.getTools();

    // 1. Basis-Route von Google Maps holen
    const routeData = await tools.googleMaps_getDirections.execute({
      context: {
        origin: context.origin,
        destination: context.destination
      }
    });

    // 2. Wetter entlang der Route prüfen
    const weatherData = await tools.weather_getForecast.execute({
      context: { coordinates: routeData.waypoints }
    });

    // 3. Erweiterte Route mit Wetterwarnungen zurückgeben
    return {
      ...routeData,
      weatherAlerts: weatherData.alerts,
      recommendation: weatherData.severe ? 'Reise verschieben' : 'Sicher zu reisen'
    };
  },
});
```

Dies gibt dir feinkörnige Kontrolle darüber, wie Tools interagieren, während du trotzdem das MCP-Protokoll für das schwere Heben nutzt.

## Wohin das führt

Benutzerdefinierte API-Clients für jeden Dienst zu schreiben, mit dem dein KI-Agent kommunizieren muss, war nie nachhaltig. Es skaliert schlecht, bricht oft, und bindet deine Plattform an spezifische Implementierungen.

MCP löst nicht jede Integrationsherausforderung—Authentifizierung ist immer noch komplex, Rate-Limiting ist immer noch wichtig, und nicht jeder Dienst hat bereits einen MCP-Server. Aber es etabliert eine Grundlage, die den Bau von Agent-Plattformen deutlich weniger schmerzhaft macht.

Wenn du ein KI-System architektonisch gestaltest, das mit externen Diensten interagieren muss, lohnt es sich wahrscheinlich, MCP zu verstehen.
