チームが AI 機能を追加するとき、最初に専用のベクターデータベースに手を伸ばすことがよくあります。

Pinecone、Weaviate、Qdrant、Chroma。新しいサービス、新しい依存関係、新しいコネクションプール、新しい請求先—そして今や整合させるべき真実のソースが二つになりました。

一方で、多くは既に PostgreSQL を既に使っています。PostgreSQL には `pgvector` があり、さらに 2008 年から優れたフルテキスト検索も組み込まれています。

専用のベクターストアは大規模で高クエリ量の下では価値を発揮します。しかし大半のアプリケーションは、本当に最初の検索システムを十分に負荷をかける前に二つ目のシステムに手を出してしまいます。そうして将来のスケーリング問題が今日の同期バグになります。

では：いつ FTS を使い、いつ pgvector を使い、いつ両方を使うべきでしょうか？

---

## それぞれが実際にやっていること

フルテキスト検索（`tsvector` / `GIN` インデックス）は語彙的（Lexical）です。テキストをトークン化してレキシームに整形し、クエリをインデックスと照合します。"running" と "runs" は同じレキシームにまとまり、"dog" と "dogs" も同様です。ランキング関数（`ts_rank`）は、クエリ語が文書内で頻出または目立つ場所に現れる文書を高く評価します。

`pgvector` は意味的（Semantic）です。チャンクの「意味」を埋め込みモデルが数値ベクトルで表し、密なベクトル（数の列）を格納します。類似度検索は高次元空間で近いベクトルを探します。"Dog" と "canine" は近くに来ます。スポーツとしての "running" とプロセスとしての "running" は離れるかもしれません。

実用的な差異：FTS は「どの文書がこれらの単語を含むか？」に答え、ベクトル検索は「どの文書が概ねこの意味を持つか？」に答えます。

![pg_trgm は短いあいまいな文字列向け、フルテキスト検索は正確な散文クエリ向け、pgvector は意味的マッチ向け、ハイブリッド検索は正確さと意味の両方が必要な長いコンテンツ向けを示す検索ツールの地図。](../search-tool-map.svg)

## フルテキスト検索が勝つとき

**正確に一致する語句を探しているとき。** 製品 SKU、エラーコード、型番、ユーザー名、法的条項の参照など。`SKU-AX-44192` は意味的に似たものは存在しません。合致するかしないかです。ベクター検索は自信を持って `SKU-AX-44193` を返すかもしれませんが、それは望ましくありません。

**クエリがキーワードベースのとき。** ユーザーが検索ボックスにタイプする、タグでフィルタする、ブログ投稿をキーワードで検索するような意図。FTS はそのような意図のために作られています。

**GPU や埋め込みインフラを使わずにランク付き結果が必要なとき。** FTS インデックスは高速で決定的、外部 API 呼び出しを必要としません。`tsvector` カラムを追加し、GIN インデックスを作れば完了です。

**検索と同時にブールなフィルタを行うとき。** `WHERE to_tsvector(body) @@ to_tsquery('postgres') AND category = 'tutorial' AND published_at > NOW() - INTERVAL '6 months'` — これは既存のクエリロジックと自然に合成できます。

```sql
-- Create the index
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body, '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Query
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`GENERATED ALWAYS AS` カラムでインデックスを自動更新できます。`setweight` はタイトルのマッチに本文より高い重みを与えます。それだけで基本セットアップは整います。

## トライグラムが勝つとき（pg_trgm）

Postgres にもう一つ、見落とされがちなツールがあります：`pg_trgm`。これはフルテキスト検索でもベクター検索でもありません。あいまいな文字列マッチング用で、両者が苦手とする中間の領域をカバーします。

**ユースケース：名前、住所、識別子、短い文字列の誤字・脱字の許容。**

FTS はテキストをレキシームにトークン化し語幹処理をします。散文には有効ですが、次の用途には不向きです：
- 人名（"Dan Levy" → ステミングで "dan levi" や言語設定によっては "leiv" になることがある）
- 会社名、住所、製品タイトルのように正確な綴りが重要なもの
- 誤字を含むクエリ — "Micheal Jordan", "Amaon", "javascipt"
- オートコンプリート / プレフィックス検索
- 部分文字列マッチ（"son" が "Johnson" や "Anderson" にマッチする等）

`pgvector` もここでは不適切です。"Micheal Jordan" を埋め込み、その近傍を探せば、埋め込み空間は意味で組織されているため、最も近い近傍が必ずしも綴りの近さを反映しません。近傍は "basketball legend" や "Michael B. Jordan" になり得ますが、ユーザーが探している誤字のあるレコードとは限りません。

`pg_trgm` は文字列を重なり合う3文字スライスに分割し、二つの文字列がどれだけ多くのトライグラムを共有しているかを測ります。"Dan" -> `" da"`, `"dan"`, `"an "`。"Micheal" と "Michael" は大部分のトライグラムを共有するため、類似度は高くなります。

```sql
-- Enable the extension (usually already available)
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- GIN index on names column — enables fast trigram similarity search
CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- Fuzzy name search: finds "Micheal Jordan" when searching "Michael Jordan"
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % operator = similarity threshold (default 0.3)
ORDER BY score DESC
LIMIT 10;

-- Or use ILIKE with trigram index support for contains matching
SELECT id, name
FROM users
WHERE name ILIKE '%johnson%'   -- GIN index makes this fast
LIMIT 10;
```

`%` 演算子は `pg_trgm.similarity_threshold`（デフォルト 0.3、範囲 0-1）を使います。閾値を高くするとより近い一致が必要になります。名前検索では 0.3-0.4 が誤字を拾いつつノイズを抑えるには大抵適切です。

**トライグラムはプレフィックス検索やオートコンプリートにも有効で、誤字許容や部分一致を必要とする場合に特に役立ちます：**

```sql
-- Autocomplete: prefix matching. For pure left-anchored prefixes,
-- compare trigram GIN against a B-tree pattern index on your data.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- More control: word_similarity for partial matches within longer strings
-- (useful when searching "Johnson" within "Andrew Johnson III")
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name          -- <% operator = word_similarity threshold
ORDER BY score DESC
LIMIT 10;
```

**`pg_trgm` を FTS より使うべき場合：**

| Scenario | Use |
|---|---|
| 人名・会社名検索で誤字がある場合 | `pg_trgm` |
| オートコンプリート / プレフィックス検索 | `pg_trgm`（またはプレフィックス検索を伴う FTS） |
| 短い文字列、コード、識別子の検索 | `pg_trgm` |
| 散文の記事やドキュメントの検索 | FTS |
| ログメッセージ内のキーワード検索 | FTS |
| 多言語の名前検索 | `pg_trgm`（言語非依存） |

`pg_trgm` は FTS とも併用できます。あいまいな事前フィルタにトライグラムを使い、`ts_rank` でランク付けするか、あるいはトライグラム類似度とベクトルスコアを組み合わせてください。

## pgvector が勝つとき

**RAG（Retrieval-Augmented Generation）を構築している場合。** RAG は意味的な検索を前提とします：文書チャンクの「意味」がユーザーの質問に近いものを見つける。表現が異なっていても該当箇所を見つけられるのはベクター検索の得意分野です。FTS は言い換えや同義語、概念的な一致を見逃します。

**ユーザーが欲しいものを言葉で説明するとき。** "Something light for a summer evening"（夏の夜に合う軽めのもの）のように、明確なワードが無い場合。"新任マネージャーとしての自信を築く方法に関する記事" のように、FTS では意味的にマッチさせにくい要望。

**類似アイテムを見つける場合。** 関連製品、類似のサポートチケット、重複バグレポートの検索。"この問題に似た issue を探して" はベクトル操作です。新しい issue を埋め込み、その近傍を探します。

**多言語コンテンツ。** 多言語で学習された埋め込みは言語を越えたマッチングが可能です。FTS は言語ごとの設定が必要で、クロスランゲージのクエリは苦手です。

```sql
-- Setup
CREATE EXTENSION IF NOT EXISTS vector;

ALTER TABLE documents ADD COLUMN embedding vector(1536);
CREATE INDEX documents_embedding_idx
  ON documents USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);

-- Query: semantic search
SELECT id, title, 1 - (embedding <=> $1::vector) AS similarity
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 10;
```

注意：`ivfflat` は近似インデックスです — 高速ですがリコール（取りこぼし）を一部犠牲にします。小規模データセット（およそ 1M 行未満）では `hnsw` の方が良いことが多いです：

```sql
CREATE INDEX documents_embedding_idx
  ON documents USING hnsw (embedding vector_cosine_ops);
```

## 両方必要なとき

技術文書は、単純な分岐が破綻する代表例です。ユーザーは "how to configure timeouts" のように概念的に検索する一方で、関数名 `withRetry()` やエラーコード `ECONNRESET` のように正確な語句も探します。

ベクター検索は概念的クエリを、FTS は正確語句を扱います。どちらも単独では両方をうまく扱えません。

解はハイブリッド検索：両方実行して結果を融合することです。

**Reciprocal Rank Fusion（RRF）** はここでの標準アルゴリズムです。2 つのシステムのスコアを正規化する必要がなく、ランク位置を組み合わせます。

```sql
-- Hybrid search with Reciprocal Rank Fusion
WITH fts_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM documents, to_tsquery('english', $1) query
  WHERE search_vector @@ query
  LIMIT 50
),
vector_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> $2::vector) AS rank
  FROM documents
  ORDER BY embedding <=> $2::vector
  LIMIT 50
),
rrf AS (
  SELECT
    COALESCE(f.id, v.id) AS id,
    COALESCE(1.0 / (60 + f.rank), 0) + COALESCE(1.0 / (60 + v.rank), 0) AS rrf_score
  FROM fts_results f
  FULL OUTER JOIN vector_results v ON f.id = v.id
)
SELECT d.id, d.title, rrf.rrf_score
FROM rrf
JOIN documents d ON d.id = rrf.id
ORDER BY rrf_score DESC
LIMIT 10;
```

分母の `60` は RRF の定数です — 値を大きくするとランク差の影響が小さくなり、小さくすると影響が大きくなります。多くの場合デフォルトの 60 が良い働きをします。

これにより一つのクエリで二つの検索を実行し、ランクを融合してキーワード信号と意味信号が合致する結果を報いることができます。

![1 つのクエリがフルテキスト検索と pgvector に分岐し、それぞれがランク付き結果を出し、Reciprocal Rank Fusion が二つのリストを結合するハイブリッド検索パイプライン。](../hybrid-rrf-pipeline.svg)

_RRF が有用なのは、`ts_rank` とコサイン距離の生スコアを比較可能だと見做さない点にあります。単に「各リストでどれだけ上位に出たか」を利用するだけです。_

## 実務上の意思決定ツリー

検索戦略を選ぶときは、まず**入力テキストの形**を見て、次に**ユーザーがどんなクエリをしているか**を問います。"誤字のある短い文字列" は "正確な語句が重要な長い散文" とは異なり、両者は "文書チャンクに関する質問" とはまた別の問題です。

![pg_trgm、フルテキスト検索、pgvector、ハイブリッド検索、または専用ベクターデータベースをテキストの形とユーザーのクエリ方法に応じて選ぶ決定ツリー。](../search-decision-tree.svg)

言葉でまとめると：

- **人名、住所、タイトル、オートコンプリート、誤字の多い短い文字列** → `pg_trgm`
- **既知の単語、エラーコード、SKU、関数名、タグ、カテゴリ、フィルタ** → FTS
- **質問、言い換え、推薦、関連アイテム、多言語マッチング、RAG チャンク** → pgvector
- **技術コンテンツで正確なシンボルと概念的な答えの両方が必要なとき** → RRF を使ったハイブリッド
- **主キー、正確な ID、権限制御、日付範囲、ソート付きリスト** → 普通の SQL インデックス
- **ベクター量が非常に大きく、高並列、または Postgres がベンチマークで達成できないレイテンシ目標があるとき** → 専用ベクターストアを検討

### FTS vs. Semantic（短い版）

"FTS を使うべきか、ベクター検索を使うべきか？" という質問は大抵こう縮約できます：**関連文書にどんな単語が出るか分かっていますか？**

もし分かっているなら — ユーザーは既知の語を検索する、カテゴリや関数名、製品コードなど — FTS は速く、安く、予測可能です。なぜその結果がマッチしたか（理由）も説明できます。

分からないなら — ユーザーが概念を説明する、質問する、別言語で検索する — ベクトル検索が適切です。意味にマッチし、語にマッチしません。

難しい中間のケースは、技術コンテンツに対する自然言語クエリです。例えば "how do I handle connection drops"（接続が切断されるときどう扱うか）という検索は、"implementing retry logic for network failures"（ネットワーク障害のための再試行ロジック実装）というタイトルの記事を必要とするかもしれません — 重複する単語が無くても意味的に高い関連性があります。ここでベクター検索は価値を発揮します。

もう一つの難しいケースは **固有名詞（人名）** です。FTS もベクトル検索も得意ではありません：
- FTS は "Micheal" を "Michael" で見逃すことがある — トークンが異なるため
- ベクター検索はトレーニングデータにその名前が頻出しないと見逃す
- `pg_trgm` は正しく扱える：綴りの類似度でマッチさせる

実際には、多くのコンテンツ重視の検索ボックスは速度とキーワードのために FTS を使い、必要に応じて `pg_trgm` やハイブリッドを組み合わせ、真の意味検索機能が必要なら pgvector を使います。RAG は常に pgvector を意味します。

## 専用ベクターストアが本当に必要なとき

システムが本当に pgvector を超えるとき、市場は雑多に見えます。選ぶときに重要なのは次の点です。

### 機能マトリクス

テーブルのいくつかの列は事前に説明が必要です。

**ハイブリッド検索** は BM25 キーワード検索とベクター類似度を一つのクエリで実行し、RRF 等で結合することを指します。`withRetry timeout` が関数名に正確にマッチしつつ、"retry logic for network failures" のような概念的な記事も返すことができます。ハイブリッドが無ければ、片方を選ぶか自分で二つのクエリを融合する必要があります。pgvector の "Manual (RRF via SQL)" は上に示したアプローチです：動きますが自分で書く必要があります。

**スパースベクトル** は BM25 を超えます。SPLADE のようなスパースベクトルは約 30,000 次元（語彙ごとに 1）、そのうち ~98% がゼロで、非ゼロ位置がどの語が重要かを示します。クエリ "dogs" は "canine" や "pet" を重み付けして BM25 レベルのキーワード精度をベクトル内で提供できます。この列が False なら、正確語句検索のために外部の FTS 層が必要です。

```python
# SPLADE: ~30,000 dims total, ~60 non-zero — only the relevant vocabulary positions fire
def encode_splade(text: str) -> dict:
    tokens = tokenizer(text, return_tensors="pt", truncation=True, max_length=512)
    with torch.no_grad():
        output = model(**tokens)
    vec = torch.log1p(torch.relu(output.logits)).max(dim=1).values.squeeze()
    return {"indices": vec.nonzero().squeeze().tolist(), "values": vec[vec != 0].tolist()}
```

**SQL / SQL-like** はフィルタリングに関する話です。フィルタなしのベクトル検索はデモに過ぎません：テナントスコープ、日付範囲、権限、カテゴリはまだ必要です。フル SQL（pgvector）は既存の結合の横でそれを表現できます。専用 DB は JSON フィルタ、クエリ DSL、GraphQL などを使います。機能はありますが、フィルタロジックが絡み合うほど SQL が魅力的になります。

```sql
-- pgvector: vector similarity is just another expression in WHERE
SELECT id, title, 1 - (embedding <=> $1) AS score
FROM documents
WHERE tenant_id = $2 AND category = ANY($3::text[]) AND created_at > NOW() - INTERVAL '90 days'
ORDER BY embedding <=> $1 LIMIT 10;
```

**マルチモーダルネイティブ** は "単に画像の埋め込みを保存できる" という意味ではありません。生の画像 URL を与えるとベクトル化まで内蔵で処理してくれる、という意味です。多くのデータベースはベクトルを格納できますが、埋め込みモデルを提供するかどうかが違いです。Marqo や Weaviate（CLIP/ImageBind モジュール経由）はそのループを閉じます。

```python
# Marqo: POST raw images, query with text — no external embedding step needed
mq.index("products").add_documents(
    [{"id": "shoe-001", "image": "https://cdn.example.com/shoes/001.jpg"}],
    tensor_fields=["image"]
)
results = mq.index("products").search(q="lightweight shoes for summer")
# Returns shoe-001 despite zero keyword overlap — CLIP handles the cross-modal match
```

**ディスクベースのインデックス** はコスト面のレバーです。RAM 駐在の HNSW は高速ですが、1536 次元ベクトル 1M 個あたり生ベクトル、グラフオーバーヘッド、メタデータで数 GB の RAM を要求することがあります。ディスクベースの代替（Milvus DiskANN、Elasticsearch DiskBBQ、LanceDB の Lance 形式、Turbopuffer のオブジェクトストレージ層）は遅延を一部犠牲にしますがインフラ費用を下げます。RAG ワークロードではモデルレイテンシが支配的なことも多く、このトレードオフはベンチマークに値します。Redis VSS は RAM 専用でディスク路線が無いことに注意してください。

**最大次元数** は将来の移行コストです。`text-embedding-3-large` は 3072 次元、Jina v3 はさらに大きな埋め込みを出すことがあり、研究モデルは次々と次元数を押し上げます。一部のマネージドサービスはハード上限を公開しています。決める前にドキュメントを確認してください。余裕のある選択を：次元上限に達してインデックスを移行するのは非常に面倒です。

### マトリクスの読み方

いくつかは表にうまく収まりません。

**スパースベクトル** はベクトルインデックス内で BM25 相当のキーワードマッチを得る方法です。Qdrant や Elasticsearch は成熟した実装があります。Weaviate は BM25F 経由でサポートします。ハイブリッド検索が重要で二つのシステムを運用できない場合、スパースベクトルをサポートするオプションを探してください。

**ディスクベースのインデックス** はコストの調整手段です。RAM 駐在の HNSW は高速ですが、ベクトル数や次元、メタデータ、グラフオーバーヘッドにより高価になります。ディスクベースの代替は遅延を許容する代わりにインフラ費用を下げます。大規模 RAG インデックスではこのトレードオフを試す価値があります。

**Turbopuffer のマルチテナンシー** は非常に多いネームスペース数を前提に設計されています。ユーザーや組織ごとに分離された検索が必要なワークロードでは経済性が変わりますが、自分のテナント形状で必ずベンチマークしてください。

**LanceDB の埋め込みモード** は "ベクトル検索の SQLite" に最も近いものです。プロセス内で動作し、サーバー不要で Lambda や Cloudflare Workers、エッジ環境で実用的に動きます。Lance の列指向フォーマットが埋め込み運用を現実的にします。

**Orama は検索 UX インフラ** であり、倉庫ではありません。JavaScript アプリやエッジ、管理されたサイト／アプリ検索レイヤー向けに優れています。10 億ベクトルや重い解析、複雑なフィルタ結合には適しません。

**Vespa は検索がプロダクトの半分であるときに選ぶもの** です。字句検索、近傍検索、テンソル、ランキング式、グルーピング、オンライン配信を組み合わせます。そのパワーは実用的ですが運用とモデリングの複雑さも相当です。検索／推薦チーム向けであり、単に CRUD アプリに意味検索を追加する用途とは異なります。

**ClickHouse は分析に結びつく検索の話題に入るべきです。** ソースがイベント、ログ、トレース、メトリクス、または大きなファクトテーブルであれば、ClickHouse はベクトル距離、フィルタ、集計、そしてまともなフルテキストインデックスを一つの SQL エンジンで保持できます。目的別のベクターデータベースではありませんが、分析向けの検索では時に最も堅実な回答になります。

**Chroma は開発・テストと小規模アプリに最適** です。大規模コーパス、HA、ディスク運用、ファーストクラスのハイブリッド検索を狙うなら、本番向けストアを評価してからプロトタイプをインフラに昇格させてください。

### 単純化した判断

本当に pgvector を超えたなら — 通常はベクトル数、フィルタ、書き込み率、または高並列時のレイテンシが Postgres の限界を超えたベンチマークで示されるとき — 制約に応じて選びます：

- **SaaS 製品でテナント分離が必須** → Turbopuffer
- **Rust 水準の性能と複雑なメタデータフィルタが必要** → Qdrant
- **既に Elastic/ELK スタックを使っている** → Elasticsearch with DiskBBQ
- **AWS 環境でオープンソースを好む** → OpenSearch
- **ランキング重視の検索／推薦プラットフォーム** → Vespa
- **分析・可観測性・ログが主目的** → ClickHouse
- **オンプレでの数十億規模** → Milvus
- **エッジ／サーバーレス／マルチモーダル** → LanceDB
- **小さな JS アプリやドキュメントサイト、エッジネイティブ検索 UX** → Orama
- **運用なしで動けば良く、コストが二次的** → Pinecone
- **マルチモーダル優先（画像・映像・音声）** → Marqo
- **既に MongoDB を使っている** → Atlas Vector Search
- **既に Postgres を使っていて余白が必要** → Supabase Vector や Neon（いずれも pgvector 管理版）

## やってはいけないこと

特定の答えが正しいものに対して、ベクター検索をあいまい検索として使わないでください。

"メール `dan@example.com` のユーザーを探して" はベクター検索の問題ではありません。`ORD-12345` のような注文 ID を探すのも同様です。`ORD-12345` を埋め込んで注文テーブルにコサイン類似度で照合すると *何か* は返りますが、それが正しいとは限りません。これらは厳密一致の問題です。主キーや通常のインデックスを使ってください。

ベクター検索はデータセット内の *最も類似したもの* を返します。関連が全くないときでも何かを返してしまうことがあります。関連文書の検索ではそれで構いませんが、特定レコードの検索で誤った近傍を返すのは致命的です。

各ツールが何に使われるのかを理解してください。多くは既にあなたの Postgres インストールの中にあります。適材適所で使いましょう。
