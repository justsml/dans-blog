AI機能を既存のチームに追加する際、まず最初に専用のベクトルデータベースを検討することがよくあります。

Pinecone、Weaviate、Qdrant、Chroma。新しいサービス、新しい依存関係、新しいコネクションプール、新しい請求。そして今度は、整合性を保つべき2つの信頼できる情報源（Sources of Truth）を抱えることになります。

しかし、すでにPostgreSQLがあるはずです。PostgreSQLにはすでに`pgvector`があります。そして、2008年から組み込まれている優れた全文検索機能もあります。

専用のベクトルストアは、大規模かつ高クエリボリュームにおいてその価値を発揮します。しかし、ほとんどのアプリケーションは、最初の検索システムを十分に使い切る前に、2つ目の検索システムに手を出してしまいます。こうして、将来のスケーリングの問題が、今日の同期バグへと変わっていくのです。

では、いつFTS（全文検索）を使い、いつpgvectorを使い、いつその両方を使うべきでしょうか？

---

## それぞれが実際に何をしているのか

全文検索（`tsvector` / `GIN` インデックス）は**語彙的（lexical）**です。テキストをレクセム（語彙素）にトークン化し、語幹抽出し、クエリをインデックスと照合します。「Running」と「runs」は同じレクセムに集約されます。「dog」と「dogs」も同様です。ランキング関数（`ts_rank`）は、クエリの用語が頻繁に、あるいは目立つ場所に現れるドキュメントを高く評価します。

pgvectorは**意味的（semantic）**です。埋め込みモデルによって理解されたチャンクの「意味」を表す、数値のリストである密ベクトルを保存します。類似性検索は、その高次元空間内で近くにあるベクトルを見つけます。「犬」と「イヌ科」は互いに近くに配置されます。スポーツの「Running」と、プロセスの「running」は近くならないかもしれません。

実用的な違い：FTSは「どのドキュメントにこれらの単語が含まれているか？」に答えます。ベクトル検索は「どのドキュメントが大体このようなことを意味しているか？」に答えます。

![短い曖昧な文字列にはpg_trgm、正確な散文クエリには全文検索、意味的なマッチングにはpgvector、そして正確なシグナルと意味的なシグナルの両方を必要とする長いコンテンツにはハイブリッド検索を示す検索ツールマップ。](../search-tool-map.svg)

_最初の分岐点は「古い検索 vs. AI検索」ではありません。テキストの形状と、どのような回答が正解となるかです。_

## 全文検索が勝るケース

**正確さが重要な用語を検索する場合。** 製品SKU、エラーコード、モデル番号、ユーザー名、法的な条項の参照など。`SKU-AX-44192`は意味的に何かに似ているわけではありません。一致するかしないか、それだけです。ベクトル検索は自信満々に`SKU-AX-44193`を返すかもしれませんが、それは望んでいる結果ではありません。

**クエリがキーワードベースの場合。** ユーザーが検索ボックスに入力したり、タグでフィルタリングしたり、ブログ記事をキーワードで検索したりする場合です。FTSは、そのような意図の形状に合わせて構築されています。

**GPUや埋め込みインフラなしでランク付けされた結果が必要な場合。** FTSインデックスは高速で決定論的であり、外部API呼び出しを必要としません。`tsvector`列を追加し、GINインデックスを作成すれば完了です。

**検索と並行してBooleanフィルタリングを行う場合。** `WHERE to_tsvector(body) @@ to_tsquery('postgres') AND category = 'tutorial' AND published_at > NOW() - INTERVAL '6 months'` — これは既存のクエリロジックと自然に合成できます。

```sql
-- インデックスの作成
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body, '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- クエリ
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`GENERATED ALWAYS AS`列により、インデックスは自動的に最新の状態に保たれます。`setweight`は、本文のヒットよりもタイトルのヒットに高いランクを与えます。設定はこれだけです。

## Trigramsが勝るケース (pg_trgm)

見落とされがちな3つ目のPostgresツール、`pg_trgm`があります。これは全文検索でもベクトル検索でもありません。**曖昧な文字列マッチング**であり、両方のツールが苦手とする絶妙な中間領域をカバーします。

**ユースケース：名前、住所、識別子、およびタイプミスのある短い文字列。**

FTSはテキストをレクセムにトークン化し、語幹抽出します。これは散文には適していますが、以下のようなものには不向きです。
- 人名（「Dan Levy」 → 言語設定によりますが「dan levi」や「leiv」などに語幹抽出される）
- 会社名、住所、正確な綴りが重要な製品名
- タイプミスを含むクエリ — 「Micheal Jordan」、「Amaon」、「javascipt」
- オートコンプリート / 前方一致検索
- 部分文字列一致（「son」が「Johnson」や「Anderson」に一致するなど）

pgvectorもここでは不適切な選択です。「Micheal Jordan」を埋め込んで最も近いベクトルを見つけることはできますが、埋め込み空間は名前を綴りではなく意味で整理します。最も近い隣人は「バスケットボールの伝説」や「Michael B. Jordan」になり、タイプミスのあるユーザーレコードではないかもしれません。

`pg_trgm`は文字列を連続する3文字の断片（トライグラム）に分解し、2つの文字列がどれだけのトライグラムを共有しているかを測定します。「Dan」 -> `" da"`, `"dan"`, `"an "`。「Micheal」と「Michael」はトライグラムの大部分を共有しているため、類似度は高くなります。

```sql
-- 拡張機能を有効にする（通常はすでに利用可能）
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- 名前カラムにGINインデックスを作成 — 高速なトライグラム類似度検索を可能にする
CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- 曖昧な名前検索：「Michael Jordan」で検索したときに「Micheal Jordan」を見つける
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % 演算子 = 類似度の閾値（デフォルトは0.3）
ORDER BY score DESC
LIMIT 10;

-- あるいは、トライグラムインデックスをサポートする ILIKE を使用して「含む」マッチングを行う
SELECT id, name
FROM users
WHERE name ILIKE '%johnson%'   -- GINインデックスにより高速化
LIMIT 10;
```

`%`演算子は`pg_trgm.similarity_threshold`（デフォルト0.3、範囲0〜1）を使用します。値が高いほど、より近い一致が必要になります。名前検索の場合、通常は0.3〜0.4が適切です。タイプミスを拾うのに十分寛容でありながら、ノイズを避けるのに十分厳密です。

**トライグラムは、特にオートコンプリートにタイプミス耐性や「含む」マッチングが必要な場合の前方一致検索やオートコンプリートにも役立ちます。**

```sql
-- オートコンプリート：前方一致。純粋な左固定のプレフィックスの場合、
-- トライグラムGINをデータのB-treeパターンインデックスと比較します。
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- より細かな制御：長い文字列内の部分一致のための word_similarity
-- （「Andrew Johnson III」の中から「Johnson」を検索する場合に便利）
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name          -- <% 演算子 = word_similarity の閾値
ORDER BY score DESC
LIMIT 10;
```

**FTSではなく `pg_trgm` を選ぶべき場面：**

| シナリオ | 使用ツール |
|---|---|
| タイプミスを含む人名/会社名検索 | `pg_trgm` |
| オートコンプリート / 前方一致検索 | `pg_trgm`（または前方一致クエリ付きのFTS） |
| 短い文字列、コード、識別子の検索 | `pg_trgm` |
| 散文記事、ドキュメントの検索 | FTS |
| ログメッセージのキーワード検索 | FTS |
| 多言語の名前検索 | `pg_trgm`（言語に依存しません） |

`pg_trgm`はFTSと組み合わせることも可能です。トライグラムを曖昧な事前フィルタリングに使用して`ts_rank`でランク付けしたり、トライグラムの類似度をベクトルのスコアと組み合わせたりできます。

## pgvectorが勝るケース

**RAG（検索拡張生成）を構築している場合。** RAGは意味的な検索に依存しています。言い回しが異なっていても、ユーザーの質問に意味が最も近いドキュメントの「チャンク」を見つける必要があります。ベクトル検索はこのために作られています。FTSは、言い換えや類義語、概念的な一致を見逃してしまいます。

**ユーザーが検索キーワードではなく、欲しいものを説明する場合。** 「夏の夜に合う軽いもの」には、明らかなワインのキーワードが含まれていません。「新任マネージャーとして自信をつけるための記事」には、FTSでは提供できない意味的な理解が必要です。

**類似したアイテムを見つける場合。** 関連製品、類似したサポートチケット、重複するバグ報告など。「これと似たイシューを見つけて」はベクトル操作です。新しいイシューを埋め込み、その最近傍を見つけます。

**多言語コンテンツ。** 多言語データで訓練された埋め込みモデルは、言語を跨いでマッチングできます。FTSは言語固有の設定が必要で、言語横断的なクエリをうまく扱えません。

```sql
-- セットアップ
CREATE EXTENSION IF NOT EXISTS vector;

ALTER TABLE documents ADD COLUMN embedding vector(1536);
CREATE INDEX documents_embedding_idx
  ON documents USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);

-- クエリ：意味検索
SELECT id, title, 1 - (embedding <=> $1::vector) AS similarity
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 10;
```

注：`ivfflat`は近似値です。高速ですが、速度のために再現率（recall）をいくらか犠牲にします。小規模なデータセット（約100万行未満）の場合、通常は`hnsw`の方が優れています。

```sql
CREATE INDEX documents_embedding_idx
  ON documents USING hnsw (embedding vector_cosine_ops);
```

## 両方が必要なケース

技術ドキュメントは、単純な切り分けが通用しない場所です。ユーザーは「タイムアウトの設定方法」を検索しますが、`withRetry()`のような関数名や`ECONNRESET`のようなエラーコードも検索します。

ベクトル検索は概念的なクエリを扱います。FTSは正確な用語を扱います。どちらも単独では両方をうまく処理できません。

解決策は**ハイブリッド検索**です。両方を実行して結果を融合させます。

**Reciprocal Rank Fusion (RRF)** は、ここでの標準的なアルゴリズムです。2つのシステムからのスコアを正規化する必要はなく、ランクの位置を組み合わせます。

```sql
-- Reciprocal Rank Fusion を用いたハイブリッド検索
WITH fts_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM documents, to_tsquery('english', $1) query
  WHERE search_vector @@ query
  LIMIT 50
),
vector_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> $2::vector) AS rank
  FROM documents
  ORDER BY embedding <=> $2::vector
  LIMIT 50
),
rrf AS (
  SELECT
    COALESCE(f.id, v.id) AS id,
    COALESCE(1.0 / (60 + f.rank), 0) + COALESCE(1.0 / (60 + v.rank), 0) AS rrf_score
  FROM fts_results f
  FULL OUTER JOIN vector_results v ON f.id = v.id
)
SELECT d.id, d.title, rrf.rrf_score
FROM rrf
JOIN documents d ON d.id = rrf.id
ORDER BY rrf_score DESC
LIMIT 10;
```

分母の `60` は RRF 定数です。値が高いほどランク差の影響が小さくなり、値が低いほど影響が大きくなります。ほとんどの場合、デフォルトの 60 がうまく機能します。

これにより、1つのクエリで2つの検索が実行され、ランクが融合され、キーワードと意味のシグナルが一致する結果が高く評価されます。

![1つのクエリが全文検索とpgvectorに分岐し、それぞれがランク付けされた結果を生成し、Reciprocal Rank Fusionがその2つのリストを組み合わせるハイブリッド検索パイプライン。](../hybrid-rrf-pipeline.svg)

_RRFが価値を発揮するのは、`ts_rank`とコサイン距離が比較可能な生のスコアであるかのように装う必要がないからです。「それぞれのリストでどの程度の順位に現れたか？」だけを問いかけます。_

## 実践的な決定ツリー

検索戦略を選ぶ際は、まず**入力の形状**から始め、次に**ユーザーがどのようなクエリを行っているか**を考えます。「綴りにバリエーションがある短い文字列」は「正確な用語が重要な長い散文」とは異なる問題であり、どちらも「ドキュメントチャンクに対する質問」とは異なります。

![テキストの形状とユーザーのクエリ方法に基づいて、pg_trgm、全文検索、pgvector、ハイブリッド検索、または専用のベクトルデータベースを選択する決定ツリー。](../search-decision-tree.svg)

同じツリーを言葉で説明すると：

- **名前、住所、タイトル、オートコンプリート、またはタイプミスの多い短い文字列** → `pg_trgm`
- **既知の単語、エラーコード、SKU、関数名、タグ、カテゴリ、フィルタ** → FTS
- **質問、言い換え、レコメンデーション、関連アイテム、多言語マッチング、RAGチャンク** → pgvector
- **正確なシンボルと概念的な回答の両方をユーザーが必要とする技術的コンテンツ** → RRFを用いたハイブリッド
- **主キー、正確なID、権限フィルタ、日付範囲、ソート済みリスト** → 通常のSQLインデックス
- **膨大なベクトル量、非常に高い同時実行性、またはPostgresではベンチマークで達成できないレイテンシターゲット** → 専用のベクトルストアを検討

### FTS vs. 意味検索：短縮版

「FTSとベクトル検索のどちらを使うべきか？」という問いは、通常これに集約されます。**関連するドキュメントにどのような単語が含まれているか、あらかじめ分かっていますか？**

**はい**の場合 — ユーザーが既知の用語、カテゴリ、関数名、製品コードを検索する場合 — FTSの方が高速で安価、かつ予測可能です。なぜ結果が一致したのかを説明できます。

**いいえ**の場合 — ユーザーが概念を説明したり、質問をしたり、別の言語で検索したりする場合 — ベクトル検索が適したツールです。単語ではなく意味を照合します。

トリッキーな中間領域は、技術的なコンテンツに対する自然言語クエリです。「接続の切断をどう処理するか」を検索している人は、「ネットワーク障害に対するリトライロジックの実装」というタイトルの記事を必要としているかもしれません。単語の重複はありませんが、意味的な関連性は高いです。ここにベクトル検索の使い道があります。

もう一つのトリッキーなケースは、**名前や固有名詞**です。FTSもベクトル検索もこれらは苦手です。
- FTSは「Michael」を検索したときに「Micheal」を見逃します（トークンが異なるため）。
- ベクトル検索は、訓練データに頻繁に登場しない名前の場合、完全に見逃すことがあります。
- `pg_trgm` はこれを正しく処理します。意味的や語彙的ではなく、正字法的な（綴りの）類似性に基づいているからです。

実際には、コンテンツ重視の検索ボックスの多くは、速度とキーワードのためにFTSを必要とし、ユーザーが名前を検索するかどうかに応じてハイブリッドや`pg_trgm`を必要とする場合があります。真の意味検索機能は通常、pgvectorを意味します。RAGは常にpgvectorを意味します。

## もし専用のベクトルストアが必要になったら

一部のシステムでは、本当にpgvectorでは不十分になることがあります。そうなったとき、市場は非常に騒がしくなります。主要な選択肢において重要なポイントは以下の通りです。

### 機能マトリックス

テーブルの内容を理解するために、いくつかの列を説明する必要があります。

**ハイブリッド検索**とは、BM25キーワード検索とベクトル類似度検索を1つのクエリで実行し、Reciprocal Rank Fusionで統合することを指します。「withRetry timeout」という検索で、関数名そのものに一致させつつ、意味的に「ネットワーク障害のリトライロジック」に関するドキュメントにも一致させることができます。ハイブリッド機能がない場合は、どちらか一方の検索モードを選ぶか、自分で2つのクエリを統合する必要があります。pgvectorの「手動（SQLによるRRF）」は、[上記で示したアプローチ](#両方が必要なケース)です。動作はしますが、自分でコードを書く必要があります。

**スパースベクトル（Sparse vectors）**は、BM25よりもさらに先を行きます。SPLADEスパースベクトルは約30,000次元（語彙用語ごとに1つ）を持ち、その約98%がゼロです。ゼロでない位置は、どの用語がどの程度重要かを示します。「dogs」というクエリは「canine」や「pet」にも重みを置きます。ベクトルインデックス内部での用語拡張を伴う、BM25レベルのキーワード精度が得られます。この列がfalseの場合、正確な用語クエリには外部のFTSレイヤーが必要です。

```python
# SPLADE: 合計約30,000次元、非ゼロは約60 — 関連する語彙ポジションのみが発火
def encode_splade(text: str) -> dict:
    tokens = tokenizer(text, return_tensors="pt", truncation=True, max_length=512)
    with torch.no_grad():
        output = model(**tokens)
    vec = torch.log1p(torch.relu(output.logits)).max(dim=1).values.squeeze()
    return {"indices": vec.nonzero().squeeze().tolist(), "values": vec[vec != 0].tolist()}
```

**SQL / SQL風**は、実際にはフィルタリングに関するものです。フィルタリングのないベクトル検索は、アプリケーションではなくデモに過ぎません。依然としてテナントのスコープ、日付範囲、権限、カテゴリが必要です。フルSQL（pgvector）は、既存の結合操作の隣でこれらを表現できます。専用DBでは、JSONフィルタオブジェクト（Qdrant, Pinecone）、クエリDSL（Elasticsearch, Milvus）、またはGraphQL（Weaviate）を使用します。それらも機能しますが、フィルタロジックが複雑になるにつれてSQLの方が魅力的になります。

```sql
-- pgvector: ベクトル類似度は WHERE 句内の単なる一つの式
SELECT id, title, 1 - (embedding <=> $1) AS score
FROM documents
WHERE tenant_id = $2 AND category = ANY($3::text[]) AND created_at > NOW() - INTERVAL '90 days'
ORDER BY embedding <=> $1 LIMIT 10;
```

```python
# Qdrant: Pythonオブジェクトとしての同等のフィルタ — 機能的だが手間がかかる
results = client.query_points(

        models.FieldCondition(key="category",  match=models.MatchAny(any=categories)),
        models.FieldCondition(key="created_at", range=models.DatetimeRange(gte=cutoff)),
    ]),
    limit=10,
)
```

**マルチモーダル・ネイティブ**とは、「画像の埋め込みを保存できる」という意味ではありません。どのデータベースも浮動小数点配列を保存できます。そうではなく、データベース自体が非テキストコンテンツ用の埋め込みモデルを搭載していることを意味し、生の画像URLを渡すだけでベクトル化を処理してくれることを指します。ここにあるほとんどのデータベースは埋め込みに依存しない（agnostic）ため、そのパイプラインは開発者が用意する必要があります。MarqoやWeaviate（CLIP/ImageBindモジュール経由）はそのループを完結させてくれます。

```python
# Marqo: 生の画像を POST し、テキストで検索 — 外部の埋め込みステップは不要
mq.index("products").add_documents(
    [{"id": "shoe-001", "image": "https://cdn.example.com/shoes/001.jpg"}],
    tensor_fields=["image"]
)
results = mq.index("products").search(q="lightweight shoes for summer")
# キーワードの重複がゼロでも shoe-001 を返す — CLIP がクロスモーダル一致を処理
```

**ディスクベースのインデックス**は、コストの問題です。RAM常駐型のHNSWは、生ベクトル、グラフのオーバーヘッド、メタデータを含めると、1536次元のベクトル100万個につき数GBのRAMを必要とすることがあります。ディスクネイティブな代替案（Milvus DiskANN、Elasticsearch DiskBBQ、LanceDBのLanceフォーマット、Turbopufferのオブジェクトストレージ階層）は、インフラコストを下げる代わりにクエリレイテンシをいくらか犠牲にします。モデルのレイテンシがすでに支配的であるRAGワークロードでは、多くの場合、ベンチマークを行う価値があります。Redis VSSは、RAMのみ、ディスクパスなしというハードな制約があります。

**最大次元数**は、今日の選択に潜む明日の移行リスクです。`text-embedding-3-large`は3072次元を使用し、Jina v3はさらに大きな埋め込みを出力する可能性があり、研究モデルはさらに高次元化を続けています。一部のマネージドサービスは次元数にハードキャップを設けていますが、他のサービスは高いキャップを設けていたり、典型的なモデルに対しては実質的な制限がないことを文書化しています。コミットする前に最新のドキュメントを確認してください。余裕のあるものを選びましょう。次元数の上限に達したためにベクトルインデックスを移行するのは、悲惨な作業になります。

### マトリックスの読み解き方

テーブルに収まりきらない重要な点がいくつかあります。

**スパースベクトル**は、別途全文エンジンを用意することなく、ベクトルインデックス内でBM25品質のキーワードマッチングを実現する方法です。QdrantとElasticsearchは、特に成熟した実装を持っています。WeaviateはBM25F経由でサポートしています。ハイブリッド検索が不可欠で、2つのシステムを運用できない場合は、スパースベクトルのサポートを探してください。

**ディスクベースのインデックス**は、実装の細部ではなく、コストのレバーです。RAM常駐型のHNSWインデックスは高速ですが、ベクトル数、次元数、メタデータ、グラフのオーバーヘッドが増えるにつれて非常に高価になります。ディスクベースの代替案（Milvus DiskANN、Elasticsearch DiskBBQ、Turbopufferオブジェクトストレージ、LanceDBのLanceフォーマット）は、インフラコストを下げるためにクエリレイテンシをトレードオフにします。大規模なRAGインデックスの場合、このトレードオフは多くの場合テストする価値があります。

**Turbopufferのマルチテナンシー**は、非常に多くのネームスペースを扱うことを前提に構築されています。その公開されているポジショニングや事例では、Notionのような大規模でネームスペースの多いコーパスに適したワークロードが強調されています。各ユーザーや組織が独立したベクトル検索を必要とする場合、このアーキテクチャは経済性を変える可能性がありますが、自身のテナントの形状でベンチマークを行うべきです。

**LanceDBの組み込みモード**は、「ベクトル検索版SQLite」に最も近いものです。プロセス内で実行され、サーバーは不要で、Lambda、Cloudflare Workers、エッジ環境でも動作します。Lanceカラムナ形式により、エッジでの運用が現実的なスケールで可能になります。

**Oramaは検索UIインフラであり、倉庫ではありません。** JavaScriptアプリ、エッジ、または管理されたサイト/アプリ検索レイヤーとして、小さな全文/ベクトル/ハイブリッド検索エンジンを導入したい場合には最適です。10億単位のベクトル検索、重い分析、または複雑なフィルタリングを伴う結合には選びません。

**Vespaは、検索結果の取得（Retrieval）が製品の半分に過ぎない場合に選ぶツールです。** 語彙検索、最近傍検索、テンソル、ランキング式、グループ化、およびオンラインサービングを組み合わせます。そのパワーは本物ですが、運用やモデリングの複雑さも同様です。単なる「CRUDアプリに意味検索を追加する」というよりは、検索/レコメンデーション専門のチームに向いています。

**ClickHouseは、検索が分析に付随する場合に会話に上がります。** 信頼できる情報源がイベント、ログ、トレース、メトリクス、または巨大なファクトテーブルである場合、ClickHouseはベクトル距離、フィルタリング、集計、そして今や本格的な全文インデックスを1つのSQLエンジンで保持できます。専用のベクトルデータベースではありませんが、分析的な検索においては「最高に退屈な（＝手堅い）答え」になり得ます。

**Chromaは開発/テスト、および小規模なアプリ展開に最も適しています。** 非常に大規模なコーパス、高可用性、ディスクを多用する運用、またはファーストクラスのハイブリッド検索を目指している場合は、プロトタイプをインフラに昇格させる前に、プロダクション指向のストアを評価してください。

### 簡略化された決定基準

ベンチマークの結果、ベクトル数、フィルタリング、書き込みレート、または高同時実行時のレイテンシがPostgresの限界を超え、pgvectorを卒業したと判断した場合は、制約条件から選択してください。

- **テナントごとの分離が必要なSaaS製品** → Turbopuffer
- **Rustレベルのパフォーマンス + 複雑なメタデータフィルタリングが必要** → Qdrant
- **すでにElastic/ELKスタックを使用中** → DiskBBQ搭載のElasticsearch
- **AWS環境でオープンソースを希望** → OpenSearch
- **高度なランキングが必要な検索/レコメンデーションプラットフォーム** → Vespa
- **分析、オブザーバビリティ、またはログ/イベント検索** → ClickHouse
- **10億規模でセルフホスト / オンプレミス** → Milvus
- **エッジ / サーバーレス / マルチモーダル** → LanceDB
- **小規模なJSアプリ、ドキュメントサイト、またはエッジネイティブな検索UI** → Orama
- **運用ゼロ、とにかく動くこと。コストは二の次** → Pinecone
- **マルチモーダル重視（画像、動画、音声）** → Marqo
- **すでにMongoDBを使用中** → Atlas Vector Search
- **すでにPostgresを使用中で、さらに余裕が必要** → Supabase Vector または Neon（どちらもpgvectorのマネージドで、ツールがより充実している）

## やってはいけないことの一つ

正解が一つしかないものに対して、ベクトル検索を曖昧なテキスト検索として使わないでください。

「メールアドレス `dan@example.com` のユーザーを見つける」というのはベクトル検索の問題ではありません。「ID `ORD-12345` の注文を見つける」も同様です。`ORD-12345`を埋め込んで、注文テーブルに対してコサイン類似度を計算すれば「何か」は返ってきますが、それは間違っているかもしれません。これらは完全一致の問題です。主キーや通常のインデックスを使用してください。

ベクトル検索は、たとえ適切な答えが一つもなくても、データセット内で「最も似ている」ものを返します。良い答えがないことをベクトル検索は知りません。それは関連ドキュメントの提示には問題ありませんが、特定のレコード検索においては致命的です。間違ったニアミスは、空の結果よりも質が悪いからです。

それぞれのツールが何のためのものかを知っておきましょう。その多くは、あなたのPostgresインストールにすでに備わっています。適材適所で使いましょう。
