AI機能を追加しようとするチームは、まず専用のベクターデータベースに手を伸ばしがちだ。

Pinecone、Weaviate、Qdrant、Chroma。新しいサービス、新しい依存関係、新しいコネクションプール、新しい請求書、そして2つの真実の情報源を同期し続ける必要が生まれる。

ところが、彼らはすでにPostgreSQLを持っている。PostgreSQLにはすでに`pgvector`がある。2008年からずっと優れた全文検索も組み込まれている。

専用ベクターストアは、大規模かつ高クエリボリュームで真価を発揮する。しかしほとんどのアプリケーションは、最初の検索システムを十分に使い倒す前に、2つ目の検索システムに手を出してしまう。そうして未来のスケーリング問題が、今日の同期バグに姿を変える。

では、FTSを使うべきか、pgvectorを使うべきか、あるいは両方を使うべきか？

---

## それぞれが実際にやっていること

全文検索（`tsvector` / `GIN`インデックス）は字句ベースだ。テキストをトークン化してレクシームに変換し、ステミングを行い、インデックスに対してクエリをマッチさせる。"Running"と"runs"は同じレクシームに収束する。"dog"と"dogs"も同様だ。ランキング関数（`ts_rank`）は、クエリ用語が頻繁に、または目立つ位置に出現するドキュメントを高く評価する。

pgvectorはセマンティックベースだ。チャンクの*意味*を埋め込みモデルが理解した形で表現する密ベクター（数値のリスト）を格納する。類似度検索は、その高次元空間内で近傍のベクターを見つけ出す。"dog"と"canine"は互いに近くに配置される。スポーツとしての"Running"とプロセスの"running"は、おそらく近くにはならない。

実務上の違い：FTSは「どのドキュメントにこれらの単語が含まれているか？」に答える。ベクター検索は「どのドキュメントがおおよそこんな意味か？」に答える。

![pg_trgmは短い曖昧な文字列、全文検索は正確なプロスクエリ、pgvectorはセマンティックマッチング、ハイブリッド検索は正確なシグナルとセマンティックなシグナルの両方を必要とする長いコンテンツに対応する検索ツールマップ。](../search-tool-map.svg)

_最初の分岐点は「旧来の検索 vs. AI検索」ではない。テキストの形状と、どのような答えが正しいかだ。_

## 全文検索が勝つ場面

**正確に一致してほしい用語を検索しているとき。** 製品SKU、エラーコード、モデル番号、ユーザー名、法的条項の参照。`SKU-AX-44192`が何かにセマンティックに類似することはない。マッチするかしないかだ。ベクター検索は自信満々に`SKU-AX-44193`を返すかもしれない。それはあなたが望むものではない。

**クエリがキーワードベースのとき。** ユーザーが検索ボックスに入力し、タグでフィルタリングし、キーワードでブログ記事を検索する。FTSはそういう意図の形状のために作られている。

**GPUや埋め込みインフラなしでランク付けされた結果が必要なとき。** FTSインデックスは高速で決定論的であり、外部APIコールを必要としない。`tsvector`カラムを追加してGINインデックスを構築すれば完了だ。

**検索と同時にブーリアンフィルタリングを行うとき。** `WHERE to_tsvector(body) @@ to_tsquery('postgres') AND category = 'tutorial' AND published_at > NOW() - INTERVAL '6 months'` — これは既存のクエリロジックと自然に合成できる。

```sql
-- インデックスを作成
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body, '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- クエリ
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`GENERATED ALWAYS AS`カラムはインデックスを自動的に最新に保つ。`setweight`は本文のマッチよりもタイトルのマッチに高いランクを与える。セットアップはこれだけだ。

## Trigramが勝つ場面（pg_trgm）

見落とされがちな第3のPostgresツールがある：`pg_trgm`だ。これは全文検索でもベクター検索でもない。曖昧な文字列マッチングであり、両方のツールが苦手とする中途半端な領域をカバーする。

**ユースケース：名前、住所、識別子、そしてタイプミスを含む短い文字列。**

FTSはテキストをレクシームにトークン化してステミングする。これはプロスには有効だが、以下には不向きだ：
- 人名（"Dan Levy" → 言語設定によっては"dan levi"や"leiv"にステミングされる）
- 会社名、住所、正確なスペルが重要な製品タイトル
- タイプミスを含むクエリ — "Micheal Jordan"、"Amaon"、"javascipt"
- オートコンプリート / 前方一致検索
- 部分文字列マッチング（"son"が"Johnson"や"Anderson"にマッチする）

pgvectorもここには不向きだ。"Micheal Jordan"を埋め込んで最近傍ベクターを見つけることはできるが、埋め込み空間はスペルではなく意味で名前を整理する。最近傍は"basketball legend"や"Michael B. Jordan"かもしれず、タイプミスを含むユーザーレコードではない。

`pg_trgm`は文字列を重複する3文字のスライスに分解し、2つの文字列が共有するトライグラムの数を測定する。"Dan" → `" da"`、`"dan"`、`"an "`。"Micheal"と"Michael"はほとんどのトライグラムを共有するため、類似度は高くなる。

```sql
-- 拡張機能を有効化（通常はすでに利用可能）
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- 名前カラムのGINインデックス — トライグラム類似度検索を高速化
CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- 曖昧な名前検索："Michael Jordan"の検索で"Micheal Jordan"を見つける
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % 演算子 = 類似度しきい値（デフォルト0.3）
ORDER BY score DESC
LIMIT 10;

-- またはILIKEとトライグラムインデックスサポートを使った部分一致検索
SELECT id, name
FROM users
WHERE name ILIKE '%johnson%'   -- GINインデックスで高速化
LIMIT 10;
```

`%`演算子は`pg_trgm.similarity_threshold`（デフォルト0.3、範囲0-1）を使用する。値が高いほどより厳密なマッチが必要になる。名前検索では0.3〜0.4が通常適切だ：タイプミスを拾うのに十分寛容で、ノイズを避けるのに十分厳格。

**トライグラムは前方一致検索やオートコンプリートにも役立つ。特にオートコンプリートにタイプミス耐性や部分一致が必要な場合：**

```sql
-- オートコンプリート：前方一致。純粋な左端固定の前方一致の場合は、
-- トライグラムGINとデータ上のB-treeパターンインデックスを比較すること。
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- より細かい制御：より長い文字列内の部分一致に対するword_similarity
-- （"Andrew Johnson III"内で"Johnson"を検索する場合に有用）
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name          -- <% 演算子 = word_similarityしきい値
ORDER BY score DESC
LIMIT 10;
```

**FTSより`pg_trgm`を選ぶべき場合：**

| シナリオ | 使用するもの |
|---|---|
| タイプミスがある人名・会社名の検索 | `pg_trgm` |
| オートコンプリート / 前方一致検索 | `pg_trgm`（または前方一致クエリを使うFTS） |
| 短い文字列、コード、識別子の検索 | `pg_trgm` |
| プロス記事、ドキュメントの検索 | FTS |
| ログメッセージのキーワード検索 | FTS |
| 多言語の名前検索 | `pg_trgm`（言語に依存しない） |

`pg_trgm`はFTSと組み合わせることもできる。トライグラムで曖昧な事前フィルタリングを行い`ts_rank`でランク付けするか、トライグラム類似度をベクタースコアと結合する。

## pgvectorが勝つ場面

**RAGを構築しているとき。** RAGはセマンティック検索に依存する。ユーザーの質問に意味が最も近いドキュメントの*チャンク*を、たとえ文言が異なっていても見つける。ベクター検索はこのために作られている。FTSは言い換え、同義語、概念的なマッチを見逃す。

**ユーザーが何を検索するかではなく、何が欲しいかを説明しているとき。** 「夏の夜にぴったりの軽いもの」には明確なワインのキーワードがない。「新しいマネージャーとして自信をつける方法についての記事」には、FTSでは提供できないセマンティックな理解が必要だ。

**類似アイテムを探しているとき。** 関連商品、類似サポートチケット、重複バグレポート。「このissueに似た問題を探して」はベクター演算だ。新しいissueを埋め込み、その最近傍を見つける。

**多言語コンテンツ。** 多言語データで訓練されたベクター埋め込みは、言語をまたいでマッチできる。FTSは言語固有の設定が必要で、言語横断的なクエリは苦手だ。

```sql
-- セットアップ
CREATE EXTENSION IF NOT EXISTS vector;

ALTER TABLE documents ADD COLUMN embedding vector(1536);
CREATE INDEX documents_embedding_idx
  ON documents USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);

-- クエリ：セマンティック検索
SELECT id, title, 1 - (embedding <=> $1::vector) AS similarity
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 10;
```

注：`ivfflat`は近似検索だ — 高速だが、再現率をいくらか犠牲にする。小規模データセット（約100万行未満）では、`hnsw`のほうが大抵は優れている：

```sql
CREATE INDEX documents_embedding_idx
  ON documents USING hnsw (embedding vector_cosine_ops);
```

## 両方が必要な場合

技術ドキュメントこそ、単純な二分法が崩れる場所だ。ユーザーは「タイムアウトの設定方法」を検索する一方で、`withRetry()`のような関数名や`ECONNRESET`のようなエラーコードも検索する。

ベクター検索は概念的なクエリを処理する。FTSは正確な用語を処理する。どちらか一方だけでは両方をうまく扱えない。

解決策はハイブリッド検索だ。両方を実行し、結果を融合する。

**Reciprocal Rank Fusion（RRF）**が標準アルゴリズムだ。2つのシステムからのスコアを正規化する必要はない。ランクの位置を組み合わせるだけだ。

```sql
-- Reciprocal Rank Fusionによるハイブリッド検索
WITH fts_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM documents, to_tsquery('english', $1) query
  WHERE search_vector @@ query
  LIMIT 50
),
vector_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> $2::vector) AS rank
  FROM documents
  ORDER BY embedding <=> $2::vector
  LIMIT 50
),
rrf AS (
  SELECT
    COALESCE(f.id, v.id) AS id,
    COALESCE(1.0 / (60 + f.rank), 0) + COALESCE(1.0 / (60 + v.rank), 0) AS rrf_score
  FROM fts_results f
  FULL OUTER JOIN vector_results v ON f.id = v.id
)
SELECT d.id, d.title, rrf.rrf_score
FROM rrf
JOIN documents d ON d.id = rrf.id
ORDER BY rrf_score DESC
LIMIT 10;
```

分母の`60`はRRF定数だ — 値が大きいほどランク差の影響を減らし、小さいほど増幅する。デフォルトの60はほとんどのケースでうまく機能する。

このクエリは1つのクエリで2つの検索を実行し、ランクを融合し、キーワードとセマンティックの両方のシグナルが一致する結果を高く評価する。

![1つのクエリが全文検索とpgvectorに分岐し、それぞれがランク付けされた結果を生成し、Reciprocal Rank Fusionが2つのリストを結合するハイブリッド検索パイプライン。](../hybrid-rrf-pipeline.svg)

_RRFが価値ある理由は、`ts_rank`とコサイン距離を比較可能な生スコアだと偽らないことにある。「この結果は各リストのどのくらい上位に現れたか？」だけを問う。_

## 実践的な判断ツリー

検索戦略を選ぶときは、**入力の形状**から始め、次に**ユーザーがどのようなクエリを発行しているか**を考える。「スペルに揺れのある短い文字列」は「正確な用語が重要な長文プロス」と同じ問題ではなく、どちらも「ドキュメントチャンクに対する質問」とも異なる。

![テキストの形状とユーザーのクエリ方法に基づいて、pg_trgm、全文検索、pgvector、ハイブリッド検索、または専用ベクターデータベースを選択する判断ツリー。](../search-decision-tree.svg)

同じツリーを言葉で表すと：

- **名前、住所、タイトル、オートコンプリート、またはタイプミスが発生しやすい短い文字列** → `pg_trgm`
- **既知の単語、エラーコード、SKU、関数名、タグ、カテゴリ、フィルター** → FTS
- **質問、言い換え、レコメンデーション、関連アイテム、多言語マッチング、RAGチャンク** → pgvector
- **正確なシンボルと概念的な答えの両方をユーザーが求める技術コンテンツ** → RRFによるハイブリッド
- **主キー、正確なID、権限フィルター、日付範囲、ソート済みリスト** → 通常のSQLインデックス
- **巨大なベクター量、非常に高い同時実行性、またはPostgresがベンチマークで達成できないレイテンシ目標** → 専用ベクターストアを評価

### FTS vs. セマンティック：短縮版

「FTSとベクター検索のどちらを使うべきか？」という問いは、たいてい次の問いに収束する：**関連ドキュメントにどんな単語が出現するか分かっているか？**

はい — ユーザーが既知の用語、カテゴリ、関数名、製品コードを検索する場合 — FTSはより高速で、安価で、予測可能だ。なぜ結果がマッチしたかを説明できる。

いいえ — ユーザーが概念を説明したり、質問をしたり、別の言語で検索する場合 — ベクター検索が正しいツールだ。単語ではなく意味にマッチする。

厄介な中間領域は、技術コンテンツに対する自然言語クエリだ。「接続が切れた場合の対処方法」を検索する人は、「ネットワーク障害のためのリトライロジックの実装」という記事を必要としているかもしれない — 重複する単語は一切なく、セマンティックな関連性は高い。そこでベクター検索が真価を発揮する。

もう一つの厄介なケースは**名前と固有名詞**だ。FTSもベクター検索もこれらを得意としない：
- FTSは"Michael"で検索したときに"Micheal"を見逃す — 異なるトークンだからだ
- ベクター検索は、名前が訓練データに頻繁に出現しない場合、名前を完全に見逃す可能性がある
- `pg_trgm`はこれを正しく処理する。意味的でも字句的でもなく、正書法的な類似性に基づく

実際には、ほとんどのコンテンツ重視の検索ボックスは、速度とキーワードのためにFTSを必要とし、ユーザーが名前を検索するかどうかに応じてハイブリッドまたは`pg_trgm`が必要になる可能性がある。真のセマンティック検索機能は通常pgvectorを意味する。RAGは常にpgvectorを意味する。

## 専用ベクターストアが必要な場合

一部のシステムは本当にpgvectorを超える。その場合、市場はノイズが多い。トップオプションの中で本当に重要なのはこれだ。

### 機能マトリックス

テーブルを理解する前に、いくつかの列を解説しておく必要がある。

**ハイブリッド検索**は、BM25キーワード検索とベクター類似性を1つのクエリで実行し、Reciprocal Rank Fusionでマージすることを意味する。"withRetry timeout"は関数名を正確にマッチさせ*つつ*、「ネットワーク障害のリトライロジック」に関するドキュメントをセマンティックにマッチさせることができる。ハイブリッドがなければ、どちらかの検索モードを選ぶか、2つのクエリを自分で融合する必要がある。pgvectorの「Manual（RRF via SQL）」は[上記で示したアプローチ](#両方が必要な場合)だ。機能はするが、自分で書く必要がある。

**スパースベクター**はBM25をさらに一歩進める。SPLADEスパースベクターは約30,000次元（語彙の各用語に1つ）で、約98%がゼロだ。非ゼロの位置は、どの用語が重要で、どの程度重要かを示す。"dogs"のクエリは"canine"や"pet"にも重みを付ける。ベクターインデックス内でBM25レベルのキーワード精度と用語展開を実現する。この列がfalseの場合、正確な用語クエリのために外部のFTS層が必要になる。

```python
# SPLADE：合計約30,000次元、非ゼロは約60 — 関連する語彙位置のみが発火
def encode_splade(text: str) -> dict:
    tokens = tokenizer(text, return_tensors="pt", truncation=True, max_length=512)
    with torch.no_grad():
        output = model(**tokens)
    vec = torch.log1p(torch.relu(output.logits)).max(dim=1).values.squeeze()
    return {"indices": vec.nonzero().squeeze().tolist(), "values": vec[vec != 0].tolist()}
```

**SQL / SQLライク**は、実際にはフィルタリングのことだ。フィルタリングなしのベクター検索はデモであって、アプリケーションではない。テナントスコープ、日付範囲、権限、カテゴリがまだ必要だ。完全なSQL（pgvector）は、既存の結合の隣でこれを表現できる。専用DBはJSONフィルターオブジェクト（Qdrant、Pinecone）、クエリDSL（Elasticsearch、Milvus）、またはGraphQL（Weaviate）を使用する。これらも機能するが、フィルターロジックが複雑になるにつれてSQLの魅力が増す。

```sql
-- pgvector：ベクター類似性はWHEREの中の単なる式のひとつ
SELECT id, title, 1 - (embedding <=> $1) AS score
FROM documents
WHERE tenant_id = $2 AND category = ANY($3::text[]) AND created_at > NOW() - INTERVAL '90 days'
ORDER BY embedding <=> $1 LIMIT 10;
```

**マルチモーダルネイティブ**は「画像埋め込みを保存できる」という意味ではない。どのデータベースもfloat配列を保存できる。データベースが非テキストコンテンツ用の埋め込みモデルを同梱し、生の画像URLを渡すとベクター化を処理してくれることを意味する。ここに挙げたほとんどのデータベースは埋め込みに依存しないので、そのパイプラインは自分で所有することになる。MarqoとWeaviate（CLIP/ImageBindモジュール経由）はループを閉じる。

```python
# Marqo：生の画像をPOSTし、テキストでクエリ — 外部埋め込みステップは不要
mq.index("products").add_documents(
    [{"id": "shoe-001", "image": "https://cdn.example.com/shoes/001.jpg"}],
    tensor_fields=["image"]
)
results = mq.index("products").search(q="lightweight shoes for summer")
# キーワードの重複がゼロでもshoe-001を返す — CLIPがクロスモーダルマッチを処理
```

**ディスクベースのインデックス**はコストの問題だ。RAM常駐のHNSWは、生ベクター、グラフオーバーヘッド、メタデータを合わせると、100万個の1536次元ベクターあたり数GBのRAMを必要とすることがある。ディスクネイティブの代替（Milvus DiskANN、Elasticsearch DiskBBQ、LanceDBのLance形式、Turbopufferのオブジェクトストレージ層）は、クエリレイテンシをある程度犠牲にしてインフラコストを削減する。モデルレイテンシがすでに支配的であるRAGワークロードでは、ベンチマークを取る価値がしばしばある。Redis VSSは厳格な制約がある。RAMのみでディスクパスはない。

**最大次元数**は、今日の選択に潜む明日の移行リスクだ。`text-embedding-3-large`は3072次元を使い、Jina v3はさらに大きな埋め込みを出力でき、研究モデルは次元を押し上げ続けている。一部のマネージドサービスはハードな次元上限を公表している。他のサービスは上限が高いか、典型的な埋め込みモデルには実質的な上限がないとドキュメントに記載している。コミットする前に現在のドキュメントを確認すること。余裕を持って選ぶこと。次元の天井にぶつかってベクターインデックスを移行するのは悲惨な作業だ。

### マトリックスの読み方

テーブルにきれいに収まらないものがいくつかある：

**スパースベクター**は、別個の全文エンジンなしでベクターインデックス内でBM25品質のキーワードマッチングを実現する方法だ。QdrantとElasticsearchは特に成熟した実装を持つ。WeaviateはBM25F経由でサポートする。ハイブリッド検索が重要で2つのシステムを実行できない場合は、スパースベクターのサポートを探すべきだ。

**ディスクベースのインデックス**は実装の詳細ではなく、コストのレバーだ。RAM常駐のHNSWインデックスは高速だが、ベクター数、次元数、メタデータ、グラフオーバーヘッドが増えるにつれて高価になりうる。ディスクベースの代替（Milvus DiskANN、Elasticsearch DiskBBQ、Turbopufferのオブジェクトストレージ、LanceDBのLance形式）は、クエリレイテンシと引き換えにインフラコストを削減する。大規模なRAGインデックスでは、そのトレードオフはテストする価値がある。

**Turbopufferのマルチテナンシー**は、非常に多い名前空間の数を中心に構築されている。その公開ポジショニングと顧客事例は、Notionのような大規模で名前空間の多いコーパスを強調している。各ユーザーや組織に分離されたベクター検索が必要な場合、そのアーキテクチャは経済性を変える可能性があるが、自身のテナント形状でベンチマークを取ること。

**LanceDBの組み込みモード**は、「ベクター検索のためのSQLite」に最も近いものだ。インプロセスで動作し、サーバーを必要とせず、Lambda、Cloudflare Workers、エッジ環境で動作する。Lanceのカラムナ形式により、組み込み運用が現実的なスケールで実現する。

**Oramaは検索UXインフラであって、ウェアハウスではない。** JavaScriptアプリ内、エッジ上、またはマネージドサイト/アプリ検索層として、小さな全文/ベクター/ハイブリッド検索エンジンが必要な場合に優れている。10億ベクターの検索、ヘビーな分析、複雑なフィルタリング結合には適さない。

**Vespaは、検索が製品の半分にすぎない場合に手を伸ばすものだ。** 字句検索、近傍検索、テンソル、ランキング式、グルーピング、オンラインサービングを組み合わせる。そのパワーは本物だが、運用とモデリングの複雑さも本物だ。「CRUDアプリにセマンティック検索を追加する」よりも、検索/レコメンデーションチームに適している。

**ClickHouseは、検索が分析に付随する場合に会話に加わるべきだ。** 真実の情報源がイベント、ログ、トレース、メトリクス、または大規模なファクトテーブルである場合、ClickHouseはベクター距離、フィルタリング、集約、そして現在は本格的な全文インデックスを1つのSQLエンジンで扱える。専用ベクターデータベースではないが、分析検索においては最良の意味で退屈な答えになりうる。

**Chromaは、開発/テストと小規模アプリのデプロイメントで最も強力だ。** 非常に大規模なコーパス、HA、ディスク主体の運用、または第一級のハイブリッド検索を目指しているなら、プロトタイプをインフラに昇格させる前にプロダクション向けストアを評価すること。

### 簡略化された判断基準

pgvectorを本当に超えた場合 — 通常はベンチマークがベクター数、フィルタリング、書き込みレート、または高同時実行レイテンシがPostgresの限界を超えていることを示す場合 — 制約条件に従って選ぶ：

- **テナントごとに分離されたSaaS製品** → Turbopuffer
- **Rustレベルのパフォーマンス + 複雑なメタデータフィルタリングが必要** → Qdrant
- **すでにElastic/ELKスタックを使用中** → DiskBBQ付きElasticsearch
- **AWSショップでオープンソースを希望** → OpenSearch
- **本格的なランキングニーズを持つ検索/レコメンデーションプラットフォーム** → Vespa
- **分析、オブザーバビリティ、またはログ/イベント検索** → ClickHouse
- **10億スケールのオンプレミス / セルフホスト** → Milvus
- **エッジ / サーバーレス / マルチモーダル** → LanceDB
- **小規模JSアプリ、ドキュメントサイト、またはエッジネイティブな検索UX** → Orama
- **運用不要、とにかく動くこと、コストは二の次** → Pinecone
- **マルチモーダルファースト（画像、動画、音声）** → Marqo
- **すでにMongoDBを使用** → Atlas Vector Search
- **すでにPostgresを使用、より多くの余裕が必要** → Supabase VectorまたはNeon（どちらもマネージドpgvector、より良いツール付き）

## やってはいけないこと

ベクター検索を、正解があるものに対するあいまいテキスト検索として使ってはいけない。

「メールアドレス`dan@example.com`のユーザーを探して」はベクター検索の問題ではない。「ID`ORD-12345`の注文を探して」も同様だ。`ORD-12345`を埋め込んで注文テーブルに対してコサイン類似度を計算すると*何か*は返ってくるが、間違っているかもしれない。これらは完全一致の問題だ。主キーか通常のインデックスを使うこと。

ベクター検索は、関連するものが何もない場合でも、データセット内で*最も類似した*ものを返す。良い答えがないことを知らないのだ。関連ドキュメントを探す場合にはそれで構わない。しかし、間違ったニアマッチが空の結果より悪い特定レコードの検索では、それは壊滅的だ。

それぞれのツールの役割を知ること。それらのほとんどはすでにPostgreSQLに含まれている。適切な場所で使おう。
