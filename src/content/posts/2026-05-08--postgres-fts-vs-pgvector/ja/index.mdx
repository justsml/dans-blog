AI機能を実装するチームは、まず専用ベクターデータベースに手を伸ばしがちだ。

Pinecone、Weaviate、Qdrant、Chroma。新しいサービス、新しい依存関係、新しいコネクションプール、新しい請求、そして同期を保たなければならない2つの信頼できる情報源。

その一方で、彼らはすでにPostgreSQLを持っている。PostgreSQLにはすでに`pgvector`がある。2008年から優れた全文検索機能も組み込まれている。

専用ベクターストアは、大規模かつ高クエリボリュームで真価を発揮する。しかし多くのアプリケーションは、最初のシステムが十分に使いこなされる前に2つ目の検索システムに手を伸ばしてしまう。それが将来のスケーリング問題を、今日の同期バグに変えてしまうのだ。

では: FTSを使うべきか、pgvectorを使うべきか、それとも両方使うべきか？

---

## それぞれが実際に何をするか

全文検索（`tsvector` / `GIN`インデックス）は字句ベースだ。テキストをlexemeにトークン化し、ステミングを行い、インデックスに対してクエリをマッチさせる。「running」と「runs」は同じlexemeに収束する。「dog」と「dogs」も同様だ。ランキング関数（`ts_rank`）は、クエリ用語が頻繁に、または目立つ位置に出現するドキュメントを高く評価する。

pgvectorはセマンティックだ。チャンクの*意味*を埋め込みモデルが理解した状態で表す密ベクター（数値のリスト）を格納する。類似度検索は、その高次元空間内で近接するベクターを見つける。「dog」と「canine」は近くに配置される。スポーツの「running」とプロセスを「実行する」のは、おそらく近くならない。

実務的な違い: FTSは「どのドキュメントにこれらの単語が含まれているか？」に答える。ベクター検索は「どのドキュメントがおおよそこのことを意味しているか？」に答える。

![pg_trgmは短くファジーな文字列用、全文検索は正確なプロスクエリ用、pgvectorはセマンティックマッチング用、ハイブリッド検索は正確なシグナルとセマンティックなシグナルの両方を必要とする長いコンテンツ用の検索ツールマップ。](../search-tool-map.svg)

_最初の分岐点は「古い検索対AI検索」ではない。テキストの形状と、どのような答えが正しいかだ。_

## 全文検索が勝つ場面

**正確な用語が重要なものを検索している場合。** 製品SKU、エラーコード、モデル番号、ユーザー名、法的条項の参照。`SKU-AX-44192`が何かにセマンティックに類似していることはない。マッチするかしないかだ。ベクター検索は自信を持って`SKU-AX-44193`を返すかもしれない。それはあなたが望むものではない。

**クエリがキーワードベースの場合。** ユーザーが検索ボックスに入力し、タグでフィルタリングし、キーワードでブログ記事を検索する。FTSはそのような意図の形状のために作られている。

**GPUや埋め込みインフラなしでランク付けされた結果が必要な場合。** FTSインデックスは高速で決定論的であり、外部API呼び出しを必要としない。`tsvector`カラムを追加し、GINインデックスを構築すれば完了だ。

**検索と同時にブーリアンフィルタリングを行う場合。** `WHERE to_tsvector(body) @@ to_tsquery('postgres') AND category = 'tutorial' AND published_at > NOW() - INTERVAL '6 months'` — これは既存のクエリロジックと自然に合成できる。

```sql
-- インデックスを作成
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body, '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- クエリ
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`GENERATED ALWAYS AS`カラムはインデックスを自動的に最新の状態に保つ。`setweight`は本文のマッチよりもタイトルのマッチにより高いランクを与える。それがセットアップのすべてだ。

## Trigramが勝つ場面（pg_trgm）

見落とされがちな第三のPostgresツールがある: `pg_trgm`だ。全文検索でもベクター検索でもない。ファジーマッチングであり、両方のツールが苦手とする中途半端な領域をカバーする。

**ユースケース: 名前、住所、識別子、タイプのある短文字列。**

FTSはテキストをlexemeにトークン化してステミングする。プロスには有効だが、以下には不向きだ:
- 人名（「Dan Levy」→「dan levi」「leiv」にステミングされる。言語設定に依存）
- 会社名、住所、正確なスペルが重要な製品タイトル
- タイプのあるクエリ — 「Micheal Jordan」「Amaon」「javascipt」
- 補完 / プレフィックス検索
- 部分文字列マッチング（「son」が「Johnson」「Anderson」にマッチ）

pgvectorもここには不向きだ。「Micheal Jordan」を埋め込んで最近傍ベクターを見つけることはできるが、埋め込み空間は意味で名前を整理するのであって、スペルではない。最近傍は「バスケットボールのレジェンド」や「Michael B. Jordan」かもしれないが、タイプのあるユーザーレコードではない。

`pg_trgm`は文字列を重複する3文字のスライスに分解し、2つの文字列が共有するtrigramの数を測定する。「Dan」→ `" da"`、`"dan"`、`"an "`。「Micheal」と「Michael」はtrigramのほとんどを共有するので、類似度は高くなる。

```sql
-- 拡張機能を有効化（通常はすでに利用可能）
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- 名前カラムのGINインデックス — trigram類似度検索を高速化
CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- ファジー名前検索: 「Micheal Jordan」を「Michael Jordan」で検索
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % 演算子 = 類似度閾値（デフォルト0.3）
ORDER BY score DESC
LIMIT 10;

-- またはILIKEでtrigramインデックスサポートを使用した部分一致検索
SELECT id, name
FROM users
WHERE name ILIKE '%johnson%'   -- GINインデックスで高速化
LIMIT 10;
```

`%`演算子は`pg_trgm.similarity_threshold`（デフォルト0.3、範囲0-1）を使用する。値が高いほど近いマッチが必要になる。名前検索では0.3-0.4が通常適切だ: タイプを捉えるのに十分寛容で、ノイズを避けるのに十分厳格。

**プレフィックス検索や補完にもtrigramが役立つ。特に補完にタイプ許容や部分一致が必要な場合:**

```sql
-- 補完: プレフィックスマッチング。純粋な左端プレフィックスの場合、
-- trigram GINをデータに対するB-treeパターンインデックスと比較する。
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- より細かい制御: より長い文字列内の部分一致に対するword_similarity
-- （「Andrew Johnson III」内で「Johnson」を検索する場合に有用）
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name          -- <% 演算子 = word_similarity閾値
ORDER BY score DESC
LIMIT 10;
```

**FTSより`pg_trgm`を選ぶべき場合:**

| シナリオ | 使用 |
|---|---|
| タイプのある人名/会社名検索 | `pg_trgm` |
| 補完 / プレフィックス検索 | `pg_trgm`（またはプレフィックスクエリでFTS） |
| 短文字列、コード、識別子の検索 | `pg_trgm` |
| プロス記事、ドキュメントの検索 | FTS |
| ログメッセージのキーワード検索 | FTS |
| 多言語名検索 | `pg_trgm`（言語非依存） |

`pg_trgm`はFTSと合成もできる。trigramでファジーなプレフィルタを行い、`ts_rank`でランク付けするか、trigram類似度をベクタースコアと組み合わせる。

## pgvectorが勝つ場面

**RAGを構築している場合。** RAGはセマンティック検索に依存する: ユーザーの質問に意味が近いドキュメント*チャンク*を見つける。単語が異なる場合でも。ベクター検索はこのために作られている。FTSは言い換え、同義語、概念的なマッチを見逃す。

**ユーザーが何を検索するかではなく、何を欲しいかを説明している場合。** 「夏の夜に適したライトなもの」「新しいマネジャーとしての自信を構築する方法に関する記事」 — FTSでは提供できないセマンティックな理解が必要だ。

**類似アイテムを探している場合。** 関連製品、類似サポートチケット、重複バグレポート。「この issue に類似した問題を見つけてくれ」はベクター演算だ。新しい issue を埋め込み、最近傍を見つける。

**多言語コンテンツ。** 多言語データで訓練されたベクター埋め込みは、言語間でマッチできる。FTSは言語固有の設定が必要で、クロス言語クエリは苦手だ。

```sql
-- セットアップ
CREATE EXTENSION IF NOT EXISTS vector;

ALTER TABLE documents ADD COLUMN embedding vector(1536);
CREATE INDEX documents_embedding_idx
  ON documents USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);

-- クエリ: セマンティック検索
SELECT id, title, 1 - (embedding <=> $1::vector) AS similarity
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 10;
```

注: `ivfflat`は近似検索だ — 高速だが、速度のためにいくつかの再現性を犠牲にする。小規模データセット（約100万行未満）では、`hnsw`の方が良いことが多い:

```sql
CREATE INDEX documents_embedding_idx
  ON documents USING hnsw (embedding vector_cosine_ops);
```

## 両方必要な場合

技術ドキュメントでは、単純な分割が崩れる場所だ。ユーザーは「タイムアウトの設定方法」を検索するが、`withRetry()`のような関数名や`ECONNRESET`のようなエラーコードも検索する。

ベクター検索は概念的なクエリを処理する。FTSは正確な用語を処理する。どちらか一方だけでは両方をうまく処理できない。

解決策はハイブリッド検索: 両方を実行し、結果を融合する。

**Reciprocal Rank Fusion（RRF）**がここでの標準アルゴリズムだ。2つのシステムからのスコアを正規化する必要はない。ランク位置を組み合わせるだけだ。

```sql
-- Reciprocal Rank Fusionによるハイブリッド検索
WITH fts_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM documents, to_tsquery('english', $1) query
  WHERE search_vector @@ query
  LIMIT 50
),
vector_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> $2::vector) AS rank
  FROM documents
  ORDER BY embedding <=> $2::vector
  LIMIT 50
),
rrf AS (
  SELECT
    COALESCE(f.id, v.id) AS id,
    COALESCE(1.0 / (60 + f.rank), 0) + COALESCE(1.0 / (60 + v.rank), 0) AS rrf_score
  FROM fts_results f
  FULL OUTER JOIN vector_results v ON f.id = v.id
)
SELECT d.id, d.title, rrf.rrf_score
FROM rrf
JOIN documents d ON d.id = rrf.id
ORDER BY rrf_score DESC
LIMIT 10;
```

分母の`60`はRRF定数だ — 値が高いほどランク差の影響が減少し、低いほど増幅される。デフォルトの60はほとんどの場合でうまく機能する。

これは1つのクエリで2つの検索を実行し、ランクを融合し、キーワードとセマンティックなシグナルが一致する結果に報酬を与える。

![1つのクエリが全文検索とpgvectorに分岐し、それぞれがランク付けされた結果を生成し、Reciprocal Rank Fusionが2つのリストを結合するハイブリッド検索パイプライン。](../hybrid-rrf-pipeline.svg)

_RRFが価値ある理由は、`ts_rank`とコサイン距離を比較可能な生スコアであると見なすことを回避できるからだ。これは「この結果が各リストでどのくらい高く登場したか？」だけを問う。_

## 実務的な意思決定ツリー

検索戦略を選ぶときは、**入力の形状**から始め、次に**ユーザーがどのようなクエリを行っているか**を問う。「スペル変動のある短文字列」は「正確な用語が重要な長いプロス」と同じ問題ではなく、どちらも「ドキュメントチャンクに対する質問」とは異なる。

![テキストの形状とユーザーのクエリ方法に基づいて、pg_trgm、全文検索、pgvector、ハイブリッド検索、または専用ベクターデータベースを選択する意思決定ツリー。](../search-decision-tree.svg)

同じツリーを言葉で表すと:

- **名前、住所、タイトル、補完、またはタイプ prone な短文字列** → `pg_trgm`
- **既知の単語、エラーコード、SKU、関数名、タグ、カテゴリ、フィルター** → FTS
- **質問、言い換え、推薦、関連アイテム、多言語マッチング、RAGチャンク** → pgvector
- **正確なシンボルと概念的な答えの両方をユーザーが必要とする技術コンテンツ** → RRFによるハイブリッド
- **主キー、正確なID、権限フィルター、日付範囲、ソートされたリスト** → 通常のSQLインデックス
- **巨大なベクターボリューム、非常に高い同時実行数、またはベンチマークでPostgresが達成できないレイテンシー目標** → 専用ベクターストアの評価

### FTS vs セマンティック: 短縮版

「FTSを使うべきか、ベクター検索を使うべきか？」という質問は通常これに収束する: **関連ドキュメントにどの単語が登場するか知っているか？**

はいの場合 — ユーザーが既知の用語、カテゴリ、関数名、製品コードを検索する場合 — FTSはより高速で、安価で、予測可能だ。なぜ結果がマッチしたかを示す。

いいえの場合 — ユーザーが概念を説明したり、質問をしたり、別の言語で検索する場合 — ベクター検索が正しいツールだ。単語ではなく意味にマッチする。

厄介な中間地带は、技術コンテンツに対する自然言語クエリだ。「接続ドロップをどう処理するか」を検索する人は、「ネットワーク障害の再試行ロジックの実装」というタイトルの記事を必要とするかもしれない — 重複する単語はないが、セマンティックな関連性は高い。そこでベクター検索が真価を発揮する。

もう一つの厄介なケースは**名前と固有名詞**だ。FTSもベクター検索もこれらにはあまり得意ではない:
- FTSは「Micheal」を検索する際に「Michael」を見逃す — 異なるトークン
- ベクター検索は、名前が訓練データに頻繁に登場しない場合、完全に見逃す可能性がある
- `pg_trgm`はこれを正しく処理する: 意味や字句ではなく、正字法類似性

実務的には、コンテンツ重視の検索ボックスのほとんどは、速度とキーワードのためにFTSを必要とし、ユーザーが名前を検索するかどうかによってハイブリッドや`pg_trgm`が必要になる可能性がある。真のセマンティック検索機能は通常、pgvectorを意味する。RAGは常にpgvectorを意味する。

## 専用ベクターストアが必要な場合

pgvectorを本当に超えるシステムもある。その場合、市場は騒がしい。ここで重要なのはトップオプションの中で何が重要かだ。

### 機能マトリックス

テーブルを意味のあるものにする前に、いくつかの列を紐解く必要がある。

**ハイブリッド検索**とは、BM25キーワード検索とベクター類似性を1つのクエリで実行し、Reciprocal Rank Fusionでマージすることを意味する。「withRetry timeout」は関数名を正確にマッチさせ*つつ*、「ネットワーク障害の再試行ロジック」に関するドキュメントもセマンティックにマッチさせることができる。ハイブリッドがなければ、どちらかの検索モードを選ぶか、2つのクエリを自分で融合する必要がある。pgvectorの「Manual (RRF via SQL)」は[上記で示したアプローチ](#when-you-need-both)だ: 機能するが、自分で書く必要がある。

**スパースベクター**はBM25をさらに進める。SPLADEスパースベクターは約30,000次元（語彙項目ごとに1つ）、約98%がゼロだ。非ゼロの位置は、どの用語が重要で、どの程度重要かを示す。「dogs」のクエリは「canine」や「pet」にも重みを付ける: ベクターインデックス内でBM25レベルのキーワード精度と用語展開。この列がfalseの場合、正確な用語クエリには外部FTSレイヤーが必要だ。

```python
# SPLADE: 合計約30,000次元、非ゼロは約60 — 関連する語彙位置のみが発火
def encode_splade(text: str) -> dict:
    tokens = tokenizer(text, return_tensors="pt", truncation=True, max_length=512)
    with torch.no_grad():
        output = model(**tokens)
    vec = torch.log1p(torch.relu(output.logits)).max(dim=1).values.squeeze()
    return {"indices": vec.nonzero().squeeze().tolist(), "values": vec[vec != 0].tolist()}
```

**SQL / SQLに類似**は、実際にはフィルタリングについてだ。フィルタリングなしのベクター検索はデモであって、アプリケーションではない: テナントスコープ、日付範囲、権限、カテゴリがまだ必要だ。完全なSQL（pgvector）は既存の結合の横でこれを表現する。専用DBはJSONフィルターオブジェクト（Qdrant、Pinecone）、クエリDSL（Elasticsearch、Milvus）、またはGraphQL（Weaviate）を使用する。機能する; フィルターロジックが複雑になるにつれて、SQLはより魅力的になる。

```sql
-- pgvector: ベクター類似性はWHERE内の単なる式
SELECT id, title, 1 - (embedding <=> $1) AS score
FROM documents
WHERE tenant_id = $2 AND category = ANY($3::text[]) AND created_at > NOW() - INTERVAL '90 days'
ORDER BY embedding <=> $1 LIMIT 10;
```

```python
# Qdrant: 同等のフィルターをPythonオブジェクトとして — 機能的だが、より儀式が必要
```

**マルチモーダルネイティブ**は「画像埋め込みを格納できる」ことを意味するのではない; どのデータベースも浮動小数点配列を格納できる。データベースが非テキストコンテンツの埋め込みモデルを出荷し、生の画像URLを渡すとベクター化を処理することを意味する。ここにあるほとんどのデータベースは埋め込み非依存なので、そのパイプラインはあなたが所有する。MarqoとWeaviate（CLIP/ImageBindモジュール経由）はループを閉じる。

```python
# Marqo: 生の画像をPOST、テキストでクエリ — 外部埋め込みステップは不要
mq.index("products").add_documents(
    [{"id": "shoe-001", "image": "https://cdn.example.com/shoes/001.jpg"}],
    tensor_fields=["image"]
)
results = mq.index("products").search(q="夏の軽量な靴")
```

### マトリックスを読む

テーブルにきれいに収まらないものがいくつかある:

**スパースベクター**は、別個の全文エンジンなしでベクターインデックス内でBM25品質のキーワードマッチングを得る方法だ。QdrantとElasticsearchはここで特に成熟した実装を持っている。WeaviateはBM25F経由でサポートしている。ハイブリッド検索が重要で、2つのシステムを実行できない場合、スパースベクターのサポートを探すべきだ。

**ディスクベースインデックス**はコストレバーであって、実装の詳細ではない。RAM常駐HNSWインデックスは高速だが、ベクター数、次元数、メタデータ、グラフオーバーヘッドが増えるにつれて高価になる可能性がある。ディスクベースの代替手段（Milvus DiskANN、Elasticsearch DiskBBQ、Turbopufferオブジェクトストレージ、LanceDBのLance形式）は、クエリレイテンシーを犠牲にしてインフラコストを削減する。大規模RAGインデックスでは、そのトレードオフはテストする価値があることが多い。

**Turbopufferのマルチテナンシー**は、非常に高いネームスペース数を中心に構築されている。公開されているポジショニングと顧客ストーリーは、Notionの大規模でネームスペースヘビーなコーパスのようなワークロードを強調している。各ユーザーまたは組織が分離されたベクター検索を必要とする場合、そのアーキテクチャは経済性を変える可能性があるが、独自のテナント形状でベンチマークすること。

**LanceDB組み込みモード**は「ベクター検索のためのSQLite」に最も近いものだ。インプロセスで実行され、サーバーを必要とせず、Lambda、Cloudflare Workers、エッジ環境で動作する。Lanceカラムナ形式により、実際のスケールで組み込み操作が現実的になる。

**Oramaは検索UXインフラストラクチャであって、ウェアハウスではない。** JavaScriptアプリ内、エッジ上、またはマネージドサイト/アプリ検索レイヤーとして、小さな全文/ベクター/ハイブリッド検索エンジンが必要な場合に優れている。10億ベクターの取得、重い分析、または複雑なフィルタリング結合のために私が選ぶツールではない。

**Vespaは、取得だけが製品の半分である場合に手を伸ばすものだ。** 字句取得、最近傍検索、テンソル、ランキング式、グルーピング、オンラインサービングを組み合わせる。その力は本物だが、運用およびモデリングの複雑さも同様だ。「CRUDアプリにセマンティック検索を追加する」よりも、検索/推薦チームに適している。

**ClickHouseは、検索が分析に付属している場合に会話に入るべきだ。** 真実の情報源がイベント、ログ、トレース、メトリクス、または大きなファクトテーブルである場合、ClickHouseはベクター距離、フィルタリング、集約、そして現在深刻な全文インデックスを1つのSQLエンジンに維持できる。専用ベクターデータベースではないが、分析検索では最良の意味で退屈な答えになり得る。

**Chromaはdev/testと小規模アプリデプロイメントで最も強力だ。** 非常に大規模なコーパス、HA、ディスクヘビーな運用、または第一級のハイブリッド検索を目指している場合、プロトタイプをインフラストラクチャに昇格させる前に、プロダクション向けのストアを評価すること。

### 簡略化された意思決定

pgvectorを本当に超えた場合 — 通常はベンチマークがベクター数、フィルタリング、書き込みレート、または高同時実行レイテンシーがPostgresの封筒を押し出していることを示す場合 — 制約によって選択する:

- **テナントごとの分離を持つSaaS製品** → Turbopuffer
- **Rustレベルのパフォーマンス + 複雑なメタデータフィルタリングが必要** → Qdrant
- **すでにElastic/ELKスタックを使用** → DiskBBQ付きElasticsearch
- **オープンソースを望むAWSショップ** → OpenSearch
- **深刻なランキングニーズを持つ検索/推薦プラットフォーム** → Vespa
- **分析、オブザーバビリティ、またはログ/イベント検索** → ClickHouse
- **10億スケールのオンプレミス / セルフホスト** → Milvus
- **エッジ / サーバーレス / マルチモーダル** → LanceDB
- **小規模JSアプリ、ドキュメントサイト、またはエッジネイティブ検索UX** → Orama
- **オペレーション不要、ただ動く、コストは二次的** → Pinecone
- **マルチモーダルファースト（画像、動画、音声）** → Marqo
- **すでにMongoDBを使用** → Atlas Vector Search
- **すでにPostgresを使用、より多くのヘッドルームが必要** → Supabase VectorまたはNeon（どちらもpgvectorマネージド、より良いツール付き）

## やらないこと一つ

正しい答えがあるもののファジーテキスト検索にベクター検索を使わないこと。

「メールアドレス`dan@example.com`のユーザーを見つけてくれ」はベクター検索の問題ではない。「ID`ORD-12345`の注文を見つけてくれ」も同様だ。`ORD-12345`を埋め込んで注文テーブルに対してコサイン類似性を行うと*何か*が返ってくるが、間違っている可能性がある。これらは正確一致の問題だ。主キーまたは通常のインデックスを使用すること。

ベクター検索は、何も関連性がない場合でも、データセット内で*最も類似した*ものを返す。関連ドキュメントにはそれで良い。しかし、特定のレコード検索では、間違ったニアマッチは空の結果よりも悪いので壊滅的だ。

各ツールの用途を知ること。それらのほとんどはすでにPostgresインストールに含まれている。 fitting する場所で使うこと。
