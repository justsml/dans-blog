AI फीचर्स जोड़ने वाली टीमें अक्सर सबसे पहले एक समर्पित vector database की तरफ देखती हैं।

Pinecone, Weaviate, Qdrant, Chroma। नई सर्विस, नई dependency, नया connection pool, नया बिल, और अब दो sources of truth को सिंक रखना।

इसी बीच, आपके पास पहले से PostgreSQL है। आपके पास पहले से `pgvector` है। और इसमें 2008 से built-in excellent full-text search भी है।

Dedicated vector stores बड़े scale और high query volume पर अपनी कीमत साबित करते हैं। लेकिन ज़्यादातर applications पहले search system को अच्छे से push करने से पहले ही दूसरे की तरफ बढ़ जाते हैं। ऐसे में future की scaling problem आज की synchronization bug बन जाती है।

तो: FTS कब इस्तेमाल करें, pgvector कब, और दोनों कब?

---

## हर एक असल में क्या करता है

Full-text search (`tsvector` / `GIN` index) lexical है। यह text को lexemes में tokenize करता है, उन्हें stem करता है, और queries को index के ख़िलाफ़ match करता है। "Running" और "runs" एक ही lexeme में collapse हो जाते हैं। "dog" और "dogs" भी। Ranking function (`ts_rank`) उन documents को ज़्यादा अंक देता है जहाँ query terms बार-बार या prominent जगह पर आते हैं।

pgvector semantic है। यह एक dense vector store करता है — numbers की एक list — जो एक embedding model द्वारा समझे गए chunk के *meaning* को represent करती है। Similarity search उस high-dimensional space में nearby vectors ढूंढता है। "Dog" और "canine" एक-दूसरे के करीब आ जाते हैं। "Running" (खेल) और "running" (process चलाना) शायद नहीं।

Practical difference: FTS जवाब देता है "किन documents में ये words हैं?" Vector search जवाब देता है "किन documents का मतलब लगभग यही है?"

![एक search tool map जो pg_trgm को short fuzzy strings के लिए, full-text search को exact prose queries के लिए, pgvector को semantic matching के लिए, और hybrid search को long content के लिए दिखाता है जिसे exact और semantic दोनों signals चाहिए।](../search-tool-map.svg)

_पहला split "पुराना search vs. AI search" नहीं है। यह text की shape और सही जवाब किस तरह का होगा, इस पर निर्भर करता है।_

## जब Full-Text Search जीतता है

**आप ऐसे terms search कर रहे हैं जो exactly match होने चाहिए।** Product SKUs, error codes, model numbers, usernames, legal clause references। `SKU-AX-44192` किसी से semantically similar नहीं है। यह या तो match करता है या नहीं। Vector search भरोसे से `SKU-AX-44193` return कर सकता है। यह वो नहीं है जो आप चाहते।

**आपकी queries keyword-based हैं।** Users search box में type करते हैं, tag से filter करते हैं, या keyword से blog posts search करते हैं। FTS उस shape के intent के लिए बना है।

**आपको GPU या embedding infrastructure के बिना ranked results चाहिए।** FTS indexes fast, deterministic हैं, और किसी external API call की ज़रूरत नहीं। एक `tsvector` column add करें, GIN index बनाएं, और हो गया।

**आप search के साथ boolean filtering कर रहे हैं।** `WHERE to_tsvector(body) @@ to_tsquery('postgres') AND category = 'tutorial' AND published_at > NOW() - INTERVAL '6 months'` — यह आपके existing query logic के साथ naturally compose होता है।

```sql
-- Create the index
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body, '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Query
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`GENERATED ALWAYS AS` column index को automatically updated रखता है। `setweight` title matches को body matches से ज़्यादा rank देता है। बस इतना ही setup है।

## जब Trigrams जीतते हैं (pg_trgm)

एक third Postgres tool है जिसे अक्सर छोड़ दिया जाता है: `pg_trgm`। यह न full-text search है न vector search। यह fuzzy string matching है, और उस awkward middle को cover करता है जिसे दोनों tools ठीक से handle नहीं कर पाते।

**Use case: names, addresses, identifiers, और typos वाले short strings।**

FTS text को lexemes में tokenize और stem करता है। यह prose के लिए काम करता है, लेकिन इनके लिए बुरा fit है:
- Person names ("Dan Levy" → language config पर depend करके "dan levi", "leiv" में stem हो सकता है)
- Company names, addresses, product titles जहाँ exact spelling matter करता है
- Typos वाले queries — "Micheal Jordan", "Amaon", "javascipt"
- Autocomplete / prefix search
- Partial string matching ("son" से "Johnson", "Anderson" match करना)

pgvector भी यहाँ सही choice नहीं है। आप "Micheal Jordan" embed कर सकते हैं और nearest vector ढूंढ सकते हैं, लेकिन embedding space names को meaning से organize करता है, spelling से नहीं। Nearest neighbor "basketball legend" या "Michael B. Jordan" हो सकता है, typo वाले user record नहीं।

`pg_trgm` strings को overlapping 3-character slices में तोड़ता है और measure करता है कि दो strings कितने trigrams share करते हैं। "Dan" → `" da"`, `"dan"`, `"an "`. "Micheal" और "Michael" अपने ज़्यादातर trigrams share करते हैं, इसलिए similarity high है।

```sql
-- Enable the extension (usually already available)
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- GIN index on names column — enables fast trigram similarity search
CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- Fuzzy name search: finds "Micheal Jordan" when searching "Michael Jordan"
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % operator = similarity threshold (default 0.3)
ORDER BY score DESC
LIMIT 10;

-- Or use ILIKE with trigram index support for contains matching
SELECT id, name
FROM users
WHERE name ILIKE '%johnson%'   -- GIN index makes this fast
LIMIT 10;
```

`%` operator `pg_trgm.similarity_threshold` (default 0.3, range 0-1) use करता है। Higher values closer matches मांगते हैं। Name search के लिए 0.3-0.4 आमतौर पर सही है: typos पकड़ने के लिए काफ़ी permissive, noise से बचने के लिए काफ़ी strict।

**Trigrams prefix search और autocomplete में भी मदद करते हैं, ख़ास तौर पर जब autocomplete को typo tolerance या contains matching चाहिए:**

```sql
-- Autocomplete: prefix matching. For pure left-anchored prefixes,
-- compare trigram GIN against a B-tree pattern index on your data.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- More control: word_similarity for partial matches within longer strings
-- (useful when searching "Johnson" within "Andrew Johnson III")
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name          -- <% operator = word_similarity threshold
ORDER BY score DESC
LIMIT 10;
```

**`pg_trgm` कब चुनें FTS के बजाय:**

| Scenario | Use |
|---|---|
| Person/company name search with typos | `pg_trgm` |
| Autocomplete / prefix search | `pg_trgm` (या prefix queries के साथ FTS) |
| Searching short strings, codes, identifiers | `pg_trgm` |
| Searching prose articles, documentation | FTS |
| Searching log messages for keywords | FTS |
| Multilingual name search | `pg_trgm` (यह language-agnostic है) |

`pg_trgm` FTS के साथ compose भी होता है। Trigrams को fuzzy pre-filter के लिए use करें और `ts_rank` से rank करें, या trigram similarity को vector score के साथ combine करें।

## जब pgvector जीतता है

**आप RAG बना रहे हैं।** RAG semantic retrieval पर depend करता है: उन document *chunks* को ढूंढना जिनका meaning user के question के closest है, भले ही wording अलग हो। Vector search इसके लिए purpose-built है। FTS paraphrases, synonyms, और conceptual matches miss कर देगा।

**Users बताते हैं कि उन्हें क्या चाहिए, search क्या करना है नहीं।** "Something light for a summer evening" के कोई obvious wine keywords नहीं हैं। "Articles about building confidence as a new manager" को semantic understanding चाहिए जो FTS provide नहीं कर सकता।

**आप similar items ढूंढ रहे हैं।** Related products, similar support tickets, duplicate bug reports। "Find me issues similar to this one" एक vector operation है। आप new issue को embed करते हैं और उसके nearest neighbors ढूंढते हैं।

**Multilingual content।** Multilingual data पर trained vector embeddings languages के across match कर सकते हैं। FTS को language-specific configurations चाहिए और cross-language queries को poorly handle करता है।

```sql
-- Setup
CREATE EXTENSION IF NOT EXISTS vector;

ALTER TABLE documents ADD COLUMN embedding vector(1536);
CREATE INDEX documents_embedding_idx
  ON documents USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);

-- Query: semantic search
SELECT id, title, 1 - (embedding <=> $1::vector) AS similarity
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 10;
```

Note: `ivfflat` approximate है — fast है लेकिन speed के बदले कुछ recall trade करता है। Smaller datasets (~1M rows से कम) के लिए `hnsw` अक्सर बेहतर है:

```sql
CREATE INDEX documents_embedding_idx
  ON documents USING hnsw (embedding vector_cosine_ops);
```

## जब दोनों की ज़रूरत हो

Technical documentation वो जगह है जहाँ simple split टूटता है। Users "how to configure timeouts" search करते हैं, लेकिन `withRetry()` जैसे function names और `ECONNRESET` जैसे error codes भी search करते हैं।

Vector search conceptual queries handle करता है। FTS exact terms handle करता है। अकेले दोनों में से कोई भी दोनों को अच्छे से handle नहीं कर पाता।

Solution hybrid search है: दोनों run करें और results को fuse करें।

**Reciprocal Rank Fusion (RRF)** यहाँ standard algorithm है। इसके लिए दो systems के scores को normalize करने की ज़रूरत नहीं; यह rank positions को combine करता है।

```sql
-- Hybrid search with Reciprocal Rank Fusion
WITH fts_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM documents, to_tsquery('english', $1) query
  WHERE search_vector @@ query
  LIMIT 50
),
vector_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> $2::vector) AS rank
  FROM documents
  ORDER BY embedding <=> $2::vector
  LIMIT 50
),
rrf AS (
  SELECT
    COALESCE(f.id, v.id) AS id,
    COALESCE(1.0 / (60 + f.rank), 0) + COALESCE(1.0 / (60 + v.rank), 0) AS rrf_score
  FROM fts_results f
  FULL OUTER JOIN vector_results v ON f.id = v.id
)
SELECT d.id, d.title, rrf.rrf_score
FROM rrf
JOIN documents d ON d.id = rrf.id
ORDER BY rrf_score DESC
LIMIT 10;
```

Denominator में `60` RRF constant है — higher values rank differences के influence को कम करते हैं, lower values amplify करते हैं। Default 60 ज़्यादातर cases में अच्छा काम करता है।

यह एक query में दो searches run करता है, ranks को fuse करता है, और उन results को reward करता है जहाँ keyword और semantic signals agree करते हैं।

![एक hybrid search pipeline जहाँ एक query full-text search और pgvector दोनों में fan out होता है, हर एक ranked results produce करता है, और Reciprocal Rank Fusion दोनों lists को combine करता है।](../hybrid-rrf-pipeline.svg)

_RRF इसलिए valuable है क्योंकि यह `ts_rank` और cosine distance को comparable raw scores मानने का नाटक नहीं करता। यह सिर्फ़ पूछता है, "यह result हर list में कितना ऊपर आया?"_

## Practical Decision Tree

Search strategy चुनते वक़्त, **input की shape** से शुरू करें, फिर पूछें **user किस तरह का query बना रहा है**। "Short string with spelling variation" वो problem नहीं है जो "long prose where exact terms matter" है, और दोनों "question over document chunks" से अलग हैं।

![एक decision tree जो pg_trgm, full-text search, pgvector, hybrid search, या dedicated vector database को text की shape और users के query करने के तरीके के basis पर चुनता है।](../search-decision-tree.svg)

वही tree words में:

- **Names, addresses, titles, autocomplete, या typo-prone short strings** → `pg_trgm`
- **Known words, error codes, SKUs, function names, tags, categories, filters** → FTS
- **Questions, paraphrases, recommendations, related items, multilingual match, RAG chunks** → pgvector
- **Technical content जहाँ users को exact symbols और conceptual answers दोनों चाहिए** → Hybrid with RRF
- **Primary keys, exact IDs, permission filters, date ranges, sorted lists** → normal SQL indexes
- **Huge vector volume, very high concurrency, या latency targets जो आपके benchmarks में Postgres hit नहीं कर पा रहा** → dedicated vector stores evaluate करें

### FTS vs. Semantic: Short Version

"FTS use करूं या vector search?" का सवाल आमतौर पर इस पर आ जाता है: **क्या आप जानते हैं कि relevant documents में कौन से words आएंगे?**

अगर हाँ — users known terms, categories, function names, product codes search कर रहे हैं — FTS faster, cheaper, और more predictable है। यह बताता है कि result क्यों match हुआ।

अगर नहीं — users concept describe कर रहे हैं, question पूछ रहे हैं, या different language में search कर रहे हैं — vector search सही tool है। यह words नहीं, meaning match करता है।

Tricky middle case natural language queries over technical content है। कोई "how do I handle connection drops" search कर रहा हो तो उसे "implementing retry logic for network failures" title वाला article चाहिए — कोई overlapping words नहीं, high semantic relevance। यहाँ vector search अपनी कीमत साबित करता है।

दूसरा tricky case **names और proper nouns** है। न FTS न vector search इनमें great हैं:
- FTS "Michael" search करते वक़्त "Micheal" miss कर देगा — different tokens
- Vector search name पूरी तरह miss कर सकता है अगर वह training data में frequently appear नहीं करता
- `pg_trgm` इसे सही से handle करता है: orthographic similarity, न semantic न lexical

Practice में, ज़्यादातर content-heavy search boxes को speed और keywords के लिए FTS चाहिए, और users names search करते हैं तो hybrid या `pg_trgm` की ज़रूरत पड़ सकती है। True semantic search feature का मतलब आमतौर पर pgvector है। RAG का मतलब हमेशा pgvector है।

## अगर Dedicated Vector Store की ज़रूरत पड़े

कुछ systems genuinely pgvector से बड़े हो जाते हैं। जब ऐसा होता है, market noisy है। यहाँ वो matter करता है जो top options में important है।

### Feature Matrix

Table को समझने से पहले कुछ columns का unpack ज़रूरी है।

**Hybrid search** का मतलब है BM25 keyword search और vector similarity एक query में run होते हैं, Reciprocal Rank Fusion से merged। "withRetry timeout" function name को exactly match कर सकता है *और* "retry logic for network failures" के documents semantically भी। Hybrid के बिना, आपको एक search mode चुनना पड़ता है या दो queries को खुद fuse करना पड़ता है। pgvector का "Manual (RRF via SQL)" [ऊपर दिखाया गया approach](#when-you-need-both) है: काम करता है, लेकिन आपको खुद लिखना पड़ता है।

**Sparse vectors** BM25 से आगे जाते हैं। SPLADE sparse vector में ~30,000 dimensions होते हैं (vocabulary term के लिए एक), ~98% zeros। Non-zero positions बताते हैं कि कौन से terms matter करते हैं और कितना। "dogs" query "canine" और "pet" को भी weight देती है: BM25-level keyword precision plus term expansion inside a vector index। अगर यह column false है, तो exact-term queries के लिए external FTS layer चाहिए।

```python
# SPLADE: ~30,000 dims total, ~60 non-zero — only the relevant vocabulary positions fire
def encode_splade(text: str) -> dict:
    tokens = tokenizer(text, return_tensors="pt", truncation=True, max_length=512)
    with torch.no_grad():
        output = model(**tokens)
    vec = torch.log1p(torch.relu(output.logits)).max(dim=1).values.squeeze()
    return {"indices": vec.nonzero().squeeze().tolist(), "values": vec[vec != 0].tolist()}
```

**SQL / SQL-like** really filtering के बारे में है। Filtering के बिना vector search demo है, application नहीं: आपको अभी भी tenant scope, date ranges, permissions, और categories चाहिए। Full SQL (pgvector) उसे आपके existing joins के बगल में express करता है। Purpose-built DBs JSON filter objects (Qdrant, Pinecone), query DSL (Elasticsearch, Milvus), या GraphQL (Weaviate) use करते हैं। काम करते हैं; SQL ज़्यादा attractive होता है जब filter logic उलझता है।

```sql
-- pgvector: vector similarity is just another expression in WHERE
SELECT id, title, 1 - (embedding <=> $1) AS score
FROM documents
WHERE tenant_id = $2 AND category = ANY($3::text[]) AND created_at > NOW() - INTERVAL '90 days'
ORDER BY embedding <=> $1 LIMIT 10;
```

```python
# Qdrant: equivalent filter as a Python object — functional, more ceremony
results = client.query_points(
    collection_name="documents", query=query_embedding,
    query_filter=models.Filter(must=[
        models.FieldCondition(key="tenant_id", match=models.MatchValue(value=tenant_id)),
        models.FieldCondition(key="category",  match=models.MatchAny(any=categories)),
        models.FieldCondition(key="created_at", range=models.DatetimeRange(gte=cutoff)),
    ]),
    limit=10,
)
```

**Multimodal native** का मतलब "image embeddings store कर सकता है" नहीं है; हर database float arrays store कर सकता है। इसका मतलब है database non-text content के लिए embedding models ship करता है, ताकि आप उसे raw image URL दें और वह vectorization handle करे। ज़्यादातर databases यहाँ embedding-agnostic हैं, तो आपको वह pipeline खुद own करना पड़ता है। Marqo और Weaviate (CLIP/ImageBind modules के ज़रिए) loop close करते हैं।

```python
# Marqo: POST raw images, query with text — no external embedding step needed
mq.index("products").add_documents(
    [{"id": "shoe-001", "image": "https://cdn.example.com/shoes/001.jpg"}],
    tensor_fields=["image"]
)
results = mq.index("products").search(q="lightweight shoes for summer")
# Returns shoe-001 despite zero keyword overlap — CLIP handles the cross-modal match
```

**Disk-based index** एक cost question है। RAM-resident HNSW को प्रति million 1536-dimension vectors कई GB RAM लग सकती है — raw vectors, graph overhead, और metadata count करने के बाद। Disk-native alternatives (Milvus DiskANN, Elasticsearch DiskBBQ, LanceDB का Lance format, Turbopuffer का object storage tier) अक्सर query latency के बदले cheaper infrastructure trade करते हैं। RAG workloads के जहाँ model latency already dominate करती है, benchmark करना अक्सर worth it है। Redis VSS hard constraint है: RAM-only, कोई disk path नहीं।

**Max dimensions** tomorrow's migration है जो आज की choice में छुपा है। `text-embedding-3-large` 3072 dims use करता है, Jina v3 बड़े embeddings emit कर सकता है, और research models higher push करते जा रहे हैं। कुछ managed services hard dimension caps publish करते हैं; दूसरे high caps या typical embedding models के लिए कोई practical cap document नहीं करते। Commit करने से पहले current docs check करें। Headroom वाला कुछ चुनें; dimension ceiling hit करने के बाद vector index migrate करना miserable sprint है।

### Reading the Matrix

कुछ चीज़ें table में cleanly fit नहीं होतीं:

**Sparse vectors** वो तरीका है जिससे आपको separate full-text engine के बिना vector index के अंदर BM25-quality keyword matching मिलती है। Qdrant और Elasticsearch के implementations यहाँ ख़ास mature हैं। Weaviate BM25F के ज़रिए support करता है। अगर hybrid search critical है और दो systems run नहीं कर सकते, तो sparse vector support देखें।

**Disk-based indexes** cost lever हैं, implementation detail नहीं। RAM-resident HNSW indexes fast हैं लेकिन vector count, dimension count, metadata, और graph overhead बढ़ने के साथ expensive हो सकते हैं। Disk-based alternatives (Milvus DiskANN, Elasticsearch DiskBBQ, Turbopuffer object storage, LanceDB का Lance format) query latency के बदले lower infrastructure cost trade करते हैं। बड़े RAG indexes के लिए, यह tradeoff अक्सर test करने लायक है।

**Turbopuffer का multi-tenancy** बहुत high namespace counts के around बना है। इसकी public positioning और customer stories Notion के large, namespace-heavy corpus जैसे workloads पर ज़ोर देती हैं। अगर हर user या organization को isolated vector search चाहिए, तो वह architecture economics बदल सकता है, लेकिन अपना tenant shape खुद benchmark करें।

**LanceDB embedded mode** "vector search के लिए SQLite" के सबसे करीब है। यह in-process run करता है, किसी server की ज़रूरत नहीं, और Lambda, Cloudflare Workers, और edge environments में काम करता है। Lance columnar format embedded operation को real scale पर practical बनाता है।

**Orama search UX infrastructure है, warehouse नहीं।** यह तब excellent है जब आपको JavaScript app में, edge पर, या managed site/app search layer के लिए tiny full-text/vector/hybrid search engine चाहिए। यह billion-vector retrieval, heavy analytics, या complex filtered joins के लिए वो tool नहीं है जो मैं चुनूंगा।

**Vespa वो है जिसकी तरफ आप तब देखते हैं जब retrieval सिर्फ़ आधा product है।** यह lexical retrieval, nearest-neighbor search, tensors, ranking expressions, grouping, और online serving को combine करता है। उस power real है, लेकिन operational और modeling complexity भी। यह search/recommendation teams के लिए fit है, "add semantic search to my CRUD app" के लिए नहीं।

**ClickHouse conversation में तब आता है जब search analytics से attached है।** अगर आपका source of truth events, logs, traces, metrics, या large fact tables हैं, तो ClickHouse vector distance, filtering, aggregation, और अब serious full-text indexing को एक SQL engine में रख सकता है। यह purpose-built vector database नहीं है, लेकिन analytical retrieval के लिए best way में boring answer हो सकता है।

**Chroma dev/test और small app deployments में सबसे strong है।** अगर आप very large corpora, HA, disk-heavy operation, या first-class hybrid search की तरफ बढ़ रहे हैं, तो prototype को infrastructure में promote करने से पहले production-oriented store evaluate करें।

### Simplified Decision

अगर आपने genuinely pgvector से बड़ा हो गए हैं — आमतौर पर क्योंकि benchmarks दिखाते हैं कि vector count, filtering, write rate, या high-concurrency latency आपके Postgres envelope से बाहर push कर रहे हैं — तो constraint के basis पर चुनें:

- **SaaS product with per-tenant isolation** → Turbopuffer
- **Rust-level performance + complex metadata filtering चाहिए** → Qdrant
- **Already on Elastic/ELK stack** → Elasticsearch with DiskBBQ
- **AWS shop जो open-source चाहता है** → OpenSearch
- **Search/recommendation platform with serious ranking needs** → Vespa
- **Analytics, observability, या log/event search** → ClickHouse
- **Billion-scale on-prem / self-hosted** → Milvus
- **Edge / serverless / multimodal** → LanceDB
- **Small JS app, docs site, या edge-native search UX** → Orama
- **Zero ops, just works, cost secondary है** → Pinecone
- **Multimodal-first (images, video, audio)** → Marqo
- **Already on MongoDB** → Atlas Vector Search
- **Already on Postgres, ज़्यादा headroom चाहिए** → Supabase Vector या Neon (दोनों pgvector managed, बेहतर tooling के साथ)

## एक चीज़ जो नहीं करनी

Fuzzy text search के लिए vector search use न करें उन चीज़ों के लिए जिनके correct answers हैं।

"Find me the user with email `dan@example.com`" vector search problem नहीं है। "Find me the order with ID `ORD-12345`" भी नहीं। `ORD-12345` embed करना और cosine similarity से orders table के ख़िलाफ़ match करना *कुछ* return करेगा, लेकिन ग़लत हो सकता है। ये exact-match problems हैं। Primary key या regular index use करें।

Vector search dataset में *most similar* चीज़ return करता है, तब भी जब कुछ relevant न हो। यह नहीं जानता कि कोई good answer नहीं है। Related documents के लिए यह ठीक है। Specific record lookup के लिए catastrophic है, जहाँ ग़लत near-match empty result से बदतर है।

पता हो कि हर tool किस काम के लिए है। ज़्यादातर आपके Postgres installation में पहले से मौजूद हैं। जहाँ fit हों, वहाँ use करें।
