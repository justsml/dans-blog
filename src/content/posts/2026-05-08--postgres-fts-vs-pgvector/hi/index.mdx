AI फीचर्स जोड़ने वाली टीमें अक्सर सबसे पहले एक समर्पित वेक्टर डेटाबेस की तरफ बढ़ती हैं।

Pinecone, Weaviate, Qdrant, Chroma। नई सर्विस, नई डिपेंडेंसी, नया कनेक्शन पूल, नया बिल, और अब दो sources of truth को ईमानदार रखना।

इसी बीच, उनके पास PostgreSQL पहले से मौजूद है। PostgreSQL में `pgvector` पहले से है। 2008 से इसमें बेहतरीन full-text search भी बिल्ट-इन है।

समर्पित वेक्टर स्टोर्स बड़े स्केल और हाई क्वेरी वॉल्यूम पर अपनी कीमत साबित करते हैं। लेकिन ज़्यादातर ऐप्लिकेशन पहले सिस्टम को ज़्यादा टेस्ट करने से पहले ही दूसरे सर्च सिस्टम की तरफ बढ़ जाते हैं। यही वजह है कि भविष्य की स्केलिंग समस्या आज के synchronization bug में बदल जाती है।

तो: FTS कब उपयोग करें, pgvector कब उपयोग करें, और दोनों कब उपयोग करें?

---

## हर एक वास्तव में क्या करता है

Full-text search (`tsvector` / `GIN` इंडेक्स) lexical है। यह टेक्स्ट को lexemes में tokenize करता है, उन्हें stem करता है, और क्वेरी को इंडेक्स से मैच करता है। "Running" और "runs" एक ही lexeme में collapse हो जाते हैं। "dog" और "dogs" भी ऐसा ही करते हैं। रैंकिंग फंक्शन (`ts_rank`) उन डॉक्यूमेंट्स को इनाम देता है जहाँ क्वेरी टर्म्स बार-बार या प्रमुख स्थान पर आती हैं।

pgvector semantic है। यह एक dense vector — नंबरों की एक सूची — स्टोर करता है, जो embedding model द्वारा समझे गए chunk के *अर्थ* को दर्शाता है। समानता खोज उस उच्च-आयामी स्थान में निकटतम वेक्टर ढूंढती है। "Dog" और "canine" एक-दूसरे के पास आ जाते हैं। "Running" (खेल) और "running" (प्रोसेस) शायद नहीं।

व्यावहारिक अंतर: FTS जवाब देता है "किन डॉक्यूमेंट्स में ये शब्द हैं?" वेक्टर खोज जवाब देती है "किन डॉक्यूमेंट्स का अर्थ लगभग यह है?"

![एक सर्च टूल मैप जो छोटे fuzzy strings के लिए pg_trgm, exact prose queries के लिए full-text search, semantic matching के लिए pgvector, और लंबी सामग्री के लिए hybrid search (exact और semantic दोनों सिग्नल) दिखाता है।](../search-tool-map.svg)

_पहला विभाजक "पुरानी खोज बनाम AI खोज" नहीं है। यह टेक्स्ट का आकार है और किस तरह का जवाब सही होगा।_

## जब Full-Text Search जीतता है

**आप ऐसे टर्म्स खोज रहे हैं जो बिल्कुल सही होने चाहिए।** प्रोडक्ट SKU, एरर कोड, मॉडल नंबर, यूज़रनेम, लीगल क्लॉज़ रेफ़रेंस। `SKU-AX-44192` किसी भी चीज़ से semantically similar नहीं है। यह या तो मैच करता है या नहीं। वेक्टर खोज आत्मविश्वास से `SKU-AX-44193` लौटा सकती है। यह वह नहीं है जो आप चाहते हैं।

**आपकी क्वेरीज़ keyword-आधारित हैं।** यूज़र्स सर्च बॉक्स में टाइप करते हैं, टैग से फ़िल्टर करते हैं, या कीवर्ड से ब्लॉग पोस्ट खोजते हैं। FTS उस intent के लिए बनाया गया है।

**आपको GPU या embedding infrastructure के बिना ranked results चाहिए।** FTS इंडेक्स तेज़, deterministic हैं, और बाहरी API कॉल की ज़रूरत नहीं है। एक `tsvector` कॉलम जोड़ें, GIN इंडेक्स बनाएं, और काम पूरा।

**आप सर्च के साथ boolean filtering कर रहे हैं।** `WHERE to_tsvector(body) @@ to_tsquery('postgres') AND category = 'tutorial' AND published_at > NOW() - INTERVAL '6 months'` — यह आपकी मौजूदा क्वेरी लॉजिक के साथ स्वाभाविक रूप से काम करता है।

```sql
-- Create the index
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body, '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Query
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`GENERATED ALWAYS AS` कॉलम इंडेक्स को स्वचालित रूप से अपडेट रखता है। `setweight` टाइटल मैच को बॉडी मैच से ऊँचा रैंक देता है। बस इतना ही सेटअप है।

## जब Trigrams जीतते हैं (pg_trgm)

एक तीसरा Postgres टूल है जिसे अक्सर छोड़ दिया जाता है: `pg_trgm`। यह न तो full-text search है और न ही वेक्टर खोज। यह fuzzy string matching है, और उस awkward middle को कवर करता है जिसे दोनों टूल्स ठीक से हैंडल नहीं करते।

**उपयोग का मामला: नाम, पते, आइडेंटिफ़ायर, और typos वाले छोटे strings।**

FTS टेक्स्ट को lexemes में tokenize और stem करता है। यह prose के लिए काम करता है, लेकिन इनके लिए ख़राब है:
- व्यक्ति के नाम ("Dan Levy" → "dan levi", "leiv" में stem, भाषा कॉन्फ़िगरेशन पर निर्भर)
- कंपनी के नाम, पते, प्रोडक्ट टाइटल जहाँ सही spelling मायने रखती है
- Typos वाली क्वेरीज़ — "Micheal Jordan", "Amaon", "javascipt"
- Autocomplete / prefix खोज
- आंशिक string मिलान ("son" का "Johnson", "Anderson" से मिलान)

pgvector भी यहाँ ख़राब विकल्प है। आप "Micheal Jordan" को embed करके निकटतम वेक्टर ढूंढ सकते हैं, लेकिन embedding space नामों को अर्थ से व्यवस्थित करता है, spelling से नहीं। निकटतम neighbor "basketball legend" या "Michael B. Jordan" हो सकता है, typo वाले यूज़र रिकॉर्ड नहीं।

`pg_trgm` strings को overlapping 3-character slices में तोड़ता है और मापता है कि दो strings कितने trigrams साझा करते हैं। "Dan" → `" da"`, `"dan"`, `"an "`। "Micheal" और "Michael" अपने ज़्यादातर trigrams साझा करते हैं, इसलिए समानता उच्च है।

```sql
-- Enable the extension (usually already available)
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- GIN index on names column — enables fast trigram similarity search
CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- Fuzzy name search: finds "Micheal Jordan" when searching "Michael Jordan"
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % operator = similarity threshold (default 0.3)
ORDER BY score DESC
LIMIT 10;

-- Or use ILIKE with trigram index support for contains matching
SELECT id, name
FROM users
WHERE name ILIKE '%johnson%'   -- GIN index makes this fast
LIMIT 10;
```

`%` ऑपरेटर `pg_trgm.similarity_threshold` (डिफ़ॉल्ट 0.3, रेंज 0-1) का उपयोग करता है। उच्च मान करीब के मैच की माँग करते हैं। नाम खोज के लिए 0.3-0.4 आमतौर पर सही है: typos पकड़ने के लिए पर्याप्त उदार, शोर से बचने के लिए पर्याप्त सख़्त।

**Trigrams prefix खोज और autocomplete में भी मदद करते हैं, ख़ास तब जब autocomplete को typo tolerance या contains matching की ज़रूरत हो:**

```sql
-- Autocomplete: prefix matching. For pure left-anchored prefixes,
-- compare trigram GIN against a B-tree pattern index on your data.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- More control: word_similarity for partial matches within longer strings
-- (useful when searching "Johnson" within "Andrew Johnson III")
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name          -- <% operator = word_similarity threshold
ORDER BY score DESC
LIMIT 10;
```

**FTS के ऊपर `pg_trgm` कब चुनें:**

| परिदृश्य | उपयोग |
|---|---|
| Typos के साथ व्यक्ति/कंपनी नाम खोज | `pg_trgm` |
| Autocomplete / prefix खोज | `pg_trgm` (या prefix queries के साथ FTS) |
| छोटे strings, कोड, आइडेंटिफ़ायर खोज | `pg_trgm` |
| Prose आर्टिकल्स, दस्तावेज़ीकरण खोज | FTS |
| लॉग मैसेज में कीवर्ड खोज | FTS |
| बहुभाषी नाम खोज | `pg_trgm` (भाषा-स्वतंत्र) |

`pg_trgm` FTS के साथ भी काम करता है। fuzzy pre-filter के लिए trigrams उपयोग करें और `ts_rank` से रैंक करें, या trigram समानता को वेक्टर स्कोर के साथ जोड़ें।

## जब pgvector जीतता है

**आप RAG बना रहे हैं।** RAG semantic retrieval पर निर्भर करता है: ऐसे document *chunks* ढूंढें जिनका अर्थ यूज़र के सवाल के सबसे करीब हो, भले ही शब्दावली अलग हो। वेक्टर खोज इसके लिए purpose-built है। FTS paraphrases, synonyms, और conceptual matches को छोड़ देगा।

**यूज़र्स बताते हैं कि उन्हें क्या चाहिए, न कि क्या खोजना है।** "किसी गर्मियों की शाम के लिए हल्का" — इसमें कोई स्पष्ट वाइन कीवर्ड नहीं है। "नए मैनेजर के रूप में आत्मविश्वास बनाने के बारे में आर्टिकल्स" — इसके लिए semantic understanding चाहिए जो FTS नहीं दे सकता।

**आप similar items ढूंढ रहे हैं।** संबंधित प्रोडक्ट्स, समान सपोर्ट टिकट, डुप्लीकेट बग रिपोर्ट। "मुझे इस जैसे issues ढूंढकर दें" एक वेक्टर ऑपरेशन है। आप नए issue को embed करते हैं और उसके nearest neighbors ढूंढते हैं।

**बहुभाषी सामग्री।** बहुभाषी डेटा पर ट्रेन किए गए वेक्टर embeddings भाषाओं के बीच मैच कर सकते हैं। FTS को भाषा-विशिष्ट कॉन्फ़िगरेशन की ज़रूरत होती है और cross-language क्वेरीज़ को ख़राब हैंडल करता है।

```sql
-- Setup
CREATE EXTENSION IF NOT EXISTS vector;

ALTER TABLE documents ADD COLUMN embedding vector(1536);
CREATE INDEX documents_embedding_idx
  ON documents USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);

-- Query: semantic search
SELECT id, title, 1 - (embedding <=> $1::vector) AS similarity
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 10;
```

नोट: `ivfflat` approximate है — यह तेज़ है लेकिन गति के लिए कुछ recall की कुर्बानी देता है। छोटे डेटासेट (~1M rows से कम) के लिए `hnsw` अक्सर बेहतर होता है:

```sql
CREATE INDEX documents_embedding_idx
  ON documents USING hnsw (embedding vector_cosine_ops);
```

## जब दोनों की ज़रूरत हो

तकनीकी दस्तावेज़ीकरण वह जगह है जहाँ सरल विभाजन टूटता है। यूज़र्स "how to configure timeouts" खोजते हैं, लेकिन वे `withRetry()` जैसे फंक्शन नाम और `ECONNRESET` जैसे एरर कोड भी खोजते हैं।

वेक्टर खोज conceptual queries हैंडल करती है। FTS exact terms हैंडल करता है। अकेले कोई भी दोनों को अच्छे से हैंडल नहीं करता।

समाधान है hybrid search: दोनों चलाएं और परिणामों को fuse करें।

**Reciprocal Rank Fusion (RRF)** यहाँ मानक एल्गोरिदम है। इसके लिए दो सिस्टम के स्कोर को normalize करने की ज़रूरत नहीं है; यह रैंक पोज़िशन को जोड़ता है।

```sql
-- Hybrid search with Reciprocal Rank Fusion
WITH fts_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM documents, to_tsquery('english', $1) query
  WHERE search_vector @@ query
  LIMIT 50
),
vector_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> $2::vector) AS rank
  FROM documents
  ORDER BY embedding <=> $2::vector
  LIMIT 50
),
rrf AS (
  SELECT
    COALESCE(f.id, v.id) AS id,
    COALESCE(1.0 / (60 + f.rank), 0) + COALESCE(1.0 / (60 + v.rank), 0) AS rrf_score
  FROM fts_results f
  FULL OUTER JOIN vector_results v ON f.id = v.id
)
SELECT d.id, d.title, rrf.rrf_score
FROM rrf
JOIN documents d ON d.id = rrf.id
ORDER BY rrf_score DESC
LIMIT 10;
```

हर में `60` RRF constant है — उच्च मान रैंक अंतर के प्रभाव को कम करते हैं, निम्न मान उन्हें बढ़ाते हैं। 60 का डिफ़ॉल्ट ज़्यादातर मामलों में अच्छा काम करता है।

यह एक क्वेरी में दो खोज चलाता है, रैंकों को fuse करता है, और उन परिणामों को इनाम देता है जहाँ keyword और semantic सिग्नल सहमत होते हैं।

![एक hybrid search pipeline जहाँ एक क्वेरी full-text search और pgvector दोनों में जाती है, हर एक ranked परिणाम देता है, और Reciprocal Rank Fusion दोनों सूचियों को जोड़ता है।](../hybrid-rrf-pipeline.svg)

_RRF इसलिए मूल्यवान है क्योंकि यह `ts_rank` और cosine distance को comparable raw scores मानने का नाटक करने से बचता है। यह केवल पूछता है, "यह परिणाम हर सूची में कितना ऊपर आया?"_

## व्यावहारिक निर्णय वृक्ष

खोज रणनीति चुनते समय, **इनपुट के आकार** से शुरू करें, फिर पूछें **यूज़र किस तरह की क्वेरी बना रहा है**। "Spelling variation वाला छोटा string" उतनी ही समस्या नहीं है जितनी "लंबी prose जहाँ exact terms मायने रखते हैं," और दोनों "document chunks पर सवाल" से अलग हैं।

![एक निर्णय वृक्ष जो टेक्स्ट के आकार और यूज़र्स के क्वेरी करने के तरीके के आधार पर pg_trgm, full-text search, pgvector, hybrid search, या समर्पित वेक्टर डेटाबेस चुनता है।](../search-decision-tree.svg)

शब्दों में वही ट्री:

- **नाम, पते, टाइटल, autocomplete, या typo-prone छोटे strings** → `pg_trgm`
- **ज्ञात शब्द, एरर कोड, SKU, फंक्शन नाम, टैग, कैटेगरी, फ़िल्टर** → FTS
- **सवाल, paraphrases, अनुशंसाएँ, संबंधित items, बहुभाषी मिलान, RAG chunks** → pgvector
- **तकनीकी सामग्री जहाँ यूज़र्स को exact symbols और conceptual दोनों जवाब चाहिए** → Hybrid with RRF
- **Primary keys, exact IDs, permission फ़िल्टर, date ranges, sorted lists** → सामान्य SQL इंडेक्स
- **बहुत बड़ा वेक्टर वॉल्यूम, बहुत उच्च concurrency, या वे latency targets जो Postgres आपके benchmarks में हिट नहीं कर सकता** → समर्पित वेक्टर स्टोर्स का मूल्यांकन करें

### FTS बनाम Semantic: संक्षिप्त संस्करण

"मुझे FTS उपयोग करना चाहिए या वेक्टर खोज?" का सवाल आमतौर पर इस पर आता है: **क्या आप जानते हैं कि संबंधित डॉक्यूमेंट्स में कौन से शब्द आएंगे?**

अगर हाँ — यूज़र्स ज्ञात टर्म्स, कैटेगरी, फंक्शन नाम, प्रोडक्ट कोड खोजते हैं — FTS तेज़, सस्ता, और अधिक अनुमानित है। यह बताता है कि परिणाम क्यों मैच हुआ।

अगर नहीं — यूज़र्स concept का वर्णन करते हैं, सवाल पूछते हैं, या अलग भाषा में खोजते हैं — वेक्टर खोज सही टूल है। यह शब्दों से नहीं, अर्थ से मैच करता है।

मुश्किल middle है तकनीकी सामग्री पर natural language क्वेरीज़। कोई "how do I handle connection drops" खोज रहा हो, उसे "implementing retry logic for network failures" शीर्षक वाला आर्टिकल चाहिए — कोई overlapping शब्द नहीं, उच्च semantic relevance। यहीं वेक्टर खोज अपनी कीमत साबित करती है।

दूसरा मुश्किल मामला है **नाम और proper nouns**। न तो FTS और न ही वेक्टर खोज इनमें अच्छे हैं:
- FTS "Michael" खोजते समय "Micheal" को छोड़ देगा — अलग tokens
- वेक्टर खोज नाम को पूरी तरह छोड़ देगा अगर यह ट्रेनिंग डेटा में बार-बार नहीं आता
- `pg_trgm` इसे सही तरीके से हैंडल करता है: orthographic समानता, semantic या lexical नहीं

व्यावहारिक रूप से, ज़्यादातर content-heavy सर्च बॉक्स को गति और कीवर्ड के लिए FTS चाहिए, और hybrid या `pg_trgm` की ज़रूरत हो सकती है — यह इस बात पर निर्भर करता है कि यूज़र्स नाम खोज रहे हैं या नहीं। असली semantic search फीचर आमतौर पर pgvector का मतलब है। RAG हमेशा pgvector का मतलब है।

## अगर आपको समर्पित वेक्टर स्टोर की ज़रूरत हो

कुछ सिस्टम वास्तव में pgvector से बड़े हो जाते हैं। जब ऐसा होता है, बाज़ार शोरगुल वाला होता है। यहाँ शीर्ष विकल्पों में वह चीज़ है जो मायने रखती है।

### फीचर मैट्रिक्स

टेबल को समझदार बनाने से पहले कुछ कॉलम को खोलने की ज़रूरत है।

**Hybrid search** का मतलब है BM25 keyword खोज और वेक्टर समानता एक क्वेरी में चलती हैं, Reciprocal Rank Fusion के माध्यम से merged। "withRetry timeout" फंक्शन नाम से बिल्कुल *और* "retry logic for network failures" के बारे में डॉक्यूमेंट्स से sematically मैच कर सकता है। Hybrid के बिना, आप एक खोज मोड चुनते हैं या दो क्वेरीज़ को खुद fuse करते हैं। pgvector का "Manual (RRF via SQL)" [ऊपर दिखाया गया दृष्टिकोण](#when-you-need-both) है: यह काम करता है, लेकिन आपको खुद लिखना होता है।

**Sparse vectors** BM25 से आगे जाते हैं। SPLADE sparse vector में ~30,000 dimensions (प्रति vocabulary term एक) होते हैं, ~98% zeros। गैर-शून्य पद बताते हैं कि कौन से टर्म्स मायने रखते हैं और कितना। "dogs" के लिए क्वेरी "canine" और "pet" को भी weight करती है: vector index के अंदर BM25-स्तर की keyword precision plus term expansion। अगर यह कॉलम false है, तो exact-term क्वेरीज़ के लिए आपको बाहरी FTS लेयर की ज़रूरत होगी।

```python
# SPLADE: ~30,000 dims total, ~60 non-zero — only the relevant vocabulary positions fire
def encode_splade(text: str) -> dict:
    tokens = tokenizer(text, return_tensors="pt", truncation=True, max_length=512)
    with torch.no_grad():
        output = model(**tokens)
    vec = torch.log1p(torch.relu(output.logits)).max(dim=1).values.squeeze()
    return {"indices": vec.nonzero().squeeze().tolist(), "values": vec[vec != 0].tolist()}
```

**SQL / SQL-like** वास्तव में filtering के बारे में है। Filtering के बिना वेक्टर खोज demo है, application नहीं: आपको अभी भी tenant scope, date ranges, permissions, और categories चाहिए। पूर्ण SQL (pgvector) इसे आपके मौजूदा joins के बगल में व्यक्त करता है। Purpose-built DB JSON filter objects (Qdrant, Pinecone), query DSL (Elasticsearch, Milvus), या GraphQL (Weaviate) उपयोग करते हैं। वे काम करते हैं; SQL अधिक आकर्षक होता जाता है जैसे-जैसे filter logic उलझता है।

```sql
-- pgvector: vector similarity is just another expression in WHERE
SELECT id, title, 1 - (embedding <=> $1) AS score
FROM documents
WHERE tenant_id = $2 AND category = ANY($3::text[]) AND created_at > NOW() - INTERVAL '90 days'
ORDER BY embedding <=> $1 LIMIT 10;
```

```python
# Qdrant: equivalent filter as a Python object — functional, more ceremony
results = client.query_points(
    collection_name="documents", query=query_embedding,
    query_filter=models.Filter(must=[
        models.FieldCondition(key="tenant_id", match=models.MatchValue(value=tenant_id)),
        models.FieldCondition(key="category",  match=models.MatchAny(any=categories)),
        models.FieldCondition(key="created_at", range=models.DatetimeRange(gte=cutoff)),
    ]),
    limit=10,
)
```

**Multimodal native** का मतलब "image embeddings स्टोर कर सकता है" नहीं है; हर डेटाबेस float arrays स्टोर करता है। इसका मतलब है कि डेटाबेस गैर-टेक्स्ट सामग्री के लिए embedding models के साथ आता है, ताकि आप इसे raw image URL दें और यह vectorization संभाल ले। यहाँ ज़्यादातर डेटाबेस embedding-agnostic हैं, इसलिए वह pipeline आपकी ज़िम्मेदारी है। Marqo और Weaviate (CLIP/ImageBind modules के माध्यम से) loop बंद करते हैं।

```python
# Marqo: POST raw images, query with text — no external embedding step needed
mq.index("products").add_documents(
    [{"id": "shoe-001", "image": "https://cdn.example.com/shoes/001.jpg"}],
    tensor_fields=["image"]
)
results = mq.index("products").search(q="lightweight shoes for summer")
# Returns shoe-001 despite zero keyword overlap — CLIP handles the cross-modal match
```

**Disk-based index** एक cost question है। RAM-resident HNSW को raw vectors, graph overhead, और metadata गिनने के बाद प्रति million 1536-dimension vectors कई GB RAM की ज़रूरत हो सकती है। Disk-native alternatives (Milvus DiskANN, Elasticsearch DiskBBQ, LanceDB का Lance format, Turbopuffer की object storage tier) अक्सर कुछ query latency को सस्ते infrastructure के लिए trade करते हैं। RAG workloads के लिए जहाँ model latency पहले से dominate करती है, यह benchmark करने लायक होता है। Redis VSS hard constraint है: RAM-only, कोई disk path नहीं।

**Max dimensions** कल का migration है जो आज के choice में छिपा है। `text-embedding-3-large` 3072 dims उपयोग करता है, Jina v3 बड़े embeddings emit कर सकता है, और research models लगातार ऊँचाई बढ़ा रहे हैं। कुछ managed services hard dimension caps publish करती हैं; अन्य उच्च caps या typical embedding models के लिए कोई practical cap नहीं document करतीं। Commit करने से पहले current docs जाँचें। कुछ headroom वाला चुनें; dimension ceiling hit करने के कारण vector index migrate करना एक दुखद sprint है।

_सार्वजनिक प्रोजेक्ट docs और प्रोडक्ट पेजों के आधार पर 8 मई, 2026 को अंतिम सत्यापन। मैट्रिक्स को snapshot मानें: managed-service limits, pricing, hybrid-search features, और disk-index options तेज़ी से बदलते हैं।_

### मैट्रिक्स को पढ़ना

कुछ चीज़ें टेबल में साफ़ फ़िट नहीं होतीं:

**Sparse vectors** वह तरीका है जिससे आप vector index के अंदर BM25-quality keyword matching पाते हैं, बिना किसी अलग full-text engine के। Qdrant और Elasticsearch में यहाँ ख़ास mature implementations हैं। Weaviate BM25F के माध्यम से उन्हें सपोर्ट करता है। अगर hybrid search critical है और आप दो सिस्टम नहीं चला सकते, तो sparse vector support ढूंढें।

**Disk-based indexes** एक cost lever हैं, implementation detail नहीं। RAM-resident HNSW indexes तेज़ हैं लेकिन vector count, dimension count, metadata, और graph overhead बढ़ने के साथ महंगे हो सकते हैं। Disk-based alternatives (Milvus DiskANN, Elasticsearch DiskBBQ, Turbopuffer object storage, LanceDB का Lance format) query latency को कम infrastructure cost के लिए trade करते हैं। बड़े RAG indexes के लिए, यह tradeoff अक्सर टेस्ट करने लायक होता है।

**Turbopuffer की multi-tenancy** बहुत उच्च namespace counts के इर्द-गिर्द बनी है। इसकी सार्वजनिक positioning और customer stories Notion के बड़े, namespace-heavy corpus जैसे workloads पर ज़ोर देती हैं। अगर हर यूज़र या संगठन को isolated vector search चाहिए, तो वह architecture economics बदल सकता है, लेकिन फिर भी अपने tenant shape को benchmark करें।

**LanceDB embedded mode** "वेक्टर खोज के लिए SQLite" के सबसे करीब है। यह in-process चलता है, किसी सर्वर की ज़रूरत नहीं है, और Lambda, Cloudflare Workers, और edge environments में काम करता है। Lance columnar format embedded operation को असल स्केल पर व्यावहारिक बनाता है।

**Orama search UX infrastructure है, warehouse नहीं।** यह तब उत्कृष्ट है जब आपको JavaScript app में, edge पर, या managed site/app search लेयर के रूप में एक छोटा full-text/vector/hybrid search engine चाहिए। यह वह टूल नहीं है जिसे मैं billion-vector retrieval, heavy analytics, या complex filtered joins के लिए चुनूँगा।

**Vespa वह है जिसकी तरफ आप तब बढ़ते हैं जब retrieval केवल आधा प्रोडक्ट है।** यह lexical retrieval, nearest-neighbor search, tensors, ranking expressions, grouping, और online serving को जोड़ता है। वह शक्ति असली है, लेकिन operational और modeling complexity भी उतनी ही है। यह "मेरे CRUD app में semantic search जोड़ें" से ज़्यादा search/recommendation teams के लिए फ़िट बैठता है।

**ClickHouse उस बातचीत में आता है जब search analytics से जुड़ा हो।** अगर आपका source of truth events, logs, traces, metrics, या बड़ी fact tables हैं, तो ClickHouse vector distance, filtering, aggregation, और अब गंभीर full-text indexing को एक SQL engine में रख सकता है। यह purpose-built वेक्टर डेटाबेस नहीं है, लेकिन analytical retrieval के लिए यह सबसे अच्छे तरीके से boring answer हो सकता है।

**Chroma dev/test और छोटे app deployments में सबसे मज़बूत है।** अगर आप बहुत बड़े corpora, HA, disk-heavy operation, या first-class hybrid search की तरफ बढ़ रहे हैं, तो prototype को infrastructure में promote करने से पहले production-oriented store का मूल्यांकन करें।

### सरलीकृत निर्णय

अगर आपने वास्तव में pgvector से बड़े हो गए हैं — आमतौर पर क्योंकि benchmarks दिखाते हैं कि vector count, filtering, write rate, या high-concurrency latency आपके Postgres envelope से बाहर जा रहे हैं — तो constraint के अनुसार चुनें:

- **Per-tenant isolation वाला SaaS प्रोडक्ट** → Turbopuffer
- **Rust-level performance + complex metadata filtering चाहिए** → Qdrant
- **पहले से Elastic/ELK stack पर हैं** → Elasticsearch with DiskBBQ
- **AWS shop जो open-source चाहती है** → OpenSearch
- **गंभीर ranking needs वाला search/recommendation platform** → Vespa
- **Analytics, observability, या log/event search** → ClickHouse
- **Billion-scale on-prem / self-hosted** → Milvus
- **Edge / serverless / multimodal** → LanceDB
- **छोटा JS app, docs site, या edge-native search UX** → Orama
- **Zero ops, बस काम करता है, cost secondary है** → Pinecone
- **Multimodal-first (images, video, audio)** → Marqo
- **पहले से MongoDB पर हैं** → Atlas Vector Search
- **पहले से Postgres पर हैं, ज़्यादा headroom चाहिए** → Supabase Vector या Neon (दोनों pgvector managed, बेहतर tooling के साथ)

## एक चीज़ जो नहीं करनी है

वेक्टर खोज को उन चीज़ों के लिए fuzzy text search के रूप में उपयोग न करें जिनके सही जवाब हैं।

"मुझे `dan@example.com` email वाला यूज़र ढूंढकर दें" वेक्टर खोज की समस्या नहीं है। "मुझे `ORD-12345` ID वाला ऑर्डर ढूंढकर दें" भी नहीं। `ORD-12345` को embed करना और आपके orders table के खिलाफ cosine similarity करना *कुछ* लौटाएगा, लेकिन यह ग़लत हो सकता है। ये exact-match समस्याएँ हैं। अपनी primary key या नियमित इंडेक्स उपयोग करें।

वेक्टर खोज आपके डेटासेट में *सबसे similar* चीज़ लौटाती है, तब भी जब कुछ भी relevant न हो। इसे पता नहीं होता कि कोई अच्छा जवाब नहीं है। संबंधित डॉक्यूमेंट्स के लिए यह ठीक है। यह specific record lookup के लिए विनाशकारी है, जहाँ ग़लत near-match खाली परिणाम से बदतर है।

पता लगाएं कि हर टूल किस काम के लिए है। उनमें से ज़्यादातर आपकी Postgres installation में पहले से मौजूद हैं। उन्हें वहाँ उपयोग करें जहाँ वे फ़िट बैठते हैं।
