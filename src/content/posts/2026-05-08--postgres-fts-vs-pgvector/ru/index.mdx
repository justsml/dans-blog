Команды, добавляющие AI-функции, часто первым делом обращаются к выделенной векторной базе данных.

Pinecone, Weaviate, Qdrant, Chroma. Новый сервис, новая зависимость, новый пул соединений, новый счёт и теперь два источника истины, которые нужно синхронизировать.

Тем временем у них уже есть PostgreSQL. В PostgreSQL уже есть `pgvector`. Также с 2008 года там есть отличная полнотекстовый поиск.

Выделенные векторные хранилища окупаются при больших масштабах и высоком объёме запросов. Но большинство приложений обращаются ко второй системе поиска, прежде чем первая будет достаточно испытана. Именно так будущая проблема масштабирования превращается в сегодняшнюю ошибку синхронизации.

Итак: когда использовать FTS, когда pgvector и когда оба?

---

## Что Каждый Из Инструментов Делает На Действительности

Полнотекстовый поиск (`tsvector` / `GIN` индекс) лексический. Он токенизирует текст в леммы, стеммирует их и сопоставляет запросы с индексом. "Running" и "runs" сводятся к одной лемме. То же самое с "dog" и "dogs." Функция ранжирования (`ts_rank`) награждает документы, где термины запроса встречаются часто или prominently.

pgvector семантический. Он хранит плотный вектор — список чисел — представляющий *значение* фрагмента, как его понимает модель эмбеддингов. Поиск по сходству находит близкие векторы в этом многомерном пространстве. "Dog" и "canine" оказываются рядом. "Running" как спорт и "running" как процесс могут и не оказаться.

Практическая разница: FTS отвечает "какие документы содержат эти слова?" Векторный поиск отвечает "какие документы примерно означают эту вещь?"

![Карта инструментов поиска, показывающая pg_trgm для коротких нечётких строк, полнотекстовый поиск для точных текстовых запросов, pgvector для семантического сопоставления и гибридный поиск для длинного контента, которому нужны и точные, и семантические сигналы.](../search-tool-map.svg)

_Первое разделение — это не "старый поиск vs. AI поиск". Это форма текста и тип ответа, который будет правильным._

## Когда Выигрывает Полнотекстовый Поиск

**Вы ищете термины, которые важны точно.** Продуктовые SKU, коды ошибок, номера моделей, имена пользователей, ссылки на юридические пункты. `SKU-AX-44192` не семантически похож ни на что. Он либо совпадает, либо нет. Векторный поиск может уверенно вернуть `SKU-AX-44193`. Это не то, что вам нужно.

**Ваши запросы основаны на ключевых словах.** Пользователи вводят текст в поле поиска, фильтруют по тегам или ищут посты в блоге по ключевым словам. FTS создан для этой формы намерения.

**Вам нужны ранжированные результаты без GPU или инфраструктуры эмбеддингов.** Индексы FTS быстры, детерминированы и не требуют внешних API-вызовов. Добавьте колонку `tsvector`, создайте GIN индекс, и всё готово.

**Вы выполняете булеву фильтрацию вместе с поиском.** `WHERE to_tsvector(body) @@ to_tsquery('postgres') AND category = 'tutorial' AND published_at > NOW() - INTERVAL '6 месяцев'` — это естественно сочетается с вашей существующей логикой запросов.

```sql
-- Создайте индекс
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body, '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Запрос
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

Колонка `GENERATED ALWAYS AS` автоматически обновляет индекс. `setweight` даёт заголовкам более высокий ранг, чем телу документа. Вот и вся настройка.

## Когда Выигрывают Триграммы (pg_trgm)

Есть третий инструмент Postgres, о котором забывают: `pg_trgm`. Это ни полнотекстовый поиск, ни векторный поиск. Это нечёткое сопоставление строк, и он покрывает неловкую середину, которую оба инструмента обрабатывают плохо.

**Случай использования: имена, адреса, идентификаторы и короткие строки с опечатками.**

FTS токенизирует текст в леммы и стеммирует их. Это работает для прозы, но плохо подходит для:
- Имен людей ("Dan Levy" → стеммировано в "dan levi", "leiv", зависит от языковой конфигурации)
- Названий компаний, адресов, названий продуктов, где важна точная орфография
- Запросов с опечатками — "Micheal Jordan", "Amaon", "javascipt"
- Автодополнения / префиксного поиска
- Частичного сопоставления строк ("son" совпадает с "Johnson", "Anderson")

pgvector тоже плохой выбор здесь. Вы можете создать эмбеддинг "Micheal Jordan" и найти ближайший вектор, но пространство эмбеддингов организует имена по значению, а не по орфографии. Ближайший сосед может быть "basketball legend" или "Michael B. Jordan", а не запись пользователя с опечаткой.

`pg_trgm` разбивает строки на перекрывающиеся 3-символьные срезы и измеряет, сколько триграмм две строки разделяют. "Dan" -> `" da"`, `"dan"`, `"an "`. "Micheal" и "Michael" разделяют большинство своих триграмм, поэтому сходство высоко.

```sql
-- Включите расширение (обычно уже доступно)
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- GIN индекс на колонке имени — включает быстрый триграммный поиск по сходству
CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- Нечёткий поиск по имени: находит "Micheal Jordan" при поиске "Michael Jordan"
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % оператор = порог сходства (по умолчанию 0.3)
ORDER BY score DESC
LIMIT 10;

-- Или используйте ILIKE с поддержкой триграммного индекса для сопоставления содержимого
SELECT id, name
FROM users
WHERE name ILIKE '%johnson%'   -- GIN индекс делает это быстрым
LIMIT 10;
```

Оператор `%` использует `pg_trgm.similarity_threshold` (по умолчанию 0.3, диапазон 0-1). Более высокие значения требуют более близких совпадений. Для поиска по именам 0.3-0.4 обычно подходит: достаточно permisсивно, чтобы поймать опечатки, достаточно строго, чтобы избежать шума.

**Триграммы также помогают с префиксным поиском и автодополнением, особенно когда автодополнение требует устойчивости к опечаткам или сопоставления содержимого:**

```sql
-- Автодополнение: префиксное сопоставление. Для чистых лево-заякоренных префиксов,
-- сравнивайте триграммный GIN с B-tree паттерн индексом на ваших данных.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- Больше контроля: word_similarity для частичных совпадений внутри более длинных строк
-- (полезно при поиске "Johnson" внутри "Andrew Johnson III")
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name          -- <% оператор = порог word_similarity
ORDER BY score DESC
LIMIT 10;
```

**Когда обращаться к `pg_trgm` вместо FTS:**

| Сценарий | Использовать |
|---|---|
| Поиск по имени человека/компании с опечатками | `pg_trgm` |
| Автодополнение / префиксный поиск | `pg_trgm` (или FTS с префиксными запросами) |
| Поиск коротких строк, кодов, идентификаторов | `pg_trgm` |
| Поиск статей прозы, документации | FTS |
| Поиск ключевых слов в логах | FTS |
| Мульязыковый поиск по именам | `pg_trgm` (он языково-независим) |

`pg_trgm` также комбинируется с FTS. Используйте триграммы для нечёткого предфильтра и ранжируйте с `ts_rank`, или комбинируйте триграммное сходство с векторным баллом.

## Когда Выигрывает pgvector

**Вы строите RAG.** RAG зависит от семантического извлечения: найдите *фрагменты* документа, чьё значение ближе всего к вопросу пользователя, даже когда формулировка отличается. Векторный поиск создан для этого. FTS пропустит парафразы, синонимы и концептуальные совпадения.

**Пользователи описывают, что они хотят, а не что искать.** "Что-то лёгкое для летнего вечера" не имеет очевидных винных ключевых слов. "Статьи о построении уверенности как нового менеджера" требуют семантического понимания, которое FTS не может предоставить.

**Вы находите похожие элементы.** Похожие продукты, похожие билеты поддержки, дубликаты отчётов об ошибках. "Найди мне проблемы, похожие на эту" — векторная операция. Вы создаёте эмбеддинг новой проблемы и находите её ближайших соседей.

**Мульязыковый контент.** Векторные эмбеддинги, обученные на мульязыковых данных, могут сопоставлятьAcross языки. FTS требует языково-специфичных конфигураций и плохо обрабатывает кросс-языковые запросы.

```sql
-- Настройка
CREATE EXTENSION IF NOT EXISTS vector;

ALTER TABLE documents ADD COLUMN embedding vector(1536);
CREATE INDEX documents_embedding_idx
  ON documents USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);

-- Запрос: семантический поиск
SELECT id, title, 1 - (embedding <=> $1::vector) AS similarity
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 10;
```

Примечание: `ivfflat` приближённый — он быстрый, но торгует некоторым recall ради скорости. Для меньших наборов данных (менее ~1M строк), `hnsw` часто лучше:

```sql
CREATE INDEX documents_embedding_idx
  ON documents USING hnsw (embedding vector_cosine_ops);
```

## Когда Вам Нужны Оба

Техническая документация — это место, где простое разделение ломается. Пользователи ищут "как настроить таймауты", но также ищут имена функций вроде `withRetry()` и коды ошибок вроде `ECONNRESET`.

Векторный поиск обрабатывает концептуальные запросы. FTS обрабатывает точные термины. Ни один не обрабатывает оба хорошо в одиночку.

Решение — гибридный поиск: запустите оба и объедините результаты.

**Reciprocal Rank Fusion (RRF)** — стандартный алгоритм здесь. Он не требует нормализации баллов из двух систем; он объединяет позиции ранжирования.

```sql
-- Гибридный поиск с Reciprocal Rank Fusion
WITH fts_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM documents, to_tsquery('english', $1) query
  WHERE search_vector @@ query
  LIMIT 50
),
vector_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> $2::vector) AS rank
  FROM documents
  ORDER BY embedding <=> $2::vector
  LIMIT 50
),
rrf AS (
  SELECT
    COALESCE(f.id, v.id) AS id,
    COALESCE(1.0 / (60 + f.rank), 0) + COALESCE(1.0 / (60 + v.rank), 0) AS rrf_score
  FROM fts_results f
  FULL OUTER JOIN vector_results v ON f.id = v.id
)
SELECT d.id, d.title, rrf.rrf_score
FROM rrf
JOIN documents d ON d.id = rrf.id
ORDER BY rrf_score DESC
LIMIT 10;
```

`60` в знаменателе — это константа RRF — более высокие значения уменьшают влияние различий рангов, более низкие усиливают их. Значение по умолчанию 60 работает хорошо в большинстве случаев.

Это запускает два поиска в одном запросе, объединяет ранги и награждает результаты, где ключевые слова и семантические сигналы согласуются.

![Гибридный конвейер поиска, где один запрос расходяется к полнотекстовому поиску и pgvector, каждый производит ранжированные результаты, и Reciprocal Rank Fusion объединяет два списка.](../hybrid-rrf-pipeline.svg)

_RRF ценен, потому что он избегает притворства, что `ts_rank` и косинусное расстояние — сопоставимые сырые баллы. Он только спрашивает, "как высоко этот результат появился в каждом списке?"_

## Практическое Дерево Решений

При выборе стратегии поиска начните с **формы ввода**, затем спросите **какой тип запроса делает пользователь**. "Короткая строка с вариацией орфографии" — не та же проблема, что "длинная проза, где важны точные термины", и оба отличаются от "вопроса по фрагментам документа".

![Дерево решений, которое выбирает pg_trgm, полнотекстовый поиск, pgvector, гибридный поиск или выделенную векторную базу данных на основе формы текста и способа, которым пользователи его запрашивают.](../search-decision-tree.svg)

То же дерево словами:

- **Имена, адреса, заголовки, автодополнение или предрасположенные к опечаткам короткие строки** → `pg_trgm`
- **Известные слова, коды ошибок, SKU, имена функций, теги, категории, фильтры** → FTS
- **Вопросы, парафразы, рекомендации, похожие элементы, мульязыковое сопоставление, фрагменты RAG** → pgvector
- **Технический контент, где пользователям нужны и точные символы, и концептуальные ответы** → Гибридный с RRF
- **Первичные ключи, точные ID, фильтры разрешений, диапазоны дат, отсортированные списки** → обычные SQL индексы
- **Огромный объём векторов, очень высокая конкуренция или цели по задержке, которые Postgres не может достичь в ваших бенчмарках** → оцените выделенные векторные хранилища

### FTS vs. Семантический: Короткая Версия

Вопрос "следует ли использовать FTS или векторный поиск?" обычно сводится к этому: **знаете ли вы, какие слова появятся в релевантных документах?**

Если да — пользователи ищут известные термины, категории, имена функций, коды продуктов — FTS быстрее, дешевле и более предсказуем. Он говорит, почему результат совпал.

Если нет — пользователи описывают концепцию, задают вопрос или ищут на другом языке — векторный поиск — правильный инструмент. Он сопоставляет значение, а не слова.

Трудная середина — естественные языковые запросы по техническому контенту. Кто-то, ищущий "как я обрабатываю падения соединения", может нуждаться в статье с заголовком "реализация логики повтора для сетевых сбоев" — никаких перекрывающихся слов, высокая семантическая релевантность. Вот здесь векторный поиск окупает себя.

Другой трудный случай — **имена и собственные имена**. Ни FTS, ни векторный поиск не хороши в них:
- FTS пропустит "Micheal" при поиске "Michael" — разные токены
- Векторный поиск пропустит имя целиком, если оно не часто появляется в обучающих данных
- `pg_trgm` обрабатывает это правильно: орфографическое сходство, не семантическое или лексическое

На практике, большинство контент-тяжёлых поисковых полей нуждаются в FTS для скорости и ключевых слов, и могут нуждаться в гибридном или `pg_trgm` в зависимости от того, ищут ли пользователи по именам. Настоящая функция семантического поиска обычно означает pgvector. RAG всегда означает pgvector.

## Если Вам Действительно Нужна Выделенная Векторная База Данных

Некоторые системы действительно вырастают из pgvector. Когда это происходит, рынок зашумлен. Вот что важно среди лучших опций.

### Матрица Функций

Несколько колонок нужно расшифровать, прежде чем таблица станет понятной.

**Гибридный поиск** означает BM25 поиск по ключевым словам и векторное сходство запускаются в одном запросе, объединены через Reciprocal Rank Fusion. "withRetry timeout" может совпасть с именем функции точно *и* документами о "логике повтора для сетевых сбоев" семантически. Без гибридного вы выбираете один режим поиска или объединяете два запроса сами. pgvector's "Manual (RRF via SQL)" — [подход, показанный выше](#when-you-need-both): он работает, но вы пишете его сами.

**Разреженные векторы** идут дальше BM25. SPLADE разреженный вектор имеет ~30,000 измерений (одно на термин словаря), ~98% нулей. Позиции с ненулевыми значениями говорят, какие термины важны и насколько. Запрос для "dogs" также взвешивает "canine" и "pet": точность ключевых слов уровня BM25 плюс расширение терминов внутри векторного индекса. Если эта колонка false, вам нужен внешний слой FTS для запросов по точным терминам.

```python
# SPLADE: ~30,000 dims всего, ~60 ненулевых — только релевантные позиции словаря активируются
def encode_splade(text: str) -> dict:
    tokens = tokenizer(text, return_tensors="pt", truncation=True, max_length=512)
    with torch.no_grad():
        output = model(**tokens)
    vec = torch.log1p(torch.relu(output.logits)).max(dim=1).values.squeeze()
    return {"indices": vec.nonzero().squeeze().tolist(), "values": vec[vec != 0].tolist()}
```

**SQL / SQL-подобный** — это на самом деле о фильтрации. Векторный поиск без фильтрации — демо, не приложение: вам всё ещё нужныtenant scope, диапазоны дат, разрешения и категории. Полный SQL (pgvector) выражает это рядом с вашими существующими JOIN-ами. Специализированные БД используют JSON объекты фильтрации (Qdrant, Pinecone), DSL запросов (Elasticsearch, Milvus) или GraphQL (Weaviate). Они работают; SQL становится привлекательнее, когда логика фильтрации запутывается.

```sql
-- pgvector: векторное сходство — просто ещё одно выражение в WHERE
SELECT id, title, 1 - (embedding <=> $1) AS score
FROM documents
WHERE tenant_id = $2 AND category = ANY($3::text[]) AND created_at > NOW() - INTERVAL '90 дней'
ORDER BY embedding <=> $1 LIMIT 10;
```

```python
)
```

**Мультимодальный нативный** не означает "может хранить image эмбеддинги"; каждая база данных хранит float массивы. Это означает, что база данных поставляется с моделями эмбеддингов для нетекстового контента, так что вы передаёте ей сырой URL изображения, и она обрабатывает векторизацию. Большинство баз данных здесь эмбеддинг-агностичны, так что вы владеете этим конвейером. Marqo и Weaviate (через CLIP/ImageBind модули) замыкают цикл.

```python
# Marqo: POST сырые изображения, запросите текстом — не нужен внешний шаг эмбеддинга
mq.index("products").add_documents(
    [{"id": "shoe-001", "image": "https://cdn.example.com/shoes/001.jpg"}],
    tensor_fields=["image"]
)
results = mq.index("products").search(q="lightweight shoes for summer")
# Возвращает shoe-001 несмотря на нулевое пересечение ключевых слов — CLIP обрабатывает кросс-модальное совпадение
```

**Индекс на диске** — это вопрос стоимости. RAM-резидентный HNSW может требовать несколько GB RAM на миллион 1536-измеренных векторов, когда сырые векторы, графовые накладные расходы и метаданные подсчитаны. Дисковые нативные альтернативы (Milvus DiskANN, Elasticsearch DiskBBQ, LanceDB's Lance формат, Turbopuffer's object storage tier) часто торгуют некоторой задержкой запроса ради более дешёвой инфраструктуры. Для RAG рабочих нагрузок, где задержка модели уже доминирует, это часто стоит бенчмаркать. Redis VSS — жёсткое ограничение: только RAM, нет пути на диск.

**Максимальные измерения** — это завтрашняя миграция, скрывающаяся в сегодняшнем выборе. `text-embedding-3-large` использует 3072 dims, Jina v3 может выдавать более крупные эмбеддинги, и исследовательские модели продолжают продвигаться выше. Некоторые управляемые сервисы публикуют жёсткие ограничения измерений; другие документируют высокие ограничения или нет практического ограничения для типичных моделей эмбеддингов. Проверьте текущие документ перед обязательством. Выберите что-то с запасом; миграция векторного индекса, потому что вы достигли потолка измерений — miserable спринт.

### Чтение Матрицы

Несколько вещей не помещаются чисто в таблицу:

**Разреженные векторы** — это как вы получаете BM25-качественное ключевое слово сопоставление внутри векторного индекса, без отдельного полнотекстового движка. Qdrant и Elasticsearch имеют особенно зрелые реализации здесь. Weaviate поддерживает их через BM25F. Если гибридный поиск критичен и вы не можете запустить две системы, ищите поддержку разреженных векторов.

**Индексы на диске** — это рычаг стоимости, не деталь реализации. RAM-резидентные HNSW индексы быстры, но могут стать дорогими по мере роста счёта векторов, измерения, метаданных и графовых накладных расходов. Дисковые альтернативы (Milvus DiskANN, Elasticsearch DiskBBQ, Turbopuffer object storage, LanceDB's Lance формат) торгуют задержкой запроса ради более низкой инфраструктуры стоимости. Для больших RAG индексов, этот компромисс часто стоит тестирования.

**Мульти-тенантность Turbopuffer** построена вокруг очень высоких счетов namespaces. Её публичная позиционировка и истории клиентов подчёркивают рабочие нагрузки вроде большого, namespace-тяжёлого корпуса Notion. Если каждый пользователь или организация нуждаются в изолированном векторном поиске, эта архитектура может изменить экономику, но всё ещё бенчмаркайте вашу собственную форму тенанта.

**LanceDB embedded режим** — это ближайшая вещь к "SQLite для векторного поиска". Он запускается in-process, не требует сервера, и работает в Lambda, Cloudflare Workers, и edge средах. Lance столбчатый формат делает embedded операцию практичной в реальном масштабе.

**Orama — это инфраструктура UX поиска, не склад.** Она превосходна, когда вы хотите крошечный полнотекстовый/векторный/гибридный поисковый движок в JavaScript приложении, на edge, или как управляемый сайт/app поисковый слой. Это не инструмент, который я бы выбрал для billion-vector retrieval, тяжёлой аналитики или сложных фильтрованных JOIN-ов.

**Vespa — это то, к чему вы обращаетесь, когда извлечение — только половина продукта.** Она комбинирует лексическое извлечение, nearest-neighbor поиск, тензоры, выражения ранжирования, группировку, и онлайн serving. Эта власть реальна, так же как операционная и моделирующая сложность. Она больше подходит search/recommendation командам, чем "добавить семантический поиск в мой CRUD app."

**ClickHouse принадлежит в разговоре, когда поиск attached к аналитике.** Если ваш источник истины — события, логи, трассировки, метрики, или большие fact таблицы, ClickHouse может держать векторное расстояние, фильтрацию, агрегацию, и теперь серьёзное полнотекстовое индексирование в одном SQL движке. Это не специализированная векторная база данных, но для аналитического извлечения она может быть скучным ответом в лучшем виде.

**Chroma сильна в dev/test и небольших app развёртываниях.** Если вы aiming at очень большие корпуса, HA, disk-тяжёлая операция, или first-class гибридный поиск, оцените production-ориентированное хранилище перед продвижением прототипа в инфраструктуру.

### Упрощённое Решение

Если вы действительно выросли из pgvector — обычно потому что бенчмарки показывают счёт векторов, фильтрацию, скорость записи, или high-concurrency задержку, выталкивающую за пределы вашего Postgres envelope — выбирайте по ограничению:

- **SaaS продукт с per-tenant изоляцией** → Turbopuffer
- **Нужна Rust-уровень производительности + сложные метаданные фильтрации** → Qdrant
- **Уже на Elastic/ELK stack** → Elasticsearch с DiskBBQ
- **AWS магазин, который хочет open-source** → OpenSearch
- **Search/recommendation платформа с серьёзными потребностями ранжирования** → Vespa
- **Аналитика, observability, или log/event поиск** → ClickHouse
- **Billion-scale on-prem / self-hosted** → Milvus
- **Edge / serverless / мультимодальный** → LanceDB
- **Маленькое JS приложение, сайт документации, или edge-native UX поиска** → Orama
- **Zero ops, просто работает, стоимость вторична** → Pinecone
- **Мультимодальный-first (изображения, видео, аудио)** → Marqo
- **Уже на MongoDB** → Atlas Vector Search
- **Уже на Postgres, нужна больше головы** → Supabase Vector или Neon (оба pgvector managed, с лучшим tooling)

## Одна Вещь, Которую Не Следует Делать

Не используйте векторный поиск как нечёткий текстовый поиск для вещей, у которых есть правильные ответы.

"Найди мне пользователя с email `dan@example.com`" — не проблема векторного поиска. То же самое с "Найди мне заказ с ID `ORD-12345`." Создание эмбеддинга `ORD-12345` и выполнение cosine similarity против вашей таблицы заказов вернёт *что-то*, но это может быть неправильно. Это проблемы точного совпадения. Используйте ваш первичный ключ или обычный индекс.

Векторный поиск возвращает *наиболее похожую* вещь в вашем наборе данных, даже когда ничего релевантного. Он не знает, что нет хорошего ответа. Это хорошо для похожих документов. Это катастрофично для специфичного lookup записи, где неправильное near-match хуже, чем пустой результат.

Знайте, для чего каждый инструмент. Большинство из них уже в вашей Postgres установке. Используйте их, где они подходят.
