Команды, добавляющие AI-функции, часто первым делом тянутся к выделенной векторной базе данных.

Pinecone, Weaviate, Qdrant, Chroma. Новый сервис, новая зависимость, новый пул соединений, новый счёт — и теперь два источника истины, которые нужно синхронизировать.

Между тем у них уже есть PostgreSQL. В PostgreSQL уже есть `pgvector`. И в нём уже давно (с 2008 года) встроен отличный полнотекстовый поиск.

Выделенные векторные хранилища оправдывают себя при больших масштабах и высоком объёме запросов. Но большинство приложений тянутся ко второй поисковой системе, не выжав толком первую. Вот так будущая проблема масштабирования превращается в сегодняшнюю ошибку синхронизации.

Итак: когда использовать FTS, когда pgvector, а когда оба?

---

## Что именно делает каждый из них

Полнотекстовый поиск (`tsvector` / GIN-индекс) — лексический. Он токенизирует текст на лексемы, приводит их к основе (стемминг) и сопоставляет запросы с индексом. "Бегу" и "бежит" сворачиваются в одну лексему. Как и "собака" с "собаки". Функция ранжирования (`ts_rank`) вознаграждает документы, где термины запроса встречаются часто или на видных позициях.

pgvector — семантический. Он хранит плотный вектор — список чисел, представляющий *смысл* фрагмента так, как его понимает модель эмбеддингов. Поиск по сходству находит ближайшие векторы в этом многомерном пространстве. "Собака" и "пёс" оказываются рядом. "Бег" как спорт и "бег" программы (run) — возможно, нет.

Практическая разница: FTS отвечает на вопрос «какие документы содержат эти слова?». Векторный поиск отвечает на вопрос «какие документы примерно значат то же самое?».

![Карта инструментов поиска: pg_trgm для коротких нечётких строк, полнотекстовый поиск для точных запросов, pgvector для семантического сопоставления и гибридный поиск для длинного контента, где нужны и точные, и смысловые сигналы.](../search-tool-map.svg)

_Первое разделение — не «старый поиск против AI-поиска». Это форма текста и то, какой ответ будет правильным._

## Когда побеждает полнотекстовый поиск

**Вы ищете термины, имеющие точное значение.** Артикулы товаров, коды ошибок, номера моделей, имена пользователей, ссылки на юридические пункты. `SKU-AX-44192` семантически ни на что не похож. Он либо совпадает, либо нет. Векторный поиск может уверенно вернуть `SKU-AX-44193`. А это совсем не то, что нужно.

**Ваши запросы основаны на ключевых словах.** Пользователи вводят текст в строку поиска, фильтруют по тегам или ищут посты в блоге по ключевым словам. FTS создан именно для такой формы запроса.

**Вам нужны ранжированные результаты без GPU или инфраструктуры эмбеддингов.** FTS-индексы быстры, детерминированы и не требуют внешних API-вызовов. Добавьте колонку `tsvector`, постройте GIN-индекс — и готово.

**Вы выполняете булеву фильтрацию вместе с поиском.** `WHERE to_tsvector(body) @@ to_tsquery('postgres') AND category = 'tutorial' AND published_at > NOW() - INTERVAL '6 months'` — это естественным образом сочетается с вашей существующей логикой запросов.

```sql
-- Создание индекса
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('russian', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('russian', coalesce(body, '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Запрос
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('russian', 'postgres & производительность') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

Колонка `GENERATED ALWAYS AS` автоматически поддерживает индекс в актуальном состоянии. `setweight` даёт совпадениям в заголовке больший вес, чем совпадениям в теле. Вот и вся настройка.

## Когда побеждают триграммы (pg_trgm)

Есть ещё один инструмент Postgres, который часто упускают из виду: `pg_trgm`. Это ни полнотекстовый поиск, ни векторный поиск. Это нечёткое сопоставление строк, и оно покрывает ту неловкую середину, с которой оба инструмента справляются плохо.

**Сценарий использования: имена, адреса, идентификаторы и короткие строки с опечатками.**

FTS токенизирует текст на лексемы и приводит их к основе. Это работает для прозы, но плохо подходит для:
- Имён людей («Иван Петров» → стемминг может исказить)
- Названий компаний, адресов, названий товаров, где важна точность написания
- Запросов с опечатками — «Михал», «Амазон», «жаваскрипт»
- Автодополнения / поиска по префиксу
- Частичного совпадения строк («софт» в «Майкрософт», «Яндекс.Софт»)

pgvector тоже подходит плохо. Можно эмбеддить «Михал» и найти ближайший вектор, но пространство эмбеддингов организует имена по смыслу, а не по написанию. Ближайшим соседом может оказаться «легендарный баскетболист», а не запись пользователя с опечаткой.

`pg_trgm` разбивает строки на перекрывающиеся 3-символьные фрагменты (триграммы) и измеряет, сколько триграмм у двух строк общих. «Иван» -> `" Ив"`, `"Ива"`, `"ван"`, `"ан "`. «Михал» и «Михаил» имеют много общих триграмм, поэтому сходство высокое.

```sql
-- Включаем расширение (обычно уже доступно)
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- GIN-индекс на колонке с именами — включает быстрый поиск по триграммам
CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- Нечёткий поиск имени: находит «Михал» при поиске «Михаил»
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- оператор % = порог сходства (по умолчанию 0.3)
ORDER BY score DESC
LIMIT 10;

-- Или ILIKE с поддержкой триграммного индекса для поиска вхождения
SELECT id, name
FROM users
WHERE name ILIKE '%петров%'   -- GIN-индекс ускоряет этот запрос
LIMIT 10;
```

Оператор `%` использует `pg_trgm.similarity_threshold` (по умолчанию 0.3, диапазон 0–1). Более высокие значения требуют более точного совпадения. Для поиска имён обычно подходит 0.3–0.4: достаточно свободно, чтобы ловить опечатки, и достаточно строго, чтобы избежать шума.

**Триграммы также помогают с поиском по префиксу и автодополнением, особенно когда автодополнению нужна устойчивость к опечаткам или поиск по вхождению:**

```sql
-- Автодополнение: поиск по префиксу. Для чисто левосторонних префиксов
-- сравните триграммный GIN с B-tree индексом на ваших данных.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- Более тонкий контроль: word_similarity для частичных совпадений внутри длинных строк
-- (полезно при поиске «Петров» внутри «Андрей Петров»)
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name          -- оператор <% = порог word_similarity
ORDER BY score DESC
LIMIT 10;
```

**Когда использовать `pg_trgm` вместо FTS:**

| Сценарий | Инструмент |
|---|---|
| Поиск имён/названий компаний с опечатками | `pg_trgm` |
| Автодополнение / поиск по префиксу | `pg_trgm` (или FTS с префиксными запросами) |
| Поиск по коротким строкам, кодам, идентификаторам | `pg_trgm` |
| Поиск по прозе, статьям, документации | FTS |
| Поиск по логам для ключевых слов | FTS |
| Многоязычный поиск имён | `pg_trgm` (не зависит от языка) |

`pg_trgm` также сочетается с FTS. Используйте триграммы для нечёткого предварительного фильтра и ранжируйте с помощью `ts_rank`, или комбинируйте триграммное сходство с векторной оценкой.

## Когда побеждает pgvector

**Вы строите RAG.** RAG зависит от семантического поиска: нужно найти фрагменты документов, смысл которых ближе всего к вопросу пользователя, даже если формулировки различаются. Векторный поиск создан именно для этого. FTS пропустит перефразирования, синонимы и концептуальные совпадения.

**Пользователи описывают, что они хотят, а не что искать.** У фразы «что-то лёгкое для летнего вечера» нет очевидных винодельческих ключевых слов. Фраза «статьи о том, как стать уверенным новым руководителем» требует семантического понимания, которое FTS не может обеспечить.

**Вы ищете похожие элементы.** Похожие товары, похожие тикеты поддержки, дубликаты баг-репортов. «Найди задачи, похожие на эту» — это векторная операция. Вы эмбеддите новую задачу и находите её ближайших соседей.

**Многоязычный контент.** Векторные эмбеддинги, обученные на многоязычных данных, могут сопоставлять тексты на разных языках. FTS требует языковых конфигураций и плохо справляется с кросс-языковыми запросами.

```sql
-- Настройка
CREATE EXTENSION IF NOT EXISTS vector;

ALTER TABLE documents ADD COLUMN embedding vector(1536);
CREATE INDEX documents_embedding_idx
  ON documents USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);

-- Запрос: семантический поиск
SELECT id, title, 1 - (embedding <=> $1::vector) AS similarity
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 10;
```

Примечание: `ivfflat` — приближённый: он быстр, но жертвует некоторой точностью в пользу скорости. Для небольших наборов данных (менее ~1M строк) часто лучше `hnsw`:

```sql
CREATE INDEX documents_embedding_idx
  ON documents USING hnsw (embedding vector_cosine_ops);
```

## Когда нужны оба

Техническая документация — вот где простое разделение ломается. Пользователи ищут «как настроить таймауты», но они также ищут имена функций вроде `withRetry()` и коды ошибок вроде `ECONNRESET`.

Векторный поиск обрабатывает концептуальные запросы. FTS обрабатывает точные термины. Ни один не справляется с обоими в одиночку.

Решение — гибридный поиск: запустить оба и слить результаты.

**Reciprocal Rank Fusion (RRF)** — стандартный алгоритм для этого. Он не требует нормализации оценок из двух систем; он комбинирует позиции в ранжировании.

```sql
-- Гибридный поиск с Reciprocal Rank Fusion
WITH fts_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM documents, to_tsquery('english', $1) query
  WHERE search_vector @@ query
  LIMIT 50
),
vector_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> $2::vector) AS rank
  FROM documents
  ORDER BY embedding <=> $2::vector
  LIMIT 50
),
rrf AS (
  SELECT
    COALESCE(f.id, v.id) AS id,
    COALESCE(1.0 / (60 + f.rank), 0) + COALESCE(1.0 / (60 + v.rank), 0) AS rrf_score
  FROM fts_results f
  FULL OUTER JOIN vector_results v ON f.id = v.id
)
SELECT d.id, d.title, rrf.rrf_score
FROM rrf
JOIN documents d ON d.id = rrf.id
ORDER BY rrf_score DESC
LIMIT 10;
```

`60` в знаменателе — константа RRF: более высокие значения уменьшают влияние различий в рангах, более низкие — усиливают его. Значение 60 по умолчанию хорошо работает в большинстве случаев.

Этот запрос выполняет два поиска в одном SQL, сливает ранги и вознаграждает результаты, где совпадают и ключевые слова, и семантические сигналы.

![Конвейер гибридного поиска: один запрос расходится на полнотекстовый поиск и pgvector, каждый выдаёт ранжированные результаты, а Reciprocal Rank Fusion объединяет два списка.](../hybrid-rrf-pipeline.svg)

_RRF ценен тем, что не требует делать вид, будто `ts_rank` и косинусное расстояние — сопоставимые сырые оценки. Он спрашивает лишь: «насколько высоко этот результат оказался в каждом списке?»_

## Практическое дерево решений

Выбирая стратегию поиска, начните с **формы входных данных**, затем спросите **какой запрос делает пользователь**. «Короткая строка с вариациями написания» — не та же задача, что «длинная проза, где важны точные термины», и обе отличаются от «вопроса по фрагментам документов».

![Дерево решений, выбирающее pg_trgm, полнотекстовый поиск, pgvector, гибридный поиск или выделенную векторную БД, исходя из формы текста и способа запроса пользователя.](../search-decision-tree.svg)

То же дерево словами:

- **Имена, адреса, заголовки, автодополнение или короткие строки с опечатками** → `pg_trgm`
- **Известные слова, коды ошибок, артикулы, имена функций, теги, категории, фильтры** → FTS
- **Вопросы, перефразирования, рекомендации, похожие элементы, многоязычное сопоставление, чанки для RAG** → pgvector
- **Технический контент, где пользователям нужны и точные символы, и концептуальные ответы** → Гибрид с RRF
- **Первичные ключи, точные ID, фильтры разрешений, диапазоны дат, отсортированные списки** → обычные SQL-индексы
- **Огромный объём векторов, очень высокая конкурентность или задержки, которых Postgres не может достичь по вашим бенчмаркам** → рассмотрите выделенные векторные хранилища

### FTS против семантического: краткая версия

Вопрос «использовать FTS или векторный поиск?» обычно сводится к следующему: **знаете ли вы, какие слова встретятся в релевантных документах?**

Если да — пользователи ищут известные термины, категории, имена функций, коды товаров — FTS быстрее, дешевле и предсказуемее. Он объясняет, почему результат подошёл.

Если нет — пользователи описывают концепцию, задают вопрос или ищут на другом языке — векторный поиск — правильный инструмент. Он сопоставляет смысл, а не слова.

Сложная середина — это запросы на естественном языке по техническому контенту. Кто-то ищет «как обрабатывать обрывы соединения», а нужна статья «реализация логики повторных попыток при сетевых сбоях» — ни одного общего слова, высокая семантическая релевантность. Вот где векторный поиск оправдывает себя.

Другой сложный случай — **имена и имена собственные**. Ни FTS, ни векторный поиск не хороши в этом:
- FTS пропустит «Михал» при поиске «Михаил» — разные токены
- Векторный поиск вообще не найдёт имя, если оно редко встречается в обучающих данных
- `pg_trgm` справляется с этим правильно: орфографическое сходство, не семантическое и не лексическое

На практике большинству контент-ориентированных поисковых строк нужен FTS для скорости и ключевых слов, и может понадобиться гибрид или `pg_trgm` в зависимости от того, ищут ли пользователи имена. Настоящая семантическая функция поиска почти всегда означает pgvector. RAG — всегда pgvector.

## Если вам всё же нужна выделенная векторная БД

Некоторые системы действительно перерастают pgvector. Когда это происходит, рынок шумен. Вот что имеет значение среди основных вариантов.

### Матрица возможностей

Несколько колонок требуют пояснений, прежде чем таблица обретёт смысл.

**Гибридный поиск** означает, что BM25-поиск по ключевым словам и векторное сходство выполняются в одном запросе и сливаются через Reciprocal Rank Fusion. `withRetry timeout` может совпасть точно с именем функции *и* семантически с документами о «логике повторных попыток при сетевых сбоях». Без гибрида вы выбираете один режим поиска или сливаете два запроса самостоятельно. «Вручную (RRF через SQL)» в pgvector — это [подход, показанный выше](#когда-нужны-оба): он работает, но вы пишете его сами.

**Разреженные векторы** идут дальше BM25. SPLADE-вектор имеет ~30 000 измерений (по одному на термин словаря), ~98% нулей. Ненулевые позиции показывают, какие термины важны и насколько. Запрос «собаки» также взвешивает «псы» и «питомец»: точность ключевых слов на уровне BM25 плюс расширение терминов внутри векторного индекса. Если эта колонка — false, вам понадобится внешний FTS-слой для точных запросов по терминам.

```python
# SPLADE: ~30 000 измерений всего, ~60 ненулевых — срабатывают только релевантные позиции словаря
def encode_splade(text: str) -> dict:
    tokens = tokenizer(text, return_tensors="pt", truncation=True, max_length=512)
    with torch.no_grad():
        output = model(**tokens)
    vec = torch.log1p(torch.relu(output.logits)).max(dim=1).values.squeeze()
    return {"indices": vec.nonzero().squeeze().tolist(), "values": vec[vec != 0].tolist()}
```

**SQL / SQL-подобный** — это на самом деле про фильтрацию. Векторный поиск без фильтрации — демо, а не приложение: вам всё равно нужны область видимости арендатора, диапазоны дат, разрешения и категории. Полноценный SQL (pgvector) выражает это рядом с вашими существующими JOIN. Специализированные БД используют JSON-объекты фильтров (Qdrant, Pinecone), DSL-запросы (Elasticsearch, Milvus) или GraphQL (Weaviate). Они работают; SQL становится привлекательнее, когда логика фильтрации усложняется.

```sql
-- pgvector: векторное сходство — это просто ещё одно выражение в WHERE
SELECT id, title, 1 - (embedding <=> $1) AS score
FROM documents
WHERE tenant_id = $2 AND category = ANY($3::text[]) AND created_at > NOW() - INTERVAL '90 days'
ORDER BY embedding <=> $1 LIMIT 10;
```

```python
)
```

**Мультимодальная нативность** — это не «умеет хранить эмбеддинги изображений» (любая база данных хранит массивы чисел). Это означает, что БД сама содержит модели эмбеддингов для не-текстового контента, так что вы передаёте ей URL сырого изображения, а она сама занимается векторизацией. Большинство БД из списка агностичны в плане эмбеддингов — вы владеете этим пайплайном сами. Marqo и Weaviate (через модули CLIP/ImageBind) замыкают этот цикл.

```python
# Marqo: POST сырые изображения, запрос текстом — без внешнего шага эмбеддинга
mq.index("products").add_documents(
    [{"id": "shoe-001", "image": "https://cdn.example.com/shoes/001.jpg"}],
    tensor_fields=["image"]
)
results = mq.index("products").search(q="лёгкая обувь для лета")
# Возвращает shoe-001, несмотря на нулевое совпадение ключевых слов — CLIP обрабатывает кросс-модальное сопоставление
```

**Дисковый индекс** — это вопрос стоимости. HNSW в RAM может требовать несколько ГБ ОЗУ на миллион векторов размерности 1536, если учитывать сырые векторы, накладные расходы графа и метаданные. Дисковые альтернативы (Milvus DiskANN, Elasticsearch DiskBBQ, LanceDB в формате Lance, Turbopuffer с object storage) часто жертвуют скоростью запросов ради более дешёвой инфраструктуры. Для RAG-нагрузок, где доминирует задержка модели, это часто стоит бенчмаркинга. Redis VSS — жёсткое ограничение: только RAM, дискового пути нет.

**Максимальная размерность** — это завтрашняя миграция, скрытая в сегодняшнем выборе. `text-embedding-3-large` использует 3072 измерения, Jina v3 может выдавать эмбеддинги большего размера, а исследовательские модели продолжают наращивать размерность. Некоторые управляемые сервисы публикуют жёсткие лимиты размерности; другие заявляют о высоких пределах или об их практическом отсутствии для типичных моделей эмбеддингов. Проверяйте актуальную документацию перед выбором. Выбирайте с запасом; миграция векторного индекса из-за превышения лимита размерности — это мучительный спринт.

### Как читать матрицу

Некоторые вещи не помещаются в таблицу:

**Разреженные векторы** — это способ получить поиск по ключевым словам качества BM25 внутри векторного индекса, без отдельного полнотекстового движка. У Qdrant и Elasticsearch особенно зрелые реализации. Weaviate поддерживает их через BM25F. Если гибридный поиск критичен и вы не можете запускать две системы, ищите поддержку разреженных векторов.

**Дисковые индексы** — это рычаг стоимости, а не деталь реализации. HNSW-индексы в RAM быстры, но могут стать дорогими по мере роста количества векторов, размерности, метаданных и накладных расходов графа. Дисковые альтернативы (Milvus DiskANN, Elasticsearch DiskBBQ, Turbopuffer object storage, LanceDB в формате Lance) жертвуют скоростью запросов ради более низкой стоимости инфраструктуры. Для больших RAG-индексов этот компромисс часто стоит протестировать.

**Мультитенантность Turbopuffer** построена вокруг очень большого количества пространств имён. Его публичное позиционирование и истории клиентов акцентируют нагрузки вроде большого корпуса Notion с множеством пространств имён. Если каждому пользователю или организации нужен изолированный векторный поиск, такая архитектура может изменить экономику, но всё равно тестируйте на своей форме арендаторов.

**Встраиваемый режим LanceDB** — это самое близкое к «SQLite для векторного поиска». Он работает в процессе, не требует сервера и может использоваться в Lambda, Cloudflare Workers и на границе сети. Колоночный формат Lance делает встраиваемую работу практичной в реальном масштабе.

**Orama — это инфраструктура поискового UX, а не хранилище.** Она отлично подходит, когда вам нужен крошечный полнотекстовый/векторный/гибридный поисковик в JavaScript-приложении, на границе сети или как управляемый слой поиска по сайту/приложению. Это не тот инструмент, который я бы выбрал для миллиардного векторного поиска, тяжёлой аналитики или сложных фильтрующих JOIN.

**Vespa — это то, что вы используете, когда поиск — лишь половина продукта.** Он сочетает лексический поиск, поиск ближайших соседей, тензоры, выражения ранжирования, группировку и онлайн-сервинг. Эта мощь реальна, но реальна и операционная сложность. Он подходит скорее командам, занимающимся поиском/рекомендациями, чем задаче «добавить семантический поиск в моё CRUD-приложение».

**ClickHouse стоит упомянуть, когда поиск привязан к аналитике.** Если ваш источник истины — события, логи, трейсы, метрики или большие таблицы фактов, ClickHouse может держать векторное расстояние, фильтрацию, агрегацию и теперь серьёзное полнотекстовое индексирование в одном SQL-движке. Это не специализированная векторная БД, но для аналитического поиска это может быть скучным ответом в лучшем смысле.

**Chroma сильнее всего в dev/test и небольших развёртываниях приложений.** Если вы целитесь в очень большие корпуса, отказоустойчивость, дисковую работу или первоклассный гибридный поиск — оцените production-ориентированное хранилище, прежде чем продвигать прототип в инфраструктуру.

### Упрощённое решение

Если вы действительно переросли pgvector — обычно потому, что бенчмарки показывают, что количество векторов, фильтрация, скорость записи или задержка при высокой конкурентности выходят за пределы возможностей вашего Postgres — выбирайте по ограничениям:

- **SaaS-продукт с изоляцией арендаторов** → Turbopuffer
- **Нужна производительность на уровне Rust + сложная фильтрация метаданных** → Qdrant
- **Уже на Elastic/ELK стеке** → Elasticsearch с DiskBBQ
- **AWS-компания, желающая open-source** → OpenSearch
- **Платформа поиска/рекомендаций с серьёзными потребностями в ранжировании** → Vespa
- **Аналитика, observability или поиск по логам/событиям** → ClickHouse
- **Миллиардные масштабы on-prem / self-hosted** → Milvus
- **Edge / serverless / мультимодальность** → LanceDB
- **Небольшое JS-приложение, сайт документации или поиск на границе сети** → Orama
- **Zero ops, просто работает, стоимость вторична** → Pinecone
- **Мультимодальность в первую очередь (изображения, видео, аудио)** → Marqo
- **Уже на MongoDB** → Atlas Vector Search
- **Уже на Postgres, нужно больше запаса** → Supabase Vector или Neon (оба управляют pgvector, с лучшими инструментами)

## Чего не стоит делать

Не используйте векторный поиск как нечёткий текстовый поиск для вещей, у которых есть правильные ответы.

«Найди пользователя с email `dan@example.com» — это не задача для векторного поиска. Как и «Найди заказ с ID `ORD-12345»». Эмбеддинг `ORD-12345` и косинусное сходство с вашей таблицей заказов что-то *вернут*, но это может быть неверно. Это задачи точного совпадения. Используйте первичный ключ или обычный индекс.

Векторный поиск возвращает *наиболее похожий* элемент в вашем наборе данных, даже когда ничего не релевантно. Он не знает, что хорошего ответа нет. Это нормально для поиска похожих документов. Это катастрофично для поиска конкретной записи, где неверное ближайшее совпадение хуже пустого результата.

Знайте, для чего предназначен каждый инструмент. Большинство из них уже есть в вашей установке Postgres. Используйте их там, где они подходят.
