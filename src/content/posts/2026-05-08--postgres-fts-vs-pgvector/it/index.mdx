I team che aggiungono funzionalità AI spesso ricorrono subito a un database vettoriale dedicato.

Pinecone, Weaviate, Qdrant, Chroma. Nuovo servizio, nuova dipendenza, nuova connessione, nuovo costo, e ora due fonti di verità da tenere in sincronia.

Nel frattempo, hanno già PostgreSQL. PostgreSQL ha già `pgvector`. Ha anche un eccellente full-text search integrato dal 2008.

I database vettoriali dedicati si giustificano a grande scala e con volumi elevati di query. Ma la maggior parte delle applicazioni si rivolge al secondo sistema di ricerca prima di aver spinto il primo fino in fondo. È così che un futuro problema di scalabilità diventa oggi un bug di sincronizzazione.

Quindi: quando usare FTS, quando usare pgvector, e quando usarli entrambi?

---

## Cosa Fa Ciascuno

Il full-text search (`tsvector` / indice `GIN`) è lessicale. Tokenizza il testo in lexemi, li stemmatizza e fa corrispondere le query all'indice. "Running" e "runs" collassano nello stesso lexema. Lo fanno anche "dog" e "dogs". La funzione di ranking (`ts_rank`) premia i documenti dove i termini della query appaiono spesso o in posizioni prominenti.

pgvector è semantico. Memorizza un vettore denso — una lista di numeri — che rappresenta il *significato* di un chunk così come è compreso da un modello di embedding. La ricerca per similarità trova vettori vicini in quello spazio ad alta dimensionalità. "Dog" e "canine" finiscono vicini. "Running" come sport e "running" come processo potrebbero non esserlo.

La differenza pratica: FTS risponde "quali documenti contengono queste parole?" La ricerca vettoriale risponde "quali documenti significano approssimativamente questa cosa?"

![Una mappa degli strumenti di ricerca che mostra pg_trgm per stringhe corte fuzzy, full-text search per query testuali esatte, pgvector per corrispondenza semantica, e ricerca ibrida per contenuti lunghi che richiedono sia segnali esatti che semantici.](../search-tool-map.svg)

_La prima divisione non è "ricerca vecchia vs. AI search". È la forma del testo e che tipo di risposta sarebbe corretta._

## Quando Vince il Full-Text Search

**Stai cercando termini che contano esattamente.** Codici prodotto SKU, codici errore, numeri di modello, nomi utente, riferimenti a clausole legali. `SKU-AX-44192` non è semanticamente simile a nulla. O corrisponde o non corrisponde. La ricerca vettoriale potrebbe restituire confidentemente `SKU-AX-44193`. Non è quello che vuoi.

**Le tue query sono basate su parole chiave.** Gli utenti digitano in una casella di ricerca, filtrano per tag, o cercano post del blog per parola chiave. FTS è stato costruito per quella forma di intento.

**Hai bisogno di risultati classificati senza GPU o infrastruttura di embedding.** Gli indici FTS sono veloci, deterministici e non richiedono chiamate API esterne. Aggiungi una colonna `tsvector`, costruisci un indice GIN, e hai finito.

**Stai facendo filtraggio booleano alongside la ricerca.** `WHERE to_tsvector(body) @@ to_tsquery('postgres') AND category = 'tutorial' AND published_at > NOW() - INTERVAL '6 mesi'` — questo si compone naturalmente con la tua logica di query esistente.

```sql
-- Crea l'indice
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body, '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Query
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

La colonna `GENERATED ALWAYS AS` mantiene l'indice aggiornato automaticamente. Il `setweight` dà ai match del titolo un ranking più alto rispetto ai match del corpo. Quello è l'intero setup.

## Quando Vincono i Trigrammi (pg_trgm)

C'è un terzo strumento Postgres che viene lasciato fuori: `pg_trgm`. Non è né full-text search né vettoriale. È corrispondenza fuzzy di stringhe, e copre la zona intermedia imbarazzata che entrambi gli strumenti gestiscono male.

**Il caso d'uso: nomi, indirizzi, identificatori e stringhe corte con refusi.**

FTS tokenizza il testo in lexemi e li stemmatizza. Funziona per prosa, ma è un brutto adattamento per:
- Nomi di persone ("Dan Levy" → stemmatizzato a "dan levi", "leiv", dipende dalla configurazione linguistica)
- Nomi di aziende, indirizzi, titoli di prodotti dove la ortografia esatta conta
- Query con refusi — "Micheal Jordan", "Amaon", "javascipt"
- Autocomplete / ricerca per prefisso
- Corrispondenza parziale di stringhe ("son" che corrisponde a "Johnson", "Anderson")

pgvector è anche una scelta povera qui. Puoi embeddare "Micheal Jordan" e trovare il vettore più vicino, ma lo spazio di embedding organizza i nomi per significato, non per ortografia. Il vicino più prossimo potrebbe essere "legenda del basket" o "Michael B. Jordan", non il record utente con il refuso.

`pg_trgm` rompe le stringhe in slice di 3 caratteri sovrapposti e misura quanti trigrammi due stringhe condividono. "Dan" -> `" da"`, `"dan"`, `"an "`. "Micheal" e "Michael" condividono la maggior parte dei loro trigrammi, quindi la similarità è alta.

```sql
-- Abilita l'estensione (solitamente già disponibile)
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- GIN index sulla colonna name — abilita ricerca di similarità trigramma veloce
CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- Ricerca nome fuzzy: trova "Micheal Jordan" quando si cerca "Michael Jordan"
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- operatore % = soglia di similarità (default 0.3)
ORDER BY score DESC
LIMIT 10;

-- O usa ILIKE con supporto indice trigramma per matching contiene
SELECT id, name
FROM users
WHERE name ILIKE '%johnson%'   -- GIN index rende questo veloce
LIMIT 10;
```

L'operatore `%` usa `pg_trgm.similarity_threshold` (default 0.3, range 0-1). Valori più alti richiedono match più stretti. Per ricerca nomi, 0.3-0.4 è solitamente giusto: permissivo abbastanza da catturare refusi, stretto abbastanza da evitare rumore.

**I trigrammi aiutano anche con ricerca per prefisso e autocomplete, specialmente quando autocomplete ha bisogno di tolleranza refusi o matching contiene:**

```sql
-- Autocomplete: matching prefisso. Per prefissi left-ancorati puri,
-- confronta GIN trigramma contro un indice B-tree pattern sulla tua data.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- Più controllo: word_similarity per match parziali dentro stringhe più lunghe
-- (utile quando cerchi "Johnson" dentro "Andrew Johnson III")
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name          -- operatore <% = soglia word_similarity
ORDER BY score DESC
LIMIT 10;
```

**Quando scegliere `pg_trgm` invece di FTS:**

| Scenario | Usa |
|---|---|
| Ricerca nomi di persone/aziende con refusi | `pg_trgm` |
| Autocomplete / ricerca per prefisso | `pg_trgm` (o FTS con query prefisso) |
| Ricerca stringhe corte, codici, identificatori | `pg_trgm` |
| Ricerca articoli prosa, documentazione | FTS |
| Ricerca messaggi log per parole chiave | FTS |
| Ricerca nomi multilingue | `pg_trgm` (è language-agnostic) |

`pg_trgm` si compone anche con FTS. Usa trigrammi per un pre-filter fuzzy e rank con `ts_rank`, o combina similarità trigramma con un punteggio vettoriale.

## Quando Vince pgvector

**Stai costruendo RAG.** RAG dipende da retrieval semantico: trova chunk di documenti il cui significato è più vicino alla domanda dell'utente, anche quando la formulazione differisce. La ricerca vettoriale è costruita apposta per questo. FTS mancherà parafrasi, sinonimi e corrispondenze concettuali.

**Gli utenti descrivono cosa vogliono, non cosa cercare.** "Qualcosa di leggero per una serata estiva" non ha parole chiave vino ovvie. "Articoli su costruire fiducia come nuovo manager" richiede comprensione semantica che FTS non può fornire.

**Stai trovando elementi simili.** Prodotti correlati, ticket di supporto simili, bug report duplicati. "Trova issue simili a questa" è un'operazione vettoriale. Embeddi la nuova issue e trovi i suoi vicini più prossimi.

**Contenuti multilingue.** Embeddings vettoriali addestrati su dati multilingue possono corrispondere tra lingue. FTS richiede configurazioni specifiche per lingua e gestisce male query cross-language.

```sql
-- Setup
CREATE EXTENSION IF NOT EXISTS vector;

ALTER TABLE documents ADD COLUMN embedding vector(1536);
CREATE INDEX documents_embedding_idx
  ON documents USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);

-- Query: ricerca semantica
SELECT id, title, 1 - (embedding <=> $1::vector) AS similarity
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 10;
```

Nota: `ivfflat` è approssimativo — è veloce ma scambia qualche recall per velocità. Per dataset più piccoli (sotto ~1M righe), `hnsw` è spesso meglio:

```sql
CREATE INDEX documents_embedding_idx
  ON documents USING hnsw (embedding vector_cosine_ops);
```

## Quando Ti Servono Entrambi

La documentazione tecnica è dove la divisione semplice si rompe. Gli utenti cercano "come configurare timeout", ma cercano anche nomi di funzioni come `withRetry()` e codici errore come `ECONNRESET`.

La ricerca vettoriale gestisce query concettuali. FTS gestisce termini esatti. Né gestisce entrambi bene da solo.

La soluzione è ricerca ibrida: esegui entrambe e fonde i risultati.

**Reciprocal Rank Fusion (RRF)** è l'algoritmo standard qui. Non richiede di normalizzare i punteggi da due sistemi; combina posizioni di rank.

```sql
-- Ricerca ibrida con Reciprocal Rank Fusion
WITH fts_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM documents, to_tsquery('english', $1) query
  WHERE search_vector @@ query
  LIMIT 50
),
vector_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> $2::vector) AS rank
  FROM documents
  ORDER BY embedding <=> $2::vector
  LIMIT 50
),
rrf AS (
  SELECT
    COALESCE(f.id, v.id) AS id,
    COALESCE(1.0 / (60 + f.rank), 0) + COALESCE(1.0 / (60 + v.rank), 0) AS rrf_score
  FROM fts_results f
  FULL OUTER JOIN vector_results v ON f.id = v.id
)
SELECT d.id, d.title, rrf.rrf_score
FROM rrf
JOIN documents d ON d.id = rrf.id
ORDER BY rrf_score DESC
LIMIT 10;
```

Il `60` nel denominatore è la costante RRF — valori più alti riducono l'influenza delle differenze di rank, valori più bassi le amplificano. Il default di 60 funziona bene nella maggior parte dei casi.

Questo esegue due ricerche in una query, fonde i rank, e premia risultati dove segnali keyword e semantici concordano.

![Un pipeline di ricerca ibrida dove una query si dirama a full-text search e pgvector, ciascuno produce risultati classificati, e Reciprocal Rank Fusion combina le due liste.](../hybrid-rrf-pipeline.svg)

_RRF è preziosa perché evita di pretendere che `ts_rank` e cosine distance siano punteggi raw comparabili. Chiede solo, "quanto in alto è apparso questo risultato in ciascuna lista?"_

## L'Albero Decisionale Pratico

Quando scegli una strategia di ricerca, inizia con la **forma dell'input**, poi chiediti **che tipo di query sta facendo l'utente**. "Stringa corta con variazione ortografica" non è lo stesso problema di "prosa lunga dove termini esatti contano", e entrambi differiscono da "domanda su chunk di documenti".

![Un albero decisionale che sceglie pg_trgm, full-text search, pgvector, ricerca ibrida, o un database vettoriale dedicato basato sulla forma del testo e il modo in cui gli utenti lo interrogano.](../search-decision-tree.svg)

Lo stesso albero in parole:

- **Nomi, indirizzi, titoli, autocomplete, o stringhe corte soggette a refusi** → `pg_trgm`
- **Parole note, codici errore, SKU, nomi di funzioni, tag, categorie, filtri** → FTS
- **Domande, parafrasi, raccomandazioni, elementi correlati, corrispondenza multilingue, chunk RAG** → pgvector
- **Contenuti tecnici dove gli utenti hanno bisogno sia di simboli esatti che di risposte concettuali** → Ibrido con RRF
- **Chiavi primarie, ID esatti, filtri permessi, range date, liste ordinate** → indici SQL normali
- **Volume vettoriale enorme, concorrenza molto alta, o target di latenza che Postgres non può raggiungere nei tuoi benchmark** → valuta database vettoriali dedicati

### FTS vs. Semantico: La Versione Breve

La domanda "dovrei usare FTS o ricerca vettoriale?" di solito collassa a questo: **conosci le parole che appariranno nei documenti rilevanti?**

Se sì - gli utenti cercano termini noti, categorie, nomi di funzioni, codici prodotto - FTS è più veloce, più economico, e più prevedibile. Ti dice perché un risultato ha corrisposto.

Se no - gli utenti descrivono un concetto, fanno una domanda, o cercano in una lingua diversa - la ricerca vettoriale è lo strumento giusto. Corrisponde al significato, non alle parole.

La zona intermedia difficile è query in linguaggio naturale su contenuti tecnici. Qualcuno che cerca "come gestire connessioni che cadono" potrebbe aver bisogno di un articolo intitolato "implementare logica retry per fallimenti di rete" — nessuna parola sovrapposta, alta rilevanza semantica. È qui che la ricerca vettoriale si guadagna il suo posto.

L'altra zona difficile è **nomi e nomi propri**. Né FTS né ricerca vettoriale sono bravi con loro:
- FTS mancherà "Micheal" quando cerchi "Michael" — token diversi
- La ricerca vettoriale mancherà il nome del tutto se non appare frequentemente nei dati di training
- `pg_trgm` gestisce questo correttamente: similarità ortografica, non semantica o lessicale

Nella pratica, la maggior parte delle caselle di ricerca basate su contenuti ha bisogno di FTS per velocità e parole chiave, e potrebbe aver bisogno di ibrido o `pg_trgm` a seconda se gli utenti cercano nomi. Una vera feature di ricerca semantica di solito significa pgvector. RAG significa sempre pgvector.

## Se Hai Davvero Bisogno di un Database Vettoriale Dedicato

Alcuni sistemi superano davvero pgvector. Quando succede, il mercato è rumoroso. Ecco cosa conta tra le opzioni principali.

### La Matrice delle Feature

Alcune colonne richiedono spiegazione prima che la tabella abbia senso.

**Ricerca ibrida** significa che BM25 keyword search e similarità vettoriale vengono eseguiti in una query, fusi tramite Reciprocal Rank Fusion. "withRetry timeout" può corrispondere al nome della funzione esattamente *e* documenti su "logica retry per fallimenti di rete" semanticamente. Senza ibrido, scegli una modalità di ricerca o fusi due query da solo. pgvector's "Manuale (RRF via SQL)" è [l'approccio mostrato sopra](#when-you-need-both): funziona, ma lo scrivi da solo.

**Vettori sparsi** vanno oltre BM25. Un vettore sparso SPLADE ha ~30,000 dimensioni (una per termine di vocabolario), ~98% zero. Le posizioni non-zero ti dicono quali termini contano e quanto. Una query per "dogs" pesa anche "canine" e "pet": precisione keyword livello BM25 più espansione termine dentro un indice vettoriale. Se questa colonna è false, hai bisogno di uno strato FTS esterno per query di termini esatti.

```python
# SPLADE: ~30,000 dims totali, ~60 non-zero — solo le posizioni di vocabolario rilevanti si attivano
def encode_splade(text: str) -> dict:
    tokens = tokenizer(text, return_tensors="pt", truncation=True, max_length=512)
    with torch.no_grad():
        output = model(**tokens)
    vec = torch.log1p(torch.relu(output.logits)).max(dim=1).values.squeeze()
    return {"indices": vec.nonzero().squeeze().tolist(), "values": vec[vec != 0].tolist()}
```

**SQL / SQL-like** è davvero su filtraggio. La ricerca vettoriale senza filtraggio è una demo, non un'applicazione: hai ancora bisogno di scope tenant, range date, permessi, e categorie. SQL pieno (pgvector) esprime questo accanto ai tuoi join esistenti. Database purpose-built usano oggetti di filtro JSON (Qdrant, Pinecone), un query DSL (Elasticsearch, Milvus), o GraphQL (Weaviate). Funzionano; SQL diventa più attraente quando la logica di filtraggio si complica.

```sql
-- pgvector: similarità vettoriale è solo un'altra espressione in WHERE
SELECT id, title, 1 - (embedding <=> $1) AS score
FROM documents
WHERE tenant_id = $2 AND category = ANY($3::text[]) AND created_at > NOW() - INTERVAL '90 giorni'
ORDER BY embedding <=> $1 LIMIT 10;
```

```python
# Qdrant: filtro equivalente come oggetto Python — funzionale, più cerimonia
results = client.query_points(
    collection_name="documents", query=query_embedding,
    query_filter=models.Filter(must=[
        models.FieldCondition(key="tenant_id", match=models.MatchValue(value=tenant_id)),
        models.FieldCondition(key="category",  match=models.MatchAny(any=categories)),
        models.FieldCondition(key="created_at", range=models.DatetimeRange(gte=cutoff)),
    ]),
    limit=10,
)
```

**Multimodale nativo** non significa "può memorizzare embedding di immagini"; ogni database memorizza array float. Significa che il database spedisce modelli di embedding per contenuti non-testo, così gli dai un URL immagine raw e lui gestisce la vettorizzazione. La maggior parte dei database qui sono embedding-agnostic, così possiedi quel pipeline. Marqo e Weaviate (tramite moduli CLIP/ImageBind) chiudono il loop.

```python
# Marqo: POST immagini raw, query con testo — nessun step embedding esterno necessario
mq.index("products").add_documents(
    [{"id": "shoe-001", "image": "https://cdn.example.com/shoes/001.jpg"}],
    tensor_fields=["image"]
)
results = mq.index("products").search(q="scarpe leggere per estate")
# Restituisce shoe-001 nonostante zero sovrapposizione keyword — CLIP gestisce la corrispondenza cross-modale
```

**Indice basato su disco** è una domanda di costo. HNSW resident in RAM può richiedere diversi GB di RAM per milione di vettori 1536-dimensioni una volta conteggiati vettori raw, overhead grafico, e metadata. Alternative native su disco (Milvus DiskANN, Elasticsearch DiskBBQ, formato Lance di LanceDB, tier object storage di Turbopuffer) spesso scambiano qualche latenza query per infrastruttura più economica. Per workload RAG dove la latenza del modello domina già, questo è spesso worth benchmarking. Redis VSS è il vincolo hard: solo RAM, nessun percorso disco.

**Dimensioni massime** è la migrazione di domani nascosta nella scelta di oggi. `text-embedding-3-large` usa 3072 dims, Jina v3 può emettere embedding più grandi, e modelli di ricerca continuano a spingere più in alto. Alcuni servizi managed pubblicano cap dimensionali hard; altri documentano cap alti o nessun cap pratico per modelli di embedding tipici. Controlla docs correnti prima di impegnarti. Scegli qualcosa con margine; migrare un indice vettoriale perché hai raggiunto un ceiling dimensionale è uno sprint miserabile.

### Leggendo la Matrice

Alcune cose non si adattano pulitamente in una tabella:

**Vettori sparsi** sono come ottieni matching keyword qualità BM25 dentro un indice vettoriale, senza motore full-text separato. Qdrant ed Elasticsearch hanno implementazioni specialmente mature qui. Weaviate li supporta tramite BM25F. Se ricerca ibrida è critica e non puoi eseguire due sistemi, cerca supporto vettori sparsi.

**Indici basati su disco** sono una leva di costo, non un dettaglio implementativo. Indici HNSW resident in RAM sono veloci ma possono diventare costosi mentre conteggio vettori, conteggio dimensioni, metadata, e overhead grafico crescono. Alternative basate su disco (Milvus DiskANN, Elasticsearch DiskBBQ, object storage Turbopuffer, formato Lance LanceDB) scambiano latenza query per costo infrastruttura più basso. Per indici RAG grandi, questo tradeoff è spesso worth testing.

**Multi-tenancy di Turbopuffer** è costruita attorno a conteggi namespace molto alti. La sua posizionamento pubblico e storie clienti enfatizzano workload come il corpus namespace-heavy di Notion. Se ogni utente o organizzazione ha bisogno di ricerca vettoriale isolata, quell'architettura può cambiare le economie, ma ancora benchmarka la tua forma tenant.

**LanceDB modalità embedded** è la cosa più vicina a "SQLite per ricerca vettoriale". Esegue in-process, non richiede server, e funziona in Lambda, Cloudflare Workers, e ambienti edge. Il formato columnar Lance rende operazione embedded pratica a scala reale.

**Orama è infrastruttura UX ricerca, non un warehouse.** È eccellente quando vuoi un motore di ricerca full-text/vettoriale/ibrido piccolo in un'app JavaScript, sull'edge, o come strato sito/app search managed. Non è lo strumento che sceglierei per retrieval bilioni-vettori, analytics pesanti, o joins filtrati complessi.

**Vespa è quello a cui ricorri quando retrieval è solo metà del prodotto.** Combina retrieval lessicale, nearest-neighbor search, tensori, espressioni ranking, grouping, e serving online. Quel potere è reale, ma lo è anche la complessità operativa e modeling. Si adatta a team search/recommendation più che "aggiungi search semantica alla tua CRUD app."

**ClickHouse appartiene nella conversazione quando ricerca è attaccata ad analytics.** Se la tua fonte di verità è eventi, log, tracce, metriche, o grandi tabelle fatti, ClickHouse può mantenere distanza vettoriale, filtraggio, aggregation, e ora serio indicizzazione full-text in un singolo engine SQL. Non è un database vettoriale purpose-built, ma per retrieval analitico può essere la risposta noiosa nel modo migliore.

**Chroma è più forte in dev/test e deploy app piccoli.** Se stai mirando a corpora molto grandi, HA, operazione heavy-disco, o first-class ricerca ibrida, valuta un store production-oriented prima di promuovere il prototype in infrastruttura.

### La Decisione Semplificata

Se hai davvero superato pgvector - solitamente perché benchmark mostrano conteggio vettori, filtraggio, write rate, o alta-concorrenza latenza spingendo oltre il tuo envelope Postgres - scegli per vincolo:

- **Prodotto SaaS con isolamento per-tenant** → Turbopuffer
- **Hai bisogno di performance livello Rust + filtraggio metadata complesso** → Qdrant
- **Già su stack Elastic/ELK** → Elasticsearch con DiskBBQ
- **Shop AWS che vuole open-source** → OpenSearch
- **Piattaforma search/recommendation con seri bisogno ranking** → Vespa
- **Analytics, osservabilità, o ricerca log/eventi** → ClickHouse
- **Scala bilioni on-prem / self-hosted** → Milvus
- **Edge / serverless / multimodale** → LanceDB
- **App JS piccola, sito docs, o UX search edge-native** → Orama
- **Zero ops, funziona solo, costo è secondario** → Pinecone
- **Multimodale-first (immagini, video, audio)** → Marqo
- **Già su MongoDB** → Atlas Vector Search
- **Già su Postgres, hai bisogno più margine** → Supabase Vector o Neon (entrambi pgvector managed, con migliori tooling)

## Una Cosa da Non Fare

Non usare ricerca vettoriale come ricerca testo fuzzy per cose che hanno risposte corrette.

"Trova l'utente con email `dan@example.com`" non è un problema ricerca vettoriale. Né è "Trova l'ordine con ID `ORD-12345`." Embeddare `ORD-12345` e fare cosine similarity contro la tua tabella ordini restituirà *qualcosa*, ma potrebbe essere sbagliato. Questi sono problemi match esatto. Usa la tua primary key o un indice regolare.

La ricerca vettoriale restituisce la cosa *più simile* nel tuo dataset, anche quando nulla è rilevante. Non sa che non c'è una buona risposta. Va bene per documenti correlati. È catastrofico per lookup record specifico, dove un near-match sbagliato è peggio di un risultato vuoto.

Sai cosa ciascuno strumento è per. La maggior parte sono già nella tua installazione Postgres. Usali dove si adattano.
