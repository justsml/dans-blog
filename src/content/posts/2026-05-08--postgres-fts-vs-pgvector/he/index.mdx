צוותים שמוסיפים תכונות AI לעיתים קרובות פונים תחילה למסד נתונים וקטורי ייעודי.

Pinecone, Weaviate, Qdrant, Chroma. שירות חדש, תלות חדשה, בריכת חיבורים חדשה, חשבון חדש, ועכשיו שני מקורות אמת שיש לשמור עליהם אמינים.

בינתיים, יש להם כבר PostgreSQL. ל‑PostgreSQL כבר יש `pgvector`. בנוסף, הוא כולל חיפוש טקסט מלא מצוין מאז 2008.

חסכונות וקטוריים ייעודיים מצדיקים את קיומם בקנה מידה גדול ובתדירות שאילתות גבוהה. אך רוב היישומים פונים למערכת החיפוש השנייה לפני שהראשונה נלחצת עד תום. כך בעיית קנה מידה עתידית מתהפכת לבאג סינכרון של היום.

אז: מתי להשתמש ב‑FTS, מתי ב‑pgvector, ומתי להשתמש בשניהם יחד?

## מה כל אחד עושה בפועל

חיפוש טקסט מלא (`tsvector` / אינדקס `GIN`) הוא לקסיקלי. הוא מחלק טקסט ללֶקְסִים, משורש אותם, ומשווה שאילתות מול האינדקס. „Running“ ו‑„runs“ מתמזגים לאותו לקסם. כך גם „dog“ ו‑„dogs“. פונקציית הדירוג (`ts_rank`) מעניקה משקל גבוה למסמכים שבהם מונחי השאילתה מופיעים בתדירות או במיקום בולט.

`pgvector` הוא סמנטי. הוא מאחסן וקטור צפוף – רשימת מספרים – שמייצגת את *המשמעות* של קטע כפי שמודל ההטבעה תופס. חיפוש דמיון מוצא וקטורים קרובים במרחב הרב‑ממדי הזה. „Dog“ ו‑„canine“ מסתיימים קרובים זה לזה. „Running“ כספורט ו‑„running“ כתהליך עשויים להיות רחוקים.

ההבדל המעשי: חיפוש טקסט מלא משיב „אילו מסמכים מכילים את המילים האלה?“ חיפוש וקטורי משיב „אילו מסמכים משמעותם קרובה לכאן?“

![מפת כלי חיפוש המציגה את pg_trgm למחרוזות קצרות מטושטשות, חיפוש טקסט מלא לשאילתות פרוזה מדויקות, pgvector להתאמה סמנטית, וחיפוש היברידי לתוכן ארוך שדורש גם אותות מדויקים וגם סמנטיים.](../search-tool-map.svg)

_הפיצול הראשון אינו "חיפוש ישן מול חיפוש AI". זהו צורת הטקסט וסוג התשובה שהייתה נכונה._

## כאשר חיפוש טקסט מלא מנצח

**אתה מחפש מונחים שחשובים בדיוק.** קודי SKU של מוצרים, קודי שגיאה, מספרי מודלים, שמות משתמש, הפניות לסעיפים משפטיים. `SKU-AX-44192` אינו דומה סמנטית לשום דבר. הוא או תואם או לא. חיפוש וקטורי עשוי להחזיר בביטחון `SKU-AX-44193`. זה לא מה שאתה רוצה.

**השאילתות שלך מבוססות מילות‑מפתח.** משתמשים מקלידים בתיבת חיפוש, מסננים לפי תג, או מחפשים פוסטים בבלוג לפי מילה. FTS נבנה עבור צורת כוונה כזו.

**אתה צריך תוצאות מדורגות ללא GPU או תשתית הטבעות.** אינדקסי FTS מהירים, דטרמיניסטיים, ואינם דורשים קריאות API חיצוניות. מוסיפים עמודת `tsvector`, בונים אינדקס GIN, והכול מוכן.

**אתה מבצע סינון בוליאני יחד עם החיפוש.** `WHERE to_tsvector(body) @@ to_tsquery('postgres') AND category = 'tutorial' AND published_at > NOW() - INTERVAL '6 months'` — זה משתלב באופן טבעי עם לוגיקת השאילתה הקיימת שלך.

```sql
-- Create the index
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body, '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Query
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

עמודת `GENERATED ALWAYS AS` משמרת את האינדקס מעודכן אוטומטית. `setweight` מעניקה התאמות בכותרת דירוג גבוה יותר מהתאמות בגוף. זהו כל ההגדרה.

## מתי טריגרמים מנצחים (pg_trgm)

קיים כלי שלישי ב‑Postgres שנשכח לעיתים: `pg_trgm`. הוא אינו חיפוש טקסט מלא ולא חיפוש וקטורי. הוא מתמקד בהתאמת מחרוזות מעורפלת, ומכסה את האזור המורכב שבו שני הכלים האחרים מתפקדים בחוסר יעילות.

**מקרה השימוש: שמות, כתובות, מזהים, ומחרוזות קצרות עם שגיאות הקלדה.**

FTS מחלק טקסט ללֶקְסִים ומבצע סטמינג. זה עובד לפרוזה, אך אינו מתאים ל‑:
- שמות אישיים (“Dan Levy” → “dan levi”, “leiv”, תלוי בתצורת השפה)
- שמות חברות, כתובות, כותרות מוצר שבהן האיות המדויק חשוב
- שאילתות עם שגיאות הקלדה — “Micheal Jordan”, “Amaon”, “javascipt”
- השלמה אוטומטית / חיפוש קידומת
- התאמת מחרוזות חלקיות (“son” תואם ל‑“Johnson”, “Anderson”)

גם `pgvector` אינו בחירה טובה כאן. אפשר להטביע את “Micheal Jordan” ולמצוא את הווקטור הקרוב ביותר, אך מרחב ההטבעות מארגן שמות לפי משמעות, לא לפי איות. השכן הקרוב עשוי להיות “אגדה של כדורסל” או “Michael B. Jordan”, ולא הרשומה עם השגיאה.

`pg_trgm` מחלק מחרוזות לחתיכות של 3 תווים חופפות ומודד כמה טריגרמים משותפים יש לשתי המחרוזות. “Dan” → `" da"`, `"dan"`, `"an "`. “Micheal” ו‑“Michael” חולקות את רוב הטריגרמים, ולכן הדמיון גבוה.

```sql
-- הפעלת ההרחבה (בדרך כלל כבר זמינה)
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- אינדקס GIN על עמודת השמות — מאפשר חיפוש דמיון טריגרמי מהיר
CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- חיפוש שם מעורפל: מוצא “Micheal Jordan” כשמחפשים “Michael Jordan”
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- אופרטור % = סף דמיון (ברירת מחדל 0.3)
ORDER BY score DESC
LIMIT 10;

-- או השתמש ב‑ILIKE עם תמיכת אינדקס טריגרם לחיפוש מכיל
SELECT id, name
FROM users
WHERE name ILIKE '%johnson%'   -- אינדקס GIN עושה זאת מהיר
LIMIT 10;
```

האופרטור `%` משתמש ב‑`pg_trgm.similarity_threshold` (ברירת מחדל 0.3, טווח 0‑1). ערכים גבוהים יותר דורשים התאמות קרובות יותר. לחיפוש שמות, 0.3‑0.4 הוא בדרך כלל מתאים: מספיק רפוי לתפוס שגיאות הקלדה, מספיק קפדני כדי למנוע רעש.

**טריגרמים גם מסייעים בחיפוש קידומת והשלמה אוטומטית, במיוחד כשצריך סובלנות לשגיאות הקלדה או חיפוש מכיל:**

```sql
-- השלמה אוטומטית: התאמת קידומת. עבור קידומות שמאליות טהורות,
-- השווה אינדקס GIN של טריגרמים מול אינדקס B‑tree של תבנית על הנתונים שלך.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- שליטה נוספת: word_similarity להתאמות חלקיות בתוך מחרוזות ארוכות
-- (שימושי כשמחפשים “Johnson” בתוך “Andrew Johnson III”)
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name          -- אופרטור <% = סף word_similarity
ORDER BY score DESC
LIMIT 10;
```

**מתי לבחור ב‑`pg_trgm` במקום ב‑FTS:**

| תרחיש | שימוש |
|---|---|
| חיפוש שם אדם/חברה עם שגיאות הקלדה | `pg_trgm` |
| השלמה אוטומטית / חיפוש קידומת | `pg_trgm` (או FTS עם שאילתות קידומת) |
| חיפוש מחרוזות קצרות, קודים, מזהים | `pg_trgm` |
| חיפוש מאמרים פרוזיים, תיעוד | FTS |
| חיפוש הודעות יומן למילות מפתח | FTS |
| חיפוש שמות רב‑לשוניים | `pg_trgm` (אינו תלוי בשפה) |

`pg_trgm` גם משתלב עם FTS. השתמשו ב‑trigrams כמסנן קדם מטושטש ודירגו עם `ts_rank`, או שלבו דמיון trigram עם ציון וקטור.

## מתי `pgvector` מנצח

**אתם בונים RAG.** RAG תלוי באחזור סמנטי: מציאת *חלקי* מסמכים שהמשמעות שלהם קרובה לשאלת המשתמש, אפילו שהניסוח שונה. חיפוש וקטורי נבנה במיוחד למטרה זו. FTS יפספס פרפרזות, מילות נרדפות והתאמות מושגיות.

**המשתמשים מתארים מה שהם רוצים, לא מה לחפש.** „משהו קל לערב קיץ“ אינו מכיל מילות מפתח ברורות ליין. „מאמרים על בניית ביטחון כמנהל חדש“ דורש הבנה סמנטית ש‑FTS לא יכול לספק.

**אתם מחפשים פריטים דומים.** מוצרים קשורים, כרטיסי תמיכה דומים, דוחות באגים משוכפלים. „מצא לי תקלות דומות לזו“ הוא פעולה וקטורית. אתם משקמים (embed) את התקלה החדשה ומוצאים את השכנים הקרובים ביותר.

**תוכן רב‑לשוני.** הטבעות הווקטוריות שאומנו על נתונים רב‑לשוניים יכולות להתאים בין שפות. FTS דורש תצורות ספציפיות לשפה ומתמודד בצורה גרועה עם שאילתות חוצות‑שפה.

```sql
-- Setup
CREATE EXTENSION IF NOT EXISTS vector;

ALTER TABLE documents ADD COLUMN embedding vector(1536);
CREATE INDEX documents_embedding_idx
  ON documents USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);

-- Query: semantic search
SELECT id, title, 1 - (embedding <=> $1::vector) AS similarity
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 10;
```

הערה: `ivfflat` הוא קירוב — הוא מהיר אך מחליף חלק מה‑recall לטובת המהירות. עבור קבוצות נתונים קטנות (מתחת לכ‑1 מיליון שורות), `hnsw` לרוב עדיף:

```sql
CREATE INDEX documents_embedding_idx
  ON documents USING hnsw (embedding vector_cosine_ops);
```

## מתי אתם צריכים את שניהם

תיעוד טכני הוא המקום שבו הפיצול הפשוט מתפרק. משתמשים מחפשים “how to configure timeouts”, אך הם גם מחפשים שמות פונקציות כמו `withRetry()` וקודי שגיאה כמו `ECONNRESET`.

חיפוש וקטורי מטפל בשאילתות קונספטואליות. FTS מטפל במונחים מדויקים. אף אחד מהם לא מתמודד היטב עם שני הסוגים יחד.

הפתרון הוא חיפוש היברידי: להריץ את שניהם ולמזג את התוצאות.

**Reciprocal Rank Fusion (RRF)** הוא האלגוריתם הסטנדרטי כאן. הוא אינו דורש נרמול ציונים משני המערכות; הוא משלב מיקומי דירוג.

```sql
-- Hybrid search with Reciprocal Rank Fusion
WITH fts_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM documents, to_tsquery('english', $1) query
  WHERE search_vector @@ query
  LIMIT 50
),
vector_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> $2::vector) AS rank
  FROM documents
  ORDER BY embedding <=> $2::vector
  LIMIT 50
),
rrf AS (
  SELECT
    COALESCE(f.id, v.id) AS id,
    COALESCE(1.0 / (60 + f.rank), 0) + COALESCE(1.0 / (60 + v.rank), 0) AS rrf_score
  FROM fts_results f
  FULL OUTER JOIN vector_results v ON f.id = v.id
)
SELECT d.id, d.title, rrf.rrf_score
FROM rrf
JOIN documents d ON d.id = rrf.id
ORDER BY rrf_score DESC
LIMIT 10;
```

המספר `60` במכנה הוא קבוע ה‑RRF — ערכים גבוהים מפחיתים את השפעת ההבדלים בדירוג, ערכים נמוכים מגבירים אותה. ברירת המחדל של 60 עובדת טוב ברוב המקרים.

זה מריץ שני חיפושים בשאילתה אחת, ממזג את הדירוגים, ומתגמל תוצאות שבהן האותות המילוליים והסמנטיים מתואמים.

![צינור חיפוש היברידי שבו שאילתה אחת מתפצלת לחיפוש טקסט מלא ו‑pgvector, כל אחד מייצר תוצאות מדורגות, ו‑Reciprocal Rank Fusion משלב את שתי הרשימות.](../hybrid-rrf-pipeline.svg)

_RRF בעל ערך מכיוון שהוא נמנע מהנחה ש‑`ts_rank` והמרחק הקוסינוס הם ציונים גולמיים השווים זה לזה. הוא שואל רק, “כמה גבוה הופיעה תוצאה זו בכל רשימה?”_

## עץ ההחלטות המעשי

כאשר בוחרים אסטרטגיית חיפוש, התחילו עם **צורת הקלט**, ואז שאלו **איזה סוג שאילתה המשתמש מבצע**. “מחרוזת קצרה עם וריאציות איות” איננה אותה בעיה כמו “פרוזה ארוכה שבה מונחים מדויקים חשובים”, ושניהם שונים מ-“שאלה על קטעי מסמך”.

![עץ החלטות שבוחר pg_trgm, חיפוש טקסט מלא, pgvector, חיפוש היברידי, או מסד וקטורים ייעודי על‑פי צורת הטקסט והאופן שבו משתמשים שואלים אותו.](../search-decision-tree.svg)

העץ באותו צורת מילים:

- **שמות, כתובות, כותרות, השלמה אוטומטית, או מחרוזות קצרות עם שגיאות כתיב** → `pg_trgm`
- **מילים ידועות, קודי שגיאה, SKU, שמות פונקציות, תגיות, קטגוריות, מסננים** → FTS
- **שאלות, פרפרזה, המלצות, פריטים קשורים, התאמה רב‑לשונית, קטעי RAG** → pgvector
- **תוכן טכני שבו המשתמשים זקוקים גם לסמלים מדויקים וגם לתשובות מושגיות** → היברידי עם RRF
- **מפתחות ראשיים, מזהים מדויקים, מסנני הרשאות, טווחי תאריכים, רשימות ממוינות** → אינדקסים רגילים של SQL
- **נפח וקטורים עצום, קונקורנטיות גבוהה מאוד, או דרישות השהייה ש‑Postgres לא עומד בהן בבנצ׳מרק שלכם** → העריכו חנויות וקטורים ייעודיות

### FTS מול סמנטי: הגרסה הקצרה

השאלה “האם להשתמש ב‑FTS או בחיפוש וקטורי?” מתכווצת לרוב ל‑: **האם אתם יודעים אילו מילים יופיעו במסמכים הרלוונטיים?**

אם כן – משתמשים מחפשים מונחים ידועים, קטגוריות, שמות פונקציות, קודי מוצר – FTS הוא מהיר יותר, זול יותר, וניתן לחיזוי. הוא מסביר למה תוצאה התאימה.

אם לא – משתמשים מתארים מושג, שואלים שאלה, או מחפשים בשפה אחרת – חיפוש וקטורי הוא הכלי המתאים. הוא תופס משמעות, לא מילים.

החלק המורכב באמצע הוא שאילתות שפה טבעית על תוכן טכני. מישהו שמחפש “איך להתמודד עם ניתוקים של חיבור” עשוי להזדקק למאמר שכותרתו “יישום לוגיקת נסיון חוזר לכשלי רשת” – אין מילים משותפות, אך הרלוונטיות הסמנטית גבוהה. כאן חיפוש וקטורי מרוויח את מקומו.

המקרה המורכב השני הוא **שמות ושמות עצמאיים**. גם FTS וגם חיפוש וקטורי אינם מצטיינים בהם:
- FTS יפספס “Micheal” כשמחפשים “Michael” – טוקנים שונים
- חיפוש וקטורי יפספס את השם לחלוטין אם הוא לא מופיע בתדירות גבוהה בנתוני האימון
- `pg_trgm` מטפל בזה נכון: דמיון אורפוגרפי, לא סמנטי או לקסיקלי

בפועל, רוב תיבות החיפוש העמוקות בתוכן דורשות FTS למהירות ולמילות‑מפתח, וייתכן וידרשו חיפוש היברידי או `pg_trgm` תלוי אם המשתמשים מחפשים שמות. תכונה של חיפוש סמנטי אמיתי משמעותה בדרך‑כלל pgvector. RAG תמיד משמעותו pgvector.

## אם אתה באמת צריך חנות וקטורים ייעודית

כמה מערכות חורגות באמת את היכולת של pgvector. כאשר זה קורה, השוק רועש. הנה מה שחשוב בבחירת האפשרויות המובילות.

### מטריצת התכונות

כמה עמודות דורשות פירוק לפני שהטבלה מקבלת משמעות.

**חיפוש היברידי** משמעותו חיפוש מילות‑מפתח BM25 ושלמות וקטורית באותו שאילתה, ממוזגים באמצעות Reciprocal Rank Fusion. `"withRetry timeout"` יכול להתאים בדיוק לשם הפונקציה *ול* למסמכים על “retry logic for network failures” סמנטית. ללא היברידיות, אתה בוחר במצב חיפוש אחד או ממזג שני שאילתות בעצמך. “Manual (RRF via SQL)” של pgvector הוא [הגישה שהודגמה למעלה](#when-you-need-both): היא עובדת, אבל אתה כותב אותה בעצמך.

**וקטורים דלילים** הולכים מעבר ל‑BM25. וקטור SPLADE דליל מכיל כ‑30,000 ממדים (אחד לכל מונח במילון), כש‑≈98 % מהם אפסים. המיקומים הלא‑אפסיים מצביעים על המונחים החשובים וכמה משקל יש להם. שאילתה ל‑“dogs” גם משקלת “canine” ו‑“pet”: דיוק מילות‑מפתח ברמת BM25 בתוספת הרחבת מונחים בתוך אינדקס וקטורי. אם עמודה זו שקרית, תצטרך שכבת FTS חיצונית לשאילתות מונח‑מדויק.

```python
# SPLADE: ~30,000 dims total, ~60 non-zero — only the relevant vocabulary positions fire
def encode_splade(text: str) -> dict:
    tokens = tokenizer(text, return_tensors="pt", truncation=True, max_length=512)
    with torch.no_grad():
        output = model(**tokens)
    vec = torch.log1p(torch.relu(output.logits)).max(dim=1).values.squeeze()
    return {"indices": vec.nonzero().squeeze().tolist(), "values": vec[vec != 0].tolist()}
```

**SQL / דמוי‑SQL** הוא בעצם סינון. חיפוש וקטורי ללא סינון הוא הדגמה, לא יישום: עדיין נדרש תחום שוכר, טווחי תאריכים, הרשאות וקטגוריות. SQL מלא (pgvector) מבטא זאת לצד הצטרפותיך הקיימות. מסדי נתונים ייעודיים משתמשים באובייקטים של סינון JSON (Qdrant, Pinecone), DSL של שאילתות (Elasticsearch, Milvus) או GraphQL (Weaviate). הם פועלים; SQL נעשה מושך יותר ככל שהלוגיקה של הסינון מסתבכת.

```sql
-- pgvector: vector similarity is just another expression in WHERE
SELECT id, title, 1 - (embedding <=> $1) AS score
FROM documents
WHERE tenant_id = $2 AND category = ANY($3::text[]) AND created_at > NOW() - INTERVAL '90 days'
ORDER BY embedding <=> $1 LIMIT 10;
```

```python
# Qdrant: equivalent filter as a Python object — functional, more ceremony
results = client.query_points(
    collection_name="documents", query=query_embedding,
    query_filter=models.Filter(must=[
        models.FieldCondition(key="tenant_id", match=models.MatchValue(value=tenant_id)),
        models.FieldCondition(key="category",  match=models.MatchAny(any=categories)),
        models.FieldCondition(key="created_at", range=models.DatetimeRange(gte=cutoff)),
    ]),

)
```

**תמיכה מולטימודאלית מובנית** לא אומרת “יכול לאחסן הטמעות של תמונות”; כל מסד נתונים שומר מערכי float. הכוונה היא שהמסד מספק מודלים להטמעת תוכן שאינו טקסט, כך שאתה נותן לו URL של תמונה גולמית והוא מטפל בווקטוריזציה. רוב המסדים כאן אינם תלויים במודל ההטמעה, ולכן הצינור הזה נשאר באחריותך. Marqo ו‑Weaviate (דרך מודולי CLIP/ImageBind) סוגרים את הלולאה.

```python
# Marqo: POST raw images, query with text — no external embedding step needed
mq.index("products").add_documents(
    [{"id": "shoe-001", "image": "https://cdn.example.com/shoes/001.jpg"}],
    tensor_fields=["image"]
)
results = mq.index("products").search(q="lightweight shoes for summer")
# Returns shoe-001 despite zero keyword overlap — CLIP handles the cross-modal match
```

**אינדקס מבוסס‑דיסק** הוא שיקול עלות. אינדקסי HNSW במטמון RAM יכולים לדרוש כמה ג'יגה‑בייט של RAM לכל מיליון וקטורים של 1536 ממדים כאשר מחשבים וקטורים גולמיים, עומס גרף ונתוני מטא‑דטה. אלטרנטיבות מקומיות‑דיסק (Milvus DiskANN, Elasticsearch DiskBBQ, פורמט Lance של LanceDB, שכבת האחסון של Turbopuffer) לעיתים מחליפות מעט זמן שאילתה עבור תשתית זולה יותר. בעומסי עבודה של RAG שבהם השהיית המודל כבר שולטת, זה לרוב שווה למדוד. Redis VSS הוא מגבלה קשוחה: RAM‑only, ללא נתיב דיסק.

**מימדים מרביים** הם הגירה של מחרת המוסתרת בבחירה של היום. `text-embedding-3-large` משתמש ב‑3072 ממדים, Jina v3 יכולה להפיק וקטורים גדולים יותר, ומודלים מחקריים ממשיכים לדחוף גבוה יותר. חלק משירותים מנוהלים מפרסמים תקרות קשות למימדים; אחרים מתעדים תקרות גבוהות או אין תקרת פרקטית למודלים טיפוסיים. בדוק את התיעוד העדכני לפני שמתחייבים. בחרו משהו עם מרווח ראש; מעבר לאינדקס וקטורים בגלל פגיעה בתקרת מימדים הוא ריצה בלתי‑סבירה.

### קריאת המטריצה

כמה נקודות אינן מתיישבות בטבלה:

**וקטורים מדוללים** הם הדרך לקבל התאמת מילות‑מפתח ברמת איכות BM25 בתוך אינדקס וקטורי, ללא מנוע טקסט מלא נפרד. ל‑Qdrant ול‑Elasticsearch יש מימושים בוגרים במיוחד כאן. Weaviate תומך בהם דרך BM25F. אם חיפוש היברידי קריטי ולא ניתן להריץ שני מערכות, חפשו תמיכה ב‑sparse vectors.

**אינדקסים מבוססי‑דיסק** הם מנוף עלות, לא פרט טכני. אינדקסי HNSW במטמון RAM מהירים אך יכולים להפוך יקרים ככל שמספר הווקטורים, מספר המימדים, המטא‑דטה ועומס הגרף גדלים. אלטרנטיבות מבוססות‑דיסק (Milvus DiskANN, Elasticsearch DiskBBQ, אחסון האובייקטים של Turbopuffer, פורמט Lance של LanceDB) מחליפות זמן שאילתה עבור עלות תשתית נמוכה יותר. עבור אינדקסים גדולים של RAG, המסחר הזה לרוב שווה לבדיקה.

**ה‑multi‑tenancy של Turbopuffer** נבנה סביב ספירות מרובות מאוד של מרחבי שם. המיצוב הציבורי שלו וסיפורי לקוחות מדגישים עומסים כמו הקורפוס הגדול והמרובה‑מרחבי שם של Notion. אם לכל משתמש או ארגון נדרש חיפוש וקטורי מבודד, ארכיטקטורה זו יכולה לשנות את הכלכלה, אך עדיין יש למדוד את הצורה של השוכרים שלכם.

**מצב משובץ של LanceDB** הוא הקרוב ביותר ל‑"SQLite לחיפוש וקטורי". הוא פועל בתהליך, אינו דורש שרת, ועובד ב‑Lambda, Cloudflare Workers ובסביבות edge. פורמט העמודה של Lance עושה פעולה משובצת פרקטית בקנה מידה אמיתי.

**Orama היא תשתית חוויית חיפוש, לא מחסן.** היא מצוינת כשאתם רוצים מנוע חיפוש טקסט‑מלא/וקטורי/היברידי זעיר באפליקציית JavaScript, בקצה הרשת, או כשכבת חיפוש מנוהלת לאתר/אפליקציה. היא לא הכלי שבו הייתי בוחר למשיכת מיליארדי וקטורים, אנליטיקה כבדה, או חיבורים מורכבים עם סינון.

**Vespa היא מה שאתם פונים אליו כשמשיכה היא רק חצי מהמוצר.** היא משלב חיפוש לקסיקלי, חיפוש שכנים קרובים, טנסורים, ביטויי דירוג, קיבוץ והגשה בזמן אמת. הכוח הזה קיים, אך גם המורכבות התפעולית והמודלינגית. היא מתאימה יותר לצוותי חיפוש/המלצות מאשר ל„להוסיף חיפוש סמנטי לאפליקציית CRUD שלי”.

**ClickHouse נכנסת לשיחה כשחיפוש מצורף לאנליטיקה.** אם מקור האמת שלכם הוא אירועים, לוגים, עקבות, מדדים, או טבלאות עובדות גדולות, ClickHouse יכול לשמור מרחק וקטורי, סינון, צבירה, ועכשיו גם אינדקס טקסט‑מלא רציני במנוע SQL אחד. היא לא מסד נתונים וקטורי שנבנה למטרה זו, אך עבור שליפה אנליטית היא יכולה להיות הפתרון המשעמם בצורה הטובה ביותר.

**Chroma חזקה ביותר בפיתוח/בדיקה ובפריסות אפליקציות קטנות.** אם אתם שואפים לקורפוסים גדולים מאוד, זמינות גבוהה, פעילות כבדת דיסק, או חיפוש היברידי ברמה ראשונה, יש להעריך חנות מוכוונת ייצור לפני שמקדים את האבות טיפוס לתשתית.

### ההחלטה המפושטת

אם באמת עברתם את גבולות pgvector – בדרך כלל כי מדדי ביצועים מראים שמספר הווקטורים, הסינון, קצב הכתיבה, או השהייה בתחרותיות גבוהה חורגים מהמגבלות של Postgres – בחרו לפי מגבלה:

- **מוצר SaaS עם בידוד לכל שוכר** → Turbopuffer  
- **צריך ביצועים ברמת Rust + סינון מטא‑דאטה מורכב** → Qdrant  
- **כבר על ערמת Elastic/ELK** → Elasticsearch עם DiskBBQ  
- **חנות AWS שרוצה קוד פתוח** → OpenSearch  
- **פלטפורמת חיפוש/המלצות עם דרישות דירוג רציניות** → Vespa  
- **אנליטיקה, תצפית, או חיפוש לוגים/אירועים** → ClickHouse  
- **קנה מידה של מיליארד במקומות מקומיים / אירוח עצמי** → Milvus  
- **קצה / ללא שרת / מולטימודלי** → LanceDB  
- **אפליקציית JS קטנה, אתר תיעוד, או חוויית חיפוש מקומית בקצה** → Orama  
- **אפס תפעול, פשוט עובד, העלות משנית** → Pinecone  
- **מולטימודלי ראשון (תמונות, וידאו, אודיו)** → Marqo  
- **כבר על MongoDB** → Atlas Vector Search  
- **כבר על Postgres, צריך יותר מרווח** → Supabase Vector או Neon (שניהם pgvector מנוהלים, עם כלי עבודה משופרים)

## דבר אחד שלא לעשות

אל תשתמשו בחיפוש וקטורי כחיפוש טקסט מעורפל עבור דברים שיש להם תשובות נכונות.

„מצא לי את המשתמש עם האימייל `dan@example.com`“ אינו בעיית חיפוש וקטורי. גם „מצא לי את ההזמנה עם המזהה `ORD-12345`“ אינו. הטמעת `ORD-12345` וביצוע דמיון קוסינוס מול טבלת ההזמנות שלכם תחזיר *משהו*, אך ייתכן שהוא שגוי. אלו בעיות של התאמה מדויקת. השתמשו במפתח הראשי או באינדקס רגיל.

חיפוש וקטורי מחזיר את הפריט *הדומה ביותר* במאגר שלכם, אפילו כשאין דבר רלוונטי. הוא לא יודע שאין תשובה טובה. זה מקובל עבור מסמכים קשורים, אך קטסטרופלי עבור חיפוש רשומה ספציפית, שבו התאמה קרובה שגויה גרועה מתוצאה ריקה.

הבינו עבור מה כל כלי נועד. רובם כבר מותקנים ב‑Postgres שלכם. השתמשו בהם רק במקומות שבהם הם מתאימים.
