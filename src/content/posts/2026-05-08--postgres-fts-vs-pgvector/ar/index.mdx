الفرق بين إضافة ميزات الذكاء الاصطناعي واستخدام قاعدة بيانات متجهات مخصصة هو أن الفرق غالبًا ما يلجأ إلى قاعدة بيانات متجهات مخصصة أولاً.

Pinecone, Weaviate, Qdrant, Chroma. خدمة جديدة، اعتماد جديد، مجموعة اتصالات جديدة، فاتورة جديدة، والآن مصدران للحقائق يجب الحفاظ على صدقهما.

في الوقت نفسه، لديهم بالفعل PostgreSQL. PostgreSQL يحتوي بالفعل على `pgvector`. كما أنه يمتلك بحثًا نصيًا كاملًا ممتازًا مدمجًا منذ عام 2008.

تكتسب مخازن المتجهات المخصصة قيمتها عند التعامل مع نطاقات كبيرة وحجم استعلامات مرتفع. لكن معظم التطبيقات تلجأ إلى نظام البحث الثاني قبل أن يتم الضغط على الأول بشدة. هكذا يتحول مشكلة التوسع المستقبلية إلى خطأ مزامنة اليوم.

إذن: متى تستخدم FTS، ومتى تستخدم pgvector، ومتى تستخدم كليهما؟

## ما الذي يفعله كل منهما فعليًا

البحث النصي الكامل (`tsvector` / فهرس `GIN`) هو بحث لغوي. يقوم بتقسيم النص إلى صيغ أساسية، ويجذرها، ويطابق الاستعلامات مع الفهرس. يتحول "Running" و "runs" إلى نفس الصيغة الأساسية. وكذلك "dog" و "dogs". دالة الترتيب (`ts_rank`) تكافئ المستندات التي تظهر فيها مصطلحات الاستعلام بشكل متكرر أو بارز.

pgvector هو بحث دلالي. يخزن متجهًا كثيفًا — قائمة من الأرقام — تمثل *معنى* الجزء كما يفهمه نموذج التضمين. البحث عن التشابه يجد المتجهات القريبة في ذلك الفضاء عالي الأبعاد. "Dog" و "canine" يقتربان من بعضهما. قد لا تكون "Running" كرياضة و "running" كعملية قريبة.

الفرق العملي: FTS يجيب على سؤال "أي المستندات تحتوي على هذه الكلمات؟" بينما بحث المتجهات يجيب على سؤال "أي المستندات تعني تقريبًا هذا الشيء؟"

![خريطة أدوات البحث تُظهر pg_trgm للسلاسل القصيرة غير الدقيقة، والبحث النصي الكامل للاستعلامات النثرية الدقيقة، وpgvector للمطابقة الدلالية، والبحث الهجين للمحتوى الطويل الذي يحتاج إلى إشارات دقيقة ودلالية معًا.](../search-tool-map.svg)

_الانقسام الأول ليس "بحثًا قديمًا مقابل بحثًا بالذكاء الاصطناعي". إنه شكل النص ونوع الإجابة التي ستكون صحيحة._

## عندما يربح البحث النصي الكامل

**أنت تبحث عن مصطلحات ذات معنى دقيق.** رموز المنتجات (SKU)، رموز الأخطاء، أرقام النماذج، أسماء المستخدمين، مراجع البنود القانونية. `SKU-AX-44192` لا يشبه دلاليًا أي شيء آخر. إما أن يتطابق أو لا. قد يُرجع بحث المتجهات بثقة `SKU-AX-44193`. هذا ليس ما تريده.

**استعلاماتك قائمة على الكلمات المفتاحية.** المستخدمون يكتبون في صندوق البحث، يفلترون حسب الوسم، أو يبحثون عن مقالات مدونة بالكلمة المفتاحية. تم بناء FTS لهذا الشكل من النية.

**تحتاج إلى نتائج مرتبة دون الحاجة إلى GPU أو بنية تحتية للتضمين.** فهارس FTS سريعة، حتمية، ولا تتطلب استدعاءات API خارجية. أضف عمود `tsvector`، أنشئ فهرس GIN، وانتهى الأمر.

**تقوم بفلترة منطقية إلى جانب البحث.** `WHERE to_tsvector(body) @@ to_tsquery('postgres') AND category = 'tutorial' AND published_at > NOW() - INTERVAL '6 months'` — يتكامل هذا طبيعيًا مع منطق الاستعلام الحالي لديك.

```sql
-- إنشاء الفهرس
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body, '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- الاستعلام
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

عمود `GENERATED ALWAYS AS` يبقي الفهرس محدثًا تلقائيًا. `setweight` يمنح مطابقة العنوان وزنًا أعلى من مطابقة النص. هذا هو الإعداد بالكامل.

## عندما تكون الـ Trigrams هي الفائزة (pg_trgm)

هناك أداة ثالثة في Postgres تُهمل غالبًا: `pg_trgm`. ليست بحث نص كامل ولا بحث متجهات. إنها مطابقة سلاسل غامضة، وتغطي الفجوة الصعبة التي تتعامل معها الأداتان بشكل ضعيف.

**حالة الاستخدام: الأسماء، العناوين، المعرفات، والسلاسل القصيرة التي تحتوي على أخطاء إملائية.**

يقوم FTS بتقسيم النص إلى صيغ أساسية (lexemes) وتجذيره. هذا يناسب النثر، لكنه غير ملائم لـ:
- أسماء الأشخاص ("Dan Levy" → تُجذر إلى "dan levi"، "leiv"، حسب إعداد اللغة)
- أسماء الشركات، العناوين، عناوين المنتجات حيث تهتم الدقة الإملائية
- الاستعلامات التي تحتوي على أخطاء إملائية — "Micheal Jordan"، "Amaon"، "javascipt"
- الإكمال التلقائي / بحث البادئة
- مطابقة جزئية للسلاسل ("son" تطابق "Johnson"، "Anderson")

`pgvector` أيضًا خيار ضعيف هنا. يمكنك تضمين "Micheal Jordan" والبحث عن أقرب متجه، لكن فضاء التضمين ينظم الأسماء حسب المعنى، لا حسب الإملاء. أقرب جار قد يكون "أسطورة كرة السلة" أو "Michael B. Jordan"، وليس السجل الذي يحتوي على الخطأ.

`pg_trgm` تقسم السلاسل إلى قطع متداخلة من 3 أحرف وتقيس عدد الـ trigrams المشتركة بين سلسلتين. "Dan" → `" da"`، `"dan"`، `"an "` . "Micheal" و "Michael" تشتركان في معظم الـ trigrams، لذا تكون التشابهة عالية.

```sql
-- تمكين الامتداد (عادةً متاح مسبقًا)
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- فهرس GIN على عمود الأسماء — يتيح بحث تشابه الـ trigram السريع
CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- بحث اسم غامض: يجد "Micheal Jordan" عند البحث عن "Michael Jordan"
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- عامل % = عتبة التشابه (الافتراضي 0.3)
ORDER BY score DESC
LIMIT 10;

-- أو استخدم ILIKE مع دعم فهرس الـ trigram للمطابقة داخل النص
SELECT id, name
FROM users
WHERE name ILIKE '%johnson%'   -- فهرس GIN يجعل هذا سريعًا
LIMIT 10;
```

عامل `%` يستخدم `pg_trgm.similarity_threshold` (الافتراضي 0.3، النطاق 0‑1). القيم الأعلى تتطلب تطابقًا أقرب. للبحث عن الأسماء، عادةً ما تكون 0.3‑0.4 مناسبة: تسمح بالتقاط الأخطاء الإملائية دون إحداث ضوضاء.

**تساعد الـ Trigrams أيضًا في بحث البادئة والإكمال التلقائي، خاصةً عندما يحتاج الإكمال إلى تحمل الأخطاء أو مطابقة داخل النص:**

```sql
-- إكمال تلقائي: مطابقة بادئة. للبادئات اليسارية الصرفة،
-- قارن فهرس GIN للـ trigram ضد فهرس نمط B‑tree على بياناتك.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- تحكم أكبر: word_similarity للمطابقات الجزئية داخل سلاسل أطول
-- (مفيد عند البحث عن "Johnson" داخل "Andrew Johnson III")
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name          -- عامل <% = عتبة word_similarity
ORDER BY score DESC
LIMIT 10;
```

**متى تختار `pg_trgm` بدلًا من FTS:**

| السيناريو | الاستخدام |
|---|---|
| بحث عن أسماء أشخاص/شركات مع أخطاء إملائية | `pg_trgm` |
| إكمال تلقائي / بحث بادئة | `pg_trgm` (أو FTS مع استعلامات بادئة) |
| البحث عن سلاسل قصيرة، رموز، معرفات | `pg_trgm` |
| البحث في مقالات نثرية، وثائق | FTS |
| البحث في رسائل السجلات عن كلمات مفتاحية | FTS |
| بحث متعدد اللغات عن الأسماء | `pg_trgm` (لا يعتمد على اللغة) |

`pg_trgm` يمكن أن يتكامل مع FTS. استخدم ثلاثيات الأحرف كمرشح غير دقيق مسبقًا ورتّب النتائج باستخدام `ts_rank`، أو اجمع تشابه الثلاثيات مع درجة المتجه.

## متى يفضَّل pgvector

**أنت تبني نظام RAG.** يعتمد RAG على الاسترجاع الدلالي: العثور على *قطع* المستند التي يكون معناها أقرب إلى سؤال المستخدم، حتى وإن اختلفت الصياغة. البحث بالمتجهات مُصمم لهذا الغرض. سيغفل FTS عن الصيغ المعادلة، المرادفات، والتطابقات المفهومية.

**المستخدمون يصفون ما يريدون، لا ما يجب البحث عنه.** العبارة “شيء خفيف لسهرة صيفية” لا تحتوي على كلمات مفتاحية واضحة للنبيذ. “مقالات عن بناء الثقة كمدير جديد” تتطلب فهماً دلاليًا لا يستطيع FTS توفيره.

**أنت تبحث عن عناصر مشابهة.** منتجات ذات صلة، تذاكر دعم مماثلة، تقارير أخطاء مكررة. “اعثر على القضايا المشابهة لهذه” هي عملية متجهية. تقوم بتمثيل القضية الجديدة كمتجه وتبحث عن أقرب جيران له.

**محتوى متعدد اللغات.** يمكن لتضمينات المتجهات المدربة على بيانات متعددة اللغات أن تطابق عبر اللغات. يتطلب FTS إعدادات خاصة بكل لغة ويتعامل مع الاستعلامات عبر اللغات بشكل ضعيف.

```sql
-- الإعداد
CREATE EXTENSION IF NOT EXISTS vector;

ALTER TABLE documents ADD COLUMN embedding vector(1536);
CREATE INDEX documents_embedding_idx
  ON documents USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);

-- الاستعلام: بحث دلالي
SELECT id, title, 1 - (embedding <=> $1::vector) AS similarity
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 10;
```

ملاحظة: `ivfflat` تقريبية — سريعة لكنها توازن بين الاسترجاع والدقة. للمجموعات الأصغر (أقل من ~1M صف)، غالبًا ما يكون `hnsw` خيارًا أفضل:

```sql
CREATE INDEX documents_embedding_idx
  ON documents USING hnsw (embedding vector_cosine_ops);
```

## متى تحتاج إلى كليهما

التوثيق التقني هو المكان الذي ينهار فيه التقسيم البسيط. يبحث المستخدمون عن “how to configure timeouts”، لكنهم يبحثون أيضًا عن أسماء الدوال مثل `withRetry()` وأكواد الأخطاء مثل `ECONNRESET`.

البحث المتجه يتعامل مع الاستعلامات المفهومية. البحث النصي الكامل يتعامل مع المصطلحات الدقيقة. لا أحد منهما يتعامل مع كليهما بشكل جيد بمفرده.

الحل هو البحث الهجين: تشغيل كلا النظامين ودمج النتائج.

**الدمج الترتيبي المتبادل (RRF)** هو الخوارزمية القياسية هنا. لا يتطلب منك تطبيع الدرجات من نظامين؛ إنه يجمع مواضع الترتيب.

```sql
-- بحث هجين باستخدام الدمج الترتيبي المتبادل
WITH fts_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM documents, to_tsquery('english', $1) query
  WHERE search_vector @@ query
  LIMIT 50
),
vector_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> $2::vector) AS rank
  FROM documents
  ORDER BY embedding <=> $2::vector
  LIMIT 50
),
rrf AS (
  SELECT
    COALESCE(f.id, v.id) AS id,
    COALESCE(1.0 / (60 + f.rank), 0) + COALESCE(1.0 / (60 + v.rank), 0) AS rrf_score
  FROM fts_results f
  FULL OUTER JOIN vector_results v ON f.id = v.id
)
SELECT d.id, d.title, rrf.rrf_score
FROM rrf
JOIN documents d ON d.id = rrf.id
ORDER BY rrf_score DESC
LIMIT 10;
```

العدد `60` في المقام هو ثابت RRF — القيم الأعلى تقلل من تأثير اختلاف الترتيب، والقيم الأقل تضخمها. القيمة الافتراضية 60 تعمل جيدًا في معظم الحالات.

هذا ينفّذ بحثين في استعلام واحد، يدمج الترتيب، ويكافئ النتائج التي تتطابق فيها إشارات الكلمات المفتاحية والإشارات الدلالية.

![خط أنابيب بحث هجين حيث يُفرّع استعلام واحد إلى البحث النصي الكامل وpgvector، كل منهما ينتج نتائج مرتبة، والدمج الترتيبي المتبادل يجمع القائمتين.](../hybrid-rrf-pipeline.svg)

_RRF ذو قيمة لأنه يتجنب الادعاء بأن `ts_rank` والمسافة الكوسينية قابلتان للمقارنة كدرجات خام. هو فقط يسأل: “ما مدى ارتفاع ظهور هذه النتيجة في كل قائمة؟”_

## شجرة اتخاذ القرار العملية

عند اختيار استراتيجية البحث، ابدأ بـ **شكل الإدخال**، ثم اسأل **ما نوع الاستعلام الذي يقدمه المستخدم**. “سلسلة قصيرة مع اختلاف إملائي” ليست نفس المشكلة التي “نص سردي طويل حيث تهم المصطلحات الدقيقة”، وكلاهما يختلف عن “سؤال حول أجزاء المستند”.

![شجرة قرار تختار pg_trgm أو البحث النصي الكامل أو pgvector أو البحث الهجين أو قاعدة بيانات متجهات مخصصة بناءً على شكل النص وطريقة استعلام المستخدم.](../search-decision-tree.svg)

نفس الشجرة بالكلمات:

- **الأسماء، العناوين، العناوين الفرعية، الإكمال التلقائي، أو السلاسل القصيرة المعرضة للأخطاء الإملائية** → `pg_trgm`
- **الكلمات المعروفة، رموز الأخطاء، رموز المخزون (SKU)، أسماء الدوال، الوسوم، الفئات، الفلاتر** → FTS
- **الأسئلة، الصياغات المعادلة، التوصيات، العناصر ذات الصلة، المطابقة متعددة اللغات، أجزاء RAG** → pgvector
- **المحتوى التقني حيث يحتاج المستخدمون إلى كل من الرموز الدقيقة والإجابات المفهومية** → هجين مع RRF
- **المفاتيح الأساسية، المعرفات الدقيقة، فلاتر الأذونات، نطاقات التاريخ، القوائم المرتبة** → فهارس SQL عادية
- **حجم المتجهات الضخم، معدل التزامن العالي جداً، أو أهداف الكمون التي لا يستطيع PostgreSQL تحقيقها في اختباراتك** → تقييم مخازن المتجهات المخصصة

### FTS مقابل الدلالي: النسخة المختصرة

السؤال “هل يجب أن أستخدم FTS أم البحث المتجهي?” يختزل عادة إلى ما يلي: **هل تعرف الكلمات التي ستظهر في المستندات ذات الصلة؟**

- إذا كان الجواب نعم – يبحث المستخدمون عن مصطلحات معروفة، فئات، أسماء دوال، رموز منتجات – فإن FTS أسرع، أرخص، وأكثر قابلية للتنبؤ. يوضح لك سبب مطابقة النتيجة.
- إذا كان الجواب لا – يصف المستخدم مفهومًا، يطرح سؤالًا، أو يبحث بلغة مختلفة – فإن البحث المتجهي هو الأداة المناسبة. يطابق المعنى، لا الكلمات.

الوسط الصعب هو الاستعلامات بلغة طبيعية على محتوى تقني. شخص يبحث عن “كيف أتعامل مع انقطاع الاتصال” قد يحتاج إلى مقالة بعنوان “تنفيذ منطق إعادة المحاولة لفشل الشبكة” – لا توجد كلمات متداخلة، لكن الصلة الدلالية عالية. هنا يبرهن البحث المتجهي قيمته.

الحالة الصعبة الأخرى هي **الأسماء والأسماء الخاصة**. لا FTS ولا البحث المتجهي يتعاملان معها بشكل جيد:
- FTS سيفوت “Micheal” عند البحث عن “Michael” — الرموز مختلفة
- البحث المتجهي سيفوت الاسم تمامًا إذا لم يظهر كثيرًا في بيانات التدريب
- `pg_trgm` يتعامل مع ذلك بشكل صحيح: التشابه الإملائي، ليس الدلالي أو المعجمي

في الواقع، معظم صناديق البحث التي تتعامل مع محتوى كثيف تحتاج إلى FTS للسرعة والكلمات المفتاحية، وقد تحتاج إلى بحث هجين أو `pg_trgm` حسب ما إذا كان المستخدمون يبحثون عن أسماء. ميزة البحث الدلالي الحقيقية تعني عادةً pgvector. RAG يعني دائمًا pgvector.

## إذا كنت بحاجة إلى مخزن متجهات مخصص

بعض الأنظمة تنمو فعلاً لتتجاوز قدرات pgvector. عندما يحدث ذلك، يصبح السوق مليئًا بالضوضاء. إليك ما يهم عبر الخيارات العليا.

### مصفوفة الميزات

بعض الأعمدة تحتاج إلى توضيح قبل أن يصبح الجدول مفهومًا.

**البحث الهجين** يعني بحث كلمات مفتاحية BM25 وتطابق المتجهات في استعلام واحد، يُدمج عبر Reciprocal Rank Fusion. يمكن لـ "withRetry timeout" مطابقة اسم الدالة بدقة *وأيضًا* المستندات التي تتحدث عن "منطق إعادة المحاولة لفشل الشبكة" دلاليًا. بدون البحث الهجين، تختار وضع بحث واحد أو تُدمج استعلامين بنفسك. "Manual (RRF via SQL)" في pgvector هو [النهج الموضح أعلاه](#when-you-need-both): يعمل، لكن عليك كتابة التنفيذ بنفسك.

**المتجهات المتناثرة** تتجاوز BM25. المتجه المتناثر SPLADE يحتوي على ~30,000 بُعد (واحد لكل مصطلح في المفردات)، ~98% أصفار. المواقع غير الصفرية تخبرك أي المصطلحات مهمة وكمية وزنها. استعلام "dogs" يضيف وزنًا لـ "canine" و "pet": دقة كلمات مفتاحية على مستوى BM25 مع توسيع المصطلحات داخل فهرس المتجهات. إذا كان هذا العمود غير مفعّل، ستحتاج إلى طبقة FTS خارجية لاستعلامات الكلمات الدقيقة.

```python
# SPLADE: ~30,000 dims total, ~60 non-zero — only the relevant vocabulary positions fire
def encode_splade(text: str) -> dict:
    tokens = tokenizer(text, return_tensors="pt", truncation=True, max_length=512)
    with torch.no_grad():
        output = model(**tokens)
    vec = torch.log1p(torch.relu(output.logits)).max(dim=1).values.squeeze()
    return {"indices": vec.nonzero().squeeze().tolist(), "values": vec[vec != 0].tolist()}
```

**SQL / شبيهة SQL** تتعلق أساسًا بالتصفيات. البحث المتجهي بدون تصفية هو مجرد عرض توضيحي، وليس تطبيقًا عمليًا: ما زلت بحاجة إلى نطاق المستأجر، فترات زمنية، أذونات، وفئات. SQL الكامل (pgvector) يعبر عن ذلك إلى جانب عمليات الانضمام الحالية. قواعد البيانات المصممة خصيصًا تستخدم كائنات تصفية JSON (Qdrant, Pinecone)، لغة استعلام DSL (Elasticsearch, Milvus)، أو GraphQL (Weaviate). كلها تعمل؛ يصبح SQL أكثر جاذبية كلما تعقّدت منطقية التصفيات.

```sql
-- pgvector: vector similarity is just another expression in WHERE
SELECT id, title, 1 - (embedding <=> $1) AS score
FROM documents
WHERE tenant_id = $2 AND category = ANY($3::text[]) AND created_at > NOW() - INTERVAL '90 days'
ORDER BY embedding <=> $1 LIMIT 10;
```

**الدعم المتعدد الوسائط أصلي** لا يعني مجرد "تخزين تمثيلات الصور"; كل قاعدة بيانات تخزن مصفوفات أعداد عائمة. ما يعنيه هو أن قاعدة البيانات تُوزّع نماذج تمثيل للبيانات غير النصية، بحيث تُعطيها عنوان صورة خام وتتعامل مع التحويل إلى متجه تلقائيًا. معظم قواعد البيانات هنا لا ترتبط بنوع معين من التمثيلات، لذا أنت المسؤول عن خط الأنابيب. Marqo و Weaviate (من خلال وحدات CLIP/ImageBind) يغلقان الحلقة.

**فهرس قائم على القرص** هو سؤال تكلفة. يمكن أن يتطلب HNSW المقيم في الذاكرة عدة جيجابايت من RAM لكل مليون متجه بأبعاد 1536 عندما تُحسب المتجهات الخام، وحمولة الرسم البياني، والبيانات الوصفية. البدائل الأصلية على القرص (Milvus DiskANN، Elasticsearch DiskBBQ، تنسيق Lance في LanceDB، طبقة التخزين الكائني في Turbopuffer) غالبًا ما تُقايض بعض زمن الاستعلام مقابل بنية تحتية أرخص. بالنسبة لأعباء عمل RAG التي يهيمن فيها زمن استجابة النموذج، يكون هذا القايض غالبًا جديرًا بالاختبار. Redis VSS هو القيد الصارم: ذاكرة RAM فقط، لا مسار قرص.

**الحد الأقصى للأبعاد** هو ترحيل الغد المخفي في اختيار اليوم. `text-embedding-3-large` يستخدم 3072 بُعدًا، وJina v3 يمكنه إنتاج تمثيلات أكبر، والنماذج البحثية تستمر في دفع الأبعاد أعلى. بعض الخدمات المدارة تنشر حدودًا ثابتة للأبعاد؛ أخرى توثق حدودًا عالية أو لا حدود عملية للنماذج الشائعة. تحقق من الوثائق الحالية قبل الالتزام. اختر شيئًا يترك مساحة للهوامش؛ ترحيل فهرس متجه لأنك وصلت إلى سقف الأبعاد هو سباق مرهق.

### قراءة المصفوفة

بعض الأمور لا تتناسب نظريًا مع جدول:

**المتجهات المتناثرة** هي الطريقة التي تحصل من خلالها على مطابقة كلمات مفتاحية بجودة BM25 داخل فهرس متجه، دون محرك نص كامل منفصل. Qdrant وElasticsearch لديهما تطبيقات ناضجة بشكل خاص هنا. Weaviate يدعمها عبر BM25F. إذا كان البحث الهجين أمرًا حاسمًا ولا يمكنك تشغيل نظامين، ابحث عن دعم المتجهات المتناثرة.

**الفهارس القائمة على القرص** هي رافعة تكلفة، ليست مجرد تفاصيل تنفيذ. فهارس HNSW المقيمة في RAM سريعة لكنها قد تصبح مكلفة مع زيادة عدد المتجهات، وعدد الأبعاد، والبيانات الوصفية، وحمولة الرسم البياني. البدائل القائمة على القرص (Milvus DiskANN، Elasticsearch DiskBBQ، تخزين كائنات Turbopuffer، تنسيق Lance في LanceDB) تقايض زمن الاستعلام مقابل تكلفة بنية تحتية أقل. لأحجام فهارس RAG الكبيرة، غالبًا ما يكون هذا القايض جديرًا بالاختبار.

**متعدد المستأجرين في Turbopuffer** مبني حول عدد مساحات اسمية كبير جدًا. توجيههم العام وقصص العملاء يبرزون أحمالًا مثل مجموعة Notion الضخمة ذات المساحات الاسمية الكثيفة. إذا كان كل مستخدم أو مؤسسة تحتاج إلى بحث متجه معزول، يمكن لهذا النموذج تغيير الاقتصاديات، لكن لا يزال عليك اختبار شكل المستأجر الخاص بك.

**وضعية LanceDB المدمجة** هي أقرب شيء إلى "SQLite للبحث المتجه". تعمل داخل العملية، لا تحتاج خادم، وتعمل في Lambda، Cloudflare Workers، وبيئات الحافة. يجعل تنسيق Lance العمودي العملية المدمجة عملية على نطاق حقيقي.

**Orama هو بنية تحتية لتجربة البحث، وليس مخزنًا.** إنه ممتاز عندما تريد محرك بحث صغير يدعم النص الكامل/المتجه/الهجين في تطبيق JavaScript، على الحافة، أو كطبقة بحث مُدارة للموقع/التطبيق. ليس الأداة التي سأختارها لاسترجاع مليارات المتجهات، التحليلات الثقيلة، أو عمليات الانضمام المصفاة المعقدة.

**Vespa هو ما تلجأ إليه عندما يكون الاسترجاع مجرد نصف المنتج.** يجمع بين الاسترجاع المعجمي، البحث عن أقرب الجيران، الموترات، تعبيرات الترتيب، التجميع، والخدمة عبر الإنترنت. هذه القوة حقيقية، لكن التعقيد التشغيلي والنمذجي كذلك. يناسب فرق البحث/التوصية أكثر من “إضافة بحث دلالي إلى تطبيق CRUD الخاص بي”.

**ClickHouse يندرج في النقاش عندما يكون البحث مرتبطًا بالتحليلات.** إذا كان مصدر الحقيقة لديك هو الأحداث، السجلات، الآثار، المقاييس، أو جداول الحقائق الكبيرة، يمكن لـ ClickHouse الاحتفاظ بمسافة المتجه، التصفية، التجميع، والآن فهرسة نص كامل جادة في محرك SQL واحد. ليس قاعدة بيانات متجهات مصممة خصيصًا، لكن للاسترجاع التحليلي يمكن أن يكون الجواب الممل بطريقة ممتازة.

**Chroma أقوى في بيئات التطوير/الاختبار والنشر لتطبيقات صغيرة.** إذا كنت تستهدف مجموعات نصية ضخمة، توافر عالي، عمليات تعتمد على القرص، أو بحث هجين من الدرجة الأولى، قيم مخزن موجه للإنتاج قبل ترقية النموذج الأولي إلى بنية تحتية.

### القرار المبسط

إذا كنت قد تجاوزت فعليًا حدود pgvector — عادةً لأن المعايير تُظهر أن عدد المتجهات، التصفية، معدل الكتابة، أو زمن الاستجابة تحت حمل عالي يتجاوز قدرة Postgres — اختر بناءً على القيد:

- **منتج SaaS مع عزل لكل مستأجر** → Turbopuffer  
- **تحتاج إلى أداء مستوى Rust + تصفية بيانات تعريفية معقدة** → Qdrant  
- **موجود بالفعل على مجموعة Elastic/ELK** → Elasticsearch مع DiskBBQ  
- **بيئة AWS تريد مفتوحة المصدر** → OpenSearch  
- **منصة بحث/توصية تحتاج ترتيبًا جادًا** → Vespa  
- **تحليلات، مراقبة، أو بحث في السجلات/الأحداث** → ClickHouse  
- **مقياس المليارات على البنية التحتية الداخلية / مستضاف ذاتيًا** → Milvus  
- **حافة / خادم بلا خادم / متعدد الوسائط** → LanceDB  
- **تطبيق JS صغير، موقع وثائق، أو تجربة بحث أصلية على الحافة** → Orama  
- **صفر عمليات، يعمل مباشرة، التكلفة ثانوية** → Pinecone  
- **متعدد الوسائط أولًا (صور، فيديو، صوت)** → Marqo  
- **موجود بالفعل على MongoDB** → Atlas Vector Search  
- **موجود بالفعل على Postgres، وتحتاج مساحة إضافية** → Supabase Vector أو Neon (كلاهما pgvector مُدار، مع أدوات أفضل)

## شيء واحد لا تفعله

لا تستخدم البحث المتجه كبحث نصي غير دقيق للأشياء التي لها إجابات صحيحة.

"Find me the user with email `dan@example.com`" ليست مشكلة بحث متجه. ولا كذلك "Find me the order with ID `ORD-12345`". تضمين `ORD-12345` وإجراء تشابه جيبي ضد جدول الطلبات سيعيد *شيئًا*، لكنه قد يكون غير صحيح. هذه مشكلات تطابق دقيق. استخدم المفتاح الأساسي أو فهرسًا عاديًا.

البحث المتجه يُعيد العنصر *الأكثر تشابهًا* في مجموعة البيانات، حتى عندما لا يكون هناك ما يطابق. لا يعرف أن لا توجد إجابة جيدة. هذا مقبول للوثائق ذات الصلة. لكنه كارثي للبحث عن سجل محدد، حيث أن مطابقة قريبة خاطئة أسوأ من نتيجة فارغة.

اعرف ما الهدف من كل أداة. معظمها موجود بالفعل في تثبيت Postgres الخاص بك. استخدمها حيثما تناسب.
