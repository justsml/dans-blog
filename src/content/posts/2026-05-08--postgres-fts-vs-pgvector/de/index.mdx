Teams, die AI-Funktionen hinzufügen, greifen oft zuerst zu einem dedizierten Vektor-Datenbank-Dienst.

Pinecone, Weaviate, Qdrant, Chroma. Neuer Dienst, neue Abhängigkeit, neuer Verbindungspool, neue Rechnung und jetzt zwei Quellen der Wahrheit, die man auf dem Laufenden halten muss.

Dabei haben sie bereits PostgreSQL. PostgreSQL hat bereits `pgvector`. Außerdem bietet es seit 2008 exzellenten Volltextsuche bereits eingebaut.

Dedizierte Vektor-Datenbanken rechtfertigen ihren Einsatz bei großem Maßstab und hoher Abfragevolumen. Aber die meisten Anwendungen greifen zur zweiten Suchsystem, bevor das erste überhaupt sehr hart gedrückt wurde. So wird ein zukünftiges Skalierungsproblem zu heute's Synchronisationsfehler.

Also: Wann verwenden Sie FTS, wann pgvector und wann beide?

---

## Was jede eigentlich tut

Die Volltextsuche (`tsvector` / GIN-Index) ist lexikalisch. Sie tokenisiert Text in Lexeme, leitet sie ab und vergleicht Abfragen gegen den Index. "Running" und "runs" kollabieren zum selben Lexem. So auch "dog" und "dogs." Die Rangfunktion (`ts_rank`) belohnt Dokumente, in denen Abfragebegriffe oft oder prominent auftauchen.

pgvector ist semantisch. Es speichert einen dichten Vektor - eine Liste von Zahlen - die einen Chunk *Bedeutung* repräsentiert, so verstanden von einem Embedding-Modell. Ähnlichkeitssuche findet nahe Vektoren in diesem hochdimensionalen Raum. "Dog" und "canine" landen in der Nähe. "Running" als Sport und "running" als Prozess vielleicht nicht.

Die praktische Differenz: FTS antwortet "welche Dokumente enthalten diese Wörter?" Vektor-Suche antwortet "welche Dokumente bedeuten grob das Gleiche?"

![Eine Suchwerkzeug-Karte, die pg_trgm für kurze unscharfe Strings, Volltextsuche für exakte Prosa-Abfragen, pgvector für semantische Übereinstimmung und hybride Suche für langen Inhalt, der sowohl exakte als auch semantische Signale benötigt, zeigt.](../search-tool-map.svg)

_Die erste Aufspaltung ist nicht "alte Suche vs. KI-Suche". Es ist die Form des Textes und die Art der Antwort, die korrekt sein sollte._

## Wenn Volltextsuche gewinnt

_Sie suchen nach Begriffen, die genau wichtig sind. ProduktsKCs, Fehlercodes, Modellnummern, Benutzernamen, Vertragsreferenzen. `SKU-AX-44192` ist nicht semantisch ähnlich zu etwas. Es passt entweder oder nicht. Vektor-Suche könnte zuversichtlich `SKU-AX-44193` zurückgeben. Das ist nicht das, was Sie wollen._

_Ihre Abfragen sind schlüsselbasiert. Benutzer tippen in ein Suchfeld, filtern nach Tag oder durchsuchen Blogposts nach Schlüsselwörtern. FTS wurde genau für diese Art von Absicht gebaut._

_Sie benötigen gerankte Ergebnisse ohne GPU oder Embedding-Infrastruktur. FTS-Indizes sind schnell, deterministisch und erfordern keine externen API-Aufrufe. Fügen Sie eine `tsvector`-Spalte hinzu, erstellen Sie einen GIN-Index und Sie sind fertig._

_Sie führen eine boolesche Filterung zusammen mit der Suche durch. `WHERE to_tsvector(body) @@ to_tsquery('postgres') AND category = 'tutorial' AND published_at > NOW() - INTERVAL '6 months'` — diese komponieren natürlich mit Ihrer bestehenden Abfrage-Logik._

```sql
-- Erstellen Sie den Index
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body, '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Abfrage
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

Die `GENERATED ALWAYS AS`-Spalte hält den Index automatisch aktualisiert. Die `setweight`-Funktion gibt Titel-Matches höheren Rang als Körper-Matches. Das ist die gesamte Einrichtung.

## Wenn Trigramme gewinnen (pg_trgm)

Es gibt ein drittes Postgres-Werkzeug, das häufig übergangen wird: `pg_trgm`. Es ist weder Volltextsuche noch Vektor-Suche. Es ist unscharfe Zeichenfolgenübereinstimmung und abdeckt die unscharfe Mitte, die beide Werkzeuge schlecht handhaben.

_Der Use-Case: Namen, Adressen, Identifikatoren und kurze Strings mit Tippfehlern._

FTS tokenisiert Text in Lexeme und leitet sie ab. Das funktioniert für Prosa, ist aber eine schlechte Passform für:
- Personennamen ("Dan Levy" → geleitet auf "dan levi", "leiv", hängt von Sprachkonfiguration ab)
- Unternehmensnamen, Adressen, Produkttitel, bei denen exakte Schreibweise wichtig ist
- Abfragen mit Tippfehlern — "Micheal Jordan", "Amaon", "javascipt"
- Autocomplete / Präfixsuche
- Teilzeichenfolgenabgleich ("son" entspricht "Johnson", "Anderson")

pgvector ist hier auch eine schlechte Wahl. Sie können "Micheal Jordan" embedden und den nächsten Vektor finden, aber der Embedding-Raum organisiert Namen nach Bedeutung, nicht nach Schreibweise. Der nächste Nachbar könnte sein "basketball legend" oder "Michael B. Jordan", nicht der Benutzerdatensatz mit dem Tippfehler.

`pg_trgm` bricht Strings in überlappende 3-Zeichen-Schnitte auf und misst, wie viele Trigramme zwei Strings gemeinsam haben. "Dan" -> `" da"`, `"dan"`, `"an "`. "Micheal" und "Michael" teilen die meisten ihrer Trigramme, also ist die Ähnlichkeit hoch.

```sql
-- Aktivieren Sie die Erweiterung (normalerweise bereits verfügbar)
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- GIN-Index auf Namen-Spalte — ermöglicht schnelle Trigramm-Ähnlichkeitssuche
CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- Unscharfe Namenssuche: findet "Micheal Jordan", wenn nach "Michael Jordan" gesucht wird
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % Operator = Ähnlichkeitsschwelle (Standard 0,3)
ORDER BY score DESC
LIMIT 10;

-- Oder verwenden Sie ILIKE mit Trigramm-Index-Unterstützung für Enthaltens-Matching
SELECT id, name
FROM users
WHERE name ILIKE '%johnson%'   -- GIN-Index macht dies schnell
LIMIT 10;
```

Der `%`-Operator verwendet `pg_trgm.similarity_threshold` (Standard 0,3, Bereich 0-1). Höhere Werte erfordern engere Übereinstimmungen. Für Namenssuche ist 0,3-0,4 normalerweise richtig: genug nachsichtig, um Tippfehler zu fangen, streng genug, um Geräusche zu vermeiden.

_Trigramme helfen auch bei der Präfixsuche und dem Autocomplete, insbesondere wenn der Autocomplete Tippfehler-Toleranz oder Enthaltens-Matching benötigt:_

```sql
-- Autocomplete: Präfixabgleich. Für rechteckige links-verankerte Präfixe,
-- vergleichen Sie Trigramm-GIN mit einem B-Tree-Muster-Index auf Ihren Daten.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- Mehr Kontrolle: word_similarity für teilweise Übereinstimmungen innerhalb längerer Strings
-- (nützlich beim Suchen nach "Johnson" in "Andrew Johnson III")
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name          -- <% Operator = word_similarity-Schwelle
ORDER BY score DESC
LIMIT 10;
```

_Wann Sie zu `pg_trgm` greifen sollten gegenüber FTS:_

| Szenario | Verwenden |
|---|---|
| Person/Firmen-Namenssuche mit Tippfehlern | `pg_trgm` |
| Autocomplete / Präfixsuche | `pg_trgm` (oder FTS mit Präfixabfragen) |
| Suchen nach kurzen Strings, Codes, Identifikatoren | `pg_trgm` |
| Suchen nach Artikelprosa, Dokumentation | FTS |
| Suchen nach Logmeldungen nach Schlüsselwörtern | FTS |
| Mehrsprachige Namenssuche | `pg_trgm` (es ist sprachunabhängig) |

`pg_trgm` komponiert auch mit FTS. Verwenden Sie Trigramme für eine unscharfe Vorfilterung und ranken Sie mit `ts_rank`, oder kombinieren Sie die Trigramm-Ähnlichkeit mit einem Vektor-Score.

## Wenn pgvector gewinnt

_Sie bauen RAG. RAG hängt von semantischer Retrieval-Abrufung ab: finde Dokument *Chunks* deren Bedeutung dem Benutzerfrage am nächsten ist, auch wenn sich der Wortlaut unterscheidet. Vektor-Suche ist speziell dafür gebaut. FTS wird Paraphrasen, Synonyme und konzeptionelle Matches verfehlen._

_Nutzer beschreiben, was sie wollen, nicht was sie suchen sollen. "Etwas Leichtes für einen Sommerabend" hat keine offensichtlichen Wein-Schlüsselwörter. "Artikel über das Aufbau von Selbstvertrauen als neuer Manager" erfordert semantisches Verständnis, das FTS nicht bieten kann._

_Sie finden ähnliche Artikel. Verwandte Produkte, ähnliche Support-Tickets, doppelte Bug-Reports. "Finde mir ähnliche Issues zu diesem" ist eine Vektoroperation. Sie embedden das neue Issue und finden seine nächsten Nachbarn._

_Mehrsprachiger Inhalt. Vektor-Embeddings, die mit mehrsprachigen Daten trainiert wurden, können über Sprachen abgleichen. FTS erfordert sprachspezifische Konfigurationen und verarbeitet mehrsprachige Abfragen schlecht._

```sql
-- Einrichtung
CREATE EXTENSION IF NOT EXISTS vector;

ALTER TABLE documents ADD COLUMN embedding vector(1536);
CREATE INDEX documents_embedding_idx
  ON documents USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);

-- Abfrage: semantische Suche
SELECT id, title, 1 - (embedding <=> $1::vector) AS similarity
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 10;
```

Hinweis: `ivfflat` ist ungenau — es ist schnell, tauscht aber eine gewisse Genauigkeit für Geschwindigkeit ein. Für kleinere Datensätze (unter ~1M Zeilen) ist `hnsw` oft besser:

```sql
CREATE INDEX documents_embedding_idx
  ON documents USING hnsw (embedding vector_cosine_ops);
```

## Wenn Sie beide benötigen

Technische Dokumentation ist der Ort, an dem die einfache Aufspaltung bricht. Nutzer suchen nach "how to configure timeouts", suchen aber auch nach Funktionsnamen wie `withRetry()` und Fehlercodes wie `ECONNRESET`.

Vektor-Suche behandelt konzeptionelle Abfragen. FTS behandelt exakte Begriffe. Keines der beiden verarbeitet beide gut allein.

Die Lösung ist hybride Suche: führen Sie beide aus und verschmelzen Sie die Ergebnisse.

_Reciprocal Rank Fusion (RRF) ist der Standard-Algorithmus hier. Er verlangt nicht, dass Sie Scores aus beiden Systemen normalisieren; er kombiniert Rangpositionen._

```sql
-- Hybride Suche mit Reciprocal Rank Fusion
WITH fts_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM documents, to_tsquery('english', $1) query
  WHERE search_vector @@ query
  LIMIT 50
),
vector_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> $2::vector) AS rank
  FROM documents
  ORDER BY embedding <=> $2::vector
  LIMIT 50
),
rrf AS (
  SELECT
    COALESCE(f.id, v.id) AS id,
    COALESCE(1.0 / (60 + f.rank), 0) + COALESCE(1.0 / (60 + v.rank), 0) AS rrf_score
  FROM fts_results f
  FULL OUTER JOIN vector_results v ON f.id = v.id
)
SELECT d.id, d.title, rrf.rrf_score
FROM rrf
JOIN documents d ON d.id = rrf.id
ORDER BY rrf_score DESC
LIMIT 10;
```

Die `60` im Nenner ist die RRF-Konstante — höhere Werte reduzieren den Einfluss von Rangunterschieden, niedrigere Werte verstärken sie. Der Standard von 60 funktioniert in den meisten Fällen gut.

Dies führt zwei Suchen in einer Abfrage aus, verschmilzt die Ränge und belohnt Ergebnisse, in denen Schlüsselwort- und semantische Signale übereinstimmen.

![Eine hybride Suchpipeline, bei der eine Abfrage nach Volltextsuche und pgvector aufzweigt, jeweils gerankte Ergebnisse liefert und Reciprocal Rank Fusion die beiden Listen kombiniert.](../hybrid-rrf-pipeline.svg)

_RRF ist wertvoll, weil er vermeidet, `ts_rank` und Kosinusabstand als vergleichbare Rohwerte zu betrachten. Es fragt nur: "Wie hoch ist dieses Ergebnis in jeder Liste erschienen?"_

## Die praktische Entscheidungsstruktur

Bei der Auswahl einer Suchstrategie beginnen Sie mit der **Form des Inputs**, dann fragen Sie **welche Art von Abfrage der Benutzer macht**. "Kurzer String mit Schreibvariation" ist nicht dasselbe Problem wie "lange Prosa, bei der exakte Begriffe wichtig sind", und beide unterscheiden sich von "Frage über Dokument-Chunks".

![Eine Entscheidungsstruktur, die pg_trgm, Volltextsuche, pgvector, hybride Suche oder eine dedizierte Vektor-Datenbank basierend auf der Form des Textes und der Art auswählt, wie Nutzer darin suchen.](../search-decision-tree.svg)

Das Baumdiagramm in Worten:

- **Namen, Adressen, Titel, Autocomplete oder Tippfalle- kurze Strings** → `pg_trgm`
- **Bekannte Wörter, Fehlercodes, SKUs, Funktionsnamen, Tags, Kategorien, Filter** → FTS
- **Fragen, Paraphrasen, Empfehlungen, ähnliche Artikel, mehrsprachige Übereinstimmung, RAG-Chunks** → pgvector
- **Technischer Inhalt, bei dem Nutzer sowohl exakte Symbole als auch konzeptionelle Antworten benötigen** → Hybrid mit RRF
- **Primärschlüssel, exakte IDs, Berechtigungsschutz, Datumsbereiche, sortierte Listen** → normale SQL-Indizes
- **Große Vektorvolumen, sehr hohe Konkurrenz oder Latenzziele, die Postgres in Ihren Benchmarks nicht erreichen kann** → dedizierte Vektorstores bewerten

### FTS vs. Semantik: Die Kurzversion

Die Frage "soll ich FTS oder Vektor-Suche verwenden?" kollabiert meistens zu dieser: **wissen Sie, welche Wörter in den relevanten Dokumenten erscheinen werden?**

Wenn ja - Nutzer suchen nach bekannten Begriffen, Kategorien, Funktionsnamen, Produktcodes - FTS ist schneller, günstiger und vorhersehbarer. Er sagt Ihnen, warum ein Ergebnis übereinstimmte.

Wenn nein - Nutzer beschreiben ein Konzept, stellen eine Frage oder suchen in einer anderen Sprache - Vektor-Suche ist das richtige Werkzeug. Sie entsprechen Bedeutung, nicht Wörter.

Die schwierige Mitte sind NATÜRLICHE SPRACHABFRAGEN über technischen Inhalt. Jemand, der nach "how do I handle connection drops" sucht, könnte einen Artikel mit dem Titel "implementing retry logic for network failures" benötigen - keine überlappenden Wörter, hohe semantische Relevanz. Dort verdient sich Vektor-Suche ihren Einsatz.

Das andere schwierige Szenario sind **Namen und Eigennamen**. Weder FTS noch Vektor-Suche sind gut darin:
- FTS wird "Micheal" verfehlen, wenn nach "Michael" gesucht wird — unterschiedliche Tokens
- Vektor-Suche wird den Namen völlig verfehlen, wenn er nicht häufig in den Trainingsdaten erscheint
- `pg_trgm` verarbeitet dies korrekt: orthografische Ähnlichkeit, nicht semantisch oder lexikalisch

In der Praxis benötigen die meisten inhaltsreichen Suchfelder FTS für Geschwindigkeit und Schlüsselwörter und möglicherweise hybride oder `pg_trgm` je nachdem, ob Nutzer nach Namen suchen. Eine wahre semantische Suche-Funktion bedeutet meistens pgvector. RAG bedeutet immer pgvector.

## Wenn Sie wirklich eine dedizierte Vektor-Datenbank benötigen

Einige System wachsen wirklich über pgvector hinaus. Wenn das passiert, ist der Markt laut. Hier sind die Dinge, die über die Top-Optionen wichtig sind.

### Die Feature-Matrix

Einige Spalten müssen vor der Tabelle verständlich gemacht werden.

_Hybrid Suche bedeutet BM25-Schlüsselwortsuche und Vektor-Ähnlichkeit, die in einer Abfrage ausgeführt werden und über Reciprocal Rank Fusion verschmolzen werden. "withRetry timeout" kann den Funktionsnamen genau *und* Dokumente über "retry logic for network failures" semantisch übereinstimmen. Ohne Hybrid wählen Sie einen der beiden Suchmodi oder verschmelzen Sie zwei Abfragen selbst. pgvector's "Manual (RRF via SQL)" ist [der oben gezeigte Ansatz](#wenn-sie-beide-benötigen): es funktioniert, aber Sie schreiben es selbst._

_Sparse Vektoren gehen weiter als BM25. Ein SPLADE-Sparse-Vektor hat etwa 30,000 Dimensionen (eine pro Vokabularbegriff), etwa 98 % Nullen. Die nicht-Null-Positionen sagen Ihnen, welche Begriffe wichtig sind und wie viel. Eine Abfrage für "dogs" gewichtet auch "canine" und "pet": BM25-Level Schlüsselwortpräzision plus Termausweitung innerhalb eines Vektor-Indexes. Wenn diese Spalte false ist, benötigen Sie eine externe FTS-Schicht für exakte-Begriff-Abfragen._

```python
# SPLADE: ~30,000 Gesamt-Dims, ~60 nicht-null — nur die relevanten Vokabel-Positionen feuern
def encode_splade(text: str) -> dict:
    tokens = tokenizer(text, return_tensors="pt", truncation=True, max_length=512)
    with torch.no_grad():
        output = model(**tokens)
    vec = torch.log1p(torch.relu(output.logits)).max(dim=1).values.squeeze()
    return {"indices": vec.nonzero().squeeze().tolist(), "values": vec[vec != 0].tolist()}
```

_SQL / SQL-ähnlich ist wirklich über Filterung. Vektor-Suche ohne Filterung ist eine Demo, nicht eine Anwendung: Sie benötigen immer noch Mietobjekt-Grenze, Datumsbereiche, Berechtigungen und Kategorien. Vollständiger SQL (pgvector) drückt dies neben Ihrer bestehenden Joins. Zweckorientierte Datenbanken verwenden JSON-Filter-Objekte (Qdrant, Pinecone), eine Abfrage-DSL (Elasticsearch, Milvus) oder GraphQL (Weaviate). Sie funktionieren; SQL wird attraktiver, wenn die Filterlogik verstrickt wird._

```sql
-- pgvector: Vektor-Ähnlichkeit ist nur ein weiterer Ausdruck in WHERE
SELECT id, title, 1 - (embedding <=> $1) AS score
FROM documents
WHERE tenant_id = $2 AND category = ANY($3::text[]) AND created_at > NOW() - INTERVAL '90 days'
ORDER BY embedding <=> $1 LIMIT 10;
```

```python
# Qdrant: äquivalenter Filter als Python-Objekt — funktional, mehr Zeremonie
results = client.query_points(
    collection_name="documents", query=query_embedding,
    query_filter=models.Filter(must=[
        models.FieldCondition(key="tenant_id", match=models.MatchValue(value=tenant_id)),
        models.FieldCondition(key="category",  match=models.MatchAny(any=categories)),
        models.FieldCondition(key="created_at", range=models.DatetimeRange(gte=cutoff)),
    ]),
    limit=10,
)
```

_Multimodal native heißt nicht "kann Bild-Embeddings speichern"; jede Datenbank speichert Float-Arrays. Es heißt, dass die Datenbank Embedding-Module für nicht-text Inhalt liefert, sodass Sie ihr eine rohe Bild-URL geben und es das Vektorisierung handhabt. Die meisten Datenbanken hier sind Embedding-agnostisch, also besitzen Sie diese Pipeline. Marqo und Weaviate (via CLIP/ImageBind-Module) schließen die Schleife._

```python
# Marqo: POST rohe Bilder, Abfrage mit Text — kein externer Embedding-Schritt nötig
mq.index("products").add_documents(
    [{"id": "shoe-001", "image": "https://cdn.example.com/shoes/001.jpg"}],
    tensor_fields=["image"]
)
results = mq.index("products").search(q="lightweight shoes for summer")
# Gibt shoe-001 zurück, obwohl es keine Schlüsselwortübereinstimmung gibt — CLIP behandelt den Cross-Modal-Match
```

_Disk-basierte Indizes sind eine Kosten-Schraube, kein Implementierungsdetail. RAM-resident HNSW-Indizes können mehrere GB RAM pro Million 1536-Dimension-Vektoren erfordern, wenn rohe Vektoren, Graphen-Overhead und Metadaten gezählt werden. Speicher-native Alternativen (Milvus DiskANN, Elasticsearch DiskBBQ, LanceDB's Lance-Format, Turbopuffer's Object Storage-Tier) tauschen manchmal einige Abfrage-Latenz gegen günstigere Infrastruktur. Für RAG-Workloads, bei denen die Modell-Latenz bereits dominiert, ist dieser Austausch oft worth testing. Redis VSS ist die harte Einschränkung: RAM-nur, kein Pfad zum Speicher._

_Max Dimensionen ist morgen's Migration, die in heute's Wahl versteckt ist. `text-embedding-3-large` verwendet 3072 Dims, Jina v3 kann größere Embeddings emittieren, und Forschungsmodelle drängen weiter auf höhere. Einige verwaltete Dienste veröffentlichen harte Dimension-Grenzen; andere dokumentieren hohe Grenzen oder keine praktische Grenze für typische Embedding-Modelle. Prüfen Sie die aktuellen Dokumente, bevor Sie sich binden. Wählen Sie etwas mit Pufferraum; ein Vektor-Index zu migrieren, weil Sie eine Dimension-Obergrenze erreichen, ist eine elende Sprint._

### Die Matrix lesen

Einige Dinge passen nicht sauber in eine Tabelle:

_Sparse Vektoren sind die Art, wie Sie BM25-Qualität Schlüsselwortabgleich innerhalb eines Vektor-Indexes erhalten, ohne separate Volltext-Suche. Qdrant und Elasticsearch haben besonders ausgereifte Implementierungen hier. Weaviate unterstützt sie über BM25F. Wenn Hybrid-Suche kritisch ist und Sie zwei Systeme nicht ausführen können, suchen Sie nach Sparse-Vektor-Unterstützung._

_Disk-basierte Indizes sind eine Kostenhebel, kein Implementierungsdetail. RAM-resident HNSW-Indizes sind schnell, können aber teuer werden, wenn Vektor-Anzahl, Dimensionen-Anzahl, Metadaten und Graphen-Overhead wachsen. Disk-basierte Alternativen (Milvus DiskANN, Elasticsearch DiskBBQ, Turbopuffer Object Storage, LanceDB's Lance-Format) tauschen Abfrage-Latenz gegen geringere Infrastruktur-Kosten. Für große RAG-Indizes ist dieser Austausch oft worth testing._

_Turbopuffer's Multi-Tenancy ist um sehr hohe NamespacesCounts herum aufgebaut. Sein öffentliches Positionieren und Kundengeschichten betonen Workloads wie Notion's large, namespace-heavy corpus. Wenn jeder Benutzer oder jede Organisation isolierte Vektor-Suche benötigt, kann diese Architektur die Ökonomie ändern, aber testen Sie Ihren eigenen Tenant-Shape._

_LanceDB embedded mode ist das Ähnlichste mit "SQLite für Vektor-Suche". Es läuft im Prozess, benötigt keinen Server und funktioniert in Lambda, Cloudflare Workers und Edge-Umgebungen. Das Lance columnar-Format macht eingebettete Operationen bei echtem Maßstab praktisch._

_Orama ist Such-UX-Infrastruktur, kein Lagerhaus. Es ist ausgezeichnet, wenn Sie eine winzige Volltext/Vector/Hybrid-Suchmaschine in einer JavaScript-App, auf dem Edge oder als managed site/app Suchschicht möchten. Es ist nicht das Werkzeug, das ich für Milliardenvector-Retrieval, schwere Analysen oder komplexe gefilterte Joins wählen würde._

_Vespa ist der Ort, an den Sie sich beim Retrieval nur halb des Produkts wenden. Es kombiniert lexikalisches Retrieval, nearest-neighbor Suche, Tensoren, Rangausdrücke, Gruppierung und online Serving. Diese Kraft ist real, aber auch die operationale und modellierungskomplexität. Es passt Suchen/Empfehlungs-Teams mehr als "fügen Sie semantische Suche zu meinem CRUD-App hinzu"._

_ClickHouse gehört in die Unterhaltung, wenn Suche an Analysen angehängt ist. Wenn Ihre Quelle der Wahrheit Ereignisse, Logs, Traces, Metriken oder große Fact-Tabellen sind, kann ClickHouse Vektorabstand, Filterung, Aggregation und jetzt ernsthaft Volltext-Indizierung in einer SQL-Engine halten. Es ist nicht eine zweckorientierte Vektor-Datenbank, aber für analytisches Retrieval kann es die langweilige Antwort in der besten Art sein._

_Chroma ist am stärksten bei Dev/Testing und kleinen App-Deployment. Wenn Sie auf sehr große Korpora abzielen, HA, Speicher-lastige Operation oder first-class Hybrid-Suche, bewerten Sie einen production-orientierten Store, bevor Sie das Prototyp in die Infrastruktur befördern._

### Die vereinfachte Entscheidung

Wenn Sie pgvector wirklich outgrown haben - meistens weil Benchmarks zeigen, dass Vektor-Anzahl, Filterung, Schreibrate oder hohe Konkurrenz-Latenz Ihre Postgres-Umgebung überschreitet - wählen Sie nach Einschränkung:

- **SaaS-Produkt mit pro-Mandant-Isolation** → Turbopuffer
- **Rust-Level-Leistung + komplexe Metadaten-Filterung erforderlich** → Qdrant
- **Bereits auf Elastic/ELK-Stack** → Elasticsearch mit DiskBBQ
- **AWS-Firma, die Open-Source will** → OpenSearch
- **Suchen/Empfehlungsplattform mit ernsthaften Rangierungsbedürfnissen** → Vespa
- **Analytik, Observability oder Log/Event-Suche** → ClickHouse
- **Milliardenscale on-prem / self-hosted** → Milvus
- **Edge / serverless / multimodal** → LanceDB
- **Kleine JS-App, Dokumentensite oder edge-native Such-UX** → Orama
- **Null Ops, nur funktioniert, Kosten sind sekundär** → Pinecone
- **Multimodal-first (Bilder, Video, Audio)** → Marqo
- **Bereits auf MongoDB** → Atlas Vector Search
- **Bereits auf Postgres, mehr Pufferraum benötigen** → Supabase Vector oder Neon (beide pgvector managed, mit besserem Tooling)

## Eines, das Sie NICHT tun

Verwenden Sie Vektor-Suche nicht als unscharfe Textsuche für Dinge, die korrekte Antworten haben.

"Finde den Benutzer mit E-Mail `dan@example.com`" ist kein Vektor-Suche-Problem. Ebenso "Finde die Bestellung mit ID `ORD-12345`". Embedden Sie `ORD-12345` und führen Sie Kosinus-Ähnlichkeit gegen Ihre Bestellungs-Tabelle aus, und Sie erhalten *etwas*, aber es könnte falsch sein. Dies sind exakte-Match-Probleme. Verwenden Sie Ihren Primärschlüssel oder einen normalen Index.

Vektor-Suche gibt das *am ähnlichste* Ding in Ihrem Datensatz zurück, auch wenn nichts relevant ist. Es weiß nicht, dass keine gute Antwort existiert. Das ist fein für ähnliche Dokumente. Es ist katastrophal für bestimmte Datensatz-Suche, bei der ein falsches Near-Match schlimmer ist als ein leeres Ergebnis.

Wissen Sie, was jedes Werkzeug für etwas ist. Die meisten von ihnen sind schon in Ihrer Postgres-Installation. Verwenden Sie sie dort, wo sie passen.
