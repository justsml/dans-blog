Teams, die KI-Features hinzufügen, greifen oft zuerst zu einer dedizierten Vektordatenbank.

Pinecone, Weaviate, Qdrant, Chroma. Neuer Dienst, neue Abhängigkeit, neuer Connection Pool, neue Rechnung, und jetzt zwei Sources of Truth, die es诚实 zu halten gilt.

Dabei haben sie Postgres bereits. Postgres hat bereits `pgvector`. Und es hat seit 2008 exzellente Volltextsuche eingebaut.

Dedizierte Vektorspeicher verdienen sich ihren Platz bei großem Maßstab und hohem Query-Volumen. Aber die meisten Anwendungen greifen zum zweiten Suchsystem, bevor das erste wirklich ausgereizt wurde. So wird ein zukünftiges Skalierungsproblem zum heutigen Synchronisationsbug.

Also: Wann nutzt man FTS, wann pgvector, und wann beide?

---

## Was jedes Werkzeug wirklich tut

Volltextsuche (`tsvector` / `GIN`-Index) ist lexikalisch. Sie zerlegt Text in Lexeme, stemmt sie und matched Queries gegen den Index. "Laufen" und "läuft" fallen zum gleichen Lexem zusammen. Genauso "Hund" und "Hunde." Die Ranking-Funktion (`ts_rank`) belohnt Dokumente, in denen Query-Begriffe oft oder prominent auftreten.

pgvector ist semantisch. Es speichert einen dichten Vektor – eine Liste von Zahlen –, der einen Chunk *bedeutet* so wie ihn ein Embedding-Modell versteht. Ähnlichkeitssuche findet benachbarte Vektoren in diesem hochdimensionalen Raum. "Hund" und "Canide" landen nah beieinander. "Laufen" der Sportart und "Laufen" eines Prozesses vielleicht nicht.

Der praktische Unterschied: FTS beantwortet "welche Dokumente enthalten diese Wörter?" Vektorsuche beantwortet "welche Dokumente bedeuten ungefähr dieses Ding?"

![Eine Werkzeugkarte für Suche: pg_trgm für kurze Fuzzy-Strings, Volltextsuche für exakte Prosa-Queries, pgvector für semantisches Matching, und Hybridsuche für lange Inhalte, die beide Signale brauchen.](../search-tool-map.svg)

_Der erste Schnitt ist nicht "alte Suche vs. KI-Suche." Es ist die Form des Texts und welche Art von Antwort richtig wäre._

## Wann Volltextsuche gewinnt

**Du suchst nach Begriffen, die exakt wichtig sind.** Produkt-SKUs, Fehlercodes, Modellnummern, Benutzernamen, Gesetzesklausel-Referenzen. `SKU-AX-44192` ist semantisch ähnlich zu nichts. Entweder es matcht oder nicht. Vektorsuche könnte confident `SKU-AX-44193` zurückgeben. Das ist nicht, was du willst.

**Deine Queries sind keyword-basiert.** Benutzer tippen in eine Suchbox, filtern nach Tag, oder durchsuchen Blogposts nach Keyword. FTS wurde für diese Form von Intent gebaut.

**Du brauchst rankierte Ergebnisse ohne GPU oder Embedding-Infrastruktur.** FTS-Indizes sind schnell, deterministisch und brauchen keine externen API-Aufrufe. Füge eine `tsvector`-Spalte hinzu, bau einen GIN-Index, und du bist fertig.

**Du machst boolesches Filtern neben der Suche.** `WHERE to_tsvector(body) @@ to_tsquery('postgres') AND category = 'tutorial' AND published_at > NOW() - INTERVAL '6 months'` — das komponiert natürlich mit deiner existierenden Query-Logik.

```sql
-- Create the index
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body, '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Query
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

Die `GENERATED ALWAYS AS`-Spalte hält den Index automatisch aktuell. Das `setweight` gibt Titel-Matches höheren Rank als Body-Matches. Das war's.

## Wann Trigramme gewinnen (pg_trgm)

Es gibt ein drittes Postgres-Werkzeug, das außen vor bleibt: `pg_trgm`. Es ist weder Volltextsuche noch Vektorsuche. Es ist Fuzzy-String-Matching, und es deckt die unangenehme Mitte ab, die beide Werkzeuge schlecht abdecken.

**Der Use Case: Namen, Adressen, Bezeichner und kurze Strings mit Tippfehlern.**

FTS zerlegt Text in Lexeme und stemmt sie. Das funktioniert für Prosa, aber es ist ein schlechter Fit für:
- Personennamen ("Max Mustermann" → gestemmt zu "max muster", "muster", je nach Sprachkonfiguration)
- Firmennamen, Adressen, Produkttitel, bei denen exakte Schreibweise wichtig ist
- Queries mit Tippfehlern — "Michaell Jordan", "Amazzon", "javaskript"
- Autocomplete / Prefix-Suche
- Partielles String-Matching ("Müller" matched "Mueller", "Andersson")

pgvector ist auch eine schlechte Wahl hier. Du kannst "Michaell Jordan" embedden und den nächsten Vektor finden, aber der Embedding-Space organisiert Namen nach Bedeutung, nicht nach Rechtschreibung. Der nächste Neighbor könnte "Basketball-Legende" oder "Michael B. Jordan" sein, nicht der User-Record mit dem Tippfehler.

`pg_trgm` zerlegt Strings in überlappende 3-Zeichen-Slices und misst, wie viele Trigramme zwei Strings teilen. "Dan" -> `" da"`, `"dan"`, `"an "`. "Micheal" und "Michael" teilen die meisten ihrer Trigramme, also ist die Similarity hoch.

```sql
-- Enable the extension (usually already available)
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- GIN index on names column — enables fast trigram similarity search
CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- Fuzzy name search: finds "Micheal Jordan" when searching "Michael Jordan"
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % operator = similarity threshold (default 0.3)
ORDER BY score DESC
LIMIT 10;

-- Or use ILIKE with trigram index support for contains matching
SELECT id, name
FROM users
WHERE name ILIKE '%johnson%'   -- GIN index makes this fast
LIMIT 10;
```

Der `%`-Operator nutzt `pg_trgm.similarity_threshold` (default 0.3, Range 0-1). Höhere Werte erfordern nähere Matches. Für Namenssuche ist 0.3-0.4 meistens richtig: permissiv genug für Tippfehler, streng genug um Rauschen zu vermeiden.

**Trigramme helfen auch bei Prefix-Suche und Autocomplete, besonders wenn Autocomplete Tolerantanz für Tippfehler oder Contains-Matching braucht:**

```sql
-- Autocomplete: prefix matching. For pure left-anchored prefixes,
-- compare trigram GIN against a B-tree pattern index on your data.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- More control: word_similarity for partial matches within longer strings
-- (useful when searching "Johnson" within "Andrew Johnson III")
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name          -- <% operator = word_similarity threshold
ORDER BY score DESC
LIMIT 10;
```

**Wann `pg_trgm` statt FTS verwenden:**

| Szenario | Nutzen |
|---|---|
| Personen-/Firmennamen-Suche mit Tippfehlern | `pg_trgm` |
| Autocomplete / Prefix-Suche | `pg_trgm` (oder FTS mit Prefix-Queries) |
| Kurze Strings, Codes, Bezeichner durchsuchen | `pg_trgm` |
| Prosa-Artikel, Dokumentation durchsuchen | FTS |
| Log-Meldungen nach Keywords durchsuchen | FTS |
| Mehrsprachige Namenssuche | `pg_trgm` (sprachagnostisch) |

`pg_trgm` komponiert auch mit FTS. Nutze Trigramme für ein fuzzy Pre-Filter und ranke mit `ts_rank`, oder kombiniere Trigram-Similarity mit einem Vektor-Score.

## Wann pgvector gewinnt

**Du baust RAG.** RAG hängt von semantischer Retrieval ab: finde Dokument-*Chunks*, deren Bedeutung der Frage des Users am nächsten kommt, selbst wenn die Formulierung abweicht. Vektorsuche ist dafür gebaut. FTS verpasst Paraphrasen, Synonyme und konzeptuelle Matches.

**Users beschreiben, was sie wollen, nicht wonach sie suchen sollen.** "Etwas Leichtes für einen Sommerabend" hat keine offensichtlichen Wein-Keywords. "Artikel über Selbstvertrauen als neue Führungskraft aufbauen" erfordert semantisches Verständnis, das FTS nicht liefern kann.

**Du findest ähnliche Items.** Ähnliche Produkte, ähnliche Support-Tickets, doppelte Bug-Reports. "Finde mir Issues ähnlich zu diesem hier" ist eine Vektor-Operation. Du embeddest das neue Issue und findest seine nächsten Nachbarn.

**Mehrsprachige Inhalte.** Vektor-Embeddings, die auf mehrsprachigen Daten trainiert wurden, können sprachübergreifend matchen. FTS braucht sprachspezifische Konfigurationen und handhabt abfrageübergreifende Queries schlecht.

```sql
-- Setup
CREATE EXTENSION IF NOT EXISTS vector;

ALTER TABLE documents ADD COLUMN embedding vector(1536);
CREATE INDEX documents_embedding_idx
  ON documents USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);

-- Query: semantic search
SELECT id, title, 1 - (embedding <=> $1::vector) AS similarity
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 10;
```

Beachte: `ivfflat` ist approximativ — es ist schnell, aber es tauscht etwas Recall gegen Speed. Für kleinere Datasets (unter ~1M Rows) ist `hnsw` oft besser:

```sql
CREATE INDEX documents_embedding_idx
  ON documents USING hnsw (embedding vector_cosine_ops);
```

## Wann du beide brauchst

Technische Dokumentation ist, wo der einfache Schnitt zerbricht. Users suchen nach "wie konfiguriere ich Timeouts", aber sie suchen auch nach Funktionsnamen wie `withRetry()` und Fehlercodes wie `ECONNRESET`.

Vektorsuche handhabt konzeptuelle Queries. FTS handhabt exakte Begriffe. Keines von beiden handhabt beides gut allein.

Die Lösung ist Hybridsuche: beide ausführen und die Ergebnisse fusionieren.

**Reciprocal Rank Fusion (RRF)** ist der Standard-Algorithmus hier. Er erfordert nicht, dass du Scores aus zwei Systemen normalisierst; er kombiniert Rangpositionen.

```sql
-- Hybrid search with Reciprocal Rank Fusion
WITH fts_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM documents, to_tsquery('english', $1) query
  WHERE search_vector @@ query
  LIMIT 50
),
vector_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> $2::vector) AS rank
  FROM documents
  ORDER BY embedding <=> $2::vector
  LIMIT 50
),
rrf AS (
  SELECT
    COALESCE(f.id, v.id) AS id,
    COALESCE(1.0 / (60 + f.rank), 0) + COALESCE(1.0 / (60 + v.rank), 0) AS rrf_score
  FROM fts_results f
  FULL OUTER JOIN vector_results v ON f.id = v.id
)
SELECT d.id, d.title, rrf.rrf_score
FROM rrf
JOIN documents d ON d.id = rrf.id
ORDER BY rrf_score DESC
LIMIT 10;
```

Die `60` im Nenner ist die RRF-Konstante — höhere Werte reduzieren den Einfluss von Rang-Unterschieden, niedrigere Werte verstärken sie. Der Default von 60 funktioniert in den meisten Fällen gut.

Das führt zwei Suchen in einer Query aus, fusioniert die Ränge und belohnt Ergebnisse, wo Keyword- und semantisches Signal übereinstimmen.

![Eine Hybridsuche-Pipeline, bei der eine Query sich zu Volltextsuche und pgvector verzweigt, jede rankierte Ergebnisse produziert, und Reciprocal Rank Fusion die beiden Listen kombiniert.](../hybrid-rrf-pipeline.svg)

_RRF ist wertvoll, weil es vermeidet, so zu tun, als wären `ts_rank` und Cosine-Distance vergleichbare Raw-Scores. Es fragt nur: "Wie hoch erschien dieses Ergebnis in jeder Liste?"_

## Der praktische Entscheidungsbaum

Beim Wählen einer Suchstrategie, starte mit der **Form des Inputs**, dann frag **welche Art von Query der User stellt**. "Kurzer String mit Rechtschreibvariation" ist nicht dasselbe Problem wie "lange Prosa, wo exakte Begriffe wichtig sind", und beide unterscheiden sich von "Frage über Dokument-Chunks."

![Ein Entscheidungsbaum, der pg_trgm, Volltextsuche, pgvector, Hybridsuche oder eine dedizierte Vektordatenbank wählt, basierend auf der Form des Texts und der Art, wie Users ihn abfragen.](../search-decision-tree.svg)

Derselbe Baum in Worten:

- **Namen, Adressen, Titel, Autocomplete oder fehleranfällige kurze Strings** → `pg_trgm`
- **Bekannte Wörter, Fehlercodes, SKUs, Funktionsnamen, Tags, Kategorien, Filter** → FTS
- **Fragen, Paraphrasen, Empfehlungen, ähnliche Items, mehrsprachiges Matching, RAG-Chunks** → pgvector
- **Technische Inhalte, wo Users sowohl exakte Symbole als auch konzeptuelle Antworten brauchen** → Hybrid mit RRF
- **Primary Keys, exakte IDs, Berechtigungsfilter, Datumsbereiche, sortierte Listen** → normale SQL-Indizes
- **Riesiges Vektorvolumen, sehr hohe Konkurrenz, oder Latenz-Ziele, die Postgres in deinen Benchmarks nicht erreichen kann** → evaluiere dedizierte Vektorspeicher

### FTS vs. Semantisch: Die Kurzversion

Die Frage "Soll ich FTS oder Vektorsuche nutzen?" reduziert sich meistens auf dies: **Weißt du, welche Wörter in den relevanten Dokumenten auftauchen werden?**

Wenn ja - Users suchen nach bekannten Begriffen, Kategorien, Funktionsnamen, Produktcodes - FTS ist schneller, günstiger und vorhersehbarer. Es sagt dir, warum ein Ergebnis gematcht hat.

Wenn nein - Users beschreiben ein Konzept, stellen eine Frage, oder suchen in einer anderen Sprache - Vektorsuche ist das richtige Werkzeug. Es matched Bedeutung, nicht Wörter.

Die schwierige Mitte ist natürliche Sprach-Queries über technische Inhalte. Jemand sucht "wie handle ich Connection-Abbrüche" könnte einen Artikel titled "Retry-Logik für Netzwerkfehler implementieren" brauchen - keine überlappenden Wörter, hohe semantische Relevanz. Das ist, wo Vektorsuche ihren Platz verdient.

Der andere schwierige Fall ist **Namen und Eigennamen**. Weder FTS noch Vektorsuche ist großartig darin:
- FTS wird "Micheal" verpassen, wenn nach "Michael" gesucht wird — verschiedene Tokens
- Vektorsuche wird den Namen komplett verpassen, wenn er nicht häufig genug in den Trainingsdaten auftaucht
- `pg_trgm` handhabt das korrekt: orthografische Ähnlichkeit, nicht semantisch oder lexikalisch

In der Praxis brauchen die meisten inhaltsschweren Suchboxen FTS für Speed und Keywords, und brauchen möglicherweise Hybrid oder `pg_trgm` je nachdem, ob Users nach Namen suchen. Ein echtes semantisches Such-Feature bedeutet meistens pgvector. RAG bedeutet immer pgvector.

## Wenn du einen dedizierten Vektorspeicher brauchst

Manche Systeme wachsen wirklich über pgvector hinaus. Wenn das passiert, ist der Markt laut. Hier ist, was bei den Top-Optionen zählt.

### Die Feature-Matrix

Einige Spalten brauchen Auspacken, bevor die Tabelle sinnvoll ist.

**Hybridsuche** bedeutet BM25-Keyword-Suche und Vektor-Ähnlichkeit in einer Query, zusammengeführt via Reciprocal Rank Fusion. "withRetry timeout" kann den Funktionsnamen exakt matchen *und* Dokumente über "Retry-Logik für Netzwerkfehler" semantisch. Ohne Hybrid wählst du einen Suchmodus oder fusionierst zwei Queries selbst. pgvectors "Manual (RRF via SQL)" ist [der oben gezeigte Ansatz](#when-you-need-both): es funktioniert, aber du schreibst es selbst.

**Sparse Vektoren** gehen weiter als BM25. Ein SPLADE sparse Vektor hat ~30.000 Dimensionen (eine pro Vokabular-Term), ~98% Nullen. Die Nicht-Null-Positionen sagen dir, welche Terms wichtig sind und wie viel. Eine Query nach "Hunde" gewichtet auch "Canide" und "Haustier": BM25-Level Keyword-Präzision plus Term-Expansion innerhalb eines Vektor-Index. Wenn diese Spalte false ist, brauchst du eine externe FTS-Schicht für exakte-Term-Queries.

```python
# SPLADE: ~30,000 dims total, ~60 non-zero — only the relevant vocabulary positions fire
def encode_splade(text: str) -> dict:
    tokens = tokenizer(text, return_tensors="pt", truncation=True, max_length=512)
    with torch.no_grad():
        output = model(**tokens)
    vec = torch.log1p(torch.relu(output.logits)).max(dim=1).values.squeeze()
    return {"indices": vec.nonzero().squeeze().tolist(), "values": vec[vec != 0].tolist()}
```

**SQL / SQL-like** ist wirklich über Filtern. Vektorsuche ohne Filtern ist eine Demo, keine Anwendung: du brauchst trotzdem Tenant-Scope, Datumsbereiche, Berechtigungen und Kategorien. Full SQL (pgvector) drückt das neben deinen existierenden Joins aus. Purpose-built DBs nutzen JSON-Filter-Objekte (Qdrant, Pinecone), eine Query-DSL (Elasticsearch, Milvus), oder GraphQL (Weaviate). Sie funktionieren; SQL wird attraktiver, je verwickelter die Filter-Logik wird.

```sql
-- pgvector: vector similarity is just another expression in WHERE
SELECT id, title, 1 - (embedding <=> $1) AS score
FROM documents
WHERE tenant_id = $2 AND category = ANY($3::text[]) AND created_at > NOW() - INTERVAL '90 days'
ORDER BY embedding <=> $1 LIMIT 10;
```

```python
# Qdrant: equivalent filter as a Python object — functional, more ceremony
results = client.query_points(
    collection_name="documents", query=query_embedding,
    query_filter=models.Filter(must=[
        models.FieldCondition(key="tenant_id", match=models.MatchValue(value=tenant_id)),
        models.FieldCondition(key="category",  match=models.MatchAny(any=categories)),
        models.FieldCondition(key="created_at", range=models.DatetimeRange(gte=cutoff)),
    ]),
    limit=10,
)
```

**Multimodal native** bedeutet nicht "kann Bild-Embeddings speichern"; jede Datenbank speichert Float-Arrays. Es bedeutet, die Datenbank bringt Embedding-Modelle für Nicht-Text-Inhalte mit, also übergibst du eine rohe Bild-URL und sie handhabt Vektorisierung. Die meisten Datenbanken hier sind embedding-agnostisch, also gehörst du diese Pipeline. Marqo und Weaviate (via CLIP/ImageBind-Module) schließen den Loop.

```python
# Marqo: POST raw images, query with text — no external embedding step needed
mq.index("products").add_documents(
    [{"id": "shoe-001", "image": "https://cdn.example.com/shoes/001.jpg"}],
    tensor_fields=["image"]
)
results = mq.index("products").search(q="lightweight shoes for summer")
# Returns shoe-001 despite zero keyword overlap — CLIP handles the cross-modal match
```

**Disk-based index** ist eine Kostenfrage. RAM-resident HNSW kann mehrere GB RAM pro Million 1536-Dim-Vektoren brauchen, wenn man rohe Vektoren, Graph-Overhead und Metadaten zählt. Disk-native Alternativen (Milvus DiskANN, Elasticsearch DiskBBQ, LanceDBs Lance-Format, Turbopuffers Object-Storage-Tier) tauschen oft etwas Query-Latenz für günstigere Infrastruktur. Für RAG-Workloads, wo Modell-Latenz bereits dominiert, ist das oft wert, zu benchmarken. Redis VSS ist die harte Einschränkung: RAM-only, kein Disk-Path.

**Max dimensions** ist morgen Migration, die sich in heute Choice versteckt. `text-embedding-3-large` nutzt 3072 Dims, Jina v3 kann größere Embeddings ausgeben, und Forschungs-Models schieben weiterhin höher. Einige managed Services veröffentlichen harte Dimensions-Caps; andere dokumentieren hohe Caps oder keinen praktischen Cap für typische Embedding-Models. Prüfe aktuelle Docs, bevor du dich festlegst. Wähl etwas mit Spielraum; einen Vektor-Index migrieren, weil du eine Dimensions-Deckengrenze getroffen hast, ist ein elender Sprint.

### Die Matrix lesen

Einige Dinge passen nicht sauber in eine Tabelle:

**Sparse Vektoren** sind, wie du BM25-Qualität Keyword-Matching innerhalb eines Vektor-Index bekommst, ohne separate Volltext-Engine. Qdrant und Elasticsearch haben hier besonders reife Implementierungen. Weaviate supportet sie via BM25F. Wenn Hybridsuche kritisch ist und du nicht zwei Systeme betreiben kannst, schau nach Sparse-Vektor-Support.

**Disk-basierte Indizes** sind ein Kosten-Hebel, keine Implementierungsdetail. RAM-resident HNSW-Indizes sind schnell, aber können teuer werden, wenn Vektoranzahl, Dimensionsanzahl, Metadaten und Graph-Overhead wachsen. Disk-basierte Alternativen (Milvus DiskANN, Elasticsearch DiskBBQ, Turbopuffer Object Storage, LanceDBs Lance-Format) tauschen Query-Latenz gegen niedrigere Infrastrukturkosten. Für große RAG-Indizes ist dieser Tradeoff oft testen wert.

**Turbopuffers Multi-Tenancy** ist um sehr hohe Namespace-Anzahlen gebaut. Sein öffentliches Positioning und Customer Stories betonen Workloads wie Notions große, namespace-reiche Corpus. Wenn jeder User oder jede Organisation isolierte Vektorsuche braucht, kann diese Architektur die Ökonomie ändern, aber benchmarke trotzdem deine eigene Tenant-Form.

**LanceDB Embedded Mode** ist das Ähnlichste zu "SQLite für Vektorsuche." Es läuft in-process, braucht keinen Server und funktioniert in Lambda, Cloudflare Workers und Edge-Umgebungen. Die Lance columnare Format macht Embedded-Operation bei realer Skala praktisch.

**Orama ist Search-UX-Infrastruktur, kein Warehouse.** Es ist großartig, wenn du eine tiny Full-Text/Vektor/Hybrid-Suchmaschine in einer JavaScript-App, am Edge, oder als managed Site/App-Such-Layer willst. Es ist nicht das Tool, das ich für Milliarden-Vektor-Retrieval, heavy Analytics oder komplex gefilterte Joins wählen würde.

**Vespa ist das, was du greifst, wenn Retrieval nur die Hälfte des Produkts ist.** Es kombiniert lexikalische Retrieval, Nearest-Neighbor-Suche, Tensors, Ranking-Expressions, Grouping und Online-Serving. Diese Power ist real, aber genauso die operationelle und Modellierungs-Komplexität. Es passt Search/Recommendation-Teams mehr als "semantische Suche zu meiner CRUD-App hinzufügen."

**ClickHouse gehört in die Konversation, wenn Suche an Analytics angehängt ist.** Wenn deine Source of Truth Events, Logs, Traces, Metriken oder große Faktentabellen sind, kann ClickHouse Vektor-Distanz, Filtern, Aggregation und jetzt ernsthafte Volltext-Indizierung in einer SQL-Engine halten. Es ist keine purpose-built Vektordatenbank, aber für analytische Retrieval kann es die langweilige Antwort auf die beste Art sein.

**Chroma ist am stärksten bei Dev/Test und kleinen App-Deployments.** Wenn du auf sehr große Corpora, HA, disk-schwere Operation oder first-class Hybridsuche zielst, evaluiere einen produktionsorientierten Speicher, bevor du den Prototype in Infrastruktur promotest.

### Die vereinfachte Entscheidung

Wenn du wirklich über pgvector hinausgewachsen bist - meistens weil Benchmarks Vektoranzahl, Filtern, Write-Rate oder High-Concurrency-Latenz zeigen, die dein Postgres-Envelope übersteigen - wähl nach Constraint:

- **SaaS-Produkt mit pro-Tenant-Isolation** → Turbopuffer
- **Brauchst Rust-Level-Performance + komplexe Metadata-Filterung** → Qdrant
- **Bereits auf Elastic/ELK-Stack** → Elasticsearch mit DiskBBQ
- **AWS-Shop, der Open-Source will** → OpenSearch
- **Search/Recommendation-Plattform mit ernsthaftem Ranking-Bedarf** → Vespa
- **Analytics, Observability oder Log/Event-Suche** → ClickHouse
- **Billion-Scale on-prem / self-hosted** → Milvus
- **Edge / serverless / multimodal** → LanceDB
- **Kleine JS-App, Doc-Site oder Edge-native Search-UX** → Orama
- **Zero ops, einfach, Kosten sekundär** → Pinecone
- **Multimodal-first (Bilder, Video, Audio)** → Marqo
- **Bereits auf MongoDB** → Atlas Vector Search
- **Bereits auf Postgres, brauchst mehr Spielraum** → Supabase Vector oder Neon (beide pgvector managed, mit besserem Tooling)

## Eine Sache, die du NICHT tun solltest

Nutze Vektorsuche nicht als Fuzzy-Text-Suche für Dinge, die richtige Antworten haben.

"Finde mir den User mit Email `dan@example.com`" ist kein Vektorsuche-Problem. Genauso wenig "Finde mir die Order mit ID `ORD-12345`." `ORD-12345` embedden und Cosine-Similarity gegen deine Orders-Tabelle machen wird *etwas* zurückgeben, aber es könnte falsch sein. Das sind Exact-Match-Probleme. Nutze deinen Primary Key oder einen regulären Index.

Vektorsuche gibt das *ähnlichste* Ding in deinem Dataset zurück, selbst wenn nichts relevant ist. Sie weiß nicht, dass es keine gute Antwort gibt. Das ist in Ordnung für verwandte Dokumente. Es ist katastrophal für spezifischen Record-Lookup, wo ein falscher Near-Match schlechter ist als ein leeres Ergebnis.

Wisse, wofür jedes Werkzeug da ist. Die meisten davon sind bereits in deiner Postgres-Installation. Nutze sie dort, wo sie passen.
