Sapevi una cosa strana sui modelli linguistici? Sanno spiegare la meccanica quantistica, scrivere poesia, e fare debug del tuo TypeScript... ma chiedi loro di moltiplicare 18472 per 9347 e c'è una discreta possibilità che ti diano, con totale sicurezza, qualcosa che sbaglia di migliaia.

Questo mi lasciava perplesso finché non ho capito cosa stiamo realmente chiedendo loro di fare. Stiamo chiedendo a un motore di pattern-matching di essere una calcolatrice. È come chiedere a un ginnasta di fare il bilancio familiare perché capisce il concetto di "bilancio".

Il fatto è che gli LLM non calcolano niente. Quando chiedi a GPT o Claude quanto fa 2 + 2, non stanno sommando. Stanno predendo che "4" è il token più probabile che appaia dopo "2 + 2 =". La maggior parte delle volte funziona benissimo perché questi pattern esistono nei loro dati di training. Ma spingiti oltre la semplice aritmetica verso calcoli multi-step o qualsiasi cosa con numeri che non erano comuni nel training, e stai essenzialmente lanciando dadi.

Mi sono scontrato con questo problema di recente mentre revisionavo del codice che usava un modello top-of-the-line per calcolare rate di mutuo. Il modello ha risposto con totale sicurezza. Sbagliava anche di $400/mese. Questo è il tipo di errore che conta.

Anche mentre i modelli migliorano nel ragionamento (GPT-5 apparentemente mostra miglioramenti), stanno ancora facendo sophisticated pattern matching, non calcolo simbolico. Per il lavoro creativo e i task in linguaggio naturale, questa natura probabilistica è esattamente ciò che li rende magici. Per la matematica? Non proprio.

## Cosa Risolve Davvero Questo?

La risposta non sta nell'aspettare modelli più intelligenti. Sta nel dare al modello lo strumento giusto per il lavoro.

Pensa a come risolveresti questo problema se stessi costruendo un sistema non-AI. Non scriveresti logica matematica custom, useresti una libreria. Lo stesso principio si applica qui, tranne che ora stiamo insegnando all'LLM quando e come usare quella libreria.

Il tool calling negli AI SDK moderni ci permette di dare al modello funzioni strutturate che può invocare. Invece di costringere l'LLM a fingere di sapere la matematica, gli diamo qualcosa che effettivamente la sa fare: un motore matematico simbolico.

Ho usato [AI SDK v5 e v6](https://ai-sdk.vercel.ai/) per questo, insieme a CortexJS Compute Engine. L'SDK gestisce l'orchestrazione e il routing dei tool, mentre CortexJS gestisce qualsiasi cosa, dall'aritmetica base fino al calcolo. È una separazione delle responsabilità sorprendentemente pulita.

```bash
bun add ai @ai-sdk/anthropic @cortex-js/compute-engine zod
```

## Costruire il Math Tool

L'implementazione è più diretta di quanto potresti aspettarti. Quello che stiamo costruendo è un ponte tra la comprensione del linguaggio naturale dell'LLM e il calcolo matematico effettivo.

```typescript
import { generateText, stepCountIs, tool } from 'ai';
import { ComputeEngine } from '@cortex-js/compute-engine';
import { z } from 'zod';

// Initialize the engine once
const ce = new ComputeEngine();

const mathTool = tool({
  description: 'Evaluate mathematical expressions and solve equations with guaranteed accuracy. MUST be used for all mathematical operations to verify correctness - do not attempt mental math. Supports arithmetic, algebra, calculus, and complex operations. Can process multiple expressions at once.',
  parameters: z.object({
    expressions: z.array(z.string()).describe(
      'Array of mathematical expressions in LaTeX or plain notation, e.g. ["2 + 2", "\\frac{x^2 + 1}{x - 1}", "\\int x^2 dx"]'
    ),
  }),
  execute: async ({ expressions }) => {
    // Process all expressions in parallel (or detailed batch)
    return expressions.map(expression => {
      try {
        const result = ce.parse(expression).evaluate();
        return {
          expression,
          result: result.toString(),
          latex: result.latex,
        };
      } catch (error) {
        return { 
          expression,
          error: (error as Error).message 
        };
      }
    });
  },
});
```

Alcune cose che vale la pena notare:

La description fa un lavoro pesante. Quel linguaggio "MUST be used" potrebbe sembrare aggressivo, ma nella mia esperienza, essere espliciti con il modello su quando usare un tool è la differenza tra farlo funzionare a volte e farlo funzionare sempre. Consideratela prompt engineering a livello di tool.

Il batch processing tramite array `expressions` conta più di quanto potresti pensare. Ogni chiamata al modello ha latenza. Se stai risolvendo un sistema di equazioni o facendo matematica multi-step, processarli individualmente crea una pessima esperienza utente. Il batching significa un solo round-trip per risolvere dieci problemi.

Usare un motore simbolico invece di solo `eval()` (per favore non usare `eval()`) ci dà una vera comprensione matematica. Il motore parsifica l'intento, gestisce la formattazione LaTeX, e può lavorare con derivate e integrali. Non stiamo solo facendo calcoli, stiamo facendo matematica.

L'error handling è scoped per espressione. Se un calcolo fallisce, restituiamo quell'errore ma continuiamo con il resto. Questo permette al modello di vedere cosa ha funzionato e cosa no, potenzialmente auto-correggendosi nel prossimo step.

## Metterlo al Lavoro

Proviamo qualcosa che tipicamente farebbe hallucinare un modello raw:

```typescript
import { anthropic } from '@ai-sdk/anthropic';

const { text } = await generateText({
  model: anthropic('claude-sonnet-4-5'),
  prompt: 'Calculate 18472 × 9347, divide by 127, then take the square root of the result.',
  tools: { mathTool },
  stopWhen: stepCountIs(5), // Allow up to five model/tool steps
});

console.log(text);
```

Il modello vede la matematica, riconosce che serve precisione, chiama il tool, ottiene il risultato accurato, e poi lo spiega in linguaggio naturale. Ogni componente che fa ciò che sa fare meglio.

## Oltre l'Aritmetica Base

Dal momento che stiamo usando un motore simbolico, questo approccio gestisce cose che semplici tool calcolatrici non possono affrontare.

Vuoi risolvere equazioni algebriche? "Risolvi queste equazioni: 3x + 7 = 22 e 2y - 5 = 13" funziona bene.

Serve il calcolo? "Trova la derivata di x^3 + 2x^2 e valutala in x = 2" è solo un altro tool call.

Il supporto LaTeX è particolarmente utile se stai costruendo app educative. Il motore capisce intrinsecamente l'input LaTeX e può restituire risultati formattati per il rendering. Nessun parsing aggiuntivo necessario.

## Il Quadro Più Ampio

Penso che questo pattern conti oltre la semplice matematica. Quello che stiamo realmente facendo è riconoscere le limitazioni degli LLM mentre sfruttiamo i loro punti di forza. Sono incredibili nel capire l'intento, parsare il linguaggio naturale, e orchestrare workflow. Non sono calcolatori, database, o file system.

Ogni volta che proviamo a far fare a un LLM qualcosa di deterministico, stiamo combattendo contro la sua natura. Ma quando accoppiamo quella comprensione del linguaggio naturale con tool specializzati che gestiscono le parti deterministiche? È lì che le cose si fanno interessanti.

Il math tool è solo un esempio. Lo stesso principio si applica alla manipolazione di date, calcoli finanziari, processing di immagini, query a database... ovunque la precisione conta più della creatività. Lascia che il modello capisca cosa vuole l'utente, poi passa il lavoro effettivo a qualcosa costruito per quello.

È un cambiamento nel modo di pensare il building con AI. Non "il modello può fare questo?" ma "il modello può orchestrare questo?" Piccola differenza nella formulazione, significativa differenza nell'affidabilità.
