C'è qualcosa di strano nei modelli linguistici? Sanno spiegare la meccanica quantistica, scrivere poesie e fare il debug del tuo TypeScript... ma chiedi loro di moltiplicare 18472 per 9347 e c'è una buona probabilità che ti diano con sicurezza un risultato sbagliato di migliaia.

Questa cosa mi sconcertava, finché non ho capito cosa stiamo davvero chiedendo loro di fare. Stiamo chiedendo a un motore di pattern-matching di fare da calcolatrice. È come chiedere a una ginnasta di tenere la contabilità perché capisce il concetto di "bilancio".

Il fatto è che gli LLM non calcolano nulla. Quando chiedi a GPT o Claude quanto fa 2 + 2, non stanno sommando. Stanno prevedendo che "4" è il token più probabile dopo "2 + 2 =". La maggior parte delle volte funziona perché questi pattern esistono nei dati di addestramento. Ma se vai oltre l'aritmetica semplice in calcoli multi-step o con numeri poco comuni nei dati di training, stai essenzialmente tirando i dadi.

Recentemente mi sono imbattuto in questo problema mentre revisionavo del codice che usava un modello di prima categoria per calcolare rate di mutui. Il modello rispondeva con assoluta sicurezza. Ed era sbagliato di 400$ al mese. È il tipo di errore che conta.

Anche se i modelli migliorano nel ragionamento (GPT-5 mostra supposti miglioramenti), stanno comunque facendo pattern-matching sofisticato, non calcolo simbolico. Per lavori creativi e compiti di linguaggio naturale, questa natura probabilistica è proprio ciò che li rende magici. Per la matematica? Non proprio.

## Cosa Risolve Davvero il Problema?

La risposta non è aspettare modelli più intelligenti. È dare al modello lo strumento giusto per il lavoro.

Pensa a come risolveresti questo problema se stessi costruendo un sistema non basato su AI. Non scriveresti logiche matematiche personalizzate, useresti una libreria. Lo stesso principio vale qui, solo che ora insegniamo all'LLM quando e come usare quella libreria.

Il tool calling negli AI SDK moderni ci permette di dare al modello funzioni strutturate che può invocare. Invece di forzare l'LLM a fingersi competente in matematica, gli diamo qualcosa che lo è davvero: un motore di calcolo simbolico.

Ho usato [AI SDK v5 e v6](https://ai-sdk.vercel.ai/) per questo, insieme a CortexJS Compute Engine. L'SDK gestisce orchestrazione e routing dei tool, mentre CortexJS gestisce qualsiasi cosa dall'aritmetica di base fino al calcolo infinitesimale. È una separazione di responsabilità sorprendentemente pulita.

```bash
bun add ai @ai-sdk/anthropic @cortex-js/compute-engine zod
```

## Costruire il Tool Matematico

L'implementazione è più semplice di quanto ti aspetteresti. Quello che stiamo costruendo è un ponte tra la comprensione del linguaggio naturale dell'LLM e il calcolo matematico reale.

```typescript
import { generateText, stepCountIs, tool } from 'ai';
import { ComputeEngine } from '@cortex-js/compute-engine';
import { z } from 'zod';

// Initialize the engine once
const ce = new ComputeEngine();

const mathTool = tool({
  description: 'Evaluate mathematical expressions and solve equations with guaranteed accuracy. MUST be used for all mathematical operations to verify correctness - do not attempt mental math. Supports arithmetic, algebra, calculus, and complex operations. Can process multiple expressions at once.',
  parameters: z.object({
    expressions: z.array(z.string()).describe(
      'Array of mathematical expressions in LaTeX or plain notation, e.g. ["2 + 2", "\\frac{x^2 + 1}{x - 1}", "\\int x^2 dx"]'
    ),
  }),
  execute: async ({ expressions }) => {
    // Process all expressions in parallel (or detailed batch)
    return expressions.map(expression => {
      try {
        const result = ce.parse(expression).evaluate();
        return {
          expression,
          result: result.toString(),
          latex: result.latex,
        };
      } catch (error) {
        return { 
          expression,
          error: (error as Error).message 
        };
      }
    });
  },
});
```

Vale la pena notare alcune cose:

La descrizione è fondamentale. Quel linguaggio "MUST be used" può sembrare aggressivo, ma per esperienza, essere espliciti con il modello su quando usare un tool è la differenza tra un funzionamento saltuario e uno affidabile. Consideralo come prompt engineering a livello di tool.

L'elaborazione in batch tramite un array `expressions` è più importante di quanto sembri. Ogni chiamata al modello ha una latenza. Se stai risolvendo un sistema di equazioni o facendo calcoli multi-step, elaborarli individualmente crea un'esperienza utente pessima. Il batch significa un solo round-trip per risolvere dieci problemi.

Usare un motore simbolico invece di un semplice `eval()` (per favore, non usare `eval()`) ci dà una vera comprensione matematica. Il motore analizza l'intento, gestisce la formattazione LaTeX e può lavorare con derivate e integrali. Non stiamo solo facendo calcoli, stiamo facendo matematica.

La gestione degli errori è per singola espressione. Se un calcolo fallisce, restituiamo quell'errore ma continuiamo con il resto. Questo permette al modello di vedere cosa ha funzionato e cosa no, potenzialmente autocorreggendosi al passo successivo.

## Metterlo in Pratica

Lanciamogli qualcosa che farebbe allucinare un modello grezzo:

```typescript
import { anthropic } from '@ai-sdk/anthropic';

const { text } = await generateText({
  model: anthropic('claude-sonnet-4-5'),
  prompt: 'Calculate 18472 × 9347, divide by 127, then take the square root of the result.',
  tools: { mathTool },
  stopWhen: stepCountIs(5), // Allow up to five model/tool steps
});

console.log(text);
```

Il modello vede la matematica, riconosce che serve precisione, chiama il tool, ottiene il risultato accurato e poi lo spiega in linguaggio naturale. Ogni componente fa ciò che sa fare meglio.

## Oltre l'Aritmetica di Base

Dato che usiamo un motore simbolico, questo approccio gestisce cose che i semplici tool calcolatrice non possono toccare.

Vuoi risolvere equazioni algebriche? "Solve these equations: 3x + 7 = 22 and 2y - 5 = 13" funziona perfettamente.

Serve il calcolo infinitesimale? "Find the derivative of x^3 + 2x^2 and evaluate it at x = 2" è solo un'altra chiamata al tool.

Il supporto LaTeX è particolarmente utile se stai costruendo app educative. Il motore comprende nativamente input LaTeX e può restituire risultati pronti per il rendering. Nessun parsing aggiuntivo necessario.

## Il Quadro Generale

Penso che questo pattern sia importante ben oltre la matematica. Quello che stiamo facendo è riconoscere i limiti degli LLM sfruttando al contempo i loro punti di forza. Sono incredibili nel comprendere l'intento, fare parsing del linguaggio naturale e orchestrare flussi di lavoro. Non sono calcolatrici o database o filesystem.

Ogni volta che cerchiamo di far fare a un LLM qualcosa di deterministico, stiamo combattendo la sua natura. Ma quando abbiniamo quella comprensione del linguaggio naturale a tool specializzati che gestiscono le parti deterministiche? È lì che le cose si fanno interessanti.

Il tool matematico è solo un esempio. Lo stesso principio si applica alla manipolazione delle date, ai calcoli finanziari, all'elaborazione delle immagini, alle query di database... ovunque la precisione è più importante della creatività. Lascia che il modello capisca cosa vuole l'utente, poi passa il lavoro vero a qualcosa costruito per quello scopo.

È un cambio di prospettiva su come pensiamo di costruire con l'AI. Non "il modello può farlo?" ma "il modello può orchestrarlo?" Piccola differenza di parole, differenza enorme in affidabilità.
