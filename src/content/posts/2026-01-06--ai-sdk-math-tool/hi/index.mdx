भाषा मॉडलों के बारे में जो अजीब बात है? वे क्वांटम मैकेनिक्स समझा सकते हैं, कविता लिख सकते हैं, और आपका TypeScript डीबग कर सकते हैं... लेकिन उनसे 18472 को 9347 से गुणा करने को कहो तो काफी संभावना है कि वे आत्मविश्वास से ऐसा जवाब दें जो हज़ारों तक गलत हो।

यह मुझे तब तक हैरान करता रहा जब तक मुझे एहसास नहीं हुआ कि हम वास्तव में उनसे क्या करवा रहे हैं। हम एक पैटर्न-मैचिंग इंजन से कैलकुलेटर बनने को कह रहे हैं। ये ऐसा है जैसे एक जिमनास्ट से आपका बैंक बैलेंस चेक करवाना क्योंकि उन्हें "बैलेंस" का कॉन्सेप्ट समझता है।

सच तो यह है कि LLMs कुछ भी कम्प्यूट नहीं करते। जब आप GPT या Claude से पूछते हैं कि 2 + 2 कितना होता है, तो वे जोड़ नहीं रहे हैं। वे यह प्रेडिक्ट कर रहे हैं कि "2 + 2 =" के बाद "4" सबसे ज़्यादा संभावित टोकन है। ज़्यादातर मामलों में यह काम करता है क्योंकि ये पैटर्न उनके ट्रेनिंग डेटा में मौजूद हैं। लेकिन साधारण अंकगणित से आगे बढ़कर मल्टी-स्टेप कैलकुलेशन या ऐसे नंबर जो ट्रेनिंग में कम देखे गए हों, तो आप असल में पासा फेंक रहे हैं।

मैं इससे हाल ही में तब टकराया जब मैंने कुछ कोड रिव्यू किया जो एक टॉप-टियर मॉडल का इस्तेमाल मॉर्टगेज पेमेंट कैलकुलेट करने के लिए कर रहा था। मॉडल ने पूरे आत्मविश्वास से जवाब दिया। लेकिन वह $400/महीना तक गलत भी था। ऐसी गलती जो मायने रखती है।

भले ही मॉडल्स रीज़निंग में बेहतर हो रहे हैं (कथित तौर पर GPT-5 में सुधार दिख रहे हैं), वे अभी भी सोफिस्टिकेटेड पैटर्न मैचिंग कर रहे हैं, सिम्बॉलिक कम्प्यूटेशन नहीं। रचनात्मक काम और प्राकृतिक भाषा के कामों के लिए, यही प्रोबेबिलिस्टिक प्रकृति उन्हें जादुई बनाती है। गणित के लिए? इतना नहीं।

## असल में क्या इसे हल करता है?

जवाब स्मार्ट मॉडल्स का इंतज़ार करना नहीं है। बल्कि मॉडल को सही टूल देना है।

सोचिए कि आप एक गैर-AI सिस्टम बना रहे होते तो यह प्रॉब्लम कैसे सुलझाते। आप कस्टम मैथ लॉजिक नहीं लिखते, आप एक लाइब्रेरी का इस्तेमाल करते। यही सिद्धांत यहाँ भी लागू होता है, सिवाय इसके कि अब हम LLM को सिखा रहे हैं कि वह कब और कैसे उस लाइब्रेरी का इस्तेमाल करे।

मॉडर्न AI SDK में टूल कॉलिंग हमें मॉडल को स्ट्रक्चर्ड फंक्शन्स देने की सुविधा देती है जो वह इनवोक कर सकता है। LLM को गणित जानने का दिखावा करने के बजाय, हम उसे कुछ ऐसा दे रहे हैं जो वास्तव में जानता है: एक सिम्बॉलिक मैथ इंजन।

मैं इसके लिए [AI SDK v5 और v6](https://ai-sdk.vercel.ai/) का इस्तेमाल कर रहा हूँ, CortexJS Compute Engine के साथ। SDK ऑर्केस्ट्रेशन और टूल रूटिंग संभालता है, जबकि CortexJS बेसिक अंकगणित से लेकर कैलकुलस तक सब कुछ हैंडल करता है। यह एक आश्चर्यजनक रूप से साफ कॉन्सर्न्स का विभाजन है।

```bash
bun add ai @ai-sdk/anthropic @cortex-js/compute-engine zod
```

## मैथ टूल बनाना

इम्प्लीमेंटेशन उतनी ही सरल है जितनी आप उम्मीद कर सकते हैं। हम जो बना रहे हैं वह LLM की प्राकृतिक भाषा समझ और वास्तविक गणितीय कम्प्यूटेशन के बीच एक पुल है।

```typescript
import { generateText, stepCountIs, tool } from 'ai';
import { ComputeEngine } from '@cortex-js/compute-engine';
import { z } from 'zod';

// Initialize the engine once
const ce = new ComputeEngine();

const mathTool = tool({
  description: 'Evaluate mathematical expressions and solve equations with guaranteed accuracy. MUST be used for all mathematical operations to verify correctness - do not attempt mental math. Supports arithmetic, algebra, calculus, and complex operations. Can process multiple expressions at once.',
  parameters: z.object({
    expressions: z.array(z.string()).describe(
      'Array of mathematical expressions in LaTeX or plain notation, e.g. ["2 + 2", "\\frac{x^2 + 1}{x - 1}", "\\int x^2 dx"]'
    ),
  }),
  execute: async ({ expressions }) => {
    // Process all expressions in parallel (or detailed batch)
    return expressions.map(expression => {
      try {
        const result = ce.parse(expression).evaluate();
        return {
          expression,
          result: result.toString(),
          latex: result.latex,
        };
      } catch (error) {
        return { 
          expression,
          error: (error as Error).message 
        };
      }
    });
  },
});
```

इसके बारे में कुछ बातें ध्यान देने लायक हैं:

डिस्क्रिप्शन भारी काम कर रही है। वह "MUST be used" वाली भाषा आक्रामक लग सकती है, लेकिन मेरे अनुभव में, मॉडल को टूल कब इस्तेमाल करना है यह स्पष्ट बताना ही अंतर है कि वह कभी-कभी काम करे या हमेशा काम करे। इसे टूल लेवल पर प्रॉम्प्ट इंजीनियरिंग मानिए।

`expressions` ऐरे के ज़रिए बैच प्रोसेसिंग आप जितना सोच सकते हैं उससे ज़्यादा मायने रखती है। हर मॉडल कॉल में लेटेंसी होती है। अगर आप इक्वेशन सिस्टम सॉल्व कर रहे हैं या मल्टी-स्टेप मैथ कर रहे हैं, तो उन्हें अलग-अलग प्रोसेस करना एक भयान यूज़र एक्सपीरियंस बनाता है। बैचिंग का मतलब है दस प्रॉब्लम सॉल्व करने में एक राउंड-ट्रिप।

सिर्फ `eval()` की जगह सिम्बॉलिक इंजन का इस्तेमाल (कृपया `eval()` मत इस्तेमाल करिए) हमें असली गणितीय समझ देता है। इंजन इंटेंट पार्स करता है, LaTeX फॉर्मेटिंग हैंडल करता है, और डेरिवेटिव और इंटीग्रल के साथ काम कर सकता है। हम सिर्फ कैलकुलेशन नहीं कर रहे, हम गणित कर रहे हैं।

एरर हैंडलिंग प्रति एक्सप्रेशन स्कोप्ड है। अगर एक कैलकुलेशन फेल होता है, तो हम उस एरर को रिटर्न करते हैं लेकिन बाकी के साथ जारी रखते हैं। इससे मॉडल देख सकता है कि क्या काम किया और क्या नहीं, और संभावित रूप से अगले स्टेप में सेल्फ-करेक्ट कर सकता है।

## इसे काम में लगाना

आइए उसे कुछ ऐसा दें जो आमतौर पर एक रॉ मॉडल को हैल्यूसिनेट करवा दे:

```typescript
import { anthropic } from '@ai-sdk/anthropic';

const { text } = await generateText({
  model: anthropic('claude-sonnet-4-5'),
  prompt: 'Calculate 18472 × 9347, divide by 127, then take the square root of the result.',
  tools: { mathTool },
  stopWhen: stepCountIs(5), // Allow up to five model/tool steps
});
```

मॉडल गणित देखता है, प्रिसिशन की ज़रूरत पहचानता है, टूल को कॉल करता है, सही रिज़ल्ट पाता है, और फिर उसे प्राकृतिक भाषा में समझाता है। हर कंपोनेंट वही कर रहा है जो वह सबसे अच्छा करता है।

## बेसिक अंकगणित से परे

चूंकि हम सिम्बॉलिक इंजन इस्तेमाल कर रहे हैं, यह तरीका ऐसी चीज़ें हैंडल कर सकता है जो साधारण कैलकुलेटर टूल छू भी नहीं सकते।

अलजेब्रिक इक्वेशन सॉल्व करनी हैं? "Solve these equations: 3x + 7 = 22 and 2y - 5 = 13" काम करता है।

कैलकुलस चाहिए? "Find the derivative of x^3 + 2x^2 and evaluate it at x = 2" बस एक और टूल कॉल है।

LaTeX सपोर्ट खास तौर पर उपयोगी है अगर आप एजुकेशनल ऐप्स बना रहे हैं। इंजन स्वाभाविक रूप से LaTeX इनपुट समझता है और रेंडरिंग के लिए फॉर्मेटेड रिज़ल्ट रिटर्न कर सकता है। कोई अतिरिक्त पार्सिंग ज़रूरी नहीं।

## बड़ी तस्वीर

मुझे लगता है कि यह पैटर्न सिर्फ गणित से परे भी मायने रखता है। हम वास्तव में LLMs की सीमाओं को स्वीकार कर रहे हैं जबकि उनकी ताकतों का लाभ उठा रहे हैं। वे इंटेंट समझने, प्राकृतिक भाषा पार्स करने, और वर्कफ़्लो ऑर्केस्ट्रेट करने में अविश्वसनीय हैं। वे कैलकुलेटर नहीं हैं, डेटाबेस नहीं हैं, फाइल सिस्टम नहीं हैं।

जब भी हम LLM से कुछ डिटरमिनिस्टिक करवाने की कोशिश करते हैं, हम उसकी प्रकृति के खिलाफ जा रहे हैं। लेकिन जब हम उस प्राकृतिक भाषा समझ को स्पेशलाइज़्ड टूल्स के साथ जोड़ते हैं जो डिटरमिनिस्टिक हिस्से संभालते हैं? तब चीज़ें दिलचस्प हो जाती हैं।

मैथ टूल सिर्फ एक उदाहरण है। यही सिद्धांत डेट मैनिपुलेशन, फाइनेंशियल कैलकुलेशन, इमेज प्रोसेसिंग, डेटाबेस क्वेरी... हर जगह लागू होता है जहाँ प्रिसिशन क्रिएटिविटी से ज़्यादा मायने रखती है। मॉडल को यह समझने दें कि यूज़र क्या चाहता है, फिर असली काम कुछ ऐसे को सौंपें जो उस काम के लिए बना हो।

यह AI के साथ बिल्डिंग के बारे में सोचने का एक बदलाव है। "क्या मॉडल यह कर सकता है?" नहीं, बल्कि "क्या मॉडल इसे ऑर्केस्ट्रेट कर सकता है?" शब्दों में छोटा फर्क, विश्वसनीयता में बड़ा फर्क।
