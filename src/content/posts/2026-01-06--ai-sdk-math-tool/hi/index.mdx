आप जानते हैं language models के बारे में क्या अजीब है? वे quantum mechanics समझा सकते हैं, poetry लिख सकते हैं, और आपका TypeScript debug कर सकते हैं... लेकिन अगर आप उनसे 18472 को 9347 से गुणा करने को कहें, तो एक अच्छा मौका है कि वे आत्मविश्वास से कुछ ऐसा देंगे जो हज़ारों से गलत हो।

यह मुझे तब तक हैरान करता रहा जब तक मुझे नहीं पता चला कि हम वास्तव में उनसे क्या करने को कह रहे हैं। हम एक pattern-matching engine से calculator बनाने को कह रहे हैं। यह वैसे ही है जैसे किसी gymnast से अपनी checkbook balance करने को कहना क्योंकि वह "balance" की अवधारणा समझता है।

बात यह है, LLMs कुछ भी compute नहीं करते। जब आप GPT या Claude से पूछते हैं कि 2 + 2 कितने होते हैं, वे जोड़ नहीं रहे होते। वे यह अनुमान लगा रहे होते हैं कि "2 + 2 =" के बाद सबसे अधिक संभावना "4" token आने वाला है। अधिकांश समय, यह अच्छा काम करता है क्योंकि ये patterns उनके training data में मौजूद होते हैं। लेकिन जब आप multi-step calculations या उन numbers के साथ कुछ करने को कहते हैं जो training में common नहीं थे, तो आप मूल रूप से dice roll कर रहे होते हैं।

मैंने यह हाल ही में तब देखा जब मैं कुछ code review कर रहा था जिसने mortgage payments calculate करने के लिए एक top-tier model का उपयोग किया। model ने पूरे आत्मविश्वास के साथ जवाब दिया। यह भी गलत था $400/month। यह वो तरह की error है जो मायने रखती है।

भले ही models reasoning में बेहतर होते जा रहे हैं (GPT-5 में कहा जाता है सुधार है), वे अभी भी symbolic computation नहीं कर रहे हैं, वे sophisticated pattern matching कर रहे हैं। creative work और natural language tasks के लिए, यह probabilistic nature वही है जो उन्हें magical बनाता है। गणित के लिए? उतना नहीं।

## What Actually Solves This?

जवाब smarter models का इंतज़ार करना नहीं है। यह model को job के लिए सही tool देना है।

सोचिए कि आप इस problem को कैसे solve करेंगे अगर आप कोई non-AI system बना रहे हों। आप custom math logic नहीं लिखेंगे, आप एक library पकड़ेंगे। यहां भी same principle apply होती है, सिवाय इसके कि अब हम LLM को सिखा रहे हैं कि कब और कैसे उस library का उपयोग करना है।

Modern AI SDKs में tool calling हमें structured functions देती है जिन्हें model invoke कर सकता है। LLM को यह जबरदस्ती नहीं डालना कि वह math जानता है, बल्कि उसे कुछ ऐसा देना जो वास्तव में करता है: एक symbolic math engine.

मैं इसके लिए AI SDK v5 और v6 का उपयोग कर रहा हूं, CortexJS Compute Engine के साथ जोड़ी बनाकर। SDK orchestration और tool routing संभालता है, जबकि CortexJS basic arithmetic से लेकर calculus तक anything handle करता है। यह surprising रूप से clean separation of concerns है।

```bash
bun add ai @ai-sdk/anthropic @cortex-js/compute-engine zod
```

## Building the Math Tool

Implementation आपके सोचने से ज्यादा straightforward है। जो हम बना रहे हैं वह LLM की natural language understanding और actual mathematical computation के बीच एक bridge है।

```typescript
import { generateText, stepCountIs, tool } from 'ai';
import { ComputeEngine } from '@cortex-js/compute-engine';
import { z } from 'zod';

// Initialize the engine once
const ce = new ComputeEngine();

const mathTool = tool({
  description: 'Evaluate mathematical expressions and solve equations with guaranteed accuracy. MUST be used for all mathematical operations to verify correctness - do not attempt mental math. Supports arithmetic, algebra, calculus, and complex operations. Can process multiple expressions at once.',
  parameters: z.object({
    expressions: z.array(z.string()).describe(
      'Array of mathematical expressions in LaTeX or plain notation, e.g. ["2 + 2", "\\frac{x^2 + 1}{x - 1}", "\\int x^2 dx"]'
    ),
  }),
  execute: async ({ expressions }) => {
    // Process all expressions in parallel (or detailed batch)
    return expressions.map(expression => {
      try {
        const result = ce.parse(expression).evaluate();
        return {
          expression,
          result: result.toString(),
          latex: result.latex,
        };
      } catch (error) {
        return { 
          expression,
          error: (error as Error).message 
        };
      }
    });
  },
});
```

इसके बारे में कुछ बातें ध्यान देने योग्य हैं:

Description बहुत काम कर रहा है। वह "MUST be used" language aggressive लग सकती है, लेकिन मेरे अनुभव में, model को बताना कि कब tool use करना है - यह कभी-कभी काम करने और reliably काम करने के बीच का अंतर है। इसे tool level पर prompt engineering मानें।

`expressions` array के माध्यम से batch processing इससे ज्यादा महत्वपूर्ण है जितना आप सोच सकते हैं। प्रत्येक model call का latency होता है। अगर आप system of equations solve कर रहे हैं या multi-step math कर रहे हैं, तो उन्हें individually process करना user experience को बुरा बनाता है। Batching का मतलब है दस problems solve करने के लिए एक round-trip।

Symbolic engine का उपयोग करके `eval()` के बजाय (कृपया `eval()` का उपयोग न करें) हमें real mathematical understanding मिलती है। Engine intent को parse करता है, LaTeX formatting handle करता है, और derivatives और integrals के साथ काम कर सकता है। हम केवल calculations नहीं कर रहे, हम mathematics कर रहे हैं।

Error handling per expression scoped है। अगर एक calculation fail होती है, तो हम उस error को return करते हैं लेकिन बाकी के साथ जारी रखते हैं। यह model को यह देखने देता है कि क्या काम किया और क्या नहीं, संभवतः अगले step में self-correcting।

## Putting It to Work

चलिए इस पर कुछ ऐसा扔 करते हैं जो typically एक raw model को hallucinate करता:

```typescript
import { anthropic } from '@ai-sdk/anthropic';

const { text } = await generateText({
  model: anthropic('claude-sonnet-4-5'),
  prompt: 'Calculate 18472 × 9347, divide by 127, then take the square root of the result.',
  tools: { mathTool },
  stopWhen: stepCountIs(5), // Allow up to five model/tool steps
});

console.log(text);
```

Model math देखता है, पहचानता है कि precision चाहिए, tool को call करता है, सही result देता है, और फिर natural language में explain करता है। प्रत्येक component अपना सबसे अच्छा कर रहा है।

## Beyond Basic Arithmetic

चूंकि हम symbolic engine का उपयोग कर रहे हैं, यह approach उन चीजों को handle करती है जिन्हें simple calculator tools छू नहीं सकते।

Algebraic equations solve करने हैं? "Solve these equations: 3x + 7 = 22 and 2y - 5 = 13" ठीक काम करता है।

Calculus चाहिए? "Find the derivative of x^3 + 2x^2 and evaluate it at x = 2" सिर्फ एक और tool call है।

LaTeX support especially useful है अगर आप educational apps बना रहे हैं। Engine inherently LaTeX input समझता है और results को rendering के लिए formatted return कर सकता है। कोई additional parsing required नहीं।

## The Bigger Picture

मुझे लगता है कि यह pattern सिर्फ math से आगे मायने रखता है। जो हम वास्तव में कर रहे हैं वह LLMs की limitations को acknowledge करना है जबकि उनकी strengths का leverage करना। वे intent समझने, natural language parse करने, और workflows orchestrate करने में अविश्वसनीय हैं। वे calculators या databases या file systems नहीं हैं।

हर बार जब हम एक LLM को कुछ deterministic करने को मजबूर करते हैं, हम उसका nature fight कर रहे होते हैं। लेकिन जब हम उस natural language understanding को specialized tools के साथ जोड़ते हैं जो deterministic parts handle करते हैं? तब चीजें interesting होती हैं।

Math tool सिर्फ एक example है। Same principle date manipulation, financial calculations, image processing, database queries... कहीं भी precision creativity से ज्यादा मायने रखती है, वहां apply होती है। Model को यह समझने दें कि user क्या चाहता है, फिर actual work कुछ ऐसी चीज को सौंपें जो उस job के लिए बनी है।

यह AI के साथ build करने के तरीके में एक shift है। नहीं "can the model do this?" बल्कि "can the model orchestrate this?" शब्दों में छोटा अंतर, reliability में significant अंतर।
