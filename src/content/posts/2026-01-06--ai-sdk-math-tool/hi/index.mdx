title: "LLMs से गणित पूछना बंद करें"
subTitle: "वे इसमें खराब हैं। यहाँ बताया गया है कि इसे कैसे ठीक किया जाए।"
---

भाषा मॉडल (language models) के बारे में क्या अजीब है जानते हैं? वे क्वांटम मैकेनिक्स (quantum mechanics) समझा सकते हैं, कविता लिख सकते हैं, और आपके TypeScript को डीबग कर सकते हैं... लेकिन उनसे 18472 को 9347 से गुणा करने के लिए कहें और इस बात की पूरी संभावना है कि वे आत्मविश्वास के साथ आपको कुछ ऐसा उत्तर देंगे जो हजारों से गलत होगा।

यह मुझे तब तक हैरान करता था जब तक मुझे समझ नहीं आया कि हम वास्तव में उन्हें क्या करने के लिए कह रहे हैं। हम एक पैटर्न-मैचिंग इंजन (pattern-matching engine) से कैलकुलेटर बनने के लिए कह रहे हैं। यह एक जिमनास्ट से अपनी चेकबुक को संतुलित करने के लिए कहने जैसा है क्योंकि वे "संतुलन" (balance) की अवधारणा को समझते हैं।

बात यह है कि, LLMs कुछ भी कंप्यूट नहीं करते हैं। जब आप GPT या Claude से पूछते हैं कि 2 + 2 कितना होता है, तो वे जोड़ नहीं रहे होते हैं। वे भविष्यवाणी कर रहे होते हैं कि "2 + 2 =" के बाद "4" टोकन आने की सबसे अधिक संभावना है। ज्यादातर समय, यह बहुत अच्छा काम करता है क्योंकि ये पैटर्न उनके ट्रेनिंग डेटा में मौजूद होते हैं। लेकिन साधारण अंकगणित (arithmetic) से आगे बढ़कर मल्टी-स्टेप कैलकुलेशन या ऐसी संख्याओं पर जाएं जो ट्रेनिंग में सामान्य नहीं थीं, तो आप मूल रूप से पासा फेंक रहे हैं।

हाल ही में मुझे इसका सामना तब करना पड़ा जब मैं कुछ ऐसे कोड की समीक्षा कर रहा था जो बंधक भुगतान (mortgage payments) की गणना करने के लिए एक टॉप-टियर मॉडल का उपयोग कर रहा था। मॉडल ने पूरे आत्मविश्वास के साथ जवाब दिया। लेकिन वह $400/महीने से गलत था। यह उस तरह की गलती है जो मायने रखती है।

भले ही मॉडल रीजनिंग (reasoning) में बेहतर हो रहे हैं (माना जाता है कि GPT-5 में सुधार दिखते हैं), फिर भी वे परिष्कृत पैटर्न मैचिंग कर रहे हैं, न कि सिम्बॉलिक कंप्यूटेशन (symbolic computation)। रचनात्मक कार्य और प्राकृतिक भाषा कार्यों के लिए, यह संभाव्य प्रकृति (probabilistic nature) ही उन्हें जादुई बनाती है। गणित के लिए? बिल्कुल नहीं।

## वास्तव में इसका समाधान क्या है?

इसका जवाब स्मार्ट मॉडल का इंतज़ार करना नहीं है। यह मॉडल को काम के लिए सही उपकरण (tool) देना है।

सोचें कि यदि आप एक गैर-एआई (non-AI) सिस्टम बना रहे होते तो आप इस समस्या को कैसे हल करते। आप कस्टम गणित लॉजिक नहीं लिखते, आप एक लाइब्रेरी का उपयोग करते। वही सिद्धांत यहाँ भी लागू होता है, बस अब हम LLM को यह सिखा रहे हैं कि उस लाइब्रेरी का उपयोग कब और कैसे करना है।

आधुनिक AI SDKs में टूल कॉलिंग हमें मॉडल को संरचित फ़ंक्शन (structured functions) सौंपने की अनुमति देती है जिन्हें वह कॉल कर सकता है। LLM को यह दिखावा करने के लिए मजबूर करने के बजाय कि उसे गणित आता है, हम उसे कुछ ऐसा देते हैं जो वास्तव में गणित करता है: एक सिम्बॉलिक मैथ इंजन (symbolic math engine)।

मैं इसके लिए [AI SDK v5 और v6](https://ai-sdk.vercel.ai/) का उपयोग कर रहा हूँ, जिसे CortexJS Compute Engine के साथ जोड़ा गया है। SDK ऑर्केस्ट्रेशन (orchestration) और टूल रूटिंग को संभालता है, जबकि CortexJS बेसिक अंकगणित से लेकर कैलकुलस (calculus) तक सब कुछ संभालता है। यह वास्तव में चिंताओं का एक स्पष्ट अलगाव (separation of concerns) है।

```bash
bun add ai @ai-sdk/anthropic @cortex-js/compute-engine zod
```

## मैथ टूल का निर्माण

कार्यान्वयन (implementation) आपकी अपेक्षा से अधिक सरल है। हम जो बना रहे हैं वह LLM की प्राकृतिक भाषा समझ और वास्तविक गणितीय गणना के बीच एक पुल है।

```typescript
import { generateText, stepCountIs, tool } from 'ai';
import { ComputeEngine } from '@cortex-js/compute-engine';
import { z } from 'zod';

// Initialize the engine once
const ce = new ComputeEngine();

const mathTool = tool({
  description: 'Evaluate mathematical expressions and solve equations with guaranteed accuracy. MUST be used for all mathematical operations to verify correctness - do not attempt mental math. Supports arithmetic, algebra, calculus, and complex operations. Can process multiple expressions at once.',
  parameters: z.object({
    expressions: z.array(z.string()).describe(
      'Array of mathematical expressions in LaTeX or plain notation, e.g. ["2 + 2", "\\frac{x^2 + 1}{x - 1}", "\\int x^2 dx"]'
    ),
  }),
  execute: async ({ expressions }) => {
    // Process all expressions in parallel (or detailed batch)
    return expressions.map(expression => {
      try {
        const result = ce.parse(expression).evaluate();
        return {
          expression,
          result: result.toString(),
          latex: result.latex,
        };
      } catch (error) {
        return { 
          expression,
          error: (error as Error).message 
        };
      }
    });
  },
});
```

यहाँ कुछ बातें ध्यान देने योग्य हैं:

विवरण (description) बहुत महत्वपूर्ण भूमिका निभा रहा है। वह "MUST be used" भाषा आक्रामक लग सकती है, लेकिन मेरे अनुभव में, मॉडल के साथ स्पष्ट होना कि टूल का उपयोग कब करना है, इसके कभी-कभी काम करने और इसके विश्वसनीय रूप से काम करने के बीच का अंतर है। इसे टूल स्तर पर प्रॉम्प्ट इंजीनियरिंग (prompt engineering) मानें।

`expressions` ऐरे के माध्यम से बैच प्रोसेसिंग आपकी सोच से कहीं अधिक मायने रखती है। प्रत्येक मॉडल कॉल में लेटेंसी (latency) होती है। यदि आप समीकरणों के सिस्टम को हल कर रहे हैं या मल्टी-स्टेप गणित कर रहे हैं, तो उन्हें व्यक्तिगत रूप से प्रोसेस करना एक खराब उपयोगकर्ता अनुभव बनाता है। बैचिंग का मतलब है दस समस्याओं को हल करने के लिए एक चक्कर (round-trip)।

सिर्फ `eval()` के बजाय एक सिम्बॉलिक इंजन का उपयोग करना (कृपया `eval()` का उपयोग न करें) हमें वास्तविक गणितीय समझ देता है। इंजन उद्देश्य को पार्स करता है, LaTeX फॉर्मेटिंग को संभालता है, और डेरिवेटिव और इंटीग्रल के साथ काम कर सकता है। हम केवल कैलकुलेशन नहीं कर रहे हैं, हम गणित कर रहे हैं।

एरर हैंडलिंग प्रत्येक एक्सप्रेशन के हिसाब से स्कोप की गई है। यदि एक कैलकुलेशन विफल हो जाती है, तो हम वह एरर वापस कर देते हैं लेकिन बाकी के साथ आगे बढ़ते रहते हैं। यह मॉडल को यह देखने की अनुमति देता है कि क्या काम किया और क्या नहीं, जिससे संभावित रूप से अगले चरण में स्व-सुधार (self-correcting) हो सके।

## इसे काम में लाना

आइए इसके सामने कुछ ऐसा रखें जो आमतौर पर एक रॉ मॉडल (raw model) को मतिभ्रम (hallucinate) करने के लिए मजबूर कर देगा:

```typescript
import { anthropic } from '@ai-sdk/anthropic';

const { text } = await generateText({
  model: anthropic('claude-sonnet-4-5'),
  prompt: 'Calculate 18472 × 9347, divide by 127, then take the square root of the result.',
  tools: { mathTool },
  stopWhen: stepCountIs(5), // Allow up to five model/tool steps
});

console.log(text);
```

मॉडल गणित को देखता है, पहचानता है कि उसे सटीकता की आवश्यकता है, टूल को कॉल करता है, सटीक परिणाम प्राप्त करता है, और फिर उसे प्राकृतिक भाषा में समझाता है। प्रत्येक घटक वह कर रहा है जो वह सबसे अच्छा करता है।

## बेसिक अंकगणित से परे

चूंकि हम एक सिम्बॉलिक इंजन का उपयोग कर रहे हैं, इसलिए यह दृष्टिकोण उन चीजों को संभालता है जिन्हें साधारण कैलकुलेटर टूल छू भी नहीं सकते।

बीजगणितीय समीकरणों (algebraic equations) को हल करना चाहते हैं? "Solve these equations: 3x + 7 = 22 and 2y - 5 = 13" ठीक काम करता है।

कैलकुलस (calculus) की जरूरत है? "Find the derivative of x^3 + 2x^2 and evaluate it at x = 2" बस एक और टूल कॉल है।

यदि आप शैक्षिक ऐप (educational apps) बना रहे हैं तो LaTeX सपोर्ट विशेष रूप से उपयोगी है। इंजन स्वाभाविक रूप से LaTeX इनपुट को समझता है और रेंडरिंग के लिए फॉर्मेटेड परिणाम दे सकता है। किसी अतिरिक्त पार्सिंग की आवश्यकता नहीं है।

## बड़ी तस्वीर

मुझे लगता है कि यह पैटर्न सिर्फ गणित से परे मायने रखता है। हम वास्तव में उनकी ताकत का लाभ उठाते हुए LLMs की सीमाओं को स्वीकार कर रहे हैं। वे इरादे को समझने, प्राकृतिक भाषा को पार्स करने और वर्कफ़्लो को व्यवस्थित करने में अविश्वसनीय हैं। वे कैलकुलेटर या डेटाबेस या फाइल सिस्टम नहीं हैं।

हर बार जब हम किसी LLM से कुछ डिटरमिनिस्टिक (deterministic) करवाने की कोशिश करते हैं, तो हम उसकी प्रकृति के खिलाफ लड़ रहे होते हैं। लेकिन जब हम उस प्राकृतिक भाषा की समझ को विशेष उपकरणों के साथ जोड़ते हैं जो डिटरमिनिस्टिक हिस्सों को संभालते हैं? तभी चीजें दिलचस्प हो जाती हैं।

मैथ टूल सिर्फ एक उदाहरण है। वही सिद्धांत तारीखों में हेरफेर (date manipulation), वित्तीय गणनाओं (financial calculations), इमेज प्रोसेसिंग, डेटाबेस क्वेरी... जहां भी रचनात्मकता से अधिक सटीकता (precision) मायने रखती है, वहां लागू होता है। मॉडल को यह समझने दें कि उपयोगकर्ता क्या चाहता है, फिर वास्तविक काम को उस चीज़ को सौंप दें जो उस काम के लिए बनी है।

यह एआई के साथ निर्माण के बारे में हमारी सोच में एक बदलाव है। "क्या मॉडल यह कर सकता है?" नहीं, बल्कि "क्या मॉडल इसे ऑर्केस्ट्रेट कर सकता है?" वाक्यांश में छोटा सा अंतर, लेकिन विश्वसनीयता में महत्वपूर्ण अंतर।
