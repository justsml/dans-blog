Du weißt, was seltsam an Sprachmodellen ist? Sie können Quantenmechanik erklären, Gedichte schreiben und deinen TypeScript-Code debuggen … aber bitte sie, 18472 mit 9347 zu multiplizieren, und mit anständiger Wahrscheinlichkeit bekommst du etwas, das um Tausender daneben liegt.

Das hat mich früher verwirrt, bis ich verstanden habe, was wir da eigentlich von ihnen verlangen. Wir bitten eine Mustererkennungs-Engine, ein Taschenrechner zu sein. Das ist, als würdest du einen Turner bitten, dein Kontoauszug zu prüfen, nur weil er das Konzept von „Gleichgewicht" versteht.

Die Sache ist: LLMs rechnen gar nichts. Wenn du GPT oder Claude fragst, was 2 + 2 ergibt, addieren sie nicht. Sie sagen voraus, dass „4" das Token ist, das am wahrscheinlichsten nach „2 + 2 =" kommt. Meistens funktioniert das prima, weil diese Muster in ihren Trainingsdaten existieren. Aber gehst du über einfache Arithmetik hinaus zu mehrschrittigen Berechnungen oder Zahlen, die im Training nicht üblich waren, würfelst du im Grunde nur.

Ich bin vor Kurzem bei der Durchsicht von Code, der ein Top-Modell zur Berechnung von Hypothekenzahlungen verwendete, direkt darauf gestoßen. Das Modell antwortete mit voller Überzeugung. Es lag trotzdem um 400 $ pro Monat daneben. Das ist die Art von Fehler, die zählt.

Selbst wenn Modelle besser im logischen Denken werden (GPT-5 zeigt angeblich Verbesserungen), betreiben sie immer noch anspruchsvolle Mustererkennung, kein symbolisches Rechnen. Für kreative Arbeit und natürliche Sprachaufgaben ist diese probabilistische Natur genau das, was sie magisch macht. Für Mathematik? Eher nicht.

## Was löst das tatsächlich?

Die Antwort ist nicht, auf schlauere Modelle zu warten. Sondern dem Modell das richtige Werkzeug für die Aufgabe zu geben.

Überleg mal, wie du dieses Problem lösen würdest, wenn du ein System ohne KI bauen würdest. Du würdest keine eigene Mathematiklogik schreiben, sondern eine Bibliothek heranziehen. Gleiches Prinzip hier – nur dass wir dem LLM jetzt beibringen, wann und wie es diese Bibliothek nutzen soll.

Tool Calling in modernen AI SDKs erlaubt es uns, dem Modell strukturierte Funktionen zu übergeben, die es aufrufen kann. Statt das LLM zu zwingen, so zu tun, als könne es rechnen, geben wir ihm etwas, das es wirklich kann: eine symbolische Mathematik-Engine.

Ich nutze dafür [AI SDK v5 und v6](https://ai-sdk.vercel.ai/) in Kombination mit CortexJS Compute Engine. Das SDK übernimmt Orchestrierung und Tool-Routing, während CortexJS alles von einfacher Arithmetik bis zur Analysis abdeckt. Eine überraschend saubere Trennung der Verantwortlichkeiten.

```bash
bun add ai @ai-sdk/anthropic @cortex-js/compute-engine zod
```

## Das Math-Tool bauen

Die Implementierung ist einfacher, als du vielleicht denkst. Wir bauen im Grunde eine Brücke zwischen dem natürlichsprachlichen Verständnis des LLMs und echter mathematischer Berechnung.

```typescript
import { generateText, stepCountIs, tool } from 'ai';
import { ComputeEngine } from '@cortex-js/compute-engine';
import { z } from 'zod';

// Engine einmal initialisieren
const ce = new ComputeEngine();

const mathTool = tool({
  description: 'Evaluate mathematical expressions and solve equations with guaranteed accuracy. MUST be used for all mathematical operations to verify correctness - do not attempt mental math. Supports arithmetic, algebra, calculus, and complex operations. Can process multiple expressions at once.',
  parameters: z.object({
    expressions: z.array(z.string()).describe(
      'Array of mathematical expressions in LaTeX or plain notation, e.g. ["2 + 2", "\\frac{x^2 + 1}{x - 1}", "\\int x^2 dx"]'
    ),
  }),
  execute: async ({ expressions }) => {
    // Alle Ausdrücke parallel verarbeiten
    return expressions.map(expression => {
      try {
        const result = ce.parse(expression).evaluate();
        return {
          expression,
          result: result.toString(),
          latex: result.latex,
        };
      } catch (error) {
        return { 
          expression,
          error: (error as Error).message 
        };
      }
    });
  },
});
```

Ein paar Dinge, die hier erwähnenswert sind:

Die Beschreibung leistet Schwerstarbeit. Dieses „MUST be used" mag aggressiv klingen, aber meiner Erfahrung nach ist die explizite Ansage, wann das Modell ein Tool nutzen soll, der Unterschied zwischen „funktioniert manchmal" und „funktioniert zuverlässig". Betrachte es als Prompt Engineering auf Tool-Ebene.

Die Batch-Verarbeitung über ein `expressions`-Array ist wichtiger, als du denkst. Jeder Modellaufruf hat Latenz. Wenn du ein Gleichungssystem löst oder mehrschrittige Mathematik betreibst, erzeugt die Einzelverarbeitung eine miserable Benutzererfahrung. Batching bedeutet einen Round-Trip, um zehn Probleme zu lösen.

Der Einsatz einer symbolischen Engine statt bloßem `eval()` (bitte benutze kein `eval()`) gibt uns echtes mathematisches Verständnis. Die Engine analysiert die Absicht, verarbeitet LaTeX-Formatierung und kann mit Ableitungen und Integralen arbeiten. Wir machen nicht nur Berechnungen, wir betreiben Mathematik.

Die Fehlerbehandlung ist pro Ausdruck begrenzt. Schlägt eine Berechnung fehl, geben wir diesen Fehler zurück, machen aber mit dem Rest weiter. So sieht das Modell, was funktioniert hat und was nicht, und kann sich im nächsten Schritt möglicherweise selbst korrigieren.

## In der Praxis

Werfen wir etwas darauf, bei dem ein reines Modell normalerweise hallucinieren würde:

```typescript
import { anthropic } from '@ai-sdk/anthropic';

const { text } = await generateText({
  model: anthropic('claude-sonnet-4-5'),
  prompt: 'Calculate 18472 × 9347, divide by 127, then take the square root of the result.',
  tools: { mathTool },
  stopWhen: stepCountIs(5), // Bis zu fünf Modell-/Tool-Schritte erlauben
});

console.log(text);
```

Das Modell sieht die Mathematik, erkennt, dass es Präzision braucht, ruft das Tool auf, bekommt das genaue Ergebnis und erklärt es dann in natürlicher Sprache. Jede Komponente macht das, was sie am besten kann.

## Jenseits einfacher Arithmetik

Da wir eine symbolische Engine verwenden, bewältigt dieser Ansatz Dinge, die einfache Rechner-Tools nicht anrühren können.

Algebraische Gleichungen lösen? „Solve these equations: 3x + 7 = 22 and 2y - 5 = 13" funktioniert problemlos.

Analysis nötig? „Find the derivative of x^3 + 2x^2 and evaluate it at x = 2" ist nur ein weiterer Tool-Aufruf.

Die LaTeX-Unterstützung ist besonders nützlich, wenn du Bildungs-Apps baust. Die Engine versteht LaTeX-Eingabe von Haus aus und kann Ergebnisse formatierter für die Darstellung zurückgeben. Kein zusätzliches Parsen nötig.

## Das große Ganze

Ich denke, dieses Muster geht über Mathematik hinaus. Was wir hier wirklich tun, ist, die Grenzen von LLMs anzuerkennen und gleichzeitig ihre Stärken zu nutzen. Sie sind unglaublich gut darin, Absichten zu verstehen, natürliche Sprache zu parsen und Arbeitsabläufe zu orchestrieren. Sie sind keine Taschenrechner, Datenbanken oder Dateisysteme.

Jedes Mal, wenn wir versuchen, ein LLM etwas Deterministisches tun zu lassen, kämpfen wir gegen seine Natur. Aber wenn wir dieses natürlichsprachliche Verständnis mit spezialisierten Werkzeugen kombinieren, die die deterministischen Teile übernehmen? Dann wird es interessant.

Das Math-Tool ist nur ein Beispiel. Das gleiche Prinzip gilt für Datumsmanipulation, Finanzberechnungen, Bildverarbeitung, Datenbankabfragen … überall dort, wo Präzision mehr zählt als Kreativität. Lass das Modell verstehen, was der Benutzer will, und übergib die eigentliche Arbeit dann an etwas, das dafür gebaut wurde.

Es ist ein Perspektivwechsel in der Art, wie wir über das Bauen mit KI denken. Nicht „Kann das Modell das?" sondern „Kann das Modell das orchestrieren?" Kleiner Unterschied in der Formulierung, großer Unterschied in der Zuverlässigkeit.
