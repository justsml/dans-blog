Du weißt, was an Sprachmodellen seltsam ist? Sie können Quantenmechanik erklären, Gedichte schreiben und dein TypeScript debuggen... aber frag sie, 18472 mit 9347 zu multiplizieren, und es besteht eine gute Chance, dass sie dir selbstbewusst etwas liefern, das um Tausende falsch liegt.

Das hat mich früher verwundert, bis ich realized habe, was wir sie wirklich bitten zu tun. Wir bitten ein Pattern-Matching-System, ein Taschenrechner zu sein. Das ist, als würdest du einen Turner bitten, dein Konto auszugleichen, weil er das Konzept von "Gleichgewicht" versteht.

Die Sache ist: LLMs berechnen nichts. Wenn du GPT oder Claude fragst, was 2 + 2 ergibt, rechnen sie nicht. Sie sagen vorher, dass "4" das wahrscheinlichste Token ist, das nach "2 + 2 =" erscheint. Meistens funktioniert das großartig, weil diese Muster in ihren Trainingsdaten vorhanden sind. Aber wenn du über einfache Arithmetik hinausgehst oder mit Zahlen arbeitest, die im Training nicht häufig vorkamen, würfelst du im Grunde.

Ich bin kürzlich dagegen gestoßen, als ich Code überprüft habe, der ein Top-Modell zur Berechnung von Hypothekenzahlungen nutzte. Das Modell antwortete mit voller Überzeugung. Es war auch um 400€/Monat falsch. Das ist die Art von Fehler, die relevant ist.

Selbst wenn Modelle beim Reasoning besser werden ( angeblich zeigt GPT-5 Verbesserungen), machen sie immer noch ausgefeiltes Pattern Matching, keine symbolische Berechnung. Für kreative Arbeit und natürliche Sprachaufgaben ist diese probabilistische Natur genau das, was sie magisch macht. Für Mathe? Nicht so sehr.

## Was das Problem tatsächlich löst

Die Antwort ist nicht, auf klügere Modelle zu warten. Es geht darum, dem Modell das richtige Werkzeug für die Aufgabe zu geben.

Denk daran, wie du dieses Problem lösen würdest, wenn du ein Nicht-KI-System bauen würdest. Du würdest keine benutzerdefinierte Mathe-Logik schreiben, du würdest eine Bibliothek verwenden. Dasselbe Prinzip gilt hier, außer dass wir jetzt dem LLM beibringen, wann und wie es diese Bibliothek nutzt.

Tool-Calling in modernen AI SDKs ermöglicht es uns, dem Modell strukturierte Funktionen zu geben, die es aufrufen kann. Anstatt das LLM zu zwingen, so zu tun, als wüsste es Mathe, geben wir ihm etwas, das es tatsächlich kann: eine symbolische Mathe-Engine.

Ich habe [AI SDK v5 und v6](https://ai-sdk.vercel.ai/) dafür verwendet, gepaart mit CortexJS Compute Engine. Das SDK kümmert sich um Orchestrierung und Tool-Routing, während CortexJS alles von grundlegender Arithmetik bis hin zur Analysis verarbeitet. Es ist eine überraschend saubere Aufgabenteilung.

```bash
bun add ai @ai-sdk/anthropic @cortex-js/compute-engine zod
```

## Den Mathe-Tool bauen

Die Implementierung ist unerwartet straightforward. Was wir bauen, ist eine Brücke zwischen dem natürlichen Sprachverständnis des LLM und tatsächlicher mathematischer Berechnung.

```typescript
import { generateText, stepCountIs, tool } from 'ai';
import { ComputeEngine } from '@cortex-js/compute-engine';
import { z } from 'zod';

// Initialize the engine once
const ce = new ComputeEngine();

const mathTool = tool({
  description: 'Evaluate mathematical expressions and solve equations with guaranteed accuracy. MUST be used for all mathematical operations to verify correctness - do not attempt mental math. Supports arithmetic, algebra, calculus, and complex operations. Can process multiple expressions at once.',
  parameters: z.object({
    expressions: z.array(z.string()).describe(
      'Array of mathematical expressions in LaTeX or plain notation, e.g. ["2 + 2", "\\frac{x^2 + 1}{x - 1}", "\\int x^2 dx"]'
    ),
  }),
  execute: async ({ expressions }) => {
    // Process all expressions in parallel (or detailed batch)
    return expressions.map(expression => {
      try {
        const result = ce.parse(expression).evaluate();
        return {
          expression,
          result: result.toString(),
          latex: result.latex,
        };
      } catch (error) {
        return { 
          expression,
          error: (error as Error).message 
        };
      }
    });
  },
});
```

Ein paar Dinge, die wert sind, beachtet zu werden:

Die Beschreibung leistet schwere Arbeit. Diese "MUST be used"-Sprache könnte aggressiv wirken, aber nach meiner Erfahrung ist das Explizite dem Modell gegenüber darüber, wann ein Tool verwendet werden soll, der Unterschied zwischen "funktioniert manchmal" und "funktioniert zuverlässig". Betrachte es als Prompt-Engineering auf Tool-Ebene.

Die Stapelverarbeitung über ein `expressions`-Array ist wichtiger, als du vielleicht denkst. Jeder Modellaufruf hat Latenz. Wenn du ein Gleichungssystem oder mehrstufige Mathematik löst, verursacht die individuelle Verarbeitung eine schlechte Nutzererfahrung. Stapelverarbeitung bedeutet eine Anfrage, um zehn Probleme zu lösen.

Eine symbolische Engine zu verwenden anstatt nur `eval()` (bitte nutze kein `eval()`) gibt uns echtes mathematisches Verständnis. Die Engine parst Absicht, handhabt LaTeX-Formatierung und kann mit Ableitungen und Integralen arbeiten. Wir machen nicht nur Berechnungen, wir machen Mathematik.

Die Fehlerbehandlung ist pro Ausdruck abgegrenzt. Wenn eine Berechnung fehlschlägt, geben wir diesen Fehler zurück, fahren aber mit dem Rest fort. Das ermöglicht es dem Modell zu sehen, was funktioniert hat und was nicht, und möglicherweise im nächsten Schritt selbst zu korrigieren.

## Den Tool einsetzen

Lass uns etwas daran ausprobieren, das ein normales Modell normalerweise halluzinieren lassen würde:

```typescript
import { anthropic } from '@ai-sdk/anthropic';

const { text } = await generateText({
  model: anthropic('claude-sonnet-4-5'),
  prompt: 'Calculate 18472 × 9347, divide by 127, then take the square root of the result.',
  tools: { mathTool },
  stopWhen: stepCountIs(5), // Allow up to five model/tool steps
});

console.log(text);
```

Das Modell sieht die Mathematik, erkennt, dass Präzision benötigt wird, ruft den Tool auf, bekommt das akkurate Ergebnis und erklärt es dann in natürlicher Sprache. Jede Komponente macht das, was sie am besten kann.

## Über grundlegende Arithmetik hinaus

Da wir eine symbolische Engine verwenden, handhabt dieser Ansatz Dinge, die einfache Taschenrechner-Tools nicht anfassen können.

Willst du algebraische Gleichungen lösen? "Löse diese Gleichungen: 3x + 7 = 22 und 2y - 5 = 13" funktioniert einwandfrei.

Brauchst du Analysis? "Finde die Ableitung von x^3 + 2x^2 und berechne sie an der Stelle x = 2" ist nur ein weiterer Tool-Aufruf.

Die LaTeX-Unterstützung ist besonders nützlich, wenn du Bildungs-Apps baust. Die Engine versteht LaTeX-Eingaben inherent und kann Ergebnisse für das Rendering formatiert zurückgeben. Keine zusätzliche Parsing erforderlich.

## Das große Ganze

Ich denke, dieses Muster ist über Mathemaktik hinaus relevant. Was wir wirklich tun, ist die Grenzen von LLMs anzuerkennen und gleichzeitig ihre Stärken zu nutzen. Sie sind unglaublich darin, Absicht zu verstehen, natürliche Sprache zu parsen und Workflows zu orchestrieren. Sie sind keine Taschenrechner, Datenbanken oder Dateisysteme.

Jedes Mal, wenn wir versuchen, ein LLM dazu zu bringen, etwas Deterministisches zu tun, kämpfen wir gegen seine Natur. Aber wenn wir dieses natürliche Sprachverständnis mit spezialisierten Tools kombinieren, die die deterministischen Teile erledigen? Dann wird es interessant.

Der Mathe-Tool ist nur ein Beispiel. Dasselbe Prinzip gilt für Datummanipulation, Finanzberechnungen, Bildverarbeitung, Datenbankabfragen... überall dort, wo Präzision wichtiger ist als Kreativität. Lass das Modell verstehen, was der Nutzer will, und gib dann die eigentliche Arbeit an etwas weiter, das dafür gebaut wurde.

Es ist ein Shift darin, wie wir darüber nachdenken, mit KI zu bauen. Nicht "kann das Modell das?" sondern "kann das Modell das orchestrieren?" Kleiner Unterschied in der Formulierung, signifikanter Unterschied in der Zuverlässigkeit.
