Знаете, что странного в языковых моделях? Они могут объяснить квантовую механику, написать стихотворение и отладить ваш TypeScript... но попросите их умножить 18472 на 9347, и есть неплохой шанс, что они с полной уверенностью дадут вам результат, отличающийся на тысячи.

Раньше это ставило меня в тупик, пока я не понял, что мы actually просим их делать. Мы просим движок распознавания образов быть калькулятором. Это всё равно что просить гимнаста свести ваш бюджет, потому что он понимает концепцию «баланса».

Дело в том, что LLM не вычисляют ничего. Когда вы спрашиваете GPT или Claude, чему равно 2 + 2, они не складывают. Они предсказывают, что «4» — это токен, который с наибольшей вероятностью появится после «2 + 2 =». Большую часть времени это отлично работает, потому что эти паттерны есть в их обучающих данных. Но выйдите за пределы простой арифметики к многошаговым вычислениям или чему-то с числами, которые не были распространены в обучении, и вы по сути бросаете кости.

Я столкнулся с этим лицом к лицу недавно, когда рассматривал код, который использовал модель top-tier для расчёта ипотечных платежей. Модель ответила с полной уверенностью. Она также была неправа на $400/месяц. Это тот тип ошибки, которая имеет значение.

Даже когда модели становятся лучше в рассуждениях (GPT-5 якобы показывает улучшения), они всё ещё делают продвинутое распознавание образов, а не символьные вычисления. Для творческой работы и задач обработки естественного языка эта вероятностная природа — это именно то, что делает их волшебными. Для математики? Не совсем.

## Что на самом деле это решает?

Ответ не в том, чтобы ждать более умных моделей. Это дать модели правильный инструмент для работы.

Подумайте, как бы вы решили эту проблему, если бы строили систему без AI. Вы бы не писали собственную математическую логику, вы бы взяли библиотеку. Тот же принцип применим здесь, за исключением того, что сейчас мы учим LLM, когда и как использовать эту библиотеку.

Вызов инструментов в современных AI SDK позволяет передать модели структурированные функции, которые она может вызывать. Вместо того чтобы заставлять LLM притворяться, что она знает математику, мы даём ей то, что действительно умеет: символьный математический движок.

Я использовал AI SDK v5 и v6 для этого, в паре с CortexJS Compute Engine. SDK обрабатывает оркестрацию и маршрутизацию инструментов, в то время как CortexJS обрабатывает что угодно от базовой арифметики до математического анализа. Это удивительно чистое разделение задач.

```bash
bun add ai @ai-sdk/anthropic @cortex-js/compute-engine zod
```

## Создание математического инструмента

Реализация более прямолинейна, чем вы можете ожидать. То, что мы строим, — это мост между пониманием естественного языка LLM и реальными математическими вычислениями.

```typescript
import { generateText, stepCountIs, tool } from 'ai';
import { ComputeEngine } from '@cortex-js/compute-engine';
import { z } from 'zod';

// Initialize the engine once
const ce = new ComputeEngine();

const mathTool = tool({
  description: 'Evaluate mathematical expressions and solve equations with guaranteed accuracy. MUST be used for all mathematical operations to verify correctness - do not attempt mental math. Supports arithmetic, algebra, calculus, and complex operations. Can process multiple expressions at once.',
  parameters: z.object({
    expressions: z.array(z.string()).describe(
      'Array of mathematical expressions in LaTeX or plain notation, e.g. ["2 + 2", "\\frac{x^2 + 1}{x - 1}", "\\int x^2 dx"]'
    ),
  }),
  execute: async ({ expressions }) => {
    // Process all expressions in parallel (or detailed batch)
    return expressions.map(expression => {
      try {
        const result = ce.parse(expression).evaluate();
        return {
          expression,
          result: result.toString(),
          latex: result.latex,
        };
      } catch (error) {
        return {
          expression,
          error: (error as Error).message
        };
      }
    });
  },
});
```

Несколько моментов, на которые стоит обратить внимание:

Описание выполняет тяжёлую работу. Язык «MUST be used» может показаться агрессивным, но по моему опыту, быть явным с моделью о том, когда использовать инструмент, — это разница между работой иногда и надёжной работой. Считайте это промпт-инженерией на уровне инструмента.

Пакетная обработка через массив `expressions` имеет значение больше, чем вы можете подумать. Каждый вызов модели имеет задержку. Если вы решаете систему уравнений или делаете многошаговую математику, обработка их по отдельности создаёт ужасный пользовательский опыт. Пакетная обработка означает один раунд для решения десяти задач.

Использование символьного движка вместо просто `eval()` (пожалуйста, не используйте `eval()`) даёт нам реальное математическое понимание. Движок парсит смысл, обрабатывает форматирование LaTeX и может работать с производными и интегралами. Мы не просто делаем вычисления, мы делаем математику.

Обработка ошибок scoped по выражению. Если одно вычисление не удаётся, мы возвращаем эту ошибку, но продолжаем с остальными. Это позволяет модели увидеть, что сработало, а что нет, потенциально самоисправляясь на следующем шаге.

## Применяем на практике

Давайте дадим ему что-то, что обычно заставляет сырую модель галлюцинировать:

```typescript
import { anthropic } from '@ai-sdk/anthropic';

const { text } = await generateText({
  model: anthropic('claude-sonnet-4-5'),
  prompt: 'Calculate 18472 × 9347, divide by 127, then take the square root of the result.',
  tools: { mathTool },
  stopWhen: stepCountIs(5), // Allow up to five model/tool steps
});

console.log(text);
```

Модель видит математику, распознаёт, что нужна точность, вызывает инструмент, получает точный результат, а затем объясняет его на естественном языке. Каждый компонент делает то, что умеет лучше всего.

## За пределами базовой арифметики

Поскольку мы используем символьный движок, этот подход обрабатывает вещи, которых простые калькуляторы не могут коснуться.

Хотите решить алгебраические уравнения? «Решите эти уравнения: 3x + 7 = 22 и 2y - 5 = 13» отлично работает.

Нужен матан? «Найдите производную x^3 + 2x^2 и вычислите её при x = 2» — это просто ещё один вызов инструмента.

Поддержка LaTeX особенно полезна, если вы создаёте образовательные приложения. Движок естественным образом понимает ввод LaTeX и может возвращать результаты, отформатированные для рендеринга. Дополнительный парсинг не требуется.

## Общая картина

Я думаю, этот паттерн важен за пределами просто математики. Что мы на самом деле делаем, — это признаём ограничения LLM, используя их преимущества. Они невероятны в понимании смысла, парсинге естественного языка и оркестрации рабочих процессов. Они не калькуляторы, не базы данных и не файловые системы.

Каждый раз, когда мы пытаемся заставить LLM делать что-то детерминированное, мы боремся с её природой. Но когда мы объединяем это понимание естественного языка со специализированными инструментами, которые обрабатывают детерминированные части? Вот тогда становится интересно.

Математический инструмент — это лишь один пример. Тот же принцип применим к манипуляциям с датами, финансовым расчётам, обработке изображений, запросам к базам данных... везде, где точность важнее креативности. Пусть модель понимает, что хочет пользователь, а затем передаёт реальную работу тому, что создано для этого.

Это сдвиг в том, как мы думаем о построении с AI. Не «может ли модель это сделать?», а «может ли модель это оркестрировать?» Маленькая разница в формулировке, значительная разница в надёжности.
