Знаете, что странного в языковых моделях? Они могут объяснять квантовую механику, писать стихи и отлаживать ваш TypeScript... но попросите их умножить 18472 на 9347 — и с большой вероятностью они уверенно выдадут результат, ошибающийся на тысячи.

Раньше это ставило меня в тупик, пока я не понял, что мы на самом деле от них просим. Мы просим движок поиска шаблонов быть калькулятором. Это всё равно что попросить гимнаста вести вашу чековую книжку, потому что он понимает концепцию «баланса».

Дело в том, что LLM ничего не вычисляют. Когда вы спрашиваете GPT или Claude, сколько будет 2 + 2, они не складывают. Они предсказывают, что «4» — это токен, наиболее вероятный после «2 + 2 =». В большинстве случаев это отлично работает, потому что эти паттерны есть в их обучающих данных. Но выйдите за рамки простой арифметики в многошаговые вычисления или в числа, которые редко встречались в обучении, — и вы по сути бросаете кости.

Я столкнулся с этим в лоб недавно, проверяя код, который использовал топовую модель для расчёта ипотечных платежей. Модель ответила с абсолютной уверенностью. И ошиблась на 400 долларов в месяц. Это та ошибка, которая имеет значение.

Даже когда модели становятся лучше в рассуждениях (GPT-5 якобы показывает улучшения), они всё равно занимаются сложным поиском шаблонов, а не символьными вычислениями. Для творческой работы и задач на естественном языке эта вероятностная природа — именно то, что делает их волшебными. Для математики? Не очень.

## Что на самом деле решает эту проблему?

Ответ — не в ожидании более умных моделей. А в том, чтобы дать модели правильный инструмент для работы.

Подумайте, как вы бы решили эту задачу, создавая обычную (не AI) систему. Вы бы не стали писать свою математическую логику, а воспользовались бы библиотекой. Тот же принцип применим и здесь, только теперь мы учим LLM, когда и как использовать эту библиотеку.

Вызов инструментов (tool calling) в современных AI SDK позволяет передать модели структурированные функции, которые она может вызывать. Вместо того чтобы заставлять LLM притворяться, что она знает математику, мы даём ей то, что действительно её знает: символьный математический движок.

Я использую для этого [AI SDK v5 и v6](https://ai-sdk.vercel.ai/) в паре с CortexJS Compute Engine. SDK занимается оркестрацией и маршрутизацией инструментов, а CortexJS — всем от базовой арифметики до матанализа. Это на удивление чистое разделение ответственности.

```bash
bun add ai @ai-sdk/anthropic @cortex-js/compute-engine zod
```

## Создание математического инструмента

Реализация проще, чем можно было бы ожидать. По сути, мы строим мост между пониманием естественного языка LLM и реальными математическими вычислениями.

```typescript
import { generateText, stepCountIs, tool } from 'ai';
import { ComputeEngine } from '@cortex-js/compute-engine';
import { z } from 'zod';

// Инициализируем движок один раз
const ce = new ComputeEngine();

const mathTool = tool({
  description: 'Вычисляет математические выражения и решает уравнения с гарантированной точностью. ДОЛЖЕН использоваться для всех математических операций для проверки корректности — не пытайтесь считать в уме. Поддерживает арифметику, алгебру, матанализ и сложные операции. Может обрабатывать несколько выражений за раз.',
  parameters: z.object({
    expressions: z.array(z.string()).describe(
      'Массив математических выражений в LaTeX или обычной нотации, например ["2 + 2", "\\frac{x^2 + 1}{x - 1}", "\\int x^2 dx"]'
    ),
  }),
  execute: async ({ expressions }) => {
    // Обрабатываем все выражения параллельно (или детальным пакетом)
    return expressions.map(expression => {
      try {
        const result = ce.parse(expression).evaluate();
        return {
          expression,
          result: result.toString(),
          latex: result.latex,
        };
      } catch (error) {
        return { 
          expression,
          error: (error as Error).message 
        };
      }
    });
  },
});
```

Несколько моментов, которые стоит отметить:

Описание делает большую часть работы. Этот язык с «ДОЛЖЕН использоваться» может показаться агрессивным, но, по моему опыту, чёткое указание модели, когда использовать инструмент, — это разница между работой «иногда» и «надёжно». Считайте это промпт-инжинирингом на уровне инструмента.

Пакетная обработка через массив `expressions` важнее, чем кажется. Каждый вызов модели добавляет задержку. Если вы решаете систему уравнений или делаете многошаговые вычисления, обработка каждого выражения по отдельности создаёт ужасный пользовательский опыт. Пакетная обработка означает один обмен на решение десятка задач.

Использование символьного движка вместо `eval()` (пожалуйста, не используйте `eval()`) даёт нам настоящее математическое понимание. Движок разбирает намерение, обрабатывает LaTeX-форматирование и может работать с производными и интегралами. Мы не просто вычисляем — мы занимаемся математикой.

Обработка ошибок выполняется для каждого выражения отдельно. Если одно вычисление падает, мы возвращаем эту ошибку, но продолжаем с остальными. Это позволяет модели увидеть, что сработало, а что нет, и, возможно, самокорректироваться на следующем шаге.

## Применение на практике

Давайте попробуем то, с чем сырая модель обычно справляется через галлюцинации:

```typescript
import { anthropic } from '@ai-sdk/anthropic';

const { text } = await generateText({
  model: anthropic('claude-sonnet-4-5'),
  prompt: 'Вычисли 18472 × 9347, раздели на 127, затем извлеки квадратный корень из результата.',
  tools: { mathTool },
  stopWhen: stepCountIs(5), // Разрешаем до пяти шагов модель/инструмент
});

console.log(text);
```

Модель видит математику, понимает, что нужна точность, вызывает инструмент, получает точный результат и объясняет его на естественном языке. Каждый компонент делает то, что у него получается лучше всего.

## За пределами базовой арифметики

Поскольку мы используем символьный движок, этот подход справляется с тем, что простые калькуляторы не могут.

Хотите решить алгебраические уравнения? «Реши эти уравнения: 3x + 7 = 22 и 2y - 5 = 13» — работает отлично.

Нужен матанализ? «Найди производную x^3 + 2x^2 и вычисли её в точке x = 2» — это просто ещё один вызов инструмента.

Поддержка LaTeX особенно полезна, если вы создаёте образовательные приложения. Движок изначально понимает LaTeX на входе и может возвращать результаты, готовые для рендеринга. Никакого дополнительного парсинга не требуется.

## Общая картина

Думаю, этот паттерн выходит далеко за рамки математики. По сути, мы признаём ограничения LLM, одновременно используя их сильные стороны. Они невероятны в понимании намерений, разборе естественного языка и оркестрации рабочих процессов. Но они не калькуляторы, не базы данных и не файловые системы.

Каждый раз, когда мы пытаемся заставить LLM делать что-то детерминированное, мы боремся с её природой. Но когда мы соединяем это понимание естественного языка со специализированными инструментами, которые отвечают за детерминированные задачи? Вот тогда всё становится интересно.

Математический инструмент — лишь один пример. Тот же принцип применим к работе с датами, финансовым расчётам, обработке изображений, запросам к базам данных... везде, где точность важнее креативности. Пусть модель понимает, чего хочет пользователь, а затем передаёт реальную работу тому, кто для неё создан.

Это сдвиг в том, как мы мыслим создание систем с AI. Не «может ли модель это сделать?», а «может ли модель этим управлять?» Маленькая разница в формулировке, значительная — в надёжности.
