言語モデルの奇妙な点を知っていますか？量子力学を説明し、詩を書き、TypeScriptのデバッグまでこなすのに……「18472 × 9347」の計算を頼むと、平然と数千もズレた答えを返してくることがよくあります。

かつて私はこれに困惑していましたが、私たちが実際にモデルに何をさせているのかを理解した瞬間に納得がいきました。私たちは「パターンマッチング・エンジン」に「計算機」になるよう求めているのです。それは、体操選手が「バランス」という概念を理解しているからといって、彼らに小切手帳の残高調整（バランス）を頼むようなものです。

実は、LLMは何の計算もしていません。GPTやClaudeに「2 + 2」の答えを聞いたとき、彼らは足し算をしているのではなく、「2 + 2 = 」の後に続くトークンとして最も可能性が高いのは「4」であると予測しているだけです。訓練データに存在する一般的なパターンであれば、ほとんどの場合はこれでうまくいきます。しかし、単純な算数を超えた多段階の計算や、訓練データにあまり登場しない数値が出てくると、それは本質的にサイコロを振っているのと変わりません。

最近、住宅ローンの支払い計算にトップティアのモデルを使用しているコードをレビューした際に、この問題に直面しました。モデルは完璧な自信を持って回答しましたが、月々の支払額は400ドルも間違っていました。これは、決して無視できない種類のエラーです。

（GPT-5などで）モデルの推論能力が向上していると言われていても、それらは依然として高度なパターンマッチングを行っているのであり、記号計算を行っているわけではありません。クリエイティブな作業や自然言語のタスクにおいて、この確率的な性質こそが魔法を生み出す源泉ですが、数学においてはそうではありません。

## 実際の解決策は？

その答えは、より賢いモデルを待つことではなく、モデルに「仕事に適した道具」を与えることです。

AIを使わないシステムを構築する場合、この問題をどう解決するか考えてみてください。独自の計算ロジックをゼロから書くのではなく、ライブラリを使うはずです。同じ原則がここでも当てはまります。違うのは、LLMに「いつ、どのようにそのライブラリを使うか」を教えるという点です。

現代のAI SDKにおけるツール呼び出し（Tool calling）機能を使えば、モデルが呼び出せる構造化された関数を渡すことができます。LLMに数学が得意なふりをさせる代わりに、実際に数学ができるもの、つまり「記号数学エンジン」を渡すのです。

私はこれに [AI SDK v5 および v6](https://ai-sdk.vercel.ai/) を使い、CortexJS Compute Engine と組み合わせています。SDKがオーケストレーションとツールルーティングを担当し、CortexJSが基本的な算術から微積分までを処理します。これは驚くほどクリーンな関心の分離です。

```bash
bun add ai @ai-sdk/anthropic @cortex-js/compute-engine zod
```

## 数学ツールの構築

実装は予想以上にシンプルです。私たちが構築するのは、LLMの自然言語理解と、実際の数学的計算の間の架け橋です。

```typescript
import { generateText, stepCountIs, tool } from 'ai';
import { ComputeEngine } from '@cortex-js/compute-engine';
import { z } from 'zod';

// エンジンを一度だけ初期化
const ce = new ComputeEngine();

const mathTool = tool({
  description: '数式を評価し、正確性を保証して方程式を解きます。正確性を検証するために、すべての数学的操作に必ず使用しなければなりません。暗算を試みてはいけません。四則演算、代数、微積分、および複雑な操作をサポートしています。複数の式を一度に処理できます。',
  parameters: z.object({
    expressions: z.array(z.string()).describe(
      'LaTeXまたはプレーンテキスト形式の数式の配列。例: ["2 + 2", "\\frac{x^2 + 1}{x - 1}", "\\int x^2 dx"]'
    ),
  }),
  execute: async ({ expressions }) => {
    // すべての式を並列（または詳細なバッチ）で処理
    return expressions.map(expression => {
      try {
        const result = ce.parse(expression).evaluate();
        return {
          expression,
          result: result.toString(),
          latex: result.latex,
        };
      } catch (error) {
        return { 
          expression,
          error: (error as Error).message 
        };
      }
    });
  },
});
```

ここで注目すべき点がいくつかあります：

`description`（説明文）が重要な役割を果たしています。「必ず使用しなければなりません（MUST be used）」という表現は少し強引に見えるかもしれませんが、私の経験上、ツールを使うべきタイミングをモデルに明示することは、ツールが「たまに動く」か「確実に動く」かの分かれ目になります。これはツールレベルでのプロンプトエンジニアリングと言えます。

`expressions` 配列によるバッチ処理は、見た目以上に重要です。各モデル呼び出しにはレイテンシが発生します。連立方程式を解いたり、多段階の数学を行ったりする場合、一つずつ処理するとユーザーエクスペリエンスが最悪になります。バッチ処理にすることで、1回の往復で10個の問題を解決できます。

単なる `eval()` ではなく記号エンジンを使用することで（お願いですから `eval()` は使わないでください）、真の数学的理解が得られます。エンジンは意図を解析し、LaTeX形式を扱い、微分や積分も処理できます。単なる「計算」をしているのではなく、「数学」をしているのです。

エラーハンドリングは式ごとにスコープされています。一つの計算が失敗しても、そのエラーを返しつつ他の計算を続行します。これにより、モデルは何がうまくいき、何がうまくいかなかったかを確認でき、次のステップで自己修正できる可能性があります。

## 実際に動かしてみる

生のモデルなら高確率でハルシネーションを起こすような問題を投げてみましょう。

```typescript
import { anthropic } from '@ai-sdk/anthropic';

const { text } = await generateText({
  model: anthropic('claude-sonnet-4-5'),
  prompt: '18472 × 9347 を計算し、127 で割り、その結果の平方根を求めてください。',
  tools: { mathTool },
  stopWhen: stepCountIs(5), // 最大5ステップのモデル/ツール実行を許可
});

console.log(text);
```

モデルは数学であることを認識し、精度が必要だと判断してツールを呼び出し、正確な結果を取得してから、それを自然言語で説明します。各コンポーネントが、それぞれの得意分野をこなしています。

## 基本的な算術を超えて

記号エンジンを使用しているため、このアプローチは単純な電卓ツールでは太刀打ちできないものまで処理できます。

代数方程式を解きたい？ 「次の方程式を解いてください: 3x + 7 = 22 と 2y - 5 = 13」は問題なく動作します。

微積分が必要？ 「x^3 + 2x^2 の導関数を求め、x = 2 で評価してください」も、ただのツール呼び出しの一つです。

教育用アプリを構築している場合、LaTeXのサポートは特に便利です。エンジンは本質的にLaTeX入力を理解し、レンダリング用にフォーマットされた結果を返すことができます。追加のパースは必要ありません。

## 大きな視野で見る

このパターンは、単なる数学以上に重要だと考えています。私たちが本当に行っているのは、LLMの限界を認めつつ、その強みを活用することです。彼らは意図の理解、自然言語のパース、そしてワークフローのオーケストレーションにおいて驚異的な能力を発揮します。しかし、彼らは計算機でもデータベースでもファイルシステムでもありません。

LLMに決定論的なことをさせようとするたびに、私たちはその本質と戦っていることになります。しかし、その自然言語理解を、決定論的な部分を処理する専門的なツールと組み合わせたとき、事態は非常に面白くなります。

数学ツールはほんの一例に過ぎません。同じ原則は、日付操作、財務計算、画像処理、データベースクエリなど、創造性よりも正確さが重要なあらゆる場面に適用できます。モデルには「ユーザーが何を望んでいるか」を理解させ、実際の作業は専用に作られたものに任せましょう。

これは、AIを使った構築における考え方の転換です。「モデルにこれができるか？」ではなく、「モデルはこれをオーケストレートできるか？」という問いです。言い回しの小さな違いですが、信頼性においては大きな違いを生みます。
