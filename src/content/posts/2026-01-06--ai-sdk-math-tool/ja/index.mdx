言語モデルの変なところって知ってる？　量子力学を説明できたり、詩を書いたり、TypeScriptのデバッグだってできるのに、18472×9347って聞くと、何千もずれた答えを自信満々で返すことがあるんだ。

これには最初は首を傾げてた。でも、自分たちが実際に何を頼んでるのか考えてみると納得がいった。パターンマッチングエンジンに電卓の仕事をさせてるんだ。まるで「バランス」という概念を理解してるからって、体操選手に家計簿をつけさせるようなものだ。

実のところ、LLMは何も計算してない。GPTやClaudeに2+2は何かって聞いても、足し算してるわけじゃない。"2 + 2 =" の後に "4" というトークンが最も出現しやすいって予測してるだけなんだ。ほとんどの場合、これはうまくいく。学習データに存在するパターンだから。でも、単純な計算を超えて多段階の計算に進んだり、学習データにあまり出てこない数字が絡むと、サイコロを振ってるようなものだ。

最近、トップクラスのモデルを使って住宅ローンの支払い額を計算するコードをレビューしたときに、これを痛いほど実感した。モデルは満々の自信で答えた。月額400ドルもずれてた。これは重大なエラーだ。

モデルの推論能力が向上しても（GPT-5は改善しているらしい）、それでも高度なパターンマッチングをやってるだけで、記号的計算をしてるわけじゃない。創造的な仕事や自然言語タスクでは、この確率的な性質がまさに魔法のようなところなんだ。でも数学には？　そうはいかない。

## じゃあどうすればいい？

答えは、もっと賢いモデルを待つことじゃない。モデルに適切な道具を与えることだ。

AIを使わないシステムを作るとき、この問題をどう解決するか考えてみて。独自の数学ロジックを書かないだろ？　ライブラリを使う。ここでも同じ原則が適用される。ただし、今度はLLMにそのライブラリをいつ・どう使うか教えてあげるんだ。

現代のAI SDKでのツール呼び出しは、モデルに構造化された関数を呼び出させることができる。LLMに数学を知っているふりをさせるんじゃなくて、実際に数学ができるもの——記号的数学エンジン——を与えるんだ。

このために、[AI SDK v5とv6](https://ai-sdk.vercel.ai/)をCortexJS Compute Engineと組み合わせて使ってる。SDKがオーケストレーションとツールルーティングを担当して、CortexJSが基本的な算術から微積分まで何でも処理する。驚くほどきれいな関心事の分離だ。

```bash
bun add ai @ai-sdk/anthropic @cortex-js/compute-engine zod
```

## 数学ツールの構築

実装は思ったよりシンプルだ。作ってるのは、LLMの自然言語理解と実際の数学的計算をつなぐ橋だ。

```typescript
import { generateText, stepCountIs, tool } from 'ai';
import { ComputeEngine } from '@cortex-js/compute-engine';
import { z } from 'zod';

// Initialize the engine once
const ce = new ComputeEngine();

const mathTool = tool({
  description: 'Evaluate mathematical expressions and solve equations with guaranteed accuracy. MUST be used for all mathematical operations to verify correctness - do not attempt mental math. Supports arithmetic, algebra, calculus, and complex operations. Can process multiple expressions at once.',
  parameters: z.object({
    expressions: z.array(z.string()).describe(
      'Array of mathematical expressions in LaTeX or plain notation, e.g. ["2 + 2", "\\frac{x^2 + 1}{x - 1}", "\\int x^2 dx"]'
    ),
  }),
  execute: async ({ expressions }) => {
    // Process all expressions in parallel (or detailed batch)
    return expressions.map(expression => {
      try {
        const result = ce.parse(expression).evaluate();
        return {
          expression,
          result: result.toString(),
          latex: result.latex,
        };
      } catch (error) {
        return { 
          expression,
          error: (error as Error).message 
        };
      }
    });
  },
});
```

いくつか注目すべき点がある:

説明文が重要な役割を果たしてる。"MUST be used" という表現は攻撃的に聞こえるかもしれないけど、経験上、モデルにツールをいつ使うか明示的に伝えることが、時々動くか確実に動くかの違いになる。ツールレベルでのプロンプトエンジニアリングだと考えてほしい。

`expressions` 配列によるバッチ処理は、思ってるより重要だ。モデル呼び出しにはレイテンシがある。連立方程式を解いたり多段階の計算をしたりするとき、個別に処理するとユーザー体験が最悪になる。バッチ処理なら、10個の問題を1往復で解ける。

記号的エンジンを使うことで、単なる `eval()`（絶対に使わないでほしい）では得られない本物の数学的理解が得られる。エンジンは意図を解析し、LaTeX書式を処理し、微分や積分も扱える。計算をしてるんじゃなくて、数学をしてるんだ。

エラーハンドリングは式ごとにスコープされている。1つの計算が失敗しても、そのエラーを返しつつ残りは続行する。これでモデルは何がうまくいって何がダメだったかを把握できて、次のステップで自己修正する可能性もある。

## 実際に動かしてみる

普通のモデルなら幻覚を起こしがちなものを投げてみよう:

```typescript
import { anthropic } from '@ai-sdk/anthropic';

const { text } = await generateText({
  model: anthropic('claude-sonnet-4-5'),
  prompt: 'Calculate 18472 × 9347, divide by 127, then take the square root of the result.',
  tools: { mathTool },
  stopWhen: stepCountIs(5), // Allow up to five model/tool steps
});

console.log(text);
```

モデルは計算を見て、精度が必要だと認識し、ツールを呼び出して正確な結果を得て、それを自然言語で説明する。それぞれのコンポーネントが得意なことをやってるだけだ。

## 基本計算を超えて

記号的エンジンを使ってるので、単純な電卓ツールでは対応できないことも処理できる。

代数方程式を解きたい？　「3x + 7 = 22 と 2y - 5 = 13 を解いて」も問題なくいける。

微積分が必要？　「x^3 + 2x^2 の微分を求めて、x = 2 で評価して」もただのツール呼び出しだ。

LaTeX対応は、教育アプリを作るときに特に便利だ。エンジンはLaTeX入力を本質的に理解して、レンダリング用に書式化された結果を返せる。追加のパースは不要だ。

## もっと大きな視点で

このパターンは数学だけじゃなくて重要だと思う。やってるのは、LLMの限界を認識しつつ、その強みを活かすことだ。意図を理解したり、自然言語を解析したり、ワークフローをオーケストレーションしたりするのは驚異的に得意。でも電卓でもデータベースでもファイルシステムでもない。

LLMに決定論的なことをさせようとするたびに、その性質と戦ってることになる。でも自然言語理解を、決定論的な部分を扱う専門ツールと組み合わせたら？　そこからが面白くなる。

数学ツールは一例に過ぎない。同じ原則が日付操作、財務計算、画像処理、データベースクエリ……精度が創造性より重要なあらゆる場所に適用できる。モデルにユーザーが何を望んでるか理解させて、実際の作業はその仕事のために作られたものに任せるんだ。

AIを使って構築する考え方の転換だ。「モデルはこれができるか？」じゃなくて「モデルはこれをオーケストレーションできるか？」と問う。言い回しの小さな違いだけど、信頼性には大きな違いが出る。
