言語モデルについて奇妙だと思うのは何だろう？量子力学を説明し、詩を書き、TypeScriptのデバッグまでできるのに、18472×9347を掛け算させてみると、数千単位でズレた答えを自信満々で返してくることが結構ある。

これが僕を困惑させていた。言語モデルに実際に何をさせているのかを理解するまでは。「バランス」という概念を理解しているからといって、体操選手に家計簿の計算を頼むようなものだ。

実際、LLMは何も計算していない。GPTやClaudeに2+2は何かと聞いても、足し算をしているわけではない。「2 + 2 = 」の後に最も出現しやすいトークンが「4」だと予測しているだけだ。単純な算数ならたいてい上手くいく。これらのパターンが訓練データに含まれているからだ。しかし多段階の計算や、訓練データに頻繁に出てこなかった数字になると、もはやサイコロを振っているようなものだ。

最近、トップクラスのモデルを使って住宅ローンの月々返済額を計算するコードをレビューしていて、これに直面した。モデルは自信満々に答えてきた。しかし月400ドルも間違っていた。これは無視できないレベルの誤差だ。

モデルが推論能力を向上させているとしても（GPT-5は改善が見られると言われている）、それでも高度なパターンマッチングを行っているだけで、記号的計算をしているわけではない。創造的な作業や自然言語処理において、この確率的な性質こそがLLMを魔法のようにしている。しかし数学についてはどうか？そう簡単にはいかない。

## 実際に解決する方法

答えは、より賢いモデルを待つことではない。モデルに適切なツールを与えることだ。

AIを使わないシステムでこの問題をどう解決するか考えてみよう。カスタムの数学ロジックを書くのではなく、ライブラリを使うはずだ。ここで適用される原則は同じだ。ただし今回は、LLMにそのライブラリをいつ・どう使うかを教えることになる。

モダンなAI SDKのツール呼び出し機能を使えば、モデルに構造化された関数を持たせて呼び出させることができる。LLMに数学のふりをさせるのではなく、実際に計算できるものを与える。記号的数学エンジンだ。

僕は[AI SDK v5およびv6](https://ai-sdk.vercel.ai/)とCortexJS Compute Engineの組み合わせを使っている。SDKがオーケストレーションとツールルーティングを担当し、CortexJSが基本的な算数から微積分まで何でも処理する。驚くほどクリーンな責務分離だ。

```bash
bun add ai @ai-sdk/anthropic @cortex-js/compute-engine zod
```

## 数学ツールの構築

実装は思ったよりシンプルだ。LLMの自然言語理解と実際の数学計算の間に橋渡しを作る。

```typescript
import { generateText, stepCountIs, tool } from 'ai';
import { ComputeEngine } from '@cortex-js/compute-engine';
import { z } from 'zod';

// Initialize the engine once
const ce = new ComputeEngine();

const mathTool = tool({
  description: 'Evaluate mathematical expressions and solve equations with guaranteed accuracy. MUST be used for all mathematical operations to verify correctness - do not attempt mental math. Supports arithmetic, algebra, calculus, and complex operations. Can process multiple expressions at once.',
  parameters: z.object({
    expressions: z.array(z.string()).describe(
      'Array of mathematical expressions in LaTeX or plain notation, e.g. ["2 + 2", "\\frac{x^2 + 1}{x - 1}", "\\int x^2 dx"]'
    ),
  }),
  execute: async ({ expressions }) => {
    // Process all expressions in parallel (or detailed batch)
    return expressions.map(expression => {
      try {
        const result = ce.parse(expression).evaluate();
        return {
          expression,
          result: result.toString(),
          latex: result.latex,
        };
      } catch (error) {
        return { 
          expression,
          error: (error as Error).message 
        };
      }
    });
  },
});
```

いくつか注目すべき点がある。

説明（description）が重要な役割を果たしている。「MUST be used」という表現は攻撃的に見えるかもしれないが、経験上、ツールをいつ使うかをモデルに明示的に伝えることが、たまに動く状態と確実に動く状態の違いになる。ツールレベルでのプロンプトエンジニアリングだと考えてほしい。

`expressions`配列によるバッチ処理も、思った以上に重要だ。モデル呼び出しにはそれぞれレイテンシがかかる。連立方程式を解いたり多段階の計算をしたりする場合、個別に処理するとユーザー体験がひどいことになる。バッチ処理なら、1回のラウンドトリップで10問解ける。

`eval()`を使うのではなく記号的エンジンを使うこと（どうか`eval()`は使わないでほしい）により、本当の数学的理解が得られる。エンジンは意図を解析し、LaTeXフォーマットを処理し、微分や積分も扱える。単なる計算ではなく、数学そのものを扱っている。

エラー処理は式ごとにスコープされている。1つの計算が失敗しても、そのエラーを返しつつ残りの処理は続行される。これにより、モデルは何が成功して何が失敗したかを確認でき、次のステップで自己修正できる可能性がある。

## 実際に使ってみる

生のモデルならたいていハルシネーションを起こすようなものを投げてみよう。

```typescript
import { anthropic } from '@ai-sdk/anthropic';

const { text } = await generateText({
  model: anthropic('claude-sonnet-4-5'),
  prompt: 'Calculate 18472 × 9347, divide by 127, then take the square root of the result.',
  tools: { mathTool },
  stopWhen: stepCountIs(5), // Allow up to five model/tool steps
});

console.log(text);
```

モデルは数学の問題を見て、精度が必要だと認識し、ツールを呼び出し、正確な結果を取得し、それを自然言語で説明する。各コンポーネントが、それぞれ最も得意なことをやっている。

## 基本的な算数を超えて

記号的エンジンを使っているので、このアプローチは単純な計算ツールでは扱えないことも処理できる。

代数方程式を解きたい？「次の方程式を解いて: 3x + 7 = 22 および 2y - 5 = 13」は問題なく動作する。

微積分が必要？「x^3 + 2x^2の導関数を求め、x = 2で評価して」も、別のツール呼び出しにすぎない。

LaTeXのサポートは、教育アプリを構築する場合に特に有用だ。エンジンはLaTeX入力を本質的に理解しており、レンダリング用にフォーマットされた結果を返すことができる。追加の解析は不要だ。

## より大きな視点

このパターンが重要だと思うのは、数学だけにとどまらない。本当のところ、LLMの限界を認めつつ、その強みを活かしているのだ。LLMは意図の理解、自然言語の解析、ワークフローのオーケストレーションに優れている。しかし電卓でもデータベースでもファイルシステムでもない。

LLMに決定的な処理をさせようとすると、その性質と戦うことになる。しかし自然言語理解を、決定的な部分を処理する専用ツールと組み合わせるとどうなるか？そこで面白くなる。

数学ツールはその一例にすぎない。同じ原則が日付操作、財務計算、画像処理、データベースクエリにも適用される。精度が創造性よりも重要な場面ならどこでも。モデルにユーザーの意図を理解させ、実際の作業はそのために作られたものに任せる。

AIを使った構築の考え方におけるシフトだ。「モデルにできるか？」ではなく「モデルにオーケストレーションできるか？」表現の小さな違いが、信頼性に大きな違いを生む。
