## ⚠️ Reti locali a rischio

Siamo onesti, lo abbiamo fatto tutti. Ti sei connesso al Wi-Fi di un bar a caso o hai lasciato che qualcuno usasse la tua rete di casa senza pensarci due volte. Forse ti fidi persino del tuo frigorifero smart perché non comprometta la tua rete. La realtà? Queste decisioni casuali possono esporre il tuo ambiente di sviluppo locale a rischi inutili. Gli attaccanti non prendono di mira solo i sistemi di produzione: gli ambienti locali sono spesso bersagli più morbidi, che offrono un modo per accedere a progetti sensibili.

### Scenari di attacco

1. **Traffico intercettato:** Il traffico non crittografato può essere facilmente catturato e letto.
2. **Servizi non protetti:** Database locali o API esposti su `0.0.0.0`.
3. **Spoofing di rete:** Reindirizza il traffico verso il dispositivo di un attaccante.

### Soluzioni rapide

- Preferisci reti Docker private anziché firewall per limitare l'esposizione della rete.
- Evita reti Wi-Fi pubbliche o condivise; preferisci usare l'hotspot del tuo telefono.
- Monitora la tua rete locale per dispositivi sconosciuti usando strumenti come `arp-scan` e `nmap`.

## 🛡️ Configurazione del firewall

### UFW con Docker (Ubuntu)

> ⚠️ **Attenzione:** Di default Docker su Ubuntu/Debian ignora le regole UFW/iptables, esponendo potenzialmente il tuo sistema ad attacchi.
> Non importa se associ le porte a indirizzi IP locali (es. `-p 127.0.0.1:8080:80`).

Questa cosa mi sorprende ogni volta che la scopro! [Docker ignora le regole UFW di default](https://github.com/moby/moby/issues/4737), permettendo ai container di comunicare con l'host e con altri container senza restrizioni.

### Buona pratica

1. 🥇 **Usa reti Docker** per isolare e controllare cosa può connettersi a ciascun container o rete.

###
2. 🥉 **Aggiorna iptables** se devi usare una rete `host` o non puoi usare reti personalizzate, puoi mitigare il rischio configurando iptables. Non per i deboli di cuore, [dai un'occhiata all'utility qui sotto.](#uf)

#### Isolamento della rete Docker

```bash
# Crea una nuova rete Docker
docker network create my-network

# Esegui il container con la nuova rete
docker run --network my-network my-container
```

#### Configurazione UFW (per reti `host`)

Ci sono molti consigli sbagliati su come risolvere questo problema. Configurare UFW per funzionare con Docker usando UFW in modo simile a quello che ci si aspetterebbe.

Ho usato `ufw-docker` per configurare un sistema self-hosted e sembra funzionare bene.

```bash title="install-ufw-docker.sh"
# Installa il binario come root (richiede comunque permessi di root)
sudo wget -O /usr/local/bin/ufw-docker \
   https://github.com/chaifeng/ufw-docker/raw/master/ufw-docker
sudo chmod +x /usr/local/bin/ufw-docker
# Installa e modifica il file `after.rules` di `ufw`
ufw-docker install

ufw-docker help

```

Questo comando esegue le seguenti operazioni:

- Esegue il backup del file `/etc/ufw/after.rules`.
- Aggiunge le regole relative a Docker alla fine del file per integrarsi correttamente con UFW.

**Fonte:** [ufw-docker GitHub](https://github.com/chaifeng/ufw-docker/tree/master#install)

**Esempio di utilizzo:**

```bash

# Consenti al container Docker sulla porta 8080
ufw-docker allow <nome_container> 8080/tcp

# Gestisci le regole in sicurezza insieme alla configurazione UFW
ufw-docker status

```

**Nota:** La maggior parte delle "soluzioni" per i conflitti Docker-UFW coinvolgono regole iptables manuali, che possono essere soggette a errori e fragili durante gli aggiornamenti.

### Firewall macOS

1. Vai su **Preferenze di Sistema > Sicurezza e Privacy > Firewall**.
2. Attiva il firewall e fai clic su "Opzioni Firewall".
3. Blocca tutte le connessioni in entrata tranne i servizi essenziali.

**Nota:** Potrebbe essere necessario consultare la configurazione del firewall per consentire alcuni dispositivi smart che utilizzi, ad esempio Google Cast/AirPlay e altri servizi.

### Comandi per utenti avanzati (macOS e Linux)

#### macOS:

```bash

sudo /usr/libexec/ApplicationFirewall/socketfilterfw --setblockall on  # Blocca tutto
sudo /usr/libexec/ApplicationFirewall/socketfilterfw --add /percorso/app  # Consenti app specifica

```

#### Linux (ufw):

```bash

ufw default deny incoming  # Blocca tutto in entrata
ufw allow ssh  # Consenti SSH
# Consenti 443 e 80 per traffico web
ufw allow 80/tcp
ufw allow 443/tcp
ufw enable  # Attiva firewall

```

**Consiglio:** Utilizza strumenti come [Little Snitch](https://www.obdev.at/products/littlesnitch/index.html) su macOS e [ufw](https://help.ubuntu.com/community/UFW) su Linux per configurazioni più intuitive.

## 🔐 Gestione dei segreti per lo sviluppo locale

### Validazione proattiva dei placeholder

<p>💡 Assicurati che i segreti siano configurati correttamente con valori reali prima di eseguire l'applicazione.</p>

Se usi placeholder come `__WARNING_REPLACE_ME__` nei tuoi segreti, bene, forse qualcuno se ne accorgerà. Per sicurezza, puoi anche aggiungere una piccola validazione per garantire la sicurezza a runtime.

Non crederesti a quanto sia facile hackerare completamente (modificare e ri-firmare) un token JWT quando gli aggressori possono indovinare il segreto!

<CodeTabs client:load tabs={["JavaScript", "Rust", "Go"]}>

```javascript
// validateSecrets.js
const validateSecrets = () => {
  const unsafePlaceholder = /__WARNING_REPLACE_ME__/;
  const missingSecrets = Object.entries(process.env).filter(
    ([key, value]) => unsafePlaceholder.test(value)
  );

  if (missingSecrets.length) {
    console.error("Unsafe secrets detected:", missingSecrets);
    process.exit(1);
  }
};

validateSecrets();
```

```rust
// validate_secrets.rs
use std::env;

fn validate_secrets() {
    let unsafe_placeholder = "__WARNING_REPLACE_ME__";
    for (key, value) in env::vars() {
        if value.contains(unsafe_placeholder) {
            panic!("Unsafe secret in {}", key);
        }
    }
}

fn main() {
    validate_secrets();
}
```

```go
// validate_secrets.go
package main

import (
	"fmt"
	"os"
	"strings"
)

func validateSecrets() {
	placeholder := "__WARNING_REPLACE_ME__"
	for _, env := range os.Environ() {
		pair := strings.SplitN(env, "=", 2)
		if len(pair) == 2 && strings.Contains(pair[1], placeholder) {
			panic(fmt.Sprintf("Unsafe secret in %s", pair[0]))
		}
	}
}

func main() {
	validateSecrets();
}
```

### Generazione e archiviazione dei segreti

<p class="inset">Non hardcodare mai i segreti nel tuo codice. Preferisci variabili d'ambiente e vault sicuri.</p>


Invece di `.env.example`, usa `.env.generate.sh` per permettere agli utenti di ottenere facilmente un file `.env` con "default" sicuri.

#### Esempio `.env.generate.sh`

```bash title=".env.generate.sh" frame="code"
#!/bin/bash
# Generates a secure .env file for local development

generate_secret() {
    local length=${1:-30}
    # add 4 bytes to account for padding
    local generate_length=$((length + 4))
    openssl rand -base64 "$generate_length" | tr -d '+=/\n' | cut -c1-"$length"
}
# Bail out if .env file already exists
[ -f .env ] && { echo ".env file already exists!"; exit 1; }

cat <<EOL > .env
# Database settings & secrets
DB_USER=app_user
DB_PASSWORD=$(generate_secret 30)
REDIS_PASSWORD=$(generate_secret 20)
# Session secrets
SESSION_KEY=$(generate_secret 32)
JWT_SECRET=$(generate_secret 64)
EOL

echo "New .env file generated!"
```

## 🕵️‍ Monitoraggio e doppia verifica

### Esempi con `nmap`

#### Test all'interno della tua rete

```bash

# Scan your localhost for all open ports
nmap -sT localhost

# Scan your machine’s private IP for services
nmap -sV 192.168.1.10

# Detect devices on your network
nmap -sn 192.168.0.0/24
nmap -sn 10.0.0.0/24

```

#### Test al di fuori della tua rete

Puoi ottenere il tuo IP pubblico corrente facilmente con servizi come `ifconfig.me`: `curl https://ifconfig.me`.

Usa una rete esterna o un server remoto per testare i tuoi IP pubblici:

```bash

print_current_ip() {
  curl https://ifconfig.me
}

print_current_ip
# --> 123.456.789.012

# Cambia target_host con il tuo IP pubblico o hostname
# Controlla l'host usando tecniche avanzate
nmap -A --open --reason $target_host
nmap -A -F --open --reason $target_host
nmap -A -p1-65535 --open --reason $target_host

```

**Perché testare entrambi?**
I test dall'interno rivelano l'esposizione interna, mentre i test esterni identificano i servizi accessibili agli aggressori.

```
