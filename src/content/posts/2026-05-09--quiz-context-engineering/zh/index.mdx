    一个合理的经验法则：先确定期望的输出大小，然后确保输入 comfortably 低于剩余预算。对多数 RAG 任务而言，这意味着 **输入占总上下文窗口的 60–70%**（系统提示 + 对话历史 + 检索到的上下文）不宜超过。将其余部分留给生成并预留安全余量。

    此外，模型在接近上下文窗口边缘时表现会变差——理解力和指令遵循都会随上下文填满而下降。虽然技术上可以运行到 95%，但体验远不如在 50% 左右时的表现。

  <slot name='explanation'>
  <div className="explanation">
    LLM 并没有持久记忆。每次 API 调用都是无状态的——你发送完整上下文，得到响应。对话的“记忆”完全取决于你在每个请求中包含的消息历史。

    当历史超出预算时，可选方案有：
    1. **Summarize**：将较早的回合压缩成持续更新的摘要，保留最近的回合原文

    为什么多样性重要：如果所有示例都是同一种输入，模型在边缘情况上的泛化能力会很差。请包含覆盖最重要变体的示例。

  <div className="explanation">

    真正有效的办法是：**基于事实的定位**。给模型提供它正确回答所需的信息，并将其约束在这些信息范围内：
    ```

        If the answer isn't in the documents, say: "I don't have enough information to answer that."
    ```
    随后验证输出：检查响应中的断言是否出现在检索到的上下文中。这就是引用定位检查——实现细节请参阅 RAG 评估讨论。

    更大的模型平均幻觉更少，但所有模型都会幻觉。定位是缓解策略，而非模型规模。
  </div>
  </slot>

    **微调** 在你的数据上更新模型权重。更改是永久的（针对该检查点），并作用于每一次推理。

    微调真正更好的情况包括：
    - 需要模型保持一致的风格/格式，而仅靠指令无法可靠实现
    - 任务需要在特定领域模式上重复行为，提示和检索无法解决
    - 需要缩短提示长度——微调后的行为无需在每次请求中解释
    - 处理大量请求时，一致的少量示例会消耗大量 token

    微调显得多余的情况有：
    - 指令可以放入系统提示中
    - 主要需要当前或专有的事实，可在请求时检索
    - 需求经常变化（需要重新微调）
    - 还未先尝试尽可能的上下文工程

    正确的顺序：先把上下文工程做到位。只有在确实不足时才进行微调。
