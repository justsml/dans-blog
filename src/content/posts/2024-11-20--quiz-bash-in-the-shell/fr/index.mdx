    Note : `$name` est utilisé pour **référencer** ou lire la valeur d'une variable.

<Challenge
  client:visible={{rootMargin: "150px"}}

  options={[

  <slot name="question">
  <div className="question">
    Que se passe-t-il ici ?
    ```bash
        str="Bark bark"
        echo ${str/bark/meow}
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La syntaxe `${var/pattern/replacement}` remplace la première occurrence de `pattern` par `replacement`. Ici, le résultat est :
    ```plaintext
        Bark meow
    ```
    Il est sensible à la casse. Pour gérer à la fois `bark` et `Bark`, utilisez un motif comme `${var/[Bb]ark/Bark}` ou normalisez la chaîne avant le remplacement.

    Pour remplacer toutes les occurrences, utilisez `${var//pattern/replacement}`.

    Pour remplacer depuis le début de la chaîne, utilisez `${var/#pattern/replacement}`.

    Pour remplacer depuis la fin de la chaîne, utilisez `${var/%pattern/replacement}`.
  </div>
  </slot>

    Que produit ce script si le fichier `cats.txt` EXISTE ?
    ```bash
        if [ -e cats.txt]; then
          echo "File exists"
        else
          echo "File does not exist"
        fi
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Avez-vous remarqué l'espace manquant avant le crochet fermant ?

    Bash est très pointilleux ici : les espaces sont obligatoires à l'intérieur des expressions entre crochets.

    Parce que l'espace manquant fait que la commande `[` ne voit pas de `]` fermant, Bash affiche un diagnostic, considère le test comme échoué et passe à la branche `else`.

    La syntaxe correcte est :
    ```bash
        if [ -e example.txt ]; then
          echo "File exists"
        else
          echo "File does not exist"
        fi
    ```
    Remarque : Les doubles crochets `[[ ]]` sont **recommandés** pour les expressions conditionnelles. [Voir BashFAQ.](https://mywiki.wooledge.org/BashFAQ/031)
  </div>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={14}
  group="Pièges"
  title="Substitution de commande"
  options={[
    {text: '\'ls -l\''},
    {text: '% ls -l'},
    {text: '$ ls -l'},
    {text: '$(ls -l)', isAnswer: true},
    {text: '${ls -l}'},
  ]}
>
  <slot name="question">
  <div className="question">
    Quelle option exécutera la commande `ls -l` et retournera la sortie ?
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La syntaxe `$(ls -l)` exécute la commande entre **parenthèses** et substitue la sortie. Par exemple :
    ```bash
        echo "Today is $(date +%F)"
        # => Today is 2029-12-31
    ```
    La première option utilise des guillemets simples `'`, **pas des backticks.** Cela empêche l'expansion, donc `'$(date +%F)'` afficherait simplement la chaîne littérale `$(date +%F)`.

    Bien que l'utilisation des backticks (`` `ls -l` ``) pour l'exécution de commandes soit toujours prise en charge, elle est récemment devenue un anti-patron (dans certains contextes). La plupart recommandent d'utiliser `$(command)` pour une meilleure lisibilité et cohérence avec différents shells et versions.

    Les accolades `${}` sont utilisées pour l'expansion de variables, pas pour la substitution de commande.

    Le caractère `%` n'est pas utilisé pour la substitution de commande.
  </div>
  </slot>
