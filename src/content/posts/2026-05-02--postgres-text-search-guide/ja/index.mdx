ほとんどのチームはPostgresの検索ツールを1つだけ使っています。3つ全てを掌握したチームは、よりシンプルな構成でより優れた検索を実現し、まだ必要のない専用検索サービスへのコストのかかる遠回りも避けています。

本ガイドでは、Postgres標準の検索オプション全容をカバーします。各ツールの役割、適切なユースケース、そしてそれらをレイヤリングする方法です。

---

## 3つのツール

**全文検索**（`tsvector` / `GIN`インデックス）は字句解析ベースです。テキストをレлем（語素）に分解し、ステミングを行い、クエリをインデックスと照合します。「running」と「runs」は同じレлемに集約されます。「dog」と「dogs」も同様です。ランキング関数（`ts_rank`）は、クエリ用語が多く出現하거나重要な位置に出現するドキュメントほど高く評価します。

**トリグラム**（`pg_trgm`）は文字列を重複する3文字のスライスに分割し、2つの文字列が共有するスライスの数を測定します。「Dan」→「 da」、「dan」、「an "」。「Micheal」と「Michael」はトリグラムの大部分を共有するため、類似度が高くなります。これにより、`pg_trgm`は模糊な名前検索、タイプ耐性、自动補完に優れています——FTSが不得意とする分野です。

**完全一致インデックス**（Bツリー、ハッシュ）は、主キー、メールドレス、ID、SKUなど、答えがバイナリなものを扱います：一致するかしないか。これらは「検索」とは感じられないかもしれませんが、最も悪いパターンは、正しい答えがある問題を模糊検索やセマンティック検索で解決しようとすることなので、この議論に含まれます。

選択は洗練度についてではありません。ツールをクエリの形状にマッチさせることです。

## 全文検索が有効な場合

**キーワード検索でプロサーを検索。** ブログ記事、ドキュメント、商品説明、支持チケット、法曹文書。FTSはこの形のコンテンツのために設計されました：自然言語テキストに対するインデックス付きランキング検索。

**キーワードベースのユーザークエリ。** ユーザーは検索語を入力し、タグでフィルタリングするか、キーワードでブラウズします。FTSは埋め込みインフラなしでその意図を組み込みます。

**外部依存関係なしのランキング結果。** FTSインデックスは高速で决定的であり、API呼び出しを一切必要としません。関連性シグナルはフィールド位置で重み付けされた用語頻度から得られます。

**検索を伴うブールフィルタリング。** FTSは既存のクエリロジックと自然に構成されます：

```sql
SELECT * FROM posts
WHERE search_vector @@ to_tsquery('english', 'postgres & performance')
  AND category = 'tutorial'
  AND published_at > NOW() - INTERVAL '6 months';
```

### FTSの設定

```sql
-- 生成列でインデックスを常に最新の状態に保つ
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body,  '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- クエリ
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`setweight`で重要性を割り当て：A（タイトル）はB（ボディ）を上回ります。これはほとんどのコンテンツ検索ユースケースにおける関連性モデルの全体です。

### FTSが得意でないこと

- クエリのタイプミス——「javascipt」は「javascript」と一致しない
- 人名、住所、辞書のステミングで予測通りに処理されない固有名詞
- 特別な設定なしではプレフィックス/自動補完
- ユーザーが概念を説明するのではなく名前を付けるクエリ

## トリグラムが有効な場合（`pg_trgm`）

`pg_trgm`はFTSが一貫して苦手とする厄介な中間をカバーします。

FTSはテキストをレлемに分解してステミングします。プロサーにはこれで正解です。しかし名前や短い識別子には往々にしてそうではありません：

- 人名（「Dan Levy」→辞書と言語設定に依存してstemmingが不安定）
- 正確なスペルが重要な会社名、住所、商品タイトル
- タイプミスを含むクエリ——「Micheal Jordan」、「Amaon」、「javascipt」
- 自動補完 / プレフィックス検索
- 部分文字列照合（「son」が「Johnson」、「Anderson」に一致）

`pg_trgm`は言語に依存しない，这也是多様な言語的背景を持つ名前にとって重要です。FTSは言語ごとに辞書設定が必要です。

### 模糊な名前検索

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- 「Michael Jordan」で検索すると「Micheal Jordan」を見つける
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % 演算子 = 類似度閾値（デフォルト 0.3）
ORDER BY score DESC
LIMIT 10;
```

`%` 演算子は `pg_trgm.similarity_threshold`（デフォルト 0.3、範囲 0–1）を使用します。名前検索では 0.3–0.4 がタイプミスをキャッチし、ノイズを低く保ちます。

### 自動補完、プレフィックス、包含検索

```sql
-- 自動補完用のプレフィックス一致。トリグラムGINインデックスが役立つ、
-- しかし純粋な左アンカー付きプレフィックスにはBツリーパターンインデックスがより良い場合がある。
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- word_similarityで長い文字列内の部分一致
--（「Andrew Johnson III」内の「Johnson」）
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name
ORDER BY score DESC
LIMIT 10;
```

トリグラムGINインデックスは `ILIKE '%pattern%'` 包含クエリとタイプ耐性照合 особенно有用——トリグラムインデックスなしでは通常フルテーブルスキャンになります。

### FTSではなくpg_trgmを使うべき場面

| シナリオ | 使用するツール |
|---|---|
| タイプミスを含む人名/会社名検索 | `pg_trgm` |
| 自動補完 / プレフィックス検索 | `pg_trgm`（またはプレフィックスクエリ付きFTS） |
| 短い文字列、識別子、コード | `pg_trgm` |
| プロサー記事、ドキュメント、チケット | FTS |
| キーワード検索のログメッセージ | FTS |
| 多言語名前検索 | `pg_trgm`（言語非依存） |

## 完全一致SQLが有効な場合

一部の「検索」問題はそもそも検索ではありません。

「メール `dan@example.com` のユーザーを探す」は等価チェックです。「注文 `ORD-12345` を探す」は主キー検索です。「`tutorial`カテゴリで日付順にソートされた投稿を一覧表示」はフィルタ付きクエリです。これらはBツリーまたはハッシュインデックスに属します。

ここでFTSやトリグラムを使用すると複雑さが増すだけで正しさは向上しません——そして正確な識別子では、近似一致は一致しないより悪い場合があります。

```sql
CREATE INDEX users_email_idx ON users (email);

-- 完全一致ルックアップ：高速で明確
SELECT id, name FROM users WHERE email = $1;
```

より広い教訓：正しい答えがある問題で近似検索することはカテゴリエラーです。何かを返します—— уверенно間違うかもしれません。

## これらのツールの組み合わせ

これらのツールはクリーンに構成されます。1つだけを選ぶ必要はありません。

**FTS + pg_trgm：キーワードのタイプミスを許容する検索ボックス：**

```sql
-- タイトルのトリグラム類似度がタイプミスをキャッチ；ts_rankがボディの関連性を処理
SELECT id, title,
  ts_rank(search_vector, to_tsquery('simple', $1)) AS fts_rank,
  similarity(title, $1) AS trgm_score
FROM posts
WHERE search_vector @@ to_tsquery('simple', $1)
   OR title % $1
ORDER BY (ts_rank(search_vector, to_tsquery('simple', $1)) + similarity(title, $1)) DESC
LIMIT 10;
```

**FTS + `unaccent` 国際コンテンツ用：**

```sql
-- 発音記号を剥離して「José」が「Jose」に一致するようにする
CREATE EXTENSION IF NOT EXISTS unaccent;

CREATE TEXT SEARCH CONFIGURATION public.simple_unaccent (COPY = pg_catalog.simple);

ALTER TEXT SEARCH CONFIGURATION public.simple_unaccent
  ALTER MAPPING FOR hword, hword_part, word
  WITH unaccent, simple;

ALTER TABLE posts ADD COLUMN search_vector tsvector;

CREATE TRIGGER posts_search_vector_refresh
BEFORE INSERT OR UPDATE OF title, body ON posts
FOR EACH ROW EXECUTE FUNCTION
  tsvector_update_trigger(search_vector, 'public.simple_unaccent', title, body);
```

**`unaccent` + `pg_trgm` 国際名前検索用：**

```sql
ALTER TABLE users ADD COLUMN name_search text;

CREATE FUNCTION users_name_search_refresh()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  NEW.name_search := unaccent(coalesce(NEW.name, ''));
  RETURN NEW;
END;
$$;

CREATE TRIGGER users_name_search_refresh
BEFORE INSERT OR UPDATE OF name ON users
FOR EACH ROW EXECUTE FUNCTION users_name_search_refresh();

CREATE INDEX users_name_search_trgm_idx
  ON users USING GIN (name_search gin_trgm_ops);

SELECT id, name
FROM users
WHERE name_search % unaccent($1)
ORDER BY similarity(name_search, unaccent($1)) DESC
LIMIT 10;
```

トリガーの例では、生成列やインデックス式内で `unaccent()` を使用する際、PostgreSQLの不変性ルールが問題になるため、このような回避策を使用しています。独自の不変関数内で `unaccent()` をラップする場合は、アップグレード/設定リスクを受け入れていることを文書化してください。

## 注目すべき拡張機能

**`pg_trgm`** はほとんどのPostgresディストリビューションにバンドルされていますが、明示的な有効化が必要です。Postgresでの模糊文字列照合の基礎です。

**`unaccent`** はインデックス作成とクエリの前に発音記号を剥離します。`pg_trgm`とFTSの両方に最適なペアリング。European言語コンテンツ向け。Postgresにバンドルされています。

**`pg_bigm`** はトリグラムアプローチをバイグラム（2文字スライス）に拡張します。これによりCJK（中国語、日本語、韓国語）言語で `pg_trgm` が性能を出す場合に結果が大幅に向上します。別途インストールが必要；バンドルされていません。

**`pg_search`**（[ParadeDB](https://www.paradedb.com/)から）は標準の `GIN` / `tsvector` スタックをTantivyベースのBM25インデックスに置き換えます。これによりBM25スコアリング（FTSより優れた場合がある）、FTSクエリ内の模糊照合、ファセット検索、大規模テーブルでの劇的に高速なインデックス作成が得られます。標準FTSがランキングやパフォーマンス限界を示し始めたときのドロップインアップグレードパスです。

```sql
-- pg_search: BM25全文検索と模糊照合
CREATE INDEX posts_bm25_idx ON posts
  USING bm25 (id, title, body)
  WITH (key_field = 'id', text_fields = '{"title": {}, "body": {}}');

-- BM25スコアリング + 模糊照合でのクエリ（「javascipt」をキャッチ）
SELECT id, title, paradedb.score(id) AS rank
FROM posts
WHERE posts @@@ paradedb.fuzzy_phrase(field => 'title', value => 'postgres performnce')
ORDER BY rank DESC
LIMIT 10;
```

**`pgvector`** は密なベクトルストレージと類似性検索を追加します。ユーザーが欲しいものを説明する而不是名前を付けるときの正しいツールです——セマンティック検索、RAG、関連コンテンツ推薦、多言語クエリ。の詳細な解説は [セマンティックベクトル検索とハイブリッド戦略](/semantic-vector-search-landscape) を参照。

## 決定表

| 検索対象 | 推奨 |
|---|---|
| プロサー記事、ドキュメント、チケット | FTS |
| タイプミスを含む人名/会社名 | `pg_trgm` |
| 自動補完、プレフィックス検索 | `pg_trgm` |
| 短いコード、識別子 | `pg_trgm` |
| キーワード検索のログメッセージ | FTS |
| 国際的な名前 | `pg_trgm` + `unaccent` |
| 大規模コンテンツ、より良いランキング | `pg_search`（ParadeDB BM25） |
| 主キー、完全なメール、ID | Bツリーインデックス |
| 日付、範囲、ソート済みリスト | Bツリーインデックス |
| 権限、カテゴリ、フィルタ | 通常のWHERE句 |
| 質問、言い換え、概念 | pgvector（次の記事を参照） |

迷う場合は：スペル変化のある短い文字列 → トリグラム。キーワードクエリの長いプロサー → FTS。構造化識別子 → 通常のインデックス。概念的または自然言語クエリ → pgvector。

## ハイブリッド検索：2つのシグナル、1つのランク

`"withRetry timeout errors"` のようなクエリが検索ボックスに入ると、2種類の意図を持ちます：ユーザーが知っている正確なシンボル名（`withRetry`）と概念的説明（`timeout errors`）。単一のプリミティブでは両方をカバーできません。FTSとベクトル検索を並行して実行し——それからのランク付きリストをReciprocal Rank Fusionでマージ——ことで可能です。

RRFは各結果のスコアをリストごとに `1 / (60 + rank)` で計算し、リスト間で合計します，定数60はトップランクの優位性を減衰させます，所以第二位连续获胜的结果可以击败在单个列表中获胜但在另一个列表中完全没有出现的结果。重要なことに、RRF決して raw score を跨いで平均化しません——FTS rankとコサイン距離は異なる通貨であり、算術的に組み合わせることができません。

<figcaption>クエリがFTSとpgvectorに並行して分散。各々は独自のランク付きリストを生成。RRFは各ドキュメントを各リストでの位置でスコアリングしスコアを合計——结果是兩個シグナルが一致するドキュメントが表面に現れます。</figcaption>

```sql
-- ハイブリッド検索：FTS + pgvector をRRFでマージ
WITH fts AS (
  SELECT id, ts_rank(search_vector, query) AS score,
         ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM docs, to_tsquery('english', 'withRetry & timeout') query
  WHERE search_vector @@ query
  LIMIT 60
),
vec AS (
  SELECT id,
         ROW_NUMBER() OVER (ORDER BY embedding <=> $embedding) AS rank
  FROM docs
  ORDER BY embedding <=> $embedding
  LIMIT 60
)
SELECT COALESCE(fts.id, vec.id) AS id,
       (COALESCE(1.0 / (60 + fts.rank), 0) +
        COALESCE(1.0 / (60 + vec.rank), 0)) AS rrf_score
FROM fts FULL JOIN vec ON fts.id = vec.id
ORDER BY rrf_score DESC
LIMIT 10;
```

各ブランチの60ドキュメント候補プール（`LIMIT 60`）は一般的な開始点です。召回率が低い場合は広げる；速度の場合は狭めます。
