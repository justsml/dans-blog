ほとんどのチームは Postgres の検索ツールを一つしか使わない。三つとも知っているチームは、より少ない複雑さでより良い検索を提供できる — そして、まだ必要のない専用検索サービスへの高価な寄り道を避けられる。

このガイドでは、Postgres ネイティブの選択肢をすべてカバーする：それぞれが何をするものか、いつ適合するのか、そしてどう組み合わせるのか。

---

## 3つのツール

**フルテキスト検索**（`tsvector` / `GIN` インデックス）は語彙的である。テキストを字句（レンマ）にトークン化し、語幹を抽出し、クエリをインデックスと照合する。「Running」と「runs」は同じ字句に変換される。「dog」と「dogs」も同様。ランキング関数（`ts_rank`）は、クエリ語が頻繁に、または目立つ場所に出現する文書を高く評価する。

**トライグラム**（`pg_trgm`）は文字列を重なり合う3文字のスライスに分解し、2つの文字列がいくつのスライスを共有しているかを測る。「Dan」→ `" da"`, `"dan"`, `"an "`。「Micheal」と「Michael」はトライグラムの大部分を共有するため、類似度は高くなる。これにより、`pg_trgm` はファジーな名前マッチング、タイプ許容、オートコンプリート — まさに FTS が苦手とする領域 — で優れた性能を発揮する。

**完全一致インデックス**（B-tree、ハッシュ）は主キー、メールアドレス、ID、SKU、答えが「一致するかしないか」の二値であるすべてのものを扱う。これらは「検索」には見えないかもしれないが、正解がある問題にファジー検索やセマンティック検索を使うのが最悪のパターンであるため、この議論に含めるべきだ。

選択は洗練さについてではない。クエリの形状にツールを合わせることだ。

## フルテキスト検索が勝つ場合

**キーワードで文章を検索する場合。** ブログ記事、ドキュメント、商品説明、サポートチケット、法的文書。FTS はこのコンテンツ形状のために設計された：自然言語テキストに対するインデックス付き・ランク付き検索。

**キーワードベースのユー�ークエリ。** ユーザーが検索語を入力し、タグでフィルタリングし、キーワードで閲覧する。FTS はこの意図を埋め込みインフラなしにネイティブに処理する。

**外部依存のないランキング付き結果。** FTS インデックスは高速で決定的で、APIコールを必要としない。関連性シグナルは、フィールド位置で重み付けされた語頻度から来る。

**検索と並ぶブールフィルタリング。** FTS は既存のクエリロジックと自然に組み合わさる：

```sql
SELECT * FROM posts
WHERE search_vector @@ to_tsquery('english', 'postgres & performance')
  AND category = 'tutorial'
  AND published_at > NOW() - INTERVAL '6 months';
```

### FTS のセットアップ

```sql
-- 生成列でインデックスを自動的に最新に保つ
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body,  '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- クエリ
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`setweight` は重要度を割り当てる：`A`（タイトル）は `B`（本文）より上位。これがほとんどのコンテンツ検索ユースケースにおける関連性モデルのすべてだ。

### FTS が苦手なこと

- クエリのタイプミス — 「javascipt」は「javascript」にマッチしない
- 人名、住所、語幹抽出が予測不可能な固有名詞
- 特別な設定なしのプレフィックス/オートコンプリート
- ユーザーが概念を説明し、名前を挙げないクエリ

## トライグラムが勝つ場合（`pg_trgm`）

`pg_trgm` は FTS が一貫して苦手とする微妙な領域をカバーする。

FTS はテキストを字句にトークン化し語幹を抽出する。文章に対しては正しい。名前や短い識別子に対してはそうではない：

- 人名（「Dan Levy」→ 辞書と言語設定によって語幹が異なる）
- 正確なスペルが重要な企業名、住所、商品タイトル
- タイポのあるクエリ — 「Micheal Jordan」「Amaon」「javascipt」
- オートコンプリート / プレフィックス検索
- 部分文字列マッチング（「son」が「Johnson」「Anderson」にマッチ）

`pg_trgm` は言語非依存でもある。これは多様な言語的背景を持つ名前にとって重要だ。FTS は言語ごとに辞書設定が必要である。

### ファジー名前検索

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- "Michael Jordan" で検索して "Micheal Jordan" を見つける
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % 演算子 = 類似度しきい値（デフォルト 0.3）
ORDER BY score DESC
LIMIT 10;
```

`%` 演算子は `pg_trgm.similarity_threshold`（デフォルト 0.3、範囲 0–1）を使用する。名前検索では、0.3–0.4 がタイプミスを捕捉しつつノイズを低く抑える。

### オートコンプリート、プレフィックス、コンテインズ検索

```sql
-- オートコンプリートのプレフィックスマッチング。
-- トライグラム GIN インデックスは役立つが、
-- 純粋な左固定プレフィックスには B-tree パターンインデックスの方が良い場合がある。
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- より長い文字列内の部分マッチに対する word_similarity
-- （"Andrew Johnson III" の中の "Johnson"）
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name
ORDER BY score DESC
LIMIT 10;
```

トライグラム GIN インデックスは、`ILIKE '%pattern%'` コンテインズクエリとタイプミス許容マッチングに特に有用 — トライグラムインデックスなしでは通常フルテーブルスキャンになるパターンだ。

### FTS より pg_trgm に頼るべき場面

| シナリオ | 使用ツール |
|---|---|
| タイポを含む人名/企業名検索 | `pg_trgm` |
| オートコンプリート / プレフィックス検索 | `pg_trgm`（またはプレフィックスクエリ付きFTS） |
| 短い文字列、識別子、コード | `pg_trgm` |
| 文章の記事、ドキュメント、チケット | FTS |
| キーワードによるログメッセージ | FTS |
| 多言語名前検索 | `pg_trgm`（言語非依存） |

## 完全一致 SQL が勝つ場合

「検索」問題の中には、そもそも検索ではないものがある。

「メール `dan@example.com` のユーザーを見つける」は等値チェックだ。「注文 `ORD-12345` を見つける」は主キールックアップだ。「`tutorial` カテゴリの投稿を日付順に一覧する」はフィルタ付きクエリだ。これらは B-tree またはハッシュインデックスに属する。

ここで FTS やトライグラムを使うと、正確性を向上させずに複雑さが増す — そして正確な識別子に対して、近いマッチはマッチなしより悪い。

```sql
CREATE INDEX users_email_idx ON users (email);

-- 完全ルックアップ：高速で明確
SELECT id, name FROM users WHERE email = $1;
```

より広い教訓：正解のある問題に近似検索を使うのはカテゴリエラーだ。それは*何か*を返す — しかし自信満々に間違っている可能性がある。

## ツールの組み合わせ

これらのツールはきれいに組み合わさる。ちょうど一つを選ぶ必要はない。

**キーワードのタイプミスを許容する検索ボックスに FTS + pg_trgm：**

```sql
-- タイトルのトライグラム類似度がタイプミスを捕捉；ts_rank が本文の関連性を処理
SELECT id, title,
  ts_rank(search_vector, to_tsquery('simple', $1)) AS fts_rank,
  similarity(title, $1) AS trgm_score
FROM posts
WHERE search_vector @@ to_tsquery('simple', $1)
   OR title % $1
ORDER BY (ts_rank(search_vector, to_tsquery('simple', $1)) + similarity(title, $1)) DESC
LIMIT 10;
```

**国際コンテンツに FTS + `unaccent`：**

```sql
-- ダイアクリティカルマークを除去して "José" が "Jose" にマッチするようにする
CREATE EXTENSION IF NOT EXISTS unaccent;

CREATE TEXT SEARCH CONFIGURATION public.simple_unaccent (COPY = pg_catalog.simple);

ALTER TEXT SEARCH CONFIGURATION public.simple_unaccent
  ALTER MAPPING FOR hword, hword_part, word
  WITH unaccent, simple;

ALTER TABLE posts ADD COLUMN search_vector tsvector;

CREATE TRIGGER posts_search_vector_refresh
BEFORE INSERT OR UPDATE OF title, body ON posts
FOR EACH ROW EXECUTE FUNCTION
  tsvector_update_trigger(search_vector, 'public.simple_unaccent', title, body);
```

**国際名前検索に `unaccent` + `pg_trgm`：**

```sql
ALTER TABLE users ADD COLUMN name_search text;

CREATE FUNCTION users_name_search_refresh()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  NEW.name_search := unaccent(coalesce(NEW.name, ''));
  RETURN NEW;
END;
$$;

CREATE TRIGGER users_name_search_refresh
BEFORE INSERT OR UPDATE OF name ON users
FOR EACH ROW EXECUTE FUNCTION users_name_search_refresh();

CREATE INDEX users_name_search_trgm_idx
  ON users USING GIN (name_search gin_trgm_ops);

SELECT id, name
FROM users
WHERE name_search % unaccent($1)
ORDER BY similarity(name_search, unaccent($1)) DESC
LIMIT 10;
```

トリガーの例は、`unaccent()` を生成列やインデックス式の中で使うことを避けている。そこでは PostgreSQL の不変性ルールが重要になるからだ。`unaccent()` を独自の不変関数でラップする場合、アップグレードや設定のリスクを受け入れていることを文書化すること。

## 注目の拡張機能

**`pg_trgm`** はほとんどの Postgres ディストリビューションにバンドルされているが、明示的な有効化が必要。Postgres におけるファジー文字列マッチングの基盤。

**`unaccent`** はインデックスとクエリの前にダイアクリティカルマークを除去する。`pg_trgm` と FTS の両方とヨーロッパ言語コンテンツでうまく組み合わさる。Postgres にバンドルされている。

**`pg_bigm`** はトライグラムアプローチをバイグラム（2文字スライス）に拡張する。`pg_trgm` が苦手な CJK（中国語、日本語、韓国語）で大幅に結果が向上する。個別インストールが必要。バンドルされていない。

**`pg_search`**（[ParadeDB](https://www.paradedb.com/) 製）は標準の `GIN` / `tsvector` スタックを Tantivy ベースの BM25 インデックスに置き換える。BM25 スコアリング（`ts_rank` より優れることが多い）、FTS クエリ内のファジーマッチング、ファセット検索、大規模テーブルでの劇的に高速なインデックス構築を提供する。標準の FTS がランキングやパフォーマンスの限界に達したときのドロップインアップグレードパス。

```sql
-- pg_search: ファジーマッチング付き BM25 フルテキスト検索
CREATE INDEX posts_bm25_idx ON posts
  USING bm25 (id, title, body)
  WITH (key_field = 'id', text_fields = '{"title": {}, "body": {}}');

-- BM25 スコアリング + ファジーマッチングでクエリ（"javascipt" を捕捉）
SELECT id, title, paradedb.score(id) AS rank
FROM posts
WHERE posts @@@ paradedb.fuzzy_phrase(field => 'title', value => 'postgres performnce')
ORDER BY rank DESC
LIMIT 10;
```

**`pgvector`** は密ベクトルの保存と類似性検索を追加する。ユーザーが名前を挙げるのではなく欲しいものを説明する場合 — セマンティック検索、RAG、関連コンテンツのレコメンデーション、多言語クエリ — に適したツールだ。[Semantic Vector Search and Hybrid Strategies](/semantic-vector-search-landscape) で深く扱っている。

## 意思決定テーブル

| 検索対象 | 推奨ツール |
|---|---|
| 文章の記事、ドキュメント、チケット | FTS |
| タイポを含む人名/企業名 | `pg_trgm` |
| オートコンプリート、プレフィックス検索 | `pg_trgm` |
| 短いコード、識別子 | `pg_trgm` |
| キーワードによるログメッセージ | FTS |
| 国際的な名前 | `pg_trgm` + `unaccent` |
| 大規模コンテンツ、より良いランキング | `pg_search`（ParadeDB BM25） |
| 主キー、正確なメール、ID | B-tree インデックス |
| 日付、範囲、ソート済みリスト | B-tree インデックス |
| パーミッション、カテゴリ、フィルタ | 通常の WHERE 句 |
| 質問、パラフレーズ、概念 | pgvector（次の記事を参照） |

迷ったら：スペルのばらつきがある短い文字列 → トライグラム。キーワードクエリ向けの長い文章 → FTS。構造化識別子 → 通常のインデックス。概念的・自然言語クエリ → pgvector。

## ハイブリッド検索：2つのシグナル、1つのランク

`"withRetry timeout errors"` のようなクエリが検索ボックスに入力されたとき、2種類の意図を含んでいる：ユーザーが知っている正確なシンボル名（`withRetry`）と、概念的な説明（`timeout errors`）。単一のプリミティブでは両方をカバーできない。FTS とベクトル検索を並列に実行し — その後 Reciprocal Rank Fusion でランク付きリストをマージする — ことで対応できる。

RRF は各リストの各結果を `1 / (60 + rank)` としてスコアリングし、リスト間で合計する。定数 60 は上位ランクの有利さを緩和するため、両方のリストで2位の結果が、片方のリストで1位でもう片方に不在の結果を上回れる。重要なのは、RRF は決して生スコアをメソッド間で平均しない — FTS ランクとコサイン距離は異なる通貨であり、算術的に組み合わせることはできない。

```sql
-- ハイブリッド検索：FTS + pgvector を RRF で統合
WITH fts AS (
  SELECT id, ts_rank(search_vector, query) AS score,
         ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM docs, to_tsquery('english', 'withRetry & timeout') query
  WHERE search_vector @@ query
  LIMIT 60
),
vec AS (
  SELECT id,
         ROW_NUMBER() OVER (ORDER BY embedding <=> $embedding) AS rank
  FROM docs
  ORDER BY embedding <=> $embedding
  LIMIT 60
)
SELECT COALESCE(fts.id, vec.id) AS id,
       (COALESCE(1.0 / (60 + fts.rank), 0) +
        COALESCE(1.0 / (60 + vec.rank), 0)) AS rrf_score
FROM fts FULL JOIN vec ON fts.id = vec.id
ORDER BY rrf_score DESC
LIMIT 10;
```

ブランチごとの60文書の候補プール（`LIMIT 60`）は一般的な出発点。再現率が低い場合は広げ、速度が必要な場合は狭める。

## 次に読むべき記事

Postgres テキスト検索は多くをカバーするが、天井がある。ユーザーが名前を挙げるのではなく欲しいものを説明するとき — 「フライトで眠るのに役立つ何か」「新人エンジニアのデバッグ信頼性に関する記事」 — 語彙的検索もトライグラム検索も失敗する。

それがベクトル埋め込み、セマンティック検索、ハイブリッドアーキテクチャの領域だ。[Semantic Vector Search and Hybrid Strategies](/semantic-vector-search-landscape) で詳しく扱っている。
