ほとんどのチームは、Postgresの検索ツールを1つしか使いません。3つすべてを知っているチームは、複雑さを抑えながらより優れた検索機能をリリースし、まだ必要のない専用の検索サービスへのコストのかかる寄り道を回避できます。

このガイドでは、Postgresネイティブのオプション一式について、それぞれが何をするのか、いつ適しているのか、そしてそれらをどのように階層化するのかを解説します。

---

## 3つのツール

**全文検索**（`tsvector` / `GIN` インデックス）は語彙的（lexical）なものです。テキストをレクシーム（lexemes）にトークン化し、それらをステミング（語幹化）して、インデックスに対してクエリを照合します。「Running」と「runs」は同じレクシームに集約されます。「dog」と「dogs」も同様です。ランキング関数（`ts_rank`）は、クエリの用語が頻繁に、あるいは目立つ場所に現れるドキュメントを評価します。

**トライグラム**（`pg_trgm`）は、文字列を重なり合う3文字の断片（スライス）に分割し、2つの文字列がどれだけの断片を共有しているかを測定します。「Dan」は `" da"`, `"dan"`, `"an "` となります。「Micheal」と「Michael」はトライグラムの大部分を共有しているため、類似度が高くなります。これにより、`pg_trgm` は、全文検索が苦手とする曖昧な名前の一致、タイポの許容、オートコンプリートにおいて非常に優れた性能を発揮します。

**完全一致インデックス**（B-tree、ハッシュ）は、主キー、メールアドレス、ID、SKUなど、答えがバイナリ（一致するかしないか）であるものを処理します。これらは「検索」とは感じられないかもしれませんが、正解がある問題に対して曖昧な検索やセマンティック検索を使用してしまうという最悪のパターンを避けるために、この議論に含めるべきものです。

選択肢は洗練されているかどうかではなく、クエリの形状にツールを合わせることです。

## 全文検索が勝る場合

**文章からキーワードを探す場合。** ブログ記事、ドキュメント、製品説明、サポートチケット、法的文書など。全文検索（FTS）は、自然言語テキストに対してインデックスを作成し、ランク付けされた検索を行うために設計されました。

**キーワードベースのユーザークエリ。** ユーザーが検索語を入力したり、タグでフィルタリングしたり、キーワードでブラウズしたりする場合。FTSは、埋め込みインフラを必要とせずに、その意図をネイティブに処理します。

**外部依存のないランク付けされた結果。** FTSインデックスは高速で、決定的であり、APIコールを必要としません。関連性のシグナルは、フィールドの位置によって重み付けされた用語の頻度から得られます。

**検索と並行したブールフィルタリング。** FTSは既存のクエリロジックと自然に調和します。

```sql
SELECT * FROM posts
WHERE search_vector @@ to_tsquery('english', 'postgres & performance')
  AND category = 'tutorial'
  AND published_at > NOW() - INTERVAL '6 months';
```

### FTSのセットアップ

```sql
-- 生成列により、インデックスを自動的に最新に保ちます
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body,  '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- クエリ
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`setweight` は重要度を割り当てます。`A`（タイトル）は `B`（本文）よりも優先されます。これが、ほとんどのコンテンツ検索のユースケースにおける関連性モデルのすべてです。

### FTSが苦手なこと

- クエリ内のタイポ — 「javascipt」は「javascript」に一致しません。
- 人名、住所、辞書や言語設定によってステミングが予測しにくい固有名詞。
- 特別な設定なしでの接頭辞/オートコンプリート。
- ユーザーが用語を挙げるのではなく、概念を説明するクエリ。

## トライグラム（`pg_trgm`）が勝る場合

`pg_trgm` は、FTSが常に失敗する厄介な中間領域をカバーします。

FTSはテキストをレクシームにトークン化し、それらをステミングします。文章の場合、これは正しいアプローチです。しかし、名前や短い識別子の場合、そうでないことが多いのです。

- 人名（「Dan Levy」 → 辞書と言語設定によってステミング結果が変わる）
- 会社名、住所、正確なスペルが重要な製品タイトル
- タイポを含むクエリ — 「Micheal Jordan」、「Amaon」、「javascipt」
- オートコンプリート / 接頭辞検索
- 部分文字列一致（「Johnson」、「Anderson」に一致する「son」）

また、`pg_trgm` は言語に依存しません。これは、多様な言語的背景を持つ名前を扱う場合に重要です。FTSは言語ごとに辞書の設定が必要です。

### 曖昧な名前検索

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- 「Michael Jordan」で検索したときに「Micheal Jordan」を見つけます
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % 演算子 = 類似度の閾値（デフォルト 0.3）
ORDER BY score DESC
LIMIT 10;
```

`%` 演算子は `pg_trgm.similarity_threshold`（デフォルト 0.3、範囲 0–1）を使用します。名前検索の場合、0.3–0.4 あたりが、ノイズを抑えつつタイポを拾うのに適しています。

### オートコンプリート、接頭辞、包含検索

```sql
-- オートコンプリートのための前方一致。トライグラムGINインデックスも役立ちますが、
-- 純粋な左固定の接頭辞にはB-treeパターンインデックスの方が適している場合があります。
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- 長い文字列内の部分一致のための word_similarity
-- （「Andrew Johnson III」内の「Johnson」）
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name
ORDER BY score DESC
LIMIT 10;
```

トライグラムGINインデックスは、特に `ILIKE '%pattern%'` の包含クエリやタイポを許容するマッチング（トライグラムインデックスがない場合は通常フルテーブルスキャンになるパターン）で威力を発揮します。

### FTSよりも pg_trgm を選ぶべき場面

| シナリオ | 使用ツール |
|---|---|
| タイポを含む人名/会社名検索 | `pg_trgm` |
| オートコンプリート / 接頭辞検索 | `pg_trgm`（または接頭辞クエリ付きFTS） |
| 短い文字列、識別子、コード | `pg_trgm` |
| 記事、ドキュメント、チケットなどの文章 | FTS |
| キーワードによるログメッセージ検索 | FTS |
| 多言語の名前検索 | `pg_trgm`（言語に依存しない） |

## 完全一致SQLが勝る場合

「検索」問題の中には、実際には検索ではないものもあります。

「メールアドレスが `dan@example.com` のユーザーを探す」のは等価性のチェックです。「注文番号 `ORD-12345` を探す」のは主キーのルックアップです。「`tutorial` カテゴリの投稿を日付順にリストする」のはフィルタリングされたクエリです。これらは B-tree またはハッシュインデックスで行うべきものです。

ここで FTS やトライグラムを使用すると、正確性が向上することなく複雑さだけが増します。また、正確な識別子の場合、ニアミスは一致しないことよりも悪い結果を招きます。

```sql
CREATE INDEX users_email_idx ON users (email);

-- 完全一致ルックアップ：高速で曖昧さがない
SELECT id, name FROM users WHERE email = $1;
```

より広い教訓として、正解がある問題に対して近似検索を行うのはカテゴリーエラーです。それは*何か*を返しますが、それは自信満々に間違っている可能性があります。

## これらのツールの組み合わせ

これらのツールはきれいに構成できます。どれか1つだけを選ぶ必要はありません。

**キーワードのタイポを許容する検索ボックスのための FTS + pg_trgm：**

```sql
-- タイトルのトライグラム類似度でタイポを拾い、ts_rankで本文の関連性を処理します
SELECT id, title,
  ts_rank(search_vector, to_tsquery('simple', $1)) AS fts_rank,
  similarity(title, $1) AS trgm_score
FROM posts
WHERE search_vector @@ to_tsquery('simple', $1)
   OR title % $1
ORDER BY (ts_rank(search_vector, to_tsquery('simple', $1)) + similarity(title, $1)) DESC
LIMIT 10;
```

**国際的なコンテンツのための FTS + `unaccent`：**

```sql
-- 発音区別符号を取り除き、「José」が「Jose」に一致するようにします
CREATE EXTENSION IF NOT EXISTS unaccent;

CREATE TEXT SEARCH CONFIGURATION public.simple_unaccent (COPY = pg_catalog.simple);

ALTER TEXT SEARCH CONFIGURATION public.simple_unaccent
  ALTER MAPPING FOR hword, hword_part, word
  WITH unaccent, simple;

ALTER TABLE posts ADD COLUMN search_vector tsvector;

CREATE TRIGGER posts_search_vector_refresh
BEFORE INSERT OR UPDATE OF title, body ON posts
FOR EACH ROW EXECUTE FUNCTION
  tsvector_update_trigger(search_vector, 'public.simple_unaccent', title, body);
```

**国際的な名前検索のための `unaccent` + `pg_trgm`：**

```sql
ALTER TABLE users ADD COLUMN name_search text;

CREATE FUNCTION users_name_search_refresh()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  NEW.name_search := unaccent(coalesce(NEW.name, ''));
  RETURN NEW;
END;
$$;

CREATE TRIGGER users_name_search_refresh
BEFORE INSERT OR UPDATE OF name ON users
FOR EACH ROW EXECUTE FUNCTION users_name_search_refresh();

CREATE INDEX users_name_search_trgm_idx
  ON users USING GIN (name_search gin_trgm_ops);

SELECT id, name
FROM users
WHERE name_search % unaccent($1)
ORDER BY similarity(name_search, unaccent($1)) DESC
LIMIT 10;
```

トリガーの例では、生成列やインデックス式の中で `unaccent()` を直接使うのを避けています。これは PostgreSQL の不変性（immutability）ルールが重要だからです。`unaccent()` を独自の不変関数でラップする場合は、アップグレードや設定のリスクを許容することを文書化してください。

## 注目すべき拡張機能

**`pg_trgm`** はほとんどの Postgres ディストリビューションに同梱されていますが、明示的な有効化が必要です。Postgres における曖昧な文字列マッチングの基盤です。

**`unaccent`** は、インデックス作成とクエリの前に発音区別符号を取り除きます。欧州言語のコンテンツに対して、`pg_trgm` と FTS の両方と相性が良いです。Postgres に同梱されています。

**`pg_bigm`** は、トライグラムのアプローチをバイグラム（2文字の断片）に拡張したもので、`pg_trgm` が苦手とする CJK（中国語、日本語、韓国語）言語の結果を大幅に改善します。別途インストールが必要で、同梱はされていません。

**`pg_search`**（[ParadeDB](https://www.paradedb.com/) 製）は、標準の `GIN` / `tsvector` スタックを Tantivy ベースの BM25 インデックスに置き換えます。これにより、BM25 スコアリング（多くの場合 `ts_rank` より優れています）、FTS クエリ内での曖昧一致、ファセット検索、そして大規模テーブルでの劇的に高速なインデックス作成が可能になります。標準の FTS でランキングやパフォーマンスの限界が見えてきたときの、ドロップインのアップグレードパスとなります。

```sql
-- pg_search: 曖昧一致機能を備えた BM25 全文検索
CREATE INDEX posts_bm25_idx ON posts
  USING bm25 (id, title, body)
  WITH (key_field = 'id', text_fields = '{"title": {}, "body": {}}');

-- BM25 スコアリング + 曖昧一致（「javascipt」を拾う）でのクエリ
SELECT id, title, paradedb.score(id) AS rank
FROM posts
WHERE posts @@@ paradedb.fuzzy_phrase(field => 'title', value => 'postgres performnce')
ORDER BY rank DESC
LIMIT 10;
```

**`pgvector`** は、密ベクトルの保存と類似性検索を追加します。ユーザーが名前を挙げるのではなく、欲しいものを説明する場合（セマンティック検索、RAG、関連コンテンツのレコメンデーション、多言語クエリ）に適したツールです。詳細は「[セマンティックベクトル検索の展望とハイブリッド戦略](/semantic-vector-search-landscape)」で詳しく解説しています。

## 決定表

| 検索対象 | 推奨ツール |
|---|---|
| 文章、記事、ドキュメント、チケット | FTS |
| タイポを含む人名/会社名 | `pg_trgm` |
| オートコンプリート、接頭辞検索 | `pg_trgm` |
| 短いコード、識別子 | `pg_trgm` |
| キーワードによるログメッセージ | FTS |
| 国際的な名前 | `pg_trgm` + `unaccent` |
| 大規模コンテンツ、より良いランキング | `pg_search` (ParadeDB BM25) |
| 主キー、正確なメールアドレス、ID | B-tree インデックス |
| 日付、範囲、ソート済みリスト | B-tree インデックス |
| 権限、カテゴリ、フィルタ | 通常の WHERE 句 |
| 質問、言い換え、概念 | pgvector（次の記事を参照） |

迷ったときは：スペルのばらつきがある短い文字列 → トライグラム。キーワードクエリ用の長い文章 → FTS。構造化された識別子 → 通常のインデックス。概念的または自然言語のクエリ → pgvector。

## ハイブリッド検索：2つのシグナル、1つのランク

「`withRetry timeout errors`」のようなクエリが検索ボックスに入力されたとき、そこには2種類の意図が含まれています。ユーザーが知っている正確なシンボル名（`withRetry`）と、概念的な説明（`timeout errors`）です。単一のプリミティブでは両方をカバーできません。FTS とベクトル検索を並行して実行し、そのランク付けされたリストを Reciprocal Rank Fusion でマージすることで、これが可能になります。

RRF は、各リストにおける各結果のスコアを `1 / (60 + rank)` として計算し、リスト間で合計します。定数 60 は上位ランクの優位性を和らげるため、両方のリストで2位になった結果が、一方のリストで1位になり他方で圏外になった結果に勝つことができます。重要なのは、RRF は決して手法間で生のスコアを平均しないことです。FTS のランクとコサイン距離は異なる通貨であり、算術的に組み合わせることはできません。

```sql
-- ハイブリッド検索：FTS + pgvector を RRF でマージ
WITH fts AS (
  SELECT id, ts_rank(search_vector, query) AS score,
         ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM docs, to_tsquery('english', 'withRetry & timeout') query
  WHERE search_vector @@ query
  LIMIT 60
),
vec AS (
  SELECT id,
         ROW_NUMBER() OVER (ORDER BY embedding <=> $embedding) AS rank
  FROM docs
  ORDER BY embedding <=> $embedding
  LIMIT 60
)
SELECT COALESCE(fts.id, vec.id) AS id,
       (COALESCE(1.0 / (60 + fts.rank), 0) +
        COALESCE(1.0 / (60 + vec.rank), 0)) AS rrf_score
FROM fts FULL JOIN vec ON fts.id = vec.id
ORDER BY rrf_score DESC
LIMIT 10;
```

各ブランチあたりの候補プールを 60 ドキュメント（`LIMIT 60`）に設定するのが一般的な出発点です。再現率が低い場合は広げ、速度を優先する場合は狭めてください。
