Die meisten Teams nutzen nur ein PostgreSQL-Suchwerkzeug. Teams, die alle drei beherrschen, liefern bessere Suchergebnisse mit weniger Komplexität – und vermeiden den kostspieligen Umweg zu einem dedizierten Suchdienst, den sie noch nicht brauchten.

Dieser Leitfaden deckt die vollständige Menge an PostgreSQL-nativen Optionen ab: was jedes Werkzeug macht, wann es seine Berechtigung hat, und wie man sie kombiniert.

---

## Die drei Werkzeuge

**Textsuche** (`tsvector` / GIN-Index) ist lexikalisch. Sie tokenisiert Text in Lexeme, stemmt sie und matcht Abfragen gegen den Index. „Running" und „runs" fallen auf dasselbe Lexem zusammen. Ebenso „dog" und „dogs". Die Rangierfunktion (`ts_rank`) belohnt Dokumente, in denen Suchbegriffe häufig oder prominent erscheinen.

**Trigramme** (`pg_trgm`) zerlegen Strings in überlappende 3-Zeichen-Schnipsel und messen, wie viele Schnipsel zwei Strings gemeinsam haben. „Dan" → `" da"`, `"dan"`, `"an "`. „Micheal" und „Michael" teilen die meisten Trigramme, also ist die Ähnlichkeit hoch. Das macht `pg_trgm` hervorragend für unscharfes Namensmatching, Tippfehler-Toleranz und Autocomplete – den Bereich, in dem FTS schlecht performt.

**Exakt-Match-Indexes** (B-tree, Hash) behandeln Primärschlüssel, E-Mail-Adressen, IDs, SKUs und alles, wo die Antwort binär ist: es matcht oder es matcht nicht. Das fühlt sich nicht wie „Suche" an, gehört aber zu dieser Diskussion, weil das schlimmste Muster ist, unscharfe oder semantische Suche für Probleme einzusetzen, die korrekte Antworten haben.

Die Wahl geht nicht um Raffinesse. Es geht darum, das Werkzeug der Form der Abfrage anzupassen.

## Wenn Textsuche gewinnt

**Suchen in Fließtext nach Stichworten.** Blogartikel, Dokumentationen, Produktbeschreibungen, Support-Tickets, juristische Dokumente. FTS wurde dafür entworfen: indizierte, rangierte Abfrage über natürlichen Text.

**Stichwort-basierte Nutzerabfragen.** Nutzer tippen einen Suchbegriff, filtern nach Tag oder browsen nach Keyword. FTS behandelt diese Absicht nativ ohne jede Embedding-Infrastruktur.

**Rangierte Ergebnisse ohne externe Abhängigkeiten.** FTS-Indizes sind schnell, deterministisch und benötigen keine API-Calls. Das Relevanzsignal kommt aus Termfrequenz gewichtet nach Feldposition.

**Boolesche Filterung alongside Suche.** FTS komponiert natürlich mit Ihrer bestehenden Query-Logik:

```sql
SELECT * FROM posts
WHERE search_vector @@ to_tsquery('english', 'postgres & performance')
  AND category = 'tutorial'
  AND published_at > NOW() - INTERVAL '6 months';
```

### FTS einrichten

```sql
-- Generated Column hält den Index automatisch aktuell
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body,  '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Query
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`setweight` vergibt Wichtigkeit: `A` (Titel) rangiert über `B` (Body). Das ist das gesamte Relevanzmodell für die meisten Content-Suche-Fälle.

### Was FTS schlecht handhabt

- Tippfehler in Abfragen — „javascipt" matcht nicht „javascript"
- Personennamen, Adressen, Eigennamen, die nicht vorhersehbar stemmen
- Prefix/Autocomplete ohne spezielle Konfiguration
- Abfragen, wo Nutzer ein Konzept beschreiben statt es zu benennen

## Wenn Trigramme gewinnen (`pg_trgm`)

`pg_trgm` deckt die unangenehme Mitte ab, die FTS konsistent fummelt.

FTS tokenisiert Text in Lexeme und stemmt sie. Für Fließtext ist das korrekt. Für Namen und kurze Identifikatoren oft nicht:

- Personennamen („Dan Levy" → anders gestemmt je nach Dictionary und Sprachkonfiguration)
- Firmennamen, Adressen, Produkttitel, wo exakte Schreibweise zählt
- Abfragen mit Tippfehlern — „Micheal Jordan", „Amaon", „javascipt"
- Autocomplete / Prefix-Suche
- Teilstring-Matching („son" matcht „Johnson", „Anderson")

`pg_trgm` ist auch sprachagnostisch, was für Namen aus diversen linguistischen Hintergründen zählt. FTS benötigt Dictionary-Konfiguration pro Sprache.

### Unscharfer Namenssuche

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- Findet „Micheal Jordan" bei Suche nach „Michael Jordan"
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % Operator = Ähnlichkeitsschwelle (default 0.3)
ORDER BY score DESC
LIMIT 10;
```

Der `%` Operator nutzt `pg_trgm.similarity_threshold` (default 0.3, Bereich 0–1). Für Namenssuche fängt 0.3–0.4 Tippfehler ab bei niedrigem Rauschen.

### Autocomplete, Prefix und Contains-Suche

```sql
-- Prefix-Matching für Autocomplete. Ein Trigram GIN Index kann helfen,
-- aber ein B-tree Pattern Index ist oft besser für rein links-ankern Prefixes.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- word_similarity für Teil-Matches innerhalb längerer Strings
-- („Johnson" innerhalb von „Andrew Johnson III")
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name
ORDER BY score DESC
LIMIT 10;
```

Der Trigram GIN Index ist besonders nützlich für `ILIKE '%pattern%'` Contains-Abfragen und Tippfehler-tolerantes Matching – Muster, die normalerweise Full-Table-Scans ohne Trigram Index sind.

### Wann pg_trgm statt FTS

| Szenario | Nutzen |
|---|---|
| Personen-/Firmennamen mit Tippfehlern | `pg_trgm` |
| Autocomplete / Prefix-Suche | `pg_trgm` (oder FTS mit Prefix-Queries) |
| Kurze Strings, Identifikatoren, Codes | `pg_trgm` |
| Fließartikel, Docs, Tickets | FTS |
| Log-Nachrichten nach Stichworten | FTS |
| Mehrsprachige Namenssuche | `pg_trgm` (sprachagnostisch) |

## Wenn Exakt-Match SQL gewinnt

Manche „Suche"-Probleme sind gar keine Suche.

„Finde den Nutzer mit E-Mail `dan@example.com`" ist ein Equality-Check. „Finde Bestellung `ORD-12345`" ist ein Primary-Key-Lookup. „Liste Posts in der `tutorial` Kategorie sortiert nach Datum" ist eine gefilterte Query. Diese gehören auf B-tree oder Hash Indizes.

FTS oder Trigramme hier hinzuzufügen fügt Komplexität hinzu ohne Korrektheit zu verbessern – und für exakte Identifikatoren ist ein Near-Match schlimmer als kein Match.

```sql
CREATE INDEX users_email_idx ON users (email);

-- Exakter Lookup: schnell und eindeutig
SELECT id, name FROM users WHERE email = $1;
```

Die breitere Lektion: Approximationssuche für Probleme mit korrekten Antworten ist ein Kategorienfehler. Er liefert *etwas* – was möglicherweise selbstsicher falsch ist.

## Diese Werkzeuge kombinieren

Diese Werkzeuge komponieren sauber. Sie wählen nicht genau eines.

**FTS + pg_trgm für eine Suchbox, die Tippfehler in Keywords toleriert:**

```sql
-- Trigram Ähnlichkeit auf Titel fängt Tippfehler; ts_rank behandelt Body Relevanz
SELECT id, title,
  ts_rank(search_vector, to_tsquery('simple', $1)) AS fts_rank,
  similarity(title, $1) AS trgm_score
FROM posts
WHERE search_vector @@ to_tsquery('simple', $1)
   OR title % $1
ORDER BY (ts_rank(search_vector, to_tsquery('simple', $1)) + similarity(title, $1)) DESC
LIMIT 10;
```

**FTS + `unaccent` für internationale Inhalte:**

```sql
-- Akzente entfernen, damit „José" „Jose" matcht
CREATE EXTENSION IF NOT EXISTS unaccent;

CREATE TEXT SEARCH CONFIGURATION public.simple_unaccent (COPY = pg_catalog.simple);

ALTER TEXT SEARCH CONFIGURATION public.simple_unaccent
  ALTER MAPPING FOR hword, hword_part, word
  WITH unaccent, simple;

ALTER TABLE posts ADD COLUMN search_vector tsvector;

CREATE TRIGGER posts_search_vector_refresh
BEFORE INSERT OR UPDATE OF title, body ON posts
FOR EACH ROW EXECUTE FUNCTION
  tsvector_update_trigger(search_vector, 'public.simple_unaccent', title, body);
```

**`unaccent` + `pg_trgm` für internationale Namenssuche:**

```sql
ALTER TABLE users ADD COLUMN name_search text;

CREATE FUNCTION users_name_search_refresh()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  NEW.name_search := unaccent(coalesce(NEW.name, ''));
  RETURN NEW;
END;
$$;

CREATE TRIGGER users_name_search_refresh
BEFORE INSERT OR UPDATE OF name ON users
FOR EACH ROW EXECUTE FUNCTION users_name_search_refresh();

CREATE INDEX users_name_search_trgm_idx
  ON users USING GIN (name_search gin_trgm_ops);

SELECT id, name
FROM users
WHERE name_search % unaccent($1)
ORDER BY similarity(name_search, unaccent($1)) DESC
LIMIT 10;
```

Die Trigger-Beispiele vermeiden `unaccent()` in generated-column oder Index-Expressionen, wo Postgress Immutability-Regeln zählen. Wenn Sie `unaccent()` in Ihre eigene immutable Funktion wickeln, dokumentieren Sie, dass Sie Upgrade-/Konfigurationsrisiko akzeptieren.

## Bemerkenswerte Erweiterungen

**`pg_trgm`** ist in den meisten PostgreSQL-Distributionen gebündelt, benötigt aber explizite Aktivierung. Die Basis für unscharfes String-Matching in PostgreSQL.

**`unaccent`** entfernt diakritische Zeichen vor Indexierung und Abfrage. Paart gut mit sowohl `pg_trgm` als auch FTS für europäisch-sprachige Inhalte. Mit PostgreSQL gebündelt.

**`pg_bigm`** erweitert den Trigram-Ansatz zu Bigrammen (2-Zeichen-Schnipsel), was Ergebnisse für CJK (Chinesisch, Japanisch, Koreanisch) Sprachen signifikant verbessert, wo `pg_trgm` unterperformt. Muss separat installiert werden; nicht gebündelt.

**`pg_search`** (von [ParadeDB](https://www.paradedb.com/)) ersetzt den Standard `GIN` / `tsvector` Stack mit einem Tantivy-basierten BM25-Index. Das gibt Ihnen BM25 Scoring (oft besser als `ts_rank`), fuzzy matching innerhalb FTS-Queries, faceted search und dramatisch schnelleres Indexieren auf großen Tabellen. Es ist ein drop-in Upgrade-Pfad, wenn Standard FTS Ranking oder Performance-Grenzen zeigt.

```sql
-- pg_search: BM25 Textsuche mit fuzzy matching
CREATE INDEX posts_bm25_idx ON posts
  USING bm25 (id, title, body)
  WITH (key_field = 'id', text_fields = '{"title": {}, "body": {}}');

-- Query mit BM25 Scoring + fuzzy matching (fängt „javascipt" ab)
SELECT id, title, paradedb.score(id) AS rank
FROM posts
WHERE posts @@@ paradedb.fuzzy_phrase(field => 'title', value => 'postgres performnce')
ORDER BY rank DESC
LIMIT 10;
```

**`pgvector`** fügt dense vector storage und similarity search hinzu. Es ist das richtige Werkzeug, wenn Nutzer beschreiben, was sie wollen statt es zu benennen – semantische Suche, RAG, Related-Content Recommendations, mehrsprachige Queries. Tief abgedeckt in [Semantische Vektorsuche und Hybrid-Strategien](/semantic-vector-search-landscape).

## Entscheidungstabelle

| Wonach Sie suchen | Empfohlen |
|---|---|
| Fließartikel, Docs, Tickets | FTS |
| Personen-/Firmennamen mit Tippfehlern | `pg_trgm` |
| Autocomplete, Prefix-Suche | `pg_trgm` |
| Kurze Codes, Identifikatoren | `pg_trgm` |
| Log-Nachrichten nach Stichworten | FTS |
| Internationale Namen | `pg_trgm` + `unaccent` |
| Große Inhalte, besseres Ranking | `pg_search` (ParadeDB BM25) |
| Primärschlüssel, exakte E-Mails, IDs | B-tree Index |
| Datumsbereiche, sortierte Listen | B-tree Index |
| Berechtigungen, Kategorien, Filter | Normale WHERE Clause |
| Fragen, Paraphrasen, Konzepte | pgvector (siehe nächster Artikel) |

Wenn unsicher: kurze Strings mit Rechtschreibvariation → Trigramme. Langer Fließtext für Keyword-Queries → FTS. Strukturierte Identifikatoren → normale Indizes. Konzeptionelle oder natürliche Sprach-Queries → pgvector.

## Hybrid-Suche: Zwei Signale, ein Rang

Wenn eine Query wie `"withRetry timeout errors"` eine Suchbox trifft, trägt sie zwei Arten von Absicht: exakte Symbols, die Nutzer kennen (`withRetry`) und eine konzeptionelle Beschreibung (`timeout errors`). Keine einzelne Primitive deckt beides ab. FTS und Vektorsuche parallel laufen lassen – dann ihre rangierten Listen mit Reciprocal Rank Fusion fusionieren – das tut es.

RRF scoret jedes Ergebnis als `1 / (60 + rank)` in jeder Liste und summiert über Listen. Die Konstante 60 dämpft den Vorteil von Top-Ranks, also kann ein Ergebnis, das in beiden Listen zweitplatziert ist, ein Ergebnis schlagen, das eine Liste gewinnt und die andere verpasst. Wichtig: RRF mittelt nie Rohscores über Methoden hinweg – FTS rank und cosine distance sind unterschiedliche Währungen und können arithmetisch nicht kombiniert werden.

<figcaption>Eine Query verzweigt parallel zu FTS und pgvector. Jede produziert ihre eigene rangierte Liste. RRF scoret jedes Dokument nach seiner Position in jeder Liste und summiert Scores – das Ergebnis surfet Dokumente, bei denen beide Signale übereinstimmen.</figcaption>

```sql
-- Hybrid-Suche: FTS + pgvector fusioniert mit RRF
WITH fts AS (
  SELECT id, ts_rank(search_vector, query) AS score,
         ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM docs, to_tsquery('english', 'withRetry & timeout') query
  WHERE search_vector @@ query
  LIMIT 60
),
vec AS (
  SELECT id,
         ROW_NUMBER() OVER (ORDER BY embedding <=> $embedding) AS rank
  FROM docs
  ORDER BY embedding <=> $embedding
  LIMIT 60
)
SELECT COALESCE(fts.id, vec.id) AS id,
       (COALESCE(1.0 / (60 + fts.rank), 0) +
        COALESCE(1.0 / (60 + vec.rank), 0)) AS rrf_score
FROM fts FULL JOIN vec ON fts.id = vec.id
ORDER BY rrf_score DESC
LIMIT 10;
```

Der 60-Dokumenten-Kandidatenpool pro Branch (`LIMIT 60`) ist ein häufiger Startpunkt. Erweitern Sie ihn, wenn Recall niedrig ist; verengen Sie ihn für Geschwindigkeit.
