Die meisten Teams nutzen nur ein Postgres-Suchwerkzeug. Teams, die alle drei kennen, liefern bessere Suchergebnisse mit weniger Komplexität – und vermeiden den kostspieligen Umweg zu einem dedizierten Suchdienst, den sie eigentlich noch nicht brauchen.

Dieser Leitfaden deckt alle Postgres-nativen Optionen ab: was sie leisten, wann sie die richtige Wahl sind und wie man sie kombiniert.

---

## Die drei Werkzeuge

**Volltextsuche** (`tsvector` / `GIN`-Index) arbeitet lexikalisch. Sie zerlegt Text in Token (Lexeme), führt ein Stemming (Stammformenbildung) durch und gleicht Abfragen mit dem Index ab. „Laufen“ und „läuft“ werden auf dasselbe Lexem reduziert. Ebenso „Hund“ und „Hunde“. Die Ranking-Funktion (`ts_rank`) belohnt Dokumente, in denen Suchbegriffe häufig oder an prominenter Stelle erscheinen.

**Trigramme** (`pg_trgm`) zerlegen Zeichenfolgen in überlappende 3-Zeichen-Abschnitte und messen, wie viele Abschnitte zwei Zeichenfolgen gemeinsam haben. „Dan“ → `" da"`, `"dan"`, `"an "`. „Micheal“ und „Michael“ teilen die meisten ihrer Trigramme, daher ist die Ähnlichkeit hoch. Das macht `pg_trgm` hervorragend für unscharfe Namenssuche (Fuzzy Matching), Fehlertoleranz und Autovervollständigung – genau dort, wo die Volltextsuche schwächelt.

**Exakte Treffer-Indizes** (B-Tree, Hash) verarbeiten Primärschlüssel, E-Mail-Adressen, IDs, SKUs und alles, bei dem die Antwort binär ist: Es passt oder es passt nicht. Das fühlt sich nicht nach „Suche“ an, gehört aber in diese Diskussion, weil das schlimmste Muster die Verwendung von unscharfer oder semantischer Suche für Probleme ist, die eindeutige Antworten haben.

Die Wahl hängt nicht von der Raffinesse ab. Es geht darum, das Werkzeug an die Form der Abfrage anzupassen.

## Wann die Volltextsuche gewinnt

**Suche in Prosa nach Schlüsselwörtern.** Blog-Posts, Dokumentationen, Produktbeschreibungen, Support-Tickets, rechtliche Dokumente. FTS wurde für diese Form von Inhalten entwickelt: indexierter, gerankter Abruf über natürlichsprachlichem Text.

**Schlüsselwortbasierte Nutzeranfragen.** Nutzer geben einen Suchbegriff ein, filtern nach Tags oder browsen nach Keywords. FTS verarbeitet diese Absicht nativ ohne jegliche Embedding-Infrastruktur.

**Gerankte Ergebnisse ohne externe Abhängigkeiten.** FTS-Indizes sind schnell, deterministisch und erfordern keine API-Aufrufe. Das Relevanzsignal stammt aus der Termfrequenz, gewichtet nach der Feldposition.

**Boolesche Filterung neben der Suche.** FTS lässt sich natürlich mit Ihrer bestehenden Abfragelogik kombinieren:

```sql
SELECT * FROM posts
WHERE search_vector @@ to_tsquery('german', 'postgres & performance')
  AND category = 'tutorial'
  AND published_at > NOW() - INTERVAL '6 months';
```

### Einrichten von FTS

```sql
-- Generierte Spalte hält den Index automatisch aktuell
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('german', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('german', coalesce(body,  '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Abfrage
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('german', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`setweight` weist Wichtigkeit zu: `A` (Titel) übertrifft `B` (Textkörper). Das ist das gesamte Relevanzmodell für die meisten Content-Suchanwendungsfälle.

### Was FTS nicht gut beherrscht

- Tippfehler in Abfragen – „javascipt“ findet „javascript“ nicht.
- Personennamen, Adressen, Eigennamen, die nicht vorhersehbar gestemmt werden.
- Präfix-/Autovervollständigung ohne spezielle Konfiguration.
- Abfragen, bei denen der Nutzer ein Konzept beschreibt, anstatt es beim Namen zu nennen.

## Wann Trigramme gewinnen (`pg_trgm`)

`pg_trgm` deckt die schwierige Mitte ab, an der FTS regelmäßig scheitert.

FTS zerlegt Text in Lexeme und führt Stemming durch. Für Prosa ist das korrekt. Für Namen und kurze Identifikatoren oft nicht:

- Personennamen („Dan Levy“ → wird je nach Wörterbuch und Sprachkonfiguration unterschiedlich gestemmt).
- Firmennamen, Adressen, Produkttitel, bei denen die exakte Schreibweise zählt.
- Abfragen mit Tippfehlern – „Micheal Jordan“, „Amaon“, „javascipt“.
- Autovervollständigung / Präfixsuche.
- Teilstring-Matching (Finden von „son“ in „Johnson“, „Anderson“).

`pg_trgm` ist zudem sprachunabhängig, was bei Namen aus unterschiedlichen sprachlichen Hintergründen wichtig ist. FTS erfordert eine Wörterbuchkonfiguration pro Sprache.

### Unscharfe Namenssuche

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- Findet „Micheal Jordan“, wenn nach „Michael Jordan“ gesucht wird
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % Operator = Ähnlichkeitsschwellenwert (Standard 0.3)
ORDER BY score DESC
LIMIT 10;
```

Der `%`-Operator verwendet `pg_trgm.similarity_threshold` (Standard 0,3, Bereich 0–1). Für die Namenssuche fängt 0,3–0,4 Tippfehler ab, während das Rauschen niedrig bleibt.

### Autovervollständigung, Präfix- und Enthält-Suche

```sql
-- Präfix-Matching für Autovervollständigung. Ein Trigramm-GIN-Index kann helfen,
-- aber ein B-Tree-Pattern-Index ist für rein linksbündige Präfixe oft besser.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- word_similarity für Teilübereinstimmungen innerhalb längerer Zeichenfolgen
-- („Johnson“ innerhalb von „Andrew Johnson III“)
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name
ORDER BY score DESC
LIMIT 10;
```

Der Trigramm-GIN-Index ist besonders nützlich für `ILIKE '%muster%'`-Abfragen und tippfehlertolerantes Matching – Muster, die ohne Trigramm-Index normalerweise Full-Table-Scans verursachen würden.

### Wann man zu pg_trgm statt FTS greift

| Szenario | Verwendung |
|---|---|
| Personen-/Firmennamensuche mit Tippfehlern | `pg_trgm` |
| Autovervollständigung / Präfixsuche | `pg_trgm` (oder FTS mit Präfix-Abfragen) |
| Kurze Zeichenfolgen, Identifikatoren, Codes | `pg_trgm` |
| Prosa-Artikel, Dokumentationen, Tickets | FTS |
| Log-Nachrichten nach Schlüsselwörtern | FTS |
| Mehrsprachige Namenssuche | `pg_trgm` (sprachunabhängig) |

## Wann exaktes SQL gewinnt

Manche „Suchprobleme“ sind gar keine Suche.

„Finde den Nutzer mit der E-Mail `dan@example.com`“ ist eine Gleichheitsprüfung. „Finde Bestellung `ORD-12345`“ ist ein Primärschlüssel-Lookup. „Liste Beiträge in der Kategorie `tutorial` sortiert nach Datum auf“ ist eine gefilterte Abfrage. Diese gehören auf B-Tree- oder Hash-Indizes.

Die Verwendung von FTS oder Trigrammen fügt hier Komplexität hinzu, ohne die Korrektheit zu verbessern – und bei exakten Identifikatoren ist ein Beinahe-Treffer schlimmer als gar kein Treffer.

```sql
CREATE INDEX users_email_idx ON users (email);

-- Exakter Lookup: schnell und eindeutig
SELECT id, name FROM users WHERE email = $1;
```

Die übergeordnete Lektion: Eine ungefähre Suche für Probleme mit exakten Antworten ist ein Kategorienfehler. Sie liefert *etwas* zurück – das aber überzeugt falsch sein kann.

## Kombination dieser Werkzeuge

Diese Werkzeuge lassen sich sauber kombinieren. Man muss sich nicht für genau eines entscheiden.

**FTS + pg_trgm für ein Suchfeld, das Tippfehler in Schlüsselwörtern toleriert:**

```sql
-- Trigramm-Ähnlichkeit auf dem Titel fängt Tippfehler ab; ts_rank regelt die Relevanz im Textkörper
SELECT id, title,
  ts_rank(search_vector, to_tsquery('simple', $1)) AS fts_rank,
  similarity(title, $1) AS trgm_score
FROM posts
WHERE search_vector @@ to_tsquery('simple', $1)
   OR title % $1
ORDER BY (ts_rank(search_vector, to_tsquery('simple', $1)) + similarity(title, $1)) DESC
LIMIT 10;
```

**FTS + `unaccent` für internationale Inhalte:**

```sql
-- Entfernt diakritische Zeichen, damit „José“ auf „Jose“ passt
CREATE EXTENSION IF NOT EXISTS unaccent;

CREATE TEXT SEARCH CONFIGURATION public.simple_unaccent (COPY = pg_catalog.simple);

ALTER TEXT SEARCH CONFIGURATION public.simple_unaccent
  ALTER MAPPING FOR hword, hword_part, word
  WITH unaccent, simple;

ALTER TABLE posts ADD COLUMN search_vector tsvector;

CREATE TRIGGER posts_search_vector_refresh
BEFORE INSERT OR UPDATE OF title, body ON posts
FOR EACH ROW EXECUTE FUNCTION
  tsvector_update_trigger(search_vector, 'public.simple_unaccent', title, body);
```

**`unaccent` + `pg_trgm` für die internationale Namenssuche:**

```sql
ALTER TABLE users ADD COLUMN name_search text;

CREATE FUNCTION users_name_search_refresh()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  NEW.name_search := unaccent(coalesce(NEW.name, ''));
  RETURN NEW;
END;
$$;

CREATE TRIGGER users_name_search_refresh
BEFORE INSERT OR UPDATE OF name ON users
FOR EACH ROW EXECUTE FUNCTION users_name_search_refresh();

CREATE INDEX users_name_search_trgm_idx
  ON users USING GIN (name_search gin_trgm_ops);

SELECT id, name
FROM users
WHERE name_search % unaccent($1)
ORDER BY similarity(name_search, unaccent($1)) DESC
LIMIT 10;
```

Die Trigger-Beispiele vermeiden die Verwendung von `unaccent()` innerhalb von generierten Spalten oder Index-Ausdrücken, wo die Immutabilitätsregeln von PostgreSQL eine Rolle spielen. Wenn Sie `unaccent()` in einer eigenen immutablen Funktion kapseln, dokumentieren Sie, dass Sie ein Upgrade-/Konfigurationsrisiko eingehen.

## Bemerkenswerte Erweiterungen

**`pg_trgm`** wird mit den meisten Postgres-Distributionen ausgeliefert, erfordert aber eine explizite Aktivierung. Die Basis für unscharfes String-Matching in Postgres.

**`unaccent`** entfernt diakritische Zeichen vor dem Indexieren und Abfragen. Lässt sich gut mit sowohl `pg_trgm` als auch FTS für europäischsprachige Inhalte kombinieren. In Postgres enthalten.

**`pg_bigm`** erweitert den Trigramm-Ansatz auf Bigramme (2-Zeichen-Abschnitte), was die Ergebnisse für CJK-Sprachen (Chinesisch, Japanisch, Koreanisch) erheblich verbessert, wo `pg_trgm` unterperformt. Muss separat installiert werden.

**`pg_search`** (von [ParadeDB](https://www.paradedb.com/)) ersetzt den Standard-`GIN` / `tsvector`-Stack durch einen Tantivy-basierten BM25-Index. Dies bietet BM25-Scoring (oft besser als `ts_rank`), unscharfes Matching innerhalb von FTS-Abfragen, facettierte Suche und drastisch schnelleres Indexieren bei großen Tabellen. Es ist ein Upgrade-Pfad, wenn die Standard-FTS an ihre Ranking- oder Performance-Grenzen stößt.

```sql
-- pg_search: BM25-Volltextsuche mit unscharfem Matching
CREATE INDEX posts_bm25_idx ON posts
  USING bm25 (id, title, body)
  WITH (key_field = 'id', text_fields = '{"title": {}, "body": {}}');

-- Abfrage mit BM25-Scoring + unscharfem Matching (findet „javascipt“)
SELECT id, title, paradedb.score(id) AS rank
FROM posts
WHERE posts @@@ paradedb.fuzzy_phrase(field => 'title', value => 'postgres performnce')
ORDER BY rank DESC
LIMIT 10;
```

**`pgvector`** fügt Speicherung von dichten Vektoren und Ähnlichkeitssuche hinzu. Es ist das richtige Werkzeug, wenn Nutzer beschreiben, was sie wollen, anstatt es zu benennen – semantische Suche, RAG, Empfehlungen für ähnliche Inhalte, mehrsprachige Abfragen. Vertieft in [Semantische Vektorsuche und Hybrid-Strategien](/semantic-vector-search-landscape).

## Entscheidungstabelle

| Was Sie suchen | Empfohlen |
|---|---|
| Prosa-Artikel, Dokus, Tickets | FTS |
| Personen-/Firmennamen mit Tippfehlern | `pg_trgm` |
| Autovervollständigung, Präfixsuche | `pg_trgm` |
| Kurze Codes, Identifikatoren | `pg_trgm` |
| Log-Nachrichten nach Schlüsselwörtern | FTS |
| Internationale Namen | `pg_trgm` + `unaccent` |
| Große Inhalte, besseres Ranking | `pg_search` (ParadeDB BM25) |
| Primärschlüssel, exakte E-Mails, IDs | B-Tree-Index |
| Daten, Zeiträume, sortierte Listen | B-Tree-Index |
| Berechtigungen, Kategorien, Filter | Reguläre WHERE-Klausel |
| Fragen, Umschreibungen, Konzepte | pgvector (siehe nächster Artikel) |

Im Zweifelsfall: Kurze Zeichenfolgen mit Schreibvarianten → Trigramme. Lange Prosa für Schlüsselwortabfragen → FTS. Strukturierte Identifikatoren → reguläre Indizes. Konzeptuelle oder natürlichsprachliche Abfragen → pgvector.

## Hybridsuche: Zwei Signale, ein Rang

Wenn eine Abfrage wie `"withRetry timeout errors"` auf ein Suchfeld trifft, trägt sie zwei Arten von Absichten in sich: exakte Symbolnamen, die der Nutzer kennt (`withRetry`), und eine konzeptuelle Beschreibung (`timeout errors`). Kein einzelnes Primitiv deckt beides ab. FTS und Vektorsuche parallel auszuführen – und dann ihre Ranglisten mit Reciprocal Rank Fusion zu mergen – hingegen schon.

RRF bewertet jedes Ergebnis als `1 / (60 + Rang)` in jeder Liste und summiert diese über die Listen hinweg. Die Konstante 60 dämpft den Vorteil von Spitzenplätzen, sodass ein Ergebnis, das in beiden Listen auf Platz zwei landet, ein Ergebnis schlagen kann, das eine Liste gewinnt, in der anderen aber gar nicht vorkommt. Entscheidend ist: RRF bildet niemals den Durchschnitt der Rohwerte über die Methoden hinweg – FTS-Rang und Cosinus-Distanz sind unterschiedliche Währungen und können nicht arithmetisch kombiniert werden.

```sql
-- Hybridsuche: FTS + pgvector mit RRF gemergt
WITH fts AS (
  SELECT id, ts_rank(search_vector, query) AS score,
         ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM docs, to_tsquery('german', 'withRetry & timeout') query
  WHERE search_vector @@ query
  LIMIT 60
),
vec AS (
  SELECT id,
         ROW_NUMBER() OVER (ORDER BY embedding <=> $embedding) AS rank
  FROM docs
  ORDER BY embedding <=> $embedding
  LIMIT 60
)
SELECT COALESCE(fts.id, vec.id) AS id,
       (COALESCE(1.0 / (60 + fts.rank), 0) +
        COALESCE(1.0 / (60 + vec.rank), 0)) AS rrf_score
FROM fts FULL JOIN vec ON fts.id = vec.id
ORDER BY rrf_score DESC
LIMIT 10;
```

Der Kandidatenpool von 60 Dokumenten pro Zweig (`LIMIT 60`) ist ein üblicher Ausgangspunkt. Erweitern Sie ihn, wenn die Trefferquote (Recall) zu niedrig ist; verkleinern Sie ihn für mehr Geschwindigkeit.

## Wie es weitergeht

Die Postgres-Textsuche deckt viel Boden ab, hat aber ihre Grenzen. Wenn Nutzer beschreiben, was sie wollen, anstatt es zu benennen – „etwas, das mir hilft, auf einem Flug zu schlafen“, „Artikel über das Vertrauen beim Debuggen als neuer Entwickler“ – scheitern sowohl die lexikalische als auch die Trigramm-Suche.

Das ist das Territorium von Vektor-Embeddings, semantischer Suche und Hybrid-Architekturen. Behandelt in [Semantische Vektorsuche und Hybrid-Strategien](/semantic-vector-search-landscape).
