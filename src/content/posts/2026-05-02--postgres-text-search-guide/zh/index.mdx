大多数团队只会使用一种 Postgres 搜索工具。熟悉全部三种原语的团队能够以更低的复杂度交付更好的搜索——并且避免走上尚未真正需要的专用搜索服务这条昂贵的弯路。

本指南覆盖了 Postgres 原生的全部选项：每种原语的功能、何时适用以及如何层叠使用。

## 三种工具

**全文搜索**（`tsvector` / `GIN` 索引）是词法层面的。它会把文本切分为词元，进行词干提取，并在索引上匹配查询。“Running” 与 “runs” 会归为同一词元，“dog” 与 “dogs” 亦然。排序函数（`ts_rank`）会提升那些查询词出现频繁或位置突出的文档。

**Trigram**（`pg_trgm`）把字符串拆分为重叠的 3‑字符片段，并衡量两个字符串共享了多少片段。比如 “Dan” → `" da"`、`"dan"`、`"an "`。`"Micheal"` 与 `"Michael"` 共享大多数 trigram，因而相似度很高。这使得 `pg_trgm` 在模糊人名匹配、容错拼写和自动补全方面表现出色——正是全文搜索薄弱的场景。

**精确匹配索引**（B‑tree、hash）用于主键、电子邮件、ID、SKU 等二元答案的字段：要么匹配，要么不匹配。它们看起来不像“搜索”，但必须放进本讨论，因为最糟糕的模式是把模糊或语义搜索用于本应有明确答案的问题。

选择的关键不在于技术的高级程度，而在于把工具匹配到查询的形状。

## 当全文搜索占优势时

**在正文中搜索关键词。** 博客文章、文档、产品描述、支持工单、法律文件。全文搜索（FTS）正是为这种内容形态设计的：对自然语言文本进行索引、排序检索。

**基于关键词的用户查询。** 用户输入搜索词、按标签过滤或按关键词浏览。FTS 能原生处理这种意图，无需任何嵌入基础设施。

**无需外部依赖的排序结果。** FTS 索引快速、确定性强，且不需要 API 调用。相关性信号来源于词频，并根据字段位置加权。

**布尔过滤与搜索并行。** FTS 能自然地与现有查询逻辑组合：

```sql
SELECT * FROM posts
WHERE search_vector @@ to_tsquery('english', 'postgres & performance')
  AND category = 'tutorial'
  AND published_at > NOW() - INTERVAL '6 months';
```

### 搭建 FTS

```sql
-- 生成列会自动保持索引最新
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body,  '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- 查询
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`setweight` 用来分配重要性：`A`（标题）优先于 `B`（正文）。这几乎覆盖了大多数内容搜索场景的全部相关性模型。

### FTS 的局限

- 查询中的拼写错误——“javascipt”无法匹配“javascript”
- 人名、地址、专有名词等不易预测词干的内容
- 未经特殊配置的前缀/自动完成
- 用户描述概念而非直接命名的查询

## 当 Trigram 优势明显 (`pg_trgm`)

`pg_trgm` 填补了 FTS 一直处理不好的尴尬中间地带。

FTS 会把文本切分为词元并进行词干提取。对自然语言段落来说这很合适，但对人名和短标识符往往不适用：

- 人名（例如 “Dan Levy” → 根据词典和语言配置会得到不同的词干）
- 公司名、地址、产品标题等对拼写敏感的场景
- 含有拼写错误的查询——“Micheal Jordan”、 “Amaon”、 “javascipt”
- 自动完成 / 前缀搜索
- 部分字符串匹配（“son” 能匹配 “Johnson”、 “Anderson”）

`pg_trgm` 同时是语言无关的，这对来自多语言背景的姓名尤为重要。FTS 则需要为每种语言单独配置词典。

### 模糊姓名搜索

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- 在搜索 “Michael Jordan” 时找到 “Micheal Jordan”
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % 运算符 = 相似度阈值（默认 0.3）
ORDER BY score DESC
LIMIT 10;
```

`%` 运算符使用 `pg_trgm.similarity_threshold`（默认 0.3，取值范围 0–1）。在姓名搜索场景下，0.3–0.4 能捕获拼写错误，同时将噪声控制在较低水平。

### 自动完成、前缀和包含搜索

```sql
-- 自动完成的前缀匹配。三元组 GIN 索引可以提供帮助，
-- 但对纯左侧锚定的前缀，B‑tree 模式索引可能更合适。
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- word_similarity 用于在更长字符串内部的部分匹配
-- （例如在 “Andrew Johnson III” 中匹配 “Johnson”）
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name
ORDER BY score DESC
LIMIT 10;
```

三元组 GIN 索引在 `ILIKE '%pattern%'` 的包含查询以及容错匹配时尤为有用——这些模式在没有三元组索引时通常会导致全表扫描。

### 何时在 FTS 之上选择 pg_trgm

（此处后续内容请继续阅读下一章节）

| 场景 | 使用 |
|---|---|
| 人名/公司名搜索且可能有拼写错误 | `pg_trgm` |
| 自动补全 / 前缀搜索 | `pg_trgm`（或使用前缀查询的 FTS） |
| 短字符串、标识符、代码 | `pg_trgm` |
| 文章、文档、工单等正文 | FTS |
| 日志消息中的关键字 | FTS |
| 多语言姓名搜索 | `pg_trgm`（语言无关） |

## 当精确匹配 SQL 更合适时

有些所谓的“搜索”根本不是搜索。

“查找邮箱为 `dan@example.com` 的用户”是等值判断。  
“查找订单 `ORD-12345`”是主键查找。  
“列出 `tutorial` 分类下按日期排序的帖子”是过滤查询。这类场景应使用 B‑tree 或 hash 索引。

在这里使用 FTS 或三元组会增加复杂度，却并未提升正确性——而对于精确标识符，模糊匹配甚至比不匹配更糟。

```sql
CREATE INDEX users_email_idx ON users (email);

-- 精确查找：快速且唯一
SELECT id, name FROM users WHERE email = $1;
```

更广的教训是：对本应有唯一正确答案的问题使用近似搜索属于类别错误。它会返回*某个*结果——而这个结果可能是自信的错误。

## 组合使用这些工具

这些工具可以干净地组合使用。你不必只能选其一。

**FTS + pg_trgm 用于容错关键词搜索框：**

```sql
-- Trigram 相似度捕捉标题中的错别字；ts_rank 负责正文相关性
SELECT id, title,
  ts_rank(search_vector, to_tsquery('simple', $1)) AS fts_rank,
  similarity(title, $1) AS trgm_score
FROM posts
WHERE search_vector @@ to_tsquery('simple', $1)
   OR title % $1
ORDER BY (ts_rank(search_vector, to_tsquery('simple', $1)) + similarity(title, $1)) DESC
LIMIT 10;
```

**FTS + `unaccent` 处理国际化内容：**

```sql
-- 去除变音符，使 “José” 能匹配 “Jose”
CREATE EXTENSION IF NOT EXISTS unaccent;

CREATE TEXT SEARCH CONFIGURATION public.simple_unaccent (COPY = pg_catalog.simple);

ALTER TEXT SEARCH CONFIGURATION public.simple_unaccent
  ALTER MAPPING FOR hword, hword_part, word
  WITH unaccent, simple;

ALTER TABLE posts ADD COLUMN search_vector tsvector;

CREATE TRIGGER posts_search_vector_refresh
BEFORE INSERT OR UPDATE OF title, body ON posts
FOR EACH ROW EXECUTE FUNCTION
  tsvector_update_trigger(search_vector, 'public.simple_unaccent', title, body);
```

**`unaccent` + `pg_trgm` 用于国际化姓名搜索：**

```sql
ALTER TABLE users ADD COLUMN name_search text;

CREATE FUNCTION users_name_search_refresh()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  NEW.name_search := unaccent(coalesce(NEW.name, ''));
  RETURN NEW;
END;
$$;

CREATE TRIGGER users_name_search_refresh
BEFORE INSERT OR UPDATE OF name ON users
FOR EACH ROW EXECUTE FUNCTION users_name_search_refresh();

CREATE INDEX users_name_search_trgm_idx
  ON users USING GIN (name_search gin_trgm_ops);

SELECT id, name
FROM users
WHERE name_search % unaccent($1)
ORDER BY similarity(name_search, unaccent($1)) DESC
LIMIT 10;
```

触发器示例避免在生成列或索引表达式中直接使用 `unaccent()`，因为 PostgreSQL 的不可变性规则会受影响。如果你将 `unaccent()` 包装在自定义的不可变函数中，请在文档中说明这会带来升级或配置风险。

## 值得关注的扩展

**`pg_trgm`** 随大多数 PostgreSQL 发行版一起提供，但需要显式启用。它是 Postgres 中模糊字符串匹配的基石。

**`unaccent`** 在索引和查询前去除变音符。与 `pg_trgm` 和 FTS 配合使用时，对欧洲语言内容效果尤佳。随 PostgreSQL 捆绑。

**`pg_bigm`** 将三元组扩展为二元组（2 字符切片），在中文、日文、韩文等 CJK 语言上显著提升匹配效果，因为 `pg_trgm` 在这些语言上表现不佳。需单独安装，未随 PostgreSQL 捆绑。

**`pg_search`**（来自 [ParadeDB](https://www.paradedb.com/)）用基于 Tantivy 的 BM25 索引取代标准的 `GIN`/`tsvector` 组合。它提供 BM25 评分（通常优于 `ts_rank`）、FTS 查询内的模糊匹配、分面搜索，以及在大表上显著更快的索引速度。当标准 FTS 在排序或性能上出现瓶颈时，这是一个即插即用的升级路径。

```sql
-- pg_search: BM25 全文搜索并支持模糊匹配
CREATE INDEX posts_bm25_idx ON posts
  USING bm25 (id, title, body)
  WITH (key_field = 'id', text_fields = '{"title": {}, "body": {}}');

-- 使用 BM25 评分 + 模糊匹配的查询（捕获 “javascipt”）
SELECT id, title, paradedb.score(id) AS rank
FROM posts
WHERE posts @@@ paradedb.fuzzy_phrase(field => 'title', value => 'postgres performnce')
ORDER BY rank DESC
LIMIT 10;
```

**`pgvector`** 增加了密集向量存储和相似度搜索。当用户用描述而非明确名称来表达需求时——语义搜索、RAG、相关内容推荐、多语言查询——它是合适的工具。详见 [语义向量搜索与混合策略](/semantic-vector-search-landscape)（已加 `../` 前缀）。

## 决策表

| 查询对象 | 推荐方案 |
|---|---|
| 文章、文档、工单正文 | FTS |
| 人名/公司名带拼写错误 | `pg_trgm` |
| 自动完成、前缀搜索 | `pg_trgm` |
| 短码、标识符 | `pg_trgm` |
| 日志关键字 | FTS |
| 国际化姓名 | `pg_trgm` + `unaccent` |
| 大块内容、需要更好排序 | `pg_search`（ParadeDB BM25） |
| 主键、精确邮箱、ID | B‑tree 索引 |
| 日期、范围、已排序列表 | B‑tree 索引 |
| 权限、分类、过滤条件 | 常规 WHERE 子句 |
| 问题、同义句、概念 | pgvector（见下一篇） |

不确定时：拼写有变体的短字符串 → 使用 trigram。关键词查询的长篇正文 → 使用 FTS。结构化标识符 → 常规索引。概念或自然语言查询 → 使用 pgvector。

## 混合搜索：双信号，单排序

当查询 `"withRetry timeout errors"` 进入搜索框时，包含两类意图：用户已知的精确符号 (`withRetry`) 与概念性描述 (`timeout errors`)。单一原语无法同时覆盖两者。将 FTS 与向量搜索并行运行——随后使用 Reciprocal Rank Fusion（RRF）合并它们的排序列表即可。

RRF 对每个列表中的结果计分为 `1 / (60 + rank)`，并在所有列表上求和。常数 60 抑制了最高排名的优势，使得在两份列表中均排名第二的结果能够击败在一份列表中第一、另一份列表中缺失的结果。关键是，RRF 从不对原始分数做平均——FTS 排名和余弦距离是不同的度量，不能直接相加。

<figcaption>查询同时发往 FTS 与 pgvector，各自生成排序列表。RRF 根据每个文档在各列表中的位置打分并求和——最终呈现两种信号都认同的文档。</figcaption>

```sql
-- Hybrid search: FTS + pgvector merged with RRF
WITH fts AS (
  SELECT id, ts_rank(search_vector, query) AS score,
         ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM docs, to_tsquery('english', 'withRetry & timeout') query
  WHERE search_vector @@ query
  LIMIT 60
),
vec AS (
  SELECT id,
         ROW_NUMBER() OVER (ORDER BY embedding <=> $embedding) AS rank
  FROM docs
  ORDER BY embedding <=> $embedding
  LIMIT 60
)
SELECT COALESCE(fts.id, vec.id) AS id,
       (COALESCE(1.0 / (60 + fts.rank), 0) +
        COALESCE(1.0 / (60 + vec.rank), 0)) AS rrf_score
FROM fts FULL JOIN vec ON fts.id = vec.id
ORDER BY rrf_score DESC
LIMIT 10;
```

每个分支的 60 条候选文档（`LIMIT 60`）是常用的起点。召回率不足时可以扩大；追求速度时可以收窄。

## 接下来该做什么

PostgreSQL 文本搜索已经覆盖了大量场景，但它有上限。当用户用描述而不是明确的关键词来表达需求时——比如 “在航班上帮助我入睡的东西”、 “关于新工程师调试信心的文章”——词法搜索和 trigram 匹配都会失效。

这正是向量嵌入、语义搜索以及混合架构发挥作用的领域。相关内容请参阅 [语义向量搜索与混合策略](/semantic-vector-search-landscape)。
