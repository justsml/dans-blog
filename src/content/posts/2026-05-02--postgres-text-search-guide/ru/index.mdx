Большинство команд используют один инструмент поиска в Postgres. Команды, которые знают все три, реализуют лучший поиск с меньшей сложностью — и избегают затратного отвлечения на выделенный поисковый сервис, который им ещё не нужен.

Это руководство охватывает весь набор нативных для Postgres вариантов: что каждый из них делает, когда он подходит и как их комбинировать.

---

## Три инструмента

**Полнотекстовый поиск** (`tsvector` / индекс GIN) лексический. Он токенизирует текст в леммы, стеммирует их и сопоставляет запросы с индексом. «Running» и «runs» сводятся к одной лемме. То же самое с «dog» и «dogs». Функция ранжирования (`ts_rank`) учитывает частоту появления терминов запроса и их заметность.

**Триграммы** (`pg_trgm`) разбивают строки на перекрывающиеся трёхсимвольные фрагменты и измеряют, сколько фрагментов две строки разделяют. «Dan» → `" da"`, `"dan"`, `"an "`. «Micheal» и «Michael» разделяют большинство триграмм, поэтому схожесть высока. Это делает `pg_trgm` отличным инструментом для нечёткого сопоставления имён, толерантности к опечаткам и автодополнения — там, где FTS работает плохо.

**Индексы точного совпадения** (B-tree, hash) обрабатывают первичные ключи, адреса электронной почты, ID, артикулы и всё, где ответ бинарный: совпадает или нет. Это не кажется «поиском», но они принадлежат этой дискуссии, потому что худшая практика — использовать нечёткий или семантический поиск для задач, имеющих правильный ответ.

Выбор не в sophistication. Это в сопоставлении инструмента с формой запроса.

## Когда побеждает полнотекстовый поиск

**Поиск прозы по ключевым словам.** Блог-посты, документация, описания продуктов, тикеты поддержки, юридические документы. FTS создан для этой формы контента: индексированное, ранжированное извлечение по естественному языку.

**Запросы пользователей на основе ключевых слов.** Пользователи вводят поисковый термин, фильтруют по тегу или просматривают по ключевому слову. FTS обрабатывает это намерение нативно, без любой инфраструктуры встраиваний.

**Результаты ранжирования без внешних зависимостей.** Индексы FTS быстры, детерминированы и не требуют вызовов API. Сигнал важности приходит от частоты терминов, взвешенной по позиции поля.

**Булево фильтрация вместе с поиском.** FTS естественно компонуется с вашей существующей поисковой логикой:

```sql
SELECT * FROM posts
WHERE search_vector @@ to_tsquery('english', 'postgres & performance')
  AND category = 'tutorial'
  AND published_at > NOW() - INTERVAL '6 месяцев';
```

### Настройка FTS

```sql
-- Генерируемая колонка держит индекс актуальным автоматически
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body,  '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Запрос
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`setweight` присваивает важность: `A` (заголовок) перевешивает `B` (тело). Это вся модель важности для большинства задач поиска контента.

### Что FTS плохо обрабатывает

- Опечатки в запросах — «javascipt» не совпадёт с «javascript»
- Имена людей, адреса, собственные имена, которые не стеммируются предсказуемо
- Префикс / автодополнение без специальной конфигурации
- Запросы, где пользователь описывает концепцию, а не называет её

## Когда побеждают триграммы (`pg_trgm`)

`pg_trgm` покрывает неловкую середину, которую FTS последовательно проваливает.

FTS токенизирует текст в леммы и стеммирует их. Для прозы это правильно. Для имён и коротких идентификаторов это часто неправильно:

- Имена людей («Dan Levy» → стеммируется по-разному в зависимости от словаря и конфигурации языка)
- Имена компаний, адреса, заголовки продуктов, где точное написание важно
- Запросы с опечатками — «Micheal Jordan», «Amaon», «javascipt»
- Автодополнение / поиск по префиксу
- Частичное сопоставление строк («son» совпадает с «Johnson», «Anderson»)

`pg_trgm` также не зависит от языка, что важно для имён из разнообразных лингвистических фонов. FTS требует конфигурации словаря для каждого языка.

### Поиск по имени с опечатками

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- Находит «Micheal Jordan» при поиске «Michael Jordan»
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- оператор % = порог схожести (по умолчанию 0.3)
ORDER BY score DESC
LIMIT 10;
```

Оператор `%` использует `pg_trgm.similarity_threshold` (по умолчанию 0.3, диапазон 0–1). Для поиска по имени 0.3–0.4 ловит опечатки, сохраняя шум низким.

### Автодополнение, префикс и поиск «содержит»

```sql
-- Совпадение по префиксу для автодополнения. Индекс GIN триграмм может помочь,
-- но для чистых левых анкерных префиксов лучше может подойти индекс паттерна B-tree.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- word_similarity для частичных совпадений внутри более длинных строк
-- («Johnson» внутри «Andrew Johnson III»)
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name
ORDER BY score DESC
LIMIT 10;
```

Индекс GIN триграмм особенно полезен для запросов `ILIKE '%pattern%'` «содержит» и толерантного к опечаткам сопоставления — паттернов, которые обычно являются полным сканированием таблицы без индекса триграмм.

### Когда обращаться к pg_trgm вместо FTS

| Сценарий | Использовать |
|---|---|
| Поиск по именам людей/компаний с опечатками | `pg_trgm` |
| Автодополнение / поиск по префиксу | `pg_trgm` (или FTS с префикс-запросами) |
| Короткие строки, идентификаторы, коды | `pg_trgm` |
| Прозаические статьи, документация, тикеты | FTS |
| Сообщения логов для ключевых слов | FTS |
| Многоязычный поиск по именам | `pg_trgm` (не зависит от языка) |

## Когда побеждает точное совпадение SQL

Некоторые задачи «поиска» вообще не являются поиском.

«Найти пользователя с email `dan@example.com`» — проверка равенства. «Найти заказ `ORD-12345`» — lookup первичного ключа. «Список постов в категории `tutorial`, отсортированный по дате» — фильтрованный запрос. Эти задачи принадлежат индексам B-tree или hash.

Использование FTS или триграмм здесь добавляет сложность без улучшения правильности — и для точных идентификаторов близкое совпадение хуже, чем отсутствие совпадения.

```sql
CREATE INDEX users_email_idx ON users (email);

-- Точный lookup: быстро и однозначно
SELECT id, name FROM users WHERE email = $1;
```

Более широкий урок: приближённый поиск для задач с правильными ответами — это ошибка категории. Он возвращает *что-то* — что может быть уверенно неверным.

## Комбинирование этих инструментов

Эти инструменты чистимо компонуется. Вам не нужно выбирать ровно один.

**FTS + pg_trgm для поисковой строки, толерантной к опечаткам в ключевых словах:**

```sql
-- Схожесть триграмм по заголовку ловит опечатки; ts_rank обрабатывает важность тела
SELECT id, title,
  ts_rank(search_vector, to_tsquery('simple', $1)) AS fts_rank,
  similarity(title, $1) AS trgm_score
FROM posts
WHERE search_vector @@ to_tsquery('simple', $1)
   OR title % $1
ORDER BY (ts_rank(search_vector, to_tsquery('simple', $1)) + similarity(title, $1)) DESC
LIMIT 10;
```

**FTS + `unaccent` для международного контента:**

```sql
-- Убирайте диакритические знаки, чтобы «José» совпадал с «Jose»
CREATE EXTENSION IF NOT EXISTS unaccent;

CREATE TEXT SEARCH CONFIGURATION public.simple_unaccent (COPY = pg_catalog.simple);

ALTER TEXT SEARCH CONFIGURATION public.simple_unaccent
  ALTER MAPPING FOR hword, hword_part, word
  WITH unaccent, simple;

ALTER TABLE posts ADD COLUMN search_vector tsvector;

CREATE TRIGGER posts_search_vector_refresh
BEFORE INSERT OR UPDATE OF title, body ON posts
FOR EACH ROW EXECUTE FUNCTION
  tsvector_update_trigger(search_vector, 'public.simple_unaccent', title, body);
```

**`unaccent` + `pg_trgm` для международного поиска по именам:**

```sql
ALTER TABLE users ADD COLUMN name_search text;

CREATE FUNCTION users_name_search_refresh()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  NEW.name_search := unaccent(coalesce(NEW.name, ''));
  RETURN NEW;
END;
$$;

CREATE TRIGGER users_name_search_refresh
BEFORE INSERT OR UPDATE OF name ON users
FOR EACH ROW EXECUTE FUNCTION users_name_search_refresh();

CREATE INDEX users_name_search_trgm_idx
  ON users USING GIN (name_search gin_trgm_ops);

SELECT id, name
FROM users
WHERE name_search % unaccent($1)
ORDER BY similarity(name_search, unaccent($1)) DESC
LIMIT 10;
```

Примеры с триггерами избегают использования `unaccent()` внутри выражений генерируемой колонки или индекса, где важны правила неизменности PostgreSQL. Если вы оборачиваете `unaccent()` в свою собственную неизменную функцию, документируйте, что принимаете риск обновления/конфигурации.

## Примечательные расширения

**`pg_trgm`** входит в большинство дистрибутивов Postgres, но требует явного включения. Основа для нечёткого сопоставления строк в Postgres.

**`unaccent`** убирает диакритические знаки перед индексацией и запросами. Хорошо сочетается с обоими, `pg_trgm` и FTS для европейско-язычного контента. Входит в Postgres.

**`pg_bigm`** расширяет подход триграмм до биграмм (двухсимвольных фрагментов), что значительно улучшает результаты для языков CJK (китайский, японский, корейский), где `pg_trgm` работает недостаточно хорошо. Должен быть установлен отдельно; не входит в состав.

**`pg_search`** (от [ParadeDB](https://www.paradedb.com/)) заменяет стандартный стек `GIN` / `tsvector` на индекс BM25 на основе Tantivy. Это даёт вам оценку BM25 (часто лучшую, чем `ts_rank`), нечёткое совпадение внутри запросов FTS, фасетированный поиск и драматически более быструю индексацию на больших таблицах. Это путь плавного обновления, когда стандартный FTS начинает показывать пределы ранжирования или производительности.

```sql
-- pg_search: BM25 полнотекстовый поиск с нечётким совпадением
CREATE INDEX posts_bm25_idx ON posts
  USING bm25 (id, title, body)
  WITH (key_field = 'id', text_fields = '{"title": {}, "body": {}}');

-- Запрос с оценкой BM25 + нечёткое совпадение (ловит «javascipt»)
SELECT id, title, paradedb.score(id) AS rank
FROM posts
WHERE posts @@@ paradedb.fuzzy_phrase(field => 'title', value => 'postgres performnce')
ORDER BY rank DESC
LIMIT 10;
```

**`pgvector`** добавляет хранение плотных векторов и поиск по схожести. Это правильный инструмент, когда пользователи описывают, что хотят, а не называют это — семантический поиск, RAG, рекомендации связанного контента, многоязычные запросы. Подробно описано в [Semantic Vector Search and Hybrid Strategies](/semantic-vector-search-landscape).

| Разрешения, категории, фильтры | Обычный WHERE clause |
| Вопросы, парафразы, концепции | pgvector (смотрите следующую статью) |

Когда сомневаетесь: короткие строки с вариациями правописания → триграммы. Длинная проза для запросов по ключевым словам → FTS. Структурированные идентификаторы → обычные индексы. Концептуальные или естественные языковые запросы → pgvector.

## Гибридный поиск: два сигнала, один ранг

Когда запрос вроде `"withRetry timeout errors"` попадает в поисковую строку, он несёт два вида намерения: точные имена символов, которые пользователь знает (`withRetry`) и концептуальное описание (`timeout errors`). Ни один примитив не покрывает оба. Запуск FTS и векторного поиска параллельно — затем слияние их ранжированных списков с Reciprocal Rank Fusion — делает это.

RRF оценивает каждый результат как `1 / (60 + rank)` в каждом списке и суммирует по спискам. Константа 60 сглаживает преимущество верхних рангов, поэтому результат, который занимает второе место в обоих списках, может победить результат, который выигрывает один список и пропускает другой. Важно: RRF никогда не усредняет сырые оценки по методам — ранг FTS и косинусное расстояние — разные валюты и не могут быть комбинированы арифметически.

```sql
-- Гибридный поиск: FTS + pgvector, объединённые с RRF
WITH fts AS (
  SELECT id, ts_rank(search_vector, query) AS score,
         ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM docs, to_tsquery('english', 'withRetry & timeout') query
  WHERE search_vector @@ query
  LIMIT 60
),
vec AS (
  SELECT id,
         ROW_NUMBER() OVER (ORDER BY embedding <=> $embedding) AS rank
  FROM docs
  ORDER BY embedding <=> $embedding
  LIMIT 60
)
SELECT COALESCE(fts.id, vec.id) AS id,
       (COALESCE(1.0 / (60 + fts.rank), 0) +
        COALESCE(1.0 / (60 + vec.rank), 0)) AS rrf_score
FROM fts FULL JOIN vec ON fts.id = vec.id
ORDER BY rrf_score DESC
LIMIT 10;
```

Кандидатский пул из 60 документов на ветку (`LIMIT 60`) — общая отправная точка. Расширьте его, если recall низкий; сузьте для скорости.

## Что дальше

Поиск текста в Postgres охватывает много земли, но имеет потолок. Когда пользователи описывают, что хотят, а не называют это — «что-то, чтобы помочь мне уснуть в полёте», «статьи о дебаггинге уверенности как новому инженеру» — лексический и триграммный поиск оба проваливаются.

Это территория векторных встраиваний, семантического поиска и гибридных архитектур. Описано в [Semantic Vector Search and Hybrid Strategies](/semantic-vector-search-landscape).
