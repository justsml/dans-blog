La maggior parte dei team usa un solo strumento di ricerca in Postgres. I team che li conoscono tutti e tre offrono una ricerca migliore con meno complessità — ed evitano la costosa deviazione verso un servizio di ricerca dedicato di cui non avevano ancora bisogno.

Questa guida copre l'intero set di opzioni native di Postgres: cosa fa ciascuna, quando è la scelta giusta e come combinarle.

---

## I Tre Strumenti

**La ricerca full-text** (`tsvector` / indice `GIN`) è lessicale. Tokenizza il testo in lessemi, li riconduce alla radice e confronta le query con l'indice. "Correre" e "corre" collassano nello stesso lessemo. Così come "cane" e "cani". La funzione di ranking (`ts_rank`) premia i documenti in cui i termini della query compaiono spesso o in posizioni di rilievo.

**I trigrammi** (`pg_trgm`) suddividono le stringhe in slice sovrapposte di 3 caratteri e misurano quante slice due stringhe condividono. "Dan" → `" da"`, `"dan"`, `"an "`. "Micheal" e "Michael" condividono la maggior parte dei loro trigrammi, quindi la similarità è alta. Questo rende `pg_trgm` eccellente per il fuzzy matching di nomi, la tolleranza ai refusi e l'autocompletamento — lo spazio in cui il full-text si comporta male.

**Gli indici per corrispondenza esatta** (B-tree, hash) gestiscono chiavi primarie, indirizzi email, ID, SKU e tutto ciò per cui la risposta è binaria: corrisponde o non corrisponde. Questi non sembrano "ricerca", ma appartengono a questa conversazione perché il pattern peggiore è usare una ricerca fuzzy o semantica per problemi che hanno risposte corrette.

La scelta non riguarda la sofisticatezza. Riguarda l'abbinare lo strumento alla forma della query.

## Quando il Full-Text Search Vince

**Ricerca di parole chiave nella prosa.** Post di blog, documentazione, descrizioni di prodotti, ticket di supporto, documenti legali. FTS è stato progettato per questa forma di contenuto: recupero indicizzato e classificato su testo in linguaggio naturale.

**Query utente basate su parole chiave.** Gli utenti digitano un termine di ricerca, filtrano per tag o navigano per parola chiave. FTS gestisce quell'intento nativamente, senza infrastruttura di embedding.

**Risultati classificati senza dipendenze esterne.** Gli indici FTS sono veloci, deterministici e non richiedono chiamate API. Il segnale di rilevanza proviene dalla frequenza del termine ponderata per posizione nel campo.

**Filtraggio booleano insieme alla ricerca.** FTS si compone naturalmente con la logica di query esistente:

```sql
SELECT * FROM posts
WHERE search_vector @@ to_tsquery('italian', 'postgres & prestazioni')
  AND category = 'tutorial'
  AND published_at > NOW() - INTERVAL '6 months';
```

### Configurazione di FTS

```sql
-- La colonna generata mantiene l'indice aggiornato automaticamente
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('italian', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('italian', coalesce(body,  '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Query
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('italian', 'postgres & prestazioni') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`setweight` assegna l'importanza: `A` (titolo) supera `B` (corpo). Questo è l'intero modello di rilevanza per la maggior parte dei casi d'uso di ricerca di contenuti.

### Cosa FTS Non Gestisce Bene

- Refusi nelle query — "javascipt" non corrisponderà a "javascript"
- Nomi di persona, indirizzi, nomi propri che non si riconducono prevedibilmente alla radice
- Prefisso/autocompletamento senza configurazione speciale
- Query in cui l'utente descrive un concetto invece di nominarlo

## Quando Vincono i Trigrammi (`pg_trgm`)

`pg_trgm` copre quella zona intermedia scomoda che FTS sbaglia sistematicamente.

FTS tokenizza il testo in lessemi e li riconduce alla radice. Per la prosa è corretto. Per nomi e identificatori brevi spesso non lo è:

- Nomi di persona ("Dan Levy" → radice diversa a seconda del dizionario e della configurazione linguistica)
- Nomi di aziende, indirizzi, titoli di prodotto in cui l'ortografia esatta conta
- Query con refusi — "Micheal Jordan", "Amaon", "javascipt"
- Autocompletamento / ricerca con prefisso
- Corrispondenza parziale di stringhe ("son" che corrisponde a "Johnson", "Anderson")

`pg_trgm` è anche indipendente dalla lingua, il che conta per nomi provenienti da background linguistici diversi. FTS richiede la configurazione del dizionario per ogni lingua.

### Ricerca Fuzzy di Nomi

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- Trova "Micheal Jordan" quando si cerca "Michael Jordan"
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- operatore % = soglia di similarità (default 0.3)
ORDER BY score DESC
LIMIT 10;
```

L'operatore `%` usa `pg_trgm.similarity_threshold` (default 0.3, range 0–1). Per la ricerca di nomi, 0.3–0.4 cattura i refusi mantenendo basso il rumore.

### Autocompletamento, Prefisso e Ricerca per Contenuto

```sql
-- Corrispondenza con prefisso per autocompletamento. Un indice GIN di trigrammi può aiutare,
-- ma un indice B-tree pattern può essere migliore per prefissi puramente ancorati a sinistra.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- word_similarity per corrispondenze parziali all'interno di stringhe più lunghe
-- ("Johnson" dentro "Andrew Johnson III")
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name
ORDER BY score DESC
LIMIT 10;
```

L'indice GIN di trigrammi è particolarmente utile per query `ILIKE '%pattern%'` di contenuto e corrispondenza tollerante ai refusi — pattern che di solito sono scansioni complete di tabella senza un indice di trigrammi.

### Quando Scegliere pg_trgm invece di FTS

| Scenario | Usa |
|---|---|
| Ricerca di nomi di persona/azienda con refusi | `pg_trgm` |
| Autocompletamento / ricerca con prefisso | `pg_trgm` (o FTS con query di prefisso) |
| Stringhe brevi, identificatori, codici | `pg_trgm` |
| Articoli in prosa, documentazione, ticket | FTS |
| Messaggi di log per parole chiave | FTS |
| Ricerca di nomi multilingue | `pg_trgm` (indipendente dalla lingua) |

## Quando Vince la SQL per Corrispondenza Esatta

Alcuni problemi di "ricerca" non sono affatto ricerca.

"Trova l'utente con email `dan@example.com`" è un controllo di uguaglianza. "Trova l'ordine `ORD-12345`" è una ricerca per chiave primaria. "Elenca i post nella categoria `tutorial` ordinati per data" è una query filtrata. Questi appartengono a indici B-tree o hash.

Usare FTS o trigrammi qui aggiunge complessità senza migliorare la correttezza — e per identificatori esatti, una corrispondenza approssimativa è peggio di nessuna corrispondenza.

```sql
CREATE INDEX users_email_idx ON users (email);

-- Ricerca esatta: veloce e senza ambiguità
SELECT id, name FROM users WHERE email = $1;
```

La lezione più ampia: la ricerca approssimativa per problemi con risposte corrette è un errore categoriale. Restituisce *qualcosa* — che può essere sicuramente sbagliato.

## Combinare Questi Strumenti

Questi strumenti si compongono in modo pulito. Non devi sceglierne solo uno.

**FTS + pg_trgm per una casella di ricerca che tollera i refusi nelle parole chiave:**

```sql
-- La similarità di trigrammi sul titolo cattura i refusi; ts_rank gestisce la rilevanza del corpo
SELECT id, title,
  ts_rank(search_vector, to_tsquery('simple', $1)) AS fts_rank,
  similarity(title, $1) AS trgm_score
FROM posts
WHERE search_vector @@ to_tsquery('simple', $1)
   OR title % $1
ORDER BY (ts_rank(search_vector, to_tsquery('simple', $1)) + similarity(title, $1)) DESC
LIMIT 10;
```

**FTS + `unaccent` per contenuti internazionali:**

```sql
-- Rimuove i segni diacritici in modo che "José" corrisponda a "Jose"
CREATE EXTENSION IF NOT EXISTS unaccent;

CREATE TEXT SEARCH CONFIGURATION public.simple_unaccent (COPY = pg_catalog.simple);

ALTER TEXT SEARCH CONFIGURATION public.simple_unaccent
  ALTER MAPPING FOR hword, hword_part, word
  WITH unaccent, simple;

ALTER TABLE posts ADD COLUMN search_vector tsvector;

CREATE TRIGGER posts_search_vector_refresh
BEFORE INSERT OR UPDATE OF title, body ON posts
FOR EACH ROW EXECUTE FUNCTION
  tsvector_update_trigger(search_vector, 'public.simple_unaccent', title, body);
```

**`unaccent` + `pg_trgm` per la ricerca internazionale di nomi:**

```sql
ALTER TABLE users ADD COLUMN name_search text;

CREATE FUNCTION users_name_search_refresh()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  NEW.name_search := unaccent(coalesce(NEW.name, ''));
  RETURN NEW;
END;
$$;

CREATE TRIGGER users_name_search_refresh
BEFORE INSERT OR UPDATE OF name ON users
FOR EACH ROW EXECUTE FUNCTION users_name_search_refresh();

CREATE INDEX users_name_search_trgm_idx
  ON users USING GIN (name_search gin_trgm_ops);

SELECT id, name
FROM users
WHERE name_search % unaccent($1)
ORDER BY similarity(name_search, unaccent($1)) DESC
LIMIT 10;
```

Gli esempi con trigger evitano di usare `unaccent()` all'interno di espressioni di colonne generate o di indici, dove entrano in gioco le regole di immutabilità di PostgreSQL. Se racchiudi `unaccent()` in una tua funzione immutabile, documenta che stai accettando rischi di aggiornamento/configurazione.

## Estensioni Degne di Nota

**`pg_trgm`** è inclusa nella maggior parte delle distribuzioni Postgres ma richiede l'attivazione esplicita. La fondazione per il fuzzy matching di stringhe in Postgres.

**`unaccent`** rimuove i segni diacritici prima dell'indicizzazione e della query. Si abbina bene sia con `pg_trgm` che con FTS per contenuti in lingue europee. Inclusa con Postgres.

**`pg_bigm`** estende l'approccio dei trigrammi ai bigrammi (slice di 2 caratteri), migliorando significativamente i risultati per le lingue CJK (cinese, giapponese, coreano) dove `pg_trgm` dà risultati scadenti. Deve essere installata separatamente; non è inclusa.

**`pg_search`** (di [ParadeDB](https://www.paradedb.com/)) sostituisce lo stack standard `GIN` / `tsvector` con un indice BM25 basato su Tantivy. Offre punteggi BM25 (spesso migliori di `ts_rank`), fuzzy matching all'interno di query FTS, ricerca sfaccettata e indicizzazione drasticamente più veloce su tabelle grandi. È un percorso di aggiornamento drop-in quando il FTS standard inizia a mostrare limiti di ranking o prestazioni.

```sql
-- pg_search: ricerca full-text BM25 con fuzzy matching
CREATE INDEX posts_bm25_idx ON posts
  USING bm25 (id, title, body)
  WITH (key_field = 'id', text_fields = '{"title": {}, "body": {}}');

-- Query con punteggio BM25 + fuzzy matching (cattura "javascipt")
SELECT id, title, paradedb.score(id) AS rank
FROM posts
WHERE posts @@@ paradedb.fuzzy_phrase(field => 'title', value => 'postgres performnce')
ORDER BY rank DESC
LIMIT 10;
```

**`pgvector`** aggiunge l'archiviazione di vettori densi e la ricerca per similarità. È lo strumento giusto quando gli utenti descrivono ciò che vogliono invece di nominarlo — ricerca semantica, RAG, raccomandazioni di contenuti correlati, query multilingue. Trattato in dettaglio in [Ricerca Vettoriale Semantica e Strategie Ibride](/semantic-vector-search-landscape).

## Tabella delle Decisioni

| Cosa stai cercando | Consigliato |
|---|---|
| Articoli in prosa, documenti, ticket | FTS |
| Nomi di persona/azienda con refusi | `pg_trgm` |
| Autocompletamento, ricerca con prefisso | `pg_trgm` |
| Codici brevi, identificatori | `pg_trgm` |
| Messaggi di log per parole chiave | FTS |
| Nomi internazionali | `pg_trgm` + `unaccent` |
| Contenuti grandi, ranking migliore | `pg_search` (ParadeDB BM25) |
| Chiavi primarie, email esatte, ID | Indice B-tree |
| Date, intervalli, elenchi ordinati | Indice B-tree |
| Permessi, categorie, filtri | Normale clausola WHERE |
| Domande, parafrasi, concetti | pgvector (vedi il prossimo articolo) |

Nel dubbio: stringhe brevi con variazioni ortografiche → trigrammi. Prosa lunga per query di parole chiave → FTS. Identificatori strutturati → indici regolari. Query concettuali o in linguaggio naturale → pgvector.

## Ricerca Ibrida: Due Segnali, Un Solo Ranking

Quando una query come `"withRetry timeout errors"` colpisce una casella di ricerca, porta con sé due tipi di intento: nomi di simboli esatti che l'utente conosce (`withRetry`) e una descrizione concettuale (`timeout errors`). Nessun singolo primitivo copre entrambi. Eseguire FTS e la ricerca vettoriale in parallelo — e poi unire le loro liste classificate con Reciprocal Rank Fusion (RRF) — lo fa.

RRF assegna a ogni risultato un punteggio di `1 / (60 + rango)` in ciascuna lista e somma i punteggi tra le liste. La costante 60 attenua il vantaggio dei primi ranghi, così un risultato che si piazza secondo in entrambe le liste può battere un risultato che vince una lista ma manca completamente l'altra. Fondamentalmente, RRF non fa mai la media dei punteggi grezzi tra metodi diversi — il rango FTS e la distanza coseno sono valute diverse e non possono essere combinate aritmeticamente.

```sql
-- Ricerca ibrida: FTS + pgvector fusi con RRF
WITH fts AS (
  SELECT id, ts_rank(search_vector, query) AS score,
         ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM docs, to_tsquery('english', 'withRetry & timeout') query
  WHERE search_vector @@ query
  LIMIT 60
),
vec AS (
  SELECT id,
         ROW_NUMBER() OVER (ORDER BY embedding <=> $embedding) AS rank
  FROM docs
  ORDER BY embedding <=> $embedding
  LIMIT 60
)
SELECT COALESCE(fts.id, vec.id) AS id,
       (COALESCE(1.0 / (60 + fts.rank), 0) +
        COALESCE(1.0 / (60 + vec.rank), 0)) AS rrf_score
FROM fts FULL JOIN vec ON fts.id = vec.id
ORDER BY rrf_score DESC
LIMIT 10;
```

Il pool di 60 documenti candidati per ramo (`LIMIT 60`) è un punto di partenza comune. Allargalo se il recall è basso; restringilo per velocità.
