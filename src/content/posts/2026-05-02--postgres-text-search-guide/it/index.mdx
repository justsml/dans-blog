La maggior parte dei team usa un solo strumento di ricerca in Postgres. I team che conoscono tutti e tre realizzano una ricerca migliore con meno complessità — ed evitano la costosa deviazione verso un servizio di ricerca dedicato che non serviva ancora.

Questa guida copre tutte le opzioni native di Postgres: cosa fa ciascuno, quando è la scelta giusta e come combinarli.

---

## I Tre Strumenti

**La ricerca full-text** (`tsvector` / indice `GIN`) è lessicale. Tokenizza il testo in lexemi, li stemmatizza e corrisponde le query all'indice. "Running" e "runs" collassano nello stesso lexema. Lo fanno anche "dog" e "dogs". La funzione di ranking (`ts_rank`) favorisce i documenti dove i termini della query appaiono spesso o in modo prominente.

**I trigrammi** (`pg_trgm`) suddividono le stringhe in sovrapposizioni di 3 caratteri e misurano quanti trigrammi due stringhe condividono. "Dan" → `" da"`, `"dan"`, `"an "`. "Micheal" e "Michael" condividono la maggior parte dei trigrammi, quindi la similarità è alta. Questo rende `pg_trgm` eccellente per la corrispondenza fuzzy dei nomi, la tolleranza agli errori di battitura e l'autocompletamento — lo spazio dove FTS performa male.

**Gli indici di corrispondenza esatta** (B-tree, hash) gestiscono chiavi primarie, indirizzi email, ID, SKU e tutto dove la risposta è binaria: corrisponde o non corrisponde. Questi non sembrano "ricerca", ma appartengono a questa conversazione perché il pattern peggiore è usare la ricerca fuzzy o semantica per problemi che hanno risposte corrette.

La scelta non riguarda la sofisticazione. Si tratta di abbinare lo strumento alla forma della query.

## Quando Vince la Ricerca Full-Text

**Cercare prosa per parole chiave.** Blog post, documentazione, descrizioni di prodotti, ticket di supporto, documenti legali. FTS è stato progettato per questa forma di contenuto: recupero indicizzato e rankizzato su testo di linguaggio naturale.

**Query utente basate su parole chiave.** Gli utenti digitano un termine di ricerca, filtrano per tag o navigano per parola chiave. FTS gestisce quell'intento nativamente senza alcuna infrastruttura di embedding.

**Risultati rankizzati senza dipendenze esterne.** Gli indici FTS sono veloci, deterministici e non richiedono chiamate API. Il segnale di rilevanza viene dalla frequenza dei termini pesata per posizione di campo.

**Filtraggio booleano accanto alla ricerca.** FTS si compone naturalmente con la tua logica di query esistente:

```sql
SELECT * FROM posts
WHERE search_vector @@ to_tsquery('english', 'postgres & performance')
  AND category = 'tutorial'
  AND published_at > NOW() - INTERVAL '6 months';
```

### Configurazione FTS

```sql
-- Colonna generata mantiene l'indice aggiornato automaticamente
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body,  '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Query
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`setweight` assegna importanza: `A` (title) supera `B` (body). Questo è l'intero modello di rilevanza per la maggior parte dei casi d'uso di ricerca di contenuti.

### Cosa FTS Non Gestisce Bene

- Errori di battitura nelle query — "javascipt" non corrisponderà a "javascript"
- Nomi di persone, indirizzi, nomi propri che non stemmatizzano in modo prevedibile
- Prefix/autocompletamento senza configurazione speciale
- Query dove l'utente descrive un concetto invece di nominarlo

## Quando Vincono i Trigrammi (`pg_trgm`)

`pg_trgm` copre la zona imbarazzante che FTS fallisce sistematicamente.

FTS tokenizza il testo in lexemi e li stemmatizza. Per la prosa questo è corretto. Per nomi e identificatori brevi spesso non lo è:

- Nomi di persone ("Dan Levy" → stemmatizzato diversamente a seconda di dizionario e configurazione linguistica)
- Nomi di aziende, indirizzi, titoli di prodotti dove lo spelling esatto conta
- Query con errori di battitura — "Micheal Jordan", "Amaon", "javascipt"
- Autocompletamento / ricerca prefix
- Corrispondenza parziale di stringhe ("son" che corrisponde a "Johnson", "Anderson")

`pg_trgm` è anche language-agnostic, il che conta per nomi da background linguistici diversificati. FTS richiede configurazione di dizionario per ogni lingua.

### Ricerca Fuzzy Nomi

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- Trova "Micheal Jordan" quando cerchi "Michael Jordan"
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- operatore % = soglia di similarità (default 0.3)
ORDER BY score DESC
LIMIT 10;
```

L'operatore `%` usa `pg_trgm.similarity_threshold` (default 0.3, range 0–1). Per la ricerca nomi, 0.3–0.4 cattura errori di battitura mantenendo il rumore basso.

### Autocompletamento, Prefix e Cerca Contiene

```sql
-- Corrispondenza prefix per autocompletamento. Un indice GIN trigram può aiutare,
-- ma un indice B-tree pattern può essere meglio per prefix left-ancorati puri.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- word_similarity per corrispondenze parziali all'interno di stringhe più lunghe
-- ("Johnson" all'interno di "Andrew Johnson III")
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name
ORDER BY score DESC
LIMIT 10;
```

L'indice GIN trigram è specialmente utile per query `ILIKE '%pattern%'` contiene e corrispondenza tollerante errori di battitura — pattern che sono solitamente scansioni full-table senza indice trigram.

### Quando Usare pg_trgm Invece di FTS

| Scenario | Usa |
|---|---|
| Ricerca nomi di persone/aziende con errori di battitura | `pg_trgm` |
| Autocompletamento / ricerca prefix | `pg_trgm` (o FTS con query prefix) |
| Stringhe brevi, identificatori, codici | `pg_trgm` |
| Articoli prosa, documentazione, ticket | FTS |
| Messaggi log per parole chiave | FTS |
| Ricerca nomi multilingue | `pg_trgm` (language-agnostic) |

## Quando Vince la SQL di Corrispondenza Esatta

Alcuni problemi di "ricerca" non sono ricerca affatto.

"Trova l'utente con email `dan@example.com`" è un controllo di uguaglianza. "Trova ordine `ORD-12345`" è una lookup di primary key. "Lista post nella categoria `tutorial` ordinati per data" è una query filtrata. Questi appartengono a indici B-tree o hash.

Usare FTS o trigram qui aggiunge complessità senza migliorare la correttezza — e per identificatori esatti, un near-match è peggio di nessun match.

```sql
CREATE INDEX users_email_idx ON users (email);

-- Lookup esatto: veloce e ambiguo
SELECT id, name FROM users WHERE email = $1;
```

La lezione più ampia: ricerca approssimativa per problemi con risposte corrette è un errore di categoria. Restituisce *qualcosa* — che può essere confidentemente sbagliato.

## Combinare Questi Strumenti

Questi strumenti si compongono pulitamente. Non scegli esattamente uno.

**FTS + pg_trgm per una casella di ricerca che tollera errori di battitura nelle parole chiave:**

```sql
-- Similarità trigram sul titolo cattura errori di battitura; ts_rank gestisce rilevanza body
SELECT id, title,
  ts_rank(search_vector, to_tsquery('simple', $1)) AS fts_rank,
  similarity(title, $1) AS trgm_score
FROM posts
WHERE search_vector @@ to_tsquery('simple', $1)
   OR title % $1
ORDER BY (ts_rank(search_vector, to_tsquery('simple', $1)) + similarity(title, $1)) DESC
LIMIT 10;
```

**FTS + `unaccent` per contenuti internazionali:**

```sql
-- Rimuovi segni diacritici così "José" corrisponde a "Jose"
CREATE EXTENSION IF NOT EXISTS unaccent;

CREATE TEXT SEARCH CONFIGURATION public.simple_unaccent (COPY = pg_catalog.simple);

ALTER TEXT SEARCH CONFIGURATION public.simple_unaccent
  ALTER MAPPING FOR hword, hword_part, word
  WITH unaccent, simple;

ALTER TABLE posts ADD COLUMN search_vector tsvector;

CREATE TRIGGER posts_search_vector_refresh
BEFORE INSERT OR UPDATE OF title, body ON posts
FOR EACH ROW EXECUTE FUNCTION
  tsvector_update_trigger(search_vector, 'public.simple_unaccent', title, body);
```

**`unaccent` + `pg_trgm` per ricerca nomi internazionali:**

```sql
ALTER TABLE users ADD COLUMN name_search text;

CREATE FUNCTION users_name_search_refresh()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  NEW.name_search := unaccent(coalesce(NEW.name, ''));
  RETURN NEW;
END;
$$;

CREATE TRIGGER users_name_search_refresh
BEFORE INSERT OR UPDATE OF name ON users
FOR EACH ROW EXECUTE FUNCTION users_name_search_refresh();

CREATE INDEX users_name_search_trgm_idx
  ON users USING GIN (name_search gin_trgm_ops);

SELECT id, name
FROM users
WHERE name_search % unaccent($1)
ORDER BY similarity(name_search, unaccent($1)) DESC
LIMIT 10;
```

Gli esempi di trigger evitano di usare `unaccent()` all'interno di generated-column o espressioni di indice, dove le regole di immutabilità di PostgreSQL contano. Se avvolgi `unaccent()` nella tua funzione immutable, documenta che stai accettando rischio di upgrade/configurazione.

## Estensioni Notevoli

**`pg_trgm`** è bundled con la maggior parte delle distribuzioni Postgres ma richiede abilitazione esplicita. Il fondamento per corrispondenza stringa fuzzy in Postgres.

**`unaccent`** rimuove segni diacritici prima di indicizzare e query. Si abbina bene sia con `pg_trgm` che FTS per contenuti europei. Bundled con Postgres.

**`pg_bigm`** estende l'approccio trigram a bigram (slice di 2 caratteri), che migliora significativamente risultati per lingue CJK (Cinese, Giapponese, Coreano) dove `pg_trgm` performa male. Deve essere installato separatamente; non bundled.

**`pg_search`** (da [ParadeDB](https://www.paradedb.com/)) sostituisce lo stack standard `GIN` / `tsvector` con un indice BM25 basato su Tantivy. Questo ti dà scoring BM25 (spesso migliore di `ts_rank`), corrispondenza fuzzy all'interno query FTS, search facetato e indicizzazione drammaticamente più veloce su tabelle grandi. È un percorso di upgrade drop-in quando FTS standard inizia a mostrare limiti di ranking o performance.

```sql
-- pg_search: BM25 full-text search con corrispondenza fuzzy
CREATE INDEX posts_bm25_idx ON posts
  USING bm25 (id, title, body)
  WITH (key_field = 'id', text_fields = '{"title": {}, "body": {}}');

-- Query con scoring BM25 + corrispondenza fuzzy (cattura "javascipt")
SELECT id, title, paradedb.score(id) AS rank
FROM posts
WHERE posts @@@ paradedb.fuzzy_phrase(field => 'title', value => 'postgres performnce')
ORDER BY rank DESC
LIMIT 10;
```

**`pgvector`** aggiunge storage vettoriale denso e search di similarità. È lo strumento giusto quando utenti descrivono cosa vogliono invece di nominarlo — ricerca semantica, RAG, raccomandazioni contenuti correlati, query multilingue. Coperto in profondità in [Ricerca Vettoriale Semantica e Strategie Ibride](/semantic-vector-search-landscape).

## Tabella Decisionale

| Cosa stai cercando | Raccomandato |
|---|---|
| Articoli prosa, docs, ticket | FTS |
| Nomi di persone/aziende con errori di battitura | `pg_trgm` |
| Autocompletamento, ricerca prefix | `pg_trgm` |
| Codici brevi, identificatori | `pg_trgm` |
| Messaggi log per parole chiave | FTS |
| Nomi internazionali | `pg_trgm` + `unaccent` |
| Contenuti grandi, ranking migliore | `pg_search` (BM25 ParadeDB) |
| Chiavi primarie, email esatte, ID | Indice B-tree |
| Date, range, liste ordinate | Indice B-tree |
| Permessi, categorie, filtri | WHERE clause regolare |
| Domande, parafrasi, concetti | pgvector (vedi prossimo articolo) |

Quando sei nel dubbio: stringhe brevi con variazione spelling → trigram. Prosa lunga per query keyword → FTS. Identificatori strutturati → indici regolari. Query concettuali o di linguaggio naturale → pgvector.

## Ricerca Ibrida: Due Segnali, Un Rank

Quando una query come `"withRetry timeout errors"` colpisce una casella di ricerca, porta due tipi di intento: nomi di simboli esatti che l'utente conosce (`withRetry`) e una descrizione concettuale (`timeout errors`). Nessun singolo primitivo copre entrambi. Esegui FTS e vector search in parallelo — poi fusi i loro ranking con Reciprocal Rank Fusion — fa.

RRF scores ogni risultato come `1 / (60 + rank)` in ogni lista e somma attraverso le liste. La costante 60 smorza il vantaggio dei top ranking, così un risultato che si posiziona secondo in entrambe le liste può battere un risultato che vince una lista e manca l'altra completamente. Crucialmente, RRF non media mai raw scores attraverso metodi — FTS rank e cosine distance sono valute diverse e non possono essere combinati aritmeticamente.

```sql
-- Hybrid search: FTS + pgvector fusi con RRF
WITH fts AS (
  SELECT id, ts_rank(search_vector, query) AS score,
         ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM docs, to_tsquery('english', 'withRetry & timeout') query
  WHERE search_vector @@ query
  LIMIT 60
),
vec AS (
  SELECT id,
         ROW_NUMBER() OVER (ORDER BY embedding <=> $embedding) AS rank
  FROM docs
  ORDER BY embedding <=> $embedding
  LIMIT 60
)
SELECT COALESCE(fts.id, vec.id) AS id,
       (COALESCE(1.0 / (60 + fts.rank), 0) +
        COALESCE(1.0 / (60 + vec.rank), 0)) AS rrf_score
FROM fts FULL JOIN vec ON fts.id = vec.id
ORDER BY rrf_score DESC
LIMIT 10;
```

Il pool di candidati di 60 documenti per branch (`LIMIT 60`) è un punto di partenza comune. Allargalo se recall è basso; restringilo per velocità.
