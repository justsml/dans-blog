ज़्यादातर टीमें सिर्फ़ एक पोस्टग्रेस सर्च टूल इस्तेमाल करती हैं। जो टीमें तीनों को जानती हैं, वे कम जटिलता के साथ बेहतर सर्च बनाती हैं — और किसी अलग सर्च सर्विस की महँगी अनावश्यक यात्रा से बचती हैं।

यह गाइड पोस्टग्रेस-नेटिव विकल्पों की पूरी रेंज को कवर करता है: हर एक क्या करता है, कब यह सही फ़िट है, और इन्हें कैसे परतबद्ध किया जाए।

---

## तीन टूल

**फुल-टेक्स्ट सर्च** (`tsvector` / `GIN` इंडेक्स) शाब्दिक (lexical) है। यह टेक्स्ट को लेक्सीम में टोकनाइज़ करता है, उन्हें स्टेम करता है, और क्वेरी को इंडेक्स के विरुद्ध मैच करता है। "Running" और "runs" एक ही लेक्सीम में समाहित हो जाते हैं। "dog" और "dogs" भी। रैंकिंग फ़ंक्शन (`ts_rank`) उन दस्तावेज़ों को पुरस्कृत करता है जहाँ क्वेरी शब्द अक्सर या प्रमुखता से दिखाई देते हैं।

**ट्राइग्राम** (`pg_trgm`) स्ट्रिंग को ओवरलैपिंग 3-अक्षरों के स्लाइस में तोड़ते हैं और मापते हैं कि दो स्ट्रिंग्स कितने स्लाइस साझा करती हैं। "Dan" → `" da"`, `"dan"`, `"an "`। "Micheal" और "Michael" अपने ज़्यादातर ट्राइग्राम साझा करते हैं, इसलिए समानता अधिक होती है। इस कारण `pg_trgm` फजी नाम मिलान, टाइपो सहनशीलता, और ऑटोकंप्लीट के लिए उत्कृष्ट है — वह क्षेत्र जहाँ FTS कमज़ोर होता है।

**एक्जैक्ट-मैच इंडेक्स** (B-tree, hash) प्राइमरी की, ईमेल पते, आईडी, SKU, और कुछ भी संभालते हैं जहाँ जवाब बाइनरी है: यह मैच करता है या नहीं। ये "सर्च" जैसे महसूस नहीं होते, लेकिन ये इस बातचीत में इसलिए हैं क्योंकि सबसे खराब पैटर्न उन समस्याओं के लिए फजी या सेमांटिक सर्च का उपयोग करना है जिनके सही जवाब होते हैं।

यह विकल्प परिष्करण के बारे में नहीं है। यह क्वेरी के आकार के अनुसार टूल मिलाने के बारे में है।

## जब फुल-टेक्स्ट सर्च बेहतर हो

**कीवर्ड के लिए गद्य में खोजना।** ब्लॉग पोस्ट, दस्तावेज़ीकरण, उत्पाद विवरण, सपोर्ट टिकट, कानूनी दस्तावेज़। FTS इस कंटेंट के आकार के लिए डिज़ाइन किया गया था: प्राकृतिक भाषा टेक्स्ट पर इंडेक्स्ड, रैंक्ड पुनर्प्राप्ति।

**कीवर्ड-आधारी उपयोगकर्ता क्वेरीज़।** उपयोगकर्ता एक सर्च शब्द टाइप करते हैं, टैग द्वारा फ़िल्टर करते हैं, या कीवर्ड द्वारा ब्राउज़ करते हैं। FTS इस इंटेंट को किसी भी एम्बेडिंग इन्फ्रास्ट्रक्चर के बिना स्वाभाविक रूप से संभालता है।

**बाहरी निर्भरता के बिना रैंक्ड परिणाम।** FTS इंडेक्स तेज़, निश्चित हैं, और उन्हें कोई API कॉल की आवश्यकता नहीं होती। प्रासंगिकता सिग्नल फ़ील्ड स्थिति द्वारा भारित शब्द आवृत्ति से आता है।

**सर्च के साथ बूलियन फ़िल्टरिंग।** FTS आपकी मौजूदा क्वेरी लॉजिक के साथ स्वाभाविक रूप से संयोजित होता है:

```sql
SELECT * FROM posts
WHERE search_vector @@ to_tsquery('english', 'postgres & performance')
  AND category = 'tutorial'
  AND published_at > NOW() - INTERVAL '6 months';
```

### FTS सेटअप करना

```sql
-- जनरेटेड कॉलम इंडेक्स को स्वचालित रूप से अप-टू-डेट रखता है
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body,  '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- क्वेरी
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`setweight` महत्व निर्धारित करता है: `A` (शीर्षक) `B` (बॉडी) से ऊपर होता है। यह अधिकांश कंटेंट-सर्च उपयोग मामलों के लिए पूरा प्रासंगिकता मॉडल है।

### FTS क्या अच्छी तरह नहीं संभालता

- क्वेरी में टाइपो — "javascipt" "javascript" से मैच नहीं करेगा
- व्यक्ति के नाम, पते, ऐसे संज्ञा जो पूर्वानुमेय रूप से स्टेम नहीं होते
- विशेष कॉन्फ़िगरेशन के बिना प्रीफिक्स/ऑटोकंप्लीट
- क्वेरीज़ जहाँ उपयोगकर्ता एक अवधारणा का वर्णन करता है बजाय इसे नाम देने के

## जब ट्राइग्राम बेहतर हों (`pg_trgm`)

`pg_trgm` उस अजीब मध्य को कवर करता है जहाँ FTS लगातार फिसलता है।

FTS टेक्स्ट को लेक्सीम में टोकनाइज़ करता है और उन्हें स्टेम करता है। गद्य के लिए यह सही है। नामों और छोटे पहचानकर्ताओं के लिए अक्सर नहीं:

- व्यक्ति के नाम ("Dan Levy" → शब्दकोश और भाषा कॉन्फ़िगरेशन के आधार पर अलग-अलग स्टेम होते हैं)
- कंपनी के नाम, पते, उत्पाद शीर्षक जहाँ सटीक वर्तनी मायने रखती है
- टाइपो वाली क्वेरीज़ — "Micheal Jordan", "Amaon", "javascipt"
- ऑटोकंप्लीट / प्रीफिक्स सर्च
- आंशिक स्ट्रिंग मिलान ("son" का "Johnson", "Anderson" से मिलान)

`pg_trgm` भाषा-agnostic भी है, जो विविध भाषाई पृष्ठभूमि वाले नामों के लिए मायने रखता है। FTS को प्रति भाषा शब्दकोश कॉन्फ़िगरेशन की आवश्यकता होती है।

### फजी नाम सर्च

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- "Michael Jordan" खोजने पर "Micheal Jordan" ढूंढता है
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % ऑपरेटर = समानता सीमा (डिफ़ॉल्ट 0.3)
ORDER BY score DESC
LIMIT 10;
```

`%` ऑपरेटर `pg_trgm.similarity_threshold` (डिफ़ॉल्ट 0.3, रेंज 0–1) का उपयोग करता है। नाम सर्च के लिए, 0.3–0.4 टाइपो पकड़ता है जबकि शोर कम रखता है।

### ऑटोकंप्लीट, प्रीफिक्स, और कंटेंस सर्च

```sql
-- ऑटोकंप्लीट के लिए प्रीफिक्स मिलान। एक ट्राइग्राम GIN इंडेक्स मदद कर सकता है,
-- लेकिन शुद्ध लेफ्ट-एंकर प्रीफिक्स के लिए B-tree पैटर्न इंडेक्स बेहतर हो सकता है।
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- लंबी स्ट्रिंग्स के भीतर आंशिक मैच के लिए word_similarity
-- ("Johnson" "Andrew Johnson III" के भीतर)
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name
ORDER BY score DESC
LIMIT 10;
```

ट्राइग्राम GIN इंडेक्स विशेष रूप से `ILIKE '%pattern%'` कंटेंस क्वेरीज़ और टाइपो-सहिष्णु मिलान के लिए उपयोगी है — वे पैटर्न जो ट्राइग्राम इंडेक्स के बिना आमतौर पर पूरी तालिका स्कैन होते हैं।

### कब pg_trgm, FTS से बेहतर विकल्प है

| परिदृश्य | उपयोग करें |
|---|---|
| टाइपो वाले व्यक्ति/कंपनी नाम सर्च | `pg_trgm` |
| ऑटोकंप्लीट / प्रीफिक्स सर्च | `pg_trgm` (या प्रीफिक्स क्वेरीज़ के साथ FTS) |
| छोटी स्ट्रिंग्स, पहचानकर्ता, कोड | `pg_trgm` |
| गद्य लेख, दस्तावेज़ीकरण, टिकट | FTS |
| कीवर्ड के लिए लॉग संदेश | FTS |
| बहुभाषी नाम सर्च | `pg_trgm` (भाषा-agnostic) |

## जब एक्जैक्ट-मैच SQL बेहतर हो

कुछ "सर्च" समस्याएँ वास्तव में सर्च नहीं हैं।

"ईमेल `dan@example.com` वाले उपयोगकर्ता को ढूंढें" एक समानता जांच है। "ऑर्डर `ORD-12345` ढूंढें" एक प्राइमरी की लुकअप है। "`tutorial` श्रेणी में पोस्ट दिनांक के अनुसार सूचीबद्ध करें" एक फ़िल्टर्ड क्वेरी है। ये B-tree या hash इंडेक्स पर हैं।

यहाँ FTS या ट्राइग्राम का उपयोग करने से सही होने में सुधार के बिना जटिलता बढ़ती है — और सटीक पहचानकर्ताओं के लिए, एक निकट-मैच किसी मैच से बदतर है।

```sql
CREATE INDEX users_email_idx ON users (email);

-- सटीक लुकअप: तेज़ और स्पष्ट
SELECT id, name FROM users WHERE email = $1;
```

व्यापक सबक: सही जवाब वाली समस्याओं के लिए अनुमानित सर्च एक श्रेणी त्रुटि है। यह *कुछ* लौटाता है — जो आत्मविश्वास से गलत हो सकता है।

## इन टूल्स को मिलाना

ये टूल स्वच्छ रूप से संयोजित होते हैं। आप सटीक रूप से एक नहीं चुनते।

**FTS + pg_trgm कीवर्ड में टाइपो सहन करने वाले सर्च बॉक्स के लिए:**

```sql
-- शीर्षक पर ट्राइग्राम समानता टाइपो पकड़ती है; ts_rank बॉडी प्रासंगिकता संभालता है
SELECT id, title,
  ts_rank(search_vector, to_tsquery('simple', $1)) AS fts_rank,
  similarity(title, $1) AS trgm_score
FROM posts
WHERE search_vector @@ to_tsquery('simple', $1)
   OR title % $1
ORDER BY (ts_rank(search_vector, to_tsquery('simple', $1)) + similarity(title, $1)) DESC
LIMIT 10;
```

**FTS + `unaccent` अंतर्राष्ट्रीय कंटेंट के लिए:**

```sql
-- डायाक्रिटिकल निशान हटाएं ताकि "José" "Jose" से मैच करे
CREATE EXTENSION IF NOT EXISTS unaccent;

CREATE TEXT SEARCH CONFIGURATION public.simple_unaccent (COPY = pg_catalog.simple);

ALTER TEXT SEARCH CONFIGURATION public.simple_unaccent
  ALTER MAPPING FOR hword, hword_part, word
  WITH unaccent, simple;

ALTER TABLE posts ADD COLUMN search_vector tsvector;

CREATE TRIGGER posts_search_vector_refresh
BEFORE INSERT OR UPDATE OF title, body ON posts
FOR EACH ROW EXECUTE FUNCTION
  tsvector_update_trigger(search_vector, 'public.simple_unaccent', title, body);
```

**`unaccent` + `pg_trgm` अंतर्राष्ट्रीय नाम सर्च के लिए:**

```sql
ALTER TABLE users ADD COLUMN name_search text;

CREATE FUNCTION users_name_search_refresh()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  NEW.name_search := unaccent(coalesce(NEW.name, ''));
  RETURN NEW;
END;
$$;

CREATE TRIGGER users_name_search_refresh
BEFORE INSERT OR UPDATE OF name ON users
FOR EACH ROW EXECUTE FUNCTION users_name_search_refresh();

CREATE INDEX users_name_search_trgm_idx
  ON users USING GIN (name_search gin_trgm_ops);

SELECT id, name
FROM users
WHERE name_search % unaccent($1)
ORDER BY similarity(name_search, unaccent($1)) DESC
LIMIT 10;
```

ट्रिगर उदाहरण जनरेटेड-कॉलम या इंडेक्स एक्सप्रेशन के अंदर `unaccent()` का उपयोग करने से बचते हैं, जहाँ PostgreSQL की अपरिवर्तनीयता नियम मायने रखते हैं। यदि आप `unaccent()` को अपनी स्वयं की अपरिवर्तनीय फ़ंक्शन में लपेटते हैं, तो दस्तावेज़ बनाएं कि आप अपग्रेड/कॉन्फ़िगरेशन जोखिम स्वीकार कर रहे हैं।

## उल्लेखनीय एक्सटेंशन

**`pg_trgm`** अधिकांश Postgres वितरणों के साथ बंडल किया जाता है लेकिन स्पष्ट सक्षमता की आवश्यकता होती है। Postgres में फजी स्ट्रिंग मिलान की नींव।

**`unaccent`** इंडेक्सिंग और क्वेरी करने से पहले डायाक्रिटिकल निशान हटाता है। यूरोपीय भाषा कंटेंट के लिए `pg_trgm` और FTS दोनों के साथ अच्छी तरह से जोड़ता है। Postgres के साथ बंडल किया गया।

**`pg_bigm`** ट्राइग्राम दृष्टिकोण को बाइग्राम (2-अक्षर स्लाइस) तक बढ़ाता है, जो CJK (चीनी, जापानी, कोरियाई) भाषाओं के लिए परिणामों में काफी सुधार करता है जहाँ `pg_trgm` कमज़ोर होता है। अलग से इंस्टॉल करना होगा; बंडल नहीं किया गया।

**`pg_search`** ([ParadeDB](https://www.paradedb.com/) से) मानक `GIN` / `tsvector` स्टैक को Tantivy-आधारित BM25 इंडेक्स से बदलता है। यह आपको BM25 स्कोरिंग (अक्सर `ts_rank` से बेहतर), FTS क्वेरीज़ के भीतर फजी मिलान, फैसेटेड सर्च, और बड़ी तालिकाओं पर नाटकीय रूप से तेज़ इंडेक्सिंग देता है। जब मानक FTS रैंकिंग या प्रदर्शन सीमाएं दिखाने लगे तो यह एक ड्रॉप-इन अपग्रेड पथ है।

```sql
-- pg_search: फजी मिलान के साथ BM25 फुल-टेक्स्ट सर्च
CREATE INDEX posts_bm25_idx ON posts
  USING bm25 (id, title, body)
  WITH (key_field = 'id', text_fields = '{"title": {}, "body": {}}');

-- BM25 स्कोरिंग + फजी मिलान के साथ क्वेरी ("javascipt" पकड़ता है)
SELECT id, title, paradedb.score(id) AS rank
FROM posts
WHERE posts @@@ paradedb.fuzzy_phrase(field => 'title', value => 'postgres performnce')
ORDER BY rank DESC
LIMIT 10;
```

**`pgvector`** घना वेक्टर संग्रहण और समानता सर्च जोड़ता है। यह सही टूल है जब उपयोगकर्ता जो वे चाहते हैं उसका वर्णन करते हैं बजाय इसे नाम देने के — सेमांटिक सर्च, RAG, संबंधित-कंटेंट सिफारिशें, बहुभाषी क्वेरीज़। [Semantic Vector Search and Hybrid Strategies](/semantic-vector-search-landscape) में गहराई से कवर किया गया।

## निर्णय तालिका

| आप क्या खोज रहे हैं | अनुशंसित |
|---|---|
| गद्य लेख, दस्तावेज़, टिकट | FTS |
| टाइपो वाले व्यक्ति/कंपनी नाम | `pg_trgm` |
| ऑटोकंप्लीट, प्रीफिक्स सर्च | `pg_trgm` |
| छोटे कोड, पहचानकर्ता | `pg_trgm` |
| कीवर्ड के लिए लॉग संदेश | FTS |
| अंतर्राष्ट्रीय नाम | `pg_trgm` + `unaccent` |
| बड़ा कंटेंट, बेहतर रैंकिंग | `pg_search` (ParadeDB BM25) |
| प्राइमरी की, सटीक ईमेल, आईडी | B-tree इंडेक्स |
| दिनांक, रेंज, क्रमबद्ध सूची | B-tree इंडेक्स |
| अनुमति, श्रेणी, फ़िल्टर | नियमित WHERE क्लॉज |
| प्रश्न, पैराफ्रेज़, अवधारणाएँ | pgvector (अगला लेख देखें) |

संदेह होने पर: वर्तनी भिन्नता वाली छोटी स्ट्रिंग्स → ट्राइग्राम। कीवर्ड क्वेरीज़ के लिए लंबा गद्य → FTS। संरचित पहचानकर्ता → नियमित इंडेक्स। संकल्पनात्मक या प्राकृतिक भाषा क्वेरीज़ → pgvector।

## हाइब्रिड सर्च: दो सिग्नल, एक रैंक

जब एक क्वेरी जैसे `"withRetry timeout errors"` सर्च बॉक्स में आती है, तो यह दो तरह की इंटेंट लेकर आती है: उपयोगकर्ता द्वारा जाने जाने वाले सटीक प्रतीक नाम (`withRetry`) और एक संकल्पनात्मक विवरण (`timeout errors`)। कोई एकल प्रिमिटिव दोनों को कवर नहीं करता। FTS और वेक्टर सर्च को समानांतर में चलाना — फिर रैंक की सूची को Reciprocal Rank Fusion से मिलाना — करता है।

RRF हर परिणाम को प्रत्येक सूची में `1 / (60 + rank)` के रूप में स्कोर करता है और सूचियों में योग करता है। स्थिरांक 60 शीर्ष रैंक के लाभ को कम करता है, इसलिए एक परिणाम जो दोनों सूचियों में दूसरा स्थान रखता है, उसे हरा सकता है जो एक सूची जीतता है और दूसरी से पूरी तरह चूक जाता है। महत्वपूर्ण रूप से, RRF कभी भी विधियों में कच्चे स्कोर का औसत नहीं करता — FTS रैंक और कोसाइन दूरी अलग मुद्राएँ हैं और उन्हें अंकगणितीय रूप से संयोजित नहीं किया जा सकता।

```sql
-- हाइब्रिड सर्च: RRF से मिलाया गया FTS + pgvector
WITH fts AS (
  SELECT id, ts_rank(search_vector, query) AS score,
         ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM docs, to_tsquery('english', 'withRetry & timeout') query
  WHERE search_vector @@ query
  LIMIT 60
),
vec AS (
  SELECT id,
         ROW_NUMBER() OVER (ORDER BY embedding <=> $embedding) AS rank
  FROM docs
  ORDER BY embedding <=> $embedding
  LIMIT 60
)
SELECT COALESCE(fts.id, vec.id) AS id,
       (COALESCE(1.0 / (60 + fts.rank), 0) +
        COALESCE(1.0 / (60 + vec.rank), 0)) AS rrf_score
FROM fts FULL JOIN vec ON fts.id = vec.id
ORDER BY rrf_score DESC
LIMIT 10;
```

प्रति शाखा 60-दस्तावेज़ उम्मीदवार पूल (`LIMIT 60`) एक सामान्य प्रारंभिक बिंदु है। यदि रिकॉल कम है तो इसे चौड़ा करें; गति के लिए संकीर्ण करें।

## आगे क्या है

पोस्टग्रेस टेक्स्ट सर्च बहुत कुछ कवर करता है, लेकिन इसकी एक सीमा है। जब उपयोगकर्ता जो वे चाहते हैं उसका वर्णन करते हैं बजाय इसे नाम देने के — "फ़्लाइट में सोने में मदद करने वाली कोई चीज़," "नए इंजीनियर के रूप में डीबगिंग आत्मविश्वास पर लेख" — शाब्दिक और ट्राइग्राम दोनों सर्च विफल हो जाते हैं।

यह वेक्टर एम्बेडिंग, सेमांटिक सर्च, और हाइब्रिड वास्तुकला का क्षेत्र है। [Semantic Vector Search and Hybrid Strategies](/semantic-vector-search-landscape) में गहराई से कवर किया गया।
