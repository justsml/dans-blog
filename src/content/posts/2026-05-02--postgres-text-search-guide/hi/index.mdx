ज़्यादातर टीमें एक ही Postgres खोज टूल का इस्तेमाल करती हैं। जो टीमें तीनों को जानती हैं, वे कम जटिलता के साथ बेहतर खोज शिप करती हैं — और एक महंगे समर्पित खोज सेविस की ट्रेन से बचती हैं जिसकी उन्हें अभी ज़रूरत नहीं थी।

यह गाइड Postgres-नेटिव विकल्पों के पूरे सेट को कवर करता है: हर एक क्या करता है, कब वह सही फिट है, और उन्हें कैसे लेयर किया जाए।

---

## तीन टूल

**फुल-टेक्स्ट खोज** (`tsvector` / `GIN` इंडेक्स) लेक्सिकल है। यह टेक्स्ट को लेक्सम में टोकनाइज़ करता है, उन्हें स्टेम करता है, और क्वेरीज़ को इंडेक्स के खिलाफ मैच करता है। "Running" और "runs" एक ही लेक्सम में सिकुड़ जाते हैं। "dog" और "dogs" भी ऐसे ही। रैंकिंग फंक्शन (`ts_rank`) उन डॉक्यूमेंट्स को प्राथमिकता देता है जहाँ क्वेरी शब्द अक्सर या प्रमुखता से दिखते हैं।

**ट्राइग्राम** (`pg_trgm`) स्ट्रिंग्स को ओवरलैपिंग 3-कैरेक्टर स्लाइस में तोड़ते हैं और मापते हैं कि दो स्ट्रिंग्स कितने स्लाइस शेयर करती हैं। "Dan" → `" da"`, `"dan"`, `"an "`। "Micheal" और "Michael" अपने ज़्यादातर ट्राइग्राम शेयर करते हैं, इसलिए समानता ज़्यादा है। इससे `pg_trgm` फज़ी नेम मैचिंग, टाइपो टॉलरेंस, और ऑटोकम्प्लीट में बेहतरीन बनता है — उस जगह जहाँ FTS कमज़ोर प्रदर्शन करता है।

**एग्ज़ैक्ट-मैच इंडेक्स** (B-tree, hash) प्राइमरी कीज़, ईमेल पते, IDs, SKUs, और ऐसी चीज़ें संभालते हैं जहाँ जवाब बाइनरी है: यह मैच होता है या नहीं। इन्हें "खोज" जैसा महसूस नहीं होता, लेकिन इनका इस कंवर्सेशन में होना ज़रूरी है क्योंकि सबसे बुरा पैटर्न फज़ी या सिमेंटिक खोज का इस्तेमाल ऐसी समस्याओं के लिए करना है जिनके सही जवाब हैं।

चुनाव सोफिस्टिकेशन के बारे में नहीं है। यह टूल को क्वेरी के आकार से मैच करने के बारे में है।

## जब फुल-टेक्स्ट खोज जीतती है

**कीवर्ड के लिए गद्य खोजना।** ब्लॉग पोस्ट, डॉक्यूमेंटेशन, प्रोडक्ट विवरण, सपोर्ट टिकट, कानूनी दस्तावेज़। FTS इस कंटेंट आकार के लिए डिज़ाइन किया गया था: इंडेक्स्ड, रैंक्ड रिट्रीवल नेचुरल-लैंग्वेज टेक्स्ट पर।

**कीवर्ड-आधारित उपयोगकर्ता क्वेरीज़।** उपयोगकर्ता सर्च टर्म टाइप करते हैं, टैग से फिल्टर करते हैं, या कीवर्ड से ब्राउज़ करते हैं। FTS इस इरादे को बिना किसी एम्बेडिंग इंफ्रास्ट्रक्चर के नेटिवली संभालता है।

**बिना बाहरी डिपेंडेंसी के रैंक्ड परिणाम।** FTS इंडेक्स तेज़, डिटरमिनिस्टिक हैं, और इन्हें किसी API कॉल की ज़रूरत नहीं। रिलेवेंस सिग्नल टर्म फ्रीक्वेंसी से आता है जिसे फील्ड पोज़िशन से वेट किया जाता है।

**खोज के साथ बूलियन फिल्टरिंग।** FTS आपके मौजूदा क्वेरी लॉजिक के साथ नेचुरली कम्पोज़ होता है:

```sql
SELECT * FROM posts
WHERE search_vector @@ to_tsquery('english', 'postgres & performance')
  AND category = 'tutorial'
  AND published_at > NOW() - INTERVAL '6 months';
```

### FTS सेटअप करना

```sql
-- Generated column keeps the index current automatically
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body,  '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Query
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`setweight` इम्पोर्टेंस असाइन करता है: `A` (शीर्षक) `B` (बॉडी) से ऊपर है। यह ज़्यादातर कंटेंट-सर्च यूज़ केस के लिए पूरा रिलेवेंस मॉडल है।

### FTS क्या अच्छे से नहीं संभालता

- क्वेरीज़ में टाइपो — "javascipt" "javascript" से मैच नहीं होगा
- व्यक्ति के नाम, पते, प्रोपर नाउन जो प्रेडिक्टेबली स्टेम नहीं होते
- बिना स्पेशल कॉन्फ़िगरेशन के प्रीफ़िक्स/ऑटोकम्प्लीट
- क्वेरीज़ जहाँ उपयोगकर्ता किसी कॉन्सेप्ट का वर्णन करता है बजाय उसे नाम देने के

## जब ट्राइग्राम जीतते हैं (`pg_trgm`)

`pg_trgm` उस अजीब मिडल को कवर करता है जिसे FTS लगातार गिराता है।

FTS टेक्स्ट को लेक्सम में टोकनाइज़ करता है और स्टेम करता है। गद्य के लिए यह सही है। नामों और छोटे आइडेंटिफ़ायर के लिए अक्सर नहीं:

- व्यक्ति के नाम ("Dan Levy" → डिक्शनरी और भाषा कॉन्फ़िग के आधार पर अलग तरह स्टेम्ड)
- कंपनी नाम, पते, प्रोडक्ट शीर्षक जहाँ सटीक वर्तनी मायने रखती है
- टाइपो वाले क्वेरीज़ — "Micheal Jordan", "Amaon", "javascipt"
- ऑटोकम्प्लीट / प्रीफ़िक्स खोज
- आंशिक स्ट्रिंग मैचिंग ("son" मैचिंग "Johnson", "Anderson")

`pg_trgm` भाषा-अज्ञेय भी है, जो विविध भाषाई पृष्ठभूमि वाले नामों के लिए मायने रखता है। FTS को प्रति भाषा डिक्शनरी कॉन्फ़िगरेशन की ज़रूरत होती है।

### फज़ी नेम सर्च

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- Finds "Micheal Jordan" when searching "Michael Jordan"
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % operator = similarity threshold (default 0.3)
ORDER BY score DESC
LIMIT 10;
```

`%` ऑपरेटर `pg_trgm.similarity_threshold` (डिफ़ॉल्ट 0.3, रेंज 0–1) का इस्तेमाल करता है। नेम सर्च के लिए, 0.3–0.4 टाइपो को पकड़ता है जबकि नॉइज़ कम रखता है।

### ऑटोकम्प्लीट, प्रीफ़िक्स, और कंटेन्स सर्च

```sql
-- Prefix matching for autocomplete. A trigram GIN index can help,
-- but a B-tree pattern index may be better for pure left-anchored prefixes.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- word_similarity for partial matches within longer strings
-- ("Johnson" within "Andrew Johnson III")
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name
ORDER BY score DESC
LIMIT 10;
```

ट्राइग्राम GIN इंडेक्स खासकर `ILIKE '%pattern%'` कंटेन्स क्वेरीज़ और टाइपो-टॉलरेंट मैचिंग के लिए उपयोगी है — पैटर्न जो बिना ट्राइग्राम इंडेक्स के आमतौर पर फुल-टेबल स्कैन होते हैं।

### कब FTS के बजाय pg_trgm का इस्तेमाल करें

| परिदृश्य | इस्तेमाल करें |
|---|---|
| टाइपो के साथ व्यक्ति/कंपनी नाम खोज | `pg_trgm` |
| ऑटोकम्प्लीट / प्रीफ़िक्स खोज | `pg_trgm` (या प्रीफ़िक्स क्वेरीज़ के साथ FTS) |
| छोटी स्ट्रिंग्स, आइडेंटिफ़ायर, कोड | `pg_trgm` |
| गद्य लेख, डॉक्यूमेंटेशन, टिकट | FTS |
| कीवर्ड के लिए लॉग मैसेज | FTS |
| बहुभाषी नाम खोज | `pg_trgm` (भाषा-अज्ञेय) |

## जब एग्ज़ैक्ट-मैच SQL जीतता है

कुछ "खोज" समस्याएँ खोज ही नहीं हैं।

"ईमेल `dan@example.com` वाले उपयोगकर्ता खोजें" एक इक्वैलिटी चेक है। "ऑर्डर `ORD-12345` खोजें" प्राइमरी की लुकअप है। "`tutorial` कैटेगरी में पोस्ट तारीख से सॉर्ट करें" फ़िल्टर्ड क्वेरी है। ये B-tree या hash इंडेक्स पर हैं।

यहाँ FTS या ट्राइग्राम इस्तेमाल करने से जटिलता बढ़ती है बिना सटीकता सुधार के — और एग्ज़ैक्ट आइडेंटिफ़ायर के लिए, नियर-मैच कोई मैच न होने से बदतर है।

```sql
CREATE INDEX users_email_idx ON users (email);

-- Exact lookup: fast and unambiguous
SELECT id, name FROM users WHERE email = $1;
```

व्यापक सबक: सही जवाब वाली समस्याओं के लिए अनुमानित खोज एक कैटेगरी एरर है। यह *कुछ* रिटर्न करती है — जो आत्मविश्वास से गलत हो सकता है।

## इन टूल्स को कॉम्बाइन करना

ये टूल साफ़-साफ़ कम्पोज़ होते हैं। आप एक ही नहीं चुनते।

**कीवर्ड में टाइपो टॉलरें्स वाले सर्च बॉक्स के लिए FTS + pg_trgm:**

```sql
-- Trigram similarity on title catches typos; ts_rank handles body relevance
SELECT id, title,
  ts_rank(search_vector, to_tsquery('simple', $1)) AS fts_rank,
  similarity(title, $1) AS trgm_score
FROM posts
WHERE search_vector @@ to_tsquery('simple', $1)
   OR title % $1
ORDER BY (ts_rank(search_vector, to_tsquery('simple', $1)) + similarity(title, $1)) DESC
LIMIT 10;
```

**इंटरनेशनल कंटेंट के लिए FTS + `unaccent`:**

```sql
-- Strip diacritical marks so "José" matches "Jose"
CREATE EXTENSION IF NOT EXISTS unaccent;

CREATE TEXT SEARCH CONFIGURATION public.simple_unaccent (COPY = pg_catalog.simple);

ALTER TEXT SEARCH CONFIGURATION public.simple_unaccent
  ALTER MAPPING FOR hword, hword_part, word
  WITH unaccent, simple;

ALTER TABLE posts ADD COLUMN search_vector tsvector;

CREATE TRIGGER posts_search_vector_refresh
BEFORE INSERT OR UPDATE OF title, body ON posts
FOR EACH ROW EXECUTE FUNCTION
  tsvector_update_trigger(search_vector, 'public.simple_unaccent', title, body);
```

**इंटरनेशनल नेम सर्च के लिए `unaccent` + `pg_trgm`:**

```sql
ALTER TABLE users ADD COLUMN name_search text;

CREATE FUNCTION users_name_search_refresh()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  NEW.name_search := unaccent(coalesce(NEW.name, ''));
  RETURN NEW;
END;
$$;

CREATE TRIGGER users_name_search_refresh
BEFORE INSERT OR UPDATE OF name ON users
FOR EACH ROW EXECUTE FUNCTION users_name_search_refresh();

CREATE INDEX users_name_search_trgm_idx
  ON users USING GIN (name_search gin_trgm_ops);

SELECT id, name
FROM users
WHERE name_search % unaccent($1)
ORDER BY similarity(name_search, unaccent($1)) DESC
LIMIT 10;
```

ट्रिगर उदाहरण `unaccent()` को जेनरेटेड-कॉलम या इंडेक्स एक्सप्रेशन के अंदर इस्तेमाल करने से बचते हैं, जहाँ PostgreSQL के इम्युटेबिलिटी नियम मायने रखते हैं। अगर आप `unaccent()` को अपने इम्यूटेबल फंक्शन में रैप करते हैं, तो डॉक्यूमेंट करें कि आप अपग्रेड/कॉन्फ़िगरेशन रिस्क स्वीकार कर रहे हैं।

## उल्लेखनीय एक्सटेंशन

**`pg_trgm`** ज़्यादातर Postgres डिस्ट्रिब्यूशन के साथ बंडल है लेकिन इसे स्पष्ट रूप से एनेबल करना ज़रूरी है। Postgres में फज़ी स्ट्रिंग मैचिंग की नींव।

**`unaccent`** इंडेक्सिंग और क्वेरीिंग से पहले डायक्रिटिकल मार्क्स हटाता है। यूरोपीय-भाषा कंटेंट के लिए `pg_trgm` और FTS दोनों के साथ अच्छा काम करता है। Postgres के साथ बंडल।

**`pg_bigm`** ट्राइग्राम अप्रोच को बाइग्राम (2-कैरेक्टर स्लाइस) तक बढ़ाता है, जो CJK (चीनी, जापानी, कोरियन) भाषाओं में जहाँ `pg_trgm` कम प्रदर्शन करता है, परिणामों में काफ़ी सुधार करता है। अलग से इंस्टॉल करना ज़रूरी; बंडल नहीं।

**`pg_search`** ([ParadeDB](https://www.paradedb.com/) से) स्टैंडर्ड `GIN` / `tsvector` स्टैक को Tantivy-आधारित BM25 इंडेक्स से बदलता है। यह आपको BM25 स्कोरिंग (अक्सर `ts_rank` से बेहतर), FTS क्वेरीज़ के अंदर फज़ी मैचिंग, फैसेटेड सर्च, और बड़ी टेबल्स पर काफ़ी तेज़ इंडेक्सिंग देता है। यह एक ड्रॉप-इन अपग्रेड पाथ है जब स्टैंडर्ड FTS रैंकिंग या प्रदर्शन सीमाएँ दिखाने लगता है।

```sql
-- pg_search: BM25 full-text search with fuzzy matching
CREATE INDEX posts_bm25_idx ON posts
  USING bm25 (id, title, body)
  WITH (key_field = 'id', text_fields = '{"title": {}, "body": {}}');

-- Query with BM25 scoring + fuzzy matching (catches "javascipt")
SELECT id, title, paradedb.score(id) AS rank
FROM posts
WHERE posts @@@ paradedb.fuzzy_phrase(field => 'title', value => 'postgres performnce')
ORDER BY rank DESC
LIMIT 10;
```

**`pgvector`** डेंस वेक्टर स्टोरेज और सिमिलैरिटी सर्च जोड़ता है। यह सही टूल है जब उपयोगकर्ता वर्णन करते हैं कि वे क्या चाहते हैं बजाय उसे नाम देने के — सिमेंटिक सर्च, RAG, रिलेटेड-कंटेंट रिकमेंडेशन, बहुभाषी क्वेरीज़। [सिमेंटिक वेक्टर खोज और हाइब्रिड रणनीतियाँ](/semantic-vector-search-landscape) में विस्तार से कवर किया गया है।

## निर्णय तालिका

| आप क्या खोज रहे हैं | अनुशंसित |
|---|---|
| गद्य लेख, डॉक्स, टिकट | FTS |
| टाइपो के साथ व्यक्ति/कंपनी नाम | `pg_trgm` |
| ऑटोकम्प्लीट, प्रीफ़िक्स खोज | `pg_trgm` |
| छोटे कोड, आइडेंटिफ़ायर | `pg_trgm` |
| कीवर्ड के लिए लॉग मैसेज | FTS |
| इंटरनेशनल नाम | `pg_trgm` + `unaccent` |
| बड़ा कंटेंट, बेहतर रैंकिंग | `pg_search` (ParadeDB BM25) |
| प्राइमरी कीज़, एग्ज़ैक्ट ईमेल, IDs | B-tree इंडेक्स |
| तारीखें, रेंज, सॉर्टेड लिस्ट | B-tree इंडेक्स |
| परमिशन, कैटेगरी, फिल्टर | सामान्य WHERE क्लॉज़ |
| सवाल, पैराफ्रेज़, कॉन्सेप्ट | pgvector (अगला लेख देखें) |

संदेह हो तो: वर्तनी वैरिएशन वाली छोटी स्ट्रिंग्स → ट्राइग्राम। कीवर्ड क्वेरीज़ के लिए लंबा गद्य → FTS। संरचित आइडेंटिफ़ायर → रेगुलर इंडेक्स। कॉन्सेप्चुअल या नेचुरल-लैंग्वेज क्वेरीज़ → pgvector।

## हाइब्रिड खोज: दो सिग्नल, एक रैंक

जब `"withRetry timeout errors"` जैसी क्वेरी सर्च बॉक्स से टकराती है, तो यह दो तरह का इरादा लेकर आती है: एग्ज़ैक्ट सिम्बल नाम जो उपयोगकर्ता जानता है (`withRetry`) और एक कॉन्सेप्चुअल विवरण (`timeout errors`)। कोई एक प्रिमिटिव दोनों को कवर नहीं करता। FTS और वेक्टर सर्च को समानांतर में चलाकर — फिर उनकी रैंक्ड लिस्ट को रेसिप्रोकल रैंक फ़्यूज़न से मर्ज करके — यह हो जाता है।

RRF हर परिणाम को हर लिस्ट में `1 / (60 + rank)` से स्कोर करता है और लिस्ट्स में सम लगाता है। कॉन्स्टेंट 60 टॉप रैंक्स के फ़ायदे को कम करता है, ताकि एक परिणाम जो दोनों लिस्ट्स में दूसरा है, एक परिणाम को हरा सके जो एक लिस्ट जीतता है और दूसरी में पूरी तरह ग़ायब है। सबसे अहम, RRF कभी भी रॉ स्कोर को मेथड्स में औसत नहीं करता — FTS रैंक और cosine distance अलग-अलग मुद्राएँ हैं और अंकगणितीय रूप से कॉम्बाइन नहीं हो सकतीं।

```sql
-- Hybrid search: FTS + pgvector merged with RRF
WITH fts AS (
  SELECT id, ts_rank(search_vector, query) AS score,
         ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM docs, to_tsquery('english', 'withRetry & timeout') query
  WHERE search_vector @@ query
  LIMIT 60
),
vec AS (
  SELECT id,
         ROW_NUMBER() OVER (ORDER BY embedding <=> $embedding) AS rank
  FROM docs
  ORDER BY embedding <=> $embedding
  LIMIT 60
)
SELECT COALESCE(fts.id, vec.id) AS id,
       (COALESCE(1.0 / (60 + fts.rank), 0) +
        COALESCE(1.0 / (60 + vec.rank), 0)) AS rrf_score
FROM fts FULL JOIN vec ON fts.id = vec.id
ORDER BY rrf_score DESC
LIMIT 10;
```

प्रति ब्रांच 60-डॉक्यूमेंट कैंडिडेट पूल (`LIMIT 60`) एक सामान्य शुरुआती बिंदु है। अगर रिकॉल कम है तो इसे चौड़ा करें; स्पीड के लिए इसे संकरा करें।

## आगे क्या

Postgres टेक्स्ट खोज बहुत कुछ कवर करती है, लेकिन इसकी एक सीलिंग है। जब उपयोगकर्ता वर्णन करते हैं कि वे क्या चाहते हैं बजाय उसे नाम देने के — "फ्लाइट पर सोने में मदद करने वाली कुछ", "एक नए इंजीनियर के रूप में डिबगिंग कॉन्फिडेंस के बारे में लेख" — तो लेक्सिकल और ट्राइग्राम खोज दोनों विफल होती हैं।

यह वेक्टर एम्बेडिंग, सिमेंटिक खोज, और हाइब्रिड आर्किटेक्चर का क्षेत्र है। [सिमेंटिक वेक्टर खोज और हाइब्रिड रणनीतियाँ](/semantic-vector-search-landscape) में कवर किया गया है।
