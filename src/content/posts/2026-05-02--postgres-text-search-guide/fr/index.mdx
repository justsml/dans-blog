La plupart des équipes utilisent un seul outil de recherche Postgres. Les équipes qui maîtrisent les trois livrent une meilleure recherche avec moins de complexité — et évitent le détour coûteux vers un service de recherche dédié dont elles n'avaient pas encore besoin.

Ce guide couvre l'ensemble des options natives Postgres : ce que fait chaque outil, quand il est le bon choix, et comment les combiner.

---

## Les Trois Outils

**La recherche textuelle** (`tsvector` / index GIN) est lexicale. Elle tokenise le texte en lexèmes, les décline, et fait correspondre les requêtes contre l'index. "Running" et "runs" se réduisent au même lexème. Il en va de même pour "dog" et "dogs". La fonction de classement (`ts_rank`) récompense les documents où les termes de la requête apparaissent fréquemment ou de manière prominente.

**Les trigrammes** (`pg_trgm`) découpent les chaînes en trigrammes (segments de 3 caractères) et mesurent combien de trigrammes deux chaînes partagent. "Dan" → `" da"`, `"dan"`, `"an "`. "Micheal" et "Michael" partagent la plupart de leurs trigrammes, donc la similarité est élevée. Cela rend `pg_trgm` excellent pour la correspondance floue de noms, la tolérance aux fautes de frappe, et l'autocomplétion — l'espace où la RFT performe mal.

**Les index à correspondance exacte** (B-tree, hash) gèrent les clés primaires, les adresses e-mail, les IDs, les SKUs, et tout ce où la réponse est binaire : cela correspond ou cela ne correspond pas. Ces index ne semblent pas être de la "recherche", mais ils appartiennent à cette conversation car la pire des habitudes est d'utiliser une recherche floue ou sémantique pour des problèmes qui ont des réponses correctes.

Le choix ne concerne pas la sophistication. Il s'agit d'adapter l'outil à la forme de la requête.

## Quand la Recherche Textuelle Gagne

**Rechercher de la prose pour des mots-clés.** Articles de blog, documentation, descriptions de produits, tickets de support, documents juridiques. La RFT a été conçue pour cette forme de contenu : récupération indexée et classée sur du texte en langage naturel.

**Requêtes utilisateur basées sur des mots-clés.** Les utilisateurs tapent un terme de recherche, filtrent par tag, ou naviguent par mot-clé. La RFT gère cette intention nativement sans aucune infrastructure d'embedding.

**Résultats classés sans dépendances externes.** Les index de RFT sont rapides, déterministes, et ne nécessitent aucun appel API. Le signal de pertinence vient de la fréquence des termes pondérée par la position du champ.

**Filtrage booléen en parallèle avec la recherche.** La RFT se compose naturellement avec votre logique de requête existante :

```sql
SELECT * FROM posts
WHERE search_vector @@ to_tsquery('english', 'postgres & performance')
  AND category = 'tutorial'
  AND published_at > NOW() - INTERVAL '6 months';
```

### Configuration de la RFT

```sql
-- Colonne générée garde l'index à jour automatiquement
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body,  '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Requête
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`setweight` assigne l'importance : `A` (titre) prime sur `B` (corps). C'est tout le modèle de pertinence pour la plupart des cas d'usage de recherche de contenu.

### Ce que la RFT Ne Gère Pas Bien

- Fautes de frappe dans les requêtes — "javascipt" ne correspondra pas à "javascript"
- Noms de personnes, adresses, noms propres qui ne se déclinent pas de manière prévisible
- Préfixe / autocomplétion sans configuration spéciale
- Requêtes où l'utilisateur décrit un concept plutôt que de le nommer

## Quand les Trigrammes Gagnent (`pg_trgm`)

`pg_trgm` couvre le milieu inconfortable que la RFT échoue constamment à gérer.

La RFT tokenise le texte en lexèmes et les décline. Pour la prose c'est correct. Pour les noms et identifiants courts cela ne l'est souvent pas :

- Noms de personnes ("Dan Levy" → décliné différemment selon le dictionnaire et la configuration de langue)
- Noms de sociétés, adresses, titres de produits où l'orthographe exacte compte
- Requêtes avec fautes de frappe — "Micheal Jordan", "Amaon", "javascipt"
- Autocomplétion / recherche par préfixe
- Correspondance de chaînes partielles ("son" correspondant à "Johnson", "Anderson")

`pg_trgm` est aussi agnostique de la langue, ce qui compte pour les noms de contextes linguistiques divers. La RFT nécessite une configuration de dictionnaire par langue.

### Recherche Floue de Noms

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- Trouve "Micheal Jordan" quand on recherche "Michael Jordan"
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- Opérateur % = seuil de similarité (défaut 0.3)
ORDER BY score DESC
LIMIT 10;
```

L'opérateur `%` utilise `pg_trgm.similarity_threshold` (défaut 0.3, plage 0–1). Pour la recherche de noms, 0.3–0.4 attrape les fautes de frappe tout en gardant le bruit faible.

### Autocomplétion, Préfixe, et Recherche de Contenance

```sql
-- Correspondance par préfixe pour l'autocomplétion. Un index GIN trigramme peut aider,
-- mais un index de motif B-tree peut être meilleur pour les préfixes ancrés à gauche purs.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- word_similarity pour les correspondances partielles dans des chaînes plus longues
-- ("Johnson" dans "Andrew Johnson III")
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name
ORDER BY score DESC
LIMIT 10;
```

L'index GIN trigramme est particulièrement utile pour les requêtes de contenance `ILIKE '%pattern%'` et la correspondance tolérante aux fautes — des motifs qui sont habituellement des scans de table complète sans index trigramme.

### Quand Recourir à pg_trgm Plutôt qu'à la RFT

| Scénario | Usage |
|---|---|
| Recherche de noms de personnes/sociétés avec fautes de frappe | `pg_trgm` |
| Autocomplétion / recherche par préfixe | `pg_trgm` (ou RFT avec requêtes par préfixe) |
| Chaînes courtes, identifiants, codes | `pg_trgm` |
| Articles de prose, documentation, tickets | RFT |
| Logs de messages pour mots-clés | RFT |
| Recherche de noms multilingues | `pg_trgm` (agnostique de la langue) |

## Quand la SQL à Correspondance Exacte Gagne

Certains problèmes de "recherche" ne sont pas de la recherche du tout.

"Trouver l'utilisateur avec l'email `dan@example.com`" est une vérification d'égalité. "Trouver la commande `ORD-12345`" est une recherche de clé primaire. "Lister les posts dans la catégorie `tutorial` triés par date" est une requête filtrée. Ceux-ci appartiennent sur des index B-tree ou hash.

Utiliser la RFT ou les trigrammes ici ajoute de la complexité sans améliorer la correction — et pour les identifiants exacts, une correspondance approximative est pire qu'aucune correspondance.

```sql
CREATE INDEX users_email_idx ON users (email);

-- Recherche exacte : rapide et sans ambiguïté
SELECT id, name FROM users WHERE email = $1;
```

La leçon plus large : la recherche approximative pour des problèmes avec des réponses correctes est une erreur de catégorie. Elle retourne *quelque chose* — ce qui peut être sûrement faux.

## Combiner Ces Outils

Ces outils se composent proprement. Vous ne choisissez pas exactement un.

**RFT + pg_trgm pour une boîte de recherche qui tolère les fautes de frappe dans les mots-clés :**

```sql
-- Similarité trigramme sur le titre attrape les fautes de frappe ; ts_rank gère la pertinence du corps
SELECT id, title,
  ts_rank(search_vector, to_tsquery('simple', $1)) AS fts_rank,
  similarity(title, $1) AS trgm_score
FROM posts
WHERE search_vector @@ to_tsquery('simple', $1)
   OR title % $1
ORDER BY (ts_rank(search_vector, to_tsquery('simple', $1)) + similarity(title, $1)) DESC
LIMIT 10;
```

**RFT + `unaccent` pour un contenu international :**

```sql
-- Retirer les accents avant l'indexation et la requête. "José" correspond à "Jose"
CREATE EXTENSION IF NOT EXISTS unaccent;

CREATE TEXT SEARCH CONFIGURATION public.simple_unaccent (COPY = pg_catalog.simple);

ALTER TEXT SEARCH CONFIGURATION public.simple_unaccent
  ALTER MAPPING FOR hword, hword_part, word
  WITH unaccent, simple;

ALTER TABLE posts ADD COLUMN search_vector tsvector;

CREATE TRIGGER posts_search_vector_refresh
BEFORE INSERT OR UPDATE OF title, body ON posts
FOR EACH ROW EXECUTE FUNCTION
  tsvector_update_trigger(search_vector, 'public.simple_unaccent', title, body);
```

**`unaccent` + `pg_trgm` pour la recherche de noms internationaux :**

```sql
ALTER TABLE users ADD COLUMN name_search text;

CREATE FUNCTION users_name_search_refresh()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  NEW.name_search := unaccent(coalesce(NEW.name, ''));
  RETURN NEW;
END;
$$;

CREATE TRIGGER users_name_search_refresh
BEFORE INSERT OR UPDATE OF name ON users
FOR EACH ROW EXECUTE FUNCTION users_name_search_refresh();

CREATE INDEX users_name_search_trgm_idx
  ON users USING GIN (name_search gin_trgm_ops);

SELECT id, name
FROM users
WHERE name_search % unaccent($1)
ORDER BY similarity(name_search, unaccent($1)) DESC
LIMIT 10;
```

Les exemples de trigger évitent d'utiliser `unaccent()` dans des expressions de colonne générée ou d'index, où les règles d'immutabilité de PostgreSQL comptent. Si vous enveloppez `unaccent()` dans votre propre fonction immuable, documentez que vous acceptez le risque de mise à jour/configuration.

## Extensions Notables

**`pg_trgm`** est inclus avec la plupart des distributions Postgres mais nécessite une activation explicite. Le fondement pour la correspondance de chaînes floue dans Postgres.

**`unaccent`** retire les marques diacritiques avant l'indexation et la requête. S'associe bien avec `pg_trgm` et la RFT pour le contenu de langues européennes. Inclus avec Postgres.

**`pg_bigm`** étend l'approche de trigramme aux bigrammes (segments de 2 caractères), ce qui améliore significativement les résultats pour les langues CJK (Chinois, Japonais, Coréen) où `pg_trgm` sous-performe. Doit être installé séparément ; non inclus.

**`pg_search`** (de [ParadeDB](https://www.paradedb.com/)) remplace l'empilement standard `GIN` / `tsvector` par un index BM25 basé sur Tantivy. Cela vous donne le scoring BM25 (souvent meilleur que `ts_rank`), la correspondance floue dans les requêtes RFT, la recherche facettée, et un indexation considérablement plus rapide sur les grandes tables. C'est un chemin de mise à niveau sans changement quand la RFT standard commence à montrer des limites de classement ou de performance.

```sql
-- pg_search : recherche textuelle BM25 avec correspondance floue
CREATE INDEX posts_bm25_idx ON posts
  USING bm25 (id, title, body)
  WITH (key_field = 'id', text_fields = '{"title": {}, "body": {}}');

-- Requête avec scoring BM25 + correspondance floue (attrape "javascipt")
SELECT id, title, paradedb.score(id) AS rank
FROM posts
WHERE posts @@@ paradedb.fuzzy_phrase(field => 'title', value => 'postgres performnce')
ORDER BY rank DESC
LIMIT 10;
```

**`pgvector`** ajoute le stockage de vecteurs denses et la recherche de similarité. C'est le bon outil quand les utilisateurs décrivent ce qu'ils veulent plutôt que de le nommer — recherche sémantique, RAG, recommandations de contenu connexe, requêtes multilingues. Couvert en profondeur dans [Recherche Vectorielle Sémantique et Stratégies Hybrides](/fr/semantic-vector-search-landscape).

## Tableau de Décision

| Ce que vous recherchez | Recommandé |
|---|---|
| Articles de prose, docs, tickets | RFT |
| Noms de personnes/sociétés avec fautes de frappe | `pg_trgm` |
| Autocomplétion, recherche par préfixe | `pg_trgm` |
| Codes courts, identifiants | `pg_trgm` |
| Logs de messages pour mots-clés | RFT |
| Noms internationaux | `pg_trgm` + `unaccent` |
| Contenu large, meilleur classement | `pg_search` (BM25 ParadeDB) |
| Clés primaires, emails exacts, IDs | Index B-tree |
| Dates, plages, listes triées | Index B-tree |
| Permissions, catégories, filtres | Clause WHERE régulière |
| Questions, paraphrases, concepts | pgvector (voir article suivant) |

Quand vous hésitez : chaînes courtes avec variation d'orthographe → trigrammes. Prose longue pour requêtes par mots-clés → RFT. Identifiants structurés → index réguliers. Requêtes conceptuelles ou en langage naturel → pgvector.

## Recherche Hybride : Deux Signaux, Un Classement

Quand une requête comme `"withRetry timeout errors"` frappe une boîte de recherche, elle porte deux kinds d'intention : des noms de symboles exacts que l'utilisateur connaît (`withRetry`) et une description conceptuelle (`timeout errors`). Aucun primitif unique ne couvre les deux. Exécuter la RFT et la recherche vectorielle en parallèle — puis fusionner leurs listes classées avec la Fusion de Rang Réciproque — fait.

Le scoring RRF score chaque résultat comme `1 / (60 + rank)` dans chaque liste et somme à travers les listes. La constante 60 amortit l'avantage des rangs supérieurs, donc un résultat qui se place deuxième dans les deux listes peut battre un résultat qui gagne une liste et rate l'autre entièrement. Crucialement, RRF ne moyenne jamais les scores bruts à travers les méthodes — le rang RFT et la distance cosinus sont des devises différentes et ne peuvent pas être combinés arithmétiquement.

```sql
-- Recherche hybride : RFT + pgvector fusionné avec RRF
WITH fts AS (
  SELECT id, ts_rank(search_vector, query) AS score,
         ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM docs, to_tsquery('english', 'withRetry & timeout') query
  WHERE search_vector @@ query
  LIMIT 60
),
vec AS (
  SELECT id,
         ROW_NUMBER() OVER (ORDER BY embedding <=> $embedding) AS rank
  FROM docs
  ORDER BY embedding <=> $embedding
  LIMIT 60
)
SELECT COALESCE(fts.id, vec.id) AS id,
       (COALESCE(1.0 / (60 + fts.rank), 0) +
        COALESCE(1.0 / (60 + vec.rank), 0)) AS rrf_score
FROM fts FULL JOIN vec ON fts.id = vec.id
ORDER BY rrf_score DESC
LIMIT 10;
```

La pool de candidats de 60 documents par branche (`LIMIT 60`) est un point de départ commun. Élargissez-le si le rappel est faible ; rétrécissez-le pour la vitesse.

## Ce Qui Vient Après

La recherche textuelle Postgres couvre beaucoup de terrain, mais elle a un plafond. Quand les utilisateurs décrivent ce qu'ils veulent plutôt que de le nommer — "quelque chose pour m'aider à dormir en vol", "articles sur le débogage de la confiance en tant que nouvel ingénieur" — la recherche lexicale et trigramme échouent toutes les deux.

C'est le territoire des embeddings vectoriels, de la recherche sémantique, et des architectures hybrides. Couvert dans [Recherche Vectorielle Sémantique et Stratégies Hybrides](/fr/semantic-vector-search-landscape).
