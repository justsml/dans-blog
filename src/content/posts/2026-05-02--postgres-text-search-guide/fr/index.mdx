La plupart des équipes utilisent un seul outil de recherche PostgreSQL. Les équipes qui connaissent les trois livrent une meilleure recherche avec moins de complexité — et évitent le détour coûteux vers un service de recherche dédié dont elles n'avaient pas encore besoin.

Ce guide couvre l'ensemble des options natives de PostgreSQL : ce que chacune fait, quand elle est adaptée, et comment les superposer.

---

## Les Trois Outils

**La recherche plein texte** (`tsvector` / index `GIN`) est lexicale. Elle tokenise le texte en lexèmes, les normalise, et fait correspondre les requêtes à l'index. « Courir » et « couru » se réduisent au même lexème. Idem pour « chien » et « chiens ». La fonction de classement (`ts_rank`) récompense les documents où les termes de la requête apparaissent souvent ou de manière proéminente.

**Les trigrammes** (`pg_trgm`) découpent les chaînes en tranches chevauchantes de 3 caractères et mesurent combien de tranches deux chaînes partagent. « Dan » → `" da"`, `"dan"`, `"an "`. « Micheal » et « Michael » partagent la plupart de leurs trigrammes, donc la similarité est élevée. Cela rend `pg_trgm` excellent pour la recherche approximative de noms, la tolérance aux fautes de frappe et l'autocomplétion — le domaine où la recherche plein texte performe mal.

**Les index de correspondance exacte** (B-tree, hash) gèrent les clés primaires, les adresses email, les identifiants, les SKU, et tout ce où la réponse est binaire : ça correspond ou pas. Ce ne ressemble pas à de la « recherche », mais ces index ont leur place dans cette discussion car la pire pratique est d'utiliser une recherche floue ou sémantique pour des problèmes qui ont des réponses correctes.

Le choix n'est pas une question de sophistication. Il s'agit d'associer l'outil à la forme de la requête.

## Quand la Recherche Plein Texte Gagne

**Recherche de mots-clés dans de la prose.** Articles de blog, documentation, descriptions de produits, tickets d'assistance, documents juridiques. La recherche plein texte a été conçue pour cette forme de contenu : recherche indexée et classée sur du texte en langage naturel.

**Requêtes utilisateur basées sur des mots-clés.** Les utilisateurs tapent un terme de recherche, filtrent par catégorie, ou parcourent par mot-clé. La recherche plein texte gère cette intention nativement, sans infrastructure d'embedding.

**Résultats classés sans dépendances externes.** Les index de recherche plein texte sont rapides, déterministes et ne nécessitent aucun appel API. Le signal de pertinence provient de la fréquence des termes pondérée par la position dans le champ.

**Filtrage booléen combiné à la recherche.** La recherche plein texte se compose naturellement avec votre logique de requête existante :

```sql
SELECT * FROM posts
WHERE search_vector @@ to_tsquery('english', 'postgres & performance')
  AND category = 'tutorial'
  AND published_at > NOW() - INTERVAL '6 months';
```

### Configuration de la Recherche Plein Texte

```sql
-- Une colonne générée maintient l'index à jour automatiquement
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body,  '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Requête
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`setweight` assigne une importance : `A` (titre) surclasse `B` (corps). C'est tout le modèle de pertinence pour la plupart des cas de recherche de contenu.

### Ce que la Recherche Plein Texte Gère Mal

- Les fautes de frappe dans les requêtes — « javascipt » ne correspondra pas à « javascript »
- Les noms de personnes, adresses, noms propres qui ne se normalisent pas de manière prévisible
- La recherche par préfixe/autocomplétion sans configuration spéciale
- Les requêtes où l'utilisateur décrit un concept plutôt que de le nommer

## Quand les Trigrammes Gagnent (`pg_trgm`)

`pg_trgm` couvre le terrain médian difficile que la recherche plein texte rate systématiquement.

La recherche plein texte tokenise le texte en lexèmes et les normalise. Pour la prose c'est correct. Pour les noms et les identifiants courts, ça ne l'est souvent pas :

- Noms de personnes (« Dan Levy » → normalisé différemment selon le dictionnaire et la configuration linguistique)
- Noms d'entreprises, adresses, titres de produits où l'orthographe exacte importe
- Requêtes avec fautes de frappe — « Micheal Jordan », « Amaon », « javascipt »
- Autocomplétion / recherche par préfixe
- Correspondance partielle de chaîne (« son » correspondant à « Johnson », « Anderson »)

`pg_trgm` est également indépendant de la langue, ce qui importe pour les noms issus de diverses origines linguistiques. La recherche plein texte nécessite une configuration de dictionnaire par langue.

### Recherche Approximative de Noms

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- Trouve « Micheal Jordan » quand on cherche « Michael Jordan »
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- opérateur % = seuil de similarité (défaut 0.3)
ORDER BY score DESC
LIMIT 10;
```

L'opérateur `%` utilise `pg_trgm.similarity_threshold` (défaut 0.3, plage 0–1). Pour la recherche de noms, 0.3–0.4 capte les fautes de frappe tout en maintenant un bruit faible.

### Autocomplétion, Préfixe et Recherche par Contenu

```sql
-- Recherche par préfixe pour l'autocomplétion. Un index GIN de trigrammes peut aider,
-- mais un index B-tree avec pattern peut être meilleur pour les préfixes ancrés à gauche.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- word_similarity pour les correspondances partielles dans des chaînes plus longues
-- (« Johnson » dans « Andrew Johnson III »)
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name
ORDER BY score DESC
LIMIT 10;
```

L'index GIN de trigrammes est particulièrement utile pour les requêtes de type `ILIKE '%pattern%'` et la correspondance tolérante aux fautes — des motifs qui sont généralement des parcours de table complets sans index de trigrammes.

### Quand Utiliser pg_trgm Plutôt que la Recherche Plein Texte

| Scénario | Utilisez |
|---|---|
| Recherche de noms de personnes/entreprises avec fautes | `pg_trgm` |
| Autocomplétion / recherche par préfixe | `pg_trgm` (ou FTS avec requêtes de préfixe) |
| Chaînes courtes, identifiants, codes | `pg_trgm` |
| Articles en prose, documentation, tickets | FTS |
| Messages de journal pour mots-clés | FTS |
| Recherche multilingue de noms | `pg_trgm` (indépendant de la langue) |

## Quand les Requêtes SQL de Correspondance Exacte Gagnent

Certains problèmes de « recherche » n'en sont pas du tout.

« Trouver l'utilisateur avec l'email `dan@example.com` » est une vérification d'égalité. « Trouver la commande `ORD-12345` » est une recherche par clé primaire. « Lister les articles dans la catégorie `tutorial` triés par date » est une requête filtrée. Ces cas relèvent des index B-tree ou hash.

Utiliser la recherche plein texte ou les trigrammes ici ajoute de la complexité sans améliorer la correction — et pour les identifiants exacts, une correspondance approximative est pire qu'aucune correspondance.

```sql
CREATE INDEX users_email_idx ON users (email);

-- Recherche exacte : rapide et sans ambiguïté
SELECT id, name FROM users WHERE email = $1;
```

La leçon plus large : la recherche approximative pour des problèmes qui ont des réponses correctes est une erreur de catégorie. Elle retourne *quelque chose* — qui peut être confiant et faux.

## Combiner Ces Outils

Ces outils se composent proprement. Vous n'en choisissez pas un seul.

**FTS + pg_trgm pour une barre de recherche tolérante aux fautes dans les mots-clés :**

```sql
-- La similarité de trigrammes sur le titre capte les fautes ; ts_rank gère la pertinence du corps
SELECT id, title,
  ts_rank(search_vector, to_tsquery('simple', $1)) AS fts_rank,
  similarity(title, $1) AS trgm_score
FROM posts
WHERE search_vector @@ to_tsquery('simple', $1)
   OR title % $1
ORDER BY (ts_rank(search_vector, to_tsquery('simple', $1)) + similarity(title, $1)) DESC
LIMIT 10;
```

**FTS + `unaccent` pour le contenu international :**

```sql
-- Supprime les signes diacritiques pour que « José » corresponde à « Jose »
CREATE EXTENSION IF NOT EXISTS unaccent;

CREATE TEXT SEARCH CONFIGURATION public.simple_unaccent (COPY = pg_catalog.simple);

ALTER TEXT SEARCH CONFIGURATION public.simple_unaccent
  ALTER MAPPING FOR hword, hword_part, word
  WITH unaccent, simple;

ALTER TABLE posts ADD COLUMN search_vector tsvector;

CREATE TRIGGER posts_search_vector_refresh
BEFORE INSERT OR UPDATE OF title, body ON posts
FOR EACH ROW EXECUTE FUNCTION
  tsvector_update_trigger(search_vector, 'public.simple_unaccent', title, body);
```

**`unaccent` + `pg_trgm` pour la recherche internationale de noms :**

```sql
ALTER TABLE users ADD COLUMN name_search text;

CREATE FUNCTION users_name_search_refresh()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  NEW.name_search := unaccent(coalesce(NEW.name, ''));
  RETURN NEW;
END;
$$;

CREATE TRIGGER users_name_search_refresh
BEFORE INSERT OR UPDATE OF name ON users
FOR EACH ROW EXECUTE FUNCTION users_name_search_refresh();

CREATE INDEX users_name_search_trgm_idx
  ON users USING GIN (name_search gin_trgm_ops);

SELECT id, name
FROM users
WHERE name_search % unaccent($1)
ORDER BY similarity(name_search, unaccent($1)) DESC
LIMIT 10;
```

Les exemples avec trigger évitent d'utiliser `unaccent()` dans des expressions de colonnes générées ou d'index, là où les règles d'immutabilité de PostgreSQL s'appliquent. Si vous encapsulez `unaccent()` dans votre propre fonction immutable, documentez que vous acceptez un risque de mise à jour/configuration.

## Extensions Notables

**`pg_trgm`** est fourni avec la plupart des distributions PostgreSQL mais nécessite une activation explicite. Le fondement de la correspondance approximative de chaînes dans Postgres.

**`unaccent`** supprime les signes diacritiques avant l'indexation et l'interrogation. Se marie bien avec `pg_trgm` et la recherche plein texte pour le contenu en langues européennes. Fourni avec PostgreSQL.

**`pg_bigm`** étend l'approche des trigrammes aux bigrammes (tranches de 2 caractères), ce qui améliore significativement les résultats pour les langues CJK (chinois, japonais, coréen) où `pg_trgm` est moins performant. Doit être installé séparément ; non fourni.

**`pg_search`** (de [ParadeDB](https://www.paradedb.com/)) remplace la pile standard `GIN` / `tsvector` par un index BM25 basé sur Tantivy. Cela offre un score BM25 (souvent meilleur que `ts_rank`), une correspondance floue dans les requêtes plein texte, une recherche à facettes, et une indexation considérablement plus rapide sur les grandes tables. C'est une voie de mise à niveau directe lorsque la recherche plein texte standard commence à montrer ses limites en matière de classement ou de performance.

```sql
-- pg_search : recherche plein texte BM25 avec correspondance floue
CREATE INDEX posts_bm25_idx ON posts
  USING bm25 (id, title, body)
  WITH (key_field = 'id', text_fields = '{"title": {}, "body": {}}');

-- Requête avec score BM25 + correspondance floue (capte « javascipt »)
SELECT id, title, paradedb.score(id) AS rank
FROM posts
WHERE posts @@@ paradedb.fuzzy_phrase(field => 'title', value => 'postgres performnce')
ORDER BY rank DESC
LIMIT 10;
```

**`pgvector`** ajoute le stockage de vecteurs denses et la recherche par similarité. C'est l'outil adapté quand les utilisateurs décrivent ce qu'ils veulent plutôt que de le nommer — recherche sémantique, RAG, recommandations de contenu connexe, requêtes multilingues. Couvert en détail dans [Semantic Vector Search and Hybrid Strategies](/semantic-vector-search-landscape).

| Ce que vous recherchez | Recommandé |
|---|---|
| Articles en prose, documentation, tickets | FTS |
| Noms de personnes/entreprises avec fautes | `pg_trgm` |
| Autocomplétion, recherche par préfixe | `pg_trgm` |

En cas de doute : chaînes courtes avec variation orthographique → trigrammes. Prose longue pour requêtes par mots-clés → FTS. Identifiants structurés → index classiques. Requêtes conceptuelles ou en langage naturel → pgvector.

## Recherche Hybride : Deux Signaux, Un Classement

Quand une requête comme `"withRetry timeout errors"` arrive dans une barre de recherche, elle porte deux types d'intention : des noms de symboles exacts que l'utilisateur connaît (`withRetry`) et une description conceptuelle (`timeout errors`). Aucune primitive seule ne couvre les deux. Exécuter la recherche plein texte et la recherche vectorielle en parallèle — puis fusionner leurs listes classées avec le Reciprocal Rank Fusion — le fait.

RRF score chaque résultat comme `1 / (60 + rang)` dans chaque liste et somme les scores entre listes. La constante 60 atténue l'avantage des premiers rangs, de sorte qu'un résultat classé deuxième dans les deux listes peut battre un résultat qui gagne une liste et manque complètement l'autre. Crucialement, RRF ne moyenne jamais les scores bruts entre méthodes — le rang FTS et la distance cosinus sont des devises différentes et ne peuvent pas être combinés arithmétiquement.

```sql
-- Recherche hybride : FTS + pgvector fusionnés avec RRF
WITH fts AS (
  SELECT id, ts_rank(search_vector, query) AS score,
         ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM docs, to_tsquery('english', 'withRetry & timeout') query
  WHERE search_vector @@ query
  LIMIT 60
),
vec AS (
  SELECT id,
         ROW_NUMBER() OVER (ORDER BY embedding <=> $embedding) AS rank
  FROM docs
  ORDER BY embedding <=> $embedding
  LIMIT 60
)
SELECT COALESCE(fts.id, vec.id) AS id,
       (COALESCE(1.0 / (60 + fts.rank), 0) +
        COALESCE(1.0 / (60 + vec.rank), 0)) AS rrf_score
FROM fts FULL JOIN vec ON fts.id = vec.id
ORDER BY rrf_score DESC
LIMIT 10;
```

Le pool de 60 documents candidats par branche (`LIMIT 60`) est un point de départ courant. Élargissez-le si le rappel est faible ; réduisez-le pour la vitesse.
