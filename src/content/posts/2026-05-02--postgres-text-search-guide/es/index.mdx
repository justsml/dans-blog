La mayoría de los equipos utilizan una única herramienta de búsqueda en Postgres. Los equipos que conocen las tres ofrecen una mejor búsqueda con menos complejidad —y evitan el costoso desvío hacia un servicio de búsqueda dedicado que aún no necesitaban.

Esta guía cubre el conjunto completo de opciones nativas de Postgres: qué hace cada una, cuándo es la opción adecuada y cómo combinarlas.

---

## Las tres herramientas

**La búsqueda de texto completo** (índice `tsvector` / `GIN`) es léxica. Divide el texto en lexemas, los reduce a su raíz (stemming) y compara las consultas con el índice. "Corriendo" y "correr" se reducen al mismo lexema. Lo mismo ocurre con "perro" y "perros". La función de clasificación (`ts_rank`) premia los documentos donde los términos de la consulta aparecen con frecuencia o de forma destacada.

**Los trigramas** (`pg_trgm`) dividen las cadenas en fragmentos superpuestos de 3 caracteres y miden cuántos fragmentos comparten dos cadenas. "Dan" → `" da"`, `"dan"`, `"an "`. "Micheal" y "Michael" comparten la mayoría de sus trigramas, por lo que la similitud es alta. Esto hace que `pg_trgm` sea excelente para la coincidencia difusa de nombres, la tolerancia a erratas y el autocompletado —el espacio donde FTS rinde mal.

**Los índices de coincidencia exacta** (B-tree, hash) manejan claves primarias, direcciones de correo electrónico, IDs, SKUs y cualquier cosa donde la respuesta sea binaria: coincide o no coincide. Estos no parecen "búsqueda", pero pertenecen a esta conversación porque el peor patrón es usar búsqueda difusa o semántica para problemas que tienen respuestas exactas.

La elección no depende de la sofisticación. Se trata de ajustar la herramienta a la forma de la consulta.

## Cuándo gana la búsqueda de texto completo (FTS)

**Buscando palabras clave en prosa.** Artículos de blog, documentación, descripciones de productos, tickets de soporte, documentos legales. FTS fue diseñado para esta forma de contenido: recuperación indexada y clasificada sobre texto en lenguaje natural.

**Consultas de usuario basadas en palabras clave.** Los usuarios escriben un término de búsqueda, filtran por etiqueta o navegan por palabra clave. FTS maneja esa intención de forma nativa sin ninguna infraestructura de embeddings.

**Resultados clasificados sin dependencias externas.** Los índices FTS son rápidos, deterministas y no requieren llamadas a APIs. La señal de relevancia proviene de la frecuencia de los términos ponderada por la posición del campo.

**Filtrado booleano junto con la búsqueda.** FTS se compone de forma natural con tu lógica de consulta existente:

```sql
SELECT * FROM posts
WHERE search_vector @@ to_tsquery('spanish', 'postgres & rendimiento')
  AND category = 'tutorial'
  AND published_at > NOW() - INTERVAL '6 months';
```

### Configurando FTS

```sql
-- La columna generada mantiene el índice actualizado automáticamente
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('spanish', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('spanish', coalesce(body,  '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Consulta
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('spanish', 'postgres & rendimiento') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`setweight` asigna importancia: `A` (título) supera a `B` (cuerpo). Ese es el modelo de relevancia completo para la mayoría de los casos de uso de búsqueda de contenido.

### Qué no maneja bien FTS

- Erratas en las consultas — "javascipt" no coincidirá con "javascript".
- Nombres de personas, direcciones, nombres propios que no se reducen a la raíz de forma predecible.
- Prefijos/autocompletado sin una configuración especial.
- Consultas donde el usuario describe un concepto en lugar de nombrarlo.

## Cuándo ganan los trigramas (`pg_trgm`)

`pg_trgm` cubre el terreno incómodo en el que FTS suele fallar.

FTS tokeniza el texto en lexemas y los reduce a su raíz. Para la prosa, esto es correcto. Para nombres e identificadores cortos, a menudo no lo es:

- Nombres de personas ("Dan Levy" → se reduce de forma diferente según el diccionario y la configuración de idioma).
- Nombres de empresas, direcciones, títulos de productos donde la ortografía exacta importa.
- Consultas con erratas — "Micheal Jordan", "Amaon", "javascipt".
- Autocompletado / búsqueda por prefijo.
- Coincidencia parcial de cadenas ("son" coincidiendo con "Johnson", "Anderson").

`pg_trgm` también es agnóstico al idioma, lo que importa para nombres de diversos orígenes lingüísticos. FTS requiere configuración de diccionario por idioma.

### Búsqueda difusa de nombres

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- Encuentra "Micheal Jordan" cuando se busca "Michael Jordan"
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- operador % = umbral de similitud (por defecto 0.3)
ORDER BY score DESC
LIMIT 10;
```

El operador `%` utiliza `pg_trgm.similarity_threshold` (por defecto 0.3, rango 0–1). Para la búsqueda de nombres, 0.3–0.4 captura erratas manteniendo el ruido bajo.

### Autocompletado, prefijo y búsqueda de contenido

```sql
-- Coincidencia de prefijo para autocompletado. Un índice GIN de trigramas puede ayudar,
-- pero un índice de patrón B-tree puede ser mejor para prefijos puros anclados a la izquierda.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- word_similarity para coincidencias parciales dentro de cadenas más largas
-- ("Johnson" dentro de "Andrew Johnson III")
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name
ORDER BY score DESC
LIMIT 10;
```

El índice GIN de trigramas es especialmente útil para consultas `ILIKE '%patrón%'` y coincidencias tolerantes a erratas —patrones que suelen ser escaneos de tabla completa sin un índice de trigramas.

### Cuándo recurrir a pg_trgm en lugar de FTS

| Escenario | Uso |
|---|---|
| Búsqueda de nombres de personas/empresas con erratas | `pg_trgm` |
| Autocompletado / búsqueda por prefijo | `pg_trgm` (o FTS con consultas de prefijo) |
| Cadenas cortas, identificadores, códigos | `pg_trgm` |
| Artículos en prosa, documentación, tickets | FTS |
| Mensajes de log para palabras clave | FTS |
| Búsqueda de nombres multilingües | `pg_trgm` (agnóstico al idioma) |

## Cuándo gana el SQL de coincidencia exacta

Algunos problemas de "búsqueda" no son búsqueda en absoluto.

"Encontrar el usuario con el correo electrónico `dan@example.com`" es una comprobación de igualdad. "Encontrar el pedido `ORD-12345`" es una búsqueda por clave primaria. "Listar publicaciones en la categoría `tutorial` ordenadas por fecha" es una consulta filtrada. Estas pertenecen a índices B-tree o hash.

Usar FTS o trigramas aquí añade complejidad sin mejorar la corrección —y para identificadores exactos, una coincidencia cercana es peor que ninguna coincidencia.

```sql
CREATE INDEX users_email_idx ON users (email);

-- Búsqueda exacta: rápida y sin ambigüedades
SELECT id, name FROM users WHERE email = $1;
```

La lección más amplia: la búsqueda aproximada para problemas con respuestas exactas es un error de categoría. Devuelve *algo*, que puede estar equivocado con total confianza.

## Combinando estas herramientas

Estas herramientas se componen limpiamente. No tienes que elegir exactamente una.

**FTS + pg_trgm para un cuadro de búsqueda que tolera erratas en palabras clave:**

```sql
-- La similitud de trigramas en el título captura erratas; ts_rank maneja la relevancia del cuerpo
SELECT id, title,
  ts_rank(search_vector, to_tsquery('simple', $1)) AS fts_rank,
  similarity(title, $1) AS trgm_score
FROM posts
WHERE search_vector @@ to_tsquery('simple', $1)
   OR title % $1
ORDER BY (ts_rank(search_vector, to_tsquery('simple', $1)) + similarity(title, $1)) DESC
LIMIT 10;
```

**FTS + `unaccent` para contenido internacional:**

```sql
-- Elimina los signos diacríticos para que "José" coincida con "Jose"
CREATE EXTENSION IF NOT EXISTS unaccent;

CREATE TEXT SEARCH CONFIGURATION public.simple_unaccent (COPY = pg_catalog.simple);

ALTER TEXT SEARCH CONFIGURATION public.simple_unaccent
  ALTER MAPPING FOR hword, hword_part, word
  WITH unaccent, simple;

ALTER TABLE posts ADD COLUMN search_vector tsvector;

CREATE TRIGGER posts_search_vector_refresh
BEFORE INSERT OR UPDATE OF title, body ON posts
FOR EACH ROW EXECUTE FUNCTION
  tsvector_update_trigger(search_vector, 'public.simple_unaccent', title, body);
```

**`unaccent` + `pg_trgm` para búsqueda de nombres internacionales:**

```sql
ALTER TABLE users ADD COLUMN name_search text;

CREATE FUNCTION users_name_search_refresh()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  NEW.name_search := unaccent(coalesce(NEW.name, ''));
  RETURN NEW;
END;
$$;

CREATE TRIGGER users_name_search_refresh
BEFORE INSERT OR UPDATE OF name ON users
FOR EACH ROW EXECUTE FUNCTION users_name_search_refresh();

CREATE INDEX users_name_search_trgm_idx
  ON users USING GIN (name_search gin_trgm_ops);

SELECT id, name
FROM users
WHERE name_search % unaccent($1)
ORDER BY similarity(name_search, unaccent($1)) DESC
LIMIT 10;
```

Los ejemplos de disparadores evitan el uso de `unaccent()` dentro de columnas generadas o expresiones de índice, donde las reglas de inmutabilidad de PostgreSQL son importantes. Si envuelves `unaccent()` en tu propia función inmutable, documenta que estás aceptando riesgos de actualización/configuración.

## Extensiones notables

**`pg_trgm`** se incluye en la mayoría de las distribuciones de Postgres, pero requiere habilitación explícita. Es la base para la coincidencia de cadenas difusas en Postgres.

**`unaccent`** elimina los signos diacríticos antes de indexar y consultar. Combina bien tanto con `pg_trgm` como con FTS para contenido en idiomas europeos. Se incluye con Postgres.

**`pg_bigm`** amplía el enfoque de trigramas a bigramas (fragmentos de 2 caracteres), lo que mejora significativamente los resultados para idiomas CJK (chino, japonés, coreano) donde `pg_trgm` rinde por debajo de lo esperado. Debe instalarse por separado; no viene incluido.

**`pg_search`** (de [ParadeDB](https://www.paradedb.com/)) sustituye la pila estándar `GIN` / `tsvector` por un índice BM25 basado en Tantivy. Esto te ofrece puntuación BM25 (a menudo mejor que `ts_rank`), coincidencia difusa dentro de las consultas FTS, búsqueda facetada e indexación drásticamente más rápida en tablas grandes. Es un camino de actualización directo cuando el FTS estándar empieza a mostrar límites de clasificación o rendimiento.

```sql
-- pg_search: búsqueda de texto completo BM25 con coincidencia difusa
CREATE INDEX posts_bm25_idx ON posts
  USING bm25 (id, title, body)
  WITH (key_field = 'id', text_fields = '{"title": {}, "body": {}}');

-- Consulta con puntuación BM25 + coincidencia difusa (captura "javascipt")
SELECT id, title, paradedb.score(id) AS rank
FROM posts
WHERE posts @@@ paradedb.fuzzy_phrase(field => 'title', value => 'postgres performnce')
ORDER BY rank DESC
LIMIT 10;
```

**`pgvector`** añade almacenamiento de vectores densos y búsqueda de similitud. Es la herramienta adecuada cuando los usuarios describen lo que quieren en lugar de nombrarlo —búsqueda semántica, RAG, recomendaciones de contenido relacionado, consultas multilingües. Cubierto en profundidad en [Panorama de la búsqueda semántica de vectores y estrategias híbridas](/semantic-vector-search-landscape).

## Tabla de decisión

| Qué estás buscando | Recomendado |
|---|---|
| Artículos en prosa, docs, tickets | FTS |
| Nombres de personas/empresas con erratas | `pg_trgm` |
| Autocompletado, búsqueda por prefijo | `pg_trgm` |
| Códigos cortos, identificadores | `pg_trgm` |
| Mensajes de log para palabras clave | FTS |
| Nombres internacionales | `pg_trgm` + `unaccent` |
| Contenido grande, mejor clasificación | `pg_search` (ParadeDB BM25) |
| Claves primarias, correos exactos, IDs | Índice B-tree |
| Fechas, rangos, listas ordenadas | Índice B-tree |
| Permisos, categorías, filtros | Cláusula WHERE regular |
| Preguntas, paráfrasis, conceptos | pgvector (ver siguiente artículo) |

En caso de duda: cadenas cortas con variación ortográfica → trigramas. Prosa larga para consultas por palabras clave → FTS. Identificadores estructurados → índices regulares. Consultas conceptuales o en lenguaje natural → pgvector.

## Búsqueda híbrida: dos señales, un rango

Cuando una consulta como `"withRetry timeout errors"` llega a un cuadro de búsqueda, lleva dos tipos de intención: nombres de símbolos exactos que el usuario conoce (`withRetry`) y una descripción conceptual (`timeout errors`). Ninguna primitiva por sí sola cubre ambos. Ejecutar FTS y búsqueda vectorial en paralelo —y luego fusionar sus listas clasificadas con Reciprocal Rank Fusion— sí lo hace.

RRF puntúa cada resultado como `1 / (60 + rango)` en cada lista y suma las listas. La constante 60 amortigua la ventaja de los primeros rangos, de modo que un resultado que ocupe el segundo lugar en ambas listas puede superar a un resultado que gane una lista y no aparezca en la otra. Crucialmente, RRF nunca promedia puntuaciones brutas entre métodos —el rango FTS y la distancia del coseno son monedas diferentes y no pueden combinarse aritméticamente.

```sql
-- Búsqueda híbrida: FTS + pgvector fusionados con RRF
WITH fts AS (
  SELECT id, ts_rank(search_vector, query) AS score,
         ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM docs, to_tsquery('spanish', 'withRetry & timeout') query
  WHERE search_vector @@ query
  LIMIT 60
),
vec AS (
  SELECT id,
         ROW_NUMBER() OVER (ORDER BY embedding <=> $embedding) AS rank
  FROM docs
  ORDER BY embedding <=> $embedding
  LIMIT 60
)
SELECT COALESCE(fts.id, vec.id) AS id,
       (COALESCE(1.0 / (60 + fts.rank), 0) +
        COALESCE(1.0 / (60 + vec.rank), 0)) AS rrf_score
FROM fts FULL JOIN vec ON fts.id = vec.id
ORDER BY rrf_score DESC
LIMIT 10;
```

El grupo de candidatos de 60 documentos por rama (`LIMIT 60`) es un punto de partida común. Amplíalo si la recuperación es baja; redúcelo para ganar velocidad.

## Qué sigue

La búsqueda de texto en Postgres cubre mucho terreno, pero tiene un límite. Cuando los usuarios describen lo que quieren en lugar de nombrarlo —"algo que me ayude a dormir en un vuelo", "artículos sobre la confianza en la depuración como ingeniero novel"— tanto la búsqueda léxica como la de trigramas fallan.

Ese es el territorio de los embeddings vectoriales, la búsqueda semántica y las arquitecturas híbridas. Cubierto en [Panorama de la búsqueda semántica de vectores y estrategias híbridas](/semantic-vector-search-landscape).
