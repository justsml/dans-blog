La mayoría de equipos usan una sola herramienta de búsqueda de Postgres. Los equipos que conocen las tres envían mejor búsqueda con menos complejidad — y evitan el desvío costoso hacia un servicio de búsqueda dedicado que aún no necesitaban.

Esta guía cubre el conjunto completo de opciones nativas de Postgres: qué hace cada una, cuándo es la elección correcta, y cómo combinarlas.

---

## Las Tres Herramientas

**Búsqueda de texto completo** (`tsvector` / índice `GIN`) es léxica. Tokeniza el texto en lexemas, los stemming, y hace coincidir consultas contra el índice. "Corriendo" y "corre" se colapsan al mismo lexema. Lo mismo pasa con "perro" y "perros." La función de ranking (`ts_rank`) premia documentos donde los términos de la consulta aparecen frecuentemente o en posiciones prominentes.

**Trigrams** (`pg_trgm`) dividen cadenas en fragmentos superpuestos de 3 caracteres y miden cuántos fragmentos comparten dos cadenas. "Dan" → `" da"`, `"dan"`, `"an "`. "Micheal" y "Michael" comparten la mayoría de sus trigramas, por lo que la similitud es alta. Esto hace que `pg_trgm` sea excelente para coincidencia difusa de nombres, tolerancia a errores tipográficos, y autocompletado — el espacio donde FTS tiene bajo rendimiento.

**Índices de coincidencia exacta** (B-tree, hash) manejan claves primarias, direcciones de email, IDs, SKUs, y cualquier cosa donde la respuesta es binaria: coincide o no coincide. Estos no se sienten como "búsqueda," pero pertenecen a esta conversación porque el peor patrón es usar búsqueda difusa o semántica para problemas que tienen respuestas correctas.

La elección no se trata de sofisticación. Se trata de hacer coincidir la herramienta con la forma de la consulta.

## Cuándo la Búsqueda de Texto Completo Gana

**Buscar en prosa palabras clave.** Publicaciones de blog, documentación, descripciones de productos, tickets de soporte, documentos legales. FTS fue diseñada para esta forma de contenido: recuperación indexada y rankeada sobre texto de lenguaje natural.

**Consultas de usuario basadas en palabras clave.** Los usuarios escriben un término de búsqueda, filtran por etiqueta, o navegan por palabra clave. FTS maneja esa intención de forma nativa sin ninguna infraestructura de embeddings.

**Resultados rankeados sin dependencias externas.** Los índices FTS son rápidos, determinísticos, y no requieren llamadas a API. La señal de relevancia viene de la frecuencia de términos ponderada por la posición del campo.

**Filtrado booleano junto con la búsqueda.** FTS se composa naturalmente con tu lógica de consulta existente:

```sql
SELECT * FROM posts
WHERE search_vector @@ to_tsquery('english', 'postgres & performance')
  AND category = 'tutorial'
  AND published_at > NOW() - INTERVAL '6 months';
```

### Configurando FTS

```sql
-- La columna generada mantiene el índice actualizado automáticamente
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body,  '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Consulta
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`setweight` asigna importancia: `A` (título) supera a `B` (cuerpo). Ese es el modelo de relevancia completo para la mayoría de casos de búsqueda de contenido.

### Lo que FTS No Maneja Bien

- Errores tipográficos en consultas — "javascipt" no encontrará "javascript"
- Nombres de personas, direcciones, sustantivos propios que no se stemming de forma predecible
- Autocompletado/prefijo sin configuración especial
- Consultas donde el usuario describe un concepto en lugar de nombrarlo

## Cuándo los Trigrams Ganen (`pg_trgm`)

`pg_trgm` cubre el awkward middle que FTS consistentemente falla.

FTS tokeniza texto en lexemas y los stemming. Para prosa esto es correcto. Para nombres y identificadores cortos frecuentemente no lo es:

- Nombres de personas ("Dan Levy" → stemming diferente dependiendo del diccionario y configuración de idioma)
- Nombres de empresas, direcciones, títulos de productos donde la ortografía exacta importa
- Consultas con errores tipográficos — "Micheal Jordan", "Amaon", "javascipt"
- Autocompletado / búsqueda de prefijos
- Coincidencia de cadenas parciales ("son" coincide con "Johnson", "Anderson")

`pg_trgm` también es agnóstico de idioma, lo que importa para nombres de fondos lingüísticos diversos. FTS requiere configuración de diccionario por idioma.

### Búsqueda Difusa de Nombres

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- Encuentra "Micheal Jordan" cuando se busca "Michael Jordan"
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % operador = umbral de similitud (default 0.3)
ORDER BY score DESC
LIMIT 10;
```

El operador `%` usa `pg_trgm.similarity_threshold` (default 0.3, rango 0–1). Para búsqueda de nombres, 0.3–0.4 captura errores tipográficos mientras mantiene el ruido bajo.

### Autocompletado, Prefijo y Búsqueda Contains

```sql
-- Coincidencia de prefijo para autocompletado. Un índice GIN de trigramas puede ayudar,
-- pero un índice B-tree de patrón puede ser mejor para prefijos anclados a la izquierda.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- word_similarity para coincidencias parciales dentro de cadenas más largas
-- ("Johnson" dentro de "Andrew Johnson III")
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name
ORDER BY score DESC
LIMIT 10;
```

El índice GIN de trigramas es especialmente útil para consultas contains `ILIKE '%pattern%'` y coincidencia tolerante a errores tipográficos — patrones que usualmente son scans de tabla completa sin un índice de trigramas.

### Cuándo Preferir pg_trgm sobre FTS

| Escenario | Usar |
|---|---|
| Búsqueda de nombres de personas/empresas con errores tipográficos | `pg_trgm` |
| Autocompletado / búsqueda de prefijos | `pg_trgm` (o FTS con consultas de prefijo) |
| Cadenas cortas, identificadores, códigos | `pg_trgm` |
| Artículos en prosa, documentación, tickets | FTS |
| Mensajes de log por palabras clave | FTS |
| Búsqueda de nombres multilingüe | `pg_trgm` (agnóstico de idioma) |

## Cuándo la Coincidencia Exacta SQL Gana

Algunos problemas de "búsqueda" no son búsqueda en absoluto.

"Encontrar el usuario con email `dan@example.com`" es una verificación de igualdad. "Encontrar orden `ORD-12345`" es una búsqueda de clave primaria. "Listar posts en la categoría `tutorial` ordenados por fecha" es una consulta filtrada. Estos pertenecen a índices B-tree o hash.

Usar FTS o trigrams aquí añade complejidad sin mejorar la corrección — y para identificadores exactos, una coincidencia cercana es peor que ninguna coincidencia.

```sql
CREATE INDEX users_email_idx ON users (email);

-- Búsqueda exacta: rápida y sin ambigüedad
SELECT id, name FROM users WHERE email = $1;
```

La lección más amplia: búsqueda aproximada para problemas con respuestas correctas es un error de categoría. Devuelve *algo* — que puede estar confidently wrong.

## Combinando Estas Herramientas

Estas herramientas se composan limpiamente. No eliges exactamente una.

**FTS + pg_trgm para una caja de búsqueda que tolera errores tipográficos en palabras clave:**

```sql
-- Similitud de trigrama en título captura errores; ts_rank maneja relevancia del cuerpo
SELECT id, title,
  ts_rank(search_vector, to_tsquery('simple', $1)) AS fts_rank,
  similarity(title, $1) AS trgm_score
FROM posts
WHERE search_vector @@ to_tsquery('simple', $1)
   OR title % $1
ORDER BY (ts_rank(search_vector, to_tsquery('simple', $1)) + similarity(title, $1)) DESC
LIMIT 10;
```

**FTS + `unaccent` para contenido internacional:**

```sql
-- Elimina marcas diacríticas para que "José" coincida con "Jose"
CREATE EXTENSION IF NOT EXISTS unaccent;

CREATE TEXT SEARCH CONFIGURATION public.simple_unaccent (COPY = pg_catalog.simple);

ALTER TEXT SEARCH CONFIGURATION public.simple_unaccent
  ALTER MAPPING FOR hword, hword_part, word
  WITH unaccent, simple;

ALTER TABLE posts ADD COLUMN search_vector tsvector;

CREATE TRIGGER posts_search_vector_refresh
BEFORE INSERT OR UPDATE OF title, body ON posts
FOR EACH ROW EXECUTE FUNCTION
  tsvector_update_trigger(search_vector, 'public.simple_unaccent', title, body);
```

**`unaccent` + `pg_trgm` para búsqueda de nombres internacional:**

```sql
ALTER TABLE users ADD COLUMN name_search text;

CREATE FUNCTION users_name_search_refresh()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  NEW.name_search := unaccent(coalesce(NEW.name, ''));
  RETURN NEW;
END;
$$;

CREATE TRIGGER users_name_search_refresh
BEFORE INSERT OR UPDATE OF name ON users
FOR EACH ROW EXECUTE FUNCTION users_name_search_refresh();

CREATE INDEX users_name_search_trgm_idx
  ON users USING GIN (name_search gin_trgm_ops);

SELECT id, name
FROM users
WHERE name_search % unaccent($1)
ORDER BY similarity(name_search, unaccent($1)) DESC
LIMIT 10;
```

Los ejemplos de triggers evitan usar `unaccent()` dentro de expresiones de columnas generadas o índices, donde las reglas de inmutabilidad de PostgreSQL importan. Si envuelves `unaccent()` en tu propia función inmutable, documenta que estás aceptando riesgo de actualización/configuración.

## Extensiones Notables

**`pg_trgm`** viene incluido con la mayoría de distribuciones de Postgres pero requiere habilitación explícita. La base para búsqueda difusa de cadenas en Postgres.

**`unaccent`** elimina marcas diacríticas antes de indexar y consultar. Combina bien con ambos `pg_trgm` y FTS para contenido en idiomas europeos. Incluye con Postgres.

**`pg_bigm`** extiende el enfoque de trigramas a bigramas (fragmentos de 2 caracteres), lo cual mejora significativamente los resultados para idiomas CJK (chino, japonés, coreano) donde `pg_trgm` tiene bajo rendimiento. Debe instalarse por separado; no viene incluido.

**`pg_search`** (de [ParadeDB](https://www.paradedb.com/)) reemplaza el stack estándar de `GIN` / `tsvector` con un índice BM25 basado en Tantivy. Esto te da scoring BM25 (a menudo mejor que `ts_rank`), coincidencia difusa dentro de consultas FTS, búsqueda facetada, e indexación dramáticamente más rápida en tablas grandes. Es un camino de actualización directo cuando FTS estándar empieza a mostrar límites de ranking o rendimiento.

```sql
-- pg_search: búsqueda de texto completo BM25 con coincidencia difusa
CREATE INDEX posts_bm25_idx ON posts
  USING bm25 (id, title, body)
  WITH (key_field = 'id', text_fields = '{"title": {}, "body": {}}');

-- Consulta con scoring BM25 + coincidencia difusa (captura "javascipt")
SELECT id, title, paradedb.score(id) AS rank
FROM posts
WHERE posts @@@ paradedb.fuzzy_phrase(field => 'title', value => 'postgres performnce')
ORDER BY rank DESC
LIMIT 10;
```

**`pgvector`** añade almacenamiento de vectores densos y búsqueda de similitud. Es la herramienta correcta cuando los usuarios describen lo que quieren en lugar de nombrarlo — búsqueda semántica, RAG, recomendaciones de contenido relacionado, consultas multilingüe. Cubierto en profundidad en [Búsqueda Semántica de Vectores y Estrategias Híbridas](/semantic-vector-search-landscape).

## Tabla de Decisión

| Qué estás buscando | Recomendado |
|---|---|
| Artículos en prosa, docs, tickets | FTS |
| Nombres de personas/empresas con errores tipográficos | `pg_trgm` |
| Autocompletado, búsqueda de prefijos | `pg_trgm` |
| Códigos cortos, identificadores | `pg_trgm` |
| Mensajes de log por palabras clave | FTS |
| Nombres internacionales | `pg_trgm` + `unaccent` |
| Contenido grande, mejor ranking | `pg_search` (BM25 de ParadeDB) |
| Claves primarias, emails exactos, IDs | índice B-tree |
| Fechas, rangos, listas ordenadas | índice B-tree |
| Permisos, categorías, filtros | cláusula WHERE regular |
| Preguntas, paráfrasis, conceptos | pgvector (ver siguiente artículo) |

Cuando tengas duda: cadenas cortas con variación de spelling → trigrams. Prosa larga para consultas de palabras clave → FTS. Identificadores estructurados → índices regulares. Consultas conceptuales o de lenguaje natural → pgvector.

## Búsqueda Híbrida: Dos Señales, Un Ranking

Cuando una consulta como `"withRetry timeout errors"` llega a una caja de búsqueda, lleva dos tipos de intención: nombres de símbolos exactos que el usuario conoce (`withRetry`) y una descripción conceptual (`timeout errors`). Ninguna primitiva única cubre ambas. Ejecutar FTS y búsqueda vectorial en paralelo — y luego fusionar sus listas rankeadas con Fusión de Ranking Recíproco — sí lo hace.

RRF puntúa cada resultado como `1 / (60 + rank)` en cada lista y suma a través de listas. La constante 60 amortigua la ventaja de los primeros rankings, así que un resultado que queda segundo en ambas listas puede vencer a uno que gana una lista y pierde la otra enteramente. Crucialmente, RRF nunca promedia scores crudos entre métodos — rank FTS y distancia coseno son monedas diferentes y no pueden combinarse aritméticamente.

<figcaption>Una consulta se dispersa a FTS y pgvector en paralelo. Cada una produce su propia lista rankeada. RRF puntúa cada documento por su posición en cada lista y suma los scores — el resultado superfice documentos donde ambas señales están de acuerdo.</figcaption>

```sql
-- Búsqueda híbrida: FTS + pgvector fusionada con RRF
WITH fts AS (
  SELECT id, ts_rank(search_vector, query) AS score,
         ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM docs, to_tsquery('english', 'withRetry & timeout') query
  WHERE search_vector @@ query
  LIMIT 60
),
vec AS (
  SELECT id,
         ROW_NUMBER() OVER (ORDER BY embedding <=> $embedding) AS rank
  FROM docs
  ORDER BY embedding <=> $embedding
  LIMIT 60
)
SELECT COALESCE(fts.id, vec.id) AS id,
       (COALESCE(1.0 / (60 + fts.rank), 0) +
        COALESCE(1.0 / (60 + vec.rank), 0)) AS rrf_score
FROM fts FULL JOIN vec ON fts.id = vec.id
ORDER BY rrf_score DESC
LIMIT 10;
```

El pool de candidatos de 60 documentos por rama (`LIMIT 60`) es un punto de partida común. Expándelo si el recall es bajo; estreéchalo para velocidad.

## Qué Sigue

La búsqueda de texto de Postgres cubre mucho terreno, pero tiene un techo. Cuando los usuarios describen lo que quieren en lugar de nombrarlo — "algo que me ayude a dormir en un vuelo," "artículos sobre depurar confianza como nuevo ingeniero" — tanto la búsqueda léxica como la de trigrams fallan.

Ese es el territorio de los embeddings vectoriales, búsqueda semántica, y arquitecturas híbridas. Cubierto en [Búsqueda Semántica de Vectores y Estrategias Híbridas](/semantic-vector-search-landscape).
