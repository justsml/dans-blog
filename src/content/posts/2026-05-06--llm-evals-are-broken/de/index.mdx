## Was »Vibes-basierte Evaluierung« wirklich kostet

Der Standardansatz: eine Modelländerung ausrollen, die Beschwerdekanäle beobachten, zurückrollen, wenn es laut wird.

Das übersieht fast alles Interessante:

**Du erwischst nur laute Fehler.** Nutzer, die eine selbstbewusst falsche Antwort bekommen und es nicht merken? Still. Nutzer, die eine schlechtere Antwort bekommen und das Feature aufgeben? Still. Support-Tickets und Fehlerraten erfassen nur einen Bruchteil der Qualitätsverschlechterung.

**Du kannst Regressionen nicht von Verbesserungen unterscheiden.** Wenn das neue Modell bei Aufgabe A besser und bei Aufgabe B schlechter ist, sehen Beschwerden über B identisch aus wie generisches »die KI ist schlechter geworden«-Feedback. Du weißt nicht, was zu beheben ist.

**Du benutzt deine Nutzer als Testinfrastruktur.** Dafür haben sie sich nicht angemeldet.

## Das Eval-Spektrum (und wo die meisten Teams falsch abbiegen)

Evaluierungsansätze liegen auf einem Spektrum von »schnell aber fragwürdig« bis »teuer aber valide«.

<figure class="breakout">

![Ein Spektrum-Diagramm, das deterministische Prüfungen, LLM-as-Judge und menschliche Evaluierung nach Geschwindigkeit, Kosten und Validität vergleicht.](../eval-spectrum.svg)

<figcaption>Verwende die billigste Evaluierungsmethode, die den Fehler ehrlich erkennen kann.</figcaption>
</figure>

**LLM-as-Judge** ist der aktuelle Liebling: Ein leistungsstarkes Modell bewertet die Ausgaben eines anderen Modells. Schnell, skalierbar, billig. Das Problem: Es übernimmt die Verzerrungen des bewertenden Modells, kann manipuliert werden und erzeugt eine zirkuläre Abhängigkeit. Wenn du GPT-5 nutzt, um GPT-5s Ausgaben zu bewerten, misst du so etwas wie »wie sehr stimmt GPT-5 mit GPT-5 überein.« Das ist nicht nichts, aber es ist nicht das, was du denkst.

**Menschliche Evaluierung** ist der Goldstandard, den alle zu umgehen versuchen. Menschen für die Bewertung von Ausgaben zu gewinnen ist teuer, langsam, inkonsistent zwischen den Bewertern und nervig zu planen. Aber es ist das Einzige, das validiert, ob dein System für echte Menschen nützlich ist.

**Aufgabenspezifische automatisierte Prüfungen** sind der Bereich, in dem die meisten Teams mehr Zeit investieren sollten. Sie sind nicht glamourös, aber sie sind schnell, deterministisch und an das gebunden, was in deinem System zählt.

## Was wirklich funktioniert

### 1. Definiere Fehler, bevor du auslieferst

Bevor du ein Modell oder einen Prompt änderst, schreibe auf, wie »schlecht« konkret aussieht.

Nicht »die Ausgabe sollte genau sein.« Das ist kein Test. Eher so:

- Strukturierte JSON-Ausgabe muss fehlerfrei parsbar sein
- Alle Zitate in der Antwort müssen wortwörtlich im abgerufenen Kontext vorkommen
- Antworten dürfen keine Produktnamen von Wettbewerbern erwähnen
- SQL-Abfragen müssen syntaktisch gültig sein und nur im Schema vorhandene Tabellen referenzieren
- Die Sentiment-Klassifikation darf auf dem bestehenden Testsatz nicht öfter als 3 % der Fälle von positiv zu negativ kippen

Diese Dinge kannst du programmatisch prüfen. Kein Judge-Modell nötig.

**Eval-Harness: Deterministische Prüfungen**

```typescript
type EvalResult = { passed: boolean; reason?: string };

const evals: Record<string, (output: string, context: EvalContext) => EvalResult> = {
  // JSON muss parsen
  validJson: (output) => {
    try {
      JSON.parse(output);
      return { passed: true };
    } catch (e) {
      return { passed: false, reason: `Invalid JSON: ${e.message}` };
    }
  },

  // Keine halluzinierten Zitate – jede Behauptung muss im Kontext vorkommen
  groundedCitations: (output, { retrievedChunks }) => {
    const claims = extractCitations(output);
    const ungrounded = claims.filter(
      (claim) => !retrievedChunks.some((chunk) => chunk.includes(claim))
    );
    return ungrounded.length === 0
      ? { passed: true }
      : { passed: false, reason: `Ungrounded claims: ${ungrounded.join(', ')}` };
  },

  // Antwortlängen-Plausibilitätsprüfung – Kürzung oder weglaufende Generierung abfangen
  reasonableLength: (output) => {
    const words = output.split(/\s+/).length;
    return words >= 10 && words <= 2000
      ? { passed: true }
      : { passed: false, reason: `Word count ${words} out of bounds` };
  },
};
```

### 2. Baue einen Golden Set aus deinen schlimmsten Tagen

Deine besten Evaluierungsdaten sind die peinlichen Dinge: die Ausgaben, die jemanden ein Ticket aufmachen, einen Screenshot einer Halluzination machen oder das Feature stillschweigend aufgeben ließen.

Jedes Mal, wenn ein Nutzer eine schlechte Ausgabe meldet, eine Halluzination anzeigt oder dir ein Fehler manuell auffällt, füge ihn zu deinem Golden Set hinzu: die Eingabe, der Kontext und das korrekte Verhalten. Halte 50–100 Fälle vor und führe sie bei jeder Modelländerung aus.

Das fühlt sich anfangs manuell an. Nach sechs Monaten hast du eine Testsuite, die kein öffentlicher Benchmark aushebeln kann, weil jeder Fall aus deiner eigenen Fehlergeschichte stammt.

<figure class="breakout">

![Ein Workflow-Diagramm, das zeigt, wie schlimme Produktionsvorfälle zu Golden Cases, dann zu CI-Eval-Läufen und dann zu blockierten Regressionen oder freigegebenen Releases werden.](../golden-set-lifecycle.svg)

<figcaption>Ein Golden Set verwandelt das Peinliche in eine Regressionstestsuite.</figcaption>
</figure>

**Golden-Case-Struktur**

```typescript
interface GoldenCase {
  id: string;
  input: string;
  context: Record<string, unknown>;
  expectedBehavior: {
    mustContain?: string[];
    mustNotContain?: string[];
    structureCheck?: (output: string) => boolean;
    minSimilarityToReference?: number; // Kosinus-Ähnlichkeit zu einer Referenzantwort
  };
  sourceIncident?: string; // Link zurück zum Bug-Report oder Ticket
}
```

### 3. Regressionstests, nicht nur Akzeptanztests

Die meisten Teams führen Evals nur durch, wenn sie einen Modellwechsel in Betracht ziehen. Das ist Akzeptanztest: »Ist dieses neue Ding gut genug?«

Du brauchst auch Regressionstests: »Hat das etwas kaputtgemacht, was vorher funktioniert hat?«

Führe deinen Golden Set bei jeder Prompt-Änderung aus, nicht nur bei Modelländerungen. Ein Prompt, der einwandfrei funktioniert hat, kann sich stillschweigend verschlechtern, wenn du ein neues Tool hinzufügst, eine RAG-Retrieval-Strategie änderst oder deine Context-Vorlage aktualisierst. Du wirst es ohne Baseline nicht merken. Tools wie [Langfuse](https://langfuse.com/) hängen Eval-Scores an Produktionstraces, sodass Regressionen in Dashboards auftauchen, nicht nur in Vorfallberichten.

<details>
<summary>Eval-Harness: Baseline-vs-Kandidaten-Vergleich</summary>

```typescript
async function compareModelVersions(
  goldenCases: GoldenCase[],
  baselinePipeline: Pipeline,
  candidatePipeline: Pipeline
) {
  const results = await Promise.all(
    goldenCases.map(async (tc) => {
      const [baseline, candidate] = await Promise.all([
        baselinePipeline.run(tc.input, tc.context),
        candidatePipeline.run(tc.input, tc.context),
      ]);

      return {
        id: tc.id,
        baselinePassed: runEvals(baseline, tc.expectedBehavior),
        candidatePassed: runEvals(candidate, tc.expectedBehavior),
        regression: /* baseline passed */ && /* candidate failed */,
        improvement: /* baseline failed */ && /* candidate passed */,
      };
    })
  );

  const regressions = results.filter((r) => r.regression);
  const improvements = results.filter((r) => r.improvement);

  console.log(`Regressions: ${regressions.length} / ${goldenCases.length}`);
  console.log(`Improvements: ${improvements.length} / ${goldenCases.length}`);

  if (regressions.length > 0) {
    console.error('Blocking regressions found:');
    regressions.forEach((r) => console.error(` - ${r.id}`));
  }

  return { regressions, improvements };
}
```

</details>

Wenn ein Kandidat bei bekannten Fehlern regrediert, wird das Upgrade-Gespräch wunderbar konkret: welche Fälle sich verbessert haben, welche kaputtgegangen sind und ob der Tausch es wert ist.

### 4. Nutze LLM-as-Judge für genau eine Sache

LLM-as-Judge ist nützlich für offene Ausgaben, bei denen es keine deterministische richtige Antwort gibt: »Ist diese Antwort hilfreich?«, »Bewahrt diese Zusammenfassung die Kernpunkte?«, »Ist diese Erklärung für einen Anfänger geeignet?«

Nutze es dafür. Verwende es nicht für deterministische Antworten. Wenn du es einsetzt, mache den Bewertungsrubrik explizit:

**Eval-Harness: Rubrik-basierter Judge**

```typescript
async function judgeHelpfulness(
  userQuery: string,
  modelResponse: string
): Promise<{ score: number; reasoning: string }> {
  const judgePrompt = `
You are evaluating a customer support response.

User question: ${userQuery}
Response: ${modelResponse}

Rate the response on a scale of 1-5:
5 = Directly answers the question with accurate, actionable information
4 = Answers the question but could be more specific or actionable
3 = Partially addresses the question; key information is missing
2 = Tangentially related but doesn't answer the question
1 = Off-topic, factually wrong, or harmful

Respond with JSON: {"score": <number>, "reasoning": "<one sentence>"}
`;

  const result = await judgeModel.generate(judgePrompt);
  return JSON.parse(result);
}
```

Ein expliziter Rubrik reduziert die Varianz zwischen Bewertern, liefert interpretierbare Ausgaben und macht es einfacher zu prüfen, wenn der Judge falsch liegt. Bibliotheken wie [Autoevals](https://github.com/braintrustdata/autoevals) und [Braintrust](https://www.braintrust.dev/) liefern vorgefertigte Rubriken für gängige Aufgaben – es lohnt sich, sie zu übernehmen, bevor man eigene von Grund auf schreibt.

## Wissenswerte Tools

Du musst das alles nicht von Grund auf neu bauen. Einige Tools haben ernsthafte Fortschritte bei der Eval-Infrastruktur gemacht:

**[Braintrust](https://www.braintrust.dev/)** — Vollständige Eval-Plattform mit Experiment-Tracking, Dataset-Management und Scoring-Funktionen. Ordnet Eval-Läufe nach Prompt, Modell und Deployment, sodass du die Qualität über die Zeit hinweg differenzieren kannst, nicht nur über Releases. Passt gut zu ihrer Open-Source-Bibliothek **[Autoevals](https://github.com/braintrustdata/autoevals)**, die vorgefertigte modellbasierte Scoring-Funktionen für häufige Aufgaben liefert (faktische Genauigkeit, Hilfreichkeit, Toxizität, semantische Ähnlichkeit).

**[Langfuse](https://langfuse.com/)** — Open-Source-LLM-Beobachtbarkeit, die zwischen deiner App und deinen Modellen sitzt. Zeichnet jeden Aufruf auf, hängt Eval-Scores (menschlich oder automatisiert) an einzelne Spans und zeigt Qualitätstrends über den Produktionsverkehr an. Gute Wahl, wenn du Beobachtbarkeit und Evals im selben Werkzeug haben willst, statt eines separaten Eval-Harnesses.

**[Evalite](https://www.evalite.dev/)** — TypeScript-natives Eval-Framework von Matt Pocock. Geringer Overhead: Definiere eine Aufgabe, definiere einen Scorer, führe es in deiner bestehenden Testumgebung aus. Richtet sich an Teams, die Evals wollen, die sich wie Unit-Tests anfühlen, nicht wie eine separate ML-Experimentierplattform.

**[promptfoo](https://www.promptfoo.dev/)** — CLI-erstes Eval-Tool, das sich auf Prompt-Vergleich und Red-Teaming konzentriert. Einfach über YAML konfigurierbar, integriert sich mit den meisten Modellanbietern und hat eingebaute Unterstützung zur Erkennung von Prompt-Injection und anderen adversarialen Eingaben.

**[deepeval](https://docs.confident-ai.com/)** — Python-Eval-Framework mit einer großen Bibliothek eingebauter Metriken (G-Eval, RAG-Faithfulness, Antwortrelevanz, Halluzinationserkennung). Nützlich für RAG-Pipelines, bei denen du spezifische Bewertungen für die Retrieval-Qualität benötigst, nicht nur für die Generierungsqualität.

Das richtige Tool hängt von deinem Stack ab und davon, wo du anfängst. Wichtiger als die Wahl des Frameworks ist die Disziplin, Evals überhaupt durchzuführen – konsistent, bei jeder bedeutenden Änderung.

## Der unangenehme Teil

Die meisten Teams lassen das aus, weil es eine lästige Frage früh stellt: Wie würde »gut« hier eigentlich aussehen?

Das ist für ein neues KI-Feature wirklich schwer. Es ist aber auch nicht optional, wenn dir Zuverlässigkeit wichtig ist. Teams, die vertrauenswürdige KI ausliefern, machen dasselbe, was sie für jeden kritischen Codepfad tun würden: erwartetes Verhalten definieren, testen und diese Tests kontinuierlich ausführen.

Die Benchmarks lügen nicht. Sie beantworten die Fragen von jemand anderem. Hör auf, sie als Produkt-Roadmap zu lesen, und fang an, Tests zu schreiben, die zu deinem System passen.

Deine Nutzer werden es merken, bevor deine Dashboards es anzeigen. Baue zuerst die Testsuite.
