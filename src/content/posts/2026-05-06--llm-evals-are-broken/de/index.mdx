Jedes neue Modell kommt im Smoking von Benchmarks daher.

MMLU: 92,4 %. HumanEval: 87,2 %. LLeMU: 88,7 %. MATH: 73,6 %. AGI: 127 %!

Doch für 99 % der Unternehmen, die mit KI Prozesse und Produkte aufbauen, **bedeutet das alles nichts.**

Was zählt? Wie geht es DEINEN Workloads? Werden sie besser oder schlechter? Der einzige vernünftige Weg, das zu wissen, ist, Evals (Tests für LLMs) zu schreiben, die die spezifischen Aufgaben, Daten und Ausfallmodi deines Systems widerspiegeln.

## Was „Vibes-Based Evaluation" Tatsächlich Kostet

Der Standardansatz: Eine Modelländerung shippen, die Beschwerdekanäle beobachten, zurückrollen, wenn es im Raum laut wird.

Das verpasst fast alles Interessante:

**Du fängst nur laute Ausfälle ab.** Nutzer, die eine selbstbewusst falsche Antwort bekommen und es nicht merken? Stumm. Nutzer, die eine schlechtere Antwort bekommen und die Funktion aufgeben? Stumm. Support-Tickets und Fehlerquoten erfassen nur einen Bruchteil von Qualitätsregressionen.

**Du kannst Regressionen von Verbesserungen nicht unterscheiden.** Wenn das neue Modell bei Aufgabe A besser und bei Aufgabe B schlechter ist, sehen Beschwerden über B identisch aus wie generisches „Die KI wurde schlechter"-Feedback. Du weißt nicht, was du reparieren sollst.

**Du benutzt deine Nutzer als Testinfrastruktur.** Dafür haben sie sich nicht eingeschrieben.

## Das Eval-Spektrum (und wo die meisten Teams es falsch machen)

Eval-Ansätze sitzen auf einem Spektrum von „schnell aber wackelig" bis „teuer aber valide".

<figure class="breakout">

![Ein Spektrumsdiagramm, das deterministische Checks, LLM-as-Judge und menschliche Evaluation nach Geschwindigkeit, Kosten und Validität vergleicht.](../eval-spectrum.svg)

<figcaption>Verwende die günstigste Evaluierungsmethode, die den Fehler ehrlich fangen kann.</figcaption>
</figure>

**LLM-as-Judge** ist der aktuelle Liebling: Ein leistungsstarkes Modell eine andere Modellausgabe bewerten lassen. Schnell, skalierbar, günstig. Das Problem: Es backt die Verzerrungen des Bewertungsmodells ein, kann manipuliert werden und schafft eine zirkuläre Abhängigkeit. Wenn du GPT-5 benutzt, um GPT-5-Ausgaben zu bewerten, misst du etwas wie „wie sehr stimmt GPT-5 mit GPT-5 überein." Das ist nicht nichts, aber es ist nicht das, was du denkst.

**Human Eval** ist der Goldstandard, den jeder zu umgehen versucht. Menschen Ausgaben bewerten zu lassen, ist teuer, langsam, inkonsistent über Evaluierer hinweg und nervig zu planen. Aber es ist das Einzige, was validiert, ob dein System für echte Menschen nützlich ist.

**Aufgabenspezifische automatisierte Checks** sind, wo die meisten Teams mehr Zeit verbringen sollten. Sie sind nicht glamourös, aber sie sind schnell, deterministisch und an das gebunden, was in deinem System zählt.

## Was Tatsächlich Funktioniert

### 1. Definiere Scheitern, Bevor Du Shipst

Bevor du ein Modell oder Prompt änderst, schreibe auf, wie schlecht aussieht. Spezifisch.

Nicht „die Ausgabe sollte genau sein." Das ist kein Test. Eher:

- Strukturierte JSON-Ausgabe muss ohne Fehler parsen
- Alle Zitate in der Antwort müssen wörtlich im abgerufenen Kontext erscheinen
- Antworten dürfen keine Produktnamen von Wettbewerbern erwähnen
- SQL-Abfragen müssen syntaktisch gültig sein und nur Tabellen referenzieren, die im Schema existieren
- Sentiment-Klassifikation darf nicht mehr als 3 % der Zeit auf dem bestehenden Testset von positiv zu negativ umkippen

Das kannst du programmatisch prüfen. Kein Bewertungsmodell erforderlich.

**Eval-Harness: deterministische Checks**

```typescript
type EvalResult = { passed: boolean; reason?: string };

const evals: Record<string, (output: string, context: EvalContext) => EvalResult> = {
  // JSON muss parsen
  validJson: (output) => {
    try {
      JSON.parse(output);
      return { passed: true };
    } catch (e) {
      return { passed: false, reason: `Ungültiges JSON: ${e.message}` };
    }
  },

  // Keine halluzinierten Zitate — jeder Anspruch muss im Kontext erscheinen
  groundedCitations: (output, { retrievedChunks }) => {
    const claims = extractCitations(output);
    const ungrounded = claims.filter(
      (claim) => !retrievedChunks.some((chunk) => chunk.includes(claim))
    );
    return ungrounded.length === 0
      ? { passed: true }
      : { passed: false, reason: `Ungesicherte Ansprüche: ${ungrounded.join(', ')}` };
  },

  // Antwort-Längen-Sanity-Check — Trunkierung oder Laufweg-Generierung abfangen
  reasonableLength: (output) => {
    const words = output.split(/\s+/).length;
    return words >= 10 && words <= 2000
      ? { passed: true }
      : { passed: false, reason: `Wortanzahl ${words} außerhalb der Grenzen` };
  },
};
```

### 2. Baue einen Golden Set Aus Deinen Schlechtesten Tagen

Deine beste Evaluierungsdaten sind die peinlichen Sachen: die Ausgaben, die jemanden dazu brachten, ein Ticket zu filingen, eine Halluzination zu screenshoten, oder die Funktion still aufzugeben.

Jedes Mal, wenn ein Nutzer eine schlechte Ausgabe meldet, eine Halluzination markiert, oder du einen Fehler manuell bemerkst, füge ihn deinem goldenen Set hinzu: die Eingabe, der Kontext, und das korrekte Verhalten. Halte 50-100 Fälle und laufe sie bei jeder Modelländerung durch.

Das fühlt sich zunächst manuell an. Nach sechs Monaten hast du eine Testsuite, die kein öffentlicher Benchmark manipulieren kann, weil jeder Fall aus deiner eigenen Fehlerhistorie kam.

<figure class="breakout">

![Ein Workflow-Diagramm, das zeigt, wie schlechte Produktionsvorfälle zu Golden Cases werden, dann CI-Eval-Läufe, dann blockierte Regressionen oder freigegebene Releases.](../golden-set-lifecycle.svg)

<figcaption>Ein goldenes Set verwandelt die peinlichen Sachen in eine Regressionssuite.</figcaption>
</figure>

**Golden Case Shape**

```typescript
interface GoldenCase {
  id: string;
  input: string;
  context: Record<string, unknown>;
  expectedBehavior: {
    mustContain?: string[];
    mustNotContain?: string[];
    structureCheck?: (output: string) => boolean;
    minSimilarityToReference?: number; // cosine similarity to a reference answer
  };
  sourceIncident?: string; // link back to the bug report or ticket
}
```

### 3. Regression Testing, Nicht Nur Acceptance Testing

Die meisten Teams laufen Evals nur beim Überlegen einer Modelländerung. Das ist Acceptance Testing: „Ist dieses neue Ding gut genug?"

Du brauchst auch Regression Testing: „Hat das etwas kaputt gemacht, das früher funktioniert hat?"

Laufe dein goldenes Set bei jeder Prompt-Änderung durch, nicht nur Modelländerungen. Ein Prompt, der gut funktionierte, kann still degradieren, wenn du ein neues Tool hinzufügst, eine RAG-Retrieval-Strategie änderst, oder deine Context-Templat aktualisierst. Das wirst du nicht wissen ohne eine Baseline. Tools wie [Langfuse](https://langfuse.com/) hängen Eval-Scores an Production Traces, damit Regression in Dashboards erscheint, nicht nur in Incident Reports.

<details>
<summary>Eval-Harness: Baseline vs Candidate Comparison</summary>

```typescript
async function compareModelVersions(
  goldenCases: GoldenCase[],
  baselinePipeline: Pipeline,
  candidatePipeline: Pipeline
) {
  const results = await Promise.all(
    goldenCases.map(async (tc) => {
      const [baseline, candidate] = await Promise.all([
        baselinePipeline.run(tc.input, tc.context),
        candidatePipeline.run(tc.input, tc.context),
      ]);

      return {
        id: tc.id,
        baselinePassed: runEvals(baseline, tc.expectedBehavior),
        candidatePassed: runEvals(candidate, tc.expectedBehavior),
        regression: /* baseline passed */ && /* candidate failed */,
        improvement: /* baseline failed */ && /* candidate passed */,
      };
    })
  );

  const regressions = results.filter((r) => r.regression);
  const improvements = results.filter((r) => r.improvement);

  console.log(`Regressionen: ${regressions.length} / ${goldenCases.length}`);
  console.log(`Verbesserungen: ${improvements.length} / ${goldenCases.length}`);

  if (regressions.length > 0) {
    console.error('Blockierende Regressionen gefunden:');
    regressions.forEach((r) => console.error(` - ${r.id}`));
  }

  return { regressions, improvements };
}
```

</details>

Wenn ein Candidate bei bekannten Fehlern regressiert, wird das Upgrade-Gespräch wunderbar spezifisch: welche Fälle sich verbesserten, welche Fälle kaputtgingen, und ob der Trade es wert ist.

### 4. Benutze LLM-as-Judge Für Genau Ein Ding

LLM-as-Judge ist nützlich für offene Ausgaben, wo es keine deterministische richtige Antwort gibt: „hilft diese Antwort?", „erhält diese Zusammenfassung die Schlüsselpunkte?", „ist diese Erklärung richtig für einen Anfänger?"

Benutze es dort. Benutze es nicht für deterministische Antworten. Wenn du es benutzt, mache das Bewertungsrubrik explizit:

**Eval-Harness: Rubrik-basierter Judge**

```typescript
async function judgeHelpfulness(
  userQuery: string,
  modelResponse: string
): Promise<{ score: number; reasoning: string }> {
  const judgePrompt = `
You are evaluating a customer support response.

User question: ${userQuery}
Response: ${modelResponse}

Rate the response on a scale of 1-5:
5 = Directly answers the question with accurate, actionable information
4 = Answers the question but could be more specific or actionable
3 = Partially addresses the question; key information is missing
2 = Tangentially related but doesn't answer the question
1 = Off-topic, factually wrong, or harmful

Respond with JSON: {"score": <number>, "reasoning": "<one sentence>"}
`;

  const result = await judgeModel.generate(judgePrompt);
  return JSON.parse(result);
}
```

Eine explizite Rubrik reduziert Evaluierer-Varianz, gibt dir interpretierbare Ausgabe, und macht es einfacher zu auditieren, wenn der Judge falsch liegt. Bibliotheken wie [Autoevals](https://github.com/braintrustdata/autoevals) und [Braintrust](https://www.braintrust.dev/) shippen vorgebaute Rubriken für häufige Aufgaben — stehlen wert vor dem Schreiben deiner eigenen von vornherein.

## Tools Das Es Wert Ist Zu Kennen

Du musst das nicht alles von Grund auf neu bauen. Mehrere Tools haben ernsthafte Fortschritte auf dem Eval-Infrastruktur-Problem gemacht:

**[Braintrust](https://www.braintrust.dev/)** — Vollständige Eval-Plattform mit Experiment-Tracking, Dataset-Management, und Scoring-Funktionen. Organisiert Eval-Läufe nach Prompt, Modell, und Deployment, damit du Qualität über Zeit diffen kannst, nicht nur über Releases hinweg. Passt gut mit ihrer Open-Source-**[Autoevals](https://github.com/braintrustdata/autoevals)** Bibliothek, die vorgebaute model-graded Scoring-Funktionen für häufige Aufgaben (faktische Genauigkeit, Nützlichkeit, Toxizität, semantische Ähnlichkeit) shippt.

**[Langfuse](https://langfuse.com/)** — Open-Source LLM Observability, das zwischen deiner App und deinen Modellen sitzt. Trace jeden Aufruf, hänge Eval-Scores (menschlich oder automatisiert) an einzelne Spans, und bringe Qualitätstrends über Production Traffic an die Oberfläche. Gute Wahl, wenn du Observability und Evals im gleichen Tool willst statt eines separaten Eval-Harness.

**[Evalite](https://www.evalite.dev/)** — TypeScript-natives Eval-Framework von Matt Pocock. Wenig Zeremonie: definiere eine Aufgabe, definiere einen Scorer, laufe es in deinem bestehenden Test-Setup. Zielt auf Teams, die Evals wollen, die sich wie Unit-Tests anfühlen statt einer separaten ML-Experiment-Plattform.

**[promptfoo](https://www.promptfoo.dev/)** — CLI-first Eval-Läufer, fokussiert auf Prompt-Vergleich und Red-Teaming. Einfach via YAML konfigurieren, integriert mit den meisten Model-Providern, und hat eingebaute Unterstützung für das Erkennen von Prompt-Injection und anderen adversarial Inputs.

**[deepeval](https://docs.confident-ai.com/)** — Python Eval-Framework mit einer großen Bibliothek eingebaute Metriken (G-Eval, RAG Faithfulness, Answer Relevancy, Hallucination Detection). Nützlich für RAG-Pipelines, wo du spezifisches Grading für Retrieval-Qualität willst, nicht nur Generierungs-Qualität.

Das richtige Tool hängt von deinem Stack und davon ab, wo du startest. Was mehr zählt als die Wahl des Frameworks ist die Disziplin, Evals überhaupt zu laufen — konsistent, bei jeder signifikanten Änderung.

## Der Unangenehme Teil

Die meisten Teams überspringen das, weil es eine irritierende Frage früh stellt: was würde „gut" hier aussehen?

Das ist echt schwer für ein neues KI-Feature. Es ist auch nicht optional, wenn du an Zuverlässigkeit interessiert bist. Teams, die vertrauenswürdige KI shippen, machen das Gleiche, was sie für jeden kritischen Code-Pfad machen würden: erwartetes Verhalten definieren, es testen, und diese Tests kontinuierlich laufen lassen.

Die Benchmarks lügen nicht. Sie beantworten eine andere Frage. Hör auf, sie als Produkt-Roadmaps zu lesen, und fang an, Tests zu schreiben, die zu deinem System passen.

Deine Nutzer werden es merken, bevor deine Dashboards es tun. Baue die Testsuite zuerst.
