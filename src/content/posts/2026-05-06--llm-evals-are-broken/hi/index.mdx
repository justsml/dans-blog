हर नया मॉडल बेहतर बेंचमार्क स्कोर के दावे के साथ आता है।

MMLU: 92.4%। HumanEval: 87.2%। LLeMU: 88.7%। MATH: 73.6%। AGI: 127%!

फिर भी, 99% व्यवसायों के लिए जो AI के साथ प्रक्रिया और उत्पाद बना रहे हैं, **इनमें से कोई मायने नहीं रखता।**

क्या मायने रखता है? आपके वर्कलोड कैसा प्रदर्शन कर रहे हैं? बेहतर हो रहे हैं या बदतर? यह जानने का एकमात्र समझदार तरीका है ईवल्स (LLMs के लिए टेस्ट) लिखना जो आपके सिस्टम के विशिष्ट कार्यों, डेटा और फेलियर मोड को दर्शाते हों।

## "वाइब्स-आधारित मूल्यांकन" वास्तव में क्या लागत आता है

मानक दृष्टिकोण: एक मॉडल परिवर्तन शिप करें, शिकायत चैनलों को देखें, अगर कमरा जोर से चिल्लाए तो वापस रोल करें।

यह लगभग सब कुछ मिस कर देता है:

**आप केवल जोरदार फेलियर पकड़ते हैं।** उपयोगकर्ता जो एक आत्मविश्वास से गलत जवाब प्राप्त करते हैं और उन्हें एहसास नहीं होता? शांत। उपयोगकर्ता जो एक बदतर जवाब प्राप्त करते हैं और फीचर छोड़ देते हैं? शांत। सपोर्ट टिकट और एरर रेट्स क्वालिटी रिग्रेशन का केवल एक अंश पकड़ते हैं।

**आप रिग्रेशन और सुधार में अंतर नहीं कर सकते।** अगर नया मॉडल टास्क A में बेहतर है और टास्क B में बदतर, तो B के बारे में शिकायतें सामान्य "AI बदतर हो गया" फीडबैक जैसी ही दिखती हैं। आपको पता नहीं चलता कि क्या ठीक करना है।

**आप अपने उपयोगकर्ताओं को टेस्ट इन्फ्रास्ट्रक्चर के रूप में उपयोग कर रहे हैं।** उन्होंने इसके लिए साइन अप नहीं किया था।

## ईवल स्पेक्ट्रम (और अधिकांश टीमें इसे गलत कहाँ करती हैं)

मूल्यांकन दृष्टिकोण "तेज लेकिन कमजोर" से "महंगे लेकिन मान्य" तक एक स्पेक्ट्रम पर बैठते हैं।

<figure class="breakout">

![डिटरमिनिस्टिक चेक्स, LLM-एज़-जज और मानव मूल्यांकन की तुलना करने वाला एक स्पेक्ट्रम डायग्राम गति, लागत और वैधता के आधार पर।](./eval-spectrum.svg)

<figcaption>वह सबसे सस्ता मूल्यांकन विधि उपयोग करें जो फेलियर को ईमानदारी से पकड़ सके।</figcaption>
</figure>

**LLM-एज़-जज** वर्तमान पसंदीदा है: एक शक्तिशाली मॉडल से दूसरे मॉडल के आउटपुट को ग्रेड करने के लिए कहें। तेज, स्केलेबल, सस्ता। समस्या: यह ग्रेडर मॉडल के पूर्वाग्रहों को बनाता है, इसे गेम किया जा सकता है, और एक चक्रीय निर्भरता बनाता है। यदि आप GPT-5 के आउटपुट को ग्रेड करने के लिए GPT-5 का उपयोग करते हैं, तो आप कुछ माप रहे हैं जैसे "GPT-5, GPT-5 से कितना सहमत है।" यह कुछ नहीं है, लेकिन यह वह नहीं है जो आप सोचते हैं।

**मानव मूल्यांकन** स्वर्ण मानक है जिसे सभी छोड़ने की कोशिश करते हैं। आउटपुट का मूल्यांकन करने के लिए मानव प्राप्त करना महंगा, धीमा, मूल्यांकनकर्ताओं के बीच असंगत है, और शेड्यूल करने में कष्टप्रद है। लेकिन यह एकमात्र चीज है जो यह सत्यापित करती है कि आपका सिस्टम वास्तविक मानवों के लिए उपयोगी है या नहीं।

**टास्क-विशिष्ट स्वचालित चेक्स** वह जगह है जहां अधिकांश टीमों को अधिक समय बिताना चाहिए। वे ग्लैमरस नहीं हैं, लेकिन वे तेज, डिटरमिनिस्टिक हैं, और आपके सिस्टम में जो मायने रखता है उससे जुड़े हैं।

## वास्तव में क्या काम करता है

### 1. शिप करने से पहले फेलियर को परिभाषित करें

मॉडल या प्रॉम्प्ट बदलने से पहले, लिखें कि बुरा कैसा दिखता है। विशिष्ट रूप से।

"आउटपुट सटीक होना चाहिए" नहीं। यह एक टेस्ट नहीं है। इसके बजाय:

- स्ट्रक्चर्ड JSON आउटपुट बिना एरर के पार्स होना चाहिए
- रिस्पॉन्स में सभी संदर्भ पुनर्प्राप्त संदर्भ में शाब्दिक रूप से दिखाई देने चाहिए
- रिस्पॉन्स में प्रतिस्पर्धी उत्पाद नाम का उल्लेख नहीं होना चाहिए
- SQL क्वेरीज सिंटैक्टिक रूप से मान्य होनी चाहिए और केवल स्कीमा में मौजूद टेबल्स को संदर्भित करनी चाहिए
- सेंटीमेंट क्लासिफिकेशन मौजूदा टेस्ट सेट पर सकारात्मक से नकारात्मक में 3% से अधिक बार फ्लिप नहीं होना चाहिए

आप इन्हें प्रोग्रामेटिक रूप से चेक कर सकते हैं। किसी जज मॉडल की आवश्यकता नहीं।

**ईवल हार्नेस: डिटरमिनिस्टिक चेक्स**

```typescript
type EvalResult = { passed: boolean; reason?: string };

const evals: Record<string, (output: string, context: EvalContext) => EvalResult> = {
  // JSON must parse
  validJson: (output) => {
    try {
      JSON.parse(output);
      return { passed: true };
    } catch (e) {
      return { passed: false, reason: `Invalid JSON: ${e.message}` };
    }
  },

  // No hallucinated citations — every claim must appear in context
  groundedCitations: (output, { retrievedChunks }) => {
    const claims = extractCitations(output);
    const ungrounded = claims.filter(
      (claim) => !retrievedChunks.some((chunk) => chunk.includes(claim))
    );
    return ungrounded.length === 0
      ? { passed: true }
      : { passed: false, reason: `Ungrounded claims: ${ungrounded.join(', ')}` };
  },

  // Response length sanity check — catch truncation or runaway generation
  reasonableLength: (output) => {
    const words = output.split(/\s+/).length;
    return words >= 10 && words <= 2000
      ? { passed: true }
      : { passed: false, reason: `Word count ${words} out of bounds` };
  },
};
```

### 2. अपने सबसे बुरे दिनों से एक गोल्डन सेट बनाएं

आपका सर्वोत्तम मूल्यांकन डेटा वह शर्मनाक सामग्री है: वे आउटपुट जिसने किसी को टिकट फाइल करने, हैलुसिनेशन का स्क्रीनशॉट लेने, या फीचर को चुपचाप बंद करने के लिए मजबूर किया।

हर बार जब कोई उपयोगकर्ता एक खराब आउटपुट रिपोर्ट करता है, हैलुसिनेशन को फ्लैग करता है, या आप मैन्युअल रूप से एक फेलियर नोटिस करते हैं, तो इसे अपने गोल्डन सेट में जोड़ें: इनपुट, संदर्भ, और सही व्यवहार। 50-100 केस रखें और हर मॉडल परिवर्तन पर उन्हें चलाएं।

यह पहले मैन्युअल लगता है। छह महीने बाद, आपके पास एक टेस्ट सूट होता है जिसे कोई भी सार्वजनिक बेंचमार्क गेम नहीं कर सकता, क्योंकि हर केस आपकी अपनी फेलियर हिस्ट्री से आया था।

<figure class="breakout">

![एक वर्कफ्लो डायग्राम जो दिखाता है कि कैसे खराब प्रोडक्शन घटनाएं गोल्डन केस बनती हैं, फिर CI ईवल रन, फिर ब्लॉक्ड रिग्रेशन या अप्रूव्ड रिलीज़।](./golden-set-lifecycle.svg)

<figcaption>एक गोल्डन सेट शर्मनाक सामग्री को एक रिग्रेशन सूट में बदल देता है।</figcaption>
</figure>

**गोल्डन केस शेप**

```typescript
interface GoldenCase {
  id: string;
  input: string;
  context: Record<string, unknown>;
  expectedBehavior: {
    mustContain?: string[];
    mustNotContain?: string[];
    structureCheck?: (output: string) => boolean;
    minSimilarityToReference?: number; // cosine similarity to a reference answer
  };
  sourceIncident?: string; // link back to the bug report or ticket
}
```

### 3. रिग्रेशन टेस्टिंग, न केवल एक्सेप्टेंस टेस्टिंग

अधिकांश टीमें केवल तब ईवल्स चलाती हैं जब वे एक मॉडल परिवर्तन पर विचार कर रही होती हैं। यह एक्सेप्टेंस टेस्टिंग है: "क्या यह नई चीज काफी अच्छी है?"

आपको रिग्रेशन टेस्टिंग की भी आवश्यकता है: "क्या इसने कुछ ऐसा तोड़ दिया जो पहले काम करता था?"

अपने गोल्डन सेट को हर प्रॉम्प्ट परिवर्तन पर चलाएं, न कि केवल मॉडल परिवर्तन पर। एक प्रॉम्प्ट जो ठीक काम कर रहा था, वह चुपचाप डिग्रेड हो सकता है जब आप एक नया टूल जोड़ते हैं, RAG रीट्रीवल स्ट्रैटेजी बदलते हैं, या अपने कॉन्टेक्स्ट टेम्पलेट को अपडेट करते हैं। बेसलाइन के बिना आपको पता नहीं चलेगा। [Langfuse](https://langfuse.com/) जैसे टूल्स प्रोडक्शन ट्रेस से ईवल स्कोर जोड़ते हैं ताकि रिग्रेशन डैशबोर्ड में दिखाई दे, न कि केवल इंसिडेंट रिपोर्ट में।

<details>
<summary>ईवल हार्नेस: बेसलाइन बनाम कैंडिडेट तुलना</summary>

```typescript
async function compareModelVersions(
  goldenCases: GoldenCase[],
  baselinePipeline: Pipeline,
  candidatePipeline: Pipeline
) {
  const results = await Promise.all(
    goldenCases.map(async (tc) => {
      const [baseline, candidate] = await Promise.all([
        baselinePipeline.run(tc.input, tc.context),
        candidatePipeline.run(tc.input, tc.context),
      ]);

      return {
        id: tc.id,
        baselinePassed: runEvals(baseline, tc.expectedBehavior),
        candidatePassed: runEvals(candidate, tc.expectedBehavior),
        regression: /* baseline passed */ && /* candidate failed */,
        improvement: /* baseline failed */ && /* candidate passed */,
      };
    })
  );

  const regressions = results.filter((r) => r.regression);
  const improvements = results.filter((r) => r.improvement);

  console.log(`Regressions: ${regressions.length} / ${goldenCases.length}`);
  console.log(`Improvements: ${improvements.length} / ${goldenCases.length}`);

  if (regressions.length > 0) {
    console.error('Blocking regressions found:');
    regressions.forEach((r) => console.error(` - ${r.id}`));
  }

  return { regressions, improvements };
}
```

</details>

यदि एक कैंडिडेट ज्ञात फेलियर पर रिग्रेस करता है, तो अपग्रेड वार्तालाप अद्भुत रूप से विशिष्ट हो जाता है: कौन से केस सुधरे, कौन से केस टूटे, और क्या ट्रेड इसके लायक है।

### 4. LLM-एज़-जज का उपयोग ठीक एक चीज के लिए करें

LLM-एज़-जज उन ओपन-एंडेड आउटपुट के लिए उपयोगी है जहां कोई डिटरमिनिस्टिक सही उत्तर नहीं है: "क्या यह रिस्पॉन्स सहायक है?", "क्या यह सारांश मुख्य बिंदुओं को संरक्षित करता है?", "क्या यह स्पष्टीकरण एक शुरुआती के लिए सही है?"

वहां इसका उपयोग करें। डिटरमिनिस्टिक उत्तरों के लिए इसका उपयोग न करें। जब आप इसका उपयोग करते हैं, तो ग्रेडिंग रूब्रिक को स्पष्ट बनाएं:

**ईवल हार्नेस: रूब्रिक-आधारित जज**

```typescript
async function judgeHelpfulness(
  userQuery: string,
  modelResponse: string
): Promise<{ score: number; reasoning: string }> {
  const judgePrompt = `
You are evaluating a customer support response.

User question: ${userQuery}
Response: ${modelResponse}

Rate the response on a scale of 1-5:
5 = Directly answers the question with accurate, actionable information
4 = Answers the question but could be more specific or actionable
3 = Partially addresses the question; key information is missing
2 = Tangentially related but doesn't answer the question
1 = Off-topic, factually wrong, or harmful

Respond with JSON: {"score": <number>, "reasoning": "<one sentence>"}
`;

  const result = await judgeModel.generate(judgePrompt);
  return JSON.parse(result);
}
```

एक स्पष्ट रूब्रिक मूल्यांकनकर्ता भिन्नता को कम करता है, आपको व्याख्या योग्य आउटपुट देता है, और यह ऑडिट करना आसान बनाता है जब जज गलत है। [Autoevals](https://github.com/braintrustdata/autoevals) और [Braintrust](https://www.braintrust.dev/) जैसी लाइब्रेरी सामान्य कार्यों के लिए प्रीबिल्ट रूब्रिक भेजती हैं — अपनी खुद की स्क्रैच से लिखने से पहले चुराने योग्य।

## जानने लायक टूल्स

आपको यह सब स्क्रैच से बनाने की जरूरत नहीं है। कई टूल्स ने ईवल इन्फ्रास्ट्रक्चर समस्या पर गंभीर प्रगति की है:

**[Braintrust](https://www.braintrust.dev/)** — प्रयोग ट्रैकिंग, डेटासेट प्रबंधन और स्कोरिंग फंक्शन्स के साथ पूर्ण ईवल प्लेटफॉर्म। ईवल रन को प्रॉम्प्ट, मॉडल और डिप्लॉयमेंट द्वारा व्यवस्थित करता है ताकि आप केवल रिलीज के across ही नहीं, बल्कि समय के साथ क्वालिटी को डिफ कर सकें। उनके ओपन-सोर्स **[Autoevals](https://github.com/braintrustdata/autoevals)** लाइब्रेरी के साथ अच्छी तरह से जोड़ी बनाता है, जो सामान्य कार्यों के लिए प्रीबिल्ट मॉडल-ग्रेडेड स्कोरिंग फंक्शन्स भेजता है (तथ्यात्मक सटीकता, सहायकता, विषाक्तता, सिमेंटिक समानता)।

**[Langfuse](https://langfuse.com/)** — ओपन-सोर्स LLM ऑब्जर्वेबिलिटी जो आपके ऐप और आपके मॉडल्स के बीच बैठता है। हर कॉल को ट्रेस करता है, व्यक्तिगत स्पैन से ईवल स्कोर (मानव या स्वचालित) जोड़ता है, और प्रोडक्शन ट्रैफिक पर क्वालिटी ट्रेंड को सामने लाता है। अच्छा विकल्प यदि आप अलग ईवल हार्नेस के बजाय एक ही टूल में ऑब्जर्वेबिलिटी और ईवल्स चाहते हैं।

**[Evalite](https://www.evalite.dev/)** — Matt Pocock द्वारा टाइपस्क्रिप्ट-नेटिव ईवल फ्रेमवर्क। कम औपचारिकता: एक टास्क परिभाषित करें, एक स्कोरर परिभाषित करें, इसे अपनी मौजूदा टेस्ट सेटअप में चलाएं। उन टीमों को लक्षित करता है जो ईवल्स चाहते हैं जो यूनिट टेस्ट जैसे महसूस हों न कि एक अलग ML प्रयोग प्लेटफॉर्म।

**[promptfoo](https://www.promptfoo.dev/)** — CLI-फर्स्ट ईवल रनर जो प्रॉम्प्ट तुलना और रेड-टीमिंग पर केंद्रित है। YAML के माध्यम से कॉन्फ़िगर करने में आसान, अधिकांश मॉडल प्रदाताओं के साथ एकीकृत होता है, और प्रॉम्प्ट इंजेक्शन और अन्य एडवर्सरियल इनपुट का पता लगाने के लिए बिल्ट-इन सपोर्ट है।

**[deepeval](https://docs.confident-ai.com/)** — पायथन ईवल फ्रेमवर्क जिसमें बिल्ट-इन मेट्रिक्स की एक बड़ी लाइब्रेरी है (G-Eval, RAG फेथफुलनेस, उत्तर प्रासंगिकता, हैलुसिनेशन डिटेक्शन)। RAG पाइपलाइन के लिए उपयोगी जहां आप केवल जनरेशन क्वालिटी नहीं, बल्कि विशिष्ट रीट्रीवल क्वालिटी ग्रेडिंग चाहते हैं।

सही टूल आपके स्टैक और आप कहां से शुरू कर रहे हैं उस पर निर्भर करता है। फ्रेमवर्क की पसंद से अधिक जो मायने रखता है वह है ईवल्स चलाने का अनुशासन — लगातार, हर महत्वपूर्ण परिवर्तन पर।

## असहज हिस्सा

अधिकांश टीमें इसे छोड़ देती हैं क्योंकि यह एक उत्तेजक सवाल जल्दी पूछता है: यहां "अच्छा" कैसा दिखेगा?

एक नए AI फीचर के लिए यह वास्तव में कठिन है। यह गैर-वैकल्पिक भी है यदि आप विश्वसनीयता की परवाह करते हैं। विश्वसनीय AI शिप करने वाली टीमें वही कर रही हैं जो वे किसी भी क्रिटिकल कोड पाथ के लिए करेंगी: अपेक्षित व्यवहार को परिभाषित करें, इसका परीक्षण करें, और उन परीक्षणों को लगातार चलाएं।

बेंचमार्क झूठ नहीं बोल रहे। वे किसी और का सवाल जवाब दे रहे हैं। उन्हें उत्पाद रोडमैप के रूप में पढ़ना बंद करें और अपने सिस्टम से मेल खाने वाले टेस्ट लिखना शुरू करें।

आपके उपयोगकर्ता आपके डैशबोर्ड से पहले नोटिस करेंगे। पहले टेस्ट सूट बनाएं।
