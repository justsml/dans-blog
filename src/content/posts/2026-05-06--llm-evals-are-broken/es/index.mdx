Cada nuevo modelo llega vestido de gala con sus benchmarks.

MMLU: 92,4%. HumanEval: 87,2%. LLeMU: 88,7%. MATH: 73,6%. ¡AGI: 127%!

Sin embargo, para el 99% de las empresas que construyen procesos y productos con IA, **nada de esto importa.**

¿Qué es lo que importa? ¿Cómo les va a TUS cargas de trabajo? ¿Están mejorando o empeorando? La única forma sensata de saberlo es escribir Evals (pruebas para LLMs) que reflejen las tareas específicas, los datos y los modos de fallo de tu sistema.

<blockquote class="breakout">
  <p>Los benchmarks no mienten. Están respondiendo a la pregunta de alguien más.</p>
</blockquote>

## Lo que cuesta realmente la "evaluación basada en vibras"

El enfoque estándar: lanzas un cambio de modelo, observas los canales de quejas, reviertes si el ambiente se caldea.

Eso omite casi todo lo interesante:

**Solo detectas los fallos ruidosos.** ¿Usuarios que reciben una respuesta incorrecta con total confianza y no se dan cuenta? Silenciosos. ¿Usuarios que reciben una respuesta peor y abandonan la funcionalidad? Silenciosos. Los tickets de soporte y las tasas de error capturan solo una fracción de la regresión de calidad.

**No puedes distinguir las regresiones de las mejoras.** Si el nuevo modelo es mejor en la tarea A y peor en la tarea B, las quejas sobre B parecen idénticas al feedback genérico de "la IA empeoró". No sabes qué arreglar.

**Estás usando a tus usuarios como infraestructura de pruebas.** Ellos no se inscribieron para eso.

---

## El espectro de las Evals (y dónde fallan la mayoría de los equipos)

Los enfoques de evaluación se sitúan en un espectro que va desde lo "rápido pero frágil" hasta lo "caro pero válido".

<figure class="breakout">

![Un diagrama de espectro que compara comprobaciones deterministas, LLM-como-juez y evaluación humana por velocidad, coste y validez.](../eval-spectrum.svg)

<figcaption>Usa el método de evaluación más barato que pueda detectar honestamente el fallo.</figcaption>
</figure>

**LLM-como-juez** (LLM-as-judge) es el favorito actual: pedirle a un modelo potente que califique los resultados de otro modelo. Rápido, escalable, barato. El problema: arrastra los sesgos del modelo evaluador, puede ser manipulado y crea una dependencia circular. Si usas GPT-5 para calificar los resultados de GPT-5, estás midiendo algo así como "cuánto está de acuerdo GPT-5 con GPT-5". No es nada, pero no es lo que crees.

**Evaluación humana** es el estándar de oro que todos intentan saltarse. Conseguir que humanos evalúen los resultados es caro, lento, inconsistente entre evaluadores y molesto de programar. Pero es lo único que valida si tu sistema es útil para humanos reales.

**Comprobaciones automatizadas específicas para la tarea** es donde la mayoría de los equipos deberían pasar más tiempo. No son glamurosas, pero son rápidas, deterministas y están ligadas a lo que importa en tu sistema.

## Lo que realmente funciona

### 1. Define el fallo antes de lanzar

Antes de cambiar un modelo o un prompt, escribe qué aspecto tiene un mal resultado. Específicamente.

No digas "el resultado debe ser preciso". Eso no es una prueba. Mejor algo como:

- El resultado en formato JSON estructurado debe parsearse sin errores.
- Todas las citas en la respuesta deben aparecer textualmente en el contexto recuperado.
- Las respuestas no deben mencionar nombres de productos de la competencia.
- Las consultas SQL deben ser sintácticamente válidas y referenciar solo tablas que existan en el esquema.
- La clasificación de sentimiento no debe cambiar de positivo a negativo más del 3% de las veces en el conjunto de prueba existente.

Puedes comprobar esto programáticamente. No se requiere ningún modelo como juez.

**Arnés de evals: comprobaciones deterministas**

```typescript
type EvalResult = { passed: boolean; reason?: string };

const evals: Record<string, (output: string, context: EvalContext) => EvalResult> = {
  // El JSON debe parsear
  validJson: (output) => {
    try {
      JSON.parse(output);
      return { passed: true };
    } catch (e) {
      return { passed: false, reason: `JSON inválido: ${e.message}` };
    }
  },

  // Sin citas alucinadas — cada afirmación debe aparecer en el contexto
  groundedCitations: (output, { retrievedChunks }) => {
    const claims = extractCitations(output);
    const ungrounded = claims.filter(
      (claim) => !retrievedChunks.some((chunk) => chunk.includes(claim))
    );
    return ungrounded.length === 0
      ? { passed: true }
      : { passed: false, reason: `Afirmaciones no ancladas: ${ungrounded.join(', ')}` };
  },

  // Comprobación de cordura de la longitud de la respuesta — detectar truncamiento o generación descontrolada
  reasonableLength: (output) => {
    const words = output.split(/\s+/).length;
    return words >= 10 && words <= 2000
      ? { passed: true }
      : { passed: false, reason: `Recuento de palabras ${words} fuera de los límites` };
  },
};
```

### 2. Construye un "Golden Set" a partir de tus peores días

Tus mejores datos de evaluación son las cosas embarazosas: los resultados que hicieron que alguien abriera un ticket, capturara una alucinación o dejara de usar la funcionalidad silenciosamente.

Cada vez que un usuario informe de un mal resultado, marque una alucinación o tú mismo notes un fallo manualmente, añádelo a tu "golden set": la entrada (input), el contexto y el comportamiento correcto. Mantén 50-100 casos y ejecútalos en cada cambio de modelo.

Esto parece manual al principio. Después de seis meses, tendrás una suite de pruebas que ningún benchmark público podrá manipular, porque cada caso provino de tu propio historial de fallos.

<figure class="breakout">

![Un diagrama de flujo que muestra cómo los incidentes de producción se convierten en casos "golden", luego en ejecuciones de CI eval, y finalmente en regresiones bloqueadas o lanzamientos aprobados.](../golden-set-lifecycle.svg)

<figcaption>Un golden set convierte las cosas embarazosas en una suite de regresión.</figcaption>
</figure>

**Estructura del Golden case**

```typescript
interface GoldenCase {
  id: string;
  input: string;
  context: Record<string, unknown>;
  expectedBehavior: {
    mustContain?: string[];
    mustNotContain?: string[];
    structureCheck?: (output: string) => boolean;
    minSimilarityToReference?: number; // similitud de coseno con una respuesta de referencia
  };
  sourceIncident?: string; // enlace al informe de error o ticket
}
```

### 3. Pruebas de regresión, no solo de aceptación

La mayoría de los equipos ejecutan evals solo cuando consideran un cambio de modelo. Eso es una prueba de aceptación: "¿es esto nuevo lo suficientemente bueno?".

También necesitas pruebas de regresión: "¿rompió esto algo que solía funcionar?".

Ejecuta tu golden set en cada cambio de prompt, no solo en los cambios de modelo. Un prompt que funcionaba bien puede degradarse silenciosamente cuando añades una nueva herramienta, cambias una estrategia de recuperación RAG o actualizas tu plantilla de contexto. No lo sabrás sin una línea de base. Herramientas como [Langfuse](https://langfuse.com/) adjuntan puntuaciones de eval a las trazas de producción para que la regresión aparezca en los paneles (dashboards), no solo en los informes de incidentes.

<details>
<summary>Arnés de evals: comparación entre línea de base y candidato</summary>

```typescript
async function compareModelVersions(
  goldenCases: GoldenCase[],
  baselinePipeline: Pipeline,
  candidatePipeline: Pipeline
) {
  const results = await Promise.all(
    goldenCases.map(async (tc) => {
      const [baseline, candidate] = await Promise.all([
        baselinePipeline.run(tc.input, tc.context),
        candidatePipeline.run(tc.input, tc.context),
      ]);

      return {
        id: tc.id,
        baselinePassed: runEvals(baseline, tc.expectedBehavior),
        candidatePassed: runEvals(candidate, tc.expectedBehavior),
        regression: /* el baseline pasó */ && /* el candidato falló */,
        improvement: /* el baseline falló */ && /* el candidato pasó */,
      };
    })
  );

  const regressions = results.filter((r) => r.regression);
  const improvements = results.filter((r) => r.improvement);

  console.log(`Regresiones: ${regressions.length} / ${goldenCases.length}`);
  console.log(`Mejoras: ${improvements.length} / ${goldenCases.length}`);

  if (regressions.length > 0) {
    console.error('Se encontraron regresiones bloqueantes:');
    regressions.forEach((r) => console.error(` - ${r.id}`));
  }

  return { regressions, improvements };
}
```

</details>

Si un candidato presenta una regresión en fallos conocidos, la conversación sobre la actualización se vuelve maravillosamente específica: qué casos mejoraron, qué casos se rompieron y si el intercambio vale la pena.

### 4. Usa LLM-as-Judge para exactamente una cosa

El enfoque de LLM-como-juez es útil para resultados abiertos donde no hay una respuesta correcta determinista: "¿es esta respuesta útil?", "¿conserva este resumen los puntos clave?", "¿es esta explicación adecuada para un principiante?".

Úsalo ahí. No lo uses para respuestas deterministas. Cuando lo uses, haz que la rúbrica de calificación sea explícita:

**Arnés de evals: juez basado en rúbrica**

```typescript
async function judgeHelpfulness(
  userQuery: string,
  modelResponse: string
): Promise<{ score: number; reasoning: string }> {
  const judgePrompt = `
Eres un evaluador de respuestas de atención al cliente.

Pregunta del usuario: ${userQuery}
Respuesta: ${modelResponse}

Califica la respuesta en una escala del 1 al 5:
5 = Responde directamente a la pregunta con información precisa y accionable
4 = Responde a la pregunta pero podría ser más específico o accionable
3 = Responde parcialmente a la pregunta; falta información clave
2 = Tangencialmente relacionado pero no responde a la pregunta
1 = Fuera de tema, factualmente incorrecto o dañino

Responde con JSON: {"score": <número>, "reasoning": "<una oración>"}
`;

  const result = await judgeModel.generate(judgePrompt);
  return JSON.parse(result);
}
```

Una rúbrica explícita reduce la varianza del evaluador, te da resultados interpretables y facilita la auditoría cuando el juez se equivoca. Librerías como [Autoevals](https://github.com/braintrustdata/autoevals) y [Braintrust](https://www.braintrust.dev/) ofrecen rúbricas preconstruidas para tareas comunes; vale la pena usarlas antes de escribir las tuyas desde cero.

## Herramientas que vale la pena conocer

No tienes que construir todo esto desde cero. Varias herramientas han hecho progresos serios en el problema de la infraestructura de evals:

**[Braintrust](https://www.braintrust.dev/)** — Plataforma completa de evals con seguimiento de experimentos, gestión de conjuntos de datos y funciones de puntuación. Organiza las ejecuciones de eval por prompt, modelo y despliegue para que puedas comparar la calidad a lo largo del tiempo, no solo entre lanzamientos. Combina bien con su librería de código abierto **[Autoevals](https://github.com/braintrustdata/autoevals)**, que incluye funciones de puntuación calificadas por modelos para tareas comunes (precisión factual, utilidad, toxicidad, similitud semántica).

**[Langfuse](https://langfuse.com/)** — Observabilidad de LLM de código abierto que se sitúa entre tu aplicación y tus modelos. Traza cada llamada, adjunta puntuaciones de eval (humanas o automatizadas) a tramos individuales y muestra tendencias de calidad sobre el tráfico de producción. Una buena opción si quieres observabilidad y evals en la misma herramienta en lugar de un arnés de eval separado.

**[Evalite](https://www.evalite.dev/)** — Framework de evals nativo para TypeScript creado por Matt Pocock. Sin complicaciones: define una tarea, define un evaluador, ejecútalo en tu configuración de pruebas existente. Dirigido a equipos que quieren evals que se sientan como pruebas unitarias en lugar de una plataforma de experimentos de ML separada.

**[promptfoo](https://www.promptfoo.dev/)** — Ejecutor de evals centrado en CLI enfocado en la comparación de prompts y el red-teaming. Fácil de configurar mediante YAML, se integra con la mayoría de los proveedores de modelos y tiene soporte integrado para detectar inyección de prompts y otras entradas adversas.

**[deepeval](https://docs.confident-ai.com/)** — Framework de evals en Python con una gran librería de métricas integradas (G-Eval, fidelidad RAG, relevancia de respuesta, detección de alucinaciones). Útil para pipelines RAG donde quieres una calificación específica para la calidad de recuperación, no solo la de generación.

La herramienta adecuada depende de tu stack y de dónde estés empezando. Lo que importa más que la elección del framework es la disciplina de ejecutar evals: de manera consistente, en cada cambio significativo.

## La parte incómoda

La mayoría de los equipos se saltan esto porque plantea una pregunta irritante muy pronto: ¿qué significaría que algo sea "bueno" aquí?

Eso es genuinamente difícil para una nueva funcionalidad de IA. También es obligatorio si te importa la fiabilidad. Los equipos que lanzan IA confiable están haciendo lo mismo que harían para cualquier ruta de código crítica: definir el comportamiento esperado, probarlo y ejecutar esas pruebas continuamente.

Los benchmarks no mienten. Están respondiendo a la pregunta de alguien más. Deja de leerlos como hojas de ruta de producto y empieza a escribir pruebas que coincidan con tu sistema.

Tus usuarios lo notarán antes que tus paneles de control. Construye primero la suite de pruebas.
