每个新模型都穿着基准的燕尾服登场。

MMLU：92.4%。HumanEval：87.2%。LLeMU：88.7%。MATH：73.6%。AGI：127%！

然而，对于 99% 用 AI 构建流程和产品的企业来说，**这些根本不重要**。

重要的是什么？你的工作负载表现如何？在提升还是在退步？唯一理性的办法是编写 Evals（LLM 测试），让它们映射系统的具体任务、数据和失效模式。

## “基于氛围”的评估到底花了什么成本

标准做法：发布模型改动，监控投诉渠道，若舆论沸腾就回滚。

这种方式几乎遗漏了所有有价值的信号：

**你只能捕捉到噪音大的失效。** 那些自信给出错误答案却未察觉的用户？沉默。 那些得到更差答案后直接放弃功能的用户？沉默。 支持工单和错误率只能捕获质量回退的一小部分。

**你无法区分回退和改进。** 如果新模型在任务 A 上表现更好，却在任务 B 上变差，关于 B 的投诉看起来和“一般的 AI 变差”反馈完全一样。你根本不知道该修哪儿。

**你把用户当成了测试基础设施。** 他们可没签约来当你的 QA。

## Eval 评估光谱（以及大多数团队出错的地方）

评估方法分布在从“快但不可靠”到“昂贵但有效”的光谱上。

<figure class="breakout">

![一张光谱图，比较确定性检查、LLM‑as‑judge 与人工评估在速度、成本和有效性上的差异。](../eval-spectrum.svg)

<figcaption>使用最便宜且能够真实捕获失效的评估方法。</figcaption>
</figure>

**LLM‑as‑judge** 是当下的宠儿：让强大的模型给另一个模型的输出打分。快、可扩展、便宜。问题在于它把评分模型的偏见硬编码进去，容易被游戏化，并且形成循环依赖。如果你用 GPT‑5 去给 GPT‑5 的输出打分，你实际上测的是“GPT‑5 与 GPT‑5 的一致程度”。这并非毫无价值，但也不是你想要的度量。

**人工评估** 是大家都想绕开的金标准。让真人评估输出成本高、速度慢、评估者之间不一致，而且排期麻烦。但它是唯一能验证系统对真实人类是否有用的手段。

**任务特定的自动化检查** 才是大多数团队应该投入更多时间的地方。它们不光鲜，却快速、确定，并且直接关联到系统真正关心的指标。

## 实际有效的做法

### 1. 在发布前定义失效

在修改模型或提示词之前，先把“坏”到底是什么写下来。要具体。

不是“输出应该准确”。那不是测试。更像是：

- 结构化的 JSON 输出必须能够解析且不报错  
- 响应中的所有引用必须在检索到的上下文中逐字出现  
- 回答不得提及竞争对手的产品名称  
- SQL 查询必须语法合法，并且只引用模式中实际存在的表  
- 在现有测试集上，情感分类的正向→负向翻转率不能超过 3%

这些都可以用程序自动检查。无需评判模型。

**Eval harness：确定性检查**

```typescript
type EvalResult = { passed: boolean; reason?: string };

const evals: Record<string, (output: string, context: EvalContext) => EvalResult> = {
  // JSON 必须可解析
  validJson: (output) => {
    try {
      JSON.parse(output);
      return { passed: true };
    } catch (e) {
      return { passed: false, reason: `Invalid JSON: ${e.message}` };
    }
  },

  // 没有凭空捏造的引用——每条主张必须出现在上下文中
  groundedCitations: (output, { retrievedChunks }) => {
    const claims = extractCitations(output);
    const ungrounded = claims.filter(
      (claim) => !retrievedChunks.some((chunk) => chunk.includes(claim))
    );
    return ungrounded.length === 0
      ? { passed: true }
      : { passed: false, reason: `Ungrounded claims: ${ungrounded.join(', ')}` };
  },

  // 响应长度合理性检查——捕获截断或失控生成
  reasonableLength: (output) => {
    const words = output.split(/\s+/).length;
    return words >= 10 && words <= 2000
      ? { passed: true }
      : { passed: false, reason: `Word count ${words} out of bounds` };
  },
};
```

### 2. 从最糟糕的日子里构建 Golden 集

最有价值的评估数据往往是那些让人尴尬的案例：导致用户提交工单、截屏出现幻觉，或悄然停止使用功能的输出。

每当用户报告错误输出、标记幻觉，或你手动发现失效时，都把它加入 Golden 集：包括输入、上下文以及正确的行为。保持 50‑100 条案例，并在每次模型变更时全部跑一遍。

这乍看起来像是手工活。六个月后，你会拥有一套任何公开基准都无法操控的测试套件，因为每个案例都来源于你自己的故障历史。

<figure class="breakout">

![展示生产事故如何转化为 golden case、随后 CI 评估运行、再到阻止回归或批准发布的工作流图。](../golden-set-lifecycle.svg)

<figcaption>Golden 集把尴尬的案例变成回归套件。</figcaption>
</figure>

**Golden case 结构**

```typescript
interface GoldenCase {
  id: string;
  input: string;
  context: Record<string, unknown>;
  expectedBehavior: {
    mustContain?: string[];
    mustNotContain?: string[];
    structureCheck?: (output: string) => boolean;
    minSimilarityToReference?: number; // cosine similarity to a reference answer
  };
  sourceIncident?: string; // link back to the bug report or ticket
}
```

### 3. 回归测试，而不仅是验收测试

大多数团队只在考虑模型变更时运行评估。这属于验收测试：“这个新东西够好吗？”

你还需要回归测试：“这是否破坏了之前能正常工作的功能？”

在每次提示词变更时都运行 golden 集，而不仅仅是模型变更。一个原本正常的提示词在你添加新工具、改动 RAG 检索策略或更新上下文模板时，可能会悄然退化。没有基线，你根本不知道。像 [Langfuse](https://langfuse.com/) 这样的工具会把评估分数附加到生产追踪上，使回归在仪表盘中显现，而不仅仅出现在事故报告里。

<details>
<summary>Eval harness：基线 vs 候选对比</summary>

```typescript
async function compareModelVersions(
  goldenCases: GoldenCase[],
  baselinePipeline: Pipeline,
  candidatePipeline: Pipeline
) {
  const results = await Promise.all(
    goldenCases.map(async (tc) => {
      const [baseline, candidate] = await Promise.all([
        baselinePipeline.run(tc.input, tc.context),
        candidatePipeline.run(tc.input, tc.context),
      ]);

      return {
        id: tc.id,
        baselinePassed: runEvals(baseline, tc.expectedBehavior),
        candidatePassed: runEvals(candidate, tc.expectedBehavior),
        regression: /* baseline passed */ && /* candidate failed */,
        improvement: /* baseline failed */ && /* candidate passed */,
      };
    })
  );

  const regressions = results.filter((r) => r.regression);
  const improvements = results.filter((r) => r.improvement);

  console.log(`Regressions: ${regressions.length} / ${goldenCases.length}`);
  console.log(`Improvements: ${improvements.length} / ${goldenCases.length}`);

  if (regressions.length > 0) {
    console.error('Blocking regressions found:');
    regressions.forEach((r) => console.error(` - ${r.id}`));
  }

  return { regressions, improvements };
}
```

</details>

如果候选模型在已知故障上出现回归，升级讨论会变得异常具体：哪些案例得到改进，哪些案例被破坏，以及这种权衡是否值得。

### 4. 将 LLM-as-Judge 用于唯一的那件事

LLM‑as‑Judge 适用于没有确定性正确答案的开放式输出，例如：“这个回复有帮助吗？”，“这个摘要保留了关键要点吗？”，“这个解释对初学者来说是否正确？”  
在这些场景使用它。不要把它用于确定性答案。若必须使用，请明确评分标准：

**Eval harness: 基于 rubric 的评审器**

```typescript
async function judgeHelpfulness(
  userQuery: string,
  modelResponse: string
): Promise<{ score: number; reasoning: string }> {
  const judgePrompt = `
You are evaluating a customer support response.

User question: ${userQuery}
Response: ${modelResponse}

Rate the response on a scale of 1-5:
5 = Directly answers the question with accurate, actionable information
4 = Answers the question but could be more specific or actionable
3 = Partially addresses the question; key information is missing
2 = Tangentially related but doesn't answer the question
1 = Off-topic, factually wrong, or harmful

Respond with JSON: {"score": <number>, "reasoning": "<one sentence>"}
`;

  const result = await judgeModel.generate(judgePrompt);
  return JSON.parse(result);
}
```

明确的 rubric 能降低评审者的方差，提供可解释的输出，并在评审出错时更易审计。像 [Autoevals](https://github.com/braintrustdata/autoevals) 和 [Braintrust](https://www.braintrust.dev/) 这样的库已经提供了常见任务的预构建 rubric——在自己从零编写之前不妨直接使用。

## 值得了解的工具

你不必从头全部实现。已有多款工具在评估基础设施上取得了实质性进展：

**[Braintrust](https://www.braintrust.dev/)** — 完整的评估平台，支持实验追踪、数据集管理和评分函数。按 Prompt、模型、部署组织评估运行，能够对质量随时间的变化进行对比，而不仅仅是跨版本对比。配合其开源的 **[Autoevals](https://github.com/braintrustdata/autoevals)** 库使用效果更佳，后者提供了常见任务（事实准确性、帮助程度、毒性、语义相似度）的预构建模型评分函数。

**[Langfuse](https://langfuse.com/)** — 开源的 LLM 可观测性平台，位于你的应用与模型之间。追踪每一次调用，将评估分数（人工或自动）附加到单独的 span 上，并在生产流量中展示质量趋势。如果你希望在同一工具中同时拥有可观测性和评估功能，而不是使用独立的评估套件，这是个不错的选择。

**[Evalite](https://www.evalite.dev/)** — 由 Matt Pocock 开发的 TypeScript 原生评估框架。低侵入性：定义任务、定义评分器，然后在现有测试环境中运行。面向希望评估体验像单元测试而非独立机器学习实验平台的团队。

**[promptfoo](https://www.promptfoo.dev/)** — 以 CLI 为主的评估运行器，侧重提示比较和红队测试。通过 YAML 轻松配置，兼容大多数模型提供商，并内置对提示注入等对抗性输入的检测支持。

**[deepeval](https://docs.confident-ai.com/)** — Python 评估框架，提供丰富的内置指标库（G‑Eval、RAG 可信度、答案相关性、幻觉检测）。在需要对检索质量进行专门打分的 RAG 流水线中尤为有用，而不仅仅是生成质量。

合适的工具取决于你的技术栈和起点。比起框架本身，更关键的是养成始终运行评估的纪律——对每一次重要改动都保持一致的测试。

## 让人不舒服的部分

大多数团队会跳过这一步，因为它在一开始就抛出一个让人恼火的问题：这里的“好”到底是什么样子？

对一个新 AI 功能来说，这确实很难回答。但如果你在乎可靠性，这一步绝对不可省略。交付可信 AI 的团队会像对待任何关键代码路径一样：定义预期行为、编写测试，并持续运行这些测试。

基准本身并没有撒谎，它们只是在回答别人的问题。别把它们当作产品路线图来看待，转而编写与你系统匹配的测试。

用户会在仪表盘之前发现问题。先构建测试套件吧。
