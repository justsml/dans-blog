Каждая новая модель приезжает в смокинге из бенчмарков.

MMLU: 92.4%. HumanEval: 87.2%. LLeMU: 88.7%. MATH: 73.6%. AGI: 127%!

Но для 99% компаний, которые строят процессы и продукты на базе ИИ, **всё это не имеет значения.**

Что имеет значение? Как работают ВАШИ нагрузки? Улучшаются или ухудшаются? Единственный здравый способ узнать это — написать eval-ы (тесты для LLM), которые отражают конкретные задачи, данные и режимы отказов вашей системы.

<blockquote class="breakout">
  <p>Бенчмарки не лгут. Они отвечают на чужой вопрос.</p>
</blockquote>

## Сколько На самом Деле Стоит "Оценочка На Ощупь"

Стандартный подход: выпустил изменение модели, смотришь каналы жалоб, откатываешь, если в комнате стало шумно.

Это упускает почти всё интересное:

**Вы ловите только громкие отказы.** Пользователи, которые получают уверенно неверный ответ и не понимают этого? Тишина. Пользователи, которые получают хуже и уходят с фичи? Тишина. Тикеты поддержки и уровни ошибок захватывают лишь долю регресса качества.

**Вы не можете отличить регресс от улучшений.** Если новая модель лучше в задаче A и хуже в задаче B, жалобы на B выглядят идентично общему фидбеку "ИИ стал хуже". Вы не знаете, что чинить.

**Вы используете своих пользователей как тестовую инфраструктуру.** Они на это не подписывались.

## Спектр Eval-ов (и Где Большинство Команд Промахивается)

Подходы к оценке находятся на спектре от "быстро но шатко" до "дорого но валидно".

<figure class="breakout">

![Диаграмма спектра, сравнивающая детерминированные проверки, LLM-as-judge и человеческую оценку по скорости, стоимости и валидности.](../eval-spectrum.svg)

<figcaption>Используйте самый дешёвый метод оценки, который может честно поймать отказ.</figcaption>
</figure>

**LLM-as-judge** — нынешний любимчик: попросите мощную модель оценить вывод другой модели. Быстро, масштабируемо, дёшево. Проблема: он вшивает предвзятость модели-судьи, его можно обмануть, и он создаёт циклическую зависимость. Если вы используете GPT-5 для оценки выводов GPT-5, вы измеряете что-то вроде "насколько GPT-5 согласна с GPT-5". Это не ноль, но это не то, что вы думаете.

**Human eval** — золотой стандарт, который все пытаются пропустить. Заставить людей оценивать выводы дорого, медленно, inconsistently между оценщиками, и неприятно планировать. Но это единственное, что валидирует, полезна ли ваша система реальным людям.

**Task-specific automated checks** — там, где большинству команд стоит тратить больше времени. Они не гламурные, но они быстрые, детерминированные и привязаны к тому, что важно в вашей системе.

## Что На Самом Деле Работает

### 1. Определите Отказ До Выпуска

Перед изменением модели или промпта, запишите, как выглядит плохо. Конкретно.

Не "вывод должен быть точным". Это не тест. Чаще вроде:

- Структурированный JSON вывод должен парситься без ошибок
- Все цитаты в ответе должны появляться дословно в полученном контексте
- Ответы не должны упоминать названия продуктов конкурентов
- SQL запросы должны быть синтаксически валидными и ссылаться только на таблицы, существующие в схеме
- Классификация тональности не должна переключаться с позитивной на негативную более чем на 3% времени на существующем тестовом наборе

Вы можете проверить это программно. Никакой модели-судьи не требуется.

**Eval хаб: детерминированные проверки**

```typescript
type EvalResult = { passed: boolean; reason?: string };

const evals: Record<string, (output: string, context: EvalContext) => EvalResult> = {
  // JSON должен парситься
  validJson: (output) => {
    try {
      JSON.parse(output);
      return { passed: true };
    } catch (e) {
      return { passed: false, reason: `Invalid JSON: ${e.message}` };
    }
  },

  // Нет галлюцинированных цитат — каждое утверждение должно появляться в контексте
  groundedCitations: (output, { retrievedChunks }) => {
    const claims = extractCitations(output);
    const ungrounded = claims.filter(
      (claim) => !retrievedChunks.some((chunk) => chunk.includes(claim))
    );
    return ungrounded.length === 0
      ? { passed: true }
      : { passed: false, reason: `Ungrounded claims: ${ungrounded.join(', ')}` };
  },

  // Проверка разумной длины ответа — поймайте усечение или runaway генерацию
  reasonableLength: (output) => {
    const words = output.split(/\s+/).length;
    return words >= 10 && words <= 2000
      ? { passed: true }
      : { passed: false, reason: `Word count ${words} out of bounds` };
  },
};
```

### 2. Соберите Золотой Набор Из Ваших Худших Дней

Ваши лучшие данные для оценки — это смехотворное: выводы, которые заставили кого-то подать тикет, сделать скриншот галлюцинации, или тихо перестать использовать фичу.

Каждый раз, когда пользователь сообщает о плохом выводе, отмечает галлюцинацию, или вы замечаете отказ вручную, добавьте его в ваш золотой набор: ввод, контекст, и правильное поведение. Держите 50-100 случаев и запускайте их на каждое изменение модели.

Это кажется ручным сначала. Через шесть месяцев у вас есть тестовый набор, который ни один публичный бенчмарк не может обмануть, потому что каждый случай пришёл из вашей собственной истории отказов.

<figure class="breakout">

![Диаграмма рабочего процесса, показывающая, как плохие продакшн инциденты становятся золотыми случаями, затем CI eval запусками, затем заблокированными регрессами или одобренными релизами.](../golden-set-lifecycle.svg)

<figcaption>Золотой набор превращает смехотворное в регрессионный набор.</figcaption>
</figure>

**Форма золотого случая**

```typescript
interface GoldenCase {
  id: string;
  input: string;
  context: Record<string, unknown>;
  expectedBehavior: {
    mustContain?: string[];
    mustNotContain?: string[];
    structureCheck?: (output: string) => boolean;
    minSimilarityToReference?: number; // cosine similarity to a reference answer
  };
  sourceIncident?: string; // link back to the bug report or ticket
}
```

### 3. Регрессионное Тестирование, Не Только Acceptance Тестирование

Большинство команд запускают eval-ы только при рассмотрении изменения модели. Это acceptance тестирование: "достаточно ли хороша эта новая штука?"

Вам также нужно регрессионное тестирование: "не сломало ли это что-то, что раньше работало?"

Запускайте ваш золотой набор на каждое изменение промпта, не только изменения модели. Промпт, который работал нормально, может тихо деградировать, когда вы добавляете новый инструмент, меняете RAG стратегию извлечения, или обновляете ваш контекстный шаблон. Вы не узнаете без базовой линии. Инструменты вроде [Langfuse](https://langfuse.com/) прикрепляют eval оценки к продакшн трассам, так что регресс появляется в дашбордах, не только в отчётах об инцидентах.

<details>
<summary>Eval хаб: сравнение базовой линии с кандидатом</summary>

```typescript
async function compareModelVersions(
  goldenCases: GoldenCase[],
  baselinePipeline: Pipeline,
  candidatePipeline: Pipeline
) {
  const results = await Promise.all(
    goldenCases.map(async (tc) => {
      const [baseline, candidate] = await Promise.all([
        baselinePipeline.run(tc.input, tc.context),
        candidatePipeline.run(tc.input, tc.context),
      ]);

      return {
        id: tc.id,
        baselinePassed: runEvals(baseline, tc.expectedBehavior),
        candidatePassed: runEvals(candidate, tc.expectedBehavior),
        regression: /* baseline passed */ && /* candidate failed */,
        improvement: /* baseline failed */ && /* candidate passed */,
      };
    })
  );

  const regressions = results.filter((r) => r.regression);
  const improvements = results.filter((r) => r.improvement);

  console.log(`Regressions: ${regressions.length} / ${goldenCases.length}`);
  console.log(`Improvements: ${improvements.length} / ${goldenCases.length}`);

  if (regressions.length > 0) {
    console.error('Blocking regressions found:');
    regressions.forEach((r) => console.error(` - ${r.id}`));
  }

  return { regressions, improvements };
}
```

</details>

Если кандидат регрессирует на известных отказах, разговор об обновлении становится wonderfully конкретным: какие случаи улучшились, какие сломались, и стоит ли та сделка того.

### 4. Используйте LLM-as-Judge Ровно Для Одной Вещи

LLM-as-judge полезен для open-ended выводов, где нет детерминированного правильного ответа: "помогателен ли этот ответ?", "сохраняет ли это резюме ключевые точки?", "правильно ли это объяснение для новичка?"

Используйте его там. Не используйте его для детерминированных ответов. Когда вы всё же используете его, сделайте rubric оценки явной:

**Eval хаб: rubric-based судья**

```typescript
async function judgeHelpfulness(
  userQuery: string,
  modelResponse: string
): Promise<{ score: number; reasoning: string }> {
  const judgePrompt = `
You are evaluating a customer support response.

User question: ${userQuery}
Response: ${modelResponse}

Rate the response on a scale of 1-5:
5 = Directly answers the question with accurate, actionable information
4 = Answers the question but could be more specific or actionable
3 = Partially addresses the question; key information is missing
2 = Tangentially related but doesn't answer the question
1 = Off-topic, factually wrong, or harmful

Respond with JSON: {"score": <number>, "reasoning": "<one sentence>"}
`;

  const result = await judgeModel.generate(judgePrompt);
  return JSON.parse(result);
}
```

Явная rubric уменьшает variance оценщика, даёт вам интерпретируемый вывод, и делает проще аудировать, когда судья ошибается. Библиотеки вроде [Autoevals](https://github.com/braintrustdata/autoevals) и [Braintrust](https://www.braintrust.dev/) поставляют prebuilt rubrics для общих задач — стоит украсть перед написанием своих с нуля.

## Инструменты, Стоит Знать

Вам не нужно строить всё это с нуля. Несколько инструментов сделали серьёзный прогресс в проблеме eval инфраструктуры:

**[Braintrust](https://www.braintrust.dev/)** — Полная eval платформа с experiment tracking, управлением датасетами, и scoring функциями. Организует eval запуски по промпту, модели, и деплою, так что вы можете diff качество со временем, не только между релизами. Хорошо сочетается с их open-source **[Autoevals](https://github.com/braintrustdata/autoevals)** библиотекой, которая поставляет prebuilt model-graded scoring функции для общих задач (фактическая точность, helpfulness, toxicity, semantic similarity).

**[Langfuse](https://langfuse.com/)** — Open-source LLM observability, которая сидит между вашим приложением и вашими моделями. Трассирует каждый вызов, прикрепляет eval оценки (человеческие или автоматические) к individual spans, и surface trends качества поверх продакшн трафика. Хороший выбор, если вы хотите observability и eval-ы в том же инструменте, а не отдельный eval хаб.

**[Evalite](https://www.evalite.dev/)** — TypeScript-native eval фреймворк от Matt Pocock. Low ceremony: определите задачу, определите scorer, запустите его в вашей существующей тестовой настройке. Целится в команды, которые хотят eval-ы, которые чувствуются как unit тесты, а не отдельная ML experiment платформа.

**[promptfoo](https://www.promptfoo.dev/)** — CLI-first eval runner, сфокусированный на сравнении промптов и red-teaming. Лёгко конфигурировать через YAML, интегрируется с большинством провайдеров моделей, и имеет built-in поддержку обнаружения prompt injection и других adversarial вводов.

**[deepeval](https://docs.confident-ai.com/)** — Python eval фреймворк с большой библиотекой built-in метрик (G-Eval, RAG faithfulness, answer relevancy, hallucination detection). Полезен для RAG pipelines, где вы хотите specific grading для качества извлечения, не только качества генерации.

Правильный инструмент зависит от вашего стека и того, откуда вы начинаете. Что важнее выбора фреймворка — дисциплина запуска eval-ов вообще — consistently, на каждое значительное изменение.

## Неприятная Часть

Большинство команд пропускают это, потому что оно задаёт раздражающий вопрос рано: как бы выглядело "хорошо" здесь?

Это genuinely трудно для новой ИИ фичи. Это также non-optional, если вы заботитесь о reliability. Команды, которые ship trustworthy ИИ, делают ту же вещь, что они бы сделали для любого critical code path: определите ожидаемое поведение, протестируйте его, и запускайте эти тесты continuously.

Бенчмарки не лгут. Они отвечают на чужой вопрос. Перестаньте читать их как product roadmaps и начните писать тесты, которые соответствуют вашей системе.

Ваши пользователи заметят раньше, чем ваши дашборды. Постройте тестовый набор сначала.
