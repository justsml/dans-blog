Каждая новая модель появляется в смокинге из бенчмарков.

MMLU: 92.4%. HumanEval: 87.2%. LLeMU: 88.7%. MATH: 73.6%. AGI: 127%!

Однако для 99% бизнесов, строящих процессы и продукты с ИИ, **всё это не имеет значения.**

Что имеет значение? Как справляются ВАШИ задачи? Становится лучше или хуже? Единственный разумный способ узнать это — написать Evals (тесты для LLM), которые отражают конкретные задачи, данные и сценарии отказов вашей системы.

<blockquote class="breakout">
  <p>Бенчмарки не врут. Они отвечают на чей-то чужой вопрос.</p>
</blockquote>

## Сколько на самом деле стоит «оценка по ощущениям»

Стандартный подход: выкатить новую модель, следить за каналами жалоб, откатить, если шумно.

Это упускает почти всё интересное:

**Вы ловите только громкие сбои.** Пользователи, получившие уверенный неверный ответ и не осознавшие этого? Молчат. Пользователи, получившие ответ хуже и бросившие функцию? Молчат. Тикеты в поддержку и частота ошибок улавливают лишь малую часть ухудшения качества.

**Вы не можете отличить регрессии от улучшений.** Если новая модель лучше в задаче A и хуже в задаче B, жалобы на B выглядят так же, как общая обратная связь «ИИ стал хуже». Вы не знаете, что исправлять.

**Вы используете своих пользователей как тестовую инфраструктуру.** Они на это не подписывались.

## Спектр Evals (и где большинство команд ошибается)

Подходы к оценке располагаются на спектре от «быстро, но ненадёжно» до «дорого, но валидно».

<figure class="breakout">

![Спектральная диаграмма, сравнивающая детерминированные проверки, LLM-как-судья и человеческую оценку по скорости, стоимости и валидности.](../eval-spectrum.svg)

<figcaption>Используйте самый дешёвый метод оценки, который может честно выявить сбой.</figcaption>
</figure>

**LLM-как-судья** — текущий фаворит: попросите мощную модель оценить результаты другой модели. Быстро, масштабируемо, дёшево. Проблема: он встраивает предвзятости модели-оценщика, его можно обыграть, и он создаёт циклическую зависимость. Если вы используете GPT-5 для оценки результатов GPT-5, вы измеряете что-то вроде «насколько GPT-5 согласен с GPT-5». Это не ничего, но это не то, что вы думаете.

**Человеческая оценка** — это золотой стандарт, который все пытаются пропустить. Привлечение людей для оценки результатов дорого, медленно, непоследовательно между оценщиками и раздражает при планировании. Но это единственное, что подтверждает, полезна ли ваша система реальным людям.

**Автоматизированные проверки для конкретных задач** — вот на что большинству команд стоит тратить больше времени. Они не гламурны, но они быстры, детерминированы и привязаны к тому, что важно в вашей системе.

## Что действительно работает

### 1. Определите отказ до выкатки

Прежде чем менять модель или промпт, запишите, как выглядит плохой результат. Конкретно.

Не «вывод должен быть точным». Это не тест. А вот так:

- Структурированный JSON-вывод должен парситься без ошибок
- Все цитаты в ответе должны дословно присутствовать в полученном контексте
- Ответы не должны упоминать названия продуктов конкурентов
- SQL-запросы должны быть синтаксически валидны и ссылаться только на таблицы, существующие в схеме
- Классификация тональности не должна переключаться с положительной на отрицательную более чем в 3% случаев на существующем тестовом наборе

Это можно проверить программно. Модель-судья не требуется.

**Инфраструктура Evals: детерминированные проверки**

```typescript
type EvalResult = { passed: boolean; reason?: string };

const evals: Record<string, (output: string, context: EvalContext) => EvalResult> = {
  // JSON must parse
  validJson: (output) => {
    try {
      JSON.parse(output);
      return { passed: true };
    } catch (e) {
      return { passed: false, reason: `Invalid JSON: ${e.message}` };
    }
  },

  // No hallucinated citations — every claim must appear in context
  groundedCitations: (output, { retrievedChunks }) => {
    const claims = extractCitations(output);
    const ungrounded = claims.filter(
      (claim) => !retrievedChunks.some((chunk) => chunk.includes(claim))
    );
    return ungrounded.length === 0
      ? { passed: true }
      : { passed: false, reason: `Ungrounded claims: ${ungrounded.join(', ')}` };
  },

  // Response length sanity check — catch truncation or runaway generation
  reasonableLength: (output) => {
    const words = output.split(/\s+/).length;
    return words >= 10 && words <= 2000
      ? { passed: true }
      : { passed: false, reason: `Word count ${words} out of bounds` };
  },
};
```

### 2. Соберите золотой набор из ваших худших дней

Ваши лучшие данные для оценки — это то, что вызывает неловкость: результаты, из-за которых кто-то создал тикет, сделал скриншот галлюцинации или тихо перестал использовать функцию.

Каждый раз, когда пользователь сообщает о плохом результате, отмечает галлюцинацию или вы вручную замечаете сбой, добавляйте это в свой золотой набор: входные данные, контекст и правильное поведение. Храните 50-100 кейсов и запускайте их при каждом изменении модели.

Сначала это кажется ручным трудом. Через шесть месяцев у вас есть тестовый набор, который не может обыграть ни один публичный бенчмарк, потому что каждый кейс взят из вашей собственной истории отказов.

<figure class="breakout">

![Диаграмма рабочего процесса, показывающая, как плохие производственные инциденты становятся золотыми кейсами, затем прогонами CI-оценки, затем заблокированными регрессиями или одобренными релизами.](../golden-set-lifecycle.svg)

<figcaption>Золотой набор превращает неловкие моменты в набор регрессионных тестов.</figcaption>
</figure>

**Структура золотого кейса**

```typescript
interface GoldenCase {
  id: string;
  input: string;
  context: Record<string, unknown>;
  expectedBehavior: {
    mustContain?: string[];
    mustNotContain?: string[];
    structureCheck?: (output: string) => boolean;
    minSimilarityToReference?: number; // cosine similarity to a reference answer
  };
  sourceIncident?: string; // link back to the bug report or ticket
}
```

### 3. Регрессионное тестирование, а не только приёмочное

Большинство команд запускает evals только при рассмотрении смены модели. Это приёмочное тестирование: «достаточно ли хороша эта новая штука?»

Вам также нужно регрессионное тестирование: «не сломалось ли что-то, что раньше работало?»

Запускайте свой золотой набор при каждом изменении промпта, а не только модели. Промпт, который отлично работал, может незаметно ухудшиться, когда вы добавляете новый инструмент, меняете стратегию поиска RAG или обновляете шаблон контекста. Вы не узнаете об этом без базовой линии. Инструменты вроде [Langfuse](https://langfuse.com/) прикрепляют оценки evals к производственным трейсам, так что регрессии отображаются в дашбордах, а не только в отчётах об инцидентах.

<details>
<summary>Инфраструктура Evals: сравнение базовой и кандидатской версий</summary>

```typescript
async function compareModelVersions(
  goldenCases: GoldenCase[],
  baselinePipeline: Pipeline,
  candidatePipeline: Pipeline
) {
  const results = await Promise.all(
    goldenCases.map(async (tc) => {
      const [baseline, candidate] = await Promise.all([
        baselinePipeline.run(tc.input, tc.context),
        candidatePipeline.run(tc.input, tc.context),
      ]);

      return {
        id: tc.id,
        baselinePassed: runEvals(baseline, tc.expectedBehavior),
        candidatePassed: runEvals(candidate, tc.expectedBehavior),
        regression: /* baseline passed */ && /* candidate failed */,
        improvement: /* baseline failed */ && /* candidate passed */,
      };
    })
  );

  const regressions = results.filter((r) => r.regression);
  const improvements = results.filter((r) => r.improvement);

  console.log(`Regressions: ${regressions.length} / ${goldenCases.length}`);
  console.log(`Improvements: ${improvements.length} / ${goldenCases.length}`);

  if (regressions.length > 0) {
    console.error('Blocking regressions found:');
    regressions.forEach((r) => console.error(` - ${r.id}`));
  }

  return { regressions, improvements };
}
```

</details>

Если кандидат регрессирует на известных сбоях, разговор об обновлении становится wonderfully конкретным: какие кейсы улучшились, какие сломались и стоит ли игра свеч.

### 4. Используйте LLM-как-судью ровно для одной вещи

LLM-как-судья полезен для открытых результатов, где нет детерминированного правильного ответа: «полезен ли этот ответ?», «сохраняет ли это резюме ключевые моменты?», «подходит ли это объяснение для новичка?»

Используйте его для этого. Не используйте для детерминированных ответов. Когда вы его используете, сделайте рубрику оценки явной:

**Инфраструктура Evals: судья на основе рубрики**

```typescript
async function judgeHelpfulness(
  userQuery: string,
  modelResponse: string
): Promise<{ score: number; reasoning: string }> {
  const judgePrompt = `
You are evaluating a customer support response.

User question: ${userQuery}
Response: ${modelResponse}

Rate the response on a scale of 1-5:
5 = Directly answers the question with accurate, actionable information
4 = Answers the question but could be more specific or actionable
3 = Partially addresses the question; key information is missing
2 = Tangentially related but doesn't answer the question
1 = Off-topic, factually wrong, or harmful

Respond with JSON: {"score": <number>, "reasoning": "<one sentence>"}
`;

  const result = await judgeModel.generate(judgePrompt);
  return JSON.parse(result);
}
```

Явная рубрика снижает вариативность оценки, даёт интерпретируемый результат и упрощает аудит, когда судья ошибается. Библиотеки вроде [Autoevals](https://github.com/braintrustdata/autoevals) и [Braintrust](https://www.braintrust.dev/) поставляют готовые рубрики для типовых задач — стоит позаимствовать, прежде чем писать свои с нуля.

## Инструменты, которые стоит знать

Вам не обязательно строить всё это с нуля. Несколько инструментов добились серьёзного прогресса в проблеме инфраструктуры evals:

**[Braintrust](https://www.braintrust.dev/)** — Полноценная платформа evals с отслеживанием экспериментов, управлением наборами данных и функциями оценки. Организует прогоны evals по промпту, модели и развёртыванию, чтобы вы могли сравнивать качество во времени, а не только между релизами. Хорошо сочетается с их open-source библиотекой **[Autoevals](https://github.com/braintrustdata/autoevals)**, которая содержит готовые функции оценки на основе модели для типовых задач (фактическая точность, полезность, токсичность, семантическое сходство).

**[Langfuse](https://langfuse.com/)** — Open-source LLM-обсервабилити, которая находится между вашим приложением и моделями. Трассирует каждый вызов, прикрепляет оценки evals (человеческие или автоматические) к отдельным спанам и показывает тренды качества на основе production-трафика. Хороший выбор, если вы хотите обсервабилити и evals в одном инструменте, а не в отдельных инфраструктурах.

**[Evalite](https://www.evalite.dev/)** — TypeScript-native eval-фреймворк от Мэтта Покока. Мало церемоний: определите задачу, определите оценщика, запустите в существующей тестовой среде. Ориентирован на команды, которые хотят, чтобы evals ощущались как модульные тесты, а не как отдельная ML-экспериментальная платформа.

**[promptfoo](https://www.promptfoo.dev/)** — CLI-ориентированный eval-раннер, сфокусированный на сравнении промптов и red-teaming. Легко настраивается через YAML, интегрируется с большинством провайдеров моделей и имеет встроенную поддержку обнаружения prompt-инъекций и других состязательных входных данных.

**[deepeval](https://docs.confident-ai.com/)** — Python eval-фреймворк с большой библиотекой встроенных метрик (G-Eval, RAG faithfulness, релевантность ответа, обнаружение галлюцинаций). Полезен для RAG-пайплайнов, где нужна специфическая оценка качества поиска, а не только качества генерации.

Правильный инструмент зависит от вашего стека и точки старта. Важнее выбора фреймворка — дисциплина запуска evals вообще: последовательно, при каждом значимом изменении.

## Неудобная часть

Большинство команд пропускает это, потому что он задаёт раздражающий вопрос на раннем этапе: как здесь выглядит «хорошо»?

Это действительно сложно для новой AI-функции. Но это не опционально, если вы заботитесь о надёжности. Команды, которые выпускают надёжный AI, делают то же самое, что и для любого критического пути в коде: определяют ожидаемое поведение, тестируют его и запускают эти тесты непрерывно.

Бенчмарки не врут. Они отвечают на чей-то чужой вопрос. Перестаньте читать их как дорожные карты продуктов и начните писать тесты, которые соответствуют вашей системе.

Ваши пользователи заметят раньше, чем ваши дашборды. Сначала создайте тестовый набор.
