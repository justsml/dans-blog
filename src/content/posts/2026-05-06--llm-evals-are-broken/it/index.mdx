Ogni nuovo modello arriva vestito con un smoking di benchmark.

MMLU: 92.4%. HumanEval: 87.2%. LLeMU: 88.7%. MATH: 73.6%. AGI: 127%!

Eppure, per il 99% delle aziende che costruiscono processi e prodotti con l'IA, **nulla di tutto ciò conta**.

Cosa conta? Come stanno funzionando I TUOI carichi di lavoro? Meglio o peggio? L'unico modo sensato per saperlo è scrivere Evals (test per LLM) che riflettano i compiti, i dati e i modi di fallimento specifici del tuo sistema.

## Quanto Costa Davvero la "Valutazione basata sulle Vibrazioni"

L'approccio standard: rilasciare un cambiamento del modello, osservare i canali di reclamo, tornare indietro se la stanza diventa rumorosa.

Questo perde quasi tutto ciò che è interessante:

**Catturi solo i guasti rumorosi.** Gli utenti che ricevono una risposta correttamente sbagliata e non se ne accorgono? Silenziosi. Gli utenti che ricevono una risposta peggiore e abbandonano la funzionalità? Silenziosi. I ticket di supporto e i tassi di errore catturano solo una frazione del regresso di qualità.

**Non puoi distinguere i regressi dai miglioramenti.** Se il nuovo modello è migliore nel compito A e peggio nel compito B, i reclami su B appaiono identici al generico feedback "l'IA è peggiorata". Non sai cosa sistemare.

**Stai usando i tuoi utenti come infrastruttura di test.** Non hanno accettato per quello.

## Lo Spettro degli Evals (e dove la maggior parte dei team sbaglia)

Gli approcci di valutazione si trovano su uno spettro da "veloce ma fragile" a "costoso ma valido".

<figure class="breakout">

![Uno spettro di diagramma che confronta i controlli deterministici, LLM-as-judge e la valutazione umana per velocità, costo e validità.](../eval-spectrum.svg)

<figcaption>Usa il metodo di valutazione più economico che può onestamente catturare il fallimento.</figcaption>
</figure>

**LLM-as-judge** è l'attuale preferito: chiedere a un modello potente di valutare i output di un altro modello. Veloce, scalabile, economico. Il problema: incorpora i pregiudizi del modello valutatore, può essere ingannato e crea una dipendenza circolare. Se usi GPT-5 per valutare i output di GPT-5, stai misurando qualcosa come "quanto GPT-5 è d'accordo con GPT-5". Non è nulla, ma non è quello che pensi.

**La valutazione umana** è lo standard d'oro che tutti cercano di evitare. Ottenere esseri umani per valutare i output è costoso, lento, inconsistente tra gli valutatori e fastidioso da programmare. Ma è l'unica cosa che convalida se il tuo sistema è utile per gli esseri umani veri.

**I controlli automatizzati specifici per compito** è dove la maggior parte dei team dovrebbe spendere più tempo. Non sono glamour, ma sono veloci, deterministici e collegati a ciò che conta nel tuo sistema.

## Cosa Funziona Davvero

### 1. Definisci il Fallimento Prima di Rilasciare

Prima di cambiare un modello o un prompt, scrivi cosa significa cattivo. Specificamente.

Non "il output dovrebbe essere accurato." Quello non è un test. Più come:

- L'output JSON strutturato deve essere analizzato senza errori
- Tutte le citazioni nella risposta devono apparire verbatim nel contesto recuperato
- Le risposte non devono menzionare i nomi dei prodotti dei concorrenti
- Le query SQL devono essere sintatticamente valide e riferirsi solo ai tabelle che esistono nello schema
- La classificazione del sentiment non deve cambiare da positivo a negativo più del 3% del tempo sull'insieme di test esistente

Puoi controllare questi programmaticamente. Nessun modello giudice richiesto.

**Harness eval: controlli deterministici**

```typescript
type EvalResult = { passed: boolean; reason?: string };

const evals: Record<string, (output: string, context: EvalContext) => EvalResult> = {
  // JSON deve essere analizzato
  validJson: (output) => {
    try {
      JSON.parse(output);
      return { passed: true };
    } catch (e) {
      return { passed: false, reason: `JSON non valido: ${e.message}` };
    }
  },

  // Nessuna citazione allucinata — ogni affermazione deve apparire nel contesto
  groundedCitations: (output, { retrievedChunks }) => {
    const claims = extractCitations(output);
    const ungrounded = claims.filter(
      (claim) => !retrievedChunks.some((chunk) => chunk.includes(claim))
    );
    return ungrounded.length === 0
      ? { passed: true }
      : { passed: false, reason: `Affermazioni non fondate: ${ungrounded.join(', ')}` };
  },

  // Controllo della lunghezza della risposta — cattura la troncatura o la generazione runaway
  reasonableLength: (output) => {
    const words = output.split(/\s+/).length;
    return words >= 10 && words <= 2000
      ? { passed: true }
      : { passed: false, reason: `Conteggio parole ${words} fuori dai limiti` };
  },
};
```

### 2. Costruisci un Insieme Dorato dai Tuoi Giorni Peggiori

I tuoi dati di valutazione migliori sono le cose imbarazzanti: i output che hanno fatto a qualcuno presentare un ticket, screenshot di un'allucinazione, o fermare silenziosamente l'uso della funzionalità.

Ogni volta che un utente riporta un output cattivo, segnala un'allucinazione o noti un fallimento manualmente, aggiungilo al tuo insieme dorato: l'input, il contesto e il comportamento corretto. Mantieni 50-100 casi ed eseguili su ogni cambiamento del modello.

Questo sembra manuale all'inizio. Dopo sei mesi, hai una suite di test che nessun benchmark pubblico può ingannare, perché ogni caso è venuto dalla tua propria storia di fallimenti.

<figure class="breakout">

![Un diagramma di flusso che mostra come i guasti di produzione imbarazzanti diventano casi dorati, poi gli eval CI, poi i regressi bloccati o i rilasci approvati.](../golden-set-lifecycle.svg)

<figcaption>Un insieme dorato trasforma le cose imbarazzanti in una suite di regressione.</figcaption>
</figure>

**Forma del caso dorato**

```typescript
interface GoldenCase {
  id: string;
  input: string;
  context: Record<string, unknown>;
  expectedBehavior: {
    mustContain?: string[];
    mustNotContain?: string[];
    structureCheck?: (output: string) => boolean;
    minSimilarityToReference?: number; // similarità coseno a una risposta di riferimento
  };
  sourceIncident?: string; // link indietro al report del bug o al ticket
}
```

### 3. Testing di Regresso, Non Solo Testing di Accettazione

La maggior parte dei team esegue gli evals solo quando considera un cambiamento del modello. Quello è testing di accettazione: "questa nuova cosa è abbastanza buona?"

Hai anche bisogno di testing di regresso: "questo ha rotto qualcosa che funzionava prima?"

Esegui il tuo insieme dorato su ogni cambiamento del prompt, non solo sui cambiamenti del modello. Un prompt che funzionava bene può degradare silenziosamente quando aggiungi un nuovo strumento, cambi una strategia di recupero RAG o aggiorni il tuo template di contesto. Non lo saprai senza una baseline. Strumenti come [Langfuse](https://langfuse.com/) attaccano i punteggi eval ai traccia di produzione così il regresso appare nei dashboard, non solo nei report di incidente.

<details>
<summary>Harness eval: confronto baseline vs candidato</summary>

```typescript
async function compareModelVersions(
  goldenCases: GoldenCase[],
  baselinePipeline: Pipeline,
  candidatePipeline: Pipeline
) {
  const results = await Promise.all(
    goldenCases.map(async (tc) => {
      const [baseline, candidate] = await Promise.all([
        baselinePipeline.run(tc.input, tc.context),
        candidatePipeline.run(tc.input, tc.context),
      ]);

      return {
        id: tc.id,
        baselinePassed: runEvals(baseline, tc.expectedBehavior),
        candidatePassed: runEvals(candidate, tc.expectedBehavior),
        regression: /* baseline passato */ && /* candidato fallito */,
        improvement: /* baseline fallito */ && /* candidato passato */,
      };
    })
  );

  const regressions = results.filter((r) => r.regression);
  const improvements = results.filter((r) => r.improvement);

  console.log(`Regressi: ${regressions.length} / ${goldenCases.length}`);
  console.log(`Miglioramenti: ${improvements.length} / ${goldenCases.length}`);

  if (regressions.length > 0) {
    console.error('Regressi bloccati trovati:');
    regressions.forEach((r) => console.error(` - ${r.id}`));
  }

  return { regressions, improvements };
}
```

</details>

Se un candidato regredisce su fallimenti noti, la conversazione di upgrade diventa meravigliosamente specifica: quali casi sono migliorati, quali casi hanno rotto, e se il compromesso vale la pena.

### 4. Usa LLM-as-Judge per Esattamente Una Cosa

LLM-as-judge è utile per i output open-ended dove non c'è una risposta deterministica giusta: "è questa risposta utile?", "questo riassunto preserva i punti chiave?", "è questa spiegazione giusta per un principiante?"

Usalo lì. Non usarlo per le risposte deterministiche. Quando lo usi, rendi esplicito il rubrica di valutazione:

**Harness eval: giudice basato su rubrica**

```typescript
async function judgeHelpfulness(
  userQuery: string,
  modelResponse: string
): Promise<{ score: number; reasoning: string }> {
  const judgePrompt = `
Stai valutando una risposta di supporto clienti.

Domanda dell'utente: ${userQuery}
Risposta: ${modelResponse}

Valuta la risposta su una scala di 1-5:
5 = Risponde direttamente alla domanda con informazioni accurate e azionabili
4 = Risponde alla domanda ma potrebbe essere più specifico o azionabile
3 = Affronta parzialmente la domanda; le informazioni chiave mancano
2 = Tangenzialmente correlato ma non risponde alla domanda
1 = Fuori argomento, factualmente sbagliato o dannoso

Rispondi con JSON: {"score": <numero>, "reasoning": "<una frase>"}
`;

  const result = await judgeModel.generate(judgePrompt);
  return JSON.parse(result);
}
```

Una rubrica esplicita riduce la varianza del valutatore, ti dà un output interpretabile e rende più facile fare audit quando il giudice sbaglia. Librerie come [Autoevals](https://github.com/braintrustdata/autoevals) e [Braintrust](https://www.braintrust.dev/) spediscono rubriche precompilate per compiti comuni — vale la pena rubare prima di scrivere la tua da zero.

## Strumenti che Vale la Pena Conoscere

Non hai bisogno di costruire tutto questo da zero. Diversi strumenti hanno fatto seri progressi sul problema dell'infrastruttura eval:

**[Braintrust](https://www.braintrust.dev/)** — Piattaforma eval completa con tracciamento dell'esperimento, gestione del dataset e funzioni di punteggio. Organizza le esecuzioni eval per prompt, modello e deployment così puoi diffire la qualità nel tempo, non solo tra i rilasci. Si abbina bene con la loro libreria open-source **[Autoevals](https://github.com/braintrustdata/autoevals)**, che spedisce funzioni di punteggio model-graded precompilate per compiti comuni (accuratezza fattuale, utilità, tossicità, similarità semantica).

**[Langfuse](https://langfuse.com/)** — Osservabilità LLM open-source che si siede tra la tua app e i tuoi modelli. Traccia ogni chiamata, attacca i punteggi eval (umani o automatizzati) alle singole span e fa emergere le tendenze di qualità sul traffico di produzione. Buona scelta se vuoi osservabilità ed evals nello stesso strumento piuttosto che un harness eval separato.

**[Evalite](https://www.evalite.dev/)** — Framework eval nativo TypeScript di Matt Pocock. Cerimonia bassa: definisci un compito, definisci un valutatore, eseguilo nel tuo setup di test esistente. Si rivolge ai team che vogliono evals che sembrano unit test piuttosto che una piattaforma di esperimento ML separata.

**[promptfoo](https://www.promptfoo.dev/)** — Esecutore eval CLI-first focalizzato sul confronto del prompt e sul red-teaming. Facile da configurare via YAML, si integra con la maggior parte dei provider di modelli e ha supporto incorporato per rilevare l'iniezione del prompt e altri input avversariali.

**[deepeval](https://docs.confident-ai.com/)** — Framework eval Python con una grande libreria di metriche incorporate (G-Eval, fedeltà RAG, rilevanza della risposta, rilevamento di allucinazione). Utile per i pipeline RAG dove vuoi un grading specifico per la qualità del recupero, non solo la qualità della generazione.

Il giusto strumento dipende dal tuo stack e da dove stai partendo. Ciò che conta più della scelta del framework è la disciplina di eseguire gli evals a tutti i costi — consistentemente, su ogni cambiamento significativo.
