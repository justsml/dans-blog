新しいモデルが登場するたび、ベンチマークのタキシードを身にまとって現れる。

MMLU: 92.4%。HumanEval: 87.2%。LLeMU: 88.7%。MATH: 73.6%。AGI: 127%！

しかし、AIを使ってプロセスやプロダクトを構築する企業の99%にとって、**どれも意味がない。**

何が重要か？あなたのワークロードはどうなっているか？良くなっているか、悪くなっているか？それを知る唯一まともな方法は、あなたのシステム固有のタスク、データ、失敗パターンを反映したEvals（LLM用のテスト）を書くことだ。

  <p>ベンチマークは嘘をついていない。ただ、別人の質問に答えているだけだ。</p>
</blockquote>

## 「勘ベースの評価」がもたらす現実のコスト

一般的なアプローチ：モデルを変更してデプロイし、苦情チャンネルを監視し、騒ぎが大きくなったらロールバックする。

これでは、ほぼすべての興味深い問題を見逃す：

**目立つ失敗しか捉えられない。** 自信満々に間違った回答を受け取り、それに気づかないユーザー？沈黙。より悪い回答を受け取り、機能を使わなくなったユーザー？沈黙。サポートチケットやエラー率は、品質の低下のごく一部しか捉えられない。

**リグレッションと改善を区別できない。** 新しいモデルがタスクAでは優れているが、タスクBでは劣っている場合、Bに関する不満は「AIが悪くなった」という一般的なフィードバックと区別がつかない。何を直せばいいかわからない。

**ユーザーをテストインフラとして使っている。** 彼らはそれに同意していない。

## Evalのスペクトラム（と、多くのチームが間違える場所）

評価アプローチは「速いが脆弱」から「高価だが有効」までのスペクトラムに位置する。

<figure class="breakout">

![決定論的チェック、LLM-as-judge、人間の評価を速度・コスト・有効性で比較したスペクトラム図。](../eval-spectrum.svg)

<figcaption>誠実に失敗を捉えられる最も安い評価方法を使え。</figcaption>
</figure>

**LLM-as-judge** は現在の寵児だ。強力なモデルに別のモデルの出力を採点させる。速い、スケーラブル、安い。問題は、採点モデルのバイアスが焼き込まれ、ゲーム化されやすく、循環依存を生むことだ。GPT-5でGPT-5の出力を採点すれば、「GPT-5がGPT-5にどれだけ同意するか」のようなものを測定していることになる。それはゼロではないが、あなたが思っているものではない。

**人間の評価** は誰もが省略しようとするゴールドスタンダードだ。人間に output を評価させるのは、高価で遅く、評価者間でばらつきがあり、スケジューリングが面倒だ。しかし、あなたのシステムが実在の人間にとって有用かどうかを検証する唯一の方法だ。

**タスク固有の自動チェック** は、多くのチームがもっと時間をかけるべき領域だ。派手ではないが、速く、決定論的で、あなたのシステムにとって重要なことに直結している。

## 実際に効果があること

### 1. 失敗を定義してからデプロイする

モデルやプロンプトを変更する前に、悪い出力が何かを具体的に書き下す。

「出力は正確であるべき」とはテストにならない。もっとこうだ：

- 構造化されたJSON出力はエラーなくパースされなければならない
- 回答中のすべての引用が、検索されたコンテキストに原文のまま存在しなければならない
- 回答に競合製品名を言及してはならない
- SQLクエリは構文的に有効であり、スキーマに存在するテーブルのみを参照しなければならない
- 感情分類が既存のテストセットでポジティブからネガティブに3%以上反転してはならない

これらはプログラムでチェックできる。判定モデルは不要だ。

**Evalハーネス：決定論的チェック**

```typescript
type EvalResult = { passed: boolean; reason?: string };

const evals: Record<string, (output: string, context: EvalContext) => EvalResult> = {
  // JSON must parse
  validJson: (output) => {
    try {
      JSON.parse(output);
      return { passed: true };
    } catch (e) {
      return { passed: false, reason: `Invalid JSON: ${e.message}` };
    }
  },

  // No hallucinated citations — every claim must appear in context
  groundedCitations: (output, { retrievedChunks }) => {
    const claims = extractCitations(output);
    const ungrounded = claims.filter(
      (claim) => !retrievedChunks.some((chunk) => chunk.includes(claim))
    );
    return ungrounded.length === 0
      ? { passed: true }
      : { passed: false, reason: `Ungrounded claims: ${ungrounded.join(', ')}` };
  },

  // Response length sanity check — catch truncation or runaway generation
  reasonableLength: (output) => {
    const words = output.split(/\s+/).length;
    return words >= 10 && words <= 2000
      ? { passed: true }
      : { passed: false, reason: `Word count ${words} out of bounds` };
  },
};
```

### 2. 最悪の日々からゴールデンセットを構築する

最高の評価データは、恥ずかしいものだ。誰かがチケットを切るきっかけになった出力、ハルシネーションをスクリーンショットしたもの、誰かがこっそり機能を使わなくなった出来事。

ユーザーが悪い出力を報告するたび、ハルシネーションをフラグするたび、手動で失敗に気づくたびに、それをゴールデンセットに追加する：入力、コンテキスト、正しい振る舞いを記録する。50〜100ケースを維持し、すべてのモデル変更で実行する。

最初は手作業に感じる。半年後、あなたはパブリックベンチマークではゲーム化できないテストスイートを持っている。なぜならすべてのケースがあなた自身の失敗履歴から来ているからだ。

**ゴールデンケースの構造**

```typescript
interface GoldenCase {
  id: string;
  input: string;
  context: Record<string, unknown>;
  expectedBehavior: {
    mustContain?: string[];
    mustNotContain?: string[];
    structureCheck?: (output: string) => boolean;
    minSimilarityToReference?: number; // cosine similarity to a reference answer
  };
  sourceIncident?: string; // link back to the bug report or ticket
}
```

### 3. 受入テストだけでなく、リグレッションテストも

多くのチームはモデル変更を検討する時だけevalを実行する。それは受入テストだ：「この新しいものは十分良いか？」

リグレッションテストも必要だ：「以前動いていたものが壊れていないか？」

ゴールデンセットはモデル変更だけでなく、すべてのプロンプト変更で実行する。問題なく動いていたプロンプトも、新しいツールを追加したり、RAG検索戦略を変更したり、コンテキストテンプレートを更新したりすると、気づかないうちに劣化する。ベースラインがなければ気づけない。[Langfuse](https://langfuse.com/) のようなツールは、evalスコアを本番のトレースに紐付けるので、リグレッションがインシデントレポートだけでなくダッシュボードに表示される。

<details>
<summary>Evalハーネス：ベースラインと候補の比較</summary>

```typescript
async function compareModelVersions(
  goldenCases: GoldenCase[],
  baselinePipeline: Pipeline,
  candidatePipeline: Pipeline
) {
  const results = await Promise.all(
    goldenCases.map(async (tc) => {
      const [baseline, candidate] = await Promise.all([
        baselinePipeline.run(tc.input, tc.context),
        candidatePipeline.run(tc.input, tc.context),
      ]);

      return {
        id: tc.id,
        baselinePassed: runEvals(baseline, tc.expectedBehavior),
        candidatePassed: runEvals(candidate, tc.expectedBehavior),
        regression: /* baseline passed */ && /* candidate failed */,
        improvement: /* baseline failed */ && /* candidate passed */,
      };
    })
  );

  const regressions = results.filter((r) => r.regression);
  const improvements = results.filter((r) => r.improvement);

  console.log(`Regressions: ${regressions.length} / ${goldenCases.length}`);
  console.log(`Improvements: ${improvements.length} / ${goldenCases.length}`);

  if (regressions.length > 0) {
    console.error('Blocking regressions found:');
    regressions.forEach((r) => console.error(` - ${r.id}`));
  }

  return { regressions, improvements };
}
```

</details>

候補が既知の失敗ケースでリグレッションを起こした場合、アップグレードの議論は具体的になる：どのケースが改善し、どのケースが壊れ、そのトレードオフが価値があるかどうかが明確になる。

### 4. LLM-as-judgeは一つのことだけに使え

LLM-as-judgeは、決定論的な正解がないオープンエンドな出力に有用だ：「この回答は役に立つか？」「この要約は重要なポイントを保持しているか？」「この説明は初心者にとって適切か？」

そこで使え。決定論的な答えには使うな。使うときは、採点ルーブリックを明示的にせよ：

**Evalハーネス：ルーブリックベースの判定**

```typescript
async function judgeHelpfulness(
  userQuery: string,
  modelResponse: string
): Promise<{ score: number; reasoning: string }> {
  const judgePrompt = `
You are evaluating a customer support response.

User question: ${userQuery}
Response: ${modelResponse}

Rate the response on a scale of 1-5:
5 = Directly answers the question with accurate, actionable information
4 = Answers the question but could be more specific or actionable
3 = Partially addresses the question; key information is missing
2 = Tangentially related but doesn't answer the question
1 = Off-topic, factually wrong, or harmful

Respond with JSON: {"score": <number>, "reasoning": "<one sentence>"}
`;

  const result = await judgeModel.generate(judgePrompt);
  return JSON.parse(result);
}
```

明示的なルーブリックは評価者のばらつきを減らし、解釈可能な出力を提供し、判定が間違っているときに監査しやすくする。[Autoevals](https://github.com/braintrustdata/autoevals) や [Braintrust](https://www.braintrust.dev/) のようなライブラリは、一般的なタスク向けのビルトインルーブリックを提供している。ゼロから書く前に盗む価値がある。

## 知っておくべきツール

これらすべてをゼロから構築する必要はない。いくつかのツールがevalインフラの問題で大きな進歩を遂げている：

**[Braintrust](https://www.braintrust.dev/)** — 実験トラッキング、データセット管理、スコアリング関数を備えた包括的なevalプラットフォーム。eval実行をプロンプト、モデル、デプロイメントごとに整理するので、リリース間だけでなく、時間経過に伴う品質の変化をdiffできる。オープンソースの **[Autoevals](https://github.com/braintrustdata/autoevals)** ライブラリとよく組み合わせて使われる。こちらは事実の正確性、有用性、有害性、意味的類似性などの一般的なタスク向けにビルトインのモデル採点関数を提供している。

**[Langfuse](https://langfuse.com/)** — アプリとモデルの間に配置するオープンソースのLLMオブザーバビリティツール。すべての呼び出しをトレースし、evalスコア（人間または自動）を個々のスパンに紐付け、本番トラフィック上の品質トレンドを可視化する。オブザーバビリティとevalを別々のツールではなく同じツールで扱いたい場合に良い選択。

**[Evalite](https://www.evalite.dev/)** — Matt PocockによるTypeScriptネイティブのevalフレームワーク。低オーバーヘッド：タスクを定義し、スコアラーを定義し、既存のテストセットアップで実行する。evalを別のML実験プラットフォームではなく、ユニットテストのように扱いたいチームをターゲットにしている。

**[promptfoo](https://www.promptfoo.dev/)** — プロンプト比較とレッドチーミングに特化したCLIファーストのevalランナー。YAMLで簡単に設定でき、ほとんどのモデルプロバイダーと統合でき、プロンプトインジェクションやその他の攻撃的入力の検出をビルトインでサポートしている。

**[deepeval](https://docs.confident-ai.com/)** — 大規模なビルトインメトリックライブラリ（G-Eval、RAG忠実性、回答関連性、ハルシネーション検出）を備えたPython evalフレームワーク。生成品質だけでなく、検索品質の具体的な採点が欲しいRAGパイプラインで有用。

適切なツールはスタックと出発点に依存する。フレームワークの選択よりも重要なのは、evalを一貫して、すべての重要な変更で実行するという規律だ。

## 居心地の悪い部分

多くのチームがこれをスキップする理由は、早い段階で厄介な問いを投げかけられるからだ：「ここでの『良い』とは何か？」

新しいAI機能にとって、それは本当に難しい。しかし、信頼性を気にするなら、それはオプションではない。信頼できるAIを提供するチームは、クリティカルなコードパスに対してやるのと同じことをしている：期待される振る舞いを定義し、テストし、それらのテストを継続的に実行する。

ベンチマークは嘘をついていない。ただ、別人の質問に答えているだけだ。それらをプロダクトロードマップとして読むのをやめ、あなたのシステムに合ったテストを書き始めよう。

ダッシュボードより先にユーザーが気づく。テストスイートを先に構築しろ。
