私が見た中で最も高額なデータベース最適化は、すべての外部キーを削除することから始まった。

ボトルネックを測定したからではない。書き込みが実際に遅かったからでもない。「外部キーはスケールしない」とどこかで読んだからだ。6ヶ月後、20億件の孤立レコード、削除されたユーザーへの請求を続ける課金システム、そして40%ずれた分析データが残っていた。

制約を追加し直そうとしたら？既に破損しているデータを検証しようとして、データベースは停止した。

Web開発には、外部キーは本質的に遅い、という蔓延した考えがある。「本当の」システムに卒業したら外す補助輪みたいなものだという見方だ。でも、それは制約の目的を完全に見落としている。速いか遅いかを選んでいるのではない。異なる障害モードを選んでいるのだ。

考えてみてほしい。安全ガラス、シートベルト、エアバッグはすべて車の重量を増やす。間違いなく車は遅くなり、燃費も悪くなる。でも、0-60のタイムを最適化するためにそれらを引き剥がす人はいない。なぜなら、全く別のものを最適化しているからだ。

問うべきは「外部キーが遅くするかどうか」ではない。当然遅くする。問うべきは、その見返りに何が得られるか、そして自分にそれが本当に必要かどうかだ。

## 本当にトレードオフしているもの

具体的な例を挙げよう。気象観測システムを構築していて、観測所、センサーデバイス、センサー読み取り値、米国の州のテーブルがある。

すべてを外部キーで繋ぐべきか？何が実際に変わり、結果は何かを考えてみよう。

米国の州はおそらく変わらない。ワイオミング州がすぐに改名されることはない。参照データが静的だと分かっている場合、挿入のたびに州コードを検証する外部キーは不要なオーバーヘッドだ。

観測所は追加、移動、廃止される。しかし、ここで一つの問いがある。誰かが誤って観測所レコードを削除した場合、過去の読み取り値が観測所を「見失う」ことを望むか？おそらく、観測所がなくなってもそのデータは残ってほしいはずだ。それは読み取り値をライブ参照ではなく歴史的スナップショットとして扱っていることを意味し、外部キーがそもそも意味をなすかどうかが変わってくる。

センサー読み取り値は毎分数千回挿入されている。外部キーのチェックはすべて検索を意味し、検索はテーブルの競合を生む。検証が遅いことによって挿入キューが溢れ、リアルタイムデータを失うなら、それは孤立レコードとは別の種類のデータ損失だ。

見えてきたか。選択はパフォーマンス対正確性という抽象的な概念の対立ではない。実際の制約と実際の結果を前提に、どちらの特定の障害をより許容できるかの選択だ。

誤った参照が破損した請求データや規制違反を意味するなら、パフォーマンスのコストに関わらず外部キーで守るべきだ。遅い検証がキューの溢れによってリアルタイムのセンサーデータを永遠に失うなら、検証は間違ったトレードオフかもしれない。

## 書き込み速度が本当に重要な時

最大の書き込み速度が必要だと決めたとしよう。キューが積み上がり、トランザクションがタイムアウトし、外部キーのチェックが実際に（理論ではなく）測定可能な問題を引き起こしている。

いくつかの選択肢がある。トランザクション分離レベルを`SERIALIZABLE`から`READ COMMITTED`に変更するのは速いが、一部の整合性保証を手放すことになる。コミットをバッチ化して1回のトランザクションで1000行を挿入し、FKのオーバーヘッドを償却することもできる。あるいは、参照の検証すら試みない追記専用のログ構造に非正規化することもできる。

3つ目の選択肢はズルではない。単に別の設計だ。

```sql
CREATE TABLE sensor_log (
  id BIGSERIAL PRIMARY KEY,
  recorded_at TIMESTAMPTZ NOT NULL,
  data JSONB NOT NULL  -- { station_id, sensor_id, temp, humidity, ... }
);

CREATE INDEX ON sensor_log USING GIN (data);
CREATE INDEX ON sensor_log (recorded_at);
```

ジョインなし。外部キーのチェックなし。データを追記して、時間範囲またはJSONBブロブのGINインデックスでクエリする。これは「ベストプラクティス」か？データベースの教科書が教える意味ではおそらく違う。Raspberry Piで毎分50,000行を挿入する場合に動くか？確かに動く。

断絶が生じるのは、人が「ベストプラクティス」を道徳的命令として扱い、一般的なシナリオでうまく機能するパターンとしてではなく、自分の状況に合わないかもしれないものとして扱わない時だ。

## 正規化の罠

データベースの授業は正規化を教えるのが大好きだ。重複をどんな犠牲を払っても避けろ。第三正規形でなければ終わりだ。

だから、こういうものになる：`Orders` → `OrderItems` → `Products` → `Variants` → `Colors` → `Sizes`

「去年のクリスマスに赤いシャツと青いシャツのどちらを注文したか？」に答えるためだけに6テーブルのジョイン。商品名を含めようものなら、カタログ階層のさらに3つ先のジョインが必要だ。

でも待って。その正当化は通常「ブランドがBlueのラベルを変更したらどうする？」だ。それが起きた場合、過去の注文の色を遡って変更したいか？当然違う。誰かがその注文を行った時、その時点で存在していた「青いTシャツ、Mサイズ」として購入したのであって、後で更新されるかもしれないカタログエントリへの抽象的な参照としてではない。

これは重要なので深く考えたい。データの中には根本的にスナップショットであって参照ではないものがある。スナップショットデータをライブ参照であるかのように扱うと、書き込み時に非正規化しておけばよかったものを再構築するための、この不条理なジョインの増殖に陥る。

`{"color": "blue", "size": "M"}`を注文に直接格納すればいい。それで終わりだ。

### スナップショットデータの見分け方

何がスナップショットであるべきかをどう見分けるか？それが特定時点の記録かどうかを自問する。

注文は購入時点の商品詳細をキャプチャする。監査ログはユーザーがアクションを実行した時のユーザー状態を記録する。履歴テーブルは更新前のレコード状態を保存する。イベントストリームは何が、いつ、どんなデータで起きたかを記録する。

答えが「はい、これは特定時点の記録だ」なら、正規化を止めろ。スナップショットを始めろ。

### 不透明なブロブ

スナップショットを超えたもう一つのカテゴリーがある。内部をクエリしないデータだ。丸ごと格納して丸ごと取得する。

`{"model": "gpt-4", "temperature": 0.7, "max_tokens": 2000}`のようなLLMモデル設定は、temperatureでクエリするものではない。必要な時にリクエストIDで設定全体を取得する。デコード後のJWTペイロード、デバッグ用のAPIリクエスト/レスポンスログ、テーマ設定や通知フラグを含むユーザー設定オブジェクト。これらはすべて不透明なブロブだ。正規化はいらない。外部キーもいらない。JSONBに押し込んで次に進め。

注文されたシャツの色を知るための6テーブルジョイン？それは適切な正規化ではない。参照と値のどちらを格納しているかについての混乱だ。

（ただし注意：これは後からそのデータをクエリする必要が出た時に猛烈に逆火する可能性がある。このアプローチが独自の悪夢を生み出す時については[JSONBの誘惑](/the-jsonb-seduction)を参照。）

## スケールとは文脈である

「外部キーはスケールしない」と言う人がいる。しかしスケールはハードウェアとアーキテクチャに対して完全に相対的だ。

Raspberry Piで毎分10,000件のセンサー読み取りをmicroSDカードに記録している？そのハードウェアでは正当に高スケールだ。プロビジョンドIOPSのAWS Auroraで数十億行を処理している？外部キーを使っても息切れしない。

実際のハードリミットは行数や書き込み量についてではない。シャーディングだ。

`Users`テーブルがサーバーAに、`Orders`テーブルがサーバーBにある場合、外部キーは物理的に機能しない。データベースにはネットワーク境界を越えて制約を強制する仕組みがない。その時点で、孤立レコードを見つけるバックグラウンドジョブや結果的整合性パターンを既に実装していることになる。

これは、コンプライアンス要件で各テナントが独自の分離データベースを持つマルチテナントSaaSや、50,000台のエッジデバイスがそれぞれローカルでSQLiteを実行しているIoTデプロイメントで起こる。そこに到達すると、パフォーマンスの考慮に関わらず外部キーは論理的に不可能だ。

でも、そのアーキテクチャの境界に到達するまで、10ユーザーの内部ツールを構築しているのにNetflixの問題を前提に最適化するのはやめよう。

## 実際にはどう見えるか

「外部キーを使うべきか」の代わりに、これら3つを問ってみよう：

この参照が間違っていたら何が壊れるか？訴訟か、破損した請求か、規制違反か？それとも分析ダッシュボードでnullを返す見逃し可能なジョインか？

検証が遅かったら何が壊れるか？取り戻せないリアルタイムデータを失うか？それともクエリが50ミリ秒余分にかかるだけか？

このデータはスナップショットか参照か？特定の時点での見た目を記録しているのか、それとも権威ある現在値へのポインタか？

そこから、パターンは自然に現れる：

金融取引、認証セッション、データ破損が法的責任を意味するものは、パフォーマンスのオーバーヘッドに関わらず外部キーが欲しいだろう。

大量のログ、追記専用の時系列データ、毎分100万イベントを書き込むものは、各書き込みに検証のオーバーヘッドをかける必要がないだろう。

注文や監査ログのような歴史的スナップショット、ユーザー設定のように常に完全なブロブとして取得するデータ、外部APIからのWebhookペイロードのような自分で制御できないスキーマ……これらは非正規化の方がうまく機能することが多い。

ただし、「おそらく」や「ことが多い」と言ったことに注意してほしい。文脈が重要で、あなたの文脈は私の文脈とは違うからだ。

## 最後に

外部キーはパフォーマンスの問題ではない。書き込み速度とデータ整合性のトレードオフだ。そのトレードオフが意味をなすかどうかは、あなたの特定のボトルネックと特定の結果に完全に依存する。

本当の問題は、「Webスケール」について読んだことを理由に、実際に書き込みパフォーマンスの問題があるか測定せず、何を手放しているかも考慮せずに外部キーを削除する人たちだ。1日100トランザクションを処理する新規プロジェクトにNetflixのアーキテクチャをカーゴカルト的に適用してしまう。

パフォーマンスのコストはあなたのユースケースにとって価値があるかもしれない。そうではないかもしれない。でも、その決定を、最適化していると思っているものではなく、実際に最適化したいものに基づいて行ってほしい。

あなたは何を最適化しているのか？
