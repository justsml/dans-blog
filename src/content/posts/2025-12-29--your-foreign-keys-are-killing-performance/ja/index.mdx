私が見た中で最も高額なデータベース最適化は、すべての外部キーを削除することから始まった。

計測してボトルネックを特定したわけでも、書き込みが実際に遅かったわけでもない。「外部キーはスケールしない」とどこかで読んだからだ。6ヶ月後、彼らは20億件の孤立レコードを持ち、削除されたユーザーに請求する請求システムを抱え、分析が40%も狂っている状態だった。

制約を元に戻そうとした時？データベースは、既に破損していた既存のデータを検証しようとして完全に停止した。

Web開発の世界には、外部キーは本質的に遅い 것이라는根強い考えがある。外部キーというのは「本当の」システムになれば外す練習用のwheelのようなものだ。しかしそれは、制約が何のためにあるのかという点を完全に误解している。高速か低速かを選んでいるのではない。異なる失敗モードの間で選んでいるのだ。

こう考えてみてほしい：安全ガラス、シートベルト airbag はすべて車に重さを加える。確かに車両を遅くし、燃費を悪くする。でも0-60タイムを最適化するためにそれらを引き裂く人はいない。なぜなら、別のものを最適化しているからだ。

問題は外部キーが足を引っ張るかどうかではない。もちろんだ。問題は代わりに何を得られるかで、そしてそれが実際に必要かどうかだ。

## 本当にトレードオフ的是什么

具体的な例を出そう。天気監視システムを作っていて、気象観測所、センサーデバイス、センサー測定値、米国の州のテーブルがある。

すべて外部キーで繋ぐべきか？実際に何が変わりうるか、后果は何かを考えてみよう：

米国の州はおそらく変わらない。ワイオミングがすぐに名前が変わることはない。参照データが静的なのが分かっている時に、挿入のたびに州コードを検証するために外部キーが必要だということはない。無駄なオーバーヘッドだ。

気象観測소는追加、移动、廃止される。だがここで вопрос：過去の測定値が、誰かが誤って観測所レコードを削除した場合に「観測所を失う」ことを望みますか？観測소가消えても過去のデータはそのまま残したいのかもしれません。そうであれば、測定値を履歴スナップショットとして扱っており、外部キーすら適切ではない可能性がある。

センサー測定値は毎分何千回も挿入されている。外部キー檢查마다ルックアップが発生する。すべてのルックアップがテーブル上で競合を作る。遅い検証がキュー溢れを招き、リアルタイムデータを失うなら、それは孤立レコードがある更能にデータ loss とは違う種類のデータ loss だ。

ここから先は分かるだろう。選択はパフォーマンス対正確性という抽象概念の話ではない。特定の失敗哪一种更能容忍 given あなたの actual 制約と actual 結果だ。

误った参照が請求データの破損や規制違反意味着するなら、パフォーマンスコストに関係なく、おそらく外部キー want すべきだ。遅い検証がリアルタイムセンサーデータを失うことを意味するなら（キュー溢れで永遠に失う）、検証は間違ったトレードオフかもしれない。

## 高速書き込みが本当に重要な時

では、最大書き込み速度が必要だと決めたわけだ。キューが積み上がり、トランザクションがタイムアウトし、外部キー檢查が実際に計測した問題（単に理論化したものではなく！）を引き起こしている。

いくつかのオプションがある。トランザクション分離レベルを `SERIALIZABLE` から `READ COMMITTED` に変更できるmdash;更快だが一部の整合性保証を捨てる。或者、バッチコミットを変えて、1つずつではなく1000行ずつトランザクションで挿入して外部キーオーバーヘッドを amortize できる。或者、非正規化して-app-only log 構造体に変えて、参照を検証すら試みない。

3番目のオプションはチートじゃない。これはただの別デザインだ：

```sql
CREATE TABLE sensor_log (
  id BIGSERIAL PRIMARY KEY,
  recorded_at TIMESTAMPTZ NOT NULL,
  data JSONB NOT NULL  -- { station_id, sensor_id, temp, humidity, ... }
);

CREATE INDEX ON sensor_log USING GIN (data);
CREATE INDEX ON sensor_log (recorded_at);
```

結合なし。外部キー檢查なし。ただデータを追加して、時間範囲またはJSONB blob 上でGINインデックスでクエリする。 これは「ベストプラクティス」か？データベース教科量が教える意味では、おそらく違う。Rasberry Pi で毎分50,000行を挿入している時に動作するか？もちろん動作する。

人々が「ベストプラクティス」を道徳的命令としてではなく、一般的なシナリオでうまくいくパターンとして扱い、それがあなたのシナリオに当てはまらないかもしれないものとして扱う時に起きる食い違いがある。

## 正規化の罠

データベースコース正規化を教えるのが好きだ。重複をあらゆるコストで避けろ。第3正規形かそれか。

そのため这样的人会成为：`Orders` → `OrderItems` → `Products` → `Variants` → `Colors` → `Sizes`

「去年のクリスマスに赤いシャツと青いシャツどちらを注文した？」答えるのに6テーブル結合が必要なのだ。さらに商品名は必要なら、カタログ階層からさらに3つ結合遠くだ。

しかし待っていただければ。 正当性は通常「ブランドがBlueのラベル付け方を変更したら怎么办？」だ。それが 起ったら、本当に過去の注文が遡及的に色改变你想要吗？当然ない。誰かが注文した時、彼らは「 момента времени に存在していた Mサイズの青いTシャツ」を買ったのであって、後で更新されるかもしれないカタログエントリへの抽象的な参照ではない。

これは因為に払う価値があるmdash;それは subtile からだ。あるデータは本質的にスナップショットであり、参照ではない。スナップショットデータをライブ参照であるかのように扱った場合、書き込み時に単に非正規化すべきだったものを再構築するために、このばかげた結合の増加に陥る。

`{"color": "blue", "size": "M"}` を注文に直接 저장하라。終わりだ。

### スナップショットデータの認識

何かがスナップショットであるべき時をどう知るか？自分に問いかけてみて whether これは point-in-time record か：

注文は購入時の商品詳細をキャプチャする。監査ログはアクション実行時のユーザー状態を記録する。 history テーブルは更新前のレコード状態を保存する。イベントストリームは何が起きたか、いつ起きたか、どのようなデータで起きたかをキャプチャする。

答えが「はい、これは瞬間を記録しています」なら、正規化を止めてください。スナップショット化し始めましょう。

### 非透過ブロブ

スナップショット以外にもう1つのカテゴリがある：クエリ никогда しないデータ。ただ存储して全体 retrieved する。

LLMモデル設定 like `{"model": "gpt-4", "temperature": 0.7, "max_tokens": 2000}` は温度でクエリするものではない。必要時リクエストIDで 전체設定を取得する。デコード後のJWTペイロード、デバッグ用のAPIリクエスト/レスポンスログ、テーマ設定と通知フラグを含むユーザー preference オブジェクト。これらはすべて非透過ブロブだ。正規化は不要。外部キー也不要。JSONBに詰め込んで先に進もう。

6テーブル結合してどの色のシャツが注文されたか調べる？それは適切な正規化ではない。それは参照と値のどちらを保存しているかの混乱した思考だ。

（ただし注意：後でそのデータをクエリする必要がある場合、 spectacularly 裏目に出る可能性がある。[The JSONB Seduction](/the-jsonb-seduction) 参照mdash;このアプローチが本身就 nightmare を作成するタイミングについて。）

## スケールはコンテキストだ

「外部キーはスケールしない」と聞いたことがあるだろう。しかしスケールはハードウェアとアーキテクチャ完全に relative だ。

microSDカードに毎分10,000件のセンサー測定値を記録するRaspberry Pi？それは確かにそのハードウェアにとっては高音量のスケールだ。プロビジョンドIOPSを持つAWS Auroraが数十億行を処理？ внешниеキー で通過 可以 without breaking a sweat。

本当の hard limit は行数や書き込みボリューム关于ではない。シャーディング关于だ。

`Users` テーブルがサーバーAにあり、`Orders` テーブルがサーバーBにある場合、 внешниеキー physically 实现できない。データベースにはネットワーク境界全体で制約を強制するメカニズムがない。その時点で、あなたは既に孤児を見つけるバックグラウンドジョブを実行し、结果整合性パターンを実装している。

これはコンプライアンス 이유로各テナントが分離されたデータベースを持つマルチテナントSaaS、または50,000個のエッジデバイス每个がローカルでSQLiteを実行するIoT展開で发生する。一旦そこに到达したら、パフォーマンス考量 Whether を待たずに внешниеキー はテーブルから除外される（文字通り）。

しかしそのアーキテクチャ境界に到达するまでは、10ユーザーの内部ツールを構築しているときにNetflixの問題のために prematurely 最適化하지 마라。

## 実際にこれは何看起来像

「外部キーを使用するべきか」ではなく、この3つのことを問いかけてみましょう：

この参照が間違っていると何が壊れますか？訴訟、請求データの破損、規制違反？それとも分析ダッシュボードでnullを返すだけの欠落結合？

検証が遅いと何が壊れますか？取り返しのつかないリアルタイムデータを失いますか？それともクエリがただ50ミリ秒余分にかかるだけですか？

このデータはスナップショット还是参照ですか？ある瞬間の样子を記録していますか？それとも权威的な current 値を指していますか？

そこから、パターンは pretty naturally 浮现する：

データ破損が法的責任を意味する金融トランザクション、認証セッションmdash;これらはパフォーマンスオーバーヘッドに関係なく外部キーをご希望だろう。

毎分100万イベントを書く高周波ログ、app-only 時系列データmdash;これらは書くたびに検証オーバーヘッドを必要としない可能性がある。

注文や監査ログのような履歴スナップショット、完全なブロブとして常にフェッチするデータ like ユーザーpreference、webhook ペイロードのような外部APIから制御しないスキーマ…これらはしばしば非正規化が会更好だ。

しかし「、おそらく」と「、しばしば」を言ったことに注意到了。コンテキストが重要で、あなたのコンテキストは私のと違う。

## 最終所想

外部キーはパフォーマンス問題ではない。書き込み速度とデータ整合性の間のトレードオフであり、そのトレードオフが意味があるかどうかは、あなたの specific ボトルネックと specific 結果完全に depend する。

本当の問題は、人々が「Webスケール」について読んだからという理由で外部キーを削除することですndash;実際に書き込みパフォーマンスの問題があるか、または何をあきらめているかを考慮せずに。 greenfield プロジェクトで 每秒100トランザクションを処理するものにNetflixのアーキテクチャを貨物 культ することになる。

パフォーマンスコストがあなたのユースケース的价值があるかもしれない。あるいは价值がないかもしれない。しかし少なくとも、あなたが 실제로最適化しているもののために决定を下して、それ，你觉得你应该优化 for のためにではない。

何を最適化していますか？
