私が今まで見た中で最も高価なデータベース最適化は、誰かがすべての外部キーを削除したことから始まった。

ボトルネックを計測したわけでもない。実際に書き込みが遅かったわけでもない。ただどこかで「外部キーはスケールしない」と読んだからだ。6か月後、彼らには20億の孤立レコード、削除済みユーザーに課金を続けるシステム、そして40%もずれた分析データがあった。

制約を元に戻そうとしたとき？ 既に破損した既存データの検証にデータベースが足を取られ、動きが止まってしまった。

ウェブ開発には、外部キーは本質的に遅いものであり、「本物」のシステムに到達したら外す補助輪だという蔓延した考えがある。しかし、それは制約の本質的な目的を見失っている。あなたは速さと遅さの間で選択しているのではない。異なる障害モードの間で選択しているのだ。

こう考えてみよう。防犯ガラス、シートベルト、エアバッグはすべて車に重さを加える。それらは間違いなく車両を遅くし、燃費を悪くする。しかし、0-60加速を最適化するためにそれらを取り外すことはない。なぜなら、あなたは全く別のものを最適化しているからだ。

問題は外部キーが遅くなるかどうかではない。もちろん遅くなる。問題はその見返りに何を得られるか、そして本当にそれが必要かどうかだ。

## あなたが本当に引き換えにしているもの

具体的な例を示そう。気象観測システムを構築していて、観測所、センサーデバイス、センサー読み値、そして米国の州を管理するテーブルがあるとする。

すべてを外部キーで結ぶのか？ 実際に何が変化し、その結果が何であるかを考えてみよう。

米国の州名はおそらく変わらない。ワイオミング州が改名されることは当面ない。参照データが静的であることがわかっているのに、毎回の挿入時に州コードを検証するために外部キーが必要なわけではない。それは無意味なオーバーヘッドだ。

気象観測所は追加、移動、廃止される。しかしここに問いがある：誰かが観測所レコードを誤って削除した場合、過去の読み値がその観測所を「失う」ことを望んでいるのだろうか？ おそらく、観測所がなくなったとしてもそのデータはそのまま残っていてほしいのかもしれない。つまり、読み値をライブ参照ではなく履歴スナップショットとして扱っていることになり、それは外部キーが意味を持つかどうかを変える。

センサー読み値は毎分何千回も挿入される。すべての外部キーチェックはルックアップを意味する。すべてのルックアップはテーブルへの競合を生む。遅い検証が挿入キューの滞留を招き、リアルタイムデータを失うことになるなら、それは孤立レコードを持つこととは異なる種類のデータ損失だ。

どこに向かっているかわかるだろう。選択はパフォーマンス対正確性という抽象的な概念の話ではない。それは、実際の制約と実際の結果を考慮した上で、どの具体的な障害をより許容できるかということだ。

間違った参照が破損した課金データや規制違反を意味するなら、パフォーマンスコストがどれほどかかろうと外部キーによる保護が欲しいだろう。遅い検証がキューが溢れてリアルタイムセンサーデータを永遠に失うことを意味するなら、検証は間違ったトレードオフかもしれない。

## 高速な書き込みが実際に重要な場合

最大限の書き込み速度が必要だと決めたとしよう。キューが蓄積し、トランザクションがタイムアウトし、外部キーチェックが実際に問題を引き起こしていて、それを実際に計測した（単に推測しただけではない）。

いくつかの選択肢がある。トランザクション分離レベルを`SERIALIZABLE`から`READ COMMITTED`に変更できる。これは高速だが、一貫性の保証をある程度犠牲にする。コミットをバッチ化でき、1行ずつではなく1000行ずつ挿入して外部キーのオーバーヘッドを償却できる。あるいは、参照を検証しようとしない追記専用のログ構造に非正規化することもできる。

3番目の選択肢はカンニングではない、ちなみに。それは単に異なる設計だ：

```sql
CREATE TABLE sensor_log (
  id BIGSERIAL PRIMARY KEY,
  recorded_at TIMESTAMPTZ NOT NULL,
  data JSONB NOT NULL  -- { station_id, sensor_id, temp, humidity, ... }
);

CREATE INDEX ON sensor_log USING GIN (data);
CREATE INDEX ON sensor_log (recorded_at);
```

JOINなし。外部キーチェックなし。データを追記して、時間範囲やJSONB BLOBのGINインデックスでクエリするだけだ。これは「ベストプラクティス」なのか？ データベースの教科書が教える意味では、おそらく違う。Raspberry Piで毎分5万行を挿入する場合に機能するのか？ 間違いなく。

分断が起きるのは、人々が「ベストプラクティス」を一般的なシナリオでうまく機能するパターンではあるが、自分の状況には合わないかもしれないという理解ではなく、道徳的な imperative として扱うときだ。

## 正規化の罠

データベースの講義では正規化を教えるのが好きだ。あらゆるコストを避けろ。第三正規形でなければならない。

その結果、こうなる：`Orders` → `OrderItems` → `Products` → `Variants` → `Colors` → `Sizes`

「去年のクリスマスに赤いシャツを注文したのか、それとも青いシャツだったのか？」という問いに答えるだけで6テーブルのJOINが必要だ。そして商品名も必要なら、カタログ階層でさらに3つのJOINが待っている。

しかし待ってほしい。正当化は通常「ブランドが『青』のラベル付けを変更したらどうするか？」というものだ。もしそれが起きたら、過去の注文が遡及的に色を変えることを本当に望んでいるのか？ もちろん違う。誰かが注文をした時、彼らは後でカタログエントリが更新されるかもしれない抽象的な参照ではなく、その時点で存在していた「青いTシャツ、サイズM」を買ったのだ。

これは微妙なのでじっくり考えてほしい。一部のデータは本質的にスナップショットであり、参照ではない。スナップショットデータをまるでライブ参照であるかのように扱うと、書き込み時に非正規化しておけばよかったものを再構成するために、_JOINの過剰な繁殖_に陥る。

注文に`{"color": "blue", "size": "M"}`を直接格納しよう。それで終わりだ。

### スナップショットデータの認識

いつスナップショットにすべきかをどうやって知るか？ それが時点記録かどうか自問してみよう：

注文は購入時に存在していた商品詳細を捉える。監査ログはユーザーがアクションを実行した時のユーザーステートを記録する。履歴テーブルは更新前のレコードの状態を保存する。イベントストリームは何が起きたか、いつ、どのデータで、を捉える。

答えが「はい、これは特定の瞬間を記録している」なら、正規化するのをやめよう。スナップショット化しよう。

### 不透明なBLOB

スナップショットの他にもう一つのカテゴリーがある：中身にクエリをかけず、まるごと格納して取り出すだけのデータだ。

`{"model": "gpt-4", "temperature": 0.7, "max_tokens": 2000}`のようなLLMモデル設定は、temperatureでクエリをかけるものではない。必要な時にリクエストIDで全体を取得する。デコード後のJWTペイロード、デバッグ用のAPIリクエスト/レスポンスログ、テーマ設定や通知フラグを含むユーザー設定オブジェクト。これらはすべて不透明なBLOBだ。正規化は不要だ。外部キーも必要ない。JSONBに放り込んで先に進もう。

注文されたシャツの色を知るために6テーブルJOINが必要なのは、適切な正規化ではない。参照を格納しているのか値を格納しているのか、という考えが混乱しているだけだ。

（ただし注意：後でそのデータにクエリをかける必要が出てきた場合、これは壮絶に裏目に出る。詳細は[The JSONB Seduction](/the-jsonb-seduction)を参照。）

## スケールとは文脈だ

「外部キーはスケールしない」と言う人がいるだろう。しかし、スケールはハードウェアとアーキテクチャに完全に相対的なものだ。

microSDカードに毎分1万センサー読み値を記録するRaspberry Pi？ そのハードウェアにとってそれは正当に高スケールだ。プロビジョンドIOPSで数十億行を処理するAWS Auroraなら？ 外部キーを気にせず使い続けても、平気で処理できる。

実際のハードな限界は行数や書き込み量の話ではない。シャーディングだ。

`Users`テーブルがサーバーAにあり、`Orders`テーブルがサーバーBにある時、外部キーは物理的に機能しない。データベースにはネットワーク境界を越えて制約を強制するメカニズムがない。その時点で、すでに孤立レコードを見つけるバックグラウンドジョブを実行し、結果整合性のパターンを実装していることだろう。

これは、マルチテナントSaaSでテナントごとにコンプライアンスのために分離されたデータベースを持つ場合や、5万台のエッジデバイスそれぞれがローカルでSQLiteを実行しているIoT展開で起きる。そこに到達したら、パフォーマンスの考慮に関係なく、外部キーは文字通りテーブルから消える。

しかし、そのアーキテクチャ的境界に到達するまで、10ユーザーの社内ツールを作っている時にNetflixの問題に先回りして最適化するのはやめよう。

## 実際に実践ではどう見えるか

「外部キーを使うべきか」という問いの代わりに、次の3つを問ってみよう：

この参照が間違っていたら何が壊れるか？ 訴訟、破損した課金データ、規制違反か？ それとも分析ダッシュボードでJOINが欠落してnullが返るだけか？

検証が遅いと何が壊れるか？ 取り返しのつかないリアルタイムデータを失うのか？ それともクエリがただ50ミリ秒長くなるだけか？

このデータはスナップショットか参照か？ 特定の瞬間の何かの様子を記録しているのか、それとも権威ある現在値を指しているのか？

そこから、パターンは自然に現れる：

金融取引、認証セッション、データ破損が法的責任を意味するものは、パフォーマンスオーバーヘッドがあっても外部キーが欲しいだろう。

高ボリュームのログ、追記専用の時系列データ、毎分100万イベントを書き込むものは、おそらくすべての書き込みでの検証オーバーヘッドは必要ない。

注文や監査ログのような履歴スナップショット、常に完全な塊として取得するデータ（ユーザー設定など）、外部APIからのWebhookペイロードのような自分で制御できないスキーマ……これらはしばしば非正規化の方がうまく機能する。

しかし「おそらく」や「しばしば」と言ったことに注目してほしい。文脈が問題で、あなたの文脈は私の文脈と異なるからだ。

## 最後に

外部キーはパフォーマンス問題ではない。書き込み速度とデータ整合性の間のトレードオフであり、そのトレードオフが意味を持つかどうかは、あなたの具体的なボトルネックと具体的な結果に完全に依存する。

本当の問題は、人々が「ウェブスケール」について読んだ何かのために外部キーを削除し、実際に書き込みパフォーマンスの問題があるかを計測せず、何を諦めるかを考慮せずに行うことだ。Netflixのアーキテクチャを、1日100トランザクションを処理するグリーンフィールドプロジェクトにカーゴカルトする結果になる。

おそらくパフォーマンスコストはあなたのユースケースにとって受け入れられるだろう。おそらく違うだろう。しかし、その決定は少なくとも、あなたが最適化すべきだと思っていることではなく、実際に最適化したいことに基づいて行おう。

あなたは何を最適化しているのか？
