---
title: "外部キー：速いかどうか聞くのはもうやめよう"
subTitle: "本当に最適化すべきものは何かを考えよう"
---

私が見てきた中で最も高くついたデータベースの最適化は、誰かがすべての外部キー（Foreign Keys）を削除したことから始まりました。

ボトルネックを測定したからではありません。書き込みが実際に遅かったからでもありません。どこかで「外部キーはスケールしない」という記事を読んだからです。半年後、そこには20億件の孤立したレコードがあり、削除されたユーザーに課金し続ける請求システムがあり、40%も数値が狂ったアナリティクスが残されていました。

彼らが制約を元に戻そうとしたとき、何が起きたか？ すでに破損してしまった既存データの整合性を検証しようとして、データベースは立ち往生してしまいました。

Web開発の世界には、外部キーは本質的に遅いものであり、「本物の」システムへとステップアップする際には取り外すべき補助輪のようなものだ、という考えが蔓延しています。しかし、それは制約（Constraint）の目的を完全に見失っています。あなたが選んでいるのは「速いか遅いか」ではありません。「どの失敗モード（Failure Mode）を許容するか」なのです。

こう考えてみてください。強化ガラス、シートベルト、エアバッグはすべて車を重くします。それらは間違いなく加速を鈍らせ、燃費を悪くします。しかし、0-60マイル加速を最適化するためにそれらを剥ぎ取ることはしません。なぜなら、あなたは全く別のものを最適化しているからです。

問題は「外部キーが速度を低下させるか」ではありません。当然、低下させます。重要なのは、その代償として何を得るのか、そしてそれが本当に必要なのか、という点です。

## あなたが本当にトレードオフにしているもの

具体的な例を挙げましょう。気象観測所、センサーデバイス、センサーの読み取り値、および米国の州のテーブルを持つ気象監視システムを構築しているとします。

すべてを外部キーで繋ぎますか？ 何が実際に変化し、どのような結果を招くかを考えてみましょう。

**米国の州**はおそらく変わりません。ワイオミング州の名前が近いうちに変わることはないでしょう。静的な参照データであることが分かっているのに、挿入のたびに州コードを検証するために外部キーを使うのは、無意味なオーバーヘッドです。

**気象観測所**は追加されたり、移動したり、廃止されたりします。ここで質問です。誰かが誤って観測所の記録を削除したとき、過去の読み取り値がその観測所との「繋がり」を失うことを望みますか？ おそらく、観測所がなくなったとしても、そのデータはそのまま残しておきたいはずです。つまり、読み取り値は「ライブな参照」ではなく「履歴のスナップショット」として扱っていることになり、そもそも外部キーが適切かどうかが変わってきます。

**センサーの読み取り値**は、1分間に数千回も挿入されます。外部キーのチェックは、そのたびにルックアップが発生することを意味します。ルックアップが増えれば、テーブルへの競合（Contention）が発生します。もし検証が遅いために挿入キューが滞り、リアルタイムデータが失われるとしたら、それは「孤立したレコードが残る」ことよりも深刻なデータ損失かもしれません。

もうお分かりでしょう。選択すべきは「パフォーマンスか正確性か」という抽象的な概念ではありません。あなたの実際の制約と実際の影響を考慮した上で、「どの特定の失敗ならより許容できるか」なのです。

参照の誤りが請求データの破損や規制違反を意味する場合、パフォーマンスコストに関わらず、外部キーで保護したいと思うでしょう。もし検証の遅延が、キューのオーバーフローによってリアルタイムのセンサーデータを永遠に失うことを意味するなら、検証は間違ったトレードオフかもしれません。

## 書き込み速度が本当に重要なとき

さて、あなたは最大限の書き込み速度が必要だと判断しました。キューは積み上がり、トランザクションはタイムアウトし、外部キーのチェックが（理論上ではなく）実際に測定可能な問題を引き起こしています。

いくつかの選択肢があります。トランザクションの分離レベルを `SERIALIZABLE` から `READ COMMITTED` に変更すれば、整合性の保証と引き換えに速度が上がります。あるいは、1回に1行ずつではなく1000行ずつコミットするバッチ処理を行い、外部キーのオーバーヘッドを分散させることもできます。あるいは、参照を検証することすらしない、追記専用（Append-only）のログ構造に非正規化することもできます。

ちなみに、3番目の選択肢は「手抜き」ではありません。単なる異なる設計です。

```sql
CREATE TABLE sensor_log (
  id BIGSERIAL PRIMARY KEY,
  recorded_at TIMESTAMPTZ NOT NULL,
  data JSONB NOT NULL  -- { station_id, sensor_id, temp, humidity, ... }
);

CREATE INDEX ON sensor_log USING GIN (data);
CREATE INDEX ON sensor_log (recorded_at);
```

結合（Join）も外部キーチェックもありません。ただデータを追加し、時間範囲やJSONBのGINインデックスでクエリを投げるだけです。これは「ベストプラクティス」でしょうか？ データベースの教科書が教える意味では、おそらく違うでしょう。しかし、Raspberry Piで1分間に5万行の挿入を処理する場合には、間違いなく機能します。

「ベストプラティス」を、個別の状況に合わせたパターンではなく、道徳的な義務として扱ってしまうときに、この断絶は起こります。

## 正規化の罠

データベースの講義では正規化が好まれます。いかなる犠牲を払っても重複を避けよ。第三正規形（3NF）でなければならない、と。

その結果、このような構成になります： `Orders` → `OrderItems` → `Products` → `Variants` → `Colors` → `Sizes`

「去年のクリスマスに注文したのは赤いシャツだったか、青いシャツだったか？」という問いに答えるためだけに、6つのテーブルを結合しなければなりません。さらに製品名まで含めるとなると、カタログ階層をさらに3つ辿る必要があります。

しかし、ちょっと待ってください。その正当化の理由は通常、「ブランドが『ブルー』のラベルの付け方を変えたらどうするか？」というものです。もしそれが起きたとき、過去の注文まで遡って色が変更されてほしいでしょうか？ もちろんNOです。誰かがその注文をしたとき、彼らが買ったのは「その瞬間に存在した青いTシャツのMサイズ」であり、後で更新されるかもしれないカタログ項目の抽象的な参照ではありません。

これは非常に繊細で、よく考える価値のあるポイントです。データの中には、参照ではなく、根本的に**スナップショット**であるべきものが存在します。スナップショットデータをライブな参照として扱ってしまうと、書き込み時に非正規化しておけば済むはずのものを再構築するために、不条理なほど結合を繰り返すことになります。

注文データに `{"color": "blue", "size": "M"}` を直接保存してください。それで終わりです。

### スナップショットデータの見極め方

あるデータがスナップショットであるべきかどうかを、どう判断すればいいでしょうか？ それが「ある時点の記録（point-in-time record）」であるかどうかを自問してみてください。

注文は、購入時の製品詳細をキャプチャします。監査ログは、ユーザーがアクションを実行した時の状態を記録します。履歴テーブルは、更新前のレコードの状態を保存します。イベントストリームは、いつ、何が、どんなデータで起きたかを捉えます。

もし答えが「はい、これはある一瞬を記録するものです」であれば、正規化をやめてください。スナップショットとしての保存を開始してください。

### 不透明な塊（Opaque Blobs）

スナップショット以外にもう一つ、中身を細かくクエリすることのないデータのカテゴリーがあります。ただ保存し、丸ごと取り出すだけのデータです。

`{"model": "gpt-4", "temperature": 0.7, "max_tokens": 2000}` のようなLLMモデルの設定を、temperature（温度）でクエリすることはないでしょう。必要な時にリクエストIDで設定全体を取得するだけです。デコード後のJWTペイロード、デバッグ用のAPIリクエスト/レスポンスログ、テーマ設定や通知フラグを持つユーザー設定オブジェクト。これらはすべて「不透明な塊」です。正規化も外部キーも必要ありません。JSONBに放り込んで、先に進みましょう。

シャツの色を知るための6テーブル結合？ それは適切な正規化ではありません。参照を保存しているのか、値を保存しているのか、考えが混乱している証拠です。

（ただし、注意してください。後でそのデータをクエリする必要が出てくると、これは悲惨なしっぺ返しを食らわせます。このアプローチが独自の悪夢を生むケースについては、[JSONBの誘惑](/the-jsonb-seduction) を参照してください。）

## スケールとは文脈である

「外部キーはスケールしない」という言葉を耳にするでしょう。しかし、スケールとは、お使いのハードウェアとアーキテクチャに対して完全に相対的なものです。

microSDカードを刺したRaspberry Piで、1分間に1万件のセンサー読み取りを記録する？ それはそのハードウェアにとって、紛れもなく「ハイ・スケール」です。プロビジョンドIOPSを備えたAWS Auroraで数十億行を処理する？ 外部キーを使っても、汗一つかかずに処理できます。

実際のハードリミットは、行数や書き込み量ではなく、**シャーディング（Sharding）**にあります。

`Users` テーブルがサーバーAにあり、`Orders` テーブルがサーバーBにある場合、外部キーは物理的に機能しません。ネットワークの境界を越えて制約を強制するメカニズムを、データベースは持っていないからです。その時点では、すでに孤立したレコードを探すバックグラウンドジョブを実行したり、結果整合性（Eventual Consistency）のパターンを実装したりしているはずです。

これは、コンプライアンスのために各テナントに独立したデータベースを割り当てるマルチテナントSaaSや、5万台のエッジデバイスがそれぞれローカルでSQLiteを実行するIoTデプロイメントで起こります。そうなれば、パフォーマンスの考慮に関わらず、外部キーは選択肢から外れます（文字通り）。

しかし、そのアーキテクチャ上の境界に達するまでは、10人しか使わない社内ツールを作っているのに、Netflixが抱える問題のために時期尚早な最適化をするのはやめましょう。

## 実践的にはどうすべきか

「外部キーを使うべきか」と問う代わりに、次の3つのことを考えてみてください。

1.  **その参照が間違っていたら何が壊れるか？** 訴訟、請求ミス、規制違反に繋がりますか？ それとも、分析ダッシュボードの結合でNULLが返るだけですか？
2.  **検証が遅かったら何が壊れるか？** かけがえのないリアルタイムデータを失いますか？ それとも、クエリに50ミリ秒余計にかかるだけですか？
3.  **そのデータはスナップショットか、参照か？** ある特定の瞬間の見え方を記録していますか、それとも権威ある現在の値を指し示していますか？

ここから、パターンは自然と浮かび上がってきます。

- 金融取引、認証セッション、データの破損が法的責任を意味するものは、パフォーマンスのオーバーヘッドに関わらず、外部キーが必要です。
- 大量のログ、追記専用の時系列データ、1分間に100万件のイベントを書き込むようなものは、書き込みごとの検証オーバーヘッドは不要かもしれません。
- 注文や監査ログのような履歴スナップショット、ユーザー設定のように常に丸ごと取得するデータ、外部APIからのウェブフックペイロードのように制御できないスキーマ……これらは非正規化の方がうまくいくことが多いです。

しかし、私は「おそらく」や「ことが多い」と言いました。なぜなら文脈（Context）が重要であり、あなたの文脈は私のものとは異なるからです。

## 最後に

外部キーはパフォーマンスの問題ではありません。書き込み速度とデータ整合性の間のトレードオフであり、そのトレードオフが理にかなっているかどうかは、あなたの特定のボトルネックと特定の結果に完全に依存します。

本当の問題は、書き込みパフォーマンスの問題があるかどうかを実際に測定もせず、何を失うかも考慮せずに、「Webスケール」について読んだ知識だけで外部キーを削除してしまうことです。その結果、1日に100件の取引しか処理しない新規プロジェクトに、Netflixのアーキテクチャを「カーゴ・カルト（貨物崇拝）」のように持ち込んでしまうのです。

あなたのユースケースにおいて、パフォーマンスコストを払う価値はあるかもしれません。ないかもしれません。しかし、少なくともその決定は、「最適化すべきだと思い込んでいるもの」ではなく、「実際に最適化すべきもの」に基づいて行ってください。
