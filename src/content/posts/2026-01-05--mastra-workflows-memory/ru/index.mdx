У LLMs есть странное свойство: они блестяще понимают нюансы, но ужасно следуют рецептам. Дайте GPT-4 расплывчатую проблему — он рассудит о возможностях. Дайте ему точную последовательность шагов — и он может пропустить шаг 3, потому что шаг 5 "показался более релевантным."

Это не баг модели. Это фундаментальная характеристика вероятностных систем, пытающихся решать детерминированные задачи.

Я наблюдал, как команды борются с этим несоответствием. Они строят агента для обработки возвратов клиентов, дают ему десяток инструментов и ожидают надёжного выполнения бизнес-процесса. Иногда всё работает идеально. Иногда он галлюцинирует одобрения, которых никогда не было. Иногда застревает, трижды запрашивая одну и ту же информацию.

Решение — не лучшие промпты. Это понимание того, когда нужно перестать просить LLM "думать" и начать заставлять его "подчиняться."

## Когда детерминированное побеждает креативное

Подумайте о том, что происходит, когда нужно обработать тикет поддержки. Реальная бизнес-логика выглядит примерно так:

1. Получить детали тикета из базы данных
2. Проверить, имеет ли пользователь право на возврат (правила политики)
3. Убедиться, что транзакция существует и ещё не была возвращена
4. Рассчитать сумму возврата
5. Обработать возврат платежа
6. Обновить статус тикета
7. Отправить подтверждение по email

Можно передать это LLM как упражнение на вызов инструментов. По моему опыту — это просьба о неприятностях. Модель может решить, что шаги 2 и 3 — " basically одно и то же" — и пропустить один. Или обработать возврат до проверки права на него, потому что пользователь казался расстроенным.

Workflows существуют именно для этого сценария. Они не захватывающие, но в этом и смысл.

### Строим Weather Activity Planner

Вот практический пример, демонстрирующий паттерн. Нам нужны жёсткие фактические данные о погоде в паре с креативными предложениями активностей. Fetch погоды никогда не должен быть креативным, а предложения — должны.

```typescript
// src/mastra/workflows/activity-planner.ts
import { createWorkflow, createStep } from '@mastra/core/workflows';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

// Step 1: Fetch weather data (Deterministic)
const fetchWeather = createStep({
  id: 'fetch-weather',
  description: 'Fetches weather forecast for a given city',
  inputSchema: z.object({
    city: z.string(),
  }),
  outputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  execute: async ({ inputData }) => {
    // ... (fetch logic) ...
    const weather = await fetch(`https://api.open-meteo.com/v1/forecast?latitude=52.52&longitude=13.41&current=temperature_2m,weather_code&daily=precipitation_probability_mean`).then(r => r.json());

    return {
      location: inputData.city,
      temperature: weather.current.temperature_2m,
      conditions: getWeatherCondition(weather.current.weather_code),
      precipitationChance: weather.daily.precipitation_probability_mean[0],
    };
  },
});

// Step 2: Agent suggests activities (Creative)
const activityPlanner = new Agent({
  id: 'activity-planner-agent',
  name: 'Activity Planner',
  instructions: `You are a local activities expert. Based on weather conditions, suggest 3-5 appropriate activities.
    - For rain (>50% precipitation), prioritize indoor activities
    - For extreme temperatures, consider climate-appropriate options
    - Always include one adventurous and one relaxing option`,
  model: openai('gpt-5'),
});

const planActivities = createStep({
  id: 'plan-activities',
  description: 'Uses AI to suggest activities based on weather',
  inputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  outputSchema: z.object({
    activities: z.string(),
  }),
  execute: async ({ inputData }) => {
    const prompt = `Weather in ${inputData.location}: ${inputData.temperature}°C...`;
    const response = await activityPlanner.generate(prompt);
    return { activities: response.text };
  },
});

// The Pipeline
export const activityPlannerWorkflow = createWorkflow({
  id: 'activity-planner',
  inputSchema: z.object({ city: z.string() }),
  outputSchema: z.object({ activities: z.string() }),
})
  .then(fetchWeather)
  .then(planActivities);

activityPlannerWorkflow.commit();
```

L LM никогда не касается weather API. Он получает проверенные данные как входные, затем делает то, что он реально умеет: контекстуальные предложения. Если поменять это местами и позволить агенту получать данные о погоде — в итоге вы получите солнечный прогноз, когда на самом деле идёт дождь.

**Когда стоит考虑 workflows:**
- У вас есть известная последовательность шагов, которая должна выполняться по порядку
- Вам нужна наблюдаемость на каждом этапе (логи, метрики, тайминги)
- Вам нужна логика повторных попыток для нестабильных внешних API
- Бизнес-правила нельзя "интерпретировать" — их нужно точно соблюдать

## Проблема контекстного окна, о которой никто не говорит

Есть паттерн, который я продолжаю видеть. Кто-то строит чатбота. Во время тестирования всё работает отлично. Затем в продакшене пользователи ведут более длинные разговоры, и бот внезапно теряется.

Разработчик смотрит на логи и понимает, что отправляет всю историю разговора с каждым запросом. Все 47 сообщений. Он сжигает токены и контекстное пространство для информации, которая mostly нерелевантна.

Хуже того, есть феномен, который исследователи называют "lost in the middle" — модели хуже работают, когда релевантная информация зарыта в длинном контексте. Модель буквально не видит леса за деревьями.

Отправка всей истории разговора кажется безопасной. Вы даёте модели "всю информацию." Но вы实际上是 усложняете модели фокусировку на том, что важно.

### Рабочая память vs. Долгосрочное хранение

Система памяти Mastra даёт вам оба варианта. Рабочая память хранит последние сообщения в контекстном окне. Семантический поиск ищет в исторических сообщениях, когда текущий запрос кажется связанным.

```typescript
// src/mastra/agents/memory-agent.ts
import { Agent } from '@mastra/core/agent';
import { Memory } from '@mastra/memory';
import { LibSQLStore } from '@mastra/libsql';

export const memoryAgent = new Agent({
  id: 'memory-agent',
  name: 'Memory Agent',
  instructions: 'You are a helpful assistant with perfect recall of our conversations.',
  model: openai('gpt-5'),
  memory: new Memory({
    storage: new LibSQLStore({
      id: 'memory-agent-store',
      url: 'file:../mastra.db',
    }),
    options: {
      lastMessages: 20,  // Keep last 20 messages in context
      semanticRecall: {
        enabled: true,  // Use embeddings to find old stuff
        topK: 5,
        threshold: 0.7,
      },
    },
  }),
});
```

Вот как это работает на практике. Пользователь спрашивает: "Какой итальянский ресторан ты рекомендовал в прошлом месяце?"

Без семантического поиска агент видит последние 20 сообщений. Рекомендация ресторана была сообщением 487 из 506. Оно потеряно. Агент говорит: "У меня нет этой информации."

С семантическим поиском:
1. Запрос эмбеддится: `[0.234, -0.567, 0.891, ...]`
2. Эмбеддинг сравнивается с историческими сообщениями
3. Сообщение 487 ("Я бы порекомендовал Trattoria Bella — их карбонара невероятная") набирает 0.89 сходства
4. То сообщение внедряется в текущий контекст
5. Агент отвечает: "Я рекомендовал Trattoria Bella. Их карбонара — вот что привлекло моё внимание."

Агент кажется обладающим идеальной памятью, при этом используя лишь часть контекстного окна. Это не просто умная инженерия — это функциональная необходимость, как только разговоры расширяются за пределы нескольких десятков сообщений.

## Координация через Agent Networks

Иногда вам нужны и структура, и гибкость. Чистые workflows слишком жёсткие. Чистые агенты слишком непредсказуемые.

Agent networks дают вам координатора, который решает, какой специализированный агент или workflow вызвать на основе задачи. Думайте об этом как об умном load balancer для AI-возможностей.

```typescript
export const coordinatorAgent = new Agent({
  id: 'coordinator-agent',
  name: 'Research Coordinator',
  instructions: `You are a network of researchers and writers.
    - Use researchAgent for gathering facts
    - Use writingAgent for producing final content
    - Use weatherTool for current weather data
    - Use activityPlannerWorkflow for location-based planning

    Always produce comprehensive, well-structured responses.`,
  model: openai('gpt-5'),

  // Available primitives
  agents: { researchAgent, writingAgent },
  workflows: { activityPlannerWorkflow },
  tools: { weatherTool },

  // Network requires memory
  memory: new Memory({
    storage: new LibSQLStore({ id: 'network-store', url: 'file:../network.db' }),
  }),
});
```

Когда вы запрашиваете эту сеть, координатор анализирует запрос и маршрутизирует соответственно:
- "Мне нужны факты о X" триггерит research agent
- "Спланируй выходные в Сиэтле" запускает activity planner workflow
- "Напиши отчёт о Y" задействует writing agent

Этот паттерн масштабируется лучше, чем попытка впихнуть всё в одного мега-агента. Специализированные агенты развивают сфокусированную экспертизу. Координатор управляет маршрутизацией. Каждая часть делает то, что у неё лучше всего.

## Всё вместе

Реальные production AI-системы нуждаются в архитектуре, а не просто в промптах. Вы строите распределённые системы, где некоторые узлы являются LLMs.

Workflows дают вам гарантии, когда нужно, чтобы всё происходило именно правильно. Memory даёт вам контекст без сжигания бюджета токенов. Agent networks позволяют компоновать сложность из более простых частей.

Ничто из этого не гламурно. Но после наблюдения за достаточным количеством "полностью автономных агентов", терпящих неудачу в production, я пришёл к тому, что скучная надёжность ценнее захватывающей непредсказуемости.

Ваши результаты могут отличаться, но по моему опыту системы, которые реально доставляют и остаются работающими — это те, что относятся к LLMs как к компонентам в более крупной архитектуре, а не как к волшебным коробкам, решающим всё.
