У LLM есть странное свойство: они блестяще понимают нюансы, но ужасно следуют инструкциям. Дайте GPT-4 расплывчатую задачу — и она обдумает возможные варианты. Дайте точную последовательность шагов — и она может пропустить шаг 3, потому что шаг 5 «показался более уместным».

Это не баг модели. Это фундаментальная характеристика вероятностных систем, пытающихся решать детерминированные задачи.

Я наблюдал, как команды борются с этим несоответствием. Они строят агента для обработки возвратов клиентам, дают ему дюжину инструментов и ожидают, что он будет надёжно выполнять бизнес-процесс. Иногда работает идеально. Иногда галлюцинирует одобрения, которых никогда не было. Иногда застревает, запрашивая одну и ту же информацию трижды.

Решение — не в улучшении промптов. Это понимание, когда пора перестать просить LLM «думать» и начать говорить ей «исполнять».

## Когда детерминированность побеждает творчество

Подумайте, что происходит, когда нужно обработать тикет в поддержку. Реальная бизнес-логика выглядит примерно так:

1. Получить данные тикета из базы данных
2. Проверить, имеет ли пользователь право на возврат (правила политики)
3. Подтвердить, что транзакция существует и ещё не была возвращена
4. Рассчитать сумму возврата
5. Обработать возврат платежа
6. Обновить статус тикета
7. Отправить подтверждение по email

Можно поручить это LLM как упражнение по вызову инструментов. По моему опыту, это путь к проблемам. Модель может решить, что шаги 2 и 3 — «по сути одно и то же», и пропустить один из них. Или обработать возврат до проверки права, потому что пользователь показался расстроенным.

Workflows существуют именно для этого сценария. Это не захватывающе, но в этом и суть.

### Создание планировщика активностей по погоде

Вот практический пример, демонстрирующий паттерн. Нам нужны точные, фактические данные о погоде в паре с креативными предложениями активностей. Запрос погоды никогда не должен быть творческим, а вот предложения — должны.

```typescript
// src/mastra/workflows/activity-planner.ts
import { createWorkflow, createStep } from '@mastra/core/workflows';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

// Step 1: Fetch weather data (Deterministic)
const fetchWeather = createStep({
  id: 'fetch-weather',
  description: 'Fetches weather forecast for a given city',
  inputSchema: z.object({
    city: z.string(),
  }),
  outputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  execute: async ({ inputData }) => {
    // ... (fetch logic) ...
    const weather = await fetch(`https://api.open-meteo.com/v1/forecast?latitude=52.52&longitude=13.41&current=temperature_2m,weather_code&daily=precipitation_probability_mean`).then(r => r.json());
    
    return {
      location: inputData.city,
      temperature: weather.current.temperature_2m,
      conditions: getWeatherCondition(weather.current.weather_code),
      precipitationChance: weather.daily.precipitation_probability_mean[0],
    };
  },
});

// Step 2: Agent suggests activities (Creative)
const activityPlanner = new Agent({
  id: 'activity-planner-agent',
  name: 'Activity Planner',
  instructions: `You are a local activities expert. Based on weather conditions, suggest 3-5 appropriate activities.
    - For rain (>50% precipitation), prioritize indoor activities
    - For extreme temperatures, consider climate-appropriate options
    - Always include one adventurous and one relaxing option`,
  model: openai('gpt-5'),
});

const planActivities = createStep({
  id: 'plan-activities',
  description: 'Uses AI to suggest activities based on weather',
  inputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  outputSchema: z.object({
    activities: z.string(),
  }),
  execute: async ({ inputData }) => {
    const prompt = `Weather in ${inputData.location}: ${inputData.temperature}°C...`;
    const response = await activityPlanner.generate(prompt);
    return { activities: response.text };
  },
});

// The Pipeline
export const activityPlannerWorkflow = createWorkflow({
  id: 'activity-planner',
  inputSchema: z.object({ city: z.string() }),
  outputSchema: z.object({ activities: z.string() }),
})
  .then(fetchWeather)
  .then(planActivities);

activityPlannerWorkflow.commit();
```

LLM никогда не касается API погоды. Она получает проверенные данные на входе, а затем делает то, в чём действительно хороша: даёт контекстные рекомендации. Если перевернуть этот подход и позволить агенту самому запрашивать погоду, рано или поздно вы получите солнечный прогноз, когда на самом деле идёт дождь.

**Когда стоит рассматривать workflows:**
- У вас есть известная последовательность шагов, которые должны выполняться по порядку
- Нужна наблюдаемость на каждом этапе (логи, метрики, тайминги)
- Нужна логика повторных попыток для нестабильных внешних API
- Бизнес-правила нельзя «интерпретировать» — они должны выполняться точно

## Проблема контекстного окна, о которой никто не говорит

Есть паттерн, который я постоянно вижу. Кто-то создаёт чат-бота. Он отлично работает во время тестирования. Затем в продакшене у пользователей возникают более длинные разговоры, и внезапно бот теряется.

Разработчик смотрит в логи и понимает, что отправляет всю историю разговора с каждым запросом. Все 47 сообщений. Токены и контекстное пространство сжигаются на информацию, которая по большей части нерелевантна.

Хуже того, существует феномен, который исследователи называют «потеря в середине»: модели работают хуже, когда релевантная информация погребена в длинном контексте. Модель буквально не видит лес за деревьями.

Отправка полной истории разговора кажется безопасной. Вы даёте модели «всю информацию». Но на самом деле вы затрудняете модели фокусировку на важном.

### Рабочая память vs. долговременное хранение

Система памяти Mastra даёт и то, и другое. Рабочая память хранит недавние сообщения в контекстном окне. Семантический поиск обращается к историческим сообщениям, когда текущий запрос кажется связанным.

```typescript
// src/mastra/agents/memory-agent.ts
import { Agent } from '@mastra/core/agent';
import { Memory } from '@mastra/memory';
import { LibSQLStore } from '@mastra/libsql';

export const memoryAgent = new Agent({
  id: 'memory-agent',
  name: 'Memory Agent',
  instructions: 'You are a helpful assistant with perfect recall of our conversations.',
  model: openai('gpt-5'),
  memory: new Memory({
    storage: new LibSQLStore({
      id: 'memory-agent-store',
      url: 'file:../mastra.db',
    }),
    options: {
      lastMessages: 20,  // Keep last 20 messages in context
      semanticRecall: {
        enabled: true,  // Use embeddings to find old stuff
        topK: 5,
        threshold: 0.7,
      },
    },
  }),
});
```

Вот как это работает на практике. Пользователь спрашивает: «Какое итальянское заведение ты рекомендовал в прошлом месяце?»

Без семантического поиска агент видит только последние 20 сообщений. Рекомендация ресторана была в сообщении 487 из 506. Она потеряна. Агент отвечает: «У меня нет этой информации».

С семантическим поиском:
1. Запрос векторизуется: `[0.234, -0.567, 0.891, ...]`
2. Вектор сравнивается с историческими сообщениями
3. Сообщение 487 («Я бы порекомендовал Trattoria Bella — их карбонара великолепна») получает оценку схожести 0.89
4. Это сообщение добавляется в текущий контекст
5. Агент отвечает: «Я рекомендовал Trattoria Bella. Их карбонара привлекла моё внимание.»

Агент выглядит так, будто обладает идеальной памятью, используя при этом лишь малую часть контекстного окна. Это не просто умная инженерия — это функциональная необходимость, когда разговоры выходят за пределы нескольких десятков сообщений.

## Координация через сети агентов

Иногда нужны и структура, и гибкость. Чистые workflows слишком жёсткие. Чистые агенты слишком непредсказуемы.

Сети агентов предоставляют координатора, который решает, какого специализированного агента или workflow вызвать в зависимости от задачи. Представьте это как умный балансировщик нагрузки для AI-возможностей.

```typescript
export const coordinatorAgent = new Agent({
  id: 'coordinator-agent',
  name: 'Research Coordinator',
  instructions: `You are a network of researchers and writers.
    - Use researchAgent for gathering facts
    - Use writingAgent for producing final content
    - Use weatherTool for current weather data
    - Use activityPlannerWorkflow for location-based planning
    
    Always produce comprehensive, well-structured responses.`,
  model: openai('gpt-5'),
  
  // Available primitives
  agents: { researchAgent, writingAgent },
  workflows: { activityPlannerWorkflow },
  tools: { weatherTool },
  
  // Network requires memory
  memory: new Memory({
    storage: new LibSQLStore({ id: 'network-store', url: 'file:../network.db' }),
  }),
});
```

Когда вы отправляете запрос в эту сеть, координатор анализирует запрос и направляет его соответственно:
- «Мне нужны факты об X» запускает исследовательского агента
- «Спланируй выходные в Сиэтле» запускает workflow планирования активностей
- «Напиши отчёт о Y» привлекает агента-писателя

Этот паттерн масштабируется лучше, чем попытка втиснуть всё в одного мега-агента. Специализированные агенты развивают сфокусированную экспертизу. Координатор занимается маршрутизацией. Каждый компонент делает то, в чём он хорош.

## Собираем всё вместе

Настоящие продакшен-системы на AI требуют архитектуры, а не просто промптов. Вы строите распределённые системы, где некоторые узлы оказываются LLM.

Workflows дают гарантии, когда нужно, чтобы всё происходило точно. Память даёт контекст, не сжигая токенный бюджет. Сети агентов позволяют собирать сложность из более простых частей.

Ничто из этого не гламурно. Но после того, как я наблюдал достаточно «полностью автономных агентов», терпящих неудачу в продакшене, я научился ценить скучную надёжность больше, чем захватывающую непредсказуемость.

Ваш опыт может отличаться, но, по моему опыту, системы, которые действительно попадают в продакшен и остаются в работе — это те, которые рассматривают LLM как компоненты более крупной архитектуры, а не как магические коробки, решающие всё.
