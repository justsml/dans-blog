LLMs haben eine seltsame Eigenschaft: Sie sind brillant darin, Nuancen zu verstehen, aber schlecht darin, Rezepte zu befolgen. Gib GPT-4 ein vages Problem und es denkt über Möglichkeiten nach. Gib ihm eine präzise Abfolge von Schritten, und es überspringt vielleicht Schritt 3, weil Schritt 5 „sich relevanter anfühlte."

Das ist kein Bug im Modell. Es ist eine grundlegende Eigenschaft probabilistischer Systeme, die versuchen, deterministische Probleme zu lösen.

Ich habe Teams beobachtet, die mit diesem Missverhältnis kämpfen. Sie bauen einen Agenten für Kundenrückerstattungen, geben ihm ein Dutzend Tools und erwarten, dass er zuverlässig einen Geschäftsprozess ausführt. Manchmal funktioniert es perfekt. Manchmal halluziniert er Genehmigungen, die nie stattfanden. Manchmal fordert er dreimal dieselbe Information an.

Die Lösung sind nicht bessere Prompts. Es geht darum zu wissen, wann man aufhört, das LLM zu bitten zu „denken", und anfängt, ihm zu sagen, es solle „gehorchen."

---

## Wenn Deterministisch besser ist als Kreativ

Überlege, was passiert, wenn ein Support-Ticket bearbeitet werden muss. Echte Geschäftslogik sieht in etwa so aus:

1. Ticketdetails aus der Datenbank abrufen
2. Prüfen, ob der Benutzer für eine Rückerstattung berechtigt ist (Richtlinienregeln)
3. Überprüfen, ob die Transaktion existiert und noch nicht erstattet wurde
4. Rückerstattungsbetrag berechnen
5. Zahlungsrückabwicklung durchführen
6. Ticketstatus aktualisieren
7. Bestätigungs-E-Mail senden

Du könntest das einem LLM als Tool-Calling-Übung übergeben. Meiner Erfahrung nach ist das ein Rezept für Ärger. Das Modell könnte entscheiden, dass Schritt 2 und 3 „im Grunde dasselbe" sind und einen überspringen. Oder es bearbeitet die Rückerstattung vor der Berechtigungsprüfung, weil der Benutzer verärgert wirkte.

Workflows existieren genau für dieses Szenario. Sie sind nicht aufregend, aber das ist der Punkt.

### Erstellen eines Wetter-Aktivitätsplaners

Hier ist ein praktisches Beispiel, das das Muster zeigt. Wir brauchen harte, faktische Wetterdaten gepaart mit kreativen Aktivitätsvorschlägen. Der Wetterabruf sollte niemals kreativ sein, aber die Vorschläge sollten es sein.

```typescript
// src/mastra/workflows/activity-planner.ts
import { createWorkflow, createStep } from '@mastra/core/workflows';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

// Step 1: Fetch weather data (Deterministic)
const fetchWeather = createStep({
  id: 'fetch-weather',
  description: 'Fetches weather forecast for a given city',
  inputSchema: z.object({
    city: z.string(),
  }),
  outputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  execute: async ({ inputData }) => {
    // ... (fetch logic) ...
    const weather = await fetch(`https://api.open-meteo.com/v1/forecast?latitude=52.52&longitude=13.41&current=temperature_2m,weather_code&daily=precipitation_probability_mean`).then(r => r.json());
    
    return {
      location: inputData.city,
      temperature: weather.current.temperature_2m,
      conditions: getWeatherCondition(weather.current.weather_code),
      precipitationChance: weather.daily.precipitation_probability_mean[0],
    };
  },
});

// Step 2: Agent suggests activities (Creative)
const activityPlanner = new Agent({
  id: 'activity-planner-agent',
  name: 'Activity Planner',
  instructions: `You are a local activities expert. Based on weather conditions, suggest 3-5 appropriate activities.
    - For rain (>50% precipitation), prioritize indoor activities
    - For extreme temperatures, consider climate-appropriate options
    - Always include one adventurous and one relaxing option`,
  model: openai('gpt-5'),
});

const planActivities = createStep({
  id: 'plan-activities',
  description: 'Uses AI to suggest activities based on weather',
  inputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  outputSchema: z.object({
    activities: z.string(),
  }),
  execute: async ({ inputData }) => {
    const prompt = `Weather in ${inputData.location}: ${inputData.temperature}°C...`;
    const response = await activityPlanner.generate(prompt);
    return { activities: response.text };
  },
});

// The Pipeline
export const activityPlannerWorkflow = createWorkflow({
  id: 'activity-planner',
  inputSchema: z.object({ city: z.string() }),
  outputSchema: z.object({ activities: z.string() }),
})
  .then(fetchWeather)
  .then(planActivities);

activityPlannerWorkflow.commit();
```

Das LLM berührt niemals die Wetter-API. Es erhält faktische Daten als Eingabe und macht dann das, was es tatsächlich gut kann: kontextbezogene Vorschläge machen. Wenn man es umdreht und den Agenten selbst Wetterdaten abrufen lässt, bekommt man irgendwann eine sonnige Vorhersage, während es tatsächlich regnet.

**Wann man Workflows in Betracht ziehen sollte:**
- Du hast eine bekannte Abfolge von Schritten, die in einer bestimmten Reihenfolge ablaufen müssen
- Du brauchst Beobachtbarkeit auf jeder Stufe (Logs, Metriken, Timing)
- Du brauchst Wiederholungslogik für unzuverlässige externe APIs
- Geschäftsregeln dürfen nicht „interpretiert" werden – sie müssen exakt befolgt werden

## Das Kontextfenster-Problem, über das niemand spricht

Es gibt ein Muster, das ich immer wieder sehe. Jemand baut einen Chatbot. Er funktioniert während der Tests großartig. In der Produktion haben die Benutzer längere Unterhaltungen, und plötzlich verliert der Bot den Faden.

Der Entwickler schaut in die Logs und stellt fest, dass die gesamte Konversationshistorie mit jeder Anfrage mitgesendet wird. Alle 47 Nachrichten. Sie verbrennen Token und Kontextplatz für Informationen, die größtenteils irrelevant sind.

Schlimmer noch, es gibt ein Phänomen, das Forscher als „Lost in the Middle" bezeichnen: Modelle liefern schlechtere Ergebnisse, wenn relevante Informationen in einem langen Kontext vergraben sind. Das Modell kann buchstäblich den Wald vor lauter Bäumen nicht sehen.

Die gesamte Konversationshistorie zu senden, fühlt sich sicher an. Du gibst dem Modell „alle Informationen." Aber du machst es dem Modell tatsächlich schwerer, sich auf das Wesentliche zu konzentrieren.

### Arbeitsgedächtnis vs. Langzeitspeicher

Mastras Speichersystem bietet beides. Das Arbeitsgedächtnis behält aktuelle Nachrichten im Kontextfenster. Die semantische Suche durchsucht historische Nachrichten, wenn die aktuelle Anfrage verwandt erscheint.

```typescript
// src/mastra/agents/memory-agent.ts
import { Agent } from '@mastra/core/agent';
import { Memory } from '@mastra/memory';
import { LibSQLStore } from '@mastra/libsql';

export const memoryAgent = new Agent({
  id: 'memory-agent',
  name: 'Memory Agent',
  instructions: 'You are a helpful assistant with perfect recall of our conversations.',
  model: openai('gpt-5'),
  memory: new Memory({
    storage: new LibSQLStore({
      id: 'memory-agent-store',
      url: 'file:../mastra.db',
    }),
    options: {
      lastMessages: 20,  // Keep last 20 messages in context
      semanticRecall: {
        enabled: true,  // Use embeddings to find old stuff
        topK: 5,
        threshold: 0.7,
      },
    },
  }),
});
```

So sieht das in der Praxis aus. Ein Benutzer fragt: „Was war das italienische Restaurant, das du letztes Monat empfohlen hast?"

Ohne semantische Suche sieht der Agent die letzten 20 Nachrichten. Die Restaurantempfehlung war Nachricht 487 von 506. Sie ist weg. Der Agent sagt: „Diese Information habe ich nicht."

Mit semantischer Suche:
1. Die Anfrage wird eingebettet: `[0.234, -0.567, 0.891, ...]`
2. Die Einbettung wird mit historischen Nachrichten verglichen
3. Nachricht 487 („Ich würde die Trattoria Bella empfehlen – ihre Carbonara ist unglaublich") erzielt eine Ähnlichkeit von 0.89
4. Diese Nachricht wird in den aktuellen Kontext eingefügt
5. Der Agent antwortet: „Ich habe Trattoria Bella empfohlen. Ihre Carbonara hat meine Aufmerksamkeit erregt."

Der Agent scheint ein perfektes Gedächtnis zu haben, während er nur einen Bruchteil des Kontextfensters nutzt. Das ist nicht nur clevere Technik – es ist funktional notwendig, sobald Unterhaltungen über ein paar Dutzend Nachrichten hinausgehen.

## Koordination durch Agentennetzwerke

Manchmal braucht man sowohl Struktur als auch Flexibilität. Reine Workflows sind zu starr. Reine Agenten sind zu unberechenbar.

Agentennetzwerke bieten einen Koordinator, der je nach Aufgabe entscheidet, welcher spezialisierte Agent oder Workflow aufgerufen wird. Stell es dir als intelligenten Lastverteiler für KI-Fähigkeiten vor.

```typescript
export const coordinatorAgent = new Agent({
  id: 'coordinator-agent',
  name: 'Research Coordinator',
  instructions: `You are a network of researchers and writers.
    - Use researchAgent for gathering facts
    - Use writingAgent for producing final content
    - Use weatherTool for current weather data
    - Use activityPlannerWorkflow for location-based planning
    
    Always produce comprehensive, well-structured responses.`,
  model: openai('gpt-5'),
  
  // Available primitives
  agents: { researchAgent, writingAgent },
  workflows: { activityPlannerWorkflow },
  tools: { weatherTool },
  
  // Network requires memory
  memory: new Memory({
    storage: new LibSQLStore({ id: 'network-store', url: 'file:../network.db' }),
  }),
});
```

Wenn du dieses Netzwerk abfragst, analysiert der Koordinator die Anfrage und leitet entsprechend weiter:
- „Ich brauche Fakten über X" löst den Recherche-Agenten aus
- „Plane ein Wochenende in Seattle" führt den Aktivitätsplaner-Workflow aus
- „Schreibe einen Bericht über Y" aktiviert den Schreib-Agenten

Dieses Muster skaliert besser als der Versuch, alles in einen einzigen Mega-Agenten zu stopfen. Spezialisierte Agenten entwickeln fokussierte Expertise. Der Koordinator übernimmt das Routing. Jeder Teil macht das, was er am besten kann.

## Alles zusammennehmen

Echte KI-Produktionssysteme brauchen Architektur, nicht nur Prompts. Du baust verteilte Systeme, in denen einige Knoten zufällig LLMs sind.

Workflows geben dir Garantien, wenn Dinge exakt richtig laufen müssen. Speicher gibt dir Kontext, ohne dein Token-Budget zu verbrennen. Agentennetzwerke erlauben dir, Komplexität aus einfacheren Teilen zusammenzusetzen.

Nichts davon ist glamourös. Aber nachdem ich genug „vollautonome Agenten" in der Produktion habe scheitern sehen, habe ich gelernt, langweilige Zuverlässigkeit aufregender Unberechenbarkeit vorzuziehen.

Deine Erfahrung mag abweichen, aber meiner Erfahrung nach sind die Systeme, die tatsächlich ausgeliefert werden und laufen bleiben, diejenigen, die LLMs als Komponenten in einer größeren Architektur behandeln – und nicht als magische Boxen, die alles lösen.
