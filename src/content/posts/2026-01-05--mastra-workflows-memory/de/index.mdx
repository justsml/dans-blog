LLMs haben diese seltsame Eigenschaft: Sie sind brillant im Umgang mit Nuancen, aber schrecklich darin, Rezepte zu befolgen. Gib GPT-4 ein vages Problem und es wird Möglichkeiten durchdenken. Gib ihm eine präzise Abfolge von Schritten und es überspringt vielleicht Schritt 3, weil Schritt 5 "relevanter erschien".

Das ist kein Bug im Modell. Es ist eine grundlegende Eigenschaft probabilistischer Systeme, die deterministische Probleme lösen sollen.

Ich habe zugesehen, wie Teams mit dieser Diskrepanz kämpfen. Sie bauen einen Agenten für Kunden-Rückerstattungen, geben ihm ein Dutzend Tools und erwarten, dass er zuverlässig einen Geschäftsprozess ausführt. Manchmal funktioniert es perfekt. Manchmal halluziniert es Genehmigungen, die nie stattgefunden haben. Manchmal bleibt es hängen und fragt dreimal nach denselben Informationen.

Die Lösung sind keine besseren Prompts. Es geht darum zu wissen, wann man aufhören soll, den LLM "denken" zu lassen, und anfangen soll, ihm zu "gehorchen".

## Wann Deterministisch Kreativität schlägt

Denk daran, was passiert, wenn du ein Support-Ticket bearbeiten musst. Echte Geschäftslogik sieht ungefähr so aus:

1. Ticket-Details aus der Datenbank abrufen
2. Prüfen, ob der Benutzer für eine Rückerstattung berechtigt ist (Policy-Regeln)
3. Überprüfen, ob die Transaktion existiert und noch nicht zurückerstattet wurde
4. Rückerstattungsbetrag berechnen
5. Zahlungsrückgabe verarbeiten
6. Ticket-Status aktualisieren
7. Bestätigungs-E-Mail senden

Du könntest dies einem LLM als Tool-Calling-Übung übergeben. In meiner Erfahrung ist das Trouble. Das Modell könnte entscheiden, dass Schritte 2 und 3 "im Grunde dieselbe Sache" sind und einen überspringen. Oder es könnte die Rückerstattung bearbeiten, bevor es die Berechtigung prüft, weil der Benutzer upset schien.

Workflows existieren genau für dieses Szenario. Sie sind nicht aufregend, aber darum geht es.

### Einen Weather Activity Planner bauen

Hier ist ein praktisches Beispiel, das das Muster zeigt. Wir brauchen harte, faktische Wetterdaten, gepaart mit kreativen Aktivitätsvorschlägen. Der Wetterabruf sollte nie kreativ sein, aber die Vorschläge sollten es sein.

```typescript
// src/mastra/workflows/activity-planner.ts
import { createWorkflow, createStep } from '@mastra/core/workflows';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

// Schritt 1: Wetterdaten abrufen (Deterministisch)
const fetchWeather = createStep({
  id: 'fetch-weather',
  description: 'Ruft Wettervorhersage für eine gegebene Stadt ab',
  inputSchema: z.object({
    city: z.string(),
  }),
  outputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  execute: async ({ inputData }) => {
    // ... (Abruf-Logik) ...
    const weather = await fetch(`https://api.open-meteo.com/v1/forecast?latitude=52.52&longitude=13.41&current=temperature_2m,weather_code&daily=precipitation_probability_mean`).then(r => r.json());

    return {
      location: inputData.city,
      temperature: weather.current.temperature_2m,
      conditions: getWeatherCondition(weather.current.weather_code),
      precipitationChance: weather.daily.precipitation_probability_mean[0],
    };
  },
});

// Schritt 2: Agent schlägt Aktivitäten vor (Kreativ)
const activityPlanner = new Agent({
  id: 'activity-planner-agent',
  name: 'Activity Planner',
  instructions: `Du bist ein lokaler Aktivitätsexperte. Basierend auf Wetterbedingungen schlage 3-5 passende Aktivitäten vor.
    - Bei Regen (>50% Niederschlag) priorisiere Indoor-Aktivitäten
    - Bei extremen Temperaturen考虑气候适当的选项
    - Immer eine abenteuerliche und eine entspannende Option einbeziehen`,
  model: openai('gpt-5'),
});

const planActivities = createStep({
  id: 'plan-activities',
  description: 'Nutzt KI um Aktivitäten basierend auf dem Wetter vorzuschlagen',
  inputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  outputSchema: z.object({
    activities: z.string(),
  }),
  execute: async ({ inputData }) => {
    const prompt = `Wetter in ${inputData.location}: ${inputData.temperature}°C...`;
    const response = await activityPlanner.generate(prompt);
    return { activities: response.text };
  },
});

// Die Pipeline
export const activityPlannerWorkflow = createWorkflow({
  id: 'activity-planner',
  inputSchema: z.object({ city: z.string() }),
  outputSchema: z.object({ activities: z.string() }),
})
  .then(fetchWeather)
  .then(planActivities);

activityPlannerWorkflow.commit();
```

Der LLM ruft nie die Wetter-API auf. Er erhält Ground-Truth-Daten als Input, dann macht er das, was er tatsächlich gut kann: kontextuelle Vorschläge. Wenn du dies umdrehst und den Agenten Wetterdaten abrufen lässt, wirst du irgendwann eine sunny Vorhersage bekommen, obwohl es tatsächlich regnet.

**Wann du Workflows in Betracht ziehen solltest:**
- Du hast eine bekannte Abfolge von Schritten, die in einer bestimmten Reihenfolge passieren müssen
- Du brauchst Observability in jeder Phase (Logs, Metriken, Timing)
- Du brauchst Retry-Logik für instabile externe APIs
- Geschäftsregeln können nicht "interpretiert" werden - sie müssen genau befolgt werden

## Das Context-Window-Problem, über das niemand spricht

Es gibt dieses Muster, das ich immer wieder sehe. Jemand baut einen Chatbot. Er funktioniert großartig während des Testens. Dann in Produktion haben Benutzer längere Gespräche und plötzlich ist der Bot lost.

Der Entwickler schaut sich die Logs an und merkt, dass er die gesamte Gesprächshistorie mit jeder Anfrage sendet. Alle 47 Nachrichten. Er verbrennt Tokens und Context-Space für Informationen, die größtenteils irrelevant sind.

Noch schlimmer: Es gibt ein Phänomen, das Forscher "lost in the middle" nennen, bei dem Modelle schlechter abschneiden, wenn relevante Informationen in einem langen Kontext vergraben sind. Das Modell kann buchstäblich den Wald vor lauter Bäumen nicht sehen.

Das Senden der vollständigen Gesprächshistorie fühlt sich sicher an. Du gibst dem Modell "alle Informationen". Aber du machst es tatsächlich schwerer für das Modell, sich auf das zu konzentrieren, was wichtig ist.

### Working Memory vs. Langzeitspeicher

Mastras Memory-System bietet beides. Working Memory behält aktuelle Nachrichten im Context Window. Semantic Recall durchsucht historische Nachrichten, wenn die aktuelle Anfrage verwandt erscheint.

```typescript
// src/mastra/agents/memory-agent.ts
import { Agent } from '@mastra/core/agent';
import { Memory } from '@mastra/memory';
import { LibSQLStore } from '@mastra/libsql';

export const memoryAgent = new Agent({
  id: 'memory-agent',
  name: 'Memory Agent',
  instructions: 'Du bist ein hilfreicher Assistent mit perfektem Erinnerungsvermögen an unsere Gespräche.',
  model: openai('gpt-5'),
  memory: new Memory({
    storage: new LibSQLStore({
      id: 'memory-agent-store',
      url: 'file:../mastra.db',
    }),
    options: {
      lastMessages: 20,  // Behalte die letzten 20 Nachrichten im Kontext
      semanticRecall: {
        enabled: true,  // Nutze Embeddings um alte Stuff zu finden
        topK: 5,
        threshold: 0.7,
      },
    },
  }),
});
```

So spielt sich dies in der Praxis ab. Ein Benutzer fragt: "Welches italienische Restaurant hast du letztens empfohlen?"

Ohne Semantic Recall sieht der Agent die letzten 20 Nachrichten. Die Restaurant-Empfehlung war Nachricht 487 von 506. Sie ist weg. Der Agent sagt "Ich habe diese Information nicht."

Mit Semantic Recall:
1. Die Anfrage wird eingebettet: `[0.234, -0.567, 0.891, ...]`
2. Das Embedding wird mit historischen Nachrichten verglichen
3. Nachricht 487 ("Ich würde Trattoria Bella empfehlen - ihre Carbonara ist unglaublich") erzielt 0.89 Ähnlichkeit
4. Diese Nachricht wird in den aktuellen Kontext injiziert
5. Der Agent antwortet: "Ich habe Trattoria Bella empfohlen. Ihre Carbonara hat mich beeindruckt."

Der Agent scheint perfektes Erinnerungsvermögen zu haben, während er nur einen Bruchteil des Context Windows nutzt. Das ist nicht nur cleveres Engineering - es ist funktional notwendig, sobald Gespräche sich über ein paar Dutzend Nachrichten hinaus erstrecken.

## Koordination durch Agent Networks

Manchmal brauchst du beides: Struktur und Flexibilität. Pure Workflows sind zu starr. Pure Agents sind zu unvorhersehbar.

Agent Networks geben dir einen Coordinator, der entscheidet, welchen spezialisierten Agenten oder Workflow basierend auf der Aufgabe aufzurufen. Denk daran als intelligenten Load Balancer für KI-Fähigkeiten.

```typescript
export const coordinatorAgent = new Agent({
  id: 'coordinator-agent',
  name: 'Research Coordinator',
  instructions: `Du bist ein Netzwerk von Forschern und Autoren.
    - Nutze researchAgent für das Sammeln von Fakten
    - Nutze writingAgent für die Produktion finaler Inhalte
    - Nutze weatherTool für aktuelle Wetterdaten
    - Nutze activityPlannerWorkflow für ortsbasierte Planung

    Produziere immer umfassende, gut strukturierte Antworten.`,
  model: openai('gpt-5'),

  // Verfügbare Primitive
  agents: { researchAgent, writingAgent },
  workflows: { activityPlannerWorkflow },
  tools: { weatherTool },

  // Network requires Memory
  memory: new Memory({
    storage: new LibSQLStore({ id: 'network-store', url: 'file:../network.db' }),
  }),
});
```

Wenn du dieses Network abfragst, analysiert der Coordinator die Anfrage und routed entsprechend:
- "Ich brauche Fakten über X" löst den Research Agent aus
- "Plane ein Wochenende in Seattle" führt den Activity Planner Workflow aus
- "Schreibe einen Bericht über Y" aktiviert den Writing Agent

Dieses Muster skaliert besser als zu versuchen, alles in einen einzelnen Mega-Agenten zu quetschen. Spezialisierte Agents entwickeln fokussierte Expertise. Der Coordinator kümmert sich ums Routing. Jedes Teil macht das, was es gut kann.

## Alles zusammenfügen

Echte Produktions-KI-Systeme brauchen Architektur, nicht nur Prompts. Du baust verteilte Systeme, wo manche Nodes zufällig LLMs sind.

Workflows geben dir Guarantees, wenn du brauchst, dass Dinge genau richtig passieren. Memory gibt dir Kontext, ohne dein Token-Budget zu verbrennen. Agent Networks lassen dich Komplexität aus einfacheren Teilen zusammensetzen.

Nichts davon ist glamourös. Aber nach dem Zusehen von genug "vollautonomen Agents", die in Produktion versagen, habe ich angefangen, langweilige Zuverlässigkeit über aufregende Unvorhersehbarkeit zu schätzen.

Dein Kilometerstand kann variieren, aber in meiner Erfahrung sind die Systeme, die tatsächlich ausgeliefert werden und weiterlaufen, diejenigen, die LLMs als Komponenten in einer größeren Architektur behandeln, anstatt als magische Boxen, die alles lösen.
