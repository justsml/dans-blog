LLMには奇妙な性質がある：ニュアンスを理解するのは天才的だが、手順に従うのは絶望的に苦手だ。GPT-4に曖昧な問題を与えれば、可能性を推論してくれる。正確な手順を与えても、ステップ3を飛ばしてステップ5が「もっと関連性が高い」と判断するかもしれない。

これはモデルのバグではない。決定的な問題を解こうとする確率的システムの根本的な特性だ。

このミスマッチに苦戦するチームを何度も見てきた。顧客の返金を処理するエージェントを作り、十数個のツールを与え、ビジネスプロセスを確実に実行できると期待する。完璧に動くこともある。存在しない承認を幻覚することもある。同じ情報を3回聞き続けることもある。

解決策はプロンプトの改善ではない。LLMに「考えさせる」のをやめ、「従わせる」タイミングを知ることだ。

## 決定的アプローチが創造的アプローチに勝つとき

サポートチケットを処理する場面を想像してみよう。現実のビジネスロジックはだいたいこうなる：

1. データベースからチケットの詳細を取得する
2. ユーザーが返金対象かどうかを確認する（ポリシールール）
3. 取引が存在し、まだ返金されていないことを検証する
4. 返金額を計算する
5. 支払いの取り消しを処理する
6. チケットのステータスを更新する
7. 確認メールを送信する

これをツール呼び出しの課題としてLLMに任せることもできる。私の経験では、それはトラブルを招くだけだ。モデルはステップ2と3を「基本的に同じこと」と判断して片方を飛ばすかもしれない。あるいは、ユーザーが怒っているように見えたからといって、適格性を確認する前に返金を処理するかもしれない。

ワークフローはまさにこのシナリオのために存在する。エキサイティングではないが、それがポイントだ。

### 天気アクティビティプランナーの構築

このパターンを示す実践的な例を見よう。確かで事実的な天気データと、創造的なアクティビティ提案を組み合わせる必要がある。天気の取得は決して創造的であってはならないが、提案は創造的であるべきだ。

```typescript
// src/mastra/workflows/activity-planner.ts
import { createWorkflow, createStep } from '@mastra/core/workflows';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

// Step 1: Fetch weather data (Deterministic)
const fetchWeather = createStep({
  id: 'fetch-weather',
  description: 'Fetches weather forecast for a given city',
  inputSchema: z.object({
    city: z.string(),
  }),
  outputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  execute: async ({ inputData }) => {
    // ... (fetch logic) ...
    const weather = await fetch(`https://api.open-meteo.com/v1/forecast?latitude=52.52&longitude=13.41&current=temperature_2m,weather_code&daily=precipitation_probability_mean`).then(r => r.json());
    
    return {
      location: inputData.city,
      temperature: weather.current.temperature_2m,
      conditions: getWeatherCondition(weather.current.weather_code),
      precipitationChance: weather.daily.precipitation_probability_mean[0],
    };
  },
});

// Step 2: Agent suggests activities (Creative)
const activityPlanner = new Agent({
  id: 'activity-planner-agent',
  name: 'Activity Planner',
  instructions: `You are a local activities expert. Based on weather conditions, suggest 3-5 appropriate activities.
    - For rain (>50% precipitation), prioritize indoor activities
    - For extreme temperatures, consider climate-appropriate options
    - Always include one adventurous and one relaxing option`,
  model: openai('gpt-5'),
});

const planActivities = createStep({
  id: 'plan-activities',
  description: 'Uses AI to suggest activities based on weather',
  inputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  outputSchema: z.object({
    activities: z.string(),
  }),
  execute: async ({ inputData }) => {
    const prompt = `Weather in ${inputData.location}: ${inputData.temperature}°C...`;
    const response = await activityPlanner.generate(prompt);
    return { activities: response.text };
  },
});

// The Pipeline
export const activityPlannerWorkflow = createWorkflow({
  id: 'activity-planner',
  inputSchema: z.object({ city: z.string() }),
  outputSchema: z.object({ activities: z.string() }),
})
  .then(fetchWeather)
  .then(planActivities);

activityPlannerWorkflow.commit();
```

LLMは天気APIに一切触れない。グラウンドトゥルースのデータを入力として受け取り、本当に得意なこと——文脈に応じた提案——に集中する。これを逆にしてエージェントに天気データを取得させると、実際は雨なのに晴れの予報を返すことがいずれ起きる。

**ワークフローを検討すべきケース：**
- 実行順序が決まっている一連のステップがある
- 各ステージでの可観測性が必要（ログ、メトリクス、タイミング）
- 不安定な外部APIにリトライロジックが必要
- ビジネスルールを「解釈」してはならない——正確に従う必要がある

## 誰も語らないコンテキストウィンドウの問題

こんなパターンを何度も見かける。チャットボットを作る。テスト中は素晴らしく動く。しかし本番に入ると、ユーザーとの会話が長くなってボットが迷子になる。

開発者がログを見ると、リクエストごとに会話履歴全体を送っていることに気づく。47メッセージ全部。ほとんど無関係な情報のためにトークンとコンテキスト領域を浪費している。

さらに悪いことに、研究者が「真ん中で見失う」と呼ぶ現象がある。関連情報が長いコンテキストの奥に埋もれていると、モデルのパフォーマンスが低下するのだ。モデルは木を見て森を見失う。

会話履歴を全部送るのは安心感がある。「すべての情報」をモデルに与えているつもりだから。しかし実際には、モデルが重要なことに集中するのを難しくしている。

### ワーキングメモリと長期記憶

Mastraのメモリシステムは両方を提供する。ワーキングメモリは最近のメッセージをコンテキストウィンドウに保持する。セマンティックリコールは、現在のクエリに関連しそうな過去のメッセージを検索する。

```typescript
// src/mastra/agents/memory-agent.ts
import { Agent } from '@mastra/core/agent';
import { Memory } from '@mastra/memory';
import { LibSQLStore } from '@mastra/libsql';

export const memoryAgent = new Agent({
  id: 'memory-agent',
  name: 'Memory Agent',
  instructions: 'You are a helpful assistant with perfect recall of our conversations.',
  model: openai('gpt-5'),
  memory: new Memory({
    storage: new LibSQLStore({
      id: 'memory-agent-store',
      url: 'file:../mastra.db',
    }),
    options: {
      lastMessages: 20,  // Keep last 20 messages in context
      semanticRecall: {
        enabled: true,  // Use embeddings to find old stuff
        topK: 5,
        threshold: 0.7,
      },
    },
  }),
});
```

実際にどう動くか見よう。ユーザーがこう聞く：「先月おすすめしてくれたあのイタリアンレストランは何だったっけ？」

セマンティックリコールなしでは、エージェントは直近の20メッセージしか見ない。レストランのおすすめは506件中487件目のメッセージだった。もう消えている。エージェントは「その情報は持っていません」と答える。

セマンティックリコールありなら：
1. クエリが埋め込みベクトル化される：`[0.234, -0.567, 0.891, ...]`
2. その埋め込みが過去のメッセージと比較される
3. メッセージ487（「Trattoria Bellaをおすすめします——カルボナーラが素晴らしい」）が0.89の類似度スコアを獲得
4. そのメッセージが現在のコンテキストに注入される
5. エージェントが答える：「Trattoria Bellaをおすすめしました。カルボナーラが気に入りましたね。」

エージェントは完璧な記憶を持っているように見えるが、コンテキストウィンドウのほんの一部しか使っていない。これは単なる賢いエンジニアリングではなく、会話が数十メッセージを超えると機能的に必要になる。

## エージェントネットワークによる協調

構造と柔軟性の両方が必要な場面もある。純粋なワークフローは硬直しすぎ、純粋なエージェントは予測不可能すぎる。

エージェントネットワークは、タスクに基づいてどの専門エージェントやワークフローを呼び出すかを決定するコーディネーターを提供する。AI機能のためのスマートなロードバランサーと考えていい。

```typescript
export const coordinatorAgent = new Agent({
  id: 'coordinator-agent',
  name: 'Research Coordinator',
  instructions: `You are a network of researchers and writers.
    - Use researchAgent for gathering facts
    - Use writingAgent for producing final content
    - Use weatherTool for current weather data
    - Use activityPlannerWorkflow for location-based planning
    
    Always produce comprehensive, well-structured responses.`,
  model: openai('gpt-5'),
  
  // Available primitives
  agents: { researchAgent, writingAgent },
  workflows: { activityPlannerWorkflow },
  tools: { weatherTool },
  
  // Network requires memory
  memory: new Memory({
    storage: new LibSQLStore({ id: 'network-store', url: 'file:../network.db' }),
  }),
});
```

このネットワークにクエリを送ると、コーディネーターがリクエストを分析してルーティングする：
- 「Xについての事実が知りたい」→ リサーチエージェントが起動
- 「シアトルでの週末を計画して」→ アクティビティプランナーワークフローが実行
- 「Yについてのレポートを書いて」→ ライティングエージェントが対応

このパターンは、すべてを単一のメガエージェントに詰め込むよりスケールする。専門化されたエージェントは焦点を絞った専門性を発揮し、コーディネーターがルーティングを担当する。各パーツは得意なことをやる。

## まとめ

本番のAIシステムに必要なのはアーキテクチャであり、プロンプトだけではない。あなたが構築しているのは分散システムであり、そのノードの一部がたまたまLLMなのだ。

ワークフローは、正確に実行しなければならない処理に保証を与える。メモリは、トークン予算を燃やさずにコンテキストを提供する。エージェントネットワークは、シンプルなパーツから複雑さを構成できる。

どれも華やかではない。しかし「完全自律エージェント」が本番で失敗するのを何度も見てきた後、私は刺激的な不安定さよりも退屈な信頼性を重んじるようになった。

状況によりけりだが、私の経験では、実際にリリースされて動き続けるシステムは、LLMを万能の魔法の箱としてではなく、より大きなアーキテクチャの構成要素として扱うものだ。
