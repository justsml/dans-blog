LLMには奇妙な性質があります。ニュアンスを理解するのは非常に得意ですが、「レシピ」に従うのは苦手なのです。GPT-4に曖昧な問題を与えれば、可能性を論理的に推論してくれます。しかし、正確な手順のシーケンスを与えると、ステップ5の方が「関連性が高いと感じた」という理由でステップ3をスキップしてしまうことがあります。

これはモデルのバグではありません。決定論的な問題を解決しようとする確率論的システムの根本的な特性なのです。

多くのチームがこの不一致に苦労しているのを見てきました。カスタマー返金処理のためにエージェントを構築し、ダース単位のツールを与え、ビジネスプロセスを確実に実行することを期待します。うまくいくこともあれば、実際には行われていない承認を「捏造（ハルシネーション）」したり、同じ情報を3回も聞き返して立ち往生したりすることもあります。

解決策は、より良いプロンプトではありません。いつLLMに「思考」させるのをやめ、「指示に従わせる」べきかを知ることです。

## 決定論が創造性に勝る時

サポートチケットを処理する際に何が起こるか考えてみましょう。現実世界のビジネスロジックは次のようなものです。

1. データベースからチケットの詳細を取得する
2. ユーザーが返金対象かどうかを確認する（ポリシー規則）
3. 取引が存在し、まだ返金されていないことを確認する
4. 返金額を計算する
5. 支払いの取り消しを処理する
6. チケットのステータスを更新する
7. 確認メールを送信する

これをツール呼び出し（tool-calling）の演習としてLLMに丸投げすることもできます。しかし、私の経験上、それはトラブルを招くだけです。モデルはステップ2と3を「基本的には同じこと」だと判断して一方をスキップするかもしれません。あるいは、ユーザーが怒っているように見えたので、資格確認の前に返金を処理してしまうかもしれません。

ワークフローは、まさにこのようなシナリオのために存在します。刺激的ではありませんが、それがポイントなのです。

### お天気アクティビティプランナーの構築

パターンを示す具体的な例を挙げます。ここでは、「厳密で事実に基づいた気象データ」と「創造的なアクティビティの提案」を組み合わせる必要があります。天気の取得は決して創造的であってはなりませんが、提案は創造的であるべきです。

```typescript
// src/mastra/workflows/activity-planner.ts
import { createWorkflow, createStep } from '@mastra/core/workflows';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

// ステップ 1: 天気データを取得 (決定論的)
const fetchWeather = createStep({
  id: 'fetch-weather',
  description: 'Fetches weather forecast for a given city',
  inputSchema: z.object({
    city: z.string(),
  }),
  outputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  execute: async ({ inputData }) => {
    // ... (取得ロジック) ...
    const weather = await fetch(`https://api.open-meteo.com/v1/forecast?latitude=52.52&longitude=13.41&current=temperature_2m,weather_code&daily=precipitation_probability_mean`).then(r => r.json());
    
    return {
      location: inputData.city,
      temperature: weather.current.temperature_2m,
      conditions: getWeatherCondition(weather.current.weather_code),
      precipitationChance: weather.daily.precipitation_probability_mean[0],
    };
  },
});

// ステップ 2: エージェントがアクティビティを提案 (創造的)
const activityPlanner = new Agent({
  id: 'activity-planner-agent',
  name: 'Activity Planner',
  instructions: `You are a local activities expert. Based on weather conditions, suggest 3-5 appropriate activities.
    - For rain (>50% precipitation), prioritize indoor activities
    - For extreme temperatures, consider climate-appropriate options
    - Always include one adventurous and one relaxing option`,
  model: openai('gpt-5'),
});

const planActivities = createStep({
  id: 'plan-activities',
  description: 'Uses AI to suggest activities based on weather',
  inputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  outputSchema: z.object({
    activities: z.string(),
  }),
  execute: async ({ inputData }) => {
    const prompt = `Weather in ${inputData.location}: ${inputData.temperature}°C...`;
    const response = await activityPlanner.generate(prompt);
    return { activities: response.text };
  },
});

// パイプライン
export const activityPlannerWorkflow = createWorkflow({
  id: 'activity-planner',
  inputSchema: z.object({ city: z.string() }),
  outputSchema: z.object({ activities: z.string() }),
})
  .then(fetchWeather)
  .then(planActivities);

activityPlannerWorkflow.commit();
```

LLMは天気APIに直接触れることはありません。グラウンドトゥルース（真実のデータ）を入力として受け取り、その上で得意なこと、つまり「文脈に応じた提案」を行います。これを逆にしてエージェントに天気データを取得させると、実際には雨が降っているのに「晴れの予報」をでっち上げる事態がいずれ起こります。

**ワークフローを検討すべき場合:**
- 順番通りに実行しなければならない既知のステップがある
- 各段階での可観測性（ログ、メトリクス、タイミング）が必要
- 不安定な外部APIに対するリトライロジックが必要
- ビジネスルールを「解釈」させるのではなく、厳密に従わせる必要がある

## 誰も語らないコンテキストウィンドウの問題

よく見かけるパターンがあります。チャットボットを構築し、テスト中は完璧に動作します。しかし本番環境では、ユーザーとの会話が長くなるにつれ、ボットが突然迷走し始めます。

開発者がログを確認すると、リクエストのたびに会話履歴全体を送信していることに気づきます。47件ものメッセージすべてを。ほとんど無関係な情報のせいで、トークンとコンテキストスペースを浪費しているのです。

さらに悪いことに、研究者が「中間での喪失（Lost in the Middle）」と呼ぶ現象があります。関連する情報が長いコンテキストの中間に埋もれていると、モデルのパフォーマンスが低下するというものです。モデルは文字通り、木を見て森を見ずの状態になります。

会話履歴をすべて送ることは「安全」に感じられます。「すべての情報」をモデルに与えているからです。しかし、実際にはモデルが重要なことに集中するのを難しくしています。

### ワーキングメモリ vs 長期ストレージ

Mastraのメモリシステムは、その両方を提供します。「ワーキングメモリ」は最近のメッセージをコンテキストウィンドウ内に保持します。「セマンティックリコール」は、現在のクエリに関連がありそうな場合に、過去のメッセージを検索します。

```typescript
// src/mastra/agents/memory-agent.ts
import { Agent } from '@mastra/core/agent';
import { Memory } from '@mastra/memory';
import { LibSQLStore } from '@mastra/libsql';

export const memoryAgent = new Agent({
  id: 'memory-agent',
  name: 'Memory Agent',
  instructions: 'You are a helpful assistant with perfect recall of our conversations.',
  model: openai('gpt-5'),
  memory: new Memory({
    storage: new LibSQLStore({
      id: 'memory-agent-store',
      url: 'file:../mastra.db',
    }),
    options: {
      lastMessages: 20,  // 直近20メッセージをコンテキストに保持
      semanticRecall: {
        enabled: true,  // 埋め込み（embeddings）を使用して古い情報を探す
        topK: 5,
        threshold: 0.7,
      },
    },
  }),
});
```

これが実際にどう機能するか見てみましょう。ユーザーがこう尋ねます。「先月おすすめしてくれたイタリアンレストランは何だっけ？」

セマンティックリコールがない場合、エージェントは直近の20メッセージしか見えません。レストランの推奨は506件中の487件目のメッセージでした。もう消えています。エージェントは「その情報は持っていません」と答えます。

セマンティックリコールがある場合:
1. クエリが埋め込みベクトル化されます: `[0.234, -0.567, 0.891, ...]`
2. その埋め込みが過去のメッセージと比較されます
3. 487件目のメッセージ（「Trattoria Bellaがおすすめです。カルボナーラが絶品ですよ」）が0.89の類似度を記録します
4. そのメッセージが現在のコンテキストに注入されます
5. エージェントが回答します: 「Trattoria Bellaをおすすめしました。特にカルボナーラが印象的でしたね」

エージェントは完璧な記憶を持っているように見えながら、コンテキストウィンドウのほんの一部しか使用していません。これは単なる賢いエンジニアリングではなく、会話が数十件を超えると機能的に不可欠なものになります。

## エージェントネットワークによる調整

構造と柔軟性の両方が必要な場合があります。純粋なワークフローは硬直すぎます。純粋なエージェントは予測不能すぎます。

エージェントネットワークは、タスクに基づいてどの専門エージェントやワークフローを呼び出すかを決定する「コーディネーター」を提供します。AI機能のためのスマートなロードバランサーのようなものだと考えてください。

```typescript
export const coordinatorAgent = new Agent({
  id: 'coordinator-agent',
  name: 'Research Coordinator',
  instructions: `You are a network of researchers and writers.
    - Use researchAgent for gathering facts
    - Use writingAgent for producing final content
    - Use weatherTool for current weather data
    - Use activityPlannerWorkflow for location-based planning
    
    Always produce comprehensive, well-structured responses.`,
  model: openai('gpt-5'),
  
  // 利用可能なプリミティブ
  agents: { researchAgent, writingAgent },
  workflows: { activityPlannerWorkflow },
  tools: { weatherTool },
  
  // ネットワークにはメモリが必要
  memory: new Memory({
    storage: new LibSQLStore({ id: 'network-store', url: 'file:../network.db' }),
  }),
});
```

このネットワークにクエリを投げると、コーディネーターがリクエストを分析し、適切にルートを振り分けます。
- 「Xに関する事実が必要」→ リサーチエージェントを起動
- 「シアトルでの週末をプランニングして」→ アクティビティプランナーワークフローを実行
- 「Yに関するレポートを書いて」→ ライティングエージェントを従事

このパターンは、すべてを一つのメガエージェントに詰め込もうとするよりもはるかに拡張性があります。専門のエージェントは特定の分野に特化し、コーディネーターがルーティングを担当します。各パーツが得意なことを行うのです。

## まとめ

本番レベルのAIシステムには、プロンプトだけでなくアーキテクチャが必要です。あなたは、一部のノードがたまたまLLMである分散システムを構築しているのです。

ワークフローは、物事を正確に進める必要がある場合に保証を与えてくれます。メモリは、トークン予算を浪費することなく文脈を与えてくれます。エージェントネットワークは、単純な部品から複雑な構成を可能にします。

これらは決して派手な技術ではありません。しかし、多くの「完全自律型エージェント」が本番環境で失敗するのを見てきた結果、私は刺激的な予測不能さよりも、退屈なほどの信頼性を高く評価するようになりました。

LLMをすべてを解決する魔法の箱としてではなく、より大きなアーキテクチャの中のコンポーネントとして扱うシステムこそが、実際にリリースされ、動き続けるシステムなのです。
