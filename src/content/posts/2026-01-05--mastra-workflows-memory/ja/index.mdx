LLMには奇妙な特性がある。微妙なニュアンスの理解스는 뛰어남하지만、レイヤーを踏むのが苦手だ。GPT-4に曖昧な問題を与方法与えると可能性について推論してくれる。精密なステップの連鎖与方法与えると、「ステップ5の方がRelevanceが高かった」とステップ3をスキップする可能性がある。

これはモデルのバグではない。確率的システムが確定的問題を解決しようとしていることの根本的的特性だ。

チームがこのミスマッチに苦しむのを何度も見た。顧客返金対応エージェントを構築し、 десятокのツール与方法与えて、ビジネスプロセスを確実に実行することを期待する。時には完璧に動作する。時には存在しない承認を幻覚する。時には同じ情報を3回求め続けて立ち往生する。

解決策はより良いプロンプトではない。いつ「考える」ことをLLMに求めるのを止めて「従う」ことを始めるかを知ることだ。

## 決定論的がクリエイティブに勝る時

サポートチケットを処理する必要があると考えた時を考える。本物のビジネスロジックは以下のようになる：

1. データベースからチケット詳細を取得
2. ユーザーが返金対象かどうかを確認（ポリシールール）
3. 取引が存在し、まだ返金されていないことを確認
4. 返金額を計算
5. 支払い巻き戻しを処理
6. チケットステータスを更新
7. 確認メールを送信

これをツール呼び出しの問題としてLLMに渡すこともできる。私の経験では、それは麻烦をそれだけだ。モデルはステップ2とステップ3が「本質的に同じもの」と判断して片方をスキップする可能性がある。或者は、ユーザーが不快そうに見えた，所以在 eligibility をチェックする前に返金処理をされる可能性がある。

ワークフローはまさにこのシナリオのために存在する。 excitingではないが、それが必要なんだ。

###  Weather Activity Planner の構築

パターンを示す実用的な例してみよう。天気データはhardで客観的に取得し、アクティビティ提案はクリエイティブにする必要がある。天気取得は決してクリエイティブにならないべきだが、提案はそうであるべきだ。

```typescript
// src/mastra/workflows/activity-planner.ts
import { createWorkflow, createStep } from '@mastra/core/workflows';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

// Step 1: 天気データを取得（決定論的）
const fetchWeather = createStep({
  id: 'fetch-weather',
  description: '指定された都市の天気予報を取得',
  inputSchema: z.object({
    city: z.string(),
  }),
  outputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  execute: async ({ inputData }) => {
    // ... (取得ロジック) ...
    const weather = await fetch(`https://api.open-meteo.com/v1/forecast?latitude=52.52&longitude=13.41&current=temperature_2m,weather_code&daily=precipitation_probability_mean`).then(r => r.json());

    return {
      location: inputData.city,
      temperature: weather.current.temperature_2m,
      conditions: getWeatherCondition(weather.current.weather_code),
      precipitationChance: weather.daily.precipitation_probability_mean[0],
    };
  },
});

// Step 2: エージェントがアクティビティを提案（クリエイティブ）
const activityPlanner = new Agent({
  id: 'activity-planner-agent',
  name: 'Activity Planner',
  instructions: `あなたはローカルアクティビティの専門家です。天気状況に基づいて3〜5個の適切なアクティビティを提案してください。
    - 雨の可能性が高い場合（降水確率>50%）、室内アクティビティを優先
    - 極端な気温の場合、気候に応じたオプションを検討
    - 常常に1つの冒険的なオプションと1つのリラックスできるオプションを含める`,
  model: openai('gpt-5'),
});

const planActivities = createStep({
  id: 'plan-activities',
  description: 'AIを使用して天気に基づいてアクティビティを提案',
  inputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  outputSchema: z.object({
    activities: z.string(),
  }),
  execute: async ({ inputData }) => {
    const prompt = `${inputData.location}の天気: ${inputData.temperature}°C...`;
    const response = await activityPlanner.generate(prompt);
    return { activities: response.text };
  },
});

// パイプライン
export const activityPlannerWorkflow = createWorkflow({
  id: 'activity-planner',
  inputSchema: z.object({ city: z.string() }),
  outputSchema: z.object({ activities: z.string() }),
})
  .then(fetchWeather)
  .then(planActivities);

activityPlannerWorkflow.commit();
```

LLMは天気APIに決して触れない。GROUND-TRUTHデータを入力として受け取り、次に実際には得意としていること：文脈に応じた提案を行う。如果你反过来 让 エージェントに天気データを取得させると、最終的には実際には雨なのに晴れの予報を得ることになる。

**ワークフローを検討するタイミング：**
- 順番대로発生する必要がある既知のステップのシーケンスがある場合
- 各ステージで可観測性が必要な場合（ログ、メトリクス、タイミング）
- 不安定な外部APIにリトライロジックが必要な場合
- ビジネスルールが「解釈」されてはならず、厳密に.follow されなければならない場合

## 誰も語らないコンテキストウィンドウ問題

繰り返し見るパターンがある。誰かがチャットボットを構築する。テスト中は正常に動作する。運用に入ると、ユーザーの会話が長くなり、突然ボットが迷子になる。

開発者はログを見て、すべてのお 请求で会話履歴全体を送っていることに気づく。47件のメッセージすべて。Mostly irrelevant な情報のためにトークンとコンテキスト空間を消費している。

さらに悪いことに、研究者が「middleで迷う」と呼ぶ現象がある。関連する情報が長いコンテキストの奥に埋もれていると、モデルのパフォーマンスが低下する。モデルは文字通り木を見て森を見 못하는。

会話履歴全体を送るのは安全に感じる。「すべての情報」をモデルに与えている。だが実際には、モデルが重要なことに集中更难くなっている。

###  working Memory とlong-term Storage

Mastraのメモリシステムは両方を提供する。working memoryは最近のメッセージをコンテキストウィンドウに保つ。semantic recallは、現在のクエリが関係しているように見える場合にHistoricalメッセージをsearchする。

```typescript
// src/mastra/agents/memory-agent.ts
import { Agent } from '@mastra/core/agent';
import { Memory } from '@mastra/memory';
import { LibSQLStore } from '@mastra/libsql';

export const memoryAgent = new Agent({
  id: 'memory-agent',
  name: 'Memory Agent',
  instructions: 'あなたは私たちの会話の完全な記憶を持つ有帮助なアシスタントです。',
  model: openai('gpt-5'),
  memory: new Memory({
    storage: new LibSQLStore({
      id: 'memory-agent-store',
      url: 'file:../mastra.db',
    }),
    options: {
      lastMessages: 20,  // 最後の20件のメッセージをコンテキストに保つ
      semanticRecall: {
        enabled: true,  // embeddingsを使用して古い情報を検索
        topK: 5,
        threshold: 0.7,
      },
    },
  }),
});
```

これが実践でどのように機能するかを示す。ユーザーが尋ねる：「先月推奨してくれたあのイタリアンレストラン何て言ったっけ？」

semantic recallなしでは、エージェントは最後の20件のメッセージだけを見る。レストランの推奨は506件中487件目だった。それは消えている。エージェントは「その情報を持っていません」と言う。

semantic recallあり：
1. クエリが埋め込まれる：`[0.234, -0.567, 0.891, ...]`
2. 埋め込みが過去のメッセージと比較される
3. メッセージ487（「Trattoria Bellaをお勧めします—他们的carbonaraは素晴らしい」）が0.89の類似度を獲得
4. そのメッセージが現在のコンテキストに注入される
5. エージェントが応答：「Trattoria Bellaをお勧めします。Their carbonaraが私の注目を引きました。」

エージェントはコンテキストウィンドウのごく一部しか使用しないながら、完全な記憶を持っているように見える。これは単なる巧妙なエンジニアリングではない— conversationが数十件のメッセージを 超える，就必须のものだ。

## エージェントネットワークを通じた調整

Sometimes both structure and flexibility are needed. Pure workflows are too rigid. Pure agents are too unpredictable.

エージェントネットワークはタスクに基づいて特別なエージェントまたはワークフローを呼び出すことを決めるディズ páticc coordinatorを提供する。AI capabilities用のsmart load balancer想象してほしい。

```typescript
export const coordinatorAgent = new Agent({
  id: 'coordinator-agent',
  name: 'Research Coordinator',
  instructions: `あなたは研究者 and ライターのネットワークです。
    - 事実収集には researchAgent を使用
    - 最終コンテンツの制作には writingAgent を使用
    - 現在の天気データには weatherTool を使用
    - 場所ベースの planning には activityPlannerWorkflow を使用

    常常に包括的で well-structured な応答を生成してください。`,
  model: openai('gpt-5'),

  // 利用可能なprimitive
  agents: { researchAgent, writingAgent },
  workflows: { activityPlannerWorkflow },
  tools: { weatherTool },

  // ネットワークにはメモリが必要
  memory: new Memory({
    storage: new LibSQLStore({ id: 'network-store', url: 'file:../network.db' }),
  }),
});
```

このネットワークにクエリすると、coordinatorがリクエストを分析して適切にルーティングする：
- 「Xについての事実が必要」→ research agentがトリガー
- 「シアトルで週末を計画」→ activity planner workflowが実行
- 「Yについてのレポートを書く」→ writing agentが起動

このパターンはすべてを1つの巨大的エージェントに押し込もうとするよりも 확장性が高い。特別なエージェントは焦点を当てた専門性を開発する。coordinatorがルーティングを処理する。各部分が得意なことを выполнять.

## まとめ

本物のproduction AIシステムにはプロンプトだけでなくアーキテクチャが必要だ。いくつかの 노드가たまたまLLMである分散システムを構築している。

ワークフローは物が正確に正しく発生する必要があるときに保証を与える。メモリはトークンバジェットを消費することなくコンテキストを与える。エージェントネットワークはより単純な部分から複雑さを構成することを可能にする。

これらはどれも glamorouse ではない。しかし、十分な「完全に自律的なエージェント」がproductionで失敗するのを見た後、 exciting な不安定性よりも退屈な信頼性を高く評価するようになった。

あなたの結果は異なる場合がありますが、私の経験では、実際にshipして稼働し続けたシステムは、LLMをすべてを解決するmagic boxとして扱うのではなく、より大きなアーキテクチャのコンポーネントとして扱うものが었습니다。
