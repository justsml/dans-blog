Gli LLM hanno questa strana proprietà: sono brillanti nel comprendere le sfumature ma terribili nel seguire le ricette. Dai a GPT-4 un problema vago e ragionerà sulle possibilità. Dagli una sequenza precisa di passaggi e potrebbe saltare il passaggio 3 perché il passaggio 5 "sembrava più rilevante."

Non è un bug nel modello. È una caratteristica fondamentale dei sistemi probabilistici che cercano di risolvere problemi deterministici.

Ho visto team che lottano con questa incongruenza. Costruiscono un agente per gestire rimborsi clienti, gli danno una dozzina di strumenti e si aspettano che esegua in modo affidabile un processo aziendale. A volte funziona perfettamente. A volte allucina approvazioni che non sono mai avvenute. A volte si blocca chiedendo le stesse informazioni tre volte.

La soluzione non sono prompt migliori. È sapere quando smettere di chiedere all'LLM di "pensare" e iniziare a dirgli di "obbedire."

---

## Quando il Determinismo Vince sulla Creatività

Pensa a cosa succede quando devi elaborare un ticket di supporto. La logica di business reale assomiglia a qualcosa del genere:

1. Recupera i dettagli del ticket dal database
2. Verifica se l'utente ha diritto a un rimborso (regole di policy)
3. Verifica che la transazione esista e non sia già stata rimborsata
4. Calcola l'importo del rimborso
5. Elabora la reversibilità del pagamento
6. Aggiorna lo stato del ticket
7. Invia email di conferma

Potresti affidare questo a un LLM come esercizio di tool-calling. Nella mia esperienza, chiedere questo significa cercarsi guai. Il modello potrebbe decidere che i passaggi 2 e 3 sono "praticamente la stessa cosa" e saltarne uno. Oppure potrebbe elaborare il rimborso prima di verificare l'idoneità perché l'utente sembrava arrabbiato.

I workflow esistono esattamente per questo scenario. Non sono emozionanti, ma questo è il punto.

### Costruire un Activity Planner Meteo

Ecco un esempio pratico che mostra il pattern. Servono dati meteorologici hardcoded e fattuali abbinati a suggerimenti creativi per attività. Il recupero del meteo non deve mai essere creativo, ma i suggerimenti sì.

```typescript
// src/mastra/workflows/activity-planner.ts
import { createWorkflow, createStep } from '@mastra/core/workflows';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

// Step 1: Fetch weather data (Deterministic)
const fetchWeather = createStep({
  id: 'fetch-weather',
  description: 'Fetches weather forecast for a given city',
  inputSchema: z.object({
    city: z.string(),
  }),
  outputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  execute: async ({ inputData }) => {
    // ... (fetch logic) ...
    const weather = await fetch(`https://api.open-meteo.com/v1/forecast?latitude=52.52&longitude=13.41&current=temperature_2m,weather_code&daily=precipitation_probability_mean`).then(r => r.json());
    
    return {
      location: inputData.city,
      temperature: weather.current.temperature_2m,
      conditions: getWeatherCondition(weather.current.weather_code),
      precipitationChance: weather.daily.precipitation_probability_mean[0],
    };
  },
});

// Step 2: Agent suggests activities (Creative)
const activityPlanner = new Agent({
  id: 'activity-planner-agent',
  name: 'Activity Planner',
  instructions: `You are a local activities expert. Based on weather conditions, suggest 3-5 appropriate activities.
    - For rain (>50% precipitation), prioritize indoor activities
    - For extreme temperatures, consider climate-appropriate options
    - Always include one adventurous and one relaxing option`,
  model: openai('gpt-5'),
});

const planActivities = createStep({
  id: 'plan-activities',
  description: 'Uses AI to suggest activities based on weather',
  inputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  outputSchema: z.object({
    activities: z.string(),
  }),
  execute: async ({ inputData }) => {
    const prompt = `Weather in ${inputData.location}: ${inputData.temperature}°C...`;
    const response = await activityPlanner.generate(prompt);
    return { activities: response.text };
  },
});

// The Pipeline
export const activityPlannerWorkflow = createWorkflow({
  id: 'activity-planner',
  inputSchema: z.object({ city: z.string() }),
  outputSchema: z.object({ activities: z.string() }),
})
  .then(fetchWeather)
  .then(planActivities);

activityPlannerWorkflow.commit();
```

L'LLM non tocca mai l'API meteo. Riceve dati verificati come input, poi fa quello che effettivamente sa fare bene: fare suggerimenti contestuali. Se giri questo e lasci che l'agente recuperi i dati meteorologici, alla fine otterrai un pronostico soleggiato quando in realtà piove.

**Quando considerare i workflow:**
- Hai una sequenza nota di passaggi che devono accadere in ordine
- Hai bisogno di osservabilità in ogni fase (log, metriche, timing)
- Hai bisogno di logica di retry per API esterne instabili
- Le regole di business non possono essere "interpretate" - devono essere seguite esattamente

## Il Problema del Context Window di cui Nessuno Parla

C'è questo pattern che continuo a vedere. Qualcuno costruisce un chatbot. Funziona benissimo durante i test. Poi in produzione, gli utenti hanno conversazioni più lunghe e improvvisamente il bot si perde.

Lo sviluppatore guarda i log e si rende conto che sta inviando l'intera cronologia della conversazione con ogni richiesta. Tutti i 47 messaggi. Sta bruciando token e spazio nel context per informazioni che sonomostly irrilevanti.

Peggio ancora, c'è un fenomeno che i ricercatori chiamano "lost in the middle" dove i modelli performano peggio quando le informazioni rilevanti sono sepolte in un contesto lungo. Il modello letteralmente non riesce a vedere il bosco per gli alberi.

Inviare la cronologia completa della conversazione sembra sicuro. Stai dando al modello "tutte le informazioni". Ma in realtà stai rendendo più difficile per il modello concentrarsi su ciò che conta.

### Memoria di Lavoro vs. Archiviazione a Lungo Termine

Il sistema di memoria di Mastra ti fornisce entrambi. La memoria di lavoro mantiene i messaggi recenti nel context window. Il richiamo semantico cerca nei messaggi storici quando la query corrente sembra correlata.

```typescript
// src/mastra/agents/memory-agent.ts
import { Agent } from '@mastra/core/agent';
import { Memory } from '@mastra/memory';
import { LibSQLStore } from '@mastra/libsql';

export const memoryAgent = new Agent({
  id: 'memory-agent',
  name: 'Memory Agent',
  instructions: 'You are a helpful assistant with perfect recall of our conversations.',
  model: openai('gpt-5'),
  memory: new Memory({
    storage: new LibSQLStore({
      id: 'memory-agent-store',
      url: 'file:../mastra.db',
    }),
    options: {
      lastMessages: 20,  // Keep last 20 messages in context
      semanticRecall: {
        enabled: true,  // Use embeddings to find old stuff
        topK: 5,
        threshold: 0.7,
      },
    },
  }),
});
```

Ecco come questo si verifica nella pratica. Un utente chiede: "Quale era quel ristorante italiano che mi hai consigliato il mese scorso?"

Senza richiamo semantico, l'agente vede gli ultimi 20 messaggi. Il consiglio del ristorante era il messaggio 487 di 506. È andato perduto. L'agente dice "Non ho questa informazione."

Con il richiamo semantico:
1. La query viene embedded: `[0.234, -0.567, 0.891, ...]`
2. L'embedding viene comparato con i messaggi storici
3. Il messaggio 487 ("Consiglierei Trattoria Bella - il loro carbonara è incredibile") ottiene un punteggio di similarità di 0.89
4. Quel messaggio viene iniettato nel contesto corrente
5. L'agente risponde: "Ho consigliato Trattoria Bella. Il loro carbonara è quello che ha attirato la mia attenzione."

L'agente sembra avere una memoria perfetta mentre usa solo una frazione del context window. Non è solo ingegneria intelligente - è funzionalmente necessario una volta che le conversazioni si estendono oltre qualche decina di messaggi.

## Coordinazione attraverso Reti di Agenti

A volte hai bisogno sia di struttura che di flessibilità. I workflow puri sono troppo rigidi. Gli agenti puri sono troppo imprevedibili.

Le reti di agenti ti danno un coordinatore che decide quale agente specializzato o workflow invocare basandosi sul task. Pensalo come un load balancer intelligente per le capacità AI.

```typescript
export const coordinatorAgent = new Agent({
  id: 'coordinator-agent',
  name: 'Research Coordinator',
  instructions: `You are a network of researchers and writers.
    - Use researchAgent for gathering facts
    - Use writingAgent for producing final content
    - Use weatherTool for current weather data
    - Use activityPlannerWorkflow for location-based planning
    
    Always produce comprehensive, well-structured responses.`,
  model: openai('gpt-5'),
  
  // Available primitives
  agents: { researchAgent, writingAgent },
  workflows: { activityPlannerWorkflow },
  tools: { weatherTool },
  
  // Network requires memory
  memory: new Memory({
    storage: new LibSQLStore({ id: 'network-store', url: 'file:../network.db' }),
  }),
});
```

Quando fai una query a questa rete, il coordinatore analizza la richiesta e instrada di conseguenza:
- "Ho bisogno di fatti su X" triggera l'agente di ricerca
- "Pianifica un weekend a Seattle" esegue il workflow dell activity planner
- "Scrivi un report su Y" impegna l'agente di scrittura

Questo pattern scala meglio che cercare di infilare tutto in un singolo mega-agente. Gli agenti specializzati sviluppano competenze focalizzate. Il coordinatore gestisce l'instradamento. Ogni componente fa ciò che sa fare bene.

## Mettere Tutto Insieme

I sistemi AI di produzione reali hanno bisogno di architettura, non solo di prompt. Stai costruendo sistemi distribuiti dove alcuni nodi sono casualmente LLM.

I workflow ti danno garanzie quando hai bisogno che le cose accadano esattamente bene. La memoria ti dà contesto senza bruciare il tuo budget di token. Le reti di agenti ti permettono di comporre la complessità da parti più semplici.

Niente di tutto questo è glamour. Ma dopo aver guardato abbastanza "agenti completamente autonomi" fallire in produzione, ho imparato ad apprezzare l'affidabilità noiosa rispetto all'imprevedibilità eccitante.

I tuoi risultati possono variare, ma nella mia esperienza, i sistemi che effettivamente vengono spediti e continuano a funzionare sono quelli che trattano gli LLM come componenti in un'architettura più ampia piuttosto che scatole magiche che risolvono tutto.
