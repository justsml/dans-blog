LLMs की एक अजीब विशेषता है: वे बारीकियों को समझने में माहिर हैं लेकिन रेसिपी फॉलो करने में बुरे। GPT-4 को एक अस्पष्ट समस्या दें और यह संभावनाओं के बारे में तर्क करेगा। इसे चरणों का एक सटीक क्रम दें, और यह चरण 3 को छोड़ सकता है क्योंकि चरण 5 "अधिक प्रासंगिक लगा"।

यह मॉडल में बग नहीं है। यह संभाव्यता प्रणालियों का गैर-संभावित समस्याओं को हल करने का एक मौलिक गुण है।

मैंने टीमों को इस मismatch से जूझते देखा है। वे एक agent बनाते हैं ग्राहक refunds संभालने के लिए, इसे एक दर्जन tools देते हैं, और एक व्यावसायिक प्रक्रिया को विश्वसनीय रूप से निष्पादित करने की उम्मीद करते हैं। कभी-कभी यह पूरी तरह से काम करता है। कभी-कभी यह उन अनुमोदनों को hallucinations करता है जो कभी नहीं हुए। कभी-कभी यह तीन बार एक ही जानकारी मांगते हुए फंस जाता है।

समाधान बेहतर prompts नहीं है। यह जानना है कि LLM से "think" करने के लिए कहना कब बंद करें और "obey" कहना कब शुरू करें।

## जब Deterministic Creative को हराता है

सोचें कि जब आपको एक support ticket प्रोसेस करने की जरूरत होती है। वास्तविक दुनिया का व्यावसायिक तर्क कुछ इस तरह दिखता है:

1. डेटाबेस से ticket विवरण लाएं
2. जांचें कि उपयोगकर्ता refund के लिए पात्र है या नहीं (policy rules)
3. सत्यापित करें कि लेनदेन मौजूद है और पहले से refund नहीं हुआ है
4. Refund राशि की गणना करें
5. भुगतान रिवर्सल प्रोसेस करें
6. Ticket स्थिति अपडेट करें
7. पुष्टििा ईमेल भेजें

आप इसे एक tool-calling अभ्यास के रूप में LLM को सौंप सकते हैं। मेरे अनुभव में, यह परेशानी के लिए कहना है। मॉडल तय कर सकता है कि चरण 2 और 3 "अनिवार्य रूप से एक ही चीज हैं" और एक को छोड़ दें। या यह refund eligibility जांचने से पहले refund process कर सकता है क्योंकि उपयोगकर्ता परेशान लग रहा था।

Workflows exactly इस scenario के लिए मौजूद हैं। वे रोमांचक नहीं हैं, लेकिन यही बात है।

### एक Weather Activity Planner बनाना

यह पैटर्न दिखाने वाला एक व्यावहारिक उदाहरण है। हमें कठोर, तथ्यात्मक weather डेटा को रचनात्मक गतिविधि सुझावों के साथ जोड़ना होता है। Weather fetch कभी भी रचनात्मक नहीं होना चाहिए, लेकिन सुझाव होने चाहिए।

```typescript
// src/mastra/workflows/activity-planner.ts
import { createWorkflow, createStep } from '@mastra/core/workflows';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

// Step 1: Fetch weather data (Deterministic)
const fetchWeather = createStep({
  id: 'fetch-weather',
  description: 'Fetches weather forecast for a given city',
  inputSchema: z.object({
    city: z.string(),
  }),
  outputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  execute: async ({ inputData }) => {
    // ... (fetch logic) ...
    const weather = await fetch(`https://api.open-meteo.com/v1/forecast?latitude=52.52&longitude=13.41&current=temperature_2m,weather_code&daily=precipitation_probability_mean`).then(r => r.json());

    return {
      location: inputData.city,
      temperature: weather.current.temperature_2m,
      conditions: getWeatherCondition(weather.current.weather_code),
      precipitationChance: weather.daily.precipitation_probability_mean[0],
    };
  },
});

// Step 2: Agent suggests activities (Creative)
const activityPlanner = new Agent({
  id: 'activity-planner-agent',
  name: 'Activity Planner',
  instructions: `You are a local activities expert. Based on weather conditions, suggest 3-5 appropriate activities.
    - For rain (>50% precipitation), prioritize indoor activities
    - For extreme temperatures, consider climate-appropriate options
    - Always include one adventurous and one relaxing option`,
  model: openai('gpt-5'),
});

const planActivities = createStep({
  id: 'plan-activities',
  description: 'Uses AI to suggest activities based on weather',
  inputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  outputSchema: z.object({
    activities: z.string(),
  }),
  execute: async ({ inputData }) => {
    const prompt = `Weather in ${inputData.location}: ${inputData.temperature}°C...`;
    const response = await activityPlanner.generate(prompt);
    return { activities: response.text };
  },
});

// The Pipeline
export const activityPlannerWorkflow = createWorkflow({
  id: 'activity-planner',
  inputSchema: z.object({ city: z.string() }),
  outputSchema: z.object({ activities: z.string() }),
})
  .then(fetchWeather)
  .then(planActivities);

activityPlannerWorkflow.commit();
```

LLM कभी भी weather API को नहीं छूता। यह इनपुट के रूप में ground-truth डेटा प्राप्त करता है, फिर वह करता है जो यह वास्तव में अच्छा है: संदर्भगत सुझाव देना। यदि आप इसे पलट देते हैं और agent को weather data fetch करने देते हैं, तो आप eventually एक धूप का पूर्वानुमान पाएंगे जब वास्तव में बारिश हो रही हो।

**Workflows पर विचार करने का समय:**
- जब आपके पास चरणों का ज्ञात क्रम हो जो क्रम में होना चाहिए
- जब आपको प्रत्येक चरण पर observability चाहिए (logs, metrics, timing)
- जब आपको flaky external APIs के लिए retry logic चाहिए
- व्यावसायिक नियमों को "interpreted" नहीं किया जा सकता - उन्हें exactly follow किया जाना चाहिए

## Context Window समस्या जिसके बारे में कोई बात नहीं करता

एक पैटर्न है जो मैं देखता रहता हूं। कोई एक chatbot बनाता है। यह परीक्षण के दौरान बढ़िया काम करता है। फिर production में, उपयोगकर्ताओं के पास लंबी बातचीत होती है और अचानक bot लापता हो जाता है।

डेवलपर logs देखता है और realize करता है कि वे हर request के साथ पूरी बातचीत history भेज रहे हैं। सभी 47 संदेश। वे tokens और context space जला रहे हैं जानकारी के लिए जो ज्यादातर irrelevant है।

 worse, researchers इसे "lost in the middle" phenomenon कहते हैं जहां models तब खराब प्रदर्शन करते हैं जब relevant information लंबे context में दफन होती है। मॉडल literally जंगल नहीं देख सकता वृक्षों के लिए।

पूरी बातचीत history भेजना सुरक्षित लगता है। आप मॉडल को "सभी जानकारी" दे रहे हैं। लेकिन आप actually मॉडल के लिए जो मायने रखती है उस पर ध्यान केंद्रित करना कठिन बना रहे हैं।

### Working Memory बनाम Long-Term Storage

Mastra की memory system आपको दोनों देती है। Working memory context window में recent संदेश रखती है। Semantic recall historical संदेशों को खोजती है जब current query संबंधित लगती है।

```typescript
// src/mastra/agents/memory-agent.ts
import { Agent } from '@mastra/core/agent';
import { Memory } from '@mastra/memory';
import { LibSQLStore } from '@mastra/libsql';

export const memoryAgent = new Agent({
  id: 'memory-agent',
  name: 'Memory Agent',
  instructions: 'You are a helpful assistant with perfect recall of our conversations.',
  model: openai('gpt-5'),
  memory: new Memory({
    storage: new LibSQLStore({
      id: 'memory-agent-store',
      url: 'file:../mastra.db',
    }),
    options: {
      lastMessages: 20,  // Keep last 20 messages in context
      semanticRecall: {
        enabled: true,  // Use embeddings to find old stuff
        topK: 5,
        threshold: 0.7,
      },
    },
  }),
});
```

यहां बताया गया है कि यह व्यवहार में कैसे काम करता है। एक उपयोगकर्ता पूछता है: "पिछले महीने आपने वह Italian restaurant क्या recommend किया था?"

बिना semantic recall के, agent последние 20 संदेश देखता है। Restaurant recommendation संदेश 487 था 506 में से। यह चला गया। Agent कहता है "मेरे पास वह जानकारी नहीं है।"

Semantic recall के साथ:
1. Query embedded होती है: `[0.234, -0.567, 0.891, ...]`
2. Embedding historical संदेशों के विरुद्ध तुलना की जाती है
3. संदेश 487 ("मैं Trattoria Bella recommend करूंगा - उनका carbonara incredible है") 0.89 similarity स्कोर करता है
4. वह संदेश current context में injected होता है
5. Agent जवाब देता है: "मैंने Trattoria Bella recommend किया था। उनका carbonara ही वह था जिसने मेरा ध्यान खींचा।"

Agent को perfect memory जैसी प्रतीत होती है जबकि context window का केवल एक अंश उपयोग कर रहा है। यह केवल clever engineering नहीं है - यह functionally necessary है एक बार जब बातचीत कुछ दर्जन संदेशों से आगे बढ़ती है।

## Agent Networks के माध्यम से समन्वय

कभी-कभी आपको both structure और flexibility चाहिए। Pure workflows बहुत rigid हैं। Pure agents बहुत unpredictable हैं।

Agent networks आपको एक coordinator देती हैं जो task के आधार पर decide करता है कि कौन से specialized agent या workflow को invoke करना है। इसे AI क्षमताओं के लिए एक smart load balancer के रूप में think करें।

```typescript
export const coordinatorAgent = new Agent({
  id: 'coordinator-agent',
  name: 'Research Coordinator',
  instructions: `You are a network of researchers and writers.
    - Use researchAgent for gathering facts
    - Use writingAgent for producing final content
    - Use weatherTool for current weather data
    - Use activityPlannerWorkflow for location-based planning

    Always produce comprehensive, well-structured responses.`,
  model: openai('gpt-5'),

  // Available primitives
  agents: { researchAgent, writingAgent },
  workflows: { activityPlannerWorkflow },
  tools: { weatherTool },

  // Network requires memory
  memory: new Memory({
    storage: new LibSQLStore({ id: 'network-store', url: 'file:../network.db' }),
  }),
});
```

जब आप इस network से query करते हैं, तो coordinator request का विश्लेषण करता है और accordingly route करता है:
- "मुझे X के बारे में तथ्य चाहिए" research agent trigger करता है
- "Seattle में सप्ताहांत की योजना बनाएं" activity planner workflow चलाता है
- "Y पर रिपोर्ट लिखें" writing agent engaged करता है

यह पैटर्न एक single mega-agent में सब कुछ crammed करने की कोशिश करने से बेहतर scale होता है। Specialized agents focused expertise विकसित करते हैं। Coordinator routing handle करता है। प्रत्येक भाग वह करता है जो यह अच्छा है।

## सब को एक साथ रखना

वास्तविक production AI systems को architecture चाहिए, सिर्फ prompts नहीं। आप distributed systems बना रहे हैं जहां कुछ nodes happen to be LLMs।

Workflows आपको guarantees देती हैं जब आपको चीजों को exactly सही होने की जरूरत होती है। Memory आपको context देती है without your token budget जलाना। Agent networks आपको simpler parts से complexity compose करने देती हैं।

इनमें से कोई भी glamorous नहीं है। लेकिन production में पर्याप्त "fully autonomous agents" fail होते देखने के बाद, मैं exciting unpredictability पर boring reliability को appreciate करना सीख गया हूं।

आपका mileage may vary, लेकिन मेरे अनुभव में, वे systems जो actually ship करते हैं और running रहते हैं, वे हैं जो LLMs को एक बड़े architecture में components के रूप में treat करते हैं rather than magic boxes जो सब कुछ solve करते हैं।
