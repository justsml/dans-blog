LLMs में एक अजीब विशेषता होती है: वे बारीकियों को समझने में शानदार होते हैं लेकिन निर्देशों का पालन करने में खराब। GPT-4 को एक अस्पष्ट समस्या दें और वह संभावनाओं पर विचार करेगा। उसे चरणों का एक सटीक क्रम दें, और हो सकता है कि वह चरण 3 को छोड़ दे क्योंकि उसे चरण 5 "अधिक प्रासंगिक" लगा।

यह मॉडल में कोई बग नहीं है। यह नियतात्मक (deterministic) समस्याओं को हल करने की कोशिश करने वाले संभाव्यतावादी (probabilistic) प्रणालियों की एक मौलिक विशेषता है।

मैंने टीमों को इस असंतुलन से जूझते देखा है। वे कस्टमर रिफंड को संभालने के लिए एक एजेंट बनाते हैं, उसे एक दर्जन टूल्स देते हैं, और उम्मीद करते हैं कि वह मज़बूती से एक व्यावसायिक प्रक्रिया को निष्पादित करेगा। कभी-कभी यह पूरी तरह से काम करता है। कभी-कभी यह उन अनुमोदनों (approvals) की कल्पना (hallucinate) करता है जो कभी हुए ही नहीं। कभी-कभी यह एक ही जानकारी को तीन बार माँगने में फंस जाता है।

समाधान बेहतर प्रॉम्प्ट नहीं है। यह जानना है कि कब LLM से "सोचने" के लिए कहना बंद करना है और कब उसे "आज्ञा मानने" के लिए कहना शुरू करना है।

## जब नियतात्मकता (Deterministic) रचनात्मकता (Creative) से बेहतर होती है

सोचिए कि तब क्या होता है जब आपको एक सपोर्ट टिकट को प्रोसेस करने की आवश्यकता होती है। वास्तविक दुनिया का बिजनेस लॉजिक कुछ इस तरह दिखता है:

1. डेटाबेस से टिकट विवरण प्राप्त करें
2. जांचें कि क्या उपयोगकर्ता रिफंड के लिए पात्र है (पॉलिसी नियम)
3. सत्यापित करें कि ट्रांजेक्शन मौजूद है और पहले रिफंड नहीं किया गया है
4. रिफंड राशि की गणना करें
5. पेमेंट रिवर्सल को प्रोसेस करें
6. टिकट की स्थिति अपडेट करें
7. पुष्टिकरण ईमेल भेजें

आप इसे टूल-कॉलिंग अभ्यास के रूप में LLM को सौंप सकते हैं। मेरे अनुभव में, यह मुसीबत को न्योता देना है। मॉडल यह तय कर सकता है कि चरण 2 और 3 "मूल रूप से एक ही बात" हैं और एक को छोड़ सकता है। या वह पात्रता की जांच करने से पहले रिफंड प्रोसेस कर सकता है क्योंकि उपयोगकर्ता परेशान लग रहा था।

वर्कफ़्लो (Workflows) बिल्कुल इसी परिदृश्य के लिए मौजूद हैं। वे रोमांचक नहीं हैं, लेकिन यही उनका मुख्य बिंदु है।

### एक मौसम गतिविधि योजनाकार (Weather Activity Planner) बनाना

यहाँ एक व्यावहारिक उदाहरण है जो इस पैटर्न को दिखाता है। हमें रचनात्मक गतिविधि सुझावों के साथ कठिन, तथ्यात्मक मौसम डेटा की आवश्यकता है। मौसम की जानकारी प्राप्त करना कभी भी रचनात्मक नहीं होना चाहिए, लेकिन सुझाव होने चाहिए।

```typescript
// src/mastra/workflows/activity-planner.ts
import { createWorkflow, createStep } from '@mastra/core/workflows';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

// Step 1: Fetch weather data (Deterministic)
const fetchWeather = createStep({
  id: 'fetch-weather',
  description: 'Fetches weather forecast for a given city',
  inputSchema: z.object({
    city: z.string(),
  }),
  outputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  execute: async ({ inputData }) => {
    // ... (fetch logic) ...
    const weather = await fetch(`https://api.open-meteo.com/v1/forecast?latitude=52.52&longitude=13.41&current=temperature_2m,weather_code&daily=precipitation_probability_mean`).then(r => r.json());
    
    return {
      location: inputData.city,
      temperature: weather.current.temperature_2m,
      conditions: getWeatherCondition(weather.current.weather_code),
      precipitationChance: weather.daily.precipitation_probability_mean[0],
    };
  },
});

// Step 2: Agent suggests activities (Creative)
const activityPlanner = new Agent({
  id: 'activity-planner-agent',
  name: 'Activity Planner',
  instructions: `You are a local activities expert. Based on weather conditions, suggest 3-5 appropriate activities.
    - For rain (>50% precipitation), prioritize indoor activities
    - For extreme temperatures, consider climate-appropriate options
    - Always include one adventurous and one relaxing option`,
  model: openai('gpt-5'),
});

const planActivities = createStep({
  id: 'plan-activities',
  description: 'Uses AI to suggest activities based on weather',
  inputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  outputSchema: z.object({
    activities: z.string(),
  }),
  execute: async ({ inputData }) => {
    const prompt = `Weather in ${inputData.location}: ${inputData.temperature}°C...`;
    const response = await activityPlanner.generate(prompt);
    return { activities: response.text };
  },
});

// The Pipeline
export const activityPlannerWorkflow = createWorkflow({
  id: 'activity-planner',
  inputSchema: z.object({ city: z.string() }),
  outputSchema: z.object({ activities: z.string() }),
})
  .then(fetchWeather)
  .then(planActivities);

activityPlannerWorkflow.commit();
```

LLM कभी भी मौसम API को स्पर्श नहीं करता है। इसे इनपुट के रूप में ठोस-सत्य (ground-truth) डेटा मिलता है, फिर यह वह करता है जो यह वास्तव में अच्छा है: संदर्भ के अनुसार सुझाव देना। यदि आप इसे उलट देते हैं और एजेंट को मौसम डेटा लाने देते हैं, तो अंततः आपको तब भी धूप का पूर्वानुमान मिल सकता है जब वास्तव में बारिश हो रही हो।

**वर्कफ़्लो पर कब विचार करें:**
- आपके पास चरणों का एक ज्ञात क्रम है जो क्रमबद्ध तरीके से होना चाहिए
- आपको प्रत्येक चरण में अवलोकन क्षमता (लॉग, मेट्रिक्स, समय) की आवश्यकता है
- आपको अविश्वसनीय बाहरी API के लिए पुन: प्रयास (retry) लॉजिक की आवश्यकता है
- व्यावसायिक नियमों की "व्याख्या" नहीं की जा सकती - उनका सटीक पालन किया जाना चाहिए

## संदर्भ विंडो (Context Window) की समस्या जिसके बारे में कोई बात नहीं करता

एक पैटर्न है जिसे मैं अक्सर देखता हूँ। कोई एक चैटबॉट बनाता है। टेस्टिंग के दौरान यह बहुत अच्छा काम करता है। फिर प्रोडक्शन में, उपयोगकर्ता लंबी बातचीत करते हैं और अचानक बॉट भटक जाता है।

डेवलपर लॉग्स देखता है और महसूस करता है कि वे हर अनुरोध के साथ पूरी बातचीत का इतिहास भेज रहे हैं। सभी 47 संदेश। वे उन जानकारियों के लिए टोकन और संदर्भ स्थान जला रहे हैं जो ज्यादातर अप्रासंगिक हैं।

इससे भी बुरा यह है कि एक घटना है जिसे शोधकर्ता "बीच में खो जाना" (lost in the middle) कहते हैं, जहाँ मॉडल तब खराब प्रदर्शन करते हैं जब प्रासंगिक जानकारी एक लंबे संदर्भ में दबी होती है। मॉडल शाब्दिक रूप से "पेड़ों के कारण जंगल नहीं देख पाता"।

पूरी बातचीत का इतिहास भेजना सुरक्षित लगता है। आप मॉडल को "पूरी जानकारी" दे रहे हैं। लेकिन आप वास्तव में मॉडल के लिए उस चीज़ पर ध्यान केंद्रित करना कठिन बना रहे हैं जो मायने रखती है।

### वर्किंग मेमोरी बनाम लॉन्ग-टर्म स्टोरेज

Mastra का मेमोरी सिस्टम आपको दोनों देता है। वर्किंग मेमोरी हाल के संदेशों को संदर्भ विंडो में रखती है। सिमेंटिक रिकॉल ऐतिहासिक संदेशों को खोजता है जब वर्तमान क्वेरी संबंधित लगती है।

```typescript
// src/mastra/agents/memory-agent.ts
import { Agent } from '@mastra/core/agent';
import { Memory } from '@mastra/memory';
import { LibSQLStore } from '@mastra/libsql';

export const memoryAgent = new Agent({
  id: 'memory-agent',
  name: 'Memory Agent',
  instructions: 'You are a helpful assistant with perfect recall of our conversations.',
  model: openai('gpt-5'),
  memory: new Memory({
    storage: new LibSQLStore({
      id: 'memory-agent-store',
      url: 'file:../mastra.db',
    }),
    options: {
      lastMessages: 20,  // Keep last 20 messages in context
      semanticRecall: {
        enabled: true,  // Use embeddings to find old stuff
        topK: 5,
        threshold: 0.7,
      },
    },
  }),
});
```

यहाँ बताया गया है कि यह व्यवहार में कैसे काम करता है। एक उपयोगकर्ता पूछता है: "पिछले महीने आपने जिस इतालवी रेस्तरां की सिफारिश की थी, वह क्या था?"

सिमेंटिक रिकॉल के बिना, एजेंट पिछले 20 संदेश देखता है। रेस्तरां की सिफारिश 506 में से 487वां संदेश थी। वह चला गया है। एजेंट कहता है "मेरे पास वह जानकारी नहीं है।"

सिमेंटिक रिकॉल के साथ:
1. क्वेरी एम्बेड हो जाती है: `[0.234, -0.567, 0.891, ...]`
2. एम्बेडिंग की तुलना ऐतिहासिक संदेशों से की जाती है
3. संदेश 487 ("मैं Trattoria Bella की सिफारिश करूँगा - उनका कार्बोनेरा अविश्वसनीय है") का स्कोर 0.89 समानता है
4. वह संदेश वर्तमान संदर्भ में इंजेक्ट किया जाता है
5. एजेंट जवाब देता है: "मैंने Trattoria Bella की सिफारिश की थी। उनके कार्बोनेरा ने मेरा ध्यान खींचा था।"

एजेंट के पास पूर्ण स्मृति (perfect memory) प्रतीत होती है, जबकि वह संदर्भ विंडो के केवल एक अंश का उपयोग कर रहा है। यह केवल चतुर इंजीनियरिंग नहीं है - बातचीत के कुछ दर्जन संदेशों से आगे बढ़ने के बाद यह कार्यात्मक रूप से आवश्यक है।

## एजेंट नेटवर्क के माध्यम से समन्वय (Coordination)

कभी-कभी आपको संरचना और लचीलेपन दोनों की आवश्यकता होती है। शुद्ध वर्कफ़्लो बहुत कठोर होते हैं। शुद्ध एजेंट बहुत अप्रत्याशित होते हैं।

एजेंट नेटवर्क आपको एक समन्वयक (coordinator) देते हैं जो कार्य के आधार पर यह तय करता है कि किस विशेष एजेंट या वर्कफ़्लो को लागू करना है। इसे AI क्षमताओं के लिए एक स्मार्ट लोड बैलेंसर के रूप में सोचें।

```typescript
export const coordinatorAgent = new Agent({
  id: 'coordinator-agent',
  name: 'Research Coordinator',
  instructions: `You are a network of researchers and writers.
    - Use researchAgent for gathering facts
    - Use writingAgent for producing final content
    - Use weatherTool for current weather data
    - Use activityPlannerWorkflow for location-based planning
    
    Always produce comprehensive, well-structured responses.`,
  model: openai('gpt-5'),
  
  // Available primitives
  agents: { researchAgent, writingAgent },
  workflows: { activityPlannerWorkflow },
  tools: { weatherTool },
  
  // Network requires memory
  memory: new Memory({
    storage: new LibSQLStore({ id: 'network-store', url: 'file:../network.db' }),
  }),
});
```

जब आप इस नेटवर्क से कोई सवाल पूछते हैं, तो समन्वयक अनुरोध का विश्लेषण करता है और तदनुसार रूट करता है:
- "मुझे X के बारे में तथ्यों की आवश्यकता है" रिसर्च एजेंट को ट्रिगर करता है
- "सिएटल में एक सप्ताहांत की योजना बनाएं" एक्टिविटी प्लानर वर्कफ़्लो चलाता है
- "Y पर एक रिपोर्ट लिखें" राइटिंग एजेंट को जोड़ता है

यह पैटर्न सब कुछ एक ही मेगा-एजेंट में ठूंसने की कोशिश करने से बेहतर स्केल करता है। विशिष्ट एजेंट केंद्रित विशेषज्ञता विकसित करते हैं। समन्वयक राउटिंग संभालता है। प्रत्येक हिस्सा वह करता है जिसमें वह अच्छा है।

## सब कुछ एक साथ लाना

वास्तविक प्रोडक्शन AI सिस्टम को आर्किटेक्चर की आवश्यकता होती है, न कि केवल प्रॉम्प्ट्स की। आप वितरित सिस्टम (distributed systems) बना रहे हैं जहाँ कुछ नोड्स संयोग से LLMs हैं।

वर्कफ़्लो आपको तब गारंटी देते हैं जब आपको चीजों को बिल्कुल सही तरीके से करने की आवश्यकता होती है। मेमोरी आपको अपने टोकन बजट को जलाए बिना संदर्भ देती है। एजेंट नेटवर्क आपको सरल भागों से जटिलता बनाने देते हैं।

इसमें से कुछ भी ग्लैमरस नहीं है। लेकिन पर्याप्त "पूर्णतः स्वायत्त एजेंटों" को प्रोडक्शन में विफल होते देखने के बाद, मैंने रोमांचक अप्रत्याशितता के बजाय उबाऊ विश्वसनीयता की सराहना करना सीख लिया है।

आपके अनुभव भिन्न हो सकते हैं, लेकिन मेरे अनुभव में, जो सिस्टम वास्तव में शिप होते हैं और चलते रहते हैं, वे वही हैं जो LLMs को एक बड़े आर्किटेक्चर में घटकों (components) के रूप में मानते हैं, न कि जादू के बक्से के रूप में जो सब कुछ हल कर देते हैं।
