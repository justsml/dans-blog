LLMs की एक अजीब विशेषता है: वे बारीकियों को समझने में शानदार हैं, लेकिन निर्देशों का पालन करने में बहुत खराब। GPT-4 को एक अस्पष्ट समस्या दो और वह संभावनाओं के बीच तर्क करेगा। लेकिन इसे सटीक कदमों का एक क्रम दो, और वह कदम 3 को छोड़ सकता है क्योंकि कदम 5 "अधिक प्रासंगिक" लगा।

यह मॉडल में कोई बग नहीं है। यह प्रायिकता-आधारित सिस्टम की एक बुनियादी विशेषता है जो नियमित समस्याओं को हल करने की कोशिश कर रहे हैं।

मैंने टीमों को इस बेमेल से जूझते हुए देखा है। वे कस्टमर रिफंड हैंडल करने के लिए एजेंट बनाते हैं, इसे दर्जन भर टूल देते हैं, और एक व्यापारिक प्रक्रिया को विश्वसनीय रूप से निष्पादित करने की उम्मीद करते हैं। कभी-कभी यह पूरी तरह से काम करता है। कभी-कभी यह ऐसे अप्रूवल का आभास करता है जो कभी हुए नहीं। कभी-कभी यह एक ही जानकारी तीन बार पूछते हुए फँस जाता है।

समाधान बेहतर प्रॉम्प्ट नहीं है। यह यह जानना है कि LLM से "सोचने" को कब रोकना है और इसे "अनुसरण करने" को कब कहना है।

---

## जब नियमित, रचनात्मक से बेहतर हो

सोचिए क्या होता है जब आपको एक सपोर्ट टिकट प्रोसेस करना है। वास्तविक व्यापारिक तर्क कुछ इस तरह दिखता है:

1. डेटाबेस से टिकट विवरण लाएं
2. जांचें कि उपयोगकर्ता रिफंड के लिए पात्र है (पॉलिसी नियम)
3. सत्यापित करें कि लेनदेन मौजूद है और पहले से रिफंड नहीं हुआ
4. रिफंड राशि की गणना करें
5. पेमेंट रिवर्सल प्रोसेस करें
6. टिकट स्टेटस अपडेट करें
7. पुष्टिकरण ईमेल भेजें

आप इसे LLM को टूल-कॉलिंग अभ्यास के रूप में दे सकते हैं। मेरे अनुभव में, यह परेशानी मांगना है। मॉडल कदम 2 और 3 को "लगभग एक ही बात" समझ सकता है और एक को छोड़ सकता है। या यह पात्रता जांचने से पहले रिफंड प्रोसेस कर सकता है क्योंकि उपयोगकर्ता दुखी लग रहा था।

वर्कफ़लो ठीक इसी परिदृश्य के लिए मौजूद हैं। वे रोमांचक नहीं हैं, लेकिन यही उनका उद्देश्य है।

### वेदर एक्टिविटी प्लानर बनाना

यह एक व्यावहारिक उदाहरण है जो पैटर्न दिखाता है। हमें कठिन, तथ्यात्मक मौसम डेटा की जरूरत है, जो रचनात्मक एक्टिविटी सुझावों के साथ जुड़ा हो। मौसम डेटा लाना कभी रचनात्मक नहीं होना चाहिए, लेकिन सुझाव रचनात्मक होने चाहिए।

```typescript
// src/mastra/workflows/activity-planner.ts
import { createWorkflow, createStep } from '@mastra/core/workflows';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

// Step 1: Fetch weather data (Deterministic)
const fetchWeather = createStep({
  id: 'fetch-weather',
  description: 'Fetches weather forecast for a given city',
  inputSchema: z.object({
    city: z.string(),
  }),
  outputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  execute: async ({ inputData }) => {
    // ... (fetch logic) ...
    const weather = await fetch(`https://api.open-meteo.com/v1/forecast?latitude=52.52&longitude=13.41&current=temperature_2m,weather_code&daily=precipitation_probability_mean`).then(r => r.json());
    
    return {
      location: inputData.city,
      temperature: weather.current.temperature_2m,
      conditions: getWeatherCondition(weather.current.weather_code),
      precipitationChance: weather.daily.precipitation_probability_mean[0],
    };
  },
});

// Step 2: Agent suggests activities (Creative)
const activityPlanner = new Agent({
  id: 'activity-planner-agent',
  name: 'Activity Planner',
  instructions: `You are a local activities expert. Based on weather conditions, suggest 3-5 appropriate activities.
    - For rain (>50% precipitation), prioritize indoor activities
    - For extreme temperatures, consider climate-appropriate options
    - Always include one adventurous and one relaxing option`,
  model: openai('gpt-5'),
});

const planActivities = createStep({
  id: 'plan-activities',
  description: 'Uses AI to suggest activities based on weather',
  inputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  outputSchema: z.object({
    activities: z.string(),
  }),
  execute: async ({ inputData }) => {
    const prompt = `Weather in ${inputData.location}: ${inputData.temperature}°C...`;
    const response = await activityPlanner.generate(prompt);
    return { activities: response.text };
  },
});

// The Pipeline
export const activityPlannerWorkflow = createWorkflow({
  id: 'activity-planner',
  inputSchema: z.object({ city: z.string() }),
  outputSchema: z.object({ activities: z.string() }),
})
  .then(fetchWeather)
  .then(planActivities);

activityPlannerWorkflow.commit();
```

LLLM कभी वेदर API को नहीं छुता। इसे इनपुट के रूप में ग्राउंड-ट्रुथ डेटा मिलता है, फिर वह वह करता है जो वह वास्तव में अच्छा करता है: संदर्भ-आधारित सुझाव देना। यदि आप इसे उल्टा करें और एजेंट को मौसम डेटा लाने दें, तो आपको अंततः जब वास्तव में बारिश हो रही हो, एक धूप का पूर्वानुमान मिलेगा।

**वर्कफ़लो कब विचार करें:**
- आपके पास कदमों का एक ज्ञात क्रम है जो क्रम में होना चाहिए
- आपको हर स्टेज पर ऑब्जर्वेबिलिटी चाहिए (लॉग, मेट्रिक्स, टाइमिंग)
- आपको अनियमित एक्सटर्नल API के लिए रिट्राई लॉजिक चाहिए
- व्यापारिक नियमों को "व्याख्या" नहीं किया जा सकता - उनका ठीक से पालन होना चाहिए

## कॉन्टेक्स्ट विंडो प्रॉब्लम जिसके बारे में कोई बात नहीं करता

यह एक पैटर्न है जो मैं बार-बार देखता है। कोई चैटबॉट बनाता है। टेस्टिंग में यह बहुत अच्छा काम करता है। फिर प्रोडक्शन में, उपयोगकर्ता लंबी बातचीत करते हैं और अचानक बॉट भटक जाता है।

डेवलपर लॉग देखता है और समझता है कि वह हर रिक्वेस्ट के साथ पूरी बातचीत का इतिहास भेज रहा है। सभी 47 मैसेज। वे टोकन और कॉन्टेक्स्ट स्पेस जला रहे हैं जानकारी के लिए जो ज्यादातर अप्रासंगिक है।

और भी बुरा, शोधकर्ता एक घटना को "लॉस्ट इन द मिडल" कहते हैं जहाँ मॉडल खराब प्रदर्शन करते हैं जब प्रासंगिक जानकारी लंबे कॉन्टेक्स्ट में दबी होती है। मॉडल सार देखने से विवरण में ही उलझ जाता है।

पूरा बातचीत इतिहास भेजना सुरक्षित लगता है। आप मॉडल को "सारी जानकारी" दे रहे हैं। लेकिन वास्तव में आप मॉडल के लिए महत्वपूर्ण बात पर फोकस करना और भी कठिन बना रहे हैं।

### वर्किंग मेमोरी बनाम लॉन्ग-टर्म स्टोरेज

Mastra की मेमोरी सिस्टम आपको दोनों देती है। वर्किंग मेमोरी हाल के मैसेज कॉन्टेक्स्ट विंडो में रखती है। सेमेंटिक रिकॉल ऐतिहासिक मैसेज से सर्च करता है जब वर्तमान क्वेरी संबंधित लगती है।

```typescript
// src/mastra/agents/memory-agent.ts
import { Agent } from '@mastra/core/agent';
import { Memory } from '@mastra/memory';
import { LibSQLStore } from '@mastra/libsql';

export const memoryAgent = new Agent({
  id: 'memory-agent',
  name: 'Memory Agent',
  instructions: 'You are a helpful assistant with perfect recall of our conversations.',
  model: openai('gpt-5'),
  memory: new Memory({
    storage: new LibSQLStore({
      id: 'memory-agent-store',
      url: 'file:../mastra.db',
    }),
    options: {
      lastMessages: 20,  // Keep last 20 messages in context
      semanticRecall: {
        enabled: true,  // Use embeddings to find old stuff
        topK: 5,
        threshold: 0.7,
      },
    },
  }),
});
```

यह प्रैक्टिस में कैसे काम करता है। एक उपयोगकर्ता पूछता है: "पिछले महीने आपने जो इतालवी रेस्टोरेंट सुझाया था, वो क्या था?"

सेमेंटिक रिकॉल के बिना, एजेंट को पिछले 20 मैसेज दिखते हैं। रेस्टोरेंट सुझाव 506 मैसेज में से मैसेज 487 था। वो गया। एजेंट कहता है "मेरे पास यह जानकारी नहीं है।"

सेमेंटिक रिकॉल के साथ:
1. क्वेरी एम्बेड होती है: `[0.234, -0.567, 0.891, ...]`
2. एम्बेडिंग की ऐतिहासिक मैसेज से तुलना होती है
3. मैसेज 487 ("मैं Trattoria Bella सुझाव दूंगा - उनकी कार्बोनारा अद्भुत है") 0.89 समानता स्कोर करता है
4. वो मैसेज वर्तमान कॉन्टेक्स्ट में इंजेक्ट होता है
5. एजेंट जवाब देता है: "मैंने Trattoria Bella सुझाया था। उनकी कार्बोनारा मेरी नजर में आई थी।"

एजेंट को पूर्ण मेमोरी दिखती है जबकि कॉन्टेक्स्ट विंडो का एक छोटा सा हिस्सा ही उपयोग होता है। यह सिर्फ चतुर इंजीनियरिंग नहीं है - यह प्रायोगिक रूप से आवश्यक है जब बातचीत कुछ दर्जन मैसेज से आगे बढ़ती है।

## एजेंट नेटवर्क से कोऑर्डिनेशन

कभी-कभी आपको दोनों - संरचना और लचीलापन - चाहिए। शुद्ध वर्कफ़लो बहुत कठोर हैं। शुद्ध एजेंट बहुत अनिश्चित हैं।

एजेंट नेटवर्क आपको एक कोऑर्डिनेटर देता है जो टास्क के आधार पर यह निर्णय करता है कि कौन सा विशेषज्ञ एजेंट या वर्कफ़लो आमंत्रित करना है। इसे AI क्षमताओं के लिए एक स्मार्ट लोड बैलेंसर समझिए।

```typescript
export const coordinatorAgent = new Agent({
  id: 'coordinator-agent',
  name: 'Research Coordinator',
  instructions: `You are a network of researchers and writers.
    - Use researchAgent for gathering facts
    - Use writingAgent for producing final content
    - Use weatherTool for current weather data
    - Use activityPlannerWorkflow for location-based planning
    
    Always produce comprehensive, well-structured responses.`,
  model: openai('gpt-5'),
  
  // Available primitives
  agents: { researchAgent, writingAgent },
  workflows: { activityPlannerWorkflow },
  tools: { weatherTool },
  
  // Network requires memory
  memory: new Memory({
    storage: new LibSQLStore({ id: 'network-store', url: 'file:../network.db' }),
  }),
});
```

जब आप इस नेटवर्क को क्वेरी करते हैं, कोऑर्डिनेटर रिक्वेस्ट का विश्लेषण करता है और उसी के अनुसार रूट करता है:
- "मुझे X के बारे में तथ्य चाहिए" रिसर्च एजेंट को ट्रिगर करता है
- "सिएटल में एक वीकेंड प्लान करो" एक्टिविटी प्लानर वर्कफ़लो चलाता है
- "Y पर एक रिपोर्ट लिखो" राइटिंग एजेंट को एंगेज करता है

यह पैटर्न हर चीज को एक सिंगल मेगा-एजेंट में भरने की कोशिश करने से बेहतर स्केल होता है। विशेषज्ञ एजेंट केंद्रित दक्षता विकसित करते हैं। कोऑर्डिनेटर रूटिंग हैंडल करता है। हर टुकड़ा वह करता है जो वह अच्छा करता है।

## सब को जोड़ना

वास्तविक प्रोडक्शन AI सिस्टम को आर्किटेक्चर चाहिए, सिर्फ प्रॉम्प्ट नहीं। आप वितरित सिस्टम बना रहे हैं जहाँ कुछ नोड LLM हैं।

वर्कफ़लो आपको गारंटी देते हैं जब आपको चीजों को ठीक से होना चाहिए। मेमोरी आपको कॉन्टेक्स्ट देती है बिना टोकन बजट जलाए। एजेंट नेटवर्क आपको सरल भागों से जटिलता रचने देते हैं।

इनमें से कुछ भी आकर्षक नहीं है। लेकिन काफी "फुली ऑटोनोमस एजेंट" प्रोडक्शन में फेल होने देखने के बाद, मैं रोमांचक अनिश्चितता से बोरिंग विश्वसनीयता को पसंद करने लगा है।

आपका अनुभव अलग हो सकता है, लेकिन मेरे अनुभव में, वे सिस्टम जो वास्तव में शिप होते हैं और चलते रहते हैं, वे हैं जो LLM को एक बड़ी आर्किटेक्चर में कंपोनेंट के रूप में ट्रीट करते हैं बजाय ऐसे मैजिक बॉक्स के जो सब कुछ सॉल्व करते हैं।
