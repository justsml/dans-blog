LLMs में यह अजीब गुण होता है: वे नuance को समझने में बेहतरीन होते हैं लेकिन नुस्खों का पालन करने में बेकार। GPT-4 को एक अस्पष्ट समस्या दें और यह संभावनाओं के माध्यम से तर्क करेगा। इसे सटीक चरणों की एक अनुक्रम दें, और यह चरण 3 को छोड़ सकता है क्योंकि चरण 5 "अधिक प्रासंगिक लगा।"

यह मॉडल में कोई बग नहीं है। यह संभाव्यता प्रणालियों की एक मौलिक विशेषता है जो निर्धारित समस्याओं को हल करने का प्रयास कर रही हैं।

मैंने टीमों को इस असमानता के साथ संघर्ष करते देखा है। वे ग्राहक धनवापसी को संभालने के लिए एक एजेंट बनाते हैं, इसे दर्जनों टूल देते हैं, और उम्मीद करते हैं कि यह एक व्यावसायिक प्रक्रिया को विश्वसनीय रूप से निष्पादित करेगा। कभी-कभी यह पूरी तरह से काम करता है। कभी-कभी यह उन अनुमोदनों का भ्रम पैदा करता है जो कभी नहीं हुए। कभी-कभी यह एक ही जानकारी तीन बार मांगने में फंस जाता है।

समाधान बेहतर प्रॉम्प्ट नहीं हैं। यह जानना है कि कब LLM से "सोचने" के लिए कहना बंद करें और "आज्ञा पालन" करने के लिए कहना शुरू करें।

## जब निर्धारित रचनात्मक को हराता है

सोचें कि जब आपको एक सहायता टिकट संसाधित करने की आवश्यकता होती है तो क्या होता है। वास्तविक दुनिया का व्यावसायिक तर्क कुछ इस तरह दिखता है:

1. डेटाबेस से टिकट विवरण प्राप्त करें
2. जांचें कि क्या उपयोगकर्ता धनवापसी के लिए पात्र है (नीति नियम)
3. सत्यापित करें कि लेनदेन मौजूद है और पहले से धनवापसी नहीं हुई है
4. धनवापसी राशि की गणना करें
5. भुगतान उलटी प्रक्रिया करें
6. टिकट स्थिति अपडेट करें
7. पुष्टि ईमेल भेजें

आप इसे LLM को एक टूल-कॉलिंग अभ्यास के रूप में सौंप सकते हैं। मेरे अनुभव में, यह मुसीबत मोल लेना है। मॉडल यह तय कर सकता है कि चरण 2 और 3 "मूल रूप से एक ही चीज़" हैं और एक को छोड़ दें। या यह पात्रता की जांच करने से पहले धनवापसी संसाधित कर सकता है क्योंकि उपयोगकर्ता परेशान लग रहा था।

वर्कफ़्लोज़ ठीक इसी परिदृश्य के लिए मौजूद हैं। वे रोमांचक नहीं हैं, लेकिन यही बात है।

### एक मौसम गतिविधि योजनाकार बनाना

यहाँ एक व्यावहारिक उदाहरण है जो पैटर्न दिखाता है। हमें कठोर, तथ्यात्मक मौसम डेटा की आवश्यकता है जो रचनात्मक गतिविधि सुझावों के साथ जोड़ा जाए। मौसम प्राप्ति कभी भी रचनात्मक नहीं होनी चाहिए, लेकिन सुझाव रचनात्मक होने चाहिए।

```typescript
// src/mastra/workflows/activity-planner.ts
import { createWorkflow, createStep } from '@mastra/core/workflows';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

// Step 1: Fetch weather data (Deterministic)
const fetchWeather = createStep({
  id: 'fetch-weather',
  description: 'Fetches weather forecast for a given city',
  inputSchema: z.object({
    city: z.string(),
  }),
  outputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  execute: async ({ inputData }) => {
    // ... (fetch logic) ...
    const weather = await fetch(`https://api.open-meteo.com/v1/forecast?latitude=52.52&longitude=13.41&current=temperature_2m,weather_code&daily=precipitation_probability_mean`).then(r => r.json());
    
    return {
      location: inputData.city,
      temperature: weather.current.temperature_2m,
      conditions: getWeatherCondition(weather.current.weather_code),
      precipitationChance: weather.daily.precipitation_probability_mean[0],
    };
  },
});

// Step 2: Agent suggests activities (Creative)
const activityPlanner = new Agent({
  id: 'activity-planner-agent',
  name: 'Activity Planner',
  instructions: `You are a local activities expert. Based on weather conditions, suggest 3-5 appropriate activities.
    - For rain (>50% precipitation), prioritize indoor activities
    - For extreme temperatures, consider climate-appropriate options
    - Always include one adventurous and one relaxing option`,
  model: openai('gpt-5'),
});

const planActivities = createStep({
  id: 'plan-activities',
  description: 'Uses AI to suggest activities based on weather',
  inputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  outputSchema: z.object({
    activities: z.string(),
  }),
  execute: async ({ inputData }) => {
    const prompt = `Weather in ${inputData.location}: ${inputData.temperature}°C...`;
    const response = await activityPlanner.generate(prompt);
    return { activities: response.text };
  },
});

// The Pipeline
export const activityPlannerWorkflow = createWorkflow({
  id: 'activity-planner',
  inputSchema: z.object({ city: z.string() }),
  outputSchema: z.object({ activities: z.string() }),
})
  .then(fetchWeather)
  .then(planActivities);

activityPlannerWorkflow.commit();
```

LLM कभी भी मौसम API को नहीं छूता है। इसे इनपुट के रूप में भूमि-सत्य डेटा मिलता है, फिर वह करता है जिसमें वह वास्तव में अच्छा है: संदर्भिक सुझाव देना। यदि आप इसे उल्टा करते हैं और एजेंट को मौसम डेटा प्राप्त करने देते हैं, तो आपको अंततः एक धूप वाला पूर्वानुमान मिलेगा जब वास्तव में बारिश हो रही हो।

**जब वर्कफ़्लोज़ पर विचार करें:**
- जब आपके पास चरणों का एक ज्ञात क्रम है जो क्रम में होना चाहिए
- जब आपको प्रत्येक चरण पर प्रेक्षणीयता की आवश्यकता हो (लॉग, मेट्रिक्स, समय)
- जब आपको अस्थिर बाहरी APIs के लिए पुनर्प्रयास तर्क की आवश्यकता हो
- जब व्यावसायिक नियमों को "व्याख्या" नहीं किया जा सकता - उन्हें ठीक उसी तरह से पालन करना चाहिए

## वह संदर्भ विंडो समस्या जिसके बारे में कोई नहीं बात करता

वहाँ एक पैटर्न है जो मैं बार-बार देखता हूँ। कोई एक चैटबॉट बनाता है। यह परीक्षण के दौरान बढ़िया काम करता है। फिर उत्पादन में, उपयोगकर्ताओं के पास लंबी बातचीत होती हैं और अचानक बॉट खो जाता है।

डेवलपर लॉग देखता है और महसूस करता है कि वे हर अनुरोध के साथ पूरी बातचीत का इतिहास भेज रहे हैं। सभी 47 संदेश। वे जानकारी के लिए टोकन और संदर्भ स्थान जला रहे हैं जो ज्यादातर अप्रासंगिक है।

इससे भी बदतर, शोधकर्ताओं द्वारा "बीच में खो जाना" कहा जाने वाला एक घटना है जहाँ मॉडल तब प्रदर्शन खराब करते हैं जब प्रासंगिक जानकारी एक लंबे संदर्भ में दबी होती है। मॉडल सचमुच पेड़ों के लिए जंगल नहीं देख सकता।

पूरी बातचीत का इतिहास भेजना सुरक्षित लगता है। आप मॉडल को "सभी जानकारी" दे रहे हैं। लेकिन वास्तव में आप मॉडल के लिए जो मायने रखता है उस पर ध्यान केंद्रित करना कठिन बना रहे हैं।

### कार्य स्मृति बनाम दीर्घकालिक भंडारण

Mastra की मेमोरी प्रणाली आपको दोनों देती है। कार्य स्मृति हाल के संदेशों को संदर्भ विंडो में रखती है। अर्थपूर्ण स्मरण तब ऐतिहासिक संदेशों को खोजता है जब वर्तमान क्वेरी संबंधित लगती है।

```typescript
// src/mastra/agents/memory-agent.ts
import { Agent } from '@mastra/core/agent';
import { Memory } from '@mastra/memory';
import { LibSQLStore } from '@mastra/libsql';

export const memoryAgent = new Agent({
  id: 'memory-agent',
  name: 'Memory Agent',
  instructions: 'You are a helpful assistant with perfect recall of our conversations.',
  model: openai('gpt-5'),
  memory: new Memory({
    storage: new LibSQLStore({
      id: 'memory-agent-store',
      url: 'file:../mastra.db',
    }),
    options: {
      lastMessages: 20,  // Keep last 20 messages in context
      semanticRecall: {
        enabled: true,  // Use embeddings to find old stuff
        topK: 5,
        threshold: 0.7,
      },
    },
  }),
});
```

यह व्यवहार में कैसे काम करता है, यहाँ देखें। एक उपयोगकर्ता पूछता है: "वह इतालवी रेस्तरां कौन सा था जो आपने पिछले महीने सुझाया था?"

अर्थपूर्ण स्मरण के बिना, एजेंट अंतिम 20 संदेश देखता है। रेस्तरां की सिफारिश संदेश 487 थी 506 में से। यह चला गया। एजेंट कहता है "मेरे पास वह जानकारी नहीं है।"

अर्थपूर्ण स्मरण के साथ:
1. क्वेरी एम्बेड हो जाती है: `[0.234, -0.567, 0.891, ...]`
2. एम्बेडिंग की तुलना ऐतिहासिक संदेशों के विरुद्ध की जाती है
3. संदेश 487 ("मैं Trattoria Bella की सिफारिश करूँगा - उनकी कार्बोनारा अविश्वसनीय है") 0.89 समानता स्कोर करता है
4. वह संदेश वर्तमान संदर्भ में इंजेक्ट किया जाता है
5. एजेंट जवाब देता है: "मैंने Trattoria Bella की सिफारिश की थी। उनकी कार्बोनारा ही थी जिसने मेरा ध्यान आकर्षित किया।"

एजेंट को पूर्ण स्मृति प्रतीत होती है जबकि यह केवल संदर्भ विंडो का एक अंश उपयोग कर रहा है। यह केवल चतुर इंजीनियरिंग नहीं है - एक बार बातचीत कुछ दर्जन संदेशों से आगे बढ़ जाने के बाद यह कार्यात्मक रूप से आवश्यक है।

## एजेंट नेटवर्क के माध्यम से समन्वय

कभी-कभी आपको संरचना और लचीलापन दोनों की आवश्यकता होती है। शुद्ध वर्कफ़्लोज़ बहुत कठोर होते हैं। शुद्ध एजेंट बहुत अप्रत्याशित होते हैं।

एजेंट नेटवर्क आपको एक समन्वयक देते हैं जो कार्य के आधार पर किस विशेषज्ञ एजेंट या वर्कफ़्लोज को लागू करना है यह तय करता है। इसे AI क्षमताओं के लिए एक स्मार्ट लोड बैलेंसर के रूप में सोचें।

```typescript
export const coordinatorAgent = new Agent({
  id: 'coordinator-agent',
  name: 'Research Coordinator',
  instructions: `You are a network of researchers and writers.
    - Use researchAgent for gathering facts
    - Use writingAgent for producing final content
    - Use weatherTool for current weather data
    - Use activityPlannerWorkflow for location-based planning
    
    Always produce comprehensive, well-structured responses.`,
  model: openai('gpt-5'),
  
  // Available primitives
  agents: { researchAgent, writingAgent },
  workflows: { activityPlannerWorkflow },
  tools: { weatherTool },
  
  // Network requires memory
  memory: new Memory({
    storage: new LibSQLStore({ id: 'network-store', url: 'file:../network.db' }),
  }),
});
```

जब आप इस नेटवर्क को क्वेरी करते हैं, तो समन्वयक अनुरोध का विश्लेषण करता है और तदनुसार मार्ग निर्धारित करता है:
- "मुझे X के बारे में तथ्य चाहिए" अनुसंधान एजेंट को ट्रिगर करता है
- "सिएटल में एक सप्ताहांत की योजना बनाएं" गतिविधि योजनाकार वर्कफ़्लोज चलाता है
- "Y पर एक रिपोर्ट लिखें" लेखन एजेंट को संलग्न करता है

यह पैटर्न एक ही मेगा-एजेंट में सब कुछ ठूंसने की कोशिश करने की तुलना में बेहतर स्केल करता है। विशेषज्ञ एजेंट केंद्रित विशेषज्ञता विकसित करते हैं। समन्वयक मार्ग निर्धारण संभालता है। प्रत्येक टुकड़ा वह करता है जिसमें वह अच्छा है।

## इसे एक साथ रखना

वास्तविक उत्पादन AI प्रणालियों को वास्तुकला की आवश्यकता होती है, न कि केवल प्रॉम्प्ट। आप वितरित प्रणालियाँ बना रहे हैं जहाँ कुछ नोड्स संयोग से LLMs हैं।

वर्कफ़्लोज़ आपको गारंटी देते हैं जब आपको चीज़ों को ठीक उसी तरह से होने की आवश्यकता होती है। मेमोरी आपको संदर्भ देती है बिना आपके टोकन बजट को जलाए। एजेंट नेटवर्क आपको सरल भागों से जटिलता की रचना करने देते हैं।

इनमें से कोई भी ग्लैमरस नहीं है। लेकिन काफी "पूरी तरह से स्वायत्त एजेंटों" को उत्पादन में विफल होते देखने के बाद, मैं रोमांचक अप्रत्याशितता पर उबाऊ विश्वसनीयता की सराहना करने लगा हूँ।

आपका अनुभव अलग हो सकता है, लेकिन मेरे अनुभव में, वे प्रणालियाँ जो वास्तव में शिप होती हैं और चलती रहती हैं, वे हैं जो LLMs को एक बड़ी वास्तुकला में घटकों के रूप में मानती हैं न कि जादू के डिब्बे जो सब कुछ हल करते हैं।
