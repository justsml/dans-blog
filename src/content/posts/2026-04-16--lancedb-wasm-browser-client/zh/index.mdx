## 问题：无法在浏览器中搜索 Lance 表

LanceDB 是一个极佳的向量数据库。相较于传统数据库，它更简洁、成本更低。你可以发布包含向量列、全文搜索（FTS）列的表，将文件托管在 S3 或 CDN 上，并直接从 Node.js 服务器通过 S3 查询数据。无需守护进程。

在此 PR 之前，**无法**直接在浏览器中打开该表并执行搜索。你必须在中间加入服务器：Lambda、容器，或任何能够运行 Rust、Node.js/WASM 或 Python 的 *东西*。

目标虽然宏大，却聚焦于一个有意义的结果：**直接可搜索、只读、通过 HTTP 托管的向量索引**。

---

## 架构

实现分为三层：

### 1. Rust 发布产物 (`web_publish.rs`)

在浏览器能够搜索 Lance 表之前，需要先知道哪些内容是安全可搜索的。Lance 的内部格式包含一些浏览器不可移植的特性——增量文件、外部基底、WASM 运行时无法执行的索引类型。与其让浏览器在搜索时自行发现这些（糟糕），发布步骤会生成明确的 side‑car 文件：

- `_web.json` —— 声明 schema 以及哪些列可以在浏览器中搜索（支持的距离类型的向量列、FTS 列）
- `_snapshot.json` —— 数据集的时间点视图，浏览器客户端可以在不理解完整数据演进协议的情况下消费
- `_latest.manifest` / `_latest.version` —— 稳定指针，浏览器轮询以检测陈旧性

`isComplete` 标志位在快照中充当安全阀。如果数据集依赖于浏览器路径无法安全读取的外部基底或数据文件，`isComplete=false`，浏览器会提前以明确的错误拒绝加载，而不是产生细微错误的结果。

这正是我最庆幸把逻辑写在 Rust 而不是 JavaScript 中的原因。判断何为浏览器安全的逻辑位于表元数据实际所在的地方。

### 2. `lancedb-wasm` Rust crate

这就是 WebAssembly 运行时：一个基于 `fetch` 的对象存储、一个在浏览器端的搜索引擎，以及用于过滤谓词的表达式求值器。

对象存储实现 (`fetch_object_store.rs`) 通过 HTTP 范围请求实现了 Lance 的 `ObjectStore` trait。Lance 的内部读取本身已经组织为字节范围抓取，这恰好对应 `Range: bytes=N-M` 头部。这是项目中最让人满意的部分——现有架构几乎完美匹配，我只需把它接通即可。

浏览器引擎 (`browser.rs`) 负责搜索：向量最近邻、全文检索以及混合搜索（向量 + FTS，使用 RRF 融合）。过滤表达式 (`browser_expr.rs`) 在纯 Rust 中对 Lance 的过滤谓词进行客户端求值，随后编译为 WASM。

特性拒绝是严格且刻意的。`hamming` 距离？直接以明确消息拒绝——不悄悄出错，也不触发运行时 panic。对不完整快照执行 `fastSearch`？拒绝。不支持的基路径布局？拒绝。目标是 **fail‑closed**：如果浏览器路径无法满足合约，就明确告知。

### 3. `@lancedb/lancedb-web` TypeScript 包

公开的 API 故意保持精简：

```typescript
import { searchTable } from "@lancedb/lancedb-web";

const results = await searchTable("https://my-cdn.example.com/my-table", "semantic search query", {
  select: ["title", "url", "score"],
  limit: 10,
});
```

在内部，`searchTable()` 优先走 Worker‑backed 执行路径，避免搜索阻塞主线程。WASM 模块在 Worker 中运行，结果通过类型化协议 (`worker_protocol.ts`) 回传。还有可选的 `./transformers` 导出，它包装了 `@xenova/transformers`，在客户端生成查询嵌入——因此可以在浏览器内完成从原始文本查询到向量搜索结果的全链路。

## 难点

### HTTP 范围读取并非免费

Lance 的读取模式假设对象存储能够高效处理大量小的范围抓取。S3 和 GCS 正是为此设计的。浏览器…大多数情况下也能做到，但使用带 `Range` 头的 `fetch` 时会出现一些 quirks，现有的 Rust 对象存储抽象并不需要关心这些细节。

主要问题在于浏览器有时会进行缓冲或激进的重新请求，而你无法像在本地进程中那样控制 TCP 连接池。对于大型向量索引，这点尤为重要。当前实现是正确的；至于在大表生产环境下是否足够快，还需要在评审过程中进一步验证。

### sidecar 生成的抉择

这个 PR 明确把它提给维护者，因为这里存在真实的设计权衡：浏览器 sidecar 文件（`_web.json`、`_snapshot.json`）是应该在每次表提交时*自动*生成，还是需要显式调用 `publish()`？

自动生成更符合人机交互——浏览器支持可以免费获得。但这意味着每一次本地写入都会多一点开销，并且会把浏览器合约耦合到核心提交路径，可能会让未来对任一方的改动变得更复杂。

显式 publish 从合约角度来看更严谨——你在声明“此版本已准备好在浏览器使用”。但容易被忘记，从而导致本应可在浏览器搜索的表悄然不可用。

我倾向于自动生成，并在 sidecar 生成失败时记录警告而不是让提交失败。但我对这一点仍然不确定，并已在 PR 中标记出来。

### 什么算是“浏览器安全”的？

`isComplete` 标志以及 `web_publish.rs` 中的列过滤逻辑对浏览器能够处理的内容做出了判断。这一判断必须与 WASM 运行时的实际能力保持同步。如果有人向 Lance 添加了浏览器无法执行的新索引类型，发布代码需要知道将其排除——否则你会对外宣称一种浏览器在搜索时根本无法支持的能力。

这里的正确做法可能是建立一个能力注册表，在发布路径和运行时之间共享，以防两者漂移。我目前把它实现为并行的常量，虽然快捷但很脆弱。这大概是我接下来想要改进的地方。

---

## Transformers 包装器的两难

`./transformers` 导出是一个提升使用体验的特性：它接受文本查询，使用 `@xenova/transformers` 在本地运行嵌入模型，然后把得到的向量交给 LanceDB 搜索引擎。这样即可在静态 HTML 页面实现零服务器的语义搜索。

但这也是我最不确定是否应当放进本 PR 的部分。它确实很有用且已经实现，但它引入了额外的依赖和 API 表面，且与核心的浏览器搜索问题是分离的。维护者可能更倾向于先落地基础的 `searchTable()` API，再单独迭代嵌入功能。

我已经询问过，后续再看。

## 如果让我重新来过的做法

**更早的范围。** 这次 PR 有 14,500 行代码，一口气审完实在太多。我本可以先提交发布产物格式，然后是 WASM 运行时，最后是 TypeScript 包。分成三个小 PR 会更容易合并。我没有这么做，因为想在提出任何单独组件之前先验证完整的端到端流程——这对信心有帮助，但也大幅提升了审查负担。

**先编写能力注册表。** `web_publish.rs` 与 `browser.rs` 之间的漂移风险是真实存在的。若一开始就使用共享的能力类型，让双方都引用，同步起来会比我现在使用的并行常量列表更干净。

**少用巧妙的命名。** `_web.json` 作为内部格式的名字倒是可以，但一旦有人缓存它或围绕它构建工具，它就成了承载负荷的名称。我本应该在提交前多花时间考虑命名——这类东西在别人依赖之前改动成本要低得多。

## 更广的意义

让这个项目可行的关键在于 Lance 的内部架构本身已经为范围读取做好了良好结构。对象存储抽象简洁，元数据格式明确，快照概念为我提供了一个自然的地点来表达“这是该数据集的浏览器安全视图”。

良好的抽象边界不仅让内部代码更易于推理——也使得像我这样从项目外部加入的人能够插入新的执行环境而无需触碰核心代码。这是真正的设计美德，也让这 14,000 行代码更像是填充形状，而不是与代码库硬碰硬。
