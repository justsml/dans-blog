## Problem: Aap Browser se Lance Table Search Nahin Kar Sakte

LanceDB ek fantastic vector database hai. Yeh traditional databases se bahut simpler aur cheaper hai. Aap tables ko vector columns, full-text search (FTS) columns ke saath publish kar sakte hain, files ko S3 ya CDN par host kar sakte hain, aur Node.js server se directly S3 par query kar sakte hain. Koi daemon nahin chahiye.

Jo aap *nahin* kar sakte the — is PR se pehle — wo tha browser se directly table kholna aur search chalana. Aapko beech mein ek server chahiye hota: Lambda, container, *kuch* jo Rust, Node.js/WASM, ya Python chalata ho.

Goal ambitious tha, lekin focused meaningful end result par: **directly searchable, read-only, HTTP-hosted vector indexes.**

---

## Architecture

Implementation mein teen layers hain:

### 1. Rust publish artifacts (`web_publish.rs`)

Browser koi Lance table search kar sake, usse pehle use yeh jaanna zaroori hai ki kya search safe hai. Lance ka internal format kuch features rakhta hai jo browser-portable nahin hain — delta files, external bases, wo index types jo WASM runtime execute nahin kar sakta. Browser ko search time par yeh discover karne deney ki bajay (jo buri baat hai), publish step explicit sidecar files produce karta hai:

- `_web.json` — schema advertise karta hai aur konsay columns browser-searchable hain (vector columns supported distance types ke saath, FTS columns)
- `_snapshot.json` — dataset ka point-in-time view jo browser client consume kar sakta hai bina full dataset evolution protocol ko samjhey
- `_latest.manifest` / `_latest.version` — stable pointers jo browser staleness detect karne ke liye poll karta hai

Snapshot mein `isComplete` flag safety valve hai. Agar dataset external bases ya data files par depend karta hai jo browser path safely read nahin kar sakta, toh `isComplete=false` aur browser early reject kar deta hai clear error ke saath instead of subtly wrong results produce karney ki bajay.

Yeh wo part hai jiska main sabse khush hoon ki Maine Rust mein build kiya, JavaScript mein nahin. Yeh judgment about browser-safe kahan hai yeh wahi decide karta hai jahan table metadata actually rehta hai.

### 2. `lancedb-wasm` Rust crate

Yeh WebAssembly runtime hai: `fetch`-backed object store, browser-side search engine, aur filter predicates ke liye expression evaluator.

Fetch object store (`fetch_object_store.rs`) Lance ke `ObjectStore` trait ko HTTP range requests par implement karta hai. Lance ke internal reads already structured hain as ranged byte-range fetches, jo cleanly `Range: bytes=N-M` headers par map hota hai. Yeh sabse satisfying part tha — existing architecture almost perfectly designed tha iske liye, mujhe bas wire karna tha.

Browser engine (`browser.rs`) search handle karta hai: vector nearest-neighbor, full-text, aur hybrid (vector + FTS with RRF fusion). Filter expressions (`browser_expr.rs`) pure Rust mein client-side evaluate karta hai Lance filter predicates ko, WASM mein compiled.

Feature rejection strict aur intentional hai. `hamming` distance? Upfront reject with clear message — not silently wrong, not runtime panic. `fastSearch` incomplete snapshot par? Reject. Unsupported base-path layouts? Reject. Goal tha fail closed: agar browser path contract honor nahin kar sakta, toh wo bolta hai.

### 3. `@lancedb/lancedb-web` TypeScript package

Public API surface intentionally small hai:

```typescript
import { searchTable } from "@lancedb/lancedb-web";

const results = await searchTable("https://my-cdn.example.com/my-table", "semantic search query", {
  select: ["title", "url", "score"],
  limit: 10,
});
```

Under the hood, `searchTable()` Worker-backed execution path prefer karta hai taaki search main thread ko block na kare. WASM module Worker mein run hota hai; results typed protocol (`worker_protocol.ts`) par wapas aate hain. Ek optional `./transformers` export bhi hai jo query embeddings generate karne ke liye `@xenova/transformers` ko wrap karta hai client-side — taaki aap raw text query se vector search results tak kabhi browser chorh kar bhaag saken.

## The Hard Parts

### Ranged reads over HTTP free nahin hain

Lance ke read patterns ek object store assume karte hain jo efficiently many small ranged fetches handle kar sake. S3 aur GCS iske liye designed hain. Browsers... mostly bhi hain, lekin `fetch` with `Range` headers kuch quirks rakhta hai jo existing Rust object store abstractions ko care nahin karna padta.

Main issue yeh hai ki browsers kabhi-kabhi aggressively buffer ya re-request karte hain, aur aap TCP connection pooling control nahin kar sakte jaisa aap native process mein kar sakte hain. Large vector indices ke liye yeh matter karta hai. Current implementation correct hai; agar yeh large tables par production use ke liye fast enough hai ya nahin yeh review ke dauran shake niklega.

### Sidecar generation question

PR yeh explicitly question karta hai maintainers ke liye, kyunki ek real design tradeoff hai: browser sidecar files (`_web.json`, `_snapshot.json`) har table commit par *automatically* emit honi chahiye, ya explicit `publish()` call honi chahiye?

Automatic more ergonomic hai — aap free mein browser support get karte ho. Lekin iska matlab har local write mein extra overhead hai, aur yeh browser contract ko core commit path se couple karta hai aise tarike se jo future changes ko complicate kar sakta hai.

Explicit publish more correct hai contract standpoint se — aap bol rahe ho "yeh version browser-ready hai" — lekin isko forget karna easy hai, matlab jo tables *should* be browser-searchable wo silently nahin hain.

Maine automatic ki taraf lean kiya, sidecar failures warning log karke. But I'm genuinely uncertain, aur maine PR mein flag kiya.

### "Browser-safe" kya count hota hai?

`isComplete` flag aur column filtering logic `web_publish.rs` mein ek judgment encode karta hai ki browser kya handle kar sakta hai. Yeh judgment WASM runtime ki actual capabilities ke saath sync mein rehni chahiye. Agar koi Lance mein naya index type add kare jo browser execute nahin kar sakta, toh publish code ko pata hona chahiye ki ise exclude karna hai — warna aap capability advertise karenge jo browser search time par honor karne mein fail karega.

Right fix shayad ek capability registry hai jo publish path aur runtime ke beech shared ho, taaki wo drift na kar sake. Maine abhi ke liye parallel constants implement kiya, jo expedient hai lekin fragile. Yeh probably wo cheez hai jisko main next improve karna chahunga.

## The Transformers Wrapper Dilemma

 `./transformers` export ek quality-of-life feature hai: yeh text query leta hai, `@xenova/transformers` via local embedding model se run karta hai, aur resulting vector LanceDB search engine ko deta hai. Static HTML page se zero-server semantic search.

Yeh bhi wo cheez hai jiske baare mein main sabse uncertain hoon ki iska PR mein hona chahiye ya nahin. Yeh genuinely useful hai aur yeh implement bhi hai, lekin yeh ek dependency aur API surface hai jo core browser search question se alag hai. Maintainers reasonably pehle base `searchTable()` API land karne aur embeddings par separately iterate karna chaah sakte hain.

Maine poocha. Dekhte hain.

---

## Main Kya Alag Karta

**Scope pehle define karna chahiye tha.** PR 14,500 lines ka hai. Yeh ek shot mein review karna bahut hai. Maine pehle publish artifact format land kar sakta tha, phir WASM runtime, phir TypeScript package. Teen smaller PRs merge karna easier hota. Maine yeh nahin kiya kyunki maine pehle full end-to-end story validate karna chaahiye tha kisi bhi piece ko propose karne se pehle — jo confidence ke liye made sense tha, lekin review burden high banata hai.

**Capability registry pehle likhna chahiye tha.** `web_publish.rs` aur `browser.rs` ke beech drift risk real hai. Shared capability type se shuru karna jo dono sides reference karein cleanest tha parallel constant lists ki jagah jo mujhe remember karna padta hai sync mein rakhna.

**Kam clever names dena chahiye tha.** `_web.json` internal format ke liye fine name hai. Yeh load-bearing name ban jata hai jab koi isko cache kare ya iske baare mein tooling build kare. Submit karne se pehle naming par zyada time spend karna chahiye tha — yeh un cheezon mein se hai jo kisi aur ke depend hone se pehle change karna bahut cheaper hai.

---

## Broader Point

Jo is project ko tractable banaya wo yeh hai ki Lance ka internal architecture ranged reads ke liye pehle se well-structured tha. Object store abstraction clean hai. Metadata format explicit hai. Snapshot concept ne mujhe natural jagah di "yeh dataset ka browser-safe view hai" express karne ke liye.

Good abstraction boundaries sirf internal code ko easier nahi banate reasoning ke liye — wo isko possible bhi banate hain ke koi outside project se aaye aur new execution environment plug in kar sake bina core ko touch kiye. Yeh real design virtue hai, aur isne 14,000 lines ko shape fill karne jaisa feel karaya codebase se ladhne ki bajay.

PR [yahan](https://github.com/lancedb/lancedb/pull/3247) hai agar aap details dekhna chahte hain. Review ka intezaar hai. Maintainers warm rahe hain aur main cautiously optimistic hoon.
