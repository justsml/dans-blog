## 問題：ブラウザからLanceテーブルを検索できない

LanceDBは素晴らしいベクターデータベースです。従来のデータベースよりもはるかにシンプルで低コストです。ベクターカラムや全文検索（FTS）カラムを持つテーブルを公開し、ファイルをS3やCDNでホスティングし、Node.jsサーバーからS3経由で直接データをクエリできます。デーモンは不要です。

しかし、このPR以前は、ブラウザから直接そのテーブルを開いて検索することは**できませんでした**。間にLambda、コンテナ、あるいはRust、Node.js/WASM、Pythonを実行できる**何らかの**サーバーが必要だったのです。

目標は野心的でありながらも、意味のある最終成果に焦点を当てていました。**HTTPでホストされた、直接検索可能で読み取り専用のベクターインデックス**です。

---

## アーキテクチャ

実装には3つのレイヤーがあります。

### 1. Rust公開アーティファクト（`web_publish.rs`）

ブラウザがLanceテーブルを検索できるようになる前に、何が安全に検索できるかを知る必要があります。Lanceの内部フォーマットには、ブラウザに移植できない機能があります。デルタファイル、外部ベース、WASMランタイムが実行できないインデックスタイプなどです。ブラウザに検索時にこれを発見させる（悪い設計）のではなく、公開ステップで明示的なサイドカーファイルを生成します。

- `_web.json` — スキーマと、ブラウザで検索可能なカラム（対応する距離タイプのベクターカラム、FTSカラム）を通知する
- `_snapshot.json` — ブラウザクライアントがデータセット全体の進化プロトコルを理解しなくても消費できる、データセットのポイントインタイムビュー
- `_latest.manifest` / `_latest.version` — ブラウザがポーリングして陳腐化を検出するための安定したポインタ

スナップショットの`isComplete`フラグは安全弁です。データセットがブラウザパスで安全に読み取れない外部ベースやデータファイルに依存している場合、`isComplete=false`となり、ブラウザは微妙に間違った結果を生成する代わりに、早期に明確なエラーで拒否します。

この判断をRust側に組み込んだことは、JavaScriptではなくRustで実装して本当に良かったと感じている点です。ブラウザセーフかどうかの判断は、テーブルメタデータが実際に存在する場所に存在するべきなのです。

### 2. `lancedb-wasm` Rustクレート

これはWebAssemblyランタイムです。`fetch`ベースのオブジェクトストア、ブラウザサイドの検索エンジン、フィルター述語の式評価器です。

fetchオブジェクトストア（`fetch_object_store.rs`）は、HTTPレンジリクエストを介してLanceの`ObjectStore`トレイトを実装します。Lanceの内部読み取りはすでに範囲指定されたバイトレンジフェッチとして構成されており、これは`Range: bytes=N-M`ヘッダーにきれいにマッピングされます。これがこのプロジェクトで最も満足感のあった部分です。既存のアーキテクチャがこの用途にほぼ完璧に設計されていたので、私は配線するだけでよかったのです。

ブラウザエンジン（`browser.rs`）は、ベクター近傍検索、全文検索、およびハイブリッド（ベクター＋FTS、RRF融合）の検索を処理します。フィルター式（`browser_expr.rs`）は、Lanceのフィルター述語をクライアントサイドの純Rustで評価し、WASMにコンパイルします。

機能の拒否は厳格かつ意図的です。`hamming`距離？ 明確なメッセージとともに事前に拒否されます。静かに間違ったり、ランタイムパニックになったりしません。不完全なスナップショットでの`fastSearch`？ 拒否されます。サポートされていないベースパスのレイアウト？ 拒否されます。目標はフェイルクローズドでした。ブラウザパスが契約を守れない場合は、そう伝えるのです。

### 3. `@lancedb/lancedb-web` TypeScriptパッケージ

公開APIサーフェスは意図的に小さく保たれています。

```typescript
import { searchTable } from "@lancedb/lancedb-web";

const results = await searchTable("https://my-cdn.example.com/my-table", "semantic search query", {
  select: ["title", "url", "score"],
  limit: 10,
});
```

内部的には、`searchTable()`はWorkerベースの実行パスを優先するため、検索がメインスレッドをブロックしません。WASMモジュールはWorker内で実行され、結果は型付けされたプロトコル（`worker_protocol.ts`）を介して返されます。また、オプションの`./transformers`エクスポートもあり、`@xenova/transformers`をラップしてクエリ埋め込みをクライアントサイドで生成します。つまり、生のテキストクエリからベクター検索結果まで、ブラウザを離れることなく完結できます。

## 困難だった点

### HTTP経由のレンジ読み取りはコストがかかる

Lanceの読み取りパターンは、オブジェクトストアが多数の小さな範囲指定フェッチを効率的に処理できることを前提としています。S3やGCSはこれに最適化されています。ブラウザも…ほとんどは同様ですが、`Range`ヘッダーを使った`fetch`には、既存のRustオブジェクトストア抽象化では気にする必要のないいくつかの癖があります。

主な問題は、ブラウザが積極的にバッファリングや再リクエストを行うことがあり、ネイティブプロセスのようにTCP接続プールを制御できないことです。大規模なベクターインデックスではこれが重要になります。現在の実装は正しいです。本番環境の大規模テーブルで十分に高速かどうかは、レビューの中で明らかになるでしょう。

### サイドカー生成の問題

PRでは、メンテナーへの質問としてこれを明示的に提起しています。なぜなら、実際の設計トレードオフがあるからです。ブラウザ用サイドカーファイル（`_web.json`、`_snapshot.json`）は、すべてのテーブルコミットで*自動的に*生成されるべきか、それとも明示的な`publish()`呼び出しが必要なのか？

自動生成の方がエルゴノミックです。ブラウザサポートが無料で得られます。しかし、すべてのローカル書き込みに少し余分なオーバーヘッドが発生し、ブラウザ契約がコアのコミットパスに結合されるため、将来のどちらかの変更が複雑になる可能性があります。

明示的な公開は契約の観点からはより正しいです。「このバージョンはブラウザ対応です」と明示するからです。しかし、忘れやすいという欠点があり、*ブラウザで検索できるべき*テーブルが静かに検索できなくなります。

私は自動生成に傾いています。サイドカーの失敗はコミットを失敗させるのではなく、警告をログに記録するようにします。しかし、本当に確信があるわけではなく、PR内でフラグを立てました。

### 「ブラウザセーフ」とは何か？

`isComplete`フラグと`web_publish.rs`のカラムフィルタリングロジックは、ブラウザが何を処理できるかという判断をエンコードしています。その判断はWASMランタイムの実際の能力と同期している必要があります。誰かがLanceにブラウザが実行できない新しいインデックスタイプを追加した場合、公開コードはそれを除外することを知っている必要があります。そうでなければ、検索時にブラウザが対応できない能力を宣伝することになります。

ここでの正しい修正は、おそらく公開パスとランタイムの間で共有される能力レジストリです。そうすれば両者が乖離することはありません。今は並列の定数として実装しましたが、これはその場しのぎであって脆弱です。おそらくこれが次に改善したい点です。

## Transformersラッパーのジレンマ

`./transformers`エクスポートはQoL機能です。テキストクエリを受け取り、`@xenova/transformers`を介してローカル埋め込みモデルで実行し、結果のベクターをLanceDB検索エンジンに渡します。静的なHTMLページから、サーバーレスでセマンティック検索が実現します。

しかし同時に、このPRに本当に属するのか最も確信が持てない部分でもあります。確かに便利で、すでに実装済みですが、依存関係であり、コアのブラウザ検索の問題とは別のAPIサーフェスです。メンテナーとしては、まずベースの`searchTable()` APIを確定させ、埋め込みは別途反復したいと考えるのが妥当でしょう。

## 次回はこうする

**もっと早くスコープを絞る。** このPRは14,500行あります。一度にレビューするには多いです。公開アーティファクトのフォーマット、WASMランタイム、TypeScriptパッケージの順に、3つの小さなPRに分割できたはずです。3つの小さなPRの方がマージしやすかったでしょう。そうしなかったのは、どの部分を提案する前に完全なエンドツーエンドのストーリーを検証したかったからです。確信を持つためには理にかなっていましたが、レビューの負担は大きくなりました。

**まず能力レジストリを書く。** `web_publish.rs`と`browser.rs`の間の乖離リスクは現実のものです。両側が参照する共有の能力タイプから始めていれば、同期を忘れないようにしなければならない並列の定数リストよりもクリーンだったでしょう。

**命名をあまり clever にしない。** `_web.json`は内部フォーマットとしては問題のない名前です。しかし、誰かがそれをキャッシュしたり、ツールを構築したりした瞬間に、負荷のかかる名前になります。誰も依存する前に変更する方がはるかに安上がりなので、もっと命名に時間をかけるべきでした。

## より広い視点

このプロジェクトを実現可能にしたのは、Lanceの内部アーキテクチャがすでに範囲指定読み取りに適した構造になっていたことです。オブジェクトストアの抽象化はきれいです。メタデータフォーマットは明示的です。スナップショットの概念は、「これはこのデータセットのブラウザセーフなビューです」と表現する自然な場所を与えてくれました。

良い抽象化境界は、内部コードの推論を容易にするだけでなく、私のようなプロジェクト外の人間がコアに触れることなく新しい実行環境をプラグインすることを可能にします。これは真の設計の美徳であり、14,000行がコードベースとの戦いというよりも、与えられた形を埋める作業のように感じさせてくれました。
