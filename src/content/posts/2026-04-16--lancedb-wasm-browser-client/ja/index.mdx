## The Problem: You Can't Search a Lance Table From a Browser

LanceDB は素晴らしいベクトルデータベースです。従来のデータベースより遥かにシンプルで安価です。ベクトルカラム、全文検索（FTS）カラムを持つテーブルを公開し、S3やCDNでファイルをホストし、Node.jsサーバーからS3直接データをクエリできます。デーモンは不要です。

以前は — このPR以前に — ブラウザから直接そのテーブルを開いて検索を実行することは**できませんでした**。そこにはサーバーが必要でした：Lambda、コンテナ、Rust、Node.js/WASM、またはPythonを実行できる何か。

この目標は野心的でしたが、意味のある最終結果に焦点を当てていました：**直接検索可能、読み取り専用、HTTPホスト済みのベクトルインデックス。**

## The Architecture

実装には3層があります：

### 1. Rust publish artifacts (`web_publish.rs`)

ブラウザがLanceテーブルを検索するには、何を検索してよいかを知っておく必要があります。Lanceの内部フォーマットには、ブラウザに移植できない機能がいくつかあります：deltaファイル、外部ベース、WASMランタイムで実行できないインデックスの種類。ブラウザが検索時に発見したら（悪い）ではなく、`publish`段階で明示的なサイドカーファイルを生成するため：

- `_web.json` — スキーマとどのカラムがブラウザ検索可能かをアドバタイズする（サポートされた距離タイプを持つベクトルカラム、FTSカラム）
- `_snapshot.json` — ブラウザクライアントが完全なデータセット進化プロトコルを理解せずに消費できる、データセットの時点からの一時的なビュー
- `_latest.manifest` / `_latest.version` — ブラウザが過去に劣化していることを検知するためにポーリングする安定したポインタ

スナップショット内の`isComplete`フラグが安全バルブです。データセットがブラウザのパスが安全に読み取れない外部ベースやデータファイルに依存している場合、`isComplete=false`になり、ブラウザは明確なエラーを返して先に拒否し、曖昧に間違った結果を生み出しません。

これが、RustではなくJavaScriptにこれをビルドドする部分として一番 glad です。ブラウザセーフかどうかの判断は、テーブルのメタデータが実際にいるところに住んでいます。

### 2. `lancedb-wasm` Rust crate

これはWebAssemblyランタイムです：`fetch`バックドオブジェクトストア、ブラウザ側検索エンジン、フィルター述語の式評価器です。

fetchオブジェクトストア（`fetch_object_store.rs`）は、HTTP範囲要求上でLanceの`ObjectStore`トレイトを実装します。Lanceの内部読み取りは既に範囲バイト範囲フェッチとして構造化されていますが、これは`Range: bytes=N-M`ヘッダにクリーンにマップします。このプロジェクトで一番満足した部分です — 既存のアーキテクチャはほぼ完璧にこのために設計されていて、それに接続する必要がありました。

ブラウザエンジン（`browser.rs`）は検索を処理します：ベクトル最近傍、全文検索、およびハイブリッド（ベクトル + FTSとRRF融合）。フィルター式（`browser_expr.rs`）は、純粋なRustでコンパイルされたWASMでクライアント側でLanceフィルター述語を評価します。

機能拒否は厳密で意図的です。`hamming`距離？先に明確なメッセージで拒否 — 静かに間違っているわけでも、ランタイムパニックでもありません。不完全なスナップショット上の`fastSearch`？拒否します。サポートされていないベースパスレイアウト？拒否します。目標は閉じることです：ブラウザのパスが契約を尊重できない場合、それを示します。

### 3. `@lancedb/lancedb-web` TypeScriptパッケージ

パブリックAPIスurfaceは意図的に小さいです：

```typescript
import { searchTable } from "@lancedb/lancedb-web";

const results = await searchTable("https://my-cdn.example.com/my-table", "semantic search query", {
  select: ["title", "url", "score"],
  limit: 10,
});
```

裏側で、`searchTable()`は検索がメインスレッドをブロックしないように、Workerバックド実行パスを優先します。WASMモジュールはWorker内で実行されます。結果は型付きプロトコル（`worker_protocol.ts`）を通して流れます。また、ローカル埋め込みモデルを介して`@xenova/transformers`をラップするオプションの`./transformers`エクスポートもあります — だからあなたは生のテキストクエリからベクトル検索結果まで、ブラウザから離れることなく移動できます。

## The Hard Parts

### Ranged reads over HTTP are not free

Lanceの読み取りパターンは、多くの小さい範囲フェッチを効率的に処理できるオブジェクトストアを想定しています。S3とGCSはこれのために設計されています。ブラウザは... mostly はそうですが、`fetch`と`Range`ヘッダには`fetch_object_store.rs`が気にする必要のないいくつかの癖があります。

主な問題は、ブラウザが時々バッファリングや再要求を積極的に行い、ネイティブプロセスではできるのと同じようにTCP接続プールリングを制御できないことです。大きなベクトルインデックスについてはこれが重要です。現在の実装は正しいです。大きなテーブルで本番利用するのに十分速いかどうかは、レビュー中に明らかになるでしょう。

### The sidecar generation question

このPRは明確に質問としていますが、本当に設計トレードオフがあります：ブラウザサイドカーファイル（`_web.json`、`_snapshot.json`）は毎回のテーブルコミットで自動的に出力するべきか、それとも明示的な`publish()`呼び出しがあるべきです。

自動はより快適です — ブラウザサポートが無料で得られます。しかし、それはすべてのローカル書き込みに少し余分なオーバーヘッドがあり、ブラウザ契約をコアコミットパスに結合して、将来のどちらかの変更を複雑にする可能性があります。

明示的パブリッシュは契約という点から正しい — 「このバージョンはブラウザ準備ができている」と言う — しかし、忘れやすいので、検索可能であるべきテーブルが静かに検索可能ではなくなる可能性があります。

私は自動に傾きましたが、サイドカーフェイルでコミットを失敗させるのではなく、警告としてログ記録しました。しかし私は本当に確信しておらず、PRでフラグ付けしました。

### What counts as "browser-safe"?

`isComplete`フラグと`web_publish.rs`内のカラムフィルタリングロジックは、ブラウザが処理できるものに対する判断を符号化しています。その判断はWASMランタイムの実際の機能と同期を保つ必要があります。もし誰かがブラウザが実行できない新しいインデックスタイプをLanceに追加したら、公開コードはそれを除外する必要があります — さもなければ、ブラウザが検索時にそれを尊重できずに失敗することになります。

ここでの正しい修正は、おそらく公開パスとランタイム間で共有できる機能レジストリです、彼らが離れることができません。私は今のところ並列定数として実装しましたが、速達的ですが脆弱です。これが改善する必要があるものの1つです。

## The Transformers Wrapper Dilemma

`./transformers`エクスポートは品質オプション機能です：テキストクエリを取り、`@xenova/transformers`を介してローカル埋め込みモデルを走らせ、結果として得られるベクトルをLanceDB検索エンジンに手渡します。静的HTMLページからサーバーなしのセマンティック検索。

また、このPRに入れるべきではないものでもあります。確かに役に立ち、実装済みですが、コアブラウザ検索質問から分離された依存関係とAPIスurfaceです。メンテナーが根本的な`searchTable()`APIを先に着陸し、埋め込みを別に反復するのは合理的かもしれません。

## What I'd Do Differently

**スコープを先に。** このPRは14,500行です。一度にレビューするには多いです。私はパブリッシュアーティファクト形式を先に、次にWASMランタイム、そしてTypeScriptパッケージに着陸できました。3つの小さいPRの方がマージしやすかったでしょう。私は最初に完全なエンドツーエンドストーリーを検証するために着陸したかったからです — これは自信にとって意味がありましたが、レビュー負荷が高くなります。

**機能レジストリを先に書く。** `web_publish.rs`と`browser.rs`間の逸脱リスクは本物です。両方が参照する共有機能型から始める方が、変更を同期する必要がある並列定数リストよりクリーンでした。

**名前をあまり賢くつけない。** `_web.json`は内部フォーマットに対する良い名前です。誰かがそれをキャッシュしたり、ツールリングをビルドしたりし始めると、これは負荷支持名になります。誰かがそれに依存する前に命名に時間を費やすべきでした — それが誰かがそれに依存する前に変更するのがずっと安いものの1つです。

## The Broader Point

このプロジェクトが取り扱い可能になったのは、Lanceの内部アーキテクチャが既に範囲読み取りのためにうまく構造化されていたからです。オブジェクトストア抽象化はクリーンです。メタデータフォーマットは明示です。スナップショットコンセプトは「これはこのデータセットのブラウザセーフなビューだ」と表現するための自然な場所を私に与えました。

良い抽象境界は単に内部コードを理解しやすくするだけ — それが誰かが入ってきて、コアを触らないで新しい実行環境をプラグインすることを可能にします。それは本当の設計美徳であり、14,000行を形状を埋めることよりもコードベースと戦うことよりも多く感じさせました。
