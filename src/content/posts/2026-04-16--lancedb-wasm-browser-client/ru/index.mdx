## Проблема: из браузера нельзя искать по таблице Lance

LanceDB — отличная векторная база данных. Она значительно проще и дешевле традиционных баз данных. Вы можете публиковать таблицы с векторными столбцами, столбцами полнотекстового поиска (FTS), размещать файлы на S3 или CDN и запрашивать данные напрямую через S3 с сервера Node.js. Демон не требуется.

Чего вы *не могли* сделать — до этого PR — так это открыть эту таблицу прямо из браузера и выполнить поиск. Вам понадобился бы промежуточный сервер: Lambda, контейнер, *что-то*, способное запускать Rust, Node.js/WASM или Python.

Цель была амбициозной, но сфокусированной на конкретном результате: **доступные для прямого поиска, доступные только для чтения, размещённые по HTTP векторные индексы.**

## Архитектура

Реализация состоит из трёх уровней:

### 1. Артефакты публикации на Rust (`web_publish.rs`)

Прежде чем браузер сможет искать по таблице Lance, ему нужно знать, что именно безопасно искать. Внутренний формат Lance содержит функции, не переносимые в браузер — дельта-файлы, внешние базы, типы индексов, которые среда выполнения WASM не может выполнить. Вместо того чтобы браузер обнаруживал это во время поиска (плохо), этап публикации создаёт явные сопроводительные файлы:

- `_web.json` — описывает схему и указывает, какие столбцы доступны для поиска из браузера (векторные столбцы с поддерживаемыми типами расстояний, столбцы FTS)
- `_snapshot.json` — снимок набора данных на определённый момент времени, который браузерный клиент может использовать без понимания полного протокола эволюции набора данных
- `_latest.manifest` / `_latest.version` — стабильные указатели, которые браузер опрашивает для обнаружения устаревания

Флаг `isComplete` в снимке — это предохранитель. Если набор данных зависит от внешних баз или файлов данных, которые браузерный путь не может безопасно прочитать, `isComplete=false`, и браузер отклоняет его на раннем этапе с понятной ошибкой, вместо того чтобы выдавать незаметно неверные результаты.

Это та часть, которой я больше всего рад, что написал её на Rust, а не на JavaScript. Суждение о том, что безопасно для браузера, живёт там, где на самом деле находятся метаданные таблицы.

### 2. Крейт `lancedb-wasm` на Rust

Это среда выполнения WebAssembly: хранилище объектов на базе `fetch`, поисковый механизм для браузера и вычислитель выражений для фильтрующих предикатов.

Хранилище объектов на базе fetch (`fetch_object_store.rs`) реализует трейт `ObjectStore` из Lance поверх HTTP-запросов с диапазоном. Внутренние чтения Lance уже структурированы как выборки байтовых диапазонов, что хорошо ложится на заголовки `Range: bytes=N-M`. Это была самая удовлетворительная часть проекта — существующая архитектура была почти идеально спроектирована для этого, мне просто нужно было всё правильно соединить.

Браузерный движок (`browser.rs`) обрабатывает поиск: векторный поиск ближайших соседей, полнотекстовый и гибридный (векторный + FTS с RRF-фузией). Фильтрующие выражения (`browser_expr.rs`) вычисляют предикаты фильтров Lance на стороне клиента на чистом Rust, скомпилированном в WASM.

Отклонение функций строгое и намеренное. Расстояние `hamming`? Отклоняется сразу с понятным сообщением — не молча даёт неверный результат, не вызывает панику времени выполнения. `fastSearch` на неполном снимке? Отклоняется. Неподдерживаемые структуры базовых путей? Отклоняются. Цель была закрыться с ошибкой: если браузерный путь не может соблюсти контракт, он так и говорит.

### 3. Пакет TypeScript `@lancedb/lancedb-web`

Поверхность публичного API намеренно мала:

```typescript
import { searchTable } from "@lancedb/lancedb-web";

const results = await searchTable("https://my-cdn.example.com/my-table", "семантический поисковый запрос", {
  select: ["title", "url", "score"],
  limit: 10,
});
```

Под капотом `searchTable()` предпочитает путь выполнения через Worker, чтобы поиск не блокировал основной поток. Модуль WASM работает в Worker; результаты передаются обратно по типизированному протоколу (`worker_protocol.ts`). Есть также опциональный экспорт `./transformers`, который оборачивает `@xenova/transformers` для генерации эмбеддингов запросов на стороне клиента — так что можно перейти от текстового запроса к результатам векторного поиска, ни разу не покинув браузер.

## Сложные моменты

### Потоковые чтения по HTTP не бесплатны

Паттерны чтения Lance предполагают хранилище объектов, способное эффективно обрабатывать множество мелких выборки по диапазону. S3 и GCS для этого и спроектированы. Браузеры... в основном тоже, но `fetch` с заголовками `Range` имеет некоторые причуды, о которых существующие абстракции хранилищ объектов на Rust могут не заботиться.

Главная проблема в том, что браузеры иногда агрессивно буферизуют или повторно запрашивают данные, и вы не можете управлять пулингом TCP-соединений так, как в нативном процессе. Для больших векторных индексов это имеет значение. Текущая реализация корректна; хватит ли её скорости для продакшена на больших таблицах — проявится во время ревью.

### Вопрос генерации сопроводительных файлов

PR поднимает этот вопрос явно для мейнтейнеров, потому что здесь есть реальный дизайнерский tradeoff: должны ли браузерные сопроводительные файлы (`_web.json`, `_snapshot.json`) создаваться *автоматически* при каждом коммите таблицы, или должен быть явный вызов `publish()`?

Автоматический режим более эргономичен — вы получаете поддержку браузера бесплатно. Но это означает, что каждая локальная запись имеет дополнительную нагрузку, и это связывает браузерный контракт с основным путём коммита способами, которые могут усложнить будущие изменения любого из них.

Явный publish более корректен с точки зрения контракта — вы говорите «эта версия готова для браузера» — но его легко забыть, что означает таблицы, которые *должны* быть доступны для поиска из браузера, молча ими не являются.

Я склонялся к автоматическому режиму, с логированием предупреждения при ошибках сопроводительных файлов, а не падением коммита. Но я искренне не уверен и отметил это в PR.

### Что считается «безопасным для браузера»?

Флаг `isComplete` и логика фильтрации столбцов в `web_publish.rs` кодируют суждение о том, что браузер может обработать. Это суждение должно оставаться синхронизированным с фактическими возможностями среды выполнения WASM. Если кто-то добавит новый тип индекса в Lance, который браузер не может выполнить, код публикации должен знать, чтобы исключить его — иначе вы будете рекламировать возможность, которую браузер не сможет реализовать во время поиска.

Правильное решение здесь — вероятно, реестр возможностей, общий для пути публикации и среды выполнения, чтобы они не могли разойтись. Я реализовал это как параллельные константы на данный момент, что expedient, но хрупко. Это, вероятно, то, что я хотел бы улучшить в первую очередь.

## Дилемма обёртки Transformers

Экспорт `./transformers` — это функция для удобства: она берёт текстовый запрос, прогоняет его через локальную модель эмбеддингов via `@xenova/transformers` и передаёт полученный вектор поисковому движку LanceDB. Семантический поиск без сервера со статической HTML-страницы.

Это также то, в чём я наименее уверен, что принадлежит этому PR. Оно действительно полезно и уже реализовано, но это зависимость и поверхность API, отдельная от основного вопроса браузерного поиска. Мейнтейнеры могут разумно захотеть сначала принять базовый API `searchTable()`, а итерировать на эмбеддингах отдельно.

Я спросил. Посмотрим.

## Что я сделал бы иначе

**Определил бы_scope_ раньше.** PR — 14 500 строк. Это много для ревью за один раз. Я мог бы сначала принять формат артефактов публикации, затем среду выполнения WASM, затем пакет TypeScript. Три меньших PR было бы легче смёржить. Я не сделал этого, потому что хотел проверить полную сквозную историю, прежде чем предлагать какие-либо части — что имело смысл для уверенности, но делает нагрузку ревью высокой.

**Написал бы реестр возможностей первым.** Риск расхождения между `web_publish.rs` и `browser.rs` реален. Начало с общего типа возможности, на который ссылаются обе стороны, было бы чище, чем параллельные списки констант, которые нужно не забыть синхронизировать.

**Назвал бы вещи менее хитроумно.** `_web.json` — нормальное имя для внутреннего формата. Оно становится именем с нагрузкой в тот момент, когда кто-то начинает его кэшировать или строить вокруг него инструментарий. Мне следовало потратить больше времени на именование до отправки — это одна из тех вещей, которые гораздо дешевле изменить, пока никто ещё от них не зависит.

## Более широкая мысль

То, что сделало этот проект выполнимым, — это то, что внутренняя архитектура Lance уже была хорошо структурирована для чтений по диапазону. Абстракция хранилища объектов чистая. Формат метаданных явный. Концепция снимка дала мне естественное место, где выразить «это безопасный для браузера взгляд на этот набор данных».

Хорошие границы абстракций не просто упрощают рассуждения о внутреннем коде — они делают возможным для кого-то вроде меня прийти извне проекта и подключить новую среду выполнения, не трогая ядро. Это реальное достоинство дизайна, и благодаря ему 14 000 строк ощущались скорее как заполнение формы, чем как борьба с кодовой базой.
