## Проблема: вы не можете искать по таблице Lance из браузера

LanceDB — потрясающая векторная база данных. Она значительно проще и дешевле традиционных БД. Вы можете публиковать таблицы с векторными колонками, колонками для полнотекстового поиска (FTS), хранить файлы в S3 или CDN и запрашивать данные из S3 напрямую через Node.js сервер. Никаких демонов не требуется.

Что вы *не могли* делать до этого PR — так это открыть таблицу напрямую из браузера и запустить поиск. Вам требовался сервер-посредник: Lambda, контейнер или что-то еще, способное исполнять Rust, Node.js/WASM или Python.

Цель была амбициозной, но сфокусированной на конкретном результате: **векторные индексы с прямым поиском, доступные только для чтения и размещенные по HTTP.**

## Архитектура

Реализация состоит из трех уровней:

### 1. Артефакты публикации на Rust (`web_publish.rs`)

Прежде чем браузер сможет искать по таблице Lance, он должен знать, что искать безопасно. Внутренний формат Lance имеет особенности, не переносимые в браузер: дельта-файлы, внешние базы (external bases), типы индексов, которые среда выполнения WASM не может исполнить. Вместо того чтобы браузер выяснял это во время поиска (что плохо), этап публикации создает явные вспомогательные файлы (sidecar files):

- `_web.json` — объявляет схему и указывает, какие колонки доступны для поиска в браузере (векторные колонки с поддерживаемыми типами расстояний, колонки FTS).
- `_snapshot.json` — моментальный снимок данных на определенный момент времени, который браузерный клиент может использовать без необходимости понимать полный протокол эволюции набора данных.
- `_latest.manifest` / `_latest.version` — стабильные указатели, которые браузер опрашивает для обнаружения устаревания данных.

Флаг `isComplete` в снапшоте служит предохранителем. Если набор данных зависит от внешних баз или файлов, которые браузер не может безопасно прочитать, `isComplete=false`, и браузер отклоняет его на раннем этапе с понятной ошибкой, вместо того чтобы выдавать неверные результаты.

Это именно та часть, которую я рад был реализовать на Rust, а не на JavaScript. Решение о том, что безопасно для браузера, принимается там, где фактически живут метаданные таблицы.

### 2. Крейт `lancedb-wasm` на Rust

Это среда выполнения WebAssembly: хранилище объектов на базе `fetch`, поисковый движок на стороне браузера и вычислитель выражений для фильтрации предикатов.

Хранилище объектов `fetch` (`fetch_object_store.rs`) реализует трейт `ObjectStore` из Lance через HTTP Range-запросы. Внутренние операции чтения в Lance уже структурированы как запросы диапазонов байтов, что идеально ложится на заголовки `Range: bytes=N-M`. Это была самая приятная часть проекта — существующая архитектура была спроектирована почти идеально для этой задачи, мне оставалось только их соединить.

Браузерный движок (`browser.rs`) управляет поиском: поиск ближайших соседей по векторам, полнотекстовый поиск и гибридный поиск (вектор + FTS с использованием RRF-слияния). Выражения фильтрации (`browser_expr.rs`) вычисляют предикаты фильтров Lance на стороне клиента на чистом Rust, скомпилированном в WASM.

Отказ от неподдерживаемых функций строг и намерен. Расстояние `hamming`? Отклоняется сразу с четким сообщением — никакой скрытой некорректности или паники во время выполнения. `fastSearch` на неполном снапшоте? Отказ. Неподдерживаемые структуры путей? Отказ. Цель состояла в том, чтобы система была закрытой: если браузерный путь не может выполнить контракт, он так и говорит.

### 3. TypeScript пакет `@lancedb/lancedb-web`

Публичный API намеренно лаконичен:

```typescript
import { searchTable } from "@lancedb/lancedb-web";

const results = await searchTable("https://my-cdn.example.com/my-table", "semantic search query", {
  select: ["title", "url", "score"],
  limit: 10,
});
```

Внутри `searchTable()` предпочитает путь исполнения через Web Worker, чтобы поиск не блокировал основной поток. Модуль WASM запускается в воркере; результаты возвращаются через типизированный протокол (`worker_protocol.ts`). Также есть опциональный экспорт `./transformers`, который оборачивает `@xenova/transformers` для генерации эмбеддингов запроса на стороне клиента — так что вы можете пройти путь от текстового запроса до результатов векторного поиска, вообще не покидая браузер.

## Трудности

### Чтение диапазонов по HTTP не обходится бесплатно

Шаблоны чтения Lance подразумевают хранилище объектов, которое может эффективно обрабатывать множество мелких Range-запросов. S3 и GCS спроектированы для этого. Браузеры... в основном тоже, но у `fetch` с заголовками `Range` есть свои причуды, о которых абстракциям нативного объектного хранилища Rust заботиться не приходится.

Основная проблема в том, что браузеры иногда агрессивно буферизируют или повторяют запросы, и вы не можете контролировать пулинг TCP-соединений так же, как в нативном процессе. Для больших векторных индексов это имеет значение. Текущая реализация корректна; насколько она быстра для промышленного использования на огромных таблицах — покажет ревью.

### Вопрос генерации вспомогательных файлов

В PR этот вопрос вынесен отдельно для обсуждения с мейнтейнерами, так как здесь есть реальный компромисс в архитектуре: должны ли вспомогательные файлы для браузера (`_web.json`, `_snapshot.json`) создаваться *автоматически* при каждом комите в таблицу или должен быть явный вызов функции `publish()`?

Автоматический режим удобнее — вы получаете поддержку браузера «бесплатно». Но это означает дополнительные накладные расходы при каждой локальной записи и связывает контракт браузера с основным путем фиксации данных так, что это может усложнить будущие изменения в любой из частей.

Явная публикация более корректна с точки зрения контракта — вы подтверждаете: «эта версия готова для браузера» — но об этом легко забыть, и тогда таблицы, которые *должны* быть доступны для поиска, внезапно перестанут работать.

Я склонился к автоматическому варианту, где ошибки создания вспомогательных файлов записываются как предупреждения в лог, а не прерывают комит. Но я не уверен до конца и пометил этот момент в PR.

### Что считать «безопасным для браузера»?

Флаг `isComplete` и логика фильтрации колонок в `web_publish.rs` кодируют суждение о том, что браузер может обработать. Это суждение должно оставаться синхронизированным с реальными возможностями WASM-рантайма. Если кто-то добавит в Lance новый тип индекса, который браузер не сможет исполнить, код публикации должен уметь его исключить — иначе вы объявите о возможности, которую браузер не сможет реализовать во время поиска.

Правильным решением здесь, вероятно, будет реестр возможностей (capability registry), общий для пути публикации и рантайма, чтобы они не могли разойтись. Пока я реализовал это в виде параллельных констант, что быстро, но хрупко. Это именно то, что я бы хотел улучшить в следующую очередь.

## Дилемма обертки Transformers

Экспорт `./transformers` — это функция для удобства: она берет текстовый запрос, прогоняет его через локальную модель эмбеддингов с помощью `@xenova/transformers` и передает полученный вектор поисковому движку LanceDB. Семантический поиск с нулевыми затратами на сервер прямо со статической HTML-страницы.

И это же та часть, в необходимости которой в данном PR я уверен меньше всего. Это действительно полезно и уже реализовано, но это зависимость и API-поверхность, отдельная от основного вопроса браузерного поиска. Мейнтейнеры могут резонно захотеть сначала принять базовый API `searchTable()`, а эмбеддинги доработать отдельно.

Я спросил. Посмотрим.

## Что бы я сделал иначе

**Разделил бы на этапы раньше.** В этом PR 14 500 строк кода. Это слишком много для одного обзора. Я мог бы сначала предложить формат артефактов публикации, затем рантайм WASM, а после — пакет TypeScript. Три небольших PR было бы гораздо проще принять. Я не сделал этого, потому что хотел проверить всю историю от начала до конца, прежде чем предлагать какие-либо части — это имело смысл для уверенности в решении, но создало огромную нагрузку на ревьюеров.

**Сначала написал бы реестр возможностей.** Риск расхождения между `web_publish.rs` и `browser.rs` реален. Использование общего типа возможностей, на который ссылаются обе стороны, было бы чище, чем параллельные списки констант, которые мне нужно не забывать обновлять синхронно.

**Давал бы менее «умные» названия.** `_web.json` — нормальное название для внутреннего формата. Оно становится нагруженным в тот момент, когда кто-то начинает его кэшировать или строить инструменты вокруг него. Мне стоило уделить больше времени неймингу до отправки — это одна из тех вещей, которые гораздо дешевле менять, пока от них никто не зависит.

## Общий вывод

Проект стал выполнимым благодаря тому, что внутренняя архитектура Lance уже была хорошо структурирована для чтения диапазонов. Абстракция объектного хранилища чиста. Формат метаданных прозрачен. Концепция снапшота дала естественное место для выражения идеи «это безопасный для браузера вид этого набора данных».

Хорошие границы абстракций не только упрощают понимание внутреннего кода — они дают возможность кому-то вроде меня прийти извне проекта и подключить новую среду исполнения, не затрагивая ядро. Это настоящее архитектурное достоинство, благодаря которому 14 000 строк кода ощущались скорее как заполнение формы, чем как борьба с кодовой базой.

PR находится [здесь](https://github.com/lancedb/lancedb/pull/3247), если хотите изучить детали. Все еще жду ревью. Мейнтейнеры настроены дружелюбно, и я сохраняю осторожный оптимизм.
