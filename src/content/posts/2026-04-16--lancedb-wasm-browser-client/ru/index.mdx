## Проблема: таблицу Lance невозможно искать из браузера

LanceDB — фантастическая векторная база данных. Она значительно проще и дешевле традиционных баз. Вы можете публиковать таблицы с векторными колонками, колонками полнотекстового поиска (FTS), размещать файлы на S3 или CDN и выполнять запросы к данным через S3 напрямую с Node.js-сервера. Никакого демона не требуется.

Чего *нельзя* было сделать — до этого PR — так это открыть ту же таблицу прямо из браузера и выполнить поиск. Понадобился бы промежуточный сервер: Lambda, контейнер, *что-то*, способное запускать Rust, Node.js/WASM или Python.

Цель была амбициозной, но сфокусированной на осмысленном результате: **векторные индексы, доступные для поиска напрямую через HTTP, только на чтение.**

---

## Архитектура

Реализация состоит из трёх уровней:

### 1. Артефакты публикации на Rust (`web_publish.rs`)

Прежде чем браузер сможет выполнить поиск по Lance-таблице, ему нужно знать, что именно безопасно искать. Внутренний формат Lance содержит возможности, непереносимые в браузер — delta-файлы, внешние базы, типы индексов, которые среда выполнения WASM не может обработать. Вместо того чтобы браузер обнаруживал это в момент поиска (плохо), этап публикации создаёт явные вспомогательные файлы:

- `_web.json` — объявляет схему и какие колонки доступны для поиска из браузера (векторные колонки с поддерживаемыми типами расстояний, FTS-колонки)
- `_snapshot.json` — представление набора данных на конкретный момент времени, которое браузер может использовать без необходимости понимать весь протокол эволюции данных
- `_latest.manifest` / `_latest.version` — стабильные указатели, которые браузер опрашивает для обнаружения устаревания

Флаг `isComplete` в снимке — предохранительный клапан. Если набор данных зависит от внешних баз или файлов данных, которые браузерный путь не может безопасно прочитать, выставляется `isComplete=false`, и браузер отклоняет его на раннем этапе с понятной ошибкой, вместо того чтобы выдавать незаметно неверные результаты.

Это та часть, которую я больше всего рад, что реализовал на Rust, а не на JavaScript. Суждение о том, что безопасно для браузера, находится там, где на самом деле живут метаданные таблицы.

### 2. Крейт `lancedb-wasm` на Rust

Это среда выполнения WebAssembly: хранилище объектов на основе `fetch`, поисковый движок на стороне браузера и вычислитель выражений для фильтров-предикатов.

Хранилище объектов fetch (`fetch_object_store.rs`) реализует трейт `ObjectStore` от Lance через HTTP-запросы с диапазонами. Внутренние чтения Lance уже структурированы как ranged-запросы байтовых диапазонов, что отлично ложится на заголовки `Range: bytes=N-M`. Это была самая приятная часть проекта — существующая архитектура была почти идеально спроектирована для этого, мне оставалось только подключить её.

Браузерный движок (`browser.rs`) обрабатывает поиск: по векторной близости, полнотекстовый и гибридный (векторный + FTS с RRF-слиянием). Выражения фильтров (`browser_expr.rs`) вычисляют Lance-предикаты на стороне клиента на чистом Rust, скомпилированном в WASM.

Отклонение неподдерживаемых возможностей — строгое и намеренное. Расстояние `hamming`? Отклоняется сразу с понятным сообщением — не безмолвно неверно, не паника в рантайме. `fastSearch` на неполном снимке? Отклоняется. Неподдерживаемые схемы базовых путей? Отклоняются. Цель — закрытый отказ: если браузерный путь не может выполнить контракт, он говорит об этом.

### 3. TypeScript-пакет `@lancedb/lancedb-web`

Публичная поверхность API намеренно мала:

```typescript
import { searchTable } from "@lancedb/lancedb-web";

const results = await searchTable("https://my-cdn.example.com/my-table", "семантический поисковый запрос", {
  select: ["title", "url", "score"],
  limit: 10,
});
```

Под капотом `searchTable()` предпочитает путь выполнения через Worker, чтобы поиск не блокировал главный поток. Модуль WASM выполняется в Worker; результаты возвращаются по типизированному протоколу (`worker_protocol.ts`). Также присутствует опциональный экспорт `./transformers`, оборачивающий `@xenova/transformers` для генерации эмбеддингов запроса на стороне клиента — так что вы можете перейти от текстового запроса к результатам векторного поиска, ни разу не покинув браузер.

## Сложные моменты

### Чтение с диапазонами по HTTP — не бесплатно

Шаблоны чтения Lance предполагают хранилище объектов, способное эффективно обрабатывать множество мелких ranged-запросов. S3 и GCS спроектированы для этого. Браузеры... в основном тоже, но у `fetch` с заголовками `Range` есть особенности, о которых существующим абстракциям хранилища Rust не приходится заботиться.

Основная проблема в том, что браузеры иногда агрессивно буферизируют или повторно запрашивают данные, и вы не можете контролировать пул TCP-соединений так, как в нативном процессе. Для больших векторных индексов это имеет значение. Текущая реализация корректна; достаточно ли она быстра для production-использования на больших таблицах — выяснится в ходе ревью.

### Вопрос генерации вспомогательных файлов

PR явно поднимает этот вопрос для мейнтейнеров, поскольку есть реальный компромисс в дизайне: должны ли вспомогательные файлы для браузера (`_web.json`, `_snapshot.json`) создаваться *автоматически* при каждом коммите таблицы, или нужен явный вызов `publish()`?

Автоматический режим эргономичнее — вы получаете поддержку браузера бесплатно. Но это означает небольшие накладные расходы при каждой локальной записи и связывает браузерный контракт с основным путём коммита таким образом, что может усложнить будущие изменения в любой из сторон.

Явный publish более корректен с точки зрения контракта — вы утверждаете: «эта версия готова для браузера» — но его легко забыть, из-за чего таблицы, которые *должны* быть доступны для поиска из браузера, молча таковыми не становятся.

Я склонялся к автоматическому режиму, при котором ошибки создания вспомогательных файлов логируются как предупреждения, а не прерывают коммит. Но я искренне не уверен и указал это в PR.

### Что считать «безопасным для браузера»?

Флаг `isComplete` и логика фильтрации колонок в `web_publish.rs` кодируют суждение о том, что браузер может обработать. Это суждение должно оставаться синхронизированным с реальными возможностями среды выполнения WASM. Если кто-то добавит в Lance новый тип индекса, который браузер не может выполнить, код публикации должен знать, что его нужно исключить — иначе вы объявите о возможности, которую браузер не сможет обеспечить в момент поиска.

Правильное решение здесь, вероятно, — реестр возможностей, общий для пути публикации и среды выполнения, чтобы они не могли рассинхронизироваться. Я реализовал это в виде параллельных констант — это практично, но хрупко. Это, вероятно, то, что я хотел бы улучшить в первую очередь.

## Дилемма обёртки Transformers

Экспорт `./transformers` — это функция качества жизни: он принимает текстовый запрос, прогоняет его через локальную модель эмбеддингов с помощью `@xenova/transformers` и передаёт полученный вектор поисковому движку LanceDB. Полностью семантический поиск без сервера из статической HTML-страницы.

И это же — то, в чём я меньше всего уверен относительно включения в этот PR. Это действительно полезно и уже реализовано, но это зависимость и поверхность API, отдельные от основного вопроса браузерного поиска. Мейнтейнеры вполне обоснованно могут захотеть сначала принять базовый API `searchTable()` и итерировать эмбеддинги отдельно.

Я спросил. Посмотрим.

## Что бы я сделал иначе

**Сузить объём раньше.** PR составляет 14 500 строк. Это много для ревью за один раз. Я мог бы сначала опубликовать формат артефактов, затем среду выполнения WASM, затем TypeScript-пакет. Три небольших PR было бы легче смержить. Я этого не сделал, потому что хотел проверить полную сквозную историю, прежде чем предлагать какие-либо части — это имело смысл для уверенности, но увеличило нагрузку на ревью.

**Написать реестр возможностей в первую очередь.** Риск рассинхронизации между `web_publish.rs` и `browser.rs` реален. Начало с общего типа возможностей, на который ссылаются обе стороны, было бы чище, чем параллельные списки констант, которые нужно не забывать синхронизировать.

**Называть вещи менее заумно.** `_web.json` — нормальное имя для внутреннего формата. Оно становится критичным в тот момент, когда кто-то начинает его кешировать или строить вокруг него инструментарий. Я должен был потратить больше времени на именование перед отправкой — это одна из тех вещей, которые гораздо дешевле изменить, пока от них никто не зависит.

## Более широкая мысль

То, что сделало этот проект реализуемым — это то, что внутренняя архитектура Lance уже была хорошо структурирована для ranged-чтения. Абстракция хранилища объектов чиста. Формат метаданных явный. Концепция снимка дала мне естественное место для выражения «это безопасное для браузера представление этого набора данных».

Хорошие границы абстракций не только облегчают понимание внутреннего кода — они позволяют кому-то вроде меня прийти извне проекта и подключить новую среду выполнения, не затрагивая ядро. Это настоящее архитектурное достоинство, и оно сделало эти 14 000 строк скорее заполнением готовой формы, чем борьбой с кодом.

PR находится [здесь](https://github.com/lancedb/lancedb/pull/3247), если хотите увидеть детали. Всё ещё жду ревью. Мейнтейнеры были радушны, и я осторожно оптимистичен.
