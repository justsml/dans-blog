## Il Problema: Non si Può Cercare una Tabella Lance dal Browser

LanceDB è un database vettoriale fantastico. È enormemente più semplice ed economico dei database tradizionali. Puoi pubblicare tabelle con colonne vettoriali e colonne per la ricerca full-text (FTS), ospitare i file su S3 o un CDN, e interrogare i dati direttamente da un server Node.js via S3. Nessun demone richiesto.

Quello che *non* si poteva fare — prima di questa PR — era aprire quella tabella direttamente dal browser ed eseguire una ricerca. Serviva un server intermediario: una Lambda, un container, *qualcosa* capace di eseguire Rust, Node.js/WASM o Python.

L'obiettivo era ambizioso, ma focalizzato su un risultato concreto: **indici vettoriali ospitati via HTTP, direttamente interrogabili e in sola lettura.**

---

## L'Architettura

L'implementazione si articola su tre livelli:

### 1. Artefatti di pubblicazione Rust (`web_publish.rs`)

Prima che un browser possa cercare in una tabella Lance, deve sapere cosa è sicuro cercare. Il formato interno di Lance ha alcune funzionalità che non sono portabili nel browser — file delta, basi esterne, tipi di indice che il runtime WASM non può eseguire. Invece di far scoprire tutto questo al browser al momento della ricerca (male), il passaggio di pubblicazione produce dei file sidecar espliciti:

- `_web.json` — dichiara lo schema e quali colonne sono ricercabili dal browser (colonne vettoriali con tipi di distanza supportati, colonne FTS)
- `_snapshot.json` — una vista puntuale del dataset che il client browser può consumare senza dover comprendere l'intero protocollo di evoluzione del dataset
- `_latest.manifest` / `_latest.version` — puntatori stabili che il browser interroga periodicamente per rilevare dati obsoleti

Il flag `isComplete` nello snapshot è la valvola di sicurezza. Se il dataset dipende da basi esterne o file di dati che il percorso browser non può leggere in modo sicuro, `isComplete=false` e il browser lo rifiuta immediatamente con un errore chiaro, invece di produrre risultati silenziosamente errati.

Questa è la parte di cui sono più felice di aver implementato in Rust anziché in JavaScript. La logica che stabilisce cosa è sicuro per il browser vive dove risiedono effettivamente i metadati della tabella.

### 2. Crate Rust `lancedb-wasm`

Questo è il runtime WebAssembly: un object store basato su `fetch`, un motore di ricerca lato browser e un valutatore di espressioni per i predicati di filtro.

L'object store fetch (`fetch_object_store.rs`) implementa il trait `ObjectStore` di Lance tramite richieste HTTP con range. Le letture interne di Lance sono già strutturate come recuperi di intervalli di byte, che si mappano in modo pulito sugli header `Range: bytes=N-M`. Questa è stata la parte più soddisfacente del progetto — l'architettura esistente era quasi perfettamente progettata per questo, bastava collegare i pezzi.

Il motore browser (`browser.rs`) gestisce la ricerca: nearest-neighbor vettoriale, full-text e ibrida (vettoriale + FTS con fusione RRF). Le espressioni di filtro (`browser_expr.rs`) valutano i predicati di filtro Lance lato client in Rust puro, compilato in WASM.

Il rifiuto delle funzionalità non supportate è rigoroso e intenzionale. Distanza `hamming`? Rifiutata subito con un messaggio chiaro — non un risultato silenziosamente errato, non un panic a runtime. `fastSearch` su uno snapshot incompleto? Rifiutata. Layout di percorso base non supportati? Rifiutati. L'obiettivo era fallire in modo conservativo: se il percorso browser non può rispettare il contratto, lo dichiara esplicitamente.

### 3. Pacchetto TypeScript `@lancedb/lancedb-web`

La superficie API pubblica è intenzionalmente ridotta:

```typescript
import { searchTable } from "@lancedb/lancedb-web";

const results = await searchTable("https://my-cdn.example.com/my-table", "semantic search query", {
  select: ["title", "url", "score"],
  limit: 10,
});
```

Sotto il cofano, `searchTable()` preferisce un percorso di esecuzione basato su Worker in modo che la ricerca non blocchi il thread principale. Il modulo WASM gira nel Worker; i risultati tornano tramite un protocollo tipizzato (`worker_protocol.ts`). C'è anche un export opzionale `./transformers` che incapsula `@xenova/transformers` per generare embedding di query lato client — così puoi passare da una query testuale grezza a risultati di ricerca vettoriale senza mai uscire dal browser.

## Le Parti Difficili

### Le letture con range su HTTP non sono gratuite

I pattern di lettura di Lance presuppongono un object store in grado di gestire in modo efficiente molti piccoli recuperi con range. S3 e GCS sono progettati per questo. I browser... per lo più anche, ma `fetch` con header `Range` ha alcune peculiarità di cui le astrazioni Rust esistenti per gli object store non devono preoccuparsi.

Il problema principale è che i browser a volte bufferizzano o richiedono in modo aggressivo, e non puoi controllare il connection pooling TCP come faresti in un processo nativo. Per indici vettoriali di grandi dimensioni questo conta. L'implementazione attuale è corretta; se sia sufficientemente veloce per l'uso in produzione su tabelle grandi è qualcosa che emergerà durante la revisione.

### La questione della generazione dei sidecar

La PR lo solleva esplicitamente come domanda aperta per i manutentori, perché c'è un reale compromesso progettuale: i file sidecar del browser (`_web.json`, `_snapshot.json`) dovrebbero essere generati *automaticamente* ad ogni commit sulla tabella, o dovrebbe esserci una chiamata esplicita a `publish()`?

L'approccio automatico è più ergonomico — ottieni il supporto browser gratis. Ma significa che ogni scrittura locale ha un piccolo overhead aggiuntivo, e accoppia il contratto browser al percorso principale di commit in modi che potrebbero complicare future modifiche a entrambi.

La pubblicazione esplicita è più corretta dal punto di vista del contratto — stai dicendo "questa versione è pronta per il browser" — ma è facile dimenticarsene, il che significa che tabelle che *dovrebbero* essere ricercabili dal browser in realtà non lo sono, senza dare alcun segnale.

Io ho proposto l'approccio automatico, con eventuali errori nella generazione dei sidecar che registrano un avviso anziché far fallire il commit. Ma ho dei dubbi reali, e l'ho segnalato nella PR.

### Cosa si intende per "sicuro per il browser"?

Il flag `isComplete` e la logica di filtraggio delle colonne in `web_publish.rs` codificano un giudizio su ciò che il browser può gestire. Quel giudizio deve rimanere sincronizzato con le capacità reali del runtime WASM. Se qualcuno aggiunge un nuovo tipo di indice a Lance che il browser non può eseguire, il codice di pubblicazione deve sapere di escluderlo — altrimenti ti ritrovi a dichiarare una capacità che il browser non riuscirà a soddisfare al momento della ricerca.

La soluzione corretta qui sarebbe probabilmente un registro delle capacità condiviso tra il percorso di pubblicazione e il runtime, in modo che non possano divergere. Per ora l'ho implementato con costanti parallele, che è pragmatico ma fragile. Questa è probabilmente la prima cosa che mi piacerebbe migliorare.

## Il Dilemma del Wrapper Transformers

L'export `./transformers` è una comodità: prende una query testuale, la passa attraverso un modello di embedding locale tramite `@xenova/transformers`, e consegna il vettore risultante al motore di ricerca LanceDB. Ricerca semantica senza alcun server, da una pagina HTML statica.

È anche la cosa di cui sono meno sicuro che appartenga a questa PR. È genuinamente utile ed è già implementata, ma è una dipendenza e una superficie API separata dalla questione fondamentale della ricerca nel browser. I manutentori potrebbero ragionevolmente voler integrare prima l'API base `searchTable()` e iterare sugli embedding separatamente.

Ho chiesto. Vedremo.

## Cosa Farei Diversamente

**Delimitare l'ambito prima.** La PR è di 14.500 righe. È molto da revisionare in un solo colpo. Avrei potuto integrare prima il formato degli artefatti di pubblicazione, poi il runtime WASM, poi il pacchetto TypeScript. Tre PR più piccole sarebbero state più facili da mergiare. Non l'ho fatto perché volevo validare l'intera storia end-to-end prima di proporre qualsiasi singolo pezzo — sensato per la sicurezza, ma rende il carico di revisione elevato.

**Scrivere prima il registro delle capacità.** Il rischio di divergenza tra `web_publish.rs` e `browser.rs` è reale. Partire con un tipo di capacità condiviso a cui entrambi i lati fanno riferimento sarebbe stato più pulito rispetto a liste di costanti parallele che devo ricordare di mantenere sincronizzate.

**Dare nomi meno "furbi".** `_web.json` va bene come nome per un formato interno. Diventa un nome strutturale nel momento in cui qualcuno lo mette in cache o ci costruisce strumenti attorno. Avrei dovuto dedicare più tempo alla nomenclatura prima di inviare la PR — è una di quelle cose molto più economiche da cambiare prima che qualcun altro ci faccia affidamento.

## Il Quadro Più Ampio

Ciò che ha reso questo progetto realizzabile è che l'architettura interna di Lance era già ben strutturata per le letture con range. L'astrazione dell'object store è pulita. Il formato dei metadati è esplicito. Il concetto di snapshot mi ha dato un punto naturale in cui esprimere "questa è una vista sicura per il browser di questo dataset."

Dei buoni confini di astrazione non rendono solo il codice interno più facile da ragionare — rendono possibile per qualcuno come me arrivare da fuori dal progetto e collegare un nuovo ambiente di esecuzione senza toccare il nucleo. Questa è una vera virtù progettuale, e ha fatto sì che le 14.000 righe sembrassero più un riempire una forma già tracciata che un combattere contro la codebase.

La PR è [qui](https://github.com/lancedb/lancedb/pull/3247) se vuoi vedere i dettagli. Aspetto ancora la revisione. I manutentori sono stati accoglienti e sono cautamente ottimista.
