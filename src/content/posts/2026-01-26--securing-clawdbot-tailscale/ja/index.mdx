OpenClaw（旧Clawdbot/Moltbot）は、WhatsApp、Slack、Discord、iMessageなどのチャンネルを横断して動作するパーソナルAIアシスタントを提供します。しかし、ゲートウェイ、ノードコントロール、SSHを強力な認証なしにパブリックインターネット上に晒すと、見知らぬ人にあなたのマシンへのシェルアクセスへの経路を与えることになります。

このガイドでは、最も安全なデフォルト設定を紹介します。OpenClawのゲートウェイをループバック上に留め、Tailscale Serveでのみテールネットに公開し、SSHをロックダウンし、外部からゲートウェイが公開されていないことを検証します。

プロジェクトの急激な採用は現実のセキュリティ懸念を浮き彫りにしました。[Shodanスキャンは最初の数週間で2,847件の公開インスタンスを発見](https://socradar.io/blog/clawdbot-is-it-safe/)し、[GitHubセキュリティ監査イシューではコードベースで512件の発見が報告](https://github.com/moltbot/moltbot/issues/1796)されました。その一部は自動スキャナーの出力であり、2026年1月のOpenClawへのリネーム以降に変更されたものもあるため、この数字は正確な現在の脆弱性カウントというより警告サインとして捉えてください。セキュリティの専門家になる必要はありません。デプロイする前にオペレーター向けサーフェスを公開しないようにすればいいのです。

## 実際に何を公開しているのか

インストール方法と公開方法に応じて、チェックすべきサーフェスが3つあります。

- **ポート22**: VPS上のSSHアクセス
- **ポート18789**: ゲートウェイコントロールUIとWebSocket API
- **ブラウザ/ノードコントロール**: ゲートウェイ/ノードペアリングモデルを通じたリモートノード実行とブラウザ自動化

現在の[OpenClawリモートアクセスドキュメント](https://docs.molt.bot/gateway/remote)によれば、ゲートウェイのWebSocketはデフォルトでループバックにバインドされ、意図的にLAN/テールネット/カスタムバインドを選択しない限りループバックのみの使用が推奨されています。これは良い設計です。リスクは、そのデフォルトを上書きした場合、Dockerポートを公開した場合、リバースプロキシを追加した場合、Funnelをオンにした場合、またはSSHを全世界に開放した場合に発生します。

ゲートウェイが最も重要です。アシスタントのオペレーター向けサーフェスであり、ツール呼び出しパスを含みます。インターネットから到達可能で認証が欠如、脆弱、バイパス、または漏洩している場合、攻撃者がエージェントを操作したり、あなたのユーザー権限でツールを呼び出したりできる可能性があります。

ブラウザコントロールも同様に機密性が高いです。現在のOpenClawドキュメントでは、ブラウザコントロールをブラウザマシン上のペアリングされたノードホスト経由で実行し、ノードペアリングをオペレーターアクセスとして扱うことが推奨されています。ゲートウェイがペアリングされたノードで `system.run` を呼び出せる場合、それはゲートウェイのノードポリシーとノード自身のexec承認に従った上で、そのノード上のリモートコード実行となります。

SSHはSSHです。パスワード認証を有効にして実行している場合、パブリックVPSではブルートフォース攻撃が不可避です。

## Tailscaleソリューション

OpenClawにとって、Tailscaleはオペレーターサービスを公開せずにリモートアクセスを提供します。

1. OpenClawインスタンスがVPSまたはローカルマシン上で動作
2. ゲートウェイはループバック上に留まり、Tailscale Serve経由でのみ公開されるか、明示的な認証とともにテールネットIPに直接バインド
3. サーバーとパーソナルデバイスの両方にTailscaleをインストール
4. Tailscale IPまたはMagicDNS名経由でOpenClawにアクセス
5. インターネット上の他の全員は何も見えません。Funnelや他のパブリックプロキシを意図的に有効にしない限り

### OpenClawにTailscaleを管理させるべきか？

OpenClawには[組み込みのTailscale統合](https://docs.molt.bot/gateway/tailscale)があり、ゲートウェイ用に `tailscale serve` または `tailscale funnel` を設定できます。

**Serveモード**はテールネット内だけに留めます。ゲートウェイは `127.0.0.1` にバインドされたまま、TailscaleがルーティングとHTTPSを処理します。`gateway.auth.allowTailscale` が有効な場合、OpenClawはTailscale IDヘッダーを使用してコントロールUI/WebSocketトラフィックを認証し、`tailscale whois` で送信元を検証できます。これはほとんどのパーソナルデプロイメントにおいて適切なモードです。

**Funnelモード**はTailscaleのパブリックエンドポイント機能を通じてゲートウェイを公開します。Tailscale自身のドキュメントでは、Funnelはより広範なインターネットからのトラフィックをローカルサービスにルーティングするものと説明されています。OpenClawはゲートウェイ認証モードが `password` でない限りFunnelの起動を拒否しますが、それでもオペレーター向けサーフェスの公開暴露を選択することになります。

OpenClawの[セキュリティドキュメント](https://docs.molt.bot/gateway/security)は、プロンプトインジェクションとツールアクセスがパーソナルアシスタントのコアリスクであることを明確にしています。エージェントが静かに自身を公開にする経路を与えないでください。Serveは意図的に使用し、本当に公開アクセスが必要でない限りFunnelは避け、`tailscale` コマンドにはexec承認を要求してください。

## OpenClawをセキュアに設定する

### ステップ1: Tailscaleをインストール

VPSまたはローカルサーバー上:

```bash
# Tailscaleをインストール
curl -fsSL https://tailscale.com/install.sh | sh

# 認証（ログイン用のブラウザが開く）
sudo tailscale up

# Tailscale IPを取得
tailscale ip -4
# 出力: 100.x.x.x
```

クライアントマシンでは、公式ダウンロードページからTailscaleをインストールし、同じテールネットにサインインしてください。

これで両方のマシンが同じプライベートネットワーク上にあります。Tailscale IPを使用してVPSにpingを送ることができ、暗号化されたトンネルを経由してルーティングされます。

### ステップ2: OpenClawをTailscale使用に設定

現在の最も安全なパターンは、ゲートウェイをループバック上に留め、Tailscale Serveでテールネットに公開することです。

OpenClawの設定で:

```js
{
  gateway: {
    bind: "loopback",
    tailscale: { mode: "serve" },
  },
}
```

その後、Serveでゲートウェイを起動:

```bash
openclaw gateway --tailscale serve
```

OpenClawのドキュメントによれば、これによりゲートウェイは `127.0.0.1` 上に留まり、TailscaleがHTTPSとテールネットルーティングを提供します。パブリックVPS IPではなく `https://<magicdns-name>/` で開きます。

Serveの代わりに直接テールネットバインドを好む場合は、明示的なゲートウェイ認証を使用します:

```js
{
  gateway: {
    bind: "tailnet",
    auth: {
      mode: "token",
      token: "replace-with-a-long-random-token",
    },
  },
}
```

別のテールネットデバイスから接続:

```text
http://<tailscale-ip>:18789/
ws://<tailscale-ip>:18789
```

Dockerや他のコンテナランタイムで実行している場合、ポート公開には特に注意してください。`-p 18789:18789` のような公開は通常、すべてのホストインターフェースにバインドします。ループバック＋Tailscale Serveを優先するか、コンテナが引き続きトラフィックを受信することを確認した後、ホスト側をTailscale IPに明示的にバインドしてください:

```bash
TAILSCALE_IP=$(tailscale ip -4)
docker run ... -p "$TAILSCALE_IP:18789:18789" ...
```

Docker変更後は、外部から `nmap` で、ローカルから `ss` で確認してください。Dockerは、考慮しないとホストファイアウォールの想定をバイパスまたは順序変更する可能性があります。

### ステップ3: SSHをロックダウン

Tailscaleを使用していても、SSHを適切にセキュア化する必要があります:

```bash
# これを行っている間、現在のSSHセッションを開いたままにしてください。
# まず、クライアントマシンから、Tailscale経由でSSHできることを確認:
ssh your-user@SERVER_TAILSCALE_IP

# sshd_configを書き換えるのではなく、ドロップインファイルにハードニングを配置。
sudo tee /etc/ssh/sshd_config.d/99-openclaw-hardening.conf >/dev/null <<'EOF'
PasswordAuthentication no
PermitRootLogin no
KbdInteractiveAuthentication no
EOF

# リロード前に検証。これをスキップしないでください。
sudo sshd -t
sudo systemctl reload ssh || sudo systemctl reload sshd
```

これにより、パスワードベースのログインとrootログインが無効になります。次のステップではUFWを使用してパブリックSSHを完全に防ぎつつ、`tailscale0` 経由のSSHは許可します。

### ステップ4: ファイアウォールルール

2番目のレイヤーとしてファイアウォールを設定:

```bash
# UFWを使用（Ubuntu/Debian）
sudo ufw allow in on tailscale0
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw enable
sudo ufw delete allow 22/tcp || true
sudo ufw reload
sudo ufw status verbose
```

Tailscale自身のUbuntuハードニングガイドも同じ形状を使用しています。`tailscale0` を許可し、他の受信トラフィックを拒否し、パブリックSSHがタイムアウトする一方で `100.x.y.z` アドレスへのSSHがまだ機能することを確認します。同じVPSでパブリックウェブサイトを実行している場合、本当に必要なパブリックルール（`80/tcp` や `443/tcp` など）のみを保持してください。

## 公開状態を確認する

### 外部から開いているポートを確認

Tailscaleネットワーク上に**ない**マシンから:

```bash
# 一般的な公開ポートが公開されているか確認
nmap -p 22,80,443,18789 YOUR_PUBLIC_IP

# セキュアなインスタンスの期待される出力:
# 22/tcp   filtered ssh
# 18789/tcp filtered unknown
```

`22` または `18789` が `filtered` や `closed` ではなく `open` と表示される場合、問題があります。`80` や `443` が開いている場合、それが意図的なパブリックウェブサイトやTailscale Funnelエンドポイントのみであり、誤ってOpenClawゲートウェイでないことを確認してください。

### ローカルで何がリッスンしているか確認

OpenClawサーバー上:

```bash
# リッスンしているすべてのポートとバインド先を表示
sudo ss -tulpn | grep LISTEN

# このような行を探す（Serveの場合は良い）:
# tcp   LISTEN 0   128   127.0.0.1:18789   *:*
#
# またはこれ（認証付きの直接テールネットバインドの場合は許容）:
# tcp   LISTEN 0   128   100.x.y.z:18789   *:*
#
# これはダメ（悪い）:
# tcp   LISTEN 0   128   0.0.0.0:18789     *:*
```

`0.0.0.0` または `:::`（IPv6同等）が表示される場合、そのサービスは全世界に公開されています。

### 組み込みセキュリティ監査

OpenClawには、設定をセキュリティベストプラクティスに対してチェックする[セキュリティ監査コマンド](https://docs.molt.bot/gateway/security)が含まれています:

```bash
openclaw security audit --deep
openclaw security audit --deep --fix
```

監査は、ゲートウェイの公開状態、Tailscaleモード、認証設定、チャンネルアクセス、ツールポリシー、プラグインインベントリ、ファイル権限をチェックします。`--fix` は有用なヘルパーとして扱ってください。発見事項を読むことの代わりにはなりません。

## これが解決しないこと

Tailscaleは最大のミス、つまりオペレーター向けの公開暴露を除去します。しかし、すべてを解決するわけではありません:

**認証情報の保存**: OpenClawはセッション記録、OAuthトークン、APIキーをディスクに保存します。適切なファイル権限（ファイルには `chmod 600`、プライベート設定ディレクトリには `chmod 700`）が設定されており、バージョン管理に含まれていないことを確認してください。組み込みの監査ツールがこれをチェックします。

**プラグインのサンドボックス化**: プラグインはあなたのユーザーの完全な権限で実行されます。信頼できるソースからのみプラグインをインストールし、要求する機能をレビューしてください。監査ツールはインストールされたプラグインをインベントリ化します。

**デバイスのセキュリティ**: 誰かがあなたのTailscaleアカウントを侵害したり、テールネット上のデバイスを盗んだりした場合、OpenClawインスタンスにアクセスできます。[Tailscaleデバイス認証](https://tailscale.com/kb/1099/device-authorization/)を有効にして、新しいデバイスに承認を要求してください。

## デプロイメントチェックリスト

OpenClaw/Moltbotインスタンスを本番対応と見なす前に:

- [ ] Tailscaleがサーバーとクライアントの両方にインストールされ、認証済み
- [ ] ゲートウェイはループバック上に留まりTailscale Serveを使用、または明示的な認証付きで `tailnet` にバインド
- [ ] SSHはパスワード認証とrootログインを無効にするように設定
- [ ] ファイアウォール（UFWまたはiptables/nftables）が `tailscale0` を許可し、不要なパブリックイングレスを拒否するように設定
- [ ] 外部nmapスキャンですべてのポートが `filtered` または `closed` を示す
- [ ] 内部 `ss -tulpn` でゲートウェイが `127.0.0.1`、`::1`、またはTailscale IPのみにバインドされていることを確認
- [ ] 認証情報ファイルの権限が600、プライベート設定ディレクトリの権限が700
- [ ] `openclaw security audit --deep` を実行し、すべての発見事項に対処
- [ ] OpenClawのTailscale管理を使用している場合、exec承認が有効
- [ ] 定期バックアップが設定済み（OpenClawデータ＋設定）
