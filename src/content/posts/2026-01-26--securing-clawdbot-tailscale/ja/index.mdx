OpenClaw（旧Clawdbot/Moltbot）は、WhatsApp、Slack、Discord、iMessage、その他のチャンネルで動作するパーソナルAIアシスタントを提供します。しかしゲートウェイ、ノードコントロール、SSHを認証なしでパブリックインターネットに公開すると、見知らぬ人にマシンへのシェルアクセスへの道を許してしまうことになります。

このガイドでは最も安全なデフォルトを示します：OpenClawのゲートウェイをループバックに保ち、Tailscale Serveでだけtailnetに公開し、SSHをロックダウンし、ゲートウェイが公開されていないことを外部から検証します。

プロジェクトの急速な普及により 실제のセキュリティ上の懸念が浮かび上がりました：[Shodanスキャンで最初の数週間で2,847の露出インスタンスが見つかりました](https://socradar.io/blog/clawdbot-is-it-safe/)。また[GitHubセキュリティ監査でコードベースに512件の發現](https://github.com/moltbot/moltbot/issues/1796)がありました。報告の一部は自動スキャナーの出力であり、2026年1月のOpenClawへの改名以降に変化した部分もあります。したがって、数字は正確な現在の脆弱性数ではなく警告信号として扱ってください。セキュリティエキスパートになる必要はありませんmdash デプロイ前にオペレーターサーフェスを公開しないだけで十分です。

## 本当に公開しているもの

インストール方法和公開方法によっては、確認すべき3つのサーフェスがあります：

- **Port 22**: VPS上のSSHアクセス
- **Port 18789**: ゲートウェイコントロールUIとWebSocket API
- **ブラウザ/ノードコントロール**: ゲートウェイ/ノードペアリングモデルを通じたリモートノード実行とブラウザ自動化

現在の[OpenClawリモートアクセスドキュメント](https://docs.molt.bot/gateway/remote)は、ゲートウェイWebSocketがデフォルトでループバックにバインドされ、LAN/tailnet/カスタムバインドを意図的に選択しない限り、ループバックのみに保つことを推奨しています。これは正しいです。リスクは、デフォルトをオーバーライドし、Dockerポートを公開しリバースプロキシを追加し、Funnelを有効にし、SSHを世界に公開したままにした時に発生します。

ゲートウェイが重要です。それはアシスタントのオペレーターサーフェスであり、ツール呼び出しパスを含みます。インターネットから到達可能で認証がない、薄弱的、回避可能、または漏洩している場合、攻撃者はエージェントを操ったり、ユーザーの権限でツールを呼び出したりできるかもしれません。

ブラウザコントロールもほぼ同等に敏感です。現在のOpenClawドキュメントは、ブラウザコントロールをブラウザマシン上のペアリングされたノードホスト経由で実行し、ノードペアリングをオペレーターアクセスと同様に扱うことを推奨しています。ゲートウェイがペアリングされたノードで`system.run`を呼び出せる場合、それはそのノードでのリモートコード実行であり、ゲートウェイのノードポリシーとそのノード自体のexec承認に従います。

SSHはSSHです。パスワード認証有効な状態で実行している場合、パブリックVPSでのブルートフォース試行は避けられません。

## Tailscaleソリューション

OpenClawにとって、Tailscaleはオペレーターサービスを公開せずにリモートアクセスを提供します：

1. OpenClawインスタンスがVPSまたはローカルマシンで動作
2. ゲートウェイがループバックにバインドされたまま、Tailscale Serve経由で到達、または明示的な認証でtailnet IPに直接バインド
3. サーバーとパーソナルデバイスの両方にTailscaleをインストール
4. Tailscale IPまたはMagicDNS名でOpenClawにアクセス
5. Funnelまたはその他のパブリックプロキシを意図的に有効にしない限り、インターネット上の他のユーザーは何も見えない

### OpenClawにTailscale管理させるべきか？

OpenClawにはゲートウェイ用に`tailscale serve`または`tailscale funnel`を構成できる[組み込みTailscale統合](https://docs.molt.bot/gateway/tailscale)があります。

**Serveモード**はtailnet上に留めます。ゲートウェイは`127.0.0.1`にバインドされたまま、TailscaleがルーティングとHTTPSを処理します。`gateway.auth.allowTailscale`が有効な場合、OpenClawはTailscale IDヘッダーを使用してControl UI/WebSocketトラフィックを認証し、`tailscale whois`でソースを検証できます。これは大多数のパーソナルデプロイメントに適したモードです。

**Funnelモード**はTailscaleのパブリックエンドポイント機能でゲートウェイを公的に公開します。Tailscale自らのドキュメントはFunnelを公のインターネットからローカルサービスへのトラフィックルーティングとして説明しています。OpenClawはゲートウェイ認証モードが`password`でない限りFunnelでの起動を拒否しますが、オペレーターサーフェスの公開を選んだことになります。

OpenClawの[セキュリティドキュメント](https://docs.molt.bot/gateway/security)は、プロンプトインジェクションとツールアクセスがパーソナルアシスタントのコアリスクであることを明示しています。エージェントに静かに自身を公開する経路を与えないでください。Serveは意図的に使用し、真正にパブリックアクセスを必要とする場合を除いてFunnelを避け、`tailscale`コマンドのexec承認が必要です。

## OpenClawを安全にセットアップする

### Step 1: Tailscaleのインストール

VPSまたはローカルサーバー上：

```bash
# Tailscaleのインストール
curl -fsSL https://tailscale.com/install.sh | sh

# 認証（ブラウザが開いてログイン）
sudo tailscale up

# Tailscale IPの取得
tailscale ip -4
# 出力: 100.x.x.x
```

クライアントマシンでは、公式サイトダウンロードページからTailscaleをインストールし、同じtailnetにサインインします。

これで両方のマシンが同じプライベートネットワーク上にあります。Tailscale IPを使用してVPSにpingでき、暗号化されたトンネルを経由してルーティングされます。

### Step 2: OpenClawがTailscaleを使用するよう構成する

現在最も安全なパターンは：ゲートウェイをループバックに保ち、Tailscale Serveでtailnetに公開することです。

OpenClaw設定で：

```js
{
  gateway: {
    bind: "loopback",
    tailscale: { mode: "serve" },
  },
}
```

次にServeでゲートウェイを起動します：

```bash
openclaw gateway --tailscale serve
```

OpenClawのドキュメントによると、これによりゲートウェイは`127.0.0.1`のまま、TailscaleがHTTPSとtailnetルーティングを提供します。パブリックVPS IPではなく`https://<magicdns-name>/`で開きます。

Serveの代わりに直接tailnetバインドを好む場合、明示的なゲートウェイ認証を使用します：

```js
{
  gateway: {
    bind: "tailnet",
    auth: {
      mode: "token",
      token: "replace-with-a-long-random-token",
    },
  },
}
```

次に別のtailnetデバイスから接続します：

```text
http://<tailscale-ip>:18789/
ws://<tailscale-ip>:18789
```

Dockerまたはその他のコンテナランタイムで実行している場合、ポート公開には特に注意を払ってください。`-p 18789:18789`のような公開は通常すべてのホストインターフェースにバインドします。ループバック+Tailscale Serve、またはコンテナがまだトラフィックを受信することを確認してから明示的にTailscale IPにホスト側をバインドすることを優先してください：

```bash
TAILSCALE_IP=$(tailscale ip -4)
docker run ... -p "$TAILSCALE_IP:18789:18789" ...
```

Dockerの変更後は必ず外側から`nmap`で、ローカルで`ss`で確認してください。Dockerはホストファイアウォールの前提を考慮しない場合거나 순서를逆にすることができる場合があります。

### Step 3: SSHのロックダウン

TailscaleがあってもSSHは適切に保護する必要があります：

```bash
# この作業中は現在のSSHセッションを開いたままにします。
# まずクライアントマシンから、Tailscale経由でSSHできることを確認します：
ssh your-user@SERVER_TAILSCALE_IP

# リバースプロキシではなくドロップインファイルにハードニングを書きます。
sudo tee /etc/ssh/sshd_config.d/99-openclaw-hardening.conf >/dev/null <<'EOF'
PasswordAuthentication no
PermitRootLogin no
KbdInteractiveAuthentication no
EOF

# リロード前に検証します。これはスキップしないでください。
sudo sshd -t
sudo systemctl reload ssh || sudo systemctl reload sshd
```

これによりパスワードベースログインとrootログインが無効になります。次のステップでは、パブリックSSHを完全にブロックしながら`tailscale0`経由のSSHは許可するためにUFWを使用します。

### Step 4: ファイアウォールルール

第2のレイヤーとしてファイアウォールを設定します：

```bash
# UFWを使用（Ubuntu/Debian）
sudo ufw allow in on tailscale0
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw enable
sudo ufw delete allow 22/tcp || true
sudo ufw reload
sudo ufw status verbose
```

Tailscale自らのUbuntuハードニングガイドは同じ形を使用しています：`tailscale0`を許可し、他の受信トラフィックを拒否し、パブリックSSHがタイムアウトする一方`100.x.y.z`アドレスへのSSHが動作することを確認します。同じVPSでパブリックWebサイトを運行している場合、`80/tcp`と`443/tcp`など、本当に必要なパブリックルールのみを維持します。

## 露出の確認

### 外部からオープンポートを確認

Tailscaleネットワーク上に**ない**マシンから：

```bash
# 一般的なパブリックポートが露出しているか確認
nmap -p 22,80,443,18789 YOUR_PUBLIC_IP

# セキュリティされたインスタンスの期待される出力：
# 22/tcp   filtered ssh
# 18789/tcp filtered unknown
```

`22`または`18789`が`open`而不是`filtered`または`closed`の場合、問題があります。`80`または`443`がオープン場合偶発的にOpenClawゲートウェイではなく意図したパブリックWeb сайードまたはTailscale Funnelエンドポイントのみであることを確認してください。

### ローカルでリッスンしているもの確認

OpenClawサーバー上：

```bash
# すべてのリッスンポートとバインド先を表示
sudo ss -tulpn | grep LISTEN

# 次のような行を探します（Serveでは良好）：
# tcp   LISTEN 0   128   127.0.0.1:18789   *:*
#
# またはこれ（認証付き直接tailnetバインドでは許容可能）：
# tcp   LISTEN 0   128   100.x.y.z:18789   *:*
#
# これのようではありません（不良）：
# tcp   LISTEN 0   128   0.0.0.0:18789     *:*
```

`0.0.0.0`または`::`（IPv6同等）を見ると、そのサービスは世界に公開されています。

### 組み込みセキュリティ監査

OpenClawにはセキュリティベストプラクティスに対して設定を проверкаする[セキュリティ監査コマンド](https://docs.molt.bot/gateway/security)があります：

```bash
openclaw security audit --deep
openclaw security audit --deep --fix
```

監査はゲートウェイ露出、Tailscaleモード、認証設定、チャンネルアクセス、ツールポリシー、プラグインインベントリ、ファイルパーミissionsをチェックします。`--fix`を有用なヘルパーとして扱い、Findings読むことの代わりにはしません。

## これでは解決しないもの

Tailscaleは最大のミスをなくします：パブリックオペレーター露出。しかしすべてを解決するわけではありません：

**認証情報保存**: OpenClawはセッショントランスクリプト、OAuthトークン、APIキーをディスクに保存します。これらが適切なファイルパーミッション（ファイルには`chmod 600`、プライベート設定ディレクトリには`chmod 700`）を持ち、バージョン管理に含まれていないことを確認します。組み込み監査はこれをチェックします。

**プラグインサンドボックス化**: プラグインはユーザーの全権限で実行されます。信頼できるソースからのプラグインのみをインストールし、彼らが要求するCapabilitiesを確認します。監査ツールはインストールされたプラグインをインベントリします。

**デバイスセキュリティ**: Tailscaleアカウントが侵害された場合、またはtailnet上のデバイスが盗まれた場合、OpenClawインスタンスにアクセスできます。新しいデバイスの承認要件を有効にするには[Tailscaleデバイス承認](https://tailscale.com/kb/1099/device-authorization/)を有効にしてください。

## デプロイメントチェックリスト

OpenClaw/Moltbotインスタンスを本番対応と見なす前に：

- [ ] Tailscaleがサーバーとクライアントの両方にインストールされ認証済み
- [ ] ゲートウェイがTailscale Serveでループバックに保たれているか、明示的な認証で`tailnet`にバインド
- [ ] SSHがパスワード認証とrootログインを無効にするよう構成
- [ ] ファイアウォール（UFWまたはiptables/nftables）が`tailscale0`を許可し、不要なパブリック着信を拒否するよう構成
- [ ] 外部nmapスキャンで全ポートが`filtered`または`closed`であることを確認
- [ ] 内部`ss -tulpn`でゲートウェイが`127.0.0.1`、`::1`、またはTailscale IPのみにバインドされていることを確認
- [ ] 認証情報ファイルが600パーミッション、プライベート設定ディレクトリが700パーミッションであることを確認
- [ ] `openclaw security audit --deep`を実行し、すべての發現に対処
- [ ] OpenClaw Tailscale管理を使用する場合、exec承認が有効であることを確認
- [ ] 定期的なバックアップが構成済み（OpenClawデータ+設定）
