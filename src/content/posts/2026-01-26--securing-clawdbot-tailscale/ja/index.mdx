---
title: "AIアシスタント経由でシェルアクセスを奪われないために"
subTitle: "ローカルやVPS上のOpenClaw/Moltbot環境をセキュアにする方法"
---

OpenClaw（旧称 Clawdbot/Moltbot）は、WhatsApp、Slack、Discord、iMessageなどのチャンネルを越えて動作するパーソナルAIアシスタントを提供します。しかし、そのゲートウェイ、ノード制御、またはSSHを強力な認証なしに公開インターネットに晒してしまうと、第三者にあなたのマシンへのシェルアクセスを許してしまう道を作ることになります。

このガイドでは、最も安全なデフォルト設定を紹介します。OpenClawのゲートウェイをループバック（127.0.0.1）に留め、Tailscale Serveを使用して自分のtailnetにのみ公開し、SSHをロックダウンした上で、外部からゲートウェイが公開されていないことを確認します。

このプロジェクトが急速に普及したことで、現実的なセキュリティ上の懸念が明らかになりました。[Shodanの調査では、最初の数週間で2,847件の露出したインスタンスが発見され](https://socradar.io/blog/clawdbot-is-it-safe/)、[GitHubのセキュリティ監査の問題では、コードベースで512件の指摘事項が報告されました](https://github.com/moltbot/moltbot/issues/1796)。その中には自動スキャナーによる出力や、2026年1月のOpenClawへの改称以降に変更されたものも含まれますが、これらの数字は正確な脆弱性数というよりも、警告サインとして捉えるべきです。セキュリティの専門家である必要はありません。運用環境をデプロイする前に、操作画面を不用意に公開しないようにすればよいのです。

---

## 実際に何を公開しているのか

インストール方法や公開方法によりますが、確認すべき3つのインターフェースがあります。

- **ポート 22**: VPS上のSSHアクセス
- **ポート 18789**: ゲートウェイ管理UIおよびWebSocket API
- **ブラウザ/ノード制御**: ゲートウェイとノードのペアリングモデルを通じた、リモートノードの実行およびブラウザ自動化

現在の[OpenClawのリモートアクセスに関するドキュメント](https://docs.molt.bot/gateway/remote)では、ゲートウェイのWebSocketはデフォルトでループバックにバインドされるようになっており、意図的にLAN、tailnet、またはカスタムバインドを選択しない限り、ループバック専用に保つことが推奨されています。これは良い傾向です。リスクが生じるのは、このデフォルトを上書きしたり、Dockerのポートを公開したり、リバースプロキシを追加したり、Funnelを有効にしたり、あるいはSSHを世界中に開放したままにしたりする場合です。

特にゲートウェイは重要です。これはアシスタントの操作画面であり、ツールの実行パスが含まれています。もしインターネットから到達可能で、認証が欠落していたり、弱かったり、回避されたり、漏洩したりした場合、攻撃者がエージェントを操作したり、あなたのユーザー権限でツールを実行したりできる可能性があります。

ブラウザ制御も同様に極めて機密性が高いものです。現在のOpenClawのドキュメントでは、ブラウザを操作するマシンのペアリング済みノードを通じてブラウザ制御を実行し、ノードのペアリングを操作権限と同等に扱うことを推奨しています。ゲートウェイがペアリング済みノードで `system.run` を実行できる場合、それはそのノード上でのリモートコード実行を意味し、ゲートウェイのノードポリシーとノード自身の実行承認設定に依存することになります。

SSHについても同様です。パスワード認証が有効な状態で公開VPSを運用していれば、総当たり攻撃（ブルートフォース）を受けるのは避けられません。

## Tailscaleによる解決策

OpenClawにおいて、Tailscaleは操作サービスを公開することなくリモートアクセスを可能にします。

1. OpenClawのインスタンスをVPSまたはローカルマシンで実行する
2. ゲートウェイはループバックにバインドしたまま、Tailscale Serveを通じてアクセスするか、明示的な認証を設けてtailnetのIPに直接バインドする
3. サーバーと個人のデバイスの両方にTailscaleをインストールする
4. OpenClawのTailscale IPまたはMagicDNS名を使用してアクセスする
5. 意図的にFunnelや他の公開プロキシを有効にしない限り、インターネット上の他の誰からも何も見えない

### OpenClawにTailscaleを管理させるべきか？

OpenClawには、ゲートウェイに対して `tailscale serve` や `tailscale funnel` を設定できる[組み込みのTailscale統合](https://docs.molt.bot/gateway/tailscale)があります。

**Serveモード**は、アクセスを自分のtailnet内のみに限定します。ゲートウェイは `127.0.0.1` にバインドされたままで、TailscaleがルーティングとHTTPSを処理します。`gateway.auth.allowTailscale` が有効な場合、OpenClawはTailscaleのアイデンティティヘッダーを使用して管理UIやWebSocketのトラフィックを認証し、`tailscale whois` で送信元を検証できます。これは、ほとんどの個人デプロイメントにおいて最適なモードです。

**Funnelモード**は、Tailscaleの公開エンドポイント機能を使用してゲートウェイを一般公開します。Tailscaleのドキュメントによれば、Funnelは広いインターネットからのトラフィックをローカルサービスにルーティングするためのものです。OpenClawは、ゲートウェイの認証モードが `password` でない限りFunnelの起動を拒否しますが、それでも操作画面を公に晒すという選択であることに変わりはありません。

OpenClawの[セキュリティドキュメント](https://docs.molt.bot/gateway/security)では、プロンプトインジェクションとツールへのアクセスがパーソナルアシスタントの核心的なリスクであると明記されています。エージェントが自らを勝手に公開するような経路を与えないでください。Serveを慎重に使用し、本当に公開アクセスが必要でない限りFunnelは避け、あらゆる `tailscale` コマンドには実行承認を必須にしてください。

## OpenClawを安全にセットアップする

### ステップ 1: Tailscaleのインストール

VPSまたはローカルサーバー上で実行します。

```bash
# Tailscaleのインストール
curl -fsSL https://tailscale.com/install.sh | sh

# 認証（ブラウザが開くのでログインする）
sudo tailscale up

# TailscaleのIPを確認
tailscale ip -4
# 出力例: 100.x.x.x
```

クライアント側のマシンでも、公式サイトからTailscaleをダウンロードしてインストールし、同じtailnetにサインインします。

これで、両方のマシンが同じプライベートネットワーク上に存在することになります。VPSのTailscale IPに対してpingを打つことができ、トラフィックは暗号化されたトンネルを経由してルーティングされます。

### ステップ 2: Tailscaleを使用するようにOpenClawを設定する

現在最も安全なパターンは、ゲートウェイをループバックに維持し、Tailscale Serveを使用してtailnetに公開することです。

OpenClawの設定ファイルにて：

```js
{
  gateway: {
    bind: "loopback",
    tailscale: { mode: "serve" },
  },
}
```

そして、Serveを有効にしてゲートウェイを起動します。

```bash
openclaw gateway --tailscale serve
```

OpenClawのドキュメントによれば、これによりゲートウェイは `127.0.0.1` で待機しつつ、TailscaleがHTTPSとtailnet内のルーティングを提供します。アクセスには公開VPSのIPではなく、`https://<magicdns-name>/` を使用します。

ServeではなくtailnetのIPに直接バインドしたい場合は、明示的なゲートウェイ認証を使用してください。

```js
{
  gateway: {
    bind: "tailnet",
    auth: {
      mode: "token",
      token: "ここに長くランダムなトークンを入力",
    },
  },
}
```

その後、別のtailnetデバイスから接続します。

```text
http://<tailscale-ip>:18789/
ws://<tailscale-ip>:18789
```

Dockerや他のコンテナランタイムを使用している場合は、ポートの公開設定に細心の注意を払ってください。`-p 18789:18789` のような設定は、通常ホストのすべてのインターフェースでバインドされます。ループバックとTailscale Serveを組み合わせるか、コンテナが正しくトラフィックを受け取れることを確認した上で、ホスト側を明示的にTailscale IPにバインドしてください。

```bash
TAILSCALE_IP=$(tailscale ip -4)
docker run ... -p "$TAILSCALE_IP:18789:18789" ...
```

Dockerの設定を変更した後は、常に外部からは `nmap`、ローカルからは `ss` を使用して確認してください。Dockerは、適切に対処しないとホストのファイアウォール設定をバイパスしたり、順序を変えたりすることがあります。

### ステップ 3: SSHのロックダウン

Tailscaleを使用していても、SSHを適切に保護すべきです。

```bash
# 作業中は、現在のSSHセッションを維持したままにしてください。
# まず、クライアントマシンからTailscale経由でSSHができることを確認します。
ssh your-user@SERVER_TAILSCALE_IP

# sshd_configを直接書き換えるのではなく、設定用ディレクトリにファイルを作成します。
sudo tee /etc/ssh/sshd_config.d/99-openclaw-hardening.conf >/dev/null <<'EOF'
PasswordAuthentication no
PermitRootLogin no
KbdInteractiveAuthentication no
EOF

# リロード前に設定を検証します。このステップは省略しないでください。
sudo sshd -t
sudo systemctl reload ssh || sudo systemctl reload sshd
```

これにより、パスワード認証とルートログインが無効になります。次のステップでは、UFWを使用して `tailscale0` 経由のSSHは許可しつつ、公開SSHを完全に遮断します。

### ステップ 4: ファイアウォールルール

第2の防御層としてファイアウォールをセットアップします。

```bash
# UFW (Ubuntu/Debian) を使用する場合
sudo ufw allow in on tailscale0
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw enable
sudo ufw delete allow 22/tcp || true
sudo ufw reload
sudo ufw status verbose
```

Tailscaleの公式Ubuntu硬化ガイドも、同様の構成を推奨しています。`tailscale0` を許可し、他のインバウンドトラフィックを拒否します。その後、公開IPへのSSHはタイムアウトし、`100.x.y.z` アドレスへのSSHは引き続き機能することを確認してください。同じVPSで公開ウェブサイトを運営している場合は、`80/tcp` や `443/tcp` など、本当に必要な公開ルールのみを保持してください。

## 露出状況の確認

### 外部からのオープンポート確認

Tailscaleネットワークに参加して**いない**マシンから実行します。

```bash
# 公開ポートが露出していないか確認
nmap -p 22,80,443,18789 YOUR_PUBLIC_IP

# セキュアなインスタンスでの期待される出力:
# 22/tcp   filtered ssh
# 18789/tcp filtered unknown
```

`22` や `18789` が `filtered` や `closed` ではなく `open` と表示される場合は、問題があります。`80` や `443` が開いている場合は、それが意図した公開ウェブサイトやTailscale Funnelエンドポイントであり、誤ってOpenClawゲートウェイが公開されていないことを確認してください。

### ローカルでのリスニング状況確認

OpenClawサーバー上で実行します。

```bash
# すべてのリスニングポートとバインド先を表示
sudo ss -tulpn | grep LISTEN

# Serveを使用している場合の正しい表示例:
# tcp   LISTEN 0   128   127.0.0.1:18789   *:*
#
# 認証付きでtailnetに直接バインドしている場合の許容される表示例:
# tcp   LISTEN 0   128   100.x.y.z:18789   *:*
#
# 危険な表示例（悪い例）:
# tcp   LISTEN 0   128   0.0.0.0:18789     *:*
```

`0.0.0.0` や `:::`（IPv6の同等物）が表示されている場合、そのサービスは世界中に公開されています。

### 組み込みのセキュリティ監査

OpenClawには、セキュリティのベストプラクティスに照らして設定をチェックする[セキュリティ監査コマンド](https://docs.molt.bot/gateway/security)が含まれています。

```bash
openclaw security audit --deep
openclaw security audit --deep --fix
```

監査では、ゲートウェイの露出、Tailscaleモード、認証設定、チャンネルアクセス、ツールポリシー、プラグインの状況、ファイル権限などがチェックされます。`--fix` は便利な補助機能として扱い、指摘された内容を自身で理解することの代わりにはしないでください。

## これで解決しないこと

Tailscaleは「操作画面の不注意な公開」という最大のミスを排除しますが、すべてを解決するわけではありません。

**資格情報の保存**: OpenClawはセッション履歴、OAuthトークン、APIキーをディスクに保存します。これらが適切なファイル権限（ファイルは `chmod 600`、非公開設定ディレクトリは `chmod 700`）を持ち、バージョン管理に含まれていないことを確認してください。組み込みの監査ツールでこれらを確認できます。

**プラグインのサンドボックス**: プラグインはあなたのユーザー権限で実行されます。信頼できるソースのプラグインのみをインストールし、要求される機能を精査してください。監査ツールでインストール済みのプラグインを一覧確認できます。

**デバイスのセキュリティ**: 誰かがあなたのTailscaleアカウントを乗っ取ったり、tailnet内のデバイスを盗んだりした場合、あなたのOpenClawインスタンスにアクセスできてしまいます。[Tailscaleのデバイス承認機能](https://tailscale.com/kb/1099/device-authorization/)を有効にして、新しいデバイスには承認を必須にするようにしてください。

## デプロイメント・チェックリスト

OpenClaw/Moltbotインスタンスを運用環境として整える前の確認事項です。

- [ ] サーバーとクライアントの両方でTailscaleがインストールされ、認証されている
- [ ] ゲートウェイがループバックでTailscale Serveを使用しているか、明示的な認証を設けて `tailnet` にバインドされている
- [ ] SSHでパスワード認証とルートログインが無効化されている
- [ ] ファイアウォール（UFW、iptables/nftables等）が `tailscale0` を許可し、不要な公開インバウンドを拒否するように設定されている
- [ ] 外部からの `nmap` スキャンで、すべてのポートが `filtered` または `closed` になっている
- [ ] 内部の `ss -tulpn` で、ゲートウェイが `127.0.0.1`、`::1`、またはTailscale IPのみにバインドされている
- [ ] 資格情報ファイルが600、非公開設定ディレクトリが700の権限に設定されている
- [ ] `openclaw security audit --deep` を実行し、すべての指摘事項に対処した
- [ ] OpenClawのTailscale管理機能を使用している場合、実行承認が有効になっている
- [ ] 定期的なバックアップが設定されている（OpenClawデータ + 設定ファイル）
