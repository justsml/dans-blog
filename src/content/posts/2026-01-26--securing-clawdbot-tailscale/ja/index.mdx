subTitle: "ローカルまたはVPSのOpenClaw/Moltbotセットアップを安全に保つ方法"
---

OpenClaw（旧称 Clawdbot/Moltbot）は、WhatsApp、Slack、Discord、iMessage、その他のチャンネルで動作するパーソナルAIアシスタントを提供します。しかし、そのゲートウェイ、ノードコントロール、またはSSHを強力な認証なしにパブリックインターネットに晒すと、見知らぬ者にあなたのマシンへのシェルアクセスの道を開くことになります。

このガイドでは、最も安全なデフォルト設定を示します。OpenClawのゲートウェイをループバックに保ち、Tailscale Serveを使ってtailnetのみに公開し、SSHをロックダウンし、外部からゲートウェイが公開されていないことを確認します。

このプロジェクトの急速な普及により、深刻なセキュリティ懸念が浮き彫りになりました。最初の数週間で[Shodanスキャンにより2,847件の公開インスタンス](https://socradar.io/blog/clawdbot-is-it-safe/)が検出され、コードベースでは[GitHubのセキュリティ監査イシューで512件の発見](https://github.com/moltbot/moltbot/issues/1796)が報告されました。その一部は自動化されたスキャナーの出力であり、2026年1月のOpenClawへの改名以降に変化している部分もあるため、これらの数字は正確な現在の脆弱性数ではなく警告サインとして捉えてください。あなたはセキュリティの専門家である必要はありません。デプロイ前にオペレーターサーフェスを公開しないようにするだけです。

## 実際に何を公開しているか

インストール方法と公開方法によって、確認すべき3つのサーフェイスがあります：

- **ポート22**: VPS上のSSHアクセス
- **ポート18789**: ゲートウェイコントロールUIとWebSocket API
- **ブラウザ/ノードコントロール**: ゲートウェイ/ノードペアリングモデルを介したリモートノード実行とブラウザ自動化

現在の[OpenClawリモートアクセスドキュメント](https://docs.molt.bot/gateway/remote)では、ゲートウェイのWebSocketはデフォルトでループバックにバインドされ、意図的にLAN/tailnet/カスタムバインドを選択しない限り、ループバックのみに保つことが推奨されています。これは良いことです。リスクは、そのデフォルトを上書きしたり、Dockerポートを公開したり、リバースプロキシを追加したり、Funnelを有効にしたり、SSHを全世界に公開したりしたときに現れます。

最大の懸念はゲートウェイです。これはあなたのアシスタントのオペレーターサーフェスであり、ツール呼び出しパスも含まれます。インターネットから到達可能で、認証が欠けていたり脆弱だったり、バイパスされたり漏洩したりしている場合、攻撃者はエージェントを操作したり、あなたのユーザーの権限でツールを呼び出したりできる可能性があります。

ブラウザコントロールもほぼ同じくらい敏感です。現在のOpenClawドキュメントでは、ブラウザマシン上のペアリングされたノードホストを介してブラウザコントロールを実行し、ノードペアリングをオペレーターアクセスと同様に扱うことが推奨されています。ゲートウェイがペアリングされたノード上で`system.run`を呼び出せる場合、それはゲートウェイのノードポリシーとノード自身の実行承認に従った、そのノード上でのリモートコード実行となります。

SSHはSSHです。パスワード認証を有効にして公開VPSを運用している場合、ブルートフォース攻撃は避けられません。

## Tailscaleによる解決策

OpenClawにとって、Tailscaleはオペレーターサービスを公開することなくリモートアクセスを提供します：

1. OpenClawインスタンスはVPSまたはローカルマシンで実行される
2. ゲートウェイはループバックにバインドされたまま、Tailscale Serveを介して到達するか、明示的な認証付きでtailnet IPに直接バインドする
3. サーバーと個人デバイスの両方にTailscaleをインストールする
4. Tailscale IPまたはMagicDNS名を介してOpenClawにアクセスする
5. インターネット上の他のすべての人は何も見えません。意図的にFunnelまたは別のパブリックプロキシを有効にしない限り

### OpenClawにTailscaleの管理を任せるべきか

OpenClawには、ゲートウェイのために`tailscale serve`または`tailscale funnel`を設定できる[組み込みのTailscale統合機能](https://docs.molt.bot/gateway/tailscale)があります。

**Serveモード**では、すべてをtailnetのみに保ちます。ゲートウェイは`127.0.0.1`にバインドされたままで、TailscaleがルーティングとHTTPSを処理します。`gateway.auth.allowTailscale`が有効になっている場合、OpenClawはTailscaleのアイデンティティヘッダーを使用してコントロールUI/WebSocketトラフィックを認証し、`tailscale whois`でソースを検証できます。これはほとんどの個人デプロイにとって適切なモードです。

**Funnelモード**は、Tailscaleのパブリックエンドポイント機能を介してゲートウェイを公開します。Tailscale自身のドキュメントでは、Funnelはより広範なインターネットからローカルサービスへのトラフィックをルーティングするものとして説明されています。OpenClawは、ゲートウェイの認証モードが`password`でない限りFunnelの起動を拒否しますが、それでもオペレーターサーフェスを公開することを選択していることになります。

OpenClawの[セキュリティドキュメント](https://docs.molt.bot/gateway/security)では、プロンプトインジェクションとツールアクセスがパーソナルアシスタントの中核的なリスクであることが明確に述べられています。エージェントに自分自身を静かに公開する道を与えないでください。意図的にServeを使用し、本当にパブリックアクセスが必要でない限りFunnelを避け、すべての`tailscale`コマンドに実行承認を要求してください。

## OpenClawを安全にセットアップする

### ステップ1: Tailscaleのインストール

VPSまたはローカルサーバーで：

```bash
# Install Tailscale
curl -fsSL https://tailscale.com/install.sh | sh

# Authenticate (opens a browser to log in)
sudo tailscale up

# Get your Tailscale IP
tailscale ip -4
# Output: 100.x.x.x
```

クライアントマシンでは、公式ダウンロードページからTailscaleをインストールし、同じtailnetにサインインしてください。

これで両方のマシンが同じプライベートネットワーク上にあります。Tailscale IPを使用してVPSにpingを実行でき、暗号化されたトンネルを介してルーティングされます。

### ステップ2: Tailscaleを使用するようOpenClawを設定する

現在最も安全なパターンは、ゲートウェイをループバックに保ち、Tailscale Serveを使ってtailnetに公開することです。

OpenClawの設定で：

```js
{
  gateway: {
    bind: "loopback",
    tailscale: { mode: "serve" },
  },
}
```

その後、Serveでゲートウェイを起動します：

```bash
openclaw gateway --tailscale serve
```

OpenClawのドキュメントでは、これによりゲートウェイは`127.0.0.1`に保たれ、TailscaleがHTTPSとtailnetルーティングを提供すると述べられています。これにはパブリックVPS IPではなく、`https://<magicdns-name>/`でアクセスします。

Serveの代わりに直接のtailnetバインドを好む場合は、明示的なゲートウェイ認証を使用します：

```js
{
  gateway: {
    bind: "tailnet",
    auth: {
      mode: "token",
      token: "replace-with-a-long-random-token",
    },
  },
}
```

その後、別のtailnetデバイスから接続します：

```text
http://<tailscale-ip>:18789/
ws://<tailscale-ip>:18789
```

Dockerまたは別のコンテナランタイムで実行している場合は、ポート公開に特に注意してください。`-p 18789:18789`のような公開は、通常すべてのホストインターフェイスにバインドされます。ループバックに加えてTailscale Serveを優先するか、コンテナがまだトラフィックを受信することを確認した後、ホスト側を明示的にTailscale IPにバインドしてください：

```bash
TAILSCALE_IP=$(tailscale ip -4)
docker run ... -p "$TAILSCALE_IP:18789:18789" ...
```

Dockerの変更後は、`nmap`で外部から、`ss`でローカルから確認してください。考慮しないと、Dockerはホストファイアウォールの想定をバイパスまたは再配置できることがあります。

### ステップ3: SSHのロックダウン

Tailscaleを使用していても、SSHを適切に保護する必要があります：

```bash
# Keep your current SSH session open while doing this.
# First, from your client machine, confirm you can SSH over Tailscale:
ssh your-user@SERVER_TAILSCALE_IP

# Put hardening in a drop-in file instead of rewriting sshd_config.
sudo tee /etc/ssh/sshd_config.d/99-openclaw-hardening.conf >/dev/null <<'EOF'
PasswordAuthentication no
PermitRootLogin no
KbdInteractiveAuthentication no
EOF

# Validate before reloading. Do not skip this.
sudo sshd -t
sudo systemctl reload ssh || sudo systemctl reload sshd
```

これにより、パスワードベースのログインとrootログインが無効になります。次のステップでは、UFWを使用してパブリックSSHを完全に防ぎながら、`tailscale0`経由のSSHは許可します。

### ステップ4: ファイアウォールルール

第2の層としてファイアウォールを設定します：

```bash
# Using UFW (Ubuntu/Debian)
sudo ufw allow in on tailscale0
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw enable
sudo ufw delete allow 22/tcp || true
sudo ufw reload
sudo ufw status verbose
```

Tailscale自身のUbuntuハードニングガイドも、この同じ形を使用しています。`tailscale0`を許可し、その他の受信トラフィックを拒否し、その後、パブリックSSHがタイムアウトし、`100.x.y.z`アドレスへのSSHがまだ動作することを確認します。同じVPSでパブリックウェブサイトを運用している場合は、`80/tcp`や`443/tcp`など、本当に必要なパブリックルールのみを保持してください。

## 公開状況の確認

### 外部からの開放ポートの確認

Tailscaleネットワークに接続されていないマシンから：

```bash
# Check if common public ports are exposed
nmap -p 22,80,443,18789 YOUR_PUBLIC_IP

# Expected output for a secured instance:
# 22/tcp   filtered ssh
# 18789/tcp filtered unknown
```

`22`または`18789`が`filtered`または`closed`ではなく`open`と表示された場合、問題があります。`80`または`443`が開いている場合は、それが意図的なパブリックウェブサイトまたはTailscale Funnelエンドポイントのみであり、誤ってOpenClawゲートウェイではないことを確認してください。

### ローカルでリッスンしているものの確認

OpenClawサーバー上で：

```bash
# Show all listening ports and what they're bound to
sudo ss -tulpn | grep LISTEN

# Look for lines like this (good for Serve):
# tcp   LISTEN 0   128   127.0.0.1:18789   *:*
#
# Or this (acceptable for direct tailnet bind with auth):
# tcp   LISTEN 0   128   100.x.y.z:18789   *:*
#
# NOT like this (bad):
# tcp   LISTEN 0   128   0.0.0.0:18789     *:*
```

`0.0.0.0`または`:::`（IPv6相当）が表示された場合、そのサービスは全世界に公開されています。

### 組み込みのセキュリティ監査

OpenClawには、セキュリティのベストプラクティスに対して設定をチェックする[セキュリティ監査コマンド](https://docs.molt.bot/gateway/security)が含まれています：

```bash
openclaw security audit --deep
openclaw security audit --deep --fix
```

監査は、ゲートウェイの公開状況、Tailscaleモード、認証設定、チャンネルアクセス、ツールポリシー、プラグインインベントリ、ファイルパーミッションをチェックします。`--fix`は便利な補助ツールとして扱い、発見内容を読むことの代替ではありません。

## これが解決しないこと

Tailscaleは最大の過ち、つまりパブリックなオペレーター公開を排除します。しかし、すべてを解決するわけではありません：

**認証情報の保存**: OpenClawは、セッショントランスクリプト、OAuthトークン、APIキーをディスクに保存します。これらに適切なファイルパーミッション（ファイルは`chmod 600`、プライベート設定ディレクトリは`chmod 700`）を設定し、バージョン管理に含まれていないことを確認してください。組み込みの監査はこれをチェックします。

**プラグインのサンドボックス化**: プラグインはあなたのユーザーの完全な権限で実行されます。信頼できるソースからのみプラグインをインストールし、要求する機能を確認してください。監査ツールはインストールされたプラグインのインベントリを作成します。

**デバイスのセキュリティ**: 誰かがあなたのTailscaleアカウントを侵害したり、tailnet上のデバイスを盗んだりした場合、その人はあなたのOpenClawインスタンスにアクセスできます。新しいデバイスに承認を要求するために、[Tailscaleデバイス認証](https://tailscale.com/kb/1099/device-authorization/)を有効にしてください。

## デプロイメントチェックリスト

OpenClaw/Moltbotインスタンスを本番環境対応とみなす前に：

- [ ] サーバーとクライアントの両方にTailscaleがインストールされ、認証されている
- [ ] ゲートウェイがTailscale Serveでループバックに保たれているか、明示的な認証で`tailnet`にバインドされている
- [ ] SSHがパスワード認証とrootログインを無効にするよう設定されている
- [ ] ファイアウォール（UFWまたはiptables/nftables）が`tailscale0`を許可し、不要なパブリック受信を拒否するよう設定されている
- [ ] 外部nmapスキャンですべてのポートが`filtered`または`closed`と表示される
- [ ] 内部の`ss -tulpn`でゲートウェイが`127.0.0.1`、`::1`、またはTailscale IPのみにバインドされている
- [ ] 認証情報ファイルに600のパーミッションが、プライベート設定ディレクトリに700のパーミッションが設定されている
- [ ] `openclaw security audit --deep`を実行し、すべての発見事項に対処している
- [ ] OpenClawのTailscale管理を使用している場合、実行承認が有効になっている
- [ ] 定期バックアップが設定されている（OpenClawデータ＋設定）
