OpenClaw (पहले Clawdbot/Moltbot) आपको एक पर्सनल AI असिस्टेंट देता है जो WhatsApp, Slack, Discord, iMessage और अन्य चैनलों पर काम करता है। लेकिन अगर आप इसका गेटवे, नोड कंट्रोल या SSH को बिना मजबूत ऑथेंटिकेशन के पब्लिक इंटरनेट पर छोड़ दें, तो आप strangers को अपनी मशीन पर shell access का रास्ता दे रहे हैं।

यह गाइड सबसे सुरक्षित डिफ़ॉल्ट दिखाती है: OpenClaw का गेटवे loopback पर रखें, इसे सिर्फ अपने tailnet में Tailscale Serve के साथ एक्सपोज करें, SSH को लॉक डाउन करें, और बाहर से वेरिफाई करें कि गेटवे पब्लिक नहीं है।

प्रोजेक्ट की तेज़ एडॉप्शन ने असली सुरक्षा चिंताएं दिखाई: पहले कुछ हफ्तों में [Shodan स्कैन में 2,847 एक्सपोज़्ड इंस्टेंस मिले](https://socradar.io/blog/clawdbot-is-it-safe/), और एक [GitHub सुरक्षा ऑडिट इश्यू में 512 फाइंडिंग्स रिपोर्ट हुईं](https://github.com/moltbot/moltbot/issues/1796)। कुछ ऑटोमेटेड स्कैनर आउटपुट थे और कुछ जनवरी 2026 के OpenClaw में रीनेम के बाद बदल गए हैं, इसलिए नंबर को वॉर्निंग साइन मानें न कि सही करंट वल्नरेबिलिटी काउंट। आपको सुरक्षा विशेषज्ञ नहीं होना है—आपको बस डिप्लॉय करने से पहले operator surfaces को पब्लिश करने से बचना है।

## आप वास्तव में क्या एक्सपोज कर रहे हैं

इंस्टॉल और एक्सपोज करने के तरीके पर निर्भर करते हुए, तीन सर्फेस चेक करने लायक हैं:

- **पोर्ट 22**: VPS पर SSH एक्सेस
- **पोर्ट 18789**: गेटवे कंट्रोल UI और WebSocket API
- **ब्राउज़र/नोड कंट्रोल**: गेटवे/नोड पेयरिंग मॉडल के जरिए रिमोट नोड एक्जीक्यूशन और ब्राउज़र ऑटोमेशन

वर्तमान [OpenClaw रिमोट-एक्सेस डॉक्स](https://docs.molt.bot/gateway/remote) कहते हैं कि गेटवे WebSocket डिफ़ॉल्ट रूप से loopback से बाइंड होता है और सिफारिश करते हैं कि इसे loopback-only रखें जब तक आप जानबूझकर LAN/tailnet/custom bind न चुनें। यह अच्छा है। रिस्क तब दिखता है जब आप वह डिफ़ॉल्ट ओवरराइड करते हैं, Docker पोर्ट पब्लिश करते हैं, रिवर्स प्रॉक्सी जोड़ते हैं, Funnel चालू करते हैं, या SSH को दुनिया के लिए खुला छोड़ देते हैं।

गेटवे बड़ा वाला है। यह आपके असिस्टेंट का operator surface है, जिसमें टूल इनवोकेशन पाथ शामिल हैं। अगर यह इंटरनेट से पहुंच योग्य है और auth गायब है, कमजोर है, बाइपास हुआ है, या लीक हुआ है, तो एक अटैकर एजेंट को ड्राइव करने या आपके यूज़र की परमिशन के साथ टूल्स इनवोक करने में सक्षम हो सकता है।

ब्राउज़र कंट्रोल लगभग उतना ही सेंसिटिव है। वर्तमान OpenClaw डॉक्स ब्राउज़र कंट्रोल को पेयर्ड नोड होस्ट पर रन करने की सिफारिश करते हैं और नोड पेयरिंग को operator एक्सेस की तरह ट्रीट करने को कहते हैं। अगर गेटवे पेयर्ड नोड पर `system.run` इनवोक कर सकता है, तो वह उस नोड पर रिमोट कोड एक्जीक्यूशन है, जो गेटवे के नोड पॉलिसी और नोड के अपने exec अप्रूवल्स के अधीन है।

SSH है SSH। अगर आप पासवर्ड ऑथेंटिकेशन चालू करके रन कर रहे हैं, तो पब्लिक VPS पर ब्रूट फोर्स अटेंप्ट्स अनिवार्य हैं।

## Tailscale समाधान

OpenClaw के लिए, Tailscale आपको operator services को पब्लिश किए बिना रिमोट एक्सेस देता है:

1. आपका OpenClaw इंस्टेंस VPS या लोकल मशीन पर रन होता है
2. गेटवे loopback से बाइंड रहता है और Tailscale Serve के जरिए tailnet तक पहुंचा जाता है, या यह स्पष्ट auth के साथ सीधे tailnet IP से बाइंड होता है
3. आप सर्वर और अपनी पर्सनल डिवाइस दोनों पर Tailscale इंस्टॉल करते हैं
4. आप इसके Tailscale IP या MagicDNS नाम से OpenClaw एक्सेस करते हैं
5. इंटरनेट पर बाकी सब कुछ कुछ नहीं देखता, जब तक आप जानबूझकर Funnel या दूसरा पब्लिक प्रॉक्सी चालू नहीं करते

### क्या आपको OpenClaw को Tailscale मैनेज करने देना चाहिए?

OpenClaw में [बिल्ट-इन Tailscale इंटीग्रेशन](https://docs.molt.bot/gateway/tailscale) है जो गेटवे के लिए `tailscale serve` या `tailscale funnel` कॉन्फ़िगर कर सकता है।

**Serve मोड** चीजों को सिर्फ आपके tailnet पर रखता है। गेटवे `127.0.0.1` से बाइंड रहता है जबकि Tailscale राउटिंग और HTTPS संभालता है। जब `gateway.auth.allowTailscale` चालू होता है, OpenClaw Tailscale identity headers का उपयोग करके Control UI/WebSocket ट्रैफिक ऑथेंटिकेट कर सकता है और `tailscale whois` से सोर्स वेरिफाई कर सकता है। यह अधिकांश पर्सनल डिप्लॉयमेंट के लिए सही मोड है।

**Funnel मोड** Tailscale के पब्लिक एंडपॉइंट फीचर के जरिए गेटवे को पब्लिकली एक्सपोज करता है। Tailscale के अपने डॉक्स Funnel को ब्रॉडर इंटरनेट से लोकल सर्विस तक ट्रैफिक राउट करने के रूप में वर्णित करते हैं। OpenClaw Funnel को तब तक शुरू करने से मना करता है जब तक gateway auth मोड `password` न हो, लेकिन आप फिर भी एक operator surface के लिए पब्लिक एक्सपोजर चुन रहे हैं।

OpenClaw के [सुरक्षा डॉक्यूमेंटेशन](https://docs.molt.bot/gateway/security) स्पष्ट हैं कि प्रॉम्प्ट इनजेक्शन और टूल एक्सेस एक पर्सनल असिस्टेंट के लिए कोर रिस्क हैं। एजेंट को खुद को पब्लिक बनाने का रास्ता न दें। Serve को जानबूझकर उपयोग करें, Funnel से बचें जब तक आपको सच में पब्लिक एक्सेस की जरूरत न हो, और किसी भी `tailscale` कमांड के लिए exec अप्रूवल जरूरी करें।

## OpenClaw को सुरक्षित रूप से सेटअप करना

### स्टेप 1: Tailscale इंस्टॉल करें

अपने VPS या लोकल सर्वर पर:

```bash
# Tailscale इंस्टॉल करें
curl -fsSL https://tailscale.com/install.sh | sh

# ऑथेंटिकेट करें (लॉगिन करने के लिए ब्राउज़र खोलता है)
sudo tailscale up

# अपना Tailscale IP पाएं
tailscale ip -4
# आउटपुट: 100.x.x.x
```

अपनी क्लाइंट मशीन पर, आधिकारिक डाउनलोड पेज से Tailscale इंस्टॉल करें और उसी tailnet में साइन इन करें।

अब दोनों मशीनें एक ही प्राइवेट नेटवर्क पर हैं। आप अपने VPS को उसके Tailscale IP से पिंग कर सकते हैं, और यह एन्क्रिप्टेड टनल के जरिए राउट होगा।

### स्टेप 2: OpenClaw को Tailscale उपयोग करने के लिए कॉन्फ़िगर करें

सबसे सुरक्षित वर्तमान पैटर्न है: गेटवे को loopback पर रखें और Tailscale Serve के साथ इसे अपने tailnet में एक्सपोज करें।

OpenClaw कॉन्फ़िग में:

```js
{
  gateway: {
    bind: "loopback",
    tailscale: { mode: "serve" },
  },
}
```

फिर Serve के साथ गेटवे शुरू करें:

```bash
openclaw gateway --tailscale serve
```

OpenClaw के डॉक्स कहते हैं यह गेटवे को `127.0.0.1` पर रखता है जबकि Tailscale HTTPS और tailnet राउटिंग देता है। आप इसे `https://<magicdns-name>/` पर खोलते हैं, अपने पब्लिक VPS IP पर नहीं।

अगर आप Serve के बजाय सीधे tailnet bind पसंद करते हैं, तो स्पष्ट gateway auth का उपयोग करें:

```js
{
  gateway: {
    bind: "tailnet",
    auth: {
      mode: "token",
      token: "replace-with-a-long-random-token",
    },
  },
}
```

फिर दूसरे tailnet डिवाइस से कनेक्ट करें:

```text
http://<tailscale-ip>:18789/
ws://<tailscale-ip>:18789
```

अगर आप Docker या किसी दूसरे कंटेनर रनटाइम में रन कर रहे हैं, तो पोर्ट पब्लिशिंग को लेकर अतिरिक्त सावधानी बरतें। `-p 18789:18789` जैसा पब्लिश आमतौर पर सभी होस्ट इंटरफेस पर बाइंड होता है। Loopback प्लस Tailscale Serve को प्राथमिकता दें, या कंफर्म करने के बाद कि कंटेनर को अभी भी ट्रैफिक मिलता है, होस्ट साइड को स्पष्ट रूप से Tailscale IP से बाइंड करें:

```bash
TAILSCALE_IP=$(tailscale ip -4)
docker run ... -p "$TAILSCALE_IP:18789:18789" ...
```

किसी भी Docker बदलाव के बाद, बाहर से `nmap` से और लोकली `ss` से चेक करें। Docker आपके होस्ट फायरवॉल अनुमानों को बाइपास या रीऑर्डर कर सकता है अगर आप इसके लिए खाते नहीं हैं।

### स्टेप 3: SSH को लॉक डाउन करें

Tailscale के साथ भी, आपको SSH को ठीक से सुरक्षित करनी चाहिए:

```bash
# इस करते समय अपना वर्तमान SSH सत्र खुला रखें।
# पहले, अपनी क्लाइंट मशीन से, पुष्टि करें कि आप Tailscale पर SSH कर सकते हैं:
ssh your-user@SERVER_TAILSCALE_IP

# sshd_config को फिर से लिखने के बजाय ड्रॉप-इन फाइल में हार्डनिंग रखें।
sudo tee /etc/ssh/sshd_config.d/99-openclaw-hardening.conf >/dev/null <<'EOF'
PasswordAuthentication no
PermitRootLogin no
KbdInteractiveAuthentication no
EOF

# रीलोड करने से पहले वैलिडेट करें। यह स्किप न करें।
sudo sshd -t
sudo systemctl reload ssh || sudo systemctl reload sshd
```

यह पासवर्ड-बेस्ड लॉगिन और रूट लॉगिन को डिसेबल करता है। अगला स्टेप UFW का उपयोग करके पब्लिक SSH को पूरी तरह ब्लॉक करता है जबकि `tailscale0` पर SSH अभी भी काम करता है।

### स्टेप 4: फायरवॉल रूल्स

दूसरी लेयर के रूप में फायरवॉल सेटअप करें:

```bash
# UFW का उपयोग करें (Ubuntu/Debian)
sudo ufw allow in on tailscale0
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw enable
sudo ufw delete allow 22/tcp || true
sudo ufw reload
sudo ufw status verbose
```

Tailscale के अपने Ubuntu हार्डनिंग गाइड इसी shape का उपयोग करते हैं: `tailscale0` को अनुमति दें, दूसरे इनबाउंड ट्रैफिक को अस्वीकार करें, फिर वेरिफाई करें कि पब्लिक SSH टाइमआउट होता है जबकि `100.x.y.z` पते पर SSH अभी भी काम करता है। अगर आप उसी VPS पर एक पब्लिक वेबसाइट चला रहे हैं, तो सिर्फ वही पब्लिक रूल्स रखें जो आपको सच में चाहिए, जैसे `80/tcp` और `443/tcp`।

## अपना एक्सपोजर चेक करना

### बाहर से ओपन पोर्ट चेक करें

किसी मशीन से जो आपके Tailscale नेटवर्क पर नहीं है:

```bash
# चेक करें कि कॉमन पब्लिक पोर्ट एक्सपोज़ हैं या नहीं
nmap -p 22,80,443,18789 YOUR_PUBLIC_IP

# सुरक्षित इंस्टेंस के लिए अपेक्षित आउटपुट:
# 22/tcp   filtered ssh
# 18789/tcp filtered unknown
```

अगर `22` या `18789` `open` दिखाता है `filtered` या `closed` के बजाय, तो आपकी समस्या है। अगर `80` या `443` खुला है, तो सुनिश्चित करें कि वह सिर्फ आपका जानबूझकर पब्लिक वेबसाइट या Tailscale Funnel एंडपॉइंट है, गलती से OpenClaw गेटवे नहीं।

### लोकली क्या लिसन कर रहा है चेक करें

अपने OpenClaw सर्वर पर:

```bash
# सभी लिसनिंग पोर्ट और वे क्या बाइंड हैं दिखाएं
sudo ss -tulpn | grep LISTEN

# ऐसी लाइनें देखें जैसे यह (Serve के लिए अच्छा):
# tcp   LISTEN 0   128   127.0.0.1:18789   *:*
#
# या यह (auth के साथ सीधे tailnet bind के लिए स्वीकार्य):
# tcp   LISTEN 0   128   100.x.y.z:18789   *:*
#
# ऐसा नहीं (खराब):
# tcp   LISTEN 0   128   0.0.0.0:18789     *:*
```

अगर आप `0.0.0.0` या `:::` (IPv6 समकक्ष) देखते हैं, तो वह सर्विस दुनिया के लिए एक्सपोज़ है।

### बिल्ट-इन सुरक्षा ऑडिट

OpenClaw में एक [सुरक्षा ऑडिट कमांड](https://docs.molt.bot/gateway/security) है जो सुरक्षा best practices के खिलाफ आपके कॉन्फ़िगरेशन की जांच करता है:

```bash
openclaw security audit --deep
openclaw security audit --deep --fix
```

ऑडिट गेटवे एक्सपोजर, Tailscale मोड, auth सेटिंग्स, चैनल एक्सेस, टूल पॉलिसी, प्लगइन इन्वेंटरी और फाइल पर्मिशन्स चेक करता है। `--fix` को उपयोगी helper मानें, फाइंडिंग्स पढ़ने का विकल्प नहीं।

## यह क्या हल नहीं करता

Tailscale सबसे बड़ी गलती को हटा देता है: पब्लिक operator एक्सपोजर। यह सब कुछ हल नहीं करता:

**क्रेडेंशियल स्टोरेज**: OpenClaw डिस्क पर सत्र ट्रांसक्रिप्ट, OAuth टोकन और API keys स्टोर करता है। सुनिश्चित करें कि इनकी फाइल पर्मिशन्स सही हैं (`chmod 600` फाइलों के लिए, `chmod 700` प्राइवेट कॉन्फ़िग डायरेक्टरीज़ के लिए) और ये वर्जन कंट्रोल में नहीं हैं। बिल्ट-इन ऑडिट इसकी जांच करता है।

**प्लगइन सैंडबॉक्सिंग**: प्लगइन आपके यूज़र की पूरी परमिशन के साथ रन होते हैं। सिर्फ उन सोर्सेज से प्लगइन इंस्टॉल करें जिन पर आप भरोसा करते हैं, और समीक्षा करें कि वे क्या क्षमताएं मांगते हैं। ऑडिट टूल इंस्टॉल किए गए प्लगइन की इन्वेंटरी बनाता है।

**डिवाइस सुरक्षा**: अगर कोई आपके Tailscale अकाउंट को समझौता करता है या आपके tailnet पर एक डिवाइस चोरी करता है, तो वे आपके OpenClaw इंस्टेंस तक पहुंच सकते हैं। [Tailscale डिवाइस ऑथोराइज़ेशन](https://tailscale.com/kb/1099/device-authorization/) चालू करें ताकि नए डिवाइस के लिए अप्रूवल जरूरी हो।

## डिप्लॉयमेंट चेकलिस्ट

जब तक आप अपने OpenClaw/Moltbot इंस्टेंस को प्रोडक्शन-रेडी मानें:

- [ ] Tailscale इंस्टॉल और सर्वर और क्लाइंट दोनों पर ऑथेंटिकेट किया गया
- [ ] गेटवे Tailscale Serve के साथ loopback पर रखा, या explicit auth के साथ `tailnet` से बाइंड
- [ ] SSH कॉन्फ़िगर किया गया ताकि पासवर्ड auth और रूट लॉगिन डिसेबल हों
- [ ] फायरवॉल (UFW या iptables/nftables) कॉन्फ़िगर किया गया ताकि `tailscale0` को अनुमति और अनावश्यक पब्लिक इनग्रेस को अस्वीकार करे
- [ ] बाहरी nmap स्कैन दिखाता है कि सभी पोर्ट `filtered` या `closed` हैं
- [ ] आंतरिक `ss -tulpn` दिखाता है कि गेटवे सिर्फ `127.0.0.1`, `::1`, या Tailscale IP से बाइंड है
- [ ] क्रेडेंशियल फाइलों की 600 पर्मिशन और प्राइवेट कॉन्फ़िग डायरेक्टरीज़ की 700 पर्मिशन है
- [ ] `openclaw security audit --deep` चलाएं और सभी फाइंडिंग्स संबोधित करें
- [ ] अगर OpenClaw Tailscale मैनेजमेंट का उपयोग कर रहे हैं, तो exec अप्रूवल चालू हैं
- [ ] नियमित बैकअप कॉन्फ़िगर किए गए (OpenClaw डेटा + कॉन्फ़िग्स)
