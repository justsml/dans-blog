OpenClaw (पहले Clawdbot/Moltbot) आपको एक व्यक्तिगत AI असिस्टेंट देता है जो WhatsApp, Slack, Discord, iMessage और अन्य चैनलों पर काम करता है। लेकिन अगर आप इसके गेटवे, नोड कंट्रोल या SSH को मजबूत प्रमाणीकरण के बिना सार्वजनिक इंटरनेट पर रखते हैं, तो आप अजनबियों को अपनी मशीन पर शेल एक्सेस का रास्ता दे रहे हैं।

यह गाइड सबसे सुरक्षित डिफॉल्ट दिखाता है: OpenClaw के गेटवे को लूपबैक पर रखें, इसे केवल अपने टेलनेट पर Tailscale Serve के साथ एक्सपोज़ करें, SSH को लॉक करें, और बाहर से सत्यापित करें कि गेटवे सार्वजनिक नहीं है।

प्रोजेक्ट की तेज़ अपनावट ने वास्तविक सुरक्षा चिंताएं उजागर कीं: [Shodan स्कैन्स में पहले कुछ हफ्तों में 2,847 एक्सपोज़्ड इंस्टेंस मिले](https://socradar.io/blog/clawdbot-is-it-safe/), और एक [GitHub सुरक्षा ऑडिट इश्यू ने कोडबेस में 512 निष्कर्ष रिपोर्ट किए](https://github.com/moltbot/moltbot/issues/1796)। इनमें से कुछ ऑटोमेटेड स्कैनर आउटपुट था और कुछ जनवरी 2026 में OpenClaw में रीनेम के बाद बदल चुका है, इसलिए इस संख्या को एक चेतावनी के रूप में लें, न कि सटीक वर्तमान भेद्यता गणना के रूप में। आपको सुरक्षा विशेषज्ञ होने की जरूरत नहीं है—आपको बस डिप्लॉय करने से पहले ऑपरेटर सतहों को सार्वजनिक करने से बचना है।

## आप वास्तव में क्या एक्सपोज़ कर रहे हैं

आपके इंस्टॉलेशन और एक्सपोज़र के आधार पर, तीन सतहें जांचने लायक हैं:

- **पोर्ट 22**: VPS पर SSH एक्सेस
- **पोर्ट 18789**: गेटवे कंट्रोल UI और WebSocket API
- **ब्राउज़र/नोड कंट्रोल**: गेटवे/नोड पेयरिंग मॉडल के माध्यम से रिमोट नोड एग्जीक्यूशन और ब्राउज़र ऑटोमेशन

वर्तमान [OpenClaw रिमोट-एक्सेस डॉक्स](https://docs.molt.bot/gateway/remote) कहते हैं कि गेटवे WebSocket डिफॉल्ट रूप से लूपबैक पर बाइंड होता है और इसे लूपबैक-ओनली रखने की सलाह देता है जब तक आप जानबूझकर LAN/टेलनेट/कस्टम बाइंड न चुनें। यह अच्छा है। जोखिम तब सामने आता है जब आप इस डिफॉल्ट को ओवरराइड करते हैं, Docker पोर्ट्स पब्लिश करते हैं, रिवर्स प्रॉक्सी जोड़ते हैं, Funnel चालू करते हैं, या SSH को दुनिया के लिए खुला छोड़ देते हैं।

गेटवे सबसे बड़ा है। यह आपके असिस्टेंट के ऑपरेटर सतह है, जिसमें टूल इनवोकेशन पाथ शामिल हैं। अगर यह इंटरनेट से पहुंच योग्य है और प्रमाणीकरण गायब, कमजोर, बाईपास्ड या लीक्ड है, तो एक हमलावर एजेंट को चला सकता है या आपके यूज़र की अनुमतियों के साथ टूल्स इनवोक कर सकता है।

ब्राउज़र कंट्रोल लगभग उतना ही संवेदनशील है। वर्तमान OpenClaw डॉक्स ब्राउज़र कंट्रोल को ब्राउज़र मशीन पर पेयर्ड नोड होस्ट के माध्यम से चलाने और नोड पेयरिंग को ऑपरेटर एक्सेस की तरह ट्रीट करने की सलाह देते हैं। अगर एक गेटवे पेयर्ड नोड पर `system.run` इनवोक कर सकता है, तो वह उस नोड पर रिमोट कोड एग्जीक्यूशन है, जो गेटवे की नोड पॉलिसी और नोड की अपनी एग्जीक्यूशन अप्रूवल्स के अधीन है।

SSH, SSH है। अगर आप पासवर्ड प्रमाणीकरण सक्षम के साथ चला रहे हैं, तो ब्रूट फोर्स प्रयास सार्वजनिक VPS पर अपरिहार्य हैं।

## Tailscale समाधान

OpenClaw के लिए, Tailscale आपको ऑपरेटर सेवाओं को पब्लिश किए बिना रिमोट एक्सेस देता है:

1. आपका OpenClaw इंस्टेंस VPS या लोकल मशीन पर चलता है
2. गेटवे लूपबैक पर बाउंड रहता है और Tailscale Serve के माध्यम से पहुंचा जाता है, या स्पष्ट प्रमाणीकरण के साथ सीधे टेलनेट IP पर बाइंड होता है
3. आप सर्वर और अपने व्यक्तिगत डिवाइस दोनों पर Tailscale इंस्टॉल करते हैं
4. आप OpenClaw को इसके Tailscale IP या MagicDNS नाम से एक्सेस करते हैं
5. इंटरनेट पर बाकी सब कुछ कुछ नहीं देखता, जब तक आप जानबूझकर Funnel या कोई अन्य सार्वजनिक प्रॉक्सी सक्षम नहीं करते

### क्या आपको OpenClaw को Tailscale मैनेज करने देना चाहिए?

OpenClaw में [बिल्ट-इन Tailscale इंटीग्रेशन](https://docs.molt.bot/gateway/tailscale) है जो गेटवे के लिए `tailscale serve` या `tailscale funnel` कॉन्फ़िगर कर सकता है।

**Serve मोड** चीजों को केवल आपके टेलनेट पर रखता है। गेटवे `127.0.0.1` पर बाउंड रहता है जबकि Tailscale रूटिंग और HTTPS हैंडल करता है। जब `gateway.auth.allowTailscale` सक्षम होता है, तो OpenClaw Tailscale पहचान हेडर का उपयोग करके कंट्रोल UI/WebSocket ट्रैफ़िक को प्रमाणित कर सकता है और स्रोत को `tailscale whois` से सत्यापित कर सकता है। यह अधिकांश व्यक्तिगत डिप्लॉयमेंट के लिए सही मोड है।

**Funnel मोड** गेटवे को Tailscale के सार्वजनिक एंडपॉइंट फीचर के माध्यम से सार्वजनिक रूप से एक्सपोज़ करता है। Tailscale के अपने डॉक्स Funnel को ब्रॉडर इंटरनेट से लोकल सेवा तक ट्रैफ़िक रूट करने के रूप में वर्णित करते हैं। OpenClaw Funnel को तब तक शुरू करने से इनकार करता है जब तक गेटवे ऑथ मोड `password` है, लेकिन आप अभी भी एक ऑपरेटर सतह के लिए सार्वजनिक एक्सपोज़र चुन रहे हैं।

OpenClaw का [सुरक्षा डॉक्यूमेंटेशन](https://docs.molt.bot/gateway/security) स्पष्ट है कि प्रॉम्प्ट इंजेक्शन और टूल एक्सेस व्यक्तिगत असिस्टेंट के लिए मूल जोखिम हैं। एजेंट को खुद को सार्वजनिक बनाने का क्वाइट पाथ न दें। Serve को जानबूझकर उपयोग करें, Funnel से बचें जब तक आपको सच में सार्वजनिक एक्सेस न चाहिए, और किसी भी `tailscale` कमांड के लिए एग्जीक्यूशन अप्रूवल की आवश्यकता करें।

## OpenClaw को सुरक्षित रूप से सेटअप करना

### चरण 1: Tailscale इंस्टॉल करें

अपने VPS या लोकल सर्वर पर:

```bash
# Tailscale इंस्टॉल करें
curl -fsSL https://tailscale.com/install.sh | sh

# प्रमाणीकरण (लॉगिन के लिए ब्राउज़र खोलता है)
sudo tailscale up

# अपना Tailscale IP प्राप्त करें
tailscale ip -4
# आउटपुट: 100.x.x.x
```

अपने क्लाइंट मशीन पर, आधिकारिक डाउनलोड पेज से Tailscale इंस्टॉल करें और उसी टेलनेट में साइन इन करें।

अब दोनों मशीनें एक ही निजी नेटवर्क पर हैं। आप अपने VPS को इसके Tailscale IP का उपयोग करके पिंग कर सकते हैं, और यह एन्क्रिप्टेड टनल के माध्यम से रूट होगा।

### चरण 2: OpenClaw को Tailscale उपयोग के लिए कॉन्फ़िगर करें

वर्तमान सबसे सुरक्षित पैटर्न है: गेटवे को लूपबैक पर रखें और इसे Tailscale Serve के साथ अपने टेलनेट पर एक्सपोज़ करें।

OpenClaw कॉन्फ़िग में:

```js
{
  gateway: {
    bind: "loopback",
    tailscale: { mode: "serve" },
  },
}
```

फिर Serve के साथ गेटवे शुरू करें:

```bash
openclaw gateway --tailscale serve
```

OpenClaw के डॉक्स कहते हैं कि यह गेटवे को `127.0.0.1` पर रखता है जबकि Tailscale HTTPS और टेलनेट रूटिंग प्रदान करता है। आप इसे `https://<magicdns-name>/` पर खोलते हैं, अपने सार्वजनिक VPS IP पर नहीं।

अगर आप Serve के बजाय सीधे टेलनेट बाइंड पसंद करते हैं, तो स्पष्ट गेटवे ऑथ का उपयोग करें:

```js
{
  gateway: {
    bind: "tailnet",
    auth: {
      mode: "token",
      token: "replace-with-a-long-random-token",
    },
  },
}
```

फिर दूसरे टेलनेट डिवाइस से कनेक्ट करें:

```text
http://<tailscale-ip>:18789/
ws://<tailscale-ip>:18789
```

अगर आप Docker या अन्य कंटेनर रनटाइम में चला रहे हैं, तो पोर्ट पब्लिशिंग के साथ विशेष सावधानी बरतें। `-p 18789:18789` जैसा पब्लिश आमतौर पर सभी होस्ट इंटरफेस पर बाइंड होता है। लूपबैक प्लस Tailscale Serve को प्राथमिकता दें, या कंटेनर को अभी भी ट्रैफ़िक प्राप्त करने की पुष्टि के बाद होस्ट साइड को स्पष्ट रूप से Tailscale IP पर बाइंड करें:

```bash
TAILSCALE_IP=$(tailscale ip -4)
docker run ... -p "$TAILSCALE_IP:18789:18789" ...
```

किसी भी Docker बदलाव के बाद, बाहर से `nmap` से और लोकली `ss` से जांचें। Docker आपकी होस्ट फ़ायरवॉल धारणाओं को बाईपास या रीऑर्डर कर सकता है अगर आप इसका हिसाब नहीं रखते।

### चरण 3: SSH को लॉकडाउन करें

Tailscale के साथ भी, आपको SSH को ठीक से सुरक्षित करना चाहिए:

```bash
# इसे करते समय अपना वर्तमान SSH सेशन खुला रखें।
# पहले, अपनी क्लाइंट मशीन से, पुष्टि करें कि आप Tailscale पर SSH कर सकते हैं:
ssh your-user@SERVER_TAILSCALE_IP

# हार्डनिंग को ड्रॉप-इन फ़ाइल में डालें, sshd_config को फिर से लिखने के बजाय।
sudo tee /etc/ssh/sshd_config.d/99-openclaw-hardening.conf >/dev/null <<'EOF'
PasswordAuthentication no
PerRootLogin no
KbdInteractiveAuthentication no
EOF

# रीलोड करने से पहले सत्यापित करें। इसे स्किप न करें।
sudo sshd -t
sudo systemctl reload ssh || sudo systemctl reload sshd
```

यह पासवर्ड-आधारित लॉगिन और रूट लॉगिन को अक्षम करता है। अगला चरण UFW का उपयोग करके सार्वजनिक SSH को पूरी तरह से रोकना है जबकि `tailscale0` पर SSH को अनुमत करना जारी रखना है।

### चरण 4: फ़ायरवॉल नियम

फ़ायरवॉल को दूसरी परत के रूप में सेटअप करें:

```bash
# UFW का उपयोग करके (Ubuntu/Debian)
sudo ufw allow in on tailscale0
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw enable
sudo ufw delete allow 22/tcp || true
sudo ufw reload
sudo ufw status verbose
```

Tailscale का अपना Ubuntu हार्डनिंग गाइड इसी शेप का उपयोग करता है: `tailscale0` को अनुमत करें, अन्य इनबाउंड ट्रैफ़िक को अस्वीकार करें, फिर सत्यापित करें कि सार्वजनिक SSH टाइम आउट होता है जबकि `100.x.y.z` पते पर SSH अभी भी काम करता है। अगर आप उसी VPS पर सार्वजनिक वेबसाइट चलाते हैं, तो केवल उन सार्वजनिक नियमों को रखें जो आपको सच में चाहिए, जैसे `80/tcp` और `443/tcp`।

## अपनी एक्सपोज़र जांचना

### बाहर से ओपन पोर्ट जांचें

एक ऐसी मशीन से जो आपके Tailscale नेटवर्क पर नहीं है:

```bash
# जांचें कि सामान्य सार्वजनिक पोर्ट एक्सपोज़्ड हैं या नहीं
nmap -p 22,80,443,18789 YOUR_PUBLIC_IP

# एक सुरक्षित इंस्टेंस के लिए अपेक्षित आउटपुट:
# 22/tcp   filtered ssh
# 18789/tcp filtered unknown
```

अगर `22` या `18789` `filtered` या `closed` के बजाय `open` दिखाता है, तो आपके पास समस्या है। अगर `80` या `443` ओपन है, तो सुनिश्चित करें कि वह केवल आपका जानबूझकर रखा गया सार्वजनिक वेबसाइट या Tailscale Funnel एंडपॉइंट है, गलती से OpenClaw गेटवे नहीं।

### लोकली क्या लिसन कर रहा है जांचें

अपने OpenClaw सर्वर पर:

```bash
# सभी लिसनिंग पोर्ट और वे किससे बाइंड हैं दिखाएं
sudo ss -tulpn | grep LISTEN

# ऐसी लाइनें देखें (Serve के लिए अच्छा):
# tcp   LISTEN 0   128   127.0.0.1:18789   *:*
#
# या ऐसा (ऑथ के साथ सीधे टेलनेट बाइंड के लिए स्वीकार्य):
# tcp   LISTEN 0   128   100.x.y.z:18789   *:*
#
# ऐसा नहीं (बुरा):
# tcp   LISTEN 0   128   0.0.0.0:18789     *:*
```

अगर आप `0.0.0.0` या `:::` (IPv6 समतुल्य) देखते हैं, तो वह सेवा दुनिया के लिए एक्सपोज़्ड है।

### बिल्ट-इन सुरक्षा ऑडिट

OpenClaw में एक [सुरक्षा ऑडिट कमांड](https://docs.molt.bot/gateway/security) शामिल है जो आपके कॉन्फ़िगरेशन को सुरक्षा सर्वोत्तम प्रथाओं के विरुद्ध जांचता है:

```bash
openclaw security audit --deep
openclaw security audit --deep --fix
```

ऑडिट गेटवे एक्सपोज़र, Tailscale मोड, ऑथ सेटिंग्स, चैनल एक्सेस, टूल पॉलिसी, प्लगइन इन्वेंट्री और फ़ाइल अनुमतियों की जांच करता है। `--fix` को एक उपयोगी सहायक मानें, निष्कर्षों को पढ़ने का विकल्प नहीं।

---

## यह क्या हल नहीं करता

Tailscale सबसे बड़ी गलती को हटाता है: सार्वजनिक ऑपरेटर एक्सपोज़र। यह सब कुछ हल नहीं करता:

**क्रेडेंशियल स्टोरेज**: OpenClaw सेशन ट्रांसक्रिप्ट्स, OAuth टोकन और API कीज़ को डिस्क पर स्टोर करता है। सुनिश्चित करें कि इनमें उचित फ़ाइल अनुमतियां हैं (फ़ाइलों के लिए `chmod 600`, प्राइवेट कॉन्फ़िग डायरेक्ट्रीज़ के लिए `chmod 700`) और वे वर्ज़न कंट्रोल में नहीं हैं। बिल्ट-इन ऑडिट इसकी जांच करता है।

**प्लगइन सैंडबॉक्सिंग**: प्लगइन्स आपके यूज़र की पूरी अनुमतियों के साथ चलते हैं। केवल उन स्रोतों से प्लगइन्स इंस्टॉल करें जिन पर आप भरोसा करते हैं, और समीक्षा करें कि वे कौन सी क्षमताएं अनुरोध करते हैं। ऑडिट टूल इंस्टॉल्ड प्लगइन्स की सूची बनाता है।

**डिवाइस सुरक्षा**: अगर कोई आपका Tailscale खाता कंप्रमाइज़ करता है या आपके टेलनेट पर कोई डिवाइस चुराता है, तो वे आपके OpenClaw इंस्टेंस को एक्सेस कर सकते हैं। नए डिवाइस के लिए अप्रूवल की आवश्यकता के लिए [Tailscale डिवाइस ऑथराइज़ेशन](https://tailscale.com/kb/1099/device-authorization/) सक्षम करें।

## डिप्लॉयमेंट चेकलिस्ट

इससे पहले कि आप अपने OpenClaw/Moltbot इंस्टेंस को प्रोडक्शन-रेडी मानें:

- [ ] सर्वर और क्लाइंट दोनों पर Tailscale इंस्टॉल और प्रमाणित
- [ ] गेटवे Tailscale Serve के साथ लूपबैक पर रखा गया, या स्पष्ट ऑथ के साथ `tailnet` पर बाउंड
- [ ] SSH को पासवर्ड ऑथ और रूट लॉगिन अक्षम करने के लिए कॉन्फ़िगर किया गया
- [ ] फ़ायरवॉल (UFW या iptables/nftables) `tailscale0` को अनुमत और अनावश्यक सार्वजनिक इनग्रेस को अस्वीकार करने के लिए कॉन्फ़िगर
- [ ] बाहरी nmap स्कैन सभी पोर्ट्स को `filtered` या `closed` दिखाता है
- [ ] आंतरिक `ss -tulpn` गेटवे को केवल `127.0.0.1`, `::1`, या Tailscale IP पर बाउंड दिखाता है
- [ ] क्रेडेंशियल फ़ाइलों में 600 अनुमतियां और प्राइवेट कॉन्फ़िग डायरेक्ट्रीज़ में 700 अनुमतियां
- [ ] `openclaw security audit --deep` चलाएं और सभी निष्कर्षों का समाधान करें
- [ ] अगर OpenClaw Tailscale मैनेजमेंट उपयोग कर रहे हैं, तो एग्जीक्यूशन अप्रूवल्स सक्षम हैं
- [ ] नियमित बैकअप कॉन्फ़िगर (OpenClaw डेटा + कॉन्फ़िग)
