OpenClaw (ранее Clawdbot/Moltbot) даёт персонального ИИ-ассистента, который работает через WhatsApp, Slack, Discord, iMessage и другие каналы. Но если вы выставите его шлюз, управления ноды или SSH в публичный интернет без строгой аутентификации — вы даёте незнакомцам путь к shell-доступу на вашей машине.

Этот гайд показывает самый безопасный вариант: оставить шлюз OpenClaw на loopback, открывать его только для своего tailnet через Tailscale Serve, закрыть SSH и проверить извне, что шлюз не публичен.

Быстрый рост проекта выявил реальные проблемы безопасности: [сканы Shodan нашли 2 847 открытых инстансов](https://socradar.io/blog/clawdbot-is-it-safe/) в первые недели, а [аудит безопасности на GitHub показал 512 проблем](https://github.com/moltbot/moltbot/issues/1796) в кодовой базе. Часть была выводом автоматических сканеров, часть изменилась после переименования в OpenClaw в январе 2026, так что воспринимайте цифры как предупреждающий знак, а не точную текущую уязвимость. Не нужно быть экспертом по безопасности — просто не публикуйте операторские поверхности до того, как задеплоите.

## Что вы реально открываете

В зависимости от способа установки и публикации есть три поверхности, которые стоит проверить:

- **Порт 22**: SSH-доступ на VPS
- **Порт 18789**: Gateway Control UI и WebSocket API
- **Браузер/управление нодой**: удалённое выполнение ноды и автоматизация браузера через модель связки шлюз/нода

Текущая [документация OpenClaw по удалённому доступу](https://docs.molt.bot/gateway/remote) говорит, что WebSocket шлюза по умолчанию привязывается к loopback и рекомендует держать его только на loopback, если вы целенаправленно не выбираете LAN/tailnet/кастомный бинд. Это правильно. Риск возникает, когда вы переопределяете это значение по умолчанию, публикуете Docker-порты, добавляете обратный прокси, включаете Funnel или оставляете SSH открытым для всех.

Шлюз — главное. Это операторская поверхность вашего ассистента, включая пути вызова инструментов. Если он доступен из интернета и аутентификация отсутствует, слабая, обходится или утекла — атакующий может управлять агентом или вызывать инструменты с правами вашего пользователя.

Управление браузером почти так же чувствительно. Текущая документация OpenClaw рекомендует запускать управление браузером через спаренный хост ноды на машине с браузером и относиться к спариванию нод как к операторскому доступу. Если шлюз может вызвать `system.run` на спаренной ноде — это удалённое выполнение кода на этой ноде, с учётом политики нод шлюза и собственных подтверждений exec ноды.

SSH — это SSH. Если работаете с включённой парольной аутентификацией, брутфорс-атаки на публичном VPS неизбежны.

## Решение через Tailscale

Для OpenClaw Tailscale даёт удалённый доступ без публикации операторских сервисов:

1. Ваш инстанс OpenClaw работает на VPS или локальной машине
2. Шлюз остаётся привязанным к loopback и достигается через Tailscale Serve, либо привязывается напрямую к tailnet IP с явной аутентификацией
3. Вы устанавливаете Tailscale и на сервер, и на свои личные устройства
4. Вы получаете доступ к OpenClaw через его Tailscale IP или MagicDNS-имя
5. Все остальные в интернете не видят ничего, если вы целенаправленно не включите Funnel или другой публичный прокси

### Стоит ли позволять OpenClaw управлять Tailscale?

OpenClaw имеет [встроенную интеграцию с Tailscale](https://docs.molt.bot/gateway/tailscale), которая может настроить `tailscale serve` или `tailscale funnel` для шлюза.

**Режим Serve** держит всё только на вашем tailnet. Шлюз остаётся привязанным к `127.0.0.1`, а Tailscale обеспечивает маршрутизацию и HTTPS. Когда `gateway.auth.allowTailscale` включен, OpenClaw может аутентифицировать трафик Control UI/WebSocket через заголовки идентичности Tailscale и проверять источник через `tailscale whois`. Это правильный режим для большинства персональных деплоев.

**Режим Funnel** публично открывает шлюз через функцию публичной конечной точки Tailscale. Собственная документация Tailscale описывает Funnel как маршрутизацию трафика из широкого интернета к локальному сервису. OpenClaw отказывается запускать Funnel, пока режим аутентификации шлюза не выставлен в `password`, но вы всё равно выбираете публичное раскрытие операторской поверхности.

[Документация по безопасности](https://docs.molt.bot/gateway/security) OpenClaw прямо говорит, что инъекция промптов и доступ к инструментам — ключевые риски для персонального ассистента. Не давайте агенту путь тихо сделать себя публичным. Используйте Serve осознанно, избегайте Funnel, пока вам действительно не нужен публичный доступ, и требуйте подтверждения exec для любой команды `tailscale`.

## Настройка OpenClaw безопасно

### Шаг 1: Установите Tailscale

На VPS или локальном сервере:

```bash
# Установка Tailscale
curl -fsSL https://tailscale.com/install.sh | sh

# Аутентификация (откроется браузер для входа)
sudo tailscale up

# Получите ваш Tailscale IP
tailscale ip -4
# Вывод: 100.x.x.x
```

На клиентской машине установите Tailscale с официальной страницы загрузки и войдите в тот же tailnet.

Теперь обе машины в одной приватной сети. Можно пинговать VPS через его Tailscale IP, и трафик пойдёт через зашифрованный туннель.

### Шаг 2: Настройте OpenClaw на использование Tailscale

Самый безопасный текущий паттерн: оставить шлюз на loopback и открыть его для своего tailnet через Tailscale Serve.

В конфиге OpenClaw:

```js
{
  gateway: {
    bind: "loopback",
    tailscale: { mode: "serve" },
  },
}
```

Затем запустите шлюз с Serve:

```bash
openclaw gateway --tailscale serve
```

Документация OpenClaw говорит, что это держит шлюз на `127.0.0.1`, а Tailscale обеспечивает HTTPS и маршрутизацию по tailnet. Вы открываете его по `https://<magicdns-name>/`, а не по публичному IP VPS.

Если предпочитаете прямой бинд к tailnet вместо Serve, используйте явную аутентификацию шлюза:

```js
{
  gateway: {
    bind: "tailnet",
    auth: {
      mode: "token",
      token: "замените-на-длинный-случайный-токен",
    },
  },
}
```

Затем подключайтесь с другого устройства в tailnet:

```text
http://<tailscale-ip>:18789/
ws://<tailscale-ip>:18789
```

Если запускаете в Docker или другом container runtime, будьте особенно осторожны с публикацией портов. Публикация типа `-p 18789:18789` обычно биндится на все хостовые интерфейсы. Предпочитайте loopback плюс Tailscale Serve, или привяжите хостовую сторону явно к Tailscale IP после подтверждения, что контейнер всё ещё получает трафик:

```bash
TAILSCALE_IP=$(tailscale ip -4)
docker run ... -p "$TAILSCALE_IP:18789:18789" ...
```

После любого изменения Docker проверяйте снаружи через `nmap` и локально через `ss`. Docker может обходить или переупорядочивать хостовые файрвольные допущения, если вы не учитываете это.

### Шаг 3: Закройте SSH

Даже с Tailscale нужно правильно защитить SSH:

```bash
# Держите текущую SSH-сессию открытой пока делаете это.
# Сначала, с клиентской машины, подтвердите, что можете SSH через Tailscale:
ssh your-user@SERVER_TAILSCALE_IP

# Поместите харденинг в отдельный файл вместо переписывания sshd_config.
sudo tee /etc/ssh/sshd_config.d/99-openclaw-hardening.conf >/dev/null <<'EOF'
PasswordAuthentication no
PermitRootLogin no
KbdInteractiveAuthentication no
EOF

# Валидируйте перед перезагрузкой. Не пропускайте это.
sudo sshd -t
sudo systemctl reload ssh || sudo systemctl reload sshd
```

Это отключает парольный вход и вход под root. Следующий шаг использует UFW, чтобы не допустить публичный SSH, при этом всё ещё позволяя SSH через `tailscale0`.

### Шаг 4: Правила файрвола

Настройте файрвол как второй слой:

```bash
# Используя UFW (Ubuntu/Debian)
sudo ufw allow in on tailscale0
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw enable
sudo ufw delete allow 22/tcp || true
sudo ufw reload
sudo ufw status verbose
```

Собственный гайд по харденингу Ubuntu от Tailscale использует ту же схему: разрешить `tailscale0`, запретить прочий входящий трафик, затем убедиться, что публичный SSH отваливается по таймауту, а SSH на адрес `100.x.y.z` всё ещё работает. Если запускаете публичный веб-сайт на том же VPS, оставьте только действительно нужные публичные правила, например `80/tcp` и `443/tcp`.

## Проверка вашей экспозиции

### Проверьте открытые порты извне

С машины, которая НЕ в вашей сети Tailscale:

```bash
# Проверьте, открыты ли обычные публичные порты
nmap -p 22,80,443,18789 YOUR_PUBLIC_IP

# Ожидаемый вывод для защищённого инстанса:
# 22/tcp   filtered ssh
# 18789/tcp filtered unknown
```

Если `22` или `18789` показывает `open` вместо `filtered` или `closed` — у вас проблема. Если `80` или `443` открыт, убедитесь, что это только ваш намеренный публичный веб-сайт или конечная точка Tailscale Funnel, а не шлюз OpenClaw случайно.

### Проверьте, что слушает локально

На вашем сервере OpenClaw:

```bash
# Показать все слушающие порты и к чему они привязаны
sudo ss -tulpn | grep LISTEN

# Ищите строки вроде этой (хорошо для Serve):
# tcp   LISTEN 0   128   127.0.0.1:18789   *:*
#
# Или такой (приемлемо для прямого tailnet-бинда с аутентификацией):
# tcp   LISTEN 0   128   100.x.y.z:18789   *:*
#
# НЕ такой (плохо):
# tcp   LISTEN 0   128   0.0.0.0:18789     *:*
```

Если видите `0.0.0.0` или `:::` (IPv6-эквивалент) — этот сервис открыт миру.

### Встроенный аудит безопасности

OpenClaw включает [команду аудита безопасности](https://docs.molt.bot/gateway/security), которая проверяет вашу конфигурацию против лучших практик:

```bash
openclaw security audit --deep
openclaw security audit --deep --fix
```

Аудит проверяет экспозицию шлюза, режим Tailscale, настройки аутентификации, доступ к каналам, политику инструментов, инвентарь плагинов и права на файлы. Воспринимайте `--fix` как полезного помощника, а не замену чтению результатов.

## Что это не решает

Tailscale убирает главную ошибку: публичную экспозицию оператора. Это не решает всё:

**Хранение учётных данных**: OpenClaw хранит транскрипты сессий, OAuth-токены и API-ключи на диске. Убедитесь, что на эти файлы правильные права (`chmod 600` для файлов, `chmod 700` для приватных конфигурационных директорий) и они не в версионном контроле. Встроенный аудит это проверяет.

**Изоляция плагинов**: Плагины работают с полными правами вашего пользователя. Устанавливайте плагины только из источников, которым доверяете, и проверяйте, какие возможности они запрашивают. Инструмент аудита инвентаризирует установленные плагины.

**Безопасность устройств**: Если кто-то скомпрометирует ваш аккаунт Tailscale или украдёт устройство в вашем tailnet, он получит доступ к вашему инстансу OpenClaw. Включите [авторизацию устройств Tailscale](https://tailscale.com/kb/1099/device-authorization/), чтобы требовать подтверждения для новых устройств.

## Чеклист деплоя

Перед тем как считать ваш инстанс OpenClaw/Moltbot готовым к продакшену:

- [ ] Tailscale установлен и аутентифицирован и на сервере, и на клиенте
- [ ] Шлюз держится на loopback с Tailscale Serve, или привязан к `tailnet` с явной аутентификацией
- [ ] SSH настроен на отключение парольной аутентификации и входа под root
- [ ] Файрвол (UFW или iptables/nftables) настроен на разрешение `tailscale0` и запрет ненужного публичного входящего трафика
- [ ] Внешний nmap-скан показывает все порты `filtered` или `closed`
- [ ] Внутренний `ss -tulpn` показывает шлюз, привязанный только к `127.0.0.1`, `::1` или Tailscale IP
- [ ] Файлы учётных данных имеют права 600, а приватные конфигурационные директории — 700
- [ ] Запустите `openclaw security audit --deep` и исправьте все находки
- [ ] Если используете управление Tailscale от OpenClaw, подтверждения exec включены
- [ ] Настроены регулярные бэкапы (данные и конфиги OpenClaw)
