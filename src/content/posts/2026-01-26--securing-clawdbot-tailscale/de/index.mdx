OpenClaw (früher Clawdbot/Moltbot) gibt dir einen persönlichen KI-Assistenten, der über WhatsApp, Slack, Discord, iMessage und andere Kanäle funktioniert. Aber wenn du sein Gateway, Node-Steuerung oder SSH ohne starke Authentifizierung ins öffentliche Internet stellst, gibst du Fremden einen Weg zu Shell-Zugriff auf deine Maschine.

Dieser Leitfaden zeigt die sicherste Standardkonfiguration: Halte OpenClaws Gateway auf Loopback, exponiere es nur über dein Tailnet mit Tailscale Serve, schalte SSH ab, und verifiziere von außen, dass das Gateway nicht öffentlich erreichbar ist.

Die schnelle Adoption des Projekts offenbarte echte Sicherheitsbedenken: [Shodan-Scans fanden 2.847 exponierte Instanzen](https://socradar.io/blog/clawdbot-is-it-safe/) in den ersten Wochen, und ein [GitHub Security Audit Issue meldete 512 Findings](https://github.com/moltbot/moltbot/issues/1796) im Codebase. Einiges davon war automatisierter Scanner-Output und einiges hat sich seit dem Januar 2026 Rename zu OpenClaw geändert, also behandle die Zahl als Warnsignal statt als präzisen aktuellen Vulnerability-Count. Du musst kein Sicherheitsexperte sein – du musst nur vermeiden, Operator-Flächen zu veröffentlichen, bevor du deployst.

---

## Was du tatsächlich exponierst

Je nach Installations- und Expositionsart gibt es drei Flächen, die du prüfen solltest:

- **Port 22**: SSH-Zugriff auf einem VPS
- **Port 18789**: Gateway Control UI und WebSocket API
- **Browser/Node-Steuerung**: Remote-Node-Ausführung und Browser-Automatisierung durch das Gateway/Node-Pairing-Modell

Die aktuellen [OpenClaw Remote-Access-Docs](https://docs.molt.bot/gateway/remote) sagen, dass das Gateway WebSocket standardmäßig an Loopback bindet und empfehlen, es Loopback-only zu halten, bis du bewusst LAN/Tailnet/Custom-Bind wählst. Das ist gut. Das Risiko entsteht, wenn du diesen Standard überschreibst, Docker-Ports veröffentlichst, einen Reverse Proxy hinzufügst, Funnel einschaltest oder SSH offen zur Welt lässt.

Das Gateway ist das große Thema. Es ist die Operator-Fläche für deinen Assistenten, inklusive Tool-Invocation-Pfaden. Wenn es aus dem Internet erreichbar ist und Auth fehlt, schwach ist, umgangen wurde oder geleakt ist, könnte ein Angreifer den Agenten steuern oder Tools mit den Berechtigungen deines Users aufrufen.

Browser-Steuerung ist fast ebenso sensibel. Die aktuellen OpenClaw-Docs empfehlen, Browser-Steuerung durch ein gepaartes Node-Host auf der Browser-Maschine laufen zu lassen und Node-Pairing wie Operator-Zugriff zu behandeln. Wenn ein Gateway `system.run` auf einem gepaarten Node aufrufen kann, ist das Remote Code Execution auf diesem Node, abhängig von der Gateway-Node-Policy und den eigenen Exec-Approvals des Nodes.

SSH ist SSH. Wenn du mit aktivierter Passwort-Authentifizierung arbeitest, sind Brute-Force-Versuche auf einem öffentlichen VPS unvermeidlich.

## Die Tailscale-Lösung

Für OpenClaw gibt dir Tailscale Fernzugriff ohne Veröffentlichung von Operator-Services:

1. Deine OpenClaw-Instanz läuft auf einem VPS oder lokaler Maschine
2. Das Gateway bleibt an Loopback gebunden und wird durch Tailscale Serve erreicht, oder es bindet direkt an die Tailnet-IP mit expliziter Auth
3. Du installierst Tailscale sowohl auf dem Server als auch auf deinen persönlichen Geräten
4. Du greifst auf OpenClaw über seine Tailscale-IP oder MagicDNS-Namen zu
5. Alle anderen im Internet sehen nichts, es sei denn, du aktivierst bewusst Funnel oder einen anderen öffentlichen Proxy

### Sollte OpenClaw Tailscale verwalten?

OpenClaw hat eine [eingebaute Tailscale-Integration](https://docs.molt.bot/gateway/tailscale), die `tailscale serve` oder `tailscale funnel` für das Gateway konfigurieren kann.

**Serve-Modus** hält alles nur auf deinem Tailnet. Das Gateway bleibt an `127.0.0.1` gebunden, während Tailscale Routing und HTTPS übernimmt. Wenn `gateway.auth.allowTailscale` aktiviert ist, kann OpenClaw Control-UI/WebSocket-Traffic mit Tailscale-Identity-Headern authentifizieren und die Quelle mit `tailscale whois` verifizieren. Das ist der richtige Modus für die meisten persönlichen Deployments.

**Funnel-Modus** exponiert das Gateway öffentlich über Tailscales öffentliches Endpoint-Feature. Tailscales eigene Docs beschreiben Funnel als Routing von Traffic aus dem breiteren Internet zu einem lokalen Service. OpenClaw weigert sich, Funnel zu starten, wenn der Gateway-Auth-Modus nicht `password` ist, aber du wählst trotzdem öffentliche Exposition für eine Operator-Fläche.

OpenClaws [Security-Dokumentation](https://docs.molt.bot/gateway/security) ist klar, dass Prompt Injection und Tool-Zugriff Kernrisiken für einen persönlichen Assistenten sind. Gib dem Agenten keinen Pfad, sich leise öffentlich zu machen. Nutze Serve bewusst, vermeide Funnel, es sei denn du brauchst wirklich öffentlichen Zugriff, und verlange Exec-Approval für jeden `tailscale`-Befehl.

## OpenClaw sicher einrichten

### Schritt 1: Tailscale installieren

Auf deinem VPS oder lokalen Server:

```bash
# Tailscale installieren
curl -fsSL https://tailscale.com/install.sh | sh

# Authentifizieren (öffnet einen Browser zum Einloggen)
sudo tailscale up

# Deine Tailscale-IP abrufen
tailscale ip -4
# Output: 100.x.x.x
```

Auf deiner Client-Maschine Tailscale von der offiziellen Download-Seite installieren und im gleichen Tailnet einloggen.

Jetzt sind beide Maschinen im gleichen privaten Netzwerk. Du kannst deinen VPS über seine Tailscale-IP anpingen, und es wird durch den verschlüsselten Tunnel geroutet.

### Schritt 2: OpenClaw für Tailscale konfigurieren

Das sicherste aktuelle Pattern ist: Das Gateway auf Loopback halten und über Tailscale Serve zu deinem Tailnet exponieren.

In OpenClaw Config:

```js
{
  gateway: {
    bind: "loopback",
    tailscale: { mode: "serve" },
  },
}
```

Dann das Gateway mit Serve starten:

```bash
openclaw gateway --tailscale serve
```

OpenClaws Docs sagen, dies hält das Gateway auf `127.0.0.1`, während Tailscale HTTPS und Tailnet-Routing bereitstellt. Du öffnest es unter `https://<magicdns-name>/`, nicht unter deiner öffentlichen VPS-IP.

Wenn du direktes Tailnet-Bind statt Serve bevorzugst, nutze explizite Gateway-Auth:

```js
{
  gateway: {
    bind: "tailnet",
    auth: {
      mode: "token",
      token: "replace-with-a-long-random-token",
    },
  },
}
```

Dann von einem anderen Tailnet-Gerät verbinden:

```text
http://<tailscale-ip>:18789/
ws://<tailscale-ip>:18789
```

Wenn du in Docker oder einem anderen Container-Runtime läufst, sei besonders vorsichtig mit Port-Veröffentlichung. Ein Publish wie `-p 18789:18789` bindet normalerweise auf allen Host-Interfaces. Bevorzuge Loopback plus Tailscale Serve, oder binde die Host-Seite explizit an die Tailscale-IP, nachdem du bestätigt hast, dass der Container noch Traffic empfängt:

```bash
TAILSCALE_IP=$(tailscale ip -4)
docker run ... -p "$TAILSCALE_IP:18789:18789" ...
```

Nach jeder Docker-Änderung von außen mit `nmap` und lokal mit `ss` prüfen. Docker kann Host-Firewall-Annahmen umgehen oder neu ordnen, wenn du das nicht berücksichtigst.

### Schritt 3: SSH absichern

Auch mit Tailscale solltest du SSH richtig absichern:

```bash
# Halte deine aktuelle SSH-Session offen, während du das machst.
# Zuerst, von deiner Client-Maschine, bestätige, dass du über Tailscale SSH kannst:
ssh your-user@SERVER_TAILSCALE_IP

# Die Härtung in eine Drop-in-Datei statt sshd_config umzuschreiben.
sudo tee /etc/ssh/sshd_config.d/99-openclaw-hardening.conf >/dev/null <<'EOF'
PasswordAuthentication no
PermitRootLogin no
KbdInteractiveAuthentication no
EOF

# Vor dem Neuladen validieren. Dies nicht überspringen.
sudo sshd -t
sudo systemctl reload ssh || sudo systemctl reload sshd
```

Dies deaktiviert passwortbasierte Anmeldung und Root-Login. Der nächste Schritt nutzt UFW, um öffentliches SSH komplett zu verhindern, während SSH über `tailscale0` noch funktioniert.

### Schritt 4: Firewall-Regeln

Firewall als zweite Schicht einrichten:

```bash
# UFW verwenden (Ubuntu/Debian)
sudo ufw allow in on tailscale0
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw enable
sudo ufw delete allow 22/tcp || true
sudo ufw reload
sudo ufw status verbose
```

Tailscales eigener Ubuntu-Härtungsleitfaden nutzt diese gleiche Form: `tailscale0` erlauben, anderen eingehenden Traffic verweigern, dann verifizieren, dass öffentliches SSH timeout, während SSH zur `100.x.y.z`-Adresse noch funktioniert. Wenn du eine öffentliche Website auf dem gleichen VPS betreibst, halte nur die öffentlichen Regeln, die du wirklich brauchst, wie `80/tcp` und `443/tcp`.

## Deine Exposition prüfen

### Offene Ports von außen prüfen

Von einer Maschine, die NICHT in deinem Tailscale-Netzwerk ist:

```bash
# Prüfen, ob gängige öffentliche Ports exponiert sind
nmap -p 22,80,443,18789 YOUR_PUBLIC_IP

# Erwartete Ausgabe für eine gesicherte Instanz:
# 22/tcp   filtered ssh
# 18789/tcp filtered unknown
```

Wenn `22` oder `18789` `open` statt `filtered` oder `closed` zeigt, hast du ein Problem. Wenn `80` oder `443` offen ist, stelle sicher, dass das nur dein beabsichtigter öffentlicher Website- oder Tailscale-Funnel-Endpoint ist, nicht das OpenClaw-Gateway aus Versehen.

### Lokal prüfen, was lauscht

Auf deinem OpenClaw-Server:

```bash
# Alle lauschenden Ports anzeigen und womit sie gebunden sind
sudo ss -tulpn | grep LISTEN

# Nach Zeilen wie dieser suchen (gut für Serve):
# tcp   LISTEN 0   128   127.0.0.1:18789   *:*
#
# Oder dieser (akzeptabel für direktes Tailnet-Bind mit Auth):
# tcp   LISTEN 0   128   100.x.y.z:18789   *:*
#
# NICHT so (schlecht):
# tcp   LISTEN 0   128   0.0.0.0:18789     *:*
```

Wenn du `0.0.0.0` oder `:::` (IPv6-Äquivalent) siehst, ist dieser Service zur Welt exponiert.

### Eingebautes Security Audit

OpenClaw enthält einen [Security-Audit-Befehl](https://docs.molt.bot/gateway/security), der deine Konfiguration gegen Security-Best-Practices prüft:

```bash
openclaw security audit --deep
openclaw security audit --deep --fix
```

Das Audit prüft Gateway-Exposition, Tailscale-Modus, Auth-Einstellungen, Kanal-Zugriff, Tool-Policy, Plugin-Inventar und Dateiberechtigungen. Behandle `--fix` als nützlichen Helfer, nicht als Ersatz für das Lesen der Findings.

## Was das nicht löst

Tailscale entfernt den größten Fehler: öffentliche Operator-Exposition. Es löst nicht alles:

**Credential-Speicherung**: OpenClaw speichert Session-Transkripte, OAuth-Tokens und API-Keys auf Disk. Stell sicher, dass diese korrekte Dateiberechtigungen haben (`chmod 600` für Dateien, `chmod 700` für private Config-Verzeichnisse) und nicht in der Versionskontrolle sind. Das eingebaute Audit prüft dies.

**Plugin-Sandboxing**: Plugins laufen mit den vollen Berechtigungen deines Users. Installiere nur Plugins aus Quellen, denen du vertraust, und prüfe, welche Fähigkeiten sie anfordern. Das Audit-Tool inventarisiert installierte Plugins.

**Geräte-Sicherheit**: Wenn jemand dein Tailscale-Konto kompromittiert oder ein Gerät in deinem Tailnet stiehlt, kann er auf deine OpenClaw-Instanz zugreifen. Aktiviere [Tailscale Device Authorization](https://tailscale.com/kb/1099/device-authorization/), um Genehmigung für neue Geräte zu require.

## Deployment-Checkliste

Bevor du deine OpenClaw/Moltbot-Instanz als produktionsreif betrachtest:

- [ ] Tailscale auf Server und Client installiert und authentifiziert
- [ ] Gateway auf Loopback mit Tailscale Serve gehalten, oder an `tailnet` mit expliziter Auth gebunden
- [ ] SSH konfiguriert, um Passwort-Auth und Root-Login zu deaktivieren
- [ ] Firewall (UFW oder iptables/nftables) konfiguriert, um `tailscale0` zu erlauben und unnötigen öffentlichen Ingress zu verweigern
- [ ] Externer nmap-Scan zeigt alle Ports `filtered` oder `closed`
- [ ] Internes `ss -tulpn` zeigt Gateway gebunden an `127.0.0.1`, `::1` oder nur die Tailscale-IP
- [ ] Credential-Dateien haben 600-Berechtigungen und private Config-Verzeichnisse haben 700-Berechtigungen
- [ ] `openclaw security audit --deep` ausführen und alle Findings adressieren
- [ ] Wenn OpenClaw Tailscale-Management nutzt, Exec-Approvals aktiviert
- [ ] Regelmäßige Backups konfiguriert (OpenClaw-Daten + Configs)
