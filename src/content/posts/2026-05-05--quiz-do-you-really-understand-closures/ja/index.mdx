<p class="inset">クロージャは、すべてのJavaScript開発者が「理解している」と思い込んでいる概念の1つです——理解していない瞬間までね。このクイズで、あなたが実際に知っていることの境界線を見つけ出します。</p>

クロージャはイベントハンドラ、ファクトリ関数、React Hooks、モジュールパターン、そしてほぼあらゆる場所に登場します。決して特殊なものではありません。JavaScriptの織りなす生地そのものです。

最初の問題は手頃で、そこから床板を剥がし始めます。

    ```javascript
    for (var i = 0; i < 5; i++) {

    ```

  <slot name='explanation'>
  <div className="explanation">
    `var` は関数スコープです。`i` は**1つだけ**——5つのコールバックはすべて同じバインディングをクロージャとして捕捉します。タイムアウトが発火する頃には、ループは終了しており `i` は `5` です。

    5つのコールバック。1つの `i`。値は `5`。5回ログ出力されます。

    修正方法は `let`（ブロックスコープ、反復ごとに新しいバインディング）を使うか、IIFEでラップして現在の値をキャプチャすることです：
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100); // 0, 1, 2, 3, 4 ✓
    }
    ```

    クロージャが「値を凍結する」と仮定する人がよくつまずきます。凍結しません。**リファレンス**を捕捉します。

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={5}
  group="上級"
  title="Reactの古いクロージャ"
  options={[
    {text: 'クリックするたびに現在のカウントをログ出力'},
    {text: '常に0をログ出力', isAnswer: true},
    {text: 'NaNをログ出力'},
    {text: 'コンパイルできない'},
  ]}
>
  <slot name="question">
  <div className="question">
    このReactコンポーネントでボタンを複数回クリックしたとき、どう動作しますか？
    ```jsx
    function Counter() {
      const [count, setCount] = useState(0);

      useEffect(() => {
        const id = setInterval(() => {
          console.log('count is', count);
        }, 1000);
        return () => clearInterval(id);
      }, []); // 空の依存配列

      return <button onClick={() => setCount(c => c + 1)}>
        Click ({count})
      </button>;
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    古典的な古いクロージャ（stale closure）です。`useEffect` は一度だけ実行され（空の依存配列）、マウント時の `count`、つまり `0` をクロージャとして捕捉したインターバルを作成します。何回クリックしても、インターバルのクロージャは更新された値を見ることはありません。

    ボタンのラベルは正しく更新されます——それは新しいレンダーだからです。しかし `console.log` は永遠に `0` に閉じ込められます。

    修正するには `count` を依存配列に追加する（インターバルが再作成される）か、refを使います：
    ```jsx
    const countRef = useRef(count);
    useEffect(() => { countRef.current = count; }, [count]);
    // インターバル内では countRef.current を使う
    ```
  </div>
  </slot>
</Challenge>

  <div className="question">
    待って——よく読んで。実際にどうなりますか？
    ```javascript
    const bank = (() => {
      let balance = 0;
      return {
        deposit: (n) => (balance += n),
        withdraw: (n) => (balance -= n),
        getBalance: () => balance,
      };
    })();

    bank.withdraw(40);

    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    答えは `60` です。

    これは**モジュールパターン**——クロージャのオブジェクトを返すIIFEです。3つのメソッドはすべて同じ `balance` バインディングをクロージャとして捕捉しています。本当の意味でプライベートです：外部から `balance` にアクセスすることはできません。

    ```javascript
    console.log(bank.balance); // undefined
    ```

    クラスが登場する前、これはJavaScriptで状態をカプセル化する*決定的な*方法でした。レガシーコードベースやユーティリティライブラリでは今でも見かけます。

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={7}
  group="エキスパート"
  title="クロージャのメモリリーク"
  options={[
    {text: 'data配列のlengthをログ出力する'},
    {text: 'メモリリークを引き起こす', isAnswer: true},
    {text: 'RangeErrorをスローする'},
    {text: '特に問題ない——これは正常'},
  ]}
>
  <slot name="question">
  <div className="question">
    このコードの問題は何ですか？
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      element.addEventListener('click', () => {
        console.log('clicked!', data.length);
      });
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    イベントリスナーが `data` をクロージャとして捕捉しているため、リスナーが到達可能な限り配列は到達可能なままです。`element` が長寿命で、リスナーが削除されなければ、大きな配列も保持され続けます。

    `element` はDOMにあります。リスナーが `data` を保持しています。GCは回収できません。

    修正方法：長寿命のリスナーで大きなデータを捕捉しないようにし、所有するUIが破棄されるときにリスナーを削除します：
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      // dataを使って何かする...

      const onClick = () => {
        console.log('clicked!');
      };

      element.addEventListener('click', onClick);
      return () => element.removeEventListener('click', onClick);
    }
    ```

    最新のV8はかつてよりこの点で優れています。しかし、多くのイベントリスナーを持つ長寿命のSPAでは、このパターンは依然として人を困らせます。
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={8}
  group="エキスパート"
  title="thisの罠"
  options={[
    {text: 'Dan, Dan'},
    {text: 'Dan, undefined'},
    {text: 'undefined, Dan'},
    {text: 'undefined, undefined', isAnswer: true},
  ]}
>
  <slot name="question">
  <div className="question">
    これは何を出力しますか？
    ```javascript
    const user = {
      name: 'Dan',
      greetArrow: () => console.log(this?.name),
      greetRegular() {
        const inner = () => console.log(this?.name);
        inner();
      },
    };

    user.greetArrow();
    const fn = user.greetRegular;
    fn();
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    2つの巧妙な罠があります：

    **`greetArrow`**：アロー関数は自身の `this` を持ちません。囲むレキシカルスコープから `this` を捕捉します——ここでは `user` ではなく、モジュール/グローバルスコープです。`this?.name` は `undefined` です。

    **`fn()`**：`greetRegular` はレシーバなしで呼び出されています（`user.greetRegular()` ではなく `fn()`）。strict modeでは `this` は `undefined` です。sloppy modeではグローバルオブジェクトです。どちらにしても `this?.name` は `undefined` です。

    `user.greetRegular()` を直接呼び出せば `'Dan'` になります——そのとき `this` が `user` にバインドされるからです。

    結果：`undefined, undefined`。
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={9}
  group="ボーナス"
  title="クロージャによるメモ化"
  options={[
    {text: 'expensive()を3回呼び出す'},
    {text: 'expensive()を1回呼び出す', isAnswer: true},
    {text: 'expensive()を2回呼び出す'},
    {text: 'スロー：Maximum call stack exceeded'},
  ]}
>
  <slot name="question">
  <div className="question">
    `expensive()` は実際に何回呼び出されますか？
    ```javascript
    function memoize(fn) {
      const cache = new Map();
      return (arg) => {
        if (cache.has(arg)) return cache.get(arg);
        const result = fn(arg);
        cache.set(arg, result);
        return result;
      };
    }

    let calls = 0;
    const expensive = (n) => { calls++; return n * 2; };
    const fast = memoize(expensive);

    fast(5);
    fast(5);
    fast(5);
    console.log(calls);
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    メモ化された関数は `cache` をクロージャとして捕捉しています。最初の呼び出し（`fast(5)`）はキャッシュにミスし、`expensive` を呼び出して結果を格納します。2回目と3回目の呼び出しはキャッシュにヒットし、即座に返します。

    `calls` は `1` です。

    これがクロージャの実用的な仕事です。`cache` はプライベート（`memoize` の外からアクセス不可）、永続的（`fast` が存在する限り存続）、そして `fast` のすべての呼び出しで共有されます。
  </div>
  </slot>
</Challenge>

どうでしたか？

- **9-10問正解**：クロージャを本当に理解しています。ペアプログラミング相手だとたぶん怖い。
- **6-8問正解**： solid です。古いクロージャ / `this` の問題でつまずいたのは正直なところです。
- **3-5問正解**：概念は知っているが、境界線が曖昧です。MDNのスコープチェーンドキュメントを読み直しましょう。
- **0-2問正解**：クロージャを知っているつもりでした。今知りました。それがポイントです。

Reactの古いクロージャ（#6）と `this` の罠（#8）が、実際のプロダクションバグを引き起こす問題です。これらが馴染みがないと感じたなら、見直す価値があります——コードレビューで他のクロージャ関連の問題よりも頻繁に登場するからです。
