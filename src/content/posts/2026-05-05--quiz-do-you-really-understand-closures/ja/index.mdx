<p class="inset">クロージャは、すべてのJavaScript開発者が*理解している*と思い込んでいる概念のひとつだ——実際には理解が曖昧なままであることに気づくその瞬間までは。このクイズであなたの知識の輪郭を探ってみよう。</p>

クロージャはイベントハンドラ、ファクトリ関数、Reactフック、モジュールパターン、そしてあらゆる場所に現れる。特別なものではない。JavaScriptが織りなす布地そのものだ。

    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100); // 0, 1, 2, 3, 4 ✓
    }
    ```

  client:visible={{rootMargin: "150px"}}
  index={5}
  group="アドバンスド"
  title="ReactのStale Closure"
  options={[
    {text: 'クリックするたびに現在のカウントを出力する'},
    {text: '常に0を出力する', isAnswer: true},
    {text: 'NaNを出力する'},
    {text: 'コンパイルエラーになる'},
  ]}
>
  <slot name="question">
  <div className="question">
    このReactコンポーネントは、ボタンが複数回クリックされたときどのような振る舞いを示すか？
    ```jsx
    function Counter() {
      const [count, setCount] = useState(0);

      useEffect(() => {
        const id = setInterval(() => {
          console.log('count is', count);
        }, 1000);
        return () => clearInterval(id);
      }, []); // 空の依存配列

      return <button onClick={() => setCount(c => c + 1)}>
        Click ({count})
      </button>;
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    古典的なStale Closure問題。`useEffect` は一度だけ実行され（空の依存配列）、マウント時の `count`（`0`）をクロージャとして閉じ込めたインターバルを作成する。何回クリックしても、インターバルのクロージャは更新された値を決して参照しない。

    ボタンのラベルは正しく更新される——それは新しいレンダリングだからだ。しかし `console.log` は永遠に `0` に閉じ込められたままになる。

    修正方法は `count` を依存配列に追加するか（インターバルを再作成）、refを使うことだ：
    ```jsx
    const countRef = useRef(count);
    useEffect(() => { countRef.current = count; }, [count]);
    // そしてインターバル内で countRef.current を使う
    ```
  </div>
  </slot>
</Challenge>

        withdraw: (n) => (balance -= n),

    ```javascript
    console.log(bank.balance); // undefined
    ```

    クラスが登場する前は、これがJavaScriptで状態をカプセル化する*主流の*方法だった。レガシーコードベースやユーティリティライブラリで今でも目にするだろう。

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={7}
  group="エキスパート"
  title="クロージャのメモリリーク"
  options={[
    {text: 'data配列の長さを出力する'},
    {text: 'メモリリークを引き起こす', isAnswer: true},
    {text: 'RangeErrorをスローする'},
    {text: '何も問題ない——これで正常'},
  ]}
>
  <slot name="question">
  <div className="question">
    このコードの問題は何か？
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      element.addEventListener('click', () => {
        console.log('clicked!', data.length);
      });
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    イベントリスナーが `data` をクロージャとして閉じ込めるため、リスナーが到達可能である限り配列も保持され続ける。`element` が長命でリスナーが削除されない場合、その大きな配列も保持され続ける。

    `element` はDOM内にある。リスナーが `data` を保持している。GCはそれを回収できない。

    修正方法：長命のリスナー内で大きなデータをキャプチャしないこと（本当に必要な場合を除く）、そしてUIが破棄されるときにリスナーを削除すること：
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      // dataで何か処理をする...

      const onClick = () => {
        console.log('clicked!');
      };

      element.addEventListener('click', onClick);
      return () => element.removeEventListener('click', onClick);
    }
    ```

    現代のV8は以前よりこれに強くなっている。しかし、多くのイベントリスナーを持つ長命のSPAでは、このパターンは今でも人々を悩ませる。
  </div>
  </slot>
</Challenge>

  client:visible={{rootMargin: "150px"}}
  index={8}
  group="エキスパート"
  title="thisの罠"
  options={[
    {text: 'Dan, Dan'},
    {text: 'Dan, undefined'},
    {text: 'undefined, Dan'},
    {text: 'undefined, undefined', isAnswer: true},
  ]}
>
  <slot name="question">
  <div className="question">
    このコードは何を出力する？
    ```javascript
    const user = {
      name: 'Dan',
      greetArrow: () => console.log(this?.name),
      greetRegular() {
        const inner = () => console.log(this?.name);
        inner();
      },
    };

    user.greetArrow();
    const fn = user.greetRegular;
    fn();
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    2つの微妙な罠：

    **`greetArrow`**：アロー関数は自身の `this` を持たない。囲んでいる語彙スコープ（ここではモジュール/グローバルスコープであって、`user` ではない）から `this` をキャプチャする。したがって `this?.name` は `undefined` になる。

    **`fn()`**：`greetRegular` はレシーバなしで呼び出されている（`user.greetRegular()` ではなく `fn()`）。strictモードでは `this` は `undefined`。sloppyモードではグローバルオブジェクトになる。どちらにせよ、`this?.name` は `undefined` だ。

    `user.greetRegular()` と直接呼び出せば `'Dan'` が得られた——その場合 `this` は `user` にバインドされるからだ。

    結果：`undefined, undefined`。
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={9}
  group="ボーナス"
  title="クロージャによるメモ化"
  options={[
    {text: 'expensive()が3回呼ばれる'},
    {text: 'expensive()が1回呼ばれる', isAnswer: true},
    {text: 'expensive()が2回呼ばれる'},
    {text: 'エラー：Maximum call stack exceeded'},
  ]}
>
  <slot name="question">
  <div className="question">
    `expensive()` は実際に何回呼び出されるか？
    ```javascript
    function memoize(fn) {
      const cache = new Map();
      return (arg) => {
        if (cache.has(arg)) return cache.get(arg);
        const result = fn(arg);
        cache.set(arg, result);
        return result;
      };
    }

    let calls = 0;
    const expensive = (n) => { calls++; return n * 2; };
    const fast = memoize(expensive);

    fast(5);
    fast(5);
    fast(5);
    console.log(calls);
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    メモ化された関数は `cache` をクロージャとして閉じ込める。最初の呼び出し（`fast(5)`）はキャッシュミスし、`expensive` を呼び出して結果を保存する。2回目と3回目の呼び出しはキャッシュにヒットして即座に返る。

    `calls` は `1` だ。

    これがクロージャの実際の仕事だ。`cache` はプライベートで（`memoize` の外部からアクセスできない）、永続的で（`fast` が存在する限り生き続ける）、`fast` へのすべての呼び出しで共有される。
  </div>
  </slot>
</Challenge>

どうだった？

- **9-10問正解**：あなたは本当にクロージャを理解している。おそらくペアプロすると怖い人。
- **6-8問正解**：堅実。Stale Closure／`this` の問題に引っかかったなら、それは正直なところだ。
- **3-5問正解**：概念はわかっているが、境界があいまい。MDNのスコープチェーンのドキュメントを読もう。
- **0-2問正解**：クロージャを理解しているつもりだった。今は理解した。それでいい。

ReactのStale Closure（#6）と `this` の罠（#8）は、実際のプロダクションバグを引き起こす問題だ。それらに馴染みがなかったなら、再訪する価値がある——クロージャ関連の問題としては、コードレビューでこれ以上に頻繁に登場するものはない。
