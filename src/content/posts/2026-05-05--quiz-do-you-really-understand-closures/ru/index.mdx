<p class="inset">Замыкания (closures) — это одна из тех концепций, которую каждый JavaScript-разработчик *думает*, что понимает — ровно до того момента, пока не столкнется с ними на практике. Этот квиз поможет нащупать границы того, что вы знаете на самом деле.</p>

Замыкания встречаются повсюду: в обработчиках событий, фабричных функциях, хуках React, паттерне «модуль» и практически во всех остальных частях языка. Это не какая-то экзотика. Это сама ткань, из которой соткан JavaScript.

Вопросы начинаются с довольно простых, но постепенно «пол под ногами начинает проваливаться».

<Challenge

  title="Классический счетчик"
  options={[

  <slot name="question">
  <div className="question">
    Что выведет этот код?
    ```javascript
    function makeCounter() {
      let count = 0;
      return () => ++count;
    }
    const tick = makeCounter();
    console.log(tick(), tick(), tick());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Функция `tick` замыкает переменную `count`. Каждый вызов инкрементирует и возвращает *ту же самую* переменную `count`. Три вызова подряд дадут `1, 2, 3`.

    Ключевое слово: *ту же самую*. `count` не копируется в замыкание. Замыкание хранит **живую ссылку** на привязку переменной. В этом вся суть.
  </div>
  </slot>

  options={[
    {text: '1, 1', isAnswer: true},
    {text: '1, 2'},
    {text: '2, 2'},
    {text: '0, 0'},
  ]}
>
  <slot name="question">
  <div className="question">
    Что выведет этот код?
    ```javascript
    function makeCounter() {
      let count = 0;
      return () => ++count;
    }
    const a = makeCounter();
    const b = makeCounter();
    console.log(a(), b());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Каждый вызов `makeCounter()` создает **новый контекст выполнения** со своей собственной свежей переменной `count`. Функции `a` и `b` замыкают *разные* привязки `count`. Они не делят общее состояние.

    Результат: `1, 1`

    Именно поэтому замыкания используются для создания приватного состояния — каждый «экземпляр» независим.

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={2}
  group="Классическая ловушка"
  title="var в цикле"
  options={[
    {text: '0, 1, 2, 3, 4'},
    {text: '5, 5, 5, 5, 5', isAnswer: true},
    {text: '1, 2, 3, 4, 5'},
    {text: 'undefined x5'},
  ]}
>
  <slot name="question">
  <div className="question">
    Что будет выведено через 100 мс?
    ```javascript
    for (var i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Ключевое слово `var` имеет функциональную область видимости. Здесь существует ровно **одна** переменная `i` — все пять колбэков замыкают одну и ту же привязку. К моменту срабатывания таймеров цикл уже завершился, и значение `i` равно `5`.

    Пять колбэков. Одна `i`. Значение: `5`. Выведено пять раз.

    Решение — использовать `let` (блочная область видимости, создающая новую привязку на каждой итерации) или обернуть вызов в IIFE для захвата текущего значения:
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100); // 0, 1, 2, 3, 4 ✓
    }
    ```
  </div>
  </slot>
</Challenge>

  title="let все исправляет (или нет?)"
  options={[
    {text: '0, 1, 2, 3, 4', isAnswer: true},
    {text: '5, 5, 5, 5, 5'},
    {text: '4, 4, 4, 4, 4'},
    {text: 'ReferenceError: i is not defined'},
  ]}
>
  <slot name="question">
  <div className="question">
    Тот же цикл, другое ключевое слово. Что в консоли?
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `let` в цикле `for` ведет себя по-особенному. Согласно спецификации, каждая итерация получает свою **новую привязку** `i`. Это работает так, будто цикл внутри делает следующее:

    ```javascript
    { let i = 0; setTimeout(() => console.log(i), 100) }
    { let i = 1; setTimeout(() => console.log(i), 100) }
    // ...
    ```

    Таким образом, пять колбэков замыкают пять различных переменных. Результат: `0, 1, 2, 3, 4`.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={4}
  group="Замыкания и объекты"
  title="Мутация объекта"
  options={[
    {text: '{ x: 1 }, { x: 1 }'},
    {text: '{ x: 1 }, { x: 99 }'},
    {text: '{ x: 99 }, { x: 99 }', isAnswer: true},
    {text: '{ x: 99 }, { x: 1 }'},
  ]}
>
  <slot name="question">
  <div className="question">
    Что выведет этот код?
    ```javascript
    function wrap() {
      const obj = { x: 1 };
      const getObj = () => obj;
      obj.x = 99;
      return getObj;
    }
    const get = wrap();
    console.log(get(), get());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Замыкание захватывает **привязку** `obj`, а не слепок (snapshot) объекта. Сам объект мутирует уже после создания замыкания, но `obj` по-прежнему указывает на ту же ссылку. Оба вызова возвращают один и тот же измененный объект: `{ x: 99 }`.

    Это часто сбивает с толку тех, кто думает, что замыкания «замораживают» значения. Это не так. Они захватывают **ссылки**.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={5}
  group="Продвинутый уровень"
  title="Устаревшее замыкание в React"
  options={[
    {text: 'Выводит актуальное значение count при каждом клике'},
    {text: 'Всегда выводит 0', isAnswer: true},
    {text: 'Выводит NaN'},
    {text: "Код не скомпилируется"},
  ]}
>
  <slot name="question">
  <div className="question">
    Как поведет себя этот React-компонент при многократном нажатии на кнопку?
    ```jsx
    function Counter() {
      const [count, setCount] = useState(0);

      useEffect(() => {
        const id = setInterval(() => {
          console.log('count is', count);
        }, 1000);
        return () => clearInterval(id);
      }, []); // пустой массив зависимостей

      return <button onClick={() => setCount(c => c + 1)}>
        Нажать ({count})
      </button>;
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Классическое «устаревшее замыкание» (stale closure). `useEffect` выполняется один раз (пустой массив зависимостей) и создает интервал, который замыкает `count` таким, каким он был в момент монтирования: `0`. Сколько бы раз вы ни кликали, замыкание внутри интервала никогда не увидит обновленное значение.

    Текст на кнопке обновляется правильно — это происходит при каждом новом рендере. Но `console.log` навсегда заперт на значении `0`.

    Чтобы исправить это, нужно либо добавить `count` в массив зависимостей (что будет пересоздавать интервал), либо использовать `ref`:
    ```jsx
    const countRef = useRef(count);
    useEffect(() => { countRef.current = count; }, [count]);
    // затем используйте countRef.current внутри интервала
    ```
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={6}
  group="Продвинутый уровень"
  title="Паттерн «Модуль»"
  options={[
    {text: '60', isAnswer: true},
    {text: 'TypeError: balance is not defined'},
    {text: '-40'},
    {text: '100'},
  ]}
>
  <slot name="question">
  <div className="question">
    Внимательно прочитайте код. Что произойдет на самом деле?
    ```javascript
    const bank = (() => {
      let balance = 0;
      return {
        deposit: (n) => (balance += n),
        withdraw: (n) => (balance -= n),
        getBalance: () => balance,
      };
    })();

    bank.deposit(100);
    bank.withdraw(40);
    console.log(bank.getBalance());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Ответ: `60`.

    Это паттерн **«Модуль»** — IIFE, который возвращает объект с замыканиями. Все три метода замыкают одну и ту же привязку `balance`. Она по-настоящему приватна: вы не можете получить доступ к `balance` извне.

    ```javascript
    console.log(bank.balance); // undefined
    ```

    До появления классов в JS это был *основной* способ инкапсуляции состояния. Вы до сих пор встретите этот паттерн в старом коде и во многих вспомогательных библиотеках.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={7}
  group="Эксперт"
  title="Утечка памяти в замыкании"
  options={[
    {text: 'Выводит длину массива data'},
    {text: 'Создает утечку памяти', isAnswer: true},
    {text: 'Выбрасывает RangeError'},
    {text: 'Ничего особенного — нормальный код'},
  ]}
>
  <slot name="question">
  <div className="question">
    В чем проблема с этим кодом?
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      element.addEventListener('click', () => {
        console.log('clicked!', data.length);
      });
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Обработчик события замыкает `data`, поэтому массив остается доступным до тех пор, пока доступен сам обработчик. Если `element` живет долго, а обработчик никогда не удаляется, то и этот огромный массив не будет очищен сборщиком мусора.

    `element` находится в DOM. Обработчик держит `data`. Сборщик мусора (GC) не может его удалить.

    Решение: не захватывайте большие объемы данных в долгоживущих обработчиках без необходимости, и удаляйте обработчики, когда UI-компонент уничтожается:
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      // какие-то действия с data...

      const onClick = () => {
        console.log('clicked!');
      };

      element.addEventListener('click', onClick);
      return () => element.removeEventListener('click', onClick);
    }
    ```

    Современный движок V8 справляется с этим лучше, чем раньше. Но в долгоживущих SPA с множеством обработчиков этот паттерн все еще может стать причиной проблем.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={8}
  group="Эксперт"
  title="Ловушка this"
  options={[
    {text: 'Dan, Dan'},
    {text: 'Dan, undefined'},
    {text: 'undefined, Dan'},
    {text: 'undefined, undefined', isAnswer: true},
  ]}
>
  <slot name="question">
  <div className="question">
    Что выведет этот код?
    ```javascript
    const user = {
      name: 'Dan',
      greetArrow: () => console.log(this?.name),
      greetRegular() {
        const inner = () => console.log(this?.name);
        inner();
      },
    };

    user.greetArrow();
    const fn = user.greetRegular;
    fn();
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Здесь сразу две тонкие ловушки:

    **`greetArrow`**: У стрелочных функций нет своего `this`. Они захватывают `this` из окружающего лексического окружения — в данном случае это глобальная область видимости (или модуль), а не объект `user`. Поэтому `this?.name` будет `undefined`.

    **`fn()`**: Метод `greetRegular` вызывается без контекста (просто `fn()`, а не `user.greetRegular()`). В строгом режиме (strict mode) `this` будет `undefined`. В обычном — глобальный объект. В обоих случаях `this?.name` даст `undefined`.

    Если бы вы вызвали `user.greetRegular()` напрямую, вы бы получили `'Dan'`, так как в этом случае `this` был бы привязан к `user`.

    Результат: `undefined, undefined`.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={9}
  group="Бонус"
  title="Мемоизация через замыкание"
  options={[
    {text: 'Вызывает expensive() 3 раза'},
    {text: 'Вызывает expensive() 1 раз', isAnswer: true},
    {text: 'Вызывает expensive() 2 раза'},
    {text: 'Ошибка: Maximum call stack exceeded'},
  ]}
>
  <slot name="question">
  <div className="question">
    Сколько раз на самом деле будет вызвана функция `expensive()`?
    ```javascript
    function memoize(fn) {
      const cache = new Map();
      return (arg) => {
        if (cache.has(arg)) return cache.get(arg);
        const result = fn(arg);
        cache.set(arg, result);
        return result;
      };
    }

    let calls = 0;
    const expensive = (n) => { calls++; return n * 2; };
    const fast = memoize(expensive);

    fast(5);
    fast(5);
    fast(5);
    console.log(calls);
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Мемоизированная функция замыкает `cache`. Первый вызов (`fast(5)`) не находит данных в кэше, вызывает `expensive` и сохраняет результат. Второй и третий вызовы попадают в кэш и возвращают результат немедленно.

    Переменная `calls` будет равна `1`.

    Это пример того, как замыкания выполняют реальную работу. `cache` является приватным (к нему нельзя обратиться извне `memoize`), персистентным (он живет столько же, сколько существует функция `fast`) и общим для всех вызовов `fast`.
  </div>
  </slot>
</Challenge>

Ну как успехи?

- **9-10**: Вы действительно знаете, как работают замыкания. Наверное, с вами страшно программировать в паре.
- **6-8**: Хороший уровень. Вопросы про «устаревшие замыкания» и `this` могли вас запутать, и это нормально.
- **3-5**: Вы понимаете концепцию, но на краях всё еще туманно. Перечитайте документацию MDN про цепочку областей видимости (scope chain).
- **0-2**: Вы думали, что знаете замыкания. Теперь — знаете. В этом и был смысл.

Устаревшие замыкания в React (#6) и ловушка с `this` (#8) — это то, что чаще всего приводит к реальным багам в продакшене. Если эти вопросы показались вам незнакомыми, к ним стоит вернуться: они встречаются на ревью чаще любых других проблем, связанных с замыканиями.
