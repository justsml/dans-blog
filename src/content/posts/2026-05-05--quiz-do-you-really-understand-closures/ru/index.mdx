<p class="inset">Замыкания — одна из тех концепций, которую каждый разработчик JavaScript *думает*, что понимает — ровно до момента, когда это не так. Этот тест покажет границы того, что вы на самом деле знаете.</p>

Замыкания встречаются в обработчиках событий, фабричных функциях, React-хуках, паттерне модулей и практически везде. Они не экзотические. Это ткань, из которой соткан JavaScript.

Вопросы начинаются разумными, а затем начинают убирать пол.

<Challenge

  title="Классический счётчик"
  options={[

  <slot name="question">
  <div className="question">
    Что это выведет?
    ```javascript
    function makeCounter() {
      let count = 0;
      return () => ++count;
    }
    const tick = makeCounter();
    console.log(tick(), tick(), tick());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `tick` замыкает на `count`. Каждый вызов инкрементирует и возвращает **одну и ту же** переменную `count`. Три вызова дают `1, 2, 3`.

    Ключевое слово: *одна и та же*. `count` не копируется в замыкание. Замыкание хранит **живую ссылку** на привязку переменной. Вот в этом и суть.
  </div>
  </slot>

  options={[
    {text: '1, 1', isAnswer: true},
    {text: '1, 2'},
    {text: '2, 2'},
    {text: '0, 0'},
  ]}
>
  <slot name="question">
  <div className="question">
    Что это выведет?
    ```javascript
    function makeCounter() {
      let count = 0;
      return () => ++count;
    }
    const a = makeCounter();
    const b = makeCounter();
    console.log(a(), b());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Каждый вызов `makeCounter()` создаёт **новый контекст выполнения** со своей собственной свежей `count`. `a` и `b` замыкают на *разные* привязки `count`. Они не делятся состоянием.

    Результат: `1, 1`

    Вот почему замыкания используются для создания приватного состояния — каждый «экземпляр» независим.

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={2}
  group="Классическая ловушка"
  title="var в цикле"
  options={[
    {text: '0, 1, 2, 3, 4'},
    {text: '5, 5, 5, 5, 5', isAnswer: true},
    {text: '1, 2, 3, 4, 5'},
    {text: 'undefined x5'},
  ]}
>
  <slot name="question">
  <div className="question">
    Что будет записано через 100мс?
    ```javascript
    for (var i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `var` имеет функционный уровень области видимости. Есть ровно **одна** `i` — все пять колбэков замыкают на одну и ту же привязку. К моменту, когда таймауты сработают, цикл закончится и `i` будет равен `5`.

    Пять колбэков. Одна `i`. Значение: `5`. Записано пять раз.

    Решение — `let` (область видимости блока, новая привязка каждой итерации) или обёртка в IIFE для захвата текущего значения:
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100); // 0, 1, 2, 3, 4 ✓
    }
    ```
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={3}
  group="Классическая ловушка"
  title="let решает (или нет?)"
  options={[
    {text: '0, 1, 2, 3, 4', isAnswer: true},
    {text: '5, 5, 5, 5, 5'},
    {text: '4, 4, 4, 4, 4'},
    {text: 'ReferenceError: i is not defined'},
  ]}
>
  <slot name="question">
  <div className="question">
    Тот же цикл, другое ключевое слово. Что запишется?
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `let` в цикле `for` — особый случай. Спецификация говорит: каждая итерация получает свою **новую привязку** `i`. Как будто цикл тайно делает это:
    
    ```javascript
    { let i = 0; setTimeout(() => console.log(i), 100) }
    { let i = 1; setTimeout(() => console.log(i), 100) }
    // ...
    ```
    
    Так что пять колбэков замыкают на пять различных переменных. Результат: `0, 1, 2, 3, 4`.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={4}
  group="Замыкания и объекты"
  title="Мутация объекта"
  options={[
    {text: '{ x: 1 }, { x: 1 }'},
    {text: '{ x: 1 }, { x: 99 }'},
    {text: '{ x: 99 }, { x: 99 }', isAnswer: true},
    {text: '{ x: 99 }, { x: 1 }'},
  ]}
>
  <slot name="question">
  <div className="question">
    Что это выведет?
    ```javascript
    function wrap() {
      const obj = { x: 1 };
      const getObj = () => obj;
      obj.x = 99;
      return getObj;
    }
    const get = wrap();
    console.log(get(), get());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Замыкание захватывает **привязку** `obj`, а не снимок объекта. Сам объект мутится после создания замыкания — но `obj` всё ещё указывает на ту же ссылку. Оба вызова возвращают один и тот же мутированный объект: `{ x: 99 }`.
    
    Это ловит людей, которые думают, что замыкания «замораживают» значения. Они не замораживают. Они захватывают **ссылки**.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={5}
  group="Продвинуто"
  title="Устаревшее замыкание в React"
  options={[
    {text: 'Записывает текущее count при каждом клике'},
    {text: 'Всегда записывает 0', isAnswer: true},
    {text: 'Записывает NaN'},
    {text: "Это не скомпилируется"},
  ]}
>
  <slot name="question">
  <div className="question">
    Какое поведение демонстрирует этот React-компонент при многократном нажатии на кнопку?
    ```jsx
    function Counter() {
      const [count, setCount] = useState(0);

      useEffect(() => {
        const id = setInterval(() => {
          console.log('count is', count);
        }, 1000);
        return () => clearInterval(id);
      }, []); // пустой массив зависимостей

      return <button onClick={() => setCount(c => c + 1)}>
        Клик ({count})
      </button>;
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Классическое устаревшее замыкание. `useEffect` выполняется один раз (пустой массив зависимостей) и создаёт интервал, который замыкает на `count` таким, каким он был при монтировании: `0`. Сколько раз вы ни нажимаете, интервал никогда не видит обновлённое значение.
    
    Метка кнопки обновляется корректно — это новый рендер. Но `console.log` навсегда застрял на `0`.
    
    Исправьте, добавив `count` в массив зависимостей (пересоздаст интервал) или используя ref:
    ```jsx
    const countRef = useRef(count);
    useEffect(() => { countRef.current = count; }, [count]);
    // затем используйте countRef.current внутри интервала
    ```
  </div>
  </slot>
</Challenge>

  client:visible={{rootMargin: "150px"}}
  index={6}
  group="Продвинуто"
  title="Паттерн модуля"
  options={[
    {text: '60', isAnswer: true},
    {text: 'TypeError: balance is not defined'},
    {text: '-40'},
    {text: '100'},
  ]}
>
  <slot name="question">
  <div className="question">
    Подождите — читайте внимательно. Что происходит на самом деле?
    ```javascript
    const bank = (() => {
      let balance = 0;
      return {
        deposit: (n) => (balance += n),
        withdraw: (n) => (balance -= n),
        getBalance: () => balance,
      };
    })();

    bank.deposit(100);
    bank.withdraw(40);
    console.log(bank.getBalance());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Ответ — `60`.
    
    Это **паттерн модуля** — IIFE, возвращающая объект замыканий. Все три метода замыкают на одну и ту же привязку `balance`. Она genuinely приватна: вы не можете получить доступ к `balance` снаружи.
    
    ```javascript
    console.log(bank.balance); // undefined
    ```
    
    До классов это был *единственный* способ инкапсулировать состояние в JavaScript. Вы всё ещё увидите это в легаси-кодобазах и утилитарных библиотеках.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={7}
  group="Эксперт"
  title="Утечка памяти через замыкание"
  options={[
    {text: 'Он записывает длину массива data'},
    {text: 'Он создаёт утечку памяти', isAnswer: true},
    {text: 'Он выбрасывает RangeError'},
    {text: 'Ничего необычного — это нормально'},
  ]}
>
  <slot name="question">
  <div className="question">
    В чём проблема этого кода?
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      element.addEventListener('click', () => {
        console.log('clicked!', data.length);
      });
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Обработчик событий замыкает на `data`, поэтому массив остаётся доступен, пока доступен обработчик. Если `element` долгоживущий и обработчик никогда не удаляется, этот большой массив тоже остаётся.
    
    `element` в DOM. Обработчик держит `data`. GC не может его собрать.
    
    Исправьте: не захватывайте большие данные в долгоживущих обработчиках, если они вам действительно не нужны, и удаляйте обработчики, когда владеющий UI разбирается:
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      // что-то сделайте с data...

      const onClick = () => {
        console.log('clicked!');
      };

      element.addEventListener('click', onClick);
      return () => element.removeEventListener('click', onClick);
    }
    ```
    
    Современные V8 лучше с этим справляются, чем раньше. Но в долгоживущих SPA с множеством обработчиков событий этот паттерн всё ещё ловит людей.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={8}
  group="Эксперт"
  title="Ловушка this"
  options={[
    {text: 'Dan, Dan'},
    {text: 'Dan, undefined'},
    {text: 'undefined, Dan'},
    {text: 'undefined, undefined', isAnswer: true},
  ]}
>
  <slot name="question">
  <div className="question">
    Что это выведет?
    ```javascript
    const user = {
      name: 'Dan',
      greetArrow: () => console.log(this?.name),
      greetRegular() {
        const inner = () => console.log(this?.name);
        inner();
      },
    };

    user.greetArrow();
    const fn = user.greetRegular;
    fn();
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Две тонкие ловушки:
    
    **`greetArrow`**: Стрелочные функции не имеют своего собственного `this`. Они захватывают `this` из окружающего лексического окружения — здесь это не `user`, а модуль/глобальная область. Так что `this?.name` — это `undefined`.
    
    **`fn()`**: `greetRegular` вызывается без получателя (`fn()` не `user.greetRegular()`). В строгом режиме `this` равен `undefined`. В нестрогом режиме — глобальный объект. В любом случае `this?.name` равен `undefined`.
    
    Если бы вы вызвали `user.greetRegular()` напрямую, вы бы получили `'Dan'` — потому что тогда `this` привязан к `user`.
    
    Результат: `undefined, undefined`.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={9}
  group="Бонус"
  title="Мемоизация через замыкание"
  options={[
    {text: 'Вызывает expensive() 3 раза'},
    {text: 'Вызывает expensive() 1 раз', isAnswer: true},
    {text: 'Вызывает expensive() 2 раза'},
    {text: 'Выбрасывает: Maximum call stack exceeded'},
  ]}
>
  <slot name="question">
  <div className="question">
    Сколько раз на самом деле вызывается `expensive()`?
    ```javascript
    function memoize(fn) {
      const cache = new Map();
      return (arg) => {
        if (cache.has(arg)) return cache.get(arg);
        const result = fn(arg);
        cache.set(arg, result);
        return result;
      };
    }

    let calls = 0;
    const expensive = (n) => { calls++; return n * 2; };
    const fast = memoize(expensive);

    fast(5);
    fast(5);
    fast(5);
    console.log(calls);
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Мемоизированная функция замыкает на `cache`. Первый вызов (`fast(5)`) промахивается по кэшу, вызывает `expensive` и хранит результат. Второй и третий вызовы попадают в кэш и возвращаются немедленно.
    
    `calls` равен `1`.
    
    Это замыкания, делающие реальную работу. `cache` приватна (недоступна снаружи `memoize`), устойчива (живёт, пока живёт `fast`), и общая для всех вызовов `fast`.
  </div>
  </slot>
</Challenge>

Как справились?

- **9-10**: Вы действительно понимаете замыкания. Вероятно, страшно работать с вами в паре.
- **6-8**: Твёрдо. Вопросы про устаревшее замыкание / `this` вас зацепили, что честно.
- **3-5**: Вы знаете концепцию, но края размыты. Прочитайте документацию MDN по цепочке областей видимости.
- **0-2**: Вы думали, что понимаете замыкания. Теперь понимаете. В этом и смысл.

Устаревшее замыкание в React (#6) и ловушка `this` (#8) — это те, что вызывают реальные production-баги. Если они были вам незнакомы, стоит вернуться к ним — они встречаются в код-ревью чаще любых других проблем, связанных с замыканиями.
