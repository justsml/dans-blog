<p class="inset">Les fermetures sont l'un de ces concepts que chaque développeur JavaScript *pense* comprendre — jusqu'au moment où il ne les comprend plus. Ce quiz révélera les limites de ce que vous savez vraiment.</p>

Les fermetures apparaissent dans les gestionnaires d'événements, les fonctions usines, les hooks React, les motifs de module, et pretty much partout ailleurs. Elles ne sont pas exotiques. Elles sont le tissu même dont JavaScript est tissé.

Les questions commencent de manière raisonnable, puis commencent à retirer les planches du plancher.

  <slot name="question">
  <div className="question">
    Que produit ce `console.log` ?
    ```javascript
    function makeCounter() {
      let count = 0;
      return () => ++count;
    }
    const tick = makeCounter();
    console.log(tick(), tick(), tick());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `tick` ferme sur `count`. Chaque appel incrémente et retourne la **même** variable `count`. Les trois appels produisent `1, 2, 3`.

    Le mot-clé : *même*. `count` n'est pas copié dans la fermeture. La fermeture conserve une **référence vivante** à la variable. C'est tout le principe.
  </div>
  </slot>

  group="Échauffement"
  title="Fermetures séparées"
  options={[
    {text: '1, 1', isAnswer: true},
    {text: '1, 2'},
    {text: '2, 2'},
    {text: '0, 0'},
  ]}
>
  <slot name="question">
  <div className="question">
    Que produit ce `console.log` ?
    ```javascript
    function makeCounter() {
      let count = 0;
      return () => ++count;
    }
    const a = makeCounter();
    const b = makeCounter();
    console.log(a(), b());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Chaque appel à `makeCounter()` crée un **nouveau contexte d'exécution** avec son propre `count` frais. `a` et `b` ferment sur des liaisons `count` **différentes**. Elles ne partagent pas l'état.

    Résultat : `1, 1`

    C'est pourquoi les fermetures sont utilisées pour construire un état privé — chaque "instance" est indépendante.

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={2}
  group="Le piège classique"
  title="var dans une boucle"
  options={[
    {text: '0, 1, 2, 3, 4'},
    {text: '5, 5, 5, 5, 5', isAnswer: true},
    {text: '1, 2, 3, 4, 5'},
    {text: 'undefined x5'},
  ]}
>
  <slot name="question">
  <div className="question">
    Que produit ce `console.log` après 100ms ?
    ```javascript
    for (var i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `var` est à portée fonctionnelle. Il y a exactement **un** `i` — les cinq callbacks ferment sur la même liaison. Quand les timeouts se déclenchent, la boucle est terminée et `i` vaut `5`.

    Cinq callbacks. Un `i`. Valeur : `5`. Affiché cinq fois.

    La solution est `let` (à portée de bloc, nouvelle liaison à chaque itération) ou envelopper dans une IIFE pour capturer la valeur actuelle :
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100); // 0, 1, 2, 3, 4 ✓
    }
    ```
  </div>
  </slot>
</Challenge>

>
  <slot name="question">
  <div className="question">
    Même boucle, mot-clé différent. Que produit `console.log` ?
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `let` dans une boucle `for` est spécial. La spécification dit : chaque itération obtient sa propre **nouvelle liaison** de `i`. C'est comme si la boucle faisait secrètement ceci :

    ```javascript
    { let i = 0; setTimeout(() => console.log(i), 100) }
    { let i = 1; setTimeout(() => console.log(i), 100) }
    // ...
    ```

    Les cinq callbacks ferment donc sur cinq variables distinctes. Résultat : `0, 1, 2, 3, 4`.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={4}
  group="Fermetures & objets"
  title="Mutation d'objet"
  options={[
    {text: '{ x: 1 }, { x: 1 }'},
    {text: '{ x: 1 }, { x: 99 }'},
    {text: '{ x: 99 }, { x: 99 }', isAnswer: true},
    {text: '{ x: 99 }, { x: 1 }'},
  ]}
>
  <slot name="question">
  <div className="question">
    Que produit ce `console.log` ?
    ```javascript
    function wrap() {
      const obj = { x: 1 };
      const getObj = () => obj;
      obj.x = 99;
      return getObj;
    }
    const get = wrap();
    console.log(get(), get());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La fermeture capture la **liaison** `obj`, pas un instantané de l'objet. L'objet lui-même est muté après que la fermeture a été créée — mais `obj` pointe toujours vers la même référence. Les deux appels retournent le même objet muté : `{ x: 99 }`.

    Cela trompe les gens qui supposent que les fermetures "figent" les valeurs. Elles ne le font pas. Elles capturent des **références**.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={5}
  group="Avancé"
  title="Fermeture obsolète dans React"
  options={[
    {text: 'Affiche la valeur actuelle de count à chaque clic'},
    {text: 'Affiche toujours 0', isAnswer: true},
    {text: 'Affiche NaN'},
    {text: 'Cela ne compile pas'},
  ]}
>
  <slot name="question">
  <div className="question">
    Quel comportement ce composant React exhibe-t-il quand on clique plusieurs fois sur le bouton ?
    ```jsx
    function Counter() {
      const [count, setCount] = useState(0);

      useEffect(() => {
        const id = setInterval(() => {
          console.log('count is', count);
        }, 1000);
        return () => clearInterval(id);
      }, []); // deps vide

      return <button onClick={() => setCount(c => c + 1)}>
        Cliquez ({count})
      </button>;
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Fermeture obsolète classique. `useEffect` s'exécute une fois (tableau de deps vide) et crée un intervalle qui ferme sur `count` tel qu'il était au montage : `0`. Peu importe combien de fois vous cliquez, le closure de l'intervalle ne voit jamais la valeur mise à jour.

    L'étiquette du bouton se met à jour correctement — c'est un nouveau rendu. Mais `console.log` est à jamais coincé à `0`.

    Corrigez-le en ajoutant `count` au tableau de deps (recrée l'intervalle) ou en utilisant un ref :
    ```jsx
    const countRef = useRef(count);
    useEffect(() => { countRef.current = count; }, [count]);
    // puis utilisez countRef.current à l'intérieur de l'intervalle
    ```
  </div>
  </slot>
</Challenge>

  client:visible={{rootMargin: "150px"}}
  index={6}
  group="Avancé"
  title="Motif de module"
  options={[
    {text: '60', isAnswer: true},
    {text: 'TypeError: balance is not defined'},
    {text: '-40'},
    {text: '100'},
  ]}
>
  <slot name="question">
  <div className="question">
    Attendez — lisez attentivement. Que se passe-t-il réellement ?
    ```javascript
    const bank = (() => {
      let balance = 0;
      return {
        deposit: (n) => (balance += n),
        withdraw: (n) => (balance -= n),
        getBalance: () => balance,
      };
    })();

    bank.deposit(100);
    bank.withdraw(40);
    console.log(bank.getBalance());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La réponse est `60`.

    C'est le **motif de module** — une IIFE qui retourne un objet de fermetures. Les trois méthodes ferment sur la même liaison `balance`. C'est véritablement privé : vous ne pouvez pas accéder à `balance` depuis l'extérieur.

    ```javascript
    console.log(bank.balance); // undefined
    ```

    Avant les classes, c'était *la* façon d'encapsuler l'état en JavaScript. Vous le verrez encore dans les bases de code héritées et les bibliothèques utilitaires.
  </div>
  </slot>
</Challenge>

  client:visible={{rootMargin: "150px"}}
  index={7}
  group="Expert"
  title="Fuite de mémoire par fermeture"
  options={[
    {text: 'Il affiche la longueur du tableau data'},
    {text: 'Il crée une fuite de mémoire', isAnswer: true},
    {text: 'Il lance un RangeError'},
    {text: 'Rien d'anormal — c'est correct'},
  ]}
>
  <slot name="question">
  <div className="question">
    Quel est le problème avec ce code ?
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      element.addEventListener('click', () => {
        console.log('clicked!', data.length);
      });
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Le gestionnaire d'événements ferme sur `data`, donc le tableau reste accessible tant que le gestionnaire est accessible. Si `element` est de longue durée et que le gestionnaire n'est jamais supprimé, ce grand tableau est conservé aussi.

    `element` est dans le DOM. Le gestionnaire maintient `data`. GC ne peut pas le collecter.

    Corrigé : ne capturez pas de grandes données dans des gestionnaires de longue durée sauf si vous en avez vraiment besoin, et supprimez les gestionnaires quand l'interface utilisateur propriétaire est détruite :
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      // faites quelque chose avec data...

      const onClick = () => {
        console.log('clicked!');
      };

      element.addEventListener('click', onClick);
      return () => element.removeEventListener('click', onClick);
    }
    ```

    V8 moderne est meilleur que par le passé. Mais dans les SPAs de longue durée avec beaucoup de gestionnaires d'événements, ce motif continue de tromper des gens.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={8}
  group="Expert"
  title="Le piège de this"
  options={[
    {text: 'Dan, Dan'},
    {text: 'Dan, undefined'},
    {text: 'undefined, Dan'},
    {text: 'undefined, undefined', isAnswer: true},
  ]}
>
  <slot name="question">
  <div className="question">
    Que produit ce `console.log` ?
    ```javascript
    const user = {
      name: 'Dan',
      greetArrow: () => console.log(this?.name),
      greetRegular() {
        const inner = () => console.log(this?.name);
        inner();
      },
    };

    user.greetArrow();
    const fn = user.greetRegular;
    fn();
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Deux pièges subtils :

    **`greetArrow`** : Les fonctions fléchées n'ont pas leur propre `this`. Elles capturent `this` du contexte lexical environnant — qui est ici le contexte module/global, pas `user`. Donc `this?.name` est `undefined`.

    **`fn()`** : `greetRegular` est appelé sans récepteur (`fn()` pas `user.greetRegular()`). En mode strict, `this` est `undefined`. En mode sloppy, c'est l'objet global. Dans les deux cas, `this?.name` est `undefined`.

    Si vous aviez appelé `user.greetRegular()` directement, vous auriez obtenu `'Dan'` — car alors `this` est lié à `user`.

    Résultat : `undefined, undefined`.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={9}
  group="Bonus"
  title="Mémorisation par fermeture"
  options={[
    {text: 'Appelle expensive() 3 fois'},
    {text: 'Appelle expensive() 1 fois', isAnswer: true},
    {text: 'Appelle expensive() 2 fois'},
    {text: 'Lance : Maximum call stack exceeded'},
  ]}
>
  <slot name="question">
  <div className="question">
    Combien de fois `expensive()` est-il réellement appelé ?
    ```javascript
    function memoize(fn) {
      const cache = new Map();
      return (arg) => {
        if (cache.has(arg)) return cache.get(arg);
        const result = fn(arg);
        cache.set(arg, result);
        return result;
      };
    }

    let calls = 0;
    const expensive = (n) => { calls++; return n * 2; };
    const fast = memoize(expensive);

    fast(5);
    fast(5);
    fast(5);
    console.log(calls);
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La fonction mémorisée ferme sur `cache`. Le premier appel (`fast(5)`) rate le cache, appelle `expensive`, et stocke le résultat. Les deuxième et troisième appels touchent le cache et retournent immédiatement.

    `calls` est `1`.

    C'est les fermetures qui font un travail réel. `cache` est privé (inaccessible depuis l'extérieur de `memoize`), persistant (existe tant que `fast` existe), et partagé entre tous les appels à `fast`.
  </div>
  </slot>
</Challenge>

Comment s'est-il passé ?

- **9-10** : Vous comprenez vraiment les fermetures. Probablement effrayant à faire du pair-programming avec.
- **6-8** : Solide. Les questions sur la fermeture obsolète / `this` vous ont pris au dépourvu, ce qui est honnête.
- **3-5** : Vous connaissez le concept mais les contours sont flous. Allez lire les docs MDN sur la chaîne de portée.
- **0-2** : Vous pensiez connaître les fermetures. Maintenant vous les connaissez. C'est le but.

La fermeture obsolète dans React (#6) et le piège `this` (#8) sont ceux qui causent de vrais bugs de production. Si ceux-ci vous sont restés étrangers, ils méritent d'être revisités — ils apparaissent en revue de code plus que tout autre problème lié aux fermetures.
