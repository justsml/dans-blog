<p class="inset">Les fermetures sont l'un de ces concepts que tout développeur JavaScript *pense* comprendre — jusqu'au moment où ce n'est plus le cas. Ce quiz va explorer les limites de ce que vous savez vraiment.</p>

Les fermetures apparaissent dans les gestionnaires d'événements, les fonctions d'usine, les hooks React, les motifs de module, et à peu près partout ailleurs. Ce n'est pas exotique. C'est le tissu même dont JavaScript est tissé.

Les questions commencent raisonnablement puis retirent les planchers les unes après les autres.

    ```javascript
    function makeCounter() {

    ```

  title="Fermetures Séparées"
  options={[
    {text: '1, 1', isAnswer: true},
    {text: '1, 2'},
    {text: '2, 2'},
    {text: '0, 0'},
  ]}
>
  <slot name="question">
  <div className="question">
    Qu'affiche ce code ?
    ```javascript
    function makeCounter() {
      let count = 0;
      return () => ++count;
    }
    const a = makeCounter();
    const b = makeCounter();
    console.log(a(), b());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Chaque appel à `makeCounter()` crée un **nouveau contexte d'exécution** avec son propre `count` tout neuf. `a` et `b` capturent des liaisons `count` *différentes*. Elles ne partagent pas d'état.

    Résultat : `1, 1`

    C'est pourquoi les fermetures sont utilisées pour construire un état privé — chaque « instance » est indépendante.

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={2}
  group="Le Piège Classique"
  title="var dans une Boucle"
  options={[
    {text: '0, 1, 2, 3, 4'},
    {text: '5, 5, 5, 5, 5', isAnswer: true},
    {text: '1, 2, 3, 4, 5'},
    {text: 'undefined x5'},
  ]}
>
  <slot name="question">
  <div className="question">
    Qu'est-ce qui est affiché après 100 ms ?
    ```javascript
    for (var i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `var` a une portée de fonction. Il y a exactement **un** `i` — les cinq callbacks capturent la même liaison. Quand les minuteries se déclenchent, la boucle est terminée et `i` vaut `5`.

    Cinq callbacks. Un `i`. Valeur : `5`. Affiché cinq fois.

    La solution est `let` (portée de bloc, nouvelle liaison à chaque itération) ou l'utilisation d'une IIFE pour capturer la valeur courante :
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100); // 0, 1, 2, 3, 4 ✓
    }
    ```
  </div>
  </slot>
</Challenge>

>
  <slot name="question">
  <div className="question">
    Même boucle, mot-clé différent. Qu'affiche-t-elle ?
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `let` dans une boucle `for` est spécial. La spécification dit : chaque itération obtient sa **nouvelle liaison** de `i`. C'est comme si la boucle faisait secrètement ceci :

    ```javascript
    { let i = 0; setTimeout(() => console.log(i), 100) }
    { let i = 1; setTimeout(() => console.log(i), 100) }
    // ...
    ```

    Les cinq callbacks capturent donc cinq variables distinctes. Résultat : `0, 1, 2, 3, 4`.
  </div>
  </slot>
</Challenge>

>
  <slot name="question">
  <div className="question">
    Qu'affiche ce code ?
    ```javascript
    function wrap() {
      const obj = { x: 1 };
      const getObj = () => obj;
      obj.x = 99;
      return getObj;
    }
    const get = wrap();
    console.log(get(), get());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La fermeture capture la **liaison** `obj`, pas un instantané de l'objet. L'objet lui-même est muté après la création de la fermeture — mais `obj` pointe toujours vers la même référence. Les deux appels retournent le même objet muté : `{ x: 99 }`.

    Cet écueil piège ceux qui pensent que les fermetures « figent » les valeurs. Ce n'est pas le cas. Elles capturent des **références**.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={5}
  group="Avancé"
  title="Fermeture Obsolète dans React"
  options={[
    {text: 'Affiche le count actuel à chaque clic'},
    {text: 'Affiche toujours 0', isAnswer: true},
    {text: 'Affiche NaN'},
    {text: 'Ce code ne compile pas'},
  ]}
>
  <slot name="question">
  <div className="question">
    Quel comportement ce composant React a-t-il lorsqu'on clique plusieurs fois sur le bouton ?
    ```jsx
    function Counter() {
      const [count, setCount] = useState(0);

      useEffect(() => {
        const id = setInterval(() => {
          console.log('count is', count);
        }, 1000);
        return () => clearInterval(id);
      }, []); // empty deps

      return <button onClick={() => setCount(c => c + 1)}>
        Click ({count})
      </button>;
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Fermeture obsolète classique. Le `useEffect` s'exécute une fois (tableau de dépendances vide) et crée un intervalle qui capture `count` tel qu'il était au montage : `0`. Peu importe combien de fois vous cliquez, la fermeture de l'intervalle ne voit jamais la valeur mise à jour.

    Le libellé du bouton se met à jour correctement — c'est un nouveau rendu. Mais `console.log` reste piégé à `0` pour toujours.

    Correction : ajouter `count` au tableau de dépendances (recrée l'intervalle) ou utiliser une ref :
    ```jsx
    const countRef = useRef(count);
    useEffect(() => { countRef.current = count; }, [count]);
    // puis utiliser countRef.current dans l'intervalle
    ```
  </div>
  </slot>
</Challenge>

    {text: 'TypeError : balance n\'est pas définie'},
    {text: '-40'},
    {text: '100'},
  ]}
>
  <slot name="question">
  <div className="question">
    Attendez — lisez attentivement. Que se passe-t-il vraiment ?
    ```javascript
    const bank = (() => {
      let balance = 0;
      return {
        deposit: (n) => (balance += n),
        withdraw: (n) => (balance -= n),
        getBalance: () => balance,
      };
    })();

    bank.deposit(100);
    bank.withdraw(40);
    console.log(bank.getBalance());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La réponse est `60`.

    C'est le **Motif de Module** — une IIFE qui retourne un objet de fermetures. Les trois méthodes capturent la même liaison `balance`. C'est réellement privé : vous ne pouvez pas accéder à `balance` de l'extérieur.

    ```javascript
    console.log(bank.balance); // undefined
    ```

    Avant les classes, c'était *la* façon d'encapsuler l'état en JavaScript. On le voit encore dans les bases de code legacy et les bibliothèques utilitaires.
  </div>
  </slot>
</Challenge>

  client:visible={{rootMargin: "150px"}}

  group="Expert"
  title="Fuite Mémoire par Fermeture"
  options={[
    {text: 'Elle affiche la longueur du tableau data'},
    {text: 'Elle crée une fuite mémoire', isAnswer: true},
    {text: 'Elle lève une RangeError'},
    {text: 'Rien d\'anormal — ce code est correct'},
  ]}
>
  <slot name="question">
  <div className="question">
    Quel est le problème avec ce code ?
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      element.addEventListener('click', () => {
        console.log('clicked!', data.length);
      });
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    L'écouteur d'événement capture `data`, donc le tableau reste accessible tant que l'écouteur est accessible. Si `element` a une longue durée de vie et que l'écouteur n'est jamais retiré, ce grand tableau est conservé aussi.

    `element` est dans le DOM. L'écouteur retient `data`. Le GC ne peut pas la collecter.

    Correction : ne capturez pas de grandes données dans des écouteurs longue durée sauf si nécessaire, et retirez les écouteurs quand l'UI propriétaire est détruite :
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      // faire quelque chose avec data...

      const onClick = () => {
        console.log('clicked!');
      };

      element.addEventListener('click', onClick);
      return () => element.removeEventListener('click', onClick);
    }
    ```

    V8 moderne gère mieux cela qu'avant. Mais dans les SPA longue durée avec beaucoup d'écouteurs d'événements, ce motif piège encore du monde.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={8}
  group="Expert"
  title="Le Piège du this"
  options={[
    {text: 'Dan, Dan'},
    {text: 'Dan, undefined'},
    {text: 'undefined, Dan'},
    {text: 'undefined, undefined', isAnswer: true},
  ]}
>
  <slot name="question">
  <div className="question">
    Qu'affiche ce code ?
    ```javascript
    const user = {
      name: 'Dan',
      greetArrow: () => console.log(this?.name),
      greetRegular() {
        const inner = () => console.log(this?.name);
        inner();
      },
    };

    user.greetArrow();
    const fn = user.greetRegular;
    fn();
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Deux pièges subtils :

    **`greetArrow`** : Les fonctions fléchées n'ont pas leur propre `this`. Elles capturent `this` de la portée lexicale englobante — ici la portée du module/globale, pas `user`. Donc `this?.name` est `undefined`.

    **`fn()`** : `greetRegular` est appelée sans récepteur (`fn()` et non `user.greetRegular()`). En mode strict, `this` est `undefined`. En mode non-strict, c'est l'objet global. Dans les deux cas, `this?.name` est `undefined`.

    Si vous aviez appelé `user.greetRegular()` directement, vous auriez obtenu `'Dan'` — car `this` serait alors lié à `user`.

    Résultat : `undefined, undefined`.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={9}
  group="Bonus"
  title="Mémoïsation par Fermeture"
  options={[
    {text: 'Appelle expensive() 3 fois'},
    {text: 'Appelle expensive() 1 fois', isAnswer: true},
    {text: 'Appelle expensive() 2 fois'},
    {text: 'Lève une erreur : Maximum call stack exceeded'},
  ]}
>
  <slot name="question">
  <div className="question">
    Combien de fois `expensive()` est-elle réellement appelée ?
    ```javascript
    function memoize(fn) {
      const cache = new Map();
      return (arg) => {
        if (cache.has(arg)) return cache.get(arg);
        const result = fn(arg);
        cache.set(arg, result);
        return result;
      };
    }

    let calls = 0;
    const expensive = (n) => { calls++; return n * 2; };
    const fast = memoize(expensive);

    fast(5);
    fast(5);
    fast(5);
    console.log(calls);
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La fonction mémoïsée capture `cache`. Le premier appel (`fast(5)`) rate le cache, appelle `expensive`, et stocke le résultat. Les deuxième et troisième appels touchent le cache et retournent immédiatement.

    `calls` vaut `1`.

    C'est le travail concret des fermetures. `cache` est privé (inaccessible depuis l'extérieur de `memoize`), persistant (vit aussi longtemps que `fast` existe), et partagé entre tous les appels à `fast`.
  </div>
  </slot>
</Challenge>

Comment vous en êtes-vous sortis ?

- **9–10** : Vous maîtrisez vraiment les fermetures. Probablement intimidant en programmation en binôme.
- **6–8** : Solide. La fermeture obsolète / les questions sur `this` vous ont piégé, ce qui est honnête.
- **3–5** : Vous connaissez le concept mais les limites sont floues. Allez lire la documentation MDN sur la chaîne de portée.
- **0–2** : Vous pensiez connaître les fermetures. Maintenant c'est le cas. C'est tout l'intérêt.

La fermeture obsolète dans React (#6) et le piège du `this` (#8) sont ceux qui causent de vrais bugs en production. Si ceux-ci vous semblaient peu familiers, ils méritent qu'on s'y attarde — ils apparaissent en revue de code plus que tout autre problème lié aux fermetures.
