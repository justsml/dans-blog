<p class="inset">Les closures sont l'un de ces concepts que chaque développeur JavaScript *pense* comprendre — jusqu'au moment où il ne comprend plus. Ce quiz révélera les limites de ce que vous savez réellement.</p>

Les closures apparaissent dans les gestionnaires d'événements, les factory functions, les hooks React, les patterns module, et à peu près partout ailleurs. Elles ne sont pas exotiques. Elles constituent le tissu dont le JavaScript est fait.

Les questions commencent raisonnablement, puis commencent à retirer le plancher.

    Qu'est-ce que cela affiche ?
    ```javascript
    function makeCounter() {
      let count = 0;
      return () => ++count;
    }
    const a = makeCounter();
    const b = makeCounter();
    console.log(a(), b());
    ```
  </div>

  <div className="explanation">
    Chaque appel à `makeCounter()` crée un **nouveau contexte d'exécution** avec son propre `count`. `a` et `b` ferment sur des liaisons `count` *différentes`. Ils ne partagent pas d'état.

    Résultat : `1, 1`

    C'est pourquoi les closures servent à construire un état privé — chaque « instance » est indépendante.

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={2}
  group="Le Piège Classique"
  title="var dans une Boucle"
  options={[
    {text: '0, 1, 2, 3, 4'},
    {text: '5, 5, 5, 5, 5', isAnswer: true},
    {text: '1, 2, 3, 4, 5'},
    {text: 'undefined x5'},
  ]}
>
  <slot name="question">
  <div className="question">
    Qu'est-ce qui est affiché après 100ms ?
    ```javascript
    for (var i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `var` est scopé à la fonction. Il y a exactement **un seul** `i` — les cinq callbacks ferment sur la même liaison. Quand les timeouts se déclenchent, la boucle est terminée et `i` vaut `5`.

    Cinq callbacks. Un `i`. Valeur : `5`. Affiché cinq fois.

    La solution est `let` (scopé au bloc, nouvelle liaison à chaque itération) ou l'enveloppement dans une IIFE pour capturer la valeur courante :
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100); // 0, 1, 2, 3, 4 ✓
    }
    ```
  </div>
  </slot>
</Challenge>

>
  <slot name="question">
  <div className="question">
    Même boucle, mot-clé différent. Qu'est-ce qui s'affiche ?
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `let` dans une boucle `for` est spécial. Le spec dit : chaque itération obtient sa propre **nouvelle liaison** de `i`. C'est comme si la boucle faisait secrètement ceci :

    ```javascript
    { let i = 0; setTimeout(() => console.log(i), 100) }
    { let i = 1; setTimeout(() => console.log(i), 100) }
    // ...
    ```

    Les cinq callbacks ferment donc sur cinq variables distinctes. Résultat : `0, 1, 2, 3, 4`.
  </div>
  </slot>
</Challenge>

  client:visible={{rootMargin: "150px"}}
  index={4}
  group="Closures & Objets"
  title="Mutation d'Objet"
  options={[
    {text: '{ x: 1 }, { x: 1 }'},
    {text: '{ x: 1 }, { x: 99 }'},
    {text: '{ x: 99 }, { x: 99 }', isAnswer: true},

  ]}
>
  <slot name="question">
  <div className="question">
    Qu'est-ce que cela affiche ?
    ```javascript
    function wrap() {
      const obj = { x: 1 };
      const getObj = () => obj;
      obj.x = 99;
      return getObj;
    }
    const get = wrap();
    console.log(get(), get());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La closure capture la **liaison** `obj`, pas un instantané de l'objet. L'objet lui-même est muté après la création de la closure — mais `obj` pointe toujours vers la même référence. Les deux appels retournent le même objet muté : `{ x: 99 }`.

    Cela piège ceux qui supposent que les closures « gèlent » les valeurs. Elles ne le font pas. Elles capturent des **références**.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={5}
  group="Avancé"
  title="Closure Obsolète dans React"
  options={[
    {text: 'Affiche le count courant à chaque clic'},
    {text: 'Affiche toujours 0', isAnswer: true},
    {text: 'Affiche NaN'},
    {text: "Ne compile pas"},
  ]}
>
  <slot name="question">
  <div className="question">
    Quel comportement ce composant React présente-t-il quand on clique plusieurs fois sur le bouton ?
    ```jsx
    function Counter() {
      const [count, setCount] = useState(0);

      useEffect(() => {
        const id = setInterval(() => {
          console.log('count is', count);
        }, 1000);
        return () => clearInterval(id);
      }, []); // empty deps

      return <button onClick={() => setCount(c => c + 1)}>
        Click ({count})
      </button>;
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Closure obsolète classique. Le `useEffect` s'exécute une fois (dep array vide) et crée un interval qui ferme sur `count` tel qu'il était au moment du montage : `0`. Peu importe combien de fois vous cliquez, la closure de l'intervalle ne voit jamais la valeur mise à jour.

    Le label du bouton se met à jour correctement — c'est un rendu frais. Mais `console.log` reste piégé à `0` pour toujours.

    Corrigez-le en ajoutant `count` au dep array (recrée l'intervalle) ou en utilisant une ref :
    ```jsx
    const countRef = useRef(count);
    useEffect(() => { countRef.current = count; }, [count]);
    // puis utilisez countRef.current dans l'intervalle
    ```
  </div>
  </slot>
</Challenge>

  title="Pattern Module"

    {text: 'TypeError: balance is not defined'},
    {text: '-40'},
    {text: '100'},
  ]}
>
  <slot name="question">
  <div className="question">
    Attendez — lisez attentivement. Que se passe-t-il réellement ?
    ```javascript
    const bank = (() => {
      let balance = 0;
      return {
        deposit: (n) => (balance += n),
        withdraw: (n) => (balance -= n),
        getBalance: () => balance,
      };
    })();

    bank.deposit(100);
    bank.withdraw(40);
    console.log(bank.getBalance());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La réponse est `60`.

    C'est le **Pattern Module** — une IIFE qui retourne un objet de closures. Les trois méthodes ferment sur la même liaison `balance`. Elle est vraiment privée : vous ne pouvez pas accéder à `balance` depuis l'extérieur.

    ```javascript
    console.log(bank.balance); // undefined
    ```

    Avant les classes, c'était *la* façon d'encapsuler l'état en JavaScript. Vous le verrez encore dans les codebases legacy et les bibliothèques utilitaires.
  </div>
  </slot>
</Challenge>

  group="Expert"
  title="Fuite Mémoire par Closure"
  options={[
    {text: 'Il affiche la longueur du tableau de données'},
    {text: 'Il crée une fuite mémoire', isAnswer: true},
    {text: 'Il lance une RangeError'},
    {text: 'Rien d\'inhabituel — c\'est correct'},
  ]}
>
  <slot name="question">
  <div className="question">
    Quel est le problème avec ce code ?
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      element.addEventListener('click', () => {
        console.log('clicked!', data.length);
      });
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    L'event listener ferme sur `data`, donc le tableau reste joignable aussi longtemps que le listener est joignable. Si `element` est de longue durée et que le listener n'est jamais supprimé, ce gros tableau est conservé aussi.

    `element` est dans le DOM. Le listener retient `data`. Le GC ne peut pas le collecter.

    Correction : ne capturez pas de gros données dans des listeners de longue durée sauf si vous en avez vraiment besoin, et supprimez les listeners quand l'UI propriétaire est démontée :
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      // faites quelque chose avec data...

      const onClick = () => {
        console.log('clicked!');
      };

      element.addEventListener('click', onClick);
      return () => element.removeEventListener('click', onClick);
    }
    ```

    V8 moderne est meilleur là-dessus qu'avant. Mais dans les SPAs de longue durée avec beaucoup d'event listeners, ce pattern mord encore les gens.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={8}
  group="Expert"
  title="Le Piège du this"
  options={[
    {text: 'Dan, Dan'},
    {text: 'Dan, undefined'},
    {text: 'undefined, Dan'},
    {text: 'undefined, undefined', isAnswer: true},
  ]}
>
  <slot name="question">
  <div className="question">
    Qu'est-ce que cela affiche ?
    ```javascript
    const user = {
      name: 'Dan',
      greetArrow: () => console.log(this?.name),
      greetRegular() {
        const inner = () => console.log(this?.name);
        inner();
      },
    };

    user.greetArrow();
    const fn = user.greetRegular;
    fn();
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Deux pièges subtils :

    **`greetArrow`** : Les fonctions fléchées n'ont pas leur propre `this`. Elles capturent `this` du scope lexical englobant — ici le scope module/global, pas `user`. Donc `this?.name` est `undefined`.

    **`fn()`** : `greetRegular` est appelé sans récepteur (`fn()` et non `user.greetRegular()`). En mode strict, `this` est `undefined`. En mode sloppy, c'est l'objet global. Dans les deux cas, `this?.name` est `undefined`.

    Si vous aviez appelé `user.greetRegular()` directement, vous obtiendriez `'Dan'` — parce que `this` serait lié à `user`.

    Résultat : `undefined, undefined`.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={9}
  group="Bonus"
  title="Mémorisation via Closure"
  options={[
    {text: 'Appelle expensive() 3 fois'},
    {text: 'Appelle expensive() 1 fois', isAnswer: true},
    {text: 'Appelle expensive() 2 fois'},
    {text: 'Lance : Maximum call stack exceeded'},
  ]}
>
  <slot name="question">
  <div className="question">
    Combien de fois `expensive()` est-elle réellement appelée ?
    ```javascript
    function memoize(fn) {
      const cache = new Map();
      return (arg) => {
        if (cache.has(arg)) return cache.get(arg);
        const result = fn(arg);
        cache.set(arg, result);
        return result;
      };
    }

    let calls = 0;
    const expensive = (n) => { calls++; return n * 2; };
    const fast = memoize(expensive);

    fast(5);
    fast(5);
    fast(5);
    console.log(calls);
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La fonction mémorisée ferme sur `cache`. Le premier appel (`fast(5)`) manque le cache, appelle `expensive`, et stocke le résultat. Les deuxième et troisième appels touchent le cache et retournent immédiatement.

    `calls` vaut `1`.

    Ici, les closures font du vrai travail. `cache` est privé (inaccessible depuis l'extérieur de `memoize`), persistant (vit aussi longtemps que `fast` existe), et partagé entre tous les appels à `fast`.
  </div>
  </slot>
</Challenge>

Comment avez-vous réussi ?

- **9-10** : Vous connaissez vraiment les closures. Probablement effrayant en pair-programming.
- **6-8** : Solide. Les questions closure obsolète / `this` vous ont piégé, ce qui est honnête.
- **3-5** : Vous connaissez le concept mais les bords sont flous. Allez lire la doc MDN sur la chaîne de scopes.
- **0-2** : Vous pensiez connaître les closures. Maintenant vous savez. C'est le but.

La closure obsolète dans React (n°6) et le piège du `this` (n°8) sont ceux qui causent de vrais bugs en production. Si ceux-là vous semblent peu familiers, ils valent la peine d'être revus — ils apparaissent en code review plus que tout autre problème lié aux closures.
