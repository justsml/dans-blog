<p class="inset">Los closures son uno de esos conceptos que todo desarrollador de JavaScript *cree* entender — justo hasta el momento en que no lo hace. Este quiz encontrará los límites de lo que realmente sabes.</p>

Los closures aparecen en event handlers, factory functions, React hooks, patrones de módulo, y prácticamente en todas partes. No son exóticos. Son la tela con la que está tejido JavaScript.

Las preguntas comienzan razonables y luego empiezan a quitar tablones del piso.

  title="Contador Clásico"
  options={[
    {text: '0, 0, 0'},

  <slot name="question">
  <div className="question">
    ¿Qué imprime esto?
    ```javascript
    function makeCounter() {
      let count = 0;
      return () => ++count;
    }
    const tick = makeCounter();
    console.log(tick(), tick(), tick());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `tick` cierra sobre `count`. Cada llamada incrementa y devuelve la *misma* variable `count`. Las tres llamadas producen `1, 2, 3`.

    La palabra clave: *misma*. `count` no se copia en el closure. El closure mantiene una **referencia en vivo** al binding de la variable. Ahí está todo el asunto.
  </div>

  group="Calentamiento"
  title="Closures Separados"
  options={[
    {text: '1, 1', isAnswer: true},
    {text: '1, 2'},
    {text: '2, 2'},
    {text: '0, 0'},
  ]}
>
  <slot name="question">
  <div className="question">
    ¿Qué imprime esto?
    ```javascript
    function makeCounter() {
      let count = 0;
      return () => ++count;
    }
    const a = makeCounter();
    const b = makeCounter();
    console.log(a(), b());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Cada llamada a `makeCounter()` crea un **nuevo contexto de ejecución** con su propio `count` fresco. `a` y `b` cierran sobre bindings de `count` *diferentes*. No comparten estado.

    Resultado: `1, 1`

    Por esto los closures se usan para construir estado privado — cada "instancia" es independiente.

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={2}
  group="La Trampa Clásica"
  title="var en un Bucle"
  options={[
    {text: '0, 1, 2, 3, 4'},
    {text: '5, 5, 5, 5, 5', isAnswer: true},
    {text: '1, 2, 3, 4, 5'},
    {text: 'undefined x5'},
  ]}
>
  <slot name="question">
  <div className="question">
    ¿Qué se imprime después de 100ms?
    ```javascript
    for (var i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `var` tiene ámbito de función. Hay exactamente **un** `i` — los cinco callbacks cierran sobre el mismo binding. Para cuando los timeouts disparan, el bucle ha terminado y `i` es `5`.

    Cinco callbacks. Un `i`. Valor: `5`. Impreso cinco veces.

    La solución es `let` ( ámbito de bloque, nuevo binding en cada iteración) o encerrar en un IIFE para capturar el valor actual:
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100); // 0, 1, 2, 3, 4 ✓
    }
    ```
  </div>
  </slot>
</Challenge>

  <div className="question">
    Mismo bucle, diferente keyword. ¿Qué se imprime?
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `let` en un bucle `for` es especial. La especificación dice: cada iteración obtiene su propio **nuevo binding** de `i`. Es como si el bucle secretamente hiciera esto:

    ```javascript
    { let i = 0; setTimeout(() => console.log(i), 100) }
    { let i = 1; setTimeout(() => console.log(i), 100) }
    // ...
    ```

    Así que los cinco callbacks cierran sobre cinco variables distintas. Resultado: `0, 1, 2, 3, 4`.
  </div>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={4}
  group="Closures y Objetos"
  title="Mutación de Objetos"
  options={[
    {text: '{ x: 1 }, { x: 1 }'},
    {text: '{ x: 1 }, { x: 99 }'},
    {text: '{ x: 99 }, { x: 99 }', isAnswer: true},
    {text: '{ x: 99 }, { x: 1 }'},
  ]}
>
  <slot name="question">
  <div className="question">
    ¿Qué imprime esto?
    ```javascript
    function wrap() {
      const obj = { x: 1 };
      const getObj = () => obj;
      obj.x = 99;
      return getObj;
    }
    const get = wrap();
    console.log(get(), get());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    El closure captura el **binding** `obj`, no una instantánea del objeto. El objeto mismo es mutado después de que el closure se crea — pero `obj` todavía apunta a la misma referencia. Ambas llamadas devuelven el mismo objeto mutado: `{ x: 99 }`.

    Esto muerde a personas que asumen que los closures "congelan" valores. No lo hacen. Capturan **referencias**.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={5}
  group="Avanzado"
  title="Stale Closure en React"
  options={[
    {text: 'Imprime el count actual en cada click'},
    {text: 'Siempre imprime 0', isAnswer: true},
    {text: 'Imprime NaN'},
    {text: "Esto no compila"},
  ]}
>
  <slot name="question">
  <div className="question">
    ¿Qué comportamiento muestra este componente de React cuando el botón se presiona múltiples veces?
    ```jsx
    function Counter() {
      const [count, setCount] = useState(0);

      useEffect(() => {
        const id = setInterval(() => {
          console.log('count is', count);
        }, 1000);
        return () => clearInterval(id);
      }, []); // empty deps

      return <button onClick={() => setCount(c => c + 1)}>
        Click ({count})
      </button>;
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Stale closure clásico. El `useEffect` se ejecuta una vez (array de deps vacío) y crea un intervalo que cierra sobre `count` como era en el momento del mount: `0`. No importa cuántas veces hagas click, el closure del intervalo nunca ve el valor actualizado.

    El label del botón se actualiza correctamente — eso es un render fresco. Pero `console.log` está atrapado en `0` para siempre.

    Solución: agregar `count` al array de deps (recrea el intervalo) o usar un ref:
    ```jsx
    const countRef = useRef(count);
    useEffect(() => { countRef.current = count; }, [count]);
    // luego usar countRef.current dentro del intervalo
    ```
  </div>
  </slot>
</Challenge>

  index={6}
  group="Avanzado"
  title="Patrón de Módulo"
  options={[
    {text: '60', isAnswer: true},
    {text: 'TypeError: balance is not defined'},
    {text: '-40'},
    {text: '100'},
  ]}
>
  <slot name="question">
  <div className="question">
    Espera — lee con cuidado. ¿Qué sucede realmente?
    ```javascript
    const bank = (() => {
      let balance = 0;
      return {
        deposit: (n) => (balance += n),
        withdraw: (n) => (balance -= n),
        getBalance: () => balance,
      };
    })();

    bank.deposit(100);
    bank.withdraw(40);
    console.log(bank.getBalance());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La respuesta es `60`.

    Este es el **Patrón de Módulo** — un IIFE que retorna un objeto de closures. Los tres métodos cierran sobre el mismo binding `balance`. Es genuinamente privado: no puedes acceder a `balance` desde fuera.

    ```javascript
    console.log(bank.balance); // undefined
    ```

    Antes de las clases, esta era *la* forma de encapsular estado en JavaScript. Todavía la verás en codebases legacy y librerías de utilidades.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={7}
  group="Experto"
  title="Memory Leak de Closure"
  options={[
    {text: 'Imprime la longitud del array data'},
    {text: 'Crea un memory leak', isAnswer: true},
    {text: 'Lanza un RangeError'},
    {text: 'Nada inusual — esto está bien'},
  ]}
>
  <slot name="question">
  <div className="question">
    ¿Cuál es el problema con este código?
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      element.addEventListener('click', () => {
        console.log('clicked!', data.length);
      });
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    El event listener cierra sobre `data`, así que el array permanece alcanzable mientras el listener sea alcanzable. Si `element` tiene larga vida y el listener nunca se remueve, ese gran array también se retiene.

    `element` está en el DOM. El listener sostiene `data`. GC no puede recolectarlo.

    Solución: no capturar datos grandes en listeners de larga vida a menos que realmente los necesites, y remover listeners cuando el UI que los posee se destruye:
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      // hacer algo con data...

      const onClick = () => {
        console.log('clicked!');
      };

      element.addEventListener('click', onClick);
      return () => element.removeEventListener('click', onClick);
    }
    ```

    V8 moderno es mejor en esto de lo que solía ser. Pero en SPAs de larga vida con muchos event listeners, este patrón todavía muerde a las personas.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={8}
  group="Experto"
  title="La Trampa del this"
  options={[
    {text: 'Dan, Dan'},
    {text: 'Dan, undefined'},
    {text: 'undefined, Dan'},
    {text: 'undefined, undefined', isAnswer: true},
  ]}
>
  <slot name="question">
  <div className="question">
    ¿Qué imprime esto?
    ```javascript
    const user = {
      name: 'Dan',
      greetArrow: () => console.log(this?.name),
      greetRegular() {
        const inner = () => console.log(this?.name);
        inner();
      },
    };

    user.greetArrow();
    const fn = user.greetRegular;
    fn();
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Dos trampas sutiles:

    **`greetArrow`**: Las arrow functions no tienen su propio `this`. Capturan `this` del ámbito léxico que las encierra — que aquí es el ámbito global/del módulo, no `user`. Así que `this?.name` es `undefined`.

    **`fn()`**: `greetRegular` se llama sin receptor (`fn()` no `user.greetRegular()`). En modo strict, `this` es `undefined`. En modo sloppy, es el objeto global. De cualquier manera, `this?.name` es `undefined`.

    Si hubieras llamado `user.greetRegular()` directamente, obtendrías `'Dan'` — porque entonces `this` está ligado a `user`.

    Resultado: `undefined, undefined`.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={9}
  group="Bonus"
  title="Memoización via Closure"
  options={[
    {text: 'Llama a expensive() 3 veces'},
    {text: 'Llama a expensive() 1 vez', isAnswer: true},
    {text: 'Llama a expensive() 2 veces'},
    {text: 'Lanza: Maximum call stack exceeded'},
  ]}
>
  <slot name="question">
  <div className="question">
    ¿Cuántas veces se llama realmente a `expensive()`?
    ```javascript
    function memoize(fn) {
      const cache = new Map();
      return (arg) => {
        if (cache.has(arg)) return cache.get(arg);
        const result = fn(arg);
        cache.set(arg, result);
        return result;
      };
    }

    let calls = 0;
    const expensive = (n) => { calls++; return n * 2; };
    const fast = memoize(expensive);

    fast(5);
    fast(5);
    fast(5);
    console.log(calls);
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La función memoizada cierra sobre `cache`. La primera llamada (`fast(5)`) falla en cache, llama a `expensive`, y almacena el resultado. Las segunda y tercera llamadas golpean el cache y retornan inmediatamente.

    `calls` es `1`.

    Esto es closures haciendo trabajo real. `cache` es privado (no puede accederse desde fuera de `memoize`), persistente (vive mientras `fast` exista), y compartido entre todas las llamadas a `fast`.
  </div>
  </slot>

¿Cómo te fue?

- **9-10**: Realmente entiendes closures. Probablemente da miedo programar en pair contigo.
- **6-8**: Solido. Las preguntas de stale closure / `this` te hicieron tropezar, lo cual es honesto.
- **3-5**: Conoces el concepto pero los bordes están borrosos. Ve a leer los docs de MDN sobre scope chain.
- **0-2**: Creías conocer closures. Ahora sí los conoces. Ese es el punto.

El stale closure en React (#6) y la trampa del `this` (#8) son las que causan bugs reales en producción. Si te parecieron desconocidos, valen la pena revisarlos — aparecen en code review más que cualquier otro tema relacionado con closures.
