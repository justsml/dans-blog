<p class="inset">Los closures son uno de esos conceptos que todo desarrollador de JavaScript *cree* entender, justo hasta el momento en que no es así. Este cuestionario va a trazar los límites de lo que realmente sabes.</p>

Los closures aparecen en manejadores de eventos, funciones factory, hooks de React, patrones de módulo y en prácticamente todo lo demás. No son exóticos. Son la trama de la que está tejido JavaScript.

Las preguntas empiezan siendo razonables y luego empiezan a quitar los tablones del suelo.

  <slot name="question">
  <div className="question">
¿Qué se registra en la consola después de 100 ms?
```javascript
    ```javascript
    for (var i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
`var` tiene alcance de función. Existe exactamente **una** `i` — los cinco callbacks cierran sobre la misma referencia. Cuando se ejecutan los timeouts, el bucle ya terminó y `i` vale `5`.

    Cinco callbacks. Una `i`. Valor: `5`. Se registra cinco veces.

    La solución es `let` (alcance de bloque, nueva referencia en cada iteración) o envolverlo en una IIFE para capturar el valor actual:
```javascript
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100); // 0, 1, 2, 3, 4 ✓
    }
    ```
```

>
  <slot name="question">
  <div className="question">
¿Qué imprime esto?
```javascript
    ```javascript
    function wrap() {
      const obj = { x: 1 };
      const getObj = () => obj;
      obj.x = 99;
      return getObj;
    }
    const get = wrap();
    console.log(get(), get());
    ```
```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
El closure captura el **enlace** `obj`, no una instantánea del objeto. El objeto en sí se muta después de crear el closure, pero `obj` sigue apuntando a la misma referencia. Ambas llamadas devuelven el mismo objeto mutado: `{ x: 99 }`.

    Esto suele pillar a quienes asumen que los closures "congelan" los valores. No lo hacen. Capturan **referencias**.

  <div className="question">
¿Qué comportamiento exhibe este componente de React al hacer clic en el botón varias veces?
```jsx
    ```jsx
    function Counter() {
      const [count, setCount] = useState(0);

      useEffect(() => {
        const id = setInterval(() => {
          console.log('count is', count);
        }, 1000);
        return () => clearInterval(id);
      }, []); // empty deps

      return <button onClick={() => setCount(c => c + 1)}>
        Click ({count})
      </button>;
    }
    ```
```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
Clausura obsoleta clásica. El `useEffect` se ejecuta una sola vez (array de dependencias vacío) y crea un intervalo que cierra sobre `count` tal como estaba en el montaje: `0`. No importa cuántas veces hagas clic, la clausura del intervalo nunca verá el valor actualizado.

    La etiqueta del botón se actualiza correctamente — eso es un renderizado nuevo. Pero `console.log` queda atrapado para siempre en `0`.

    Arréglalo agregando `count` al array de dependencias (recrea el intervalo) o usando una ref:
```jsx
    ```jsx
    const countRef = useRef(count);
    useEffect(() => { countRef.current = count; }, [count]);
    // then use countRef.current inside the interval
    ```
```

  </slot>

    const bank = (() => {

    ```javascript
    console.log(bank.balance); // undefined

>
  <slot name="question">
  <div className="question">
¿Cuál es el problema con este código?
```javascript
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      element.addEventListener('click', () => {
        console.log('clicked!', data.length);
      });
    }
    ```
```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
El *event listener* cierra sobre `data`, por lo que el array permanece accesible mientras el listener siga existiendo. Si `element` tiene un ciclo de vida largo y el listener nunca se elimina, ese array enorme también se queda retenido.

    `element` vive en el DOM. El listener mantiene viva `data`. El GC no puede liberarlo.

    Solución: no captures datos pesados en listeners de larga duración a menos que sea estrictamente necesario, y elimina los listeners cuando desmontes la interfaz que los usa:
```javascript
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      // do something with data...

      const onClick = () => {
        console.log('clicked!');
      };

      element.addEventListener('click', onClick);
      return () => element.removeEventListener('click', onClick);
    }
    ```
```
El motor V8 moderno gestiona esto mucho mejor que antes. Pero en SPAs de larga ejecución con muchos *event listeners*, este patrón sigue mordiendo a más de uno.

  </slot>
</Challenge>

  client:visible={{rootMargin: "150px"}}
  index={8}
  group="Experto"
  title="La trampa del `this`"
  options={[
    {text: 'Dan, Dan'},
    {text: 'Dan, undefined'},
    {text: 'undefined, Dan'},
    {text: 'undefined, undefined', isAnswer: true},
  ]}
>
  <slot name="question">
  <div className="question">
¿Qué imprime esto?
```javascript
    ```javascript
    const user = {
      name: 'Dan',
      greetArrow: () => console.log(this?.name),
      greetRegular() {
        const inner = () => console.log(this?.name);
        inner();
      },
    };

    user.greetArrow();
    const fn = user.greetRegular;
    fn();
    ```
```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
Dos trampas sutiles:

    **`greetArrow`**: Las funciones flecha no tienen su propio `this`. Capturan `this` del ámbito léxico circundante — que en este caso es el ámbito del módulo/global, no `user`. Por lo tanto, `this?.name` es `undefined`.

    **`fn()`**: `greetRegular` se invoca sin un receptor (`fn()` y no `user.greetRegular()`). En modo estricto, `this` es `undefined`. En modo no estricto, apunta al objeto global. En cualquier caso, `this?.name` es `undefined`.

    Si hubieras llamado a `user.greetRegular()` directamente, obtendrías `'Dan'` — porque entonces `this` estaría vinculado a `user`.

    Resultado: `undefined, undefined`.

  </slot>

  client:visible={{rootMargin: "150px"}}
  index={9}
  group="Bonus"
  title="Memoización mediante closure"
  options={[
    {text: 'Llama a `expensive()` 3 veces'},
    {text: 'Llama a `expensive()` 1 vez', isAnswer: true},
    {text: 'Llama a `expensive()` 2 veces'},
    {text: 'Lanza: Maximum call stack exceeded'},
  ]}
>
  <slot name="question">
  <div className="question">
¿Cuántas veces se llama realmente a `expensive()`?
```javascript
    ```javascript
    function memoize(fn) {
      const cache = new Map();
      return (arg) => {
        if (cache.has(arg)) return cache.get(arg);
        const result = fn(arg);
        cache.set(arg, result);
        return result;
      };
    }

    let calls = 0;
    const expensive = (n) => { calls++; return n * 2; };
    const fast = memoize(expensive);

    fast(5);
    fast(5);
    fast(5);
    console.log(calls);
    ```
```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
La función memoizada cierra sobre `cache`. La primera llamada (`fast(5)`) no encuentra el valor en caché, ejecuta `expensive` y almacena el resultado. La segunda y tercera llamadas aciertan en caché y devuelven el resultado de inmediato.

    `calls` es `1`.

    Así es como los closures hacen trabajo real. `cache` es privado (no se puede acceder desde fuera de `memoize`), persistente (vive mientras `fast` exista) y se comparte entre todas las llamadas a `fast`.
