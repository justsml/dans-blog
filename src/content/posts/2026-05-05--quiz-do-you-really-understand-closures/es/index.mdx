<p class="inset">Los cierres son uno de esos conceptos que cada desarrollador de JavaScript *piensa* que comprende — justo hasta que no. Este cuestionario encontrará los límites de lo que realmente sabes.</p>

Los cierres aparecen en los controladores de eventos, funciones de fábrica, React hooks, patrones de módulos y casi en cualquier otro lugar. No son exóticos. Son la tela de la que está tejido JavaScript.

Las preguntas empiezan de forma razonable y luego empiezan a quitar las tablas de los pisos.


    La palabra clave: *misma*. `count` no se copia en el cierre. El cierre mantiene una **referencia en vivo** a la variable de enlace. Esa es la única trama.

    ¿Qué se registra?
    ```javascript
    function makeCounter() {
      let count = 0;
      return () => ++count;
    }
    const a = makeCounter();
    const b = makeCounter();
    console.log(a(), b());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Cada llamada a `makeCounter()` crea un **contexto de ejecución** nuevo con su propio `count` fresco. `a` y `b` cierran sobre *diferentes* enlaces de `count`. No comparten estado.

    Resultado: `1, 1`

    Es por eso que los cierres se usan para construir estado privado — cada "instancia" es independiente.

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={2}
  group="La Trampa Clásica"
  title="var en un Bucle"
  options={[
    {text: '0, 1, 2, 3, 4'},
    {text: '5, 5, 5, 5, 5', isAnswer: true},
    {text: '1, 2, 3, 4, 5'},
    {text: 'undefined x5'},
  ]}
>
  <slot name="question">
  <div className="question">
    ¿Qué se registra después de 100ms?
    ```javascript
    for (var i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `var` tiene alcance de función. Hay **exactamente una** `i` — los cinco callbacks cierran sobre el mismo enlace. Para cuando los timeouts se disparan, el bucle ha terminado y `i` es `5`.

    Cinco callbacks. Una `i`. Valor: `5`. Se registra cinco veces.

    La solución es `let` (alcance de bloque, nuevo enlace en cada iteración) o envolver en una IIFE para capturar el valor actual:
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100); // 0, 1, 2, 3, 4 ✓
    }
    ```
  </div>
  </slot>
</Challenge>

>
  <slot name="question">
  <div className="question">
    El mismo bucle, palabra clave diferente. ¿Qué se registra?
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `let` en un bucle `for` es especial. La especificación dice: cada iteración obtiene su propio **nuevo enlace** de `i`. Es como si el bucle hiciera algo secreto:

    ```javascript
    { let i = 0; setTimeout(() => console.log(i), 100) }
    { let i = 1; setTimeout(() => console.log(i), 100) }
    // ...
    ```

    Así que los cinco callbacks cierran sobre cinco variables distintas. Resultado: `0, 1, 2, 3, 4`.
  </div>
  </slot>
</Challenge>

>
  <slot name="question">
  <div className="question">
    ¿Qué se registra?
    ```javascript
    function wrap() {
      const obj = { x: 1 };
      const getObj = () => obj;
      obj.x = 99;
      return getObj;
    }
    const get = wrap();
    console.log(get(), get());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    El cierre captura el **enlace** `obj`, no una instantánea del objeto. El objeto en sí se muta después de que se crea el cierre — pero `obj` todavía apunta a la misma referencia. Ambas llamadas devuelven el mismo objeto mutado: `{ x: 99 }`.

    Esto pega a la gente que asume que los cierres "congelan" los valores. No lo hacen. Capturan **referencias**.
  </div>
  </slot>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={5}
  group="Avanzado"
  title="Cierre Caducado en React"
  options={[
    {text: 'Registra el recuento actual en cada clic'},
    {text: 'Siempre registra 0', isAnswer: true},
    {text: 'Registra NaN'},
    {text: "Esto no compila"},
  ]}
>
  <slot name="question">
  <div className="question">
    ¿Qué comportamiento exhibe este componente React cuando se hace clic varias veces en el botón?
    ```jsx
    function Counter() {
      const [count, setCount] = useState(0);

      useEffect(() => {
        const id = setInterval(() => {
          console.log('count is', count);
        }, 1000);
        return () => clearInterval(id);
      }, []); // array de deps vacía

      return <button onClick={() => setCount(c => c + 1)}>
        Click ({count})
      </button>;
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Cierre caducado clásico. El `useEffect` se ejecuta una vez (array de deps vacío) y crea un intervalo que cierra sobre `count` como estaba en el momento del montaje: `0`. Sin importar cuántas veces hagas clic, el cierre del intervalo nunca ve el valor actualizado.

    La etiqueta del botón se actualiza correctamente — eso es un render fresco. Pero `console.log` está atrapado para siempre en `0`.

    Solucionalo agregando `count` al array de deps (recrea el intervalo) o usando un ref:
    ```jsx
    const countRef = useRef(count);
    useEffect(() => { countRef.current = count; }, [count]);
    // luego usa countRef.current dentro del intervalo
    ```
  </div>
  </slot>
</Challenge>

  options={[
    {text: '60', isAnswer: true},
    {text: 'TypeError: balance is not defined'},
    {text: '-40'},
    {text: '100'},
  ]}
>
  <slot name="question">
  <div className="question">
    Espera — lee con cuidado. ¿Qué realmente pasa?
    ```javascript
    const bank = (() => {
      let balance = 0;
      return {
        deposit: (n) => (balance += n),
        withdraw: (n) => (balance -= n),
        getBalance: () => balance,
      };
    })();

    bank.deposit(100);
    bank.withdraw(40);
    console.log(bank.getBalance());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La respuesta es `60`.

    Este es el **Patrón de Módulo** — una IIFE que devuelve un objeto de cierres. Los tres métodos cierran sobre el mismo enlace `balance`. Es genuinamente privado: no puedes acceder a `balance` desde fuera.

    ```javascript
    console.log(bank.balance); // undefined
    ```

    Antes de las clases, esta era *la* forma de encapsular estado en JavaScript. Aún la verás en códigobase legado y librerías de utilidad.
  </div>
  </slot>
</Challenge>

  client:visible={{rootMargin: "150px"}}
  index={7}
  group="Experto"
  title="Fuga de Memoria con Cierre"
  options={[
    {text: 'Registra la longitud del array de datos'},
    {text: 'Crea una fuga de memoria', isAnswer: true},
    {text: 'Lanza un RangeError'},
    {text: 'Nada inusual — esto está bien'},
  ]}
>
  <slot name="question">
  <div className="question">
    ¿Cuál es el problema con este código?
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      element.addEventListener('click', () => {
        console.log('clicked!', data.length);
      });
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    El controlador de eventos cierra sobre `data`, así que el array se mantiene accesible mientras el controlador esté accesible. Si `element` es de larga duración y el controlador nunca se elimina, ese array grande también se retiene.

    `element` está en el DOM. El controlador mantiene `data`. GC no puede recolectarlo.

    Solución: no captures datos grandes en controladores de eventos de larga duración a menos que realmente los necesites, y elimina los controladores cuando se destruye la interfaz de usuario que los posee:
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      // haz algo con data...

      const onClick = () => {
        console.log('clicked!');
      };

      element.addEventListener('click', onClick);
      return () => element.removeEventListener('click', onClick);
    }
    ```

    V8 moderno es mejor con esto que lo que solía ser. Pero en SPAs de larga duración con muchos controladores de eventos, este patrón sigue picando a la gente.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={8}
  group="Experto"
  title="La Trampa this"
  options={[
    {text: 'Dan, Dan'},
    {text: 'Dan, undefined'},
    {text: 'undefined, Dan'},
    {text: 'undefined, undefined', isAnswer: true},
  ]}
>
  <slot name="question">
  <div className="question">
    ¿Qué se registra?
    ```javascript
    const user = {
      name: 'Dan',
      greetArrow: () => console.log(this?.name),
      greetRegular() {
        const inner = () => console.log(this?.name);
        inner();
      },
    };

    user.greetArrow();
    const fn = user.greetRegular;
    fn();
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Dos trampas sutiles:

    **`greetArrow`**: Las funciones flecha no tienen su propio `this`. Capturan `this` desde el alcance léxico circundante — aquí el alcance de módulo/global, no `user`. Así que `this?.name` es `undefined`.

    **`fn()`**: `greetRegular` se llama sin receptor (`fn()` no `user.greetRegular()`). En modo estricto, `this` es `undefined`. En modo relajado, es el objeto global. En ambos casos, `this?.name` es `undefined`.

    Si hubieras llamado `user.greetRegular()` directamente, habrías recibido `'Dan'` — porque entonces `this` está vinculado a `user`.

    Resultado: `undefined, undefined`.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={9}
  group="Bonus"
  title="Memoización a través de Cierre"
  options={[
    {text: 'Llama a expensive() 3 veces'},
    {text: 'Llama a expensive() 1 vez', isAnswer: true},
    {text: 'Llama a expensive() 2 veces'},
    {text: 'Lanza: Maximum call stack exceeded'},
  ]}
>
  <slot name="question">
  <div className="question">
    ¿Cuántas veces se llama realmente `expensive()`?
    ```javascript
    function memoize(fn) {
      const cache = new Map();
      return (arg) => {
        if (cache.has(arg)) return cache.get(arg);
        const result = fn(arg);
        cache.set(arg, result);
        return result;
      };
    }

    let calls = 0;
    const expensive = (n) => { calls++; return n * 2; };
    const fast = memoize(expensive);

    fast(5);
    fast(5);
    fast(5);
    console.log(calls);
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La función memoizada cierra sobre `cache`. La primera llamada (`fast(5)`) falla en el caché, llama a `expensive` y almacena el resultado. Las segundas y terceras llamadas golpean el caché y devuelven inmediatamente.

    `calls` es `1`.

    Esto es cierres haciendo trabajo real. `cache` es privado (no se puede acceder desde fuera `memoize`), persistente (vive tanto como `fast` existe) y compartido entre todas las llamadas a `fast`.
  </div>
  </slot>
</Challenge>

¿Cómo lo hiciste?

- **9-10**: Realmente conoces cierres. Probablemente aterrador para programar en pareja.
- **6-8**: Sólido. Las preguntas de cierre caducado / `this` te confundieron, lo cual es honesto.
- **3-5**: Conoces el concepto pero los bordes son borrosos. Ve los docs de alcance de cadena MDN.
- **0-2**: Pensaste que conocías cierres. Ahora sí. Eso es el punto.

El cierre caducado en React (#6) y la trampa `this` (#8) son los que causan errores reales en producción. Si esas te parecieron desconocidas, vale la pena repasarlas — aparecen más en la revisión de código que cualquier otro problema relacionado con cierres.
