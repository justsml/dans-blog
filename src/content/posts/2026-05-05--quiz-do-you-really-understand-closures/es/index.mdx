<p class="inset">Los closures son uno de esos conceptos que todo desarrollador de JavaScript *cree* entender — hasta el momento en que no lo hace. Este quiz encontrará los bordes de lo que realmente sabes.</p>

Los closures aparecen en manejadores de eventos, factory functions, hooks de React, patrones de módulos y en casi todos los demás lugares. No son exóticos. Son el tejido del que está hecho JavaScript.

Las preguntas empiezan de forma razonable y luego comienzan a quitar las tablas del suelo.

    ¿Qué registra esto?
    ```javascript
    function makeCounter() {

    ```
  </div>
  </slot>

    ¿Qué registra esto?
    ```javascript
    function makeCounter() {
      let count = 0;
      return () => ++count;
    }
    const a = makeCounter();
    const b = makeCounter();
    console.log(a(), b());
    ```
  </div>

  <slot name='explanation'>
  <div className="explanation">
    Cada llamada a `makeCounter()` crea un **nuevo contexto de ejecución** con su propio `count` fresco. `a` y `b` hacen closure sobre enlaces `count` *diferentes*. No comparten estado.

    Resultado: `1, 1`

    Por eso se usan closures para construir estado privado: cada "instancia" es independiente.

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={2}
  group="La trampa clásica"
  title="var en un bucle"
  options={[
    {text: '0, 1, 2, 3, 4'},
    {text: '5, 5, 5, 5, 5', isAnswer: true},
    {text: '1, 2, 3, 4, 5'},
    {text: 'undefined x5'},
  ]}
>
  <slot name="question">
  <div className="question">
    ¿Qué se registra después de 100ms?
    ```javascript
    for (var i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `var` tiene alcance de función. Hay exactamente **un** `i` — los cinco callbacks hacen closure sobre el mismo enlace. Cuando los timeouts se ejecutan, el bucle ha terminado y `i` vale `5`.

    Cinco callbacks. Un `i`. Valor: `5`. Registrado cinco veces.

    La solución es `let` (alcance de bloque, nuevo enlace en cada iteración) o envolver en una IIFE para capturar el valor actual:
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100); // 0, 1, 2, 3, 4 ✓
    }
    ```
  </div>
  </slot>
</Challenge>

>
  <slot name="question">
  <div className="question">
    Mismo bucle, distinta palabra clave. ¿Qué se registra?
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `let` en un bucle `for` es especial. La especificación dice: cada iteración obtiene su propio **nuevo enlace** de `i`. Es como si el bucle hiciera esto en secreto:

    ```javascript
    { let i = 0; setTimeout(() => console.log(i), 100) }
    { let i = 1; setTimeout(() => console.log(i), 100) }
    // ...
    ```

    Así que los cinco callbacks hacen closure sobre cinco variables distintas. Resultado: `0, 1, 2, 3, 4`.
  </div>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={4}
  group="Closures y objetos"
  title="Mutación de objetos"
  options={[
    {text: '{ x: 1 }, { x: 1 }'},
    {text: '{ x: 1 }, { x: 99 }'},
    {text: '{ x: 99 }, { x: 99 }', isAnswer: true},
    {text: '{ x: 99 }, { x: 1 }'},
  ]}
>
  <slot name="question">
  <div className="question">
    ¿Qué registra esto?
    ```javascript
    function wrap() {
      const obj = { x: 1 };
      const getObj = () => obj;
      obj.x = 99;
      return getObj;
    }
    const get = wrap();
    console.log(get(), get());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    El closure captura el **enlace** `obj`, no una instantánea del objeto. El objeto se muta después de crear el closure, pero `obj` sigue apuntando a la misma referencia. Ambas llamadas devuelven el mismo objeto mutado: `{ x: 99 }`.

    Esto pilla a quienes asumen que los closures "congelan" valores. No lo hacen. Capturan **referencias**.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={5}
  group="Avanzado"
  title="Closure stale en React"
  options={[
    {text: 'Registra el count actual en cada clic'},
    {text: 'Siempre registra 0', isAnswer: true},
    {text: 'Registra NaN'},
    {text: "Esto no compila"},
  ]}
>
  <slot name="question">
  <div className="question">
    ¿Qué comportamiento muestra este componente de React al hacer clic en el botón varias veces?
    ```jsx
    function Counter() {
      const [count, setCount] = useState(0);

      useEffect(() => {
        const id = setInterval(() => {
          console.log('count is', count);
        }, 1000);
        return () => clearInterval(id);
      }, []); // empty deps

      return <button onClick={() => setCount(c => c + 1)}>
        Click ({count})
      </button>;
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Closure stale clásico. El `useEffect` se ejecuta una vez (array de dependencias vacío) y crea un intervalo que hace closure sobre `count` tal como estaba en el montaje: `0`. No importa cuántas veces hagas clic, el closure del intervalo nunca ve el valor actualizado.

    La etiqueta del botón se actualiza correctamente — eso es un render fresco. Pero `console.log` queda atrapado para siempre en `0`.

    Arréglalo añadiendo `count` al array de dependencias (recrea el intervalo) o usando una ref:
    ```jsx
    const countRef = useRef(count);
    useEffect(() => { countRef.current = count; }, [count]);
    // luego usa countRef.current dentro del intervalo
    ```
  </div>
  </slot>
</Challenge>

  group="Avanzado"
  title="Patrón módulo"
  options={[
    {text: '60', isAnswer: true},
    {text: 'TypeError: balance is not defined'},
    {text: '-40'},
    {text: '100'},
  ]}
>
  <slot name="question">
  <div className="question">
    Espera — lee con atención. ¿Qué ocurre realmente?
    ```javascript
    const bank = (() => {
      let balance = 0;
      return {
        deposit: (n) => (balance += n),
        withdraw: (n) => (balance -= n),
        getBalance: () => balance,
      };
    })();

    bank.deposit(100);
    bank.withdraw(40);
    console.log(bank.getBalance());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La respuesta es `60`.

    Este es el **Patrón Módulo** — una IIFE que devuelve un objeto de closures. Los tres métodos hacen closure sobre el mismo enlace `balance`. Es genuinamente privado: no puedes acceder a `balance` desde fuera.

    ```javascript
    console.log(bank.balance); // undefined
    ```

    Antes de las clases, esta era *la* forma de encapsular estado en JavaScript. Todavía lo verás en codebases legacy y bibliotecas de utilidades.
  </div>
  </slot>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={7}
  group="Experto"
  title="Fuga de memoria por closure"
  options={[
    {text: 'Registra la longitud del array de datos'},
    {text: 'Crea una fuga de memoria', isAnswer: true},
    {text: 'Lanza un RangeError'},
    {text: 'Nada inusual — esto está bien'},
  ]}
>
  <slot name="question">
  <div className="question">
    ¿Cuál es el problema con este código?
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      element.addEventListener('click', () => {
        console.log('clicked!', data.length);
      });
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    El listener de eventos hace closure sobre `data`, así que el array permanece accesible mientras el listener sea accesible. Si `element` es de larga vida y el listener nunca se elimina, ese gran array también se mantiene.

    `element` está en el DOM. El listener retiene `data`. El GC no puede recogerlo.

    Solución: no captures datos grandes en listeners de larga vida a menos que realmente los necesites, y elimina los listeners cuando se destruya la UI propietaria:
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      // haz algo con data...

      const onClick = () => {
        console.log('clicked!');
      };

      element.addEventListener('click', onClick);
      return () => element.removeEventListener('click', onClick);
    }
    ```

    V8 moderno es mejor en esto que antes. Pero en SPAs de larga vida con muchos listeners de eventos, este patrón sigue mordiendo a la gente.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={8}
  group="Experto"
  title="La trampa de this"
  options={[
    {text: 'Dan, Dan'},
    {text: 'Dan, undefined'},
    {text: 'undefined, Dan'},
    {text: 'undefined, undefined', isAnswer: true},
  ]}
>
  <slot name="question">
  <div className="question">
    ¿Qué registra esto?
    ```javascript
    const user = {
      name: 'Dan',
      greetArrow: () => console.log(this?.name),
      greetRegular() {
        const inner = () => console.log(this?.name);
        inner();
      },
    };

    user.greetArrow();
    const fn = user.greetRegular;
    fn();
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Dos trampas sutiles:

    **`greetArrow`**: Las arrow functions no tienen su propio `this`. Capturan `this` del ámbito léxico envolvente — que aquí es el ámbito módulo/global, no `user`. Así que `this?.name` es `undefined`.

    **`fn()`**: `greetRegular` se llama sin receptor (`fn()` no `user.greetRegular()`). En modo estricto, `this` es `undefined`. En modo sloppy, es el objeto global. De cualquier forma, `this?.name` es `undefined`.

    Si hubieras llamado `user.greetRegular()` directamente, obtendrías `'Dan'` — porque entonces `this` se enlaza a `user`.

    Resultado: `undefined, undefined`.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={9}
  group="Bonus"
  title="Memoización vía closure"
  options={[
    {text: 'Llama a expensive() 3 veces'},
    {text: 'Llama a expensive() 1 vez', isAnswer: true},
    {text: 'Llama a expensive() 2 veces'},
    {text: 'Lanza: Maximum call stack exceeded'},
  ]}
>
  <slot name="question">
  <div className="question">
    ¿Cuántas veces se llama realmente a `expensive()`?
    ```javascript
    function memoize(fn) {
      const cache = new Map();
      return (arg) => {
        if (cache.has(arg)) return cache.get(arg);
        const result = fn(arg);
        cache.set(arg, result);
        return result;
      };
    }

    let calls = 0;
    const expensive = (n) => { calls++; return n * 2; };
    const fast = memoize(expensive);

    fast(5);
    fast(5);
    fast(5);
    console.log(calls);
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La función memoizada hace closure sobre `cache`. La primera llamada (`fast(5)`) falla en el cache, llama a `expensive` y almacena el resultado. La segunda y tercera llamadas aciertan en el cache y retornan inmediatamente.

    `calls` es `1`.

    Aquí los closures hacen trabajo real. `cache` es privado (no se puede acceder desde fuera de `memoize`), persistente (vive mientras `fast` exista) y compartido entre todas las llamadas a `fast`.
  </div>
  </slot>

¿Qué tal lo hiciste?

- **9-10**: Realmente entiendes los closures. Probablemente da miedo hacer pair-programming contigo.
- **6-8**: Sólido. Las preguntas de closure stale / `this` te pillaron, lo cual es honesto.
- **3-5**: Conoces el concepto pero los bordes son borrosos. Ve a leer la documentación de cadena de alcances de MDN.
- **0-2**: Creías que entendías los closures. Ahora sí. Ese es el punto.

El closure stale en React (#6) y la trampa de `this` (#8) son los que causan bugs reales en producción. Si te resultaron desconocidos, vale la pena revisarlos — aparecen en code review más que cualquier otro tema relacionado con closures.
