<p class="inset">Le closure sono uno di quei concetti che ogni sviluppatore JavaScript *pensa* di capire — giusto fino al momento in cui non lo fa. Questo quiz troverà i limiti di ciò che realmente sai.</p>

Le closure appaiono negli event handler, nelle factory function, nelle React hooks, nei pattern di modulo e praticamente ovunque. Non sono esotiche. Sono il tessuto stesso con cui JavaScript è intrecciato.

Le domande partono ragionevoli e poi iniziano a rimuovere il pavimento.

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={2}
  group="La Trappola Classica"
  title="var in un Loop"
  options={[
    {text: '0, 1, 2, 3, 4'},
    {text: '5, 5, 5, 5, 5', isAnswer: true},
    {text: '1, 2, 3, 4, 5'},
    {text: 'undefined x5'},
  ]}
>
  <slot name="question">
  <div className="question">
    Cosa viene stampato dopo 100ms?
    ```javascript
    for (var i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `var` è function-scoped. C'è esattamente **uno** `i` — tutti e cinque i callback chiudono sullo stesso legame. Quando i timeout scattano, il loop è finito e `i` è `5`.

    Cinque callback. Un `i`. Valore: `5`. Stampato cinque volte.

    La soluzione è `let` (block-scoped, nuovo legame ogni iterazione) o avvolgere in un IIFE per catturare il valore corrente:
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100); // 0, 1, 2, 3, 4 ✓
    }
    ```
  </div>
  </slot>
</Challenge>

    {text: '4, 4, 4, 4, 4'},

  ]}
>
  <slot name="question">
  <div className="question">
    Stesso loop, parola chiave diversa. Cosa stampa?
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `let` in un loop `for` è speciale. Lo spec dice: ogni iterazione ottiene il proprio **nuovo legame** di `i`. È come se il loop segretamente facesse questo:

    ```javascript
    { let i = 0; setTimeout(() => console.log(i), 100) }
    { let i = 1; setTimeout(() => console.log(i), 100) }
    // ...
    ```

    Quindi i cinque callback chiudono su cinque variabili distinte. Risultato: `0, 1, 2, 3, 4`.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={4}
  group="Closure & Oggetti"
  title="Mutazione Oggetto"
  options={[
    {text: '{ x: 1 }, { x: 1 }'},
    {text: '{ x: 1 }, { x: 99 }'},
    {text: '{ x: 99 }, { x: 99 }', isAnswer: true},

  ]}
>
  <slot name="question">
  <div className="question">
    Cosa stampa questo?
    ```javascript
    function wrap() {
      const obj = { x: 1 };
      const getObj = () => obj;
      obj.x = 99;
      return getObj;
    }
    const get = wrap();
    console.log(get(), get());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La closure cattura il **legame** `obj`, non uno snapshot dell'oggetto. L'oggetto stesso viene mutato dopo che la closure è stata creata — ma `obj` punta ancora alla stessa reference. Entrambe le chiamate ritornano lo stesso oggetto mutato: `{ x: 99 }`.

    Questo morde persone che assumono che le closure "congelino" i valori. Non lo fanno. Catturano **riferimenti**.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={5}
  group="Avanzato"
  title="Stale Closure in React"
  options={[
    {text: 'Stampa il count corrente ad ogni click'},
    {text: 'Stampa sempre 0', isAnswer: true},
    {text: 'Stampa NaN'},
    {text: "Questo non compilerà"},
  ]}
>
  <slot name="question">
  <div className="question">
    Che comportamento mostra questo componente React quando il bottone viene cliccato più volte?
    ```jsx
    function Counter() {
      const [count, setCount] = useState(0);

      useEffect(() => {
        const id = setInterval(() => {
          console.log('count è', count);
        }, 1000);
        return () => clearInterval(id);
      }, []); // deps vuoto

      return <button onClick={() => setCount(c => c + 1)}>
        Click ({count})
      </button>;
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Classica stale closure. L'`useEffect` gira una volta (array deps vuoto) e crea un intervallo che chiude su `count` come era al mount: `0`. Non importa quante volte clicchi, la closure dell'intervallo non vede mai il valore aggiornato.

    L'etichetta del bottone si aggiorna correttamente — è un render fresco. Ma `console.log` è per sempre intrappolato a `0`.

    Risolvilo aggiungendo `count` all'array deps (ricrea l'intervallo) o usando un ref:
    ```jsx
    const countRef = useRef(count);
    useEffect(() => { countRef.current = count; }, [count]);
    // poi usa countRef.current dentro l'intervallo
    ```
  </div>
  </slot>
</Challenge>

  group="Avanzato"
  title="Pattern Modulo"
  options={[
    {text: '60', isAnswer: true},
    {text: 'TypeError: balance non è definito'},
    {text: '-40'},
    {text: '100'},
  ]}
>
  <slot name="question">
  <div className="question">
    Aspetta — leggi attentamente. Cosa succede davvero?
    ```javascript
    const bank = (() => {
      let balance = 0;
      return {
        deposit: (n) => (balance += n),
        withdraw: (n) => (balance -= n),
        getBalance: () => balance,
      };
    })();

    bank.deposit(100);
    bank.withdraw(40);
    console.log(bank.getBalance());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La risposta è `60`.

    Questo è il **Pattern Modulo** — un IIFE che ritorna un oggetto di closure. Tutti e tre i metodi chiudono sullo stesso legame `balance`. È genuinamente privato: non puoi accedere a `balance` dall'esterno.

    ```javascript
    console.log(bank.balance); // undefined
    ```

    Prima delle class, questo era *il* modo per incapsulare stato in JavaScript. Lo vedrai ancora in codebase legacy e librerie utility.
  </div>
  </slot>
</Challenge>

  client:visible={{rootMargin: "150px"}}
  index={7}
  group="Esperto"
  title="Memory Leak da Closure"
  options={[
    {text: 'Stampa la lunghezza dell\'array data'},
    {text: 'Crea un memory leak', isAnswer: true},
    {text: 'Lancia un RangeError'},
    {text: 'Niente di insolito — questo va bene'},
  ]}
>
  <slot name="question">
  <div className="question">
    Qual è il problema con questo codice?
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      element.addEventListener('click', () => {
        console.log('cliccato!', data.length);
      });
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    L'event listener chiude su `data`, quindi l'array rimane raggiungibile finché l'listener è raggiungibile. Se `element` è a vita lunga e l'listener non viene mai rimosso, quell'array grande viene mantenuto comunque.

    `element` è nel DOM. L'listener tiene `data`. GC non può raccoglierlo.

    Risolvi: non catturare dati grandi in listener a vita lunga a meno che non ti servano davvero, e rimuovi listener quando l'UI che li possiede viene distrutta:
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      // fai qualcosa con data...

      const onClick = () => {
        console.log('cliccato!');
      };

      element.addEventListener('click', onClick);
      return () => element.removeEventListener('click', onClick);
    }
    ```

    I moderni V8 sono migliori di quanto fossero una volta. Ma in SPA a vita lunga con molti event listener, questo pattern morde ancora le persone.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={8}
  group="Esperto"
  title="La Trappola this"
  options={[
    {text: 'Dan, Dan'},
    {text: 'Dan, undefined'},
    {text: 'undefined, Dan'},
    {text: 'undefined, undefined', isAnswer: true},
  ]}
>
  <slot name="question">
  <div className="question">
    Cosa stampa questo?
    ```javascript
    const user = {
      name: 'Dan',
      greetArrow: () => console.log(this?.name),
      greetRegular() {
        const inner = () => console.log(this?.name);
        inner();
      },
    };

    user.greetArrow();
    const fn = user.greetRegular;
    fn();
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Due trappole sottili:

    **`greetArrow`**: Le funzioni arrow non hanno il proprio `this`. Catturano `this` dal scope lessicale circostante — che qui è lo scope del modulo/global, non `user`. Quindi `this?.name` è `undefined`.

    **`fn()`**: `greetRegular` viene chiamata senza receiver (`fn()` non `user.greetRegular()`). In strict mode, `this` è `undefined`. In sloppy mode, è l'oggetto global. Comunque, `this?.name` è `undefined`.

    Se avessi chiamato `user.greetRegular()` direttamente, avresti ottenuto `'Dan'` — perché allora `this` è legato a `user`.

    Risultato: `undefined, undefined`.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={9}
  group="Bonus"
  title="Memoization via Closure"
  options={[
    {text: 'Chiama expensive() 3 volte'},
    {text: 'Chiama expensive() 1 volta', isAnswer: true},
    {text: 'Chiama expensive() 2 volte'},
    {text: 'Lancia: Maximum call stack exceeded'},
  ]}
>
  <slot name="question">
  <div className="question">
    Quante volte viene davvero chiamata `expensive()`?
    ```javascript
    function memoize(fn) {
      const cache = new Map();
      return (arg) => {
        if (cache.has(arg)) return cache.get(arg);
        const result = fn(arg);
        cache.set(arg, result);
        return result;
      };
    }

    let calls = 0;
    const expensive = (n) => { calls++; return n * 2; };
    const fast = memoize(expensive);

    fast(5);
    fast(5);
    fast(5);
    console.log(calls);
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La funzione memoizzata chiude su `cache`. La prima chiamata (`fast(5)`) manca la cache, chiama `expensive`, e memorizza il risultato. La seconda e terza chiamata colpiscono la cache e ritornano immediatamente.

    `calls` è `1`.

    Questo sono closure che fanno lavoro reale. `cache` è privato (non può essere acceso da fuori `memoize`), persistente (vive finché `fast` esiste), e condiviso tra tutte le chiamate a `fast`.
  </div>
  </slot>

---

Com'è andata?

- **9-10**: Conosci davvero le closure. Probabilmente spaventoso da fare pair-programming con.
- **6-8**: Solido. Le domande stale closure / `this` ti hanno preso, che è onesto.
- **3-5**: Conosci il concetto ma i bordi sono sfocati. Vai a leggere i docs MDN sulla scope chain.
- **0-2**: Pensavi di conoscere le closure. Ora le conosci. Questo è il punto.

La stale closure in React (#6) e la trappola `this` (#8) sono quelle che causano veri bug di produzione. Se quelle ti sono sembrate familiari, vale la pena rivederle — appaiono in code review più di qualsiasi altro issue legato alle closure.
