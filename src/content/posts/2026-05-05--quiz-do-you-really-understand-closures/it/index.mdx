<p class="inset">Le closure sono uno di quei concetti che ogni sviluppatore JavaScript *pensa* di capire — fino al momento in cui scopre di no. Questo quiz troverà i limiti di ciò che sai davvero.</p>

Le closure si trovano negli event handler, nelle factory function, negli hook di React, nei pattern modulari e praticamente ovunque. Non sono esotiche. Sono il tessuto di cui è fatto JavaScript.

Le domande iniziano in modo ragionevole e poi iniziano a rimuovere il pavimento.

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={2}
  group="La Trappola Classica"
  title="var in un Ciclo"
  options={[
    {text: '0, 1, 2, 3, 4'},
    {text: '5, 5, 5, 5, 5', isAnswer: true},
    {text: '1, 2, 3, 4, 5'},
    {text: 'undefined x5'},
  ]}
>
  <slot name="question">
  <div className="question">
    Cosa viene loggato dopo 100ms?
    ```javascript
    for (var i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `var` ha scope di funzione. C'è esattamente **un** `i` — tutte e cinque le callback creano una closure sullo stesso binding. Quando i timeout si attivano, il ciclo è terminato e `i` vale `5`.

    Cinque callback. Un `i`. Valore: `5`. Loggato cinque volte.

    La soluzione è `let` (scope di blocco, un nuovo binding per ogni iterazione) o racchiudere in un IIFE per catturare il valore corrente:
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100); // 0, 1, 2, 3, 4 ✓
    }
    ```
  </div>
  </slot>
</Challenge>

  options={[
    {text: '0, 1, 2, 3, 4', isAnswer: true},

>
  <slot name="question">
  <div className="question">
    Stesso ciclo, keyword diversa. Cosa viene loggato?
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `let` in un ciclo `for` è speciale. La specifica dice: ogni iterazione ottiene il proprio **nuovo binding** di `i`. È come se il ciclo facesse segretamente così:

    ```javascript
    { let i = 0; setTimeout(() => console.log(i), 100) }
    { let i = 1; setTimeout(() => console.log(i), 100) }
    // ...
    ```

    Quindi le cinque callback creano closure su cinque variabili distinte. Risultato: `0, 1, 2, 3, 4`.
  </div>
  </slot>
</Challenge>

  title="Mutazione di Oggetti"
  options={[
    {text: '{ x: 1 }, { x: 1 }'},
    {text: '{ x: 1 }, { x: 99 }'},

>
  <slot name="question">
  <div className="question">
    Cosa viene loggato?
    ```javascript
    function wrap() {
      const obj = { x: 1 };
      const getObj = () => obj;
      obj.x = 99;
      return getObj;
    }
    const get = wrap();
    console.log(get(), get());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La closure cattura il **binding** `obj`, non un'istantanea dell'oggetto. L'oggetto stesso viene mutato dopo la creazione della closure — ma `obj` punta ancora allo stesso riferimento. Entrambe le chiamate restituiscono lo stesso oggetto mutato: `{ x: 99 }`.

    Questo inganna chi presume che le closure "congelino" i valori. Non lo fanno. Catturano **riferimenti**.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={5}
  group="Avanzato"
  title="Stale Closure in React"
  options={[
    {text: 'Logga il count corrente a ogni click'},
    {text: 'Logga sempre 0', isAnswer: true},
    {text: 'Logga NaN'},
    {text: "Non compila"},
  ]}
>
  <slot name="question">
  <div className="question">
    Che comportamento ha questo componente React quando il pulsante viene cliccato più volte?
    ```jsx
    function Counter() {
      const [count, setCount] = useState(0);

      useEffect(() => {
        const id = setInterval(() => {
          console.log('count is', count);
        }, 1000);
        return () => clearInterval(id);
      }, []); // empty deps

      return <button onClick={() => setCount(c => c + 1)}>
        Click ({count})
      </button>;
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Classica stale closure. `useEffect` viene eseguito una volta (array di dipendenze vuoto) e crea un intervallo che cattura `count` com'era al momento del mount: `0`. Non importa quante volte clicchi, la closure dell'intervallo non vede mai il valore aggiornato.

    L'etichetta del pulsante si aggiorna correttamente — quello è un nuovo render. Ma il `console.log` è intrappolato per sempre a `0`.

    Soluzione: aggiungere `count` all'array di dipendenze (ricrea l'intervallo) o usare un ref:
    ```jsx
    const countRef = useRef(count);
    useEffect(() => { countRef.current = count; }, [count]);
    // poi usa countRef.current dentro l'intervallo
    ```
  </div>
  </slot>
</Challenge>

  group="Avanzato"
  title="Pattern Modulare"
  options={[
    {text: '60', isAnswer: true},
    {text: 'TypeError: balance is not defined'},
    {text: '-40'},
    {text: '100'},
  ]}
>
  <slot name="question">
  <div className="question">
    Aspetta — leggi attentamente. Cosa succede realmente?
    ```javascript
    const bank = (() => {
      let balance = 0;
      return {
        deposit: (n) => (balance += n),
        withdraw: (n) => (balance -= n),
        getBalance: () => balance,
      };
    })();

    bank.deposit(100);
    bank.withdraw(40);
    console.log(bank.getBalance());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La risposta è `60`.

    Questo è il **Module Pattern** — un IIFE che restituisce un oggetto di closure. Tutti e tre i metodi catturano lo stesso binding di `balance`. È genuinamente privato: non puoi accedere a `balance` dall'esterno.

    ```javascript
    console.log(bank.balance); // undefined
    ```

    Prima delle classi, questo era *il* modo per incapsulare lo stato in JavaScript. Lo vedrai ancora in codebase legacy e librerie di utilità.
  </div>
  </slot>
</Challenge>

  client:visible={{rootMargin: "150px"}}
  index={7}
  group="Esperto"
  title="Memory Leak da Closure"
  options={[
    {text: 'Logga la lunghezza dell\'array data'},
    {text: 'Crea un memory leak', isAnswer: true},
    {text: 'Lancia un RangeError'},
    {text: 'Nulla di insolito — va bene così'},
  ]}
>
  <slot name="question">
  <div className="question">
    Qual è il problema con questo codice?
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      element.addEventListener('click', () => {
        console.log('clicked!', data.length);
      });
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    L'event listener cattura `data` con una closure, quindi l'array rimane raggiungibile finché il listener è raggiungibile. Se `element` è longevo e il listener non viene mai rimosso, anche quel grande array viene trattenuto.

    `element` è nel DOM. Il listener trattiene `data`. Il GC non può raccoglierlo.

    Soluzione: non catturare grandi dati in listener longevi a meno che non sia davvero necessario, e rimuovi i listener quando la UI che li possiede viene smantellata:
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      // fai qualcosa con data...

      const onClick = () => {
        console.log('clicked!');
      };

      element.addEventListener('click', onClick);
      return () => element.removeEventListener('click', onClick);
    }
    ```

    V8 moderno è migliorato rispetto al passato. Ma nelle SPA longeve con molti event listener, questo pattern morde ancora.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={8}
  group="Esperto"
  title="La Trappola di this"
  options={[
    {text: 'Dan, Dan'},
    {text: 'Dan, undefined'},
    {text: 'undefined, Dan'},
    {text: 'undefined, undefined', isAnswer: true},
  ]}
>
  <slot name="question">
  <div className="question">
    Cosa viene loggato?
    ```javascript
    const user = {
      name: 'Dan',
      greetArrow: () => console.log(this?.name),
      greetRegular() {
        const inner = () => console.log(this?.name);
        inner();
      },
    };

    user.greetArrow();
    const fn = user.greetRegular;
    fn();
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Due trappole sottili:

    **`greetArrow`**: Le arrow function non hanno un proprio `this`. Catturano `this` dallo scope lessicale circostante — che qui è lo scope del modulo/globale, non `user`. Quindi `this?.name` è `undefined`.

    **`fn()`**: `greetRegular` viene chiamata senza un ricevente (`fn()` non `user.greetRegular()`). In strict mode, `this` è `undefined`. In sloppy mode, è l'oggetto globale. In ogni caso, `this?.name` è `undefined`.

    Se avessi chiamato `user.greetRegular()` direttamente, avresti ottenuto `'Dan'` — perché allora `this` è legato a `user`.

    Risultato: `undefined, undefined`.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={9}
  group="Bonus"
  title="Memoizzazione tramite Closure"
  options={[
    {text: 'Chiama expensive() 3 volte'},
    {text: 'Chiama expensive() 1 volta', isAnswer: true},
    {text: 'Chiama expensive() 2 volte'},
    {text: 'Lancia: Maximum call stack exceeded'},
  ]}
>
  <slot name="question">
  <div className="question">
    Quante volte viene effettivamente chiamata `expensive()`?
    ```javascript
    function memoize(fn) {
      const cache = new Map();
      return (arg) => {
        if (cache.has(arg)) return cache.get(arg);
        const result = fn(arg);
        cache.set(arg, result);
        return result;
      };
    }

    let calls = 0;
    const expensive = (n) => { calls++; return n * 2; };
    const fast = memoize(expensive);

    fast(5);
    fast(5);
    fast(5);
    console.log(calls);
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La funzione memoizzata cattura `cache` con una closure. La prima chiamata (`fast(5)`) non trova il valore in cache, chiama `expensive` e memorizza il risultato. La seconda e terza chiamata trovano il valore in cache e restituiscono immediatamente.

    `calls` vale `1`.

    Questo sono le closure al lavoro. `cache` è privata (non accessibile dall'esterno di `memoize`), persistente (vive finché `fast` esiste), e condivisa tra tutte le chiamate a `fast`.
  </div>
  </slot>

Com'è andata?

- **9-10**: Conosci davvero le closure. Probabilmente fai paura nel pair-programming.
- **6-8**: Solido. Le domande sulla stale closure e su `this` ti hanno inciampato, il che è onesto.
- **3-5**: Conosci il concetto ma i bordi sono sfocati. Vai a leggerti la documentazione MDN sulla scope chain.
- **0-2**: Pensavi di conoscere le closure. Ora è così. Questo è il punto.

La stale closure in React (#6) e la trappola di `this` (#8) sono quelle che causano veri bug in produzione. Se ti sono sembrate poco familiari, vale la pena rivederle — appaiono in code review più di qualsiasi altro problema legato alle closure.
