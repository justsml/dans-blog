<p class="inset">Closures sind eines dieser Konzepte, von denen jeder JavaScript-Entwickler *glaubt*, sie zu verstehen — bis zu dem Moment, in dem er es eben nicht tut. Dieses Quiz findet die Grenzen von dem, was du wirklich weißt.</p>

Closures tauchen in Event-Handlern, Fabriksfunktionen, React Hooks, Modul-Mustern und praktisch überall sonst auf. Sie sind nicht exotisch. Sie sind der Stoff, aus dem JavaScript gemacht ist.

Die Fragen starten harmlos und werden dann zunehmend haarig.

    Was wird geloggt?
    ```javascript
    function makeCounter() {
      let count = 0;
      return () => ++count;
    }

    const b = makeCounter();
    console.log(a(), b());
    ```
  </div>

  <slot name='explanation'>
  <div className="explanation">
    Jeder Aufruf von `makeCounter()` erzeugt einen **neuen Ausführungskontext** mit seinem eigenen frischen `count`. `a` und `b` schließen über *unterschiedliche* `count`-Bindungen. Sie teilen keinen Zustand.

    Ergebnis: `1, 1`

    Das ist der Grund, warum Closures verwendet werden, um privaten Zustand aufzubauen — jede "Instanz" ist unabhängig.

  client:visible={{rootMargin: "150px"}}
  index={2}
  group="Die klassische Falle"
  title="var in einer Schleife"
  options={[
    {text: '0, 1, 2, 3, 4'},
    {text: '5, 5, 5, 5, 5', isAnswer: true},
    {text: '1, 2, 3, 4, 5'},
    {text: 'undefined x5'},
  ]}
>
  <slot name="question">
  <div className="question">
    Was wird nach 100ms geloggt?
    ```javascript
    for (var i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `var` ist funktions-scoped. Es gibt genau **eine** `i` — alle fünf Callbacks schließen über dieselbe Bindung. Wenn die Timeouts feuern, ist die Schleife längst fertig und `i` ist `5`.

    Fünf Callbacks. Eine `i`. Wert: `5`. Fünf Mal geloggt.

    Die Lösung ist `let` (block-scoped, neue Bindung pro Iteration) oder ein IIFE, das den aktuellen Wert einfängt:
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100); // 0, 1, 2, 3, 4 ✓
    }
    ```
  </div>
  </slot>
</Challenge>

    ```javascript
    { let i = 0; setTimeout(() => console.log(i), 100) }
    { let i = 1; setTimeout(() => console.log(i), 100) }
    // ...
    ```

    Also schließen die fünf Callbacks über fünf verschiedene Variablen. Ergebnis: `0, 1, 2, 3, 4`.

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={4}
  group="Closures & Objekte"
  title="Objekt-Mutation"
  options={[
    {text: '{ x: 1 }, { x: 1 }'},
    {text: '{ x: 1 }, { x: 99 }'},
    {text: '{ x: 99 }, { x: 99 }', isAnswer: true},
    {text: '{ x: 99 }, { x: 1 }'},
  ]}
>
  <slot name="question">
  <div className="question">
    Was wird geloggt?
    ```javascript
    function wrap() {
      const obj = { x: 1 };
      const getObj = () => obj;
      obj.x = 99;
      return getObj;
    }
    const get = wrap();
    console.log(get(), get());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Die Closure erfasst die **Bindung** `obj`, nicht eine Momentaufnahme des Objekts. Das Objekt selbst wird mutiert, nachdem die Closure erstellt wurde — aber `obj` zeigt immer noch auf dieselbe Referenz. Beide Aufrufe geben dasselbe mutierte Objekt zurück: `{ x: 99 }`.

    Das erwischt Leute, die annehmen, Closures "einfrieren" Werte. Tun sie nicht. Sie erfassen **Referenzen**.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={5}
  group="Fortgeschritten"
  title="Stale Closure in React"
  options={[
    {text: 'Loggt den aktuellen Count bei jedem Klick'},
    {text: 'Loggt immer 0', isAnswer: true},
    {text: 'Loggt NaN'},
    {text: "Das kompiliert nicht"},
  ]}
>
  <slot name="question">
  <div className="question">
    Welches Verhalten zeigt diese React-Komponente, wenn der Button mehrfach geklickt wird?
    ```jsx
    function Counter() {
      const [count, setCount] = useState(0);

      useEffect(() => {
        const id = setInterval(() => {
          console.log('count is', count);
        }, 1000);
        return () => clearInterval(id);
      }, []); // leere deps

      return <button onClick={() => setCount(c => c + 1)}>
        Click ({count})
      </button>;
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Klassische stale Closure. Der `useEffect` läuft einmal (leerer Dep-Array) und erstellt ein Intervall, das über `count` schließt, wie es zum Mount-Zeitpunkt war: `0`. Egal wie oft du klickst, das Intervall in der Closure sieht den aktualisierten Wert nie.

    Das Button-Label aktualisiert sich korrekt — das ist ein neuer Render. Aber `console.log` ist für immer bei `0` gefangen.

    Lösung: `count` zum Dep-Array hinzufügen (erstellt das Intervall neu) oder einen Ref verwenden:
    ```jsx
    const countRef = useRef(count);
    useEffect(() => { countRef.current = count; }, [count]);
    // dann countRef.current im Intervall verwenden
    ```
  </div>
  </slot>
</Challenge>

  title="Module Pattern"
  options={[

    {text: 'TypeError: balance is not defined'},
    {text: '-40'},
    {text: '100'},
  ]}
>
  <slot name="question">
  <div className="question">
    Moment — genau lesen. Was passiert wirklich?
    ```javascript
    const bank = (() => {
      let balance = 0;
      return {
        deposit: (n) => (balance += n),
        withdraw: (n) => (balance -= n),
        getBalance: () => balance,
      };
    })();

    bank.deposit(100);
    bank.withdraw(40);
    console.log(bank.getBalance());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Die Antwort ist `60`.

    Das ist das **Module Pattern** — eine IIFE, die ein Objekt aus Closures zurückgibt. Alle drei Methoden schließen über dieselbe `balance`-Bindung. Es ist wirklich privat: du kannst `balance` von außen nicht zugreifen.

    ```javascript
    console.log(bank.balance); // undefined
    ```

    Bevor es Klassen gab, war das *der* Weg, Zustand in JavaScript zu kapseln. Du wirst es noch in Legacy-Codebasen und Utility-Bibliotheken sehen.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={7}
  group="Experte"
  title="Closure Memory Leak"
  options={[
    {text: 'Es loggt die Länge des data-Arrays'},
    {text: 'Es erzeugt ein Memory Leak', isAnswer: true},
    {text: 'Es wirft einen RangeError'},
    {text: 'Nichts Ungewöhnliches — das ist in Ordnung'},
  ]}
>
  <slot name="question">
  <div className="question">
    Was ist das Problem mit diesem Code?
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      element.addEventListener('click', () => {
        console.log('clicked!', data.length);
      });
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Der Event-Listener schließt über `data`, also bleibt das Array erreichbar, solange der Listener erreichbar ist. Wenn `element` langlebig ist und der Listener nie entfernt wird, wird dieses große Array ebenfalls retained.

    `element` ist im DOM. Der Listener hält `data`. GC kann es nicht einsammeln.

    Lösung: Große Daten nicht in langlebigen Listenern einfangen, wenn du sie nicht wirklich brauchst, und Listener entfernen, wenn die zugehörige UI abgebaut wird:
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      // do something with data...

      const onClick = () => {
        console.log('clicked!');
      };

      element.addEventListener('click', onClick);
      return () => element.removeEventListener('click', onClick);
    }
    ```

    Modernes V8 ist besser darin als früher. Aber in langlebigen SPAs mit vielen Event-Listenern beißt dieses Muster immer noch zu.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={8}
  group="Experte"
  title="The this Trap"
  options={[
    {text: 'Dan, Dan'},
    {text: 'Dan, undefined'},
    {text: 'undefined, Dan'},
    {text: 'undefined, undefined', isAnswer: true},
  ]}
>
  <slot name="question">
  <div className="question">
    Was wird geloggt?
    ```javascript
    const user = {
      name: 'Dan',
      greetArrow: () => console.log(this?.name),
      greetRegular() {
        const inner = () => console.log(this?.name);
        inner();
      },
    };

    user.greetArrow();
    const fn = user.greetRegular;
    fn();
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Zwei subtile Fallen:

    **`greetArrow`**: Arrow Functions haben kein eigenes `this`. Sie erfassen `this` aus dem umgebenden lexikalischen Scope — der hier der Modul-/Global-Scope ist, nicht `user`. Also ist `this?.name` `undefined`.

    **`fn()`**: `greetRegular` wird ohne Receiver aufgerufen (`fn()` nicht `user.greetRegular()`). Im Strict Mode ist `this` `undefined`. Im Sloppy Mode ist es das globale Objekt. So oder so ist `this?.name` `undefined`.

    Wenn du `user.greetRegular()` direkt aufgerufen hättest, würdest du `'Dan'` bekommen — weil dann `this` an `user` gebunden ist.

    Ergebnis: `undefined, undefined`.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={9}
  group="Bonus"
  title="Memoization via Closure"
  options={[
    {text: 'Ruft expensive() 3 Mal auf'},
    {text: 'Ruft expensive() 1 Mal auf', isAnswer: true},
    {text: 'Ruft expensive() 2 Mal auf'},
    {text: 'Wirft: Maximum call stack exceeded'},
  ]}
>
  <slot name="question">
  <div className="question">
    Wie oft wird `expensive()` wirklich aufgerufen?
    ```javascript
    function memoize(fn) {
      const cache = new Map();
      return (arg) => {
        if (cache.has(arg)) return cache.get(arg);
        const result = fn(arg);
        cache.set(arg, result);
        return result;
      };
    }

    let calls = 0;
    const expensive = (n) => { calls++; return n * 2; };
    const fast = memoize(expensive);

    fast(5);
    fast(5);
    fast(5);
    console.log(calls);
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Die memoizierte Funktion schließt über `cache`. Der erste Aufruf (`fast(5)`) verfehlt den Cache, ruft `expensive` auf und speichert das Ergebnis. Der zweite und dritte Aufruf treffen den Cache und geben sofort zurück.

    `calls` ist `1`.

    Das sind Closures, die echte Arbeit leisten. `cache` ist privat (kann von außerhalb `memoize` nicht zugegriffen werden), persistent (lebt solange `fast` existiert) und wird von allen Aufrufen von `fast` geteilt.
  </div>
  </slot>

Wie bist du zurechtgekommen?

- **9-10**: Du verstehst wirklich Closures. Programmier-Partner sollten sich vor dir fürchten.
- **6-8**: Solide. Die stale Closure / `this`-Fragen haben dich erwischt, was ehrlich ist.
- **3-5**: Du kennst das Konzept, aber die Ränder sind unscharf. Geh die MDN Scope-Chain-Docs durch.
- **0-2**: Du dachtest, du kennst Closures. Jetzt tust du es. Dafür ist das hier.

Die stale Closure in React (#6) und die `this`-Falle (#8) sind die, die echte Produktions-Bugs verursachen. Wenn die dir unbekannt vorkamen, sind sie eine Wiederholung wert — sie tauchen im Code-Review häufiger auf als jedes andere Closure-bezogene Problem.
