<p class="inset">Closures sind eines jener Konzepte, die jeder JavaScript-Entwickler *zu verstehen glaubt* — bis der Moment kommt, in dem er es nicht tut. Dieses Quiz findet die Grenzen dessen, was Sie wirklich wissen.</p>

Closures tauchen in Event-Handlern, Factory-Funktionen, React-Hooks, Modul-Mustern und so ziemlich überall sonst auf. Sie sind nicht exotisch. Sie sind der Stoff, aus dem JavaScript gewebt ist.

Die Fragen beginnen vernünftig und fangen dann an, den Boden unter den Füßen wegzuziehen.

    ```javascript
    function makeCounter() {
    ```

    ```javascript
    console.log(a(), b());
    ```

    Jeder Aufruf von `makeCounter()` erstellt einen **neuen Ausführungskontext** mit einer eigenen frischen `count`. `a` und `b` erfassen *verschiedene* `count`-Bindings. Sie teilen sich keinen Zustand.

    Ergebnis: `1, 1`

    Das ist der Grund, warum Closures verwendet werden, um privaten Zustand zu bauen — jede "Instanz" ist unabhängig.

  group="Die klassische Falle"
  title="var in einer Schleife"
  options={[
    {text: '0, 1, 2, 3, 4'},
    {text: '5, 5, 5, 5, 5', isAnswer: true},
    {text: '1, 2, 3, 4, 5'},
    {text: 'undefined x5'},
  ]}
>
  <slot name="question">
  <div className="question">
    Was wird nach 100ms ausgegeben?
    ```javascript
    for (var i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `var` hat Funktions-Scope. Es gibt genau **ein** `i` — alle fünf Callbacks erfassen dasselbe Binding. Wenn die Timeouts feuern, ist die Schleife fertig und `i` ist `5`.

    Fünf Callbacks. Ein `i`. Wert: `5`. Fünfmal ausgegeben.

    Die Lösung ist `let` (Block-Scope, neues Binding pro Iteration) oder das Umschließen in einem IIFE, um den aktuellen Wert zu erfassen:
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100); // 0, 1, 2, 3, 4 ✓
    }
    ```
  </div>
  </slot>
</Challenge>

    ```javascript
    for (let i = 0; i < 5; i++) {

    }
    ```

  <div className="explanation">
    `let` in einer `for`-Schleife ist speziell. Die Spezifikation sagt: Jede Iteration bekommt ihr eigenes **neues Binding** von `i`. Es ist, als würde die Schleife heimlich folgendes tun:

    ```javascript
    { let i = 0; setTimeout(() => console.log(i), 100) }
    { let i = 1; setTimeout(() => console.log(i), 100) }
    // ...
    ```

    Also erfassen die fünf Callbacks fünf verschiedene Variablen. Ergebnis: `0, 1, 2, 3, 4`.

  index={4}
  group="Closures & Objekte"
  title="Objekt-Mutation"
  options={[
    {text: '{ x: 1 }, { x: 1 }'},
    {text: '{ x: 1 }, { x: 99 }'},
    {text: '{ x: 99 }, { x: 99 }', isAnswer: true},

  ]}
>
  <slot name="question">
  <div className="question">
    Was wird hier ausgegeben?
    ```javascript
    function wrap() {
      const obj = { x: 1 };
      const getObj = () => obj;
      obj.x = 99;
      return getObj;
    }
    const get = wrap();
    console.log(get(), get());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Der Closure erfasst das **Binding** `obj`, nicht eine Momentaufnahme des Objekts. Das Objekt selbst wird nach dem Erstellen des Closure mutiert — aber `obj` zeigt immer noch auf dieselbe Referenz. Beide Aufrufe geben dasselbe mutierte Objekt zurück: `{ x: 99 }`.

    Das trifft Leute, die annehmen, Closures "frieren" Werte ein. Tun sie nicht. Sie erfassen **Referenzen**.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={5}
  group="Fortgeschritten"
  title="Stale Closure in React"
  options={[
    {text: 'Gibt den aktuellen count bei jedem Klick aus'},
    {text: 'Gibt immer 0 aus', isAnswer: true},
    {text: 'Gibt NaN aus'},
    {text: "Das wird nicht kompiliert"},
  ]}
>
  <slot name="question">
  <div className="question">
    Welches Verhalten zeigt diese React-Komponente, wenn der Button mehrmals geklickt wird?
    ```jsx
    function Counter() {
      const [count, setCount] = useState(0);

      useEffect(() => {
        const id = setInterval(() => {
          console.log('count is', count);
        }, 1000);
        return () => clearInterval(id);
      }, []); // empty deps

      return <button onClick={() => setCount(c => c + 1)}>
        Click ({count})
      </button>;
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Klassische "Stale Closure". Der `useEffect` läuft einmal (leeres Dep-Array) und erstellt ein Intervall, das `count` so erfasst, wie es zum Mount-Zeitpunkt war: `0`. Egal wie oft Sie klicken, die Closure des Intervalls sieht niemals den aktualisierten Wert.

    Die Button-Beschriftung aktualisiert sich korrekt — das ist ein frischer Render. Aber `console.log` ist für immer bei `0` gefangen.

    Fixen Sie es, indem Sie `count` zum Dep-Array hinzufügen (erstellt das Intervall neu) oder eine Ref verwenden:
    ```jsx
    const countRef = useRef(count);
    useEffect(() => { countRef.current = count; }, [count]);
    // then use countRef.current inside the interval
    ```
  </div>
  </slot>
</Challenge>

    Warten Sie — lesen Sie sorgfältig. Was passiert tatsächlich?
    ```javascript
    const bank = (() => {
      let balance = 0;
      return {
        deposit: (n) => (balance += n),
        withdraw: (n) => (balance -= n),
        getBalance: () => balance,
      };
    })();

    bank.deposit(100);
    bank.withdraw(40);
    console.log(bank.getBalance());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Die Antwort ist `60`.

    Das ist das **Modul-Muster** — eine IIFE, die ein Objekt aus Closures zurückgibt. Alle drei Methoden erfassen dasselbe `balance`-Binding. Es ist wirklich privat: Sie können von außen nicht auf `balance` zugreifen.

    ```javascript
    console.log(bank.balance); // undefined
    ```

    Vor der Einführung von Klassen war dies *der* Weg, um Zustand in JavaScript zu kapseln. Sie werden es immer noch in Legacy-Codebasen und Utility-Bibliotheken sehen.
  </div>
  </slot>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={7}
  group="Experte"
  title="Closure Memory Leak"
  options={[
    {text: 'Es gibt die Länge des data-Arrays aus'},
    {text: 'Es erzeugt ein Speicherleck', isAnswer: true},
    {text: 'Es wirft einen RangeError'},
    {text: 'Nichts Ungewöhnliches — das ist okay'},
  ]}
>
  <slot name="question">
  <div className="question">
    Was ist das Problem mit diesem Code?
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      element.addEventListener('click', () => {
        console.log('clicked!', data.length);
      });
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Der Event-Listener erfasst `data`, also bleibt das Array erreichbar, solange der Listener erreichbar ist. Wenn `element` langlebig ist und der Listener niemals entfernt wird, bleibt auch dieses große Array im Speicher.

    `element` ist im DOM. Der Listener hält `data`. Der Garbage Collector kann es nicht einsammeln.

    Lösung: Erfassen Sie keine großen Datenmengen in langlebigen Listenern, es sei denn, Sie brauchen sie wirklich, und entfernen Sie Listener, wenn die zugehörige UI abgebaut wird:
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      // do something with data...

      const onClick = () => {
        console.log('clicked!');
      };

      element.addEventListener('click', onClick);
      return () => element.removeEventListener('click', onClick);
    }
    ```

    Moderne V8-Engines sind darin besser geworden als früher. Aber in langlebigen SPAs mit vielen Event-Handlern trifft dieses Muster die Leute immer noch.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={8}
  group="Experte"
  title="Die this-Falle"
  options={[
    {text: 'Dan, Dan'},
    {text: 'Dan, undefined'},
    {text: 'undefined, Dan'},
    {text: 'undefined, undefined', isAnswer: true},
  ]}
>
  <slot name="question">
  <div className="question">
    Was wird hier ausgegeben?
    ```javascript
    const user = {
      name: 'Dan',
      greetArrow: () => console.log(this?.name),
      greetRegular() {
        const inner = () => console.log(this?.name);
        inner();
      },
    };

    user.greetArrow();
    const fn = user.greetRegular;
    fn();
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Zwei subtile Fallen:

    **`greetArrow`**: Arrow-Funktionen haben kein eigenes `this`. Sie erfassen `this` aus dem umgebenden lexikalischen Scope — was hier der Modul-/global-Scope ist, nicht `user`. Also ist `this?.name` `undefined`.

    **`fn()`**: `greetRegular` wird ohne Empfänger aufgerufen (`fn()` nicht `user.greetRegular()`). Im Strict Mode ist `this` `undefined`. Im Sloppy Mode ist es das globale Objekt. In beiden Fällen ist `this?.name` `undefined`.

    Wenn Sie `user.greetRegular()` direkt aufgerufen hätten, würden Sie `'Dan'` erhalten — weil dann `this` an `user` gebunden ist.

    Ergebnis: `undefined, undefined`.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={9}
  group="Bonus"
  title="Memoization via Closure"
  options={[
    {text: 'Ruft expensive() 3 Mal auf'},
    {text: 'Ruft expensive() 1 Mal auf', isAnswer: true},
    {text: 'Ruft expensive() 2 Mal auf'},
    {text: 'Wirft: Maximum call stack exceeded'},
  ]}
>
  <slot name="question">
  <div className="question">
    Wie oft wird `expensive()` tatsächlich aufgerufen?
    ```javascript
    function memoize(fn) {
      const cache = new Map();
      return (arg) => {
        if (cache.has(arg)) return cache.get(arg);
        const result = fn(arg);
        cache.set(arg, result);
        return result;
      };
    }

    let calls = 0;
    const expensive = (n) => { calls++; return n * 2; };
    const fast = memoize(expensive);

    fast(5);
    fast(5);
    fast(5);
    console.log(calls);
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Die memoisierte Funktion erfasst `cache`. Der erste Aufruf (`fast(5)`) verfehlt den Cache, ruft `expensive` auf und speichert das Ergebnis. Der zweite und dritte Aufruf treffen den Cache und kehren sofort zurück.

    `calls` ist `1`.

    Hier leisten Closures echte Arbeit. `cache` ist privat (kann nicht von außerhalb von `memoize` zugegriffen werden), persistent (lebt so lange wie `fast` existiert) und wird über alle Aufrufe von `fast` hinweg geteilt.
  </div>
  </slot>
</Challenge>

Wie war's?

- **9-10**: Sie kennen Closures tatsächlich. Wahrscheinlich erschreckend, mit Ihnen zu pair-programmieren.
- **6-8**: Solide. Die Fragen zu Stale Closures / `this` haben Sie erwischt, was ehrlich ist.
- **3-5**: Sie kennen das Konzept, aber die Ränder sind verschwommen. Lesen Sie die MDN Scope-Chain-Dokumentation.
- **0-2**: Sie dachten, Sie kennen Closures. Jetzt kennen Sie sie. Das ist der Punkt.

Die "Stale Closure" in React (#6) und die `this`-Falle (#8) sind diejenigen, die echte Produktions-Bugs verursachen. Wenn diese ungewohnt wirkten, lohnt es sich, sie noch einmal anzuschauen — sie tauchen in Code-Reviews häufiger auf als jedes andere Closure-bezogene Problem.
