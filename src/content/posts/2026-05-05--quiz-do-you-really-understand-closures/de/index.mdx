<p class="inset">Closures (Abschlüsse) sind eines dieser Konzepte, von denen jeder JavaScript-Entwickler *glaubt*, sie zu verstehen – bis zu dem Moment, in dem er es nicht mehr tut. Dieses Quiz wird die Grenzen dessen finden, was du wirklich weißt.</p>

Closures begegnen dir in Event-Handlern, Factory-Funktionen, React-Hooks, Modul-Mustern und so ziemlich überall sonst. Sie sind nicht exotisch. Sie sind das Gewebe, aus dem JavaScript gewoben ist.

Die Fragen fangen vernünftig an und beginnen dann, den Boden unter den Füßen wegzuziehen.

    ```javascript
    function makeCounter() {

    ```

    Das Schlüsselwort: *dieselbe*. `count` wird nicht in die Closure kopiert. Die Closure hält eine **Live-Referenz** auf das Variablen-Binding. Das ist das ganze Geheimnis.

  ]}

  <slot name="question">
  <div className="question">
    Was wird hier ausgegeben?
    ```javascript
    function makeCounter() {
      let count = 0;
      return () => ++count;
    }
    const a = makeCounter();
    const b = makeCounter();
    console.log(a(), b());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Jeder Aufruf von `makeCounter()` erstellt einen **neuen Ausführungskontext** mit seinem eigenen frischen `count`. `a` und `b` schließen über *verschiedene* `count`-Bindings ab. Sie teilen sich keinen Zustand.

    Ergebnis: `1, 1`

    Das ist der Grund, warum Closures verwendet werden, um privaten Zustand zu erzeugen – jede "Instanz" ist unabhängig.

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={2}
  group="Die klassische Falle"
  title="var in einer Schleife"
  options={[
    {text: '0, 1, 2, 3, 4'},
    {text: '5, 5, 5, 5, 5', isAnswer: true},
    {text: '1, 2, 3, 4, 5'},
    {text: 'undefined x5'},
  ]}
>
  <slot name="question">
  <div className="question">
    Was wird nach 100ms protokolliert?
    ```javascript
    for (var i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `var` hat einen Funktions-Scope. Es gibt genau **ein** `i` — alle fünf Callbacks schließen über dasselbe Binding ab. Bis die Timeouts feuern, ist die Schleife beendet und `i` ist `5`.

    Fünf Callbacks. Ein `i`. Wert: `5`. Fünfmal ausgegeben.

    Die Lösung ist `let` (Block-Scope, neues Binding bei jeder Iteration) oder das Einwickeln in eine IIFE, um den aktuellen Wert zu erfassen:
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100); // 0, 1, 2, 3, 4 ✓
    }
    ```
  </div>
  </slot>
</Challenge>

  <slot name="question">
  <div className="question">
    Gleiche Schleife, anderes Schlüsselwort. Was wird ausgegeben?
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `let` in einer `for`-Schleife ist speziell. Die Spezifikation besagt: Jede Iteration erhält ihr eigenes **neues Binding** von `i`. Es ist so, als würde die Schleife heimlich Folgendes tun:

    ```javascript
    { let i = 0; setTimeout(() => console.log(i), 100) }
    { let i = 1; setTimeout(() => console.log(i), 100) }
    // ...
    ```

    Die fünf Callbacks schließen also über fünf verschiedene Variablen ab. Ergebnis: `0, 1, 2, 3, 4`.
  </div>
  </slot>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={4}
  group="Closures & Objekte"
  title="Objekt-Mutation"
  options={[
    {text: '{ x: 1 }, { x: 1 }'},
    {text: '{ x: 1 }, { x: 99 }'},
    {text: '{ x: 99 }, { x: 99 }', isAnswer: true},
    {text: '{ x: 99 }, { x: 1 }'},
  ]}
>
  <slot name="question">
  <div className="question">
    Was wird hier ausgegeben?
    ```javascript
    function wrap() {
      const obj = { x: 1 };
      const getObj = () => obj;
      obj.x = 99;
      return getObj;
    }
    const get = wrap();
    console.log(get(), get());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Die Closure erfasst das **Binding** `obj`, nicht einen Schnappschuss des Objekts. Das Objekt selbst wird mutiert, nachdem die Closure erstellt wurde – aber `obj` zeigt immer noch auf dieselbe Referenz. Beide Aufrufe geben dasselbe mutierte Objekt zurück: `{ x: 99 }`.

    Das trifft Leute, die annehmen, dass Closures Werte "einfrieren". Das tun sie nicht. Sie erfassen **Referenzen**.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={5}
  group="Fortgeschritten"
  title="Veraltete Closure in React"
  options={[
    {text: 'Gibt bei jedem Klick den aktuellen Stand aus'},
    {text: 'Gibt immer 0 aus', isAnswer: true},
    {text: 'Gibt NaN aus'},
    {text: "Das lässt sich nicht kompilieren"},
  ]}
>
  <slot name="question">
  <div className="question">
    Welches Verhalten zeigt diese React-Komponente, wenn der Button mehrmals angeklickt wird?
    ```jsx
    function Counter() {
      const [count, setCount] = useState(0);

      useEffect(() => {
        const id = setInterval(() => {
          console.log('count ist', count);
        }, 1000);
        return () => clearInterval(id);
      }, []); // leeres Abhängigkeitsarray

      return <button onClick={() => setCount(c => c + 1)}>
        Klick ({count})
      </button>;
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Ein klassischer Fall von "Stale Closure" (veralteter Abschluss). Der `useEffect` wird einmal ausgeführt (leeres Dep-Array) und erstellt ein Intervall, das über `count` so abschließt, wie es zum Zeitpunkt des Mountens war: `0`. Egal wie oft du klickst, die Closure des Intervalls sieht nie den aktualisierten Wert.

    Die Beschriftung des Buttons aktualisiert sich korrekt – das ist ein frischer Render-Vorgang. Aber das `console.log` ist für immer bei `0` gefangen.

    Behebe es, indem du `count` zum Dep-Array hinzufügst (reaktiviert das Intervall) oder ein Ref verwendest:
    ```jsx
    const countRef = useRef(count);
    useEffect(() => { countRef.current = count; }, [count]);
    // dann countRef.current innerhalb des Intervalls verwenden
    ```
  </div>
  </slot>
</Challenge>

  group="Fortgeschritten"
  title="Modul-Muster"
  options={[
    {text: '60', isAnswer: true},
    {text: 'TypeError: balance is not defined'},
    {text: '-40'},
    {text: '100'},
  ]}
>
  <slot name="question">
  <div className="question">
    Warte – lies genau. Was passiert hier wirklich?
    ```javascript
    const bank = (() => {
      let balance = 0;
      return {
        deposit: (n) => (balance += n),
        withdraw: (n) => (balance -= n),
        getBalance: () => balance,
      };
    })();

    bank.deposit(100);
    bank.withdraw(40);
    console.log(bank.getBalance());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Die Antwort ist `60`.

    Dies ist das **Module Pattern** – eine IIFE, die ein Objekt aus Closures zurückgibt. Alle drei Methoden schließen über dasselbe `balance`-Binding ab. Es ist wirklich privat: Du kannst nicht von außen auf `balance` zugreifen.

    ```javascript
    console.log(bank.balance); // undefined
    ```

    Vor der Einführung von Klassen war dies *der* Weg, um Zustand in JavaScript zu kapseln. Man sieht es immer noch in älteren Codebasen und Utility-Bibliotheken.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={7}
  group="Experte"
  title="Closure Memory Leak"
  options={[
    {text: 'Es protokolliert die Länge des Daten-Arrays'},
    {text: 'Es erzeugt ein Speicherleck', isAnswer: true},
    {text: 'Es wirft einen RangeError'},
    {text: 'Nichts Ungewöhnliches – das ist in Ordnung'},
  ]}
>
  <slot name="question">
  <div className="question">
    Was ist das Problem mit diesem Code?
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      element.addEventListener('click', () => {
        console.log('clicked!', data.length);
      });
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Der Event-Listener schließt über `data` ab, sodass das Array so lange erreichbar bleibt, wie der Listener erreichbar ist. Wenn `element` langlebig ist und der Listener nie entfernt wird, bleibt auch dieses große Array im Speicher.

    `element` ist im DOM. Der Listener hält `data`. Der Garbage Collector kann es nicht einsammeln.

    Lösung: Erfasse keine großen Datenmengen in langlebigen Listenern, es sei denn, du brauchst sie wirklich, und entferne Listener, wenn die zugehörige UI abgebaut wird:
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      // mache etwas mit data...

      const onClick = () => {
        console.log('clicked!');
      };

      element.addEventListener('click', onClick);
      return () => element.removeEventListener('click', onClick);
    }
    ```

    Moderne V8-Engines sind darin besser geworden als früher. Aber in langlebigen SPAs mit vielen Event-Listenern trifft dieses Muster die Leute immer noch.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={8}
  group="Experte"
  title="Die this-Falle"
  options={[
    {text: 'Dan, Dan'},
    {text: 'Dan, undefined'},
    {text: 'undefined, Dan'},
    {text: 'undefined, undefined', isAnswer: true},
  ]}
>
  <slot name="question">
  <div className="question">
    Was wird hier ausgegeben?
    ```javascript
    const user = {
      name: 'Dan',
      greetArrow: () => console.log(this?.name),
      greetRegular() {
        const inner = () => console.log(this?.name);
        inner();
      },
    };

    user.greetArrow();
    const fn = user.greetRegular;
    fn();
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Zwei subtile Fallen:

    **`greetArrow`**: Arrow-Funktionen haben kein eigenes `this`. Sie erfassen `this` aus dem umschließenden lexikalischen Scope – was hier der Modul-/globale Scope ist, nicht `user`. Daher ist `this?.name` gleich `undefined`.

    **`fn()`**: `greetRegular` wird ohne Empfänger aufgerufen (`fn()` statt `user.greetRegular()`). Im Strict Mode ist `this` gleich `undefined`. Im Sloppy Mode ist es das globale Objekt. In jedem Fall ist `this?.name` gleich `undefined`.

    Hättest du `user.greetRegular()` direkt aufgerufen, hättest du `'Dan'` erhalten – denn dann ist `this` an `user` gebunden.

    Ergebnis: `undefined, undefined`.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={9}
  group="Bonus"
  title="Memoisierung via Closure"
  options={[
    {text: 'Ruft expensive() 3 Mal auf'},
    {text: 'Ruft expensive() 1 Mal auf', isAnswer: true},
    {text: 'Ruft expensive() 2 Mal auf'},
    {text: 'Wirft: Maximum call stack exceeded'},
  ]}
>
  <slot name="question">
  <div className="question">
    Wie oft wird `expensive()` tatsächlich aufgerufen?
    ```javascript
    function memoize(fn) {
      const cache = new Map();
      return (arg) => {
        if (cache.has(arg)) return cache.get(arg);
        const result = fn(arg);
        cache.set(arg, result);
        return result;
      };
    }

    let calls = 0;
    const expensive = (n) => { calls++; return n * 2; };
    const fast = memoize(expensive);

    fast(5);
    fast(5);
    fast(5);
    console.log(calls);
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Die memoisierte Funktion schließt über `cache` ab. Der erste Aufruf (`fast(5)`) verfehlt den Cache, ruft `expensive` auf und speichert das Ergebnis. Der zweite und dritte Aufruf treffen den Cache und geben sofort das Ergebnis zurück.

    `calls` ist `1`.

    Hier leisten Closures echte Arbeit. `cache` ist privat (kann nicht von außerhalb von `memoize` zugegriffen werden), persistent (lebt so lange wie `fast` existiert) und wird über alle Aufrufe von `fast` hinweg geteilt.
  </div>
  </slot>
</Challenge>
