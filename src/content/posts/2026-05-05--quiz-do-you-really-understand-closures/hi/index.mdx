    `var` function-scoped है। बिल्कुल **एक** ही `i` है — सभी पाँच callbacks उसी binding पर close होते हैं। जब तक timeouts fire होते हैं, loop खत्म हो चुका होता है और `i` की value `5` है।

    पाँच callbacks। एक `i`। value: `5`। पाँच बार log हुआ।

    Fix है `let` (block-scoped, हर iteration में नई binding) या current value capture करने के लिए IIFE में wrap करना:
    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100); // 0, 1, 2, 3, 4 ✓
    }
    ```

    ```javascript
    function wrap() {
      const obj = { x: 1 };

    console.log(get(), get());
    ```

  <slot name='explanation'>

    Closure `obj` की **binding** capture करता है, object का snapshot नहीं। Object को closure बनने के बाद mutate किया जाता है — लेकिन `obj` अभी भी उसी reference को point करता है। दोनों calls उसी mutated object को return करती हैं: `{ x: 99 }`।

    यह उन लोगों को काटता है जो सोचते हैं कि closures values "freeze" करते हैं। ऐसा नहीं होता। ये **references** capture करते हैं।

  group="Advanced"
  title="Stale Closure in React"
  options={[
    {text: 'हर click पर current count log करता है'},
    {text: 'हमेशा 0 log करता है', isAnswer: true},
    {text: 'NaN log करता है'},
    {text: 'यह compile नहीं होगा'},
  ]}
>
  <slot name="question">
  <div className="question">
    बटन को कई बार click करने पर यह React component क्या व्यवहार दिखाता है?
    ```jsx
    function Counter() {
      const [count, setCount] = useState(0);

      useEffect(() => {
        const id = setInterval(() => {
          console.log('count is', count);
        }, 1000);
        return () => clearInterval(id);
      }, []); // empty deps

      return <button onClick={() => setCount(c => c + 1)}>
        Click ({count})
      </button>;
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Classic stale closure। `useEffect` एक बार चलता है (empty dep array) और एक interval बनाता है जो mount होने के समय के `count` पर close होता है: `0`। चाहे कितनी भी बार click करो, interval का closure updated value कभी नहीं देखता।

    Button label सही अपडेट होता है — वो एक fresh render है। लेकिन `console.log` हमेशा `0` पर फँसा रहता है।

    Fix करने के लिए dep array में `count` जोड़ें (interval दोबारा बनेगा) या ref इस्तेमाल करें:
    ```jsx
    const countRef = useRef(count);
    useEffect(() => { countRef.current = count; }, [count]);
    // फिर interval में countRef.current इस्तेमाल करें
    ```
  </div>
  </slot>
</Challenge>

    ```javascript
    console.log(bank.balance); // undefined
    ```

    Classes से पहले, JavaScript में state encapsulate करने का यही *तरीका* था। Legacy codebases और utility libraries में आज भी ये दिखता है।

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={7}
  group="Expert"
  title="Closure Memory Leak"
  options={[
    {text: 'यह data array की length log करता है'},
    {text: 'यह एक memory leak बनाता है', isAnswer: true},
    {text: 'यह RangeError throw करता है'},
    {text: 'कुछ भी असामान्य नहीं — यह ठीक है'},
  ]}
>
  <slot name="question">
  <div className="question">
    इस code में क्या समस्या है?
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      element.addEventListener('click', () => {
        console.log('clicked!', data.length);
      });
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Event listener `data` पर close होता है, इसलिए array तब तक reachable रहती है जब तक listener reachable है। अगर `element` long-lived है और listener कभी हटाया नहीं जाता, तो वो बड़ी array भी retain रहती है।

    `element` DOM में है। Listener `data` को पकड़े हुए है। GC इसे collect नहीं कर सकता।

    Fix: long-lived listeners में बड़ा data तब तक capture मत करो जब तक ज़रूरत न हो, और जब owning UI tear down हो तो listeners हटाओ:
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      // data के साथ कुछ करें...

      const onClick = () => {
        console.log('clicked!');
      };

      element.addEventListener('click', onClick);
      return () => element.removeEventListener('click', onClick);
    }
    ```

    Modern V8 पहले से बेहतर है। लेकिन बहुत सारे event listeners वाले long-lived SPAs में, ये pattern अभी भी लोगों को काटता है।
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={8}
  group="Expert"
  title="The this Trap"
  options={[
    {text: 'Dan, Dan'},
    {text: 'Dan, undefined'},
    {text: 'undefined, Dan'},
    {text: 'undefined, undefined', isAnswer: true},
  ]}
>
  <slot name="question">
  <div className="question">
    यह क्या log करता है?
    ```javascript
    const user = {
      name: 'Dan',
      greetArrow: () => console.log(this?.name),
      greetRegular() {
        const inner = () => console.log(this?.name);
        inner();
      },
    };

    user.greetArrow();
    const fn = user.greetRegular;
    fn();
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    दो सूक्ष्म traps:

    **`greetArrow`**: Arrow functions का अपना `this` नहीं होता। ये `this` को enclosing lexical scope से capture करते हैं — जो यहाँ `user` नहीं, module/global scope है। तो `this?.name` `undefined` है।

    **`fn()`**: `greetRegular` को receiver के बिना call किया जाता है (`fn()` न कि `user.greetRegular()`)। Strict mode में, `this` `undefined` है। Sloppy mode में, ये global object है। दोनों ही स्थिति में, `this?.name` `undefined` है।

    अगर आप सीधे `user.greetRegular()` call करते, तो आपको `'Dan'` मिलता — क्योंकि तब `this` `user` से bound होता।

    परिणाम: `undefined, undefined`।
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={9}
  group="Bonus"
  title="Memoization via Closure"
  options={[
    {text: 'expensive() को 3 बार call करता है'},
    {text: 'expensive() को 1 बार call करता है', isAnswer: true},
    {text: 'expensive() को 2 बार call करता है'},
    {text: 'Throws: Maximum call stack exceeded'},
  ]}
>
  <slot name="question">
  <div className="question">
    `expensive()` असल में कितनी बार call होता है?
    ```javascript
    function memoize(fn) {
      const cache = new Map();
      return (arg) => {
        if (cache.has(arg)) return cache.get(arg);
        const result = fn(arg);
        cache.set(arg, result);
        return result;
      };
    }

    let calls = 0;
    const expensive = (n) => { calls++; return n * 2; };
    const fast = memoize(expensive);

    fast(5);
    fast(5);
    fast(5);
    console.log(calls);
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Memoized function `cache` पर close होता है। पहली call (`fast(5)`) cache miss होती है, `expensive` को call करती है, और result store करती है। दूसरी और तीसरी call cache hit होती हैं और तुरंत return होती हैं।

    `calls` है `1`।

    यही closures का असली काम है। `cache` private है (`memoize` के बाहर से access नहीं हो सकता), persistent है (`fast` जितना रहता है तब तक ज़िंदा रहता है), और `fast` की सभी calls में shared है।

कैसा रहा?

- **9-10**: आप सच में closures समझते हैं। शायद pair-program करना डरावना होगा।
- **6-8**: मज़बूत। Stale closure / `this` वाले सवालों में अटके, जो ईमानदारी है।
- **3-5**: Concept तो समझते हैं लेकिन किनारे धुंधले हैं। MDN scope chain docs पढ़ें।
- **0-2**: आप सोचते थे कि closures समझते हैं। अब समझ गए। यही तो मकसद था।

React में stale closure (#6) और `this` trap (#8) वे हैं जो असली production bugs पैदा करते हैं। अगर ये अपरिचित लगे, तो इन्हें दोबारा देखने लायक हैं — code review में ये closure-related किसी भी दूसरी issue से ज़्यादा आते हैं।
