Al comenzar este artículo (finales de 2018, actualizado en 2024) supuse que terminaría con una tabla de casillas de verificación mixtas.

Seguramente hay _Casos de Uso_ especiales que justificaron [`axios`](https://www.npmjs.com/package/axios), [`request`](https://www.npmjs.com/package/request), [`r2`](https://www.npmjs.com/package/r2), [`superagent`](https://www.npmjs.com/package/superagent), [`got`](https://www.npmjs.com/package/got), etc.

Bueno, resulta que **sobreestimé la necesidad de bibliotecas HTTP de terceros.**

A pesar de usar `fetch` durante varios años (incluso para tareas no triviales: subidas de archivos y soporte de errores/reintentos) todavía tenía conceptos erróneos sobre las capacidades y limitaciones de `fetch`.

El `fetch` nativo no parsea automáticamente las respuestas JSON ni serializa los cuerpos de las solicitudes JSON.

Llamas a `response.json()` a la vuelta y a `JSON.stringify()` a la ida. Axios sigue ganando en ese aspecto de ergonomía; el argumento a favor de `fetch` es que un pequeño helper a menudo cubre la diferencia.

Bueno, veamos lo que `fetch` puede hacer...

### Compatibilidad

A partir de 2022, la API `fetch` tiene [amplio soporte](https://caniuse.com/#feat=fetch) en todos los navegadores modernos y en versiones más recientes de NodeJS v18+.

Si debes dar soporte a IE, puedes usar un [polyfill para fetch](https://github.com/github/fetch#browser-support) con el paquete `github/fetch` (mantenido por un equipo increíble en GitHub).

Es posible retroceder hasta [IE8](https://github.com/camsong/fetch-ie8) - _Los resultados pueden variar_.

Versiones anteriores de NodeJS pueden aprovechar la API `fetch` con el paquete [`node-fetch`](https://www.npmjs.com/package/node-fetch):

```sh
npm install node-fetch
```

_Después de polyfill+node-fetch: 99.99% compatible_ ✅
