Was passiert, wenn du diesen Code ausführst? Versuche, die Ausgabe oder den Fehler vorherzusagen:
```rust
      fn main() {
          let philosopher =

          let greeting = philosopher;

      }
```
  </div>

Dieser Code lässt sich nicht kompilieren, weil er gegen Rusts Ownership-Regeln verstößt. Wenn wir `philosopher` an `greeting` zuweisen, wird das Ownership des Strings an `greeting` verschoben. Nach diesem Move ist `philosopher` nicht mehr gültig und kann nicht mehr verwendet werden.

```rust
      let greeting = philosopher.clone();
```

```rust
      let greeting = &philosopher;
```
3. Einen String-Slice verwenden (leiht einen Teil des Strings aus):
```rust
      let greeting = &philosopher[..];
```
Jede Lösung hat unterschiedliche Anwendungsfälle und Auswirkungen auf die Performance. Klonen ist teurer, gibt dir aber volles Ownership, während Referenzen günstiger sind, aber an Lebenszeiten (Lifetimes) gebunden sind.

  <slot name="question">
  <div className="question">
Was passiert, wenn du diesen Code ausführst? Denk genau an die Ownership-Übertragung:
```rust
      fn take_knowledge(knowledge: String) {
          println!("Knowledge: {}", knowledge);
      }

      fn main() {
          let wisdom = String::from("know thyself");
          take_knowledge(wisdom);
          // What happens to our wisdom?
          println!("Do you {}", wisdom);
      }
```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
Der Code lässt sich nicht kompilieren, weil der Besitz von `wisdom` an `take_knowledge` übergeben wurde und die Variable danach nicht mehr verwendet werden kann.

      Hier sind drei Möglichkeiten, das Problem zu lösen:

      1. Übergabe per Referenz (den Wert ausleihen):
```rust
      fn borrow_it(text: &String) {
          println!("Inside: {}", text);
      }
      borrow_it(&wisdom);  // Now wisdom can be used after
```
2. Den Wert klonen (eine neue Kopie erstellen):
```rust
      take_knowledge(wisdom.clone());  // Original wisdom remains valid
```
3. Den Besitz aus der Funktion zurückgeben:
```rust
      fn take_and_return(text: String) -> String {
          println!("Inside: {}", text);
          text  // Return ownership back
      }
      let wisdom = take_and_return(wisdom);  // Reassign returned ownership
```
Jeder Ansatz hat seine eigenen Anwendungsfälle:
      - Referenzen: Am effizientesten, erfordern aber Lebensdauer-Management
      - Klonen: Einfach, aber potenziell ressourcenintensiv
      - Besitz zurückgeben: Nützlich zum Transformieren von Werten

      Best Practice: Nutze Referenzen, solange du keine Ownership-Übertragung benötigst.

  <slot name="question">
  <div className="question">
Was passiert bei mehreren veränderbaren Referenzen?
```rust
      fn main() {
          let mut wisdom = String::from("He who laughs at");
          let ref1 = &mut wisdom;  // First mutable borrow
          let ref2 = &mut wisdom;  // Second mutable borrow
          ref1.push_str(" himself never runs");
          ref2.push_str(" out of things to laugh at.");
      }
```
Denk an Rusts strikte Regeln für veränderbare Referenzen.
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
Dieser Code verstößt gegen Rusts fundamentale Ausleihregeln:
      - Nur EINE veränderbare Referenz auf einen Wert gleichzeitig
      - ODER beliebig viele unveränderbare Referenzen
      - Referenzen dürfen nicht länger leben als der Wert, auf den sie zeigen

      So behebst du den Code:

      1. Nutze sequenzielle Scopes:
```rust
      let mut wisdom = String::from("He who laughs at");
      {
          let ref1 = &mut wisdom;
          ref1.push_str(" himself never runs");
      }  // ref1 goes out of scope
      let ref2 = &mut wisdom;  // Now this is valid
      ref2.push_str(" out of things to laugh at.");
```
2. Oder ändere den String innerhalb eines einzigen Borrows:
```rust
      let mut wisdom = String::from("He who laughs at");
      let ref1 = &mut wisdom;
      ref1.push_str(" himself never runs out of things to laugh at.");
```
Diese Regeln verhindern Data Races bereits zur Kompilierzeit und machen Rust standardmäßig thread-sicher.

      Häufige Falle: Der Versuch, mehrere veränderbare Referenzen zu nutzen, um Cloning zu vermeiden oder
      verschiedene Teile desselben Werts gleichzeitig zu ändern.

  </slot>

    {text: 'Fehler: explizite Lifetime erforderlich', hint: 'Der Compiler kann das hier automatisch ableiten.'},
    {text: 'Fehler: Lifetime-Konflikt', hint: 'Die Lifetimes passen hier perfekt zusammen.'},
  ]}
>
  <slot name="question">
  <div className="question">
Kompiliert dieser Code? Wenn ja, warum? Wenn nein, was ist das Problem?
```rust
      fn first_word(s: &str) -> &str {  // No explicit lifetimes?
          match s.find(' ') {
              Some(pos) => &s[0..pos],
              None => s,
          }
      }

      fn main() {
          let name = String::from("Seneca the Younger");
          let first = first_word(&name);
          println!("Hello, {}", first);
      }
```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
Dieser Code kompiliert erfolgreich dank Rusts Lifetime-Elisionsregeln. 
      Diese Regeln ermöglichen es dem Compiler, Lifetimes in gängigen Mustern automatisch abzuleiten.

      Die drei Lifetime-Elisionsregeln lauten:
      1. Jeder Parameter erhält seinen eigenen Lifetime-Parameter.
      2. Gibt es genau einen Eingabe-Lifetime-Parameter, wird dieser Lifetime allen Ausgabe-Lifetime-Parametern zugewiesen.
      3. Gibt es mehrere Eingabe-Lifetime-Parameter, aber einer davon ist &self oder &mut self, wird der Lifetime von self allen Ausgabe-Lifetime-Parametern zugewiesen.

      Diese Funktion ist äquivalent zu:
```rust
      fn first_word<'a>(s: &'a str) -> &'a str {
          // ... same implementation
      }
```
Häufige Muster, bei denen Elision funktioniert:
```rust
      // These don't need explicit lifetimes
      fn get_str(s: &str) -> &str { s }
      fn get_first(s: &str) -> &str { &s[0..1] }

      // These would need explicit lifetimes
      fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
          if x.len() > y.len() { x } else { y }
      }
```
Best Practice: Lass die Elision für dich arbeiten, wann immer möglich, aber verstehe, wann explizite Lifetimes nötig sind.
  </div>
  </slot>
</Challenge>

  <slot name="question">
  <div className="question">
Was ist an dieser rekursiven Typdefinition falsch?
```rust
      #[derive(Debug)]
      enum CatList {
          Cons(i32, CatList),  // Recursive without indirection
          Nil,
      }

      fn main() {
          let catlist = CatList::Cons(1,
              CatList::Cons(2,
                  CatList::Cons(3,
                      CatList::Nil)));
      }
```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
Dieser Code scheitert, weil der Compiler die Größe von `CatList` zur Kompilierzeit nicht bestimmen kann. Die rekursive Natur des Typs bedeutet, dass er unendlich groß sein könnte!

      So behebst du das Problem mit `Box<T>`:
```rust
      #[derive(Debug)]
      enum CatList {
          Cons(i32, Box<CatList>),  // Box provides a fixed-size pointer
          Nil,
      }

      fn main() {
          let catlist = CatList::Cons(1,
              Box::new(CatList::Cons(2,
                  Box::new(CatList::Cons(3,
                      Box::new(CatList::Nil))))));
      }
```
Warum `Box<T>` funktioniert:
      1. Box stellt einen Zeiger fester Größe bereit (normalerweise 8 Byte auf 64-Bit-Systemen)
      2. Die eigentlichen Daten werden auf dem Heap gespeichert
      3. Der Compiler weiß nun exakt, wie viel Speicher allokiert werden muss

      Häufige Anwendungsfälle für `Box<T>`:
      - Rekursive Datenstrukturen (verkettete Listen, Bäume)
      - Große Daten, die garantiert auf dem Heap liegen sollen
      - Trait-Objekte, wenn du dynamisches Dispatch benötigst

      Best Practice: Verwende `Box<T>`, wenn du Folgendes benötigst:
      - Rekursive Typen
      - Eine garantierte Heap-Allokation
      - Das Verschieben großer Daten ohne Kopieren
  </div>
  </slot>
</Challenge>

So funktioniert `Rc` im Detail:

      1. Erstes Erstellen mit `Rc::new()`: count = 1
      2. Erster Klon für `marcus`: count = 2
      3. Zweiter Klon für `aurelius`: count = 3

      Wichtige Eigenschaften von `Rc`:
```rust
      use std::rc::Rc;

          let original = Rc::new(String::from("Shared"));
          println!("Count after creation: {}", Rc::strong_count(&original)); // 1
          
          {
              let copy = Rc::clone(&original);
```

      - `Rc::clone()` ist günstig – es erhöht lediglich einen Zähler

      - Nutze `Weak`-Referenzen, um Referenzzyklen zu vermeiden

      - Verwende `Rc`, wenn du gemeinsame Ownership benötigst
      - Greife auf `Arc` zurück, wenn Thread-Sicherheit erforderlich ist
      - Vermeide das Erstellen von Referenzzyklen

  <div className="question">
Wird diese Struct-Definition kompilieren? Warum oder warum nicht?
```rust
      struct Philosopher {
          name: &str,    // Reference without lifetime
          quote: &str,   // Another reference without lifetime
      }

      fn main() {
          let phil = Philosopher {
              name: "Seneca",
              quote: "Luck happens when preparation meets opportunity",
          };
      }
```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
Der Code scheitert, weil Structs mit Referenzen zwingend Lebensdauern angeben müssen. So reparierst du es:
```rust
      // Single lifetime parameter
      struct Philosopher<'a> {
          name: &'a str,
          quote: &'a str,
      }

      // Or different lifetimes if needed
      struct PhilosopherFlex<'n, 'q> {
          name: &'n str,
          quote: &'q str,
      }
```
Typische Muster:
```rust
      // Own the data instead
      struct PhilosopherOwned {
          name: String,
          quote: String,
      }

      // Mixed ownership
      struct PhilosopherMixed<'a> {
          name: String,      // Owned
          quote: &'a str,    // Borrowed
      }
```
Best Practices:
      1. Verwende besitzende Typen (String), wenn du Daten dauerhaft speichern musst
      2. Nutze Referenzen, wenn die Lebensdauer des Structs eindeutig kürzer ist als die der Daten
      3. Erwäge mehrere Lebensdauer-Parameter, wenn Referenzen unterschiedliche Gültigkeitsbereiche haben können
      4. Dokumentiere Lebensdauer-Beziehungen in komplexen Strukturen

  <div className="question">
Was passiert bei dieser Funktion, die die längere von zwei String-Slices zurückgibt?
```rust
      fn longest(text1: &str, text2: &str) -> &str {
          if text1.len() > text2.len() {
              text1    // Returning a reference, but which lifetime?
          } else {
              text2    // Could be this reference instead
          }
      }

      fn main() {
          println!("{}", longest(
              "Seneca the Younger",
              "Marcus Aurelius"
          ));
      }
```
  </div>
  </slot>

  <div className="explanation">
Dieser Code schlägt fehl, weil der Compiler die Beziehung zwischen den Lebensdauern der Eingaben und der Ausgabe nicht bestimmen kann. Hier ist der Grund und die Lösung:
```rust
      // Correct version with explicit lifetime annotation
      fn longest<'a>(text1: &'a str, text2: &'a str) -> &'a str {
          if text1.len() > text2.len() {
              text1
          } else {
              text2
          }
      }

      // Alternative with different lifetimes
      fn longest_flex<'a, 'b>(text1: &'a str, text2: &'b str) -> &'a str {
          if text1.len() > text2.len() {
              text1
          } else {
              text2.to_string().as_str() // Won't compile! Shows why we need same lifetime
          }
      }
```
Warum Lebensdauern hier nötig sind:
      1. Mehrere Eingabe-Referenzen können unterschiedliche Lebensdauern haben
      2. Der Rückgabewert muss mindestens so lange leben wie beide Eingaben
      3. Der Compiler muss diese Beziehungen überprüfen

      Häufige Muster:
```rust
      // Single input reference - elision works
      fn first_word(s: &str) -> &str { /* ... */ }

      // Multiple references, same lifetime needed
      fn compare_str<'a>(s1: &'a str, s2: &'a str) -> &'a str { /* ... */ }

      // Different lifetimes possible
      fn combine<'a, 'b>(s1: &'a str, s2: &'b str) -> String { /* ... */ }
```
Best Practices:
      1. Lass Lifetime Elision wirken, wenn möglich
      2. Nutze explizite Lebensdauern, wenn Beziehungen klar sein müssen
      3. Erwäge die Rückgabe von besitzenden Typen, um Lebensdauer-Komplexität zu vermeiden
      4. Dokumentiere komplexe Lebensdauer-Beziehungen
  </div>

Was passiert, wenn dieser Code läuft?
```rust
      use std::cell::RefCell;

      fn main() {
          let data = RefCell::new(42);
          let _borrow1 = data.borrow_mut();  // First mutable borrow
          let _borrow2 = data.borrow_mut();  // Second mutable borrow
```

RefCell bietet Interior Mutability, erzwingt aber dennoch Rusts Borrowing-Regeln zur Laufzeit:
```rust
      use std::cell::RefCell;

      fn demonstrate_refcell() {
          let data = RefCell::new(42);
          
          // Correct way to use RefCell
          {
              let mut first = data.borrow_mut();
              *first += 1;
          } // first is dropped here
          
          // Now we can borrow again
          let second = data.borrow_mut();
          
          // Or multiple immutable borrows
          let read1 = data.borrow();
          let read2 = data.borrow(); // This is OK
      }
```

      1. RefCell verlagert Borrowing-Prüfungen zur Laufzeit
      2. Kann Panics auslösen, wenn Regeln verletzt werden
      3. Nützlich für das Interior-Mutability-Pattern

      - Mock-Objekte in Tests
      - Implementierung selbstreferenzieller Strukturen
      - Wenn Daten hinter einer gemeinsamen Referenz mutiert werden müssen
      
      Best Practices:
      1. Bevorzuge wann immer möglich Borrowing zur Kompilierzeit
      2. Halte RefCell-Borrows in engen Gültigkeitsbereichen
      3. Erwäge die Nutzung von drop(), um Borrows explizit zu beenden
      4. Verwende RefCell, wenn du Interior Mutability benötigst

Was gibt dieser Code aus?
```rust
      use std::cell::Cell;

      fn main() {
          let life = Cell::new(42);
          let meaning = &life;        // Shared reference
          println!("{}", life.get()); // What prints here?
```

Cell und RefCell dienen unterschiedlichen Zwecken bei der inneren Veränderbarkeit:
```rust
      use std::cell::{Cell, RefCell};

      // Cell for Copy types
      struct Counter {
          count: Cell<i32>,
      }

      impl Counter {
          fn increment(&self) {
              self.count.set(self.count.get() + 1);
          }
```
