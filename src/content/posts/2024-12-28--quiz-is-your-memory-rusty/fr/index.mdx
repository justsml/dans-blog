    1. Passer par référence (emprunter la valeur) :

    ```rust
    fn borrow_it(text: &String) {
    ```

    Que se passe-t-il avec plusieurs références mutables ?
    ```rust
    fn main() {
        let mut wisdom = String::from("He who laughs at");
        let ref1 = &mut wisdom;  // First mutable borrow
        let ref2 = &mut wisdom;  // Second mutable borrow
        ref1.push_str(" himself never runs");
        ref2.push_str(" out of things to laugh at.");
    }
    ```
    Pensez aux règles de Rust concernant les références mutables.
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Ce code viole les règles fondamentales d'emprunt de Rust :
    - Une SEULE référence mutable à une valeur à la fois
    - OU un nombre quelconque de références immuables
    - Les références ne peuvent pas survivre à leur référent

    Voici comment corriger le code :

    1. Utilisez un cadrage séquentiel :
    ```rust
    let mut wisdom = String::from("He who laughs at");
    {
        let ref1 = &mut wisdom;
        ref1.push_str(" himself never runs");
    }  // ref1 goes out of scope
    let ref2 = &mut wisdom;  // Now this is valid
    ref2.push_str(" out of things to laugh at.");
    ```
    2. Ou modifiez la chaîne en un seul emprunt :
    ```rust
    let mut wisdom = String::from("He who laughs at");
    let ref1 = &mut wisdom;
    ref1.push_str(" himself never runs out of things to laugh at.");
    ```
    These rules prevent data races at compile time, making Rust thread-safe by default.

    Common pitfall: Trying to use multiple mutable references to avoid cloning or
    to modify different parts of the same value simultaneously.

  <slot name="question">
  <div className="question">
    Ce code va-t-il compiler ? Si oui, pourquoi ? Si non, quel est le problème ?
    ```rust
    fn first_word(s: &str) -> &str {  // No explicit lifetimes?
        match s.find(' ') {
            Some(pos) => &s[0..pos],
            None => s,
        }
    }

    fn main() {
        let name = String::from("Seneca the Younger");
        let first = first_word(&name);
        println!("Hello, {}", first);
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Ce code compile avec succès grâce aux règles d'élision de durée de vie de Rust.
    Ces règles permettent au compilateur de déduire automatiquement les durées de vie dans les motifs courants.

    Les trois règles d'élision de durée de vie sont :
    1. Chaque paramètre reçoit son propre paramètre de durée de vie
    2. S'il y a exactement un paramètre de durée de vie en entrée, cette durée de vie est assignée à tous les paramètres de durée de vie en sortie
    3. S'il y a plusieurs paramètres de durée de vie en entrée, mais que l'un d'eux est &self ou &mut self, la durée de vie de self est assignée à tous les paramètres de durée de vie en sortie

    Cette fonction est équivalente à :
    ```rust
    fn first_word<'a>(s: &'a str) -> &'a str {
        // ... same implementation
    }
    ```
    Motifs courants où l'élision fonctionne :
    ```rust
    // These don't need explicit lifetimes
    fn get_str(s: &str) -> &str { s }
    fn get_first(s: &str) -> &str { &s[0..1] }

    // These would need explicit lifetimes
    fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
        if x.len() > y.len() { x } else { y }
    }
    ```
    Meilleure pratique : Laissez l'élision travailler pour vous quand c'est possible, mais comprenez quand des durées de vie explicites sont nécessaires.
  </div>
  </slot>
</Challenge>

    Quel est le problème avec cette définition de type récursive ?
    ```rust
    #[derive(Debug)]
    enum CatList {
        Cons(i32, CatList),  // Recursive without indirection
        Nil,
    }

    fn main() {
        let catlist = CatList::Cons(1,
            CatList::Cons(2,
                CatList::Cons(3,
                    CatList::Nil)));
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Ce code échoue car le compilateur ne peut pas déterminer la taille de `CatList` à la compilation. La nature récursive du type signifie qu'il pourrait être infiniment grand !

    Voici comment le corriger en utilisant `Box<T>` :
    ```rust
    #[derive(Debug)]
    enum CatList {
        Cons(i32, Box<CatList>),  // Box provides a fixed-size pointer
        Nil,
    }

    fn main() {
        let catlist = CatList::Cons(1,
            Box::new(CatList::Cons(2,
                Box::new(CatList::Cons(3,
                    Box::new(CatList::Nil))))));
    }
    ```
    Pourquoi `Box<T>` fonctionne :
    1. Box fournit un pointeur de taille fixe (généralement 8 octets sur les systèmes 64 bits)
    2. Les données réelles sont stockées sur le tas
    3. Le compilateur sait maintenant exactement combien d'espace allouer

    Cas d'utilisation courants de `Box<T>` :
    - Structures de données récursives (listes chaînées, arbres)
    - Grandes données que vous voulez garantir allouées sur le tas
    - Objets de trait lorsque vous avez besoin de dispatch dynamique

    Bonne pratique : Utilisez `Box<T>` lorsque vous avez besoin :
    - De types récursifs
    - De garantir l'allocation sur le tas
    - De déplacer de grandes données sans copie

  </slot>

    Que se passe-t-il avec cette fonction qui retourne la plus longue de deux tranches de chaîne ?
    ```rust
    fn longest(text1: &str, text2: &str) -> &str {
        if text1.len() > text2.len() {
            text1    // Returning a reference, but which lifetime?
        } else {
            text2    // Could be this reference instead
    ```

    ```rust
    // Correct version with explicit lifetime annotation
    fn longest<'a>(text1: &'a str, text2: &'a str) -> &'a str {
        if text1.len() > text2.len() {
            text1
        } else {
            text2
        }
    ```

    1. Plusieurs références d'entrée peuvent avoir des durées de vie différentes

    3. Le compilateur doit vérifier ces relations

    ```rust

    fn first_word(s: &str) -> &str { /* ... */ }

    // Different lifetimes possible
    fn combine<'a, 'b>(s1: &'a str, s2: &'b str) -> String { /* ... */ }
    ```
    Bonnes pratiques :
    1. Laissez l'élision de durée de vie fonctionner quand c'est possible

    3. Envisagez de retourner des types possédés pour éviter la complexité des durées de vie

    ```rust
            self.count.set(self.count.get() + 1);
        }
    }

    struct Logger {
    ```

    ```rust
    // Correct way - Release lock before acquiring it again
    fn safe_mutex() {
        let lock = Arc::new(Mutex::new(42));

            let mut data = lock.lock().unwrap();
    ```

    ```rust
        path: String,

                file: File::create(path)?,
                path: path.to_string(),

            self.file.write_all(content.as_bytes())

        }
    }
    ```
