क्या आपने कभी वो एहसास महसूस किया है जब एक लाइन का बदलाव मेमोरी उपयोग आधा कर देता है? मैंने यह Chrome के DevTools Performance Monitor में देखा, जब एक डैशबोर्ड ऐप हर घंटे 100MB लीक करने से पूरी दोपहर स्थिर चलने लगा।

सिर्फ एक लाइन का बदलाव: `new Map()` बन गया `new WeakMap()`।

बस इतना ही। वही API, वही इस्तेमाल का तरीका, लेकिन अंदर से पूरी तरह अलग व्यवहार। इसे समझने के लिए आपको वह चीज़ समझनी होगी जिसके बारे में ज़्यादातर JavaScript डेवलपर कभी नहीं सोचते: जब आपके डेटा को कोई नहीं देख रहा होता, तब क्या होता है?

## जब रेफ़रेंसेस एंकर बन जाते हैं

JavaScript में एक सामान्य Map अपनी keys को कीमती सामान की तरह पकड़ता है। एक बार कुछ डाल दिया, Map उसे मज़बूती से पकड़े रखता है। Garbage Collector यह रिश्ता देखकर सोचता है, "साफ़ है, इन्हें अभी भी इस ऑब्जेक्ट की ज़रूरत है, बेहतर है इसे न छेड़ूँ।"

यह सुरक्षात्मक स्वभाव तब समस्या बनता है जब आप अस्थायी चीज़ों के बारे में metadata स्टोर कर रहे हों। DOM nodes जो हटा दिए जाते हैं। user sessions जो expire हो जाते हैं। component instances जो unmount हो जाते हैं। Map को नहीं पता कि ये ऑब्जेक्ट्स अब काम के नहीं रहे। उसे बस इतना पता है कि उसके पास एक रेफ़रेंस है, इसलिए वह उन्हें ज़िंदा रखता है।

```javascript
const cache = new Map();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// एलीमेंट DOM से चला गया, लेकिन cache उसे मेमोरी में पकड़े हुए है
```

Garbage Collector `element` को साफ़ नहीं कर सकता क्योंकि `cache` अभी भी उसकी तरफ़ इशारा कर रहा है। इसे "strong reference" कहते हैं, और लंबे समय तक चलने वाले Single Page Apps में यह एक leak बन जाता है जो आखिरकार browser को crash कर देता है।

## WeakMap नियम बदल देता है

WeakMap अलग तरह से काम करता है। यह अपनी keys को स्थायी residents के बजाय अस्थायी नागरिकों की तरह मानता है। जब आप कुछ WeakMap में स्टोर करते हैं, तो आप मूल रूप से कह रहे होते हैं: "मैं इस डेटा को इस ऑब्जेक्ट से जोड़ना चाहता हूँ, लेकिन मैं उस वजह से इसे ज़िंदा नहीं रखना चाहता।"

यदि किसी ऑब्जेक्ट को मेमोरी में रखने वाली एकमात्र चीज़ WeakMap है, तो Garbage Collector उसे हटा सकता है। जब ऑब्जेक्ट गायब हो जाता है, WeakMap entry भी अपने आप गायब हो जाती है। किसी manual cleanup की ज़रूरत नहीं।

```javascript
const cache = new WeakMap();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// एलीमेंट Garbage Collected हो जाता है
// cache entry अपने आप गायब हो जाती है
```

मैंने एक benchmark चलाया जिसमें 100,000 DOM nodes बनाए, हर एक के बारे में metadata स्टोर किया, फिर सबको हटा दिया। Map के साथ browser ने 150-200MB पकड़े रखा। WeakMap के साथ यह 70-80MB तक गिर गया। वही कोड, वही functionality, आधा memory footprint।

## आप क्या छोड़ते हैं

WeakMap की कुछ सीमाएँ तब तक limitations लगती हैं जब तक आपको एहसास नहीं होता कि यही चीज़ इसे possible बनाती है।

**आप WeakMap पर iterate नहीं कर सकते।** कोई `forEach` नहीं, कोई `keys()` नहीं, कोई `values()` नहीं। जब आप इसके बारे में सोचते हैं तो यह समझ में आता है: Garbage Collector आपके loop के बीच में एक entry delete कर सकता है। क्या आप सच में उससे निपटना चाहेंगे?

आप size नहीं देख सकते। कोई `.size` property नहीं, कोई `.length` नहीं। फिर से, यह एक moving target है। संख्या आपके पूछने और जवाब मिलने के बीच बदल सकती है।

**Keys केवल ऑब्जेक्ट्स हो सकते हैं।** न strings, न numbers, न primitives। यह weak references के काम करने के तरीके के लिए मौलिक है: primitive values की कोई ऐसी identity नहीं होती जिसे उनके value से अलग track किया जा सके।

ये bugs नहीं हैं। यह design है। WeakMap एक खास काम के लिए बना है: ऑब्जेक्ट्स को साफ़ होने से रोके बिना उनसे metadata जोड़ना। अगर आपको iteration, primitive keys, या entries की गिनती चाहिए, तो आप शायद कोई अलग problem solve कर रहे हैं और आपको regular Map इस्तेमाल करना चाहिए।

## यह वास्तव में कहाँ मदद करता है

"private data" pattern WeakMap का मूल use case था, उस समय जब JavaScript में `#private` fields नहीं थे। libraries class के बाहर एक WeakMap बनाती थीं और उसका उपयोग उस data को स्टोर करने के लिए करती थीं जो instance पर accessible नहीं होना चाहिए।

```javascript
const privateData = new WeakMap();

class User {
  constructor(name) {
    privateData.set(this, { name });
  }

  getName() {
    return privateData.get(this).name;
  }
}
```

जब User instance Garbage Collected हो जाता है, private data भी उसके साथ चला जाता है। किसी cleanup code की ज़रूरत नहीं।

Memoization एक और natural fit है, खासकर जब आप primitive values के बजाय object inputs के आधार पर results cache कर रहे हों। अगर आपकी expensive calculation input के रूप में config object लेती है, तो WeakMap का मतलब है कि आपको इस बात की चिंता नहीं करनी होगी कि cache configs से ज़्यादा समय तक ज़िंदा रहे।

```javascript
const cache = new WeakMap();

function expensiveCalc(obj) {
  if (cache.has(obj)) return cache.get(obj);
  
  const result = heavyMath(obj);
  cache.set(obj, result);
  return result;
}
```

Cache केवल तब तक जीवित रहता है जब तक cache किए जा रहे ऑब्जेक्ट्स जीवित हैं। एक बार `obj` कहीं और referenced नहीं रहता, cached result और cache entry दोनों एक साथ गायब हो जाते हैं।

## इसे कब उपयोग करें

modern web apps में memory leaks आमतौर पर stale references से आते हैं जिन्हें cleanup हो जाना चाहिए था। अगर आप कुछ भी long-running बना रहे हैं, एक dashboard जो पूरा दिन खुला रहता है, एक chat app जो घंटों चलता है, एक admin panel जो कभी refresh नहीं होता, तो आपको यह सोचना होगा कि पुराने data के साथ क्या होता है।

WeakMap खास तौर पर तब उपयोगी है जब आप DOM nodes, component instances, या किसी भी ऐसे ऑब्जेक्ट के साथ data जोड़ रहे हों जिसके lifetime को आप control नहीं करते। अगर आप किसी reference के आधार पर कुछ store कर रहे हैं और वह reference गायब हो सकता है, तो WeakMap cleanup को बहुत सरल बना देता है।

Regular Map अभी भी सही choice है जब आप eviction policies के साथ actual cache बना रहे हों, जब आपको entries पर iterate करना हो, जब आप primitive keys इस्तेमाल कर रहे हों, या जब data itself मायने रखता हो न कि किसी object के साथ उसका association।

`WeakMap` के बारे में अच्छी बात यह है कि आमतौर पर साफ़ होता है कि आपको इसकी ज़रूरत है। अगर आप cleanup code लिख रहे हैं ताकि objects destroy होने पर Map entries हट सकें, तो यह एक संकेत है। अगर आप चिंतित हैं कि memory अनियंत्रित बढ़ रही है क्योंकि आपको पता नहीं चीज़ें कब delete करें, तो यह भी एक संकेत है।

कभी-कभी सबसे अच्छी feature वह होती है जो बस काम कर जाए, बिना आपको उसके बारे में सोचने पर मजबूर किए।
