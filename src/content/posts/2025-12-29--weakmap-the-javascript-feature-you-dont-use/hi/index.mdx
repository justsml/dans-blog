subTitle: "कमजोर संदर्भों (weak references) के साथ कमजोर कोड को ठीक करना!"
---

क्या आप उस एहसास को जानते हैं जब आप कोड की एक लाइन बदलते हैं और अपने मेमोरी उपयोग को 50% गिरते हुए देखते हैं? मुझे वह पल तब महसूस हुआ जब मैं क्रोम के DevTools Performance Monitor को देख रहा था, जहाँ एक डैशबोर्ड ऐप हर घंटे 100MB खर्च कर रहा था, लेकिन बदलाव के बाद वह पूरी दोपहर सफाई से चला।

सिर्फ एक लाइन का बदलाव: `new Map()` बन गया `new WeakMap()`।

बस इतना ही। वही API, वही उपयोग का तरीका, लेकिन अंदरूनी तौर पर पूरी तरह से अलग व्यवहार। लेकिन यह क्यों काम करता है, इसे समझने का मतलब है उस चीज़ को समझना जिसके बारे में अधिकांश जावास्क्रिप्ट डेवलपर्स कभी नहीं सोचते: तब क्या होता है जब आपके डेटा की अब किसी को ज़रूरत नहीं होती?

## जब संदर्भ (References) लंगर (Anchors) बन जाते हैं

जावास्क्रिप्ट में एक साधारण Map अपनी कुंजियों (keys) को कीमती सामान की तरह मानता है। एक बार जब आप इसमें कुछ डाल देते हैं, तो Map उसे मजबूती से पकड़े रखता है। Garbage Collector इस रिश्ते को देखता है और सोचता है, "साफ है कि उन्हें अभी भी इस ऑब्जेक्ट की ज़रूरत है, बेहतर होगा कि इसे न छुआ जाए।"

यह सुरक्षात्मक स्वभाव तब समस्या बन जाता है जब आप अस्थायी चीजों के बारे में मेटाडेटा स्टोर कर रहे होते हैं। जैसे DOM नोड्स जिन्हें हटा दिया गया है, यूजर सेशन जो समाप्त हो गए हैं, या कॉम्पोनेंट इंस्टेंस जो अनमाउंट (unmount) हो गए हैं। Map को यह नहीं पता होता कि ये ऑब्जेक्ट अब उपयोगी नहीं रहे। उसे बस इतना पता है कि उसके पास एक संदर्भ (reference) है, इसलिए वह उन्हें जीवित रखता है।

```javascript
const cache = new Map();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// एलीमेंट DOM से हट गया है, लेकिन cache अभी भी इसे मेमोरी में रख रहा है
```

Garbage Collector `element` को साफ नहीं कर सकता क्योंकि `cache` अभी भी उसकी ओर इशारा कर रहा है। इसे "स्ट्रांग रेफरेंस" (strong reference) कहा जाता है, और लंबे समय तक चलने वाले सिंगल पेज ऐप्स (SPAs) में, यह एक लीक बन जाता है जो अंततः ब्राउज़र को क्रैश कर देता है।

## WeakMap नियम बदल देता है

एक WeakMap अलग तरह से काम करता है। यह अपनी कुंजियों को स्थायी निवासियों के बजाय अस्थायी नागरिकों की तरह मानता है। जब आप WeakMap में कुछ स्टोर करते हैं, तो आप मूल रूप से कह रहे होते हैं: "मैं इस डेटा को इस ऑब्जेक्ट के साथ जोड़ना चाहता हूँ, लेकिन मैं इसके जीवित रहने का कारण नहीं बनना चाहता।"

यदि किसी ऑब्जेक्ट को मेमोरी में रखने वाली एकमात्र चीज़ एक WeakMap है, तो Garbage Collector को उसे हटाने की अनुमति होती है। जब ऑब्जेक्ट गायब हो जाता है, तो WeakMap प्रविष्टि (entry) भी उसके साथ गायब हो जाती है। किसी मैनुअल सफाई की आवश्यकता नहीं होती।

```javascript
const cache = new WeakMap();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// एलीमेंट Garbage Collect हो जाता है
// कैशे एंट्री अपने आप गायब हो जाती है
```

मैंने 100,000 DOM नोड्स बनाकर, प्रत्येक के बारे में मेटाडेटा स्टोर करके और फिर उन सभी को हटाकर एक बेंचमार्क चलाया। Map के साथ, ब्राउज़र ने 150-200MB रोके रखा। WeakMap के साथ, यह घटकर 70-80MB रह गया। वही कोड, वही कार्यक्षमता, लेकिन मेमोरी का आधा हिस्सा।

## आपको क्या छोड़ना पड़ता है

WeakMap में कुछ ऐसी बाधाएं हैं जो तब तक सीमाएं लगती हैं जब तक आपको यह एहसास नहीं होता कि वे ही इसे जादुई बनाती हैं।

**आप WeakMap पर इटरेशन (iteration) नहीं कर सकते।** कोई `forEach` नहीं, कोई `keys()` नहीं, कोई `values()` नहीं। यह तब तर्कसंगत लगता है जब आप इसके बारे में सोचते हैं: Garbage Collector आपके लूप के बीच में ही किसी एंट्री को हटा सकता है। क्या आप वास्तव में इससे निपटना चाहते हैं?

**आप साइज़ (size) चेक नहीं कर सकते।** कोई `.size` प्रॉपर्टी नहीं, कोई `.length` नहीं। फिर से, यह एक बदलता हुआ लक्ष्य है। आपके पूछने और जवाब मिलने के बीच संख्या बदल सकती है।

**कुंजियाँ (Keys) ऑब्जेक्ट होनी चाहिए।** कोई स्ट्रिंग नहीं, कोई नंबर नहीं, कोई प्रिमिटिव (primitive) नहीं। यह इस बात के लिए मौलिक है कि कमजोर संदर्भ (weak references) कैसे काम करते हैं: प्रिमिटिव वैल्यूज़ की अपनी कोई अलग पहचान नहीं होती जिसे उनके मूल्य से अलग ट्रैक किया जा सके।

ये बग नहीं हैं। ये डिज़ाइन हैं। WeakMap को एक विशिष्ट काम के लिए बनाया गया है: ऑब्जेक्ट्स को साफ होने से रोके बिना उनके साथ मेटाडेटा जोड़ना। यदि आपको इटरेशन या प्रिमिटिव कुंजियों या प्रविष्टियों की गिनती की आवश्यकता है, तो आप शायद एक अलग समस्या का समाधान कर रहे हैं और आपको एक नियमित Map का उपयोग करना चाहिए।

## यह वास्तव में कहाँ मदद करता है

"प्राइवेट डेटा" पैटर्न WeakMap का मूल उपयोग मामला था, इससे पहले कि जावास्क्रिप्ट में `#private` फ़ील्ड्स आए। लाइब्रेरी क्लास के बाहर एक WeakMap बनाती थीं और इसका उपयोग उस डेटा को स्टोर करने के लिए करती थीं जो इंस्टेंस पर सुलभ नहीं होना चाहिए।

```javascript
const privateData = new WeakMap();

class User {
  constructor(name) {
    privateData.set(this, { name });
  }

  getName() {
    return privateData.get(this).name;
  }
}
```

जब कोई User इंस्टेंस Garbage Collect होता है, तो उसका प्राइवेट डेटा भी उसके साथ चला जाता है। किसी क्लीनअप कोड की ज़रूरत नहीं होती।

मेमोइज़ेशन (Memoization) एक और बेहतरीन फिट है, खासकर जब आप प्रिमिटिव वैल्यूज़ के बजाय ऑब्जेक्ट इनपुट के आधार पर परिणामों को कैश कर रहे हों। यदि आपकी महंगी गणना (expensive calculation) इनपुट के रूप में एक कॉन्फ़िग ऑब्जेक्ट लेती है, तो WeakMap का मतलब है कि आपको कैशे के कॉन्फ़िग से अधिक समय तक जीवित रहने की चिंता करने की ज़रूरत नहीं है।

```javascript
const cache = new WeakMap();

function expensiveCalc(obj) {
  if (cache.has(obj)) return cache.get(obj);
  
  const result = heavyMath(obj);
  cache.set(obj, result);
  return result;
}
```

कैशे केवल तब तक जीवित रहता है जब तक कैश किए जा रहे ऑब्जेक्ट जीवित रहते हैं। एक बार जब `obj` का कहीं और संदर्भ नहीं रहता, तो कैश किया गया परिणाम और कैशे प्रविष्टि दोनों एक साथ गायब हो जाते हैं।

## इसे कब इस्तेमाल करें

आधुनिक वेब ऐप्स में मेमोरी लीक आमतौर पर उन चीजों के बासी संदर्भों (stale references) से आते हैं जिन्हें साफ कर दिया जाना चाहिए था। यदि आप कुछ ऐसा बना रहे हैं जो लंबे समय तक चलता है - एक डैशबोर्ड जो पूरे दिन खुला रहता है, एक चैट ऐप जो घंटों चलता है, एक एडमिन पैनल जो कभी रिफ्रेश नहीं होता - तो आपको यह सोचने की ज़रूरत है कि पुराने डेटा का क्या होता है।

WeakMap विशेष रूप से तब उपयोगी होता है जब आप DOM नोड्स, कॉम्पोनेंट इंस्टेंस, या किसी ऐसे ऑब्जेक्ट के साथ डेटा जोड़ रहे होते हैं जिसका जीवनकाल आपके नियंत्रण में नहीं है। यदि आप किसी संदर्भ के आधार पर कुछ स्टोर कर रहे हैं और वह संदर्भ गायब हो सकता है, तो WeakMap सफाई को बहुत सरल बना देता है।

नियमित Map अभी भी सही विकल्प है जब आप बेदखली नीतियों (eviction policies) के साथ एक वास्तविक कैशे बना रहे हों, जब आपको प्रविष्टियों पर इटरेशन करने की आवश्यकता हो, जब आप प्रिमिटिव कुंजियों का उपयोग कर रहे हों, या जब डेटा स्वयं मायने रखता हो न कि किसी ऑब्जेक्ट के साथ उसका जुड़ाव।

`WeakMap` के बारे में अच्छी बात यह है कि आमतौर पर यह स्पष्ट होता है कि आपको इसकी आवश्यकता कब है। यदि आप ऑब्जेक्ट के नष्ट होने पर मैप प्रविष्टियों को हटाने के लिए क्लीनअप कोड लिख रहे हैं, तो यह एक संकेत है। यदि आप मेमोरी के अनियंत्रित रूप से बढ़ने को लेकर चिंतित हैं क्योंकि आप सुनिश्चित नहीं हैं कि चीजों को कब हटाना है, तो यह एक और संकेत है।

कभी-कभी सबसे अच्छी विशेषता वह होती है जो बिना आपके सोचे ही काम कर जाती है।
