क्या आपको वह एहसास याद है जब आप एक लाइन कोड बदलते हैं और अपनी मेमोरी उपयोग को 50% गिरते हुए देखते हैं? मुझे वह पल तब मिला जब मैं Chrome के DevTools Performance Monitor को देख रहा था और एक डैशबोर्ड ऐप हर घंटे 100MB खोने से लेकर पूरे दोपहर तक साफ चल रहा था।

एक लाइन का बदलाव: `new Map()` became `new WeakMap()`.

बस इतना ही। वही API सरफेस, वही उपयोग पैटर्न, हुड के नीचे पूरी तरह से अलग व्यवहार। लेकिन यह समझना कि यह क्यों काम करता है, इसका मतलब है कि कुछ ऐसा समझना जिसके बारे में अधिकांश JavaScript डेवलपर्स कभी नहीं सोचते: जब कुछ भी आपके डेटा को नहीं देख रहा होता तब क्या होता है।

## जब रेफरेंसेस एंकर बन जाते हैं

JavaScript में एक सामान्य Map अपनी कुंजियों को कीमती माल की तरह मानता है। एक बार जब आप कुछ उसमें डाल देते हैं, Map उसे लोहे की पकड़ से पकड़े रखेगा। Garbage Collector यह रिश्ता देखता है और सोचता है, "स्पष्ट रूप से उन्हें अभी भी इस ऑब्जेक्ट की जरूरत है, बेहतर है कि इसे न छुएं।"

यह सुरक्षात्मक स्वभाव तब समस्या बन जाता है जब आप अस्थायी चीजों के बारे में मेटाडेटा संग्रहीत कर रहे हों। DOM नोड्स जो हटा दिए जाते हैं। उपयोगकर्ता सत्र जो समाप्त हो जाते हैं। कंपोनेंट इंस्टेंस जो अनमाउंट हो जाते हैं। Map को नहीं पता कि ये ऑब्जेक्ट्स उपयोगी होना बंद कर चुके हैं। उसे बस इतना पता है कि उसके पास एक रेफरेंस है, इसलिए वह उन्हें जीवित रखता है।

```javascript
const cache = new Map();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// The element is gone from the DOM, but cache is keeping it in memory
```

Garbage Collector `element` को साफ नहीं कर सकता क्योंकि `cache` अभी भी उसकी ओर इशारा कर रहा है। इसे "स्ट्रांग रेफरेंस" कहा जाता है, और लंबे समय तक चलने वाले Single Page Apps में, यह एक लीक बन जाता है जो अंततः ब्राउज़र को क्रैश कर देता है।

## WeakMap नियम बदल देता है

WeakMap अलग तरह से काम करता है। यह अपनी कुंजियों को स्थायी निवासियों के बजाय अस्थायी नागरिकों के रूप में मानता है। जब आप कुछ WeakMap में संग्रहीत करते हैं, तो आप मूल रूप से यह कह रहे हैं: "मैं इस डेटा को इस ऑब्जेक्ट के साथ जोड़ना चाहता हूं, लेकिन मैं उस कारण नहीं बनना चाहता जिससे यह जीवित रहे।"

यदि किसी ऑब्जेक्ट को मेमोरी में रखने वाली एकमात्र चीज WeakMap है, तो Garbage Collector को इसे लेने की अनुमति है। जब ऑब्जेक्ट गायब हो जाता है, WeakMap एंट्री भी उसके साथ गायब हो जाती है। कोई मैनुअल सफाई की आवश्यकता नहीं।

```javascript
const cache = new WeakMap();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// The element gets Garbage Collected
// The cache entry vanishes automatically
```

मैंने एक बेंचमार्क चलाया जिसमें 100,000 DOM नोड्स बनाए, प्रत्येक के बारे में मेटाडेटा संग्रहीत किया, फिर उन सभी को हटा दिया। Map के साथ, ब्राउज़र ने 150-200MB पकड़कर रखा। WeakMap के साथ, यह 70-80MB तक गिर गया। वही कोड, वही कार्यक्षमता, आधी मेमोरी फुटप्रिंट।

## आप क्या छोड़ देते हैं

WeakMap में कुछ बंधन हैं जो सीमाओं की तरह महसूस होते हैं जब तक आपको एहसास नहीं होता कि ये वही हैं जो जादू काम करते हैं।

**आप WeakMap पर इटरेट नहीं कर सकते।** कोई `forEach`, कोई `keys()`, कोई `values()`। जब आप इसके बारे में सोचते हैं तो यह समझ में आता है: Garbage Collector आपके लूप के बीच में एक एंट्री हटा सकता है। क्या आप वास्तव में उससे निपटना चाहते हैं?

आप आकार की जांच नहीं कर सकते। कोई `.size` प्रॉपर्टी, कोई `.length`। फिर से, यह एक चलता हुआ लक्ष्य है। संख्या आपके पूछने और जवाब पाने के बीच बदल सकती है।

**कुंजियाँ ऑब्जेक्ट्स होनी चाहिए।** कोई स्ट्रिंग्स, कोई नंबर, कोई प्रिमिटिव्स। यह कमजोर रेफरेंसेस के काम करने के तरीके के लिए मौलिक है: प्रिमिटिव मानों की कोई पहचान नहीं होती जो उनके मान से अलग से ट्रैक की जा सके।

ये बग नहीं हैं। ये डिज़ाइन हैं। WeakMap एक विशिष्ट काम के लिए बनाया गया है: ऑब्जेक्ट्स पर मेटाडेटा जोड़ना बिना उन ऑब्जेक्ट्स को साफ किए जाने से रोके। यदि आपको इटरेशन या प्रिमिटिव कुंजियों या एंट्री की गिनती की आवश्यकता है, तो आप शायद एक अलग समस्या हल कर रहे हैं और एक सामान्य Map का उपयोग करना चाहिए।

## यह वास्तव में कहाँ मदद करता है

"प्राइवेट डेटा" पैटर्न WeakMap का मूल उपयोग केस था, JavaScript में `#private` फ़ील्ड्स होने से पहले। लाइब्रेरीज़ क्लास के बाहर एक WeakMap बनाती थीं और इसका उपयोग उस डेटा को संग्रहीत करने के लिए करती थीं जो इंस्टेंस पर सुलभ नहीं होना चाहिए।

```javascript
const privateData = new WeakMap();

class User {
  constructor(name) {
    privateData.set(this, { name });
  }

  getName() {
    return privateData.get(this).name;
  }
}
```

जब एक User इंस्टेंस Garbage Collected हो जाता है, प्राइवेट डेटा भी उसके साथ चला जाता है। कोई सफाई कोड की आवश्यकता नहीं।

मेमोइज़ेशन एक और प्राकृतिक फिट है, विशेष रूप से जब आप प्रिमिटिव मानों के बजाय ऑब्जेक्ट इनपुट्स के आधार पर परिणामों को कैश कर रहे हों। यदि आपकी महंगी गणना एक कॉन्फ़िग ऑब्जेक्ट को इनपुट के रूप में लेती है, तो WeakMap का अर्थ है कि आपको कैश के कॉन्फ़िग्स से जीवित रहने की चिंता नहीं करनी होगी।

```javascript
const cache = new WeakMap();

function expensiveCalc(obj) {
  if (cache.has(obj)) return cache.get(obj);
  
  const result = heavyMath(obj);
  cache.set(obj, result);
  return result;
}
```

कैश तब तक जीवित रहता है जब तक कैश किए जा रहे ऑब्जेक्ट्स जीवित हैं। एक बार `obj` कहीं और रेफरेंस नहीं रहता, तो कैश किया गया परिणाम और कैश एंट्री दोनों एक साथ गायब हो जाते हैं।

## इसे कब उपयोग करें

आधुनिक वेब ऐप्स में मेमोरी लीक्स आमतौर पर उन चीजों के स्टेल रेफरेंसेस से आते हैं जिन्हें साफ कर दिया जाना चाहिए था। यदि आप कुछ भी लंबे समय तक चलने वाला बना रहे हैं, एक डैशबोर्ड जो पूरे दिन खुला रहता है, एक चैट ऐप जो घंटों तक चलता है, एक एडमिन पैनल जो कभी रिफ्रेश नहीं होता, आपको यह सोचने की जरूरत है कि पुराने डेटा का क्या होता है।

WeakMap विशेष रूप से उपयोगी है जब आप DOM नोड्स, कंपोनेंट इंस्टेंस, या किसी भी ऑब्जेक्ट के साथ डेटा जोड़ रहे हों जिसकी लाइफटाइम आप नियंत्रित नहीं करते। यदि आप किसी रेफरेंस के आधार पर कुछ संग्रहीत कर रहे हैं और वह रेफरेंस जा सकता है, WeakMap सफाई को बहुत आसान बना देता है।

सामान्य Map अभी भी सही विकल्प है जब आप वास्तविक eviction नीतियों के साथ एक कैश बना रहे हों, जब आपको एंट्री पर इटरेट करने की आवश्यकता हो, जब आप प्रिमिटिव कुंजियों का उपयोग कर रहे हों, या जब डेटा स्वयं मायने रखता हो न कि इसका किसी ऑब्जेक्ट के साथ संबंध।

`WeakMap` के बारे में अच्छी बात यह है कि यह आमतौर पर स्पष्ट होता है कि आपको इसकी आवश्यकता कब है। यदि आपको क्लीनअप कोड लिखते हुए पाते हैं जो ऑब्जेक्ट्स के नष्ट होने पर Map एंट्री हटाता है, तो यह एक संकेत है। यदि आप चिंतित हैं कि मेमोरी अनबाउंडेड बढ़ रही है क्योंकि आपको निश्चित नहीं है कि चीजों को कब हटाना है, तो यह एक और संकेत है।

कभी-कभी सबसे अच्छी सुविधा वह होती है जो बिना सोचे काम करती है।
