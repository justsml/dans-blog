---

1行コードを変えただけで、メモリ使用量が50%減る感覚を知っていますか？ダッシュボードアプリが1時間ごとに100MBも垂れ流していたのが、ChromeのDevToolsパフォーマンスモニターで見ている間に、午後ずっとクリーンに動作するようになった瞬間を私は目撃しました。

その1行の変更とは、`new Map()` を `new WeakMap()` に変えたこと。

それだけです。APIの表面も使い方も同じですが、内部の挙動はまったく異なります。なぜこれが機能するのかを理解するには、ほとんどのJavaScript開発者が普段意識しないこと、つまり「データがどこからも見られなくなったときに何が起きるか」を理解する必要があります。

## 参照がアンカーになるとき

JavaScriptの通常の `Map` は、キーを貴重な貨物のように扱います。一度何かを入れると、`Map` はそれを鉄の握力で保持し続けます。ガベージコレクター（GC）はこの関係を見て、「明らかにまだこのオブジェクトが必要なのだな、触れないでおこう」と判断します。

この保護本能は、一時的なものに関するメタデータを保存している場合に問題となります。削除されたDOMノード、期限切れのユーザーセッション、アンマウントされたコンポーネントインスタンスなどです。`Map` はそれらのオブジェクトがもう役に立たなくなったことを知りません。ただ参照を持っていることだけを知っているので、それらを生かし続けます。

```javascript
const cache = new Map();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// 要素はDOMから消えましたが、cacheがメモリ内に保持し続けています
```

ガベージコレクターは `element` をクリーンアップできません。なぜなら `cache` がまだそれを指し示しているからです。これは「強い参照（Strong Reference）」と呼ばれ、長時間実行されるシングルページアプリ（SPA）では、最終的にブラウザをクラッシュさせるメモリリークの原因となります。

## WeakMapがルールを変える

`WeakMap` の仕組みは異なります。キーを永久の居住者ではなく、一時的な市民として扱います。`WeakMap` に何かを保存するとき、あなたは本質的にこう言っているのです。「このデータをこのオブジェクトに関連付けたいけれど、私がそのオブジェクトを生かし続ける理由にはなりたくない」と。

オブジェクトをメモリに保持している唯一のものが `WeakMap` である場合、ガベージコレクターはそのオブジェクトを回収することが許されます。オブジェクトが消えると、`WeakMap` のエントリも一緒に消滅します。手動でのクリーンアップは不要です。

```javascript
const cache = new WeakMap();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// 要素はガベージコレクションされます
// キャッシュエントリは自動的に消滅します
```

10万個のDOMノードを作成し、それぞれのメタデータを保存してからすべて削除するというベンチマークを実行しました。`Map` の場合、ブラウザは150〜200MBを保持し続けました。`WeakMap` の場合、それは70〜80MBまで低下しました。同じコード、同じ機能で、メモリフットプリントは半分になりました。

## 引き換えに失うもの

`WeakMap` には制約がありますが、それこそが魔法を実現するための鍵であると気づくまでは「制限」のように感じられるかもしれません。

**WeakMapは反復処理（イテレーション）ができません。** `forEach` も `keys()` も `values()` もありません。これは理にかなっています。ループの途中でガベージコレクターがエントリを削除する可能性があるからです。そんな状況に対処したい人はいないでしょう。

**サイズを確認できません。** `.size` プロパティも `.length` もありません。これも、ターゲットが常に動いているためです。数値を問い合わせてから答えを受け取るまでの間に、その数値が変わってしまう可能性があるからです。

**キーはオブジェクトである必要があります。** 文字列や数値、プリミティブは使えません。これは「弱い参照」の仕組みの根本に関わります。プリミティブ値には、その値自体と切り離して追跡できる「アイデンティティ（同一性）」がないためです。

これらはバグではなく、設計です。`WeakMap` は特定の仕事、つまり「オブジェクトのクリーンアップを妨げることなく、オブジェクトにメタデータを付与する」ために作られています。反復処理やプリミティブなキー、エントリのカウントが必要な場合は、おそらく別の問題を解こうとしているので、通常の `Map` を使うべきです。

## 実際に役立つ場面

「プライベートデータ」パターンは、JavaScriptに `#private` フィールドが登場する前の `WeakMap` の元々のユースケースでした。ライブラリはクラスの外側に `WeakMap` を作成し、インスタンスからアクセスできないデータを保存するためにそれを使用していました。

```javascript
const privateData = new WeakMap();

class User {
  constructor(name) {
    privateData.set(this, { name });
  }

  getName() {
    return privateData.get(this).name;
  }
}
```

`User` インスタンスがガベージコレクションされると、プライベートデータも一緒に消えます。クリーンアップ用のコードは必要ありません。

「メモ化（Memoization）」も相性が良く、特にプリミティブ値ではなくオブジェクトの入力に基づいて結果をキャッシュする場合に最適です。高コストな計算が設定オブジェクトを引数に取る場合、`WeakMap` を使えばキャッシュが設定オブジェクトより長生きしてしまう心配をする必要がありません。

```javascript
const cache = new WeakMap();

function expensiveCalc(obj) {
  if (cache.has(obj)) return cache.get(obj);
  
  const result = heavyMath(obj);
  cache.set(obj, result);
  return result;
}
```

キャッシュは、キャッシュされているオブジェクトが生きている間だけ存在します。`obj` がどこからも参照されなくなれば、キャッシュされた結果もキャッシュエントリも一緒に消滅します。

## いつ使うべきか

現代のウェブアプリにおけるメモリリークは、通常、クリーンアップされるべきものへの古い参照が残っていることに起因します。一日中開いたままのダッシュボード、何時間も動かし続けるチャットアプリ、リフレッシュされることのない管理パネルなどを構築しているなら、古いデータがどうなるかを考える必要があります。

`WeakMap` は、DOMノードやコンポーネントインスタンスなど、自分では寿命をコントロールできないオブジェクトにデータを関連付けるときに特に便利です。参照に基づいて何かを保存していて、その参照がなくなる可能性がある場合、`WeakMap` はクリーンアップを大幅に簡素化してくれます。

一方、削除ポリシー（エビクションポリシー）を持つ本格的なキャッシュを構築する場合、エントリを反復処理する必要がある場合、プリミティブなキーを使用する場合、あるいはオブジェクトとの関連付けではなくデータそのものが重要である場合は、通常の `Map` が依然として正しい選択です。

`WeakMap` の良いところは、それが必要なときは大抵すぐにわかることです。オブジェクトが破棄されたときにマップのエントリを削除するクリーンアップコードを書いている自分に気づいたら、それが合図です。いつ削除すればいいかわからないためにメモリが際限なく増えていくのが心配なら、それもまた合図です。

時として、最高の機能とは、何も考えなくても「ただ正しく動く」ものなのです。
