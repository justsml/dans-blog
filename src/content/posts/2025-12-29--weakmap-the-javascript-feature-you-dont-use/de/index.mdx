Kennt man das Gefühl, wenn man eine einzige Codezeile ändert und zusehen kann, wie der Speicherverbrauch um 50 % sinkt? Ich hatte diesen Moment beim Beobachten von Chromes DevTools Performance Monitor – eine Dashboard-Anwendung, die vorher 100 MB pro Stunde verschleuderte, lief plötzlich den ganzen Nachmittag sauber durch.

Die eine Zeile: `new Map()` wurde zu `new WeakMap()`.

Das war's. Gleiche API-Oberfläche, gleiches Nutzungsmuster, aber völlig anderes Verhalten unter der Haube. Zu verstehen, warum das funktioniert, bedeutet jedoch, etwas zu begreifen, woran die meisten JavaScript-Entwickler nie denken: Was passiert, wenn nichts mehr auf Ihre Daten schaut.

## Wenn Referenzen zu Ankern werden

Ein normales Map in JavaScript behandelt seine Schlüssel wie wertvolle Fracht. Sobald Sie etwas dort hineinlegen, hält das Map es mit eisernem Griff fest. Der Garbage Collector sieht diese Beziehung und denkt sich: „Offensichtlich brauchen sie dieses Objekt noch, besser fasse ich es nicht an."

Dieser Schutzinstinkt wird zum Problem, wenn Sie Metadaten über temporäre Dinge speichern. DOM-Knoten, die entfernt werden. Benutzersitzungen, die ablaufen. Komponenteninstanzen, die unmounten. Das Map weiß nicht, dass diese Objekte ihren Dienst erfüllt haben. Es weiß nur, dass es eine Referenz hat, also hält es sie am Leben.

```javascript
const cache = new Map();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// Das Element ist aus dem DOM verschwunden, aber der Cache hält es im Speicher
```

Der Garbage Collector kann `element` nicht aufräumen, weil `cache` immer noch darauf verweist. Das nennt man eine „starke Referenz", und in langlaufenden Single-Page-Apps wird sie zu einem Leck, das schließlich den Browser zum Absturz bringt.

## WeakMap ändert die Spielregeln

Ein WeakMap funktioniert anders. Es behandelt seine Schlüssel als vorübergehende Bewohner statt als permanente. Wenn Sie etwas in einem WeakMap speichern, sagen Sie im Grunde: „Ich möchte diese Daten mit diesem Objekt verknüpfen, aber ich will nicht der Grund sein, warum es am Leben bleibt."

Wenn das einzige, das ein Objekt im Speicher hält, ein WeakMap ist, darf der Garbage Collector es entfernen. Wenn das Objekt verschwindet, verschwindet auch der WeakMap-Eintrag. Keine manuelle Aufräumaktion nötig.

```javascript
const cache = new WeakMap();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// Das Element wird vom Garbage Collector eingesammelt
// Der Cache-Eintrag verschwindet automatisch
```

Ich habe einen Benchmark durchgeführt, bei dem 100.000 DOM-Knoten erstellt, Metadaten zu jedem gespeichert und dann alle entfernt wurden. Mit einem Map hielt der Browser 150–200 MB fest. Mit einem WeakMap sank der Wert auf 70–80 MB. Gleicher Code, gleiche Funktionalität, halber Speicherbedarf.

## Was Sie aufgeben

WeakMap hat Einschränkungen, die sich wie Nachteile anfühlen – bis man erkennt, dass sie erst die Magie möglich machen.

**Sie können nicht über ein WeakMap iterieren.** Kein `forEach`, keine `keys()`, keine `values()`. Das ergibt Sinn, wenn man darüber nachdenkt: Der Garbage Collector könnte mitten in Ihrer Schleife einen Eintrag löschen. Wollen Sie wirklich damit umgehen?

Sie können die Größe nicht prüfen. Keine `.size`-Eigenschaft, kein `.length`. Auch hier ist es ein bewegliches Ziel. Die Zahl könnte sich zwischen der Frage und der Antwort ändern.

**Schlüssel müssen Objekte sein.** Keine Strings, keine Zahlen, keine Primitives. Das ist grundlegend für die Funktionsweise schwacher Referenzen: Primitive Werte haben keine Identität, die unabhängig von ihrem Wert verfolgt werden kann.

Das sind keine Fehler. Das ist das Design. WeakMap ist für eine einzige Aufgabe gebaut: Metadaten an Objekte zu heften, ohne deren Bereinigung zu verhindern. Wenn Sie Iteration, primitive Schlüssel oder eine Zählung der Einträge benötigen, lösen Sie wahrscheinlich ein anderes Problem und sollten ein normales Map verwenden.

## Wo das wirklich hilft

Das „Private-Daten"-Muster war der ursprüngliche Anwendungsfall für WeakMap, zurück aus der Zeit bevor JavaScript `#private`-Felder hatte. Bibliotheken erstellten ein WeakMap außerhalb der Klasse und nutzten es, um Daten zu speichern, die auf der Instanz nicht zugänglich sein sollten.

```javascript
const privateData = new WeakMap();

class User {
  constructor(name) {
    privateData.set(this, { name });
  }

  getName() {
    return privateData.get(this).name;
  }
}
```

Wenn eine User-Instanz vom Garbage Collector eingesammelt wird, geht die privaten Daten mit ihr. Kein Aufräumcode nötig.

Memoisierung ist ein weiterer natürlicher Einsatzfall, besonders wenn Sie Ergebnisse auf Basis von Objekteingaben statt primitiver Werte zwischenspeichern. Wenn Ihre aufwendige Berechnung ein Konfigurationsobjekt als Eingabe nimmt, bedeutet ein WeakMap, dass Sie sich keine Gedanken darüber machen müssen, dass der Cache die Konfigurationen überlebt.

```javascript
const cache = new WeakMap();

function expensiveCalc(obj) {
  if (cache.has(obj)) return cache.get(obj);
  
  const result = heavyMath(obj);
  cache.set(obj, result);
  return result;
}
```

Der Cache existiert nur so lange wie die zwischengespeicherten Objekte. Sobald `obj` nirgendwo sonst mehr referenziert wird, verschwinden sowohl das zwischengespeicherte Ergebnis als auch der Cache-Eintrag gemeinsam.

## Wann Sie danach greifen sollten

Speicherlecks in modernen Web-Apps entstehen meist durch veraltete Referenzen auf Dinge, die längst aufgeräumt sein sollten. Wenn Sie etwas Langlaufendes bauen – ein Dashboard, den ganzen Tag geöffnet, eine Chat-App, die stundenlang läuft, ein Admin-Panel, das sich nie neu lädt –, müssen Sie sich Gedanken darüber machen, was mit alten Daten passiert.

WeakMap ist besonders nützlich, wenn Sie Daten mit DOM-Knoten, Komponenteninstanzen oder einem beliebigen Objekt verknüpfen, dessen Lebensdauer Sie nicht kontrollieren. Wenn Sie etwas basierend auf einer Referenz speichern und diese Referenz möglicherweise verschwindet, macht WeakMap die Bereinigung viel einfacher.

Ein normales Map ist weiterhin die richtige Wahl, wenn Sie einen echten Cache mit Eviction-Richtlinien aufbauen, wenn Sie über Einträge iterieren müssen, wenn Sie primitive Schlüssel verwenden oder wenn die Daten selbst das Wichtige sind und nicht ihre Verknüpfung mit einem Objekt.

Das Schöne an `WeakMap` ist, dass es meistens offensichtlich ist, wenn man es braucht. Wenn Sie feststellen, dass Sie Aufräumcode schreiben, um Map-Einträge zu entfernen, wenn Objekte zerstört werden, ist das ein Zeichen. Wenn Sie sich Sorgen machen, dass der Speicher unbegrenzt wächst, weil Sie nicht wissen, wann Sie etwas löschen sollen, ist das ein weiteres Zeichen.

Manchmal ist das beste Feature eines, das einfach funktioniert, ohne dass man darüber nachdenken muss.
