Conosci quella sensazione quando cambi una sola riga di codice e vedi l'uso della memoria crollare del 50%? L'ho provata guardando il Performance Monitor di Chrome DevTools, mentre un'app dashboard passava dal perdere 100 MB all'ora a funzionare pulita per un intero pomeriggio.

La modifica? `new Map()` è diventato `new WeakMap()`.

Tutto qui. Stessa superficie API, stesso pattern d'uso, comportamento completamente diverso sotto il cofano. Ma capire perché funziona significa comprendere qualcosa a cui la maggior parte degli sviluppatori JavaScript non pensa mai: cosa succede quando nessuno guarda più i tuoi dati.

## Quando i Riferimenti Diventano Ancore

Una `Map` normale in JavaScript tratta le proprie chiavi come un carico prezioso. Una volta inserito qualcosa, la Map se lo tiene stretto con una presa d'acciaio. Il Garbage Collector vede questa relazione e pensa: "Chiaramente gli serve ancora questo oggetto, meglio non toccarlo."

Questo istinto protettivo diventa un problema quando memorizzi metadati su cose temporanee. Nodi DOM che vengono rimossi. Sessioni utente che scadono. Istanze di componenti che smontano. La Map non sa che questi oggetti hanno finito il loro compito. Sa solo di avere un riferimento, quindi li tiene in vita.

```javascript
const cache = new Map();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// L'elemento non è più nel DOM, ma cache lo mantiene in memoria
```

Il Garbage Collector non può ripulire `element` perché `cache` punta ancora a esso. Si chiama "riferimento forte", e nelle Single Page App a lunga esecuzione diventa una perdita di memoria che prima o poi manda in crash il browser.

## WeakMap Cambia le Regole

Una `WeakMap` funziona in modo diverso. Tratta le proprie chiavi come cittadini temporanei anziché residenti permanenti. Quando memorizzi qualcosa in una WeakMap, stai essenzialmente dicendo: "Voglio associare questi dati a questo oggetto, ma non voglio essere il motivo per cui resta in vita."

Se l'unica cosa che tiene un oggetto in memoria è una WeakMap, il Garbage Collector è autorizzato a rimuoverlo. Quando l'oggetto scompare, la voce nella WeakMap scompare con lui. Nessuna pulizia manuale necessaria.

```javascript
const cache = new WeakMap();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// L'elemento viene deallocato dal Garbage Collector
// La voce nella cache scompare automaticamente
```

Ho eseguito un benchmark creando 100.000 nodi DOM, memorizzando metadati per ciascuno, e poi rimuovendoli tutti. Con una Map, il browser tratteneva 150-200 MB. Con una WeakMap, scendeva a 70-80 MB. Stesso codice, stessa funzionalità, metà dell'impronta di memoria.

## Cosa Rinunci

WeakMap ha dei vincoli che sembrano limitazioni, finché non capisci che sono proprio ciò che rende possibile la magia.

**Non puoi iterare su una WeakMap.** Niente `forEach`, niente `keys()`, niente `values()`. Ha senso se ci pensi: il Garbage Collector potrebbe eliminare una voce a metà del tuo ciclo. Ti andrebbe di gestire una situazione del genere?

Non puoi verificarne la dimensione. Niente proprietà `.size`, niente `.length`. Anche qui, è un bersaglio mobile: il numero potrebbe cambiare tra il momento in cui lo chiedi e quello in cui ottieni la risposta.

**Le chiavi devono essere oggetti.** Niente stringhe, niente numeri, niente primitivi. È un punto fondamentale per come funzionano i riferimenti deboli: i valori primitivi non hanno un'identità separata dal loro valore che possa essere tracciata.

Questi non sono bug. È il design. WeakMap è pensato per un solo compito: associare metadati a oggetti senza impedire che vengano deallocati. Se ti serve iterare, usare chiavi primitive o contare le voci, probabilmente stai risolvendo un problema diverso e dovresti usare una Map normale.

## Dove Serve Davvero

Il pattern dei "dati privati" era il caso d'uso originario di WeakMap, prima che JavaScript introducesse i campi `#private`. Le librerie creavano una WeakMap esterna alla classe e la usavano per memorizzare dati che non dovevano essere accessibili dall'istanza.

```javascript
const privateData = new WeakMap();

class User {
  constructor(name) {
    privateData.set(this, { name });
  }

  getName() {
    return privateData.get(this).name;
  }
}
```

Quando un'istanza di `User` viene deallocata dal Garbage Collector, i dati privati scompaiono con lei. Nessun codice di pulizia richiesto.

La memoizzazione è un altro impiego naturale, specialmente quando metti in cache risultati calcolati a partire da oggetti anziché da valori primitivi. Se la tua funzione costosa riceve un oggetto di configurazione come input, una WeakMap significa non doverti preoccupare che la cache sopravviva ai config stessi.

```javascript
const cache = new WeakMap();

function expensiveCalc(obj) {
  if (cache.has(obj)) return cache.get(obj);
  
  const result = heavyMath(obj);
  cache.set(obj, result);
  return result;
}
```

La cache vive solo finché vivono gli oggetti in cache. Quando `obj` non è più referenziato da nessuna parte, sia il risultato messo in cache che la voce nella cache scompaiono insieme.

## Quando Sceglierla

Le perdite di memoria nelle web app moderne di solito derivano da riferimenti obsoleti a cose che avrebbero dovuto essere ripulite. Se stai costruendo qualcosa che gira a lungo — una dashboard aperta tutto il giorno, un'app di chat attiva per ore, un pannello admin che non si ricarica mai — devi pensare a cosa succede ai dati vecchi.

WeakMap è particolarmente utile quando associ dati a nodi DOM, istanze di componenti, o qualsiasi oggetto il cui ciclo di vita non controlli. Se memorizzi qualcosa in base a un riferimento e quel riferimento potrebbe sparire, WeakMap rende la pulizia molto più semplice.

Una Map normale resta la scelta giusta quando costruisci una cache con politiche di evizione vere e proprie, quando devi iterare sulle voci, quando usi chiavi primitive, o quando i dati in sé sono ciò che conta — non la loro associazione con un oggetto.

La cosa bella di `WeakMap` è che di solito è evidente quando ti serve. Se ti ritrovi a scrivere codice di pulizia per rimuovere voci dalla map quando gli oggetti vengono distrutti, è un segnale. Se ti preoccupi che la memoria cresca senza limiti perché non sai quando cancellare le cose, è un altro segnale.

A volte la caratteristica migliore è quella che funziona senza che tu debba pensarci.
