הדמו של RAG תמיד נראה מרהיב.

אתה מטמיע כמה מסמכים, מפעיל חנות וקטורים, מחבר אותה לצ'אט, וצופה במודל מצטט את מאגר הידע הפנימי שלך כאילו הוא עובד שם כבר שנים. יפהפה. בעלי העניין מתרשמים. מישהו אומר "בואו נשחרר את זה."

שישה שבועות אחרי, המשתמשים מקבלים תשובות בטוחות אך שגויות. כרטיסי תמיכה מצטברים. המערכת *פועלת*, רק לא באופן שהאנשים צריכים אותה.

הכשלון בדרך כלל אינו טעות דרמטית אחת. מדובר בחמישה כשלונות משעממים שמצטברים יחד.

---

## 1. החלקים שלך בגודל הלא נכון

הכשלון הזה לא גורם לקריסה. הוא רק מחליש כל תשובה במקצת עד שהפיצ'ר כולו מרגיש לא אמין.

חיפוש וקטורי מחזיר *חלקים*, לא מסמכים. כל מה שאתה מחלק את חומר המקור שלך אליו הופך ליחידת האמת של המאתר. אם החלקים אינם נכונים, המודל משיב עם הקטעים השגויים.

**קטן מדי**: החלק מכיל תשובה חלקית. ההטמעה קולטת את הנושא הנכון, אך הטקסט שהתקבל חסר הקשר. אתה מושך "הזמן המרבי הוא 30 שניות" בלי המשפט שלפניו שאומר "כאשר משתמשים ב‑API הישן."

**גדול מדי**: ההטמעה הופכת לממוצע מטושטש של רעיונות רבים. חיפוש סמנטי מתבלבל מכיוון שהחלק מתייחס למספר נושאים, והווקטור המתקבל אינו מייצג אף אחד מהם בצורה נקייה.

גודל הקטע הנכון תלוי לחלוטין בתוכן שלך. תיעוד טכני, חוזים משפטיים ותמלילי תמיכה מחולקים לחלקים באופן שונה. אין תשובה אוניברסלית.

מה לעשות: למדוד. בנה קבוצת הערכה של זוגות שאלה/תשובה מהמאגר שלך. נסה קטעים של 256, 512 ו‑1024 טוקנים. מדוד את דיוק האחזור: האם הקטע הנכון מופיע בחמישה הראשונים? תגלה מהר שהגודל של הקטע משפיע יותר מהמודל ההטמעה שהשקעת בו זמן.

גם השתמש בחפיפה. קטע של 512 טוקנים עם 64 טוקנים חפיפה בכל צד מאפשר לתשובות החוצות גבולות עדיין להיאחז. רוב ספריות הווקטורים תומכות בזה. רוב האנשים מדלגים על זה.

## 2. ההטמעות שלך מתיישנות (ולא תשים לב)

דמיין שהחברה שלך משנה שם. או משנה שם מוצר. או מעדכנת את תמחוריה. או מפסיקה להשתמש ב‑API.

אתה מעדכן את התיעוד אך לא מבצע הטמעה מחדש של הקטעים. אינדקס הווקטורים עדיין מייצג את התוכן הישן.

משתמשים שואלים על התמחור החדש. ההטמעות מכוונות אותם לתוכן הישן. המודל קורא את התוכן הישן ומסביר בביטחון את התמחור הישן. התמיכה מקבלת כרטיס.

כל מערכת RAG רצינית נתקלת בזה בסופו של דבר. הפתרון נשמע ברור – לבצע הטמעה מחדש כשמתעדכן תוכן – אך צוותים כמעט ולא בונים את הצינור הזה לפני האירוע הראשון.

אתה צריך אינדקס מחדש אינקרמנטלי עם טביעת אצבע על תוכן:

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Content unchanged, skip re-embedding
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

העדכון מחדש בזמן כתיבה, טביעת אצבע על התוכן, לא על חותמות זמן. המסמכים במערכת ניהול התוכן מתעדכנים לעיתים קרובות מבלי שהטקסט בפועל משתנה.

## 3. דיוק של אחזור מול שליפה: אתה ממטב את הלא‑נכון

רוב מדריכי ה‑RAG מציגים איך לאחזר את ה‑K‑החלקים העליונים. הם לא מסבירים את הפשרה בין שני היעדים שפונים בכיוונים מנוגדים.

**שליפה גבוהה**: מחזירים כל מה שיכול להיות רלוונטי. המשתמשים תמיד מקבלים תשובה. אבל חלון ההקשר של המודל מתמלא ברעש משני, והוא מתחיל להמציא כדי למלא את הפערים בין הקטעים.

**דיוק גבוה**: מחזירים רק את הקטעים הכי רלוונטיים. המודל עובד עם הקשר נקי וממוקד. אם הקטע הנכון לא נמצא בין שלושת הראשונים, המודל חסר מידע ויוצר תשובה משוערת בביטחון.

מצבי הכשל נראים זהים למשתמשים: תשובות שגויות. אבל הסיבות והפתרונות הם הפוכים.

שתי טכניקות שמביאות תועלת ממשית:

**דירוג מחדש**: מאחרים יותר מועמדים (top‑20), ואז משתמשים במודל cross‑encoder כדי לדרג אותם מחדש לפי רלוונטיות לפני העברתם ל‑LLM. מודלים מסוג cross‑encoder איטיים יותר משוויון וקטורי, אך מדויקים בהרבה בשלב הדירוג הסופי.

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // עכשיו top‑5 באמת משמעותי
```

**חיפוש היברידי**: משלב חיפוש וקטורי (דמיון סמנטי) עם חיפוש מילות‑מפתח (BM25). כל אחד נכשל בצורה שונה. חיפוש וקטורי מתקשה במונחים ספציפיים, שמות מודלים ו‑ID‑ים. חיפוש מילות‑מפתח מתקשה בפרפרזה ובמילים נרדפות. יחד הם מכסים את נקודות העיוור של השני.

## 4. חלון ההקשר שלך הוא בצורה הלא נכונה

קיבלת את החלקים הנכונים. מזל טוב. המודל עדיין יטעה.

הבעיה אינה רק מה שאתה משחזר. היא איפה שאתה שם את זה.

LLM‑ים יכולים לסבול מבעיה של “איבוד באמצע”. Liu ואחרים מדדו שמודלים עם הקשר ארוך משתמשים במידע רלוונטי פחות באופן אמין כאשר הוא מופיע באמצע הפרומפט במקום קרוב להתחלה או הסוף.

אם אתה דוחס 20 חלקים לרשימה שטוחה ומקווה שהמודל יסנתז אותם נכון, אתה משאיר ביצועים על השולחן.

דברים שבאמת עוזרים:

**הערך מיקום תחילה/סוף עבור החלקים הרלוונטיים ביותר שלך.** הנחה נפוצה היא שהחלק המדורג ראשונה יופיע ראשון, השני אחרון, והשאר במרכז. זה מנוגד לאינטואיציה, אבל שווה לבדוק מול המודל וצורת הפרומפט שלך.

**מספר ותיוג במפורש את מקטעי ההקשר שלך.** `[Source 1]` … `[Source 2]` נותן למודל עוגנים לחשוב עליהם.

**הוסף אות סיגנל של ביטחון השחזור.** אם ציון הדמיון שלך הוא 0.65 בקנה מידה של 0‑1, אמור למודל: “הקשר הבא שוחזר עם ביטחון מתון. הודה בחוסר וודאות אם התשובה אינה ברורה.”

**קבע תקציב הקשר.** אל תעביר פשוט את כל מה ששחזרת. ספור טוקנים, תעדף לפי ציון הרלוונטיות, וקצץ בחומרה ב‑60‑70% מחלון ההקשר של המודל. השאר מקום למודל לחשוב מבלי להיתקע.

Reference: [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172).

## 5. אין לך מושג מתי זה שגוי

זה הכשל השקט: התשובה חוזרת, הממשק נראה בריא, והתוכן שגוי.

ב‑API מסורתי, הכשלים נראים: HTTP 500, timeout, שגיאת אימות סכימה. אתה יודע מיד. כשלונות RAG שקטים יותר: המערכת מחזירה תגובה, היא נראית משכנעת, והיא שגויה.

אולי לא תדע שהצינור RAG שלך נכשל עד שהמשתמשים יגידו לך. לרוב הם לא עושים זאת. הם פשוט מפסיקים לסמוך עליו וחוזרים ל‑Ctrl+F.

הגדרת תצורת תצפית מינימלית ברת-קיימא למערכת RAG בתפעול:

**תעד את שרשרת השחזור שלך.** כל שאילתה, מה שנשחזר (מזהי מקטעים + ציונים), ומה שהמודל הפיק. אתה צריך זאת כדי לאתר באגים.

**עקוב אחרי מדדי השחזור.** דירוג ממוצע הפוך (MRR) ו‑NDCG אם יש לך תוויות אמתיות. לפחות, עקוב אחרי התפלגויות ציוני הדמיון — אם ציון ה‑P50 של השחזור יורד, איכות האינדקס ירדה.

**בנה לולאת משוב.** אפילו סימון של‑העלה/הורדה על תגובות, שמקושר חזרה לשאילתה ולמקטעים שהושגו, מספק איתות אימון. בלי זה, אתה טס בעיניים מכוסות.

**הפעל הערכות תקופתיות.** סט מבחן של 50‑100 שאלות עם תשובות נכונות ידועות, שמורץ שבועית, יזהה רגרסיה לפני שהמשתמשים יגלו. גליון אלקטרוני וסקריפט מספיקים כדי להתחיל.

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // Alert if score drops below threshold
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## הבעיה האמיתית

הכשלונות האלה אינם נובעים בעיקר ממודל ההטבעה או ממסד הנתונים הווקטורי. הם נובעים מהמערכת שמסביבם.

הדגמות RAG עובדות כי תנאי ההדגמה נשלטים: מסמכים נקיים, שאלות מובנות, מעריכים סובלניים. בייצור הכשלונות מתרחשים כי אף אחד מהתנאים האלה אינו מתקיים.

כל כשלון שהוזכר ניתן לאבחון, אך רק אם מודדים. צוותי RAG אמינים אינם עושים משהו יוצא דופן. הם מתייחסים לאיכות השחזור כמערכת משנית אמיתית ולא כפריט הדגמה.

הקם תחילה את לולאת ההערכה. כל השאר הופך לקל יותר ברגע שאתה יכול למדוד זאת.
