  <div className="question">
    איזו התנהגות מציג הרכיב React הזה כאשר לוחצים על הכפתור מספר פעמים?
    ```jsx
    function Counter() {
      const [count, setCount] = useState(0);

      useEffect(() => {
        const id = setInterval(() => {
          console.log('count is', count);
        }, 1000);
        return () => clearInterval(id);
      }, []); // empty deps

      return <button onClick={() => setCount(c => c + 1)}>
        Click ({count})
      </button>;
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    סגירה ישנה קלאסית. ה-`useEffect` רץ פעם אחת (מערך תלות ריק) ויוצר אינטרוול שסוגר על `count` כפי שהיה בזמן ההרצה: `0`. לא משנה כמה פעמים תלחץ, הסגירה של האינטרוול לעולם לא רואה את הערך המעודכן.

    תווית הכפתור מתעדכנת נכון — זה רינדור חדש. אבל `console.log` נלכד לעד ב-`0`.

    תקן זאת על‑ידי הוספת `count` למערך התלות (יוצר אינטרוול מחדש) או באמצעות ref:
    ```jsx
    const countRef = useRef(count);
    useEffect(() => { countRef.current = count; }, [count]);
    // then use countRef.current inside the interval
    ```
  </div>
  </slot>
</Challenge>

    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      element.addEventListener('click', () => {

      });
    }
    ```

    `element` נמצא ב‑DOM. המאזין מחזיק ב‑`data`. ה‑GC לא יכול לאסוף אותו.

    פתרון: אל תתפס נתונים גדולים במאזינים ארוכי‑חיים אלא אם באמת צריך, והסר מאזינים כאשר ממשק המשתמש שבבעלותם מתפרק:
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      // do something with data...

      const onClick = () => {
        console.log('clicked!');
      };

      element.addEventListener('click', onClick);
      return () => element.removeEventListener('click', onClick);
    }
    ```
    V8 מודרני משפר את ההתמודדות עם זה לעומת בעבר. אבל באפליקציות SPA ארוכות‑חיים עם הרבה מאזיני אירועים, תבנית זו עדיין גורמת לבעיות.
  </div>

  <div className="question">
    מה מודפס?
    ```javascript
    const user = {
      name: 'Dan',
      greetArrow: () => console.log(this?.name),
      greetRegular() {
        const inner = () => console.log(this?.name);
        inner();
      },
    };

    user.greetArrow();
    const fn = user.greetRegular;
    fn();
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    שתי מלכודות עדינות:

    **`greetArrow`**: פונקציות חץ אין להן `this` משלהן. הן לוכדות את `this` מההקשר הלקסי הסביבתי — כאן זה תחום המודול/הגלובלי, ולא `user`. ולכן `this?.name` הוא `undefined`.

    **`fn()`**: `greetRegular` נקראת ללא מקבל (`fn()` ולא `user.greetRegular()`). במצב strict, `this` הוא `undefined`. במצב sloppy, זה האובייקט הגלובלי. בכל מקרה, `this?.name` הוא `undefined`.

    אם היית קורא ישירות `user.greetRegular()` היית מקבל `'Dan'` — כי אז `this` קשור ל‑`user`.

    תוצאה: `undefined, undefined`.
  </div>
  </slot>
</Challenge>

    כמה פעמים בפועל נקראת הפונקציה `expensive()`?
    ```javascript
    function memoize(fn) {
      const cache = new Map();
      return (arg) => {
        if (cache.has(arg)) return cache.get(arg);
        const result = fn(arg);
        cache.set(arg, result);
        return result;
      };
    }

    let calls = 0;
    const expensive = (n) => { calls++; return n * 2; };
    const fast = memoize(expensive);

    fast(5);
    fast(5);
    fast(5);
    console.log(calls);
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    הפונקציה הממוזגת סוגרת על `cache`. הקריאה הראשונה (`fast(5)`) מחמיצה את המטמון, קוראת ל‑`expensive` ושומרת את התוצאה. הקריאות השנייה והשלישית פוגעות במטמון ומחזירות מיד.

    `calls` הוא `1`.

    זה סגירות שעושות עבודה אמיתית. `cache` פרטית (לא ניתן לגשת אליה מחוץ ל‑`memoize`), מתמשכת (חיה כל עוד `fast` קיימת), ומשותפת לכל הקריאות ל‑`fast`.

איך הלך?

- **9‑10**: אתה באמת מבין סגירות. כנראה מפחיד לשתף בתכנות זוגי.
- **6‑8**: חזק. שאלות הסגירה ה„stale“ / `this` הקשו עליך, וזה לגמרי הגיוני.
- **3‑5**: אתה יודע את הקונספט, אבל הקצוות מטושטשים. קרא את תיעוד שרשרת ההקשר (scope chain) ב‑MDN.
- **0‑2**: חשבת שאתה יודע סגירות. עכשיו אתה באמת יודע. זה הכוונה.

הסגירה ה‑stale ב‑React (שאלה #6) והמלכודת של `this` (שאלה #8) הן אלו שגורמות לבאגים אמיתיים במוצרים. אם הן הרגישו לך לא מוכרות, כדאי לחזור אליהן — הן מופיעות בביקורות קוד יותר מכל בעיית סגירה אחרת.
