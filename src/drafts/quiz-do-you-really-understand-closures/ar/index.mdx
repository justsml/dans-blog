<p class="inset">الإغلاقات هي أحد تلك المفاهيم التي يعتقد كل مطور جافاسكريبت أنه يفهمها — حتى اللحظة التي لا يفهمها فيها. هذا الاختبار سيكشف حدود ما تعرفه فعلاً.</p>

تظهر الإغلاقات في معالجات الأحداث، ودوال المصنع، وReact hooks، وأنماط الوحدات، وفي كل مكان تقريبًا. ليست غريبة. إنها النسيج الذي يُنسج منه جافاسكريبت.

    خمسة استدعاءات راجعة. متغيّر `i` واحد. القيمة: `5`. تُسجَّل خمس مرات.

    ```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100); // 0, 1, 2, 3, 4 ✓
    }
    ```

  </slot>

    هذا يُفاجئ الأشخاص الذين يظنون أن الإغلاقات "تجمّد" القيم. فهي لا تفعل ذلك. إنها تلتقط **المراجع**.

  client:visible={{rootMargin: "150px"}}
  index={5}
  group="متقدم"
  title="إغلاق قديم في React"
  options={[
    {text: 'يسجل العدد الحالي في كل نقرة'},
    {text: 'دائمًا يسجل 0', isAnswer: true},
    {text: 'يسجل NaN'},
    {text: 'هذا لن يُترجم'},
  ]}
>
  <slot name="question">
  <div className="question">
    ما السلوك الذي يظهره هذا المكوّن React عندما يتم النقر على الزر عدة مرات؟
    ```jsx
    function Counter() {
      const [count, setCount] = useState(0);

      useEffect(() => {
        const id = setInterval(() => {
          console.log('count is', count);
        }, 1000);
        return () => clearInterval(id);
      }, []); // empty deps

      return <button onClick={() => setCount(c => c + 1)}>
        Click ({count})
      </button>;
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    إغلاق قديم كلاسيكي. الـ `useEffect` يُنفّذ مرة واحدة (مصفوفة الاعتماديات فارغة) وينشئ فاصلًا يلتقط `count` كما كان عند التركيب: `0`. بغض النظر عن عدد النقرات، لا يرى إغلاق الفاصل القيمة المحدثة.

    تسمية الزر تُحدَّث بشكل صحيح — هذا إعادة تصيير جديدة. لكن `console.log` محبوس إلى الأبد على `0`.

    أصلحه بإضافة `count` إلى مصفوفة الاعتماديات (يعيد إنشاء الفاصل) أو باستخدام مرجع (`ref`):
    ```jsx
    const countRef = useRef(count);
    useEffect(() => { countRef.current = count; }, [count]);
    // then use countRef.current inside the interval
    ```
  </div>
  </slot>
</Challenge>

        deposit: (n) => (balance += n),

    ```javascript
    console.log(bank.balance); // undefined
    ```
    قبل وجود الفئات، كان هذا هو *الطريقة* لتغليف الحالة في جافاسكريبت. لا يزال يظهر في قواعد الشيفرة القديمة ومكتبات الأدوات.

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={7}
  group="خبير"
  title="تسرب الذاكرة في الإغلاق"
  options={[
    {text: 'يقوم بتسجيل طول مصفوفة البيانات'},
    {text: 'ينشئ تسربًا في الذاكرة', isAnswer: true},
    {text: 'يرمي خطأ نطاق (RangeError)'},
    {text: 'لا شيء غير عادي — هذا جيد'},
  ]}
>
  <slot name="question">
  <div className="question">
    ما المشكلة في هذا الكود؟
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      element.addEventListener('click', () => {
        console.log('clicked!', data.length);
      });
    }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    مستمع الحدث يغلق على `data`، لذا تظل المصفوفة قابلة للوصول طالما كان المستمع قابلًا للوصول. إذا كان `element` طويل العمر ولم يُزال المستمع أبدًا، فإن تلك المصفوفة الكبيرة تُحتفظ بها أيضًا.

    `element` موجود في الـ DOM. المستمع يحتفظ بـ `data`. لا يستطيع جامع القمامة (GC) جمعه.

    الحل: لا تلتقط بيانات كبيرة في مستمعين طويلي العمر إلا إذا كنت بحاجة فعلًا إليها، وأزل المستمعين عندما يتم تفكيك واجهة المستخدم المالكة:
    ```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      // do something with data...

      const onClick = () => {
        console.log('clicked!');
      };

      element.addEventListener('click', onClick);
      return () => element.removeEventListener('click', onClick);
    }
    ```
    محرك V8 الحديث أفضل في هذا الأمر مقارنةً بالماضي. لكن في تطبيقات SPA طويلة العمر مع العديد من مستمعي الأحداث، لا يزال هذا النمط يسبب مشاكل للناس.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={8}
  group="خبير"
  title="فخ الـ this"
  options={[
    {text: 'Dan, Dan'},
    {text: 'Dan, undefined'},
    {text: 'undefined, Dan'},
    {text: 'undefined, undefined', isAnswer: true},
  ]}
>
  <slot name="question">
  <div className="question">
    ماذا يُسجِّل هذا؟
    ```javascript
    const user = {
      name: 'Dan',
      greetArrow: () => console.log(this?.name),
      greetRegular() {
        const inner = () => console.log(this?.name);
        inner();
      },
    };

    user.greetArrow();
    const fn = user.greetRegular;
    fn();
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    فخّان دقيقان:

    **`greetArrow`**: الدوال السهمية لا تملك `this` خاصًا بها. إنها تلتقط `this` من النطاق اللفظي المحيط — وهو هنا نطاق الوحدة/العالمية، وليس `user`. لذا `this?.name` يكون `undefined`.

    **`fn()`**: يتم استدعاء `greetRegular` بدون مستقبل (`fn()` وليس `user.greetRegular()`). في الوضع الصارم، `this` يكون `undefined`. في الوضع غير الصارم، يكون كائنًا عالميًا. في كلتا الحالتين، `this?.name` يكون `undefined`.

    إذا استدعيت `user.greetRegular()` مباشرةً، ستحصل على `'Dan'` — لأن `this` حينئذٍ مرتبط بـ `user`.

    النتيجة: `undefined, undefined`.
  </div>
  </slot>
</Challenge>

  client:visible={{rootMargin: "150px"}}
  index={9}
  group="مكافأة"
  title="التخزين المؤقت عبر الإغلاق"
  options={[
    {text: 'ينادي expensive() 3 مرات'},
    {text: 'ينادي expensive() مرة واحدة', isAnswer: true},
    {text: 'ينادي expensive() مرتين'},
    {text: 'يرمي استثناء: تجاوز الحد الأقصى لسلسلة الاستدعاءات'},
  ]}
>
  <slot name="question">
  <div className="question">
    كم مرة يتم نداء `expensive()` فعليًا؟
    ```javascript
    function memoize(fn) {
      const cache = new Map();
      return (arg) => {
        if (cache.has(arg)) return cache.get(arg);
        const result = fn(arg);
        cache.set(arg, result);
        return result;
      };
    }

    let calls = 0;
    const expensive = (n) => { calls++; return n * 2; };
    const fast = memoize(expensive);

    fast(5);
    fast(5);
    fast(5);
    console.log(calls);
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    الدالة المخزنة مؤقتًا تغلق على `cache`. الاستدعاء الأول (`fast(5)`) لا يجد القيمة في الذاكرة المؤقتة، ينادي `expensive`، ويخزن النتيجة. الاستدعاء الثاني والثالث يجدان القيمة في الذاكرة المؤقتة وتعيدان النتيجة فورًا.

    `calls` هو `1`.

    هذا مثال على عمل الإغلاقات. `cache` خاصة (لا يمكن الوصول إليها من خارج `memoize`)، مستمرة (تعيش طالما `fast` موجودة)، ومشتركة بين جميع استدعاءات `fast`.
  </div>
  </slot>
</Challenge>

- **9-10**: أنت فعلاً تفهم الـ closures. ربما يكون من المخيف البرمجة الزوجية معك.
- **6-8**: جيد. أسئلة الـ stale closure / `this` أوقعتك، وهذا طبيعي.
- **3-5**: تعرف الفكرة لكن الحواف غير واضحة. اطلع على وثائق سلسلة النطاق في MDN.
- **0-2**: كنت تعتقد أنك تعرف الـ closures. الآن عرفت. هذا هو الهدف.

الـ stale closure في React (السؤال #6) وفخ الـ `this` (السؤال #8) هما ما يسببان أخطاء حقيقية في الإنتاج. إذا كانا غير مألوفين لك، فاستحقان المراجعة — يظهران في مراجعات الكود أكثر من أي مشكلة أخرى متعلقة بالـ closures.---
