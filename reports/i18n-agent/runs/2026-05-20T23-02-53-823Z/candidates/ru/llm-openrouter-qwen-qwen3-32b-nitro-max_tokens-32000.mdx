## Вы — склад учетных данных

Ваш ноутбук не является ноутбуком. Это склад учетных данных с клавиатурой — сеансы браузера, SSH-ключи, файлы `.env`, токены GitHub, облачные CLI, AI-инструменты кодирования с доступом к оболочке, экспорты баз данных, о которых вы забыли.

Старая модель: продакшн — опасно, локально — удобно. Эта модель умерла.

<p class="inset">
Вопрос не в том, сможете ли вы избежать каждой плохой кликнутой ссылки. Вопрос в том, может ли одна плохая кликнутая ссылка прочитать всё, использовать всё и уйти до того, как вы это заметите.
</p>

Разработчик сталкивается с чем-то, что выглядит достаточно нормально: PDF от подрядчика, фальшивый CAPTCHA, просящий вставить что-то в терминал, пакет с `postinstall`-скриптом, сессия AI-кодирования, которая заглянула глубже в файловую систему, чем требовалось. Некоторые пути устанавливают вредоносное ПО. Некоторые крадут учетные данные. Некоторым не нужна локальная эксплуатация — пользователь сам запускает команду атакующего.

Это современная атакующая поверхность. Иногда вы сами являетесь уязвимостью.

## Проблема цепочки поставок невероятно масштабна

Вот веселая часть. Чтобы быть полностью безопасным, вам нужно выполнить глубокую многоуровневую оценку безопасности каждой зависимости, на которую вы полагаетесь — их поддерживателей, историю, транзитивные зависимости — во всех пакетных реестрах. Затем повторите оценку каждый раз, когда ваше дерево зависимостей изменяется или получает обновление, потому что именно так работают атаки на цепочку поставок: они эксплуатируют цепочку доверия.

Легко.

О, и злоумышленнику нужно лишь один раз преуспеть. Вам нужно поддерживать идеальную защиту каждый раз.

Lumma Stealer — широко используемый инфостеалер, который тихо собирает пароли, куки браузера, API-ключи и облачные учетные данные — достиг жертв через поддельные CAPTCHA, отравленные рекламные объявления и троянизированные приложения. Расследование Snowflake, проведенное Mandiant, проследило каскад корпоративных взломов до украденных инфостеалерами учетных данных, некоторые из которых датируются 2020 годом. По меньшей мере 79,7% учетных записей, использованных в атаке, имели известное предыдущее воздействие. Замки никогда не менялись.

Атакующему не нужно было взламывать склад. Он нашел старые ключи в ящике стола.

Для разработчиков этот ящик стола выглядит так:

| Локальный артефакт | Почему атакующие заинтересованы |
| --- | --- |
| Куки браузера | Могут обойти вход и иногда пропускать МФА. |
| Файлы `.env` | API-ключи, URL баз данных, JWT-секреты. |
| Конфигурация облачного CLI | Переводит компрометацию ноутбука в полный доступ к инфраструктуре. |
| SSH-ключи | Все еще повсюду, все еще мощные, все еще копируются между машинами. |
| Токены менеджера пакетов | Ваш токен публикации в npm или PyPI — это доступ к цепочке поставок. |
| Дампы баз данных | Менее защищены, чем продакшн, часто более полные. |
| Контекст AI-кодирования | Ассистент мог получить чувствительные файлы "для контекста". |

А затем есть резервные копии — экспорт продакшн, который кто-то бросил в `~/Downloads` и забыл. Резервная копия не безопаснее из-за своей инертности. Это просто продакшн без системы оповещений.

## Несолюция "Будьте осторожны"

Совет "Будьте осторожны" слаб. Он требует, чтобы человек был границей.

Люди не являются границами. Люди — это трафик.

Границы скучны: изоляция файловой системы, шифрование секретов, краткосрочные учетные данные, аутентификация с поддержкой аппаратных средств и оповещения, которые срабатывают в момент касания поддельного секрета.

Если запускается вредоносный процесс, вопросы, которые решают, будет ли у вас плохой день или инцидент по всей компании, следующие:
1. Что этот процесс может **прочитать**?
2. Какие учетные данные он может **использовать**?
3. Куда он может **отправить данные**?

## Наиболее эффективные действия прямо сейчас

### Dev Containers — По умолчанию

[Development Containers](https://github.com/devcontainers/spec) — это самый эффективный изменение, который большинство команд не внедряет. Dev Container запускает проектную работу внутри изолированного Docker-контейнера. `npm install`, `pip install`, `postinstall`-скрипты, команды AI-оболочки, расширения VS Code — все это происходит в "рабочей области" или контейнере, который не видит остальную часть вашей машины.

<p class="inset">Попросите Claude Code настроить DevContainers в любом проекте.</p>

Подключите репозиторий. Включите только те секреты, которые нужны для этого проекта. Не подключайте `~/.ssh`, `~/.aws` или ваш домашний каталог из-за удобства. Инструкция с подстановкой приглашения может достичь только того, что может достичь агент — сделайте это скучным.

```jsonc
// .devcontainer/devcontainer.json
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ]
}
```

### Canary Tokens — Агрессивно Развернутые

[Canarytokens](https://canarytokens.org) — это бесплатные цифровые сигнальные устройства. Посадите фальшивый, но убедительный секрет где-нибудь, где атакующий будет искать. В момент касания вы получите оповещение — часто в течение нескольких секунд. Представьте себе, как оставляют красящий пакет в поддельной стопке денег.

Атакующие инвентаризируют, прежде чем украсть. Этот этап разведки — ваше окно.

Поместите сигнальные устройства в самые привлекательные для кражи файлы:

```text
~/.aws/credentials          ← добавьте фальшивый профиль [billing-prod-legacy] с сигнальным ключом
~/backups/customer-export-2024.sql   ← сигнальный URL внутри
~/.env.canary               ← фальшивые учетные данные в каждом репозитории
```

Сигнальные устройства доступны бесплатно на [canarytokens.org](https://canarytokens.org), их можно развернуть локально, а также получить как платный SaaS-сервис через [Thinkst Canary](https://canary.tools). Нет никаких оправданий не использовать их повсеместно, где вор будет искать.

### Инструменты безопасности пакетов

Инструменты вроде [Socket.dev](https://socket.dev), [Snyk](https://snyk.io) и [Wiz](https://wiz.io) часто первыми обнаруживают и блокируют атаки через цепочки поставок. Они отслеживают реестры пакетов, которые вы не можете мониторить самостоятельно. Для команд, не имеющих возможности поддерживать полноценную программу безопасности, это высокодоходные системы раннего предупреждения.

### Минимальный возраст пакетов в PNPM

Если вы используете PNPM, установите минимальный возраст публикации. Недавно опубликованные пакеты представляют наибольший риск для атак по цепочке поставок — пакет, существующий менее 24 часов, практически не подвергался вниманию сообщества. Установите `minimumReleaseAge` в минутах: минимум `1440` (один день), а лучше `2880` (два дня).

```yaml
minimumReleaseAge: 2880
minimumReleaseAgeStrict: true
minimumReleaseAgeIgnoreMissingTime: false
minimumReleaseAgeExclude:
  - 'typescript'
```

Эта конфигурация блокирует множество атак через недавно опубликованные пакеты, особенно те, которые обнаруживаются и удаляются до вашего следующего установки. Используйте `minimumReleaseAgeExclude` осторожно для пакетов, где немедленные обновления важнее задержки, например, для компилятора или зависимостей среды выполнения, которые вы активно отслеживаете.

### Для самых критичных с точки зрения безопасности сред

Интеллектуальные агентства, правоохранительные органы, финансовая торговая инфраструктура, медицинские записи — эти среды иногда внедряют строгие процессы оценки и одобрения пакетов. Это звучит безопасно. Но компромисс серьезен: ваша зависимость постепенно превращается в устаревшее ПО.

Время здесь не нейтрально. Старые версии накапливают известные уязвимости. Атакующие изучают исправленные версии, чтобы найти неисправленные экземпляры. И "лучше дьявола, которого знаешь" не является спасением, на которое вы надеялись — это просто показывает, какие уязвимости атакующий дольше всего изучал.

Строгие белые списки работают, если у вас есть персонал для их поддержки. Большинству команд это не под силу. Для всех остальных подход с многослойной защитой — Dev Containers, сигнальные устройства, инструменты безопасности пакетов, краткосрочные учетные данные — обеспечивает более реалистичную защиту, чем иллюзия, что вы можете вручную аудитировать каждую зависимость.

## У вас есть минуты

Когда сигнальное устройство срабатывает — или GitHub предупреждает вас, что токен использовался из неожиданного IP — у вас есть окно. Минуты, может быть, несколько часов. Не неделю.

- **Сначала измените, потом расследуйте.** Отмените токены до того, как поймете, что произошло.
- **Проверьте на наличие устойчивости атакующего.** Новые OAuth-приложения, пользователи IAM, ключи деплоя, API-токены, созданные до их ухода.
- **Завершите активные сеансы браузера.** Принудительно выйдите из всего, что важно.
- **Сообщите кому-нибудь.** Инциденты безопасности улучшаются с наблюдателями и временными метками.

Индустрия безопасности много говорит о детектировании. Меньше говорит о том, что происходит в двадцать минут после обнаружения, когда вы один за своим столом пытаетесь вспомнить, для каких сервисов у вас есть токены.

Этот список должен существовать до срабатывания сигнала.

## Стандарт, заслуживающий наличия

Стандарт — это не "никогда не нажимайте на что-то странное". Это совет для плаката, а не для системы.

Ненадежная зависимость не должна иметь доступ к учетным данным в облаке из других проектов. Документ с внедренным приглашением не должен перенаправлять агента в ваш домашний каталог. Инфостеалер не должен находить резервные копии в открытом виде и долгоживущие токены без активации сигнализации. Украденный токен должен истекать, проваливать двухфакторную аутентификацию или попадать в canary-токен до того, как привести к полному захвату.

Безопасность улучшается, когда мы перестаем требовать от людей идеальности и начинаем делать компромисс менее выгодным.

Ваш ноутбук теперь часть инфраструктуры. Наделите его скучными границами, которые поймают как атакующего, проникшего в систему, так и того, кого вы случайно пропустили сами.
