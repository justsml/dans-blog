## You Are the Credential Warehouse

Ваш ноутбук — это не просто ноутбук. Это склад учётных данных с клавиатурой — сессии браузера, SSH‑ключи, файлы `.env`, токены GitHub, облачные CLI, инструменты ИИ‑кодинга с доступом к оболочке, экспортированные базы данных, о которых вы забыли.

Старая модель была такова: продакшн опасен, локальная работа удобна. Эта модель завершена.

<p class="inset">
Вопрос не в том, сможете ли вы избежать каждой ошибочной щелчка. Вопрос в том, может ли один ошибочный щелчок прочитать всё, использовать всё и уйти, пока вы ещё не заметили.
</p>

Разработчик сталкивается с тем, что выглядит достаточно обычным: PDF от подрядчика, поддельная CAPTCHA с просьбой вставить что‑то в терминал, пакет с скриптом `postinstall`, сессия ИИ‑кодинга, которая зашла дальше в файловую систему, чем требовалось. Некоторые пути устанавливают вредоносное ПО. Некоторые крадут учётные данные. Некоторые не требуют локального эксплойта — пользователь сам исполняет команду злоумышленника.

Это современная поверхность атаки. Иногда вы сами становитесь утечкой.

## The Supply Chain Problem Is Impossibly Large

Вот интересная часть. Чтобы быть полностью защищённым, вам нужно провести глубокую, многоплатформенную оценку безопасности каждой зависимости, на которой вы полагаетесь — их поддерживающих, их истории, их транзитивных зависимостей — во всех реестрах пакетов. Затем повторять эту оценку каждый раз, когда меняется дерево зависимостей или появляется обновление, потому что именно так работают атаки на цепочку поставок: они используют цепочку доверия.

Легко.

И, кстати, злоумышленнику достаточно один раз добиться успеха. Вам же нужно поддерживать безупречную защиту каждый раз.

Lumma Stealer — широко используемый инфостилер, который тихо собирает пароли, куки браузеров, API‑ключи и облачные учётные данные — доставал жертв через поддельные CAPTCHA, отравленные рекламные объявления и троянские приложения. Расследование Snowflake от Mandiant проследило каскад корпоративных нарушений до учётных данных, украденных инфостилерами, некоторые из которых датируются 2020 годом. По крайней мере 79,7 % аккаунтов, использованных в атаке, уже были известны ранее. Замки никогда не менялись.

Злоумышленник не взломал склад. Он нашёл старые ключи в ящике стола.

Для разработчиков этот ящик выглядит так:

| Локальный артефакт | Почему он интересен злоумышленникам |
| --- | --- |
| Куки браузера | Позволяют обойти вход и иногда миновать MFA. |
| Файлы `.env` | API‑ключи, URL баз данных, JWT‑секреты. |
| Конфигурация облачного CLI | Превращает компрометацию ноутбука в полный доступ к инфраструктуре. |
| SSH‑ключи | Всё ещё повсюду, всё ещё мощные, всё ещё копируются между машинами. |
| Токены менеджеров пакетов | Ваш токен публикации в npm или PyPI — доступ к цепочке поставок. |
| Дамп базы данных | Защищён меньше, чем продакшн, часто более полон. |
| Контекст кода ИИ | Помощник мог получить чувствительные файлы «для контекста». |

И потом идут резервные копии — экспорт продакшна, который кто‑то бросил в `~/Downloads` и забыл. Резервная копия не безопаснее, просто потому что она инертна. Это просто продакшн без системы сигнализации.

## «Будьте осторожны» — не решение

«Будьте осторожны» — слабый совет. Он перекладывает границу на человека.

Человек — не граница. Человек — трафик.

Границы скучны: изоляция файловой системы, шифрование секретов в покое, кратковременные учётные данные, аутентификация с поддержкой аппаратуры и оповещения, срабатывающие в момент касания поддельного секрета.

Если запускается вредоносный процесс, вопросы, определяющие, будет ли у вас плохой вечер или масштабный инцидент компании, таковы:
1. Что этот процесс **может читать**?
2. Какие учётные данные он **может использовать**?
3. Куда он **может отправлять данные**?

## Наиболее эффективные действия прямо сейчас

### Dev Containers — по умолчанию

[Development Containers](https://github.com/devcontainers/spec) — единственное изменение с наибольшим рычагом воздействия, которое большинство команд не применяют. Dev Container запускает работу проекта внутри изолированного Docker‑контейнера. `npm install`, `pip install`, скрипты `postinstall`, команды ИИ‑оболочки, расширения VS Code — всё это происходит в «рабочем пространстве» или контейнере, который не видит остальную часть вашей машины.

<p class="inset">Попросите Claude Code настроить DevContainers в любом проекте.</p>

Монтируйте репозиторий. Добавляйте только те секреты, которые нужны конкретному проекту. Не монтируйте `~/.ssh`, `~/.aws` и ваш домашний каталог из удобства. Инструкция, внедрённая через подсказку, может достать только то, к чему агент имеет доступ — сделайте это скучным.

```jsonc
// .devcontainer/devcontainer.json
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ]
}
```

### Canary Tokens — агрессивно развернутые

[Canarytokens](https://canarytokens.org) — бесплатные цифровые ловушки. Посадите поддельный, но правдоподобный секрет там, где злоумышленник будет искать. В момент касания вы получаете оповещение — часто в течение нескольких секунд. Представьте это как оставление пакета‑краски в фальшивой пачке банкнот.

Злоумышленники проводят инвентаризацию перед кражей. Этот разведывательный проход — ваше окно.

Бросьте канарейки в самые «привлекательные» файлы:

```text
~/.aws/credentials          ← добавьте поддельный профиль [billing-prod-legacy] с канарейским ключом
~/backups/customer-export-2024.sql   ← канарейский URL внутри
~/.env.canary               ← поддельные учётные данные в каждом репозитории
```

Canary‑токены бесплатны на [canarytokens.org](https://canarytokens.org), их можно развернуть самостоятельно, а также приобрести как SaaS‑решение у [Thinkst Canary](https://canary.tools). Нет веской причины не использовать их везде, где может заглянуть злоумышленник.

### Инструменты безопасности пакетов

Инструменты вроде [Socket.dev](https://socket.dev), [Snyk](https://snyk.io) и [Wiz](https://wiz.io) часто первыми обнаруживают и блокируют атаки на цепочку поставок. Они мониторят реестры пакетов, за которыми вы сами не успеваете следить. Для команд, которым не хватает ресурсов на полноценную программу безопасности, такие сервисы — высокоэффективные системы раннего предупреждения.

### Настройки минимального возраста пакетов в PNPM

Если вы используете PNPM, задайте минимальный возраст релиза. Новоопубликованные пакеты — самое уязвимое окно для атак на цепочку поставок: пакет, существующий менее 24 часов, практически не прошёл общественной проверки. Установите `minimumReleaseAge` в минутах: минимум `1440` (один день) и, желательно, `2880` (два дня).

```yaml
minimumReleaseAge: 2880
minimumReleaseAgeStrict: true
minimumReleaseAgeIgnoreMissingTime: false
minimumReleaseAgeExclude:
  - 'typescript'
```

Эта конфигурация блокирует многие атаки на только‑что опубликованные пакеты, особенно те, которые обнаруживаются и вытягиваются до вашего следующего `install`. Параметр `minimumReleaseAgeExclude` используйте экономно, только для пакетов, где мгновенные обновления важнее задержки, например компилятор или runtime‑зависимость, которую вы активно отслеживаете.

### Для самых критичных к безопасности окружений

Разведывательные агентства, правоохранительные органы, инфраструктура финансовой торговли, медицинские записи — в этих средах иногда вводят строгий процесс оценки и одобрения пакетов. На первый взгляд это выглядит безопасно, но цена высока: дерево зависимостей медленно превращается в устаревшее программное обеспечение.

Время здесь не нейтрально. Старые версии накапливают известные CVE. Атакующие изучают исправленные версии, чтобы найти непатченные экземпляры. И «лучше дьявол, которого знаешь» не спасёт вас — это лишь указывает, какие уязвимости злоумышленник имел возможность изучить дольше всех.

Строгие allow‑листы работают, если у вас есть персонал для их поддержки. У большинства команд такой возможности нет. Для остальных более реалистичная защита — слоистый подход: Dev Containers, канарейские токены, инструменты безопасности пакетов, краткоживущие учётные данные — превосходит попытки аудировать каждую зависимость вручную.

## У вас есть минуты

Когда канарейка срабатывает — или GitHub оповещает, что токен использован с неожиданного IP — у вас есть окно. Минуты, возможно, несколько часов. Не неделя.

- **Сначала замените, потом расследуйте.** Отзовите токены, пока не ясно, что произошло.
- **Проверьте устойчивость атакующего.** Новые OAuth‑приложения, IAM‑пользователи, deploy‑ключи, API‑токены, созданные до его ухода.
- **Прервите активные браузерные сессии.** Принудительно разлогиньте всё, что вам важно.
- **Сообщите кому‑нибудь.** Инциденты в безопасности лучше решаются при наличии свидетелей и меток времени.

Индустрия безопасности много говорит о детекции. Меньше — о том, что происходит в двадцать минут после обнаружения, когда вы одни за столом пытаетесь вспомнить, какие сервисы используют ваши токены.

Этот список должен быть готов до того, как сработает сигнал тревоги.

## Стандарт, который стоит иметь

Стандарт — это не «никогда не нажимать на что‑то странное». Такое правило подходит для плаката, а не для системы.

Плохая зависимость не должна иметь возможности достать облачные учётные данные из других проектов. Документ, внедрённый через подсказку, не должен перенаправлять агент в ваш домашний каталог. Инфостилер не должен находить открытые резервные копии и долговременные токены без срабатывания тревоги. Украденные учётные данные должны истекать, проваливать MFA или попадать в канарейку до того, как они позволят полностью захватить систему.

Безопасность повышается, когда мы перестаём требовать от людей безупречности и начинаем делать компрометацию менее выгодной.

Ваш ноутбук теперь часть продакшена. Оборудуйте его скучными границами, которые поймают как злоумышленника, пробившего защиту, так и того, кого вы сами случайно впустили внутрь.
