## 君自身がクレデンシャル倉庫だ

君のラップトップはラップトップではない。キーボード付きのクレデンシャル倉庫だ——ブラウザセッション、SSH鍵、`.env`ファイル、GitHubトークン、クラウドCLI、シェルアクセスを持つAIコーディングツール、存在を忘れていたデータベースのエクスポート。

昔のモデルは「本番は危険、ローカルは便利」だった。そのモデルは終わった。

<p class="inset">
問題は、悪いクリックをすべて回避できるかどうかではない。問題は、たった一度の悪いクリックで、すべてを読み、すべてを使い、気づかれる前に去られるかどうかだ。
</p>

開発者は、一見ごく普通に見えるものに出会う。請負業者からのPDF、ターミナルに何かを貼り付けるよう求める偽のCAPTCHA、`postinstall`スクリプトを含むパッケージ、タスクに必要な範囲を超えてファイルシステムにまで踏み込んだAIコーディングセッション。ある経路はマルウェアをインストールする。ある経路は認証情報を盗む。ローカルエクスプロイトすら不要なものもある——ユーザー自身が攻撃者のコマンドを実行する。

これが現代の攻撃対象領域だ。時には、君自身が侵害そのものになる。

## サプライチェーン問題は不可能なほど大きい

ここからが面白いところだ。完全に安全になるために必要なことは、依存するすべての依存関係——そのメンテナー、履歴、推移的依存関係——に対して、すべてのパッケージレジストリにわたって深く、マルチプラットフォームなセキュリティ評価を実施することだ。その後、依存関係ツリーが変更されたり更新があったりするたびに評価を繰り返す。なぜなら、まさにそれがサプライチェーン攻撃の仕組みだからだ：信頼の連鎖を悪用する。

簡単だ。

ああ、攻撃者は一度成功すればいい。君は毎回完全な防御を維持しなければならない。

Lumma Stealer — パスワード、ブラウザのクッキー、APIキー、クラウド認証情報を静かに収集する広く使われているインフォスティーラー — は、偽のCAPTCHA、改ざんされた検索広告、トロイの木馬化されたアプリを通じて被害者に到達した。MandiantのSnowflake調査は、企業侵害の連鎖をインフォスティーラーによって盗まれた認証情報にまで遡り、中には2020年までさかのぼるものもあった。攻撃で使用されたアカウントの少なくとも79.7%は、以前に漏洩が確認されていた。鍵は一度も交換されていなかった。

攻撃者は倉庫を破ったわけではない。机の引き出しに置き忘れた古い鍵を見つけただけだ。

開発者にとって、その机の引き出しは次のような姿をしている：

| ローカル成果物 | 攻撃者が狙う理由 |
| --- | --- |
| ブラウザのクッキー | ログインをバイパスし、場合によっては多要素認証をスキップできる。 |
| `.env`ファイル | APIキー、データベースURL、JWTシークレット。 |
| クラウドCLI設定 | ラップトップの侵害を完全なインフラストラクチャアクセスに変える。 |
| SSHキー | 今もどこにでもあり、強力で、マシン間でコピーされ続けている。 |
| パッケージマネージャートークン | npmやPyPIの公開トークンはサプライチェーンへのアクセスだ。 |
| データベースダンプ | 本番環境よりも保護が不十分で、多くの場合より完全なデータを含む。 |
| AIコーディングコンテキスト | アシスタントに「コンテキストとして」機密ファイルが渡されている可能性がある。 |

そしてバックアップもある——誰かが`~/Downloads`に置き忘れた本番環境のエクスポートだ。バックアップは非アクティブだから安全というわけではない。警報システムのない本番環境に過ぎない。

## 「気をつけろ」は解決策にならない

「気をつけろ」は弱いアドバイスだ。人間を境界にしようとしている。人間は境界ではない。人間はトラフィックだ。境界は地味だ。ファイルシステムの分離、保存時の暗号化、短命な認証情報、ハードウェアベースの認証、そして偽のシークレットに触れた瞬間に発報するアラート。悪意のあるプロセスが実行された場合、午後がひどくなるだけか、会社全体のインシデントになるかを決める質問は次のとおりだ：
1. このプロセスは何を **読み取れる** か？
2. どの認証情報を **使用できる** か？
3. どこに **データを送信できる** か？

## 今すぐできる最大の効果を発揮する対策

### Dev Containers — デフォルトで使う

[Development Containers](https://github.com/devcontainers/spec) は、ほとんどのチームが導入していない、単一で最大の効果を発揮する変更だ。Dev Containerは、プロジェクトの作業を隔離されたDockerコンテナ内で実行する。`npm install`、`pip install`、`postinstall`スクリプト、AIシェルコマンド、VS Code拡張機能——これらすべてが、マシンの他の部分を見ることができない「ワークスペース」またはコンテナ内で行われる。

<p class="inset">Claude Codeに任意のプロジェクトでDevContainersをセットアップするよう依頼してください。</p>

リポジトリをマウントする。そのプロジェクトに必要なシークレットだけを含める。`~/.ssh`、`~/.aws`、またはホームディレクトリを便宜的にマウントしてはならない。プロンプトインジェクションされた命令は、エージェントが到達できるものにしか到達できない——それを地味にしておく。

```jsonc
// .devcontainer/devcontainer.json
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ]
}
```

### Canary Tokens — 積極的に配置する

[Canarytokens](https://canarytokens.org) は無料のデジタルワイヤーだ。攻撃者が探しそうな場所に、偽物だが信憑性のあるシークレットを仕掛ける。触れられた瞬間、アラートが届く——多くの場合、数秒以内に。偽の札束に染料パックを仕込むイメージだ。攻撃者は盗む前に在庫調査を行う。その偵察の一瞬があなたのチャンスだ。

最も魅力的に見えるファイルにカナリアを仕込め：

```text
~/.aws/credentials          ← 偽の [billing-prod-legacy] プロファイルにカナリアキーを追加
~/backups/customer-export-2024.sql   ← 内部にカナリアURL
~/.env.canary               ← 全リポジトリに偽の認証情報
```

Canaryトークンは [canarytokens.org](https://canarytokens.org) で無料提供されており、セルフホストも可能、有料SaaSとしては [Thinkst Canary](https://canary.tools) からも利用できる。泥棒が探しそうな場所には、どこにでも仕掛けておくべきだ。それをしない理由はない。

### パッケージセキュリティツール

[Socket.dev](https://socket.dev)、[Snyk](https://snyk.io)、[Wiz](https://wiz.io) といったツールは、進行中のサプライチェーン攻撃をいち早く発見しブロックする。自前では監視しきれないパッケージレジストリを監視してくれる。フルタイムのセキュリティプログラムを組めないチームにとっては、高いレバレッジを持つ早期警戒システムだ。

### PNPMの最低リリース期間設定

PNPMを使っているなら、最低リリース期間を設定しよう。新しく公開されたパッケージはサプライチェーン攻撃のリスクが最も高い。公開から24時間未満のパッケージは、コミュニティによる実質的な精査がゼロに等しい。`minimumReleaseAge` を分単位で設定する：最低 `1440`（1日）、理想的には `2880`（2日）だ。

```yaml
minimumReleaseAge: 2880
minimumReleaseAgeStrict: true
minimumReleaseAgeIgnoreMissingTime: false
minimumReleaseAgeExclude:
  - 'typescript'
```

この設定により、特に公開後に発見されて次のインストール前に削除されるような、新規公開パッケージを利用した攻撃の多くを防げる。`minimumReleaseAgeExclude` は、コンパイラやランタイム依存関係など、遅延よりも即時更新が重要なパッケージに限定して使うこと。

### 最もセキュリティを重視する環境向け

諜報機関、法執行機関、金融取引基盤、医療記録——こうした環境では、厳格なパッケージ評価・承認プロセスを採用することがある。安全に聞こえる。しかしその代償は深刻だ：依存関係ツリーが徐々に古いソフトウェアへと硬化していく。

時間は中立ではない。古いバージョンは既知のCVEを蓄積する。攻撃者は修正版を研究して未パッチのインスタンスを狙う。「知っている悪魔の方がまし」という考えは、望むほどの救いにはならない——それは単に、攻撃者が最も習熟している脆弱性が何かを教えているだけだ。

厳格な許可リストは、それを維持できる人員がいるチームには有効だ。ほとんどのチームにはいない。それ以外の全員にとっては、Dev Containers、カナリアトークン、パッケージセキュリティツール、短命な認証情報——というレイヤードアプローチが、すべての依存関係を手作業で監査できるふりをするよりも現実的な防御を提供する。

## あなたには数分しかない

カナリアが発報したとき——あるいはGitHubからトークンが予期しないIPで使用されたと通知が来たとき——あなたには猶予期間がある。数分、せいぜい数時間だ。一週間ではない。

- **先にローテーション、後で調査。** 何が起きたか理解する前にトークンを無効化せよ。
- **攻撃者の永続化を確認せよ。** 攻撃者が去る前に作成した新しいOAuthアプリ、IAMユーザー、デプロイキー、APIトークンがないか。
- **アクティブなブラウザセッションを強制終了せよ。** 重要なすべてのサービスで強制ログアウトを実行せよ。
- **誰かに伝えよ。** セキュリティインシデントは証人とタイムスタンプがあれば改善される。

セキュリティ業界は検知について多く語る。しかし、検知から20分後、机の前で一人きりで「どのサービスにトークンを持っていたっけ」と必死に思い出そうとしているときに何が起きるかについては、ほとんど語らない。そのリストはアラートが発報する前に存在していなければならない。

## 持つべき基準

基準とは「変なものを絶対にクリックしない」ではない。それはポスターのためのアドバイスであり、システムのためのものではない。

不正な依存関係が他のプロジェクトのクラウド認証情報に到達できてはならない。プロンプトインジェクションされたドキュメントがエージェントをホームディレクトリに誘導してはならない。インフォスティーラーが平文のバックアップや長期有効なトークンをアラームもなく発見できてはならない。窃取された認証情報は、完全な乗っ取りに至る前に期限切れになるか、MFAに失敗するか、カナリアに引っかかるべきである。

人間に完璧を求めず、侵害の利益を減らす——その方向に進むほどセキュリティは向上する。

あなたのラップトップは今や本番環境の一部だ。侵入した攻撃者も、うっかり自分で招き入れた脆弱性も、どちらも捕まえる地味な境界を与えよ。
