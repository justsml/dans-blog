## あなたは「認証情報の倉庫」である

あなたのラップトップは、単なるラップトップではない。キーボードが付いた「認証情報の倉庫」だ。ブラウザセッション、SSHキー、`.env` ファイル、GitHubトークン、クラウドCLI、シェルアクセス権を持つAIコーディングツール、存在すら忘れていたデータベースのエクスポートファイルなどが詰まっている。

かつてのモデルは「本番環境は危険だが、ローカル環境は便利」というものだった。そのモデルはもう終わりだ。

<p class="inset">
問題は、すべての「悪いクリック」を回避できるかどうかではない。問題は、たった一度の「悪いクリック」で、すべてを読み取られ、すべてを悪用され、気づかれる前に立ち去られてしまうかどうかだ。
</p>

開発者は、一見すると何の問題もなさそうなものに遭遇する。契約業者からのPDF、ターミナルへの貼り付けを要求する偽のCAPTCHA、`postinstall` スクリプトを含むパッケージ、タスクに必要な範囲を超えてファイルシステムにアクセスするAIコーディングセッションなどだ。マルウェアをインストールするものもあれば、認証情報を盗むものもある。ローカルでのエクスプロイトすら必要としないものもある。ユーザー自身が攻撃者のコマンドを実行してしまうのだから。

これが現代の攻撃対象領域（アタックサーフェス）だ。時として、あなた自身が侵害の入り口（ブリーチ）となる。

## サプライチェーン問題はあまりにも巨大すぎる

ここからが面白いところだ。完全に安全であるためには、依存しているすべての依存関係（メンテナー、履歴、推移的依存関係）について、あらゆるパッケージレジストリを横断して、深く、マルチプラットフォームなセキュリティ評価を行う必要がある。そして、依存関係ツリーが変更されたり更新されたりするたびに、その評価を繰り返さなければならない。なぜなら、それこそがサプライチェーン攻撃の仕組みそのものだからだ。彼らは「信頼の連鎖」を悪用する。

簡単だろう？

ああ、それから、攻撃者は一度成功するだけでいい。あなたは毎回、完璧な防御を維持しなければならない。

Lumma Stealerは、パスワード、ブラウザのCookie、APIキー、クラウド認証情報を密かに収集する広く使われているインフォスティーラー（情報窃取マルウェア）だ。偽のCAPTCHA、汚染された検索広告、トロイの木馬化したアプリを通じて被害者に到達する。MandiantによるSnowflakeの調査では、一連の企業侵害がインフォスティーラーによって盗まれた認証情報に遡ることが判明しており、中には2020年まで遡るものもあった。攻撃に使用されたアカウントの少なくとも79.7%は、以前から侵害が判明していたものだった。鍵は一度も交換されていなかったのだ。

攻撃者は倉庫を破壊したわけではない。机の引き出しの中にあった古い鍵を見つけただけだ。

開発者にとって、その「机の引き出し」は以下のようなものだ。

| ローカルアーティファクト | 攻撃者が狙う理由 |
| --- | --- |
| ブラウザのCookie | ログインをバイパスし、MFAをスキップできる場合がある。 |
| `.env`ファイル | APIキー、データベースURL、JWTシークレット。 |
| クラウドCLI設定 | ラップトップの侵害がインフラ全体へのアクセスに直結する。 |
| SSHキー | 至る所に存在し、強力で、マシン間でコピーされ続けている。 |
| パッケージマネージャのトークン | npmやPyPIの公開トークンはサプライチェーンへのアクセス権そのもの。 |
| データベースダンプ | 本番環境より保護が甘く、多くの場合より完全なデータが含まれる。 |
| AIコーディングのコンテキスト | アシスタントに「コンテキスト用」として機密ファイルを渡している可能性がある。 |

さらにバックアップもある。誰かが `~/Downloads` に放り込んで忘れた本番環境のエクスポートデータだ。バックアップは「動かない」からといって安全なわけではない。それは単に、警報システムのない本番環境に過ぎない。

## 「気をつける」という解決策にならない解決策

「気をつける」というのは弱いアドバイスだ。人間に境界線になれと言っているようなものだ。

人間は境界線ではない。人間はトラフィック（通信）そのものだ。

境界線とは退屈なものだ。ファイルシステムの分離、保存時の暗号化、短命な認証情報、ハードウェアベースの認証、そして偽のシークレットが触れられた瞬間に発報するアラート。これらが境界線だ。

悪意のあるプロセスが実行されたとき、それが「最悪の午後」で済むか「全社的なインシデント」になるかを決める問いは以下の通りだ。
1. このプロセスは何を**読み取れる**か？
2. このプロセスはどの認証情報を**使用できる**か？
3. このプロセスはどこに**データを送信できる**か？

## 今すぐ実行すべき最もレバレッジの高い対策

### Dev Containers — デフォルトで導入する

[Development Containers](https://github.com/devcontainers/spec)は、ほとんどのチームが導入していない、最もレバレッジの高い変更だ。Dev Containerは、プロジェクトの作業を分離されたDockerコンテナ内で実行する。`npm install`、`pip install`、`postinstall`スクリプト、AIシェルコマンド、VS Code拡張機能など、すべてがマシンの他の部分からは見えない「ワークスペース」またはコンテナ内で完結する。

<p class="inset">Claude Codeに、あらゆるプロジェクトでDevContainersをセットアップするよう指示しよう。</p>

リポジトリをマウントする際は、そのプロジェクトに必要なシークレットのみを含めること。利便性を求めて `~/.ssh` や `~/.aws`、あるいはホームディレクトリ全体をマウントしてはならない。プロンプトインジェクションによる命令が到達できるのは、エージェントがアクセスできる範囲だけだ。その範囲を「退屈なもの」にしておけ。

```jsonc
// .devcontainer/devcontainer.json
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ]
}
```

### Canary Tokens — 積極的に展開する

[Canarytokens](https://canarytokens.org)は、無料のデジタルな仕掛け罠だ。攻撃者が探しそうな場所に、偽物だが本物らしく見えるシークレットを仕込んでおく。それが触れられた瞬間にアラートが届く。多くの場合、数秒以内だ。偽の札束にカラーボール（染料パック）を仕込んでおくようなものだと考えればいい。

攻撃者は盗む前にインベントリ（資産の棚卸し）を行う。その偵察の瞬間こそが、こちらのチャンスだ。

攻撃者が目をつけそうなファイルに、Canary（仕掛け罠）を仕込んでおく。

```text
~/.aws/credentials          ← add a fake [billing-prod-legacy] profile with a canary key
~/backups/customer-export-2024.sql   ← canary URL inside
~/.env.canary               ← fake credentials in every repo
```

Canarytokensは[canarytokens.org](https://canarytokens.org)で無料で利用でき、セルフホストも可能だ。また、[Thinkst Canary](https://canary.tools)経由で有償のSaaSとしても提供されている。泥棒が覗きそうな場所すべてにこれを展開しない理由はない。

### パッケージセキュリティツール

[Socket.dev](https://socket.dev)、[Snyk](https://snyk.io)、[Wiz](https://wiz.io)といったツールは、進行中のサプライチェーン攻撃をいち早く発見し、ブロックしてくれることが多い。これらは、自分では監視しきれないパッケージレジストリを監視してくれる。専任のセキュリティチームを抱える余裕のないチームにとって、これらは極めてレバレッジの高い早期警戒システムとなる。

### PNPMの最小リリース経過時間設定

PNPMを使っているなら、最小リリース経過時間を設定しよう。公開されたばかりのパッケージは、サプライチェーン攻撃のリスクが最も高い。公開から24時間も経っていないパッケージは、コミュニティによる精査が事実上ゼロだからだ。`minimumReleaseAge`を分単位で設定しよう。最低でも`1440`（1日）、理想的には`2880`（2日）だ。

```yaml
minimumReleaseAge: 2880
minimumReleaseAgeStrict: true
minimumReleaseAgeIgnoreMissingTime: false
minimumReleaseAgeExclude:
  - 'typescript'
```

この設定により、公開直後のパッケージを悪用した攻撃の多くを防げる。特に、インストールする前に発見されて削除されるような攻撃には有効だ。`minimumReleaseAgeExclude`は、コンパイラやランタイムの依存関係など、遅延よりも即時のアップデートが重要なパッケージに対してのみ、控えめに使用すること。

### セキュリティが極めて重要な環境のために

情報機関、法執行機関、金融取引インフラ、医療記録。こうした環境では、厳格なパッケージ評価・承認プロセスを採用することがある。一見安全そうだが、その代償は大きい。依存関係ツリーが徐々に硬直化し、時代遅れのソフトウェアになっていくのだ。

ここで時間は中立ではない。古いバージョンには既知のCVEが蓄積される。攻撃者は修正済みのバージョンを研究し、パッチが当たっていないインスタンスを探し出す。「見知らぬ悪魔より知っている悪魔の方がマシ（better the devil you know）」などという考えは、期待した救いにはならない。それは単に、攻撃者が最も長い時間をかけて習熟した脆弱性がどれであるかを教えているに過ぎない。

厳格な許可リスト（allowlist）は、それを維持する人員がいれば機能する。だが、ほとんどのチームにはそんな余裕はない。それ以外のチームにとっては、Dev Containers、Canary tokens、パッケージセキュリティツール、短命な認証情報といった多層的なアプローチの方が、すべての依存関係を手作業で監査できると思い込むよりも、はるかに現実的な防御策となる。

## 残された時間は数分

Canaryが発火したとき、あるいはGitHubから予期せぬIPアドレスでトークンが使用されたという警告が届いたとき、あなたに残された時間はわずかだ。数分、長くても数時間。1週間などという猶予はない。

- **まずローテーションし、調査は後回しにする。** 何が起きたのかを理解する前に、トークンを無効化せよ。
- **攻撃者の永続化の痕跡を確認する。** 攻撃者が立ち去る前に作成された新しいOAuthアプリ、IAMユーザー、デプロイキー、APIトークンがないか探せ。
- **アクティブなブラウザセッションを強制終了する。** 重要なものはすべて強制ログアウトさせる。
- **誰かに報告する。** セキュリティインシデントは、証人とタイムスタンプがあることで対応が改善する。

セキュリティ業界は検知についてよく語るが、検知後の20分間に何が起きるかについてはあまり語らない。デスクで一人、どのサービスにトークンを発行していたかを必死に思い出そうとしている、あの20分間だ。

そのリストは、アラートが鳴る前に存在していなければならない。

## 目指すべき基準

目指すべき基準は「怪しいリンクは絶対にクリックしない」などではない。それはポスターに書くような標語であって、システムではない。

悪意のある依存関係が、他のプロジェクトのクラウド認証情報にアクセスできてはならない。プロンプトインジェクションされたドキュメントが、エージェントをホームディレクトリに誘導してはならない。インフォスティーラーが、アラートを鳴らさずにプレーンテキストのバックアップや長期有効なトークンを見つけてはならない。盗まれた認証情報は、完全な乗っ取りに至る前に、期限切れになるか、MFAで失敗するか、カナリアトークンにヒットしなければならない。

人間に完璧さを求めるのをやめ、侵害を割に合わないものにすることで、セキュリティは向上する。

今や、あなたのラップトップは本番環境の一部だ。侵入した攻撃者と、あなたが誤って招き入れた攻撃者の両方を捕まえられるような、退屈だが堅実な境界線を設定せよ。
