## あなたは資格情報倉庫だ

あなたのノートPCはノートPCではない。それはキーボード付きの資格情報倉庫だ——ブラウザセッション、SSHキー、`.env`ファイル、GitHubトークン、クラウドCLI、シェルアクセスのあるAIコーディングツール、存在を忘れていたデータベースエクスポート。

古いモデルは「本番環境は危険で、ローカルは便利」というものだった。そのモデルは終わっている。

<p class="inset">
避けるべき悪いクリックをすべて回避できるかが問題ではなく、1つの悪いクリックがすべてを読み取り、すべてを利用し、気づく前に退出できるかが問題だ。
</p>

開発者は、通常に見えるものを遭遇する——下請け業者のPDF、ターミナルに何かを貼り付けろと求める偽のCAPTCHA、`postinstall`スクリプトを含むパッケージ、タスクに必要な範囲を超えてファイルシステムを探索したAIコーディングセッション。あるパスはマルウェアをインストールする。あるパスは資格情報を盗む。あるパスはローカルのエクスプロイトすら必要とせず、ユーザー自身が攻撃者のコマンドを実行する。

これが現代の攻撃面だ。時にはあなた自身が漏洩だ。

## サプライチェーン問題は不可能に大きい

楽しみなのはここだ。完全に安全であるためには、依存するすべてのパッケージレジストリで、すべての依存関係——メンテナ、履歴、推移依存関係——を深く多プラットフォームでセキュリティ評価する必要がある。その後、依存ツリーが変更されたり更新が入るたびに評価を繰り返す必要がある。それがまさにサプライチェーン攻撃の仕組みだ——信頼の連鎖を悪用するのだ。

簡単だ。

ああ、そして攻撃者は1度成功すればいい。あなたは毎回完璧な防御を維持しなければならない。

Lumma Stealer — 拡散性の高いインフォスティーラーで、パスワードやブラウザのクッキー、APIキー、クラウド資格情報を静かに収集するもの — は、偽のCAPTCHAや毒された検索広告、トロイの木馬化されたアプリを通じて被害者に到達しました。MandiantのSnowflake調査は、インフォスティーラーによって盗まれた資格情報が企業の侵害連鎖の起点となった事例を追跡し、その一部は2020年まで遡っていました。攻撃に使用されたアカウントの79.7%は既知の過去の露出がありました。鍵は一度も交換されていません。

攻撃者は金庫を破ったわけではありません。机の引き出しに古い鍵が見つかったのです。

開発者にとってその引き出しはこう見えます：

| ローカルアーティファクト | 攻撃者が関心を持つ理由 |
| --- | --- |
| ブラウザのクッキー | ログインをバイパスし、場合によってはMFAをスキップできる。 |
| `.env`ファイル | APIキー、データベースURL、JWTシークレットが含まれる。 |
| クラウドCLI設定 | ラップトップの侵害がインフラ構造へのフルアクセスに繋がる。 |
| SSHキー | 依然として至る所に存在し、依然として強力で、依然としてマシン間でコピーされる。 |
| パッケージマネージャーのトークン | npmやPyPIの公開トークンはサプライチェーンへのアクセス権となる。 |
| データベースダンプ | 本番環境より保護が弱く、多くの場合より詳細な情報が含まれる。 |
| AIコーディングコンテキスト | アシスタントに「コンテキストのために」機密ファイルが渡されている可能性がある。 |

そしてバックアップもあります——誰かが`~/Downloads`に置き去りにした本番環境のエクスポートです。バックアップが安全でないのは不活性だからではありません。それは単に警報システムのない本番環境なのです。

## 「注意してください」という非解決策

「注意してください」というアドバイスは弱いです。これでは人間を境界と見なしています。

人間は境界ではありません。人間はトラフィックです。

境界は単調です：ファイルシステムの分離、静的暗号化されたシークレット、短時間の資格情報、ハードウェアベースの認証、そして偽のシークレットが触れた瞬間に発火するアラートです。

悪意のあるプロセスが実行された場合、あなたが午後の悪い出来事に終わるのか、それとも企業全体のインシデントになるのかを決める質問は以下の3つです：
1. このプロセスが**読み取れる**のは何ですか？
2. どの資格情報を**使用できる**ですか？
3. データを**どこに送信できる**ですか？

## 今すぐ最も効果的な対策

### デベロッパー・コンテナ — デフォルトで使用

[Development Containers](https://github.com/devcontainers/spec)は、多くのチームが実施していない最も効果的な変更です。デベロッパー・コンテナはプロジェクトの作業を分離されたDockerコンテナ内で実行します。`npm install`、`pip install`、`postinstall`スクリプト、AIシェルコマンド、VS Code拡張機能——これらすべては、他のマシンのファイルを見えないようにする「ワークスペース」やコンテナ内で処理されます。

<p class="inset">Claude Codeに任意のプロジェクトでDevContainersの設定を依頼してください。</p>

リポジトリをマウントします。そのプロジェクトに必要なシークレットのみを含めます。`~/.ssh`や`~/.aws`、ホームディレクトリを利便性のためにマウントしないでください。プロンプトに注入された指示は、エージェントが到達できるものにしかアクセスできません——それを単調なものにしましょう。

```jsonc
// .devcontainer/devcontainer.json
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ]
}
```

### キャンバートークン — 積極的に展開

[Canarytokens](https://canarytokens.org)は無料のデジタルトラップです。攻撃者が探す場所に偽でも説得力のあるシークレットを植えます。それが触れた瞬間にアラートが発生します——多くの場合数秒以内です。これは偽の現金束に染料パックを置くようなものです。

攻撃者は盗み出す前にまず調査します。その情報収集のパスがあなたのチャンスです。

最も魅力的に見えるファイルにキャンバートークンを配置してください：

```text
~/.aws/credentials          ← [billing-prod-legacy] プロファイルにキャンバーキーを追加
~/backups/customer-export-2024.sql   ← 内部にキャンバーURLを配置
~/.env.canary               ← すべてのリポジトリに偽の資格情報を配置
```

キャンバートークンは[canarytokens.org](https://canarytokens.org)で無料で入手可能で、セルフホスト可能であり、[Thinkst Canary](https://canary.tools)を通じて有料SaaSとしても利用できます。盗まれる可能性のある場所に展開しない正当な理由はありません。

### パッケージセキュリティツール

[Socket.dev](https://socket.dev)、[Snyk](https://snyk.io)、[Wiz](https://wiz.io)などのツールは、サプライチェーン攻撃が進行中に発見・ブロックされることが多いです。あなた自身が監視できないパッケージレジストリを監視します。フルタイムのセキュリティプログラムを維持できないチームにとって、これらは高リターンの早期警戒システムです。

### PNPMの最小リリース年齢設定

PNPMを使用する場合、最小リリース年齢を設定してください。新しく公開されたパッケージはサプライチェーン攻撃のリスクが最も高くなります——24時間未満のパッケージは実質的にコミュニティの検証を受けていません。`minimumReleaseAge`を分単位で設定してください：最低`1440`（1日）、理想は`2880`（2日）。

```yaml
minimumReleaseAge: 2880
minimumReleaseAgeStrict: true
minimumReleaseAgeIgnoreMissingTime: false
minimumReleaseAgeExclude:
  - 'typescript'
```

この設定は、特に次にインストールする前に発見・削除される新規パッケージ攻撃をブロックします。`minimumReleaseAgeExclude`は、即時更新が重要なパッケージ（コンパイラーやランタイム依存など）にのみ慎重に使用してください。

### 最もセキュリティが重要な環境の場合

国家機関、司法機関、金融取引インフラ、医療記録——これらの環境では、厳格なパッケージ評価・承認プロセスを採用することがあります。これは安全そうに見えます。トレードオフは深刻です：依存関係ツリーが徐々に古くなったソフトウェアに固まります。

ここでは時間は中立ではありません。古いバージョンは既知のCVEが蓄積されます。攻撃者は修正済みバージョンを研究して未修正のインスタンスを見つけるのです。「知っている悪魔の方が…」という考えは救い主になりません——単に攻撃者が最も長く習熟した脆弱性を教えてくれるだけです。

厳格なホワイトリストは運用スタッフが維持できる場合にのみ機能します。ほとんどのチームはそうではありません。他のすべてのチームにとって、レイヤードアプローチ——Dev Containers、キャンバートークン、パッケージセキュリティツール、短期間の資格情報——は、すべての依存関係を手動で監査できるかのように装うよりも現実的な防御です。

## あなたには数分しかない

キャンバーが作動したとき——またはGitHubが予期しないIPからトークン使用を通知したとき——あなたには窓があります。数分、多くて数時間です。1週間ではありません。

- **まず回転、その後で調査。** 何が起きたか理解する前にトークンを無効化してください。
- **攻撃者の永続性を確認。** 退職前に作成された新しいOAuthアプリ、IAMユーザー、デプロイキー、APIトークン。
- **アクティブなブラウザセッションを終了。** 重要なものすべてを強制ログアウトしてください。
- **誰かに報告。** 見証者とタイムスタンプがあるとセキュリティインシデントは改善します。

セキュリティ業界は検出について多く話します。検出後の20分間に何をすべきかについては話しません。あなたがデスクに座り、どのサービスにトークンがあるかを思い出そうとしているときです。

そのリストはアラートが発生する前に存在しているべきです。

## 価値のある基準

価値のある基準は「何でもかんでもクリックしないこと」ではありません。これはポスター向けのアドバイスであり、システムではありません。

悪い依存関係が他のプロジェクトのクラウド資格情報を取得できるべきではありません。プロンプト注入されたドキュメントがエージェントをあなたのホームディレクトリにリダイレクトすべきではありません。情報窃取ツールが平文のバックアップや長期有効なトークンを発見してもアラートをトリガーしないべきではありません。盗まれた資格情報は、完全な乗っ取りになる前に期限切れになる、MFA認証に失敗する、またはCanaryトークンにヒットする必要があります。

セキュリティは、人間が完璧であることを求めるのをやめ、妥協が利益を生まなくなるようにするときに向上します。

あなたのノートPCは今や運用環境の一部です。侵入者を追い出すだけでなく、自分がうっかり許可してしまった相手も捕まえるための単調な境界を与えてください。
