## You Are the Credential Warehouse

あなたのノートパソコンは単なるノートパソコンではない。キーボードを備えた認証情報倉庫だ――ブラウザセッション、SSH 鍵、`.env` ファイル、GitHub トークン、クラウド CLI、シェルアクセスを持つ AI コーディングツール、忘れ去られたデータベースエクスポートなどがそこにある。

従来のモデルは「本番は危険、ローカルは便利」だったが、そのモデルは終わった。

<p class="inset">
問題は「すべての悪いクリックを回避できるか」ではない。「たった一回の悪いクリックで全てを読み取り、全てを利用し、気付かれる前に去ってしまえるか」だ。
</p>

開発者は、外注先からの PDF、ターミナルに貼り付けさせる偽 CAPTCHA、`postinstall` スクリプトを持つパッケージ、タスク以上にファイルシステムへアクセスする AI コーディングセッションなど、見た目は普通に見えるものに遭遇する。あるパスはマルウェアをインストールし、あるパスは認証情報を盗み、あるパスはローカルでのエクスプロイトを必要としない――ユーザー自身が攻撃者のコマンドを実行してしまう。

これが現代の攻撃面だ。時に自分が侵害そのものになる。

## The Supply Chain Problem Is Impossibly Large

ここからが面白い。完全に安全になるために必要なのは、依存しているすべてのコンポーネント――メンテナ、履歴、トランジティブ依存関係――をマルチプラットフォームで徹底的に評価し、すべてのパッケージレジストリを横断して調査することだけだ。その後、依存関係ツリーが変化するたび、または更新が入るたびに同じ評価を繰り返す。サプライチェーン攻撃はまさに信頼の連鎖を突くものだからだ。

簡単だ。

そして、攻撃者は一度成功すればよい。一方、あなたは毎回完璧な防御を維持しなければならない。

LummaStealer — 広く利用されている情報窃盗マルウェアで、パスワード、ブラウザのクッキー、API キー、クラウド認証情報を静かに収集する — は、偽の CAPTCHA、毒された検索広告、トロイナイズされたアプリを通じて被害者に届いた。Mandiant の Snowflake 調査は、企業侵害の連鎖が情報窃盗マルウェアで盗まれた認証情報に遡ることを明らかにし、その一部は 2020 年までさかのぼる。攻撃に使用されたアカウントの少なくとも 79.7% が以前に漏洩が確認されていた。ロックは一度も変更されなかった。

攻撃者は倉庫を破ったわけではない。彼らは机の引き出しに残された古い鍵を見つけただけだ。

開発者にとって、その机の引き出しは次のような形になる：

| ローカルアーティファクト | 攻撃者が関心を持つ理由 |
| --- | --- |
| ブラウザクッキー | ログインをバイパスでき、時には MFA をスキップできる。 |
| `.env` ファイル | API キー、データベース URL、JWT シークレット。 |
| Cloud CLI 設定 | ラップトップの侵害がフルインフラへのアクセスに変わる。 |
| SSH 鍵 | 依然として至る所にあり、強力で、マシン間でコピーされることが多い。 |
| パッケージマネージャートークン | npm や PyPI の公開トークンはサプライチェーンへのアクセスになる。 |
| データベースダンプ | 本番環境より保護が甘く、しばしばより完全な情報が含まれる。 |
| AI コーディングコンテキスト | アシスタントに「コンテキスト用」として機密ファイルが渡されている可能性がある。 |

さらにバックアップがある――`~/Downloads` に落としたまま忘れた本番エクスポートだ。バックアップが安全になるわけではない。バックアップは惰性のままの本番環境であり、警報システムが付いていないだけだ。

## 「注意」だけでは解決にならない

「注意」は弱い助言だ。人間に境界を担わせようとする。

人間は境界ではない。人間はトラフィックだ。

境界は退屈である：ファイルシステムの分離、保存時暗号化されたシークレット、短命クレデンシャル、ハードウェアバックド認証、そして偽シークレットが触れられた瞬間に発火するアラート。

悪意あるプロセスが実行されたとき、午後の小さなトラブルか企業全体のインシデントかを決める質問は次の３つだ：
1. このプロセスは **何を読み取れる** か？
2. どの認証情報を **使用できる** か？
3. データを **どこへ送れる** か？

## 今すぐ取るべき最も効果的な手段

### Dev Containers — デフォルトで

[Development Containers](https://github.com/devcontainers/spec) は、ほとんどのチームがまだ実施していない、最もインパクトの大きい変更である。Dev Container はプロジェクト作業を分離された Docker コンテナ内で実行する。`npm install`、`pip install`、`postinstall` スクリプト、AI シェルコマンド、VS Code 拡張機能――すべてが「ワークスペース」またはコンテナ内で行われ、マシンの他の部分を見ることはできない。

<p class="inset">Claude Code に任せて、任意のプロジェクトで DevContainers を設定させよう。</p>

リポジトリをマウントする。プロジェクトに必要なシークレットだけを含める。利便性のために `~/.ssh`、`~/.aws`、ホームディレクトリ全体をマウントしない。プロンプトで注入された指示はエージェントが到達できる範囲だけに届く――それを退屈に保て。

```jsonc
// .devcontainer/devcontainer.json
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ]
}
```

### Canary Tokens — 積極的に配置

[Canarytokens](https://canarytokens.org) は無料のデジタルトリップワイヤーだ。攻撃者が目を付けそうな場所に、偽だが説得力のあるシークレットを配置する。そのシークレットが触れられた瞬間にアラートが届く――多くの場合数秒以内だ。偽の紙幣の束に染み込み式の爆弾を忍ばせるようなものと考えればよい。

攻撃者は盗む前に情報収集を行う。その偵察フェーズがあなたのウィンドウになる。

### カナリートークン — 積極的に配置

[Canarytokens](https://canarytokens.org) は無料のデジタルトリップワイヤーです。攻撃者が狙いそうな場所に、偽だが説得力のあるシークレットを置きます。そのシークレットが触れられた瞬間にアラートが届きます――多くの場合数秒以内です。偽の紙幣の束に染み込み式の爆弾を忍ばせるようなものと考えてください。

攻撃者は盗む前に情報収集を行います。その偵察フェーズがあなたのウィンドウになります。

```text
~/.aws/credentials          ← カナリキー付きの偽 [billing-prod-legacy] プロファイルを追加
~/backups/customer-export-2024.sql   ← カナリURLを埋め込む
~/.env.canary               ← すべてのリポジトリに偽クレデンシャルを配置
```

Canary tokens は [canarytokens.org](https://canarytokens.org) で無料で利用でき、セルフホストも可能です。また、[Thinkst Canary](https://canary.tools) の有料 SaaS でも提供されています。盗人が目を向ける場所ならどこでも展開しない理由はありません。

### パッケージセキュリティツール

[Socket.dev](https://socket.dev)、[Snyk](https://snyk.io)、[Wiz](https://wiz.io) などのツールは、サプライチェーン攻撃が進行中であることをいち早く検知し、ブロックします。自分たちで監視できないパッケージレジストリを監視してくれるのです。フルタイムのセキュリティチームを抱えられない組織にとって、これらは高いレバレッジを持つ早期警告システムとなります。

### PNPM の最小リリース年齢設定

PNPM を使っている場合は、最小リリース年齢を設定しましょう。新しく公開されたパッケージはサプライチェーン攻撃のリスクが最も高いウィンドウです――リリースから 24 時間未満のパッケージは事実上、コミュニティの目を通っていません。`minimumReleaseAge` を分単位で設定します：最低でも `1440`（1 日）、理想は `2880`（2 日）です。

```yaml
minimumReleaseAge: 2880
minimumReleaseAgeStrict: true
minimumReleaseAgeIgnoreMissingTime: false
minimumReleaseAgeExclude:
  - 'typescript'
```

この設定は、特に次のインストールまでに検出・取得される新規パッケージ攻撃を多く防ぎます。即時更新が重要なコンパイラやランタイム依存関係など、遅延よりも更新が優先されるパッケージに対しては `minimumReleaseAgeExclude` を慎重に使用してください。

### 最もセキュリティが重要な環境向け

情報機関、法執行機関、金融取引インフラ、医療記録――これらの環境では厳格なパッケージ評価・承認プロセスが採用されることがあります。安全に思えますが、トレードオフは深刻です：依存関係のツリーが徐々に老朽化し、古いソフトウェアへと固まっていきます。

時間は中立ではありません。古いバージョンは既知の CVE を蓄積します。攻撃者は修正済みバージョンを研究し、未パッチのインスタンスを狙います。また「知っている悪魔の方がマシ」という考えは救いになりません――それは単に攻撃者が最も長く熟知している脆弱性を示すだけです。

厳格な許可リストは、維持できる人員がいる場合にのみ機能します。多くのチームはそれを実行できません。ほとんどの組織にとっては、Dev Containers、カナリートークン、パッケージセキュリティツール、短命クレデンシャルという層状の防御が、すべての依存関係を手作業で監査できると信じるより現実的です。

## 数分の猶予

カナリートークンが作動したとき――あるいは GitHub が予期しない IP からトークンが使用されたと警告したとき――あなたにはウィンドウがあります。数分、場合によっては数時間です。1 週間ではありません。

- **まずローテーションし、後で調査する。** 何が起きたかを把握する前にトークンを失効させます。  
- **攻撃者の永続性を確認する。** 新規 OAuth アプリ、IAM ユーザー、デプロイキー、API トークンが残っていないか調べます。  
- **アクティブなブラウザセッションを終了する。** 重要なサービスからすべて強制ログアウトさせます。  
- **関係者に報告する。** インシデントは目撃者とタイムスタンプがあるほど対処しやすくなります。

セキュリティ業界は検知については多く語りますが、検知後 20 分間に何をすべきかについてはあまり語りません。デスクにひとりで座り、どのサービスにトークンが残っているか思い出そうとしているときです。

このリストはアラートが鳴る前に用意しておくべきです。

## 持っておく価値のある標準

標準とは「決して変なものをクリックしない」ことではありません。これはポスター向けの助言であり、システム向けの指針ではありません。

悪質な依存関係が他のプロジェクトのクラウド認証情報にアクセスできてはいけません。プロンプト注入されたドキュメントがエージェントをホームディレクトリへリダイレクトしてはいけません。情報窃盗マルウェアが平文のバックアップや長期間有効なトークンを見つけても、アラームが鳴らなければなりません。盗まれた認証情報は期限切れになるか、MFA に失敗するか、カナリアに引っ掛かるまでに、完全な乗っ取りに至らないようにすべきです。

人間に完璧さを求めるのをやめ、侵害の利益を減らす設計に転換したとき、セキュリティは向上します。

あなたのノートPC はすでに本番環境の一部です。侵入した攻撃者と、誤って自分で入れてしまった攻撃者の両方を捕らえる、退屈な境界を設定しましょう。
