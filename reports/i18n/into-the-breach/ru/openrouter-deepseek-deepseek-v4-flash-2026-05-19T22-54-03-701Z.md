## Вы — хранилище учётных данных

Ваш ноутбук — не ноутбук. Это хранилище учётных данных с клавиатурой: сессии браузера, SSH-ключи, `.env`-файлы, токены GitHub, облачные CLI, AI-инструменты для кода с доступом к оболочке, дампы баз данных, о которых вы забыли.

Раньше думали: продакшен — это опасно, локальная среда — удобно. Эта модель мертва.

<p class="inset">
Вопрос не в том, можете ли вы избежать каждого плохого клика. Вопрос в том, может ли один плохой клик прочитать всё, использовать всё и уйти, пока вы не заметили.
</p>

Разработчик сталкивается с тем, что выглядит достаточно обыденно: PDF от подрядчика, поддельная капча, которая просит вставить что-то в терминал, пакет с `postinstall`-скриптом, сессия AI-программирования, залезшая в файловую систему глубже, чем требовала задача. ## Проблема цепочки поставок непомерно велика

Вот самое смешное. Чтобы быть абсолютно защищённым, нужно провести глубокую многоплатформенную проверку безопасности каждой зависимости, на которую вы полагаетесь — их мейнтейнеров, их историю, их транзитивные зависимости — во всех реестрах пакетов. А потом повторять проверку каждый раз, когда ваше дерево зависимостей меняется или получает обновление, потому что именно так работают атаки на цепочку поставок: они эксплуатируют цепочку доверия.

Легко.

Ах да, атакующему нужно преуспеть лишь однажды. Вам нужно поддерживать безупречную защиту каждый раз.

Lumma Stealer — широко распространённый инфостилер, который тихо собирает пароли, куки браузера, ключи API и облачные учётные данные — попадал к жертвам через поддельные капчи, отравленные рекламные объявления и троянизированные приложения. Расследование Mandiant по Snowflake выявило каскад корпоративных утечек, восходящий к учётным данным, украденным инфостилерами, некоторые аж с 2020 года. Как минимум 79,7% аккаунтов, задействованных в атаке, были ранее скомпрометированы. Замки так и не сменили.

Злоумышленник не взломал хранилище. Он нашёл старые ключи в ящике стола.

Для разработчиков этот ящик стола выглядит так:

| Локальный артефакт | Почему это важно для атакующих |
| --- | --- |
| Куки браузера | Могут обойти вход и иногда пропустить MFA. |
| `.env` файлы | Ключи API, URL баз данных, секреты JWT. |
| Конфигурация облачного CLI | Превращает компрометацию ноутбука в полный доступ к инфраструктуре. |
| SSH-ключи | Всё ещё повсюду, всё ещё мощные, всё ещё копируются между машинами. |
| Токены менеджера пакетов | Ваш токен публикации npm или PyPI — это доступ к цепочке поставок. |
| Дампы баз данных | Менее защищены, чем продакшн, часто более полные. |
| Контекст AI-кодинга | Ассистенту могли передать чувствительные файлы «для контекста». |

А ещё есть резервные копии — экспорты продакшена, которые кто-то сбросил в `~/Downloads` и забыл. Резервная копия не безопаснее, потому что она инертна. Это просто продакшн без системы сигнализации.

## Псевдорешение «будь осторожен»

«Будь осторожен» — это слабый совет. Он возлагает на человека роль границы.

Люди — не границы. Люди — это трафик.

Границы — это скучно: изоляция файловой системы, секреты с шифрованием в покое, недолговечные учётные данные, аутентификация на аппаратном ключе и оповещения, которые срабатывают в момент обращения к поддельному секрету.

Если запускается вредоносный процесс, вопросы, которые определяют, будет ли у вас плохой день или инцидент масштаба всей компании, следующие:
1. Что этот процесс может **прочитать**?
2. Какие учётные данные он может **использовать**?
3. Куда он может **отправить данные**?

## Действия с максимальным рычагом прямо сейчас

### Dev Containers — по умолчанию

[Development Containers](https://github.com/devcontainers/spec) — это единственное изменение с наибольшим рычагом, которое большинство команд не внедряет. Dev Container запускает работу над проектом внутри изолированного Docker-контейнера. `npm install`, `pip install`, скрипты `postinstall`, AI-команды в оболочке, расширения VS Code — всё это происходит в «рабочей области» или контейнере, который не видит остальную часть вашей машины.

<p class="inset">Попросите Claude Code настроить DevContainers в любом проекте.</p>

Монтируйте репозиторий. Включайте только те секреты, которые нужны для этого проекта. Не монтируйте `~/.ssh`, `~/.aws` или вашу домашнюю директорию из соображений удобства. Инструкция, внедрённая через промпт, может добраться только до того, до чего может добраться агент — сделайте это скучным.

```jsonc
// .devcontainer/devcontainer.json
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ]
}
```

### Canary Tokens — агрессивное развёртывание

[Canarytokens](https://canarytokens.org) — это бесплатные цифровые растяжки. Посадите поддельный, но убедительный секрет там, где злоумышленник будет искать. Как только к нему прикоснутся, вы получите оповещение — часто в течение секунд. Думайте об этом как о красящем пакете в пачке фальшивых купюр.

Злоумышленники проводят инвентаризацию перед кражей. Этот разведывательный проход — ваше окно.

Разместите canary-токены в ваших самых заманчивых файлах:

```text
~/.aws/credentials          ← add a fake [billing-prod-legacy] profile with a canary key
~/backups/customer-export-2024.sql   ← canary URL inside
~/.env.canary               ← fake credentials in every repo
```

Canary-токены бесплатны на [canarytokens.org](https://canarytokens.org), могут быть размещены самостоятельно, а также доступны как платный SaaS через [Thinkst Canary](https://canary.tools). Нет ни одной веской причины не развернуть их везде, куда заглянет вор.

### Инструменты безопасности пакетов

Такие инструменты, как [Socket.dev](https://socket.dev), [Snyk](https://snyk.io) и [Wiz](https://wiz.io), часто первыми обнаруживают и блокируют атаки на цепочки поставок в процессе их выполнения. Они мониторят реестры пакетов, которые вы не можете отслеживать самостоятельно. Для команд, которые не могут позволить себе полноценную программу безопасности, это высокоэффективные системы раннего предупреждения.

### Настройки минимального возраста в PNPM

Если вы используете PNPM, установите минимальный возраст релиза. Недавно опубликованные пакеты представляют собой окно наибольшего риска для атак на цепочки поставок — пакет, существующий менее 24 часов, практически не подвергался общественному контролю. Установите `minimumReleaseAge` в минутах: не менее `1440` (один день), а в идеале `2880` (два дня).

```yaml
minimumReleaseAge: 2880
minimumReleaseAgeStrict: true
minimumReleaseAgeIgnoreMissingTime: false
minimumReleaseAgeExclude:
  - 'typescript'
```

Эта конфигурация блокирует множество атак на недавно опубликованные пакеты, особенно те, которые обнаруживают и отзывают до вашей следующей установки. Используйте `minimumReleaseAgeExclude` экономно для пакетов, где немедленные обновления важнее задержки, например для компилятора или зависимости времени выполнения, за которыми вы активно следите.

### Для наиболее критичных с точки зрения безопасности сред

Разведывательные агентства, правоохранительные органы, инфраструктура финансовой торговли, медицинские записи — в этих средах иногда применяют строгий процесс оценки и утверждения пакетов. Это звучит безопасно. Однако компромисс серьёзен: ваше дерево зависимостей медленно окаменевает, превращаясь в устаревшее программное обеспечение.

Время здесь не нейтрально. Старые версии накапливают известные CVE. Злоумышленники изучают исправленные версии, чтобы найти незаплатированные экземпляры. И «лучше знакомый дьявол» — не то спасение, на которое вы надеялись — это лишь говорит вам, какие уязвимости злоумышленник дольше всего осваивал.

Строгие белые списки работают, если у вас есть персонал для их поддержания. У большинства команд его нет. Для всех остальных многоуровневый подход — Dev Containers, canary-токены, инструменты безопасности пакетов, краткосрочные учётные данные — обеспечивает более реалистичную защиту, чем притворство, что вы можете вручную проверить каждую зависимость.

## У вас есть минуты

Когда срабатывает canary — или GitHub оповещает вас, что токен был использован с неожиданного IP — у вас есть окно. Минуты, возможно, несколько часов. Не неделя.

- **Сначала ротируйте, потом расследуйте.** Отзывайте токены до того, как поймёте, что произошло.
- **Проверьте на сохранение доступа злоумышленником.** Новые OAuth-приложения, пользователи IAM, ключи развёртывания, API-токены, созданные до того, как они ушли.
- **Завершите активные сессии браузера.** Принудительно выйдите из всего, что вам дорого.
- **Сообщите кому-то.** Инциденты безопасности улучшаются при наличии свидетелей и временных меток.

Индустрия безопасности много говорит об обнаружении. Она меньше говорит о том, что происходит в двадцать минут после обнаружения, когда вы одни за своим столом и пытаетесь вспомнить, для каких сервисов у вас есть токены.

Этот список должен существовать до того, как сработает оповещение.

## Стандарт, который стоит иметь

Стандарт — это не «никогда не кликай на странное». Это совет для плаката, а не для системы.

Плохая зависимость не должна иметь доступа к облачным учётным данным из других проектов. Документ с внедрённым промптом не должен перенаправлять агента в вашу домашнюю директорию. Инфостилер не должен находить резервные копии в открытом виде и долгоживущие токены без срабатывания тревоги. Украденные учётные данные должны истекать, проваливать MFA или натыкаться на canary, прежде чем превратиться в полный захват.

Безопасность повышается, когда мы перестаём требовать от людей безупречности и начинаем делать компрометацию менее выгодной.

Ваш ноутбук теперь — часть продакшена. Дайте ему те скучные границы, которые поймают как взломщика, так и того, кого вы случайно впустили сами.
