## Вы — это хранилище учетных данных

Ваш ноутбук — это не ноутбук. Это хранилище учетных данных с клавиатурой — сеансы браузера, SSH-ключи, файлы `.env`, токены GitHub, облачные CLI, инструменты ИИ с доступом к оболочке, экспорты баз данных, которые вы забыли существуют.

Старая модель: производство — опасно, локальная среда — удобна. Эта модель устарела.

<p class="inset">
Вопрос не в том, сможете ли вы избежать каждой плохой кликаний. Вопрос в том, может ли одна плохая кликания прочитать всё, использовать всё и уйти, прежде чем вы заметите.
</p>

Разработчик сталкивается с чем-то, что выглядит достаточно нормально: PDF от подрядчика, фальшивый CAPTCHA, требующий вставить что-то в терминал, пакет с `postinstall`-скриптом, сеанс кодирования ИИ, который глубже проник в файловую систему, чем требовалось задачей. Некоторые пути устанавливают вредоносное ПО. Некоторые крадут учетные данные. Некоторым не нужна локальная эксплуатация — пользователь сам запускает команду атакующего.

Это современная атакующая поверхность. Иногда вы сами являетесь уязвимостью.

## Проблема цепочки поставок колоссальна

Вот веселая часть. Чтобы быть полностью безопасным, вам нужно провести глубокую многоуровневую оценку безопасности каждой зависимости, на которую вы полагаетесь — их поддерживателей, историю, транзитивные зависимости — по всем пакетным реестрам. Затем повторите оценку каждый раз, когда ваше дерево зависимостей изменится или получит обновление, потому что именно так работают атаки на цепочку поставок: они эксплуатируют цепочку доверия.

Легко.

О, и атакующему нужно лишь один раз успешно атаковать. Вам нужно поддерживать идеальную защиту каждый раз.

Lumma Stealer — широко используемый инфостеалер, тихо собирающий пароли, куки браузера, API-ключи и облачные учетные данные — достиг жертв через фальшивые CAPTCHA, отравленные рекламные объявления и троянские приложения. Исследование Snowflake Mandianta проследило каскад корпоративных взломов до учетных данных, украденных инфостеалерами, некоторые из которых датируются 2020 годом. По меньшей мере 79,7% учетных записей, использованных в атаке, имели известное предыдущее воздействие. Замки никогда не менялись.

Атакующий не взломал склад. Он нашел старые ключи в ящике стола.

Для разработчиков этот ящик стола выглядит так:

| Локальный артефакт | Почему атакующие заинтересованы |
| --- | --- |
| Куки браузера | Могут обойти вход и иногда пропустить МФА. |
| `.env` файлы | Ключи API, URL баз данных, секреты JWT. |
| Конфигурация CLI облака | Превращает компрометацию ноутбука в полный доступ к инфраструктуре. |
| Ключи SSH | Все еще повсюду, все еще мощные, все еще копируются между машинами. |
| Токены менеджера пакетов | Ваш токен публикации в npm или PyPI — это доступ к цепочке поставок. |
| Выгрузки баз данных | Менее защищены, чем продакшн, часто более полные. |
| Контекст AI-кодирования | Ассистент мог получить чувствительные файлы "для контекста". |

А затем есть резервные копии — экспорт продакшн, который кто-то сбросил в `~/Downloads` и забыл. Резервная копия не безопаснее из-за своей пассивности. Это просто продакшн без системы оповещения.

## Несолюция "Будьте осторожны"

"Будьте осторожны" — слабый совет. Он требует, чтобы человек был границей.

Люди не являются границами. Люди — это трафик.

Границы скучны: изоляция файловой системы, шифрование секретов, краткосрочные учетные данные, аутентификация с поддержкой аппаратных ключей и оповещения, которые срабатывают в момент касания поддельного секрета.

Если запущен вредоносный процесс, эти три вопроса определят, будет ли у вас плохой день или инцидент, затронувший всю компанию:
1. Что этот процесс может **прочитать**?
2. Какие учетные данные он может **использовать**?
3. Куда он может **отправить данные**?

## Наиболее эффективные действия прямо сейчас

### Dev Containers — По умолчанию

[Development Containers](https://github.com/devcontainers/spec) — это самый эффективный сдвиг, который большинство команд не внедряет. Dev Container запускает работу над проектом в изолированном Docker-контейнере. `npm install`, `pip install`, скрипты `postinstall`, команды AI-оболочки, расширения VS Code — все это происходит в "рабочей области" или контейнере, который не видит остальную часть вашей машины.

<p class="inset">Попросите Claude Code настроить DevContainers в любом проекте.</p>

Подключите репозиторий. Включите только те секреты, которые нужны для этого проекта. Не подключайте `~/.ssh`, `~/.aws` или домашнюю директорию ради удобства. Инструкция, внедренная в приглашение, может достичь только то, что может достичь агент — сделайте это скучным.

```jsonc
// .devcontainer/devcontainer.json
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ]
}
```

### Canary Tokens — Агрессивно Развернутые

[Canarytokens](https://canarytokens.org) — это бесплатные цифровые сигнализации. Разместите поддельный, но убедительный секрет где-нибудь, где атакующий будет искать. В момент его касания вы получите оповещение — часто в течение нескольких секунд. Представьте, что вы оставляете маркировочную жидкость в поддельной стопке денег.

Атакующие проводят инвентаризацию перед кражей. Этот этап разведки — ваш шанс.

Разместите сигнализации в ваших самых привлекательных по виду файлах:

```text
~/.aws/credentials          ← добавьте поддельный профиль [billing-prod-legacy] с токеном-канарейкой
~/backups/customer-export-2024.sql   ← вставьте канарейский URL внутрь
~/.env.canary               ← поддельные учетные данные во всех репозиториях
```

Канарейки доступны бесплатно на [canarytokens.org](https://canarytokens.org), их можно развернуть самостоятельно, а также использовать как платный SaaS-сервис через [Thinkst Canary](https://canary.tools). Нет никаких причин не развертывать их везде, где злоумышленник будет искать.

### Инструменты безопасности для пакетов

Инструменты вроде [Socket.dev](https://socket.dev), [Snyk](https://snyk.io) и [Wiz](https://wiz.io) часто первыми обнаруживают и блокируют атаки на цепочки поставок. Они отслеживают реестры пакетов, которые вы не можете следить вручную. Для команд, не имеющих возможности поддерживать полноценную программу безопасности, это высокоприбыльные системы раннего предупреждения.

### Минимальные настройки возраста для PNPM

Если вы используете PNPM, установите минимальный возраст публикации. Новые пакеты представляют наибольший риск для атак на цепочки поставок — пакет, существующий менее 24 часов, практически не подвергался общественному контролю. Установите `minimumReleaseAge` в минутах: минимум `1440` (один день), а лучше `2880` (два дня).

```yaml
minimumReleaseAge: 2880
minimumReleaseAgeStrict: true
minimumReleaseAgeIgnoreMissingTime: false
minimumReleaseAgeExclude:
  - 'typescript'
```

Эта конфигурация блокирует множество атак с новыми пакетами, особенно те, которые обнаруживаются и удаляются до вашего следующего обновления. Используйте `minimumReleaseAgeExclude` осторожно для пакетов, где немедленные обновления важнее задержки, например, для компилятора или зависимостей среды выполнения, которые вы активно отслеживаете.

### Для самых критичных с точки зрения безопасности сред

Государственные разведывательные агентства, органы правопорядка, инфраструктура финансовых торгов, медицинские записи — эти среды иногда применяют строгие процессы оценки и одобрения пакетов. Это звучит безопасно. Но есть жертва: ваша зависимость постепенно превращается в устаревшее ПО.

Время здесь не нейтрально. Старые версии накапливают известные уязвимости. Атакующие изучают исправленные версии, чтобы найти непатченные экземпляры. И "лучше дьявола, которого знаешь" не спасет — это просто говорит, какие уязвимости атакующий изучал дольше всего.

Строгие allowlists работают, если у вас есть персонал для их поддержки. Большинству команд это не подходит. Для всех остальных подход с многослойной защитой — Dev Containers, канарейки, инструменты безопасности пакетов, краткосрочные учетные данные — обеспечивает более реалистичную защиту, чем иллюзия возможности вручную проверить каждую зависимость.

## У вас есть минуты

Когда канарейка сработает — или GitHub уведомит вас, что токен использовали с неожиданного IP — у вас будет окошко. Минуты, может быть, несколько часов. Не неделю.

- **Сначала поворачивайте, потом расследуйте.** Отменяйте токены до того, как поймете, что произошло.
- **Проверяйте устойчивость атакующего.** Новые OAuth-приложения, IAM-пользователи, ключи деплоя, API-токены, созданные до их ухода.
- **Убивайте активные сеансы браузера.** Принудительно выходите из всего, что важно.
- **Сообщите кому-нибудь.** Инциденты безопасности улучшаются при свидетелях и временных метках.

Индустрия безопасности много говорит о детектировании. Меньше говорит о том, что происходит в двадцать минут после обнаружения, когда вы один за своим столом, пытаясь вспомнить, для каких сервисов у вас есть токены.

Этот список должен существовать до того, как сработает тревога.

## Стандарт, заслуживающий внимания

Стандарт — это не "никогда не нажимайте на что-то странное". Это совет для плаката, а не для системы.

Плохая зависимость не должна иметь доступа к учетным данным других проектов. Документ с внедренным приглашением не должен перенаправлять агента в ваш домашний каталог. Инфостеалер не должен находить текстовые резервные копии и долговременные токены без срабатывания тревоги. Украденные учетные данные должны истечь, провалить MFA или сработать как канарейка, прежде чем привести к полному захвату.

Безопасность улучшается, когда мы перестаем требовать от людей идеальности и начинаем делать компромиссы менее выгодными.  
Ваш ноутбук теперь часть производства. Ограничьте его простыми правилами, которые блокируют как злоумышленника, проникшего внутрь, так и вас самих, если вы случайно открыли доступ.
