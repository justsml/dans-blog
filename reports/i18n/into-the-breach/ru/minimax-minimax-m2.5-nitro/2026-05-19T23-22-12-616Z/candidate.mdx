## Вы — хранилище учетных данных

Ваш ноутбук — это не ноутбук. Это хранилище учетных данных с клавиатурой — браузерные сессии, SSH-ключи, файлы `.env`, токены GitHub, облачные CLI-инструменты, AI-инструменты для написания кода с доступом к shell, дампы баз данных, о существовании которых вы забыли.

Старая модель звучала так: продакшен опасен, локальная среда удобна. Эта модель умер

Атакующий не взломал склад. Он нашёл старые ключи в ящике стола.

Для разработчиков этот ящик стола выглядит так:

| Локальный артефакт | Почему атакующим это важно |
| --- | --- |
| Куки браузера | Могут обойти вход и иногда пропустить MFA. |
| Файлы `.env` | API-ключи, URL баз данных, секреты JWT. |
| Конфигурация облачного CLI | Превращает компрометацию ноутбука в полный доступ к инфраструктуре. |
| SSH-ключи | Везде, всё ещё мощные, всё ещё копируются между машинами. |
| Токены менеджера пакетов | Ваш токен npm или PyPI — это доступ к цепочке поставок. |
| Дампы баз данных | Менее защищены, чем продакшен, но часто полнее. |
| AI-контекст для написания кода | Ассистенту могли передать чувствительные файлы «для контекста». |

А ещё есть бэкапы — экспорты продакшена, которые кто-то сбросил в `~/Downloads` и забыл. Бэкап не безопаснее просто потому, что он инертен. Это просто продакшен без системы оповещения.

## «Будьте осторожны» — не решение

«Будьте осторожны» — слабый совет. Он просит человека быть границей.

Люди — не границы. Люди — трафик.

Границы скучны: изоляция файловой системы, шифрование секретов при хранении, краткосрочные учётные данные, аутентификация на основе оборудования и оповещения, которые срабатывают в момент касания поддельного секрета.

Если запущен вредоносный процесс, вопросы, определяющие, будет ли у вас плохой день или инцидент масштаба компании:

1. Что этот процесс может **прочитать**?
2. Какие учётные данные он может **использовать**?
3. Куда он может **отправить данные**?

## Самые эффективные действия прямо сейчас

### Dev Containers — по умолчанию

[Development Containers](https://github.com/devcontainers/spec) — это единственное самое высокое Leverage изменение, которое большинство команд не применяет. Dev Container запускает проектную работу внутри изолированного Docker-контейнера. `npm install`, `pip install`, postinstall-скрипты, AI-команды shell, расширения VS Code — всё это происходит в «workspace» или контейнере, который не видит остальную часть вашей

```text
~/.aws/credentials          ← add a fake [billing-prod-legacy] profile with a canary key
~/backups/customer-export-2024.sql   ← canary URL inside
~/.env.canary               ← fake credentials in every repo
```

Канареечные токены бесплатны на [canarytokens.org](https://canarytokens.org), их можно развернуть на собственном хосте или использовать как платный SaaS через [Thinkst Canary](https://canary.tools). Нет веской причины не разместить их везде, где вор мог бы искать.

### Инструменты безопасности пакентов

Такие инструменты, как [Socket.dev](https://socket.dev), [Snyk](https://snyk.io) и [Wiz](https://wiz.io), часто первыми обнаруживают и блокируют атаки на цепочку поставок в процессе. Они следят за реестрами пакетов, которые вы не можете отслеживать самостоятельно. Для команд, которые не могут позволить себе полноценную программу безопасности, это системы раннего предупреждения с высоким эффектом.

### Настройки минимального возраста PNPM

Если вы используете PNPM, установите минимальный возраст release. Недавно опубликованные пакеты — это окно наивысшего риска для атаки на цепочку поставок — пакет, который существует менее 24 часов, по сути не прошёл никакого scrutiny со стороны сообщества. Установите `minimumReleaseAge` в минутах: минимум `1440` (один день), а лучше `2880` (два дня).

```yaml
minimumReleaseAge: 2880
minimumReleaseAgeStrict: true
minimumReleaseAgeIgnoreMissingTime: false
minimumReleaseAgeExclude:
  - 'typescript'
```

Эта конфигурация блокирует многие атаки с использованием вновь опубликованных пакетов, особенно те, которые обнаруживаются и удаляются до вашего следующего установки. Используйте `minimumReleaseAgeExclude` экономно для пакетов, где немедленные обновления важнее задержки, например для компилятора или зависимости времени выполнения, которую вы активно отслеживаете.

### Для наиболее критически важных сред

Разведывательные агентства, правоохранительные органы, инфраструктура финансовой торговли, медицинские записи — в этих срелях иногда применяется строгий процесс оценки и одобрения пакетов. Это звучит безопасно. Компромисс суров: ваше дерево зависимостей медленно кальцифицируется в устаревшее программное обеспечение.

Время здесь не нейтрально. Старые версии накапливают известные CVE. Атаки изучают исправленные версии, чтобы найти не пропатченные экземпляры. И "лучше знакомый дьявол" — это не то спасение, на которое вы надеялись — оно просто говорит вам, какими уязвимостями атака имела больше всего времени овладеть.

Строгие whitelist'ы работают, если у вас есть персонал для их поддержки. У большинства команд его нет. Для всех остальных многоуровневый подход — Dev Containers, канареечные токены, инструменты безопасности пакетов, краткосрочные учетные данные — обеспечивает более реалистичную защиту, чем притворяться, что вы можете проверить каждую зависимость вручную.

## У вас минуты

Когда срабатывает канарейка — или GitHub предупреждает вас, что токен был использован с неожиданного IP — у вас есть окно. Минуты, может быть несколько часов. Не неделя.

- **Сначала ротируйте, потом расследуйте.** Отзывайте токены, прежде чем поймете, что произошло.
- **Проверьте наличие persistance атакующего.** Новые OAuth-приложения, IAM-пользователи, deploy-ключи, API-токены, созданные до того, как они ушли.
- **Убейте активные браузерные сессии.** Принудительный logout везде, что вам важно.
- **Кому-нибудь расскажите.** Инциденты безопасности улучшаются со свидетелями и временными метками.

Индустрия безопасности много говорит о детекции. Она меньше говорит о том, что происходит в двадцать минут после детекции, когда вы сидите один за своим столом, пытаясь вспомнить, для каких сервисов у вас есть токены.

Этот список должен существовать до срабатывания алерта.

## Стандарт, который стоит иметь

Стандарт — это не "никогда не нажимайте на что-то странное". Это совет для плаката, а не для системы.

Плохая зависимость не должна иметь доступ к облачным учетным данным из других проектов. Документ с prompt-injection не должен перенаправлять агента в ваш домашний каталог. Инфостелер не должен находить plaintext-бэкапы и долгоживущие токены без срабатывания сигнализации. Скомпрометированные учетные данные должны истекать, не проходить MFA или попадать на канарейку до того, как произойдет полный захват.

Безопасность улучшается, когда мы перестаем требовать от людей совершенства и начинаем делать компрометацию менее выгодной.

Ваш ноутбук — это теперь продакшен. Дайте ему скучные границы, которые поймают и атакующего, который пробрался внутрь, и того, кого вы случайно впустили сами.
