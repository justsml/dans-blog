## Вы — это хранилище учетных данных

Ваш ноутбук — это не ноутбук. Это хранилище учетных данных с клавиатурой — сеансы браузера, SSH-ключи, файлы `.env`, токены GitHub, облачные CLI, инструменты ИИ с доступом к терминалу, экспорты баз данных, о которых вы забыли.

Старая модель: «производство опасно, локальная среда удобна». Эта модель устарела.

<p class="inset">
Вопрос не в том, сможете ли вы избежать каждой плохой кликанины. Вопрос в том, может ли одна плохая кликанина прочитать всё, использовать всё и уйти, прежде чем вы это заметите.
</p>

Разработчик сталкивается с чем-то, что выглядит достаточно обычно: PDF от подрядчика, фальшивый CAPTCHA, требующий вставить что-то в терминал, пакет с `postinstall`-скриптом, сессия ИИ, которая глубже залезла в файловую систему, чем это было необходимо. Некоторые пути устанавливают вредоносное ПО. Некоторые крадут учетные данные. Некоторым не нужна локальная эксплуатация — пользователь сам запускает команду атакующего.

Это современная атакующая поверхность. Иногда вы сами являетесь уязвимостью.

## Проблема цепочки поставок чрезвычайно масштабна

Вот веселая часть. Чтобы быть полностью безопасным, вам нужно выполнить глубокую многоуровневую оценку безопасности каждой зависимости, на которую вы полагаетесь — её поддерживателей, историю, транзитивные зависимости — во всех пакетных реестрах. Затем повторите оценку каждый раз, когда ваше дерево зависимостей изменится или получит обновление, потому что именно так работают атаки на цепочку поставок: они эксплуатируют цепочку доверия.

Просто.

О, и атакующему нужно преуспеть всего один раз. Вам нужно поддерживать идеальную защиту каждый раз.

Lumma Stealer — широко используемый инфостеалер, который бесшумно собирает пароли, куки браузера, API-ключи и учетные данные облака — достиг жертв через фальшивые CAPTCHA, отравленные рекламные объявления и троянизированные приложения. Расследование Snowflake компании Mandiant проследило каскад атак на предприятия до учетных данных, украденных инфостеалерами, некоторые из которых датируются 2020 годом. По меньшей мере 79,7% учётных записей, использованных в атаке, имели известное предыдущее воздействие. Замки никогда не меняли.

Атакующий не взломал склад. Он нашёл старые ключи в ящике стола.

Для разработчиков этот ящик стола выглядит так:

| Локальный артефакт | Почему атакующие заинтересованы |
| --- | --- |
| Куки браузера | Могут обойти вход и иногда пропустить МВА. |
| Файлы `.env` | API-ключи, URL баз данных, секреты JWT. |
| Конфигурация облачного CLI | Переводит компрометацию ноутбука в полный доступ к инфраструктуре. |
| SSH-ключи | По-прежнему повсюду, по-прежнему мощные, по-прежнему копируются между машинами. |
| Токены менеджера пакетов | Ваш токен npm или PyPI — это доступ к цепочке поставок. |
| Выгрузки баз данных | Менее защищены, чем продакшн, часто более полные. |
| Контекст ИИ-кодирования | Ассистент мог получить чувствительные файлы "для контекста". |

А затем идут резервные копии — экспорт продакшн-данных, который кто-то бросил в `~/Downloads` и забыл. Резервная копия не безопаснее из-за своей инертности. Это просто продакшн без системы оповещения.

## Несолидное решение "Будьте осторожны"

Совет "будьте осторожны" слаб. Он требует, чтобы человек был границей.

Люди — не границы. Люди — это трафик.

Границы — это скучно: изоляция файловой системы, шифрование секретов, краткосрочные учетные данные, аутентификация с аппаратной поддержкой и оповещения, срабатывающие в момент касания поддельного секрета.

Если запущен вредоносный процесс, вопросы, которые определяют, превратится ли это в плохой день или в инцидент по всей компании:
1. Что этот процесс может **прочитать**?
2. Какие учетные данные он может **использовать**?
3. Куда он может **отправить данные**?

## Наиболее эффективные меры прямо сейчас

### Dev Containers — По умолчанию

[Development Containers](https://github.com/devcontainers/spec) — это самый эффективный шаг, который большинство команд не выполняет. Dev Container запускает проектную работу внутри изолированного Docker-контейнера. `npm install`, `pip install`, скрипты `postinstall`, команды AI-оболочки, расширения VS Code — всё это происходит в «рабочей области» или контейнере, который не видит остальную часть вашей машины.

<p class="inset">Попросите Claude Code настроить DevContainers в любом проекте.</p>

Подмонтируйте репозиторий. Включите только те секреты, которые нужны для этого проекта. Не подключайте `~/.ssh`, `~/.aws` или домашнюю директорию ради удобства. Инструкция с подстановкой приглашения может достичь только того, что может достичь агент — сделайте это скучным.

```jsonc
// .devcontainer/devcontainer.json
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ]
}
```

### Canary Tokens — Агрессивно Развернутые

[Canarytokens](https://canarytokens.org) — это бесплатные цифровые сигнальные устройства. Заведите поддельный, но убедительный секрет куда-нибудь, где атакующий будет искать. В момент его касания вы получите оповещение — часто в течение нескольких секунд. Представьте, что оставили маркировочную краску в поддельной стопке денег.

Атакующие инвентаризируют, прежде чем украсть. Этот этап разведки — ваше окно возможностей.

Разместите сигнальные устройства в самых приметных файлах:

```text
~/.aws/credentials          ← добавьте поддельный профиль [billing-prod-legacy] с сигнальным ключом
~/backups/customer-export-2024.sql   ← вставьте сигнальный URL внутрь
~/.env.canary               ← поддельные учетные данные в каждом репозитории
```

Сигнальные устройства можно получить бесплатно на [canarytokens.org](https://canarytokens.org), они доступны для саморазмещения и как платная SaaS-сервис через [Thinkst Canary](https://canary.tools). Нет никаких причин не развертывать их повсеместно, где бы вор тоже смотрел.

### Инструменты безопасности для пакетов

Инструменты вроде [Socket.dev](https://socket.dev), [Snyk](https://snyk.io) и [Wiz](https://wiz.io) часто первыми обнаруживают и блокируют атаки через цепочки поставок. Они отслеживают реестры пакетов, которые вы не можете контролировать самостоятельно. Для команд, не имеющих возможности содержать полноценную программу безопасности, это высокоприбыльные системы раннего предупреждения.

### Минимальный возраст публикации в PNPM

Если вы используете PNPM, установите минимальный возраст публикации. Недавно опубликованные пакеты представляют наибольший риск для атак через цепочки поставок — пакет, существующий менее 24 часов, практически не прошел никакого сообщественного тестирования. Установите `minimumReleaseAge` в минутах: как минимум `1440` (один день), а лучше `2880` (два дня).

```yaml
minimumReleaseAge: 2880
minimumReleaseAgeStrict: true
minimumReleaseAgeIgnoreMissingTime: false
minimumReleaseAgeExclude:
  - 'typescript'
```

Эта конфигурация блокирует множество атак через недавно опубликованные пакеты, особенно те, которые обнаруживаются и удаляются до следующей установки. Используйте `minimumReleaseAgeExclude` осторожно для пакетов, где своевременные обновления важнее задержки, например, для компилятора или зависимости среды выполнения, которую вы активно отслеживаете.

### Для самых критичных с точки зрения безопасности сред

Разведывательные агентства, органы правопорядка, инфраструктура финансовых торгов, медицинские записи — эти среды иногда применяют строгие процессы оценки и одобрения пакетов. Это звучит безопасно. Но у этого есть сильный недостаток: ваша зависимость дерево медленно превращается в устаревшее ПО.

Здесь время не нейтрально. Старые версии накапливают известные уязвимости. Атакующие изучают исправленные версии, чтобы найти необновленные экземпляры. И "лучше дьявола, которого знаешь" не является спасением, на которое вы надеялись — это просто говорит, какие уязвимости атакующий изучал дольше всего.

Строгие белые списки работают, если у вас есть персонал для их поддержки. Большинству команд это не подходит. Для всех остальных подход с многослойной защитой — Dev Containers, сигнальные устройства, инструменты безопасности пакетов, краткосрочные учетные данные — обеспечивает более реалистичную защиту, чем иллюзия, что вы можете вручную проверить каждую зависимость.

## У вас есть минуты

Когда сигнальное устройство срабатывает — или GitHub предупреждает вас, что токен использовали с неожиданного IP — у вас есть окно. Минуты, может быть, несколько часов. Не неделю.

- **Сначала обновите, потом расследуйте.** Отмените токены, прежде чем понять, что произошло.
- **Проверьте настойчивость атакующего.** Новые OAuth-приложения, пользователи IAM, ключи деплоя, API-токены, созданные до их ухода.
- **Завершите активные сеансы браузера.** Принудительно выйдите из всего, что вам важно.
- **Сообщите кому-нибудь.** Инциденты безопасности улучшаются с наблюдателями и временными метками.

Индустрия безопасности много говорит о детектировании. Меньше говорит о том, что происходит в двадцать минут после обнаружения, когда вы одни за своим столом, пытаясь вспомнить, какие у вас есть токены.

Этот список должен существовать до того, как сработает предупреждение.

## Стандарт, заслуживающий внимания

Стандарт не "никогда не нажимайте на что-то странное". Это совет для плаката, а не для системы.

Плохая зависимость не должна иметь доступа к учетным данным облака из других проектов. Документ с внедренным приглашением не должен перенаправить агента в ваш домашний каталог. Инфостеалер не должен найти текстовые резервные копии и долговременные токены без срабатывания тревоги. Украденные учетные данные должны истечь, провалить MFA или сработать сигнальное устройство, прежде чем они приведут к полному захвату.

Безопасность улучшается, когда мы перестаем требовать от людей идеальности и начинаем делать компромиссы менее выгодными.  

Ваш ноутбук теперь часть инфраструктуры. Наделите его стандартными границами, которые обнаружат как атакующего, который проник, так и того, кого вы случайно сами пустили.
