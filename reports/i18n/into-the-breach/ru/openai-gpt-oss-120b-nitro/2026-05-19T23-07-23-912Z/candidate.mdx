## You Are the Credential Warehouse

Ваш ноутбук — это не просто ноутбук. Это склад учётных данных с клавиатурой — сессии браузера, SSH‑ключи, файлы `.env`, токены GitHub, облачные CLI, инструменты ИИ‑кодинга с доступом к оболочке, экспортированные базы данных, о которых вы забыли.

Старая модель была такова: продакшн опасен, локальная работа удобна. Эта модель завершена.

<p class="inset">
Вопрос не в том, сможете ли вы избежать каждой ошибочной щелчка. Вопрос в том, может ли один ошибочный щелчок прочитать всё, использовать всё и уйти, пока вы не заметите.
</p>

Разработчик сталкивается с тем, что выглядит достаточно обычным: PDF от подрядчика, поддельная CAPTCHA, предлагающая вставить что‑то в терминал, пакет с `postinstall`‑скриптом, сессия ИИ‑кодинга, зайдя глубже в файловую систему, чем требовалось задаче. Некоторые пути устанавливают вредоносное ПО. Некоторые крадут учётные данные. Некоторые не требуют локального эксплойта — пользователь сам исполняет команду злоумышленника.

Это современная поверхность атаки. Иногда вы сами становитесь утечкой.

## The Supply Chain Problem Is Impossibly Large

Вот интересная часть. Чтобы быть полностью в безопасности, вам нужно провести глубокую, многоплатформенную оценку безопасности каждой зависимости, на которой вы полагаетесь — их поддерживающих, их истории, их транзитивных зависимостей — во всех реестрах пакетов. Затем повторять эту оценку каждый раз, когда меняется дерево зависимостей или появляется обновление, потому что именно так работают атаки цепочки поставок: они используют цепочку доверия.

Легко.

О, и злоумышленнику достаточно один раз добиться успеха. Вам же нужно поддерживать безупречную защиту каждый раз.

Lumma Stealer — широко используемый инфостилер, который тихо собирает пароли, куки браузера, API‑ключи и облачные учётные данные — попал к жертвам через поддельные CAPTCHA, отравленные рекламные объявления и троянские приложения. Расследование Snowflake от Mandiant проследило каскад корпоративных утечек до учётных данных, украденных инфостилерами, некоторые из которых датируются 2020 годом. По крайней мере 79,7 % учётных записей, использованных в атаке, имели известные предыдущие компрометации. Замки никогда не менялись.

Атакующий не взломал склад. Он нашёл старые ключи в ящике стола.

Для разработчиков этот ящик выглядит так:

| Локальный артефакт | Почему это интересно атакующим |
| --- | --- |
| Куки браузера | Позволяют обойти вход и иногда миновать MFA. |
| `.env` файлы | API‑ключи, URL баз данных, JWT‑секреты. |
| Конфигурация облачных CLI | Превращает компрометацию ноутбука в полный доступ к инфраструктуре. |
| SSH‑ключи | Всё ещё везде, всё ещё мощные, всё ещё копируются между машинами. |
| Токены менеджеров пакетов | Ваш токен публикации npm или PyPI — доступ к цепочке поставок. |
| Дамп‑базы данных | Менее защищён, чем продакшн, часто более полон. |
| Контекст кода ИИ | Помощник мог получить чувствительные файлы «для контекста». |

И потом есть резервные копии — экспорт продакшна, оставленный кем‑то в `~/Downloads` и забытый. Резервная копия не безопаснее, просто потому что она инертна. Это просто продакшн без системы сигнализации.

## Нерешение «Будьте осторожны»

«Будьте осторожны» — слабый совет. Он перекладывает границу на человека.

Человек — не граница. Человек — трафик.

Границы скучны: изоляция файловой системы, шифрование‑на‑диске секретов, краткоживущие учётные данные, аппаратно‑поддерживаемая аутентификация и оповещения, срабатывающие в момент касания поддельного секрета.

Если запускается вредоносный процесс, вопросы, решающие, будет ли это лишь плохим днём или масштабным инцидентом компании, такие:
1. Что этот процесс **может читать**?
2. Какие учётные данные он **может использовать**?
3. Куда он **может отправлять данные**?

## Действия с наибольшим рычагом прямо сейчас

### Dev Containers — по умолчанию

[Development Containers](https://github.com/devcontainers/spec) — единственное изменение с наибольшим рычагом, которое большинство команд не применяют. Dev Container запускает работу над проектом внутри изолированного Docker‑контейнера. `npm install`, `pip install`, скрипты `postinstall`, команды ИИ‑оболочки, расширения VS Code — всё это происходит в «рабочем пространстве» или контейнере, который не видит остальную часть вашей машины.

<p class="inset">Попросите Claude Code настроить DevContainers в любом проекте.</p>

Монтируйте репозиторий. Добавляйте только те секреты, которые нужны конкретному проекту. Не монтируйте `~/.ssh`, `~/.aws` и ваш домашний каталог из удобства. Инструкция, внедрённая через подсказку, может достичь только того, к чему агент имеет доступ — сделайте это скучным.

```jsonc
// .devcontainer/devcontainer.json
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ]
}
```

### Canary Tokens — агрессивно развернутые

[Canarytokens](https://canarytokens.org) — бесплатные цифровые ловушки. Посадите поддельный, но правдоподобный секрет туда, где атакующий будет искать. Как только он будет затронут, вы получите оповещение — часто в течение секунд. Представьте себе, что это крошка‑краска в поддельной стопке банкнот.

Атакующие проводят инвентаризацию перед кражей. Этот разведывательный проход — ваше окно возможностей.

Разбросайте канарейки в самые заманчиво выглядящие файлы:

```text
~/.aws/credentials          ← добавьте фиктивный профиль [billing-prod-legacy] с канарейским ключом
~/backups/customer-export-2024.sql   ← канарейский URL внутри
~/.env.canary               ← поддельные учётные данные в каждом репозитории
```

Канарейские токены бесплатны на [canarytokens.org](https://canarytokens.org), их можно развернуть самостоятельно, а также приобрести как SaaS у [Thinkst Canary](https://canary.tools). Нет веской причины не использовать их везде, где потенциальный злоумышленник будет искать.

### Инструменты безопасности пакетов

Инструменты вроде [Socket.dev](https://socket.dev), [Snyk](https://snyk.io) и [Wiz](https://wiz.io) часто первыми обнаруживают и блокируют атаки на цепочку поставок в процессе их выполнения. Они мониторят реестры пакетов, которые вы сами не в состоянии отслеживать. Для команд, не способных содержать полноценную программу безопасности, такие сервисы являются высокоэффективными системами раннего предупреждения.

### Настройки минимального возраста в PNPM

Если вы используете PNPM, задайте минимальный возраст выпуска. Ново опубликованные пакеты — самая уязвимая точка для атак на цепочку поставок: пакет, существующий менее 24 часов, практически не прошёл проверку сообщества. Установите `minimumReleaseAge` в минутах: минимум `1440` (один день) и, желательно, `2880` (два дня).

```yaml
minimumReleaseAge: 2880
minimumReleaseAgeStrict: true
minimumReleaseAgeIgnoreMissingTime: false
minimumReleaseAgeExclude:
  - 'typescript'
```

Эта конфигурация блокирует многие атаки с новыми пакетами, особенно те, которые обнаруживаются и вытягиваются до вашего следующего `install`. Используйте `minimumReleaseAgeExclude` экономно — только для пакетов, где мгновенные обновления важнее задержки, например компилятор или runtime‑зависимость, которую вы активно отслеживаете.

### Для самых критичных с точки зрения безопасности окружений

Разведывательные агентства, правоохранительные органы, инфраструктура финансовой торговли, медицинские записи — в этих средах иногда вводят строгий процесс оценки и одобрения пакетов. На первый взгляд это выглядит безопасно. Но компромисс тяжёлый: дерево зависимостей медленно затвердевает в устаревшее ПО.

Время здесь нейтральным не является. Старые версии накапливают известные CVE. Атакующие изучают исправленные версии, чтобы найти непатченные экземпляры. И поговорка «лучше дьявол, которого знаешь», не спасает — она лишь указывает, какие уязвимости злоумышленник имел возможность изучить дольше всех.

Строгие белые списки работают, если у вас есть персонал для их поддержки. Большинство команд этого не имеют. Для остальных более реалистичная защита — слоистый подход: Dev Containers, канарейские токены, инструменты безопасности пакетов, краткоживущие учётные данные — лучше, чем пытаться вручную аудировать каждую зависимость.

## У вас есть минуты

Когда канарейка срабатывает — или GitHub оповещает, что токен использован с неожиданного IP — у вас есть окно. Минуты, возможно, несколько часов. Не неделя.

- **Сначала ротация, потом расследование.** Отзовите токены, пока не поймёте, что произошло.
- **Проверьте наличие устойчивости атакующего.** Новые OAuth‑приложения, IAM‑пользователи, deploy‑ключи, API‑токены, созданные до их ухода.
- **Прервите активные браузерные сессии.** Принудительно разлогиньте всё, что вам важно.
- **Сообщите кому‑нибудь.** Инциденты в безопасности улучшаются, когда есть свидетели и метки времени.

Индустрия безопасности много говорит о детекции. Меньше — о том, что происходит в двадцать минут после обнаружения, когда вы одни за столом пытаетесь вспомнить, для каких сервисов у вас есть токены.

Этот список должен быть готов задолго до срабатывания сигнала.

## Стандарт, который стоит иметь

Стандарт — не «никогда не нажимать ничего подозрительного». Это совет для плакатов, а не для систем.

Плохая зависимость не должна иметь доступа к облачным учётным данным из других проектов. Документ, внедрённый через подсказку, не должен перенаправлять агент в ваш домашний каталог. Инфостилер не должен находить открытые бэкапы и долгоживущие токены без срабатывания тревоги. Украденные учётные данные должны истекать, проваливаться MFA или попадать в канарейку до того, как превратятся в полный захват.

Безопасность повышается, когда мы перестаём требовать от людей совершенства и начинаем делать компромисс менее выгодным.

Ваш ноутбук теперь является частью продакшена. Оборудуйте его скучными границами, которые поймают как злоумышленника, пробившегося внутрь, — так и того, кого вы случайно пропустили сами.
