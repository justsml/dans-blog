## Вы — хранилище учётных данных

Ваш ноутбук — это не ноутбук. Это хранилище учётных данных с клавиатурой — сессии браузера, SSH-ключи, `.env` файлы, GitHub токены, облачные CLI, AI-инструменты для программирования с доступом к shell, экспорты базы данных, которые вы забыли существовали.

Старая модель была: production опасен, локальная разработка удобна. Эта модель закончилась.

<p class="inset">
Вопрос не в том, можете ли вы избежать каждого плохого клика. Вопрос в том, может ли один плохой клик прочитать всё, использовать всё и уйти до того, как вы заметите.
</p>

Разработчик встречает что-то, что выглядит достаточно нормальным: PDF от контрактора, фальшивая CAPTCHA, просящая вставить что-то в терминал, пакет с скриптом `postinstall`, сессия AI-программирования, которая зашла дальше в файловую систему, чем требовала задача. Некоторые пути устанавливают вредоносное ПО. Некоторые крадут учётные данные. Некоторые не нуждаются в локальном exploit — пользователь сам запускает команду атакующего.

Это современная поверхность атаки. Иногда вы — точка взлома.

## Проблема цепочки поставок невозможно велика

А вот интересная часть. Чтобы быть полностью безопасным, всё что вам нужно сделать — это провести глубокую, многоплатформенную оценку безопасности каждой зависимости, которую вы используете — их поддерживающих, их истории, их транзитивных зависимостей — через каждый пакетный реестр. Затем повторять оценку каждый раз, когда ваше дерево зависимостей меняется или получает обновление, потому что именно так работают атаки на цепочку поставок: они используют цепочку доверия.

Легко.

А ещё, атакующий должен успешно сделать это только один раз. Вы должны поддерживать идеальную защиту каждый раз.

Lumma Stealer — широко используемый инфостилер, который тихо собирает пароли, браузерные куки, API-ключи и облачные учётные данные — достигал жертв через фальшивые CAPTCHA, отравленные поисковые рекламные объявления и троянские приложения. Исследование Mandiant по Snowflake отследило каскад взломов предприятий до учётных данных, украденных инфостилерами, некоторые ещё в 2020 году. По крайней мере 79.7% аккаунтов, использованных в атаке, имели известное предыдущее раскрытие. Замки никогда не менялись.

Злоумышленник не взламывал склады. Он просто нашёл старые ключи в ящике стола.

Для разработчиков этот ящик выглядит так:

| Локальный артефакт | Почему это важно для атакующих |
| --- | --- |
| Cookies браузера | Обход логина, иногда и MFA. |
| Файлы `.env` | API-ключи, URL базы данных, JWT-секреты. |
| Конфигурация CLI облачных сервисов | Компрометация ноутбука превращается в полный доступ к инфраструктуре. |
| SSH-ключи | Ещё повсеместны, ещё мощны, ещё копируются между машинами. |
| Токены менеджеров пакетов | Токен публикации в npm или PyPI — это доступ к цепочке поставок. |
| Дампы базы данных | Часто менее защищены, чем production, но более полные. |
| Контекст AI-помощников для кода | Ассистенту могли передать чувствительные файлы "для контекста". |

А ещё есть резервные копии — экспорты из production, которые кто-то бросил в `~/Downloads` и забыл. Резервная копия не безопаснее просто потому, что она неактивна. Это просто production без системы сигнализации.

## Не-решение «Будьте осторожны»

«Будьте осторожны» — слабый совет. Он просит человека быть границей.

Люди — не границы. Люди — это трафик.

Границы — это скучно: изоляция файловой системы, секреты, зашифрованные при хранении, краткосрочные учётные данные, аутентификация на основе аппаратного обеспечения и алерты, которые срабатывают в момент обращения к фальшивому секрету.

Если запускается вредоносный процесс, вопросы, которые определяют, будет у вас плохой день или инцидент в масштабе компании, следующие:
1. Что этот процесс может **читать**?
2. Какие учётные данные он может **использовать**?
3. Куда он может **передать данные**?

## Самые эффективные меры прямо сейчас

### Dev Containers — По умолчанию

[Development Containers](https://github.com/devcontainers/spec) — это самое эффективное изменение, которое большинство команд не применяет. Dev Container запускает работу над проектом внутри изолированного Docker-контейнера. `npm install`, `pip install`, скрипты `postinstall`, команды AI в shell, расширения VS Code — всё это происходит в «рабочем пространстве» или контейнере, который не видит остальную часть вашей машины.

<p class="inset">Попросите Claude Code настроить DevContainers в любом проекте.</p>

Подключите репозиторий. Включите только секреты, необходимые для этого проекта. Не подключайте `~/.ssh`, `~/.aws` или домашнюю директорию ради удобства. Инструкция, внедренная через промпт, может достичь только того, что доступно агент — сделайте это скучным.

```jsonc
// .devcontainer/devcontainer.json
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ]
}
```

### Canary Tokens — Агрессивное размещение

[Canarytokens](https://canarytokens.org) — бесплатные цифровые сигнальные устройства. Поместите фальшивый, но убедительный секрет там, где злоумышленник будет его искать. В момент обращения к нему вы получите алерт — часто в течение секунд. Представьте это как оставление красящего пакета в фальшивой стопке банкнот.

Атакующие проводят инвентаризацию перед тем, как украсть. Этот этап reconnaissance — ваш шанс.

Размещайте канарейки в ваших самых привлекательных файлах:

```text
~/.aws/credentials          ← add a fake [billing-prod-legacy] profile with a canary key
~/backups/customer-export-2024.sql   ← canary URL inside
~/.env.canary               ← fake credentials in every repo
```

Canary tokens бесплатны на [canarytokens.org](https://canarytokens.org), их можно разместить самостоятельно, а также доступны как платный SaaS через [Thinkst Canary](https://canary.tools). Нет никакой веской причины не размещать их там, где будет искать вор.

### Инструменты безопасности пакетов

Инструменты, такие как [Socket.dev](https://socket.dev), [Snyk](https://snyk.io) и [Wiz](https://wiz.io), часто первыми обнаруживают и блокируют атаки на цепочку поставок в процессе. Они отслеживают реестры пакетов, которые вы не можете контролировать самостоятельно. Для команд, которые не могут позволить себе полноценную программу безопасности, это высокоэффективные системы раннего предупреждения.

### Настройки минимального возраста для PNPM

Если вы используете PNPM, установите минимальный возраст выпуска. Новые опубликованные пакеты представляют самый высокий риск для атак на цепочку поставок — пакет, существующий менее 24 часов, практически не подвергался проверке сообщества. Установите `minimumReleaseAge` в минутах: минимум `1440` (один день), и идеально `2880` (два дня).

```yaml
minimumReleaseAge: 2880
minimumReleaseAgeStrict: true
minimumReleaseAgeIgnoreMissingTime: false
minimumReleaseAgeExclude:
  - 'typescript'
```

Эта конфигурация блокирует многие атаки через новые пакеты, особенно те, которые обнаруживаются и удаляются до вашей следующей установки. Используйте `minimumReleaseAgeExclude` с осторожностью для пакетов, где немедленные обновления важнее, чем отсрочка, например, для компилятора или зависимости среды выполнения, которую вы активно отслеживаете.

### Для наиболее критичных к безопасности окружений

Агентства разведки, правоохранительные органы, инфраструктура финансовых торгов, медицинские записи — эти окружения иногда применяют строгий процесс оценки и утверждения пакетов. Это звучит безопасно. Но компромисс серьезен: ваше дерево зависимых пакетов медленно превращается в устаревшее программное обеспечение.

Время здесь не нейтрально. Старые версии накапливают известные CVE. Атакующие изучают исправленные версии, чтобы найти непатченные инстансы. И «лучше тот дьявол, которого ты знаешь» — не то спасение, на которое вы надеялись — это лишь указывает, какие уязвимости атакующий изучал наиболее долго.

Строгие списки разрешений работают, если у вас есть персонал для их поддержки. Большинство команд не имеют. Для всех остальных многоуровневый подход — Dev Containers, canary tokens, инструменты безопасности пакетов, кратковременные credentials — обеспечивает более реалистичную защиту, чем попытка аудита каждой зависимости вручную.

## У вас есть минуты

Когда canary срабатывает — или GitHub предупреждает вас, что токен был использован с неожиданного IP — у вас есть окно. Минуты, возможно, несколько часов. Не неделя.

- **Сначала замените, потом исследуйте.** Отзовите токены до того, как вы поняли, что произошло.
- **Проверьте наличие persistence атакующего.** Новые OAuth приложения, IAM пользователи, deploy keys, API токены, созданные перед их уходом.
- **Убить активные сессии браузера.** Принудительно выйти из всего, что вам важно.
- **Сообщите кому-то.** Инциденты безопасности улучшаются с наличием witnesses и timestamps.

Индустрия безопасности много говорит об обнаружении. Она меньше говорит о том, что происходит в течение двадцати минут после обнаружения, когда вы один у своего стола, пытаясь вспомнить, для каких сервисов у вас есть токены.

Этот список должен существовать до того, как сработает алерт.

## Стандарт, который стоит иметь

Стандарт не «никогда не кликать на странное». Это совет для плаката, не для системы.

Плохая зависимость не должна получать доступ к облачным учетным данным из других проектов. Документ с инъекцией prompt не должен перенаправлять агента в ваш домашний директорий. Информационный сборщик не должен обнаруживать резервные копии в виде plaintext и долгоживущие токены без срабатывания сигнала тревоги. Украденные учетные данные должны истекать, не пройти MFA или встретить canary до того, как они позволят полный захват.

Безопасность улучшается, когда мы прекращаем требовать от людей быть идеальными и начинаем делать компромисс менее прибыльным.

Ваш ноутбук теперь часть производственной среды. Обеспечьте ему простые границы, которые захватят и атакующего, который прорвался внутрь, и того, которого вы случайно сами допустили.
