## Установите уровень угрозы

Большинство людей представляют себе драматическую атаку — zero-day, государственный актор с приглашением в календаре. Что-то настолько экзотическое, что обычная инженерная дисциплина кажется неуместной.

Скучная версия полезнее.

Разработчик сталкивается с чем-то, что выглядит достаточно обыденно:

- PDF-счёт от подрядчика
- SMS о доставке или предупреждение об аккаунте
- поддельная CAPTCHA, которая просит вставить команду в терминал
- отравленная реклама в поиске для инструмента, который он всё равно собирался установить
- расширение браузера, которое тихо запрашивает чуть больше, чем нужно
- pull request, добавляющий dev-зависимость с postinstall-скриптом
- сессия AI-кодинга, которая читает больше файловой системы, чем требовала задача
- workflow GitHub Actions, который сливает секреты через переменную окружения, которой их никогда не должны были видеть
- инъекция промпта в документ, веб-страницу или репозиторий, перенаправляющая следующее действие AI-агента

Некоторые из этих путей устанавливают вредоносное ПО. Некоторые крадут учётные данные через фишинг. Некоторым вообще не нужна локальная уязвимость — пользователь сам запускает команду атакующего вручную.

Отчёт Microsoft о Lumma Stealer — полезный снимок. Lumma — широко используемый *инфостилер* — вредоносное ПО, которое незаметно собирает пароли, куки браузера, ключи API и криптокошельки с заражённой машины. Он попадает к жертвам через фишинговые письма, вредоносную рекламу, поддельные CAPTCHA и троянизированные приложения. Интересна не марка Lumma — интересна стратегия: атакующим не нужна одна идеальная дверь, когда пользователи весь день перемещаются по городу полунадёжных дверей.

### Инъекция промптов

Вредоносная инструкция, спрятанная в файле, README, описании PR или комментарии, может перенаправить поведение агента. Агент читает документ как содержимое. Скрытая инструкция — тоже содержимое. Если модель обрабатывает внедрённый текст как команду, агент может выполнить действия, которые пользователь никогда не задумывал — читать файлы, вызывать инструменты или следовать цепочке инструкций, которая ему не принадлежала.

Для этого не нужна скомпрометированная модель. Нужен документ, который агенту поручили обработать.

Практические выводы:

- Не давайте агентам неограниченный доступ к файловой системе «для контекста». Контекст не бесплатен.
- Проверяйте, что агент предлагает, прежде чем он выполнит действие, особенно для файлов, к которым он обратился без явного запроса.
- Относитесь скептически, если агент внезапно хочет прочитать учётные данные, отправить сетевые запросы или выполнить что-то, «что он нашёл, изучая проект».
- Держите сессии AI-оболочки внутри Dev Containers с узкими монтированиями. Внедрённая инструкция может воздействовать только на то, до чего агент может дотянуться.

### GitHub CI/CD

GitHub Actions — мощный, надёжный и часто неправильно настраиваемый инструмент. Последствия часто оказываются теми же, что и при компрометации ноутбука: учётные данные, исходный код и доступ к развёртыванию.

**Отравленные сторонние действия.** Ваш рабочий процесс использует `uses: some-org/some-action@v2`. Теги версий, такие как `@v2`, — это перемещаемые метки: если вышестоящий репозиторий скомпрометирован или этот тег перенаправлен на вредоносный коммит, ваш рабочий процесс выполнит код атакующего с секретами вашего репозитория. Исправление: фиксируйте действия на полный SHA коммита.

**Злоупотребление триггером pull request.** `pull_request_target` — это триггер, который запускает рабочие процессы с доступом к секретам базового репозитория — даже если PR приходит от внешнего участника. Неосторожные рабочие процессы могут раскрыть эти секреты ненадёжному коду. Это задокументированная «грабля» GitHub.

**Внедрение в рабочий процесс через ненадёжный ввод.** Интерполяция `${{ github.event.pull_request.title }}` непосредственно в шаг `run:` позволяет атакующему создать заголовок PR, который внедряет команды оболочки. Всегда передавайте значения, контролируемые пользователем, через промежуточную переменную окружения.

**Эксфильтрация секретов из форков.** Форкнутые PR по умолчанию не получают секреты репозитория, но неправильные настройки вокруг `pull_request_target` и правил защиты окружения могут это изменить.

Практический минимум:

- Фиксируйте сторонние действия на полные SHA коммитов.
- Никогда не интерполируйте поля `github.event` непосредственно в шаги `run:`.
- Храните производственные секреты в окружениях с правилами защиты и обязательными рецензентами.
- Аудируйте, кто может запускать рабочие процессы с доступом к чувствительным секретам.
- Используйте кратковременный обмен учётными данными (OIDC) для доступа к облаку вместо хранения долгоживущих секретов в CI.

## Полный сценарий захвата

Фраза «утечка данных» слишком узка для того, что последует.

1. **Первое касание**: пользователь открывает файл, кликает по ссылке, устанавливает инструмент, запускает скопированную команду или попадает на скомпрометированную страницу.
2. **Инвентаризация**: вредоносный процесс обследует машину — каталоги, конфигурационные файлы, данные браузера, переменные окружения. Он выясняет, что у него есть.
3. **Локальный сбор**: сессии браузера, конфигурационные файлы, файлы `.env`, токены, SSH-ключи, история команд и каталоги проектов копируются наружу.
4. **Облачный переход**: украденные учётные данные используются для входа в облачные аккаунты, GitHub, CI-системы или SaaS-инструменты — часто в течение нескольких минут.
5. **Зачистка резервных копий**: локальные экспорты, облачные хранилища, CI-артефакты и снимки баз данных становятся целями, потому что они менее защищены, чем продакшен.
6. **Постоянство**: прежде чем окно возможностей закроется, атакующий создаёт новые API-ключи, OAuth-приложения или сервисные аккаунты — чтобы иметь возможность вернуться даже после смены паролей.
7. **Вымогательство или перепродажа**: данные монетизируются напрямую, продаются как доступ или сохраняются для будущей кампании.

Ваш ноутбук — это брокер удостоверений. Он доказывает вашу личность каждой системе, которой вы пользуетесь. Если атакующий украдёт достаточно таких доказательств, он сможет явиться, выдавая себя за вас.

Обратите внимание на второй шаг: **сначала инвентаризация**. Большинство атакующих просматривают, прежде чем украсть. Они осматриваются, открывают каталоги, проверяют, какие учётные данные присутствуют.

Именно это окно и призваны использовать канареечные токены.

## Лучшая рамка: Читать, Использовать, Эксфильтровать

Любая защита рабочей станции должна отвечать на три вопроса:

1. Что этот процесс может **читать**?
2. Какие учётные данные он может **использовать**?
3. Куда он может **отправлять данные**?

Большинство советов по безопасности рабочих станций останавливаются на первом. Обновляйте программное обеспечение. Не открывайте подозрительные вложения. Используйте антивирус. Хорошо, да, очевидно.

Но если вредоносный процесс всё же запускается, вопросы два и три решают, будет ли у вас плохой день или инцидент уровня компании.

Может ли он прочитать `~/.aws/credentials`? Может ли использовать GitHub-токен? Может ли открыть расширение менеджера паролей? Может ли загрузить 3 ГБ на случайный хост, пока никто не заметил?

Эта рамка превращает угрозу из туманной машины в чек-лист с зубами.

## Что бы я сделал в первую очередь

Если бы я ужесточал программу защиты рабочих станций разработчиков, не превращая компанию в грустный аэропорт, я бы начал с этого.

### 1. Перенесите рискованную работу в Dev Containers

Используйте [Development Containers](https://github.com/devcontainers/spec) для проектной работы, требующей зависимостей, инструментов сборки, установки пакетов или AI-помощи в командной строке. Dev Container — это локальный Docker-контейнер, который выступает изолированным рабочим пространством для вашего проекта: он не видит остальную часть вашей машины, если вы явно не смонтируете её.

Выигрыш: `npm install`, `pip install`, `go generate`, `cargo build` и всё, что модель захочет запустить, происходит в рабочем пространстве, которое не владеет автоматически всей вашей домашней директорией.

Монтируйте репозиторий. Монтируйте только те секреты, которые нужны для этого проекта. Избегайте монтирования `~/.ssh`, `~/.aws`, `~/Downloads` и всей домашней папки ради удобства.

```jsonc
// .devcontainer/devcontainer.json — только узкие монтирования
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "workspaceFolder": "/workspaces/app",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ],
  "containerEnv": {
    "NODE_ENV": "development"
  },
  "postCreateCommand": "bun install"
}
```

Внедряйте ограниченные учётные данные. Предпочитайте короткоживущие токены. Где возможно — доступ только на чтение. Инструкция, внедрённая через промпт, может добраться лишь до того, до чего может добраться агент — сделайте это скучным.

### 2. Шифруйте локальные секреты вместо поклонения `.env`

Файлы `.env` в открытом виде удобны, потому что файлы удобны. Атакующие тоже любят файлы.

[VarLock](https://varlock.dev/guides/secrets/) рассматривает чувствительность как структурированные метаданные — вы помечаете, какие значения являются чувствительными, он шифрует их локально, скрывает из вывода консоли и сканирует на наличие открытых вхождений значений, которые должны были быть секретными.

```dotenv
# .env.schema
# @sensitive
STRIPE_SECRET_KEY=

# @sensitive
DATABASE_URL=
```

Секреты должны знать, что они секреты. Это не защитит секрет, уже загруженный в скомпрометированный процесс, но уменьшит количество ценных файлов в открытом виде, ожидающих, когда станут чьим-то инвентарём.

### 3. Разместите канареечные токены везде, куда заглянет вор

Это уровень, который большинство команд пропускает, и, пожалуй, самый полезный немедленно.

[Canarytokens](https://help.canary.tools/hc/en-gb/articles/10905485310109-Canarytoken-Overview-and-Use-Cases) — это цифровые растяжки. Разместите поддельный, но убедительный секрет, API-ключ или URL там, куда может заглянуть атакующий. Если его когда-нибудь затронут, вы получите оповещение — часто в течение секунд. Думайте об этом как о пакете с краской внутри фальшивой пачки купюр: как только кто-то её открывает, вы узнаёте.

Вспомните второй шаг паттерна захвата: **сначала инвентаризация**. Атакующие просматривают, прежде чем украсть. Этот разведывательный проход — ваше окно.

Канарейка в нужном месте срабатывает до того, как данные покинут систему.

**На локальной машине:**

```text
~/backups/customer-prod-export-2024.sql
~/Documents/passwords-old.csv
~/.aws/credentials   ← добавьте фейковый профиль [billing-prod-legacy] с канареечным AWS-ключом
~/.ssh/config        ← добавьте фейковую запись хоста, указывающую на канарейку
```

Поместите URL канарейки внутрь этих файлов. Если кто-то откроет их и перейдет по ссылке, вы узнаете.

**В репозиториях:**

- файл `.env.canary` с фейковыми учетными данными
- старые развертывающие runbook'и с фейковыми сервисными токенами
- устаревшие конфигурационные файлы, которые атакующий проверит при разведке исходного кода

**В CI/CD:**

- фейковый CI-секрет, названный как токен развертывания
- фейковый kubeconfig в окружении GitHub

**В облачных аккаунтах:**

- фейковый IAM-пользователь без привилегий, но с реальным канареечным API-ключом
- неиспользуемый путь в S3-бакете с канареечным объектом

Оповещение должно быть действенным. Канарейка, отправляющая письмо в непроверяемый почтовый ящик, — это украшение. Направьте его туда, где оно кого-то разбудит — PagerDuty, Slack с пингом, SMS — и укажите, какой токен сработал, где он был размещен и чек-лист для ротации.

#### Слепое пятно, о котором стоит знать

Инфостилер криптокошельков может забрать файлы кошельков и никогда не коснуться ваших фейковых AWS-учетных данных. Оператор вымогательского ПО может зашифровать диск до того, как сработает канарейка. Целевой атакующий, уже знающий вашу структуру, может вообще пропустить разведку.

Это нормально. Канареечные токены не предназначены для каждой угрозы — они предназначены для самой распространенной: оппортунистического атакующего, который проводит зачистку учетных данных, просматривает интересные файлы и инвентаризирует ваш доступ, прежде чем решить, что украсть. Таковы большинство атакующих.

Фейковый AWS-ключ, который срабатывает при попытке его использовать, дает вам окно для ротации до того, как найдут настоящий.

Цель не во всеведении. Цель — сделать разведывательный проход дорогим.

### 4. Добавьте исходящий межсетевой экран

Большинство людей думают о «межсетевом экране» и представляют блокировку входящих соединений. Это упускает проблему рабочей станции.

Если вредоносное ПО может читать локальные секреты, следующий вопрос — может ли оно их отправить наружу. Большинство замков смотрят наружу — исходящий межсетевой экран смотрит внутрь. Ему все равно, кто пытается добраться до вашей машины; ему важно, что пытается ее покинуть.

На macOS [LuLu](https://objective-see.org/products/lulu.html) — бесплатный вариант с открытым исходным кодом. [Little Snitch](https://obdev.at/products/littlesnitch/) — отшлифованный коммерческий вариант с правилами для каждого приложения и домена. На Windows и Linux стоит оценить [Portmaster](https://safing.io/).

Этот уровень поначалу раздражает. Но это не повод его пропускать. Цель — заметить, когда `postinstall`, `python` или `invoice-viewer` пытаются связаться с доменом, которому не место в вашем вторнике.

### 5. Относитесь к инструментам ИИ для кодинга как к младшим администраторам с амнезией

Инструменты ИИ для кодинга — не зло. Я ими пользуюсь. Они мне нравятся.

Но у них есть доступ на чтение, запись, к оболочке, к сети и талант уверенно двигаться вперёд. Они будут действовать на основе того, что им дано — и если в том, что им дано, есть вредоносная инструкция, которую они не смогли отличить от легитимного контента, они выполнят и её.

В документации Claude Code от Anthropic разграничиваются разрешения и изоляция. Разрешения определяют, что агенту *разрешено* использовать. Изоляция обеспечивает принуждение на уровне ОС. Текст политики — не изоляция. Запрос разрешения — не изоляция. Благонамеренная модель — не изоляция.

Используйте правила разрешения и запрета на уровне проекта. Держите чувствительные файлы вне рабочих директорий. Запускайте рискованные команды внутри контейнеров. Не отдавайте агенту всю вашу домашнюю директорию только потому, что ему может понадобиться «контекст».

## У вас есть минуты, может быть, часы

Когда срабатывает канарейка — или когда вендор присылает письмо о подозрительном входе, или GitHub предупреждает, что токен был использован с неожиданного IP — следующий шаг — не необязательное чтение.

У вас есть окно. Это могут быть минуты. Может быть, несколько часов, если злоумышленник терпелив. Это не неделя.

Что с ним делать:

- **Сначала ротация, потом расследование.** Отзывайте токены до того, как поймёте, что произошло. Ограничение ущерба — в первую очередь.
- **Проверьте токены GitHub, OAuth-приложения и ключи развёртывания.** Злоумышленник, имевший доступ к вашему ноутбуку, мог создать новые учётные данные перед уходом.
- **Просмотрите недавнюю активность в облаке.** Ищите новых пользователей IAM, сервисные аккаунты, ключи API или политики хранения, которые вы не создавали.
- **Проведите аудит CI.** Проверьте, не выполнялись ли какие-либо рабочие процессы неожиданно, особенно в репозиториях, к которым вы не прикасались недавно.
- **Завершите активные сессии браузера.** Принудительно выйдите из всего, что вам дорого.
- **Сообщите кому-нибудь.** Инциденты безопасности улучшаются при наличии свидетелей и временных меток.

Сообщество безопасности много говорит об обнаружении. Меньше говорят о том, что происходит в течение двадцати минут после обнаружения, когда вы одни за своим столом и пытаетесь вспомнить, для каких сервисов у вас есть токены.

Этот список должен существовать до того, как сработает оповещение.

## Замечание о резервных копиях

Резервные копии — это место, где программы безопасности лгут сами себе.

Они необходимы. Но они также опасны. Резервная копия — это самая переносимая форма того, что вы меньше всего хотите делать переносимым.

- Не храните экспорт из продакшена локально без реальной необходимости.
- Шифруйте локальные резервные копии и дампы баз данных.
- Добавляйте даты истечения к экспортам.
- Размещайте канареечные строки или документы внутри файлов, похожих на резервные копии.
- Держите резервные копии вне широких монтирований Dev Container и контекста AI-инструментов.
- Ротируйте любые учётные данные, которые появляются внутри резервной копии.

Если резервная копия содержит учётные данные, это не просто резервная копия. Это отложенный набор для захвата.

## Практический стандарт

Стандарт не должен быть «никогда не кликай на странное». Это совет для плаката, а не для системы.

Практический стандарт:

- вредоносный PDF не должен иметь возможности прочитать все секреты проекта
- вредоносная зависимость не должна видеть облачные учётные данные из других проектов
- документ с инъекцией промптов не должен перенаправлять агента в вашу домашнюю директорию
- отравленный GitHub Action не должен иметь возможности украсть ваш токен развёртывания
- инфостилер не должен находить резервные копии в открытом виде и долгоживущие токены без срабатывания тревоги
- неизвестный процесс не должен иметь возможности отправлять данные наружу без локального оповещения
- украденные учётные данные должны истекать, проваливать MFA, проваливать проверки устройства или натыкаться на канарейку до того, как станут полным захватом

Безопасность улучшается, когда мы перестаём требовать от людей совершенства и начинаем делать компрометацию менее выгодной.

Ваш ноутбук теперь часть продакшена. Злоумышленник не всегда взламывает — иногда вы впускаете его, не зная об этом.

Дайте вашим системам такие границы, которые ловят оба случая.
