## より良いフレーム：Read, Use, Exfiltrate

すべてのワークステーション防御は次の三つの質問に答えるべきだ。

1. このプロセスは **何を読み取れるか**？
2. どの認証情報を **使用できるか**？
3. データを **どこへ送信できるか**？

ほとんどのワークステーション向けセキュリティ助言は最初の質問で止まっている。ソフトウェアを更新し続ける。疑わしい添付ファイルを開かない。アンチウイルスを使う。良いことだ、もちろん。

しかし、悪意あるプロセスが実行された場合、2 と 3 の質問が、ただの悪い午後になるか、全社的なインシデントになるかを決定します。

`~/.aws/credentials` を読み取れるか？ GitHub トークンを使用できるか？ パスワードマネージャ拡張を開けるか？ 誰にも気付かれずに 3 GB をランダムなホストへアップロードできるか？

このフレームは脅威をフォグマシンから、歯のあるチェックリストへと変換します。

## 私が最初にやること

会社を「悲しい空港」にしないで開発者ワークステーションのプログラムを堅牢化するなら、まずここから手を付けます。

### 1. リスクの高い作業を Dev Containers に移す

依存関係、ビルドツール、パッケージインストール、あるいは AI 補助シェルコマンドが必要なプロジェクト作業には、[Development Containers](https://github.com/devcontainers/spec) を使用します。Dev Container はローカルの Docker コンテナで、プロジェクトの分離された作業領域として機能します — 明示的にマウントしない限り、マシンの他の部分を見ることはできません。

利点: `npm install`、`pip install`、`go generate`、`cargo build`、そしてモデルが実行したがるものはすべて、ホームディレクトリ全体を自動的に所有しない作業領域で行われます。

リポジトリをマウントする。プロジェクトに必要なシークレットだけをマウントする。利便性のために `~/.ssh`、`~/.aws`、`~/Downloads`、ホームフォルダ全体をマウントしないようにする。

```jsonc
// .devcontainer/devcontainer.json — narrow mounts only
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "workspaceFolder": "/workspaces/app",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ],
  "containerEnv": {
    "NODE_ENV": "development"
  },
  "postCreateCommand": "bun install"
}
```

スコープ化された認証情報を注入する。短命トークンを優先する。可能な限り読み取り専用アクセスを選ぶ。プロンプト注入された指示はエージェントが到達できる範囲だけに届くので、そこを退屈に保ちます。

### 2. `.env` を崇拝する代わりにローカルシークレットを暗号化する

平文の `.env` ファイルは「ファイルが便利」だから便利です。攻撃者もファイルが好きです。

[VarLock](https://varlock.dev/guides/secrets/) は感度を構造化メタデータとして扱います — 敏感な値にマークを付けると、ローカルで暗号化し、コンソール出力からはマスクし、秘密であるはずの値の平文出現をスキャンします。

```dotenv
# .env.schema
# @sensitive
STRIPE_SECRET_KEY=

# @sensitive
DATABASE_URL=
```

シークレットは自分がシークレットであることを認識すべきです。すでに侵害されたプロセスにロードされたシークレットを保護するわけではありませんが、価値ある平文ファイルが誰かのインベントリになるリスクを減らします。

### 3. 泥棒が目を向ける場所すべてにカナリアトークンを配置する

これは多くのチームが飛ばしがちで、かつ最も即効性のある層です。

[Canarytokens](https://help.canary.tools/hc/en-gb/articles/10905485310109-Canarytoken-Overview-and-Use-Cases) はデジタルのトリップワイヤです。攻撃者が目を向けそうな場所に、偽だけど説得力のあるシークレット、API キー、URL を置きます。触れられた瞬間にアラートが発生し、しばしば数秒以内に通知されます。偽札の束に染料入りパックを忍ばせるイメージです：開封と同時に分かります。

テイクオーバーパターンのステップ二、**インベントリの取得** を思い出してください。攻撃者は盗む前に閲覧します。その偵察フェーズがあなたのウィンドウです。

適切な場所に置いたカナリアは、データが外部へ出る前に作動します。

**ローカルマシン上で:**

```text
~/backups/customer-prod-export-2024.sql
~/Documents/passwords-old.csv
~/.aws/credentials   ← 偽の [billing-prod-legacy] プロファイルにカナリア AWS キーを追加
~/.ssh/config        ← カナリアへ向かう偽のホストエントリを追加
```

これらのファイルにカナリア URL を埋め込みます。何かがそれらを開いてリンクをたどった瞬間に検知できます。

**リポジトリ内で:**

- 偽の認証情報を入れた `.env.canary` ファイル  
- 偽のサービストークンを含む古いデプロイ手順書  
- ソース偵察時に攻撃者が目を通すであろう廃止済み設定ファイル  

**CI/CD で:**

- デプロイトークンのような名前の偽 CI シークレット  
- GitHub 環境に置く偽の kubeconfig  

**クラウドアカウントで:**

- 権限を持たない偽 IAM ユーザーに実際のカナリア API キーを付与  
- 使われていない S3 バケットパスにカナリアオブジェクトを配置  

アラートは実行可能である必要があります。未監視の受信トレイにメールが届くだけのカナリアは装飾に過ぎません。誰かがすぐに対応できる先へ送ります — PagerDuty、Slack の ping、SMS など — そして、どのトークンが作動したか、どこに設置したか、ローテーションチェックリストを添えて通知します。

#### 知っておくべき盲点

暗号ウォレットの情報窃取ツールはウォレットファイルだけを取得し、偽の AWS 認証情報には触れないかもしれません。ランサムウェアの実行者は、カナリアが作動する前にディスク全体を暗号化することがあります。レイアウトをすでに把握している標的型攻撃者は、偵察工程を完全にスキップすることもあり得ます。

それで構いません。カナリアトークンはすべての脅威に対応するためのものではなく、最も一般的なシナリオ、すなわち「資格情報をざっくりと走査し、興味深いファイルを探し出し、何を盗むか決める前にアクセスをインベントリ化する」機会主義的な攻撃者を想定しています。実際に多くの攻撃者はこのパターンです。

誰かが偽の AWS キーを使用しようとした瞬間に作動させれば、実際のキーが見つかる前にローテーションできるウィンドウが得られます。

目的は全知全能になることではありません。目的は偵察フェーズをコスト高にすることです。

### 4. アウトバウンドファイアウォールを追加する

多くの人は「ファイアウォール」と聞くとインバウンド接続を遮断するイメージを持ちますが、これはワークステーション問題を見落としています。

マルウェアがローカルのシークレットを読み取れるなら、次に問うべきはそれらを外部へ送信できるかどうかです。ほとんどのロックは外向きに設置されますが、アウトバウンドファイアウォールは内向きに機能します。誰がマシンにアクセスしようとしているかは問わず、何がマシンから外へ出ようとしているかを監視します。

macOS では、[LuLu](https://objective-see.org/products/lulu.html) が無料でオープンソースの選択肢です。[Little Snitch](https://obdev.at/products/littlesnitch/) は、アプリごと・ドメインごとのルールを設定できる洗練された商用オプションです。Windows と Linux では、[Portmaster](https://safing.io/) を検討する価値があります。

この層は最初は面倒です。だからといって省いてはいけません。目的は、`postinstall`、`python`、`invoice‑viewer` などが、あなたの火曜日に関係のないドメインへ通信しようとしたときに気付くことです。

### 5. AI コーディングツールを記憶喪失の新人管理者と同等に扱う

AI コーディングツールが悪いわけではありません。私も使っていますし、好きです。

しかし、読み取り権限、書き込み権限、シェルアクセス、ネットワークアクセスをすべて持ち、かつ自信満々に動く才能があります。与えられたものに対しては必ず行動します――そして、与えられたものに悪意ある指示が混入していてそれが正当なコンテンツと区別できなければ、同様に実行します。

Anthropic の Claude Code ドキュメントは、権限とサンドボックス化を区別しています。権限はエージェントが *使用できる* ものを決め、サンドボックスは OS レベルでの強制を提供します。ポリシーテキストはサンドボックスではありません。権限プロンプトもサンドボックスではありません。善意のモデルもサンドボックスではありません。

プロジェクト単位の許可・拒否ルールを使用しましょう。機密ファイルは作業ディレクトリから除外します。リスクの高いコマンドはコンテナ内で実行します。エージェントにホームディレクトリ全体を渡さないでください。「コンテキスト」が必要になるかもしれませんが、最小限に抑えるべきです。

## 数分、場合によっては数時間がある

カナリアが鳴ったとき――あるいはベンダーから疑わしいログインのメールが来たとき、GitHub が予期しない IP からトークンが使用されたと警告したとき――次のステップは任意の読書ではありません。

ウィンドウがあります。数分かもしれません。攻撃者が忍耐強く待っている場合は数時間になることもあります。1 週間はありません。

取るべき行動:

- **まずローテーションし、後で調査する。** 何が起きたかを把握する前にトークンを失効させます。被害の限定が最優先です。
- **GitHub のトークン、OAuth アプリ、デプロイキーを確認する。** ラップトップを奪われた攻撃者は、離脱前に新しい認証情報を作成している可能性があります。
- **最近のクラウドアクティビティをレビューする。** 作成していない IAM ユーザー、サービスアカウント、API キー、ストレージポリシーがないか探します。
- **CI を監査する。** 特に最近触っていないリポジトリで、予期しないワークフローが走っていないか確認します。
- **アクティブなブラウザセッションを終了する。** 重要なものはすべて強制ログアウトします。
- **誰かに報告する。** インシデントは目撃者とタイムスタンプがあるほど対処しやすくなります。

セキュリティコミュニティは検知については多く語りますが、検知後の 20 分間、デスクに一人で座り「どのサービスにトークンを持っているか」を思い出そうとしているときに何が起きるかについてはあまり語りません。

このリストはアラートが鳴る前に用意しておくべきです。

## 私がすべてのチーム Wiki に欲しい表

| Layer | Bad default | Better default |
| --- | --- | --- |
| Filesystem | Projects, secrets, downloads, backups, and tools all share one user context. | Run project work in Dev Containers with narrow mounts. |
| Secrets | Plaintext `.env` files and long-lived tokens. | Encrypted local secrets, scoped tokens, short lifetimes, hardware-backed auth. |
| Detection | Hope security software catches the exfiltration in time. | Canary tokens in high-value local, CI, cloud, and documentation locations. |
| Network | Any process can reach out unless blocked by reputation. | Outbound application firewall with per-app rules. |
| AI agents | Broad read/write/shell permissions in the main workstation context. | Project-scoped permissions, prompt-injection awareness, sandboxed commands. |
| Backups | Local dumps and exports treated like dead files. | Encrypt, expire, isolate, and monitor access to backup artifacts. |
| CI/CD | Mutable action tags, broad secret access, unsafe input interpolation. | Pinned commit SHAs, scoped environments, short-lived credential exchange, no interpolation of untrusted input. |

## バックアップに関する注意

バックアップは、セキュリティプログラムが自分自身に嘘をつく場所です。

それらは必要不可欠です。同時に危険でもあります。バックアップは、最も持ち出したくないものを最も持ち運びやすくした形です。

- 本当に必要な場合以外は、プロダクションのエクスポートをローカルに保存しないこと。
- ローカルのバックアップやデータベースダンプは暗号化すること。
- エクスポートに有効期限を設定すること。
- バックアップらしきファイルにカナリア行やドキュメントを埋め込むこと。
- バックアップを広範な Dev Container のマウントや AI ツールのコンテキストから除外すること。
- バックアップ内に現れた認証情報はすべてローテーションすること。

バックアップに認証情報が含まれている場合、それは単なるバックアップではなく、遅延型の乗っ取りキットです。

## 実用的な標準

標準は「変なものは決してクリックしない」ではありません。これはポスター向けの助言であり、システム向けではありません。

実用的な標準:

- 悪質な PDF がプロジェクトのすべてのシークレットを読み取れないこと
- 悪意ある依存関係が他プロジェクトのクラウド認証情報を参照できないこと
- プロンプトインジェクションされた文書がエージェントをホームディレクトリへリダイレクトできないこと
- 毒された GitHub Action がデプロイトークンを盗めないこと
- 情報窃盗ツールが平文のバックアップや長期間有効なトークンを検知アラームなしで見つけられないこと
- 不明なプロセスがローカルアラートなしにデータを外部へ送信できないこと
- 盗まれた認証情報が期限切れになる、MFA に失敗する、デバイスチェックに引っ掛かる、またはカナリアに捕まるまでに完全な乗っ取りに至らないこと

人間に完璧を求めるのをやめ、妥協を利益しにくくすることでセキュリティは向上します。

あなたのラップトップは今やプロダクションの一部です。攻撃者が必ずしも侵入に成功するわけではなく、知らずに入口を開けてしまうこともあります。

両方を捕らえるような境界をシステムに設けましょう。
