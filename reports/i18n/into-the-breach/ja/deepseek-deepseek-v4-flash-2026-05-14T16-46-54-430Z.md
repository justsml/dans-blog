### GitHub CI/CD

GitHub Actionsは強力で信頼されているが、しばしば誤設定される。その結果は、ラップトップの侵害と同じ場所にたどり着くことが多い。すなわち、認証情報、ソースコード、デプロイメントアクセスである。

**サードパーティアクションの汚染。** ワークフローが `uses: some-org/some-action@v2` をプルしているとする。`@v2` のようなバージョンタグは移動可能なラベルであり、上流リポジトリが侵害されたり、そのタグが悪意のあるコミットにリダイレクトされたりすると、ワークフローはリポジトリのシークレットを使って攻撃者のコードを実行する。対策：アクションを完全なコミットSHAに固定する。

**プルリクエストトリガーの悪用。** `pull_request_target` は、外部コントリビューターからのPRであっても、ベースリポジトリのシークレットにアクセスできるワークフローを実行するトリガーである。不注意なワークフローは、信頼されていないコードにそれらのシークレットを露出させる可能性がある。これは文書化されたGitHubの自爆機能である。

**信頼できない入力によるワークフローインジェクション。** `${{ github.event.pull_request.title }}` を `run:` ステップに直接展開すると、攻撃者がシェルコマンドを注入するPRタイトルを作成できる。ユーザー制御の値は常に中間環境変数を介して渡すこと。

**フォークからのシークレット流出。** フォークされたPRはデフォルトではリポジトリシークレットを受け取らないが、`pull_request_target` や環境保護ルールの誤設定により、それが変わる可能性がある。

実用的な最低限の対策：

- サードパーティアクションを完全なコミットSHAに固定する。
- `github.event` フィールドを `run:` ステップに直接展開しない。
- 本番シークレットは、保護ルールと必須レビュアーがある環境に保管する。
- 機密シークレットにアクセスできるワークフローをトリガーできる人を監査する。
- CIに長期間有効なシークレットを保存する代わりに、クラウドアクセスには短命な認証情報交換（OIDC）を使用する。

### 1. リスクの高い作業をDev Containerに移す

依存関係、ビルドツール、パッケージインストール、AI支援シェルコマンドを必要とするプロジェクト作業には、[Development Containers](https://github.com/devcontainers/spec) を使用する。Dev Containerは、プロジェクトの隔離されたワークスペースとして機能するローカルDockerコンテナであり、明示的にマウントしない限り、マシンの他の部分を見ることはできない。

利点: `npm install`、`pip install`、`go generate`、`cargo build`、そしてモデルが実行したいものはすべて、自動的にホームディレクトリ全体を所有しないワークスペースで実行される。

リポジトリをマウントする。そのプロジェクトに必要なシークレットのみをマウントする。便利だからといって `~/.ssh`、`~/.aws`、`~/Downloads`、ホームフォルダ全体をマウントしないようにする。

```jsonc
// .devcontainer/devcontainer.json — narrow mounts only
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "workspaceFolder": "/workspaces/app",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ],
  "containerEnv": {
    "NODE_ENV": "development"
  },
  "postCreateCommand": "bun install"
}
```

スコープ付きの認証情報を注入する。短命なトークンを優先する。可能な限り読み取り専用アクセスを優先する。プロンプトインジェクションされた命令は、エージェントが到達できるものにしか到達できない — それを退屈なものにする。

### 2. `.env` を崇拝する代わりにローカルシークレットを暗号化する

平文の `.env` ファイルは便利だ。なぜならファイルは便利だからだ。攻撃者もファイルを好む。

[VarLock](https://varlock.dev/guides/secrets/) は機密性を構造化メタデータとして扱う — どの値が機密かをマークすると、ローカルで暗号化し、コンソール出力から編集し、秘密にすべき値の平文出現をスキャンする。

```dotenv
# .env.schema
# @sensitive
STRIPE_SECRET_KEY=

# @sensitive
DATABASE_URL=
```

シークレットは自分がシークレットであることを認識すべきだ。これは既に侵害されたプロセスにロードされたシークレットを保護するわけではないが、誰かの在庫になるのを待っている貴重な平文ファイルの数を減らす。

### 3. 泥棒が見そうな場所にカナリアトークンを仕掛ける

これはほとんどのチームがスキップする層であり、おそらく最も即座に有用である。

[Canarytokens](https://help.canary.tools/hc/en-gb/articles/10905485310109-Canarytoken-Overview-and-Use-Cases) はデジタルトリップワイヤーだ。偽物だが説得力のあるシークレット、APIキー、またはURLを攻撃者が見そうな場所に配置する。もし触れられると、アラートが届く — 多くの場合数秒以内だ。偽の札束の中に染料パックを仕込むようなものだ: 誰かが開けた瞬間に、あなたは知る。

乗っ取りパターンのステップ2を思い出してほしい: **まずインベントリ**。攻撃者は盗む前に閲覧する。その偵察パスがあなたのチャンスだ。

適切な場所に仕掛けたカナリアは、データが流出する前に発報する。

**ローカルマシン上:**

```text
~/backups/customer-prod-export-2024.sql
~/Documents/passwords-old.csv
~/.aws/credentials   ← add a fake [billing-prod-legacy] profile with a canary AWS key
~/.ssh/config        ← add a fake host entry pointing to a canary
```

それらのファイルの中にカナリアURLを仕込む。何かがそのファイルを開いてリンクを辿ったら、すぐに分かる。

**リポジトリ内:**

- 偽の認証情報を含む `.env.canary` ファイル
- 偽のサービストークンを含む古いデプロイメントランブック
- 攻撃者がソース偵察中に調べるであろう廃止された設定ファイル

**CI/CD内:**

- デプロイトークンのような名前の偽のCIシークレット
- GitHub環境内の偽のkubeconfig

**クラウドアカウント内:**

- 権限はないが本物のカナリアAPIキーを持つ偽のIAMユーザー
- カナリアオブジェクトが置かれた未使用のS3バケットパス

アラートはアクション可能であるべきだ。誰も見ていない受信箱にメールを送るだけのカナリアは飾りに過ぎない。誰かを起こせる場所 — PagerDuty、Slackのメンション、SMS — にルーティングし、どのトークンが発報したか、どこに仕掛けられたか、そしてローテーションのチェックリストを含める。

#### 知っておくべき盲点

暗号資産ウォレットの情報窃取型マルウェアは、ウォレットファイルを奪い、あなたの偽のAWS認証情報には一切触れないかもしれない。ランサムウェアの運営者は、カナリアが発報する前にディスクを暗号化するかもしれない。あなたの構成を既に知っている標的型攻撃者は、偵察を完全にスキップするかもしれない。

それで構わない。カナリアトークンはあらゆる脅威のために設計されているわけではない — 最も一般的な脅威のために設計されている: 認証情報をスイープし、興味深そうなファイルを閲覧し、何を盗むか決める前にアクセス権を棚卸しする日和見的な攻撃者だ。それがほとんどの攻撃者である。

誰かが使おうとした時に発報する偽のAWSキーは、攻撃者が本物を見つける前にローテーションする猶予を与える。

目標は全知になることではない。目標は偵察のパスを高コストにすることだ。

### 4. アウトバウンドファイアウォールを追加する

ほとんどの人は「ファイアウォール」と聞くと、受信接続をブロックするイメージを持つ。それはワークステーションの問題を見逃している。

マルウェアがローカルのシークレットを読み取れるなら、次の問題はそれを外部に送信できるかどうかだ。ほとんどのロックは外側を向いている — アウトバウンドファイアウォールは内側を向く。誰があなたのマシンに到達しようとしているかは気にしない。何がマシンから出ていこうとしているかを気にする。

macOSでは、[LuLu](https://objective-see.org/products/lulu.html)が無料のオープンソース選択肢だ。[Little Snitch](https://obdev.at/products/littlesnitch/)はアプリ単位・ドメイン単位のルールを備えた洗練された商用製品で、WindowsとLinuxでは[Portmaster](https://safing.io/)が検討に値する。

このレイヤーは最初は煩わしい。それは飛ばす理由にはならない。目的は、`postinstall`や`python`、`invoice-viewer`が、火曜日のあなたの作業とは無関係なドメインと通信しようとしたときに気づくことだ。

### 5. AIコーディングツールを記憶喪失のジュニア管理者のように扱う

AIコーディングツールは悪くない。私も使っているし、好きだ。

しかし、それらは読み取りアクセス、書き込みアクセス、シェルアクセス、ネットワークアクセスを持ち、自信過剰な勢いで動く才能がある。与えられたものに従って行動する——そして、もし与えられたものに、正規のコンテンツと区別できない悪意ある指示が含まれていれば、それにも従ってしまう。

AnthropicのClaude Codeのドキュメントでは、パーミッションとサンドボックス化を区別している。パーミッションはエージェントが*使用を許可される*ものを決める。サンドボックス化はOSレベルの強制を提供する。ポリシーテキストはサンドボックスではない。許可プロンプトはサンドボックスではない。善意のモデルはサンドボックスではない。

プロジェクトレベルの許可・拒否ルールを使う。機密ファイルを作業ディレクトリに置かない。リスクのあるコマンドはコンテナ内で実行する。「コンテキスト」が必要だからといって、エージェントにホームディレクトリ全体を渡さない。

## 猶予は数分、せいぜい数時間

カナリーが発報したとき——あるいはベンダーから不審なログインについてメールが届いたとき、GitHubが予期しないIPからトークンが使われたと通知したとき——次のステップは任意の読み物ではない。

あなたには猶予期間がある。数分かもしれない。攻撃者が忍耐強ければ数時間かもしれない。一週間ではない。

何をすべきか：

- **ローテーションを先に、調査は後で。** 何が起きたか理解する前にトークンを失効させる。被害の局限が最優先だ。
- **GitHubトークン、OAuthアプリ、デプロイキーを確認する。** あなたのラップトップにアクセスした攻撃者は、去る前に新しい認証情報を作成しているかもしれない。
- **最近のクラウドアクティビティを確認する。** 自分が作成していない新しいIAMユーザー、サービスアカウント、APIキー、ストレージポリシーがないか探す。
- **CIを監査する。** 特に最近触っていないリポジトリで、予期しないワークフローが実行されていないか確認する。
- **アクティブなブラウザセッションを強制終了する。** 気にかかるサービスすべてでログアウトを強制する。
- **誰かに伝える。** セキュリティインシデントは、目撃者とタイムスタンプがあれば改善される。

セキュリティコミュニティは検出について多く語る。しかし、検出後の20分間に、あなたが一人で机に向かい、どのサービスにトークンを持っているかを思い出そうとしている間に何が起きるかについては、あまり語られない。

そのリストは、アラートが発報する前に存在しているべきだ。

## バックアップについての注意

バックアップは、セキュリティプログラムが自分自身に嘘をつく場所だ。

バックアップは必要だ。同時に危険でもある。バックアップとは、最も持ち運びたくないものを最も持ち運びやすい形にしたものだ。

- 本当に必要な場合を除き、本番環境のエクスポートをローカルに保存しない。
- ローカルのバックアップやデータベースダンプは暗号化する。
- エクスポートに有効期限を設定する。
- バックアップのようなファイルの中にカナリア行やカナリア文書を仕込む。
- バックアップを広範なDev ContainerマウントやAIツールのコンテキストから遠ざける。
- バックアップ内に現れた認証情報はすべてローテーションする。

バックアップに認証情報が含まれているなら、それは単なるバックアップではない。それは遅延発動型の乗っ取りキットだ。

## 実用的な基準

基準は「怪しいものを絶対にクリックしない」であってはならない。それはポスター向けのアドバイスであって、システム向けではない。

実用的な基準とは：

- 悪意のあるPDFがプロジェクトの全シークレットを読めないこと
- 悪意のある依存関係が他のプロジェクトのクラウド認証情報を参照できないこと
- プロンプトインジェクションされたドキュメントがエージェントをホームディレクトリに誘導できないこと
- 毒されたGitHub Actionがデプロイトークンを盗めないこと
- 情報窃取型マルウェアが平文のバックアップや長期トークンをアラームなしに見つけられないこと
- 未知のプロセスがローカルアラートなしにデータを外部送信できないこと
- 盗まれた認証情報が完全な乗っ取りに至る前に、期限切れ、MFA失敗、デバイスチェック失敗、カナリア発動のいずれかを経ること

人間に完璧を求めるのをやめ、侵害の収益性を下げることに注力すれば、セキュリティは向上する。

あなたのラップトップは今や本番環境の一部だ。攻撃者は必ずしも侵入するとは限らない——時にはあなたが知らないうちに招き入れてしまう。

両方に対処できる境界を、あなたのシステムに与えよう。
