## あなたは認証情報の倉庫である

あなたのラップトップはラップトップではありません。キーボード付きの認証情報倉庫です——ブラウザセッション、SSH鍵、`.env`ファイル、GitHubトークン、クラウドCLI、シェルアクセスを持つAIコーディングツール、存在を忘れていたデータベースエクスポートなどが詰め込まれています。

旧来のモデルは：プロダクションは危険、ローカルは便利。そのモデルは終わりました。

<p class="inset">
問題は、すべての悪意あるクリックを避けることができるかではありません。問題は、一つの悪意あるクリックがすべてを読み取り、すべてを使用し、あなたが気付く前に去ることができるかです。
</p>

開発者は、十分に普通に見える何かに遭遇します：契約者からのPDF、端末に何かを貼り付けるように要求する偽のCAPTCHA、`postinstall`スクリプトを持つパッケージ、タスクの必要以上にファイルシステムに深くアクセスしたAIコーディングセッション。いくつかの経路はマルウェアをインストールします。いくつかは認証情報を盗みます。いくつかはローカル exploit を必要としません——ユーザー自身が攻撃者のコマンドを実行します。

これが現代の攻撃対象領域です。時には、あなた自身が侵害です。

## サプライチェーン問題は不可能なほど巨大

ここが面白い部分です。完全に安全であるために必要なのは、すべての依存関係——そのメンテナー、履歴、推移的依存関係——に対して、すべてのパッケージレジストリを横断した深い、マルチプラットフォームのセキュリティ評価を行うことです。そして、依存関係ツリーが変更されるか更新されるたびに評価を繰り返します。なぜなら、それがサプライチェーン攻撃が機能する仕組みそのものです：信頼の連鎖を悪用します。

簡単ですね。

そして、攻撃者は一度成功すればよいのです。あなたは毎回完璧な防御を維持しなければなりません。

Lumma Stealer——パスワード、ブラウザクッキー、APIキー、クラウド認証情報を静かに収集する広く使用されている情報窃取ツール——は、偽のCAPTCHA、毒された検索広告、トロイ化されたアプリを通じて被害者に到達しました。MandiantのSnowflake調査は、企業侵害の連鎖を情報窃取ツールによって盗まれた認証情報に遡り、一部は2020年までさかのぼります。攻撃で使用されたアカウントの少なくとも79.7%は以前に既知の露出がありました。鍵は一度も変更されていませんでした。

攻撃者は倉庫を破壊しなかった。机の引き出しで古い鍵を見つけた。

開発者にとって、その机の引き出しは以下のようなものです：

| ローカルアーティファクト | 攻撃者が注目する理由 |
| --- | --- |
| ブラウザクッキー | ログインをバイパスし、場合によってはMFAをスキップできる。 |
| `.env` ファイル | APIキー、データベースURL、JWTシークレット。 |
| クラウドCLI設定 | ラップトップの侵害を完全なインフラアクセスに変える。 |
| SSHキー | まだどこにでもあり、強力で、マシン間でコピーされる。 |
| パッケージマネージャトークン | npmやPyPIの公開トークンはサプライチェーンへのアクセスである。 |
| データベースダンプ | 本番環境より保護が弱く、往往にしてより完全なデータを含む。 |
| AIコーディングコンテキスト | アシスタントが「コンテキスト用」に敏感なファイルを渡されている可能性がある。 |

そして、バックアップがある——誰かが `~/Downloads` に落として忘れた本番環境のエクスポート。バックアップは無害だから安全ではない。それはただ警報システムがない本番環境だ。

## 「注意してください」という非解決策

「注意してください」は弱いアドバイスだ。それは人間を境界として要求する。

人間は境界ではない。人間はトラフィックだ。

境界は退屈なものだ：ファイルシステムの隔離、静止時に暗号化されたシークレット、短期間の認証情報、ハードウェアベースの認証、そして偽のシークレットが触られた瞬間に発動するアラート。

もし悪意のあるプロセスが実行されたら、あなたが悪い午後を経験するか会社全体のインシデントになるかを決定する質問は：
1. このプロセスは何を**読み取る**ことができるか？
2. どの認証情報を**使用**できるか？
3. どこにデータを**送信**できるか？

## 現在最も効果的な対策

### 開発コンテナ —— 既定で

[開発コンテナ](https://github.com/devcontainers/spec)は、ほとんどのチームが実施していない単一で最も効果的な変更だ。開発コンテナはプロジェクト作業を隔離されたDockerコンテナ内で実行する。`npm install`、`pip install`、`postinstall` スクリプト、AIシェルコマンド、VS Code拡張機能——すべてがあなたのマシンの他の部分を見ることができない「ワークスペース」またはコンテナ内で発生する。

<p class="inset">Claude Codeに任意のプロジェクトで開発コンテナを設定するように依頼する。</p>

リポジトリをマウントする。そのプロジェクトに必要なシークレットのみを含める。便利さのために `~/.ssh`、`~/.aws`、またはホームディレクトリをマウントしない。プロンプトインジェクションされた指示は、エージェントが到達できるもののみに到達できる——それを退屈なものにする。

```jsonc
// .devcontainer/devcontainer.json
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ]
}
```

### カナリアトークン —— 積極的に展開

[カナリアトークン](https://canarytokens.org)は無料のデジタルトリップワイヤだ。攻撃者が探す場所に偽物だが説得力のあるシークレットを仕掛ける。それが触られた瞬間、あなたはアラートを受け取る——往往にして数秒以内だ。偽札の束に染料パックを残すと考えてみよう。

攻撃者は盗む前にインベントリを作成する。その偵察パスはあなたの窓だ。

最も誘惑的なファイルにカナリアを仕掛ける：

```text
~/.aws/credentials          ← [billing-prod-legacy]という偽のプロファイルにカナリアキーを追加
~/backups/customer-export-2024.sql   ← 内部にカナリアURLを仕込む
~/.env.canary               ← すべてのリポジトリに偽の認証情報を配置
```

カナリアトークンは[canarytokens.org](https://canarytokens.org)で無料で利用でき、自己ホストも可能であり、有料SaaSとして[Thinkst Canary](https://canary.tools)からも提供されている。盗み見される可能性のある場所すべてに展開しない理由はない。

### パッケージセキュリティツール

[Socket.dev](https://socket.dev)、[Snyk](https://snyk.io)、[Wiz](https://wiz.io)のようなツールは、進行中のサプライチェーン攻撃を最初に発見しブロックすることが多い。これらは自分では監視できないパッケージレジストリを監視する。専任のセキュリティプログラムを維持できないチームにとって、これらは高い効果を持つ早期警告システムだ。

### PNPMの最小リリース年齢設定

PNPMを使用している場合、最小リリース年齢を設定する。新しく公開されたパッケージはサプライチェーン攻撃のリスクが最も高い期間である——24時間以内に公開されたパッケージは、基本的にコミュニティの審査を受けていない。`minimumReleaseAge`を分単位で設定する：最低`1440`（1日）、理想的には`2880`（2日）。

```yaml
minimumReleaseAge: 2880
minimumReleaseAgeStrict: true
minimumReleaseAgeIgnoreMissingTime: false
minimumReleaseAgeExclude:
  - 'typescript'
```

この設定は、特に次のインストール前に発見され削除されるような、新規公開パッケージ攻撃の多くをブロックする。`minimumReleaseAgeExclude`は、遅延よりも即時の更新が重要なパッケージ（例えば積極的に追跡するコンパイラやランタイム依存関係）に対してのみ控えめに使用する。

### 最もセキュリティが重要な環境の場合

情報機関、法執行機関、金融取引インフラ、医療記録——これらの環境では厳格なパッケージ評価と承認プロセスを採用することがある。それは安全に聞こえる。しかし代償は深刻だ：依存関係ツリーが徐々に古いソフトウェアで固まってしまう。

時間は中立ではない。古いバージョンは既知のCVEを蓄積する。攻撃者は修正済みバージョンを研究し、未修正のインスタンスを探す。そして「知っている悪魔は良い」という考えは期待した救済ではない——それは攻撃者が最も長く習熟した脆弱性を教えるだけだ。

厳格な許可リストは、維持する人員がいる場合に機能する。ほとんどのチームにはない。他のすべての人にとって、階層化アプローチ——開発コンテナ、カナリアトークン、パッケージセキュリティツール、短期認証情報——は、すべての依存関係を手動で監査できると仮定するよりも現実的な防御を提供する。

## あなたには数分しかない

カナリアが発動したとき——またはGitHubが予期しないIPからトークンが使用されたと警告したとき——あなたには時間がある。数分、あるいは数時間。一週間ではない。

- **まずローテーションし、後で調査する。** 何が起こったかを理解する前にトークンを無効化する。
- **攻撃者の永続性を確認する。** 離脱前に作成された新しいOAuthアプリ、IAMユーザー、デプロイキー、APIトークン。
- **アクティブなブラウザセッションを終了する。** 重要なすべてのものに対して強制ログアウト。
- **誰かに伝える。** セキュリティインシデントは証人とタイムスタンプによって改善される。

セキュリティ業界は検出について多く語る。検出後の20分間、あなたが机に一人で座り、どのサービスにトークンを持っているかを思い出そうとしているときに何が起こるかについては、あまり語らない。

そのリストはアラートが発動する前に存在しているべきだ。

## 持つべき標準

標準は「怪しいものを一切クリックしない」ではない。それはポスターのためのアドバイスであり、システムではない。

悪い依存関係が他のプロジェクトからのクラウド認証情報にアクセスできないようにする。プロンプトインジェクションされたドキュメントがエージェントをあなたのホームディレクトリにリダイレクトできないようにする。情報窃取ツールが平文のバックアップと長期トークンを発見しても、警告をトリガーしないようにする。盗まれた認証情報は、完全な乗っ取りになる前に、期限切れ、MFA失敗、またはカナリアにヒットするべきだ。

セキュリティは、人間に完璧を求めず、侵害の利益を減らすことで向上する。

あなたのラップトップは今、プロダクション環境の一部だ。侵入した攻撃者と、自分で誤って招いた攻撃者を両方捕捉する、退屈な境界線を与えよ。
