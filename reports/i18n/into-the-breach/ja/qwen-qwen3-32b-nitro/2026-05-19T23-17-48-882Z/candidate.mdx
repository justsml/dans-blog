## あなたこそが資格情報倉庫

あなたのノートPCはノートPCではありません。それはキーボード付きの資格情報倉庫です——ブラウザセッション、SSHキー、`.env`ファイル、GitHubトークン、クラウドCLI、シェルアクセス付きのAIコーディングツール、存在を忘れていたデータベースエクスポートが含まれます。

古いモデルは「本番環境は危険で、ローカルは便利」というものでした。そのモデルは終わりました。

<p class="inset">
避けるべきはすべての悪いクリックではなく、1つの悪いクリックがすべてを読み取り、すべてを活用し、気づく前に退出できるかどうかです。
</p>

開発者は、通常に見えるものを遭遇します：コンタクターからのPDF、ターミナルに何かを貼り付けるよう求める偽のCAPTCHA、`postinstall`スクリプトを含むパッケージ、タスクに必要な範囲を超えてファイルシステムにアクセスするAIコーディングセッション。一部のパスはマルウェアをインストールします。一部は資格情報を盗みます。一部はローカルのエクスプロイトでなく、ユーザー自身が攻撃者のコマンドを実行します。

これが現代の攻撃面です。ときにはあなた自身が漏洩経路になります。

## サプライチェーン問題は不可能に広い

面白い点はここです。完全に安全であるためには、依存するすべてのパッケージレジストリで、すべての依存関係（メンテナ、履歴、推移的依存関係）を深く多プラットフォームでセキュリティ評価する必要があります。依存ツリーが変化したり、更新が入るたびにこの評価を繰り返す必要があります。なぜならそれがまさにサプライチェーン攻撃の仕組みだからです——信頼の連鎖を悪用するからです。

簡単ですね。

ああ、そうそう。攻撃者は一度成功すればいい。あなたは常に完璧な防御を維持しなければなりません。

Lumma Stealer——パスワード、ブラウザのクッキー、APIキー、クラウド資格情報を静かに収集する広く使われる情報窃取ツール——は、偽のCAPTCHA、毒された検索広告、トロイの木馬化されたアプリを通じて被害者に到達しました。MandiantのSnowflake調査は、インフォスティーラーによって盗まれた資格情報を起点にした企業の連鎖的な漏洩をたどり、一部は2020年まで遡ることが明らかにされました。攻撃で使用されたアカウントの79.7%は既知の過去の暴露歴がありました。錠は決して変更されませんでした。

攻撃者は倉庫を破壊しませんでした。古い鍵を机の引き出しから見つけたのです。

開発者にとってその引き出しは次のように見えます：

| ローカルアーティファクト | 攻撃者が関心を持つ理由 |
| --- | --- |
| ブラウザのクッキー | ログインをバイパスし、場合によってはMFAをスキップできる。 |
| `.env`ファイル | APIキー、データベースURL、JWTシークレットが含まれる。 |
| クラウドCLI設定 | ラップトップの侵害がインフラストラクチャへのフルアクセスに繋がる。 |
| SSHキー | 依然としていたるところに存在し、依然として強力で、依然としてマシン間でコピーされる。 |
| パッケージマネージャートークン | あなたのnpmまたはPyPI公開トークンはサプライチェーンへのアクセス権である。 |
| データベースダンプ | 本番環境より保護が不十分で、しばしばより完全な情報が含まれる。 |
| AIコーディングコンテキスト | アシスタントが「コンテキストのために」機密ファイルを渡された可能性がある。 |

そしてバックアップもあります——誰かが`~/Downloads`に置き忘れた本番環境のエクスポートです。バックアップが安全でないのは不活性だからではありません。単にアラームシステムのない本番環境に過ぎません。

## 「気をつけて」では解決しない

「気をつけて」は弱いアドバイスです。人間を境界として頼りにしています。

人間は境界ではありません。人間はトラフィックです。

境界は退屈です：ファイルシステムの隔離、静的暗号化されたシークレット、短時間有効な資格情報、ハードウェアベースの認証、そして偽のシークレットが触れた瞬間に発火するアラートです。

もし悪意のあるプロセスが実行された場合、あなたが午後の悪い出来事か企業全体のインシデントかを決定する質問は：
1. このプロセスが**読み取れる**のは何ですか？
2. どの資格情報を**使用できる**ですか？
3. どこに**データを送信できる**ですか？

## 今すぐ実施すべき最高リバレッジ戦略

### デベロッパーラン — デフォルトで使用

[Development Containers](https://github.com/devcontainers/spec)は、ほとんどのチームが実施していない最もリバレッジの高い変更です。Dev Containerはプロジェクトの作業を隔離されたDockerコンテナ内で実行します。`npm install`、`pip install`、`postinstall`スクリプト、AIシェルコマンド、VS Code拡張機能——これらすべては、他のマシンを参照できない「ワークスペース」またはコンテナ内で実行されます。

<p class="inset">Claude Codeに任意のプロジェクトでDevContainersを設定させることができます。</p>

リポジトリをマウントします。そのプロジェクトに必要なシークレットのみを含めます。`~/.ssh`、`~/.aws`、またはホームディレクトリを利便性のためにマウントしないでください。プロンプト注入された指示はエージェントが到達できるものにしかアクセスできません——それを退屈なものにしましょう。

```jsonc
// .devcontainer/devcontainer.json
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ]
}
```

### カナリートークン — 積極的に展開

[Canarytokens](https://canarytokens.org)は無料のデジタルトリップワイヤーです。攻撃者が探すであろう場所に偽装されたが説得力のあるシークレットを植えます。それが触れられた瞬間、あなたはアラートを受け取り——多くの場合数秒以内です。これは偽の現金束に染料パックを置くようなものです。

攻撃者は盗みに先立ってインベントリを行います。その偵察のパスがあなたの窓口です。

あなたの最も魅力的に見えるファイルにカナリートークンを配置してください：

```text
~/.aws/credentials          ← [billing-prod-legacy] プロファイルに偽のカナリートークンを追加
~/backups/customer-export-2024.sql   ← 内部にカナリートークンURLを配置
~/.env.canary               ← 各リポジトリに偽の資格情報を配置
```

カナリートークンは [canarytokens.org](https://canarytokens.org) で無料で利用可能で、セルフホストも可能であり、[Thinkst Canary](https://canary.tools) 経由で有料SaaSとして提供されています。盗み手が探すであろう場所に展開しない正当な理由はありません。

### パッケージセキュリティツール

[Socket.dev](https://socket.dev)、[Snyk](https://snyk.io)、[Wiz](https://wiz.io) などのツールは、サプライチェーン攻撃の進行を発見・ブロックする上で最初の警戒ラインとなることが多いです。これらは、自身で監視できないパッケージレジストリを監視します。フルタイムのセキュリティプログラムを維持できないチームにとって、これらは高リターンの早期警告システムです。

### PNPM の最小リリース年齢設定

PNPM を使用する場合、最小リリース年齢を設定してください。新規公開されたパッケージはサプライチェーン攻撃のリスクが最も高く、24時間未満の存在期間のパッケージは実質的にコミュニティの検証を受けていません。分単位で `minimumReleaseAge` を設定し、最低 `1440`（1日）以上、理想は `2880`（2日）を設定してください。

```yaml
minimumReleaseAge: 2880
minimumReleaseAgeStrict: true
minimumReleaseAgeIgnoreMissingTime: false
minimumReleaseAgeExclude:
  - 'typescript'
```

この設定は、特に次のインストール前に削除される新規公開パッケージの攻撃を多くの場合ブロックします。`minimumReleaseAgeExclude` は、即時更新が重要なコンパイラーやランタイム依存など、遅延よりも更新が優先されるパッケージにのみ限定的に使用してください。

### セキュリティが最優先の環境向け

情報機関、法執行機関、金融取引インフラ、医療記録 — これらの環境では、厳格なパッケージ評価と承認プロセスを採用することがあります。これは安全そうに聞こえますが、トレードオフは深刻です。依存関係ツリーが徐々に古いソフトウェアに石化してしまうのです。

ここでは時間は中立ではありません。古いバージョンは既知のCVEが蓄積されます。攻撃者は修正済みバージョンを研究して未修正のインスタンスを見つけます。そして「知っている悪魔ほど恐ろしいものはない」というのは救いではなく、単に攻撃者がどの脆弱性を最も長く研究しているかを教えてくれるだけです。

厳格なホワイトリストは、メンテナンススタッフが十分いる場合にのみ機能します。ほとんどのチームにはそのリソースはありません。他のすべてのチームにとって、レイヤードアプローチ — Dev Containers、カナリートークン、パッケージセキュリティツール、短期間の資格情報 — が、すべての依存関係を手動で監査するという幻想よりも現実的な防御です。

## あなたには数分しかない

カナリートークンが作動したとき、またはGitHubが予期しないIPからのトークン使用を通知したとき、あなたには窓があります。数分、あるいは数時間。1週間ではありません。

- **まずはローテーション、後で調査。** 何が起きたか理解する前にトークンを無効化してください。
- **攻撃者の持続性を確認。** 退職前に作成された新しいOAuthアプリ、IAMユーザー、デプロイキー、APIトークンを確認。
- **アクティブなブラウザセッションを終了。** 重要資産すべてから強制ログアウト。
- **誰かに知らせる。** 見証者とタイムスタンプがあるとセキュリティインシデントの改善につながります。

セキュリティ業界は検出について多く語ります。しかし検出後20分間、あなたが机に座ってどのサービスにトークンがあるかを思い出している間に何が起こるかについては語りません。

そのリストはアラートが発生する前に存在している必要があります。

## 価値のある基準

価値のある基準は「何でもかんでもクリックしないこと」ではありません。これはポスターのためのアドバイスであり、システムではありません。

悪い依存関係は他のプロジェクトのクラウド資格情報を取得してはいけません。プロンプト注入されたドキュメントはエージェントをホームディレクトリにリダイレクトしてはいけません。インフォステーラーは平文のバックアップや長期有効なトークンを見つけずにアラームを発動させなければなりません。盗まれた資格情報は、完全な乗っ取りになる前に期限切れになる、MFAを失敗させる、またはカナリートークンにヒットしなければなりません。

セキュリティは、人間が完璧であることを求めるのをやめて、妥協が利益を生まなくなるようにするときに向上します。  
あなたのノートPCは今や運用環境の一部です。侵入した攻撃者だけでなく、うっかり自分自身が許可してしまった存在も捕まえる、単調な境界を与えてください。
