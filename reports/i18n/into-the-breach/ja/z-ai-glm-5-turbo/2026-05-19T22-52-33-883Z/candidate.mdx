## あなたは認証情報の倉庫である

あなたのノートPCはノートPCではない。キーボードのついた認証情報の倉庫だ——ブラウザセッション、SSH鍵、`.env`ファイル、GitHubトークン、クラウドCLI、シェルアクセスを持つAIコーディングツール、存在すら忘れたデータベースエクスポート。

かつてのモデルはこうだった：本番環境は危険で、ローカルは便利。そのモデルは終わった。

<p class="inset">
問題は、すべての悪意あるクリックを回避できるかではない。問題は、たった一度の悪意あるクリックですべてを読み取り、すべてを使いこなし、気づく前に消え去れるかどうかだ。
</p>

開発者は、ごく普通に見えるものに出会う：請負業者からのPDF、ターミナルに何かを貼り付けるよう求める偽のCAPTCHA、`postinstall`スクリプトを含むパッケージ、タスクの必要以上にファイルシステムの奥深くまで踏み込んだAIコーディングセッション。一部の経路はマルウェアをインストールし、一部は認証情報を窃取する。ローカルエクスプロイトすら不要なものもある——ユーザー自身が攻撃者のコマンドを実行するのだ。

これが現代の攻撃面だ。時には、あなた自身が侵害の起点になる。

## サプライチェーン問題は広大すぎる

ここからが面白い部分だ。完全に安全になるために必要なのは、依存しているすべてのパッケージ——メンテナー、履歴、推移的な依存関係——について、すべてのパッケージレジストリを横断した深く多角的なセキュリティ評価を行うことだけだ。そして、依存関係ツリーが変わるか更新が入るたびにその評価を繰り返す。なぜなら、それこそがサプライチェーン攻撃の仕組みだからだ：信頼の連鎖を突くのだ。

簡単だろう。

ああ、それと攻撃者は一度だけ成功すればいい。あなたは毎回完璧な防御を維持しなければならない。

Lumma Stealer——パスワード、ブラウザのCookie、APIキー、クラウドの認証情報を密かに収集する広く使われる情報窃取マルウェア——は、偽のCAPTCHA、毒入りの検索広告、トロイの木馬化されたアプリを通じて被害者に到達した。MandiantのSnowflake調査は、企業への連鎖的な侵害を情報窃取マルウェアによって盗まれた認証情報にまで遡り、中には2020年まで遡るものもあった。攻撃に使われたアカウントの少なくとも79.7%が、以前の情報漏洩の記録があった。鍵は替えられていなかった。

攻撃者は倉庫を破ったわけではない。机の引き出しに古い鍵を見つけただけだ。

開発者にとって、その引き出しはこんな感じだ：

| ローカルのアーティファクト | 攻撃者が狙う理由 |
| --- | --- |
| ブラウザのCookie | ログインをバイパスし、場合によってはMFAをスキップできる。 |
| `.env`ファイル | APIキー、データベースURL、JWTシークレット。 |
| クラウドCLIの設定 | ラップトップの侵害をインフラ全体へのアクセスに昇格させる。 |
| SSH鍵 | まだ至る所にあり、依然として強力で、マシン間でコピーされ続けている。 |
| パッケージマネージャーのトークン | npmやPyPIの公開トークンは、サプライチェーンへのアクセス権そのもの。 |
| データベースのダンプ | 本番環境より保護が弱く、しかもより完全であることが多い。 |
| AIコーディングのコンテキスト | アシスタントに「コンテキスト用」として機密ファイルが渡されているかもしれない。 |

そしてバックアップもある——誰かが`~/Downloads`に置きっぱなしにして忘れた本番環境のエクスポートだ。バックアップは不活性だからといって安全ではない。単に警報システムのない本番環境にすぎない。

## 「気をつけて」は解決にならない

「気をつけて」は弱いアドバイスだ。人間に境界線になれと求めている。

人間は境界線ではない。人間はトラフィックだ。

境界線は退屈なものだ：ファイルシステムの分離、保管時暗号化されたシークレット、短命な認証情報、ハードウェア基盤の認証、そして偽のシークレットに触れられた瞬間に発火するアラート。

悪意のあるプロセスが実行されたとき、午後を台無しにされるだけか、社全体のインシデントになるかを決める問いは次の通りだ：
1. このプロセスは何を**読み取れる**か？
2. どんな認証情報を**使える**か？
3. データを**どこに送れる**か？

## 今すぐできる最も効果の高い対策

### Dev Containers —— デフォルトで導入する

[Development Containers](https://github.com/devcontainers/spec)は、ほとんどのチームがまだやっていない、単一で最も効果の高い変更だ。Dev Containerはプロジェクトの作業を分離されたDockerコンテナ内で実行する。`npm install`、`pip install`、`postinstall`スクリプト、AIのシェルコマンド、VS Code拡張機能——すべてがマシンの残りの部分を見られない「ワークスペース」またはコンテナ内で行われる。

<p class="inset">Claude Codeに任意のプロジェクトでDevContainersのセットアップを依頼する。</p>

リポジトリをマウントする。そのプロジェクトに必要なシークレットだけを含める。便利だからといって`~/.ssh`や`~/.aws`、ホームディレクトリをマウントしない。プロンプトインジェクションされた命令は、エージェントが届く範囲にしか届かない——その範囲を退屈なものにしろ。

```jsonc
// .devcontainer/devcontainer.json
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ]
}
```

### Canary Tokens —— 積極的に配置する

[Canarytokens](https://canarytokens.org)は無料のデジタルトリップワイヤーだ。攻撃者が探しそうな場所に、偽物だがもっともらしいシークレットを仕込む。触れられた瞬間にアラートが飛ぶ——多くの場合、数秒以内に。偽の札束に染料パックを仕込んでおくようなものだ。

攻撃者は盗む前にインベントリを作る。その偵察のパスがあなたの窓だ。

最も魅力的に見えるファイルにカナリアを仕込め：

```text
~/.aws/credentials          ← add a fake [billing-prod-legacy] profile with a canary key
~/backups/customer-export-2024.sql   ← canary URL inside
~/.env.canary               ← fake credentials in every repo
```

Canarytokenは[canarytokens.org](https://canarytokens.org)で無料で利用でき、セルフホストも可能だ。有料SaaSとして[Thinkst Canary](https://canary.tools)からも提供されている。泥棒が覗き込みそうな場所すべてに配置しない正当な理由はない。

### パッケージセキュリティツール

[Socket.dev](https://socket.dev)、[Snyk](https://snyk.io)、[Wiz](https://wiz.io)などのツールは、進行中のサプライチェーン攻撃を発見し、ブロックする最初のラインになりうる。自分では監視しきれないパッケージレジストリを監視してくれる。専任のセキュリティ体制を維持できないチームにとって、これらはレバレッジの高い早期警戒システムだ。

### PNPMの最小リリース年齢設定

PNPMを使っているなら、最小リリース年齢（minimum release age）を設定しろ。新しく公開されたパッケージは、サプライチェーン攻撃のリスクが最も高い期間だ——存在して24時間未満のパッケージには、実質的にコミュニティのレビューが一切入っていない。`minimumReleaseAge`は分単位で設定する：最低でも`1440`（1日）、理想を言えば`2880`（2日）だ。

```yaml
minimumReleaseAge: 2880
minimumReleaseAgeStrict: true
minimumReleaseAgeIgnoreMissingTime: false
minimumReleaseAgeExclude:
  - 'typescript'
```

この設定は、新規公開パッケージの攻撃の多くをブロックする——特に、次回インストールの前に発見されて削除されるようなものだ。`minimumReleaseAgeExclude`は慎重に使え。コンパイラやランタイムなど、遅延よりも即時アップデートが重要な、能動的に追跡している依存関係に限定すること。

### 最もセキュリティクリティカルな環境の場合

情報機関、法執行機関、金融取引インフラ、医療記録——こうした環境では、厳格なパッケージ評価・承認プロセスを採用することがある。安全に聞こえるかもしれない。しかしトレードオフは深刻だ：依存関係ツリーがゆっくりと古いソフトウェアへと石化していく。

時間はここでは中立ではない。古いバージョンには既知のCVEが蓄積していく。攻撃者は修正版を研究して、未修正のインスタンスを見つけ出す。「知っている悪魔の方がマシ」という考えは、あなたが期待した救いにはならない——それは単に、攻撃者が最も長く熟練してきた脆弱性がどれかを教えてくれるだけだ。

厳格な許可リストは、それを維持する人員がいれば機能する。だがほとんどのチームにはいない。その他のすべてのチームにとって、多層防御アプローチ——Dev Containers、カナリアトークン、パッケージセキュリティツール、短期間の認証情報——は、すべての依存関係を手作業で監査できると装うよりも、はるかに現実的な防御となる。

## あなたに残された時間は分単位だ

カナリアが発火したとき——あるいはGitHubが、トークンが予期しないIPから使われたと警告してきたとき——あなたにはわずかな時間の窓しかない。数分、せいぜい数時間だ。1週間ではない。

- **まずローテートし、後で調査しろ。** 何が起きたか理解する前にトークンを失効させろ。
- **攻撃者の永続化を確認しろ。** 新規のOAuthアプリ、IAMユーザー、デプロイキー、APIトークン——攻撃者が去る前に作成されたものがないか。
- **アクティブなブラウザセッションを切断しろ。** 重要なすべてのサービスから強制ログアウトさせろ。
- **誰かに伝えろ。** セキュリティインシデントは、目撃者とタイムスタンプがある方がマシになる。

セキュリティ業界は検知についてよく語る。しかし、検知から20分後、デスクに一人で座って、自分がどのサービスにトークンを持っているか思い出そうとするあの時間のことについては、あまり語られない。

そのリストは、アラートが鳴る前から存在しているべきだ。

## 持つべき基準

基準とは「怪しいものは一切クリックするな」ではない。それはポスターに書く助言であって、システムの基準ではない。

悪意のある依存関係が、他のプロジェクトのクラウド認証情報に到達できるようであってはならない。プロンプトインジェクションされたドキュメントが、エージェントをホームディレクトリに誘導できるようであってはならない。情報スティーラーが、アラートを引き起こすことなく、平文のバックアップと長期間有効なトークンを見つけられるようであってはならない。盗まれた認証情報は、完全な乗っ取りに至る前に、期限切れになり、MFAに弾かれ、あるいはカナリアに引っかかるべきだ。

人間に完璧を求めるのをやめ、侵害のリターンを下げる方向にシフトすると、セキュリティは改善する。

今やあなたのノートPCは本番環境の一部だ。外部から侵入してきた攻撃者も、あなたがうっかり招き入れてしまった者も、どちらも弾く「退屈な」境界線を張れ。
