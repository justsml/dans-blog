## 「注意して」という非ソリューション

「注意して」は弱いアドバイスです。人間を境界にすることを要求しています。

人間は境界ではありません。人間はトラフィックです。

境界は退屈です：ファイルシステム分離、保存時暗号化シークレット、短命な資格情報、ハードウェア зад-backed auth、そして偽のシークレットが触られた瞬間に発火するアラート。

恶意プロセスが実行された場合、午後の悲剧か会社全体のインシデントを決定する質問は：
1. このプロセスは何を**読める**か？
2. このプロセスはどの資格情報を**使える**か？
3. データはどこに**送れる**か？

## PNPM minimumReleaseAge Configuration

When implementing PNPM, establishing a minimum release age threshold proves crucial. Newly published packages represent the highest vulnerability window for supply chain exploits—a package under 24 hours old has received virtually no community examination. Configure minimumReleaseAge in minutes: recommend setting at minimum `1440` (one day), ideally `2880` (two days).

```yaml
minimumReleaseAge: 2880
minimumReleaseAgeStrict: true
minimumReleaseAgeIgnoreMissingTime: false
minimumReleaseAgeExclude:
  - 'typescript'
```

This configuration prevents most fresh-package attack vectors, especially those identified and removed before subsequent installations. Use minimumReleaseAgeExclude sparingly for dependencies like compilers or actively monitored runtime components where immediate updates outweigh security delays.

## For Maximum Security-Sensitive Contexts

Intelligence agencies, law enforcement, financial trading infrastructure, and health records sectors sometimes implement rigorous package evaluation protocols. While this approach sounds secure, it carries significant tradeoffs: dependency trees gradually consolidate into outdated software versions.

Time isn't neutral here. Older versions accumulate documented CVEs. Attackers analyze patched versions to identify unpatched targets. The "devil you know" approach doesn't provide the expected security—it simply indicates which vulnerabilities attackers have had extended time to exploit.

Strict allowlists function only with dedicated maintenance staff. For most organizations, the layered strategy—Dev Containers, canary tokens, package security tooling, short-lived credentials—delivers more practical protection than pretending to manually audit every dependency.

## Response Window

When a canary triggers or GitHub alerts to unauthorized token usage from unexpected locations, action time is limited. Minutes, possibly hours—not weeks.

- **Rotate credentials immediately, investigate later.** Invalidate tokens before understanding the full context.
- **Check for attacker persistence.** Examine any recently created OAuth applications, IAM users, deployment keys, or API tokens.
- **Terminate active sessions.** Force logout from all sensitive services.
- **Report the incident.** Security breaches improve with witnesses and documented timestamps.

The security sector emphasizes detection but undercommunicates what happens in the twenty minutes post-detection when you're alone at your desk trying to recall which services hold your tokens.

That inventory should exist before any alert triggers.

## Achievable Security Standard

The standard isn't "never interact with suspicious content." That's poster advice, not operational strategy.

Insecure dependencies shouldn't access cloud credentials across projects. Prompt-injected documents shouldn't redirect agents into home directories. Information stealers shouldn't locate plaintext backups and persistent tokens without triggering detection. Compromised credentials should expire, fail multi-factor authentication, or trip a canary before full system compromise occurs.
