## 设定威胁等级

大多数人想象的是戏剧性的攻击——一个零日漏洞、一个带着日历邀请的国家级黑客。某种足够新奇的东西，让普通的工程纪律显得无关紧要。

更平淡的版本反而更有用。

开发者遇到一些看起来足够正常的东西：

- 来自承包商的 PDF 发票
- 关于快递或账户警告的短信
- 一个伪造的 CAPTCHA，要求他们将命令粘贴到终端
- 一个投毒的搜索广告，指向他们本来就要安装的工具
- 一个浏览器扩展，悄悄请求过多权限
- 一个拉取请求，添加了一个带有 postinstall 脚本的开发依赖
- 一次 AI 编码会话，读取了比任务所需更多的文件系统内容
- 一个 GitHub Actions 工作流，通过一个本不该看到的环境变量泄露了密钥
- 一个注入到文档、网页或仓库中的提示，重定向了 AI 代理的下一个动作

其中一些路径会安装恶意软件。一些通过钓鱼窃取凭据。一些根本不需要本地漏洞——用户亲手运行了攻击者的命令。

微软关于 Lumma Stealer 的分析报告是一个有用的快照。Lumma 是一种广泛使用的*信息窃取器*——一种从受感染机器上静默收集密码、浏览器 Cookie、API 密钥和加密货币钱包的恶意软件。它通过钓鱼邮件、恶意广告、伪造 CAPTCHA 和木马化应用到达受害者手中。有趣的部分不是 Lumma 这个品牌——而是策略：当用户整天穿梭在一座充满半信任门的城市中时，攻击者不需要一扇完美的门。

这样设定威胁等级：

> 假设一个进程可以以你的身份运行几分钟。

不是以 root 身份。不是永远。只是以你的身份。

## 你就是入侵

“我的笔记本电脑被入侵了”这句话带有被动语态，并不总是适用。

有时故事是：我克隆了仓库，运行了安装，然后 postinstall 脚本在测试开始前就回传了数据。我打开了别人发送的文件。我批准了工作流触发器。我粘贴了那个东西。我给了代理“完整上下文”，因为那比指定它需要哪些文件更容易。

现代攻击面包括那些你作为行为者的地方。

### 提示注入

隐藏在文件、README、PR 描述或评论中的恶意指令可以重定向代理的行为。代理将文档视为内容。隐藏的指令也是内容。如果模型将注入的文本视为命令，代理可能会执行用户从未意图的操作——读取文件、调用工具或遵循一条本不属于他们的指令链。

这不需要一个被攻破的模型。它需要一个代理被要求处理的文档。

实际影响：

- 不要为了“上下文”而给代理无限制的文件系统访问权限。上下文不是免费的。
- 在代理执行之前审查它提议的操作，尤其是当它未经明确请求就访问了文件时。
- 如果代理突然想要读取凭据、发送网络请求或对“它在查看项目时发现的东西”采取行动，要保持怀疑。
- 将 AI shell 会话保持在具有狭窄挂载点的 Dev Container 内。注入的指令只能作用于代理能够触及的内容。

### GitHub CI/CD

GitHub Actions 功能强大、值得信赖，但经常配置不当。其后果往往与笔记本电脑被攻破殊途同归：凭据、源代码和部署权限。

**被投毒的第三方 Action。** 你的工作流引用了 `uses: some-org/some-action@v2`。像 `@v2` 这样的版本标签是可移动的标签——如果上游仓库被攻破，或者该标签被重定向到恶意提交，你的工作流就会用你仓库的密钥运行攻击者的代码。修复方法：将 Action 固定到完整的提交 SHA。

**Pull Request 触发器滥用。** `pull_request_target` 是一个触发器，即使 PR 来自外部贡献者，它也能以访问基础仓库密钥的权限运行工作流。粗心的工作流会将这些密钥暴露给不可信代码。这是 GitHub 文档中已知的脚枪。

**通过不可信输入注入工作流。** 将 `${{ github.event.pull_request.title }}` 直接插值到 `run:` 步骤中，攻击者就能构造一个注入 shell 命令的 PR 标题。始终通过中间环境变量传递用户控制的值。

**从 Fork 中窃取密钥。** Fork 的 PR 默认不会收到仓库密钥，但围绕 `pull_request_target` 和环境保护规则的错误配置可能会改变这一点。

实际底线：

- 将第三方 Action 固定到完整的提交 SHA。
- 永远不要将 `github.event` 字段直接插值到 `run:` 步骤中。
- 将生产密钥保存在具有保护规则和必需审查者的环境中。
- 审计谁可以触发具有敏感密钥访问权限的工作流。
- 使用短期凭据交换（OIDC）进行云访问，而不是在 CI 中存储长期密钥。

## 硬盘就是奖品

信息窃取器想要你的磁盘——具体来说，是那些多年来悄然积累了大量受信访问权限的地方。

微软在 2025 年 3 月至 5 月期间识别出超过 394,000 台感染了 Lumma 的 Windows 计算机，该恶意软件收集了密码、信用卡和金融账户凭据。

Mandiant 对 Snowflake 的调查揭示了更可怕的商业层面。该活动中的每一起事件都追溯到被攻破的客户凭据——而不是 Snowflake 自身基础设施的漏洞。这些凭据来自无关机器上的信息窃取器感染，有些早在 2020 年就被窃取了。攻击中使用的账户至少有 79.7% 已知此前已暴露——这意味着密码早已被盗，却无人更改。

攻击者并没有攻破仓库。他们只是在办公桌抽屉里找到了旧钥匙，然后发现锁从未换过。

对开发者来说，这个办公桌抽屉就是一个杂物间：

| 本地工件 | 攻击者关心的原因 |
| --- | --- |
| 浏览器 Cookie 和已保存的会话 | 可以绕过登录页面，有时还能跳过多因素认证（MFA）。 |
| `.env` 文件 | API 密钥、数据库连接字符串、JWT 密钥、第三方令牌。 |
| 云 CLI 配置 | 将笔记本电脑被攻破转化为完整的基础设施访问权限（AWS、GCP、Azure）。 |
| Git 凭据 | 源代码映射系统、密钥和部署路径。 |
| SSH 密钥 | 仍然无处不在，仍然强大，仍然在机器之间复制。 |
| 数据库转储 | 保护不如生产环境，通常更完整。 |
| AI 编码上下文 | 助手可能被提供了敏感文件或额外目录。 |
| 包管理器令牌 | 如果你的 npm 或 PyPI 发布令牌在本地，供应链访问权限也在。 |
| GitHub 令牌 | 个人访问令牌可以读取仓库、触发工作流和发布包。 |

备份值得特别关注。

团队用访问控制和审计日志保护生产数据库。然后有人将相同的数据导出为 `customer-backup-final-2.sql.gz`，放在工作站上，然后忘记它的存在。

那个文件可能包含比生产环境更敏感的数据——更容易复制、更容易搜索，而且更不可能被监控。

备份并不因为静止而更安全。它们只是没有警报系统的生产环境。

## 完整接管模式

“数据泄露”这个词不足以描述接下来发生的事情。

1. **初始接触**：用户打开文件、点击链接、安装工具、运行复制的命令，或访问被攻破的页面。
2. **清点**：恶意进程扫描机器——目录、配置文件、浏览器数据、环境变量。它弄清楚自己有什么。
3. **本地抓取**：浏览器会话、配置文件、`.env`文件、令牌、SSH密钥、shell历史记录和项目目录被复制出去。
4. **云侧跳转**：被盗凭据被用于登录云账户、GitHub、CI系统或SaaS工具——通常只需几分钟。
5. **备份扫描**：本地导出、云存储桶、CI制品和数据库快照成为目标，因为它们比生产环境更脆弱。
6. **持久化**：在窗口关闭之前，攻击者创建新的API密钥、OAuth应用或服务账户——这样即使密码更改后他们也能返回。
7. **勒索或转售**：数据直接变现、作为访问权限出售，或为未来的攻击保存。

你的笔记本电脑是一个身份代理。它向你所使用的每个系统证明你是谁。如果攻击者窃取了足够的证明，他们就能以你的身份出现。

注意第二步：**先清点**。大多数攻击者在窃取之前会先浏览。他们四处查看，打开目录，检查存在哪些凭据。

这正是金丝雀令牌设计用来利用的窗口。

## 开发者工具扩大了爆炸半径

容器使本地环境可复现。包管理器使依赖安装无摩擦。云CLI使基础设施可编程。AI编码工具使终端变得对话式。

都是好事。但当指向一个充满秘密的工作站时，也都危险。

开发依赖中的供应链攻破不需要进入生产环境就足以造成影响。一个恶意的`postinstall`脚本——安装包时自动运行的代码——可以在你运行任何测试之前读取本地文件、检查环境变量并将它们发送出去。一个拥有广泛文件系统和shell权限的AI代理可以放大一个错误的指令或错误的假设。

这就是为什么“小心”是如此无力的建议。它要求人类成为边界。

人类不是边界。人类是流量。

边界是些无聊的东西：文件系统隔离、静态加密的秘密、默认拒绝的出站规则、短期凭据、硬件支持的身份验证，以及当假秘密被触碰时触发的警报。

## 更好的框架：读取、使用、外泄

每个工作站防御应该回答三个问题：

1. 这个进程能**读取**什么？
2. 它能**使用**什么凭据？
3. 它能将数据**发送**到哪里？

大多数工作站安全建议止步于第一个。保持软件更新。不要打开可疑附件。使用杀毒软件。好的，是的，显然。

但如果一个恶意进程确实运行了，问题二和问题三将决定你只是度过一个糟糕的下午，还是引发一场全公司范围的事故。

它能读取 `~/.aws/credentials` 吗？它能使用 GitHub 令牌吗？它能打开你的密码管理器扩展吗？它能悄无声息地向随机主机上传 3 GB 数据吗？

这个框架把威胁从迷雾机变成了带牙齿的检查清单。

## 我会优先做什么

如果我要收紧开发者工作站方案，但又不想把公司变成一座悲伤的机场，我会从这里开始。

### 1. 将有风险的工作移入开发容器

对于需要依赖、构建工具、包安装或 AI 辅助 shell 命令的项目工作，使用[开发容器](https://github.com/devcontainers/spec)。开发容器是一个本地 Docker 容器，作为项目的隔离工作空间——除非你显式挂载，否则它无法看到你机器的其余部分。

好处：`npm install`、`pip install`、`go generate`、`cargo build` 以及模型想运行的任何东西，都在一个不会自动拥有你整个主目录的工作空间内执行。

挂载仓库。只挂载该项目所需的密钥。避免为了方便而挂载 `~/.ssh`、`~/.aws`、`~/Downloads` 以及整个主文件夹。

```jsonc
// .devcontainer/devcontainer.json — 仅窄挂载
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "workspaceFolder": "/workspaces/app",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ],
  "containerEnv": {
    "NODE_ENV": "development"
  },
  "postCreateCommand": "bun install"
}
```

注入有作用域的凭据。优先使用短期令牌。尽可能使用只读访问。一个提示注入的指令只能触及代理能触及的东西——让那部分变得无聊。

### 2. 加密本地密钥，而不是崇拜 `.env`

明文 `.env` 文件很方便，因为文件本身就方便。攻击者也喜欢文件。

[VarLock](https://varlock.dev/guides/secrets/) 将敏感性视为结构化元数据——你标记哪些值是敏感的，它会在本地加密，从控制台输出中隐去，并扫描本应是秘密的值的明文出现。

```dotenv
# .env.schema
# @sensitive
STRIPE_SECRET_KEY=

# @sensitive
DATABASE_URL=
```

密钥应该知道自己是被保密的。它无法保护已经加载到被入侵进程中的密钥，但能减少那些等着变成别人库存的有价值明文文件的数量。

### 3. 在窃贼可能查看的每个地方放置金丝雀令牌

这是大多数团队跳过的一层，而且可以说是最立竿见影的一层。

[Canarytokens](https://help.canary.tools/hc/en-gb/articles/10905485310109-Canarytoken-Overview-and-Use-Cases) 是数字绊线。在攻击者可能查看的地方放置一个虚假但令人信服的密钥、API 密钥或 URL。一旦它被触碰，你就会收到警报——通常只需几秒钟。可以把它想象成在一叠假钞里放一个染料包：一旦有人打开它，你就知道了。

回想接管模式的第二步：**先清点库存**。攻击者在窃取之前会先浏览。那次侦察扫描就是你的窗口。

一个放在正确位置的金丝雀令牌会在数据离开之前触发警报。

**在本地机器上：**

```text
~/backups/customer-prod-export-2024.sql
~/Documents/passwords-old.csv
~/.aws/credentials   ← 添加一个假的 [billing-prod-legacy] 配置段，内含金丝雀 AWS 密钥
~/.ssh/config        ← 添加一个指向金丝雀的假主机条目
```

在这些文件里放一个金丝雀 URL。如果有人打开文件并访问了链接，你就会知道。

**在仓库中：**

- 一个包含假凭据的 `.env.canary` 文件
- 带有假服务令牌的旧部署手册
- 攻击者在源码侦察期间会检查的废弃配置文件

**在 CI/CD 中：**

- 一个命名类似部署令牌的假 CI 密钥
- GitHub 环境中的假 kubeconfig

**在云账户中：**

- 一个没有权限但持有真实金丝雀 API 密钥的假 IAM 用户
- 一个包含金丝雀对象的未使用 S3 存储桶路径

警报应该是可操作的。一个发送到无人值守收件箱的金丝雀只是装饰。把它路由到能叫醒人的地方——PagerDuty、带 @ 提醒的 Slack、短信——并注明哪个令牌触发了、它被埋在哪里，以及轮换检查清单。

#### 值得了解的盲点

加密货币钱包窃密程序可能会抓取钱包文件，而从不碰你的假 AWS 凭据。勒索软件操作者可能在金丝雀触发之前就加密了磁盘。一个已经了解你布局的定向攻击者可能会完全跳过侦察阶段。

这没关系。金丝雀令牌并非为所有威胁而设计——它们是为最常见的威胁设计的：一个机会主义的攻击者，他会进行凭据扫描、浏览看起来有趣的文件、盘点你的访问权限，然后决定偷什么。大多数攻击者都是这样。

一个假的 AWS 密钥在有人试图使用时触发，这给了你在他们找到真实密钥之前进行轮换的时间窗口。

目标不是全知全能。目标是让侦察阶段变得代价高昂。

### 4. 添加出站防火墙

大多数人想到“防火墙”时，会想象阻止入站连接。这忽略了工作站的问题。

如果恶意软件能读取本地密钥，下一个问题是它能否把它们发送出去。大多数锁朝外——出站防火墙朝内。它不关心谁在试图访问你的机器；它关心什么在试图离开它。

在 macOS 上，[LuLu](https://objective-see.org/products/lulu.html) 是免费的开源选择。[Little Snitch](https://obdev.at/products/littlesnitch/) 是打磨精致的商业版，支持按应用和按域名规则。在 Windows 和 Linux 上，[Portmaster](https://safing.io/) 值得评估。

这一层起初会有些烦人。但这不能成为跳过它的理由。目标是当 `postinstall`、`python` 或 `invoice-viewer` 想要联系一个与你的周二毫无关系的域名时，你能注意到。

### 5. 把 AI 编码工具当作健忘的初级管理员

AI 编码工具本身不坏。我也用，我也喜欢。

但它们拥有读权限、写权限、shell 权限、网络权限，以及一种自信推进的天赋。它们会按收到的指令行事——而如果收到的指令包含一条它们无法与合法内容区分的恶意指令，它们也会照做。

Anthropic 的 Claude Code 文档区分了权限与沙箱。权限决定代理*允许*使用什么。沙箱提供操作系统级别的强制隔离。策略文本不是沙箱。权限提示不是沙箱。善意模型也不是沙箱。

使用项目级别的允许和拒绝规则。将敏感文件排除在工作目录之外。在容器内运行有风险的命令。不要因为代理可能需要“上下文”就把整个家目录交给它。

## 你只有几分钟，也许几小时

当金丝雀触发时——或者当供应商发邮件通知可疑登录，或者 GitHub 提醒你某个令牌从未知 IP 被使用时——下一步不是可选的阅读材料。

你有一个时间窗口。可能是几分钟。如果攻击者足够耐心，可能是几个小时。但绝不是一周。

该做什么：

- **先轮换，再调查。** 在搞清楚发生了什么之前就撤销令牌。损害控制优先。
- **检查 GitHub 令牌、OAuth 应用和部署密钥。** 攻击者如果接触过你的笔记本，可能在离开前创建了新的凭据。
- **审查最近的云活动。** 查找不是你创建的 IAM 用户、服务账号、API 密钥或存储策略。
- **审计 CI。** 检查是否有工作流意外运行，尤其是在你最近没有碰过的仓库中。
- **终止活跃的浏览器会话。** 强制登出所有你在意的服务。
- **告诉别人。** 安全事件有目击者和时间戳会更好。

安全社区谈论检测很多，但很少谈论检测后的二十分钟里发生了什么——那时你独自坐在工位上，努力回忆自己有哪些服务的令牌。

那份清单应该在警报触发之前就存在。

## 关于备份的说明

备份是安全计划中自我欺骗的地方。

备份是必要的。它们也是危险的。备份是你最不希望被便携的东西中最便携的形式。

- 除非确有需要，否则不要将生产环境的导出文件存储在本地。
- 对本地备份和数据库转储进行加密。
- 为导出文件添加过期日期。
- 在类似备份的文件中放置金丝雀行或文档。
- 将备份排除在宽泛的 Dev Container 挂载和 AI 工具上下文之外。
- 轮换任何出现在备份中的凭据。

如果备份包含凭据，那它就不只是备份——它是一个延迟的接管工具包。

## 实用标准

标准不应该是“永远不要点击任何奇怪的东西”。那是贴在墙上的建议，不是系统应有的标准。

实用标准是：

- 一个恶意的 PDF 不应能读取所有项目密钥
- 一个恶意的依赖项不应能看到其他项目的云凭据
- 一个被提示注入的文档不应将代理重定向到你的主目录
- 一个被投毒的 GitHub Action 不应能窃取你的部署令牌
- 一个信息窃取器不应能在不触发警报的情况下找到明文备份和长期有效的令牌
- 一个未知进程不应能在没有本地警报的情况下向外发送数据
- 一个被盗的凭据应在成为完全接管之前过期、MFA 失败、设备检查失败或触发金丝雀

当我们不再要求人类完美无缺，而是让妥协变得不那么有利可图时，安全性才会提升。

你的笔记本电脑现在已经是生产环境的一部分。攻击者并不总是破门而入——有时你会在不知情的情况下让他们进来。

给你的系统设置那种能同时拦住两者的边界。
