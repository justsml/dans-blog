## 你就是凭据仓库

你的笔记本电脑不是笔记本电脑。它是一个带键盘的凭据仓库——浏览器会话、SSH 密钥、.env 文件、GitHub 令牌、云 CLI、拥有 Shell 访问权限的 AI 编码工具，以及那些你早已遗忘的数据库导出文件。

旧模式是：生产环境危险，本地环境方便。那套模式已经结束了。

<p class="inset">
问题不在于你是否能避免每一次错误的点击。问题在于，一次错误的点击是否能在你察觉之前，读取所有内容、利用所有内容并悄然离开。
</p>

开发者遇到了一些看起来足够正常的东西：承包商的 PDF、一个要求他们将内容粘贴到终端的虚假验证码、一个带有 postinstall 脚本的软件包、一个深入文件系统程度超出任务要求的 AI 编码会话。有些路径会安装恶意软件。有些会窃取凭据。有些不需要本地漏洞利用——用户自己运行了攻击者的命令。

这就是现代攻击面。有时候你就是那个被入侵点。

## 供应链问题大得不可思议

这就是有趣的部分。要完全安全，你需要做的就是对你依赖的每一个依赖项进行深度、多平台的安全评估——包括它们的维护者、历史记录、传递依赖——覆盖每一个软件包注册表。然后，每次依赖树发生变化或收到更新时，都要重复进行评估，因为这就是供应链攻击的工作原理：它们利用信任链。

简单。

哦，而且攻击者只需要成功一次。而你每次都必须维持完美的防御。

Lumma Stealer（一种广泛使用的窃密软件，静默收集密码、浏览器 Cookie、API 密钥和云凭据）——通过虚假验证码、中毒的搜索广告和特洛伊木马应用程序攻击受害者。Mandiant 的 Snowflake 调查将一系列企业入侵追溯到由窃密软件窃取的凭据，有些甚至可以追溯到 2020 年。攻击中使用的账户中，至少有 79.7% 曾有过已知的先前暴露记录。锁从未被更换过。

攻击者并没有攻破仓库。他们只是在抽屉里找到了旧钥匙。

对开发者而言，那个抽屉长这样：

| 本地产物 | 攻击者为何在意 |
| --- | --- |
| 浏览器 Cookie | 可绕过登录，有时甚至能跳过 MFA。 |
| `.env` 文件 | API 密钥、数据库 URL、JWT 密钥。 |
| 云端 CLI 配置 | 将笔记本电脑沦陷转化为对整个基础设施的访问权限。 |
| SSH 密钥 | 依然无处不在，依然强大，依然在机器间复制。 |
| 包管理器令牌 | 你的 npm 或 PyPI 发布令牌就是供应链访问权限。 |
| 数据库转储 | 比生产环境受保护程度低，但数据往往更完整。 |
| AI 编码上下文 | 助手可能被传入了敏感文件“用于上下文”。 |

还有备份——有人把生产环境的导出文件扔进 `~/Downloads` 目录后就忘了。备份之所以安全，并不是因为它“惰性”或静止不动。它只是没有报警系统的生产环境。

## “小心”这个伪解决方案

“小心”是弱建议。它要求人充当边界。

人不是边界。人是流量。

边界是枯燥的：文件系统隔离、静态加密密钥、短期凭证、硬件支持认证，以及一旦假密钥被触碰就立即触发的警报。

如果恶意进程运行，决定你是度过一个糟糕的下午，还是引发一场公司级事故的问题是：
1. 这个进程能**读取**什么？
2. 它能**使用**什么凭证？
3. 它能**发送数据**到哪里？

## 现在杠杆率最高的举措

### 开发容器——默认启用

[开发容器](https://github.com/devcontainers/spec) 是大多数团队尚未采用的杠杆率最高的举措。开发容器在隔离的 Docker 容器内运行项目工作。`npm install`、`pip install`、`postinstall` 脚本、AI Shell 命令、VS Code 扩展——所有这些操作都在一个无法看到你机器其余部分的“工作区”或容器中进行。

<p class="inset">让 Claude Code 在任何项目中设置 DevContainers。</p>

挂载仓库。只包含该项目所需的密钥。不要为了图方便而挂载 `~/.ssh`、`~/.aws` 或你的主目录。提示注入的指令只能到达代理能触及的范围——让环境变得枯燥乏味。

```jsonc
// .devcontainer/devcontainer.json
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ]
}
```

### 金丝雀令牌——激进部署

[金丝雀令牌](https://canarytokens.org) 是免费的数字绊网。在攻击者会搜寻的地方种下一个虚假但极具迷惑性的“秘密”。一旦它被触碰，你就会收到警报——通常在几秒之内。把它想象成在假钞堆里藏了一个染料包。

攻击者在动手前会先进行资产盘点。那次侦察就是你的窗口。

在你最诱人的文件里埋下金丝雀：

```text
~/.aws/credentials          ← add a fake [billing-prod-legacy] profile with a canary key
~/backups/customer-export-2024.sql   ← canary URL inside
~/.env.canary               ← fake credentials in every repo
```

金丝雀令牌在 [canarytokens.org](https://canarytokens.org) 上免费提供，可自托管，也可通过 [Thinkst Canary](https://canary.tools) 作为付费 SaaS 使用。没有理由不在窃贼会搜查的地方部署它们。

### 包安全工具

像 [Socket.dev](https://socket.dev)、[Snyk](https://snyk.io) 和 [Wiz](https://wiz.io) 这样的工具通常是第一个发现并阻止正在进行的供应链攻击的工具。它们监控你自己无法监视的包注册表。对于无力承担全职安全项目的团队来说，这些是高杠杆的早期预警系统。

### PNPM 最小发布年龄设置

如果你使用 PNPM，请设置最小发布年龄。新发布的包是供应链攻击风险最高的窗口期——存在不到 24 小时的包基本上没有经过社区审查。以分钟为单位设置 `minimumReleaseAge`：至少 `1440`（一天），理想情况下是 `2880`（两天）。

```yaml
minimumReleaseAge: 2880
minimumReleaseAgeStrict: true
minimumReleaseAgeIgnoreMissingTime: false
minimumReleaseAgeExclude:
  - 'typescript'
```

该配置会阻止许多新发布的包攻击，特别是那些在你下次安装前被发现并拉取的攻击。对于更新速度比延迟更重要的包，请谨慎使用 `minimumReleaseAgeExclude`，例如你积极跟踪的编译器或运行时依赖。

### 针对最关键的安全环境

情报机构、执法部门、金融交易基础设施、健康记录——这些环境有时会采用严格的包评估和审批流程。这听起来很安全。代价很严重：你的依赖树会慢慢钙化成过时的软件。

在这里，时间并不中立。旧版本会积累已知的 CVE。攻击者研究已修复的版本以发现未修补的实例。而且“宁可信其有”并不是你希望的救赎——它只是告诉你攻击者已经掌握了最长时间的漏洞。

严格的允许列表只有在有人力维护时才有效。大多数团队没有。对于其他人来说，分层方法——Dev Containers、金丝雀令牌、包安全工具、短期凭证——提供了比假装可以手动审计每个依赖项更现实的防御。

## 你只有几分钟

当金丝雀触发——或者 GitHub 警告你令牌从意外的 IP 使用——你有一个窗口期。几分钟，也许几小时。不是一周。

- **先轮换，后调查。** 在理解发生了什么之前撤销令牌。
- **检查攻击者持久性。** 新的 OAuth 应用、IAM 用户、部署密钥、在他们离开前创建的 API 令牌。
- **终止活跃的浏览器会话。** 在你关心的所有内容上强制登出。
- **告诉某人。** 安全事件在有目击者和时间戳的情况下会得到改善。

安全行业谈论很多关于检测。它很少谈论在检测后独自坐在办公桌前试图回忆自己有哪些服务令牌的那二十分钟里发生了什么。

那个列表应该在警报触发前就存在。

## 值得拥有的标准

标准不是“永远不要点击任何奇怪的东西”。那是给普通人的建议，不是给系统的。

一个糟糕的依赖项不应该能够从其他项目访问云凭证。一个被注入提示的文档不应该将代理重定向到你的主目录。一个信息窃取程序不应该在未触发警报的情况下找到明文备份和长期令牌。被盗的凭证应该在成为完全接管之前过期、失败 MFA 或触发金丝雀。

当我们不再指望人类完美无缺，转而让入侵变得无利可图时，安全性才会提升。

你的笔记本电脑现在也是生产环境的一部分了。给它加上那些乏味的边界防护，既能拦截闯入的攻击者，也能拦截你自己不小心放进去的那个。
