## 你是凭证仓库

你的笔记本电脑不是笔记本电脑。它是带有键盘的凭证仓库——浏览器会话、SSH密钥、`.env`文件、GitHub令牌、云CLI、具有shell访问权限的AI编码工具、你忘记存在的数据库导出。

旧模型是：生产环境危险，本地环境便利。这个模型已经终结。

<p class="inset">
问题不是你能否避免所有恶意点击。问题是：一次恶意点击是否能读取一切、利用一切，并在你察觉前离开。
</p>

开发者遇到看似正常的东西：承包商的PDF、要求他们将内容粘贴到终端的假验证码、带有`postinstall`脚本的包、AI编码会话中访问文件系统超出任务需求的部分。某些路径安装恶意软件。某些窃取凭证。有些甚至不需要本地漏洞——用户自己运行了攻击者的命令。

这就是现代攻击面。有时，你就是漏洞。

## 供应链问题大到难以想象

有趣的是，要完全安全，你只需对依赖的每个包进行深度、跨平台的安全评估——包括维护者、历史记录、传递依赖——覆盖每个包注册表。然后每次依赖树变化或更新时重复评估，因为供应链攻击正是这样运作的：它们利用信任链。

简单。

哦，而且攻击者只需成功一次。你必须每次保持完美防御。

Lumma Stealer——一种广泛使用的窃密工具，可静默收集密码、浏览器cookie、API密钥和云凭证——通过伪造验证码、中毒的搜索广告和特洛伊化应用感染受害者。Mandiant对Snowflake的调查追溯到一系列企业入侵事件，源头是信息窃取工具盗取的凭证，部分事件可追溯至2020年。至少79.7%的攻击中使用的账户此前已知存在泄露。锁从未更换过。

攻击者并未闯入仓库。他们只是在抽屉里找到了旧钥匙。

对开发者而言，这个抽屉长这样：

| 本地工件 | 攻击者为何在意 |
| --- | --- |
| 浏览器 Cookie | 可绕过登录，有时甚至能跳过 MFA（多因素认证）。 |
| `.env` 文件 | 包含 API 密钥、数据库 URL 和 JWT 密钥。 |
| 云 CLI 配置 | 一旦笔记本被攻破，就等于获得了整个基础设施的访问权限。 |
| SSH 密钥 | 依然无处不在，依然强大，依然在不同机器间复制传播。 |
| 包管理器令牌 | 你的 npm 或 PyPI 发布令牌就是供应链的入口。 |
| 数据库备份 | 比生产环境保护得更差，但往往包含更完整的数据。 |
| AI 编程上下文 | 助手可能被“上下文”为由获取了敏感文件。 |

还有备份文件——有人把生产环境导出的数据丢在 `~/Downloads` 文件夹里就忘了。备份并不更安全，因为它是静态的。它只是没有警报系统的生产环境。

## “小心谨慎”的无效方案

“小心谨慎”是无效的建议。它要求人类成为防线。

人类不是防线。人类是流量。

真正的防线很无聊：文件系统隔离、静态加密的密钥、短时效凭证、硬件认证，以及在伪造密钥被触碰的瞬间触发的警报。

如果恶意进程正在运行，决定你只是度过糟糕下午还是遭遇全公司级事故的问题有三个：
1. 这个进程能**读取**什么？
2. 它能**使用**哪些凭证？
3. 它能**发送数据**到哪里？

## 当前最高效的安全措施

### 开箱即用的 Dev Containers

[开发容器](https://github.com/devcontainers/spec) 是大多数团队尚未实施的最高价值安全措施。Dev Container 在隔离的 Docker 容器中运行项目工作。`npm install`、`pip install`、`postinstall` 脚本、AI 命令、VS Code 扩展——所有操作都在一个无法访问你机器其他部分的“工作区”或容器中执行。

<p class="inset">让 Claude Code 为任何项目配置 DevContainers。</p>

挂载代码仓库。仅包含该项目所需的密钥。不要为了方便而挂载 `~/.ssh`、`~/.aws` 或你的主目录。提示注入式攻击只能访问代理可见的内容——让它尽可能无聊。

```jsonc
// .devcontainer/devcontainer.json
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ]
}
```

### 全面部署 Canary Tokens

[Canarytokens](https://canarytokens.org) 是免费的数字陷阱。在攻击者可能查找的位置植入一个看似真实但伪造的密钥。一旦被访问，你将在几秒内收到警报。这就像在假钞堆里藏染料包。

攻击者在盗窃前会先侦查。这正是你的窗口期。

将 Canary Tokens 部署在最诱人的文件中：

```text
~/.aws/credentials          ← 添加一个包含金丝雀密钥的虚假 [billing-prod-legacy] 配置文件
~/backups/customer-export-2024.sql   ← 内嵌金丝雀 URL
~/.env.canary               ← 每个仓库中放置伪造凭证
```

金丝雀令牌可在 [canarytokens.org](https://canarytokens.org) 免费获取，支持自托管，付费 SaaS 版本通过 [Thinkst Canary](https://canary.tools) 提供。没有正当理由不在任何窃贼可能查看的位置部署它们。

### 包安全工具

[Socket.dev](https://socket.dev)、[Snyk](https://snyk.io) 和 [Wiz](https://wiz.io) 等工具往往能率先发现并阻断正在进行的供应链攻击。它们监控你无法亲自监视的包注册表。对于无法负担专职安全团队的组织，这些工具是高杠杆的早期预警系统。

### PNPM 最小发布年龄设置

若使用 PNPM，请设置最小发布年龄。新发布的包是供应链攻击的高风险窗口——存在不足24小时的包几乎未经过社区审查。以分钟为单位设置 `minimumReleaseAge`：至少 `1440`（一天），理想值为 `2880`（两天）。

```yaml
minimumReleaseAge: 2880
minimumReleaseAgeStrict: true
minimumReleaseAgeIgnoreMissingTime: false
minimumReleaseAgeExclude:
  - 'typescript'
```

该配置可阻止许多新发布包攻击，尤其是那些在下次安装前被发现并撤回的恶意包。对编译器或运行时依赖等需要即时更新的包，谨慎使用 `minimumReleaseAgeExclude`。

### 对于最高安全级别的环境

情报机构、执法部门、金融交易基础设施、医疗记录——这些环境有时会采用严格的包评估和审批流程。这听起来很安全。代价是依赖树会逐渐僵化为过时软件。

此处时间并非中立。旧版本会累积已知漏洞。攻击者研究已修复版本以寻找未修补的实例。"已知的恶魔总比未知的好"并非救赎——它只是告诉你攻击者掌握哪些漏洞最久。

严格白名单只有在有足够人力维护时才有效。大多数团队不具备这种条件。对其他所有人而言，分层防御策略——Dev Containers、金丝雀令牌、包安全工具、短期凭证——比幻想手动审计每个依赖更现实。

## 你只有几分钟

当金丝雀触发——或 GitHub 提示令牌从异常IP被使用时——你有窗口期。几分钟，或许几小时。不是一周。

- **先轮换，后调查。** 在理解发生了什么前先撤销令牌。
- **检查攻击者持久性。** 查看攻击者离开前创建的新 OAuth 应用、IAM 用户、部署密钥、API 令牌。
- **终止活跃浏览器会话。** 强制登出所有重要系统。
- **通知他人。** 安全事件需要见证人和时间戳才能改善。

安全行业谈论很多检测，却很少讨论检测后二十分钟你在办公桌前独自挣扎时该怎么办——试图回忆自己有哪些服务的令牌。

这份清单应在警报触发前就存在。

## 值得拥有的标准

标准不是"永远不要点击任何可疑内容"。这是海报标语，不是系统。

恶意依赖不应能访问其他项目的云凭证。提示注入的文档不应能将代理重定向到你的家目录。信息窃取者不应能在不触发警报的情况下找到明文备份和长期令牌。被盗凭证应在完全接管前过期、失败 MFA 或触发金丝雀。

当不再要求人类完美，而是让妥协变得不那么有利可图时，安全性会提高。

你的笔记本电脑现在已经是生产环境的一部分。为它设置枯燥但可靠的边界，既能拦截闯入者——也能拦截你自己不小心放进来的人。
