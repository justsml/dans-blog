## אתה מחסן האישורים

הלפטופ שלך אינו לפטופ. הוא מחסן אישורים עם מקלדת – הפעלות דפדפן, מפתחות SSH, קבצי `.env`, טוקנים של GitHub, כלי CLI של ענן, כלי תכנות בינה מלאכותית עם גישה למעטפת, ייצואי מסדי נתונים ששכחת שקיימים.

המודל הישן היה: הייצור מסוכן, המקומי נוח. המודל הזה גמור.

<p class="inset">
השאלה איננה האם תוכל להימנע מכל קליק רע. השאלה היא האם קליק רע אחד יכול לקרוא הכול, להשתמש בכול, ולעזוב לפני שתשים לב.
</p>

מפתח נתקל במשהו שנראה מספיק נורמלי: PDF מקבלן, CAPTCHA מזויפת שמבקשת להדביק משהו בטרמינל, חבילה עם סקריפט `postinstall`, הפעלת תכנות בינה מלאכותית שהגיעה רחוק יותר למערכת הקבצים ממה שהמשימה דרשה. נתיבים מסוימים מתקינים תוכנה זדונית. אחרים גונבים אישורים. חלקם לא זקוקים לניצול מקומי — המשתמש מריץ בעצמו את הפקודה של התוקף.

זהו משטח התקיפה המודרני. לפעמים אתה הוא הפרצה.

## בעיית שרשרת האספקה היא גדולה באופן בלתי אפשרי

הנה החלק המצחיק. כדי להיות בטוח לחלוטין, כל מה שצריך לעשות הוא לבצע הערכת אבטחה מעמיקה ורב־פלטפורמית של כל תלות שאתה מסתמך עליה — המתחזקים שלה, ההיסטוריה שלה, התלות הטרנזיטיבית שלה — על פני כל רשם חבילות. ואז לחזור על ההערכה בכל פעם שעץ התלות משתנה או מקבל עדכון, משום שזה בדיוק איך פועלות התקפות שרשרת אספקה: הן מנצלות שרשרת אמון.

קל.

אה, והתוקף צריך להצליח רק פעם אחת. אתה צריך לשמור על הגנה מושלמת בכל פעם.

Lumma Stealer — גנב מידע נפוץ שאוסף בשקט סיסמאות, עוגיות דפדפן, מפתחות API ואישורי ענן — הגיע לקורבנות דרך CAPTCHAs מזויפות, מודעות חיפוש מורעלות ואפליקציות שעברו טרויאניזציה. חקירת Snowflake של Mandiant איתרה שרשרת של פרצות ארגוניות חזרה לאישורים שנגנבו על ידי גנבי מידע, חלקם עוד מ-2020. לפחות 79.7% מהחשבונות ששימשו בהתקפה היו בעלי חשיפה קודמת ידועה. המנעולים מעולם לא הוחלפו.

התוקף לא פרץ למחסן. הוא מצא מפתחות ישנים במגירת שולחן.

עבור מפתחים, המגירה הזו נראית כך:

| פריט מקומי | למה לתוקפים אכפת |
| --- | --- |
| עוגיות דפדפן | יכולות לעקוף כניסה, ולפעמים לדלג על אימות רב-שלבי. |
| קבצי `.env` | מפתחות API, כתובות URL לבסיסי נתונים, סודות JWT. |
| תצורת CLI לענן | הופך פגיעה במחשב מקומי לגישה מלאה לתשתית. |
| מפתחות SSH | עדיין בכל מקום, עדיין חזקים, עדיין מועתקים בין מכונות. |
| אסימוני מנהל חבילות | אסימון פרסום npm או PyPI שלך הוא גישה לשרשרת האספקה. |
| דאמפים של בסיסי נתונים | פחות מוגנים מהייצור, לרוב שלמים יותר. |
| הקשר קוד בינה מלאכותית | העוזר אולי קיבל קבצים רגישים "בשביל הקשר". |

ואז יש גיבויים — ייצואי ייצור שמישהו זרק לתיקיית `~/Downloads` ושכח. גיבוי לא בטוח יותר רק כי הוא לא פעיל. הוא פשוט ייצור בלי מערכת אזעקה.

## הפתרון ה"תהיה זהיר" — לא פתרון

"תהיה זהיר" זו עצה חלשה. היא דורשת מהאדם להיות הגבול.

בני אדם אינם גבולות. בני אדם הם תנועה.

גבולות הם משעממים: בידוד מערכת קבצים, סודות מוצפנים במנוחה, אישורים קצרי טווח, אימות מבוסס חומרה, והתראות שנשלחות ברגע שנוגעים בסוד מזויף.

אם תהליך זדוני רץ, השאלות שקובעות אם יהיה לך אחר צהריים רע או אירוע בכל החברה הן:
1. מה התהליך הזה יכול **לקרוא**?
2. באילו אישורים הוא יכול **להשתמש**?
3. לאן הוא יכול **לשלוח מידע**?

## המהלכים בעלי ההשפעה הגבוהה ביותר, ממש עכשיו

### Dev Containers — כברירת מחדל

[Development Containers](https://github.com/devcontainers/spec) הם השינוי היחיד בעל ההשפעה הגבוהה ביותר שרוב הצוותים לא עושים. Dev Container מריץ עבודת פרויקט בתוך קונטיינר Docker מבודד. `npm install`, `pip install`, סקריפטים של `postinstall`, פקודות מעטפת של AI, הרחבות VS Code — כל זה קורה בתוך 'סביבת עבודה' או קונטיינר שאינו יכול לראות את שאר המחשב שלך.

<p class="inset">בקשו מ-Claude Code להגדיר DevContainers בכל פרויקט.</p>

הרכיבו את הריפו. כללו רק את הסודות הדרושים לאותו פרויקט. אל תרכיבו את `~/.ssh`, `~/.aws` או את תיקיית הבית שלכם מתוך נוחות. הוראה שהוחדרה לפרומפט יכולה להגיע רק למה שהסוכן יכול להגיע — תהפכו את זה למשעמם.

```jsonc
// .devcontainer/devcontainer.json
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ]
}
```

### אסימוני קנרית — פריסה אגרסיבית

[Canarytokens](https://canarytokens.org) הם מוקשי דיגיטל חינמיים. תטמעו סוד מזויף-אך-משכנע במקום שתוקף יחפש. ברגע שנוגעים בו, תקבלו התראה — לעיתים תוך שניות. דמיינו ששתלתם שקית צבע נסתר בערימת שטרות מזויפים.

תוקפים עושים מיפוי לפני שהם גונבים. המעבר הסיור הזה הוא החלון שלכם.

פזרו קנריות בקבצים הכי מפתים שלכם:

```text
~/.aws/credentials          ← הוסף פרופיל מזויף [billing-prod-legacy] עם מפתח קנרית
~/backups/customer-export-2024.sql   ← כתובת קנרית בתוך הקובץ
~/.env.canary               ← פרטי גישה מזויפים בכל ריפו
```

אסימוני קנרית זמינים בחינם ב‑[canarytokens.org](https://canarytokens.org), ניתנים לאירוח עצמי, וקיימים גם כ‑SaaS בתשלום דרך [Thinkst Canary](https://canary.tools). אין שום סיבה טובה לא לפרוס אותם בכל מקום שגנב יסתכל.

### כלי אבטחת חבילות

כלים כמו [Socket.dev](https://socket.dev), [Snyk](https://snyk.io) ו‑[Wiz](https://wiz.io) הם לרוב הראשונים שמגלים וחוסמים התקפות שרשרת אספקה בזמן אמת. הם מנטרים את מאגרי החבילות שאתם לא יכולים לעקוב אחריהם בעצמכם. עבור צוותים שאינם יכולים להרשות לעצמם תוכנית אבטחה במשרה מלאה, אלו מערכות התרעה מוקדמת בעלות מינוף גבוה.

### הגדרות גיל מינימלי ב‑PNPM

אם אתם משתמשים ב‑PNPM, הגדירו גיל מינימלי לשחרור. חבילות שפורסמו לאחרונה הן חלון הסיכון הגבוה ביותר להתקפות שרשרת אספקה — חבילה שקיימת פחות מ‑24 שעות כמעט ולא עברה ביקורת קהילתית. הגדירו `minimumReleaseAge` בדקות: לפחות `1440` (יום אחד), ובאופן אידיאלי `2880` (יומיים).

```yaml
minimumReleaseAge: 2880
minimumReleaseAgeStrict: true
minimumReleaseAgeIgnoreMissingTime: false
minimumReleaseAgeExclude:
  - 'typescript'
```

הגדרה זו חוסמת התקפות רבות שמבוססות על חבילות שפורסמו זה עתה, במיוחד כאלו שמתגלות ונמשכות לפני ההתקנה הבאה שלכם. השתמשו ב‑`minimumReleaseAgeExclude` במשורה, עבור חבילות שבהן עדכונים מיידיים חשובים יותר מהעיכוב, כמו קומפיילר או תלות ריצה שאתם עוקבים אחריה באופן פעיל.

### עבור הסביבות הקריטיות ביותר מבחינת אבטחה

סוכנויות מודיעין, רשויות אכיפת חוק, תשתיות מסחר פיננסי, רשומות בריאות — סביבות אלו מאמצות לעיתים תהליך קפדני של הערכה ואישור חבילות. זה נשמע בטוח. המחיר הוא חמור: עץ התלות שלכם מתאבן לאט לתוכנה מיושנת.

הזמן אינו ניטרלי כאן. גרסאות ישנות צוברות CVEs ידועות. תוקפים חוקרים גרסאות מתוקנות כדי למצוא מופעים שלא תוקנו. ו"עדיף שטן שאתה מכיר" אינו הישועה שקיוויתם לה — הוא רק אומר לכם אילו פרצות התוקף הספיק להכיר הכי טוב.

רשימות לבנות קפדניות עובדות אם יש לכם כוח אדם לתחזק אותן. לרוב הצוותים אין. עבור כולם, הגישה השכבתית — מכולות פיתוח, אסימוני קנריה, כלי אבטחת חבילות, פרטי גישה קצרי מועד — מספקת הגנה ריאלית יותר מאשר להעמיד פנים שאתם יכולים לבקר כל תלות ידנית.

## יש לכם דקות

כשקנרית מופעלת — או שגיטהאב מתריעה שטוקן שימש מכתובת IP לא צפויה — יש לכם חלון. דקות, אולי כמה שעות. לא שבוע.

- **סובבו תחילה, חקרו אחר כך.** בטלו טוקנים לפני שאתם מבינים מה קרה.
- **בדקו נוכחות מתמשכת של תוקף.** אפליקציות OAuth חדשות, משתמשי IAM, מפתחי פריסה, טוקני API שנוצרו לפני שהתוקף עזב.
- **הרגו הפעלות דפדפן פעילות.** בצעו התנתקות כפויה מכל מה שחשוב לכם.
- **ספרו למישהו.** אירועי אבטחה משתפרים עם עדים וחותמות זמן.

תעשיית האבטחה מדברת הרבה על זיהוי. היא מדברת פחות על מה שקורה בעשרים הדקות אחרי הזיהוי, כשאתם לבד ליד השולחן מנסים להיזכר באילו שירותים יש לכם טוקנים.

הרשימה הזו צריכה להתקיים לפני שההתראה נשלחת.

## הסטנדרט הראוי

הסטנדרט הוא לא "לעולם אל תלחץ על שום דבר מוזר." זו עצה לפוסטר, לא למערכת.

תלות גרועה לא אמורה להיות מסוגלת להגיע לפרטי גישה בענן מפרויקטים אחרים. מסמך שהוחדרה לו פרומפט לא אמור להפנות סוכן לתיקיית הבית שלכם. גנב מידע לא אמור למצוא גיבויים בטקסט רגיל וטוקנים ארוכי טווח מבלי להפעיל אזעקה. פרטי גישה גנובים אמורים לפקוע, להיכשל ב‑MFA, או לפגוע בקנרית לפני שהם הופכים להשתלטות מלאה.

Security improves when we stop asking humans to be perfect and start making compromise less profitable.

האבטחה משתפרת כשמפסיקים לדרוש מבני אדם להיות מושלמים ומתחילים להפוך פשרה לפחות משתלמת.

הלפטופ שלך הוא חלק מהייצור עכשיו. תן לו גבולות משעממים שיתפסו גם את התוקף שפרץ פנימה — וגם את זה שאתה בעצם הכנסת בטעות.
