## 🚀 Введение

Извлечение URL из необработанного текста иногда напоминает утомительную игру в «Whack-a-Mole». Знаки препинания, скобки и неоднозначное форматирование — всё это сговаривается, чтобы усложнить задачу. Создаёте ли вы веб-скрапер, анализатор данных или чат-приложение — точное извлечение URL необходимо.

В этой статье мы решим проблему напрямую, используя гибкий двухэтапный подход. Наша цель — **сначала захватить все _потенциальные_ строки, похожие на URL**, а затем выполнить валидацию на следующем этапе.

> 💡 **Примечание:** Этот шаблон не предназначен для **_валидации_** URL! Он намеренно допускает знаки препинания и опечатки.

## 🔍 Цель: Извлечение URL из текста

При извлечении URL из необработанного текста эффективен двухэтапный подход:

1. **Захватить всё, похожее на URL**: Закинуть широкую сеть, чтобы собрать все строки, которые *могут* быть URL. Здесь и блистает наше «регулярное выражение на 120+ байт».
2. **Валидация**: После захвата кандидатов используйте дополнительные проверки (например, разрешение DNS, сравнение с известными доменами), чтобы отсеять недействительные записи.

## Регулярное выражение на 120+ байт

Ниже приведено компактное регулярное выражение, предназначенное для извлечения и разбора URL за один проход. Оно поддерживает различные протоколы, домены, пути и опциональные секции запроса/фрагмента.

Не волнуйтесь — мы разберём его по шагам!

```js title="120+ Byte URL Regex" frame="code"
const urlRegex = /([-.a-z0-9]+:\/{1,3})([^-\/\.[\](|)\s?][^`\/\s\]?]+)([-_a-z0-9!@$%^&*()=+;/~\.]*)[?]?([^#\s`?]*)[#]?([^#\s'"`\.,!]*)/gi;
// Compatibility: ES5+

// Same pattern, split on newlines for readability:
([-.a-z0-9]+:\/{1,3})
([^-\/\.[\](|)\s?][^`\/\s\]?]+)
([-_a-z0-9!@$%^&*()=+;/~\.]*)
[?]?([^#\s`?]*)
[#]?([^#\s'"`\.,!]*)

```

## 🧩 Разбираем по шагам

Давайте разберём регулярное выражение на составляющие, чтобы понять, как оно работает:

<h3>1. Протокол (Группа 1): <code>{`([-.a-z0-9]+:\/{1,3})`}</code></h3>

<ul>
  <li>**Назначение:** Соответствует части URL, отвечающей за протокол (например, `http://`, `ftp://`, `custom-scheme://`).</li>
  <li>
    **Пояснение:**
    <ul>
      <li><code>[-.a-z0-9]+</code>: Один или более символов: строчные буквы, цифры, дефисы или точки (типично для схем протоколов).</li>
      <li><code>{`:\/{1,3}`}</code>: Двоеточие, за которым следует от одного до трёх слешей (<code>:/</code>, <code>://</code> или <code>:///</code>).</li>
    </ul>
  </li>
</ul>

<h3>2. Домен (Группа 2): <code>{`([^-\/\.[\](|)\s?][^\`\/\s\]?]+)`}</code></h3>

<ul>
  <li>**Назначение:** Захватывает доменную часть или хост URL.</li>
  <li>
    **Пояснение:**
    <ul>
      <li><code>[^-\/\.[\](|)\s?]</code>: Любой символ, кроме указанных специальных символов и пробелов.</li>
      <li><code>[^`\/\s\]?]+</code>: Один или более символов, кроме обратных кавычек, слешей, пробелов или закрывающих квадратных скобок.</li>
    </ul>
  </li>
</ul>

<h3>3. Путь (Группа 3): <code>{`([-_a-z0-9!@$%^&*()=+;/~\\.]*)`}</code></h3>

<ul>
  <li>**Назначение:** Соответствует компоненту пути URL.</li>
  <li>
    **Пояснение:**
    <ul>
      <li><code>[-_a-z0-9!@$%^&*()=+;/~\.]*</code>: Соответствует нулю или более URL-безопасным символам, обычно встречающимся в путях.</li>
    </ul>
  </li>
</ul>

<h3>4. Запрос (Группа 4): <code>[?]?([^#\s`?]*)</code></h3>

<ul>
  <li>**Назначение:** Опционально соответствует строке запроса, начинающейся с любого символа <code>?</code>.</li>
  <li>
    **Пояснение:**
    <ul>
      <li><code>[?]?</code>: Опционально соответствует <code>?</code>. (Квадратные скобки не строго обязательны, однако они немного яснее, чем ультра-краткое двойное <code>??</code>. Также это создаёт визуальную параллель с (похожей) следующей группой <code>[#]?</code>.)</li>
      <li><code>([^#\s`?]*)</code>: Соответствует нулю или более символов, которые не являются решёткой, пробелом, обратным апострофом или вопросительным знаком.</li>
    </ul>
  </li>
</ul>

<h3>5. Фрагмент (Группа 5): <code>[#]?([^#\s'"`\.,!]*)</code></h3>

<ul>
  <li>**Назначение:** Опционально соответствует идентификатору фрагмента, начинающемуся с <code>#</code>.</li>
  <li>
    **Пояснение:**
    <ul>
      <li><code>[#]?</code>: Опционально соответствует <code>#</code>.</li>
      <li><code>([^#\s'"`\.,!]*)</code>: Соответствует нулю или более символов, которые не являются запрещёнными знаками препинания или пробелами.</li>
    </ul>
  </li>
</ul>

```js title="extract-urls.js" frame="code"
const text = `
Check this out: https://example.com/path?query=123#section
And also (ftp://files.server.org/index).
Plus a weird one: custom-scheme://host/param;weird^stuff
`;

const urlRegex =
  /([-.a-z0-9]+:\/{1,3})([^-\/\.[\](|)\s?][^`\/\s\]?]+)([-_a-z0-9!@$%^&*()=+;/~\.]*)[?]?([^#\s`?]*)[#]?([^#\s'"`\.,!]*)/gi;

const matches = [
  ...text.matchAll(urlRegex),
].map((match) => match[0]);
console.log("Extracted URLs:", matches);

const parts = [
  ...text.matchAll(urlRegex),
].map((match) => match.slice(1));
console.log("Extracted Parts:", parts);
```

## ☑️ Дальнейшие шаги

В зависимости от вашего сценария использования может потребоваться доработать это регулярное выражение или добавить дополнительные шаги валидации и постобработки.

### Разные проекты — разные потребности

У проектов разные требования и соображения безопасности:

1. **Веб-скрапинг**: Проверяйте URL на доступность и надёжность.
2. **Обработка данных**: Извлекайте URL из пользовательского контента, обеспечивая безопасность.
3. **Анализ данных**: Отфильтровывайте дубликаты или нерелевантные ссылки для исследований или маркетинга.
4. **Пользовательские приложения**: Автоматически превращайте URL в гиперссылки в чатах или на форумах.

### Постобработка и валидация

После сбора потенциальных URL применяйте дополнительные проверки:

- **DNS-запрос**: Убедитесь, что домены разрешаются.
- **Проверка безопасности**: Используйте сервисы для выявления вредоносных или фишинговых сайтов.
- **Пользовательские правила**: Применяйте фильтры, специфичные для проекта (например, разрешённые TLD, максимальная длина URL).
