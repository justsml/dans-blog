## 🚀 Введение

Извлечение URL из необработанного текста иногда напоминает утомительную игру «вбить крота». Пунктуация, скобочные обёртки и неоднозначное форматирование постоянно мешают вам. Независимо от того, пишете ли вы веб‑скрейпер, аналитический модуль или чат‑приложение, точное извлечение URL критично.

В этом посте мы решим задачу напрямую, используя гибкий двухэтапный подход. Наша цель — **сначала захватить все _потенциальные_ строкоподобные URL**, а затем выполнить проверку на этапе последующей обработки.

> 💡 **Примечание:** Этот шаблон **не предназначен для _валидации_** URL! Он намеренно допускает пунктуацию и ошибки в написании.

## 🔍 Цель: извлекать URL из текста

При извлечении URL из необработанного текста эффективен двухэтапный метод:

1. **Захватить всё, что выглядит как URL**: Бросьте широкую сеть, чтобы собрать все строки, которые *могут* быть URL. Здесь в дело вступает наш «регулярное выражение 120+ байт».
2. **Проверка**: После того как вы собрали эти кандидаты, выполните вторичные проверки (например, разрешение DNS, сравнение с известными доменами), чтобы отсеять недействительные записи.

Термины `extract` и `parse` часто употребляются как синонимы, однако они обозначают разные процессы. Извлечение URL — это поиск и захват потенциальных URL в большом тексте. Парсинг, в свою очередь, подразумевает разбор этих URL на отдельные компоненты.

## Регулярное выражение в 120 + байт

Ниже представлен компактный регекс, предназначенный для одновременного извлечения и парсинга URL. Он поддерживает различные протоколы, домены, пути и опциональные секции запроса/фрагмента.

Не волнуйтесь — разберём всё пошагово!

```js title="120+ Byte URL Regex" frame="code"
const urlRegex = /([-.a-z0-9]+:\/{1,3})([^-\/\.[\](|)\s?][^`\/\s\]?]+)([-_a-z0-9!@$%^&*()=+;/~\.]*)[?]?([^#\s`?]*)[#]?([^#\s'"`\.,!]*)/gi;
// Compatibility: ES5+

// Same pattern, split on newlines for readability:
([-.a-z0-9]+:\/{1,3})
([^-\/\.[\](|)\s?][^`\/\s\]?]+)
([-_a-z0-9!@$%^&*()=+;/~\.]*)
[?]?([^#\s`?]*)
[#]?([^#\s'"`\.,!]*)

```

## 🧩 Разбираем по шагам

Разберём регулярное выражение по компонентам, чтобы понять, как оно работает:

<h3>1. Протокол (Группа 1): <code>{`([-.a-z0-9]+:\/{1,3})`}</code></h3>

<ul>
  <li>**Назначение:** Совпадает с частью URL, отвечающей за протокол (например, `http://`, `ftp://`, `custom-scheme://`).</li>
  <li>
    **Объяснение:**
    <ul>
      <li><code>[-.a-z0-9]+</code>: Совпадает с одной или более строчными буквами, цифрами, дефисами или точками (часто встречающимися в схемах протокола).</li>
      <li><code>{`:\/{1,3}`}</code>: Совпадает с двоеточием, за которым следует от одной до трёх косых черт (<code>:/</code>, <code>://</code> или <code>:///</code>).</li>
    </ul>
  </li>
</ul>

<h3>2. Домен (Группа 2): <code>{`([^-\/\.[\](|)\s?][^\`\/\s\]?]+)`}</code></h3>

<ul>
  <li>**Назначение:** Захватывает часть URL, содержащую домен или хост.</li>
  <li>
    **Объяснение:**
    <ul>
      <li><code>[^-\/\.[\](|)\s?]</code>: Совпадает с любым символом, кроме указанных специальных символов и пробельных символов.</li>
      <li><code>[^`\/\s\]?]+</code>: Совпадает с одной или более последовательностью символов, исключая обратные кавычки, косые черты, пробелы и закрывающие квадратные скобки.</li>
    </ul>
  </li>
</ul>

<h3>3. Путь (Группа 3): <code>{`([-_a-z0-9!@$%^&*()=+;/~\\.]*)`}</code></h3>

<ul>
  <li>**Назначение:** Совпадает с компонентом пути URL.</li>
  <li>
    **Объяснение:**
    <ul>
      <li><code>[-_a-z0-9!@$%^&*()=+;/~\.]*</code>: Совпадает с нулём или более символов, безопасных для URL и обычно встречающихся в путях.</li>
    </ul>
  </li>
</ul>

<h3>4. Query (Группа 4): <code>[?]?([^#\s`?]*)</code></h3>

<ul>
  <li>**Назначение:** Необязательно совпадает со строкой запроса, начинающейся с любого символа <code>?</code>.</li>
  <li>
    **Объяснение:**
    <ul>
      <li><code>[?]?</code>: Необязательно совпадает с <code>?</code>. (Квадратные скобки не обязательны, но они чуть более наглядны, чем ультра‑краткое двойное <code>??</code>. Кроме того, они создают визуальный параллелизм с следующей группой <code>[#]?</code>.)</li>
      <li><code>([^#\s`?]*)</code>: Совпадает с нулём или более символов, которые не являются решёткой, пробельным символом, обратной кавычкой или вопросительным знаком.</li>
    </ul>
  </li>
</ul>

<h3>5. Fragment (Группа 5): <code>[#]?([^#\s'"`\.,!]*)</code></h3>

<ul>
  <li>**Назначение:** Необязательно совпадает с идентификатором фрагмента, начинающимся с <code>#</code>.</li>
  <li>
    **Объяснение:**
    <ul>
      <li><code>[#]?</code>: Необязательно совпадает с <code>#</code>.</li>
      <li><code>([^#\s'"`\.,!]*)</code>: Совпадает с нулём или более символов, которые не являются запрещённой пунктуацией или пробельными символами.</li>
    </ul>
  </li>
</ul>

```js title="extract-urls.js" frame="code"
const text = `
Check this out: https://example.com/path?query=123#section
And also (ftp://files.server.org/index).
Plus a weird one: custom-scheme://host/param;weird^stuff
`;

const urlRegex =
  /([-.a-z0-9]+:\/{1,3})([^-\/\.[\](|)\s?][^`\/\s\]?]+)([-_a-z0-9!@$%^&*()=+;/~\.]*)[?]?([^#\s`?]*)[#]?([^#\s'"`\.,!]*)/gi;

const matches = [
  ...text.matchAll(urlRegex),
].map((match) => match[0]);
console.log("Extracted URLs:", matches);

const parts = [
  ...text.matchAll(urlRegex),
].map((match) => match.slice(1));
console.log("Extracted Parts:", parts);
```

## ☑️ Следующие шаги

В зависимости от задачи, возможно придётся уточнить этот регекс или добавить дополнительные проверки и пост‑обработку.

### Разные проекты — разные требования

У проектов различаются требования и соображения безопасности:

1. **Веб‑скрейпинг**: проверять URL‑ы, чтобы убедиться, что они доступны и надёжны.  
2. **Обработка данных**: извлекать URL‑ы из пользовательского контента, при этом соблюдая безопасность.  
3. **Анализ данных**: отфильтровывать дубликаты или нерелевантные ссылки для исследований или маркетинга.  
4. **Приложения с пользовательским интерфейсом**: автоматически превращать URL‑ы в гиперссылки в чат‑приложениях или форумах.  

### Пост‑обработка и проверка

После сбора потенциальных URL‑ов выполните дополнительные проверки:

- **DNS‑запрос**: убедитесь, что домены резолвятся.  
- **Проверки безопасности**: используйте сервисы для обнаружения вредоносных или фишинговых сайтов.  
- **Пользовательские правила**: применяйте фильтры, специфичные для проекта (например, разрешённые доменные зоны, максимальная длина URL).
