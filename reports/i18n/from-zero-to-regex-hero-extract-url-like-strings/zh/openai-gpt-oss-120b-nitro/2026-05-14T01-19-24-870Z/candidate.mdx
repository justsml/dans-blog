## 🚀 介绍

从原始文本中提取 URL 有时会像玩一场乏味的打地鼠游戏。标点、括号包装以及模糊的格式都会让人抓狂。无论你是在构建网页爬虫、数据分析器，还是聊天应用，准确提取 URL 都是必不可少的。

本文将采用灵活的两步法直接解决这个问题。我们的目标是 **先捕获所有 _潜在_ 类 URL 字符串**，随后在后续流程中进行验证。

> 💡 **注意：** 该模式并非用于 **_验证_** URL！它对标点和拼写错误故意保持宽松。

## 🔍 目标：从文本中提取 URL

在从原始文本中提取 URL 时，两步法是有效的：

1. **捕获所有类似 URL 的字符串**：先抛出宽网，抓取所有*可能*是 URL 的文本。这正是我们的“120+ 字节正则”发挥作用的地方。  
2. **验证**：捕获到候选后，使用二次检查（例如 DNS 解析、与已知域名对比）剔除无效条目。

### 可视化挑战

`extract` 与 `parse` 常被混用，但它们指的是不同的过程。**提取 URL** 是在大块文本中识别并捕获潜在的 URL；**解析 URL** 则是把这些 URL 拆解成各个组成部分。

## 120+ 字节正则

下面是一段简洁的正则，用于一次性提取并解析 URL。它支持多种协议、域名、路径以及可选的查询/片段部分。

别担心——我们会一步步拆解它！

```js title="120+ Byte URL Regex" frame="code"
const urlRegex = /([-.a-z0-9]+:\/{1,3})([^-\/\.[\](|)\s?][^`\/\s\]?]+)([-_a-z0-9!@$%^&*()=+;/~\.]*)[?]?([^#\s`?]*)[#]?([^#\s'"`\.,!]*)/gi;
// Compatibility: ES5+
```

```
// 同样的模式，为了可读性在换行处拆分：
([-.a-z0-9]+:\/{1,3})
([^-\/\.[\](|)\s?][^`\/\s\]?]+)
([-_a-z0-9!@$%^&*()=+;/~\.]*)
[?]?([^#\s`?]*)
[#]?([^#\s'"`\.,!]*)

```

## 🧩 逐步拆解

让我们把正则拆成各个子表达式，弄清它的工作原理：

<h3>1. 协议（第 1 组）：<code>{`([-.a-z0-9]+:\/{1,3})`}</code></h3>

<ul>
  <li>**目的：** 匹配 URL 的协议部分（例如 <code>http://</code>、<code>ftp://</code>、<code>custom-scheme://</code>）。</li>
  <li>
    **解释：**
    <ul>
      <li><code>[-.a-z0-9]+</code>：匹配一个或多个小写字母、数字、连字符或点（协议名中常见的字符）。</li>
      <li><code>{`:\/{1,3}`}</code>：匹配冒号后跟 1 到 3 个斜杠（<code>:/</code>、<code>://</code> 或 <code>:///</code>）。</li>
    </ul>
  </li>
</ul>

<h3>2. 域名（第 2 组）：<code>{`([^-\/\.[\](|)\s?][^\`\/\s\]?]+)`}</code></h3>

<ul>
  <li>**目的：** 捕获 URL 的域名或主机部分。</li>
  <li>
    **解释：**
    <ul>
      <li><code>[^-\/\.[\](|)\s?]</code>：匹配除列出的特殊字符和空白之外的任意字符。</li>
      <li><code>[^`\/\s\]?]+</code>：匹配一个或多个字符，排除反引号、斜杠、空白或右方括号。</li>
    </ul>
  </li>
</ul>

<h3>3. 路径（第 3 组）：<code>{`([-_a-z0-9!@$%^&*()=+;/~\\.]*)`}</code></h3>

<ul>
  <li>**目的：** 匹配 URL 的路径部分。</li>
  <li>
    **解释：**
    <ul>
      <li><code>[-_a-z0-9!@$%^&*()=+;/~\.]*</code>：匹配零个或多个在路径中常见的 URL 安全字符。</li>
    </ul>
  </li>
</ul>

<h3>4. 查询（第 4 组）：<code>[?]?([^#\s`?]*)</code></h3>

<ul>
  <li>**目的：** 可选地匹配查询字符串，以任意 <code>?</code> 开头。</li>
  <li>
    **解释：**
    <ul>
      <li><code>[?]?</code>：可选匹配一个 <code>?</code>。（方括号并非必需，但比超简写的双 <code>??</code> 更直观，也与下一个匹配组 <code>[#]?</code> 在视觉上形成对应。）</li>
      <li><code>([^#\s`?]*)</code>：匹配零个或多个既不是井号、空白、反引号也不是问号的字符。</li>
    </ul>
  </li>
</ul>

<h3>5. 片段（第 5 组）：<code>[#]?([^#\s'"`\.,!]*)</code></h3>

<ul>
  <li>**目的：** 可选地匹配以 <code>#</code> 开头的片段标识符。</li>
  <li>
    **解释：**
    <ul>
      <li><code>[#]?</code>：可选匹配一个 <code>#</code>。</li>
      <li><code>([^#\s'"`\.,!]*)</code>：匹配零个或多个既不是禁止的标点也不是空白的字符。</li>
    </ul>
  </li>
</ul>

## 🛠️ 解析示例

下面展示如何在 JavaScript 中使用这段“怪兽”正则：

```js title="extract-urls.js" frame="code"
const text = `
Check this out: https://example.com/path?query=123#section
And also (ftp://files.server.org/index).
Plus a weird one: custom-scheme://host/param;weird^stuff
`;

const urlRegex =
  /([-.a-z0-9]+:\/{1,3})([^-\/\.[\](|)\s?][^`\/\s\]?]+)([-_a-z0-9!@$%^&*()=+;/~\.]*)[?]?([^#\s`?]*)[#]?([^#\s'"`\.,!]*)/gi;

const matches = [
  ...text.matchAll(urlRegex),
].map((match) => match[0]);
console.log("Extracted URLs:", matches);

const parts = [
  ...text.matchAll(urlRegex),
].map((match) => match.slice(1));
console.log("Extracted Parts:", parts);
```

```json title="extracted-urls.json"
]
```

```json title="urls-parts.json"
  [

    "/param;weird^stuff", // Path
    "",                   // Query

]
```

## ☑️ 后续步骤

根据具体使用场景，你可能需要进一步细化正则，或加入额外的校验与后处理步骤。

### 不同项目，不同需求

项目的要求和安全关注点各不相同：

1. **网页抓取**：验证 URL，确保其可访问且可信。
2. **数据处理**：从用户生成的内容中提取 URL，同时保证安全。
3. **数据分析**：过滤掉重复或无关的链接，用于研究或营销目的。
4. **面向用户的应用**：在聊天应用或论坛中自动为 URL 添加超链接。

### 后处理与校验

收集到潜在 URL 后，执行额外检查：

- **DNS 查询**：确认域名能够解析。
- **安全检查**：使用服务检测是否为恶意或钓鱼站点。
- **自定义规则**：应用项目特定的过滤器（例如，允许的顶级域名、最大 URL 长度）。

## 📝 总结

提取半结构化字符串数据可能是正则掌握中最让人满足的环节。

以下是关键要点回顾：

- **使用可视化工具编写、测试并理解你的[正则表达式模式](https://regex101.com/r/jO8bC4/69)。**
- **将挑战拆解为多个部分**，分别解决。可以把捕获组视作正则的“路标”。  
- **使用‘宽松’匹配表达式，避免在数据摄取时严格遵循规范。**
- **在初始提取后进行校验步骤**至关重要——始终考虑项目的安全性和具体需求。

通过遵循这些步骤，你可以高效地提取任意半结构化字符串数据，为后续的处理和校验奠定基础。
