## 🚀 מבוא

חילוץ כתובות URL מטקסט גולמי יכול לפעמים להרגיש כמו משחק מתיש של "מכה חפרפרת". סימני פיסוק, עטיפות בסוגריים ועיצוב דו-משמעי כולם מתאגדים יחד כדי לסכל את מאמציך. בין אם אתה בונה סורק רשת, מנתח נתונים או יישום צ'אט, חילוץ מדויק של כתובות URL הוא חיוני.

בפוסט זה, נתמודד עם הבעיה חזיתית באמצעות גישה גמישה דו-שלבית. המטרה שלנו היא **לתפוס תחילה את כל המחרוזות _הפוטנציאליות_ דמויות URL** ולטפל באימות בתהליך נפרד לאחר מכן.

> 💡 **הערה:** תבנית זו אינה מיועדת **_לאימות_** כתובות URL! היא מתירנית בכוונה לגבי סימני פיסוק ואיות שגוי.

## 🔍 מטרה: חילוץ כתובות URL מטקסט

בעת חילוץ כתובות URL מטקסט גולמי, גישה דו-שלבית היא יעילה:

1. **לכידת כל מה שנראה כמו URL**: הטל רשת רחבה כדי לתפוס את כל המחרוזות ש*עשויות* להיות URLs. כאן הביטוי הרגולרי "120+ בתים" שלנו זורח.
2. **אימות**: לאחר שלכדתם מועמדים אלה, השתמשו בבדיקות משניות (למשל, רזולוציית DNS, השוואה מול דומיינים ידועים) כדי לסנן ערכים לא תקינים.

### הדמיית האתגר

מונחים כמו `extract` ו-`parse` משמשים לעתים קרובות לסירוגין, אך הם מתייחסים לתהליכים נפרדים. חילוץ URLs כולל זיהוי ולכידה של URLs פוטנציאליים מתוך גוף טקסט גדול יותר. לעומת זאת, ניתוח (parsing) כולל פירוק URLs אלה לחלקיהם המרכיבים.

## הביטוי הרגולרי באורך 120+ בתים

להלן ביטוי רגולרי תמציתי שנועד לחלץ ולנתח URLs בבת אחת. הוא תומך במגוון פרוטוקולים, דומיינים, נתיבים, וקטעי שאילתה/פרגמנט אופציונליים.

אל דאגה — נפרק אותו צעד אחר צעד!

```js title="ביטוי רגולרי לאיתור URL באורך 120+ בתים" frame="code"
const urlRegex = /([-.a-z0-9]+:\/{1,3})([^-\/\.[\](|)\s?][^`\/\s\]?]+)([-_a-z0-9!@$%^&*()=+;/~\.]*)[?]?([^#\s`?]*)[#]?([^#\s'"`\.,!]*)/gi;
// Compatibility: ES5+
```

// Same pattern, split on newlines for readability:
([-.a-z0-9]+:\/{1,3})
([^-\/\.[\](|)\s?][^`\/\s\]?]+)
([-_a-z0-9!@$%^&*()=+;/~\.]*)
[?]?([^#\s`?]*)
[#]?([^#\s'"`\.,!]*)

```

<blockquote class="inset">שתפו את הביטויים הרגולריים הכי פרועים שנתקלתם בהם (או שכתבתם) בתגובות למטה! 🚀</blockquote>

## 🧩 פירוק שלב אחר שלב

בואו נפרק את הביטוי הרגולרי למרכיביו כדי להבין איך הוא עובד:

<h3>1. פרוטוקול (קבוצה 1): <code>{`([-.a-z0-9]+:\/{1,3})`}</code></h3>

<ul>
  <li>**מטרה:** תואם את חלק הפרוטוקול של ה-URL (למשל, `http://`, `ftp://`, `custom-scheme://`).</li>
  <li>
    **הסבר:**
    <ul>
      <li><code>[-.a-z0-9]+</code>: תואם תו אחד או יותר של אותיות קטנות, ספרות, מקפים או נקודות (נפוץ בסכמות פרוטוקול).</li>
      <li><code>{`:\/{1,3}`}</code>: תואם נקודתיים ואחריה אחת עד שלוש לוכסנים (<code>:/</code>, <code>://</code>, או <code>:///</code>).</li>
    </ul>
  </li>
</ul>

<h3>2. דומיין (קבוצה 2): <code>{`([^-\/\.[\](|)\s?][^\`\/\s\]?]+)`}</code></h3>

<ul>
  <li>**מטרה:** לוכד את חלק הדומיין או המארח של ה-URL.</li>
  <li>
    **הסבר:**
    <ul>
      <li><code>[^-\/\.[\](|)\s?]</code>: תואם כל תו פרט לתווים מיוחדים מוגדרים ורווח.</li>
      <li><code>[^`\/\s\]?]+</code>: תואם תו אחד או יותר פרט לגרשיים אחוריים, לוכסנים, רווח או סוגריים מרובעים סוגרים.</li>
    </ul>
  </li>
</ul>

<h3>3. נתיב (קבוצה 3): <code>{`([-_a-z0-9!@$%^&*()=+;/~\\.]*)`}</code></h3>

<ul>
  <li>**מטרה:** תואם את חלק הנתיב של ה-URL.</li>
  <li>
    **הסבר:**
    <ul>
      <li><code>[-_a-z0-9!@$%^&*()=+;/~\.]*</code>: תואם אפס תווים או יותר מתווים בטוחים ל-URL הנפוצים בנתיבים.</li>
    </ul>
  </li>
</ul>

<h3>4. שאילתה (קבוצה 4): <code>[?]?([^#\s`?]*)</code></h3>

<ul>
  <li>**מטרה:** תואם באופן אופציונלי מחרוזת שאילתה, המתחילה בתו <code>?</code> כלשהו.</li>
  <li>
    **הסבר:**
    <ul>
      <li><code>[?]?</code>: תואם באופן אופציונלי <code>?</code>. (הסוגריים המרובעים אינם הכרחיים, אך הם מעט ברורים יותר מאשר ה-<code>??</code> הכפול והקצרצר. הם גם מספקים מקבילה חזותית לקבוצת ההתאמה הבאה (הדומה) <code>[#]?</code>.)</li>
      <li><code>([^#\s`?]*)</code>: תואם אפס תווים או יותר שאינם סולמית, רווח, גרש אחורי או סימן שאלה.</li>
    </ul>
  </li>
</ul>

<h3>5. קטע (קבוצה 5): <code>[#]?([^#\s'"`\.,!]*)</code></h3>

<ul>
  <li>**מטרה:** תואם באופן אופציונלי את מזהה הקטע המתחיל ב-<code>#</code>.</li>
  <li>
    **הסבר:**
    <ul>
      <li><code>[#]?</code>: תואם באופן אופציונלי <code>#</code>.</li>
      <li><code>([^#\s'"`\.,!]*)</code>: תואם אפס תווים או יותר שאינם סימני פיסוק אסורים או רווח.</li>
    </ul>
  </li>
</ul>

## 🛠️ דוגמת ניתוח

הנה איך אפשר להפעיל את הרגקס המפלצתי הזה, עם קצת JavaScript:

<CodeTabs client:only
 tabs={[
    "קוד: חילוץ URLs",
    "תוצאות: URLs שחולצו",
    "תוצאות: חלקי URL",
  ]} >
```js title="extract-urls.js" frame="code"
const text = `
Check this out: https://example.com/path?query=123#section
And also (ftp://files.server.org/index).
Plus a weird one: custom-scheme://host/param;weird^stuff
`;

const urlRegex =
  /([-.a-z0-9]+:\/{1,3})([^-\/\.[\](|)\s?][^`\/\s\]?]+)([-_a-z0-9!@$%^&*()=+;/~\.]*)[?]?([^#\s`?]*)[#]?([^#\s'"`\.,!]*)/gi;

const matches = [
  ...text.matchAll(urlRegex),
].map((match) => match[0]);
console.log("Extracted URLs:", matches);

const parts = [
  ...text.matchAll(urlRegex),
].map((match) => match.slice(1));
console.log("Extracted Parts:", parts);
```

  "https://example.com/path?query=123#section",
  "ftp://files.server.org/index",
  "custom-scheme://host/param;weird^stuff"
]
```

    "section"      // Fragment
  ],
  [
    "ftp://",           // Protocol
    "files.server.org", // Domain
    "/index",           // Path
    "",                 // Query
    ""                  // Fragment
  ],
  [
    "custom-scheme://",   // Protocol
    "host",               // Domain
    "/param;weird^stuff", // Path
    "",                   // Query
    ""                    // Fragment
  ]
]
```

## ☑️ צעדים הבאים

בהתאם למקרה השימוש שלך, ייתכן שתצטרך לחדד את הרגקס הזה או להוסיף שלבי אימות ועיבוד נוספים.

### פרויקטים שונים, צרכים שונים

לפרויקטים יש דרישות וחששות אבטחה מגוונים:

1. **גירוד אתרים**: אמת כתובות URL כדי לוודא שהן נגישות ואמינות.
2. **עיבוד נתונים**: חלץ כתובות URL מתוכן שנוצר על ידי משתמשים תוך הבטחת בטיחות.
3. **ניתוח נתונים**: סנן כפילויות או קישורים לא רלוונטיים למטרות מחקר או שיווק.
4. **יישומים מול משתמש**: קשר אוטומטית כתובות URL באפליקציות צ'אט או פורומים.

### עיבוד ואימות לאחר החילוץ

לאחר איסוף כתובות URL פוטנציאליות, החל בדיקות נוספות:

- **בדיקת DNS**: וודא שהדומיינים נפתרים.
- **בדיקות בטיחות**: השתמש בשירותים לבדיקת אתרים זדוניים או דיוג.
- **חוקים מותאמים אישית**: החל מסננים ספציפיים לפרויקט (למשל, TLDs מותרים, אורך URL מקסימלי).

## 📝 סיכום

חילוץ נתוני מחרוזת חצי-מובנים עשוי להיות החלק המספק ביותר בשליטה ברגקס.

הנה תקציר הנקודות העיקריות:

- **השתמש בכלי חזותי לכתיבה, בדיקה** והבנה של [תבניות הרגקס שלך.](https://regex101.com/r/jO8bC4/69)
- **פרק את האתגר לחלקים** ופתור כל חלק בנפרד. במובן מסוים, קבוצות לכידה מספקות לנו 'סימני דרך' פיגורטיביים לרגקס שלנו.
- **השתמש בביטויי התאמה 'רפויים', הימנע מהקפדה על תאימות מפרט** בעת ביצוע הזנת נתונים.
- **יישום שלבי אימות** לאחר החילוץ הראשוני הוא חיוני—תמיד שקול את אבטחת הפרויקט ואת הצרכים הספציפיים שלו.

על ידי ביצוע השלבים האלה, תוכל לחלץ ביעילות כל נתוני מחרוזת חצי-מובנית, ולהניח את הבסיס לעיבוד ואימות נוספים.
