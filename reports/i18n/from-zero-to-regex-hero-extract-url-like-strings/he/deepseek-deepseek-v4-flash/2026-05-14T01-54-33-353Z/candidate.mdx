## 🚀 מבוא

חילוץ כתובות URL מטקסט גולמי יכול לפעמים להרגיש כמו משחק מייגע של 'הכה את החפרפרת'. סימני פיסוק, עטיפות בסוגריים ועיצוב מעורפל מתאחדים כדי לסכל את מאמציך. בין אם אתה בונה גורף רשת, מנתח נתונים או יישום צ'אט, חילוץ מדויק של כתובות URL הוא חיוני.

בפוסט זה, נתמודד עם הבעיה ישירות באמצעות גישה גמישה דו-שלבית. המטרה שלנו היא **לתפוס תחילה את כל המחרוזות _הפוטנציאליות_ דמויות URL** ולאחר מכן לטפל באימות בתהליך נפרד.

> 💡 **הערה:** תבנית זו אינה מיועדת **_לאימות_** כתובות URL! היא מתירנית בכוונה לגבי סימני פיסוק ואיות שגוי.

## 🔍 מטרה: חילוץ כתובות URL מטקסט

בעת חילוץ כתובות URL מטקסט גולמי, גישה דו-שלבית יעילה:

1. **לכידת כל מה שדומה ל-URL**: הטל רשת רחבה כדי לתפוס את כל המחרוזות ש*יכולות* להיות URLs. כאן ה-"regex של 120+ בתים" שלנו זורח.
2. **אימות**: לאחר שתפסת את המועמדים הללו, השתמש בבדיקות משניות (למשל, רזולוציית DNS, השוואה מול דומיינים ידועים) כדי לסנן ערכים לא חוקיים.

### המחשת האתגר

מונחים כמו `extract` ו-`parse` משמשים לעתים קרובות כמילים נרדפות, אך הם מתייחסים לתהליכים נפרדים. חילוץ URLs כרוך בזיהוי ולכידת URLs פוטנציאליים מתוך גוף טקסט גדול יותר. לעומת זאת, ניתוח (parsing) כרוך בפירוק URLs אלה לחלקיהם המרכיבים.

## ה-Regex של 120+ בתים

להלן regex תמציתי שנועד לחלץ ולנתח URLs בבת אחת. הוא תומך במגוון פרוטוקולים, דומיינים, נתיבים, וסעיפי שאילתה/פרגמנט אופציונליים.

אל דאגה — נפרק אותו צעד אחר צעד!

```js title="Regex של 120+ בתים לכתובות URL" frame="code"
const urlRegex = /([-.a-z0-9]+:\/{1,3})([^-\/\.[\](|)\s?][^`\/\s\]?]+)([-_a-z0-9!@$%^&*()=+;/~\.]*)[?]?([^#\s`?]*)[#]?([^#\s'"`\.,!]*)/gi;
// Compatibility: ES5+
```

// Same pattern, split on newlines for readability:
([-.a-z0-9]+:\/{1,3})
([^-\/\.[\](|)\s?][^`\/\s\]?]+)
([-_a-z0-9!@$%^&*()=+;/~\.]*)
[?]?([^#\s`?]*)
[#]?([^#\s'"`\.,!]*)

```

## 🧩 פירוק צעד אחר צעד

בואו נפרק את הרג'קס למרכיביו כדי להבין איך הוא עובד:

<h3>1. פרוטוקול (קבוצה 1): <code>{`([-.a-z0-9]+:\/{1,3})`}</code></h3>

<ul>
  <li>**מטרה:** תואם את חלק הפרוטוקול של ה-URL (למשל, <code>http://</code>, <code>ftp://</code>, <code>custom-scheme://</code>).</li>
  <li>
    **הסבר:**
    <ul>
      <li><code>[-.a-z0-9]+</code>: תואם תו אחד או יותר של אותיות קטנות, ספרות, מקפים או נקודות (נפוץ בסכמות פרוטוקול).</li>
      <li><code>{`:\/{1,3}`}</code>: תואם נקודתיים ואחריה אחת עד שלוש לוכסנים (<code>:/</code>, <code>://</code>, או <code>:///</code>).</li>
    </ul>
  </li>
</ul>

<h3>2. דומיין (קבוצה 2): <code>{`([^-\/\.[\](|)\s?][^\`\/\s\]?]+)`}</code></h3>

<ul>
  <li>**מטרה:** לוכד את חלק הדומיין או המארח של ה-URL.</li>
  <li>
    **הסבר:**
    <ul>
      <li><code>[^-\/\.[\](|)\s?]</code>: תואם כל תו מלבד תווים מיוחדים שצוינו ורווחים.</li>
      <li><code>[^`\/\s\]?]+</code>: תואם תו אחד או יותר מלבד גרשיים אחוריים, לוכסנים, רווחים או סוגריים מרובעים סוגרים.</li>
    </ul>
  </li>
</ul>

### 3. נתיב (קבוצה 3): <code>{`([-_a-z0-9!@$%^&*()=+;/~\\.]*)`}</code>

<ul>
  <li>**מטרה:** תואם את רכיב הנתיב של ה-URL.</li>
  <li>
    **הסבר:**
    <ul>
      <li><code>[-_a-z0-9!@$%^&*()=+;/~\.]*</code>: תואם אפס או יותר תווים בטוחים ל-URL הנפוצים בנתיבים.</li>
    </ul>
  </li>
</ul>

<h3>4. שאילתה (קבוצה 4): <code>[?]?([^#\s`?]*)</code></h3>

<ul>
  <li>**מטרה:** תואם באופן אופציונלי מחרוזת שאילתה, המתחילה בתו <code>?</code> כלשהו.</li>
  <li>
    **הסבר:**
    <ul>
      <li><code>[?]?</code>: תואם באופן אופציונלי <code>?</code>. (הסוגריים המרובעים אינם הכרחיים, אך הם מעט ברורים יותר מאשר ה-<code>??</code> הכפול והתמציתי. הם גם מספקים הקבלה ויזואלית לקבוצה הבאה הדומה <code>[#]?</code>.)</li>
      <li><code>([^#\s`?]*)</code>: תואם אפס או יותר תווים שאינם סולמית, רווח, גרש אחורי או סימן שאלה.</li>
    </ul>
  </li>
</ul>

<h3>5. קטע (קבוצה 5): <code>[#]?([^#\s'"`\.,!]*)</code></h3>

<ul>
  <li>**מטרה:** תואם באופן אופציונלי את מזהה הקטע המתחיל ב-<code>#</code>.</li>
  <li>
    **הסבר:**
    <ul>
      <li><code>[#]?</code>: תואם באופן אופציונלי <code>#</code>.</li>
      <li><code>([^#\s'"`\.,!]*)</code>: תואם אפס או יותר תווים שאינם סימני פיסוק אסורים או רווחים.</li>
    </ul>
  </li>
</ul>

## 🛠️ דוגמת ניתוח

הנה איך אפשר להפעיל את הביטוי הרגולרי המפלצתי הזה, עם קצת JavaScript:

<CodeTabs client:only
 tabs={[
    "קוד: חילוץ URLs",
    "תוצאות: URLs שחולצו",
    "תוצאות: חלקי URL",
  ]} >
```js title="extract-urls.js" frame="code"
const text = `
Check this out: https://example.com/path?query=123#section
And also (ftp://files.server.org/index).
Plus a weird one: custom-scheme://host/param;weird^stuff
`;

const urlRegex =
  /([-.a-z0-9]+:\/{1,3})([^-\/\.[\](|)\s?][^`\/\s\]?]+)([-_a-z0-9!@$%^&*()=+;/~\.]*)[?]?([^#\s`?]*)[#]?([^#\s'"`\.,!]*)/gi;

const matches = [
  ...text.matchAll(urlRegex),
].map((match) => match[0]);
console.log("Extracted URLs:", matches);

const parts = [
  ...text.matchAll(urlRegex),
].map((match) => match.slice(1));
console.log("Extracted Parts:", parts);
```

```json title="extracted-urls.json"
[
  "https://example.com/path?query=123#section",
  "ftp://files.server.org/index",
  "custom-scheme://host/param;weird^stuff"
]
```

```json title="urls-parts.json"
[
  [
    "https://",    // Protocol
    "example.com", // Domain
    "/path",       // Path
    "query=123",   // Query
    "section"      // Fragment
  ],
  [
    "ftp://",           // Protocol
    "files.server.org", // Domain
    "/index",           // Path
    "",                 // Query
    ""                  // Fragment
  ],
  [
    "custom-scheme://",   // Protocol
    "host",               // Domain
    "/param;weird^stuff", // Path
    "",                   // Query
    ""                    // Fragment
  ]
]
```

## ☑️ שלבים הבאים

בהתאם למקרה השימוש שלך, ייתכן שתצטרך לחדד את הביטוי הרגולרי הזה או להוסיף עוד שלבי אימות ועיבוד לאחר מכן.

### פרויקטים שונים, צרכים שונים

לפרויקטים יש דרישות וחששות אבטחה מגוונים:

1. **גריפת אתרים (Web Scraping)**: אמת כתובות URL כדי לוודא שהן ניתנות להשגה ואמינות.
2. **עיבוד נתונים**: חלץ כתובות URL מתוכן שנוצר על ידי משתמשים תוך הבטחת בטיחות.
3. **ניתוח נתונים**: סנן כפילויות או קישורים לא רלוונטיים לצורכי מחקר או שיווק.
4. **יישומים מול משתמשים**: קשר אוטומטי של כתובות URL ביישומי צ'אט או פורומים.

### עיבוד ואימות לאחר מכן

לאחר איסוף כתובות URL פוטנציאליות, יש להחיל בדיקות נוספות:

- **בדיקת DNS**: ודא שהדומיינים נפתרים.
- **בדיקות בטיחות**: השתמש בשירותים לבדיקת אתרים זדוניים או דיוג.
- **כללים מותאמים אישית**: החל מסננים ספציפיים לפרויקט (למשל, TLD מורשים, אורך URL מקסימלי).

## 📝 סיכום

חילוץ נתוני מחרוזת חצי-מובנים עשוי להיות החלק המספק ביותר בשליטה בביטויים רגולריים.

הנה תקציר של הנקודות העיקריות:

- **השתמש בכלי חזותי לכתיבה, בדיקה** והבנה של [תבניות Regex שלך.](https://regex101.com/r/jO8bC4/69)
- **פרק את האתגר לחלקים** ופתור כל חלק בנפרד. במובן מסוים, קבוצות לכידה מספקות לנו 'סימני דרך' פיגורטיביים עבור הביטוי הרגולרי שלנו.
- **השתמש בביטויי התאמה 'רפויים', הימנע מהתאמה מחמירה לתקן** בעת ביצוע הזנת נתונים.
- **יישום שלבי אימות** לאחר החילוץ הראשוני הוא חיוני – תמיד שקול את האבטחה של הפרויקט שלך ואת הצרכים הספציפיים שלו.

על ידי ביצוע השלבים האלה, תוכל לחלץ ביעילות כל נתוני מחרוזת חצי-מובנים, ולהניח את הבסיס לעיבוד ואימות נוספים.
