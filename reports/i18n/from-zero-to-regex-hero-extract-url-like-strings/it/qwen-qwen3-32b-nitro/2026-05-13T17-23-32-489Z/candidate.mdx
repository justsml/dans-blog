## 🚀 Introduzione

Estrarre URL da testo grezzo può sembrare talvolta un noioso gioco di colpire le talpe. La punteggiatura, gli avvolgimenti parentetici e la formattazione ambigua si combinano per rendere frustranti i tuoi sforzi. Che tu stia sviluppando uno scraper web, un analizzatore di dati o un'applicazione di chat, estrarre correttamente le URL è essenziale.

In questo articolo affronteremo il problema frontalmente con un approccio flessibile a due passaggi. Il nostro obiettivo è **catturare prima tutte le _potenziali_ stringhe simili a URL** e gestire la validazione in un processo successivo.

> 💡 **Nota:** Questo modello non è per **_validare_** le URL! È intenzionalmente permissivo riguardo alla punteggiatura e all'ortografia errata.

## 🔍 Obiettivo: Estrarre URL dal Testo

Quando si estraggono URL da testo grezzo, un approccio a due passaggi è efficace:

1. **Cattura Tutto URL-simile**: Lancia una rete larga per acchiappare tutte le stringhe che *potrebbero* essere URL. È qui che la nostra "regex da 120+ byte" eccelle.
2. **Valida**: Una volta catturati questi candidati, utilizza controlli secondari (es. risoluzione DNS, confronto con domini noti) per scartare le voci non valide.

### Visualizzare la Sfida

Termini come `estrarre` e `analizzare` vengono spesso usati come sinonimi, anche se indicano processi distinti. Estrarre gli URL consiste nell'identificare e catturare potenziali URL all'interno di un testo più ampio. Analizzare, invece, implica scomporre questi URL nei loro componenti.

## La regex da 120+ byte

Di seguito una regex concisa progettata per estrarre e analizzare gli URL in un'unica operazione. Supporta vari protocolli, domini, percorsi e sezioni opzionali di query/fragment.

Non preoccuparti—scomponiamo passo per passo!

```js title="120+ Byte URL Regex" frame="code"
const urlRegex = /([-.a-z0-9]+:\/{1,3})([^-\/\.[\](|)\s?][^`\/\s\]?]+)([-_a-z0-9!@$%^&*()=+;/~\.]*)[?]?([^#\s`?]*)[#]?([^#\s'"`\.,!]*)/gi;
// Compatibility: ES5+

// Same pattern, split on newlines for readability:
([-.a-z0-9]+:\/{1,3})
([^-\/\.[\](|)\s?][^`\/\s\]?]+)
([-_a-z0-9!@$%^&*()=+;/~\.]*)
[?]?([^#\s`?]*)
[#]?([^#\s'"`\.,!]*)

```

## 🧩 Scomposizione passo per passo

Analizziamo il regex nei suoi componenti per comprendere come funziona:

<h3>1. Protocollo (Gruppo 1): <code>{`([-.a-z0-9]+:\/{1,3})`}</code></h3>

<ul>
  <li>**Scopo:** Corrisponde al protocollo dell'URL (es. `http://`, `ftp://`, `custom-scheme://`).</li>
  <li>
    **Spiegazione:**
    <ul>
      <li><code>[-.a-z0-9]+</code>: Corrisponde a uno o più caratteri minuscoli, cifre, trattini o punti (comuni nei protocolli).</li>
      <li><code>{`:\/{1,3}`}</code>: Corrisponde a un doppio punto seguito da uno a tre slash (<code>:/</code>, <code>://</code> o <code>:///</code>).</li>
    </ul>
  </li>
</ul>

<h3>2. Dominio (Gruppo 2): <code>{`([^-\/\.[\](|)\s?][^\`\/\s\]?]+)`}</code></h3>

<ul>
  <li>**Scopo:** Cattura il dominio o l'host dell'URL.</li>
  <li>
    **Spiegazione:**
    <ul>
      <li><code>[^-\/\.[\](|)\s?]</code>: Corrisponde a qualsiasi carattere tranne quelli speciali specificati e gli spazi bianchi.</li>
      <li><code>[^`\/\s\]?]+</code>: Corrisponde a uno o più caratteri tranne backtick, slash, spazi bianchi o parentesi quadrate chiuse.</li>
    </ul>
  </li>
</ul>

<h3>3. Percorso (Gruppo 3): <code>{`([-_a-z0-9!@$%^&*()=+;/~\\.]*)`}</code></h3>

<ul>
  <li><strong>Scopo:</strong> Corrisponde al componente path dell'URL.</li>
  <li>
    <strong>Spiegazione:</strong>
    <ul>
      <li><code>[-_a-z0-9!@$%^&*()=+;/~\.]*</code>: Corrisponde a zero o più caratteri sicuri per URL comunemente presenti nei path.</li>
    </ul>
  </li>
</ul>

<h3>4. Query (Gruppo 4): <code>[?]?([^#\s`?]*)</code></h3>

<ul>
  <li><strong>Scopo:</strong> Corrisponde opzionalmente alla stringa di query, iniziando con il carattere <code>?</code>.</li>
  <li>
    <strong>Spiegazione:</strong>
    <ul>
      <li><code>[?]?</code>: Corrisponde opzionalmente a <code>?</code>. (Le parentesi quadrate non sono strettamente necessarie, tuttavia sono leggermente più chiare rispetto alla forma estremamente concisa <code>??</code>. Forniscono anche un parallelo visivo per il gruppo successivo <code>[#]?</code>.)</li>
      <li><code>([^#\s`?]*)</code>: Corrisponde a zero o più caratteri che non siano hash, spazi bianchi, backtick o punto interrogativo.</li>
    </ul>
  </li>
</ul>

<h3>5. Fragment (Gruppo 5): <code>[#]?([^#\s'"`\.,!]*)</code></h3>

<ul>
  <li><strong>Scopo:</strong> Corrisponde opzionalmente all'identificatore di frammento iniziante con <code>#</code>.</li>
  <li>
    <strong>Spiegazione:</strong>
    <ul>
      <li><code>[#]?</code>: Corrisponde opzionalmente a <code>#</code>.</li>
      <li><code>([^#\s'"`\.,!]*)</code>: Corrisponde a zero o più caratteri che non siano punteggiatura vietata o spazi bianchi.</li>
    </ul>
  </li>
</ul>

## 🛠️ Esempio di Parsing

Ecco come puoi mettere a lavoro questa espressione regolare, con un po' di JavaScript:

<CodeTabs client:only
 tabs={[

    "Risultati: Parti URL",

```js title="extract-urls.js" frame="code"
const text = `
Controlla questo: https://example.com/path?query=123#section
E anche (ftp://files.server.org/index).
Un altro strano: custom-scheme://host/param;weird^stuff
`;

const urlRegex =
  /([-.a-z0-9]+:\/{1,3})([^-\/\.[\](|)\s?][^`\/\s\]?]+)([-_a-z0-9!@$%^&*()=+;/~\.]*)[?]?([^#\s`?]*)[#]?([^#\s'"`\.,!]*)/gi;

const matches = [
  ...text.matchAll(urlRegex),
].map((match) => match[0]);
console.log("URL Estratti:", matches);

const parts = [
  ...text.matchAll(urlRegex),
].map((match) => match.slice(1));
console.log("Parti Estratte:", parts);
```

## ☑️ Passi Successivi

A seconda del tuo utilizzo, potresti dover raffinare questa espressione regolare o aggiungere ulteriori passaggi di validazione e post-elaborazione.

### Progetti Diversi, Bisogni Diversi

I progetti hanno requisiti e preoccupazioni di sicurezza variabili:

1. **Web Scraping**: Convalidare gli URL per assicurarsi che siano raggiungibili e attendibili.  
2. **Data Processing**: Estrarre gli URL da contenuti generati dagli utenti garantendo la sicurezza.  
3. **Data Analysis**: Filtrare duplicati o collegamenti irrilevanti per ricerche o scopi di marketing.  
4. **Applicazioni a interfaccia utente**: Collegare automaticamente gli URL in app di chat o forum.  

### Post-elaborazione e convalida  

Dopo aver raccolto potenziali URL, applicare controlli aggiuntivi:  

- **Verifica DNS**: Verificare che i domini risolvano.  
- **Controlli di sicurezza**: Usare servizi per rilevare siti dannosi o phishing.  
- **Regole personalizzate**: Applicare filtri specifici del progetto (es. TLD consentiti, lunghezza massima URL).  

## 📝 Riepilogo  

Estrarre dati di stringhe semistrutturati potrebbe essere la parte più soddisfacente della padronanza delle espressioni regolari.

Ecco un riepilogo dei punti chiave:

- **Usa uno strumento visivo per scrivere, testare** e comprendere le tue [espressioni regex.](https://regex101.com/r/jO8bC4/69)  
- **Suddividi il problema in parti** e risolvi ciascuna parte separatamente. In un certo senso, i gruppi di cattura ci forniscono "indicazioni stradali figurative" per le nostre espressioni regex.  
- **Usa espressioni di corrispondenza "lasse", evita la conformità rigorosa agli standard** quando si tratta di ingestione dati.  
- **Applicare passaggi di convalida** dopo l'estrazione iniziale è essenziale—considera sempre la sicurezza e i bisogni specifici del tuo progetto.  

Seguendo questi passaggi, potrai estrarre efficacemente qualsiasi dato di stringa semistrutturato, creando la base per ulteriori elaborazioni e convalida.
