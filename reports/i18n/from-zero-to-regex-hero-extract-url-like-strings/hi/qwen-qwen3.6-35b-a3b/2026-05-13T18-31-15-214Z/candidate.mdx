## 🚀 परिचय

कच्चे टेक्स्ट स्ट्रिंग्स से URLs निकालना अक्सर एक थका देने वाला वाक-ए-मोल खेल बन जाता है। विराम चिह्न, कोष्ठक से लिपटे पैरेंथेसिस, और अस्पष्ट फॉर्मेटिंग स्ट्रक्चर आपकी पार्सिंग लॉजिक को बाधित करते हैं। चाहे आप वेब स्क्रैपर, डेटा एनालाइजर, या चैट एप्लिकेशन डेवलप कर रहे हों, URLs को सटीक रूप से कैप्चर करना सिस्टम की सटीकता और डेटा इंटिग्रिटी के लिए अनिवार्य है। इस ट्यूटोरियल में हम एक फ्लेक्सिबल, दो-चरणीय दृष्टिकोण अपनाएंगे जो स्ट्रिक्ट वैलिडेशन से ज्यादा फ्लेक्सिबिलिटी और रीडेबिलिटी पर फोकस करता है।

इस पोस्ट में, हम एक लचीले, दो-चरणीय दृष्टिकोण के साथ इस समस्या का सीधा समाधान करेंगे। हमारा लक्ष्य है **सबसे पहले सभी _संभावित_ URL-जैसे स्ट्रिंग्स को कैप्चर करना** और वैलिडेशन को अगले चरण में हैंडल करना।

> 💡 **नोट:** यह पैटर्न URLs को **_वैलिडेट_** करने के लिए नहीं है! यह जानबूझकर पंक्चुएशन और खराब स्पेलिंग के लिए परमिसिव (permissive) है।

## 🔍 लक्ष्य: टेक्स्ट से URLs निकालना

रॉ टेक्स्ट से URLs निकालते समय एक दो-चरणीय दृष्टिकोण प्रभावी साबित होता है:

1. **URL-जैसे सब कुछ कैप्चर करें**: एक चौड़ी जाल बिछाएं ताकि वे सभी स्ट्रिंग्स मिल सकें जो *URL हो सकती हैं*। यहीं पर हमारा '120+ बाइट regex' काम आता है।
2. **वैलिडेट करें**: एक बार आपने इन उम्मीदवारों को कैप्चर कर लिया हो, तो अमान्य एंट्रीज़ को बाहर निकालने के लिए सेकेंडरी चेक्स (जैसे DNS रेज़ोल्यूशन, ज्ञात डोमेन के साथ तुलना) का उपयोग करें।

### चुनौती को विज़ुअलाइज़ करना

`extract` और `parse` जैसे शब्दों का अक्सर परस्पर प्रयोग किया जाता है, हालाँकि ये दो अलग-अलग प्रक्रियाओं की ओर इशारा करते हैं। `extract` (URL निकालना) एक बड़े टेक्स्ट ब्लॉक से संभावित URL को पहचानने और कैप्चर करने की प्रक्रिया है। वहीं, `parse` (URL पार्स करना) इन URL को उनके घटक भागों में तोड़ने की प्रक्रिया है।

## 120+ बाइट का Regex

नीचे एक संक्षिप्त regex दिया गया है जो एक ही चरण में URL निकालने और पार्स करने के लिए तैयार किया गया है। यह विभिन्न प्रोटोकॉल, डोमेन, पाथ और वैकल्पिक क्वेरी/फ्रैगमेंट सेक्शन को सपोर्ट करता है।

चिंता न करें—हम इसे चरण-दर-चरण विघटित करेंगे।

```js title="120+ बाइट URL Regex" frame="code"
const urlRegex = /([-.a-z0-9]+:\/{1,3})([^-\/\.[\](|)\s?][^`\/\s\]?]+)([-_a-z0-9!@$%^&*()=+;/~\.]*)[?]?([^#\s`?]*)[#]?([^#\s'"`\.,!]*)/gi;
// Compatibility: ES5+

// Same pattern, split on newlines for readability:
([-.a-z0-9]+:\/{1,3})
([^-\/\.[\](|)\s?][^`\/\s\]?]+)
([-_a-z0-9!@$%^&*()=+;/~\.]*)
[?]?([^#\s`?]*)
[#]?([^#\s'"`\.,!]*)

```

<blockquote class="inset">आपने जिन सबसे जटिल regex (या स्वयं लिखे regex) का सामना किया है, उन्हें <a href="#post-comments">नीचे कमेंट्स में</a> साझा करें! 🚀</blockquote>

## 🧩 इसे चरण-दर-चरण तोड़ना

regex के घटकों को अलग-अलग करके देखते हैं कि यह आंतरिक रूप से कैसे काम करता है:

<h3>1. प्रोटोकॉल (ग्रुप 1): <code>{`([-.a-z0-9]+:\/{1,3})`}</code></h3>

<ul>
  <li>**उद्देश्य:** URL के प्रोटोकॉल भाग (जैसे `http://`, `ftp://`, `custom-scheme://`) को मैच करता है।</li>
  <li>
    **व्याख्या:**
    <ul>
      <li><code>[-.a-z0-9]+</code>: प्रोटोकॉल स्कीम में आम तौर पर मिलने वाले एक या अधिक लघु अक्षरों, अंकों, हाइफ़न या डॉट्स को मैच करता है।</li>
      <li><code>{`:\/{1,3}`}</code>: एक कालन (colon) के बाद एक से तीन स्लैश (<code>:/</code>, <code>://</code>, या <code>:///</code>) को मैच करता है।</li>
    </ul>
  </li>
</ul>

<h3>2. डोमेन (ग्रुप 2): <code>{`([^-\/\.[\](|)\s?][^\`\/\s\]?]+)`}</code></h3>

<ul>
  <li>**उद्देश्य:** URL के डोमेन या होस्ट भाग को कैप्चर करता है।</li>
  <li>
    **व्याख्या:**
    <ul>
      <li><code>[^-\/\.[\](|)\s?]</code>: निर्दिष्ट विशेष अक्षरों और व्हाइटस्पेस को छोड़कर किसी भी अक्षर को मैच करता है।</li>
      <li><code>[^`\/\s\]?]+</code>: बैकटिक, स्लैश, व्हाइटस्पेस या बंद स्क्वायर ब्रैकेट को छोड़कर एक या अधिक अक्षरों को मैच करता है।</li>
    </ul>
  </li>
</ul>

<h3>3. पथ (ग्रुप 3): <code>{`([-_a-z0-9!@$%^&*()=+;/~\\.]*)`}</code></h3>

<ul>
  <li>**उद्देश्य:** URL के पथ घटक को मैच करता है।</li>
  <li>
    **व्याख्या:**
    <ul>
      <li><code>[-_a-z0-9!@$%^&*()=+;/~\.]*</code>: पथों में आम तौर पर पाए जाने वाले शून्य या अधिक URL-सुरक्षित अक्षरों को मैच करता है।</li>
    </ul>
  </li>
</ul>

<h3>4. क्वेरी (ग्रुप 4): <code>[?]?([^#\s`?]*)</code></h3>

<ul>
  <li>**उद्देश्य:** वैकल्पिक रूप से एक क्वेरी स्ट्रिंग को मैच करता है, जो किसी भी <code>?</code> अक्षर से शुरू होती है।</li>
  <li>
    **व्याख्या:**
    <ul>
      <li><code>[?]?</code>: वैकल्पिक रूप से एक <code>?</code> को मैच करता है। (वर्ग ब्रैकेट सख्ती से आवश्यक नहीं हैं, हालांकि वे अत्यंत संक्षिप्त डबल <code>??</code> की तुलना में थोड़े स्पष्ट हैं। यह अगले समान मैचिंग ग्रुप <code>[#]?</code> के लिए एक दृश्य समानता भी प्रदान करता है।)</li>
      <li><code>([^#\s`?]*)</code>: हैश, व्हाइटस्पेस, बैकटिक या प्रश्नवाचक चिह्न को छोड़कर शून्य या अधिक अक्षरों को मैच करता है।</li>
    </ul>
  </li>
</ul>

<h3>5. फ्रैगमेंट (ग्रुप 5): <code>[#]?([^#\s'"`\.,!]*)</code></h3>

<ul>
  <li>**उद्देश्य:** वैकल्पिक रूप से <code>#</code> से शुरू होने वाले फ्रैगमेंट पहचानकर्ता को मैच करता है।</li>
  <li>
    **व्याख्या:**
    <ul>
      <li><code>[#]?</code>: वैकल्पिक रूप से एक <code>#</code> को मैच करता है।</li>
      <li><code>([^#\s'"`\.,!]*)</code>: अनुपयुक्त विराम चिह्नों या व्हाइटस्पेस को छोड़कर शून्य या अधिक अक्षरों को मैच करता है।</li>
    </ul>
  </li>
</ul>

## 🛠️ पार्सिंग उदाहरण

यहाँ बताया गया है कि आप इस जटिल रेगेक्स को जावास्क्रिप्ट के साथ कैसे लागू कर सकते हैं:

<CodeTabs client:only
 tabs={[
    "कोड: URL निकालें",
    "परिणाम: निकाले गए URL",
    "परिणाम: URL के भाग",
  ]} >
```js title="extract-urls.js" frame="code"
const text = `
Check this out: https://example.com/path?query=123#section
And also (ftp://files.server.org/index).
Plus a weird one: custom-scheme://host/param;weird^stuff
`;

const urlRegex =
  /([-.a-z0-9]+:\/{1,3})([^-\/\.[\](|)\s?][^`\/\s\]?]+)([-_a-z0-9!@$%^&*()=+;/~\.]*)[?]?([^#\s`?]*)[#]?([^#\s'"`\.,!]*)/gi;

const matches = [
  ...text.matchAll(urlRegex),
].map((match) => match[0]);
console.log("Extracted URLs:", matches);

const parts = [
  ...text.matchAll(urlRegex),
].map((match) => match.slice(1));
console.log("Extracted Parts:", parts);
```

```json title="extracted-urls.json"
[
  "https://example.com/path?query=123#section",
  "ftp://files.server.org/index",
  "custom-scheme://host/param;weird^stuff"
]
```

```json title="urls-parts.json"
[
  [
    "https://",    // Protocol
    "example.com", // Domain
    "/path",       // Path
    "query=123",   // Query
    "section"      // Fragment
  ],
  [
    "ftp://",           // Protocol
    "files.server.org", // Domain
    "/index",           // Path
    "",                 // Query
    ""                  // Fragment
  ],
  [
    "custom-scheme://",   // Protocol
    "host",               // Domain
    "/param;weird^stuff", // Path
    "",                   // Query
    ""                    // Fragment
  ]
]
```

## ☑️ अगले कदम

आपके उपयोग के मामले के आधार पर, आपको इस रेगेक्स को परिष्कृत करने या अधिक सत्यापन और पोस्ट-प्रोसेसिंग चरण जोड़ने की आवश्यकता हो सकती है।

### विभिन्न परियोजनाएँ, विभिन्न आवश्यकताएँ

परियोजनाओं की आवश्यकताएँ और सुरक्षा चिंताएँ भिन्न होती हैं:

1. **वेब स्क्रैपिंग**: URL सत्यापित करें ताकि वे पहुँच योग्य और विश्वसनीय हों।
2. **डेटा प्रोसेसिंग**: उपयोगकर्ता सामग्री से URL निकालते समय सुरक्षा जोखिम सीमित करें।
3. **डेटा विश्लेषण**: शोध या मार्केटिंग के लिए डुप्लिकेट/असंबंधित लिंक फ़िल्टर करें।
4. **यूज़र-फेसिंग एप्लिकेशन**: चैट ऐप्स या फोरम में URL को स्वचालित हाइपरलिंक बनाएं।

### पोस्ट-प्रोसेसिंग और सत्यापन

संभावित URL मिलने के बाद अतिरिक्त जाँच लागू करें:

- **DNS लुकअप**: डोमेन रिज़ॉल्व होने की पुष्टि करें।
- **सुरक्षा जाँच**: दुर्भावनापूर्ण या फ़िशिंग साइटों के जोखिम को कम करने के लिए सेवाओं का उपयोग करें।
- **कस्टम नियम**: परियोजना-विशिष्ट फ़िल्टर लागू करें (जैसे, अनुमत TLD, अधिकतम URL लंबाई)।

## 📝 सारांश

अर्ध-संरचित स्ट्रिंग डेटा निकालना रेगेक्स में निपुणता का सबसे संतोषजनक पहलू है।

मुख्य निष्कर्षों का सारांश यहाँ दिया गया है:

- **रेगेक्स पैटर्न लिखने, परखने और समझने के लिए** एक विज़ुअल टूल का उपयोग करें।[रेगेक्स पैटर्न।](https://regex101.com/r/jO8bC4/69)
- **चुनौती को भागों में बाँटें** और प्रत्येक भाग को अलग-अलग हल करें। इस हिसाब से, कैप्चर ग्रुप्स रेगेक्स के लिए हमें एक रूपक 'ट्रेल मार्कर' प्रदान करते हैं।
- **डेटा इनटेक के दौरान 'लूज़' मैच एक्सप्रेशन का उपयोग करें, और सख्त स्पेसिफिकेशन अनुपालन से बचें।**
- **प्रारंभिक एक्सट्रैक्शन के बाद सत्यापन चरण लागू करना** आवश्यक है—हमेशा अपने प्रोजेक्ट की सुरक्षा और विशिष्ट आवश्यकताओं को ध्यान में रखें।

इन चरणों का पालन करने से आप किसी भी अर्ध-संरचित स्ट्रिंग डेटा को प्रभावी ढंग से निकाल सकते हैं, जिससे आगे के प्रसंस्करण और सत्यापन के लिए एक मजबूत नींव तैयार होती है।
