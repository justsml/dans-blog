## 🚀 परिचय

कच्चे पाठ से URL निकालना कभी-कभी एक बोरिंग खेल जैसा महसूस हो सकता है। विराम चिह्न, पैरेंथेटिकल लपेट, और अस्पष्ट फॉरमेटिंग सभी आपके प्रयासों को निराश करने के लिए मिले हुए लगते हैं। चाहे आप एक वेब स्क्रैपर, डेटा विश्लेषक, या चैट एप्लिकेशन बना रहे हों, URL को सटीक रूप से निकालना आवश्यक है।

इस पोस्ट में, हम एक सुगम, दो-चरण दृष्टिकोण के साथ इस समस्या का सीधा सामना करेंगे। हमारा लक्ष्य **सभी _संभावित_ URL-जैसी स्ट्रिंग्स को पहले पकड़ना है** और फिर अगले प्रक्रिया में मान्यता प्राप्त करेंगे।

> 💡 **नोट:** यह पैटर्न **_मान्यता प्राप्त_** करने के लिए नहीं है! यह विराम चिह्नों और खराब वर्तनी के साथ बर्दाश्त करता है।

## 🔍 लक्ष्य: टेक्स्ट से URL निकालें

जब कच्चे टेक्स्ट से URL निकालने की बात आती है, तो एक दो-चरण दृष्टिकोण प्रभावी होता है:

1. **सभी URL-जैसी चीजें पकड़ें**: एक चौड़ा जाल बिछाएं जिससे सभी स्ट्रिंग्स जो *हो सकता है* URL हों, उन्हें पकड़ लिया जाए। यहां हमारा "120+ बाइट रेगेक्स" उज्ज्वल काम करता है।
2. **मान्यता प्राप्त करें**: एक बार जब आपने इन उम्मीदवारों को पकड़ लिया हो, तो द्वितीयक जांच (जैसे, डीएनएस रिज़ॉल्यूशन, ज्ञात डोमेन के साथ तुलना) का उपयोग करके अमान्य प्रविष्टियों को छाँट लें।

### चुनौती को देखना

`निकालें` और `अनुक्रमित करें` जैसे शब्दों का अक्सर एक-दूसरे के स्थान पर उपयोग किया जाता है, हालांकि वे अलग-अलग प्रक्रियाओं को संदर्भित करते हैं। URL निकालना एक बड़े पाठ के शरीर से संभावित URL की पहचान और पकड़ करने को संदर्भित करता है। दूसरी ओर, अनुक्रमित करना इन URL को उनके घटक भागों में तोड़ने को संदर्भित करता है।

## 120+ बाइट रेगेक्स

नीचे दिए गए संक्षिप्त रेगेक्स को एकल शॉट में URL निकालने और अनुक्रमित करने के लिए डिज़ाइन किया गया है। यह विभिन्न प्रोटोकॉल, डोमेन, पाथ और वैकल्पिक प्रश्न/खंड अनुभागों का समर्थन करता है।

चिंता न करें—हम इसे धीरे-धीरे तोड़कर समझेंगे!

```js title="120+ Byte URL Regex" frame="code"
const urlRegex = /([-.a-z0-9]+:\/{1,3})([^-\/\.[\](|)\s?][^`\/\s\]?]+)([-_a-z0-9!@$%^&*()=+;/~\.]*)[?]?([^#\s`?]*)[#]?([^#\s'"`\.,!]*)/gi;
// संगतता: ES5+

// एक ही पैटर्न, पठनीयता के लिए नई पंक्ति पर विभाजित:
([-.a-z0-9]+:\/{1,3})
([^-\/\.[\](|)\s?][^`\/\s\]?]+)
([-_a-z0-9!@$%^&*()=+;/~\.]*)
[?]?([^#\s`?]*)
[#]?([^#\s'"`\.,!]*)

```

## 🧩 धागा-दर-धागा विश्लेषण

चलिए इस रेगेक्स को उसके घटकों में विभाजित करके देखते हैं कि यह कैसे काम करता है:

<h3>1. प्रोटोकॉल (ग्रुप 1): <code>{`([-.a-z0-9]+:\/{1,3})`}</code></h3>

<ul>
  <li>**उद्देश्य:** URL के प्रोटोकॉल भाग के मिलान के लिए (जैसे `http://`, `ftp://`, `custom-scheme://`).</li>
  <li>
    **स्पष्टीकरण:**
    <ul>
      <li><code>[-.a-z0-9]+</code>: एक या अधिक छोटे अक्षर, अंक, हाइफ़न, या डॉट के मिलान (प्रोटोकॉल स्कीम में सामान्य होते हैं).</li>
      <li><code>{`:\/{1,3}`}</code>: एक कोलन के बाद एक से तीन स्लैश के मिलान (<code>:/</code>, <code>://</code>, या <code>:///</code>).</li>
    </ul>
  </li>
</ul>

<h3>2. डोमेन (ग्रुप 2): <code>{`([^-\/\.[\](|)\s?][^\`\/\s\]?]+)`}</code></h3>

<ul>
  <li>**उद्देश्य:** URL के डोमेन या होस्ट भाग को पकड़ता है।</li>
  <li>
    **स्पष्टीकरण:**
    <ul>
      <li><code>[^-\/\.[\](|)\s?]</code>: निर्दिष्ट विशेष वर्णों और व्हाइटस्पेस के अलावा किसी भी वर्ण के मिलान।</li>
      <li><code>[^`\/\s\]?]+</code>: बैकटिक, स्लैश, व्हाइटस्पेस, या बंद वर्ग ब्रैकेट के अलावा एक या अधिक वर्णों के मिलान।</li>
    </ul>
  </li>
</ul>

<h3>3. पाथ (ग्रुप 3): <code>{`([-_a-z0-9!@$%^&*()=+;/~\\.]*)`}</code></h3>

<ul>
  <li>**उद्देश्य:** URL के पाथ घटक के मिलान।</li>
  <li>
    **स्पष्टीकरण:**
    <ul>
      <li><code>[-_a-z0-9!@$%^&*()=+;/~\.]*</code>: पाथ में आमतौर पर पाए जाने वाले शून्य या अधिक URL-सुरक्षित वर्णों के मिलान।</li>
    </ul>
  </li>
</ul>

<h3>4. क्वेरी (ग्रुप 4): <code>[?]?([^#\s`?]*)</code></h3>

<ul>
  <li>**उद्देश्य:** एक वैकल्पिक क्वेरी स्ट्रिंग के मिलान, जो कोई <code>?</code> चिह्न से शुरू होती है।</li>
  <li>
    **स्पष्टीकरण:**
    <ul>
      <li><code>[?]?</code>: वैकल्पिक रूप से एक <code>?</code> के मिलान। (वर्ग ब्रैकेट पूरी तरह आवश्यक नहीं हैं, हालांकि वे अत्यंत संक्षिप्त डबल <code>??</code> की तुलना में थोड़ा स्पष्ट हैं। यह अगले मिलान ग्रुप <code>[#]?</code> के साथ दृश्यमान समानता भी प्रदान करता है।)</li>
      <li><code>([^#\s`?]*)</code>: हैश, व्हाइटस्पेस, बैकटिक, या प्रश्न चिह्न के अलावा शून्य या अधिक वर्णों के मिलान।</li>
    </ul>
  </li>
</ul>

<h3>5. फ्रैगमेंट (ग्रुप 5): <code>[#]?([^#\s'"`\.,!]*)</code></h3>

<ul>
  <li>**उद्देश्य:** एक वैकल्पिक फ्रैगमेंट आइडेंटिफायर के मिलान, जो <code>#</code> से शुरू होता है।</li>
  <li>
    **स्पष्टीकरण:**
    <ul>
      <li><code>[#]?</code>: वैकल्पिक रूप से एक <code>#</code> के मिलान।</li>
      <li><code>([^#\s'"`\.,!]*)</code>: प्रतिबंधित पंक्ति चिह्न या व्हाइटस्पेस के अलावा शून्य या अधिक वर्णों के मिलान।</li>
    </ul>
  </li>
</ul>

## 🛠️ पार्सिंग उदाहरण

यहाँ आप थोड़ा जावास्क्रिप्ट उपयोग करके इस भयानक रेगेक्स को काम में ला सकते हैं:

<CodeTabs client:only
 tabs={[
    "कोड: URL निकालें",
    "परिणाम: अपवाहित URLs",
    "परिणाम: URL भाग",
  ]} >
```js title="extract-urls.js" frame="code"
const text = `
Check this out: https://example.com/path?query=123#section
And also (ftp://files.server.org/index).
Plus a weird one: custom-scheme://host/param;weird^stuff
`;

const urlRegex =
  /([-.a-z0-9]+:\/{1,3})([^-\/\.[\](|)\s?][^`\/\s\]?]+)([-_a-z0-9!@$%^&*()=+;/~\.]*)[?]?([^#\s`?]*)[#]?([^#\s'"`\.,!]*)/gi;

const matches = [
  ...text.matchAll(urlRegex),
].map((match) => match[0]);
console.log("अपवाहित URLs:", matches);

const parts = [
  ...text.matchAll(urlRegex),
].map((match) => match.slice(1));
console.log("अपवाहित भाग:", parts);
```

  "https://example.com/path?query=123#section",
  "ftp://files.server.org/index",
  "custom-scheme://host/param;weird^stuff"
]
```

  [
    "https://",    // प्रोटोकॉल
    "example.com", // डोमेन
    "/path",       // पथ
    "query=123",   // प्रश्न
    "section"      // अंश
  ],
  [
    "ftp://",           // प्रोटोकॉल
    "files.server.org", // डोमेन
    "/index",           // पथ
    "",                 // प्रश्न
    ""                  // अंश
  ],
  [
    "custom-scheme://",   // प्रोटोकॉल
    "host",               // डोमेन
    "/param;weird^stuff", // पथ
    "",                   // प्रश्न
    ""                    // अंश
  ]
]
```

## ☑️ अगले कदम

अपने उपयोग के मामले के आधार पर, आपको इस रेगेक्स को सुधारने की आवश्यकता हो सकती है या अधिक वैधता और पोस्ट-प्रोसेसिंग कदम जोड़े।

### अलग-अलग परियोजनाएँ, अलग-अलग आवश्यकताएँ

परियोजनाओं में आवश्यकताएँ और सुरक्षा चिंताएँ भिन्न होती हैं:

1. **वेब स्क्रैपिंग**: URLs की जाँच करें ताकि यह सुनिश्चित हो कि वे पहुँचे जा सकते हैं और भरोसेमंद हैं।  
2. **डेटा प्रोसेसिंग**: उपयोगकर्ता-जनित सामग्री से URLs निकालें जबकि सुरक्षा की गारंटी दें।  
3. **डेटा विश्लेषण**: अनुसंधान या विपणन उद्देश्यों के लिए डुप्लिकेट या अप्रासंगिक लिंक फ़िल्टर करें।  
4. **उपयोगकर्ता-अभिमुखीकृत एप्लिकेशन**: चैट ऐप या फ़ोरम में URLs को स्वचालित रूप से हाइपरलिंक करें।  

### पोस्ट-प्रोसेसिंग और वैधता  

संभावित URLs एकत्रित करने के बाद, अतिरिक्त जाँच लागू करें:  

- **DNS लुकअप**: यह सुनिश्चित करें कि डोमेन हल हो रहे हैं।  
- **सुरक्षा जाँच**: नुकसानदायक या फिशिंग साइटों की जाँच करने के लिए सेवाओं का उपयोग करें।  
- **कस्टम नियम**: परियोजना-विशिष्ट फ़िल्टर लागू करें (जैसे, अनुमत TLD, अधिकतम URL लंबाई)।  

## 📝 सारांश  

अर्ध-संरचित स्ट्रिंग डेटा निकालना रेगेक्स में महारत हासिल करने का सबसे संतोषजनक हिस्सा हो सकता है।

मुख्य बिंदुओं की एक झलक:  

- **[रेगेक्स पैटर्न](https://regex101.com/r/jO8bC4/69) लिखने, टेस्ट करने और समझने के लिए एक विजुअल टूल का उपयोग करें।**  
- **चुनौतियों को भागों में तोड़ें** और प्रत्येक भाग को अलग-अलग हल करें। एक तरह से, कैप्चर ग्रुप हमारे रेगेक्स के लिए आकर्षक 'संकेत चिह्न' प्रदान करते हैं।  
- **डेटा इंजेस्ट करते समय 'लूज' मैच एक्सप्रेशन का उपयोग करें, सख्त विनियमों की अनुपालन टालें।**  
- **प्रारंभिक निकाल के बाद वैधता कदम लागू करना आवश्यक है** – हमेशा अपनी परियोजना की सुरक्षा और विशिष्ट आवश्यकताओं को ध्यान में रखें।  

इन चरणों का पालन करके, आप किसी भी अर्ध-संरचित स्ट्रिंग डेटा को प्रभावी ढंग से निकाल सकते हैं, जिससे आगे की प्रक्रिया और वैधता के लिए आधार बनाया जा सकता है।
