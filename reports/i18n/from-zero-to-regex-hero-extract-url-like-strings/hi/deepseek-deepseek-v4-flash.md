## 🚀 परिचय

कच्चे पाठ से URL निकालना कभी-कभी एक थकाऊ 'व्हैक-ए-मोल' खेल जैसा लग सकता है। विराम चिह्न, कोष्ठकों में लिपटे हुए टुकड़े, और अस्पष्ट स्वरूपण—ये सब मिलकर आपके प्रयासों को विफल करने की साजिश रचते हैं। चाहे आप वेब स्क्रैपर, डेटा विश्लेषक, या चैट एप्लिकेशन बना रहे हों, URL को सटीक रूप से निकालना आवश्यक है।

इस पोस्ट में, हम समस्या का सीधा मुकाबला एक लचीले, दो-चरणीय दृष्टिकोण से करेंगे। हमारा लक्ष्य है **पहले सभी _संभावित_ URL-जैसी स्ट्रिंग्स को कैप्चर करना** और फिर बाद की प्रक्रिया में मान्यकरण करना।

> 💡 **नोट:** यह पैटर्न URL को **_मान्य_** करने के लिए नहीं है! यह जानबूझकर विराम चिह्नों और गलत वर्तनी के प्रति उदार है।

## 🔍 लक्ष्य: पाठ से URL निकालना

जब कच्चे पाठ से URL निकालते हैं, तो दो-चरणीय दृष्टिकोण प्रभावी होता है:

1. **सभी URL-जैसी चीज़ों को कैप्चर करें**: एक बड़ा जाल बिछाएँ ताकि उन सभी स्ट्रिंग्स को पकड़ा जा सके जो *संभावित* URL हो सकती हैं। यहीं पर हमारा "120+ बाइट रेगेक्स" चमकता है।
2. **मान्य करें**: एक बार जब आप इन उम्मीदवारों को कैप्चर कर लें, तो अमान्य प्रविष्टियों को हटाने के लिए द्वितीयक जाँचों (जैसे DNS रिज़ॉल्यूशन, ज्ञात डोमेन से तुलना) का उपयोग करें।

## 120+ बाइट रेगेक्स

नीचे एक संक्षिप्त रेगेक्स दिया गया है जो एक ही बार में URL निकालने और पार्स करने के लिए डिज़ाइन किया गया है। यह विभिन्न प्रोटोकॉल, डोमेन, पथ, और वैकल्पिक क्वेरी/फ़्रैगमेंट अनुभागों का समर्थन करता है।

चिंता न करें—हम इसे चरण दर चरण समझाएंगे!

```js title="120+ Byte URL Regex" frame="code"
const urlRegex = /([-.a-z0-9]+:\/{1,3})([^-\/\.[\](|)\s?][^`\/\s\]?]+)([-_a-z0-9!@$%^&*()=+;/~\.]*)[?]?([^#\s`?]*)[#]?([^#\s'"`\.,!]*)/gi;
// Compatibility: ES5+

// Same pattern, split on newlines for readability:
([-.a-z0-9]+:\/{1,3})
([^-\/\.[\](|)\s?][^`\/\s\]?]+)
([-_a-z0-9!@$%^&*()=+;/~\.]*)
[?]?([^#\s`?]*)
[#]?([^#\s'"`\.,!]*)

```

## 🧩 चरण दर चरण समझना

आइए regex को उसके घटकों में विभाजित करके समझें कि यह कैसे काम करता है:

<h3>1. प्रोटोकॉल (समूह 1): <code>{`([-.a-z0-9]+:\/{1,3})`}</code></h3>

<ul>
  <li>**उद्देश्य:** URL के प्रोटोकॉल भाग को मैच करता है (जैसे `http://`, `ftp://`, `custom-scheme://`).</li>
  <li>
    **व्याख्या:**
    <ul>
      <li><code>[-.a-z0-9]+</code>: एक या अधिक लोअरकेस अक्षरों, अंकों, हाइफ़न या डॉट्स को मैच करता है (प्रोटोकॉल स्कीम में सामान्य).</li>
      <li><code>{`:\/{1,3}`}</code>: एक कोलन के बाद एक से तीन स्लैश (<code>:/</code>, <code>://</code>, या <code>:///</code>) को मैच करता है.</li>
    </ul>
  </li>
</ul>

<h3>2. डोमेन (समूह 2): <code>{`([^-\/\.[\](|)\s?][^\`\/\s\]?]+)`}</code></h3>

<ul>
  <li>**उद्देश्य:** URL के डोमेन या होस्ट भाग को कैप्चर करता है.</li>
  <li>
    **व्याख्या:**
    <ul>
      <li><code>[^-\/\.[\](|)\s?]</code>: निर्दिष्ट विशेष वर्णों और व्हाइटस्पेस को छोड़कर किसी भी वर्ण को मैच करता है.</li>
      <li><code>[^`\/\s\]?]+</code>: बैकटिक, स्लैश, व्हाइटस्पेस या बंद वर्गाकार ब्रैकेट को छोड़कर एक या अधिक वर्णों को मैच करता है.</li>
    </ul>
  </li>
</ul>

<h3>3. पथ (समूह 3): <code>{`([-_a-z0-9!@$%^&*()=+;/~\\.]*)`}</code></h3>

<ul>
  <li>**उद्देश्य:** URL के पथ घटक को मैच करता है।</li>
  <li>
    **व्याख्या:**
    <ul>
      <li><code>[-_a-z0-9!@$%^&*()=+;/~\.]*</code>: पथों में सामान्यतः पाए जाने वाले शून्य या अधिक URL-सुरक्षित वर्णों को मैच करता है।</li>
    </ul>
  </li>
</ul>

<h3>4. क्वेरी (समूह 4): <code>[?]?([^#\s`?]*)</code></h3>

<ul>
  <li>**उद्देश्य:** वैकल्पिक रूप से किसी <code>?</code> वर्ण से शुरू होने वाली क्वेरी स्ट्रिंग को मैच करता है।</li>
  <li>
    **व्याख्या:**
    <ul>
      <li><code>[?]?</code>: वैकल्पिक रूप से <code>?</code> को मैच करता है। (वर्गाकार ब्रैकेट सख्ती से आवश्यक नहीं हैं, हालांकि वे अति संक्षिप्त दोहरे <code>??</code> की तुलना में थोड़े अधिक स्पष्ट हैं। यह समान अगले मैचिंग समूह <code>[#]?</code> के लिए एक दृश्य समानता भी प्रदान करता है।)</li>
      <li><code>([^#\s`?]*)</code>: शून्य या अधिक ऐसे वर्णों को मैच करता है जो हैश, व्हाइटस्पेस, बैकटिक या प्रश्न चिह्न नहीं हैं।</li>
    </ul>
  </li>
</ul>

<h3>5. फ़्रैगमेंट (समूह 5): <code>[#]?([^#\s'"`\.,!]*)</code></h3>

<ul>
  <li>**उद्देश्य:** वैकल्पिक रूप से <code>#</code> से शुरू होने वाले फ़्रैगमेंट पहचानकर्ता को मैच करता है।</li>
  <li>
    **व्याख्या:**
    <ul>
      <li><code>[#]?</code>: वैकल्पिक रूप से <code>#</code> को मैच करता है।</li>
      <li><code>([^#\s'"`\.,!]*)</code>: शून्य या अधिक ऐसे वर्णों को मैच करता है जो निषिद्ध विराम चिह्न या व्हाइटस्पेस नहीं हैं।</li>
    </ul>
  </li>
</ul>

## ☑️ अगले कदम

आपके उपयोग के मामले के आधार पर, आपको इस रेगेक्स को परिष्कृत करने या अधिक सत्यापन और पोस्ट-प्रोसेसिंग चरण जोड़ने की आवश्यकता हो सकती है।

### अलग-अलग प्रोजेक्ट, अलग-अलग ज़रूरतें

प्रोजेक्ट्स की अलग-अलग आवश्यकताएँ और सुरक्षा संबंधी चिंताएँ होती हैं:

1. **वेब स्क्रैपिंग**: यूआरएल को मान्य करें ताकि वे पहुंच योग्य और विश्वसनीय हों।
2. **डेटा प्रोसेसिंग**: उपयोगकर्ता-निर्मित सामग्री से यूआरएल निकालें, साथ ही सुरक्षा सुनिश्चित करें।
3. **डेटा विश्लेषण**: शोध या मार्केटिंग उद्देश्यों के लिए डुप्लिकेट या अप्रासंगिक लिंक को फ़िल्टर करें।
4. **उपयोगकर्ता-सामना करने वाले एप्लिकेशन**: चैट ऐप या फ़ोरम में यूआरएल को स्वचालित रूप से हाइपरलिंक करें।

### पोस्ट-प्रोसेसिंग और सत्यापन

संभावित यूआरएल एकत्र करने के बाद, अतिरिक्त जाँच लागू करें:

- **DNS लुकअप**: सत्यापित करें कि डोमेन हल होते हैं।
- **सुरक्षा जाँच**: दुर्भावनापूर्ण या फ़िशिंग साइटों की जाँच के लिए सेवाओं का उपयोग करें।
- **कस्टम नियम**: प्रोजेक्ट-विशिष्ट फ़िल्टर लागू करें (जैसे, अनुमत TLD, अधिकतम URL लंबाई)।

## 📝 सारांश

अर्ध-संरचित स्ट्रिंग डेटा निकालना शायद रेगेक्स निपुणता का सबसे संतोषजनक हिस्सा है।

यहाँ मुख्य बातों का सारांश दिया गया है:

- **रेगेक्स पैटर्न लिखने, परीक्षण करने और समझने के लिए विज़ुअल टूल का उपयोग करें** — [Regex101](https://regex101.com/r/jO8bC4/69) देखें।
- **चुनौती को भागों में बाँटें** और प्रत्येक भाग को अलग-अलग हल करें। एक अर्थ में, कैप्चर ग्रुप हमारे रेगेक्स के लिए आलंकारिक 'पथ-चिह्न' प्रदान करते हैं।
- **डेटा इंजेशन करते समय 'ढीले' मिलान एक्सप्रेशन का उपयोग करें**, सख्त स्पेक अनुरूपता से बचें।
- **प्रारंभिक निष्कर्षण के बाद सत्यापन चरण लागू करना** आवश्यक है—अपने प्रोजेक्ट की सुरक्षा और विशिष्ट आवश्यकताओं पर हमेशा विचार करें।

इन चरणों का पालन करके, आप किसी भी अर्ध-संरचित स्ट्रिंग डेटा को प्रभावी ढंग से निकाल सकते हैं, जो आगे की प्रक्रिया और सत्यापन की नींव रखता है।
