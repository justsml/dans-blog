> Если процесс работает от вашего имени, он не должен автоматически наследовать всё, что вам разрешено делать.

Это и есть проблема «взрывного радиуса» рабочей станции.

## 1. Помещайте проекты в Dev Containers

[Dev Containers](https://github.com/devcontainers/spec) позволяют использовать контейнер как полнофункциональную среду разработки. Это звучит как инфраструктура для улучшения опыта разработчиков — и так оно и есть. Но при правильном подходе это также граница безопасности.

Ленивая конфигурация монтирует слишком много:

```jsonc
// Слишком удобно. Слишком большой радиус поражения.
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "mounts": [
    "source=${localEnv:HOME},target=/host-home,type=bind"
  ]
}
```

В результате контейнер превращается в странно‑формированную копию вашей учетной записи хоста.

Используйте узкие монтирования:

```jsonc
// .devcontainer/devcontainer.json
{
  "name": "app",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-22",
  "workspaceFolder": "/workspaces/app",
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/app,type=bind,consistency=cached"
  ],
  "containerEnv": {
    "NODE_ENV": "development"
  },
  "postCreateCommand": "bun install"
}
```

Это не идеальная изолированная среда. Контейнеры разделяют ядро. Docker имеет свои острия. Монтирования могут пробивать дырки сквозь модель.

Но для большинства рабочих процессов разработчиков выгода очевидна: команды проекта видят только проект, а не ваш весь цифровой чердак.

### Что монтировать

Монтировать репозиторий.

Возможно, монтировать кэш, специфичный для проекта.

Не монтируйте их по умолчанию:

- `~/.ssh`
- `~/.aws`
- `~/.config/gcloud`
- `~/.azure`
- `~/Downloads`
- `~/Documents`
- экспорты менеджеров паролей
- дампы баз данных
- папки резервных копий
- произвольные «temp»‑папки, существующие с 2021 года

Если проект требует доступа к облаку, внедрите учётные данные, созданные специально для этого проекта. Краткоживущие предпочтительнее. Только для чтения — ещё лучше. Токен, который может работать лишь с учёткой разработчика, лучше, чем ваша личная административная идентификация, блуждающая в контейнере с крошечным чемоданчиком.

### Инструменты ИИ‑кодинга тоже сюда относятся

Инструменты ИИ‑кодинга делают Dev Containers ещё более важными, а не менее.

Документация по разрешениям [Claude Code от Anthropic](https://code.claude.com/docs/en/permissions) делит мир на разрешения и изоляцию: разрешения управляют инструментами, файлами и доменами; изоляция обеспечивает принудительное выполнение на уровне ОС для доступа Bash к файловой системе и сети.

Это различие — всё дело.

Если агент может выполнять shell‑команды, устанавливать пакеты, просматривать файлы и следовать инструкциям, помещайте эту работу в ограниченную среду проекта. Хост оставляйте простым.

Хороший вариант по умолчанию:

- запускайте агент в репозитории, а не в домашнем каталоге  
- явно запрещайте доступ к чувствительным путям  
- используйте Dev Container для команд установки/сборки/тестов  
- избегайте добавления широких «дополнительных каталогов» в контекст  
- проверяйте любые сгенерированные команды, которые работают с учётными данными, конфигурацией аутентификации, публикацией пакетов или облачными ресурсами  

Модели не нужен ваш каталог `~/Documents`, чтобы исправить ошибку TypeScript.

## 2. Замените открытый текст в `.env` на более безопасный вариант

`.env` файлы не являются злом.

Это всего лишь файлы. В этом и заключается проблема.

Файлы копируются. Файлы индексируются. Файлы монтируются. Файлы читаются скриптами, которым изначально нужно было только проверять CSS. Файлы попадают в отладочные zip‑архивы. Файлы вставляются в чат, потому что кто‑то попросил помощи и забыл последние двенадцать строк.

Используйте скучную иерархию:

1. Секрет не нужен: поместите значение в `.env.example`.
2. Секрет только для локального использования: шифруйте его в состоянии покоя.
3. Общий секрет разработки: храните его в настоящем менеджере секретов или менеджере паролей.
4. Секрет продакшна: не размещайте его на ноутбуках разработчиков, если только нет очень специфической причины.

[VarLock](https://varlock.dev/guides/secrets/) привлекателен тем, что делает чувствительность явной. В его документации описано помечать значения как `@sensitive`, шифровать локальные значения с помощью `varlock()`, скрывать чувствительные значения в выводе консоли и сканировать файлы проекта на предмет вхождений открытого текста известных чувствительных значений.

Форма лучше, чем «запустить регулярное выражение по репозиторию и надеяться, что секрет выглядит как секрет».

Пример направления:

```dotenv
# .env.schema
# @defaultSensitive=false

PUBLIC_APP_NAME=

# @sensitive
STRIPE_SECRET_KEY=

# @sensitive
DATABASE_URL=
```

Локальное переопределение:

```dotenv
# .env.local
PUBLIC_APP_NAME=demo
STRIPE_SECRET_KEY=varlock(local:...)
DATABASE_URL=varlock(local:...)
```

Это не делает секреты безопасными после загрузки в скомпрометированный процесс. Ничего не делает. Но это значит, что в файловой системе остаётся меньше открытого текста.

Это важно против инфостилеров, вредоносных зависимостей, слишком широкого контекста ИИ, случайных коммитов и скромного момента `console.log(process.env)`.

## 3. Добавьте канарейковые токены туда, где бы воровал бы злоумышленник


Большинство систем мониторинга сообщает вам, когда произошло что‑то известное как плохое.

Канарейковые токены сигнализируют, когда что‑то странное коснулось объекта, о котором он не должен был знать.

[Thinkst Canarytokens](https://help.canary.tools/hc/en-gb/articles/10905485310109-Canarytoken-Overview-and-Use-Cases) описывает их как цифровые ловушки. Это могут быть документы, URL‑адреса, API‑ключи, VPN‑профили, QR‑коды и другие поддельные ресурсы, которые вызывают оповещение при доступе.

Размещение — это искусство.

Не рассыпайте случайные приманки и не объявляйте победу. Размещайте канарейки там, где естественно окажутся кража учётных данных, кража резервных копий или разведка.

### Локальные канарейки

Создайте поддельную резервную копию:

```text
~/backups/customer-prod-export-2024.sql
```

Поместите в неё URL‑канарейку или токен:

```sql
-- legacy analytics webhook
-- https://canarytokens.example.invalid/static/abc123
```

Создайте поддельный файл учётных данных:

```text
~/Documents/passwords-old.csv
```

Или поддельный профиль AWS:

```ini
# ~/.aws/credentials
[billing-prod-legacy]
aws_access_key_id = AKIA...
aws_secret_access_key = ...
```

По возможности используйте реальный тип токена‑канарейки AWS, чтобы оповещение срабатывало при попытке использования, а не только при открытии файла.

### Канарейки в репозитории

Разместите канарейки рядом с теми местами, которые злоумышленники проверяют после получения доступа к исходному коду:

- внутренние runbook‑ы
- устаревшая документация по развертыванию
- старые заметки о миграциях
- фиктивные учётные данные сервиса в явно не‑продакшн‑файле `.env.canary`
- поддельные инструкции по восстановлению из резервной копии

Это не «безопасность через скрытность». Это сигнал тревоги в коридоре.

### CI и облачные канарейки

Хорошие места для облачных ловушек:

- поддельный CI‑секрет
- поддельный токен развертывания
- поддельный пользователь базы данных без привилегий
- неиспользуемый путь к объектному хранилищу
- поддельный kubeconfig
- поддельный API‑ключ, задокументированный в runbook‑е

Сделайте оповещение практичным. Канарейка, которая просто шлёт письмо в неотслеживаемый ящик, — лишь декоративный элемент.

Как минимум, оповещение должно содержать:

- какой токен сработал  
- где он был размещён  
- какая система с ним взаимодействовала  
- что необходимо обновить  
- кто отвечает за реакцию

## 4. Ограничьте исходящий трафик

Если на локальной машине запущен вредоносный код, для вывода данных нужен сетевой путь.

Большинство ноутбуков разработчиков по умолчанию разрешают исходящий трафик. Это удобно. Но это также означает, что неизвестный процесс может отправлять данные в неизвестное место без локального контрольного пункта.

Исходящие файрволы — это слой ремня безопасности.

Они не остановят каждый сбой. Они сделают некоторые сбои выживаемыми. Они также будут жаловаться в неудобные моменты, пока вы не научите их, как выглядит нормальное поведение.

### macOS

[LuLu](https://objective-see.org/products/lulu.html) — бесплатный и открытый. Objective‑See описывает его как блокирующий неизвестные исходящие соединения, и в документации отмечается, что LuLu мониторит только исходящий трафик.

Это хороший первый вариант, если вам нужны простые запросы на исходящие соединения и вы готовы мириться с небольшим трением при настройке.

[Little Snitch](https://obdev.at/products/littlesnitch/) — коммерческий и более отполированный. Он выводит оповещения о соединениях, позволяет разрешать или блокировать соединения приложений и предоставляет сетевой монитор с видимостью по приложению, домену, стране, порту, протоколу и объёму трафика.

Это болеенадёжный вариант, если вам нужны профили, управление правилами и интерфейс, которым люди действительно будут пользоваться после второй недели.

### Windows

Брандмауэр Windows Defender поддерживает правила исходящего трафика и приоритет правил для входящего и исходящего трафика. Руководство Microsoft предельно практично: изменение правил исходящего трафика на «заблокировано» может рассматриваться в средах с высоким уровнем безопасности, но требует инвентаризации приложений и создания правил для тех, кто действительно нуждается в сетевом подключении.

Перевод: возможно, мощно и легко раздражать.

[Portmaster](https://safing.io/) также стоит оценить в Windows. Safing описывает его как открытый межсетевой экран приложений, который отслеживает сетевые соединения и задаёт правила блокировки для каждого приложения.

### Linux

Portmasterподдерживает распространённые пакеты Linux. OpenSnitch — ещё один межсетевой экран приложений для Linux, который стоит оценить, хотя перед стандартизацией следует проверить состояние проекта и наличие пакетов в дистрибутивах.

Для серверов используйте обычные серверные средства управления. Для ноутбуков разработчиков ключевая функция — видимость на уровне приложений. «Блокировать весь исходящий трафик, кроме 443» недостаточно, когда каждый интересный путь утечки данных также использует 443.

## 5. Дайте резервным копиям взрослый надзор

Резервныекопии не являются «холодными». Это чувствительные данные в переносимом виде.

Рабочие станции разработчиков не должны превращаться в архивы резервных копий, если только это не их прямая задача.

Правила, которые я действительно бы применил:

- Для экспортов в продакшн требуется указать владельца и дату истечения срока.
- Локальные дампы баз данных должны быть зашифрованы.
- Любой экспорт, содержащий учётные данные, инициирует их ротацию или очистку.
- Папки резервных копий по умолчанию не монтируются в Dev Containers.
- Папки резервных копий по умолчанию недоступны инструментам AI‑кодинга.
- По крайней мере один канарейка должна находиться в хранилище, похожем на резервную копию.
- Старые экспорты удаляются автоматикой, а не «по настроению».

Простая локальная конвенция:

```bash
mkdir -p ~/sensitive-exports
chmod 700 ~/sensitive-exports
```

Более продвинутая конвенция:

- зашифрованный том или зашифрованный архив  
- чёткое именование с указанием срока действия  
- задокументированное удаление  
- без синхронизации в облачные хранища потребителей, если только нет одобрения  

Пример:

```bash
age -r age1yourpublickeyhere -o customer-export-2026-05-09.sql.gz.age customer-export.sql.gz
shred -u customer-export.sql.gz
```

Не превращайте это в ритуал. Лучшая политика резервного копирования — та, при которой разработчикам редко приходится запрашивать экспорт из продакшена.

## 6. Создайте базовый набор настроек рабочей станции

Ниже предложен разумный базовый набор для отдельного разработчика:

| Область | Базовый набор |
| --- | --- |
| Браузер | Не сохранять пароли продакшн. Пользоваться менеджером паролей и аппаратным MFA для важных аккаунтов. |
| Проекты | Использовать Dev Containers для проектов с установкой пакетов, недоверенным кодом или AI‑управляемой работой в оболочке. |
| Секреты | Не хранить открытые продакшн‑секреты на диске. При возможности шифровать локальные dev‑секреты. |
| Облако | Краткоживущие учётные данные. Разделять идентичности dev и prod. По умолчанию не использовать личный токен администратора. |
| GitHub | Токены с тонкой гранулярностью. Проверять токены публикации пакетов. Использовать SSO организации и аппаратные ключи. |
| AI‑инструменты | Доступ, ограниченный проектом, запрещать доступ к чувствительным путям, запускать команды в контейнерах, когда это практично. |
| Резервные копии | Шифровать, задавать срок действия, изолировать и мониторить. Держать их вне широких монтирований и контекста AI. |
| Сеть | Исходящий firewall в режиме оповещения или мониторинга сначала, затем правила для рискованных инструментов. |
| Обнаружение | Канарейковые токены в резервных копиях, учётных данных, CI, облаке и местах документации. |

Для команды добавить:

- стандартный шаблон `.devcontainer`
- политику секретов, различающую локальные, общие для разработки, staging и prod
- конвенции размещения канарейковых токенов
- задокументированные профили исходящего firewall
- быстрые сценарии ротации учётных данных
- вводный материал, объясняющий модель угроз без излишних драматизаций

Цель — не превратить каждого разработчика в инженера‑по‑безопасности.

Цель — сделать более безопасный путь обычным.

## Что сделать на этой неделе

Если это кажется слишком масштабным, выполните пять действий:

1. Выберите один репозиторий с высоким уровнем риска и добавьте в него Dev Container с ограниченными монтированиями.  
2. Перенесите один открытый секрет из файла `.env.local` в зашифрованное локальное хранилище или менеджер паролей.  
3. Разместите один канарейский токен в поддельном файле резервной копии и направьте оповещения в видимое место.  
4. Установите LuLu, Little Snitch, Portmaster или аналогичный инструмент в режим мониторинга и посмотрите, с кем действительно происходит обмен данными.  
5. Найдите локальные экспортированные данные продакшн и либо удалите их, либо зашифруйте, либо задайте срок действия.

Этого достаточно, чтобы начать.

Работа по безопасности часто проваливается, потому что её пытаются построить как собор. Сначала установите дверь. Затем замок. Затем сигнализацию. Затем привычку.

Рабочая станция не обязана быть полностью доверенной.

Она должна перестать быть бесконечно доверенной по ошибке.

Возможные варианты обложки:

- Диаграммная карта: ноутбук в центре, окружённый четырьмя ограниченными кольцами с подписями изоляция, секреты, обнаружение и исход. Лучше всего подходит для практического руководства.
