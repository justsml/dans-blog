大多数站点搜索的决策都来得太晚。

等到有人说“我们应该使用 Algolia”时，团队通常已经跳过了一个关键问题：我们到底在搜索什么内容？

如果答案是“我们已经构建好的 HTML 页面”，[Pagefind](https://pagefind.app/) 应该是你首先尝试的工具。并不是说 Algolia 不好，Algolia 在许多棘手问题上表现非常出色。但如果你的搜索索引随站点部署而变化，托管搜索服务可能只是一层基础设施的“cosplay”。

<p class="inset">当你的可搜索内容在构建时生成时使用 Pagefind。当搜索需要接受实时写入、业务规则、用户特定排序或静态构建无法提供的运营保证时，再考虑 Algolia。</p>

这个规则适用的站点比人们想象的要多：博客、文档、营销站点、内部手册、产品指南、课程目录，以及大量主要发布页面的“应用”。

## 问题的形态

Algolia 为你提供一个外部搜索系统。你创建记录、推送到索引、配置排序、接入 UI，并且要保持它与真实数据源同步。

Pagefind 则直接读取你已经发布的 HTML，并在旁边生成一个静态搜索索引。

这个区别听起来乏味，直到你需要维护集成时才会感受到差异。

使用 Algolia 时，你的站点会拥有内容的第二份拷贝。此时你需要回答诸如以下问题：

- 部署完成但索引更新失败了？
- 哪些字段是规范的：CMS 字段、渲染后的页面，还是搜索记录？
- 当排名微调不再匹配页面时，谁来负责？
- 当免费层并非你的真实流量形态时会怎样？

有时这些问题值得深究。对于市场平台、支持门户或大型电商目录，它们几乎是必不可少的。对于静态文档站点，这类问题往往是自找的复杂度。

## Pagefind 能工作是因为它拒绝了额外系统

Pagefind 的技巧并非魔法，而是品味。

它等到页面已经生成，索引完成的 HTML，并生成一组可以和站点其余资源一起部署到同一 CDN 的静态资产。浏览器只下载它需要的块。没有需要保温的搜索服务器，没有爬虫配额需要监控，也没有 webhook 流水线去记忆哪些内容变更。

这让故障模式更易理解：

- 如果页面已部署，索引内容就来源于该页面。
- 如果页面未部署，用户根本看不到它。
- 如果搜索结果错误，问题通常出在你的渲染标记或 Pagefind 配置，而不是远端的同步任务。

这也是我偏爱它用于内容站点的原因。索引紧随产出。

## 实际的设置长什么样

对于普通的静态站点，工作流相当平淡：

- **CLI**：扫描站点的 HTML 文件，生成索引，并在几分钟内将其部署到全局 CDN。
- **静态站点生成器**：使用 PageFind 为 Astro 或 Hugo 提供的插件，实现索引过程的自动化。
- **自定义方案**：利用 PageFind 的 API 构建符合你独特需求的定制搜索体验。

## Algolia 仍然占优势的场景

Pagefind 并不是穿着风衣的微型 Algolia。它提供的是另一种解答。

当你的搜索索引需要独立于站点部署而变化时，使用 Algolia、OpenSearch、Postgres 搜索或其他实时系统。

这包括：

- 每几分钟就会变动的库存数量
- 基于用户的权限或私有结果
- 受收入、时效、热度或实验驱动的自定义排序
- 跨系统的联合搜索，这些系统的内容并未渲染成单一的静态站点
- 业务方期望的由托管供应商提供的分析与运维支持

这些都是实际需求。因为 Pagefind 快就假装它能满足这些需求，只会沦为另一种供应商博客的口吻。

## 我采用的决策方法

先问自己一个问题：

> 能否从用户正在浏览的同一套静态输出中重新构建搜索索引？

如果答案是肯定的，就直接使用 Pagefind。它默认私有化搜索、友好的 CDN 资源，并且省去一个需要额外配置的服务账号。

如果答案是否定的，明确导致索引需要实时更新的因素：库存、权限、个性化、分析、排序或写入频率。然后选择专门负责这些任务的数据库或搜索服务。

Algolia 并不是这里的反派。真正的反派是，在尚未证明首个方案不足之前，就匆忙引入第二套系统。
