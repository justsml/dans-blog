<p class="inset">Closures sind eines dieser Konzepte, von denen jeder JavaScript-Entwickler *glaubt*, sie zu verstehen – bis zu dem Moment, wo man es nicht tut. Dieses Quiz findet die Grenzen dessen, was du wirklich weißt.</p>

Closures tauchen in Event-Handlern, Factory-Funktionen, React-Hooks, Modul-Patterns und so ziemlich überall auf. Sie sind nicht exotisch. Sie sind das Gewebe, aus dem JavaScript gewebt ist.

Die Fragen beginnen vernünftig und fangen dann an, die Dielenbretter zu entfernen.

    {text: '1, 2, 3, 4, 5'},

>
  <slot name="question">
  <div className="question">
Was wird nach 100ms geloggt?
```javascript
    for (var i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100);
    }
```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
`var` ist funktionsbezogen (function-scoped). Es gibt genau **ein** `i` – alle fünf Callbacks schließen über dieselbe Bindung. Wenn die Timeouts feuern, ist die Schleife bereits beendet und `i` ist `5`.

    Fünf Callbacks. Ein `i`. Wert: `5`. Fünfmal geloggt.

    Die Lösung ist `let` (block-scoped, neue Bindung pro Iteration) oder das Einwickeln in eine IIFE, um den aktuellen Wert einzufangen:
```javascript
    for (let i = 0; i < 5; i++) {
      setTimeout(() => console.log(i), 100); // 0, 1, 2, 3, 4 ✓
    }
```

>
  <slot name="question">
  <div className="question">
Was wird hier geloggt?
```javascript
    function wrap() {
      const obj = { x: 1 };
      const getObj = () => obj;
      obj.x = 99;
      return getObj;
    }
    const get = wrap();
    console.log(get(), get());
```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
Die Closure erfasst die **Bindung** `obj`, nicht eine Momentaufnahme des Objekts. Das Objekt selbst wird nach der Erstellung der Closure mutiert – aber `obj` zeigt immer noch auf dieselbe Referenz. Beide Aufrufe geben dasselbe mutierte Objekt zurück: `{ x: 99 }`.

    Das überrascht Leute, die annehmen, dass Closures Werte "einfrieren". Tun sie nicht. Sie erfassen **Referenzen**.
  </div>
  </slot>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={5}
  group="Fortgeschritten"
  title="Stale Closure in React"
  options={[
    {text: 'Protokolliert den aktuellen Zähler bei jedem Klick'},
    {text: 'Protokolliert immer 0', isAnswer: true},
    {text: 'Protokolliert NaN'},
    {text: 'Das lässt sich nicht kompilieren'},
  ]}
>
  <slot name="question">
  <div className="question">
Welches Verhalten zeigt diese React-Komponente, wenn der Button mehrmals geklickt wird?
```jsx
    function Counter() {
      const [count, setCount] = useState(0);

      useEffect(() => {
        const id = setInterval(() => {
          console.log('count is', count);
        }, 1000);
        return () => clearInterval(id);
      }, []); // empty deps

      return <button onClick={() => setCount(c => c + 1)}>
        Click ({count})
      </button>;
    }
```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
Klassischer Stale Closure. Der `useEffect` wird einmal ausgeführt (leeres Abhängigkeitsarray) und erstellt ein Intervall, das `count` so einfängt, wie es zum Zeitpunkt des Mountens war: `0`. Egal wie oft du klickst, der Closure des Intervalls sieht niemals den aktualisierten Wert.

    Der Button-Label aktualisiert sich korrekt – das ist ein neuer Render. Aber `console.log` ist für immer bei `0` gefangen.

    Behebe es, indem du `count` zum Abhängigkeitsarray hinzufügst (das Intervall wird neu erstellt) oder einen Ref verwendest:
```jsx
    const countRef = useRef(count);
    useEffect(() => { countRef.current = count; }, [count]);
    // then use countRef.current inside the interval
```
  </div>
  </slot>
</Challenge>

Warte — lies genau. Was passiert eigentlich?
```javascript
    const bank = (() => {
      let balance = 0;
      return {
        deposit: (n) => (balance += n),
        withdraw: (n) => (balance -= n),
        getBalance: () => balance,
      };
```

    Das ist das **Modul-Pattern** — eine IIFE, die ein Objekt mit Closures zurückgibt. Alle drei Methoden schließen über dieselbe `balance`-Bindung. Es ist wirklich privat: Du kannst von außen nicht auf `balance` zugreifen.
```javascript
    console.log(bank.balance); // undefined
```
Vor den Klassen war das *der* Weg, um Zustand in JavaScript zu kapseln. Du wirst es immer noch in Legacy-Codebasen und Utility-Bibliotheken sehen.

  client:visible={{rootMargin: "150px"}}
  index={7}
  group="Experte"
  title="Closure-Speicherleck"
  options={[
    {text: 'Es protokolliert die Länge des Daten-Arrays'},
    {text: 'Es verursacht ein Speicherleck', isAnswer: true},
    {text: 'Es wirft einen RangeError'},
    {text: 'Nichts Ungewöhnliches – das ist in Ordnung'},
  ]}
>
  <slot name="question">
  <div className="question">
Was ist das Problem mit diesem Code?
```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      element.addEventListener('click', () => {
        console.log('clicked!', data.length);
      });
    }
```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
Der Event-Listener schließt `data` ein, sodass das Array erreichbar bleibt, solange der Listener erreichbar ist. Wenn `element` langlebig ist und der Listener nie entfernt wird, wird auch dieses große Array zurückgehalten.

    `element` ist im DOM. Der Listener hält `data`. Der GC kann es nicht einsammeln.

    Lösung: Erfasse keine großen Daten in langlebigen Listenern, es sei denn, du brauchst sie wirklich, und entferne Listener, wenn die zugehörige UI abgebaut wird:
```javascript
    function attachHandler(element) {
      const data = new Array(100_000).fill('🔥');
      // do something with data...

      const onClick = () => {
        console.log('clicked!');
      };

      element.addEventListener('click', onClick);
      return () => element.removeEventListener('click', onClick);
    }
```
Moderne V8 ist darin besser als früher. Aber in langlebigen SPAs mit vielen Event-Listenern beißt dieses Muster die Leute immer noch.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={8}
  group="Experte"
  title="Die this-Falle"
  options={[
    {text: 'Dan, Dan'},
    {text: 'Dan, undefined'},
    {text: 'undefined, Dan'},
    {text: 'undefined, undefined', isAnswer: true},
  ]}
>
  <slot name="question">
  <div className="question">
Was wird hier ausgegeben?
```javascript
    const user = {
      name: 'Dan',
      greetArrow: () => console.log(this?.name),
      greetRegular() {
        const inner = () => console.log(this?.name);
        inner();
      },
    };

    user.greetArrow();
    const fn = user.greetRegular;
    fn();
```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
Zwei subtile Fallen:

    **`greetArrow`**: Pfeilfunktionen haben kein eigenes `this`. Sie übernehmen `this` aus dem umschließenden lexikalischen Scope – hier ist das der Modul-/globale Scope, nicht `user`. Also ist `this?.name` `undefined`.

    **`fn()`**: `greetRegular` wird ohne Empfänger aufgerufen (`fn()` nicht `user.greetRegular()`). Im Strict Mode ist `this` `undefined`. Im Sloppy Mode ist es das globale Objekt. In beiden Fällen ist `this?.name` `undefined`.

    Wenn du `user.greetRegular()` direkt aufgerufen hättest, würdest du `'Dan'` erhalten – weil dann `this` an `user` gebunden ist.

    Ergebnis: `undefined, undefined`.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={9}
  group="Bonus"
  title="Memoization durch Closure"
  options={[
    {text: 'Ruft expensive() 3 Mal auf'},
    {text: 'Ruft expensive() 1 Mal auf', isAnswer: true},
    {text: 'Ruft expensive() 2 Mal auf'},
    {text: 'Wirft: Maximale Aufrufstapelgröße überschritten'},
  ]}
>
  <slot name="question">
  <div className="question">
Wie oft wird `expensive()` tatsächlich aufgerufen?
```javascript
    function memoize(fn) {
      const cache = new Map();
      return (arg) => {
        if (cache.has(arg)) return cache.get(arg);
        const result = fn(arg);
        cache.set(arg, result);
        return result;
      };
    }

    let calls = 0;
    const expensive = (n) => { calls++; return n * 2; };
    const fast = memoize(expensive);

    fast(5);
    fast(5);
    fast(5);
    console.log(calls);
```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
Die memoized-Funktion schließt über `cache`. Der erste Aufruf (`fast(5)`) verfehlt den Cache, ruft `expensive` auf und speichert das Ergebnis. Der zweite und dritte Aufruf treffen den Cache und geben sofort zurück.

    `calls` ist `1`.

    Das sind Closures, die echte Arbeit leisten. `cache` ist privat (kann von außerhalb von `memoize` nicht zugegriffen werden), persistent (lebt so lange wie `fast` existiert) und wird über alle Aufrufe von `fast` hinweg geteilt.
  </div>
  </slot>
</Challenge>
