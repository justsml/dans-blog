<p class="inset">Los closures son uno de esos conceptos que todo desarrollador JavaScript *cree* entender — hasta que llega el momento en que no lo hacen. Este quiz detectará los límites de lo que realmente sabes.</p>

Los closures aparecen en manejadores de eventos, funciones fábrica, hooks de React, patrones de módulos y, básicamente, en cualquier otro sitio. No son exóticos. Son la tela de la que está tejido JavaScript.

Las preguntas empiezan razonables y luego empiezan a arrancar tablas del piso.

    ¿Qué muestra esto?
    ```javascript

          let count = 0;

    ```
  </div>

  <slot name="question">
  <div className="question">
    ¿Qué muestra este registro?
    ```javascript
        function makeCounter() {
          let count = 0;
          return () => ++count;
        }
        const a = makeCounter();
        const b = makeCounter();
        console.log(a(), b());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Cada llamada a `makeCounter()` crea un **nuevo contexto de ejecución** con su propio `count` fresco. `a` y `b` cierran sobre *diferentes* vinculaciones de `count`. No comparten estado.

    Resultado: `1, 1`

    Por eso se usan los closures para construir estado privado: cada "instancia" es independiente.

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={2}
  group="La Trampa Clásica"
  title="var en unBucle"
  options={[
    {text: '0, 1, 2, 3, 4'},
    {text: '5, 5, 5, 5, 5', isAnswer: true},
    {text: '1, 2, 3, 4, 5'},
    {text: 'undefined x5'},
  ]}
>
  <slot name="question">
  <div className="question">
    ¿Qué se registra después de 100 ms?
    ```javascript
        for (var i = 0; i < 5; i++) {
          setTimeout(() => console.log(i), 100);
        }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `var` es de alcance de función. Hay exactamente **uno** `i` — los cinco callbacks cierran sobre la misma referencia. Cuando los timeouts se disparan, el bucle ya terminó y `i` vale `5`.

    Cinco callbacks. Un `i`. Valor: `5`. Registrado cinco veces.

    La solución es `let` (de alcance de bloque, nueva referencia en cada iteración) o envolver en un IIFE para capturar el valor actual:
    ```javascript
        for (let i = 0; i < 5; i++) {
          setTimeout(() => console.log(i), 100); // 0, 1, 2, 3, 4 ✓
        }
    ```
  </div>
  </slot>
</Challenge>

  <slot name="question">
  <div className="question">
    Mismo bucle, diferente palabra clave. ¿Qué se registra?

    ```javascript
        for (let i = 0; i < 5; i++) {

        }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `let` en un bucle `for` es especial. La especificación dice: cada iteración obtiene su propio **nuevo enlace** de `i`. Es como si el bucle secretamente hiciera esto:
    ```javascript
        { let i = 0; setTimeout(() => console.log(i), 100) }
        { let i = 1; setTimeout(() => console.log(i), 100) }
        // ...
    ```
    Así que los cinco callbacks cierran sobre cinco variables distintas. Resultado: `0, 1, 2, 3, 4`.
  </div>

  client:visible={{rootMargin: "150px"}}
  index={4}
  group="Clausuras y Objetos"
  title="Mutación de Objetos"
  options={[
    {text: '{ x: 1 }, { x: 1 }'},
    {text: '{ x: 1 }, { x: 99 }'},
    {text: '{ x: 99 }, { x: 99 }', isAnswer: true},
    {text: '{ x: 99 }, { x: 1 }'},
  ]}
>
  <slot name="question">
  <div className="question">
    ¿Qué imprime esto?
    ```javascript
        function wrap() {
          const obj = { x: 1 };
          const getObj = () => obj;
          obj.x = 99;
          return getObj;
        }
        const get = wrap();
        console.log(get(), get());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La clausura captura el **enlace** `obj`, no una instantánea del objeto. El objeto mismo se muta después de crear la clausura — pero `obj` sigue apuntando a la misma referencia. Ambas llamadas devuelven el mismo objeto mutado: `{ x: 99 }`.

    Esto atrapa a la gente que asume que las clausuras "congelan" valores. No lo hacen. Capturan **referencias**.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={5}
  group="Avanzado"
  title="Cierre obsoleto en React"
  options={[
    {text: 'Registra el recuento actual en cada clic'},
    {text: 'Siempre registra 0', isAnswer: true},
    {text: 'Registra NaN'},
    {text: 'Esto no compilará'},
  ]}
>
  <slot name="question">
  <div className="question">
    ¿Qué comportamiento muestra este componente de React cuando se hace clic en el botón varias veces?
    ```jsx
        function Counter() {
          const [count, setCount] = useState(0);

          useEffect(() => {
            const id = setInterval(() => {
              console.log('count is', count);
            }, 1000);
            return () => clearInterval(id);
          }, []); // empty deps

          return <button onClick={() => setCount(c => c + 1)}>
            Click ({count})
          </button>;
        }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Cierre obsoleto clásico. El `useEffect` se ejecuta una vez (array de dependencias vacío) y crea un intervalo que cierra sobre `count` tal como estaba al montar: `0`. No importa cuántas veces hagas clic, el cierre del intervalo nunca ve el valor actualizado.

    La etiqueta del botón se actualiza correctamente — eso es una renderización fresca. Pero `console.log` queda atrapado para siempre en `0`.

    Soluciona añadiendo `count` al array de dependencias (recrea el intervalo) o usando una ref:
    ```jsx
        const countRef = useRef(count);
        useEffect(() => { countRef.current = count; }, [count]);
        // then use countRef.current inside the interval
    ```
  </div>
  </slot>
</Challenge>

  index={6}
  group="Avanzado"
  title="Patrón de Módulo"
  options={[
    {text: '60', isAnswer: true},
    {text: 'TypeError: balance no está definido'},
    {text: '-40'},
    {text: '100'},
  ]}
>
  <slot name="question">
  <div className="question">
    Espera — lee con atención. ¿Qué ocurre realmente?
    ```javascript
        const bank = (() => {
          let balance = 0;
          return {
            deposit: (n) => (balance += n),
            withdraw: (n) => (balance -= n),
            getBalance: () => balance,
          };
        })();

        bank.deposit(100);
        bank.withdraw(40);
        console.log(bank.getBalance());
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La respuesta es `60`.

    Este es el **Patrón de Módulo** — una IIFE que devuelve un objeto de closures. Los tres métodos cierran sobre la misma variable `balance`. Es verdaderamente privado: no puedes acceder a `balance` desde fuera.
    ```javascript
        console.log(bank.balance); // undefined
    ```
    Antes de las clases, esto era *la* forma de encapsular estado en JavaScript. todavía lo verás en bases de código heredadas y bibliotecas de utilidades.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={7}
  group="Experto"
  title="Fuga de Memoria por Cierre"
  options={[
    {text: 'Registra la longitud del arreglo data'},
    {text: 'Crea una fuga de memoria', isAnswer: true},
    {text: 'Lanza un RangeError'},
    {text: 'Nada inusual — esto está bien'},
  ]}
>
  <slot name="question">
  <div className="question">
    ¿Cuál es el problema con este código?
    ```javascript
        function attachHandler(element) {
          const data = new Array(100_000).fill('🔥');
          element.addEventListener('click', () => {
            console.log('clicked!', data.length);
          });
        }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    El listener del evento cierra sobre `data`, por lo que el arreglo sigue siendo accesible mientras el listener lo sea. Si `element` tiene una vida larga y el listener nunca se elimina, ese gran arreglo también se retiene.

    `element` está en el DOM. El listener mantiene `data`. El GC no puede recolectarlo.

    Solución: no captures datos grandes en listeners de larga vida a menos que realmente lo necesites, y elimina los listeners cuando la UI propietaria se desmonte:
    ```javascript
        function attachHandler(element) {
          const data = new Array(100_000).fill('🔥');
          // do something with data...

          const onClick = () => {
            console.log('clicked!');
          };

          element.addEventListener('click', onClick);
          return () => element.removeEventListener('click', onClick);
        }
    ```
    Modern V8 es mejor en esto que antes. Pero en SPA de larga duración con muchos listeners de eventos, este patrón todavía atrapa a la gente.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={8}
  group="Experto"
  title="La trampa de this"
  options={[
    {text: 'Dan, Dan'},
    {text: 'Dan, undefined'},
    {text: 'undefined, Dan'},
    {text: 'undefined, undefined', isAnswer: true},
  ]}
>
  <slot name="question">
  <div className="question">
    ¿Qué muestra este registro?
    ```javascript
        const user = {
          name: 'Dan',
          greetArrow: () => console.log(this?.name),
          greetRegular() {
            const inner = () => console.log(this?.name);
            inner();
          },
        };

        user.greetArrow();
        const fn = user.greetRegular;
        fn();
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Dos trampas sutiles:

    **`greetArrow`**: Las funciones arrow no tienen su propio `this`. Capturan `this` del ámbito léxico circundante — que aquí es el ámbito del módulo/global, no `user`. Así que `this?.name` es `undefined`.

    **`fn()`**: `greetRegular` se llama sin un receptor (`fn()` no `user.greetRegular()`). En modo estricto, `this` es `undefined`. En modo no estricto, es el objeto global. De cualquier forma, `this?.name` es `undefined`.

    Si hubieras llamado `user.greetRegular()` directamente, obtendrías `'Dan'` — porque entonces `this` está ligado a `user`.

    Resultado: `undefined, undefined`.
  </div>
  </slot>
</Challenge>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={9}
  group="Bonificación"
  title="Memoización mediante Cierre"
  options={[
    {text: 'Llama a `expensive()` 3 veces'},
    {text: 'Llama a `expensive()` 1 vez', isAnswer: true},
    {text: 'Llama a `expensive()` 2 veces'},
    {text: 'Lanza: Maximum call stack exceeded'},
  ]}
>
  <slot name="question">
  <div className="question">
    ¿Cuántas veces se llama realmente a `expensive()`?
    ```javascript
        function memoize(fn) {
          const cache = new Map();
          return (arg) => {
            if (cache.has(arg)) return cache.get(arg);
            const result = fn(arg);
            cache.set(arg, result);
            return result;
          };
        }

        let calls = 0;
        const expensive = (n) => { calls++; return n * 2; };
        const fast = memoize(expensive);

        fast(5);
        fast(5);
        fast(5);
        console.log(calls);
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    La función memoizada cierra sobre `cache`. La primera llamada (`fast(5)`) falla en la caché, llama a `expensive` y almacena el resultado. La segunda y tercera llamadas encuentran la caché y devuelven inmediatamente.

    `calls` es `1`.

    Esto muestra cierres haciendo trabajo real. `cache` es privada (no se puede acceder desde fuera de `memoize`), persistente (vive mientras exista `fast`) y compartida entre todas las llamadas a `fast`.
  </div>
  </slot>
</Challenge>

¿Cómo te fue?

- **9‑10**: En realidad dominas los closures. Probablemente sea intimidante hacer pair‑programming contigo.
- **6‑8**: Sólido. Las preguntas de closure obsoleto / `this` te pillaron, lo cual es honesto.
- **3‑5**: Conoces el concepto pero los bordes están borrosos. Ve a leer la documentación de la cadena de alcance en MDN.
- **0‑2**: Creías que sabías de closures. Ahora sí los sabes. Ese es el objetivo.

Los closures obsoletos en React (#6) y la trampa de `this` (#8) son los que generan errores reales en producción. Si esos temas te resultaron desconocidos, vale la pena revisarlos — aparecen en revisiones de código más que cualquier otro problema relacionado con closures.
