## Solltest du `named` oder `default` Exports in JavaScript verwenden?

Es mangelt es nicht an stark formulierten Artikeln zu diesem Thema.

Die Mehrheit beurteilt `default export` als „schreckbar“. Andere behaupten, dass `default` gewinnen sollte (z. B. der AirBnb-Stilguide).

Sie werfen oft **völlig vorübergehenden** Dingen die Schuld zu: IDE-Auto-Import-Bugs, die Tree-Shaking-Fähigkeiten eines bestimmten Bundlers oder bloß die Möglichkeit von Tippfehlern beim Benennen eines Imports.

Haben wir den eigentlichen Zweck des `export`ierens übersehen?

**Code ist Kommunikation. ✨**

> Wir senden importierenden Modulen ein Signal, wie sie etwas verwenden sollen.

### Also, was sagen wir?

Im Wesentlichen gibt es zwei Arten, Dinge in modernem JavaScript zu exportieren:

- Ein `export default` erklärt klar und deutlich: „Das ist **DER EINZIGE WICHTIGSTE** Punkt.“ Gleichzeitig: „Nebenexports spielen nur eine unterstützende Rolle.“
- Ein `named export` sagt klipp und klar: „Das ist definitiv **EINE SACHA!**“ Und fragt gleich mit: „Hast du noch andere Freunde dabei?“ Nachfrage: „Sind die eingeladen oder verpflichtend?“

Natürlich kannst du beides kombinieren oder unterschiedliche Ansätze für verschiedene Teile deines Codebases verwenden. [Weitere Beispiele findest du am Ende des Artikels.](#summary)

### Schwache Argumente, Alter

Schwache Argumente, Alter

Lassen Sie uns einige der gängigen „vorübergehenden Probleme“ ansprechen, mit denen Leute aufstoßen.

- Arg #1: Named Exports gewährleisten Namenskonsistenz. [Quelle](https://blog.neufund.org/why-we-have-banned-default-exports-and-you-should-do-the-same-d51fdc2cf2ad)  
  - Nein, tun sie nicht. Suchst du vielleicht eine Linter-Regel?  
  - (Ich hasse es, dir das sagen zu müssen, aber warte, bis du erfährst, was Variablen alles können!)  

```tsx
// Du kannst beides aliassen!  
import { Knife as Handle } from "./knife.js"; // 🔪  
import { default as Handle } from "./knife.js"; // 🔪  
import Handle from "./knife.js"; // 🔪  
```

- Arg #2: Verwende `import * as soManyKnives from './kinves.js'`, um named Exports zu kombinieren. (Nicht verlinkt, Autor zurückgezogen.)  
  - Nette Funktion. Nicht der Punkt.  
  - Sag mir jetzt, wie halt ich deinen Kram wieder fest? Kein Autorabsicht.  
- Arg #3: Named Exports haben besseren IDE-Import- oder -Umbenennungs-Unterstützung. [Quelle](https://www.bundleapps.io/blog/use-named-exports-over-default-exports-in-javascript)  

- Falsch (heute nicht mehr). Konfiguriere/deine Tools aktualisieren.  
  - Unterstützung existiert seit 3+ Jahren in [VS Code](https://github.com/microsoft/vscode/pull/94480), IntelliJ, etc.  
  - Dennoch gibt es einige „Best Practices“, um mit `default exports` die beste IDE- und Refaktoriererfahrung zu erzielen.  
  - ✅ `export default function UserService() {}` - immer benannte Funktionen bevorzugen.  
  - ❌ `export default function() { }` - anonyme Funktionen sind nicht implizit mit ihrem Dateinamen verknüpft. Wenn du die Sache nicht benennst, ist es schwer, den Computer bitten zu können, sie zu ändern.  
  - **Hinweis:** Aus historischen Gründen kann man `export default` nicht mit einem `const`-Ausdruck verknüpfen.  

```tsx
export default const Knife = () => {...blade, ...handle}  
// ^ ❌ Nicht unterstützt ❌ ^  
// Export default const ... nicht möglich  
// ==========================  

// Sobald deklariert, kannst du eine const-Variabel als default exportieren.  
const Knife = () => {...blade, ...handle}  
export default Knife;  
// ^ ✅ Gültig  

// Für Vollständigkeit:  
export default class anyoneStillUseThese {}  
// ^ ✅ Auch gültig, um eine Klasse als default zu exportieren  
```

## Zusammenfassung  

Es gibt tatsächlich viele Kombinationen, wie wir Dinge exportieren können, jede erzählt eine andere Geschichte:  

| Default (Exports) | Named (Exports) | Private Fns | Pattern                                                   | Meaning                                                       |  
| ----------------- | --------------- | ----------- | --------------------------------------------------------- | ------------------------------------------------------------- |  
| ✅                | ❌              | ❌          | Ein default Export.                                       | „Hier ist EINE Funktion mit Einem Zweck!“                   |  
| ❌                | ✅              | ❌          | Ein named Export.                                         | „Bitte benenne mich nicht um.“                              |  
| ✅                | ✅              | ✅          | Default Export + mehrere 'private' nicht-exportierte Funktionen | „Hier ist einige verwandte Logik. Erwarte auch Klassen-artiges Verhalten.“ |  
| ❌                | ❌              | ✅          | Mehrere named Exports, generischer Dateiname.             | „Ein Sammelsurium lose verbundener Dinge, keine Hierarchie impliziert.“ |  
| ✅                | ✅              | ❌          | Ein named Export, der auch als default exportiert wird.   | „Du kannst nicht falsch importieren.“                         |  
</section>  

**Etwas zum Nachdenken:** Was sagen wir, wenn der Dateiname mit einem seiner Exports übereinstimmt oder nicht? (Zum Beispiel eine `utils.js` mit vielen Funktionen.)

### Fazit

Wenn Code Kommunikation ist, dann `export` wie du es verdammt nochmal ernst meinst. 💞
