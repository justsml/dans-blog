## JavaScriptで `named` と `default` のどちらのエクスポートを使うべきか？

このテーマについては、熱烈な意見が溢れています。

大半は `default export` を「ひどい」と評し、別の立場では `default` が優先すべきだと主張します（例：AirBnb のスタイルガイド）。

批判の多くは **一時的な** 問題にすり替えています。IDE の自動インポートのバグ、特定のバンドラのツリーシェイキング性能、あるいはインポート名のタイプミスの可能性といったものです。

そもそも `export` の本来の目的を見失ってはいませんか？

**コードはコミュニケーションです。 ✨**

> `import` する側に **「このものはどう使うべきか」** というシグナルを送っている。

### では、何を伝えているのか？

現代の JavaScript では、エクスポートの方法は大きく 2 種類に分けられます。

- `export default` は「**この **_最重要_** なもの**」であることを大胆に宣言します。併せて「名前付きエクスポートは補助的な役割にすぎない」と示唆します。
- `named export` は「**これは **_あるもの_** だ」 と断言し、さらに「他に仲間はいるか？」 といった疑問を投げかけます。続いて「それらは招待されているのか、必須なのか？」 といったニュアンスが付随します。

もちろん、両者を組み合わせたり、コードベースの領域ごとに使い分けることも可能です。 [記事の最後にさらに例があります。](../#summary)

### 弱い議論、マン

一般的に遭遇する「一時的な問題」について見ていきましょう。

- **議論 #1:** 名前付きエクスポートは名前の一貫性を保証する。 [source](https://blog.neufund.org/why-we-have-banned-default-exports-and-you-should-do-the-same-d51fdc2cf2ad)
  - いいえ、保証しません。 lint ルールを探しているのかもしれません。
  - （残念ながら、変数が何ができるかを学ぶまで待ってください！）

```tsx
// 両方でエイリアス付け可能!
import { Knife as Handle } from "./knife.js"; // 🔪
import { default as Handle } from "./knife.js"; // 🔪
import Handle from "./knife.js"; // 🔪
```

- **議論 #2:** `import * as soManyKnives from './kinves.js'` で名前付きエクスポートをまとめる。（リンクなし、作者が撤回）
  - 便利な機能です。ポイントではありません。
  - では、改めてどうやってその装置を握るのか教えてください。作者の意図が見えません。

- **議論 #3:** 名前付きエクスポートは IDE のインポートやリネームサポートが優れている。 [source](https://www.bundleapps.io/blog/use-named-exports-over-default-exports-in-javascript)

- **誤り（その他）:** ツールの設定・更新を行ってください。
  - 3 年以上前から [VS Code](https://github.com/microsoft/vscode/pull/94480)、IntelliJ などでサポートされています。
  - それでも、`default export` を快適に扱うための「ベストプラクティス」があります。
  - ✅ `export default function UserService() {}` — 常に名前付き関数を推奨。
  - ❌ `export default function() { }` — 匿名関数はファイル名と暗黙的に結びつきません。名前を付けないと、リファクタリングが困難です。
  - **注意:** 歴史的経緯で `export default` と `const` 式の併用はできません。

    ```tsx
    export default const Knife = () => {...blade, ...handle}
    // ^ ❌ Not Supported ❌ ^
    // Cannot export default const ....
    // ==========================

    // ただし、一度宣言すれば const 変数を default としてエクスポート可能。
    const Knife = () => {...blade, ...handle}
    export default Knife;
    // ^ ✅ Valid

    // 完全版:
    export default class anyoneStillUseThese {}
    // ^ ✅ Also valid to export a class as default
    ```

## Summary

エクスポートの組み合わせは実に多様で、各パターンが異なるストーリーを語ります。

| Default (Exports) | Named (Exports) | Private Fns | Pattern                                                   | Meaning                                                       |
| ----------------- | --------------- | ----------- | --------------------------------------------------------- | ------------------------------------------------------------- |
| ✅                | ❌              | ❌          | One default export.                                       | “単一目的の関数を 1 つ提示しています！”                     |
| ❌                | ✅              | ❌          | One named export.                                         | “名前を変えないでください。”                                 |
| ✅                | ✅              | ✅          | Default export + multiple 'private' un-exported functions | “関連ロジックがここにあります。クラス的な振る舞いも期待してください。” |
| ❌                | ❌              | ✅          | Multiple named exports, generic file name.                | “緩く関連したものの寄せ集めで、階層は暗示されていません。” |
| ✅                | ✅              | ❌          | Single named export ALSO exported as default.             | “インポートを間違える余地がありません。”                     |
</section>

**考えてみるべきこと:** ファイル名がエクスポートのいずれかと一致するかしないかで、何を暗示しているでしょうか？（例: 多数の関数を持つ `utils.js`）

### 結論

コードがコミュニケーションなら、`export` は本気でやれ。 💞
