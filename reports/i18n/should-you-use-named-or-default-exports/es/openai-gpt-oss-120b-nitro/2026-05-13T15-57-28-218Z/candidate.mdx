## ¿Deberías usar exportaciones `named` o `default` en JavaScript?

No faltan los artículos con tono enérgico sobre este tema.

La mayoría juzgan el `default export` como “terrible”. Otros sostienen que el `default` debe ganar (p. ej., la guía de estilo de AirBnb).

A menudo culpan cosas **totalmente temporales**: errores de auto‑importación del IDE, las capacidades de tree‑shaking de un empaquetador concreto, o la mera posibilidad de errores tipográficos al nombrar una importación.

¿Nos hemos perdido el objetivo de `export`ar en primer lugar?

**El código es comunicación. ✨**

> Estamos enviando una señal a los `import`ers _sobre cómo usar algo._

### Entonces, ¿qué estamos diciendo?

En términos generales, hay 2 formas de exportar cosas en el JavaScript moderno:

- Un `export default` declara con firmeza “Esto es **_LO MÁS IMPORTANTE_**.” Además, “cualquier exportación nombrada solo tiene un papel secundario.”
- Una `named export` dice que “definitivamente **_ES UNA COSA!_**” y también plantea preguntas como “¿hay otros compañeros allí?” y, como seguimiento, “¿Están invitados o son obligatorios?”

Por supuesto puedes combinar ambos, o usar enfoques diferentes para distintas partes de tu base de código. [Ver más ejemplos al final del artículo.](../#summary)

### Weak Args, Man

Abordemos algunos de los “problemas temporales” que la gente suele encontrar.

- Arg #1: Las exportaciones nombradas garantizan la consistencia de nombres. [source](https://blog.neufund.org/why-we-have-banned-default-exports-and-you-should-do-the-same-d51fdc2cf2ad)
  - No, no lo hacen. ¿Quizás estás buscando una regla de lint?
  - (¡Odio romperte la ilusión, pero espera a ver lo que pueden hacer las variables!)

```tsx
// ¡Puedes aliasar usando ambos!
import { Knife as Handle } from "./knife.js"; // 🔪
import { default as Handle } from "./knife.js"; // 🔪
import Handle from "./knife.js"; // 🔪
```

- Arg #2: Usa `import * as soManyKnives from './kinves.js'` para combinar exportaciones nombradas. (No enlazado, autor retractado.)
  - Función interesante. No es el punto.
  - Ahora dime, ¿cómo vuelvo a sujetar tu artilugio? No hay intención del autor.

- Arg #3: Las exportaciones nombradas tienen mejor soporte de importación o renombrado en el IDE. [source](https://www.bundleapps.io/blog/use-named-exports-over-default-exports-in-javascript)

- Incorrecto (cualquier otro). Configura/actualiza tus herramientas.
  - El soporte ha existido por más de 3 años en [VS Code](https://github.com/microsoft/vscode/pull/94480), IntelliJ, etc.
  - Aún así, hay algunas “mejores prácticas” para usar con `export default` y obtener la mejor experiencia de IDE y refactorización.
  - ✅ `export default function UserService() {}` – siempre prefiere funciones nombradas.
  - ❌ `export default function() { }` – las funciones anónimas no están implícitamente vinculadas a su nombre de archivo. Si no nombras la cosa, es difícil pedirle a la computadora que la cambie.
  - **Nota:** Por razones históricas no puedes combinar `export default` con una expresión `const`.

    ```tsx
    export default const Knife = () => {...blade, ...handle}
    // ^ ❌ No soportado ❌ ^
    // No se puede exportar default const ....
    // ==========================

    // Sin embargo, una vez declarada puedes exportar una variable const como default.
    const Knife = () => {...blade, ...handle}
    export default Knife;
    // ^ ✅ Válido

    // Para completitud:
    export default class anyoneStillUseThese {}
    // ^ ✅ También válido exportar una clase como default
    ```

## Resumen

En realidad existen muchísimas combinaciones de formas de exportar cosas, cada una cuenta una historia distinta:

| Default (Exports) | Named (Exports) | Private Fns | Pattern                                                   | Meaning                                                       |
| ----------------- | --------------- | ----------- | --------------------------------------------------------- | ------------------------------------------------------------- |
| ✅                | ❌              | ❌          | Una exportación default.                                   | “¡Presentando UNA función con Propósito Único!”               |
| ❌                | ✅              | ❌          | Una exportación nombrada.                                 | “Por favor, no me renombres.”                                 |
| ✅                | ✅              | ✅          | Exportación default + múltiples funciones ‘privadas’ no exportadas | “Aquí hay lógica relacionada. Además, espera comportamiento tipo clase.” |
| ❌                | ❌              | ✅          | Múltiples exportaciones nombradas, nombre de archivo genérico. | “Una mezcolanza de cosas vagamente relacionadas, sin jerarquía implícita.” |
| ✅                | ✅              | ❌          | Exportación nombrada única TAMBIÉN exportada como default. | “No puedes equivocarte al importarme.”                        |
</section>

**Algo en qué pensar:** ¿Qué estamos diciendo cuando el nombre del archivo coincide o no con una de sus exportaciones? (Por ejemplo, un `utils.js` con muchas funciones.)

### Conclusión

Si el código es comunicación, por favor `export` como si realmente lo quisieras. 💞
