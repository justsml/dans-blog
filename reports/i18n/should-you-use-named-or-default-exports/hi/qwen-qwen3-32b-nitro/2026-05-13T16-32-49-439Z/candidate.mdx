## जावास्क्रिप्ट में आपको `named` या `default` एक्सपोर्ट का उपयोग करना चाहिए?

इस विषय पर तीखे शब्दों वाले लेखों की कमी नहीं है।

अधिकांश `default export` को “बुरा” ठहराते हैं। दूसरे `default` के निकलने का समर्थन करते हैं (जैसे, एयरबीएनबी स्टाइल गाइड।)

वे अक्सर **पूरी तरह से अस्थायी** चीजों को दोषी ठहराते हैं: IDE ऑटो-इम्पोर्ट बग, किसी विशेष बंडलर की ट्री-शेकिंग क्षमता, या नामकरण के समय टाइपो की संभावना।

क्या हमने पहले से ही `export` करने के मुख्य बिंदु को भूल गए हैं?

**कोड संचार है। ✨**

> हम `import`र्स को एक संकेत भेज रहे हैं कि किस तरह से किसी चीज़ का उपयोग करना है।

### तो, हम क्या कह रहे हैं?

सामान्य रूप से, आधुनिक जावास्क्रिप्ट में चीजों को निर्यात करने के 2 तरीके हैं:

- एक `export default` बोल्डी ढंग से घोषित करता है कि "**एकमात्र सबसे महत्वपूर्ण चीज़** है।" इसके साथ-साथ, "कोई भी नामित निर्यात केवल सहायक भूमिका निभाते हैं।"
- एक `named export` कहता है कि यह "**बिल्कुल एक चीज़ है!**" इसके साथ-साथ कुछ सवाल उठाता है, "क्या आपके पास अन्य दोस्त हैं?" अगला प्रश्न, "क्या वे आमंत्रित हैं या आवश्यक हैं?"

बेशक, आप दोनों को जोड़ सकते हैं, या अपने कोडबेस के अलग-अलग हिस्सों के लिए अलग-अलग दृष्टिकोण अपना सकते हैं। [अंत में अधिक उदाहरण देखें।](#summary)

### कमजोर तर्क, दोस्त

चलिए कुछ सामान्य "अस्थायी समस्याओं" के बारे में बात करते हैं जिनसे लोग टकराते हैं।

- तर्क #1: नामित निर्यात नाम एकरूपता की गारंटी देते हैं। [स्रोत](https://blog.neufund.org/why-we-have-banned-default-exports-and-you-should-do-the-same-d51fdc2cf2ad)
  - नहीं, वे नहीं करते। शायद आप एक लिंट नियम की तलाश में हैं?
  - (मैं आपको बुरी तरह से बताना नहीं चाहता, लेकिन पहले आपको जानना होगा कि चर क्या कर सकते हैं!)

```tsx
// आप दोनों का उपयोग करके एलियस कर सकते हैं!
import { Knife as Handle } from "./knife.js"; // 🔪
import { default as Handle } from "./knife.js"; // 🔪
import Handle from "./knife.js"; // 🔪
```

- तर्क #2: `import * as soManyKnives from './kinves.js'` का उपयोग नामित निर्यात को जोड़ने के लिए करें। (स्रोत नहीं जुड़ा, लेखक ने वापस ले लिया।)
  - एक सुंदर विशेषता। यह बिंदु नहीं है।
  - अब बताएं, मैं आपके उपकरण को फिर से कैसे पकड़ू? कोई लेखक का इरादा नहीं।
- तर्क #3: नामित निर्यातों में बेहतर IDE आयात या पुन:नामकरण समर्थन है। [स्रोत](https://www.bundleapps.io/blog/use-named-exports-over-default-exports-in-javascript)

- गलत (अब नहीं)। अपने उपकरणों को निर्मित/अपडेट करें।
  - [VS Code](https://github.com/microsoft/vscode/pull/94480), IntelliJ आदि में 3+ वर्षों से समर्थन है।
  - फिर भी, कुछ "सर्वोत्तम प्रथाएँ" हैं जिनका उपयोग `default exports` के साथ करके आपके IDE और पुनर्गठन अनुभव को बेहतर बनाया जा सकता है।
  - ✅ `export default function UserService() {}` - हमेशा नामित फ़ंक्शन का प्राथमिकता दें।
  - ❌ `export default function() { }` - अज्ञात नामक फ़ंक्शन अपने फ़ाइल नाम से अनियमित रूप से जुड़े होते हैं। यदि आप चीज़ का नाम नहीं देते, तो कंप्यूटर को उसे बदलने के लिए पूछना मुश्किल होता है।
  - **नोट:** ऐतिहासिक कारणों से आप `export default` को एक `const` अभिव्यक्ति के साथ जोड़ नहीं सकते हैं।

    ```tsx
    export default const Knife = () => {...blade, ...handle}
    // ^ ❌ समर्थित नहीं ❌ ^
    // डिफ़ॉल्ट कॉन्स्ट निर्यात नहीं किया जा सकता
    // ==========================

    // हालाँकि, एक बार घोषित करने के बाद आप एक कॉन्स्ट वैरिएबल को डिफ़ॉल्ट के रूप में निर्यात कर सकते हैं।
    const Knife = () => {...blade, ...handle}
    export default Knife;
    // ^ ✅ मान्य

    // पूर्णता के लिए:
    export default class anyoneStillUseThese {}
    // ^ ✅ डिफ़ॉल्ट के रूप में एक क्लास निर्यात करना भी मान्य है
    ```

## सारांश

हम वास्तव में निर्यात करने के कई संयोजन हैं, प्रत्येक अलग-अलग कहानी बताता है:

| डिफ़ॉल्ट (निर्यात) | नामित (निर्यात) | निजी फ़ंक्शन | पैटर्न                                                   | अर्थ                                                       |
| ----------------- | --------------- | ----------- | --------------------------------------------------------- | ------------------------------------------------------------- |
| ✅                | ❌              | ❌          | एक डिफ़ॉल्ट निर्यात।                                       | “एकल उद्देश्य वाले एक फ़ंक्शन का प्रस्तुति!”                  |
| ❌                | ✅              | ❌          | एक नामित निर्यात।                                         | “कृपया मुझे पुन:नामित न करें।”                                     |
| ✅                | ✅              | ✅          | डिफ़ॉल्ट निर्यात + कई 'निजी' अनिर्यात फ़ंक्शन | “यहाँ कुछ संबंधित तर्क हैं। भी, क्लास-जैसा व्यवहार अपेक्षित है।” |
| ❌                | ❌              | ✅          | कई नामित निर्यात, सामान्य फ़ाइल नाम।                | “लोगों के बिना जुड़े चीजों का एक ग्रैब-बैग, कोई श्रेणीकरण नहीं।” |
| ✅                | ✅              | ❌          | एक नामित निर्यात जो डिफ़ॉल्ट के रूप में भी निर्यात किया जाता है।             | “मुझे आयात करने में गलती नहीं होगी।”                             |
</section>

**सोचने के लिए कुछ बातें:** जब फ़ाइल नाम अपने निर्यातों में से एक के साथ मेल खाता है या नहीं, तो हम क्या कह रहे हैं? (उदाहरण के लिए, `utils.js` जिसमें कई फ़ंक्शन हैं।)

### निष्कर्ष
