## Следует ли использовать `named` или `default` экспорт в JavaScript?

Не не хватает статей с резко сформулированными мнениями на эту тему.

Большинство называют `default export` ужасным. Другие утверждают, что `default` должен быть предпочтительным (например, стиль AirBnb).

Они часто обвиняют **чисто временные** вещи: ошибки автозагрузки импортов в IDE, способности конкретного бандлера к tree-shaking, или даже возможность опечаток при именовании импорта.

Забыли ли мы о главном в самом экспорте?

**Код — это коммуникация. ✨**

> Мы отправляем сигнал _импортирующим_, как использовать сущность.

### Итак, что мы имеем в виду?

В современном JavaScript существует два основных способа экспорта:

- `export default` смело декларирует: «Это **_ЕДИНСТВЕННОЕ И ВАЖНЕЙШЕЕ_**». Также подразумевает: «именные экспорт лишь играют вспомогательную роль».
- Именной экспорт говорит: «**_ЭТО СУЩНОСТЬ!_**» и задаёт вопросы: «Есть ли у вас ещё коллеги?» Следом: «Они приглашены или обязательны?»

Конечно, вы можете комбинировать оба подхода или использовать разные методы в разных частях кода. [Узнайте больше примеров в конце статьи.](#summary)

### Слабые аргументы, чувак

Давайте рассмотрим некоторые из распространённых «временных проблем», с которыми сталкиваются люди.

- Аргумент #1: Именные экспорт обеспечивают согласованность названий. [источник](https://blog.neufund.org/why-we-have-banned-default-exports-and-you-should-do-the-same-d51fdc2cf2ad)
  - Нет, не обеспечивают. Возможно, вы имели в виду правило проверки (lint rule)?
  - (Я сожалею, что должен это сказать, но подождите, пока вы узнаете, что могут переменные!)

```tsx
// Вы можете использовать псевдонимы обоими способами!
import { Knife as Handle } from "./knife.js"; // 🔪
import { default as Handle } from "./knife.js"; // 🔪
import Handle from "./knife.js"; // 🔪
```

- Аргумент #2: Используйте `import * as soManyKnives from './kinves.js'` для объединения именных экспорт. (Не указан источник, автор отозвал утверждение.)
  - Полезная функция. Не по теме.
  - Теперь скажите мне, как я снова могу использовать ваше устройство? Нет намёка на авторский намерение.
- Аргумент #3: Именные экспорт лучше поддерживаются IDE при импорте или переименовании. [источник](https://www.bundleapps.io/blog/use-named-exports-over-default-exports-in-javascript)

- Неправильно (больше не актуально). Настройте/обновите свои инструменты.
  - Поддержка существует более 3 лет в [VS Code](https://github.com/microsoft/vscode/pull/94480), IntelliJ и т.д.
  - Тем не менее, есть некоторые «лучшие практики» при использовании `default exports` для получения лучшего опыта IDE и рефакторинга.
  - ✅ `export default function UserService() {}` - всегда предпочитайте именованные функции.
  - ❌ `export default function() { }` - анонимные функции не связаны с именем файла. Если вы не называете сущность, сложно попросить компьютер изменить её.
  - **Примечание:** По историческим причинам нельзя комбинировать `export default` с выражением `const`.

    ```tsx
    export default const Knife = () => {...blade, ...handle}
    // ^ ❌ Не поддерживается ❌ ^
    // Нельзя экспортировать default const ...
    // ==========================

    // Однако, после объявления можно экспортировать const как default.
    const Knife = () => {...blade, ...handle}
    export default Knife;
    // ^ ✅ Допустимо

    // Для полноты:
    export default class anyoneStillUseThese {}
    // ^ ✅ Также допустимо экспортировать класс как default
    ```

## Сводка

На самом деле существует множество комбинаций экспорта, каждая из которых рассказывает свою историю:

| Default (Экспорт) | Named (Экспорт) | Частные функции | Шаблон                                                   | Значение                                                       |
| ----------------- | --------------- | --------------- | --------------------------------------------------------- | ------------------------------------------------------------- |
| ✅                | ❌              | ❌              | Один default экспорт.                                     | «Представляю ОДНУ функцию с Единым Назначением!»                  |
| ❌                | ✅              | ❌              | Один named экспорт.                                         | «Не переименовывайте меня.»                                     |
| ✅                | ✅              | ✅              | Default экспорт + несколько 'частных' неэкспортируемых функций | «Вот некоторые связанные логики. Также, ожидайте классоподобное поведение» |
| ❌                | ❌              | ✅              | Множество named экспорт, общий файл.                        | «Корзина несвязанных вещей, без иерархии.»                        |
| ✅                | ✅              | ❌              | Один named экспорт, также экспортированный как default.     | «Вы не можете неправильно импортировать меня.»                             |
</section>

**Над чем стоит подумать:** Что мы говорим, когда имя файла совпадает или не совпадает с одним из его экспортов? (Например, `utils.js` со множеством функций.)

### Заключение

Если код — это коммуникация, то `export` делайте по-настоящему серьезно. 💞
