  <slot name='explanation'>
  <div className="explanation">
    Debido a errores de precisión en punto flotante, `0.1 + 0.2` no es exactamente igual a `0.3`. Debido a la forma en que se almacenan los números de punto flotante en la memoria, el resultado es `0.30000000000000004`. El estándar IEEE 754 para la aritmética de punto flotante es el culpable: no puede representar ciertos números con exactitud. Este es un problema común en todos los lenguajes de programación. Eventualmente te encontrarás con un decimal periódico, y sin importar el lenguaje, la computadora simplemente tiene que dejar de perseguir dígitos infinitos.

    Algunos lenguajes como Python y Java tienen `Decimal` o `BigDecimal` para manejar esto, pero no está integrado en JavaScript. Puedes usar bibliotecas como `big.js` o `decimal.js` para gestionar estos casos.

    (Nota: Algunos lenguajes están diseñados para manejar fracciones, números imaginarios, etc., a un nivel lógico superior, preservando las expresiones literales. Pero aún así tienen que lidiar con los mismos problemas de precisión de punto flotante a nivel de hardware.)
  </div>

    ```jsx
        parseInt("42") === parseFloat("42")
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    En JavaScript, tanto `parseInt` como `parseFloat` interpretarán la cadena `"42"` como el número `42`. Por lo tanto, la comparación `parseInt("42") === parseFloat("42")` se evalúa como `true`. Aunque `parseInt` detiene el análisis en el primer carácter no numérico, `parseFloat` continúa hasta encontrar un carácter que no forme parte de un número de punto flotante. Sin embargo, dado que `"42"` no contiene puntos decimales ni otros caracteres no numéricos, ambas funciones devuelven el mismo valor.
  </div>
  </slot>

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={9}
  group="Number[]"
  title="Usar `.map(parseInt)`"
  options={[
    {text: '[24, NaN, NaN]', isAnswer: true},
    {text: '[24, NaN, 42]'},
    {text: '[24, 42]'},
    {text: 'NaN'},
  ]}
>
  <slot name="question">
  <div className="question">
    ```jsx
        [24, 'One', 42].map(parseInt)
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    El segundo argumento de `parseInt` (la base numérica) se alinea con el argumento `index` de los métodos de array. Esto genera resultados inesperados, ya que `parseInt("One", 1)` devuelve `NaN` debido a una entrada inválida.

    El primer elemento, `24`, se analiza como `24` en base 0 (detección automática), por lo que se mantiene igual. El segundo elemento, `'One'`, se analiza como `NaN` en base 1. El tercer elemento, `42`, se analiza usando base 2. En base 2, `'42'` es `NaN`, así que el resultado es `[24, NaN, NaN]`.

    Esta es una trampa clásica al usar `parseInt` con `map`. Si deseas convertir un array de strings a números, el único método "integrado" seguro es `.map(Number)` o agregar un callback como `.map(x => parseInt(x, 10))`.
  </div>
  </slot>
</Challenge>
