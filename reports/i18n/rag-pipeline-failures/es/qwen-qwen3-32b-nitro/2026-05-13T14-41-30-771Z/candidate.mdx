Los demostraciones de RAG siempre se ven geniales.  

Incrustas algunos documentos, configuras un almacén vectorial, lo conectas a un chat y ves cómo el modelo cita tu base de conocimiento interno como si hubiera trabajado allí durante años. Bonito. Las partes interesadas quedan impresionadas. Alguien dice: "vamos a lanzar esto".  

Seis semanas después, los usuarios reciben respuestas claramente incorrectas. Los tickets de soporte se acumulan. El sistema *funciona*, solo que no de la manera en que la gente lo necesita.  

El fallo suele no ser un error dramático. Son cinco aburridos apilados juntos.  

## 1. Tus fragmentos tienen el tamaño incorrecto

Este fallo no provoca un error. Simplemente hace que cada respuesta sea ligeramente peor hasta que la característica completa se sienta poco confiable.  

La búsqueda vectorial recupera *fragmentos*, no documentos. Lo que dividas tu material fuente se convierte en la unidad de verdad del recuperador. Si los fragmentos están equivocados, el modelo responde con fragmentos incorrectos.  

**Demasiado pequeño**: El fragmento contiene una respuesta parcial. El incrustado captura el tema correcto, pero el texto recuperado carece de contexto. Recuperas "El tiempo máximo de espera es de 30 segundos" sin la oración anterior que dice "cuando se usa la API legada".  

**Demasiado grande**: El incrustado se convierte en un promedio borroso de muchas ideas. La búsqueda semántica se confunde porque el fragmento trata sobre múltiples temas, y el vector resultante no representa claramente ninguno.  

El tamaño correcto de fragmento depende completamente de tu contenido. Las documentaciones técnicas, los contratos legales y los transcritos de soporte se dividen de maneras distintas. No existe una respuesta universal.  

¿Qué hacer? Mide. Construye un conjunto de evaluación de pares pregunta/respuesta desde tu corpus. Prueba fragmentos de 256, 512 y 1024 tokens. Mide la precisión de recuperación: ¿aparece el fragmento correcto entre los 5 primeros? Aprenderás rápidamente que el tamaño de los fragmentos importa más que el modelo de incrustación sobre el que te desgarraste.

También usa solapamiento. Un fragmento de 512 tokens con 64 tokens de solapamiento en cada lado significa que las respuestas que abarcan límites de fragmentos aún se recuperan. La mayoría de las bibliotecas de vectores admiten esto. La mayoría de la gente lo omite.

## 2. Tus incrustaciones se vuelven obsoletas (Y no te darás cuenta)

Imagina que tu empresa rebrandea. O renombra un producto. O actualiza sus precios. O deprecia una API.

Actualizas la documentación pero no re-incrustas los fragmentos. El índice de vectores todavía representa el contenido antiguo.

Los usuarios preguntan por los nuevos precios. Las incrustaciones las dirigen al contenido antiguo. El modelo lee el contenido antiguo y explica confiadamente los precios antiguos. Soporte recibe un ticket.

Todo sistema RAG serio eventualmente choca con este problema. La solución parece obvia: re-incrustar cuando cambie el contenido, pero los equipos rara vez construyen esa tubería antes del primer incidente.

Necesitas reindexado incremental con huella digital del contenido:

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Contenido sin cambios, omitir re-incrustación
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

Reindexa al escribir, usa huella digital basada en el contenido, no en timestamps. Los documentos se actualizan en tu CMS constantemente sin que el contenido real haya cambiado.

## 3. Precisión vs. Recall en la Recuperación: Estás Optimizando la Incorrecta

La mayoría de tutoriales de RAG muestran cómo recuperar los K fragmentos más relevantes. No explican el tradeoff entre dos objetivos que apuntan en direcciones opuestas.

**Alta recuperación**: Devolver todo lo que podría ser relevante. Los usuarios siempre obtienen una respuesta. Pero la ventana de contexto del modelo está llena de ruido periférico, y el modelo genera información falsa para rellenar los huecos entre fragmentos.

**Alta precisión**: Solo devolver los fragmentos más relevantes. El modelo trabaja con un contexto limpio y enfocado. Pero si el fragmento correcto no está en los primeros 3, el modelo no tiene la información y de todas formas genera algo confiado.

Los modos de fallo parecen idénticos para los usuarios: respuestas incorrectas. Pero las causas y las soluciones son opuestas.

Dos técnicas que realmente ayudan:

**Reclasificación**: Recuperar más candidatos (top-20), luego usar un modelo cross-encoder para reordenarlos por relevancia antes de pasárselos al LLM. Los cross-encoders son más lentos que la similitud vectorial, pero significativamente más precisos en el paso final de clasificación.

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // Ahora top-5 realmente significa algo
```

**Búsqueda híbrida**: Combinar búsqueda vectorial (similitud semántica) con búsqueda de palabras clave (BM25). Fallan de maneras diferentes. La búsqueda vectorial tiene dificultades con términos específicos, nombres de modelos e IDs. La búsqueda de palabras clave tiene dificultades con paráfrasis y sinónimos. Juntas, cubren las ciegas de cada una.

## 4. Tu ventana de contexto tiene la forma incorrecta

Has recuperado los fragmentos correctos. Felicitaciones. El modelo aún lo hará mal.

El problema no es solo lo que recuperas. Es también dónde lo colocas.

Los Modelos de Lenguaje de Gran Tamaño (LLMs) pueden sufrir del problema de "perdido en el medio". Liu et al. midieron que los modelos de contexto largo utilizan información relevante de manera menos confiable cuando aparece en el medio del prompt en lugar de cerca del inicio o el final.

Si estás empujando 20 fragmentos en una lista plana y esperando que el modelo los sintetice correctamente, estás dejando rendimiento en la mesa.

Cosas que realmente ayudan:

**Evalúa la colocación de inicio y fin para tus fragmentos más relevantes.** Una heurística común es: primero el de mayor rango, luego el segundo de último lugar y el resto en el medio. Contraintuitivo, pero vale la pena probarlo con tu modelo y la forma de tu prompt.

**Numerar y etiquetar explícitamente tus secciones de contexto.** `[Fuente 1]` ... `[Fuente 2]` da al modelo anclajes para razonar.

**Añade una señal de confianza en la recuperación.** Si tu puntuación de similitud es 0.65 en una escala de 0-1, dile al modelo: "El siguiente contexto se recuperó con moderada confianza. Reconoce la incertidumbre si la respuesta no está clara."

**Establece un presupuesto de contexto.** No pases todo lo que recuperaste. Cuenta los tokens, prioriza por puntuación de relevancia y corta de forma rígida al 60-70% de la ventana de contexto del modelo. Deja espacio para que el modelo razone sin saturarse.

Referencia: [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172).

## 5. No tienes ni idea de cuándo está equivocado

Este es el fracaso silencioso: la respuesta se devuelve, la interfaz parece saludable y el contenido está equivocado.

Con una API tradicional, los fallos son visibles: HTTP 500, tiempo de espera agotado, error de validación de esquema. Lo sabes inmediatamente. Los fallos en RAG son más silenciosos: el sistema devuelve una respuesta, parece plausible y está equivocado.

Es posible que no sepas que tu pipeline RAG está fallando hasta que los usuarios te lo digan. A menudo, no lo hacen. Simplemente dejan de confiar en él y regresan a Ctrl+F.

El conjunto mínimo viable de observabilidad para un sistema RAG en producción:

**Registra tu cadena de recuperación.** Cada consulta, lo que se recuperó (IDs de fragmentos + puntuaciones) y lo que produjo el modelo. Necesitas esto para depurar cualquier problema.

**Sigue métricas de recuperación.** Rango recíproco medio (MRR) y NDCG si tienes etiquetas de verdad terrestre. Como mínimo, sigue distribuciones de puntuaciones de similitud — si tu puntuación P50 de recuperación cae, la calidad de tu índice disminuyó.

**Construye un bucle de retroalimentación.** Incluso una valoración de pulgar arriba/abajo en las respuestas, conectada de vuelta a la consulta y los fragmentos recuperados, te da una señal de entrenamiento. Sin ella, estás volando a ciegas.

**Ejecuta evaluaciones periódicas.** Un conjunto de pruebas de 50-100 preguntas con respuestas conocidas, ejecutado semanalmente, detectará regresiones antes que los usuarios. Una hoja de cálculo y un script son suficientes para empezar.

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // Alert if score drops below threshold
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## El Problema Real

Estos fallos no son principalmente sobre el modelo de incrustación o la base de datos de vectores. Son sobre el sistema que los rodea.

Las demostraciones de RAG funcionan porque las condiciones de la demo están controladas: documentos limpios, preguntas bien formuladas, evaluadores tolerantes. El entorno de producción falla porque ninguna de esas condiciones se cumple.

Cada fallo mencionado es detectable, pero solo si estás midiendo. Los equipos de RAG confiables no están haciendo nada exótico. Están tratando la calidad de la recuperación como un subsistema real en lugar de un artefacto de demostración.

Configura primero el bucle de evaluación. Todo lo demás es más fácil una vez que puedes medirlo.
