La demo de RAG siempre se ve genial.

Incrustas algunos documentos, levantas una tienda vectorial, la conectas al chat y observas cómo el modelo cita tu base de conocimiento interna como si hubiera trabajado allí durante años. Hermoso. Los interesados quedan impresionados. Alguien dice “vamos a lanzar esto”.

Seis semanas después, los usuarios reciben respuestas erróneas con confianza. Los tickets de soporte se acumulan. El sistema *funciona*, solo que no de la manera que la gente necesita.

El fallo normalmente no es un error dramático. Son cinco errores aburridos apilados.

---

## 1. Tus fragmentos tienen el tamaño incorrecto
---

Esta falla no provoca un bloqueo. Simplemente empeora cada respuesta ligeramente hasta que toda la funcionalidad resulta poco confiable.

La búsqueda vectorial recupera *fragmentos*, no documentos. Lo que sea que dividas tu material fuente se convierte en la unidad de verdad del recuperador. Si los fragmentos son incorrectos, el modelo responderá con los fragmentos equivocados.

**Demasiado pequeño**: El fragmento contiene una respuesta parcial. La incrustación captura el tema correcto, pero el texto recuperado carece de contexto. Obtienes “El tiempo máximo de espera es 30 segundos” sin la frase anterior que dice “cuando se usa la API heredada”.

**Demasiado grande**: La incrustación se vuelve un promedio borroso de muchas ideas. La búsqueda semántica se confunde porque el fragmento trata de varios temas, y el vector resultante no representa ninguno de ellos de forma limpia.

El tamaño correcto del fragmento depende totalmente de tu contenido. La documentación técnica, los contratos legales y las transcripciones de soporte se fragmentan de manera distinta. No existe una respuesta universal.

Qué hacer: medir. Construye un conjunto de evaluación con pares pregunta/respuesta a partir de tu corpus. Prueba fragmentos de 256, 512 y 1024 tokens. Mide la precisión de recuperación: ¿aparece el fragmento correcto entre los 5 primeros? Aprenderás rápido que el tamaño del fragmento influye más que el modelo de incrustación que tanto te costó elegir.

También usa solapamiento. Un fragmento de 512 tokens con 64 tokens de solapamiento a cada lado permite que las respuestas que cruzan límites aún se recuperen. La mayoría de las bibliotecas vectoriales lo admiten. La mayoría de la gente lo omite.

---

## 2. Tus incrustaciones se vuelven obsoletas (y no lo notarás)

Imagina que tu empresa cambia de marca. O renombra un producto. O actualiza sus precios. O deprecia una API.

Actualizas la documentación pero no vuelves a incrustar los fragmentos. El índice vectorial sigue representando el contenido antiguo.

Los usuarios preguntan por los nuevos precios. Las incrustaciones los dirigen al contenido viejo. El modelo lee el contenido antiguo y explica con confianza la tarificación anterior. Soporte recibe un ticket.

Cada sistema RAG serio acaba topándose con esto. La solución parece obvia: volver a generar incrustaciones cuando el contenido cambia, pero los equipos rara vez construyen ese pipeline antes del primer incidente.

Necesitas re‑indexado incremental con huellas de contenido:

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Contenido sin cambios, omitir re‑incrustación
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

Re‑indexa en el momento de la escritura, genera la huella sobre el contenido, no sobre marcas de tiempo. Los documentos se actualizan en tu CMS todo el tiempo sin que el contenido real cambie.

## 3. Precisión vs. Recall en la Recuperación: Estás Optimizando el Parámetro Incorrecto

La mayoría de los tutoriales de RAG enseñan cómo recuperar los *top‑K* fragmentos. No explican el compromiso entre dos objetivos que van en direcciones opuestas.

**Alto recall**: Devuelve todo lo que pueda ser relevante. Los usuarios siempre obtienen una respuesta. Pero la ventana de contexto del modelo se llena de ruido tangencial y el modelo alucina para cubrir los vacíos entre fragmentos.

**Alta precisión**: Sólo devuelve los fragmentos más relevantes. El modelo trabaja con contexto limpio y enfocado. Pero si el fragmento correcto no está entre los top 3, el modelo no tiene la información y, sin embargo, inventa algo con confianza.

Los modos de falla se ven idénticos para los usuarios: respuestas incorrectas. Sin embargo, las causas y las soluciones son opuestas.

Dos técnicas que realmente ayudan:

**Reranking**: Recupera más candidatos (top‑20) y luego usa un modelo cross‑encoder para volver a ordenarlos por relevancia antes de pasarlos al LLM. Los cross‑encoders son más lentos que la similitud vectorial, pero mucho más precisos en el paso final de clasificación.

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // Ahora top‑5 realmente significa algo
```

**Búsqueda híbrida**: Combina búsqueda vectorial (similitud semántica) con búsqueda por palabras clave (BM25). Fallan de maneras distintas. La búsqueda vectorial tiene problemas con términos específicos, nombres de modelos e IDs. La búsqueda por palabras clave falla con paráfrasis y sinónimos. Juntas, cubren los puntos ciegos de la otra.

## 4. Tu ventana de contexto tiene la forma equivocada

Has recuperado los fragmentos correctos. Felicidades. El modelo seguirá equivocándose.

El problema no es solo lo que recuperas. Es dónde lo colocas.

Los LLM pueden sufrir el problema de “perderse en el medio”. Liu et al. midieron que los modelos de contexto largo usan la información relevante de forma menos fiable cuando aparece en medio del prompt en lugar de cerca del inicio o del final.

Si estás apretando 20 fragmentos en una lista plana y esperando que el modelo los sintetice correctamente, estás dejando rendimiento sobre la mesa.

Cosas que realmente ayudan:

**Evalúa la posición de inicio/final para tus fragmentos más relevantes.** Una heurística común es colocar el de mayor rango al principio, el de segundo rango al final y el resto en el medio. Contrario a la intuición, pero vale la pena probarlo con tu modelo y la forma de tu prompt.

**Numera y etiqueta explícitamente tus secciones de contexto.** `[Source 1]` ... `[Source 2]` le brinda al modelo anclajes para razonar.

**Añade una señal de confianza de recuperación.** Si tu puntuación de similitud es 0.65 en una escala de 0‑1, indica al modelo: "El contexto siguiente se recuperó con confianza moderada. Reconoce la incertidumbre si la respuesta no está clara."

**Establece un presupuesto de contexto.** No pases todo lo que recuperaste. Cuenta los tokens, prioriza por puntuación de relevancia y corta de forma rígida al 60‑70 % de la ventana de contexto del modelo. Deja espacio para que el modelo razone sin sobrecargarse.

Referencia: [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172).

## 5. No sabes cuándo está equivocado

Este es el fallo silencioso: la respuesta llega, la UI parece saludable y el contenido está mal.

Con una API tradicional, los fallos son visibles: HTTP 500, timeout, error de validación de esquema. Lo sabes al instante. Los fallos de RAG son más discretos: el sistema devuelve una respuesta, parece plausible y está equivocada.

Puede que no sepas que tu pipeline RAG está fallando hasta que los usuarios te lo indiquen. A menudo no lo hacen. Simplemente dejan de confiar en él y vuelven a usar Ctrl + F.

La configuración mínima viable de observabilidad para un sistema RAG en producción:

**Registra tu cadena de recuperación.** Cada consulta, lo que se recuperó (IDs de fragmentos + puntuaciones) y lo que produjo el modelo. Necesitas esto para depurar cualquier cosa.

**Rastrea métricas de recuperación.** Mean Reciprocal Rank (MRR) y NDCG si dispones de etiquetas de referencia. Como mínimo, sigue la distribución de puntuaciones de similitud: si tu puntuación P50 de recuperación cae, la calidad del índice ha disminuido.

**Construye un bucle de retroalimentación.** Incluso un pulgar arriba/abajo en las respuestas, vinculado a la consulta y a los fragmentos recuperados, te brinda señal de entrenamiento. Sin ello, vuelas a ciegas.

**Ejecuta evaluaciones periódicas.** Un conjunto de prueba de 50‑100 preguntas con respuestas correctas conocidas, ejecutado semanalmente, atrapará regresiones antes de que los usuarios lo noten. Una hoja de cálculo y un script son suficientes para comenzar.

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // Alert if score drops below threshold
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## El problema real

Estas fallas no se deben principalmente al modelo de incrustaciones ni a la base de datos vectorial. Se trata del sistema que los rodea.

Las demos de RAG funcionan porque las condiciones de la demo están controladas: documentos limpios, preguntas bien formadas, evaluadores indulgentes. La producción falla porque ninguna de esas condiciones se cumple.

Cada falla descrita es diagnosticable, pero solo si se está midiendo. Los equipos de RAG confiables no hacen nada exótico. Tratan la calidad de la recuperación como un subsistema real, no como un artefacto de demo.

Configure primero el bucle de evaluación. Todo lo demás será más sencillo una vez que pueda medirlo.
