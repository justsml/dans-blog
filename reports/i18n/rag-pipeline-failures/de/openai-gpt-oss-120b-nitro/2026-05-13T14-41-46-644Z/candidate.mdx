Das RAG‑Demo wirkt immer großartig.

Sie betten einige Dokumente ein, starten einen Vektor‑Store, verbinden ihn mit dem Chat und beobachten, wie das Modell Ihre interne Wissensdatenbank zitiert, als hätte es dort jahrelang gearbeitet. Schön. Die Stakeholder sind beeindruckt. Jemand sagt: „Lassen Sie das rausbringen.“

Sechs Wochen später geben die Nutzer selbstbewusst falsche Antworten. Support‑Tickets häufen sich. Das System *funktioniert*, nur nicht so, wie die Nutzer es benötigen.

Der Fehler ist meist nicht ein dramatischer Fehltritt. Es sind fünf langweilige Fehler, die sich addieren.

---

## 1. Ihre Chunks haben die falsche Größe
---

Dieses Versagen führt nicht zu einem Absturz. Es verschlechtert lediglich jede Antwort ein wenig, bis das gesamte Feature als unzuverlässig empfunden wird.

Die Vektorsuche ruft *Chunks* ab, nicht ganze Dokumente. Was immer Sie Ihr Quellmaterial zerlegen, wird zur Wahrheitseinheit des Retrievers. Sind die Chunks falsch, liefert das Modell die falschen Fragmente.

**Zu klein**: Der Chunk enthält nur einen Teil einer Antwort. Das Embedding erfasst das richtige Thema, aber der abgerufene Text fehlt der Kontext. Sie erhalten „Der maximale Timeout beträgt 30 Sekunden“ ohne den vorhergehenden Satz, der lautet „bei Verwendung der Legacy‑API“.

**Zu groß**: Das Embedding wird zu einem unscharfen Durchschnitt vieler Ideen. Die semantische Suche gerät durcheinander, weil der Chunk mehrere Themen abdeckt, und der resultierende Vektor repräsentiert keines davon sauber.

Die passende Chunk‑Größe hängt vollständig von Ihrem Inhalt ab. Technische Dokumentationen, Rechtsverträge und Support‑Transkripte werden jeweils anders chunked. Es gibt keine universelle Lösung.

**Was zu tun ist**: Messen. Erstellen Sie ein Evaluationsset aus Frage‑Antwort‑Paaren aus Ihrem Korpus. Probieren Sie Chunks mit 256, 512 und 1024 Token. Messen Sie die Retrieval‑Präzision: erscheint der richtige Chunk in den Top 5? Sie werden schnell erkennen, dass die Chunk‑Größe wichtiger ist als das Embedding‑Modell, über das Sie sich den Kopf zerbrochen haben.

Auch Überlappungen verwenden. Ein 512‑Token‑Chunk mit 64 Token Überlappung auf jeder Seite sorgt dafür, dass Antworten, die über Chunk‑Grenzen hinweggehen, trotzdem gefunden werden. Die meisten Vektorbibliotheken unterstützen das. Die meisten Leute überspringen es.

---

## 2. Ihre Embeddings werden veraltet (und Sie merken es nicht)

Stellen Sie sich vor, Ihr Unternehmen führt ein Re‑Branding durch. Oder ein Produkt wird umbenannt. Oder die Preisgestaltung wird aktualisiert. Oder eine API wird eingestellt.

Sie aktualisieren die Dokumentation, aber betten die Chunks nicht neu ein. Der Vektor‑Index spiegelt weiterhin den alten Inhalt wider.

Nutzer fragen nach dem neuen Preis. Die Embeddings leiten sie zum alten Inhalt. Das Modell liest den alten Inhalt und erklärt selbstsicher die alte Preisgestaltung. Der Support bekommt ein Ticket.

Jedes ernsthafte RAG‑System stößt irgendwann darauf. Die Lösung klingt offensichtlich – bei Inhaltsänderungen neu einbetten – aber Teams bauen diese Pipeline selten, bevor der erste Vorfall eintritt.

Man braucht inkrementelles Re‑Indexieren mit Inhalts‑Fingerprinting:

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Content unchanged, skip re-embedding
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

Re‑Indexieren beim Schreiben, Fingerprint auf den Inhalt, nicht auf Zeitstempel. Dokumente werden in Ihrem CMS ständig aktualisiert, ohne dass sich der eigentliche Inhalt ändert.

## 3. Retrieval‑Präzision vs. Recall: Sie optimieren das falsche Ziel

Die meisten RAG‑Tutorials zeigen, wie man die Top‑K‑Chunks abruft. Sie erklären nicht den Trade‑off zwischen zwei Zielen, die in entgegengesetzte Richtungen weisen.

**Hoher Recall**: Alles zurückgeben, was potenziell relevant sein könnte. Die Nutzer erhalten immer eine Antwort. Doch das Kontextfenster des Modells ist voller tangentialer Geräusche, und das Modell halluziniert, um die Lücken zwischen den Fragmenten zu füllen.

**Hohe Präzision**: Nur die relevantesten Chunks zurückgeben. Das Modell arbeitet mit sauberem, fokussiertem Kontext. Aber wenn der richtige Chunk nicht zu den Top‑3 gehört, fehlt dem Modell die Information und es erfindet trotzdem selbstbewusst etwas.

Die Fehlermodi sehen für die Nutzer identisch aus: falsche Antworten. Die Ursachen und die Gegenmaßnahmen liegen jedoch diametral gegensätzlich.

Zwei Techniken, die tatsächlich helfen:

**Reranking**: Mehr Kandidaten abrufen (Top‑20) und dann ein Cross‑Encoder‑Modell verwenden, um sie nach Relevanz neu zu ordnen, bevor sie an das LLM übergeben werden. Cross‑Encoder sind langsamer als Vektor‑Ähnlichkeit, liefern aber beim finalen Ranking Schritt deutlich höhere Genauigkeit.

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // Jetzt bedeutet Top‑5 tatsächlich etwas
```

**Hybrid‑Suche**: Vektor‑Suche (semantische Ähnlichkeit) mit Keyword‑Suche (BM25) kombinieren. Sie versagen auf unterschiedliche Weise. Vektor‑Suche hat Probleme mit spezifischen Begriffen, Modellnamen und IDs. Keyword‑Suche hat Schwierigkeiten mit Paraphrasierungen und Synonymen. Zusammen decken sie die blinden Flecken des jeweils anderen ab.

## 4. Ihr Kontextfenster hat die falsche Form

Sie haben die richtigen Chunks abgerufen. Glückwunsch. Das Modell wird das trotzdem falsch machen.

Das Problem ist nicht nur, was Sie abrufen. Es ist, wo Sie es platzieren.

LLMs können das „verloren‑im‑Mitte“-Problem haben. Liu et al. haben gezeigt, dass Langkontext‑Modelle relevante Informationen weniger zuverlässig nutzen, wenn sie in der Mitte des Prompts auftauchen statt nahe am Anfang oder Ende.

Wenn Sie 20 Chunks in eine flache Liste stopfen und hoffen, dass das Modell sie korrekt synthetisiert, lassen Sie Leistung auf dem Tisch.

Dinge, die tatsächlich helfen:

**Bewerten Sie die Start‑/End‑Platzierung Ihrer relevantesten Chunks.** Eine gängige Heuristik ist: am besten bewerteter Chunk zuerst, zweitbester zuletzt und der Rest in der Mitte. Gegenintuitiv, aber einen Test gegenüber Ihrem Modell und Prompt‑Layout wert.

**Nummerieren und beschriften Sie Ihre Kontext‑Abschnitte explizit.** `[Source 1]` … `[Source 2]` gibt dem Modell Anker, an denen es argumentieren kann.

**Fügen Sie ein Retrieval‑Vertrauenssignal hinzu.** Wenn Ihr Ähnlichkeitswert 0,65 auf einer Skala von 0‑1 beträgt, sagen Sie dem Modell: „Der folgende Kontext wurde mit mittlerer Sicherheit abgerufen. Zeigen Sie Unsicherheit, falls die Antwort unklar ist.“

**Setzen Sie ein Kontext‑Budget.** Geben Sie nicht einfach alles weiter, was Sie abgerufen haben. Zählen Sie Tokens, priorisieren Sie nach Relevanz‑Score und schneiden Sie hart bei 60‑70 % des Kontext‑Fensters des Modells zu. Lassen Sie Raum für das Modell, zu argumentieren, ohne zu überladen.

Reference: [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172).

## 5. Sie haben keine Ahnung, wann es falsch ist

Das ist das stille Versagen: Die Antwort kommt, die UI wirkt einwandfrei, aber der Inhalt ist falsch.

Bei einer klassischen API sind Fehler sichtbar: HTTP 500, Timeout, Schema‑Validierungsfehler. Man erkennt sie sofort. RAG‑Fehler sind leiser: Das System liefert eine Antwort, sie wirkt plausibel, und sie ist falsch.

Vielleicht merken Sie erst, dass Ihre RAG‑Pipeline scheitert, wenn Nutzer es Ihnen melden. Oft tun sie das nicht. Sie verlieren einfach das Vertrauen und greifen wieder zu Ctrl + F.

Das minimal funktionsfähige Observability‑Setup für ein produktives RAG‑System:

**Protokollieren Sie Ihre Retrieval‑Kette.** Jede Anfrage, was abgerufen wurde (Chunk‑IDs + Scores) und was das Modell erzeugt hat. Das benötigen Sie, um irgendetwas zu debuggen.

**Verfolgen Sie Retrieval‑Metriken.** Mean Reciprocal Rank (MRR) und NDCG, falls Sie Ground‑Truth‑Labels besitzen. Mindestens sollten Sie die Verteilung der Ähnlichkeits‑Scores beobachten — wenn Ihr P50‑Retrieval‑Score fällt, hat die Index‑Qualität nachgelassen.

**Bauen Sie einen Feedback‑Loop.** Selbst ein Daumen‑hoch/runter‑Feedback zu den Antworten, zurückverknüpft mit der Anfrage und den abgerufenen Chunks, liefert ein Trainingssignal. Ohne das fliegen Sie blind.

**Führen Sie periodische Evaluierungen durch.** Ein Test‑Set von 50‑100 Fragen mit bekannten korrekten Antworten, wöchentlich ausgeführt, fängt Regressionen, bevor Nutzer sie bemerken. Eine Tabellenkalkulation und ein Skript reichen für den Anfang.

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // Alert if score drops below threshold
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## Das eigentliche Problem

Diese Fehler betreffen nicht primär das Embedding‑Modell oder die Vektordatenbank. Sie liegen im System, das sie umgibt.

RAG‑Demos funktionieren, weil die Demo‑Bedingungen kontrolliert sind: saubere Dokumente, wohlgeformte Fragen, nachsichtige Evaluatoren. In der Produktion scheitert es, weil keine dieser Bedingungen mehr zutrifft.

Jeder der oben genannten Fehler ist diagnostizierbar – vorausgesetzt, man misst. Zuverlässige RAG‑Teams setzen nichts Besonderes ein. Sie behandeln die Retrieval‑Qualität wie ein echtes Subsystem statt wie ein Demo‑Artefakt.

Richten Sie zuerst die Evaluationsschleife ein. Alles andere wird einfacher, sobald Sie es messen können.
