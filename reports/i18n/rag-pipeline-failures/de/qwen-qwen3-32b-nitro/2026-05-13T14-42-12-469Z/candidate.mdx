Die RAG-Demo sieht immer großartig aus.  

Sie betten einige Dokumente ein, starten einen Vektorstore, verbinden ihn mit einem Chat und beobachten, wie das Modell auf Ihre interne Wissensbasis verweist, als hätte es dort seit Jahren gearbeitet. Herrlich. Die Stakeholder sind beeindruckt. Jemand sagt: „Lassen Sie uns das veröffentlichen.“  

Sechs Wochen später erhalten die Benutzer überzeugend falsche Antworten. Support-Tickets häufen sich. Das System *funktioniert*, nur nicht in der Weise, wie die Menschen es benötigen.  

Der Fehler ist in der Regel kein dramatischer Einzelfehler. Es sind fünf langweilige Fehler, die sich überschneiden.  

## 1. Ihre Chunk-Größe ist falsch

Dieser Fehler stürzt nicht ab. Er macht nur jede Antwort leicht schlechter, bis das gesamte Feature unvertrauenswürdig wirkt.  

Vektorsearch ruft *Chunks* ab, nicht Dokumente. Jede Aufteilung Ihrer Quellen wird zur Einheit der Wahrheit für den Retriever. Wenn die Chunks falsch sind, antwortet das Modell mit den falschen Fragmenten.  

**Zu klein**: Der Chunk enthält eine unvollständige Antwort. Der Embedding fängt das richtige Thema ein, aber der abgerufene Text fehlt den Kontext. Sie erhalten „Die maximale Timeout-Dauer beträgt 30 Sekunden“, ohne den vorherigen Satz, der „bei Verwendung der Legacy-API“ lautet.  

**Zu groß**: Der Embedding wird zu einem verschwommenen Durchschnitt vieler Ideen. Die semantische Suche wird verwirrt, weil der Chunk über mehrere Dinge spricht, und der resultierende Vektor repräsentiert keins davon sauber.  

Die richtige Chunk-Größe hängt ganz von Ihrem Inhalt ab. Technische Dokumente, Rechtsverträge und Support-Transkripte werden alle anders in Chunks unterteilt. Es gibt keine universelle Antwort.  

Was tun: messen. Erstellen Sie eine Evaluationsmenge aus Frage/Antwort-Paaren aus Ihrem Korpus. Testen Sie 256-, 512- und 1024-Token-Chunks. Messen Sie die Retrieval-Präzision: Erscheint der richtige Chunk in den Top 5? Sie werden schnell lernen, dass die Chunk-Größe wichtiger ist als der Embedding-Modell, über den Sie sich den Kopf zerbrochen haben.

Auch Überlappung nutzen. Ein 512-Token-Chunk mit 64 Token Überlappung auf jeder Seite bedeutet, dass Antworten, die Grenzen überschreiten, dennoch abgerufen werden. Die meisten Vektorbibliotheken unterstützen dies. Die meisten Leute meiden es.

## 2. Ihre Embeddings veralten (Und Sie merken es nicht)

Stellen Sie sich vor, Ihr Unternehmen rebrandet. Oder benennt ein Produkt um. Oder aktualisiert seine Preise. Oder deaktiviert eine API.

Sie aktualisieren die Dokumentation, aber nicht die Embeddings. Der Vektorindex repräsentiert weiterhin den alten Inhalt.

Benutzer fragen nach den neuen Preisen. Die Embeddings leiten sie zum alten Inhalt weiter. Das Modell liest den alten Inhalt und erklärt konsequent die alten Preise. Support erhält eine Anfrage.

Jeder ernsthafte RAG-System stößt früher oder später auf dieses Problem. Die Lösung erscheint offensichtlich – Embeddings neu generieren, wenn sich der Inhalt ändert –, doch Teams bauen diesen Pipeline selten vor dem ersten Zwischenfall aus.

Sie benötigen inkrementelles Re-Indexing mit Inhaltshashierung:

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Inhalt unverändert, Re-Embedding überspringen
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

Indexieren Sie bei Schreibvorgängen neu, hashen Sie den Inhalt, nicht Zeitstempel. Dokumente werden in Ihrem CMS ständig aktualisiert, ohne dass sich der eigentliche Inhalt geändert hat.

## 3. Retrieval-Präzision vs. Recall: Sie optimieren die falsche

Die meisten RAG-Tutorials zeigen, wie man die Top-K-Chunk abruft. Sie erklären aber nicht den Tradeoff zwischen zwei Zielen, die in entgegengesetzte Richtungen zeigen.

**Hohe Recall-Rate**: Gibt alles zurück, was möglicherweise relevant ist. Benutzer erhalten immer eine Antwort. Doch der Kontextfenster des Modells ist voller peripherer Rauschinformationen, und es erfindet Dinge, um die Lücken zwischen Fragmenten zu füllen.

**Hohe Präzision**: Gibt nur die relevantesten Chunks zurück. Das Modell arbeitet mit sauberem, fokussiertem Kontext. Doch wenn der richtige Chunk nicht in den Top-3 ist, hat das Modell die Information nicht und erfindet dennoch etwas, obwohl es sich sicher anmutet.

Die Fehlermodi sehen für Benutzer gleich aus: falsche Antworten. Doch die Ursachen und Lösungen sind gegensätzlich.

Zwei Techniken, die tatsächlich helfen:

**Reranking**: Rufen Sie mehr Kandidaten ab (Top-20), dann verwenden Sie ein Cross-Encoder-Modell, um sie nach Relevanz neu zu rangieren, bevor sie an das LLM weitergegeben werden. Cross-Encoder sind langsamer als Vektorsimilarität, liefern aber deutlich genauere Rangfolgen am Ende.

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // Jetzt bedeutet Top-5 tatsächlich etwas
```

**Hybrid-Suche**: Kombinieren Sie Vektorsuche (semantische Similarität) mit Keywordsuche (BM25). Sie scheitern auf unterschiedliche Weise. Vektorsuche hat Probleme mit spezifischen Begriffen, Modellnamen und IDs. Keywordsuche hat Probleme mit Paraphrasen und Synonymen. Zusammen decken sie sich gegenseitig die Blind spots.

## 4. Ihr Kontextfenster hat die falsche Form  

Sie haben die richtigen Chunke abgerufen. Glückwunsch. Das Modell wird dies dennoch falsch verstehen.  

Das Problem betrifft nicht nur die abgerufenen Inhalte. Es betrifft auch, wo Sie sie platzieren.  

Großsprachmodelle leiden unter dem „lost in the middle“-Problem. Liu et al. stellten fest, dass Modelle mit langen Kontextfenstern relevante Informationen weniger verlässlich verarbeiten, wenn sie sich in der Mitte des Prompts befinden, statt nahe Beginn oder Ende.  

Wenn Sie 20 Chunke in eine flache Liste packen und hoffen, dass das Modell sie korrekt zusammenfasst, lassen Sie Leistungspotenzial ungenutzt.

Dinge, die tatsächlich helfen:

**Bewerten Sie die Platzierung am Anfang/Ende Ihrer wichtigsten Chunks.** Eine gängige Heuristik ist: Top-Ranking zuerst, zweit-Ranking zuletzt und den Rest dazwischen. Kontraintuitiv, aber es lohnt sich, dies an Ihrem Modell und Prompt-Format zu testen.

**Nummerieren und beschriften Sie Ihre Kontextabschnitte explizit.** `[Quelle 1]` ... `[Quelle 2]` gibt dem Modell Ankerpunkte für die Analogisierung.

**Fügen Sie ein Retrieval-Vertrauenssignal hinzu.** Wenn Ihr Ähnlichkeitswert 0,65 auf einer 0-1-Skala beträgt, informieren Sie das Modell: „Der folgende Kontext wurde mit mittlerer Sicherheit abgerufen. Anerkennen Sie Unsicherheit, wenn die Antwort unklar ist.“

**Setzen Sie ein Kontextbudget.** Übergeben Sie nicht einfach alles, was Sie abgerufen haben. Zählen Sie Tokens, priorisieren Sie nach Relevanzscore und schneiden Sie bei 60–70 % des Modellkontextfensters scharf ab. Lassen Sie Platz, damit das Modell ohne unnötiges Herumirren denken kann.

Referenz: [Verloren in der Mitte: Wie Sprachmodelle lange Kontexte nutzen](https://arxiv.org/abs/2307.03172).

## 5. Sie wissen nicht, wann es falsch liegt  

Das ist der leise Fehler: Die Antwort wird zurückgegeben, die Benutzeroberfläche sieht gesund aus und der Inhalt ist falsch.  

Bei einer traditionellen API sind Fehler sichtbar: HTTP 500, Timeout, Schema-Validierungsfehler. Sie erkennen sie sofort. RAG-Fehler sind leiser: Das System gibt eine Antwort zurück, sie wirkt plausibel und ist dennoch falsch.  

Sie erfahren möglicherweise erst dann, dass Ihre RAG-Pipeline fehlschlägt, wenn Benutzer es Ihnen mitteilen. Oft tun sie das nicht. Sie verlieren einfach das Vertrauen und wechseln zurück zu Strg+F.  

Das mindestens erforderliche Beobachtbarkeits-Setup für ein Produktivsystem mit RAG:

**Protokollierung der Retrieval-Kette.** Jede Abfrage, was abgerufen wurde (Chunk-IDs + Scores) und was das Modell erzeugt hat. Dies benötigen Sie, um alles zu debuggen.  

**Tracking von Retrieval-Metriken.** Mittelwert des reziproken Ranks (MRR) und NDCG, wenn Sie Ground-Truth-Labels haben. Mindestens die Verteilung der Ähnlichkeitspunkte verfolgen – fällt der P50-Retrieval-Score ab, sank die Index-Qualität ab.  

**Ein Feedback-Loop aufbauen.** Selbst ein Daumen-hoch/Daumen-runter für Antworten, zurück an die Abfrage und die abgerufenen Chunk-IDs verknüpfen, gibt Ihnen ein Trainingsignal. Ohne das fliegen Sie im Blindflug.  

**Periodische Evaluierungen durchführen.** Eine Testmenge mit 50-100 Fragen mit bekannten korrekten Antworten, wöchentlich durchlaufen, erwischt Regressionsprobleme, bevor Benutzer sie bemerken. Ein Tabellenkalkulationsprogramm und ein Skript reichen aus, um zu beginnen.  

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Bewertungsergebnis: ${(avgScore * 100).toFixed(1)}%`);
  
  // Wenn das Ergebnis unter den Schwellenwert fällt, eine Warnung senden
  if (avgScore < 0.75) {
    await notifyTeam(`RAG-Bewertungsergebnis sank auf ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## Das eigentliche Problem

Diese Fehler sind nicht hauptsächlich auf das Embedding-Modell oder die Vektor-Datenbank zurückzuführen. Es geht um das System, das sie umgibt.

RAG-Demos funktionieren, weil die Demonstrationsbedingungen kontrolliert sind: saubere Dokumente, gut formulierte Fragen, tolerante Bewertung. Im Produktivbetrieb scheitert es, weil keine dieser Bedingungen gilt.

Jeder der oben genannte Fehler ist diagnostizierbar, aber nur, wenn Sie messen. Zuverlässige RAG-Teams tun nichts Exotisches. Sie betrachten die Retrieval-Qualität als echtes Unter-system statt als Demonstrationsartefakt.

Richten Sie zuerst die Evaluierungs-Schleife ein. Alles andere ist einfacher, sobald Sie messen können.
