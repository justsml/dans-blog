Le démon RAG a toujours l’air parfait.

Vous indexez quelques documents, lancez un magasin de vecteurs, le branchez au chat, et vous voyez le modèle citer votre base de connaissances interne comme s’il y travaillait depuis des années. Magnifique. Les parties prenantes sont impressionnées. Quelqu’un lance « déployons ça ».

Six semaines plus tard, les utilisateurs reçoivent des réponses confiantement erronées. Les tickets de support s’accumulent. Le système *fonctionne*, mais pas de la façon dont les gens en ont besoin.

L’échec n’est généralement pas le résultat d’une erreur spectaculaire. C’est cinq erreurs ennuyeuses empilées les unes sur les autres.

---

## 1. Vos fragments ont la mauvaise taille

Cette défaillance ne fait pas planter le système. Elle se contente d’alourdir chaque réponse de façon imperceptible jusqu’à ce que la fonctionnalité entière devienne peu fiable.

La recherche vectorielle récupère *des fragments*, pas des documents entiers. Ce que vous découpez dans votre source devient l’unité de vérité du récupérateur. Si les fragments sont mal dimensionnés, le modèle répond avec les mauvais morceaux.

**Trop petit** : le fragment ne contient qu’une partie de la réponse. L’embedding saisit le bon sujet, mais le texte récupéré manque de contexte. Vous obtenez « Le délai d’expiration maximal est de 30 secondes » sans la phrase précédente qui précise « lors de l’utilisation de l’API hérité ».

**Trop grand** : l’embedding devient une moyenne floue de nombreuses idées. La recherche sémantique se perd parce que le fragment couvre plusieurs sujets, et le vecteur résultant ne représente aucun d’eux de façon nette.

La taille correcte du fragment dépend entièrement de votre contenu. La documentation technique, les contrats juridiques et les transcriptions de support se découpent différemment. Il n’existe pas de réponse universelle.

**À faire** : mesurer. Constituez un jeu d’évaluation de paires question/réponse à partir de votre corpus. Testez des fragments de 256, 512 et 1024 tokens. Mesurez la précision de récupération : le bon fragment apparaît‑il parmi les 5 premiers ? Vous constaterez rapidement que la taille du fragment compte plus que le modèle d’embedding sur lequel vous avez tant insisté.

Utilisez également le chevauchement. Un fragment de 512 tokens avec 64 tokens de chevauchement de chaque côté permet aux réponses qui traversent les limites d’être toujours récupérées. La plupart des bibliothèques de vecteurs le supportent. La plupart des gens l’ignorent.

## 2. Vos embeddings deviennent obsolètes (et vous ne le remarquez pas)

Imaginez que votre entreprise change de marque. Ou renomme un produit. Ou met à jour ses tarifs. Ou déprécie une API.

Vous mettez à jour la documentation mais ne ré‑embeddiez pas les fragments. L’index vectoriel continue de refléter l’ancien contenu.

Les utilisateurs posent des questions sur les nouveaux tarifs. Les embeddings les dirigent vers l’ancien contenu. Le modèle lit cet ancien contenu et explique avec assurance les anciens tarifs. Le support reçoit un ticket.

Chaque système RAG sérieux finit par rencontrer ce problème. La solution paraît évidente : ré‑embeder dès que le contenu change, mais les équipes construisent rarement ce pipeline avant le premier incident.

Il faut un ré‑indexage incrémental avec empreinte du contenu :

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Content unchanged, skip re-embedding
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

Ré‑indexez lors de l’écriture, créez l’empreinte sur le contenu, pas sur les horodatages. Les documents sont mis à jour dans votre CMS en permanence sans que le contenu réel ne change.

## 3. Précision de la récupération vs. Rappel : vous optimisez le mauvais indicateur

La plupart des tutoriels RAG montrent comment récupérer les K meilleurs fragments. Ils n’expliquent pas le compromis entre deux objectifs qui vont dans des directions opposées.

**Rappel élevé** : Retourner tout ce qui pourrait être pertinent. L'utilisateur obtient toujours une réponse. Mais la fenêtre de contexte du modèle est remplie de bruit tangent, et il hallucine pour combler les lacunes entre les fragments.

**Précision élevée** : Ne retourner que les fragments les plus pertinents. Le modèle travaille avec un contexte propre et ciblé. Mais si le bon fragment n'est pas dans le top 3, le modèle n'a pas l'information et invente quand même quelque chose avec assurance.

Les modes d'échec se traduisent de la même façon pour les utilisateurs : réponses erronées. En revanche, les causes et les correctifs sont opposés.

Deux techniques qui fonctionnent réellement :

**Reranking** : récupérer davantage de candidats (top‑20), puis utiliser un modèle cross‑encoder pour les ré‑ordonner par pertinence avant de les transmettre au LLM. Les cross‑encoders sont plus lents que la similarité vectorielle mais nettement plus précis à l'étape finale de classement.

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // Maintenant le top‑5 a réellement du sens
```

**Recherche hybride** : combiner la recherche vectorielle (similarité sémantique) avec la recherche par mots‑clés (BM25). Elles échouent de manières différentes. La recherche vectorielle a du mal avec les termes spécifiques, les noms de modèles et les identifiants. La recherche par mots‑clés a du mal avec les paraphrases et les synonymes. Ensemble, elles comblent les angles morts de l'autre.

## 4. Votre fenêtre de contexte a la mauvaise forme

Vous avez récupéré les bons morceaux. Félicitations. Le modèle va quand même se tromper.

Le problème ne réside pas seulement dans ce que vous récupérez. C’est où vous le placez.

Les LLM peuvent souffrir du problème du « perdu au milieu ». Liu et al. ont constaté que les modèles à long contexte utilisent l’information pertinente de façon moins fiable lorsqu’elle apparaît au milieu de l’invite plutôt qu’au début ou à la fin.

Si vous bourrez 20 morceaux dans une liste plate en espérant que le modèle les synthétise correctement, vous laissez des performances sur la table.

Things that actually help:

**Évaluez le placement début/fin pour vos morceaux les plus pertinents.** Une heuristique courante consiste à placer le mieux classé en premier, le deuxième classé en dernier, et le reste au milieu. Contre‑intuitif, mais cela vaut la peine de le tester avec votre modèle et la forme de votre invite.

**Numérotez et étiquetez explicitement vos sections de contexte.** `[Source 1]` … `[Source 2]` fournit au modèle des ancres pour raisonner.

**Ajoutez un signal de confiance de récupération.** Si votre score de similarité est de 0,65 sur une échelle de 0 à 1, indiquez au modèle : « Le contexte suivant a été récupéré avec une confiance modérée. Reconnaissez l’incertitude si la réponse n’est pas claire. »

**Définissez un budget de contexte.** Ne transmettez pas tout ce que vous avez récupéré. Comptez les jetons, priorisez par score de pertinence, et imposez une coupe stricte à 60‑70 % de la fenêtre de contexte du modèle. Laissez de la marge au modèle pour raisonner sans être submergé.

Reference : [Perdu au milieu : comment les modèles de langage utilisent les longs contextes](https://arxiv.org/abs/2307.03172).

## 5. Vous ne savez pas quand c’est faux

C’est l’échec silencieux : la réponse arrive, l’interface a l’air correcte, mais le contenu est erroné.

Avec une API traditionnelle, les échecs sont visibles : HTTP 500, timeout, erreur de validation de schéma. Vous le savez immédiatement. Les échecs de RAG sont plus discrets : le système renvoie une réponse, elle paraît plausible, et elle est fausse.

Vous ne vous rendrez peut‑être pas compte que votre pipeline RAG échoue avant que les utilisateurs ne le signalent. Souvent, ils ne le font pas. Ils cessent simplement de lui faire confiance et retournent à Ctrl + F.

Le minimum viable d’observabilité pour un système RAG en production :

**Consignez votre chaîne de récupération.** Chaque requête, ce qui a été récupéré (identifiants de morceaux + scores), et ce que le modèle a produit. Vous avez besoin de ces traces pour déboguer quoi que ce soit.

**Suivez les métriques de récupération.** MRR (Mean Reciprocal Rank) et NDCG si vous disposez d’étiquettes de vérité terrain. Au minimum, suivez la distribution des scores de similarité — si votre score de récupération au P50 chute, la qualité de votre index a baissé.

**Mettez en place une boucle de rétroaction.** Même un simple pouce ↑/↓ sur les réponses, rattaché à la requête et aux morceaux récupérés, vous fournit un signal d’entraînement. Sans cela, vous naviguez à l’aveugle.

**Exécutez des évaluations périodiques.** Un jeu de test de 50 à 100 questions avec réponses correctes connues, lancé chaque semaine, détectera les régressions avant les utilisateurs. Un tableau Excel et un script suffisent pour commencer.

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // Alert if score drops below threshold
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## Le vrai problème

Ces échecs ne concernent pas principalement le modèle d’embedding ou la base de vecteurs. Ils proviennent du système qui les entoure.

Les démos RAG fonctionnent parce que les conditions de démonstration sont contrôlées : documents propres, questions bien formées, évaluateurs indulgents. En production, aucune de ces conditions n’est remplie.

Chaque problème ci‑dessus est diagnostiquable, mais seulement si vous le mesurez. Les équipes RAG fiables n’utilisent pas de techniques exotiques. Elles traitent la qualité de la récupération comme un vrai sous‑système plutôt que comme un artefact de démonstration.

Installez d’abord la boucle d’évaluation. Tout le reste devient plus simple une fois que vous pouvez le mesurer.
