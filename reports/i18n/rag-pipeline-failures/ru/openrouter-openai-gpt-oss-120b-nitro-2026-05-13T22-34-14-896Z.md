RAG‑демо всегда выглядит впечатляюще.

Вы встраиваете несколько документов, поднимаете векторное хранилище, подключаете его к чату и наблюдаете, как модель ссылается на вашу внутреннюю базу знаний, будто она работает с ней годами. Красиво. Заинтересованные стороны в восторге. Кто‑то говорит: «давайте выпустим это».

Через шесть недель пользователи получают уверенно неправильные ответы. Тикеты в поддержку растут. Система *работает*, просто не так, как этого требуют пользователи.

Проблема обычно не в одной драматичной ошибке. Это пять скучных ошибок, наложившихся друг на друга.

## 1. Размер ваших чанков неверен

Эта ошибка не приводит к краху. Она просто делает каждый ответ чуть хуже, пока вся функция не начнёт казаться ненадёжной.

Векторный поиск извлекает *чанки*, а не документы. Всё, на что вы разбиваете исходный материал, становится единицей истины для ретривера. Если чанки неверны, модель отвечает неправильными фрагментами.

**Слишком маленькие**: Чанк содержит лишь часть ответа. Эмбеддинг захватывает правильную тему, но извлечённый текст лишён контекста. Вы получаете «Максимальный тайм‑аут — 30 секунд» без предшествующего предложения, в котором говорится «при использовании устаревшего API».

**Слишком большие**: Эмбеддинг превращается в размытый средний вектор множества идей. Семантический поиск путается, потому что чанк охватывает несколько тем, и полученный вектор не представляет ни одну из них чётко.

Размер чанка зависит полностью от вашего контента. Техническая документация, юридические контракты и транскрипты поддержки требуют разных подходов к разбиению. Универсального решения нет.

Что делать: измерять. Сформируйте набор тестовых вопросов/ответов из вашего корпуса. Попробуйте чанки в 256, 512 и 1024 токена. Оцените точность поиска: попадает ли нужный чанк в топ‑5? Вы быстро поймёте, что размер чанка важнее, чем выбранная модель эмбеддингов, над которой вы так мучились.

Также используйте перекрытие. Чанк в 512 токенов с 64‑токенным перекрытием по обе стороны гарантирует, что ответы, охватывающие границу, всё равно будут найдены. Большинство векторных библиотек поддерживают эту возможность. Большинство людей её игнорируют.

## 2. Ваши эмбеддинги устаревают (и вы этого не заметите)

Представьте, что ваша компания меняет бренд. Или переименовывает продукт. Или обновляет цены. Или выводит из эксплуатации API.

Вы обновляете документацию, но не переэмбеддите чанки. Векторный индекс всё ещё отражает старый контент.

Пользователи спрашивают о новых ценах. Эмбеддинги направляют их к старой информации. Модель читает старый контент и уверенно объясняет устаревшие цены. Служба поддержки получает тикет.

Каждая серьёзная RAG‑система рано или поздно сталкивается с этим. Решение кажется очевидным — переэмбеддить при изменении контента — но команды редко строят такой конвейер до первого инцидента.

Нужна инкрементальная переиндексация с отпечатками содержимого:


```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Content unchanged, skip re-embedding
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

Переиндексация при записи, отпечаток — по содержимому, а не по меткам времени. Документы в вашей CMS часто обновляются, хотя фактическое содержание остаётся тем же.

## 3. Точность vs. Полнота поиска: Вы оптимизируете не то

Большинство руководств по RAG показывают, как получить топ‑K фрагментов. Они не объясняют компромисс между двумя целями, которые идут в противоположных направлениях.

**Высокая полнота**: Возвращать всё, что может быть релевантным. Пользователь всегда получает ответ. Но окно контекста модели заполняется побочным шумом, и модель начинает «галлюцинировать», заполняя пробелы между фрагментами.

**Высокая точность**: Возвращать только самые релевантные фрагменты. Модель работает с чистым, сфокусированным контекстом. Однако если нужный фрагмент не попал в топ‑3, у модели нет информации и она всё равно уверенно придумывает ответ.

С точки зрения пользователя оба режима выглядят одинаково — неправильные ответы. Причины и решения, однако, противоположны.

Два подхода, которые действительно помогают:

**Перерейтинг**: Получить больше кандидатов (топ‑20), а затем с помощью кросс‑энкодера переупорядочить их по релевантности перед передачей в LLM. Кросс‑энкодеры медленнее векторного сравнения, но значительно точнее на финальном этапе ранжирования.

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // Теперь топ‑5 действительно имеет смысл
```

**Гибридный поиск**: Сочетать векторный поиск (семантическое сходство) с поиском по ключевым словам (BM25). Они дают сбои по‑разному. Векторный поиск хуже справляется с конкретными терминами, названиями моделей и идентификаторами. Поиск по ключевым словам хуже работает с перефразировками и синонимами. Вместе они покрывают слепые зоны друг друга.

##4. Ваше окно контекста имеет неправильную форму

Вы получили нужные фрагменты. Поздравляем. Модель всё равно выдаст ошибку.

Проблема заключается не только в том, что вы извлекаете, а в том, где вы это размещаете.

LLM могут сталкиваться с проблемой «потерялись в середине». Liu et al. показали, что модели с длинным контекстом используют релевантную информацию менее надёжно, когда она находится в середине подсказки, а не ближе к началу или концу.

Если вы просто складываете 20 фрагментов в плоский список и надеетесь, что модель правильно их синтезирует, вы теряете производительность.

То, что действительно помогает:

**Оцените размещение начала/конца для самых релевантных фрагментов.** Распространённый эвристический приём — помещать фрагмент с наивысшим рангом в начало, второй по рангу — в конец, а остальные — в середину. Противоинтуитивно, но стоит проверить на вашей модели и с вашей формой подсказки.

**Явно нумеруйте и помечайте секции контекста.** `[Source 1]` … `[Source 2]` дают модели якоря для рассуждений.

**Добавьте сигнал уверенности извлечения.** Если ваш коэффициент сходства = 0,65 по шкале от 0 до 1, сообщите модели: «Следующий контекст был получен с умеренной уверенностью. Признайте неопределённость, если ответ не ясен».

**Установите бюджет контекста.** Не передавайте модели всё, что вы нашли. Считайте токены, приоритизируйте по оценке релевантности и жёстко обрезайте до 60‑70 % окна контекста модели. Оставьте место модели для рассуждений, а не для «переполнения».

Reference: [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172).

## 5. Вы Не Знаете, Когда Всё Не Так

Это тихий провал: ответ возвращён, UI выглядит нормально, а содержание ошибочно.

При традиционном API сбои видны: HTTP 500, тайм‑аут, ошибка валидации схемы. Вы узнаёте об этом сразу. Ошибки RAG тише: система отдаёт ответ, он выглядит правдоподобно, но неверен.

Вы можете не понять, что ваш конвейер RAG падает, пока пользователи не укажут на это. Часто они этого не делают. Они просто перестают доверять системе и возвращаются к Ctrl+F.

Минимальный набор наблюдаемости для продакшн‑системы RAG:

**Логируйте цепочку извлечения.** Каждый запрос, что было извлечено (идентификаторы чанков + оценки) и что сгенерировала модель. Это необходимо для отладки любой проблемы.

**Отслеживайте метрики извлечения.** Средний обратный ранг (MRR) и NDCG, если у вас есть эталонные метки. Как минимум, следите за распределением оценок сходства — если ваш P50‑балл снижается, качество индекса ухудшилось.

**Создайте обратную связь.** Даже простое «палец вверх/вниз» к ответам, привязанное к запросу и извлечённым чанкам, даёт вам обучающий сигнал. Без этого вы летите вслепую.

**Проводите периодические оценки.** Тестовый набор из 50‑100 вопросов с известными правильными ответами, запускаемый еженедельно, поймает регрессию раньше, чем пользователи. Достаточно таблицы и небольшого скрипта, чтобы начать.

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // Alert if score drops below threshold
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## Реальная проблема

Эти сбои в основном не связаны с моделью эмбеддингов или векторной базой данных. Они касаются окружающей системы.

Демонстрации RAG работают, потому что условия демо контролируются: чистые документы, корректно сформулированные вопросы, снисходительные оценщики. На продакшене всё падает, потому что ни одно из этих условий не соблюдается.

Каждый из перечисленных выше провалов диагностируем, но только при наличии измерений. Надёжные команды RAG не используют ничего экзотического. Они рассматривают качество извлечения как реальный подсистемный компонент, а не артефакт демо.

Сначала настройте цикл оценки. Всё остальное станет проще, как только появятся измерения.
