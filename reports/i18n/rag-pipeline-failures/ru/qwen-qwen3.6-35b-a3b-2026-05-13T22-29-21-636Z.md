Демонстрация RAG всегда выглядит безупречно.

Вы делаете эмбеддинги документов, поднимаете векторное хранилище, подключаете это к чату и смотрите, как модель цитирует вашу внутреннюю базу знаний так, будто она там работает годами. Красиво. Стейкхолдеры впечатлены. Кто-то говорит: «выкатываем в прод».

Спустя шесть недель пользователи получают уверенные, но неверные ответы. Тикеты в поддержку копятся. Система *работает*, просто не так, как нужно людям.

Проблема обычно не в одной драматичной ошибке. Это пять скучных недочётов, сложенных друг на друга.

## 1. Неправильный размер чанков

Эта ошибка не вызывает краш. Она просто постепенно ухудшает каждый ответ, пока вся фича не начнёт казаться ненадёжной.

Векторный поиск извлекает *чанки*, а не документы. Всё, на что вы разбили исходные материалы, становится для ривера единственной точкой истины. Если чанки подобраны неверно, модель будет отвечать на основе неправильных фрагментов.

**Слишком маленькие**: чанк содержит лишь часть ответа. Эмбеддинг захватывает правильную тему, но в извлечённом тексте не хватает контекста. Вы получаете «Максимальный таймаут составляет 30 секунд», упуская предшествующее предложение «при использовании legacy API».

**Слишком большие**: эмбеддинг превращается в размытое усреднение множества идей. Семантический поиск сбивается с толку, потому что чанк охватывает сразу несколько тем, а итоговый вектор не отражает ни одну из них чётко.

Правильный размер чанка полностью зависит от вашего контента. Технические документы, юридические контракты и транскрипты поддержки разбиваются по-разному. Универсального решения нет.

Что делать: замеряйте. Соберите eval-набор пар «вопрос-ответ» на основе вашего корпуса. Попробуйте чанки на 256, 512 и 1024 токена. Оцените precision при поиске: попадает ли релевантный чанк в топ-5? Вы быстро поймёте, что размер чанка влияет сильнее, чем модель эмбеддинга, которой вы так долго придавали значение.

Используйте перекрытие. Чанк на 512 токенов с перекрытием по 64 токена с каждой стороны гарантирует, что ответы, разорванные границей чанка, всё равно попадут в выдачу. Большинство векторных библиотек это поддерживают. Большинство разработчиков этим пренебрегают.

## 2. Ваши эмбеддинги устаревают (и вы этого не заметите)

Представьте, что компания меняет бренд. Или переименовывает продукт. Или обновляет тарифы. Или снимает API с поддержки.

Вы обновляете документацию, но не переэмбеддите чанки. Векторный индекс по-прежнему отражает старые данные.

Пользователи спрашивают о новых тарифах. Эмбеддинги направляют их к старому контенту. Модель читает устаревшие данные и уверенно объясняет старые цены. В поддержку приходит тикет.

Каждой серьёзной RAG-системе рано или поздно приходится с этим столкнуться. Решение звучит очевидно — переэмбеддите при изменении контента — но команды редко выстраивают этот пайплайн до первого инцидента.

Вам потребуется инкрементальная переиндексация с контентным хешированием:

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Content unchanged, skip re-embedding
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

Переиндексируйте при записи, вычисляйте отпечаток по содержимому, а не по меткам времени. Документы в CMS часто обновляются без фактического изменения текста.

## 3. Precision vs. Recall: вы оптимизируете не ту метрику

В большинстве туториалов по RAG показывают, как извлекать топ-K чанков. При этом не объясняется компромисс между двумя целями, которые работают в противоположных направлениях.

**Высокий recall**: Возвращайте всё, что хоть как-то может быть релевантно. Пользователь всегда получает ответ. Но контекстное окно модели забивается шумом с косвенным отношением, и она начинает галлюцинировать, пытаясь заполнить пробелы между фрагментами.

**Высокий precision**: Возвращайте только наиболее релевантные чанки. Модель оперирует чистым, сфокусированным контекстом. Но если нужный чанк не попадает в топ-3, у модели нет данных, и она всё равно уверенно генерирует выдуманный ответ.

Для пользователя режимы сбоев выглядят одинаково: неверные ответы. Но причины и способы их устранения — диаметрально противоположны.

Две техники, которые реально работают:

**Реранкинг**: Забирайте больше кандидатов (top-20), а затем используйте модель cross-encoder для переранжирования по релевантности перед передачей в LLM. Cross-encoders работают медленнее, чем векторное сходство, но дают значительно более точный результат на финальном этапе ранжирования.

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // Now top-5 actually means something
```

**Гибридный поиск**: Комбинируйте векторный поиск (семантическое сходство) с поиском по ключевым словам (BM25). Они дают сбои по-разному. Векторный поиск плохо справляется с конкретными терминами, названиями моделей и идентификаторами. Поиск по ключевым словам плохо работает с парафразами и синонимами. Вместе они закрывают слепые зоны друг друга.

## 4. Окно контекста имеет неправильную структуру

Вы извлекли нужные чанки. Поздравляю. Модель всё равно допустит ошибку.

Проблема не только в том, что вы извлекаете. Она в том, куда вы это помещаете.

LLM подвержены эффекту «потеря в середине» (lost in the middle). Liu et al. зафиксировали, что модели с длинным контекстом реже корректно используют релевантную информацию, если она находится в середине промпта, а не в начале или в конце.

Если вы просто сваливаете 20 чанков в плоский список и надеетесь, что модель их корректно синтезирует, вы теряете в качестве ответов.

Что реально помогает:

**Оцените размещение самых релевантных чанков в начале и в конце.** Распространённое эвристическое правило: фрагмент с наивысшим рангом — в начало, второй по рангу — в конец, остальные — в середину. Контринтуитивно, но стоит протестировать на вашей модели и структуре промпта.

**Явно нумеруйте и подписывайте секции контекста.** `[Source 1]` ... `[Source 2]` даёт модели якоря для рассуждений.

**Добавьте сигнал уверенности при извлечении.** Если ваш скор сходства равен 0.65 по шкале от 0 до 1, сообщите модели: «Следующий контекст извлечён с умеренной уверенностью. Укажите на неопределённость, если ответ неочевиден».

**Задайте бюджет контекста.** Не передавайте модели всё, что извлекли. Считайте токены, сортируйте по баллу релевантности и жёстко обрезайте на 60–70% от окна контекста модели. Оставьте запас для рассуждений, не доводя до перегрузки контекста.

Ссылка: [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172).

## 5. Вы не понимаете, когда система ошибается

Это тихий сбой: ответ возвращается, интерфейс выглядит стабильно, а содержание неверно.

В традиционном API сбои заметны: HTTP 500, таймаут, ошибка валидации схемы. Вы узнаете об этом мгновенно. Сбои в RAG тише: система возвращает ответ, он выглядит правдоподобно, но он неверен.

Вы можете не узнать о падении качества пайплайна RAG, пока пользователи не скажут об этом. А часто они молчат. Они просто перестают доверять системе и возвращаются к Ctrl+F.

Минимально жизнеспособный набор средств наблюдаемости для продакшн-системы RAG:

**Логгируйте цепочку извлечения.** Каждый запрос, что было извлечено (ID чанков + баллы) и что сгенерировала модель. Без этого отладка невозможна.

**Отслеживайте метрики извлечения.** Mean Reciprocal Rank (MRR) и NDCG, если есть эталонная разметка (ground-truth). Как минимум отслеживайте распределение баллов сходства: если P50 падает, значит, упало качество индекса.

**Настройте цикл обратной связи.** Даже простая оценка ответа «лайк/дизлайк», привязанная к запросу и извлечённым чанкам, даёт обучающий сигнал. Без него вы работаете вслепую.

**Запускайте периодические оценки.** Набор из 50–100 вопросов с известными правильными ответами, запускаемый еженедельно, позволит выявить регрессию раньше пользователей. Для старта достаточно таблицы и скрипта.

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // Alert if score drops below threshold
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## Реальная проблема

Эти сбои связаны не столько с моделью эмбеддингов или векторной БД, сколько с системой вокруг них.

Демо RAG работают, потому что условия строго контролируемые: чистые документы, корректные вопросы, снисходительные оценщики. В продакшене ни одно из этих условий не соблюдается.

Каждый описанный выше сбой диагностируем, но только если вы ведёте измерения. Команды, выстраивающие надёжный RAG, не используют ничего экзотического. Они рассматривают качество извлечения как полноценную подсистему, а не как демонстрационный артефакт.

Сначала настройте цикл оценки. Всё остальное станет проще, как только вы начнёте измерять.
