## 1. Размер ваших чанков неверный

Эта ошибка не приводит к падению системы. Она просто делает каждый ответ чуть хуже, пока вся функция не начинает казаться ненадёжной.

Векторный поиск извлекает *чанки*, а не документы. То, на что вы разбиваете исходный материал, становится единицей истины для поисковика. Если чанки неправильные, модель отвечает неверными фрагментами.

**Слишком маленький**: Чанк содержит частичный ответ. Эмбеддинг захватывает правильную тему, но извлечённому тексту не хватает контекста. Вы получаете «Максимальный тайм-аут составляет 30 секунд» без предшествующего предложения «при использовании устаревшего API».

**Слишком большой**: Эмбеддинг становится размытым средним многих идей. Семантический поиск путается, потому что чанк охватывает несколько тем, и результирующий вектор не отражает ни одной из них чётко.

Правильный размер чанка полностью зависит от вашего контента. Техническая документация, юридические договоры и записи поддержки — всё требует разного дробления. Универсального ответа нет.

Что делать: измерять. Соберите eval-набор пар «вопрос/ответ» из вашего корпуса. Попробуйте чанки по 256, 512 и 1024 токена. Замерьте точность поиска: попадает ли нужный чанк в топ-5? Вы быстро поймёте, что размер чанка важнее, чем модель эмбеддингов, над выбором которой вы мучились.

Также используйте перекрытие. Чанк в 512 токенов с перекрытием 64 токена с каждой стороны гарантирует, что ответы, задевающие границы, всё равно будут найдены. Большинство векторных библиотек это поддерживают. Большинство людей этим не пользуются.

## 2. Ваши эмбеддинги устаревают (и вы этого не заметите)

Представьте: компания провела ребрендинг. Или переименовала продукт. Или обновила цены. Или объявила API устаревшим.

Вы обновляете документацию, но не перестраиваете эмбеддинги для чанков. Векторный индекс по-прежнему содержит старый контент.

Пользователи спрашивают про новые цены. Эмбеддинги направляют их к старому контенту. Модель читает старый контент и уверенно объясняет старые цены. Служба поддержки получает тикет.

Любая серьёзная RAG-система рано или поздно с этим сталкивается. Исправление звучит очевидно — перестраивать эмбеддинги при изменении контента, — но команды редко строят такой конвейер до первого инцидента.

Вам нужно инкрементальное переиндексирование с fingerprinting контента:

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Content unchanged, skip re-embedding
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

Переиндексация на запись, fingerprint по содержимому, а не по временным меткам. Документы в вашей CMS обновляются постоянно без изменения самого контента.

## 3. Точность и полнота поиска: вы оптимизируете не то

Большинство RAG-туториалов показывают, как извлекать top-K чанков. Они не объясняют компромисс между двумя противоположно направленными целями.

**Высокая полнота**: вернуть всё, что может быть релевантно. Пользователь всегда получает ответ. Но контекстное окно модели забивается шумом по касательной, и она галлюцинирует, заполняя пробелы между фрагментами.

**Высокая точность**: вернуть только самые релевантные чанки. Модель работает с чистым, сфокусированным контекстом. Но если нужный чанк не попал в top-3, у модели нет информации, и она с уверенностью сочиняет ответ.

Пользователи видят одинаковый сбой: неверные ответы. Но причины — и способы исправления — противоположны.

Две техники, которые действительно помогают:

**Ранжирование (Reranking)**: извлеките больше кандидатов (top-20), затем используйте cross-encoder модель, чтобы переранжировать их по релевантности перед передачей в LLM. Cross-encoder'ы медленнее векторного поиска, но на финальном шаге ранжирования работают значительно точнее.

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // Теперь top-5 действительно что-то значит
```

**Гибридный поиск**: объедините векторный поиск (семантическая близость) с поиском по ключевым словам (BM25). Они ошибаются по-разному. Векторный поиск плохо справляется с конкретными терминами, названиями моделей и идентификаторами. Поиск по ключевым словам — с перефразированием и синонимами. Вместе они закрывают слепые зоны друг друга.

## 4. Ваше окно контекста имеет неправильную форму

Вы получили правильные чанки. Поздравляю. Модель всё равно ошибётся.

Проблема не только в том, что вы извлекли. А в том, куда вы это поместили.

LLM могут страдать от проблемы «потеряно в середине». Лю и коллеги измерили, что модели с длинным контекстом менее надёжно используют релевантную информацию, когда она появляется в середине промпта, а не в начале или конце.

Если вы заталкиваете 20 чанков в плоский список и надеетесь, что модель правильно их синтезирует, вы оставляете производительность на столе.

Что действительно помогает:

**Оцените размещение наиболее релевантных чанков в начале/конце.** Распространённая эвристика — самый релевантный первым, второй по релевантности последним, остальные в середине. Неожиданно, но стоит протестировать на вашей модели и форме промпта.

**Явно нумеруйте и помечайте секции контекста.** `[Source 1]` ... `[Source 2]` даёт модели якоря для рассуждения.

**Добавьте сигнал уверенности в результате поиска.** Если ваш балл схожести составляет 0.65 по шкале от 0 до 1, скажите модели: «Следующий контекст был получен с умеренной уверенностью. Укажите на неопределённость, если ответ неясен.»

**Установите бюджет контекста.** Не передавайте всё, что получили. Считайте токены, сортируйте по баллу релевантности и жёстко обрезайте на 60–70% контекстного окна модели. Оставьте пространство для рассуждений без перегрузки.

Reference: [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172)

## 5. Вы понятия не имеете, когда он ошибается

Это тихий сбой: ответ возвращается, UI выглядит здоровым, а содержимое неверно.

С традиционным API сбои видны: HTTP 500, таймаут, ошибка валидации схемы. Вы знаете сразу. Сбои RAG тише: система возвращает ответ, он выглядит правдоподобно, но неверен.

Вы можете не знать, что ваш RAG-конвейер даёт сбой, пока пользователи не скажут вам. Часто они этого не делают. Они просто перестают ему доверять и возвращаются к Ctrl+F.

Минимальный жизнеспособный набор наблюдения для продакшен RAG-системы:

**Логируйте цепочку поиска.** Каждый запрос, что было извлечено (ID чанков + баллы) и что модель сгенерировала. Это нужно для отладки.

**Отслеживайте метрики поиска.** Средний обратный ранг (MRR) и NDCG, если у вас есть эталонные метки. Как минимум, отслеживайте распределение баллов схожести — если ваш P50 retrieval score падает, качество вашего индекса упало.

**Создайте цикл обратной связи.** Даже оценка «палец вверх/вниз» к ответам, привязанная к запросу и извлечённым чанкам, даёт сигнал для обучения. Без этого вы летите вслепую.

**Запускайте периодические оценки.** Тестовый набор из 50–100 вопросов с известными правильными ответами, запускаемый еженедельно, выявит регрессию до того, как её заметят пользователи. Для начала хватит таблицы и скрипта.

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // Alert if score drops below threshold
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## Настоящая проблема

Эти сбои связаны не столько с моделью эмбеддингов или векторной базой данных. Они связаны с системой вокруг них.

Демо RAG работают, потому что демо-условия контролируются: чистые документы, хорошо сформулированные вопросы, снисходительные оценщики. Продакшен ломается, потому что ни одно из этих условий не выполняется.

Каждый из описанных выше сбоев диагностируем, но только если вы измеряете. Надёжные RAG-команды не делают ничего экзотического. Они относятся к качеству поиска как к реальной подсистеме, а не как к демо-артефакту.

Сначала настройте цикл оценки. Всё остальное становится проще, когда вы можете это измерить.
