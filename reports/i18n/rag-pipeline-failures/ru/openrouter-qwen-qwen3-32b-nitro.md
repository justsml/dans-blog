Демонстрация RAG всегда выглядит отлично. Вы встраиваете несколько документов, запускаете векторное хранилище, подключаете его к чату и наблюдаете, как модель ссылается на вашу внутреннюю базу знаний, как будто работает там много лет. Прекрасно. Заинтересованные стороны остаются впечатлены. Кто-то говорит: «Давайте запустим это».  

Через шесть недель пользователи получают уверенно неправильные ответы. Заявки в службу поддержки растут. Система *работает*, просто не так, как люди от нее ожидают.  

Сбой обычно не является одной драматичной ошибкой. Это пять скучных ошибок, накопленных вместе.  

## 1. Ваш размер фрагментов неправильный

Эта ошибка не вызывает сбоев. Она просто делает каждый ответ немного хуже, пока вся функция не станет ненадежной.

Поиск векторов извлекает *фрагменты*, а не документы. То, на что вы разбили исходный материал, становится единицей правды для извлекающего. Если фрагменты неправильные, модель отвечает неправильными частями.

**Слишком маленькие**: Фрагмент содержит частичный ответ. Вложение отражает правильную тему, но извлеченный текст теряет контекст. Вы получаете «Максимальный таймаут — 30 секунд», не имея предыдущего предложения, в котором говорится «при использовании устаревшего API».

**Слишком большие**: Вложение становится расплывчатым средним множества идей. Семантический поиск запутывается, потому что фрагмент охватывает несколько тем, а результирующий вектор не представляет ни одну из них четко.

Правильный размер фрагментов зависит исключительно от содержимого. Техническая документация, юридические контракты и транскрипты поддержки требуют разных подходов к разбивке. Универсального решения нет.

Что делать: измеряйте. Создайте набор оценок (eval set) пар вопрос-ответ из вашего корпуса. Попробуйте фрагменты размером 256, 512 и 1024 токена. Измерьте точность извлечения: появляется ли правильный фрагмент в топ-5? Вы быстро поймете, что размер фрагментов важнее, чем модель вложений, над которой вы мучились.

Также используйте перекрытие. Фрагмент из 512 токенов с 64 токенами перекрытия с каждой стороны гарантирует, что ответы, пересекающие границы фрагментов, всё ещё будут извлечены. Большинство векторных библиотек поддерживают это. Большинство людей пропускают этот шаг.

## 2. Ваши векторные представления устаревают (И вы этого не заметите)

Представьте, что ваша компания меняет бренд. Или переименовывает продукт. Или обновляет цены. Или устаревает API.

Вы обновляете документацию, но не пересоздаёте векторные представления фрагментов. Векторный индекс всё ещё представляет старый контент.

Пользователи спрашивают о новых ценах. Векторные представления направляют их к старому контенту. Модель читает старый контент и уверенно объясняет старые цены. Поддержка получает заявку.

Каждая серьезная система RAG рано или поздно сталкивается с этим. Фикс кажется очевидным - пересоздавать векторы при изменении контента - но редко строят этот пайплайн до первого инцидента.

Вам нужна постепенная переиндексация с отпечатками контента:

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Content unchanged, skip re-embedding
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

Переиндексируйте при записи, отпечатывайте по содержимому, а не по временным меткам. Документы обновляются в вашем CMS постоянно, без изменения реального контента.

## 3. Точность и полнота поиска: Вы оптимизируете неправильную из них

Большинство туториалов по RAG показывают, как извлекать топ-K фрагментов. Они не объясняют компромисс между двумя целями, которые указывают в противоположные стороны.

**Высокая полнота**: Возвращайте всё, что может быть связано. Пользователи всегда получают ответ. Но контекстное окно модели переполнено косвенно связанным шумом, и модель выдумывает, чтобы заполнить промежутки между фрагментами.

**Высокая точность**: Возвращайте только самые релевантные фрагменты. Модель работает с чистым, сфокусированным контекстом. Но если нужный фрагмент не входит в топ-3, модель не имеет информации и всё равно уверенно выдумывает что-то своё.

Режимы сбоя выглядят одинаково для пользователей: неправильные ответы. Но причины и решения противоположны.

Два действительно полезных метода:

**Пересортировка**: Извлеките больше кандидатов (топ-20), затем используйте модель cross-encoder для пересортировки их по релевантности перед передачей в LLM. Cross-encoderы медленнее, чем векторная схожесть, но значительно точнее на финальном этапе ранжирования.

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // Теперь топ-5 действительно что-то значит
```

**Гибридный поиск**: Соедините векторный поиск (семантическую схожесть) с ключевым поиском (BM25). Они сбоют по-разному. Векторный поиск плохо справляется со специфическими терминами, названиями моделей и ID. Ключевой поиск плохо справляется с перефразировкой и синонимами. Вместе они покрывают слепые пятна друг друга.

## 4. Ваше контекстное окно имеет неправильную форму

Вы извлекли правильные чанки. Поздравляем. Модель всё равно ошибётся.

Проблема не только в том, что вы извлекаете. Это в том, где вы размещаете.

LLM могут страдать от проблемы «потеря в середине». Liu и др. измеряли долгие контекстные модели, которые менее надёжно использовали релевантную информацию, когда она появлялась в середине подсказки, а не в начале или конце.

Если вы загоняете 20 чанков в плоский список и надеетесь, что модель правильно их синтезирует, вы оставляете производительность на столе.

Факты, которые действительно помогают:

**Оцените размещение начала/конца для ваших самых релевантных чанков.** Часто используемый эвристический подход — самый высокий ранг в начале, второй по рангу в конце, а остальные в середине. Контринтуитивно, но стоит протестировать это с вашей моделью и формой подсказки.

**Явно нумеруйте и подписывайте разделы контекста.** `[Источник 1]` ... `[Источник 2]` даёт модели якоря для рассуждений.

**Добавьте сигнал уверенности в извлечение.** Если ваш показатель сходства составляет 0.65 на шкале от 0 до 1, сообщите модели: "Следующий контекст был извлечён с умеренной уверенностью. Признайте неопределённость, если ответ неясен."

**Установите бюджет контекста.** Не передавайте всё, что вы извлекли. Считайте токены, приоритизируйте по показателю релевантности и строго ограничьте на 60-70% окна контекста модели. Оставьте пространство для рассуждений модели без перегрузки.

Ссылка: [Потеря в середине: Как языковые модели используют длинные контексты](https://arxiv.org/abs/2307.03172).

## 5. Вы не знаете, когда это неправильно

Это тихий сбой: ответ возвращается, интерфейс выглядит исправно, а содержание ошибочно.

В традиционных API сбои явны: HTTP 50, таймаут, ошибка валидации схемы. Вы сразу понимаете проблему. Сбои RAG тише: система возвращает ответ, он кажется правдоподобным, и он неправильный.

Вы можете не узнать, что ваш RAG-пайплайн сломался, пока пользователи не сообщат об этом. Часто они не сообщают. Они просто перестают доверять системе и возвращаются к Ctrl+F.

Минимальная жизнеспособная настройка наблюдаемости для RAG-системы в production:
---

**Записывайте цепочку извлечения.** Для каждой запроса фиксируйте, что было извлечено (ID фрагментов + оценки), и что модель выдала. Это необходимо для отладки чего-либо.

**Отслеживайте метрики извлечения.** Среднее обратное ранг (MRR) и NDCG, если у вас есть метки правильных ответов. В крайнем случае, отслеживайте распределения оценок схожести — если медианный показатель извлечения падает, качество индекса ухудшается.

**Постройте обратную связь.** Даже простой лайк/дизлайк к ответам, связанный с запросом и извлеченными фрагментами, дает сигнал для обучения. Без этого вы действуете вслепую.

**Запускайте периодическую оценку.** Тестовый набор из 50-100 вопросов с известными правильными ответами, запускаемый еженедельно, выявит регрессию до того, как пользователи. Таблица и скрипт — достаточно для старта.

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // Alert if score drops below threshold
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## Самая настоящая проблема

Эти сбои в основном не связаны с моделью встраивания или векторной базой данных. Они связаны с системой вокруг них.

Демонстрации RAG работают, потому что условия демонстрации контролируются: чистые документы, хорошо сформулированные вопросы, доброжелательные оценщики. Сбои в производстве происходят потому, что ни одно из этих условий не выполняется.

Каждый из вышеуказанных сбоев можно диагностировать, но только если вы измеряете. Надежные команды RAG не делают ничего экзотического. Они рассматривают качество извлечения как реальную подсистему, а не как артефакт демонстрации.

Установите цикл оценки первым делом. Все остальное становится проще, когда вы можете измерить это.
