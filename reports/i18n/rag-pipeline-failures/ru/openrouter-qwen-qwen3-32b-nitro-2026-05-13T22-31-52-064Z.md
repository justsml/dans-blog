Демонстрация RAG всегда выглядит отлично.  

Вы встраиваете несколько документов, запускаете векторное хранилище, подключаете к чату и наблюдаете, как модель цитирует вашу внутреннюю базу знаний, словно она там работала годами. Прекрасно. Заказчики впечатлены. Кто-то говорит: «Давайте запустим это в продакшн».  

Через шесть недель пользователи получают уверенно неправильные ответы. Поток обращений в поддержку растет. Система *работает*, просто не так, как люди в ней нуждаются.  

Ошибка почти никогда не заключается в одной драматичной ошибке. Это пять скучных, но систематических проблем.  

## 1. Размер ваших фрагментов неправильный  

Эта ошибка не приводит к сбою. Она просто немного ухудшает каждый ответ, пока вся функция не кажется ненадежной.  

Векторный поиск извлекает *фрагменты*, а не документы. Всё, что вы разбили из исходного материала, становится единицей истины для ретривера. Если фрагменты неправильные, модель отвечает с неправильными частями.  

**Слишком маленькие**: Фрагмент содержит частичный ответ. Вложение захватывает правильную тему, но извлеченный текст теряет контекст. Вы получаете «Максимальный таймаут — 30 секунд» без предыдущего предложения, в котором говорится «при использовании legacy API».  

**Слишком большие**: Вложение становится размытым средним множества идей. Семантический поиск запутывается, потому что фрагмент охватывает несколько тем, и результирующий вектор не представляет ни одну из них чётко.

Правильный размер фрагмента зависит исключительно от вашего контента. Технические документы, юридические контракты и транскрипты поддержки фрагментируются по-разному. Универсального ответа нет.

Что делать: измеряйте. Создайте набор оценки пар вопрос/ответ из вашего корпуса. Попробуйте фрагменты из 256, 512 и 1024 токенов. Измерьте точность извлечения: появляется ли правильный фрагмент в топ-5? Вы быстро поймёте, что размер фрагмента важнее, чем модель вложений, которую вы так тщательно выбирали.

Также используйте перекрытие. Фрагмент из 512 токенов с перекрытием в 64 токена с каждой стороны означает, что ответы, пересекающие границы, всё ещё извлекаются. Большинство векторных библиотек поддерживают это. Большинство пропускают этот шаг.

## 2. Ваши вложения устаревают (и вы этого не заметите)

Представьте, что ваша компания меняет бренд. Или переименовывает продукт. Или обновляет цены. Или устаревает API.

Вы обновляете документацию, но не пересоздаёте вложения фрагментов. Векторный индекс всё ещё представляет старый контент.

Пользователи спрашивают о новых ценах. Вложения направляют их к старому контенту. Модель читает старый контент и уверенно объясняет старые цены. Поддержка получает тикет.

Каждая серьёзная система RAG сталкивается с этим в конечном итоге. Исправление звучит очевидно - пересоздавайте вложения при изменении контента - но редко строят такую конвейерную линию до первого инцидента.

Вам нужна пошаговая переиндексация с отпечатками контента:

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Content unchanged, skip re-embedding
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

Переиндексируйте на этапе записи, вычисляйте отпечатки по содержимому, а не по временным меткам. Документы в вашей CMS обновляются постоянно, даже если содержимое не меняется.

## 3. Точность и полнота извлечения: вы оптимизируете неправильную

Большинство обучающих материалов по RAG показывают, как извлекать топ-K фрагментов. Они не объясняют, что две цели противоречат друг другу.

**Высокая полнота**: Возвращайте всё, что может быть связано. Пользователи всегда получают ответ. Но контекстовая окно модели переполнено косвенными шумами, и модель придумывает ответы, чтобы заполнить промежутки между фрагментами.

**Высокая точность**: Возвращайте только самые релевантные фрагменты. Модель работает с чистым, сфокусированным контекстом. Но если нужный фрагмент не входит в топ-3, модель не имеет информации и всё равно уверенно придумывает ответ.

Для пользователей режимы сбоя выглядят одинаково: неправильные ответы. Но причины и способы исправления противоположны.

Два техники, которые действительно помогают:

**Переупорядочивание**: Извлекайте больше кандидатов (топ-20), затем используйте модель cross-encoder для переупорядочивания по релевантности перед передачей в LLM. Cross-encoderы медленнее, чем векторное сходство, но значительно точнее на финальном этапе ранжирования.

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // Теперь топ-5 действительно значит что-то
```

**Гибридный поиск**: Комбинируйте векторный поиск (семантическое сходство) с ключевым поиском (BM25). Они ошибаются по-разному. Векторный поиск бороться с конкретными терминами, названиями моделей и идентификаторами. Ключевой поиск бороться с перефразированием и синонимами. Вместе они покрывают слепые пятна друг друга.

## 4. Ваш контекстное окно имеет неправильную форму

Вы извлекли правильные фрагменты. Поздравляем. Модель всё равно ошибётся.

Проблема не только в том, что вы извлекаете. Это и в том, где вы его размещаете.

LLM-модели могут страдать от проблемы "потерянного в середине". Лиу и др. измерили, что долгоконтекстные модели используют релевантную информацию менее надёжно, когда она появляется в середине запроса, а не в начале или конце.

Если вы втискиваете 20 фрагментов в плоский список и надеетесь, что модель правильно их синтезирует, вы оставляете производительность на столе.

Что действительно помогает:

**Оцените размещение начала/конца для ваших самых релевантных фрагментов.** Общая эвристика — первым идёт топ-рангированный, последним — второй, а остальные в середине. Контр-интуитивно, но стоит протестировать на вашей модели и форме запроса.

**Явно нумеруйте и маркируйте разделы контекста.** `[Источник 1]` ... `[Источник 2]` дают модели якоря для рассуждений.

**Добавьте сигнал уверенности в извлечение.** Если ваш показатель сходства 0.65 на шкале от 0 до 1, сообщите модели: «Следующий контекст был извлечён с умеренной уверенностью. Признайте неопределённость, если ответ неясен».

**Установите бюджет контекста.** Не передавайте всё, что вы извлекли. Считайте токены, приоритет по баллу релевантности и жестко ограничьте на 60-70% окна контекста модели. Оставьте место для рассуждений модели без перегрузки.

Ссылка: [Потеряно посередине: Как языковые модели используют длинные контексты](https://arxiv.org/abs/2307.03172).

## 5. Вы не знаете, когда это неправильно

Это тихая ошибка: ответ возвращается, интерфейс выглядит здоровым, а содержание — неправильным.

С традиционным API ошибки видны: HTTP 500, таймаут, ошибка проверки схемы. Вы сразу понимаете. Ошибки RAG тише: система возвращает ответ, он кажется правдоподобным, и он неправильный.

Вы можете не знать, что ваша RAG-конвейер не работает, пока пользователи не сообщат вам. Часто они не сообщают. Просто перестают доверять и возвращаются к Ctrl+F.

Минимальная жизнеспособная система наблюдения для RAG в продакшене:

**Ведите журнал извлечения.** Каждый запрос, что было извлечено (ID фрагментов + баллы) и что произвела модель. Вам это нужно для отладки чего-либо.

**Отслеживайте метрики извлечения.** Средний обратный ранг (MRR) и NDCG, если у вас есть метки истинного положения. В крайнем случае, отслеживайте распределение баллов сходства — если ваша медиана балла извлечения падает, качество индекса ухудшается.

**Создайте обратную связь.** Даже простой лайк/дизлайк к ответам, связанный с исходным запросом и извлеченными фрагментами, дает сигнал для обучения. Без этого вы летите вслепую.

**Запускайте периодические оценки.** Тестовый набор из 50-100 вопросов с известными правильными ответами, запускаемый еженедельно, выявит регрессию до того, как пользователи это заметят. Таблица Excel и скрипт — достаточно для старта.

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // Оповестить, если балл упадет ниже порога
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## Реальная проблема

Эти сбои в основном не связаны с моделью встраивания или векторной базой данных. Они связаны с системой вокруг них.

Демо RAG работает, потому что условия демонстрации контролируются: чистые документы, хорошо сформулированные вопросы, терпимые оценщики. Производственные системы не работают, потому что ни одно из этих условий не выполняется.

Каждый из вышеперечисленных сбоев можно диагностировать, но только если вы измеряете. Надежные команды RAG ничего экзотического не делают. Они рассматривают качество извлечения как реальную подсистему, а не артефакт демонстрации.

Сначала настройте оценочный цикл. Всё остальное станет проще, как только вы сможете его измерить.
