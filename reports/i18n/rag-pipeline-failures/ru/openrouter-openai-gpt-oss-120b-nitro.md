Вы встраиваете документы, поднимаете векторное хранилище, подключаете его к чату и наблюдаете, как модель ссылается на вашу внутреннюю базу знаний, будто она работает с ней годами. Красиво. Заинтересованные стороны впечатлены. Кто‑то говорит: «давайте выпустим это».

## 1. Ваши фрагменты неправильного размера

Это сбой, который не приводит к падению системы. Он просто делает каждый ответ чуть хуже, пока вся функция не перестаёт вызывать доверие.

Векторный поиск извлекает *фрагменты*, а не документы. Что бы вы ни делили свой исходный материал, это становится единицей истины для поисковика. Если фрагменты неправильные, модель отвечает неверными кусками.

**Слишком маленькие**: Фрагмент содержит лишь часть ответа. Эмбеддинг захватывает правильную тему, но извлечённый текст лишён контекста. Вы получаете «Максимальный тайм‑аут — 30 секунд» без предшествующего предложения «при использовании устаревшего API».

**Слишком большие**: Эмбеддинг превращается в размытое среднее множество идей. Семантический поиск путается, потому что фрагмент охватывает несколько тем, и полученный вектор не представляет их чисто.

Оптимальный размер фрагмента полностью зависит от вашего контента. Техническая документация, юридические контракты и транскрипты поддержки требуют разных подходов к фрагментации. Универсального ответа нет.

Что делать: измерять. Соберите набор оценочных вопросов/ответов из вашего корпуса. Попробуйте фрагменты в 256, 512 и 1024 токена. Оцените точность поиска: появляется ли нужный фрагмент в топ‑5? Вы быстро поймёте, что размер фрагмента важнее, чем выбранная модель эмбеддингов.

Также используйте перекрытие. Фрагмент в 512 токенов с 64‑токенным перекрытием с каждой стороны гарантирует, что ответы, охватывающие границу, всё равно будут извлечены. Большинство векторных библиотек поддерживают это. Большинство людей просто пропускают.

## 2. Ваши эмбеддинги устаревают (и вы этого не заметите)

Представьте, что ваша компания меняет бренд. Или переименовывает продукт. Или обновляет цены. Или выводит из эксплуатации API.

Вы обновляете документацию, но не переэмбеддите фрагменты. Векторный индекс всё ещё отражает старое содержание.

Пользователи спрашивают о новых ценах. Эмбеддинги направляют их к старому контенту. Модель читает старый материал и уверенно объясняет старые цены. Служба поддержки получает тикет.

Каждая серьёзная RAG‑система в конце концов сталкивается с этим. Решение кажется очевидным — перепроецировать эмбеддинги при изменении контента, — но команды редко строят такой конвейер до первого инцидента.

Нужна инкрементальная переиндексация с отпечатками содержимого:

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Content unchanged, skip re-embedding
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

Переиндексировать при записи, использовать отпечаток для самого содержимого, а не для меток времени. Документы в вашей CMS обновляются постоянно, хотя фактическое содержание может оставаться тем же.

## 3. Точность vs. полнота извлечения: вы оптимизируете не то

Большинство руководств по RAG показывают, как получить топ‑K фрагментов. Они не объясняют компромисс между двумя целями, движущимися в противоположные стороны.

**Высокий охват**: Возвращать всё, что может быть релевантным. Пользователь всегда получает ответ. Но окно контекста модели заполняется косвенно связанным шумом, и модель начинает «галлюцинировать», заполняя пробелы между фрагментами.

**Высокая точность**: Возвращать только самые релевантные фрагменты. Модель работает с чистым, сфокусированным контекстом. Но если нужный фрагмент не попал в топ‑3, информации нет, и модель всё равно уверенно придумывает ответ.

С точки зрения пользователя оба режима выглядят одинаково: неправильные ответы. Причины и решения, однако, противоположны.

Две техники, действительно помогающие:

**Перерейтигинг**: Получить больше кандидатов (top‑20), а затем с помощью кросс‑энкодера переупорядочить их по релевантности перед передачей в LLM. Кросс‑энкодеры медленнее векторного сходства, но существенно точнее на этапе финального ранжирования.

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // Теперь top‑5 действительно что‑то значит
```

**Гибридный поиск**: Сочетать векторный поиск (семантическое сходство) с поиском по ключевым словам (BM25). Они дают сбои по‑разному. Векторный поиск плохо работает с конкретными терминами, названиями моделей и идентификаторами. Поиск по ключевым словам страдает при перефразировании и синонимах. Вместе они покрывают слепые зоны друг друга.

## 4. Окно контекста имеет неправильную форму

Вы получили нужные фрагменты. Поздравляем. Модель всё равно выдаст ошибку.

Проблема заключается не только в том, что вы извлекаете, а в том, куда это помещаете.

LLM могут сталкиваться с проблемой «потерялись в середине». Liu et al. обнаружили, что модели с длинным контекстом используют релевантную информацию менее надёжно, когда она находится в середине подсказки, а не ближе к её началу или концу.

Если вы просто складываете 20 фрагментов в один плоский список и надеетесь, что модель правильно их синтезирует, вы теряете производительность.

То, что действительно помогает:

**Оцените размещение начала/конца для самых релевантных фрагментов.** Распространённый эвристический приём — ставить фрагмент с наивысшим ранжированием первым, второй по рангу — последним, а остальные — посередине. Противоинтуитивно, но стоит проверить на вашей модели и в контексте подсказки.

**Явно нумеруйте и помечайте секции контекста.** `[Source 1]` … `[Source 2]` даёт модели якоря для рассуждений.

**Добавьте сигнал уверенности извлечения.** Если ваш коэффициент сходства = 0.65 по шкале 0‑1, сообщите модели: «Следующий контекст был получен с умеренной уверенностью. Признайте неопределённость, если ответ не ясен.»

**Установите бюджет контекста.** Не передавайте модели всё, что вы извлекли. Считайте токены, приоритизируйте по оценке релевантности и жёстко обрезайте до 60‑70 % окна контекста модели. Оставьте место для рассуждений без перегрузки.

Ссылка: [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172).

## 5. Вы Не Понимаете, Когда Всё Не Так

Это тихий сбой: ответ возвращается, UI выглядит нормально, а содержимое ошибочно.

При традиционном API сбои видны: HTTP 500, тайм‑аут, ошибка валидации схемы. Вы узнаёте об этом сразу. Сбои RAG тише: система отдаёт ответ, он кажется правдоподобным, но неверен.

Вы можете не заметить, что ваш RAG‑конвейер падает, пока пользователи не укажут на проблему. Чаще всего они просто перестают ему доверять и возвращаются к Ctrl+F.

Минимальный набор наблюдаемости для production‑RAG‑системы:

**Логируйте цепочку поиска.** Записывайте каждый запрос, что было найдено (идентификаторы чанков + оценки) и что сгенерировала модель. Это необходимо для отладки любой проблемы.

**Отслеживайте метрики поиска.** Средний обратный ранг (MRR) и NDCG, если у вас есть метки‑истина. Как минимум, следите за распределением оценок схожести — если ваш P50‑score падает, качество индекса ухудшилось.

**Создайте обратный цикл.** Даже простое «палец вверх/вниз» к ответам, привязанное к запросу и найденным чанкам, даёт сигнал для обучения. Без этого вы летите вслепую.

**Проводите периодические оценки.** Тестовый набор из 50‑100 вопросов с известными правильными ответами, запускаемый еженедельно, выявит регрессии раньше, чем пользователи. Достаточно таблицы и небольшого скрипта, чтобы начать.

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // Alert if score drops below threshold
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## Реальная проблема

Эти сбои в основном не связаны с моделью эмбеддингов или векторной базой данных. Они касаются окружающей их системы.

Демонстрации RAG работают, потому что условия демонстрации контролируемы: чистые документы, корректно сформулированные вопросы, снисходительные оценщики. В продакшене всё падает, потому что ни одно из этих условий не соблюдается.

Каждый из перечисленных сбойных сценариев можно диагностировать, но только при наличии измерений. Надёжные команды RAG не используют никаких экзотических приёмов. Они рассматривают качество поиска как реальный подсистемный компонент, а не как артефакт демонстрации.

Сначала настройте цикл оценки. Всё остальное станет проще, как только появятся измеримые данные.
