````mdx
## 2. आपके एम्बेडिंग्स पुराने हो जाते हैं (और आपको पता भी नहीं चलता)

कल्पना कीजिए आपकी कंपनी का रीब्रांडिंग हो गया। या किसी प्रोडक्ट का नाम बदल दिया गया। या प्राइसिंग अपडेट हुई। या कोई API डिप्रिकेट हो गया।

उपयोगकर्ता नई प्राइसिंग के बारे में पूछते हैं। एम्बेडिंग्स उन्हें पुराने कंटेंट की ओर रूट कर देती हैं। मॉडल पुराने कंटेंट को पढ़ता है और आत्मविश्वास से पुरानी प्राइसिंग समझा देता है। सपोर्ट को एक टिकट मिलता है।

हर गंभीर RAG सिस्टम अंततः इस समस्या से टकराता है। समाधान तो स्पष्ट लगता है – जब कंटेंट बदलता है तो फिर से एम्बेड करें – लेकिन टीमें अक्सर पहली घटना से पहले ही यह पाइपलाइन नहीं बनातीं।

आपको कंटेंट फ़िंगरप्रिंटिंग के साथ इन्क्रिमेंटल री‑इंडेक्सिंग चाहिए:

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // कंटेंट नहीं बदला, फिर से एम्बेडिंग स्किप करें
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

राइट पर री‑इंडेक्स करें, फ़िंगरप्रिंट कंटेंट पर रखें, टाइमस्टैम्प पर नहीं। आपके CMS में दस्तावेज़ लगातार अपडेट होते रहते हैं, लेकिन वास्तविक कंटेंट बदलता नहीं है।

## 5. आपको नहीं पता कब यह गलत है

यह वह चुप्पी वाली विफलता है: उत्तर लौटता है, UI ठीक दिखता है, लेकिन सामग्री गलत होती है।

परम्परागत API में विफलताएँ स्पष्ट होती हैं: HTTP 500, टाइम‑आउट, स्कीमा वैधता त्रुटि। आपको तुरंत पता चल जाता है। RAG विफलताएँ चुपचाप आती हैं: सिस्टम एक प्रतिक्रिया देता है, वह यथार्थ लगती है, फिर भी वह गलत होती है।

आपको यह नहीं पता चल सकता कि आपका RAG पाइपलाइन फेल हो रहा है, जब तक उपयोगकर्ता आपको नहीं बताते। अक्सर वे नहीं बताते। वे बस उस पर भरोसा करना बंद कर देते हैं और Ctrl+F से वापस खोजते हैं।

एक प्रोडक्शन RAG सिस्टम के लिए न्यूनतम कार्यात्मक ऑब्ज़र्वेबिलिटी सेटअप:

**अपनी रिट्रीवल चेन को लॉग करें।** हर क्वेरी, क्या रिट्रीव हुआ (चंक आईडी + स्कोर), और मॉडल ने क्या आउटपुट दिया। डिबग करने के लिए यह आवश्यक है।

**रिट्रीवल मेट्रिक्स को ट्रैक करें।** यदि आपके पास ग्राउंड‑ट्रुथ लेबल हैं तो मीन रेसिप्रोकल रैंक (MRR) और NDCG उपयोग करें। न्यूनतम रूप में, सिमिलैरिटी स्कोर वितरण को ट्रैक करें — अगर आपका P50 रिट्रीवल स्कोर गिरता है, तो आपका इंडेक्स गुणवत्ता घट गई है।

**एक फीडबैक लूप बनाएं।** प्रतिक्रिया पर थंब्स अप/डाउन, क्वेरी और रिट्रीव्ड चंक्स से जुड़ा हुआ, आपको प्रशिक्षण संकेत देता है। बिना इस के आप अंधेरे में उड़ रहे हैं।

**नियमित रूप से इवैल्युएशन चलाएँ।** ज्ञात सही उत्तरों वाले 50‑100 प्रश्नों का टेस्ट सेट, साप्ताहिक चलाएँ, ताकि उपयोगकर्ता नोटिस करने से पहले रिग्रेशन पकड़ा जा सके। एक स्प्रेडशीट और एक स्क्रिप्ट शुरू करने के लिए पर्याप्त है।

```typescript

  testCases: { query: string; expectedAnswer: string }[],

  // Alert if score drops below threshold
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
````
