RAGのデモはいつだって完璧に見える。

ドキュメントを埋め込み、ベクトルストアを立ち上げ、チャットに接続すれば、モデルが内部のナレッジベースをあたかも長年そこで働いてきたかのように引用してくれる。美しい。ステークホルダーも感心する。「これを出荷しよう」と誰かが言う。

6週間後、ユーザーは自信満々の誤った回答を受け取っている。サポートチケットが山積みになる。システムは*動作している*。ただ、人々が必要とする方法では動いていない。

失敗は、たいてい一つの劇的なミスではない。退屈なミスが五つ積み重なった結果だ。

---

## 1. チャンクのサイズが間違っている

この失敗はクラッシュを引き起こさない。ただ、すべての回答を少しずつ悪化させ、やがて機能全体が信頼できなくなる。

ベクトル検索が取得するのは*チャンク*であって、ドキュメントではない。ソース素材をどのように分割するかが、検索器の真実の単位となる。チャンクが間違っていれば、モデルは誤った断片で回答する。

**小さすぎる場合**: チャンクに部分的な回答しか含まれていない。埋め込みは正しいトピックを捉えているが、取得されたテキストには文脈が欠けている。「最大タイムアウトは30秒です」という文を取得しても、その前に「レガシーAPIを使用する場合」という文がない。

**大きすぎる場合**: 埋め込みが複数のアイデアのぼやけた平均値になる。チャンクが複数のトピックにわたっているため、セマンティック検索が混乱し、結果として得られるベクトルはどのトピックもきれいに表現できなくなる。

適切なチャンクサイズはコンテンツに完全に依存する。技術ドキュメント、法的契約書、サポートトランスクリプトはすべて異なるチャンク分割が必要だ。万能な答えは存在しない。

対策：測定する。コーパスから質問/回答ペアの評価セットを構築する。256、512、1024トークンのチャンクを試す。検索精度を測定する：正しいチャンクがトップ5に入っているか？ チャンクサイズが、悩み抜いた埋め込みモデルよりも重要だとすぐにわかる。

さらにオーバーラップを使う。両側に64トークンのオーバーラップを持つ512トークンチャンクは、境界をまたぐ回答でも取得される。ほとんどのベクトルライブラリはこれをサポートしているが、ほとんどの人はスキップする。

---

## 2. 埋め込みが陳腐化する（そして気づかない）

会社がリブランドする、製品名を変更する、価格を更新する、APIを非推奨にする——そういった状況を想像してほしい。

ドキュメントは更新するが、チャンクを再埋め込みしない。ベクトルインデックスは古いコンテンツを表したままになる。

ユーザーが新しい価格について質問する。埋め込みは古いコンテンツにルーティングする。モデルは古いコンテンツを読み、古い価格を自信満々に説明する。サポートにチケットが届く。

真面目なRAGシステムはすべて、いつかこれに直面する。修正方法は明白に聞こえる——コンテンツが変更されたら再埋め込みする——だが、チームは最初のインシデントが発生する前にそのパイプラインを構築することはほとんどない。

コンテンツフィンガープリンティングを用いたインクリメンタルな再インデックス化が必要だ：

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // コンテンツ未変更、再埋め込みをスキップ
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

書き込み時に再インデックスし、タイムスタンプではなくコンテンツのフィンガープリントを使う。CMS上でドキュメントが更新されても、実際のコンテンツが変わっていないことはよくある。

## 3. 検索の精度 vs. 再現率：間違った方を最適化している

ほとんどのRAGチュートリアルは、上位K個のチャンクを取得する方法を示すだけだ。相反する方向を向いた二つの目標のトレードオフについては説明しない。

**高再現率**：関連しそうなものをすべて返す。ユーザーは常に回答を得られる。しかしモデルのコンテキストウィンドウは関連性の薄いノイズで埋め尽くされ、断片間の隙間を埋めるために幻覚を起こす。

**高精度**：最も関連性の高いチャンクだけを返す。モデルはクリーンで焦点の絞られたコンテキストで動作する。しかし適切なチャンクが上位3件に入っていなければ、モデルは情報を持たず、自信満々にでっち上げる。

どちらの障害モードもユーザーからは同じに見える：間違った回答。しかし原因と修正方法は正反対だ。

実際に役立つ二つのテクニック：

**リランキング**：より多くの候補（上位20件）を取得し、クロスエンコーダモデルで関連性に基づいて再ランク付けしてからLLMに渡す。クロスエンコーダはベクトル類似度より遅いが、最終的なランキング精度は格段に高い。

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // これで上位5件が実際に意味を持つ
```

**ハイブリッド検索**：ベクトル検索（意味的類似度）とキーワード検索（BM25）を組み合わせる。それぞれ異なる形で失敗する。ベクトル検索は特定の用語、モデル名、IDに弱い。キーワード検索は言い換えや類義語に弱い。組み合わせることで、互いの盲点を補完する。

## 4. コンテキストウィンドウの形が間違っている

正しいチャンクを取得できた。おめでとう。しかし、モデルは依然として間違えるだろう。

問題は取得した内容だけではない。どこに配置するかだ。

LLMは「lost in the middle」問題に悩まされることがある。Liuらは、長文コンテキストモデルにおいて、関連情報がプロンプトの先頭や末尾ではなく中央に現れると、その情報を信頼性高く利用できないことを測定した。

20個のチャンクをフラットなリストに詰め込んで、モデルが正しく合成してくれると期待しているなら、パフォーマンスを無駄にしている。

実際に役立つこと：

**最も関連性の高いチャンクの先頭/末尾配置を評価する。** 一般的なヒューリスティックは、最もスコアの高いものを先頭に、2番目を末尾に、残りを中央に配置する。直感に反するが、自分のモデルとプロンプトの形状でテストする価値はある。

**コンテキストセクションに明示的に番号とラベルを付ける。** `[Source 1]` ... `[Source 2]` のようにすると、モデルが推論するためのアンカーとなる。

**検索の信頼度シグナルを追加する。** 類似度スコアが0-1スケールで0.65の場合、モデルに次のように伝える：「以下のコンテキストは中程度の信頼度で取得されました。回答が不明確な場合は不確実性を認識してください。」

**コンテキスト予算を設定する。** 取得したものをすべてそのまま渡してはいけない。トークンをカウントし、関連性スコアで優先順位をつけ、モデルのコンテキストウィンドウの60～70%でハードカットする。モデルが推論する余地を残し、無駄な処理をさせない。

参考: [中間に埋もれる: 言語モデルが長いコンテキストをどのように利用するか](https://arxiv.org/abs/2307.03172)

## 5. いつ間違っているかわからない

これが静かな障害だ。回答は返ってきて、UIは正常に見え、内容が間違っている。

従来のAPIでは、障害は目に見える。HTTP 500、タイムアウト、スキーマ検証エラー。すぐにわかる。RAGの障害はもっと静かだ。システムは応答を返し、もっともらしく見え、そして間違っている。

RAGパイプラインが失敗していることに、ユーザーから知らされるまで気づかないかもしれない。多くの場合、ユーザーは知らせない。ただ信頼を失い、Ctrl+Fに戻るだけだ。

プロダクションRAGシステムのための最小限の可観測性セットアップ:

**検索チェーンをログに記録する。** すべてのクエリ、何が取得されたか（チャンクID＋スコア）、そしてモデルが生成したもの。これがなければ何もデバッグできない。

**検索メトリクスを追跡する。** 正解ラベルがある場合は平均相互ランク（MRR）とNDCG。最低限、類似度スコアの分布を追跡する。P50の検索スコアが低下したら、インデックスの品質が落ちている。

**フィードバックループを構築する。** 応答に対するサムズアップ/ダウンさえ、クエリと取得されたチャンクに紐付ければ、トレーニングシグナルになる。それがなければ、手探りで進むしかない。

**定期的な評価を実行する。** 既知の正解を持つ50～100問のテストセットを毎週実行すれば、ユーザーが気づく前に回帰を検出できる。スプレッドシートとスクリプトがあれば十分だ。

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // Alert if score drops below threshold
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## 実際の問題

これらの障害は、主に埋め込みモデルやベクトルデータベースに起因するものではない。その周辺のシステムに起因する。

RAGのデモが機能するのは、デモ環境が制御されているからだ。クリーンなドキュメント、適切に形成された質問、寛容な評価者。本番環境で失敗するのは、それらの条件がどれも成立しないからだ。

上記のすべての障害は診断可能だが、それは測定している場合に限る。信頼性の高いRAGチームは特別なことをしているわけではない。彼らは検索品質をデモの成果物ではなく、実際のサブシステムとして扱っている。

まず評価ループを構築せよ。測定できれば、他のすべては容易になる。
