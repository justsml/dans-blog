RAG のデモは常に見栄えが良い。

いくつかのドキュメントを埋め込み、ベクトルストアを立ち上げ、チャットに接続すれば、モデルが内部ナレッジベースを何年も使ってきたかのように引用してくれる。美しい。ステークホルダーは感心し、誰かが「これを出荷しよう」と言う。

しかし 6 週間後、ユーザーは自信満々で間違った回答を受け取るようになる。サポートチケットが山積みになる。システムは *動作* しているが、求められている形で機能していない。

失敗の原因は通常、劇的な単一ミスではない。5 つの退屈なミスが積み重なっているだけだ。

---

## 1. チャンクサイズが間違っている

この失敗はクラッシュさせるわけではない。ただ、すべての回答が少しずつ劣化し、機能全体が信頼できないと感じられるようになる。

ベクトル検索は *ドキュメント* ではなく *チャンク* を取得する。ソース素材をどのように分割するかが、検索エンジンの真実単位になる。チャンクが適切でなければ、モデルは誤った断片で回答してしまう。

**小さすぎる**：チャンクに部分的な回答しか含まれない。埋め込みは正しいトピックを捉えるが、取得されたテキストは文脈が欠落している。たとえば「最大タイムアウトは 30 秒です」という文だけが取得され、前の文「レガシー API を使用する場合に限る」が抜け落ちている。

**大きすぎる**：埋め込みが多数のアイデアのぼやけた平均になる。意味検索はチャンクが複数のテーマを含むため混乱し、結果として得られるベクトルはどれもきれいに表現できない。

適切なチャンクサイズはコンテンツ次第です。技術文書、法的契約書、サポートの文字起こしはそれぞれ異なる方法でチャンク化されます。普遍的な答えはありません。

**やるべきこと**: 計測する。コーパスから質問‑回答ペアの評価セットを作成し、256、512、1024 トークンのチャンクで試します。取得精度を測定します――正しいチャンクが上位 5 件に入っているか？　ここですぐに、チャンクサイズが埋め込みモデルよりもはるかに重要であることが分かります。

また、オーバーラップを利用します。512 トークンのチャンクに各側 64 トークンのオーバーラップを持たせれば、境界を跨ぐ回答も取得対象になります。ほとんどのベクトルライブラリがこの機能をサポートしていますが、実装を省く人が多いです。

## 2. 埋め込みが古くなる（そして気付かれない）

会社がリブランドしたり、製品名を変更したり、価格を改定したり、API を廃止したりしたと想像してください。

ドキュメントは更新しても、チャンクを再埋め込みしなければベクトルインデックスは旧コンテンツを指し続けます。

ユーザーが新しい価格について質問すると、埋め込みは古いコンテンツへと誘導します。モデルは古い情報を読み取り、自信満々に旧価格を説明します。サポートはチケットを受け取ります。

真剣に RAG を運用するシステムは必ずこの壁にぶつかります。解決策は明白です――コンテンツが変わったら再埋め込みする――しかし、最初のインシデントが起きる前にそのパイプラインを構築するチームはほとんどいません。

必要なのは、コンテンツ指紋を用いたインクリメンタルな再インデックスです。

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Content unchanged, skip re-embedding
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

書き込み時に再インデックスし、指紋はコンテンツそのものに基づくべきで、タイムスタンプではありません。CMS 上でドキュメントは頻繁に更新されますが、実際の内容が変わっていないことも多いです。

## 3. 検索の精度 vs. 再現率: 間違った方を最適化している

多くの RAG チュートリアルは top‑K チャンクの取得方法だけを示しますが、互いに逆方向に働く二つの目標間のトレードオフは説明しません。

**高再現率**: 関連し得るすべてを返す。ユーザーは必ず回答を得られますが、モデルのコンテキストウィンドウは周辺ノイズで埋め尽くされ、断片間のギャップを埋めようと幻覚を起こします。

**高精度**: 最も関連性の高いチャンクだけを返す。モデルはクリーンで焦点が絞られたコンテキストで動作しますが、正しいチャンクが top‑3 に入っていなければ、情報が欠如したまま自信満々に妄想を生成します。

失敗モードはユーザーにとっては同じです：誤った回答。しかし原因と解決策は正反対です。

実際に効果がある二つの手法：

**再ランク**: 候補を多めに取得（top‑20）し、クロスエンコーダモデルで関連度順に再ランクしてから LLM に渡す。クロスエンコーダはベクトル類似度より遅いですが、最終ランク付けでは格段に正確です。

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // Now top-5 actually means something
```

**ハイブリッド検索**: ベクトル検索（意味的類似度）とキーワード検索（BM25）を組み合わせる。両者は異なる失敗パターンを持ちます。ベクトル検索は特定の用語、モデル名、ID に弱く、キーワード検索はパラフレーズや同義語に弱い。組み合わせることで互いの盲点を補完します。

## 4. コンテキストウィンドウの形が間違っている

正しいチャンクを取得できました。おめでとうございます。モデルは依然として誤答を出す可能性があります。

問題は「何を取得するか」だけではありません。**どこに配置するか** が鍵です。

LLM は「途中で失われる」問題に陥りやすいことが報告されています。Liu らは、長文コンテキストモデルが、情報がプロンプトの途中に現れたときに、冒頭や末尾に比べて信頼性が低下することを測定しました。

20 個のチャンクを平坦なリストに詰め込み、モデルが正しく統合できると期待しても、性能は犠牲になります。

実際に効果がある手法:

- **最も関連性の高いチャンクの開始/終了位置を評価する**。一般的なヒューリスティックは、上位ランクのチャンクを最初に、次位を最後に、残りを中間に配置することです。直感に反しますが、モデルとプロンプトの形状に対してテストする価値があります。

- **コンテキストセクションに明示的に番号とラベルを付ける**。`[Source 1]` … `[Source 2]` のように記すことで、モデルは参照ポイントを持って推論できます。

- **取得信頼度のシグナルを付加する**。類似度スコアが 0〜1 のスケールで 0.65 の場合、モデルに対して「以下のコンテキストは中程度の信頼度で取得されました。回答が不明瞭な場合は不確実性を認めてください」と伝えます。

**コンテキスト予算を設定する。** 取得したすべてをそのまま渡すのはやめましょう。トークン数を数え、関連スコアで優先順位を付け、モデルのコンテキストウィンドウの 60〜70% でハードカットします。モデルが余裕を持って推論できるように、余白を残しておくことが重要です。

Reference: [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172).

## 5. 何が間違っているか分からない

これが静かな失敗です。回答が返ってきて UI は正常に見えるが、内容は誤っている。

従来の API では失敗が可視化されます。HTTP 500、タイムアウト、スキーマ検証エラーなど、すぐに分かります。RAG の失敗は静かです。システムはレスポンスを返し、見た目は妥当に思えるが、実際は間違っています。

ユーザーが指摘するまで、RAG パイプラインが失敗していることに気付かないことが多いです。多くの場合、ユーザーは何も言わずに信頼を失い、Ctrl+F に戻ります。

本番環境の RAG システムに最低限必要な可観測性セットアップ:

**取得チェーンをログに残す。** すべてのクエリ、取得されたもの（チャンク ID とスコア）、そしてモデルが生成した出力を記録します。これがなければデバッグは不可能です。

**取得メトリクスを追跡する。** 真のラベルがある場合は平均逆順位 (MRR) と NDCG を測ります。最低限、類似度スコアの分布を追跡しましょう。P50 の取得スコアが低下したら、インデックスの品質が低下したことを意味します。

**フィードバックループを構築する。** レスポンスに対する「いいね」/「よくない」だけでも、クエリと取得されたチャンクに紐付ければ学習シグナルになります。これがなければ盲目的に飛び回っている状態です。

**定期的に評価を実行する。** 正解が分かっている 50〜100 件の質問をテストセットとして用意し、週次で走らせればユーザーが気付く前にリグレッションを検出できます。スプレッドシートとスクリプトがあれば十分です。

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // Alert if score drops below threshold
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## 実際の問題

これらの失敗は主に埋め込みモデルやベクトルデータベースそのものに起因するわけではありません。問題はそれらを取り巻くシステムにあります。

RAG デモがうまくいくのは、デモ環境が管理された条件下にあるからです。クリーンな文書、整形式の質問、寛容な評価者… これらが揃っているからこそ成功します。実運用ではそのような条件はほとんど存在しません。

上記の各失敗は測定すれば診断可能です。測定しない限り、原因は見えません。信頼できる RAG チームは特別なことをしているわけではなく、取得品質をデモの付属品ではなく、実際のサブシステムとして扱っています。

まずは評価ループを構築しましょう。測定できれば、他のすべてが格段に楽になります。
