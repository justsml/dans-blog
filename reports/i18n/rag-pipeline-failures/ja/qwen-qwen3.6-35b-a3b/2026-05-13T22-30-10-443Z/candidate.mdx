RAGのデモは往々にして完璧に見える。

ドキュメントを埋め込み、ベクトルストアを起動し、チャットに接続して、モデルが社内ナレッジベースを数年勤めたかのように引用するのを見る。素晴らしい。ステークホルダーは納得し、「これをリリースしよう」という声が上がる。

6週間後、ユーザーは自信満々に間違った回答を受け取るようになる。サポートチケットが溜まり始める。システムは確かに「動作している」が、人々が求めている用途では動いていない。

失敗は通常、劇的な一つのミスから来るものではない。5つの地味なミスが積み重なった結果だ。

---

## 1. チャンクサイズが間違っている

この失敗はシステムをダウンさせない。ただ、すべての回答をわずかに劣化させ、最終的に機能全体が信頼できなくなるまでにする。

ベクトル検索はドキュメントではなく*チャンク*を検索する。ソースデータをどのように分割するかによって、それが検索システムの「真実の単位」になる。チャンクが間違っていれば、モデルは間違った断片を基に回答する。

**小さすぎる**: チャンクに回答の一部しか含まれていない。埋め込みは正しいトピックを捉えているが、取得されたテキストには文脈が欠けている。「最大タイムアウトは30秒です」という記述だけを取得し、「レガシーAPIを使用する場合」という直前の文が欠落している状態だ。

**大きすぎる**: 埋め込みが多数のアイデアのぼやけた平均値になってしまう。セマンティック検索が混乱し、チャンクが複数のトピックを混在させているため、生成されるベクトルがどれ一つとして明確に表現できない。

適切なチャンクサイズは、コンテンツの種類に完全に依存する。技術ドキュメント、法的契約書、サポート通話録音は、すべて異なる方法でチャンク分割される必要がある。万能な正解は存在しない。

対応策は測定だ。コーパスから質問と回答のペアの評価セット（eval）を構築し、256、512、1024トークンのチャンクで試行せよ。検索精度を測定する：正しいチャンクが上位5件に含まれるか？ チャンクサイズが、頭を悩ませた埋め込みモデルよりも重要であることを、すぐに理解するだろう。

重複（オーバーラップ）も活用するべきだ。両側に64トークンの重複を持たせた512トークンのチャンクなら、境界をまたぐ回答も確実に取得できる。多くのベクトルライブラリがこの機能に対応している。だが、多くの開発者がこれをスキップしている。

## 2. 埋め込みが古くなる（そしてそれに気づかない）

会社がリブランディングする。あるいは製品名を変更する。価格を更新する。APIを非推奨にする。

ドキュメントは更新したが、チャンクの再埋め込みは行わなかった。ベクトルインデックスは依然として古いコンテンツを反映している。

ユーザーが新価格について質問する。埋め込みベクトルは古いコンテンツへルーティングする。モデルは古いコンテンツを読み取り、自信を持って旧価格を説明する。サポートチームにはチケットが飛ぶ。

本格的なRAGシステムは、いずれ必ずこの問題に直面する。解決策は明白だ——コンテンツが変更されたら再埋め込みすればいい。だが、最初のインシデントが発生する前に、そのパイプラインを構築するチームは稀だ。

必要なのは、コンテンツフィンガープリントを用いた増分再インデックスだ：

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Content unchanged, skip re-embedding
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

書き込み時に再インデックスし、タイムスタンプではなくコンテンツそのものでフィンガープリントを生成する。CMSでは実際のコンテンツが変更されていないのに、文書の更新が頻繁に走ることはよくある。

## 3. 検索精度と再現率：あなたは間違った指標を最適化している

大半のRAGチュートリアルは上位K件のチャンク取得手法を示すだけで、正反対を向く2つの目標間のトレードオフには触れない。

**再現率（High recall）**: 関連する可能性のあるものはすべて返す。ユーザーは常に回答を得られる。だがモデルのコンテキストウィンドウは関連性の薄いノイズで埋め尽くされ、欠落部分を埋めるためにモデルは幻覚（ハルシネーション）を起こす。

**精度（High precision）**: 最も関連性の高いチャンクのみを返す。モデルはクリーンで焦点を絞ったコンテキストで動作する。ただし正解のチャンクが上位3件に入っていない場合、モデルは情報がないにもかかわらず、それでも自信を持ってでたらめな回答を生成してしまう。

ユーザーから見れば失敗モードは同じだ。間違った回答が返ってくる。しかし原因と解決策は正反対である。

実際に効果を発揮する2つの技術がある。

**リランキング（Reranking）**: より多くの候補（上位20件）を取得し、LLMに渡す前にクロスエンコーダーで関連性に基づいて再ランク付けする。クロスエンコーダーはベクトル類似度よりも遅いが、最終ランク付けステップでは圧倒的に精度が高い。

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // Now top-5 actually means something
```

**ハイブリッド検索（Hybrid search）**: ベクトル検索（意味的類似度）とキーワード検索（BM25）を組み合わせる。これらは異なる方法で失敗する。ベクトル検索は固有名詞、モデル名、IDの処理が苦手だ。キーワード検索は言い換えや同義語の処理が苦手だ。組み合わせることで互いの盲点を補完し合う。

## 4. コンテキストウィンドウの形状が間違っている

適切なチャンクを取得できた。おめでとう。だが、モデルはそれでもこれを誤る。

問題は取得する内容だけではない。どこに配置するかだ。

LLMは「真ん中で見失う（lost in the middle）」現象に陥りやすい。Liuらの研究では、プロンプトの冒頭や末尾ではなく中央部に配置されると、長文脈モデルが関連情報をより不安定に処理することが測定されている。

20個のチャンクをフラットなリストに放り込んで、モデルが正しく統合してくれることを期待しているなら、パフォーマンスを大幅に損なっていることになる。

実際に効果的な対策は以下の通りだ：

**最も関連性の高いチャンクの配置（冒頭/末尾）を評価せよ。** 一般的なヒューリスティックでは、最も関連度の高いチャンクを冒頭に、次点のチャンクを末尾に、残りを中央に配置する。直感に反するが、モデルとプロンプトの形状に合わせてテストする価値は十分にある。

**コンテキストセクションに明示的な番号とラベルを付与せよ。** `[Source 1]` ... `[Source 2]` のようにすると、モデルが推論するためのアンカーが得られる。

**検索の信頼度シグナルを追加せよ。** 類似度スコアが0〜1の尺度で0.65の場合、モデルに次のように指示する：「以下のコンテキストは中程度の信頼度で取得されました。回答が不明確な場合は不確実性を明記してください。」

**コンテキスト予算を設定せよ。** 取得したものをすべて渡すな。トークンをカウントし、関連度スコアで優先順位をつけて、モデルのコンテキストウィンドウの60〜70%で硬く切り捨てろ。スラッシング（パフォーマンス劣化）を起こさずに推論できるよう、余白を残せ。

参考文献: [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172).

## 5. 誤りを検知する仕組みがない

これは静かな失敗だ。回答は返ってくるし、UIは正常に見える。だが中身は間違っている。

従来のAPIなら失敗は目に見える。HTTP 500、タイムアウト、スキーマ検証エラー。即座にわかる。RAGの失敗はもっと静かだ。システムは応答を返し、それらしく見え、そして間違っている。

ユーザーが指摘するまで、RAGパイプラインが壊れていることに気づかないかもしれない。多くの場合、指摘すらされない。単に信頼を失い、Ctrl+Fに戻って手動検索し始めるだけだ。

本番環境のRAGシステムに必要な最低限の観測性（observability）セットアップは以下の通りだ：

**検索チェーンをログに記録せよ。** 各クエリ、何を取得したか（チャンクID＋スコア）、モデルが何を生成したか。これを残さなければ、デバッグのしようがない。

**検索メトリクスを追跡せよ。** 正解ラベルがあるなら、平均逆位取り（MRR）やNDCGを計測せよ。少なくとも、類似度スコアの分布を追跡しろ。P50の検索スコアが低下すれば、それはインデックス品質の低下を意味する。

**フィードバックループを構築せよ。** 回答に対する高評価・低評価を、クエリと取得チャンクに紐付けて記録するだけでも、学習信号が得られる。これがないと、盲目で操縦しているようなものだ。

**定期的な評価を実行せよ。** 正解が分かっている50〜100問のテストセットを週次で回せば、ユーザーが気づく前に回帰を検知できる。始めるのに、スプレッドシートとスクリプトがあれば十分だ。

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // Alert if score drops below threshold
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## 本質的な問題

これらの失敗は、主に埋め込みモデルやベクトルデータベースのせいではない。それらを囲むシステム全体のせいだ。

RAGのデモが機能するのは、デモ条件が管理されているからだ：クリーンなドキュメント、構造化された質問、寛容な評価者。本番環境で失敗するのは、それらの条件が一つも満たされないからだ。

上記の失敗はすべて診断可能だが、測定している場合に限りだ。信頼性の高いRAGチームがやっているのは、特段変わったことではない。検索品質をデモの産物ではなく、本格的なサブシステムとして扱っている。

まず評価ループを構築せよ。測定可能になれば、他はすべて簡単になる。
