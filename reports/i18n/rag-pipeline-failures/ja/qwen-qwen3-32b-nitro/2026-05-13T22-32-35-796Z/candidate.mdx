RAGのデモはいつも素晴らしいように見えます。  

ドキュメントを埋め込み、ベクターDBを立ち上げ、チャットに接続して、モデルが内部の知識ベースを何年も使い慣れたかのように引用するのを観察します。美しい。ステークホルダーたちは感心します。誰かが「これをリリースしましょう」と言います。  

6週間後、ユーザーは自信満々に間違った回答を得ています。サポートチケットが積み上がります。システムは「動いています」が、人々が本当に必要としている方法では動いていません。  

失敗は通常、1つの劇的なミスではなく、5つの地味なミスが重なったものです。  

---  

## 1. チャンクのサイズが適切でない  

この失敗はクラッシュしません。ただ、すべての回答がわずかずつ悪化してしまい、最終的に機能全体が信頼できなくなるのです。  

ベクターサーチが取得するのは*チャンク*であり、ドキュメントではありません。ソースデータを分割した単位が、リトリーバーの真実の単位になります。チャンクが間違っていると、モデルは間違ったフラグメントで回答します。  

**小さすぎる**: チャンクに部分的な回答しか含まれていません。埋め込みは正しいトピックを捉えますが、取得されたテキストに文脈が欠けています。前の文「レガシーアプリケーションを使用する場合に」を抜かして「最大タイムアウトは30秒です」と取得します。  

**大きすぎる**: 埋め込みは多くのアイデアのぼんやりとした平均になります。セマンティックサーチが混乱する原因となり、チャンクが複数の話題を扱っているため、結果として得られるベクターはどれも正確に表していません。

チャンクサイズの適切な選択はコンテンツに完全に依存します。技術文書、法的契約書、サポートの転記はすべて異なるチャンク分けが必要です。普遍的な答えはありません。

対処法：測定しましょう。自社のコーパスから質問/回答ペアの評価セットを構築します。256トークン、512トークン、1024トークンのチャンクを試してください。リトリーバル精度を測定します：正しいチャンクが上位5位に現れますか？すぐに学ぶことになるでしょうが、チャンクサイズの選択は悩んだ末に選んだ埋め込みモデルよりも重要です。

またオーバーラップを活用しましょう。512トークンのチャンクで各境界に64トークンのオーバーラップを設定すると、境界をまたぐ回答も取得できます。ほとんどのベクターライブラリがこの機能をサポートしていますが、多くの人はこれを飛ばしてしまいます。

## 2. 埋め込みが陳腐化する（そして気づかない）

会社がブランド戦略を刷新したと想像してください。または製品名を変更した。価格を更新した。APIを非推奨にした。

ドキュメントは更新したものの、チャンクの再埋め込みは行っていません。ベクターインデックスは依然として古いコンテンツを表しています。

ユーザーが新しい価格について尋ねると、埋め込みは古いコンテンツに誘導します。モデルは古いコンテンツを読み、自信たっぷりに古い価格を説明します。サポートチームにはチケットが届きます。

すべての本格的なRAGシステムは最終的にこの問題に直面します。解決策は明らかに聞こえます—コンテンツ変更時に再埋め込みする—が、多くのチームは最初のインシデントが発生するまでそのパイプラインを構築しません。

インクリメンタルな再インデックスとコンテンツフィンガープリントが必要です：

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Content unchanged, skip re-embedding
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

書き込み時に再インデックスを実行し、コンテンツのフィンガープリントを取得してタイムスタンプではなくコンテンツに基づいて判定します。CMSではコンテンツが変更されていないにもかかわらずドキュメントが頻繁に更新されます。

## 3. Retrievalの適合率と再現率: 間違った指標を最適化している

ほとんどのRAGチュートリアルは、トップ-Kチャンクの取得方法を示します。しかし、反対方向に働く2つの目的のトレードオフを説明しません。

**高再現率**: 関連性がありそうなものをすべて返却する。ユーザーは常に回答を得る。しかしコンテキストウィンドウが関連性の低いノイズで満たされ、モデルは断片間のギャップを埋めるために幻覚を起こす。

**高適合率**: 最も関連性の高いチャンクのみを返却する。モデルはクリーンで焦点を合わせたコンテキストで動作する。しかし正しいチャンクがトップ3に入っていなければ、モデルは情報がないにもかかわらず自信満々に何かを生成する。

ユーザーにとっての失敗モードは同じように見えます: 間違った回答。しかし原因と修正策は正反対です。

実際に役立つ2つの技法:

**再ランク付け**: より多くの候補を取得（top-20）し、クロスエンコーダモデルを使って関連性で再ランク付けしてからLLMに渡す。クロスエンコーダはベクターサイミリティより遅いが、最終ランク付けでははるかに正確。

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // 今やtop-5に意味がある
```

**ハイブリッド検索**: ベクターセルチ（意味的類似性）とキーワード検索（BM25）を組み合わせる。それぞれが異なる方法で失敗する。ベクターセルチは特定の語句、モデル名、IDに弱く、キーワード検索は言い換えや同義語に弱い。合わせることでそれぞれの盲点を補い合う。

## 4. コンテキストウィンドウの形状が適切でない

適切なチャンクを取得しました。おめでとうございます。それでもモデルはこれを誤る可能性があります。

取得する内容だけでなく、それをどこに配置するかも問題です。

LLM（大規模言語モデル）は「中央で迷う（lost in the middle）」という問題に悩まされることがあります。Liuらは、長文コンテキストにおいて、関連情報がプロンプトの中央に配置された場合、冒頭や末尾に配置された場合よりモデルが信頼性を失うことを測定しました。

20のチャンクをフラットリストに詰め込み、モデルが正しく統合してくれるのを期待している場合、パフォーマンスを放棄していることになります。

実際に役立つ方法：

**最も関連性の高いチャンクの配置位置を評価する。** 一般的なヒューリスティクスは「1位を先頭に、2位を末尾に、残りを中央に配置する」です。直感に反するかもしれませんが、モデルとプロンプトの形状にテストしてみることをおすすめします。

**コンテキストセクションを明示的に番号とラベル付する。** `[Source 1]` ... `[Source 2]` とモデルにアンカーを提供します。

**取得信頼度のシグナルを追加する。** 類似度スコアが0-1のスケールで0.65の場合、モデルに「以下のコンテキストは中程度の信頼度で取得されました。答えが不明確な場合は不確実性を認識してください」と伝えるとよいでしょう。

**コンテキスト予算を設定する。** 検索したすべての情報をそのまま渡さない。トークン数をカウントし、関連性スコアで優先順位をつけて、モデルのコンテキストウィンドウの60〜70%で厳密にカットする。モデルが計算に時間を浪費せずに推論できる余地を残す。

参考: [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172)

## 5. 正確でないときの判断ができない

これは静かな失敗です: 答えが返ってきますが、UIは正常に見え、内容は誤っています。

従来のAPIでは失敗が明確です: HTTP 500、タイムアウト、スキーマ検証エラー。すぐにわかります。RAGの失敗は静かです: システムが応答を返し、それが妥当に見え、そして誤っています。

ユーザーがRAGパイプラインが失敗していることを伝えるまで、自分が失敗していることに気づかないかもしれません。多くの場合、ユーザーは伝えません。ただ信頼を失い、Ctrl+Fに戻るだけです。

運用中のRAGシステムで最低限必要な観測設定:

**取得チェーンをログする。** 各クエリで何が取得されたか（チャンクID + スコア）と、モデルが生成した内容を記録する。これらがないと何のデバッグもできない。

**取得メトリクスを追跡する。** グラウンドトゥルースラベルがあれば平均逆位数（MRR）やNDCGを追跡。最低限、類似度スコアの分布を追跡する——P50取得スコアが下がれば、インデックスの品質が下がっている。

**フィードバックループを構築する。** 応答に対する単純な「いいね/よくない」のフィードバックでも、クエリと取得されたチャンクに紐付ければトレーニングシグナルになります。これを欠くと、盲目で運用することになります。

**定期的な評価を実行する。** 正解がわかっている50〜100問のテストセットを週単位で実行すれば、ユーザーが気づく前に回帰を検出できます。スプレッドシートとスクリプトがあれば十分です。

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // Alert if score drops below threshold
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## 実際の問題

これらの失敗は主に埋め込みモデルやベクトルデータベースの問題ではありません。これらを取り囲むシステムの問題です。

RAGのデモが機能するのは、デモの条件が制御されているからです：綺麗なドキュメント、整った質問、寛容な評価者。運用が失敗するのは、これらの条件が一切成立しないからです。

上記のすべての失敗は診断可能です。ただし、測定していないと意味がありません。信頼できるRAGチームは特別なことをやっていません。取得品質をデモの副産物ではなく、本格的なサブシステムとして扱っているだけです。

まず評価ループを構築してください。測定できるようになれば、他のすべては簡単に解決できます。
