RAGのデモは常に素晴らしいように見えます。  

ドキュメントを埋め込み、ベクター格納庫を立ち上げ、チャットに接続して、モデルがまるで何年も自社の知識ベースを扱ってきたかのように参照するのを観ます。美しい。関係者は感動します。誰かが「これをリリースしましょう」と言います。  

6週間後、ユーザーは自信満々で誤った回答を受け取ります。サポートチケットが増殖します。システムは「動作しています」が、人々が本当に必要としている形ではありません。  

失敗は通常、劇的な1つのミスではなく、5つの地味な失敗が積み重なったものです。  

---

## 1. チャンクのサイズが不適切な場合  
---

この失敗はクラッシュしません。ただ、すべての回答がわずかずつ悪化し、最終的に全体の機能が信頼できなくなります。

ベクター検索が取得するのは*チャンク*であり、ドキュメントではありません。ソースデータを分割した単位がリトリーバーの真実単位になります。チャンクが誤っていると、モデルは間違った断片で回答します。

**小さすぎる場合**：チャンクに部分的な回答しか含まれていません。埋め込みは正しいトピックを捉えていますが、取得されたテキストに文脈が欠けています。たとえば「最大タイムアウトは30秒です」という文だけが取得され、「レガシーアピを使用する場合に限る」という前の文が含まれていない場合があります。

**大きすぎる場合**：埋め込みは多くのアイデアのぼんやりとした平均になります。意味検索が混乱する原因になります。チャンクが複数の話題を含んでいるため、結果として得られるベクトルはどれも明確に表していません。

正しいチャンクサイズはコンテンツに完全に依存します。技術文書、法的契約、サポートのトランスクリプトはそれぞれ異なるチャンク分けが必要です。普遍的な答えはありません。

対処法：測定しましょう。コーパスから質問/回答ペアの評価セットを構築しましょう。256、512、1024トークンのチャンクを試してください。リトリーバル精度を測定しましょう：正しいチャンクが上位5位に表示されますか？すぐに学ぶことになるでしょうが、チャンクサイズは悩んで選んだ埋め込みモデルよりも重要です。

オーバーラップも活用しましょう。512トークンのチャンクで、各側に64トークンのオーバーラップがあると、境界にまたがる回答も取得されます。ほとんどのベクトルライブラリがこの機能をサポートしていますが、多くの人はこれをスキップしてしまいます。

## 2. 埋め込みが古くなる（気づかないまま）

企業がリブランドしたとしましょう。または製品名を変更した。価格を更新した。APIを廃止した。  
ドキュメントは更新したものの、チャンクの再埋め込みは行いません。ベクトルインデックスは古いコンテンツを依然として反映しています。  

ユーザーが新価格について尋ねると、埋め込みは古いコンテンツに誘導します。モデルは古いコンテンツを読み、自信たっぷりに古い価格を説明します。サポートにチケットが届きます。

## 2. 埋め込みが古くなる（気づかないまま）  

企業がリブランドしたとしましょう。または製品名を変更した。価格を更新した。APIを廃止した。  
ドキュメントは更新したものの、チャンクの再埋め込みは行いません。ベクトルインデックスは古いコンテンツを依然として反映しています。  

ユーザーが新価格について尋ねると、埋め込みは古いコンテンツに誘導します。モデルは古いコンテンツを読み、自信たっぷりに古い価格を説明します。サポートにチケットが届きます。  

すべての真剣なRAGシステムは最終的にこの問題に直面します。対処法は明らかに思えます——コンテンツ変更時に再埋め込みする——が、チームは最初のインシデントが発生するまで、このパイプラインを構築しません。  

インクリメンタルな再インデックスとコンテンツフィンガープリントが必要です：  

```typescript
import { createHash } from 'crypto';

async function upsertDocument(doc: Document, vectorStore: VectorStore) {
  const fingerprint = createHash('sha256')
    .update(doc.content)
    .digest('hex');

  const existing = await vectorStore.getBySourceId(doc.id);

  if (existing?.fingerprint === fingerprint) {
    return; // Content unchanged, skip re-embedding
  }

  const chunks = chunkDocument(doc);
  const embeddings = await embedBatch(chunks);

  await vectorStore.upsert(
    chunks.map((chunk, i) => ({
      id: `${doc.id}:${i}`,
      sourceId: doc.id,
      fingerprint,
      vector: embeddings[i],
      text: chunk.text,
      metadata: { ...doc.metadata, updatedAt: new Date() },
    }))
  );
}
```

書き込み時に再インデックスし、フィンガープリントはコンテンツに基づいて行う。タイムスタンプではなく。CMSでは、実際のコンテンツが変更されていないにもかかわらず、ドキュメントが頻繁に更新されます。

## 3. Retrieval Precision vs. Recall: 最適化しているのが間違っている  

ほとんどのRAGチュートリアルは、上位Kチャンクを取得する方法を示します。しかし、2つの目標が相反する方向を指しているというトレードオフについては説明しません。  

精度（Precision）と再現率（Recall）の間でトレードオフが発生します。精度は「関連性のある結果を正しく返す率」、再現率は「すべての関連結果を返す率」です。RAGシステムでは、この2つの指標のバランスが重要です。過度に精度を追求すると、必要な情報が漏れる可能性が高まります。逆に再現率を優先すると、文脈に無関係なノイズが混入し、モデルが誤った推論を生じるリスクが増加します。  

このトレードオフを意識せずに設計されたシステムは、実運用で予期せぬエラーを引き起こします。例えば、再現率を過剰に高めた場合、文脈に無関係なチャンクが混入し、モデルが「隙間埋め」のために虚偽情報を生成する可能性があります。一方、精度を過剰に重視すると、重要な情報が除外され、ユーザーの質問に応えられません。  

この問題を解決するには、**再現率を確保しつつ精度を最適化する**アプローチが必要です。具体的には、ベクター検索とキーワード検索を組み合わせるハイブリッド検索、またはクロスエンコーダーによるリランク（Reranking）を導入することで、精度と再現率のバランスを調整できます。

**高い再現率（High recall）**: 関連しそうなすべてのチャンクを返す。ユーザーは常に回答を得るが、モデルの文脈ウィンドウが関連性の薄いノイズで満たされ、断片間の隙間を埋めるために虚偽情報を生成する可能性がある。

**高い精度（High precision）**: 最も関連性の高いチャンクのみを返す。モデルはクリーンで焦点を絞った文脈で動作する。しかし、正しいチャンクが上位3位に含まれていなければ、モデルは情報が不足しているにもかかわらず自信を持って誤った回答を生成する。

これらの失敗モードはユーザーにとって見た目が同じ（誤った回答）だが、原因と修正方法は正反対である。

実際的に役立つ2つの手法:

**リランク（Reranking）**: より多くの候補（上位20件）を取得し、クロスエンコーダーモデルで関連性に基づいて再順位付けし、LLMに渡す前に最終的な順位を調整する。クロスエンコーダーはベクターランクより遅いが、最終的な順位付けの精度が大幅に向上する。

```typescript
import { Reranker } from '@mastra/rag';

const results = await vectorStore.search(queryEmbedding, { topK: 20 });
const reranked = await reranker.rank(query, results);
const context = reranked.slice(0, 5); // 今や上位5件が実質的な意味を持つ
```

**ハイブリッド検索（Hybrid search）**: ベクターセルチ（意味的類似性）とキーワード検索（BM25）を組み合わせる。それぞれが異なる弱点を持つ。ベクターセルチは特定の語句やモデル名、IDの検索に弱く、キーワード検索は言い換えや同義語に弱い。これらを組み合わせることで、お互いの盲点を補完する。

## 4. コンテキストウィンドウの形状が間違っている

適切なチャンクを取得しました。おめでとうございます。モデルがこれを誤る原因はここにあります。

取得する内容だけでなく、その配置場所が問題です。

LLMは「真ん中に埋もれる問題」に悩まされることがあります。Liuらは、長文脈モデルにおいて、関連情報がプロンプトの真ん中に位置する場合、冒頭や末尾に配置される場合に比べて信頼性が低下することを測定しました。

20のチャンクをフラットリストに詰め込み、モデルがそれらを正しく合成してくれると期待しているなら、パフォーマンスを意図外に損なっています。

---

実際に役立つもの：

**最も関連性の高いチャンクの配置場所を評価してください。** 一般的なヒューリスティックは、上位のランクを最初に配置し、2番目のランクを最後に配置し、残りを真ん中に配置することです。直感に反するかもしれませんが、自分のモデルとプロンプトの形状に対してテストすることをおすすめします。

**コンテキストセクションを明示的に番号付きラベルで区切ってください。** `[ソース 1]` ... `[ソース 2]` とすることで、モデルに論理的基準を提供できます。

**取得信頼度のシグナルを追加してください。** 類似度スコアが0〜1のスケールで0.65の場合、モデルに対して次のように伝えます：「以下に示すコンテキストは中程度の信頼度で取得されました。答えが曖昧な場合は不確実性を認識してください」。

**コンテキスト予算を設定してください。** 取得したすべてのデータを単に渡すだけにしないでください。トークン数をカウントし、関連性スコアで優先順位を付け、モデルのコンテキストウィンドウの60〜70％でハードカットしてください。モデルが混乱することなく推論できるスペースを確保してください。

参考：[Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172)

## 本質的な問題

**検索連鎖のログを記録する。** すべてのクエリ、取得された内容（チャンクID + スコア）、およびモデルが生成した内容をログに記録する必要があります。これがないと何をデバッグするにもできません。

**検索メトリクスを追跡する。** 正解ラベルがあれば、平均逆位数（MRR）や正規化済み割引累積適合度（NDCG）を追跡しましょう。最低限、類似度スコアの分布を追跡する必要があります — P50検索スコアが下落すれば、インデックスの品質が低下しています。

**フィードバックループを構築する。** 回答に対する単純な「いいね/悪い」のフィードバックでも、クエリと取得チャンクに結びつけることでトレーニングシグナルになります。これを欠くと、盲目で運用することになります。

**定期的な評価を実行する。** 50〜100問の正解が分かっているテストケースを週単位で実行し、ユーザーが気づく前に回帰を検出しましょう。スプレッドシートとスクリプトがあれば十分です。

```typescript
async function runEval(
  testCases: { query: string; expectedAnswer: string }[],
  pipeline: RAGPipeline
) {
  const results = await Promise.all(
    testCases.map(async ({ query, expectedAnswer }) => {
      const response = await pipeline.query(query);
      const score = await scoreResponse(response, expectedAnswer);
      return { query, score, response };
    })
  );

  const avgScore = results.reduce((s, r) => s + r.score, 0) / results.length;
  console.log(`Eval score: ${(avgScore * 100).toFixed(1)}%`);
  
  // スコアがしきい値を下回ったら通知
  if (avgScore < 0.75) {
    await notifyTeam(`RAG eval score dropped to ${(avgScore * 100).toFixed(1)}%`);
  }

  return results;
}
```

## 本質的な問題

これらの失敗は、主に埋め込みモデルやベクターDBの問題ではありません。これらは、それらを取り囲むシステムの問題なのです。

RAGのデモが機能するのは、デモの条件が制御されているからです：クリーンなドキュメント、整った質問、寛容な評価者。本番環境での失敗は、これらの条件がすべて成立しないからです。

上記のすべての失敗は診断可能ですが、測定を行わなければなりません。信頼性のあるRAGチームは特異な手法を行っていません。彼らはリトリーブの品質をデモの副産物ではなく、本物のサブシステムとして取り扱っているのです。

まず評価ループを構築してください。測定できるようになれば、それ以降のすべてが容易になります。
