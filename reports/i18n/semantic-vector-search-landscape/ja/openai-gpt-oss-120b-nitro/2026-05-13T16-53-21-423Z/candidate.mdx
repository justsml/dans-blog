検索は単一の機能ではなく、セマンティック検索は残りの検索手法の置き換えでもありません。

「メールアドレス `dan@example.com` のユーザーを探す」と「新入エンジニア向けにデバッグに関する記事を探す」はどちらも検索と呼ばれますが、エンジニアリング上の問題としてはほとんど共通点がありません。前者は正解が存在し、`O(log n)` のインデックス検索で答えが得られます。後者は正解が存在せず、関連性のみが求められ、言語・意図・意味の理解が必要です。

検索に関する意思決定で最も説得力を持つエンジニア――議論に勝ち、適切なシステムを出荷できる人――は全体像を把握しています。どのツールをいつ使うべきかを知り、その理由を明確に説明できるのです。

本稿ではセマンティック層、すなわちベクトル検索が実際に何を行い、どのようなケースで有効で、どこで手を引くべきかを解説します。実用的なアプローチは「すべてを埋め込む」ことではなく、ベクトルがレキシカル検索、ファジー検索、完全一致検索とハイブリッドアーキテクチャでどの位置にくるべきかを見極めることです。

レキシカルおよびファジー検索（`tsvector`、`pg_trgm`、`pg_search`）については、[Postgres Text Searching Guide 2026](../postgres-text-search-guide) を参照してください。

## 用語一覧

**Embedding** — モデルが生成する浮動小数点数の密なリストで、テキスト（画像、音声など）を高次元空間の一点として表現します。意味的に関連するコンテンツは近くに配置され、無関係なコンテンツは遠くに配置されます。

**Lexical search** — 正確な単語やトークンの一致に基づく検索。高速で決定論的、既知の語句に対しては正確です。シノニム、パラフレーズ、あるいは言語間の同等語は理解できません。

**Semantic search** — トークンではなく意味に基づく検索。たとえば「タイムアウトの対処方法」というクエリが、共通語が全くない「リトライポリシーの設定」という文書タイトルと一致することがあります。これは埋め込みベクトルが幾何学的に近いからです。

**Vector** — 数値のリスト。検索文脈では埋め込みモデルの出力を指します。「ベクトル検索」はクエリベクトルに最も近いベクトルを幾何学的距離で見つけます。

**FTS (Full-Text Search)** — Postgres が提供する組み込みのレキシカル検索機能で、`tsvector` / `tsquery` によって駆動されます。テキストをトークン化、ステミング、インデックス化し、キーワードクエリに応答します。散文や正確な語句検索には強力ですが、意味は考慮しません。

**BM25** — レキシカル検索向けのランキングアルゴリズム（Elasticsearch、Qdrant などで使用）。語彙頻度をコーパス全体での希少性で重み付けしてスコアを算出する。単純なキーワードマッチより優れるが、依然としてレキシカル。

**HNSW (Hierarchical Navigable Small World)** — ベクトル検索で標準的に用いられる近似最近傍インデックス。高速で高リコールな類似度クエリを実現するために、層状の近接グラフを構築する。pgvector、Qdrant、Weaviate など多くの実装が採用している。

**RRF (Reciprocal Rank Fusion)** — 複数の検索システムから得られた順位付き結果リストを統合するアルゴリズム。スコアの正規化は不要で、順位位置だけを利用する。FTS とベクトルの両方で上位に位置する結果は、どちらか一方だけで支配的な結果よりも強い結合スコアを得る。

## セマンティック検索が実際に行うこと

ベクトル埋め込みはテキスト（画像、音声なども含む）を数値のリストに変換し、高次元空間上の点として表現する。埋め込みモデルは、意味的に関連するテキストがその空間で近接するように学習される。「Dog」と「canine」は近くに配置されるが、「Running a marathon」と「running a Python script」は同じ単語を共有していても遠くに配置される。

その空間での類似検索は、クエリの意味に最も近い文書を、正確な単語の重複に関係なく見つけ出します。

具体例:
- 「リクエストのタイムアウトを設定する方法？」という質問は、タイトルが「接続上限とリトライポリシーの設定」の記事とマッチする可能性があります――キーワードは重複しませんが、概念的な関連性は高いです
- 「夏の夜に飲みたい軽めのもの」という要望は、商品説明にキーワードがなくてもワインの推薦とマッチします
- 英語のクエリが、埋め込みモデルが多言語で学習されていれば、フランス語・スペイン語・日本語の関連文書とマッチします

**Lexical search**（`tsvector`、`pg_trgm`）ではこれらは実現できません。文字や単語単位で動作し、意味を考慮しません。ツールは置き換え可能ではなく、解決すべき問題が異なります。

## pgvector が有利になるケース

**RAG の構築**。Retrieval‑Augmented Generation は、ユーザーの質問に最も近い意味を持つ文書チャンクを取得し、コンテキストとして言語モデルに渡します。この取得ステップ自体がベクトル演算です。全文検索（FTS）だけでは、パラフレーズや同義語、概念的なマッチングを見逃すことがあります。スタンドアロンのベクトルストアに対する pgvector の利点は、既存の Postgres インスタンス内で動作する点です――別途サービスをデプロイしたり、運用したり、データを同期したりする必要がありません。

**ユーザーは何を検索したいかではなく、何をしたいかを記述する。** 「新しいマネージャとして自信をつける方法に関する記事」には、関連する投稿に確実に現れるキーワードが存在しません。「副作用を扱う軽量フレームワーク」も、ドキュメント内で正確にその語句が使われているとは限りません。ベクトル検索は綴りではなく意図にマッチします。

**類似アイテムの検索。** 関連商品、類似サポートチケット、重複バグレポート、あなたが好きかもしれない記事。 「このチケットに似た問題を探す」は最近傍検索です — アイテムを埋め込み、幾何学的な近傍を探します。重要な注意点として、ベクトル検索は常に結果を返します。たとえ実質的に類似していない場合でもです。重複除去やレコメンデーションのユースケースでは、最低類似度閾値（例：コサイン類似度 ≥ 0.80）でフィルタリングし、低信頼度のマッチが意味のあるものとして表面化するのを防ぎます。

**セマンティック重複除去。** RAG や検索用にコンテンツをインデックス化する前に、コーパス内のほぼ重複した文書を特定する必要があります — 複数回改訂された記事、二重で登録されたサポートチケット、かなり重複するナレッジベースエントリなどです。文書を埋め込み、コサイン類似度で閾値フィルタをかけて、インデックスを汚染する前に近似重複をフラグ付けまたはマージします。これにより、検索時にほぼ同一のチャンクが複数返され、コンテキストウィンドウが希釈されるのを防げます。

**多言語検索。** 多言語埋め込みモデルは、異なる言語間で意味的に等価なコンテンツを近いベクトルにマッピングします。スペイン語のクエリ「perder peso」は、英語の記事「sustainable weight loss habits」とマッチします — 共有トークンはなくても、根底にある意味は同じです。FTS は言語ごとの辞書設定が必要で、クロス言語クエリの取り扱いは苦手です。`pg_trgm` は言語に依存しませんが、文字列レベルの比較でありセマンティックではありません。

### Setting Up pgvector

拡張機能のインストールから類似度クエリまで、セットアップは数行の SQL 文で完了します。

```sqlCREATE EXTENSION IF NOT EXISTS vector;

ALTER TABLE documents ADD COLUMN embedding vector(1536);

-- HNSW は中規模データセットで最初に試すべきインデックスです
CREATE INDEX documents_embedding_idx
  ON documents USING hnsw (embedding vector_cosine_ops);

-- セマンティック検索クエリ
SELECT id, title, 1 - (embedding <=> $1::vector) AS similarity
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 10;
```

`<=>` はコサイン距離です。`1 - cosine_distance` がコサイン類似度を表し、1.0 が完全に同一、0.0 が直交を意味します。`ivfflat`（構築が速い従来型の代替手段）を使う場合は、`lists = sqrt(row_count)` を初期設定として試してください。

### pgvector が苦手とするケース

- 正確なトークン一致 ― 製品 SKU、エラーコード、関数名など。`ORD-12345` は意味的に類似するものがありません。埋め込みベースの検索は `ORD-12344` を返すか、関連性のない結果になる可能性があります。FTS または B ツリーインデックスを併用してください。
- 名前や固有名詞。埋め込み空間は意味で整理されるため、綴りが異なるだけの「Micheal Jordan」レコードが「Michael Jordan」の近くに来るとは限りません。
- 意味より文字レベルの類似が重要になる短い文字列。`pg_trgm` が適しています。
- クエリに正確な語句が必ず含まれる必要がある場合。既知語句のマッチングには BM25 や FTS の方が信頼性が高いです。

## ハイブリッド検索：両者が必要な理由

技術ドキュメントは、どちらか一方だけでは十分でない典型的な例です。

ユーザーが「how to configure timeouts」を検索するときは、概念的なマッチングが必要です。キーワードが重複しない「Setting retry policies and connection limits」という記事が、正確に求めている情報です。

同じユーザーは `withRetry()`、`ECONNRESET`、`ERR_SOCKET_TIMEOUT` も検索します。これらの文字列は必ず結果に含まれる必要があり、意味的マッチングだけでは確実に見つけられない可能性があり、概念的に似ているが正しい API でない結果は誤解を招きます。

ベクトル検索は概念的なクエリを処理し、FTS は正確な語句を処理します。どちらも単独で両方をうまく扱うことはできません。

解決策はハイブリッド検索です。両方を実行し、結果を統合します。

### Reciprocal Rank Fusion

**Reciprocal Rank Fusion (RRF)** は、異なる検索システムからのランク付けされたリストを結合する標準アルゴリズムです。システム間でスコアを正規化する必要はなく、ランク位置だけを使用します。*両方* のリストで上位に現れる結果は、どちらか一方で支配的な結果よりも強い結合スコアを得ます。

```sql
WITH fts_results AS (
  SELECT id,
    ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM documents, to_tsquery('english', $1) query
  WHERE search_vector @@ query
  LIMIT 50
),
vector_results AS (
  SELECT id,
    ROW_NUMBER() OVER (ORDER BY embedding <=> $2::vector) AS rank
  FROM documents
  ORDER BY embedding <=> $2::vector
  LIMIT 50
),
rrf AS (
  SELECT
    COALESCE(f.id, v.id) AS id,
    COALESCE(1.0 / (60 + f.rank), 0) +
    COALESCE(1.0 / (60 + v.rank), 0) AS rrf_score
  FROM fts_results f
  FULL OUTER JOIN vector_results v ON f.id = v.id
)
SELECT d.id, d.title, rrf.rrf_score
FROM rrf
JOIN documents d ON d.id = rrf.id
ORDER BY rrf_score DESC
LIMIT 10;
```

分母の `60` は RRF の定数です。値が大きいほどランク位置の差異が抑えられ、値が小さいほど差異が強調されます。デフォルトの 60 はほとんどのコンテンツタイプで十分に機能します。

RRF は `ts_rank`（対数頻度スコア）とコサイン距離（幾何学的測度）を正規化して比較するという、より難しい問題を回避します。両者は直接比較できません。RRF が問うのは「この結果が各リストでどれだけ上位に現れたか？」だけです。

### トリグラムも使ったハイブリッド検索

ユーザーが同一セッション内で人物名、概念、あるいは正確な語句を検索する可能性がある混在コンテンツに対して、3 方向の融合で全てに対応できます。

```sql
WITH trgm_results AS (
  SELECT id,
    ROW_NUMBER() OVER (ORDER BY similarity(title, $1) DESC) AS rank
  FROM documents
  WHERE title % $1
  LIMIT 50
),
fts_results AS (
  SELECT id,
    ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, to_tsquery('english', $1)) DESC) AS rank
  FROM documents
  WHERE search_vector @@ to_tsquery('english', $1)
  LIMIT 50
),
vector_results AS (
  SELECT id,
    ROW_NUMBER() OVER (ORDER BY embedding <=> $2::vector) AS rank
  FROM documents
  ORDER BY embedding <=> $2::vector
  LIMIT 50
),
rrf AS (
  SELECT
    COALESCE(t.id, f.id, v.id) AS id,
    COALESCE(1.0 / (60 + t.rank), 0) +
    COALESCE(1.0 / (60 + f.rank), 0) +
    COALESCE(1.0 / (60 + v.rank), 0) AS rrf_score
  FROM trgm_results t
  FULL OUTER JOIN fts_results f ON t.id = f.id
  FULL OUTER JOIN vector_results v ON COALESCE(t.id, f.id) = v.id
)
SELECT d.id, d.title, rrf.rrf_score
FROM rrf
JOIN documents d ON d.id = rrf.id
ORDER BY rrf_score DESC
LIMIT 10;
```

이 쿼리는 퍼지 이름 매치(트리그램), 정확한 키워드 매치(FTS), 그리고 개념 기반 쿼리(벡터)를 모두 처리합니다. 하나의 검색 입력창으로 세 가지 사용자 의도를 모두 만족시킬 수 있습니다。

## マルチレイヤー ハイブリッド アーキテクチャ

実際のアプリケーションはほとんどが単一の検索対象を持ちません。複数の検索対象があり、それぞれが異なる要件を持ちます。

| 検索対象 | ユーザーが検索するもの | 推奨レイヤー |
|---|---|---|
| ブログ / ドキュメント検索 | キーワード + コンセプト | FTS + pgvector (RRF) |
| ユーザー／顧客名検索 | 誤字を含む名前 | `pg_trgm` |
| 商品検索 | 名前、説明文、「似ている」 | `pg_trgm` + FTS + pgvector |
| サポートチケット重複検出 | 「このチケットに似た問題」 | pgvector のみ |
| 社内 SKU／注文検索 | 正確な識別子 | B-tree インデックス |
| 大規模ナレッジベース上の RAG | 自然言語の質問 | pgvector (チャンク化された文書) |
| E コマース「あなたにおすすめ」 | 行動履歴 + セマンティック類似度 | pgvector |
| オートコンプリート | プレフィックス、スペル許容 | `pg_trgm` |

これらは仮想的な例ではありません。コンテンツが多いアプリケーションの多くは、クエリ形状が異なる少なくとも二つの検索対象を必要とします。誘惑に駆られて「すべてに同じ手法」を適用しがちですが、現在は流行のベクトル検索を選びがちです。結果として、トリグラムインデックスで十分に高速・低コスト・正確に処理できる問題に対して、不要な埋め込み計算が走り、コストが嵩んでしまいます。

### 経験則

現在のレイヤーで対処できない失敗モードが現れたときに、別のレイヤーを追加します。

- ユーザーがタイプミスでヒットしない → `pg_trgm` を追加  
- ユーザーが概念で検索し、関連結果が漏れる → pgvector を追加  
- ユーザーが正確なシンボルやコードで検索したのに概念的な結果が返る → FTS を追加するか、ベクトル検索に依存しすぎていないか確認  
- レイテンシが問題になる → プリフィルタリング、近似インデックス、または専用ストアの導入を検討  

## 専用ベクトルストアが本当に必要な場合

pgvector は、別のデータベースが必要になる前に多くのアプリケーション検索を処理します。実際の境界はベクトル数、インデックス設定、書き込みレート、フィルタ、ハードウェア、同時実行数に依存するため、 「10 M 未満のベクトル」 というルールはベンチマークの出発点と捉え、製品上限と見なさないでください。真にスケールアウトが必要になるケース ―‑ 非常に高い同時実行数、p99 レイテンシが極めて低い要件、数十億のベクトル、または本格的なマルチテナント分離が求められる場合 ―‑ は、専用ベクトルデータベースの選択肢が多数あり、把握しておく価値があります。

### マトリックス列の実際の意味

**ハイブリッド検索** とは、BM25 キーワード検索とベクトル類似度検索を同一クエリで実行し、RRF によって結果を統合することを指します。これがない場合は、検索モードを一方に絞るか、二つのクエリを自前で融合する必要があります。

**Sparse vectors** は BM25 を超えた機能を提供します。SPLADE のスパースベクトルは約 30,000 次元（語彙項目ごとに 1 つ）で、約 98% がゼロです。非ゼロの位置は、どの語が重要か、どれだけ重要かを示します。「dogs」というクエリは「canine」や「pet」も重み付けします ―‑ BM25 レベルの精度に加えて、ベクトルインデックス内での語句展開が行われます。この列が false の場合、正確な語句検索のために別途 FTS 層が必要になります。

```python
# SPLADE: ~30,000 dims, ~60 non-zero — only relevant vocabulary positions fire
def encode_splade(text: str) -> dict:
    tokens = tokenizer(text, return_tensors="pt", truncation=True, max_length=512)
    with torch.no_grad():
        output = model(**tokens)
    vec = torch.log1p(torch.relu(output.logits)).max(dim=1).values.squeeze()
    return {"indices": vec.nonzero().squeeze().tolist(), "values": vec[vec != 0].tolist()}
```

**SQL / SQL-like** は実際にはフィルタリングに関するものです。フィルタなしのベクトル検索はデモに過ぎません。テナントスコープ、日付範囲、権限、カテゴリフィルタは依然として必要です。フル SQL（pgvector、LanceDB）では既存の JOIN の横にこれらを記述できます。目的特化型データベースは JSON フィルタオブジェクト（Qdrant、Pinecone）、クエリ DSL（Elasticsearch、Milvus）、あるいは GraphQL（Weaviate）を使用します。いずれも機能しますが、フィルタロジックが複雑になるほど SQL の方が魅力的になります。

```sql
-- pgvector: vector similarity is just another expression
SELECT id, title, 1 - (embedding <=> $1) AS score
FROM documents
WHERE tenant_id = $2
  AND category = ANY($3::text[])
  AND created_at > NOW() - INTERVAL '90 days'
ORDER BY embedding <=> $1
LIMIT 10;
```

```python
)
```

**Multimodal native** とは、データベースが非テキストコンテンツ用の埋め込みモデルを同梱していることを指します。生の画像 URL を渡すだけでベクトル化が行われます。多くのデータベースは埋め込みに依存しない設計で、埋め込みパイプラインは利用者が所有します。Marqo と Weaviate（CLIP/ImageBind モジュール経由）はこのループを閉じます。

```python
# Marqo: POST raw images, query with text — no external embedding step
mq.index("products").add_documents(
    [{"id": "shoe-001", "image": "https://cdn.example.com/shoes/001.jpg"}],
    tensor_fields=["image"]
)
results = mq.index("products").search(q="lightweight shoes for summer")
# Returns shoe-001 despite zero keyword overlap — CLIP handles the cross-modal match
```

**Disk-based index** はコスト調整のレバーです。RAM 常駐の HNSW インデックスは、元のベクトル、グラフオーバーヘッド、メタデータをすべて含めると、100 万件の 1536 次元ベクトルにつき数 GB の RAM が必要になることがあります。ディスクネイティブの代替手段（Milvus DiskANN、Elasticsearch DiskBBQ、LanceDB の Lance 形式、Turbopuffer のオブジェクトストレージ層）では、クエリ遅延を若干犠牲にしてインフラコストを削減できます。モデル遅延がすでに支配的な RAG ワークロードでは、そのトレードオフはベンチマークする価値があります。

**Max dimensions** はアーキテクチャに潜む移行リスクです。`text-embedding-3-large` は 3072 次元を使用し、Jina v3 はさらに大きな埋め込みを出力できます。研究モデルは次々に次元数を増やしています。いくつかのマネージドサービスはハードな次元上限を公開していますが、他は実務上の上限が高いか、典型的な埋め込みモデルに対して実質的な上限がありません。導入前に最新のドキュメントを確認してください。余裕を持った次元数を選択すべきです。次元上限に達したことでベクトルインデックスを移行するのは、痛みを伴うスプリント作業になります。

### 表に収まらないいくつかのポイント

**Turbopuffer のマルチテナンシー** は、非常に多くの名前空間を前提に設計されています。公開されているポジショニングや顧客事例は、Notion のように名前空間が大量にある大規模コーパスを強調しています。ユーザーや組織ごとに検索を分離する必要がある場合、アーキテクチャがコスト構造を変える可能性がありますが、必ず自分のテナント形状でベンチマークを取ってください。

**LanceDB の埋め込みモード** は「ベクトル検索用 SQLite」に最も近いものです。プロセス内で動作しサーバー不要で、Lambda、Cloudflare Workers、エッジ環境でも利用できます。Lance のカラム指向フォーマットにより、実規模でも埋め込み運用が実用的になります。

**Chroma は開発/テストや小規模アプリのデプロイに最適です。** もし非常に大規模なコーパス、HA、ディスク中心の運用、あるいは本格的なハイブリッド検索が必要なら、プロトタイプをインフラに昇格させる前に本番向けストアを評価すべきです。

**Vespa は、検索が製品の半分に過ぎない場合に選択すべきものです。** 語彙検索、最近傍検索、テンソル、ランク式、グルーピング、オンライン配信をすべて統合しています。その威力は本物ですが、運用とモデリングの複雑さも同様に大きいです。CRUD アプリに「セマンティック検索を追加したい」よりも、検索・レコメンデーションチーム向きです。

**ClickHouse は、検索が分析と結びつくときに議論に入ります。** 真実の情報源がイベント、ログ、トレース、メトリクスである場合、ClickHouse はベクトル距離、フィルタリング、集計、そして本格的な全文インデックスを 1 つの SQL エンジンで提供します。専用のベクトルデータベースではありませんが、分析系検索に対してはしばしば「退屈だけど正解」な選択肢です。

**スパースベクトルは、ベクトルインデックス内で BM25 レベルのキーワードマッチングを実現する手段です** — 別個の全文エンジンを走らせる必要がありません。Qdrant と Elasticsearch が特に成熟した実装を持っています。ハイブリッド検索が必須で、二系統構成が許容できない場合は、スパースベクトルのサポートが重要な指標になります。

## やってはいけないこと

ベクトル検索を、正解が決まっているものに対するファジーテキスト検索として使わないこと。

「メールアドレス `dan@example.com` のユーザーを探して」や「注文ID `ORD-12345` の注文を探して」はベクトル検索の問題ではありません。`ORD-12345` を埋め込み、コサイン類似度で検索すると何かは返ってきますが、間違っている可能性があります。識別子には正しい答えが一つだけあります。識別子に対する近似一致はバグです。

ベクトル検索はデータセット内で *最も類似した* 項目を返しますが、実際に関連するものが存在しない場合でも何かを返してしまいます。答えが存在しないことを判断できません。これは関連文書を探す場合には許容できても、正確なレコード検索では深刻な問題です。自信のある誤答は、空結果よりもはるかに悪影響を及ぼします。

逆方向も同様です。ユーザーが概念を説明しているクエリに対して全文検索（FTS）を使わないこと。「不確実性の下で難しい意思決定をする方法に関する記事」のようなクエリは信頼できるキーワードがありません。全文検索はノイズを返すか何も返さないかのどちらかです。クエリの形状に合った適切なツールを選択してください。

## 全体像

ほとんどの本番検索システムは複数のレイヤーを必要とします：

- **`pg_trgm`** – 名前、タイプミス、オートコンプリート用
- **FTS / `pg_search`** – キーワードベースの文章検索用
- **pgvector** – セマンティック・概念検索用
- **RRF fusion** – ユーザーがクエリタイプを混在させる場面での統合用
- **通常のインデックス** – 正確な識別子、フィルタ、ソート済みリスト用

これらは競合するツールではなく、相補的です。適切に構築された検索システムはクエリの形状に応じて最適なレイヤーを選択し、形状が重なる場合は複数レイヤーを同時に走らせて結果を融合します。

優れた検索機能を提供できるチームはスタック全体を把握しています。そうでないチームはベクトルデータベースに手を出し、すべてを埋め込み、正確な検索が時々誤ったレコードを返す理由が分からないままです。
