还记得过去连接数据库时要把一堆零散的环境变量拼在一起的日子吗？

那是一座脆弱的配置塔。`DB_HOST`、`DB_PORT`、`DB_USER`、`DB_PASSWORD`、`DB_NAME`……等等，甚至还有 `DB_USERNAME`、`DB_PASS` 或 `DB_PWD` 的争论。到底要不要加上 `PG_*` 前缀？超时设置该放在哪儿？

这座纸牌屋随时可能因大小写错误（比如忘记把 `HOST` 大写）而崩塌。

于是有人灵机一动，直接使用 URL¹：

```bash
postgres://user:pass@host:5432/dbname
```

一行字符串，所需信息全都有。通用可解析、可移植。敢说……优雅？

那我们为何还把 LLM 当成 1999 年的技术？

## 环境变量的爆炸

现在我的 `.env` 文件看起来像是被废弃的 API 密钥墓地：`OPENAI_API_KEY`、`ANTHROPIC_API_KEY`、`MISTRAL_API_KEY`、`GROQ_API_KEY`。更别提 Azure——光是说声“你好”，就得提供端点、部署名称、API 版本和密钥。

这不仅难看，更是摩擦点。每次想换模型或试新提供商，我都得改写初始化代码，去翻特定参数名的文档，还要往环境配置里再添几行。

要不我们直接…… ~~偷走~~ 借用一下数据库 URL 的思路？

## 引入 LLM 连接字符串

想象一下，只用一行就能配置整个模型接口：

```bash
llm://api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7&max_tokens=1500
llm://api.z.ai/glm-4.7?top_p=0.9&cache=true
```

---
<br />

### LLM 连接字符串的结构

![the parts of a LLM connection string](../inline-url-diagram-dark.svg)

方案是 `llm://`。主机是提供商的 API 基础 URL。路径是模型名称。查询参数则承担了通常会把代码弄得乱七八糟的运行时选项。

## 需要认证？那就加上吧。

和 `postgres://` 一样，我们可以把认证信息直接写进去：

```bash
llm://app-name:sk-proj-123456@api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7
```

*注意：把凭证写进 URL 确实有安全风险，尤其是粘贴到公开日志时。但现代日志服务已经相当擅长清除这类模式。说实话，你的 `.env` 文件真的更安全吗？请务必验证、清理，并谨慎使用。*

## 稳定性？为什么不。

很多数据库库通过列出多个主机实现轮询容错。我们的 AI 代理为何不能拥有同等可靠性？

```bash
llms://primary.gpt,backup.gpt/gpt-6?temp=0.9
```

`llms://` 里的那个 `s` 不是笔误，而是复数。若 `primary.gpt` 卡死，客户端会自动重试 `backup.gpt`。不需要额外的路由逻辑。

<blockquote class="inset">一行字符串，涵盖 **认证**、**端点** 以及 **超参数** 的全部信息。</blockquote>

## 替代格式

我并不执着于 `llm://`。具体的 scheme 并不重要，标准本身才是关键。

可以想象一种情形：使用提供商专属的 scheme 来简化，同时保持标准结构：

```bash
ollama://localhost:11434/llama3
vercel://anthropic/sonnet-4.5?temp=0.8&web_search={"maxUses":3}
bedrock://us-west-2.aws/anthropic/sonnet-4.5?temp=0.8&cacheControl=ephemeral
```

不论语法如何，核心收益显而易见：

1.  **可移植性：** 从本地脚本复制粘贴整套配置到云函数，毫无障碍。
2.  **CLI 友好：** 单个参数即可传入脚本。`my-agent --model "llm://..."` 远胜于 `my-agent --model gpt-4 --temp 0.7 --key $KEY --host ...`。
3.  **语言无关：** 各编程语言都有成熟的 URL 解析器。我们天然获得验证、解析和清理功能。

<blockquote class="ai-response inset">数据库领域花了数十年才弄清楚这件事。<br /><b>好消息是，在 AI 时间线里，这只相当于半个 vibe-year 前的事。</b></blockquote>

## 结论

我们不需要再引入另一套复杂的配置标准，也不必新建基于 YAML 的清单文件。只要使用过去 30 年里互联网一直在用的这把钥匙即可。

别再重复造轮子，给 LLM 连接也配上和数据库同等的尊重。你的 `.env` 文件（以及你的理智）会感谢你的。
