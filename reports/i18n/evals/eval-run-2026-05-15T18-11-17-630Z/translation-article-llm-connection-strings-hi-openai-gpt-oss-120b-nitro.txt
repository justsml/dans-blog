क्या आपको वह पुराना दर्द याद है, जब डेटाबेस से कनेक्ट करना मतलब अनगिनत पर्यावरण वेरिएबल्स को जुगलबंदी से संभालना?

यह एक नाज़ुक कॉन्फ़िग टॉवर था। `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`… या फिर `DB_USERNAME`? `DB_PASS` या `DB_PWD`? इस बार `PG_*` प्रीफ़िक्स चाहिए? टाइमआउट सेटिंग कहाँ रखनी है?

यह एक नाज़ुक कार्डों का घर था, जो आपके प्रोडक्शन बिल्ड को तब गिरा सकता था जब आप `HOST` को कैपिटलाइज़ करना भूल जाएँ।

फिर किसी ने सोचा, क्यों न सिर्फ एक URL¹ इस्तेमाल करें:

```bash
postgres://user:pass@host:5432/dbname
```

एक स्ट्रिंग। सब कुछ जो चाहिए। सार्वभौमिक रूप से पार्सेबल। पोर्टेबल। कहूँ तो… खूबसूरत?

तो फिर हम LLM को 1999 जैसा क्यों ट्रीट कर रहे हैं?

## Env Var विस्फोट

अभी मेरा `.env` फ़ाइल abandoned API keys की कब्रिस्तान जैसा दिखता है। `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `MISTRAL_API_KEY`, `GROQ_API_KEY`। और Azure की बात न करें—एक endpoint, एक deployment name, एक API version, और एक key चाहिए बस “हैलो” कहने के लिए।

यह सिर्फ बदसूरत नहीं, बल्कि घर्षण पैदा करता है। हर बार जब मैं मॉडल बदलता हूँ या नया प्रोवाइडर टेस्ट करता हूँ, मुझे इनिशियलाइज़ेशन कोड फिर से लिखना पड़ता है, विशिष्ट पैरामीटर नामों की डॉक्यूमेंटेशन ढूँढनी पड़ती है, और अपने environment कॉन्फ़िग में तीन और लाइनें जोड़नी पड़ती हैं।

क्या अगर हम… ~~चुराएँ~~ DB URL आइडिया को उधार ले लें?

## LLM कनेक्शन स्ट्रिंग्स का परिचय

कल्पना करें कि आप पूरे मॉडल इंटरफ़ेस को एक ही लाइन में कॉन्फ़िगर कर सकते हैं:

```bash
llm://api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7&max_tokens=1500
llm://api.z.ai/glm-4.7?top_p=0.9&cache=true
```

---
<br />

### LLM कनेक्शन स्ट्रिंग की संरचना

![LLM कनेक्शन स्ट्रिंग के भाग](../inline-url-diagram-dark.svg)

स्कीम `llm://` है। होस्ट प्रोवाइडर की API बेस URL है। पाथ मॉडल का नाम है। और क्वेरी पैरामीटर उन सभी रन‑टाइम विकल्पों को संभालते हैं जो आमतौर पर आपके कोड को गंदा कर देते हैं।

## ऑथ चाहिए? बढ़िया, जोड़ दें।

`postgres://` की तरह, हम ऑथेंटिकेशन सीधे URL में एम्बेड कर सकते हैं:

```bash
llm://app-name:sk-proj-123456@api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7
```

*नोट: हाँ, URLs में क्रेडेंशियल्स रखना सार्वजनिक लॉग में पेस्ट करने पर सुरक्षा जोखिम बन सकता है। लेकिन आधुनिक लॉगिंग सर्विसेज इन पैटर्न को स्क्रब करने में काफी सक्षम हैं, और ईमानदारी से कहें तो, क्या आप अपनी `.env` फ़ाइल को बेहतर तरीके से ट्रीट कर रहे हैं? वैरिफ़ाई करें, सैनिटाइज़ करें, और सावधानी से उपयोग करें।*

## रेज़िलिएंसी? क्यों नहीं।

कई डेटाबेस लाइब्रेरीज़ मल्टी‑हॉस्ट निर्दिष्ट करके राउंड‑रॉबिन फेलओवर सपोर्ट करती हैं। हमारे AI एजेंट्स को वही भरोसेमंदता क्यों नहीं मिलनी चाहिए?

```bash
llms://primary.gpt,backup.gpt/gpt-6?temp=0.9
```

`llms://` में `s` टाइपो नहीं, यह बहुवचन है। अगर `primary.gpt` फँस जाता है, क्लाइंट स्वचालित रूप से `backup.gpt` को रिट्राई करता है। जटिल राउटर लॉजिक की जरूरत नहीं।

<blockquote class="inset">एक स्ट्रिंग में **ऑथ**, **एंडपॉइंट**, और **हाइपरपैरामीटर्स** सब कुछ।</blockquote>

## वैकल्पिक फॉर्मेट्स

मैं `llm://` से बंधा नहीं हूँ। स्कीम का नाम कम महत्वपूर्ण है, मानक खुद ही मायने रखता है।

मैं एक ऐसी दुनिया की कल्पना कर सकता हूँ जहाँ हम संक्षिप्तता के लिए प्रोवाइडर‑स्पेसिफिक स्कीम इस्तेमाल करें, लेकिन मानक संरचना वही रहे:

```bash
ollama://localhost:11434/llama3
vercel://anthropic/sonnet-4.5?temp=0.8&web_search={"maxUses":3}
bedrock://us-west-2.aws/anthropic/sonnet-4.5?temp=0.8&cacheControl=ephemeral
```

सिंटैक्स चाहे जैसा भी हो, मुख्य लाभ अनिवार्य हैं:

1.  **पोर्टेबिलिटी:** स्थानीय स्क्रिप्ट से क्लाउड वर्कर तक पूरी कॉन्फ़िग को कॉपी‑पेस्ट करें।
2.  **CLI‑फ्रेंडली:** अपने स्क्रिप्ट्स को एक ही आर्गुमेंट पास करें। `my-agent --model "llm://..."` बनाम `my-agent --model gpt-4 --temp 0.7 --key $KEY --host ...`।
3.  **भाषा‑अज्ञेय:** हर प्रोग्रामिंग भाषा में एक मजबूत URL पार्सर होता है। हमें वैलिडेशन, पार्सिंग, और सैनिटाइज़ेशन मुफ्त में मिल जाता है।

<blockquote class="ai-response inset">डेटाबेस दुनिया को यह समझने में दशकों लगे।<br /><b>अच्छी खबर, AI टाइमलाइन में यह सिर्फ आधा “वाइब‑ईयर” पहले हुआ।</b></blockquote>

## निष्कर्ष

हमें कोई नया जटिल कॉन्फ़िगरेशन मानक या नया YAML‑आधारित मैनिफेस्ट फ़ाइल नहीं चाहिए। हमें सिर्फ वही टूल चाहिए जो पिछले 30 सालों से इंटरनेट पर काम कर रहा है।

पहिया फिर से बनाने से बचें और अपने LLM कनेक्शन को वही सम्मान दें जो आप डेटाबेस को देते हैं। आपका `.env` फ़ाइल (और आपका दिमाग) आपका धन्यवाद करेगा।
