<blockquote class="inset">
**עדכון:** מאמר זה הוביל ל[טיוטת אינטרנט עבור סכמת ה-`llm://` URI](https://datatracker.ietf.org/doc/draft-levy-llm-uri-scheme/).
</blockquote>


זוכרים את הימים הקשים כשהתחברות למסד נתונים דרשה ניהול של חבילה מבולגנת של משתני סביבה?

זה היה מגדל של קונפיגורציה עדינה. `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`… או חכו, זה היה `DB_USERNAME`? זה `DB_PASS` או `DB_PWD`? האם הפעם צריך את הקידומות `PG_*`? ואיפה לעזאזל מגיע הגדרת ה‑timeout?

זה היה בית קלפים שבור, מוכן להפיל את בניית ה‑production שלך רק בגלל ששכחת להגדיר `HOST` באותיות גדולות.

ואז מישהו חשב ברגע של ברק להשתמש ב‑URL¹:

```bash
postgres://user:pass@host:5432/dbname
```

מחרוזת אחת. כל מה שאתה צריך. ניתנת לפירוש אוניברסלי. ניידת. אפשר לומר… יפה?

אז למה אנחנו מתייחסים ל‑LLM כאילו זה 1999?

## פיצוץ משתני הסביבה

כרגע קובץ ה‑`.env` שלי נראה כמו בית קברות של מפתחות API שננטשו. `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `MISTRAL_API_KEY`, `GROQ_API_KEY`. ואל תתחילו עם Azure—צריך נקודת קצה, שם פריסה, גרסת API, ומפתח רק כדי לומר "היי".

זה לא רק מכוער; זה גורם לחיכוך. בכל פעם שאני רוצה להחליף מודל או לבדוק ספק חדש, אני משכתב קוד אתחול, מחפש בתיעוד שמות פרמטרים ספציפיים, ומוסיף שלוש שורות נוספות לקונפיגורציית הסביבה.

מה אם פשוט… ~~גנבנו~~ השאלנו את רעיון URL של מסד הנתונים?

## מציגים מחרוזות חיבור LLM

דמיינו שאתם מגדירים את כל ממשק המודל שלכם בשורה אחת:

```bash
llm://api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7&max_tokens=1500
llm://api.z.ai/glm-4.7?top_p=0.9&cache=true
```

### אנטומיה של מחרוזת חיבור LLM

![the parts of a LLM connection string](../inline-url-diagram-dark.svg)

הסכימה היא `llm://`. המארח הוא כתובת הבסיס של ה‑API של הספק. הנתיב הוא שם המודל. פרמטרי השאילתה מטפלים בכל אפשרויות הריצה שמסתימות בדרך כלל את הקוד שלכם.

## צריך אימות? מצוין, הוסיפו אותו.

בדיוק כמו `postgres://`, אפשר לשלב אימות ישירות במחרוזת:

```bash
llm://app-name:sk-proj-123456@api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7
```

*הערה: כן, הכנסת אישורים ל‑URL יכולה להיות סיכון אבטחה אם אתם מדביקים אותם ללוגים ציבוריים. אבל שירותי לוגים מודרניים טובים למדי בניקוי תבניות כאלה, ובאמת, האם אתם מתייחסים לקובץ ה‑`.env` שלכם בצורה הרבה יותר זהירה? אמתו, נקהו, והשתמשו בזהירות.*

## עמידות? למה שלא.

ספריות מסד נתונים רבות תומכות במעבר סיבובי (round‑robin) על‑ידי ציון כמה מארחים. למה שלא נעניק לסוכני ה‑AI שלנו את אותה רמת אמינות?

```bash
llms://primary.gpt,backup.gpt/gpt-6?temp=0.9
```

אות ה‑`s` ב‑`llms://` איננה טעות כתיב. היא ריבוי. אם `primary.gpt` נתקע, הלקוח מנסה אוטומטית את `backup.gpt`. אין צורך בלוגיקה מורכבת של ראוטר.

<blockquote class="inset">מחרוזת אחת עם הכל – מה‑**אימות** ועד **נקודת הקצה** ועד **הפרמטרים ההיפר‑**.</blockquote>

## פורמטים חלופיים

אני לא קשור ל‑`llm://`. הסכימה הספציפית חשובה פחות מהסטנדרט עצמו.

אפשר לדמיין עולם שבו נשתמש בסכימות ספציפיות לספק כדי לחסוך תווים, תוך שמירה על המבנה הסטנדרטי:

```bash
ollama://localhost:11434/llama3
vercel://anthropic/sonnet-4.5?temp=0.8&web_search={"maxUses":3}
bedrock://us-west-2.aws/anthropic/sonnet-4.5?temp=0.8&cacheControl=ephemeral
```

לא משנה התחביר המדויק, היתרונות המרכזיים בלתי ניתנים להכחשה:

1.  **ניידות:** העתקו והדביקו את כל הקונפיגורציה שלכם מסקריפט מקומי לעובד ענן.
2.  **ידידותי ל‑CLI:** העבירו ארגומנט יחיד לסקריפטים שלכם. `my-agent --model "llm://..."` מנצח על `my-agent --model gpt-4 --temp 0.7 --key $KEY --host ...`.
3.  **שפה‑אגרגטיבית:** לכל שפת תכנות יש מפרש URL חזק. אנחנו מקבלים ולידציה, פירוש, וניקוי בחינם.

<blockquote class="ai-response inset">עולם מסדי הנתונים לקח עשורים להבין זאת.<br /><b>חדשות טובות, בקנה מידה של AI זה רק חצי שנה לפני היום.</b></blockquote>

## המסקנה

איננו זקוקים לסטנדרט קונפיגורציה מורכב נוסף או לקובץ מניפסט מבוסס YAML. אנחנו רק צריכים להשתמש בכלי שכבר עובד עבור רוב האינטרנט במשך שלושים השנים האחרונות.

בואו נפסיק להמציא את הגלגל ונתחיל להתייחס לחיבורי ה‑LLM שלנו באותה כבוד שאנו מעניקים למסדי הנתונים. קובץ ה‑`.env` שלכם (והשפיות שלכם) יודה לכם.
