还记得连接数据库时需要处理各种杂乱环境变量的糟糕年代吗？

那时的配置就像精妙的积木塔。`DB_HOST`，`DB_PORT`，`DB_USER`，`DB_PASSWORD`，`DB_NAME`... 或者是 `DB_USERNAME`？是 `DB_PASS` 还是 `DB_PWD`？这次需要 `PG_*` 前缀吗？还有该死的超时设置该放在哪？

这脆弱的纸牌屋，只因你忘记大写 `HOST` 就可能让生产构建崩溃。

直到有人灵光一闪，提出了使用 URL¹ 的方案：

```bash
postgres://user:pass@host:5432/dbname
```

一个字符串。包含所有你需要的信息。通用可解析。可移植。敢说...很美？

那为什么我们还在用1999年的方式对待LLM？

## 环境变量爆炸

现在我的 `.env` 文件看起来像一堆废弃API密钥的坟场。`OPENAI_API_KEY`，`ANTHROPIC_API_KEY`，`MISTRAL_API_KEY`，`GROQ_API_KEY`。更别提Azure了——你得准备端点、部署名称、API版本和密钥才能说句"你好"。

这不只是丑陋；这是摩擦。每次我想切换模型或测试新提供商时，都要重写初始化代码，查找特定参数名的文档，并在环境配置中添加三行新内容。

如果我们只是... ~~偷走~~ 借鉴数据库URL的思路会怎样？

## 引入LLM连接字符串

想象用单行代码配置整个模型接口：

```bash
llm://api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7&max_tokens=1500
llm://api.z.ai/glm-4.7?top_p=0.9&cache=true
```

### LLM连接字符串的结构

![LLM连接字符串的组成部分](../inline-url-diagram-dark.svg)

方案是 `llm://`。主机是提供商的API基础URL。路径是模型名称。查询参数处理通常会弄乱代码的所有运行时选项。

## 需要认证？很好，加上去。

就像 `postgres://` 一样，我们可以直接内置认证：

```bash
llm://app-name:sk-proj-123456@api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7
```

*注意：是的，把凭证放在URL中可能会带来安全风险，如果你把它们粘贴到公共日志里。但现代日志服务在清理这些模式上做得相当不错，而且说实话，你对待 `.env` 文件真的更好吗？验证、清理并谨慎使用。*

## 需要容错？为什么不行。

许多数据库库通过指定多个主机支持轮询故障转移。为什么我们的AI代理不能有同样的可靠性？

```bash
llms://primary.gpt,backup.gpt/gpt-6?temp=0.9
```

那个 `llms://` 中的 `s` 不是拼写错误。它是复数。如果 `primary.gpt` 挂了，客户端会自动重试 `backup.gpt`。不需要复杂的路由逻辑。

<blockquote class="inset">一个字符串包含从你的 **认证** 到 **端点** 再到 **超参数** 的所有内容。</blockquote>

## 替代格式

我对 `llm://` 并不执着。具体方案不如标准本身重要。

我想象一个世界，我们可以使用提供商特定的方案来缩短名称，同时保持标准结构：

```bash
ollama://localhost:11434/llama3
vercel://anthropic/sonnet-4.5?temp=0.8&web_search={"maxUses":3}
bedrock://us-west-2.aws/anthropic/sonnet-4.5?temp=0.8&cacheControl=ephemeral
```

无论具体语法如何，核心优势是不可否认的：

1. **可移植性：** 从本地脚本复制粘贴整个配置到云工作者。
2. **CLI友好：** 向脚本传递单个参数。`my-agent --model "llm://..."` 比 `my-agent --model gpt-4 --temp 0.7 --key $KEY --host ...` 更简洁。
3. **语言无关：** 每种编程语言都有强大的URL解析器。我们免费获得验证、解析和清理功能。

<blockquote class="ai-response inset">数据库世界花了数十年才搞明白这一点。<br /><b>好消息，在AI时间线上，这不过是半年前的事。</b></blockquote>

## 结论

我们不需要另一个复杂的配置标准或新的YAML清单文件。我们只需要使用过去30年来互联网其他部分一直在使用的工具。

让我们停止重复造轮子，开始像对待数据库一样对待LLM连接。你的 `.env` 文件（和你的理智）会感谢你的。
