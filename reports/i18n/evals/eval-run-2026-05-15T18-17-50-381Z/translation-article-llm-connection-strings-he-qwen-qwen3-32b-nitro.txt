**עדכון:** המאמר גרם ל-[Internet-Draft של סכמת URI `llm://`](https://datatracker.ietf.org/doc/draft-levy-llm-uri-scheme/).

זכור את ימי הלא-טובים כשהתחברת למסד נתונים וסידרת סביבה של משתני סביבה מוסגרים?

זה היה מגדל רגיש של תצורת. `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`... או אולי זה היה `DB_USERNAME`? האם זה `DB_PASS` או `DB_PWD`? האם צריך префиксы `PG_*` הפעם? ולמה לעזאזל הולך הגדרת הזמן החיכוך?

זה היה בית כרטיסים רך, מוכן להפיל את הבניית היצור שלך בגלל ששכחת לגדול `HOST`.

ואז, מישהו היה גאוני מספיק להציע פשוט להשתמש ב-URL¹:

```bash
postgres://user:pass@host:5432/dbname
```

строка אחת. כל מה שדרוש. ניתנת לנתח באופן אוניברסלי. ניידת. אומרים, יפה?

אז למה אנחנו מטפלים ב-LLM כאילו זה 1999?

## התפוצצות של משתני הסביבה

נכון עכשיו, קובץ `.env` שלי נראה כמו קבר ל-API Keys עזבי. `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `MISTRAL_API_KEY`, `GROQ_API_KEY`. ואל תתחילו עם Azure – אתה צריך נקודת קצה, שם ה депלוי, גרסה API וключ רק כדי לומר "שלום".

זה לא רק醜陋; זה פגיעה. בכל פעם שאני רוצה להחליף מודל או לבדוק ספק חדש, אני כותב מחדש קוד אתחול, מחפש מסמך על שמות פרמטרים ספציפיים, ומוסיף שלוש שורות נוספות לקובץ הסביבה.

מה אם פשוט... ~~נسرב~~ נגנוב את הרעיון של URL מסד?

## היכרות עם מיתרי חיבור LLM

דמיינו להגדיר את כל ממשק המודל שלך בשורה אחת:

```bash
llm://api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7&max_tokens=1500
llm://api.z.ai/glm-4.7?top_p=0.9&cache=true
```

### אנטומיה של מיתר חיבור LLM

![חלקי מיתר חיבור LLM](../inline-url-diagram-dark.svg)

הסכמה היא `llm://`. המארח הוא הבסיס של ה-API של הספק. המסלול הוא שם המודל. והפרמטרים של שאילת ה-URL מטפלים בכל האפשרויות הריצה שבעתיד יתבלגן את הקוד שלך.

## צריך אימות? מושלם, הוסף את זה.

בדיוק כמו `postgres://`, נוכל לשלב אימות ישירות:

```bash
llm://app-name:sk-proj-123456@api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7
```

*הערה: כן, הוספת מזדהים ל-URL יכולה להיות סיכון אבטחה אם אתה מכניס אותם ללוגים ציבוריים. אבל שירותים מודרניים של לוגים די טובים במחיקת דפוסים אלו, והonestly, האם אתה מטפל בקובץ `.env` שלך טוב יותר? בדוק, סניף, ו השתמש בזהירות.*

## עמידות? למה לא.

ריבועי מסד רבים תומכים בפיטור עיגוליים על ידי ציון מספר מארחים. למה לא לספק את אותה עמידות ל代办 agents?

```bash
llms://primary.gpt,backup.gpt/gpt-6?temp=0.9
```

ה-`s` ב-`llms://` איננו טעות. זו פלורל. אם `primary.gpt` נתקע, הלקוח מנסה מחדש `backup.gpt` אוטומטית. אין צורך בตรיקי לוגיקה מסובכים.

<blockquote class="inset">строка אחת הכוללת את כל מה שדרוש לך – מהאימות דרך הכתובת ועד ההיפר-פרמטרים.</blockquote>

## פורמטים חלופיים

אני לא נשוי ל-`llm://`. הסכמה הספציפית פחות חשובה מהתקן עצמו.

אני יכול לדמיין עולם שבו נשתמש בסכימות ספק ספציפיות לצורך קיצור, תוך שמירה על מבנה התקן:

```bash
ollama://localhost:11434/llama3
vercel://anthropic/sonnet-4.5?temp=0.8&web_search={"maxUses":3}
bedrock://us-west-2.aws/anthropic/sonnet-4.5?temp=0.8&cacheControl=ephemeral
```

בכל מקרה, התועלות הבסיסיות אינן ספק:

1.  **ניידות:** העתק וסחוב את כל התצורה שלך מהתוכנית המקומית ל-Worker הענן.
2.  **ידידותי CLI:** העבר פרמטר אחד לск립טים שלך. `my-agent --model "llm://..."` מנצח `my-agent --model gpt-4 --temp 0.7 --key $KEY --host ...`.
3.  **שפת נייטרליות:** כל שפת תכנות מכילה מפרaser URL מתקדם. אנחנו מקבלים אימות, נתח, וניקיון מجانית.

<blockquote class="ai-response inset">העולם של מסד לקח עשורים להבין את זה.<br /><b>חדשות טובות, בזמנים של AI, זה רק חצי vibe-year אחורה.</b></blockquote>

## החלט

אנחנו לא צריכים סטנדרט תצורה מורכב נוסף או קובץ תצורה חדש על בסיס YAML. אנחנו רק צריכים להשתמש בเครื่อง היחידי שעובד עבור שאר האינטרנט בשלושים השנים האחרונות.

נפסיק להמציא מחדש את הגלגל ונתחיל לטפל בקonneקציות LLM שלנו באותה מידה שבה אנחנו מטפלים במסדים. קובץ `.env` שלך (וגם מינוחך) יתודה לך.
