Помните те старые времена, когда подключение к базе данных означало жонглирование разношёрстным набором переменных окружения?

Это была башня из хрупких настроек. `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`... или погодите, это было `DB_USERNAME`? `DB_PASS` или `DB_PWD`? Нужны ли на этот раз префиксы `PG_*`? И куда, чёрт возьми, девать настройку таймаута?

Это был карточный домик, готовый рухнуть в продакшене из-за того, что вы забыли написать `HOST` с большой буквы.

А потом кому-то пришла гениальная идея — просто использовать URL¹:

```bash
postgres://user:pass@host:5432/dbname
```

Одна строка. Всё, что нужно. Универсально разбирается. Переносимо. Осмелюсь сказать... красиво?

Так почему же мы обращаемся с LLM так, будто на дворе 1999 год?

## Взрыв переменных окружения

Прямо сейчас мой `.env` файл выглядит как кладбище заброшенных API-ключей. `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `MISTRAL_API_KEY`, `GROQ_API_KEY`. И даже не заставляйте меня начинать про Azure — вам нужны endpoint, имя развёртывания, версия API и ключ, чтобы просто сказать «привет».

Это не просто уродливо; это трение. Каждый раз, когда я хочу сменить модель или протестировать нового провайдера, я переписываю код инициализации, охочусь за документацией по конкретным именам параметров и добавляю ещё три строки в конфигурацию окружения.

Что если мы просто... ~~украдём~~ позаимствуем идею URL для БД?

## Знакомьтесь: строки подключения для LLM

Представьте, что вы настраиваете весь интерфейс модели одной строкой:

```bash
llm://api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7&max_tokens=1500
llm://api.z.ai/glm-4.7?top_p=0.9&cache=true
```

### Анатомия строки подключения LLM

![части строки подключения LLM](../inline-url-diagram-dark.svg)

Схема — `llm://`. Хост — базовый URL API провайдера. Путь — имя модели. А параметры запроса обрабатывают все опции времени выполнения, которые обычно засоряют ваш код.

## Нужна аутентификация? Отлично, добавляем.

Как и в `postgres://`, мы можем встроить аутентификацию прямо в URL:

```bash
llm://app-name:sk-proj-123456@api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7
```

*Примечание: да, размещение учётных данных в URL может быть риском безопасности, если вы вставляете их в публичные логи. Но современные сервисы логирования неплохо умеют вычищать такие паттерны, и, честно говоря, вы лучше обращаетесь со своим `.env` файлом? Проверяйте, очищайте и используйте с осторожностью.*

## Отказоустойчивость? А почему бы и нет.

Многие библиотеки для работы с базами данных поддерживают round-robin failover, указывая несколько хостов. Почему наши AI-агенты не должны иметь такую же надёжность?

```bash
llms://primary.gpt,backup.gpt/gpt-6?temp=0.9
```

Буква `s` в `llms://` — не опечатка. Это множественное число. Если `primary.gpt` зависает, клиент автоматически повторяет запрос к `backup.gpt`. Никакой сложной логики роутера.

<blockquote class="inset">Одна строка со всем: от **аутентификации** до **endpoint** и **гиперпараметров**.</blockquote>

## Альтернативные форматы

Я не привязан к `llm://`. Конкретная схема менее важна, чем сам стандарт.

Можно представить мир, где мы используем специфичные для провайдера схемы для краткости, сохраняя стандартную структуру:

```bash
ollama://localhost:11434/llama3
vercel://anthropic/sonnet-4.5?temp=0.8&web_search={"maxUses":3}
bedrock://us-west-2.aws/anthropic/sonnet-4.5?temp=0.8&cacheControl=ephemeral
```

Независимо от точного синтаксиса, основные преимущества неоспоримы:

1.  **Переносимость:** Скопируйте и вставьте всю конфигурацию из локального скрипта в облачный воркер.
2.  **Удобство для CLI:** Передайте один аргумент в свои скрипты. `my-agent --model "llm://..."` лучше, чем `my-agent --model gpt-4 --temp 0.7 --key $KEY --host ...`.
3.  **Языковая независимость:** Каждый язык программирования имеет надёжный парсер URL. Мы получаем валидацию, разбор и очистку бесплатно.
