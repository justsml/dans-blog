データベースに接続するのに、環境変数の寄せ集めを扱わなければならなかった悪しき時代を覚えていますか？

それは繊細な設定の塔でした。`DB_HOST`、`DB_PORT`、`DB_USER`、`DB_PASSWORD`、`DB_NAME`……いや待て、`DB_USERNAME` だったか？ `DB_PASS` か `DB_PWD` か？ 今回は `PG_*` プレフィックスが必要なのか？ そしてタイムアウト設定はどこに書くんだ？

`HOST` の大文字小文字を間違えただけでプロダクションビルドが吹き飛ぶ、カードの家のように脆い代物でした。

そこである賢者が、URL を使えばいいんじゃないかとひらめいたのです¹：

```bash
postgres://user:pass@host:5432/dbname
```

たった一つの文字列。必要なものはすべて揃っている。どこでもパースできる。ポータブル。あえて言おう、美しい、と。

なのに、なぜ LLM に対しては 1999 年方式を続けているのでしょうか？

## 環境変数の爆発

今、私の `.env` ファイルは放棄された API キーの墓場と化しています。`OPENAI_API_KEY`、`ANTHROPIC_API_KEY`、`MISTRAL_API_KEY`、`GROQ_API_KEY`。Azure に至っては、エンドポイント、デプロイ名、API バージョン、キーが揃わなければ「こんにちは」すら言えません。

醜いだけでなく、摩擦でもあります。モデルを切り替えたり、新しいプロバイダを試すたびに、初期化コードを書き直し、特定のパラメータ名のドキュメントを探し回り、環境設定にさらに 3 行追加することになります。

だったら、DB URL のアイデアを……~~パク~~ 借用してみてはどうでしょう？

## LLM コネクション文字列の紹介

モデルインターフェース全体をたった一行で設定できる世界を想像してみてください：

```bash
llm://api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7&max_tokens=1500
llm://api.z.ai/glm-4.7?top_p=0.9&cache=true
```

---
<br />

### LLM コネクション文字列の構造

![LLM コネクション文字列の各部分](./inline-url-diagram-dark.svg)

スキームは `llm://`。ホストはプロバイダの API ベース URL。パスはモデル名。そしてクエリパラメータで、通常コードを散らかす実行時オプションをすべて処理します。

## 認証が必要？ よし、追加しよう。

`postgres://` と同じように、認証情報を URL に直接埋め込めます：

```bash
llm://app-name:sk-proj-123456@api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7
```

*注：確かに、URL に資格情報を入れるのは、公開ログに貼り付ける場合にはセキュリティリスクになります。しかし、最近のログサービスはこれらのパターンをうまく除去してくれますし、正直なところ、あなたの `.env` ファイルの扱いはもっとマシですか？ 検証し、サニタイズし、注意して使用してください。*

## レジリエンシー？ なぜダメなのか。

多くのデータベースライブラリは、複数のホストを指定することでラウンドロビンによるフェイルオーバーをサポートしています。AI エージェントにも同じ信頼性があってしかるべきでしょう？

```bash
llms://primary.gpt,backup.gpt/gpt-6?temp=0.9
```

`llms://` の `s` はタイポではありません。複数形です。`primary.gpt` がハングしたら、クライアントは自動的に `backup.gpt` にリトライします。複雑なルーターロジックは不要です。

<blockquote class="inset">**認証**から**エンドポイント**、**ハイパーパラメータ**まで、すべてを一つの文字列に。</blockquote>

## 代替フォーマット

私は `llm://` に固執しているわけではありません。具体的なスキームよりも、標準そのものが重要です。

標準構造を保ちつつ、簡潔さのためにプロバイダ固有のスキームを使う世界もありえるでしょう：

```bash
ollama://localhost:11434/llama3
vercel://anthropic/sonnet-4.5?temp=0.8&web_search={"maxUses":3}
bedrock://us-west-2.aws/anthropic/sonnet-4.5?temp=0.8&cacheControl=ephemeral
```

正確な構文はともかく、中核的な利点は否定できません：

1.  **ポータビリティ:** ローカルスクリプトからクラウドワーカーへ、設定全体をコピペ。
2.  **CLI フレンドリー:** スクリプトに単一の引数を渡すだけ。`my-agent --model "llm://..."` は `my-agent --model gpt-4 --temp 0.7 --key $KEY --host ...` より優れています。
3.  **言語非依存:** あらゆるプログラミング言語に堅牢な URL パーサーが存在します。バリデーション、パース、サニタイゼーションを無料で手に入れられます。

<blockquote class="ai-response inset">データベースの世界がこれを理解するのに数十年かかりました。<br /><b>良い知らせは、AI のタイムラインでは、それはほんの半バイブ年前のことだということです。</b></blockquote>

## 結論

複雑な設定標準や新しい YAML マニフェストファイルは必要ありません。必要なのは、過去 30 年間インターネットの他の部分で機能してきたツールを活用することだけです。

車輪の再発明はやめて、LLM 接続をデータベースと同じ敬意をもって扱いましょう。あなたの `.env` ファイル（そしてあなたの正気）が感謝してくれるはずです。
