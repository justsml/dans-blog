<blockquote class="inset">
**עדכון:** מאמר זה הוביל ל-[טיוטת אינטרנט עבור תכנית ה-URI `llm://`](https://datatracker.ietf.org/doc/draft-levy-llm-uri-scheme/).
</blockquote>

זוכרים את הימים הרעים ההם, כשהתחברות לבסיס נתונים פירושה להתמודד עם אוסף אקראי של משתני סביבה?

זה היה מגדל קלפים של קונפיגורציה עדינה. `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`... או רגע, אולי זה `DB_USERNAME`? זה `DB_PASS` או `DB_PWD`? הפעם צריך את הקידומות `PG_*`? ואיפה לעזאזל שמים את הגדרת ה-timeout?

זה היה בית קלפים רעוע, מוכן להפיל את ה-build שלך בייצור רק כי שכחת לאות רישית ב-`HOST`.

ואז, למישהו היה רעיון מבריק: פשוט להשתמש בכתובת URL¹:

```bash
postgres://user:pass@host:5432/dbname
```

מחרוזת אחת. כל מה שצריך. ניתנת לפרסור אוניברסלי. נייד. ואעז לומר... יפה?

אז למה אנחנו מתייחסים ל-LLMs כאילו אנחנו ב-1999?

## פיצוץ משתני הסביבה

כרגע, קובץ `.env` שלי נראה כמו בית קברות של מפתחות API נטושים. `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `MISTRAL_API_KEY`, `GROQ_API_KEY`. ואל תתחילו אותי עם Azure – צריך endpoint, שם deployment, גרסת API, ומפתח רק כדי להגיד "שלום".

זה לא רק מכוער; זה חיכוך. בכל פעם שאני רוצה להחליף מודל או לבדוק ספק חדש, אני כותב מחדש קוד אתחול, מחפש תיעוד לשמות פרמטרים ספציפיים, ומוסיף עוד שלוש שורות לקונפיגורציית הסביבה.

מה אם פשוט... ~~נגנוב~~ נשאיל את הרעיון של כתובות URL לבסיסי נתונים?

## היכרות עם מחרוזות חיבור LLM

דמיינו שאתם מגדירים את כל ממשק המודל שלכם בשורה אחת:

```bash
llm://api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7&max_tokens=1500
llm://api.z.ai/glm-4.7?top_p=0.9&cache=true
```

### האנטומיה של מחרוזת חיבור LLM

![החלקים של מחרוזת חיבור LLM](../inline-url-diagram-dark.svg)

התכנית היא `llm://`. ה-host הוא כתובת ה-API הבסיסית של הספק. ה-path הוא שם המודל. ופרמטרי השאילתה מטפלים בכל אפשרויות הריצה שבדרך כלל מלכלכות את הקוד שלך.

## צריך אימות? מעולה, הוסיפו אותו.

בדיוק כמו `postgres://`, אפשר להטמיע אימות ישירות בכתובת:

```bash
llm://app-name:sk-proj-123456@api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7
```

*הערה: כן, הכנסת פרטי התחברות לכתובות URL עלולה להיות סיכון אבטחתי אם אתם מדביקים אותן בלוגים ציבוריים. אבל שירותי לוגינג מודרניים די טובים בניקוי תבניות כאלה, ובכנות, אתם מתייחסים הרבה יותר טוב לקובץ `.env` שלכם? אמתו, נטרלו, והשתמשו בזהירות.*

## עמידות? למה שלא.

ספריות רבות של בסיסי נתונים תומכות ב-failover בצורה של round-robin על ידי ציון מספר hosts. למה שהסוכנים ה-AI שלנו לא ייהנו מאותה אמינות?

```bash
llms://primary.gpt,backup.gpt/gpt-6?temp=0.9
```

ה-`s` ב-`llms://` הוא לא טעות. זה רבים. אם `primary.gpt` נתקע, הלקוח מנסה אוטומטית את `backup.gpt`. אין צורך בלוגיקת ראוטר מורכבת.

<blockquote class="inset">מחרוזת אחת עם הכל – מה-**auth** שלך, דרך ה-**endpoint** ועד ל-**hyperparameters** שלך.</blockquote>

## פורמטים חלופיים

אני לא כבול ל-`llm://`. התכנית הספציפית פחות חשובה מהסטנדרט עצמו.

אני יכול לדמיין עולם שבו משתמשים בתכניות ספציפיות לספק לקיצור, תוך שמירה על המבנה הסטנדרטי:

```bash
ollama://localhost:11434/llama3
vercel://anthropic/sonnet-4.5?temp=0.8&web_search={"maxUses":3}
bedrock://us-west-2.aws/anthropic/sonnet-4.5?temp=0.8&cacheControl=ephemeral
```

ללא קשר לתחביר המדויק, היתרונות המרכזיים הם בלתי ניתנים להכחשה:

1.  **ניידות:** העתק-הדבק את כל הקונפיגורציה שלך מסקריפט מקומי לעובד ענן.
2.  **ידידותי לשורת הפקודה:** העבר ארגומנט בודד לסקריפטים שלך. `my-agent --model "llm://..."` מנצח את `my-agent --model gpt-4 --temp 0.7 --key $KEY --host ...`.
3.  **בלתי תלוי בשפה:** לכל שפת תכנות יש מפענח URL חזק. אנחנו מקבלים אימות, פרסור וניטרול בחינם.

<blockquote class="ai-response inset">לעולם בסיסי הנתונים לקח עשורים להבין את זה.<br /><b>חדשות טובות, בצירי זמן של AI, זה רק בערך חצי שנת וייב.</b></blockquote>

## פסק הדין

אנחנו לא צריכים עוד סטנדרט קונפיגורציה מורכב או קובץ מניפסט מבוסס YAML. אנחנו פשוט צריכים להשתמש בכלי האחד שעובד עבור שאר האינטרנט כבר 30 שנה.

בואו נפסיק להמציא את הגלגל מחדש ונתחיל להתייחס לחיבורי ה-LLM שלנו באותו כבוד שאנחנו נותנים לבסיסי הנתונים שלנו. קובץ `.env` שלך (והשפיות שלך) יודו לך.
