<blockquote class="inset">
**עדכון:** מאמר זה הוביל ל[טיוטת אינטרנט עבור סכמת ה-URI `llm://`](https://datatracker.ietf.org/doc/draft-levy-llm-uri-scheme/).
</blockquote>


זוכרים את הימים הקשים שבהם חיבור למסד נתונים היה משימה של ניהול קופסה של משתני סביבה שונים?

היה זה מגדל של קונפיגורציה עדינה. `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`... או רגע, האם זה היה `DB_USERNAME`? האם זה `DB_PASS` או `DB_PWD`? האם הפעם צריך קידומות `PG_*`? ואיפה בכלל הולך הגדרת ה‑timeout?

זה היה בית קרטון שביר, מוכן להפיל את בניית ה‑production שלך רק בגלל ששכחת להגדיל את האות ב‑`HOST`.

ואז מישהו חשב על רעיון מבריק – להשתמש פשוט ב‑URL¹:

```bash
postgres://user:pass@host:5432/dbname
```

מחרוזת אחת. כל מה שצריך. ניתנת לפירוש אוניברסלי. ניידת. אפשר לומר... יפה?

אז למה אנחנו מתייחסים ל‑LLM כאילו אנחנו ב‑1999?

## פיצוץ משתני הסביבה

כרגע קובץ ה‑`.env` שלי נראה כמו בית קברות של מפתחות API שננטשו. `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `MISTRAL_API_KEY`, `GROQ_API_KEY`. ולא נדבר על Azure – צריך נקודת קצה, שם פריסה, גרסת API ומפתח רק כדי לומר "היי".

זה לא רק מכוער; זה גורם לחיכוך. בכל פעם שאני רוצה להחליף מודל או לבדוק ספק חדש, אני משכתב קוד אתחול, מחפש תיעוד לשמות פרמטרים ספציפיים, ומוסיף שלוש שורות נוספות לקונפיגורציית הסביבה.

מה אם פשוט... ~~גנבנו~~ השאלנו את רעיון ה‑DB URL?

## הצגת מחרוזות חיבור LLM

דמיינו שאתם מגדירים את כל ממשק המודל שלכם בשורה אחת:

```bash
llm://api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7&max_tokens=1500
llm://api.z.ai/glm-4.7?top_p=0.9&cache=true
```

### מבנה של מחרוזת חיבור LLM

![החלקים של מחרוזת חיבור LLM](../inline-url-diagram-dark.svg)

הסכמת היא `llm://`. המארח הוא כתובת הבסיס של ה‑API של הספק. הנתיב הוא שם המודל. פרמטרי השאילתה מטפלים בכל אפשרויות הריצה שלרוב מציפות את הקוד שלכם.

## צריך אימות? מצוין, הוסיפו אותו.

בדיוק כמו `postgres://`, אפשר לשלב אימות ישירות ב‑URL:

```bash
llm://app-name:sk-proj-123456@api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7
```

*הערה: כן, הכנסת אישורים ב‑URL יכולה להיות סיכון אבטחה אם אתם מדביקים אותם ללוגים ציבוריים. אך שירותי לוג מודרניים טובים למדי בניקוי תבניות כאלה, ובכנות, האם אתם מתייחסים לקובץ ה‑`.env` שלכם הרבה יותר טוב? אמתו, נקהו, והשתמשו בזהירות.*

## עמידות? למה שלא.

רבות מספריות מסדי נתונים תומכות במעבר סיבובי (round‑robin) על‑ידי ציון כמה מארחים. למה שלא סוכני ה‑AI שלנו יהיו באותה רמת אמינות?

```bash
llms://primary.gpt,backup.gpt/gpt-6?temp=0.9
```

אות ה‑`s` ב‑`llms://` איננה טעות כתיב. היא ריבוי. אם `primary.gpt` נתקע, הלקוח מנסה אוטומטית את `backup.gpt`. אין צורך בלוגיקה מורכבת של נתב.

<blockquote class="inset">מחרוזת אחת עם הכול – מה‑**אימות** ועד **נקודת הקצה** ועד **הפרמטרים ההיפריים**.</blockquote>

## פורמטים חלופיים

אני לא קשור רק ל‑`llm://`. הסכמת הספציפית חשובה פחות מהתקן עצמו.

אפשר לדמיין עולם שבו נשתמש בסכמות ספציפיות לספקים לצורך קיצור, תוך שמירה על המבנה הסטנדרטי:

```bash
ollama://localhost:11434/llama3
vercel://anthropic/sonnet-4.5?temp=0.8&web_search={"maxUses":3}
bedrock://us-west-2.aws/anthropic/sonnet-4.5?temp=0.8&cacheControl=ephemeral
```

לא משנה התחביר המדויק, היתרונות המרכזיים בלתי ניתנים להכחשה:

1.  **ניידות:** העתקו והדביקו את כל הקונפיגורציה שלכם מסקריפט מקומי לעובד ענן.
2.  **ידידותי ל‑CLI:** העבירו ארגומנט יחיד לסקריפטים שלכם. `my-agent --model "llm://..."` מנצח על `my-agent --model gpt-4 --temp 0.7 --key $KEY --host ...`.
3.  **שפה‑אגרגטית:** לכל שפת תכנות יש מפרש URL חזק. אנחנו מקבלים ולידציה, פירוש, וניקוי בחינם.

<blockquote class="ai-response inset">עולם מסדי הנתונים לקח עשורים להבין זאת.<br /><b>חדשות טובות, בקנה מידה של AI זה רק לפני חצי שנה.</b></blockquote>

## המסקנה

איננו זקוקים לתקן קונפיגורציה מורכב נוסף או לקובץ מניפסט מבוסס YAML חדש. אנחנו רק צריכים להשתמש בכלי שכבר עובד עבור רוב האינטרנט במשך שלושים השנים האחרונות.

בואו נפסיק להמציא את הגלגל ונתחיל להתייחס לחיבורי ה‑LLM שלנו באותה כבוד שאנו מעניקים למסדי הנתונים. קובץ ה‑`.env` שלכם (והשפיות שלכם) יודה לכם.
