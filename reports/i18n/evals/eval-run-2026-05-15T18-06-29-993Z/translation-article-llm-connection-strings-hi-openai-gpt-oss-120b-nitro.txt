<blockquote class="inset">
**अद्यतन:** इस लेख ने `llm://` URI स्कीम के लिए एक [Internet-Draft for the `llm://` URI scheme](https://datatracker.ietf.org/doc/draft-levy-llm-uri-scheme/) का मार्ग प्रशस्त किया।
</blockquote>


पुराने दिनों को याद करें जब डेटाबेस से कनेक्ट करना मतलब पर्यावरण वेरिएबल्स की बेतरतीब जार में हाथ डालना था?

यह एक नाज़ुक कॉन्फ़िग टॉवर था। `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`... या फिर `DB_USERNAME`? क्या यह `DB_PASS` है या `DB_PWD`? इस बार मुझे `PG_*` प्रीफ़िक्स चाहिए? और टाइमआउट सेटिंग कहाँ रखूँ?

यह एक नाज़ुक कार्डों का घर था, जो आपके प्रोडक्शन बिल्ड को तब गिरा देता जब आप `HOST` को कैपिटलाइज़ करना भूल जाते।

फिर किसी ने एक ही URL¹ का उपयोग करने का शानदार विचार किया:

```bash
postgres://user:pass@host:5432/dbname
```

एक स्ट्रिंग। सब कुछ जो चाहिए। सार्वभौमिक रूप से पार्स करने योग्य। पोर्टेबल। कहूँ तो... सुंदर?

तो फिर हम LLMs को 1999 जैसा क्यों मान रहे हैं?

## एनव वेरिएबल विस्फोट

अभी मेरा `.env` फ़ाइल abandoned API keys की कब्रगाह जैसा दिखता है। `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `MISTRAL_API_KEY`, `GROQ_API_KEY`। और Azure की बात न करें—आपको एक endpoint, एक deployment name, एक API version, और एक key चाहिए बस “हैलो” कहने के लिए।

यह सिर्फ बदसूरत नहीं; यह घर्षण है। हर बार जब मैं मॉडल बदलना या नया प्रोवाइडर टेस्ट करना चाहता हूँ, मुझे इनिशियलाइज़ेशन कोड फिर से लिखना पड़ता है, विशिष्ट पैरामीटर नामों की डॉक्यूमेंटेशन ढूँढ़नी पड़ती है, और अपने environment कॉन्फ़िग में तीन और लाइनें जोड़नी पड़ती हैं।

क्या अगर हम बस... ~~चुराएँ~~ DB URL विचार को उधार ले लें?

## LLM कनेक्शन स्ट्रिंग्स का परिचय

कल्पना करें कि आप अपने पूरे मॉडल इंटरफ़ेस को एक ही लाइन में कॉन्फ़िगर कर रहे हैं:

```bash
llm://api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7&max_tokens=1500
llm://api.z.ai/glm-4.7?top_p=0.9&cache=true
```

---
<br />

### LLM कनेक्शन स्ट्रिंग की संरचना

![LLM कनेक्शन स्ट्रिंग के भाग](../inline-url-diagram-dark.svg)

स्कीम `llm://` है। होस्ट प्रोवाइडर की API बेस URL है। पाथ मॉडल का नाम है। और क्वेरी पैरामीटर सभी रनटाइम विकल्पों को संभालते हैं जो आमतौर पर आपके कोड को गंदा करते हैं।

## ऑथ चाहिए? बढ़िया, जोड़ दें।

`postgres://` की तरह, हम ऑथेंटिकेशन को सीधे अंदर बेक कर सकते हैं:

```bash
llm://app-name:sk-proj-123456@api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7
```

*नोट: हाँ, URLs में क्रेडेंशियल्स डालना सुरक्षा जोखिम हो सकता है अगर आप उन्हें सार्वजनिक लॉग में पेस्ट करते हैं। लेकिन आधुनिक लॉगिंग सर्विसेज इन पैटर्न को साफ़ करने में काफी अच्छी हैं, और ईमानदारी से कहें तो, क्या आप अपनी `.env` फ़ाइल को बेहतर तरीके से संभाल रहे हैं? सत्यापित करें, साफ़ करें, और सावधानी से उपयोग करें।*

## रेजिलिएंसी? क्यों नहीं।

कई डेटाबेस लाइब्रेरीज़ कई होस्ट निर्दिष्ट करके राउंड‑रॉबिन फेलओवर सपोर्ट करती हैं। हमारे AI एजेंट्स को वही भरोसेमंदता क्यों नहीं मिलनी चाहिए?

```bash
llms://primary.gpt,backup.gpt/gpt-6?temp=0.9
```

`llms://` में वह `s` टाइपो नहीं है। यह बहुवचन है। अगर `primary.gpt` फँस जाता है, क्लाइंट स्वचालित रूप से `backup.gpt` को रीट्राई करता है। जटिल राउटर लॉजिक की ज़रूरत नहीं।

<blockquote class="inset">एक स्ट्रिंग जिसमें **ऑथ**, **एंडपॉइंट**, और **हाइपरपैरामीटर्स** सब कुछ शामिल है।</blockquote>

## वैकल्पिक फ़ॉर्मेट्स

मैं `llm://` से बंधा नहीं हूँ। विशिष्ट स्कीम स्वयं से कम महत्वपूर्ण है, मानक ही मायने रखता है।

मैं एक ऐसी दुनिया की कल्पना कर सकता हूँ जहाँ हम संक्षिप्तता के लिए प्रोवाइडर‑स्पेसिफिक स्कीम का उपयोग करें, जबकि मानक संरचना को बनाए रखें:

```bash
ollama://localhost:11434/llama3
vercel://anthropic/sonnet-4.5?temp=0.8&web_search={"maxUses":3}
bedrock://us-west-2.aws/anthropic/sonnet-4.5?temp=0.8&cacheControl=ephemeral
```

सटीक सिंटैक्स चाहे जो भी हो, मुख्य लाभ स्पष्ट हैं:

1.  **पोर्टेबिलिटी:** स्थानीय स्क्रिप्ट से क्लाउड वर्कर तक अपनी पूरी कॉन्फ़िग को कॉपी‑पेस्ट करें।
2.  **CLI‑फ्रेंडली:** अपने स्क्रिप्ट को एक ही आर्ग्यूमेंट पास करें। `my-agent --model "llm://..."` बनाम `my-agent --model gpt-4 --temp 0.7 --key $KEY --host ...`।
3.  **भाषा‑अज्ञेय:** हर प्रोग्रामिंग भाषा में एक मजबूत URL पार्सर होता है। हमें वैलिडेशन, पार्सिंग, और सैनिटाइज़ेशन मुफ्त में मिल जाता है।

<blockquote class="ai-response inset">डेटाबेस दुनिया को इसे समझने में दशकों लगे।<br /><b>अच्छी खबर, AI टाइमलाइन में यह सिर्फ आधा “वाइब‑ईयर” पहले हुआ है।</b></blockquote>

## निष्कर्ष

हमें कोई नया जटिल कॉन्फ़िगरेशन मानक या नया YAML‑आधारित मैनिफेस्ट फ़ाइल की जरूरत नहीं है। हमें सिर्फ वही टूल इस्तेमाल करना है जो पिछले 30 सालों से इंटरनेट के बाकी हिस्सों में काम कर रहा है।

पहिया फिर से न बनाएं, और अपने LLM कनेक्शन को वही सम्मान दें जो हम डेटाबेस को देते हैं। आपकी `.env` फ़ाइल (और आपका दिमाग) आपका धन्यवाद करेगा।
