你知道那种改一行代码，内存占用就降一半的感觉吗？我就在Chrome DevTools性能监视器里体验过——一个仪表盘应用从每小时泄漏100MB，变成整个下午都干干净净。

那一行改动：`new Map()` 变成了 `new WeakMap()`。

就这。同样的API，同样的用法，底层行为却完全不同。但要理解为什么有效，就得理解大多数JavaScript开发者从没想过的事：当没人再盯着你的数据时，会发生什么。

## 当引用变成锚点

JavaScript里的普通Map把它的键当成宝贝。一旦放进去，Map就会死死攥住不放。垃圾回收器看到这种关系，心想：“显然他们还需要这个对象，最好别碰。”

这种保护欲在你存储临时对象的元数据时就变成了问题。被移除的DOM节点、过期的用户会话、卸载的组件实例。Map不知道这些对象已经没用了，它只知道它有个引用，所以一直留着它们。

```javascript
const cache = new Map();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// 元素已经从DOM中移除，但cache还在内存中保留它
```

垃圾回收器没法清理`element`，因为`cache`还指着它。这叫“强引用”，在长期运行的单页应用里，它就成了最终让浏览器崩溃的泄漏。

## WeakMap改变了规则

WeakMap的工作方式不同。它把键当作临时居民，而不是永久住户。当你把东西存进WeakMap时，你实际上是在说：“我想把这个数据和这个对象关联起来，但我不想成为它存活的原因。”

如果唯一让一个对象留在内存里的东西是WeakMap，垃圾回收器就可以把它带走。对象消失时，WeakMap里的条目也跟着消失。不需要手动清理。

```javascript
const cache = new WeakMap();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// 元素被垃圾回收
// 缓存条目自动消失
```

我跑过一个基准测试：创建10万个DOM节点，为每个节点存储元数据，然后全部移除。用Map时，浏览器保留了150-200MB内存；用WeakMap时，降到了70-80MB。同样的代码，同样的功能，内存占用减半。

## 你放弃了什么

WeakMap有一些约束，乍看像是限制，但你会发现正是这些约束成就了它的魔力。

**你不能遍历WeakMap。** 没有`forEach`，没有`keys()`，没有`values()`。细想一下这很合理：垃圾回收器可能在你循环的中途删除一个条目。你真的想处理这种情况吗？

你无法检查大小。没有`.size`属性，没有`.length`。同样，这是一个移动的目标——在你询问和得到答案之间，数量可能已经变了。

**键必须是对象。** 不能是字符串、数字或原始值。这是弱引用工作的基础：原始值没有独立于其值的可追踪标识。

这些不是缺陷，而是设计。WeakMap为一项特定工作而生：将元数据附加到对象上，同时不阻止这些对象被清理。如果你需要遍历、原始键或条目计数，那你可能是在解决另一个问题，应该用普通的Map。

## 这在实际中帮到了什么

“私有数据”模式是WeakMap最初的用例，早在JavaScript拥有`#private`字段之前。库会在类外部创建一个WeakMap，用它存储不应在实例上访问的数据。

```javascript
const privateData = new WeakMap();

class User {
  constructor(name) {
    privateData.set(this, { name });
  }

  getName() {
    return privateData.get(this).name;
  }
}
```

当User实例被垃圾回收时，私有数据也随之消失。不需要清理代码。
```

记忆化是另一个自然适用场景，尤其是当你基于对象输入而非原始值缓存结果时。如果你的昂贵计算接受一个配置对象作为输入，使用WeakMap意味着你无需担心缓存比配置存活得更久。

```javascript
const cache = new WeakMap();

function expensiveCalc(obj) {
  if (cache.has(obj)) return cache.get(obj);
  
  const result = heavyMath(obj);
  cache.set(obj, result);
  return result;
}
```

缓存的生命周期与被缓存的对象相同。一旦`obj`在其他地方不再被引用，缓存的结果和缓存条目会一起消失。

## 何时使用它

现代Web应用中的内存泄漏通常源于对应该被清理的对象的陈旧引用。如果你在构建任何长期运行的应用——一个全天保持打开的控制面板、一个运行数小时的聊天应用、一个从不刷新的管理面板——你需要考虑旧数据会怎样。

WeakMap在将数据与DOM节点、组件实例或任何你无法控制其生命周期的对象关联时特别有用。如果你基于某个引用存储数据，而该引用可能会消失，WeakMap能让清理变得简单得多。

当你构建一个具有淘汰策略的实际缓存、需要遍历条目、使用原始键，或者数据本身比它与对象的关联更重要时，普通的Map仍然是正确的选择。

`WeakMap`的好处在于，何时需要它通常是显而易见的。如果你发现自己编写清理代码来在对象销毁时移除映射条目，那是一个信号。如果你担心内存无限增长，因为不确定何时删除东西，那是另一个信号。

有时最好的特性就是那种无需你思考就能正常工作的特性。
