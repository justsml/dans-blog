أنت تعرف ذلك الإحساس عندما تغير سطرًا واحدًا من الشيفرة وتلاحظ انخفاض استهلاك الذاكرة بنسبة 50٪؟ حصل لي ذلك وأنا أراقب مراقب الأداء في أدوات مطوري Chrome بينما كان تطبيق لوحة تحكم يفرغ 100 ميغابايت في الساعة ثم يعمل بنظافة طوال بعد ظهر كامل.

التغيير بسطر واحد: `new Map()` أصبح `new WeakMap()`.

وهذا كل ما في الأمر. نفس واجهة الـ API، نفس نمط الاستخدام، سلوك مختلف تمامًا تحت الغطاء. لكن فهم سبب عمل ذلك يتطلب فهم شيء نادراً ما يفكر فيه معظم مطوري جافاسكريبت: ما الذي يحدث عندما لا أحد ينظر إلى بياناتك بعد الآن.

## عندما تتحول المراجع إلى مراسٍ

تتعامل الـ Map العادية في جافاسكريبت مع مفاتيحها كحمولة ثمينة. بمجرد وضع شيء فيها، ستمسك الـ Map به بقبضة من حديد. يرى جامع القمامة هذه العلاقة ويفكر: "من الواضح أنهم لا يزالون بحاجة إلى هذا الكائن، من الأفضل عدم لمسه."

هذا الغريزة الوقائية تصبح مشكلة عندما تخزن بيانات وصفية لأشياء مؤقتة. عقد DOM التي تُحذف. جلسات المستخدم التي تنتهي. مثيلات المكوّن التي تُفكّ. الـ Map لا تعرف أن هذه الكائنات انتهى دورها. إنها فقط تعرف أن لديها مرجعًا، لذا تبقيها حية.

```javascript
const cache = new Map();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// العنصر اختفى من DOM، لكن الـ cache يبقيه في الذاكرة
```

لا يستطيع جامع القمامة تنظيف `element` لأن `cache` لا يزال يشير إليه. يُطلق على هذا "المرجع القوي"، وفي تطبيقات الصفحة الواحدة طويلة الأمد يتحول إلى تسرب ينهار المتصفح في النهاية.

## WeakMap يغيّر القواعد

تعمل الـ WeakMap بطريقة مختلفة. تتعامل مع مفاتيحها كمواطنين مؤقتين بدلاً من سكان دائمين. عندما تخزن شيئًا في WeakMap، فأنت تقول أساسًا: "أريد ربط هذه البيانات بهذا الكائن، لكن لا أريد أن أكون السبب في بقاءه حيًا."

إذا كان الشيء الوحيد الذي يبقي كائنًا في الذاكرة هو WeakMap، يُسمح لجامع القمامة بأخذه. عندما يختفي الكائن، تختفي أيضًا مدخلة الـ WeakMap معه. لا حاجة لتنظيف يدوي.

```javascript
const cache = new WeakMap();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// The element gets Garbage Collected
// The cache entry vanishes automatically
```

قمت بتشغيل اختبار يخلق 100,000 عقدة DOM، يخزن بيانات وصفية لكل واحدة، ثم يزيلها جميعًا. باستخدام `Map`، احتفظ المتصفح بذاكرة تتراوح بين 150‑200 ميغابايت. باستخدام `WeakMap`، انخفض الاستخدام إلى 70‑80 ميغابايت. نفس الشيفرة، نفس الوظيفة، نصف البصمة الذاكرية.

## ما الذي تتنازل عنه

لـ `WeakMap` قيود قد تبدو كعقبات حتى تدرك أنها ما يجعل السحر يعمل.

**لا يمكنك التجول داخل `WeakMap`.** لا `forEach`، لا `keys()`، لا `values()`. هذا منطقي عندما تفكر في الأمر: قد يحذف جامع القمامة مدخلة أثناء حلقتك. هل تريد حقًا التعامل مع ذلك؟

لا يمكنك التحقق من الحجم. لا خاصية `.size`، لا `.length`. مرة أخرى، الهدف متحرك. قد يتغير العدد بين لحظة سؤالك ولحظة حصولك على الجواب.

**المفاتيح يجب أن تكون كائنات.** لا سلاسل، لا أرقام، لا قيم بدائية. هذا أساسي لكيفية عمل المراجع الضعيفة: القيم البدائية لا تملك هوية يمكن تتبعها بشكل منفصل عن قيمتها.

هذه ليست أخطاء. إنها التصميم. `WeakMap` مبنية لمهمة محددة: إرفاق بيانات وصفية بالكائنات دون منع تلك الكائنات من أن تُنظف. إذا كنت بحاجة إلى التجول أو مفاتيح بدائية أو عدد المدخلات، فأنت على الأرجح تحل مشكلة مختلفة ويجب أن تستخدم `Map` عاديًا.

## أين يساعد هذا فعليًا

نمط “البيانات الخاصة” كان الاستخدام الأصلي لـ `WeakMap`، قبل أن تدعم JavaScript الحقول `#private`. كانت المكتبات تنشئ `WeakMap` خارج الفئة وتستخدمه لتخزين بيانات لا ينبغي أن تكون قابلة للوصول عبر المثيل.

```javascript
const privateData = new WeakMap();

class User {
  constructor(name) {
    privateData.set(this, { name });
  }

  getName() {
    return privateData.get(this).name;
  }
}
```

عندما يُجمع مثيل `User` بواسطة جامع القمامة، تُحذف البيانات الخاصة معه. لا حاجة لكود تنظيف.

الميموزيشن هو تطبيق طبيعي آخر، خاصة عندما تقوم بتخزين نتائج بناءً على مدخلات كائنات بدلاً من القيم الأولية. إذا كانت حساباتك المكلفة تستقبل كائن تكوين كمدخل، فإن `WeakMap` يعني أنك لا تحتاج للقلق من أن الذاكرة المؤقتة تتجاوز عمر تكويناتك.

```javascript
const cache = new WeakMap();

function expensiveCalc(obj) {
  if (cache.has(obj)) return cache.get(obj);
  
  const result = heavyMath(obj);
  cache.set(obj, result);
  return result;
}
```

الذاكرة المؤقتة تبقى موجودة فقط طالما الكائنات المخزنة موجودة. بمجرد أن يصبح `obj` غير مُشار إليه في أي مكان آخر، يختفي كل من النتيجة المخزنة وإدخال الذاكرة المؤقتة معًا.

## متى نستخدمها

تسربات الذاكرة في تطبيقات الويب الحديثة عادةً تنبع من مراجع قديمة لأشياء كان يجب تنظيفها. إذا كنت تبني أي شيء يعمل لفترات طويلة—لوحة تحكم تبقى مفتوحة طوال اليوم، تطبيق دردشة يعمل لساعات، أو لوحة إدارية لا تُعيد التحميل أبداً—فإنك تحتاج إلى التفكير فيما يحدث للبيانات القديمة.

`WeakMap` مفيدة بشكل خاص عندما ترتبط بياناتك بعناصر DOM أو مثيلات مكوّنات أو أي كائن لا تتحكم في عمره. إذا كنت تخزن شيئًا بناءً على مرجع وقد يختفي ذلك المرجع، فإن `WeakMap` تجعل عملية التنظيف أبسط بكثير.

`Map` العادي يظل الخيار الصحيح عندما تبني ذاكرة مؤقتة فعلية مع سياسات إخلاء، أو عندما تحتاج إلى التجول عبر الإدخالات، أو عندما تستخدم مفاتيح أولية، أو عندما تكون البيانات نفسها هي المهمة وليس ارتباطها بكائن.

الشيء الجيد في `WeakMap` هو أنه عادةً ما يكون واضحًا متى تحتاجه. إذا وجدت نفسك تكتب كود تنظيف لإزالة إدخالات الخريطة عندما تُدمّر الكائنات، فهذا إشارة. إذا كنت قلقًا من نمو الذاكرة بلا حدود لأنك غير متأكد متى تحذف الأشياء، فهذه إشارة أخرى.

أحيانًا تكون أفضل ميزة هي تلك التي تعمل ببساطة دون أن تحتاج إلى التفكير فيها.
