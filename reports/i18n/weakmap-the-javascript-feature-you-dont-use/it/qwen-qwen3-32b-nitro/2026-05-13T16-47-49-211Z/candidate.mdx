Hai mai provato quella sensazione quando cambi una riga di codice e guardi l'uso della memoria scendere del 50%? L'ho provata io guardando la Performance Monitor di Chrome DevTools mentre un'app dashboard passava da un'emorragia di 100MB all'ora a funzionare pulita per l'intera giornata.

La modifica a una riga: `new Map()` diventato `new WeakMap()`.

Ecco. Stessa superficie API, stesso pattern di utilizzo, comportamento completamente diverso sotto i cofani. Ma capire perché funziona richiede di comprendere qualcosa che la maggior parte degli sviluppatori JavaScript non considera mai: cosa succede quando niente sta più guardando ai tuoi dati.

## Quando le Referenze Diventano Ancore

Un Map normale in JavaScript tratta le sue chiavi come merce preziosa. Una volta che metti qualcosa lì dentro, il Map ci si aggrappa con un'attenzione ferrea. Il Garbage Collector vede questa relazione e pensa: "Chiaramente gli serve ancora quell'oggetto, meglio non toccarlo."

Questo istinto protettivo diventa problematico quando stai memorizzando metadati su cose temporanee. Nodi DOM che vengono rimossi. Sessioni utente che scadono. Istanze di componenti che smontano. Il Map non sa che questi oggetti non sono più utili. Sa solo di avere una referenza, quindi li mantiene vivi.

```javascript
const cache = new Map();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// L'elemento è stato rimosso dal DOM, ma la cache lo mantiene in memoria
```

Il Garbage Collector non può pulire `element` perché `cache` continua a puntare ad esso. Questo è chiamato un "riferimento forte", e in applicazioni Single Page Apps di lunga durata, diventa una fuga che alla fine crasha il browser.

## WeakMap Cambia le Regole

Un WeakMap funziona diversamente. Tratta le sue chiavi come cittadini temporanei invece che residenti permanenti. Quando immagazzini qualcosa in un WeakMap, stai essenzialmente dicendo: "Voglio associare questi dati a questo oggetto, ma non voglio essere il motivo per cui rimane in vita".

Se l'unico elemento che mantiene un oggetto in memoria è un WeakMap, al Garbage Collector è permesso prenderlo. Quando l'oggetto scompare, l'ingresso del WeakMap scompare con esso. Non è necessaria alcuna pulizia manuale.

```javascript
const cache = new WeakMap();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// L'elemento viene Garbage Collected
// L'ingresso della cache scompare automaticamente
```

Ho eseguito un benchmark creando 100.000 nodi DOM, memorizzando i metadati di ciascuno, poi rimuovendoli tutti. Con un Map, il browser manteneva 150-200MB. Con un WeakMap, scendeva a 70-80MB. Stesso codice, stessa funzionalità, metà dell'impronta di memoria.

## Cosa Perdi

**Non puoi iterare su un WeakMap.** Nessun `forEach`, nessun `keys()`, nessun `values()`. Ti sembrerà strano, ma ha senso: il Garbage Collector potrebbe cancellare un'entrata nel mezzo del tuo ciclo. Vuoi davvero gestire questa situazione?

Non puoi controllare la dimensione. Nessuna proprietà `.size`, nessun `.length`. Anche questo è ovvio: il numero potrebbe cambiare tra il momento in cui chiedi e quando ricevi la risposta.

**I keys devono essere oggetti.** Nessun stringhe, numeri, primitivi. È fondamentale per come funzionano le referenze deboli: i valori primitivi non hanno un'identità separata dal loro valore.

Questi non sono bug. Sono la progettazione. WeakMap è stato costruito per un compito specifico: allegare metadati agli oggetti senza impedire che questi vengano puliti. Se hai bisogno di iterazione, chiavi primitive o contare le entrate, probabilmente stai risolvendo un problema diverso e dovresti usare un Map normale.

## Dove Questo Aiuta Davvero

Il "pattern dati privati" fu il primo utilizzo di WeakMap, prima che JavaScript avesse i campi `#private`. Le librerie creavano un WeakMap esterno alla classe e lo usavano per memorizzare dati non accessibili sull'istanza.

```javascript
const privateData = new WeakMap();

class User {
  constructor(name) {
    privateData.set(this, { name });
  }

  getName() {
    return privateData.get(this).name;
  }
}
```

Quando un'istanza User viene Garbage Collected, i dati privati vengono eliminati insieme ad essa. Nessun codice di pulizia necessario.

La memoizzazione è un altro caso naturale, soprattutto quando si memorizza una cache basata su input oggetto piuttosto che su valori primitivi. Se il tuo calcolo costoso richiede un oggetto di configurazione come input, un WeakMap ti permette di non preoccuparti che la cache sopravviva alle configurazioni.

```javascript
const cache = new WeakMap();

function expensiveCalc(obj) {
  if (cache.has(obj)) return cache.get(obj);
  
  const result = heavyMath(obj);
  cache.set(obj, result);
  return result;
}
```

La cache vive solo finché gli oggetti memorizzati sono riferiti. Una volta che `obj` non è più riferito da nessuna parte, sia il risultato memorizzato che l'ingresso della cache scompaiono insieme.

## Quando Sceglierlo

Le perdite di memoria nelle moderne applicazioni web derivano spesso da riferimenti obsoleti a oggetti che avrebbero dovuto essere puliti. Se stai costruendo qualcosa di a lunga durata, un dashboard aperto tutto il giorno, un'app di chat che funziona per ore, un pannello di amministrazione che non si ricarica mai, devi considerare cosa succede ai dati vecchi.

`WeakMap` è particolarmente utile quando devi associare dati a nodi DOM, istanze di componenti o qualsiasi oggetto la cui durata non controlli. Se stai memorizzando qualcosa basandoti su un riferimento e quel riferimento potrebbe sparire, `WeakMap` rende la pulizia molto più semplice.

`Map` è comunque la scelta giusta quando stai costruendo un vero cache con politiche di scadenza, quando devi iterare su voci, quando usi chiavi primitive o quando i dati stessi sono più importanti della loro associazione a un oggetto.

L'aspetto positivo di `WeakMap` è che spesso è evidente quando serve. Se ti ritrovi a scrivere codice di pulizia per rimuovere voci da una mappa quando gli oggetti vengono distrutti, è un segnale. Se ti preoccupi che la memoria cresca indefinitamente perché non sai quando cancellare le cose, è un altro segnale.

A volte la migliore funzionalità è quella che funziona senza che tu debba preoccupartene.
