Sai quella sensazione quando cambi una riga di codice e vedi il consumo di memoria scendere del 50 %? L’ho provata guardando il Performance Monitor di Chrome DevTools mentre un’app dashboard passava da un’emorragia di 100 MB all’ora a un funzionamento pulito per un’intera pomeriggio.

Il cambiamento di una riga: `new Map()` è diventato `new WeakMap()`.

Tutto qui. Stessa interfaccia API, stesso schema d’uso, comportamento completamente diverso sotto il cofano. Ma capire perché funzioni significa comprendere qualcosa su cui la maggior parte degli sviluppatori JavaScript non riflette mai: cosa succede quando nessuno guarda più i tuoi dati.

## Quando i Riferimenti Diventano Ancore

Una Map normale in JavaScript tratta le chiavi come carico prezioso. Una volta inserito qualcosa, la Map la trattiene con una presa di ferro. Il Garbage Collector vede questa relazione e pensa: “È chiaro che hanno ancora bisogno di questo oggetto, meglio non toccarlo”.

Questo istinto protettivo diventa un problema quando memorizzi metadati su elementi temporanei. Nodi DOM che vengono rimossi. Sessioni utente che scadono. Istanze di componenti che si smontano. La Map non sa che questi oggetti hanno terminato la loro utilità. Sa solo di avere un riferimento, quindi li mantiene in vita.

```javascript
const cache = new Map();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// The element is gone from the DOM, but cache is keeping it in memory
```

Il Garbage Collector non può liberare `element` perché `cache` continua a puntarlo. Questo è un “riferimento forte” e, nelle Single Page App a lunga durata, si trasforma in una perdita che alla fine blocca il browser.

## WeakMap Cambia le Regole

Una WeakMap funziona in modo diverso. Considera le sue chiavi come cittadini temporanei anziché residenti permanenti. Quando memorizzi qualcosa in una WeakMap, stai essenzialmente dicendo: “Voglio associare questi dati a questo oggetto, ma non voglio essere io il motivo per cui rimane vivo”.

Se l’unico riferimento che mantiene un oggetto in memoria è una WeakMap, il Garbage Collector è libero di eliminarlo. Quando l’oggetto scompare, anche la voce nella WeakMap scompare. Nessuna pulizia manuale è necessaria.

```javascript
const cache = new WeakMap();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// The element gets Garbage Collected
// The cache entry vanishes automatically
```

Ho eseguito un benchmark creando 100 000 nodi DOM, memorizzando metadati per ciascuno, poi rimuovendoli tutti. Con una Map, il browser tratteneva 150‑200 MB. Con una WeakMap, il consumo scendeva a 70‑80 MB. Stesso codice, stessa funzionalità, metà dell’ingombro di memoria.

## Cosa Si Rinuncia


WeakMap ha dei vincoli che sembrano limitazioni finché non ti rendi conto che sono proprio questi a far funzionare la magia.

**Non è possibile iterare su una WeakMap.** Nessun `forEach`, nessun `keys()`, nessun `values()`. Ha senso se ci pensi: il Garbage Collector potrebbe cancellare una voce nel bel mezzo del tuo ciclo. Vuoi davvero gestire una situazione del genere?

Non è possibile controllare la dimensione. Nessuna proprietà `.size`, nessun `.length`. Anche qui si tratta di un bersaglio mobile. Il numero potrebbe cambiare tra il momento in cui chiedi e quello in cui ottieni la risposta.

**Le chiavi devono essere oggetti.** Niente stringhe, niente numeri, niente primitive. Questo è fondamentale per il funzionamento dei riferimenti deboli: i valori primitivi non hanno un’identità che possa essere tracciata separatamente dal loro valore.

Questi non sono bug. Sono il design. WeakMap è costruita per un compito specifico: allegare metadati agli oggetti senza impedire che tali oggetti vengano puliti. Se hai bisogno di iterazione, chiavi primitive o di un conteggio delle voci, probabilmente stai risolvendo un problema diverso e dovresti usare una Map normale.

## Dove Questo È Davvero Utile

Il pattern dei “dati privati” era l’uso originale di WeakMap, prima che JavaScript introducesse i campi `#private`. Le librerie creavano una WeakMap al di fuori della classe e la usavano per memorizzare dati che non dovevano essere accessibili sull’istanza.

```javascript
const privateData = new WeakMap();

class User {
  constructor(name) {
    privateData.set(this, { name });
  }

  getName() {
    return privateData.get(this).name;
  }
}
```

Quando un’istanza di User viene raccolta dal garbage collector, anche i dati privati vengono eliminati. Nessun codice di pulizia necessario.

La memoizzazione è un altro caso d’uso naturale, soprattutto quando si memorizzano risultati basati su input oggetto anziché su valori primitivi. Se il tuo calcolo costoso accetta un oggetto di configurazione come input, una WeakMap ti permette di non preoccuparti che la cache sopravviva più a lungo delle configurazioni.

```javascript
const cache = new WeakMap();

function expensiveCalc(obj) {
  if (cache.has(obj)) return cache.get(obj);
  
  const result = heavyMath(obj);
  cache.set(obj, result);
  return result;
}
```

La cache vive solo finché vivono gli oggetti memorizzati. Non appena `obj` non è più referenziato da nessun’altra parte, sia il risultato memorizzato sia la voce della cache scompaiono insieme.

## Quando Usarla

Le perdite di memoria nelle moderne web app di solito derivano da riferimenti obsoleti a risorse che avrebbero dovuto essere pulite. Se stai costruendo qualcosa di a lungo termine — una dashboard che resta aperta tutto il giorno, un’app di chat che gira per ore, un pannello di amministrazione che non si ricarica mai — devi considerare cosa succede ai dati vecchi.

WeakMap è particolarmente utile quando si associano dati a nodi DOM, istanze di componenti o a qualsiasi oggetto il cui ciclo di vita non è sotto il proprio controllo. Se memorizzi qualcosa basandoti su un riferimento e quel riferimento può scomparire, WeakMap semplifica notevolmente la pulizia.

Una `Map` normale rimane la scelta corretta quando si costruisce una vera cache con politiche di eviction, quando è necessario iterare sulle voci, quando si usano chiavi primitive, o quando è il dato stesso a contare più che la sua associazione a un oggetto.

Il vantaggio di `WeakMap` è che di solito è evidente quando serve. Se ti trovi a scrivere codice di pulizia per rimuovere voci dalla mappa quando gli oggetti vengono distrutti, è un segnale. Se temi che la memoria cresca indefinitamente perché non sai quando cancellare le cose, è un altro segnale.

A volte la migliore caratteristica è proprio quella che funziona senza che tu debba pensarci.
