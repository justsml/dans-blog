Вы когда‑нибудь меняли одну строку кода и видели, как потребление памяти падает на 50 %? У меня был такой момент, когда в Chrome DevTools Performance Monitor приложение‑дашборд перестало «кровоточить» 100 МБ в час и проработало чисто целый полдня.

Однострочное изменение: `new Map()` заменилось на `new WeakMap()`.

И всё. Тот же API, тот же паттерн использования, но совершенно иное поведение под капотом. Понимать, почему это работает, значит понимать то, о чём почти ни один JavaScript‑разработчик не задумывается: что происходит, когда на ваши данные больше никто не смотрит.

## Когда ссылки становятся якорями

Обычный `Map` в JavaScript относится к своим ключам как к ценному грузу. Как только вы помещаете туда что‑то, `Map` держит это с железным захватом. Сборщик мусора видит эту связь и думает: «Очевидно, им всё ещё нужен этот объект, лучше не трогать его».

Этот защитный рефлекс становится проблемой, когда вы храните метаданные о временных вещах: удалённые узлы DOM, истёкшие пользовательские сессии, размонтированные экземпляры компонентов. `Map` не знает, что эти объекты больше не нужны. Он просто знает, что у него есть ссылка, и поэтому оставляет их живыми.

```javascript
const cache = new Map();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// The element is gone from the DOM, but cache is keeping it in memory
```

Сборщик мусора не может освободить `element`, потому что `cache` всё ещё ссылается на него. Это «сильная ссылка», и в длительно работающих одностраничных приложениях она превращается в утечку, которая в конечном итоге приводит к падению браузера.

## WeakMap меняет правила

WeakMap работает иначе. Он рассматривает свои ключи как временных граждан, а не постоянных резидентов. Когда вы помещаете что‑то в WeakMap, вы по сути говорите: «Я хочу связать эти данные с объектом, но не хочу быть причиной его удержания в памяти».

Если единственная ссылка, удерживающая объект, находится в WeakMap, сборщик мусора может его собрать. Как только объект исчезает, запись в WeakMap исчезает вместе с ним. Ручная очистка не требуется.

```javascript
const cache = new WeakMap();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// The element gets Garbage Collected
// The cache entry vanishes automatically
```

Я провёл бенчмарк: создал 100 000 DOM‑узлов, хранил метаданные для каждого, затем удалил их все. С `Map` браузер удерживал 150–200 МБ памяти. С `WeakMap` — 70–80 МБ. Тот же код, та же функциональность, вдвое меньше потребления памяти.

## Что вы теряете

WeakMap имеет ограничения, которые сначала кажутся неудобствами, пока не поймёте, что именно они позволяют магии работать.

**Нельзя итерировать WeakMap.** Нет `forEach`, нет `keys()`, нет `values()`. Это логично: сборщик мусора может удалить запись посреди вашего цикла. Вы действительно хотите иметь дело с такой ситуацией?

Нельзя проверить размер. Нет свойства `.size`, нет `.length`. Снова же, это подвижная цель. Количество может измениться между моментом запроса и получением ответа.

**Ключи должны быть объектами.** Нет строк, нет чисел, нет примитивов. Это фундаментально для работы слабых ссылок: у примитивных значений нет идентичности, которую можно отслеживать отдельно от их значения.

Это не баги. Это дизайн. WeakMap создан для одной конкретной задачи: привязывать метаданные к объектам, не препятствуя их очистке сборщиком мусора. Если вам нужна итерация, примитивные ключи или подсчёт записей, скорее всего вы решаете другую проблему и вам следует использовать обычный Map.

## Где это действительно помогает

Паттерн «приватные данные» изначально задумывался для WeakMap, задолго до появления в JavaScript полей `#private`. Библиотеки создавали WeakMap вне класса и использовали его для хранения информации, недоступной напрямую через экземпляр.

```javascript
const privateData = new WeakMap();

class User {
  constructor(name) {
    privateData.set(this, { name });
  }

  getName() {
    return privateData.get(this).name;
  }
}
```

Когда экземпляр `User` собирается сборщиком мусора, приватные данные исчезают вместе с ним. Никакой дополнительный код очистки не требуется.

Мемоизация — ещё один естественный случай применения, особенно когда кэшируются результаты, зависящие от объектных входных параметров, а не от примитивов. Если ваша дорогая вычислительная функция принимает объект‑конфигурацию, WeakMap избавит от необходимости следить, чтобы кэш не пережил жизнь этих конфигураций.

```javascript
const cache = new WeakMap();

function expensiveCalc(obj) {
  if (cache.has(obj)) return cache.get(obj);
  
  const result = heavyMath(obj);
  cache.set(obj, result);
  return result;
}
```

Кэш живёт ровно столько, сколько живут кэшируемые объекты. Как только `obj` перестаёт быть где‑либо ссылкой, и результат, и запись в кэше исчезают одновременно.

## Когда стоит использовать

Утечки памяти в современных веб‑приложениях обычно возникают из‑за устаревших ссылок на объекты, которые уже должны были быть удалены. Если вы создаёте что‑то длительно работающего: панель мониторинга, открытая весь день; чат‑приложение, работающее часами; административный интерфейс, который никогда не перезагружается — вам необходимо продумать, что происходит со старыми данными.

WeakMap особенно полезен, когда вы связываете данные с DOM‑узлами, экземплярами компонентов или любыми объектами, чьё время жизни вы не контролируете. Если вы храните что‑то, опираясь на ссылку, и эта ссылка может исчезнуть, WeakMap упрощает очистку.

Обычный Map остаётся правильным выбором, когда вы строите реальный кэш с политиками вытеснения, когда нужно итерировать записи, когда вы используете примитивные ключи или когда важны сами данные, а не их привязка к объекту.

Хорошая сторона `WeakMap` в том, что её необходимость обычно очевидна. Если вы находите себя пишущим код очистки для удаления записей из карты при уничтожении объектов, это сигнал. Если вас беспокоит, что память растёт без границ, потому что вы не уверены, когда удалять вещи, это другой сигнал.

Иногда лучшая функция — та, которая просто работает, не заставляя вас думать.
