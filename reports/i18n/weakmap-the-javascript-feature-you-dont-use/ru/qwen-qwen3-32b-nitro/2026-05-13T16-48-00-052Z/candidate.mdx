Вы знакомы с ощущением, когда меняете одну строку кода и видите, как использование памяти падает на 50%? Я испытал это, наблюдая за монитором производительности Chrome DevTools, когда приложение-доска обновлений перестало терять по 100 МБ в час и вдруг просидело целый послеобеденный период без утечек.  

Одна строка: `new Map()` превратилась в `new WeakMap()`.  

Вот и всё. Та же API-поверхность, те же паттерны использования, но абсолютно другое поведение внутри. Но понимание, почему это работает, требует понимания чего-то, о чём большинство разработчиков на JavaScript никогда не задумываются: что происходит, когда никто уже не смотрит на ваши данные.  

## Когда ссылки становятся якорями  

Обычный Map в JavaScript относится к своим ключам как к драгоценному грузу. Как только вы что-то туда положите, Map будет держать это с железным захватом. Сборщик мусора видит эту связь и думает: «Очевидно, они всё ещё нуждаются в этом объекте, лучше не трогать его».  

Этот защитный инстинкт становится проблемой, когда вы храните метаданные о временных вещах. Узлы DOM, которые удаляются. Сессии пользователей, которые истекают. Экземпляры компонентов, которые открепляются. Map не знает, что эти объекты больше не нужны. Он просто знает, что у него есть ссылка, и держит их в памяти.

```javascript
const cache = new Map();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// The element is gone from the DOM, but cache is keeping it in memory
```

Сборщик мусора не может очистить `element`, потому что `cache` всё ещё указывает на него. Это называется "сильная ссылка", и в долгоживущих одностраничных приложениях (SPA) это становится утечкой памяти, которая в конечном итоге приводит к краху браузера.

## WeakMap Меняет Правила

WeakMap работает иначе. Он рассматривает свои ключи как временных граждан, а не постоянных жителей. Когда вы храните что-то в WeakMap, вы фактически говорите: "Я хочу ассоциировать эти данные с этим объектом, но не хочу быть причиной его выживания".

Если единственной причиной, по которой объект остаётся в памяти, является WeakMap, Сборщику мусора разрешено его убрать. Когда объект исчезает, запись в WeakMap исчезает вместе с ним. Нет необходимости вручную очищать.

```javascript
const cache = new WeakMap();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// The element gets Garbage Collected
// The cache entry vanishes automatically
```

Я запустил бенчмарк, создающий 100 000 узлов DOM, хранящих метаданные о каждом из них, а затем удаляющих их все. С Map браузер держал 150-200МБ. С WeakMap объём снизился до 70-80МБ. Та же функциональность, половина объёма памяти.

## Что Вы Теряете

WeakMap имеет ограничения, которые сначала кажутся недостатками, пока вы не осознаете, что именно они делают его возможным.

**Вы не можете итерировать WeakMap.** Нет `forEach`, нет `keys()`, нет `values()`. Это логично, если подумать: сборщик мусора может удалить запись в середине вашего цикла. Вы действительно хотите с этим справляться?

Вы не можете проверить размер. Нет свойства `.size`, нет `.length`. Опять же, это подвижная цель. Количество может измениться между моментом, когда вы задаёте вопрос, и получением ответа.

**Ключами могут быть только объекты.** Нет строк, нет чисел, нет примитивов. Это фундаментально для работы слабых ссылок: примитивные значения не имеют идентичности, которую можно отслеживать отдельно от их значения.

Это не ошибки. Это дизайн. WeakMap создан для одной конкретной задачи: привязки метаданных к объектам без блокировки их утилизации. Если вам нужны итерация или примитивные ключи или подсчёт записей, вы, вероятно, решаете другую задачу и должны использовать обычный Map.

## Где Это На Самом Деле Помогает

Паттерн "приватных данных" был исходным применением WeakMap, задолго до появления в JavaScript приватных полей `#private`. Библиотеки создавали WeakMap вне класса и использовали его для хранения данных, к которым не должно быть доступа из экземпляра.

```javascript
const privateData = new WeakMap();

class User {
  constructor(name) {
    privateData.set(this, { name });
  }

  getName() {
    return privateData.get(this).name;
  }
}
```

Когда экземпляр User подвергается сборке мусора, приватные данные исчезают вместе с ним. Нет необходимости в коде очистки.

Мемоизация — ещё один естественный сценарий, особенно если кэшируете результаты на основе объектных входных данных, а не примитивных значений. Если ваша дорогостоящая операция принимает объект конфигурации, WeakMap гарантирует, что кэш не будет существовать дольше, чем сами конфигурации.

```javascript
const cache = new WeakMap();

function expensiveCalc(obj) {
  if (cache.has(obj)) return cache.get(obj);
  
  const result = heavyMath(obj);
  cache.set(obj, result);
  return result;
}
```

Кэш существует только до тех пор, пока существуют объекты, на которые он ссылается. Как только `obj` больше не упоминается где-либо, оба — результат кэша и запись в кэше — исчезают одновременно.

## Когда стоит использовать

Утечки памяти в современных веб-приложениях обычно происходят из-за устаревших ссылок на объекты, которые должны были быть очищены. Если вы разрабатываете приложения с длительным циклом жизни — панель управления, открытую целый день, чат-приложение, работающее часы, или административную панель, которая никогда не перезагружается — вам нужно задуматься о судьбе старых данных.

`WeakMap` особенно полезен, когда вы связываете данные с DOM-узлами, экземплярами компонентов или любым объектом, чей жизненный цикл вы не контролируете. Если вы храните что-то на основе ссылки, и эта ссылка может исчезнуть, `WeakMap` значительно упрощает очистку.

Обычный `Map` всё ещё правильный выбор, когда вы строите настоящий кэш с политиками вытеснения, когда вам нужно итерироваться по записям, когда вы используете примитивные ключи или когда важно само значение данных, а не их связь с объектом.

Хорошо в `WeakMap` то, что обычно очевидно, когда он нужен. Если вы пишете код для очистки, чтобы удалять записи из карты при уничтожении объектов — это признак. Если вы беспокоитесь о неограниченном росте памяти, не зная, когда удалять данные — это ещё один признак.

Иногда лучшая черта — это то, что она просто работает, не требуя от вас думать об этом.
