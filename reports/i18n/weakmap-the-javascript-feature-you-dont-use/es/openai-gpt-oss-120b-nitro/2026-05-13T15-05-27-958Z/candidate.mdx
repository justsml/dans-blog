Sabes esa sensación cuando cambias una línea de código y ves que tu uso de memoria se reduce un 50 %? Yo la tuve al observar el Monitor de Rendimiento de Chrome DevTools mientras una aplicación de panel pasaba de perder 100 MB por hora a ejecutarse limpia durante toda una tarde.

El cambio de una sola línea: `new Map()` pasó a `new WeakMap()`.

Eso es todo. Mismo API, mismo patrón de uso, comportamiento completamente distinto bajo el capó. Pero entender por qué funciona implica comprender algo que la mayoría de los desarrolladores de JavaScript nunca consideran: qué ocurre cuando ya nadie está mirando tus datos.

## Cuando las referencias se convierten en anclas

Un `Map` regular en JavaScript trata sus claves como carga preciosa. Una vez que colocas algo allí, el `Map` lo retendrá con un agarre de hierro. El recolector de basura ve esa relación y piensa: “Claramente todavía necesitan este objeto, mejor no tocarlo”.

Este instinto protector se vuelve un problema cuando almacenas metadatos sobre cosas temporales. Nodos del DOM que se eliminan. Sesiones de usuario que expiran. Instancias de componentes que se desmontan. El `Map` no sabe que esos objetos ya no son útiles. Sólo sabe que tiene una referencia, así que los mantiene vivos.

```javascript
const cache = new Map();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// The element is gone from the DOM, but cache is keeping it in memory
```

El recolector de basura no puede liberar `element` porque `cache` sigue apuntando a él. Eso es una *referencia fuerte* y, en aplicaciones de una sola página de larga duración, se convierte en una fuga que eventualmente bloquea el navegador.

## WeakMap cambia las reglas

Un WeakMap funciona de forma distinta. Trata sus claves como ciudadanos temporales en lugar de residentes permanentes. Cuando almacenas algo en un WeakMap, esencialmente dices: “Quiero asociar estos datos con este objeto, pero no quiero ser la razón por la que siga vivo”.

Si lo único que mantiene un objeto en memoria es un WeakMap, el recolector de basura puede liberarlo. Cuando el objeto desaparece, la entrada del WeakMap desaparece con él. No se necesita limpieza manual.

```javascript
const cache = new WeakMap();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// The element gets Garbage Collected
// The cache entry vanishes automatically
```

Realicé una prueba creando 100 000 nodos del DOM, almacenando metadatos para cada uno y luego eliminándolos todos. Con un Map, el navegador retenía entre 150 y 200 MB. Con un WeakMap, bajó a 70‑80 MB. Mismo código, misma funcionalidad, la mitad del consumo de memoria.

## Lo que pierdes

WeakMap tiene restricciones que parecen limitaciones hasta que te das cuenta de que son lo que hace que la magia funcione.

**No puedes iterar sobre un WeakMap.** No hay `forEach`, ni `keys()`, ni `values()`. Tiene sentido si lo piensas: el recolector de basura podría eliminar una entrada en medio de tu bucle. ¿Realmente quieres lidiar con eso?

No puedes consultar el tamaño. No hay propiedad `.size`, ni `.length`. De nuevo, es un objetivo móvil. El número podría cambiar entre el momento en que lo preguntas y el momento en que recibes la respuesta.

**Las claves deben ser objetos.** No se permiten strings, números ni primitivos. Esto es fundamental para el funcionamiento de las referencias débiles: los valores primitivos no tienen una identidad que pueda rastrearse por separado de su valor.

Estos no son errores. Son el diseño. WeakMap está construido para una tarea específica: adjuntar metadatos a objetos sin impedir que esos objetos sean limpiados. Si necesitas iteración, claves primitivas o un recuento de entradas, probablemente estés resolviendo un problema diferente y deberías usar un Map regular.

## Donde esto realmente ayuda

El patrón de “datos privados” fue el caso de uso original de WeakMap, antes de que JavaScript tuviera campos `#private`. Las bibliotecas creaban un WeakMap fuera de la clase y lo usaban para almacenar datos que no debían ser accesibles desde la instancia.

```javascript
const privateData = new WeakMap();

class User {
  constructor(name) {
    privateData.set(this, { name });
  }

  getName() {
    return privateData.get(this).name;
  }
}
```

Cuando una instancia de User es recolectada por el Garbage Collector, los datos privados desaparecen con ella. No se necesita código de limpieza.

La memoización encaja de forma natural, sobre todo cuando se cachean resultados basados en objetos de entrada en lugar de valores primitivos. Si tu cálculo costoso recibe un objeto de configuración como parámetro, un WeakMap evita que la caché sobreviva más allá de la vida de esas configuraciones.

```javascript
const cache = new WeakMap();

function expensiveCalc(obj) {
  if (cache.has(obj)) return cache.get(obj);
  
  const result = heavyMath(obj);
  cache.set(obj, result);
  return result;
}
```

La caché solo vive mientras existan los objetos que se están cacheando. Cuando `obj` deja de estar referenciado en cualquier otro sitio, tanto el resultado cacheado como la entrada de la caché desaparecen simultáneamente.

## Cuándo usarlo

Las fugas de memoria en aplicaciones web modernas suelen originarse en referencias obsoletas a elementos que deberían haberse limpiado. Si construyes algo de larga duración —un panel de control que permanece abierto todo el día, una aplicación de chat que funciona durante horas, un panel de administración que nunca se recarga— debes considerar qué ocurre con los datos antiguos.

WeakMap es particularmente útil cuando asocias datos con nodos del DOM, instancias de componentes o cualquier objeto cuyo ciclo de vida no controlas. Si almacenas algo basado en una referencia y esa referencia puede desaparecer, WeakMap simplifica enormemente la limpieza.

Map regular sigue siendo la elección correcta cuando construyes una caché real con políticas de expulsión, cuando necesitas iterar sobre las entradas, cuando usas claves primitivas, o cuando lo que importa es el propio dato más que su asociación con un objeto.

Lo bueno de `WeakMap` es que suele ser evidente cuándo lo necesitas. Si te encuentras escribiendo código de limpieza para eliminar entradas del mapa cuando los objetos se destruyen, eso es una señal. Si te preocupa que la memoria crezca sin límite porque no sabes cuándo borrar cosas, esa es otra señal.

A veces la mejor característica es la que simplemente funciona sin que tengas que pensar en ella.
