¿Conoces esa sensación cuando cambias una línea de código y ves que tu uso de memoria disminuye un 50%? Tuve ese momento observando el monitor de rendimiento de las herramientas de desarrollo de Chrome mientras una aplicación de dashboard pasó de perder 100MB por hora a funcionar sin problemas durante toda la tarde.

El cambio de una línea: `new Map()` se convirtió en `new WeakMap()`.

Eso es todo. La misma superficie de API, el mismo patrón de uso, un comportamiento completamente diferente bajo el capó. Pero entender por qué esto funciona significa comprender algo que la mayoría de los desarrolladores de JavaScript nunca consideran: ¿qué sucede cuando nadie más está observando tus datos?

## Cuando las referencias se convierten en anclas

Un Map normal en JavaScript trata sus claves como si fueran objetos de gran valor. Una vez que introduces algo allí, el Map se aferrará a ello con una fuerza inquebrantable. El recolector de basura ve esta relación y piensa: "Claramente aún necesitan este objeto, mejor no tocarlo".

Este instinto protector se convierte en un problema cuando almacenas metadatos sobre cosas temporales. Nodos DOM que se eliminan. Sesiones de usuario que expiran. Instancias de componentes que se desmontan. El Map no sabe que estos objetos ya no son útiles. Solo sabe que tiene una referencia, así que los mantiene vivos.

```javascript
const cache = new Map();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// El elemento ya no está en el DOM, pero el cache lo mantiene en memoria
```

El recolector de basura no puede limpiar `element` porque `cache` aún apunta a él. Esto se llama una "referencia fuerte", y en aplicaciones de una sola página (SPAs) de larga duración, se convierte en una fuga que eventualmente crashea el navegador.

## WeakMap Cambia las Reglas

Un WeakMap funciona de manera diferente. Trata sus claves como ciudadanos temporales en lugar de residentes permanentes. Cuando almacenas algo en un WeakMap, estás diciendo esencialmente: "Quiero asociar estos datos con este objeto, pero no quiero que sea la razón por la que permanece en memoria".

Si lo único que mantiene un objeto en memoria es un WeakMap, al recolector de basura se le permite recogerlo. Cuando el objeto desaparece, la entrada del WeakMap desaparece con él. No se requiere limpieza manual.

```javascript
const cache = new WeakMap();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// El elemento es recolectado por el recolector de basura
// La entrada del cache desaparece automáticamente
```

Ejecuté una prueba creando 100,000 nodos DOM, almacenando metadatos sobre cada uno y luego eliminándolos todos. Con un Map, el navegador se quedaba con 150-200MB. Con un WeakMap, bajaba a 70-80MB. Mismo código, misma funcionalidad, la mitad del uso de memoria.

## Lo que Sacrificas

`WeakMap` tiene restricciones que parecen limitaciones hasta que te das cuenta de que son precisamente lo que hace posible la magia.

**No puedes iterar sobre un `WeakMap`.** No hay `forEach`, ni `keys()`, ni `values()`. Esto tiene sentido si lo piensas: el Recolector de Basura podría eliminar una entrada en medio de tu bucle. ¿Realmente quieres lidiar con eso?

No puedes comprobar el tamaño. No hay propiedad `.size`, ni `.length`. Otra vez, es un objetivo móvil. La cantidad podría cambiar entre el momento en que le preguntas y el que recibes la respuesta.

**Las claves deben ser objetos.** No cadenas, no números, no primitivos. Esto es fundamental para cómo funcionan las referencias débiles: los valores primitivos no tienen identidad que pueda ser rastreada de forma independiente a su valor.

Estas no son fallos. Son el diseño. `WeakMap` está construido para un trabajo específico: adjuntar metadatos a objetos sin impedir que esos objetos sean limpiados. Si necesitas iteración, claves primitivas o contar entradas, probablemente estés resolviendo un problema diferente y deberías usar un `Map` normal.

## Dónde Esto Realmente Ayuda

El patrón de "datos privados" fue el caso de uso original de `WeakMap`, antes de que JavaScript tuviera campos privados `#`. Las bibliotecas crearían un `WeakMap` fuera de la clase y lo usarían para almacenar datos que no deberían ser accesibles en la instancia.

```javascript
const privateData = new WeakMap();

class User {
  constructor(name) {
    privateData.set(this, { name });
  }

  getName() {
    return privateData.get(this).name;
  }
}
```

Cuando una instancia de `User` es recolectada por el Garbage Collector, los datos privados también se van. No se necesita código de limpieza.

La memoización es otro ajuste natural, especialmente cuando estás cachando resultados basados en entradas de objetos en lugar de valores primitivos. Si tu cálculo costoso toma un objeto de configuración como entrada, un `WeakMap` significa que no tienes que preocuparte por que la caché viva más que las configuraciones.

```javascript
const cache = new WeakMap();

function expensiveCalc(obj) {
  if (cache.has(obj)) return cache.get(obj);
  
  const result = heavyMath(obj);
  cache.set(obj, result);
  return result;
}
```

La caché vive solo mientras los objetos cacheados estén vivos. Una vez que `obj` no es referenciado en ningún otro lugar, tanto el resultado cachéado como la entrada de la caché desaparecen juntos.

## Cuándo Usarlo

Las fugas de memoria en aplicaciones web modernas suelen provenir de referencias obsoletas a cosas que deberían haber sido limpiadas. Si estás construyendo algo que se ejecuta durante mucho tiempo, un panel de control que permanece abierto todo el día, una aplicación de chat que corre por horas, un panel de administración que nunca se refresca, debes pensar en qué ocurre con los datos antiguos.

`WeakMap` es especialmente útil cuando estás asociando datos con nodos del DOM, instancias de componentes o cualquier objeto cuya duración no controlas. Si estás almacenando algo basado en una referencia y esa referencia podría desaparecer, `WeakMap` hace que la limpieza sea mucho más sencilla.

`Map` regular sigue siendo la opción correcta cuando estás construyendo un caché real con políticas de expulsión, cuando necesitas iterar sobre las entradas, cuando usas claves primitivas o cuando los datos en sí mismos son lo que importa, no su asociación con un objeto.

Lo bueno de `WeakMap` es que suele ser evidente cuándo lo necesitas. Si te encuentras escribiendo código de limpieza para eliminar entradas del mapa cuando los objetos se destruyen, esa es una señal. Si te preocupas por que la memoria crezca sin límite porque no estás seguro de cuándo eliminar cosas, esa es otra señal.

A veces la mejor característica es una que simplemente funciona sin que tengas que pensarlo.
