אתה יודעאת התחושה כשאתה משנה שורה אחת בקוד ורואה את השימוש בזיכרון שלך יורד ב‑50%? היה לי הרגע הזה כשצפיתי במוניטור הביצועים של Chrome DevTools בזמן שאפליקציית לוח מחוונים הייתה עוברת מדימום של 100 מ״ב לשעה להרצה נקייה של כל אחר הצהריים.

השינוי בקו אחד: `new Map()` הפך ל‑`new WeakMap()`.

זה הכל. ממשק API זהה, דפוס שימוש זהה, אבל התנהגות שונה לחלוטין מתחת למכסה. אבל להבין למה זה עובד דורש להבין משהו שרוב מפתחי JavaScript אף פעם לא חושבים עליו: מה קורה כשאף אחד כבר לא מסתכל על הנתונים שלך.

## כשקישורים הופכים לעוגנים

מפה רגילה ב‑JavaScript מתייחסת למפתחות שלה כמו למטען יקר ערך. ברגע שאתה שם משהו בתוכה, המפה תחזיק בו עם אחיזה ברזל. אספן הזבל רואה את הקשר הזה וחושב, “ברור שהם עדיין צריכים את האובייקט הזה, עדיף שלא לגעת בו.”

הגנה זו הופכת לבעיה כשאתה מאחסן מטא‑נתונים על דברים זמניים. צמתים של DOM שמוסרים. סשנים של משתמשים שפג תוקפם. מופעי רכיבים שמתפרקים. המפה לא יודעת שהאובייקטים האלה כבר לא שימושיים. היא רק יודעת שיש לה הפניה, ולכן משאירה אותם בחיים.

```javascript
const cache = new Map();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// האלמנט נעלם מה‑DOM, אבל ה‑cache משאיר אותו בזיכרון
```

אספן הזבל לא יכול לנקות את `element` מכיוון שה‑`cache` עדיין מצביע עליו. זה נקרא “הפניה חזקה”, ובאפליקציות SPA ארוכות‑טווח זה הופך לדליפת זיכרון שמסתיימת בקריסת הדפדפן.

## WeakMap משנה את הכללים

WeakMap פועל אחרת. הוא מתייחס למפתחות שלו כאזרחים זמניים ולא כתושבים קבועים. כשאתה מאחסן משהו ב‑WeakMap, אתה בעצם אומר: “אני רוצה לשייך את הנתונים האלה לאובייקט הזה, אבל אני לא רוצה להיות הסיבה שהוא נשאר בחיים.”

אם הדבר היחיד שמחזיק אובייקט בזיכרון הוא WeakMap, אספן הזבל מורשה לקחת אותו. כאשר האובייקט נעלם, רשומת ה‑WeakMap נעלמת יחד איתו. אין צורך בניקוי ידני.

```javascript
const cache= new WeakMap();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// האלמנט נלקח על‑ידי אספן הזבל
// רשומת ה‑WeakMap נעלמת אוטומטית
```

הרצתי מדד שבו יצרתי 100,000 צמתים ב‑DOM, שמרתי מטא‑נתונים על כל אחד, ואז הסרתי את כולם. עם `Map`, הדפדפן החזיק בזיכרון 150‑200 מ״ב. עם `WeakMap`, הצריכה ירדה ל‑70‑80 מ״ב. אותו קוד, אותה פונקציונליות, חצי מהזיכרון.

## מה אתה מוותר

ל‑WeakMap יש מגבלות שמרגישות כמו הגבלות עד שמבינים שהן מה שמאפשרות את הקסם.

**לא ניתן לעבור על WeakMap.** אין `forEach`, אין `keys()`, אין `values()`. זה הגיוני כשחושבים על זה: אספן הזבל עשוי למחוק רשומה באמצע הלולאה שלך. האם באמת תרצה להתמודד עם זה?

לא ניתן לבדוק את הגודל. אין מאפיין `.size`, אין `.length`. שוב, זה יעד בתנועה. המספר יכול להשתנות בין הרגע שבו אתה שואל ובין הרגע שבו אתה מקבל תשובה.

**המפתחות חייבים להיות אובייקטים.** אין מחרוזות, אין מספרים, אין ערכים פרימיטיביים. זהו יסוד של איך עובדות הפניות החלשות: ערכים פרימיטיביים אינם בעלי זהות שניתן לעקוב אחריה בנפרד מהערך שלהם.

אלו לא באגים. הן העיצוב. WeakMap נבנה למטרה ספציפית אחת: לצרף מטא‑נתונים לאובייקטים מבלי למנוע מהם להימחק. אם אתה צריך איטרציה, מפתחות פרימיטיביים או ספירת רשומות, כנראה שאתה פותר בעיה אחרת ויש להשתמש ב‑Map רגיל.

## איפה זה באמת עוזר

תבנית ה‑“נתונים פרטיים” הייתה השימוש המקורי של WeakMap, לפני ש‑JavaScript קיבל שדות `#private`. ספריות היו יוצרות WeakMap מחוץ למחלקה ומשתמשות בו לאחסן נתונים שלא אמורים להיות נגישים על‑ידי המופע.

```javascript
const privateData = new WeakMap();

class User {
  constructor(name) {
    privateData.set(this, { name });
  }

  getName() {
    return privateData.get(this).name;
  }
}
```

כאשר מופע של `User` נלקח על‑ידי אספן הזבל, הנתונים הפרטיים נעלמים יחד איתו. אין צורך בקוד ניקוי.

הזיכרון הממוזער הוא התאמה טבעית נוספת, במיוחד כשאתה מאחסן תוצאות במטמון על בסיס קלט של אובייקטים ולא ערכים פרימיטיביים. אם החישוב היקר שלך מקבל אובייקט תצורה כקלט, WeakMap משחרר אותך מדאגה שהמטמון ימשיך להתקיים אחרי שהקונפיגים ייעלמו.

```javascript
const cache = new WeakMap();

function expensiveCalc(obj) {
  if (cache.has(obj)) return cache.get(obj);
  
  const result = heavyMath(obj);
  cache.set(obj, result);
  return result;
}
```

המטמון קיים רק כל עוד האובייקטים הממוזערים קיימים. ברגע שה‑`obj` אינו מצוין במקומות אחרים, הן התוצאה הממוזערת והן רשומת המטמון נעלמים יחד.

## מתי להשתמש בו

דליפות זיכרון באפליקציות ווב מודרניות נובעות בדרך כלל מהפניות ישנות לדברים שאמורים היה להימחק. אם אתה בונה משהו שמריץ זמן ארוך — לוח מחוונים שנשאר פתוח כל היום, אפליקציית צ׳אט שפועלת שעות, פאנל ניהול שלעולם לא מתרענן — עליך לחשוב מה קורה לנתונים הישנים.

WeakMap שימושי במיוחד כשאתה משייך נתונים לצמתים של DOM, למופעי רכיבים, או לכל אובייקט שהחיים שלו אינם בשליטתך. אם אתה שומר משהו על בסיס הפנייה והפנייה זו עשויה להיעלם, WeakMap מפשט משמעותית את הניקוי.

Map רגיל עדיין הבחירה הנכונה כשאתה בונה מטמון אמיתי עם מדיניות פינוי, כשאתה צריך לעבור על הרשומות, כשאתה משתמש במפתחות פרימיטיביים, או כשהנתונים עצמם הם החשובים ולא הקשר שלהם לאובייקט.

הדבר הנחמד ב‑`WeakMap` הוא שלרוב הוא ברור מתי הוא נדרש. אם אתה מוצא את עצמך כותב קוד ניקוי כדי להסיר רשומות ממפה כשאובייקטים מתים, זה סימן. אם אתה מודאג שהזיכרון יגדל ללא גבול כי אינך בטוח מתי למחוק דברים, זה סימן נוסף.

לפעמים הפיצ'ר הטוב ביותר הוא זה שפשוט עובד מבלי שתצטרך לחשוב עליו.
