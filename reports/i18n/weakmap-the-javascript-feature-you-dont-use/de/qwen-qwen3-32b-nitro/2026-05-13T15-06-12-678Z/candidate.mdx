Kennen Sie das Gefühl, wenn Sie eine Zeile Code ändern und Ihre Speichernutzung um 50 % sinkt? Ich hatte diesen Moment, als ich in den Chrome DevTools Performance Monitor sah, wie eine Dashboard-App von 100 MB pro Stunde, die sie verlor, plötzlich sauber für den ganzen Nachmittag lief.

Die Änderung in einer Zeile: `new Map()` wurde zu `new WeakMap()`.

Das war's. Selbe API-Oberfläche, gleicher Verwendungsmuster, völlig unterschiedenes Verhalten im Inneren. Aber um zu verstehen, warum das funktioniert, muss man etwas verstehen, das die meisten JavaScript-Entwickler nie in Betracht ziehen: Was passiert, wenn niemand mehr auf Ihre Daten schaut.

## Wenn Referenzen zu Ankerpunkten werden

Ein regulärer Map in JavaScript behandelt seine Schlüssel wie wertvolles Gut. Sobald Sie etwas hineinstecken, hält der Map es mit einer Eisenfaust fest. Der Garbage Collector sieht diese Beziehung und denkt: „Klar, dass sie dieses Objekt noch benötigen, besser nicht daran herumfummeln.“

Dieser Schutzinstinkt wird zum Problem, wenn Sie Metadaten zu temporären Dingen speichern. DOM-Knoten, die entfernt werden. Benutzersitzungen, die ablaufen. Komponenteninstanzen, die abgemountet werden. Der Map weiß nicht, dass diese Objekte nicht mehr nützlich sind. Er weiß nur, dass er eine Referenz hat, also hält er sie am Leben.

```javascript
const cache = new Map();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// Das Element ist vom DOM weg, aber der cache hält es weiterhin im Speicher
```

Der Garbage Collector kann `element` nicht bereinigen, weil `cache` immer noch darauf zeigt. Dies wird als "starker Verweis" bezeichnet, und in lang laufenden Single Page Apps wird daraus ein Leck, das schließlich den Browser abstürzen lässt.

## WeakMap ändert die Regeln

Ein WeakMap funktioniert anders. Es betrachtet seine Schlüssel als temporäre Bürger statt als ständige Einwohner. Wenn Sie etwas in einem WeakMap speichern, sagen Sie im Grunde: "Ich möchte diese Daten mit diesem Objekt verknüpfen, aber ich möchte nicht der Grund sein, warum es am Leben bleibt."

Wenn das einzige, was ein Objekt im Speicher hält, ein WeakMap ist, darf der Garbage Collector es entfernen. Wenn das Objekt verschwindet, verschwindet auch der WeakMap-Eintrag. Keine manuelle Bereinigung nötig.

```javascript
const cache = new WeakMap();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// Das Element wird vom Garbage Collector bereinigt
// Der Cache-Eintrag verschwindet automatisch
```

Ich habe einen Benchmark durchgeführt, bei dem 100.000 DOM-Elemente erstellt wurden, Metadaten zu jedem gespeichert und anschließend alle entfernt wurden. Mit einem Map hielt der Browser 150-200MB fest. Mit einem WeakMap sank der Speicherbedarf auf 70-80MB. Selber Code, selbe Funktionalität, aber nur die Hälfte des Speicherbedarfs.

## Was Sie opfern

**Was Sie opfern**  
WeakMap hat Einschränkungen, die wie Begrenzungen wirken, bis man erkennt, dass sie genau das sind, was die Magie ermöglicht.  

**Sie können eine WeakMap nicht iterieren.** Kein `forEach`, kein `keys()`, kein `values()`. Das ergibt Sinn, wenn man darüber nachdenkt: Der Garbage Collector könnte während Ihres Loops einen Eintrag löschen. Wollen Sie das wirklich bewältigen?  

Sie können die Größe nicht prüfen. Kein `.size`-Attribut, kein `.length`. Wieder: Es ist ein bewegliches Ziel. Die Anzahl könnte sich zwischen der Anfrage und der Antwort ändern.  

**Schlüssel müssen Objekte sein.** Keine Strings, keine Zahlen, keine Primitiven. Das ist grundlegend für die Funktionsweise von schwachen Referenzen: Primitive Werte haben keine Identität, die unabhängig vom Wert verfolgt werden kann.  

Das sind keine Fehler. Das ist die Designentscheidung. WeakMap ist für eine spezifische Aufgabe gebaut: Metadaten an Objekte anzuhängen, ohne dass diese Objekte nicht gelöscht werden können. Wenn Sie Iteration oder primitive Schlüssel oder eine Eintragsanzahl benötigen, lösen Sie wahrscheinlich ein anderes Problem und sollten eine reguläre Map verwenden.  

## Wo dies tatsächlich hilft

Das „private Daten“-Muster war ursprünglicher Anwendungsfall für WeakMap, noch bevor JavaScript `#private`-Felder hatte. Bibliotheken erstellten außerhalb der Klasse eine WeakMap und verwendeten sie, um Daten zu speichern, die nicht über die Instanz zugänglich sein sollten.  

```javascript
const privateData = new WeakMap();

class User {
  constructor(name) {
    privateData.set(this, { name });
  }

  getName() {
    return privateData.get(this).name;
  }
}
```

Wenn eine User-Instanz vom Garbage Collector bereinigt wird, verschwindet auch die private Daten. Kein Bereinigungscode nötig.  

Memoisierung ist ein weiterer natürlicher Anwendungsfall, besonders wenn Sie Ergebnisse basierend auf Objekt-Eingaben zwischenspeichern, anstatt primitive Werte. Wenn Ihre kostspielige Berechnung ein Konfigurationsobjekt als Eingabe verwendet, bedeutet WeakMap, dass Sie sich um die Zwischenspeicherung nicht länger als die Konfigurationen sorgen müssen.  

```javascript
const cache = new WeakMap();

function expensiveCalc(obj) {
  if (cache.has(obj)) return cache.get(obj);
  
  const result = heavyMath(obj);
  cache.set(obj, result);
  return result;
}
```

Der Cache existiert nur so lange wie die Objekte, die zwischengespeichert werden. Sobald `obj` an keiner anderen Stelle referenziert wird, verschwinden sowohl das zwischengespeicherte Ergebnis als auch der Cache-Eintrag gemeinsam.  

## Wann Sie danach greifen sollten  

Speicherlecks in modernen Webanwendungen entstehen üblicherweise aus veralteten Verweisen auf Dinge, die bereinigt werden sollten. Wenn Sie etwas Langfristiges bauen – ein Dashboard, das den ganzen Tag geöffnet bleibt, eine Chat-App, die Stunden läuft, eine Admin-Oberfläche, die niemals neu geladen wird –, müssen Sie darüber nachdenken, was mit alten Daten passiert.

`WeakMap` ist besonders nützlich, wenn Sie Daten mit DOM-Elementen, Komponenten-Instanzen oder jedem Objekt verknüpfen, dessen Lebensdauer Sie nicht kontrollieren. Wenn Sie etwas basierend auf einem Verweis speichern und dieser Verweis wegfallen könnte, vereinfacht `WeakMap` die Bereinigung erheblich.  

Reguläre `Map` ist immer noch die richtige Wahl, wenn Sie einen echten Cache mit Entfernungspolitiken bauen, wenn Sie Einträge durchlaufen müssen, wenn Sie primitive Schlüssel verwenden oder wenn die Daten selbst im Vordergrund stehen, nicht ihre Verknüpfung mit einem Objekt.  

Das Schöne an `WeakMap` ist, dass es meist offensichtlich ist, wann Sie es benötigen. Wenn Sie sich selbst Bereinigungscode schreiben, um `Map`-Einträge zu entfernen, wenn Objekte zerstört werden, ist das ein Hinweis. Wenn Sie Sorge um unbeschränktes Wachstum des Speichers haben, weil Sie unsicher sind, wann Sie Dinge löschen sollen, ist das ein weiterer Hinweis.  

Manchmal ist das beste Feature das, das einfach funktioniert, ohne dass Sie darüber nachdenken müssen.
