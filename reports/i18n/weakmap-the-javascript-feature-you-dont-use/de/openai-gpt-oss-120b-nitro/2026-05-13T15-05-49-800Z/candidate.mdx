Sie kennen das Gefühl, wenn Sie eine Zeile Code ändern und Ihr Speicherverbrauch um 50 % sinkt? Ich hatte diesen Moment, als ich Chrome DevTools Performance Monitor beobachtete, wie eine Dashboard‑App von einem Blutverlust von 100 MB pro Stunde zu einem sauberen Lauf über einen ganzen Nachmittag wechselte.

Die Ein‑Zeilen‑Änderung: `new Map()` wurde zu `new WeakMap()`.

Das war alles. Gleiche API‑Oberfläche, gleiches Nutzungsmuster, völlig anderes Verhalten im Inneren. Aber zu verstehen, warum das funktioniert, bedeutet, etwas zu begreifen, über das die meisten JavaScript‑Entwickler nie nachdenken: was passiert, wenn niemand mehr auf Ihre Daten schaut.

## Wenn Referenzen zu Ankern werden

Eine reguläre Map in JavaScript behandelt ihre Schlüssel wie wertvolle Fracht. Sobald Sie etwas dort hineinlegen, hält die Map es mit eisernem Griff fest. Der Garbage Collector sieht diese Beziehung und denkt: „Offensichtlich wird dieses Objekt noch benötigt, besser nicht anfassen.“

Dieser Schutzinstinkt wird zum Problem, wenn Sie Metadaten über temporäre Dinge speichern. DOM‑Knoten, die entfernt werden. Benutzersitzungen, die ablaufen. Komponenteninstanzen, die unmounten. Die Map weiß nicht, dass diese Objekte nicht mehr nützlich sind. Sie weiß nur, dass sie eine Referenz hat, also hält sie sie am Leben.

```javascript
const cache = new Map();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// The element is gone from the DOM, but cache is keeping it in memory
```

Der Garbage Collector kann `element` nicht freigeben, weil `cache` noch darauf verweist. Das ist eine **starke Referenz**; in langlaufenden Single‑Page‑Apps wird daraus ein Leak, das schließlich den Browser zum Absturz bringt.

## WeakMap ändert die Regeln

Eine WeakMap funktioniert anders. Sie behandelt ihre Schlüssel wie temporäre Bürger statt als permanente Bewohner. Wenn Sie etwas in einer WeakMap speichern, sagen Sie im Wesentlichen: „Ich möchte diese Daten mit diesem Objekt verknüpfen, aber ich will nicht der Grund sein, warum es am Leben bleibt.“

Ist die WeakMap die einzige Stelle, an der ein Objekt im Speicher gehalten wird, darf der Garbage Collector es entfernen. Sobald das Objekt verschwindet, verschwindet auch der WeakMap‑Eintrag. Keine manuelle Aufräumaktion nötig.

```javascript
const cache = new WeakMap();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// The element gets Garbage Collected
// The cache entry vanishes automatically
```

Ich habe einen Benchmark durchgeführt, bei dem 100 000 DOM‑Knoten erzeugt, Metadaten für jeden gespeichert und anschließend alle entfernt wurden. Mit einer Map hielt der Browser 150‑200 MB im Speicher. Mit einer WeakMap sank das auf 70‑80 MB. Same code, same functionality, half the memory footprint.

## Was Sie aufgeben

WeakMap hat Einschränkungen, die zunächst wie Beschränkungen wirken, bis man erkennt, dass genau diese das „Magische“ ermöglichen.

**Man kann nicht über eine WeakMap iterieren.** Kein `forEach`, keine `keys()`, keine `values()`. Das ist logisch, wenn man darüber nachdenkt: Der Garbage Collector könnte mitten in Ihrer Schleife einen Eintrag löschen. Will man das wirklich handhaben?

Man kann die Größe nicht abfragen. Keine `.size`‑Eigenschaft, kein `.length`. Auch das ist ein bewegliches Ziel. Die Anzahl kann sich zwischen der Anfrage und der Rückgabe ändern.

**Schlüssel müssen Objekte sein.** Keine Strings, keine Zahlen, keine Primitiven. Das ist grundlegend für die Funktionsweise schwacher Referenzen: Primitive Werte besitzen keine Identität, die getrennt von ihrem Wert verfolgt werden könnte.

Das sind keine Bugs. Das ist das Design. WeakMap ist für eine ganz bestimmte Aufgabe gebaut: Metadaten an Objekte zu hängen, ohne zu verhindern, dass diese Objekte aufgeräumt werden. Wenn Sie Iteration, primitive Schlüssel oder eine Zählung der Einträge benötigen, lösen Sie wahrscheinlich ein anderes Problem und sollten eine reguläre Map verwenden.

## Wo das wirklich hilft

Das ursprüngliche Einsatzszenario für das „private data“-Muster war die WeakMap, noch bevor JavaScript `#private`‑Felder hatte. Bibliotheken legten eine WeakMap außerhalb der Klasse an und nutzten sie, um Daten zu speichern, die nicht über die Instanz zugänglich sein sollten.

```javascript
const privateData = new WeakMap();

class User {
  constructor(name) {
    privateData.set(this, { name });
  }

  getName() {
    return privateData.get(this).name;
  }
}
```

Wird eine User‑Instanz vom Garbage Collector entfernt, verschwinden die privaten Daten mit ihr. Kein Aufräum‑Code nötig.

Memoisierung passt ebenfalls hervorragend, insbesondere wenn Ergebnisse anhand von Objekt‑Inputs und nicht anhand primitiver Werte zwischengespeichert werden. Wenn Ihre rechenintensive Berechnung ein Konfigurationsobjekt als Eingabe erhält, sorgt eine WeakMap dafür, dass der Cache nicht länger lebt als die Konfigurationen.

```javascript
const cache = new WeakMap();

function expensiveCalc(obj) {
  if (cache.has(obj)) return cache.get(obj);
  
  const result = heavyMath(obj);
  cache.set(obj, result);
  return result;
}
```

Der Cache existiert nur solange wie die zu cachenden Objekte. Sobald `obj` nirgendwo mehr referenziert wird, verschwinden sowohl das zwischengespeicherte Ergebnis als auch der Cache‑Eintrag gleichzeitig.

## Wann man darauf zurückgreifen sollte

Speicherlecks in modernen Web‑Apps entstehen meist durch verwaiste Referenzen auf Dinge, die eigentlich bereits aufgeräumt sein sollten. Wenn Sie etwas Langzeit‑Laufendes bauen – ein Dashboard, das den ganzen Tag geöffnet bleibt, eine Chat‑App, die stundenlang läuft, ein Admin‑Panel, das nie neu geladen wird – müssen Sie überlegen, was mit alten Daten passiert.

WeakMap ist besonders nützlich, wenn Sie Daten an DOM‑Knoten, Komponenten‑Instanzen oder beliebige Objekte binden, deren Lebensdauer Sie nicht steuern. Wenn Sie etwas basierend auf einer Referenz speichern und diese Referenz verschwinden kann, macht WeakMap die Bereinigung deutlich einfacher.

Ein reguläres Map bleibt die richtige Wahl, wenn Sie einen echten Cache mit Eviktions‑Strategien bauen, wenn Sie über Einträge iterieren müssen, primitive Schlüssel verwenden oder wenn die Daten selbst wichtiger sind als ihre Assoziation zu einem Objekt.

Der Vorteil von `WeakMap` ist, dass ihr Bedarf meist sofort erkennbar ist. Wenn Sie Code schreiben, der Map‑Einträge entfernt, sobald Objekte zerstört werden, ist das ein Hinweis. Wenn Sie befürchten, dass der Speicher unbegrenzt wächst, weil Sie nicht wissen, wann Sie Dinge löschen sollen, ist das ein weiterer Hinweis.

Manchmal ist das beste Feature jenes, das einfach funktioniert, ohne dass Sie darüber nachdenken müssen.
