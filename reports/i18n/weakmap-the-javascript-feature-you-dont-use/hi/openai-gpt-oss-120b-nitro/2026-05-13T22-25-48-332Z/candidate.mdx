आप जानते हैं वह एहसास जब आप कोड की एक लाइन बदलते हैं और देखते हैं कि आपकी मेमोरी उपयोग 50 % घट गई? मैं वही क्षण देख रहा था Chrome के DevTools Performance Monitor में, जब एक डैशबोर्ड ऐप एक घंटे में 100 MB तक मेमोरी लीक कर रहा था और फिर पूरे दोपहर तक साफ‑सुथरा चल रहा था।

एक‑लाइन का बदलाव: `new Map()` को `new WeakMap()` कर दिया।

बस इतना ही। वही API सतह, वही उपयोग पैटर्न, लेकिन अंदर‑बाहर बिल्कुल अलग व्यवहार। लेकिन यह समझना कि यह क्यों काम करता है, इसका मतलब है वह बात समझना जो अधिकांश जावास्क्रिप्ट डेवलपर्स कभी नहीं सोचते: जब आपके डेटा को अब कोई नहीं देख रहा होता तो क्या होता है।

## जब रेफ़रेंसेज़ एंकर बन जाती हैं

जावास्क्रिप्ट में एक सामान्य Map अपनी कुंजियों को कीमती माल की तरह मानता है। एक बार आप उसमें कुछ डाल देते हैं, तो Map इसे लोहे की पकड़ से पकड़ लेता है। गार्बेज कलेक्टर इस रिश्ते को देखता है और सोचता है, “स्पष्ट है कि उन्हें अभी भी इस ऑब्जेक्ट की ज़रूरत है, इसे नहीं छेड़ना चाहिए।”

यह सुरक्षा प्रवृत्ति तब समस्या बन जाती है जब आप अस्थायी चीज़ों के बारे में मेटाडेटा स्टोर कर रहे हों। जैसे हटाए गए DOM नोड्स, समाप्त हो रहे यूज़र सत्र, या अनमाउंट हो रहे कंपोनेंट इंस्टेंस। Map को नहीं पता कि ये ऑब्जेक्ट अब उपयोगी नहीं रहे; उसे सिर्फ़ पता है कि उसके पास एक रेफ़रेंस है, इसलिए वह उन्हें जीवित रखता है।

```javascript
const cache = new Map();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// एलिमेंट DOM से हट गया है, लेकिन cache अभी भी उसे मेमोरी में रख रहा है
```

गार्बेज कलेक्टर `element` को साफ़ नहीं कर पाता क्योंकि `cache` अभी भी उस पर इशारा कर रहा है। इसे “स्ट्रॉन्ग रेफ़रेंस” कहा जाता है, और लंबे‑समय तक चलने वाले सिंगल‑पेज ऐप्स में यह एक लीकेज बन जाता है जो अंततः ब्राउज़र को क्रैश कर देता है।

## WeakMap नियम बदल देता है

WeakMap अलग ढंग से काम करता है। यह अपनी कुंजियों को स्थायी निवासी की बजाय अस्थायी नागरिक मानता है। जब आप WeakMap में कुछ स्टोर करते हैं, तो आप मूल रूप से कह रहे होते हैं: “मैं इस डेटा को इस ऑब्जेक्ट से जोड़ना चाहता हूँ, लेकिन मैं नहीं चाहता कि मैं ही कारण बनूँ कि वह जीवित रहे।”

यदि किसी ऑब्जेक्ट को मेमोरी में रखने वाला एकमात्र रेफ़रेंस WeakMap ही है, तो गार्बेज कलेक्टर उसे ले सकता है। जब ऑब्जेक्ट गायब हो जाता है, तो WeakMap एंट्री भी उसके साथ गायब हो जाती है। कोई मैनुअल क्लीन‑अप की जरूरत नहीं।

```javascript
const cache = new WeakMap();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// The element gets Garbage Collected
// The cache entry vanishes automatically
```

मैंने 100,000 DOM नोड्स बनाकर, प्रत्येक पर मेटाडेटा स्टोर करके, फिर सबको हटाकर एक बेंचमार्क चलाया। Map के साथ, ब्राउज़र ने 150‑200 MB मेमोरी रखी। WeakMap के साथ, यह 70‑80 MB तक गिर गया। कोड और कार्यक्षमता समान, लेकिन मेमोरी फुटप्रिंट आधा।

## आप क्या त्यागते हैं

WeakMap में कुछ प्रतिबंध हैं जो पहले सीमाएँ लगते हैं, जब तक आप नहीं समझते कि वही जादू को संभव बनाते हैं।

**आप WeakMap पर इटररेट नहीं कर सकते।** `forEach`, `keys()`, `values()` नहीं। यह तब समझ में आता है जब आप सोचते हैं: गार्बेज कलेक्टर आपके लूप के बीच में एंट्री को हटा सकता है। क्या आप वास्तव में ऐसा संभालना चाहते हैं?

आप आकार (size) नहीं देख सकते। `.size` प्रॉपर्टी नहीं, `.length` नहीं। फिर भी, यह एक चलती हुई लक्ष्य है। संख्या आपके पूछने और उत्तर मिलने के बीच बदल सकती है।

**कीज़ ऑब्जेक्ट ही होना चाहिए।** स्ट्रिंग, नंबर या कोई प्रिमिटिव नहीं। यह कमजोर रेफ़रेंसेज़ के काम करने का मूल सिद्धांत है: प्रिमिटिव वैल्यूज़ की कोई अलग पहचान नहीं होती जिसे वैल्यू से अलग ट्रैक किया जा सके।

ये बग नहीं हैं। ये डिज़ाइन का हिस्सा हैं। WeakMap का एक ही काम है: ऑब्जेक्ट्स पर मेटाडेटा जोड़ना बिना उन ऑब्जेक्ट्स को क्लीन‑अप से रोकने के। यदि आपको इटररेशन, प्रिमिटिव कीज़ या एंट्री काउंट चाहिए, तो आप शायद किसी अलग समस्या को हल कर रहे हैं और नियमित Map का उपयोग करना चाहिए।

## जहाँ यह वास्तव में मदद करता है

“प्राइवेट डेटा” पैटर्न WeakMap का मूल उपयोग मामला था, जब जावास्क्रिप्ट में `#private` फ़ील्ड नहीं थे। लाइब्रेरीज़ क्लास के बाहर एक WeakMap बनाती थीं और उसे उन डेटा को स्टोर करने के लिए इस्तेमाल करती थीं जो इंस्टेंस पर एक्सेस नहीं होना चाहिए।

```javascript
const privateData = new WeakMap();

class User {
  constructor(name) {
    privateData.set(this, { name });
  }

  getName() {
    return privateData.get(this).name;
  }
}
```

जब User इंस्टेंस गार्बेज कलेक्ट हो जाता है, तो प्राइवेट डेटा भी उसके साथ हट जाता है। कोई क्लीन‑अप कोड की आवश्यकता नहीं।

Memoization भी एक स्वाभाविक फिट है, विशेषकर जब आप ऑब्जेक्ट इनपुट के आधार पर परिणामों को कैश कर रहे हों न कि प्रिमिटिव मानों पर। यदि आपका महँगा गणना फ़ंक्शन एक कॉन्फ़िग ऑब्जेक्ट को इनपुट लेता है, तो WeakMap आपको यह चिंता नहीं करने देता कि कैश कॉन्फ़िग्स से अधिक समय तक जीवित रहे।

```javascript
const cache = new WeakMap();

function expensiveCalc(obj) {
  if (cache.has(obj)) return cache.get(obj);
  
  const result = heavyMath(obj);
  cache.set(obj, result);
  return result;
}
```

कैश केवल उन ऑब्जेक्ट्स के जीवनकाल तक रहता है जो कैश किए गए हैं। एक बार `obj` कहीं और रेफ़रेंस नहीं रहा, तो कैश किया गया परिणाम और कैश एंट्री दोनों एक साथ गायब हो जाते हैं।

## When to Reach for It

आधुनिक वेब ऐप्स में मेमोरी लीक आमतौर पर उन पुराने रेफ़रेंसेज़ से आती हैं जो साफ़ होनी चाहिए थी। यदि आप कोई लंबा‑चलने वाला सिस्टम बना रहे हैं—एक डैशबोर्ड जो पूरे दिन खुला रहता है, एक चैट ऐप जो घंटों चलती है, या एक एडमिन पैनल जो कभी रिफ़्रेश नहीं होता—तो आपको यह सोचना पड़ेगा कि पुराना डेटा क्या होगा।

WeakMap विशेष रूप से उपयोगी है जब आप डेटा को DOM नोड्स, कंपोनेंट इंस्टेंस, या किसी भी ऑब्जेक्ट के साथ जोड़ रहे हों जिसकी लाइफ़टाइम आप नियंत्रित नहीं करते। यदि आप किसी रेफ़रेंस के आधार पर कुछ स्टोर कर रहे हैं और वह रेफ़रेंस हट सकता है, तो WeakMap क्लीन‑अप को बहुत सरल बनाता है।

Regular Map अभी भी सही विकल्प है जब आप वास्तविक कैश बना रहे हैं जिसमें eviction पॉलिसी हो, जब आपको एंट्रीज़ पर इटररेट करना हो, जब आप प्रिमिटिव कीज़ का उपयोग कर रहे हों, या जब डेटा स्वयं मायने रखता हो न कि उसकी ऑब्जेक्ट के साथ एसोसिएशन।

`WeakMap` की अच्छी बात यह है कि अक्सर यह स्पष्ट हो जाता है कि आपको इसकी ज़रूरत है। यदि आप ऑब्जेक्ट नष्ट होने पर मैप एंट्रीज़ को हटाने के लिए क्लीन‑अप कोड लिख रहे हैं, तो यह एक संकेत है। यदि आप मेमोरी के अनियंत्रित रूप से बढ़ने को लेकर चिंतित हैं क्योंकि आपको नहीं पता कब चीज़ें डिलीट करनी हैं, तो यह भी एक संकेत है।

कभी‑कभी सबसे अच्छा फीचर वही होता है जो बिना सोचे‑समझे काम करता है।
