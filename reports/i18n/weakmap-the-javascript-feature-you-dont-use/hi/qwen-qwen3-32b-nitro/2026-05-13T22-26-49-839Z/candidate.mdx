आपने उस महसूरत के बारे में तो सुना ही होगा जब आप एक पंक्ति कोड में बदलाव करते हैं और अपने स्मृमिति उपयोग में 50% तक कमी देखते हैं। मुझे ऐसा अनुभव क्रोम के डेवटूल्स प्रदरम्शन मॉनिटर के साथ हुआ जब एक डैशबोर्ड एप्लिकेशन 100 मेगाबाइट प्रति घंटा रक्तस्राव करने से एक पूरे दोपहर तक स्वच्छ चलने लगा।

एकल पंक्ति का बदलाव: `new Map()` बदलकर `new WeakMap()` बन गया।

यही है। एपीआई का वही सतह, उपयोग का वही पैटर्न, लेकिन अंदरूनी तौर पर पूरी तरह अलग व्यवहार। लेकिन इसके काम करने का कारण समझना हमें उस चीज़ को समझने की आवश्यकता है जिसे जावास्क्रिप्ट डेवलपर्स अक्सर नज़रअंदाज कर देते हैं: जब आपके डेटा पर कोई नज़र नहीं रह जाती।

## जब संदर्भ एंकर बन जाते हैं

जावास्क्रिप्ट में एक सामान्य `Map` अपनी कुंजियों को बहुत महत्वपूर्ण आइटम के रूप में देखता है। एक बार आप कुछ डाल देते हैं, तो `Map` उसे लौह काबू में रखता है। गारबेज कलेक्टर इस संबंध को देखकर सोचता है, "स्पष्ट रूप से उन्हें अभी भी इस ऑब्जेक्ट की आवश्यकता है, बेहतर है कि इसे छूए नहीं।"

यह रक्षात्मक इच्छा तब समस्या बन जाती है जब आप अस्थायी चीजों के मेटाडेटा को संग्रहीत कर रहे होते हैं। डीओएम नोड्स जो हटा दिए जाते हैं। उपयोगकर्ता सत्र जो समाप्त हो जाते हैं। कॉम्पोनेंट इंस्टेंस जो अनमाउंट हो जाते हैं। `Map` को इन ऑब्जेक्ट्स के उपयोग न होने के बारे में पता नहीं होता। यह केवल जानता है कि उसके पास एक संदर्भ है, इसलिए यह उन्हें जीवित रखता है।

```javascript
const cache = new Map();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// एलिमेंट डीओएम से हटा दिया गया है, लेकिन कैश इसे स्मृमिति में रख रहा है
```

`गारबेज कलेक्टर` `एलिमेंट` को साफ़ कर नहीं सकता क्योंकि `कैश` इस पर अभी तक नज़र रख रहा है। इसे "मजबूत संदर्भ" कहा जाता है, और लंबे समय तक चलने वाले सिंगल पेज एप्लिकेशन में यह एक रिसेप्शन बन जाता है जो अंततः ब्राउज़र को क्रैश कर देता है।

## वीकमैप नियम बदल देता है

एक `वीकमैप` अलग तरीके से काम करता है। यह अपनी कुंजियों को स्थायी नागरिकों के बजाय अस्थायी नागरिकों के रूप में देखता है। जब आप कुछ डालते हैं तो आप वास्तव में कहते हैं: "मैं इस डेटा को इस ऑब्जेक्ट से जोड़ना चाहता हूं, लेकिन मैं इसके जीवित रहने का कारण नहीं बनना चाहता।"

अगर किसी ऑब्जेक्ट को स्मृमिति में रखे रखने का एकमात्र कारण `वीकमैप` है, तो गारबेज कलेक्टर उसे ले ले जा सकता है। जब ऑब्जेक्ट गायब हो जाता है, तो `वीकमैप` की प्रविष्टि भी गायब हो जाती है। कोई मैन्युअल सफाई आवश्यक नहीं है।

```javascript
const cache = new WeakMap();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// ऑब्जेक्ट गारबेज कलेक्शन हो जाता है
// कैश प्रविष्टि स्वत: गायब हो जाती है
```

मैंने 1 लाख DOM नोड्स बनाकर उनके बारे में मेटाडेटा संग्रहित करने और फिर उन्हें हटाने का बेंचमार्क चलाया। `Map` के साथ ब्राउज़र 150-200MB तक मेमोरी ले रहा था। `WeakMap` के साथ यह 70-80MB तक घट गया। एक ही कोड, एक ही फ़ंक्शनलिटी, लेकिन मेमोरी उपयोग आधा।

## आप क्या छोड़ देते हैं

`WeakMap` में ऐसे सीमाएँ हैं जो लिमिटेशन लगती हैं जब तक आपको यह नहीं समझ में आ जाता कि यही वो चीज़ है जो इसके जादू को काम करता है।

**आप `WeakMap` को इटरेट नहीं कर सकते।** कोई `forEach`, कोई `keys()`, कोई `values()` नहीं। इसे समझने पर यह बनता है: गारबेज कलेक्टर आपके लूप के बीच में एंट्री मिटा सकता है। क्या आप वाकई ऐसा संभालना चाहेंगे?

आप आकार नहीं चेक कर सकते। कोई `.size` प्रॉपर्टी, कोई `.length` नहीं। फिर से, यह एक चलती निशाना है। आपके पूछे से लेकर जवाब आने तक संख्या बदल सकती है।

**कुंजियाँ ऑब्जेक्ट होनी चाहिए।** कोई स्ट्रिंग, कोई नंबर, कोई प्राइमिटिव नहीं। यह वीक रेफरेंस के काम करने के तरीके के लिए मूल है: प्राइमिटिव मानों के पास अपने मान से अलग पहचान नहीं होती है।

ये बग नहीं हैं। ये डिज़ाइन है। `WeakMap` को एक विशिष्ट काम के लिए बनाया गया है: ऑब्जेक्ट्स से मेटाडेटा जोड़ना बिना उन ऑब्जेक्ट्स के सफ़ाई से रोके। अगर आपको इटरेशन चाहिए या प्राइमिटिव कुंजियाँ या एंट्री की गिनती, तो आपको शायद एक अलग समस्या ही हल करनी है और आपको एक सामान्य `Map` का उपयोग करना चाहिए।

## जहाँ यह वास्तव में मदद करता है

"निजी डेटा" पैटर्न `WeakMap` का मूल उपयोग था, जब तक जावास्क्रिप्ट में `#private` फील्ड्स नहीं आए। लाइब्रेरियाँ क्लास के बाहर `WeakMap` बनाती थीं और उसका उपयोग उदाहरण पर उपलब्ध नहीं होने वाले डेटा संग्रहित करने के लिए करती थीं।

```javascript
const privateData = new WeakMap();

class User {
  constructor(name) {
    privateData.set(this, { name });
  }

  getName() {
    return privateData.get(this).name;
  }
}
```

जब `User` उदाहरण गारबेज कलेक्शन हो जाता है, तो निजी डेटा भी उसके साथ चला जाता है। कोई सफ़ाई कोड आवश्यक नहीं है।

मेमोइजेशन एक और प्राकृतिक फिट है, खासकर जब आप प्राथमिक मानों के बजाय ऑब्जेक्ट इनपुट्स के आधार पर परिणामों को कैश कर रहे हों। अगर आपकी महंगी गणना एक कॉन्फ़िग ऑब्जेक्ट के रूप में इनपुट लेती है, तो वीकमैप का उपयोग करने से आपको कैश के कॉन्फ़िग्स के जीवनकाल के बारे में चिंता करने की आवश्यकता नहीं होती है।

```javascript
const cache = new WeakMap();

function expensiveCalc(obj) {
  if (cache.has(obj)) return cache.get(obj);
  
  const result = heavyMath(obj);
  cache.set(obj, result);
  return result;
}
```

कैश केवल उन ऑब्जेक्ट्स के जीवनकाल के बराबर रहता है जिनके आधार पर यह कैश किया गया है। एक बार `obj` कहीं भी अन्यत्र संदर्भित नहीं होता, तो कैश किया गया परिणाम और कैश प्रविष्टि एक साथ गारबेज कलेक्शन हो जाते हैं।

## जब इसका उपयोग करें

मॉडर्न वेब एप्लिकेशन में मेमोरी लीक आमतौर पर उन चीजों के स्थायी संदर्भों से आते हैं जिन्हें साफ़ कर दिया जाना चाहिए था। अगर आप कुछ ऐसा बना रहे हैं जो लंबे समय तक चलता रहे, जैसे एक डैशबोर्ड जो पूरे दिन खुला रहे, एक चैट एप्लिकेशन जो घंटों चलता रहे, या एक एडमिन पैनल जो कभी रिफ्रेश नहीं होता, तो आपको पुराने डेटा के बारे में विचार करने की आवश्यकता होती है।

वीकमैप तब खासकर उपयोगी होता है जब आप डीएन नोड्स, कॉम्पोनेंट उदाहरणों, या ऐसे ऑब्जेक्ट्स के साथ डेटा संग्रहित कर रहे हों जिनके जीवनकाल पर आपका नियंत्रण नहीं है। अगर आप किसी संदर्भ के आधार पर कुछ संग्रहित कर रहे हैं और वह संदर्भ चला जा सकता है, तो वीकमैप साफ़ करने को बहुत सरल बनाता है।

सामान्य मैप तब अभी भी सही चयन होता है जब आप वास्तविक कैश बना रहे हों जिसमें एविक्शन पॉलिसी हो, जब आप प्रविष्टियों पर पुनरावृत्ति करने की आवश्यकता हो, जब आप प्राथमिक कुंजियों का उपयोग कर रहे हों, या जब डेटा खुद के बजाय ऑब्जेक्ट के संगति के बारे में चीजें महत्वपूर्ण हों।

`वीकमैप` की अच्छी बात यह है कि आमतौर पर आपको जानकर इसकी आवश्यकता स्पष्ट हो जाती है। अगर आप ऐसा महसूस करते हैं कि ऑब्जेक्ट्स के नष्ट होने पर मैप प्रविष्टियों को हटाने के लिए साफ़ करने का कोड लिख रहे हैं, तो यह एक संकेत है। अगर आप मेमोरी के अनियंत्रित वृद्धि के बारे में चिंतित हैं क्योंकि आपको नहीं पता कि कब कुछ हटाना है, तो यह एक और संकेत है।

कभी-कभी सबसे अच्छी सुविधा वह होती है जो आपके द्वारा इसके बारे में सोचे बिना ही काम करती है।
