आपको वह एहसास पता है जब आप कोड की एक लाइन बदलते हैं और अपनी मेमोरी उपयोग को 50% गिरते देखते हैं? मुझे वह पल तब आया जब मैं Chrome के DevTools Performance Monitor में एक डैशबोर्ड ऐप को प्रति घंटे 100MB मेमोरी गंवाते हुए से लेकर पूरी दोपहर साफ चलते देख रहा था।

एक लाइन का बदलाव: `new Map()` को `new WeakMap()` में बदल दिया।

बस इतना ही। एक ही API सतह, एक ही उपयोग पैटर्न, लेकिन हुड के नीचे पूरी तरह से अलग व्यवहार। लेकिन यह समझना कि यह क्यों काम करता है, इसका मतलब है कुछ ऐसा समझना जिसके बारे में अधिकांश JavaScript डेवलपर्स कभी नहीं सोचते: जब आपके डेटा को कोई नहीं देख रहा होता है तो क्या होता है।

## जब संदर्भ एंकर बन जाते हैं

JavaScript में एक सामान्य Map अपनी कुंजियों को कीमती सामान की तरह मानता है। एक बार जब आप उसमें कुछ डालते हैं, तो Map उसे लोहे की पकड़ से पकड़े रखता है। Garbage Collector इस संबंध को देखता है और सोचता है, "स्पष्ट रूप से उन्हें अभी भी इस ऑब्जेक्ट की जरूरत है, बेहतर होगा कि इसे न छूएं।"

यह सुरक्षात्मक प्रवृत्ति तब समस्या बन जाती है जब आप अस्थायी चीजों के बारे में मेटाडेटा संग्रहीत कर रहे होते हैं। DOM नोड जो हटा दिए जाते हैं। उपयोगकर्ता सत्र जो समाप्त हो जाते हैं। कंपोनेंट इंस्टेंस जो अनमाउंट हो जाते हैं। Map को नहीं पता कि ये ऑब्जेक्ट उपयोगी होना बंद कर चुके हैं। उसे बस पता है कि उसके पास एक संदर्भ है, इसलिए वह उन्हें जीवित रखता है।

```javascript
const cache = new Map();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// The element is gone from the DOM, but cache is keeping it in memory
```

Garbage Collector `element` को साफ नहीं कर सकता क्योंकि `cache` अभी भी उसकी ओर इशारा कर रहा है। इसे "strong reference" कहा जाता है, और लंबे समय तक चलने वाले Single Page Apps में, यह एक लीक बन जाता है जो अंततः ब्राउज़र को क्रैश कर देता है।

## WeakMap नियम बदलता है

WeakMap अलग तरह से काम करता है। यह अपनी कुंजियों को स्थायी निवासियों के बजाय अस्थायी नागरिकों के रूप में मानता है। जब आप WeakMap में कुछ संग्रहीत करते हैं, तो आप मूल रूप से कह रहे होते हैं: "मैं इस डेटा को इस ऑब्जेक्ट से जोड़ना चाहता हूं, लेकिन मैं इसके जीवित रहने का कारण नहीं बनना चाहता।"

यदि किसी ऑब्जेक्ट को मेमोरी में रखने वाली एकमात्र चीज WeakMap है, तो Garbage Collector को इसे लेने की अनुमति है। जब ऑब्जेक्ट गायब हो जाता है, तो WeakMap एंट्री भी उसके साथ गायब हो जाती है। किसी मैन्युअल सफाई की आवश्यकता नहीं है।

```javascript
const cache = new WeakMap();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// एलिमेंट Garbage Collect हो जाता है
// कैश एंट्री अपने आप गायब हो जाती है
```

मैंने एक बेंचमार्क चलाया जिसमें 100,000 DOM नोड बनाए, हर एक के बारे में मेटाडेटा स्टोर किया, फिर सबको हटा दिया। Map के साथ, ब्राउज़र 150-200MB पर टिका रहा। WeakMap के साथ, यह घटकर 70-80MB रह गया। एक ही कोड, एक ही कार्यक्षमता, आधी मेमोरी फुटप्रिंट।

## आप क्या छोड़ते हैं

WeakMap की कुछ सीमाएँ हैं जो तब तक बाधाओं जैसी लगती हैं जब तक आपको एहसास न हो कि ये वही चीज़ें हैं जो इसके जादू को काम करने देती हैं।

**आप WeakMap पर इटरेट नहीं कर सकते।** कोई `forEach` नहीं, कोई `keys()` नहीं, कोई `values()` नहीं। यह समझ में आता है जब आप इसके बारे में सोचते हैं: Garbage Collector आपके लूप के बीच में एक एंट्री डिलीट कर सकता है। क्या आप सच में उससे निपटना चाहते हैं?

आप साइज़ नहीं जाँच सकते। कोई `.size` प्रॉपर्टी नहीं, कोई `.length` नहीं। फिर से, यह एक चलता-फिरता लक्ष्य है। जब आप पूछते हैं और जब आपको जवाब मिलता है, तब के बीच संख्या बदल सकती है।

**कीज़ को ऑब्जेक्ट होना चाहिए।** कोई स्ट्रिंग नहीं, कोई नंबर नहीं, कोई प्रिमिटिव नहीं। यह इस बात का मूल है कि वीक रेफरेंस कैसे काम करते हैं: प्रिमिटिव वैल्यूज़ की कोई ऐसी पहचान नहीं होती जिसे उनके मान से अलग ट्रैक किया जा सके।

ये बग नहीं हैं। ये डिज़ाइन हैं। WeakMap एक विशिष्ट काम के लिए बना है: ऑब्जेक्ट्स से मेटाडेटा जोड़ना बिना उन ऑब्जेक्ट्स को साफ होने से रोके। अगर आपको इटरेशन या प्रिमिटिव कीज़ या एंट्रीज़ की गिनती चाहिए, तो आप शायद कोई अलग समस्या हल कर रहे हैं और आपको एक सामान्य Map का उपयोग करना चाहिए।

## यह वास्तव में कहाँ मदद करता है

"प्राइवेट डेटा" पैटर्न WeakMap का मूल उपयोग मामला था, उस समय से पहले जब जावास्क्रिप्ट में `#private` फील्ड्स आए। लाइब्रेरीज़ क्लास के बाहर एक WeakMap बनाती थीं और उसका उपयोग उस डेटा को स्टोर करने के लिए करती थीं जो इंस्टेंस पर एक्सेसिबल नहीं होना चाहिए।

```javascript
const privateData = new WeakMap();

class User {
  constructor(name) {
    privateData.set(this, { name });
  }

  getName() {
    return privateData.get(this).name;
  }
}
```

जब कोई User इंस्टेंस Garbage Collect हो जाता है, तो प्राइवेट डेटा भी उसके साथ चला जाता है। कोई क्लीनअप कोड ज़रूरी नहीं।

मेमोइज़ेशन एक और स्वाभाविक उपयोग है, खासकर जब आप प्रिमिटिव वैल्यू के बजाय ऑब्जेक्ट इनपुट के आधार पर परिणाम कैश कर रहे हों। यदि आपकी महँगी गणना एक कॉन्फ़िग ऑब्जेक्ट को इनपुट के रूप में लेती है, तो WeakMap का मतलब है कि आपको कैश के कॉन्फ़िग से अधिक जीवित रहने की चिंता नहीं करनी पड़ेगी।

```javascript
const cache = new WeakMap();

function expensiveCalc(obj) {
  if (cache.has(obj)) return cache.get(obj);
  
  const result = heavyMath(obj);
  cache.set(obj, result);
  return result;
}
```

कैश केवल तब तक जीवित रहता है जब तक कैश किए जा रहे ऑब्जेक्ट जीवित हैं। एक बार जब `obj` का कहीं और कोई संदर्भ नहीं रहता, तो कैश किया गया परिणाम और कैश एंट्री दोनों एक साथ गायब हो जाते हैं।

## इसका उपयोग कब करें

आधुनिक वेब ऐप्स में मेमोरी लीक आमतौर पर उन चीज़ों के पुराने संदर्भों से आते हैं जिन्हें साफ़ कर दिया जाना चाहिए था। यदि आप कोई लंबे समय तक चलने वाला एप्लिकेशन बना रहे हैं, एक डैशबोर्ड जो पूरे दिन खुला रहता है, एक चैट ऐप जो घंटों चलता है, एक एडमिन पैनल जो कभी रिफ्रेश नहीं होता, तो आपको यह सोचने की ज़रूरत है कि पुराने डेटा का क्या होता है।

WeakMap विशेष रूप से तब उपयोगी होता है जब आप DOM नोड्स, कंपोनेंट इंस्टेंस, या किसी भी ऐसे ऑब्जेक्ट के साथ डेटा जोड़ रहे हों जिसका जीवनकाल आप नियंत्रित नहीं करते। यदि आप किसी संदर्भ के आधार पर कुछ स्टोर कर रहे हैं और वह संदर्भ खत्म हो सकता है, तो WeakMap सफाई को बहुत सरल बना देता है।

नियमित Map अभी भी सही विकल्प है जब आप निष्कासन नीतियों वाला एक वास्तविक कैश बना रहे हों, जब आपको एंट्रीज़ पर इटरेट करने की आवश्यकता हो, जब आप प्रिमिटिव कीज़ का उपयोग कर रहे हों, या जब डेटा स्वयं महत्वपूर्ण हो न कि किसी ऑब्जेक्ट के साथ उसका जुड़ाव।

`WeakMap` के बारे में अच्छी बात यह है कि आमतौर पर यह स्पष्ट होता है कि आपको इसकी आवश्यकता कब है। यदि आप स्वयं को ऑब्जेक्ट नष्ट होने पर मैप एंट्रीज़ को हटाने के लिए क्लीनअप कोड लिखते हुए पाते हैं, तो यह एक संकेत है। यदि आप इस बात को लेकर चिंतित हैं कि मेमोरी अनियंत्रित रूप से बढ़ रही है क्योंकि आपको नहीं पता कि कब चीज़ें हटानी हैं, तो यह एक और संकेत है।

कभी-कभी सबसे अच्छी सुविधा वह होती है जो आपके बिना सोचे ही काम करती है।
