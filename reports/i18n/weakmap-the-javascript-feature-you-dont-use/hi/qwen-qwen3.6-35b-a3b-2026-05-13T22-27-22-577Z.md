## जब रेफरेंस एंकर बन जाते हैं

JavaScript में एक साधारण Map अपने कीज (keys) को कीमती माल की तरह पेश आता है। एक बार डेटा डालने के बाद, Map उसे लोहे की मुट्ठी में थामे रखता है। Garbage Collector इस रिश्ते को देखकर यह निष्कर्ष निकालता है कि 'डेवलपर को अभी भी इस ऑब्जेक्ट की जरूरत है, इसे मेमोरी से हटाना बेहतर नहीं होगा।'

यह संरक्षण की प्रवृत्ति तब समस्या बन जाती है जब आप अस्थायी ऑब्जेक्ट्स के मेटाडेटा स्टोर कर रहे हों। हटाए गए DOM नोड्स। एक्सपायर हो चुके यूजर सेशन। अनमाउंट हो चुके कंपोनेंट इंस्टेंस। Map को यह नहीं पता कि ये ऑब्जेक्ट्स अब उपयोग के लिए निष्क्रिय हैं। उसे बस इतना पता है कि उसके पास उनका रेफरेंस है, इसलिए वह उन्हें मेमोरी में जीवित रखता है।

```javascript
const cache = new Map();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// The element is gone from the DOM, but cache is keeping it in memory
```

Garbage Collector `element` को मेमोरी से हटा नहीं सकता क्योंकि `cache` अभी भी उसका रेफरेंस रख रहा है। इसे 'स्ट्रॉंग रेफरेंस' कहा जाता है। लंबे समय तक चलने वाले Single Page Apps में, यह एक मेमोरी लीक बन जाता है जो आखिरकार ब्राउज़र को क्रैश कर देता है।

## WeakMap नियम बदल देता है

WeakMap अलग तर्क पर काम करता है। यह अपने कीज को स्थायी निवासियों के बजाय अस्थायी नागरिक मानता है। WeakMap में डेटा स्टोर करते समय आप मूल रूप से यह कह रहे होते हैं: 'मैं इस डेटा को इस ऑब्जेक्ट से जोड़ना चाहता हूं, लेकिन मैं यह नहीं चाहता कि मेरा रेफरेंस ही इसे मेमोरी में जीवित रखे।'

अगर किसी ऑब्जेक्ट को मेमोरी में रखने वाली एकमात्र चीज WeakMap का रेफरेंस है, तो Garbage Collector उसे हटाने की अनुमति पाता है। जब ऑब्जेक्ट का बाहरी रेफरेंस खत्म होता है, तो WeakMap का एंट्री भी अपने आप गायब हो जाता है। मैनुअल क्लीनअप की कोई जरूरत नहीं।

```javascript
const cache = new WeakMap();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// The element gets Garbage Collected
// The cache entry vanishes automatically
```

मैंने 100,000 DOM नोड्स बनाकर, प्रत्येक के बारे में मेटाडेटा स्टोर करके और फिर उन्हें हटाकर एक बेंचमार्क चलाया। Map के साथ, ब्राउज़र 150-200MB मेमोरी पकड़े रखता था। WeakMap के साथ, यह घटकर 70-80MB रह गया। कोड वही था, फंक्शनलिटी वही थी, लेकिन मेमोरी फुटप्रिंट आधा रह गया।

## आप क्या त्याग रहे हैं

WeakMap में ऐसे प्रतिबंध हैं जो सीमाओं जैसे लगते हैं, जब तक कि आपको एहसास न हो कि यही वो चीज़ें हैं जो जादू को काम करवाती हैं।

**आप WeakMap पर इटरेट नहीं कर सकते।** कोई `forEach`, कोई `keys()`, कोई `values()` नहीं। अगर गहराई से सोचें तो यह तर्कसंगत है: Garbage Collector आपके लूप के बीच में ही एक एंट्री डिलीट कर सकता है। क्या आप सच में उसका सामना करना चाहेंगे?

आप साइज़ चेक नहीं कर सकते। कोई `.size` प्रॉपर्टी नहीं, कोई `.length` नहीं। फिर से, यह स्थिर नहीं रहता। आप पूछने और जवाब पाने के बीच में ही नंबर बदल सकता है।

**कीज़ ऑब्जेक्ट्स ही होनी चाहिए।** कोई स्ट्रिंग नहीं, कोई नंबर नहीं, कोई प्रिमिटिव नहीं। यह वेक रेफरेंस के काम करने के तरीके का मूल सिद्धांत है: प्रिमिटिव वैल्यूज की अपनी वैल्यू से अलग ट्रैक की जा सकने वाली कोई पहचान (identity) नहीं होती।

ये बग्स नहीं हैं। ये डिज़ाइन है। WeakMap को एक ही खास काम के लिए बनाया गया है: ऑब्जेक्ट्स से मेटाडेटा जोड़ना, बिना उन ऑब्जेक्ट्स को क्लीनअप होने से रोकें। अगर आपको इटेरेशन, प्रिमिटिव कीज़ या एंट्रीज़ की गिनती चाहिए, तो आप शायद किसी अलग समस्या को सॉल्व कर रहे हैं और आपको एक रेगुलर Map का इस्तेमाल करना चाहिए।

## यह असल में कहाँ काम आता है

“प्राइवेट डेटा” पैटर्न WeakMap का असली यूज़ केस था, जब JavaScript में `#private` फील्ड्स नहीं थे। लाइब्रेरीज़ क्लास के बाहर एक WeakMap बनाती थीं और उसका इस्तेमाल उन डेटा को स्टोर करने के लिए करती थीं जो इंस्टेंस पर एक्सेस योग्य नहीं होने चाहिए थे।

```javascript
const privateData = new WeakMap();

class User {
  constructor(name) {
    privateData.set(this, { name });
  }

  getName() {
    return privateData.get(this).name;
  }
}
```

जब एक User इंस्टेंस Garbage Collector द्वारा कलेक्ट हो जाता है, तो प्राइवेट डेटा भी उसके साथ चला जाता है। क्लीनअप कोड की कोई जरूरत नहीं।

मेमोइज़ेशन भी एक प्राकृतिक उपयोग है, खासकर जब आप प्राइमिटिव मानों के बजाय ऑब्जेक्ट इनपुट के आधार पर परिणाम कैश कर रहे हों। अगर आपका भारी कैलकुलेशन एक कॉन्फ़िग ऑब्जेक्ट को इनपुट के रूप में लेता है, तो WeakMap का मतलब है कि आपको यह चिंता नहीं करनी पड़ेगी कि कैश कॉन्फ़िग से ज्यादा देर तक जीवित रहेगा।

```javascript
const cache = new WeakMap();

function expensiveCalc(obj) {
  if (cache.has(obj)) return cache.get(obj);
  
  const result = heavyMath(obj);
  cache.set(obj, result);
  return result;
}
```

कैश केवल उतने ही समय तक जीवित रहता है जितने समय तक कैश किए गए ऑब्जेक्ट्स मौजूद होते हैं। एक बार जब `obj` को कहीं और रेफरेंस नहीं किया जाता, तो कैश किया गया परिणाम और कैश एंट्री दोनों एक साथ गायब हो जाते हैं।

## कब इसका उपयोग करें

आधुनिक वेब ऐप्स में मेमोरी लीक्स आमतौर पर उन चीज़ों के पुराने रेफरेंस से आते हैं जिन्हें क्लीनअप किया जाना चाहिए था। अगर आप कुछ लंबे समय तक चलने वाला बना रहे हैं—जिस दिन भर खुला रहने वाला डैशबोर्ड, घंटों चलने वाला चैट ऐप, या कभी रीफ्रेश न होने वाला एडमिन पैनल—तो आपको यह सोचना होगा कि पुराना डेटा क्या होगा।

WeakMap तब विशेष रूप से उपयोगी होता है जब आप डेटा को DOM नोड्स, कंपोनेंट इंस्टेंस, या किसी ऐसे ऑब्जेक्ट से जोड़ रहे हों जिसका जीवनकाल आप कंट्रोल नहीं करते। अगर आप किसी रेफरेंस के आधार पर कुछ स्टोर कर रहे हैं और वह रेफरेंस गायब भी हो सकता है, तो WeakMap क्लीनअप को काफी आसान बना देता है।

रेगुलर Map अभी भी सही विकल्प है जब आप एविकशन पॉलिसीज़ के साथ एक असली कैश बना रहे हों, जब आपको एंट्रीज़ पर इटरेट करने की जरूरत हो, जब आप प्राइमिटिव कीज़ का उपयोग कर रहे हों, या जब डेटा स्वयं मायने रखता हो न कि उसका किसी ऑब्जेक्ट से जुड़ाव।

`WeakMap` की अच्छी बात यह है कि आमतौर पर यह स्पष्ट होता है कि आपको इसकी कब जरूरत है। अगर आपको ऑब्जेक्ट्स डस्ट्रॉय होने पर मैप एंट्रीज़ हटाने के लिए क्लीनअप कोड लिखते हुए पाते हैं, तो यह एक संकेत है। अगर आपको डर है कि मेमोरी बिना किसी सीमा के बढ़ रही है क्योंकि आपको यह नहीं पता कि कब डिलीट करना है, तो यह एक और संकेत है।
