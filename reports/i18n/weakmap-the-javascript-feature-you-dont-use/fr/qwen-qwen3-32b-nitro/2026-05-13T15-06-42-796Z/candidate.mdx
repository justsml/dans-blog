Vous connaissez cette sensation où vous modifiez une seule ligne de code et voyez votre utilisation de la mémoire baisser de 50 % ? C'est ce que j'ai vécu en observant le moniteur de performance des outils de développement Chrome alors qu'une application de tableau de bord passait d'un saignement de 100 Mo par heure à une exécution propre pendant toute l'après-midi.

Le changement d'une ligne : `new Map()` est devenu `new WeakMap()`.

C'est tout. Même surface d'API, même schéma d'utilisation, comportement complètement différent en interne. Mais comprendre pourquoi cela fonctionne nécessite de comprendre quelque chose que la plupart des développeurs JavaScript n'ont jamais envisagé : ce qui se passe quand plus rien ne regarde vos données.

## Quand les références deviennent des ancrages

Un Map classique en JavaScript traite ses clés comme un trésor précieux. Une fois que vous y placez quelque chose, le Map s'en accroche avec une poigne de fer. Le Garbage Collector voit cette relation et pense : « Évidemment, ils ont encore besoin de cet objet, mieux vaut ne pas y toucher ».

Ce réflexe protecteur devient un problème lorsque vous stockez des métadonnées sur des éléments temporaires. Des nœuds DOM supprimés. Des sessions d'utilisateurs expirées. Des instances de composants qui se désinstallent. Le Map n'a aucun moyen de savoir que ces objets ne sont plus utiles. Il sait juste qu'il a une référence, donc il les maintient en vie.

```javascript
const cache = new Map();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// The element is gone from the DOM, but cache is keeping it in memory
```

Le Garbage Collector ne peut pas nettoyer `element` car `cache` fait toujours référence à cet objet. On appelle cela une « référence forte », et dans les Single Page Apps longues en cours d'exécution, cela devient un fuit qui finit par crasher le navigateur.

## WeakMap Change les Règles

Un WeakMap fonctionne différemment. Il traite ses clés comme des citoyens temporaires plutôt que des résidents permanents. Quand vous stockez quelque chose dans un WeakMap, vous dites essentiellement : « Je veux associer ces données à cet objet, mais je ne veux pas être la raison pour laquelle il reste en mémoire ».

Si le seul élément qui maintient un objet en mémoire est un WeakMap, le Garbage Collector est autorisé à le supprimer. Quand l'objet disparaît, l'entrée du WeakMap disparaît avec lui. Aucun nettoyage manuel n'est nécessaire.

```javascript
const cache = new WeakMap();

function trackClick(element) {
  cache.set(element, { clicks: 0 });
}

document.body.removeChild(element);
// The element gets Garbage Collected
// The cache entry vanishes automatically
```

J'ai effectué un benchmark en créant 100 000 nœuds DOM, en stockant des métadonnées sur chacun, puis en les supprimant tous. Avec un Map, le navigateur conservait 150-200 Mo. Avec un WeakMap, il est passé à 70-80 Mo. Même code, même fonctionnalité, une empreinte mémoire réduite de moitié.

## Ce que vous abandonnez

Le WeakMap a des contraintes qui semblent être des limites jusqu'à ce que vous réalisiez qu'elles sont ce qui rend le fonctionnement possible.

**Vous ne pouvez pas itérer sur un WeakMap.** Aucun `forEach`, aucune `keys()`, aucune `values()`. Cela a du sens si vous y réfléchissez : le Garbage Collector pourrait supprimer une entrée pendant que vous bouclez. Souhaitez-vous vraiment gérer cela ?

Vous ne pouvez pas vérifier la taille. Aucune propriété `.size`, aucune `.length`. Encore une fois, c'est une cible en mouvement. Le nombre pourrait changer entre le moment où vous posez la question et celui où vous obtenez la réponse.

**Les clés doivent être des objets.** Aucune chaîne, aucun nombre, aucune valeur primitive. C'est fondamental au fonctionnement des références faibles : les valeurs primitives n'ont pas d'identité pouvant être suivie séparément de leur valeur.

Ce ne sont pas des bugs. C'est la conception. Le WeakMap est conçu pour une tâche spécifique : attacher des métadonnées à des objets sans empêcher ces objets d'être nettoyés. Si vous avez besoin d'itération, de clés primitives ou d'un décompte des entrées, vous résolvez probablement un problème différent et devriez utiliser un Map standard.

## Où cela aide réellement

Le motif de **données privées** était le cas d'utilisation initial de `WeakMap`, avant que JavaScript n'ait des champs privés (`#private`). Des bibliothèques créaient un `WeakMap` en dehors de la classe pour stocker des données qui ne devaient pas être accessibles sur l'instance.

```javascript
const privateData = new WeakMap();

class User {
  constructor(name) {
    privateData.set(this, { name });
  }

  getName() {
    return privateData.get(this).name;
  }
}
```

Quand une instance `User` est recueillie par le Garbage Collector, les données privées disparaissent avec elle. Aucun code de nettoyage n'est nécessaire.

La **mémorisation** est un autre cas d'usage naturel, surtout quand vous cachez des résultats basés sur des objets plutôt que des valeurs primitives. Si votre calcul coûteux prend un objet de configuration en entrée, un `WeakMap` signifie que vous n'avez pas à vous inquiéter du cache qui survit aux configurations.

```javascript
const cache = new WeakMap();

function expensiveCalc(obj) {
  if (cache.has(obj)) return cache.get(obj);
  
  const result = heavyMath(obj);
  cache.set(obj, result);
  return result;
}
```

Le cache existe uniquement aussi longtemps que les objets mis en cache. Une fois que `obj` n'est plus référencé nulle part ailleurs, à la fois le résultat mis en cache et l'entrée du cache disparaissent ensemble.

## Quand l'utiliser

Les fuites de mémoire dans les applications web modernes viennent souvent de références obsolètes à des objets qui auraient dû être nettoyés. Si vous construisez une application longue durée — un tableau de bord ouvert toute la journée, une application de chat fonctionnant pendant des heures, un panneau d'administration qui ne se recharge jamais — vous devez réfléchir à ce qui advient des anciennes données.

`WeakMap` est particulièrement utile lorsque vous associez des données à des nœuds DOM, des instances de composants ou tout objet dont la durée de vie que vous ne contrôlez pas. Si vous stockez quelque chose en fonction d'une référence et que cette référence pourrait disparaître, `WeakMap` rend le nettoyage bien plus simple.  

`Map` classique reste le bon choix lorsque vous construisez un cache réel avec des politiques d'éviction, lorsque vous avez besoin d'itérer sur les entrées, lorsque vous utilisez des clés primitives, ou lorsque les données elles-mêmes sont importantes plutôt que leur association à un objet.  

L'avantage de `WeakMap` est qu'il est généralement évident quand vous en avez besoin. Si vous vous trouvez à écrire du code de nettoyage pour supprimer des entrées de la carte lorsque les objets sont détruits, c'est un signe. Si vous êtes préoccupé par une croissance incontrôlée de la mémoire car vous ne savez pas quand supprimer des éléments, c'est un autre signe.  

Parfois, la meilleure fonctionnalité est celle qui fonctionne sans que vous ayez à y penser.
