Chaque nouveau modèle arrive dans un smoking de benchmarks.  

MMLU : 92,4 %. HumanEval : 87,2 %. LLeMU : 88,7 %. MATH : 73,6 %. AGI : 127 % !  

Pourtant, pour 99 % des entreprises construisant des processus et des produits avec l’IA, **rien de tout cela n’a d’importance**.  

Qu’est-ce qui importe ? Comment se comportent VOS charges de travail ? Amélioration ou détérioration ? La seule façon raisonnable de le savoir est d’écrire des Evals (tests pour les LLM) reflétant les tâches, les données et les modes de défaillance spécifiques à votre système.  

<blockquote class="breakout">  
  <p>Les benchmarks ne mentent pas. Ils répondent à une autre question.</p>  
</blockquote>

## Ce que coûte vraiment l'évaluation basée sur les "vibes"

L'approche standard : déployer un changement de modèle, surveiller les canaux de signalement, et revenir en arrière si le niveau de bruit augmente.  

Cela manque presque tout ce qui est pertinent :

**Vous ne détectez que les échecs bruyants.** Les utilisateurs qui reçoivent une réponse clairement fausse sans s'en rendre compte ? Silencieux. Les utilisateurs qui obtiennent une réponse pire et abandonnent la fonctionnalité ? Silencieux. Les tickets de support et les taux d'erreur capturent uniquement une fraction des régressions de qualité.  

**Vous ne pouvez pas distinguer les régressions des améliorations.** Si le nouveau modèle est meilleur pour la tâche A et pire pour la tâche B, les plaintes sur B ressemblent à des commentaires génériques du type « l'IA est devenue pire ». Vous ne savez pas quoi corriger.  

**Vous utilisez vos utilisateurs comme infrastructure de test.** Ils n'ont pas signé pour ça.

## Le spectre des évaluations (et où la plupart des équipes se trompent)

Les approches d'évaluation s'inscrivent sur un spectre allant des méthodes « rapides mais fragiles » aux méthodes « coûteuses mais valides ».

<figure class="breakout">

![Un diagramme comparant les vérifications déterministes, LLM-en-tant-que-juge et l'évaluation humaine en fonction de la vitesse, du coût et de la validité.](../eval-spectrum.svg)

<figcaption>Utilisez la méthode d'évaluation la moins coûteuse capable de détecter honnêtement l'échec.</figcaption>
</figure>

**LLM-en-tant-que-juge** est le chouchou actuel : demandez à un modèle puissant de noter les sorties d'un autre modèle. Rapide, échelonnable, peu coûteux. Le problème : cela intègre les biais du modèle correcteur, peut être manipulé et crée une dépendance circulaire. Si vous utilisez GPT-5 pour noter les sorties de GPT-5, vous mesurez quelque chose comme « dans quelle mesure GPT-5 est d'accord avec GPT-5 ». Ce n'est pas rien, mais ce n'est pas ce que vous croyez.

**L'évaluation humaine** est le standard-or que tout le monde tente d'éviter. Faire évaluer les sorties par des humains est coûteux, lent, incohérent entre les évaluateurs et ennuyeux à organiser. Mais c'est la seule façon de valider si votre système est utile aux vrais humains.

**Vérifications automatisées spécifiques aux tâches** sont là où la plupart des équipes devraient passer plus de temps. Elles ne sont pas glamour, mais elles sont rapides, déterministes et liées à ce qui compte dans votre système.

## Ce qui fonctionne vraiment  

### 1. Définir l'échec avant le déploiement  

Avant de modifier un modèle ou un prompt, notez ce à quoi ressemble un mauvais résultat. Spécifiquement.  
Pas « la sortie doit être précise ». Ce n'est pas un test. Plutôt :

- La sortie au format JSON structuré doit être analysable sans erreur  
- Toutes les citations dans la réponse doivent apparaître verbatim dans le contexte récupéré  
- Les réponses ne doivent pas mentionner les noms des produits concurrents  
- Les requêtes SQL doivent être syntaxiquement valides et ne faire référence qu'aux tables existant dans le schéma  
- La classification de sentiment ne doit pas passer du positif au négatif plus de 3 % du temps sur l'ensemble de test existant  

Vous pouvez vérifier ces points de manière programmable. Aucun modèle juge n'est nécessaire.  

**Harnais d'évaluation : vérifications déterministes**  

```typescript
type EvalResult = { passed: boolean; reason?: string };

const evals: Record<string, (output: string, context: EvalContext) => EvalResult> = {
  // JSON must parse
  validJson: (output) => {
    try {
      JSON.parse(output);
      return { passed: true };
    } catch (e) {
      return { passed: false, reason: `Invalid JSON: ${e.message}` };
    }
  },

  // No hallucinated citations — every claim must appear in context
  groundedCitations: (output, { retrievedChunks }) => {
    const claims = extractCitations(output);
    const ungrounded = claims.filter(
      (claim) => !retrievedChunks.some((chunk) => chunk.includes(claim))
    );
    return ungrounded.length === 0
      ? { passed: true }
      : { passed: false, reason: `Ungrounded claims: ${ungrounded.join(', ')}` };
  },

  // Response length sanity check — catch truncation or runaway generation
  reasonableLength: (output) => {
    const words = output.split(/\s+/).length;
    return words >= 10 && words <= 2000
      ? { passed: true }
      : { passed: false, reason: `Word count ${words} out of bounds` };
  },
};
```

### 2. Créer un ensemble d'or à partir de vos pires jours  

Vos meilleures données d'évaluation sont les situations embarrassantes : les sorties qui ont poussé quelqu'un à ouvrir un ticket, à capturer une hallucination en capture d'écran, ou à arrêter discrètement d'utiliser la fonctionnalité.  

À chaque fois qu'un utilisateur signale une sortie incorrecte, signale une hallucination, ou que vous remarquez manuellement une échec, ajoutez-le à votre ensemble d'or : l'entrée, le contexte, et le comportement correct. Conservez 50 à 100 cas et exécutez-les à chaque modification de modèle.

Cela semble manuel au début. Après six mois, vous disposez d'une série de tests que aucun benchmark public ne peut tricher, car chaque cas provient de votre propre historique d'échecs.

<figure class="breakout">

![Un diagramme de flux illustrant comment les incidents problématiques en production deviennent des cas d'or, puis des exécutions d'évaluations CI, puis des régressions bloquées ou des versions approuvées.](../golden-set-lifecycle.svg)

<figcaption>Un ensemble d'or transforme les situations embarrassantes en un ensemble de tests de régression.</figcaption>
</figure>

**Forme d'un cas d'or**

```typescript
interface GoldenCase {
  id: string;
  input: string;
  context: Record<string, unknown>;
  expectedBehavior: {
    mustContain?: string[];
    mustNotContain?: string[];
    structureCheck?: (output: string) => boolean;
    minSimilarityToReference?: number; // similarité cosinus avec une réponse de référence
  };
  sourceIncident?: string; // lien vers le rapport de bogue ou le ticket
}
```

### 3. Tests de régression, pas seulement de validation

La plupart des équipes exécutent des évaluations uniquement lorsqu'elles envisagent un changement de modèle. C'est le test de validation : « est-ce que cette nouvelle chose est suffisamment bonne ? »

Vous avez également besoin de tests de régression : « a-t-elle cassé quelque chose qui fonctionnait auparavant ? »

Exécutez votre ensemble d'or à chaque modification de prompt, pas seulement à chaque changement de modèle. Un prompt qui fonctionnait correctement peut se dégrader silencieusement lorsque vous ajoutez un nouvel outil, modifiez une stratégie de récupération RAG ou mettez à jour votre modèle de contexte. Vous ne saurez pas sans une ligne de base. Des outils comme [Langfuse](https://langfuse.com/) associent des scores d'évaluation aux traces de production, de sorte que les régressions apparaissent dans les tableaux de bord, et non seulement dans les rapports d'incidents.

<details>
<summary>Harness d'évaluation : comparaison entre la version de référence et la candidate</summary>

```typescript
async function compareModelVersions(
  goldenCases: GoldenCase[],
  baselinePipeline: Pipeline,
  candidatePipeline: Pipeline
) {
  const results = await Promise.all(
    goldenCases.map(async (tc) => {
      const [baseline, candidate] = await Promise.all([
        baselinePipeline.run(tc.input, tc.context),
        candidatePipeline.run(tc.input, tc.context),
      ]);

      return {
        id: tc.id,
        baselinePassed: runEvals(baseline, tc.expectedBehavior),
        candidatePassed: runEvals(candidate, tc.expectedBehavior),
        regression: /* la version de référence a réussi */ && /* la candidate a échoué */,
        improvement: /* la version de référence a échoué */ && /* la candidate a réussi */,
      };
    })
  );

  const regressions = results.filter((r) => r.regression);
  const improvements = results.filter((r) => r.improvement);

  console.log(`Régressions : ${regressions.length} / ${goldenCases.length}`);
  console.log(`Améliorations : ${improvements.length} / ${goldenCases.length}`);

  if (regressions.length > 0) {
    console.error('Régressions bloquantes détectées :');
    regressions.forEach((r) => console.error(` - ${r.id}`));
  }

  return { regressions, improvements };
}
```

</details>

Si une candidate régresse sur des échecs connus, la discussion d'upgrade devient délicatement spécifique : quels cas ont été améliorés, quels cas ont été cassés, et si le compromis vaut la peine d'être fait.

### 4. Utilisez LLM-as-Juge pour Exactement Une Chose

LLM-en-juge est utile pour les sorties non déterministes où il n'existe pas de réponse correcte : « est-ce que cette réponse est utile ? », « cette synthèse préserve-t-elle les points clés ? », « cette explication est-elle adaptée à un débutant ? »

Utilisez-le là. N'utilisez-le pas pour les réponses déterministes. Quand vous l'utilisez, rendez le barème d'évaluation explicite :

**Juge basé sur une grille d'évaluation**

```typescript
async function judgeHelpfulness(
  userQuery: string,
  modelResponse: string
): Promise<{ score: number; reasoning: string }> {
  const judgePrompt = `
You are evaluating a customer support response.

User question: ${userQuery}
Response: ${modelResponse}

Rate the response on a scale of 1-5:
5 = Directly answers the question with accurate, actionable information
4 = Answers the question but could be more specific or actionable
3 = Partially addresses the question; key information is missing
2 = Tangentially related but doesn't answer the question
1 = Off-topic, factually wrong, or harmful

Respond with JSON: {"score": <number>, "reasoning": "<one sentence>"}
`;

  const result = await judgeModel.generate(judgePrompt);
  return JSON.parse(result);
}
```

Une grille d'évaluation explicite réduit la variance des évaluateurs, vous donne des résultats interprétables et facilite l'audit lorsque le juge se trompe. Des bibliothèques comme [Autoevals](https://github.com/braintrustdata/autoevals) et [Braintrust](https://www.braintrust.dev/) proposent des grilles d'évaluation prêtes à l'emploi pour des tâches courantes — utiles à recopier avant de les réinventer de zéro.

## Outils à Connaître

Vous n'avez pas besoin de tout construire depuis zéro. Plusieurs outils ont fait des progrès significatifs sur le problème de l'infrastructure d'évaluation :

**[Braintrust](https://www.braintrust.dev/)** — Plateforme d'évaluation complète avec suivi des expériences, gestion des jeux de données et fonctions de notation. Organise les exécutions d'évaluation par invite, modèle et déploiement pour que vous puissiez comparer la qualité dans le temps, pas seulement entre les versions. Fonctionne bien avec leur bibliothèque open-source **[Autoevals](https://github.com/braintrustdata/autoevals)**, qui fournit des fonctions de notation évaluées par modèle pour des tâches courantes (exactitude factuelle, utilité, toxicité, similarité sémantique).

**[Langfuse](https://langfuse.com/)** — Observabilité open-source des modèles LLM située entre votre application et vos modèles. Traque chaque appel, attache des scores d'évaluation (humains ou automatisés) à des spans individuels, et met en évidence les tendances de qualité sur le trafic de production. Bon choix si vous souhaitez une observabilité et des évaluations dans le même outil plutôt qu'un harnais d'évaluation séparé.

**[Evalite](https://www.evalite.dev/)** — Framework d'évaluation natif TypeScript par Matt Pocock. Peu de cérémonie : définissez une tâche, définissez un évaluateur, exécutez-le dans votre configuration de test existante. Cible les équipes souhaitant des évaluations qui ressemblent à des tests unitaires plutôt qu'à une plateforme d'expériences ML distincte.

**[promptfoo](https://www.promptfoo.dev/)** — Exécuteur d'évaluations CLI-first axé sur la comparaison de prompts et le red teaming. Facile à configurer via YAML, s'intègre à la plupart des fournisseurs de modèles, et inclut un support intégré pour détecter les injections de prompt et autres entrées adverses.

**[deepeval](https://docs.confident-ai.com/)** — Framework d'évaluation Python avec une vaste bibliothèque de métriques intégrées (G-Eval, fidélité RAG, pertinence des réponses, détection des hallucinations). Utile pour les pipelines RAG où vous souhaitez un classement spécifique sur la qualité du recouvrement, pas seulement sur la génération.

Le bon outil dépend de votre stack et de votre point de départ. Ce qui compte plus que le choix du framework, c'est la discipline d'exécuter des évaluations de manière cohérente — systématiquement, à chaque changement significatif.

## La Partie Désagréable

La plupart des équipes sautent cette étape car elle pose une question agaçante dès le départ : quel serait le « bon » ici ?

C'est vraiment difficile pour une nouvelle fonctionnalité d'IA. C'est aussi non négociable si vous tenez à la fiabilité. Les équipes qui déployent une IA fiable font la même chose qu'elles le feraient pour tout chemin de code critique : définir le comportement attendu, le tester et exécuter ces tests de manière continue.

Les benchmarks ne mentent pas. Ils répondent à une autre question. Arrêtez de les lire comme des roadmaps produit et commencez à écrire des tests qui correspondent à votre système.

Vos utilisateurs s'en apercevront avant que vos tableaux de bord ne le fassent. Construisez d'abord le jeu de tests.
