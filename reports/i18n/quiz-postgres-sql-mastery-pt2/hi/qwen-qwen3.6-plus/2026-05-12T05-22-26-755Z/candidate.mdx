  <div className="explanation">
हालाँकि `SERIAL` का उपयोग अभी भी आम है, लेकिन **SQL Standard** के अनुसार सही तरीका यह है:
- `id INT IDENTITY(1,1)` SQL Server सिंटैक्स है।
- `id INT AUTO_INCREMENT` MySQL सिंटैक्स है।
- `id INTEGER AUTO_INCREMENT PRIMARY KEY` MySQL सिंटैक्स है।


इसके `SERIAL` पर कई स्पष्ट फायदे हैं:
- यह SQL Standard के पूर्णतः अनुपालन में है
- सीक्वेंस ओनरशिप का बेहतर प्रबंधन
- मैनुअल इंसर्ट्स के लिए स्पष्ट सेमेंटिक्स
- रेप्लिकेशन के साथ अधिक सुगमता से काम करता है

वास्तव में, `SERIAL` केवल एक सीक्वेंस बनाने और डिफ़ॉल्ट वैल्यू सेट करने का एक शॉर्टहैंड है, जो डंप और रिस्टोर प्रक्रियाओं में समस्याएँ पैदा कर सकता है।
```sql
    id int GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY
```
- `id INT IDENTITY(1,1)` is SQL Server syntax.
    - `id INT AUTO_INCREMENT` is MySQL syntax.
    - `id INTEGER AUTO_INCREMENT PRIMARY KEY` is MySQL syntax.


    This has several advantages over `SERIAL`:
    - It's SQL Standard compliant
    - Better handling of sequence ownership
    - Clearer semantics around manual inserts
    - Works better with replication

    `SERIAL` is actually just shorthand for creating a sequence and setting a default value, which can lead to issues with dumps and restores.

मेरे लिए यहाँ सबसे आश्चर्यजनक बात यह है कि एक मल्टी-कॉलम इंडेक्स का उपयोग तब भी किया जा सकता है जब क्वेरी टेक्स्ट में कॉलम का क्रम इंडेक्स डेफिनिशन से मेल नहीं खाता।

**नोट:** B-tree इंडेक्स में, पहला कॉलम (leading column) सबसे मायने रखता है। चूंकि यह क्वेरी `grade_level` और `last_name` के आधार पर फ़िल्टर करती है, इसलिए `grade_level` से शुरू होकर `last_name` वाला इंडेक्स सबसे उपयुक्त रहेगा।

SQL जितनी जल्दी कम से कम पंक्तियों (rows) को चुन लेगा, प्रदर्शन उतना ही बेहतर होगा।

उदाहरण के लिए, यदि आपके पास एक `task` टेबल है और आप जानते हैं कि `status`, `created_at` की तुलना में अधिक चयनात्मक (selective) है, तो आप `status = 'done' AND created_at > '2024-01-01'` जैसी क्वेरी के लिए `status` से शुरू होने वाले इंडेक्स को चुन सकते हैं। `WHERE` क्लॉज़ के टेक्स्ट में शर्तों का क्रम मायने नहीं रखता; मायने रखता है इंडेक्स कॉलम का क्रम और प्रेडिकेट का स्वरूप।

B+Tree इंडेक्स का उपयोग इनके लिए किया जा सकता है:
- पहले कॉलम, या कॉलम के लेफ्टमोस्ट प्रीफ़िक्स (leftmost prefix) के लिए।
- क्वेरी के स्वरूप के आधार पर `=`, `>`, `<`, `BETWEEN`, और `IN` जैसे ऑपरेटर्स के लिए।
- प्रीफ़िक्स मैचिंग, जैसे `LIKE 'prefix%'`।

    एक और समस्या, `ORDER BY RANDOM() > 10` रैंडम नंबर के बजाय बूलियन वैल्यू पर सॉर्ट करेगा। लेकिन वह `> 10` वाकई परसेंटेज जैसा लग रहा था, है ना?
    4. `BERNOULLI` वैध है, हालाँकि `SAMPLETABLE` नहीं है।
    5. `WHERE RANDOM() >= 0.1` एक ट्रिक है! `RANDOM()` `0.0 <= x < 1.0` रेंज में वैल्यू रिटर्न करता है, इसलिए `>= 0.1` लगभग 90% रो रिटर्न करेगा, 10% नहीं।

    `TABLESAMPLE` मेथड की बात करें तो, PostgreSQL दो सैंपलिंग मेथड्स सपोर्ट करता है:
    - `BERNOULLI`: हर रो की संभावना बराबर होती है
    - `SYSTEM`: ब्लॉक-लेवल सैंपलिंग (तेज़ लेकिन कम रैंडम)

    सैंपलिंग के बारे में और जानने के लिए, [Render](https://render.com/blog/postgresql-random-samples-big-tables) का यह आर्टिकल चेक करें।

    उदाहरण:
```sql
    -- Sample 10% of rows
    SELECT * FROM students TABLESAMPLE BERNOULLI (10);

    -- For repeatable results, use REPEATABLE:
    SELECT * FROM students
    TABLESAMPLE BERNOULLI (10) REPEATABLE (42);
```
यह भी ध्यान देने योग्य है कि `TABLESAMPLE` [बहुत रैंडम नहीं होने की रिपोर्ट](https://stackoverflow.com/questions/8674718/best-way-to-select-random-rows-postgresql) दी गई है। यदि आपको _अधिक_ रैंडम फिक्स्ड-साइज़ सैंपल चाहिए, तो `ORDER BY RANDOM() LIMIT 10` का उपयोग करने पर विचार करें। लेकिन सावधान रहें, यह 10% नहीं बल्कि 10 रो रिटर्न करता है, और बड़ी टेबल्स पर यह धीमा हो सकता है।

    कुछ थर्ड-पार्टी एक्सटेंशन भी हैं जो बेहतर रैंडम सैंपलिंग मेथड्स प्रदान करते हैं, लेकिन वह इस क्विज़ के दायरे से बाहर है! 😅
  </div>
  </slot>
</Challenge>
