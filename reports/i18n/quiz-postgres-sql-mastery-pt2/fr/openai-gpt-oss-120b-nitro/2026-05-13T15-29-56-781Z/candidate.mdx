  <div className="explanation">
    Bien que `SERIAL` soit encore couramment utilisé, la façon **SQL Standard** est :
    ```sql

    ```
    - `id INT IDENTITY(1,1)` est la syntaxe SQL Server.
    - `id INT AUTO_INCREMENT` est la syntaxe MySQL.

    - Meilleure gestion de la propriété de la séquence
    - Sémantique plus claire pour les insertions manuelles
    - Fonctionne mieux avec la réplication

    `SERIAL` n'est en fait qu'un raccourci pour créer une séquence et définir une valeur par défaut, ce qui peut poser des problèmes lors des exportations et restaurations.

  </div>

  <slot name='explanation'>
  <div className="explanation">
    Pour moi, la chose surprenante ici est qu'un index multi‑colonnes peut être utilisé même lorsque l'ordre des colonnes dans le texte de la requête ne correspond pas à la définition de l'index.

    **Note :** Avec les index B‑tree, la colonne principale compte. Puisque cette requête filtre sur `grade_level` et `last_name`, un index commençant par `grade_level` puis `last_name` convient parfaitement.

    Plus SQL peut sélectionner le moins de lignes possible, meilleure est la performance.

    Par exemple, si vous avez une table `task` et que vous savez que `status` est plus sélectif que `created_at`, vous pourriez choisir un index commençant par `status` pour une requête comme `status = 'done' AND created_at > '2024-01-01'`. Ce n'est pas l'ordre des conditions dans le texte du `WHERE` qui importe ; c'est l'ordre des colonnes de l'index et la forme des prédicats.

    Un index B+Tree peut être utilisé pour :
    - La colonne principale, ou un préfixe gauche des colonnes.
    - Des opérateurs tels que `=`, `>`, `<`, `BETWEEN` et `IN`, selon la forme de la requête.
    - La correspondance de préfixe, `LIKE 'prefix%'`.

    Merci à [u/mwdb2](https://www.reddit.com/r/PostgreSQL/comments/1h6kw87/comment/m0fovn0/?utm_source=share&utm_medium=web3x&utm_name=web3xcss&utm_term=1&utm_content=share_button) d'avoir signalé une erreur antérieure dans cette explication !

    Comment spécifier un **nom de colonne contenant un point ?**
    ```sql
        CREATE TABLE teachers (
          id INT GENERATED BY DEFAULT AS IDENTITY,

          -- How do you include the column: first.name?
          salary NUMERIC
        );
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    PostgreSQL utilise les guillemets doubles pour citer les identifiants :
    - Guillemets doubles `"..."` pour les identifiants (noms de colonnes, noms de tables)
    - Guillemets simples `'...'` pour les littéraux de chaîne
    - La forme `first.name` est interprétée comme `table_name.column_name`

    Sans guillemets, un point dans un nom de colonne serait interprété comme un séparateur de schéma/table ! De plus, tous les identifiants non cités sont mis en minuscules par défaut.

    Pour éviter les mauvaises surprises, il vaut mieux utiliser le `snake_case` et éviter les caractères spéciaux.

<Challenge
  client:visible={{rootMargin: "150px"}}

  group="Échantillonnage de données"
  title="Syntaxe TABLESAMPLE"
  options={[
  ]}
>
  <slot name="question">
  <div className="question">
    Quelle est la syntaxe valide pour **échantillonner aléatoirement 10 %** des étudiants ?
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Celle‑ci est délicate ! Elle comporte environ 6 pièges, dont :

    1. Description de haut niveau + fragments de syntaxe. Teste votre capacité à adapter et assembler des informations incomplètes.
    2. Nécessite une bonne connaissance de la syntaxe `TABLESAMPLE`, car aucune des options `TABLESAMPLE` n’utilise la bonne syntaxe ! (Sinistre ! Je sais ! Continuez, je crois en vous !!! ❤️)
    3. Les expressions `ROW_NUMBER() OVER (ORDER BY RAND())` et `ORDER BY RANDOM() > 10` sont de fausses pistes. Elles semblent correctes, mais ne le sont pas. `RAND()` n’existe pas. Et `RANDOM() > 10` est une expression booléenne qui sera toujours fausse puisque `RANDOM()` renvoie toujours une valeur entre `0.0` et `1.0`.
    Un autre problème : `ORDER BY RANDOM() > 10` trierait sur une valeur booléenne, pas sur un nombre aléatoire. Mais ce `> 10` donne l’impression d’être un pourcentage, non ?
    4. `BERNOULLI` est valide, cependant `SAMPLETABLE` ne l’est pas.
    5. `WHERE RANDOM() >= 0.1` est un leurre ! `RANDOM()` renvoie une valeur dans l’intervalle `0.0 <= x < 1.0`, donc `>= 0.1` retournerait environ 90 % des lignes, pas 10 %.

    Concernant la méthode `TABLESAMPLE`, PostgreSQL supporte deux méthodes d’échantillonnage :
    - `BERNOULLI` : chaque ligne a une probabilité égale
    - `SYSTEM` : échantillonnage au niveau des blocs (plus rapide mais moins aléatoire)

    Pour en savoir plus sur l’échantillonnage, consultez cet article de [Render](https://render.com/blog/postgresql-random-samples-big-tables).

    Exemple :
    ```sql
        -- Sample 10% of rows
        SELECT * FROM students TABLESAMPLE BERNOULLI (10);

        -- For repeatable results, use REPEATABLE:
        SELECT * FROM students
        TABLESAMPLE BERNOULLI (10) REPEATABLE (42);
    ```
    Il vaut aussi la peine de mentionner que `TABLESAMPLE` [a été signalé](https://stackoverflow.com/questions/8674718/best-way-to-select-random-rows-postgresql) comme n’étant pas très aléatoire. Si vous avez besoin d’un échantillon fixe _plus_ aléatoire, envisagez d’utiliser `ORDER BY RANDOM() LIMIT 10`. Mais attention, cela renvoie 10 lignes, pas 10 %, et peut être lent sur de grandes tables.

    Il existe également des extensions tierces qui offrent de meilleures méthodes d’échantillonnage aléatoire, mais cela dépasse le cadre de cet exercice ! 😅
  </div>
  </slot>
</Challenge>

  group="Comportement de NULL"
  title="COUNT et NULL"
  options={[
    {text: '100'},
    {text: '90'},
    {text: '10'},
    {text: '1', isAnswer: true},
    {text: '0'},
  ]}
>
  <slot name="question">
  <div className="question">
    Combien de *lignes* cela renverra-t-il ?
    ```sql
        SELECT COUNT(*)
        FROM students
        WHERE grade = NULL;
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Ok, c'était un peu piégeux. Le truc, c'est que `x = NULL` ne correspondra jamais à aucune ligne, et il renverra TOUJOURS une ligne avec un count de `0`.

    Comme `NULL` est une valeur inconnue, on ne peut la comparer à quoi que ce soit, même pas à elle-même.

    La bonne façon de vérifier les valeurs `NULL` est d'utiliser `IS NULL`.

    Exemple :
    ```sql
        postgres=# SELECT count(*)
          FROM students
          WHERE age = null;
        count 
        -------
            0
        (1 row)

        postgres=# 
        postgres=# SELECT count(*)
          FROM students
          WHERE age is null;
        count 
        -------
            3
        (1 row)
    ```
  </div>
  </slot>
</Challenge>
