Auch wenn `SERIAL` noch häufig verwendet wird, lautet die korrekte **SQL-Standard**-Syntax:
- `id INT IDENTITY(1,1)` ist SQL Server-Syntax.
    - `id INT AUTO_INCREMENT` ist MySQL-Syntax.
    - `id INTEGER AUTO_INCREMENT PRIMARY KEY` ist MySQL-Syntax.


    Das bietet mehrere Vorteile gegenüber `SERIAL`:
    - Es ist SQL-Standard-konform
    - Bessere Handhabung der Sequenz-Zugehörigkeit
    - Klarere Semantik bei manuellen INSERTs
    - Funktioniert zuverlässiger mit Replikation

    `SERIAL` ist im Grunde nur syntaktischer Zucker, der eine Sequenz erstellt und einen Standardwert setzt. Das kann bei Dumps und Restores schnell zu Problemen führen.
```sql
    id int GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY
```
- `id INT IDENTITY(1,1)` is SQL Server syntax.
    - `id INT AUTO_INCREMENT` is MySQL syntax.
    - `id INTEGER AUTO_INCREMENT PRIMARY KEY` is MySQL syntax.


    This has several advantages over `SERIAL`:
    - It's SQL Standard compliant
    - Better handling of sequence ownership
    - Clearer semantics around manual inserts
    - Works better with replication

    `SERIAL` is actually just shorthand for creating a sequence and setting a default value, which can lead to issues with dumps and restores.

Welcher Index ist für diese Abfrage besser geeignet?

  </div>

  <slot name='explanation'>
  <div className="explanation">
Das Überraschende hier ist für mich, dass ein mehrspaltiger Index auch dann genutzt werden kann, wenn die Spaltenreihenfolge im Abfragetext nicht mit der Indexdefinition übereinstimmt.

    **Hinweis:** Bei B-Tree-Indizes ist die führende Spalte entscheidend. Da diese Abfrage nach `grade_level` und `last_name` filtert, ist ein Index, der mit `grade_level` und dann `last_name` beginnt, eine gute Wahl.

    Je schneller SQL die kleinstmögliche Zeilenmenge auswählen kann, desto besser ist die Performance.

    Wenn du beispielsweise eine `task`-Tabelle hast und weißt, dass `status` selektiver ist als `created_at`, würdest du für eine Abfrage wie `status = 'done' AND created_at > '2024-01-01'` wahrscheinlich einen Index wählen, der mit `status` beginnt. Entscheidend ist nicht die Reihenfolge der Bedingungen im `WHERE`-Klausel-Text, sondern die Spaltenreihenfolge des Index und die Form der Prädikate.

    Ein B+Tree-Index kann genutzt werden für:
    - Die führende Spalte oder ein Präfix der linken Spalten.
    - Operatoren wie `=`, `>`, `<`, `BETWEEN` und `IN`, abhängig von der Abfragestruktur.
    - Präfix-Matching, z. B. `LIKE 'prefix%'`.

    Danke an [u/mwdb2](https://www.reddit.com/r/PostgreSQL/comments/1h6kw87/comment/m0fovn0/?utm_source=share&utm_medium=web3x&utm_name=web3xcss&utm_term=1&utm_content=share_button) für den Hinweis auf einen früheren Fehler in dieser Erklärung!

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={8}
  group="Datensampling"
  title="TABLESAMPLE-Syntax"
  options={[
  ]}
>
  <slot name="question">
  <div className="question">
Welche Syntax ist gültig, um **10 % der Studenten zufällig auszuwählen**?
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
Diese Frage ist knifflig! Sie enthält etwa 6 Fallen, darunter:

    1. Eine grobe Beschreibung + Syntax-Fragmente. Testet, wie du dich anpasst und unvollständige Informationen zusammensetzt.
    2. Erfordert fundierte Kenntnisse der `TABLESAMPLE`-Syntax, da keine der `TABLESAMPLE`-Optionen die korrekte Syntax verwendet! (Hinterhältig! Ich weiß! Mach weiter, ich glaube an dich!!! ❤️)
    3. `ROW_NUMBER() OVER (ORDER BY RAND())` und `ORDER BY RANDOM() > 10` sind weitere Ablenkungsmanöver. Sie sehen korrekt aus, sind es aber nicht. `RAND()` existiert nicht. Und `RANDOM() > 10` ist ein boolescher Ausdruck, der immer `false` ergibt, da `RANDOM()` immer im Bereich `0.0-1.0` liegt.
    Ein weiteres Problem: `ORDER BY RANDOM() > 10` würde zudem nach einem booleschen Wert sortieren, nicht nach einer Zufallszahl. Aber dieses `> 10` fühlte sich doch irgendwie nach einem Prozentsatz an, nicht wahr?
    4. `BERNOULLI` ist gültig, `SAMPLETABLE` jedoch nicht.
    5. `WHERE RANDOM() >= 0.1` ist ein Trick! `RANDOM()` gibt einen Wert im Bereich `0.0 <= x < 1.0` zurück, sodass `>= 0.1` etwa 90 % der Zeilen zurückgeben würde, nicht 10 %.

    Was die `TABLESAMPLE`-Methode betrifft, unterstützt PostgreSQL zwei Sampling-Verfahren:
    - `BERNOULLI`: Jede Zeile hat die gleiche Wahrscheinlichkeit
    - `SYSTEM`: Blockweises Sampling (schneller, aber weniger zufällig)

    Um mehr über Sampling zu erfahren, schau dir diesen Artikel von [Render](https://render.com/blog/postgresql-random-samples-big-tables) an.

    Beispiel:
```sql
    -- Sample 10% of rows
    SELECT * FROM students TABLESAMPLE BERNOULLI (10);

    -- For repeatable results, use REPEATABLE:
    SELECT * FROM students
    TABLESAMPLE BERNOULLI (10) REPEATABLE (42);
```
Erwähnenswert ist auch, dass `TABLESAMPLE` [als nicht besonders zufällig gemeldet wurde](https://stackoverflow.com/questions/8674718/best-way-to-select-random-rows-postgresql). Wenn du eine _zufälligere_ Stichprobe fester Größe benötigst, erwäge die Verwendung von `ORDER BY RANDOM() LIMIT 10`. Beachte jedoch, dass dies 10 Zeilen und nicht 10 % zurückgibt und bei großen Tabellen langsam sein kann.

    Es gibt auch einige Drittanbieter-Erweiterungen, die bessere Verfahren für zufällige Stichproben bieten, aber das sprengt hier den Rahmen! 😅
  </div>
  </slot>

  title="COUNT und NULL"
  options={[
    {text: '100'},
    {text: '90'},
    {text: '10'},
    {text: '1', isAnswer: true},
    {text: '0'},
  ]}
>
  <slot name="question">
  <div className="question">
Wie viele *Zeilen* wird das zurückgeben?
```sql
    SELECT COUNT(*)
    FROM students
    WHERE grade = NULL;
```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
Ok, das war ein bisschen knifflig. Der Knackpunkt ist, dass `x = NULL` niemals auf Zeilen zutrifft und IMMER eine einzelne Zeile mit dem Zählergebnis `0` zurückgibt.

    Da `NULL` einen unbekannten Wert darstellt, kann er mit nichts verglichen werden – nicht einmal mit sich selbst.
    
    Der korrekte Weg, um auf `NULL`-Werte zu prüfen, ist die Verwendung von `IS NULL`.

    Beispiel:
```sql
    postgres=# SELECT count(*)
      FROM students
      WHERE age = null;
    count 
    -------
        0
    (1 row)

    postgres=# 
    postgres=# SELECT count(*)
      FROM students
      WHERE age is null;
    count 
    -------
        3
    (1 row)
```
  </div>
  </slot>
