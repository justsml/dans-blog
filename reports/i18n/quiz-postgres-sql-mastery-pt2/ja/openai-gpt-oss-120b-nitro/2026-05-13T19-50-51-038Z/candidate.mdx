  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `SERIAL` はまだ一般的に使われていますが、**SQL Standard** の方法は次のとおりです：
    - `id INT IDENTITY(1,1)` は SQL Server の構文です。
    - `id INT AUTO_INCREMENT` は MySQL の構文です。
    - `id INTEGER AUTO_INCREMENT PRIMARY KEY` は MySQL の構文です。


    これには `SERIAL` よりいくつかの利点があります：
    - SQL Standard に準拠している
    - シーケンス所有権の取り扱いが改善される
    - 手動挿入時の意味が明確になる
    - レプリケーションでの動作が向上する

    `SERIAL` は実際にはシーケンスを作成しデフォルト値を設定するための省略形で、ダンプやリストア時に問題を引き起こすことがあります。
    ```sql
        id int GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY
    ```
    - `id INT IDENTITY(1,1)` is SQL Server syntax.
    - `id INT AUTO_INCREMENT` is MySQL syntax.
    - `id INTEGER AUTO_INCREMENT PRIMARY KEY` is MySQL syntax.


    This has several advantages over `SERIAL`:
    - It's SQL Standard compliant
    - Better handling of sequence ownership
    - Clearer semantics around manual inserts
    - Works better with replication

    `SERIAL` is actually just shorthand for creating a sequence and setting a default value, which can lead to issues with dumps and restores.

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={8}
  group="データサンプリング"
  title="TABLESAMPLE 構文"
  options={[
  ]}
>
  <slot name="question">
  <div className="question">
    学生を **ランダムに 10% サンプリング** する有効な構文はどれですか？
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    これはちょっとややこしいです！およそ 6 つの罠があり、例えば：

    1. 高レベルの説明と構文の断片が混在。不完全な情報をどう組み立てるかをテストします。
    2. `TABLESAMPLE` の構文に慣れている必要がありますが、提示された `TABLESAMPLE` の選択肢はすべて正しくありません！(怖いですよね！でも頑張ってください!!! ❤️)
    3. `ROW_NUMBER() OVER (ORDER BY RAND())` と `ORDER BY RANDOM() > 10` は赤鯛です。正しそうに見えますが、`RAND()` は存在しません。また `RANDOM() > 10` は常に false になるブール式です。`RANDOM()` の範囲は `0.0-1.0` です。
    さらに `ORDER BY RANDOM() > 10` はブール値でソートしようとしており、パーセンテージとは関係ありません。
    4. `BERNOULLI` は有効ですが、`SAMPLETABLE` は無効です。
    5. `WHERE RANDOM() >= 0.1` はトリックです！`RANDOM()` は `0.0 <= x < 1.0` の範囲を返すので、`>= 0.1` は約 90% の行を取得し、10% にはなりません。

    PostgreSQL がサポートする `TABLESAMPLE` のサンプリング手法は次の 2 つです：
    - `BERNOULLI`: 各行が等しい確率で選ばれる
    - `SYSTEM`: ブロック単位のサンプリング（高速だがランダム性は低い）

    サンプリングについて詳しくは、[Render の記事](https://render.com/blog/postgresql-random-samples-big-tables) を参照してください。

    例:
    ```sql
        -- Sample 10% of rows
        SELECT * FROM students TABLESAMPLE BERNOULLI (10);

        -- For repeatable results, use REPEATABLE:
        SELECT * FROM students
        TABLESAMPLE BERNOULLI (10) REPEATABLE (42);
    ```
    もうひとつ付け加えると、`TABLESAMPLE` は [報告されています](https://stackoverflow.com/questions/8674718/best-way-to-select-random-rows-postgresql) が、あまりランダムではありません。よりランダムな固定サイズのサンプルが必要なら、`ORDER BY RANDOM() LIMIT 10` を検討してください。ただし、これは 10 行を返すだけで 10% ではなく、テーブルが大きいと遅くなる可能性があります。

    なお、サードパーティ製の拡張機能でより良いランダムサンプリング手法が提供されていますが、今回は範囲外です 😅
  </div>
  </slot>
</Challenge>

  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    さて、少しトリッキーでしたね。ポイントは `x = NULL` は決して行にマッチせず、常に `0` のカウントで 1 行が返されます。

    `NULL` は未知の値なので、何とも比較できません。自分自身ですら比較できません。

    `NULL` 値をチェックする正しい方法は `IS NULL` を使うことです。

    例:
    ```sql
        postgres=# SELECT count(*)
          FROM students
          WHERE age = null;
        count 
        -------
            0
        (1 row)

        postgres=# 
        postgres=# SELECT count(*)
          FROM students
          WHERE age is null;
        count 
        -------
            3
        (1 row)
    ```

    ベストプラクティス:
    - UPDATE/DELETE/INSERT には `EXPLAIN`（ANALYZE なし）を使用
    - 変更はトランザクションでラップする:
    ```sql
        BEGIN;
        EXPLAIN ANALYZE ...;
        ROLLBACK;
    ```
