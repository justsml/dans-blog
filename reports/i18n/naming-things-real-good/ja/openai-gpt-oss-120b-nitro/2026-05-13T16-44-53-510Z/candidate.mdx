## Naming stuff: Object Oriented Basics

例を使ってオブジェクト/クラス設計を見てみましょう...

### The Situation

`data model`（コード、SQL、あるいは Excel シート）を設計したことがありますか？  
以下のようなものを見たことがありませんか？

```
*** anti-pattern - don't copy-paste ***
* User
  - id
  - avatarUrl
  - name
  - email
  - password

* Agent
  - id
  - primaryPhoto
  - name
  - email
  - agentEmail
  - agentPhoneMain
  - agentEmailPrimary
  - agentPhonePrimary
  - agentAddressFull
  - agentCompanyName
  - agentCompanyAddress
  - *userEmail* - 'Pointer' to User table ^^^
```

### Where's the bug?

実際にはバグはありません。単に再構築が必要なデータがあるだけです。

**以下のような状況に心当たりはありませんか？**

1. アプリに何らかの変更を加えるたびに、数時間に及ぶ骨の折れるデバッグが必要になる。  
2. 要件が変わるたびに、次のような事態が発生する：

![schema refactor][schema_refactor]

なぜ `agentEmailPrimary` というフィールド名が **それほど悪い** のか？

まず前提として、あなたが宇宙に全く新しい概念を作り出しているわけではありません。過度に具体的な命名にはいくつかの落とし穴があります。

1. 非常に特化した名前に「ロック」されてしまうと、`agentEmailPrimary` はビューや関連コードの **再利用性を 0%** にし、以下のような煩わしいバグが繰り返し発生します：

- テーブル間でデータが同期しない（`user.email` を `agent.agentEmail` に伝搬させるべきか、逆にすべきかが不明瞭——この「ロジック」をどこで、どのように実装するかの複雑さは別として…）
- バリデーションルールやロジックが重複し、整合性が取れなくなる可能性が高い。
- プロジェクトは次第に揺らぐ Jenga タワーのようになる。
- 些細な変更でも極端に細部への注意が必要になるため、ファイルを増やすたびに脆弱性が蓄積する。

1. `agentEmailPrimary` はいくつかの意味を持ち得る。**短い名前**で曖昧さを排除せよ。

- 不要な冗長語に注意。`Primary` だけでも余計な疑問を呼ぶ。セカンダリはあるのか？ それは主要な近親者用か？

言葉はもう十分だ、実際にどうすべきか？

### 解決策

```
// Consolidated Schema:

User
  - id
  - role: ['agent', 'lead', 'admin']
  - name
  - phone
  - address
  - email
  - password
  - company
    - name
    - address
```

`Agent` テーブルを削除した。`Agent` に固有のフィールドが存在しなかったためだ。また、`User.company` オブジェクト（`.name`、`.address` を含む）は、命名を整理した結果として一度だけ現れた。

いくつかの指針：

1. 不要なテーブルは削除する。`statuses` テーブルが本当に必要か？`User` テーブルに `status::VARCHAR(8)` カラムを追加できるのであれば、行あたりの余分なバイト数は許容できる。
2. 関連テーブルは統合を試みる。**Data**
3. 冗長なデータ収集は削除する（例：Analytics ソリューションで代替できるなら `ActivityLogs` テーブルを削除）。
4. **すべてのフィールド名は単語1つ／名詞／代名詞に抑える**。テーブルが提供するコンテキストに依存してよい。（例：`PersonalAccount.email` と `BusinessAccount.email` はテーブル名がコンテキストを提供している。）
5. `Agent.agentEmail` や `Agent.agentPhonePrimary` といった名前は **存在しない**。句点。口に出して言おう：「`email` と `phone`」だ。
6. 高度に具体的な名前を使うと、`code‑reusability` と `durability` のレベルが **ゼロパーセント** に固定される。
7. `User.profileSummaryEmail` のような無意味な命名で自分に好意を与えることはない。💞
