## Naming stuff: Object Oriented Basics

Давайте посмотрим на проектирование объектов/классов на примере…

### The Situation

Вы когда‑нибудь проектировали `data model` (в коде, SQL или Excel‑таблицах)?
Звучит ли следующее знакомо?

```
*** anti-pattern - don't copy-paste ***
* User
  - id
  - avatarUrl
  - name
  - email
  - password

* Agent
  - id
  - primaryPhoto
  - name
  - email
  - agentEmail
  - agentPhoneMain
  - agentEmailPrimary
  - agentPhonePrimary
  - agentAddressFull
  - agentCompanyName
  - agentCompanyAddress
  - *userEmail* - 'Pointer' to User table ^^^
```

### Where's the bug?

Технически багов нет, просто данные нуждаются в реорганизации.

**Звучит ли это знакомо?**

1. Любое изменение в вашем приложении потребует часы изнурительной отладки.  
2. Любые меняющиеся требования приведут к:

![schema refactor][schema_refactor]

Почему название поля `agentEmailPrimary` _так плохо_?

Во-первых, вы **не** создаёте совершенно новую сущность во вселенной. Чрезмерная специфичность имеет свои подводные камни:

1. «Заперто» в слишком конкретном имени, `agentEmailPrimary` скорее всего сделает ваши представления и связанный код **на 0 % переиспользуемым**, а также будет порождать назойливые повторяющиеся баги, такие как:

- Данные не синхронизируются между таблицами (неочевидно, нужно ли распространять `user.email` в `agent.agentEmail` или наоборот — не считая сложности ручной реализации места и способа принудительного применения этой «логики» ...)
- Правила/логика валидации, скорее всего, дублируются и несогласованы.
- С каждым шагом ваш проект всё больше напоминает шаткую башню Дженга.
- Хрупкость накапливается с каждым новым файлом, поскольку даже для тривиальных изменений требуется предельно тщательное внимание к деталям.

1. `agentEmailPrimary` может означать несколько разных вещей. Избавьтесь от двусмысленности **короче названиями**.

- Остерегайтесь бессмысленного нагромождения слов. `Primary`? Это лишь порождает новые вопросы: есть ли Secondary? Это для их основного контакта ближайшего родственника?

Хватит слов, Дэн, как это должно выглядеть на практике?

### Решение

```
// Consolidated Schema:

User
  - id
  - role: ['agent', 'lead', 'admin']
  - name
  - phone
  - address
  - email
  - password
  - company
    - name
    - address
```

Я удалил таблицу `Agent`, так как в ней не было полей, уникальных для агентов. А объект `User.company` (с полями `.name`, `.address`) появился после того, как названия были приведены в порядок.

Некоторые руководящие принципы:

1.  Удаляйте ненужные таблицы. Действительно ли вам нужна таблица `statuses`? Когда можно просто добавить поле `status::VARCHAR(8)` в таблицу `User`? Можно, небольшое увеличение размера строки не страшно.
2.  Старайтесь объединять связанные таблицы. **Data**
3.  Удаляйте избыточные коллекции данных (например, уберите таблицу `ActivityLogs`, если её заменило аналитическое решение).
4.  Старайтесь, чтобы **все имена полей** состояли из **одного слова/существительного/местоимения**. Можно полагаться на контекст, задаваемый таблицей (например, `PersonalAccount.email` vs `BusinessAccount.email` — контекст уже дан именем таблицы).
5.  Не существует `Agent.agentEmail` или `Agent.agentPhonePrimary`. Точка. Повторяйте за мной: «это `email` и `phone`».
6.  Используя чрезмерно специфичные имена, вы застываете уровень `code-reusability` и `durability` на конкретном уровне, а именно **0 %**.
7.  Вы не делаете себе одолжения, придумывая такие названия, как `User.profileSummaryEmail`. 💞
