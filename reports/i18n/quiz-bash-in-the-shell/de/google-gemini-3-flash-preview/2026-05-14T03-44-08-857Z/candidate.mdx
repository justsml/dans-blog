    Variablen in Bash werden ohne Leerzeichen um das `=` Zeichen deklariert. Zum Beispiel:

    Dies weist der Variable `name` den Wert `"Alice"` zu.

    Hinweis: `$name` wird verwendet, um den Wert einer Variable zu **referenzieren** oder zu lesen.

    Das Hinzufügen von Leerzeichen führt dazu, dass die Shell den Befehl als ein auszuführendes Programm interpretiert, was beim Setzen einer Variable nicht erwünscht ist.

    Außerdem unterscheidet Bash zwischen Groß- und Kleinschreibung, sodass `name`, `NAME` und `Name` unterschiedliche Variablen sind.

    Schließlich dürfen Variablennamen keine Leerzeichen oder Bindestriche (`-`) enthalten. Verwende stattdessen Unterstriche (`_`) oder camelCase.

    ```bash
        price="\$100"
        echo "Cost: $price"
    ```

    Die Syntax `${var/pattern/replacement}` ersetzt das erste Vorkommen von `pattern` durch `replacement`. Hier ist die Ausgabe:
    ```plaintext
        Bark meow
    ```
    Die Suche unterscheidet zwischen Groß- und Kleinschreibung. Um sowohl `bark` als auch `Bark` zu erfassen, verwende ein Muster wie `${var/[Bb]ark/Bark}` oder normalisiere den String vor dem Ersetzen.

    Um nur am Anfang des Strings zu ersetzen, verwende `${var/#pattern/replacement}`.

    Um nur am Ende des Strings zu ersetzen, verwende `${var/%pattern/replacement}`.

  <div className="question">
    Was gibt dieses Skript aus, wenn die Datei `cats.txt` EXISTIERT?
    ```bash
        if [ -e cats.txt]; then
          echo "File exists"
        else
          echo "File does not exist"
        fi
    ```
  </div>
  </slot>

  <slot name='explanation'>
  <div className="explanation">
    Hast du das fehlende Leerzeichen vor der schließenden Klammer bemerkt?

    Bash ist hier ziemlich eigen: Leerzeichen innerhalb von Klammerausdrücken sind zwingend erforderlich.

    Da durch das fehlende Leerzeichen der Befehl `[` kein schließendes `]` findet, gibt Bash eine Diagnosemeldung aus, wertet den Test als fehlgeschlagen und fährt mit dem `else`-Zweig fort.

    Die korrekte Syntax lautet:
    ```bash
        if [ -e example.txt ]; then
          echo "File exists"
        else
          echo "File does not exist"
        fi
    ```
    Hinweis: Doppelte Klammern `[[ ]]` werden für bedingte Ausdrücke **empfohlen**. [Siehe BashFAQ.](https://mywiki.wooledge.org/BashFAQ/031)

<Challenge
  client:visible={{rootMargin: "150px"}}
  index={14}
  group="Stolperfallen"
  title="Befehlssubstitution"
  options={[
    {text: '\'ls -l\''},
    {text: '% ls -l'},
    {text: '$ ls -l'},
    {text: '$(ls -l)', isAnswer: true},
    {text: '${ls -l}'},
  ]}
>
  <slot name="question">
  <div className="question">
    Welche Option führt den Befehl `ls -l` aus und gibt das Ergebnis zurück?
  </div>
  </slot>

  <slot name='explanation'>
  <div className="explanation">
    Die Syntax `$(ls -l)` führt den Befehl innerhalb der **Klammern** aus und ersetzt den Ausdruck durch die Ausgabe. Zum Beispiel:
    ```bash
        echo "Today is $(date +%F)"
        # => Today is 2029-12-31
    ```
    Die erste Option verwendet einfache Anführungszeichen `'`, **keine Backticks.** Dies verhindert die Expansion, sodass `'$(date +%F)'` einfach die Zeichenfolge `$(date +%F)` wörtlich ausgeben würde.

    Obwohl die Verwendung von Backticks (`` `ls -l` ``) zur Befehlsausführung weiterhin unterstützt wird, gilt dies mittlerweile in manchen Kontexten als Anti-Pattern. Meist wird die Verwendung von `$(command)` empfohlen, da dies die Lesbarkeit verbessert und die Konsistenz über verschiedene Shells und Versionen hinweg erhöht.

    Geschweifte Klammern `${}` werden für die Variablenexpansion verwendet, nicht für die Befehlssubstitution.

    Das Zeichen `%` wird nicht für die Befehlssubstitution verwendet.
  </div>
  </slot>
