<br /><br />

Cuando empecé este artículo (finales de 2018, actualizado 2024) asumí que terminaría con una tabla de casillas marcadas de forma mixta. Seguramente existían casos de uso específicos que justificaban [`axios`](https://www.npmjs.com/package/axios), [`request`](https://www.npmjs.com/package/request), [`r2`](https://www.npmjs.com/package/r2), [`superagent`](https://www.npmjs.com/package/superagent), [`got`](https://www.npmjs.com/package/got), etc.

Bueno, resulta que **sobreestimé la necesidad de bibliotecas HTTP de terceros**.

A pesar de haber usado `fetch` durante varios años (incluyendo tareas no triviales: subidas de archivos y soporte de errores/reintentos), aún tenía malentendidos sobre sus capacidades y límites.

`fetch` nativo no analiza automáticamente respuestas JSON ni convierte cuerpos de solicitud JSON en cadenas. Tú llamas a `response.json()` al regresar y `JSON.stringify()` al salir. Axios aún gana en esta cuestión de ergonomía; el argumento a favor de `fetch` es que a menudo un pequeño ayudante cubre esta brecha.

Bueno, veamos qué puede hacer `fetch`...

### Compatibilidad

Hasta 2022, la API `fetch` es [ampliamente compatible](https://caniuse.com/#feat=fetch) en todos los navegadores modernos y en versiones más recientes de NodeJS v18+.

Si debes soportar IE, puedes [rellenar `fetch`](https://github.com/github/fetch#browser-support) con el paquete `github/fetch` (mantenido por un excelente equipo en GitHub). Es posible retroceder hasta [IE8](https://github.com/camsong/fetch-ie8) - _A tu criterio_.

Versiones anteriores de NodeJS pueden aprovechar la API `fetch` con el paquete [`node-fetch`](https://www.npmjs.com/package/node-fetch):

```sh
npm install node-fetch
```

_Después de polyfill+node-fetch: 99,99% compatible_ ✅
