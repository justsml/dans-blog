### 機能比較

|                                                 | fetch    | axios    | request |
|-------------------------------------------------|:--------:|:--------:|:-------:|
| リクエストとレスポンスのインターセプト          |✅        |✅         |✅       |
| リクエストとレスポンスデータのトランスフォーム  |✅        |✅         |✅       |
| リクエストのキャンセル                          |✅        |✅         |❌       |
| JSONデータの自動トランスフォーム                 |手動ヘルパー|✅         |✅       |
| XSRF対策のクライアントサイドサポート           |✅        |✅         |✅       |
| 進捗                                            |✅        |✅         |✅       |
| ストリーミング                                  |✅        |✅         |✅       |
| リダイレクト                                    |✅        |✅         |✅       |

<br /><br />

この記事を執筆し始めた頃（2018年末、2024年に更新）は、最終的にチェックボックスが混在した表になるものだと思っていた。確かに、[`axios`](https://www.npmjs.com/package/axios)、[`request`](https://www.npmjs.com/package/request)、[`r2`](https://www.npmjs.com/package/r2)、[`superagent`](https://www.npmjs.com/package/superagent)、[`got`](https://www.npmjs.com/package/got) などのサードパーティ製ライブラリを正当化する特別な _ユースケース_ が存在するに違いない。

しかし結果として、**サードパーティ製HTTPライブラリに対する需要を過大評価していた**ことがわかった。

`fetch`を数年間使用してきたにもかかわらず（ファイルアップロードやエラー/リトライ対応といった実務レベルのタスクも含めて）、`fetch`の能力と制限に対する誤解はまだ残っていた。

ネイティブの `fetch` は、JSONレスポンスの自動パースやJSONリクエストボディの文字列化を行わない。受信時は `response.json()` を呼び出し、送信時は `JSON.stringify()` を使う。その点の操作性ではAxiosがまだ勝るが、`fetch` の主張は、小さなヘルパー関数でほとんど差を埋められるという点にある。

さて、`fetch` が何ができるか見ていこう。

### 互換性

2022年現在、`fetch` APIはすべての主要ブラウザおよびNodeJS v18以降で[広くサポートされています](https://caniuse.com/#feat=fetch)。

IEのサポートが必須の場合、`github/fetch`パッケージを使用して`fetch`をポリフィルできます（GitHubのチームが保守しています）。[IE8](https://github.com/camsong/fetch-ie8)まで対応させることも可能ですが、_環境や要件によって結果は異なります_。

より古いNodeJSでも、[`node-fetch`](https://www.npmjs.com/package/node-fetch)パッケージを導入すれば`fetch` APIを利用可能です：

```sh
npm install node-fetch
```

_ポリフィルとnode-fetchの併用で、互換性は99.99%に到達_ ✅
