<br /><br />

Lorsque j’ai commencé cet article (fin 2018, mis à jour 2024), je pensais finir avec un tableau de cases à cocher mélangées. Il doit bien exister des _cas d’usage_ particuliers qui justifient l’utilisation de [`axios`](https://www.npmjs.com/package/axios), [`request`](https://www.npmjs.com/package/request), [`r2`](https://www.npmjs.com/package/r2), [`superagent`](https://www.npmjs.com/package/superagent), [`got`](https://www.npmjs.com/package/got), etc.  

Eh bien, il s’avère que **j’ai surestimé le besoin de bibliothèques HTTP tierces**.  

Même après plusieurs années d’utilisation de `fetch` (y compris pour des tâches non triviales : téléversements de fichiers et prise en charge des erreurs/reprises), je conservais des idées reçues sur les capacités et les limites de `fetch`.  

Le `fetch` natif ne parse pas automatiquement les réponses JSON ni ne sérialise les corps de requête JSON. Il faut appeler `response.json()` à la réception et `JSON.stringify()` à l’envoi. Axios reste plus ergonomique sur ce point ; l’argument en faveur de `fetch` est qu’un petit helper comble généralement cet écart.  

Voyons donc ce que `fetch` peut réellement faire…

### Compatibilité

À partir de 2022, l’API `fetch` est [largement prise en charge](https://caniuse.com/#feat=fetch) par tous les navigateurs modernes et par les versions récentes de Node JS v18+.

Si vous devez prendre en charge IE, vous pouvez [polyfill fetch](https://github.com/github/fetch#browser-support) avec le paquet `github/fetch` (maintenu par une équipe formidable chez GitHub). Il est possible de remonter jusqu’à [IE8](https://github.com/camsong/fetch-ie8) – _les résultats peuvent varier_.

Les versions antérieures de Node JS peuvent exploiter l’API `fetch` via le paquet [`node-fetch`](https://www.npmjs.com/package/node-fetch) :

```sh
npm install node-fetch
```

_Après polyfill + node-fetch : 99,99 % compatible_ ✅
