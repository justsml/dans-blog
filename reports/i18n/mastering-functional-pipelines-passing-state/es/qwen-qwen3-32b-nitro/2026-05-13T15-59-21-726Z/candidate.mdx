## Maestro de tuberías: Pasar estado

¿Has tenido desafíos al pasar estado usando tuberías funcionales?

La organización (o su falta) de tu código impacta directamente en la facilidad con la que se pasa el estado.

En este artículo exploraremos una técnica efectiva para pasar estado a través de una tubería. Mientras tanto, mejoraremos la organización y legibilidad de nuestro código.

El siguiente fragmento "real" será nuestro enfoque en este artículo: Una función de checkout que acepta un `userId` y un array de `products`. Devuelve una cadena de Promesas que ejecuta 4 funciones en secuencia.

```tsx
const checkout = (userId: number, products: number[]) => {
  return getProductsSubtotal(userId, products)
    .then(subTotal => applyTaxes(userId, subTotal))
    .then(total => purchaseProducts(userId, total))
    .then(result => sendReceipt(userId, result));
};
```

Espera un momento, este código es bastante bueno, considerando las tuberías en JS.

Tiene algunos problemas sutiles que pueden combinarse en problemas más sustanciales.

Un problema es que estamos pasando repetidamente `userId` a cada función (lógicamente relacionada).  
Ahora combina eso con otro problema que fácilmente se le escapa a los desarrolladores y también a TypeScript: invertir los argumentos numéricos crea fácilmente un error silencioso. (Ve `applyTaxes` y `purchaseProducts`. _¿Fue `userId` o `amount` lo que va primero?_)

Antes de decidir cómo mejorar este código, identifiquemos algunas ventajas y desventajas.

### Ventajas y desventajas

#### Ventajas

- ¡Uso adecuado de un cierre! Pasar `userId` y `products` una sola vez.
- Nomenclatura consistente de argumentos.
- Composición relativamente efectiva y concisa de 4 funciones clave para el checkout.
- Control del flujo de errores "gratuito". (Los errores se propagan desde cualquier función anidada, rechazando la Promesa devuelta por `checkout()`.)

#### Desventajas

- Pasar repetidamente `userId` es tedioso.
- Las funciones no son de un solo parámetro (es decir, unarias). _Esto afecta la componibilidad. Vea [el ejemplo final](#checkout-with-further-improvements) para entender por qué._
- Puede no ser obvio qué devuelve cada función. (¿Es el resultado del envío del correo, o esa variable `result`? ¿O algo más?)
- No es evidente cómo agregar funcionalidad (por ejemplo, supongamos que necesitamos cargar descuentos, créditos o puntos del cliente).
- A veces los nombres de parámetros temporales (como en cada `.then(param => {})`) añaden contexto. Sin embargo, con el tiempo, probablemente se conviertan en un almacén de nombres innecesarios.

### Solución, Parte 1: ¡Cree un módulo!

Esta técnica consiste en organizar funciones relacionadas en un solo módulo (por ejemplo, `CartHelpers`). No exige un patrón específico. Explore [funciones de fábrica](#carthelpers-factory), [Clases](#carthelpers-class), Cierres, Mixins, etc. Elija lo que más sentido tenga para su proyecto y equipo.

#### CartHelpers Factory

Ejemplo de un módulo `CartHelpers`, donde `userId` se pasa una sola vez y todos los métodos son de un solo argumento.

```tsx
const CartHelpers = (userId: number) => {
  return {
    getProductsSubtotal: products => getProductsSubtotal(userId, products),
    applyTaxes: subTotal => applyTaxes(userId, subTotal),
    purchaseProducts: total => purchaseProducts(userId, total),
    sendReceipt: invoice => sendReceipt(userId, invoice)
  };
};
```

#### Clase CartHelpers

Si las clases son su enfoque, es fácil adaptar:

```tsx
class CartHelpers {
  constructor(userId) {
    this.userId = userId;
  }
  getProductsSubtotal = products => getProductsSubtotal(this.userId, products);
  applyTaxes = subTotal => applyTaxes(this.userId, subTotal);
  purchaseProducts = total => purchaseProducts(this.userId, total);
  sendReceipt = invoice => sendReceipt(this.userId, invoice);
}
```

Beneficios inmediatos:

- Elimina la pasada repetitiva de variables.
  - DRY: `CartHelpers` abstrae el argumento repetido `userId`.
  - Cada método acepta **_solo_** los argumentos necesarios. Haciendo que `cart.applyTaxes(subTotal)` sea completamente intuitivo de leer.
- Las funciones de un solo argumento en `CartHelpers` son más legibles, con propósito claro.

Agrupando funciones relacionadas, creamos la oportunidad de reducir el área expuesta (ej. `checkout()`, métodos públicos de `CartHelpers`).

> Menor área expuesta === menor carga cognitiva, mejor testing y mantenibilidad.
> _Sistemas de diseño con intención y enfoque. ✨_

#### Uso de checkout y CartHelpers

Veamos cómo se ve ahora la función `checkout()`:

```tsx
export const checkout = ({ userId, products }) => {
  const cart = CartHelpers(userId);

  return Promise.resolve(products)
    .then(products => cart.getProductsSubtotal(products))
    .then(subTotal => cart.applyTaxes(subTotal))
    .then(total => cart.purchaseProducts(total))
    .then(result => cart.sendReceipt(result));
};
```

##### Checkout con mejoras adicionales

> ¿Se puede mejorar aún más? ¡Sí! No necesitamos repetir argumentos en absoluto.

Cuando los argumentos de una función son proporcionados por la salida de funciones anteriores, puedes simplificar aún más el código.

```tsx
export const checkout = ({ userId, products }) => {
  const cart = CartHelpers(userId);

  // 🌈 Las funciones se apilan como Legos y se leen como palabras normales para los humanos 💅
  return Promise.resolve(products)
    .then(cart.getProductsSubtotal)
    .then(cart.applyTaxes)
    .then(cart.purchaseProducts)
    .then(cart.sendReceipt);
};
```

**Si te parece inadecuado combinar parámetros en un solo (objeto) argumento,** considera dividir tus funciones **O** combinarlas en módulos con un alcance más adecuado.

#### ¿Dónde empezar?

Encuentra funciones relacionadas y agrúpalas. (p. ej., `CartHelpers`.)

Una parte del desafío al encontrar posibles módulos lógicos es identificar el código relacionado en primer lugar.

##### ¿Qué hace que las funciones estén relacionadas?

Una técnica útil: Busca repeticiones en los parámetros de las funciones. ¿Hay una relación en juego? ¿O una responsabilidad subyacente?

- ✅ Funciones con argumentos repetidos y comunes. (p. ej., Si 4 métodos aceptan `userRewards`, es probable que necesites un módulo `Rewards` u otro.)
- ✅ Funciones cuyos argumentos son proporcionados directamente por la salida de funciones anteriores. (Secuencias de pasos. p. ej., `Extract`, `Transform`, `Load`.)
- ❌ Cualquier cosa vagamente relacionada con el área de la característica, "¿compra de productos?"
- ❌ Funciones con nombres que comparten un prefijo o sufijo común?
- ❌ Funciones que requieren objetos grandes como argumentos, a pesar de usar solo unos pocos valores de dentro de esos objeto(s). (p. ej., `applyTaxes({ user, business, rewards, kitchenSink })` vs `applyTaxes({ subTotal })`)

Aunque no existe una única "respuesta correcta" al diseñar módulos, ayuda identificar 2-3 opciones para la organización: dibuja un esquema, escribe código "fantasía", pregunta "¿te hace feliz?".

<aside>
📌 Suele tomar varios intentos de organización de módulos antes de que tu Modelo de Dominio se aclare. No te obsesiones por lograrlo perfecto.
</aside>

> Podrías sentir que `cart.sendReceipt()` no encaja con métodos relacionados con pagos. Quizás `customerNotifications.sendReceipt()` sea un mejor lugar para mensajes al cliente. Si `CartHelper` es lo suficientemente importante, podría actuar como **_controlador_** llamando internamente a todos los **_servicios_** necesarios, como `customerNotifications`.

#### ¿Cómo sabes que estás ayudando?

Si la legibilidad no sufre al eliminar argumentos ad-hoc, **¡FELICIDADES!!!** Probablemente hayas construido un módulo con un alcance claro y duradero.

- Eliminar argumentos intermedios tiene la forma de forzar a que "capas" emerjan.
- ¡Debería ser difícil meter código ad-hoc en el lugar incorrecto!

Entonces, surge la pregunta, ¿dónde agregamos nueva funcionalidad?  

En mi experiencia, hay 2 estrategias principales para evaluar al agregar funcionalidad:  

1. Extender/refactorizar un método existente. (Cuando el nuevo código es lo suficientemente similar al existente.)  
2. Crear una nueva función (quinta) en el lugar deseado de la cadena. (Suponiendo que el nuevo código no está relacionado con las funciones existentes.)  

En última instancia, esto facilita decidir dónde pertenece la nueva funcionalidad. (p. ej., `cart.applyDiscounts()`, `cart.applyTaxes()`, `rewards.getBalance()`.)  

### Conclusión  

Pasar el estado a través de una tubería compleja puede ser complicado. Sin embargo, con un poco de práctica en refactorización, te encontrarás escribiendo código más legible, con menor carga cognitiva.
