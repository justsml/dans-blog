## Мастер пайплайнов: передача состояния

Вы сталкивались с трудностями при передаче состояния через функциональные пайплайны?

Организация (или её отсутствие) вашего кода напрямую влияет на то, насколько легко передавать состояние.

В этой статье мы рассмотрим эффективную технику передачи состояния через пайплайн. Попутно улучшим организацию и читаемость кода.

Следующий «реальный» фрагмент будет в центре нашего внимания: функция `checkout`, которая принимает `userId` и массив `products`. Она возвращает цепочку Promise, выполняющую 4 функции последовательно.

```tsx
const checkout = (userId: number, products: number[]) => {
  return getProductsSubtotal(userId, products)
    .then(subTotal => applyTaxes(userId, subTotal))
    .then(total => purchaseProducts(userId, total))
    .then(result => sendReceipt(userId, result));
};
```

Погодите-ка, этот код на самом деле довольно неплох, насколько это вообще возможно для пайплайнов в JS!

Тем не менее, у него есть несколько тонких проблем, которые в совокупности могут перерасти в более серьёзные.

Одна из проблем — мы постоянно передаём `userId` каждой (логически связанной) функции.
Теперь добавьте к этому ещё одну проблему, которую легко пропустить как разработчикам, так и TypeScript: перестановка числовых аргументов легко создаёт скрытую ошибку. (См. `applyTaxes` и `purchaseProducts`. _Что идёт первым — `userId` или `amount`?_)

Прежде чем решать, как улучшить этот код, давайте определим плюсы и минусы.

### Плюсы и минусы

#### Плюсы

- Хорошее использование замыкания! Передача `userId` и `products` один раз!
- Согласованное именование аргументов.
- Относительно эффективная и лаконичная композиция 4 ключевых функций для оформления заказа.
- «Бесплатное» управление потоком ошибок. (Ошибки всплывают из любых вложенных функций, отклоняя Promise, возвращаемый `checkout()`.)

#### Минусы

- Многократная передача `userId` утомительна.
- Функции не являются одноместными (унарными). _Это влияет на композируемость. См. [финальный пример](#checkout-with-further-improvements) для пояснения._
- Не всегда очевидно, что возвращает каждая функция. (Результат отправки email, или переменная `result`? Или что-то ещё?)
- Неочевидно, как добавлять функциональность (например, если нужно загрузить скидку/кредит/баллы клиента и т.д.)
- Иногда «временные» имена параметров (как в каждом `.then(param => {})`) добавляют контекст. Однако со временем они, скорее всего, станут источником мусора в именах.

### Решение, часть 1: Создаём модуль!

Этот подход заключается в организации связанных функций в единый модуль (например, `CartHelpers`). Он не требует определённого паттерна. Изучите [фабричные функции](#carthelpers-factory), [классы](#carthelpers-class), замыкания, примеси и т.д. Найдите то, что подходит для вашего проекта и команды.

#### Фабрика CartHelpers

Пример модуля `CartHelpers`, где `userId` передаётся один раз, а все методы являются одноместными (с одним аргументом).

```tsx
const CartHelpers = (userId: number) => {
  return {
    getProductsSubtotal: products => getProductsSubtotal(userId, products),
    applyTaxes: subTotal => applyTaxes(userId, subTotal),
    purchaseProducts: total => purchaseProducts(userId, total),
    sendReceipt: invoice => sendReceipt(userId, invoice)
  };
};
```

#### Класс CartHelpers

Если вам ближе классы, адаптировать несложно:

```tsx
class CartHelpers {
  constructor(userId) {
    this.userId = userId;
  }
  getProductsSubtotal = products => getProductsSubtotal(this.userId, products);
  applyTaxes = subTotal => applyTaxes(this.userId, subTotal);
  purchaseProducts = total => purchaseProducts(this.userId, total);
  sendReceipt = invoice => sendReceipt(this.userId, invoice);
}
```

Вот некоторые преимущества:

- Устраняет повторяющуюся передачу переменных.
  - DRY: `CartHelpers` абстрагирует повторяющийся аргумент `userId`.
  - Каждый метод принимает **_только_** необходимые аргументы. Это делает `cart.applyTaxes(subTotal)` совершенно предсказуемым для чтения.
- Одноместные функции в `CartHelpers` более читаемы и имеют более ясное назначение.

Группируя связанные функции, мы создаём возможность уменьшить открытую поверхность (например, `checkout()`, публичные методы `CartHelpers`).

> Меньше поверхности === меньше когнитивная нагрузка, лучше тестирование и поддерживаемость.
> _Проектируйте системы с намерением и фокусом. ✨_

#### Использование Checkout и CartHelpers

Давайте посмотрим, как теперь выглядит функция `checkout()`:

```tsx
export const checkout = ({ userId, products }) => {
  const cart = CartHelpers(userId);

  return Promise.resolve(products)
    .then(products => cart.getProductsSubtotal(products))
    .then(subTotal => cart.applyTaxes(subTotal))
    .then(total => cart.purchaseProducts(total))
    .then(result => cart.sendReceipt(result));
};
```

##### Checkout с дальнейшими улучшениями

> Можно ли улучшить ещё? Да! Нам вообще не нужно повторять аргументы!

Когда аргументы функции предоставляются выходными данными предыдущих функций, вы можете упростить код ещё больше.

```tsx
export const checkout = ({ userId, products }) => {
  const cart = CartHelpers(userId);

  // 🌈 Functions stack like Lego & read like normal "Human Words!" 💅
  return Promise.resolve(products)
    .then(cart.getProductsSubtotal)
    .then(cart.applyTaxes)
    .then(cart.purchaseProducts)
    .then(cart.sendReceipt);
};
```

**Если объединение параметров в один (объектный) аргумент кажется неестественным,** подумайте о разделении функций **ИЛИ** объединении их в модули с более подходящей областью видимости.

#### С чего начать?

Найдите связанные функции и сгруппируйте их вместе (например, `CartHelpers`).

Часть сложности при поиске возможных логических модулей заключается в том, чтобы сначала выявить связанный код.

##### Что делает функции связанными?

Один полезный трюк: найдите повторяющиеся параметры функций. Спросите себя: есть ли здесь взаимосвязь? Или скрытая ответственность?

- ✅ Функции с повторяющимися общими аргументами. (Например, если 4 метода принимают `userRewards`, скорее всего, вам нужен модуль `Rewards` или другой.)
- ✅ Функции, чьи аргументы напрямую предоставляются выводом предыдущих функций. (Последовательности шагов, например, `Extract`, `Transform`, `Load`.)
- ❌ Всё, что смутно связано с функциональной областью «покупка товаров».
- ❌ Функции, имеющие общие префиксы или суффиксы в именах.
- ❌ Функции, которые требуют большие объекты в качестве аргументов, несмотря на использование лишь нескольких значений из этих объектов. (Например, `applyTaxes({ user, business, rewards, kitchenSink })` против `applyTaxes({ subTotal })`)

Хотя не существует единственного «правильного ответа» при проектировании модулей, полезно наметить 2–3 варианта организации — набросать план, написать «фантазийный» код, спросить «приносит ли это радость?»

<aside>
📌 Часто требуется несколько попыток организации модулей, прежде чем ваша доменная модель устаканится. Не мучайтесь, пытаясь сделать идеально.
</aside>

> Возможно, вы почувствуете, что `cart.sendReceipt()` не относится к методам, связанным с оплатой. Возможно, `customerNotifications.sendReceipt()` — лучшее место для уведомлений клиентов. Если `CartHelper` достаточно важен, он может выступать в роли **_контроллера_**, внутренне вызывая все необходимые **_сервисы_**, такие как `customerNotifications`.

#### Как понять, что вы помогаете?

Если читаемость не страдает при устранении ситуативных аргументов, **ПОЗДРАВЛЯЮ!!!** Скорее всего, вы построили модуль с четкой и устойчивой областью ответственности!

- Удаление промежуточных аргументов часто заставляет «слои» проявляться.
- Должно быть _сложно_ запихнуть ситуативный код не в то место!

Так возникает вопрос: где же добавлять функциональность?

По моему опыту, есть две основные стратегии, которые стоит рассмотреть при добавлении функциональности:

1.  Расширить/рефакторить существующий метод. (Когда новый код достаточно близок к существующему.)
2.  Создать новую (пятую) функцию в нужном месте цепочки. (Предполагая, что новый код не связан с существующими функциями.)

В конечном итоге это упрощает принятие решения о том, куда должна попасть новая функциональность. (например, `cart.applyDiscounts()`, `cart.applyTaxes()`, `rewards.getBalance()`.)

### Заключение
