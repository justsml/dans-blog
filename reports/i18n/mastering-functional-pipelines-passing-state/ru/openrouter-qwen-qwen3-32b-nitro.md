## Мастер пайплайнов: Передача состояния

Встречались ли вы с проблемами передачи состояния с помощью функциональных пайплайнов?

Организация (или её отсутствие) вашего кода напрямую влияет на удобство передачи состояния.

В этой статье мы рассмотрим эффективную технику передачи состояния через пайплайн. По ходу дела мы улучшим организацию и читаемость нашего кода.

Следующий «реальный» фрагмент станет фокусом этой статьи: функция checkout, которая принимает `userId` и массив `products`. Она возвращает Promise-цепочку, выполняющую 4 функции последовательно.

```tsx
const checkout = (userId: number, products: number[]) => {
  return getProductsSubtotal(userId, products)
    .then(subTotal => applyTaxes(userId, subTotal))
    .then(total => purchaseProducts(userId, total))
    .then(result => sendReceipt(userId, result));
};
```

Подождите секунду, этот код на самом деле довольно неплох для пайплайнов в JS!

Он страдает от нескольких тонких проблем, которые могут объединиться в более серьезные сложности.  

Одна из проблем в том, что мы многократно передаем `userId` каждой (логически связанным) функции.  
Теперь объедините это с еще одной проблемой, которую легко упустить разработчикам и даже TypeScript: перестановка числовых аргументов легко создает скрытый баг. (См. `applyTaxes` и `purchaseProducts`. _Был ли первый аргумент `userId` или `amount`?_ )  

Прежде чем решать, как улучшить этот код, определим его плюсы и минусы.  

### Плюсы и минусы  

#### Плюсы  

- Хорошее использование замыкания! Передача `userId` и `products` один раз!  
- Единообразное именование аргументов.  
- Относительно эффективное и лаконичное составление 4 ключевых функций для оформления заказа.  
- «Бесплатный» контроль потока ошибок. (Ошибки распространяются из любых вложенных функций, отклоняя промис, возвращаемый `checkout()`.)

#### Минусы  

- Повторная передача `userId` в разных местах занимает много времени.  
- Функции не являются однопараметровыми (т.е. унарными). _Это влияет на составляемость. Посмотрите [окончательный пример](#checkout-with-further-improvements), чтобы понять почему?_  
- Иногда неясно, что возвращает каждая функция. (Результат отправки email или переменная `result`? Или что-то ещё?)  
- Неочевидно, как добавить функциональность (например, если нужно загрузить скидку/кредит/очки клиента и т.д.).  
- Иногда временные имена параметров (как в каждом `.then(param => {})`) добавляют контекст. Однако со временем они, скорее всего, станут беспорядочными.  

### Решение, Часть 1: Создайте модуль!  

Эта техника заключается в организации связанных функций в один модуль (например, `CartHelpers`). Она не требует конкретного шаблона. Исследуйте [фабричные функции](#carthelpers-factory), [Классы](#carthelpers-class), Замыкания, Смешанные модули и т.д. Найдите подходящее для вашего проекта и команды.  

#### CartHelpers Factory  

Пример модуля `CartHelpers`, где `userId` передаётся один раз, а все методы имеют один аргумент.

```tsx
const CartHelpers = (userId: number) => {
  return {
    getProductsSubtotal: products => getProductsSubtotal(userId, products),
    applyTaxes: subTotal => applyTaxes(userId, subTotal),
    purchaseProducts: total => purchaseProducts(userId, total),
    sendReceipt: invoice => sendReceipt(userId, invoice)
  };
};
```

#### CartHelpers Класс

Если вы предпочитаете классы, адаптация проста:

```tsx
class CartHelpers {
  constructor(userId) {
    this.userId = userId;
  }
  getProductsSubtotal = products => getProductsSubtotal(this.userId, products);
  applyTaxes = subTotal => applyTaxes(this.userId, subTotal);
  purchaseProducts = total => purchaseProducts(this.userId, total);
  sendReceipt = invoice => sendReceipt(this.userId, invoice);
}
```

Непосредственные преимущества:

- Устранение повторяющейся передачи переменных.
  - DRY: `CartHelpers` абстрагирует повторяющийся аргумент `userId`.
  - Каждый метод принимает **_только_** необходимые аргументы. Вызов `cart.applyTaxes(subTotal)` становится полностью предсказуемым.
- Функции с одним аргументом в `CartHelpers` более читаемы и имеют чёткую цель.

Группировка связанных функций создаёт возможность сократить экспонированную поверхность (например, `checkout()`, публичные методы `CartHelpers`).

> Меньше поверхности === меньше когнитивной нагрузки, лучшее тестирование и поддержка.
> _Проектируйте системы с намеренностью и фокусом. ✨_

#### Оформление заказа и использование CartHelpers

Давайте посмотрим, как теперь выглядит функция `checkout()`:

```tsx
export const checkout = ({ userId, products }) => {
  const cart = CartHelpers(userId);

  return Promise.resolve(products)
    .then(products => cart.getProductsSubtotal(products))
    .then(subTotal => cart.applyTaxes(subTotal))
    .then(total => cart.purchaseProducts(total))
    .then(result => cart.sendReceipt(result));
};
```

##### Оформление заказа с дальнейшими улучшениями

> Можно ли улучшить ещё больше? Да! Нам не нужно вообще повторять аргументы!

Когда аргументы функции предоставляются результатом предыдущих функций, можно ещё больше упростить код.

```tsx
export const checkout = ({ userId, products }) => {
  const cart = CartHelpers(userId);

  // 🌈 Функции складываются как конструктор LEGO и читаются как обычные "человеческие слова!" 💅
  return Promise.resolve(products)
    .then(cart.getProductsSubtotal)
    .then(cart.applyTaxes)
    .then(cart.purchaseProducts)
    .then(cart.sendReceipt);
};
```

**Если объединение параметров в одном (объектном) аргументе кажется неестественным,** рассмотрите возможность разбиения функций **ИЛИ** объединения их в более подходящие по области модули.

#### С чего начать?

Найдите связанные функции и сгруппируйте их вместе. (например, `CartHelpers`.)

Одна из сложностей при поиске возможных логических модулей — это идентификация связанных функций на начальном этапе.

##### Что делает функции связанными?

Один из полезных приёмов: ищите повторения в параметрах функций. Спросите, есть ли между ними связь? Или скрытая ответственность?

- ✅ Функции с повторяющимися, общими аргументами. (например, если 4 метода принимают `userRewards`, высока вероятность, что вам нужен модуль `Rewards` или другой модуль.)
- ✅ Функции, аргументы которых напрямую предоставляются выходными данными предыдущих функций. (Последовательности шагов. Например, `Extract`, `Transform`, `Load`.)
- ❌ Что-либо смутно связанное с областью функционала, «покупка продуктов»?
- ❌ Функции с общим префиксом или суффиксом в названии?
- ❌ Функции, требующие больших объектов в качестве аргументов, несмотря на использование лишь нескольких значений из этих объектов. (например, `applyTaxes({ user, business, rewards, kitchenSink })` vs `applyTaxes({ subTotal })`)

Хотя нет однозначного «правильного ответа» при проектировании модулей, полезно выделить 2-3 варианта организации — нарисуйте контуры, напишите «фантазийный» код, спросите: «Вдохновляет ли это?»

#### Как понять, что вы помогаете?  

Если читаемость не страдает при исключении случайных аргументов, **ПОЗДРАВЛЯЕМ!!!** Вы, скорее всего, создали модуль с чёткой и устойчивой областью ответственности!  

- Устранение промежуточных аргументов способствует выявлению «слоёв» в архитектуре.  
- Помещать случайный код в неправильное место **_должно быть_** сложно!

Так возникает вопрос: куда добавлять новую функциональность?

В моем опыте при добавлении функциональности оцениваются две основные стратегии:

1. Расширить/рефакторить существующий метод. (Когда новый код достаточно близок к существующему.)
2. Создать новую (5-ю) функцию в нужной точке цепочки. (Предполагая, что новый код не связан с существующими функциями.)

В конечном итоге это упрощает решение, куда принадлежит новая функциональность. (например, `cart.applyDiscounts()`, `cart.applyTaxes()`, `rewards.getBalance()`.)
