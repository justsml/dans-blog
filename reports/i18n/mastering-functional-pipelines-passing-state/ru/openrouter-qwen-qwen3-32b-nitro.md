## Мастерство пайплайнов: Передача состояния

Случалось ли вам сталкиваться с проблемами передачи состояния с помощью функциональных пайплайнов?

Организация (или её отсутствие) вашего кода напрямую влияет на удобство передачи состояния.

В этой статье мы рассмотрим эффективную технику передачи состояния через пайплайн. По пути мы улучшим организацию и читаемость нашего кода.

Следующий «реальный» фрагмент станет фокусом этой статьи: функция checkout, которая принимает `userId` и массив `products`. Она возвращает цепочку Promise, выполняющую 4 функции последовательно.

```tsx
const checkout = (userId: number, products: number[]) => {
  return getProductsSubtotal(userId, products)
    .then(subTotal => applyTaxes(userId, subTotal))
    .then(total => purchaseProducts(userId, total))
    .then(result => sendReceipt(userId, result));
};
```

Подождите секунду, этот код на самом деле довольно неплох для пайплайнов в JS!

Этот код имеет несколько тонких проблем, которые могут объединиться в более серьезные последствия.

Одна проблема заключается в том, что мы постоянно передаем `userId` в каждую (логически связанную) функцию.  
Соедините это с еще одной проблемой, которую легко упустить как разработчики, так и TypeScript: перестановка числовых аргументов легко приводит к тихой ошибке. (См. `applyTaxes` и `purchaseProducts`. _Кто был первым: `userId` или `amount`?_)

Перед тем как решить, как улучшить этот код, определим некоторые плюсы и минусы.

### Плюсы и минусы

#### Плюсы

- Хорошее использование замыкания! Передача `userId` и `products` всего один раз.  
- Согласованное именование аргументов.  
- Относительно эффективное и лаконичное объединение 4 ключевых функций для оформления заказа.  
- «Бесплатный» контроль потока ошибок. (Ошибки распространяются из любых вложенных функций, отклоняя промис, возвращаемый `checkout()`.)

#### Минусы

- Повторная передача `userId` утомительна.  
- Функции не являются одноаргументными (однопараметровыми). _Это влияет на составляемость. См. [окончательный пример](#checkout-with-further-improvements), почему?_  
- Не всегда очевидно, что возвращает каждая функция. (Это результат отправки email, или переменная `result`, или что-то ещё?)  
- Неясно, как добавить функциональность (например, предположим, что нужно загрузить скидку/кредит/баллы клиента и т.д.).  
- Иногда временные имена параметров (как в каждом `.then(param => {})`) добавляют контекст. Однако со временем они, вероятно, станут хранилищем для нечитаемых имен.  

### Решение, часть 1: Создайте модуль!  

Этот подход заключается в организации связанных функций в один модуль (например, `CartHelpers`). Он не требует конкретного шаблона. Изучите [фабричные функции](#carthelpers-factory), [Классы](#carthelpers-class), Замыкания, Миксины и т.д. Найдите то, что подходит для вашего проекта и команды.  

#### Фабричная функция `CartHelpers`  

Пример модуля `CartHelpers`, где `userId` передаётся один раз, а все методы являются одноаргументными.

```tsx
const CartHelpers = (userId: number) => {
  return {
    getProductsSubtotal: products => getProductsSubtotal(userId, products),
    applyTaxes: subTotal => applyTaxes(userId, subTotal),
    purchaseProducts: total => purchaseProducts(userId, total),
    sendReceipt: invoice => sendReceipt(userId, invoice)
  };
};
```

#### Фабричная функция `CartHelpers`

Если вы предпочитаете классы, их легко адаптировать:

```tsx
class CartHelpers {
  constructor(userId) {
    this.userId = userId;
  }
  getProductsSubtotal = products => getProductsSubtotal(this.userId, products);
  applyTaxes = subTotal => applyTaxes(this.userId, subTotal);
  purchaseProducts = total => purchaseProducts(this.userId, total);
  sendReceipt = invoice => sendReceipt(this.userId, invoice);
}
```

Непосредственные преимущества:

- Устранение повторяющейся передачи переменных.
  - DRY: `CartHelpers` абстрагирует повторяющийся аргумент `userId`.
  - Каждый метод принимает **_только_** необходимые аргументы. Вызов `cart.applyTaxes(subTotal)` становится полностью предсказуемым.
- Одноаргументные функции в `CartHelpers` более читаемы и имеют явную цель.

Группировка связанных функций создаёт возможность сократить экспонированную поверхность (например, `checkout()`, публичные методы `CartHelpers`).

> Меньше поверхности === меньше когнитивной нагрузки, лучшее тестирование и поддержка.
> _Проектируйте системы с намерением и фокусом. ✨_

#### Использование checkout и CartHelpers

Давайте посмотрим, как теперь выглядит функция `checkout()`:

```tsx
export const checkout = ({ userId, products }) => {
  const cart = CartHelpers(userId);

  return Promise.resolve(products)
    .then(products => cart.getProductsSubtotal(products))
    .then(subTotal => cart.applyTaxes(subTotal))
    .then(total => cart.purchaseProducts(total))
    .then(result => cart.sendReceipt(result));
};
```

##### Дальнейшие улучшения checkout

> Можно ли улучшить ещё больше? Да! Нам не нужно повторять аргументы вообще!

Когда аргументы функции предоставляются результатом предыдущих функций, код можно упростить ещё больше.

```tsx
export const checkout = ({ userId, products }) => {
  const cart = CartHelpers(userId);

  // 🌈 Функции складываются как Лего и читаются как обычные "Человеческие слова!" 💅
  return Promise.resolve(products)
    .then(cart.getProductsSubtotal)
    .then(cart.applyTaxes)
    .then(cart.purchaseProducts)
    .then(cart.sendReceipt);
};
```

**Если объединение параметров в одиночные (объектные) аргументы кажется неестественным,** рассмотрите возможность разбиения функций **ИЛИ** объединения их в более подходящие по масштабу модули.

#### С чего начать?

Найдите связанные функции и сгруппируйте их вместе. (например, `CartHelpers`.)

Одной из сложностей при поиске возможных логических модулей является идентификация связанных функций в первую очередь.

##### Что делает функции связанными?

Один полезный трюк: ищите повторения в параметрах функций. Спросите себя: есть ли здесь какая-то взаимосвязь? Или скрытая ответственность?

- ✅ Функции с повторяющимися, общими аргументами. (например, если 4 метода принимают `userRewards`, высока вероятность, что вам нужен модуль `Rewards` или аналогичный.)
- ✅ Функции, аргументы которых напрямую предоставляются выходными данными предыдущих функций. (Последовательности шагов. Например: `Extract`, `Transform`, `Load`.)
- ❌ Что-либо вагно связанное с областью функциональности, типа "покупка продукта?"
- ❌ Функции с общими префиксами или суффиксами в именах?
- ❌ Функции, требующие больших объектов в качестве аргументов, несмотря на использование лишь нескольких значений из этих объектов. (например, `applyTaxes({ user, business, rewards, kitchenSink })` vs `applyTaxes({ subTotal })`)

Хотя нет однозначного «правильного ответа» при проектировании модулей, полезно выделить 2-3 варианта организации — нарисовать контуры, написать «фантазийный» код, спросить: «приносит ли это радость?»

#### Как понять, что вы помогаете?

Если при удалении промежуточных аргументов читаемость не страдает, **ПОЗДРАВЛЯЕМ!!!** Вы, скорее всего, создали модуль с чёткой и устойчивой областью ответственности!

- Удаление промежуточных аргументов способствует возникновению «слоёв».
- Вставить произвольный код в неправильное место должно быть сложно!

Так возникает вопрос: где мы добавляем новую функциональность?

В моем опыте при добавлении функциональности есть 2 основные стратегии:

1. Расширить/рефакторить существующий метод (когда новый код достаточно близок к существующему).
2. Создать новую (5-ю) функцию в нужном месте цепочки (предполагая, что новый код не связан с существующими функциями).

В конечном итоге это упрощает определение того, где должна находиться новая функциональность. (Например: `cart.applyDiscounts()`, `cart.applyTaxes()`, `rewards.getBalance()`.)
