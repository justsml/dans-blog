## Maître des pipelines : Passage d’état

Avez‑vous déjà rencontré des difficultés à faire passer de l’état à travers des pipelines fonctionnels ?

L’organisation (ou son absence) de votre code influe directement sur la facilité avec laquelle l’état circule.

Dans cet article, nous explorerons une technique efficace pour transmettre l’état à travers un pipeline. En chemin, nous améliorerons l’organisation et la lisibilité de notre code.

L’extrait « réel » suivant sera notre sujet : une fonction de paiement, qui accepte un `userId` et un tableau de `products`. Elle renvoie une chaîne de promesses qui exécute 4 fonctions séquentiellement.

```tsx
const checkout = (userId: number, products: number[]) => {
  return getProductsSubtotal(userId, products)
    .then(subTotal => applyTaxes(userId, subTotal))
    .then(total => purchaseProducts(userId, total))
    .then(result => sendReceipt(userId, result));
};
```

Attendez une seconde, ce code est en fait assez correct, pour ce qui concerne les pipelines en JS !

Il présente toutefois quelques problèmes subtils qui peuvent se combiner en difficultés plus importantes.

Un problème est que nous transmettons re‑définiment `userId` à chaque fonction (logiquement liée).  
Ajoutez à cela un autre souci facilement négligé par les développeurs et même par TypeScript : inverser les arguments numériques crée facilement un bug silencieux. (Voir `applyTaxes` et `purchaseProducts`. _Était‑ce `userId` ou `amount` qui devait venir en premier ?_)

Avant de décider comment améliorer ce code, identifions quelques avantages et inconvénients.

### Avantages & Inconvénients

#### Avantages

- Bonne utilisation d’une fermeture ! `userId` et `products` ne sont passés qu’une fois !
- Nommage d’arguments cohérent.
- Composition relativement efficace et concise de 4 fonctions clés pour le checkout.
- Gestion d’erreur « gratuite ». (Les erreurs remontent depuis n’importe quelle fonction imbriquée, rejetant la promesse renvoyée par `checkout()`.)

#### Inconvénients

- Passer `userId` à plusieurs reprises devient fastidieux.  
- Les fonctions ne sont pas à paramètre unique (c’est‑à‑dire unary). _Cela nuit à la composabilité. Voir l'[exemple final](#checkout-with-further-improvements) pour comprendre pourquoi._  
- Il n’est pas toujours clair ce que chaque fonction renvoie. (Est‑ce le résultat de l’envoi d’email, ou la variable `result` ? Ou autre ?)  
- Il n’est pas évident comment ajouter de la fonctionnalité (par ex. : charger la remise, le crédit, les points client, etc.).  
- Parfois, les noms de paramètres « temporaire » (comme dans chaque `.then(param => {})`) apportent du contexte. Cependant, avec le temps, ils finissent souvent par devenir du bruit de nommage.

### Solution, partie 1 : créer un module !

Cette technique consiste à regrouper les fonctions liées dans un même module (par ex. `CartHelpers`). Elle n’impose aucun modèle particulier. Explorez les [fonctions usine](#carthelpers-factory), les [classes](#carthelpers-class), les fermetures, les mixins, etc. Choisissez ce qui a du sens pour votre projet et votre équipe.

#### Factory CartHelpers

Exemple d’un module `CartHelpers`, où `userId` est fourni une seule fois et toutes les méthodes sont à argument unique.

```tsx
const CartHelpers = (userId: number) => {
  return {
    getProductsSubtotal: products => getProductsSubtotal(userId, products),
    applyTaxes: subTotal => applyTaxes(userId, subTotal),
    purchaseProducts: total => purchaseProducts(userId, total),
    sendReceipt: invoice => sendReceipt(userId, invoice)
  };
};
```

#### Classe CartHelpers

Si les classes vous conviennent, l’adaptation est triviale :

```tsx
class CartHelpers {
  constructor(userId) {
    this.userId = userId;
  }
  getProductsSubtotal = products => getProductsSubtotal(this.userId, products);
  applyTaxes = subTotal => applyTaxes(this.userId, subTotal);
  purchaseProducts = total => purchaseProducts(this.userId, total);
  sendReceipt = invoice => sendReceipt(this.userId, invoice);
}
```

Quelques bénéfices immédiats :

- Suppression du passage répétitif de variables.  
  - DRY : `CartHelpers` encapsule l’argument récurrent `userId`.  
  - Chaque méthode n’accepte **_que_** les paramètres nécessaires. `cart.applyTaxes(subTotal)` devient alors totalement prévisible à la lecture.  
- Les fonctions à un seul argument dans `CartHelpers` sont plus lisibles, avec un objectif plus clair.

En regroupant les fonctions connexes, on crée une opportunité de réduire la surface exposée (par ex. `checkout()`, méthodes « publiques » de `CartHelpers`).

> Moins de surface exposée === moins de charge cognitive, meilleurs tests et maintenabilité.  
> _Concevez les systèmes avec intention et focus. ✨_

#### Checkout & Utilisation de CartHelpers

Voyons à quoi ressemble maintenant la fonction `checkout()` :

```tsx
export const checkout = ({ userId, products }) => {
  const cart = CartHelpers(userId);

  return Promise.resolve(products)
    .then(products => cart.getProductsSubtotal(products))
    .then(subTotal => cart.applyTaxes(subTotal))
    .then(total => cart.purchaseProducts(total))
    .then(result => cart.sendReceipt(result));
};
```

##### Checkout avec des améliorations supplémentaires

> Peut‑on encore l’améliorer ? Oui ! On n’a plus besoin de répéter les arguments !

Lorsque les arguments d’une fonction sont fournis par la sortie des fonctions précédentes, on peut simplifier le code davantage.

```tsx
export const checkout = ({ userId, products }) => {
  const cart = CartHelpers(userId);

  // 🌈 Les fonctions s’empilent comme des Lego & se lisent comme des « Mots humains ! » 💅
  return Promise.resolve(products)
    .then(cart.getProductsSubtotal)
    .then(cart.applyTaxes)
    .then(cart.purchaseProducts)
    .then(cart.sendReceipt);
};
```

**Si cela vous paraît contre‑naturel de combiner les paramètres en un seul argument (objet),** envisagez de scinder vos fonctions **OU** de les regrouper dans des modules mieux délimités.

#### Par où commencer ?

Identifiez les fonctions connexes et regroupez‑les. (par ex. `CartHelpers`.)

Une partie du défi lorsqu’on cherche des modules logiques possibles est de repérer le code lié dès le départ.

##### Qu’est‑ce qui rend les fonctions liées ?

Une astuce pratique : cherchez les répétitions dans les paramètres de fonction. Y a‑t‑il une relation sous‑jacente ? Une responsabilité commune ?

- ✅ Fonctions avec des arguments répétés et communs. (par ex. si 4 méthodes acceptent `userRewards`, il y a de fortes chances que vous ayez besoin d’un module `Rewards` ou similaire.)
- ✅ Fonctions dont les arguments sont fournis directement par la sortie de fonctions précédentes. (Séquences d’étapes, par ex. `Extract`, `Transform`, `Load`.)
- ❌ Tout ce qui est vaguement lié à la zone fonctionnelle, « achat de produit » ?
- ❌ Fonctions dont le nom partage un préfixe ou suffixe commun ?
- ❌ Fonctions qui exigent de gros objets comme arguments, alors qu’elles n’utilisent que quelques valeurs de ces objets. (par ex. `applyTaxes({ user, business, rewards, kitchenSink })` contre `applyTaxes({ subTotal })`)

Bien qu’il n’existe pas de « bonne réponse » unique pour concevoir des modules, il est utile d’identifier 2‑3 options d’organisation — esquisser un plan, écrire du code « fantaisie », se demander « ça me donne envie ? »

<aside>
📌 Il faut souvent plusieurs essais d’organisation de modules avant que votre modèle de domaine ne prenne forme. Ne vous tourmentez pas à le rendre parfait du premier coup.
</aside>

> Vous avez peut‑être l’impression que `cart.sendReceipt()` n’appartient pas aux méthodes liées au paiement. Peut‑être que `customerNotifications.sendReceipt()` serait un meilleur endroit pour la messagerie client. Si `CartHelper` est suffisamment important, il peut jouer le rôle de **_contrôleur_** en appelant en interne tous les **_services_** nécessaires, comme `customerNotifications`.

#### Comment savoir si vous êtes sur la bonne voie ?

Si la lisibilité ne se dégrade pas lorsque vous éliminez les arguments ad‑hoc, **FÉLICITATIONS !!!** Vous avez probablement construit un module avec une portée claire et durable !

- Supprimer les arguments intermédiaires a tendance à faire émerger des « couches ».
- Il _devrait_ être difficile de déposer du code ad‑hoc au mauvais endroit !

Alors, laquestion qui se pose est : où ajouter la fonctionnalité ?

D’après mon expérience, deux stratégies principales doivent être évaluées lorsqu’on ajoute une fonctionnalité :

1.  Étendre / refactoriser la méthode existante. (Lorsque le nouveau code est suffisamment proche du code existant.)  
2.  Créer une nouvelle fonction (la 5ᵉ) à l’endroit souhaité dans la chaîne. (En supposant que le nouveau code n’est pas lié aux fonctions existantes.)

En fin de compte, cela simplifie la décision sur l’emplacement de la nouvelle fonctionnalité. (par ex. `cart.applyDiscounts()`, `cart.applyTaxes()`, `rewards.getBalance()`.)

### Conclusion

Faire passer l’état à travers une chaîne complexe peut être délicat. Cependant, avec un peu de pratique de refactorisation, vous finirez par écrire du code plus lisible, avec une charge cognitive réduite.
