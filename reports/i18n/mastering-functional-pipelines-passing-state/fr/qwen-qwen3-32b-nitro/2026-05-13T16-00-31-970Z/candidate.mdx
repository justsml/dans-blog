## Maître des pipelines : Passage de l'état

Avez-vous rencontré des défis liés au passage de l'état à l'aide de pipelines fonctionnels ?

L'organisation (ou son absence) de votre code influence directement la facilité avec laquelle l'état est transmis.

Dans cet article, nous explorerons une technique efficace pour transmettre l'état à travers un pipeline. Au passage, nous améliorerons l'organisation et la lisibilité de notre code.

L'extrait suivant, « réel », sera au cœur de cet article : une fonction de paiement, qui accepte un `userId` et un tableau de `products`. Elle retourne une chaîne de promesses qui exécute 4 fonctions en séquence.

```tsx
const checkout = (userId: number, products: number[]) => {
  return getProductsSubtotal(userId, products)
    .then(subTotal => applyTaxes(userId, subTotal))
    .then(total => purchaseProducts(userId, total))
    .then(result => sendReceipt(userId, result));
};
```

Attendez une seconde, ce code est en réalité assez bon, en ce qui concerne les pipelines en JS !

Il souffre de quelques problèmes subtils qui peuvent se combiner pour créer des problèmes plus importants.  

Un problème est que nous passons répétitivement `userId` à chaque fonction (logiquement liée).  
Ajoutons à cela un autre problème souvent manqué par les développeurs et même par TypeScript : inverser l'ordre des arguments numériques crée facilement un bug silencieux. (Voir `applyTaxes` et `purchaseProducts`. _Est-ce `userId` ou `amount` qui vient en premier ?_)  

Avant de décider comment améliorer ce code, identifions quelques avantages/inconvénients.  

### Avantages et inconvénients  

#### Avantages  

- Une bonne utilisation d'une fermeture ! Passage de `userId` & `products` une seule fois !  
- Nommage cohérent des arguments.  
- Composition relativement efficace et concise des 4 fonctions clés pour le processus d'achat.  
- Gestion "gratuite" du flux d'erreurs. (Les erreurs remontent depuis n'importe quelle fonction imbriquée, rejetant la Promise retournée par `checkout()`.)

#### Inconvénients  

- Passage répétitif de `userId` est fastidieux.  
- Les fonctions ne sont pas à un seul paramètre (aka unaire). _Cela affecte la composable. Voir [exemple final](#checkout-with-further-improvements) pourquoi ?_  
- Il peut être non évident ce que retourne chaque fonction. (Est-ce le résultat de l'envoi de l'email, ou cette variable `result` ? Ou autre ?)  
- Ajout de fonctionnalités non évident (ex. : supposons qu'il faille charger les remises, crédits, points client, etc.).  
- Parfois des noms de paramètres temporaires (comme dans chaque `.then(param => {})`) ajoutent du contexte. Cependant, avec le temps, ils deviendront probablement le refuge d'un encombrement de noms.  

### Solution, Partie 1 : Créer un module !  

Cette technique consiste à organiser des fonctions liées en un seul module (ex. : `CartHelpers`). Elle ne nécessite pas un schéma spécifique. Explorez [fonctions usine](#carthelpers-factory), [Classes](#carthelpers-class), Closures, Mixins, etc. Choisissez ce qui convient à votre projet & équipe.  

#### Fonction usine CartHelpers  

Exemple d'un module `CartHelpers` où `userId` est passé une seule fois, et toutes les méthodes sont à un seul argument.

```tsx
const CartHelpers = (userId: number) => {
  return {
    getProductsSubtotal: products => getProductsSubtotal(userId, products),
    applyTaxes: subTotal => applyTaxes(userId, subTotal),
    purchaseProducts: total => purchaseProducts(userId, total),
    sendReceipt: invoice => sendReceipt(userId, invoice)
  };
};
```

#### Classe CartHelpers

Si vous préférez les classes, l'adaptation est facile :

```tsx
class CartHelpers {
  constructor(userId) {
    this.userId = userId;
  }
  getProductsSubtotal = products => getProductsSubtotal(this.userId, products);
  applyTaxes = subTotal => applyTaxes(this.userId, subTotal);
  purchaseProducts = total => purchaseProducts(this.userId, total);
  sendReceipt = invoice => sendReceipt(this.userId, invoice);
}
```

Bénéfices immédiats :

- Éliminer le passage répétitif des variables.
  - DRY : `CartHelpers` abstrait l'argument répétitif `userId`.
  - Chaque méthode accepte **uniquement** les arguments nécessaires. Rendant `cart.applyTaxes(subTotal)` entièrement prévisible à lire.
- Les fonctions à un seul argument dans `CartHelpers` sont plus lisibles, avec un objectif clair.

En regroupant les fonctions liées, on crée l'opportunité de réduire la superficie exposée (ex. : `checkout()`, méthodes publiques de `CartHelpers`.)

> Moins de superficie exposée === moins de charge cognitive, meilleure testabilité et maintenabilité.
> _Concevez des systèmes avec intention et focus. ✨_

#### Utilisation de checkout & CartHelpers

Voyons à quoi ressemble maintenant la fonction `checkout()` :

```tsx
export const checkout = ({ userId, products }) => {
  const cart = CartHelpers(userId);

  return Promise.resolve(products)
    .then(products => cart.getProductsSubtotal(products))
    .then(subTotal => cart.applyTaxes(subTotal))
    .then(total => cart.purchaseProducts(total))
    .then(result => cart.sendReceipt(result));
};
```

##### Améliorations supplémentaires pour checkout

> Peut-on l'améliorer davantage ? Oui ! Nous n'avons même pas besoin de répéter les arguments !

Quand les arguments d'une fonction sont fournis par la sortie des fonctions précédentes, vous pouvez simplifier encore davantage le code.

```tsx
export const checkout = ({ userId, products }) => {
  const cart = CartHelpers(userId);

  // 🌈 Les fonctions s'empilent comme des Lego & lisent comme des "Mots humains" normaux ! 💅
  return Promise.resolve(products)
    .then(cart.getProductsSubtotal)
    .then(cart.applyTaxes)
    .then(cart.purchaseProducts)
    .then(cart.sendReceipt);
};
```

**Si cela semble peu naturel de regrouper les paramètres en un seul argument (objet),** envisagez de diviser vos fonctions **OU** de les regrouper dans des modules mieux adaptés à leur portée.

#### Où commencer ?

Groupez les fonctions liées entre elles. (ex. `CartHelpers`).

Une partie du défi lors de la recherche de modules logiques possibles consiste à identifier d'abord le code lié.  

##### Qu'est-ce qui rend des fonctions liées ?

Une astuce utile : trouvez les répétitions dans les paramètres des fonctions. Demandez-vous s'il existe une relation sous-jacente ou une responsabilité commune ?

- ✅ Fonctions partageant des arguments répétés et communs. (ex. Si 4 méthodes acceptent `userRewards`, il est probable que vous ayez besoin d'un module `Rewards` ou similaire.)
- ✅ Fonctions dont les arguments sont fournis directement par la sortie de fonctions précédentes. (Séquences d'étapes. ex. `Extract`, `Transform`, `Load`)
- ❌ N'importe quoi lié de manière vague à la zone fonctionnelle, "achat de produit ?"
- ❌ Fonctions portant des préfixes/suffixes communs dans leurs noms ?
- ❌ Fonctions nécessitant de grands objets en arguments, malgré l'utilisation de seulement quelques valeurs dedans. (ex. `applyTaxes({ user, business, rewards, kitchenSink })` vs `applyTaxes({ subTotal })`)

Il n’existe pas de réponse unique pour concevoir des modules, mais il est utile d’identifier 2 à 3 options d’organisation – tracer un plan, écrire du code « fantaisiste », se demander « est-ce que ça fait plaisir ? »

<aside>
📌 Il arrive souvent qu’il faille plusieurs tentatives pour organiser les modules avant que votre modèle de domaine ne se révèle. Ne vous torturez pas pour obtenir la perfection.
</aside>

> Vous pourriez sentir que `cart.sendReceipt()` n’appartient pas aux méthodes liées au paiement. Peut-être que `customerNotifications.sendReceipt()` serait un meilleur endroit pour gérer les notifications client. Si `CartHelper` est suffisamment important, il pourrait agir en tant que **_contrôleur_** interne appelant tous les **_services_** nécessaires, comme `customerNotifications`.

#### Comment savoir si vous améliorez les choses ?

Si la lisibilité ne souffre pas lorsque vous éliminez les arguments ad-hoc, **FÉLICITATIONS !!!** Vous avez probablement construit un module avec une portée claire et durable !

- Supprimer les arguments intermédiaires force souvent l’émergence de « couches » logiques.
- Il _devrait_ être difficile de coller du code ad-hoc dans le mauvais endroit !

Alors, cela soulève la question : où ajoutons-nous la fonctionnalité ?  

D'après mon expérience, il existe deux stratégies principales à évaluer lors de l'ajout de fonctionnalités :  

1. Étendre/refactoriser une méthode existante. (Lorsque le nouveau code est suffisamment proche du code existant.)  
2. Créer une nouvelle fonction (cinquième) à l'endroit souhaité dans la chaîne. (En supposant que le nouveau code n'est pas lié aux fonctions existantes.)  

Cela rend finalement plus facile de décider où appartient la nouvelle fonctionnalité. (Par exemple : `cart.applyDiscounts()`, `cart.applyTaxes()`, `rewards.getBalance()`.)  

### Conclusion  

Passer l'état à travers un pipeline complexe peut être délicat. Cependant, avec un peu d'entraînement en refactorisation, vous vous retrouverez à écrire un code plus lisible, avec une charge cognitive réduite.
