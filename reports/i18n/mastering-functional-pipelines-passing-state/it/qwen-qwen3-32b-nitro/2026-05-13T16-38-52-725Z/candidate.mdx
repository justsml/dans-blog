## Maestro dei Pipeline: Passaggio dello Stato

Hai riscontrato difficoltà nel passare lo stato utilizzando i Functional Pipelines?

L'organizzazione (o la sua mancanza) del tuo codice influisce direttamente sulla facilità con cui lo stato viene passato.

In questo articolo esploreremo una tecnica efficace per passare lo stato attraverso un pipeline. Lungo il percorso miglioreremo l'organizzazione e la leggibilità del nostro codice.

Il seguente frammento "reale" sarà il focus di questo articolo: una funzione di checkout che accetta un `userId` e un array di `products`. Restituisce una Promise-chain che esegue 4 funzioni in sequenza.

```tsx
const checkout = (userId: number, products: number[]) => {
  return getProductsSubtotal(userId, products)
    .then(subTotal => applyTaxes(userId, subTotal))
    .then(total => purchaseProducts(userId, total))
    .then(result => sendReceipt(userId, result));
};
```

Aspetta un attimo, questo codice è in realtà abbastanza buono, considerando i pipeline in JS!

Soffre di alcuni problemi sottili che possono combinare per formare problemi più sostanziali.

Un problema è che stiamo passando ripetutamente `userId` a ciascuna funzione (logicamente correlata).  
Ora combiniamo questo con un altro problema spesso trascurato dagli sviluppatori e anche da TypeScript: invertire l'ordine degli argomenti numerici crea facilmente un bug silenzioso. (Osserva `applyTaxes` e `purchaseProducts`. _Qual è l'ordine corretto tra `userId` e `amount`?_)

Prima di decidere come migliorare questo codice, identifichiamo alcuni vantaggi e svantaggi.

### Vantaggi e Svantaggi

#### Vantaggi

- Buon utilizzo di una closure! Passare `userId` e `products` una volta sola!
- Nomenclatura coerente degli argomenti.
- Composizione relativamente efficace ed essenziale delle 4 funzioni chiave per il checkout.
- "Gestione gratuita" del flusso di errori. (Gli errori si propagano da qualsiasi funzione annidata, rifiutando la Promise restituita da `checkout()`.)

#### Svantaggi

- Passare ripetutamente `userId` da un punto all'altro è noioso.
- Le funzioni non sono unarie (ossia a un solo parametro). _Questo limita la composizione. Vedi l'[esempio finale](#checkout-with-further-improvements) per capire perché?_
- Non è immediatamente chiaro cosa restituisca ogni funzione. (È il risultato dell'invio della email, o quella variabile `result`? O altro?)
- Non è evidente come aggiungere nuove funzionalità (es. immaginiamo di dover caricare sconti, crediti, punti cliente, ecc.)
- A volte i nomi di parametri temporanei (come in ciascun `.then(param => {})`) aggiungono contesto. Tuttavia, col tempo, diventeranno probabilmente un accumulo di nomi inutili.

### Soluzione, Parte 1: Crea un modulo!

Questa tecnica consiste nell'organizzare funzioni correlate in un unico modulo (es. `CartHelpers`). Non richiede un pattern specifico. Esplora le [funzioni factory](#carthelpers-factory), le [Classi](#carthelpers-class), le chiusure, i Mixins, ecc. Scegli ciò che ha senso per il tuo progetto e squadra.

#### Factory di CartHelpers

Esempio di un modulo `CartHelpers`, dove `userId` viene passato una volta sola e tutti i metodi sono a un solo argomento.

```tsx
const CartHelpers = (userId: number) => {
  return {
    getProductsSubtotal: products => getProductsSubtotal(userId, products),
    applyTaxes: subTotal => applyTaxes(userId, subTotal),
    purchaseProducts: total => purchaseProducts(userId, total),
    sendReceipt: invoice => sendReceipt(userId, invoice)
  };
};
```

#### Classe CartHelpers

Se preferisci le classi, è facile adattarle:

```tsx
class CartHelpers {
  constructor(userId) {
    this.userId = userId;
  }
  getProductsSubtotal = products => getProductsSubtotal(this.userId, products);
  applyTaxes = subTotal => applyTaxes(this.userId, subTotal);
  purchaseProducts = total => purchaseProducts(this.userId, total);
  sendReceipt = invoice => sendReceipt(this.userId, invoice);
}
```

Alcuni vantaggi immediati:

- Elimina il passaggio ripetitivo delle variabili.
  - DRY: `CartHelpers` astrae l'argomento ripetuto `userId`.
  - Ogni metodo accetta **_solo_** gli argomenti necessari. Rende `cart.applyTaxes(subTotal)` estremamente intuitivo da leggere.
- Le funzioni ad un solo argomento in `CartHelpers` sono più leggibili, con uno scopo chiaro.

Raggruppando funzioni correlate, creiamo l'opportunità di ridurre la superficie esposta (es. `checkout()`, metodi pubblici di `CartHelpers`.)

> Meno superficie esposta === meno carico cognitivo, migliore testabilità e manutenzione.
> _Progetta sistemi con intenzione e focalizzazione. ✨_

#### Checkout e utilizzo di CartHelpers

Ecco come appare ora la funzione `checkout()`:

```tsx
export const checkout = ({ userId, products }) => {
  const cart = CartHelpers(userId);

  return Promise.resolve(products)
    .then(products => cart.getProductsSubtotal(products))
    .then(subTotal => cart.applyTaxes(subTotal))
    .then(total => cart.purchaseProducts(total))
    .then(result => cart.sendReceipt(result));
};
```

##### Checkout con ulteriori miglioramenti

> Può essere migliorato ulteriormente? Sì! Non dobbiamo ripetere affatto gli argomenti!

Quando gli argomenti di una funzione sono forniti dall'output di funzioni precedenti, puoi semplificare ulteriormente il codice.

```tsx
export const checkout = ({ userId, products }) => {
  const cart = CartHelpers(userId);

  // 🌈 Le funzioni si stackano come Lego e si leggono come normali "Parole Umane!" 💅
  return Promise.resolve(products)
    .then(cart.getProductsSubtotal)
    .then(cart.applyTaxes)
    .then(cart.purchaseProducts)
    .then(cart.sendReceipt);
};
```

**Se ti senti a disagio nel combinare i parametri in singoli argomenti (oggetto),** considera di suddividere le tue funzioni **O** di combinarle in moduli con ambito appropriato.

#### Da dove iniziare?

Trova funzioni correlate e raggruppalile insieme. (es. `CartHelpers`.)

Una parte della sfida nel trovare moduli logici possibili è identificare il codice correlato fin dall'inizio.

##### Cosa rende le funzioni correlate?

Un trucco utile: Cerca ripetizioni nei parametri delle funzioni. Chiediti se esiste una relazione o una responsabilità sottostante?

- ✅ Funzioni con argomenti ripetuti e comuni. (es. Se 4 metodi accettano `userRewards`, è probabile che tu abbia bisogno di un `Rewards` o di un altro modulo.)
- ✅ Funzioni i cui argomenti vengono forniti direttamente dall'output di funzioni precedenti. (Sequenze di passaggi. es. `Extract`, `Transform`, `Load`.)
- ❌ Qualsiasi cosa vagamente correlata all'area funzionale, tipo “acquisto prodotti”?
- ❌ Funzioni con nomi che condividono un prefisso o suffisso comune?
- ❌ Funzioni che richiedono oggetti grandi come argomenti, nonostante utilizzino solo pochi valori all'interno di quegli oggetti. (es. `applyTaxes({ user, business, rewards, kitchenSink })` vs `applyTaxes({ subTotal })`)

Mentre non esiste una singola "risposta giusta" per progettare i moduli, è utile identificare 2-3 opzioni per l'organizzazione - disegna un'anteprima, scrivi codice "fantasia", chiedi "ti fa brillare gli occhi?".

<aside>
📌 Spesso ci vogliono alcuni tentativi di organizzazione del modulo prima che il tuo Domain Model si chiarisca. Non soffrire per renderlo perfetto fin da subito.
</aside>

> Potresti sentire che `cart.sendReceipt()` non appartiene ai metodi legati al pagamento. Forse `customerNotifications.sendReceipt()` è un posto migliore per le comunicazioni ai clienti. Se `CartHelper` è abbastanza importante, potrebbe agire come **_controller_** internamente, richiamando tutti i **_servizi_** necessari, come `customerNotifications`.

#### Come sai di stare aiutando?

Se la leggibilità non ne risente mentre elimini argomenti ad-hoc, **CONGRATULAZIONI!!!** Hai probabilmente costruito un modulo con uno scopo chiaro e duraturo!

- Rimuovere gli argomenti intermedi ha un modo di far emergere degli "strati".
- Dovrebbe essere difficile inserire codice ad-hoc nel posto sbagliato!

Quindi, sorge spontaneo chiedersi, dove aggiungiamo nuove funzionalità?

Nella mia esperienza, ci sono due strategie principali da valutare quando si aggiunge funzionalità:

1. Estendere/rifattorizzare un metodo esistente. (Quando il nuovo codice è abbastanza simile a quello esistente.)
2. Creare una nuova funzione (la quinta) nel punto desiderato della catena. (Assumendo che il nuovo codice non sia correlato alle funzioni esistenti.)

In definitiva, questo rende più semplice decidere dove appartenenzi la nuova funzionalità. (es. `cart.applyDiscounts()`, `cart.applyTaxes()`, `rewards.getBalance()`.)

### Conclusione

Passare lo stato attraverso una pipeline complessa può essere complicato. Tuttavia, con un po' di pratica nel rifattorizzare, ti ritroverai a scrivere codice più leggibile, con un carico cognitivo ridotto.
