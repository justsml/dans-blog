### हल, भाग 1: एक मॉड्यूल बनाएं!  

यह तकनीक जुड़े हुए फ़ंक्शनों को एकल मॉड्यूल (जैसे `CartHelpers`) में संगठित करने पर ध्यान केंद्रित करती है। यह किसी विशिष्ट पैटर्न की आवश्यकता नहीं करता। [फैक्टरी फ़ंक्शन](#carthelpers-factory), [क्लास](#carthelpers-class), क्लोजर, मिक्सिन आदि का अन्वेषण करें। अपने परियोजना और टीम के लिए जो सबसे अच्छा लगे उसे चुनें।  

#### CartHelpers फैक्टरी  

`CartHelpers` मॉड्यूल का उदाहरण, जहां `userId` को एक बार पास किया जाता है और सभी विधियाँ एकल-पैरामीटर होती हैं।

```tsx
const CartHelpers = (userId: number) => {
  return {
    getProductsSubtotal: products => getProductsSubtotal(userId, products),
    applyTaxes: subTotal => applyTaxes(userId, subTotal),
    purchaseProducts: total => purchaseProducts(userId, total),
    sendReceipt: invoice => sendReceipt(userId, invoice)
  };
};
```

#### कार्टहेल्पर्स क्लास

अगर आपके पास क्लास है तो यह आसानी से अनुकूलित कर सकते हैं:

```tsx
class CartHelpers {
  constructor(userId) {
    this.userId = userId;
  }
  getProductsSubtotal = products => getProductsSubtotal(this.userId, products);
  applyTaxes = subTotal => applyTaxes(this.userId, subTotal);
  purchaseProducts = total => purchaseProducts(this.userId, total);
  sendReceipt = invoice => sendReceipt(this.userId, invoice);
}
```

कुछ तत्काल लाभ:

- पुनरावृत्ति चर पासिंग को समाप्त करें।
  - DRY: `CartHelpers` पुनरावृत्ति आर्ग्यूमेंट `userId` को छिपा देता है।
  - प्रत्येक विधि केवल आवश्यक आर्ग्यूमेंट्स को स्वीकार करती है। `cart.applyTaxes(subTotal)` पढ़ने में पूरी तरह से अप्रत्याशित नहीं है।
- `CartHelpers` में एकल-आर्ग्यूमेंट फ़ंक्शन अधिक पठनीय होते हैं और उद्देश्य स्पष्ट होता है।

संबंधित फ़ंक्शन्स के समूहन द्वारा हम उग्रित अनुमानित क्षेत्र (exposed surface area) को कम करने का अवसर बनाते हैं (उदाहरण: `checkout()`, `CartHelpers` के 'सार्वजनिक' विधियाँ।)

> कम अनुमानित क्षेत्र = कम बौद्धिक भार, बेहतर परीक्षण और रखरखाव।  
> _इरादा और ध्यान के साथ डिज़ाइन सिस्टम बनाएं। ✨_

#### चेकआउट और CartHelpers का उपयोग  

चलिए देखते हैं कि `checkout()` फ़ंक्शन अब कैसा दिखता है:  

```tsx
export const checkout = ({ userId, products }) => {
  const cart = CartHelpers(userId);

  return Promise.resolve(products)
    .then(products => cart.getProductsSubtotal(products))
    .then(subTotal => cart.applyTaxes(subTotal))
    .then(total => cart.purchaseProducts(total))
    .then(result => cart.sendReceipt(result));
};
```

##### आगे के सुधार के साथ चेकआउट  

> क्या इसे आगे सुधारा जा सकता है? हाँ! हमें तर्क (arguments) को दोहराने की आवश्यकता नहीं है।  

जब किसी फ़ंक्शन के तर्क पिछले फ़ंक्शनों के आउटपुट द्वारा प्रदान किए जाते हैं, तो आप फ़ंक्शन को और भी सरल बना सकते हैं।  

```tsx
export const checkout = ({ userId, products }) => {
  const cart = CartHelpers(userId);

  // 🌈 फ़ंक्शन्स लेगो की तरह ढलते हैं और सामान्य "मानव शब्दों" की तरह पढ़ते हैं! 💅
  return Promise.resolve(products)
    .then(cart.getProductsSubtotal)
    .then(cart.applyTaxes)
    .then(cart.purchaseProducts)
    .then(cart.sendReceipt);
};
```

**यदि तर्कों को एकल (ऑब्जेक्ट) तर्कों में जोड़ना प्राकृतिक नहीं लगता है,** तो अपने फ़ंक्शन्स को तोड़ने **या** उन्हें अधिक उचित रूप से विस्तृत मॉड्यूल्स में जोड़ने पर विचार करें।

#### कहाँ से शुरू करें?

संबंधित फ़ंक्शन्स खोजें और उन्हें एक साथ समूहित करें। (उदा। `CartHelpers`।)

संभावित तार्किक मॉड्यूल्स खोजते समय चुनौतियों का हिस्सा यह पहचानना है कि पहले से ही संबंधित कोड कहाँ है।

##### फ़ंक्शन्स को संबंधित क्या बनाता है?

एक अच्छी चाल: फ़ंक्शन तर्कों में दोहराव खोजें। क्या कोई संबंध या अंतर्निहित जिम्मेदारी खेल रही है?

- ✅ फ़ंक्शन्स जिनमें दोहराई गई, सामान्य तर्क हैं। (उदा। यदि 4 विधियाँ `userRewards` स्वीकार करती हैं, तो संभवतः आपको एक `Rewards` या अन्य मॉड्यूल की आवश्यकता है।)
- ✅ तर्क जो पूर्व फ़ंक्शन्स के आउटपुट द्वारा सीधे प्रदान किए जाते हैं। (चरणों की अनुक्रम। उदा। `Extract`, `Transform`, `Load`।)

- ❌ उन फ़ंक्शन्स में आम प्रिफ़िक्स या सुफ़िक्स नामकरण होना?
