## पाइपलाइन के मास्टर: स्टेट पास करना  

क्या आपने फ़ंक्शनल पाइपलाइन में स्टेट पास करने में कठिनाइयों का सामना किया है?  

आपके कोड का संगठन (या उसकी कमी) सीधे इस बात को प्रभावित करता है कि स्टेट को इधर‑उधर ले जाना कितना आसान है।  

इस लेख में हम पाइपलाइन के माध्यम से स्टेट पास करने की एक प्रभावी तकनीक का अन्वेषण करेंगे। साथ‑साथ हम अपने कोड की संरचना और पठनीयता को भी बेहतर बनाएँगे।  

नीचे दिया गया “वास्तविक” स्निपेट इस लेख का फोकस होगा: एक चेकआउट फ़ंक्शन, जो `userId` और `products` की एरे लेता है। यह एक Promise‑चेन लौटाता है जो क्रमशः 4 फ़ंक्शन चलाता है।

```tsx
const checkout = (userId: number, products: number[]) => {
  return getProductsSubtotal(userId, products)
    .then(subTotal => applyTaxes(userId, subTotal))
    .then(total => purchaseProducts(userId, total))
    .then(result => sendReceipt(userId, result));
};
```

रुको ज़रा, इस कोड को देखिए—जावास्क्रिप्ट में पाइपलाइन की बात करें तो यह काफी ठीक‑ठाक है!

यह कुछ सूक्ष्म समस्याओं से ग्रस्त है, जो मिलकर अधिक गंभीर समस्याओं में बदल सकती हैं।

एक समस्या यह है कि हम `userId` को प्रत्येक (तार्किक रूप से संबंधित) फ़ंक्शन में बार‑बार पास कर रहे हैं।  
अब इसे किसी अन्य मुद्दे के साथ जोड़ें, जिसे डेवलपर्स और TypeScript दोनों आसानी से नज़रअंदाज़ कर देते हैं: संख्यात्मक आर्ग्यूमेंट्स का क्रम उलट देना एक चुपचाप बग पैदा कर सकता है। (देखिए `applyTaxes` और `purchaseProducts`। _क्या `userId` पहले आता है या `amount`?_)

इस कोड को कैसे सुधारा जाए, इस पर निर्णय लेने से पहले, चलिए कुछ फायदे‑नुकसान की पहचान करते हैं।

### Pros & Cons

#### Pros

- क्लोज़र का अच्छा उपयोग! `userId` और `products` को एक बार पास किया गया!
- आर्ग्यूमेंट नामकरण में निरंतरता।
- चेकआउट के लिए 4 मुख्य फ़ंक्शन की अपेक्षाकृत प्रभावी और संक्षिप्त संयोजन।
- “फ़्री” एरर फ़्लो कंट्रोल। (किसी भी नेस्टेड फ़ंक्शन से त्रुटियाँ ऊपर तक उठती हैं, `checkout()` द्वारा लौटाए गए Promise को रिजेक्ट करती हैं.)

#### Cons

- `userId` को बार‑बार पास करना थकाऊ है।
- फ़ंक्शन एक‑पैरामीटर (यूनरी) नहीं हैं। _यह संयोजकता को प्रभावित करता है। क्यों, देखिए [अंतिम उदाहरण](#checkout-with-further-improvements) में।_
- प्रत्येक फ़ंक्शन क्या रिटर्न करता है, यह स्पष्ट नहीं हो सकता। (क्या यह ई‑मेल भेजने का परिणाम है, या वह `result` वैरिएबल? या कुछ और?)
- नई कार्यक्षमता जोड़ना स्पष्ट नहीं है (उदाहरण : मान लीजिए हमें ग्राहक की डिस्काउंट/क्रेडिट/पॉइंट्स आदि लोड करनी हों)।
- कभी‑कभी “temp” पैरामीटर नाम (जैसे प्रत्येक `.then(param => {})` में) संदर्भ जोड़ते हैं। लेकिन समय के साथ ये नामकरण का बोझ बन सकते हैं।

### Solution, Part 1: Make a module!

यह तकनीक संबंधित फ़ंक्शनों को एक ही मॉड्यूल (जैसे `CartHelpers`) में व्यवस्थित करने के बारे में है। यह किसी विशिष्ट पैटर्न को बाध्य नहीं करती। [फ़ैक्टरी फ़ंक्शन](#carthelpers-factory), [क्लासेज़](#carthelpers-class), क्लोज़र, मिक्सिन आदि का अन्वेषण करें। अपनी परियोजना & टीम के लिये जो समझ में आए, वही चुनें।

#### CartHelpers Factory

`CartHelpers` मॉड्यूल का उदाहरण, जहाँ `userId` को एक बार पास किया जाता है, और सभी मेथड्स एक‑आर्ग्यूमेंट वाले होते हैं।

```tsx
const CartHelpers = (userId: number) => {
  return {
    getProductsSubtotal: products => getProductsSubtotal(userId, products),
    applyTaxes: subTotal => applyTaxes(userId, subTotal),
    purchaseProducts: total => purchaseProducts(userId, total),
    sendReceipt: invoice => sendReceipt(userId, invoice)
  };
};
```

#### CartHelpers क्लास

यदि क्लासेज़ आपका पसंदीदा तरीका है, तो इसे आसानी से अनुकूलित किया जा सकता है:

```tsx
class CartHelpers {
  constructor(userId) {
    this.userId = userId;
  }
  getProductsSubtotal = products => getProductsSubtotal(this.userId, products);
  applyTaxes = subTotal => applyTaxes(this.userId, subTotal);
  purchaseProducts = total => purchaseProducts(this.userId, total);
  sendReceipt = invoice => sendReceipt(this.userId, invoice);
}
```

कुछ त्वरित लाभ:

- दोहराए जाने वाले वेरिएबल पास करने को समाप्त करता है।
  - DRY: `CartHelpers` बार‑बार पास होने वाले `userId` तर्क को अमूर्त बनाता है।
  - प्रत्येक मेथड **केवल** आवश्यक तर्क लेता है। इसलिए `cart.applyTaxes(subTotal)` पढ़ने में पूरी तरह से स्पष्ट रहता है।
- `CartHelpers` में एक‑आर्ग्यूमेंट वाले फ़ंक्शन अधिक पठनीय होते हैं और उनका उद्देश्य स्पष्ट रहता है।

संबंधित फ़ंक्शन को समूहित करके, हम एक्सपोज़्ड सतह क्षेत्र को घटाने का अवसर बनाते हैं (जैसे `checkout()`, `CartHelpers` के ‘public’ मेथड्स)।

> कम सतह क्षेत्र === कम संज्ञानात्मक भार, बेहतर परीक्षण और रख‑रखाव।
> _इरादा और फोकस के साथ डिज़ाइन सिस्टम बनाएं। ✨_

#### Checkout & CartHelpers उपयोग

आइए देखें कि `checkout()` फ़ंक्शन अब कैसे दिखता है:

```tsx
export const checkout = ({ userId, products }) => {
  const cart = CartHelpers(userId);

  return Promise.resolve(products)
    .then(products => cart.getProductsSubtotal(products))
    .then(subTotal => cart.applyTaxes(subTotal))
    .then(total => cart.purchaseProducts(total))
    .then(result => cart.sendReceipt(result));
};
```

##### आगे के सुधारों के साथ Checkout

> क्या इसे और बेहतर बनाया जा सकता है? हाँ! हमें आर्ग्यूमेंट दोहराने की ज़रूरत नहीं है!

जब फ़ंक्शन के आर्ग्यूमेंट पहले वाले फ़ंक्शन के आउटपुट से आते हैं, तो कोड को और भी सरल किया जा सकता है।

```tsx
export const checkout = ({ userId, products }) => {
  const cart = CartHelpers(userId);

  // 🌈 फ़ंक्शन लेगो की तरह स्टैक होते हैं और सामान्य "मानव शब्दों" की तरह पढ़ते हैं! 💅
  return Promise.resolve(products)
    .then(cart.getProductsSubtotal)
    .then(cart.applyTaxes)
    .then(cart.purchaseProducts)
    .then(cart.sendReceipt);
};
```

**यदि एकल (ऑब्जेक्ट) आर्ग्यूमेंट में पैरामीटर मिलाना असहज लगता है,** तो अपने फ़ंक्शन को **तोड़ने** या उन्हें अधिक उपयुक्त स्कोप वाले मॉड्यूल में **एकत्रित करने** पर विचार करें।

#### कहाँ से शुरू करें?

संबंधित फ़ंक्शन खोजें और उन्हें एक साथ समूहित करें (जैसे `CartHelpers`)।

संभव तर्कसंगत मॉड्यूल खोजते समय सबसे बड़ी चुनौती अक्सर यह होती है कि पहले ही संबंधित कोड की पहचान कैसे की जाए।

##### फ़ंक्शन को संबंधित क्या बनाता है?

एक सरल ट्रिक: फ़ंक्शन पैरामीटर में दोहराव देखें। क्या कोई संबंध मौजूद है? या कोई अंतर्निहित ज़िम्मेदारी?

- ✅ फ़ंक्शन जिनके पास दोहराए गए, सामान्य आर्ग्यूमेंट होते हैं। (उदाहरण : यदि 4 मेथड `userRewards` लेते हैं, तो संभवतः आपको एक `Rewards` या अन्य मॉड्यूल चाहिए।)
- ✅ फ़ंक्शन जिनके आर्ग्यूमेंट सीधे पिछले फ़ंक्शन के आउटपुट से आते हैं। (स्टेप की श्रृंखला : उदाहरण `Extract`, `Transform`, `Load`.)
- ❌ कोई भी चीज़ जो फीचर क्षेत्र से मोटे तौर पर जुड़ी हो, “प्रोडक्ट खरीदना?”  
- ❌ फ़ंक्शन जिनके नाम में सामान्य प्रीफ़िक्स या सफ़िक्स हो?  
- ❌ फ़ंक्शन जो बड़े ऑब्जेक्ट को आर्ग्यूमेंट के रूप में लेते हैं, जबकि अंदर से केवल कुछ ही मानों का उपयोग करते हैं। (उदाहरण : `applyTaxes({ user, business, rewards, kitchenSink })` बनाम `applyTaxes({ subTotal })`)

जबकि मॉड्यूल डिज़ाइन के लिए कोई एकल “सही उत्तर” नहीं होता, 2‑3 संगठन विकल्पों की पहचान करना मददगार रहता है — रूपरेखा बनाएँ, “फ़ैंटेसी” कोड लिखें, पूछें “क्या यह खुशी देता है?”

<aside>
📌 अक्सर मॉड्यूल संगठन में कुछ प्रयास लगते हैं जब तक आपका डोमेन मॉडल स्पष्ट नहीं हो जाता। इसे परिपूर्ण बनाने के लिए ज़्यादा तनाव न लें।
</aside>

> आपको लग सकता है `cart.sendReceipt()` भुगतान‑संबंधी मेथड्स के साथ नहीं जाना चाहिए। शायद `customerNotifications.sendReceipt()` ग्राहक संदेशों के लिए बेहतर जगह है। यदि `CartHelper` पर्याप्त महत्व रखता है, तो वह आंतरिक रूप से सभी आवश्यक **_services_** को कॉल करते हुए एक **_controller_** बन सकता है, जैसे `customerNotifications`।

#### आप कैसे जानेंगे कि आप मदद कर रहे हैं?

यदि पढ़ने में आसानी बनी रहती है जबकि आप एडे‑हॉक आर्ग्यूमेंट्स को हटाते हैं, **बधाई हो!!!** आपने संभवतः एक स्पष्ट और टिकाऊ स्कोप वाला मॉड्यूल बना लिया है!

- मध्यवर्ती आर्ग्यूमेंट्स को हटाने से “लेयर” स्वाभाविक रूप से उभरते हैं।
- एडे‑हॉक कोड को गलत जगह पर डालना _कठिन_ होना चाहिए!

तो, सवाल उठता है, हम कार्यक्षमता कहाँ जोड़ें?

मेरे अनुभव में, कार्यक्षमता जोड़ते समय दो मुख्य रणनीतियों का मूल्यांकन किया जाता है:

1.  मौजूदा मेथड को विस्तारित/रिफैक्टर करें। (जब नया कोड मौजूदा कोड के पर्याप्त करीब हो।)
2.  चेन में इच्छित स्थान पर एक नया (पाँचवाँ) फ़ंक्शन बनाएं। (जब नया कोड मौजूदा फ़ंक्शनों से असंबंधित हो।)

आखिरकार यह तय करना आसान हो जाता है कि नई कार्यक्षमता कहाँ रखनी है। (उदाहरण के लिए `cart.applyDiscounts()`, `cart.applyTaxes()`, `rewards.getBalance()`।)
