## パイプラインの達人：状態の受け渡し

関数型パイプラインを使って状態を受け渡す際に課題に直面したことはありませんか？

コードの構成（あるいはその欠如）は、状態の受け渡しのしやすさに直接影響します。

この記事では、パイプラインを通じて状態を受け渡す効果的なテクニックを探ります。その過程で、コードの構成と可読性を改善していきます。

この記事の焦点となる「実際の」スニペットは以下の通りです。`userId`と`products`の配列を受け取るチェックアウト関数です。4つの関数を順次実行するPromiseチェーンを返します。

```tsx
const checkout = (userId: number, products: number[]) => {
  return getProductsSubtotal(userId, products)
    .then(subTotal => applyTaxes(userId, subTotal))
    .then(total => purchaseProducts(userId, total))
    .then(result => sendReceipt(userId, result));
};
```

ちょっと待ってください、このコードはJSのパイプラインとしてはかなりまともです！

このコードには、いくつかの微妙な問題があり、それらが組み合わさってより深刻な問題に発展する可能性があります。

1つ目の問題は、論理的に関連する各関数に`userId`を繰り返し渡していることです。さらに、開発者やTypeScriptでも見逃しがちな別の問題と組み合わせると、数値引数の順序を間違えると静かなバグが発生しやすくなります。（`applyTaxes`と`purchaseProducts`を見てください。_最初に来るのは`userId`ですか、それとも`amount`ですか？_）

このコードを改善する方法を決める前に、長所と短所を確認しましょう。

### 長所と短所

#### 長所

- クロージャの良い使い方！`userId`と`products`を一度だけ渡している！
- 一貫した引数名。
- チェックアウトの4つの主要関数を比較的効果的かつ簡潔に合成している。
- 「無料」のエラーフロー制御。（エラーはネストされた関数からバブルアップし、`checkout()`が返すPromiseをrejectする。）

#### 短所

- `userId` をあちこちに繰り返し渡すのが面倒。
- 関数が単一パラメータ（別名 `unary`）ではない。_これが合成性に影響する。理由は[最終例](#checkout-with-further-improvements)を参照。_
- 各関数が何を返すかがわかりにくい。（メール送信結果か、それとも `result` 変数か？ それとも？）
- 機能追加がわかりにくい（例：顧客の割引/クレジット/ポイントなどを読み込む必要がある場合）。
- 時には「一時的な」パラメータ名（各 `.then(param => {})` のように）がコンテキストを追加する。しかし時間が経つにつれて、命名のゴミ溜めになりがちだ。

### 解決策、パート1: モジュールを作成する！

このテクニックは、関連する関数を単一のモジュール（例：`CartHelpers`）に整理することです。特定のパターンを強制しません。[ファクトリ関数](#carthelpers-factory)、[クラス](#carthelpers-class)、クロージャ、ミックスインなどを探求してください。あなたのプロジェクトとチームに合ったものを見つけてください。

#### CartHelpers ファクトリ

`CartHelpers` モジュールの例。`userId` を一度だけ渡し、すべてのメソッドは単一引数です。

```tsx
const CartHelpers = (userId: number) => {
  return {
    getProductsSubtotal: products => getProductsSubtotal(userId, products),
    applyTaxes: subTotal => applyTaxes(userId, subTotal),
    purchaseProducts: total => purchaseProducts(userId, total),
    sendReceipt: invoice => sendReceipt(userId, invoice)
  };
};
```

#### CartHelpers クラス

クラスが好みなら、簡単に適応できる：

```tsx
class CartHelpers {
  constructor(userId) {
    this.userId = userId;
  }
  getProductsSubtotal = products => getProductsSubtotal(this.userId, products);
  applyTaxes = subTotal => applyTaxes(this.userId, subTotal);
  purchaseProducts = total => purchaseProducts(this.userId, total);
  sendReceipt = invoice => sendReceipt(this.userId, invoice);
}
```

すぐに得られる利点：

- 繰り返し変数を渡す手間がなくなる。
  - DRY: `CartHelpers` が繰り返し引数 `userId` を抽象化する。
  - 各メソッドは**_必要な引数だけ_**を受け取る。`cart.applyTaxes(subTotal)` は読んでいてまったく驚かない。
- `CartHelpers` 内の単一引数関数は可読性が高く、目的が明確になる。

関連関数をグループ化することで、露出する表面積を減らす機会が生まれる（例：`checkout()`、`CartHelpers` の「公開」メソッド）。

> 表面積が小さい === 認知負荷が減り、テストと保守性が向上する。
> _意図と焦点を持ってシステムを設計しよう。✨_

#### チェックアウトとCartHelpersの使用例

では、`checkout()` 関数がどのようになったか見てみましょう。

```tsx
export const checkout = ({ userId, products }) => {
  const cart = CartHelpers(userId);

  return Promise.resolve(products)
    .then(products => cart.getProductsSubtotal(products))
    .then(subTotal => cart.applyTaxes(subTotal))
    .then(total => cart.purchaseProducts(total))
    .then(result => cart.sendReceipt(result));
};
```

##### さらなる改善を施したチェックアウト

> さらに改善できるでしょうか？ できます！ 引数をまったく繰り返す必要はありません。

関数の引数が前の関数の出力によって提供される場合、コードをさらに簡略化できます。

```tsx
export const checkout = ({ userId, products }) => {
  const cart = CartHelpers(userId);

  // 🌈 Functions stack like Lego & read like normal "Human Words!" 💅
  return Promise.resolve(products)
    .then(cart.getProductsSubtotal)
    .then(cart.applyTaxes)
    .then(cart.purchaseProducts)
    .then(cart.sendReceipt);
};
```

**パラメータを単一の（オブジェクト）引数にまとめるのが不自然に感じる場合は、** 関数を分割するか、**あるいは** より適切なスコープのモジュールにまとめることを検討してください。

#### どこから始めるか？

関連する関数を見つけて、それらをまとめる（例：`CartHelpers`）。

論理的なモジュールを見つける際の課題の一つは、そもそも関連するコードを特定することにある。

##### 関数が関連しているとはどういうことか？

便利なコツ：関数のパラメータに繰り返しがないか探す。そこに関係性があるか？あるいは根底にある責務は何か？と自問する。

- ✅ 繰り返し現れる共通の引数を持つ関数。（例：4つのメソッドが `userRewards` を受け取るなら、`Rewards` などのモジュールが必要な可能性が高い。）
- ✅ 引数が先行する関数の出力によって直接提供される関数。（一連のステップ。例：`Extract`、`Transform`、`Load`。）
- ❌ 機能領域に漠然と関連しているもの、「製品購入？」
- ❌ 共通の接頭辞や接尾辞の命名を持つ関数？
- ❌ オブジェクト内の一部の値しか使わないにもかかわらず、大きなオブジェクトを引数として必要とする関数。（例：`applyTaxes({ user, business, rewards, kitchenSink })` 対 `applyTaxes({ subTotal })`）

モジュール設計に唯一の「正解」はありませんが、2〜3の構成案を考え、アウトラインを描き、「理想」のコードを書き、「ときめくか？」と自問するのが役立ちます。

<aside>
📌 ドメインモデルが固まるまでには、何度かモジュール構成を試行錯誤するのが普通です。完璧を求めすぎないでください。
</aside>

> `cart.sendReceipt()` は支払い関連のメソッドと一緒にすべきではないと感じるかもしれません。顧客へのメッセージ送信は `customerNotifications.sendReceipt()` の方が適切でしょう。`CartHelper` の重要度が十分に高ければ、内部で `customerNotifications` などの必要な**_サービス_**を呼び出す**_コントローラー_**として機能する可能性があります。

#### 役に立っているかどうかはどう判断するか？

アドホックな引数を排除しても可読性が損なわれなければ、**おめでとうございます！！！** 明確で堅牢なスコープを持つモジュールが構築できた可能性が高いです！

- 中間的な引数を排除することで、自然と「レイヤー」が浮かび上がってきます。
- アドホックなコードを間違った場所に放り込むのが難しくなる**_はず_**です！

では、当然の疑問として、機能をどこに追加すべきでしょうか？

私の経験では、機能追加時に検討すべき主要な戦略が2つあります。

1. 既存メソッドの拡張・リファクタリング（新しいコードが既存コードに十分近い場合）。
2. チェーンの適切な位置に新しい（5番目の）関数を作成する（新しいコードが既存関数と無関係な場合）。

最終的にこれにより、新しい機能の所属先を決めやすくなります（例：`cart.applyDiscounts()`、`cart.applyTaxes()`、`rewards.getBalance()`）。

### 結論

複雑なパイプラインを通じて状態を渡すのは厄介な場合があります。しかし、少しリファクタリングの練習を積めば、より読みやすく、認知的負荷の低いコードを書けるようになるでしょう。
