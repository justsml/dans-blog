Sono le2 del mattino. L’allarme di PagerDuty suona a tutto volume.

Apri i log e trovi questo:

```
Error: Cannot read properties of undefined (reading 'id')
    at processTicksAndRejections (node:internal/process/task_queues:96:5)
```

È tutto. Nessun nome di funzione. Nessun numero di riga. Nessun percorso file. Solo “processTicksAndRejections”.

Benvenuti in JavaScript asincrono, dove le tracce di stack sono inventate e i numeri di riga non contano.

---

## Perché le tracce di stack si rompono


Nel codice sincrono, lo Stack delle Chiamate è una genealogia ben definita: A chiama B, B chiama C. Quando C va in crash, è possibile vedere esattamente come ci si è arrivati.

Nel codice asincrono (`async/await`), ogni parola chiave `await` è un punto di sospensione.

Quando esegui `await`, la tua funzione viene staccata dallo stack. Viene inserita in un congelatore criogenico chiamato Coda dei Microtask. Lo stack è ora vuoto (o occupato da qualcos’altro).

Quando la Promise si risolve, la tua funzione viene scongelata e rimessa sullo stack. Ma la cronologia è sparita.

Il motore non ha idea di chi ha chiamato `await` 500 millisecondi fa. Sa solo di avere un compito da eseguire.

## Tentativi di V8 di risolvere il problema

Node.js cerca di aiutare. Abbiamo:

1.  `Error.captureStackTrace()`: cattura lo stack *al momento della creazione*. Inutile se l'errore viene lanciato più tardi.  
2.  `--async-stack-traces`: un flag che fa sì che Node.js mantenga uno “shadow stack” delle catene di promise.  
    *   **Costo**: rende la tua app circa il 30 % più lenta.  
    *   **Risultato**: è d’aiuto, ma il rumore cresce rapidamente.

---

## La vera soluzione: AsyncLocalStorage

Se vuoi sopravvivere in produzione, smetti di guardare gli stack trace. Concentrati sulla causalità.

Dobbiamo collegare il contesto (User ID, Request ID) al “filo” di esecuzione, anche quando salta tra lo Stack e la Microtask Queue.

Node.js dispone di uno strumento integrato per questo: `AsyncLocalStorage`.

```javascript
import { AsyncLocalStorage } from 'async_hooks';

const context = new AsyncLocalStorage();

// 1. Wrap the request
context.run({ requestId: '123' }, () => {
  // 2. Call deep async code
  await processOrder();
});

// 3. Deep inside processOrder:
async function processOrder() {
  await db.query();
  
  // Magic! We can still see the requestId
  const { requestId } = context.getStore();
  console.log(`[${requestId}] Failed to process order`);
}
```

Non importa quanti `await` si verifichino nel frattempo. Il contesto sopravvive.

---

## Playbook per la produzione

1.  Smetti di fidarti di `err.stack`. È incompleto per progettazione.  
2.  Usa il logging strutturato. Allegati `requestId` a ogni singola riga di log tramite `AsyncLocalStorage`.  
3.  Traccia, non stack. Usa OpenTelemetry. Visualizza la catena causale tra i servizi, che è ciò di cui hai realmente bisogno.

Il tuo codice è asincrono. Il tuo contesto di debug non dovrebbe esserlo.
