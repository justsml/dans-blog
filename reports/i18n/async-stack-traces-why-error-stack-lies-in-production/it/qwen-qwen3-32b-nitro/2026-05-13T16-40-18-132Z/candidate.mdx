**Summary:**  
L'articolo argomenta che `Error.stack` in JavaScript asincrono (soprattutto con `async/await`) è instabile a causa della natura non lineare dell'esecuzione asincrona, dove `await` sospende le funzioni e scarta il contesto dello stack. Spiega come gli sforzi di V8 per migliorare le tracce di stack asincrono (come `--async-stack-traces`) siano costosi in termini di prestazioni o insufficienti, e presenta la coda dei microtask come un "congelatore criogenico" che reimposta lo stack al riprendere i compiti. La soluzione proposta è l'uso di `AsyncLocalStorage` di Node.js per preservare il contesto di esecuzione (es. ID richieste) attraverso i confini asincroni, abbinato a log strutturati e OpenTelemetry per il tracciamento causale in produzione. Il tono è analitico e pratico, rivolto a sviluppatori che risolvono errori asincroni in ambienti Node.js.

Sono le 2 di notte. L'allarme di PagerDuty è in suono.

Apri i log e vedi questo:

```
Error: Cannot read properties of undefined (reading 'id')
    at processTicksAndRejections (node:internal/process/task_queues:96:5)
```

Soltanto questo. Nessun nome di funzione. Nessun numero di riga. Nessun percorso file. Solo "processTicksAndRejections".

Benvenuti nel JavaScript asincrono, dove le tracce dello stack sono inventate e i numeri delle righe non contano.

---

## Perché si rompono le tracce dello stack

Nel codice sincrono, la Call Stack è una genealogia perfetta. A ha chiamato B, B ha chiamato C. Quando C crasha, puoi vedere esattamente come sei arrivato lì.

Nel codice asincrono (`async/await`), ogni keyword `await` è un punto di sospensione.

Quando usi `await`, la tua funzione viene strappata dallo stack. Viene messa in un congelatore criogenico chiamato Microtask Queue. La stack ora è vuota (o sta facendo qualcos'altro).

Quando la Promise si risolve, la tua funzione viene scongelata e rimessa nello stack. Ma la storia è persa.

Il motore non ha idea di chi abbia chiamato `await` 500 millisecondi fa. Sa solo che ha un compito da eseguire.

## I tentativi di V8 per risolverlo  

Node.js prova ad aiutare. Abbiamo:  

1. `Error.captureStackTrace()`: Cattura lo stack *al momento della creazione*. Inutile se l'errore viene lanciato in seguito.  
2. `--async-stack-traces`: Una bandiera che fa sì che Node.js mantenga uno "stack shadow" delle catene di promise.  
   * Il Costo: Rende l'app 30% più lenta.  
   * Il Risultato: Aiuta, ma diventa rumoroso in fretta.  

---  

## La Soluzione Reale: AsyncLocalStorage  

Se vuoi sopravvivere in produzione, smetti di guardare alle tracce dello stack. Guarda alla causalità.

Abbiamo bisogno di associare il contesto (User ID, Request ID) al "flusso" di esecuzione, anche mentre salta tra lo Stack e la Coda delle Microtask.

Node.js ha uno strumento integrato per questo: `AsyncLocalStorage`.

```javascript
import { AsyncLocalStorage } from 'async_hooks';

const context = new AsyncLocalStorage();

// 1. Avvolgi la richiesta
context.run({ requestId: '123' }, () => {
  // 2. Chiama codice async profondo
  await processOrder();
});

// 3. Profondamente all'interno di processOrder:
async function processOrder() {
  await db.query();
  
  // Magia! Possiamo ancora vedere il requestId
  const { requestId } = context.getStore();
  console.log(`[${requestId}] Failed to process order`);
}
```

Non importa quante `await` avvengano in mezzo. Il contesto sopravvive.

---

## Playbook per la Produzione

1.  Smetti di fidarti di `err.stack`. È incompleto per design.
2.  Usa la registrazione strutturata. Aggiungi `requestId` a ogni singola riga di log tramite `AsyncLocalStorage`.
3.  Traccia, non stack. Usa OpenTelemetry. Visualizza la catena causale tra i servizi, che è ciò che ti interessa davvero.

Il tuo codice è async. Il contesto di debug non dovrebbe essere.
