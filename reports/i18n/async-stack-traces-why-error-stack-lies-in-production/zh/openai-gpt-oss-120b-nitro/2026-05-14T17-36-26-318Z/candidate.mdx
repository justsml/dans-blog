现在是凌晨2点，PagerDuty 警报声响个不停。

你打开日志，看到：

```
Error: Cannot read properties of undefined (reading 'id')
    at processTicksAndRejections (node:internal/process/task_queues:96:5)
```

仅此而已。没有函数名、没有行号、没有文件路径。只有 “processTicksAndRejections”。

欢迎来到 async JavaScript 的世界，这里堆栈跟踪是凭空编造的，行号根本不重要。

---

## 为什么堆栈跟踪会失效

在同步代码中，调用栈是一条清晰的血缘链。A 调用了 B，B 调用了 C。当 C 崩溃时，你可以准确看到是怎么到达这里的。

在 async 代码（`async/await`）里，每个 `await` 关键字都是一个挂起点。

当你 `await` 时，函数会从栈中被撕下，放进一个叫做微任务队列的低温冷冻库。此时栈为空（或正忙于别的事）。

当 Promise 解决后，函数被解冻并重新推回栈上。但历史记录已经消失。

引擎根本不知道 500 毫秒前是谁调用了 `await`，它只知道有个任务要执行。

---

## V8 的修复尝试

Node.js 试图帮忙。我们有：

1.  `Error.captureStackTrace()`：在错误创建时捕获栈。若错误稍后才抛出，则毫无用处。
2.  `--async-stack-traces`：一个标志，使 Node.js 保留 promise 链的“影子栈”。
    *   成本：会让你的应用慢约 30%。
    *   结果：有帮助，但很快就会产生大量噪声。

---

## 真正的解决方案：AsyncLocalStorage

如果你想在生产环境中生存下来，别再盯着堆栈跟踪了。关注因果关系。

我们需要把上下文（用户 ID、请求 ID）绑定到执行的“线程”，即使它在栈和微任务队列之间跳转。

Node.js 内置了一个工具：`AsyncLocalStorage`。

```javascript
import { AsyncLocalStorage } from 'async_hooks';

const context = new AsyncLocalStorage();

// 1. 包裹请求
context.run({ requestId: '123' }, () => {
  // 2. 调用深层异步代码
  await processOrder();
});

// 3. 在 processOrder 的深处：
async function processOrder() {
  await db.query();
  
  // 魔法！我们仍然能看到 requestId
  const { requestId } = context.getStore();
  console.log(`[${requestId}] Failed to process order`);
}
```

中间出现多少个 `await` 都无关紧要。上下文会一直保留下来。

## 生产实战手册

1. 停止信任 `err.stack`。它本身就不完整。
2. 使用结构化日志。通过 `AsyncLocalStorage` 将 `requestId` 附加到每一条日志上。
3. 用追踪取代堆栈。采用 OpenTelemetry。它能够可视化跨服务的因果链，这才是你真正关心的。

你的代码是异步的，调试上下文也不该例外。
