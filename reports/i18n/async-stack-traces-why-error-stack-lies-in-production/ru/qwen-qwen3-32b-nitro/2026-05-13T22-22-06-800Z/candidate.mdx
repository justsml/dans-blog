2 часа ночи. Сирена PagerDuty ревёт.  
Вы открываете логи и видите это:  

```
Error: Cannot read properties of undefined (reading 'id')
    at processTicksAndRejections (node:internal/process/task_queues:96:5)
```  

Это всё. Нет имени функции. Нет номера строки. Нет пути к файлу. Только "processTicksAndRejections".  

Добро пожаловать в асинхронный JavaScript, где трассировки стека составлены искусственно, а номера строк не имеют значения.  

---  

## Почему разрываются трассировки стека  

В синхронном коде стек вызовов — это красивая генеалогия. A вызвал B, B вызвал C. Когда C падает, вы видите, как именно вы оказались там.  

В асинхронном коде (`async/await`) каждое ключевое слово `await` — это точка приостановки.  

Когда вы используете `await`, ваша функция отрывается от стека. Её помещают в криогенный холодильник под названием Microtask Queue. Стек теперь пуст (или делает что-то другое).  

Когда промис разрешается, ваша функция оттаивает и бросается обратно на стек. Но история утеряна.

Двигок не имеет понятия, кто вызвал `await` 500 миллисекунд назад. Он просто знает, что у него есть задача, которую нужно выполнить.

---

## Попытки V8 решить проблему

Node.js пытается помочь. У нас есть:

1. `Error.captureStackTrace()`: Захватывает стек *при создании*. Невероятно бесполезен, если ошибка выброшена позже.
2. `--async-stack-traces`: Флаг, заставляющий Node.js вести "теневой стек" цепочек промисов.
    *   Стоимость: Уменьшает производительность приложения на 30%.
    *   Результат: Помогает, но быстро становится шумным.

---

## Реальное решение: AsyncLocalStorage

Если вы хотите выжить в продакшене, перестаньте смотреть на трассировки стека. Следите за причинно-следственными связями.

Нам нужно привязать контекст (ID пользователя, ID запроса) к "потоку" выполнения, даже если он перескакивает между Стеком и Микрозадачной очередью.

Node.js предоставляет для этого встроенную утилиту: `AsyncLocalStorage`.

```javascript
import { AsyncLocalStorage } from 'async_hooks';

const context = new AsyncLocalStorage();

// 1. Оберните запрос
context.run({ requestId: '123' }, () => {
  // 2. Вызовите глубокий асинхронный код
  await processOrder();
});

// 3. Глубоко внутри processOrder:
async function processOrder() {
  await db.query();
  
  // Волшебство! Мы всё ещё видим requestId
  const { requestId } = context.getStore();
  console.log(`[${requestId}] Не удалось обработать заказ`);
}
```

Не имеет значения, сколько `await` происходит между ними. Контекст выживает.

---

## План действий для продакшена

1.  Перестаньте доверять `err.stack`. Он неполон по дизайну.
2.  Используйте структурированный логгинг. Прикрепляйте `requestId` к каждой строке логов через `AsyncLocalStorage`.
3.  Трассируйте, а не трассируйте стек. Используйте OpenTelemetry. Он визуализирует причинно-следственные цепочки между сервисами, что действительно важно.

Ваш код асинхронный. Ваш контекст отладки не должен быть.
