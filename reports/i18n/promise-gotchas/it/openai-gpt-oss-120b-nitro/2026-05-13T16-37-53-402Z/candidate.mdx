###Le promesse non funzionano come gli altri valori

Non puoi stampare il loro valore come faresti con la maggior parte dei valori:

```js
// Questo non ha senso con le promesse:
console.log(Promise.resolve(42));

// Dobbiamo usare l'interfaccia `.then`:
Promise.resolve(42).then(value => console.log(value));
```

### Le promesse non ti avvertono quando commetti un errore

Beh, un errore probabile.

Per vari motivi, il TC39 ha deciso che `.then` e `.catch` possono ricevere `null`. Ad esempio, `.then(null, null)` è valido e il comportamento richiesto è saltare quel “passo” nella catena.

La sfortunata conseguenza è che è molto facile incasinare le cose.

##### Per Esempio

Vediamo una mini sfida: quale delle seguenti opzione(i) eseguirà `console.log` con 42?

```js
// Option #1:
Promise.resolve(42).then(console.log());

// Option #2:
Promise.resolve(42).then(console.log);

// Option #3:
Promise.resolve(42).then(value => console.log(value));

// Option #4:
Promise.resolve(42)
  .then(console.log())
  .then(console.log);
```

##### La Risposta

La risposta è #2, #3 e #4.

Perché? Esaminiamo i **tipi** di ciò che è stato passato a `.then()`:

```js
var arg1 = console.log();
var arg2 = console.log;
var arg3 = value => console.log(value);

typeof arg1 === "undefined";
typeof arg2 === "function";
typeof arg3 === "function";
```

Ti chiedi ancora come funzioni la quarta Opzione?

È effettivamente eseguito così:

```js
// Option #4 - effectively
Promise.resolve(42)
  .then(undefined) // this has no affect on the value, it will be handed to following `.then(fn)`
  .then(console.log);
```
