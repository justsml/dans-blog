### Les promesses ne fonctionnent pas comme les autres valeurs

Vous ne pouvez pas imprimer leur valeur comme la plupart des valeurs :

```js
// Cela n'a aucun sens avec les promesses :
console.log(Promise.resolve(42));

// Nous devons utiliser l'interface `.then` :
Promise.resolve(42).then(value => console.log(value));
```

### Les promesses ne vous avertissent pas en cas d'erreur

Eh bien, une erreur probable.

Pour diverses raisons, le TC39 a décidé que `.then` et `.catch` peuvent recevoir `null`. Par exemple, `.then(null, null)` est valide et le comportement requis est d'ignorer cette « étape » dans la chaîne.

La conséquence fâcheuse de cela est qu'il est très facile de tout gâcher.

##### Par l'exemple

Examinons un petit défi : quelle(s) option(s) suivante(s) vont afficher `42` avec `console.log` ?

```js
// Option #1:
Promise.resolve(42).then(console.log());

// Option #2:
Promise.resolve(42).then(console.log);

// Option #3:
Promise.resolve(42).then(value => console.log(value));

// Option #4:
Promise.resolve(42)
  .then(console.log())
  .then(console.log);
```

##### La réponse

La réponse est #2, #3 et #4.

Pourquoi ? Regardons les **types** de ce qui a été passé à `.then()` :

```js
var arg1 = console.log();
var arg2 = console.log;
var arg3 = value => console.log(value);

typeof arg1 === "undefined";
typeof arg2 === "function";
typeof arg3 === "function";
```

Vous vous demandez encore comment fonctionne la 4ᵉ option ?

C’est effectivement exécuté ainsi :

```js
// Option #4 - effectively
Promise.resolve(42)
  .then(undefined) // this has no affect on the value, it will be handed to following `.then(fn)`
  .then(console.log);
```
