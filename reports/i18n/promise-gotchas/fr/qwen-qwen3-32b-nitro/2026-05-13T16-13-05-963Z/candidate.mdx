### Les Promesses ne fonctionnent pas comme d'autres valeurs

Vous ne pouvez pas afficher leur valeur comme la plupart des valeurs :

```js
// Cela n'a pas de sens avec les promesses :
console.log(Promise.resolve(42));

// Nous devons utiliser l'interface `.then` :
Promise.resolve(42).then(value => console.log(value));
```

### Les Promesses ne vous avertissent pas quand vous commettez une erreur

Une erreur probable.

Pour diverses raisons, TC39 a décidé que `.then` et `.catch` pouvaient accepter `null`. Par exemple, `.then(null, null)` est valide et le comportement requis est de sauter cette « étape » dans la chaîne.

La conséquence malheureuse de cela est qu'il est très facile de se tromper.

##### Par exemple

Examinons un petit défi : laquelle(s) des options suivantes affichera(nt) `console.log` 42 ?

```js
// Option #1 :
Promise.resolve(42).then(console.log());

// Option #2 :
Promise.resolve(42).then(console.log);

// Option #3 :
Promise.resolve(42).then(value => console.log(value));

// Option #4 :
Promise.resolve(42)
  .then(console.log())
  .then(console.log);
```

##### La réponse

La réponse est #2, #3 et #4.

Pourquoi ? Examinons les **types** des arguments passés à `.then()` :

```js
var arg1 = console.log();
var arg2 = console.log;
var arg3 = value => console.log(value);

typeof arg1 === "undefined";
typeof arg2 === "function";
typeof arg3 === "function";
```

Toujours se demander comment fonctionne cette 4e option ?

Cela fonctionne en fait comme suit :

```js
// Option #4 - effectively
Promise.resolve(42)
  .then(undefined) // this has no affect on the value, it will be handed to following `.then(fn)`
  .then(console.log);
```

Le `undefined` passé à `.then(undefined)` n'affecte pas la valeur, qui est donc transmise au `.then(console.log)` suivant.
