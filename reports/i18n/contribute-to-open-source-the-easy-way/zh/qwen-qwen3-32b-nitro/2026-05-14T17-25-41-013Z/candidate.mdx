### 现实情况

我们都听过为开源项目**_`贡献`_**有多有益。但实际操作起来并不总是容易。

> 在我的职业生涯中，我为数十个拥有1万颗星标的仓库提交过许多PR。我的微小贡献已进入Node.js、Docker、Lodash、Bluebird、Gatsby、Rancher、Angular、React Router、Minio、MDN（Mozilla开发者网络）等项目。

我将分享我的轻松贡献秘诀，但在此之前，我们先简要讨论传统建议存在的问题。

### 这（不是）正确的方式

为什么**_`贡献`_**开源项目如此困难？

最常见的建议实际上介于无用和糟糕之间：找到一个`GFI`问题（`Good First Issue`标签）并解决它。或者，纯粹出于热爱去贡献一个项目。

这些初衷虽好，但实践中`GFI`标签高度主观，且往往涉及令人惊讶的工作量。

如果我告诉你，最好的起点是**你已经找到的解决方案**，你会怎么想？

### 更好的方式

✅ 扫描你的项目依赖文件。哪些库让你怒火中烧？什么导致了截止日期的错过？你是如何克服的？

💪 从你已经解决的问题开始，**你无需担心自己是否能解决**。你已经**精通且熟悉**上下文！

既然**你已经解决了问题，大部分工作已经完成**。接下来，你需要思考如何**帮助他人完全避免你的挣扎**。

也许一条推文或 Stack Overflow 回答就足够了，但如果你想产生**持久的影响，就向项目本身_`contribute`_**。

### 头脑风暴

最好在经历还比较新鲜时，思考一下你的_笨笨大脑_最初是如何迷失的。

你**最先**尝试了什么？为什么？你做了哪些假设？或者误解了什么？

💪 你不需要想出完美的解决方案，通常只需对 README 或文档进行简单更新，就能为他人节省无数小时的挣扎。

- README 过时了？缺少或糟糕的示例？缺少设置步骤？_简单修复，补充缺失的信息！_
- API 文档在你的 Google 搜索结果中未出现。_调整或翻译过于技术化的语言。_
- 也许是技术疏忽，文档站点缺少必要的 `<meta/>` 标签。_如果你知道如何解决，就动手修复，或者提交一个包含你的发现的工单。_
- 如果是技能问题... 就去提升这些技能！

这类问题维护者很容易忽略！但对项目及其用户可能产生意想不到的巨大影响。

下次你克服某个挑战时，_不要删除你绝望时的临时解决方案！_ **相反，反思你的挣扎并公开分享你的解决方案！**

### 附言

始终遵循项目规范，永远不要当混蛋。✨

这些都是公开的。所以，请保持友善、出色和感恩。

如果你还需要更多说服：`contribute`是为了学习！新的流程、语言、框架、自动化！
