### 现实

我们都听说过为开源做**贡献**有多好。但，起步并不总是那么容易。

> 在我的职业生涯中，我为几十个拥有上万 Star 的仓库写过很多 PR。我那些不起眼的贡献已经进入了 Node.js、Docker、Lodash、Bluebird、Gatsby、Rancher、Angular、React Router、Minio、MDN（Mozilla 开发者网络）等等。

我要分享我轻松贡献的秘诀，但首先我们得简单聊聊传统建议的问题。

### 这不是（正确的）路

为什么为开源做**贡献**这么难？

最常见的建议其实介于无用和糟糕之间：找一个 `GFI` 议题（`Good First Issue` 标签）然后解决它。或者，出于纯粹的热爱去贡献。

这些建议都是好意，但实际上 `GFI` 标签非常主观，而且往往工作量惊人。

如果我告诉你，最好的地方其实是**你已经找到的解决方案**呢？

### 更好的方法

✅ 扫描你的项目依赖文件。哪些库让你火冒三丈？什么导致了错过截止日期？你是如何克服的？

💪 从你已经解决的事情开始，**不用担心你能不能做到。** 你已经**精通且熟悉**上下文了！

既然**你已经解决了问题，大部分工作已经完成。** 接下来，你需要弄清楚如何**帮助其他人完全避免你的挣扎。**

也许一条推文或 Stack Overflow 的回答就够了，但如果你想产生**持久的影响，那就`贡献`到项目本身吧。**

### 头脑风暴

最好趁体验还新鲜的时候，回想一下你那颗**笨笨**的脑袋当初是怎么迷路的。

你**首先**尝试了什么？为什么？你假设了什么？或者误解了什么？

💪 你不需要想出完美的解决方案，通常只需简单更新一下 readme 或文档，就能为他人节省无数小时的挣扎。

- README 过时了？缺少或错误的示例？遗漏了设置步骤？_简单修复，补上缺失的信息！_
- API 文档没有出现在你的谷歌搜索结果中。_调整或翻译过于技术性的语言。_
- 也许是技术疏忽，文档站点缺少必要的 `<meta/>` 标签。_如果你知道怎么修就修，或者根据你的发现写一个工单。_
- 如果是技能问题……那就练好你的技能！

这类问题维护者很容易忽略！而且对项目及其用户的影响可能出奇地大。

下次你攻克某个挑战时，**不要 rebase 掉你那些绝望的 hack！** **相反，反思你的挣扎，并公开分享你的解决方案！**

### 细则

始终遵循项目指南，永远不要当混蛋。✨

一切都是公开的。所以，要亲切、优秀、感恩。

如果你还需要更多说服：`贡献`是为了学习！新的流程、语言、框架、自动化！
