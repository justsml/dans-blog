### 现实

我们都听说过 **_`contribute`_** 开源的好处。但要真正起步并不总是那么容易。

> 在我的职业生涯中，我为数十个星标 10,000+ 的仓库提交了大量 PR。我的 modest contributions 已经出现在 Node.js、Docker、Lodash、Bluebird、Gatsby、Rancher、Angular、React Router、Minio、MDN（Mozilla Developer Network）等项目中。

我将分享我的轻松贡献秘籍，但首先需要简要讨论一下传统建议的问题。

### 这不是（也不是）正确的方式

为什么 **_`contribute`_** 开源如此困难？

最常见的建议实际上介于 **无用** 与 **糟糕** 之间：寻找带有 `GFI`（Good First Issue）标签的问题并解决它，或者出于纯粹的热爱贡献项目。

这些建议出发点良好，但在实践中 `GFI` 标签高度主观，往往隐藏着意想不到的工作量。

如果我告诉你，最佳的切入点是 **你已经找到的解决方案**，你会怎么想？

### 更好的方式

✅ 扫描你的项目依赖文件。哪些库让你火冒三丈？是什么导致了错过截止日期？你是如何克服的？

💪 从已经解决的事情入手，**根本不必担心自己能否完成**。你已经 **流畅且熟悉** 那个上下文！

既然 **你已经解决了问题，大部分工作已经完成**。接下来，只需想办法 **帮助其他人彻底避免你的痛点**。

也许一条 Tweet 或 Stack Overflow 回答就够了，但如果你想产生 **持久影响**，就 **`contribute`** 到项目本身吧。

### 头脑风暴

最好在经验还稍新鲜时，回想一下你的 **笨笨的大脑** 当初是怎么陷入困境的。

你**首先**尝试了什么？为什么？你有什么假设？或者误解了什么？

💪 你不必想出完美的方案，往往对 README 或文档的简单更新就能为他人省下无数小时的折腾。

- README 过时了？示例缺失或写得糟糕？安装步骤遗漏？_简单修复，补全所有缺失信息！_
- API 文档在 Google 搜索里找不到？_调整或翻译过于技术化的语言。_
- 也许是技术疏漏，文档站缺少必要的 `<meta/>` 标签。_如果你会，就修复它；否则写一条包含你发现的 ticket。_
- 如果是技能问题……提升你的 skillz 吧！

这些问题很容易被维护者忽视，却能对项目及其用户产生意想不到的巨大影响。

下次你攻克了某个挑战，_别把你的急救 hack 直接 rebase 掉！_ **而是反思你的痛点并公开分享你的解决方案！**

### 细则

始终遵循项目指南，永远不要做混蛋。✨

一切都是公开的。所以，要保持亲切、优秀且感恩。

如果你还需要更多说服力：**`contribute`** 为了学习！新流程、语言、框架、自动化！
