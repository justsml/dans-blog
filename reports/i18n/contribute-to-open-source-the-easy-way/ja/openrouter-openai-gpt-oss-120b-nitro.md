私たちは皆、オープンソースへ **_`contribute`_** することの利点を聞いたことがありますが、実際に始めるのは簡単ではありません。

### これが（正しくない）やり方

なぜオープンソースへ **_`contribute`_** するのが難しいのでしょうか？

最も一般的なアドバイスは、実際には「役に立たない」から「ひどい」までの間に位置します。`GFI` ラベル（Good First Issue）付きの課題を見つけて解決する、あるいは純粋な愛情だけでプロジェクトに貢献する、というものです。

しかし実際には、`GFI` ラベルは主観的であり、驚くほど多くの作業が必要になることが多いです。

**すでに見つけた解決策** が最適な出発点だとしたら、どう思いますか？

### より良い方法

✅ プロジェクトの依存ファイルをスキャンする。どのライブラリが怒りを引き起こしたか？どの問題で納期に遅れたか？それをどう克服したか？

💪 すでに解決したものから始めれば、**「自分にできるか」** を心配する必要はありません。コンテキストに**流暢で、慣れ親しんで**いるのですから。

**すでに問題を解決しているので、作業の大半は完了しています。** 次にすべきは、**あなたの苦労を他の人が回避できるようにする** 方法を考えることです。

もしかしたらツイートや Stack Overflow の回答で済むかもしれませんが、**永続的なインパクト** を残したいなら、_`contribute`_ してプロジェクト自体に貢献しましょう。

### ブレインストーム

できれば経験がまだ新しいうちに、なぜ自分の **ぼんやりした** 頭が最初に迷子になったのかを考えてみてください。

最初に **何を試した** か？ そしてなぜ？ 何を前提にしたか？ あるいは何を勘違いしたか？

💪 完璧な解決策を考える必要はありません。README やドキュメントのちょっとした更新だけで、他の開発者の苦労を何時間も削減できることがあります。

- 古くなった README？ 例が足りない、または不適切？ セットアップ手順が抜けている？ _シンプルに修正し、欠けている情報を追加しましょう！_
- API ドキュメントが Google 検索に出てこなかった。 _過度に技術的な表現を調整または翻訳してください。_
- あるいは技術的な見落としで、ドキュメントサイトに必要な `<meta/>` タグが欠けている。 _方法が分かれば修正するか、調査結果を添えてチケットを作成してください。_
- スキルの問題であれば… そのスキルを磨きましょう！

これらの問題はメンテナナーが見落としがちで、プロジェクトやユーザーに意外に大きな影響を与えることがあります。

次に何か課題を解決したときは、_絶望的なハックをリベースで消さないで！_ **むしろ、苦労した過程を振り返り、解決策を公開しましょう！**
