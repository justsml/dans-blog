## Сломаны ли JavaScript Promises?

### В эпоху до этого

Одним из самых распространенных мифов о Promises является утверждение о предполагаемых недостатках обработки ошибок.

**Многие годы** Promises действительно были ужасны в обработке ошибок. **Потребовалось много работы, чтобы это исправить.**

> И вот, **это было исправлено**, даже **широко внедрено**.

#### #1 Промисы нуждаются в чем-то, к чему можно привязаться  

Критически важно **всегда возвращать** значения из функций.  

Функции-колбэки для промисов следуют определённому паттерну в `.then(callback)` и `.catch(callback)`.

Каждое возвращаемое значение передаётся в колбэк следующего `.then()`.

```js
function addTen(number) {
  return number + 10;
}

Promise.resolve(10)  // 10
  .then(addTen)      // 20
  .then(addTen)      // 30
  .then(addTen)      // 40
  .then(console.log) // логирует "40"
```

> Бонус «всегда возвращать»: код намного проще тестировать на уровне модулей.

#### #2 Используйте настоящие экземпляры `Error`

JavaScript имеет интересное поведение относительно ошибок (оно применяется к асинхронному **и** синхронному коду).

<a href="https://repl.it/@justsml/throwing-errors-in-javascript" target="_blank">[<i>посмотреть пример на repl.it: `throwing errors in javascript`</i>]</a>
<img alt="бросание ошибок в JavaScript" src="../throwing-errors-in-javascript.webp" />

Чтобы получить полезные сведения о номере строки и стеке вызовов, необходимо использовать экземпляры `Error`. Бросание строк не работает так, как в Python или Ruby.

Хотя JavaScript **кажется**, что он обрабатывает `throw "string"`, как вы увидите, строку можно увидеть в вашем обработчике `catch`. Однако, данные — это всё, что вы увидите*. Никаких предыдущих [кадров стека](https://en.wikipedia.org/wiki/Call_stack#Stack_and_frame_pointers) не будет включено.

Правильные примеры с `new Error`:

```js
throw new Error('message')           // ✅
Promise.reject(new Error('message')) // ✅
throw Error('message')               // ✅
Promise.reject(Error('message'))     // ✅
```

Следующие являются распространёнными антипаттернами:

```js
throw 'error message'  // ❌
Promise.reject(-42)    // ❌
```

#### #3 Обрабатывайте ошибки там, где это имеет смысл

Promises предоставляют удобный способ обработки ошибок с помощью `.catch()`. Это по сути специальный вид `.then()` — где обрабатываются любые ошибки из предыдущих `.then()`. Рассмотрим пример...

```js
Promise.resolve(42)
  .then(() => 'hello')
  .catch(() => console.log('will not get hit'))
  .then(() => throw new Error('totes fail'))
  .catch(() => console.log('WILL get hit'))
```

Хотя `.catch()` может казаться обработчиком событий DOM (например, `click`, `keypress`), его расположение важно, поскольку он может "поймать" ошибки, возникшие **выше него**.

**Переопределение ошибок относительно тривиально**. Верните неошибочное значение в вашем колбэке `.catch()`, и цепочка Promise перейдет к выполнению `.then()` колбэков последовательно. (Фактически.)

Попробуйте проследить последовательность в следующем примере:

```js
Promise.resolve(42)
  .then(() => 'hello')
  .then(() => throw new Error('totes fail'))
  .catch(() => {
    return 99
  })
  .then(num => num + 1)
  .then(console.log) // ожидаемый вывод: 100
```

**Важно понимать последовательность.**

Хотя пример и кажется глупым, он предназначен для **показа того, как ошибки и данные передаются** в Promise.

Вот общий ход событий:

1. 42 — начальное значение.
2. `hello` всегда возвращается следующим методом.
3. мы игнорируем предыдущее значение и выбрасываем ошибку с сообщением `'totes fail'`.
4. `.catch()` перехватывает ошибку, вместо этого возвращает `99`, который будет обработан любым последующим `.then()`.
5. увеличиваем `num`, получаем `100`.
6. метод `console.log` получает `100` и выводит его! :tada:

#### #4 Добавляйте ясность с именованными функциями 🦄✨

Сравните **читаемость** следующих двух примеров:

**Анонимные:** ❌

```js
Promise.resolve(10)          // 10
  .then(x => x * 2)          // 20
  .then(x => x / 4)          // 5
  .then(x => x * x)          // 25
  .then(x => x.toFixed(2))   // "25.00"
  .then(x => console.log(x)) // ожидаемый вывод: "25.00"
```

**Именованные:** ✅

```js
Promise.resolve(10) // 10
  .then(double)     // 20
  .then(quarter)    // 5
  .then(square)     // 25
  .then(format)     // "25.00"
  .then(log)        // ожидаемый вывод: "25.00"

const double = x => x * 2
const quarter = x => x / 4
const square = x => x * x
const format = x => x.toFixed(2)
const log = x => console.log(x)
```

**Бонус:** ✅

> Совместимо с методами массивов!!!

Вы можете переиспользовать свои именованные функции вместе с нашими друзьями из `Array.prototype`. Включая `.map()`, `.filter()`, `.every()`, `.some()`, `.find()`!

Collection pipelines #FTW:

```js
// IT'S LIKE THE SAME THING :mindblown:

[10, 20]           // [ 10, 20 ]
  .map(double)     // [ 20, 40 ]
  .map(quarter)    // [ 5, 10 ]
  .map(square)     // [ 25, 100 ]
  .map(format)     // [ "25.00", "100.00" ]
  .map(log)        // expected 2 lines of output: "25.00", "100.00"

```

И если вы не хотите использовать этот линейный стиль программирования... У вас есть простые функции!

Вы можете использовать их по своему усмотрению:

```js
// Вложенный паттерн
// ❌ пожалуйста, не делайте этого

const result = format(square(quarter(double(10))))

log(result)
// expected output: "25.00"
```

**Почему вложенные функции являются антипаттерном?**

1. Меньше людей могут легко читать такой код
2. git-диффы не показывают, кто вносил изменения
3. Трудно отлаживать или логировать промежуточные результаты вложенных функций
