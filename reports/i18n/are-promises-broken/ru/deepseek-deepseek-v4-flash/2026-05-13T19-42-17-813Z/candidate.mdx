#### #1 Промисам нужно за что-то зацепиться

Крайне важно **всегда `return`** из ваших функций.

Функции обратного вызова промисов следуют определённому шаблону в `.then(callback)` и `.catch(callback)`.

Каждое возвращаемое значение передается в следующий колбэк `.then()`.

```js
function addTen(number) {
  return number + 10;
}

Promise.resolve(10)  // 10
  .then(addTen)      // 20
  .then(addTen)      // 30
  .then(addTen)      // 40
  .then(console.log) // выводит "40"
```

> Бонус «всегда возвращайте»: код становится гораздо проще для модульного тестирования.

#### #2 Используйте настоящие экземпляры `Error`

У JavaScript есть интересное поведение, связанное с ошибками (которое применимо как к асинхронному, так и к синхронному коду).

<a href="https://repl.it/@justsml/throwing-errors-in-javascript" target="_blank">[<i>смотрите пример на repl.it: `throwing errors in javascript`</i>]</a>
<img alt="выброс ошибок в javascript" src="../throwing-errors-in-javascript.webp" />

Чтобы **получить полезные сведения о номере строки** и стеке вызовов, необходимо использовать экземпляры `Error`. Выброс строк не работает, как в Python или Ruby.

Хотя JavaScript **кажется** обрабатывает `throw "string"`, и вы увидите строку в обработчике `catch`, однако это всё, что вы увидите*. Никакие предыдущие [фреймы стека](https://en.wikipedia.org/wiki/Call_stack#Stack_and_frame_pointers) включены не будут.

Правильные примеры с `new Error`:

```js
throw new Error('message')           // ✅
Promise.reject(new Error('message')) // ✅
throw Error('message')               // ✅
Promise.reject(Error('message'))     // ✅
```

Следующие примеры — распространённые антипаттерны:

```js
throw 'error message'  // ❌
Promise.reject(-42)    // ❌
```

#### #3 Обрабатывайте ошибки там, где это имеет смысл

Promises предоставляют элегантный способ обработки ошибок с помощью `.catch()`. Это по сути особый вид `.then()` — он перехватывает любые ошибки из предыдущих `.then()`. Рассмотрим пример...

```js
Promise.resolve(42)
  .then(() => 'hello')
  .catch(() => console.log('will not get hit'))
  .then(() => throw new Error('totes fail'))
  .catch(() => console.log('WILL get hit'))
```

Хотя `.catch()` может напоминать обработчик событий DOM (например, `click`, `keypress`), его расположение важно: он может «поймать» только ошибки, выброшенные **выше него.**

**Переопределить ошибки относительно просто** — верните не-ошибочное значение в колбэке `.catch()`, и цепочка Promise переключится на последовательное выполнение колбэков `.then()`. (По сути.)

Попробуйте проследить последовательность в следующем примере:

```js
Promise.resolve(42)
  .then(() => 'hello')
  .then(() => throw new Error('totes fail'))
  .catch(() => {
    return 99
  })
  .then(num => num + 1)
  .then(console.log) // expected output: 100
```

**Последовательность — вот что важно понять.**

Хотя это глупый пример, он призван **проиллюстрировать, как ошибки и данные передаются** в Promise.

Вот схема последовательности:

1. 42 — начальное значение.
1. `hello` всегда возвращается следующим методом.
1. мы игнорируем предыдущее значение и выбрасываем ошибку с сообщением `'totes fail'`.
1. `.catch()` перехватывает ошибку, вместо этого возвращает `99`, которое будет обработано любым последующим `.then()`.
1. увеличиваем `num`, возвращая `100`.
1. метод `console.log` получает `100` и выводит его! :tada:

#### #4 Добавьте ясности с помощью именованных функций 🦄✨

Сравните **читаемость** следующих двух примеров:

**Анонимные:** ❌

```js
Promise.resolve(10)          // 10
  .then(x => x * 2)          // 20
  .then(x => x / 4)          // 5
  .then(x => x * x)          // 25
  .then(x => x.toFixed(2))   // "25.00"
  .then(x => console.log(x)) // expected output: "25.00"
```

**Именованные:** ✅

```js
Promise.resolve(10) // 10
  .then(double)     // 20
  .then(quarter)    // 5
  .then(square)     // 25
  .then(format)     // "25.00"
  .then(log)        // expected output: "25.00"

const double = x => x * 2
const quarter = x => x / 4
const square = x => x * x
const format = x => x.toFixed(2)
const log = x => console.log(x)

```

**БОНУС:** ✅

> Совместимость с методами массивов!!!

Вы можете переиспользовать свои именованные функции с нашими друзьями из `Array.prototype.` Включая `.map()`, `.filter()`, `.every()`, `.some()`, `.find()`!

Конвейеры коллекций #FTW:

```js
// ЭТО ЖЕ ТО ЖЕ САМОЕ :mindblown:

[10, 20]           // [ 10, 20 ]
  .map(double)     // [ 20, 40 ]
  .map(quarter)    // [ 5, 10 ]
  .map(square)     // [ 25, 100 ]
  .map(format)     // [ "25.00", "100.00" ]
  .map(log)        // ожидаемый вывод: 2 строки — "25.00", "100.00"

```

А если вам не нравится такой линейный стиль… Что ж, у вас есть простые функции!

Вы можете использовать их как угодно:

```js
// Паттерн вложенности

log(result)
// ожидаемый вывод: "25.00"
```

**Почему вложенность функций — антипаттерн?**

1. Не так читаемо для многих людей
2. git-диффы не показывают, кто что менял
3. Сложно отлаживать или логировать из середины вложенных функций
