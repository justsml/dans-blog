#### #1 Promises brauchen etwas, an das sie sich klammern können  

Es ist entscheidend, dass Sie **immer `return`** aus Ihren Funktionen verwenden.  

Rückruffunktionen von Promises folgen einem bestimmten Muster in `.then(callback)` und `.catch(callback)`.

Jeder zurückgegebene Wert wird an den Callback der nächsten `.then()`-Methode weitergegeben.

```js
function addTen(number) {
  return number + 10;
}

Promise.resolve(10)  // 10
  .then(addTen)      // 20
  .then(addTen)      // 30
  .then(addTen)      // 40
  .then(console.log) // logs "40"
```

> Vorteil von „immer zurückgeben“: Der Code ist wesentlich einfacher zu unit-testen.

#### #2 Nutzen Sie echte `Error`-Instanzen

JavaScript hat ein interessantes Verhalten bei Fehlern (das gilt für asynchronen **und** synchronen Code).

<a href="https://repl.it/@justsml/throwing-errors-in-javascript" target="_blank">[<i>Beispiel auf repl.it ansehen: `throwing errors in javascript`</i>]</a>  
<img alt="Fehler werfen in JavaScript" src="../throwing-errors-in-javascript.webp" />  

Um **nützliche Details zur Zeilennummer** und dem Aufrufstapel zu erhalten, **müssen** Sie `Error`-Instanzen verwenden. Das Werfen von Zeichenketten funktioniert nicht wie in Python oder Ruby.  

Obwohl JavaScript **scheinbar** `throw "string"` akzeptiert, da Sie die Zeichenkette in Ihrem `catch`-Handler sehen. Die Daten sind jedoch alles, was Sie sehen*. Es werden keine vorherigen [Stackrahmen](https://en.wikipedia.org/wiki/Call_stack#Stack_and_frame_pointers) enthalten sein.  

Korrekte `new Error`-Beispiele:  

```js  
throw new Error('message')           // ✅  
Promise.reject(new Error('message')) // ✅  
throw Error('message')               // ✅  
Promise.reject(Error('message'))     // ✅  
```  

Die folgenden sind gängige Anti-Muster:  

```js  
throw 'error message'  // ❌  
Promise.reject(-42)    // ❌  
```

#### #3 Fehler an sinnvollen Stellen behandeln

Promises bieten eine elegante Methode zur Fehlerbehandlung durch `.catch()`. Es handelt sich im Grunde um eine spezielle Form von `.then()` – bei der Fehler aus vorhergehenden `.then()`-Aufrufen behandelt werden. Schauen wir uns ein Beispiel an...

```js
Promise.resolve(42)
  .then(() => 'hello')
  .catch(() => console.log('wird nicht getroffen'))
  .then(() => throw new Error('totes Fail'))
  .catch(() => console.log('WIRD getroffen'))
```

Obwohl `.catch()` auf den ersten Blick wie ein DOM-Ereignis-Handler (z. B. `click`, `keypress`) wirkt, ist seine Platzierung entscheidend, da es nur Fehler behandeln kann, die **darüber** geworfen wurden.

**Das Überschreiben von Fehlern ist relativ einfach**  
Geben Sie in Ihrem `.catch()`-Callback einen nicht-fehlerhaften Wert zurück, und die Promise-Kette wechselt automatisch in die Ausführung der `.then()`-Callbacks in Reihenfolge. (Effektiv.)

Versuchen Sie, die Abfolge des folgenden Beispiels zu verfolgen:

```js
Promise.resolve(42)
  .then(() => 'hello')
  .then(() => throw new Error('totes Fail'))
  .catch(() => {
    return 99
  })
  .then(num => num + 1)
  .then(console.log) // erwartete Ausgabe: 100
```

**Die Abfolge ist das Wichtige zu verstehen.**

Obwohl ein einfaches Beispiel, ist es so konzipiert, **zu veranschaulichen, wie Fehler und Daten in Promises fließen**.  

Hier ist ein Überblick über die Abfolge:  

1. **42** ist der Ausgangswert.  
2. **`'hello'`** wird immer von der nächsten Methode zurückgegeben.  
3. Wir ignorieren den vorherigen Wert und werfen einen Fehler mit der Nachricht **`'totes fail'`**.  
4. **`.catch()`** fängt den Fehler ab und gibt stattdessen **`99`** zurück, der von jeder nachfolgenden `.then()`-Methode verarbeitet wird.  
5. Der **`num`**-Wert wird um 1 erhöht, wodurch **`100`** entsteht.  
6. Die Methode **`console.log`** empfängt **`100`** und gibt ihn aus! :tada:

#### #4 Klarheit mit benannten Funktionen schaffen 🦄✨

Vergleichen Sie die **Lesbarkeit** der folgenden beiden Beispiele:

**Anonym:** ❌

```js
Promise.resolve(10)          // 10
  .then(x => x * 2)          // 20
  .then(x => x / 4)          // 5
  .then(x => x * x)          // 25
  .then(x => x.toFixed(2))   // "25.00"
  .then(x => console.log(x)) // erwartete Ausgabe: "25.00"
```

**Benannt:** ✅

```js
Promise.resolve(10) // 10
  .then(double)     // 20
  .then(quarter)    // 5
  .then(square)     // 25
  .then(format)     // "25.00"
  .then(log)        // erwartete Ausgabe: "25.00"

const double = x => x * 2
const quarter = x => x / 4
const square = x => x * x
const format = x => x.toFixed(2)
const log = x => console.log(x)

```

**BONUS:** ✅

> Array-Methoden-kompatibel!!!

Sie können Ihre benannten Funktionen wiederverwenden mit unseren Freunden aus `Array.prototype.`. Dazu gehören `.map()`, `.filter()`, `.every()`, `.some()`, `.find()`!

Collection pipelines #GEIL:

```js
// ES IST GENAU DAS SELBE :mindblown:

[10, 20]           // [ 10, 20 ]
  .map(double)     // [ 20, 40 ]
  .map(quarter)    // [ 5, 10 ]
  .map(square)     // [ 25, 100 ]
  .map(format)     // [ "25.00", "100.00" ]
  .map(log)        // erwartete Ausgabe: "25.00", "100.00"

```

Und wenn Sie diese lineare Codestyle-Programmierung nicht mögen... Nun, Sie haben einfache Funktionen!

Sie können sie nach Bedarf verwenden:

```js

// erwartete Ausgabe: "25.00"
```

**Warum ist die Verschachtelung von Funktionen ein Anti-Muster?**

1. Nicht für so viele Menschen lesbar  
2. git-Diffs zeigen nicht sofort an, wer was geändert hat  
3. schwer zu debuggen oder zu protokollieren aus der Mitte der verschachtelten Funktionen
