#### #1 Promises need something to hang on to

It is critical that you **always `return`** from your functions.

Promise callback functions follow a certain pattern in `.then(callback)` and `.catch(callback)`.

各 `return` された値は次の `.then()` のコールバックへ渡されます。

```js
function addTen(number) {
  return number + 10;
}

Promise.resolve(10)  // 10
  .then(addTen)      // 20
  .then(addTen)      // 30
  .then(addTen)      // 40
  .then(console.log) // logs "40"
```

> 「常に `return` する」ことの副産物: コードのユニットテストが格段に楽になる。

#### #2 実際の `Error` インスタンスを使用する

JavaScript にはエラーに関して興味深い挙動があります（非同期 **および** 同期コードの両方に適用されます）。

<a href="https://repl.it/@justsml/throwing-errors-in-javascript" target="_blank">[<i>repl.it の例を参照: `throwing errors in javascript`</i>]</a>
<img alt="JavaScript でエラーを投げる例" src="../throwing-errors-in-javascript.webp" />

**行番号やコールスタックの有用な情報を取得するには、`Error` インスタンスを使用する必要があります。** 文字列を投げるだけでは、Python や Ruby のようには機能しません。

JavaScript は **`throw "string"`** を受け付けるように見えますが、`catch` ハンドラで文字列がそのまま渡されるだけです。取得できるのはその文字列だけで、*スタックフレームは一切含まれません*。事前の [スタックフレーム](https://en.wikipedia.org/wiki/Call_stack#Stack_and_frame_pointers) が付与されることはありません。

正しい `new Error` の例:

```js
throw new Error('message')           // ✅
Promise.reject(new Error('message')) // ✅
throw Error('message')               // ✅
Promise.reject(Error('message'))     // ✅
```

以下は典型的なアンチパターンです:

```js
throw 'error message'  // ❌
Promise.reject(-42)    // ❌
```

#### #3 エラーは意味のある場所で処理する

Promise は `.catch()` を使ってエラー処理をスマートに行える。これは基本的に特別な `.then()` で、前の `.then()` で発生したエラーを捕捉する。例を見てみよう…

```js
Promise.resolve(42)
  .then(() => 'hello')
  .catch(() => console.log('will not get hit'))
  .then(() => throw new Error('totes fail'))
  .catch(() => console.log('WILL get hit'))
```

`.catch()` は DOM のイベントハンドラ（例: `click`、`keypress`）のように見えるかもしれないが、**配置が重要** で、**それより上で投げられた** エラーしか捕まえられない。

**エラーを上書きするのは比較的簡単** `.catch()` コールバックでエラーでない値を返すと、Promise チェーンは `.then()` コールバックの実行に切り替わる（実質的に）。

以下の例の流れを追ってみよう:

```js
Promise.resolve(42)
  .then(() => 'hello')
  .then(() => throw new Error('totes fail'))
  .catch(() => {
    return 99
  })
  .then(num => num + 1)
  .then(console.log) // expected output: 100
```

**重要なのはシーケンスそのものだ。**

冗談めいた例ですが、**Promise におけるエラーとデータの流れを示す** 目的で作られています。

シーケンスの概要は次のとおりです:

1. `42` が初期値です。  
1. 次のメソッドは常に `hello` を返します。  
1. 前の値は無視し、メッセージ `'totes fail'` のエラーを投げます。  
1. `.catch()` がエラーを捕捉し、代わりに `99` を返します。これ以降の `.then()` で処理されます。  
1. `num` をインクリメントし、`100` を返します。  
1. メソッド `console.log` が `100` を受け取り、出力します！ :tada:

#### #4 名前付き関数で明瞭さを向上させる 🦄✨

以下の 2 つの例の **可読性** を比較してください。

**匿名:** ❌

```js
Promise.resolve(10)          // 10
  .then(x => x * 2)          // 20
  .then(x => x / 4)          // 5
  .then(x => x * x)          // 25
  .then(x => x.toFixed(2))   // "25.00"
  .then(x => console.log(x)) // expected output: "25.00"
```

**名前付き:** ✅

```js
Promise.resolve(10) // 10
  .then(double)     // 20
  .then(quarter)    // 5
  .then(square)     // 25
  .then(format)     // "25.00"
  .then(log)        // expected output: "25.00"

const double = x => x * 2
const quarter = x => x / 4
const square = x => x * x
const format = x => x.toFixed(2)
const log = x => console.log(x)

```

**ボーナス:** ✅

> 配列メソッドと相性抜群!!!

`Array.prototype` 系のメソッド（`.map()`、`.filter()`、`.every()`、`.some()`、`.find()` など）と組み合わせて、名前付き関数を再利用できます。

コレクションパイプライン #FTW:

```js
// IT'S LIKE THE SAME THING :mindblown:

[10, 20]           // [ 10, 20 ]
  .map(double)     // [ 20, 40 ]
  .map(quarter)    // [ 5, 10 ]
  .map(square)     // [ 25, 100 ]
  .map(format)     // [ "25.00", "100.00" ]
  .map(log)        // expected 2 lines of output: "25.00", "100.00"
```

そして、線形スタイルのコーディングが嫌なら… シンプルな関数が用意されています！

必要に応じて使い回せます:

```js
// ネストパターン
// ❌ できるだけやめてください

const result = format(square(quarter(double(10))))

log(result)
// expected output: "25.00"
```

**なぜ関数のネストはアンチパターンなのか？**

1. 多くの人にとって可読性が低い  
2. `git diff` で誰が何を変更したかがすぐに分からない  
3. ネストされた関数の途中でデバッグやロギングがしにくい
