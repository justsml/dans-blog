## ¿Están rotas las Promesas de JavaScript?

### En los tiempos anteriores

Uno de los mitos más comunes sobre las Promesas es su supuesta deficiencia con los errores.

**Hace muchos años** las Promesas _eran_ realmente terribles en el manejo de errores. **Se invirtió mucho trabajo en corregirlo.**

> Y he aquí, **se arregló**, incluso **se desplegó ampliamente**.

#### La gente se regocijó

Y, lamentablemente, algunos no se dieron cuenta.

#### #1 Las promesas necesitan algo a lo que aferrarse

Es crítico que **siempre `return`** desde tus funciones.

Las funciones de devolución de llamada de una promesa siguen un patrón determinado en `.then(callback)` y `.catch(callback)`.

Cada valor devuelto se pasa al callback del siguiente `.then()`.

```js
function addTen(number) {
  return number + 10;
}

Promise.resolve(10)  // 10
  .then(addTen)      // 20
  .then(addTen)      // 30
  .then(addTen)      // 40
  .then(console.log) // logs "40"
```

> Ventaja de “siempre devolver”: el código es mucho más fácil de probar unitariamente.

#### #2 Use real `Error` Instances

JavaScript tiene un comportamiento interesante respecto a los errores (que se aplican tanto al código asíncrono **como** al síncrono).

<a href="https://repl.it/@justsml/throwing-errors-in-javascript" target="_blank">[<i>ver ejemplo en repl.it: `throwing errors in javascript`</i>]</a>
<img alt="lanzando errores en javascript" src="../throwing-errors-in-javascript.webp" />

Para **obtener detalles útiles sobre el número de línea** y la pila de llamadas, debe usar instancias de `Error`. Lanzar cadenas no funciona como en Python o Ruby.

Aunque JavaScript **parece** manejar `throw "string"`, como verá la cadena en su manejador `catch`. Sin embargo, los datos son todo lo que verá*. No se incluirán marcos de pila previos[^1].

Ejemplos correctos con `new Error`:

```js
throw new Error('message')           // ✅
Promise.reject(new Error('message')) // ✅
throw Error('message')               // ✅
Promise.reject(Error('message'))     // ✅
```

Los siguientes son anti‑patrones comunes:

```js
throw 'error message'  // ❌
Promise.reject(-42)    // ❌
```

#### #3 Manejar errores donde tenga sentido

Las Promesas ofrecen una forma elegante de gestionar errores, usando `.catch()`. Es básicamente un tipo especial de `.then()`—donde se manejan los errores que provienen de los `.then()` anteriores. Veamos un ejemplo...

```js
Promise.resolve(42)
  .then(() => 'hello')
  .catch(() => console.log('will not get hit'))
  .then(() => throw new Error('totes fail'))
  .catch(() => console.log('WILL get hit'))
```

Aunque `.catch()` pueda parecerse a un manejador de eventos del DOM (p. ej., `click`, `keypress`), su ubicación es importante, ya que solo puede *capturar* errores lanzados **por encima de él**.

**Sobrescribir errores es relativamente trivial**. Si devuelves un valor que no sea un error en el callback de tu `.catch()`, la cadena de Promesas vuelve a ejecutar los callbacks de `.then()` en secuencia. (Efectivamente.)

Intenta seguir la secuencia del siguiente ejemplo:

```js
Promise.resolve(42)
  .then(() => 'hello')
  .then(() => throw new Error('totes fail'))
  .catch(() => {
    return 99
  })
  .then(num => num + 1)
  .then(console.log) // salida esperada: 100
```

**La secuencia es lo que importa comprender.**

Mientras que el ejemplo es tonto, está diseñado para **ilustrar cómo fluyen los errores y los datos** en las Promesas.

He aquí el esquema de la secuencia:

1. 42 es el valor inicial.  
1. `hello` siempre es devuelto por el método siguiente.  
1. descartamos el valor anterior y lanzamos un error con el mensaje `'totes fail'`.  
1. `.catch()` intercepta el error y, en su lugar, devuelve `99`, que será manejado por cualquier `.then()` posterior.  
1. incrementamos `num`, devolviendo `100`.  
1. el método `console.log` recibe `100` y lo imprime! :tada:

**Pregunta:** ¿Qué ocurre cuando hay 2 `.catch()` en secuencia? ¿Puede el segundo ejecutarse alguna vez? ¿Puedes pensar en un caso de uso?

#### #4 Añade claridad con funciones nombradas 🦄✨

Compárese la **legibilidad** de los siguientes 2 ejemplos:

**Anónima:** ❌

```js
Promise.resolve(10)          // 10
  .then(x => x * 2)          // 20
  .then(x => x / 4)          // 5
  .then(x => x * x)          // 25
  .then(x => x.toFixed(2))   // "25.00"
  .then(x => console.log(x)) // expected output: "25.00"
```

**Nombrada:** ✅

```js
Promise.resolve(10) // 10
  .then(double)     // 20
  .then(quarter)    // 5
  .then(square)     // 25
  .then(format)     // "25.00"
  .then(log)        // expected output: "25.00"

const double = x => x * 2
const quarter = x => x / 4
const square = x => x * x
const format = x => x.toFixed(2)
const log = x => console.log(x)

```

**BONUS:** ✅

> ¡Compatible con métodos de arrays!

Puedes reutilizar tus funciones nombradas con nuestros amigos de `Array.prototype`. ¡Incluyendo `.map()`, `.filter()`, `.every()`, `.some()`, `.find()`!

**¿Por qué anidar funciones es un anti‑patrón?**

1. No resulta legible para muchas personas  
2. Los diffs de git no revelan fácilmente quién cambió qué  
3. Es difícil depurar o registrar desde el medio de las funciones anidadas
