## Explorando técnicas de canalización basadas en matrices y conjuntos

### ¿Un anti patrón?

Esta es una exploración de las ventajas obtenidas al codificar todo como una matriz. (Usando conceptos Jedi de Smalltalk)

Aquí hay algunos principios guía:

1. Toda entrada es semejante a una matriz. Incluso si es una matriz de 1.
1. Las funciones de nivel superior deben aceptar y devolver generalmente matrices. (Excepto para métodos de callback en bucles: map/filtrar/reducir/cada)
1. 99 de cada 100 desarrolladores sufren lo que llamo el síndrome de `exceso agudo de esquema`.
1. Tengan cuidado con los modelos basados en clases hinchados - con todos sus efectos predecibles: estado de instancia frágil con tantas palancas y pernos para tocar, transacciones de base de datos, bloqueos SQL, asincronía/mutex (que siempre funcionan a la primera), uso de getters/setters idiomáticos, y su uso de `público/privado/final/etc` es sólido, ¿verdad?

1. Así que tomemos un problema común y _forcemos_ ~~~agregar~~~ algunas reflexiones basadas en conjuntos.
1. Un sitio web de blog hipotético tiene muchos Artículos y aún más Publicaciones (Comentarios).
1. Añadamos un método `delete` (abajo) - pero con soporte tanto para elementos individuales como para matrices.

```java
package net.danlevy.why.java___why.you.got.all.the.dots____it.must.be.all.the.factories;

public class Post {
  public String   title;
  public Date     created;
  public String   message;

  public Post(String title, String message) {
    this.title    = title;
    this.message  = message;
    this.created  = new Date();
  }

  public Date isArchived() {
    return this.created < new Date(2015, 0, 1);
  }

  // Post.delete` can be called w/ a singular Post or an array of Post[]
  public static int delete(Post post) {
    List<Post> posts = new List<Post>(post);
    return delete(posts)
  }

  public static int delete(List<Post> posts) {
    return posts.map(Dao.remove);
  }
}
```
