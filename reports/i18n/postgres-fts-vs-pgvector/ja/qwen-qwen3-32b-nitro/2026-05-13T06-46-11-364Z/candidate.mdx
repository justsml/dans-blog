AI機能を追加するチームは、まず専用のベクトルデータベースに手を伸ばすことが多い。  

Pinecone, Weaviate, Qdrant, Chroma。新しいサービス、新しい依存関係、新しい接続プール、新しい請求書。そして今や、整合性を保つべき真実の源が2つになる。  

一方で、彼らはすでにPostgreSQLを持っている。PostgreSQLにはすでに`pgvector`が備わっている。2008年以来、優れた全文検索機能が組み込まれている。

専用のベクトルストアは大規模なデータ量と高頻度のクエリ処理においてその価値を発揮する。しかし多くのアプリケーションでは、最初の検索システムが十分に活用される前に、すでに2番目の検索システムを採用してしまう。それが、将来のスケーラビリティの問題が今日の同期バグに変わってしまう仕組みだ。

では：全文検索（FTS）はいつ使うべきか？ pgvectorはいつ使うべきか？そして両方を併用するべき状況はいつなのか？

## 実際の機能

全文検索（`tsvector` / `GIN` インデックス）は語彙に基づく処理です。テキストをトークン化して語彙に変換し、それらを語幹に還元し、インデックスと照合します。「Running」と「runs」は同じ語彙にまとめられます。同様に「dog」と「dogs」も同じ語彙になります。順位付け関数（`ts_rank`）は、検索語が頻繁に出現したり、目立つ位置にあったりするドキュメントに高得点を付与します。

pgvectorは意味に基づく処理です。チャンクの「意味」を埋め込みモデルが解釈し、その結果を密なベクトル（数値のリスト）として保存します。類似性検索はその高次元空間で近いベクトルを検出します。「dog」と「canine」は近接しますが、「running」（スポーツ）と「running」（プロセスの実行）はそうではないかもしれません。

実用的な違いは次の通りです。全文検索は「これらの語を含むドキュメントはどれですか？」という質問に答えるのに対し、ベクトル検索は「だいたいこの意味のドキュメントはどれですか？」という質問に答えるものです。

![pg_trgmによる短い曖昧文字列、全文検索による正確な文章クエリ、pgvectorによる意味一致、そして正確性と意味の両方が必要な長いコンテンツ向けのハイブリッド検索を示す検索ツールマップ](../search-tool-map.svg)

_最初の分岐は「古い検索 vs AI検索」というわけではありません。それはテキストの形状と、どの種類の回答が正しいかという問題です。_

## 全文検索が適している場面

**正確に一致する語を検索している**  
製品のSKUコード、エラーコード、モデル番号、ユーザー名、法律条文の参照番号など、語の意味的類似性が不要なケースです。`SKU-AX-44192` は他の何よりも意味的に類似しません。一致するか、しないかだけです。ベクトル検索は自信を持って `SKU-AX-44193` を返すかもしれませんが、それは求めている結果ではありません。

**検索クエリはキーワードベースです。** ユーザーが検索ボックスに文字を入力する、タグでフィルタリングする、ブログ投稿をキーワードで検索する——FTSはこうした意図の形状に設計されています。

**GPUや埋め込みインフラなしでランキング付きの結果が必要です。** FTSインデックスは高速で決定論的であり、外部API呼び出しが不要です。`tsvector`列を追加し、GINインデックスを構築すれば完了です。

**検索とブールフィルタリングを併用しています。** `WHERE to_tsvector(body) @@ to_tsquery('postgres') AND category = 'tutorial' AND published_at > NOW() - INTERVAL '6 months'` — これは既存のクエリロジックと自然に組み合わせられます。

```sql
-- インデックスの作成
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body, '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- クエリ
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`GENERATED ALWAYS AS`列はインデックスを自動的に更新します。`setweight`によってタイトルのマッチが本文のマッチよりもランクが高くなります。これで設定は完了です。

## トリグラムが活躍する場面 (pg_trgm)

PostgreSQLには見過ごされがちな3番目のツールがあります：`pg_trgm`。これは全文検索（FTS）でもベクトル検索でもありません。これは曖昧な文字列マッチングであり、両方のツールがうまく対応できない中途半端な領域をカバーします。

**適用ケース：名前、住所、ID、タイポを含む短い文字列**

FTSはテキストをトークン化して語幹に変換します。これは文章向けですが、以下のケースには不向きです：
- 人物名（「Dan Levy」→ ステミングされて「dan levi」、「leiv」など、言語設定に依存）
- 企業名、住所、製品タイトルなど正確な綴りが重要なケース
- タイポを含むクエリ — 「Micheal Jordan」、「Amaon」、「javascipt」
- オートコンプリート / プレフィックス検索
- 部分文字列マッチ（「son」が「Johnson」、「Anderson」にマッチ）

pgvectorもここでは不向きです。"Micheal Jordan"を埋め込み（embedding）して最も近いベクトルを検索しても、埋め込み空間は意味で名前を組織化しますが、綴りではありません。最も近い近傍は「バスケットボールの伝説」や「Michael B. Jordan」になる可能性があり、タイポを含むユーザー記録にはマッチしません。  

`pg_trgm`は文字列を重複する3文字のスライスに分割し、2つの文字列が共有するトリグラムの数を測定します。「Dan」→ `" da"`、`"dan"`、`"an "`。"Micheal"と"Michael"は多くのトリグラムを共有するため、類似度が高くなります。  

```sql
-- 拡張機能の有効化（通常はすでに利用可能）
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- 名前の列にGINインデックス — トリグラム類似度検索を高速化
CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- ファジーな名前検索：「Michael Jordan」で「Micheal Jordan」を検索
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % 演算子 = 類似度しきい値（デフォルト0.3）
ORDER BY score DESC
LIMIT 10;

-- トリグラムインデックスをサポートして部分一致検索
SELECT id, name
FROM users
WHERE name ILIKE '%johnson%'   -- GINインデックスが高速化
LIMIT 10;
```

`%`演算子は`pg_trgm.similarity_threshold`（デフォルト0.3、範囲0-1）を使用します。値を高くすると、より厳密なマッチが必要になります。名前検索では0.3-0.4が通常適切です：タイポを検出するには十分に柔軟であり、ノイズを避けるには十分に厳密です。

**トリグラムはプレフィクス検索やオートコンプリートにも役立ち、特にオートコンプリートがタイポ対応や部分一致が必要な場合に有効です：**

```sql
-- オートコンプリート：プレフィクス一致。純粋な左固定プレフィクスの場合、
-- トリグラムのGINインデックスをデータのB-treeパターンインデックスと比較してください。
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- より細かい制御：長文内での部分一致にword_similarityを使用
-- （例："Johnson"を"Andrew Johnson III"内で検索する場合に有用）
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name          -- <% 演算子 = word_similarityしきい値
ORDER BY score DESC
LIMIT 10;
```

**`pg_trgm`をFTSよりも選ぶべき状況：**

| シナリオ | 使用するもの |
|---|---|
| 人名/企業名検索（タイポあり） | `pg_trgm` |
| オートコンプリート/プレフィクス検索 | `pg_trgm`（またはFTSのプレフィクスクエリ） |
| 短い文字列、コード、識別子の検索 | `pg_trgm` |
| 記事やドキュメントの検索 | FTS |
| ログメッセージのキーワード検索 | FTS |
| 多言語名検索 | `pg_trgm`（言語に依存しない） |

`pg_trgm`はFTSと組み合わせて使用できます。3-グラムを使用してファジーな事前フィルタを行い、`ts_rank`でランク付けする、または3-グラムの類似度とベクトルスコアを組み合わせることもできます。

## pgvectorが適しているケース

**RAGの構築中です。** RAGは意味論的検索に依存します。ユーザーの質問と意味が最も近いドキュメントの*チャンク*を検索します。語彙が異なる場合でもです。ベクトル検索はこの目的に特化しています。FTSは言い換え表現や類義語、概念的なマッチを検出できません。

**ユーザーは欲しいものではなく検索語を指定しません。** 「夏の夕方に軽い何か」には明確なワインキーワードが含まれません。「新任マネージャーが自信を築く方法に関する記事」はFTSでは提供できない意味論的理解を必要とします。

**類似項目の検索を行います。** 関連製品、類似サポートチケット、重複するバグ報告。"このチケットに類似した問題を検索して"という要求はベクトル演算です。新しい問題を埋め込み、最近傍の類似物を検索します。

**多言語コンテンツ。** 多言語データで訓練されたベクトル埋め込みは、言語間でマッチングできます。FTSは言語固有の設定を必要とし、クロス言語クエリの処理が困難です。

```sql
-- 設定
CREATE EXTENSION IF NOT EXISTS vector;

ALTER TABLE documents ADD COLUMN embedding vector(1536);
CREATE INDEX documents_embedding_idx
  ON documents USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);

-- クエリ: 語義検索
SELECT id, title, 1 - (embedding <=> $1::vector) AS similarity
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 10;
```

注意: `ivfflat` は近似検索です。高速ですが、再現率を速度にトレードしています。データセットが小さい場合（約100万行未満）、`hnsw` が通常より適しています:

```sql
CREATE INDEX documents_embedding_idx
  ON documents USING hnsw (embedding vector_cosine_ops);
```

## 両方が必要な場合

技術ドキュメントでは、単純な分割では対応できません。ユーザーは「タイムアウトを設定する方法」を検索する一方で、`withRetry()` といった関数名や `ECONNRESET` といったエラーコードも検索します。

ベクターセルチは概念的なクエリを処理します。全文検索 (FTS) は正確な用語を処理します。どちらも単独では両方のクエリを効果的に処理できません。

解決策はハイブリッド検索です。両方を実行し、結果を統合します。

**Reciprocal Rank Fusion (RRF)** はここでの標準アルゴリズムです。2つのシステムからのスコアを正規化する必要はありません。代わりに、順位の位置を組み合わせます。

```sql
-- Hybrid search with Reciprocal Rank Fusion
WITH fts_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM documents, to_tsquery('english', $1) query
  WHERE search_vector @@ query
  LIMIT 50
),
vector_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> $2::vector) AS rank
  FROM documents
  ORDER BY embedding <=> $2::vector
  LIMIT 50
),
rrf AS (
  SELECT
    COALESCE(f.id, v.id) AS id,
    COALESCE(1.0 / (60 + f.rank), 0) + COALESCE(1.0 / (60 + v.rank), 0) AS rrf_score
  FROM fts_results f
  FULL OUTER JOIN vector_results v ON f.id = v.id
)
SELECT d.id, d.title, rrf.rrf_score
FROM rrf
JOIN documents d ON d.id = rrf.id
ORDER BY rrf_score DESC
LIMIT 10;
```

分母にある `60` はRRF定数です。値が大きいほど順位の差の影響が小さくなり、値が小さいほど影響が大きくなります。60のデフォルト値は、ほとんどの場合で良好に動作します。

このクエリは、1つのクエリで全文検索とベクトル類似度検索の2つの検索を実行し、順位を融合し、キーワードと意味的信号が一致する結果に報酬を与えます。

![ハイブリッド検索パイプライン。1つのクエリが全文検索とpgvectorに分岐し、それぞれが順位付けられた結果を生成し、Reciprocal Rank Fusionが2つのリストを統合する。](../hybrid-rrf-pipeline.svg)

_Reciprocal Rank Fusion（RRF）は、`ts_rank`とコサイン距離を直接比較する偽装を避ける価値があります。RRFは単に「この結果がそれぞれのリストでどのくらい上位に現れたか？」を尋ねるだけです。_

## 実用的な決定木

検索戦略を選ぶ際には、まず**入力の形状**から考え、次に**ユーザーが行っているクエリの種類**を尋ねる必要があります。「継字の変種を含む短い文字列」と「正確な用語が重要な長い文章」は異なる問題であり、これらは「ドキュメントチャンクに関する質問」とも異なります。

![名前、住所、タイトル、オートコンプリート、またはタイプミスの多い短い文字列などのテキスト形状とユーザーのクエリ方法に基づいて、pg_trgm、全文検索、pgvector、ハイブリッド検索、または専用ベクトルデータベースを選択する決定木](../search-decision-tree.svg)

同じ決定木を言葉で表したバージョン:

- **名前、住所、タイトル、オートコンプリート、またはタイプミスの多い短い文字列** → `pg_trgm`
- **既知の単語、エラーコード、SKU、関数名、タグ、カテゴリ、フィルター** → 全文検索(FTS)
- **質問、言い換え、おすすめ、関連アイテム、マルチリンガルマッチング、RAGチャンク** → pgvector
- **ユーザーが正確な記号と概念的な答えの両方を必要とする技術的コンテンツ** → RRFを使用したハイブリッド検索
- **プライマリキー、正確なID、権限フィルター、日付範囲、ソート済みリスト** → 通常のSQLインデックス
- **ベクトルデータの量が非常に多く、非常に高い同時接続性、またはPostgreSQLがベンチマークで達成できない遅延目標がある場合** → 専用ベクトルストアの評価

### 全文検索 vs. セマンティック検索: 簡潔なバージョン

「FTSとベクター検索のどちらを使うべきか？」という質問は、通常次のように簡略化されます。**関連するドキュメントに何の単語が含まれるかわかっていますか？**

はいの場合 - ユーザーが既知の用語、カテゴリ、関数名、製品コードを検索する場合 - FTSはより高速で、コストが安く、予測可能です。検索結果がマッチした理由を教えてくれます。

If no - users describe a concept, ask a question, or search in a different language - vector search is the right tool. It matches meaning, not words.  
ベクター検索が適切なのは、ユーザーが概念を説明したり、質問をしたり、異なる言語で検索したりするケースです。これは単語ではなく意味をマッチさせます。  

The tricky middle is natural language queries over technical content. Someone searching "how do I handle connection drops" might need an article titled "implementing retry logic for network failures" - no overlapping words, high semantic relevance. That's where vector search earns its keep.  
技術的なコンテンツに対する自然言語クエリは難しい中間領域です。例えば「接続が切断された場合の対処方法」を検索しても、タイトルが「ネットワーク障害時のリトライロジックの実装」の記事が必要になるかもしれません。このケースでは単語の重複がなく、セマンティックな関連性が高いため、ベクター検索が真の価値を発揮します。  

The other tricky case is **names and proper nouns**. Neither FTS nor vector search is great at them:  
- FTS will miss "Micheal" when searching "Michael" — different tokens  
- Vector search will miss the名 entirely if it doesn't appear frequently in training data  
- `pg_trgm` handles this correctly: orthographic similarity, not semantic or lexical  
もう1つの難しいケースは**名前や固有名詞**です。全文検索(FTS)もベクター検索もこれらを正確に扱うのは難しいです：  
- FTSは「Michael」を検索しても「Micheal」を見逃す場合があります（トークンが異なるため）  
- ベクター検索はトレーニングデータに頻出しない名前を完全に見逃す可能性があります  
- `pg_trgm`はこれを正しく処理します：表記の類似性を評価し、セマンティックや語彙的類似性は評価しません

実際には、コンテンツ中心の検索ボックスにはFTSが必要で、速度とキーワード検索を実現します。ユーザーが名前を検索する必要があるかどうかによって、ハイブリッド検索または`pg_trgm`が必要になる場合もあります。真のセマンティック検索機能が必要な場合は通常、pgvectorが使われます。RAGが必要な場合は必ずpgvectorが使われます。

## 専用ベクトルストアが必要な場合

一部のシステムは本当にpgvectorを成長しすぎることがあります。そうなった場合、市場は混雑しています。トップオプションで何が重要かを以下に示します。

### 機能マトリクス

表が理解できる前にいくつかの列を解説する必要があります。

**ハイブリッド検索**とは、BM25キーワード検索とベクトル類似度を1つのクエリで実行し、Reciprocal Rank Fusion（逆数ランク融合）でマージすることを意味します。"withRetry timeout"は関数名を正確に一致させるだけでなく、"ネットワーク障害時のリトライロジック"に関するドキュメントをセマンティックにマッチングします。ハイブリッド検索がない場合、検索モードを1つ選ぶか、自分で2つのクエリを統合する必要があります。pgvectorの「手動（SQL経由のRRF）」は[上記で示したアプローチ](#when-you-need-both)です。機能はありますが、自分で実装する必要があります。

**スパースベクトル**はBM25よりもさらに進んでいます。SPLADEスパースベクトルは約30,000次元（語彙語1語あたり1次元）を持ち、約98%がゼロです。非ゼロの位置はどの語彙語が重要で、どの程度重要かを示します。"dogs"というクエリは"canine"や"pet"も重み付けします。これにより、BM25レベルのキーワード精度に加えて、ベクトルインデックス内で語彙拡張が可能になります。この列がfalseの場合、正確な語彙語クエリには外部のFTSレイヤーが必要です。

```python
# SPLADE: 約30,000次元、約60非ゼロ — 関連する語彙語の位置のみが発火
def encode_splade(text: str) -> dict:
    tokens = tokenizer(text, return_tensors="pt", truncation=True, max_length=512)
    with torch.no_grad():
        output = model(**tokens)
    vec = torch.log1p(torch.relu(output.logits)).max(dim=1).values.squeeze()
    return {"indices": vec.nonzero().squeeze().tolist(), "values": vec[vec != 0].tolist()}
```

**SQL / SQLライク**は実際にはフィルタリングについてです。フィルタリングなしのベクトル検索はデモであり、アプリケーションではありません。テナントスコープ、日付範囲、権限、カテゴリなど、検索に加えてフィルタリングが必要です。フルSQL（pgvector）は既存の結合と併用してこれを表現できます。特化されたDBはJSONフィルターオブジェクト（Qdrant、Pinecone）、クエリDSL（Elasticsearch、Milvus）、またはGraphQL（Weaviate）を使用します。これらは機能しますが、フィルタロジックが複雑になるほどSQLの魅力が高まります。

```sql
-- pgvector: ベクタ類似度はWHERE句の別の表現に過ぎない
SELECT id, title, 1 - (embedding <=> $1) AS score
FROM documents
WHERE tenant_id = $2 AND category = ANY($3::text[]) AND created_at > NOW() - INTERVAL '90 days'
ORDER BY embedding <=> $1 LIMIT 10;
```

```python
# Qdrant: 同等のフィルタをPythonオブジェクトとして表現 — 機能はするが儀礼的な手順が多い
results = client.query_points(
    collection_name="documents", query=query_embedding,
    query_filter=models.Filter(must=[
        models.FieldCondition(key="tenant_id", match=models.MatchValue(value=tenant_id)),
        models.FieldCondition(key="category",  match=models.MatchAny(any=categories)),
        models.FieldCondition(key="created_at", range=models.DatetimeRange(gte=cutoff)),
    ]),
    limit=10,
)
```

**マルチモーダルネイティブ**は「画像の埋め込みベクトルを保存できる」ことを意味しない。すべてのデータベースはfloat配列を保存できる。これは、データベースが非テキストコンテンツの埋め込みモデルを提供することを意味する。つまり、生の画像URLを渡すだけでベクトル化を処理してくれる。ここではほとんどのデータベースが埋め込みに依存せず、パイプラインはユーザー自身が管理する必要がある。MarqoとWeaviate（CLIP/ImageBindモジュール経由）はこのプロセスを閉じている。

**ディスクベースのインデックス**はコストの問題である。生のベクトル、グラフオーバーヘッド、メタデータを含めると、100万件の1536次元ベクトルあたりRAMにHNSWを保持するには数GBのRAMが必要になる。ディスクネイティブな代替案（Milvus DiskANN、Elasticsearch DiskBBQ、LanceDBのLance形式、Turbopufferのオブジェクトストレージ階層）は、多くの場合クエリ遅延を少し犠牲にしてインフラコストを削減する。モデル遅延が支配的なRAGワークロードでは、ベンチマーク検証が価値がある。Redis VSSは厳密な制約：RAM専用で、ディスクパスは存在しない。

**最大次元数**は今日の選択が明日のマイグレーションを隠している。`text-embedding-3-large`は3072次元を使用し、Jina v3はさらに大きい埋め込みを出力可能で、研究モデルはさらに高次元を追求している。一部のマネージドサービスは硬い次元上限を公表しているが、他のサービスは高次元の上限や実用的な上限なしを記載している。コミットする前に最新のドキュメントを確認せよ。十分な余裕があるものを選べ — 次元上限に達したためにベクターインデックスをマイグレートするのは地獄のようなスプリントになる。

### マトリクスの読み方

いくつかの点は表にうまく収まらない:  

**スパースベクトル**は、ベクトルインデックス内でBM25品質のキーワード一致を実現し、別途の全文検索エンジンを必要としない方法である。QdrantやElasticsearchは特に成熟した実装を持っている。WeaviateはBM25Fを通じてサポートしている。ハイブリッド検索が重要であり、2つのシステムを運用できない場合は、スパースベクトルのサポートを検討すべきである。  

**ディスクベースのインデックス**はコスト調整の手段であり、実装の詳細ではない。RAMに常駐するHNSWインデックスは高速だが、ベクトル数、次元数、メタデータ、グラフのオーバーヘッドが増加するにつれてコストが高くなる。ディスクベースの代替案（Milvus DiskANN、Elasticsearch DiskBBQ、Turbopufferオブジェクトストレージ、LanceDBのLance形式）はクエリ遅延とインフラコストのトレードオフを提供する。大規模なRAGインデックスでは、このトレードオフは多くの場合テストに値する。

**Turbopufferのマルチテナントアーキテクチャ**は、非常に高い名前空間数を基盤としている。その公開的なポジショニングや顧客事例は、Notionのような大規模な名前空間中心のコルパスを含むワークロードを強調している。ユーザーまたは組織ごとにベクトル検索の分離が必要な場合、このアーキテクチャは経済性に変化をもたらす可能性があるが、依然として自社のテナント構造をベンチマークする必要がある。  

**LanceDBの埋め込みモード**は、「ベクトル検索用のSQLite」とも言える存在である。プロセス内で動作し、サーバーを必要とせず、LambdaやCloudflare Workers、エッジ環境でも利用可能である。Lanceのカラム形式により、実際の大規模なワークロードにおいても埋め込み運用が現実的になる。  

**Oramaは検索UXのインフラストラクチャであり、データウェアハウスではない**。JavaScriptアプリケーション内、エッジ環境、または管理されたサイト/アプリ検索レイヤーで、コンパクトな全文検索/ベクトル検索/ハイブリッド検索エンジンを構築したい場合に最適である。10億ベクトル規模の検索、大規模な分析、複雑なフィルタリング付き結合を必要とするワークロードには、このツールは選択しないだろう。

**Vespaは検索が製品の一部に過ぎない場合に選ぶツールである。** レキシカル検索、最近傍探索、テンソル、ランキング式、グルーピング、オンラインサーブを組み合わせている。その強力さは実在するが、運用やモデリングの複雑さも同様に現実的である。検索/レコメンドチーム向けに適しているが、「CRUDアプリにセマンティック検索を追加する」ようなケースには向きにくい。  

**ClickHouseは検索が分析と結びついている場合に検討すべき存在である。** イベント、ログ、トレース、メトリクス、または大規模なファクトテーブルが真のソースである場合、ClickHouseはベクトル距離、フィルタリング、集約、そして最近の本格的な全文インデックスを1つのSQLエンジンで処理できる。これは目的別に設計されたベクトルデータベースではないが、分析的な検索においては「最も良い方法で堅実な答え」となる可能性がある。  

**Chromaは開発/テストや小規模アプリの展開に最も強い。** 非常に大きなコーパス、HA（高可用性）、ディスク中心の運用、または本格的なハイブリッド検索を目標とする場合、プロトタイプをインフラに昇格させる前に生産性を重視したストアの評価を行うべきである。

### 簡略化された決定フロー  

PostgreSQLのpgvectorが本当に限界に達した場合（通常はベンチマークでベクトル数、フィルタリング、書き込みレート、高同時性時の遅延がPostgreSQLの限界を超える場合）は、制約に応じて以下のように選択してください：  

- **テナントごとの隔離が必要なSaaS製品** → Turbopuffer  
- **Rustレベルのパフォーマンス＋複雑なメタデータフィルタリングが必要** → Qdrant  
- **Elastic/ELKスタックを使用中** → DiskBBQ付きElasticsearch  
- **AWS環境でオープンソースを希望** → OpenSearch  
- **検索/レコメンドプラットフォームで高度なランキングが必要** → Vespa  
- **分析、オブザーバビリティ、ログ/イベント検索** → ClickHouse  
- **オンプレ/セルフホストで10億スケール** → Milvus  
- **エッジ/サーバーレス/マルチモーダル** → LanceDB  
- **小型JSアプリ、ドキュメントサイト、エッジネイティブ検索UI** → Orama  
- **運用コストを無視し、ゼロオペレーションで動作** → Pinecone  
- **マルチモーダル優先（画像、動画、音声）** → Marqo  
- **MongoDBを使用中** → Atlas Vector Search  
- **PostgreSQL環境でさらにスケーラビリティが必要** → Supabase VectorまたはNeon（どちらもpgvectorを管理し、ツールが充実）

## 避けるべきこと

正解が明確な検索タスクに対して、ベクターサーチをファジーなテキスト検索として使用しないでください。  
---

"メールアドレス `dan@example.com` を持つユーザーを検索する" はベクターサーチの問題ではありません。"注文ID `ORD-12345` を持つ注文を検索する" も同様です。`ORD-12345` を埋め込みベクトルに変換して注文テーブルに対してコサイン類似度を計算しても、*何か* は返ってきますが、それは正確ではありません。これらは完全一致の問題です。主キーまたは通常のインデックスを使用してください。

ベクターサーチはデータセット内で*最も類似している*ものを返します。関連性がまったくない場合でも例外ではありません。良い答えがないことを認識しません。関連性のあるドキュメント検索には適していますが、特定のレコード検索では、誤った近似が空の結果よりもはるかに悪影響を及ぼします。

各ツールが何のために存在するのかを理解してください。これらはすでにPostgreSQLにインストールされています。それぞれが適した場面で活用しましょう。
