チームが AI 機能を追加するときは、まず専用のベクトルデータベースに手を伸ばすことが多い。

Pinecone、Weaviate、Qdrant、Chroma。新しいサービス、新しい依存関係、新しい接続プール、新しいコスト、そして正確さを保つための二つの真実のソース。

一方で、すでに PostgreSQL がある。PostgreSQL には `pgvector` が組み込まれている。さらに、2008 年から優れたフルテキスト検索機能も備わっている。

専用のベクトルストアは、大規模かつ高クエリ負荷の環境で初めてコストに見合う価値を発揮する。しかし多くのアプリケーションは、第一の検索システムを本格的に使い込む前に、第二の検索システムに手を伸ばす。これが、将来のスケール問題が今日の同期バグへと変わる原因だ。

つまり、FTS を使うべきタイミング、pgvector を使うべきタイミング、そして両者を併用すべきタイミングはいつか、ということになる。

## 実際に何をするか

フルテキスト検索（`tsvector` / `GIN` インデックス）は字句的です。テキストをレキシームに分割し、語幹化してインデックスに格納し、クエリをそのインデックスと照合します。たとえば “Running” と “runs” は同じレキシームに統合されます。 “dog” と “dogs” も同様です。ランキング関数（`ts_rank`）は、クエリ語が頻繁に出現する、または目立つ位置にある文書に対してスコアを上げます。

pgvector は意味的です。密なベクトル――数値のリスト――を格納し、埋め込みモデルが捉えるチャンクの *意味* を表現します。類似度検索は、その高次元空間で近接するベクトルを探し出します。 “Dog” と “canine” は互いに近くなりますが、スポーツとしての “Running” とプロセスとしての “running” は必ずしも近くなりません。

実用的な違いは次の通りです。FTS は「これらの単語を含む文書はどれか？」に答え、ベクトル検索は「この内容に概ね近い意味を持つ文書はどれか？」に答えます。

![A search tool map showing pg_trgm for short fuzzy strings, full-text search for exact prose queries, pgvector for semantic matching, and hybrid search for long content that needs both exact and semantic signals.](../search-tool-map.svg)

_最初の分割は「従来の検索 vs. AI検索」ではありません。テキストの形状と、どのような回答が正しいかという観点です。_

## Full‑Text Search が優位になるケース

**正確に一致する語句を検索している** 場面です。製品 SKU、エラーコード、型番、ユーザー名、法的条項の参照などです。`SKU-AX-44192` は意味的に類似するものとは無関係です。マッチすればマッチするだけで、そうでなければマッチしません。ベクトル検索は自信を持って `SKU-AX-44193` を返すかもしれませんが、これは求めている結果ではありません。

**クエリがキーワードベースです。** ユーザーは検索ボックスに入力し、タグで絞り込み、またはキーワードでブログ記事を検索します。FTS はそのような意図の形に合わせて設計されています。

**GPU や埋め込みインフラなしでランク付けされた結果が必要です。** FTS インデックスは高速で決定論的であり、外部 API 呼び出しを必要としません。`tsvector` カラムを追加し、GIN インデックスを作成すれば完了です。

**検索と同時にブールフィルタリングを行います。** `WHERE to_tsvector(body) @@ to_tsquery('postgres') AND category = 'tutorial' AND published_at > NOW() - INTERVAL '6 months'` — 既存のクエリロジックと自然に組み合わせられます。

```sql
-- Create the index
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body, '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Query
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`GENERATED ALWAYS AS` カラムはインデックスを自動的に最新の状態に保ちます。`setweight` により、タイトルの一致が本文の一致よりも高いランクを付与されます。これで設定は完了です。

## Trigram が有利になるケース (pg_trgm)

PostgreSQL にはもう一つ、見落とされがちなツールがあります: `pg_trgm`。これはフルテキスト検索でもベクトル検索でもなく、あいまい文字列マッチングです。両者が苦手とする中間領域をカバーします。

**利用ケース: 誤字が混じる名前、住所、識別子、短い文字列。**

FTS はテキストを字句に分解しステミングします。文章には有効ですが、次のようなケースには不向きです。
- 人名（例: "Dan Levy" が言語設定により "dan levi"、"leiv" へステム化される）
- 企業名、住所、製品名など、正確な綴りが重要な文字列
- 誤字を含むクエリ — "Micheal Jordan"、"Amaon"、"javascipt"
- オートコンプリート／プレフィックス検索
- 部分文字列マッチング（"son" が "Johnson"、"Anderson" に一致する）

pgvector もここでは適切ではありません。たとえば「Micheal Jordan」をベクトル化して最近傍ベクトルを検索すると、埋め込み空間は綴りではなく意味で名前を整理します。そのため、最も近い近傍は「basketball legend」や「Michael B. Jordan」になる可能性が高く、誤字を含むユーザーレコードにはヒットしません。

`pg_trgm` は文字列を重複する 3 文字スライスに分割し、2 つの文字列が共有するトリグラム数で類似度を測ります。たとえば「Dan」→`" da"`, `"dan"`, `"an "` のようになります。「Micheal」と「Michael」はほとんどのトリグラムを共有するため、類似度が高くなります。

```sql
-- Enable the extension (usually already available)
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- GIN index on names column — enables fast trigram similarity search
CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- Fuzzy name search: finds "Micheal Jordan" when searching "Michael Jordan"
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % operator = similarity threshold (default 0.3)
ORDER BY score DESC
LIMIT 10;

-- Or use ILIKE with trigram index support for contains matching
SELECT id, name
FROM users
WHERE name ILIKE '%johnson%'   -- GIN index makes this fast
LIMIT 10;
```

`%` 演算子は `pg_trgm.similarity_threshold`（デフォルト 0.3、範囲 0‑1）を使用します。値が高いほどより厳密な一致が要求されます。名前検索では 0.3‑0.4 が実務上妥当です。許容範囲は誤字を捕捉できる程度に緩く、ノイズを排除するほどに厳しくなります。

**Trigram はプレフィックス検索やオートコンプリートでも役立ちます。特にオートコンプリートで誤字許容や部分一致が必要な場合は有効です。**

```sql
-- オートコンプリート: プレフィックス一致。左側に固定されたプレフィックスだけを対象にする場合は、
-- データ上の B ツリー パターンインデックスに対して trigram GIN を比較します。
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- もっと細かく制御したい場合: 長い文字列中の部分一致に word_similarity を使用
-- （例: "Andrew Johnson III" の中から "Johnson" を検索）
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name          -- <% 演算子 = word_similarity 閾値
ORDER BY score DESC
LIMIT 10;
```

**`pg_trgm` を FTS より選択すべきケース:**

| シナリオ | 使用すべきツール |
|---|---|
| 人名・社名検索で誤字が出る可能性がある | `pg_trgm` |
| オートコンプリート / プレフィックス検索 | `pg_trgm`（またはプレフィックスクエリを使った FTS） |
| 短い文字列、コード、識別子の検索 | `pg_trgm` |
| 本文記事やドキュメントの検索 | FTS |
| ログメッセージ内のキーワード検索 | FTS |
| 多言語の名前検索 | `pg_trgm`（言語非依存） |

`pg_trgm` は FTS とも組み合わせて使用できます。トリグラムであいまいな事前フィルタをかけ、`ts_rank` でランク付けするか、トリグラム類似度とベクトルスコアを組み合わせます。

## pgvector が優位になるケース


**RAG を構築している場合。** RAG は意味的検索に依存します。ユーザーの質問と意味が最も近い文書 *チャンク* を見つける必要があり、表現が異なっていても検索できなければなりません。ベクトル検索はこの目的のために設計されています。FTS ではパラフレーズや同義語、概念的な一致は拾いきれません。

**ユーザーは「何を検索すべきか」ではなく「何を求めているか」を記述します。** 「夏の夜に飲む軽めのもの」には明確なワインキーワードがありません。「新しいマネージャーとして自信をつける方法」についての記事を探すには、FTS が提供できない意味的理解が必要です。

**類似アイテムを探す場合。** 関連商品、類似したサポートチケット、重複したバグレポートなどです。「このチケットに似た問題を探す」 はベクトル演算です。新しいチケットを埋め込み、最も近い近傍を検索します。

**多言語コンテンツ。** 多言語データで学習されたベクトル埋め込みは、言語を跨いだマッチングが可能です。FTS は言語ごとの設定が必要で、跨言語クエリの取り扱いは苦手です。

```sql
-- Setup
CREATE EXTENSION IF NOT EXISTS vector;

ALTER TABLE documents ADD COLUMN embedding vector(1536);
CREATE INDEX documents_embedding_idx
  ON documents USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);

-- Query: semantic search
SELECT id, title, 1 - (embedding <=> $1::vector) AS similarity
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 10;
```

※ `ivfflat` は近似検索です。高速ですが、リコールを若干犠牲にして速度を得ます。データセットが小規模（約 100 万行未満）であれば、`hnsw` の方が一般的に適しています。

```sql
CREATE INDEX documents_embedding_idx
  ON documents USING hnsw (embedding vector_cosine_ops);
```

## 両方が必要なとき

技術ドキュメントでは、単純な分割が通用しません。ユーザーは「タイムアウトの設定方法」のような概念的な検索を行うと同時に、`withRetry()` のような関数名や `ECONNRESET` といったエラーコードも検索します。

ベクトル検索は概念的なクエリに対応します。FTS は正確な語句に対応します。どちらも単独で両方をうまく処理できません。

ハイブリッド検索が解決策です：両方を実行し結果を融合します。

**Reciprocal Rank Fusion (RRF)** はここで標準的に使われるアルゴリズムです。2 つのシステムからスコアを正規化する必要はなく、ランク位置だけを組み合わせます。

```sql
-- Hybrid search with Reciprocal Rank Fusion
WITH fts_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM documents, to_tsquery('english', $1) query
  WHERE search_vector @@ query
  LIMIT 50
),
vector_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> $2::vector) AS rank
  FROM documents
  ORDER BY embedding <=> $2::vector
  LIMIT 50
),
rrf AS (
  SELECT
    COALESCE(f.id, v.id) AS id,
    COALESCE(1.0 / (60 + f.rank), 0) + COALESCE(1.0 / (60 + v.rank), 0) AS rrf_score
  FROM fts_results f
  FULL OUTER JOIN vector_results v ON f.id = v.id
)
SELECT d.id, d.title, rrf.rrf_score
FROM rrf
JOIN documents d ON d.id = rrf.id
ORDER BY rrf_score DESC
LIMIT 10;
```

分母の `60` は RRF の定数です。値が大きいほどランク差の影響が減少し、値が小さいほど差が強調されます。デフォルトの 60 は多くのケースで十分に機能します。

このクエリは 1 つの問い合わせで 2 つの検索を実行し、ランクを融合させ、キーワードとセマンティックのシグナルが一致する結果に報酬を与えます。

![ハイブリッド検索パイプライン：1 つのクエリがフルテキスト検索と pgvector に分岐し、各々がランク付けされた結果を生成し、Reciprocal Rank Fusion が 2 つのリストを結合する様子。](../hybrid-rrf-pipeline.svg)

_RRF が有用なのは、`ts_rank` とコサイン距離を直接比較可能なスコアとして扱うのを避ける点にあります。RRF は単に「この結果が各リストのどれだけ上位に現れたか？」を問うだけです。

## 実践的な意思決定ツリー

検索戦略を選ぶときは、まず **入力の形状** を確認し、次に **ユーザーがどんなクエリを行っているか** を問います。 「綴りの揺れがある短い文字列」は、 「正確な語句が重要な長文」 と同じ問題ではなく、さらに 「文書のチャンクに対する質問」 とは別のケースです。

![テキストの形状とユーザーのクエリ方法に基づいて、pg_trgm、全文検索、pgvector、ハイブリッド検索、または専用ベクトルデータベースを選択する意思決定ツリー。](../search-decision-tree.svg)

同じ内容を文章で示すと:

- **名前、住所、タイトル、オートコンプリート、または綴りミスが起きやすい短い文字列** → `pg_trgm`
- **既知の単語、エラーコード、SKU、関数名、タグ、カテゴリ、フィルタ** → FTS
- **質問、パラフレーズ、レコメンデーション、関連アイテム、多言語マッチング、RAGチャンク** → pgvector
- **ユーザーが正確なシンボルと概念的な回答の両方を必要とする技術コンテンツ** → ハイブリッド（RRF 使用）
- **主キー、正確な ID、権限フィルタ、日付範囲、ソート済みリスト** → 通常の SQL インデックス
- **ベクトル量が膨大、同時実行数が非常に高い、またはベンチマークで Postgres が達成できないレイテンシ目標** → 専用ベクトルストアの導入を検討


### FTS vs. Semantic: The Short Version

「FTS を使うべきかベクトル検索を使うべきか？」という質問は、ほとんど次の点に集約されます。**関連文書にどんな単語が出てくるか、事前に把握していますか？**

もし把握しているなら—ユーザーが既知の用語、カテゴリ、関数名、製品コードで検索する場合—FTS の方が高速でコストも低く、予測可能です。結果がマッチした理由も示してくれます。

もし「いいえ」‑ ユーザーが概念を説明したり、質問したり、別の言語で検索したりする場合‑ ベクトル検索が適切なツールです。意味でマッチし、単語そのものではありません。

ややこしい中間ケースは、技術コンテンツに対する自然言語クエリです。たとえば「接続が切れたときの対処方法」を検索したユーザーが、実際には「ネットワーク障害時のリトライロジック実装」という記事を必要とする場合です。単語の重なりはなく、意味的な関連性が高い。このようなケースでベクトル検索が価値を発揮します。

もう一つのややこしいケースは **名前や固有名詞** です。FTS もベクトル検索もこれらには弱いです：
- FTS は「Michael」を検索したときに「Micheal」を見逃します ― トークンが異なるため
- ベクトル検索は、名前が学習データに頻繁に現れない場合、名前自体を見落とします
- `pg_trgm` はこれを正しく処理します：語形的類似性であり、意味的でも語彙的でもありません。

実務では、コンテンツが大量にある検索ボックスの多くは高速なキーワード検索のために FTS が必要で、ユーザーが名前で検索するかどうかに応じてハイブリッド検索や `pg_trgm` を併用することがあります。真に意味的な検索機能を実装する場合は通常 pgvector が必要です。RAG（Retrieval‑Augmented Generation）を行う場合は必ず pgvector が前提となります。

## 専用のベクトルストアが本当に必要な場合


一部のシステムは実際に pgvector の限界を超えてしまいます。そのようなケースになると、市場には多種多様な選択肢があふれますが、重要なのは次の点です。

### 機能比較表

表を理解するために、いくつかの項目を事前に説明しておきます。

**Hybrid search** は BM25 キーワード検索とベクトル類似度を同一クエリで実行し、Reciprocal Rank Fusion でマージすることを指します。`withRetry timeout` のように、関数名を正確に一致させつつ、意味的に「ネットワーク障害時のリトライロジック」についてのドキュメントもヒットさせられます。ハイブリッド検索を使わない場合は、検索モードを一つ選ぶか、二つのクエリを自前で融合する必要があります。pgvector の「Manual (RRF via SQL)」は[上記のアプローチ](#when-you-need-both)と同様で、動作はしますが実装は自分で行うことになります。

**Sparse vectors** は BM25 を超える表現力を持ちます。SPLADE のスパースベクトルは約 30,000 次元（語彙項目ごとに 1 つ）で、約 98% がゼロです。非ゼロの位置が、どの語が重要か、どれだけ重要かを示します。たとえば「dogs」というクエリは「canine」や「pet」も重み付けされます。これは BM25 レベルのキーワード精度に加えて、ベクトルインデックス内部での語彙展開を実現します。このカラムが false の場合、正確な語句検索のために外部の FTS レイヤが必要になります。

```python
# SPLADE: ~30,000 dims total, ~60 non-zero — only the relevant vocabulary positions fire
def encode_splade(text: str) -> dict:
    tokens = tokenizer(text, return_tensors="pt", truncation=True, max_length=512)
    with torch.no_grad():
        output = model(**tokens)
    vec = torch.log1p(torch.relu(output.logits)).max(dim=1).values.squeeze()
    return {"indices": vec.nonzero().squeeze().tolist(), "values": vec[vec != 0].tolist()}
```

**SQL / SQL-like** は実質的にフィルタリングを指します。フィルタなしのベクトル検索はデモに過ぎず、実運用ではテナントスコープ、日付範囲、権限、カテゴリといった条件が必須です。フル SQL（pgvector）であれば、既存の JOIN と同時にこれらを記述できます。専用 DB は JSON フィルタオブジェクト（Qdrant、Pinecone）やクエリ DSL（Elasticsearch、Milvus）、GraphQL（Weaviate）を提供します。いずれも機能しますが、フィルタロジックが複雑になるほど SQL の方が魅力的になります。

```sql
-- pgvector: ベクトル類似度は WHERE の別の式にすぎない
SELECT id, title, 1 - (embedding <=> $1) AS score
FROM documents
WHERE tenant_id = $2 AND category = ANY($3::text[]) AND created_at > NOW() - INTERVAL '90 days'
ORDER BY embedding <=> $1 LIMIT 10;
```

```python
# Qdrant: 同等のフィルタを Python オブジェクトで表現 — 関数型でやや冗長
results = client.query_points(
    collection_name="documents", query=query_embedding,
    query_filter=models.Filter(must=[
        models.FieldCondition(key="tenant_id", match=models.MatchValue(value=tenant_id)),
        models.FieldCondition(key="category",  match=models.MatchAny(any=categories)),
        models.FieldCondition(key="created_at", range=models.DatetimeRange(gte=cutoff)),
    ]),
    limit=10,
)
```

**マルチモーダルネイティブ** は「画像埋め込みを保存できる」ことを意味するわけではない。すべてのデータベースが浮動小数点配列を格納できるだけだ。重要なのは、データベースがテキスト以外のコンテンツ用に埋め込みモデルを同梱しており、生の画像 URL を渡すだけでベクトル化を処理してくれる点だ。ここに列挙された多くのデータベースは埋め込みに依存しない設計なので、パイプラインは自前で用意する必要がある。Marqo と Weaviate（CLIP / ImageBind モジュール経由）はこのループを閉じてくれる。

```python
# Marqo: 生画像を POST し、テキストでクエリ — 外部の埋め込みステップ不要
mq.index("products").add_documents(
    [{"id": "shoe-001", "image": "https://cdn.example.com/shoes/001.jpg"}],
    tensor_fields=["image"]
)
results = mq.index("products").search(q="lightweight shoes for summer")
# キーワードが全く重なっていなくても shoe-001 が返る — CLIP がクロスモーダルマッチを処理
```

**ディスクベースのインデックス** はコストの問題だ。RAM 常駐の HNSW は、素のベクトル、グラフオーバーヘッド、メタデータをすべて合計すると、100 万件の 1536 次元ベクトルで数 GB の RAM が必要になることがある。ディスクネイティブの代替手段（Milvus DiskANN、Elasticsearch DiskBBQ、LanceDB の Lance 形式、Turbopuffer のオブジェクトストレージ層） は、クエリ遅延を若干犠牲にしてインフラコストを下げることが多い。RAG ワークロードのようにモデル遅延がすでに支配的なケースでは、ベンチマークに値することが多い。Redis VSS はハードな制約がある：RAM のみでディスクパスは存在しない。

**最大次元数** は、今日の選択に潜む明日のマイグレーションリスクだ。`text-embedding-3-large` は 3072 次元を使用し、Jina v3 はさらに大きな埋め込みを出力でき、研究モデルは常に次元数を押し上げている。いくつかのマネージドサービスはハードな次元上限を公開しているが、他は実務上の上限が高いか、典型的な埋め込みモデルに対して実質的な上限を設けていない。導入前に最新ドキュメントを確認し、余裕を持った選択をすべきだ。次元上限に突き当たってベクトルインデックスを移行するのは、非常に苦痛なスプリントになる。

### マトリックスの読み方

テーブルにすっきり収まらない点がいくつかあります。

**スパースベクトル**は、別個の全文検索エンジンを用意せずにベクトルインデックス内で BM25 クオリティのキーワードマッチングを実現する手法です。Qdrant や Elasticsearch が特に成熟した実装を提供しています。Weaviate も BM25F を通じてサポートしています。ハイブリッド検索が必須で、二つのシステムを同時に走らせられない場合は、スパースベクトルの対応有無を確認してください。

**ディスクベースのインデックス**は実装上の詳細ではなく、コスト調整のレバーです。RAM 上に保持する HNSW インデックスは高速ですが、ベクトル数・次元数・メタデータ・グラフオーバーヘッドが増えるにつれてコストが高くなります。ディスクベースの代替手段（Milvus DiskANN、Elasticsearch DiskBBQ、Turbopuffer オブジェクトストレージ、LanceDB の Lance 形式）では、クエリ遅延とインフラコストのトレードオフが行われます。大規模な RAG インデックスの場合、このトレードオフはテストに値することが多いです。

**Turbopuffer のマルチテナンシー**は、非常に多数の名前空間を前提に設計されています。公開されているポジショニングや顧客事例は、Notion のように大規模で名前空間が多いコーパスを対象としたワークロードを強調しています。ユーザーや組織ごとに検索を分離する必要がある場合、このアーキテクチャはコスト構造を変える可能性がありますが、やはり自分のテナント構成でベンチマークを取るべきです。

**LanceDB の埋め込みモード**は「ベクトル検索用 SQLite」に最も近い存在です。プロセス内で動作し、サーバーは不要で、Lambda、Cloudflare Workers、エッジ環境でも利用できます。Lance のカラム指向フォーマットにより、実際のスケールでも埋め込み運用が実用的になります。

**Orama は検索 UX インフラであり、データウェアハウスではありません。** JavaScript アプリ内、エッジ、あるいはマネージドなサイト/アプリ検索層として、軽量な全文検索・ベクトル・ハイブリッド検索エンジンが欲しいときに優れています。ただし、十億規模のベクトル取得や重い分析、複雑なフィルタ付き結合が必要なケースには私が選ぶツールではありません。

**Vespa は、検索が製品の半分に過ぎない場合に選択するものです。** 語彙検索、最近傍検索、テンソル、ランク式、グルーピング、オンラインサービングをすべて統合しています。その性能は確かですが、運用やモデリングの複雑さも同様に大きいです。CRUD アプリに「セマンティック検索」を追加したいというケースよりも、検索・レコメンデーションチーム向きです。

**ClickHouse は、検索が分析と結びつくときに議論に入ります。** 真実の情報源がイベント、ログ、トレース、メトリクス、あるいは大規模なファクトテーブルである場合、ClickHouse はベクトル距離、フィルタリング、集計、そして現在は本格的な全文インデックスを 1 つの SQL エンジンで扱えます。専用のベクトルデータベースではありませんが、分析的な検索においては最も堅実な選択肢になり得ます。

**Chroma は、開発/テストや小規模アプリのデプロイに最も適しています。** 非常に大規模なコーパスや高可用性、ディスク中心の運用、あるいは本格的なハイブリッド検索が必要な場合は、プロトタイプをインフラに昇格させる前に、プロダクション向けのストアを評価すべきです。

### 簡易的な判断基準

実際に pgvector の限界に達した場合—ベンチマークでベクトル数、フィルタリング、書き込みレート、または高同時実行レイテンシが PostgreSQL の許容範囲を超えていると判断したとき—制約に基づいて選択します。

- **テナントごとに分離された SaaS 製品** → Turbopuffer  
- **Rust レベルの性能と複雑なメタデータフィルタが必要** → Qdrant  
- **既に Elastic/ELK スタックを使用中** → Elasticsearch + DiskBBQ  
- **オープンソース志向の AWS 環境** → OpenSearch  
- **高度なランキングが求められる検索/レコメンデーションプラットフォーム** → Vespa  
- **分析、可観測性、またはログ/イベント検索** → ClickHouse  
- **オンプレミス／セルフホストで十億規模** → Milvus  
- **エッジ／サーバーレス／マルチモーダル** → LanceDB  
- **小規模な JS アプリ、ドキュメントサイト、またはエッジネイティブ検索 UI** → Orama  
- **運用負荷ゼロ、動作すれば OK、コストは二の次** → Pinecone  
- **マルチモーダル優先（画像、動画、音声）** → Marqo  
- **既に MongoDB を使用中** → Atlas Vector Search  
- **既に Postgres を使用中で余裕が必要** → Supabase Vector または Neon（どちらも pgvector 管理版で、ツールチェーンが充実）

## やってはいけないこと

正解が決まっているテキストに対して、ベクトル検索をあいまい検索として使わないこと。

「メールアドレス `dan@example.com` を持つユーザーを探して」や「ID が `ORD-12345` の注文を探して」というクエリはベクトル検索の問題ではありません。`ORD-12345` を埋め込みに変換し、コサイン類似度で注文テーブルに対して検索しても何かは返りますが、正しくない可能性があります。これらは正確な一致が求められるケースです。主キーや通常のインデックスを使いましょう。

ベクトル検索はデータセット内で *最も類似した* 項目を返します。たとえ関連性のあるものが全く存在しなくても、何かしらの結果を出します。これは関連文書の検索には問題ありませんが、特定レコードの検索では致命的です。誤った近似一致が空結果よりもはるかに悪い結果を招くことがあります。

各ツールの用途を正しく理解してください。ほとんどはすでに Postgres にインストールされています。適材適所で活用しましょう。
