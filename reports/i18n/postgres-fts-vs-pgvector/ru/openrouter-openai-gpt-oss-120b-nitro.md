Команды, добавляющие функции ИИ, часто сразу берут в работу специализированную векторную базу данных.  

Pinecone, Weaviate, Qdrant, Chroma. Новый сервис, новая зависимость, новый пул соединений, новый счёт, и теперь два источника правды, которые нужно поддерживать в согласованном виде.  

При этом у них уже есть PostgreSQL. В PostgreSQL уже есть `pgvector`. Кроме того, с 2008 года в нём встроен отличный полнотекстовый поиск.

Выделенные векторные хранилища оправдывают себя только при действительно больших масштабах и высокой нагрузке запросов. Но большинство приложений берут вторую поисковую систему, пока первая ещё не была сильно нагружена. Так потенциальная проблема масштабирования превращается в текущий баг синхронизации.

Итак: когда использовать FTS, когда — pgvector, а когда — их вместе?

## Что делает каждый из них на самом деле

Полнотекстовый поиск (`tsvector` / индекс `GIN`) — лексический. Он разбивает текст на лексемы, приводит их к основе и сопоставляет запросы с индексом. «Running» и «runs» сводятся к одной лексеме. То же самое с «dog» и «dogs». Функция ранжирования (`ts_rank`) отдаёт предпочтение документам, где термины запроса встречаются часто или находятся в заметных позициях.

pgvector — семантический. Он хранит плотный вектор — список чисел, представляющих *значение* фрагмента согласно модели эмбеддингов. Поиск по схожести находит близкие векторы в этом высокоразмерном пространстве. «Dog» и «canine» оказываются рядом. «Running» в контексте спорта и «running» как процесс могут оказаться далеко друг от друга.

Практическая разница: FTS отвечает на вопрос «в каких документах встречаются эти слова?», а векторный поиск отвечает «какие документы примерно означают эту вещь?»

![Карта инструментов поиска, показывающая pg_trgm для коротких нечётких строк, полнотекстовый поиск для точных запросов в прозе, pgvector для семантического сопоставления и гибридный поиск для длинного контента, требующего как точных, так и семантических сигналов.](../search-tool-map.svg)

_Первое разделение — это не «старый поиск vs. AI‑поиск». Это форма текста и тип ответа, который будет корректным._

## Когда выигрывает полнотекстовый поиск

**Вы ищете точные термины.** SKU‑товаров, коды ошибок, номера моделей, имена пользователей, ссылки на юридические пункты. `SKU-AX-44192` семантически ничему не похож. Он либо совпадает, либо нет. Векторный поиск может уверенно вернуть `SKU-AX-44193`. Это не то, что вам нужно.

**Ваши запросы основаны на ключевых словах.** Пользователи вводят запрос в строку поиска, фильтруют по тегу или ищут записи блога по ключевому слову. FTS создан именно для такой формы намерения.

**Вам нужны ранжированные результаты без GPU или инфраструктуры эмбеддингов.** Индексы FTS быстры, детерминированы и не требуют внешних API‑вызовов. Добавьте колонку `tsvector`, построьте GIN‑индекс — и всё готово.

**Вы комбинируете булеву фильтрацию с поиском.** `WHERE to_tsvector(body) @@ to_tsquery('postgres') AND category = 'tutorial' AND published_at > NOW() - INTERVAL '6 months'` — естественно вписывается в вашу существующую логику запросов.

```sql
-- Create the index
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body, '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Query
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

Колонка `GENERATED ALWAYS AS` автоматически поддерживает актуальность индекса. Функция `setweight` присваивает более высокий вес совпадениям в заголовке по сравнению с содержимым тела. Это и есть вся настройка.

## Когда выигрывают триграммы (pg_trgm)

Есть третий инструмент PostgreSQL, который часто упускают из виду: `pg_trgm`. Это не полнотекстовый поиск и не векторный поиск. Это нечеткое сопоставление строк, которое покрывает неудобный промежуток, где оба остальных инструмента работают плохо.

**Сценарий использования: имена, адреса, идентификаторы и короткие строки с опечатками.**

FTS разбивает текст на лексемы и приводит их к основе. Это работает для прозы, но плохо подходит для:
- Имен людей («Dan Levy» → стемминг даст «dan levi», «leiv», в зависимости от языковой конфигурации)
- Названий компаний, адресов, названий продуктов, где важна точная орфография
- Запросов с опечатками — «Micheal Jordan», «Amaon», «javascipt»
- Автодополнения / поиска по префиксу
- Частичного совпадения строк («son» в «Johnson», «Anderson»)

pgvector также здесь плохой выбор. Можно заэмбедить «Micheal Jordan» и найти ближайший вектор, но пространство эмбеддингов упорядочивает имена по смыслу, а не по написанию. Ближайший сосед может оказаться «basketball legend» или «Michael B. Jordan», а не запись пользователя с опечаткой.

`pg_trgm` разбивает строки на перекрывающиеся 3‑символьные срезы и измеряет, сколько триграмм у двух строк совпадает. «Dan» → `" da"`, `"dan"`, `"an "`. «Micheal» и «Michael» имеют почти одинаковые триграммы, поэтому сходство высоко.

```sql
-- Enable the extension (usually already available)
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- GIN index on names column — enables fast trigram similarity search
CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- Fuzzy name search: finds "Micheal Jordan" when searching "Michael Jordan"
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % operator = similarity threshold (default 0.3)
ORDER BY score DESC
LIMIT 10;

-- Or use ILIKE with trigram index support for contains matching
SELECT id, name
FROM users
WHERE name ILIKE '%johnson%'   -- GIN index makes this fast
LIMIT 10;
```

Оператор `%` использует `pg_trgm.similarity_threshold` (по умолчанию 0.3, диапазон 0‑1). Более высокие значения требуют более близкого совпадения. Для поиска по именам обычно подходит 0.3‑0.4: достаточно permissive, чтобы ловить опечатки, но достаточно строгий, чтобы исключать шум.

**Триграммы также полезны для поиска по префиксу и автодополнения, особенно когда автодополнение требует толерантности к опечаткам или поиска совпадений:**

```sql
-- Autocomplete: prefix matching. For pure left-anchored prefixes,
-- compare trigram GIN against a B-tree pattern index on your data.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- More control: word_similarity for partial matches within longer strings
-- (useful when searching "Johnson" within "Andrew Johnson III")
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name          -- <% operator = word_similarity threshold
ORDER BY score DESC
LIMIT 10;
```

**Когда стоит выбирать `pg_trgm` вместо FTS:**

| Сценарий | Использовать |
|---|---|
| Поиск имени человека/компании с опечатками | `pg_trgm` |
| Автодополнение / поиск по префиксу | `pg_trgm` (или FTS с префиксными запросами) |
| Поиск коротких строк, кодов, идентификаторов | `pg_trgm` |
| Поиск по статьям, документации | FTS |
| Поиск ключевых слов в лог‑сообщениях | FTS |
| Многоязычный поиск имён | `pg_trgm` (не зависит от языка) |

`pg_trgm` также может работать совместно с FTS. Используйте триграммы как нечёткий предварительный фильтр и ранжируйте с помощью `ts_rank`, либо комбинируйте сходство триграмм со векторным скором.

## Когда выигрывает pgvector

**Вы строите RAG.** RAG опирается на семантическое извлечение: находить фрагменты документов, смысл которых максимально близок к вопросу пользователя, даже если формулировка отличается. Поиск по векторам создан именно для этого. FTS пропустит парафразы, синонимы и концептуальные совпадения.

**Пользователи описывают, чего хотят, а не как искать.** «Что‑то лёгкое для летнего вечера» не содержит очевидных винных ключевых слов. «Статьи о том, как построить уверенность в роли нового менеджера» требует семантического понимания, которое FTS не обеспечивает.

**Вы ищете похожие элементы.** Схожие товары, аналогичные заявки в службу поддержки, дублирующие баг‑репорты. «Найдите мне проблемы, похожие на эту» — это векторная операция. Вы встраиваете новую заявку в вектор и ищете её ближайших соседей.

**Многоязычное содержимое.** Векторные эмбеддинги, обученные на многоязычных данных, могут сопоставлять тексты между разными языками. FTS требует языко‑специфичных конфигураций и плохо справляется с кросс‑языковыми запросами.

## Когда нужны оба подхода

Техническая документация — это место, где простое деление рушится. Пользователи ищут «how to configure timeouts», но также ищут имена функций вроде `withRetry()` и коды ошибок вроде `ECONNRESET`.

Векторный поиск обрабатывает концептуальные запросы. FTS — точные термины. Ни один из них не справляется с обеими задачами в одиночку.

Решение — гибридный поиск: выполнить оба запроса и объединить результаты.

**Reciprocal Rank Fusion (RRF)** — стандартный алгоритм для этой задачи. Он не требует нормализации оценок из двух систем; объединяет позиции в ранжировании.

```sql
-- Hybrid search with Reciprocal Rank Fusion
WITH fts_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM documents, to_tsquery('english', $1) query
  WHERE search_vector @@ query
  LIMIT 50
),
vector_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> $2::vector) AS rank
  FROM documents
  ORDER BY embedding <=> $2::vector
  LIMIT 50
),
rrf AS (
  SELECT
    COALESCE(f.id, v.id) AS id,
    COALESCE(1.0 / (60 + f.rank), 0) + COALESCE(1.0 / (60 + v.rank), 0) AS rrf_score
  FROM fts_results f
  FULL OUTER JOIN vector_results v ON f.id = v.id
)
SELECT d.id, d.title, rrf.rrf_score
FROM rrf
JOIN documents d ON d.id = rrf.id
ORDER BY rrf_score DESC
LIMIT 10;
```

Число `60` в знаменателе — константа RRF: большие значения снижают влияние различий в рангах, меньшие — усиливают их. Значение по умолчанию — 60, обычно работает достаточно хорошо.

Это выполняет два поиска в одном запросе, объединяет ранги и отдаёт предпочтение результатам, где согласуются ключевые слова и семантические сигналы.

![Гибридный конвейер поиска, где один запрос распыляется на полнотекстовый поиск и pgvector, каждый из которых выдаёт ранжированные результаты, а Reciprocal Rank Fusion объединяет два списка.](../hybrid-rrf-pipeline.svg)

_RRF полезен, потому что избавляет от необходимости приравнивать `ts_rank` и косинусное расстояние как сравнимые сырые оценки. Он задаёт лишь вопрос: «на какой позиции появился этот результат в каждом списке?»_

## Практическое дерево решений  

При выборе стратегии поиска сначала ориентируйтесь на **форму входных данных**, затем уточняйте **какой запрос делает пользователь**. «Короткая строка с орфографическими вариациями» — это не то же самое, что «длинный текст, где важны точные термины», и оба случая отличаются от «вопроса к фрагментам документа».

![Дерево решений, выбирающее pg_trgm, полнотекстовый поиск, pgvector, гибридный поиск или специализированную векторную базу данных в зависимости от формы текста и способа запросов пользователей.](../search-decision-tree.svg)

То же самое дерево словами:

- **Имена, адреса, названия, автодополнение или короткие строки, склонные к опечаткам** → `pg_trgm`
- **Известные слова, коды ошибок, SKU, имена функций, теги, категории, фильтры** → FTS
- **Вопросы, перефразировки, рекомендации, связанные элементы, многоязычное сопоставление, фрагменты RAG** → pgvector
- **Техническое содержание, где пользователям нужны и точные символы, и концептуальные ответы** → гибридный поиск с RRF
- **Первичные ключи, точные идентификаторы, фильтры прав, диапазоны дат, отсортированные списки** → обычные индексы SQL
- **Огромный объём векторов, очень высокая конкурентность или требования к задержке, которые Postgres не может удовлетворить в ваших тестах** → оценить специализированные векторные хранилища

### FTS vs. Semantic: The Short Version

Вопрос «использовать FTS или векторный поиск?» обычно сводится к следующему: **знаете ли вы, какие слова появятся в релевантных документах?**

Если да — пользователи ищут известные термины, категории, имена функций, коды продуктов — FTS быстрее, дешевле и предсказуемее. Он показывает, почему результат совпал.

Если нет — пользователи описывают концепцию, задают вопрос или ищут на другом языке — векторный поиск будет правильным инструментом. Он сопоставляет смысл, а не отдельные слова.

Сложный промежуточный случай — запросы на естественном языке к техническому контенту. Кто‑то ищет «how do I handle connection drops», а нужная статья называется «implementing retry logic for network failures» — слов нет в пересечении, но высокая семантическая релевантность. Именно здесь векторный поиск оправдывает себя.

Другой сложный случай — **имена и собственные существительные**. Ни FTS, ни векторный поиск с ними не справляются:
- FTS пропустит «Micheal», когда ищут «Michael» — разные токены
- Векторный поиск полностью упустит имя, если оно редко встречается в обучающих данных
- `pg_trgm` решает задачу корректно: орфографическое сходство, а не семантическое или лексическое.

На практике большинство поисковых полей, насыщенных контентом, используют FTS для скорости и ключевых слов, а при необходимости гибридного поиска или `pg_trgm` — в зависимости от того, ищут ли пользователи имена. По‑настоящему семантическая функция поиска обычно подразумевает pgvector. RAG всегда требует pgvector.

## Если нужен отдельный векторный магазин


Некоторые системы действительно перерастают pgvector. Когда это происходит, рынок становится шумным. Вот что имеет значение среди ведущих вариантов.

### Матрица функций

Некоторые столбцы требуют разъяснения, прежде чем таблица станет понятной.

**Гибридный поиск** означает, что BM25‑поиск по ключевым словам и векторное сходство выполняются в одном запросе, объединённые через Reciprocal Rank Fusion. «withRetry timeout» может точно совпасть с именем функции *и* семантически находить документы о «логике повторных попыток при сетевых сбоях». Без гибрида вы выбираете один режим поиска или сами объединяете два запроса. «Manual (RRF via SQL)» в pgvector — это [подход, показанный выше](#when-you-need-both): он работает, но вы пишете его вручную.

**Разреженные векторы** идут дальше BM25. Разреженный вектор SPLADE имеет ~30 000 измерений (по одному на каждый термин словаря), ~98 % нулей. Ненулевые позиции показывают, какие термины важны и насколько. Запрос «dogs» также учитывает «canine» и «pet»: точность ключевых слов уровня BM25 плюс расширение терминов внутри векторного индекса. Если этот столбец ложен, понадобится внешняя FTS‑слой для точных терминных запросов.

```python
# SPLADE: ~30,000 dims total, ~60 non-zero — only the relevant vocabulary positions fire
def encode_splade(text: str) -> dict:
    tokens = tokenizer(text, return_tensors="pt", truncation=True, max_length=512)
    with torch.no_grad():
        output = model(**tokens)
    vec = torch.log1p(torch.relu(output.logits)).max(dim=1).values.squeeze()
    return {"indices": vec.nonzero().squeeze().tolist(), "values": vec[vec != 0].tolist()}
```

**SQL / SQL‑подобный** на самом деле касается фильтрации. Векторный поиск без фильтрации — это демонстрация, а не приложение: всё равно нужны ограничения по арендаторам, диапазонам дат, правам доступа и категориям. Полный SQL (pgvector) позволяет выразить это рядом с вашими существующими JOIN‑ами. Специализированные БД используют JSON‑объекты фильтров (Qdrant, Pinecone), DSL запросов (Elasticsearch, Milvus) или GraphQL (Weaviate). Они работают; SQL становится более привлекательным, когда логика фильтрации запутывается.

```sql
-- pgvector: векторное сходство — это просто ещё одно выражение в WHERE

FROM documents
WHERE tenant_id = $2 AND category = ANY($3::text[]) AND created_at > NOW() - INTERVAL '90 days'
ORDER BY embedding <=> $1 LIMIT 10;
```

**Multimodal native** не означает «может хранить эмбеддинги изображений»; любой движок хранит массивы чисел с плавающей точкой. Это значит, что база поставляется с моделями эмбеддинга для нелинейного контента, так что вы передаёте ей прямой URL изображения, а она сама векторизует его. Большинство рассматриваемых СУБД нейтральны к типу эмбеддингов, поэтому построение конвейера остаётся на вашей стороне. Marqo и Weaviate (через модули CLIP/ImageBind) закрывают этот цикл.

```python
# Marqo: POST необработанные изображения, запросить текстом — внешний шаг эмбеддинга не нужен
mq.index("products").add_documents(
    [{"id": "shoe-001", "image": "https://cdn.example.com/shoes/001.jpg"}],
    tensor_fields=["image"]
)
results = mq.index("products").search(q="lightweight shoes for summer")
# Возвращает shoe-001, несмотря на отсутствие совпадения по ключевым словам — CLIP решает кросс‑модальное сопоставление
```

**Disk-based index** — вопрос стоимости. HNSW в RAM может потреблять несколько гигабайт оперативной памяти на миллион векторов размером 1536 измерений, если учитывать сами векторы, графовую надстройку и метаданные. Дисковые варианты (Milvus DiskANN, Elasticsearch DiskBBQ, формат Lance в LanceDB, объектный слой хранения Turbopuffer) часто жертвуют небольшим увеличением задержки запросов ради более дешёвой инфраструктуры. Для RAG‑нагрузок, где задержка модели уже доминирует, такой компромисс обычно оправдан и требует бенчмаркинга. Redis VSS — жёсткое ограничение: только RAM, без дискового пути.

**Max dimensions** — завтрашняя миграция, скрытая в сегодняшнем выборе. `text-embedding-3-large` использует 3072 измерения, Jina v3 может выдавать более крупные эмбеддинги, а исследовательские модели продолжают расти. Некоторые управляемые сервисы публикуют жёсткие лимиты на размерность; другие указывают лишь «высокие» или «отсутствие практического ограничения» для типовых моделей. Проверьте актуальную документацию перед тем, как фиксировать решение. Выбирайте с запасом; миграция векторного индекса из‑за достижения потолка размерности превращается в мучительный спринт.

### Как читать матрицу

Несколько пунктов не укладываются в таблицу:

**Разреженные векторы** позволяют получать BM25‑качество сопоставления ключевых слов внутри векторного индекса, без отдельного полнотекстового движка. Qdrant и Elasticsearch имеют особенно зрелые реализации. Weaviate поддерживает их через BM25F. Если гибридный поиск критичен и нельзя запускать две системы, ищите поддержку разреженных векторов.

**Индексы на диске** — это рычаг стоимости, а не деталь реализации. HNSW‑индексы, хранящиеся в RAM, быстры, но могут стать дорогими по мере роста количества векторов, размерности, метаданных и накладных расходов графа. Дисковые альтернативы (Milvus DiskANN, Elasticsearch DiskBBQ, объектное хранилище Turbopuffer, формат Lance от LanceDB) меняют задержку запросов на более низкую инфраструктурную стоимость. Для больших RAG‑индексов такой компромисс часто стоит проверить.

**Мультиаренда Turbopuffer** построена вокруг огромного количества пространств имён. В публичных позиционировании и историях клиентов делается акцент на нагрузки вроде крупного, насыщенного пространствами имён корпуса Notion. Если каждому пользователю или организации требуется изолированный векторный поиск, такая архитектура может изменить экономику, но всё равно измеряйте собственную форму арендатора.

**Встроенный режим LanceDB** — самое близкое к «SQLite для векторного поиска». Он работает в процессе, не требует сервера и подходит для Lambda, Cloudflare Workers и edge‑окружений. Колоночный формат Lance делает встроенную работу практичной даже при реальном масштабе.

**Orama — инфраструктура UX‑поиска, а не склад данных.** Она превосходна, когда нужен крошечный полнотекстовый/векторный/гибридный поисковый движок в JavaScript‑приложении, на edge или как управляемый слой поиска сайта/приложения. Это не тот инструмент, который я бы выбрал для поиска среди миллиарда векторов, тяжёлой аналитики или сложных фильтруемых соединений.

**Vespa — это то, к чему вы прибегаете, когда поиск составляет лишь половину продукта.** Она объединяет лексический поиск, поиск ближайших соседей, тензоры, ранжирующие выражения, группировку и онлайн‑обслуживание. Эта мощь реальна, но также реальны и операционная, и модельная сложность. Подходит командам, занимающимся поиском/рекомендациями, а не тем, кто просто «добавляет семантический поиск в своё CRUD‑приложение».

**ClickHouse имеет смысл упоминать, когда поиск тесно связан с аналитикой.** Если ваш источник правды — это события, логи, трассировки, метрики или огромные факт‑таблицы, ClickHouse может хранить векторные расстояния, фильтрацию, агрегацию и теперь серьёзный полнотекстовый индекс в одном SQL‑движке. Это не специализированная векторная БД, но для аналитического извлечения она может стать скучным, но надёжным решением.

**Chroma сильна в разработке/тестировании и небольших развертываниях приложений.** Если вы планируете работать с очень большими корпусами, высокой доступностью, дискоёмкой или полноценным гибридным поиском, оцените продакшн‑ориентированное хранилище, прежде чем продвигать прототип в инфраструктуру.

### Упрощённый выбор

Если вы действительно переросли pgvector — обычно потому, что бенчмарки показывают, что количество векторов, фильтрация, скорость записи или задержка при высокой конкуренции выходят за пределы возможностей вашего Postgres — выбирайте по ограничению:

- **SaaS‑продукт с изоляцией арендаторов** → Turbopuffer  
- **Требуется производительность уровня Rust + сложная фильтрация метаданных** → Qdrant  
- **Уже используете стек Elastic/ELK** → Elasticsearch с DiskBBQ  
- **AWS‑ориентированная инфраструктура, желающая открытый код** → OpenSearch  
- **Платформа поиска/рекомендаций с серьёзными требованиями к ранжированию** → Vespa  
- **Аналитика, наблюдаемость или поиск по логам/событиям** → ClickHouse  
- **Масштаб в миллиарды записей on‑prem / self‑hosted** → Milvus  
- **Edge / serverless / мультимодальный** → LanceDB  
- **Небольшое JS‑приложение, сайт документации или edge‑нативный поисковый UI** → Orama  
- **Zero‑ops, просто работает, стоимость вторична** → Pinecone  
- **Мультимодальный в первую очередь (изображения, видео, аудио)** → Marqo  
- **Уже на MongoDB** → Atlas Vector Search  
- **Уже на Postgres, нужен дополнительный запас ресурсов** → Supabase Vector или Neon (оба управляют pgvector, с лучшими инструментами)

## Одна вещь, которую нельзя делать

Не используйте векторный поиск как нечеткий текстовый поиск для задач, где есть однозначный правильный ответ.

«Найдите пользователя с email `dan@example.com`» — это не задача векторного поиска. То же самое относится к запросу «Найдите заказ с ID `ORD-12345`». Встраивание `ORD-12345` и вычисление косинусного сходства по таблице заказов вернёт *что‑то*, но это может быть неверно. Это задачи точного совпадения. Используйте первичный ключ или обычный индекс.

Векторный поиск возвращает *наиболее похожий* элемент в наборе данных, даже если ничего релевантного нет. Он не умеет определить, что хорошего ответа нет. Для связанных документов это приемлемо, но для поиска конкретной записи катастрофически: неверное приближённое совпадение хуже, чем пустой результат.

Знайте, для чего предназначен каждый инструмент. Большинство из них уже присутствует в вашей установке Postgres. Применяйте их там, где они действительно подходят.
