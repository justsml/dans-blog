## जब पूर्ण-पाठ खोज जीतती है

**आप ठीक ठीक शब्दों की तलाश में हैं।** उत्पाद एसकेयू, त्रुटि कोड, मॉडल संख्या, उपयोगकर्ता नाम, कानूनी धारा संदर्भ। `SKU-AX-44192` कुछ भी अर्थात्मक रूप से समान नहीं है। या तो यह मेल खाता है या नहीं। वैक्टर खोज आश्वस्त रूप से `SKU-AX-44193` लौटा सकती है। यह आपकी आवश्यकता नहीं है।

**आपके प्रश्न शब्द-आधारित हैं।** उपयोगकर्ता एक खोज बॉक्स में टाइप करते हैं, टैग द्वारा फ़िल्टर करते हैं, या शब्दांकन द्वारा ब्लॉग पोस्ट की खोज करते हैं। एफटीएस के लिए इस प्रकार का इरादा डिज़ाइन किया गया है।

**आपको रैंकिंग परिणामों की आवश्यकता है बिना जीपीयू या एम्बेडिंग बुनियादी ढांचे के।** एफटीएस इंडेक्स त्वरित, निर्धारणात्मक, और कोई बाहरी एपीआई कॉल नहीं चाहिए। एक `tsvector` स्तंभ जोड़ें, एक GIN इंडेक्स बनाएं, और आप तैयार हैं।

**आप खोज के साथ बूलियन फ़िल्टरिंग कर रहे हैं।** `WHERE to_tsvector(body) @@ to_tsquery('postgres') AND category = 'tutorial' AND published_at > NOW() - INTERVAL '6 months'` — यह आपके मौजूदा प्रश्न तर्क के साथ प्राकृतिक रूप से संगठित होता है।

```sql
-- Create the index
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body, '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Query
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`GENERATED ALWAYS AS` स्तंभ इंडेक्स को स्वचालित रूप से अपडेट रखता है। `setweight` शीर्षक के मिलान को शरीर के मिलान की तुलना में अधिक रैंक देता है। यह पूरी स्थापना है।

## जब ट्राइग्राम्स जीतते हैं (pg_trgm)

`pg_trgm` एक तीसरा पोस्टग्रेस उपकरण है जिसे अक्सर ओवरलॉक कर दिया जाता है। यह न तो पूर्ण-पाठ खोज है और न ही वेक्टर खोज। यह फेंजी स्ट्रिंग मैचिंग है, और यह दोनों उपकरणों द्वारा खराब ढंग से संभाले गए कठिन मध्य भाग को कवर करता है।

**उपयोग का मामला: नाम, पते, पहचानकर्ता, और अक्षरों में त्रुटियाँ वाले छोटे स्ट्रिंग्स।**

FTS पाठ को लेक्समेंस में टोकनाइज़ करता है और उन्हें स्टेम करता है। यह प्रोज़ के लिए काम करता है, लेकिन निम्नलिखित मामलों में खराब फिट होता है:  
- व्यक्ति के नाम ("डैन लेवी" → स्टेम करके "डैन लेवी", "लेवी", भाषा कॉन्फ़िगरेशन पर निर्भर करता है)  
- कंपनी के नाम, पते, उत्पाद शीर्षक जहाँ ठीक वर्तनी महत्वपूर्ण है  
- त्रुटियों वाले प्रश्न - "मिचेल जॉर्डन", "अमान", "जावास्क्रिप्ट"  
- स्वचालित पूरा करें / प्रिफिक्स खोज  
- आंशिक स्ट्रिंग मैचिंग ("सन" मैच करता है "जॉनसन", "एंडरसन")

यहां पर `pgvector` भी एक खराब चयन है। आप "मिचेल जॉर्डन" को एम्बेड कर सकते हैं और निकटतम वेक्टर ढूंढ सकते हैं, लेकिन एम्बेडिंग स्पेस नामों को अर्थ के आधार पर व्यवस्थित करता है, वर्तनी के आधार पर नहीं। निकटतम पड़ोसी "बास्केटबॉल लेजेंड" या "माइकल बी. जॉर्डन" हो सकता है, न कि उपयोगकर्ता रिकॉर्ड जिसमें त्रुटि है।

`pg_trgm` स्ट्रिंग्स को ओवरलैपिंग 3-चरक स्लाइस में तोड़ता है और दो स्ट्रिंग्स के बीच कितने ट्राइग्राम साझा किए जाते हैं, उसका मापन करता है। "डैन" -> `" da"`, `"dan"`, `"an "`. "मिचेल" और "माइकल" अपने ट्राइग्राम के अधिकांश भाग शेयर करते हैं, इसलिए समानता उच्च है।

```sql
-- एक्सटेंशन को सक्षम करें (आमतौर पर पहले से उपलब्ध होता है)
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- नाम कॉलम पर GIN इंडेक्स — तेज ट्राइग्राम समानता खोज को सक्षम करता है
CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- नाम की गैर-सटीक खोज: "मिचेल जॉर्डन" को खोजते समय "माइकल जॉर्डन" ढूंढेगा
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % ऑपरेटर = समानता थ्रेशोल्ड (डिफ़ॉल्ट 0.3)
ORDER BY score DESC
LIMIT 10;

-- या शामिल मिलान के लिए trigram इंडेक्स समर्थन के साथ ILIKE का उपयोग करें
SELECT id, name
FROM users
WHERE name ILIKE '%johnson%'   -- GIN इंडेक्स इसे तेज करता है
LIMIT 10;
```

`%` ऑपरेटर `pg_trgm.similarity_threshold` (डिफ़ॉल्ट 0.3, रेंज 0-1) का उपयोग करता है। उच्च मान निकटता के मैच की आवश्यकता होती है। नाम खोज के लिए 0.3-0.4 आमतौर पर सही होता है: त्रुटियों को पकड़ने के लिए पर्याप्त आवाहक, शोर से बचने के लिए पर्याप्त सख्त।

**ट्राइग्राम्स अटूट खोज और स्वचालित पूर्णता में भी मदद करते हैं, खासकर जब स्वचालित पूर्णता में त्रुटि सहिष्णुता या शामिल मैचिंग की आवश्यकता होती है:**  

```sql
-- स्वचालित पूर्णता: पूर्वानुमान मैचिंग। शुद्ध बाएं-आधारित पूर्वानुमान के लिए,  
-- अपने डेटा पर ट्राइग्राम GIN की तुलना B-tree पैटर्न इंडेक्स से करें।
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- अधिक नियंत्रण: लंबे स्ट्रिंग्स में आंशिक मैचिंग के लिए word_similarity
-- ("Andrew Johnson III" में "Johnson" खोजने के लिए उपयोगी)
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name          -- <% ऑपरेटर = word_similarity थ्रेशोल्ड
ORDER BY score DESC
LIMIT 10;
```

**जब `pg_trgm` का उपयोग FTS के बजाय करें:**  

| परिदृश्य | उपयोग |
|---|---|
| व्यक्ति/कंपनी नाम खोज (त्रुटियों के साथ) | `pg_trgm` |
| स्वचालित पूर्णता / पूर्वानुमान खोज | `pg_trgm` (या पूर्वानुमान प्रश्नों के साथ FTS) |
| छोटे स्ट्रिंग्स, कोड, पहचानकर्ता खोज | `pg_trgm` |
| निबंध लेखों/दस्तावेजों की खोज | FTS |
| लॉग संदेशों में कीवर्ड खोज | FTS |
| बहुभाषिक नाम खोज | `pg_trgm` (यह भाषा-अनजान है) |

`pg_trgm` FTS के साथ भी संयोजित होता है। त्रिग्राम का उपयोग फजी पूर्व फ़िल्टर के रूप में करें और `ts_rank` के साथ रैंक करें, या त्रिग्राम समानता को वेक्टर स्कोर के साथ जोड़ें।

## जब pgvector जीतता है

**आप RAG बना रहे हैं।** RAG अर्थात्मक पुनर्प्राप्ति पर निर्भर करता है: उपयोगकर्ता के प्रश्न के अर्थ के सबसे निकटतम दस्तावेज़ के खंडों को खोजें, भले ही शब्द भिन्न हों। वेक्टर खोज इसके लिए विशेष रूप से डिज़ाइन की गई है। FTS पुनर्व्याख्यान, समानार्थी शब्द और अवधारणात्मक मेल को छूट जाएगा।

**उपयोगकर्ता अपनी आवश्यकता के बारे में वर्णन करते हैं, न कि खोजने के लिए क्या है।** "गर्मी की रात के लिए कुछ हल्का" में स्पष्ट शराब के शब्द नहीं हैं। "नए प्रबंधक के रूप में आत्मविश्वास बनाने वाले लेख" के लिए अर्थात्मक समझ आवश्यक है जो FTS प्रदान नहीं कर सकता।

**आप समान वस्तुओं को खोज रहे हैं।** संबंधित उत्पाद, समान समर्थन टिकट, डुप्लिकेट बग रिपोर्ट। "इस तरह की समस्याओं के समान मुद्दों को खोजें" एक वेक्टर संचालन है। आप नई समस्या को एम्बेड करते हैं और उसके निकटतम पड़ोसी खोजते हैं।

**बहुभाषी सामग्री।** बहुभाषी डेटा पर प्रशिक्षित वेक्टर एम्बेडिंग भाषाओं के मध्य मेल खाते हैं। एफटीएस भाषा-विशिष्ट निर्माणों की आवश्यकता होती है और अन्तर-भाषा प्रश्नों को खराब ढंग से संभालता है।

```sql
-- स्थापना
CREATE EXTENSION IF NOT EXISTS vector;

ALTER TABLE documents ADD COLUMN embedding vector(1536);
CREATE INDEX documents_embedding_idx
  ON documents USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);

-- प्रश्न: अर्थात्मक खोज
SELECT id, title, 1 - (embedding <=> $1::vector) AS similarity
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 10;
```

टिप्पणी: `ivfflat` अनुमान है — यह तेज़ है लेकिन गति के बदले कुछ याद दिलाने की बलि देता है। छोटे डेटासेट (लगभग 1 मिलियन पंक्तियों के नीचे) के लिए, `hnsw` अक्सर बेहतर होता है:

```sql
CREATE INDEX documents_embedding_idx
  ON documents USING hnsw (embedding vector_cosine_ops);
```

## जब आपको दोनों की आवश्यकता होती है

तकनीकी दस्तावेज़न वह जगह है जहां सरल विभाजन टूट जाता है। उपयोगकर्ता "टाइमआउट कॉन्फ़िगर करने के तरीके" की खोज करते हैं, लेकिन वे फ़ंक्शन नामों जैसे `withRetry()` और त्रुटि कोडों जैसे `ECONNRESET` की भी खोज करते हैं।

वैक्टर खोज अवधारणात्मक प्रश्नों का संचालन करती है। FTS ठीक शब्दों का संचालन करती है। कोई भी एकल रूप से दोनों को अच्छी तरह से संचालित नहीं करता है।

हल हाइब्रिड सर्च है: दोनों को चलाएं और परिणामों को जोड़ें।

**पारस्परिक रैंक फ्यूजन (Reciprocal Rank Fusion - RRF)** यहां मानक एल्गोरिथ्म है। यह आपको दोनों प्रणालियों से स्कोर को सामान्यीकृत करने की आवश्यकता नहीं होती; यह रैंक स्थिति को जोड़ता है।

```sql
-- Hybrid search with Reciprocal Rank Fusion
WITH fts_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM documents, to_tsquery('english', $1) query
  WHERE search_vector @@ query
  LIMIT 50
),
vector_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> $2::vector) AS rank
  FROM documents
  ORDER BY embedding <=> $2::vector
  LIMIT 50
),
rrf AS (
  SELECT
    COALESCE(f.id, v.id) AS id,
    COALESCE(1.0 / (60 + f.rank), 0) + COALESCE(1.0 / (60 + v.rank), 0) AS rrf_score
  FROM fts_results f
  FULL OUTER JOIN vector_results v ON f.id = v.id
)
SELECT d.id, d.title, rrf.rrf_score
FROM rrf
JOIN documents d ON d.id = rrf.id
ORDER BY rrf_score DESC
LIMIT 10;
```

अंश में `60` RRF स्थिरांक है — उच्च मान रैंक अंतर के प्रभाव को कम करता है, कम मान उन्हें बढ़ाता है। अधिकांश मामलों में 60 का डिफ़ॉल्ट मान अच्छा काम करता है।

यह एक प्रश्न में दो खोजें चलाता है, रैंकों को मिश्रित करता है, और उन परिणामों को प्रोत्साहित करता है जहां शब्दकोश और अर्थात्मक संकेत सहमत होते हैं।

![एक मिश्रित खोज पाइपलाइन जहां एक प्रश्न पूर्ण-पाठ खोज और pgvector में विस्तारित होता है, प्रत्येक रैंकित परिणाम उत्पन्न करता है, और पारस्परिक रैंक फ्यूजन दोनों सूचियों को मिश्रित करता है।](../hybrid-rrf-pipeline.svg)

_पीआरएफ की मूल्यवान बात इसके द्वारा यह नहीं मान लेना है कि `ts_rank` और कोसाइन दूरी तुलनीय कच्चे अंक हैं। यह केवल पूछता है, "प्रत्येक सूची में यह परिणाम कितना ऊपर दिखाई दिया?"_

## व्यावहारिक निर्णय वृक्ष

जब खोज रणनीति का चयन कर रहे हों, तो **इनपुट के आकार** से शुरू करें, फिर **उपयोगकर्ता किस प्रकार की प्रश्न बना रहा है** इसे पूछें। "लघु स्ट्रिंग जिसमें वर्तनी परिवर्तन है" समस्या "लंबा प्रोज़ जहां ठीक-ठीक शब्द महत्वपूर्ण हैं" से अलग है, और दोनों "दस्तावेज़ खंडों पर प्रश्न" से भिन्न हैं।

![एक निर्णय वृक्ष जो पाठ के आकार और उपयोगकर्ता के प्रश्न के आधार पर pg_trgm, फुल-टेक्स्ट सर्च, pgvector, हाइब्रिड सर्च या विशिष्ट सदिश डेटाबेस का चयन करता है।](../search-decision-tree.svg)

शब्दों में वही वृक्ष:

- **नाम, पते, शीर्षक, स्वतः पूर्ण, या वर्तनी त्रुटि वाली लघु स्ट्रिंग** → `pg_trgm`  
- **पहचाने गए शब्द, त्रुटि कोड, SKUs, फ़ंक्शन नाम, टैग, श्रेणियाँ, फ़िल्टर** → FTS  
- **प्रश्न, पुनर्व्याख्यान, सुझाव, संबंधित वस्तुएँ, बहुभाषिक मिलान, RAG खंड** → pgvector  
- **तकनीकी सामग्रि जहाँ उपयोगकर्ता को ठीक-ठीक प्रतीक और अवधारणात्मक उत्तर दोनों की आवश्यकता हो** → RRF के साथ हाइब्रिड  
- **मुख्य कुंजियाँ, ठीक-ठीक ID, अनुमति फ़िल्टर, तारीख श्रेणियाँ, क्रमबद्ध सूचियाँ** → सामान्य SQL इंडेक्स  
- **बड़ी मात्रा में सदिश, बहुत अधिक समानांतरता, या लेटेंसी लक्ष्य जो आपके बेंचमार्क में Postgres प्राप्त नहीं कर सकता** → विशिष्ट सदिश संग्रहों का मूल्यांकन करें

### FTS बनाम सेमेंटिक: संक्षिप्त संस्करण  

"क्या मैं FTS या वेक्टर सर्च का उपयोग करूँ?" यह प्रश्न आमतौर पर इस बात पर संकुचित हो जाता है: **क्या आप जानते हैं कि प्रासंगिक दस्तावेज़ में कौन-से शब्द दिखाई देंगे?**  

यदि हाँ - उपयोगकर्ता ज्ञात पद, श्रेणियाँ, फ़ंक्शन नाम, उत्पाद कोड आदि की खोज कर रहे हैं - तो FTS तेज़, सस्ता और अधिक भविष्यसंगत होता है। यह आपको बताता है कि क्यों एक परिणाम मेल खाता है।

अगर नहीं - उपयोगकर्ता एक अवधारणा का वर्णन कर रहे हैं, एक प्रश्न पूछ रहे हैं या एक अलग भाषा में खोज रहे हैं - वेक्टर सर्च सही उपकरण है। यह अर्थ के मेल की तलाश करता है, शब्दों के नहीं।  

प्राकृतिक भाषा प्रश्न वाले तकनीकी सामग्री पर एक झेल वाला मामला है। कोई व्यक्ति "मैं नेटवर्क विफलताओं के लिए रीट्राय लॉजिक कैसे लागू करूँ?" की खोज कर रहा है, जिसे एक लेख की आवश्यकता हो सकती है जिसका शीर्षक "नेटवर्क विफलताओं के लिए रीट्राय लॉजिक का लागू करना" है - कोई भी शब्द ओवरलैप नहीं करते, लेकिन उच्च सेमेंटिक प्रासंगिकता। यहीं वेक्टर सर्च अपना जीवन जीत लेता है।  

दूसरा झेल वाला मामला **नाम और संज्ञा** है। न तो FTS और न ही वेक्टर सर्च इन्हें अच्छी तरह से संभालते हैं:  
- FTS "माइकल" खोजने पर "मिचेल" को छूट जाएगा - अलग टोकन  
- वेक्टर सर्च पूरी तरह से नाम को छूट जाएगा यदि यह प्रशिक्षण डेटा में अक्सर नहीं दिखाई देता  
- `pg_trgm` इसे सही ढंग से संभालता है: लिप्यन्तर समानता, सेमेंटिक या व्याकरणिक नहीं

अभ्यास में, अधिकांश सामग्री-भारी खोज बॉक्स के लिए FTS की आवश्यकता होती है गति और कीवर्ड्स के लिए, और यह मिश्रित या `pg_trgm` की आवश्यकता हो सकती है यदि उपयोगकर्ता नाम खोज रहे हैं। एक असली सेमेंटिक सर्च फीचर आमतौर पर pgvector का अर्थ होता है। RAG हमेशा pgvector का अर्थ होता है।

## यदि आपको विशेष वेक्टर स्टोर की आवश्यकता है

कुछ प्रणालियाँ वास्तव में pgvector से बाहर निकल जाती हैं। जब ऐसा होता है, तो बाजार शोरभरा होता है। शीर्ष विकल्पों में यही महत्वपूर्ण है।

### विशेषता मैट्रिक्स

तालिका के अर्थ आने से पहले कुछ स्तंभों को समझने की आवश्यकता है।

**मिश्रित खोज** का अर्थ है BM25 कीवर्ड खोज और वेक्टर समानता एक प्रश्न में चलाई जाती है, जिसे रिसीप्रोकल रैंक फ्यूजन (Reciprocal Rank Fusion) के माध्यम से जोड़ा जाता है। "withRetry timeout" फ़ंक्शन नाम के साथ पूर्ण रूप से मेल खाता है *और* "नेटवर्क विफलताओं के लिए रीट्राई लॉजिक" के बारे में दस्तावेज़ों को अर्थात्मक रूप से जोड़ देता है। मिश्रित नहीं होने पर, आप एक खोज मोड चुनते हैं या खुद दो प्रश्नों को जोड़ते हैं। pgvector का "मैनुअल (RRF वियोग SQL के माध्यम से)" [ऊपर दिखाए गए तरीके](#when-you-need-both) है: यह काम करता है, लेकिन आप खुद इसे लिखते हैं।

**विरल वेक्टर्स** BM25 से आगे बढ़ते हैं। एक SPLADE विरल वेक्टर में ~30,000 विमान (एक शब्दकोश शब्द प्रति) होते हैं, ~98% शून्य। शून्य नहीं होने वाली स्थितियाँ आपको बताती हैं कि कौन से शब्द महत्वपूर्ण हैं और कितने। "कुत्ते" के लिए एक प्रश्न भी "कुत्ता" और "पालतू जानवर" को भारित करता है: BM25 स्तर की कीवर्ड सटीकता और शब्द विस्तार एक वेक्टर अनुक्रमण में। यदि यह स्तंभ असत्य है, तो आपको एक्सैक्ट-टर्म प्रश्नों के लिए बाहरी FTS परत की आवश्यकता होती है।

```python
# SPLADE: ~30,000 विमान कुल, ~60 शून्य नहीं — केवल प्रासंगिक शब्दकोश स्थितियाँ फायर होती हैं
def encode_splade(text: str) -> dict:
    tokens = tokenizer(text, return_tensors="pt", truncation=True, max_length=512)
    with torch.no_grad():
        output = model(**tokens)
    vec = torch.log1p(torch.relu(output.logits)).max(dim=1).values.squeeze()
    return {"indices": vec.nonzero().squeeze().tolist(), "values": vec[vec != 0].tolist()}
```

**एसक्यूएल / एसक्यूएल-जैसा** वास्तव में फ़िल्टरिंग के बारे में है। फ़िल्टरिंग के बिना वेक्टर खोज एक प्रदर्शन है, न कि एक अनुप्रयोग: आपको अभी भी ठाउक की दृष्टि, तिथि अवधि, अनुमति, और श्रेणियों की आवश्यकता होती है। पूर्ण एसक्यूएल (pgvector) अपने मौजूदा जॉइन के साथ इसे व्यक्त करता है। उद्देश्य-निर्मित डीबी जॉन्स (Qdrant, Pinecone) के जावा फ़िल्टर ऑब्जेक्ट, एक प्रश्न डीएसएल (Elasticsearch, Milvus), या ग्राफ़क्यूएल (Weaviate) का उपयोग करते हैं। वे काम करते हैं; जैसे-जैसे फ़िल्टर तर्क जटिल होता जाता है, एसक्यूएल अधिक आकर्षक होता जाता है।

```sql
-- pgvector: सदिश समानता WHERE में एक अभिव्यक्ति है
SELECT id, title, 1 - (embedding <=> $1) AS score
FROM documents
WHERE tenant_id = $2 AND category = ANY($3::text[]) AND created_at > NOW() - INTERVAL '90 days'
ORDER BY embedding <=> $1 LIMIT 10;
```

```python
# Qdrant: समतुल्य फ़िल्टर एक पायथन ऑब्जेक्ट के रूप में — कार्यात्मक, अधिक रस्म
results = client.query_points(

```

**बहु-मोडली आंतरिक** इसका अर्थ नहीं है कि "छवि समानता संग्रहीत कर सकता है"; हर डेटाबेस फ्लोट सरणियाँ संग्रहीत करता है। इसका अर्थ यह है कि डेटाबेस गैर-पाठ विषयों के लिए समानता मॉडल शामिल करता है, इसलिए आप उसे एक खुली छवि URL देते हैं और यह सदिशीकरण करता है। यहाँ के अधिकांश डेटाबेस समानता-निरपेक्ष हैं, इसलिए आप उस पाइपलाइन के मालिक हैं। मार्को और वेविएट (CLIP/छवि-बाइंड मॉड्यूल के माध्यम से) लूप को बंद करते हैं।

```python
# मार्को: खुली छवियाँ पोस्ट करें, पाठ के साथ प्रश्न करें — बाहरी समानता चरण की आवश्यकता नहीं है
mq.index("products").add_documents(
    [{"id": "shoe-001", "image": "https://cdn.example.com/shoes/001.jpg"}],
    tensor_fields=["image"]
)
results = mq.index("products").search(q="lightweight shoes for summer")
# शूज़-001 लौटाता है भले ही कोई भी शब्द संयोग न हो — CLIP अन्तः-मोडल समानता को संभालता है
```

**डिस्क-आधारित सूचकांक** एक लागत प्रश्न है। RAM-रिज़िडेंट HNSW को ग्राही सदिश, ग्राफ़ अतिरिक्त और मेटाडेटा की गणना के बाद प्रति मिलियन 1536-आयामी सदिशों के लिए कई जीबी रैम की आवश्यकता हो सकती है। डिस्क-निवासी विकल्प (मिल्वस डिस्कएएनएन, एलास्टिकसर्च डिस्कबीक्यू, लेंसडीबी का लेंस फॉरमेट, टर्बोपफ़र का ऑब्जेक्ट संग्रहण टियर) अक्सर कुछ प्रश्न विलंब के बदले सस्ते बुनियादी ढांचे के लिए व्यापार करते हैं। आरएजी कार्यभार में जहां मॉडल विलंब पहले से ही प्रभावी होता है, वहां यह अक्सर बेंचमार्क करने लायक होता है। रेडिस वीएसएस कठोर प्रतिबंध है: केवल रैम, कोई डिस्क पथ नहीं।

**अधिकतम आयाम** आज के चयन में छिपा कल का माइग्रेशन है। `text-embedding-3-large` 3072 आयामों का उपयोग करता है, जिना वी3 बड़े समानता उत्पन्न कर सकता है, और अनुसंधान मॉडल अधिक ऊँचाई तक धकेलते रहते हैं। कुछ प्रबंधित सेवाएँ कठोर आयाम सीमा प्रकाशित करती हैं; अन्य उच्च सीमा या सामान्य समानता मॉडल के लिए कोई व्यावहारिक सीमा देखाई देती हैं। समर्पण से पहले वर्तमान दस्तावेज़ जांचें। कुछ अंतराल चुनें; समानता सूचकांक को आयाम सीमा पर पहुँचने के कारण माइग्रेट करना एक दुःखद स्प्रिंट होता है।

2026 के मई 8 को सार्वजनिक परियोजना दस्तावेज़ और उत्पाद पृष्ठों के साथ अंतिम सत्यापन किया गया। मैट्रिक्स को एक तस्वीर के रूप में लें: प्रबंधित-सेवा सीमाएं, मूल्य, हाइब्रिड-खोज विशेषताएं, और डिस्क-सूचकांक विकल्प तेजी से बदल रहे हैं।

### मैट्रिक्स को पढ़ें

कुछ चीजें एक तालिका में स्पष्ट रूप से नहीं फिट होती हैं:

**विरल सदिश (Sparse vectors)** एक वेक्टर इंडेक्स के भीतर BM25 गुणवत्ता वाला कीवर्ड मिलान प्राप्त करने का तरीका है, जिसमें कोई अलग पूर्ण-पाठ इंजन नहीं होता। Qdrant और Elasticsearch यहां विशेष रूप से परिपक्व कार्यान्वयन प्रदान करते हैं। Weaviate उन्हें BM25F के माध्यम से समर्थन करता है। यदि हाइब्रिड खोज आवश्यक है और आप दो प्रणालियां नहीं चला सकते, तो विरल सदिश समर्थन की तलाश करें।

**डिस्क-आधारित सूचकांक (Disk-based indexes)** एक लागत नियंत्रक हैं, न कि एक कार्यान्वयन विवरण। RAM-वासी HNSW सूचकांक तेज़ होते हैं लेकिन वेक्टर गिनती, आयाम गिनती, मेटाडेटा और ग्राफ ओवरहेड बढ़ने पर महंगे हो सकते हैं। डिस्क-आधारित विकल्प (Milvus DiskANN, Elasticsearch DiskBBQ, Turbopuffer ऑब्जेक्ट स्टोरेज, LanceDB का Lance फॉरमैट) प्रश्न लेटेंसी के बदले कम बुनियादी ढांचा लागत पर व्यापार करते हैं। बड़े RAG सूचकांक के लिए, यह व्यापार अक्सर परीक्षण करने लायक होता है।

**टर्बोपफ़र का बहु-किरायेदार** बहुत अधिक नेमस्पेस गिनती पर बनाया गया है। इसकी सार्वजनिक स्थिति और ग्राहक कहानियाँ नोशन के बड़े, नेमस्पेस-भारी कॉर्पस जैसे कार्यों पर बल देती हैं। अगर प्रत्येक उपयोगकर्ता या संगठन को अलग-अलग वेक्टर सर्च की आवश्यकता है, तो यह आर्किटेक्चर अर्थव्यवस्था में परिवर्तन कर सकता है, लेकिन अपने टेनेंट आकार का अभी भी बेंचमार्क करें।

**लेंसडीबी एम्बेडेड मोड** "वेक्टर सर्च के लिए एसक्यूएलआईटी" के सबसे करीब है। यह प्रक्रिया में चलता है, कोई सर्वर आवश्यकता नहीं है, और लैम्ब्डा, क्लाउडफ़ेयर वर्कर्स और एज वातावरण में काम करता है। लेंस कॉलमार फॉरमेट वास्तविक पैमाने पर एम्बेडेड ऑपरेशन को व्यावहारिक बनाता है।

**ओरमा सर्च यूएक्स बुनियादी ढांचा है, न कि एक डेटा भंडार।** यह तब अच्छा होता है जब आपको एक छोटे फुल-टेक्स्ट/वेक्टर/हाइब्रिड सर्च इंजन की आवश्यकता होती है, जेएस एप्प में, एज पर, या प्रबंधित साइट/एप्प सर्च लेयर के रूप में। बिलियन-वेक्टर रिट्रीवल, भारी विश्लेषण या जटिल फ़िल्टर्ड जॉइंस के लिए यह उपकरण मैं अपन चुनाव नहीं करूंगा।

**वेस्पा तब उपयोग करने योग्य होता है जब रिट्रीवल केवल उत्पाद का आधा हिस्सा होता है।** यह लेक्सिकल रिट्रीवल, निकटतम-पड़ोसी खोज, टेंसर, रैंकिंग अभिव्यक्तियाँ, समूहीकरण, और ऑनलाइन सर्विसिंग को जोड़ता है। यह शक्ति वास्तविक है, लेकिन संचालन और मॉडलिंग की जटिलता भी है। यह खोज/अनुशंसा टीमों के लिए अधिक उपयुक्त है जबकि "अपने CRUD एप्प में सेमेंटिक खोज जोड़ें" के लिए नहीं।

**जब खोज विश्लेषण से जुड़ी होती है तो क्लिकहाउस चर्चा में शामिल होना चाहिए।** अगर आपका सत्य का स्रोत घटनाएँ, लॉग, ट्रेस, मेट्रिक्स, या बड़े तथ्य तालिकाएँ हैं, तो क्लिकहाउस वेक्टर दूरी, फ़िल्टरिंग, एग्रीगेशन, और अब गंभीर पूर्ण-पाठ इंडेक्सिंग को एक SQL इंजन में रख सकता है। यह एक उद्देश्य-निर्मित वेक्टर डेटाबेस नहीं है, लेकिन विश्लेषणात्मक रिट्रीवल के लिए यह सबसे सामान्य उत्तर हो सकता है।

**क्रोमा डेव/टेस्ट और छोटे एप्प डिप्लोमेंट में सबसे मजबूत होता है।** अगर आप बहुत बड़े कॉर्पोरा, उच्च उपलब्धता (HA), डिस्क-भारी संचालन, या प्रथम श्रेणी की हाइब्रिड खोज के लक्ष्य पर काम कर रहे हैं, तो प्रोटोटाइप को बुनियादी ढांचे में बढ़ाने से पहले एक उत्पादन-उन्मुख संग्रह का मूल्यांकन करें।

### सरल निर्णय

अगर आपने वास्तव में pgvector के बाहर बढ़ लिया है - आमतौर पर बेंचमार्क्स दिखाते हैं कि वेक्टर काउंट, फ़िल्टरिंग, लिखने की दर, या उच्च-समान्तरता लेटेंसी आपके पोस्टग्रेस एनवेलोप के बाहर जा रही है - तो प्रतिबंध द्वारा चुनें:

- **SaaS उत्पाद जिसमें प्रति-किरायेदार अलगाव हो** → Turbopuffer  
- **Rust-स्तर का प्रदर्शन + जटिल मेटाडेटा फ़िल्टरिंग की आवश्यकता हो** → Qdrant  
- **पहले से ही Elastic/ELK स्टैक पर हैं** → Elasticsearch सहित DiskBBQ  
- **AWS शॉप जो ओपन-सोर्स चाहता है** → OpenSearch  
- **खोज/अनुशंसा प्लेटफॉर्म जिसमें गंभीर रैंकिंग आवश्यकताएँ हैं** → Vespa  
- **विश्लेषण, अवलोकनीयता, या लॉग/घटना खोज** → ClickHouse  
- **अरब-स्केल ऑन-प्रीम / स्व-होस्टेड** → Milvus  
- **किनारा / सर्वरलेस / बहु-संवेदी** → LanceDB  
- **छोटा JS एप्प, दस्तावेज़ साइट, या किनारा-नैतिक खोज UX** → Orama  
- **शून्य ऑपरेशन, केवल काम करता है, लागत द्वितीयक है** → Pinecone  
- **बहु-संवेदी प्रथम (चित्र, वीडियो, ऑडियो)** → Marqo  
- **पहले से ही MongoDB पर हैं** → Atlas Vector Search  
- **पहले से ही Postgres पर हैं, अधिक गैर-मौजूदा आवश्यकता है** → Supabase Vector या Neon (दोनों pgvector मैनेज्ड, बेहतर टूलिंग के साथ)

## एक चीज जिसे नहीं करना चाहिए

वेक्टर खोज का ओझल पाठ खोज के रूप में उन चीजों के लिए उपयोग न करें जिनके सही उत्तर होते हैं।

"मुझे उस उपयोगकर्ता को ढूंढें जिसका ईमेल `dan@example.com` है" एक वेक्टर खोज समस्या नहीं है। न ही "मुझे उस ऑर्डर को ढूंढें जिसका ID `ORD-12345` है" है। `ORD-12345` को एम्बेड करना और आपकी ऑर्डर तालिका के खिलाफ कोसाइन समानता करना *कुछ* लौटाएगा, लेकिन यह गलत हो सकता है। ये एक्सैक्ट-मैच समस्याएँ हैं। अपनी मुख्य कुंजी या एक सामान्य सूचकांक का उपयोग करें।
