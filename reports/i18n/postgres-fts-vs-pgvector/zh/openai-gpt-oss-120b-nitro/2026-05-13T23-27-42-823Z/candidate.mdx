团队在添加 AI 功能时，往往首先会选择专用的向量数据库。

Pinecone、Weaviate、Qdrant、Chroma。新服务、新依赖、新连接池、新账单，同时还要维护两套真相的同步。

与此同时，他们已经拥有 PostgreSQL。PostgreSQL 本身已经提供了 `pgvector`。自 2008 年起，它还内置了出色的全文搜索功能。

专用向量存储在大规模和高查询量的场景下才显得必要。但大多数应用在第一套系统还未被压到极限时，就已经去寻找第二套搜索系统。这正是把未来的扩展问题变成今天的同步 bug 的典型路径。

所以：什么时候该使用 FTS，什么时候该使用 pgvector，什么时候两者都要用？

---

## 每个工具到底做了什么

全文搜索（`tsvector` / `GIN` 索引）是词法层面的。它会把文本切分为词元，进行词干提取，然后在索引上匹配查询。“Running” 与 “runs” 会归为同一个词元，“dog” 与 “dogs” 亦然。排名函数（`ts_rank`）会提升那些查询词出现频率高或位置突出的文档。

`pgvector` 则是语义层面的。它存储一个密集向量——一串数值——表示块的*意义*，由嵌入模型生成。相似度搜索在高维空间中寻找相邻向量。“Dog” 与 “canine” 会彼此靠近，而 “running” 作为运动和 “running” 作为进程可能并不相近。

实际区别在于：全文搜索回答的是“哪些文档包含这些词？” 向量搜索回答的是“哪些文档的含义大致是这个？”

![显示 pg_trgm 处理短模糊字符串、全文搜索处理精确文本查询、pgvector 进行语义匹配，以及混合搜索处理需要同时考虑精确和语义信号的长内容的搜索工具地图。](../search-tool-map.svg)

_第一次划分并不是“传统搜索 vs. AI 搜索”。而是文本的形态以及哪种答案才是正确的。_

## 当全文搜索占优时

**你在寻找完全匹配的关键字。** 商品 SKU、错误码、型号、用户名、法律条款引用。`SKU-AX-44192` 在语义上与任何内容都不相似。它要么匹配，要么不匹配。向量搜索可能会自信地返回 `SKU-AX-44193`，这显然不是你想要的结果。

**你的查询基于关键词。** 用户在搜索框中输入、按标签过滤，或按关键字检索博客文章。全文搜索正是为这种意图而生。

**你需要在没有 GPU 或嵌入基础设施的情况下获得排序结果。** 全文搜索索引快速、确定性强，且不需要外部 API 调用。添加一个 `tsvector` 列，构建 GIN 索引，工作即可完成。

**你在搜索的同时还要做布尔过滤。**  

```sql
WHERE to_tsvector(body) @@ to_tsquery('postgres') 
  AND category = 'tutorial' 
  AND published_at > NOW() - INTERVAL '6 months'
```  

这种写法可以自然地与现有查询逻辑组合。

```sql
-- 创建索引
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body, '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- 查询
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`GENERATED ALWAYS AS` 列会自动保持索引同步。`setweight` 让标题匹配的权重高于正文匹配。这就是完整的配置。

## 当Trigram 获胜时 (pg_trgm)

还有第三个 Postgres 工具常被忽视：`pg_trgm`。它既不是全文搜索，也不是向量搜索，而是模糊字符串匹配，填补了两者都处理不佳的尴尬中间地带。

**使用场景：姓名、地址、标识符以及带有拼写错误的短字符串。**

FTS 会把文本切分为词元并进行词干提取。这对自然语言段落有效，但在以下情况下表现不佳：
- 人名（例如 “Dan Levy” → 词干化为 “dan levi”、 “leiv”，取决于语言配置）
- 公司名称、地址、产品标题等对拼写精确度有要求的场景
- 含有拼写错误的查询 — “Micheal Jordan”、 “Amaon”、 “javascipt”
- 自动完成 / 前缀搜索
- 部分字符串匹配（如 “son” 匹配 “Johnson”、 “Anderson”）

`pgvector` 在这些场景下同样不适合。你可以为 “Micheal Jordan” 生成向量并寻找最近邻，但向量空间是按意义而非拼写组织的。最近邻可能是 “basketball legend” 或 “Michael B. Jordan”，而不是带有拼写错误的用户记录。

`pg_trgm` 将字符串拆分为重叠的 3‑字符切片，并衡量两个字符串共享多少 trigram。例如 “Dan” → `" da"`、`"dan"`、`"an "`。“Micheal” 与 “Michael” 共享大多数 trigram，因而相似度很高。

```sql
-- 启用扩展（通常已可用）
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- 在 name 列上创建 GIN 索引 — 支持快速 trigram 相似度搜索
CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- 模糊姓名搜索：在搜索 “Michael Jordan” 时找到 “Micheal Jordan”
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % 运算符 = 相似度阈值（默认 0.3）
ORDER BY score DESC
LIMIT 10;

-- 或者使用 ILIKE 并借助 trigram 索引进行包含匹配
SELECT id, name
FROM users
WHERE name ILIKE '%johnson%'   -- GIN 索引让此查询快速
LIMIT 10;
```

`%` 运算符使用 `pg_trgm.similarity_threshold`（默认 0.3，取值范围 0‑1）。阈值越高，匹配越严格。对姓名搜索而言，0.3‑0.4 通常是合适的：足够宽松以捕捉拼写错误，又足够严格以避免噪声。

**Trigram 还能帮助前缀搜索和自动完成，尤其是当自动完成需要容错或包含匹配时：**

```sql
-- 自动完成：前缀匹配。对于纯左侧前缀，
-- 将 trigram GIN 与数据上的 B‑tree 模式索引进行比较。
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- 更细粒度控制：word_similarity 用于在更长字符串内部的部分匹配
-- （在搜索 “Johnson” 于 “Andrew Johnson III” 时很有用）
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name          -- <% 运算符 = word_similarity 阈值
ORDER BY score DESC
LIMIT 10;
```

**何时优先选择 `pg_trgm` 而非 FTS：**

| 场景 | 使用 |
|---|---|
| 带拼写错误的人名/公司名搜索 | `pg_trgm` |
| 自动完成 / 前缀搜索 | `pg_trgm`（或使用前缀查询的 FTS） |
| 短字符串、代码、标识符搜索 | `pg_trgm` |
| 长篇文章、文档搜索 | FTS |
| 日志消息关键字搜索 | FTS |
| 多语言姓名搜索 | `pg_trgm`（语言无关） |

`pg_trgm` 也可以与 FTS 组合使用。先用三元组进行模糊预过滤，再用 `ts_rank` 排序，或将三元组相似度与向量分数结合。

## 当 pgvector 获胜时

**你在构建 RAG。** RAG 依赖语义检索：找出意义最接近用户提问的文档 *块*，即使表述不同。向量搜索正是为此而生。FTS 会漏掉同义改写、概念匹配等情况。

**用户描述的是需求，而不是搜索词。** “适合夏夜的清爽饮品”没有明显的葡萄酒关键词。 “关于新经理如何建立自信的文章”需要语义理解，FTS 无法提供。

**你在寻找相似项。** 相关产品、相似的支持工单、重复的 bug 报告。 “找出与此问题相似的工单”是向量操作。先对新问题生成嵌入，再寻找最近邻。

**多语言内容。** 在多语言数据上训练的向量嵌入可以跨语言匹配。FTS 需要针对每种语言单独配置，跨语言查询效果差。

```sql
-- Setup
CREATE EXTENSION IF NOT EXISTS vector;

ALTER TABLE documents ADD COLUMN embedding vector(1536);
CREATE INDEX documents_embedding_idx
  ON documents USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);

-- Query: semantic search
SELECT id, title, 1 - (embedding <=> $1::vector) AS similarity
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 10;
```

注意：`ivfflat` 是近似算法——速度快，但会在召回率上做出一定牺牲。对于较小的数据集（约 100 万行以下），`hnsw` 往往更合适：

```sql
CREATE INDEX documents_embedding_idx
  ON documents USING hnsw (embedding vector_cosine_ops);
```

## 当你需要两者兼备时

技术文档是简单二分法失效的地方。用户可能搜索 “how to configure timeouts”，但也会搜索函数名如 `withRetry()` 或错误码 `ECONNRESET`。

向量搜索处理概念性查询，全文检索处理精确词项。单独使用时，两者都难以兼顾。

解决方案是混合搜索：同时运行两种搜索并融合结果。

**Reciprocal Rank Fusion (RRF)** 是此处的标准算法。它不要求你对两个系统的分数进行归一化，只合并排名位置。

```sql
-- Hybrid search with Reciprocal Rank Fusion
WITH fts_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM documents, to_tsquery('english', $1) query
  WHERE search_vector @@ query
  LIMIT 50
),
vector_results AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> $2::vector) AS rank
  FROM documents
  ORDER BY embedding <=> $2::vector
  LIMIT 50
),
rrf AS (
  SELECT
    COALESCE(f.id, v.id) AS id,
    COALESCE(1.0 / (60 + f.rank), 0) + COALESCE(1.0 / (60 + v.rank), 0) AS rrf_score
  FROM fts_results f
  FULL OUTER JOIN vector_results v ON f.id = v.id
)
SELECT d.id, d.title, rrf.rrf_score
FROM rrf
JOIN documents d ON d.id = rrf.id
ORDER BY rrf_score DESC
LIMIT 10;
```

分母中的 `60` 是 RRF 常数——数值越大，排名差异的影响越小，数值越小则放大差异。默认 60 在大多数场景下表现良好。

该查询在一次 SQL 中执行两种搜索，融合排名，并对关键词和语义信号一致的结果给予额外奖励。

![A hybrid search pipeline where one query fans out to full-text search and pgvector, each produces ranked results, and Reciprocal Rank Fusion combines the two lists.](../hybrid-rrf-pipeline.svg)

_RRF 的价值在于它避免了把 `ts_rank` 与余弦距离当作可比的原始分数，只关心 “每个列表中该结果出现的排名有多高”。_

## 实际决策树

当选择搜索策略时，先看 **输入的形态**，再问 **用户到底在做什么样的查询**。 “带有拼写变化的短字符串” 与 “精确词汇很重要的长篇文字” 并不是同一个问题，两者也都不同于 “针对文档块的提问”。

![基于文本形态和用户查询方式，选择 pg_trgm、全文搜索、pgvector、混合搜索或专用向量数据库的决策树。](../search-decision-tree.svg)

同一棵树的文字描述：

- **姓名、地址、标题、自动完成或易错的短字符串** → `pg_trgm`
- **已知词、错误码、SKU、函数名、标签、分类、过滤器** → 全文搜索 (FTS)
- **问题、改写、推荐、相关项目、多语言匹配、RAG 块** → `pgvector`
- **技术内容，用户既需要精确符号又需要概念性答案** → 使用 RRF 的混合搜索
- **主键、精确 ID、权限过滤、日期范围、排序列表** → 普通 SQL 索引
- **向量规模巨大、并发极高或延迟目标超出 Postgres 基准** → 评估专用向量存储

### FTS 与语义搜索：简要对比

“我该用全文搜索还是向量搜索？”这个问题通常可以归结为：**你是否知道相关文档中会出现哪些词？**

- **如果知道**——用户搜索已知的术语、分类、函数名、产品代码——全文搜索更快、更便宜且更可预测。它还能告诉你为何匹配到了该结果。
- **如果不知道**——用户描述概念、提问或使用不同语言——向量搜索才是合适的工具。它匹配的是意义，而不是字面词汇。

中间的棘手情况是对技术内容的自然语言查询。比如有人搜索 “how do I handle connection drops”，可能需要一篇标题为 “implementing retry logic for network failures” 的文章——两者几乎没有重叠词，却在语义上高度相关。这正是向量搜索发挥价值的场景。

另一个棘手案例是 **人名和专有名词**。全文搜索和向量搜索都对它们表现不佳：
- 全文搜索在搜索 “Michael” 时会漏掉 “Micheal”——因为分词不同
- 向量搜索如果在训练数据中出现频率低，可能根本找不到该名称
- `pg_trgm` 能正确处理：基于正字法相似度，而非语义或词汇匹配

在实际使用中，大多数内容丰富的搜索框需要全文搜索（FTS）来保证速度和关键词匹配，并且可能需要混合搜索或 `pg_trgm`，具体取决于用户是否会搜索人名。真正的语义搜索功能通常意味着使用 pgvector。RAG 场景则必然使用 pgvector。

## 如果确实需要专用的向量存储

有些系统的需求会超出 pgvector 的承载范围。此时市场上选择繁多，关键在于以下几个维度的比较。

### 功能矩阵

在表格能说明问题之前，需要先解释几列的含义。

**混合搜索** 指在同一次查询中同时运行 BM25 关键词搜索和向量相似度搜索，并通过 Reciprocal Rank Fusion（倒数秩融合）合并结果。比如 `"withRetry timeout"` 可以同时精准匹配函数名 *and* 语义上与 “retry logic for network failures” 相关的文档。若不使用混合搜索，你只能选择单一搜索模式，或自行将两个查询的结果融合。pgvector 的 “Manual (RRF via SQL)” 正是[上文示例](#when-you-need-both) 中的做法：可行，但需要自行实现。

**稀疏向量** 超越了 BM25 的表达能力。SPLADE 稀疏向量大约有 30,000 维（对应词表中的每个词），约 98% 为零。非零位置指示哪些词重要以及其权重。例如查询 “dogs” 时，同样会给 “canine” 与 “pet” 加权：兼具 BM25 级别的关键词精度和向量索引内部的词项扩展。如果此列为 false，则需要外部的全文搜索层来处理精确词项查询。

```python
# SPLADE: ~30,000 dims total, ~60 non-zero — only the relevant vocabulary positions fire
def encode_splade(text: str) -> dict:
    tokens = tokenizer(text, return_tensors="pt", truncation=True, max_length=512)
    with torch.no_grad():
        output = model(**tokens)
    vec = torch.log1p(torch.relu(output.logits)).max(dim=1).values.squeeze()
    return {"indices": vec.nonzero().squeeze().tolist(), "values": vec[vec != 0].tolist()}
```

**SQL / 类 SQL** 实际上是指过滤能力。没有过滤的向量搜索只能算作演示，无法直接用于生产：仍然需要租户范围、时间区间、权限和类别等约束。完整的 SQL（pgvector）可以把这些约束写在现有的 JOIN 之中。专用向量数据库则通常使用 JSON 过滤对象（Qdrant、Pinecone）、查询 DSL（Elasticsearch、Milvus）或 GraphQL（Weaviate）。它们都能工作，但当过滤逻辑变得错综复杂时，SQL 的吸引力会提升。

```sql
-- pgvector: vector similarity is just another expression in WHERE
SELECT id, title, 1 - (embedding <=> $1) AS score
FROM documents
WHERE tenant_id = $2 AND category = ANY($3::text[]) AND created_at > NOW() - INTERVAL '90 days'
ORDER BY embedding <=> $1 LIMIT 10;
```

```python
# Qdrant: equivalent filter as a Python object — functional, more ceremony
results = client.query_points(
    collection_name="documents", query=query_embedding,
    query_filter=models.Filter(must=[
        models.FieldCondition(key="tenant_id", match=models.MatchValue(value=tenant_id)),
        models.FieldCondition(key="category",  match=models.MatchAny(any=categories)),
        models.FieldCondition(key="created_at", range=models.DatetimeRange(gte=cutoff)),
    ]),
    limit=10,
)
```

**多模态原生** 并不等同于 “能存储图像嵌入”；所有数据库都能存放浮点数组。这里指的是数据库自带非文本内容的嵌入模型，你只需提供原始图像 URL，数据库即可完成向量化。大多数数据库在这点上保持模型无关，需要自行构建管道。Marqo 和 Weaviate（通过 CLIP / ImageBind 模块）提供了端到端的闭环方案。

```python
# Marqo: POST raw images, query with text — no external embedding step needed
mq.index("products").add_documents(
    [{"id": "shoe-001", "image": "https://cdn.example.com/shoes/001.jpg"}],
    tensor_fields=["image"]
)
results = mq.index("products").search(q="lightweight shoes for summer")
# Returns shoe-001 despite zero keyword overlap — CLIP handles the cross-modal match
```

**基于磁盘的索引** 是成本问题。每百万条 1536 维原始向量在 RAM 中的 HNSW 索引往往需要数 GB 内存，考虑到原始向量、图结构开销以及元数据后更是如此。磁盘原生的替代方案（Milvus DiskANN、Elasticsearch DiskBBQ、LanceDB 的 Lance 格式、Turbopuffer 的对象存储层）通常会以稍高的查询延迟换取更低的基础设施开支。对于模型延迟已经主导的 RAG 工作负载，这种权衡往往值得基准测试。Redis VSS 则是硬性限制：只能在 RAM 中运行，暂无磁盘路径。

**最大维度** 是今天的选择埋下的明日迁移风险。`text-embedding-3-large` 使用 3072 维，Jina v3 能输出更大的向量，研究模型也在不断推高维度。部分托管服务会公布硬性维度上限；其他则只给出高上限或对常见模型没有实际限制。务必在决定前查阅最新文档。选一个有余量的方案；因为维度上限而迁移向量索引往往是一场痛苦的冲刺。

### 阅读矩阵

有些信息在表格里难以完整呈现：

**稀疏向量** 让你在向量索引内部获得 BM25 级别的关键词匹配，而无需单独的全文检索引擎。Qdrant 和 Elasticsearch 在这方面实现尤为成熟。Weaviate 通过 BM25F 提供支持。如果混合搜索是关键且无法运行两套系统，务必寻找稀疏向量支持。

**基于磁盘的索引** 是成本杠杆，而非实现细节。RAM 中的 HNSW 索引查询快，但随着向量数量、维度、元数据以及图结构开销增长，成本会迅速上升。磁盘原生方案（Milvus DiskANN、Elasticsearch DiskBBQ、Turbopuffer 对象存储、LanceDB 的 Lance 格式）以稍高的查询延迟换取更低的基础设施支出。对于大规模 RAG 索引，这种权衡常常值得实验。

**Turbopuffer 的多租户** 设计围绕极高的命名空间数量。其公开定位和客户案例强调类似 Notion 那样的庞大、命名空间密集型语料。如果每个用户或组织都需要独立的向量搜索，这种架构会显著影响成本模型，但仍需自行基准测试实际租户形态。

**LanceDB 的嵌入模式** 是最接近 “SQLite for vector search” 的方案。它在进程内运行，无需服务器，可在 Lambda、Cloudflare Workers 以及边缘环境中使用。Lance 的列式格式让嵌入式操作在真实规模下也变得可行。

**Orama 是搜索 UX 基础设施，而不是数据仓库。** 当你需要在 JavaScript 应用、边缘或作为托管站点/应用搜索层中部署一个小型的全文/向量/混合搜索引擎时，它表现出色。但它并不是我会选来处理十亿向量检索、重度分析或复杂过滤联接的工具。

**Vespa 是在检索只是产品一半时的首选。** 它把词汇检索、最近邻搜索、张量、排序表达式、分组以及在线服务融合在一起。这种能力确实强大，但运维和建模的复杂度同样不容小觑。它更适合搜索/推荐团队，而不是“给我的 CRUD 应用加个语义搜索”。

**ClickHouse 在搜索与分析结合时值得一提。** 如果你的真相数据是事件、日志、追踪、指标或大规模事实表，ClickHouse 能在同一个 SQL 引擎里保存向量距离、过滤、聚合，甚至现在还能做相当完整的全文索引。它不是专门的向量数据库，但在分析检索场景下，它往往是最踏实的答案。

**Chroma 在开发/测试和小型应用部署时最为强大。** 若你的目标是超大语料库、高可用、磁盘密集型运行或一流的混合搜索，请在把原型提升为基础设施之前，先评估面向生产的存储方案。

### 简化决策

如果你真的已经超出了 pgvector 的能力——通常是因为基准测试显示向量数量、过滤、写入速率或高并发延迟已经突破了 PostgreSQL 的上限——请按约束条件选择：

- **SaaS 产品需要租户隔离** → Turbopuffer  
- **需要 Rust 级别性能 + 复杂元数据过滤** → Qdrant  
- **已经在使用 Elastic/ELK 堆栈** → 带 DiskBBQ 的 Elasticsearch  
- **AWS 环境且想要开源方案** → OpenSearch  
- **搜索/推荐平台需要严肃的排序** → Vespa  
- **分析、可观测性或日志/事件搜索** → ClickHouse  
- **千亿级规模的本地部署** → Milvus  
- **边缘 / 无服务器 / 多模态** → LanceDB  
- **小型 JS 应用、文档站点或边缘原生搜索 UX** → Orama  
- **零运维、开箱即用、成本次要** → Pinecone  
- **多模态优先（图像、视频、音频）** → Marqo  
- **已经在使用 MongoDB** → Atlas Vector Search  
- **已经在使用 Postgres，需要更多余量** → Supabase Vector 或 Neon（两者都是托管的 pgvector，工具链更完善）

## 一件绝不能做的事

不要把向量搜索当作模糊文本搜索来处理那些有唯一正确答案的场景。

“找出电子邮件为 `dan@example.com` 的用户”并不是向量搜索的场景。 “找出订单号为 `ORD-12345` 的订单”同样不适合向量搜索。把 `ORD-12345` 进行嵌入并在订单表上做余弦相似度会得到*某个结果*，但它可能是错误的。这类问题要求精确匹配，应该使用主键或普通索引。

向量搜索会返回数据集中*最相似*的条目，即使根本没有相关项。它并不知道没有合适的答案。这在查找相关文档时还能接受，但在精确记录查找时就会灾难性——一个错误的近似匹配比空结果更糟。

要清楚每种工具的定位。大多数工具已经随你的 Postgres 安装而来，按适用场景使用即可。
