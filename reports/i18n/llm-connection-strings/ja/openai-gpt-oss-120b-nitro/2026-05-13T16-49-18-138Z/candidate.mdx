データベースに接続する際に、環境変数のごちゃ混ぜをいちいち扱わなければならなかった、あの悪名高い昔を覚えていますか？

それは繊細な設定の塔でした。`DB_HOST`、`DB_PORT`、`DB_USER`、`DB_PASSWORD`、`DB_NAME`… いや、`DB_USERNAME`だったかな？ `DB_PASS` か `DB_PWD` か？ 今回は `PG_*` プレフィックスが必要なのか？ タイムアウト設定はどこに入れるんだ？

脆いカードの家のようで、`HOST` の大文字小文字を忘れただけで本番ビルドが崩壊する危険が常にありました。

そこで、誰かが「URL だけで済ませよう」という brilliant なアイデアを思いつきました¹：

```bash
postgres://user:pass@host:5432/dbname
```

たった一つの文字列。必要な情報はすべてここに。普遍的にパース可能で、ポータブル。敢えて言うなら… 美しい。

なぜ私たちは LLM を 1999 年のまま扱っているのでしょうか？

## 環境変数の爆発

現在、私の `.env` ファイルは放棄された API キーの墓場のようです。`OPENAI_API_KEY`、`ANTHROPIC_API_KEY`、`MISTRAL_API_KEY`、`GROQ_API_KEY`。そして Azure については語るに及びません――エンドポイント、デプロイ名、API バージョン、キーのすべてが「こんにちは」と言うだけで必要です。

見た目が悪いだけでなく、摩擦も生みます。モデルを入れ替えたり新しいプロバイダーをテストしたりするたびに、初期化コードを書き直し、特定のパラメータ名のドキュメントを探し回り、環境設定にさらに 3 行を追加しなければなりません。

もし… ~~盗む~~ 借りてくるだけで、データベース URL のアイデアを取り入れたらどうでしょうか？

## LLM 接続文字列の紹介

Imagine configuring your entire model interface with a single line:

```bash
llm://api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7&max_tokens=1500
llm://api.z.ai/glm-4.7?top_p=0.9&cache=true
```

### LLM 接続文字列の構成要素

![LLM 接続文字列の各部品](../inline-url-diagram-dark.svg)

スキームは `llm://`。ホストはプロバイダーの API ベース URL。パスがモデル名を表し、クエリパラメータが通常コードを散らかすランタイムオプションすべてを処理します。

## 認証が必要ですか？ では、URLに埋め込みましょう。

`postgres://` と同様に、認証情報を直接埋め込めます。

```bash
llm://app-name:sk-proj-123456@api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7
```

*注: URL に認証情報を入れると、公開ログに貼り付けた際に情報漏洩のリスクがあります。ただし、最近のログサービスはこのパターンをかなり上手くマスクしますし、正直なところ `.env` ファイルをもっと厳重に管理しているでしょうか？検証・サニタイズし、注意して使用してください。*

## レジリエンスが欲しい？ もちろんです。

多くのデータベースライブラリは、複数ホストを指定してラウンドロビンフェイルオーバーをサポートしています。AI エージェントにも同等の信頼性を求めるべきでは？

```bash
llms://primary.gpt,backup.gpt/gpt-6?temp=0.9
```

`llms://` の末尾の `s` はタイプミスではなく、複数形を示しています。`primary.gpt` がハングした場合、クライアントは自動的に `backup.gpt` にリトライします。複雑なルーターロジックは不要です。

<blockquote class="inset">認証からエンドポイント、ハイパーパラメータまで、すべてを **1 つの文字列** にまとめたもの。</blockquote>

## Alternative Formats

`llm://` に固執しているわけではありません。スキーム自体は重要ではなく、標準化された構造が鍵です。

プロバイダー固有のスキームを使って記述を簡潔にしつつ、標準フォーマットは維持するといった使い方も想像できます。

```bash
ollama://localhost:11434/llama3
vercel://anthropic/sonnet-4.5?temp=0.8&web_search={"maxUses":3}
bedrock://us-west-2.aws/anthropic/sonnet-4.5?temp=0.8&cacheControl=ephemeral
```

正確な構文はどうであれ、核心的な利点は揺るぎません。

1.  **Portability:** ローカルスクリプトからクラウドワーカーへ、設定全体をコピー＆ペーストできる。
2.  **CLI Friendly:** スクリプトに単一の引数を渡すだけで済む。`my-agent --model "llm://..."` は `my-agent --model gpt-4 --temp 0.7 --key $KEY --host ...` より圧倒的に扱いやすい。
3.  **Language Agnostic:** すべてのプログラミング言語に堅牢な URL パーサーが備わっているため、バリデーション・パース・サニタイズが自動的に得られる。

<blockquote class="ai-response inset">データベース界はこれを解決するまでに何十年もかかった。<br /><b>良いニュースとして、AI のタイムラインではそれがたった半年ほど前のことだ。</b></blockquote>

## 結論

新たな複雑な設定標準や YAML ベースのマニフェストファイルは不要だ。過去 30 年間インターネット全体で実績のある、あのツールをそのまま使えばいい。

車輪の再発明はやめて、LLM 接続をデータベースと同等の敬意で扱おう。`.env` ファイル（そして自分の正気）も感謝してくれるはずだ。
