昔、データベースに接続するのにさまざまな環境変数を扱う必要があったことを覚えていますか？

それはデリケートな構成の塔でした。`DB_HOST`、`DB_PORT`、`DB_USER`、`DB_PASSWORD`、`DB_NAME`... いや、`DB_USERNAME`だったか？`DB_PASS`か`DB_PWD`か？今回は`PG_*`の接頭辞が必要か？タイムアウト設定はどこへ入れればいいんだ？

`HOST`のキャピタライズを忘れただけで、プロダクションビルドが崩壊するという危険なカード建てでした。

その後、誰かがURL¹を使うという素晴らしいアイデアを思いつきました:

```bash
postgres://user:pass@host:5432/dbname
```

1つの文字列。必要なすべてが揃っています。普遍的に解析可能。移動可能。僭越ですが...美しくありませんか？

LLMを扱う際に、なぜまるで1999年の技術のように扱っているのでしょうか？

## 環境変数の爆発

現在、私の`.env`ファイルは放浪されたAPIキーの墓場のように見えます。`OPENAI_API_KEY`、`ANTHROPIC_API_KEY`、`MISTRAL_API_KEY`、`GROQ_API_KEY`...。Azureについては特に言いたくないですが、単に「こんにちは」と言うためにもエンドポイント、デプロイ名、APIバージョン、キーが必要です。

これは単に見苦しいだけでなく、作業の摩擦を生み出しています。モデルを切り替えるたびに初期化コードを書き直し、特定のパラメータ名のドキュメントを探し回り、環境設定に3行も追加する必要があります。

では、なぜDB URLのアイデアを...借用してみないでしょうか？  

## LLM接続文字列の紹介

モデルインターフェース全体を1行で設定できると想像してみましょう：

```bash
llm://api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7&max_tokens=1500
llm://api.z.ai/glm-4.7?top_p=0.9&cache=true
```

---

### LLM接続文字列の構造

![LLM接続文字列の構成要素](../inline-url-diagram-dark.svg)

スキームは`llm://`です。ホストはプロバイダのAPIベースURLです。パスはモデル名です。クエリパラメータは、通常コードに散らばっているランタイムオプションをすべて処理します。

## 認証が必要？いいよ、追加するだけ。

`postgres://`のように、認証情報を文字列に直接組み込むことができます：

```bash
llm://app-name:sk-proj-123456@api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7
```

*注：はい、URLに資格情報を埋め込むのは、パブリックログに貼り付けた場合のセキュリティリスクがあります。しかし現代のログサービスはこのパターンのクリーニングがかなり上手です。正直に言いますが、`.env`ファイルの取り扱いがそれより安全ですか？使用前には検証し、サニタイズし、注意して使用してください。*

## 高可用性？なぜ不可能なのか。

多くのデータベースライブラリは複数のホストを指定することでロードバランシングやフェイルオーバーをサポートしています。なぜAIエージェントにも同じ信頼性を要求できないのでしょうか？

```bash
llms://primary.gpt,backup.gpt/gpt-6?temp=0.9
```

`llms://`の`s`はタイプミスではありません。複数形です。`primary.gpt`が応答しなくなった場合、クライアントは自動的に`backup.gpt`をリトライします。複雑なルーター論理は必要ありません。

**auth** から **endpoint** から **hyperparameters** まで、すべてを含む1つの文字列です。

## 代替形式

`llm://` に固執しているわけではありません。具体的なスキームよりも標準自体が重要です。

プロバイダごとのスキームを簡潔さのために使用しつつ、標準的な構造を維持する世界を想像できます：

```bash
ollama://localhost:11434/llama3
vercel://anthropic/sonnet-4.5?temp=0.8&web_search={"maxUses":3}
bedrock://us-west-2.aws/anthropic/sonnet-4.5?temp=0.8&cacheControl=ephemeral
```

正確な構文に関わらず、コアの利点は否定できません：

1.  **ポータビリティ:** ローカルスクリプトからクラウドワーカーに設定をコピー＆ペーストできます。
2.  **CLIに親和的:** スクリプトに1つの引数を渡せます。`my-agent --model "llm://..."` は `my-agent --model gpt-4 --temp 0.7 --key $KEY --host ...` よりも優れています。
3.  **言語に依存しない:** すべてのプログラミング言語には頑健なURLパーサーがあります。検証、解析、サニタイズが無料で得られます。

## 結論

複雑な設定規格や新たなYAMLベースのマニフェストファイルは必要ありません。インターネットの30年間で確実に機能してきたツールを使えば十分です。

再発明をやめましょう。LLM接続をデータベースと同じ尊重で扱い始めることにしましょう。あなたの`.env`ファイル（そして精神的健康）が感謝します。
