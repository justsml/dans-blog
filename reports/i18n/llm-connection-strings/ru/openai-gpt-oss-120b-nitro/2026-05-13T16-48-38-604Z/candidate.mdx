Помните те плохие старые времена, когда подключение к базе данных означало жонглирование бесконечным набором переменных окружения?

Это была башня из хрупкой конфигурации. `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`… а может быть, `DB_USERNAME`? Это `DB_PASS` или `DB_PWD`? Нужно ли сейчас добавлять префиксы `PG_*`? И куда, черт возьми, помещать настройку таймаута?

Это был хрупкий дом из карт, готовый обрушить ваш продакшн‑билд из‑за того, что вы забыли написать `HOST` в верхнем регистре.

Потом кто‑то придумал гениальную идею — просто использовать URL¹:

```bash
postgres://user:pass@host:5432/dbname
```

Одна строка. Всё, что нужно. Универсально разбираемая. Портативная. Позвольте сказать… красивая?

Так почему мы продолжаем обращатьсяс LLM‑ами, как будто нам снова 1999 год?

## Взрыв переменных окружения

Сейчас мой файл `.env` выглядит как кладбище забытых API‑ключей: `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `MISTRAL_API_KEY`, `GROQ_API_KEY`. И не заставляйте меня говорить про Azure — для простого «привет» вам нужен endpoint, имя развертывания, версия API и ключ.

Это не просто некрасиво, это тормоз. Каждый раз, когда я хочу переключить модель или протестировать нового провайдера, мне приходится переписывать инициализационный код, искать в документации названия конкретных параметров и добавлять ещё три строки в конфигурацию окружения.

А что если просто… ~~украсть~~ позаимствовать идею URL‑а базы данных?

## Представляем строку подключения LLM


Представьте, что всю конфигурацию интерфейса модели можно задать одной строкой:

```bash
llm://api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7&max_tokens=1500
llm://api.z.ai/glm-4.7?top_p=0.9&cache=true
```

Схема — `llm://`. Хост — базовый URL API провайдера. Путь — название модели. Параметры запроса — все опции выполнения, которые обычно захламляют ваш код.

## Нужно аутентифицироваться? Отлично, добавьте её.

Точно так же, как `postgres://`, мы можем запихнуть аутентификацию прямо в строку:

```bash
llm://app-name:sk-proj-123456@api.openai.com/gpt-5.2?reasoning_effort=none&temp=0.7
```

*Примечание: Да, размещение учётных данных в URL может представлять риск безопасности, если вы выводите их в публичные логи. Но современные сервисы логирования довольно хорошо очищают такие шаблоны, и, честно говоря, вы же лучше относитесь к своему файлу `.env`, правда? Проверяйте, санитизируйте и используйте с осторожностью.*

## Надёжность? Почему бы и нет.

Многие библиотеки для баз данных поддерживают отказоустойчивый раунд‑робин, указывая несколько хостов. Почему наши AI‑агенты не должны иметь такой же уровень надёжности?

```bash
llms://primary.gpt,backup.gpt/gpt-6?temp=0.9
```

Эта `s` в `llms://` — не опечатка. Это множественное число. Если `primary.gpt` зависнет, клиент автоматически переключится на `backup.gpt`. Никакой сложной логики роутера не требуется.

<blockquote class="inset">Одна строка, содержащая всё: от **аутентификации** до **конечного пункта** и **гиперпараметров**.</blockquote>

## Альтернативные форматы

Я не привязан к `llm://`. Важнее сам стандарт, а не конкретная схема.

Можно представить мир, где для краткости используются схемы, специфичные для провайдера, но сохраняется стандартная структура:

```bash
ollama://localhost:11434/llama3
vercel://anthropic/sonnet-4.5?temp=0.8&web_search={"maxUses":3}
bedrock://us-west-2.aws/anthropic/sonnet-4.5?temp=0.8&cacheControl=ephemeral
```

Независимо от точного синтаксиса, основные преимущества неоспоримы:

1.  **Переносимость:** Копируете‑вставляете всю конфигурацию из локального скрипта в облачный воркер.
2.  **Удобство для CLI:** Передаёте один аргумент скриптам. `my-agent --model "llm://..."` гораздо проще, чем `my-agent --model gpt-4 --temp 0.7 --key $KEY --host ...`.
3.  **Языковая независимость:** В каждом языке программирования есть надёжный парсер URL. Мы получаем валидацию, разбор и санитизацию «в подарок».

## Вердикт

Нам не нужен ещё один сложный стандарт конфигурации или новый манифест‑файл на YAML. Достаточно воспользоваться тем единственным инструментом, который служит всему интернету уже 30 лет.

Прекратим изобретать велосипед и начнём относиться к подключениям LLM с тем же уважением, которое мы оказываем нашим базам данных. Ваш файл `.env` (и ваше здравомыслие) будут вам благодарны.
