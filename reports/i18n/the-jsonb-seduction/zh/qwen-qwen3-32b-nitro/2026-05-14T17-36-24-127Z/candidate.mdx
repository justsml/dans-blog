PostgreSQL 引入 JSONB 的目的是让您在不预先定义严格模式的情况下存储半结构化数据。这个想法本身是合理的：有时您确实不知道数据会是什么样子，或者数据变化过于频繁，传统列的设计反而不再适用。

这很重要，因为 JSONB 并不是个错误。在许多系统中，它恰恰是问题空间最清晰的表示方式。如果您需要存储第三方 webhook 负载、版本化事件体、功能标志，或是 LLM 配置对象（每个供应商和模型都暴露略有不同且持续变化的选项集），强行将所有内容塞入第一类列反而会更麻烦。

真正的问题在于 JSONB 成为了推迟模式决策的最简单方式，而无需承认自己在推迟决策。在意图与实现之间，它变成了数据库界的"我稍后会整理房间"。六个月前你随手添加的那个临时解决方案？它还在那里，现在生产环境已经依赖它了。

我总能看到同样的模式。一个团队添加 JSONB 列是因为他们不确定需求。他们向自己承诺等事情稳定后就规范化。三年后，这个列里包含了原本应该统一的用户资料的四十种不同版本，被十五个服务以不同假设进行查询。

技术债务本身并非来自 JSONB。而是来自你告诉自己要构建的内容与实际构建内容之间的差距：一个未文档化的读时模式系统。

## 通常会发生什么

你正在添加一个功能，不确定用户需要的是 `twitter_handle`、`bluesky_handle` 还是其他什么。与其思考模式设计，你直接这样做了：

```sql
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  profile JSONB
);
```

这能工作。你上线了这个功能，继续下一个，再下一个。JSONB 列在后台悄然增长。

这是岔路口。如果 `profile` 始终作为通过 `user.id` 获取的不透明数据块存在，你可能没问题。但如果它开始成为业务数据的主要存储位置，权衡会迅速改变。

产品问："纽约有多少用户？"

你写出：

```sql
SELECT count(*) FROM users WHERE profile->>'location' = 'New York';
```

PostgreSQL 执行全表扫描。每一行都会被扫描。

于是你添加了 GIN 索引。这可能还是可以接受的。有时确实如此。但你现在要为一个本应是第一类关系数据的字段支付真实复杂度和存储成本，而它从未成为真正的列。

### 第一年：模式漂移

同一列中存在三个版本的数据。

*   行1: `{"city": "NYC"}`
*   行1000: `{"location": "NYC"}`
*   行5000: `{"address": {"city": "New York"}}`

你的应用代码现在变成这样：

```javascript
const city = user.location || user.city || user.address?.city || "Unknown";
```

你并没有删除模式。你只是把验证和一致性检查从数据库转移到了分散的应用代码中。

## 实际使用 JSONB 的场景

JSONB 有其合理的使用场景。很多时候它完全合适，有时甚至是最佳选择。

关键的区分点并非“结构化数据好，JSON 差”。更接近的判断标准是：

- 数据是否主要通过稳定的主键整体获取？
- 键在不同提供方、版本、租户或时间上是否有显著差异？
- 你是查询几个已知字段，还是每次迭代都发明新的路径查询？
- 应用是否主动管理版本和验证，还是只是随意处理？

### 合法的 JSONB 使用场景

1.  **Webhook 载荷**：你从 Stripe、Slack 或 GitHub 接收数据。你对模式无任何控制权。你可能永远不会查询它。你只需要存储用于调试或重放。**JSONB 非常适合。**

2.  **日志与事件流**：应用程序日志、审计跟踪、错误上下文。这些是写密集型的，很少按特定字段查询，通常批量分析或导出到分析平台。**JSONB 在这里适用。**

3.  **用户偏好与设置**：包含 100+ 布尔标志的设置对象，大多数为 false，你总是按用户 ID 整体获取整个对象。你不会运行 `WHERE preferences->>'theme' = 'dark'`。**JSONB 可行。**

4.  **LLM 提供方/模型配置**：这是现代最清晰的例子之一。OpenAI、Anthropic、Gemini、开源本地模型和供应商特定网关都暴露重叠但不同的参数。即使在单一提供方内，模型能力和选项名称也在演变。一个 JSONB 配置块通常比假装 `temperature`、`top_p`、`reasoning_effort`、`json_schema`、`tool_choice` 和二十多个旋钮都应作为通用列更诚实。**JSONB 在这里往往是正确的抽象。**

5.  **API 响应缓存**：你缓存整个 API 响应。数据库只是更快的 Redis。你按缓存键获取，从不按嵌套属性。**JSONB 适用。**

6.  **事件溯源**：你存储不可变的事件载荷。你的查询始终是“给我聚合 X 的所有事件”，按时间排序。你从不在事件属性上运行 `WHERE` 子句。**JSONB 适用。**

7.  **扩展性接口**：集成、插件设置、租户特定覆盖、市场元数据、提供方能力或“额外”字段，你明确预期其形状会因子类型而异。**JSONB 可以是正确的契约，而非妥协。**

经验法则：如果应用通过已知键获取文档并理解如何验证/版本化它，JSONB 可能非常出色。如果业务持续对嵌套键提出关系型问题，这些字段正试图成为列。

## 最佳模式往往是混合型

许多成熟系统采用这种模式：

```sql
CREATE TABLE llm_requests (
  id UUID PRIMARY KEY,
  provider TEXT NOT NULL,
  model TEXT NOT NULL,
  status TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  config JSONB NOT NULL
);
```

这通常比两种极端更好。

- `provider`、`model`、`status` 和 `created_at` 是第一类列，因为你需要过滤、连接、聚合和索引它们。
- `config` 保持为 JSONB，因为确切的选项界面是模型特定的、提供方特定的，并且很可能演变。

这并不是“未规范化”。这是在正确的地方划清界限。

### 在大规模场景下：对象版本控制 > 规范化

有趣的地方在这里。在足够大的规模下，"正确"的解决方案并不是规范化——而是对象版本控制。

如果你有数十亿行数据且需要频繁模式演进，迁移字段会变得昂贵。Stripe、GitHub 和 Netflix 这类公司并不会完全规范化。他们采用：

```sql
CREATE TABLE entities (
  id UUID PRIMARY KEY,
  version INT NOT NULL,
  data JSONB NOT NULL
);
```

你的应用程序知道如何读取 `version: 1`、`version: 2`、`version: 3`。新增字段不需要数据库迁移。代码处理向后兼容性。

这是一个架构决策，而非懒惰。它用应用复杂度换取数据库复杂度。当文档本身具有天然版本特性且应用程序是权威解释者时，这往往正是正确的权衡。

失败的模式不是"使用 JSONB"。失败的模式是使用 JSONB 时没有版本控制、验证规则、晋升机制，或没有明确区分文档数据和关系型数据的边界。

## 真正重要的问题

在添加 JSONB 字段前，先问：

1. 我们会频繁在 `WHERE`、`JOIN`、`GROUP BY` 或 `ORDER BY` 中查询嵌套字段吗？
2. 这个模式是受我们控制的，还是外部定义且易变的？
3. 记录的结构是否设计为异构的？
4. 我们有应用层的验证和版本控制吗？
5. 哪些字段可能会演变为操作维度？

如果第1个问题的答案是"是的，经常如此"，这强烈指向传统列。

如果第2和第3个问题的答案是"是的"，JSONB 很可能确实在为你工作。

## 逃离陷阱

如果你已经陷入这个坑，立即停止继续挖掘。

1. 审计：运行 `jsonb_object_keys` 检查实际的结构漂移，而不是你假设的结构。
2. 提升：识别你最常过滤、连接、排序或报告的字段。将这些字段转为真实列。
3. 验证：为 JSONB 中剩余的数据添加应用层或数据库层的验证。
4. 版本控制：如果数据块是真实的领域数据，显式添加版本控制。
5. 修剪：在提升字段为真实列后，从数据块中移除重复的键。

不要告诉自己每个数据块都必须规范化。也不要告诉自己具有永久业务语义的数据块是"临时的"。

当文档确实是文档结构时，JSONB 非常强大。当它只是披着文档外衣的关系型模式时，就非常危险。
