PostgreSQLはJSONBを追加し、厳格なスキーマを事前に定義せずに半構造化データを保存できるようにした。そのアイデアは妥当だった。データの形が本当にわからない場合や、従来のカラムでは追いつかないほど頻繁に変化する場合があるからだ。

これは重要だ。JSONBは間違いではない。多くのシステムでは、問題領域を最もクリーンに表現できる方法だ。サードパーティのWebhookペイロード、バージョン管理されたイベントボディ、フィーチャーフラグ、LLM設定オブジェクトなど、プロバイダやモデルごとに微妙に異なり常に変化するオプションセットを扱う場合、すべてを第一級のカラムに押し込むのは、かえって不便で役に立たない。

問題は、JSONBがスキーマの決定を先延ばしにしていることを認めずに先延ばしにする最も簡単な方法でもあることだ。意図と実装の間のどこかで、それは「後で部屋を掃除する」のデータベース版になった。半年前に手を付けた一時的な解決策？ それはまだそこにあり、今や本番環境がそれに依存している。

私は同じパターンを何度も見てきた。チームが要件に確信が持てずにJSONBカラムを追加する。事態が落ち着いたら正規化しようと自分たちに約束する。3年後、そのカラムにはユーザープロファイルであるはずの40種類の異なるバージョンが含まれ、15のサービスがそれぞれ内部の内容について異なる仮定をしてクエリを発行している。

技術的負債はJSONB自体ではない。自分が構築していると思っていたものと実際に構築したもののギャップ、すなわち文書化されていないスキーマオンリードシステムだ。

## よくあるパターン

機能を追加していて、ユーザーが `twitter_handle` を必要とするのか `bluesky_handle` なのか、あるいはまったく別のものなのか確信が持てない。スキーマを熟考する代わりに、こうする：

```sql
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  profile JSONB
);
```

これで動く。機能をリリースし、次の機能、さらに次の機能へと進む。JSONBカラムは静かにバックグラウンドで成長する。

これが分かれ道だ。`profile` が `user.id` で取得される不透明なブロブのままなら、おそらく問題ない。しかし、ビジネスデータが存在する主要な場所になり始めると、トレードオフは急速に変化する。

プロダクトが尋ねる：*「ニューヨークのユーザーは何人？」*

あなたはこう書く：

```sql
SELECT count(*) FROM users WHERE profile->>'location' = 'New York';
```

Postgresは全テーブルスキャンを実行する。すべての行に対して。

そこでGINインデックスを追加する。それでも許容できる場合もある。時にはそれで十分だ。しかし今や、第一級のリレーショナルデータのように振る舞うフィールドが決して第一級のカラムにならなかったために、実際の複雑性とストレージコストを支払っている。

### 1年目：スキーマの漂流

同じカラムに3種類のデータが存在する。

*   行1: `{"city": "NYC"}`
*   行1000: `{"location": "NYC"}`
*   行5000: `{"address": {"city": "New York"}}`

アプリケーションコードはこうなる：

```javascript
const city = user.location || user.city || user.address?.city || "Unknown";
```

スキーマを削除したわけではない。バリデーションと一貫性チェックをデータベースから散らばったアプリケーションコードに移しただけだ。

## JSONBを実際に使うべき時

JSONBには有効なユースケースがある。多くの場合、まったく問題なく、時には最良の選択肢となる。

重要な区別は「構造化は良く、JSONは悪い」ではない。むしろ次のようなものだ：

- データは主に安定した主キーで全体として取得されるか？
- キーはプロバイダ、バージョン、テナント、時間によって実質的に異なるか？
- 既知のフィールドをいくつかクエリしているのか、それともスプリントごとに新しいパスクエリを発明しているのか？
- アプリケーションは意図的にバージョン管理とバリデーションを所有しているのか、それとも行き当たりばったりなのか？

### 正当なJSONBのユースケース

1.  **Webhookペイロード**: Stripe、Slack、GitHubからデータを受け取る。スキーマをまったく制御できない。クエリすることはまずない。デバッグやリプレイのために保存するだけ。**JSONBに最適。**

2.  **ログとイベントストリーム**: アプリケーションログ、監査証跡、エラーコンテキスト。これらは書き込みが多く、特定のフィールドでクエリされることは稀で、バルク分析や分析プラットフォームへのエクスポートが一般的。**JSONBで問題ない。**

3.  **ユーザー設定とプリファレンス**: 100以上のブールフラグがあり、ほとんどがfalseで、常にユーザーIDでブロブ全体を取得する設定オブジェクト。`WHERE preferences->>'theme' = 'dark'` のようなクエリは実行しない。**JSONBで機能する。**

4.  **LLMプロバイダ/モデル設定**: これは最も明確な現代の例の一つだ。OpenAI、Anthropic、Gemini、オープンウェイトのローカルモデル、ベンダー固有のゲートウェイは、重複しつつも異なるパラメータを公開している。単一プロバイダ内でも、モデルの機能やオプション名は進化する。`temperature`、`top_p`、`reasoning_effort`、`json_schema`、`tool_choice`、その他20ものノブをすべて普遍的なカラムにするふりをするよりも、JSONB設定ブロブの方がはるかに正直な場合が多い。**JSONBはここではしばしば正しい抽象化である。**

5.  **APIレスポンスキャッシュ**: APIレスポンス全体をキャッシュしている。データベースは高速なRedisに過ぎない。キャッシュキーで取得し、ネストされたプロパティでは決して取得しない。**JSONBは適切。**

6.  **イベントソーシング**: 不変のイベントペイロードを保存している。クエリは常に「集約Xの全イベントを時間順で取得」であり、イベントプロパティにWHERE句を実行することはない。**JSONBが適合する。**

7.  **拡張性サーフェス**: 統合、プラグイン設定、テナントごとのオーバーライド、マーケットプレイスメタデータ、プロバイダ機能、またはサブタイプごとに形状が異なることを明示的に想定している「追加」フィールド。**JSONBは妥協ではなく、適切な契約となり得る。**

経験則: アプリケーションが既知のキーでドキュメントを取得し、その検証・バージョン管理方法を理解している場合、JSONBは優れた選択肢となる。ビジネス側がネストされたキーについてリレーショナルな質問を繰り返すなら、それらのフィールドはカラムになろうとしている。

## 最良のパターンは多くの場合ハイブリッド

成熟したシステムの多くはここに落ち着く:

```sql
CREATE TABLE llm_requests (
  id UUID PRIMARY KEY,
  provider TEXT NOT NULL,
  model TEXT NOT NULL,
  status TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  config JSONB NOT NULL
);
```

これは通常、どちらかの極端な方法よりも優れている。

- `provider`、`model`、`status`、`created_at`は第一級のカラムである。なぜなら、フィルタリング、結合、集約、インデックス作成に使用するからだ。
- `config`はJSONBのままにする。正確なオプションの表面はモデル固有、プロバイダ固有であり、進化する可能性が高いからだ。

それは「正規化に失敗している」わけではない。適切な場所に線を引いているのだ。

### 大規模環境では: オブジェクトバージョニング > 正規化

ここからが面白いところだ。十分に大規模な環境では、「正しい」解決策は正規化ではなく、オブジェクトバージョニングである。

何十億もの行と頻繁なスキーマ進化がある場合、カラムのマイグレーションは高コストになる。Stripe、GitHub、Netflixのような企業はすべてを正規化していない。代わりに:

```sql
CREATE TABLE entities (
  id UUID PRIMARY KEY,
  version INT NOT NULL,
  data JSONB NOT NULL
);
```

アプリケーションは `version: 1`、`version: 2`、`version: 3` を読み取る方法を知っている。新しいフィールドのためにデータベースのマイグレーションは不要。コードが後方互換性を処理する。

これはアーキテクチャ上の判断であり、怠惰ではない。データベースの複雑さをアプリケーションの複雑さと引き換えにしている。時にはそれがまさに正しいトレードオフであり、特にドキュメントが自然にバージョン管理され、アプリが正規のインタプリタである場合にそうだ。

失敗のモードは「JSONBを使うこと」ではない。失敗のモードは、バージョン管理、バリデーション、昇格ルール、ドキュメントデータとリレーショナルデータの明確な境界なしにJSONBを使うことだ。

## 本当に重要な質問

JSONBカラムを追加する前に、次のことを問いかけてほしい。

1.  `WHERE`、`JOIN`、`GROUP BY`、`ORDER BY` でネストされたフィールドを定期的にクエリすることになるか？
2.  このスキーマを自分たちで制御しているのか、それとも外部で定義され変動しやすいものか？
3.  形状は意図的にレコード間で異種混合になっているか？
4.  アプリケーションレベルのバリデーションとバージョン管理はあるか？
5.  どのフィールドが後で運用上のディメンションになりそうか？

#1 の答えが「はい、頻繁に」なら、それはカラムを使うべき強いシグナルだ。

#2 と #3 の答えが「はい」なら、JSONBはおそらく実際に役立っている。

## 罠からの脱出

すでにこの穴にいるなら、掘るのをやめろ。

1.  監査：`jsonb_object_keys` を実行し、想定している形状ではなく、実際の形状のドリフトを検査する。
2.  昇格：最も頻繁にフィルタ、結合、ソート、レポートに使用するフィールドを特定し、それらを実際のカラムにする。
3.  検証：JSONBに残ったものに対して、アプリケーションまたはデータベースレベルのバリデーションを追加する。
4.  バージョン管理：ブロブが実際のドメインデータである場合、明示的にバージョン管理する。
5.  トリミング：昇格したカラムが確立されたら、ブロブから重複したキーを削除する。

すべてのブロブを正規化しなければならない、と思い込んではいけない。また、永続的なビジネスセマンティクスを持つブロブを「一時的なもの」と決めつけてもいけない。

JSONBは、ドキュメントが真にドキュメント形状である場合に威力を発揮する。しかし、リレーショナルスキーマが偽の口ひげを生やしているだけの場合は危険だ。
