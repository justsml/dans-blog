PostgreSQLはJSONBを追加して、柔軟なスキーマを事前に定義せずに半構造化データを保存できるようにしました。この考え方は妥当です。実際にデータがどのような形になるか分からない場合や、伝統的なカラムで意味があるほど頻繁に変化しない場合もあります。  

重要なのはJSONBが間違いではないということです。多くのシステムにおいて、これは問題領域を最も明確に表現する方法です。サードパーティのウェブフックペイロードやバージョン付きイベントボディ、機能フラグ、LLMの構成オブジェクト（各プロバイダやモデルがわずかに異なり、常に変化するオプションセットを持つ場合）を保存する際、すべてをファーストクラスのカラムに押し込むことは、逆に面倒で役に立たない場合があります。  

問題はJSONBがスキーマの決定を先延ばしにする最も簡単な方法であることです。意図と実装の間で、これは「あとで部屋を掃除する」というデータベース的な表現になってしまいました。6か月前に手っ取り早く導入した一時的な解決策は、今や本番環境に依存されています。  

私は常に同じパターンを目撃します。チームは要件が不確実なためJSONBカラムを追加します。落ち着いたら正規化すると自分に約束します。3年後には、そのカラムに40種類ものバージョンが詰まっており、15のサービスがそれぞれ異なる仮定に基づいてクエリを実行しています。  

技術的負債はJSONB自体にありません。あなたが建設したはずのスキーマと、実際に構築した読み込み時にスキーマを定義しないシステムのギャップにあります。  

## 通常何が起こるか  

新機能を追加していて、ユーザーが`twitter_handle`や`bluesky_handle`、あるいはそれ以外の何かが必要かどうか分からないとします。スキーマを深く考えずに次のようにします：  

```sql
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  profile JSONB
);
```  

これは動きます。機能をリリースし、次のタスクに進みます。JSONBカラムは静かに背景で増殖していきます。  

これが分岐点です。`profile`が`user.id`でフェッチされる不透明なバイト列のままであれば、おそらく問題ありません。それがビジネスデータの主要な保管場所になるようになると、トレードオフは急激に変化します。  

プロダクトが尋ねます：「ニューヨークにユーザーはどれだけいますか？」

```sql
SELECT count(*) FROM users WHERE profile->>'location' = 'New York';
```

PostgreSQLはフルテーブルスキャンを実行します。すべての行をスキャンします。

GINインデックスを追加します。まだ許容できるかもしれません。場合によってはそうですが、今や「第一級のリレーショナルデータ」として振る舞うフィールドが、第一級のカラムとして定義されなかったために、実際の複雑さとストレージコストを支払うことになります。

### 年1: スキーマのドリフト

同じカラムに3つのデータバージョンが存在します。

*   行1: `{"city": "NYC"}`
*   行1000: `{"location": "NYC"}`
*   行5000: `{"address": {"city": "New York"}}`

アプリケーションコードはこうなります：

```javascript
const city = user.location || user.city || user.address?.city || "Unknown";
```

スキーマを削除しなかった。単に検証と一貫性チェックをデータベースから散らばったアプリケーションコードに移動しただけだ。

## JSONBを実際に使用すべきとき

JSONBには有効な使用ケースがあります。多くの場合、JSONBは完全に適切であり、時には利用可能な最良の選択肢です。

重要な区別は「構造化データが良い、JSONが悪い」というものではありません。より正確な視点は以下の通りです：

- データは安定したプライマリキーで全体として取得されることが多いですか？
- キーはプロバイダー、バージョン、テナント、時間によって大きく異なりますか？
- 特定のフィールドではなく、スプリントごとに新しいパスクエリを考案していますか？
- アプリケーションはバージョン管理と検証を意図的に所有していますか？それとも適当に扱っていますか？

### JSONBの正当な使用ケース

1.  **ウェブフックペイロード**：Stripe、Slack、GitHubからデータを受信します。スキーマの制御はできません。クエリもほとんど行いません。デバッグやリプレイのために保存するだけです。**JSONBに最適です。**

2.  **ログ & イベントストリーム**：アプリケーションログ、監査トレイル、エラーコンテキスト。これらは書き込みが頻繁で、特定のフィールドでクエリされることが少なく、通常はバルク処理や分析プラットフォームへのエクスポートのために分析されます。**JSONBはここでも問題ありません。**

3.  **ユーザーの好み & 設定**：100以上のブール値フラグを持つ設定オブジェクトで、ほとんどの値がfalseで、ユーザーIDで常に全体を取得します。`WHERE preferences->>'theme' = 'dark'` のようなクエリは実行しません。**JSONBは適しています。**

4.  **LLMプロバイダー/モデル設定**：これは現代で最も明確な例の一つです。OpenAI、Anthropic、Gemini、オープンウェイトローカルモデル、ベンダーオリジナルのゲートウェイはすべて、重複しながらも異なるパラメータを公開します。たとえ同じプロバイダー内でも、モデルの機能やオプション名は進化します。`temperature`、`top_p`、`reasoning_effort`、`json_schema`、`tool_choice`、そして20以上のその他のノブをすべてユニバーサルなカラムとして扱うことを装うよりも、JSONB設定ボルブの方がはるかに正直です。**JSONBはここではしばしば正しい抽象化です。**

5.  **APIレスポンスのキャッシュ**：完全なAPIレスポンスをキャッシュしています。データベースは単に高速なRedisです。キャッシュキーで取得し、ネストされたプロパティでクエリしません。**JSONBは適切です。**

6.  **イベントソーシング**：不変のイベントペイロードを保存しています。クエリは常に「アグリゲートXのすべてのイベントを時系列順に取得する」ものです。イベントプロパティで`WHERE`句を実行しません。**JSONBは適合します。**

7.  **拡張性インターフェース**：統合、プラグイン設定、テナントごとのオーバーライド、マーケットプレイスメタデータ、プロバイダの機能、またはサブタイプごとに形状が異なることを明示的に想定する「追加フィールド」など。**JSONBは妥協ではなく、正しい契約となる可能性があります。**

実用ルール：アプリケーションが既知のキーでドキュメントを取得し、検証/バージョン管理方法を理解できる場合、JSONBは優れた選択肢です。ビジネスがネストされたキーに関する関係性を問うようになるたびに、そのフィールドはカラムとして独立する準備ができています。

## 最適なパターンは多くの場合ハイブリッド

多くの成熟したシステムがこの形に落ち着きます：

```sql
CREATE TABLE llm_requests (
  id UUID PRIMARY KEY,
  provider TEXT NOT NULL,
  model TEXT NOT NULL,
  status TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  config JSONB NOT NULL
);
```

これは極端などちらよりもバランスが取れています。

- `provider`、`model`、`status`、`created_at` は1等市民カラムとして扱われます。フィルタリング、結合、集約、インデックス作成を頻繁に行うためです。
- `config` はJSONBのままです。オプションの正確な構成はモデル固有・プロバイダ固有であり、進化し続けるからです。

これは「正規化されていない」という失敗ではありません。これは適切な境界を引くことです。

### 大規模なケース：オブジェクトバージョニング＞正規化

ここで興味深い点があります。十分に大規模なケースでは、「正しい」解決策は正規化ではなく、オブジェクトバージョニングです。

数十億行のデータと頻繁なスキーマ進化がある場合、カラムのマイグレーションはコストが高くなります。StripeやGitHub、Netflixのような企業はすべてを正規化しません。代わりに：

```sql
CREATE TABLE entities (
  id UUID PRIMARY KEY,
  version INT NOT NULL,
  data JSONB NOT NULL
);
```

アプリケーションは`version: 1`、`version: 2`、`version: 3`を読み込む方法を把握しています。新規フィールドのためのデータベースマイグレーションは不要です。コードがバックワード互換性を処理します。

これは怠惰ではなく、アーキテクチャ上の決定です。データベースの複雑さをアプリケーションの複雑さとトレードしています。ドキュメントが自然にバージョン化されており、アプリケーションが正規の解釈者である場合、これはまさに正しいトレードオフです。

失敗パターンは「JSONBを使用すること」ではありません。失敗パターンは、JSONBを使用する際にバージョニング、検証、プロモーションルール、ドキュメントデータとリレーショナルデータの明確な境界を持たないことです。

## 本当に重要な質問

JSONBカラムを追加する前に以下の質問を検討してください：

1.  `WHERE`、`JOIN`、`GROUP BY`、`ORDER BY`でネストされたフィールドを頻繁にクエリしますか？
2.  このスキーマは我々が管理していますか？それとも外部定義で変動性が高いですか？
3.  レコード間で意図的に構造が異なりますか？
4.  アプリケーションレベルでの検証とバージョニングを持っていますか？
5.  どのフィールドが後に運用次元として扱われる可能性がありますか？

1番目の質問に「はい、頻繁に」と答えた場合、それはカラムを使用すべき強いシグナルです。

2番目と3番目の質問に「はい」と答えた場合、JSONBはおそらくあなたにとって実際的な仕事をしています。

## 親和性の罠からの脱出

すでにこの穴に足を取られている場合は、掘り下げを止めてください。

1. 点検: `jsonb_object_keys`を実行し、仮定している構造ではなく実際に存在する構造の変化を検査してください。
2. 昇格: フィルタリング、結合、ソート、レポートで最も頻繁に使用するフィールドを特定してください。それらを実際のカラムに昇格させましょう。
3. 検証: JSONBに残っているデータに対してアプリケーションまたはデータベースレベルの検証を追加してください。
4. バージョン管理: ブロブが本当にドメインデータである場合、明示的にバージョン管理を行ってください。
5. トリム: 昇格したカラムが確立された後、ブロブから重複したキーを削除してください。

すべてのブロブを正規化しなければならないとは思わないでください。また、永久的なビジネスセマンティクスを持つブロブを「一時的なもの」と思わないでください。

JSONBはドキュメントが本当にドキュメント型である場合に最適です。リレーショナルスキーマが偽のヒゲをつけて装っている場合、それは危険です。
