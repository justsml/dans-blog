PostgreSQLはJSONBを追加し、事前に厳格なスキーマを定義せずに半構造化データを保存できるようにした。この考え方は妥当だ：時にはデータがどのような形状になるか本当に分からない場合や、従来の列では追いつかないほど頻繁に変更される場合がある。

重要なのは、JSONBが間違いではないということだ。多くのシステムにおいて、それは問題空間を最もクリーンに表現する手段である。サードパーティのWebhookペイロード、バージョン管理されたイベントボディ、フィーチャーフラグ、あるいは各プロバイダやモデルがわずかに異なり絶えず変化するオプションセットを公開するLLM設定オブジェクトを保存する場合、すべてを個別の列に強制するのは、役立つよりも不格好になることが多い。

問題は、JSONBがスキーマの決定を先送りしていることを自覚せずに、それを先送りする最も手軽な手段でもあることだ。意図と実装の間に挟まれたどこかで、それはデータベースにおける「後で片付ける」の代名詞になった。6ヶ月前に飛びついたその一時的な解決策は？ まだそこにあり、今では本番環境がそれに依存している。

僕は同じパターンを何度も目にする。チームは要件が不明確なためJSONB列を追加する。状況が落ち着いたら正規化する自分たちに約束する。3年後、その列には本来ユーザープロファイルだったものの40種類の異なるバージョンが含まれており、それぞれ内部の構造について異なる前提を置く15のサービスからクエリされている。

技術的負債そのものはJSONB自体ではない。あなたが自分自身に「こう作る」と言い聞かせていたものと、実際に構築したものの間のギャップだ：ドキュメント化されていないスキーマオンリード（読み込み時スキーマ）システム。

## 通常どうなるか

機能を追加していて、ユーザーに `twitter_handle` が必要なのか `bluesky_handle` が必要なのか、あるいは全く別のものが必要なのか分からない場合がある。スキーマを十分に検討する代わりに、あなたはこうする：

```sql
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  profile JSONB
);
```

動作する。機能をリリースし、次の機能に進み、さらに次の機能に進む。JSONB列は背景でひっそりと蓄積されていく。

ここが分岐点だ。`profile` が `user.id` でフェッチされる透過性のないJSONオブジェクトのままなら、おそらく大丈夫だ。しかしそれがビジネスデータの主要な保管場所になり始めると、トレードオフは急速に変化する。

プロダクトチームが尋ねる：*「ニューヨーク在住のユーザーは何人いますか？」*

あなたはこう書く：

```sql
SELECT count(*) FROM users WHERE profile->>'location' = 'New York';
```

Postgresはフルテーブルスキャンを実行する。すべての行を。

そこでGINインデックスを追加する。まだ許容範囲かもしれない。場合によってはそれで済む。しかし今や、本来は第一級のリレーショナルデータとして振る舞うべきフィールドが第一級の列にならなかったため、実際の複雑さとストレージコストを支払うことになる。

### 1年目：スキーマドリフト

同じ列に3つのバージョンのデータが混在する。

*   行 1: `{"city": "NYC"}`
*   行 1000: `{"location": "NYC"}`
*   行 5000: `{"address": {"city": "New York"}}`

アプリケーションコードはこうなる：

```javascript
const city = user.location || user.city || user.address?.city || "Unknown";
```

スキーマを削除したわけではない。単に、データベースで行うべき検証と整合性チェックを、散在するアプリケーションコードへ移しただけだ。

## JSONBを本当に使うべきはいつか

JSONBには正当なユースケースが存在する。多くの場合、問題なく使え、時には最善の選択肢となる。

重要なのは「構造化＝良、JSON＝悪」という二分法ではない。むしろ、以下のような基準で判断すべきだ：

- データは安定した主キーで全体として取得されることが多いか？
- キーはプロバイダー、バージョン、テナント、あるいは時間の経過とともに大きく変動するか？
- 数多くの既知のフィールドを照会しているか、それとも毎スプリントで新しいパスクエリを考案しているか？
- アプリケーションが意図的にバージョン管理と検証を担っているか、それとも場当たり的にやっているか？

### JSONBの正当なユースケース

1.  **Webhookペイロード**: Stripe、Slack、GitHubなどからデータを受信する。スキーマには一切関与できない。クエリすることもないかもしれない。デバッグやリプレイのために保存する必要があるだけだ。**JSONBに最適。**

2.  **ログとイベントストリーム**: アプリケーションログ、監査証跡、エラーコンテキスト。これらは書き込みが主で、特定フィールドでのクエリは稀であり、バッチ処理で分析するか分析プラットフォームへエクスポートされることが多い。**JSONBで問題ない。**

3.  **ユーザー設定とプリファレンス**: 100以上のブールフラグを持つ設定オブジェクトで、大半はfalseであり、ユーザーIDで常に全体を取得する。`WHERE preferences->>'theme' = 'dark'`のようなクエリは実行しない。**JSONBで十分機能する。**

4.  **LLMプロバイダー／モデル設定**: 現代における最も明確な例の一つだ。OpenAI、Anthropic、Gemini、オープンウェイトのローカルモデル、ベンダー固有のゲートウェイはすべて、重複しつつも異なるパラメーターを公開している。単一プロバイダー内でも、モデルの機能やオプション名は進化し続ける。`temperature`、`top_p`、`reasoning_effort`、`json_schema`、`tool_choice`、その他二十個ほどのノブがすべてユニバーサルな列であるかのように振る舞うよりも、JSONB設定ブロブの方がはるかにありのままを反映している。**JSONBはここで適切な抽象化となることが多い。**

5.  **APIレスポンスキャッシュ**: APIレスポンス全体をキャッシュしている。データベースは単なる高速なRedisだ。キャッシュキーで取得し、ネストされたプロパティで取得することはない。**JSONBが適切。**

6.  **イベントソーシング**: 不変のイベントペイロードを保存している。クエリは常に「集約Xの全イベントを時間順に返せ」だ。イベントプロパティに対して`WHERE`句を実行することはない。**JSONBに適合する。**

7. **拡張可能なインターフェース**: 外部連携、プラグイン設定、テナントごとの上書き設定、マーケットプレイスのメタデータ、プロバイダーの機能、あるいはサブタイプによって構造が明確に変わることを前提とした「追加」フィールド。**JSONB は妥協ではなく、適切なデータ契約となり得る。**

目安：アプリケーションが既知のキーでドキュメントを取得し、その検証やバージョン管理の仕組みを理解しているなら、JSONB は非常に有効だ。一方、ビジネス側がネストされたキーに対してリレーショナルなクエリを繰り返し要求する場合、そのフィールドは列として扱われるべきなのだ。

## 最適なパターンは往々にしてハイブリッドである

成熟したシステムでは、多くの場合ここで落ち着く。

```sql
CREATE TABLE llm_requests (
  id UUID PRIMARY KEY,
  provider TEXT NOT NULL,
  model TEXT NOT NULL,
  status TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  config JSONB NOT NULL
);
```

これは通常、どちらか片寄ったアプローチよりも優れている。

- `provider`、`model`、`status`、`created_at` は第一級のカラムだ。これらはフィルタリング、結合、集計、インデックスの対象になるためである。
- `config` は JSONB のままにする。正確なオプションの構造はモデル固有であり、プロバイダー固有であり、かつ進化し続ける可能性が高いからだ。

これは「正規化に失敗した」のではない。適切な境界線を引きただけだ。

### 大規模スケール時：オブジェクトバージョン管理 > 正規化

ここからが面白い。十分に大規模なスケールでは、「正解」は正規化ではない。それはオブジェクトバージョン管理だ。

数十億行のデータがあり、スキーマの進化が頻繁に起こる場合、カラムのマイグレーションは高コストになる。Stripe、GitHub、Netflix といった企業はすべてを正規化しない。その代わりに：

```sql
CREATE TABLE entities (
  id UUID PRIMARY KEY,
  version INT NOT NULL,
  data JSONB NOT NULL
);
```

アプリケーションが `version: 1`、`version: 2`、`version: 3` の解釈方法を知っていれば十分だ。新しいフィールドのために DB マイグレーションを実行する必要はない。後方互換性はコード側で担保する。

これは怠慢ではなく、明確なアーキテクチャ上の判断だ。DBの複雑さをアプリケーションの複雑さに転嫁している。ドキュメントが本質的にバージョン管理されており、アプリケーションが唯一の正規解釈系である場合、このトレードオフはまさに正解になり得る。

失敗する原因は「JSONBを使っていること」ではない。失敗するのは、バージョン管理、バリデーション、カラム昇格ルールを欠いたままJSONBを使い、ドキュメントデータとリレーショナルデータの境界を明確にしない場合だ。

## 本当に問うべき質問

JSONBカラムを追加する前に、自問せよ：

1.  `WHERE`、`JOIN`、`GROUP BY`、`ORDER BY` でネストされたフィールドを定期的にクエリする必要があるか？
2.  このスキーマを我々が管理しているか、それとも外部定義で揮発性が高いか？
3.  レコード間で意図的に形状が異なっているか？
4.  アプリケーションレベルでのバリデーションとバージョン管理を持っているか？
5.  どのフィールドが将来的に運用上のディメンションになる可能性があるか？

#1 の答えが「常に yes」なら、それはカラム化すべき強いシグナルだ。

#2 と #3 の答えが「yes」なら、JSONB はおそらく本当にあなたの役に立っている。

## 罠から抜け出す

すでにこの穴に落ちているなら、掘り続けるな。

1. 監査：`jsonb_object_keys`を実行し、思い込んでいる形状ではなく、実際の構造ドリフトを確認せよ。
2. 昇格：フィルタ、結合、ソート、レポートで最も頻繁に使用するフィールドを特定し、それらを正式なカラムにせよ。
3. 検証：JSONBに残すフィールドについては、アプリケーションまたはデータベースレベルで検証を追加せよ。
4. バージョン管理：そのデータブロックが本物のドメインデータであれば、明示的にバージョン管理を行え。
5. 削減：昇格したカラムが確立されたら、データブロックから重複するキーを削除せよ。

全てのデータブロックを正規化しなければならないと自分に言い聞かせるな。また、永続的なビジネス意味を持つデータを「一時的」と自己正当化してはならない。

JSONBは、データが本当にドキュメント形状である場合に素晴らしい。偽のヒゲをかぶったリレーショナルスキーマである場合は危険だ。
