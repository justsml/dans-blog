PostgreSQL は、事前に厳密なスキーマを定義せずに半構造化データを保存できるように JSONB を導入しました。考え方は正当です。データの形が分からない、あるいは頻繁に変わるため従来のカラムで表現できない場合があるからです。

この点が重要なのは、JSONB が単なるミスではないということです。多くのシステムで、問題領域を最もクリーンに表現できる手段となっています。サードパーティの webhook ペイロード、バージョン管理されたイベント本体、機能フラグ、あるいは各プロバイダーやモデルが微妙に異なるオプションセットを持つ LLM 設定オブジェクトを保存する場合、すべてを第一級カラムに詰め込もうとすると、むしろ面倒になることがあります。

しかし JSONB は、スキーマ決定を先送りする最も手軽な方法でもあり、先送りしていることを認めたくないという心理に付け込まれます。意図と実装の間で、データベース上の「あとで部屋を片付ける」的な状態になってしまったのです。6 ヶ月前に一時的な解決策として導入したものが、今も残り、プロダクションがそれに依存している、という状況です。

同じパターンを繰り返し目にします。チームは要件が不確かなので JSONB カラムを追加し、要件が落ち着いたら正規化すると自分たちに約束します。3 年後、そのカラムには本来はユーザープロファイルであるべきものが 40 種類も混在し、15 のサービスがそれぞれ異なる前提で中身を参照しています。

技術的負債の根源は JSONB そのものではありません。自分たちが「こう作る」と言い聞かせたものと、実際に作ってしまった「ドキュメント化されていないスキーマオンリード」システムとの間のギャップです。

## 通常起こること

機能を追加しようとして、ユーザーが `twitter_handle` が必要か `bluesky_handle` が必要か、あるいは全く別のものが必要か分からない場合、スキーマを熟考する代わりに次のように書きます。

```sql
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  profile JSONB
);
```

これで動きます。機能をリリースし、次の機能へ、さらに次の機能へと進むうちに、JSONB カラムは静かに肥大化していきます。

ここで分岐が生まれます。`profile` が `user.id` で取得する不透明なブロブのままであれば、問題は少ないでしょう。逆にビジネスデータの主要な格納場所になり始めると、トレードオフは急速に変化します。

プロダクト側からの質問: *「ニューヨークにいるユーザーは何人いる？」*

あなたは次のように書く:

```sql
SELECT count(*) FROM users WHERE profile->>'location' = 'New York';
```

Postgres は全テーブル走査を行う。すべての行を対象にする。

そこで GIN インデックスを追加する。まだ許容範囲かもしれない。ケースによってはそうだ。しかし、ファーストクラスのリレーショナルデータとして振る舞うべきフィールドをファーストクラス列にしなかったため、実際の複雑性とストレージコストが発生する。

### Year 1: Schema Drift

同じカラムに 3 種類のデータが混在している。

*   行 1: `{"city": "NYC"}`
*   行 1000: `{"location": "NYC"}`
*   行 5000: `{"address": {"city": "New York"}}`

アプリケーションコードは次のようになる:

```javascript
const city = user.location || user.city || user.address?.city || "Unknown";
```

スキーマを削除したわけではない。検証や一貫性チェックをデータベースからアプリケーションの散在コードへ移しただけだ。

## When to Actually Use JSONB

JSONB には正当な使用ケースがあります。多くの場合、全く問題なく、時には利用可能な中で最適な選択肢となります。

重要な区別は「構造化は良い、JSON は悪い」ではなく、次のような観点です。

- データは主キーで安定して取得されることが多いか？
- キーはプロバイダー、バージョン、テナント、または時間によって実質的に変化するか？
- いくつかの既知フィールドだけをクエリするか、あるいはスプリントごとに新しいパスクエリを考案しているか？
- アプリケーションがバージョン管理とバリデーションを意図的に所有しているか、あるいはただ手当たり次第にやっているだけか？

### 正当な JSONB の使用例

1.  **Webhook ペイロード**: Stripe、Slack、GitHub などからデータを受け取ります。スキーマは全くコントロールできません。クエリすることはほとんどなく、デバッグやリプレイのために保存するだけです。**JSONB に最適です。**

2.  **ロギング & イベントストリーム**: アプリケーションログ、監査トレイル、エラーコンテキスト。書き込みが多く、特定フィールドでクエリされることは稀で、バルクで分析したり分析プラットフォームへエクスポートしたりします。**JSONB で問題ありません。**

3.  **ユーザー設定 & プリファレンス**: 100 以上のブールフラグを持つ設定オブジェクトで、ほとんどが false であり、常にユーザー ID で全体ブロブを取得します。`WHERE preferences->>'theme' = 'dark'` のようなクエリは実行しません。**JSONB が機能します。**

4.  **LLM プロバイダー / モデル構成**: これが最も明確な現代的例の一つです。OpenAI、Anthropic、Gemini、オープンウェイトのローカルモデル、ベンダー固有のゲートウェイはすべて、重複しつつも異なるパラメータを公開します。たとえ同一プロバイダー内でも、モデル機能やオプション名は進化します。`temperature`、`top_p`、`reasoning_effort`、`json_schema`、`tool_choice` など二十以上のノブをすべて汎用カラムにするより、JSONB の構成ブロブの方が正直です。**JSONB がしばしば正しい抽象化です。**

5.  **API 応答キャッシュ**: 完全な API 応答をキャッシュしています。データベースは高速な Redis の代替で、キャッシュキーで取得し、ネストされたプロパティで検索することはありません。**JSONB が適切です。**

6.  **イベントソーシング**: 不変のイベントペイロードを保存しています。クエリは常に「集約 X のすべてのイベントを時間順に取得する」だけで、イベントプロパティに対する `WHERE` 句は使用しません。**JSONB が合致します。**

7.  **拡張性の表面**: 統合、プラグイン設定、テナントごとのオーバーライド、マーケットプレイスメタデータ、プロバイダー機能、またはサブタイプごとに形状が変わることを明示的に想定した「extras」フィールド。**JSONB は妥協ではなく、正しい契約になることがあります。**

経験則: アプリケーションが既知のキーでドキュメントを取得し、検証やバージョン管理の方法を理解しているなら、JSONB は非常に有効です。ビジネスロジックがネストされたキーに対してリレーショナルな質問をし続ける場合、そのフィールドは列化しようとしている証拠です。

## ベストパターンはハイブリッドであることが多い

成熟したシステムの多くはここに集まります:

```sql
CREATE TABLE llm_requests (
  id UUID PRIMARY KEY,
  provider TEXT NOT NULL,
  model TEXT NOT NULL,
  status TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  config JSONB NOT NULL
);
```

この形は極端などちらかよりも一般に優れています。

- `provider`、`model`、`status`、`created_at` はファーストクラスの列です。これらでフィルタ、結合、集計、インデックスを行う必要があります。
- `config` は JSONB のままにします。正確なオプションの表面はモデル固有、プロバイダー固有で、変化しやすいためです。

これは「正規化に失敗した」ことではなく、**適切な場所で線を引いた**ことです。

### 大規模環境: 正規化よりオブジェクトバージョニング

ここが興味深いポイントです。規模が十分に大きくなると、「正しい」解は正規化ではなくオブジェクトバージョニングになります。

もし数十億行のデータがあり、スキーマの進化が頻繁なら、列のマイグレーションはコストがかかります。Stripe、GitHub、Netflix といった企業はすべてを正規化していません。その代わりに:

```sql
CREATE TABLE entities (
  id UUID PRIMARY KEY,
  version INT NOT NULL,
  data JSONB NOT NULL
);
```

アプリケーションは `version: 1`、`version: 2`、`version: 3` の読み取り方法を知っている。新しいフィールドのためにデータベースマイグレーションは不要で、コード側で後方互換性を処理する。

これは怠慢ではなく設計上の選択であり、データベースの複雑さをアプリケーションの複雑さと交換している。ドキュメントが自然にバージョン管理され、アプリが唯一の解釈者である場合、まさに正しいトレードオフになることがある。

失敗モードは「JSONB を使うこと」ではない。失敗モードはバージョニングや検証、プロモーションルール、ドキュメントデータとリレーショナルデータの境界が不明確なまま JSONB を使うことだ。

## 本当に重要な質問

JSONB カラムを追加する前に、次の点を自問せよ：

1.  `WHERE`、`JOIN`、`GROUP BY`、`ORDER BY` でネストされたフィールドを定期的にクエリする必要があるか？
2.  スキーマを自分たちで管理するか、外部定義で変動しやすいか？
3.  レコード間で形状が意図的に異質であるか？
4.  アプリケーションレベルでの検証とバージョニングは整備されているか？
5.  後で運用上の指標になる可能性が高いフィールドはどれか？

#1 の答えが「はい、常に」なら、列として持たせる強いシグナルになる。

#2 と #3 の答えが「はい」なら、JSONB が実際に役立つ可能性が高い。

## 落とし穴からの脱出

すでにこの穴にハマっているなら、掘り続けるのはやめろ。

1. 監査: `jsonb_object_keys` を実行し、実際の形状ドリフトを確認せよ。想定した形状ではなく、実際に存在する形状を見ること。
2. プロモート: フィルタ、結合、ソート、またはレポートで最も頻繁に使用するフィールドを特定し、それらを実際の列にする。
3. バリデーション: JSONB に残るものについて、アプリケーションまたはデータベースレベルで検証を追加せよ。
4. バージョン管理: ブロブが実際のドメインデータである場合、明示的にバージョンを付ける。
5. トリム: プロモートされた列が確立したら、ブロブから重複したキーを削除する。

すべてのブロブを正規化しなければならないと自分に言い聞かせるな。また、永続的なビジネス意味を持つブロブを「一時的」だと自分に言い聞かせるな。

JSONB はドキュメントそのものが本当に文書形状を持つ場合に有効だ。リレーショナルスキーマに偽の口ひげを付けたような使い方をすると危険になる。
