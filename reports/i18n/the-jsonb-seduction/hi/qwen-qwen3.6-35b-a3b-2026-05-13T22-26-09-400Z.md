```sql
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  profile JSONB
);
```

यह काम करता है। आप फीचर को प्रोडक्शन में भेज देते हैं, अगले फीचर पर आगे बढ़ जाते हैं, फिर अगले पर। JSONB कॉलम पृष्ठभूमि में चुपचाप बढ़ता जाता है।

यह रास्ता का वह मोड़ है। यदि `profile` एक अपारदर्शी ब्लॉब (opaque blob) बना रहता है जिसे केवल `user.id` से फ़ेच किया जाता है, तो शायद आप सुरक्षित हैं। लेकिन यदि यह बिज़नेस डेटा के मुख्य भंडारण स्थल के रूप में उभरने लगता है, तो ट्रेडऑफ़ तेज़ी से बदल जाते हैं।

प्रोडक्ट टीम पूछती है: *"न्यू यॉर्क में कितने यूज़र्स हैं?"*

आप यह लिखते हैं:

```sql
SELECT count(*) FROM users WHERE profile->>'location' = 'New York';
```

तो आप एक GIN इंडेक्स जोड़ते हैं। शायद यह अभी भी स्वीकार्य हो। कभी-कभी यह होता भी है। लेकिन अब आप वास्तविक जटिलता और स्टोरेज लागत चुका रहे हैं, क्योंकि एक ऐसा फ़ील्ड जो प्रथम-श्रेणी संबंधित डेटा की तरह व्यवहार करता है, कभी प्रथम-श्रेणी कॉलम के रूप में नहीं बना।

## JSONB का वास्तव में कब उपयोग करें

JSONB के वैध उपयोग के मामले हैं। कई बार यह बिल्कुल उचित रहता है, और कभी-कभी यह उपलब्ध सबसे बेहतर विकल्प होता है।

निर्णायक अंतर यह नहीं है कि "स्ट्रक्चर्ड डेटा अच्छा है और JSON बुरा है।" सही दृष्टिकोण इससे मिलता-जुलता है:

- क्या डेटा को स्थिर प्राथमिक कुंजी (primary key) के आधार पर मुख्य रूप से एक ही ब्लॉब के रूप में फ़ेच किया जाता है?
- क्या कुंजियाँ प्रदाताओं, संस्करणों, टेंनेंट्स या समय के साथ महत्वपूर्ण रूप से भिन्न होती हैं?
- क्या आप कुछ ज्ञात फ़ील्ड्स पर क्वेरी चला रहे हैं, या हर स्प्रिंट में नए पाथ-आधारित क्वेरी बना रहे हैं?
- क्या एप्लिकेशन वर्ज़निंग और वैलिडेशन को जानबूझकर संभालता है, या यह बिना किसी स्पष्ट रणनीति के काम चला रहा है?

### JSONB के वैध उपयोग के मामले

1.  **वेबहुक पेलोड्स**: आप Stripe, Slack, या GitHub जैसे स्रोतों से डेटा प्राप्त करते हैं। स्कीमा पर आपका कोई नियंत्रण नहीं होता। आप कभी भी उस पर क्वेरी नहीं चलाएंगे। आपको बस डिबगिंग या रीप्ले के लिए इसे स्टोर रखने की आवश्यकता होती है। **JSONB के लिए बिल्कुल सही।**

2.  **लॉगिंग और इवेंट स्ट्रीम्स**: एप्लिकेशन लॉग, ऑडिट ट्रेल, और एरर कॉन्टेक्स्ट। इनमें लिखने (write) की मात्रा अधिक होती है, विशिष्ट फ़ील्ड्स पर क्वेरी बहुत कम होती है, और अक्सर इन्हें बल्क में विश्लेषित किया जाता है या एनालिटिक्स प्लेटफ़ॉर्म पर निर्यात किया जाता है। **यहाँ JSONB बिल्कुल उचित है।**

3.  **उपयोगकर्ता प्राथमिकताएं और सेटिंग्स**: सेटिंग्स ऑब्जेक्ट्स जहाँ आपके पास 100 से अधिक बूलियन फ्लैग्स हैं, जिनमें से ज्यादातर `false` होते हैं, और आप हमेशा यूजर आईडी के आधार पर पूरा ब्लॉब फ़ेच करते हैं। आप `WHERE preferences->>'theme' = 'dark'` जैसी क्वेरी नहीं चला रहे हैं। **JSONB यहाँ काम करता है।**

4.  **LLM प्रदाता / मॉडल कॉन्फ़िग**: यह आधुनिक उदाहरणों में सबसे स्पष्ट में से एक है। OpenAI, Anthropic, Gemini, ओपन-वेट लोकेल मॉडल्स, और वेंडर-स्पेसिफिक गेटवे सभी ओवरलैपिंग लेकिन भिन्न पैरामीटर प्रस्तुत करते हैं। एक ही प्रदाता के भीतर मॉडल क्षमताएं और विकल्प नाम बदलते रहते हैं। `temperature`, `top_p`, `reasoning_effort`, `json_schema`, `tool_choice`, और बीस अन्य नियंत्रकों को यूनिवर्सल कॉलम मानने की बजाय एक JSONB कॉन्फ़िग ब्लॉब रखना कहीं ज्यादा व्यावहारिक और ईमानदार है। **JSONB अक्सर यहाँ सही एबस्ट्रैक्शन साबित होता है।**

5.  **API रिस्पॉन्स कैशिंग**: आप पूरी API रिस्पॉन्स को कैश कर रहे हैं। डेटाबेस यहाँ बस एक तेज़ Redis की तरह काम करता है। आप कैश की (cache key) के आधार पर डेटा फ़ेच करते हैं, नेस्टेड प्रॉपर्टीज के आधार पर कभी नहीं। **JSONB यहाँ उचित है।**

6.  **इवेंट सॉर्सिंग**: आप इम्यूटेबल इवेंट पेलोड्स स्टोर कर रहे हैं। आपकी क्वेरी हमेशा 'एग्रीगेट X के लिए सभी इवेंट्स समय के क्रम में दो' जैसी होती है। आप कभी इवेंट प्रॉपर्टीज पर `WHERE` क्लॉज लागू नहीं करते। **JSONB यहाँ पूरी तरह फिट बैठता है।**

7.  **एक्सटेंसिबिलिटी सर्फेस**: इंटीग्रेशन, प्लगइन सेटिंग्स, प्रति-टेंनैंट ओवरराइड, मार्केटप्लेस मेटाडेटा, प्रोवाइडर कैपेबिलिटीज, या "एक्स्ट्रा" फ़ील्ड्स जहाँ आप स्पष्ट रूप से उम्मीद करते हैं कि डेटा की शैप सबटाइप के हिसाब से बदलेगी। **JSONB यहाँ एक सही कॉन्ट्रैक्ट हो सकता है, कोई समझौता नहीं।**

Rule of thumb: अगर एप्लिकेशन डॉक्यूमेंट को किसी ज्ञात की (key) के आधार पर फ़ेच करता है और उसे वैलिडेट/वर्शन करने का तरीका पता है, तो JSONB शानदार हो सकता है। अगर बिज़नेस नेस्टेड कीज़ के बारे में रिलेशनल सवाल पूछता रहता है, तो वो फ़ील्ड्स कॉलम बनने की कोशिश कर रहे हैं।

## सबसे बेहतर पैटर्न अक्सर हाइब्रिड होता है

कई परिपक्व सिस्टम यहीं आकर रुकते हैं:

```sql
CREATE TABLE llm_requests (
  id UUID PRIMARY KEY,
  provider TEXT NOT NULL,
  model TEXT NOT NULL,
  status TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  config JSONB NOT NULL
);
```

यह आमतौर पर किसी भी चरम दृष्टिकोण (extreme) से बेहतर होता है।

- `provider`, `model`, `status`, और `created_at` फर्स्ट-क्लास कॉलम हैं क्योंकि आप उन पर फ़िल्टर, जॉइन, एग्रीगेशन और इंडेक्सिंग करेंगे।
- `config` JSONB ही रहेगा क्योंकि सटीक ऑप्शन सर्फेस मॉडल-स्पेसिफिक, प्रोवाइडर-स्पेसिफिक है और इसमें बदलाव की संभावना रहती है।

यह "नॉर्मलाइज़ेशन में असफलता" नहीं है। यह सही जगह पर रेखा खींचना है।

### स्केल पर: ऑब्जेक्ट वर्शनिंग > नॉर्मलाइज़ेशन

यहीं से बात दिलचस्प हो जाती है। पर्याप्त बड़े स्केल पर, "सही" समाधान नॉर्मलाइज़ेशन नहीं—बल्कि ऑब्जेक्ट वर्शनिंग है।

अगर आपके पास अरबों पंक्तियाँ (rows) हैं और स्कीमा में बार-बार बदलाव आ रहे हैं, तो कॉलम माइग्रेट करना महंगा पड़ता है। Stripe, GitHub, और Netflix जैसे कंपनियाँ सब कुछ नॉर्मलाइज़ नहीं करतीं। इसके बजाय:

```sql
CREATE TABLE entities (
  id UUID PRIMARY KEY,
  version INT NOT NULL,
  data JSONB NOT NULL
);
```

आपका ऐप्लिकेशन `version: 1`, `version: 2`, `version: 3` को पढ़ना जानता है। नए फ़ील्ड्स के लिए डेटाबेस माइग्रेशन की ज़रूरत नहीं पड़ती। कोड ही बैकवर्ड कॉम्पैटिबिलिटी संभालता है।

यह एक आर्किटेक्चरल निर्णय है, आलस्य नहीं। इसमें डेटाबेस की जटिलता को ऐप्लिकेशन की जटिलता के लिए ट्रेडऑफ़ किया जाता है। कभी-कभी यही सही ट्रेडऑफ़ होता है, खासकर तब जब दस्तावेज़ स्वाभाविक रूप से वर्श्ड हो और ऐप ही कैनोनिकल इंटरप्रेटर हो।

फेल्योर मोड "JSONB का उपयोग करना" नहीं है। फेल्योर मोड है बिना वर्शनिंग, वैलिडेशन, प्रोमोशन नियमों, या दस्तावेज़ डेटा और रिलेशनल डेटा के बीच स्पष्ट सीमा के JSONB का उपयोग करना।

JSONB कॉलम जोड़ने से पहले ये सवाल खुद से पूछें:

1.  क्या हम `WHERE`, `JOIN`, `GROUP BY`, या `ORDER BY` में नेस्टेड फ़ील्ड्स को नियमित रूप से क्वेरी करेंगे?
2.  क्या हम इस स्कीमा को कंट्रोल करते हैं, या यह बाहरी रूप से परिभाषित और अस्थिर है?
3.  क्या रिकॉर्ड्स के बीच शेप (shape) जानबूझकर विषम (heterogeneous) है?
4.  क्या हमारे पास ऐप्लिकेशन-लेवल वैलिडेशन और वर्शनिंग है?
5.  कौन से फ़ील्ड्स बाद में ऑपरेशनल डायमेंशन बनने की संभावना रखते हैं?

अगर #1 का जवाब "हाँ, लगातार" है, तो यह कॉलम बनाने की मज़बूत निशानी है।

अगर #2 और #3 का जवाब "हाँ" है, तो JSONB शायद आपके लिए असली काम कर रहा है।

---
## फँद से बाहर निकलना

अगर आप पहले ही इस गड्ढे में हैं, तो खोदना बंद कर दें।

1. ऑडिट: `jsonb_object_keys` चलाएँ और वास्तविक शैप ड्रिफ्ट की जाँच करें, न कि उस संरचना का जो आप मानते हैं कि मौजूद है।
2. प्रमोट: उन फ़ील्ड्स की पहचान करें जिन पर आप सबसे अधिक filter, join, sort, या रिपोर्ट बनाते हैं। उन्हें असली कॉलम में बदल दें।
3. वैलिडेशन: JSONB में बचे डेटा के लिए एप्लिकेशन या डेटाबेस-लेवल वैलिडेशन जोड़ें।
4. वर्शन: अगर ब्लॉब वास्तविक डोमेन डेटा है, तो उसे स्पष्ट रूप से वर्शन करें।
5. ट्रिम: प्रमोट किए गए कॉलम स्थापित होने के बाद, ब्लॉब से डुप्लिकेट कीज़ हटा दें।

खुद को यह मत समझाओ कि हर ब्लॉब को नॉर्मलाइज़ करना अनिवार्य है। साथ ही, इस बात को भी न मानो कि जिस ब्लॉब में स्थायी बिज़नेस सेमैंटिक्स हैं, वह "अस्थायी" है।

JSONB तब बेहतरीन है जब दस्तावेज़ वास्तव में दस्तावेज़-आकार का हो। यह खतरनाक है जब यह एक नकली दाढ़ी पहने हुए रिलेशनल स्कीमा हो।
