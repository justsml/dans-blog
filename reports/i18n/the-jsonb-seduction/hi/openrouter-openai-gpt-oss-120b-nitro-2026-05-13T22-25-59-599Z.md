## When to Actually Use JSONB

JSONB के वैध उपयोग केस होते हैं। कई बार यह पूरी तरह ठीक रहता है, और कभी‑कभी यह सबसे अच्छा विकल्प भी हो सकता है।

महत्वपूर्ण अंतर यह नहीं है “स्ट्रक्चर्ड अच्छा, JSON बुरा।” यह इस तरह है:

- क्या डेटा मुख्य कुंजी द्वारा एक ही बार में पूरी तरह लाया जाता है?
- क्या कुंजियों में प्रोवाइडर, संस्करण, टेनेंट या समय के अनुसार बड़ा अंतर होता है?
- क्या आप कुछ ज्ञात फ़ील्ड्स को क्वेरी कर रहे हैं, या हर स्प्रिंट में नए पाथ क्वेरी बना रहे हैं?
- क्या एप्लिकेशन जानबूझकर संस्करणीकरण और वैधता संभालता है, या बस अंदाज़ा लगा रहा है?

### वैध JSONB उपयोग केस

1.  **Webhook पेलोड्स**: आपको Stripe, Slack या GitHub से डेटा मिलता है। स्कीमा पर आपका कोई नियंत्रण नहीं है। आप शायद कभी इसे क्वेरी नहीं करेंगे। आपको सिर्फ डिबगिंग या रीप्ले के लिए इसे स्टोर करना है। **JSONB के लिए परफ़ेक्ट।**

2.  **लॉगिंग और इवेंट स्ट्रीम्स**: एप्लिकेशन लॉग, ऑडिट ट्रेल, एरर कॉन्टेक्स्ट। ये लिखने‑भारी होते हैं, विशिष्ट फ़ील्ड्स द्वारा शायद ही क्वेरी किए जाते हैं, और अक्सर बड़े पैमाने पर विश्लेषण या एनालिटिक्स प्लेटफ़ॉर्म पर एक्सपोर्ट होते हैं। **JSONB यहाँ ठीक है।**

3.  **यूज़र प्रेफ़रेंसेज़ और सेटिंग्स**: सेटिंग्स ऑब्जेक्ट जहाँ 100+ बूलियन फ़्लैग होते हैं, अधिकांश false होते हैं, और आप हमेशा यूज़र आईडी द्वारा पूरा ब्लॉब फ़ेच करते हैं। आप `WHERE preferences->>'theme' = 'dark'` नहीं चलाते। **JSONB काम करता है।**

4.  **LLM प्रोवाइडर / मॉडल कॉन्फ़िग**: यह सबसे स्पष्ट आधुनिक उदाहरणों में से एक है। OpenAI, Anthropic, Gemini, ओपन‑वेट लोकल मॉडल और विक्रेता‑विशिष्ट गेटवे सभी ओवरलैपिंग लेकिन अलग पैरामीटर पेश करते हैं। एक ही प्रोवाइडर के भीतर भी मॉडल क्षमताएँ और विकल्प नाम बदलते रहते हैं। एक JSONB कॉन्फ़िग ब्लॉब अक्सर यह स्वीकार करने में अधिक ईमानदार होता है कि `temperature`, `top_p`, `reasoning_effort`, `json_schema`, `tool_choice` और बीस अन्य नॉब्स सभी सार्वभौमिक कॉलम नहीं हो सकते। **JSONB अक्सर यहाँ सही एब्स्ट्रैक्शन है।**

5.  **API रिस्पॉन्स कैशिंग**: आप पूरे API रिस्पॉन्स को कैश कर रहे हैं। डेटाबेस सिर्फ तेज़ Redis है। आप कैश की द्वारा फ़ेच करते हैं, नेस्टेड प्रॉपर्टीज़ द्वारा नहीं। **JSONB उपयुक्त है।**

6.  **इवेंट सोर्सिंग**: आप अपरिवर्तनीय इवेंट पेलोड्स स्टोर कर रहे हैं। आपके क्वेरी हमेशा “एग्रीगेट X के सभी इवेंट्स समय के क्रम में दो” होते हैं। आप इवेंट प्रॉपर्टीज़ पर `WHERE` क्लॉज़ नहीं चलाते। **JSONB फिट बैठता है।**

7.**विस्तारशील सतहें**: इंटीग्रेशन, प्लगइन सेटिंग्स, प्रति‑टेनेंट ओवरराइड, मार्केटप्लेस मेटाडेटा, प्रोवाइडर क्षमताएँ, या “extras” फ़ील्ड जहाँ आप स्पष्ट रूप से अपेक्षा करते हैं कि आकार उप‑प्रकार के अनुसार बदलता रहेगा। **JSONB सही कॉन्ट्रैक्ट हो सकता है, समझौता नहीं।**

नियम यह है: यदि एप्लिकेशन ज्ञात कुंजी से दस्तावेज़ को फ़ेच करता है और उसे वैलिडेट/वर्ज़न करना जानता है, तो JSONB उत्कृष्ट हो सकता है। यदि व्यवसाय नेस्टेड कुंजियों पर रिलेशनल प्रश्न पूछता रहता है, तो वे फ़ील्ड कॉलम बनने की कोशिश कर रहे हैं।

## सबसे अच्छा पैटर्न अक्सर हाइब्रिड होता है

कई परिपक्व सिस्टम यहाँ समाप्त होते हैं:

```sql
CREATE TABLE llm_requests (
  id UUID PRIMARY KEY,
  provider TEXT NOT NULL,
  model TEXT NOT NULL,
  status TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  config JSONB NOT NULL
);
```

यह आम तौर पर दोनों चरम स्थितियों से बेहतर है।

- `provider`, `model`, `status`, और `created_at` प्रथम‑क्लास कॉलम हैं क्योंकि आप इन्हें फ़िल्टर, जॉइन, एग्रीगेट और इंडेक्स करेंगे।
- `config` JSONB बना रहता है क्योंकि सटीक विकल्प सतह मॉडल‑विशिष्ट, प्रोवाइडर‑विशिष्ट है और संभवतः विकसित होगी।

3.  वैलिडेट: जो कुछ भी JSONB में बचा है, उसके लिए एप्लिकेशन‑स्तर या डेटाबेस‑स्तर की वैधता जोड़ें।
4.  वर्ज़न: यदि ब्लॉब वास्तविक डोमेन डेटा है, तो इसे स्पष्ट रूप से संस्करणित करें।
5.  ट्रिम: प्रोमोट किए गए कॉलम स्थापित होने के बाद ब्लॉब से डुप्लिकेट कीज़ हटा दें।

हर ब्लॉब को सामान्यीकृत करना ज़रूरी नहीं है, यह खुद को मत बताएँ। साथ ही यह भी मत कहें कि स्थायी व्यापारिक अर्थ वाले ब्लॉब “अस्थायी” हैं।

JSONB तब शानदार है जब दस्तावेज़ वास्तव में दस्तावेज़‑आकार का हो। यह खतरनाक है जब यह एक रिलेशनल स्कीमा है जो नकली मूँछ पहन रहा हो।
