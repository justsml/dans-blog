## वास्तव में JSONB का उपयोग कब करें

JSONB के वैध उपयोग के मामले हैं। कई बार यह पूरी तरह से ठीक होता है, और कभी-कभी यह उपलब्ध सबसे अच्छा विकल्प होता है।

महत्वपूर्ण अंतर "संरचित अच्छा, JSON बुरा" नहीं है। यह इसके करीब है:

- क्या डेटा अधिकतर एक स्थिर प्राथमिक कुंजी द्वारा समग्र रूप से प्राप्त किया जाता है?
- क्या कुंजियाँ प्रदाताओं, संस्करणों, टेनेंट्स या समय के अनुसार भौतिक रूप से भिन्न होती हैं?
- क्या आप कुछ ज्ञात फ़ील्ड्स पर क्वेरी कर रहे हैं, या हर स्प्रिंट में नए पथ क्वेरीज़ का आविष्कार कर रहे हैं?
- क्या एप्लिकेशन जानबूझकर वर्जनिंग और सत्यापन का स्वामित्व रखता है, या बस बिना योजना के काम चला रहा है?

### वैध JSONB उपयोग के मामले

1.  **वेबहुक पेलोड**: आप Stripe, Slack या GitHub से डेटा प्राप्त करते हैं। आपका स्कीमा पर कोई नियंत्रण नहीं है। हो सकता है कि आप उस पर कभी क्वेरी न करें। आपको बस डिबगिंग या रीप्ले के लिए इसे संग्रहीत करना है। **JSONB के लिए एकदम सही।**

2.  **लॉगिंग और इवेंट स्ट्रीम**: एप्लिकेशन लॉग, ऑडिट ट्रेल, एरर कॉन्टेक्स्ट। ये राइट-हैवी होते हैं, शायद ही कभी विशिष्ट फ़ील्ड्स द्वारा क्वेरी किए जाते हैं, और अक्सर बल्क में विश्लेषित किए जाते हैं या एनालिटिक्स प्लेटफॉर्म पर निर्यात किए जाते हैं। **यहाँ JSONB ठीक है।**

3.  **उपयोगकर्ता प्राथमिकताएँ और सेटिंग्स**: सेटिंग्स ऑब्जेक्ट जहाँ आपके पास 100+ बूलियन फ़्लैग हैं, अधिकांश गलत हैं, और आप हमेशा उपयोगकर्ता आईडी द्वारा पूरा ब्लॉब प्राप्त कर रहे हैं। आप `WHERE preferences->>'theme' = 'dark'` नहीं चला रहे हैं। **JSONB काम करता है।**

4.  **LLM प्रदाता / मॉडल कॉन्फ़िग**: यह सबसे स्पष्ट आधुनिक उदाहरणों में से एक है। OpenAI, Anthropic, Gemini, ओपन-वेट स्थानीय मॉडल और विक्रेता-विशिष्ट गेटवे सभी ओवरलैपिंग लेकिन अलग-अलग पैरामीटर प्रदर्शित करते हैं। एक ही प्रदाता के भीतर भी, मॉडल क्षमताएँ और विकल्प नाम विकसित होते हैं। एक JSONB कॉन्फ़िग ब्लॉब अक्सर यह दिखावा करने से कहीं अधिक ईमानदार होता है कि `temperature`, `top_p`, `reasoning_effort`, `json_schema`, `tool_choice` और बीस अन्य नॉब सभी सार्वभौमिक कॉलम होने चाहिए। **यहाँ JSONB अक्सर सही अमूर्तता है।**

5.  **API प्रतिक्रिया कैशिंग**: आप संपूर्ण API प्रतिक्रियाओं को कैश कर रहे हैं। डेटाबेस सिर्फ एक तेज़ Redis है। आप कैश कुंजी द्वारा प्राप्त करते हैं, कभी नेस्टेड प्रॉपर्टीज़ द्वारा नहीं। **JSONB उपयुक्त है।**

6.  **इवेंट सोर्सिंग**: आप अपरिवर्तनीय इवेंट पेलोड संग्रहीत कर रहे हैं। आपकी क्वेरीज़ हमेशा "एग्रीगेट X के लिए सभी इवेंट दें" समय के अनुसार क्रमबद्ध होती हैं। आप कभी इवेंट प्रॉपर्टीज़ पर `WHERE` क्लॉज़ नहीं चलाते। **JSONB फिट बैठता है।**

7.  **एक्सटेंसिबिलिटी सतहें**: इंटीग्रेशन, प्लगइन सेटिंग्स, प्रति-टेनेंट ओवरराइड, मार्केटप्लेस मेटाडेटा, प्रदाता क्षमताएं, या "एक्स्ट्रा" फ़ील्ड जहां आप स्पष्ट रूप से उम्मीद करते हैं कि आकार उपप्रकार के अनुसार भिन्न होगा। **JSONB सही अनुबंध हो सकता है, कोई समझौता नहीं।**

## सर्वोत्तम पैटर्न अक्सर हाइब्रिड होता है

कई परिपक्व सिस्टम यहाँ आकर रुकते हैं:

```sql
CREATE TABLE llm_requests (
  id UUID PRIMARY KEY,
  provider TEXT NOT NULL,
  model TEXT NOT NULL,
  status TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),

```

यह आमतौर पर दोनों चरम सीमाओं से बेहतर होता है।

- `provider`, `model`, `status`, और `created_at` प्रथम श्रेणी के कॉलम हैं क्योंकि आप उन पर फ़िल्टर, जॉइन, एग्रीगेट और इंडेक्स करेंगे।
- `config` JSONB ही रहता है क्योंकि सटीक विकल्प सतह मॉडल-विशिष्ट, प्रदाता-विशिष्ट है और विकसित होने की संभावना है।

### बड़े पैमाने पर: ऑब्जेक्ट वर्ज़निंग > सामान्यीकरण

यहाँ यह दिलचस्प हो जाता है। पर्याप्त बड़े पैमाने पर, "सही" समाधान सामान्यीकरण नहीं है—यह ऑब्जेक्ट वर्ज़निंग है।

यदि आपके पास अरबों पंक्तियाँ और बार-बार स्कीमा विकास है, तो कॉलम माइग्रेट करना महँगा हो जाता है। Stripe, GitHub, और Netflix जैसी कंपनियाँ सब कुछ सामान्यीकृत नहीं करती हैं। इसके बजाय:

2.  **प्रमोट करें:** उन फ़ील्ड्स की पहचान करें जिन्हें आप सबसे अधिक फ़िल्टर, जॉइन, सॉर्ट या रिपोर्ट करते हैं। उन्हें वास्तविक कॉलम बनाएँ।
3.  **मान्य करें:** JSONB में जो कुछ भी बचता है, उसके लिए एप्लिकेशन या डेटाबेस-स्तरीय मान्यकरण जोड़ें।
4.  **वर्ज़न करें:** यदि ब्लॉब वास्तविक डोमेन डेटा है, तो उसे स्पष्ट रूप से वर्ज़न करें।
5.  **ट्रिम करें:** एक बार प्रमोट किए गए कॉलम स्थापित हो जाने के बाद, ब्लॉब से डुप्लिकेट की गई कुंजियाँ हटाएँ।

अपने आप से यह न कहें कि हर ब्लॉब को सामान्यीकृत किया जाना चाहिए। साथ ही यह भी न कहें कि स्थायी व्यावसायिक अर्थ वाला ब्लॉब "अस्थायी" है।

JSONB तब बढ़िया है जब दस्तावेज़ वास्तव में दस्तावेज़-आकार का हो। यह तब खतरनाक होता है जब वह नकली मूंछें पहने हुए एक रिलेशनल स्कीमा हो।
