Решение не в улучшении промптов. Оно в том, чтобы знать, когда перестать просить LLM «думать» и начать требовать, чтобы он «следовал».

## Когда детерминированность побеждает творчество

Когда детерминированность побеждает творчество

Представьте, что происходит, когда вам нужно обработать заявку на поддержку. Реальная бизнес-логика выглядит примерно так:

1. Получить детали заявки из базы данных
2. Проверить, имеет ли пользователь право на возврат средств (политика компании)
3. Подтвердить, что транзакция существует и еще не была возвращена
4. Рассчитать сумму возврата
5. Выполнить возврат платежа
6. Обновить статус заявки
7. Отправить подтверждающее письмо

Вы можете передать это LLM как упражнение с вызовом инструментов. По моему опыту, это приглашение к проблемам. Модель может решить, что шаги 2 и 3 «в сущности одно и то же», и пропустить один. Или она может начать возврат средств до проверки права на возврат, если пользователь выглядит обеспокоенным.

Работы с четко определенными шагами существуют именно для таких сценариев. Они не захватывающие, но это и есть их назначение.

### Построение планировщика активности по погоде

Вот практический пример, демонстрирующий этот паттерн. Нам нужны точные фактические данные о погоде и творческие предложения по активности. Получение погоды никогда не должно быть творческим, но предложения должны быть.

Модель никогда не взаимодействует напрямую с API погоды. Она получает данные истинного положения дел, а затем делает то, что действительно хорошо: предлагает контекстуальные варианты. Если же вы перевернете этот процесс и позволите агенту запрашивать данные о погоде, то в какой-то момент вы получите прогноз солнечной погоды, когда на самом деле идет дождь.

**Когда стоит использовать workflow:**
- У вас есть известная последовательность шагов, которые должны выполняться в определенном порядке
- Вам нужна наблюдаемость на каждом этапе (журналы, метрики, временные рамки)
- Вам нужна логика повторных попыток для нестабильных внешних API
- Бизнес-правила не могут быть "интерпретированы" - их нужно строго соблюдать

## Проблема с контекстным окном, о которой никто не говорит

Я постоянно наблюдаю один и тот же паттерн. Кто-то создает чат-бота. Он отлично работает в тестировании. Затем в продакшене, когда пользователи ведут более длительные диалоги, бот внезапно "теряется".

Разработчик смотрит в логи и понимает, что отправляет полную историю диалога с каждым запросом. Все 47 сообщений. Он тратит токены и пространство контекста на информацию, которая в основном не имеет отношения к текущей задаче.

Хуже того, существует явление, которое исследователи называют "теряется в середине", когда модели работают хуже, если важная информация скрыта в длинном контексте. Модель буквально не может увидеть лес за деревьями.

Отправка полной истории диалога кажется безопасной. Вы даете модели "все возможные данные". Но вы фактически затрудняете ее фокусировку на том, что действительно важно.

### Рабочая память vs. Долгосрочное хранение

Система памяти Mastra предоставляет оба варианта. Рабочая память хранит последние сообщения в контекстном окне. Семантическое воспоминание ищет исторические сообщения, когда текущий запрос кажется связанным.

```typescript
// src/mastra/agents/memory-agent.ts
import { Agent } from '@mastra/core/agent';
import { Memory } from '@mastra/memory';
import { LibSQLStore } from '@mastra/libsql';

export const memoryAgent = new Agent({
  id: 'memory-agent',
  name: 'Memory Agent',
  instructions: 'You are a helpful assistant with perfect recall of our conversations.',
  model: openai('gpt-5'),
  memory: new Memory({
    storage: new LibSQLStore({
      id: 'memory-agent-store',
      url: 'file:../mastra.db',
    }),
    options: {
      lastMessages: 20,  // Keep last 20 messages in context
      semanticRecall: {
        enabled: true,  // Use embeddings to find old stuff
        topK: 5,
        threshold: 0.7,
      },
    },
  }),
});
```

Вот как это работает на практике. Пользователь спрашивает: "Какой итальянский ресторан вы рекомендовали в прошлом месяце?"

Без семантического воспоминания агент видит последние 20 сообщений. Рекомендация ресторана была 487-м сообщением из 506. Она исчезла. Агент отвечает: "У меня нет этой информации".

С семантическим воспоминанием:  
1. Запрос встраивается: `[0.234, -0.567, 0.891, ...]`  
2. Векторное представление сравнивается с историей сообщений  
3. Сообщение 487 ("I'd recommend Trattoria Bella - their carbonara is incredible") имеет 0.89 схожести  
4. Это сообщение вставляется в текущий контекст  
5. Агент отвечает: "I recommended Trattoria Bella. Their carbonara is what caught my attention."  

Агент демонстрирует идеальную память, используя лишь долю контекстного окна. Это не просто хитрый инжиниринг — это функциональная необходимость, как только диалоги превышают несколько десятков сообщений.  

## Координация через сети агентов  

Иногда нужны и структура, и гибкость. Чисто workflow-подход слишком жёсток. Чисто агентный подход слишком непредсказуем.  

Сети агентов предоставляют координатора, который решает, какой специализированный агент или workflow запускать в зависимости от задачи. Представьте это как умный балансировщик нагрузки для AI-возможностей.

```typescript
export const coordinatorAgent = new Agent({
  id: 'coordinator-agent',
  name: 'Research Coordinator',
  instructions: `You are a network of researchers and writers.
    - Use researchAgent for gathering facts
    - Use writingAgent for producing final content
    - Use weatherTool for current weather data
    - Use activityPlannerWorkflow for location-based planning
    
    Always produce comprehensive, well-structured responses.`,
  model: openai('gpt-5'),
  
  // Available primitives
  agents: { researchAgent, writingAgent },
  workflows: { activityPlannerWorkflow },
  tools: { weatherTool },
  
  // Network requires memory
  memory: new Memory({
    storage: new LibSQLStore({ id: 'network-store', url: 'file:../network.db' }),
  }),
});
```

При запросе к этой сети координатор анализирует запрос и направляет его соответствующим образом:  
- "Мне нужны факты о X" активирует исследовательского агента  
- "План выходного дня в Сиэтле" запускает workflow планирования активностей  
- "Напиши отчет о Y" задействует письменного агента  

Этот паттерн масштабируется лучше, чем попытки втиснуть всё в один гигантский агент. Специализированные агенты развивают фокусированную экспертизу. Координатор управляет маршрутизацией. Каждая часть делает то, что ей присуще.

## Сборка в единое целое  

Реальные промышленные ИИ-системы требуют архитектуры, а не только подсказок. Вы строите распределенные системы, где некоторые узлы случайно являются LLM.  

Работы дают гарантии, когда нужно, чтобы всё происходило точно. Память обеспечивает контекст без того, чтобы тратить бюджет токенов. Сети агентов позволяют составлять сложность из более простых частей.

Ничего из этого не выглядит привлекательным. Но после того, как я достаточно часто наблюдал, как «полностью автономные агенты» выходят из строя в условиях эксплуатации, я пришёл к тому, что ценю скучную надёжность больше, чем волнующую неопределённость.  

У вас могут быть разные результаты, но по моему опыту, системы, которые действительно выпускаются и остаются в работе, — это те, которые рассматривают LLM как компоненты в более крупной архитектуре, а не как волшебные коробки, решающие всё.
