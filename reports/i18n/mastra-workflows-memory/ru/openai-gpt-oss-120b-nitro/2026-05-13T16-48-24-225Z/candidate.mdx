LLM обладают странным свойством: они блестяще улавливают нюансы, но ужасно справляются с выполнением рецептов. Дайте GPT‑4 расплывчатую задачу, и он будет рассуждать о возможных вариантах. Дайте ему чёткую последовательность шагов, и он может пропустить шаг 3, потому что шаг 5 «казался более уместным».

Это не ошибка модели. Это фундаментальная характеристика вероятностных систем, пытающихся решить детерминированные задачи.

Я наблюдал, как команды сталкиваются с этим несоответствием. Они создают агента для обработки возвратов клиентам, снабжают его дюжиной инструментов и ожидают, что он надёжно выполнит бизнес‑процесс. Иногда всё работает безупречно. Иногда он «галлюцинирует» одобрения, которых не было. Иногда он застревает, запрашивая одну и ту же информацию три раза.

Решение — не в лучших подсказках. Оно — в том, чтобы знать, когда перестать просить LLM «думать» и начать говорить ему «исполнять».

## Когда детерминированное важнее креативного

Подумайте, что происходит, когда нужно обработать тикет поддержки. Реальная бизнес‑логика выглядит примерно так:

1. Получить детали тикета из базы данных  
2. Проверить, имеет ли пользователь право на возврат (правила политики)  
3. Убедиться, что транзакция существует и ещё не была возвращена  
4. Рассчитать сумму возврата  
5. Выполнить отмену платежа  
6. Обновить статус тикета  
7. Отправить подтверждающее письмо  

Можно было бы передать всё это LLM как задачу вызова инструментов. По моему опыту, это приглашение к проблемам. Модель может решить, что шаги 2 и 3 — «по сути одно и то же», и пропустить один из них. Или она может выполнить возврат до проверки права, если пользователь выглядит раздражённым.

Для именно таких случаев существуют рабочие процессы. Они не вызывают восторга, но в этом и смысл.

### Создание планировщика погодных активностей

Ниже практический пример, демонстрирующий шаблон. Нам нужны твёрдые, фактические данные о погоде, сопоставленные с креативными предложениями активностей. Запрос погоды никогда не должен быть креативным, а вот предложения — должны.

```typescript
// src/mastra/workflows/activity-planner.ts
import { createWorkflow, createStep } from '@mastra/core/workflows';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

// Step 1: Fetch weather data (Deterministic)
const fetchWeather = createStep({
  id: 'fetch-weather',
  description: 'Fetches weather forecast for a given city',
  inputSchema: z.object({
    city: z.string(),
  }),
  outputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  execute: async ({ inputData }) => {
    // ... (fetch logic) ...
    const weather = await fetch(`https://api.open-meteo.com/v1/forecast?latitude=52.52&longitude=13.41&current=temperature_2m,weather_code&daily=precipitation_probability_mean`).then(r => r.json());
    
    return {
      location: inputData.city,
      temperature: weather.current.temperature_2m,
      conditions: getWeatherCondition(weather.current.weather_code),
      precipitationChance: weather.daily.precipitation_probability_mean[0],
    };
  },
});

// Step 2: Agent suggests activities (Creative)
const activityPlanner = new Agent({
  id: 'activity-planner-agent',
  name: 'Activity Planner',
  instructions: `You are a local activities expert. Based on weather conditions, suggest 3-5 appropriate activities.
    - For rain (>50% precipitation), prioritize indoor activities
    - For extreme temperatures, consider climate-appropriate options
    - Always include one adventurous and one relaxing option`,
  model: openai('gpt-5'),
});

const planActivities = createStep({
  id: 'plan-activities',
  description: 'Uses AI to suggest activities based on weather',
  inputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  outputSchema: z.object({
    activities: z.string(),
  }),
  execute: async ({ inputData }) => {
    const prompt = `Weather in ${inputData.location}: ${inputData.temperature}°C...`;
    const response = await activityPlanner.generate(prompt);
    return { activities: response.text };
  },
});

// The Pipeline
export const activityPlannerWorkflow = createWorkflow({
  id: 'activity-planner',
  inputSchema: z.object({ city: z.string() }),
  outputSchema: z.object({ activities: z.string() }),
})
  .then(fetchWeather)
  .then(planActivities);

activityPlannerWorkflow.commit();
```

LLM никогда не взаимодействует напрямую с погодным API. Он получает проверенные данные в качестве входа и делает то, что у него действительно получается — формировать контекстуальные предложения. Если попытаться поменять местами и позволить агенту запрашивать погоду, в итоге вы получите «солнечный» прогноз, когда на самом деле идёт дождь.

**Когда стоит использовать рабочие процессы:**
- Есть известная последовательность шагов, которую необходимо выполнять строго в заданном порядке
- Требуется наблюдаемость на каждом этапе (логи, метрики, время выполнения)
- Необходимо реализовать повторные попытки при работе с ненадёжными внешними API
- Бизнес‑правила нельзя «интерпретировать» — их нужно исполнять точно

## Проблема окна контекста, о которой никто не говорит

Я часто наблюдаю одну и ту же схему. Кто‑то создаёт чат‑бота. На тестах всё работает отлично. А в продакшене пользователи ведут более длительные диалоги, и бот вдруг «теряется».

Разработчик смотрит логи и понимает, что каждый запрос отправляет полную историю разговора — все 47 сообщений. В результате токены и пространство контекста расходуются на информацию, которая в большинстве случаев не имеет значения.

Хуже того, существует явление, которое исследователи называют «потеря в середине», когда модели работают хуже, если релевантная информация зарыта в длинном контексте. Модель буквально не видит леса из‑за деревьев.

Отправка полной истории диалога кажется безопасной. Вы даёте модели «всю информацию». На деле вы лишь усложняете ей задачу сосредоточиться на том, что действительно важно.

### Рабочая память vs. Долгосрочное хранилище

Система памяти Mastra предоставляет оба варианта. Рабочая память хранит последние сообщения в окне контекста. Семантическое воспоминание ищет в исторических сообщениях, когда текущий запрос кажется связанным.

```typescript
// src/mastra/agents/memory-agent.ts
import { Agent } from '@mastra/core/agent';
import { Memory } from '@mastra/memory';
import { LibSQLStore } from '@mastra/libsql';

export const memoryAgent = new Agent({
  id: 'memory-agent',
  name: 'Memory Agent',
  instructions: 'You are a helpful assistant with perfect recall of our conversations.',
  model: openai('gpt-5'),
  memory: new Memory({
    storage: new LibSQLStore({
      id: 'memory-agent-store',
      url: 'file:../mastra.db',
    }),
    options: {
      lastMessages: 20,  // Keep last 20 messages in context
      semanticRecall: {
        enabled: true,  // Use embeddings to find old stuff
        topK: 5,
        threshold: 0.7,
      },
    },
  }),
});
```

Вот как это выглядит на практике. Пользователь спрашивает: «Какой итальянский ресторан ты рекомендовал в прошлом месяце?»

Без семантического воспоминания агент видит только последние 20 сообщений. Рекомендация ресторана была в сообщении 487 из 506. Она уже недоступна. Агент отвечает: «У меня нет такой информации».

С семантическим воспоминанием:  
1. Запрос преобразуется в эмбеддинг: `[0.234, -0.567, 0.891, ...]`  
2. Эмбеддинг сравнивается с историческими сообщениями  
3. Сообщение 487 («Я бы порекомендовал Trattoria Bella — их карбонара просто невероятна») получает коэффициент сходства 0.89  
4. Это сообщение внедряется в текущий контекст  
5. Агент отвечает: «Я рекомендовал Trattoria Bella. Их карбонара — вот что меня привлекло.»

Агент выглядит так, будто обладает идеальной памятью, используя лишь небольшую часть окна контекста. Это не просто умная инженерия — это функциональная необходимость, когда диалоги выходят за пределы нескольких десятков сообщений.  

## Координация через сети агентов  

Иногда требуется и структура, и гибкость. Чистые рабочие процессы слишком жёстки. Чистые агенты — слишком непредсказуемы.  

Сети агентов предоставляют координатор, который решает, какой специализированный агент или рабочий процесс задействовать в зависимости от задачи. Можно сравнить с умным балансировщиком нагрузки для возможностей ИИ.

```typescript
export const coordinatorAgent = new Agent({
  id: 'coordinator-agent',
  name: 'Research Coordinator',
  instructions: `You are a network of researchers and writers.
    - Use researchAgent for gathering facts
    - Use writingAgent for producing final content
    - Use weatherTool for current weather data
    - Use activityPlannerWorkflow for location-based planning
    
    Always produce comprehensive, well-structured responses.`,
  model: openai('gpt-5'),
  
  // Available primitives
  agents: { researchAgent, writingAgent },
  workflows: { activityPlannerWorkflow },
  tools: { weatherTool },
  
  // Network requires memory
  memory: new Memory({
    storage: new LibSQLStore({ id: 'network-store', url: 'file:../network.db' }),
  }),
});
```

Когда вы отправляете запрос в эту сеть, координатор анализирует запрос и направляет его соответствующим образом:
- «Мне нужны факты о X» — запускает research‑agent
- «Составьте план уик‑энда в Сиэтле» — запускает workflow планировщика активности
- «Напиши отчёт о Y» — привлекает writing‑agent

Такой подход масштабируется лучше, чем попытка уместить всё в один мегa‑агент. Специализированные агенты развивают узкую экспертизу. Координатор отвечает за маршрутизацию. Каждый компонент делает то, в чём он силён.

## Сборка всего вместе

Реальные продакшн‑системы ИИ требуют архитектуры, а не только подсказок. Вы строите распределённые системы, где некоторые узлы случайно являются LLM.

Рабочие процессы дают гарантии, когда нужно, чтобы операции происходили точно так, как задумано. Память обеспечивает контекст без излишних расходов токенов. Сети агентов позволяют собрать сложность из более простых частей.

Ниодна из этих тем не выглядит привлекательно. Но после того, как я увидел, как множество «полностью автономных агентов» ломаются в продакшене, я пришёл к выводу, что скучная надёжность важнее захватывающей непредсказуемости.

У вас могут быть свои результаты, однако, по моему опыту, системы, которые действительно поставляются и продолжают работать, — это те, которые рассматривают LLM как компонент более крупной архитектуры, а не как волшебные коробки, решающие всё.
