LLM‑ओं में एक अजीब विशेषता होती है: वे बारीकियों को समझने में तेज़ होते हैं, लेकिन रेसिपी का पालन करने में बेतुके। GPT‑4 को एक अस्पष्ट समस्या दें और वह संभावनाओं पर तर्क करेगा। उसे एक सटीक कदम‑दर‑कदम क्रम दें, और वह चरण 3 को छोड़ सकता है क्योंकि चरण 5 “ज्यादा प्रासंगिक” लगा।

यह मॉडल में बग नहीं है। यह संभाव्य (probabilistic) सिस्टमों की मूलभूत विशेषता है, जो निर्धारक (deterministic) समस्याओं को हल करने की कोशिश करते हैं।

मैंने टीमों को इस असंगति से जूझते देखा है। वे ग्राहक रिफंड संभालने के लिए एक एजेंट बनाते हैं, उसे दर्जनों टूल्स देते हैं, और उम्मीद करते हैं कि वह व्यवसाय प्रक्रिया को भरोसेमंद ढंग से निष्पादित करेगा। कभी‑कभी यह बिल्कुल सही काम करता है। कभी‑कभी यह ऐसे अनुमोदन “हैलुसिनेट” करता है जो कभी नहीं हुए। कभी‑कभी यह वही जानकारी तीन बार पूछते‑पूछते फँस जाता है।

समाधान बेहतर प्रॉम्प्ट नहीं है। यह जानना है कि LLM को “सोचने” से कब रोकें और “आज्ञा‑पालन” के लिए कहें।

## जब निर्धारक रचनात्मकता से आगे हो


जब आपको एक सपोर्ट टिकट को प्रोसेस करना हो, तो वास्तविक‑दुनिया का बिज़नेस लॉजिक कुछ इस तरह दिखता है:

1. डेटाबेस से टिकट विवरण लाएँ
2. उपयोगकर्ता रिफंड के लिए योग्य है या नहीं, नीति नियमों के आधार पर जांचें
3. लेन‑देन मौजूद है और अभी तक रिफंड नहीं हुआ, यह सत्यापित करें
4. रिफंड राशि की गणना करें
5. भुगतान उलटने की प्रक्रिया करें
6. टिकट की स्थिति अपडेट करें
7. पुष्टि ई‑मेल भेजें

आप इसे एक LLM को टूल‑कॉलिंग अभ्यास के रूप में दे सकते हैं। मेरे अनुभव में, यह समस्या का आमंत्रण है। मॉडल चरण 2 और 3 को “लगभग एक ही चीज़” मान कर एक को छोड़ सकता है। या यह उपयोगकर्ता के गुस्से को देख कर योग्यता की जाँच से पहले ही रिफंड प्रोसेस कर सकता है।

वर्कफ़्लो ठीक इसी परिदृश्य के लिए बनाए गए हैं। वे रोमांचक नहीं होते, लेकिन यही उनका उद्देश्य है।

### मौसम‑आधारित गतिविधि प्लानर बनाना

यहाँ एक व्यावहारिक उदाहरण है जो इस पैटर्न को दर्शाता है। हमें ठोस, तथ्यात्मक मौसम डेटा चाहिए, जिसके साथ रचनात्मक गतिविधि सुझाव जोड़ें। मौसम प्राप्त करना कभी रचनात्मक नहीं होना चाहिए, लेकिन सुझाव देना चाहिए।

```typescript
// src/mastra/workflows/activity-planner.ts
import { createWorkflow, createStep } from '@mastra/core/workflows';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

// Step 1: Fetch weather data (Deterministic)
const fetchWeather = createStep({
  id: 'fetch-weather',
  description: 'Fetches weather forecast for a given city',
  inputSchema: z.object({
    city: z.string(),
  }),
  outputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  execute: async ({ inputData }) => {
    // ... (fetch logic) ...
    const weather = await fetch(`https://api.open-meteo.com/v1/forecast?latitude=52.52&longitude=13.41&current=temperature_2m,weather_code&daily=precipitation_probability_mean`).then(r => r.json());
    
    return {
      location: inputData.city,
      temperature: weather.current.temperature_2m,
      conditions: getWeatherCondition(weather.current.weather_code),
      precipitationChance: weather.daily.precipitation_probability_mean[0],
    };
  },
});

// Step 2: Agent suggests activities (Creative)
const activityPlanner = new Agent({
  id: 'activity-planner-agent',
  name: 'Activity Planner',
  instructions: `You are a local activities expert. Based on weather conditions, suggest 3-5 appropriate activities.
    - For rain (>50% precipitation), prioritize indoor activities
    - For extreme temperatures, consider climate-appropriate options
    - Always include one adventurous and one relaxing option`,
  model: openai('gpt-5'),
});

const planActivities = createStep({
  id: 'plan-activities',
  description: 'Uses AI to suggest activities based on weather',
  inputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  outputSchema: z.object({
    activities: z.string(),
  }),
  execute: async ({ inputData }) => {
    const prompt = `Weather in ${inputData.location}: ${inputData.temperature}°C...`;
    const response = await activityPlanner.generate(prompt);
    return { activities: response.text };
  },
});

// The Pipeline
export const activityPlannerWorkflow = createWorkflow({
  id: 'activity-planner',
  inputSchema: z.object({ city: z.string() }),
  outputSchema: z.object({ activities: z.string() }),
})
  .then(fetchWeather)
  .then(planActivities);

activityPlannerWorkflow.commit();
```

LLM कभी भी मौसम API को नहीं छूता। इसे वास्तविक डेटा इनपुट मिलता है, फिर वह वही करता है जिसमें वह वास्तव में निपुण है: संदर्भ‑आधारित सुझाव देना। यदि आप इसे उल्टा कर एजेंट को मौसम डेटा लाने देते हैं, तो अंत में आपको धूप वाला पूर्वानुमान मिलेगा जबकि असल में बारिश हो रही होगी।

**वर्कफ़्लो कब अपनाएँ:**
- आपके पास क्रमबद्ध, ज्ञात चरणों की श्रृंखला है जिसे ठीक‑ठाक क्रम में चलना आवश्यक है
- आपको प्रत्येक चरण पर अवलोकन चाहिए (लॉग, मीट्रिक, टाइमिंग)
- बाहरी अस्थिर API के लिए री‑ट्राई लॉजिक आवश्यक है
- व्यावसायिक नियम “व्याख्या” नहीं किए जा सकते – उन्हें सटीक रूप से लागू होना चाहिए

## वह कॉन्टेक्स्ट विंडो समस्या जिसके बारे में कोई नहीं बात करता

एक पैटर्न बार‑बार दिखता है। कोई चैटबॉट बनाता है। परीक्षण में वह बढ़िया चलता है। फिर प्रोडक्शन में, उपयोगकर्ता लंबे‑लंबे संवाद शुरू करते हैं और अचानक बॉट दिशा‑भ्रमित हो जाता है।

डेवलपर लॉग देखता है और समझता है कि वह हर अनुरोध के साथ पूरी बातचीत इतिहास भेज रहा है। सभी 47 संदेश। वह टोकन और कॉन्टेक्स्ट स्पेस को उन जानकारियों पर बर्बाद कर रहा है जो अधिकांशतः अप्रासंगिक हैं।

बुरा तो यह है कि शोधकर्ता एक घटना को "मध्य में खो जाना" कहते हैं, जहाँ मॉडल तब खराब प्रदर्शन करता है जब प्रासंगिक जानकारी लंबी संदर्भ में दबी होती है। मॉडल वास्तव में पेड़ों के कारण जंगल नहीं देख पाता।

पूरी बातचीत इतिहास भेजना सुरक्षित लग सकता है। आप मॉडल को “सभी जानकारी” दे रहे हैं। लेकिन वास्तव में आप मॉडल के लिए यह कठिन बना रहे हैं कि वह वही पर ध्यान दे जो मायने रखता है।

### कार्यशील स्मृति बनाम दीर्घकालिक संग्रह

Mastra की मेमोरी सिस्टम दोनों प्रदान करती है। कार्यशील स्मृति हाल के संदेशों को कॉन्टेक्स्ट विंडो में रखती है। सेमांटिक रिकॉल तब ऐतिहासिक संदेशों को खोजता है जब वर्तमान क्वेरी संबंधित लगती है।

```typescript
// src/mastra/agents/memory-agent.ts
import { Agent } from '@mastra/core/agent';
import { Memory } from '@mastra/memory';
import { LibSQLStore } from '@mastra/libsql';

export const memoryAgent = new Agent({
  id: 'memory-agent',
  name: 'Memory Agent',
  instructions: 'You are a helpful assistant with perfect recall of our conversations.',
  model: openai('gpt-5'),
  memory: new Memory({
    storage: new LibSQLStore({
      id: 'memory-agent-store',
      url: 'file:../mastra.db',
    }),
    options: {
      lastMessages: 20,  // Keep last 20 messages in context
      semanticRecall: {
        enabled: true,  // Use embeddings to find old stuff
        topK: 5,
        threshold: 0.7,
      },
    },
  }),
});
```

व्यावहारिक रूप से यह इस प्रकार काम करता है। एक उपयोगकर्ता पूछता है: “पिछले महीने आपने कौन सा इटालियन रेस्तरां सुझाया था?”

सेमांटिक रिकॉल के बिना, एजेंट केवल अंतिम 20 संदेश देखता है। रेस्तरां की सिफ़ारिश संदेश 506 में से 487वें स्थान पर थी। वह अब उपलब्ध नहीं है। एजेंट जवाब देता है “मेरे पास वह जानकारी नहीं है।”

सेमांटिक रिकॉल के साथ:
1. क्वेरी को एम्बेड किया जाता है: `[0.234, -0.567, 0.891, ...]`
2. एम्बेडिंग को ऐतिहासिक संदेशों के विरुद्ध तुलना की जाती है
3. संदेश 487 ("मैं Trattoria Bella की सिफ़ारिश करता—उनका कार्बोनारा अद्भुत है") 0.89 समानता स्कोर प्राप्त करता है
4. वह संदेश वर्तमान संदर्भ में इंजेक्ट किया जाता है
5. एजेंट जवाब देता है: "मैंने Trattoria Bella की सिफ़ारिश की थी। उनका कार्बोनारा ही मेरी ध्यान आकर्षित करने वाला था।"

एजेंट को ऐसा लगता है कि उसके पास पूर्ण स्मृति है, जबकि वह केवल संदर्भ विंडो के एक अंश का ही उपयोग कर रहा है। यह केवल चतुर इंजीनियरिंग नहीं है—जब बातचीत कुछ दर्जन संदेशों से आगे बढ़ती है, तो यह कार्यात्मक रूप से आवश्यक हो जाता है।

## एजेंट नेटवर्क के माध्यम से समन्वय

कभी‑कभी आपको संरचना और लचीलापन दोनों की जरूरत होती है। शुद्ध वर्कफ़्लो बहुत कठोर होते हैं। शुद्ध एजेंट बहुत अप्रत्याशित होते हैं।

एजेंट नेटवर्क आपको एक समन्वयकर्ता देता है जो कार्य के आधार पर यह तय करता है कि कौन‑सा विशेषीकृत एजेंट या वर्कफ़्लो बुलाया जाए। इसे AI क्षमताओं के लिए एक स्मार्ट लोड बैलेंसर समझें।

```typescript
export const coordinatorAgent = new Agent({
  id: 'coordinator-agent',
  name: 'Research Coordinator',
  instructions: `You are a network of researchers and writers.
    - Use researchAgent for gathering facts
    - Use writingAgent for producing final content
    - Use weatherTool for current weather data
    - Use activityPlannerWorkflow for location-based planning
    
    Always produce comprehensive, well-structured responses.`,
  model: openai('gpt-5'),
  
  // Available primitives
  agents: { researchAgent, writingAgent },
  workflows: { activityPlannerWorkflow },
  tools: { weatherTool },
  
  // Network requires memory
  memory: new Memory({
    storage: new LibSQLStore({ id: 'network-store', url: 'file:../network.db' }),
  }),
});
```

जब आप इस नेटवर्क को क्वेरी करते हैं, तो कोऑर्डिनेटर अनुरोध का विश्लेषण करता है और उसी अनुसार रूट करता है:
- “मुझे X के बारे में तथ्य चाहिए” रिसर्च एजेंट को ट्रिगर करता है
- “सीएटल में एक वीकेंड प्लान करो” एक्टिविटी प्लानर वर्कफ़्लो चलाता है
- “Y पर रिपोर्ट लिखो” राइटिंग एजेंट को एंगेज करता है

यह पैटर्न एक ही मेगा‑एजेंट में सब कुछ धकेलने की तुलना में बेहतर स्केलेबल है। विशेषीकृत एजेंट्स फोकस्ड एक्सपर्टाइज़ विकसित करते हैं। कोऑर्डिनेटर रूटिंग संभालता है। प्रत्येक भाग वही करता है जिसमें वह अच्छा है।

## इसे एक साथ जोड़ना

वास्तविक प्रोडक्शन AI सिस्टम को आर्किटेक्चर चाहिए, सिर्फ प्रॉम्प्ट नहीं। आप वितरित सिस्टम बना रहे हैं जहाँ कुछ नोड्स संयोगवश LLM होते हैं।

वर्कफ़्लो आपको गारंटी देता है जब आपको चीज़ें ठीक‑ठीक होने की आवश्यकता होती है। मेमोरी आपको संदर्भ देती है बिना टोकन बजट को जलाए। एजेंट नेटवर्क आपको सरल भागों से जटिलता को संयोजित करने की सुविधा देता है।

यह कोई रोमांचक बात नहीं है। लेकिन जब मैंने पर्याप्त “पूरी तरह स्वायत्त एजेंट” को उत्पादन में फेल होते देखा, तो मैंने रोमांचक अनिश्चितता से अधिक बोरिंग विश्वसनीयता की कद्र करना सीख लिया।

आपका अनुभव अलग हो सकता है, लेकिन मेरे अनुभव में वे सिस्टम जो वास्तव में शिप होते हैं और चलते रहते हैं, वे ही हैं जो LLM को बड़े आर्किटेक्चर के घटकों के रूप में देखते हैं, न कि सभी समस्याओं को हल करने वाले जादुई बॉक्स।
