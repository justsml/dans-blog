LLMs haben diese seltsame Eigenschaft: Sie sind brillant darin, Nuancen zu verstehen, aber katastrophal darin, Rezepte zu befolgen. Geben Sie GPT-4 ein vages Problem, und es wird Möglichkeiten durchdenken. Geben Sie ihm eine präzise Abfolge von Schritten, und es könnte Schritt 3 überspringen, weil Schritt 5 „irgendwie relevanter erscheint“.  

Dies ist kein Fehler im Modell. Es ist eine fundamentale Eigenschaft von probabilistischen Systemen, die versuchen, deterministische Probleme zu lösen.  

Ich habe Teams beobachtet, wie sie mit diesem Mismatch kämpfen. Sie bauen einen Agenten, um Kundenerstattungen zu verarbeiten, geben ihm eine Handvoll Tools und erwarten, dass er zuverlässig einen Geschäftsprozess ausführt. Manchmal funktioniert es perfekt. Manchmal erfindet er Genehmigungen, die nie stattgefunden haben. Manchmal steckt er fest und bittet drei Mal um dieselbe Information.  

Die Lösung sind nicht bessere Prompts. Es ist das Wissen, wann man aufhören muss, dem LLM zu sagen, „denke“, und stattdessen zu sagen, „gehorsam sei“.  

## Wenn deterministische Systeme kreativ überlegen sind

Denken Sie daran, was passiert, wenn Sie ein Support-Ticket verarbeiten müssen. Reale Geschäftslogik sieht ungefähr so aus:  

1. Rufe die Ticketdetails aus der Datenbank ab  
2. Überprüfe, ob der Benutzer berechtigt ist, eine Rückerstattung zu erhalten (Richtlinienregeln)  
3. Bestätige, dass die Transaktion existiert und noch nicht erstattet wurde  
4. Berechne den Rückerstattungsbetrag  
5. Verarbeite die Zahlungsrückzahlung  
6. Aktualisiere den Ticketstatus  
7. Senden Sie eine Bestätigungs-E-Mail  

Sie könnten dies einem LLM als Tool-Aufruf-Übung übergeben. Aus meiner Erfahrung ist das ein Fehler, der passieren wird. Das Modell könnte entscheiden, dass Schritt 2 und 3 „im Grunde dasselbe sind“ und einen überspringen. Oder es könnte die Rückerstattung verarbeiten, bevor es die Berechtigung überprüft, weil der Benutzer verärgert schien.  

Workflows existieren genau für solche Szenarien. Sie sind nicht aufregend, aber das ist der Punkt.  

### Bau eines Wetteraktivitätsplaners  

Hier ist ein praktisches Beispiel, das das Muster zeigt. Wir benötigen harte, faktische Wetterdaten gepaart mit kreativen Aktivitätsempfehlungen. Der Wetterabruf sollte niemals kreativ sein, aber die Empfehlungen sollten es sein.

```typescript
// src/mastra/workflows/activity-planner.ts
import { createWorkflow, createStep } from '@mastra/core/workflows';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

// Schritt 1: Wetterdaten abrufen (Deterministisch)
const fetchWeather = createStep({
  id: 'fetch-weather',
  description: 'Holt die Wettervorhersage für eine gegebene Stadt',
  inputSchema: z.object({
    city: z.string(),
  }),
  outputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  execute: async ({ inputData }) => {
    // ... (Abruflogik) ...
    const weather = await fetch(`https://api.open-meteo.com/v1/forecast?latitude=52.52&longitude=13.41&current=temperature_2m,weather_code&daily=precipitation_probability_mean`).then(r => r.json());
    
    return {
      location: inputData.city,
      temperature: weather.current.temperature_2m,
      conditions: getWeatherCondition(weather.current.weather_code),
      precipitationChance: weather.daily.precipitation_probability_mean[0],
    };
  },
});

// Schritt 2: Agent schlägt Aktivitäten vor (Kreativ)
const activityPlanner = new Agent({
  id: 'activity-planner-agent',
  name: 'Activity Planner',
  instructions: `Sie sind ein lokaler Aktivitäts-Experte. Basierend auf Wetterbedingungen schlagen Sie 3-5 passende Aktivitäten vor.
    - Bei Regen (>50% Niederschlagswahrscheinlichkeit) Priorisierung von Innenaktivitäten
    - Bei Extremtemperaturen Berücksichtigung von klimatisch angepassten Optionen
    - Immer eine Abenteuer- und eine Entspannungsaktivität einbeziehen`,
  model: openai('gpt-5'),
});

const planActivities = createStep({
  id: 'plan-activities',
  description: 'Verwendet KI, um Aktivitäten basierend auf Wetterbedingungen zu vorschlagen',
  inputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  outputSchema: z.object({
    activities: z.string(),
  }),
  execute: async ({ inputData }) => {
    const prompt = `Wetter in ${inputData.location}: ${inputData.temperature}°C...`;
    const response = await activityPlanner.generate(prompt);
    return { activities: response.text };
  },
});

// Die Pipeline
export const activityPlannerWorkflow = createWorkflow({
  id: 'activity-planner',
  inputSchema: z.object({ city: z.string() }),
  outputSchema: z.object({ activities: z.string() }),
})
  .then(fetchWeather)
  .then(planActivities);

activityPlannerWorkflow.commit();
```

Das LLM berührt niemals die Wetter-API. Es erhält reale Daten als Eingabe und konzentriert sich dann auf das, was es tatsächlich gut kann: Kontextuelle Vorschläge zu machen. Wenn Sie die Reihenfolge umdrehen und den Agenten die Wetterdaten abrufen lassen, erhalten Sie letztendlich eine sonnige Vorhersage, obwohl es tatsächlich regnet.

**Wann Workflows in Betracht ziehen:**
- Sie haben eine bekannte Abfolge von Schritten, die in einer bestimmten Reihenfolge ausgeführt werden müssen
- Sie benötigen Beobachtbarkeit auf jeder Stufe (Protokolle, Metriken, Zeitmessung)
- Sie benötigen Wiederholungslogik für unzuverlässige externe APIs
- Geschäftsregeln dürfen nicht "interpretiert" werden – sie müssen exakt befolgt werden

## Das Kontextfenster-Problem, über das niemand spricht

Es gibt ein Muster, das ich immer wieder sehe. Jemand baut einen Chatbot. Während der Tests funktioniert alles prima. Im Produktivbetrieb haben die Nutzer jedoch längere Konversationen, und plötzlich verliert sich der Bot.

Der Entwickler schaut in die Protokolle und stellt fest, dass er bei jedem Anfragenaufruf die gesamte Konversationshistorie sendet. Alle 47 Nachrichten. Dadurch werden Tokens und Kontextfenster verschwendet, obwohl die meisten Informationen irrelevant sind.

Schlimmer noch, es gibt ein Phänomen, das Forscher „lost in the middle“ nennen, bei dem Modelle schlechter abschneiden, wenn relevante Informationen in einer langen Kontextzeichenkette versteckt sind. Das Modell sieht buchstäblich nicht den Wald vor lauter Bäumen.

Das Senden der gesamten Konversationshistorie fühlt sich sicher an. Man gibt dem Modell „alle Informationen“. Tatsächlich macht man es ihm aber nur schwerer, sich auf das Wichtige zu konzentrieren.

### Arbeitsgedächtnis vs. Langzeit-Speicher

Das Mastra-Gedächtnis-System bietet beides. Das Arbeitsgedächtnis hält die neuesten Nachrichten im Kontextfenster. Die semantische Rückrufsuche durchsucht historische Nachrichten, wenn die aktuelle Anfrage releviert.

```typescript
// src/mastra/agents/memory-agent.ts
import { Agent } from '@mastra/core/agent';
import { Memory } from '@mastra/memory';
import { LibSQLStore } from '@mastra/libsql';

export const memoryAgent = new Agent({
  id: 'memory-agent',
  name: 'Memory Agent',
  instructions: 'You are a helpful assistant with perfect recall of our conversations.',
  model: openai('gpt-5'),
  memory: new Memory({
    storage: new LibSQLStore({
      id: 'memory-agent-store',
      url: 'file:../mastra.db',
    }),
    options: {
      lastMessages: 20,  // Keep last 20 messages in context
      semanticRecall: {
        enabled: true,  // Use embeddings to find old stuff
        topK: 5,
        threshold: 0.7,
      },
    },
  }),
});
```

So sieht das in der Praxis aus. Ein Nutzer fragt: „Was war das italienische Restaurant, das du letzte Monat empfohlen hast?“

Ohne semantische Rückrufsuche sieht das Agent die letzten 20 Nachrichten. Die Restaurantempfehlung war Nachricht 487 von 506. Sie ist weg. Das Agent antwortet: „Ich habe diese Information nicht.“

Mit semantischer Rückrufsuche:  
1. Die Abfrage wird eingebettet: `[0.234, -0.567, 0.891, ...]`  
2. Die Einbettung wird mit historischen Nachrichten verglichen  
3. Nachricht 487 („Ich würde Trattoria Bella empfehlen – ihre Carbonara ist unglaublich“) erzielt einen Ähnlichkeitswert von 0.89  
4. Diese Nachricht wird in den aktuellen Kontext eingefügt  
5. Das Agenten antwortet: „Ich habe Trattoria Bella empfohlen. Ihre Carbonara hat meine Aufmerksamkeit erregt.“  

Das Agenten erscheint, eine perfekte Erinnerung zu haben, während es nur einen Bruchteil des Kontextfensters nutzt. Dies ist nicht nur geschicktes Engineering – es ist funktionell notwendig, sobald Gespräche über einige Dutzend Nachrichten hinausgehen.

## Koordination durch Agentennetzwerke  

Manchmal benötigen Sie sowohl Struktur als auch Flexibilität. Reine Workflows sind zu starr. Reine Agenten sind zu unvorhersehbar.  

Agentennetzwerke bieten Ihnen einen Koordinator, der entscheidet, welcher spezialisierte Agent oder Workflow basierend auf der Aufgabe aufgerufen wird. Denken Sie daran als intelligenten Lastausgleichs-Verstärker für AI-Fähigkeiten.

```typescript
export const coordinatorAgent = new Agent({
  id: 'coordinator-agent',
  name: 'Research Coordinator',
  instructions: `You are a network of researchers and writers.
    - Use researchAgent for gathering facts
    - Use writingAgent for producing final content
    - Use weatherTool for current weather data
    - Use activityPlannerWorkflow for location-based planning
    
    Always produce comprehensive, well-structured responses.`,
  model: openai('gpt-5'),
  
  // Available primitives
  agents: { researchAgent, writingAgent },
  workflows: { activityPlannerWorkflow },
  tools: { weatherTool },
  
  // Network requires memory
  memory: new Memory({
    storage: new LibSQLStore({ id: 'network-store', url: 'file:../network.db' }),
  }),
});
```

Wenn Sie diese Netzwerkabfrage ausführen, analysiert der Koordinator die Anfrage und leitet sie entsprechend weiter:  
- „I need facts about X“ löst den Forschungs-Agent aus  
- „Plan a weekend in Seattle“ führt den Aktivitätsplaner-Workflow aus  
- „Write a report on Y“ aktiviert den Schreib-Agent  

Dieses Muster skaliert besser als das Versuchen, alles in einen einzelnen Mega-Agenten zu packen. Spezialisierte Agenten entwickeln fokussiertes Know-how. Der Koordinator übernimmt die Weiterleitung. Jeder Baustein macht das, wofür er geschaffen wurde.

## Zusammenbau  

Reale Produktions-AI-Systeme benötigen Architektur, nicht nur Prompts. Sie bauen verteilte Systeme auf, bei denen einige Knoten zufällig LLMs sind.  

Workflows garantieren Ihnen Sicherheit, wenn Dinge exakt richtig ablaufen müssen. Memory bietet Kontext ohne Ihr Token-Budget zu verfeuern. Agentennetze ermöglichen es, Komplexität aus einfachen Teilen zusammenzusetzen.

Ihr Meilenstein kann variieren, aber aus meiner Erfahrung sind die Systeme, die tatsächlich ausgeliefert und weiterlaufen, diejenigen, die LLMs als Komponenten in einer größeren Architektur betrachten, anstatt sie als magische Kästen zu sehen, die alles lösen.
