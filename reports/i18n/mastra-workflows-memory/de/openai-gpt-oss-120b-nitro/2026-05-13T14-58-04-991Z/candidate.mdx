LLMshaben diese seltsame Eigenschaft: Sie sind brillant im Verstehen von Nuancen, aber miserabel beim Befolgen von Anleitungen. Gib GPT‑4 ein vages Problem und es wird verschiedene Möglichkeiten durchdenken. Gib ihm eine präzise Schritt‑folge, und es könnte Schritt 3 überspringen, weil Schritt 5 „relevanter erschien“.

Das ist kein Bug im Modell. Es ist ein grundlegendes Merkmal probabilistischer Systeme, die versuchen, deterministische Probleme zu lösen.

Ich habe Teams beobachtet, die mit dieser Diskrepanz kämpfen. Sie bauen einen Agenten, der Kundenerstattungen abwickelt, geben ihm ein Dutzend Werkzeuge und erwarten, dass er zuverlässig einen Geschäftsprozess ausführt. Manchmal funktioniert es einwandfrei. Manchmal halluziniert er Genehmigungen, die nie stattgefunden haben. Manchmal bleibt er stecken und fragt dieselbe Information dreimal nach.

Die Lösung sind nicht bessere Prompts. Es geht darum zu wissen, wann man den LLM aufhören lässt zu „denken“ und ihm sagt, er solle „befolgen“.

## Wenn Deterministisches Kreativität übertrifft

Denken Sie darüber nach, was passiert, wenn Sie ein Support‑Ticket bearbeiten müssen. Praktische Geschäftslogik sieht in etwa so aus:

1. Ticketdetails aus der Datenbank holen  
2. Prüfen, ob der Nutzer für eine Rückerstattung berechtigt ist (Richtlinienregeln)  
3. Verifizieren, dass die Transaktion existiert und noch nicht erstattet wurde  
4. Rückerstattungsbetrag berechnen  
5. Zahlungsumkehr durchführen  
6. Ticket‑Status aktualisieren  
7. Bestätigungs‑E‑Mail senden  

Man könnte das einem LLM als „Tool‑Calling“-Aufgabe übergeben. Aus meiner Erfahrung ist das ein Rezept für Probleme. Das Modell könnte die Schritte 2 und 3 als „im Wesentlichen dasselbe“ interpretieren und einen überspringen. Oder es könnte die Rückerstattung ausführen, bevor die Berechtigung geprüft wurde, weil der Nutzer verärgert wirkte.

Workflows existieren genau für solche Fälle. Sie sind nicht aufregend – und das ist beabsichtigt.

### Aufbau eines Wetter‑Aktivitäts‑Planers

Im Folgenden ein praxisnahes Beispiel, das das Muster demonstriert. Wir benötigen harte, faktische Wetterdaten kombiniert mit kreativen Aktivitätsvorschlägen. Das Abrufen des Wetters darf niemals kreativ sein, die Vorschläge jedoch schon.

```typescript
// src/mastra/workflows/activity-planner.ts
import { createWorkflow, createStep } from '@mastra/core/workflows';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

// Step 1: Fetch weather data (Deterministic)
const fetchWeather = createStep({
  id: 'fetch-weather',
  description: 'Fetches weather forecast for a given city',
  inputSchema: z.object({
    city: z.string(),
  }),
  outputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  execute: async ({ inputData }) => {
    // ... (fetch logic) ...
    const weather = await fetch(`https://api.open-meteo.com/v1/forecast?latitude=52.52&longitude=13.41&current=temperature_2m,weather_code&daily=precipitation_probability_mean`).then(r => r.json());
    
    return {
      location: inputData.city,
      temperature: weather.current.temperature_2m,
      conditions: getWeatherCondition(weather.current.weather_code),
      precipitationChance: weather.daily.precipitation_probability_mean[0],
    };
  },
});

// Step 2: Agent suggests activities (Creative)
const activityPlanner = new Agent({
  id: 'activity-planner-agent',
  name: 'Activity Planner',
  instructions: `You are a local activities expert. Based on weather conditions, suggest 3-5 appropriate activities.
    - For rain (>50% precipitation), prioritize indoor activities
    - For extreme temperatures, consider climate-appropriate options
    - Always include one adventurous and one relaxing option`,
  model: openai('gpt-5'),
});

const planActivities = createStep({
  id: 'plan-activities',
  description: 'Uses AI to suggest activities based on weather',
  inputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  outputSchema: z.object({
    activities: z.string(),
  }),
  execute: async ({ inputData }) => {
    const prompt = `Weather in ${inputData.location}: ${inputData.temperature}°C...`;
    const response = await activityPlanner.generate(prompt);
    return { activities: response.text };
  },
});

// The Pipeline
export const activityPlannerWorkflow = createWorkflow({
  id: 'activity-planner',
  inputSchema: z.object({ city: z.string() }),
  outputSchema: z.object({ activities: z.string() }),
})
  .then(fetchWeather)
  .then(planActivities);

activityPlannerWorkflow.commit();
```

Das LLM greift nie auf die Wetter‑API zu. Es erhält Ground‑Truth‑Daten als Eingabe und erledigt dann das, worin es wirklich stark ist: kontextbezogene Vorschläge zu machen. Wenn man das umdreht und dem Agenten das Wetter abrufen lässt, bekommt man irgendwann einen sonnigen Forecast, obwohl es tatsächlich regnet.

**Wann Workflows in Betracht ziehen:**
- Es gibt eine bekannte Abfolge von Schritten, die strikt in dieser Reihenfolge ausgeführt werden muss
- Pro Schritt wird Beobachtbarkeit benötigt (Logs, Metriken, Timing)
- Für unzuverlässige externe APIs ist Wiederholungslogik erforderlich
- Geschäftsregeln können nicht „interpretiert“ werden – sie müssen exakt eingehalten werden

## Das Kontext‑Fenster‑Problem, über das niemand spricht

Ein Muster, das ich immer wieder sehe: Jemand baut einen Chatbot. Im Test läuft er einwandfrei. In der Produktion führen Nutzer längere Unterhaltungen, und plötzlich verliert der Bot den Faden.

Der Entwickler schaut in die Logs und erkennt, dass bei jeder Anfrage die komplette Gesprächshistorie mitgesendet wird – alle 47 Nachrichten. Das verbrennt Tokens und Kontext‑Platz für Informationen, die meist irrelevant sind.

Schlimmer noch, es gibt ein Phänomen, das Forscher „lost in the middle“ nennen, bei dem Modelle schlechter abschneiden, wenn relevante Informationen tief in einem langen Kontext vergraben sind. Das Modell kann buchstäblich den Wald vor lauter Bäumen nicht mehr sehen.

Das Senden der gesamten Gesprächshistorie wirkt sicher. Man gibt dem Modell „alle Informationen“. In Wirklichkeit erschwert man dem Modell jedoch, sich auf das Wesentliche zu konzentrieren.

### Arbeitsgedächtnis vs. Langzeitspeicher

Mastras Speichersystem liefert beides. Das Arbeitsgedächtnis hält die letzten Nachrichten im Kontextfenster. Die semantische Erinnerung durchsucht historische Nachrichten, wenn die aktuelle Anfrage thematisch zu ihnen passt.

```typescript
// src/mastra/agents/memory-agent.ts
import { Agent } from '@mastra/core/agent';
import { Memory } from '@mastra/memory';
import { LibSQLStore } from '@mastra/libsql';

export const memoryAgent = new Agent({
  id: 'memory-agent',
  name: 'Memory Agent',
  instructions: 'You are a helpful assistant with perfect recall of our conversations.',
  model: openai('gpt-5'),
  memory: new Memory({
    storage: new LibSQLStore({
      id: 'memory-agent-store',
      url: 'file:../mastra.db',
    }),
    options: {
      lastMessages: 20,  // Keep last 20 messages in context
      semanticRecall: {
        enabled: true,  // Use embeddings to find old stuff
        topK: 5,
        threshold: 0.7,
      },
    },
  }),
});
```

So sieht das in der Praxis aus. Ein Nutzer fragt: „Welches italienische Restaurant hast du letzten Monat empfohlen?“

Ohne semantische Erinnerung sieht der Agent nur die letzten 20 Nachrichten. Die Restaurantempfehlung war Nachricht 487 von 506. Sie ist nicht mehr im Kontext. Der Agent antwortet: „Ich habe diese Information nicht.“

Mit semantischer Erinnerung:
1. Die Anfrage wird eingebettet: `[0.234, -0.567, 0.891, …]`
2. Die Einbettung wird mit historischen Nachrichten verglichen
3. Nachricht 487 („Ich würde die Trattoria Bella empfehlen – ihre Carbonara ist unglaublich“) erzielt 0,89 Ähnlichkeit
4. Diese Nachricht wird in den aktuellen Kontext injiziert
5. Der Agent antwortet: „Ich habe die Trattoria Bella empfohlen. Ihre Carbonara war das, was mir aufgefallen ist.“

Der Agent wirkt, als hätte er ein perfektes Gedächtnis, nutzt dabei aber nur einen Bruchteil des Kontextfensters. Das ist nicht nur clevere Technik – es wird funktional nötig, sobald Unterhaltungen über ein paar Dutzend Nachrichten hinausgehen.

## Koordination über Agentennetzwerke

Manchmal braucht man sowohl Struktur als auch Flexibilität. Reine Workflows sind zu starr. Reine Agenten sind zu unvorhersehbar.

Agentennetzwerke liefern einen Koordinator, der basierend auf der Aufgabe entscheidet, welchen spezialisierten Agenten oder Workflow er aufruft. Das lässt sich als intelligenter Load‑Balancer für KI‑Fähigkeiten verstehen.

```typescript
export const coordinatorAgent = new Agent({
  id: 'coordinator-agent',
  name: 'Research Coordinator',
  instructions: `You are a network of researchers and writers.
    - Use researchAgent for gathering facts
    - Use writingAgent for producing final content
    - Use weatherTool for current weather data
    - Use activityPlannerWorkflow for location-based planning
    
    Always produce comprehensive, well-structured responses.`,
  model: openai('gpt-5'),
  
  // Available primitives
  agents: { researchAgent, writingAgent },
  workflows: { activityPlannerWorkflow },
  tools: { weatherTool },
  
  // Network requires memory
  memory: new Memory({
    storage: new LibSQLStore({ id: 'network-store', url: 'file:../network.db' }),
  }),
});
```

Wenn Sie dieses Netzwerk abfragen, analysiert der Koordinator die Anfrage und leitet sie entsprechend weiter:
- „Ich brauche Fakten zu X“ löst den Research‑Agent aus
- „Plane ein Wochenende in Seattle“ startet den Activity‑Planner‑Workflow
- „Schreibe einen Bericht über Y“ aktiviert den Writing‑Agent

Dieses Muster skaliert besser, als alles in einen einzigen Mega‑Agent zu pressen. Spezial‑Agenten entwickeln fokussiertes Fachwissen. Der Koordinator übernimmt das Routing. Jeder Baustein macht das, wofür er am besten geeignet ist.

## Zusammensetzen

Echte Produktions‑KI‑Systeme benötigen Architektur, nicht nur Prompt‑Design. Sie bauen verteilte Systeme, bei denen einige Knoten zufällig LLMs sind.

Workflows liefern Garantien, wenn Abläufe exakt so ablaufen müssen, wie vorgesehen. Speicher (Memory) stellt Kontext bereit, ohne das Token‑Budget zu sprengen. Agent‑Netzwerke ermöglichen es, Komplexität aus einfacheren Bausteinen zusammenzusetzen.

Nichts davon ist glamourös. Aber nachdem ich genug „vollständig autonome Agenten“ in der Produktion scheitern sah, habe ich die langweilige Zuverlässigkeit der aufregenden Unvorhersehbarkeit vorgezogen.

Ihre Erfahrungen können abweichen, doch nach meiner Erfahrung sind die Systeme, die tatsächlich ausgeliefert werden und laufen bleiben, jene, die LLMs als Bausteine einer größeren Architektur behandeln und nicht als magische Kästen, die alles lösen.
