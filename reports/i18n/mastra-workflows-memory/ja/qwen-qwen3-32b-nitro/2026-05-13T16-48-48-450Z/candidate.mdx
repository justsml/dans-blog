LLMには奇妙な特性があります。それは、微妙なニュアンスを理解するには優れていても、レシピに従うことは苦手だということです。GPT-4に曖昧な問題を与えると、可能性を推論して解決します。しかし、明確な手順のシーケンスを与えると、ステップ5が「より関連性がある」と感じてステップ3を飛ばしてしまうかもしれません。

これはモデルのバグではありません。確率論的システムが決定論的問題を解決しようとする際の根本的な特性です。

私はこの不一致に悩まされるチームを見てきました。顧客返金を処理するエージェントを構築し、12個のツールを与えて、ビジネスプロセスを確実に実行できると期待します。場合によっては完璧に動くこともあります。しかし、時には実際には行われていない承認を幻覚して承認されたかのように振る舞ったり、同じ情報を3回も繰り返して尋ねて立ち往生したりします。

解決策は「より良いプロンプト」ではありません。LLMに「考える」ことを求めるのをやめ、'従う'ように指示を出すタイミングを知ることです。

## 決定論的が創造的を凌ぐとき

サポートチケットを処理する際、現実世界のビジネスロジックは次のようになります。

1. データベースからチケットの詳細を取得する
2. ユーザーが返金対象かをポリシー規則で確認する
3. トランザクションが存在し、すでに返金されていないことを検証する
4. 返金金額を計算する
5. 支払いを返金処理する
6. チケットステータスを更新する
7. 確認メールを送信する

これをLLMにツール呼び出しの練習として渡すこともできます。しかし経験上、これはトラブルを招く要因になります。モデルはステップ2と3が「本質的に同じもの」と判断して1つをスキップするかもしれません。あるいはユーザーが不機嫌そうだと感じて、ステップ2の資格確認をスキップして返金処理を実行するかもしれません。

ワークフローはまさにこのようなシナリオのために存在しています。それは刺激的ではありませんが、それが目的です。

### 天気アクティビティプランナーの構築

このパターンを示す実践的な例を紹介します。厳密な事実ベースの天気データと、創造的なアクティビティ提案を組み合わせる必要があります。天気データの取得は決して創造的であってはなりませんが、提案は創造的であるべきです。

```typescript
// src/mastra/workflows/activity-planner.ts
import { createWorkflow, createStep } from '@mastra/core/workflows';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

// ステップ1: 天気データの取得（決定論的）
const fetchWeather = createStep({
  id: 'fetch-weather',
  description: '指定された都市の天気予報を取得します',
  inputSchema: z.object({
    city: z.string(),
  }),
  outputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  execute: async ({ inputData }) => {
    // ... (取得ロジック) ...
    const weather = await fetch(`https://api.open-meteo.com/v1/forecast?latitude=52.52&longitude=13.41&current=temperature_2m,weather_code&daily=precipitation_probability_mean`).then(r => r.json());
    
    return {
      location: inputData.city,
      temperature: weather.current.temperature_2m,
      conditions: getWeatherCondition(weather.current.weather_code),
      precipitationChance: weather.daily.precipitation_probability_mean[0],
    };
  },
});

// ステップ2: エージェントがアクティビティを提案（創造的）
const activityPlanner = new Agent({
  id: 'activity-planner-agent',
  name: 'Activity Planner',
  instructions: `あなたは現地アクティビティの専門家です。天気条件に基づいて3〜5つの適切なアクティビティを提案してください。
    - 雨（降水量50%以上）の場合は屋内アクティビティを優先
    - 極端な気温の場合は気候に応じたオプションを検討
    - いつも冒険的でリラックスできるオプションを1つずつ含める`,
  model: openai('gpt-5'),
});

const planActivities = createStep({
  id: 'plan-activities',
  description: 'AIを使用して天気に基づくアクティビティを提案',
  inputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  outputSchema: z.object({
    activities: z.string(),
  }),
  execute: async ({ inputData }) => {
    const prompt = `天気 ${inputData.location}: ${inputData.temperature}°C...`;
    const response = await activityPlanner.generate(prompt);
    return { activities: response.text };
  },
});

// パイプライン
export const activityPlannerWorkflow = createWorkflow({
  id: 'activity-planner',
  inputSchema: z.object({ city: z.string() }),
  outputSchema: z.object({ activities: z.string() }),
})
  .then(fetchWeather)
  .then(planActivities);

activityPlannerWorkflow.commit();
```

LLMは天気APIに直接アクセスしません。基準となるデータを入力として受け取り、実際に得意な文脈に応じた提案を行います。このプロセスを逆転させ、エージェントが天気データを取得させると、実際に雨が降っているにもかかわらず晴れの予報が出るような問題が最終的に発生します。

**ワークフローを検討すべきシナリオ:**
- 実行順序が決まっているステップの既知のシーケンスがある場合
- 各ステージで観測可能性（ログ、メトリクス、タイミング）が必要な場合
- 不安定な外部APIに対して再試行ロジックが必要な場合
- 業務ルールが「解釈」ではなく正確に従わなければならない場合

## 誰も話さないコンテキストウィンドウの問題

私が繰り返し見かけるパターンがあります。誰かがチャットボットを構築します。テストでは問題なく動作しますが、本番環境ではユーザーが長い会話を進めると突然ボットが迷子になります。

開発者はログを確認し、すべての会話履歴を毎回リクエストに含めていることに気づきます。47件のメッセージを。関係ない情報のためにトークンとコンテキストスペースを浪費しています。

より悪いことに、研究者たちが「middle lost（中で迷う）」と呼ぶ現象があります。関連する情報が長すぎるコンテキストに埋もれていると、モデルの性能が低下するのです。モデルは文字通り「森を見失っている」状態です。  

すべての会話履歴を送信するのは安全に感じるかもしれません。モデルに「すべての情報を与える」ことで、より良い結果を期待できるからです。しかし実際には、モデルが重要な部分に集中するのが難しくなっているのです。  

### 作業メモリと長期保存  

Mastraのメモリシステムは、この両方を実現します。作業メモリは最近のメッセージをコンテキストウィンドウに保持します。意味論的回想は、現在のクエリが過去のメッセージに関連している場合に、履歴を検索します。  

```typescript
// src/mastra/agents/memory-agent.ts
import { Agent } from '@mastra/core/agent';
import { Memory } from '@mastra/memory';
import { LibSQLStore } from '@mastra/libsql';

export const memoryAgent = new Agent({
  id: 'memory-agent',
  name: 'Memory Agent',
  instructions: 'You are a helpful assistant with perfect recall of our conversations.',
  model: openai('gpt-5'),
  memory: new Memory({
    storage: new LibSQLStore({
      id: 'memory-agent-store',
      url: 'file:../mastra.db',
    }),
    options: {
      lastMessages: 20,  // Keep last 20 messages in context
      semanticRecall: {
        enabled: true,  // Use embeddings to find old stuff
        topK: 5,
        threshold: 0.7,
      },
    },
  }),
});
```

これは実際にどうなるかというと、ユーザーが次のように尋ねたとします。「先月おすすめしてくれたイタリアンレストランはどこでしたか？」  

意味論的回想が使われていない場合、エージェントは直近の20件のメッセージを見ます。レストランの推薦は506件中の487件目でした。それは表示されず、エージェントは「その情報を保持していません」と応答します。

意味論的回想が有効な場合:
1. クエリが埋め込みに変換される: `[0.234, -0.567, 0.891, ...]`
2. 埋め込みが過去のメッセージと比較される
3. メッセージ487（「イタリアンレストラン『トラトリアベラ』をおすすめします。そのカルボナーラは素晴らしいです」）が0.89の類似度スコアを取得
4. そのメッセージが現在のコンテキストに挿入される
5. エージェントが応答:「トラトリアベラをおすすめしました。そのカルボナーラが注目されました。」

エージェントはコンテキストウィンドウの一部しか使用していないのに、まるで完璧な記憶を持っているように見える。これは単なる巧妙なエンジニアリングではなく、数十メッセージを超える会話が続くと機能的に必要不可欠になる。

## エージェントネットワークによる調整

構造と柔軟性の両方が必要な場合もある。純粋なワークフローは過度に硬く、純粋なエージェントは過度に予測不能である。

エージェントネットワークは、タスクに基づいてどの専門化されたエージェントやワークフローを呼び出すかを決定する調整者（コーディネーター）を提供する。AI機能のスマートなロードバランサと考えてほしい。

```typescript
export const coordinatorAgent = new Agent({
  id: 'coordinator-agent',
  name: 'Research Coordinator',
  instructions: `あなたは研究者とライターのネットワークです。
    - 事実の収集にはresearchAgentを使用してください
    - 最終コンテンツの作成にはwritingAgentを使用してください
    - 現在の天気データにはweatherToolを使用してください
    - 場所ベースの計画にはactivityPlannerWorkflowを使用してください
    
    常に包括的で構造化された応答を生成してください。`,
  model: openai('gpt-5'),
  
  // 利用可能なプリミティブ
  agents: { researchAgent, writingAgent },
  workflows: { activityPlannerWorkflow },
  tools: { weatherTool },
  
  // ネットワークにはメモリが必要
  memory: new Memory({
    storage: new LibSQLStore({ id: 'network-store', url: 'file:../network.db' }),
  }),
});
```

このネットワークにクエリを送信すると、コーディネーターがリクエストを分析してルーティングします：
- "Xに関する事実が必要です" → researchAgentが起動
- "シアトルでの週末を計画してください" → activityPlannerWorkflowが実行
- "Yに関するレポートを書いてください" → writingAgentが起動

このパターンは、すべてを単一のメガエージェントに詰め込むよりもスケーラブルです。専門化されたエージェントは集中した専門知識を発展させます。コーディネーターはルーティングを担当します。それぞれのコンポーネントは得意なことを実行します。

## 全体の構成

実際の運用AIシステムにはプロンプトだけでなくアーキテクチャが必要です。構築しているのは、一部のノードがLLMである分散システムです。

ワークフローは、正確に実行する必要があるタスクに対して保証を提供します。メモリはトークン予算を燃やさずにコンテキストを提供します。エージェントネットワークは、単純なコンポーネントから複雑性を構成できます。

これらは華やかなものではありません。しかし、十分な数の「完全に自律したエージェント」が運用環境で失敗したのを見てきた後、私は興奮する予測不能性よりも地味な信頼性を重視するようになりました。  

実際に展開され、継続して動作するシステムは、LLMを包括的なアーキテクチャ内のコンポーネントとして扱い、すべてを解決する魔法の箱と見なさないものです。
