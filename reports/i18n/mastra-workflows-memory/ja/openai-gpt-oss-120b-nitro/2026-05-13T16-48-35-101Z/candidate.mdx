LLM には奇妙な特性があります。ニュアンスの理解は得意ですが、レシピ通りに実行するのは苦手です。GPT‑4 に曖昧な問題を与えると、可能性を検討しながら推論します。正確な手順のシーケンスを与えると、ステップ 3 を飛ばしてステップ 5 が「より関連性が高い」と判断してしまうことがあります。

これはモデルのバグではありません。確定的な問題を解こうとする確率的システムの根本的な特徴です。

このミスマッチに苦しむチームを見てきました。顧客の返金処理を行うエージェントを構築し、ツールを十数個与えてビジネスプロセスを確実に実行させようとします。うまくいくこともありますが、実際に存在しない承認を幻覚的に生成したり、同じ情報を三回も要求して詰まったりします。

解決策は「より良いプロンプト」ではありません。LLM に「考える」ことをやめさせて「従う」よう指示するタイミングを見極めることです。

## 決定的な処理が創造的な処理に勝るとき

サポートチケットを処理する必要がある場合に何が起きるかを考えてみてください。実際のビジネスロジックは大まかに次のようになります。

1. データベースからチケットの詳細を取得する  
2. ユーザーが返金対象かどうかをポリシー規則でチェックする  
3. 取引が存在し、まだ返金されていないことを確認する  
4. 返金額を算出する  
5. 支払いの取り消しを実行する  
6. チケットのステータスを更新する  
7. 確認メールを送信する  

これらをツール呼び出しの演習として LLM に任せることもできますが、私の経験上、それは問題を招く行為です。モデルはステップ 2 と 3 を「ほぼ同じこと」とみなして片方をスキップしたり、ユーザーが怒っているように見えるために、適格性のチェックを行う前に返金処理を実行したりします。

このようなシナリオのためにワークフローが用意されています。ワークフロー自体は派手ではありませんが、そこが狙いです。

### 天気アクティビティプランナーの構築

以下はパターンを示す実用的な例です。確実で事実ベースの天気データと、創造的なアクティビティ提案を組み合わせる必要があります。天気取得は決して創造的であってはならず、提案部分だけが創造性を発揮すべきです。

```typescript
// src/mastra/workflows/activity-planner.ts
import { createWorkflow, createStep } from '@mastra/core/workflows';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

// Step 1: Fetch weather data (Deterministic)
const fetchWeather = createStep({
  id: 'fetch-weather',
  description: 'Fetches weather forecast for a given city',
  inputSchema: z.object({
    city: z.string(),
  }),
  outputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  execute: async ({ inputData }) => {
    // ... (fetch logic) ...
    const weather = await fetch(`https://api.open-meteo.com/v1/forecast?latitude=52.52&longitude=13.41&current=temperature_2m,weather_code&daily=precipitation_probability_mean`).then(r => r.json());
    
    return {
      location: inputData.city,
      temperature: weather.current.temperature_2m,
      conditions: getWeatherCondition(weather.current.weather_code),
      precipitationChance: weather.daily.precipitation_probability_mean[0],
    };
  },
});

// Step 2: Agent suggests activities (Creative)
const activityPlanner = new Agent({
  id: 'activity-planner-agent',
  name: 'Activity Planner',
  instructions: `You are a local activities expert. Based on weather conditions, suggest 3-5 appropriate activities.
    - For rain (>50% precipitation), prioritize indoor activities
    - For extreme temperatures, consider climate-appropriate options
    - Always include one adventurous and one relaxing option`,
  model: openai('gpt-5'),
});

const planActivities = createStep({
  id: 'plan-activities',
  description: 'Uses AI to suggest activities based on weather',
  inputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  outputSchema: z.object({
    activities: z.string(),
  }),
  execute: async ({ inputData }) => {
    const prompt = `Weather in ${inputData.location}: ${inputData.temperature}°C...`;
    const response = await activityPlanner.generate(prompt);
    return { activities: response.text };
  },
});

// The Pipeline
export const activityPlannerWorkflow = createWorkflow({
  id: 'activity-planner',
  inputSchema: z.object({ city: z.string() }),
  outputSchema: z.object({ activities: z.string() }),
})
  .then(fetchWeather)
  .then(planActivities);

activityPlannerWorkflow.commit();
```

LLM は天気 API に一切触れません。確実なデータを入力として受け取り、得意な「文脈に合わせた提案」だけを行います。逆にエージェントに天気取得を任せると、実際は雨なのに晴れ予報が返ってくるような事態に陥ります。

**ワークフローを検討すべきタイミング:**
- 明確な順序で実行しなければならないステップがあるとき
- 各段階で可観測性（ログ、メトリクス、タイミング）が必要なとき
- 外部 API が不安定でリトライロジックが求められるとき
- ビジネスルールは「解釈」できない。正確に遵守しなければならないとき

## 誰も語らないコンテキストウィンドウ問題

あるパターンをよく目にします。誰かがチャットボットを作る。テストではうまく動くのに、実運用でユーザーが長い会話を続けると、いきなりボットが迷子になる。

開発者がログを確認すると、毎回リクエストに会話履歴全体（全部で 47 件）を送っていることが判明します。ほとんどが無関係な情報なのに、トークンとコンテキスト領域を浪費しているのです。

さらに悪化するのが、研究者が「ミドルで迷子になる（lost in the middle）」と呼ぶ現象です。長いコンテキストに関連情報が埋もれると、モデルの性能が低下します。文字通り、木々の中で森が見えなくなるイメージです。

会話履歴全体を送るのは安全だと感じがちです。「すべての情報」をモデルに与えているように思えますが、実際にはモデルが重要な点に集中しにくくなっています。

### 作業メモリ vs. 長期ストレージ

Mastra のメモリシステムはこの二つを提供します。作業メモリは最近のメッセージをコンテキストウィンドウに保持し、長期ストレージは現在のクエリが関連性を示すときに過去のメッセージを検索します。

```typescript
// src/mastra/agents/memory-agent.ts
import { Agent } from '@mastra/core/agent';
import { Memory } from '@mastra/memory';
import { LibSQLStore } from '@mastra/libsql';

export const memoryAgent = new Agent({
  id: 'memory-agent',
  name: 'Memory Agent',
  instructions: 'You are a helpful assistant with perfect recall of our conversations.',
  model: openai('gpt-5'),
  memory: new Memory({
    storage: new LibSQLStore({
      id: 'memory-agent-store',
      url: 'file:../mastra.db',
    }),
    options: {
      lastMessages: 20,  // Keep last 20 messages in context
      semanticRecall: {
        enabled: true,  // Use embeddings to find old stuff
        topK: 5,
        threshold: 0.7,
      },
    },
  }),
});
```

実際の動きを見てみましょう。ユーザーが「先月おすすめしたイタリアンレストランはどこだった？」と尋ねたとします。

セマンティックリコールを使わない場合、エージェントは直近の 20 件のメッセージしか見ません。レストランの推薦は 506 件中 487 件目にあり、コンテキストから消えてしまいます。結果、エージェントは「その情報は持っていません」と答えてしまいます。

セマンティックリコールを使用した場合:
1. クエリが埋め込みベクトルに変換される: `[0.234, -0.567, 0.891, ...]`
2. 埋め込みベクトルが過去のメッセージと比較される
3. メッセージ 487（「Trattoria Bella をおすすめします ― カルボナーラが最高です」）が 0.89 の類似度でヒットする
4. そのメッセージが現在のコンテキストに注入される
5. エージェントが応答する: 「Trattoria Bella をおすすめしました。カルボナーラが特に印象的でした。」

エージェントはコンテキストウィンドウのごく一部しか使用せずに、まるで完全な記憶を持っているかのように振る舞います。これは単なる巧妙な実装ではなく、会話が数十件を超える規模になると機能的に必須となります。

## エージェントネットワークによる協調

構造と柔軟性の両方が必要になることがあります。純粋なワークフローは硬直しすぎ、純粋なエージェントは予測不可能です。

エージェントネットワークは、タスクに応じてどの専門エージェントまたはワークフローを呼び出すかを決定するコーディネータを提供します。AI 能力のためのスマートなロードバランサーと考えてください。

```typescript
export const coordinatorAgent = new Agent({
  id: 'coordinator-agent',
  name: 'Research Coordinator',
  instructions: `You are a network of researchers and writers.
    - Use researchAgent for gathering facts
    - Use writingAgent for producing final content
    - Use weatherTool for current weather data
    - Use activityPlannerWorkflow for location-based planning
    
    Always produce comprehensive, well-structured responses.`,
  model: openai('gpt-5'),
  
  // Available primitives
  agents: { researchAgent, writingAgent },
  workflows: { activityPlannerWorkflow },
  tools: { weatherTool },
  
  // Network requires memory
  memory: new Memory({
    storage: new LibSQLStore({ id: 'network-store', url: 'file:../network.db' }),
  }),
});
```

このネットワークにクエリを投げると、コーディネータはリクエストを解析し、適切にルーティングします。  
- 「X に関する事実が必要だ」と言えば researchAgent が起動  
- 「シアトルで週末を計画したい」と言えば activityPlannerWorkflow が実行  
- 「Y のレポートを書いてほしい」と言えば writingAgent が介入  

すべてを単一のメガエージェントに詰め込もうとするよりも、このパターンの方がスケーラビリティが高いです。専門化されたエージェントは特定領域の専門知識を蓄積し、コーディネータはルーティングだけを担当します。各コンポーネントが得意なことに専念できる構造です。

## まとめ

実稼働の AI システムには、単なるプロンプト以上のアーキテクチャが必要です。LLM がノードとして機能する分散システムを構築していると考えてください。

- ワークフローは「正確に実行すべき」処理に対して保証を提供します。  
- メモリはトークン予算を消費せずにコンテキストを保持します。  
- エージェントネットワークは、シンプルな部品を組み合わせて複雑さを構築できる手段です。

これらは派手さはありませんが、実稼働で「完全自律エージェント」が失敗するのを十分に見てきた結果、刺激的な予測不可能性よりも退屈な信頼性の方が価値があると実感しています。

成果は人それぞれですが、私の経験では、実際に出荷され長期間稼働し続けているシステムは、LLM をすべてを解決する魔法の箱ではなく、より大きなアーキテクチャのコンポーネントとして扱っているものです。
