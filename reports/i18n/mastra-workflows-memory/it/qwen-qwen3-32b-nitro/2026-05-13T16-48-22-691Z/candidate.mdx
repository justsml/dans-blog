LLMs hanno questa proprietà strana: sono brillanti nel comprendere le sfumature ma terribili nel seguire le procedure. Date a GPT-4 un problema vago e ragionerà attraverso le possibilità. Dategli una sequenza precisa di passaggi, e potrebbe saltare il passo 3 perché il passo 5 "sembrava più rilevante".

Questo non è un bug nel modello. È una caratteristica fondamentale dei sistemi probabilistici che cercano di risolvere problemi deterministici.

Ho visto team lottare con questo mismatch. Costruiscono un agente per gestire i rimborso clienti, gli danno una dozzina di strumenti e si aspettano che esegua in modo affidabile un processo aziendale. A volte funziona perfettamente. A volte immagina approvazioni che non sono mai avvenute. A volte si blocca a chiedere le stesse informazioni tre volte.

La soluzione non sono prompt migliori. È sapere quando smettere di chiedere all’LLM di "pensare" e iniziare a dirgli di "obbedire".

---

## Quando il deterministico supera il creativo

Pensate a cosa succede quando dovete gestire un ticket di supporto. La logica aziendale reale assomiglia a qualcosa del genere:

1. Recuperare i dettagli del ticket dal database
2. Verificare se l'utente è idoneo al rimborso (regole di politica)
3. Confermare che la transazione esiste e non è già stata rimborsata
4. Calcolare l'importo del rimborso
5. Eseguire l'annullamento del pagamento
6. Aggiornare lo stato del ticket
7. Inviare l'email di conferma

Potreste affidare questo a un LLM come esercizio di chiamata a strumenti. Dalla mia esperienza, però, è chiedere guai. Il modello potrebbe decidere che i passaggi 2 e 3 sono "sostanzialmente la stessa cosa" e saltarne uno. O potrebbe processare il rimborso prima di verificare l'idoneità perché l'utente sembrava preoccupato.

I workflow esistono appunto per questo scenario. Non sono entusiasmanti, ma questo è il punto.

### Costruire un pianificatore di attività meteorologiche

Ecco un esempio pratico che illustra il pattern. Abbiamo bisogno di dati meteorologici precisi e fatti, abbinati a suggerimenti creativi sulle attività. La parte relativa al meteo non dovrebbe mai essere creativa, ma i suggerimenti sì.

```typescript
// src/mastra/workflows/activity-planner.ts
import { createWorkflow, createStep } from '@mastra/core/workflows';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

// Step 1: Fetch weather data (Deterministic)
const fetchWeather = createStep({
  id: 'fetch-weather',
  description: 'Fetches weather forecast for a given city',
  inputSchema: z.object({
    city: z.string(),
  }),
  outputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  execute: async ({ inputData }) => {
    // ... (fetch logic) ...
    const weather = await fetch(`https://api.open-meteo.com/v1/forecast?latitude=52.52&longitude=13.41&current=temperature_2m,weather_code&daily=precipitation_probability_mean`).then(r => r.json());
    
    return {
      location: inputData.city,
      temperature: weather.current.temperature_2m,
      conditions: getWeatherCondition(weather.current.weather_code),
      precipitationChance: weather.daily.precipitation_probability_mean[0],
    };
  },
});

// Step 2: Agent suggests activities (Creative)
const activityPlanner = new Agent({
  id: 'activity-planner-agent',
  name: 'Activity Planner',
  instructions: `You are a local activities expert. Based on weather conditions, suggest 3-5 appropriate activities.
    - For rain (>50% precipitation), prioritize indoor activities
    - For extreme temperatures, consider climate-appropriate options
    - Always include one adventurous and one relaxing option`,
  model: openai('gpt-5'),
});

const planActivities = createStep({
  id: 'plan-activities',
  description: 'Uses AI to suggest activities based on weather',
  inputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  outputSchema: z.object({
    activities: z.string(),
  }),
  execute: async ({ inputData }) => {
    const prompt = `Weather in ${inputData.location}: ${inputData.temperature}°C...`;
    const response = await activityPlanner.generate(prompt);
    return { activities: response.text };
  },
});

// The Pipeline
export const activityPlannerWorkflow = createWorkflow({
  id: 'activity-planner',
  inputSchema: z.object({ city: z.string() }),
  outputSchema: z.object({ activities: z.string() }),
})
  .then(fetchWeather)
  .then(planActivities);

activityPlannerWorkflow.commit();
```

Il LLM non interagisce mai direttamente con l'API meteo. Riceve dati verificabili in input, quindi si concentra su ciò per cui è effettivamente adatto: fornire suggerimenti contestuali. Se inverti il processo e lasci che l'agente prenda i dati meteo, alla fine otterrai un'previsione soleggiata quando in realtà piove.

**Quando considerare i workflow:**
- Hai una sequenza nota di passaggi che devono avvenire in ordine
- Hai bisogno di osservabilità a ogni fase (log, metriche, timing)
- Devi implementare logica di retry per API esterne instabili
- Le regole aziendali non possono essere "interpretate" - devono essere seguite esattamente

## Il problema della finestra di contesto che nessuno discute

C'è questo pattern ricorrente. Qualcuno costruisce un chatbot. Funziona bene durante i test. Poi in produzione, gli utenti hanno conversazioni più lunghe e improvvisamente il bot perde l'orientamento.

Il developer guarda i log e si accorge che sta inviando l'intera storia della conversazione con ogni richiesta. Tutte e 47 le messaggi. Consuma tokens e spazio di contesto per informazioni per lo più irrilevanti.

Peggio ancora, c'è un fenomeno che i ricercatori chiamano "lost in the middle" dove i modelli prestano peggio quando informazioni rilevanti sono seppellite in un contesto lungo. Il modello non riesce letteralmente a vedere il bosco per gli alberi.

Inviare l'intera storia della conversazione sembra sicuro. Stai dando al modello "tutte le informazioni". Ma in realtà gli stai rendendo più difficile concentrarsi su ciò che conta.

### Memoria operativa vs. Memoria a lungo termine

Il sistema di memoria di Mastra ti dà entrambi. La memoria operativa mantiene i messaggi recenti nella finestra di contesto. Il ricordo semantico cerca i messaggi storici quando la query corrente sembra correlata.

```typescript
// src/mastra/agents/memory-agent.ts
import { Agent } from '@mastra/core/agent';
import { Memory } from '@mastra/memory';
import { LibSQLStore } from '@mastra/libsql';

export const memoryAgent = new Agent({
  id: 'memory-agent',
  name: 'Memory Agent',
  instructions: 'You are a helpful assistant with perfect recall of our conversations.',
  model: openai('gpt-5'),
  memory: new Memory({
    storage: new LibSQLStore({
      id: 'memory-agent-store',
      url: 'file:../mastra.db',
    }),
    options: {
      lastMessages: 20,  // Keep last 20 messages in context
      semanticRecall: {
        enabled: true,  // Use embeddings to find old stuff
        topK: 5,
        threshold: 0.7,
      },
    },
  }),
});
```

Ecco come si presenta nella pratica. Un utente chiede: "Qual era quel ristorante italiano che hai consigliato lo scorso mese?"

Senza ricordo semantico, l'agente vede gli ultimi 20 messaggi. La raccomandazione del ristorante era il messaggio 487 di 506. È andata. L'agente risponde "Non ho quell'informazione."

Con ricordo semantico:
1. La query viene incapsulata: `[0.234, -0.567, 0.891, ...]`
2. L'incapsulamento viene confrontato con i messaggi storici
3. Il messaggio 487 ("Ti consiglierei Trattoria Bella - la loro carbonara è incredibile") ottiene un punteggio di similarità di 0.89
4. Quel messaggio viene iniettato nel contesto corrente
5. L'agente risponde: "Ho consigliato Trattoria Bella. La loro carbonara è stato ciò che ha attirato la mia attenzione."

L'agente sembra avere una memoria perfetta utilizzando solo una frazione della finestra di contesto. Questo non è solo ingegneria intelligente - diventa funzionalmente necessario non appena le conversazioni superano una dozzina di messaggi.

## Coordinamento attraverso reti di agenti

A volte hai bisogno sia di struttura che di flessibilità. I flussi di lavoro puri sono troppo rigidi. Gli agenti puri sono troppo imprevedibili.

Le reti di agenti ti danno un coordinatore che decide quale agente specializzato o workflow invocare in base al compito. Pensaci come un bilanciatore di carico intelligente per le capacità AI.

```typescript
export const coordinatorAgent = new Agent({
  id: 'coordinator-agent',
  name: 'Research Coordinator',
  instructions: `Sei una rete di ricercatori e scrittori.
    - Usa researchAgent per raccogliere fatti
    - Usa writingAgent per produrre contenuti finali
    - Usa weatherTool per dati meteo correnti
    - Usa activityPlannerWorkflow per pianificazione basata su località
    
    Sempre produce risposte complete e ben strutturate.`,
  model: openai('gpt-5'),
  
  // Primitivi disponibili
  agents: { researchAgent, writingAgent },
  workflows: { activityPlannerWorkflow },
  tools: { weatherTool },
  
  // La rete richiede memoria
  memory: new Memory({
    storage: new LibSQLStore({ id: 'network-store', url: 'file:../network.db' }),
  }),
});
```

Quando interroghi questa rete, il coordinatore analizza la richiesta e instradala di conseguenza:
- "Ho bisogno di fatti su X" attiva l'agente di ricerca
- "Pianifica un fine settimana a Seattle" esegue il workflow del pianificatore di attività
- "Scrivi un rapporto su Y" coinvolge l'agente di scrittura

Questo schema si scala meglio rispetto a tentativi di comprimere tutto in un unico mega-agente. Gli agenti specializzati sviluppano expertise focalizzata. Il coordinatore gestisce l'instradamento. Ogni componente fa ciò in cui è specializzato.

## Assemblare il Tutto

Sistemi AI reali richiedono architettura, non solo prompt. Stai costruendo sistemi distribuiti dove alcuni nodi coincidono con LLM.

I workflow ti danno garanzie quando devi assicurarti che le cose accadano esattamente come previsto. La memoria ti fornisce contesto senza esaurire il tuo budget di token. Le reti di agenti ti permettono di comporre complessità da componenti semplici.

Nessuna di queste cose è glamour. Ma dopo aver visto abbastanza "agenti completamente autonomi" fallire in produzione, ho imparato ad apprezzare la noiosa affidabilità rispetto all'inattesa eccitazione.

Il tuo risultato può variare, ma dalla mia esperienza, i sistemi che effettivamente vengono rilasciati e rimangono in esecuzione sono quelli che trattano gli LLM come componenti in un'architettura più ampia, piuttosto che come scatole magiche che risolvono tutto.
