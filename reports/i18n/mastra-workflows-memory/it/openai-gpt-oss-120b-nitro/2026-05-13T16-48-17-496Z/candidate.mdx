LLMhanno questa strana proprietà: sono brillanti nel comprendere le sfumature ma terribili nel seguire ricette. Dai a GPT‑4 un problema vago e ragionerà sulle possibilità. Dagli una sequenza precisa di passaggi, e potrebbe saltare il passo 3 perché il passo 5 “sembra più rilevante”.

Non è un bug del modello. È una caratteristica fondamentale dei sistemi probabilistici che cercano di risolvere problemi deterministici.

Ho visto team lottare con questo disallineamento. Costruiscono un agente per gestire i rimborsi dei clienti, gli danno una dozzina di strumenti e si aspettano che esegua in modo affidabile un processo aziendale. A volte funziona perfettamente. A volte allunga approvazioni che non sono mai avvenute. A volte si blocca chiedendo la stessa informazione tre volte.

La soluzione non sono prompt migliori. È sapere quando smettere di chiedere al LLM di “pensare” e cominciare a dirgli di “obbedire”.

## Quando il Deterministico Batte il Creativo

Pensate a cosa succede quando dovete gestire un ticket di supporto. La logica aziendale reale appare più o meno così:

1. Recuperare i dettagli del ticket dal database  
2. Verificare se l'utente è idoneo a un rimborso (regole di policy)  
3. Controllare che la transazione esista e non sia già stata rimborsata  
4. Calcolare l'importo del rimborso  
5. Eseguire l'annullamento del pagamento  
6. Aggiornare lo stato del ticket  
7. Inviare l'email di conferma  

Potreste affidare tutto questo a un LLM come esercizio di tool‑calling. Nella mia esperienza, è una ricetta per i guai. Il modello potrebbe decidere che i passaggi 2 e 3 sono “praticamente la stessa cosa” e saltarne uno. Oppure potrebbe processare il rimborso prima di verificare l'idoneità perché l'utente sembrava irritato.

I workflow esistono proprio per questo scenario. Non sono entusiasmanti, ma è questo il punto.

### Costruire un Planner di Attività basato sul Meteo

Ecco un esempio pratico che illustra il pattern. Abbiamo bisogno di dati meteo concreti e verificabili accoppiati a suggerimenti creativi di attività. Il recupero del meteo non deve mai essere creativo, mentre i suggerimenti sì.

```typescript
// src/mastra/workflows/activity-planner.ts
import { createWorkflow, createStep } from '@mastra/core/workflows';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

// Step 1: Fetch weather data (Deterministic)
const fetchWeather = createStep({
  id: 'fetch-weather',
  description: 'Fetches weather forecast for a given city',
  inputSchema: z.object({
    city: z.string(),
  }),
  outputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  execute: async ({ inputData }) => {
    // ... (fetch logic) ...
    const weather = await fetch(`https://api.open-meteo.com/v1/forecast?latitude=52.52&longitude=13.41&current=temperature_2m,weather_code&daily=precipitation_probability_mean`).then(r => r.json());
    
    return {
      location: inputData.city,
      temperature: weather.current.temperature_2m,
      conditions: getWeatherCondition(weather.current.weather_code),
      precipitationChance: weather.daily.precipitation_probability_mean[0],
    };
  },
});

// Step 2: Agent suggests activities (Creative)
const activityPlanner = new Agent({
  id: 'activity-planner-agent',
  name: 'Activity Planner',
  instructions: `You are a local activities expert. Based on weather conditions, suggest 3-5 appropriate activities.
    - For rain (>50% precipitation), prioritize indoor activities
    - For extreme temperatures, consider climate-appropriate options
    - Always include one adventurous and one relaxing option`,
  model: openai('gpt-5'),
});

const planActivities = createStep({
  id: 'plan-activities',
  description: 'Uses AI to suggest activities based on weather',
  inputSchema: z.object({
    location: z.string(),
    temperature: z.number(),
    conditions: z.string(),
    precipitationChance: z.number(),
  }),
  outputSchema: z.object({
    activities: z.string(),
  }),
  execute: async ({ inputData }) => {
    const prompt = `Weather in ${inputData.location}: ${inputData.temperature}°C...`;
    const response = await activityPlanner.generate(prompt);
    return { activities: response.text };
  },
});

// The Pipeline
export const activityPlannerWorkflow = createWorkflow({
  id: 'activity-planner',
  inputSchema: z.object({ city: z.string() }),
  outputSchema: z.object({ activities: z.string() }),
})
  .then(fetchWeather)
  .then(planActivities);

activityPlannerWorkflow.commit();
```

Il LLM non tocca mai l'API meteo. Riceve dati di verità di base come input, poi fa ciò in cui è realmente efficace: fornire suggerimenti contestuali. Se si inverte il flusso e si lascia che l'agente recuperi i dati meteo, prima o poi otterrete una previsione di sole quando in realtà sta piovendo.

**Quando considerare i workflow:**
- Hai una sequenza nota di passaggi che devono avvenire in ordine
- Hai bisogno di osservabilità a ogni stadio (log, metriche, tempi)
- Hai bisogno di logica di retry per API esterne inaffidabili
- Le regole di business non possono essere “interpretate” – devono essere rispettate alla lettera

## Il Problema della Finestra di Contesto di cui Nessuno Parla

C'è un pattern che vedo ripetersi. Qualcuno costruisce un chatbot. Funziona alla grande nei test. Poi, in produzione, gli utenti hanno conversazioni più lunghe e improvvisamente il bot si perde.

Il developer controlla i log e si accorge che sta inviando l'intera cronologia della conversazione ad ogni richiesta. Tutti i 47 messaggi. Sta bruciando token e spazio di contesto per informazioni che sono per lo più irrilevanti.

Peggio ancora, esiste un fenomeno che i ricercatori chiamano "lost in the middle", in cui i modelli peggiorano quando le informazioni rilevanti sono sepolte in un contesto lungo. Il modello letteralmente non riesce a vedere la foresta per gli alberi.

Inviare l'intera cronologia della conversazione sembra una scelta sicura. Stai dando al modello "tutte le informazioni". In realtà lo rendi più difficile concentrarsi su ciò che è importante.

### Memoria di lavoro vs. archiviazione a lungo termine

Il sistema di memoria di Mastra ti fornisce entrambi. La memoria di lavoro mantiene i messaggi recenti nella finestra di contesto. Il richiamo semantico ricerca nei messaggi storici quando la query corrente sembra correlata.

```typescript
// src/mastra/agents/memory-agent.ts
import { Agent } from '@mastra/core/agent';
import { Memory } from '@mastra/memory';
import { LibSQLStore } from '@mastra/libsql';

export const memoryAgent = new Agent({
  id: 'memory-agent',
  name: 'Memory Agent',
  instructions: 'You are a helpful assistant with perfect recall of our conversations.',
  model: openai('gpt-5'),
  memory: new Memory({
    storage: new LibSQLStore({
      id: 'memory-agent-store',
      url: 'file:../mastra.db',
    }),
    options: {
      lastMessages: 20,  // Keep last 20 messages in context
      semanticRecall: {
        enabled: true,  // Use embeddings to find old stuff
        topK: 5,
        threshold: 0.7,
      },
    },
  }),
});
```

Ecco come si traduce in pratica. Un utente chiede: "Qual era il ristorante italiano che mi hai consigliato il mese scorso?"

Senza richiamo semantico, l'agente vede solo gli ultimi 20 messaggi. Il consiglio del ristorante era il messaggio 487 su 506. È sparito. L'agente risponde "Non ho quell'informazione".

Con richiamo semantico:
1. La query viene embeddata: `[0.234, -0.567, 0.891, ...]`
2. L'embedding viene confrontato con i messaggi storici
3. Il messaggio 487 ("Consiglierei Trattoria Bella – la loro carbonara è incredibile") ottiene una similarità di 0,89
4. Quel messaggio viene inserito nel contesto corrente
5. L'agente risponde: "Ho consigliato Trattoria Bella. La loro carbonara è ciò che mi ha colpito."

L'agente sembra avere una memoria perfetta pur usando solo una frazione della finestra di contesto. Non è solo un trucco ingegneristico – è praticamente necessario quando le conversazioni superano qualche decina di messaggi.

## Coordinamento tramite Reti di Agenti

A volte servono sia struttura che flessibilità. I workflow puri sono troppo rigidi. Gli agenti puri sono troppo imprevedibili.

Le reti di agenti forniscono un coordinatore che decide quale agente specializzato o workflow invocare in base al compito. Pensalo come un bilanciatore di carico intelligente per le capacità AI.

```typescript
export const coordinatorAgent = new Agent({
  id: 'coordinator-agent',
  name: 'Research Coordinator',
  instructions: `You are a network of researchers and writers.
    - Use researchAgent for gathering facts
    - Use writingAgent for producing final content
    - Use weatherTool for current weather data
    - Use activityPlannerWorkflow for location-based planning
    
    Always produce comprehensive, well-structured responses.`,
  model: openai('gpt-5'),
  
  // Available primitives
  agents: { researchAgent, writingAgent },
  workflows: { activityPlannerWorkflow },
  tools: { weatherTool },
  
  // Network requires memory
  memory: new Memory({
    storage: new LibSQLStore({ id: 'network-store', url: 'file:../network.db' }),
  }),
});
```

Quando interroghi questa rete, il coordinatore analizza la richiesta e instrada di conseguenza:
- “Ho bisogno di fatti su X” attiva l'agente di ricerca
- “Pianifica un weekend a Seattle” esegue il workflow di pianificazione delle attività
- “Scrivi un rapporto su Y” coinvolge l'agente di scrittura

Questo modello scala meglio rispetto al tentativo di comprimere tutto in un unico mega‑agente. Gli agenti specializzati sviluppano competenze focalizzate. Il coordinatore gestisce il routing. Ogni componente fa ciò in cui è bravo.

## Mettere tutto insieme

I sistemi AI di produzione reale richiedono un'architettura, non solo prompt. Stai costruendo sistemi distribuiti in cui alcuni nodi sono LLM.

I workflow ti forniscono garanzie quando è necessario che le operazioni avvengano esattamente come previsto. La memoria ti dà contesto senza consumare il budget di token. Le reti di agenti ti permettono di comporre complessità a partire da parti più semplici.

Niente di glamour. Ma dopo aver visto fallire in produzione un numero sufficiente di “agent completamente autonomi”, ho imparato a preferire l'affidabilità noiosa all'imprevedibilità eccitante.

Il risultato può variare, ma nella mia esperienza i sistemi che effettivamente vengono rilasciati e rimangono operativi sono quelli che trattano i LLM come componenti di un'architettura più ampia, non come scatole magiche che risolvono tutto.
