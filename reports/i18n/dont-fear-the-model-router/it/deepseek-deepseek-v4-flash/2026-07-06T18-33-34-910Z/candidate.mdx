The first version of [Non sposare il tuo modello](/llm-routing-mastra-ai) presentava un argomento semplice: smetti di inviare ogni compito allo stesso modello solo perché ha vinto l'ultimo confronto.

Usa un modello economico per lavori economici. Usa un modello più potente dove il lavoro è effettivamente complesso. Mantieni il livello di routing abbastanza flessibile da poter cambiare provider senza trasformare il tuo codebase in un santuario.

Quella era la scelta giusta.

Ma era anche incompleta.

Perché una volta aggiunto un router, hai un nuovo comportamento di sistema da testare. La domanda non è più "qual è il modello migliore?" La domanda è "il sistema ha scelto la rotta giusta, ha usato gli strumenti giusti, ha preservato le prove giuste e si è fermato al momento giusto?"

Se non lo misuri, il tuo router di modelli è solo sensazioni con una tabella di instradamento.

<p class="inset">
Il router non è la risposta. Il router è un'ipotesi su come il tuo sistema dovrebbe comportarsi.
</p>

Mastra ci offre superfici utili per trasformare quell'ipotesi in qualcosa di testabile: [scorer](https://mastra.ai/docs/evals/overview), [`runEvals`](https://mastra.ai/reference/evals/run-evals), [dataset](https://mastra.ai/docs/evals/datasets/overview) ed [esperimenti](https://mastra.ai/docs/evals/datasets/running-experiments). I nomi delle API sembrano infrastruttura di valutazione, e lo sono, ma il vero valore è più semplice:

Rendono il comportamento dell'agente abbastanza visibile da poterlo discutere.

## Cosa stiamo testando?

Il router di modelli del post precedente ha tre rotte specialistiche ovvie:

| Rotta | Cosa dovrebbe andare qui | Cosa sarebbe una rotta sbagliata |
|---|---|---|
| `code` | implementazione, refactoring, debugging, code review | riassunto a lungo contesto, classificazione semplice |
| `long-context` | documenti disordinati, trascrizioni, sintesi di policy, molti file | formattazione meccanica breve |
| `general` | classificazione, formattazione, semplici Q&A, estrazione noiosa | codice complesso o analisi con molte prove |

Quella tabella è un inizio, ma non è una valutazione. Una valutazione ha bisogno di esempi e scorer:

| Parte | Ruolo |
|---|---|
| Dataset item | "Ecco una richiesta rappresentativa." |
| Ground truth | "Ecco la rotta o il comportamento che ci aspettavamo." |
| Scorer | "Ecco come decidiamo se l'output ha superato il controllo." |
| Experiment | "Ecco l'esecuzione che possiamo confrontare con esecuzioni future." |

La mossa importante è testare il comportamento, non solo la qualità della prosa. Un modello può scrivere una risposta bellissima dopo aver scelto lo specialista sbagliato. Un agente di sicurezza può produrre un rapporto plausibile senza preservare le prove. Un agente di supporto può sembrare empatico saltando il controllo della politica di rimborso. Il paragrafo è la parte visibile. È nella traiettoria che vivono i bug.

Per un router, di solito parto da quattro assi:

| Asse | Domanda | Scorer di esempio |
|---|---|---|---|
| Qualità | Ha scelto la rotta giusta e prodotto un risultato utile? | accuratezza della rotta, completezza della risposta, fedeltà |
| Costo | Ha evitato modelli premium per lavori noiosi? | classe di costo della rotta selezionata, budget di token |
| Velocità | Ha terminato entro il budget di latenza del prodotto? | scorer di runtime o timeout |
| Altro | Ha rispettato i vincoli di sicurezza, privacy e osservabilità? | allowlist di strumenti, preservazione delle prove, comportamento di rifiuto |

Quell'ultima colonna conta. "Altro" è dove vive il tessuto cicatriziale della produzione.

## Rendere la Decisione del Router Valutabile

Se il router produce solo una risposta finale, è difficile capire perché si è comportato in un certo modo. Puoi ancora valutare l'output, ma stai indovinando la decisione.

Per le valutazioni, dai al passo di routing un piccolo contratto strutturato:

```typescript
type RouterDecision = {
  route: "code" | "long-context" | "general";
  confidence: number;
  reason: string;
};
```

Il sistema di produzione non ha bisogno di mostrare questo JSON agli utenti. Può essere un passaggio interno, un handoff del workflow o uno span di traccia. Lo scorer ha solo bisogno di una superficie.

Ecco un agente Mastra volutamente piccolo che sceglie una rotta:

```typescript
// src/mastra/agents/router-decision-agent.ts
import { Agent } from "@mastra/core/agent";

export const routerDecisionAgent = new Agent({
  id: "router-decision-agent",
  name: "Router Decision Agent",
  instructions: `Choose the best specialist route for the user request.

Return ONLY JSON:
{
  "route": "code" | "long-context" | "general",
  "confidence": number,
  "reason": string
}

Routing rules:
- code: implementation, refactoring, debugging, code review, APIs, tests
- long-context: large documents, transcripts, policy synthesis, many files
- general: classification, formatting, extraction, simple Q&A

Do not answer the user request. Only choose the route.`,
  model: process.env.ROUTER_MODEL ?? "openai/gpt-5-mini",
});
```

Sì, è un po' artificiale. Bene. Le valutazioni premiano le giunture noiose.

Quando la decisione del router è esplicita, puoi testare la rotta prima di testare lo specialista a valle. È così che scopri se il problema è il router, il modello scelto, il prompt, la superficie degli strumenti o lo scorer della risposta finale.

## Scrivere uno Scorer che Catturi il Fallimento Noioso

Il [`createScorer`](https://mastra.ai/reference/evals/create-scorer) di Mastra può usare funzioni JavaScript, prompt giudice LLM, o entrambi. Inizia con le funzioni ogni volta che il fallimento è deterministico. Sono più economici, più veloci e meno misteriosi.

Per l'accuratezza della rotta, non abbiamo bisogno di un modello giudice. Dobbiamo parsare JSON e confrontare un campo.

```typescript
// src/mastra/scorers/route-accuracy.ts
import { createScorer } from "@mastra/core/evals";

type Route = "code" | "long-context" | "general";
type RouteGroundTruth = {
  route: Route;
  mustMention?: string[];
};

function textFromAgentOutput(output: Array<{ content?: unknown }>) {
  const content = output[0]?.content;
  return typeof content === "string" ? content : JSON.stringify(content ?? "");
}

function parseDecision(output: Array<{ content?: unknown }>) {
  try {
    return JSON.parse(textFromAgentOutput(output)) as {
      route?: string;
      confidence?: number;
      reason?: string;
    };
  } catch {
    return {};
  }
}

export const validRouterJsonScorer = createScorer({
  id: "valid-router-json",
  description: "Checks that the router emits a valid decision object.",
  type: "agent",
})
  .generateScore(({ run }) => {
    const decision = parseDecision(run.output);
    const validRoute = ["code", "long-context", "general"].includes(
      decision.route ?? "",
    );
    const validConfidence =
      typeof decision.confidence === "number" &&
      decision.confidence >= 0 &&
      decision.confidence <= 1;

    return validRoute && validConfidence && decision.reason ? 1 : 0;
  })
  .generateReason(({ score }) =>
    score === 1 ? "Valid router decision." : "Router output was not valid JSON.",
  );

export const routeAccuracyScorer = createScorer({
  id: "route-accuracy",
  description: "Checks whether the selected route matches ground truth.",
  type: "agent",
})
  .generateScore(({ run }) => {
    const expected = run.groundTruth as RouteGroundTruth;
    const decision = parseDecision(run.output);
    return decision.route === expected.route ? 1 : 0;
  })
  .generateReason(({ run, score }) => {
    const expected = run.groundTruth as RouteGroundTruth;
    const decision = parseDecision(run.output);

    return score === 1
      ? `Selected expected route: ${expected.route}.`
      : `Expected ${expected.route}, got ${decision.route ?? "nothing"}.`;
  });
```

Quel scorer non è affascinante. Questo è il punto.

Se il router non riesce a produrre costantemente JSON valido e a scegliere lo specialista ovvio su un piccolo set di test, non c'è motivo di fidarsi di lui per il traffico di produzione. Non hai bisogno di un'ontologia di valutazione di un modello filosofo. Hai bisogno dell'equivalente di un rilevatore di fumo con la batteria.

## Eseguire Prima il Piccolo Ciclo di Valutazione

Il [`runEvals`](https://mastra.ai/reference/evals/run-evals) di Mastra è il ciclo veloce. Dagli un target, casi di test, scorer e un limite di concorrenza. Esegue il target sui dati e restituisce punteggi aggregati.

```typescript
// src/mastra/evals/router.eval.ts
import { runEvals } from "@mastra/core/evals";
import { routerDecisionAgent } from "../agents/router-decision-agent";
import {
  routeAccuracyScorer,
  validRouterJsonScorer,
} from "../scorers/route-accuracy";

const routingCases = [
  {
    input: "Refactor this React component to remove duplicated state.",
    groundTruth: { route: "code" },
  },
  {
    input: "Summarize these 14 interview transcripts and find recurring objections.",
    groundTruth: { route: "long-context" },
  },
  {
    input: "Classify this ticket as billing, technical, account, or other.",
    groundTruth: { route: "general" },
  },
  {
    input: "Debug a failing Playwright test that only breaks in CI.",
    groundTruth: { route: "code" },
  },
  {
    input: "Extract the renewal date and contract value from this short paragraph.",
    groundTruth: { route: "general" },
  },
];

const result = await runEvals({
  target: routerDecisionAgent,
  data: routingCases,
  scorers: [validRouterJsonScorer, routeAccuracyScorer],
  targetOptions: {
    modelSettings: { temperature: 0 },
  },
  concurrency: 3,
});

console.log(result.scores);
console.log(result.summary.totalItems);

if (result.scores["valid-router-json"] < 1) {
  throw new Error("Router emitted invalid decision JSON.");
}

if (result.scores["route-accuracy"] < 0.9) {
  throw new Error("Router route accuracy fell below 90%.");
}
```

Questo è il ciclo che esegui mentre cambi il prompt, aggiungi una nuova rotta o provi un modello router più economico.

Non è abbastanza per un sistema maturo, ma è sufficiente per prevenire la regressione più imbarazzante: "abbiamo cambiato il prompt del router e ha iniziato a inviare compiti di classificazione al modello di codice premium."

Costo, velocità, qualità e altri aspetti compaiono qui:

- Costo: il modello router può rimanere economico se l'accuratezza è mantenuta.
- Velocità: la valutazione può imporre timeout o registrare la latenza nell'harness.
- Qualità: l'accuratezza della rotta e la qualità della risposta finale sono punteggi separati.
- Altro: validità JSON, strumenti consentiti, sicurezza e tracciabilità ottengono i propri controlli.

Non riunire tutto in un unico punteggio di "qualità". Le medie sono dove i fallimenti utili vanno in pensione.

## Aggiungi un giudice LLM solo dove vale la pena

Alcuni comportamenti del router sono soggettivi. Una richiesta può essere legittimamente ambigua:

```text
Read these logs and tell me why the deploy failed.
```

È `code` perché debugging? `long-context` perché log? `general` perché riassunto? La rotta giusta dipende dalla superficie degli strumenti e dalla promessa del tuo prodotto.

È qui che un giudice LLM può aiutare, ma solo con una rubrica rigorosa. Gli scorer Mastra possono mescolare passi di funzione e passi di prompt-object. Usa funzioni per la struttura, poi un giudice per la parte che richiede effettivamente giudizio.

```typescript
// src/mastra/scorers/route-reasonableness.ts
import { createScorer } from "@mastra/core/evals";
import { z } from "zod";

export const routeReasonablenessScorer = createScorer({
  id: "route-reasonableness",
  description: "Judges whether the route explanation matches the request.",
  type: "agent",
  judge: {
    model: process.env.JUDGE_MODEL ?? "openai/gpt-5-mini",
    instructions: "You are a strict evaluator for model-routing decisions.",
  },
})
  .analyze({
    description: "Evaluate the router's decision rationale.",
    outputSchema: z.object({
      score: z.number().min(0).max(1),
      rationale: z.string(),
    }),
    createPrompt: ({ run }) => `
User request:
${JSON.stringify(run.input)}

Router output:
${JSON.stringify(run.output)}

Score from 0 to 1.

1.0 = route is clearly appropriate and the reason cites the right task signals
0.5 = route is defensible but underspecified or ambiguous
0.0 = route is wrong, unsupported, or the reason is unrelated

Return JSON with { "score": number, "rationale": string }.
`,
  })
  .generateScore(({ results }) => results.analyzeStepResult.score)
  .generateReason(({ results }) => results.analyzeStepResult.rationale);
```

Questo scorer costa denaro perché chiama un modello giudice. Va bene quando il giudizio ne vale la pena.

Non usarlo per verificare se il JSON viene parsificato.

## Promuovi i casi validi in un dataset

Array di valutazione hard-coded vanno bene all'inizio. Col tempo, i tuoi esempi diventano risorse di prodotto. Il ticket cliente fallito, la strana conversazione di supporto, il tentativo di injection del prompt, la richiesta che prima di giovedì scorso veniva instradata correttamente.

Quello appartiene a un dataset.

I dataset Mastra sono raccolte versionate di casi di test. Ogni mutazione crea una nuova versione, il che significa che puoi rieseguire un esperimento contro l'esatto insieme di casi che esisteva quando hai preso una decisione sul modello.

Prima configura lo storage, perché i dataset necessitano di persistenza:

```typescript
// src/mastra/index.ts
import { Mastra } from "@mastra/core";
import { LibSQLStore } from "@mastra/libsql";
import { routerDecisionAgent } from "./agents/router-decision-agent";
import {
  routeAccuracyScorer,
  validRouterJsonScorer,
} from "./scorers/route-accuracy";

export const mastra = new Mastra({
  storage: new LibSQLStore({
    id: "router-evals",
    url: "file:./mastra.db",
  }),
  agents: {
    routerDecisionAgent,
  },
  scorers: {
    validRouterJson: validRouterJsonScorer,
    routeAccuracy: routeAccuracyScorer,
  },
});
```

Poi crea un dataset e aggiungi casi:

```typescript
// src/mastra/evals/create-router-dataset.ts
import { z } from "zod";
import { mastra } from "../index";

const dataset = await mastra.datasets.create({
  name: "router-decisions-v1",
  description: "Representative model-router decisions for CI and experiments.",
  inputSchema: z.string(),
  groundTruthSchema: z.object({
    route: z.enum(["code", "long-context", "general"]),
    source: z.string().optional(),
  }),
});

await dataset.addItems({
  items: [
    {
      input: "Refactor this React component to remove duplicated state.",
      groundTruth: { route: "code", source: "synthetic:happy-path" },
    },
    {
      input: "Summarize these 14 interview transcripts and find recurring objections.",
      groundTruth: { route: "long-context", source: "synthetic:happy-path" },
    },
    {
      input: "Classify this ticket as billing, technical, account, or other.",
      groundTruth: { route: "general", source: "synthetic:happy-path" },
    },
  ],
});
```

Nel momento in cui hai un dataset, puoi smettere di trattare i casi di valutazione come dati di script usa e getta. Ora hanno ID, versioni, cronologia e risultati di esperimenti.

È allora che le valutazioni iniziano a sembrare meno "file di test per prompt" e più una memoria di prodotto.

## Esegui esperimenti contro il router

Una volta che il dataset esiste, usa [`dataset.startExperiment()`](https://mastra.ai/reference/datasets/startExperiment) per eseguirlo su un agente, workflow o scorer registrato.

```typescript
// src/mastra/evals/run-router-experiment.ts
import { mastra } from "../index";

const dataset = await mastra.datasets.get({ id: process.env.ROUTER_DATASET_ID! });

const summary = await dataset.startExperiment({
  name: "router-gpt-5-mini-baseline",
  description: "Baseline router decision run before adding security route.",
  targetType: "agent",
  targetId: "router-decision-agent",
  scorers: ["validRouterJson", "routeAccuracy"],
  metadata: {
    routerModel: process.env.ROUTER_MODEL ?? "openai/gpt-5-mini",
    promptVersion: "router-2026-07-03",
  },
  maxConcurrency: 5,
  itemTimeout: 30_000,
  maxRetries: 1,
});

console.log(`${summary.succeededCount}/${summary.totalItems} items succeeded`);

for (const item of summary.results) {
  const scores = Object.fromEntries(
    item.scores.map((score) => [score.scorerId, score.score]),
  );

  console.log(item.itemId, item.output, scores);
}
```

Ora la conversazione cambia.

Invece di “il nuovo router sembra migliore”, puoi dire:

- Il vecchio router ha ottenuto un punteggio di `0.94` in accuratezza di rotta.
- Il nuovo router ha ottenuto `0.98` complessivi.
- Ha migliorato il routing a contesto lungo.
- Ha peggiorato due casi di code review.
- Ha ridotto i passaggi a modelli premium del 18%.
- Ha aggiunto 300ms di latenza del router.

Questa è una conversazione ingegneristica. Esistono compromessi. Puoi decidere se vale la pena accettarli.

## Valutare il comportamento in produzione, senza confonderlo con la verità di base

Mastra può anche collegare scorer direttamente ad agenti e step dei workflow. Gli scorer in produzione vengono eseguiti in modo asincrono e memorizzano i risultati dei punteggi nel database configurato, con controlli di campionamento per non valutare ogni risposta in produzione se non lo si desidera.

È utile, ma è un compito diverso.

```typescript
import { Agent } from "@mastra/core/agent";
import { validRouterJsonScorer } from "../scorers/route-accuracy";

export const routerDecisionAgent = new Agent({
  id: "router-decision-agent",
  instructions: "Choose the best specialist route...",
  model: process.env.ROUTER_MODEL ?? "openai/gpt-5-mini",
  scorers: {
    validRouterJson: {
      scorer: validRouterJsonScorer,
      sampling: { type: "ratio", rate: 1 },
    },
  },
});
```

La valutazione in produzione può dirti che il router sta ancora emettendo decisioni valide. Può intercettare output malformati, contenuti tossici, chiamate a tool vietate, marcatori di evidenza mancanti o fiducia sospettosamente bassa.

Di solito non può dirti l'accuratezza della rotta, perché il traffico di produzione non arriva con la verità di base allegata.

Questa distinzione è importante. La valutazione in produzione è monitoraggio. Gli esperimenti su dataset sono test controllati. Vuoi entrambi, ma rispondono a domande diverse.

## Cosa misurare dopo l'accuratezza della rotta

L'accuratezza della rotta è il primo gradino. Ti dice se la richiesta è andata allo specialista previsto. Non ti dice se lo specialista ha fatto un buon lavoro.

Dopo che il router supera le basi, valuta il sistema a strati:

| Layer | Cosa valutare | Perché è importante |
|---|---|---|
| Decisione del router | rotta selezionata, confidenza, motivo | Intercetta classificazioni errate e regole di escalation sbagliate |
| Traiettoria | sequenza prevista di tool o agent | Intercetta comportamenti “risposta giusta, percorso sbagliato” |
| Output dello specialista | correttezza, fedeltà, utilità | Intercetta lavoro di bassa qualità dopo un routing corretto |
| Costo e latenza | scelta del modello, token, runtime | Intercetta vittorie costose o lente |
| Sicurezza e ambito | tool consentiti, confini di rifiuto, evidenza | Intercetta fallimenti a rischio prodotto |

L'API `runEvals` di Mastra supporta configurazioni di scorer a livello di agente, workflow, step e traiettoria. Ciò significa che non devi fingere che la risposta finale sia l'unico artefatto.

Per un workflow, la struttura può essere così:

```typescript
const result = await runEvals({
  target: supportWorkflow,
  data: supportCases,
  scorers: {
    workflow: [finalAnswerQualityScorer],
    steps: {
      "route-request": [routeAccuracyScorer],
      "check-policy": [policyGroundingScorer],
    },
    trajectory: [expectedPathScorer],
  },
});
```

Questo è il modello mentale che voglio per gli agenti in produzione:

Valuta la decisione. Valuta il percorso. Valuta la risposta.

Se valuti solo la risposta, il modello può passare per caso.

## Il router dovrebbe diventare più noioso col tempo

Il primo prompt di routing è di solito un paragrafo di valutazioni soggettive. Va bene per un prototipo.

Man mano che impari dalle valutazioni, parti del router dovrebbero diventare meno magiche:

- I casi lessicali chiari possono diventare regole deterministiche.
- I compiti rischiosi possono richiedere un'approvazione esplicita o un ramo del workflow.
- I compiti ambigui possono porre una domanda chiarificatrice invece di indovinare.
- Le route costose possono richiedere una maggiore confidenza o un secondo segnale.
- I casi di fallimento noti possono diventare elementi del dataset.

L'obiettivo non è rendere il router "più intelligente" per sempre. L'obiettivo è rendere il sistema più facile da ragionare.

A volte significa un modello migliore. A volte significa un prompt più stringente. A volte significa un passo del workflow, uno scorer, un limite duro, o una noiosa istruzione `if` che ti fa risparmiare quattro cifre al mese.

Questo è il senso di misurare il comportamento. Smetti di discutere basandoti sul gusto e inizi a discutere basandoti sulle evidenze.

## Una checklist pratica per iniziare

Se stai costruendo un router Mastra oggi, inizierei da qui:

1. Rendi strutturata la decisione di routing, anche se gli utenti non la vedono mai.
2. Scrivi scorer deterministici per JSON valido, route prevista e route vietate.
3. Usa `runEvals` con 10-20 casi prima di cambiare prompt o modelli del router.
4. Promuovi i fallimenti reali in un dataset versionato.
5. Esegui esperimenti sul dataset per modifiche significative a prompt, modello, route o workflow.
6. Aggiungi scorer live per invarianti di produzione a basso costo.
7. Confronta gli esperimenti per route, non solo per punteggio medio.

La media conta meno del cluster di fallimenti.

Se ogni regressione è nella sintesi di policy a contesto lungo, non hai "un router peggiore". Hai un problema di confine di route. Se ogni caso fallito usa uno strumento specifico, hai un problema di contratto dello strumento. Se ogni modello economico fallisce negli stessi due casi ambigui, potresti aver bisogno di una logica di escalation invece di un default più costoso.

Qui è dove le valutazioni diventano utili. Non come una cerimonia. Non come una dashboard che fa sentire tutti temporaneamente adulti.

Come un modo per trovare la forma del sistema.
