最昂贵的数据库优化措施我曾见过，始于某人移除了所有外键。

不是因为他们测量到性能瓶颈。不是因为写入操作真的变慢了。而是因为他们读到某处说“外键无法扩展”。六个月后，他们积累了20亿条孤儿记录，计费系统错误地向已删除用户收费，分析数据偏差了40%。

当他们试图重新添加这些约束时？数据库在验证已损坏的数据时完全瘫痪了。

在Web开发中普遍存在一种观念，认为外键本质上很慢，是当你升级到“真正的”系统时会移除的训练轮。但这完全误解了约束的真正作用。你不是在速度和慢速之间做选择。你是在选择不同的失败模式。

这样思考：安全玻璃、安全带和气囊都会增加汽车的重量。它们确实会让车辆变慢且油耗增加。但你不会为了优化0-60加速时间而拆除它们，因为你在优化的是完全不同的东西。

问题不在于外键是否拖慢你。当然会。问题是你得到了什么回报，以及你是否真的需要它。

## 你真正权衡的内容

让我举个具体例子。你正在构建一个天气监测系统，包含天气站、传感器设备、传感器读数和美国各州的表格。

你是否要将所有内容通过外键关联？让我们思考实际变化的内容和后果：

美国各州的数据很可能不会改变。怀俄明州短期内不会被改名。当你知道参考数据是静态的时，不需要在每次插入时都用外键验证州代码。这完全是无意义的开销。

天气站会被添加、移动和退役。但这里有个问题：如果有人意外删除了一个天气站记录，你希望历史读数“失去”该站点吗？也许你希望即使站点不存在了，这些数据仍保持完整。这意味着你将读数视为历史快照而非实时引用，这会改变外键是否合理的判断。

传感器读数每分钟被插入数千次。每次外键检查都意味着一次查询。每次查询都会在表上产生争用。如果缓慢的验证导致插入队列积压并丢失实时数据，这与孤儿记录的丢失是完全不同的数据丢失类型。

你能看出趋势了。选择不是抽象概念上的性能与正确性之争。而是根据你的实际约束和实际后果，你更愿意容忍哪种具体失败。

如果错误的引用意味着计费数据损坏或违反监管要求，你可能需要外键保护，无论性能成本如何。如果缓慢的验证导致实时传感器数据永久丢失（因为队列溢出），那么验证可能不是正确的权衡。

## 何时写入速度真正重要

假设你决定需要最大写入速度。你的队列堆积了，事务超时了，外键检查确实造成了你实际测量到的问题（而不仅仅是理论推测）。

你有几个选项。你可以将事务隔离级别从`SERIALIZABLE`改为`READ COMMITTED`，这会更快但牺牲部分一致性保证。你可以批量提交，每次事务插入1000行而不是单行，以分摊外键的开销。或者你可以反规范化为仅追加的日志结构，甚至不再尝试验证引用。

顺便说一句，第三种选项并不是作弊。这只是不同的设计：

```sql
CREATE TABLE sensor_log (
  id BIGSERIAL PRIMARY KEY,
  recorded_at TIMESTAMPTZ NOT NULL,
  data JSONB NOT NULL  -- { station_id, sensor_id, temp, humidity, ... }
);

CREATE INDEX ON sensor_log USING GIN (data);
CREATE INDEX ON sensor_log (recorded_at);
```

没有连接操作。没有外键检查。只需追加数据并通过时间范围或JSONB字段的GIN索引进行查询。这是否符合"最佳实践"？在数据库教科书意义上可能不是。但当你在树莓派上每分钟插入5万条记录时，它确实能正常工作。

分歧往往出现在人们将"最佳实践"视为道德命令，而非在常见场景下有效但可能不适用于你特定场景的模式。

## 规范化的陷阱

数据库课程热衷于教授规范化。不惜一切代价避免重复。非到第三范式誓不罢休。

于是你得到类似这样的结构：`Orders` → `OrderItems` → `Products` → `Variants` → `Colors` → `Sizes`

仅仅为了回答"去年圣诞节我订购的是红色还是蓝色T恤？"就需要进行六次表连接。更别提如果需要包含产品名称，还要再进行三次连接才能从目录层级中获取。

但等等。这种设计的正当性通常基于"如果品牌改变了颜色标签怎么办？" 如果这种情况真的发生，你真的希望历史订单的颜色信息被追溯性修改吗？当然不。当用户下单时，他们购买的是当时存在的"蓝色T恤，M码"，而不是某个可能后续更新的目录条目抽象引用。

这一点值得深入思考，因为其中的微妙之处容易被忽视。有些数据本质上是快照，而非引用。当你把快照数据当作实时引用处理时，就会产生这种荒谬的连接链条，而这些本应在写入时就进行反规范化处理。

直接在订单中存储`{"color": "blue", "size": "M"}`。这就完成了。

### 识别快照数据

如何判断某数据应作为快照处理？问问自己这是否是某个时间点的记录：

订单捕获购买时的产品详情。审计日志记录用户执行操作时的状态。历史表保留更新前的记录状态。事件流捕获发生的事情、时间以及相关数据。

如果答案是"是的，这在记录某个时间点"，请停止规范化。开始进行快照处理。

### 不透明数据块

还有另一类数据：你永远不会查询其内部字段的数据。你只是存储和整块检索。

像`{"model": "gpt-4", "temperature": 0.7, "max_tokens": 2000}`这样的LLM模型配置，你不会按温度值查询。当需要时，你会通过请求ID获取整个配置。解码后的JWT载荷、用于调试的API请求/响应日志、包含主题设置和通知标志的用户偏好对象。这些都是不透明数据块。你不需要规范化。不需要外键。直接塞进JSONB字段继续你的生活。

为了查清订购了什么颜色的T恤而进行六表连接？这不是正确的规范化。这是混淆了你存储的是引用还是值。

（不过要小心：如果后续需要查询这些数据，这种方法可能引发灾难性后果。参见[The JSONB Seduction](../the-jsonb-seduction)了解这种方案何时会带来噩梦）

## 规模是上下文

你会听到有人说"外键无法扩展"。但规模完全取决于你的硬件和架构。

一台树莓派每分钟记录10,000条传感器读数到microSD卡？对于该硬件来说这确实是高规模。AWS Aurora处理十亿级行数且预配置IOPS？你完全可以不费吹灰之力地用外键处理这些数据。

真正的硬性限制不是行数或写入量。而是分片。

当你的`Users`表在A服务器，`Orders`表在B服务器时，外键在物理上就无法工作。数据库没有机制可以在网络边界外执行约束。这时你已经在运行后台作业查找孤儿记录，并实现最终一致性模式。

这在多租户SaaS中很常见，每个租户因合规性需要独立数据库；或在物联网部署中，50,000个边缘设备各自运行本地SQLite。一旦到达这个阶段，外键就彻底出局（字面意义上的"表"出局）了，无论性能如何。

但在触及这个架构边界前，如果你正在开发一个10用户的内部工具，也许不必提前为Netflix的问题优化。

## 实际应用中的样子

与其问"我应该使用外键吗"，不如尝试问这三个问题：

如果这个引用错误会怎样？是引发诉讼、账单损坏还是监管违规？还是只是分析仪表板中缺少连接返回null？

如果验证变慢会怎样？你会丢失不可替代的实时数据吗？还是查询只是多花50毫秒？

这些数据是快照还是引用？你是记录某个时刻的状态，还是指向权威的当前值？

从这里开始，模式自然浮现：

金融交易、认证会话、任何数据损坏会导致法律责任的场景，都需要外键，无论性能开销。

高吞吐日志、仅追加的时间序列数据、每分钟写入百万事件的场景，可能不需要每次写入都进行验证开销。

订单和审计日志等历史快照、始终以完整数据块获取的用户偏好、不受控制的外部API webhook载荷...这些通常反规范化处理更有效。

但请注意我说的是"可能"和"通常"。因为上下文很重要，而你的上下文与我不同。

## 最终思考

外键不是性能问题。它们是在写入速度与数据完整性之间的权衡，这种权衡是否合理完全取决于你的具体瓶颈和具体后果。

真正的问题在于，当人们因为读到过"网络规模"而没有实际测量写入性能问题或考虑自己放弃什么时，就移除了外键。最终你会把Netflix的架构盲目复制到一个每天只处理100笔交易的新项目上。

也许性能成本对你的用例来说是值得的。也许不值得。但至少要根据你实际优化的目标来做出这个决定，而不是你认为应该优化的目标。
