最贵的数据库优化案例，我亲眼见过的，是从有人把所有外键都删掉开始的。

并不是因为他们测到了瓶颈，也不是因为写入真的慢。只是因为他们在某处看到“外键不具备可扩展性”。六个月后，他们拥有了 20 亿条孤儿记录，一个对已删除用户仍然计费的计费系统，以及偏差达 40% 的分析结果。

当他们尝试把约束重新加回去时，数据库在尝试验证已经被破坏的现有数据时卡死了。

在 Web 开发中流行一种观念：外键本身就慢，是你毕业后才会去掉的“训练轮”。但这完全忽视了约束的根本目的。你并不是在快与慢之间做选择，而是在不同的失效模式之间权衡。

可以这么想：安全玻璃、安全带和安全气囊都会增加汽车的重量。它们确实会让你的车子变慢、油耗更高。但你不会为了优化 0‑60 加速时间而把它们全部拆掉，因为你在优化的是完全不同的目标。

问题不在于外键是否会拖慢你的系统——当然会。关键在于你能得到什么回报，以及你是否真的需要这些回报。

## 你真正在交易的是什么

让我给你一个具体的例子。你正在构建一个天气监测系统，包含天气站、传感器设备、传感器读数以及美国州份的表。

你会把所有表都用外键关联起来吗？先思考一下实际会改变的东西以及其后果：

美国州份基本上是固定不变的。怀俄明州近期不会改名。当你知道引用数据是静态的时，就没有必要在每次插入时都用外键去校验州代码。这只会带来毫无意义的开销。

天气站会被新增、迁移和退役。但这里有个问题：如果有人意外删除了站点记录，你希望历史读数“失去”它们的站点吗？也许你实际上希望这些数据即使站点不存在也保持完整。这意味着你把读数视为历史快照而不是实时引用，这会改变外键是否有意义。

传感器读数每分钟会被插入数千次。每一次外键检查都要进行一次查找。每一次查找都会在表上产生争用。如果缓慢的校验导致你的插入队列堵塞，进而丢失实时数据，这是一种不同于产生孤儿记录的数据丢失。

你可以预见到接下来会怎样。选择并不是抽象的性能 vs 正确性，而是基于你的实际约束和实际后果，决定你更能容忍哪种具体的失效。

如果错误的引用会导致账单数据损坏或违反监管规定，你可能会在乎性能代价，仍然让外键来保护你。如果缓慢的校验会让实时传感器数据永远丢失，因为队列溢出，那么也许校验本身就是错误的权衡。

## 当写入速度真的重要时

所以你已经决定需要最大写入速度。队列堆积，事务超时，外键检查确实在造成你已经测量到的问题（而不仅仅是理论上的）。

你有几种选择。可以把事务隔离级别从 `SERIALIZABLE` 改为 `READ COMMITTED`，速度更快，但会牺牲一些一致性保证。也可以批量提交，每次事务插入 1000 行，而不是一次插入一行，以摊销外键的开销。或者可以去规范化，采用追加式日志结构，根本不去校验引用。

顺便说一下，第三种方案并不是作弊。它只是另一种设计：

```sql
CREATE TABLE sensor_log (
  id BIGSERIAL PRIMARY KEY,
  recorded_at TIMESTAMPTZ NOT NULL,
  data JSONB NOT NULL  -- { station_id, sensor_id, temp, humidity, ... }
);

CREATE INDEX ON sensor_log USING GIN (data);
CREATE INDEX ON sensor_log (recorded_at);
```

没有关联查询。没有外键检查。只是在时间范围或 JSONB blob 的 GIN 索引上追加数据并查询。这是“最佳实践”吗？可能并不是教材中教的那种意义上的最佳实践。它在 Raspberry Pi 上每分钟插入 50,000 行时还能工作吗？答案是肯定的。

当人们把“最佳实践”当作道德命令而不是一种在常见场景下有效、但可能不适合你特定需求的模式时，问题就出现了。

## 规范化陷阱

数据库课程喜欢教授规范化。必须不惜一切代价避免重复。第三范式或崩溃。

于是你会得到类似的结构：`Orders` → `OrderItems` → `Products` → `Variants` → `Colors` → `Sizes`

要回答“我去年圣诞节买的是红色衬衫还是蓝色衬衫？”需要六表联接。更别提如果还要包含产品名称，那就在目录层级上再多三次联接。

但等等。常见的理由是“如果品牌改变了蓝色的标记怎么办？”如果真的发生了，你真的想让历史订单的颜色自动 retroactively 改变吗？当然不想。当时下单的人购买的是当时存在的“蓝色 T 恤，尺码 M”，而不是以后可能被更新的某个抽象目录条目。

这点值得细细思考，因为它很微妙。有些数据本质上是快照，而不是引用。当你把快照数据当作实时引用来处理时，就会出现大量荒唐的联接，只是为了重建本该在写入时就去规范化的内容。

直接在订单上存储 `{"color": "blue", "size": "M"}`。完成。

### 识别快照数据

如何判断某些数据应该是快照？问自己它是否是某个时间点的记录：

- 订单在购买时捕获的产品细节。  
- 审计日志记录用户在执行操作时的状态。  
- 历史表在更新前保存记录的状态。  
- 事件流捕获了“发生了什么、何时、使用了什么数据”。

如果答案是“是的，这在记录某个时刻”，就不要再对其进行规范化。直接把它快照化。

### 不透明 Blob

还有一种类别超出快照：你从不查询其内部的数据。你只把它整体存储，然后整体取回。

LLM 模型配置如 `{"model": "gpt-4", "temperature": 0.7, "max_tokens": 2000}` 并不是要按 temperature 查询的东西。需要时，你会通过请求 ID 把整个配置取出来。解码后的 JWT 负载、用于调试的 API 请求/响应日志、包含主题设置和通知标志的用户偏好对象，这些都是不透明的 Blob。无需规范化。无需外键。直接塞进 JSONB，继续你的生活。

要通过 6 表联接才能找出订购的衬衫颜色？这并不是正确的规范化。这是对“你是在存引用还是值”这一点的混淆思考。

（不过要小心：如果以后需要查询这些数据，这种做法可能会导致灾难性后果。参见 [JSONB 诱惑](../the-jsonb-seduction) 了解这种方法何时会自掘坟墓。）

## 规模是相对的

你会听到有人说“外键不具备可扩展性”。但规模完全取决于你的硬件和架构。

一块 Raspberry Pi 每分钟记录 10,000 条传感器读数并写入 microSD 卡？对那块硬件来说，这已经是相当高的规模。AWS Aurora 配置了 IOPS，处理数十亿行数据？你完全可以使用外键而不出汗。

真正的硬性上限并不是行数或写入量，而是分片。

当你的 `Users` 表位于服务器 A 上，而 `Orders` 表位于服务器 B 时，外键在物理上根本无法工作。数据库没有跨网络边界强制约束的机制。此时，你已经在运行后台作业来查找孤儿记录，并实现最终一致性模式。

这种情况常见于多租户 SaaS——每个租户拥有自己的隔离数据库以满足合规要求，或在物联网部署中，拥有 50,000 台边缘设备，每台设备本地运行 SQLite。一旦达到这种架构层面，外键就已经“下桌”（字面意义上）了，性能考量已不再 relevant。

但在你触及该架构边界之前，或许不必在构建一个 10 人内部工具时就提前为 Netflix 那样的高并发问题做优化。

## 实际场景中的表现

与其问“我应该使用外键吗”，不如先问下面三个问题：

- 如果这个引用错误，会导致什么后果？是诉讼、账单损坏、监管违规，还是仅仅在分析仪表盘中出现空结果？
- 如果验证变慢，会有什么影响？是失去不可替代的实时数据，还是查询仅多了 50 毫秒？
- 这段数据是快照还是引用？你是在记录某个时刻的状态，还是指向权威的当前值？

基于这些回答，模式会自然浮现：

- 金融交易、认证会话等数据腐败会产生法律责任的场景，通常仍然需要外键，即使会带来性能开销。

高吞吐量日志、追加式时间序列数据，或者每分钟写入数百万事件的场景，通常不需要在每一次写入时进行验证开销。

历史快照（如订单和审计日志）、始终以完整块方式获取的数据（例如用户偏好）、以及你无法控制的模式（比如来自外部 API 的 webhook 负载）……这些往往在去规范化后表现更好。

但请注意，我用了“可能”和“通常”。因为上下文很重要，而你的上下文与我的不同。

## 最后思考

外键本身不是性能问题。它们是写入速度与数据完整性之间的权衡，而这种权衡是否合理完全取决于你的具体瓶颈和具体后果。

真正的问题在于，人们因为看到“Web 规模”之类的说法而移除外键，却没有实际测量写入性能是否真的存在问题，也没有考虑放弃了什么。于是你把 Netflix 的架构盲目搬到一个每天只处理 100 笔交易的绿地项目上。

也许在你的使用场景下，性能代价是值得的。也许并非如此。但至少要基于你真正想优化的目标来做决定，而不是基于你认为应该优化的目标。
