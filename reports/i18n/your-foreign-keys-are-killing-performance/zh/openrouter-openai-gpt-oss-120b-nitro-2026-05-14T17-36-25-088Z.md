最贵的数据库优化案例，我见过的，就是有人把所有外键都删掉。

并不是因为他们测到了瓶颈，也不是因为写入真的慢，而是因为他们在某处看到“外键不具备可扩展性”。六个月后，系统出现了 20 亿条孤儿记录，计费系统向已删除的用户收费，分析数据偏差达 40%。

当他们想把约束重新加回来时，数据库在尝试验证已经损坏的现有数据时直接卡死。

在 Web 开发圈里有一种普遍的观念：外键本身就慢，是你毕业后才能去掉的训练轮。但这完全忽视了约束的根本目的。你并不是在快与慢之间做选择，而是在不同的失效模式之间权衡。

可以这么想：安全玻璃、安全带和安全气囊都会增加汽车的重量，确实会让车子变慢、油耗上升。但你不会为了提升 0‑60 加速时间而把它们全部拆掉，因为你的优化目标根本不是速度。

问题不在于外键是否让你变慢——它们当然会。但关键是你得到的是什么，以及你是否真的需要这些保障。

## 你真正的权衡是什么

举个具体例子。你在构建一个天气监测系统，涉及天气站、传感器设备、传感器读数以及美国州表。

要把所有表都用外键关联吗？先看看实际会变动的东西以及其后果：

美国州基本是静态的。怀俄明州近期不会改名。既然参考数据是固定的，就没有必要在每次插入时都检查州代码，这只会增加无意义的开销。

天气站会新增、迁移、退役。但有个问题：如果有人误删了某个站点记录，你希望历史读数“失去”对应的站点吗？也许你更希望这些数据在站点被删除后仍然保持完整。这就意味着读数是历史快照而非实时引用，从而决定外键是否有意义。

传感器读数每分钟会被插入成千上万次。每一次外键检查都要一次查找，而每一次查找都会在表上产生争用。如果慢速校验导致插入队列堆积，实时数据丢失，那就是一种不同于孤儿记录的数据丢失。

你可以预见接下来的讨论：选择并不是抽象的性能 vs 正确性，而是基于实际约束和后果，决定你更能容忍哪种具体的失效。

如果错误的引用会导致计费数据损坏或触犯监管，那么不论性能成本如何，你都应该让外键来保护。如果慢速校验会让实时传感器数据因队列溢出而永久丢失，那么也许外键不是正确的权衡。

## 当写入速度真的重要时

假设你已经决定必须追求最高写入速度。队列堆积，事务超时，外键检查已经被实际测量出导致问题（而不是空想）。

此时有几种可选方案。可以把事务隔离级别从 `SERIALIZABLE` 降到 `READ COMMITTED`，这样更快，但会放弃部分一致性保证。也可以批量提交，每次插入 1000 行而不是单行，以摊薄外键开销。或者可以把数据去规范化，采用追加式日志结构，根本不去校验引用。

顺便说一下，第三种方案并不是作弊，而是另一种设计思路：

```sql
CREATE TABLE sensor_log (
  id BIGSERIAL PRIMARY KEY,
  recorded_at TIMESTAMPTZ NOT NULL,
  data JSONB NOT NULL  -- { station_id, sensor_id, temp, humidity, ... }
);

CREATE INDEX ON sensor_log USING GIN (data);
CREATE INDEX ON sensor_log (recorded_at);
```

没有 JOIN。没有外键检查。只是在时间范围或 JSONB blob 的 GIN 索引上追加数据并查询。这算是“最佳实践”吗？可能并不符合教科书里所说的最佳实践。但在树莓派上每分钟插入 50,000 行时，它能跑通吗？答案是肯定的。

当人们把“最佳实践”当作道德命令，而不是在常见场景下有效、但未必适合你特定需求的模式时，问题就出现了。

## 规范化陷阱

数据库课程热衷于教授规范化。无论如何都要避免重复。第三范式或毁灭。

于是你得到类似这样的结构：`Orders` → `OrderItems` → `Products` → `Variants` → `Colors` → `Sizes`

仅仅为了回答“我去年圣诞节到底是买了红色衬衫还是蓝色衬衫？”就要进行六表连接。更别提如果还要把产品名称一起返回，那就再多三层目录层级的 JOIN。

但等等。背后的理由通常是“如果品牌改了蓝色的标签怎么办？”如果真的改了，你真的想让历史订单的颜色自动回溯修改吗？显然不想。当订单被下时，买的是当时的“蓝色 T‑恤，尺码 M”，而不是某个以后可能被更新的目录条目。

这点值得细细思考，因为它很微妙。某些数据本质上是快照，而不是引用。当你把快照数据当作活引用来处理时，就会产生大量荒唐的 JOIN，只是为了重建本该在写入时就去规范化的内容。

直接在订单上存 `{"color": "blue", "size": "M"}`。完事。

### 识别快照数据

如何判断某条记录应该是快照？问自己它是否是“某一时刻的记录”：

- 订单在购买时捕获的商品细节。  
- 审计日志记录用户执行操作时的状态。  
- 历史表保存更新前的记录状态。  
- 事件流捕获了何时、发生了什么、携带了哪些数据。

如果答案是“是的，这在记录一个时间点”，就停止对它进行规范化，改为快照存储。

### 不透明的 Blob

还有一种类别超出快照：你从不对其内部进行查询，只是整块存取。

例如 LLM 模型配置 `{"model": "gpt-4", "temperature": 0.7, "max_tokens": 2000}`，你不会按 temperature 查询，只会在需要时通过请求 ID 把整段配置取回。JWT 负载解码后、用于调试的 API 请求/响应日志、包含主题设置和通知标志的用户偏好对象，这些都是不透明的 Blob。它们不需要规范化，也不需要外键。直接塞进 JSONB，继续你的生活。

那段用于找出订购颜色的 6 表 JOIN？这并不是正确的规范化，而是对“我是在存引用还是值”这一点的混淆思考。

（注意：如果以后真的需要查询这些数据，这种做法可能会导致灾难。参见 [The JSONB Seduction](/the-jsonb-seduction) 中对该方法产生的噩梦的讨论。）

## 规模是相对的

你会听到有人说“外键不具备可扩展性”。但可扩展性完全取决于你的硬件和架构。

一块 Raspberry Pi 每分钟向 microSD 卡记录 10,000 条传感器读数？对那套硬件来说已经是相当高的规模。AWS Aurora 配置了预置 IOPS，处理数十亿行数据？你完全可以在不出汗的情况下使用外键。

真正的硬上限并不是行数或写入量，而是分片。

当你的 `Users` 表位于服务器 A，`Orders` 表位于服务器 B 时，外键在物理上根本无法工作。数据库没有跨网络边界强制约束的机制。此时，你已经在跑后台任务去查找孤儿记录，并实现最终一致性模式。

这种情况出现在多租户 SaaS 中，每个租户为了合规性拥有独立的数据库，或在物联网部署中，拥有 50,000 台边缘设备各自本地运行 SQLite。到那一步，外键已经不在讨论范围（字面意义上）——性能已经不是决定因素。

但在触及到这种架构边界之前，别急着为 Netflix 那种规模的问题做提前优化，尤其是当你只在构建一个 10 人内部工具时。

## 实际操作中这到底是什么样子

与其问“我该不该使用外键”，不如先问自己下面三个问题：

- 如果这个引用错误会导致什么后果？是诉讼、账单损坏、合规违规，还是仅仅在分析仪表盘上出现空值？
- 如果验证变慢会有什么影响？会丢失不可替代的实时数据，还是查询只慢了 50 毫秒？
- 这段数据是快照还是引用？是记录某一时刻的状态，还是指向权威的当前值？

基于这些回答，常见模式自然浮现：

- 金融交易、认证会话等数据腐败会导致法律责任的场景，几乎都应当使用外键，即使有性能开销。
- 高吞吐日志、追加式时间序列数据、每分钟写入数百万事件的场景，通常不需要在每次写入时进行验证。
- 订单、审计日志等历史快照，或者像用户偏好这种总是以完整块读取的数据，亦或是你无法控制的外部 API webhook 负载……这些往往在去规范化后表现更好。

但请注意，我用了“几乎”和“往往”。因为上下文决定选择，而你的上下文与我的不同。

## 结语

外键本身不是性能瓶颈。它们是写入速度与数据完整性之间的权衡，是否值得取决于你的具体瓶颈和可能产生的后果。

真正的问题在于，有人因为看到“Web 规模”之类的说法，就在没有测量写入性能是否真的有问题、也没有考虑放弃了什么的情况下，直接去掉外键。结果就是把 Netflix 的架构盲目搬到一个每天只处理 100 笔交易的全新项目上。

也许在你的使用场景下，性能开销是值得的；也许并不值得。无论如何，至少要基于你真正想优化的目标来做决定，而不是基于你认为应该优化的目标。
