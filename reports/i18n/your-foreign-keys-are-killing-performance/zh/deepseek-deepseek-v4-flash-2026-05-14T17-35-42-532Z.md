我见过的最昂贵的数据库优化，始于有人删除了所有外键。

不是因为他们测量到了瓶颈。也不是因为写入真的慢。而是因为他们读到过“外键无法扩展”。六个月后，他们有了20亿条孤立记录，一个向已删除用户收费的计费系统，以及偏差高达40%的分析数据。

当他们试图重新添加约束时？数据库在尝试验证已经损坏的现有数据时直接卡死。

Web开发中普遍存在一种观念：外键天生就慢，它们是你毕业进入“真正”系统后就要拆掉的辅助轮。但这完全误解了约束存在的意义。你选择的不是快与慢。你选择的是不同的故障模式。

这样想吧：安全玻璃、安全带和安全气囊都会给你的车增加重量。它们绝对会让你的车更慢、更耗油。但你不会为了优化百公里加速就把它们拆掉，因为你优化的是完全不同的东西。

问题不在于外键是否拖慢你。它们当然会。问题在于你换来了什么，以及你是否真的需要它。

## 你真正在权衡什么

我给你一个具体的例子。你正在构建一个天气监测系统，包含气象站、传感器设备、传感器读数和美国各州这几张表。

你要把所有东西都用外键关联起来吗？我们来想想实际会变化什么，以及后果是什么：

美国各州大概不会变。怀俄明州短期内不会改名。当你确定引用数据是静态的时，你不需要在每次插入时都用外键来验证州代码。那是毫无意义的开销。

气象站会被添加、移动和退役。但这里有个问题：如果有人不小心删除了一个气象站记录，你希望历史读数“丢失”该气象站吗？也许你实际上希望即使气象站被删除，数据仍然保持完整。这意味着你把读数视为历史快照而非实时引用，这改变了外键是否有意义。

传感器读数每分钟被插入数千次。每次外键检查都意味着一次查找。每次查找都会在你的表上产生争用。如果缓慢的验证导致你的插入队列积压，从而丢失实时数据，那是一种与孤立记录不同的数据丢失。

你可以看到问题所在。选择不是关于性能与正确性这些抽象概念。而是关于根据你的实际约束和实际后果，你更愿意容忍哪种具体的故障。

如果错误的引用意味着损坏的计费数据或违反法规，那么无论性能成本如何，你可能都需要外键来保护你。如果缓慢的验证意味着你的队列溢出，从而永远丢失实时传感器数据，那么验证可能就是错误的权衡。

## 当快速写入真正重要时

所以你已经决定需要最大写入速度。你的队列在堆积，事务在超时，外键检查确实在造成你实际测量到（而不仅仅是理论上的）的问题。

你有几个选择。你可以将事务隔离级别从 `SERIALIZABLE` 改为 `READ COMMITTED`，这样更快，但会牺牲一些一致性保证。你可以批量提交，每次事务插入1000行而不是一行一行插，以摊销外键开销。或者你可以反范式化为仅追加的日志结构，甚至不尝试验证引用。

顺便说一句，第三种选择不是作弊。它只是一种不同的设计：

```sql
CREATE TABLE sensor_log (
  id BIGSERIAL PRIMARY KEY,
  recorded_at TIMESTAMPTZ NOT NULL,
  data JSONB NOT NULL  -- { station_id, sensor_id, temp, humidity, ... }
);

CREATE INDEX ON sensor_log USING GIN (data);
CREATE INDEX ON sensor_log (recorded_at);
```

没有连接。没有外键检查。只是追加数据，然后按时间范围或 JSONB 数据块上的 GIN 索引查询。这是“最佳实践”吗？从数据库教科书的角度看，可能不是。但当你每分钟在树莓派上插入 5 万行时，它绝对管用。

问题在于，人们把“最佳实践”当作道德准则，而不是一种在常见场景下表现良好、但未必适合你的模式。

## 范式化陷阱

数据库课程喜欢教范式化。不惜一切代价避免重复。要么第三范式，要么拉倒。

于是你得到类似这样的结构：`Orders` → `OrderItems` → `Products` → `Variants` → `Colors` → `Sizes`

六个表连接，只是为了回答“我去年圣诞节订的是红衬衫还是蓝衬衫？”更别提你还需要包含产品名称，那又得在目录层级里多连三个表。

但等等，理由通常是“如果品牌改变了‘蓝色’的标签怎么办？”如果真发生了，你真的希望历史订单自动改变颜色吗？当然不。当某人下订单时，他们买的是当时存在的“蓝色 T 恤，M 码”，而不是某个可能后来被更新的目录条目的抽象引用。

这一点值得细想，因为它很微妙。有些数据本质上是快照，而不是引用。当你把快照数据当作实时引用处理时，就会导致这种荒谬的连表泛滥，只是为了重构那些本应在写入时反范式化的东西。

直接把 `{"color": "blue", "size": "M"}` 存储在订单上。完事。

### 识别快照数据

你怎么知道什么时候该用快照？问问自己：这是不是一条时间点记录？

订单记录购买时的产品详情。审计日志记录用户执行操作时的状态。历史表保存更新前的记录状态。事件流捕获发生了什么、何时发生、附带什么数据。

如果答案是“是的，这是在记录一个时间点”，那就别再范式化了。开始快照化。

### 不透明数据块

除了快照，还有另一类数据：你从不查询其内部结构。你只是存储它，然后整体取出。

LLM 模型配置如 `{"model": "gpt-4", "temperature": 0.7, "max_tokens": 2000}`，你不会按 temperature 去查询。你需要时，按请求 ID 取出整个配置。解码后的 JWT 负载、用于调试的 API 请求/响应日志、包含主题设置和通知标志的用户偏好对象。这些都是不透明数据块。你不需要范式化。你不需要外键。把它们塞进 JSONB，然后继续生活。

那个为了找出订了什么颜色衬衫的六表连接？那不是正确的范式化。那是混淆了你在存储引用还是值。

（不过要小心：如果后来你需要查询那些数据，这可能会适得其反。参见 [JSONB 的诱惑](../the-jsonb-seduction) 了解这种方法何时会制造自己的噩梦。）

## 规模取决于上下文

你常会听到有人说“外键无法扩展”。但扩展性完全取决于你的硬件和架构。

一台树莓派每分钟向 microSD 卡写入 10,000 条传感器读数？对那块硬件来说，这确实是高负载。而配备预置 IOPS 的 AWS Aurora 处理数十亿行数据？你可以轻松地用外键搞定，毫不费力。

真正的硬性限制不在于行数或写入量，而在于分片。

当你的 `Users` 表在服务器 A 上，而 `Orders` 表在服务器 B 上时，外键物理上就无法工作。数据库没有机制跨网络边界强制执行约束。到那时，你已经在运行后台任务来查找孤儿记录，并实现最终一致性模式。

这在多租户 SaaS 中会发生——每个租户因合规要求拥有独立的数据库；或者在物联网部署中，5 万个边缘设备各自本地运行 SQLite。一旦到了那个地步，外键就彻底出局了，与性能考量无关。

但在你遇到那个架构边界之前，也许不要为了 Netflix 的问题而提前优化，当你只是在构建一个 10 用户的内部工具时。

## 实践中实际的样子

与其问“我该不该用外键”，不如试着问这三件事：

如果这个引用错了，什么会坏？是诉讼、账单损坏、监管违规？还是仅仅一个缺失的连接，在你的分析仪表盘里返回 null？

如果验证很慢，什么会坏？你会丢失不可替代的实时数据吗？还是你的查询仅仅多花了 50 毫秒？

这些数据是快照还是引用？你是在记录某个特定时刻的样子，还是指向权威的当前值？

由此，模式就自然浮现了：

金融交易、认证会话，任何数据损坏意味着法律责任的场景，可能都需要外键，无论性能开销如何。

高吞吐量日志、仅追加的时间序列数据，任何每分钟写入百万事件的场景，可能不需要每次写入都做验证开销。

历史快照（如订单和审计日志）、你总是作为完整数据块获取的数据（如用户偏好）、你不控制的模式（如外部 API 的 webhook 负载）……这些通常非范式化效果更好。

但注意我说的是“可能”和“通常”。因为上下文很重要，而你的上下文和我的不同。

## 最后思考

外键不是性能问题。它们是写入速度与数据完整性之间的权衡，而这种权衡是否合理，完全取决于你具体的瓶颈和具体的后果。

真正的问题在于，有些人因为读到关于“Web 规模”的内容就移除外键，却没有实际测量自己是否存在写入性能问题，也没有考虑放弃的是什么。结果就是把 Netflix 的架构生搬硬套到一个每天只处理 100 笔交易的绿地项目上。

也许性能代价对你的用例来说是值得的。也许不是。但至少，你应该根据自己实际在优化的目标来做决定，而不是根据你认为自己应该优化的目标。
