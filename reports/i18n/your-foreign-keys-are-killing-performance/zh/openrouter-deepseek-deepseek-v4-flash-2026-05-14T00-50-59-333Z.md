我所见过最昂贵的数据库优化，始于有人删除了所有外键。

不是因为测量到了瓶颈。不是因为写入真的慢。而是因为他们在某处读到“外键无法扩展”。六个月后，他们有了20亿条孤儿记录，一个向已删除用户收费的计费系统，以及偏差达40%的分析数据。

当他们试图重新添加约束时？数据库在尝试验证已经损坏的现有数据时直接卡死。

Web开发中普遍存在一种观念：外键天生缓慢，是“真正”系统毕业前该卸掉的辅助轮。但这完全误解了约束存在的意义。你选择的不是快与慢，而是不同的故障模式。

这样想吧：安全玻璃、安全带和安全气囊都会增加汽车重量。它们绝对会让你的车变慢、油耗更高。但你不会为了优化百公里加速时间而把它们拆掉，因为你优化的是完全不同的东西。

问题不在于外键是否拖慢你。当然会。问题在于你换回了什么，以及你是否真的需要它。

## 你真正在交易什么

举个具体例子。你在构建一个天气监测系统，包含气象站、传感器设备、传感器读数以及美国各州这些表。

要不要把所有东西都用外键关联起来？我们来想想实际会变化什么，以及后果是什么：

美国各州大概率不会变。怀俄明州短期内不会改名。当你知道参考数据是静态的时，你不需要在每次插入时用外键验证州代码。那是毫无意义的开销。

气象站会被添加、迁移和退役。但问题来了：如果有人误删了一条气象站记录，你希望历史读数也跟着“丢失”吗？也许你更希望即使气象站被删，数据依然完好无损。这意味着你把读数当作历史快照而非实时引用，这直接决定了外键是否还有意义。

传感器读数每分钟插入数千次。每次外键检查都意味着一次查找，每次查找都会在你的表上产生争用。如果缓慢的校验导致插入队列积压，进而丢失实时数据，那这种数据丢失和产生孤立记录是两种不同的损失。

你应该能看出走向了。选择并非抽象概念上的“性能 vs 正确性”，而是根据你的实际约束和实际后果，你更愿意容忍哪一种具体的故障。

如果错误的引用意味着账单数据损坏或违反监管规定，那么无论性能代价多大，你都应该用外键来保护自己。如果缓慢的校验导致队列溢出，永久丢失实时传感器数据，那么校验可能就不是正确的权衡。

## 快速写入真正重要的时候

假设你已确定需要最大写入速度。你的队列在堆积，事务在超时，外键检查确实造成了你实际测量过（而非仅理论推测）的问题。

你有几个选择。你可以将事务隔离级别从 `SERIALIZABLE` 改为 `READ COMMITTED`，这样更快，但会牺牲一些一致性保证。你可以批量提交，每次事务插入1000行而不是一行一行插，从而分摊外键开销。或者你可以反范式化为仅追加的日志结构，甚至不去尝试验证引用。

顺便说一句，第三种选择并不是作弊。它只是另一种设计：

```sql
CREATE TABLE sensor_log (
  id BIGSERIAL PRIMARY KEY,
  recorded_at TIMESTAMPTZ NOT NULL,
  data JSONB NOT NULL  -- { station_id, sensor_id, temp, humidity, ... }
);

CREATE INDEX ON sensor_log USING GIN (data);
CREATE INDEX ON sensor_log (recorded_at);
```

没有连接，没有外键检查。只管追加数据，按时间范围或 JSONB 上的 GIN 索引查询。这是“最佳实践”吗？从数据库教科书的角度看，可能不是。但在树莓派上每分钟插入 5 万行时，它绝对管用。

问题在于，人们把“最佳实践”当作道德准则，而不是一种在常见场景下表现良好、但未必适合你的模式。

## 规范化陷阱

数据库课程热衷于教规范化。不惜一切代价避免重复。要么第三范式，要么拉倒。

于是你搞出了类似这样的结构：`订单` → `订单项` → `产品` → `变体` → `颜色` → `尺码`

六个表连接，就为了回答“去年圣诞节我买的是红衬衫还是蓝衬衫？”更别提你还需要包含产品名称，那又得在目录层级里多连三个表。

但等等。通常的辩解是：“如果品牌改了‘蓝色’的标签怎么办？”如果真改了，你希望历史订单也自动变色吗？当然不。当用户下单时，他们买的是当时存在的“蓝色T恤，M码”，而不是某个将来可能更新的目录条目的抽象引用。

这一点值得细想，因为它很微妙。有些数据本质上是快照，而不是引用。当你把快照数据当作实时引用处理时，就会陷入这种荒谬的连表泛滥——为了重建本应在写入时直接反范式化的东西。

直接把 `{"color": "blue", "size": "M"}` 存到订单上。完事。

### 识别快照数据

怎么判断某样东西该不该是快照？问问自己它是不是一个时间点的记录：

- 订单记录的是购买时刻的产品详情。
- 审计日志记录用户执行操作时的状态。
- 历史表保存更新前的记录状态。
- 事件流捕获了什么时间发生了什么、带着什么数据。

如果答案是“是的，这是记录某个时间点”，那就别再规范化了。直接快照化。

### 不透明数据块

除了快照之外，还有另一类数据：你永远不会去查询其内部结构的数据。你只是完整地存储它，再完整地取回。

LLM 模型配置如 `{"model": "gpt-4", "temperature": 0.7, "max_tokens": 2000}` 并不是那种你会按温度字段去查询的东西。你需要时，直接按请求 ID 取出整个配置。解码后的 JWT 载荷、用于调试的 API 请求/响应日志、包含主题设置和通知标志的用户偏好对象——这些都是不透明数据块。你不需要规范化，也不需要外键。把它们塞进 JSONB，然后继续干正事。

为了查出订购的衬衫是什么颜色而连 6 个表？那不是正确的规范化。那是搞不清你在存引用还是存值。

（不过要小心：如果后来你需要查询那些数据，这种做法可能会反噬。参见 [JSONB 的诱惑](../the-jsonb-seduction) 了解这种方案何时会制造自己的噩梦。）

## 规模取决于上下文

你会听到有人说“外键无法扩展”。但规模完全取决于你的硬件和架构。

一台树莓派每分钟向 microSD 卡记录 10,000 条传感器读数？对那块硬件来说，这确实是高负载。配备预置 IOPS 的 AWS Aurora 处理数十亿行数据？你可以毫无压力地使用外键。

真正的硬性限制不在于行数或写入量。而在于分片。

当你的 `Users` 表在服务器 A 上，而 `Orders` 表在服务器 B 上时，外键物理上就无法工作。数据库没有机制跨网络边界强制执行约束。到了这一步，你已经在跑后台任务来查找孤儿记录，并实现最终一致性模式。

这种情况出现在多租户 SaaS 中——每个租户因合规要求拥有独立的隔离数据库，或者出现在 IoT 部署中——5 万个边缘设备各自本地运行 SQLite。一旦到了这种架构，外键就（字面意义上）被排除在外，与性能考量无关。

但在你触及那个架构边界之前，也许不要在你构建一个只有 10 个用户的内部工具时，就过早地优化 Netflix 级别的问题。

## 实际中这是什么样子的

与其问“我该不该用外键”，不如试着问这三个问题：

如果这个引用错了，会出什么问题？是诉讼、账单错误、监管违规？还是仅仅在分析仪表盘上返回 null 的缺失连接？

如果验证慢了，会出什么问题？你会丢失不可替代的实时数据吗？还是只是查询多花了 50 毫秒？

这些数据是快照还是引用？你是在记录某个时刻的样子，还是在指向权威的当前值？

从这些出发，模式就自然浮现了：

金融交易、认证会话——任何数据损坏意味着法律责任的东西——很可能需要外键，无论性能开销如何。

高吞吐量的日志、仅追加的时间序列数据——任何每分钟写入百万级事件的数据，可能不需要在每次写入时都承担验证开销。

历史快照（如订单和审计日志）、你总是作为完整数据块获取的数据（如用户偏好）、你无法控制的模式（如外部 API 的 webhook 负载）……这些通常更适合反范式化。

但注意我说的是“可能”和“通常”。因为上下文很重要，而你的上下文和我的不同。

## 最后思考

外键不是性能问题。它们是写入速度与数据完整性之间的权衡，而这种权衡是否合理，完全取决于你的具体瓶颈和具体后果。

真正的问题在于，有人因为读到“Web 规模”相关的内容就移除外键，却没有实际测量自己是否存在写入性能问题，也没有考虑自己放弃了什么。结果就是盲目照搬 Netflix 的架构，用在一个每天处理 100 笔事务的绿地项目上。

也许性能开销对你的用例来说是值得的。也许不是。但至少，你应该基于自己实际在优化什么来做决定，而不是基于你认为自己应该优化什么。
