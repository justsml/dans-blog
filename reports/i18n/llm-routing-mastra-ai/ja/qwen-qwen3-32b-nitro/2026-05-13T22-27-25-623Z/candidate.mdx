ほとんどのエンジニアリングチームは、言語モデルを1つ選び、すべてのタスクに使い続けます。1つのプロバイダー、1つのモデル、すべてのタスク。これはまるで、最初の面接でうまくいったからといって、コーディング、ライティング、税務のすべてを1人にやらせるようなものです。

ある瞬間、あるモデルはコードに最適で、別のモデルは長くて複雑なコンテキストに強く、また別のモデルは分類作業に最適で安価です。モデル名は変化しますが、問題の構造は変わりません。1つのモデルをすべてに優れていると扱うと、単純なタスクで過剰に支払いをするか、専門的なタスクで不十分な結果を得ることになります。

私はあるチームが、感情分析に100万トークンあたり30ドルのモデルを使って数千ドルを費やすのを見てきました。0.5ドルのモデルで十分な作業を、プレミアムプロバイダーに任せていたのです。単純なJSONフォーマットや基本的な分類タスクもすべてプレミアムプロバイダーを通じて処理されていました。加熱していたのはAWSの請求額だけでした。

より良い方法があり、それは特に複雑ではありません。

## 委任 versus 感情的忠誠

特定のタスクに最も適したモデルにリクエストをルーティングできないでしょうか？難しいタスクには高価なパワフルなモデルを使い、単純なパースやフォーマットには安価なモデルに任せます。複数のプロバイダーの利点を得る必要はありません。コードベースで手動で切り替える必要がないからです。

Mastraは、まさにこのシステムを構築できるようにします。異なるタイプの作業に特化したエージェントを設定し、どの専門家が各リクエストを処理すべきかを判断するルーター・エージェントを作成します。以下に示すモデルIDは例であり、リーダーボードではありません。あなたの評価で勝ち抜いた現在のモデルに置き換えてください。

こう考えてみましょう。チームには3人の専門家がいます。

```typescript
// ./src/mastra/index.ts
import { Mastra } from '@mastra/core';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { anthropic } from '@ai-sdk/anthropic';
import { google } from '@ai-sdk/google';

export const claudeAgent = new Agent({
  id: 'claude-agent',
  instructions: 'あなたはエキスパートエンジニアです。バグを書いたら解雇されます。',
  model: anthropic(process.env.CODE_MODEL ?? 'claude-sonnet-4-5'),
});

export const geminiAgent = new Agent({
  id: 'gemini-agent',
  instructions: 'あなたはクリエイティブライターです。奇抜でなければなりません。',
  model: google(process.env.LONG_CONTEXT_MODEL ?? 'gemini-3-pro-preview'),
});

export const gptAgent = new Agent({
  id: 'gpt-agent',
  instructions: 'あなたは親切なアシスタントです。地味でなければなりません。',
  model: openai(process.env.GENERAL_MODEL ?? 'gpt-5.2'),
});
```

それぞれに役割があります。コードエージェントはリポジトリ固有のコーディング評価をパスするモデルでなければなりません。長コンテキストエージェントは、実際のドキュメントを処理しても中身がスープ状にならないモデルでなければなりません。汎用エージェントは、安価で信頼性が高く、最も良い意味で地味でなければなりません。

ここで興味深い点があります。インテリジェントプロキシとして機能するルーターを追加します。

```typescript
export const routerAgent = new Agent({
  id: 'router-agent',
  name: 'The Boss',
  instructions: `あなたはインテリジェントなルーターです。
  - コーディング -> Claude
  - 詩 -> Gemini
  - 事実 -> GPT

  自分で作業をしないでください。委任してください。`,
  model: openai(process.env.ROUTER_MODEL ?? 'gpt-5-mini'), // ルーティングには安価なモデルを使用!
  agents: {
    claudeAgent,
    geminiAgent,
    gptAgent,
  },
});

export const mastra = new Mastra({
  agents: { routerAgent, claudeAgent, geminiAgent, gptAgent },
});
```

ルーター自体は軽量モデルで実行されます。これは単にトラフィックをどこに送るかを判断しているだけです。どの高価なモデルを使うかを決めるためにプレミアム料金を払う必要はありません。この点も測定してください。悪いルーターは静かに節約を誤ったルーティングに変えていきます。

誰かがバブルソートの実装を依頼した場合、ルーターはそれをコード作業として認識し、あなたのコード専門家に引き渡します。クリエイティブライティングのプロンプト？それは声と範囲を選ばせるモデルに送られます。歴史的イベントに関する事実質問？リトリーブが必要な場合は、一般エージェントにルーティングします。

## 実用的な利点

**コスト効率はあなたが思っているよりも重要です。** 小さなルーティングモデルが委任決定を行うコストは、すべてのリクエストを最も高価なプロバイダーに回すコストの一部に過ぎません。規模が大きくなると、これは実際に大きな金額になります。本当に必要になるまで高価な知性に支払う必要はありません。

**モデルとタスクのマッチングにより品質が向上します。** 勝者モデルは月ごと、タスクごと、プロンプトの形状によって異なります。そのためルーティングレイヤーはあなたの評価に依存すべきであり、インテグレーションを書いた週にツイッターで勝っていたモデルに依存すべきではありません。

**信頼性は副次的な利益になります。** OpenAIが定期的な障害を起こしたとき（実際に起こります）、ルーターはトラフィックを他のプロバイダーにリダイレクトできます。特定のAPIがオンラインに戻るのを待っている間にシステムが停止するわけではありません。

これは単に賢く見えるためにやっているわけではありません。これは財務的にも技術的にも意味のあるシステムを構築することです。同じハンマーですべての建設作業を行うように、すべてのAIタスクに同じ言語モデルを使うべきではありません。

このアプローチの美しさはアプリケーションコードが変更不要だということです。ルーターエージェントを呼び出すだけです。どのタスクにどのモデルを使うかを決める複雑さは1か所に配置され、一度設定すればコードベース中の条件分岐の論理に散らばることはありません。
