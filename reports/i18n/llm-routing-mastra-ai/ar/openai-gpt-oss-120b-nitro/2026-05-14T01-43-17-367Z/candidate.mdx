معظم فرق الهندسة تختار نموذج لغة وتلتزم به. مزود واحد، نموذج واحد، جميع المهام. الأمر يشبه توظيف شخص واحد للقيام بالبرمجة، والكتابة الإعلانية، والضرائب لأنهم كانوا جيدين في المقابلة الأولى.

في أي لحظة، نموذج واحد يكون أفضل في كتابة الشيفرة، وآخر أفضل في التعامل مع سياق طويل وفوضوي، وآخر هو الأرخص للمهام المملة مثل التصنيف. الأسماء تتغير. شكل المشكلة لا يتغير. معاملة نموذج واحد كأنه يتقن كل شيء يعني إما أنك تدفع أكثر من اللازم للمهام البسيطة أو تحصل على نتائج دون المستوى للمهام المتخصصة.

شاهدت فريقًا يحرق آلاف الدولارات في تشغيل تحليل المشاعر عبر نموذج يكلف 30 دولارًا لكل مليون رمز بينما نموذج بـ 0.50 دولار كان سيؤدي المهمة بنفس الكفاءة. تنسيق JSON بسيط، مهام تصنيف أساسية، كلها تمر عبر مزودهم الفاخر. الشيء الوحيد الذي ارتفع هو فاتورة AWS.

هناك طريقة أفضل، وهي ليست معقدة بالذات.

## التفويض فوق الإخلاص

ماذا لو كان بإمكانك توجيه الطلبات إلى النموذج الأنسب لتلك المهمة المحددة؟ استخدم القوة الغالية للمهام الصعبة، لكن خفّض التكلفة للمهام البسيطة مثل التحليل والتنسيق إلى نموذج أرخص. احصل على فوائد تعدد المزودين دون الحاجة إلى إدارة يدوية لهم في قاعدة الشيفرة.

Mastra يتيح لك بناء هذا النوع من الأنظمة. تقوم بإنشاء وكلاء متخصصين لأنواع مختلفة من العمل، ثم تنشئ وكيل توجيه يحدد أي متخصص يجب أن يتعامل مع كل طلب. معرفات النماذج أدناه أمثلة، ليست ترتيبًا تنافسيًا. استبدلها بالنماذج الحالية التي تتفوق في تقييماتك وتناسب ميزانيتك.

فكر في الأمر هكذا: لديك ثلاثة متخصصين في فريقك.

```typescript
// ./src/mastra/index.ts
import { Mastra } from '@mastra/core';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { anthropic } from '@ai-sdk/anthropic';
import { google } from '@ai-sdk/google';

export const claudeAgent = new Agent({
  id: 'claude-agent',
  instructions: 'You are an expert engineer. Write bugs? You are fired.',
  model: anthropic(process.env.CODE_MODEL ?? 'claude-sonnet-4-5'),
});

export const geminiAgent = new Agent({
  id: 'gemini-agent',
  instructions: 'You are a creative writer. Be weird.',
  model: google(process.env.LONG_CONTEXT_MODEL ?? 'gemini-3-pro-preview'),
});

export const gptAgent = new Agent({
  id: 'gpt-agent',
  instructions: 'You are a helpful assistant. Be boring.',
  model: openai(process.env.GENERAL_MODEL ?? 'gpt-5.2'),
});
```

لكل واحد وظيفة. يجب أن يكون وكيل الشيفرة هو النموذج الذي يجتاز تقييمات البرمجة الخاصة بمستودعك. يجب أن يكون وكيل السياق الطويل هو النموذج الذي يتحمل مستنداتك الفعلية دون تحويلها إلى حساء. يجب أن يكون وكيلك العام رخيصًا، موثوقًا، ومملًا بأفضل طريقة ممكنة.

هنا يصبح الأمر مثيرًا. تضيف وكيل توجيه يعمل كوكيل ذكي للطلبات:

```typescript
export const routerAgent = new Agent({
  id: 'router-agent',
  name: 'The Boss',
  instructions: `You are an intelligent router.
  - Coding -> Claude
  - Poetry -> Gemini
  - Facts -> GPT

  Do not do the work yourself. Delegate.`,
  model: openai(process.env.ROUTER_MODEL ?? 'gpt-5-mini'), // Use a cheap model for routing!
  agents: {
    claudeAgent,
    geminiAgent,
    gptAgent,
  },
});

export const mastra = new Mastra({
  agents: { routerAgent, claudeAgent, geminiAgent, gptAgent },
});
```

الموجه نفسه يعمل على نموذج خفيف الوزن لأنه يقتصر فقط على اتخاذ قرارات حول وجهة الطلبات. لا تدفع أسعارًا متميزة لتحديد أي نموذج متميز آخر يجب استخدامه. قس هذا أيضًا؛ فالموجه السيء يحول المدخرات بهدوء إلى توجيهات خاطئة.

عندما يطلب أحدهم تنفيذ خوارزمية فرز الفقاعات، يتعرف الموجه على ذلك كعمل برمجي ويُسلم المهمة إلى أخصائي الشيفرة لديك. طلب كتابة إبداعية؟ يُرسل إلى النموذج الذي اخترته للـ«صوت» والنطاق. سؤال واقعي عن أحداث تاريخية؟ يُوجّه إلى الوكيل العام، ويفضل أن يكون مع استرجاع عندما تكون الحداثة أو الاستشهاد مهمين.

## الفوائد العملية

**كفاءة التكلفة أهم مما تتصور.** نموذج توجيه صغير يتخذ قرارات التفويض يكلف جزءًا ضئيلًا من تشغيل كل طلب عبر أغلى مزود لديك. على المدى الطويل، خاصةً عند التوسع، يتراكم ذلك إلى مبالغ حقيقية. تدفع فقط مقابل الذكاء الثقيل عندما تحتاجه فعليًا.

**تحسّن الجودة عندما تطابق النماذج مع المهام.** الفائز يتغيّر شهريًا، حسب المهمة، وشكل الطلب. لهذا يجب أن يعتمد طبقة التوجيه على تقييماتك، لا على النموذج الذي كان يهيمن على تويتر في الأسبوع الذي كتبت فيه التكامل.

**المرونة تصبح فائدة جانبية.** عندما تواجه OpenAI أحد انقطاعاتها الدورية (وهي تحدث)، يستطيع الموجه إعادة توجيه الحركة إلى مزودين آخرين. لا تبقى عالقًا في انتظار عودة واجهة برمجة تطبيقات واحدة إلى العمل.

هذا ليس مجرد محاولة للذكاء من أجل الذكاء. إنه بناء أنظمة منطقية من الناحيتين المالية والتقنية. لن تستخدم نفس المِطرقة لكل مهمة بناء، ولا ينبغي عليك استخدام نفس نموذج اللغة لكل مهمة ذكاء اصطناعي.

جمال هذا النهج هو أن شفرة التطبيق لا تتغير. ما زلت تستدعي وكيل التوجيه فقط. تعقيد اتخاذ قرار أي نموذج يُستَخدم لأي مهمة يُحفظ في مكان واحد، يُضبط مرة واحدة، بدلاً من أن يتناثر عبر قاعدة الشفرة في مجموعة من الشروط المتفرقة.
