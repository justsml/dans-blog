Die meisten Engineering‑Teams wählen ein Sprachmodell und bleiben dabei. Ein Anbieter, ein Modell, alle Aufgaben. Das ist, als würde man eine Person einstellen, die sowohl das Programmieren, das Verfassen von Texten als auch die Steuererklärung übernimmt, weil sie im ersten Vorstellungsgespräch gut abgeschnitten hat.

Zu jedem Zeitpunkt ist ein Modell besser im Codieren, ein anderes besser bei langen, unübersichtlichen Kontexten, und ein weiteres das günstigste Arbeitspferd für reine Klassifikationsaufgaben. Die Namen ändern sich. Die Struktur des Problems bleibt gleich. Ein Modell wie ein Alleskönner zu behandeln bedeutet, dass man entweder für einfache Aufgaben zu viel bezahlt oder bei spezialisierten Aufgaben unbefriedigende Ergebnisse erhält.

Ich habe ein Team beobachtet, das tausende Dollar verbrannte, weil es Sentiment‑Analysen durch ein 30 $‑pro‑Million‑Token‑Modell laufen ließ, obwohl ein 0,50 $‑Modell die Aufgabe genauso gut erledigt hätte. Einfaches JSON‑Formatting, grundlegende Klassifikationsaufgaben – alles über ihren Premium‑Anbieter. Das Einzige, das dabei heiß wurde, war ihre AWS‑Rechnung.

Es gibt einen besseren Weg, und er ist nicht besonders kompliziert.

## Delegation statt Devotion

Was wäre, wenn Sie Anfragen zu dem Modell routen könnten, das wirklich am besten für die jeweilige Aufgabe geeignet ist? Nutzen Sie Ihren teuren Hochleistungspartner für die schwierigen Fälle, aber leiten Sie einfaches Parsen und Formatieren an ein günstigeres Modell weiter. So erhalten Sie die Vorteile mehrerer Anbieter, ohne sie manuell im Codebase jonglieren zu müssen.

Mastra ermöglicht genau dieses System. Sie definieren Fach‑Agenten für verschiedene Aufgabentypen und erstellen dann einen Router‑Agent, der ermittelt, welcher Fach‑Agent jede Anfrage bearbeiten soll. Die unten gezeigten Modell‑IDs dienen nur als Beispiele, nicht als Ranking. Ersetzen Sie sie durch die aktuellen Modelle, die Ihre Evaluierungen gewinnen und in Ihr Budget passen.

Stellen Sie sich das so vor: Sie haben drei Spezialisten in Ihrem Team.

```typescript
// ./src/mastra/index.ts
import { Mastra } from '@mastra/core';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { anthropic } from '@ai-sdk/anthropic';
import { google } from '@ai-sdk/google';

export const claudeAgent = new Agent({
  id: 'claude-agent',
  instructions: 'You are an expert engineer. Write bugs? You are fired.',
  model: anthropic(process.env.CODE_MODEL ?? 'claude-sonnet-4-5'),
});

export const geminiAgent = new Agent({
  id: 'gemini-agent',
  instructions: 'You are a creative writer. Be weird.',
  model: google(process.env.LONG_CONTEXT_MODEL ?? 'gemini-3-pro-preview'),
});

export const gptAgent = new Agent({
  id: 'gpt-agent',
  instructions: 'You are a helpful assistant. Be boring.',
  model: openai(process.env.GENERAL_MODEL ?? 'gpt-5.2'),
});
```

Jeder hat seine Aufgabe. Ihr Code‑Agent sollte das Modell sein, das Ihre repository‑spezifischen Code‑Evaluierungen besteht. Ihr Long‑Context‑Agent muss die langen Dokumente verarbeiten können, ohne dass der Mittelteil zu Brei wird. Ihr General‑Agent sollte günstig, zuverlässig und auf die nüchternste Art „langweilig“ sein.

Jetzt wird es interessant. Sie fügen einen Router hinzu, der als intelligenter Proxy fungiert:

```typescript
export const routerAgent = new Agent({
  id: 'router-agent',
  name: 'The Boss',
  instructions: `You are an intelligent router.
  - Coding -> Claude
  - Poetry -> Gemini
  - Facts -> GPT

  Do not do the work yourself. Delegate.`,
  model: openai(process.env.ROUTER_MODEL ?? 'gpt-5-mini'), // Use a cheap model for routing!
  agents: {
    claudeAgent,
    geminiAgent,
    gptAgent,
  },
});

export const mastra = new Mastra({
  agents: { routerAgent, claudeAgent, geminiAgent, gptAgent },
});
```

Der Router selbst läuft auf einem leichtgewichtigen Modell, weil er nur entscheidet, wohin der Traffic geleitet wird. Sie zahlen also keine Premium‑Tarife, um zu bestimmen, welches andere Premium‑Modell verwendet werden soll. Messen Sie das ebenfalls; ein schlechter Router wandelt Einsparungen still in Fehlleitungen um.

Wenn jemand nach einer Bubble‑Sort‑Implementierung fragt, erkennt der Router die Aufgabe als Code‑Arbeit und leitet sie an Ihren Code‑Spezialisten weiter. Ein kreativer Schreib‑Prompt? Der geht an das Modell, das Sie für Stimme und Ausdruck ausgewählt haben. Eine faktische Frage zu historischen Ereignissen? Sie wird an den General‑Agent weitergeleitet, idealerweise mit Retrieval, wenn Aktualität oder Zitation wichtig sind.

## The Practical Benefits

**Kosten‑Effizienz ist wichtiger, als Sie denken.** Ein kleines Routing‑Modell, das Delegationsentscheidungen trifft, kostet nur einen Bruchteil dessen, was entsteht, wenn jede einzelne Anfrage über Ihren teuersten Anbieter läuft. Auf lange Sicht, besonders bei großem Volumen, summiert sich das zu echtem Geld. Sie zahlen nur für die Hochleistungs‑Intelligenz, wenn Sie sie tatsächlich benötigen.

**Qualität steigt, wenn Sie Modelle den Aufgaben zuordnen.** Der Sieger wechselt von Monat zu Monat, je nach Aufgabe und Prompt‑Form. Deshalb sollte die Routing‑Schicht von Ihren Evaluierungen abhängen und nicht von dem Modell, das gerade auf Twitter gerade die Nase vorn hat, als Sie die Integration geschrieben haben.

**Resilienz wird zum Nebeneffekt.** Wenn OpenAI einen seiner periodischen Ausfälle hat (und das kommt vor), kann Ihr Router den Traffic zu anderen Anbietern umleiten. Sie sind nicht hilflos am Ufer und warten darauf, dass eine bestimmte API wieder online geht.

Dabei geht es nicht um cleveres Spielerei. Es geht darum, Systeme zu bauen, die sowohl finanziell als auch technisch Sinn ergeben. Man würde nicht denselben Hammer für jede Bauaufgabe verwenden, und Sie sollten auch nicht dasselbe Sprachmodell für jede KI‑Aufgabe einsetzen.

Der Vorteil dieses Ansatzes ist, dass sich Ihr Anwendungscode nicht ändert. Sie rufen weiterhin einfach Ihren Router‑Agenten auf. Die Komplexität, welches Modell für welche Aufgabe verwendet wird, steckt an einer Stelle, wird einmal konfiguriert und bleibt dort, anstatt im gesamten Code‑Base in einer Menge von Bedingungslogik zu verstreuen.
