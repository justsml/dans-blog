רוב צוותי ההנדסה בוחרים מודל שפה אחד ונשארים איתו. ספק אחד, מודל אחד, כל המשימות. זה כמו לשכור אדם אחד שיעשה את כל הקוד שלך, את כתיבת הטקסטים שלך, ואת המיסים שלך רק בגלל שהוא הצליח בראיון הראשון.

בכל רגע נתון, מודל אחד טוב יותר בקוד, מודל אחר מתמודד טוב יותר עם הקשר ארוך ומבולבל, ומודל שלישי הוא הפשוט והזול ביותר למשימות סיווג. השמות משתנים. צורת הבעיה אינה משתנה. להתייחס למודל אחד כאילו הוא מצטיין בכל תחום משמעותו שאתה או משלם יותר מדי עבור משימות פשוטות או מקבל תוצאות מתחת לסטנדרט עבור משימות מיוחדות.

צפיתי בצוות ששרף אלפי דולרים על ניתוח רגשות באמצעות מודל שעולה 30 דולר למיליון טוקנים, בעוד שמודל של 0.50 דולר היה עושה את העבודה באותה איכות. עיצוב JSON פשוט, משימות סיווג בסיסיות, כולם עברו דרך ספק הפרימיום שלהם. הדבר היחיד שהתחמם היה חשבון ה‑AWS שלהם.

יש דרך טובה יותר, והיא לא מורכבת במיוחד.

## האצלה במקום קיבוד

מה אם תוכל לנתב בקשות למודל שהכי מתאים למשימה הספציפית? להשתמש במכונה היקרה שלך עבור המשימות הקשות, אבל להעביר את העיבוד הפשוט של ניתוח ופורמטינג למשהו זול יותר. לקבל את היתרונות של כמה ספקים מבלי לצרף אותם ידנית בקוד שלך.

Mastra מאפשרת לך לבנות בדיוק מערכת כזו. אתה מגדיר סוכני‑מומחים לסוגי עבודה שונים, ואז יוצר סוכן‑נתב שמחליט איזה מומחה צריך לטפל בכל בקשה. מזהי המודלים למטה הם דוגמאות, לא דירוג. החלף אותם במודלים העדכניים שמנצחים את ההערכות שלך ומתאימים לתקציב.

תחשוב על זה כך: יש לך שלושה מומחים בצוות.

```typescript
// ./src/mastra/index.ts
import { Mastra } from '@mastra/core';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { anthropic } from '@ai-sdk/anthropic';
import { google } from '@ai-sdk/google';

export const claudeAgent = new Agent({
  id: 'claude-agent',
  instructions: 'You are an expert engineer. Write bugs? You are fired.',
  model: anthropic(process.env.CODE_MODEL ?? 'claude-sonnet-4-5'),
});

export const geminiAgent = new Agent({
  id: 'gemini-agent',
  instructions: 'You are a creative writer. Be weird.',
  model: google(process.env.LONG_CONTEXT_MODEL ?? 'gemini-3-pro-preview'),
});

export const gptAgent = new Agent({
  id: 'gpt-agent',
  instructions: 'You are a helpful assistant. Be boring.',
  model: openai(process.env.GENERAL_MODEL ?? 'gpt-5.2'),
});
```

לכל אחד תפקיד. סוכן הקוד שלך צריך להיות המודל שעובר את מבחני הקוד הספציפיים למאגר שלך. סוכן הקשר הארוך צריך להיות זה שמצליח לעבד את המסמכים האמיתיים שלך מבלי להפוך את האמצע למרק. הסוכן הכללי צריך להיות זול, אמין, ומשעמם במובן הטוב ביותר.

הנה המקום שבו זה הופך למעניין. אתה מוסיף נתב שפועל כפרוקסי אינטיליגנטי:

```typescriptexport const routerAgent = new Agent({
  id: 'router-agent',
  name: 'The Boss',
  instructions: `You are an intelligent router.
  - Coding -> Claude
  - Poetry -> Gemini
  - Facts -> GPT

  Do not do the work yourself. Delegate.`,
  model: openai(process.env.ROUTER_MODEL ?? 'gpt-5-mini'), // Use a cheap model for routing!
  agents: {
    claudeAgent,
    geminiAgent,
    gptAgent,
  },
});

export const mastra = new Mastra({
  agents: { routerAgent, claudeAgent, geminiAgent, gptAgent },
});
```

הנתב עצמו רץ על מודל קל משקל מכיוון שהוא רק מחליט לאן לשלוח את התנועה. אתה לא משלם תעריפים פרימיום כדי לגלות איזה מודל פרימיום אחר להשתמש. יש למדוד זאת גם; נתב גרוע עלול לשקול חסכון למסלולים שגויים.

כאשר מישהו מבקש מימוש של מיון בועות, הנתב מזהה זאת כעבודה קודית ומעביר אותה למומחה הקוד שלך. בקשת כתיבה יצירתית? זה הולך למודל שבחרת עבור סגנון וטווח. שאלה עובדתית על אירועים היסטוריים? הפנה אותה לסוכן הכללי, רצוי עם שליפה כאשר רלוונטיות או ציטוט חשובים.

## היתרונות המעשיים

**יעילות על‑העלות חשובה יותר ממה שאתה חושב.** מודל ניתוב קטן שמקבל החלטות ההצבה עולה חלק מזעיר מהפעלת כל בקשה דרך הספק היקר ביותר שלך. עם הזמן, במיוחד בקנה מידה גדול, זה מצטבר לכסף אמיתי. אתה משלם רק עבור האינטליגנציה הכבדה כשאתה באמת צריך אותה.

**האיכות משתפרת כשמתאימים מודלים למשימות.** המנצח משתנה מחודש לחודש, משימה לצורה של הפרומפט. לכן שכבת הניתוב צריכה להתבסס על ההערכות שלך, ולא על מודל שזכה בתשומת לב בטוויטר באותו שבוע שבו כתבת את האינטגרציה.

**החוסן הופך לתועלת משנית.** כאשר ל‑OpenAI יש אחת מהפסקות השירות התקופתיות (והן קורות), הנתב שלך יכול להפנות תנועה לספקים אחרים. אתה לא נתקע במצב המתנה עד שה‑API הספציפי יחזור לפעולה.

זה לא עניין של להיות חכם לשם עצמו. מדובר בבניית מערכות שהן הגיוניות גם מבחינת העלות וגם מבחינת הטכנולוגיה. לא תשתמש באותו פטיש לכל משימת בנייה, וככל הנראה גם לא תשתמש באותו מודל שפה לכל משימת AI.

היופי בגישה הזו הוא שקוד האפליקציה שלך אינו משתנה. אתה עדיין קורא רק לסוכן הנתב. המורכבות של החלטת המודל המתאים לכל משימה נשארת במקום אחד, מוגדרת פעם אחת, במקום להתפזר ברחבי הקוד שלך במגוון תנאים.
