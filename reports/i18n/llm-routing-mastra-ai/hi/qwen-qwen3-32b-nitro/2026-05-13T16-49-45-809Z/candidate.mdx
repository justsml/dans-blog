अधिकांश इंजीनियरिंग टीमें एक भाषा मॉडल चुनती हैं और उससे बाहर नहीं निकलती। एक प्रदाता, एक मॉडल, सभी कार्य। यह उसी तरह है जैसे आप एक व्यक्ति को अपने कोडिंग, कॉपीराइटिंग और कर के काम करने के लिए नियुक्त कर रहे हों क्योंकि वह पहले साक्षात्कार में अच्छा रहा।

किसी भी दिए गए समय में, एक मॉडल कोड के लिए बेहतर होता है, दूसरा लंबे अस्पष्ट संदर्भ के साथ बेहतर होता है, और तीसरा वर्गीकरण के लिए सस्ता साधारण कार्य बनाता है। नाम बदल जाते हैं। समस्या का आकार नहीं बदलता। एक मॉडल को सब कुछ में शीर्ष पर रखने का तरीका यह बताता है कि आप या तो साधारण कार्यों के लिए अतिरिक्त भुगतान कर रहे हैं या विशेषज्ञता के कार्यों पर खराब परिणाम प्राप्त कर रहे हैं।

मैंने एक टीम को हजारों डॉलर बर्बर करने देखा जबकि एक $30-प्रति-मिलियन-टोकन मॉडल के माध्यम से भावनात्मक विश्लेषण चल रहा था, जबकि $0.50 के मॉडल ने इस कार्य को उतना ही अच्छा कर सकता था। सरल JSON फॉर्मेटिंग, आधारभूत वर्गीकरण कार्य, सभी अपने प्रीमियम प्रदाता के माध्यम से चल रहे थे। गर्म हो रहा था केवल उनका AWS बिल।

इसके बजाय एक बेहतर तरीका है, और यह विशेष रूप से जटिल नहीं है।

## निर्देशन के बजाय निर्देशन

क्या आप उस मॉडल के लिए अनुरोधों को पुनर्निर्देशित कर सकते हैं जो वास्तव में उस विशिष्ट कार्य के लिए अनुकूलित है? मुश्किल काम के लिए अपना महंगा शक्तिशाली उपयोग करें, लेकिन सरल विश्लेषण और फॉर्मेटिंग को कुछ अधिक सस्ता मॉडल पर डाल दें। बिना अपने कोडबेस में उन्हें मैन्युअल रूप से जूल किए बिना कई प्रदाताओं के लाभ प्राप्त करें।

मास्त्रा आपको ठीक इस तरह की प्रणाली बनाने की अनुमति देता है। आप विभिन्न प्रकार के कार्यों के लिए विशेषज्ञ एजेंट्स स्थापित करते हैं, फिर एक रूटर एजेंट बनाते हैं जो यह निर्धारित करता है कि प्रत्येक अनुरोध किस विशेषज्ञ द्वारा संभाला जाएगा। नीचे दिए गए मॉडल ID उदाहरण हैं, न कि कोई रैंकिंग। उन्हें अपने एवल्यूएशन में जीते हुए वर्तमान मॉडल और अपने बजट में फिट होने वाले मॉडलों के साथ बदल दें।

इसे इस तरह सोचें: आपकी टीम में तीन विशेषज्ञ हैं।

```typescript
// ./src/mastra/index.ts
import { Mastra } from '@mastra/core';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { anthropic } from '@ai-sdk/anthropic';
import { google } from '@ai-sdk/google';

export const claudeAgent = new Agent({
  id: 'claude-agent',
  instructions: 'You are an expert engineer. Write bugs? You are fired.',
  model: anthropic(process.env.CODE_MODEL ?? 'claude-sonnet-4-5'),
});

export const geminiAgent = new Agent({
  id: 'gemini-agent',
  instructions: 'You are a creative writer. Be weird.',
  model: google(process.env.LONG_CONTEXT_MODEL ?? 'gemini-3-pro-preview'),
});

export const gptAgent = new Agent({
  id: 'gpt-agent',
  instructions: 'You are a helpful assistant. Be boring.',
  model: openai(process.env.GENERAL_MODEL ?? 'gpt-5.2'),
});
```

प्रत्येक के पास एक कार्य है। आपका कोड एजेंट वह मॉडल होना चाहिए जो आपके रिपो-विशिष्ट कोडिंग एवल्यूएशन को पास करता है। आपका लंबे संदर्भ एजेंट वह होना चाहिए जो आपके वास्तविक दस्तावेजों को बिना बीच में घुले बिना संभाल ले। आपका सामान्य एजेंट सस्ता, विश्वसनीय और सबसे अच्छे तरीके से बोरिंग होना चाहिए।

यहां दिलचस्प बात आती है। आप एक रूटर जोड़ते हैं जो एक बुद्धिमान प्रॉक्सी के रूप में कार्य करता है:

```typescript
export const routerAgent = new Agent({
  id: 'router-agent',
  name: 'The Boss',
  instructions: `You are an intelligent router.
  - Coding -> Claude
  - Poetry -> Gemini
  - Facts -> GPT

  Do not do the work yourself. Delegate.`,
  model: openai(process.env.ROUTER_MODEL ?? 'gpt-5-mini'), // Use a cheap model for routing!
  agents: {
    claudeAgent,
    geminiAgent,
    gptAgent,
  },
});

export const mastra = new Mastra({
  agents: { routerAgent, claudeAgent, geminiAgent, gptAgent },
});
```

रूटर खुद एक हल्के मॉडल पर चलता है क्योंकि यह बस यह निर्धारित कर रहा है कि ट्रैफ़िक कहां भेजा जाए। आप उच्च दरों का भुगतान नहीं कर रहे हैं ताकि यह निर्धारित कर सकें कि कौन सा अन्य उच्च दर का मॉडल उपयोग किया जाए। इसको भी मापें; एक खराब रूटर बचत को गलत रूटिंग में बदल देता है।

जब कोई बुबल सॉर्ट की कार्यरति के लिए पूछता है, तो रूटर इसे कोड कार्य के रूप में पहचानता है और इसे आपके कोड विशेषज्ञ को सौंप देता है। क्रिएटिव राइटिंग का प्राम्प्ट? यह आपके द्वारा आवाज और विस्तार के लिए चुने गए मॉडल में जाता है। ऐतिहासिक घटनाओं पर तथ्यात्मक प्रश्न? इसे सामान्य एजेंट को रूट करें, आमतौर पर रिट्रीवल के साथ जब ताजगी या संदर्भ की आवश्यकता हो।

## अभ्यास के लाभ  

**कीमती कारकों के लाभ का महत्व आपकी कल्पना से अधिक है।** एक छोटे रूटिंग मॉडल जो निर्णय लेता है, आपके सबसे महंगे प्रदाता के माध्यम से प्रत्येक अनुरोध चलाने की तुलना में लागत का एक अंश होता है। समय के साथ, विशेष रूप से बड़े पैमाने पर, यह वास्तविक पैसा बचाता है। आप केवल तभी भारी मुख्य बुद्धि के लिए भुगतान करते हैं जब आपको वास्तव में आवश्यकता होती है।  

**क्वालिटी बेहतर होती है जब आप मॉडल को कार्यों से मेल करते हैं।** महीने, कार्य और प्राम्प्ट आकार द्वारा विजेता बदलता रहता है। यही कारण है कि रूटिंग परत आपके एवल्स पर निर्भर करनी चाहिए, न कि उस मॉडल पर जो आपने एकीकरण लिखे जाने के सप्ताह में ट्विटर पर जीता हो।  

**टिकाऊपन एक अतिरिक्त लाभ बन जाता है।** जब OpenAI के एक आवधिक बंद होने (और वे करते हैं) होते हैं, तो आपका रूटर अन्य प्रदाताओं की ओर ट्रैफ़िक को रीडायरेक्ट कर सकता है। आप किसी विशेष API के वापस ऑनलाइन आने के लिए प्रतीक्षा करते हुए पानी में नहीं डूबे रहते हैं।  

यह केवल बुद्धिमान होने के लिए नहीं है। यह ऐसे प्रणाली बनाने के बारे में है जो वित्तीय और तकनीकी दोनों तरफ से बुद्धिमान हैं। आप एक ही हथौड़ा हर निर्माण कार्य के लिए उपयोग नहीं करेंगे, और आपको संभवतः हर AI कार्य के लिए एक ही भाषा मॉडल उपयोग नहीं करना चाहिए।  

इस दृष्टिकोण की सुंदरता इस तथ्य में है कि आपका एप्लिकेशन कोड नहीं बदलता है। आप अभी भी अपने रूटर एजेंट को कॉल करते रहते हैं। कौन से मॉडल का उपयोग कौन से कार्य के लिए करने के जटिलता एक जगह में रहता है, एक बार कॉन्फ़िगर किया गया, बजाय अपने कोडबेस में फैले हुए कई स्थिति लॉजिक के।
