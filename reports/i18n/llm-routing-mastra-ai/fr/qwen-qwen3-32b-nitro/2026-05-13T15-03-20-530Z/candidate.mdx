La plupart des équipes techniques choisissent un modèle de langage et s'y tiennent. Un seul fournisseur, un seul modèle, toutes les tâches. C'est comme embaucher une seule personne pour faire votre codage, votre rédaction de contenus et vos impôts parce qu'elle s'est bien comportée lors du premier entretien.

À tout moment, un modèle est meilleur en codage, un autre excelle avec un contexte long et encombré, et un troisième est le cheval de bataille peu coûteux pour les tâches simples. Les noms changent. La forme du problème non. Traiter un modèle comme s'il était excellent dans tout signifie que vous payez trop cher pour des tâches simples ou obtenez des résultats médiocres pour des tâches spécialisées.

J'ai vu une équipe dépenser des milliers de dollars en exécutant une analyse de sentiment via un modèle à 30 dollars par million de tokens alors qu'un modèle à 0,50 dollar aurait fait le travail aussi bien. Un simple formatage JSON, des tâches de classification basiques, tout cela passant par leur fournisseur premium. La seule chose à s'échauffer était leur facture AWS.

Il existe une meilleure approche, et elle n'est pas particulièrement complexe.

## Délégation au lieu de fidélité

Imaginez pouvoir router les requêtes vers le modèle qui est effectivement le mieux adapté à cette tâche spécifique. Utilisez votre modèle puissant pour les tâches difficiles, mais déléguer les analyses simples et le formatage à quelque chose de moins coûteux. Obtenez les avantages de plusieurs fournisseurs sans avoir à les gérer manuellement dans votre code.

Mastra vous permet de construire exactement ce type de système. Vous configurez des agents spécialisés pour différents types de tâches, puis créez un agent "router" qui détermine quel spécialiste doit gérer chaque requête. Les IDs de modèles ci-dessous sont des exemples, pas un classement. Remplacez-les par les modèles actuels qui gagnent vos évaluations et s'adaptent à votre budget.

Imaginez le comme ceci : vous avez trois spécialistes dans votre équipe.

```typescript
// ./src/mastra/index.ts
import { Mastra } from '@mastra/core';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { anthropic } from '@ai-sdk/anthropic';
import { google } from '@ai-sdk/google';

export const claudeAgent = new Agent({
  id: 'claude-agent',
  instructions: 'Vous êtes un ingénieur expert. Vous écrivez des bugs ? Vous êtes viré.',
  model: anthropic(process.env.CODE_MODEL ?? 'claude-sonnet-4-5'),
});

export const geminiAgent = new Agent({
  id: 'gemini-agent',
  instructions: 'Vous êtes un écrivain créatif. Soyez étrange.',
  model: google(process.env.LONG_CONTEXT_MODEL ?? 'gemini-3-pro-preview'),
});

export const gptAgent = new Agent({
  id: 'gpt-agent',
  instructions: 'Vous êtes un assistant utile. Soyez ennuyeux.',
  model: openai(process.env.GENERAL_MODEL ?? 'gpt-5.2'),
});
```

Chacun a un rôle spécifique. Votre agent de codage devrait être le modèle qui passe vos évaluations de codage spécifiques à votre dépôt. Votre agent à long contexte devrait être celui qui gère vos documents réels sans les transformer en bouillie. Votre agent général devrait être peu coûteux, fiable, et ennuyeux dans le sens le plus positif.

Voici où cela devient intéressant. Vous ajoutez un routeur qui agit comme un proxy intelligent :

```typescript
export const routerAgent = new Agent({
  id: 'router-agent',
  name: 'The Boss',
  instructions: `Vous êtes un routeur intelligent.
  - Codage -> Claude
  - Poésie -> Gemini
  - Faits -> GPT

  Ne faites pas le travail vous-même. Déléguez.`,
  model: openai(process.env.ROUTER_MODEL ?? 'gpt-5-mini'), // Utilisez un modèle peu coûteux pour le routage !
  agents: {
    claudeAgent,
    geminiAgent,
    gptAgent,
  },
});

export const mastra = new Mastra({
  agents: { routerAgent, claudeAgent, geminiAgent, gptAgent },
});
```

Le routeur lui-même s'exécute sur un modèle léger car il ne prend que des décisions sur la destination du trafic. Vous ne payez pas des tarifs premium pour déterminer quel autre modèle premium utiliser. Mesurez aussi cela : un mauvais routeur transforme silencieusement les économies en redirections incorrectes.

Quand quelqu'un demande une implémentation de tri à bulles, le routeur le reconnaît comme un travail de codage et le transmet à votre spécialiste code. Un prompt d'écriture créative ? Cela va au modèle que vous avez choisi pour sa voix et sa gamme. Une question factuelle sur des événements historiques ? Routez-la vers l'agent général, idéalement avec une recherche quand la fraîcheur ou les citations sont importantes.

## Les avantages pratiques

**L'efficacité en termes de coûts compte plus que vous ne le pensez.** Un petit modèle de routage qui prend les décisions de délégation coûte une fraction du coût de passer chaque requête via votre fournisseur le plus cher. Au fil du temps, surtout à grande échelle, cela représente de l'argent réel. Vous ne payez pour l'intelligence lourde que lorsque vous en avez vraiment besoin.

**La qualité s'améliore lorsque vous adaptez les modèles aux tâches.** Le modèle gagnant varie selon le mois, la tâche et la forme des prompts. C'est pourquoi la couche de routage doit s'appuyer sur vos évaluations, et non sur le modèle qui dominait Twitter la semaine où vous avez intégré le code.

**La résilience devient un avantage secondaire.** Quand OpenAI connaît l'une de ses pannes périodiques (et cela arrive), votre routeur peut rediriger le trafic vers d'autres fournisseurs. Vous n'êtes pas bloqué en attendant qu'une API spécifique revienne en ligne.

Ce n'est pas question de malice pour le plaisir. C'est question de construire des systèmes qui ont du sens à la fois sur le plan financier et technique. Vous n'utiliseriez pas la même marteau pour chaque tâche de construction, et vous ne devriez probablement pas non plus utiliser le même modèle de langage pour chaque tâche d'IA.

La beauté de cette approche réside dans le fait que votre code d'application ne change pas. Vous appelez toujours votre agent routeur. La complexité de la décision du modèle à utiliser pour chaque tâche est cantonnée à un seul endroit, configuré une seule fois, plutôt que disséminée dans tout votre codebase sous forme de logique conditionnelle.
