Большинство инженерных команд выбирают одну модель и придерживаются её. Один поставщик, одна модель, все задачи. Это как нанять одного человека, чтобы он писал ваш код, писал тексты и подавал декларации о налогах, потому что он оказался хорош на первом интервью.

В любой момент времени одна модель лучше справляется с кодом, другая — с длинным запутанным контекстом, а третья — самая дешёвая и надёжная для классификации. ## Делегирование вместо преданности

А что, если вы могли бы направлять запросы к модели, которая действительно лучше всего подходит для конкретной задачи? Используйте свою дорогую мощную модель для сложных задач, но опускайте простую разборку и форматирование к чему-то дешёвому. Получайте преимущества нескольких поставщиков, не управляя ими вручную в своём коде.

Mastra позволяет создать именно такую систему. Вы настраиваете специалистов для разных типов задач, а затем создаёте роутер, который определяет, какой специалист должен обрабатывать каждый запрос. Ниже приведены идентификаторы моделей в качестве примеров, а не рейтинга. Замените их на текущие модели, которые выигрывают ваши оценки и подходят под ваш бюджет.

Представьте так: у вас три специалиста в команде.

```typescript
// ./src/mastra/index.ts
import { Mastra } from '@mastra/core';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { anthropic } from '@ai-sdk/anthropic';
import { google } from '@ai-sdk/google';

export const claudeAgent = new Agent({
  id: 'claude-agent',
  instructions: 'You are an expert engineer. Write bugs? You are fired.',
  model: anthropic(process.env.CODE_MODEL ?? 'claude-sonnet-4-5'),
});

export const geminiAgent = new Agent({
  id: 'gemini-agent',
  instructions: 'You are a creative writer. Be weird.',
  model: google(process.env.LONG_CONTEXT_MODEL ?? 'gemini-3-pro-preview'),
});

export const gptAgent = new Agent({
  id: 'gpt-agent',
  instructions: 'You are a helpful assistant. Be boring.',
  model: openai(process.env.GENERAL_MODEL ?? 'gpt-5.2'),
});
```

У каждого есть своя работа. Ваш агент для кода должен быть моделью, которая проходит ваши оценки, специфичные для репозитория. Ваш агент с длинным контекстом должен выдерживать ваши реальные документы, не превращая середину в кашу. Ваш общий агент должен быть дешёвым, надёжным и скучным в самом лучшем смысле.

Теперь интересная часть. Вы добавляете роутер, который действует как умный прокси:

```typescript
export const routerAgent = new Agent({
  id: 'router-agent',
  name: 'The Boss',
  instructions: `You are an intelligent router.
  - Coding -> Claude
  - Poetry -> Gemini
  - Facts -> GPT

  Do not do the work yourself. Delegate.`,
  model: openai(process.env.ROUTER_MODEL ?? 'gpt-5-mini'), // Use a cheap model for routing!
  agents: {
    claudeAgent,
    geminiAgent,
    gptAgent,
  },
});

export const mastra = new Mastra({
  agents: { routerAgent, claudeAgent, geminiAgent, gptAgent },
});
```

Сам роутер работает на лёгкой модели, потому что он просто принимает решения о направлении трафика. Вы не платите премиальные тарифы, чтобы определить, какую другую премиальную модель использовать. Измеряйте это тоже; плохой роутер тихо превращает сбережения в неправильные маршруты.

Когда кто-то запрашивает реализацию сортировки пузырьком, роутер распознаёт это как задачу на код и передаёт её вашему специалисту по коду. Запрос творческого письма? Это отправляется модели, которую вы выбрали для работы с голосом и диапазоном. Фактический вопрос о исторических событиях? Направьте его к общему агенту, желательно с извлечением, когда важна свежесть или цитирование.

## Практические преимущества

**Эффективность по стоимости важнее, чем вы думаете.** Модель для маршрутизации, принимающая решения о делегировании, стоит доля стоимости выполнения каждого запроса через ваш поставщика с самой высокой стоимостью. Со временем, особенно в масштабе, это превращается в реальные деньги. Вы платите за тяжёлый интеллект только тогда, когда действительно его нужно.

**Качество улучшается, когда вы сопоставляете модели с задачами.** Победитель меняется каждый месяц, задача и форма запроса. Именно поэтому слой маршрутизации должен зависеть от ваших оценок, а не от той модели, которая выиграла в Твиттере в неделю, когда вы писали интеграцию.

**Устойчивость становится побочным эффектом.** Когда у OpenAI происходит одна из периодических сбоев (и они происходят), ваш роутер может перенаправлять трафик к другим поставщикам. Вы не застреваете, ожидая возвращения конкретного API.

Это не про то, чтобы быть умным ради этого. Это про создание систем, которые имеют смысл как с финансовой, так и с технической точки зрения. Вы бы не использовали один и тот же молоток для каждой строительной задачи, и, вероятно, не должны использовать одну и ту же модель языка для каждой задачи ИИ.

Прелесть этого подхода в том, что ваш код приложения не меняется. Вы всё ещё вызываете свой роутер-агент. Сложность принятия решения о том, какую модель использовать для какой задачи, живёт в одном месте, настроена один раз, а не разбросана по вашему коду в виде условной логики.
