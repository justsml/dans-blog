Большинство инженерных команд выбирают одну языковую модель и придерживаются её. Один провайдер, одна модель, все задачи. Это как нанять одного человека для кода, копирайтинга и налогов только потому, что он хорошо прошёл первое собеседование.

В любой момент времени одна модель лучше справляется с кодом, другая — с длинным запутанным контекстом, а третья — самый дешёвый надёжный работяга для классификации. Названия меняются. Форма проблемы — нет. Относиться к одной модели как к универсальному эксперту — значит либо переплачивать за простые задачи, либо получать посредственные результаты на специализированных.

Я видел, как команда сжигала тысячи долларов, прогоняя анализ тональности через модель за 30 долларов за миллион токенов, когда модель за 0,50 доллара справилась бы не хуже. Простое форматирование JSON, базовые задачи классификации — всё шло через премиум-провайдера. Единственное, что нагревалось, — их счёт от AWS.

Есть способ лучше, и он не особенно сложен.

## Делегирование вместо преданности

Что, если бы вы могли направлять запросы к модели, которая действительно лучше всего подходит для этой конкретной задачи? Используйте дорогую мощную модель для сложных вещей, но спускайте простой парсинг и форматирование на что-то подешевле. Получайте преимущества нескольких провайдеров без необходимости вручную переключать их в кодовой базе.

Mastra позволяет построить именно такую систему. Вы настраиваете агентов-специалистов для разных типов работы, а затем создаёте агента-маршрутизатора, который определяет, какой специалист должен обработать каждый запрос. Идентификаторы моделей ниже — это примеры, а не рейтинг. Замените их на текущие модели, которые побеждают в ваших тестах и вписываются в бюджет.

Представьте это так: у вас в команде три специалиста.

```typescript
// ./src/mastra/index.ts
import { Mastra } from '@mastra/core';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { anthropic } from '@ai-sdk/anthropic';
import { google } from '@ai-sdk/google';

export const claudeAgent = new Agent({
  id: 'claude-agent',
  instructions: 'You are an expert engineer. Write bugs? You are fired.',
  model: anthropic(process.env.CODE_MODEL ?? 'claude-sonnet-4-5'),
});

export const geminiAgent = new Agent({
  id: 'gemini-agent',
  instructions: 'You are a creative writer. Be weird.',
  model: google(process.env.LONG_CONTEXT_MODEL ?? 'gemini-3-pro-preview'),
});

export const gptAgent = new Agent({
  id: 'gpt-agent',
  instructions: 'You are a helpful assistant. Be boring.',
  model: openai(process.env.GENERAL_MODEL ?? 'gpt-5.2'),
});
```

У каждого своя работа. Ваш агент для кода должен быть моделью, которая проходит ваши репозиторные тесты кода. Ваш агент для длинного контекста должен быть тем, кто выживает в ваших реальных документах, не превращая середину в кашу. Ваш общий агент должен быть дешёвым, надёжным и скучным в лучшем смысле этого слова.

Вот где становится интересно. Вы добавляете маршрутизатор, который действует как интеллектуальный прокси:

```typescript
export const routerAgent = new Agent({
  id: 'router-agent',
  name: 'The Boss',
  instructions: `You are an intelligent router.
  - Coding -> Claude
  - Poetry -> Gemini
  - Facts -> GPT

  Do not do the work yourself. Delegate.`,
  model: openai(process.env.ROUTER_MODEL ?? 'gpt-5-mini'), // Use a cheap model for routing!
  agents: {
    claudeAgent,
    geminiAgent,
    gptAgent,
  },
});

export const mastra = new Mastra({
  agents: { routerAgent, claudeAgent, geminiAgent, gptAgent },
});
```

Сам маршрутизатор работает на лёгкой модели — он всего лишь решает, куда направить запрос. Нет смысла платить премиум за то, чтобы понять, на какую премиум-модель направить трафик. Замеряйте это тоже: плохой маршрутизатор незаметно превращает экономию в неверные маршруты.

Когда кто-то просит реализовать пузырьковую сортировку, маршрутизатор распознаёт это как код и передаёт вашему специалисту по коду. Запрос на креативное письмо? Он уходит на модель, которую вы выбрали за голос и диапазон. Фактологический вопрос об исторических событиях? Маршрутизируйте его на общего агента, в идеале с поиском, когда важны свежесть или источники.

## Практические преимущества

**Экономическая эффективность важнее, чем кажется.** Маленькая модель маршрутизации, принимающая решения о делегировании, стоит долю от того, чтобы прогонять каждый запрос через самого дорогого провайдера. Со временем, особенно в масштабе, это превращается в реальные деньги. Вы платите за тяжёлую интеллектуальную работу только тогда, когда она действительно нужна.

**Качество улучшается, когда модели соответствуют задачам.** Победитель меняется каждый месяц, в зависимости от задачи и формы запроса. Именно поэтому слой маршрутизации должен опираться на ваши оценки (evals), а не на модель, которая побеждала в твиттере на неделе, когда вы писали интеграцию.

**Устойчивость становится побочным эффектом.** Когда у OpenAI случается один из периодических сбоев (а они случаются), ваш роутер может перенаправить трафик другим провайдерам. Вы не останетесь на мели в ожидании, пока один конкретный API вернётся в строй.

Это не про то, чтобы быть умным ради самого факта. Это про построение систем, которые имеют смысл и с финансовой, и с технической точки зрения. Вы ведь не будете использовать один и тот же молоток для всех строительных задач — и, скорее всего, не стоит использовать одну и ту же языковую модель для всех AI-задач.

Прелесть этого подхода в том, что ваш прикладной код не меняется. Вы всё так же просто вызываете своего агента-маршрутизатора. Вся сложность решения, какую модель использовать для какой задачи, живёт в одном месте, настраивается один раз, а не разбросана по всей кодовой базе в виде кучи условной логики.
