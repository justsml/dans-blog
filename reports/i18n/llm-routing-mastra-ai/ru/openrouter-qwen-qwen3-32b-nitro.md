## Делегирование вместо преданности

Что, если вы можете направлять запросы к модели, которая действительно лучше всего подходит для конкретной задачи? Используйте свою дорогую мощную модель для сложных задач, но опускайте простую обработку и форматирование до чего-то дешёвого. Получайте преимущества нескольких поставщиков без необходимости вручную переключаться между ними в своём коде.

Mastra позволяет создавать именно такой тип системы. Вы настраиваете специалистов для разных видов задач, а затем создаете маршрутизатор, который определяет, какой специалист должен обрабатывать каждый запрос. Ниже приведены примеры идентификаторов моделей, это не рейтинг. Замените их на текущие модели, которые выигрывают ваши оценки и соответствуют вашему бюджету.

Представьте так: у вас три специалиста в команде.

```typescript
// ./src/mastra/index.ts
import { Mastra } from '@mastra/core';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { anthropic } from '@ai-sdk/anthropic';
import { google } from '@ai-sdk/google';

export const claudeAgent = new Agent({
  id: 'claude-agent',
  instructions: 'Вы — эксперт-инженер. Пишете баги? Вас уволят.',
  model: anthropic(process.env.CODE_MODEL ?? 'claude-sonnet-4-5'),
});

export const geminiAgent = new Agent({
  id: 'gemini-agent',
  instructions: 'Вы — творческий писатель. Будьте странноваты.',
  model: google(process.env.LONG_CONTEXT_MODEL ?? 'gemini-3-pro-preview'),
});

export const gptAgent = new Agent({
  id: 'gpt-agent',
  instructions: 'Вы — полезный ассистент. Будьте скучны.',
  model: openai(process.env.GENERAL_MODEL ?? 'gpt-5.2'),
});
```

Каждый из них выполняет свою работу. Ваш агент для кода должен быть моделью, которая проходит ваши репозиторно-специфические тесты на кодирование. Ваш агент с длинным контекстом должен выдерживать ваши реальные документы, не превращая средину в кашу. Ваш общий агент должен быть дешевым, надежным и скучным в самом лучшем смысле.

Вот где становится интересно. Вы добавляете маршрутизатор, который действует как интеллектуальный прокси:

```typescript
export const routerAgent = new Agent({
  id: 'router-agent',
  name: 'Босс',
  instructions: `Вы — интеллектуальный маршрутизатор.
  - Кодирование -> Claude
  - Поэзия -> Gemini
  - Факты -> GPT

  Не делайте работу сами. Делегируйте.`,
  model: openai(process.env.ROUTER_MODEL ?? 'gpt-5-mini'), // Используйте дешевую модель для маршрутизации!
  agents: {
    claudeAgent,
    geminiAgent,
    gptAgent,
  },
});

export const mastra = new Mastra({
  agents: { routerAgent, claudeAgent, geminiAgent, gptAgent },
});
```

Сам маршрутизатор работает на легковесной модели, потому что он просто принимает решения о том, куда отправлять трафик. Вы не платите премиальные ставки, чтобы определить, какую другую премиальную модель использовать. Измеряйте это тоже; плохой маршрутизатор тихо превращает сбережения в ошибочные маршруты.

Когда кто-то запрашивает реализацию пузырьковой сортировки, маршрутизатор распознает это как задачу на кодирование и передает ее вашему специалисту по коду. Творческий запрос? Он отправляется к модели, которую вы выбрали для голоса и диапазона. Фактический вопрос о исторических событиях? Маршрутизируйте его к общему агенту, желательно с извлечением, когда важна свежесть или цитирование.

Практические преимущества

**Экономическая эффективность важнее, чем вы думаете.** Модель маршрутизации, принимающая решения о делегировании, стоит дробной части от стоимости обработки каждого запроса через самого дорогого провайдера. Со временем, особенно при масштабе, это накапливается в реальные деньги. Вы платите за тяжелую интеллектуальную мощность только тогда, когда действительно нуждаетесь в ней.

**Качество улучшается, когда вы сопоставляете модели с задачами.** Лучший выбор меняется в зависимости от месяца, задачи и формы запроса. Именно поэтому маршрутизационный слой должен зависеть от ваших оценок, а не от того, какая модель была популярна в Твиттере на момент написания интеграции.

**Устойчивость становится побочным преимуществом.** Когда OpenAI сталкивается с очередным периодическим сбоями (а они случаются), ваш маршрутизатор может перенаправлять трафик к другим провайдерам. Вы не остаетесь в бездействии, ожидая возвращения конкретного API.

Это не про умничание ради самого процесса. Это про создание систем, которые логичны как с финансовой, так и с технической точки зрения. Вы бы не использовали один и тот же молоток для всех задач по строительству, и, вероятно, не стоит использовать одну и ту же языковую модель для всех задач с ИИ.

Прелесть этого подхода в том, что ваш код приложения не меняется. Вы все равно просто вызываете агента-маршрутизатора. Сложность выбора модели для каждой задачи сосредоточена в одном месте, настроена один раз, вместо того чтобы быть разбросанной по всей кодовой базе в виде условной логики.
