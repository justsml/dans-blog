Большинство инженерных команд выбирают одну языковую модель и остаются с ней. Один провайдер, одна модель, все задачи. Это как нанять одного человека, чтобы он писал код, занимался копирайтингом и сдавал налоги только потому, что он неплохо показал себя на первом собеседовании.

В любой момент одна модель лучше справляется с кодом, другая — с длинным и неструктурированным контекстом, а третья является самым дешёвым и предсказуемым рабочим конём для классификации. Названия меняются. Форма задачи остаётся прежней. Если относиться к одной модели как к универсальному решению, вы либо переплачиваете за простые задачи, либо получаете посредственный результат на специализированных.

Я видел, как команда сожгла тысячи долларов, прогоняя анализ тональности через модель за $30 за миллион токенов, хотя модель за $0,50 справилась бы с задачей не хуже. Простое форматирование JSON, базовая классификация — всё шло через их премиум-провайдера. Единственное, что действительно «разогревалось», так это счёт от AWS.

Есть способ лучше, и он не особо сложен.

## Делегирование вместо преданности

Что, если вы сможете маршрутизировать запросы к модели, которая действительно лучше всего подходит для конкретной задачи? Использовать свою дорогую мощную модель для сложных задач, а простые парсинг и форматирование перепоручать чему-то более дешёвому. Получать выгоду от работы с несколькими провайдерами, не занимаясь ручным переключением между ними в кодовой базе.

Mastra позволяет собрать именно такую систему. Вы настраиваете агентов-специалистов для разных типов задач, а затем создаёте роутер-агент, который определяет, какой специалист должен обработать каждый запрос. Приведённые ниже идентификаторы моделей — примеры, а не рейтинг. Замените их на актуальные модели, которые показывают лучшие результаты на ваших тестах и вписываются в бюджет.

Представьте это так: в вашей команде трое специалистов.

```typescript
// ./src/mastra/index.ts
import { Mastra } from '@mastra/core';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { anthropic } from '@ai-sdk/anthropic';
import { google } from '@ai-sdk/google';

export const claudeAgent = new Agent({
  id: 'claude-agent',
  instructions: 'You are an expert engineer. Write bugs? You are fired.',
  model: anthropic(process.env.CODE_MODEL ?? 'claude-sonnet-4-5'),
});

export const geminiAgent = new Agent({
  id: 'gemini-agent',
  instructions: 'You are a creative writer. Be weird.',
  model: google(process.env.LONG_CONTEXT_MODEL ?? 'gemini-3-pro-preview'),
});

export const gptAgent = new Agent({
  id: 'gpt-agent',
  instructions: 'You are a helpful assistant. Be boring.',
  model: openai(process.env.GENERAL_MODEL ?? 'gpt-5.2'),
});
```

У каждого своя задача. Ваш код-агент должен быть той моделью, которая проходит ваши специфичные для репозитория тесты на программирование. Ваш агент для длинного контекста — той, которая выдерживает ваши реальные документы, не превращая середину текста в кашу. Ваш универсальный агент должен быть дешёвым, надёжным и максимально скучным — в хорошем смысле.

А вот здесь начинается самое интересное. Вы добавляете роутер, который работает как интеллектуальный прокси:

```typescript
export const routerAgent = new Agent({
  id: 'router-agent',
  name: 'The Boss',
  instructions: `You are an intelligent router.
  - Coding -> Claude
  - Poetry -> Gemini
  - Facts -> GPT

  Do not do the work yourself. Delegate.`,
  model: openai(process.env.ROUTER_MODEL ?? 'gpt-5-mini'), // Use a cheap model for routing!
  agents: {
    claudeAgent,
    geminiAgent,
    gptAgent,
  },
});

export const mastra = new Mastra({
  agents: { routerAgent, claudeAgent, geminiAgent, gptAgent },
});
```

Роутер сам по себе работает на лёгкой модели, потому что его задача — лишь принимать решения о том, куда направлять трафик. Вы не платите премиальные ставки, чтобы понять, какую ещё премиальную модель использовать. Это тоже стоит замерять: плохой роутер незаметно превращает экономию в перенаправления не туда.

Когда просят реализовать пузырьковую сортировку, роутер определяет это как кодовую задачу и передаёт её вашему специалисту по коду. Промпт для творческого текста? Отправляем модель, выбранную за стиль и диапазон. Фактологический вопрос по истории? Маршрутизируем на универсального агента, желательно с подключением retrieval, когда важны актуальность или точные ссылки.

## Практические преимущества

**Экономическая эффективность важнее, чем кажется.** Лёгкая модель роутера, принимающая решения о делегировании, обходится в доли стоимости прогонки каждого запроса через вашего самого дорогого провайдера. Со временем, особенно на масштабе, это выливается в реальные деньги. Вы платите за тяжёлую артиллерию только тогда, когда она действительно нужна.

**Качество растёт, когда модели соответствуют задачам.** Лидер меняется от месяца к месяцу, в зависимости от задачи и формы промпта. Именно поэтому слой роутинга должен опираться на ваши evals, а не на ту модель, которая захватывала Twitter на неделе написания интеграции.

**Отказоустойчивость становится приятным бонусом.** Когда у OpenAI случается очередной плановый сбой (а они случаются), ваш роутер может перенаправить трафик на других провайдеров. Вы не зависаете в ожидании, пока один конкретный API снова не поднимется.

Речь не о том, чтобы быть умнее ради самого процесса. Речь о построении систем, которые имеют смысл и финансово, и технически. Вы же не берёте один и тот же молоток для каждой строительной задачи, так что, вероятно, не стоит использовать одну и ту же языковую модель для каждой AI-задачи.

Красота этого подхода в том, что код приложения не меняется. Вы по-прежнему просто вызываете агента-роутер. Сложность выбора модели под конкретную задачу сосредоточена в одном месте, настроена единожды, а не размазана по кодовой базе кучей условных конструкций.
