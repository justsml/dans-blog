## Делегирование вместо преданности

А что если направлять запросы к модели, которая действительно лучше всего подходит для конкретной задачи? Использовать дорогой «мощный» вариант для сложных задач, а простое парсинг и форматирование отдавать более дешёвому. Получить преимущества нескольких провайдеров без необходимости вручную переключать их в кодовой базе.

Mastra позволяет построить именно такую систему. Вы задаёте специализированные агенты для разных типов работы, а затем создаёте роутер‑агент, который решает, какому специалисту передать каждый запрос. Идентификаторы моделей ниже приведены лишь как пример, а не как рейтинг. Подмените их актуальными моделями, которые показывают лучшие результаты в ваших оценках и укладываются в бюджет.

Представьте это так: у вас есть три специалиста в команде.

```typescript
// ./src/mastra/index.ts
import { Mastra } from '@mastra/core';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { anthropic } from '@ai-sdk/anthropic';
import { google } from '@ai-sdk/google';

export const claudeAgent = new Agent({
  id: 'claude-agent',
  instructions: 'You are an expert engineer. Write bugs? You are fired.',
  model: anthropic(process.env.CODE_MODEL ?? 'claude-sonnet-4-5'),
});

export const geminiAgent = new Agent({
  id: 'gemini-agent',
  instructions: 'You are a creative writer. Be weird.',
  model: google(process.env.LONG_CONTEXT_MODEL ?? 'gemini-3-pro-preview'),
});

export const gptAgent = new Agent({
  id: 'gpt-agent',
  instructions: 'You are a helpful assistant. Be boring.',
  model: openai(process.env.GENERAL_MODEL ?? 'gpt-5.2'),
});
```

У каждого своя роль. Агент кода должен быть моделью, проходящей ваши специфические оценки по репозиторию. Агент с длинным контекстом — тем, кто справляется с вашими реальными документами, не превращая середину в «суп». Общий агент должен быть дешёвым, надёжным и «скучным» в лучшем смысле.

И вот где начинается интересное. Вы добавляете роутер, который выступает в роли интеллектуального прокси:

```typescript
export const routerAgent = new Agent({
  id: 'router-agent',
  name: 'The Boss',
  instructions: `You are an intelligent router.
  - Coding -> Claude
  - Poetry -> Gemini
  - Facts -> GPT

  Do not do the work yourself. Delegate.`,
  model: openai(process.env.ROUTER_MODEL ?? 'gpt-5-mini'), // Use a cheap model for routing!
  agents: {
    claudeAgent,
    geminiAgent,
    gptAgent,
  },
});

export const mastra = new Mastra({
  agents: { routerAgent, claudeAgent, geminiAgent, gptAgent },
});
```

Сам роутер работает на лёгкой модели, потому что его задача — лишь решить, куда направить запрос. Вы не платите премиум‑тарифы за то, чтобы определить, какую из премиум‑моделей задействовать. Это тоже стоит измерять: плохой роутер тихо превращает экономию в неправильные маршруты.

Когда кто‑то запрашивает реализацию пузырьковой сортировки, роутер распознаёт задачу как код и передаёт её вашему специалисту по коду. Творческий запрос — отправляется к модели, выбранной за её стиль и диапазон. Фактический вопрос о исторических событиях — направляется к общему агенту, желательно с извлечением, когда важны актуальность или ссылки.

## Практические преимущества

**Экономия расходов важнее, чем кажется.** Маленькая модель‑роутер, принимающая решения о делегировании, стоит лишь доли от стоимости обработки каждого запроса через самую дорогую модель. При масштабировании это приводит к реальной экономии. Вы платите за тяжёлый интеллект только тогда, когда он действительно нужен.

**Качество повышается, когда модели соответствуют задачам.** Победитель меняется от месяца к месяцу, от задачи к задаче и от формы промпта. Поэтому слой роутинга должен опираться на ваши оценки, а не на то, какая модель «трендит» в Твиттере в ту неделю, когда вы писали интеграцию.

**Устойчивость становится побочным эффектом.** Когда у OpenAI происходит очередной сбой (а такие бывают), ваш роутер может перенаправить трафик к другим провайдерам. Вы не останетесь в бездне, ожидая восстановления конкретного API.

Это не попытка быть умным ради умения. Это построение систем, которые логичны и с финансовой, и с технической точек зрения. Вы бы не использовали один и тот же молоток для всех строительных работ, так же и не стоит использовать одну и ту же языковую модель для всех AI‑задач.

Преимущество подхода в том, что ваш прикладной код не меняется. Вы всё так же вызываете роутер‑агент. Сложность выбора модели для конкретной задачи сосредоточена в одном месте, настроена один раз, а не разбросана по всему кодовому базе в виде множества условных операторов.
