La maggior parte delle squadre di ingegneria sceglie un modello linguistico e ci resta fedele. Un unico fornitore, un unico modello, tutti i compiti. È come assumere una persona per occuparsi del tuo codice, della tua scrittura e delle tue tasse solo perché si è dimostrata brava in un primo colloquio.

In ogni momento, un modello è migliore per il codice, un altro gestisce meglio i contesti lunghi e disordinati, e un terzo è il lavoro noioso e economico per la classificazione. I nomi cambiano. La forma del problema no. Trattare un modello come se eccellessi in tutto significa che stai pagando troppo per compiti semplici o ottenendo risultati insufficienti per compiti specializzati.

Ho visto una squadra spendere migliaia di dollari per eseguire l'analisi del sentimento tramite un modello che costa $30 per un milione di token, quando un modello da $0,50 avrebbe svolto lo stesso lavoro. Formattazione JSON semplice, compiti di classificazione di base, tutto passava attraverso il loro fornitore premium. L'unica cosa che si riscaldava era la loro fattura AWS.

C'è un modo migliore, e non è particolarmente complicato.

## Delegazione al posto della fedeltà

E se potessi instradare le richieste al modello effettivamente più adatto per quel compito specifico? Usa il tuo modello potente e costoso per le cose difficili, ma abbassa i compiti semplici di parsing e formattazione a qualcosa di più economico. Ottieni i vantaggi di più fornitori senza doverli gestire manualmente nel tuo codice.

Mastra ti permette di costruire esattamente questo tipo di sistema. Configuri agenti specializzati per diversi tipi di lavoro, poi crei un agente di instradamento che determina quale specialista dovrebbe gestire ogni richiesta. Gli ID dei modelli qui sotto sono esempi, non una classifica. Sostituirli con i modelli correnti che vincono le tue valutazioni e si adattano al tuo budget.

Pensaci così: hai tre specializzati nel tuo team.

```typescript
// ./src/mastra/index.ts
import { Mastra } from '@mastra/core';
import { Agent } from '@mastra/core/agent';
import { openai } from '@ai-sdk/openai';
import { anthropic } from '@ai-sdk/anthropic';
import { google } from '@ai-sdk/google';

export const claudeAgent = new Agent({
  id: 'claude-agent',
  instructions: 'Sei un ingegnere esperto. Scrivi bug? Sei licenziato.',
  model: anthropic(process.env.CODE_MODEL ?? 'claude-sonnet-4-5'),
});

export const geminiAgent = new Agent({
  id: 'gemini-agent',
  instructions: 'Sei uno scrittore creativo. Sii strano.',
  model: google(process.env.LONG_CONTEXT_MODEL ?? 'gemini-3-pro-preview'),
});

export const gptAgent = new Agent({
  id: 'gpt-agent',
  instructions: 'Sei un assistente utile. Sii noioso.',
  model: openai(process.env.GENERAL_MODEL ?? 'gpt-5.2'),
});
```

Ogni agente ha un compito specifico. Il tuo agente per il codice dovrebbe essere il modello che supera le tue valutazioni di programmazione specifiche per il repository. L'agente per il contesto lungo dovrebbe essere quello che gestisce i tuoi documenti reali senza trasformarne il centro in una zuppa. L'agente generale dovrebbe essere economico, affidabile e noioso nel miglior modo possibile.

Ecco dove diventa interessante. Aggiungi un router che agisce come un proxy intelligente:

```typescript
export const routerAgent = new Agent({
  id: 'router-agent',
  name: 'The Boss',
  instructions: `Sei un router intelligente.
  - Codifica -> Claude
  - Poesia -> Gemini
  - Fatti -> GPT

  Non fare il lavoro da solo. Delega.`,
  model: openai(process.env.ROUTER_MODEL ?? 'gpt-5-mini'), // Usa un modello economico per l'instradamento!
  agents: {
    claudeAgent,
    geminiAgent,
    gptAgent,
  },
});

export const mastra = new Mastra({
  agents: { routerAgent, claudeAgent, geminiAgent, gptAgent },
});
```

Il router stesso funziona su un modello leggero perché si limita a prendere decisioni su dove inviare il traffico. Non stai pagando tariffe premium per decidere quale altro modello premium utilizzare. Misura anche questo: un router mal configurato trasforma silenziosamente i risparmi in instradamenti errati.

Quando qualcuno chiede un'implementazione del bubble sort, il router riconosce che si tratta di un lavoro di codifica e lo passa al tuo specialista del codice. Prompt per scrittura creativa? Vanno al modello che hai scelto per voce e gamma. Domanda fattuale su eventi storici? Instradalola all'agente generale, idealmente con recupero quando freschezza o citazione sono rilevanti.

## I benefici pratici

**L'efficienza economica è più importante di quanto si pensi.** Un modello di routing leggero che prende decisioni su delega costa una frazione rispetto all'invio di ogni richiesta al tuo provider più costoso. Nel tempo, specialmente su larga scala, questo si traduce in risparmi reali. Paghi l'intelligenza pesante solo quando ne hai realmente bisogno.

**La qualità migliora quando abbiniamo modelli ai compiti.** Il "vincitore" cambia ogni mese, per tipo di task e forma del prompt. Questo è il motivo per cui lo strato di routing deve basarsi sulle tue valutazioni, non su quale modello stia vincendo su Twitter la settimana in cui hai scritto l'integrazione.

**La resilienza diventa un vantaggio collaterale.** Quando OpenAI ha una delle sue interruzioni periodiche (e lo fa spesso), il tuo router può deviare il traffico verso altri provider. Non sei in panne ad aspettare che un'API specifica torni online.

Questo non è una questione di ingegnosità fine a sé stessa. Si tratta di costruire sistemi che siano sensati sia dal punto di vista finanziario che tecnico. Non useresti lo stesso martello per ogni compito edile, e probabilmente non dovresti usare lo stesso modello linguistico per ogni task AI nemmeno.

La bellezza di questo approccio è che il codice dell'applicazione non cambia. Chiami comunque l'agente router. La complessità di decidere quale modello utilizzare per ogni task vive in un unico punto, configurato una volta, invece di essere sparsa in tutta la base codice con logica condizionale.
