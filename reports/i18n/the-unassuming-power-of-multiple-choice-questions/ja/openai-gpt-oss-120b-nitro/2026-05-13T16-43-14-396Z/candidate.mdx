I used to think multiple-choice questions were the cheap seats of teaching.

Good for checking whether someone remembered a term. Bad for finding out whether they could actually use the idea. Fine for standardized tests, less fine for the messy work of learning.

Then I started building programming quizzes.

The surprise was not that multiple-choice questions scale. Everyone knows that. The surprise was that a good answer set can teach the lesson before the explanation appears.

<p class="inset">複数選択問題で有用なのは正解ではなく、意図的に選んだ誤答です。</p>

## The Distractor Is The Curriculum

ほとんどの質の低い多肢選択問題は、正解が1つ、ほぼ正解が1つ、そして著者がコーヒーを待っている間に書き残したような2つの選択肢で構成されています。

それはフォーマットの無駄遣いです。

優れたクイズでは、各誤答が実際の誤解を表します：

- 他の言語から借用した構文
- 旧コードに今でも残るレガシーパターン
- ある条件下で破綻する妥当な思考モデル
- 質問中の1語が変われば正解になる選択肢

ここで学習が成立します。受験者は単に選択肢を選んでいるのではなく、どのモデルを使っているかを露呈しているのです。

## 1つの質問、複数の深さ

この PostgreSQL の質問を例に取ります。

> What is the (_SQL Standard_) syntax for an auto-incrementing IDENTITY column in PostgreSQL (v10+)?
> - `id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY` ✅ (Correct)
> - `id SERIAL PRIMARY KEY`  (Legacy syntax)
> - `id INT IDENTITY(1,1)`  (SQL Server syntax)
> - `id INT AUTO_INCREMENT`  (MySQL syntax)
> - `id NUMBER GENERATED ALWAYS AS IDENTITY`  (Oracle syntax)

この質問は、受講者のスキルレベルに応じて複数の教育的目的を果たします。

- **初心者** は現在の構文とレガシー構文を区別することに集中します  
- **中級者** は類似した構文間の微妙な違いを見極めます  
- **上級者** は他プラットフォームの知識を活用しつつ、バージョン固有の実装について正確に答える必要があります  

単に構文を問うだけでなく、受験者のメンタルモデルを分類しています。

`SERIAL` を選んだ人は、古い例から Postgres を学んだ可能性が高いです。`IDENTITY(1,1)` を選んだ人は、SQL Server の筋肉記憶が残っています。`AUTO_INCREMENT` を選んだ人は概念は理解しているものの、方言（dialect）を誤っています。これらはそれぞれ異なる学習機会となります。

もし説明が「A が正解」とだけ書かれているなら、質問は最も価値のあるデータを捨ててしまっている。

## クイズ作成で学んだこと

クイズを作ることで、読者に何を学んでほしいかをより正確に定義せざるを得なくなった。

記事は滑らかな文章の中にあいまいな説明を隠すことができる。クイズはそれができない。誤答は概念に実際の境界があるかどうかを露呈する。

現在、プログラミング問題を書くときは次の点を自問する：

- 半分だけ理解している人が賢く選ぶのは何か？
- 隣接するエコシステムでは正解は何になるか？
- 一般的な誤解を維持する選択肢は何か？
- 正解を正しくするごく小さな条件は何か？

それらの質問は指導をより鋭くします。また、クイズが説明そのもののテストになるため、記事自体も鋭くなります。4つの信頼できる選択肢を書けないなら、まだ本質的な概念を見つけていない可能性が高いです。

## フィードバックループが目的

多肢選択問題はスケールで有用です。採点は自動化され、パターンが見えてきます。どの質問が簡すぎるか、曖昧すぎるか、あるいは偶然にトリビアをテストしているかが分かります。

しかし、より良いフィードバックループは学習者がクリックする前に起こります。

著者は学習者が陥りやすいミスをモデル化しなければなりません。つまり、学習者を尊重する必要があります。怠惰な誤答は「埋め合わせが必要だった」と言っているだけです。強力な誤答は「この考え方がどこで横道にそれるか正確に分かっている」と示しています。

だからこそ、このサイトで MCQ を使い続けています。効率的だからというだけでなく、文章だけでは回避できる指導上の正直さを強制してくれるからです。

次に問題を作るときは、正解から始めないでください。誤解から着手しましょう。正解は、はるかに適切な位置に置くことができます。
