我曾以为选择题是教学中的廉价座位。

适合检查某人是否记住了术语。不适合了解他们是否真的能运用概念。适合标准化考试，不太适合学习这种混乱的工作。

后来我开始编写编程测验。

令人惊讶的不是选择题可以规模化。这谁都知道。令人惊讶的是，一组好的选项可以在解释出现之前就教会课程。

<p class="inset">选择题中有用的部分往往不是正确答案。而是你故意选中的错误答案。</p>

## 干扰项即课程

大多数糟糕的选择题有一个正确答案，一个几乎正确的答案，以及两个明显是作者等咖啡时写出来的选项。

这浪费了这种形式。

在好的测验中，每个错误答案都代表一种真正的误解：

- 从另一种语言借用的语法
- 仍然出现在旧代码中的遗留模式
- 在某种条件下失效的看似合理的心理模型
- 如果问题中某个词改变就会正确的答案

这就是教学发生的地方。学习者并非仅仅在挑选选项，而是在揭示他们正在使用哪种模型。

## 一个问题，多种深度

以这个 PostgreSQL 问题为例：

> 在 PostgreSQL（v10+）中，用于自增 IDENTITY 列的（_SQL 标准_）语法是什么？
> - `id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY` ✅（正确）
> - `id SERIAL PRIMARY KEY`（旧语法）
> - `id INT IDENTITY(1,1)`（SQL Server 语法）
> - `id INT AUTO_INCREMENT`（MySQL 语法）
> - `id NUMBER GENERATED ALWAYS AS IDENTITY`（Oracle 语法）

这个问题在不同专业水平上服务于多种教学目的：

- **初学者**关注基本的 PostgreSQL 语法，主要区分当前语法和旧语法。
- **中级学习者**需要辨别相似语法之间的细微差别。
- **高级从业者**必须利用跨平台知识，同时保持对特定版本实现的精确性。

这个问题所做的不仅仅是询问语法。它是在对心理模型进行分类。

选择 `SERIAL` 的人可能是从旧示例中学的 Postgres。选择 `IDENTITY(1,1)` 的人带有 SQL Server 的肌肉记忆。选择 `AUTO_INCREMENT` 的人知道概念但不熟悉方言。这些都是不同的教学时刻。

如果解释只说“A 是正确的”，那么这个问题就浪费了它最好的数据。

## 编写测验教会我的事

编写测验迫使我更精确地思考我真正希望读者学到什么。

一篇文章可以用流畅的散文掩盖模糊的解释。但测验不能。错误的答案会暴露概念是否具有真正的边界。

现在当我编写编程问题时，我会问：

- 一个半懂不懂的聪明人会选择什么？
- 在相邻的生态系统中，哪个答案会是正确的？
- 哪个答案保留了常见的误解？
- 是什么微小的条件使得正确答案成为正确答案？

这些问题让教学更锐利。它们也让文章更锐利，因为测验变成了对解释本身的测试。如果我写不出四个令人信服的选项，我可能还没有找到真正的概念。

## 反馈循环才是关键

多选题在大规模使用时确实有用。评分是自动的。模式变得可见。你可以看到哪些问题太简单、太模糊，或者意外地测试了琐碎知识。

但更好的反馈循环发生在学习者点击之前。

作者必须模拟学习者可能犯的错误。这意味着作者必须尊重学习者。一个敷衍的错误答案说：“我需要凑数。”一个强有力的错误答案说：“我确切知道这个想法会在哪里跑偏。”

这就是为什么我一直在本站使用多选题。不是因为它们高效——尽管它们确实高效。而是因为它们迫使一种散文本身可以回避的教学诚实。

下次你写多选题时，不要从正确答案开始。从误解开始。正确答案会有一个更好的立足之地。
