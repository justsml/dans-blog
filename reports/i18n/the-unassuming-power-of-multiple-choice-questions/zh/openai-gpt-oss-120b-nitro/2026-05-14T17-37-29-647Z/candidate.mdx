我曾经认为选择题是教学中的廉价座位。

适合检查某人是否记得一个术语。却不适合判断他们是否真的能使用这个概念。标准化考试还能接受，但对混乱的学习过程就不那么合适了。

后来我开始制作编程测验。

让人惊讶的并不是选择题可以大规模使用——大家都知道这点。惊讶的是，一个好的答案集合可以在解释出现之前就传授课程。

<p class="inset">选择题的有用之处往往不是正确答案，而是你刻意设置的错误答案。</p>

## 干扰项即课程

大多数糟糕的选择题只有一个正确答案、一个几乎正确的答案，以及两个显然是作者等咖啡时随手写出的选项。

这浪费了这种形式的价值。

在一个好的测验中，每个错误答案都代表一种真实的误解：

- 来自另一种语言的语法
- 仍然出现在旧代码中的遗留模式
- 在特定条件下会失效的合理思维模型
- 如果问题中的一个词改动，答案就会变成正确的

教学正发生在这里。学习者不仅仅是选一个选项，而是在透露他们使用的模型。

## 一个问题，多层深度

以这个 PostgreSQL 问题为例：

> What is the (_SQL Standard_) syntax for an auto-incrementing IDENTITY column in PostgreSQL (v10+)?
> - `id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY` ✅ (Correct)
> - `id SERIAL PRIMARY KEY`  (Legacy syntax)
> - `id INT IDENTITY(1,1)`  (SQL Server syntax)
> - `id INT AUTO_INCREMENT`  (MySQL syntax)
> - `id NUMBER GENERATED ALWAYS AS IDENTITY`  (Oracle syntax)

该问题在不同专业水平上实现了多重教学目标：

- **初学者** 关注基本的 PostgreSQL 语法，主要区分当前与遗留的写法  
- **中级学习者** 探索相似语法之间的细微差别  
- **高级实践者** 必须利用跨平台知识，同时对版本特定实现保持精准

这个问题做的不仅仅是询问语法，它在对思维模型进行分类。

选择 `SERIAL` 的人可能是从旧示例中学习到 Postgres 的。选择 `IDENTITY(1,1)` 的人带着 SQL Server 的记忆。选择 `AUTO_INCREMENT` 的人懂概念但不熟悉方言。这些都是不同的教学时机。

如果解释只说“ A 是正确的”，那么问题就丢失了最有价值的数据。

## 构建测验给我的启示

编写测验迫使我对读者真正需要学到的内容更加精准。

文章可以把模糊的解释藏在流畅的文字里，而测验做不到。错误选项会暴露概念是否真的有边界。

当我现在撰写编程题目时，我会问：

- 如果一个人只懂一半，这个聪明的人会选什么？
- 在相邻的生态系统中，哪个答案是正确的？
- 哪个答案保留了常见的误解？
- 哪个细微条件让正确答案成立？

这些问题让教学更锋利，也让文章更锋利，因为测验本身成了对解释的检验。如果我写不出四个可信的选项，说明我还没有真正抓住核心概念。

## 反馈循环才是关键

多项选择题在大规模使用时确实有用。批改自动化，模式一目了然。你可以看到哪些题目太容易、太模糊，或意外地在考察琐碎细节。

但更好的反馈循环发生在学习者点击之前。

作者必须模拟学习者可能的错误。这意味着作者必须尊重学习者。一个敷衍的错误答案只是在填充；一个有力度的错误答案则表明“我清楚地知道这个思路会怎么偏离”。

这就是我在本站持续使用 MCQ 的原因。不是因为它们效率高（虽然确实如此），而是因为它们强迫一种教学上的诚实，而单纯的文字很容易回避。

下次写题时，别先从正确答案开始。先从误解出发。正确答案自然会有更好的落脚点。
