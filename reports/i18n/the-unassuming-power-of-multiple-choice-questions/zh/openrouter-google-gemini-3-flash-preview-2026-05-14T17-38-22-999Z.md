我以前一直认为，多选题是教学里的“廉价座”。

它们适合检查某人是否记住了某个术语，但不适合发现他们是否真的能运用这个概念。对应付标准化考试还行，但对于繁杂的实际学习过程，就没那么好使了。

后来，我开始编写编程测验。

令我惊讶的不是多选题的规模化能力——这一点人尽皆知。令我惊讶的是，一套出色的选项设计竟然能在解释出现之前，就先把课给讲了。

<p class="inset">多选题最有用的部分往往不是正确答案，而是你刻意设计的那些错误选项。</p>

## 干扰项即课程

大多数糟糕的多选题都有一个正确答案、一个接近正确的答案，以及两个显然是作者在等咖啡时随手胡诌的选项。

这是对这种形式的浪费。

在一个好的测验中，每个错误答案都代表一种真实的误解：

- 从另一种语言借用的语法
- 仍出现在旧代码中的过时模式
- 在某种特定条件下会失效的、看似合理的心理模型
- 如果题目改动一个词就会变成正确的答案

这才是教学发生的地方。学习者不仅仅是在选择一个选项，他们是在揭示自己正在使用哪种模型。

## 一个问题，多重深度

以这个 PostgreSQL 问题为例：

> 在 PostgreSQL (v10+) 中，符合（SQL 标准）的自增 IDENTITY 列语法是什么？
> - `id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY` ✅ (正确)
> - `id SERIAL PRIMARY KEY`  (旧版语法)
> - `id INT IDENTITY(1,1)`  (SQL Server 语法)
> - `id INT AUTO_INCREMENT`  (MySQL 语法)
> - `id NUMBER GENERATED ALWAYS AS IDENTITY`  (Oracle 语法)

这个问题针对不同水平的学习者起到了多种教学作用：

- **初学者**关注基础的 PostgreSQL 语法，主要是区分当前方法和旧版方法。
- **中级学习者**需要辨析相似语法之间的细微差别。
- **高级从业者**必须利用他们的跨平台知识，同时对特定版本的实现保持精确。

这个问题不仅仅是在考语法，它在对心理模型进行分类。

选 `SERIAL` 的人可能通过旧示例学习的 Postgres。选 `IDENTITY(1,1)` 的人带着 SQL Server 的肌肉记忆。选 `AUTO_INCREMENT` 的人懂这个概念但不懂这种方言。这些都是不同的教学契机。

如果解析只是一句“A 是正确答案”，那这道题就白费了它最有价值的数据。

## 编写测验教会我的事

编写测验迫使我必须精确定义我到底想让读者学到什么。

文章可以用流畅的措辞掩盖模糊的解释，但测验不行。错误选项会直接暴露概念的边界是否清晰。

现在当我编写编程题目时，我会问自己：

- 如果一个聪明人只懂了一半，他会选哪个？
- 在相邻的生态系统中，哪个答案是正确的？
- 哪个选项符合常见的误解？
- 哪个微小的条件决定了正确答案的正确性？

这些问题让教学更精准，也让文章更有深度，因为测验本身成了对“解释”的压力测试。如果我凑不齐四个看起来都像那么回事的选项，说明我还没抓到概念的核心。

## 反馈循环才是核心

没错，选择题在大规模应用中确实好用。自动评分，模式清晰。你能一眼看出哪些题太简单、太含糊，或者不小心考到了冷门偏门。

但更有价值的反馈循环，在学习者点击选项之前就已经开始了。

作者必须模拟学习者可能犯的错。这意味着你得尊重学习者。一个敷衍的错误选项在说：“我只是想凑数”；而一个高水平的错误选项则在说：“我太清楚你会在哪儿掉进坑里了。”

这就是为什么我一直坚持在这个网站上使用选择题。不只是因为高效（虽然确实高效），更因为它们逼出了一种教学上的“诚实”——这种诚实是单纯写文章很容易逃避的。

下次你出题时，别先想正确答案。先从误解入手。这样，正确答案的立足点才会更稳固。
