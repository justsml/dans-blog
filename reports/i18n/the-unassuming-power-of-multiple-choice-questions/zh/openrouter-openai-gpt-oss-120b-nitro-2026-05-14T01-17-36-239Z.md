我过去认为选择题是教学中的廉价座位。

适合检查某人是否记得一个术语。却不擅长判断他们是否真的能运用这个概念。对标准化考试还行，但对学习的混沌工作就不那么合适了。

后来我开始制作编程测验。

让人意外的不是选择题可以大规模使用——大家都知道这点。意外的是，一个好的答案集合可以在解释出现之前就传授课程。

<p class="inset">选择题的有用之处往往不是正确答案，而是你刻意设置的错误答案。</p>

## 干扰项即课程

大多数糟糕的选择题只有一个正确答案、一个几乎正确的答案，以及两个显然是作者等咖啡时随手写出的选项。

这浪费了这种形式的价值。

在一个好的测验中，每个错误答案都代表一种真实的误解：

- 来自另一种语言的语法借用  
- 仍然出现在旧代码中的遗留模式  
- 在特定条件下失效的合理思维模型  
- 若问题中的一个词改变则会正确的答案

这正是教学发生的地方。学习者并非仅仅在挑选一个选项，而是在暴露他们所使用的模型。

## 一个问题，多层深度

以这个 PostgreSQL 问题为例：

> What is the (_SQL Standard_) syntax for an auto-incrementing IDENTITY column in PostgreSQL (v10+)?
> - `id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY` ✅ (Correct)
> - `id SERIAL PRIMARY KEY`  (Legacy syntax)
> - `id INT IDENTITY(1,1)`  (SQL Server syntax)
> - `id INT AUTO_INCREMENT`  (MySQL syntax)
> - `id NUMBER GENERATED ALWAYS AS IDENTITY`  (Oracle syntax)

该问题在不同专业水平上承担了多重教学目的：

- **初学者** 关注基本的 PostgreSQL 语法，主要是区分当前写法与旧版写法  
- **中级学习者** 在相似语法之间辨别细微差别  
- **高级实践者** 必须利用跨平台知识，同时对特定版本的实现保持精准

这个问题的作用不仅仅是让人给出语法答案，它在对学习者的思维模型进行分类。

选择 `SERIAL` 的人可能是从较旧的示例中学习到 Postgres 的。选择 `IDENTITY(1,1)` 的人则带有 SQL Server 的肌肉记忆。选择 `AUTO_INCREMENT` 的人懂概念但不熟悉方言。这些都是不同的教学切入点。

如果解释只说“`A` 是正确的”，那么这个问题就浪费了它最有价值的数据。

## 构建测验让我学到的东西

编写测验迫使我对读者真正需要学习的内容更加精准。

文章可以把模糊的解释隐藏在流畅的文字里，而测验不行。错误选项会暴露概念是否真的有边界。

当我现在撰写编程题时，我会问：

- 如果一个人只懂一半，这个聪明的人会选什么？
- 在相邻的生态系统中，哪个答案是正确的？
- 哪个答案保留了常见的误解？
- 哪个细微条件让正确答案成立？

这些问题让教学更锋利，也让文章更锋利，因为测验本身成了对解释的检验。如果我写不出四个可信的选项，说明我还没有真正抓住核心概念。

## 反馈循环才是关键

多项选择题在大规模使用时确实有用。评分自动化，模式一目了然。你可以看到哪些题目太容易、太模糊，或意外地在考 trivia。

但更好的反馈循环发生在学习者点击之前。

作者必须模拟学习者可能的错误。这意味着作者必须尊重学习者。一个敷衍的错误答案在说：“我只需要填充”。一个有力度的错误答案在说：“我清楚地知道这个思路会怎么偏离”。

这就是我在本站持续使用 MCQ 的原因。不是因为它们效率高（虽然确实如此），而是因为它们强迫一种教学上的诚实，而单纯的文字很容易逃避。

下次编写题目时，别先想正确答案，而是先从误解出发。正确答案随后才会有更稳固的落脚点。
