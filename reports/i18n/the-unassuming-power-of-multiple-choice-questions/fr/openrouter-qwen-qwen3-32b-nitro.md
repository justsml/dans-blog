Je croyais autrefois que les questions à choix multiples étaient les places bon marché de l'enseignement. Utiles pour vérifier si quelqu'un se souvient d'un terme, mais mauvaises pour savoir s'il peut vraiment utiliser l'idée. Acceptables pour les tests standardisés, moins adaptées pour le travail chaotique de l'apprentissage. Puis j'ai commencé à créer des quiz de programmation. La surprise n'était pas que les questions à choix multiples s'adaptent bien – tout le monde le sait. La surprise était qu'un bon ensemble de réponses pouvait enseigner la leçon avant même que l'explication ne soit donnée.  

<p class="inset">La partie utile d'une question à choix multiples est souvent moins la bonne réponse que les mauvaises réponses que vous choisissez intentionnellement.</p>  

## Le Distracteur Est Le Programme

La plupart des mauvaises questions à choix multiples comportent une bonne réponse, une réponse presque correcte, et deux options écrites en diagonale pendant que l'auteur attendait sa tasse de café.  

Cela gaspille le potentiel du format.  

Dans un bon quiz, chaque réponse fausse incarne une incompréhension réelle :  

- une syntaxe empruntée à un autre langage  
- un schéma obsolète encore présent dans le code ancien  
- un modèle mental plausible qui échoue sous une condition précise  
- une réponse qui serait correcte si un mot dans la question changeait  

C'est là que l'apprentissage se produit. Le candidat ne choisit pas simplement une option : il révèle le modèle mental qu'il utilise.  

## Une question, plusieurs profondeurs

Prenons cette question PostgreSQL comme exemple :  

> Quelle est la syntaxe (_SQL Standard_) pour une colonne IDENTITY auto-incrémentée en PostgreSQL (v10+) ?  
> - `id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY` ✅ (Correct)  
> - `id SERIAL PRIMARY KEY`  (Syntaxe obsolète)  
> - `id INT IDENTITY(1,1)`  (Syntaxe SQL Server)  
> - `id INT AUTO_INCREMENT`  (Syntaxe MySQL)  
> - `id NUMBER GENERATED ALWAYS AS IDENTITY`  (Syntaxe Oracle)  

Cette question remplit plusieurs objectifs pédagogiques selon les niveaux d'expertise :  

- **Les débutants** se concentrent sur la syntaxe de base de PostgreSQL, en distinguant principalement les approches actuelles et obsolètes  
- **Les apprenants intermédiaires** naviguent dans les subtilités des syntaxes similaires  
- **Les praticiens avancés** doivent mobiliser leurs connaissances interplateformes tout en restant précis sur les implémentations spécifiques aux versions  

La question ne se contente pas de demander de la syntaxe. Elle classe les modèles mentaux.  

Quelqu'un qui choisit `SERIAL` a probablement appris PostgreSQL via des exemples plus anciens. Quelqu'un qui choisit `IDENTITY(1,1)` apporte la mémoire musculaire de SQL Server. Quelqu'un qui choisit `AUTO_INCREMENT` maîtrise le concept mais pas le dialecte. Ce sont des moments pédagogiques distincts.

Si l'explication ne dit que « A est correct », la question jette son meilleur élément de données.  

## Ce que m'a appris la création de quiz  

Rédiger des quiz m'a obligé à être plus précis sur ce que je voulais vraiment que les lecteurs apprennent.  

Un article peut cacher une explication floue dans un texte lisse. Un quiz ne peut pas. Les mauvaises réponses révèlent si le concept a des arêtes réelles.  

Quand je rédige maintenant une question de programmation, je me demande :  

- Quel choix ferait une personne intelligente s'ils comprenaient à moitié cela ?  
- Quelle réponse serait correcte dans un écosystème voisin ?  
- Quelle réponse préserve la mécompréhension courante ?  
- Quelle petite condition rend la réponse correcte effectivement correcte ?

Ces questions rendent l'enseignement plus précis. Elles rendent également l'article plus précis, car le quiz devient un test de l'explication elle-même. Si je ne peux pas écrire quatre options crédibles, je n'ai probablement pas encore trouvé le vrai concept.

## La boucle de rétroaction est l'objectif

Les questions à choix multiples sont utiles à grande échelle, oui. La notation est automatique. Les modèles deviennent visibles. Vous pouvez identifier les questions trop faciles, trop vagues, ou qui testent accidentellement des détails triviaux.

Mais la meilleure boucle de rétroaction se produit avant même que l'apprenant ne clique.

L'auteur doit modéliser les erreurs probables de l'apprenant. Cela signifie que l'auteur doit respecter l'apprenant. Une mauvaise réponse paresseuse dit : « J'avais besoin de remplissage ». Une mauvaise réponse solide dit : « Je sais exactement comment cette idée peut dérailler ».

C'est pourquoi je continue à utiliser les QCM sur ce site. Pas seulement parce qu'ils sont efficaces, bien qu'ils le soient. Parce qu'ils imposent une forme d'honnêteté pédagogique que la prose seule peut éviter.

La prochaine fois que vous en rédigerez un, ne commencez pas par la bonne réponse. Commencez par les malentendus. La bonne réponse aura une place bien plus solide à tenir.
