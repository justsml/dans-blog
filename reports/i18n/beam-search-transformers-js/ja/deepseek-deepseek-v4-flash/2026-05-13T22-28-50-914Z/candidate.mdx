## 問題: `num_beams` は嘘だった

`transformers.js` の生成ループの奥深くに、長い間放置されていたコメントがありました。

```js
// TODO: Support beam search
```

そしてそのすぐ下に、最初のトークンでループを静かに抜け出す `break` 文がありました。`num_beams > 1` で出荷されたすべてのモデル設定（T5、BART、Whisper）は、静かに貪欲法（greedy decoding）で動作していたのです。警告もエラーもなく、ただ間違った出力が返ってきていました。

これを発見したのは、要約パイプラインをテストしていて、出力がPythonのリファレンスと比べてひどく劣化している理由を調べていたときでした。`modeling_utils.js` をたどり、TODOを見つけ、「どれほど難しいんだろう？」と思ってしまったのが間違いでした。

答えはこうです：かなり難しいが、面白い方法で難しい。

## ビームサーチの正体

誰もが一度は

貪欲法（greedy decoding）は各ステップで最も確率の高いトークンを選ぶ。シンプルで高速だが、しばしば最適ではない——口から出る最初の言葉が文の最良の始まりとは限らないのと同じだ。

ビームサーチは代わりに、`num_beams`個の候補系列を同時に保持し、各ステップでそれぞれを全語彙で拡張し、累積対数確率で上位`num_beams`個に刈り込む。トークン空間における制限付き幅優先探索のようなものだ。

結果として、`num_beams`倍の計算コストと引き換えに、全体的により良い系列が得られる。

3つの種類がある：

- **標準ビームサーチ** — 決定論的で、argmax候補を採用し、最良の系列を得る
- **多様ビームサーチ** — ビームをグループに分割し、先行グループが既に選択したトークンにペナルティを課すことで、出力候補がすべて同じ内容にならないようにする
- **ビームサンプリング** — 確率的で、ビームフレームワーク内でtop-k + softmax + ランダムサンプリングを適用する

これら3つすべてが現在PRに含まれている。

## 実際に苦労したアーキテクチャ上の判断

既存のコードベースには`BeamSearchSampler`クラスがあった。一見関連性があるように見えた。しかし、そこには微妙な罠があった。それはビームごとに上位`num_beams`個のトークンしか返さないのだ。一見正しく聞こえるが、実際のビームサーチには不十分だと気づくまでは。

適切なビームサーチでは、バッチアイテムごとに**すべての`num_beams × vocab_size`個の候補**を考慮して、全体的に最良の継続を見つける必要がある。各ビームから上位数トークンを個別に見るだけでは不十分で、すべてのビームを横断してランク付けする必要がある。

既存のサンプラーを完全にバイパスした。処理済みロジットに直接`log_softmax`を計算し、累積ビームスコアを加算し、結合候補空間で2段階のソートを行った。よりクリーンな数学、正しいセマンティクスだ。

`BeamSearchSampler`クラスはそのまま残っており、元々の用途にはまだ使える。これは、「明らかな」再利用の道が間違った方向に導くケースの一つだ。

## 最も厄介なバグ：KVキャッシュの並べ替え

ビームサーチがシーケンスを刈り込むとき、トークンを切り詰めるだけでなく、どのビームが生き残るかを*並べ替える*。ビーム3が最良の継続を生成して複製され、ビーム0と2は破棄されるかもしれない。

問題は、トランスフォーマーのアテンション機構からのキー・バリューキャッシュがバッチ次元でビームごとにインデックス付けされていることだ。出力シーケンスを並べ替えてもキャッシュを並べ替えなければ、状態が不一致になる。モデルが間違った過去にアテンションを向けてしまう。

修正は`_reorder_cache()`メソッドである。これは、すべての過去のキー・バリューテンソルに対して`index_select`を呼び出し、新しいビーム順序に従って再インデックス化し、古いテンソルを破棄する。

CPUではこれは簡単だ。型付き配列を行ごとにスライスする。GPUテンソルではもっと面倒だ。データを非同期でダウンロードし（`ort_tensor.getData(true)`）、並べ替えてから再アップロードする必要がある。`tensor.js`に`index_select`（同期、CPU用）と`index_select_async`の両方を追加して、両方のパスを処理した。

エンコーダー・デコーダーモデル（T5、BART）には*2つ*のキャッシュがある。エンコーダーとデコーダーだ。エンコーダーのPKVはデコード中に変化しないため、そのまま通過する。並べ替えが必要なのはデコーダーのPKVだけだ。この区別を間違えると、非常に悪い出力が生成される。参照と比較するまではほぼ正しく見える、微妙な間違いだ。

## 多様性ビームサーチ：面白い方

多様性ビームサーチは、後続のビームグループが先行グループで既に選択されたトークンを選ぶのを抑制する`diversity_penalty`を追加する。その直感は、すべてのビームが同じ出力に収束してしまうと、仮説空間を真に探索したとは言えない、というものだ。

実装上、グループは各デコードステップ内で*逐次的*に処理しなければならない。並列処理はできない。各グループが自身のスコアを計算する前に、先行グループが何を選択したかを確認する必要があるからだ。

最終的に採用した構造は次の通り：

```
for each step:
  token_counts = {}
  for each group in groups:
    extract this group's beams and logits
    for each token selected by previous groups:
      logits[token] -= diversity_penalty * token_counts[token]
    score candidates, select top 2×group_size
    group_scorer.process(...)
    record newly selected tokens into token_counts
```

ここでの逐次依存性は本物だ。並列化すると多様性の保証が失われる。一時はバッチ処理を試みようかと誘惑されたが、それは間違いだっただろう。

## `BeamHypotheses`優先度キュー

ビームが`max_length`より前にEOSトークンに達すると、それは「完了」となる。しかし、単に破棄したり即座に返したりはできない。`BeamHypotheses`と呼ばれる有界優先度キューに追加する。

このキューはバッチアイテムごとに最大`num_beams`個の完了シーケンスを保持し、以下の式でスコア付けされる：

```
score = sum_logprobs / (length ^ length_penalty)
```

`length_penalty > 1.0`は長い出力を促進し、`< 1.0`は短い出力を促進する。`early_stopping`フラグは、キューが満杯になった時点でビームを完了とみなすか（`true`）、`max_length`になるまで決して完了とみなさないか（`"never"`）、あるいは残りのビームが最悪の完了仮説を上回る可能性がなくなった時点で完了とみなすか（`false`）を制御する。

`false` の場合が興味深い。残りのスコアの最大値を考慮して、現在の最悪仮説をまだ上回れる生存ビームがあるかどうかを追跡する必要がある。これは、すでに良い仮説が得られている場合に `max_length` まで実行するのを防ぐ枝刈り最適化である。

これは新しいファイル `beam_search.js` にあり、全体で約240行である。また、バッチ全体の `BeamHypotheses` インスタンスを管理し、`finalize()` を処理する `BeamSearchScorer` をエクスポートする。

## Pythonリファレンスに対するテスト

ここにあるすべての重要な実装詳細には、HuggingFaceの `transformers` ライブラリにPython版の対応物がある。私はそれを大いに活用した。

追加したテストスイートは以下をカバーする：

- エンコーダー-デコーダー（T5）およびデコーダーのみ（LLaMA風）での標準ビームサーチ
- `num_beam_groups=2, diversity_penalty=0.5` を用いた多様性ビームサーチ
- `do_sample=true, top_k=10` を用いたビームサンプリング
- `num_return_sequences > 1` — 出力形状が `[N, seq_len]` であることを確認
- 互換性のない組み合わせに対する正しいエラースロー：CFG + ビームサーチ、ストリーミング + ビームサーチ、`num_return_sequences > num_beams`

「正しいエラースロー」テストは過小評価されている。これらは意図的な制限を文書化し、誰かが組み合わせられない機能を組み合わせようとしたときに、静かに間違った出力を得るのを防ぐ。（開発中にCFGとビームサーチを組み合わせようとしたので分かる。数学が成立しない。今はエラーがスローされる。）

## まだ欠けているもの

私が明示的に除外したいくつかの機能を、`throws` で示している：

- **多様性ビームサンプリング**（`num_beam_groups > 1` + `do_sample`）：ここでの数学は本当に複雑になる。標準の多様性ビームサーチはグループ間で逐次的であり、そこにサンプリングを追加するには、確率的モードで多様性ペナルティをどのように適用するかについて慎重な検討が必要だ。実行可能ではあるが、まだ実装していない。
- **ストリーミング + ビームサーチ**：ストリーミングはトークンが生成されるたびに出力する。ビームサーチは定義上、複数ステップ先まで最良の系列がわからない。これらは根本的に相反する。現時点での最良のビームをストリーミングすることは可能だが、それは独自の設計上の問題を伴う別の機能だ。

## 誰も語らない部分：オープンソースのレイテンシ

コードは動作する。テストは通る。既存のテストスイートはクリーンだ。しかし、レビューに数ヶ月間放置されている。

これは大規模で人気のあるオープンソースプロジェクトではよくあることだ。Hugging Face チームは迅速にリリースしており、issue キューは膨大で、コアの生成ループに触れる約1000行の機能PRのレビューは簡単なコミットメントではない。彼らはコメントで応答し、実際にレビューするときは真摯に関与している。不満を言っているわけではない——記録しているのだ。

主要なOSSプロジェクトに貢献していて、すぐにマージされることを期待しているなら、その期待を調整すべきだ。この規模のものでは数ヶ月は普通だ。コードはその間ずっとあなたのフォークで動作し続ける。

## 実際に得られたもの

以前は持っていなかったいくつかのもの：

1. **ビームサーチの真のメンタルモデル** — 教科書版ではなく、エッジケースを含むもの。KVキャッシュが壊れる仕組み。2段階のソートが重要な理由。`length_penalty`がスコアに実際に与える影響。

2. **JavaScriptにおける型付き配列演算への感謝** — CPU上の型付き配列で`index_select`を実装するのは、Webコードでは滅多に触れない低レベルの作業だ。問題なく動作するが、JavaScriptが本来想定していた用途ではないと感じる。

3. **Pythonリファレンス実装への新たな敬意**。HuggingFaceの`transformers`ライブラリは大きく、時に厄介だが、ビームサーチのロジックはコメントが充実しており、設計判断は明確に意図的だ。それを読むことが、実際に何を構築すべきかを理解する最短の方法だった。

4. **野生のパッチ** — まだマージされていなくても、存在し、動作し、PRブランチから誰でも使える。それで十分だ。

このすべてのきっかけとなったTODOコメントは、私のフォークから消えた。静かでオタク的な満足感がある。
