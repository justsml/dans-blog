## 問題: `num_beams` は嘘だった

`transformers.js` の生成ループの奥深くに、長期間存在していたコメントがありました。

```js
// TODO: Support beam search
```

その直下には、最初のトークンの後にループを静かに終了させる `break` 文がありました。`num_beams > 1` で構成されたすべてのモデル（T5、BART、Whisper など）は、静かに貪欲デコードを実行していました。警告もエラーもなく、ただ...出力が間違っていたのです。

## ビームサーチが本当に何なのか

誰もが一度は

貪欲デコードは、各ステップで確率が最も高いトークンを選びます。シンプルで高速ですが、頻繁に最適でない結果になります。口から最初に出てくる言葉が、必ずしも文の最適な始まりではないのと同じです。

ビームサーチは、代わりに`num_beams`個の候補シーケンスを同時に保持します。各ステップで全語彙を用いて各候補を拡張し、累積された対数確率で上位`num_beams`に絞り込みます。これはトークン空間における幅優先探索の一種で、探索範囲を制限しています。

結果としてグローバルに優れたシーケンスが得られますが、計算コストは`num_beams`倍かかります。

3つの実装形態があります：

- **標準ビームサーチ** – 決定論的でargmax候補を採用し、全体最適なシーケンスを返す
- **多様性ビームサーチ** – ビームをグループに分割し、既に選ばれたトークンをペナルティ付与することで出力候補が重複しないようにする
- **確率的ビームサーチ** – 確率論的で、ビームフレームワーク内にtop-k + softmax + ランダムサンプリングを適用する

これら3つは現在のPRで実装済みです。

## 実際に悩んだアーキテクチャ決定

既存のコードベースには`BeamSearchSampler`クラスがありました。関連性がありそうに見えましたが、巧妙な罠がありました。このクラスは各ビームごとに上位`num_beams`トークンのみを返していました。これは一見正しいように思えますが、実際のビームサーチには不十分です。

正しいビームサーチでは、**バッチアイテムごとに`num_beams × vocab_size`個の候補**を検討してグローバル最適な継続候補を特定する必要があります。各ビームの上位数トークンだけで判断するのではなく、すべてのビームを統合して順位付けを行う必要があります。

したがって、私は既存のサンプラーを完全にバイパスしました。処理済みのlogitsに直接`log_softmax`を計算し、累積ビームスコアを加算し、統合候補空間で2段階のソートを行いました。より洗練された数学的処理で、正しいセマンティクスを実現しました。

`BeamSearchSampler`クラスは未変更で残っています。もともとの用途には引き続き有用です。これは「明らかに見える」再利用パスが誤った方向に導く典型的な事例の一つです。

## 最も面倒なバグ：KVキャッシュの再並べ替え

ビームサーチがシーケンスをトリムする際、トークンを単に短縮するだけでなく、**生存するビームの順序を再構成**します。ビーム3が最適な継続を生成した場合、複製されるかもしれません。ビーム0と2は破棄されるかもしれません。

問題は、トランスフォーマーのアテンションメカニズムのキー・バリューキャッシュがバッチ次元に沿ってビームごとにインデックス付けられていることです。出力シーケンスを再並べ替えてもキャッシュを再並べ替えなければ、状態が不一致になります。モデルは過去の誤った情報に注目してしまうのです。

修正方法は`_reorder_cache()`です——これは、新しいビーム順に従って過去のすべてのキー・バリューテンソルに`index_select`を呼び出し、再インデックス化し、古くなったテンソルを破棄するメソッドです。

CPUの場合、単純に型付き配列を行単位でスライスするだけで済みます。GPUテンソルでは面倒になります——非同期でデータをダウンロード(`ort_tensor.getData(true)`)し、再並べ替え、再アップロードする必要があります。このため、`tensor.js`に`index_select`（同期、CPU用）と`index_select_async`を追加し、両方のパスを処理できるようにしました。

エンコーダー・デコーダーモデル（T5、BART）には**2つのキャッシュ**があります——エンコーダーとデコーダー。エンコーダーのPKVはデコード中に変化しないため、そのまま通過します。変更が必要なのはデコーダーのPKVのみです。この区別を誤ると、出力が非常に悪くなり、しかも**微妙に間違った形で**——参照と比較するまでにほぼ正しいように見えます。

## 多様性のあるビームサーチ：面白いほう

多様性のあるビームサーチでは、`diversity_penalty`を追加して、後続のビームグループが先行グループが選択したトークンを再選択することを抑制します。その背後にある直感はこうです——すべてのビームが同じ出力に収束してしまうと、仮説空間を十分に探索できていないということです。

実装上では、各デコードステップ内でグループを**逐次処理**する必要があります。並列処理ではいけません。なぜなら、各グループが自身のスコアを計算する前に、先行グループが選択したトークンを参照する必要があるからです。

最終的に実装した構造は以下の通りです：

```
for each step:
  token_counts = {}
  for each group in groups:
    extract this group's beams and logits
    for each token selected by previous groups:
      logits[token] -= diversity_penalty * token_counts[token]
    score candidates, select top 2×group_size
    group_scorer.process(...)
    record newly selected tokens into token_counts
```

ここでの逐次依存関係は本質的です。並列化してしまうと、多様性の保証が失われます。一時的にバッチ処理を試みた誘惑にかられたこともありますが、それは間違いでした。

## `BeamHypotheses`プライオリティキュー

ビームが`max_length`に達する前にEOSトークンに到達した場合、それは「完了」したとみなされます——しかし、単に破棄したり即座に返したりすることはできません。代わりに、`BeamHypotheses`というバウンドプライオリティキューに追加します。

このキューは、バッチごとに最大`num_beams`個の完了済みシーケンスを保持し、以下の式でスコアリングします：

```
score = sum_logprobs / (length ^ length_penalty)
```

`length_penalty > 1.0`は長めの出力を報酬し、`< 1.0`は短めの出力を報酬します。`early_stopping`フラグは、キューが満杯になった時点でビームを「完了」扱いするかどうか（`true`）、`max_length`に達するまで決して完了しない（`"never"`）、または残りのビームが最悪の完了済み仮説を上回る可能性がなくなった時点で完了とするか（`false`）を制御します。

`false`の場合が興味深い点です。これは、最大の可能性のある残りスコアを与えられた状態で、どのライブビームが現在の最悪の仮説を打ち勝つ可能性があるかを追跡する必要があります。これは、すでに良い仮説を持っている場合に`max_length`まで実行しないようにする枝刈り最適化です。

これは`beam_search.js`に実装されており、新規ファイルで全体で約240行あります。また、バッチ全体で`BeamHypotheses`インスタンスを管理し`finalize()`を処理する`BeamSearchScorer`もエクスポートしています。

## Pythonリファレンスとのテスト

ここに含まれるすべての非自明な実装詳細には、HuggingFaceの`transformers`ライブラリにPythonの対応物があります。私はこれを非常に活用しました。

追加したテストスイートは以下の通りです：

- エンコーダー-デコーダー（T5）およびデコーダー専用（LLaMA-ish）の標準的なビームサーチ
- `num_beam_groups=2, diversity_penalty=0.5`のダイバーシビームサーチ
- `do_sample=true, top_k=10`のビームサンプリング
- `num_return_sequences > 1` — 出力形状が`[N, seq_len]`であることを検証
- 互換性のない組み合わせの正しいエラー送出：CFG + ビームサーチ、ストリーミング + ビームサーチ、`num_return_sequences > num_beams`

「正しいエラー送出」テストは見落とされがちですが重要です。これらは意図的な制限を文書化し、ユーザーが結合不可能な機能を組み合わせた際に間違って誤った出力を得ないように防ぎます。（私が開発中にCFGとビームサーチを組み合わせようとした経験から言えることです。数学的に成立しません。現在はエラーが送出されます。）

## まだ実装されていないもの

明示的に省いたいくつかの点（`throws`でマーク）：

- **ダイバーシビームサンプリング**（`num_beam_groups > 1` + `do_sample`）：ここでの数学は本格的に複雑になります。標準的なダイバーシビームサーチはグループ間で逐次処理されるため、そこにサンプリングを加えるには、確率的モードでのダイバーシティペナルティの適用方法について慎重な検討が必要です。実装可能ではありますが、まだ実施されていません。
- **ストリーミング + ビームサーチ**：ストリーミングはトークンが生成された時点で出力します。ビームサーチの定義上、複数ステップの結果を確認するまでに最良のシーケンスが不明です。これらは本質的にトレードオフがあります。現時点で生成された最良ビームをストリームする方法もありますが、それは別の機能であり、独自の設計上の課題があります。

## 誰も口にしない話：オープンソースの遅延

コードは動作します。テストもパスしています。既存のテストスイートはクリーンです。これは何ヶ月もレビュー待ちの状態です。

これは単に大規模で人気のあるオープンソースプロジェクトの日常です。Hugging Faceチームは迅速にリリースしており、イシューの数は膨大です。約1,000行にわたるコア生成ループを変更する機能追加のPRは、レビューの負担が大きく、非自明な作業です。コメントへの対応は迅速で、実際にレビューする際も真剣に取り組んでいます。私は不平を言っているわけではありません。ただ記録しているだけです。

主要なOSSプロジェクトに貢献し、迅速なマージを期待している場合：期待値を調整してください。この規模の機能追加では数カ月が通常です。フォークされたコードはその間ずっと動作します。

## 実際に得たもの

以前持っていなかったいくつかのもの：

1. **ビームサーチの実際のメンタルモデル** — 教科書的なバージョンではなく、エッジケースを含むもの。KVキャッシュがどう壊れるか。2段階のソートがなぜ重要か。`length_penalty`がスコアに実際に与える影響。  

2. **JSにおける型付き配列の数学への理解の深まり** — CPU型付き配列で`index_select`を実装する行為は、ウェブコードではめったに触れない低レベルな作業だ。構文上問題ないが、JavaScriptが設計された目的とはずれている感覚は伝わってくる。  

3. **Pythonのリファレンス実装への再評価** — HuggingFaceの`transformers`ライブラリは巨大で複雑だが、ビームサーチのロジックはコメントが丁寧で設計決定が明確に意図的だ。読むのが最も素早い学習だった。  

4. **現実世界でのパッチ** — まだマージされていなくても存在し、動作し、PRブランチから利用可能だ。それだけで十分だ。
