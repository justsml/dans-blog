## ビームサーチが実際に何であるか

誰もが気づいていないことだが、貪欲デコードは各ステップで最も確率の高いトークンを選びます。シンプルで速いですが、しばしば最適ではありません。口から出る最初の単語が必ずしも文の最良の始まりとは限りません。

ビームサーチは代わりに、`num_beams` 個の候補シーケンスを同時に維持します。各ステップでそれぞれの候補を完全な語彙で拡張し、累積された対数確率で上位 `num_beams` に絞り込みます。これはトークン空間を探索する際の限定された幅優先探索のようなものです。

## 最もうんざりするバグ：KVキャッシュの再順序付け
---

ビームサーチがシーケンスを切り捨てるとき、トークンの単なる短縮ではなく、**どのビームが生き残るかの順序付け**を行います。ビーム3が最適な継続を生成して複製され、ビーム0や2が破棄される可能性があります。

問題は、トランスフォーマーのアテンションメカニズムのキー・バリュー（KV）キャッシュが**ビームごとにバッチ次元に沿ってインデックス化されている**ことです。出力シーケンスの順序を変更してもキャッシュの順序を再調整しないと、状態が一致しなくなります。モデルが誤った過去に注目してしまうのです。

対処は`_reorder_cache()`——このメソッドはすべての過去のキー・バリューテンソルに`index_select`を呼び出して、新しいビーム順序に従って再インデックスし、古くなったテンソルを破棄します。

CPUの場合、型付き配列を行単位でスライスするだけなので単純です。GPUテンソルでは面倒になります——データを非同期ダウンロード（`ort_tensor.getData(true)`）し、順序を変更して再アップロードする必要があります。このため、`tensor.js`に`index_select`（同期、CPU用）と`index_select_async`を追加して、両方のパスを処理できるようにしました。

エンコーダー-デコーダーモデル（T5、BARTなど）は**2つのキャッシュ**を持ちます：エンコーダーとデコーダー。エンコーダーのPKV（過去のキー・バリュー）はデコード中に変化しないため、変更なしで通過します。再順序付けが必要なのはデコーダーのPKVだけです。この区別を誤ると、非常に悪い出力が**微妙に**生じます——参照と比較するまで、ほぼ正しそうに見える誤りです。

## 多様なビーム探索: 興味深い実装

多様なビーム探索では、後続のビームグループが以前のグループが選択したトークンを選ぶことを抑制する`diversity_penalty`を追加します。その直感はこうです: すべてのビームが同じ出力に収束すると、仮説空間を十分に探索していないことになります。

実装上は、各デコードステップ内でグループを**逐次**処理する必要があります。並列処理ではダメです。というのも、各グループは自身のスコアを計算する前に、以前のグループが選択した内容を確認する必要があるからです。

最終的に落ち着いた構造はこうなりました:

```
for each step:
  token_counts = {}
  for each group in groups:
    extract this group's beams and logits
    for each token selected by previous groups:
      logits[token] -= diversity_penalty * token_counts[token]
    score candidates, select top 2×group_size
    group_scorer.process(...)
    record newly selected tokens into token_counts
```

ここでシーケンシャルな依存関係は実際に必要です。これを並列化すると、多様性の保証が失われます。私は一時的にバッチ処理を試みる誘惑にかられたのですが、それは間違いでした。

## `BeamHypotheses` 用のプライオリティキュー

ビームが `max_length` に達する前に EOS トークンに到達した場合、それは「完了」したとみなされます — しかし、それを単に破棄したり直ちに返すことはできません。これを `BeamHypotheses` と呼ばれる上限付きプライオリティキューに追加します。

このキューは、バッチアイテムごとに最大 `num_beams` 個の完了済みシーケンスを保持し、以下のスコアで管理します：

```
score = sum_logprobs / (length ^ length_penalty)
```

`length_penalty > 1.0` は長めの出力を報酬し、`< 1.0` は短めの出力を報酬します。`early_stopping` フラグは、キューが満了したときにビームが「完了」とみなされるかどうか（`true`）、`max_length` に達するまで決して完了しない（`"never"`）、または現在の最悪の仮説を上回る可能性があるビームが存在しなくなった時点で完了とみなす（`false`）を制御します。

`false` のケースが特に興味深いです — これは、最大で想定される残りスコアを考慮して、どのライブビームが現在の最悪の仮説を上回る可能性があるかを追跡する必要があります。これは、すでに良い仮説が存在する場合に `max_length` まで実行する必要をなくすための枝刈り最適化です。

これは `beam_search.js` という新しいファイルに実装されています。全体で約240行あります。また、`BeamSearchScorer` もエクスポートしており、これはバッチ全体の `BeamHypotheses` インスタンスを管理し、`finalize()` を処理します。
