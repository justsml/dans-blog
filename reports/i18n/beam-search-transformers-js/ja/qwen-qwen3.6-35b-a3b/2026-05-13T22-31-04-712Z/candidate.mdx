## 問題：`num_beams` は嘘だった

`transformers.js`のトークン生成ループの深部には、長年放置されていたコメントが存在した。

```js
// TODO: Support beam search
```

その直下には、最初のトークン生成後にループを静かに終了させる`break`文があった。`num_beams > 1`で出荷されていたすべてのモデル設定（T5、BART、Whisper）は、気づかれずにグリーディデコーディングにフォールバックしていた。警告もエラーも出ない。ただ……出力が間違っているだけだった。

私は要約パイプラインのテスト中に、Pythonの参照実装と比較して自分の出力がなぜこれほど劣化しているのか疑問に思いながら、これを発見した。`modeling_utils.js`までトレースしてTODOを見つけた瞬間、「そんなに難しいわけないだろう」という過ちを犯してしまった。

答えはこうだ。かなり難しい。だが、面白いやり方で。

## ビームサーチの正体

誰だって一度は...

グリーディデコーディングは、各ステップで確率が最も高いトークンを選択する。シンプルで高速だが、しばしば最適解ではない。口から出てきた最初の単語が、必ずしも文の始まりとして最善とは限らないのだ。

ビームサーチは代わりに、`num_beams` 個の候補シーケンスを同時に保持し、各ステップで完全な語彙サイズ分だけ展開し、その後、累積対数確率に基づいて上位 `num_beams` 個に刈り込む。トークン空間における有界な幅優先探索のようなものだ。

結果として、計算コストが `num_beams` 倍かかる代わりに、よりグローバルに優れたシーケンスが得られる。

主に3つのバリエーションが存在する。

- **標準ビームサーチ** — 決定論的。argmax候補を採用し、全体として最良のシーケンスを得る
- **多様性ビームサーチ** — ビームをグループに分割し、先行グループで既に選択されたトークンにペナルティを課すことで、出力候補がすべて同じ内容になるのを防ぐ
- **ビームサンプリング** — 確率的。ビームの枠組み内で top-k + softmax + ランダムサンプリングを適用する

これら3つはすべて、現在のPRにマージ済みだ。

## 実際に頭を悩ませたアーキテクチャの判断

既存のコードベースには `BeamSearchSampler` クラスが存在していた。一見すると関連がありそうだ。だが、そこには微妙な罠があった。このクラスはビームごとに上位 `num_beams` 個のトークンしか返さないのだ。一見すると正しそうに聞こえるが、実際のビームサーチにはそれが足りないことに気づけば、話は別だ。

本来のビームサーチでは、グローバルに最適な継続シーケンスを見つけるために、バッチアイテムごとに **`num_beams × vocab_size` 個の候補すべて** を考慮する必要がある。各ビームから上位数個のトークンを孤立して見るだけでは不十分だ。すべてのビームを横断して順位付けする必要がある。

結局、既存のサンプラーを完全にスキップした。処理済みのロジットに対して直接 `log_softmax` を計算し、累積ビームスコアを加算してから、結合された候補空間全体で2段階のソートを実行した。計算式はすっきりし、意味論も正しくなった。

`BeamSearchSampler` クラスはそのまま残っている。手を加えていないが、元の用途ではまだ役に立っている。これは「明らかに再利用できる」という道が実は間違った方向へ導く、典型例の一つだ。

## 最も厄介なバグ：KVキャッシュの再順序付け

ビームサーチがシーケンスを剪定する際、単にトークンを切り捨てるだけでなく、生き残るビームの *順序* を入れ替える。ビーム3が最も良い継続シーケンスを生成して複製され、ビーム0と2が破棄される可能性がある。

問題は、トランスフォーマーの注意機構が生成するキー・バリューキャッシュが、バッチ次元に沿ってビームごとにインデックス付けされている点にある。キャッシュの順序を入れ替えずに出力シーケンスだけを入れ替えると、内部状態が一致しなくなる。モデルは過去の誤った情報を参照してしまう。

修正は `_reorder_cache()` メソッドだ。このメソッドは、各過去のキー・バリューテンソルに対して `index_select` を呼び出して新しいビーム順序で再インデックスし、その後で古いテンソルを破棄する。

CPUでは単純だ：行方向に型付き配列をスライスするだけ。GPUテンソルになると厄介になる。データを非同期でダウンロードし（`ort_tensor.getData(true)`）、順序を入れ替えて、再度アップロードする必要がある。両方のパスに対応するため、`tensor.js` に `index_select`（同期、CPU用）と `index_select_async` の両方を追加した。

エンコーダ・デコーダモデル（T5やBARTなど）は *2つ* のキャッシュを持つ：エンコーダ用とデコーダ用だ。エンコーダのPKVはデコード中に変わらないため、変更なしでそのまま通過する。順序を入れ替える必要があるのはデコーダのPKVだけだ。この区別を間違えると、非常に悪い出力が生成される。微妙にだ。参照結果と突き合わせるまで、ほぼ正しく見える、あの種の誤りだ。

## Diverse Beam Search: 面白い方

Diverse beam search は `diversity_penalty` を導入し、後のビームグループが先行グループで既に選択されたトークンを選出するのを抑制する。直感的な理由はこうだ：すべてのビームが同じ出力に収束するなら、仮説空間を本当に探索したことになるわけではない。

実装面では、各デコードステップ内でグループを並列ではなく *逐次* 処理する必要がある。各グループが自身のスコアを計算する前に、先行グループが何を選択したかを確認する必要があるからだ。

最終的にたどり着いた構造は以下の通りだ：

```
for each step:
  token_counts = {}
  for each group in groups:
    extract this group's beams and logits
    for each token selected by previous groups:
      logits[token] -= diversity_penalty * token_counts[token]
    score candidates, select top 2×group_size
    group_scorer.process(...)
    record newly selected tokens into token_counts
```

この逐次依存関係は本物だ。並列化すれば、多様性の保証は崩れる。私は一瞬、それでもバッチ処理でやってみようかと思ったが、それは間違いだった。

## `BeamHypotheses` 優先度キュー

ビームが `max_length` より前に EOS トークンに到達すると、そのビームは「完了」状態になる — ただし、単に破棄したり即座に返却したりはできない。これを `BeamHypotheses` という上限付き優先度キューに追加する。

このキューは、バッチアイテムごとに最大 `num_beams` 個の完了済みシーケンスを保持し、スコアは以下で計算される：

```
score = sum_logprobs / (length ^ length_penalty)
```

`length_penalty > 1.0` は長い出力を報酬し、`< 1.0` は短い出力を報酬する。`early_stopping` フラグは、キューがいっぱいになった時点でビームを完了とみなすか（`true`）、`max_length` に達するまで決して完了とみなさないか（`"never"`）、あるいは残りのビームが最悪の完了済み仮説のスコアを上回る可能性がまったくない時点で完了とみなすか（`false`）を制御する。

`false` のケースが興味深い。最大で残せるスコアを考慮し、まだ生存しているビームが現在の最悪の仮説を上回る可能性があるかどうかを追跡する必要があるからだ。これは、すでに良い仮説が得られている場合に `max_length` まで実行するのを防ぐ、剪定（pruning）の最適化だ。

これらは新しいファイル `beam_search.js` に実装されており、合計約240行だ。また、バッチ全体にわたって `BeamHypotheses` のインスタンスを管理し、`finalize()` を処理する `BeamSearchScorer` もエクスポートしている。

## Pythonリファレンスとのテスト

ここにある非自明な実装の詳細はすべて、HuggingFace の `transformers` ライブラリにPython版が存在する。実装の指針としてそれを強く頼りにした。

私が追加したテストスイートは以下のケースをカバーしている：

- エンコーダ・デコーダ（T5）およびデコーダのみ（LLaMA風）モデルでの標準ビームサーチ
- `num_beam_groups=2, diversity_penalty=0.5` を指定した多様性ビームサーチ
- `do_sample=true, top_k=10` を指定したビームサンプリング
- `num_return_sequences > 1` — 出力形状が `[N, seq_len]` であることを検証
- 互換性のない組み合わせに対する正しいエラースロー：CFG + ビームサーチ、ストリーミング + ビームサーチ、`num_return_sequences > num_beams`

「正しいエラースロー」のテストは過小評価されがちだ。これらは意図的な制限を文書化し、組み合わせることができない機能を無理やり結合しようとした際に、間違った出力が静かに返されるのを防ぐ。（開発中にCFGとビームサーチを結合しようとして実際に試したことがある。数学的に成立しないからだ。今はちゃんとエラーを投げる。）

## まだ実装されていない機能

`throws` で明示的に除外した機能は以下の通りだ。

- **多様性ビームサンプリング** (`num_beam_groups > 1` + `do_sample`): 数学的な処理が本格的に複雑になる。標準的な多様性ビームサーチはグループ間で逐次実行されるため、サンプリングを組み合わせるには、確率的なモードにおいて多様性ペナルティをどう適用するかを慎重に設計する必要がある。技術的には可能だが、まだ実装していない。
- **ストリーミング + ビームサーチ**: ストリーミングは生成されたトークンを逐次出力する。一方、ビームサーチは定義上、複数ステップ先まで展開しないとどのシーケンスが最適か判断できない。この二つは根本的に性質が衝突する。現時点での最良ビームだけをストリーミングすることは技術的に可能だが、それは設計上の課題が異なる別の機能として扱う必要がある。

## 誰も語らない部分：オープンソースのレビュー遅延

コードは動作する。テストはパスする。既存のテストスイートもクリーンだ。しかし、レビューに数ヶ月も引っ張られている。

これは規模が大きく人気のあるオープンソースプロジェクトではよくある話だ。Hugging Face チームは高速でリリースを進めており、イシューキューは膨大で、コアの生成ループに干渉する ~1,000 行規模の機能 PR は、それだけで多大なレビュー負荷となる。コメントへの対応は迅速で、実際にレビューに入った際には真摯に関与してくれる。文句を言っているわけではない。記録に残しているだけだ。

主要な OSS プロジェクトにコントリビュートして、すぐにマージされることを期待しているなら：期待値を調整する必要がある。この規模の PR なら数ヶ月かかるのが普通だ。フォーク上ではコードはずっと動作し続けている。

## 実際に得られたもの

以前には持っていなかったものがいくつかある。

1. **ビームサーチの実際のメンタルモデル** — 教科書版ではなく、エッジケースを含む実装版。KVキャッシュがどのように破綻するか。なぜ2段階ソートが重要なのか。`length_penalty` がスコアに実際にどのような影響を与えるか。

2. **JSにおけるTyped Array演算へのより深い理解** — CPU上のTyped Arrayで`index_select`を実装するのは、Web開発ではまず触れない低レベルな作業だ。問題ないが、JavaScriptの設計思想とは異なり、その違いは確かに体感する。

3. **Pythonリファレンス実装への再評価。** HuggingFaceの`transformers`ライブラリは規模が大きく、時に複雑怪奇な部分もあるが、ビームサーチのロジックはコメントが充実しており、設計判断は明確に意図的に行われている。それを読むのが、実際に何を構築すべきかを理解する最速の方法だった。

4. **実世界に存在するパッチ** — まだマージされていなくても、存在し、動作し、開発者はPRブランチからそれを利用できる。それで十分だ。
