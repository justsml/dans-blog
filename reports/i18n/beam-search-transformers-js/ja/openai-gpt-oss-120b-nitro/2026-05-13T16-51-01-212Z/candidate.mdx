## 多様性ビームサーチ：楽しいやつ

多様性ビームサーチは `diversity_penalty` を導入し、後続のビームグループが先行グループですでに選択されたトークンを選ばないようにします。直感としては、すべてのビームが同じ出力に収束してしまうと、仮説空間を十分に探索できていない、ということです。

実装上は、各デコードステップで **順次** にグループを処理する必要があります。各グループは自分のスコアを計算する前に、前のグループが何を選択したかを参照しなければならないためです。

最終的に構築した構造は次の通りです：

```
for each step:
  token_counts = {}
  for each group in groups:
    extract this group's beams and logits
    for each token selected by previous groups:
      logits[token] -= diversity_penalty * token_counts[token]
    score candidates, select top 2×group_size
    group_scorer.process(...)
    record newly selected tokens into token_counts
```

ここでの **順次依存性は実際に存在します**。これを並列化すると多様性の保証が失われます。私は一時的にバッチ化してみようかと考えましたが、そうすれば大きな間違いになるでしょう。

## `BeamHypotheses` 優先度キュー

ビームが `EOS` トークンに到達して `max_length` 未満で終了した場合、単に破棄したりすぐに返したりできません。代わりに `BeamHypotheses` と呼ばれる上限付き優先度キューに追加します。

キューはバッチ項目ごとに最大 `num_beams` 個の完了シーケンスを保持し、次の式でスコア付けします。

```
score = sum_logprobs / (length ^ length_penalty)
```

`length_penalty > 1.0` は長い出力を優遇し、`< 1.0` は短い出力を優遇します。`early_stopping` フラグはビームが完了とみなされるタイミングを制御します。キューが満杯になった時点で完了とする (`true`)、`max_length` に達するまで決して完了しない (`"never"`)、あるいは「残りのビームが最悪の完了仮説を上回る可能性がなくなる」まで完了しない (`false`) のいずれかです。

`false` のケースが興味深いです――これは、残りのスコア上限を考慮して、現在の最悪仮説を上回る可能性があるライブビームがまだ存在するかどうかを追跡する必要があります。これは、すでに良好な仮説が得られている場合に `max_length` まで走らせるのを防ぐ剪定最適化です。

このロジックは `beam_search.js`（新規ファイル、全体で約 240 行）に実装されます。また `BeamSearchScorer` もエクスポートされ、バッチ全体で `BeamHypotheses` インスタンスを管理し、`finalize()` を処理します。

## Python リファレンスとの比較テスト

## まだ足りないもの

明示的に除外した項目は `throws` でマークしています:

- **多様ビームサンプリング** (`num_beam_groups > 1` + `do_sample`): ここでは数式が実際に複雑になります。標準的な多様ビームサーチはグループ間で順次処理されますが、そこにサンプリングを加えると、確率的モードで多様性ペナルティをどう適用するかを慎重に設計する必要があります。実装は可能ですが、まだ手を付けていません。
- **ストリーミング + ビームサーチ**: ストリーミングはトークンが生成されるたびに出力します。一方ビームサーチは、複数ステップ先まで進んで初めてどのシーケンスが最適かを判断します。この二つは根本的に張り合う関係にあります。現在のベストビームをストリーミングすることは可能ですが、別の機能となり、独自の設計課題が生じます。

## 誰も語らない部分: オープンソースのレイテンシ
---

大規模な OSS プロジェクトに貢献し、すぐにマージされることを期待しているなら、期待値を調整してください。この規模の変更が数か月かかるのは普通です。コードはフォーク上でも常に動作し続けます。
