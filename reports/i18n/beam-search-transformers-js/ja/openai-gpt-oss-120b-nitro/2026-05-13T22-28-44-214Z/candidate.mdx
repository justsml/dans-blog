## 問題: `num_beams` は嘘だった

`transformers.js` の生成ループの奥深くには、長らく残っていたコメントがありました:

```js
// TODO: Support beam search
```

そしてそのすぐ下に、最初のトークンでループを黙って抜け出す `break` 文がありました。`num_beams > 1` を指定して出荷されたすべてのモデル設定 — T5、BART、Whisper — は、実は黙ってグリーディーデコードにフォールバックしていました。警告もエラーもなく、ただ…間違った出力が得られるだけでした。

私は要約パイプラインをテストしているときに、出力が Python のリファレンスに比べて著しく劣化していることに気づきました。`modeling_utils.js` に遡って TODO を見つけ、 「どれだけ難しくないだろう？」 と考えてしまったのが間違いでした。

答えは: 結構難しい、しかし興味深い形で。

## ビームサーチが実際に何か

誰もが...

グリーディーデコードは、各ステップで最も確率の高いトークンを選択します。シンプルで高速ですが、しばしば最適ではありません。口から出る最初の単語が、文のベストな出だしになるとは限らないのと同じです。

ビームサーチは、`num_beams` 個の候補シーケンスを同時に保持し、各ステップで全語彙に対して展開した後、累積対数確率で上位 `num_beams` に剪定します。トークン空間を幅優先で探索するが、探索幅に上限を設けたようなものです。

その結果、グローバルにより良いシーケンスが得られますが、計算コストは `num_beams` 倍になります。

3 つのバリエーションがあります：

- **Standard beam search** — 決定的で、argmax 候補を取り、全体で最良のシーケンスを生成
- **Diverse beam search** — ビームをグループに分割し、先行グループが選んだトークンにペナルティを課すことで、出力候補がすべて同じ内容になるのを防止
- **Beam sampling** — 確率的で、ビーム枠組み内で top‑k + softmax + ランダムサンプリングを適用

これらすべてが現在 PR に含まれています。

## 実際に検討したアーキテクチャの決定

既存コードベースには `BeamSearchSampler` クラスがあり、ひとまず適切に見えました。しかし微妙な落とし穴がありました：このクラスはビームごとに上位 `num_beams` トークンしか返しません。これは一見正しそうですが、実際のビームサーチには不十分です。

正しいビームサーチでは、バッチ項目ごとに **`num_beams × vocab_size` すべての候補** を考慮して、グローバルに最適な継続を見つける必要があります。各ビームから上位数トークンだけを個別に見るだけでは不十分で、すべてのビームを横断して順位付けしなければなりません。

既存のサンプラーを完全に回避し、処理済みロジットに対して直接 `log_softmax` を計算し、累積ビームスコアを加算して、結合された候補空間全体で二段階のソートを行いました。数式がすっきりし、意味論も正しくなります。

`BeamSearchSampler` クラス自体はそのまま残っており、元々の用途にまだ有用です。いわゆる「明らかな」再利用パスが逆に間違った結果を導く典型例です。

## 最も厄介なバグ: KV キャッシュの再順序付け

ビームサーチでシーケンスを剪定するとき、単にトークンを切り捨てるだけでなく、**どのビームが残るかを再順序付け**します。ビーム 3 が最良の続き方を示してクローン化され、ビーム 0 と 2 が破棄されることがあります。

問題は、トランスフォーマーの注意機構からのキー‑バリューキャッシュがバッチ次元に沿ってビーム単位でインデックス付けされている点です。出力シーケンスだけを再順序付けてキャッシュを再順序付けしなければ、状態がずれ、モデルは過去の誤った位置に注意を向けてしまいます。

この修正は `_reorder_cache()` で行います。これは各過去キー‑バリューテンソルに対して `index_select` を呼び出し、新しいビーム順序に合わせてインデックス付けし直し、古いテンソルを破棄します。

CPU 上では単に行単位で型付き配列をスライスすれば済みますが、GPU テンソルの場合はもう少し面倒です。データを非同期でダウンロード (`ort_tensor.getData(true)`) し、再順序付けしてから再アップロードする必要があります。そこで `tensor.js` に `index_select`（同期、CPU 用）と `index_select_async`（非同期、GPU 用）を追加し、両方のパスに対応させました。

エンコーダ‑デコーダモデル（T5、BART）には **2 つのキャッシュ** が存在します: エンコーダとデコーダです。エンコーダ側の PKV はデコード中に変化しないためそのまま通過させます。再順序付けが必要なのはデコーダ側の PKV だけです。この区別を誤ると、出力が極めて悪くなりますが、微妙に正しそうに見えるため、参照と比較しないと気づきにくいケースが多くあります。

## 多様ビームサーチ：楽しいやつ

多様ビームサーチは `diversity_penalty` を導入し、後続のビームグループが先行グループで既に選択されたトークンを選ばないようにします。直感としては、すべてのビームが同じ出力に収束してしまうと、仮説空間を十分に探索できていない、ということです。

実装上は、各デコードステップでグループを **順次** 処理する必要があります。各グループは自分のスコアを計算する前に、前のグループが何を選択したかを見る必要があるため、並列処理はできません。

最終的にたどり着いた構造は次の通りです：

```
for each step:
  token_counts = {}
  for each group in groups:
    extract this group's beams and logits
    for each token selected by previous groups:
      logits[token] -= diversity_penalty * token_counts[token]
    score candidates, select top 2×group_size
    group_scorer.process(...)
    record newly selected tokens into token_counts
```

ここでの順序依存は実際の問題です。これを並列化すると多様性の保証が失われます。私は一時的にバッチ化しようと考えましたが、そこは大きな間違いでした。

## `BeamHypotheses` 優先度キュー

ビームが `EOS` トークンに到達して `max_length` 未満で終了した場合、そのビームは「完了」したとみなされますが、単に破棄したりすぐに返したりしてはいけません。代わりに、`BeamHypotheses` と呼ばれる上限付き優先度キューに追加します。

このキューはバッチ内の各アイテムにつき最大 `num_beams` 個の完了シーケンスを保持し、次の式でスコア付けされます：

```
score = sum_logprobs / (length ^ length_penalty)
```

`length_penalty > 1.0` は長い出力を好み、`< 1.0` は短い出力を好みます。`early_stopping` フラグはビームが完了とみなされる条件を制御します。キューが満杯になった時点で完了とする (`true`)、`max_length` に達するまで決して完了しない (`"never"`)、あるいは残りのビームが最悪の完了仮説を上回る可能性がなくなった時点で完了とする (`false`) のいずれかです。

`false` のケースが興味深いです — これは、残りのスコア上限を考慮したときに、現在の最悪仮説を上回る可能性があるライブビームがまだ存在するかどうかを追跡する必要があります。これは、すでに良い仮説が得られている場合に `max_length` まで走らせるのを防ぐ剪定最適化です。

このロジックは `beam_search.js` に実装されます。新規ファイルで、全体で約 240 行です。また `BeamSearchScorer` をエクスポートし、バッチ全体で `BeamHypotheses` インスタンスを管理し、`finalize()` を処理します。

## Python 参照実装とのテスト

ここで扱う非自明な実装詳細はすべて、HuggingFace の `transformers` ライブラリにある Python 実装と対応しています。私はそれを強く参照しました。

追加したテストスイートは次をカバーします:

- エンコーダ‑デコーダ（T5）およびデコーダのみ（LLaMA 系）に対する標準ビームサーチ
- `num_beam_groups=2, diversity_penalty=0.5` を用いた多様ビームサーチ
- `do_sample=true, top_k=10` のビームサンプリング
- `num_return_sequences > 1` — 出力形状が `[N, seq_len]` になることを検証
- 非互換な組み合わせに対する正しい例外スロー: CFG + ビームサーチ、ストリーミング + ビームサーチ、`num_return_sequences > num_beams`

「正しい例外スロー」テストは過小評価されがちです。意図的な制限を文書化し、組み合わせ不可能な機能を同時に使用しようとしたときに、黙って誤った出力が得られるのを防ぎます。（開発中に CFG とビームサーチを組み合わせようとした経験があります。数式が合わず、今は例外が投げられます。）

## 未実装の項目


いくつか意図的に除外し、`throws` でマークした項目があります。

- **多様ビームサンプリング**（`num_beam_groups > 1` + `do_sample`）: ここでは数式が本格的に複雑になります。標準的な多様ビームサーチはグループ間で順次実行されますが、そこにサンプリングを加えると、確率的モードで多様性ペナルティをどのように適用するかを慎重に考える必要があります。実装は可能ですが、まだ手が付けられていません。
- **ストリーミング + ビームサーチ**: ストリーミングはトークンが生成されるたびに出力します。一方ビームサーチは、複数ステップ先まで進まないとどのシーケンスが最適か判断できません。この二者は根本的に張り合いがあります。現在のベストビームをストリーミングすることは可能ですが、別の機能となり、独自の設計課題が生じます。

## 誰も語らない部分：オープンソースのレイテンシ

コードは動作します。テストはすべて通ります。既存のテストスイートはきれいです。レビューに数か月も掛かっています。

大規模で人気のあるオープンソースプロジェクトでは、こういうことが普通です。Hugging Face のチームは高速でリリースを行い、課題キューは膨大です。コアの生成ループに触れる約1,000行の機能 PR は、非 trivial なレビュー作業を要します。コメントへの対応は迅速で、実際に目を通す際には真摯に関わってくれます。文句を言っているわけではなく、単に事実を記録しています。

大規模 OSS プロジェクトに貢献し、すぐにマージされることを期待しているなら、期待値を調整してください。数か月かかるのがこの規模の変更では普通です。その間、コードはフォーク上で常に動作し続けます。

## 実際に得られたもの

以前は持っていなかったいくつかの知見：

1. **ビームサーチの実際的なメンタルモデル** — 教科書的なものではなく、エッジケースを含むモデル。KV キャッシュが壊れる理由。二段階ソートが重要になる理由。`length_penalty` がスコアに実際に与える影響。

2. **JS における TypedArray 数学への理解が深まった** — CPU 上の TypedArray で `index_select` を実装するのは、ウェブコードでほとんど触れない低レベルな作業です。動作は問題ありませんが、JavaScript がそのために設計されたわけではないという感覚が得られます。

3. **Python 参考実装への敬意が新たに** — HuggingFace の `transformers` ライブラリは巨大で時に厄介ですが、ビームサーチのロジックは丁寧にコメントが付けられ、設計上の決定が明確に意図されています。実装すべき内容を把握する最速の手段は、まずこの実装を読むことでした。

4. **実際に流通しているパッチ** — たとえまだマージされていなくても、存在し、動作し、PR ブランチから利用できるだけで十分です。
