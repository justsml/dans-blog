## लड़ाई...?

### Promises बनाम Async/Await!

रुको, क्या यह एक लड़ाई है? यह सही होना चाहिए? अब हमें कॉलबैक्स के बारे में बात ही नहीं होती?

नहीं, यह कोई लड़ाई नहीं है। अंत में यह आपके टूलबॉक्स में एक और संभावित उपकरण है। लेकिन चूँकि `async`/`await` सभी Promise कार्यक्षमता (विशेष रूप से `Promise.all`, `.race`) को नहीं बदलता, इसे प्रतिस्थान के रूप में पेश करना भ्रामक है।

काफी प्रभावशाली लोग इस गलतफ़हमी को बढ़ावा दे रहे हैं कि `async`/`await` ही Promises का **[replacement](https://developers.google.com/web/fundamentals/primers/async-functions)** **[everyone's](https://hackernoon.com/6-reasons-why-javascripts-async-await-blows-promises-away-tutorial-c7ec10518dd9)** **[been](https://x.com/umaar/status/1045655069478334464)** **[waiting](http://2ality.com/2017/08/promise-try.html#why-not-just-use-async-functions)** **[for](https://dzone.com/articles/javascript-promises-and-why-asyncawait-wins-the-ba)**।

> **संकेत: नहीं, बिलकुल नहीं, और थोड़ा भी नहीं।**

VS Code में हालिया एक जोड़ ने इस पक्षपात को और बढ़ा दिया है। जैसा कि [@umaar](https://x.com/umaar) ने ट्वीट किया:

<blockquote class="twitter-tweet" data-lang="en"><p lang="en" dir="ltr">Visual Studio Code can now convert your long chains of Promise.then()&#39;s into async/await! 🎊 Works very well in both JavaScript and TypeScript files. .catch() is also correctly converted to try/catch ✅ <a href="https://t.co/xb39Lsp84V">pic.x.com/xb39Lsp84V</a></p>&mdash; Umar Hansa (@umaar) <a href="https://x.com/umaar/status/1045655069478334464?ref_src=twsrc%5Etfw">September 28, 2018</a></blockquote>

यदि आप Promises से नफ़रत करते हैं और इस रीफ़ैक्टरिंग फीचर को चाहते हैं, तो मैं आपका दोष नहीं लगाता।

मैं पहले Promises से नफ़रत करता था। आज, मैं पूरी तरह से बदल गया हूँ। **Promises शानदार हैं।** वे आपको **फ़ंक्शन कंपोज़िशन का लाभ उठाने** के लिए सक्षम/प्रोत्साहित कर सकते हैं।

मैं दो क्षेत्रों की सिफ़ारिश करता हूँ जिन्हें पहले फोकस करके आप अपनी Promise तकनीक को आगे बढ़ा सकते हैं।

<h2 id="rule-1">#1: Named Functions!</h2>

अपने अनाम मेथड्स को खत्म करें। **नामित फ़ंक्शन** का उपयोग करने से कोड आपके आवश्यकताओं की कविता जैसा पढ़ा जाता है।

आइए एक आम उदाहरण देखें:

`fetch` का उपयोग करके HTTP GET अनुरोध करना:

<!-- समाधान देखने से पहले, एक सामान्य "सिफ़ारिशित" कार्यान्वयन देखें: -->

### एंटी‑पैटर्न

```js
// ❌ अनाम इनलाइन फ़ंक्शन का उपयोग 💩
fetch(url)
  .then(response => response.status < 400
    ? response
    : Promise.reject(new Error('Request Failed: ' + response.ststus)))
  .then(response => response.text())
```

### समाधान: नामित मेथड्स

```js
// ✅ स्पष्टता उभरती है: नामित फ़ंक्शन
fetch(url)
  .then(checkResponse)
  .then(getText)


// पुन: उपयोग योग्य सामान्य‑उद्देश्य फ़ंक्शन
function checkResponse(response) {
  return response.status < 400
    ? response
    : Promise.reject(new Error('Request Failed: ' + response.ststus))
}
function getText(response) {
  return response.text()
}
```

> जैसे‑जैसे आपका कोड अधिक DRY होता है, इस दृष्टिकोण के लाभ स्पष्ट होते जाते हैं।

**अतिरिक्त संसाधन:** इस तकनीक का उपयोग करके मेरे **1 मिनट के वीडियो** देखें: [बेसिक लॉगिंग](https://youtu.be/xR_MZE1SIkk) और [एडवांस्ड डिबगिंग](https://youtu.be/P_tghqWj72M)।

<h2 id="rule-2">#2: एकल उद्देश्य (फ़ंक्शन)</h2>

यह _भ्रमित करने वाली सटीकता_ से सुनाई देता है: एकल उद्देश्य।

फिर भी यह बहुत ही व्यक्तिपरक, मनमाना, और कभी‑कभी तो निरर्थक भी होता है।

```js
// 1 point: the return & ternary are effectively a one-liner
function checkResponse(response) {
  return response.status < 400
    ? response
    : Promise.reject(new Error('Request Failed: ' + response.ststus))
}
// 1 point: the return & expression are also effectively a one-liner
function getText(response) {
  return response.text()
}
```

फ़ंक्शन के कोड को देखते हुए, प्रत्येक पंक्ति में यदि `if`, `return`, टर्नरी, `for`, `const`, `let`, `var`, `switch`, `while`, `[].map/filter/reduce/etc` में से कोई भी मौजूद हो तो 1 अंक जोड़ें। प्रत्येक निर्देश के लिए 1 अंक जोड़ें (खाली पंक्तियों को न गिनें)। कई चेन वाली अभिव्यक्तियों या मेथड कॉल्स को केवल 1 अंक माना जाएगा।

अरे, थोड़ा तकनीकी शब्दजाल हो गया।

दिलचस्प बात यह है कि अधिकांश डेवलपर्स रिपोर्ट करते हैं कि वे अपने कोड को **Single Purpose** करने में _काफी अच्छे_ हैं। यह असंबंधित नहीं है: वे खुद को बेहतरीन ड्राइवर भी मानते हैं!

आइए एक उदाहरण देखें जो (अत्यंत प्रतिभाशाली) [Jake Archibald](https://x.com/jaffathecake) ने Google Developers साइट पर अपने async/await लेख में दिखाया है (ध्यान दें: 2024, लिंक हटा दिया गया है)।

```js
// source: https://developers.google.com/web/fundamentals/primers/async-functions
function logInOrder(urls) {
  // fetch all the URLs
  const textPromises = urls.map(url => {
    return fetch(url).then(response => response.text());
  });

  // log them in order
  textPromises.reduce((chain, textPromise) => {
    return chain.then(() => textPromise)
      .then(text => console.log(text));
  }, Promise.resolve());
}
```

### Single Purpose?

मैं कहूँगा नहीं। `logInOrder` असल में क्या कर रहा है?

1. `urls` की सूची पर लूप चलाना  
2. प्रत्येक को इनलाइन HTTP GET में पास करना:  
   1. HTTP `fetch`  
   2. प्रतिक्रिया का टेक्स्ट बॉडी लौटाना  
3. प्रत्येक `textPromise` के बाद `.then(text => console.log(text))` जोड़ना  
   1. परिणाम क्रमबद्ध रूप से प्रिंट करना  

इस एकल फ़ंक्शन में 5 अनाम मेथड परिभाषित हैं। जैसा कि Jake ने भी बताया, `.reduce` बहुत जटिल है। कोड के विभिन्न हिस्सों में बारीकी से लिखे हुए मैकेनिज़्म हाथ से लिखना समझ नहीं आता। दूसरे शब्दों में, हम DOM निर्माण को अनंत `document.createElement()`, `element.setAttribute()` आदि से नहीं लिखते। इसके बजाय हम कई विकल्पों में से सबसे उपयुक्त टूल चुनते हैं: हेल्पर/यूटिलिटी फ़ंक्शन, लाइब्रेरी या फ्रेमवर्क।
