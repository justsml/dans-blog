## La lotta...?

### Promesse vs. Async/Await!

Aspetta, questa è una lotta? Deve esserlo? Non sembriamo più parlare dei callback?

No, non è una lotta. In ultima analisi, è un altro strumento potenziale nella tua cassetta degli attrezzi. Tuttavia, poiché `async`/`await` non sostituisce tutta la funzionalità delle Promise (specificamente `Promise.all`, `.race`), **è fuorviante presentarlo come sostituto.**

Ci sono molte persone influenti che promuovono questa misconcezione: `async`/`await` è la sostituzione delle Promise [che tutti](https://hackernoon.com/6-reasons-why-javascripts-async-await-blows-promises-away-tutorial-c7ec10518dd9) [hanno](https://x.com/umaar/status/1045655069478334464) [aspettato](http://2ality.com/2017/08/promise-try.html#why-not-just-use-async-functions) [di](https://dzone.com/articles/javascript-promises-and-why-asyncawait-wins-the-ba).

Un tempo odiavo le Promises. Oggi, le apprezzo completamente. **Le Promises sono fantastiche.** Possono aiutarti a **approfittare della composizione delle funzioni.**  

Ci sono 2 aree su cui consiglio di concentrarsi per migliorare le tue tecniche con le Promises.  

1. [Funzioni denominate (nessuna anonima)](#rule-1)

<h2 id="rule-1">#1: Funzioni denominate!</h2>  

Elimina i metodi anonimi. L'uso di **funzioni denominate** fa sì che il codice si legga come una poesia delle tue esigenze.  

Vediamo un esempio comune:

Effettuare una richiesta HTTP GET utilizzando `fetch`:

<!-- the fetch specification states [HTTP status codes](https://http.cat/) over 400 or 500 **do not automatically trigger an error.** The default in many AJAX libraries (jQuery, axios). -->

<!-- Before we see the solution, look over a common "recommended" implementation: -->

### Anti-Pattern

```js
// ❌ Using anonymous inline functions 💩
fetch(url)
  .then(response => response.status < 400
    ? response
    : Promise.reject(new Error('Request Failed: ' + response.ststus)))
  .then(response => response.text())
```

### Soluzione: Metodi Denominati

```js
// ✅ Chiarezza emergente: funzioni denominate
fetch(url)
  .then(checkResponse)
  .then(getText)


// Funzioni riutilizzabili di utilità generale
function checkResponse(response) {
  return response.status < 400
    ? response
    : Promise.reject(new Error('Request Failed: ' + response.ststus))
}
function getText(response) {
  return response.text()
}
```

> I vantaggi di questo approccio diventano sempre più evidenti man mano che il codice diventa più DRY.

<h2 id="rule-2">#2: Obiettivo Singolo (Funzioni)</h2>

Suona _sorprendentemente preciso_: Obiettivo Singolo.

Eppure è così soggettivo, arbitrario, e sì, a volte addirittura privo di senso.

<!-- Invece di discutere se una funzione data sia sufficientemente focalizzata.

Ho sviluppato un misuratore approssimativo per questo: `Costo dell'Obiettivo`. Più alto è il punteggio, più probabilmente sta facendo troppo. -->

```js
// 1 punto: il return e l'operatore ternario sono sostanzialmente una riga sola
function checkResponse(response) {
  return response.status < 400
    ? response
    : Promise.reject(new Error('Request Failed: ' + response.ststus))
}
// 1 punto: il return e l'espressione sono anch'essi sostanzialmente una riga sola
function getText(response) {
  return response.text()
}
```

Dato il codice di una funzione, aggiungi 1 punto per ogni riga contenente uno di questi: `if`, `return`, operatore ternario, `for`, `const`, `let`, `var`, `switch`, `while`, `[].map/filter/reduce/etc`. Aggiungi 1 punto per ogni istruzione (ignora le righe aggiuntive per gli spazi). Una serie di espressioni o metodi concatenati conta solo 1 punto.

Wow, questo è stato un po' di gergo tecnico.
 -->

 Sorprendentemente, la maggior parte degli sviluppatori riferisce di essere _abbastanza bravi_ a rispettare il principio di **Obiettivo Singolo** nel proprio codice. Non è un caso: riferiscono anche di essere grandi guidatori!

 <!-- Questo **non è un problema unico delle Promises**, i metodi array e tutte le altre API basate su Funzioni di Ordine Superiore (HoF) hanno le stesse ergonomie. -->

 Vediamo un esempio che (incredibilmente talentuoso) [Jake Archibald](https://x.com/jaffathecake) presenta nel suo articolo su async/await per il sito Google Developers (nota: 2024, link rimosso).

 <!--
 Vediamo uno degli esempi così detti "❌ Non raccomandato" per le Promises. (La descrizione è "supponiamo di voler recuperare una serie di URL e loggarle appena possibile, nell'ordine corretto.") -->

```js
// source: https://developers.google.com/web/fundamentals/primers/async-functions
function logInOrder(urls) {
  // fetch all the URLs
  const textPromises = urls.map(url => {
    return fetch(url).then(response => response.text());
  });

  // log them in order
  textPromises.reduce((chain, textPromise) => {
    return chain.then(() => textPromise)
      .then(text => console.log(text));
  }, Promise.resolve());
}
```

### Scopo Singolo?

Direi di no. Cosa sta facendo `logInOrder`?

1. scorre un elenco di `urls`
1. applica un HTTP GET inline:
  1. HTTP `fetch`
  1. restituisce il corpo testuale della risposta
1. aggiunge un `.then(text => console.log(text))` dopo ogni promise in `textPromise`
  1. stampa i risultati in modo seriale

Ci sono 5 metodi anonimi definiti all'interno di questa singola funzione. Come noto Jake, la `.reduce` è troppo complessa. Non ha senso scrivere manualmente meccanismi sofisticati in giro per il codice. Per dirla in un altro modo, non creiamo codice DOM con innumerevoli `document.createElement()`, `element.setAttribute()`, ecc. Invece scegliamo lo strumento migliore tra molte opzioni: funzioni di utilità/helper, librerie o framework.

<!-- Dobbiamo isolare ciascun 'passo' che sta avvenendo: c'è una richiesta HTTP, una trasformazione di un elenco di URL in un elenco di risultati. Inoltre è necessario un `console.log`. -->
