## Бой…?

### Promises vs. Async/Await!

Подождите, это же бой? Должно быть, да? Мы же больше не говорим о колбэках?

Нет, это не бой. В конечном счёте — ещё один потенциальный инструмент в вашем арсенале. Однако, поскольку `async`/`await` не заменяет всю функциональность Promise (в частности `Promise.all`, `.race`), **представлять его как полную замену вводит в заблуждение**.

Много влиятельных людей распространяют это заблуждение, будто `async`/`await` — это **замена** Promise — [для всех](https://developers.google.com/web/fundamentals/primers/async-functions) — [ожидаемая](https://hackernoon.com/6-reasons-why-javascripts-async-await-blows-promises-away-tutorial-c7ec10518dd9) — [долгожданная](https://x.com/umaar/status/1045655069478334464) — [ожидание](http://2ality.com/2017/08/promise-try.html#why-not-just-use-async-functions) — [для](https://dzone.com/articles/javascript-promises-and-why-asyncawait-wins-the-ba).

> **Подсказка: Нет, ни чуть‑чуть, и даже не намёк.**

Раньше я терпеть не мог Promises. Сейчас я полностью изменил своё мнение. **Promises — это потрясающе.** Они позволяют/поощряют **использовать композицию функций**.

Есть два направления, на которых я советую сосредоточиться в первую очередь, чтобы улучшить технику работы с Promise.

1. [Именованные функции (без анонимных)](#rule-1)
2. [Функции с одной задачей](#rule-2)

<h2 id="rule-1">#1: Именованные функции!</h2>

Убирайте анонимные методы. **Именованные функции** делают код читаемым, как поэма о ваших требованиях.

Рассмотрим типичный пример:

Сделаем HTTP‑GET запрос с помощью `fetch`:

<!-- спецификация fetch гласит, что [HTTP‑коды статуса](https://http.cat/) 400 и выше **не вызывают автоматическую ошибку**. По умолчанию многие AJAX‑библиотеки (jQuery, axios) делают иначе. -->

<!-- Прежде чем посмотреть решение, рассмотрим типичную «рекомендованную» реализацию: -->

### Антипаттерн

```js
// ❌ Используем анонимные встроенные функции 💩
fetch(url)
  .then(response => response.status < 400
    ? response
    : Promise.reject(new Error('Request Failed: ' + response.ststus)))
  .then(response => response.text())
```

### Решение: Именованные функции

```js
// ✅ Ясность появляется: именованные функции
fetch(url)
  .then(checkResponse)
  .then(getText)


// Переиспользуемые функции общего назначения
function checkResponse(response) {
  return response.status < 400
    ? response
    : Promise.reject(new Error('Request Failed: ' + response.ststus))
}
function getText(response) {
  return response.text()
}
```

> Преимущества такого подхода становятся всё более очевидными по мере того, как ваш код становится более DRY.

<h2 id="rule-2">#2: Одно назначение (функции)</h2>

Это звучит _обманчиво точно_: Одно назначение.

Но это настолько субъективно, произвольно и, порой, даже бессмысленно.

```js
// 1 point: the return & ternary are effectively a one-liner
function checkResponse(response) {
  return response.status < 400
    ? response
    : Promise.reject(new Error('Request Failed: ' + response.ststus))
}
// 1 point: the return & expression are also effectively a one-liner
function getText(response) {
  return response.text()
}
```

По коду функции добавляйте 1 балл за каждую строку, содержащую любой из операторов: `if`, `return`, тернарный оператор, `for`, `const`, `let`, `var`, `switch`, `while`, `[].map/filter/reduce/etc`. Добавляйте ещё 1 балл за каждую инструкцию (пробелы и пустые строки не учитываются). Последовательные вызовы методов считаются одним баллом.

Уф, это было немного жаргона.  
-->

Интересно, что большинство разработчиков считают себя _довольно крутыми_ в применении принципа **Single Purpose** к своему коду. Не без связи: они также утверждают, что отличные водители!

<!-- This **isn't a unique issue with Promises**, array methods and all other HoF-based (Higher Order Function) APIs have the same ergonomics. -->

Посмотрим пример, который (невероятно талантливый) [Jake Archibald](https://x.com/jaffathecake) привёл в своей статье об async/await для сайта Google Developers (примечание: 2024 год, ссылка удалена).

```js
// source: https://developers.google.com/web/fundamentals/primers/async-functions
function logInOrder(urls) {
  // fetch all the URLs
  const textPromises = urls.map(url => {
    return fetch(url).then(response => response.text());
  });

  // log them in order
  textPromises.reduce((chain, textPromise) => {
    return chain.then(() => textPromise)
      .then(text => console.log(text));
  }, Promise.resolve());
}
```

### Одна цель?

Скажу сразу — нет. Что делает `logInOrder`?

1. перебирает список `urls`
2. применяет к каждому встроенный HTTP‑GET:
   1. `fetch`
   2. возвращает тело ответа как текст
3. после каждого промиса в `textPromise` добавляет `.then(text => console.log(text))`
   1. выводит результаты последовательно

В этой единственной функции объявлено пять анонимных функций. Как отмечает Джейк, использование `.reduce` здесь избыточно. Нет смысла вручную писать такие тонкие механизмы по всему коду. Аналогично, мы не пишем создание DOM‑элементов через бесконечные `document.createElement()`, `element.setAttribute()` и т.п. Вместо этого выбираем лучший инструмент из доступных вариантов: вспомогательные функции, библиотеки или фреймворки.
