## Der Kampf…?

### Promises vs. Async/Await!

Moment, ist das ein Kampf? Das muss doch stimmen, oder? Wir reden doch nicht mehr über Callbacks?

Nein, es ist kein Kampf. Letztlich ist es ein weiteres potenzielles Werkzeug in Ihrem Arsenal. Da `async`/`await` jedoch nicht die gesamte Promise‑Funktionalität ersetzt (insbesondere `Promise.all`, `.race`) **ist es irreführend, es als Ersatz darzustellen.**

Zahlreiche einflussreiche Personen verbreiten dieses Missverständnis, dass `async`/`await` die Promises‑[Ersatz](https://developers.google.com/web/fundamentals/primers/async-functions) [sei, die jeder](https://hackernoon.com/6-reasons-why-javascripts-async-await-blows-promises-away-tutorial-c7ec10518dd9) [wartet](https://x.com/umaar/status/1045655069478334464) [hatte](http://2ality.com/2017/08/promise-try.html#why-not-just-use-async-functions) [zu](https://dzone.com/articles/javascript-promises-and-why-asyncawait-wins-the-ba).

<h2 id="rule-1">#1: Benannte Funktionen!</h2>

Töten Sie Ihre anonymen Methoden. Der Einsatz **benannter Funktionen** lässt den Code wie Poesie Ihrer Anforderungen lesen.

Betrachten wir ein gängiges Beispiel:

Making an HTTP GET request using `fetch`:

<!-- Bevor wir die Lösung sehen, schauen wir uns eine gängige „empfohlene“ Implementierung an: -->

### Anti‑Pattern

```js
// ❌ Using anonymous inline functions 💩
fetch(url)
  .then(response => response.status < 400
    ? response
    : Promise.reject(new Error('Request Failed: ' + response.ststus)))
  .then(response => response.text())
```

### Lösung: Benannte Methoden

```js
// ✅ Clarity emerges: named functions
fetch(url)
  .then(checkResponse)
  .then(getText)


// Reusable general-purpose functions
function checkResponse(response) {
  return response.status < 400
    ? response
    : Promise.reject(new Error('Request Failed: ' + response.ststus))
}
function getText(response) {
  return response.text()
}
```

> Die Vorteile dieses Ansatzes werden immer deutlicher, je mehr DRY‑Code Sie erhalten.

**Zusätzliche Ressourcen:** Schauen Sie sich meine **1‑Minute‑Videos** zu [basic logging](https://youtu.be/xR_MZE1SIkk) und [advanced debugging](https://youtu.be/P_tghqWj72M) an, die diese Technik demonstrieren.

<h2 id="rule-2">#2: Single Purpose (Funktionen)</h2>

Es klingt _verführerisch präzise_: Single Purpose.

Doch ist es so subjektiv, willkürlich und, ja, manchmal sogar bedeutungslos.

```js
// 1 Punkt: return & ternary sind effektiv ein Einzeiler
function checkResponse(response) {
  return response.status < 400
    ? response
    : Promise.reject(new Error('Request Failed: ' + response.ststus))
}
// 1 Punkt: return & Ausdruck sind ebenfalls ein Einzeiler
function getText(response) {
  return response.text()
}
```

Für den Code einer Funktion einen Punkt hinzufügen, wenn eine Zeile eines der Schlüsselwörter enthält: `if`, `return`, ternary, `for`, `const`, `let`, `var`, `switch`, `while`, `[].map/filter/reduce/etc`. Zusätzlich für jede Anweisung einen Punkt zählen (Leerzeilen ignorieren). Mehrere verkettete Aufrufe oder Methoden zählen nur als ein Punkt.

Puh, das war ein bisschen Fachjargon.

Interessanterweise geben die meisten Entwickler an, dass sie beim **Single‑Purpose**‑Gestalten ihres Codes _ziemlich gut_ sind. Nicht ohne Zusammenhang: Sie behaupten auch, exzellente Autofahrer zu sein!

Lassen Sie uns ein Beispiel ansehen, das der (außerordentlich talentierte) [Jake Archibald](https://x.com/jaffathecake) in seinem async/await‑Artikel für die Google‑Developers‑Seite präsentiert (Hinweis: 2024, Link entfernt).

```js
// source: https://developers.google.com/web/fundamentals/primers/async-functions
function logInOrder(urls) {
  // fetch all the URLs
  const textPromises = urls.map(url => {
    return fetch(url).then(response => response.text());
  });

  // log them in order
  textPromises.reduce((chain, textPromise) => {
    return chain.then(() => textPromise)
      .then(text => console.log(text));
  }, Promise.resolve());
}
```

### Single Purpose?

Ich würde sagen, nein. Was macht `logInOrder`?

1. Durchläuft eine Liste von `urls`
1. Wendet darauf einen Inline‑HTTP‑GET an:
  1. HTTP `fetch`
  1. Gibt den Antwort‑Textkörper zurück
1. Hängt ein `.then(text => console.log(text))` an jedes Promise in `textPromise` an
  1. Gibt die Ergebnisse seriell aus

In dieser einzigen Funktion werden fünf anonyme Methoden definiert. Wie Jake sogar betont, ist das `.reduce` zu komplex. Es macht keinen Sinn, nuancierte Mechanismen überall im Code von Hand zu schreiben. Anders gesagt, wir schreiben keinen DOM‑Erstellungscode mit endlosen `document.createElement()`, `element.setAttribute()` usw. Stattdessen wählen wir das beste Werkzeug aus vielen Optionen: Hilfs‑/Utility‑Funktionen, Bibliotheken oder Frameworks.
