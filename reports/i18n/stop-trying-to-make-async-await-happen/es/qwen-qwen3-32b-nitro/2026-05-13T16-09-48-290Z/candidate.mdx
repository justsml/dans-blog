## ¿La pelea...?

### ¡Promesas vs. Async/Await!

¿Espera, ¿esta es una pelea? ¿Debe serlo? Ya no hablamos de los callbacks, ¿verdad?

No, no es una pelea. Al final, es otra herramienta potencial en tu caja de herramientas. Sin embargo, dado que `async`/`await` no reemplaza todas las funcionalidades de las Promesas (específicamente `Promise.all`, `.race`) **es engañoso presentarla como un reemplazo.**

Hay muchas personas influyentes promoviendo esta falsa creencia de que `async`/`await` es el [reemplazo](https://developers.google.com/web/fundamentals/primers/async-functions) [de](https://hackernoon.com/6-reasons-why-javascripts-async-await-blows-promises-away-tutorial-c7ec10518dd9) [todas](https://x.com/umaar/status/1045655069478334464) [las](http://2ality.com/2017/08/promise-try.html#why-not-just-use-async-functions) [Promesas](https://dzone.com/articles/javascript-promises-and-why-asyncawait-wins-the-ba).

<h2 id="rule-1">#1: ¡Funciones con nombre!</h2>  

Elimina tus métodos anónimos. Usar **funciones con nombre** hace que el código se lea como una poesía de tus requisitos.  

Veamos un ejemplo común:

Realizando una solicitud HTTP GET usando `fetch`:

<!-- la especificación de fetch establece que los [códigos de estado HTTP](https://http.cat/) mayores a 400 o 500 **no desencadenan automáticamente un error.** El comportamiento predeterminado en muchas bibliotecas AJAX (jQuery, axios). -->

<!-- Antes de ver la solución, echa un vistazo a una implementación "recomendada" común: -->

### Patrón anti-idioma

```js
// ❌ Usando funciones anónimas inline 💩
fetch(url)
  .then(response => response.status < 400
    ? response
    : Promise.reject(new Error('Request Failed: ' + response.ststus)))
  .then(response => response.text())
```

### Solución: Métodos nombrados

```js
// ✅ Emerge la claridad: funciones con nombre
fetch(url)
  .then(checkResponse)
  .then(getText)


// Funciones reutilizables de propósito general
function checkResponse(response) {
  return response.status < 400
    ? response
    : Promise.reject(new Error('Request Failed: ' + response.ststus))
}
function getText(response) {
  return response.text()
}
```

> Los beneficios de este enfoque se vuelven cada vez más evidentes a medida que obtienes un código más DRY.

**Recursos adicionales:** Mira mis **videos de 1 minuto** sobre [registro básico](https://youtu.be/xR_MZE1SIkk) y [depuración avanzada](https://youtu.be/P_tghqWj72M) usando esta técnica.

<h2 id="rule-2">#2: Propósito único (Funciones)</h2>

Suena _engañosamente precisa_: Propósito único.

Y sin embargo, es tan subjetivo, arbitrario, y ciertamente, a veces incluso sin sentido.

<!-- En lugar de debatir si una función dada es suficientemente enfocada.

Desarrollé una medida aproximada para esto: `Purpose Cost`. Cuanto más alto sea el puntaje, más probable es que esté haciendo demasiado. -->

```js
// 1 punto: el return y el ternario son efectivamente una línea
function checkResponse(response) {
  return response.status < 400
    ? response
    : Promise.reject(new Error('Request Failed: ' + response.ststus))
}
// 1 punto: el return y la expresión también son efectivamente una línea
function getText(response) {
  return response.text()
}
```

Dado el código de una función, suma 1 punto por cada línea que contenga alguno de estos: `if`, `return`, ternario, `for`, `const`, `let`, `var`, `switch`, `while`, `[].map/filter/reduce/etc`. Suma 1 punto por cada instrucción (ignora líneas extra por espacios). Una serie de expresiones o métodos encadenados cuentan como 1 punto.

Vaya, eso fue un poco de jerga.
 -->

Curiosamente, la mayoría de los desarrolladores reportan que son _bastante buenos_ en **Propósito Único** en su código. No es casualidad: también reportan ser buenos conductores.

<!-- Este **no es un problema único con Promises**, los métodos de arrays y todas las demás APIs basadas en HoF (Higher Order Functions) tienen las mismas ergonomías. -->

Veamos un ejemplo del (incrediblemente talentoso) [Jake Archibald](https://x.com/jaffathecake) en su artículo sobre `async/await` para el sitio de Google Developers (nota: 2024, enlace eliminado).

<!--
Veamos uno de los llamados ejemplos "❌ No recomendado" de Promises. (La descripción es "digamos que queremos obtener una serie de URLs y registrarlas lo antes posible, en el orden correcto.") -->
---

```js
// fuente: https://developers.google.com/web/fundamentals/primers/async-functions
function logInOrder(urls) {
  // obtener todas las URLs
  const textPromises = urls.map(url => {
    return fetch(url).then(response => response.text());
  });

  // registrarlas en orden
  textPromises.reduce((chain, textPromise) => {
    return chain.then(() => textPromise)
      .then(text => console.log(text));
  }, Promise.resolve());
}
```

### ¿Propósito único?

Diría que no. ¿Qué está haciendo `logInOrder`?

1. Iterar sobre una lista de `urls`
1. Aplicar un HTTP GET inline:
  1. HTTP `fetch`
  1. Devolver el cuerpo de texto de la respuesta
1. Añadir un `.then(text => console.log(text))` después de cada promesa en `textPromise`
  1. Imprimir resultados de forma serial

Hay 5 métodos anónimos definidos en esta única función. Como Jake señala, la reducción con `.reduce` es demasiado compleja. No tiene sentido escribir manualmente mecanismos detallados por todo el código. Para aclararlo: no escribimos código de creación de DOM con una infinidad de `document.createElement()`, `element.setAttribute()`, etc. En su lugar, elegimos la mejor herramienta entre muchas opciones: funciones auxiliares/útiles, bibliotecas o marcos.

<!-- Necesitamos aislar cada 'paso' que está sucediendo: hay una solicitud HTTP, una transformación de una lista de URLs en una lista de resultados. También se necesita un `console.log`. -->

<!-- > 🤔 ¿Por qué los `Promises` hacen que los desarrolladores abandonen prácticas que usan en otros lugares? -->

**Nota:** Si la intención era _iniciar solicitudes_ en secuencia, en lugar de solo imprimir los resultados en orden, este código no lo hace realmente. Procederemos a refactorizarlo de manera adecuada.
