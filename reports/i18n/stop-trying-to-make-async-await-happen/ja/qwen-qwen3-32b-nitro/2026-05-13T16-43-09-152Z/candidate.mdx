## 争い…？  

### Promise と Async/Await の争い！  

待って、これは争いなの？ そうだろう？ コールバックについて話すことはもうあまりない気がする。

いいえ、これは争いではありません。最終的にはあなたのツールボックスに追加される別の可能性のあるツールです。ただし、`async`/`await`はすべてのPromise機能（具体的には`Promise.all`、`.race`）を置き換えないため、**これを置き換えとして提示することは誤解を招きます**。

多くの影響力のある人物がこの誤解を広めています。`async`/`await`がPromiseの[置き換え](https://developers.google.com/web/fundamentals/primers/async-functions) [誰もが](https://hackernoon.com/6-reasons-why-javascripts-async-await-blows-promises-away-tutorial-c7ec10518dd9) [待ち続けてきた](https://x.com/umaar/status/1045655069478334464) [もの](http://2ality.com/2017/08/promise-try.html#why-not-just-use-async-functions) [だ](https://dzone.com/articles/javascript-promises-and-why-asyncawait-wins-the-ba)。

かつてPromiseを嫌っていた。今では完全に立場を変えており、Promiseは素晴らしいと思う。関数合成を活用するための手段として、**関数合成を活かす**ことを奨励する。

1. [名前付き関数（匿名関数は使用しない）](#rule-1)

<h2 id="rule-1">#1: 名前付き関数！</h2>  

匿名メソッドはやめろ。**名前付き関数**を用いることで、コードが要件の詩のように読めるようになる。  

一般的な例を見てみよう：

HTTP GETリクエストを`fetch`で実行する：

<!-- fetchの仕様では、[HTTPステータスコード](https://http.cat/)が400または500を超える場合、**自動的にエラーが発生しない**と定義されています。多くのAJAXライブラリ（jQuery、axios）のデフォルト動作とは異なります。 -->

<!-- 解決策を見る前に、一般的な「推奨される」実装例を確認してください： -->

### 非推奨パターン

```js
// ❌ 匿名のインライン関数を使用 💩
fetch(url)
  .then(response => response.status < 400
    ? response
    : Promise.reject(new Error('Request Failed: ' + response.ststus)))
  .then(response => response.text())
```

### 解決策：名前付きメソッド

```js
// ✅ 明確さが生まれる：名前付き関数
fetch(url)
  .then(checkResponse)
  .then(getText)


// 再利用可能な汎用関数
function checkResponse(response) {
  return response.status < 400
    ? response
    : Promise.reject(new Error('Request Failed: ' + response.ststus))
}
function getText(response) {
  return response.text()
}
```

> このアプローチの利点は、コードがDRY（Don't Repeat Yourself）になるにつれてますます明らかになります。

**追加資料:** この手法を使った[基本的なログ出力](https://youtu.be/xR_MZE1SIkk)と[高度なデバッグ](https://youtu.be/P_tghqWj72M)の**1分間の動画**をご覧ください。

<h2 id="rule-2">#2: 単一目的（関数）</h2>

これは_欺瞞的なほど明確に聞こえる_：単一目的。

しかし実際には主観的で恣意的であり、たしかに時折無意味にさえなる。

<!-- ある関数が十分に焦点を持っているかを議論する代わりに -->

この概念に対する粗い測定基準を考案しました：`目的コスト`。スコアが高ければ高いほど、何かしら多くのことをやっている可能性が高いです。

```js
// 1ポイント：return と三項演算子は実質的に1行の処理
function checkResponse(response) {
  return response.status < 400
    ? response
    : Promise.reject(new Error('Request Failed: ' + response.ststus))
}
// 1ポイント：return と式も実質的に1行の処理
function getText(response) {
  return response.text()
}
```

関数のコードに対して、以下のいずれかを含む行ごとに1ポイント加算します：`if`、`return`、三項演算子、`for`、`const`、`let`、`var`、`switch`、`while`、`[].map/filter/reduce/etc`。各命令ごとに1ポイント加算します（空白行を除く）。連鎖した式やメソッドは1つのポイントとしてカウントされます。

うーむ、ちょっと専門用語が多くなりましたね。
 -->

興味深いことに、多くの開発者は自分が**単一目的**のコードを書くのが「かなりうまい」と主張しています。これは偶然ではありません。彼らは自分たちが優れた運転手であるとも主張しています！

Google Developersサイトのasync/await記事（注：2024年、リンクは削除）で、（非常に才能ある）[Jake Archibald](https://x.com/jaffathecake)が紹介している例を見てみましょう。

<!--
「❌ 推奨されない」Promiseの例の1つを見てみましょう。（説明は「複数のURLを取得し、可能な限り早く正しい順序でログ出力したい」となっています。） -->

```js
// source: https://developers.google.com/web/fundamentals/primers/async-functions
function logInOrder(urls) {
  // fetch all the URLs
  const textPromises = urls.map(url => {
    return fetch(url).then(response => response.text());
  });

  // log them in order
  textPromises.reduce((chain, textPromise) => {
    return chain.then(() => textPromise)
      .then(text => console.log(text));
  }, Promise.resolve());
}
```

### 単一の目的？

いいえ。`logInOrder`は具体的に何を行っていますか？

1. `urls`リストをループ処理
1. 各URLにHTTP GETを適用:
  1. HTTP `fetch`の実行
  1. 応答テキストボディの返却
1. `textPromise`内の各Promiseに`.then(text => console.log(text))`を追加
  1. 結果をシリアルに印刷

この単一の関数内に5つの無名メソッドが定義されています。Jakeが指摘しているように、`.reduce`は複雑すぎます。コード中にあらゆる場所で個別に複雑なメカニズムを手で書くのは適切ではありません。言い換えると、DOM生成コードを無限に`document.createElement()`、`element.setAttribute()`と手書きする代わりに、ヘルパー/ユーティリティ関数、ライブラリ、またはフレームワークといった複数のオプションの中から最適なツールを選択します。
