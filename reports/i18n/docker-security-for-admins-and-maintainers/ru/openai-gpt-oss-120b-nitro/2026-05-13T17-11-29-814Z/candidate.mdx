## ⚠️ Локальные сети под угрозой

Будем откровенны: все мы так делали. Вы подключались к случайному Wi‑Fi в кафе или позволяли кому‑то пользоваться вашей домашней сетью, не задумываясь ни секунды. Может, вы даже доверяете своему умному холодильнику, что он не скомпрометирует сеть. Реальность такова: такие беззаботные решения могут открыть ваш локальный набор разработки ненужным рискам. Злоумышленники не ограничиваются только продакшн‑системами — локальные среды часто являются более мягкой мишенью, предоставляя путь к конфиденциальным проектам.

### Сценарии атак

1. **Перехваченный трафик:** Незашифрованный трафик легко захватывается и читается.  
2. **Незащищённые сервисы:** Локальные базы данных или API, открытые на `0.0.0.0`.  
3. **Подмена сети:** Перенаправление трафика на устройство атакующего.

### Быстрые исправления

- Предпочитайте частные Docker‑сети вместо межсетевых экранов, чтобы ограничить экспозицию сети.  
- Избегайте публичных или общих Wi‑Fi; лучше используйте точку доступа вашего телефона.  
- Следите за локальной сетью на предмет неизвестных устройств с помощью таких инструментов, как `arp-scan` и `nmap`.

## 🛡️ Конфигурация межсетевого экрана

### UFW с Docker (Ubuntu)

> ⚠️ **Внимание:** По умолчанию Docker в Ubuntu/Debian обходит правила UFW/iptables, что может открыть вашу систему для атак.  
> Не имеет значения, привязываете ли вы порты к локальным IP‑адресам (например, `-p 127.0.0.1:8080:80`).

Это удивляет меня каждый раз, когда я узнаю об этом! [Docker по умолчанию обходит правила UFW](https://github.com/moby/moby/issues/4737), позволяя контейнерам общаться с хостом и другими контейнерами без ограничений.

### Лучший подход

1. 🥇 **Используйте сети Docker** для изоляции и контроля того, к чему может подключаться каждый контейнер или сеть.

###
2. 🥉 **Обновите iptables**: если вам необходимо использовать сеть `host` или вы не можете применять пользовательские сети, риск можно снизить, настроив iptables. Не для слабонервных, [см. утилиту ниже.](../#uf)

#### Изоляция сети Docker

```bash
# Создать новую сеть Docker
docker network create my-network

# Запустить контейнер в этой сети
docker run --network my-network my-container
```

#### Конфигурация UFW (для сетей `host`)

Существует множество плохих советов по исправлению этой проблемы. Настройте UFW для работы с Docker так, как вы обычно ожидаете.

Я использовал `ufw-docker` для настройки собственного сервера, и он, кажется, работает стабильно.

```bash title="install-ufw-docker.sh"
# Установить бинарник от имени root (всё равно нужны root‑права)
sudo wget -O /usr/local/bin/ufw-docker \
   https://github.com/chaifeng/ufw-docker/raw/master/ufw-docker
sudo chmod +x /usr/local/bin/ufw-docker
# Установить и изменить файл `after.rules` для `ufw`
ufw-docker install

ufw-docker help
```

```

Эта команда выполняет следующее:

- Делает резервную копию файла `/etc/ufw/after.rules`.
- Добавляет правила, связанные с Docker, в конец файла, чтобы они корректно интегрировались с UFW.

**Источник:** [ufw-docker GitHub](https://github.com/chaifeng/ufw-docker/tree/master#install)

**Пример использования:**

```bash

# Разрешить Docker‑контейнеру порт 8080
ufw-docker allow <container_name> 8080/tcp

# Управлять правилами безопасно вместе с вашей конфигурацией UFW
ufw-docker status

```

**Примечание:** Большинство «исправлений» конфликтов Docker‑UFW требуют ручного управления iptables, что может приводить к ошибкам и ломаться при обновлениях.

### Брандмауэр macOS

1. Откройте **Системные настройки > Безопасность и конфиденциальность > Брандмауэр**.
2. Включите брандмауэр и нажмите «Параметры брандмауэра».
3. Блокируйте все входящие соединения, кроме необходимых сервисов.

**Примечание:** Возможно, понадобится настроить ваш брандмауэр, чтобы разрешить работу некоторых умных устройств — например Google Cast/AirPlay и другие сервисы.

### Команды для продвинутых пользователей (macOS и Linux)

#### macOS:

```bash

sudo /usr/libexec/ApplicationFirewall/socketfilterfw --setblockall on  # Блокировать всё
sudo /usr/libexec/ApplicationFirewall/socketfilterfw --add /path/to/app  # Разрешить конкретное приложение

```

#### Linux (ufw):

```bash

ufw default deny incoming  # Блокировать все входящие
ufw allow ssh  # Разрешить SSH
# разрешить 443 и 80 для веб‑трафика
ufw allow 80/tcp
ufw allow 443/tcp
ufw enable  # Включить брандмауэр

```

**Pro Tip:** Используйте такие инструменты, как [Little Snitch](https://www.obdev.at/products/littlesnitch/index.html) на macOS и [ufw](https://help.ubuntu.com/community/UFW) на Linux для более удобных настроек.

##🔐 Управление секретами для локальной разработки

### Проактивная проверка заполнителей

<p>💡 Убедитесь, что секреты заданы реальными значениями перед запуском приложения.</p>

Если вы используете заполнители вроде `__WARNING_REPLACE_ME__` в своих секретах, здорово — возможно, кто‑то их заметит. На всякий случай можно добавить небольшую проверку, чтобы обеспечить безопасность во время выполнения.

Вы бы не поверили, насколько просто полностью взломать (модифицировать и пере‑подписать) JWT‑токен, когда злоумышленники могут угадать секрет!

<CodeTabs client:load tabs={["JavaScript", "Rust", "Go"]}>

```javascript
// validateSecrets.js
const validateSecrets = () => {
  const unsafePlaceholder = /__WARNING_REPLACE_ME__/;
  const missingSecrets = Object.entries(process.env).filter(
    ([key, value]) => unsafePlaceholder.test(value)
  );

  if (missingSecrets.length) {
    console.error("Unsafe secrets detected:", missingSecrets);
    process.exit(1);
  }
};

validateSecrets();
```

```rust
// validate_secrets.rs
use std::env;

fn validate_secrets() {
    let unsafe_placeholder = "__WARNING_REPLACE_ME__";
    for (key, value) in env::vars() {
        if value.contains(unsafe_placeholder) {
            panic!("Unsafe secret in {}", key);
        }
    }
}

fn main() {
    validate_secrets();
}
```

```go
// validate_secrets.go
package main

import (
	"fmt"
	"os"
	"strings"
)

func validateSecrets() {
	placeholder := "__WARNING_REPLACE_ME__"
	for _, env := range os.Environ() {
		pair := strings.SplitN(env, "=", 2)
		if len(pair) == 2 && strings.Contains(pair[1], placeholder) {
			panic(fmt.Sprintf("Unsafe secret in %s", pair[0]))
		}
	}
}

func main() {
	validateSecrets()
}
```

### Генерация и хранение секретов

<p class="inset">Никогда не вшивайте секреты в кодовую базу. Предпочитайте переменные окружения и защищённые хранилища.</p>


Вместо `.env.example` используйте `.env.generate.sh`, чтобы упростить пользователям получение файла `.env` с безопасными «значениями по умолчанию».

#### Пример `.env.generate.sh`

```bash title=".env.generate.sh" frame="code"
#!/bin/bash
# Generates a secure .env file for local development

generate_secret() {
    local length=${1:-30}
    # add 4 bytes to account for padding
    local generate_length=$((length + 4))
    openssl rand -base64 "$generate_length" | tr -d '+=/\n' | cut -c1-"$length"
}
# Bail out if .env file already exists
[ -f .env ] && { echo ".env file already exists!"; exit 1; }

cat <<EOL > .env
# Database settings & secrets
DB_USER=app_user
DB_PASSWORD=$(generate_secret 30)
REDIS_PASSWORD=$(generate_secret 20)
# Session secrets
SESSION_KEY=$(generate_secret 32)
JWT_SECRET=$(generate_secret 64)
EOL

echo "New .env file generated!"
```

## 🕵️‍ Monitoring & Double-checking

### Примеры `nmap`

#### Тестирование внутри вашей сети

```bash

# Scan your localhost for all open ports
nmap -sT localhost

# Scan your machine’s private IP for services
nmap -sV 192.168.1.10

# Detect devices on your network
nmap -sn 192.168.0.0/24
nmap -sn 10.0.0.0/24
```

```


#### Тестирование за пределами вашей сети

Текущий (публичный) IP можно быстро узнать через сервисы вроде `ifconfig.me`: `curl https://ifconfig.me`.

Для проверки публичных IP используйте внешнюю сеть или удалённый сервер:

```bash
print_current_ip() {
  curl https://ifconfig.me
}

print_current_ip
# --> 123.456.789.012

# Замените target_host на ваш публичный IP или имя хоста
# Проверка хоста с помощью продвинутых методов
nmap -A --open --reason $target_host
nmap -A -F --open --reason $target_host
nmap -A -p1-65535 --open --reason $target_host
```

**Зачем тестировать оба направления?**  
Тесты изнутри показывают внутренние утечки, а внешние сканирования выявляют сервисы, доступные злоумышленникам.

## 🛡️ Распространённые заблуждения
```

1. **Моёлокальное окружение не является целью.**  
   - Факт: Злоумышленники могут использовать ваш компьютер как точку опоры для доступа к продакшн‑системам.  
2. **Брандмауэры блокируют всё.**  
   - Факт: Они блокируют только то, что вы явно настроили.  
3. **Приватные IP‑адреса безопасны.**  
   - Факт: Эксплойты, такие как обходы NAT, всё равно могут затронуть вашу сеть.
