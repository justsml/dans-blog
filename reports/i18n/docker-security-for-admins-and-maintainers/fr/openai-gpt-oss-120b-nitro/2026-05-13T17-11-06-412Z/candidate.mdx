## ⚠️ Réseaux locaux à risque

Soyons honnêtes, nous l’avons tous fait. Vous vous êtes connecté à un Wi‑Fi de café aléatoire ou avez laissé quelqu’un utiliser votre réseau domestique sans y réfléchir. Peut‑être faites‑vous même confiance à votre réfrigérateur intelligent pour ne pas compromettre votre réseau. La réalité ? Ces décisions désinvoltes peuvent exposer votre environnement de développement local à des risques inutiles. Les attaquants ne ciblent pas seulement les systèmes de production — les environnements locaux sont souvent des cibles plus molles, offrant un accès aux projets sensibles.

### Scénarios d’attaque

1. **Trafic intercepté :** Le trafic non chiffré peut être capturé et lu facilement.  
2. **Services non protégés :** Bases de données ou API locales exposées sur `0.0.0.0`.  
3. **Usurpation de réseau :** Redirection du trafic vers l’appareil de l’attaquant.

### Corrections rapides

- Privilégiez les réseaux Docker privés plutôt que les pare‑feux pour limiter l’exposition réseau.  
- Évitez le Wi‑Fi public ou partagé ; utilisez de préférence le point d’accès de votre téléphone.  
- Surveillez votre réseau local à la recherche d’appareils inconnus avec des outils comme `arp-scan` et `nmap`.

## 🛡️ Configuration du pare‑feu

### UFW avec Docker (Ubuntu)

> ⚠️ **Avertissement :** Par défaut, Docker sur Ubuntu/Debian contourne les règles UFW/iptables, ce qui peut exposer votre système aux attaques.  
> Peu importe que vous liiez les ports à des adresses IP locales (par ex. `-p 127.0.0.1:8080:80`).

Cela me surprend à chaque fois que j’en apprends davantage ! [Docker contourne les règles UFW par défaut](https://github.com/moby/moby/issues/4737), permettant aux conteneurs de communiquer avec l’hôte et entre eux sans restriction.

### Meilleure pratique

1. 🥇 **Utilisez les réseaux Docker** pour isoler et contrôler ce qui peut se connecter à chaque conteneur ou réseau.

###
2. 🥉 **Mettez à jour iptables** si vous devez utiliser un réseau `host`, ou si vous ne pouvez pas recourir à des réseaux personnalisés, vous pouvez atténuer le risque en configurant iptables. Ce n’est pas pour les timorés, [consultez l’utilitaire ci‑dessous.](#uf)

#### Isolationdu réseau Docker

```bash
# Create a new Docker network
docker network create my-network

# Run your container with the new network
docker run --network my-network my-container
```

#### Configuration UFW (pour les réseaux `host`)

Il existe beaucoup de mauvais conseils sur la façon de corriger ce problème. Configurez UFW pour qu’il fonctionne avec Docker en l’utilisant comme vous le feriez normalement.

J’ai utilisé `ufw-docker` pour configurer un système auto‑hébergé et cela semble bien fonctionner.

```bash title="install-ufw-docker.sh"
# Install binary as root (needs root permissions anyway)
sudo wget -O /usr/local/bin/ufw-docker \
   https://github.com/chaifeng/ufw-docker/raw/master/ufw-docker
sudo chmod +x /usr/local/bin/ufw-docker
# Install and modify the `after.rules` file of `ufw`
ufw-docker install

ufw-docker help
```

Ce commande effectue les actions suivantes :

- Sauvegarde le fichier `/etc/ufw/after.rules`.
- Ajoute des règles liées à Docker à la fin du fichier afin de s’intégrer correctement avec UFW.

**Source :** [ufw-docker GitHub](https://github.com/chaifeng/ufw-docker/tree/master#install)

**Exemple d’utilisation :**

```bash

# Autoriser un conteneur Docker sur le port 8080
ufw-docker allow <container_name> 8080/tcp

# Gérer les règles en toute sécurité aux côtés de votre configuration UFW
ufw-docker status

```

**Note :** La plupart des « correctifs » pour les conflits Docker‑UFW reposent sur des règles iptables manuelles, ce qui peut être source d’erreurs et fragile lors des mises à jour.

### Pare‑feu macOS

1. Ouvrez **Préférences Système > Sécurité et confidentialité > Pare‑feu**.  
2. Activez le pare‑feu et cliquez sur « Options du pare‑feu ».  
3. Bloquez toutes les connexions entrantes sauf les services essentiels.

**Note :** Vous devrez peut‑être consulter la configuration de votre pare‑feu pour autoriser certains appareils intelligents que vous utilisez – par ex. Google Cast/AirPlay et d’autres services.

### Commandes pour les utilisateurs avancés (macOS et Linux)

#### macOS :

```bash

sudo /usr/libexec/ApplicationFirewall/socketfilterfw --setblockall on  # Bloquer tout
sudo /usr/libexec/ApplicationFirewall/socketfilterfw --add /path/to/app  # Autoriser une application précise

```

#### Linux (ufw) :

```bash

ufw default deny incoming  # Bloquer toutes les entrées
ufw allow ssh  # Autoriser SSH
# autoriser 443 et 80 pour le trafic web
ufw allow 80/tcp
ufw allow 443/tcp
ufw enable  # Activer le pare‑feu

```

**Pro Tip :** Utilisez des outils comme [Little Snitch](https://www.obdev.at/products/littlesnitch/index.html) sur macOS et [ufw](https://help.ubuntu.com/community/UFW) sur Linux pour des configurations plus conviviales.

## 🔐 Gestion des secrets pour le développement local

### Validation proactive des espaces réservés

<p>💡 Assurez‑vous que les secrets sont correctement configurés avec de vraies valeurs avant d’exécuter votre application.</p>

Si vous utilisez des espaces réservés comme `__WARNING_REPLACE_ME__` dans vos secrets, super, peut‑être que quelqu’un le remarquera. Au cas où, vous pouvez également ajouter une petite validation pour garantir la sécurité à l’exécution.

Vous ne croiriez pas à quel point il est facile de pirater complètement (modifier & re‑signer) un jeton JWT quand les attaquants peuvent deviner le secret !

<CodeTabs client:load tabs={["JavaScript", "Rust", "Go"]}>

```javascript
// validateSecrets.js
const validateSecrets = () => {
  const unsafePlaceholder = /__WARNING_REPLACE_ME__/;
  const missingSecrets = Object.entries(process.env).filter(
    ([key, value]) => unsafePlaceholder.test(value)
  );

  if (missingSecrets.length) {
    console.error("Unsafe secrets detected:", missingSecrets);
    process.exit(1);
  }
};

validateSecrets();
```

```rust
// validate_secrets.rs
use std::env;

fn validate_secrets() {
    let unsafe_placeholder = "__WARNING_REPLACE_ME__";
    for (key, value) in env::vars() {
        if value.contains(unsafe_placeholder) {
            panic!("Unsafe secret in {}", key);
        }
    }
}

fn main() {
    validate_secrets();
}
```

```go
// validate_secrets.go
package main

import (
	"fmt"
	"os"
	"strings"
)

func validateSecrets() {
	placeholder := "__WARNING_REPLACE_ME__"
	for _, env := range os.Environ() {
		pair := strings.SplitN(env, "=", 2)
		if len(pair) == 2 && strings.Contains(pair[1], placeholder) {
			panic(fmt.Sprintf("Unsafe secret in %s", pair[0]))
		}
	}
}

func main() {
	validateSecrets()
}
```

###Générer et stocker les secrets

<p class="inset">Ne jamais coder en dur les secrets dans votre base de code. Privilégiez les variables d’environnement et les coffres sécurisés.</p>

Au lieu de `.env.example`, utilisez `.env.generate.sh` pour faciliter aux utilisateurs l’obtention d’un fichier `.env` avec des « valeurs par défaut » sécurisées.

#### Exemple `.env.generate.sh`

```bash title=".env.generate.sh" frame="code"
#!/bin/bash
# Generates a secure .env file for local development

generate_secret() {
    local length=${1:-30}
    # add 4 bytes to account for padding
    local generate_length=$((length + 4))
    openssl rand -base64 "$generate_length" | tr -d '+=/\n' | cut -c1-"$length"
}
# Bail out if .env file already exists
[ -f .env ] && { echo ".env file already exists!"; exit 1; }

cat <<EOL > .env
# Database settings & secrets
DB_USER=app_user
DB_PASSWORD=$(generate_secret 30)
REDIS_PASSWORD=$(generate_secret 20)
# Session secrets
SESSION_KEY=$(generate_secret 32)
JWT_SECRET=$(generate_secret 64)
EOL

echo "New .env file generated!"
```

## 🕵️‍ Surveillance et double vérification

### `nmap` Exemples

#### Tester dans votre réseau

```bash

# Scan your localhost for all open ports
nmap -sT localhost

# Scan your machine’s private IP for services
nmap -sV 192.168.1.10

# Detect devices on your network
nmap -sn 192.168.0.0/24
nmap -sn 10.0.0.0/24
```

```

#### Tester depuis l’extérieur de votre réseau

Vous pouvez récupérer votre adresse IP publique en un clin d’œil avec des services comme `ifconfig.me` : `curl https://ifconfig.me`.

Utilisez un réseau externe ou un serveur distant pour tester vos adresses IP publiques :

```bash

print_current_ip() {
  curl https://ifconfig.me
}

print_current_ip
# --> 123.456.789.012

# Remplacez target_host par votre IP publique ou votre nom d’hôte
# Vérifiez l’hôte avec des techniques avancées
nmap -A --open --reason $target_host
nmap -A -F --open --reason $target_host
nmap -A -p1-65535 --open --reason $target_host

```

**Pourquoi tester les deux ?**  
Les tests depuis l’intérieur exposent les vulnérabilités internes, tandis que les tests externes identifient les services accessibles aux attaquants.

## 🛡️ Idées reçues courantes

1.**Mon environnement local n’est pas une cible.**  
   - Factuel : les attaquants peuvent pivoter depuis votre machine vers vos systèmes de production.  
2. **Les pare‑feux bloquent tout.**  
   - Factuel : ils ne bloquent que ce que vous leur avez configuré.  
3. **Les IP privées sont sécurisées.**  
   - Factuel : des exploits comme le contournement NAT peuvent tout de même affecter votre réseau.
