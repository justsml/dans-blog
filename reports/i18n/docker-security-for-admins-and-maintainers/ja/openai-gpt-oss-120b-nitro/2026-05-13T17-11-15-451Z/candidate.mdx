## ⚠️ ローカルネットワークのリスク

正直に言いましょう、誰もがやったことがあります。ランダムなカフェの Wi‑Fi に接続したり、何の疑問も抱かずに自宅ネットワークを他人に使わせたり。場合によっては、スマート冷蔵庫がネットワークを危険にさらさないと信じているかもしれません。実際には、こうした軽い判断がローカル開発環境を不必要なリスクにさらすことになります。攻撃者は本番システムだけを狙うわけではなく、ローカル環境を比較的柔らかい標的として利用し、機密プロジェクトへの入口を得ようとします。

### 攻撃シナリオ

1. **トラフィックの傍受:** 暗号化されていない通信は簡単にキャプチャされ、内容が読まれます。
2. **保護されていないサービス:** `0.0.0.0` 上に公開されたローカルデータベースや API。
3. **ネットワークスプーフィング:** トラフィックを攻撃者のデバイスへリダイレクト。

### 即効対策

- ネットワーク露出を抑えるため、ファイアウォールよりプライベート Docker ネットワークを優先します。
- 公衆 Wi‑Fi や共有ネットワークは避け、携帯電話のホットスポットを利用します。
- `arp-scan` や `nmap` といったツールで未知のデバイスを検出し、ローカルネットワークを監視します。

## 🛡️ ファイアウォール設定

### UFW と Docker (Ubuntu)

> ⚠️ **警告:** デフォルトでは Ubuntu/Debian 上の Docker は UFW/iptables のルールをバイパスし、システムが攻撃にさらされる可能性があります。  
> `-p 127.0.0.1:8080:80` のようにローカル IP アドレスにポートバインドしても影響はありません。

これを知るたびに驚かされます！ [Docker はデフォルトで UFW ルールをバイパスします](https://github.com/moby/moby/issues/4737) 、コンテナはホストや他のコンテナと制限なく通信できてしまいます。

### ベストプラクティス

1. 🥇 **Docker ネットワークを使用** して、各コンテナやネットワークに接続できる対象を分離・制御します。

###
2. 🥉 **iptables を更新** します。`host` ネットワークを使用せざるを得ない、またはカスタムネットワークが使えない場合は、iptables を設定してリスクを緩和できます。初心者向きではありません、[以下のユーティリティをご覧ください。](#uf)

#### Docker ネットワーク分離

```bash
# 新しい Docker ネットワークを作成
docker network create my-network

# コンテナを新しいネットワークで起動
docker run --network my-network my-container
```

#### UFW 設定（`host` ネットワーク用）

この問題を解決しようとする情報の中には、根拠のない悪いアドバイスが多く混在しています。UFW を Docker と連携させるには、期待通りに設定すればよいです。

私は `ufw-docker` を使って自前の環境を構築しましたが、問題なく動作しています。

```bash title="install-ufw-docker.sh"
# バイナリを root でインストール（root 権限は必須）
sudo wget -O /usr/local/bin/ufw-docker \
   https://github.com/chaifeng/ufw-docker/raw/master/ufw-docker
sudo chmod +x /usr/local/bin/ufw-docker
# `ufw` の `after.rules` ファイルをインストール・変更
ufw-docker install

ufw-docker help
```

```

このコマンドは以下を実行します。

- `/etc/ufw/after.rules` ファイルをバックアップします。  
- Docker 用のルールをファイル末尾に追記し、UFW と正しく統合します。

**出典:** [ufw-docker GitHub](https://github.com/chaifeng/ufw-docker/tree/master#install)

**使用例:**

```bash

# Docker コンテナをポート 8080 で許可
ufw-docker allow <container_name> 8080/tcp

# UFW 設定と併せてルールを安全に管理
ufw-docker status

```

**注記:** Docker と UFW の衝突を解消する「修正」の多くは手動で iptables ルールを追加する形になるため、更新時にエラーが起きやすく脆弱です。

### macOS ファイアウォール

1. **システム環境設定 > セキュリティとプライバシー > ファイアウォール** を開く。  
2. ファイアウォールを有効化し、「ファイアウォールオプション」をクリック。  
3. 必要なサービス以外のすべての受信接続をブロックする。

**注記:** 使用しているスマートデバイス（例: Google Cast、AirPlay など）を許可するために、ファイアウォール設定を個別に確認する必要があります。

### 上級者向けコマンド (macOS と Linux)

#### macOS:

```bash

sudo /usr/libexec/ApplicationFirewall/socketfilterfw --setblockall on  # すべてブロック
sudo /usr/libexec/ApplicationFirewall/socketfilterfw --add /path/to/app  # 特定アプリを許可

```

#### Linux (ufw):

```bash

ufw default deny incoming  # すべての受信をブロック
ufw allow ssh  # SSH を許可
# Web トラフィック用に 443 と 80 を許可
ufw allow 80/tcp
ufw allow 443/tcp
ufw enable  # ファイアウォールを有効化

```

**プロ・ティップ:** macOS では [Little Snitch](https://www.obdev.at/products/littlesnitch/index.html) を、Linux では [ufw](https://help.ubuntu.com/community/UFW) を使うと、よりユーザーフレンドリーに設定できます。

## 🔐 ローカル開発向けシークレット管理

### 事前プレースホルダー検証

<p>💡 アプリケーション実行前に、シークレットが実際の値で正しく設定されていることを確認します。</p>

シークレットに `__WARNING_REPLACE_ME__` のようなプレースホルダーを残したままにすると、たまたま誰かが気付くかもしれません。万が一に備えて、実行時に安全性を確保するための簡易検証を追加しておくとよいでしょう。

攻撃者がシークレットを推測できるだけで、JWT トークンを完全に改ざん（変更＆再署名）できるのがどれほど簡単か、想像できないでしょう！

<CodeTabs client:load tabs={["JavaScript", "Rust", "Go"]}>

```javascript
// validateSecrets.js
const validateSecrets = () => {
  const unsafePlaceholder = /__WARNING_REPLACE_ME__/;
  const missingSecrets = Object.entries(process.env).filter(
    ([key, value]) => unsafePlaceholder.test(value)
  );

  if (missingSecrets.length) {
    console.error("Unsafe secrets detected:", missingSecrets);
    process.exit(1);
  }
};

validateSecrets();
```

```rust
// validate_secrets.rs
use std::env;

fn validate_secrets() {
    let unsafe_placeholder = "__WARNING_REPLACE_ME__";
    for (key, value) in env::vars() {
        if value.contains(unsafe_placeholder) {
            panic!("Unsafe secret in {}", key);
        }
    }
}

fn main() {
    validate_secrets();
}
```

```go
// validate_secrets.go
package main

import (
	"fmt"
	"os"
	"strings"
)

func validateSecrets() {
	placeholder := "__WARNING_REPLACE_ME__"
	for _, env := range os.Environ() {
		pair := strings.SplitN(env, "=", 2)
		if len(pair) == 2 && strings.Contains(pair[1], placeholder) {
			panic(fmt.Sprintf("Unsafe secret in %s", pair[0]))
		}
	}
}

func main() {
	validateSecrets()
}
```

### シークレットの生成と保存

<p class="inset">コードベースにシークレットをハードコードしないこと。環境変数や安全なボールトを利用することを推奨します。</p>


`.env.example` の代わりに `.env.generate.sh` を用意し、ユーザーが安全な「デフォルト」から `.env` ファイルを簡単に取得できるようにします。

#### `.env.generate.sh` の例

```bash title=".env.generate.sh" frame="code"
#!/bin/bash
# ローカル開発用の安全な .env ファイルを生成

generate_secret() {
    local length=${1:-30}
    # パディング分として 4 バイト追加
    local generate_length=$((length + 4))
    openssl rand -base64 "$generate_length" | tr -d '+=/\n' | cut -c1-"$length"
}
# .env ファイルが既に存在する場合は中止
[ -f .env ] && { echo ".env file already exists!"; exit 1; }

cat <<EOL > .env
# データベース設定とシークレット
DB_USER=app_user
DB_PASSWORD=$(generate_secret 30)
REDIS_PASSWORD=$(generate_secret 20)
# セッションシークレット
SESSION_KEY=$(generate_secret 32)
JWT_SECRET=$(generate_secret 64)
EOL

echo "New .env file generated!"
```

## 🕵️‍ 監視と二重チェック

### `nmap` の例

#### ネットワーク内部でのテスト

```bash

# localhost のすべての開いているポートをスキャン
nmap -sT localhost

# マシンのプライベート IP に対してサービスをスキャン
nmap -sV 192.168.1.10

# ネットワーク上のデバイスを検出
nmap -sn 192.168.0.0/24
nmap -sn 10.0.0.0/24
```

#### ネットワーク外でのテスト

現在の（パブリック）IP アドレスは `ifconfig.me` などのサービスで簡単に取得できます: `curl https://ifconfig.me`.

外部ネットワークまたはリモートサーバーからパブリック IP をテストします:

```bash

print_current_ip() {
  curl https://ifconfig.me
}

print_current_ip
# --> 123.456.789.012

# target_host を自分のパブリック IP またはホスト名に置き換えてください
# 高度な手法でホストを確認
nmap -A --open --reason $target_host
nmap -A -F --open --reason $target_host
nmap -A -p1-65535 --open --reason $target_host

```

**なぜ両方テストするのか？**  
内部からのテストは内部露出を示し、外部からのテストは攻撃者が到達可能なサービスを特定します。

## 🛡️ 一般的な誤解

---

1. **ローカル環境は標的ではない**  
   - Fact: 攻撃者はあなたのマシンからプロダクションシステムへピボットできる。  
2. **ファイアウォールはすべてをブロックする**  
   - Fact: ファイアウォールは設定したものだけをブロックする。  
3. **プライベート IP は安全だ**  
   - Fact: NAT バイパスなどのエクスプロイトは依然としてネットワークに影響を及ぼす可能性がある。
