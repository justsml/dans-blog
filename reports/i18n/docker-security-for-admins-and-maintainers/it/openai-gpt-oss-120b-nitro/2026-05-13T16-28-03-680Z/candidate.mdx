## ⚠️ Reti locali a rischio

Siamo onesti, tutti noi lo abbiamo fatto. Ti sei connesso a una rete Wi‑Fi di un bar a caso o hai lasciato che qualcuno usasse la tua rete domestica senza pensarci due volte. Forse ti fidi persino del tuo frigorifero intelligente perché non comprometta la rete. La realtà? queste decisioni casuali possono esporre il tuo ambiente di sviluppo locale a rischi inutili. Gli aggressori non puntano solo ai sistemi di produzione — gli ambienti locali sono spesso bersagli più deboli, offrendo un punto d’ingresso per accedere a progetti sensibili.

### Scenari di attacco

1. **Traffico intercettato:** Il traffico non crittografato può essere catturato e letto con facilità.  
2. **Servizi non protetti:** Database o API locali esposti su `0.0.0.0`.  
3. **Spoofing di rete:** Reindirizzamento del traffico verso il dispositivo dell’attaccante.

### Soluzioni rapide

- Preferisci reti Docker private rispetto ai firewall per limitare l’esposizione della rete.  
- Evita Wi‑Fi pubblici o condivisi; usa preferibilmente l’hotspot del tuo telefono.  
- Monitora la tua rete locale alla ricerca di dispositivi sconosciuti con strumenti come `arp-scan` e `nmap`.

## 🛡️ Configurazione del firewall

### UFW con Docker (Ubuntu)

> ⚠️ **Attenzione:** Per impostazione predefinita Docker su Ubuntu/Debian aggira le regole UFW/iptables, potenzialmente esponendo il tuo sistema ad attacchi.  
> Non importa se associ le porte a indirizzi IP locali (es. `-p 127.0.0.1:8080:80`).

Questo mi sorprende ogni volta che ne sento parlare! [Docker bypasses UFW rules by default](https://github.com/moby/moby/issues/4737), consentendo ai container di comunicare con l'host e tra di loro senza restrizioni.

### Best Practice

1. 🥇 **Usa le reti Docker** per isolare e controllare cosa può connettersi a ciascun container o rete.

###
2. 🥉 **Aggiorna iptables** se devi usare una rete `host`, o non puoi utilizzare reti personalizzate, puoi mitigare il rischio configurando iptables. Non per i deboli di cuore, [dai un'occhiata all'utilità qui sotto](../#uf).

#### Isolamento della rete Docker

```bash
# Crea una nuova rete Docker
docker network create my-network

# Avvia il tuo container con la nuova rete
docker run --network my-network my-container
```

#### Configurazione di UFW (per reti `host`)

Circolano molti consigli sbagliati su come risolvere questo problema. Configura UFW per lavorare con Docker usando UFW in modo abbastanza intuitivo.

Ho usato `ufw-docker` per configurare un sistema self‑hosted e mi è sembrato funzionare bene.

```bash title="install-ufw-docker.sh"
# Installa il binario come root (richiede comunque i permessi di root)
sudo wget -O /usr/local/bin/ufw-docker \
   https://github.com/chaifeng/ufw-docker/raw/master/ufw-docker
sudo chmod +x /usr/local/bin/ufw-docker
# Installa e modifica il file `after.rules` di `ufw`
ufw-docker install

ufw-docker help
```

```

Questo comando esegue le seguenti operazioni:

- Effettua il backup del file `/etc/ufw/after.rules`.
- Aggiunge regole relative a Docker alla fine del file per integrarsi correttamente con UFW.

**Fonte:** [ufw-docker GitHub](https://github.com/chaifeng/ufw-docker/tree/master#install)

**Esempio d'uso:**

```bash

# Consenti al container Docker di ascoltare sulla porta 8080
ufw-docker allow <container_name> 8080/tcp

# Gestisci le regole in modo sicuro insieme alla tua configurazione UFW
ufw-docker status

```

**Nota:** La maggior parte delle “soluzioni” per i conflitti Docker‑UFW prevede regole iptables manuali, che possono risultare soggette a errori e fragili durante gli aggiornamenti.

### Firewall macOS

1. Vai su **Preferenze di Sistema > Sicurezza e Privacy > Firewall**.
2. Attiva il firewall e fai clic su “Opzioni Firewall”.
3. Blocca tutte le connessioni in ingresso tranne i servizi essenziali.

**Nota:** Potrebbe essere necessario consultare la configurazione del tuo firewall per consentire alcuni dispositivi intelligenti che usi, ad esempio Google Cast/AirPlay e altri servizi.

### Comandi per Utenti Avanzati (macOS e Linux)

#### macOS:

```bash

sudo /usr/libexec/ApplicationFirewall/socketfilterfw --setblockall on  # Blocca tutto
sudo /usr/libexec/ApplicationFirewall/socketfilterfw --add /path/to/app  # Consenti app specifica

```

#### Linux (ufw):

```bash

ufw default deny incoming  # Blocca tutte le connessioni in ingresso
ufw allow ssh  # Consenti SSH
# consenti 443 e 80 per il traffico web
ufw allow 80/tcp
ufw allow 443/tcp
ufw enable  # Abilita il firewall

```

**Consiglio da esperti:** Usa strumenti come [Little Snitch](https://www.obdev.at/products/littlesnitch/index.html) su macOS e [ufw](https://help.ubuntu.com/community/UFW) su Linux per configurazioni più user‑friendly.

## 🔐 Gestione dei Segreti per lo Sviluppo Locale

### Validazione Proattiva dei Segnaposto

<p>💡 Assicurati che i segreti siano impostati con valori reali prima di avviare l'applicazione.</p>

Se usi segnaposto come `__WARNING_REPLACE_ME__` nei tuoi segreti, ottimo, magari qualcuno se ne accorgerà. Per sicurezza, puoi aggiungere una piccola validazione per garantire la protezione a runtime.

Non crederesti a quanto sia semplice compromettere (modificare e risignare) un token JWT quando gli aggressori possono indovinare il segreto!

<CodeTabs client:load tabs={["JavaScript", "Rust", "Go"]}>

```javascript
// validateSecrets.js
const validateSecrets = () => {
  const unsafePlaceholder = /__WARNING_REPLACE_ME__/;
  const missingSecrets = Object.entries(process.env).filter(
    ([key, value]) => unsafePlaceholder.test(value)
  );

  if (missingSecrets.length) {
    console.error("Unsafe secrets detected:", missingSecrets);
    process.exit(1);
  }
};

validateSecrets();
```

```rust
// validate_secrets.rs
use std::env;

fn validate_secrets() {
    let unsafe_placeholder = "__WARNING_REPLACE_ME__";
    for (key, value) in env::vars() {
        if value.contains(unsafe_placeholder) {
            panic!("Unsafe secret in {}", key);
        }
    }
}

fn main() {
    validate_secrets();
}
```

```go
// validate_secrets.go
package main

import (
	"fmt"
	"os"
	"strings"
)

func validateSecrets() {
	placeholder := "__WARNING_REPLACE_ME__"
	for _, env := range os.Environ() {
		pair := strings.SplitN(env, "=", 2)
		if len(pair) == 2 && strings.Contains(pair[1], placeholder) {
			panic(fmt.Sprintf("Unsafe secret in %s", pair[0]))
		}
	}
}

func main() {
	validateSecrets()
}
```

### Generazione e memorizzazione dei segreti

<p class="inset">Mai inserire segreti direttamente nel codice. Preferisci variabili d'ambiente e vault sicuri.</p>


Invece di `.env.example`, usa `.env.generate.sh` per semplificare agli utenti la creazione di un file `.env` con “default” sicuri.

#### Esempio `.env.generate.sh`

```bash title=".env.generate.sh" frame="code"
#!/bin/bash
# Generates a secure .env file for local development

generate_secret() {
    local length=${1:-30}
    # add 4 bytes to account for padding
    local generate_length=$((length + 4))
    openssl rand -base64 "$generate_length" | tr -d '+=/\n' | cut -c1-"$length"
}
# Bail out if .env file already exists
[ -f .env ] && { echo ".env file already exists!"; exit 1; }

cat <<EOL > .env
# Database settings & secrets
DB_USER=app_user
DB_PASSWORD=$(generate_secret 30)
REDIS_PASSWORD=$(generate_secret 20)
# Session secrets
SESSION_KEY=$(generate_secret 32)
JWT_SECRET=$(generate_secret 64)
EOL

echo "New .env file generated!"
```

## 🕵️‍ Monitoraggio e verifica

### Esempi di `nmap`

#### Test all'interno della tua rete

```bash

# Scan your localhost for all open ports
nmap -sT localhost

# Scan your machine’s private IP for services
nmap -sV 192.168.1.10

# Detect devices on your network
nmap -sn 192.168.0.0/24
nmap -sn 10.0.0.0/24
```

```

#### Testare al di fuori della tua rete

Per conoscere rapidamente il tuo IP pubblico puoi usare servizi come `ifconfig.me`: `curl https://ifconfig.me`.

Usa una rete esterna o un server remoto per verificare i tuoi IP pubblici:

```bash

print_current_ip() {
  curl https://ifconfig.me
}

print_current_ip
# --> 123.456.789.012

# Cambia target_host con il tuo IP pubblico o hostname
# Verifica l'host usando tecniche avanzate
nmap -A --open --reason $target_host
nmap -A -F --open --reason $target_host
nmap -A -p1-65535 --open --reason $target_host

```

**Perché testare entrambi?**
I test dall'interno mostrano le esposizioni interne, mentre i test dall'esterno identificano i servizi accessibili agli aggressori.
