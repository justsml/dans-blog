## ⚠️ Lokale Netzwerke unter Risiko

Seien wir ehrlich: Wir haben es alle schon mal getan. Sie haben sich an einem zufälligen Café-WLAN angeschlossen oder jemandem erlaubt, Ihr Heimnetzwerk zu nutzen, ohne groß darüber nachzudenken. Vielleicht vertrauen Sie sogar Ihrem Smart-Kühlschrank, Ihr Netzwerk nicht zu gefährden. Die Realität? Diese scheinbar harmlosen Entscheidungen können Ihr lokales Entwicklungssetup unnötigen Risiken aussetzen. Angreifer zielen nicht nur auf Produktionsumgebungen ab – lokale Umgebungen sind oft weichere Ziele, die sensiblen Projekten Zugriff verschaffen können.

### Angriffsszenarien

1. **Abgefangener Datenverkehr:** Unverschlüsselter Datenverkehr kann leicht abgefangen und ausgelesen werden.
2. **Unschützte Dienste:** Lokale Datenbanken oder APIs, die auf `0.0.0.0` ausgeliefert werden.
3. **Netzwerkbetrug:** Datenverkehr wird an ein Gerät des Angreifers weitergeleitet.

### Schnelle Lösungen

- Nutzen Sie private Docker-Netzwerke statt Firewalls, um den Netzwerkzugriff zu beschränken.
- Vermeiden Sie öffentliche oder geteilte WLAN-Netzwerke; bevorzugen Sie stattdessen den Hotspot Ihres Handys.
- Überwachen Sie Ihr lokales Netzwerk nach unbekannten Geräten mit Tools wie `arp-scan` und `nmap`.

## 🛡️ Firewall-Konfiguration

### UFW mit Docker (Ubuntu)

> ⚠️ **Warnung:** Standardmäßig umgeht Docker auf Ubuntu/Debian UFW/iptables-Regeln, was Ihr System potenziell Angriffen aussetzen kann.  
> Es ist egal, ob Sie Ports an lokale IP-Adressen binden (z. B. `-p 127.0.0.1:8080:80`).  

Das überrascht mich jedes Mal, wenn ich es erlerne! [Docker umgeht UFW-Regeln standardmäßig](https://github.com/moby/moby/issues/4737), wodurch Container ungehindert mit dem Host und anderen Containern kommunizieren können.  

### Best Practices  

1. 🥇 **Nutzen Sie Docker-Netzwerke**, um Container oder Netzwerke zu isolieren und zu steuern, was sich mit ihnen verbinden kann.  

2. 🥉 **Aktualisieren Sie iptables**, falls Sie ein `host`-Netzwerk verwenden müssen oder keine benutzerdefinierten Netzwerke einsetzen können. Um das Risiko zu reduzieren, können Sie iptables konfigurieren. Nicht für schwache Nerven geeignet, [siehe untenstehendes Hilfsprogramm.](#uf)

#### Docker-Netzwerkisolation

```bash
# Erstellen Sie ein neues Docker-Netzwerk
docker network create my-network

# Führen Sie Ihren Container mit dem neuen Netzwerk aus
docker run --network my-network my-container
```

#### UFW-Konfiguration (für `host`-Netzwerke)

Es gibt viel schlechte Ratschläge, um dies zu beheben. Konfigurieren Sie UFW so, dass es mit Docker zusammenarbeitet, wie Sie es vermutlich erwarten würden.

Ich habe `ufw-docker` verwendet, um ein selbst gehostetes System zu konfigurieren, und es scheint gut zu funktionieren.

```bash title="install-ufw-docker.sh"
# Installieren Sie die Binärdatei als Root (benötigt ohnehin Root-Rechte)
sudo wget -O /usr/local/bin/ufw-docker \
   https://github.com/chaifeng/ufw-docker/raw/master/ufw-docker
sudo chmod +x /usr/local/bin/ufw-docker
# Installieren und ändern Sie die Datei `after.rules` von `ufw`
ufw-docker install

ufw-docker help
```

Dieser Befehl führt folgendes aus:

- Erstellt ein Backup der Datei `/etc/ufw/after.rules`.
- Fügt Docker-spezifische Regeln am Ende der Datei hinzu, um korrekt mit UFW zusammenzuarbeiten.

**Quelle:** [ufw-docker GitHub](https://github.com/chaifeng/ufw-docker/tree/master#install)

**Beispielverwendung:**

```bash
# Erlaubt Docker-Container auf Port 8080
ufw-docker allow <container_name> 8080/tcp

# Verwalte Regeln sicher neben deiner UFW-Konfiguration
ufw-docker status
```

**Hinweis:** Die meisten "Lösungen" für Docker-UFW-Konflikte beinhalten manuelle iptables-Regeln, die fehleranfällig und brüchig bei Updates sein können.

### macOS-Feuerwall

1. Gehe zu **Systemeinstellungen > Sicherheit und Datenschutz > Firewall**.
2. Aktiviere die Firewall und klicke auf "Firewall-Optionen".
3. Blockiere alle eingehenden Verbindungen außer den wesentlichen Diensten.

**Hinweis:** Du musst möglicherweise die Konfiguration deiner Firewall anpassen, um bestimmte Smart-Home-Geräte zu erlauben, z. B. Google Cast/AirPlay und andere Dienste.

### Befehle für fortgeschrittene Benutzer (macOS und Linux)

#### macOS:

```bash
sudo /usr/libexec/ApplicationFirewall/socketfilterfw --setblockall on  # Alles blockieren
sudo /usr/libexec/ApplicationFirewall/socketfilterfw --add /path/to/app  # Spezifische App erlauben
```

#### Linux (ufw):

```bash
ufw default deny incoming  # Alle eingehenden Verbindungen blockieren
ufw allow ssh  # SSH erlauben
# 443 und 80 für Webverkehr erlauben
ufw allow 80/tcp
ufw allow 443/tcp
ufw enable  # Firewall aktivieren
```

**Pro-Tipp:** Nutze Tools wie [Little Snitch](https://www.obdev.at/products/littlesnitch/index.html) auf macOS und [ufw](https://help.ubuntu.com/community/UFW) auf Linux für benutzerfreundlichere Konfigurationen.

## 🔐 Geheimnisse-Verwaltung für lokale Entwicklung

### Proaktive Platzhalter-Validierung

<p>💡 Stelle sicher, dass Geheimnisse vor dem Start deiner Anwendung mit echten Werten konfiguriert sind.</p>

Wenn du Platzhalter wie `__WARNING_REPLACE_ME__` in deinen Geheimnissen verwendest, gut, vielleicht bemerkt jemand. Falls nicht, kannst du auch eine kleine Validierung hinzufügen, um Sicherheit zur Laufzeit zu gewährleisten.

Du würdest überrascht sein, wie einfach es ist, einen JWT-Token vollständig zu manipulieren (zu ändern und erneut zu signieren), wenn Angreifer den Geheimnis-Schlüssel erraten können!

<CodeTabs client:load tabs={["JavaScript", "Rust", "Go"]}>

```javascript
// validateSecrets.js
const validateSecrets = () => {
  const unsafePlaceholder = /__WARNING_REPLACE_ME__/;
  const missingSecrets = Object.entries(process.env).filter(
    ([key, value]) => unsafePlaceholder.test(value)
  );

  if (missingSecrets.length) {
    console.error("Unsafe secrets detected:", missingSecrets);
    process.exit(1);
  }
};

validateSecrets();
```

```rust
// validate_secrets.rs
use std::env;

fn validate_secrets() {
    let unsafe_placeholder = "__WARNING_REPLACE_ME__";
    for (key, value) in env::vars() {
        if value.contains(unsafe_placeholder) {
            panic!("Unsafe secret in {}", key);
        }
    }
}

fn main() {
    validate_secrets();
}
```

```go
// validate_secrets.go
package main

import (
	"fmt"
	"os"
	"strings"
)

func validateSecrets() {
	placeholder := "__WARNING_REPLACE_ME__"
	for _, env := range os.Environ() {
		pair := strings.SplitN(env, "=", 2)
		if len(pair) == 2 && strings.Contains(pair[1], placeholder) {
			panic(fmt.Sprintf("Unsafe secret in %s", pair[0]))
		}
	}
}

func main() {
	validateSecrets()
}
```

### Generieren und Speichern von Geheimnissen

<p class="inset">Verwenden Sie niemals Geheimnisse im Quellcode. Verwenden Sie stattdessen Umgebungsvariablen und sichere Safes.</p>

Verwenden Sie stattdessen `.env.generate.sh`, um Benutzern zu ermöglichen, eine `.env`-Datei mit sicheren „Standardwerten“ zu generieren.

#### Beispiel `.env.generate.sh`

```bash title=".env.generate.sh" frame="code"
#!/bin/bash
# Generiert eine sichere .env-Datei für die lokale Entwicklung

generate_secret() {
    local length=${1:-30}
    # add 4 bytes to account for padding
    local generate_length=$((length + 4))
    openssl rand -base64 "$generate_length" | tr -d '+=/\n' | cut -c1-"$length"
}
# Beenden, wenn .env-Datei bereits existiert
[ -f .env ] && { echo ".env-Datei existiert bereits!"; exit 1; }

cat <<EOL > .env
# Datenbank-Einstellungen & Geheimnisse
DB_USER=app_user
DB_PASSWORD=$(generate_secret 30)
REDIS_PASSWORD=$(generate_secret 20)
# Sitzungsgeheimnisse
SESSION_KEY=$(generate_secret 32)
JWT_SECRET=$(generate_secret 64)
EOL

echo "Neue .env-Datei generiert!"
```

## 🕵️‍ Überwachung und Doppelprüfung

### `nmap`-Beispiele

#### Innerhalb Ihres Netzwerks testen

```bash

# Scannen Sie Ihren localhost nach allen offenen Ports
nmap -sT localhost

# Scannen Sie die private IP Ihres Geräts nach Diensten
nmap -sV 192.168.1.10

# Erkennen Sie Geräte in Ihrem Netzwerk
nmap -sn 192.168.0.0/24
nmap -sn 10.0.0.0/24
```

#### Tests außerhalb Ihres Netzwerks

Sie können Ihre aktuelle (öffentliche) IP-Adresse einfach mit Diensten wie `ifconfig.me` ermitteln: `curl https://ifconfig.me`.

Verwenden Sie ein externes Netzwerk oder einen Remote-Server, um Ihre öffentlichen IPs zu testen:

```bash

print_current_ip() {
  curl https://ifconfig.me
}

print_current_ip
# --> 123.456.789.012

# Ändern Sie target_host in Ihre öffentliche IP oder Hostname
# Überprüfen Sie den Host mit erweiterten Techniken
nmap -A --open --reason $target_host
nmap -A -F --open --reason $target_host
nmap -A -p1-65535 --open --reason $target_host

```

**Warum beides testen?**  
Tests aus dem Inneren zeigen die interne Exposition auf, während externe Tests Dienste identifizieren, die für Angreifer zugänglich sind.

## 🛡️ Häufige Fehlvorstellungen

1. **Mein lokales Umfeld ist kein Ziel.**  
   - Tatsache: Angreifer können von Ihrem Rechner zu Ihren Produktionsumgebungen wechseln.  
2. **Feuerwände sperren alles ab.**  
   - Tatsache: Sie sperren nur das ab, was Sie konfigurieren.  
3. **Private IPs sind sicher.**  
   - Tatsache: Ausnutzungen wie NAT-Bypass können weiterhin Ihr Netzwerk beeinflussen.
