  <slot name='explanation'>
  <div className="explanation">
    PostgreSQL supporte trois syntaxes de cast :

    1. ANSI SQL : `CAST(expression AS type)`.
    2. PostgreSQL : `expression::type`.
    3. Fonction de type : `type 'literal'`.

    Toutes sont fonctionnellement équivalentes, mais :
    - `CAST()` est la plus portable.
    - `::` est spécifique à PostgreSQL mais couramment utilisé.
    - La syntaxe `type 'literal'` en style infixe est moins courante mais valide.

  options={[
    {text: 'Ils occupent tous les deux 8 octets, mais représentent des sémantiques de timestamp différentes', isAnswer: true},
    {text: 'Ils\'},
    {text: 'timestamptz conserve n\'importe quelle zone horaire d\'entrée'},
    {text: 'timestamptz stocke le nom ou le décalage d\'origine de la zone horaire'},
    {text: 'timestamptz stocke une valeur de 2 octets pour la TZ'},
    {text: 'timestamptz est le successeur de timestamp'},
  ]}
>
  <slot name="question">
  <div className="question">
    Quelle est l'affirmation **la plus précise** concernant `timestamptz` et `timestamp` ?
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Ils sont tous deux de 8 octets, mais ils ne stockent pas le même type de valeur.

    Alors, quelle est la différence ? Elle réside dans l'analyse des entrées.

    **`timestamptz`**
    - Normalise l'entrée en un point absolu dans le temps.
    - Tient compte du paramètre `TimeZone` du serveur/connexion lors de l'analyse des entrées sans décalage explicite et lors de l'affichage de la sortie.

    **`timestamp`**
    - Stocke une date et une heure sans conversion de fuseau horaire.
    - Ne conserve ni ne normalise pas les informations de fuseau horaire.


    **`timestamp`**

    - Stocke la date et l'heure sans informations de fuseau horaire.
    - Utile pour stocker explicitement des dates standardisées, soit en UTC, soit dans un fuseau horaire spécifique.
  </div>
  </slot>
</Challenge>

    ```

          end_time timestamptz
        );
    ```
  </div>

  <div className="explanation">
    Les timestamps PostgreSQL ont une précision de microseconde (6 chiffres après la virgule).

    - Maximum : `.123456` (6 chiffres)
    - Les nanosecondes (9 chiffres) sont arrondies ou tronquées selon la précision prise en charge
    - Les décalages de fuseau horaire sont acceptés pour `timestamptz`, mais non obligatoires

    **Piège peu connu :** Certains langages/frameworks envoient une précision nanoseconde, mais PostgreSQL stocke les timestamps avec une précision microseconde.
  </div>
  </slot>

    À quel moment cette contrainte de vérification est-elle vérifiée ?
    ```sql
        ALTER TABLE students 
        ADD CONSTRAINT valid_grade 
        CHECK (
          (grade >= 0 AND grade <= 100) OR 
          grade IS NULL
        ) NOT VALID;
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `NOT VALID` contraintes :
    - Sont vérifiées immédiatement pour les nouvelles insertions et mises à jour
    - Ne valident pas les lignes existantes
    - Peuvent valider les lignes existantes plus tard avec `VALIDATE CONSTRAINT`
    - Utiles pour les grandes tables

    Sans `NOT VALID` :
    - La contrainte est vérifiée immédiatement
    - Toutes les lignes existantes sont validées
    - Peuvent être lentes sur de grandes tables
