  <div className="explanation">
    PostgreSQL は 3 つのキャスト構文をサポートしています:

    1. ANSI SQL: `CAST(expression AS type)`。
    2. PostgreSQL: `expression::type`。
    3. 型関数: `type 'literal'`。

    すべて機能的には同等ですが、
    - `CAST()` は最も移植性が高い。
    - `::` は PostgreSQL 固有だが一般的に使われる。
    - 中置スタイルの `type 'literal'` はあまり一般的でないが有効です。

        CREATE TABLE student_emails (

        );

    {text: 'timestamptz は timestamp の後継です'},
  ]}
>
  <slot name="question">
  <div className="question">
    `timestamptz` と `timestamp` について、**最も正確な**記述はどれですか？
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    どちらも8バイトですが、同じ種類の値は保存しません。

    では違いは何でしょう？入力のパースにあります。

    **`timestamptz`**
    - 入力を絶対的な時点に正規化します。
    - 明示的なオフセットがない入力や出力表示時に、サーバ/接続の `TimeZone` 設定を考慮します。

    **`timestamp`**
    - タイムゾーン変換なしで日付と時刻を保存します。
    - タイムゾーン情報を保持したり正規化したりしません。


    **`timestamp`**

    - タイムゾーン情報なしで日付と時刻を保存します。
    - UTC でも特定のタイムゾーンでも、標準化された日付を明示的に保存したいときに便利です。
  </div>
  </slot>
</Challenge>

    **あまり知られていない落とし穴:** 一部の言語/フレームワークはナノ秒精度で送りますが、PostgreSQL はマイクロ秒精度で保存します。

    このグレード制約はいつチェックされますか？
    ```sql
        ALTER TABLE students 
        ADD CONSTRAINT valid_grade 
        CHECK (
          (grade >= 0 AND grade <= 100) OR 
          grade IS NULL
        ) NOT VALID;
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `NOT VALID` 制約:
    - 新しい INSERT と UPDATE に対して即座にチェックされる
    - 既存行は検証されない
    - 後で `VALIDATE CONSTRAINT` で既存行を検証できる
    - 大規模テーブルで有用

    `NOT VALID` なしの場合:
    - 制約は即座にチェックされる
    - すべての既存行が検証される
    - 大規模テーブルでは遅くなる可能性がある
  </div>
