    次の型変換のうち、**無効**なものはどれですか ❌？
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    PostgreSQL では3つのキャスト構文がサポートされています:

    1. ANSI SQL: `CAST(expression AS type)`.
    2. PostgreSQL: `expression::type`.
    3. 型関数: `type 'literal'`.

    これらはすべて機能的に同等ですが:
    - `CAST()` は最も移植性が高いです。
    - `::` は PostgreSQL 固有ですが、よく使われます。
    - 中置スタイルの `type 'literal'` はあまり一般的ではありませんが、有効です。
  </div>

    ```sql
        );
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    PostgreSQLのUNIQUE制約:
    - 複数のNULL値を許可します。

    NULL値を防ぐには、`NOT NULL`を追加します:
    ```sql
        CREATE TABLE student_emails (
          student_id INTEGER,
          email VARCHAR(255) NOT NULL,
          UNIQUE(email)
        );
    ```

  <slot name='explanation'>
  <div className="explanation">
    どちらも8バイトですが、同じ種類の値を保存するわけではありません。

    では、違いは何でしょうか？それは入力の解析にあります。

    **`timestamptz`**
    - 入力を絶対的な時点に正規化します。
    - 明示的なオフセットなしで入力を解析する際や出力を表示する際に、サーバー/接続の `TimeZone` 設定を考慮します。

    **`timestamp`**
    - タイムゾーン変換なしで日時を保存します。
    - タイムゾーン情報を保持または正規化しません。


    **`timestamp`**

    - タイムゾーン情報なしで日時を保存します。
    - UTCまたは特定のタイムゾーンで標準化された日付を明示的に保存するのに便利です。

    ```sql
        SELECT 256::numeric * 256 * 256 * 256;
    ```

    **あまり知られていない落とし穴:** 一部の言語/フレームワークはナノ秒精度で送信しますが、PostgreSQLはタイムスタンプをマイクロ秒精度で保存します。
  </div>

    ```sql
        ALTER TABLE students 
        ADD CONSTRAINT valid_grade

          (grade >= 0 AND grade <= 100) OR 
          grade IS NULL
        ) NOT VALID;
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    `NOT VALID`制約:
    - 新しい挿入や更新に対して即時チェックされる
    - 既存の行は検証しない
    - 後で`VALIDATE CONSTRAINT`を使って既存の行を検証できる
    - 大規模テーブルに有用

    `NOT VALID`なしの場合:
    - 制約は即時チェックされる
    - すべての既存行が検証される
    - 大規模テーブルでは遅くなる可能性がある
  </div>
