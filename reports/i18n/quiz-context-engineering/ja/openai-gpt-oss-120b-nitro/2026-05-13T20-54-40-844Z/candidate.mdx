    マルチターンチャットアプリで、会話履歴が長くなったときの正しい戦略は何ですか？
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    LLMには永続的な記憶はありません。すべての API 呼び出しはステートレスで、完全なコンテキストを送信すれば応答が返ってきます。会話の「記憶」は、各リクエストに含めるメッセージ履歴だけです。

    履歴が予算を超えると、選択肢は次のとおりです:
    1. **Summarize**: 古いターンを要約に圧縮し、最近のターンはそのまま残す
    2. **Sliding window**: 直近の N ターンだけを保持し、以前のものは削除する
    3. **Selective retrieval**: 会話ターンを埋め込み、クエリごとに関連するものを取得する（複雑だが強力）

    単純な切り捨て — 収まるように古いメッセージを削る — は最悪の選択肢です。モデルが必要とするコンテキストを黙って失ってしまいます。

    会話履歴のベクトル DB 取得は理論的には魅力的ですが、ほとんどのチャットアプリでは過剰です。要約が実用的なデフォルトです。
  </div>
  </slot>
</Challenge>
