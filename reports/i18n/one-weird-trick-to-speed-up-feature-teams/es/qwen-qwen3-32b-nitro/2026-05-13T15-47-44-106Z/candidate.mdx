Cuando se diseña un nuevo sistema o característica, es fácil quedarse atascado en el diseño de esquemas. En este artículo compartiré un truco útil que me ha dado dividendos a lo largo de mi carrera.

<section class="breakout">
  _Intenta_ el almacenamiento de datos más simple posible al diseñar un nuevo sistema o característica.
</section>

Con mucha frecuencia, veo equipos que recurren a SQL o MongoDB como única opción para el almacenamiento de datos. Claro, nadie pierde el trabajo por elegir SQL. Pero ¿qué pasaría si te dijera que hay una forma más simple, rápida y económica para comenzar?

Un KV o almacén de pares clave-valor podría ser todo lo que necesitas. Algo como Redis o S3.

No siempre es la opción correcta, pero **quizás con más frecuencia de lo que piensas.**

Una capa de almacenamiento simple puede acelerar moderadamente el desarrollo *inicial* al reutilizar código de la capa de datos y evitar costos relacionados con los cambios en el diseño de esquemas y migraciones. Los cambios ocurrirán de todas formas; deja que el código se encargue de ellos tanto tiempo como sea posible. Es mejor evitar tener que lidiar con cambios en dos lugares.

Es probable que obtengas ganancias de rendimiento, ya que las búsquedas por `clave` están altamente optimizadas, y las escrituras pueden beneficiarse de actualizaciones por lotes.

{/* Evita patrones KV si necesitas JOINs o consultar por propiedades en tu conjunto de datos. O en casos donde tengas conjuntos de datos no acotados/infinitamente crecientes. (`Logs`, `Signups`, etc.) */}

## Pensar en Claves

Puede parecer extraño diseñar con un patrón Clave-Valor primero, especialmente si estás acostumbrado a diseñar sistemas con jerarquías de objetos o Diagramas de Entidad-Relación y a implementarlos directamente en SQL.

Has ***usado*** patrones clave-valor antes, ¡probablemente! Están en todas partes, desde configuraciones y URLs hasta Almacenamiento de objetos estilo S3. Cada vez que interactúes con datos mediante un valor `ID` único, ¿qué crees? ¡Otro patrón clave-valor! (Aunque no necesariamente un almacén KV.)

### Diseño con claves

Casi todos los datos _pueden_ representarse utilizando patrones KV. (De hecho, muchas bases de datos de orden superior se basan en patrones KV de nivel inferior.) Veamos algunos ejemplos:

```markdown
user/123          {id: 123, ...}
user/123/block    ['user/456', 'user/789']
user/123/groups   ['admin', 'staff']
user/420/friends  ['user/456', 'user/789']

group/admin       {user: '*:rw'}
group/default     {user: '*:r'}

product/42/discount/<UUID>	{percentOff: '10%'}
product/42/discount/<UUID>	{percentOff: '20%', minTotal: 100.0}
```

Es posible que hayas notado, pero el `ID` a menudo es una clave en sí mismo. ¡Este es un patrón común en almacenes KV! La clave suele ser una combinación del tipo de entidad e identificador único (por ejemplo, `user/123`, `user:456`).

### ¿KVs como gráficos y árboles?

Puede ser útil representar estructuras de datos complejas como Gráficos o Árboles mediante patrones KV. (Nuevamente, las URLs de REST son un gran ejemplo de esto.)

La jerarquía de claves (`user/420` -> `user/420/friends`) codifica naturalmente una relación de gráfico entre el `user` y sus `friends`.

Este es un método rápido y económico para serializar estructuras de datos de gráfico. Especialmente si no necesitas la complejidad de una base de datos de gráficos (como Neo4j).

### Cuándo usar patrones KV

- Cuando necesites escala masiva. (Miles de millones o incluso billones de pares clave-valor.)
- Cuando accedas principalmente a datos mediante una clave única.
- Cuando necesites estructuras de datos simples.
- Cuando tengas datos con una jerarquía, gráfico o estructura de árbol.

### Cuándo evitar patrones KV

No almacenes cosas como comentarios de blog en un par clave-valor _**único**_. Por ejemplo, `post/666 -> {comments: [...too many...]}`. En su lugar, podrías usar `post/666/comments/1`, o `post/666/comments/<UUID>`, etc. O optar por una tabla SQL.

- Cuando necesites buscar por propiedades (no por Clave o ID) en tu conjunto de datos.
- Cuando necesites unir datos a través de múltiples entidades.
- Cuando necesites imponer restricciones o relaciones complejas.

### Cuándo necesitas más que KV

A medida que los requisitos del proyecto evolucionen naturalmente, es posible que necesites hacer más de lo que soporta tu almacén KV. En este punto, deberás considerar migrar a un almacén de datos más complejo.

{/* La buena noticia es que puedes comenzar con un patrón KV y evolucionarlo hacia un sistema más complejo según sea necesario. S3 tiene funciones más allá del almacenamiento simple, desde Athena para buscar archivos, Glacier, y políticas de expiración hay muchas cosas que puedes hacer con él. Además, Redis ha agregado muchas funciones de alto nivel (como Pub/Sub, Geoespaciales, Streams y Conjuntos Ordenados) que pueden ayudarte a satisfacer algunos requisitos. */}

La buena noticia es que migrar un solo almacén KV a SQL es relativamente más fácil que migrar un esquema SQL complejo a un almacén KV. (Con múltiples tablas, índices, restricciones, etc.) He hecho esto muchas veces con un script de 50 líneas.

Anecdóticamente, he encontrado que la calidad de los diseños SQL es mayor si comienzas con un patrón KV primero. Te obliga a pensar realmente en los datos de una manera diferente y a comprender exactamente qué necesitas realmente de SQL.

## Pasos siguientes

La mejor forma de aprender es probarlo. Si estás interesado en explorar este patrón más a fondo, te recomiendo **construir cosas** con Redis, DynamoDB o S3.  
Todos son excelentes almacenes KV con diferentes trade-offs.

### Fact Service - Proyecto de referencia

Echa un vistazo a mi Código Abierto ["Fact Service", un proyecto de referencia en GitHub](https://github.com/justsml/fact-service).  
Es una API RESTful autónoma que implementa un servicio de datos KV.

Cuenta con muchos [adaptadores de datos](https://github.com/justsml/fact-service/tree/main/lib/providers).  
Incluidos para Postgres, Redis, DynamoDB, Firestore y Cassandra! (Con [comandos de Docker](https://github.com/justsml/fact-service/tree/main/lib/providers) para comenzar rápidamente.)

Fact Service está diseñado para ser un proyecto de inicio y aprendizaje, clónalo y construye tu propio servicio de datos KV.
