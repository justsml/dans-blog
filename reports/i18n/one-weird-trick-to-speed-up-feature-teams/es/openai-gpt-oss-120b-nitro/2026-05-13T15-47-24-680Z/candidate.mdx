Al diseñar un nuevo sistema o característica, es fácil quedar atrapado en la definición del esquema. En este artículo compartiré un truco sencillo que me ha dado frutos a lo largo de mi carrera.

<section class="breakout">
  _Intenta_ la persistencia de datos más simple posible al diseñar un nuevo sistema o característica.
</section>

Con demasiada frecuencia veo equipos que optan por SQL o MongoDB como su única opción de almacenamiento. Claro, nadie será despedido por elegir SQL. Pero, ¿qué pasaría si te dijera que hay una forma más simple, rápida y económica de comenzar?

Un almacén KV o de pares clave‑valor podría ser todo lo que necesitas. Algo como Redis o S3.

No siempre es la elección correcta, pero quizá **más a menudo de lo que crees**.

Una capa de almacenamiento simple puede acelerar moderadamente el desarrollo *temprano* reutilizando el código de la capa de datos y evitando costos relacionados con el churn en el diseño de esquemas y migraciones. El churn ocurrirá de todos modos; deja que el código lo gestione el mayor tiempo posible. Es mejor evitar lidiar con cambios en dos lugares.

Es probable que se obtengan ganancias de rendimiento, ya que las búsquedas por `key` están altamente optimizadas, y las escrituras pueden beneficiarse de actualizaciones por lotes.

## Pensando en llaves

Puede resultar extraño diseñar primero con un patrón de clave‑valor, sobre todo si estás acostumbrado a diseñar sistemas con jerarquías de objetos o diagramas entidad‑relación y a implementarlos directamente en SQL.

Probablemente ya ***has usado*** patrones clave‑valor antes. ¡Están en todas partes, desde configuraciones y URLs hasta el almacenamiento de objetos al estilo S3! Cada vez que manejas datos mediante un valor `ID` único, ¿adivinas qué? ¡Otro patrón clave‑valor! (Aunque no sea necesariamente un KV Store.)

### Diseñando con llaves

Prácticamente todos los datos _pueden_ representarse usando patrones KV. (De hecho, muchas bases de datos de nivel superior se construyen sobre patrones KV de bajo nivel.) Veamos algunos ejemplos:

```markdown
user/123          {id: 123, ...}
user/123/block    ['user/456', 'user/789']
user/123/groups   ['admin', 'staff']
user/420/friends  ['user/456', 'user/789']

group/admin       {user: '*:rw'}
group/default     {user: '*:r'}

product/42/discount/<UUID>	{percentOff: '10%'}
product/42/discount/<UUID>	{percentOff: '20%', minTotal: 100.0}
```

Seguramente lo notaste, pero el `ID` suele ser una llave por sí mismo. Este es un patrón común en los KV stores. La llave suele ser una combinación del tipo de entidad y el identificador único (p. ej., `user/123`, `user:456`).

### ¿KVs como grafos y árboles?

Puede resultar útil representar estructuras de datos complejas, como grafos o árboles, usando patrones KV. (De nuevo, las URLs REST son un gran ejemplo de esto.)

La jerarquía de llaves (`user/420` → `user/420/friends`) codifica de forma natural una relación de grafo entre el `user` y sus `friends`.

Es una manera rápida y barata de serializar estructuras de datos de grafo, sobre todo si no necesitas la complejidad de una base de datos de grafos (como Neo4j).

### Cuándo usar patrones KV

- Cuando requieres escala masiva (miles de millones o incluso billones de pares KV).
- Cuando accedes a los datos principalmente mediante una llave única.
- Cuando necesitas estructuras de datos simples.
- Cuando tus datos tienen una jerarquía, grafo o estructura de árbol.

### Cuándo evitar patrones KV

No almacenescosas como comentarios de blog en un _**único**_ par KV. Por ejemplo, `post/666 -> {comments: [...too many...]}`. En su lugar podrías usar `post/666/comments/1`, o `post/666/comments/<UUID>`, etc. O recurrir a una tabla SQL.

- Cuando necesites buscar por propiedades (no por clave o ID) en tu conjunto de datos.  
- Cuando necesites REALIZAR JOIN entre datos de múltiples entidades.  
- Cuando necesites imponer restricciones o relaciones complejas.

### Cuando necesitas más que KV

A medida que los requisitos del proyecto evolucionan naturalmente, puede que necesites hacer más de lo que tu almacén KV permite. En ese punto tendrás que considerar migrar a un almacén de datos más complejo.

{/* La buena noticia es que a menudo puedes comenzar con un patrón KV y evolucionarlo a un sistema más complejo según sea necesario. S3 ofrece funcionalidades más allá del simple almacenamiento, desde Athena para buscar archivos, Glacier y políticas de expiración; hay mucho que puedes hacer con él. Además, Redis ha añadido muchas características de alto nivel (como Pub/Sub, Geo‑spatial, Streams y Sorted Sets) que pueden ayudarte a cumplir algunos requisitos. */}

La buena noticia es que migrar un único almacén KV a SQL es relativamente más fácil que migrar un esquema SQL complejo a un almacén KV. (Con múltiples tablas, índices, restricciones, etc.) Lo he hecho muchas veces con un script de 50 líneas.

Anecdóticamente, he observado que la calidad de los diseños SQL es mayor cuando se comienza con un patrón KV. Obliga a pensar los datos de una forma distinta y a comprender _exactamente_ lo que realmente se necesita de SQL.

## Próximos pasos

La mejor manera de aprender es probarlo. Si te interesa profundizar en este patrón, te recomiendo **construir cosas** con Redis, DynamoDB o S3.  
Todos son excelentes almacenes KV con diferentes compensaciones.

### Fact Service – Proyecto de referencia

Echa un vistazo a mi proyecto de código abierto ["Fact Service", un proyecto de referencia en GitHub](https://github.com/justsml/fact-service).

Se trata de una API RESTful independiente que implementa un servicio de datos KV.

Incluye numerosos [adaptadores de datos](https://github.com/justsml/fact-service/tree/main/lib/providers), entre ellos para Postgres, Redis, DynamoDB, Firestore y Cassandra. ¡Todo con los [comandos de Docker](https://github.com/justsml/fact-service/tree/main/lib/providers) necesarios para arrancar rápidamente!

Fact Service está pensado como un proyecto de inicio y aprendizaje; ¡forkéalo y construye tu propio servicio de datos KV!
