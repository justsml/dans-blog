## Generative UI が何ではないか  
---

定義する前に、まず区別すべき3つの概念があります：

**設計時コード生成** — v0、Lovable、Bolt、Cursor が React コンポーネントを生成する処理。これらのツールは開発者がレビューしてコミットするコードを生成します。AI は開発中に動作します。ユーザー視点では、配信されるものは静的です。これは非常に有用なツールカテゴリですが、「ランタイムジェネレーティブUI」とは異なります。

**AI支援フォーム自動入力** — モデルがコンテキストからフィールド値を自動入力する処理。インターフェースの構造は固定されたままです。変化するのはコンテンツのみです。これは実用的なパターンですが、ジェネレーティブUIではありません。

**AIによる生HTMLのページ挿入** — モデルが `<div>` や `<button>` の文字列を出力し、`innerHTML` や `dangerouslySetInnerHTML` 経由で挿入される処理。これは技術的定義において**ランタイムジェネレーティブUI**に該当します。しかし最も危険な実装形態でもあり、この分野で成熟したフレームワークが存在する理由そのものです。生のAI生成マークアップはXSSリスク、アクセス不能な属性、一貫性のないスタイリング、幻覚構造を引き起こします。この記事の残りは、これを上回る方法について述べています。

## 功能的な定義

ランタイムでのジェネレーティブUIとは、**モデルが会話またはタスクの状態に基づき、ユーザーに表示するインターフェースコンポーネントまたはコンポーネントの組み合わせを決定すること**を意味します。  

言葉ではない。インターフェースです。  

最も単純なケース：フライト予約のアシスタントが`search_flights`ツールを呼び出します。通常は「3つのオプションがあります…」といった平文を返す代わりに、選択可能なフライト、座席クラスの切り替え、および「予約」ボタンを含む`<FlightResultsCard>`コンポーネントをレンダリングします。モデルがこのケースで構造化されたカードが適切な応答であると判断しました。開発者はそのカードの外観や「予約」ボタンの動作を定義しました。

より複雑なケース：財務分析エージェントがポートフォリオに関する質問を受け、`MetricGroup`で主要な数値を表示し、`RiskBreakdown`チャート、`ScenarioComparison`テーブル、`PolicyNotice`を組み合わせた応答を構成します。モデルは事前承認済みコンポーネントのカタログからそのレイアウトを組み立てました。開発者はすべてのコンポーネントを定義しました。モデルはどのコンポーネントを使用するか、そしてその中に何のデータを入れるかを選択しました。

両方のケースはジェネレーティブUIです。違いはモデルに与えられた構成の自由度にあり、これは出力の豊かさと、何かが間違える可能性の複雑さを同時に決定します。

## 3つのパターン

### パターン1: ツール呼び出しによるコンポーネントレンダリング

モデルが名前付きツールを呼び出します。アプリケーションにはツール名からコンポーネントへのマップがあります。ツール呼び出しがコンポーネントのレンダリングをトリガーします。

```tsx
// The model calls: { name: "show_flight_results", args: { flights: [...] } }

useCopilotAction({
  name: "show_flight_results",
  render: ({ args }) => <FlightResultsCard flights={args.flights} />,
});
```

これは最も安全なパターンです。レイアウトはモデルから決まることがありません。モデルはコンポーネントを表示する *タイミング* と *何のデータ* を詰め込むかを決定します。開発者は依然としてコンポーネントコード、ビジュアルデザイン、アクセシビリティ実装、レンダリングロジックのすべてのエッジケースを所有しています。

Vercel AI SDKの `useChat` は `tool` ハンドラでこれを行います。assistant-uiのツールレンダリングもこれです。CopilotKitの「Static Generative UI」がこのパターンです。明確に動作するプロダクション環境のコパイロットUIは、このパターンを採用しています。

**適切な使用場面**: 開発時に表示したいもののセットが把握できる場合に適しています。予約確認、検索結果、アカウントサマリー、承認ウィジェット。シナリオを列挙できるなら、このパターンが対応します。

### パターン 2: コンポーネントカタログ構成

モデルは、開発者が定義したカタログからコンポーネントを参照する型付きJSONツリーを出力します。フロントエンド側には、ツリーを走査して各コンポーネントをインスタンス化するレンダラーがあります。

```json
[
  { "type": "metric_group", "metrics": [
    { "label": "MRR", "value": "$82,400", "delta": "+12%" },
    { "label": "Churn", "value": "2.1%", "delta": "-0.4%" }
  ]},
  { "type": "line_chart", "title": "30-day growth", "data_ref": "mrr_series" },
  { "type": "insight_callout", "text": "Expansion revenue driving the delta — avg seat count up 18%." }
]
```

このレイアウトはモデルが構成しました。`MetricGroup`、`LineChart`、`InsightCallout`といったコンポーネントです。しかし、各コンポーネントタイプが何を意味するか、どのプロップを受け付けるか、どうレンダリングされるかはすべてあなたが定義しています。モデルが `{ "type": "custom_untested_thing" }` のような出力を試みた場合、スキーマ検証がそれを検出し、レンダラーは無視または拒否します。

これは`json-render`、`A2UI`、`Hashbrown`、`OpenUI`、`Tambo`の背後にあるパターンです。鍵となるエンジニアリングの作業は**カタログ設計**です——どのコンポーネントタイプが存在するか、そのスキーマがどのようになっているか、モデルが組み合わせてよい/悪いものを決定することです。

**適切な場面**: 表示したい構造がデータやユーザーのリクエストに応じて正当に変化する場合。数値の特徴に応じて表示構造が変化するダッシュボード。コンテキストに応じて異なるセクションを表示するレポート。エージェントがどのステップにいるかによって変化するワークフローのパネル。

### パターン3: 無制限生成

モデルは、厳格なコンテンツセキュリティポリシー（CSP）を持つサンドボックス付きiframe内でHTML、SVG、Canvas、WebGLを生成します。

これは、固定されたコンポーネントカタログでは対応できないユースケースに適しています。アルゴリズムの可視化、アーキテクチャ図、即席のチャート、ジェネレーティブアート、教育用シミュレーションなどが該当します。iframe境界がセキュリティの役割を果たしています。この境界を取り除くと、記事冒頭で述べた生HTML挿入問題に戻ってしまいます。

`CopilotKit/OpenGenerativeUI`がこのパターンにおける現時点での最良の参考実装です。サンドボックスはスクリプトを除去し、メッセージングを制限し、生成されたアーティファクトをアプリケーションの特権状態から隔離します。

**適切な使用時**: 本当に任意の視覚出力が必要な場合 — 一時的な説明図、動的シミュレーション、創造的アーティファクトなど。トランザクショナルUIには使用しないでください。チェックアウト確認画面はサンドボックス付きiframeを必要としません。

### 3つのパターンを超えて: LLMがピクセルを直接駆動

これらすべてのパターンにすっきりと収まらない新たな第4の方向性が登場しています: LLMが**没入型、ゲームのような体験**を構築するために、サンドボックス付きiframeよりも直接的に視覚出力を制御する方法です。

生成型UIにおける典型的な区別は **iframe HTML と JSON カタログ** です:

- **iframe HTML** — モデルが隔離されたサンドボックス内でレンダリングされるHTML、SVG、Canvas、WebGLを生成します。表現の自由が最大で、セキュリティはiframeの境界に完全に依存します。例: Anthropic Artifacts、OpenGenerativeUI。
- **JSON カタログ** — モデルは開発者が定義したコンポーネントカタログに制約された構造化されたペイロードを出力します。あなたのレンダラがその仕様から信頼できる事前に構築されたコンポーネントをインスタンス化します。モデルが*何を表示するか*を決定し、あなたが*どのようにレンダリングするか*を決定します。例: json-render、A2UI。

これらに加えて、非常に最近のデモはモデルがコンポーネントを選択したりサンドボックス付きHTMLを書いたりするのではなく、キャンバスをより直接的に駆動する第3のモードを示唆しています。[TencentのHunyuanWorld](https://arxiv.org/abs/2502.01999)（単一画像から探索可能な3D環境を生成）や、LLMがコンポーネントカタログを呼び出す代わりにランタイムでマップ、NPC、クエストを生成するゲームアーキテクチャは、モデルがフォームレンダラーではなくゲームディレクターに近づいていく未来を示唆しています。WebGPUを介したブラウザ内LLM推論（[WebLLM](https://mlc.ai/web-llm/)）は、同じフロンティアをローカルで押し進めています。

この分野は確かに興奮を呼び、確かに初期段階にあります。まだ本番環境の製品を構築するための安定したフレームワークはありません。それが変わるまで、このアプローチについては専用の記事で詳しく取り上げます。

## プロトコル: AG-UI と A2UI

AG-UI と A2UI はプロトコル層における主要な2つの基準です。これらは異なる問題を解決し、競合関係にはありません。  

### AG-UI

**GitHub**: [ag-ui-protocol/ag-ui](https://github.com/ag-ui-protocol/ag-ui)

AG-UI は、AIエージェントとフロントエンドアプリケーション間の通信を実現するイベント駆動型プロトコルです。`TEXT_MESSAGE_START`、`TEXT_MESSAGE_CONTENT`、`TOOL_CALL_START`、`TOOL_CALL_END`、`STATE_SNAPSHOT`、`STATE_DELTA` など、約16種類のイベントタイプを定義しています。トランスポート方式は任意で選択可能で、SSE、WebSockets、webhooks いずれも利用可能です。フォーマットは広範な採用を目的として意図的に緩やかに設計されています。

AG-UI は、UIがどのように見えるかを定義するものではありません。エージェントがフロントエンドと*どのように*通信するかを定義します。これは、ReactアプリがLangGraphエージェントをCrewAIエージェントと同じ方法で購読できるようにするワイヤプロトコル層のようなものです。フロントエンドコードを変更することなく、LangGraphエージェントとCrewAIエージェントの両方にサブスクライブできる仕組みです。

CopilotKitは、LangGraphやCrewAIでの実装経験からAG-UIを開発しました。このプロトコルはLangChain、Mastra、PydanticAIなど複数のプロジェクトで採用されており、MicrosoftもAG-UIの統合ガイドを公開しています。マルチエージェントフロントエンドを開発し、バックエンドフレームワークとフロントエンドコードを分離する必要がある場合、AG-UIが答えです。

**1つの重要な注意点**: AG-UIはUIフレームワークではありません。描画内容を指示しません。エージェントが何か発言した、ツールを呼び出した、共有状態を更新したといった事実を通知するだけです。そのイベントに対する描画内容は引き続きあなたの決定に委ねられています。

### A2UI

**GitHub**: [google/A2UI](https://github.com/google/A2UI) · Spec: [a2ui.org](https://a2ui.org/)

A2UIは、エージェントがUIを表示したいときに送信する内容を定義したGoogleの宣言型仕様です。AG-UIが「エージェントはどのように通信するか？」に答えるのに対し、A2UIは「エージェントがコンポーネントレイアウトを記述する際のフォーマットは何か？」に焦点を当てます。

A2UIはフラットなJSONL形式を使用します。1行につき1つのコンポーネント記述子があり、それぞれにID、タイプ、データが含まれます。このフラット構造は意図的な設計です。ネストされたツリー構造では、モデルが全体構造を把握した上でストリーミングを開始する必要があります。一方で、フラットリスト形式ではモデルが「思考」を進めるにつれて各コンポーネントを順次出力できるため、モデルがチャートの追加を検討している間に、フロントエンドはすでに最初のメトリクスカードのレンダリングを開始できます。

```jsonl
{"id":"h1","type":"kpi_card","title":"MRR","value":"$82,400","delta":"+12%"}
{"id":"h2","type":"kpi_card","title":"Churn","value":"2.1%","delta":"-0.4%"}
{"id":"c1","type":"line_chart","title":"30-day MRR","data_ref":"mrr_series"}
{"id":"t1","type":"data_table","cols":["Month","MRR","Net New"],"data_ref":"monthly"}
```

A2UIはセキュリティに配慮された設計です。この仕様は実行可能なコードではなく、データ形式です。コンポーネントカタログは開発者が事前に定義しており、エージェントはそのカタログに含まれるタイプのみを参照可能です。A2UIレンダラーが未知のタイプ名を受け取った場合、それを無視します。

CopilotKitの「Open-JSON-UI」形式はA2UIと互換性があります。今日コンポーネントカタログ用の仕様形式を選ぶなら、A2UIは最も広範なクロスプラットフォームサポートを持つ形式です。

**安定性に関する注意点**: A2UIは1.0リリース前で、2026年5月8日に確認した時点ではv0.9でした。マイナーバージョン間で破壊的変更を含む仕様変更が含まれています。Googleのロードマップに関するコミュニケーションは不規則で、一部のレンダラー（Lit、Flutter）は仕様更新にリードが遅れています。現在A2UIを基盤に構築する場合、仕様の変化に時間的余裕を確保してください。純粋なWeb用途では、json-renderがより完全なツールチェーンを持っているようです。A2UIの長期的な利点は、json-renderにはないクロスプラットフォーム対応（Web、Flutter、SwiftUI、Android）です。

### MCP Apps

**GitHub**: [modelcontextprotocol](https://github.com/modelcontextprotocol) · 関連: [mcp-ui](https://github.com/MCP-UI-Org/mcp-ui)

MCPは当初、大規模言語モデル(LLM)をツールやデータに接続するためのプロトコルとして設計されました。Apps拡張機能により、MCPツールは単なるデータだけでなく、インタラクティブなUIアーティファクト（Reactコンポーネント、フォーム、ダッシュボード、地図など）も返却できるようになります。

セキュリティモデルは厳格であり、それが目的です。すべてのレンダリングは権限が削除されたサンドボックス化されたiframe内で行われ、テンプレートは事前に宣言されてホストアプリがレビューできるようになっており、すべての通信は監査可能なJSON-RPCで行われます。これはツールプロバイダーにとって正しいモデルです。Shopify MCPサーバーはチェックアウトウィジェットを返すことができ、地図サービスは埋め込み可能な地図を返すことができます。ホストアプリケーションはそのウィジェットのコードを所有せず、信頼もしていません。

MCP AppsはUIが*ツールプロバイダーに所属*する場合に最適な選択肢です。アプリケーションのドメインに属するUIについては、パターン1または2を使用してください。

CopilotKit はエージェントネイティブなフロントエンドアプリケーション向けに最も包括的なフレームワークです。AG-UI 経由でエージェントバックエンドに接続し、双方向会話状態を管理し、ジェネレーティブUIコンポーネントをレンダリングし、エージェントとユーザーが同じデータを変更できる共有状態のパイプラインを提供します。

### Vercel AI SDK

**GitHub**: [vercel/ai](https://github.com/vercel/ai) · Docs: [ai-sdk.dev](https://ai-sdk.dev/)

Vercel AI SDK は AI アプリケーションにおける事実上の TypeScript 基盤です。特にジェネレーティブ UI については、

```tsx
const { object: dashboard } = useObject({
  api: "/api/generate-dashboard",
  schema: z.object({
    title: z.string(),
    metrics: z.array(z.object({ label: z.string(), value: z.number() })),
    insights: z.array(z.string()),
  }),
});
```

**ツールハンドラー付きの`useChat`** → パターン1。モデルがツールを呼び出し、ツール名をコンポーネントにマッピングします。

**AI Elements** ([elements.ai-sdk.dev](https://elements.ai-sdk.dev/)) は、SDKと組み合わせて使用できる準備済みのUIプリミティブを提供します。

**ここでの混乱を招く軌道に関する注記**: 2024年10月、Vercelは[GitHub Discussion #3251](https://github.com/vercel/ai/discussions/3251)で、SDK 3.0の主要な「Generative UI」機能として宣伝されたReact Server Components (RSC) ストリーミングパターンであるAI SDK RSCが、「いくつかの長期にわたる制約」により「直近の良い解決策が存在しない」ため、無期限に一時停止されることを発表しました。RSCストリーミングを製品戦略の基盤としていたチームは予期せぬ状況に置かれました。`generateObject`/`streamObject` APIもSDK 6.0で非推奨とされました。AI SDK RSCからの推奨される移行方法は、上記の`useObject`パターン、またはカタログベース生成のためのjson-renderです。

### assistant-ui

**GitHub**: [assistant-ui/assistant-ui](https://github.com/assistant-ui/assistant-ui)

assistant-uiは、本番品質のチャットインターフェースを構築するための組み合わせ可能なReactプリミティブのセットです。洗練されたチャットUX（メッセージバブル、ストリーミングトークン、コピー/編集/再生成アクション、思考状態など）が必要で、独自のバックエンドとツールレンダリングを提供したい場合、これが最適な選択肢です。

OpenAI、Anthropic、ローカルモデル、カスタムエンドポイントなど、任意のバックエンドと組み合わせて使用可能で、ツール呼び出しのレンダリングは馴染みのあるスロット/レンダープロップモデルで処理します。

### json-render

**GitHub**: [vercel-labs/json-render](https://github.com/vercel-labs/json-render) · Docs: [json-render.dev](https://json-render.dev/)

json-renderは、パターン2を実装するためのオピニオンが強い「すべてが揃った」アプローチで運用しています。提供されるのは、事前に構築済みのコンポーネントカタログ（shadcn/uiコンポーネントとZodスキーマ）、レンダラ、およびモデルがスキーマによってカタログに制約される生成ループです。

特徴的な点：
- **マルチターゲットレンダリング**：同じJSON仕様でReactウェブアプリ、React Nativeモバイルアプリ、PDF、HTMLメール、Remotion動画にレンダリング可能です。これはレポート生成において特に有用です。
- **進行型レンダリング**：コンポーネントはモデルがストリーミングするにつれて表示され、仕様全体が到着した後に表示されるわけではありません。
- **厳密なスキーマ制約**：カタログはモデルが「有効だが未知のコンポーネントタイプ」を生成できないように設計されています。

ダッシュボードやレポート生成機能を開発しており、独自のコンポーネントカタログ設計に必要なインフラ構築をスキップしたい場合、json-renderはウェブアプリケーションにおける最も速い実装方法です。

**動向に関して**: json-renderは2026年初頭にVercel Labsからリリースされ、標準的なReact/Next.jsプロジェクトで即座に利用可能であるため、ウェブ開発者の注目を集める速度で広がっています。ただしjson-renderはまだプレリリースであり、json-renderとA2UIの関係性は今後調整中です。VercelはA2UI互換の出力形式も試験的に実装しており、将来的に両者の収束が可能であることを示唆しています。クロスプラットフォーム（ネイティブモバイル、複数フレームワーク）を目標とする場合は、A2UIがより長期的な選択肢として適しています。

### Hashbrown

**GitHub**: [liveloveapp/hashbrown](https://github.com/liveloveapp/hashbrown)

Hashbrownは独自のアプローチを採用しています。AIインターフェースの専用レイヤーを構築するのではなく、既存のReactやAngularアプリケーションにAIコンポーネント選択を直接組み込みます。アプリケーションのコンポーネントをLLMに公開し、LLMが描画すべきコンポーネントを選択し、クライアントサイドのツールを呼び出すことも可能です。

これは「チャット」ではない製品UIに知能を注入したい場合に適したツールです。例えば、レイアウトを適応させる製品ページ、適切なオプションを表示する設定パネル、次のステップを提案するワークフロー編集器などに活用できます。

### OpenUI

**GitHub**: [thesysdev/openui](https://github.com/thesysdev/openui) · Docs: [openui.com](https://www.openui.com/)

OpenUIはJSONを「行単位のコードライク形式」（OpenUI Lang）で置き換え、プログレッシブレンダリングとトークン効率を目的としています。複雑なレイアウトにおいて同等のJSONと比較して約67%のトークン削減を主張しています。

トレードオフはエコシステムの成熟度です。OpenUIはより新しい技術であり、JSONベースのアプローチに比べてツールの選択肢が少ないです。ただし、トークンコストが重要な制約になる場合で、複雑なレイアウトを高頻度で生成する必要があるなら、フォーマット効率は確かに現実的です。

### Tambo

**GitHub**: [tambo-ai/tambo](https://github.com/tambo-ai/tambo)

Tamboは状態を持ったコンポーネント選択に焦点を当てています。AIがコンポーネントを選択し、クライアントサイドツールを通じてそれらと相互作用することで、会話全体でコンポーネントの状態を維持します。これは、UI要素がターンをまたいで保持される必要があるユースケース（ユーザーがフィルターを調整しながらAIがフィルタリングされたデータを継続的に分析するフィルターコンポーネントなど）に適しています。

## Pythonレイヤー

PythonエコシステムはAIインターフェースを異なるアプローチで扱います。これらのツールはMLモデルのデモ、データアプリケーション、内部ツールの最適化に設計されており、エージェント駆動のレイアウト構成を必要とする本番環境向け消費者アプリケーションには向いていません。  

これは否定的な意味ではありません。適切なユースケースにおいては、GradioとStreamlitが唯一必要なツールです。  

### Gradio

**GitHub**: [gradio-app/gradio](https://github.com/gradio-app/gradio) · PyPI: `gradio`  

Gradioのコア価値は、Python関数を記述すればGradioがそれをウェブUIでラップすることです。`Interface`クラスは画像分類器に3行で実装できます。`ChatInterface`はチャットボットに10行で実装可能です。`Blocks`は必要に応じてレイアウトの微調整を許容します。  

Gradioにおける「Generative UI」はモデルではなくPython開発者によって定義されます。コンポーネントの可視性や設定はモデルの出力に応じて動的に変化できますが、コンポーネントカタログ自体は静的です。モデルにレイアウト構成を依頼しているわけではありません。

GradioはHuggingFace SpacesおよびMLデモエコシステムのデフォルトです。毎月何百万ものダウンロードがあり、AIデモの広範な領域を支えています。  

**Gradioを使うべきとき**: Python開発者がMLモデルのデモ、研究プロトタイプ、または内部ツールを構築しており、JavaScriptを触りたくない場合です。  

### Streamlit

**GitHub**: [streamlit/streamlit](https://github.com/streamlit/streamlit)

Streamlitの設計はより明確で、各インタラクションごとにPythonスクリプトがエンドツーエンドで実行されます。`st.chat_message()`、`st.dataframe()`、`st.plotly_chart()`といった関数を呼び出すだけで、フレームワークがレイアウトを自動的に処理します。

フルスクリプト再実行モデルは非効率に聞こえるかもしれませんが、会話履歴を蓄積するAIチャットボットには予想外に使いやすく、スクリプト全体が再実行され、会話履歴はセッション状態に保持され、出力は決定論的になります。Streamlitは現在、主要なLLMプロバイダーのほとんどに公式サポートを提供し、Snowflake Cortexとネイティブに統合されています。

**Streamlitを活用するべき場面は**: PythonでAIを活用したデータアプリ、内部レポートツール、MLベースのダッシュボードを開発し、最もシンプルなデプロイメントパスを求める場合です。

### LangChain と Haystack

これらはバックエンドオーケストレーションフレームワークであり、UIフレームワークではありません。どの誠実なジェネレーティブUIスタックマップにも登場するのが当然です。なぜなら、これらは通常、フロントエンドに送信される前に構造化された出力が生成されるレイヤーだからです。

**LangChain** ([langchain-ai/langchain](https://github.com/langchain-ai/langchain)): 任意のLLMに`.with_structured_output()`を適用すると、Pydantic制約付きJSON生成が可能になります。`@tool`デコレータによる自動スキーマ生成は、モデルが呼び出すことのできるツールを定義する最もシンプルな方法です。LangChainは、使用するフロントエンドレイヤーに構造化結果を供給します。

**Haystack** ([deepset-ai/haystack](https://github.com/deepset-ai/haystack)): モジュール型パイプラインアーキテクチャと強力なRAGサポートを備えています。HayhooksはHaystackパイプラインをHTTPエンドポイント（MCP互換エンドポイントを含む）としてラップします。ジェネレーティブUIにリトリーバルバックボーンが必要な場合、Haystackのパイプラインアーキテクチャがこれをクリーンに処理します。

いずれのフレームワークもUI階層を所有しません。これらは、あなたのフロントエンド（パターン1、2、または3）がレンダリングするデータを生成するものです。

## エコシステムの速度と不安定な領域

この分野は急速に進化しており、いくつかのプロジェクトがコードとともに混乱を招くコミュニケーションを発表しています。最終確認日：2026年5月8日。ここに記載のプロジェクト状況に関するメモはタイムスタンプ付きの読み取り専用情報であり、永久的な判断ではありません。

**Vercel AI SDK RSC** はSDK 3.0リリース時のフラグシップとなるGenerative UI機能でした。Vercelは2024年10月にその開発を一時停止し（[Discussion #3251](https://github.com/vercel/ai/discussions/3251)）、React Server Componentsのアーキテクチャ的制限が直近の解決策を持たないことを理由にしました。これを利用して構築していたチームは当然ながら不満を抱えていました。この機能はドキュメントに残されていますが、推奨されるアプローチではありません。`useObject` が推奨されます。

**json-render**（Vercel Labs）は新しい方向性として登場しました。これはRSCの結合問題を回避する、カタログベースでフレームワークに依存しない代替案です。これは1.0リリース前であり、React/web開発者コミュニティで初期段階から強い関心が寄せられています。DX（開発者体験）の理由としては、json-renderは標準的なReact/Next.jsプロジェクトで即座に利用可能であるのに対し、A2UIのクロスプラットフォーム対応はセットアップに摩擦を生みます。両方の仕様が成熟するにつれてどう展開するかは現時点では明確ではありません。Vercelはjson-render内でA2UIの互換性を検証しており、統合が可能であることを示唆しています。

**A2UI**（Google）は1.0リリース前（最終確認時でv0.9）で、マイナーバージョン間で互換性を破る変更が発生し、Googleのロードマップに関するコミュニケーションも不一貫です。json-renderが対応していないクロスプラットフォーム対応（Web + Flutter + SwiftUI）を実現するには適した選択肢であり、企業向けの支援も明確です。純粋なWebプロジェクトにおいては現在の開発者体験（DX）はやや厳しいです。

**AG-UI**（CopilotKit）もまた1.0リリース前です。最も一般的な誤解は、名前からUIフレームワークのように思える点です。実際にはそうではありません——これはトランスポートプロトコルです。AG-UIはエージェントバックエンドとフロントエンド間のイベントフローを定義しますが、それに対するレンダリングの実装は依然として開発者の責任です。この概念モデルは堅実で広く採用されていますが、1.0リリース前の仕様であるためエッジケースがまだ整理中です。

現実的な結論：**ここに挙げられた主要プレイヤーすべてが1.0リリース前**です。API変更に備えて計画を立てる必要があります。パターン（ツールからコンポーネントへのマッピング、カタログ構成、サンドボックス生成）はすでに十分安定して構築可能です。一方、具体的なプロトコル選択についてはそうではありません。

## コンポーネントカタログ設計：本質的なエンジニアリング作業

パターン2における興味深い複雑さの多くはレンダラーにではなく、カタログにあります。

カタログは**スキーマとしてエンコードされた製品決定**です。これは「このドメインにおける意味のあるUIオブジェクトとは何か？」という問いに答えます。"既存のReactコンポーネントは何か？"ではなく、"このコンテキストにおけるユーザーが実際に視覚化・操作すべき本質的な要素とは何か？"という問いに答えます。

**過剰に細分化された失敗モード**：`Row`、`Column`、`Text`、`Button`、`Icon`といった基本要素を公開してしまうと、モデルはフロントエンドエンジニアを模倣する必要があります。結果として、設計システムと整合性のない平凡なレイアウトが生成され、空状態の考慮が欠如し、アクセシビリティに配慮されていないマークアップが生じ、カタログに制約がないため返答ごとに視覚言語が不統一になります。

**過剰に粗い失敗モード**：`WeatherCard`、`FlightCard`、`HotelCard`といった事前定義されたカードのみを公開すると、ユーザーが既存のカードにマッピングできないリクエストをした場合にモデルは適応できなくなり、テキスト出力に後退します。

**有用な中間領域**：ドメインレベルのコンポーネントと制約付きスロット。  

旅行アプリのカタログは次のような構成になるかもしれません：  

```
TripSummary         — 旅程の概要  
FlightOptionList    — 価格付きの選択可能なフライトオプション  
HotelComparison     — 並列表示されるホテルカード  
TravelerForm        — 旅行者情報を収集するフォーム  
PolicyNotice        — 規制/運賃規則の強調表示  
BookingConfirmation — 最終確認とアクションボタン  
```  

金融アプリのカタログは次のような構成になるかもしれません：

```
PortfolioSnapshot   — key positions and P&L  
TransactionTable    — filterable, paginated transactions  
RiskBreakdown       — allocation and volatility metrics  
ScenarioComparison  — side-by-side scenario modeling  
ApprovalGate        — action requiring human confirmation  
```  

このカタログはあなたの製品の語彙のように聞こえるかもしれません。UXの設計決定、アクセシビリティの要件、空状態の処理、危険なアクションのパターンをコンポーネントコードにエンコードしています。モデルはそのピースを配置しますが、それぞれのピースの見た目や許容される動作は依然としてあなたが決めます。  

**ハルシネーションを減らすスキーマ設計のルール**：  

1. enum値は短くて明確にし、`"type": "bar_chart"` ではなく `"type": "data-visualization-bar-type-vertical"` とはしない。  
2. 無効なコンポジションを不可能にせよ。`PolicyNotice` がレイアウトの最後にしか現れない場合、どこにでも現れる可能性のあるコンポーネントと同じスキーマレベルに置かない。  
3. 必須フィールドを多用せよ。オプションフィールドはモデルが省略し、レンダラがnullとして処理しなければならないフィールドである。  
4. リリース前にカタログを現実のプロンプトでテストせよ。生成された仕様を保存し、スキーマ違反、ハルシネーションされたフィールド値、技術的には有効だが意味論的に誤ったコンポジションをチェックせよ。

## 一般的なトラップ  

**トラップ: 有効なJSONを安全な動作と誤認する**。  
スキーマ検証は構造を確認します。しかし、ボタンに付随するアクションがラベルと一致するか、合計値が導出元のデータと一致するか、UIコンポーネントがユーザーが意図しなかった動作をしていないかについては何も保証しません。生成されたUI仕様はスキーマ検証だけでなく、意味論的なレビューが必要です。最低限、破壊的アクションは確認コンポーネントを経由するべきであり、そのコンポーネントのラベルはトリガーするアクションに対してテストされるべきです。

**トラップ: 設計プリミティブではなく製品プリミティブを提供しないこと。** モデルが16pxと20pxのパディングどちらを使うかを決定しなければならない場合、それは誤った抽象化レベルを与えています。ドメインコンポーネントは製品の品位をエンコードすべきです。モデルは挙動の組み合わせを行うべきであり、プレゼンテーションの詳細を管理すべきではありません。  

**トラップ: 静的UIで十分な場面でジェネレーティブUIを使用すること。** 表示する構造が開発時時点で既知である場合（通常そうですが）は、事前に構築されたコンポーネントを使用するパターン1がより速く、安全で、一貫性があります。ジェネレーティブUIは構造が実際にデータやタスクコンテキストに応じて変化する場合にのみその複雑さに値します。  

**トラップ: アクセシビリティの検証をスキップすること。** LLMはWCAG違反をハロシネーションします。インタラクティブ要素に`role="region"`を誤って付与し、ラベルのないフォームを生成し、WCAG AA基準を満たさないコントラスト比を出力します。コンポーネントライブラリ自体が完全にアクセシブルでも、そのコンポーネントのAI生成された組み合わせは自動的にアクセシブルとは限りません。完全なレンダリングパスをテストし、個別のコンポーネントだけでなく全体の構成を検証してください。

**トラップ: プロトコルとフレームワークを混同すること。** AG-UIはフロントエンドフレームワークではありません。A2UIはReactライブラリではありません。これらはワイヤーフォーマットおよびイベントプロトコルです。これらを実装するには、依然としてフロントエンドフレームワークが必要です。CopilotKitはAG-UIおよびA2UIを実装しています。json-renderはA2UI/Open-JSON-UIカタログパターンを実装しています。これらは異なるレイヤーです。

## ユースケース別の推奨事項

**既存のSaaSアプリケーションにコパイロットを追加する**: パターン1（ツール→コンポーネント）から始めること。Vercel AI SDKの`useChat`やCopilotKitを使用する。エージェントの上位5～10のアクションを事前に構築されたコンポーネントにマッピングする。それをリリースし、測定してから、ユーザーが明らかに豊かなコンポジションを必要とする場合にのみカタログを拡張する。  

**自然言語からのダッシュボード生成**: json-renderやカスタムA2UIカタログを用いたパターン2を使用する。チャートタイプ、メトリックカード、テーブルのバリエーションをカバーする8～15種類のコンポーネントタイプのカタログを定義する。スキーマをモデルに供給し、レイアウトを組み立てさせる。レンダラーに到達する前に未知のタイプを検出する検証を構築する。  

**マルチエージェントフロントエンド**: AG-UIとCopilotKitを使用する。イベントストリームはエージェントバックエンド間のリアルタイムストリーミングを処理し、共有状態はエージェント間のハンドオフを管理し、HITLパターンは承認ゲートを処理する。

**ChatGPTや他のMCPホスト内で構築**: MCPアプリを使用する。ツールをデータツールとして定義し、データの取得と推論を行う。別途、ウィジェットのリクエストを行うレンダリングツールを用意する。ウィジェットテンプレートにビジネスロジックを持たせない。  

**MLモデルのデモとデータアプリ（Pythonチーム）**: デモはGradio、HuggingFace Spacesを使用。複雑なインタラクションを必要とするデータアプリにはStreamlitを用いる。どちらもJavaScript不要。  

**視覚的アーティファクト、シミュレーション、図表**: パターン3（OpenGenerativeUIまたは同等のツール）を使用する。厳格なiframe CSPを設定する。セキュリティの観点から出力を非信頼ユーザー内容として扱う。

フレームワークは急速に成熟しています。プロトコルの収束（ストリーミング用のAG-UI、カタログ仕様用のA2UI/Open-JSON-UI）はまだ進行中ですが、形状が明確になり、既に構築を開始できるほどになっています。  

現在最も重要となるエンジニアリング上の課題はフレームワーク選定ではありません。それはカタログ設計であり、モデルが何を表示可能かを決定するプロダクトの明確さが技術スキルより重要です。セマンティック検証も同様で、生成されたUIが主張通りに動作することをテストする必要があります。スキーマ検証を通過するだけでは不十分です。また、アクセシビリティギャップも重要で、各コンポーネントおよびその組み合わせが、手書きUIと同じアクセシビリティ基準を満たすカタログを構築する必要があります。
