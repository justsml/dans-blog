「ジェネレーティブUI」とは、誰が言ったかによって少なくとも5つの異なる意味があります。

- モデルのツール呼び出しによって製品カードを埋め込むチャットインターフェース
- フロントエンドがコンポーネントツリーとしてレンダリングするランタイムJSON仕様
- チケット注文、ホテル予約、地図レンダリング、チェックアウトウィジェットなど、ホストアプリケーションにMCPツールがサンドボックス化されたiframeを返す
- エージェントの状態をフロントエンドにストリーミングするイベントプロトコル
- v0、Lovable、Bolt：デザインタイムにReactを生成するAIツール

これらは関連する概念ですが、スタックの異なるレイヤーに存在し、リスクプロファイル、実装コスト、適切なユースケースが異なります。これらを混同すると、すべてのアーキテクチャ議論が混乱します。

私はスタックのどこに到達するかを決定する際に、このようなマップが必要です。

## ジェネレーティブUIではないもの

定義する前に、3つのことを除外しておきます。

**デザインタイムのコード生成** — v0、Lovable、Bolt、CursorによるReactコンポーネントの組み合わせ。これらのツールは、開発者がレビューしてコミットするコードを生成します。AIは開発時に実行されます。ユーザーの視点では、配信されるものは静的です。これは非常に有用なツールカテゴリです。これは「ランタイムでのジェネレーティブUI」を意味するものではありません。

**AIによるフォーム自動入力の支援** — モデルがコンテキストからフィールド値を充填する。インターフェースの構造は固定されたままです。変化するのはコンテンツだけです。これは有用なパターンです。ジェネレーティブUIではありません。

**AIが生のHTMLをページに書き込む** — モデルが出力する`<div>`や`<button>`の文字列が`innerHTML`や`dangerouslySetInnerHTML`を通じて挿入される。これは技術的な意味ではランタイムジェネレーティブUIです。しかし、これは最も危険な形態であり、この分野の成熟したフレームワークが存在する理由です。生のAI生成マークアップはXSSリスク、アクセシビリティに不適な属性、不一貫なスタイリング、幻覚化された構造を意味します。この記事の残りは、これを改善する方法について述べます。

## 実用的な定義

ランタイムでのジェネレーティブUIとは、**モデルが会話やタスクの状態に基づき、ユーザーに表示するインターフェースコンポーネントまたはコンポーネントの構成を決定すること**を意味します。

重要なのは言葉ではなく、インターフェースです。

最も単純なケース: あなたの飛行機予約アシスタントが`search_flights`ツールを呼び出します。モデルが単純なテキスト("3つのオプションがあります...")ではなく、選択可能なフライト、座席クラスの切り替え、"予約"ボタンを含む`<FlightResultsCard>`コンポーネントをレンダリングします。モデルが構造化されたカードが適切な応答であると判断しました。開発者はそのカードの外観や"予約"ボタンの動作を定義しました。

より複雑なケース: 財政分析エージェントがポートフォリオに関する質問を受け取り、`MetricGroup`で主要な数値を表示し、`RiskBreakdown`チャート、`ScenarioComparison`テーブル、`PolicyNotice`を組み合わせた応答を決定します。モデルは事前に承認されたコンポーネントのカタログからこのレイアウトを構成しました。開発者はすべてのコンポーネントを定義しました。モデルはどのコンポーネントを使用し、そこにどのようなデータを入れるかを選択しました。

両方のケースはジェネレーティブUIです。違いはモデルが持つ構成の自由度にあり、これは出力の豊かさと、何かが間違える可能性の複雑さを決定します。

## 3つのパターン

この領域全体は3つのパターンに集約され、それぞれ異なる出力文法を持っています。

![A spectrum diagram showing three patterns: tool calls only on the left (safest), component catalog in the middle, and open-ended generation on the right (most expressive).](../output-grammar-spectrum.svg)

_すべてのジェネレーティブUIの決定はこのスペクトル上の一点です。左から始めてください。_

### パターン1: ツールからコンポーネントへのレンダリング

モデルは名前付きツールを呼び出します。アプリケーションはツール名からコンポーネントへのマップを持っています。ツール呼び出しがコンポーネントのレンダリングをトリガーします。

```tsx
// モデルが呼び出す: { name: "show_flight_results", args: { flights: [...] } }

useCopilotAction({
  name: "show_flight_results",
  render: ({ args }) => <FlightResultsCard flights={args.flights} />,
});
```

これは最も安全なパターンです。レイアウトはモデルから決まりません。モデルはコンポーネントを表示する*タイミング*と、そのコンポーネントに*どのデータ*を埋め込むかを決定します。開発者は依然としてコンポーネントコード、ビジュアルデザイン、アクセシビリティ実装、レンダリングロジックのすべてのエッジケースを所有しています。

Vercel AI SDKの`useChat`と`tool`ハンドラがこれを行います。assistant-uiのツールレンダリングもこれです。CopilotKitの「Static Generative UI」はこのパターンです。現実的に動作するほとんどのコパイロットUIがこれを行っています。

**適用可能ケース**: 開発時に表示したい項目の集合が明確に列挙できる場合。予約確認、検索結果、アカウントサマリー、承認ウィジェットなどです。シナリオを列挙できるなら、このパターンがカバーします。

### パターン2: コンポーネントカタログの構成

モデルは開発者が定義したカタログからコンポーネントを参照する型付きJSONツリーを出力します。フロントエンドにはツリーを歩きながら各コンポーネントをインスタンス化するレンダラーがあります。

```json
[
  { "type": "metric_group", "metrics": [
    { "label": "MRR", "value": "$82,400", "delta": "+12%" },
    { "label": "Churn", "value": "2.1%", "delta": "-0.4%" }
  ]},
  { "type": "line_chart", "title": "30-day growth", "data_ref": "mrr_series" },
  { "type": "insight_callout", "text": "Expansion revenue driving the delta — avg seat count up 18%." }
]
```

モデルがそのレイアウトを構成しました。`MetricGroup`、`LineChart`、`InsightCallout`です。しかし、各コンポーネントタイプが何を意味し、どのプロパティを受け入れ、どのようにレンダリングされるかはあなたが定義しています。モデルが`{ "type": "custom_untested_thing" }`を出力しようとすると、スキーマ検証がそれを検出し、レンダラーは無視または拒否します。

これは`json-render`、`A2UI`、`Hashbrown`、`OpenUI`、`Tambo`の背後にあるパターンです。中心的なエンジニアリング作業は**カタログ設計**です——どのコンポーネントタイプを用意するか、そのスキーマがどう見えるか、モデルが組み立ててよい/悪いものをどう定義するかを決めることです。

**適切な使用時**: 表示したい構造がデータやユーザーのリクエストによって正当に変化する場合。数値の特異点に応じてアダプティブなダッシュボード。文脈に応じてセクションを切り替えるレポート。エージェントがどのステップにいるかによって変化するワークフローのパネル。

### パターン3: オープンエンド生成

モデルは厳格なコンテンツセキュリティポリシー（CSP）を持つサンドボックスドIFRAME内でHTML、SVG、Canvas、WebGLをレンダリングします。

これは固定されたコンポーネントカタログでは対応できない状況に適しています: アルゴリズムの可視化、アーキテクチャ図、即席のチャート、ジェネレーティブアート、教育用シミュレーション。IFRAME境界がセキュリティの役割を担っています——これを外すと、この記事冒頭で述べた生HTML挿入問題に戻ってしまいます。

`CopilotKit/OpenGenerativeUI`がこのパターンの現時点で最も優れた実装例です。サンドボックスはスクリプトを除去し、メッセージングを制限し、生成されたアーティファクトをアプリケーションの特権状態から隔離します。

**適切な使用時**: 本当に任意の視覚出力が必要な場合——一時的な説明図、動的シミュレーション、クリエイティブなアートifactsに限ります。トランザクショナルUIには使用しないでください。チェックアウト確認画面にサンドボックスドIFRAMEは必要ありません。

### 3つのパターンを超えて: LLMがピクセルを直接駆動する

これらすべてのパターンにすっきりと収まらない第4の方向性が現れています: サンドボックスドIFRAMEよりも直接的に視覚出力を制御し、LLMが**没入型・ゲームライクな体験**を駆動するものです。

ジェネレーティブUIにおける典型的な区別は**IFRAME HTML vs JSONカタログ**です:

- **IFRAME HTML** — モデルがHTML、SVG、Canvas、WebGLを記述し、隔離されたサンドボックスでレンダリングします。表現の自由が最大ですが、セキュリティはIFRAME境界に完全に依存します。例: Anthropic Artifacts、OpenGenerativeUI。  
- **JSONカタログ** — モデルは開発者が定義したコンポーネントカタログに制約された構造化されたペイロードを出力します。レンダラはその仕様に基づいて信頼できる事前構築済みコンポーネントをインスタンス化します。モデルは*何を表示するか*を決定し、あなたは*どのようにレンダリングするか*を決定します。例: json-render、A2UI。  

これらに加えて、最近のデモでは3番目のモードが示唆されています。このモードではモデルがコンポーネントを選択したりサンドボックスHTMLを記述したりするのではなく、キャンバスをより直接的に駆動しています。[TencentのHunyuanWorld](https://arxiv.org/abs/2502.01999)（単一画像から探索可能な3D環境を生成）や、LLMがランタイムで地図、NPC、クエストを生成するゲームアーキテクチャ（コンポーネントカタログを呼び出す代わりに）は、モデルがフォームレンダラではなくゲームディレクターのように機能する未来を示唆しています。WebGPUを介したブラウザ内LLM推論（[WebLLM](https://mlc.ai/web-llm/)）は、同じフロンティアをローカルで推進しています。  

この領域は本当に魅力的で、本当に初期段階です。現時点で生産的な製品を構築できる安定したフレームワークはありません。このアプローチについては、状況が変化した時点で専用の記事で詳しく説明します。

## プロトコル: AG-UIとA2UI  

AG-UIとA2UIはプロトコル層の主な2つの規格です。これらは異なる問題を解決し、競合関係ではありません。

### AG-UI

**GitHub**: [ag-ui-protocol/ag-ui](https://github.com/ag-ui-protocol/ag-ui)

AG-UIは、AIエージェントとフロントエンドアプリケーション間の通信に使用されるイベントベースのプロトコルです。約16種類のイベントタイプを定義しています。`TEXT_MESSAGE_START`、`TEXT_MESSAGE_CONTENT`、`TOOL_CALL_START`、`TOOL_CALL_END`、`STATE_SNAPSHOT`、`STATE_DELTA`などです。トランスポート方式は自由に選べます。SSE、WebSocket、Webhookなどすべて対応可能です。フォーマットはあえて緩く設計されており、広範な採用を可能にしています。

AG-UIはあなたのUIがどのように見えるかを定義しません。エージェントがフロントエンドとどのように通信するかを定義します。これは、あなたのReactアプリがLangGraphエージェントをCrewAIエージェントと同じように購読できるワイヤープロトコルレイヤーと考えてください。フロントエンドコードを変更せずにです。

CopilotKitはLangGraphとCrewAIの開発経験からAG-UIを作成しました。LangChain、Mastra、PydanticAIなどによってすでに採用されています。MicrosoftはAG-UIの統合ガイドを公開しています。マルチエージェントフロントエンドを構築し、バックエンドフレームワークをフロントエンドコードから分離する必要がある場合、AG-UIが答えです。

**1つの重要な注意点**: AG-UIはUIフレームワークではありません。あなたに何をレンダリングするかを指示しません。エージェントが何かを言った、ツールを呼び出した、または共有状態を更新したことを伝えます。その応答として何をレンダリングするかは、依然としてあなたの決定です。

### A2UI

**GitHub**: [google/A2UI](https://github.com/google/A2UI) · Spec: [a2ui.org](https://a2ui.org/)

A2UIは、エージェントがUIを表示したいときに送信するGoogleの宣言型仕様です。AG-UIが「エージェントはどのように通信するか？」という質問に答えるのに対し、A2UIは「エージェントがコンポーネントレイアウトを説明するためのフォーマットは何か？」という質問に答えます。

A2UIはフラットなJSONL形式を使用します。1行に1つのコンポーネント記述子があり、それぞれにID、タイプ、データが含まれます。フラットな形式は意図的です。ネストされたツリー構造では、モデルが全体の構造を把握する前にストリーミングを開始できません。一方、フラットなリストではモデルが「考える」たびにコンポーネントを逐次出力できるため、フロントエンドはモデルがチャートを追加するかどうかを検討している間に、最初のメトリックカードのレンダリングを開始できます。

```jsonl
{"id":"h1","type":"kpi_card","title":"MRR","value":"$82,400","delta":"+12%"}
{"id":"h2","type":"kpi_card","title":"Churn","value":"2.1%","delta":"-0.4%"}
{"id":"c1","type":"line_chart","title":"30-day MRR","data_ref":"mrr_series"}
{"id":"t1","type":"data_table","cols":["Month","MRR","Net New"],"data_ref":"monthly"}
```

A2UIはセキュリティに配慮された設計です。この仕様は実行可能なコードではなくデータ形式です。コンポーネントカタログは開発者が事前に定義し、エージェントはそのカタログに登録されたタイプのみを参照できます。A2UIのレンダラが未知のタイプ名を受け取った場合、それを無視します。

CopilotKitの「Open-JSON-UI」形式はA2UIと互換性があります。今日コンポーネントカタログの仕様形式を選ぶなら、A2UIは最もクロスプラットフォームで広くサポートされている形式です。

**安定性に関する注意点**: A2UIは1.0未満のバージョンで、2026年5月8日に確認した時点ではv0.9でした。マイナーバージョン間で仕様の破壊的変更が発生しています。Googleのロードマップに関するコミュニケーションは不規則で、一部のレンダラ（Lit、Flutter）は仕様更新に遅れています。現在これを利用して開発する場合は、仕様の変化に備えて時間を見積もる必要があります。純粋なWeb用途の場合、json-renderは現在より完全なツールチェーンを持っているように見えます。A2UIの長期的な利点はjson-renderにはないクロスプラットフォーム対応（Web、Flutter、SwiftUI、Android）です。

### MCP Apps

**GitHub**: [modelcontextprotocol](https://github.com/modelcontextprotocol) · 関連: [mcp-ui](https://github.com/MCP-UI-Org/mcp-ui)

MCPはLLMをツールやデータに接続するためのプロトコルとして始まりました。Apps拡張機能により、MCPツールはデータだけでなくインタラクティブなUIアーティファクト（Reactコンポーネント、フォーム、ダッシュボード、地図）も返すことができます。

セキュリティモデルは厳格で、それが目的です。すべてが権限が削除されたサンドボックスiframeでレンダリングされ、テンプレートは事前に宣言されてホストアプリがレビューできます。すべての通信は監査可能なJSON-RPCです。これはツールプロバイダにとって正しいモデルです。Shopify MCPサーバはチェックアウトウィジェットを返すことができ、地図サービスは埋め込み可能な地図を返すことができます。ホストアプリはそのウィジェットのコードを所有したり信頼したりしません。

UIが*ツールプロバイダに所属する*場合にMCP Appsは適切な選択です。アプリケーションのドメインに属するUIにはパターン1または2を使用してください。

### CopilotKit

**GitHub**: [CopilotKit/CopilotKit](https://github.com/CopilotKit/CopilotKit) · サンプル: [CopilotKit/generative-ui](https://github.com/CopilotKit/generative-ui)

CopilotKitはエージェントネイティブなフロントエンドアプリケーション向けに最も包括的なフレームワークです。これはライフサイクル全体を処理します: AG-UI経由でエージェントバックエンドに接続し、双方向会話状態を管理し、生成UIコンポーネントをレンダリングし、エージェントとユーザーが同じデータを変更できる共有状態のパイプラインを提供します。

3つのパターンモデルはCopilotKitのAPIに明確にマッピングされます:
- `useCopilotAction` と `render` コールバック → パターン1
- A2UI/Open-JSON-UIレンダリング → パターン2
- `OpenGenerativeUI` サンドボックスアーティファクト → パターン3

CopilotKitで議論が少ないが重要な機能は**共有状態と人間ループ**です: エージェントはアプリケーション状態を読み書きでき、ユーザーも読み書き可能で、変更は双方向に流れます。これこそが、コパイロットスタイルUIがチャットボックスを製品に貼り付けたものではなく、真の協働体験に感じさせる理由です。

### Vercel AI SDK

**GitHub**: [vercel/ai](https://github.com/vercel/ai) · ドキュメント: [ai-sdk.dev](https://ai-sdk.dev/)

Vercel AI SDKはAIアプリケーション向けの事実上のTypeScript基盤です。生成UIに関して特に:

**`useObject`** はサーバーから生成される構造化されたJSONオブジェクトをストリーミングします。Zodスキーマを定義し、SDKは部分的なJSONを解析し、フィールドが到着するたびに再レンダリングをトリガーします。これはNext.jsアプリケーションにおけるパターン2への最も滑らかな実装方法です。

```tsx
const { object: dashboard } = useObject({
  api: "/api/generate-dashboard",
  schema: z.object({
    title: z.string(),
    metrics: z.array(z.object({ label: z.string(), value: z.number() })),
    insights: z.array(z.string()),
  }),
});
```

**`useChat` とツールハンドラ** → パターン1。モデルはツールを呼び出します。ツール名をコンポーネントにマッピングします。

**AI Elements** ([elements.ai-sdk.dev](https://elements.ai-sdk.dev/)) は、SDKと組み合わせて使用できる事前に作成されたUIプリミティブを提供します。

**ここでの混乱した軌跡に関する注記**: 2024年10月、Vercelは[GitHub Discussion #3251](https://github.com/vercel/ai/discussions/3251)で、SDK 3.0のハイライト機能である「Generative UI」のReact Server Components (RSC) ストリーミングパターンであるAI SDK RSCが「いくつかの長期的な制約」により無期限に一時停止されたことを発表しました。RSCストリーミングを製品戦略に組み入れていたチームは予期せずに影響を受けました。`generateObject`/`streamObject` APIもSDK 6.0で非推奨となりました。AI SDK RSCからの推奨される移行先は、上記の`useObject`パターンまたはカタログベース生成のjson-renderです。

### assistant-ui

**GitHub**: [assistant-ui/assistant-ui](https://github.com/assistant-ui/assistant-ui)

assistant-uiは、本番品質のチャットインターフェースを構築するための組み合わせ可能なReactプリミティブのセットです。メッセージバブル、ストリーミングトークン、コピー/編集/再生成アクション、思考状態などの洗練されたチャットUXが必要で、バックエンドとツールレンダリングをカスタマイズしたい場合に最適な選択です。

任意のバックエンド（OpenAI、Anthropic、ローカルモデル、カスタムエンドポイント）と組み合わせて使用でき、ツール呼び出しレンダリングは慣れたスロット/レンダープロップモデルで処理されます。

### json-render

**GitHub**: [vercel-labs/json-render](https://github.com/vercel-labs/json-render) · Docs: [json-render.dev](https://json-render.dev/)

json-renderは、Pattern 2を実装するための意見の入った「すべてが揃った」アプローチで運用します。事前に構築されたコンポーネントカタログ（shadcn/uiコンポーネントとZodスキーマ）、レンダラ、およびモデルがカタログに制約されるスキーマによる厳密な生成ループが含まれます。

特徴的な機能:
- **マルチターゲットレンダリング**: 同じJSON仕様からReactウェブアプリ、React Nativeモバイルアプリ、PDF、HTMLメール、Remotionビデオにレンダリング可能です。これはレポート生成に本当に有用です。
- **進行型レンダリング**: コンポーネントはモデルがストリームするたびに表示され、完全な仕様が到着した後ではなくなります。
- **厳密なスキーマ制約**: カタログはモデルが有効だが未知のコンポーネントタイプを生成できないように設計されています。

json-renderは、ダッシュボードやレポート生成機能を構築し、独自のカタログ設計のインフラ作業をスキップしたい場合、ウェブアプリケーションにおいて最も速い実装方法です。

**動向について**: json-renderは2026年初頭にVercel Labsからリリースされ、標準的なReact/Next.jsプロジェクトで即座に有用であるため、ウェブ開発者から急速に関心を集めておりました。ただし、json-renderはまだ1.0未満のバージョンであり、json-renderとA2UIの関係性はまだ調整中です。VercelはA2UI互換の出力も試験的に実装しており、両者の統合が進む可能性があります。クロスプラットフォーム（ネイティブモバイル、複数フレームワーク）の長期的な選択肢としてはA2UIがより適しています。

### Hashbrown

**GitHub**: [liveloveapp/hashbrown](https://github.com/liveloveapp/hashbrown)

Hashbrownは独自のアプローチを取ります。AIインターフェース層を別途構築するのではなく、既存のReactやAngularアプリケーションにAIコンポーネントの選択を直接組み込みます。アプリケーションのコンポーネントをLLMに公開し、LLMがどのコンポーネントをレンダリングするかを選択し、クライアントサイドのツールを呼び出すことが可能です。

これは「チャット」ではない製品インターフェースに知能を注入したい場合に最適なツールです。例えば、レイアウトを動的に調整する製品ページ、適切なオプションを表示する設定パネル、次のステップを提案するワークフロー編集器などです。

### OpenUI

**GitHub**: [thesysdev/openui](https://github.com/thesysdev/openui) · ドキュメント: [openui.com](https://www.openui.com/)

OpenUIはJSONを置き換え、進行型レンダリングとトークン効率を設計した行単位のコード形式（「OpenUI Lang」）を採用しています。複雑なレイアウトにおいて同等のJSON表現と比較して約67%のトークン削減が可能であると主張しています。

トレードオフはエコシステムの成熟度です。OpenUIは新しい技術であり、JSONベースのアプローチに比べてツールの整備が不十分です。ただし、トークンコストが重要な制約であり、頻繁に複雑なレイアウトを生成する必要がある場合、フォーマットの効率性は現実的な利点です。

### Tambo

**GitHub**: [tambo-ai/tambo](https://github.com/tambo-ai/tambo)

Tamboは状態を保持するコンポーネント選択に焦点を当てており、AIがコンポーネントを選択し、クライアントサイドツールを通じてそれらとやり取りできるように設計されています。会話全体でコンポーネントの状態を維持します。UI要素がターンを越えて保持されるユースケース（ユーザーがフィルターコンポーネントを調整しながらAIがフィルタリングされたデータを継続的に処理する場合）に適しています。

## Pythonレイヤー

PythonエコシステムはAIインターフェースを異なるアプローチで扱います。これらのツールはMLモデルのデモ、データアプリケーション、内部ツールに最適化されており、エージェント駆動のレイアウト構成を伴う本番環境向け消費者アプリケーションには向いていません。

これは否定的な意見ではありません。適切なユースケースにおいてGradioやStreamlitは必要なツールのすべてです。

### Gradio

**GitHub**: [gradio-app/gradio](https://github.com/gradio-app/gradio) · PyPI: `gradio`

Gradioのコア価値は、Python関数を記述すればGradioがそれをウェブUIでラップする点にあります。`Interface`クラスは画像分類器を3行で実装可能です。`ChatInterface`はチャットボットを10行で構築できます。`Blocks`は必要に応じて細かいレイアウト制御を提供します。

Gradioにおける「Generative UI」はPython開発者が定義し、モデルが定義するものではありません。コンポーネントの可視性や設定はモデルの出力に応じて動的に変化できますが、コンポーネントカタログ自体は静的です——モデルにレイアウトを構成させるわけではありません。

GradioはHuggingFace Spacesのデフォルトであり、MLデモエコシステムの基盤となっています。月間数百万のダウンロード数があり、AIデモの大部分を支えています。

**Gradioを選ぶべき状況**: Python開発者がMLモデルのデモ、研究プロトタイプ、または内部ツールを開発しており、JavaScriptを触りたくない場合です。

### Streamlit

**GitHub**: [streamlit/streamlit](https://github.com/streamlit/streamlit)

Streamlitのモデルはより意見が強いです——各インタラクションごとにPythonスクリプトがエンドツーエンドで実行されます。`st.chat_message()`、`st.dataframe()`、`st.plotly_chart()`を呼び出します。フレームワークがレイアウトを処理します。

スクリプト全体を再実行するモデルは非効率に聞こえますが、会話履歴を蓄積するAIチャットボットでは意外と操作性が良いです——スクリプト全体が再実行され、チャット履歴はセッション状態に保存され、出力は決定的です。Streamlitは主要なLLMプロバイダーの公式サポートを提供し、Snowflake Cortexとネイティブに統合されています。

**Streamlitを選ぶべき状況**: PythonでAI駆動のデータアプリ、内部レポートツール、MLベースのダッシュボードを開発し、最もシンプルなデプロイメントパスを望む場合です。

### LangChain と Haystack

これらはUIフレームワークではなく、バックエンドオーケストレーションフレームワークです。誠実なGenerative UIスタックマップには必ず含まれる理由は、構造化された出力がフロントエンドに送られる前の通常のレイヤーであるからです。

**LangChain** ([langchain-ai/langchain](https://github.com/langchain-ai/langchain)): 任意のLLMで`.with_structured_output()`を使用すると、Pydantic制約付きのJSON生成が可能です。`@tool`デコレータと自動スキーマ生成が、モデルが呼び出せるツールを定義する最もシンプルな方法です。LangChainは構造化された結果を、使用しているフロントエンド層に送信します。  

**Haystack** ([deepset-ai/haystack](https://github.com/deepset-ai/haystack)): 強力なRAGサポートを備えたモジュール型パイプラインアーキテクチャ。HayhooksはHaystackパイプラインをHTTPエンドポイントとしてラップします——MCP互換のエンドポイントも含みます。生成型UIがリトリーバルバックボーンを必要とする場合、Haystackのパイプラインアーキテクチャがこれをクリーンに処理します。  

いずれのフレームワークもUI層を所有していません。これらは、あなたのフロントエンド（パターン1, 2, または3）がレンダリングするデータを生成するだけです。

## エコシステムの速度と不安定な地盤  

この分野は急速に進化しており、いくつかのプロジェクトはコードとともに混乱を招くコミュニケーションをリリースしています。最終確認日: 2026年5月8日。ここでのプロジェクトステータスのメモは、時刻付きの読み取り情報として扱い、永久的な判定とは見なしないでください。

**Vercel AI SDK RSC** はSDK 3.0リリース時のフラグシップ機能でした。Vercelは2024年10月にReact Server Componentsのアーキテクチャ的制約により開発を一時停止しました（[ディスカッション #3251](https://github.com/vercel/ai/discussions/3251)）。これにより、既存のプロジェクトを構築していたチームは当然ながら不満を抱いていました。現在もドキュメントに記載されていますが、推奨される方法ではありません。代わりに `useObject` が推奨されています。

**json-render**（Vercel Labs）は新しい方向性で、RSCの結合問題を回避するカタログベースのフレームワーク非依存の代替案です。これは1.0リリース前で、React/web開発者間で初期段階で強い関心が寄せられています。DX（開発者経験）の理由としては、json-renderは標準的なReact/Next.jsプロジェクトで即座に有用であるのに対し、A2UIのクロスプラットフォーム対応がセットアップの摩擦を生みます。両方の仕様が成熟するにつれてどうなるかは本当に不透明です。Vercelはjson-renderでA2UIの互換性を検証しており、統合が可能であることを示唆しています。

**A2UI**（Google）は1.0未満（最終確認時v0.9）で、マイナーバージョン間で破壊的変更があり、Googleのロードマップに関するコミュニケーションが不連続です。json-renderが対応していないクロスプラットフォーム対応（web + Flutter + SwiftUI）を実現するには適した選択肢であり、企業向けの支援も実質的に行われています。ただし、純粋なwebプロジェクトでは現在のDXは厳しいです。

**AG-UI**（CopilotKit）も1.0未満です。最も一般的な誤解は、名前がUIフレームワークのように聞こえる点です。実際にはそうではなく、トランスポートプロトコルです。AG-UIはエージェントバックエンドとフロントエンド間のイベントフローを定義します。あなたがどのようにレンダリングするかは依然としてあなたの判断に委ねられています。このメンタルモデルは堅実で広く採用されていますが、1.0未満の仕様であるためエッジケースがまだ整理中です。

現実的な結論：**ここに挙げた主要プレイヤーはすべて1.0未満です**。API変更を計画してください。パターン自体（ツールからコンポーネント、カタログ構成、サンドボックス生成）は構築に十分安定しています。具体的なプロトコル選択はそうではありません。

## コンポーネントカタログ設計：本質的なエンジニアリング作業

パターン2の興味深い複雑さの多くはレンダラにではなく、カタログにあります。

カタログは**製品決定をスキーマとしてエンコードしたもの**です。それは「このドメインにおける意味のあるUIオブジェクトは何か？」を答えます。Reactコンポーネントの存在ではなく、「このコンテキストにおけるユーザーが実際に視覚化・操作すべきものとは何か？」に焦点を当てます。

**粒度が細かすぎる失敗モード**：`Row`、`Column`、`Text`、`Button`、`Icon`を公開します。これによりモデルはフロントエンドエンジニアである必要があります。結果として、あなたのデザインシステムに合致しない平凡なレイアウトが生成され、空状態が考慮されず、アクセシビリティに欠けるマークアップが生成され、カタログに制約がないため応答ごとにアプローチが変化します。

**粒度が粗すぎる失敗モード**: `WeatherCard`、`FlightCard`、`HotelCard`を公開します。ユーザーが事前に作成されたカードにマッピングされない何かを要求した場合、モデルは適応できません。代わりにテキストにフォールバックします。

**有用な中間領域**: 制約付きスロットを持つドメインレベルコンポーネント。

旅行アプリのカタログは以下のようになります：

```
TripSummary         — 旅程の概要
FlightOptionList    — 価格付きの選択可能なフライトオプション
HotelComparison     — サイドバイサイドのホテルカード
TravelerForm        — 旅行者詳細の収集
PolicyNotice        — 規制/運賃規則の強調表示
BookingConfirmation — アクションボタン付きの最終確認
```

ファイナンスアプリのカタログは以下のようになります：

```
PortfolioSnapshot   — 主要ポジションと損益
TransactionTable    — フィルタ可能、ページネーション付きの取引
RiskBreakdown       — 配分とボラティリティ指標
ScenarioComparison  — シナリオモデリングのサイドバイサイド
ApprovalGate        — 人間の確認が必要なアクション
```

カタログは製品の語彙のように聞こえます。それはUX決定、アクセシビリティ要件、空状態の処理、危険なアクションパターンをコンポーネントコードにエンコードします。モデルはこれらのピースを配置できます。あなたは依然として各ピースの外観と許容される動作を決定します。

**幻覚を減らすスキーマ設計ルール**:

1. 列挙値を短く明確に保つ。`"type": "bar_chart"` ではなく `"type": "data-visualization-bar-type-vertical"` は避ける。
2. 無効なコンポジションを不可能にさせる。`PolicyNotice`がレイアウトの終わりにしか表示できない場合、どこにでも表示できる他の要素と同じスキーマレベルに置かない。
3. 必須フィールドを多用する。オプションフィールドはモデルが省略し、レンダラがnullとして処理しなければならないフィールドである。
4. 本番リリース前に実際のプロンプトでカタログをテストする。生成された仕様を保存し、スキーマ違反、ハルシネーションされたフィールド値、技術的に有効だが意味的に誤っているコンポジションをチェックする。

## 一般的な罠

**罠: 有効なJSONを安全な動作と誤認する**。スキーマ検証は構造を確認します。それはボタンのラベルと関連するアクションが一致するか、合計がデータと一致するか、UIコンポーネントがユーザーが予期しない動作を行わないかについて何の保証もしません。生成されたUI仕様はセマンティックレビューが必要であり、スキーマ検証だけでは不十分です。最低限、破壊的アクションは確認コンポーネントを必要とし、そのコンポーネントのラベルはトリガーするアクションに対してテストされるべきです。

**罠: 設計プリミティブではなく製品プリミティブを公開する**。モデルが16pxと20pxのパディングどちらを使うかを決定しなければならない場合、あなたは間違った抽象化レベルを与えている。ドメインコンポーネントは製品の味わいをエンコードすべきである。モデルは動作の組み合わせを行うべきであり、プレゼンテーションの詳細を管理すべきではない。

**罠: 静的UIで十分な場面でジェネレーティブUIを使う**。表示したい構造が開発時に既知である場合（通常そうである）、パターン1の事前構築コンポーネントはより速く、安全で、一貫性がある。ジェネレーティブUIは構造が実際にデータやタスクコンテキストに応じて変化する場合にのみその複雑さに値する。

**罠: アクセシビリティをスキップする**。LLMはWCAG違反をハルシネーションする。インタラクティブな要素に`role="region"`を出力し、ラベルのないフォームを生成し、WCAG AA基準を満たさないコントラスト比を出す。あなたのコンポーネントライブラリが完全にアクセシブルでも、それらのコンポーネントのAI生成組み合わせは自動的にアクセシブルではない。レンダリングパス全体をテストし、個別のコンポーネントではなく。

**罠: プロトコルとフレームワークを混同する**。AG-UIはフロントエンドフレームワークではない。A2UIはReactライブラリではない。これらはワイヤーフォーマットとイベントプロトコルである。実装には依然としてフロントエンドフレームワークが必要である。CopilotKitはAG-UIとA2UIを実装する。json-renderはA2UI/Open-JSON-UIカタログパターンを実装する。これらは異なるレイヤーである。

## ケース別の推奨事項

**既存のSaaSアプリにコパイロットを追加**: パターン1（ツール→コンポーネント）から始める。Vercel AI SDKの`useChat`やCopilotKitを使用する。上位5～10のエージェントアクションを事前構築コンポーネントにマッピングする。それをリリースし、測定し、ユーザーが明らかに豊富なコンポジションを必要とする場合のみカタログを拡張する。

**自然言語によるダッシュボード生成**: json-renderやカスタムA2UIカタログを使用したパターン2を採用する。チャートタイプ、メトリックカード、テーブルバリアントをカバーする8～15のコンポーネントタイプのカタログを定義する。スキーマをモデルにフィードし、レイアウトを組み合わせさせる。未知のタイプがレンダラーに到達する前に検証を構築する。

**マルチエージェントフロントエンド**: CopilotKitとAG-UIを使用する。イベントストリームはエージェントバックエンド間のリアルタイムストリーミングを処理し、共有状態はエージェント間のハンドオフを処理し、HITLパターンは承認ゲートを処理する。

**ChatGPTや他のMCPホスト内で構築**: MCP Appsを使用する。ツールをデータ取得と推論を行うデータツールと、ウィジェットをリクエストする別々のレンダートールに定義する。ビジネスロジックはウィジェットテンプレートに含めない。

**MLモデルのデモとデータアプリ（Pythonチーム）**: デモにはGradio、HuggingFace Spacesを使用。複雑なインタラクションを必要とするデータアプリにはStreamlitを使用。どちらもJavaScriptは必要ない。

**可視化アーティファクト、シミュレーション、図表**: パターン3（OpenGenerativeUIまたは同等のツール）を使用。iframeのCSPを厳格に設定し、セキュリティ観点から出力を信頼されていないユーザー内容として扱う。

フレームワークは急速に成熟している。ストリーミング用のAG-UIやカタログ仕様用のA2UI/Open-JSON-UIといったプロトコルの収束はまだ進行中だが、構造はすでに十分明確で構築が可能である。

現在最も重要なエンジニアリング課題はフレームワーク選定ではない。それは**カタログ設計**である——モデルが何を表現可能かを決定するプロダクトの明確さが技術スキルよりも重要だ。それは**意味的検証**である——生成されたUIが宣言通りに動作することをテストし、単にスキーマ検証を通過することだけでは済まない。そしてそれは**アクセシビリティのギャップ**である——カタログ内のすべてのコンポーネント、およびその組み合わせが、手書きUIと同等のアクセシビリティ基準を満たすように構築することだ。

モデルはあなたが与えた文法の枠内で、あなたが指示した通りに行動する。その文法を意図的に設計せよ。
