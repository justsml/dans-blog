「Generative UI」には、発言者によって少なくとも5つの異なる意味があります。

- モデルツール呼び出しから生成されたプロダクトカードを埋め込むチャットインターフェース  
- フロントエンドがコンポーネントツリーとして描画するランタイムJSON仕様  
- MCPツールがホストアプリに返すサンドボックス化されたiframe（チケット予約、ホテル予約、マップ描画、チェックアウトウィジェットなど）  
- エージェント状態をフロントエンドにストリームするイベントプロトコル  
- v0、Lovable、Bolt：設計時にReactコードを書き出すAIツール  

これらは関連概念ですが、スタックの異なる層に存在し、リスクプロファイル、実装コスト、適切なユースケースがそれぞれ異なります。これらを混同すると、アーキテクチャ議論はすべて混沌とします。

このマップは、スタックのどの位置に手を伸ばすかを決める際に私が欲しいものです。

---

## Generative UI ではないもの

定義に入る前に、まず以下の3点を脇に置きます。

**設計時コード生成** — v0、Lovable、Bolt、Cursor などが React コンポーネントを組み立てるツールです。これらのツールは開発者がレビューしコミットするコードを生成します。AI は開発時に走り、ユーザー側から見ると出荷物は静的です。これは有用なツールカテゴリですが、「ランタイムでの Generative UI」ではありません。

**AI 補助型フォーム自動入力** — モデルがコンテキストからフィールド値を埋めるケースです。インターフェースの構造は固定されたままで、コンテンツだけが変わります。便利なパターンですが、Generative UI ではありません。

**AI がページに生の HTML を書き込む** — モデルが `<div>` や `<button>` といった文字列を出力し、`innerHTML` や `dangerouslySetInnerHTML` を通じて注入するケースです。これは技術的に最も純粋な「ランタイム Generative UI」ですが、同時に最も危険な形でもあります。成熟したフレームワークが回避しようとしている対象です。生の AI 生成マークアップは XSS リスク、アクセシビリティ属性の欠如、スタイルの不整合、そして構造の幻覚を招きます。本稿の残りは、これより安全な手法について議論します。

## 実装上の定義

ランタイムでの Generative UI とは、**会話やタスクの状態に応じて、モデルがユーザーに提示する UI コンポーネントまたはコンポーネント構成を決定すること** を指します。

テキストではなく、インターフェースです。

最も単純な例: フライト予約アシスタントが `search_flights` ツールを呼び出すケースです。単にテキスト ("Here are three options...") を返すのではなく、選択可能なフライト、座席クラスの切替、そして「Book」ボタンを備えた `<FlightResultsCard>` コンポーネントを描画します。ここでモデルは構造化されたカードが適切な応答であると判断しました。カードの見た目や「Book」ボタンの動作は開発者が定義しています。

もう少し複雑な例: 金融分析エージェントがポートフォリオに関する質問を受け取り、`MetricGroup` で主要数値を表示し、`RiskBreakdown` チャート、`ScenarioComparison` テーブル、そして `PolicyNotice` を組み合わせて回答を構成するとします。モデルは事前に承認されたコンポーネントカタログからそのレイアウトを組み立てました。各コンポーネントの定義は開発者が行い、どのコンポーネントを使用し、どのデータを入れるかをモデルが選択しました。

両方とも Generative UI です。モデルが持つ構成自由度の違いにより、生成可能な出力の豊かさと、失敗の可能性の複雑さが変わります。

## 3 つのパターン

この領域はすべて 3 つのパターンに集約でき、それぞれ出力文法が異なります。

![3つのパターンを示すスペクトラム図：左がツール呼び出しのみ（最も安全）、中央がコンポーネントカタログ、右がオープンエンド生成（最も表現力が高い）。](../output-grammar-spectrum.svg)

_すべての Generative UI の決定はこのスペクトラム上の一点です。左側から始めてください。_

### パターン 1: ツール → コンポーネント描画

モデルは名前付きツールを呼び出します。アプリ側でツール名とコンポーネントを紐付けるマップを用意し、ツール呼び出しがコンポーネントのレンダリングをトリガーします。

```tsx
// The model calls: { name: "show_flight_results", args: { flights: [...] } }

useCopilotAction({
  name: "show_flight_results",
  render: ({ args }) => <FlightResultsCard flights={args.flights} />,
});
```

レイアウトがモデルから来ないため、最も安全なパターンです。モデルは **いつ** コンポーネントを表示するか、**どのデータ** を埋め込むかだけを決定します。コンポーネントのコード、ビジュアルデザイン、アクセシビリティ実装、そしてレンダリングロジックのあらゆるエッジケースは依然として開発者が所有します。

Vercel AI SDK の `useChat` における `tool` ハンドラや、assistant‑ui のツールレンダリング、CopilotKit の「Static Generative UI」もこのパターンです。信頼性の高い本番環境の Copilot UI の多くはここを採用しています。

**適用シーン**: 表示したい要素が開発時に把握できる場合。予約確認、検索結果、アカウントサマリー、承認ウィジェットなど、シナリオを列挙できるならこのパターンで網羅できます。

### パターン 2: コンポーネントカタログ構成

モデルは開発者が定義したカタログのコンポーネントを参照する型付き JSON ツリーを出力します。フロントエンドはそのツリーを走査し、各コンポーネントをインスタンス化するレンダラを持ちます。

```json
[
  { "type": "metric_group", "metrics": [
    { "label": "MRR", "value": "$82,400", "delta": "+12%" },
    { "label": "Churn", "value": "2.1%", "delta": "-0.4%" }
  ]},
  { "type": "line_chart", "title": "30-day growth", "data_ref": "mrr_series" },
  { "type": "insight_callout", "text": "Expansion revenue driving the delta — avg seat count up 18%." }
]
```

このレイアウトはモデルが組み立てましたが、`MetricGroup`、`LineChart`、`InsightCallout` といった各コンポーネントの意味、受け取るプロパティ、描画方法はすべて開発者が定義しています。モデルが `{ "type": "custom_untested_thing" }` のような未定義タイプを出力しようとしても、スキーマ検証で捕捉され、レンダラはそれを無視または拒否します。

このパターンは `json-render`、`A2UI`、`Hashbrown`、`OpenUI`、`Tambo` の背後にあるものです。主なエンジニアリング作業は **カタログ設計** であり、どのコンポーネントタイプを提供するか、そのスキーマはどうなるか、モデルが組み立てて良いものと組み立ててはいけないものを決めます。

**適用シーン**: 表示したい内容の構造がデータやユーザーのリクエストに応じて正当化される場合。数値の中で注目すべきポイントに合わせて変化するダッシュボード。コンテキストに応じてセクションが変わるレポート。エージェントの現在のステップに応じてパネルが切り替わるワークフロー。

### パターン 3: オープンエンド生成

モデルが HTML、SVG、Canvas、または WebGL を生成し、厳格な Content Security Policy が設定されたサンドボックス化された iframe 内で描画します。

固定のコンポーネントカタログでは対応できないケースに適しています：アルゴリズムの可視化、アーキテクチャ図、アドホックなチャート、生成アート、教育用シミュレーション。ここでのセキュリティは iframe の境界が担い、境界を取り除くとこの記事冒頭で指摘した生の HTML 注入問題に戻ります。

`CopilotKit/OpenGenerativeUI` はこのパターンの現在最良のリファレンス実装です。サンドボックスはスクリプトを除去し、メッセージパッシングを制限し、生成されたアーティファクトをアプリケーションの特権状態から隔離します。

**適用シーン**: 任意のビジュアル出力が本当に必要な場合—一回限りの説明図、動的シミュレーション、クリエイティブなアーティファクト。取引 UI には使用しないでください。チェックアウト確認画面など、サンドボックス化された iframe が不要なケースです。

### 3 つのパターンを超えて: LLM がピクセルを直接制御するケース

ここ数年で浮上してきた第 4 の方向性があります。これは従来のパターンのいずれにもきれいに収まらないもので、LLM が **没入型でゲームのような体験** を提供するために、サンドボックス化された iframe よりも直接的にビジュアル出力を制御します。

生成 UI における典型的な区別は **iframe HTML と JSON カタログ** です：

- **IframeHTML** — モデルが HTML、SVG、Canvas、または WebGL を記述し、分離されたサンドボックスで描画します。表現の自由度は最大ですが、セキュリティは完全に iframe の境界に依存します。例: Anthropic Artifacts、OpenGenerativeUI。  
- **JSON カタログ** — モデルが開発者が定義したコンポーネントカタログに制約された構造化ペイロードを出力します。レンダラーはその仕様から信頼できる事前構築コンポーネントをインスタンス化します。モデルは *何を* 表示するかを決定し、実装側が *どのように* 描画するかを選びます。例: json-render、A2UI。  

これらに加えて、最近のデモはモデルがコンポーネントを選択したりサンドボックス化された HTML を書いたりせず、キャンバスをより直接的に操作する第 3 のモードを示唆しています。Tencent の [HunyuanWorld](https://arxiv.org/abs/2502.01999) のように、単一画像から探索可能な 3D 環境を生成したり、LLM がコンポーネントカタログを呼び出すのではなく、実行時にマップ、NPC、クエストを生成するゲームアーキテクチャなどが、モデルがフォームレンダラではなくゲームディレクタのようになる未来を示しています。WebGPU を介したブラウザ内 LLM 推論（[WebLLM](https://mlc.ai/web-llm/)）も同様のフロンティアをローカルで押し進めています。

この領域は本当にエキサイティングで、まだ黎明期です。現時点で本番向けに安定したフレームワークは存在しません。状況が変わったら、専用の記事でこのアプローチを取り上げます。

## プロトコル: AG-UI と A2UI

AG-UI と A2UI はプロトコル層の主要な標準です。解決する課題が異なり、競合関係にはありません。

###AG-UI

**GitHub**: [ag-ui-protocol/ag-ui](https://github.com/ag-ui-protocol/ag-ui)

AG-UI は AI エージェントとフロントエンドアプリケーション間の通信を行うイベントベースのプロトコルです。約 16 種類のイベントタイプを定義します：`TEXT_MESSAGE_START`、`TEXT_MESSAGE_CONTENT`、`TOOL_CALL_START`、`TOOL_CALL_END`、`STATE_SNAPSHOT`、`STATE_DELTA` など。トランスポートは自由で、SSE、WebSocket、Webhook すべてが利用可能です。フォーマットは意図的に緩く設計されており、広範な採用を想定しています。

AG-UI は UI の見た目を規定しません。エージェントが *フロントエンドと* どのように通信するかを定義します。React アプリが LangGraph エージェントや CrewAI エージェントに対して同じ方法でサブスクライブできる、いわばワイヤープロトコル層と考えてください。フロントエンドコードを変更する必要はありません。

CopilotKit が LangGraph と CrewAI の取り組みから AG-UI を作り出しました。その後 LangChain、Mastra、PydanticAI などが採用しています。Microsoft も AG-UI の統合ガイドを公開しています。マルチエージェントフロントエンドを構築し、バックエンドフレームワークとフロントエンドコードを分離したい場合、AG-UI が解決策です。

**よくある誤解**：AG-UI は UI フレームワークではありません。何を描画すべきかは指示しません。エージェントが「何かを言った」「ツールを呼び出した」「共有状態を更新した」ことを *伝える* 手段を提供するだけです。実際に何を描画するかは依然として開発者の判断です。

### A2UI

**GitHub**: [google/A2UI](https://github.com/google/A2UI) · Spec: [a2ui.org](https://a2ui.org/)

A2UI は、エージェントが UI を表示したいときに送信する内容を記述するための Google 製宣言的仕様です。AG-UI が「エージェントはどう通信するか？」に答えるのに対し、A2UI は「エージェントはコンポーネントレイアウトをどの形式で記述するか？」に答えます。

A2UI はフラットな JSONL 形式を採用します。1 行につき 1 コンポーネント記述子があり、各行は ID、type、data を持ちます。フラット化は意図的です。ネストしたツリー構造だと、モデルが全体構造を把握してからでないとストリーミングできません。一方、フラットリストならモデルは「考えている」コンポーネントを順次出力できるため、フロントエンドはモデルがまだチャート追加を検討中でも、最初のメトリックカードの描画をすぐに開始できます。

```jsonl
{"id":"h1","type":"kpi_card","title":"MRR","value":"$82,400","delta":"+12%"}
{"id":"h2","type":"kpi_card","title":"Churn","value":"2.1%","delta":"-0.4%"}
{"id":"c1","type":"line_chart","title":"30-day MRR","data_ref":"mrr_series"}
{"id":"t1","type":"data_table","cols":["Month","MRR","Net New"],"data_ref":"monthly"}
```

A2UI はセキュリティを意識した設計です。仕様は実行可能なコードではなくデータフォーマットであり、コンポーネントカタログは開発者が事前に定義したものです。エージェントはそのカタログに含まれる型しか参照できません。A2UI レンダラは未知の型名が来た場合は単に無視します。

CopilotKit の「Open-JSON-UI」形式は A2UI と互換性があります。現在コンポーネントカタログ用の仕様フォーマットを選ぶのであれば、汎用性とプラットフォーム横断のサポートが最も広いのは A2UI です。

**安定性に関する注意**: A2UI は 1.0 未満（2026 年 5 月 8 日時点で v0.9）で、マイナーバージョン間で破壊的な仕様変更が行われています。Google のロードマップに関する情報は断続的で、いくつかのレンダラ（Lit、Flutter）は仕様更新に遅れを取っています。現在この仕様上で開発する場合は、仕様のドリフトに備えて余裕を持って計画してください。純粋なウェブ向けのユースケースでは、json‑render の方がツールチェーンがより充実しているようです。A2UI の長期的な強みは、web、Flutter、SwiftUI、Android といった複数プラットフォームへの到達性であり、json‑render にはない利点です。

### MCP Apps

**GitHub**: [modelcontextprotocol](https://github.com/modelcontextprotocol) · 関連: [mcp-ui](https://github.com/MCP-UI-Org/mcp-ui)

MCP は LLM とツール・データを接続するプロトコルとして始まりました。Apps 拡張により、MCP ツールは単なるデータだけでなく、React コンポーネント、フォーム、ダッシュボード、マップといったインタラクティブ UI アーティファクトを返すことができます。

セキュリティモデルは徹底的に厳格で、これが意図された設計です。すべての UI は権限が削除されたサンドボックス化された iframe 内で描画され、テンプレートは事前に宣言されるためホストアプリがレビュー可能です。通信はすべて監査可能な JSON‑RPC で行われます。このモデルはツールプロバイダーに最適です。たとえば Shopify の MCP サーバーはチェックアウトウィジェットを返し、マッピングサービスは埋め込み可能な地図を返すことができます。ホストアプリはそのウィジェットのコードを所有したり信頼したりしません。

MCP Apps は UI が **ツールプロバイダー側に属する** 場合に適しています。UI が自アプリケーションのドメイン内に存在する場合は、パターン 1 または 2 を選択してください。

## JavaScript/TypeScript フレームワーク


### CopilotKit

**GitHub**: [CopilotKit/CopilotKit](https://github.com/CopilotKit/CopilotKit) · Examples: [CopilotKit/generative-ui](https://github.com/CopilotKit/generative-ui)

CopilotKit はエージェントネイティブなフロントエンドアプリケーション向けの最も包括的なフレームワークです。エージェントバックエンドへの AG‑UI 経由の接続、双方向の会話状態管理、生成 UI コンポーネントのレンダリング、エージェントとユーザーが同じデータを操作できるようにする共有状態の配管まで、ライフサイクル全体を扱います。

3 パターンのモデルは CopilotKit の API にきれいにマッピングされます:
- `useCopilotAction` と `render` コールバック → パターン 1  
- A2UI/Open-JSON-UI のレンダリング → パターン 2  
- `OpenGenerativeUI` のサンドボックス化アーティファクト → パターン 3  

あまり語られない重要な CopilotKit の機能は **共有状態とヒューマン・イン・ザ・ループ** です。エージェントはアプリケーション状態を読み書きでき、ユーザーも同様に読み書きでき、変更は双方向に流れます。これが Copilot 風 UI を単なるチャットボックスではなく、実際の共同作業のように感じさせる要因です。

### Vercel AI SDK

**GitHub**: [vercel/ai](https://github.com/vercel/ai) · Docs: [ai-sdk.dev](https://ai-sdk.dev/)

Vercel AI SDK は AI アプリ向けの事実上の TypeScript ベースラインです。生成 UI に特化した機能は次のとおりです。

**`useObject`** はサーバー側で生成される構造化 JSON オブジェクトをストリームします。Zod スキーマを定義すると、SDK が部分的な JSON を解析し、フィールドが到着するたびに再レンダリングをトリガーします。Next.js アプリでパターン 2 に到達する最もスムーズな手段です。

```tsx
const { object: dashboard } = useObject({
  api: "/api/generate-dashboard",
  schema: z.object({
    title: z.string(),
    metrics: z.array(z.object({ label: z.string(), value: z.number() })),
    insights: z.array(z.string()),
  }),
});
```

**`useChat`** にツールハンドラを組み合わせる → パターン 1。モデルがツールを呼び出し、開発者がツール名をコンポーネントにマッピングします。

**AI Elements**（[elements.ai-sdk.dev](https://elements.ai-sdk.dev/)）は、SDK と組み合わせて使用できる既製の UI プリミティブを提供します。

**ここで混乱しやすい経緯についての注記**：2024 年 10 月、Vercel は [GitHub Discussion #3251](https://github.com/vercel/ai/discussions/3251) で、SDK 3.0 の見出し機能として掲げられていた「Generative UI」機能である AI SDK RSC ― React Server Components のストリーミングパターン ― が「長年の制限がいくつかあり」近い将来の解決策が見込めないため、無期限で保留されたと発表しました。RSC ストリーミングを中心に製品戦略を組み立てていたチームは予期せぬ事態に直面しました。`generateObject`／`streamObject` API も SDK 6.0 で廃止されました。AI SDK RSC からの推奨移行先は上記の `useObject` パターン、あるいはカタログベース生成のための json‑render です。

### assistant-ui

**GitHub**： [assistant-ui/assistant-ui](https://github.com/assistant-ui/assistant-ui)

assistant-ui は、プロダクション品質のチャットインターフェイスを構築するためのコンポーザブルな React プリミティブの集合です。メッセージバブル、トークンのストリーミング、コピー／編集／再生成アクション、思考中ステータスといった洗練されたチャット UX が必要で、バックエンドとツール描画は自前で行いたい場合に最適な選択肢です。

任意のバックエンド（OpenAI、Anthropic、ローカルモデル、カスタムエンドポイント）と組み合わせて動作し、ツール呼び出しの描画は慣れ親しんだ slot／render prop モデルで処理します。

### json-render

**GitHub**： [vercel-labs/json-render](https://github.com/vercel-labs/json-render) ・ Docs: [json-render.dev](https://json-render.dev/)

json-render は、パターン 2 を意見の強い「バッテリー同梱」アプローチで具現化します。事前に構築されたコンポーネントカタログ（shadcn/ui コンポーネント＋ Zod スキーマ）、レンダラ、そしてモデルがスキーマでカタログに拘束されるタイトな生成ループが提供されます。

主な特徴は次のとおりです：
- **マルチターゲットレンダリング**：同一の JSON 仕様から React Web アプリ、React Native モバイルアプリ、PDF、HTML メール、あるいは Remotion ビデオへと出力できます。レポート作成に実際に役立ちます。
- **プログレッシブレンダリング**：モデルがストリーミングするたびにコンポーネントが順次表示され、全体の仕様が届くまで待つ必要がありません。
- **タイトなスキーマ制約**：カタログはモデルが「有効だが未知」のコンポーネントタイプを幻覚する余地を残さないよう設計されています。

もしダッシュボードやレポート生成機能を構築していて、独自のカタログ設計にかかるインフラ作業を省きたいなら、json‑render がウェブアプリ向けの最速パスです。

**モメンタムについて**: json‑render は 2026 年初頭に Vercel Labs からリリースされ、標準的な React/Next.js プロジェクトですぐに活用できる点がウェブ開発者の関心を集めています。ただし、json‑render はまだ 1.0 未満で、json‑render と A2UI の関係は現在調整中です — Vercel は A2UI 互換の出力を試験しており、将来的に統合される可能性があります。クロスプラットフォーム（ネイティブモバイル、複数フレームワーク）を狙うなら、A2UI の方が長期的に有利です。

### Hashbrown

**GitHub**: [liveloveapp/hashbrown](https://github.com/liveloveapp/hashbrown)

Hashbrown は独自のアプローチを取ります。別個の AI インターフェース層を構築する代わりに、既存の React または Angular アプリに AI コンポーネント選択を直接埋め込みます。アプリのコンポーネントを LLM に公開し、LLM がどれをレンダリングするかを選択し、クライアント側ツールを呼び出すことができます。

これは「チャット」以外の製品 UI にインテリジェンスを組み込みたい場合に最適です。たとえば、レイアウトを動的に変える製品ページ、適切なオプションを提示する設定パネル、次のステップを提案するワークフローエディタなどです。

### OpenUI

**GitHub**: [thesysdev/openui](https://github.com/thesysdev/openui) · Docs: [openui.com](https://www.openui.com/)

OpenUI は JSON を置き換え、行指向のコードライクなフォーマット（「OpenUI Lang」）を採用しています。このフォーマットはプログレッシブレンダリングとトークン効率を意識して設計されており、複雑なレイアウトに対して同等の JSON より約 67 % 少ないトークンで表現できると主張しています。

トレードオフはエコシステムの成熟度です — OpenUI は比較的新しく、ツールチェーンは JSON 系アプローチに比べて薄いです。しかし、トークンコストが重要な制約であり、頻繁に複雑なレイアウトを生成する場合は、フォーマット効率は実際に効果があります。

### Tambo

**GitHub**: [tambo-ai/tambo](https://github.com/tambo-ai/tambo)

Tambo は状態を保持するコンポーネント選択に焦点を当てます。AI がコンポーネントを選び、クライアント側ツールを通してそれらと対話し、会話全体でコンポーネントの状態を維持します。UI 要素が複数ターンにわたって持続するユースケースに適しています――ユーザーが微調整するフィルタコンポーネントがあり、AI がフィルタ済みデータの解析を続ける、といったケースです。

## The Python Layer

Python エコシステムは AI インターフェースを別の方法で扱います。これらのツールは ML モデルのデモ、データアプリケーション、社内ツール向けに最適化されており、エージェント主導のレイアウト構成を伴う本番向けコンシューマアプリには向いていません。

これは批判ではありません。適切なユースケースでは、Gradio と Streamlit が唯一必要なツールです。

### Gradio

**GitHub**: [gradio-app/gradio](https://github.com/gradio-app/gradio) · PyPI: `gradio`

Gradio の核心的価値は、Python 関数を書くだけで、Gradio がそれを Web UI にラップしてくれることです。`Interface` クラスは画像分類器で 3 行、`ChatInterface` はチャットボットで 10 行です。レイアウトを細かく制御したい場合は `Blocks` を使用します。

「generative UI」は Gradio では Python 開発者が定義し、モデルが決めるわけではありません。コンポーネントの表示・設定はモデルの出力に応じて動的に変化させられますが、コンポーネントカタログ自体は固定です――モデルにレイアウトを組み立てさせることはありません。

Gradio は HuggingFace Spaces や ML デモエコシステムのデフォルトです。月間ダウンロード数は数百万件に上り、AI デモ全体の大部分を支えています。

**Gradio を選ぶべきケース**: Python 開発者が ML モデルのデモ、研究プロトタイプ、あるいは社内ツールを構築し、JavaScript に触れたくない場合。

### Streamlit

**GitHub**: [../streamlit/streamlit](https://github.com/streamlit/streamlit)

Streamlit のアプローチはやや意見主導的です。Python スクリプトが各インタラクションごとに全体を実行します。`st.chat_message()`、`st.dataframe()`、`st.plotly_chart()` を呼び出すだけで、フレームワークがレイアウトを管理します。

全スクリプト再実行モデルは非効率に思えるかもしれませんが、会話履歴を保持する AI チャットボットに対しては意外に使いやすく、スクリプト全体が再実行され、チャット履歴はセッションステートに保存され、出力は決定的になります。Streamlit は主要な LLM プロバイダー向けのファーストパーティサポートを提供し、Snowflake Cortex ともネイティブに統合されています。

**Streamlit を選ぶべきケース**: Python で AI 搭載のデータアプリ、社内レポートツール、あるいは ML バックエンドのダッシュボードを構築し、できるだけシンプルなデプロイパスを求める場合。

### LangChain と Haystack

これらはバックエンドのオーケストレーションフレームワークであり、UI フレームワークではありません。構造化された出力を生成しフロントエンドへ送る層として、正直な generative UI スタックマップに必ず登場します。

**LangChain**（[langchain-ai/langchain](https://github.com/langchain-ai/langchain)）: 任意の LLM に対して `.with_structured_output()` を呼び出すと、Pydantic で制約された JSON が生成されます。`@tool` デコレータに自動スキーマ生成を組み合わせるのが、モデルが呼び出せるツールを定義する最もクリーンな方法です。LangChain は構造化された結果を、使用しているフロントエンド層へそのまま渡します。

**Haystack**（[deepset-ai/haystack](https://github.com/deepset-ai/haystack)）: モジュラーなパイプラインアーキテクチャで、RAG（Retrieval‑Augmented Generation）を強力にサポートします。Hayhooks は Haystack パイプラインを HTTP エンドポイントとしてラップし、MCP 互換のエンドポイントも提供します。生成 UI が検索バックボーンを必要とする場合、Haystack のパイプライン構造がそれをきれいに処理します。

どちらのフレームワークも UI 層を所有しているわけではありません。フロントエンド（パターン 1、2、または 3）が描画するためのデータを生成するだけです。

## エコシステムの速度と不安定な土台

この領域は急速に変化しており、いくつかのプロジェクトはコードと同時に混乱を招く情報を出しています。2026 年 5 月 8 日時点で最終確認済みです。ここに記したプロジェクトステータスは「タイムスタンプ付きの読み取り情報」として扱い、永続的な判断材料にはしないでください。

**Vercel AI SDK RSC** は SDK 3.0 がリリースされたときの旗艦 Generative UI 機能でした。Vercel は 2024 年 10 月に開発を一時停止しました（[Discussion #3251](https://github.com/vercel/ai/discussions/3251)）。React Server Components のアーキテクチャ上の制限が近い将来に解決できないと判断したためです。これを基盤に構築していたチームは当然ながらフラストレーションを抱きました。ドキュメントにはまだ記載されていますが、推奨パスではなくなっています；代わりに `useObject` が推奨されます。

**json-render**（Vercel Labs）は新しい方向性です――カタログベースでフレームワークに依存しない代替手段で、RSC との結合問題を回避します。1.0 未満の段階ですが、React／Web 開発者の間で早期関心が高いようです。考えられる DX の理由は、json-render が標準的な React/Next.js プロジェクトですぐに利用できるのに対し、A2UI のクロスプラットフォーム範囲はセットアップの摩擦を増やす点です。両方の仕様が成熟するにつれてどうなるかは正直なところ不透明です。Vercel は json-render で A2UI 互換性を試みており、収束の可能性を示唆しています。

**A2UI**（Google）は 1.0 未満（最終確認時は v0.9）で、マイナーバージョン間に破壊的変更があり、ロードマップに関する Google の情報が一貫していません。json-render がカバーしないクロスプラットフォーム展開（Web + Flutter + SwiftUI）を実現したい場合に適しており、エンタープライズの裏付けもあります。純粋な Web プロジェクトに限って言えば、現在の DX はやや厳しいです。

**AG-UI**（CopilotKit）も 1.0 未満です。最も一般的な混乱は、名前が UI フレームワークのように聞こえる点です。実際には UI フレームワークではなく、トランスポートプロトコルです。AG-UI はエージェントバックエンドとフロントエンド間のイベントフローを定義し、レスポンスとして何を描画するかは依然として開発者の判断に委ねられます。その概念は堅牢で広く採用されていますが、1.0 未満の仕様であるためエッジケースはまだ解決途上です。

実務的な結論：**ここに挙げた主要プレイヤーはすべて 1.0 未満**です。API の変更を見越して計画してください。パターン――ツール‑to‑コンポーネント、カタログ構成、サンドボックス化生成――は十分に安定しているので上に乗ることは可能です。具体的なプロトコル選択はまだ流動的です。

## Component Catalog Design: The Real Engineering Work

Pattern 2 で興味深い複雑性の多くはレンダラにあるのではなく、カタログにあります。

カタログは **スキーマとしてエンコードされた製品決定** です。次の質問に答えます：このドメインで意味のある UI オブジェクトは何か？ 「どの React コンポーネントが存在するか？」ではなく、「このコンテキストのユーザーが実際に見るべきもの、操作すべきものは何か？」ということです。

**The too-granular failure mode**: `Row`、`Column`、`Text`、`Button`、`Icon` をすべて公開したとします。するとモデルはフロントエンドエンジニアにならなければならず、デザインシステムに合わない中途半端なレイアウトを生成したり、空状態を見落としたり、アクセシビリティに欠けるマークアップを出したり、カタログに出力を制限する要素がないためレスポンスごとにアプローチが変わってしまいます。

**The too-coarse failure mode**: `WeatherCard`、`FlightCard`、`HotelCard` を公開すると、ユーザーが事前に用意されたカードにマッピングできない要求をしたときにモデルは対応できず、テキストにフォールバックします。

**The useful middle**: スロットが制限されたドメインレベルのコンポーネント。

旅行アプリのカタログは次のようになるかもしれません:

```
TripSummary         — itinerary at a glance
FlightOptionList    — selectable flight options with pricing
HotelComparison     — side-by-side hotel cards
TravelerForm        — collect traveler details
PolicyNotice        — regulatory/fare rule callout
BookingConfirmation — final confirmation with action button
```

金融アプリのカタログは次のようになるかもしれません:

```
PortfolioSnapshot   — key positions and P&L
TransactionTable    — filterable, paginated transactions
RiskBreakdown       — allocation and volatility metrics
ScenarioComparison  — side-by-side scenario modeling
ApprovalGate        — action requiring human confirmation
```

カタログは製品の語彙のように機能します。UX の決定、アクセシビリティ要件、空状態の取り扱い、危険なアクションパターンをコンポーネントコードにエンコードします。モデルはそれらの部品を組み立てるだけです。各部品がどのように見えるか、何が許可されるかは依然として開発者が決めます。

**Schema design rules that reduce hallucinations**:

1. 列挙型の値は短く分かりやすく保つ。`"type": "bar_chart"` とし、`"type": "data-visualization-bar-type-vertical"` のように長くしない。
2. 無効な構成を不可能にする。`PolicyNotice` がレイアウトの最後にしか現れない場合、どこにでも現れる要素と同じスキーマレベルに置かない。
3. 必須フィールドは積極的に使用する。オプションフィールドはモデルが省略する可能性があり、レンダラ側で null 対応が必要になる。
4. カタログを実際のプロンプトでテストしてからリリースする。生成された仕様を保存し、スキーマ違反、幻覚的なフィールド値、技術的には有効だが意味的に誤った構成がないかチェックする。

## Common Traps

**Trap: treating valid JSON as safe behavior.** スキーマ検証は構造を確認するだけで、ボタンのラベルと実際のアクションが一致しているか、合計が元データと合っているか、ユーザーが予期しない操作を UI コンポーネントが行っていないかは保証しません。生成された UI 仕様はスキーマ検証だけでなく、意味的なレビューが必要です。最低限、破壊的なアクションには確認コンポーネントが必要であり、そのラベルはトリガーするアクションと照合してテストすべきです。

**Trap:製品プリミティブではなくデザインプリミティブを公開している。** モデルに 16px か 20px のパディングを選ばせようとすると、抽象化レベルが間違っています。ドメインコンポーネントは製品の趣向をエンコードすべきです。モデルは振る舞いを組み立てるべきで、プレゼンテーションの細部を管理すべきではありません。

**Trap: 静的 UI で足りるところで生成 UI を使う。** 表示したい構造が開発時に決定できる—ほとんどの場合そうです—場合は、事前構築されたコンポーネントを用いる Pattern 1 が高速で安全かつ一貫性があります。生成 UI がその複雑さを正当化するのは、構造がデータやタスクコンテキストに応じて本当に変化する場合だけです。

**Trap: アクセシビリティを省く。** LLM は WCAG 違反を幻覚的に生成します。インタラクティブ要素に `role="region"` を付与したり、ラベルが欠落したフォームを作ったり、WCAG AA に違反するコントラスト比を出したりします。コンポーネントライブラリ自体が完全にアクセシブルであっても、AI が組み立てたそのコンポーネントの組み合わせが自動的にアクセシブルになるわけではありません。コンポーネント単体だけでなく、全体のレンダリングパスをテストしてください。

**Trap: プロトコルとフレームワークを混同する。** AG‑UI はフロントエンドフレームワークではなく、A2UI は React ライブラリでもありません。これらはワイヤーフォーマットおよびイベントプロトコルです。実装には依然としてフロントエンドフレームワークが必要です。CopilotKit は AG‑UI と A2UI を実装しています。json‑render は A2UI/Open‑JSON‑UI カタログパターンを実装しています。これらは別層です。

## ユースケース別推奨

**既存 SaaS アプリにコパイロットを追加する**: Pattern 1（tool‑to‑component）から始めます。Vercel AI SDK の `useChat` または CopilotKit を使用し、上位 5〜10 のエージェントアクションを事前構築コンポーネントにマッピングします。まずそれをリリースし、計測した上で、ユーザーが実際にリッチな構成を必要とした場合にだけカタログを拡張します。

**自然言語からのダッシュボード生成**: Pattern 2 と json‑render あるいはカスタム A2UI カタログを利用します。チャート種別、メトリックカード、テーブルバリエーションをカバーする 8〜15 種類のコンポーネントタイプをカタログ化し、スキーマをモデルに渡してレイアウトを組み立たせます。未知のタイプがレンダラに到達する前に捕捉するバリデーションを構築してください。

**マルチエージェントフロントエンド**: CopilotKit と AG‑UI を組み合わせます。イベントストリームがエージェントバックエンド間のリアルタイムストリーミングを処理し、共有状態がエージェント間のハンドオフを管理し、HITL パターンが承認ゲートを提供します。

**ChatGPT や他の MCP ホスト内で構築する**: MCP Apps を使用します。データ取得と推論を行うツールと、ウィジェットを要求する別のレンダーツールを定義します。ビジネスロジックはウィジェットテンプレートから分離してください。

**ML モデルのデモとデータアプリ（Python チーム）**: デモには Gradio、データアプリには HuggingFace Spaces を使用します。より複雑なインタラクションが必要な場合は Streamlit を選択します。どちらも JavaScript を必要としません。

**ビジュアルアーティファクト、シミュレーション、図**: パターン 3（OpenGenerativeUI または同等）を利用します。厳格な iframe CSP を設定し、出力をセキュリティ上は信頼できないユーザーコンテンツとして扱います。

---

フレームワークは急速に成熟しています。プロトコルの収束（ストリーミング向け AG-UI、カタログ仕様向け A2UI/Open-JSON-UI）はまだ進行中ですが、実装に十分な形が見えてきています。

現在最も重要なエンジニアリング課題は、フレームワークの選択ではありません。**カタログ設計**です。モデルが何を言えるかを決めることで、技術的スキルよりも製品側の明確さが求められます。**セマンティック検証**も不可欠です。生成された UI が主張通りに機能するかをテストし、単にスキーマ検証を通過するだけでなく実際の動作を確認します。さらに**アクセシビリティのギャップ**があります。すべてのコンポーネント、そしてコンポーネントの組み合わせが、手書き UI が満たすべきアクセシビリティ基準を満たすようなカタログを構築する必要があります。
