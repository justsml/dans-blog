"Generative UI" bedeutet je nach Sprecher mindestens fünf unterschiedliche Dinge.

- Chat‑Interfaces, die Produkt‑Karten aus Modell‑Tool‑Aufrufen einbetten  
- Laufzeit‑JSON‑Spezifikationen, die das Frontend als Komponenten‑Bäume rendert  
- Sandgeboxte Iframes, die von MCP‑Tools an Host‑Apps zurückgegeben werden (von Ticket‑Bestellung, Hotelbuchung bis Karten‑Rendering, Checkout‑Widgets)  
- Ereignis‑Protokolle, die Agenten‑Zustand zum Frontend streamen  
- v0, Lovable und Bolt: KI‑Tools, die React zur Design‑Zeit schreiben  

Das sind verwandte Konzepte, aber sie existieren auf unterschiedlichen Schichten des Stacks und haben unterschiedliche Risikoprofile, Implementierungskosten und passende Anwendungsfälle. Wenn man sie zusammenwürfelt, wird jede Architekturdiskussion zu einem Chaos.

Das ist die Karte, die ich brauche, wenn ich entscheide, wo im Stack ich ansetzen soll.

---

## Was Generative UI nicht ist

Bevor definiert wird, was es ist, drei Dinge beiseitelegen:

**Design‑Zeit Code‑Generierung** — v0, Lovable, Bolt, Cursor, die React‑Komponenten zusammensetzen. Diese Werkzeuge erzeugen Code, den Entwickler prüfen und committen. Die KI läuft zur Entwicklungszeit. Was ausgeliefert wird, ist aus Sicht des Nutzers statisch. Das ist eine großartige Kategorie von Tools. Es ist nicht das, was „generative UI zur Laufzeit“ bedeutet.

**KI‑unterstützte Formular‑Autovervollständigung** — das Modell füllt Feldwerte aus dem Kontext aus. Die Struktur der Oberfläche bleibt fest; nur der Inhalt ändert sich. Das ist ein nützliches Muster. Es ist kein generatives UI.

**KI schreibt rohes HTML in eine Seite** — das Modell gibt `<div>`‑ und `<button>`‑Strings aus, die über `innerHTML` oder `dangerouslySetInnerHTML` injiziert werden. Das *ist* Laufzeit‑generatives UI im strengsten technischen Sinn. Es ist zugleich die gefährlichste Variante und die, die jedes ausgereifte Framework in diesem Bereich zu vermeiden versucht. Roh von KI erzeugtes Markup bedeutet XSS‑Risiko, nicht‑barrierefreie Attribute, inkonsistentes Styling und halluzinierte Struktur. Der Rest dieses Artikels beschäftigt sich damit, wie man es besser macht.

## EineArbeitsdefinition

Generatives UI zur Laufzeit bedeutet: **das Modell bestimmt, welche Benutzeroberflächen‑Komponente oder -Komposition dem Nutzer angezeigt wird, basierend auf dem Zustand der Unterhaltung oder Aufgabe.**

Nicht die Worte. Die Oberfläche.

Der einfachste Fall: Ihr Flugbuchungs‑Assistent ruft ein `search_flights`‑Tool auf. Anstatt reinen Text zurückzugeben („Hier sind drei Optionen …“), rendert er eine `<FlightResultsCard>`‑Komponente mit auswählbaren Flügen, Sitzklassen‑Umschaltern und einem „Buchen“-Button. Das Modell hat entschieden, dass eine strukturierte Karte hier die passende Antwort ist. Der Entwickler hat festgelegt, wie diese Karte aussieht und was „Buchen“ bewirkt.

Der komplexere Fall: Ein Finanzanalyse‑Agent erhält eine Frage zu einem Portfolio und entscheidet sich, eine Antwort mit einer `MetricGroup` zu komponieren, die Schlüsselzahlen zeigt, einem `RiskBreakdown`‑Diagramm, einer `ScenarioComparison`‑Tabelle und einem `PolicyNotice`. Das Modell hat dieses Layout aus einem Katalog vorab genehmigter Komponenten zusammengebaut. Der Entwickler hat jede Komponente definiert. Das Modell wählte, welche verwendet werden und welche Daten darin platziert werden.

Beide Fälle sind generatives UI. Sie unterscheiden sich darin, wie viel Kompositionsfreiheit das Modell hat, was sowohl die Vielfalt möglicher Ausgaben als auch die Komplexität potenzieller Fehler bestimmt.

## Die drei Muster

Der gesamte Raum lässt sich auf drei Muster reduzieren, von denen jedes eine eigene Ausgabesyntax hat.

![EinSpektraldiagramm, das drei Muster zeigt: Werkzeugaufrufe nur links (am sichersten), Komponenten‑Katalog in der Mitte und offene Generierung rechts (am ausdrucksstärksten).](../output-grammar-spectrum.svg)

_Jede generative‑UI‑Entscheidung ist ein Punkt auf diesem Spektrum. Beginnen Sie links._

### Muster 1: Tool‑zu‑Komponente‑Rendering

Das Modell ruft ein benanntes Tool auf. Ihre Anwendung besitzt eine Zuordnung von Tool‑Namen zu Komponenten. Der Tool‑Aufruf löst das Rendern einer Komponente aus.

```tsx
// Das Modell ruft auf: { name: "show_flight_results", args: { flights: [...] } }

useCopilotAction({
  name: "show_flight_results",
  render: ({ args }) => <FlightResultsCard flights={args.flights} />,
});
```

Dies ist das sicherste Muster, weil das Layout niemals vom Modell stammt. Das Modell entscheidet *wann* eine Komponente angezeigt wird und *welche Daten* sie füllen. Ihre Entwickler behalten die Verantwortung für den Komponenten‑Code, das visuelle Design, die Barrierefreiheits‑Implementierung und jede Randbedingung in der Render‑Logik.

Das `useChat`‑Feature des Vercel AI‑SDKs mit `tool`‑Handlern funktioniert so. Auch das Tool‑Rendering von assistant‑ui folgt diesem Ansatz. CopilotKit’s „Static Generative UI“ ist dieses Muster. Die meisten produktiven Copilot‑UIs, die zuverlässig funktionieren, setzen darauf.

**Passend, wenn**: die Menge der Dinge, die Sie anzeigen möchten, zur Entwicklungszeit bekannt ist. Buchungsbestätigungen, Suchergebnisse, Kontozusammenfassungen, Genehmigungs‑Widgets. Wenn Sie die Szenarien aufzählen können, deckt dieses Muster sie ab.

### Muster 2: Komponenten‑Katalog‑Komposition

Das Modell gibt einen typisierten JSON‑Baum aus, der Komponenten aus einem von Entwicklern definierten Katalog referenziert. Ihr Frontend verfügt über einen Renderer, der den Baum traversiert und jede Komponente instanziiert.

```json
[
  { "type": "metric_group", "metrics": [
    { "label": "MRR", "value": "$82,400", "delta": "+12%" },
    { "label": "Churn", "value": "2.1%", "delta": "-0.4%" }
  ]},
  { "type": "line_chart", "title": "30-day growth", "data_ref": "mrr_series" },
  { "type": "insight_callout", "text": "Expansion revenue driving the delta — avg seat count up 18%." }
]
```

Das Modell hat dieses Layout zusammengesetzt: ein `MetricGroup`, ein `LineChart`, ein `InsightCallout`. Aber Sie haben definiert, was jeder Komponententyp bedeutet, welche Props er akzeptiert und wie er gerendert wird. Versucht das Modell, `{ "type": "custom_untested_thing" }` zu erzeugen, fängt Ihre Schema‑Validierung das ab und der Renderer ignoriert oder verwirft es.

Dies ist das Muster hinter `json-render`, `A2UI`, `Hashbrown`, `OpenUI` und `Tambo`. Die zentrale ingenieurtechnische Aufgabe ist **Katalogdesign** — Festlegen, welche Komponententypen existieren, wie ihre Schemata aussehen und was das Modell darf und nicht darf zu komponieren.

**Passend, wenn**: die Struktur dessen, was Sie anzeigen wollen, legitim basierend auf den Daten oder der Benutzeranfrage variiert. Dashboards, die sich an das Bemerkenswerte in den Zahlen anpassen. Berichte, die je nach Kontext unterschiedliche Abschnitte zeigen. Workflow‑Panels, die sich ändern, je nachdem, in welchem Schritt ein Agent sich befindet.

### Muster 3: Offene Generierung

Das Modell schreibt HTML, SVG, Canvas oder WebGL, das innerhalb eines sandboxed iframes mit einer strikten Content‑Security‑Policy gerendert wird.

Dies ist geeignet für Fälle, in denen kein fester Komponenten‑Katalog ausreicht: Algorithmus‑Visualisierungen, Architektur‑Diagramme, Ad‑hoc‑Charts, generative Kunst, pädagogische Simulationen. Die iframe‑Grenze übernimmt hier die Sicherheitsarbeit; entfernen Sie sie, landen Sie wieder beim reinen HTML‑Injection‑Problem, das zu Beginn dieses Artikels beschrieben wurde.

`CopilotKit/OpenGenerativeUI` ist die derzeit beste Referenzimplementierung dieses Musters. Die Sandbox entfernt Skripte, begrenzt Nachrichten‑Austausch und hält das erzeugte Artefakt von dem privilegierten Zustand Ihrer Anwendung fern.

**Passend, wenn**: Sie tatsächlich beliebige visuelle Ausgaben benötigen — einmalige erklärende Diagramme, dynamische Simulationen, kreative Artefakte. Verwenden Sie dies nicht für transaktionale UI. Eine Checkout‑Bestätigung braucht keinen sandboxed iframe.

### Jenseits der drei Muster: LLMs steuern Pixel direkt

Es entsteht ein vierter Ansatz, der in keines der bisherigen Muster passt: LLMs erzeugen **immersive, spielähnliche Erlebnisse**, indem sie die visuelle Ausgabe direkter steuern als ein sandboxed iframe.

Die kanonische Unterscheidung innerhalb generativer UI lautet **iframe‑HTML vs. JSON‑Katalog**:

- **Iframe HTML** — das Modell schreibt HTML, SVG, Canvas oder WebGL, das in einem isolierten Sandbox‑Iframe gerendert wird. Maximale Ausdrucksfreiheit; die Sicherheit hängt vollständig von der iframe‑Grenze ab. Beispiele: Anthropic Artifacts, OpenGenerativeUI.  
- **JSON‑Katalog** — das Modell gibt ein strukturiertes Payload aus, das auf einen vom Entwickler definierten Komponenten‑Katalog beschränkt ist; Ihr Renderer instanziiert vertrauenswürdige, vorgefertigte Komponenten aus dieser Spezifikation. Das Modell entscheidet *was* angezeigt wird; Sie entscheiden *wie* es gerendert wird. Beispiele: json-render, A2UI.

Darüber hinaus deuten sehr aktuelle Demos auf einen dritten Modus hin, bei dem das Modell weder Komponenten auswählt noch sandboxed HTML schreibt — es steuert die Zeichenfläche direkter. Projekte wie [Tencent's HunyuanWorld](https://arxiv.org/abs/2502.01999), das erkundbare 3D‑Umgebungen aus einem einzigen Bild erzeugt, und Spiel‑Architekturen, bei denen LLMs zur Laufzeit Karten, NPCs und Quests generieren, anstatt einen Komponenten‑Katalog aufzurufen, lassen vermuten, dass das Modell künftig eher einem Spieldirektor als einem Formular‑Renderer ähnelt. In‑Browser‑LLM‑Inference via WebGPU ([WebLLM](https://mlc.ai/web-llm/)) treibt dieselbe Grenze lokal voran.

Dieses Gebiet ist wirklich spannend und gleichzeitig noch ganz am Anfang. Es gibt noch keine stabilen Frameworks, mit denen Produktions‑Produkte gebaut werden können. Ich werde diesen Ansatz in einem eigenen Artikel behandeln, sobald sich das ändert.

## Die Protokolle: AG‑UI und A2UI

AG‑UI und A2UI sind die beiden Hauptstandards in der Protokoll‑Schicht. Sie lösen unterschiedliche Probleme und stehen nicht in Konkurrenz zueinander.

### AG-UI

**GitHub**: [ag-ui-protocol/ag-ui](https://github.com/ag-ui-protocol/ag-ui)

AG-UI ist ein ereignisbasierendes Protokoll für die Kommunikation zwischen KI‑Agenten und Frontend‑Anwendungen. Es definiert etwa 16 Ereignistypen: `TEXT_MESSAGE_START`, `TEXT_MESSAGE_CONTENT`, `TOOL_CALL_START`, `TOOL_CALL_END`, `STATE_SNAPSHOT`, `STATE_DELTA` und so weiter. Der Transport bleibt Ihnen überlassen — SSE, WebSockets, Webhooks funktionieren alle. Das Format ist bewusst locker gehalten, um eine breite Adoption zu ermöglichen.

AG-UI legt nicht fest, wie Ihre UI aussieht. Es definiert, **wie** der Agent *mit* Ihrem Frontend kommuniziert. Denken Sie an die Draht‑Protokoll‑Schicht, die Ihrer React‑App erlaubt, sich genauso wie bei einem CrewAI‑Agenten bei einem LangGraph‑Agenten anzumelden, ohne Frontend‑Code zu ändern.

CopilotKit hat AG-UI aus ihrer Arbeit mit LangGraph und CrewAI heraus entwickelt. Es wird von LangChain, Mastra, PydanticAI und anderen verwendet. Microsoft hat einen AG-UI‑Integrationsleitfaden veröffentlicht. Wenn Sie ein Multi‑Agent‑Frontend bauen und Backend‑Frameworks vom Frontend‑Code entkoppeln müssen, ist AG-UI die Lösung.

**Ein Hinweis, der häufig verwirrt**: AG-UI ist kein UI‑Framework. Es sagt Ihnen nicht, was gerendert werden soll. Es sagt Ihnen *dass* der Agent etwas gesagt, ein Tool aufgerufen oder den geteilten Zustand aktualisiert hat. Was Sie daraufhin rendern, bleibt Ihre Entscheidung.

### A2UI

**GitHub**: [google/A2UI](https://github.com/google/A2UI) · Spec: [a2ui.org](https://a2ui.org/)

A2UI ist Googles deklarative Spezifikation dafür, was Agenten senden, wenn sie UI anzeigen wollen. Während AG-UI die Frage „Wie kommuniziert der Agent?“ beantwortet, beantwortet A2UI „Welches Format verwendet der Agent, um ein Komponenten‑Layout zu beschreiben?“.

A2UI verwendet ein flaches JSONL‑Format: ein Komponenten‑Deskriptor pro Zeile, jeweils mit einer ID, einem Typ und Daten. Flach ist beabsichtigt. Verschachtelte Bäume erfordern, dass das Modell die gesamte Struktur kennt, bevor es mit dem Streamen beginnen kann. Eine flache Liste erlaubt dem Modell, jede Komponente zu emitten, sobald es darüber „nachdenkt“, was bedeutet, dass Ihr Frontend bereits die erste Metrik‑Karte rendern kann, während das Modell noch entscheidet, ob ein Diagramm hinzugefügt werden soll.

```jsonl
{"id":"h1","type":"kpi_card","title":"MRR","value":"$82,400","delta":"+12%"}
{"id":"h2","type":"kpi_card","title":"Churn","value":"2.1%","delta":"-0.4%"}
{"id":"c1","type":"line_chart","title":"30-day MRR","data_ref":"mrr_series"}
{"id":"t1","type":"data_table","cols":["Month","MRR","Net New"],"data_ref":"monthly"}
```

A2UI ist sicherheitsbewusst: Die Spezifikation ist ein Datenformat, kein ausführbarer Code. Der Komponenten‑Katalog wird vom Entwickler vorgegeben; der Agent kann nur Typen aus diesem Katalog referenzieren. Ein A2UI‑Renderer, der einen unbekannten Typnamen erhält, ignoriert ihn.

CopilotKits „Open‑JSON‑UI“-Format ist mit A2UI kompatibel. Wenn Sie heute ein Spezifikationsformat für einen Komponenten‑Katalog wählen, ist A2UI das mit der breitesten plattformübergreifenden Unterstützung.

**Ein Hinweis zur Stabilität**: A2UI befindet sich noch vor Version 1.0 – v0.9 beim letzten Check am 8. Mai 2026 – und hat zwischen Minor‑Versionen breaking changes eingeführt. Googles Kommunikation zum Fahrplan war sporadisch, und einige Renderer (Lit, Flutter) hinken den Spezifikations‑Updates hinterher. Planen Sie Zeit für Spezifikations‑Drift ein, wenn Sie heute darauf aufbauen. Für reine Web‑Anwendungsfälle scheint **json‑render** derzeit umfassendere Werkzeuge zu bieten. Der langfristige Vorteil von A2UI liegt in der plattformübergreifenden Reichweite (Web, Flutter, SwiftUI, Android), die json‑render nicht hat.

### MCP‑Apps

**GitHub**: [modelcontextprotocol](https://github.com/modelcontextprotocol) · Verwandt: [mcp-ui](https://github.com/MCP-UI-Org/mcp-ui)

MCP begann als Protokoll zum Verbinden von LLMs mit Tools und Daten. Die Apps‑Erweiterung lässt MCP‑Tools nicht nur Daten, sondern interaktive UI‑Artefakte zurückgeben: React‑Komponenten, Formulare, Dashboards, Karten.

Das Sicherheitsmodell ist strikt und das ist beabsichtigt: Alles wird in einem sandboxed iframe mit reduzierten Berechtigungen gerendert, Vorlagen sind vorab deklariert, sodass die Host‑App sie prüfen kann, und die gesamte Kommunikation erfolgt über auditable JSON‑RPC. Das ist das richtige Modell für Tool‑Provider – ein Shopify‑MCP‑Server kann ein Checkout‑Widget zurückgeben; ein Mapping‑Dienst kann eine einbettbare Karte liefern. Die Host‑App besitzt oder vertraut nicht dem Code dieses Widgets.

MCP‑Apps ist die richtige Wahl, wenn die UI *dem Tool‑Provider gehört*, nicht Ihrer Anwendung. Für UI, die im Kontext Ihrer Anwendung lebt, sollten Sie Muster 1 oder 2 verwenden.

## Die JavaScript/TypeScript‑Frameworks

### CopilotKit

**GitHub**: [CopilotKit/CopilotKit](https://github.com/CopilotKit/CopilotKit) · Beispiele: [CopilotKit/generative-ui](https://github.com/CopilotKit/generative-ui)

CopilotKit ist das umfassendste Framework für agenten‑native Frontend‑Anwendungen. Es übernimmt den gesamten Lebenszyklus: Verbindung zu Agent‑Backends über AG‑UI, Verwaltung des bidirektionalen Konversations‑State, Rendering generativer UI‑Komponenten und Bereitstellung der gemeinsamen State‑Infrastruktur, die Agenten und Nutzern erlaubt, dieselben Daten zu lesen und zu schreiben.

Das Drei‑Muster‑Modell lässt sich sauber auf die APIs von CopilotKit abbilden:
- `useCopilotAction` mit einem `render`‑Callback → Muster 1
- A2UI/Open-JSON-UI Rendering → Muster 2
- `OpenGenerativeUI` sandboxed artifacts → Muster 3

Das wichtige CopilotKit‑Feature, das zu selten diskutiert wird, ist **gemeinsamer State und Human‑in‑the‑Loop**: Der Agent kann Anwendungs‑State lesen und schreiben, der Nutzer kann ihn ebenfalls lesen und schreiben, und Änderungen fließen bidirektional. Das lässt Copilot‑artige UIs wie echte Zusammenarbeit wirken, statt wie ein an ein Produkt angehefteter Chat‑Box.

### Vercel AI SDK

**GitHub**: [vercel/ai](https://github.com/vercel/ai) · Docs: [ai-sdk.dev](https://ai-sdk.dev/)

Das Vercel AI SDK ist de‑facto das TypeScript‑Grundgerüst für KI‑Apps. Für generative UI im Speziellen:

**`useObject`** streamt ein strukturiertes JSON‑Objekt vom Server, während es erzeugt wird. Man definiert ein Zod‑Schema; das SDK parsed das partielle JSON und löst Re‑Renders aus, sobald Felder ankommen. Das ist der glatteste Weg zu Muster 2 in einer Next.js‑App.

```tsx
const { object: dashboard } = useObject({
  api: "/api/generate-dashboard",
  schema: z.object({
    title: z.string(),
    metrics: z.array(z.object({ label: z.string(), value: z.number() })),
    insights: z.array(z.string()),
  }),
});
```

**`useChat` mit Tool‑Handlern** → Muster 1. Das Modell ruft Tools auf; man mappt Tool‑Namen zu Komponenten.

**AI Elements** ([elements.ai-sdk.dev](https://elements.ai-sdk.dev/)) stellt gebrauchsfertige UI‑Primitives bereit, die zum SDK passen.

**Ein Hinweis zur verwirrenden Entwicklung**: Im Oktober 2024 kündigte Vercel in [GitHub Discussion #3251](https://github.com/vercel/ai/discussions/3251) an, dass AI SDK RSC — das React Server Components‑Streaming‑Muster, das als Haupt‑„Generative UI“‑Feature in SDK 3.0 beworben wurde — auf unbestimmte Zeit pausiert wird, weil „mehrere langjährige Einschränkungen“ ohne absehbare Kurzzeit‑Lösungen bestehen. Teams, die ihre Produktstrategie auf RSC‑Streaming gebaut hatten, wurden überrascht. Die `generateObject`/`streamObject`‑APIs wurden später in SDK 6.0 ebenfalls veraltet. Die empfohlene Migration von AI SDK RSC ist das oben gezeigte `useObject`‑Muster oder json‑render für katalogbasierte Generierung.

### assistant-ui

**GitHub**: [assistant-ui/assistant-ui](https://github.com/assistant-ui/assistant-ui)

assistant-ui ist ein Set aus komponierbaren React‑Primitives zum Aufbau von Chat‑Interfaces in Produktionsqualität. Es ist die richtige Wahl, wenn Sie ein ausgereiftes Chat‑UX benötigen — Nachrichtenblasen, Streaming‑Tokens, Kopier‑/Bearbeit‑/Neu‑Generieren‑Aktionen, Denk‑Zustände — und gleichzeitig Ihr eigenes Backend sowie Ihre eigene Tool‑Darstellung einbringen wollen.

Es lässt sich problemlos mit jedem Backend (OpenAI, Anthropic, lokale Modelle, eigene Endpunkte) kombinieren und übernimmt das Rendern von Tool‑Aufrufen über ein vertrautes Slot-/Render‑Prop‑Modell.

### json-render

**GitHub**: [vercel-labs/json-render](https://github.com/vercel-labs/json-render) · Docs: [json-render.dev](https://json-render.dev/)

json-render setzt Muster 2 mit einem meinungsstarken, batteries‑included Ansatz um. Sie erhalten einen vorgefertigten Komponenten‑Katalog (shadcn/ui‑Komponenten mit Zod‑Schemas), einen Renderer und einen engen Generierungs‑Loop, bei dem das Modell durch das Schema an den Katalog gebunden ist.

Die Unterscheidungsmerkmale:
- **Multi‑Target‑Rendering**: dieselbe JSON‑Spezifikation kann in einer React‑Web‑App, einer React‑Native‑Mobile‑App, einem PDF, einer HTML‑E‑Mail oder einem Remotion‑Video gerendert werden. Das ist für Berichte wirklich nützlich.
- **Progressives Rendering**: Komponenten erscheinen, sobald das Modell sie streamt, nicht erst nach Ankunft der kompletten Spezifikation.
- **Strenge Schema‑Beschränkungen**: Der Katalog ist so gestaltet, dass das Modell keine gültigen, aber unbekannten Komponenten‑Typen halluzinieren kann.

Wenn Sieein Dashboard‑ oder Bericht‑Generierungs‑Feature bauen und die Infrastrukturarbeit für einen eigenen Katalog überspringen wollen, ist **json‑render** der schnellste Weg für Web‑Apps.

**Zum Momentum**: json‑render wurde Anfang 2026 von Vercel Labs gestartet und scheint schnell die Aufmerksamkeit von Web‑Entwicklern zu erlangen, weil es sofort in Standard‑React/Next.js‑Projekten nutzbar ist. Trotzdem befindet sich json‑render noch in der Vor‑1.0‑Phase und die Beziehung zwischen json‑render und A2UI wird noch geklärt – Vercel hat bereits A2UI‑kompatible Ausgaben ausprobiert, sodass eine Konvergenz möglich ist. Für plattformübergreifende Szenarien (native Mobile, mehrere Frameworks) ist A2UI die langfristig bessere Wahl.

### Hashbrown

**GitHub**: [liveloveapp/hashbrown](https://github.com/liveloveapp/hashbrown)

Hashbrown verfolgt einen besonderen Ansatz: Anstatt eine separate KI‑Schnittstellenschicht aufzubauen, bettet es die KI‑Komponentenauswahl direkt in Ihre bestehende React‑ oder Angular‑App ein. Sie stellen die Komponenten Ihrer App dem LLM zur Verfügung; das LLM wählt aus, welche gerendert werden sollen, und kann client‑seitige Werkzeuge aufrufen.

Das ist das passende Werkzeug, wenn Sie Intelligenz in Produktflächen einfließen lassen wollen, die nicht „Chat“ sind – etwa eine Produktseite, die ihr Layout anpasst, ein Einstellungs‑Panel, das die richtigen Optionen anzeigt, oder ein Workflow‑Editor, der den nächsten Schritt vorschlägt.

### OpenUI

**GitHub**: [thesysdev/openui](https://github.com/thesysdev/openui) · Docs: [openui.com](https://www.openui.com/)

OpenUI ersetzt JSON durch ein zeilenorientiertes, code‑ähnliches Format („OpenUI Lang“), das für progressives Rendering und Token‑Effizienz konzipiert ist. Die Behauptung lautet, dass es etwa 67 % weniger Tokens verbraucht als äquivalentes JSON für komplexe Layouts.

Der Nachteil ist die Reife des Ökosystems – OpenUI ist neuer und das Tooling ist dünner als bei JSON‑basierten Ansätzen. Wenn jedoch Token‑Kosten ein relevanter Faktor sind und Sie häufig komplexe Layouts generieren, ist die Format‑Effizienz tatsächlich spürbar.

### Tambo

**GitHub**: [tambo-ai/tambo](https://github.com/tambo-ai/tambo)

Tambo konzentriert sich auf zustandsbehaftete Komponentenwahl: Die KI wählt Komponenten aus und kann über client‑seitige Werkzeuge mit ihnen interagieren, wobei der Komponenten‑Zustand über die gesamte Konversation hinweg erhalten bleibt. Geeignet für Anwendungsfälle, bei denen UI‑Elemente über mehrere Turns hinweg bestehen bleiben — etwa ein Filter‑Widget, das der Nutzer anpasst, während die KI weiterhin über die gefilterten Daten nachdenkt.

---

## Die Python‑Ebene

Das Python‑Ökosystem geht AI‑Schnittstellen anders an. Diese Werkzeuge sind für ML‑Modelldemos, Daten‑Apps und interne Tools optimiert — nicht für produktive Verbraucher‑Apps mit agentengesteuerter Layout‑Komposition.

Das ist kein Kritikpunkt. Für die richtigen Anwendungsfälle sind Gradio und Streamlit die einzigen Werkzeuge, die Sie benötigen.

### Gradio

**GitHub**: [gradio-app/gradio](https://github.com/gradio-app/gradio) · PyPI: `gradio`

Gradio's Kernnutzen: Sie schreiben eine Python‑Funktion; Gradio verpackt sie in eine Web‑UI. Die `Interface`‑Klasse besteht aus 3 Zeilen für einen Bildklassifikator. `ChatInterface` benötigt 10 Zeilen für einen Chatbot. `Blocks` gibt Ihnen feinkörnige Layout‑Kontrolle, wenn Sie sie benötigen.

Der „generative UI“-Ansatz in Gradio wird vom Python‑Entwickler definiert, nicht vom Modell. Die Sichtbarkeit und Konfiguration von Komponenten kann dynamisch anhand von Modellausgaben geändert werden, aber der Komponenten‑Katalog bleibt statisch – Sie lassen das Modell nicht das Layout zusammenstellen.

Gradio ist der Standard für HuggingFace Spaces und das ML‑Demo‑Ökosystem. Es hat Millionen monatlicher Downloads und treibt einen großen Teil der KI‑Demo‑Landschaft an.

**Setzen Sie Gradio ein, wenn**: Sie ein Python‑Entwickler sind, der ein ML‑Modell‑Demo, einen Forschung‑Prototyp oder ein internes Tool baut und dabei nicht mit JavaScript arbeiten möchten.

### Streamlit

**GitHub**: [streamlit/streamlit](https://github.com/streamlit/streamlit)

Streamlit verfolgt ein stärker vordefiniertes Modell: Ein Python‑Skript wird bei jeder Interaktion von Anfang bis Ende ausgeführt. Sie rufen `st.chat_message()`, `st.dataframe()`, `st.plotly_chart()` auf. Das Framework übernimmt das Layout.

Das vollständige Skript‑Neuausführungs‑Modell wirkt ineffizient, erweist sich aber überraschend ergonomisch für KI‑Chatbots, die den Gesprächsverlauf akkumulieren – das gesamte Skript wird neu ausgeführt, der Chat‑Verlauf befindet sich im Session‑State, und die Ausgabe ist deterministisch. Streamlit bietet jetzt erstklassige Unterstützung für die meisten großen LLM‑Anbieter und integriert sich nativ in Snowflake Cortex.

**Setzen Sie Streamlit ein, wenn**: Sie eine KI‑gestützte Daten‑App, ein internes Reporting‑Tool oder ein ML‑basiertes Dashboard in Python bauen und den möglichst einfachen Bereitstellungsweg wünschen.

### LangChain und Haystack

Diese sind Backend‑Orchestrierungs‑Frameworks, keine UI‑Frameworks. Sie erscheinen in jeder ehrlichen generativen‑UI‑Stack‑Karte, weil sie typischerweise die Schicht darstellen, in der strukturierte Ausgaben erzeugt werden, bevor sie an ein Frontend gesendet werden.

**LangChain** ([langchain-ai/langchain](https://github.com/langchain-ai/langchain)): `.with_structured_output()` auf jedem LLM liefert Pydantic‑ge‑konstra­niertes JSON. Der `@tool`‑Decorator mit automatischer Schema‑Generierung ist der sauberste Weg, um zu definieren, welche Werkzeuge das Modell aufrufen darf. LangChain leitet strukturierte Ergebnisse an die von Ihnen genutzte Frontend‑Schicht weiter.

**Haystack** ([deepset-ai/haystack](https://github.com/deepset-ai/haystack)): modulare Pipeline‑Architektur mit starkem RAG‑Support. Hayhooks verpackt Haystack‑Pipelines als HTTP‑Endpoints – inklusive MCP‑kompatibler Endpoints. Wenn Ihre generative UI ein Retrieval‑Backbone benötigt, übernimmt Haystack’s Pipeline‑Architektur das sauber.

Keines der beiden Frameworks besitzt die UI‑Schicht. Sie erzeugen die Daten, die Ihr Frontend (Pattern 1, 2 oder 3) rendert.

## Ökosystem‑Dynamik und instabiles Terrain

Dieses Feld entwickelt sich rasant, und mehrere Projekte haben gleichzeitig verwirrende Kommunikation zusammen mit ihrem Code veröffentlicht. Zuletzt verifiziert am 8. Mai 2026; betrachten Sie die Projekt‑Status‑Hinweise hier als zeitlich abgestempelte Lektüre, nicht als endgültiges Urteil.

**Vercel AI SDK RSC** war das Flaggschiff‑Feature für Generative UI, als SDK 3.0 veröffentlicht wurde. Vercel hat die Entwicklung im Oktober 2024 pausiert ([Discussion #3251](https://github.com/vercel/ai/discussions/3251)) und dabei architektonische Beschränkungen von React Server Components ohne kurzfristige Lösung angeführt. Teams, die darauf aufgebaut hatten, waren verständlicherweise frustriert. Es steht noch in der Dokumentation, wird aber nicht mehr empfohlen; stattdessen sollte `useObject` verwendet werden.

**json-render** (Vercel Labs) ist die neue Richtung — eine katalogbasierte, framework‑agnostische Alternative, die die Kopplungsprobleme mit RSC umgeht. Das Projekt befindet sich noch vor Version 1.0, zeigt aber starkes frühes Interesse bei React‑/Web‑Entwicklern. Der wahrscheinliche DX‑Grund: json-render lässt sich sofort in einem normalen React/Next.js‑Projekt einsetzen, während A2UI’s plattformübergreifender Ansatz zusätzlichen Setup‑Aufwand erzeugt. Wie sich das entwickelt, wenn beide Spezifikationen reifen, bleibt unklar. Vercel hat die A2UI‑Kompatibilität in json-render getestet, was eine mögliche Konvergenz andeutet.

**A2UI** (Google) ist ebenfalls vor Version 1.0 (v0.9 zum letzten Check) und weist zwischen Minor‑Versionen breaking changes sowie inkonsistente Google‑Kommunikation zum Fahrplan auf. Es ist die richtige Wahl für plattformübergreifende Reichweite (Web + Flutter + SwiftUI), die json-render nicht abdeckt, und hat bedeutende Enterprise‑Unterstützung. Für reine Web‑Projekte ist das DX momentan jedoch rauer.

**AG-UI** (CopilotKit) ist ebenfalls vor Version 1.0. Die häufigste Verwirrung: Der Name lässt an ein UI‑Framework denken. Das ist er nicht — es ist ein Transport‑Protokoll. AG-UI definiert, wie Ereignisse zwischen Agent‑Backends und Ihrem Frontend fließen; was Sie als Antwort rendern, bleibt Ihre Entscheidung. Dieses mentale Modell ist solide und wird breit übernommen, aber die pre‑1.0‑Spezifikation bedeutet, dass Randfälle noch ausgearbeitet werden.

Der praktische Schluss: **Jeder wichtige Akteur hier befindet sich noch vor Version 1.0**. Planen Sie API‑Änderungen ein. Die Muster — Tool‑zu‑Komponente, Katalog‑Komposition, sandboxed Generation — sind stabil genug, um darauf aufzubauen. Die konkreten Protokoll‑Entscheidungen sind es nicht.

## Component Catalog Design: The Real Engineering Work

Der größte Teil der interessanten Komplexität in Muster 2 liegt nicht im Renderer — sondern im Katalog.

Der Katalog ist eine **Produktentscheidung, kodiert als Schema**. Er beantwortet: Welche UI‑Objekte sind in diesem Domänen‑Kontext sinnvoll? Nicht „Welche React‑Komponenten gibt es?“, sondern „Was muss ein Nutzer in diesem Kontext tatsächlich sehen und bedienen?“

**Der zu feinkörnige Fehlermodus**: Sie stellen `Row`, `Column`, `Text`, `Button`, `Icon` bereit. Jetzt muss das Modell selbst ein Frontend‑Ingenieur sein. Es erzeugt mittelmäßige Layouts, die nicht zu Ihrem Design‑System passen, verpasst leere Zustände, produziert nicht‑barrierefreien Markup und ändert von Antwort zu Antwort seine Vorgehensweise, weil im Katalog nichts das Ergebnis an Ihre visuelle Produkt‑Sprache bindet.

**Der zu grobe Fehlermodus**: Sie stellen `WeatherCard`, `FlightCard`, `HotelCard` bereit. Das Modell kann nicht reagieren, wenn der Nutzer etwas verlangt, das nicht zu einer vorgefertigten Karte passt. Es fällt zurück auf reinen Text.

**Der nützliche Mittelweg**: Domänen‑Level‑Komponenten mit eingeschränkten Slots.

Ein Katalog für eine Reise‑App könnte so aussehen:

```
TripSummary         — itinerary at a glance
FlightOptionList    — selectable flight options with pricing
HotelComparison     — side-by-side hotel cards
TravelerForm        — collect traveler details
PolicyNotice        — regulatory/fare rule callout
BookingConfirmation — final confirmation with action button
```

Ein Katalog für eine Finanz‑App könnte so aussehen:

```
PortfolioSnapshot   — key positions and P&L
TransactionTable    — filterable, paginated transactions
RiskBreakdown       — allocation and volatility metrics
ScenarioComparison  — side-by-side scenario modeling
ApprovalGate        — action requiring human confirmation
```

Der Katalog klingt nach dem Vokabular Ihres Produkts. Er kodiert Ihre UX‑Entscheidungen, Ihre Barrierefrei‑Anforderungen, das Handling von leeren Zuständen und Ihre Muster für gefährliche Aktionen im Komponenten‑Code. Das Modell darf diese Bausteine anordnen. Sie bestimmen weiterhin, wie jedes Teil aussieht und was es dürfen darf.

**Schema‑Design‑Regeln, die Halluzinationen reduzieren**:

1. Halten Sie Enum‑Werte kurz und eindeutig. `"type": "bar_chart"` statt `"type": "data-visualization-bar-type-vertical"`.
2. Machen Sie ungültige Zusammensetzungen unmöglich. Wenn ein `PolicyNotice` nur am Ende eines Layouts erscheinen darf, setzen Sie ihn nicht auf dieselbe Schema‑Ebene wie Elemente, die überall vorkommen können.
3. Verwenden Sie Pflichtfelder großzügig. Ein optionales Feld ist ein Feld, das das Modell weglassen könnte, und Ihr Renderer müsste dann mit `null` umgehen.
4. Testen Sie den Katalog mit realen Prompt‑Beispielen, bevor Sie ihn ausliefern. Speichern Sie die generierten Spezifikationen; prüfen Sie sie auf Schema‑Verstöße, halluzinierte Feldwerte und Zusammensetzungen, die technisch gültig, aber semantisch falsch sind.

## Häufige Fallen

**Falle: gültiges JSON als sicheres Verhalten zu betrachten.** Die Schema‑Validierung bestätigt nur die Struktur. Sie sagt nichts darüber aus, ob die an einen Button gebundene Aktion zu seiner Beschriftung passt, ob ein Summenwert zu den zugrunde liegenden Daten passt oder ob eine UI‑Komponente etwas tut, das der Nutzer nicht erwartet. Generierte UI‑Spezifikationen benötigen eine semantische Prüfung, nicht nur Schema‑Validierung. Mindestens sollten destruktive Aktionen eine Bestätigungs‑Komponente erfordern, und die Beschriftungen dieser Komponenten sollten gegen die Aktionen, die sie auslösen, getestet werden.

**Falle: Design‑Primitiven statt Produkt‑Primitiven aussetzen.** Wenn das Modell entscheiden muss, ob 16 px oder 20 px Abstand verwendet werden, haben Sie die falsche Abstraktionsebene gewählt. Domänen‑Komponenten sollten den Produktgeschmack kodieren. Das Modell sollte Verhalten zusammensetzen, nicht Präsentationsdetails verwalten.

**Falle: Generative UI einsetzen, wo statische UI ausreicht.** Wenn die Struktur dessen, was Sie zeigen wollen, zur Entwicklungszeit bekannt ist — und das ist meistens der Fall — ist Muster 1 mit vorgefertigten Komponenten schneller, sicherer und konsistenter. Generative UI rechtfertigt ihre Komplexität nur, wenn die Struktur tatsächlich aufgrund von Daten‑ oder Aufgabenkontext variiert.

**Falle: Barrierefreiheit überspringen.** LLMs halluzinieren WCAG‑Verstöße. Sie geben `role="region"` bei interaktiven Elementen aus, erzeugen Formulare ohne Beschriftungen und produzieren Kontrastwerte, die WCAG AA nicht bestehen. Ihre Komponentenbibliothek mag vollständig barrierefrei sein; KI‑generierte Zusammensetzungen dieser Komponenten sind nicht automatisch barrierefrei. Testen Sie den gesamten Render‑Pfad, nicht nur die einzelnen Komponenten isoliert.

**Falle: Protokoll und Framework verwechseln.** AG‑UI ist kein Frontend‑Framework. A2UI ist keine React‑Bibliothek. Sie sind Wire‑Formate und Ereignis‑Protokolle. Sie benötigen weiterhin ein Frontend‑Framework, um sie zu implementieren. CopilotKit implementiert AG‑UI und A2UI. `json-render` implementiert das A2UI/Open-JSON-UI‑Katalog‑Muster. Das sind unterschiedliche Schichten.

## Empfehlungen nach Anwendungsfall

**Einen Copilot zu einer bestehenden SaaS‑App hinzufügen**: Beginnen Sie mit Muster 1 (Tool‑zu‑Komponente). Nutzen Sie das Vercel AI SDK `useChat` oder CopilotKit. Ordnen Sie Ihre Top‑5–10 Agent‑Aktionen vorgefertigten Komponenten zu. Ausliefern, messen und dann den Katalog nur erweitern, wenn Nutzer nachweislich komplexere Zusammensetzungen benötigen.

**Dashboard‑Generierung aus natürlicher Sprache**: Verwenden Sie Muster 2 mit `json-render` oder einem eigenen A2UI‑Katalog. Definieren Sie einen Katalog von 8–15 Komponententypen, die Ihre Diagramm‑Typen, Metrik‑Karten und Tabellen‑Varianten abdecken. Geben Sie das Schema an das Modell weiter; lassen Sie es das Layout komponieren. Implementieren Sie eine Validierung, die unbekannte Typen abfängt, bevor sie den Renderer erreichen.

**Multi‑Agent‑Frontend**: Nutzen Sie CopilotKit mit AG‑UI. Der Ereignis‑Stream verarbeitet Echtzeit‑Streaming über Agent‑Backends; geteilter Zustand steuert die Übergabe zwischen Agenten; das HITL‑Muster sorgt für Genehmigungsgates.

**Entwicklung innerhalb von ChatGPT oder einem anderen MCP‑Host**: Verwenden Sie MCP‑Apps. Definieren Sie Ihr Tool als Datentool, das abruft und schlussfolgert, und ein separates Render‑Tool, das ein Widget anfordert. Halten Sie Geschäftslogik aus der Widget‑Vorlage heraus.

**ML‑Modell‑Demos und Daten‑Apps (Python‑Team)**: Gradio für Demos und HuggingFace Spaces. Streamlit für Daten‑Apps mit komplexerer Interaktion. Keines von beidem erfordert JavaScript.

**Visuelle Artefakte, Simulationen, Diagramme**: Verwenden Sie Muster 3 (OpenGenerativeUI oder Äquivalent). Etablieren Sie eine strenge CSP für iframes. Behandeln Sie die Ausgabe aus Sicherheitsperspektive wie unzuverlässigen Benutzergenerierten Inhalt.

---

Die Frameworks reifen schnell. Die Protokoll‑Konvergenz (AG‑UI für Streaming, A2UI/Open‑JSON‑UI für Katalog‑Spezifikationen) ist noch im Gange, aber die Form wird klar genug, um darauf aufzubauen.

Die technischen Herausforderungen, die gerade am wichtigsten sind, betreffen nicht die Auswahl des Frameworks. Sie betreffen das Katalog‑Design – zu entscheiden, was das Modell sagen darf, was mehr Produktklarheit erfordert als technisches Können. Sie betreffen die semantische Validierung – zu prüfen, dass die generierte UI das tut, was sie behauptet, und nicht nur das Schema besteht. Und sie betreffen die Barrierefrei‑Lücke – Kataloge zu bauen, bei denen jede Komponente und jede Zusammensetzung von Komponenten die Barrierefrei‑Anforderungen erfüllt, die Sie an handgeschriebene UI stellen würden.

Das Modell tut, was Sie ihm im Rahmen der Grammatik, die Sie bereitstellen, vorgeben. Gestalten Sie die Grammatik bewusst.
