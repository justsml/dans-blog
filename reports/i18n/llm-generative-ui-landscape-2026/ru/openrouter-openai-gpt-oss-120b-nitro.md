«Генеративный UI» означает как минимум пять разных вещей, в зависимости от того, кто это говорит.

- Чат‑интерфейсы, которые встраивают карточки продукта из вызовов инструментов модели  
- Спецификации JSON во время выполнения, которые фронтенд рендерит как дерево компонентов  
- Песочничные iframe, возвращаемые инструментами MCP в хост‑приложения (от заказа билетов, бронирования отелей до рендеринга карт, виджетов оформления покупки)  
- Протоколы событий, которые передают состояние агента в фронтенд  
- v0, Lovable и Bolt: AI‑инструменты, которые пишут React на этапе проектирования  

Это связанные концепции, но они находятся на разных уровнях стека, имеют разные профили риска, разные затраты на реализацию и разные подходящие сценарии использования. Смешивание их превращает каждое обсуждение архитектуры в беспорядок.

Это карта, которую я использую, решая, на каком уровне стека работать.

## Что не является Generative UI

Перед тем как определить, что это такое, отложим в сторону три вещи:

**Генерация кода на этапе проектирования** — v0, Lovable, Bolt, Cursor, компоновка React‑компонентов. Эти инструменты генерируют код, который разработчики просматривают и коммитят. ИИ работает во время разработки. Всё, что попадает в продакшн, статично с точки зрения пользователя. Это отличная категория инструментов. Это не то, что означает «генеративный UI во время выполнения».

**AI‑ассистент автозаполнения форм** — модель заполняет значения полей из контекста. Структура интерфейса остаётся фиксированной; меняется только содержимое. Это полезный паттерн. Это не генеративный UI.

**AI пишет сырой HTML на страницу** — модель выводит строки `<div>` и `<button>`, которые затем вставляются через `innerHTML` или `dangerouslySetInnerHTML`. Это *runtime*‑генеративный UI в самом буквальном техническом смысле. Это также самая опасная версия, и именно ради её избежания созданы все зрелые фреймворки в этой области. Сырой AI‑сгенерированный разметочный код несёт риск XSS, недоступные атрибуты, непоследовательный стиль и галлюцинированную структуру. Остальная часть статьи посвящена тому, как сделать лучше, чем это.

## Рабочее определение

Генеративный UI во время выполнения означает: **модель определяет, какой компонент интерфейса или композицию компонентов увидит пользователь, исходя из состояния диалога или задачи**.

Не слова. Интерфейс.

Самый простой пример: ваш помощник по бронированию авиабилетов вызывает инструмент `search_flights`. Вместо того чтобы возвращать простой текст («Вот три варианта…»), он отображает компонент `<FlightResultsCard>` с выбираемыми рейсами, переключателями класса места и кнопкой «Book». Модель решила, что здесь уместна структурированная карточка. Разработчик определил, как выглядит эта карточка и что делает кнопка «Book».

Более сложный пример: агент финансового анализа получает запрос о портфеле и решает сформировать ответ с `MetricGroup`, отображающим ключевые цифры, диаграммой `RiskBreakdown`, таблицей `ScenarioComparison` и уведомлением `PolicyNotice`. Модель собрала такой макет из каталога предварительно одобренных компонентов. Разработчик определил каждый компонент. Модель выбрала, какие из них использовать и какие данные поместить в них.

Оба случая — это генеративный UI. Они различаются степенью свободы композиции, предоставляемой модели, что определяет как потенциальную насыщенность выводов, так и сложность возможных ошибок.

## Три паттерна

Весь спектр сводится к трем паттернам, каждый со своей грамматикой вывода.

![Диаграмма спектра, показывающая три паттерна: вызовы инструментов слева (самый безопасный), каталог компонентов в центре и открытая генерация справа (наиболее выразительный).](../output-grammar-spectrum.svg)

_Каждое решение в генеративном UI — это точка на этом спектре. Начинайте слева._

### Паттерн 1: Рендеринг «инструмент‑к‑компоненту»

Модель вызывает именованный инструмент. В приложении существует отображение от имён инструментов к компонентам. Вызов инструмента инициирует рендер компонента.

```tsx
// The model calls: { name: "show_flight_results", args: { flights: [...] } }

useCopilotAction({
  name: "show_flight_results",
  render: ({ args }) => <FlightResultsCard flights={args.flights} />,
});
```

Это самый безопасный паттерн, потому что разметка никогда не приходит от модели. Модель решает, *когда* показывать компонент и *какими данными* его заполнять. Разработчики по‑прежнему владеют кодом компонента, визуальным дизайном, реализацией доступности и всеми граничными случаями в логике рендера.

`useChat` из Vercel AI SDK с обработчиками `tool` реализует именно это. Рендеринг инструментов в assistant‑ui делает то же самое. «Static Generative UI» в CopilotKit соответствует этому паттерну. Большинство производственных UI‑копилотов, которые работают надёжно, используют именно его.

**Подходит, когда**: набор элементов, которые вы потенциально хотите отобразить, известен на этапе разработки. Подтверждения бронирования, результаты поиска, сводки аккаунтов, виджеты одобрения. Если вы можете перечислить сценарии, этот паттерн их покрывает.

### Pattern 2: Component catalog composition

Модель выдаёт типизированное дерево JSON, в котором ссылки на компоненты берутся из каталога, определённого разработчиком. На фронтенде есть рендерер, который проходит по дереву и создаёт каждый компонент.

```json
[
  { "type": "metric_group", "metrics": [
    { "label": "MRR", "value": "$82,400", "delta": "+12%" },
    { "label": "Churn", "value": "2.1%", "delta": "-0.4%" }
  ]},
  { "type": "line_chart", "title": "30-day growth", "data_ref": "mrr_series" },
  { "type": "insight_callout", "text": "Expansion revenue driving the delta — avg seat count up 18%." }
]
```

Модель скомпоновала этот макет: `MetricGroup`, `LineChart`, `InsightCallout`. Но вы задали, что каждый тип компонента означает, какие свойства он принимает и как рендерится. Если модель попытается выдать `{ "type": "custom_untested_thing" }`, ваша схема валидации поймает это, и рендерер проигнорирует или отклонит такой элемент.

Это паттерн, лежащий в основе `json-render`, `A2UI`, `Hashbrown`, `OpenUI` и `Tambo`. Ключевая инженерная работа — **проектирование каталога**: определение, какие типы компонентов существуют, как выглядят их схемы и что модели разрешено или запрещено компонировать.

**Когда уместно**: структура отображаемой информации меняется законно в зависимости от данных или запроса пользователя. Дашборды, адаптирующиеся к заметным в цифрах аспектам. Отчёты, показывающие разные секции в зависимости от контекста. Панели рабочего процесса, меняющиеся в зависимости от текущего шага агента.

### Pattern 3: Open-ended generation

Модель генерирует HTML, SVG, Canvas или WebGL, которые затем отображаются внутри изолированного iframe с жёсткой политикой Content Security Policy.

Это подходит для случаев, когда фиксированный каталог компонентов не справится: визуализации алгоритмов, архитектурные схемы, разово создаваемые графики, генеративное искусство, обучающие симуляции. Здесь граница iframe берёт на себя задачу обеспечения безопасности; если её убрать, вы вернётесь к проблеме прямой инъекции HTML, описанной в начале статьи.

`CopilotKit/OpenGenerativeUI` — наиболее актуальная референс‑реализация этого паттерна. Песочница удаляет скрипты, ограничивает передачу сообщений и изолирует сгенерированный артефакт от привилегированного состояния вашего приложения.

**Подходит, когда**: действительно требуется произвольный визуальный вывод — одноразовые пояснительные диаграммы, динамические симуляции, креативные артефакты. Не используйте это для транзакционного UI. Подтверждение покупки не нуждается в изолированном iframe.

### За пределами трёх паттернов: LLM управляют пикселями напрямую

Появляется четвёртое направление, которое не укладывается в существующие паттерны: LLM‑ы создают **иммерсивные, похожие на игру опыты**, контролируя визуальный вывод более напрямую, чем изолированный iframe.

Каноническое различие в генеративном UI — **iframe HTML vs. JSON‑каталог**:

- **Iframe HTML** — модель пишет HTML, SVG, Canvas или WebGL, которые отрисовываются в изолированном песочном окружении. Максимальная выразительная свобода; безопасность полностью зависит от границы iframe. Примеры: Anthropic Artifacts, OpenGenerativeUI.
- **JSON‑каталог** — модель выдаёт структурированный payload, ограниченный каталогом компонентов, определённым разработчиком; ваш рендерер создаёт доверенные, заранее построенные компоненты из этой спецификации. Модель решает, *что* показывать; вы решаете, *как* это отрисовывать. Примеры: json-render, A2UI.

Помимо этого, очень свежие демонстрации намекают на третий режим, при котором модель не выбирает компоненты и не пишет песочный HTML — она управляет канвасом более напрямую. Проекты вроде [Tencent's HunyuanWorld](https://arxiv.org/abs/2502.01999), генерирующего исследуемые 3D‑окружения из одного изображения, и игровых архитектур, где LLM генерируют карты, NPC и квесты в режиме реального времени вместо обращения к каталогу компонентов, указывают на будущее, где модель выступает скорее как режиссёр игры, а не как рендерер форм. Инференс LLM в браузере через WebGPU ([WebLLM](https://mlc.ai/web-llm/)) продвигает эту же границу локально.

Эта область действительно захватывающая и действительно ранняя. Пока нет стабильных фреймворков для создания продакшн‑продуктов. Я расскажу об этом подходе в отдельной статье, когда ситуация изменится.

## Протоколы: AG-UI и A2UI

AG-UI и A2UI — два основных стандарта на уровне протокола. Они решают разные задачи и не являются конкурентами.

### AG-UI

**GitHub**: [ag-ui-protocol/ag-ui](https://github.com/ag-ui-protocol/ag-ui)

AG-UI — это событийный протокол для взаимодействия AI‑агентов и фронтенд‑приложений. Он описывает около 16 типов событий: `TEXT_MESSAGE_START`, `TEXT_MESSAGE_CONTENT`, `TOOL_CALL_START`, `TOOL_CALL_END`, `STATE_SNAPSHOT`, `STATE_DELTA` и др. Транспорт выбираете вы — SSE, WebSockets, webhooks — все работают. Формат намеренно свободный, чтобы обеспечить широкое принятие.

AG-UI не определяет, как выглядит ваш UI. Он определяет, как агент общается *с* вашим фронтендом. Считайте его слоем проводного протокола, который позволяет вашему React‑приложению подписаться на агент LangGraph так же, как на агент CrewAI, без изменения фронтенд‑кода.

CopilotKit создал AG‑UI на основе своей работы с LangGraph и CrewAI. Его приняли LangChain, Mastra, PydanticAI и другие. Microsoft опубликовала руководство по интеграции AG‑UI. Если вы строите фронтенд для нескольких агентов и хотите отделить бэкенд‑фреймворки от кода UI, AG‑UI — это решение.

**Одна деталь, которая часто сбивает с толку**: AG‑UI — это не UI‑фреймворк. Он не указывает, что именно рендерить. Он сообщает, *что* агент что‑то сказал, вызвал инструмент или обновил совместное состояние. Что именно отображать в ответ — по‑прежнему ваше решение.

### A2UI

**GitHub**: [google/A2UI](https://github.com/google/A2UI) · Spec: [a2ui.org](https://a2ui.org/)

A2UI — это декларативная спецификация Google для того, что агенты отправляют, когда хотят отобразить UI. Если AG‑UI отвечает «как агент общается?», то A2UI отвечает «в каком формате агент описывает расположение компонентов?».

A2UI использует плоский формат JSONL: один дескриптор компонента на строку, каждый с ID, типом и данными. Плоский формат намерен. Вложенные деревья требуют, чтобы модель знала полную структуру до начала потоковой передачи. Плоский список позволяет модели выдавать каждый компонент по мере «размышления», что даёт возможность вашему фронтенду начать рендерить первую карточку метрики, пока модель ещё решает, добавить ли график.

```jsonl
{"id":"h1","type":"kpi_card","title":"MRR","value":"$82,400","delta":"+12%"}
{"id":"h2","type":"kpi_card","title":"Churn","value":"2.1%","delta":"-0.4%"}
{"id":"c1","type":"line_chart","title":"30-day MRR","data_ref":"mrr_series"}
{"id":"t1","type":"data_table","cols":["Month","MRR","Net New"],"data_ref":"monthly"}
```

A2UI ориентирован на безопасность: спецификация — это формат данных, а не исполняемый код. Каталог компонентов задаётся разработчиком; агент может ссылаться только на типы из этого каталога. Рендерер A2UI, получивший неизвестное имя типа, просто игнорирует его.

Формат «Open-JSON-UI» от CopilotKit совместим с A2UI. Если вы выбираете формат спецификации для каталога компонентов сегодня, A2UI — это вариант с наибольшей кроссплатформенной поддержкой.

**Замечание о стабильности**: A2UI находится в предрелизной версии — v0.9 на момент последней проверки 8 мая 2026 года — и между минорными версиями уже происходили несовместимые изменения спецификации. Информационные сообщения Google о дорожной карте были редки, и некоторые рендереры (Lit, Flutter) отставали от обновлений спецификации. Планируйте время на возможный дрейф спецификации, если строите решение сейчас. Для чисто веб‑ориентированных сценариев json-render пока предлагает более полную экосистему инструментов. Долгосрочное преимущество A2UI — широкий кроссплатформенный охват (web, Flutter, SwiftUI, Android), которого нет у json-render.

### MCP Apps

**GitHub**: [modelcontextprotocol](https://github.com/modelcontextprotocol) · Related: [mcp-ui](https://github.com/MCP-UI-Org/mcp-ui)

MCP изначально задумывался как протокол для соединения LLM с инструментами и данными. Расширение Apps позволяет инструментам MCP возвращать не только данные, но и интерактивные UI‑артефакты: React‑компоненты, формы, дашборды, карты.

Модель безопасности жёсткая, и в этом её смысл: всё рендерится в изолированном iframe с ограниченными правами, шаблоны заранее объявлены, чтобы хост‑приложение могло их проверить, а вся коммуникация осуществляется через аудируемый JSON‑RPC. Это правильный подход для поставщиков инструментов — сервер MCP Shopify может вернуть виджет оформления заказа; сервис карт может вернуть встраиваемую карту. Хост‑приложение не владеет и не доверяет коду этого виджета.

MCP Apps подходит, когда UI *принадлежит поставщику инструмента*, а не вашему приложению. Если интерфейс находится в домене вашего приложения, используйте Паттерн 1 или 2.

## JavaScript/TypeScript‑фреймворки

### CopilotKit

**GitHub**: [CopilotKit/CopilotKit](https://github.com/CopilotKit/CopilotKit) · Примеры: [CopilotKit/generative-ui](https://github.com/CopilotKit/generative-ui)

CopilotKit — это самый полнофункциональный фреймворк для приложений с агентно‑нативным фронтендом. Он охватывает весь жизненный цикл: подключение к бэкендам‑агентам через AG‑UI, управление двунаправленным состоянием диалога, рендеринг генеративных UI‑компонентов и предоставление общей инфраструктуры состояния, позволяющей агентам и пользователям изменять одни и те же данные.

Трёхшаблонная модель удобно сопоставляется с API CopilotKit:
- `useCopilotAction` с колбэком `render` → Шаблон 1
- рендеринг A2UI/Open-JSON-UI → Шаблон 2
- артефакты `OpenGenerativeUI` в песочнице → Шаблон 3

Важная, но часто недооценённая возможность CopilotKit — **общий статус и человек‑в‑цикле**: агент может читать и записывать состояние приложения, пользователь тоже может его читать и записывать, а изменения передаются в обе стороны. Именно это делает UI‑интерфейсы в стиле Copilot ощущаемыми как реальное совместное взаимодействие, а не просто чат‑окно, приклеенное к продукту.

### Vercel AI SDK

**GitHub**: [vercel/ai](https://github.com/vercel/ai) · Docs: [ai-sdk.dev](https://ai-sdk.dev/)

Vercel AI SDK является де‑факто базовым TypeScript‑фреймворком для AI‑приложений. Для генеративного UI в частности:

**`useObject`** передаёт структурированный JSON‑объект с сервера по мере его генерации. Вы задаёте схему Zod; SDK парсит частичный JSON и инициирует повторные рендеры по мере появления полей. Это самый гладкий путь к Паттерну 2 в приложении Next.js.

```tsx
const { object: dashboard } = useObject({
  api: "/api/generate-dashboard",
  schema: z.object({
    title: z.string(),
    metrics: z.array(z.object({ label: z.string(), value: z.number() })),
    insights: z.array(z.string()),
  }),
});
```

**`useChat` с обработчиками инструментов** → Паттерн 1. Модель вызывает инструменты; вы сопоставляете имена инструментов с компонентами.

**AI Elements** ([elements.ai-sdk.dev](https://elements.ai-sdk.dev/)) предоставляет готовые UI‑примитивы для использования вместе с SDK.

**Замечание о запутанном развитии событий**: В октябре 2024 года Vercel объявил в [GitHub Discussion #3251](https://github.com/vercel/ai/discussions/3251), что потоковый паттерн AI SDK RSC — React Server Components, продвигаемый как флагманская функция «Generative UI» в SDK 3.0 — приостановлен на неопределённый срок из‑за «нескольких давних ограничений», для которых нет хороших краткосрочных решений. Команды, построившие продуктовые стратегии вокруг потокового RSC, оказались врасплох. API `generateObject`/`streamObject` также позже были помечены устаревшими в SDK 6.0. Рекомендуемая миграция от AI SDK RSC — использовать паттерн `useObject`, описанный выше, либо json-render для генерации на основе каталога.

### assistant-ui

**GitHub**: [assistant-ui/assistant-ui](https://github.com/assistant-ui/assistant-ui)

assistant-ui — набор компонуемых примитивов React для создания чат‑интерфейсов промышленного уровня. Это правильный выбор, когда нужен отшлифованный UX чата — пузырьки сообщений, потоковые токены, действия копировать/редактировать/перегенерировать, состояния «размышляет» — и при этом хочется использовать собственный бекенд и собственный рендеринг инструментов.

Он хорошо сочетается с любым бекендом (OpenAI, Anthropic, локальные модели, кастомные эндпоинты) и обрабатывает рендеринг вызовов инструментов через привычную модель slot/render‑prop.

### json-render

**GitHub**: [vercel-labs/json-render](https://github.com/vercel-labs/json-render) · Docs: [json-render.dev](https://json-render.dev/)

json-render реализует Паттерн 2 с предвзятой, «всё-в-одном» стратегией. Вы получаете готовый каталог компонентов (компоненты shadcn/ui со схемами Zod), рендерер и плотный цикл генерации, где модель ограничена каталогом схемой.

Отличительные особенности:
- **Мульти‑таргет рендеринг**: один и тот же JSON‑спецификатор может отрисоваться как React‑веб‑приложение, React Native‑мобильное приложение, PDF, HTML‑письмо или видео Remotion. Это действительно полезно для отчётов.
- **Прогрессивный рендеринг**: компоненты появляются по мере того, как модель их стримит, а не после получения полной спецификации.
- **Жёсткие ограничения схемы**: каталог спроектирован так, чтобы модель не могла «галлюцинировать» валидные, но неизвестные типы компонентов.

Если вы создаёте панель мониторинга или функцию генерации отчётов и хотите обойти работу по построению собственного каталога, json-render — самый быстрый путь для веб‑приложений.

**О моментуме**: json-render был запущен Vercel Labs в начале 2026 года и, судя по всему, быстро привлёк внимание веб‑разработчиков, потому что сразу же полезен в типовых проектах на React/Next.js. При этом json-render всё ещё находится в версии pre‑1.0, а взаимосвязь между json-render и A2UI ещё уточняется — Vercel экспериментировал с выводом, совместимым с A2UI, так что сходимость возможна. Для кроссплатформенных решений (нативный мобильный, несколько фреймворков) A2UI остаётся более надёжной долгосрочной ставкой.

**GitHub**: [liveloveapp/hashbrown](https://github.com/liveloveapp/hashbrown)

Hashbrown использует особый подход: вместо создания отдельного слоя интерфейса ИИ он встраивает выбор ИИ‑компонентов непосредственно в ваше существующее приложение на React или Angular. Вы раскрываете компоненты приложения перед LLM; LLM выбирает, какие из них отобразить, и может вызывать клиентские инструменты.

Это подходящий инструмент, когда нужно добавить интеллект в продуктовые поверхности, не являющиеся «чатом» — например, страницу продукта, адаптирующую макет, панель настроек, показывающую релевантные опции, или редактор рабочего процесса, предлагающий следующий шаг.

### OpenUI

**GitHub**: [thesysdev/openui](https://github.com/thesysdev/openui) · Docs: [openui.com](https://www.openui.com/)

OpenUI заменяет JSON на построчный кодоподобный формат («OpenUI Lang»), разработанный для прогрессивного рендеринга и экономии токенов. По заявлению, он использует примерно на 67 % меньше токенов, чем эквивалентный JSON для сложных макетов.

Торговый компромисс — зрелость экосистемы: OpenUI новее, а набор инструментов менее развит, чем у подходов, основанных на JSON. Однако если стоимость токенов является значимым ограничением и вы генерируете сложные макеты с высокой частотой, эффективность формата действительно ощутима.

### Tambo

**GitHub**: [tambo-ai/tambo](https://github.com/tambo-ai/tambo)

Tambo ориентирован на выбор состоянийных компонентов: ИИ выбирает компоненты и может взаимодействовать с ними через клиентские инструменты, сохраняя состояние компонента в течение всего диалога. Подходит для сценариев, где UI‑элементы сохраняются между ходами — например, фильтрующий компонент, который пользователь настраивает, пока ИИ продолжает анализировать отфильтрованные данные.

## Слой Python

Экосистема Python подходит к AI‑интерфейсам иначе. Эти инструменты оптимизированы для демонстраций ML‑моделей, дата‑приложений и внутренних утилит — а не для производственных потребительских приложений с композицией макета, управляемой агентом.

Это не критика. Для подходящих сценариев Gradio и Streamlit — это всё, что вам понадобится.

### Gradio

**GitHub**: [gradio-app/gradio](https://github.com/gradio-app/gradio) · PyPI: `gradio`

Основная ценность Gradio: вы пишете функцию на Python, а Gradio оборачивает её в веб‑интерфейс. Класс `Interface` занимает 3 строки для классификатора изображений. `ChatInterface` — 10 строк для чат‑бота. `Blocks` даёт тонко‑настроечный контроль над макетом, когда это необходимо.

«Генеративный UI» в Gradio определяется Python‑разработчиком, а не моделью. Видимость и конфигурацию компонентов можно менять динамически в зависимости от выводов модели, но каталог компонентов статичен — вы не просите модель собирать макеты.

Gradio — де‑факто стандарт для HuggingFace Spaces и экосистемы демонстраций машинного обучения. У него миллионы загрузок в месяц, и он обслуживает большую часть ландшафта AI‑демо.

**Выбирайте Gradio, когда**: вы — разработчик Python, создаёте демонстрацию ML‑модели, исследовательский прототип или внутренний инструмент, и не хотите иметь дело с JavaScript.

### Streamlit

**GitHub**: [streamlit/streamlit](https://github.com/streamlit/streamlit)

Модель Streamlit более предопределённая: Python‑скрипт исполняется полностью при каждом взаимодействии. Вы вызываете `st.chat_message()`, `st.dataframe()`, `st.plotly_chart()`. Фреймворк управляет расположением элементов.

Модель полного пере‑запуска скрипта выглядит неэффективной, но оказывается удивительно удобной для AI‑чат‑ботов, которые накапливают историю диалога — скрипт полностью перезапускается, история чата хранится в session state, а вывод детерминирован. Сейчас Streamlit предоставляет официальную поддержку большинства крупных провайдеров LLM и интегрируется напрямую с Snowflake Cortex.

**Reach for Streamlit when**: вы создаёте AI‑управляемое приложение для работы с данными, внутренний инструмент отчётности или дашборд с поддержкой ML на Python и хотите максимально простую схему развертывания.

### LangChain and Haystack

Это фреймворки оркестрации бэкенда, а не UI‑фреймворки. Они появляются в любой честной карте стека генеративного UI, потому что обычно именно на этом уровне формируются структурированные выводы, которые затем передаются фронтенду.

**LangChain** ([langchain-ai/langchain](https://github.com/langchain-ai/langchain)): метод `.with_structured_output()` для любой LLM обеспечивает генерацию JSON, ограниченного схемой Pydantic. Декоратор `@tool` с автоматическим построением схемы — самый чистый способ задать, какие инструменты модель может вызывать. LangChain передаёт структурированные результаты дальше, к любой используемой вами фронтенд‑слой.

**Haystack** ([deepset-ai/haystack](https://github.com/deepset-ai/haystack)): модульная архитектура конвейера с сильной поддержкой RAG. Hayhooks оборачивает конвейеры Haystack в HTTP‑эндпоинты — включая совместимые с MCP эндпоинты. Если вашему генеративному UI нужен слой извлечения, архитектура конвейера Haystack справляется с этим чисто.

Ни один из фреймворков не владеет UI‑уровнем. Они генерируют данные, которые ваш фронтенд (паттерн 1, 2 или 3) отображает.

## Скорость экосистемы и нестабильная почва

Это пространство развиваетсястремительно, и несколько проектов выпустили запутанные сообщения вместе со своим кодом. Последняя проверка — 8 мая 2026; рассматривайте заметки о статусе проекта здесь как помечённый датой обзор, а не окончательное заключение.

**Vercel AI SDK RSC** был флагманской функцией Generative UI при запуске SDK 3.0. Vercel приостановил её разработку в октябре 2024 ([Discussion #3251](https://github.com/vercel/ai/discussions/3251)), сославшись на архитектурные ограничения React Server Components, для которых не было близкосрочного решения. Команды, построившие решения на этой базе, естественно разочаровались. Функция всё ещё присутствует в документации, но не рекомендуется к использованию; вместо неё предпочтительнее `useObject`.

**json-render** (Vercel Labs) — новое направление — альтернатива, основанная на каталоге и независимая от фреймворка, которая избегает проблем сопряжения с RSC. Проект находится в предрелизе (< 1.0) и, судя по всему, вызывает сильный интерес у разработчиков React/web. Вероятная причина удобства разработки: json-render сразу работает в стандартном проекте React/Next.js, тогда как кроссплатформенный охват A2UI добавляет накладные расходы на настройку. Как будет развиваться ситуация по мере созревания обеих спецификаций, пока действительно неясно. Vercel проверял совместимость A2UI в json-render, что намекает на возможность конвергенции.

**A2UI** (Google) находится в стадии pre‑1.0 (v0.9 на момент последней проверки) — между минорными версиями происходят несовместимые изменения, а коммуникация Google по дорожной карте продукта остаётся непоследовательной. Это правильный выбор, если нужен кроссплатформенный охват (web + Flutter + SwiftUI), который json‑render не покрывает, и у проекта есть значительная поддержка со стороны крупных предприятий. Для чисто веб‑проекта на данный момент DX (developer experience) менее удобен.

**AG‑UI** (CopilotKit) также находится в стадии pre‑1.0. Наиболее распространённая путаница: название намекает на UI‑фреймворк. На самом деле это транспортный протокол. AG‑UI определяет, как события передаются между бек‑эндами агентов и вашим фронтендом; что именно вы будете рендерить в ответ — по‑прежнему остаётся вашим решением. Такая ментальная модель надёжна и получила широкое признание, но спецификация pre‑1.0 означает, что граничные случаи ещё находятся в процессе проработки.

Практический вывод: **каждый из основных игроков находится в pre‑1.0**. Планируйте изменения API. Паттерны — tool‑to‑component, catalog composition, sandboxed generation — достаточно стабилизированы, чтобы на них строить решения. Конкретный выбор протокола пока остаётся открытым.

## Дизайн каталога компонентов: реальная инженерная работа

Большая часть интересующей сложности в Паттерне 2 находится не в рендерере — а в каталоге.

Каталог — это **продуктовое решение, зафиксированное в схеме**. Он отвечает на вопрос: какие UI‑объекты имеют смысл в данном домене? Не «какие React‑компоненты существуют?», а «что пользователю в этом контексте действительно нужно видеть и с чем взаимодействовать?».

**Слишком гранулированный режим отказа**: вы раскрываете `Row`, `Column`, `Text`, `Button`, `Icon`. Теперь модель должна выступать в роли фронтенд‑инженера. Она будет генерировать посредственный макет, не соответствующий вашей системе дизайна, пропускать состояния пустоты, создавать недоступный разметочный код и менять подход от ответа к ответу, потому что в каталоге нет ограничений, принуждающих вывод соответствовать визуальному языку продукта.

**Слишком грубый режим отказа**: вы раскрываете `WeatherCard`, `FlightCard`, `HotelCard`. Модель не сможет адаптироваться, когда пользователь запрашивает что‑то, что не укладывается в заранее подготовленную карточку. Она откатывается к простому тексту.

**Полезный средний вариант**: компоненты уровня домена с ограниченными слотами.

Каталог travel‑приложения может выглядеть так:

```
TripSummary         — itinerary at a glance
FlightOptionList    — selectable flight options with pricing
HotelComparison     — side-by-side hotel cards
TravelerForm        — collect traveler details
PolicyNotice        — regulatory/fare rule callout
BookingConfirmation — final confirmation with action button
```

Каталог финансового приложения может выглядеть так:

```
PortfolioSnapshot   — key positions and P&L
TransactionTable    — filterable, paginated transactions
RiskBreakdown       — allocation and volatility metrics
ScenarioComparison  — side-by-side scenario modeling
ApprovalGate        — action requiring human confirmation
```

Каталог отражает словарь вашего продукта. В нём зафиксированы UX‑решения, требования к доступности, обработка пустых состояний и шаблоны опасных действий в виде компонентов. Модель получает возможность комбинировать эти элементы. Вы по‑прежнему определяете, как каждый элемент выглядит и что ему разрешено делать.

**Правила проектирования схем, снижающие галлюцинации**:

1. Делайте значения перечислений короткими и однозначными. `"type": "bar_chart"` вместо `"type": "data-visualization-bar-type-vertical"`.
2. Делайте невозможной недопустимую композицию. Если `PolicyNotice` может появляться только в конце макета, не размещайте его на том же уровне схемы, где могут находиться элементы, допускающие произвольное расположение.
3. Щедро используйте обязательные поля. Опциональное поле — это поле, которое модель может опустить, и ваш рендерер придётся обрабатывать как `null`.
4. Тестируйте каталог на реальных запросах перед выпуском. Сохраняйте сгенерированные спецификации; проверяйте их на нарушения схемы, вымышленные значения полей и композиции, которые технически корректны, но семантически ошибочны.

## Общие ловушки

**Ловушка: считать валидный JSON безопасным поведением.** Проверка схемы подтверждает только структуру. Она ничего не говорит о том, соответствует ли действие, привязанное к кнопке, её подписи, совпадает ли итог с данными, из которых он вычислен, или делает ли UI‑компонент то, чего пользователь не ожидал. Сгенерированные спецификации UI требуют семантической проверки, а не только валидации схемы. Как минимум, деструктивные действия должны требовать компонента подтверждения, а подписи этих компонентов следует тестировать на соответствие вызываемым действиям.

**Ловушка: раскрытие дизайнерских примитивов вместо продуктовых примитивов.** Если модели приходится решать, использовать отступ 16 px или 20 px, вы задали ей неправильный уровень абстракции. Компоненты домена должны инкапсулировать вкусовые решения продукта. Модель должна собирать поведение, а не управлять деталями презентации.

**Ловушка: применение генеративного UI там, где достаточно статического UI.** Если структура того, что нужно отобразить, известна уже на этапе разработки — а обычно так и есть — Паттерн 1 с готовыми компонентами будет быстрее, безопаснее и стабильнее. Генеративный UI оправдывает свою сложность лишь тогда, когда структура действительно меняется в зависимости от данных или контекста задачи.

**Ловушка: игнорирование доступности.** LLM‑ы могут «галлюцинировать» нарушения WCAG. Они могут добавить `role="region"` к интерактивным элементам, сгенерировать формы без меток и вывести контрастные соотношения, не проходящие WCAG AA. Ваша библиотека компонентов может быть полностью доступной; AI‑сгенерированные композиции этих компонентов автоматически не становятся доступными. Тестируйте весь путь рендеринга, а не только отдельные компоненты.

**Ловушка: смешивание протокола и фреймворка.** AG‑UI — это не фронтенд‑фреймворк. A2UI — это не библиотека React. Это форматы передачи данных и протоколы событий. Для их реализации всё равно нужен фронтенд‑фреймворк. CopilotKit реализует AG‑UI и A2UI. json‑render реализует шаблон каталога A2UI/Open‑JSON‑UI. Это разные уровни.

## Рекомендации по случаям использования

**Добавление копилота в существующее SaaS‑приложение**: Начните с шаблона 1 (tool‑to‑component). Используйте Vercel AI SDK `useChat` или CopilotKit. Сопоставьте 5–10 основных действий агента с готовыми компонентами. Выпустите эту версию, измерьте результаты, а затем расширяйте каталог только в том случае, если пользователи явно требуют более сложную композицию.

**Генерация дашборда из естественного языка**: Примените шаблон 2 с json‑render или собственным каталогом A2UI. Определите каталог из 8–15 типов компонентов, покрывающих ваши типы графиков, карточки метрик и варианты таблиц. Передайте схему модели; пусть она собирает макет. Реализуйте проверку, которая отлавливает неизвестные типы до того, как они попадут в рендерер.

**Мульти‑агентный фронтенд**: Используйте CopilotKit с AG‑UI. Поток событий обрабатывает потоковое вещание в реальном времени между бекэндами агентов; общая состояние управляет передачей управления между агентами; паттерн HITL обеспечивает контрольные точки одобрения.

**Создание внутри ChatGPT или другого хоста MCP**: используйте MCP Apps. Определите ваш инструмент как data‑tool, который получает данные и делает вывод, и отдельный render‑tool, который запрашивает виджет. Держите бизнес‑логику вне шаблона виджета.

**Демонстрации ML‑моделей и data‑приложения (команда Python)**: Gradio для демо и HuggingFace Spaces. Streamlit для data‑приложений с более сложным взаимодействием. Ни одно из них не требует JavaScript.

**Визуальные артефакты, симуляции, диаграммы**: используйте Паттерн 3 (OpenGenerativeUI или эквивалент). Установите строгий CSP для iframe. Рассматривайте вывод как недоверенный пользовательский контент с точки зрения безопасности.

---

Фреймворки быстро созревают. Конвергенция протоколов (AG‑UI для потоковой передачи, A2UI/Open‑JSON‑UI для спецификаций каталога) всё ещё находится в процессе, но форма уже достаточно ясна, чтобы на неё опираться.

Самыми важными инженерными задачами сейчас являются не выбор фреймворка. Это проектирование каталога — определение того, что модель может выдавать, что требует большей ясности продукта, чем технической экспертизы. Это семантическая валидация — проверка того, что сгенерированный UI действительно делает то, что заявлено, а не только проходит проверку схемы. И это разрыв в доступности — создание каталогов, где каждый компонент и любая их комбинация соответствуют тем же требованиям доступности, что и вручную написанный UI.

Модель будет выполнять то, что вы ей укажете, в пределах заданной грамматики. Сделайте грамматику преднамеренной.
