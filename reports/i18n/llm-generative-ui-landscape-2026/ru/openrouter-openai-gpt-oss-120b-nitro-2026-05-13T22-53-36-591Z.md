- Чат‑интерфейсы, встраивающие карточки продукта из вызовов инструмента модели

## Что НЕ является генеративным UI

Прежде чем определить, что это такое, отложим в сторону три вещи:

**Генерация кода во время проектирования** — v0, Lovable, Bolt, Cursor, компонуя React‑компоненты. Эти инструменты генерируют код, который разработчики просматривают и коммитят. AI работает на этапе разработки. Всё, что попадает в продакшн, статично с точки зрения пользователя. Это отличная категория инструментов. Это не то, что означает «генеративный UI во время выполнения».

**AI‑ассистент автозаполнения форм** — модель заполняет значения полей из контекста. Структура интерфейса остаётся фиксированной; меняется только содержимое. Это полезный паттерн. Это не генеративный UI.

**AI, пишущий сырый HTML в страницу** — модель выводит строки `<div>` и `<button>`, которые внедряются через `innerHTML` или `dangerouslySetInnerHTML`. Это *является* генеративным UI во время выполнения в самом техническом смысле. Это также самая опасная версия и та, которой каждый зрелый фреймворк в этой области пытается избежать. Сырой AI‑сгенерированный разметка несёт риск XSS, недоступные атрибуты, несогласованное стилизование и галлюцинированную структуру. Остальная часть статьи посвящена тому, как сделать лучше, чем это.

## Рабочее определение

Генеративный UI во время выполнения означает: **модель определяет, какой компонент интерфейса или композицию компонентов увидит пользователь, исходя из состояния диалога или задачи**.

Не слова. Интерфейс.

Самый простой пример: ваш помощник по бронированию авиабилетов вызывает инструмент `search_flights`. Вместо того чтобы вернуть простой текст («Вот три варианта…»), он рендерит компонент `<FlightResultsCard>` с выбором рейсов, переключателями класса места и кнопкой «Забронировать». Модель решила, что структурированная карточка — правильный ответ в данном случае. Разработчик определил, как выглядит эта карточка и что делает кнопка «Забронировать».

Более сложный пример: агент финансового анализа получает вопрос о портфеле и решает собрать ответ из `MetricGroup`, показывающего ключевые цифры, `RiskBreakdown`‑диаграммы, таблицы `ScenarioComparison` и `PolicyNotice`. Модель собрала такой макет из каталога предварительно одобренных компонентов. Разработчик описал каждый компонент. Модель выбрала, какие из них использовать и какие данные в них поместить.

Оба случая — генеративный UI. Они различаются степенью свободы композиции, которую получает модель, что определяет как потенциальную насыщенность выводов, так и сложность возможных ошибок.

## Три паттерна

Всё пространство сводится к трем паттернам, каждый со своим грамматическим набором вывода.

![Диаграмма спектра, показывающая три паттерна: вызовы инструментов слева (самый безопасный), каталог компонентов в середине и открытая генерация справа (самая выразительная).](../output-grammar-spectrum.svg)

_Каждое решение генеративного UI — это точка на этом спектре. Начинайте слева._

### Паттерн 1: Рендеринг инструмент‑компонент

Модель вызывает именованный инструмент. В вашем приложении есть отображение от имён инструментов к компонентам. Вызов инструмента инициирует рендер компонента.

```tsx
// Модель вызывает: { name: "show_flight_results", args: { flights: [...] } }

useCopilotAction({
  name: "show_flight_results",
  render: ({ args }) => <FlightResultsCard flights={args.flights} />,
});
```

Это самый безопасный паттерн, потому что разметка никогда не приходит от модели. Модель решает, *когда* показывать компонент и *какими данными* его заполнять. Разработчики по‑прежнему владеют кодом компонента, визуальным дизайном, реализацией доступности и всеми граничными случаями в логике рендера.

`useChat` из Vercel AI SDK с обработчиками `tool` реализует именно это. Рендеринг инструментов в assistant‑ui делает то же самое. «Static Generative UI» в CopilotKit — это данный паттерн. Большинство продакшн‑интерфейсов copilot, которые работают надёжно, используют его.

**Подходит, когда**: набор возможных отображаемых элементов известен на этапе разработки. Подтверждения бронирований, результаты поиска, сводки аккаунтов, виджеты одобрения. Если вы можете перечислить сценарии, этот паттерн покрывает их.

### Паттерн 2: Композиция из каталога компонентов

Модель выдаёт типизированное дерево JSON, в котором ссылки на компоненты берутся из каталога, определённого разработчиком. На фронтенде есть рендерер, который проходит по дереву и инстанцирует каждый компонент.

```json
[
  { "type": "metric_group", "metrics": [
    { "label": "MRR", "value": "$82,400", "delta": "+12%" },
    { "label": "Churn", "value": "2.1%", "delta": "-0.4%" }
  ]},
  { "type": "line_chart", "title": "30-day growth", "data_ref": "mrr_series" },
  { "type": "insight_callout", "text": "Expansion revenue driving the delta — avg seat count up 18%." }
]
```

Разметка составлена моделью: `MetricGroup`, `LineChart`, `InsightCallout`. Но вы определяете, что каждый тип компонента означает, какие свойства он принимает и как он рендерится. Если модель попытается выдать `{ "type": "custom_untested_thing" }`, ваша схема валидации поймает это, и рендерер проигнорирует или отклонит запрос.

Это паттерн, лежащий в основе `json-render`, `A2UI`, `Hashbrown`, `OpenUI` и `Tambo`. Ключевая инженерная работа — **проектирование каталога** — определение, какие типы компонентов существуют, как выглядят их схемы и что модели разрешено или запрещено составлять.

**Подходит, когда**: структура того, что нужно показать, действительно меняется в зависимости от данных или запроса пользователя. Дашборды, адаптирующиеся к заметным цифрам. Отчёты, показывающие разные секции в зависимости от контекста. Панели рабочего процесса, меняющиеся в зависимости от того, на каком шаге находится агент.

### Паттерн 3: Открытая генерация

Модель пишет HTML, SVG, Canvas или WebGL, который рендерится внутри изолированного iframe с жёсткой политикой Content Security Policy.

Это уместно для случаев, когда фиксированный каталог компонентов не справится: визуализации алгоритмов, архитектурные схемы, разово создаваемые графики, генеративное искусство, образовательные симуляции. Здесь безопасность обеспечивается границей iframe; если её убрать, вы вернётесь к проблеме внедрения «сырого» HTML, описанной в начале статьи.

`CopilotKit/OpenGenerativeUI` — наилучший текущий референс‑реализационный пример этого паттерна. Песочница удаляет скрипты, ограничивает передачу сообщений и держит сгенерированный артефакт вдали от привилегированного состояния вашего приложения.

**Подходит, когда**: вам действительно нужен произвольный визуальный вывод — одноразовые пояснительные диаграммы, динамические симуляции, креативные артефакты. Не используйте это для транзакционного UI. Подтверждение покупки не требует изолированного iframe.

### За пределами трёх паттернов: LLM‑ы, напрямую управящие пикселями

Появляется четвёртое направление, которое не укладывается чисто в любой из этих паттернов: LLM‑ы, создающие **иммерсивный, похожий на игру опыт** путём более прямого контроля визуального вывода, чем через изолированный iframe.

Каноническое различие в генеративном UI — **iframe HTML vs. JSON catalog**:

- **Iframe HTML** — модель пишет HTML, SVG, Canvas или WebGL, который исполняется в изолированном песочном окружении. Максимальная выразительная свобода; безопасность полностью зависит от границы iframe. Примеры: Anthropic Artifacts, OpenGenerativeUI.  
- **JSON catalog** — модель выдаёт структурированный payload, ограниченный каталогом компонентов, определённым разработчиком; ваш рендерер инстанцирует доверенные, заранее построенные компоненты из этой спецификации. Модель решает, *что* показывать; вы решаете, *как* это отобразить. Примеры: json-render, A2UI.

Помимо этого, совсем новые демонстрации намекают на третий режим, при котором модель не выбирает компоненты и не пишет изолированный HTML — она управляет канвой более напрямую. Проекты вроде [Tencent's HunyuanWorld](https://arxiv.org/abs/2502.01999), генерирующего исследуемые 3‑D среды из единственного изображения, и игровых архитектур, где LLM‑ы создают карты, NPC и квесты в режиме реального времени вместо вызова каталога компонентов, предвещают будущее, где модель выступает скорее как режиссёр игры, а не как форма‑рендерер. Инференс LLM в браузере через WebGPU ([WebLLM](https://mlc.ai/web-llm/)) продвигает эту же границу локально.

Эта область действительно захватывающая и ещё только начинается. Пока нет стабильных фреймворков для построения продакшн‑продуктов. Я расскажу об этом подходе в отдельной статье, когда появятся надёжные инструменты.

## Протоколы: AG-UI и A2UI

AG-UI и A2UI — два основных стандарта на уровне протоколов. Они решают разные задачи и не являются конкурентами.

### AG-UI

**GitHub**: [ag-ui-protocol/ag-ui](https://github.com/ag-ui-protocol/ag-ui)

AG-UI — это событийный протокол для обмена данными между AI‑агентами и фронтенд‑приложениями. Он описывает около 16 типов событий: `TEXT_MESSAGE_START`, `TEXT_MESSAGE_CONTENT`, `TOOL_CALL_START`, `TOOL_CALL_END`, `STATE_SNAPSHOT`, `STATE_DELTA` и др. Транспорт выбираете вы — SSE, WebSockets, веб‑хуки — всё работает. Формат намеренно свободный, чтобы облегчить широкое принятие.

AG-UI не определяет, как выглядит ваш UI. Он описывает, как агент общается *с* вашим фронтендом. Считайте его уровнем проводного протокола, позволяющим вашему React‑приложению подписаться на агент LangGraph так же, как оно подписывается на агент CrewAI, без изменения кода фронтенда.

CopilotKit создал AG-UI на основе работы с LangGraph и CrewAI. Протокол уже приняли LangChain, Mastra, PydanticAI и другие. Microsoft опубликовала руководство по интеграции AG-UI. Если вы строите фронтенд для нескольких агентов и хотите отделить бекенд‑фреймворки от кода UI, AG-UI — ваш выбор.

**Один момент, который часто путает**: AG-UI — это не UI‑фреймворк. Он не подсказывает, что отрисовывать. Он лишь сообщает, *что* агент что‑то сказал, вызвал инструмент или обновил совместное состояние. Что именно отобразить в ответ — всё равно ваше решение.

### A2UI

**GitHub**: [google/A2UI](https://github.com/google/A2UI) · Spec: [a2ui.org](https://a2ui.org/)

A2UI — декларативная спецификация Google о том, что агенты отправляют, когда хотят показать UI. Если AG-UI отвечает на вопрос «как агент общается?», то A2UI отвечает «в каком формате агент описывает макет компонента?».

A2UI использует плоский формат JSONL: один дескриптор компонента на строку, каждый с ID, типом и данными. Плоский формат выбран намеренно. Вложенные деревья требуют, чтобы модель знала полную структуру до начала стриминга. Плоский список позволяет модели выдавать каждый компонент по мере «размышления», что даёт возможность фронтенду начать рендерить первую карточку метрики, пока модель ещё решает, добавить ли график.

```jsonl
{"id":"h1","type":"kpi_card","title":"MRR","value":"$82,400","delta":"+12%"}
{"id":"h2","type":"kpi_card","title":"Churn","value":"2.1%","delta":"-0.4%"}
{"id":"c1","type":"line_chart","title":"30-day MRR","data_ref":"mrr_series"}
{"id":"t1","type":"data_table","cols":["Month","MRR","Net New"],"data_ref":"monthly"}
```

A2UI ориентирован на безопасность: спецификация — это формат данных, а не исполняемый код. Каталог компонентов задаётся разработчиком заранее; агент может ссылаться только на типы, объявленные в этом каталоге. Если рендерер A2UI получает неизвестное имя типа, он просто игнорирует его.

Формат «Open-JSON-UI» от CopilotKit совместим с A2UI. Если вы выбираете формат спецификации для каталога компонентов сегодня, A2UI предлагает наибольшую кроссплатформенную поддержку.

**Замечание о стабильности**: A2UI находится в предрелизе — v0.9 на момент последней проверки 8 мая 2026 года — и уже претерпел несовместимые изменения между минорными версиями. Информационные сообщения Google о дорожной карте приходят нерегулярно, а некоторые рендереры (Lit, Flutter) отстают от обновлений спецификации. Планируйте время на возможный дрейф спецификации, если собираетесь использовать её сейчас. Для чисто веб‑ориентированных сценариев json-render пока выглядит более полностью поддерживаемым инструментарием. Долгосрочное преимущество A2UI — широкий кроссплатформенный охват (web, Flutter, SwiftUI, Android), которого у json-render нет.

### MCP Apps

**GitHub**: [modelcontextprotocol](https://github.com/modelcontextprotocol) · Связанные проекты: [mcp-ui](https://github.com/MCP-UI-Org/mcp-ui)

MCP изначально задумывался как протокол для соединения LLM с инструментами и данными. Расширение Apps позволяет инструментам MCP возвращать не только данные, но и интерактивные UI‑артефакты: React‑компоненты, формы, дашборды, карты.

Модель безопасности строгая, и в этом её смысл: всё рендерится в изолированном iframe с минимальными правами, шаблоны объявляются заранее, чтобы хост‑приложение могло их проверить, а вся коммуникация проходит через аудируемый JSON‑RPC. Это правильный подход для поставщиков инструментов — например, сервер MCP для Shopify может вернуть виджет оформления заказа; сервис карт может вернуть встраиваемую карту. Хост‑приложение не владеет и не доверяет коду этого виджета.

MCP Apps подходит, когда UI *принадлежит поставщику инструмента*, а не вашему приложению. Если пользовательский интерфейс находится в домене вашего приложения, лучше использовать шаблон 1 или 2.

## JavaScript/TypeScript Frameworks

### CopilotKit

**GitHub**: [CopilotKit/CopilotKit](https://github.com/CopilotKit/CopilotKit) · Примеры: [CopilotKit/generative-ui](https://github.com/CopilotKit/generative-ui)

CopilotKit — самый полный фреймворк для агентно‑нативных фронтенд‑приложений. Он покрывает весь жизненный цикл: подключение к бекэндам‑агентам через AG‑UI, управление двунаправленным состоянием диалога, рендеринг генеративных UI‑компонентов и предоставление общей инфраструктуры состояния, позволяющей агентам и пользователям изменять одни и те же данные.

Модель из трёх паттернов естественно сопоставляется с API CopilotKit:
- `useCopilotAction` с колбэком `render` → Паттерн 1
- A2UI/Open-JSON-UI рендеринг → Паттерн 2
- `OpenGenerativeUI` — песочница артефактов → Паттерн 3

Важная, но часто недооценённая возможность CopilotKit — **общий стейт и человек‑в‑цикле**: агент может читать и писать состояние приложения, пользователь тоже может его читать и писать, а изменения передаются в обе стороны. Именно это делает UI‑интерфейсы в стиле Copilot ощущаемыми как реальное сотрудничество, а не просто чат‑окно, приклеенное к продукту.

### Vercel AI SDK

**GitHub**: [vercel/ai](https://github.com/vercel/ai) · Документация: [ai-sdk.dev](https://ai-sdk.dev/)

Vercel AI SDK — де‑факто базовый TypeScript‑стек для AI‑приложений. Для генеративного UI в частности:

**`useObject`** потоково передаёт структурированный JSON‑объект с сервера по мере его генерации. Вы задаёте схему Zod; SDK парсит частичный JSON и инициирует повторные рендеры по мере поступления полей. Это самый гладкий путь к Паттерну 2 в приложении Next.js.

```tsx
const { object: dashboard } = useObject({
  api: "/api/generate-dashboard",
  schema: z.object({
    title: z.string(),
    metrics: z.array(z.object({ label: z.string(), value: z.number() })),
    insights: z.array(z.string()),
  }),
});
```

**`useChat` с обработчиками инструментов** → Паттерн 1. Модель вызывает инструменты; вы сопоставляете имена инструментов компонентам.

**AI Elements** ([elements.ai-sdk.dev](https://elements.ai-sdk.dev/)) предоставляет готовые UI‑примитивы для использования вместе с SDK.

**Замечание о запутанном развитии**: В октябре 2024 года Vercel объявил в [GitHub Discussion #3251](https://github.com/vercel/ai/discussions/3251), что AI SDK RSC — паттерн потоковой передачи React Server Components, продвигаемый как флагманская функция «Generative UI» в SDK 3.0 — приостановлен на неопределённый срок из‑за «нескольких давних ограничений», для которых нет хороших краткосрочных решений. Команды, построившие продуктовые стратегии вокруг потоковой передачи RSC, оказались неожиданно в затруднительном положении. API `generateObject`/`streamObject` также позже были помечены как устаревшие в SDK 6.0. Рекомендуемая миграция от AI SDK RSC — использовать паттерн `useObject`, описанный выше, или json-render для генерации на основе каталога.

### assistant-ui

**GitHub**: [assistant-ui/assistant-ui](https://github.com/assistant-ui/assistant-ui)

assistant-ui — набор компонуемых React‑примитивов для создания чат‑интерфейсов промышленного уровня. Это правильный выбор, когда нужен отшлифованный UX чата — пузырьки сообщений, потоковые токены, действия копировать/редактировать/перегенерировать, состояния «размышления» — и при этом хочется использовать собственный бекенд и собственный рендеринг инструментов.

Он хорошо работает в паре с любым бекендом (OpenAI, Anthropic, локальные модели, кастомные эндпоинты) и обрабатывает отображение вызовов инструментов через знакомую модель slot/render‑prop.

### json-render

**GitHub**: [vercel-labs/json-render](https://github.com/vercel-labs/json-render) · Docs: [json-render.dev](https://json-render.dev/)

json-render реализует Паттерн 2 с предвзятой, «всё‑в‑одном» стратегией. Вы получаете готовый каталог компонентов (компоненты shadcn/ui с схемами Zod), рендерер и плотный цикл генерации, где модель ограничена каталогом схемой.

Отличительные особенности:
- **Мульти‑целевая отрисовка**: один и тот же JSON‑спецификатор может отрисоваться в веб‑приложении React, мобильном приложении React Native, PDF, HTML‑письме или видео Remotion. Это действительно полезно для отчётов.
- **Прогрессивная отрисовка**: компоненты появляются по мере того, как модель их стримит, а не после получения полной спецификации.
- **Жёсткие ограничения схемы**: каталог спроектирован так, что модель не может «галлюцинировать» валидные, но неизвестные типы компонентов.

Если вы разрабатываете панель мониторинга или функцию генерации отчётов и хотите обойти инфраструктурную работу по созданию собственного каталога, **json-render** — самый быстрый путь для веб‑приложений.

**О текущем росте**: json-render был запущен Vercel Labs в начале 2026 года и, судя по всему, быстро привлёк внимание веб‑разработчиков, потому что сразу же полезен в типичных проектах React/Next.js. При этом json-render всё ещё находится в предрелизной версии < 1.0, а взаимосвязь между json-render и A2UI пока уточняется — Vercel экспериментировал с выводом, совместимым с A2UI, так что сходимость возможна. Для кроссплатформенных решений (нативный мобильный, несколько фреймворков) A2UI остаётся более надёжным долгосрочным выбором.

### Hashbrown

**GitHub**: [liveloveapp/hashbrown](https://github.com/liveloveapp/hashbrown)

Hashbrown использует отличительный подход: вместо построения отдельного слоя AI‑интерфейса он встраивает выбор AI‑компонентов непосредственно в ваше существующее приложение на React или Angular. Вы раскрываете компоненты приложения LLM; LLM выбирает, какие из них отрисовать, и может вызывать клиентские инструменты.

Это подходящий инструмент, когда нужно добавить интеллект в продуктовые поверхности, не являющиеся «чатом» — страницу продукта, адаптирующую свой макет, панель настроек, предлагающую релевантные опции, или редактор рабочего процесса, предлагающий следующий шаг.

### OpenUI

**GitHub**: [thesysdev/openui](https://github.com/thesysdev/openui) · Docs: [openui.com](https://www.openui.com/)

OpenUI заменяет JSON на построчный, похожий на код формат («OpenUI Lang»), разработанный для прогрессивной отрисовки и эффективности токенов. По заявлению, он экономит примерно 67 % токенов по сравнению с эквивалентным JSON при сложных макетах.

Компромисс — зрелость экосистемы — OpenUI новее, а инструментарий тоньше, чем у подходов, основанных на JSON. Но если стоимость токенов является значимым ограничением и вы генерируете сложные макеты с высокой частотой, эффективность формата действительно ощутима.

### Tambo

**GitHub**: [tambo-ai/tambo](https://github.com/tambo-ai/tambo)

Tambo ориентирован на выбор состояний компонентов: ИИ выбирает компоненты и может взаимодействовать с ними через клиентские инструменты, сохраняя состояние компонента в течение всего диалога. Подходит для сценариев, где UI‑элементы сохраняются между ходами — например, фильтрующий компонент, который пользователь настраивает, пока ИИ продолжает анализировать отфильтрованные данные.

## Слой Python

Экосистема Python подходит к AI‑интерфейсам иначе. Эти инструменты оптимизированы для демонстраций ML‑моделей, данных и внутренних утилит — а не для массовых потребительских приложений с композицией макетов, управляемой агентом.

Это не критика. Для подходящих задач Gradio и Streamlit — единственные инструменты, которые вам понадобятся.

### Gradio

**GitHub**: [gradio-app/gradio](https://github.com/gradio-app/gradio) · PyPI: `gradio`

Основная ценность Gradio: вы пишете Python‑функцию; Gradio оборачивает её в веб‑интерфейс. Класс `Interface` занимает 3 строки для классификатора изображений. `ChatInterface` — 10 строк для чат‑бота. `Blocks` предоставляет детальный контроль над макетом, когда он действительно нужен.

«Генеративный UI» в Gradio определяется Python‑разработчиком, а не моделью. Видимость компонентов и их конфигурация могут изменяться динамически в зависимости от выводов модели, но каталог компонентов статичен — вы не просите модель собирать макеты.

Gradio — де‑факто стандарт для HuggingFace Spaces и экосистемы демонстраций машинного обучения. У него миллионы загрузок в месяц, и он обслуживает большую часть ландшафта AI‑демо.

**Выбирайте Gradio, когда**: вы Python‑разработчик, создаёте демо‑модель ML, исследовательский прототип или внутренний инструмент и не хотите трогать JavaScript.

### Streamlit

**GitHub**: [streamlit/streamlit](https://github.com/streamlit/streamlit)

Модель Streamlit более предопределённая: Python‑скрипт исполняется полностью при каждом взаимодействии. Вы вызываете `st.chat_message()`, `st.dataframe()`, `st.plotly_chart()`. Фреймворк управляет расположением элементов.

Полный перезапуск скрипта выглядит неэффективным, но оказывается достаточно удобным для чат‑ботов с накопленной историей — скрипт переходит, история чата хранится в состоянии сессии, а вывод детерминирован. Streamlit теперь предоставляет официальную поддержку большинства крупных провайдеров LLM и интегрируется с Snowflake Cortex «из коробки».

**Выбирайте Streamlit, когда**: вы строите приложение с AI‑поддержкой данных, внутренний инструмент отчётности или панель мониторинга, основанную на ML, и хотите максимально простую цепочку развертывания в Python.

### LangChain и Haystack

Это фреймворки оркестрации бекенда, а не UI‑фреймворки. Они появляются на любой честной карте стека генеративного UI, потому что обычно именно на этом уровне генерируются структурированные выводы, которые затем передаются фронтенду.

**LangChain** ([langchain-ai/langchain](https://github.com/langchain-ai/langchain)): `.with_structured_output()` на любой LLM выдаёт JSON, ограниченный схемой Pydantic. Декоратор `@tool` с автоматической генерацией схемы — самый чистый способ задать, какие инструменты модель может вызывать. LangChain передаёт структурированные результаты дальше к любой используемой вами фронтенд‑слой.

**Haystack** ([deepset-ai/haystack](https://github.com/deepset-ai/haystack)): модульная архитектура конвейера с сильной поддержкой RAG. Hayhooks оборачивает конвейеры Haystack в HTTP‑эндпоинты — включая совместимые с MCP эндпоинты. Если вашему генеративному UI нужен слой поиска, архитектура конвейера Haystack справляется с этим аккуратно.

Ни один из этих фреймворков не владеет уровнем UI. Они генерируют данные, которые ваш фронтенд (шаблон 1, 2 или 3) отображает.

## Скорость развития экосистемы и нестабильная почва

Это направление меняется стремительно, и несколько проектов публиковали запутанные сообщения вместе с кодом. Последняя проверка — 8 мая 2026; рассматривайте заметки о статусе проектов здесь как моментальный снимок, а не как окончательное заключение.

**Vercel AI SDK RSC** был флагманской функцией Generative UI при запуске SDK 3.0. Vercel приостановил её разработку в октябре 2024 года ([Discussion #3251](https://github.com/vercel/ai/discussions/3251)), ссылаясь на архитектурные ограничения React Server Components, для которых не было близкосрочного решения. Команды, построившие свои продукты на этой базе, естественно разочаровались. Фича всё ещё присутствует в документации, но уже не рекомендуется; вместо неё следует использовать `useObject`.

**json-render** (Vercel Labs) — новое направление: каталог‑ориентированная, независимая от фреймворка альтернатива, устраняющая проблемы сопряжения с RSC. Проект находится в предрелизе 1.0 и, судя по раннему интересу среди разработчиков React/web, имеет потенциал. Вероятная причина удобства: json-render сразу работает в обычном проекте React/Next.js, тогда как кроссплатформенный охват A2UI добавляет накладные расходы на настройку. Как сложатся обе спецификации по мере их созревания, пока действительно неизвестно. Vercel уже проверял совместимость A2UI в json-render, что намекает на возможность конвергенции.

**A2UI** (Google) тоже находится в предрелизе 1.0 (v0.9 на момент последней проверки) с ломающими изменениями между минорными версиями и непоследовательными коммуникациями Google о дорожной карте. Это правильный выбор для кроссплатформенного охвата (web + Flutter + SwiftUI), который json-render не покрывает, и у проекта есть значительная поддержка со стороны предприятий. Для чисто веб‑проектов сейчас DX (опыт разработчика) менее удобен.

**AG-UI** (CopilotKit) также предрелизный. Наиболее частая путаница: название звучит как UI‑фреймворк. На самом деле это транспортный протокол. AG-UI определяет, как события передаются между бекендом агента и вашим фронтендом; то, что вы рендерите в ответ, остаётся вашим решением. Такая ментальная модель надёжна и широко принята, но предрелизная спецификация означает, что граничные случаи всё ещё прорабатываются.

Практический вывод: **каждый из основных игроков находится в предрелизе 1.0**. Планируйте изменения API. Паттерны — tool-to-component, catalog composition, sandboxed generation — достаточно стабильны для построения над ними. Конкретный выбор протокола пока не фиксирован.

## Дизайн каталога компонентов: реальная инженерная работа

Большая часть интересующей сложности во втором паттерне скрыта не в рендерере, а в каталоге.

Каталог — это **продуктовое решение, зашифрованное в схеме**. Он отвечает на вопрос: какие значимые UI‑объекты существуют в данном домене? Не «какие React‑компоненты есть?», а «что пользователю в этом контексте действительно нужно увидеть и с чем взаимодействовать?»

**Слишком гранулярный режим отказа**: вы раскрываете `Row`, `Column`, `Text`, `Button`, `Icon`. Теперь модель должна выступать в роли фронтенд‑инженера. Она будет генерировать посредственный макет, не соответствующий вашей дизайн‑системе, упустит пустые состояния, создаст недоступный markup и будет менять подход от ответа к ответу, потому что в каталоге нет ограничений, заставляющих её придерживаться визуального языка вашего продукта.

**Слишком грубый режим отказа**: вы раскрываете `WeatherCard`, `FlightCard`, `HotelCard`. Модель не может адаптироваться, когда пользователь просит что‑то, что не сопоставляется с готовой карточкой. Она откатывается к простому тексту.

**Полезный средний вариант**: компоненты уровня домена с ограниченными слотами.

Каталог travel‑приложения может выглядеть так:

```
TripSummary         — itinerary at a glance
FlightOptionList    — selectable flight options with pricing
HotelComparison     — side-by-side hotel cards
TravelerForm        — collect traveler details
PolicyNotice        — regulatory/fare rule callout
BookingConfirmation — final confirmation with action button
```

Каталог финансового приложения может выглядеть так:

```
PortfolioSnapshot   — key positions and P&L
TransactionTable    — filterable, paginated transactions
RiskBreakdown       — allocation and volatility metrics
ScenarioComparison  — side-by-side scenario modeling
ApprovalGate        — action requiring human confirmation
```

Каталог звучит как ваш продуктовый словарь. Он фиксирует UX‑решения, требования к доступности, обработку пустых состояний и шаблоны опасных действий в коде компонентов. Модель получает возможность комбинировать эти части. Вы по‑прежнему решаете, как выглядит каждый элемент и что ему разрешено делать.

**Правила проектирования схем, снижающие галлюцинации**:

1. Делайте значения enum короткими и очевидными. `"type": "bar_chart"` вместо `"type": "data-visualization-bar-type-vertical"`.
2. Делайте невозможной недопустимую композицию. Если `PolicyNotice` может появляться только в конце макета, не размещайте его на том же уровне схемы, где находятся элементы, допускающие произвольное расположение.
3. Щедро используйте обязательные поля. Опциональное поле — это поле, которое модель может опустить, и ваш рендерер придётся обрабатывать как `null`.
4. Тестируйте каталог на реальных запросах до релиза. Сохраняйте сгенерированные спецификации; проверяйте их на нарушения схемы, галлюцинированные значения полей и композиции, которые технически корректны, но семантически ошибочны.

## Общие ловушки

**Ловушка: считать валидный JSON безопасным поведением.** Проверка схемы гарантирует только структуру. Она ничего не говорит о том, соответствует ли действие, привязанное к кнопке, её подписи, совпадает ли итог с данными, из которых он вычислен, или делает ли UI‑компонент то, чего пользователь не ожидал. Сгенерированные спецификации UI требуют семантической ревью, а не только валидации схемы. Как минимум, разрушительные действия должны требовать компонента подтверждения, а подписи этих компонентов следует проверять на соответствие вызываемым действиям.

**Ловушка: раскрытие дизайнерских примитивов вместо продуктовых примитивов.** Если модели приходится решать, использовать отступ 16 px или 20 px, вы задали ей неверный уровень абстракции. Доменные компоненты должны инкапсулировать вкусовые решения продукта. Модель должна композировать поведение, а не управлять деталями презентации.

**Ловушка: применение генеративного UI там, где достаточно статического UI.** Если структура того, что нужно отобразить, известна на этапе разработки — а обычно так и есть — Паттерн 1 с готовыми компонентами будет быстрее, безопаснее и согласованнее. Сложность генеративного UI оправдывается только тогда, когда структура действительно меняется в зависимости от данных или контекста задачи.

**Ловушка: игнорирование доступности.** LLM‑ы могут «галлюцинировать» нарушения WCAG. Они могут добавить `role="region"` к интерактивным элементам, сгенерировать формы без меток и вывести контрастные соотношения, не проходящие WCAG AA. Ваша библиотека компонентов может быть полностью доступной, но AI‑сгенерированные композиции этих компонентов автоматически не становятся доступными. Тестируйте весь путь рендеринга, а не только отдельные компоненты.

**Ловушка: смешивание протокола и фреймворка.** AG‑UI — это не фронтенд‑фреймворк. A2UI — это не библиотека React. Это форматы передачи данных и протоколы событий. Для их реализации всё равно нужен фронтенд‑фреймворк. CopilotKit реализует AG‑UI и A2UI. `json-render` реализует паттерн каталога A2UI/Open-JSON-UI. Это разные уровни абстракции.

## Рекомендации по сценариям использования

**Добавление копилота в существующее SaaS‑приложение**: Начните с Паттерна 1 (tool-to-component). Используйте Vercel AI SDK `useChat` или CopilotKit. Сопоставьте 5–10 основных действий агента с готовыми компонентами. Выпустите эту версию, измерьте результаты, а затем расширяйте каталог только если пользователи явно нуждаются в более сложных композициях.

**Генерация дашбордов из естественного языка**: Примените Паттерн 2 с `json-render` или собственным каталогом A2UI. Определите каталог из 8–15 типов компонентов, покрывающих ваши типы графиков, карточки метрик и варианты таблиц. Передайте схему модели; пусть она соберёт макет. Реализуйте валидацию, которая отлавливает неизвестные типы до того, как они попадут в рендерер.

**Мульти‑агентный фронтенд**: Используйте CopilotKit с AG‑UI. Поток событий обеспечивает реальное время передачи данных между бекэндами агентов; общая состояние управляет переключением между агентами; паттерн HITL (Human‑in‑the‑Loop) реализует контрольные точки согласования.

**Разработка внутри ChatGPT или другого хоста MCP**: Используйте MCP Apps. Определите ваш инструмент как data‑tool, который получает данные и делает выводы, и отдельный render‑tool, который запрашивает виджет. Держите бизнес‑логику вне шаблона виджета.

**ML model demos and data apps (Python team)**: Gradio для демонстраций и HuggingFace Spaces. Streamlit для data‑приложений со сложным взаимодействием. Ни один из них не требует JavaScript.

**Visual artifacts, simulations, diagrams**: Используйте Pattern 3 (OpenGenerativeUI или эквивалент). Установите строгий CSP для iframe. Рассматривайте вывод как недоверенный пользовательский контент с точки зрения безопасности.

Фреймворки быстро созревают. Конвергенция протоколов (AG‑UI для потоковой передачи, A2UI/Open‑JSON‑UI для спецификаций каталога) ещё в процессе, но форма уже достаточно ясна, чтобы на неё опираться.

Самые важные инженерные задачи сейчас — не выбор фреймворка. Это проектирование каталога — определение того, что модель может генерировать, что требует большей ясности продукта, чем технической экспертизы. Это семантическая валидация — проверка того, что сгенерированный UI действительно делает заявленное, а не просто проходит схему. И это разрыв в доступности — создание каталогов, где каждый компонент и каждая их композиция соответствуют уровню доступности, который вы требовали бы от вручную написанного UI.
