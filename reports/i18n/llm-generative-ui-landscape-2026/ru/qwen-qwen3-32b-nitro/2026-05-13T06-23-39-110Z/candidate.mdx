«Генеративный интерфейс» означает как минимум пять разных вещей в зависимости от того, кто это говорит.  

- Чат-интерфейсы, встраивающие карточки продуктов через вызовы инструментов модели  
- Runtime JSON-спецификации, которые фронтенд рендерит в виде деревьев компонентов  
- Изолированные iframe, возвращаемые инструментами MCP в хост-приложения (от заказа билетов, бронирования отелей до отрисовки карт и виджетов оформления заказа)  
- Протоколы событий, стримящие состояние агента на фронтенд  
- v0, Lovable и Bolt: инструменты ИИ, пишущие React на этапе проектирования  

Это связанные концепции, но они работают на разных слоях стека, несут разные профили рисков, требуют разной сложности реализации и подходят для разных сценариев использования. Смешивать их — превращать каждое обсуждение архитектуры в хаос.

## Что такое Generative UI не является  
--- CHUNK END ---

Прежде чем определить, что это такое, три вещи, которые стоит отложить в сторону:  

**Генерация кода на этапе проектирования** — v0, Lovable, Bolt, Cursor, создающие компоненты React. Эти инструменты генерируют код, который разработчики проверяют и фиксируют. ИИ работает на этапе разработки. То, что поставляется, является статичным с точки зрения пользователя. Это отличная категория инструментов. Это не то, что подразумевается под «генеративным UI во время выполнения».  

**Автозаполнение форм с помощью ИИ** — модель заполняет значения полей на основе контекста. Структура интерфейса остаётся фиксированной; меняется только содержимое. Это полезный паттерн. Это не генеративный UI.

**ИИ, пишущий сырой HTML на страницу** — модель генерирует строки `<div>` и `<button>`, которые вставляются через `innerHTML` или `dangerouslySetInnerHTML`. Это *является* генеративным UI во время выполнения в самом техническом смысле. Это также самая опасная версия, и именно её стремится избежать каждый зрелый фреймворк в этой области. Сырая генерация HTML с помощью ИИ означает риск XSS, недоступные атрибуты, несогласованный стиль и вымышленную структуру. Остальная часть этой статьи посвящена тому, как сделать лучше, чем это.  

## Рабочее определение

Генеративный UI во время выполнения означает: **модель определяет, какой интерфейсный компонент или композиция компонентов пользователь видит, исходя из состояния разговора или задачи.**  

Не слова. Интерфейс.  

Простейший случай: ваш ассистент по бронированию авиабилетов вызывает инструмент `search_flights`. Вместо возврата простого текста ("Вот три варианта...") он отрисовывает компонент `<FlightResultsCard>` с переключаемыми рейсами, переключателями класса кабины и кнопкой "Забронировать". Модель определила, что карточка с структурированным интерфейсом — это правильный ответ в данном случае. Разработчик определил, как выглядит эта карточка и что делает кнопка "Забронировать".

Более сложный случай: агент по финансовому анализу получает вопрос о портфеле и решает составить ответ с помощью `MetricGroup`, показывающей ключевые цифры, диаграммы `RiskBreakdown`, таблицы `ScenarioComparison` и уведомления `PolicyNotice`. Модель собрала этот макет из каталога предварительно утвержденных компонентов. Разработчик определил каждый компонент. Модель выбрала, какие из них использовать, и какие данные в них разместить.

Оба случая относятся к генеративному интерфейсу. Они отличаются степенью свободы модели в составлении ответа, которая определяет как разнообразие возможных выходных данных, так и сложность потенциальных ошибок.

## Три паттерна

Все пространство сводится к трём паттернам, каждый из которых имеет свою выходную грамматику.

![Диаграмма спектра, показывающая три паттерна: только вызовы инструментов слева (самый безопасный), каталог компонентов посередине и открытая генерация справа (наиболее выразительная).](../output-grammar-spectrum.svg)

_Каждое решение в области генеративного UI — это точка на этом спектре. Начните слева._

### Паттерн 1: Рендеринг от инструмента к компоненту

Модель вызывает именованный инструмент. Ваше приложение имеет отображение от названий инструментов к компонентам. Вызов инструмента запускает рендеринг компонента.

```tsx
// The model calls: { name: "show_flight_results", args: { flights: [...] } }

useCopilotAction({
  name: "show_flight_results",
  render: ({ args }) => <FlightResultsCard flights={args.flights} />,
});
```

Это самый безопасный паттерн, потому что компоновка интерфейса никогда не поступает от модели. Модель решает *когда* отображать компонент и *какие данные* в него поместить. Разработчики всё ещё контролируют код компонента, визуальный дизайн, реализацию доступности и все граничные случаи в логике рендеринга.

SDK Vercel AI с `useChat` и обработчиками `tool` работает именно так. Рендеринг инструментов в assistant-ui реализует этот паттерн. "Статический генеративный UI" в CopilotKit — это реализация этого подхода. Большинство надежно работающих в продакшене UI-интерфейсов сопровождающих ИИ используют именно этот паттерн.

**Подходит, когда**: множество возможных сценариев отображения известно на этапе разработки. Подтверждения бронирования, результаты поиска, сводки аккаунтов, виджеты одобрения. Если вы можете перечислить сценарии, этот паттерн покроет их все.

### Паттерн 2: Составление из каталога компонентов

Модель генерирует типизированное JSON-дерево, ссылающееся на компоненты из каталога, определённого разработчиком. Ваш фронтенд имеет рендерер, который обходит дерево и инстанцирует каждый компонент.

```json
[
  { "type": "metric_group", "metrics": [
    { "label": "MRR", "value": "$82,400", "delta": "+12%" },
    { "label": "Churn", "value": "2.1%", "delta": "-0.4%" }
  ]},
  { "type": "line_chart", "title": "30-day growth", "data_ref": "mrr_series" },
  { "type": "insight_callout", "text": "Expansion revenue driving the delta — avg seat count up 18%." }
]
```

Модель составила этот макет. `MetricGroup`, `LineChart`, `InsightCallout`. Но вы определили, что означает каждый тип компонента, какие пропсы он принимает и как рендерится. Если модель попытается сгенерировать `{ "type": "custom_untested_thing" }`, валидация схемы поймает это и рендерер проигнорирует или отвергнёт его.

Это паттерн, лежащий в основе `json-render`, `A2UI`, `Hashbrown`, `OpenUI` и `Tambo`. Основная инженерная работа — **проектирование каталога** — определение типов компонентов, как выглядят их схемы и что модель может/не может составлять.

**Подходит, когда**: структура того, что вы хотите показать, действительно меняется в зависимости от данных или запроса пользователя. Дэшборды, адаптирующиеся к тому, что выделяется в числах. Отчёты, показывающие разные разделы в зависимости от контекста. Панели рабочего процесса, изменяющиеся в зависимости от того, на каком этапе находится агент.

### Паттерн 3: Открытая генерация

Модель генерирует HTML, SVG, Canvas или WebGL, которые отображаются внутри изолированного iframe с жёсткой Политикой безопасности содержимого.  

Это подходит для задач, где фиксированный набор компонентов невозможен: визуализации алгоритмов, архитектурные диаграммы, импровизированные диаграммы, генеративное искусство, образовательные симуляции. Граница iframe здесь обеспечивает безопасность; удалите её, и вы вернётесь к проблеме вставки сырого HTML, описанной в начале этой статьи.  

`CopilotKit/OpenGenerativeUI` — это лучшая на сегодняшний день референсная реализация этого паттерна. Изолированная среда удаляет скрипты, ограничивает передачу сообщений и изолирует сгенерированный артефакт от привилегированного состояния вашего приложения.

**Возможно, когда**: вам действительно нужен произвольный визуальный вывод — одноразовые поясняющие диаграммы, динамические симуляции, творческие артефакты. Не используйте это для транзакционных интерфейсов. Подтверждение заказа не требует изолированного iframe.  

### За пределами трёх паттернов: LLM, управляющие пикселями напрямую  

Существует четвертое направление, которое только начинает формироваться и не вписывается в эти паттерны: LLM, управляющие **погружными, похожими на игру опытами**, контролируя визуальный вывод напрямую, более чем изолированный iframe.

Каноническое различие в генеративном UI — это **HTML в iframe vs. JSON-каталог**:

- **HTML в iframe** — модель генерирует HTML, SVG, Canvas или WebGL, которые отображаются в изолированной песочнице. Максимальная выразительная свобода; безопасность зависит исключительно от границы iframe. Примеры: Anthropic Artifacts, OpenGenerativeUI.
- **JSON-каталог** — модель выдаёт структурированный payload, ограниченный каталогом компонентов, определённых разработчиком; ваш рендерер создаёт доверенные, предварительно собранные компоненты на основе этого описания. Модель решает *что* показывать; вы решаете *как* это отображать. Примеры: json-render, A2UI.

Помимо этого, совсем недавние демонстрации намекают на третий режим, где модель не выбирает компоненты и не пишет изолированный HTML — она управляет холстом напрямую. Проекты вроде [Tencent's HunyuanWorld](https://arxiv.org/abs/2502.01999), который генерирует интерактивные 3D-среды из одиночного изображения, и игровые архитектуры, где LLM создают карты, NPC и квесты в реальном времени, а не обращаются к каталогу компонентов, указывают на будущее, где модель будет похожа на режиссёра игры, а не на рендерер форм. Локальная инференс LLM через WebGPU ([WebLLM](https://mlc.ai/web-llm/)) расширяет те же границы локально.

Эта территория действительно волнующая и действительно ранняя. Еще нет стабильных фреймворков для создания промышленных продуктов. Я рассмотрю этот подход в отдельной статье, как только это изменится.

## Протоколы: AG-UI и A2UI

AG-UI и A2UI — это два основных стандарта в слое протоколов. Они решают разные задачи и не являются конкурентами.

### AG-UI

**GitHub**: [ag-ui-protocol/ag-ui](https://github.com/ag-ui-protocol/ag-ui)

AG-UI — это событийный протокол для взаимодействия между ИИ-агентами и фронтенд-приложениями. Он определяет около 16 типов событий: `TEXT_MESSAGE_START`, `TEXT_MESSAGE_CONTENT`, `TOOL_CALL_START`, `TOOL_CALL_END`, `STATE_SNAPSHOT`, `STATE_DELTA` и так далее. Способ передачи данных остаётся на ваше усмотрение — поддерживаются SSE, WebSockets, вебхуки. Формат умышленно не строг, чтобы обеспечить широкое принятие.

AG-UI не определяет, как будет выглядеть ваш интерфейс. Он определяет, как агент взаимодействует *с* вашим фронтендом. Представьте его как уровень проводного протокола, который позволяет вашему React-приложению подписываться на агента LangGraph так же, как на агента CrewAI, без изменения фронтенд-кода.

CopilotKit разработали AG-UI на основе их работы с LangGraph и CrewAI. Его приняли LangChain, Mastra, PydanticAI и другие. Microsoft опубликовала руководство по интеграции AG-UI. Если вы создаете мультиагентный фронтенд и вам нужно отделить фреймворки бэкенда от кода фронтенда, AG-UI — это ответ.

**Одна уточнение, которая вызывает путаницу**: AG-UI не является фреймворком пользовательского интерфейса. Он не указывает, что отображать. Он сообщает, *что* агент что-то сказал, вызвал инструмент или обновил общее состояние. Что вы отобразите в ответ — решаете вы сами.

### A2UI

**GitHub**: [google/A2UI](https://github.com/google/A2UI) · Спецификация: [a2ui.org](https://a2ui.org/)

A2UI — это декларативная спецификация Google, описывающая, как агенты отправляют данные, когда хотят отобразить пользовательский интерфейс. В то время как AG-UI отвечает на вопрос «как агент общается?», A2UI отвечает на вопрос «в каком формате агент описывает компоновку компонентов?».

A2UI использует плоский формат JSONL: один описатель компонента на строку, каждый из которых содержит идентификатор, тип и данные. Плоскость — это намеренное решение. Вложенные деревья требуют, чтобы модель знала полную структуру до начала стриминга. Плоский список позволяет модели отправлять каждый компонент по мере его «выработки», что означает, что ваш фронтенд может начать отрисовку первой карточки метрики, пока модель всё ещё решает, добавлять ли график.

```jsonl
{"id":"h1","type":"kpi_card","title":"MRR","value":"$82,400","delta":"+12%"}
{"id":"h2","type":"kpi_card","title":"Churn","value":"2.1%","delta":"-0.4%"}
{"id":"c1","type":"line_chart","title":"30-day MRR","data_ref":"mrr_series"}
{"id":"t1","type":"data_table","cols":["Month","MRR","Net New"],"data_ref":"monthly"}
```

A2UI уделяет внимание безопасности: спецификация — это формат данных, а не исполняемый код. Каталог компонентов определяется заранее разработчиком; агент может ссылаться только на типы из этого каталога. Рендерер A2UI, получивший неизвестное имя типа, игнорирует его.

Формат "Open-JSON-UI" CopilotKit совместим с A2UI. Если вы выбираете формат спецификации для каталога компонентов сегодня, A2UI — это формат с наиболее широкой поддержкой на разных платформах.

**Примечание по стабильности**: A2UI находится в предверсии 1.0 — версия v0.9 на момент проверки 8 мая 2026 года — и вносил разрывающие изменения в спецификацию между минорными версиями. Коммуникации Google по поводу дорожной карты были непостоянными, а некоторые рендереры (Lit, Flutter) отставали от обновлений спецификации. Планируйте время на изменения в спецификации, если вы используете её сегодня. Для чисто веб-приложений, json-render на данный момент имеет более полные инструменты. Долгосрочное преимущество A2UI — доступность на разных платформах (веб, Flutter, SwiftUI, Android), которой не имеет json-render.

### Приложения MCP

**GitHub**: [modelcontextprotocol](https://github.com/modelcontextprotocol) · Связанные: [mcp-ui](https://github.com/MCP-UI-Org/mcp-ui)

MCP изначально был протоколом для подключения LLM к инструментам и данным. Расширение Apps позволяет инструментам MCP возвращать не только данные, но и интерактивные UI-артефакты: компоненты React, формы, дашборды, карты.

Модель безопасности строгая, и это намеренно: все отображается в изолированном iframe с ограниченными разрешениями, шаблоны предварительно объявлены, чтобы хост-приложение могло их проверить, а вся коммуникация осуществляется через аудируемый JSON-RPC. Это правильная модель для поставщиков инструментов — сервер Shopify MCP может возвращать виджет оформления заказа; сервис карт может возвращать встраиваемую карту. Хост-приложение не владеет и не доверяет коду этого виджета.  

MCP Apps — правильный выбор, когда интерфейс *принадлежит поставщику инструментов*, а не вашему приложению. Для интерфейса, который находится в домене вашего приложения, используйте паттерн 1 или 2.

## JavaScript/TypeScript-фреймворки

### CopilotKit

**GitHub**: [CopilotKit/CopilotKit](https://github.com/CopilotKit/CopilotKit) · Примеры: [CopilotKit/generative-ui](https://github.com/CopilotKit/generative-ui)

CopilotKit — это наиболее полный фреймворк для приложений, ориентированных на агентов. Он охватывает полный жизненный цикл: подключение к backend-ам агентов через AG-UI, управление двусторонним состоянием разговора, рендеринг генеративных компонентов интерфейса и предоставление инфраструктуры управления общим состоянием, позволяющей агентам и пользователям изменять одни и те же данные.

Трехуровневая модель четко отображается в API CopilotKit:
- `useCopilotAction` с callback `render` → Паттерн 1
- Рендеринг A2UI/Open-JSON-UI → Паттерн 2
- Изолированные артефакты `OpenGenerativeUI` → Паттерн 3

Важная, но недостаточно обсуждаемая особенность CopilotKit — это **общее состояние и человек в цикле**: агент может читать и записывать состояние приложения, пользователь также может читать и записывать его, а изменения передаются в двустороннем режиме. Именно это делает интерфейсы в стиле copilot ощущаться как настоящее сотрудничество, а не чат-окно, прикреплённое к продукту.

### Vercel AI SDK

**GitHub**: [vercel/ai](https://github.com/vercel/ai) · Docs: [ai-sdk.dev](https://ai-sdk.dev/)

Vercel AI SDK — это фактический стандарт TypeScript для приложений с ИИ. В контексте генеративных интерфейсов:

**`useObject`** потоком передает структурированный JSON-объект с сервера по мере его генерации. Вы определяете схему Zod; SDK анализирует частичный JSON и запускает перерисовки по мере поступления полей. Это самый плавный путь к Паттерну 2 в приложении Next.js.

```tsx
const { object: dashboard } = useObject({
  api: "/api/generate-dashboard",
  schema: z.object({
    title: z.string(),
    metrics: z.array(z.object({ label: z.string(), value: z.number() })),
    insights: z.array(z.string()),
  }),
});
```

**`useChat` с обработчиками инструментов** → Паттерн 1. Модель вызывает инструменты; вы сопоставляете имена инструментов с компонентами.

**AI Elements** ([elements.ai-sdk.dev](https://elements.ai-sdk.dev/)) предоставляет готовые примитивы интерфейса для использования с SDK.

**Примечание по запутанному развитию**: В октябре 2024 года Vercel объявил в [обсуждении GitHub #3251](https://github.com/vercel/ai/discussions/3251), что AI SDK RSC — паттерн потоковой передачи React-компонентов сервера (React Server Components), представленный как ключевая функция "Генеративный интерфейс" в SDK 3.0 — была приостановлена бессрочно из-за "нескольких долгосрочных ограничений", для которых нет хороших решений в ближайшей перспективе. Команды, построившие стратегии продуктов вокруг потоковой передачи RSC, оказались в непростой ситуации. API `generateObject`/`streamObject` позже были устаревшими в SDK 6.0. Рекомендуемая миграция с AI SDK RSC — паттерн `useObject`, описанный выше, или `json-render` для генерации на основе каталога.

### assistant-ui

**GitHub**: [assistant-ui/assistant-ui](https://github.com/assistant-ui/assistant-ui)

assistant-ui — это набор составных React-примитивов для создания чат-интерфейсов промышленного качества. Это правильный выбор, когда вам нужен отшлифованный чат-UX: пузыри сообщений, потоковые токены, действия копирования/редактирования/перегенерации, состояния размышления — и вы хотите использовать свой собственный бэкенд и собственную логику рендеринга инструментов.  

Он хорошо работает в сочетании с любым бэкендом (OpenAI, Anthropic, локальные модели, пользовательские эндпоинты) и обрабатывает рендеринг вызовов инструментов через знакомую модель слота/рендер-пропа.  

### json-render

**GitHub**: [vercel-labs/json-render](https://github.com/vercel-labs/json-render) · Документация: [json-render.dev](https://json-render.dev/)

json-render реализует паттерн 2 с четко определенной позицией и подходом «все включено». Вы получаете готовый каталог компонентов (компоненты shadcn/ui с схемами Zod), рендерер и плотный цикл генерации, где модель ограничена каталогом схемой.

Отличительные особенности:
- **Многоцелевой рендеринг**: одна и та же JSON-спецификация может рендериться в веб-приложение на React, мобильное приложение на React Native, PDF, HTML-письмо или видео в Remotion. Это действительно полезно для отчетов.
- **Прогрессивный рендеринг**: компоненты отображаются по мере их потоковой передачи моделью, а не после получения всей спецификации.
- **Строгие схематические ограничения**: каталог спроектирован так, что модель не может выдумывать допустимые, но неизвестные типы компонентов.

Если вы разрабатываете функцию доски мониторинга или генерации отчетов и хотите обойти инфраструктурные работы по созданию собственного каталога, json-render — это самый быстрый путь для веб-приложений.  

**По динамике**: json-render был запущен из Vercel Labs в начале 2026 года и быстро привлек внимание веб-разработчиков, потому что он сразу полезен в стандартных проектах React/Next.js. При этом json-render всё ещё находится в предверсии 1.0, а взаимосвязь между json-render и A2UI ещё не урегулирована — Vercel экспериментировал с выходными данными, совместимыми с A2UI, поэтому схождение возможно. Для кроссплатформенных решений (нativaльное мобильное приложение, несколько фреймворков) A2UI — лучшая долгосрочная ставка.  

### Hashbrown

**GitHub**: [liveloveapp/hashbrown](https://github.com/liveloveapp/hashbrown)

Hashbrown использует уникальный подход: вместо создания отдельного AI-интерфейса он встраивает выбор компонентов ИИ напрямую в ваше существующее приложение React или Angular. Вы предоставляете компоненты вашего приложения модели LLM; модель LLM выбирает, какие из них отображать, и может вызывать клиентские инструменты.

Это правильный инструмент, когда вы хотите встроить интеллект в интерфейсы продуктов, которые не являются «чатами» — страница продукта, адаптирующая свою компоновку, панель настроек, отображающая правильные опции, редактор рабочего процесса, предлагающий следующий шаг.

### OpenUI

**GitHub**: [thesysdev/openui](https://github.com/thesysdev/openui) · Docs: [openui.com](https://www.openui.com/)

OpenUI заменяет JSON форматом, похожим на код, с ориентацией на строки ("OpenUI Lang"), который разработан для прогрессивного рендеринга и эффективности токенов. Утверждается, что он требует примерно на 67% меньше токенов, чем эквивалентный JSON для сложных компоновок.

Платой является зрелость экосистемы — OpenUI новее, а инструменты менее развиты, чем у JSON-подходов. Но если стоимость токенов является значимым ограничением и вы генерируете сложные компоновки с высокой частотой, эффективность формата реальна.

### Tambo

**GitHub**: [tambo-ai/tambo](https://github.com/tambo-ai/tambo)

Tambo фокусируется на выборе компонентов с сохранением состояния: ИИ выбирает компоненты и взаимодействует с ними через инструменты на стороне клиента, поддерживая состояние компонентов в течение диалога. Хорошо подходит для сценариев, где элементы интерфейса сохраняются между ходами — компонент фильтра, который пользователь настраивает, пока ИИ продолжает анализировать отфильтрованные данные.

## Python-слой

Экосистема Python подходит к интерфейсам ИИ иначе. Эти инструменты оптимизированы для демонстраций моделей ML, приложений с данными и внутренних инструментов — не для потребительских приложений в производственной среде с агент-ориентированной композицией макета.  

Это не критика. Для правильных сценариев использования Gradio и Streamlit — единственные инструменты, которые вам нужны.  

### Gradio

**GitHub**: [gradio-app/gradio](https://github.com/gradio-app/gradio) · PyPI: `gradio`

Ценность Gradio: вы пишете функцию на Python, а Gradio оборачивает её в веб-интерфейс. Класс `Interface` занимает 3 строки для классификатора изображений. `ChatInterface` — 10 строк для чат-бота. `Blocks` даёт вам тонкий контроль над компоновкой, когда это необходимо.

«Генеративный UI» в Gradio определяется разработчиком на Python, а не моделью. Видимость и конфигурация компонентов могут динамически меняться в зависимости от выходных данных модели, но каталог компонентов статичен — вы не просите модель составлять компоновку.

Gradio — это стандартный выбор для HuggingFace Spaces и экосистемы демонстраций ML. У него миллионы загрузок в месяц, и он обеспечивает значительную долю ландшафта демонстраций ИИ.

**Используйте Gradio, когда**: вы — разработчик на Python, создающий демонстрацию ML-модели, исследовательский прототип или внутренний инструмент, и не хотите иметь дело с JavaScript.

### Streamlit

**GitHub**: [streamlit/streamlit](https://github.com/streamlit/streamlit)

Модель Streamlit более строгая: Python-скрипт выполняется полностью при каждом взаимодействии. Вы вызываете `st.chat_message()`, `st.dataframe()`, `st.plotly_chart()`. Фреймворк отвечает за макет.

Модель перезапуска всего скрипта звучит как неэффективно, но оказывается неожиданно эргономичной для чат-ботов на ИИ, которые накапливают историю разговора — весь скрипт перезапускается, история чата хранится в состоянии сессии, а выход детерминирован. Streamlit теперь имеет встроенную поддержку большинства основных поставщиков LLM и интегрируется нативно с Snowflake Cortex.

**Используйте Streamlit, когда**: вы создаете приложение с ИИ, внутренний инструмент отчетности или дашборд с поддержкой машинного обучения на Python и хотите максимально простой путь для развертывания.

### LangChain и Haystack

Это фреймворки оркестрации backend, а не фреймворки интерфейса. Они появляются на любой честной карте генеративных UI-стеков, потому что обычно это уровень, где генерируются структурированные выходные данные перед отправкой на фронтенд.

**LangChain** ([langchain-ai/langchain](https://github.com/langchain-ai/langchain)): `.with_structured_output()` для любого LLM позволяет генерировать JSON с ограничениями Pydantic. Декоратор `@tool` с автоматической генерацией схемы — это самый чистый способ определить, какие инструменты может вызывать модель. LangChain передаёт структурированные результаты в любой используемый вами фронтенд.

**Haystack** ([deepset-ai/haystack](https://github.com/deepset-ai/haystack)): модульная архитектура конвейеров с сильной поддержкой RAG. Hayhooks оборачивает конвейеры Haystack в HTTP-конечные точки — включая MCP-совместимые. Если ваш генеративный UI требует основу на извлечении данных, архитектура конвейеров Haystack обрабатывает это аккуратно.

Ни один из этих фреймворков не владеет уровнем пользовательского интерфейса. Они генерируют данные, которые ваш фронтенд (Паттерн 1, 2 или 3) рендерит.

## Скорость экосистемы и непрочная почва

Это пространство быстро развивается, и несколько проектов сопровождаются запутанной коммуникацией вместе с выпуском кода. Последняя проверка: 8 мая 2026 года; рассматривайте здесь заметки о статусе проекта как временную информацию, а не постоянный вердикт.

**Vercel AI SDK RSC** была ключевой функцией Generative UI при выпуске SDK 3.0. Vercel приостановила её развитие в октябре 2024 года ([Обсуждение #3251](https://github.com/vercel/ai/discussions/3251)), указав на архитектурные ограничения React Server Components, которые не имели решения в ближайшие сроки. Команды, построившие проекты на этом, естественно разочарованы. Функция всё ещё присутствует в документации, но не является рекомендуемым путём; рекомендуется `useObject`.

**json-render** (Vercel Labs) — новое направление, независимая от фреймворка альтернатива на основе каталога, избегающая проблемы привязки к RSC. Это предварительная версия, и она уже вызывает сильный интерес среди разработчиков React/web. Вероятная причина удобства разработки: json-render немедленно полезен в стандартном проекте React/Next.js, тогда как кросс-платформенный охват A2UI добавляет сложности настройки. Как это сложится, когда оба спецификации зрелееют, остаётся неясным. Vercel исследовала совместимость A2UI в json-render, что указывает на возможность схождения.

**A2UI** (Google) находится в предварительной версии (v0.9 на момент последней проверки) с разрывающими изменениями между минорными версиями и несогласованными коммуникациями Google по поводу дорожной карты. Это правильный выбор для кросс-платформенного охвата (веб + Flutter + SwiftUI), которого json-render не охватывает, и имеет значительную корпоративную поддержку. Для чисто веб-проектов в настоящее время DX хуже.  

**AG-UI** (CopilotKit) также находится в предварительной версии. Наиболее распространённая путаница: название заставляет думать, что это фреймворк UI. Это не так — это транспортный протокол. AG-UI определяет, как события передаются между backend-ами агентов и вашим frontend; что вы рендерите в ответ — всё ещё ваше решение. Эта модель мышления твёрда и широко принята, но предварительная спецификация означает, что граничные случаи всё ещё вырабатываются.  

Практический вывод: **все основные игроки здесь находятся в предварительной версии**. Планируйте изменения API. Паттерны — инструмент-в-компонент, каталог-составление, санкционированная генерация — достаточно стабильны для построения на их основе. Конкретные выборы протоколов не являются такими.

## Проектирование каталога компонентов: Настоящая инженерная работа

Большая часть интересной сложности во Втором паттерне не в рендере — а в каталоге.

Каталог — это **принятие продукта, закодированное как схема**. Он отвечает: какие значимые объекты интерфейса существуют в этой области? Не «какие существуют компоненты React?», а «что пользователь в данном контексте действительно должен видеть и с чем взаимодействовать?»  

**Режим сбоя из-за излишней детализации**: вы предоставили `Row`, `Column`, `Text`, `Button`, `Icon`. Теперь модель должна быть фронтенд-инженером. Она будет генерировать посредственный макет, не соответствующий вашей системе дизайна, пропустит пустые состояния, создаст неудостовлетворительную разметку и будет менять подход от ответа к ответу, потому что ничего в каталоге не ограничивает вывод визуальным языком вашего продукта.  

**Режим сбоя из-за недостаточной детализации**: вы предоставили `WeatherCard`, `FlightCard`, `HotelCard`. Модель не сможет адаптироваться, когда пользователь запрашивает что-то, что не соответствует заранее приготовленной карточке. Она вернётся к тексту.

**Полезный средний уровень**: компоненты на уровне домена с ограниченными слотами.

Каталог путешествий может выглядеть так:

```
TripSummary         — обзор маршрута
FlightOptionList    — выбираемые варианты рейсов с ценами
HotelComparison     — горизонтальное сравнение карточек отелей
TravelerForm        — сбор данных о путешественнике
PolicyNotice        — уведомление о правилах/тарифах
BookingConfirmation — финальное подтверждение с кнопкой действия
```

Каталог финансового приложения может выглядеть так:

```
PortfolioSnapshot   — ключевые позиции и прибыль/убытки
TransactionTable    — фильтруемая, разбитая на страницы таблица транзакций
RiskBreakdown       — распределение и метрики волатильности
ScenarioComparison  — сравнение сценариев
ApprovalGate        — действие, требующее подтверждения
```

Каталог компонентов кодирует вашу лексику продукта. Он фиксирует решения по UX, требования к доступности, обработку пустых состояний и паттерны опасных действий в коде компонентов. Модель может комбинировать эти элементы. Вы всё ещё решаете, как выглядит каждый элемент и какие действия он может выполнять.

**Правила проектирования схемы, снижающие галлюцинации**:

1. Делайте значения enum короткими и очевидными. `"тип": "диаграмма-столбик"` вместо `"тип": "вид-визуализации-столбик-вертикальный"`.
2. Исключайте недопустимые комбинации. Если `PolicyNotice` может появляться только в конце макета, не включайте его в ту же часть схемы, где размещаются элементы, допустимые в любом месте.
3. Используйте обязательные поля по умолчанию. Необязательное поле — это поле, которое модель может пропустить, а ваш рендерер должен обработать как null.
4. Тестируйте каталог на реальных запросах перед выпуском. Сохраняйте сгенерированные спецификации; проверяйте их на нарушения схемы, галлюцинации значений полей и комбинации, которые технически корректны, но семантически неправильны.

## Частые ловушки

**Ловушка: воспринимание валидного JSON как безопасного поведения.** Проверка схемы подтверждает структуру. Она ничего не говорит о том, соответствует ли действие, прикреплённое к кнопке, её метке, соответствует ли итоговое значение данным, из которых оно получено, или о том, что компонент интерфейса делает что-то, чего пользователь не ожидал. Спецификации сгенерированного интерфейса требуют семантического обзора, а не только проверки схемы. Как минимум, разрушительные действия должны требовать компонент подтверждения, а метки на этих компонентах должны проверяться на соответствие действиям, которые они инициируют.

**Попадка: использование дизайн-примитивов вместо продуктовых примитивов.** Если модель должна решать, использовать ли отступ в 16px или 20px, вы предоставили ей неправильный уровень абстракции. Доменные компоненты должны кодировать вкус продукта. Модель должна составлять поведение, а не управлять деталями представления.

**Попадка: использование генеративного интерфейса там, где подойдёт статический.** Если структура того, что нужно показать, известна на этапе разработки — а это обычно так — Паттерн 1 с готовыми компонентами работает быстрее, безопаснее и более последовательно. Генеративный интерфейс оправдывает свою сложность, только когда структура действительно меняется в зависимости от данных или контекста задачи.

**Попадка: игнорирование доступности.** Языковые модели создают нарушения WCAG. Они добавляют `role="region"` к интерактивным элементам, генерируют формы без меток и производят соотношения контрастности, не проходящие WCAG AA. Ваша библиотека компонентов может быть полностью доступной; композиции этих компонентов, сгенерированные ИИ, автоматически доступными не становятся. Тестируйте полный путь рендеринга, а не только отдельные компоненты.

**Попадание: смешение протокола и фреймворка.** AG-UI не является фронтенд-фреймворком. A2UI не является библиотекой React. Это форматы передачи данных и протоколы событий. Вам всё равно нужен фронтенд-фреймворк для их реализации. CopilotKit реализует AG-UI и A2UI. json-render реализует паттерн каталога A2UI/Open-JSON-UI. Это разные уровни.

## Рекомендации по случаю использования

**Добавление копилота в существующее приложение SaaS**: Начните с паттерна 1 (tool-to-component). Используйте Vercel AI SDK `useChat` или CopilotKit. Сопоставьте свои 5–10 основных действий агента с предварительно созданными компонентами. Выпустите это, измерьте, а затем расширьте каталог только в случае, если пользователи явно продемонстрируют потребность в более богатом составе.  

**Генерация дашбордов с помощью естественного языка**: Используйте паттерн 2 с json-render или пользовательским каталогом A2UI. Определите каталог из 8–15 типов компонентов, охватывающих ваши типы диаграмм, карточки метрик и варианты таблиц. Передайте схему модели; пусть она составит макет. Постройте проверку, которая ловит неизвестные типы до их попадания в рендерер.  

**Фронтенд с несколькими агентами**: Используйте CopilotKit с AG-UI. Поток событий обрабатывает потоковую передачу в реальном времени между бэкендами агентов; общее состояние управляет передачей между агентами; паттерн HITL обрабатывает точки одобрения.

**Построение внутри ChatGPT или другого хоста MCP**: Используйте MCP Apps. Определите ваш инструмент как инструмент для получения данных и логических выводов, и отдельный рендер-инструмент, запрашивающий виджет. Выносите бизнес-логику из шаблона виджета.  

**Демо ML-моделей и данные-приложения (Python-команда)**: Gradio для демонстраций и HuggingFace Spaces. Streamlit для данных-приложений со сложным взаимодействием. Ни один из них не требует JavaScript.  

**Визуальные артефакты, симуляции, диаграммы**: Используйте Паттерн 3 (OpenGenerativeUI или эквивалент). Установите строгую CSP для iframe. Обрабатывайте вывод как ненадежный пользовательский контент с точки зрения безопасности.

Фреймворки быстро созревают. Сходимость протоколов (AG-UI для потоковой передачи, A2UI/Open-JSON-UI для спецификаций каталога) всё ещё в процессе, но форма становится достаточно чёткой для построения на этом.  

Наиболее важные инженерные задачи в настоящее время — это не выбор фреймворка. Это проектирование каталога — определение того, что модель может говорить, что требует ясности продукта больше, чем технических навыков. Это семантическая проверка — тестирование того, что сгенерированный интерфейс делает то, что он утверждает, а не только проходит проверку схемы. И это зазор в доступности — создание каталогов, где каждый компонент и каждая композиция компонентов соответствует стандарту доступности, который вы бы требовали от вручную написанного интерфейса.

Модель будет делать то, что вы ей скажете, в рамках грамматики, которую вы ей зададите. Сделайте грамматику осознанной.
