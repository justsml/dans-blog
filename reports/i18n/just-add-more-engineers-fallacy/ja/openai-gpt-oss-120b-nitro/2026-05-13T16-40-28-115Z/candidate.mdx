## n(n-1)/2

それはすべてのエンジニアリングマネージャーの腕に入れ墨すべき式だ。締め切り問題の解決策ではない。あなたの提案が機能しない理由を示す数学的証明だ。

あの会議にいたことがあるだろう。誰かが必ず「期限に間に合わせるために、**もっと開発者を増やす**だけだ」と提案する会議だ。スプレッドシートはとてもシンプルに見える。2人のエンジニア × 3か月 = 6エンジニア‑月の作業量、だから6人のエンジニアが1か月で終えるはず、というわけだ。VPはうなずき、PMはタイムラインを更新し、皆は生産的だと感じる。

しかし、数学はそうは働かず、これまでずっとそうだった。

## What Nobody Tells You About Communication Paths

チームに5人のエンジニアがいると、互いに話せる組み合わせは10通りある。さらに5人増やすと、管理すべきコミュニケーションチャネルは一気に45通りになる。これが式の働きだ：n(n-1)/2、ここで n はチーム規模である。

これは単なる些細な不便ではない。「昼食時に話し合う」から「ステークホルダーと会議を設定する」への違いだ。チームのキャパシティを単に倍増させたわけではない。むしろ、全員を整合させるためのオーバーヘッドを四倍以上に膨らませたことになる。

人数が20人に達した時点で、管理すべき潜在的な会話は190通りになる。スタンドアップはもはや手早い同期ではなくなる。アーキテクチャの決定には合意形成が必要になる。マージコンフリクトは、同一機能に対する競合するビジョンを掘り起こす考古学的作業に変わる。

Microsoft の DevDiv チームが 2008 年に実施した調査では、協調コストが至る所に現れることが不快なほど明確に示された。どのモジュールを誰が所有するか？誰がこの会議に参加すべきか？今週だけでマージコンフリクトが三度起きた理由は何か？これらの質問はすべて、エンジニアが本来の締め切り達成に費やすべき時間を奪う実時間である。

さらに、Forsgren、Humble、Kim の *Accelerate* に掲載された研究は、規模が小さく自律性の高いチームが速度と品質の両面で大規模チームを一貫して上回ることを示している。協調の税金は単なる概念ではなく、測定可能で高コストである。

## 時間がない三か月間の投資

しかし、協調コストを支払う覚悟があるとしよう。予算も人員も確保でき、リソースを投入する準備ができている。素晴らしい。次にやるべきことは、そのリソースが実際に効果を発揮するまで、三〜六か月待つことだ。

新しく採用したシニアエンジニア（シニアを雇うんですよね？ ジュニアだとさらに立ち上げに時間がかかります） は、最初の一週間を開発環境の構築に費やします。その後、アーキテクチャを理解しなければなりません。「マイクロサービスを使っている」だけでなく、*なぜ*そのようにサービスを分割したのか。「ここでRedisを使っている」だけでなく、二年前のインシデントでRedisを選択した経緯も把握しなければなりません。

この間、既存チームはメンターシップの税金を支払っています。すべての質問は正当で必要です。「なぜ認証フローをこの形にしたのか？」 「請求サービスの所有者は誰か？」 「2019年のTODOはまだ有効か？」 それぞれが誰かの作業を中断させます。UC Irvine の Gloria Mark の研究によれば、割り込み後にタスクに戻るまで平均23分かかります。もし新しいエンジニア5人が1日5つずつ質問すると、締め切りに向けて作業すべき人々の集中時間が何時間も失われる計算です。

Camille Fournier は *The Manager's Path* で、シニアエンジニアが複雑なコードベースで完全に生産的になるまでに 3〜6か月かかると述べています。これはすべてが順調に進んだ場合です。ドキュメントが最新で、部族知識がアクセス可能で、ドメインが特に奇妙でない場合に限ります。ほとんどのコードベースはこの条件を満たしていません。

3か月目は特に危険です。新しいエンジニアは自信を持ち、コードを書き始め、生産的になります。しかし、支払いフローの奇妙なエッジケースや、密かに置き換え作業を進めているレガシーモジュール、金曜日にデプロイしないと決めたインシデントの教訓などを知らないままです。彼らのコードの一部はリファクタリングが必要になるでしょうし、インシデントを引き起こす可能性もあります。すべてのコードは、締め切りに向かうべき同じ人々による徹底的なレビューが求められます。

## ブルックスの法則の不快な連続性

Fred Brooks は 1975 年にこのテーマを書きました。*The Mythical Man‑Month* はシンプルな真実を捉えています。「遅れているソフトウェアプロジェクトに人員を増やすと、さらに遅くなる」。本書は IBM の OS/360 プロジェクトでの経験に基づき、次の印象的な比喩を紹介しています。女性一人が赤ん坊を産むのに九か月かかるが、九人の女性でも赤ん坊を一か月で産むことはできない、というものです。

5️⃣0年後、ツールは進化しました。CI/CD パイプラインは統合コストの一部を削減します。モダンなプルリクエストワークフローは問題を早期に捕捉します。*IEEE Transactions on Software Engineering* の 2019 年の研究によれば、優れた CI/CD 実践は統合コストを最大 40% 削減できることが示されています。非同期コミュニケーションツールは時差を超えて調整を可能にし、適切に設計されたマイクロサービスはチーム間の依存関係を減らします。

しかし人間はアップグレードされていません。コンテキストが必要です。何をしたかだけでなく、なぜその決定がなされたのかを理解する必要があります。O(n²) は意見ではなく数学です。その式の定数項は最適化できても、基本的な増加率からは逃れられません。ドメイン知識の習得には時間がかかります。どんなに優れたオンボーディングドキュメントを用意しても同様です。

技術は変わりますが、数学は変わりません。

## Amazon は 2000 年代初頭にこれを知っていた

Jeff Bezos は Amazon で「two‑pizza team」ルールを導入しました。チームが二枚のピザで食べきれないなら、規模が大きすぎるということです。最適な人数は 5〜8 人でした。このルールはピザ予算や奇抜さを示すためのものではありません。小規模チームはステークホルダーが少ないため意思決定が速く、経路が少ないためコミュニケーションが効率的です。また、所有権が明確になるため、より良いシステムを構築できます。

*Team Topologies* において、Matthew Skelton と Manuel Pais は、7〜9 人のチームが能力と調整のバランスを最適化することを示す研究を引用しています。その閾値を超えると、出力の伸びが鈍くなるにもかかわらず、調整コストが指数的に増大します。数学は、認識しようとしようと、必ず追いついてきます。

## 実際に効果があること

では、プロジェクトが本当に遅れていて、締め切りが本当に重要な場合、何をすべきでしょうか？

スコープを削減します。ビジネス側が嫌がるのは承知の上です。しかし、Marty Cagan が *Inspired* で書いているように「現実として、我々のアイデアの少なくとも半分は実現不可能です」。MVP を出荷し、余計な機能は延期します。バニティメトリクスのダッシュボードなしでリリースし、v2 で改善を繰り返すことができます。

不要な会議は廃止します。承認プロセスを簡素化します。チームに実際の中断なしの集中時間を与えます。新機能のリクエストにはノーと言いましょう。トヨタ生産方式には「無駄を取り除く」という概念があり、時には最速の前進手段は、足を引っ張る作業を止めることです。

時間枠を徹底的に設定し、手元にあるものを出荷します。Facebook がローンチしたときは「Thefacebook」という名前で、ハーバードだけで動作していました。AWS が S3 をリリースしたときは、数本の API 呼び出しだけでした。Paul Graham が *Hackers & Painters* で述べているように「少数のユーザーに本当に愛される方が、多数のユーザーにまあまあ好きでもらうより効果的です」。最もシンプルで機能するものを出荷しましょう。

どうしても人員を増やす必要がある場合は、現在のプロジェクトを救うためではなく、次のプロジェクトのために採用します。3〜6 か月の立ち上がり期間を受け入れ、ペアプログラミングやメンターシップに投資します。決定理由を *why* として文書化し、何を決めたかだけでなくなぜそうしたかも残します。Brooks の法則には派生があり、プロジェクトの *早い段階* で人を増やすと、締め切り直前のオンボーディングコストを先に支払うことで効果が得られます。

## なぜこの誤謬が残るのか

「エンジニアをもっと増やすだけ」という誤謬が残るのは、行動を起こしたように見えるからだ。ステークホルダーに対して緊急性を示すことができ、スプレッドシート上でも見栄えが良い。瞬間的には感情的に満足できる。

しかしソフトウェア開発は組み立てラインの作業ではない。作業員を倍にすれば出力も倍になるようなウィジェットの大量生産とは違う。私たちが扱うのは相互に結びついた問題であり、コンテキストが重要で、コミュニケーションが重要で、調整コストはチーム規模とともに二次関数的に増加する実際のコストである。

数式は明快だ。5人なら通信経路は10本、10人なら45本、20人なら190本になる。オンボーディングには数週間ではなく数か月かかる。調整コストは追加される人一人ひとりに課される税金だ。これは会議室で議論できる意見ではなく、測定可能な現実であり、あなたが信じようと信じまいとベロシティチャートに現れる。

次に誰かが遅延プロジェクトにエンジニアを増やすことを提案したとき、何が起こるか分かるだろう。プロジェクトが遅れるのは日単位や週単位ではない。

月単位で遅れることになる。そして多くのエンジニアリング課題と違い、これをリファクタリングで解決することはできない。
