    ```rust
          fn take_knowledge(knowledge: String) {
              println!("Knowledge: {}", knowledge);
          }

          fn main() {
              let wisdom = String::from("know thyself");
              take_knowledge(wisdom);

          }
    ```
  </div>

    1. Pasar por referencia (tomar prestado el valor):
    ```rust
          fn borrow_it(text: &String) {
              println!("Inside: {}", text);
          }
          borrow_it(&wisdom);  // Now wisdom can be used after
    ```

    ```rust
          take_knowledge(wisdom.clone());  // Original wisdom remains valid
    ```

    ```rust
          fn take_and_return(text: String) -> String {
              println!("Inside: {}", text);
              text  // Return ownership back
          }
          let wisdom = take_and_return(wisdom);  // Reassign returned ownership
    ```

    - Referencias: Más eficientes, pero requieren gestión de tiempos de vida
    - Clonación: Simple pero potencialmente costosa
    - Devolver la propiedad: Útil para transformar valores

    Mejor práctica: Usa referencias a menos que necesites transferencia de propiedad.

    ```rust
          fn main() {
              let mut wisdom = String::from("He who laughs at");
              let ref1 = &mut wisdom;  // First mutable borrow
              let ref2 = &mut wisdom;  // Second mutable borrow
              ref1.push_str(" himself never runs");
              ref2.push_str(" out of things to laugh at.");
          }
    ```
    Piensa en las reglas de Rust para referencias mutables.
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Este código viola las reglas fundamentales de préstamo de Rust:
    - Solo UNA referencia mutable a un valor a la vez
    - O cualquier número de referencias inmutables
    - Las referencias no pueden sobrevivir a su referente

    Así es como se corrige el código:

    1. Usa ámbito secuencial:
    ```rust
          let mut wisdom = String::from("He who laughs at");
          {
              let ref1 = &mut wisdom;
              ref1.push_str(" himself never runs");
          }  // ref1 goes out of scope
          let ref2 = &mut wisdom;  // Now this is valid
          ref2.push_str(" out of things to laugh at.");
    ```
    2. O modifica el string en un solo préstamo:
    ```rust
          let mut wisdom = String::from("He who laughs at");
          let ref1 = &mut wisdom;
          ref1.push_str(" himself never runs out of things to laugh at.");
    ```
    These rules prevent data races at compile time, making Rust thread-safe by default.

    Common pitfall: Trying to use multiple mutable references to avoid cloning or
    to modify different parts of the same value simultaneously.

  <slot name="question">

    ¿Compilará este código? Si es así, ¿por qué? Si no, ¿qué está mal?
    ```rust
          fn first_word(s: &str) -> &str {  // No explicit lifetimes?
              match s.find(' ') {
                  Some(pos) => &s[0..pos],
                  None => s,
              }
          }

          fn main() {
              let name = String::from("Seneca the Younger");
              let first = first_word(&name);
              println!("Hello, {}", first);
          }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Este código compila correctamente gracias a las reglas de elisión de tiempos de vida de Rust.
    Estas reglas permiten al compilador inferir automáticamente los tiempos de vida en patrones comunes.

    Las tres reglas de elisión de tiempos de vida son:
    1. Cada parámetro obtiene su propio parámetro de tiempo de vida
    2. Si hay exactamente un parámetro de tiempo de vida de entrada, ese tiempo de vida se asigna a todos los parámetros de tiempo de vida de salida
    3. Si hay múltiples parámetros de tiempo de vida de entrada, pero uno de ellos es &self o &mut self, el tiempo de vida de self se asigna a todos los parámetros de tiempo de vida de salida

    Esta función es equivalente a:
    ```rust
          fn first_word<'a>(s: &'a str) -> &'a str {
              // ... same implementation
          }
    ```
    Patrones comunes donde funciona la elisión:
    ```rust
          // These don't need explicit lifetimes
          fn get_str(s: &str) -> &str { s }
          fn get_first(s: &str) -> &str { &s[0..1] }

          // These would need explicit lifetimes
          fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
              if x.len() > y.len() { x } else { y }
          }
    ```
    Mejor práctica: Deja que la elisión trabaje para ti cuando sea posible, pero entiende cuándo se necesitan tiempos de vida explícitos.
  </div>
  </slot>

    ¿Qué problema tiene esta definición de tipo recursiva?
    ```rust
          #[derive(Debug)]
          enum CatList {
              Cons(i32, CatList),  // Recursive without indirection
              Nil,
          }

          fn main() {
              let catlist = CatList::Cons(1,
                  CatList::Cons(2,
                      CatList::Cons(3,
                          CatList::Nil)));
          }
    ```
  </div>
  </slot>
  <slot name='explanation'>
  <div className="explanation">
    Este código falla porque el compilador no puede determinar el tamaño de `CatList` en tiempo de compilación. ¡La naturaleza recursiva del tipo significa que podría ser infinitamente grande!

    Así es como se soluciona usando `Box<T>`:
    ```rust
          #[derive(Debug)]
          enum CatList {
              Cons(i32, Box<CatList>),  // Box provides a fixed-size pointer
              Nil,
          }

          fn main() {
              let catlist = CatList::Cons(1,
                  Box::new(CatList::Cons(2,
                      Box::new(CatList::Cons(3,
                          Box::new(CatList::Nil))))));
          }
    ```
    Por qué funciona `Box<T>`:
    1. Box proporciona un puntero de tamaño fijo (generalmente 8 bytes en sistemas de 64 bits)
    2. Los datos reales se almacenan en el heap
    3. El compilador ahora sabe exactamente cuánto espacio asignar

    Casos de uso comunes para `Box<T>`:
    - Estructuras de datos recursivas (listas enlazadas, árboles)
    - Datos grandes que quieres asegurar que se asignen en el heap
    - Objetos trait cuando necesitas despacho dinámico

    Mejores prácticas: Usa `Box<T>` cuando necesites:
    - Tipos recursivos
    - Asegurar asignación en el heap
    - Mover datos grandes sin copiar

  </slot>

    Desglosemos cómo funciona Rc:

    1. Creación inicial con `Rc::new()`: count = 1
    2. Primer clon para `marcus`: count = 2
    3. Segundo clon para `aurelius`: count = 3

    Características importantes de Rc:
    ```rust
          use std::rc::Rc;
      
          fn demonstrate_rc() {
              let original = Rc::new(String::from("Shared"));
              println!("Count after creation: {}", Rc::strong_count(&original)); // 1
          
              {
                  let copy = Rc::clone(&original);
                  println!("Count inside scope: {}", Rc::strong_count(&original)); // 2
              } // copy is dropped here

          }
    ```

    - Rc::clone() es barato - solo incrementa un contador
    - Rc es solo para escenarios de un solo hilo
    - Cuando se elimina la última referencia, los datos se limpian
    - Usa referencias Weak para prevenir ciclos de referencia

    Buenas prácticas:
    - Usa Rc cuando necesites propiedad compartida
    - Considera Arc para escenarios seguros entre hilos
    - Evita crear ciclos de referencia

    ¿Qué sucede con esta función que devuelve la más larga de dos slices de cadena?
    ```rust
          fn longest(text1: &str, text2: &str) -> &str {
              if text1.len() > text2.len() {
                  text1    // Returning a reference, but which lifetime?
              } else {
                  text2    // Could be this reference instead
              }
    ```

    ```rust
          // Correct version with explicit lifetime annotation
          fn longest<'a>(text1: &'a str, text2: &'a str) -> &'a str {
              if text1.len() > text2.len() {
                  text1
              } else {
                  text2
              }
          }

          // Alternative with different lifetimes
          fn longest_flex<'a, 'b>(text1: &'a str, text2: &'b str) -> &'a str {
    ```

    1. Varias referencias de entrada podrían tener diferentes tiempos de vida

    ```rust

          fn first_word(s: &str) -> &str { /* ... */ }

          fn combine<'a, 'b>(s1: &'a str, s2: &'b str) -> String { /* ... */ }
    ```
    Mejores prácticas:
    1. Deja que la elisión de tiempos de vida funcione cuando sea posible
    2. Usa tiempos de vida explícitos cuando las relaciones necesiten ser claras
    3. Considera devolver tipos con propiedad para evitar la complejidad de los tiempos de vida

    ```rust
          use std::cell::Cell;

          fn main() {

              let meaning = &life;        // Shared reference
              println!("{}", life.get()); // What prints here?
    ```

  <slot name='explanation'>
  <div className="explanation">
    Cell y RefCell sirven para diferentes propósitos de mutabilidad interior:
    ```rust
          use std::cell::{Cell, RefCell};

          // Cell for Copy types
          struct Counter {
              count: Cell<i32>,
          }

          impl Counter {
              fn increment(&self) {
                  self.count.set(self.count.get() + 1);
              }
          }

          // RefCell for non-Copy types
          struct Logger {
              messages: RefCell<Vec<String>>,
          }

          impl Logger {
              fn log(&self, msg: &str) {
                  self.messages.borrow_mut().push(msg.to_string());
              }
          }
    ```
    Diferencias clave:
    1. Cell:
    - Funciona mejor con tipos Copy
    - Sin API de préstamo
    - Siempre copia o mueve valores

    2. RefCell:
    - Funciona con cualquier tipo
    - Tiene API de préstamo
    - Verificación de préstamo en tiempo de ejecución

    Mejores prácticas:
    1. Usa Cell para tipos Copy simples (números, bool, etc.)
    2. Usa RefCell cuando necesites tomar prestado el contenido
    3. Mantén las mutaciones a través de Cell/RefCell al mínimo
    4. Documenta por qué se necesita mutabilidad interior
  </div>

    ```rust
              x: i32,
              y: i32,
          }

          // Types that can only be Clone
    ```
