Was an Sprachmodellen seltsam ist? Sie können Quantenmechanik erklären, Poetry schreiben und Ihren TypeScript-Code debuggen... aber bitten Sie sie, 18472 mit 9347 zu multiplizieren, und es besteht eine gute Chance, dass sie mit Sicherheit etwas liefern, das um Tausende falsch ist.  

Das hat mich früher verwirrt, bis mir klar wurde, was wir tatsächlich von ihnen verlangen. Wir bitten eine Mustererkennungsmaschine, eine Rechenmaschine zu sein. Das ist so, als würde man eine Turnerin bitten, Ihre Kontobewegungen auszugleichen, weil sie das Konzept von "Ausgewogenheit" versteht.  

Das Problem ist, dass LLMs nichts berechnen. Wenn Sie GPT oder Claude fragen, was 2 + 2 ergibt, addieren sie nicht. Sie prognostizieren, dass „4“ der Token ist, der am wahrscheinlichsten nach „2 + 2 =“ folgt. Die meisten Male funktioniert das gut, weil diese Muster in ihrem Trainingsdatensatz existieren. Aber sobald Sie über einfache Arithmetik hinausgehen, in mehrschrittige Berechnungen oder Zahlen, die im Training nicht verbreitet waren, sind Sie im Grunde am Würfeln.  

Ich habe das kürzlich hautnah erlebt, als ich Code überprüfte, der ein Top-Modell dafür einsetzte, Hypothekenraten zu berechnen. Das Modell antwortete mit voller Überzeugung. Es war auch um 400 $/Monat falsch. Das ist eine Art Fehler, die zählt.  

Selbst wenn Modelle besser in der Argumentation werden (GPT-5 soll Verbesserungen zeigen), handelt es sich immer noch um eine fortgeschrittene Mustererkennung, nicht um symbolische Berechnung. Für kreative Arbeit und natürliche Sprachaufgaben ist diese probabilistische Natur genau das, was sie magisch macht. Für Mathematik? Nicht so sehr.  

## Was löst das Problem tatsächlich?

Die Lösung besteht nicht darin, auf intelligere Modelle zu warten. Sie besteht darin, dem Modell das richtige Werkzeug für die Aufgabe zu geben.  

Stellen Sie sich vor, wie Sie dieses Problem lösen würden, wenn Sie ein nicht-AI-System bauen. Sie würden nicht eigene Mathematik-Logik schreiben, sondern sich einer Bibliothek bedienen. Das gleiche Prinzip gilt hier, außer dass wir dem LLM jetzt beibringen, wann und wie es diese Bibliothek zu verwenden hat.  

Das Tool-Calling in modernen AI-SDKs ermöglicht es uns, dem Modell strukturierte Funktionen zu übergeben, die es aufrufen kann. Statt das LLM dazu zu zwingen, Mathematik zu simulieren, geben wir ihm etwas, das tatsächlich Mathematik beherrscht: einen symbolischen Mathematik-Engine.  

Ich verwende dafür [AI SDK v5 und v6](https://ai-sdk.vercel.ai/), kombiniert mit CortexJS Compute Engine. Das SDK übernimmt die Orchestrierung und das Tool-Routing, während CortexJS von der einfachsten Arithmetik bis hin zur Infinitesimalrechnung alles abdeckt. Es ist eine überraschend klare Trennung der Verantwortlichkeiten.  

```bash
bun add ai @ai-sdk/anthropic @cortex-js/compute-engine zod
```  

## Aufbau des Mathematik-Werkzeugs  

Die Umsetzung ist überraschenderweise einfacher, als Sie erwarten. Was wir bauen, ist eine Brücke zwischen der natürlichen Sprachverarbeitung des LLMs und der tatsächlichen mathematischen Berechnung.

Einige Punkte, die hier erwähnenswert sind:  

Die Beschreibung übernimmt eine zentrale Rolle. Diese Formulierung „MUST be used“ („MUST verwendet werden“) mag aggressiv wirken, aber aus meiner Erfahrung ist es entscheidend, dem Modell explizit mitzuteilen, wann ein Tool genutzt werden muss. Das ist der Unterschied zwischen gelegentlicher Funktionalität und zuverlässigem Betrieb. Betrachten Sie dies als Prompt Engineering auf Tool-Ebene.  

Die Batchverarbeitung über ein `expressions`-Array ist wichtiger, als Sie denken könnten. Jeder Modellaufruf verursacht Verzögerung. Wenn Sie ein Gleichungssystem lösen oder mehrschrittige Mathematik durchführen, führt die Einzelverarbeitung zu einem schlechten Benutzererlebnis. Batching bedeutet eine Rundreise, um zehn Probleme auf einmal zu lösen.  

Die Verwendung eines symbolischen Engines statt einfach `eval()` (bitte nicht `eval()` verwenden) gewährt uns ein echtes mathematisches Verständnis. Das System analysiert die Absicht, verarbeitet LaTeX-Formatierung und kann mit Ableitungen und Integralen umgehen. Wir führen nicht bloß Berechnungen durch, sondern Mathematik im eigentlichen Sinne.  

Die Fehlerbehandlung ist pro Expression begrenzt. Wenn eine Berechnung fehlschlägt, geben wir diese Fehlermeldung zurück, führen aber die verbleibenden Berechnungen fort. Dies erlaubt dem Modell, zu erkennen, was funktioniert hat und was nicht, wodurch es potenziell in den nächsten Schritten selbst korrigieren kann.  

## In die Praxis umsetzen

Wir werfen etwas vor, das einen rohen Modelltyp typischerweise in Halluzinationen versetzen würde:

```typescript
import { anthropic } from '@ai-sdk/anthropic';

const { text } = await generateText({
  model: anthropic('claude-sonnet-4-5'),
  prompt: 'Berechnen Sie 18472 × 9347, teilen Sie durch 127 und ziehen Sie anschließend die Quadratwurzel des Ergebnisses.',
  tools: { mathTool },
  stopWhen: stepCountIs(5), // Bis zu fünf Modell/Tool-Schritte erlauben
});

console.log(text);
```

Das Modell erkennt die Mathematik, identifiziert die Notwendigkeit von Genauigkeit, ruft das Tool auf, erhält das präzise Ergebnis und erklärt es anschließend in natürlicher Sprache. Jede Komponente übernimmt das, was sie am besten kann.

## Jenseits der Grundrechenarten

Da wir ein symbolisches Engine-System verwenden, kann dieser Ansatz Dinge bewältigen, die einfache Rechenwerkzeuge nicht verarbeiten können.

Algebraische Gleichungen lösen? "Lösen Sie diese Gleichungen: 3x + 7 = 22 und 2y - 5 = 13" funktioniert problemlos.

Differenzialrechnung benötigen? "Finden Sie die Ableitung von x^3 + 2x^2 und bewerten Sie sie an der Stelle x = 2" ist einfach ein weiterer Toolaufruf.

Die LaTeX-Unterstützung ist besonders nützlich, wenn Sie Bildungs-Apps entwickeln. Das System versteht LaTeX-Eingaben von Grund auf und kann Ergebnisse im richtigen Format für die Darstellung zurückgeben. Keine zusätzliche Parsing erforderlich.  

## Das größere Bild  

Ich glaube, dieses Muster ist über Mathematik hinaus relevant. Was wir eigentlich tun, ist die Grenzen von LLMs anzuerkennen, während wir ihre Stärken nutzen. Sie sind erstaunlich darin, Absichten zu verstehen, natürliche Sprache zu verarbeiten und Workflows zu orchestrieren. Sie sind keine Taschenrechner, keine Datenbanken und keine Dateisysteme.  

Jedes Mal, wenn wir versuchen, ein LLM dazu zu bringen, etwas Deterministisches zu tun, kämpfen wir gegen seine Natur. Doch wenn wir diese Fähigkeit zur natürlichen Sprachverarbeitung mit spezialisierten Tools kombinieren, die sich um die deterministischen Teile kümmern? Genau dann wird es interessant.  

Das Mathematik-Tool ist nur ein Beispiel. Das gleiche Prinzip gilt für Datumsmanipulation, Finanzberechnungen, Bildverarbeitung, Datenbankabfragen... überall dort, wo Präzision wichtiger ist als Kreativität. Lassen Sie das Modell verstehen, was der Benutzer will, und übertragen Sie die eigentliche Arbeit an etwas, das dafür gebaut wurde.  

Es ist eine Veränderung in der Art und Weise, wie wir mit KI arbeiten. Nicht „kann das Modell das tun?“, sondern „kann das Modell das orchestrieren?“ Ein kleiner Unterschied in der Formulierung, aber eine große Verbesserung in der Zuverlässigkeit.
