言語モデルの何が奇妙かと言えば、量子力学の解説や詩の作成、TypeScriptのデバッグはできても、18472 × 9347 の計算を頼むと、数千もずれた答えを堂々と提示される確率が十分にあるんです。

これは、実際に何を求めているのかを理解するまでずっと不可解だった。私たちはパターンマッチングエンジンに電卓の役割を求めている。それは、「バランス」という概念を理解しているからといって、体操選手に家計簿の収支管理を任せるようなものだ。

要するに、LLMは何も計算していない。GPTやClaudeに2 + 2がいくつだと尋ねても、足し算をしているわけではない。"2 + 2 ="の後に"4"というトークンが出現する確率が最も高いと予測しているだけだ。多くの場合、訓練データにこれらのパターンが存在するためうまくいく。しかし、単純な算術を超えて多段階の計算や、訓練データにあまり登場しない数値を含むタスクに踏み込むと、実質的にサイコロを転がしているのと同じことになる。

最近、トップティアのモデルを使って住宅ローンの返済額を計算するコードレビューをしていた際、この問題に直面した。モデルは完全な自信を持って回答していた。しかし、結果は月々400ドルの誤差があった。それが実際に問題になる種類の誤差だ。

モデルの推論能力が向上しても（GPT-5は改善が見られるとされる）、それらは高度なパターンマッチングを行っており、記号計算を行っているわけではない。創造的な作業や自然言語タスクでは、この確率的な性質こそが魔法のように機能する。しかし、数学では？ 話は別だ。

## これを本当に解決するには？

答えは、より賢いモデルを待つことではない。モデルに適切なツールを与えることだ。

もしAIシステム以外を構築しているなら、この問題をどう解決するか考えてみてほしい。独自の数値計算ロジックを書かず、ライブラリに頼るはずだ。ここでも同じ原則が適用される。ただし、今回はLLMにそのライブラリをいつ、どのように使うかを教え込む点が異なる。

現代のAI SDKにおけるツール呼び出し機能を使えば、モデルが呼び出せる構造化された関数を渡すことができる。LLMに数学の知識があるふりを強いるのではなく、実際に計算できるもの、つまり記号計算エンジンを与えるのだ。

私はこれに [AI SDK v5 と v6](https://ai-sdk.vercel.ai/) を使用しており、CortexJS Compute Engine と組み合わせている。SDK がオーケストレーションとツールルーティングを処理し、CortexJS が基本算術から微積分までを処理する。関心の分離が、驚くほど明確だ。

```bash
bun add ai @ai-sdk/anthropic @cortex-js/compute-engine zod
```

## 数学ツールの実装

実装は予想より単純だ。構築しているのは、LLMの自然言語理解と実際の数値計算を繋ぐブリッジに過ぎない。

```typescript
import { generateText, stepCountIs, tool } from 'ai';
import { ComputeEngine } from '@cortex-js/compute-engine';
import { z } from 'zod';

// Initialize the engine once
const ce = new ComputeEngine();

const mathTool = tool({
  description: 'Evaluate mathematical expressions and solve equations with guaranteed accuracy. MUST be used for all mathematical operations to verify correctness - do not attempt mental math. Supports arithmetic, algebra, calculus, and complex operations. Can process multiple expressions at once.',
  parameters: z.object({
    expressions: z.array(z.string()).describe(
      'Array of mathematical expressions in LaTeX or plain notation, e.g. ["2 + 2", "\\frac{x^2 + 1}{x - 1}", "\\int x^2 dx"]'
    ),
  }),
  execute: async ({ expressions }) => {
    // Process all expressions in parallel (or detailed batch)
    return expressions.map(expression => {
      try {
        const result = ce.parse(expression).evaluate();
        return {
          expression,
          result: result.toString(),
          latex: result.latex,
        };
      } catch (error) {
        return { 
          expression,
          error: (error as Error).message 
        };
      }
    });
  },
});
```

これについて、いくつか押さえておくべき点がある。

`description` フィールドが重い役割を担っている。「MUST be used」という表現は攻撃的に見えるかもしれないが、私の経験では、モデルに対してツールを使用するタイミングを明確に指示するかどうかで、たまに動くか確実に動くかの差が生まれる。ツールレベルのプロンプトエンジニアリングだと思えばよい。

`expressions` 配列によるバッチ処理は、思われている以上に重要だ。モデル呼び出しには必ずレイテンシが発生する。連立方程式を解く場合や多段階の計算を行う場合、個別に処理するとユーザー体験は最悪になる。バッチ処理すれば、10問解くのに往復は1回で済む。

`eval()` の代わりに記号計算エンジンを使うことで、真の意味での数学的処理が可能になる（`eval()` は絶対に使うな）。エンジンが意図を解析し、LaTeX フォーマットを処理し、微分や積分を扱える。計算をしているのではなく、数学を扱っているのだ。

エラーハンドリングは式ごとにスコープが区切られている。1つの計算で失敗しても、そのエラーを返しつつ他の処理は継続する。これによりモデルは成功した部分と失敗した部分を確認でき、次のステップで自己修正できる可能性がある。

## 実際に動かしてみる

通常、生モデルだと幻覚（ハルシネーション）を起こすような問題を投げてみよう：

```typescript
import { anthropic } from '@ai-sdk/anthropic';

const { text } = await generateText({
  model: anthropic('claude-sonnet-4-5'),
  prompt: 'Calculate 18472 × 9347, divide by 127, then take the square root of the result.',
  tools: { mathTool },
  stopWhen: stepCountIs(5), // Allow up to five model/tool steps
});

console.log(text);
```

モデルは数学的タスクを認識し、精度が必要だと判断してツールを呼び出し、正確な結果を取得した上で、それを自然言語で説明する。各コンポーネントが得意なことをやっているだけだ。

## 基礎計算の先へ

記号演算エンジンを採用しているため、このアプローチは単純な電卓ツールでは処理できない領域にも対応可能だ。

代数学の方程式求解が必要でも、「Solve these equations: 3x + 7 = 22 and 2y - 5 = 13」と指示するだけで正常に動作する。

微積分の計算が必要でも、「Find the derivative of x^3 + 2x^2 and evaluate it at x = 2」は単なる別のツール呼び出しに過ぎない。

教育用アプリケーションを構築する場合、LaTeXサポートは特に価値が高い。エンジン側がLaTeX入力をネイティブに解釈でき、レンダリング用の整形済み結果を返すため、追加のパース処理が一切不要になる。

## 全体像

このパターンが示す価値は、数学処理の範疇を超えている。私たちが実際に行っているのは、LLMの根本的な限界を認めた上で、その強みを体系的に活用することだ。意図の抽出、自然言語の構文解析、複雑なワークフローのオーケストレーションにおいて、これほど適性が高いコンポーネントは他にない。それらは電卓でもデータベースでもファイルシステムでもない。

LLMに決定論的なタスクを強制しようとするたびに、その確率的な本性に逆らうことになる。しかし、その自然言語理解能力を、決定論的な演算を専門に扱うツールと組み合わせるとなると、話は別だ。そこで初めて堅牢なシステムが構築できる。

数学ツールは単なる一例に過ぎない。日付の操作、財務計算、画像処理、データベースクエリ……創造性よりも正確性が優先されるあらゆる領域で、この原則はそのまま適用できる。モデルにはユーザーの意図を把握させ、実際の処理は当該タスク用に設計された専用コンポーネントに委譲すればよい。

これはAIを活用した開発における設計思想の転換だ。「モデルがこれを直接処理できるか？」ではなく、「モデルがこれをオーケストレーションできるか？」と問うのである。表現のわずかな違いが、システムの信頼性において決定的な差を生む。
