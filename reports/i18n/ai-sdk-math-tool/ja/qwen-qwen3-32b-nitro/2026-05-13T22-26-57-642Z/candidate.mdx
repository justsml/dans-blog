言語モデルの不思議な点をご存知ですか？量子力学を説明したり、詩を書いたり、あなたのTypeScriptをデバッグしたりできますが、18472と9347を掛けるように頼むと、結果が数千も誤差のある答えを自信満々で出す確率が結構あります。

かつて私はこれを不思議に思っていましたが、彼らに何を求めているかに気づくと理解できます。パターンマッチングエンジンに計算機をさせているのです。これは「バランス」という概念を理解しているから、ジムネーティクス選手にチェックブックのバランスを取らせるようなものです。

重要なのは、LLMは実際には何も計算していないということです。GPTやClaudeに「2+2は？」と尋ねても、足し算しているわけではありません。彼らは「2+2=」の後に「4」というトークンが最も現れやすいと予測しているのです。これらのパターンはトレーニングデータに存在するため、多くの場合正しく動作します。しかし、単純な算術を超えて複数ステップの計算やトレーニングデータにあまり含まれていない数値を扱うと、実質的にサイコロを振っているのと同じです。

最近、モルガン支払いを計算するコードをレビューしていたところ、トップクラスのモデルが400ドル/月も誤差のある答えを自信満々で出していました。これは重要なエラーです。

モデルが推論能力を向上させても（GPT-5は改善されているとされています）、依然として高度なパターンマッチングを行い、記号計算は行っていません。創造的な作業や自然言語タスクでは、この確率的な性質がまさに彼らの強みです。しかし数学ではそうではありません。

## 本当にこれを解決する方法とは？

より賢いモデルを待つことではありません。モデルに適切なツールを与えることです。

非AIシステムでこの問題を解決する方法を考えてみましょう。カスタムの数学ロジックを書くのではなく、ライブラリに頼ります。同じ原理がここにも適用されます。今度はLLMにそのライブラリを使うタイミングと方法を教えるのです。

現代のAI SDKにおけるツール呼び出しが、モデルに構造化された関数を呼び出す手段を提供します。LLMに数学を知っているふりをさせることなく、本当に計算できるもの（記号数学エンジン）を与えるのです。

私はこれに[AI SDK v5およびv6](https://ai-sdk.vercel.ai/)とCortexJS Compute Engineを組み合わせて使っています。SDKはオーケストレーションとツールルーティングを処理し、CortexJSは基本算術から微積分までを処理します。驚くほどシンプルな責務分離です。

```bash
bun add ai @ai-sdk/anthropic @cortex-js/compute-engine zod
```

## 数学ツールの構築

実装は想像よりも単純です。構築しているのはLLMの自然言語理解と実際の数学計算の間の橋渡しです。

```typescript
import { generateText, stepCountIs, tool } from 'ai';
import { ComputeEngine } from '@cortex-js/compute-engine';
import { z } from 'zod';

// エンジンを1回初期化
const ce = new ComputeEngine();

const mathTool = tool({
  description: '数学的表現の評価と方程式の解法を保証された精度で実行します。すべての数学的操作で正確さを確認するために必須です - 計算を頭の中で行わないでください。算術、代数、微積分、複雑な操作をサポートします。複数の表現を同時に処理可能です。',
  parameters: z.object({
    expressions: z.array(z.string()).describe(
      'LaTeXまたは平文表記の数学的表現の配列、例: ["2 + 2", "\\frac{x^2 + 1}{x - 1}", "\\int x^2 dx"]'
    ),
  }),
  execute: async ({ expressions }) => {
    // 並列処理（または詳細バッチ処理）
    return expressions.map(expression => {
      try {
        const result = ce.parse(expression).evaluate();
        return {
          expression,
          result: result.toString(),
          latex: result.latex,
        };
      } catch (error) {
        return { 
          expression,
          error: (error as Error).message 
        };
      }
    });
  },
});
```

この実装で注目すべき点：

説明文が重要な役割を果たしています。「必須」という表現が強硬に見えるかもしれませんが、私の経験ではモデルにツール使用のタイミングを明確に伝えることが、時折動作する状態から信頼性のある動作に差を生みます。これはツールレベルでのプロンプトエンジニアリングと考えてください。

`expressions`配列によるバッチ処理の重要性は想像以上です。各モデル呼び出しには遅延が伴います。連立方程式の解法や複数ステップの数学処理において個別に処理するとユーザー体験が悪化します。バッチ処理により1回の往復で10の問題を解決できます。

`eval()`（絶対に`eval()`は使わないでください）ではなく記号エンジンを使うことで真の数学的理解が得られます。エンジンは意図を解析し、LaTeX形式を処理し、微分積分も扱えます。単なる計算ではなく、数学そのものを実行しています。

エラーハンドリングは各表現単位で行われます。1つの計算が失敗してもエラーを返した上で他の処理は継続します。これによりモデルは成功と失敗の両方を視覚化し、次のステップで自己修正が可能になります。

## 実際の運用

通常はモデルがデタラメになるような問題を投げてみましょう：

```typescript
import { anthropic } from '@ai-sdk/anthropic';

const { text } = await generateText({
  model: anthropic('claude-sonnet-4-5'),
  prompt: '18472 × 9347を計算し、127で割って、結果の平方根を求めてください。',
  tools: { mathTool },
  stopWhen: stepCountIs(5), // 最大5回のモデル/ツールステップを許可
});

console.log(text);
```

モデルは数学を認識し、精度が必要と判断してツールを呼び出します。正確な結果を得た後、自然言語で説明します。それぞれのコンポーネントが得意なことを実行しているのがわかります。

## 基本的な算術を超えて

シンボリックエンジンを使っているため、このアプローチは単純な計算機ツールでは対応できない問題にも対処できます。

代数方程式を解きたいですか？「これらの式を解いてください：3x + 7 = 22 と 2y - 5 = 13」も問題ありません。

微積分が必要ですか？「x^3 + 2x^2 の導関数を求めてx=2で評価してください」も単なるツール呼び出しだけです。

教育用アプリケーションを構築する場合、LaTeXサポートは特に役立ちます。エンジンはLaTeX入力を本来理解しており、レンダリング用にフォーマットされた結果を返却できます。追加のパースは必要ありません。

## より広い視点

私はこのパターンが単なる数学の枠を超えて重要だと考えています。実際に行っているのは、LLMの限界を認めながらもその強みを活用することです。LLMは意図を理解したり、自然言語を解析したり、ワークフローを調整したりするのが得意です。彼らは計算機やデータベースやファイルシステムではありません。

LLMに決定論的な処理をさせようとするたびに、その本質と戦っています。しかし、その自然言語理解能力を、決定論的な部分を処理する専用ツールと組み合わせるとき、面白いことが起こります。

数学ツールは単なる例です。同じ原理は日付操作、金融計算、画像処理、データベースクエリ…創造性よりも精度が重要になるあらゆる場面に適用できます。モデルがユーザーが何を望んでいるかを理解し、実際に作業をその仕事に特化したツールに任せることを学びましょう。

これはAIを構築する考え方のシフトです。「モデルがこれを実行できるか？」ではなく「モデルがこれを調整できるか？」という問いに焦点を当てます。表現のわずかな違いですが、信頼性の差は計り知れません。
