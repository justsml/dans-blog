आपकोभाषा मॉडल्स में क्या अजीब लगता है? वे क्वांटम मैकेनिक्स समझा सकते हैं, कविता लिख सकते हैं, और आपके TypeScript को डिबग कर सकते हैं... लेकिन अगर आप उनसे 18472 को 9347 से गुणा करने को कहें तो काफी संभावना है कि वे आत्मविश्वास से ऐसा उत्तर देंगे जो हजारों में गलत हो।

यह मुझे भी उलझन में डालता था जब तक मैंने नहीं समझा कि हम वास्तव में उनसे क्या करवाना चाहते हैं। हम एक पैटर्न‑मैचिंग इंजन को कैलकुलेटर बनाने को कह रहे हैं। यह वैसा ही है जैसे जिम्नास्ट को आपका चेकबुक संतुलित करने को कहना, क्योंकि वह “संतुलन” की अवधारणा समझता है।

असल में, LLMs कुछ भी गणना नहीं करते। जब आप GPT या Claude से पूछते हैं कि 2 + 2 कितना है, तो वे जोड़ नहीं रहे होते। वे यह भविष्यवाणी कर रहे होते हैं कि “4” वह टोकन है जो “2 + 2 =” के बाद सबसे अधिक संभावना से आएगा। अधिकांश समय यह ठीक काम करता है क्योंकि ये पैटर्न उनके प्रशिक्षण डेटा में मौजूद होते हैं। लेकिन जब आप साधारण अंकगणित से आगे बढ़ते हैं, कई‑स्टेप की गणनाएँ या ऐसे नंबर जिनका प्रशिक्षण में बहुत कम उदाहरण था, तो आप मूलतः पासा फेंक रहे होते हैं।

मैंने यह समस्या हाल ही में सीधे सामना की जब मैं कुछ कोड की समीक्षा कर रहा था जो टॉप‑टियर मॉडल का उपयोग करके मॉर्टगेज भुगतान की गणना करता था। मॉडल ने पूरी आत्मविश्वास के साथ उत्तर दिया। लेकिन वह $400/माह से भी अधिक गलत था। यही वह प्रकार की त्रुटि है जो वास्तव में मायने रखती है।

भले ही मॉडल्स तर्कशक्ति में बेहतर होते जा रहे हों (GPT‑5 में सुधार दिखाया गया है), वे अभी भी जटिल पैटर्न मैचिंग कर रहे हैं, न कि प्रतीकात्मक गणना। रचनात्मक कार्य और प्राकृतिक भाषा कार्यों के लिए यह प्रायिकात्मक स्वभाव ही उन्हें जादुई बनाता है। गणित के लिए? उतना नहीं।

## वास्तव में क्या इस समस्या को हल करता है?

उत्तर अधिक बुद्धिमान मॉडलों का इंतजार करने में नहीं है। यह मॉडल को सही उपकरण प्रदान करने में है।

सोचिए कि यदि आप एक गैर‑AI सिस्टम बना रहे होते तो इस समस्या को कैसे हल करते। आप कस्टम गणितीय लॉजिक नहीं लिखते, बल्कि किसी लाइब्रेरी का उपयोग करते। यही सिद्धांत यहाँ लागू होता है, बस अब हम LLM को यह सिखा रहे हैं कि कब और कैसे उस लाइब्रेरी को बुलाए।

आधुनिक AI SDK में टूल कॉलिंग हमें मॉडल को संरचित फ़ंक्शन प्रदान करने की अनुमति देती है जिन्हें वह invoke कर सकता है। LLM को यह दिखाने के बजाय कि उसे गणित आती है, हम उसे वह चीज़ देते हैं जो वास्तव में करता है: एक प्रतीकात्मक गणित इंजन।

मैंने इसके लिए [AI SDK v5 और v6](https://ai-sdk.vercel.ai/) का उपयोग किया है, जिसे CortexJS Compute Engine के साथ जोड़ा गया है। SDK ऑर्केस्ट्रेशन और टूल रूटिंग संभालता है, जबकि CortexJS बुनियादी अंकगणित से लेकर कलन तक सब कुछ करता है। यह चिंताओं को अलग‑अलग रखने में आश्चर्यजनक रूप से साफ़ है।

```bash
bun add ai @ai-sdk/anthropic @cortex-js/compute-engine zod
```

## गणित टूल बनाना

इम्प्लीमेंटेशन उतना जटिल नहीं है जितना आप सोचते हैं। हम जो बना रहे हैं वह LLM की प्राकृतिक भाषा समझ और वास्तविक गणितीय गणना के बीच एक पुल है।

```typescript
import { generateText, stepCountIs, tool } from 'ai';
import { ComputeEngine } from '@cortex-js/compute-engine';
import { z } from 'zod';

// Initialize the engine once
const ce = new ComputeEngine();

const mathTool = tool({
  description: 'Evaluate mathematical expressions and solve equations with guaranteed accuracy. MUST be used for all mathematical operations to verify correctness - do not attempt mental math. Supports arithmetic, algebra, calculus, and complex operations. Can process multiple expressions at once.',
  parameters: z.object({
    expressions: z.array(z.string()).describe(
      'Array of mathematical expressions in LaTeX or plain notation, e.g. ["2 + 2", "\\frac{x^2 + 1}{x - 1}", "\\int x^2 dx"]'
    ),
  }),
  execute: async ({ expressions }) => {
    // Process all expressions in parallel (or detailed batch)
    return expressions.map(expression => {
      try {
        const result = ce.parse(expression).evaluate();
        return {
          expression,
          result: result.toString(),
          latex: result.latex,
        };
      } catch (error) {
        return { 
          expression,
          error: (error as Error).message 
        };
      }
    });
  },
});
```

इस बारे में कुछ महत्वपूर्ण बिंदु:

विवरण (description) मुख्य भूमिका निभाता है। वह “MUST be used” वाक्यांश थोड़ा कठोर लग सकता है, लेकिन मेरे अनुभव में मॉडल को टूल कब उपयोग करना है, इस बात को स्पष्ट रूप से बताना, कभी‑कभी काम करने और लगातार भरोसेमंद काम करने के बीच का अंतर है। इसे टूल‑स्तर पर प्रॉम्प्ट इंजीनियरिंग मानें।

`expressions` एरे के माध्यम से बैच प्रोसेसिंग का प्रभाव आप सोचते से अधिक है। प्रत्येक मॉडल कॉल में लेटेंसी जुड़ी होती है। यदि आप समीकरणों की प्रणाली हल कर रहे हैं या बहु‑चरणीय गणित कर रहे हैं, तो उन्हें एक‑एक करके प्रोसेस करना उपयोगकर्ता अनुभव को बुरा बना देता है। बैचिंग का मतलब है दस समस्याओं को हल करने के लिए केवल एक राउंड‑ट्रिप।

`eval()` के बजाय एक सिम्बॉलिक इंजन (कृपया `eval()` का प्रयोग न करें) का उपयोग करने से हमें वास्तविक गणितीय समझ मिलती है। इंजन इंटेंट को पार्स करता है, LaTeX फ़ॉर्मेटिंग को संभालता है, और डेरिवेटिव व इंटीग्रल जैसी चीज़ों को भी प्रोसेस कर सकता है। हम केवल गणना नहीं, बल्कि गणित कर रहे हैं।

त्रुटि संभालना प्रत्येक अभिव्यक्ति (expression) के स्तर पर किया गया है। यदि कोई एक गणना विफल हो जाती है, तो हम उस त्रुटि को लौटाते हैं लेकिन बाकी अभिव्यक्तियों को जारी रखते हैं। इससे मॉडल को पता चलता है कि क्या काम किया और क्या नहीं, और अगली चरण में स्वयं‑सुधार की संभावना बढ़ती है।

## इसे लागू करना

आइए कुछ ऐसा डालते हैं जो आमतौर पर एक कच्चे मॉडल को भ्रमित कर देता है:

```typescript
import { anthropic } from '@ai-sdk/anthropic';

const { text } = await generateText({
  model: anthropic('claude-sonnet-4-5'),
  prompt: 'Calculate 18472 × 9347, divide by 127, then take the square root of the result.',
  tools: { mathTool },
  stopWhen: stepCountIs(5), // Allow up to five model/tool steps
});

console.log(text);
```

मॉडल गणित देखता है, पहचानता है कि सटीकता चाहिए, टूल को कॉल करता है, सटीक परिणाम प्राप्त करता है, और फिर उसे प्राकृतिक भाषा में समझाता है। प्रत्येक घटक अपनी सबसे अच्छी क्षमता का उपयोग करता है।

## बुनियादी अंकगणित से आगे

चूँकि हम एक प्रतीकात्मक इंजन का उपयोग कर रहे हैं, यह तरीका उन चीज़ों को भी संभालता है जिन्हें साधारण कैलकुलेटर टूल नहीं छू सकते।

क्या आप बीजगणितीय समीकरण हल करना चाहते हैं? “Solve these equations: 3x + 7 = 22 and 2y - 5 = 13” ठीक काम करता है।

कैल्कुलस चाहिए? “Find the derivative of x^3 + 2x^2 and evaluate it at x = 2” बस एक और टूल कॉल है।

LaTeX समर्थन विशेष रूप से उपयोगी है जब आप शैक्षणिक ऐप्स बना रहे हों। इंजन स्वाभाविक रूप से LaTeX इनपुट को समझता है और रेंडरिंग के लिए स्वरूपित परिणाम लौटाता है। अतिरिक्त पार्सिंग की आवश्यकता नहीं।

## व्यापक परिप्रेक्ष्य

मुझे लगता है कि यह पैटर्न सिर्फ गणित से आगे तक लागू होता है। हम वास्तव में LLM की सीमाओं को स्वीकार कर रहे हैं जबकि उनकी ताकतों का उपयोग कर रहे हैं। वे इरादे को समझने, प्राकृतिक भाषा को पार्स करने, और वर्कफ़्लो को समन्वयित करने में अद्भुत हैं। वे कैलकुलेटर, डेटाबेस या फ़ाइल सिस्टम नहीं हैं।

हर बार जब हम LLM को कोई निर्धारक कार्य करवाने की कोशिश करते हैं, तो हम उसकी प्रकृति के विरुद्ध लड़ रहे होते हैं। लेकिन जब हम उस प्राकृतिक भाषा समझ को उन विशिष्ट टूल्स के साथ जोड़ते हैं जो निर्धारक भागों को संभालते हैं? तभी चीज़ें रोचक हो जाती हैं।

गणित टूल सिर्फ एक उदाहरण है। वही सिद्धांत तिथि संचालन, वित्तीय गणनाएँ, छवि प्रोसेसिंग, डेटाबेस क्वेरी… जहाँ सटीकता रचनात्मकता से अधिक मायने रखती है, उन सभी क्षेत्रों में लागू होता है। मॉडल को यह समझने दें कि उपयोगकर्ता क्या चाहता है, फिर वास्तविक कार्य को उस काम के लिए निर्मित टूल को सौंप दें।

यह AI के साथ निर्माण करने के बारे में सोचने का एक बदलाव है। “क्या मॉडल यह कर सकता है?” नहीं, बल्कि “क्या मॉडल इसे समन्वयित कर सकता है?” छोटी सी वाक्यांश में अंतर, विश्वसनीयता में बड़ा अंतर।
