Sabes qué es raro de los modelos de lenguaje? Pueden explicar mecánica cuántica, escribir poesía y depurar tu TypeScript… pero si les pides multiplicar 18472 por 9347 hay una buena probabilidad de que te den, con total confianza, un resultado que difiere en miles.

Esto me desconcertaba hasta que comprendí lo que realmente les estamos pidiendo. Le estamos pidiendo a un motor de coincidencia de patrones que sea una calculadora. Es como pedirle a un gimnasta que lleve la contabilidad porque entiende el concepto de “balance”.

La cuestión es que los LLM no calculan nada. Cuando le preguntas a GPT o a Claude cuánto es 2 + 2, no están sumando. Están prediciendo que “4” es el token más probable que aparezca después de “2 + 2 =”. La mayor parte del tiempo funciona porque esos patrones existen en sus datos de entrenamiento. Pero si vas más allá de la aritmética simple, a cálculos de varios pasos o a números que no eran comunes en el entrenamiento, esencialmente estás tirando dados.

Me topé con esto de frente hace poco al revisar un código que usaba un modelo de primera línea para calcular pagos de hipoteca. El modelo respondió con total confianza. También estaba equivocado por 400 $/mes. Ese es el tipo de error que importa.

Incluso cuando los modelos mejoran en razonamiento (se dice que GPT‑5 muestra avances), siguen haciendo coincidencia de patrones sofisticada, no cálculo simbólico. Para trabajos creativos y tareas de lenguaje natural, esa naturaleza probabilística es precisamente lo que los hace mágicos. ¿Para matemáticas? No tanto.

## ¿Qué es lo que realmente lo soluciona?

La respuesta no está esperando a modelos más inteligentes. Se trata de darle al modelo la herramienta adecuada para el trabajo.

Piensa cómo resolverías este problema si estuvieras construyendo un sistema no basado en IA. No escribirías lógica matemática a medida, buscarías una biblioteca. El mismo principio se aplica aquí, salvo que ahora le enseñamos al LLM cuándo y cómo usar esa biblioteca.

La llamada a herramientas en los SDK de IA modernos nos permite entregar al modelo funciones estructuradas que puede invocar. En lugar de obligar al LLM a fingir que sabe matemáticas, le damos algo que realmente lo hace: un motor de matemáticas simbólicas.

He estado usando [AI SDK v5 y v6](https://ai-sdk.vercel.ai/) junto con CortexJS Compute Engine. El SDK se encarga de la orquestación y el enrutamiento de herramientas, mientras que CortexJS maneja desde aritmética básica hasta cálculo. Es una separación de responsabilidades sorprendentemente limpia.

```bash
bun add ai @ai-sdk/anthropic @cortex-js/compute-engine zod
```

## Construyendo la Herramienta Matemática

La implementación es más directa de lo que podrías esperar. Lo que estamos construyendo es un puente entre la comprensión del lenguaje natural del LLM y la computación matemática real.

```typescript
import { generateText, stepCountIs, tool } from 'ai';
import { ComputeEngine } from '@cortex-js/compute-engine';
import { z } from 'zod';

// Initialize the engine once
const ce = new ComputeEngine();

const mathTool = tool({
  description: 'Evaluate mathematical expressions and solve equations with guaranteed accuracy. MUST be used for all mathematical operations to verify correctness - do not attempt mental math. Supports arithmetic, algebra, calculus, and complex operations. Can process multiple expressions at once.',
  parameters: z.object({
    expressions: z.array(z.string()).describe(
      'Array of mathematical expressions in LaTeX or plain notation, e.g. ["2 + 2", "\\frac{x^2 + 1}{x - 1}", "\\int x^2 dx"]'
    ),
  }),
  execute: async ({ expressions }) => {
    // Process all expressions in parallel (or detailed batch)
    return expressions.map(expression => {
      try {
        const result = ce.parse(expression).evaluate();
        return {
          expression,
          result: result.toString(),
          latex: result.latex,
        };
      } catch (error) {
        return { 
          expression,
          error: (error as Error).message 
        };
      }
    });
  },
});
```

Algunas cosas que vale la pena destacar:

La descripción lleva la mayor parte del trabajo. Esa frase “DEBE usarse” puede sonar agresiva, pero en mi experiencia, ser explícito con el modelo sobre cuándo invocar una herramienta marca la diferencia entre que funcione a veces y que funcione de forma fiable. Considéralo ingeniería de prompts a nivel de herramienta.

El procesamiento por lotes mediante un arreglo `expressions` importa más de lo que parece. Cada llamada al modelo tiene latencia. Si resuelves un sistema de ecuaciones o haces cálculos en varios pasos, procesarlos individualmente genera una experiencia de usuario terrible. Agruparlos permite una única ida y vuelta para resolver diez problemas.

Usar un motor simbólico en lugar de simplemente `eval()` (por favor, no uses `eval()`) nos brinda una comprensión matemática real. El motor interpreta la intención, maneja el formato LaTeX y puede trabajar con derivadas e integrales. No solo estamos haciendo cálculos, estamos haciendo matemáticas.

El manejo de errores está aislado por expresión. Si un cálculo falla, devolvemos ese error pero seguimos con el resto. Esto permite que el modelo vea qué funcionó y qué no, y potencialmente se autocorrija en el siguiente paso.

## Poniéndolo en práctica

Vamos a lanzar algo que normalmente haría que un modelo crudo alucine:

```typescript
import { anthropic } from '@ai-sdk/anthropic';

const { text } = await generateText({
  model: anthropic('claude-sonnet-4-5'),
  prompt: 'Calculate 18472 × 9347, divide by 127, then take the square root of the result.',
  tools: { mathTool },
  stopWhen: stepCountIs(5), // Allow up to five model/tool steps
});

console.log(text);
```

El modelo ve la operación, reconoce que necesita precisión, invoca la herramienta, obtiene el resultado exacto y luego lo explica en lenguaje natural. Cada componente hace lo que mejor sabe hacer.

## Más allá de la aritmética básica

Como estamos usando un motor simbólico, este enfoque maneja cosas que las herramientas de calculadora simples no pueden tocar.

¿Quieres resolver ecuaciones algebraicas? `"Solve these equations: 3x + 7 = 22 and 2y - 5 = 13"` funciona sin problemas.

¿Necesitas cálculo? `"Find the derivative of x^3 + 2x^2 and evaluate it at x = 2"` es solo otra llamada a la herramienta.

El soporte de LaTeX es especialmente útil si estás construyendo aplicaciones educativas. El motor entiende de forma nativa la entrada LaTeX y puede devolver resultados formateados para su renderizado. No se requiere análisis adicional.

## El panorama más amplio

Creo que este patrón trasciende el simple cálculo. Lo que realmente estamos haciendo es reconocer las limitaciones de los LLM y aprovechar sus fortalezas. Son excelentes para comprender la intención, interpretar lenguaje natural y orquestar flujos de trabajo. No son calculadoras, bases de datos ni sistemas de archivos.

Cada vez que intentamos que un LLM realice algo determinista, estamos luchando contra su naturaleza. Pero cuando combinamos esa comprensión del lenguaje natural con herramientas especializadas que manejan las partes deterministas, ahí es donde las cosas se ponen interesantes.

La herramienta matemática es solo un ejemplo. El mismo principio se aplica a la manipulación de fechas, cálculos financieros, procesamiento de imágenes, consultas a bases de datos… a cualquier lugar donde la precisión importe más que la creatividad. Deja que el modelo entienda lo que el usuario quiere y luego delega el trabajo real a algo construido para esa tarea.

Es un cambio en la forma de pensar al construir con IA. No “¿puede el modelo hacer esto?” sino “¿puede el modelo orquestar esto?”. Una pequeña diferencia en la redacción, una diferencia significativa en la fiabilidad.
