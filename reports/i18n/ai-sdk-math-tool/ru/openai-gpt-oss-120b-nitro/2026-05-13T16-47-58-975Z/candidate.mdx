Вы знаете, что странного в языковых моделях? Они могут объяснить квантовую механику, написать стихотворение и отладить ваш TypeScript… но если попросить их умножить 18472 на 9347, есть довольно большая вероятность, что они уверенно выдадут результат, ошибающийся на тысячи.

Меня это тоже озадачивало, пока я не понял, чего мы на самом деле от них требуем. Мы просим движок, основанный на сопоставлении шаблонов, стать калькулятором. Это всё равно что просить гимнаста вести ваш бухгалтерский учёт, потому что он понимает понятие «баланс».

Дело в том, что LLM‑ы ничего не вычисляют. Когда вы спрашиваете у GPT или Claude, чему равно 2 + 2, они не складывают. Они предсказывают, что токен «4» наиболее вероятен после строки «2 + 2 =». В большинстве случаев это работает отлично, потому что такие шаблоны присутствуют в их обучающих данных. Но если выйти за рамки простейшей арифметики, перейти к многошаговым вычислениям или к числам, которые были редки в обучении, вы по сути бросаете кости.

Я столкнулся с этим напрямую недавно, проверяя код, который использовал топовую модель для расчёта ипотечных платежей. Модель ответила с полной уверенностью. При этом она ошиблась на 400 долларов в месяц. Именно такие погрешности имеют значение.

Даже когда модели становятся лучше в рассуждениях (по слухам, GPT‑5 уже показывает улучшения), они всё равно занимаются сложным сопоставлением шаблонов, а не символьными вычислениями. Для творческих задач и работы с естественным языком эта вероятностная природа — именно то, что делает их волшебными. Для математики? Не совсем.

## Что действительно решает эту проблему?

Ответ не будет ждать более умных моделей. Мы просто даём модели правильный инструмент для задачи.

Подумайте, как бы вы решили эту проблему, если бы строили обычную систему без ИИ. Вы бы не писали собственную математическую логику, а подключили бы библиотеку. Тот же принцип применим и здесь, только теперь мы обучаем LLM, когда и как пользоваться этой библиотекой.

Вызов инструментов в современных AI‑SDK позволяет передать модели структурированные функции, которые она может вызвать. Вместо того чтобы заставлять LLM притворяться, что она умеет считать, мы даём ей то, что действительно умеет: символьный математический движок.

Я использую [AI SDK v5 и v6](https://ai-sdk.vercel.ai/) в паре с CortexJS Compute Engine. SDK берёт на себя оркестрацию и маршрутизацию вызовов, а CortexJS обрабатывает всё — от простейшей арифметики до интегралов. Получается удивительно чистое разделение ответственности.

```bash
bun add ai @ai-sdk/anthropic @cortex-js/compute-engine zod
```

## Создание математического инструмента

Реализация проще, чем может показаться. Мы строим мост между естественным языком LLM и реальными математическими вычислениями.

```typescript
import { generateText, stepCountIs, tool } from 'ai';
import { ComputeEngine } from '@cortex-js/compute-engine';
import { z } from 'zod';

// Initialize the engine once
const ce = new ComputeEngine();

const mathTool = tool({
  description: 'Evaluate mathematical expressions and solve equations with guaranteed accuracy. MUST be used for all mathematical operations to verify correctness - do not attempt mental math. Supports arithmetic, algebra, calculus, and complex operations. Can process multiple expressions at once.',
  parameters: z.object({
    expressions: z.array(z.string()).describe(
      'Array of mathematical expressions in LaTeX or plain notation, e.g. ["2 + 2", "\\frac{x^2 + 1}{x - 1}", "\\int x^2 dx"]'
    ),
  }),
  execute: async ({ expressions }) => {
    // Process all expressions in parallel (or detailed batch)
    return expressions.map(expression => {
      try {
        const result = ce.parse(expression).evaluate();
        return {
          expression,
          result: result.toString(),
          latex: result.latex,
        };
      } catch (error) {
        return { 
          expression,
          error: (error as Error).message 
        };
      }
    });
  },
});
```

Несколько моментов, на которые стоит обратить внимание:

Описание несёт большую часть нагрузки. Формулировка «MUST be used» может показаться резкой, но по моему опыту чёткое указание модели, когда использовать инструмент, отличает редкую работу от надёжной. Считайте это инженерией подсказок на уровне инструмента.

Пакетная обработка через массив `expressions` важнее, чем кажется. Каждый вызов модели влечёт задержку. Если вы решаете систему уравнений или делаете многошаговые вычисления, обработка их по отдельности создаёт ужасный пользовательский опыт. Пакетирование позволяет решить десять задач за один раунд‑трип.

Использование символьного движка вместо простого `eval()` (пожалуйста, не используйте `eval()`) даёт реальное математическое понимание. Движок разбирает намерения, обрабатывает LaTeX‑форматирование и умеет работать с производными и интегралами. Мы делаем не просто вычисления, а настоящую математику.

Обработка ошибок привязана к каждому выражению отдельно. Если одно вычисление падает, мы возвращаем ошибку, но продолжаем обработку остальных. Это позволяет модели увидеть, что сработало, а что нет, и потенциально скорректировать себя на следующем шаге.

## Применение на практике

Давайте бросим в модель задачу, которая обычно заставляет «сырой» ИИ галлюцинировать:

```typescript
import { anthropic } from '@ai-sdk/anthropic';

const { text } = await generateText({
  model: anthropic('claude-sonnet-4-5'),
  prompt: 'Calculate 18472 × 9347, divide by 127, then take the square root of the result.',
  tools: { mathTool },
  stopWhen: stepCountIs(5), // Allow up to five model/tool steps
});

console.log(text);
```

Модель видит математическую задачу, понимает, что нужна точность, вызывает инструмент, получает корректный результат и затем объясняет его на естественном языке. Каждый компонент делает то, что умеет лучше всего.

## За пределами базовой арифметики

Поскольку мы используем символьный движок, такой подход справляется с тем, что простые калькуляторные инструменты не могут обработать.

Хотите решить алгебраические уравнения? «Solve these equations: 3x + 7 = 22 and 2y - 5 = 13» работает без проблем.

Нужен анализ? «Find the derivative of x^3 + 2x^2 and evaluate it at x = 2» — это просто ещё один вызов инструмента.

Поддержка LaTeX особенно полезна, если вы создаёте образовательные приложения. Движок изначально понимает LaTeX‑ввод и может вернуть результат в формате, готовом к рендерингу. Дополнительный парсинг не требуется.

## Более широкий контекст

Я считаю, что эта схема важна не только для математики. По сути, мы признаём ограничения LLM, одновременно используя их сильные стороны. Они превосходно улавливают намерения, разбирают естественный язык и координируют рабочие процессы. Но они не являются калькуляторами, базами данных или файловыми системами.

Каждый раз, когда мы пытаемся заставить LLM выполнять детерминированную задачу, мы боремся с её природой. А когда мы комбинируем понимание естественного языка модели со специализированными инструментами, которые берут на себя детерминированные части? Именно тогда появляется реальная ценность.

Инструмент для вычислений — это лишь один пример. Тот же принцип применим к работе с датами, финансовыми расчётами, обработкой изображений, запросам к базе данных… везде, где точность важнее креативности. Позвольте модели понять, чего хочет пользователь, а затем передайте фактическую работу тому, кто создан для этой задачи.

Это изменение мышления при построении решений с ИИ. Не «может ли модель выполнить это?», а «может ли модель скоординировать это?». Небольшая разница в формулировке, но значительное влияние на надёжность.
