## Umgang mit Secrets & API‑Tokens sicher

### Verwandter Artikel: [Schütze deine Tokens](../protect-your-tokens/)

Kurz zusammengefasst: Unterschied zwischen `secret` und `non-secret`.

* 🔒 `Secret keys` MÜSSEN über einen eigenen Server (z. B. Node/Express/Heroku) laufen, um Anfragen an Dritt‑API‑Dienste zu verbergen (Proxy).
* 🌍 `Non-secret keys` bezeichnet Schlüssel, die an den Browser gesendet werden können.

> Wirkonzentrieren uns in diesem Artikel darauf, 🔒 `Secret keys` mittels **Umgebungsvariablen** zu behandeln.

[Code‑Beispiele finden Sie weiter unten.](#️-code-example)

#### Überblick

Um **Geheimnisse sicher in Ihrem NodeJS‑Code zu verwenden**:

1. Ersetzen Sie hartkodierte Schlüssel durch Umgebungsvariablen, z. B. `process.env.API_SECRET`
1. Nutzen Sie eine Bibliothek wie [`dotenv`](https://github.com/motdotla/dotenv) zusammen mit einer `.env`‑Datei. Fügen Sie Ihre zuvor hartkodierten Geheimnisse in die `.env`‑Datei ein.
1. Stellen Sie sicher, dass die Zeile für `.env` in Ihrer `.gitignore`‑Datei steht!

> **NICHT** eine `.env`‑Datei auf produktiven Servern anlegen. Verwenden Sie das von Ihrem Hosting‑Anbieter bereitgestellte Werkzeug zur Verwaltung von Umgebungsvariablen (z. B. [Heroku](https://devcenter.heroku.com/articles/config-vars), Netlify, AWS EC2): entweder über das **Dashboard** oder die **Kommandozeile**.

### Code‑Beispiel

Wir definieren ein paar Dateien.

1. `.env`
1. `./db/connection.js`
1. `./api/users.js`

Zuerst das Paket [`dotenv`](https://www.npmjs.com/package/dotenv) installieren.

```bash
npm install dotenv
```

Als nächstes eine `.env`‑Datei im Projekt‑Root anlegen.

```
# .env
PGDATABASE="postgres"
PGHOST="localhost"
PGPORT=5234
PGUSER="postgres"
PGPASSWORD="password"
```

❌ **NIEMALS** die `.env`‑Datei committen.

❌ Auf Servern keine `.env`‑Datei erzeugen.

Siehe die Dokumentation deines Hosting‑Providers, um _Umgebungsvariablen_ zu konfigurieren.

Um sicherzugehen, dass deine `.gitignore` eine Zeile für `.env` enthält:

```bash
# .gitignore automatisch aktualisieren
# Im Terminal ausführen:
[ "$(grep '^.env' .gitignore)" == "" ] && echo '.env' >> .gitignore
# Hinweis: Es wird keine Ausgabe erzeugt
```

Die Datei `./db/connection.js` stellt eine gemeinsam genutzte `pg.Pool`‑Instanz bereit. Sie wird für Datenbank‑Abfragen verwendet.

```js
// ./db/connection.js
require('dotenv').config(); // ✅ .env-Datei laden
const pg = require('pg');
const {PGUSER, PGHOST, PGPORT} = process.env;

if (process.env.NODE_ENV === 'development')
  console.log(`Connecting to ${PGUSER} @ ${PGHOST}:${PGHOST}`);
// ^^ nur zum Debuggen der Verbindungsvariablen

// pg verwendet automatisch die PG*-Umgebungsvariablen
module.exports = new pg.Pool();
```

Der Ordner `./api` enthält Schnittstellen zu deinen Tabellen/Views.

Ein Beispiel für `./api/users.js` zur `users`‑Tabelle:

```js
// ./api/users.js
const db = require('../db/connection.js');

module.exports = {
  findUsername: function(username) {
    return db.query('SELECT * FROM users WHERE username=$1', username);
  }
};
```

-  **Nie** deine `.env`‑Geheimnisse in Git committen!  
-  **Teile `.env`‑Dateien nicht im Team.** *

\* Jeder neue Entwicklungs‑Laptop oder -Desktop sollte **neue Zugangs‑Keys & Tokens erzeugen**.  
Falls das nicht möglich ist, gehe äußerst vorsichtig mit dem Teilen deiner `.env` um (z. B. wenn ein Dienst alle älteren Schlüssel ungültig macht oder du ein kostenpflichtiges API‑Token mit begrenztem Zugriff nutzt).
