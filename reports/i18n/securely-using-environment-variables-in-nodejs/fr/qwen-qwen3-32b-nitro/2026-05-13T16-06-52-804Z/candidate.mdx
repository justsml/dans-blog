## Gestion sécurisée des secrets et des jetons API

### Article lié : [Protégez vos jetons](../protect-your-tokens/)

Rappelons rapidement la différence entre `secret` et `non-secret`.

* 🔒 Les `cles secrètes` doivent utiliser un serveur personnalisé (ex. Node/Express/Heroku) afin de cacher (proxy) les requêtes vers les services API tiers.
* 🌍 Les `cles non secrètes` désignent les clés pouvant être envoyées au navigateur.

<br />

---------------------------------------------

> Nous nous concentrerons sur la gestion des 🔒 `cles secrètes` à l'aide de **Variables d'environnement** dans cet article.

[Exemples de code sont inclus ci-dessous.](#️-code-example)

#### Aperçu

Pour **accéder en toute sécurité aux secrets dans votre code NodeJS :**

1. Remplacez les clés codées en dur par des variables d'environnement. Ex. : `process.env.API_SECRET`
1. Utilisez une bibliothèque comme [`dotenv`](https://github.com/motdotla/dotenv) accompagnée d'un fichier `.env`. Ajoutez vos anciennes clés codées en dur dans le fichier `.env`.
1. Vérifiez que la ligne `.env` est présente dans votre fichier `.gitignore` !

> **NE FAITES PAS** créer un fichier `.env` sur les serveurs déployés. Utilisez les outils de gestion des variables d'environnement fournis par votre service d'hébergement (ex. [Heroku](https://devcenter.heroku.com/articles/config-vars), Netlify, AWS EC2) : par exemple, **l'interface de configuration ou la ligne de commande**.

### Exemple de code

Nous allons définir quelques fichiers.

1. `.env`
1. `./db/connection.js`
1. `./api/users.js`

Installez d'abord le package [`dotenv`](https://www.npmjs.com/package/dotenv).

```bash
npm install dotenv
```

Ensuite, créez un fichier `.env` à la racine de votre projet.

```
# .env
PGDATABASE="postgres"
PGHOST="localhost"
PGPORT=5234
PGUSER="postgres"
PGPASSWORD="password"
```

❌ **NE JAMAIS** commettre le fichier `.env`.

❌ Évitez de créer un `.env` sur les serveurs.

Vérifiez les documents de votre fournisseur d'hébergement pour configurer les _variables d'environnement_.

Pour vérifier facilement que votre `.gitignore` contient bien une ligne `.env` :

```bash
# Mise à jour automatique de .gitignore
# Exécuter dans le terminal :
[ "$(grep '^.env' .gitignore)" == "" ] && echo '.env' >> .gitignore
# note : aucune sortie ne s'affichera
```

Le fichier `./db/connection.js` fournit une instance partagée de `pg.Pool`. Il sera utilisé pour interroger la base de données.

```js
// ./db/connection.js
require('dotenv').config(); // ✅ Charger le fichier .env
const pg = require('pg');
const {PGUSER, PGHOST, PGPORT} = process.env;

if (process.env.NODE_ENV === 'development')
  console.log(`Connexion à ${PGUSER} @ ${PGHOST}:${PGHOST}`);
// ^^ uniquement pour afficher les variables de connexion en débogage

// pg utilise automatiquement les variables d'environnement PG*
module.exports = new pg.Pool();
```

Le dossier `./api` contient les interfaces vers vos tables/vues.

Voici un exemple de `./api/users.js` pour la table `users`.

```js
// ./api/users.js
const db = require('../db/connection.js');

module.exports = {
  findUsername: function(username) {
    return db.query('SELECT * FROM users WHERE username=$1', username);
  }
};
```

- **Ne jamais valider vos secrets .env dans git !**
- **Ne pas partager les fichiers .env au sein d'une équipe.** *

* Chaque nouveau ordinateur de développement doit **générer de nouvelles clés d'accès et jetons**.
Si ce n'est pas possible, prenez de nombreuses précautions lors de la partage de votre .env (dans les cas où un service pourrait invalider toutes les clés plus anciennes, ou si vous avez des API payantes avec un jeton d'accès limité.)

#### ⚠️ Important : si nécessaire, utilisez toujours un service de messagerie sécurisé (préférez ceux avec support de messages expirants.)
