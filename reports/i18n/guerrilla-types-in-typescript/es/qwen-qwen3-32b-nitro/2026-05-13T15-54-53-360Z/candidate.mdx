### Enfoques para diseñar tipos

Probablemente hayas encontrado o escrito patrones variados en torno a "implementaciones de tipos". Especialmente cuando consumes datos de APIs de terceros.

**Nota:** A propósito, estoy ignorando procesos "tradicionales" como la creación de Diagramas de Entidad-Relación (DER) o jerarquías de herencia de Programación Orientada a Objetos (POO). Aquí, estamos diseñando tipos para representar datos de API semiestructurados.

Exploraremos dos enfoques de alto nivel: **Objeto único grande** (de arriba hacia abajo) vs. **Tipos nombrados múltiples** (de abajo hacia arriba).

#### Objeto único grande

Prioriza la explicititud sobre la reutilización y la DRY.

**Bonus:** La experiencia del IDE/Desarrollador es excelente, ya que las sugerencias incluyen una vista previa más completa sin complicaciones.

```tsx
interface ProductDetails {
  name: string;
  seller: { name: string };
  availability: Array<{ warehouseId: string; quantity: number }>;
  reviews: Array<{ authorId: number; stars: number }>;
}
```

Al priorizar la legibilidad explícita, es aceptable permitir _cierta_ repetición (dentro de razonable). Cuando grupos de propiedades se repiten _muchas_ veces, sientete libre de extraer los campos repetidos a un tipo nombrado.

#### Tipos nombrados múltiples

Prioriza la reutilización y la DRY-ness.

<!-- La legibilidad es una medida curiosa. Dado que la legibilidad es buena o **excelente cuando hay pocos tipos/archivos.** **Inevitablemente los tipos tienden a proliferar,** con cada vez más propiedades. **La legibilidad sufre.** -->

Este enfoque probablemente sea el preferido con diferencia.

```ts
interface ProductDetails {
  name: string;
  seller: Seller;
  reviews: Reviews[];
  availability: Availability[];
}
interface Seller { name: string; }
interface Availability { warehouseId: string; quantity: number; }
interface Reviews { authorId: number; stars: number; }
```

En general, este enfoque es excelente. Pero no está exento de inconvenientes.

- **Legibilidad** excelente al principio; sin embargo, _puede_ sufrir a medida que aumenta el tamaño y número de tipos.  
- Extremadamente DRY, ¿pero a qué costo? (Más sobre esto más adelante.)  
- La experiencia del desarrollador puede sufrir ya que las sugerencias son menos informativas.  

> ⚠️ Desde aproximadamente TypeScript v3, el Language Server trunca las sugerencias, omitiendo propiedades anidadas.  
> 💡 Hay trucos para mejorar un poco. Intente mantener presionado `Cmd o Ctrl`, luego pase el cursor sobre varios nombres de tipo: debería ver al menos una capa adicional de propiedades en la sugerencia.  

¿Por qué tenemos que elegir entre estos dos enfoques? (Tipo grande vs. Sub-tipos nombrados.)

```tsx
export interface ProductDetails {
  name: string;
  seller: { name: string };
  reviews: Array<{ authorId: number; stars: number }>;
  availability: Array<{ warehouseId: string; quantity: number }>;

export type Seller = ProductDetails["seller"];

```

1.  Crear tipos estructurados grandes "Primarios".
2.  Exportar subtipos derivados del tipo Primario.

Este enfoque destaca en sistemas donde los objetos "de alto nivel" benefician de la documentación en un solo lugar.
También, esta técnica soporta el reuso entre muchos casos de uso: Modelos, Servicios, Resultados de Consulta, etc.

### Técnica #2: Mix-ins

Esta estrategia consiste en combinar los **campos correctos**, con los **nombres adecuados**, para **representar objetos lógicos únicos.** El objetivo es abordar eficientemente múltiples casos de uso con Utilidades de TypeScript y Uniones de tipos.

Este enfoque difiere de la herencia y jerarquías tradicionales de OOP, que busca crear capas de objetos en taxonomías estrechamente enlazadas. El **enfoque de mix-in se centra en tipos planos y ligeramente relacionados**, agrupando campos relacionados mientras se reduce la duplicación.

#### Ejemplos de mix-in

```tsx
interface TodoModel {
  text: string;
  complete: boolean;
}
interface InstanceMixin {
  id: number;
}
/** TodoDraft representa el estado del formulario, posiblemente todo indefinido */
export type TodoDraft = Partial<TodoModel>;
/** Todo representa un registro de instancia de Todo desde la base de datos */
export type Todo = TodoModel & InstanceMixin;
```

#### Ejemplo `User`

```tsx
interface User {
  id: number;
  name: string;
  bio: string;
  social: Record<"facebook" | "instagram" | "github", URL>;
}
```

Representemos al `User` antes y después de guardarlo en la base de datos.

```tsx
// Campos básicos de User (por ejemplo, para un <form>)
interface UserBase {
  name: string;
  bio: string;
  social: Record<"facebook" | "instagram" | "github", URL>;
}
// Campos desde la base de datos
interface InstanceMixin {
  id: number;
  createdAt: Date;
  updatedAt: Date;
}
// Una instancia de User - con todos los campos
type UserInstance = InstanceMixin & UserBase;
```

Ahora podemos definir los campos exactos que necesitamos (como `password` para crear/actualizar, pero no incluida en consultas de `UserInstance`).

```tsx
interface UserBase {
  name: string;
  bio: string;
  social: Record<"facebook" | "instagram" | "github", URL>;
}
interface InstanceMixin {
  id: number;
  createdAt: Date;
  updatedAt: Date;
}
/** Tipo de payload para registro, incluyendo el campo `password` */
export type UserPayload = UserBase & { password: string };
/** Representa el tipo User devuelto por el servidor. */
export type UserInstance = UserBase & InstanceMixin;
```

### Técnica #3: Organización con Namespaces

Aquí declaramos un espacio de nombres `ModelMixins`. Esto proporciona cierta organización además de un patrón de reutilización más claro.

**Formas Estándar**

- `createdAt` y `updatedAt` existen juntos.
- `id`, no `ID` ni `_id`.

```tsx
// `src/types/mixins.d.ts`
namespace ModelMixins {
  interface Identity {
    id: number;
  }
  interface Timestamp {
    createdAt: Date;
    updatedAt: Date;
  }
  type Instance = ModelMixins.Identity & ModelMixins.Timestamp;
  interface HashedPassword {
    passwordHash: string;
  }
  interface InputPassword {
    password: string;
  }
}
```

**Usando Uniones de Tipos**

```tsx
// `src/types/user.d.ts`
export interface UserBase {
  name: string;
  bio: string;
  social: Record<"facebook" | "instagram" | "github", URL>;
}
// Único tipo `User`, usando Unión de Tipos para representar
// dinámicamente los estados pre- y post-creación.
export type User =
  | (UserBase & ModelMixins.Instance & ModelMixins.HashedPassword)
  | (UserBase & ModelMixins.InputPassword);
```

Si se desea, también se pueden exportar tipos nombrados individualmente:

```tsx
/** Payload de usuario para registro, incluyendo el campo `password` */
export type UserPayload = UserBase & ModelMixins.Instance & ModelMixins.HashedPassword;
/** Representa el tipo User devuelto por el servidor. */
export type UserInstance = UserBase & ModelMixins.InputPassword;
```

#### Uso en el mundo real

Aquí hay una función `upsert()` que utiliza el operador `in` para distinguir entre los tipos `UserInstance` y `UserPayload`.

```tsx
function upsert(user: User) {
  if ("id" in user) {
    // TypeScript sabe que `user` aquí tiene campos de Instance (id, createdAt, etc)
    return updateUser(user.id, user);
  } else {
    // TypeScript sabe que esto debe ser la versión `UserBase & ModelMixins.InputPassword` de user.
    return createUser(user);
  }
}
```
