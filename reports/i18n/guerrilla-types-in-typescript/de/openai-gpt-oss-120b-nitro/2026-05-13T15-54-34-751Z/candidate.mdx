### Ansätze zum Entwerfen von Typen

Du bist wahrscheinlich bereits über verschiedene Muster zu „Typ‑Implementierungen“ gestolpert oder hast sie selbst geschrieben – besonders beim Konsumieren von Daten aus Drittanbieter‑APIs.

**Hinweis:** Ich ignoriere bewusst die „traditionellen“ Verfahren, bei denen Entity‑Relationship‑Diagramme (ERD) oder objektorientierte Vererbungshierarchien erstellt werden. Hier bauen wir Typen, um halbstrukturierte API‑Daten abzubilden.

Betrachten wir zwei grobe Vorgehensweisen: **Ein einzelnes großes Objekt** (Top‑Down) gegenüber **Mehreren benannten Typen** (Bottom‑Up).

#### Ein einzelnes großes Objekt

Setzt die explizite Definition über Wiederverwendbarkeit und DRY‑Prinzip.

**Bonus:** IDE‑/Entwickler‑Erlebnis ist hervorragend, da Tooltips eine umfassendere Vorschau bieten – ohne Aufwand.

```tsx
interface ProductDetails {
  name: string;
  seller: { name: string };
  availability: Array<{ warehouseId: string; quantity: number }>;
  reviews: Array<{ authorId: number; stars: number }>;
}
```

Da wir explizite Lesbarkeit priorisieren, ist ein gewisses Maß an Wiederholung (in vernünftigem Rahmen) akzeptabel. Wiederholen sich Gruppen von Eigenschaften **häufig**, kann man die wiederholten Felder in einen benannten Typ auslagern.

#### Mehrere benannte Typen

Wiederverwendbarkeit & DRY‑Prinzip im Fokus.

<!-- Readability is a funny measure. Since Readability is often good or **great when there are few types/files.** **Inevitably types tend to proliferate,** featuring ever more properties. **Readability suffers.** -->

Dieser Ansatz ist mit Abstand der bevorzugte Weg.

```ts
interface ProductDetails {
  name: string;
  seller: Seller;
  reviews: Reviews[];
  availability: Availability[];
}
interface Seller { name: string; }
interface Availability { warehouseId: string; quantity: number; }
interface Reviews { authorId: number; stars: number; }
```

Insgesamt ist dieser Ansatz solide. Er hat jedoch auch seine Schattenseiten.

- **Lesbarkeit** ist anfangs hervorragend; jedoch kann sie leiden, wenn die Größe & Anzahl der Typen wächst.  
- Durchgehend DRY, aber zu welchem Preis? (Mehr dazu später.)  
- Die Entwickler‑Erfahrung kann leiden, weil Tooltips weniger informativ sind.  

> ⚠️ Seit (ungefähr) TypeScript v3 kürzt der Language Server Tooltips und lässt verschachtelte Eigenschaften weg.  
> 💡 Es gibt Tricks, um das ein wenig zu verbessern. Halte `Cmd` oder `Ctrl` gedrückt und fahre über verschiedene Typnamen – du solltest mindestens eine zusätzliche „Schicht“ von Eigenschaften im Tooltip sehen.  

Warum müssen wir zwischen diesen beiden Ansätzen wählen? (Riesiger Typ vs. benannte Sub‑Typen.)  

### Technik #1: Warum nicht alles  

Können wir beides haben?  

- Klarheit von „Big‑Picture“-Typen?  
- Plus benannte Sub‑Typen?  
- Ohne Duplizierung?

<!-- Wenn man strukturierte Array‑/Objektdaten bekommt, verspüren viele TypeScript‑Entwickler den Drang, Typen zu erstellen. Unmengen von Typen. Schließlich entsteht eine Kaskade von Schichten, bestehend aus einfacheren Typen, die immer komplexere Typen aufbauen.

Oder vielleicht bist du der Typ, der beim höchst‑abstrakten Typ anfängt und genug Gerüst bereitstellt, um den nächsten Sub‑Typ im Baum zu schreiben? -->

```tsx
export interface ProductDetails {
  name: string;
  seller: { name: string };
  reviews: Array<{ authorId: number; stars: number }>;
  availability: Array<{ warehouseId: string; quantity: number }>;
}
export type Seller = ProductDetails["seller"];
export type Review = ProductDetails["reviews"][number];
export type Availability = ProductDetails["availability"][number];
```

1.  Erstelle große „Primär‑“Strukturt­ypen.  
2.  Exportiere Sub‑Typen, die aus dem Primärtyp abgeleitet sind.

Dieser Ansatz glänzt besonders in Systemen, bei denen „hoch‑level“ Objekte von einer zentralen Dokumentation profitieren. Außerdem unterstützt die Technik die Wiederverwendung in vielen Szenarien: Modelle, Services, Abfrage‑Ergebnisse usw.

### Technik #2: Mix‑ins

Bei dieser Strategie geht es darum, die **richtigen Felder** mit den **richtigen Namen** zu kombinieren, um **einzelne logische Objekte** abzubilden. Ziel ist es, mehrere Anwendungsfälle effizient mit TypeScript‑Utilities und Typ‑Unions zu bedienen.

Im Gegensatz zu klassischer OOP‑Vererbung und Hierarchien, die Schichten von Objekten in eng geknüpften Taxonomien erzeugen wollen, **setzt der Mix‑in‑Ansatz auf flache und lose verbundene Typen**, gruppiert verwandte Felder und reduziert Duplizierung.

#### Mix‑in‑Beispiele


```tsx
interface TodoModel {
  text: string;
  complete: boolean;
}
interface InstanceMixin {
  id: number;
}
/** TodoDraft represents Form state, possibly all undefined */
export type TodoDraft = Partial<TodoModel>;
/** Todo represents a Todo instance record from the database */
export type Todo = TodoModel & InstanceMixin;
```

#### Beispiel `User`

```tsx
interface User {
  id: number;
  name: string;
  bio: string;
  social: Record<"facebook" | "instagram" | "github", URL>;
}
```

Wir stellen den `User` vor und nach dem Speichern in der Datenbank dar.

```tsx
// Kernfelder des Users (z. B. für ein <form>)
interface UserBase {
  name: string;
  bio: string;
  social: Record<"facebook" | "instagram" | "github", URL>;
}
// Felder aus der Datenbank
interface InstanceMixin {
  id: number;
  createdAt: Date;
  updatedAt: Date;
}
// Ein User‑**Instanz** – mit allen Feldern
type UserInstance = InstanceMixin & UserBase;
```

Jetzt können wir exakt die Felder modellieren, die wir benötigen (z. B. `password` für Create/Update, aber nicht in Abfragen von `UserInstance` enthalten).

```tsx
interface UserBase {
  name: string;
  bio: string;
  social: Record<"facebook" | "instagram" | "github", URL>;
}
interface InstanceMixin {
  id: number;
  createdAt: Date;
  updatedAt: Date;
}
/** User‑Payload für die Registrierung, inkl. `password`‑Feld */
export type UserPayload = UserBase & { password: string };
/** Repräsentiert den vom Server zurückgegebenen User‑Typ. */
export type UserInstance = UserBase & InstanceMixin;
```

1.  „Ist das eine gute Praxis?“
2.  „Soll ich das ausprobieren?“

Keine Ahnung. Weiter geht’s!

### Technik #3: Organisation mit Namespaces

Hier deklarieren wir einen `ModelMixins`‑Namespace. Das sorgt für etwas Ordnung und ein klareres Wiederverwendungsmuster.

**Standardisierte Strukturen**

- `createdAt` & `updatedAt` treten gemeinsam auf.  
- `id`, nicht `ID` oder `_id`.

```tsx
// `src/types/mixins.d.ts`
namespace ModelMixins {
  interface Identity {
    id: number;
  }
  interface Timestamp {
    createdAt: Date;
    updatedAt: Date;
  }
  type Instance = ModelMixins.Identity & ModelMixins.Timestamp;
  interface HashedPassword {
    passwordHash: string;
  }
  interface InputPassword {
    password: string;
  }
}
```

**Verwendung von Typ‑Unions**

```tsx
// `src/types/user.d.ts`
export interface UserBase {
  name: string;
  bio: string;
  social: Record<"facebook" | "instagram" | "github", URL>;
}
// Einzelner `User`‑Typ, der mittels Typ‑Union dynamisch
// die Vor‑ und Nach‑Erstellungs‑Zustände abbildet.
export type User =
  | (UserBase & ModelMixins.Instance & ModelMixins.HashedPassword)
  | (UserBase & ModelMixins.InputPassword);
```

Falls gewünscht, können Sie auch einzelne benannte Typen exportieren:

```tsx
/** User‑Payload für die Registrierung, inklusive `password`‑Feld */
export type UserPayload = UserBase & ModelMixins.Instance & ModelMixins.HashedPassword;
/** Repräsentiert den vom Server zurückgegebenen User‑Typ. */
export type UserInstance = UserBase & ModelMixins.InputPassword;
```

#### Praxisbeispiel

---

Hier ein `upsert()`‑Beispiel, das den `in`‑Operator nutzt, um zwischen den Typen `UserInstance` und `UserPayload` zu unterscheiden.

```tsx
function upsert(user: User) {
  if ("id" in user) {
    // TypeScript weiß, dass `user` hier Felder aus Instance (id, createdAt, usw.) hat
    return updateUser(user.id, user);
  } else {
    // TypeScript erkennt, dass dies die `UserBase & ModelMixins.InputPassword`‑Variante des Benutzers ist.
    return createUser(user);
  }
}
```
