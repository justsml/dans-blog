### Approcci al Design dei Tipi

Probabilmente vi siete imbattuti o avete scritto pattern diversi per le "implementazioni dei tipi". Soprattutto quando si consumano dati da API di terze parti.

**Nota:** Sto ignorando intenzionalmente i processi "tradizionali" per la costruzione di Diagrammi di Relazione Entità (ERD) o le gerarchie di ereditarietà della Programmazione Orientata agli Oggetti (OOP). Qui stiamo costruendo tipi per rappresentare dati API semi-strutturati.

Esploriamo due approcci di alto livello: **Oggetto unico e grande** (Top-down) contro **Tipi nominati multipli** (Bottom-up).

#### Oggetto unico e grande

Privilegia l'esplicitazione rispetto alla riutilizzabilità e al principio DRY.

**Bonus:** l'esperienza IDE/Dev è ottima, poiché i tooltip includono un'anteprima più completa, senza fronzoli.

```tsx
interface ProductDetails {
  name: string;
  seller: { name: string };
  availability: Array<{ warehouseId: string; quantity: number }>;
  reviews: Array<{ authorId: number; stars: number }>;
}
```

Dato che privilegiamo la leggibilità esplicita, è lecito indulgere in _una certa_ ripetitività (nei limiti del ragionevole). Quando gruppi di proprietà si ripetono _molte_ volte, puoi estrarre i campi ripetuti in un tipo nominato.

#### Tipi nominati multipli

Privilegia la riutilizzabilità e il principio DRY.

<!-- La leggibilità è una metrica curiosa. Spesso è buona o **eccellente quando i tipi/file sono pochi.** **Inevitabilmente i tipi tendono a proliferare,** aggiungendo proprietà sempre più numerose. **La leggibilità ne risente.** -->

Questo approccio è probabilmente quello preferito, e di gran lunga.

```ts
interface ProductDetails {
  name: string;
  seller: Seller;
  reviews: Reviews[];
  availability: Availability[];
}
interface Seller { name: string; }
interface Availability { warehouseId: string; quantity: number; }
interface Reviews { authorId: number; stars: number; }
```

Nel complesso, questo approccio è ottimo. Ma non è privo di svantaggi.

- **La leggibilità** è eccellente all'inizio; tuttavia, _può_ risentirne man mano che dimensioni e numero di tipi crescono.
- Ferreo rispetto al principio DRY, ma a quale prezzo? (Ne parleremo più avanti.)
- L'esperienza di sviluppo può risentirne, poiché i tooltip sono meno informativi.

> ⚠️ A partire (circa) da TypeScript v3, il Language Server tronca i tooltip, omettendo le proprietà annidate.
> 💡 Esistono alcuni accorgimenti per migliorare la situazione. Prova a tenere premuto `Cmd o Ctrl`, quindi passa il mouse sui vari nomi di tipo: dovresti vedere almeno un ulteriore 'livello' di proprietà nel tooltip.

Perché dobbiamo scegliere tra questi due approcci? (Un tipo monolitico vs. sottotipi nominati.)

### Tecnica #1: Perché non avere tutto

Possiamo avere tutto?

- Chiarezza dei tipi "ad alto livello"?
- Più sottotipi nominati?
- Senza duplicazione?

> ✅ SÌ! 🎉

```tsx
export interface ProductDetails {
  name: string;
  seller: { name: string };
  reviews: Array<{ authorId: number; stars: number }>;
  availability: Array<{ warehouseId: string; quantity: number }>;
}
export type Seller = ProductDetails["seller"];
export type Review = ProductDetails["reviews"][number];
export type Availability = ProductDetails["availability"][number];
```

1.  Crea tipi strutturati "Primari" di grandi dimensioni.
2.  Esporta i sotto-tipi derivati dal tipo Primario.

Questo approccio brilla davvero nei sistemi in cui gli oggetti "di alto livello" beneficiano di una documentazione centralizzata. Inoltre, questa tecnica favorisce il riutilizzo tra molti casi d'uso: Model, Services, Query Results, ecc.

### Tecnica #2: Mix-in

Questa strategia consiste nel combinare i **campi giusti**, con i **nomi giusti**, per **rappresentare oggetti logici singoli**. L'obiettivo è affrontare in modo efficiente più casi d'uso utilizzando TypeScript Utilities e Type Unions.

Questo approccio si discosta dall'ereditarietà e dalle gerarchie OOP tradizionali, che mirano a creare strati di oggetti in tassonomie strettamente vincolate. L'**approccio mix-in riguarda tipi piatti e debolmente correlati**, raggruppando campi affini riducendo al contempo la duplicazione.

#### Esempi di Mix-in

```tsx
interface TodoModel {
  text: string;
  complete: boolean;
}
interface InstanceMixin {
  id: number;
}
/** TodoDraft rappresenta lo stato del Form, potenzialmente tutto undefined */
export type TodoDraft = Partial<TodoModel>;
/** Todo rappresenta un record di istanza Todo dal database */
export type Todo = TodoModel & InstanceMixin;
```

#### Esempio `User`

```tsx
interface User {
  id: number;
  name: string;
  bio: string;
  social: Record<"facebook" | "instagram" | "github", URL>;
}
```

Rappresentiamo l'`User` prima e dopo il salvataggio nel database.

```tsx
// Campi core di User (ad esempio per un <form>)
interface UserBase {
  name: string;
  bio: string;
  social: Record<"facebook" | "instagram" | "github", URL>;
}
// Campi provenienti dal database
interface InstanceMixin {
  id: number;
  createdAt: Date;
  updatedAt: Date;
}
// Un'istanza di User **con tutti i campi**
type UserInstance = InstanceMixin & UserBase;
```

Ora possiamo definire esattamente i campi di cui abbiamo bisogno (come `password` per create/update, ma non inclusi nelle query di `UserInstance`).

```tsx
interface UserBase {
  name: string;
  bio: string;
  social: Record<"facebook" | "instagram" | "github", URL>;
}
interface InstanceMixin {
  id: number;
  createdAt: Date;
  updatedAt: Date;
}
/** Payload di User per la registrazione, include il campo `password` */
export type UserPayload = UserBase & { password: string };
/** Rappresenta il tipo User restituito dal server. */
export type UserInstance = UserBase & InstanceMixin;
```

1.  "È una buona pratica?"
2.  "Dovrei provarlo?"

Non lo so. Continuiamo!

### Tecnica #3: Organizzazione tramite namespace

Qui dichiariamo un namespace `ModelMixins`. Questo offre una struttura migliore e un pattern di riutilizzo più chiaro.

**Strutture standardizzate**

- `createdAt` e `updatedAt` coesistono.
- `id`, non `ID` o `_id`.

```tsx
// `src/types/mixins.d.ts`
namespace ModelMixins {
  interface Identity {
    id: number;
  }
  interface Timestamp {
    createdAt: Date;
    updatedAt: Date;
  }
  type Instance = ModelMixins.Identity & ModelMixins.Timestamp;
  interface HashedPassword {
    passwordHash: string;
  }
  interface InputPassword {
    password: string;
  }
}
```

**Utilizzo delle unioni di tipi**

```tsx
// `src/types/user.d.ts`
export interface UserBase {
  name: string;
  bio: string;
  social: Record<"facebook" | "instagram" | "github", URL>;
}
// Single `User` type, using Type Union to dynamically
//  represent the pre- & post-creation states.
export type User =
  | (UserBase & ModelMixins.Instance & ModelMixins.HashedPassword)
  | (UserBase & ModelMixins.InputPassword);
```

Se lo desideri, puoi anche esportare tipi nominati individualmente:

```tsx
/** User payload for signup, including `password` field */
export type UserPayload = UserBase & ModelMixins.Instance & ModelMixins.HashedPassword;
/** Represents User type returned from server. */
export type UserInstance = UserBase & ModelMixins.InputPassword;
```

#### Utilizzo pratico

Ecco una funzione `upsert()` che sfrutta l'operatore `in` per distinguere tra i tipi `UserInstance` e `UserPayload`.

```tsx
function upsert(user: User) {
  if ("id" in user) {
    // TypeScript knows `user` here has fields from Instance (id, createdAt, etc)
    return updateUser(user.id, user);
  } else {
    // TypeScript knows this must be the `UserBase & ModelMixins.InputPassword` version of user.
    return createUser(user);
  }
}
```
