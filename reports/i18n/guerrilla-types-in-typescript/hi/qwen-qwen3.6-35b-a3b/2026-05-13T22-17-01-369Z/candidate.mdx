5.  टाइप के वैरिएंट्स को 'स्वचालित' रूप से मैच करने के लिए [Unions](https://www.typescriptlang.org/docs/handbook/typescript-in-5-minutes-func.html#discriminated-unions) का उपयोग करें। -->

### टाइप डिज़ाइन के दृष्टिकोण

आपने शायद "टाइप इम्प्लीमेंटेशन" के आसपास विभिन्न पैटर्न देखे या लिखे होंगे। खासकर जब 3rd party APIs से डेटा consume कर रहे हों।

**नोट:** मैं जानबूझकर "पारंपरिक" प्रक्रियाओं, जैसे Entity Relationship Diagrams (ERD) बनाना या Object Oriented Programming (OOP) इनहेरिटेंस हायरार्की, को नज़रअंदाज़ कर रहा हूँ। यहाँ, हम semi-structured API डेटा को प्रस्तुत करने के लिए टाइप्स बना रहे हैं।

आइए दो हाई-लेवल दृष्टिकोणों पर चर्चा करें: **Single large object** (Top-down) बनाम **Multiple named types** (Bottom-up.)

#### Single large object

स्पष्टता को पुनः प्रयोग और DRY सिद्धांत पर प्राथमिकता देता है।

**अतिरिक्त लाभ:** IDE/डेवलपर अनुभव बेहतरीन है, क्योंकि टूलटिप्स में अधिक संपूर्ण पूर्वावलोकन शामिल होता है — बिना किसी अतिरिक्त झंझट के।

```tsx
interface ProductDetails {
  name: string;
  seller: { name: string };
  availability: Array<{ warehouseId: string; quantity: number }>;
  reviews: Array<{ authorId: number; stars: number }>;
}
```

चूँकि हम स्पष्ट पठनीयता को प्राथमिकता दे रहे हैं, इसलिए _कुछ_ दोहराव (युक्तिसंगत सीमा के भीतर) में ढलना ठीक है। जब गुणों के समूह _कई_ बार दोहराए जाते हैं, तो दोहराए गए फ़ील्ड्स को एक नामित टाइप में निकालने में संकोच न करें।

#### कई नामित टाइप्स

पुनः प्रयोग और DRY सिद्धांत को प्राथमिकता देता है।

<!-- पठनीयता एक अजीब मापदंड है। चूँकि पठनीयता अक्सर तब अच्छी या **बेहतरीन होती है जब टाइप्स/फ़ाइलें कम हों।** **अनिवार रूप से टाइप्स का प्रसार होता जाता है,** जिसमें लगातार अधिक गुण जुड़ते जाते हैं। **पठनीयता प्रभावित होती है।** -->

यह दृष्टिकोण संभवतः काफी अंतर से सबसे अधिक पसंद किया जाने वाला विकल्प है।

```ts
interface ProductDetails {
  name: string;
  seller: Seller;
  reviews: Reviews[];
  availability: Availability[];
}
interface Seller { name: string; }
interface Availability { warehouseId: string; quantity: number; }
interface Reviews { authorId: number; stars: number; }
```

समग्र रूप से, यह दृष्टिकोण बेहतरीन है। लेकिन इसकी कुछ कमी भी हैं।

- **पठनीयता** शुरू में उत्कृष्ट होती है; हालाँकि, जैसे-जैसे टाइप्स का आकार और संख्या बढ़ती है, यह _प्रभावित_ हो सकती है।
- लगातार DRY सिद्धांत का पालन, लेकिन इसकी कीमत क्या है? (इस पर बाद में चर्चा होगी।)
- डेवलपर अनुभव प्रभावित हो सकता है, क्योंकि टूलटिप्स कम जानकारीपूर्ण होते हैं।

> ⚠️ (लगभग) TypeScript v3 से, लैंग्वेज सर्वर टूलटिप्स को काट देता है, जिसमें नेस्टेड गुण छोड़ दिए जाते हैं।
> 💡 स्थिति में थोड़ा सुधार करने के लिए कुछ तरकीबें हैं। `Cmd or Ctrl` दबाए रखें, फिर विभिन्न टाइप नामों पर होवर करें — आपको टूलटिप में कम से कम गुणों का एक अतिरिक्त 'परत' दिखाई देना चाहिए।

हमें इन दोनों दृष्टिकोणों के बीच चुनाई क्यों करनी पड़ती है? (विस्तृत टाइप बनाम नामित सब-टाइप्स।)

### तकनीक #1: क्यों न सब कुछ हो

क्या हम सब कुछ प्राप्त कर सकते हैं?

- क्या "बड़े चित्र" वाले टाइप्स की स्पष्टता?
- साथ ही नामित सब-टाइप्स?
- बिना किसी दोहराव के?

> ✅ हाँ! 🎉

<!-- ### विचार करने योग्य कुछ बातें

- `Product` -> `Seller` जैसे `one-to-one` संबंध को आप कैसे दर्शाते हैं?
- `one-to-many` संबंधों की क्या स्थिति है? जैसे `Reviews` या `Photos`?
- क्या इसे Prisma को सौंप दें? (बुरा विचार नहीं है, लेकिन यह लेख गुप्त रूप से कुछ TypeScript सीखने के बारे में है...) -->

<!-- यह दृष्टिकोण Model फ़ील्ड नामों को कभी भी दोहराने के अभ्यास पर आधारित है। इससे "बड़ा चित्र" अधिक स्पष्ट हो जाता है (एक ही स्थान पर)। सबसे बड़े, उच्च-स्तरीय टाइप से शुरू करना, और उससे सरल टाइप्स को व्युत्पन्न करना। -->

<!-- जब कुछ संरचित array/object डेटा दिया जाता है, तो कई TypeScript कोडर टाइप्स बनाने के इच्छुक महसूस करते हैं। टाइप्स की भारी मात्रा। अंततः एक परतों की जंजीर बन जाती है, जो सरल टाइप्स से बनी होती है जो और अधिक जटिल टाइप्स बनाती हैं।

या शायद आप उसी प्रकार के डेवलपर हैं जो उच्च-स्तरीय टाइप से शुरू करते हैं, और पेड़ में अगले सब-टाइप को लिखने के लिए पर्याप्त ढांचा तैयार करते हैं? -->

```tsx
export interface ProductDetails {
  name: string;
  seller: { name: string };
  reviews: Array<{ authorId: number; stars: number }>;
  availability: Array<{ warehouseId: string; quantity: number }>;
}
export type Seller = ProductDetails["seller"];
export type Review = ProductDetails["reviews"][number];
export type Availability = ProductDetails["availability"][number];
```

1. बड़े "प्राथमिक" संरचित टाइप्स बनाएं।
2. प्राथमिक टाइप से व्युत्पन्न सब-टाइप्स को एक्सपोर्ट करें।

यह दृष्टिकोण उन सिस्टम में वाकई चमकता है जहाँ "उच्च-स्तरीय" ऑब्जेक्ट्स को एक ही जगह दस्तावेज़ीकरण से लाभ होता है।
साथ ही, यह तकनीक कई उपयोग मामलों के बीच पुनः उपयोग को सपोर्ट करती है: मॉडल्स, सर्विसेस, क्वेरी रिजल्ट्स, आदि।

### तकनीक #2: मिक्स-इन

यह रणनीति मुख्य रूप से **सही फील्ड्स** को **सही नामों** के साथ जोड़कर **एकल तार्किक ऑब्जेक्ट्स** को **प्रस्तुत** करने के बारे में है। लक्ष्य टाइपस्क्रिप्ट यूटिलिटीज और टाइप यूनियन्स का उपयोग करके कई उपयोग मामलों को कुशलता से संभालना है।

यह दृष्टिकोण पारंपरिक OOP इनहेरिटेंस और हैरार्की से अलग है, जो ऑब्जेक्ट्स की परतों को कसकर बंधे टैक्सोनॉमी में बनाने का लक्ष्य रखता है। **मिक्स-इन दृष्टिकोण फ्लैट और कमजोर रूप से जुड़े टाइप्स के बारे में है**, जो संबंधित फील्ड्स को ग्रुप करते हुए डुप्लिकेशन को कम करता है।

#### मिक्स-इन उदाहरण

```tsx
interface TodoModel {
  text: string;
  complete: boolean;
}
interface InstanceMixin {
  id: number;
}
/** TodoDraft represents Form state, possibly all undefined */
export type TodoDraft = Partial<TodoModel>;
/** Todo represents a Todo instance record from the database */
export type Todo = TodoModel & InstanceMixin;
```

#### उदाहरण `User`

```tsx
interface User {
  id: number;
  name: string;
  bio: string;
  social: Record<"facebook" | "instagram" | "github", URL>;
}
```

आइए डेटाबेस में सेव करने से पहले और बाद में `User` को प्रस्तुत करें।

```tsx
// Core User fields (say for a <form>)
interface UserBase {
  name: string;
  bio: string;
  social: Record<"facebook" | "instagram" | "github", URL>;
}
// Fields from the database
interface InstanceMixin {
  id: number;
  createdAt: Date;
  updatedAt: Date;
}
// A User **instance** - with all fields
type UserInstance = InstanceMixin & UserBase;
```

अब हम अपनी आवश्यकता के सटीक फील्ड्स को ढाल सकते हैं (जैसे `password` create/update के लिए, लेकिन `UserInstance` की क्वेरीज में शामिल नहीं है)।

```tsx
interface UserBase {
  name: string;
  bio: string;
  social: Record<"facebook" | "instagram" | "github", URL>;
}
interface InstanceMixin {
  id: number;
  createdAt: Date;
  updatedAt: Date;
}
/** User payload for signup, including `password` field */
export type UserPayload = UserBase & { password: string };
/** Represents User type returned from server. */
export type UserInstance = UserBase & InstanceMixin;
```

1. "क्या यह एक अच्छी प्रैक्टिस है?"
2. "क्या मुझे इसे आज़माना चाहिए?"

पता नहीं। चलिए आगे बढ़ते हैं!

### तकनीक #3: नामस्थानों (Namespaces) के साथ संगठन

यहाँ हम `ModelMixins` नामस्थान घोषित करते हैं। इससे कोड को कुछ संरचना मिलती है और पुनः उपयोग का पैटर्न अधिक स्पष्ट हो जाता है।

**मानकीकृत संरचनाएं**

- `createdAt` और `updatedAt` साथ-साथ मौजूद हैं।
- `id`, न कि `ID` या `_id`।

```tsx
// `src/types/mixins.d.ts`
namespace ModelMixins {
  interface Identity {
    id: number;
  }
  interface Timestamp {
    createdAt: Date;
    updatedAt: Date;
  }
  type Instance = ModelMixins.Identity & ModelMixins.Timestamp;
  interface HashedPassword {
    passwordHash: string;
  }
  interface InputPassword {
    password: string;
  }
}
```

**टाइप यूनियन का उपयोग**

```tsx
// `src/types/user.d.ts`
export interface UserBase {
  name: string;
  bio: string;
  social: Record<"facebook" | "instagram" | "github", URL>;
}
// Single `User` type, using Type Union to dynamically
//  represent the pre- & post-creation states.
export type User =
  | (UserBase & ModelMixins.Instance & ModelMixins.HashedPassword)
  | (UserBase & ModelMixins.InputPassword);
```

यदि चाहें, तो आप व्यक्तिगत नामित टाइप्स को भी एक्सपोर्ट कर सकते हैं:

```tsx
/** User payload for signup, including `password` field */
export type UserPayload = UserBase & ModelMixins.Instance & ModelMixins.HashedPassword;
/** Represents User type returned from server. */
export type UserInstance = UserBase & ModelMixins.InputPassword;
```

#### वास्तविक उपयोग

यहाँ एक `upsert()` फ़ंक्शन है जो `UserInstance` और `UserPayload` टाइप्स के बीच अंतर करने के लिए `in` ऑपरेटर का उपयोग करता है।

```tsx
function upsert(user: User) {
  if ("id" in user) {
    // TypeScript knows `user` here has fields from Instance (id, createdAt, etc)
    return updateUser(user.id, user);
  } else {
    // TypeScript knows this must be the `UserBase & ModelMixins.InputPassword` version of user.
    return createUser(user);
  }
}
```
