## Todesschlacht: Git Rebase vs. (Squash) Merge!

Soll ich Rebase durchführen? Oder Squash Merge?

- Ist es eine persönliche Vorliebe?
  - _Antwort: Nicht, wenn eine oder mehrere Teams beteiligt sind! **Jede dieser Optionen wird die Benutzbarkeit der anderen beeinflussen!**_

### Warum erzeugt dieses Thema religiöse Begeisterung?

Einige Ingenieure nutzen ihr Wissen über Git (& das Terminal) als Signal für ihr relatives Können. Und jede Praxis, die mit unserem Ich/Ego verknüpft ist, kann es unmöglich machen, sie unvoreingenommen zu analysieren, geschweige denn zu ändern.

Weitere Faktoren sind wahrscheinlich Vertrautheit & Survivorship Bias, die unsere eigenen Bewertungen und Annahmen weiter verschleiern können.

### Szenario: Erstellen eines überarbeiteten Release-Tags

Lassen Sie uns den Prozess vergleichen, ein Tag-Release zu erstellen, bei dem einige kürzlich auf `main` gemachte Commits ausgeschlossen werden.

![Git Tag Releasing from main with 2 feature branches](../git-branching-with-main-simplified.svg)

### Die Rebase-Methode

Denkmodell: "Ich möchte eine alternative Version einer bestehenden Historie erstellen. (z. B. Ich habe einen kleinen Fehler 16 Merges zurück gemacht und benötige möglicherweise feingranulare Kontrolle, um ihn zu korrigieren. Dazu könnte ich in eine scheinbar endlose Schleife aus Konflikten und `--continue` geraten.)"

1. Aktualisiere: `git checkout main` && `git pull`  
2. Erstelle neuen Branch: `git checkout -b release/hot-newness-and-stuff`  
3. Starte interaktives Rebase & gib die git-Referenz für den Zeitpunkt an, zu dem du zurückreisen willst. `git rebase -i HEAD~6` (Hinweis: `HEAD~6` ist eine Kurzschreibweise für `6 Commits zurück`)  
4. Entferne gewünschte Commits, indem du ihren Präfix in `drop` änderst. Speichere und schließe den Editor.  
5. Behebe Zusammenführungs-Konflikte, `git add .` && `git rebase --continue` (NICHT `git commit` verwenden).  
6. Wiederhole den vorherigen Schritt, bis alles abgeschlossen ist.  
7. Tagge/pushe mit dem aktuellen Prozess. Beispiel: `git tag -a v1.2.3 -m 'Release v1.2.3'` && `git push --tags`  

#### Vorteile  

- 🔌 Absolute Macht. Du kannst die Historie ändern.  
  <!-- - 🎭 Trainiere deine Engineering-Theater-Fähigkeiten. -->  

#### Nachteile  

- 😰 Absolute Macht. Du kannst die Historie ändern. (Okay, ein Vorteil & Nachteil...)  
- 🔂 Du kannst in eine scheinbar endlose Schleife aus Konflikten & `--continue` geraten. (Manchmal sogar mit `git rerere`)  
- 🙀 Zerstört wichtige Zusammenarbeit-Funktionen: verlorene/verwaiste PR-Kommentare. Unhöflich.  
- 🖇️ Permalinks können nicht so permanent sein.  

### Die (Squash-)Zusammenführungs-Methode

Mentales Modell: „Ich möchte eine benutzerdefinierte Freigabe erstellen, ab einem bestimmten Punkt, und beliebige Zielzweige (Branches) einbeziehen.“

1. Aktuellste Version holen: `git checkout main` && `git pull`  
2. Neuen Zweig erstellen: `git checkout -b release/hot-newness-and-stuff`  
3. Gewünschte Zweige und/oder Commits einfügen: `git merge --no-ff feature/hot-newness bug/fix-123` (wenn möglich, den `--no-ff`-Flag verwenden.)  
4. Eventuelle Merge-Konflikte beheben (falls nötig).  
5. Mit aktuellem Prozess taggen/pushen. Beispiel: `git tag -a v1.2.3 -m 'Release v1.2.3'` && `git push --tags`  

#### Vorteile  

- 💪 Weniger Prozess, weniger Konflikte insgesamt, nutzt bestehendes Git-Kommandowissen.  
- 🚀 Erlaubt dir, auf der PR/Branch-Ebene zu denken, ignoriert Commit-Granularität (es sei denn, nötig).  
- 🦺 Nicht-zerstörerisch. Du kannst jederzeit zurückgehen und/oder neue Zweige erstellen.  
- 🎥 Behält bestehende Commits und Nachrichten als Ganzes, was weniger „blame“-Rauschen erzeugt.  

#### Nachteile  

- 🔏 Ändern von Commit-Nachrichten ist schwieriger.  
- 🤐 Verbergen deiner Arbeit ist schwieriger.

### Fazit

Am Ende des Tages **gewinnt ein einfacherer Prozess mit weniger Risiko.**

Obwohl _Rebasers_ tatsächlich Wege haben, ihre Probleme zu lösen (oder zu umgehen), **bleibt die Tatsache: Du wirst irgendwann einen Schwarzen Gürtel in Git-Fu brauchen.** (Zum Beispiel: Selbst ein einfacher `git push` kann zusätzliche Komplexität aufweisen: War es `git push --force` oder `git push --force-with-lease`? Warum sich das überhaupt antun?)

Es gibt einen weiteren Grund, warum **Rebasing** bei der Erstellung einer überarbeiteten Historie **immer einen Nachteil gegenüber `git merge ...` haben wird.** Ein `git merge` erlaubt der `git`-CLI, **fortgeschrittene Algorithmen** einzusetzen, um Konflikte zu vermeiden, indem sie den HEAD jedes Branches analysiert.

Das kann intelligenter sein, weil jede Zusammenführung sich nur um den neuesten Zustand jedes gewünschten Branches kümmert. **Rebasing hingegen muss die Commit-Historie in der angegebenen Reihenfolge wiederholen (oder verwerfen)**. Dies **beschränkt die Fähigkeit von `git`, die Zusammenführung zu optimieren**, da es **nur zwei Commits gleichzeitig vergleicht**.

Letztendlich **bedeutet Rebasing, dass Sie sich gelegentlich irrelevante alte Commits und Konflikte erneut ansehen werden** – selbst wenn Sie wissen, dass diese mittlerweile entfernt oder gelöst wurden.

### Zusammenfassung

- 💃 Antwort: **SQUASH MERGE** Ihre PRs in `main` ein.
  - Ihre Branch-Historie ist vorhanden, falls benötigt, und `main` bleibt relativ „sauber“.
- _🔤 Immer Commits erstellen!_
  - In >95 % der Unternehmensprojekte ist die „Backup-Mentalität“ der „Gestaltungskunst“-Mentalität vorzuziehen. Mit der Zeit verblasst die Bedeutung Ihrer Commit-Nachrichten schneller, als die Logik und Tests des Codes, deren Relevanz erhalten bleibt.

- Sie können das spezielle Trennzeichen "--" mit `git checkout` verwenden, um in der aktuellen Branch zu bleiben, während Sie die angegebenen Dateien kopieren:  
- `git checkout feature/half-a-feature **--** <Ordner- oder Dateipfad>`  
- Stellen Sie sicher, dass Sie alle Änderungen, die Sie behalten möchten, zuerst committen, da dies alle lokalen Änderungen überschreiben wird.
