## مباراة الموت: Git Rebase مقابل (Squash) Merge!

هل يجب أن أستخدم Rebase؟ أم Squash Merge؟

- هل هو مجرد تفضيل شخصي؟
  - _الإجابة: ليس عندما يشارك فريق أو أكثر! **كل خيار سيؤثر على قابلية استخدام الآخر**!_

### لماذا يثير هذا الموضوع حماسًا دينيًا؟

بعض المهندسين يستخدمون معرفة git (والطرفية) كإشارة إلى مستوى مهارتهم النسبي. وأي ممارسة مرتبطة بهويتنا/أنا يمكن أن تكون مستحيلة التحليل بموضوعية، ناهيك عن التغيير.

عوامل أخرى ربما تشمل الألفة وتحامل البقاء التي يمكن أن تعكر تقييمنا وافتراضاتنا الخاصة.

### السؤال الرئيسي: ما هو هدف الـ git commit؟

1.  هل تقوم بالـ commit مبكرًا وبشكل متكرر؟ باستخدام عقلية "نقطة تفتيش" أو نسخة احتياطية؟
    - حيث يُسجل كل شيء، حتى البدايات الفاشلة والتجارب؟ (مثال: `git commit -am "Updated deps" && git push`، كرر بانتظام)
    - ربما تكون رسائل الـ commit أقل أهمية بالنسبة لك من الكود؟
2.  أم أن الـ commits لديك عمل فني منظم ومصمم بعناية؟
    - ربما كل commit وحدة عمل ذاتية ومتكاملة؟ (مثال: `git add package.json && git commit -m "Updated deps"`)
    - أو أنك لا تحتمل سجلات الـ commit "الفوضوية"؟
    - هل تتضمن مراجعات الـ PR الخاصة بك عادة مراجعة كل commit على حدة؟

نظرًا لتباين العقليات حول استراتيجية الـ commit، ليس من المستغرب وجود الكثير من الالتباس حول الطريقة "الصحيحة" لاستخدام Git.

### سيناريو: إنشاء علامة إصدار مُعدَّلة

لنقارن عملية إنشاء علامة إصدار مع استبعاد بعض الـ commits الأخيرة على `main`.

### طريقة الـ Rebase

النموذج الذهني: "أريد إنشاء نسخة بديلة من تاريخ موجود. (مثلاً ارتكبت خطأً قبل 16 دمجًا، وقد أحتاج إلى تحكم دقيق لتصحيحه. وربما أعلق في دورة لا نهائية من النزاعات و`--continue`.)"

1.  احصل على أحدث نسخة: `git checkout main` && `git pull`
2.  أنشئ فرعًا جديدًا: `git checkout -b release/hot-newness-and-stuff`
3.  ابدأ rebase تفاعليًا وأدرج المرجع الذي تريد العودة إليه في الزمن. `git rebase -i HEAD~6` (ملاحظة: `HEAD~6` هو اختصار مرجعي لـ `6 commits ago`)
4.  احذف الـ commit(s) المطلوبة بتغيير بادئتها إلى `drop`. احفظ وأغلق المحرر.
5.  أصلح تعارضات الدمج، `git add .` && `git rebase --continue` (لا تقم بـ `git commit`).
6.  كرّر الخطوة السابقة حتى تكتمل العملية.
7.  ضع علامة/ادفع باستخدام العملية الحالية. مثال `git tag -a v1.2.3 -m 'Release v1.2.3'` && `git push --tags`

- 🔌 سلطة مطلقة. يمكنك تغيير التاريخ.

#### العيوب

- 😰 سلطة مطلقة. يمكنك تغيير التاريخ. (حسنًا، ميزة وعيب...)
- 🔂 قد تنتهي بك المطاف في دورة لا نهائية من النزاعات و `—-continue`. (أحيانًا حتى مع `git rerere`)
- 🙀 يكسر ميزات التعاون الأساسية: فقدان/تعليق طلبات السحب اليتيمة. أمر فظ.
- 🖇️ قد لا تكون الروابط الدائمة ثابتة كما ينبغي.

### طريقة (Squash) Merge

نموذج التفكير: "أريد إصدارًا مخصصًا، يبدأ من نقطة معينة، ويشمل أي فرع (فروع) مرغوب."

1.  احصل على أحدث نسخة: `git checkout main` && `git pull`
2.  أنشئ فرعًا جديدًا: `git checkout -b release/hot-newness-and-stuff`
3.  دمج الفروع و/أو الالتزامات المطلوبة: `git merge --no-ff feature/hot-newness bug/fix-123` (استخدم خيار `--no-ff` حيثما أمكن.)
4.  أصلح أي تعارض دمج (إذا ظهر.)
5.  ضع علامة/ادفع باستخدام العملية الحالية. مثال `git tag -a v1.2.3 -m 'Release v1.2.3'` && `git push --tags`

#### الإيجابيات

- 💪 عملية أقل، نزاعات أقل بشكل عام، وتستفيد من معرفة أوامر git الحالية.
- 🚀 تسمح لك بالتفكير على مستوى طلب السحب/الفرع الأعلى، متجاهلاً تفاصيل الالتزام الفردي (إلا إذا كان ذلك ضروريًا.)
- 🦺 غير مدمرة. يمكنك الرجوع أو إنشاء فروع جديدة في أي وقت.
- 🎥 تترك الالتزامات والرسائل الحالية كما هي، مما يقلل من ضوضاء "اللوم".

#### العيوب

- 🔏 أصعب تعديل رسائل الالتزام.
- 🤐 أصعب إخفاء عملك.

### الخلاصة

في النهاية، **عملية أبسط مع مخاطر أقل هي التي يجب أن تنتصر**.

على الرغم من أن _المعيدين_ لديهم طرق لحل (أو تجنب) مشاكلهم، **الواقع يبقى: ستحتاج في النهاية إلى حزام أسود في git fu**. (مثلاً حتى أمر `git push` المتواضع قد يواجه تعقيدًا إضافيًا: هل كان `git push --force` أم `git push --force-with-lease`؟ لماذا نتعامل مع ذلك أصلاً؟)

هناك سبب آخر يجعل **إعادة الترتيب** لإنشاء سجل مُعدَّل **سيظل دائمًا في وضع غير ملائم** مقارنةً بـ **`git merge ...`**. يسمح `git merge` لواجهة سطر أوامر `git` بتطبيق خوارزميات متقدمة لتفادي النزاعات عبر تحليل رأس كل فرع.

يمكن أن يكون هذا أذكى لأن كل دمج يهتم فقط بالحالة الأخيرة لكل فرع مطلوب، بينما **يجب على الـ rebase إعادة تشغيل (أو حذف) سجل الالتزامات بالتسلسل** المحدد. هذا **يحد من قدرة `git` على التحسين** لأن عملية الدمج **تقارن التزامين فقط** في كل مرة.

في النهاية، **يعني الـ rebase أنك ستجد نفسك بين الحين والآخر تعيد تجربة الالتزامات والنزاعات القديمة غير ذات الصلة** - حتى لو كنت تعلم أنها أُزيلت أو حُلت بالفعل.

### الملخص

- 💃 الجواب: **استخدم SQUASH MERGE** لطلبات السحب الخاصة بك على `main`.
  - سيظل تاريخ فرعك موجودًا إذا احتجت إليه، وسيبقى `main` نظيفًا نسبيًا.
- _🔤 دائمًا استمر في الالتزام!_
  - في أكثر من 95٪ من المشاريع المؤسسية، يُفضَّل "عقلية النسخ الاحتياطي" على "عقلية الفن المنحوت". مع مرور الوقت، سيتلاشى معنى رسائل الالتزام أسرع بكثير من الشيفرة التي ستظل منطقها واختباراتها تحتفظ بأهميتها.
