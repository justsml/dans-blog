## Смертельный бой: Git Rebase против (Squash) Merge!

Стоит ли делать Rebase? Или Squash Merge?

- Является ли это личным предпочтением?
  - _Ответ: Нет, если вовлечены одна или несколько команд! **Любой выбор повлияет на удобство использования другого!**_

### Почему эта тема вызывает религиозный фанатизм?

Некоторые инженеры используют знание git (& терминала) как показатель своего относительного уровня навыков. И любая практика, связанная с нашей идентичностью/эго, может быть невозможно проанализировать беспристрастно, не говоря уже о том, чтобы изменить.

Другие факторы, вероятно, включают Знакомство & Смещение выживших, которые могут дополнительно затуманивать наше собственное оценку и предположения.

### Основной вопрос: В чем цель коммита в Git?

1.  Вы коммитите рано и часто? С мыслью о "точке сохранения" или резервной копии?
    - Где записывается всё, даже неудачные начинания и эксперименты? (например, `git commit -am "Updated deps" && git push`, повторяется регулярно)
    - Возможно, сообщения коммитов важнее для вас, чем сам код?
1.  Или ваши коммиты — это тщательно отобранный, вылепленный шедевр?
    - Может быть, каждый коммит представляет собой самодостаточную, атомарную единицу работы? (например, `git add package.json && git commit -m "Updated deps"`)
    - Или вы просто не можете терпеть "загрязненные" логи коммитов?
    - Часто ли ваши обзоры PR включают проверку коммит за коммитом?

### Сценарий: Создание пересмотренного тега релиза

Сравним процесс создания тега релиза, исключающего некоторые недавние коммиты в ветке `main`.

![Релиз тега из main с 2 ветками функций](../git-branching-with-main-simplified.svg)

### Способ через Rebase

Ментальная модель: "Я хочу создать альтернативную версию существующей истории. (Например, я допустил ошибку 16 слияний назад и, возможно, потребуется тонкая настройка для её исправления. Также могу попасть в бесконечный цикл конфликтов и `--continue`.)"

1. Получить последние изменения: `git checkout main` && `git pull`  
2. Создать новую ветку: `git checkout -b release/hot-newness-and-stuff`  
3. Запустить интерактивный rebase и указать git-ссылку на точку, с которой хотите начать. `git rebase -i HEAD~6` (Примечание: `HEAD~6` — это сокращённая форма для «6 коммитов назад»)  
4. Удалить нужные коммиты, изменив их префикс на `drop`. Сохраните и закройте редактор.  
5. Исправьте конфликты слияния, `git add .` && `git rebase --continue` (не используйте `git commit`!).  
6. Повторяйте предыдущий шаг до завершения.  
7. Проставьте тег и отправьте на удалённый репозиторий, используя текущий процесс. Пример: `git tag -a v1.2.3 -m 'Release v1.2.3'` && `git push --tags`  

#### Преимущества  

- 🔌 Абсолютная власть. Вы можете изменить историю.  
  <!-- - 🎭 Попрактикуйтесь в навыках Engineering Theater. -->  

#### Недостатки  

- 😰 Абсолютная власть. Вы можете изменить историю. (Хорошо, и преимущество, и недостаток...)  
- 🔂 Вы можете попасть в бесконечный цикл конфликтов и `--continue`. (Иногда даже с `git rerere`)  
- 🙀 Разрушает ключевые элементы коллаборации: теряются/остаются висеть комментарии к PR. Неприлично.  
- 🖇️ Постоянные ссылки могут перестать быть постоянными.  

### Способ (Squash) Merge

Ментальная модель: «Хочу создать кастомный релиз, начиная с определённой точки, включающий нужные ветки».  

1. Получить последние изменения: `git checkout main` && `git pull`  
2. Создать новую ветку: `git checkout -b release/hot-newness-and-stuff`  
3. Слить нужные ветки/коммиты: `git merge --no-ff feature/hot-newness bug/fix-123` (используйте флаг `--no-ff` там, где это возможно)  
4. Решить конфликты слияния (если они возникнут).  
5. Протагировать/запушить по текущему процессу. Пример: `git tag -a v1.2.3 -m 'Release v1.2.3'` && `git push --tags`  

#### Преимущества  

- 💪 Меньше процесса, меньше конфликтов в целом, использует существующие знания команды по `git`.  
- 🚀 Позволяет думать на уровне PR/веток, игнорируя детали коммитов (если не нужно).  
- 🦺 Не разрушительный. Можно возвращаться и создавать новые ветки в любое время.  
- 🎥 Сохраняет исходные коммиты и сообщения целиком, что уменьшает «шум» в `blame`.  

#### Недостатки  

- 🔏 Сложнее изменять сообщения коммитов.  
- 🤐 Сложнее скрывать свою работу.

### Заключение

В конце концов, **более простой и менее рискованный процесс должен победить.**

Хотя _Ребейсеры_ действительно имеют способы решить (или избежать) свои проблемы, **факт остается фактом: в конечном итоге вам понадобится черный пояс в git fu.** (Например, даже скромный `git push` может столкнуться с дополнительной сложностью: был ли это `git push --force` или `git push --force-with-lease`? Зачем вообще иметь дело с этим?)

Еще одна причина, по которой **ребейз** для создания пересмотренной истории **всегда будет иметь недостаток** по сравнению с **`git merge ...`.** `git merge` позволяет CLI `git` применять продвинутые алгоритмы для избежания конфликтов, анализируя HEAD каждой ветки.

Это может быть умнее, потому что каждый мердж учитывает только последнее состояние каждой нужной ветки, в то время как **ребейз должен воспроизводить (или отбрасывать) историю коммитов в указанной последовательности**. Это **ограничивает способность `git` оптимизировать** объединение, поскольку он **сравнивает только 2 коммита** за раз.

В конечном итоге **ребейз означает, что вы время от времени будете сталкиваться с ненужными старыми коммитами и конфликтами** — даже если вы знаете, что они уже были удалены или разрешены.

### Вывод

- 💃 Ответ: **СХЛОПЫВАЮЩИЙ ОБЪЕДИНЕНИЕ** ваших PR в ветку `main`.
  - История вашей ветки останется доступной, если понадобится, а `main` останется относительно "чистой".
- _🔤 Всегда коммитируйте!_
  - В более чем 95% корпоративных проектов "резервный подход" предпочтителен по сравнению с "скульптурным искусством". Со временем смысл ваших сообщений о коммитах будет затухать гораздо быстрее, чем логика и тесты кода, которые сохранят свою значимость.
