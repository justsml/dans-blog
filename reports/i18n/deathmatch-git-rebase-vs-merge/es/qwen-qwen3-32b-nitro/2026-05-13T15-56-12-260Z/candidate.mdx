## Batalla mortal: Git Rebase vs. (Squash) Merge!

¿Debo usar Rebase? ¿O Squash Merge?

- ¿Es una preferencia personal?
  - _Respuesta: ¡No cuando están involucradas una o más equipos! **Cualquiera de las dos opciones afectará la usabilidad del otro**!_

### ¿Por qué este tema genera un fervor religioso?

Algunos ingenieros usan su conocimiento de git (& la terminal) como señal de su nivel de habilidad relativo. Y cualquier práctica ligada a nuestra identidad/ego puede ser imposible de analizar imparcialmente, ni siquiera cambiar.

Otros factores probablemente incluyan la Familiaridad y el Sesgo de Supervivencia, que pueden enturbiar aún más nuestro propio análisis y suposiciones.

### Pregunta clave: ¿Cuál es el propósito de un commit en git?

1. ¿Haces commits temprano y con frecuencia? Usando una mentalidad de "punto de control" o respaldo?  
   - ¿Dónde se registra todo, incluso intentos fallidos y experimentos? (ej. `git commit -am "Updated deps" && git push`, repetir regularmente)  
   - ¿Quizás los mensajes de commit son menos importantes para ti que el código?  
1. ¿O son tus commits una obra cuidadosamente seleccionada y tallada?  
   - ¿Tal vez cada commit es una unidad atómica y autocontenida de trabajo? (ej. `git add package.json && git commit -m "Updated deps"`)  
   - ¿O simplemente no soportas registros de commits "desordenados"?  
   - ¿Tus revisiones de PR suelen incluir revisar commit por commit?

Dado que existen mentalidades muy diferentes en torno a la estrategia de commits, no es sorprendente que haya tanta confusión sobre la "forma correcta" de usar git.

### Escenario: Crear una etiqueta de lanzamiento revisada

Comparemos el proceso para crear una etiqueta de lanzamiento excluyendo algunos commits recientes en `main`.

![Git Tag Releasing from main with 2 feature branches](../git-branching-with-main-simplified.svg)

### La forma con Rebase

Modelo mental: "Quiero crear una versión alternativa de un historial existente. (e.g. Me equivoqué hace 16 fusiones, y podría necesitar control fino para corregirlo. Además, podría quedar atrapado en un ciclo aparentemente interminable de conflictos y `--continue`.)"

1. Obtén la versión más reciente: `git checkout main` && `git pull`  
2. Crea una nueva rama: `git checkout -b release/hot-newness-and-stuff`  
3. Inicia un rebase interactivo e incluye la referencia git del punto al que deseas retroceder en el historial. `git rebase -i HEAD~6` (Nota: `HEAD~6` es una abreviatura de git para "hace 6 commits").  
4. Elimina los commits deseados cambiando su prefijo a `drop`. Guarda y cierra el editor.  
5. Resuelve los conflictos de fusión, `git add .` && `git rebase --continue` (¡NO uses `git commit`!).  
6. Repite el paso anterior hasta completar.  
7. Etiqueta y publica usando el flujo actual. Ejemplo: `git tag -a v1.2.3 -m 'Release v1.2.3'` && `git push --tags`  

#### Ventajas  

- 🔌 Poder absoluto. Puedes modificar el historial.  
  <!-- - 🎭 Practica tus habilidades de Teatro de Ingeniería. -->  

#### Desventajas  

- 😰 Poder absoluto. Puedes modificar el historial. (Bueno, una ventaja y desventaja...)  
- 🔂 Puedes quedar atrapado en un ciclo aparentemente interminable de conflictos y `--continue`. (A veces incluso con `git rerere`)  
- 🙀 Rompe características clave de colaboración: comentarios de PR perdidos/u orfandos. Muy impolite.  
- 🖇️ Los enlaces permanentes (permalinks) dejan de serlo.  

### La forma (Squash) Merge

Modelo mental: "Quiero una versión personalizada, comenzando desde un punto dado, e incluyendo cualquier rama deseada".

1.  Obtén la versión más reciente: `git checkout main` && `git pull`  
2.  Crea una nueva rama: `git checkout -b release/hot-newness-and-stuff`  
3.  Fusiona las ramas y/o confirmaciones deseadas: `git merge --no-ff feature/hot-newness bug/fix-123` (usa la bandera `--no-ff` siempre que sea posible.)  
4.  Resuelve cualquier conflicto de fusión (si surge.)  
5.  Etiqueta y publica usando el proceso actual. Ejemplo: `git tag -a v1.2.3 -m 'Release v1.2.3'` && `git push --tags`  

#### Ventajas  

- 💪 Menos proceso, menos conflictos en general y utiliza conocimientos existentes de comandos de Git.  
- 🚀 Te permite pensar a un nivel superior de PR/rama, ignorando la granularidad a nivel de confirmación (a menos que sea necesario.)  
- 🦺 No destructivo. Puedes retroceder y/o crear nuevas ramas en cualquier momento.  
- 🎥 Mantiene las confirmaciones existentes y sus mensajes como un todo, lo que reduce el ruido de 'culpabilidad'.  

#### Desventajas  

- 🔏 Más difícil cambiar los mensajes de confirmación.  
- 🤐 Más difícil ocultar tu trabajo.

### Conclusión

Al final del día, **un proceso más simple con menos riesgos debe ganar.**

Aunque los _Rebasers_ sí tienen formas de resolver (o evitar) sus problemas, **el hecho permanece: eventualmente necesitarás un cinturón negro en git fu**. (Por ejemplo, incluso un humilde `git push` puede enfrentar complejidad adicional: ¿fue `git push --force` o `git push --force-with-lease`? ¿Para qué lidiar con eso en absoluto?)

Hay otra razón por la cual **el rebase para crear una historia revisada siempre estará en desventaja** frente a **`git merge ...`**. Un `git merge` permite que la CLI de `git` aplique algoritmos avanzados para evitar conflictos al analizar el HEAD de cada rama.

Esto puede ser más inteligente porque cada fusión solo se preocupa por el estado más reciente de cada rama deseada. En cambio, **el rebase debe reproducir (o eliminar) la historia de commits en el orden** especificado. Esto **limita la capacidad de `git` para optimizar** la fusión, ya que **solo compara 2 commits a la vez**.

Al final, **el rebase significa que ocasionalmente te encontrarás reexperimentando commits y conflictos antiguos e irrelevantes** — incluso si ya sabes que han sido eliminados o resueltos.

### Resumen

- 💃 Respuesta: **HACER UN SQUASH MERGE** de tus PRs en `main`.
  - Tu historia de ramas estará allí si es necesaria, y `main` permanecerá relativamente "limpio".
- _🔤 ¡Siempre Estás Haciendo Commits!_
  - En más del 95% de los proyectos corporativos, la "mentalidad de respaldo" es preferible a la "mentalidad de arte tallado". Con el tiempo, el significado de tus mensajes de commit se desvanecerá más rápido que el código cuya lógica y pruebas mantendrán su relevancia.
