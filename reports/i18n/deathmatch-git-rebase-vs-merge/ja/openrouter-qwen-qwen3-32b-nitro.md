コミット戦略に関する考え方には非常に異なるものがあり、gitを「正しい」方法で使うことに対する混乱が生じるのは当然です。

### Rebase方式

メンタルモデル:「既存の履歴の代替バージョンを作成したい。（例: 16回のマージ前におかしなミスをしてしまい、修正するために細かい制御が必要かもしれない。また、無限に続くようなコンフリクトと`--continue`のサイクルに陥ることもある）」

1. 最新の状態を取得: `git checkout main` && `git pull`  
2. 新しいブランチを作成: `git checkout -b release/hot-newness-and-stuff`  
3. インタラクティブリベースを開始し、過去に戻りたい位置のgit refを含める。`git rebase -i HEAD~6`（注: `HEAD~6`は`6 commits ago`を意味するgit refの略記です）  
4. 除外したいコミットの接頭辞を`drop`に変更して保存し、エディタを閉じます。  
5. マージコンフリクトを解決し、`git add .` && `git rebase --continue`（`git commit`は実行しない）  
6. 前のステップを完了するまで繰り返します。  
7. 現在のプロセスでタグ/プッシュ。例: `git tag -a v1.2.3 -m 'Release v1.2.3'` && `git push --tags`  

#### 良点  
- 🔌 絶対的な力。履歴を変更できます。  
  <!-- - 🎭 エンジニアリング・シアターのスキルを練習できます。 -->  

#### 悪点  
- 😰 絶対的な力。履歴を変更できます。（これは良点でも悪点でもありますが…）  
- 🔂 無限に続くようなコンフリクトと`--continue`のサイクルに陥ることもあります。（場合によっては`git rerere`でも）  
- 🙀 キーとなるコラボレーション機能が壊れる: 失われた/孤立したPRコメント。失礼です。  
- 🖇️ パーマリンクが永久的でなくなる可能性があります。  

### (Squash)マージ方式

Mental model: "特定のポイントから始めて、任意のブランチを含むカスタムリリースを作りたい。"

1. 最新の状態にする: `git checkout main` && `git pull`
2. 新しいブランチを作成: `git checkout -b release/hot-newness-and-stuff`
3. 望むブランチやコミットをマージ: `git merge --no-ff feature/hot-newness bug/fix-123` (可能な限り`--no-ff`フラグを使用。)
4. マージコンフリクトを解決（必要に応じて。）
5. 現在のプロセスでタグ/プッシュ。例: `git tag -a v1.2.3 -m 'Release v1.2.3'` && `git push --tags`

#### 良点  
- 💪 プロセスが少なく、全体的なコンフリクトも減る。既存のGitコマンド知識を活用できる。  
- 🚀 PR/ブランチレベルで思考できる。コミットレベルの細かい粒度を気にしない（必要でない限り）。  
- 🦺 非破壊的。いつでも戻ったり、新しいブランチを作成したりできる。  
- 🎥 既存のコミットとメッセージ全体を保持するため、'blame'のノイズが減る。  

#### 悪点  
- 🔏 コミットメッセージを変更するのが難しい。  
- 🤐 作業を隠すのが難しい。

### 結論

最終的に、**シンプルでリスクの少ないプロセスが勝つべきです。**

たとえ _Rebasers_ が問題を解決（または回避）する方法を持っているとしても、**事実として、最終的にGitの達人（black belt in git fu）になる必要があります。**（たとえば、単純な `git push` でさえ、`git push --force` か `git push --force-with-lease` かの選択が必要になるなど、追加の複雑さに直面します。そもそもなぜそんなことを扱う必要があるのか？）

**リベース** で履歴を再編成する作業が、**`git merge ...`** と比較して常に不利になるもう一つの理由があります。`git merge` は、`git` CLI が各ブランチの HEAD を分析してコンフリクトを回避する高度なアルゴリズムを適用できるためです。

これはスマートである可能性があります。なぜなら、各マージは目的のブランチの最新状態だけを気にするからです。一方で、**リベースでは指定された順序でコミット履歴を再現（または削除）する必要があります**。これは、**一度に2つのコミットしか比較できない**ため、**`git`の最適化能力に制限を生じさせます**。

最終的に、**リベースは無関係な古いコミットやコンフリクトを再経験する必要があるケースが生じる**ことを意味します。それらがすでに削除または解決済みであることを知っている場合でもです。

### 要点

- 💃 答え: **スクワッシュマージ**でPRを`main`にマージする
  - ブランチの履歴は必要に応じて残るため、`main`は比較的「綺麗」に保たれます。
- _🔤 コミットを常に実行する！_
  - 95%以上の企業プロジェクトでは、「バックアップマインドセット」が「彫刻芸術マインドセット」よりも優れています。時間とともにコミットメッセージの意味は急速に薄れますが、コードのロジックやテストの重要性は維持されます。
