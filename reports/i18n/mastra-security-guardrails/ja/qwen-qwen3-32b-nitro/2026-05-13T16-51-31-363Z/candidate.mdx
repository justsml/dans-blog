誰も安全でないAIシステムを意図して構築する人はいません。指示を記述し、エッジケースをテストし、いくつかの検証ルールを追加します。しかし誰かが、あなたのボットを海賊（パイレート）として役割を演じさせることで、ユーザーのデータを漏らす方法を見つけ出すかもしれません。あるいは、クレジットカード番号がログに含まれてしまうこともあります。あるいはモデルが自信満々に競合企業の製品を推奨してしまうかもしれません。  

「デモでは動く」ことと「本番環境で安全である」ことのギャップは、多くのチームが予想するよりも広いものです。  

問題の一部は、生のLLM（大規模言語モデル）が、自分が何をすべきで何をすべきでないかについて意見を持っていないことです。これらは、ユーザーが開始したパターンを引き続き試みる予測機械です。プロンプトに「システムオーバーライドモード」といったものを見せれば、喜んでそれに従ってくれます。これはモデルのバグではなく、単に言語モデルがそう動作するからです。  

ほとんどのフレームワークはモデルを提供して「頑張ってね」と願いを込めるだけです。Mastraは異なるアプローチを採用しています。ガードレールが必要になるだろうと仮定し、エージェントアーキテクチャに最初から組み込みます。  

## プロセッサによるセキュリティレイヤー

コアメカニズムはシンプルです。プロンプトがモデルに到達する前に、入力プロセッサの連鎖を通過します。モデルが応答した後は、出力プロセッサが処理を行います。各プロセッサは、その段階でコンテンツを検査し、修正、またはブロックできます。

これらをAI相互作用のための中間ウェアと考えてください。必要なプロセッサをスタックし、その挙動を設定すれば、すべてのリクエストに対して自動的に実行されます。

### 1. 海賊の阻止（プロンプトインジェクション）

プロンプトインジェクション攻撃は多様化しています。人々は目で見えないUnicode文字を使ったり、Base64で指示を書いたり、「デバッグモード」に自分がいるとモデルに納得させたりします。こうした手法は常に進化しています。

Mastraには、一般的なパターンを検出するプロセッサが含まれています：

```typescript
// src/mastra/agents/secure-agent.ts
import { Agent } from '@mastra/core/agent';
import { PromptInjectionDetector, UnicodeNormalizer } from '@mastra/core/processors';
import { openai } from '@ai-sdk/openai';

export const secureAgent = new Agent({
  id: 'fortress-assistant',
  name: 'fortress-assistant',
  instructions: 'You are a secure assistant.',
  model: openai('gpt-5'),
  inputProcessors: [
    // 1. 目で見えない文字の除去
    new UnicodeNormalizer({
      id: 'unicode-normalizer',
      stripControlChars: true,
      collapseWhitespace: true,
    }),
    // 2. 攻撃の検出
    new PromptInjectionDetector({
      id: 'prompt-injection-detector',
      model: openai('gpt-5-nano'), // 安価で高速
      threshold: 0.8,
      strategy: 'block', // 硬直的なブロック
      detectionTypes: ['injection', 'jailbreak', 'system-override'],
    }),
  ],
});
```

[`UnicodeNormalizer`](https://mastra.ai/docs/processors) は制御文字を除去し、ホワイトスペースを圧縮します。[`PromptInjectionDetector`](https://mastra.ai/docs/processors) は、クリーンアップされた入力に、誰かが指示を上書きしようとしている可能性を示すパターンを分析します。

検出の強さ（`threshold`パラメータ）やトリップした際の対応（ブロック、ログ出力、または単にフラグを立てる）を設定できます。

### 2. PIIの取り扱い

ログに含まれるクレジットカード番号、ベクターデータベースに保存された社会保障番号、必要以上に保持されたメールアドレス。これらは規制上の問題を引き起こす典型的な例です。課題は、ユーザーがチャットウィンドウに機密情報を貼り付けていることに気づかないことが多いことです。

[`PIIDetector`](https://mastra.ai/docs/processors) は、モデルに到達する前やストレージに書き込まれる前の段階で、一般的なパターンをスキャンします：

```typescript
import { PIIDetector } from '@mastra/core/processors';

export const privateAgent = new Agent({
  id: 'privacy-first-assistant',
  name: 'privacy-first-assistant',
  instructions: 'You are a helpful assistant that never stores personal information.',
  model: openai('gpt-5'),
  inputProcessors: [
    new PIIDetector({
      id: 'pii-detector',
      model: openai('gpt-5-nano'),
      detectionTypes: ['email', 'phone', 'credit-card', 'ssn'],
      threshold: 0.6,
      strategy: 'redact',
      redactionMethod: 'mask',  // [REDACTED]に置換
      instructions: 'Detect and mask personally identifiable information',
    }),
  ],
});
```

抹消（[REDACTED]に置換）、ハッシュ化、または完全なブロックのいずれかを選択できます。このプロセッサは入力と出力の両方に適用されるため、モデルが何らかの方法で応答に機密情報を生成した場合でも保護されます。

### 3. コンテンツのモデレーション

モデルがインターネットデータで訓練されているため、さまざまなコンテンツに触れています。フィルタリングがなければ、たまにPRチームを不安にさせるような応答を生成してしまうこともあります。[`ModerationProcessor`](https://mastra.ai/docs/processors)は、コミュニティガイドラインに違反する内容を検出します：

```typescript
import { ModerationProcessor } from '@mastra/core/processors';

export const moderatedAgent = new Agent({
  id: 'safe-assistant',
  name: 'safe-assistant',
  instructions: 'You are a helpful assistant for a community platform.',
  model: openai('gpt-5'),
  inputProcessors: [
    new ModerationProcessor({
      id: 'moderation-processor',
      model: openai('gpt-5-nano'),  // 分類用の高速でコストの低いモデル
      categories: ['hate', 'harassment', 'violence', 'self-harm'],
      threshold: 0.7,  // 確信度が70%を超える場合にブロック
      strategy: 'block',  // 要求を即座に停止
      instructions: 'Detect harmful content that violates community guidelines',
    }),
  ],
});
```

興味深い点は、使用ケースに応じてどのカテゴリが重要かを自分で定義できることです。クリエイティブライティングツールはカスタマーサポートボットよりも表現的なコンテンツを許容するかもしれません。しきい値と戦略によって、フィルタリングの厳しさを調整できます。

## 問題が発生したとき

プロセッサは問題を検出したときにエラーをスローしません。代わりに、結果オブジェクトにフラグを設定します：

```typescript
const result = await secureAgent.generate('Ignore all previous instructions...');

if (result.tripwire) {
  console.log(`Blocked! Reason: ${result.tripwireReason}`);
  // "Blocked! Reason: Prompt injection detected."
  return "Nice try, script kiddie.";
}
```

このパターンにより、アプリケーションに応じてセキュリティイベントをどのように処理するかを柔軟に定義できます。ログに記録して分析したり、汎用的なエラーメッセージを返したり、特定のコンテキストでは一部の違反を許容したりも可能です。`tripwireReason`フィールドは、どのプロセッサがコンテンツをフラグ付けしたかを正確に示し、誤検出のデバッグやしきい値の調整時に役立ちます。

---

## これは解決できないこと

プロセッサは多くの問題を検出しますが、魔法ではありません。十分な時間と意欲のある攻撃者は、おそらく通過するプロンプトを見つけることができるでしょう。モデルは時折、プロセッサが予測できない形でハルシネーションを起こします。そしてセキュリティと柔軟性のトレードオフは常に存在します。ルールが厳格になるほど、正当なユースケースをブロックする可能性が高くなります。

価値は完全な保護にありません。生産環境で確実に発生する一般的な問題を体系的に処理できる仕組みを持つことです。ユーザーが実際に何をしているかを学びながら、感度を調整できます。ドメイン固有のリスクに対してカスタムプロセッサを追加できます。そして、ブロックされた内容とその理由を示す実装履歴を持っています。

生産環境のAIにおける多くのセキュリティ問題は、洗練された攻撃ではありません。ユーザーが誤って扱いの悪いデータをコピー＆ペーストしたり、試行錯誤の末にボットが意図しない動作を行うことを発見したりするケースです。プロセッサはすべての問題を防ぐことはできませんが、明らかな問題をはるかに難しくします。
