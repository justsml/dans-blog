Niemand baut absichtlich ein unsicheres KI-System. Sie schreiben Anweisungen, testen Randfälle und fügen ein paar Validierungsregeln hinzu. Dann stellt jemand fest, dass er Ihren Bot dazu bringen kann, die Rolle eines Piraten zu spielen und Benutzerdaten preiszugeben. Oder eine Kreditkartennummer taucht in Ihren Protokollen auf. Oder das Modell empfiehlt mit Überzeugung das Produkt eines Wettbewerbers.

Der Abstand zwischen „funktioniert in der Demo“ und „sicher in der Produktion“ ist größer als die meisten Teams erwarten.

Ein Teil des Problems ist, dass Roh-LLMs keine Meinung darüber haben, was sie tun sollten oder nicht. Sie sind Vorhersagemaschinen, die versuchen, jedes Muster fortzusetzen, das du begonnen hast. Gib ihnen einen Prompt, der so aussieht wie „system override mode“, und sie spielen gerne mit. Das ist kein Fehler im Modell; es ist einfach so, wie Sprachmodelle funktionieren.

Die meisten Frameworks übergeben dir das Modell und wünschen dir Glück. Mastra verfolgt einen anderen Ansatz: Es geht davon aus, dass du irgendwann Schutzmaßnahmen benötigst, und baut sie daher von Anfang an in die Agentenarchitektur ein.

Das zentrale Mechanismus ist geradlinig. Bevor dein Prompt das Modell erreicht, läuft er durch eine Kette von Eingabe-Processoren. Nachdem das Modell geantwortet hat, ist es an den Ausgabe-Processoren. Jeder Prozessor kann den Inhalt an dieser Stufe untersuchen, modifizieren oder blockieren.

Denke daran, dass sie Middleware für AI-Interaktionen sind. Du stapelst die benötigten Prozessoren, konfigurierst ihr Verhalten, und sie laufen automatisch bei jedem Anforderung.

### 1. Die Piraten stoppen (Prompt-Injektion)

Prompt-Injektionsangriffe haben kreative Wege gefunden. Menschen nutzen unsichtbare Unicode-Zeichen, schreiben Anweisungen in Base64 oder überzeugen das Modell, dass sie sich im „Debug-Modus“ befinden, in dem normale Regeln nicht gelten. Die Techniken entwickeln sich ständig weiter.

Mastra enthält Prozessoren, die gängige Muster erkennen:

```typescript
// src/mastra/agents/secure-agent.ts
import { Agent } from '@mastra/core/agent';
import { PromptInjectionDetector, UnicodeNormalizer } from '@mastra/core/processors';
import { openai } from '@ai-sdk/openai';

export const secureAgent = new Agent({
  id: 'fortress-assistant',
  name: 'fortress-assistant',
  instructions: 'You are a secure assistant.',
  model: openai('gpt-5'),
  inputProcessors: [
    // 1. Entferne unsichtbare Zeichen
    new UnicodeNormalizer({
      id: 'unicode-normalizer',
      stripControlChars: true,
      collapseWhitespace: true,
    }),
    // 2. Erkenne den Versuch
    new PromptInjectionDetector({
      id: 'prompt-injection-detector',
      model: openai('gpt-5-nano'), // Günstig, schnell
      threshold: 0.8,
      strategy: 'block', // Hartes Stoppen
      detectionTypes: ['injection', 'jailbreak', 'system-override'],
    }),
  ],
});
```

Der [`UnicodeNormalizer`](../docs/processors) entfernt Steuerzeichen und zusammengefasste Leerzeichen. Der [`PromptInjectionDetector`](../docs/processors) analysiert die bereinigten Eingaben nach Mustern, die darauf hindeuten, dass jemand versucht, deine Anweisungen zu überschreiben.

Sie konfigurieren, wie aggressiv die Erkennung sein soll (der `threshold`-Parameter) und was geschieht, wenn eine Erkennung eintritt (blockieren, protokollieren oder einfach markieren).

### 2. Umgang mit personenbezogenen Informationen (PII)

Kreditkartennummern in Logs, Sozialversicherungsnummern in Vektor-Datenbanken, E-Mail-Adressen, die länger als notwendig gespeichert werden. Diese sind die Arten von Problemen, die zu regulatorischen Konsequenzen führen. Das Problem ist, dass Nutzer nicht immer erkennen, dass sie sensible Daten in ein Chatfenster einfügen.

Der [`PIIDetector`](../docs/processors) scannt nach gängigen Mustern, bevor sie Ihr Modell erreichen oder in die Speicherung geschrieben werden:

```typescript
import { PIIDetector } from '@mastra/core/processors';

export const privateAgent = new Agent({
  id: 'privacy-first-assistant',
  name: 'privacy-first-assistant',
  instructions: 'You are a helpful assistant that never stores personal information.',
  model: openai('gpt-5'),
  inputProcessors: [
    new PIIDetector({
      id: 'pii-detector',
      model: openai('gpt-5-nano'),
      detectionTypes: ['email', 'phone', 'credit-card', 'ssn'],
      threshold: 0.6,
      strategy: 'redact',
      redactionMethod: 'mask',  // Replace with [REDACTED]
      instructions: 'Detect and mask personally identifiable information',
    }),
  ],
});
```

Sie können wählen, ob Sie zusammenfassen (mit `[REDACTED]` ersetzen), hashen oder vollständig blockieren. Der Prozessor läuft sowohl bei Eingabe als auch bei Ausgabe, sodass Sie geschützt sind, selbst wenn das Modell irgendeine Weise sensible Daten in der Antwort generiert.

### 3. Inhaltsschutz

Modelle, die anhand von Internetdaten trainiert wurden, haben vieles gesehen. Ohne Filter können sie gelegentlich Antworten generieren, die Ihr PR-Team beunruhigen würden. Der [`ModerationProcessor`](https://mastra.ai/docs/processors) erkennt Inhalte, die gegen Ihre Richtlinien verstoßen:

```typescript
import { ModerationProcessor } from '@mastra/core/processors';

export const moderatedAgent = new Agent({
  id: 'safe-assistant',
  name: 'safe-assistant',
  instructions: 'You are a helpful assistant for a community platform.',
  model: openai('gpt-5'),
  inputProcessors: [
    new ModerationProcessor({
      id: 'moderation-processor',
      model: openai('gpt-5-nano'),  // Fast, cheap model for classification
      categories: ['hate', 'harassment', 'violence', 'self-harm'],
      threshold: 0.7,  // Block if confidence > 70%
      strategy: 'block',  // Stop the request immediately
      instructions: 'Detect harmful content that violates community guidelines',
    }),
  ],
});
```

Interessant ist, dass Sie definieren können, welche Kategorien für Ihren Use Case relevant sind. Ein Kreativschreibwerkzeug könnte ausdrucksstärkere Inhalte zulassen als ein Kundendienst-Bot. Der Schwellenwert und die Strategie geben Ihnen die Kontrolle darüber, wie streng die Filterung sein soll.

## Wenn Sicherungsfäden auslösen

Processor werfen keine Fehler aus, wenn sie ein Problem erkennen. Stattdessen setzen sie ein Flag auf das Ergebnisobjekt:

```typescript
const result = await secureAgent.generate('Ignore all previous instructions...');

if (result.tripwire) {
  console.log(`Blocked! Reason: ${result.tripwireReason}`);
  // "Blocked! Reason: Prompt injection detected."
  return "Nice try, script kiddie.";
}
```

Dieses Muster erlaubt es Ihnen, Sicherheitsereignisse nach Bedarf für Ihre Anwendung zu behandeln. Sie könnten sie für die Analyse protokollieren, eine generische Fehlermeldung zurückgeben oder sogar bestimmte Verstöße in spezifischen Kontexten zulassen. Das Feld `tripwireReason` teilt Ihnen mit, welcher Processor den Inhalt gemeldet hat, was hilfreich ist, wenn Sie Fehlalarme debuggen oder Ihre Schwellenwerte anpassen.

## Was dies nicht löst  

Processoren erkennen vieles, aber sie sind nicht magisch. Ein entschlossener Angreifer mit genügend Zeit kann wahrscheinlich eine Prompt finden, die durchschlüpft. Modelle halluzinieren gelegentlich auf Weise, die Prozessoren nicht vorhersagen können. Und es gibt immer einen Kompromiss zwischen Sicherheit und Flexibilität: Je strenger Ihre Regeln, desto wahrscheinlicher blockieren Sie legitime Use Cases.  

Der Wert liegt nicht in perfekter Schutzmaßnahme. Es geht darum, eine systematische Methode zu haben, um mit den Problemen umzugehen, die in der Produktion definitiv auftreten. Sie können die Empfindlichkeit anpassen, während Sie erfahren, was Ihre Benutzer tatsächlich tun. Sie können benutzerdefinierte Prozessoren hinzufügen, um domänen-spezifische Risiken zu behandeln. Und Sie haben Audit-Trail, die zeigen, was blockiert wurde und warum.  

Die meisten Sicherheitsprobleme in Produktions-AI sind keine sophistizierten Angriffe. Es ist, wenn Menschen Daten kopieren und einfügen, die sie nicht sollten, oder herausfinden, durch Probieren, dass der Bot Dinge tut, die Sie nicht beabsichtigt haben. Prozessoren werden nicht jedes mögliche Problem stoppen, aber sie machen die offensichtlichen deutlich schwerer.
