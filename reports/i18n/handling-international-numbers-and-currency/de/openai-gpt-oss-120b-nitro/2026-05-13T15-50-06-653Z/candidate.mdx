## Money: Localization (L10n) and Internationalization (i18n)

Sie sind nicht nur für ein Scrabble‑Spiel gedacht – *Lokalisierung* und *Internationalisierung* bezeichnen den Prozess, ein Produkt **in einem anderen Land heimisch wirken zu lassen**.

<p class="breakout quote">Eine Währung im falschen lokalen Format anzuzeigen, ist ein todsicheres Indiz dafür, dass kein Aufwand betrieben wurde.<br/>Wenn Sie keinen Preis formatieren können, wie sollen Sie dann den Versand handhaben?</p>

Internationalisierung ist ein umfangreiches Thema, das von Textübersetzung bis zur Datumsformatierung reicht. In diesem Beitrag konzentrieren wir uns auf einen speziellen Teilbereich: **die Formatierung von Zahlen und Währungen**.

Betrachten wir die Formatierung in drei Eurozonen‑Ländern, den USA und Indien:

- `€1,234,567.89` Irland 🇮🇪
- `1.234.567,89 €` Deutschland 🇩🇪
- `1 234 567,89 €` Frankreich 🇫🇷
- `$1,234,567.89` USA 🇺🇸
- `₹12,34,567.89` Indien 🇮🇳

Chaos! Oder? Symbole, Leerzeichen und Interpunktion fliegen überall hin! Erstaunlich, wie wenig die EU sich einig werden kann! 😅

## Kritische Konzepte

Bevor wir zu Lösungen kommen, was meinen wir mit „Zahlen sind lokal“?

### Zahlen sind lokal 🏘️

Jede Locale ([Country per ISO 3166](https://en.wikipedia.org/wiki/List_of_ISO_3166_country_codes)) definiert Regeln zur Formatierung von Zahlen.

Zahlformatierungsregeln umfassen:

- Dezimaltrennzeichen: Komma, Punkt.
- Tausendertrennzeichen: Komma, Punkt, Leerzeichen.
- Position und Abstand des Währungssymbols.

### Currency is Global 🌎

Eine `currency` bezeichnet eine bestimmte Geldeinheit. (Siehe [ISO 4217](https://en.wikipedia.org/wiki/ISO_4217#Active_codes_(list_one)) für die Liste.)

- Definiert ein Symbol: `$`, `€`, `£`, `¥`. (Wird häufig wiederverwendet.)
- Hat immer einen 3‑Buchstaben‑Code: `USD`, `EUR`, `GBP`, `JPY`.
- Kann theoretisch in „jedem“ Land verwendet/umgetauscht werden.
- Der Umtausch zwischen Währungen erfordert Wechselkursdaten.
- Der Wert ändert sich nicht je nach Locale.

### When Locale Matters

Die meisten E‑Commerce‑/Payment‑REST‑APIs arbeiten mit `price` + `currencyCode`. Warum keine Locales?

Locales werden (typischerweise) auf Betriebssystem‑/Geräteebene festgelegt und Browser stellen sie über `navigator.language` bereit. Da jeder Ihrer Nutzer eine andere Locale haben kann, macht es nur Sinn, Zahlen und Währungen clientseitig zu formatieren.

## Eine Lösung

Gut, es gibt Neuigkeiten! Moderne Programmiersprachen bieten dafür integrierte Unterstützung. In JavaScript steht die Klasse [`Intl`](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Intl) und `Intl.NumberFormat` zur Verfügung!

Schauen wir uns etwas Code an:

```javascript
const number = 1_234_567.89;

/**
 * Formatiert eine Zahl in lokaler Währung.
 * @param {number} amount - Der zu formatierende Betrag.
 * @param {string} currency - Der 3‑Buchstaben‑Währungscode.
 * @param {string} [locale] - Der Locale‑String des Nutzers.
 */
const formatMoney = (amount, currency, locale = navigator.language) =>
  new Intl.NumberFormat(locale, { currency, style: 'currency' })
    .format(amount);

console.log('🇩🇪 ' + formatMoney(number, 'EUR', 'de-DE'));
console.log('🇮🇪 ' + formatMoney(number, 'EUR', 'ga-IE'));
console.log('🇫🇷 ' + formatMoney(number, 'EUR', 'fr-FR'));
```

Falls Sie anspruchsvollere Vorgänge benötigen, etwa Steuerberechnungen, Rabatte anwenden oder zwischen Währungen konvertieren, sollten Sie eine Bibliothek wie [dinero.js](https://v2.dinerojs.com/) einsetzen.

## Nächste Schritte

Je nach Ihren konkreten Anforderungen sollten Sie verwandte Konzepte prüfen:

- Best Practices für die Benutzer‑Locale. Erkennen + Überschreibungen zulassen (z. B. ein Länderauswahl‑Dropdown).
- Ganze Zahlen persistieren (Cents speichern, nicht Dollar).
- Geld‑Mathematik (z. B. einen `20 % Rabatt`‑Gutschein anwenden, `Zwischensumme + Steuern` berechnen usw.).
- Live‑Wechselkurse (für Einzelhandelskauf, Forex‑/Währungsumtausch).
