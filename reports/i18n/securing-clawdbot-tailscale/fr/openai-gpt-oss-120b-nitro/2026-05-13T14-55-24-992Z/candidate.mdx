OpenClaw (anciennement Clawdbot/Moltbot) vous fournit un assistant IA personnel qui fonctionne sur WhatsApp, Slack, Discord, iMessage et d’autres canaux. Mais si vous exposez son passerelle, ses contrôles de nœud ou SSH sur Internet public sans authentification robuste, vous offrez à des inconnus un accès direct au shell de votre machine.

Ce guide présente le paramètre par défaut le plus sûr : garder la passerelle d’OpenClaw liée à l’adresse de boucle locale, ne l’exposer qu’à votre tailnet via Tailscale Serve, verrouiller SSH, et vérifier depuis l’extérieur que la passerelle n’est pas publique.

L’adoption rapide du projet a mis en lumière de véritables problèmes de sécurité : [les analyses Shodan ont trouvé 2 847 instances exposées](https://socradar.io/blog/clawdbot-is-it-safe/) au cours des premières semaines, et un [audit de sécurité GitHub a signalé 512 constats](https://github.com/moltbot/moltbot/issues/1796) dans le code. Une partie provient de scanners automatisés et certains éléments ont évolué depuis le renommage de janvier 2026 en OpenClaw, il faut donc considérer ce chiffre comme un indicateur d’alerte plutôt que comme un décompte précis de vulnérabilités actuelles. Vous n’avez pas besoin d’être expert en sécurité — il suffit d’éviter de publier les surfaces d’opération avant le déploiement.

---

## Ce que vous exposez réellement

Selon la façon dont vous l’avez installé et exposé, trois surfaces méritent d’être vérifiées :

- **Port 22** : accès SSH sur un VPS  
- **Port 18789** : interface de contrôle de la passerelle et API WebSocket  
- **Contrôle navigateur/​node** : exécution distante de nœuds et automatisation du navigateur via le modèle d’appairage passerelle/nœud  

La documentation actuelle de [OpenClaw remote-access](https://docs.molt.bot/gateway/remote) indique que le WebSocket de la passerelle se lie à l’interface de bouclage (`loopback`) par défaut et recommande de le laisser en mode boucle locale sauf si vous choisissez explicitement une liaison LAN, tailnet ou personnalisée. C’est une bonne pratique. Le risque apparaît lorsque vous surchargez ce comportement par défaut, publiez les ports Docker, ajoutez un reverse proxy, activez Funnel, ou laissez le SSH ouvert à tout le monde.  

La passerelle est le point d’entrée principal. C’est la surface d’opération de votre assistant, y compris les chemins d’invocation d’outils. Si elle est accessible depuis Internet et que l’authentification est absente, faible, contournée ou divulguée, un attaquant peut piloter l’agent ou invoquer des outils avec les permissions de votre utilisateur.  

Le contrôle du navigateur est presque aussi sensible. La documentation actuelle d’OpenClaw recommande d’exécuter le contrôle du navigateur via un nœud appairé sur la machine du navigateur et de traiter l’appairage de nœuds comme un accès opérateur. Si une passerelle peut appeler `system.run` sur un nœud appairé, cela équivaut à une exécution de code à distance sur ce nœud, soumise à la politique de nœud de la passerelle et aux approbations d’exécution propres au nœud.  

SSH, c’est SSH. Si vous autorisez l’authentification par mot de passe, les tentatives de force brute sont inévitables sur un VPS public.

## The Tailscale Solution

Pour OpenClaw, Tailscale vous fournit un accès distant sans publier les services opérateur :

1. Votre instance OpenClaw tourne sur un VPS ou une machine locale  
2. La passerelle reste liée à l’interface de boucle locale et est atteinte via Tailscale Serve, ou elle se lie directement à l’adresse IP du tailnet avec une authentification explicite  
3. Vous installez Tailscale à la fois sur le serveur et sur vos appareils personnels  
4. Vous accédez à OpenClaw via son IP Tailscale ou son nom MagicDNS  
5. Tout le reste d’Internet ne voit rien, sauf si vous activez intentionnellement Funnel ou un autre proxy public  

### Should You Let OpenClaw Manage Tailscale?

OpenClaw possède une [intégration Tailscale intégrée](../docs.molt.bot/gateway/tailscale) qui peut configurer `tailscale serve` ou `tailscale funnel` pour la passerelle.

**Serve mode** garde tout au sein de votre tailnet. La passerelle reste liée à `127.0.0.1` tandis que Tailscale gère le routage et le HTTPS. Lorsque `gateway.auth.allowTailscale` est activé, OpenClaw peut authentifier le trafic UI/WebSocket de contrôle en utilisant les en‑têtes d’identité Tailscale et vérifier la source avec `tailscale whois`. C’est le mode recommandé pour la plupart des déploiements personnels.

**Funnel mode** expose la passerelle au public via la fonctionnalité d’endpoint public de Tailscale. La documentation officielle de Tailscale décrit Funnel comme le routage du trafic depuis Internet vers un service local. OpenClaw refuse de lancer Funnel tant que le mode d’authentification de la passerelle n’est pas `password`, mais vous choisissez quand même d’exposer publiquement une surface d’opération.

La [documentation de sécurité d'OpenClaw](https://docs.molt.bot/gateway/security) indique clairement que l’injection d’instructions et l’accès aux outils sont des risques majeurs pour un assistant personnel. Ne donnez pas à l’agent la possibilité de se rendre public discrètement. Utilisez Serve de façon délibérée, évitez Funnel sauf si un accès public est réellement nécessaire, et exigez une approbation d’exécution pour toute commande `tailscale`.

## Mise en place sécurisée d'OpenClaw

### Étape 1 : Installer Tailscale

Sur votre VPS ou serveur local :

```bash
# Install Tailscale
curl -fsSL https://tailscale.com/install.sh | sh

# Authenticate (opens a browser to log in)
sudo tailscale up

# Get your Tailscale IP
tailscale ip -4
# Output: 100.x.x.x
```

Sur votre machine cliente, installez Tailscale depuis la page de téléchargement officielle et connectez‑vous au même tailnet.

Les deux machines se trouvent alors sur le même réseau privé. Vous pouvez pinguer votre VPS en utilisant son adresse IP Tailscale, le trafic sera acheminé via le tunnel chiffré.

### Étape 2 : Configurer OpenClaw pour utiliser Tailscale

Le modèle le plus sûr aujourd’hui consiste à laisser la passerelle liée à l’interface de boucle locale et à l’exposer à votre tailnet avec Tailscale Serve.

Dans la configuration d’OpenClaw :

```js
{
  gateway: {
    bind: "loopback",
    tailscale: { mode: "serve" },
  },
}
```

Puis lancez la passerelle avec Serve :

```bash
openclaw gateway --tailscale serve
```

La documentation d’OpenClaw indique que cela maintient la passerelle sur `127.0.0.1` tandis que Tailscale fournit HTTPS et le routage tailnet. Vous y accédez via `https://<magicdns-name>/`, pas via l’adresse IP publique de votre VPS.

Si vous préférez lier directement le tailnet plutôt que d’utiliser Serve, activez l’authentification explicite de la passerelle :

```js
{
  gateway: {
    bind: "tailnet",
    auth: {
      mode: "token",
      token: "replace-with-a-long-random-token",
    },
  },
}
```

Puis connectez‑vous depuis un autre appareil du tailnet :

```text
http://<tailscale-ip>:18789/
ws://<tailscale-ip>:18789
```

Lorsque vous exécutez OpenClaw dans Docker ou un autre runtime de conteneurs, soyez particulièrement vigilant quant à la publication des ports. Une instruction du type `-p 18789:18789` lie généralement le port sur toutes les interfaces de l’hôte. Privilégiez la boucle locale avec Tailscale Serve, ou liez explicitement le côté hôte à l’adresse IP Tailscale après vous être assuré que le conteneur continue de recevoir du trafic :

```bash
TAILSCALE_IP=$(tailscale ip -4)
docker run ... -p "$TAILSCALE_IP:18789:18789" ...
```

Après toute modification Docker, vérifiez l’exposition depuis l’extérieur avec `nmap` et localement avec `ss`. Docker peut contourner ou réorganiser les hypothèses du pare‑feu de l’hôte si vous ne le prenez pas en compte.

### Étape 3 : Sécuriser SSH

Même avec Tailscale, vous devez sécuriser SSH correctement :

```bash
# Gardez votre session SSH actuelle ouverte pendant cette manipulation.
# D'abord, depuis votre machine cliente, confirmez que vous pouvez vous connecter en SSH via Tailscale :
ssh your-user@SERVER_TAILSCALE_IP

# Placez le durcissement dans un fichier drop‑in au lieu de réécrire sshd_config.
sudo tee /etc/ssh/sshd_config.d/99-openclaw-hardening.conf >/dev/null <<'EOF'
PasswordAuthentication no
PermitRootLogin no
KbdInteractiveAuthentication no
EOF

# Validez avant de recharger. Ne sautez pas cette étape.
sudo sshd -t
sudo systemctl reload ssh || sudo systemctl reload sshd
```

Cela désactive l’authentification par mot de passe et la connexion root. L’étape suivante utilise UFW pour bloquer complètement le SSH public tout en autorisant le SSH sur `tailscale0`.

### Étape 4 : Règles de pare‑feu

Installez un pare‑feu comme seconde couche :

```bash
# Avec UFW (Ubuntu/Debian)
sudo ufw allow in on tailscale0
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw enable
sudo ufw delete allow 22/tcp || true
sudo ufw reload
sudo ufw status verbose
```

Le guide de durcissement officiel de Tailscale pour Ubuntu utilise exactement cette approche : autoriser `tailscale0`, refuser tout le trafic entrant restant, puis vérifier que le SSH public échoue tandis que le SSH vers l’adresse `100.x.y.z` fonctionne toujours. Si vous hébergez un site web public sur le même VPS, ne conservez que les règles publiques réellement nécessaires, comme `80/tcp` et `443/tcp`.

## Vérifier votre exposition

### Vérifier les ports ouverts depuis l’extérieur

Depuis une machine qui **n’est pas** sur votre réseau Tailscale :

```bash
# Vérifier si les ports publics courants sont exposés
nmap -p 22,80,443,18789 YOUR_PUBLIC_IP

# Sortie attendue pour une instance sécurisée :
# 22/tcp   filtered ssh
# 18789/tcp filtered unknown
```

Si le port `22` ou `18789` apparaît comme `open` au lieu de `filtered` ou `closed`, il y a un problème. Si le port `80` ou `443` est ouvert, assurez‑vous qu’il s’agit uniquement de votre site web public intentionnel ou du point de terminaison Tailscale Funnel, et non du gateway OpenClaw par inadvertance.

### Vérifier ce qui écoute localement

Sur votre serveur OpenClaw :

```bash
# Show all listening ports and what they're bound to
sudo ss -tulpn | grep LISTEN

# Look for lines like this (good for Serve):
# tcp   LISTEN 0   128   127.0.0.1:18789   *:*
#
# Or this (acceptable for direct tailnet bind with auth):
# tcp   LISTEN 0   128   100.x.y.z:18789   *:*
#
# NOT like this (bad):
# tcp   LISTEN 0   128   0.0.0.0:18789     *:*
```

Si vous voyez `0.0.0.0` ou `:::` (équivalent IPv6), ce service est exposé au monde.

### Audit de sécurité intégré

OpenClaw inclut une [commande d’audit de sécurité](https://docs.molt.bot/gateway/security) qui vérifie votre configuration par rapport aux meilleures pratiques :

```bash
openclaw security audit --deep
openclaw security audit --deep --fix
```

L’audit examine l’exposition du gateway, le mode Tailscale, les paramètres d’authentification, l’accès aux canaux, la politique d’outils, l’inventaire des plugins et les permissions des fichiers. Considérez `--fix` comme un assistant pratique, pas comme un substitut à la lecture des résultats.

## Ce que cela ne résout pas


Tailscale élimine l’erreur la plus fréquente : l’exposition publique de l’opérateur. Cela ne résout pas tout :

**Stockage des identifiants** : OpenClaw conserve les transcriptions de session, les jetons OAuth et les clés API sur le disque. Veillez à ce qu’ils bénéficient des permissions de fichier appropriées (`chmod 600` pour les fichiers, `chmod 700` pour les répertoires de configuration privés) et qu’ils ne se retrouvent pas dans le contrôle de version. L’audit intégré vérifie ce point.

**Isolation des plugins** : Les plugins s’exécutent avec les permissions complètes de votre utilisateur. N’installez que des plugins provenant de sources fiables et examinez les capacités qu’ils demandent. L’outil d’audit recense les plugins installés.

**Sécurité des appareils** : Si quelqu’un compromet votre compte Tailscale ou vole un appareil sur votre tailnet, il pourra accéder à votre instance OpenClaw. Activez [l’autorisation des appareils Tailscale](https://tailscale.com/kb/1099/device-authorization/) pour exiger une approbation des nouveaux appareils.

## Checklist de déploiement

Avant de considérer votre instance OpenClaw/Moltbot prête pour la production :

- [ ] Tailscale installé et authentifié sur le serveur et le client
- [ ] Passerelle maintenue en boucle locale avec Tailscale Serve, ou liée au `tailnet` avec authentification explicite
- [ ] SSH configuré pour désactiver l’authentification par mot de passe et la connexion root
- [ ] Pare‑feu (UFW ou iptables/nftables) configuré pour autoriser `tailscale0` et refuser tout trafic public non nécessaire
- [ ] Scan nmap externe montrant tous les ports comme `filtered` ou `closed`
- [ ] `ss -tulpn` interne montrant la passerelle liée à `127.0.0.1`, `::1` ou uniquement à l’adresse IP Tailscale
- [ ] Les fichiers d’identifiants ont des permissions 600 et les répertoires de configuration privés ont des permissions 700
- [ ] Exécuter `openclaw security audit --deep` et corriger toutes les anomalies
- [ ] Si vous utilisez la gestion Tailscale d’OpenClaw, les approbations d’exécution sont activées
- [ ] Sauvegardes régulières configurées (données OpenClaw + configurations)
