OpenClaw (ранее Clawdbot/Moltbot) предоставляет вам персонального AI-ассистента, который работает через WhatsApp, Slack, Discord, iMessage и другие каналы. Но если вы выносите его шлюз, узловые контроллеры или SSH в публичный интернет без строгой аутентификации, вы даете посторонним путь к shell-доступу на вашу машину.

Этот гид демонстрирует самый безопасный вариант по умолчанию: держите шлюз OpenClaw в локальной петле, открывайте его только для вашей tailnet через Tailscale Serve, блокируйте SSH и проверяйте извне, не открыт ли шлюз публично.

Быстрое принятие проекта выявило реальные проблемы безопасности: [сканы Shodan обнаружили 2 847 открытых экземпляров](https://socradar.io/blog/clawdbot-is-it-safe/) в первые недели, а [аудит безопасности GitHub сообщил о 512 находках](https://github.com/moltbot/moltbot/issues/1796) в кодовой базе. Некоторые из них были результатом автоматических сканеров, а часть изменилась после переименования в OpenClaw в январе 2026 года, поэтому рассматривайте цифры как предупреждающий знак, а не точный счет уязвимостей. Вам не нужно быть экспертом в безопасности — вам нужно просто избегать публикации управляющих интерфейсов до деплоя.

## Что именно вы открываете

В зависимости от способа установки и публикации, есть три интерфейса, которые стоит проверить:

- **Порт 22**: SSH-доступ на VPS  
- **Порт 18789**: Управляющий интерфейс шлюза и WebSocket API  
- **Управление браузером/узлом**: удалённое выполнение на узлах и автоматизация браузера через модель парных узлов  

Текущие [документы OpenClaw по удалённому доступу](https://docs.molt.bot/gateway/remote) говорят, что WebSocket-интерфейс шлюза по умолчанию привязывается к loopback и рекомендуют сохранять эту настройку, если вы не намеренно выбираете LAN/tailnet/пользовательскую привязку. Это правильно. Риск возникает, когда вы изменяете это поведение по умолчанию, публикуете порты Docker, добавляете обратный прокси, включаете Funnel или оставляете SSH открытым для интернета.

Шлюз — это самый критичный элемент. Это управляющая поверхность для вашего ассистента, включая пути вызова инструментов. Если он достижим из интернета и аутентификация отсутствует, слаба, обходится или утекает, атакующий может управлять агентом или вызывать инструменты с вашими правами пользователя.

Управление браузером почти так же чувствительно. Текущие документы OpenClaw рекомендуют запускать управление браузером через парный узловой хост на машине браузера и рассматривать парную связку узлов как управляющий доступ. Если шлюз может вызывать `system.run` на парном узле, это удалённое выполнение кода на этом узле, ограниченное политикой узла шлюза и собственными одобрениями выполнения на узле.

SSH — это SSH. Если вы используете аутентификацию по паролю, атаки методом перебора на публичном VPS неизбежны.

## Решение Tailscale

Для OpenClaw Tailscale предоставляет удалённый доступ без публикации управляющих сервисов:

1. Ваша инстанция OpenClaw запущена на VPS или локальной машине  
2. Шлюз остаётся привязан к loopback и достигается через Tailscale Serve, или привязывается напрямую к IP tailnet с явной аутентификацией  
3. Вы устанавливаете Tailscale как на сервере, так и на своих личных устройствах  
4. Доступ к OpenClaw осуществляется через его IP Tailscale или имя MagicDNS  
5. Остальной интернет ничего не видит, если вы не намеренно включите Funnel или другой публичный прокси  

### Следует ли позволить OpenClaw управлять Tailscale?

OpenClaw имеет [встроенную интеграцию Tailscale](https://docs.molt.bot/gateway/tailscale), которая может настраивать `tailscale serve` или `tailscale funnel` для шлюза.

**Режим Serve** держит всё в вашем tailnet. Шлюз остаётся привязан к `127.0.0.1`, тогда как Tailscale обрабатывает маршрутизацию и HTTPS. Когда `gateway.auth.allowTailscale` включён, OpenClaw может аутентифицировать трафик Control UI/WebSocket с помощью заголовков идентичности Tailscale и проверять источник через `tailscale whois`. Это правильный режим для большинства личных развертываний.

**Режим Funnel** публично открывает шлюз через публичный эндпоинт Tailscale. Официальная документация Tailscale описывает Funnel как способ маршрутизации трафика из интернета к локальной службе. OpenClaw не запускает Funnel, если режим аутентификации шлюза не `password`, но вы всё равно выбираете публичное открытие управляющего интерфейса.

Документация OpenClaw по [безопасности](https://docs.molt.bot/gateway/security) прямо указывает, что инъекции промптов и доступ к инструментам — это фундаментальные риски для персонального ассистента. Не позволяйте агенту тихо сделать себя публичным. Используйте Serve осознанно, избегайте Funnel, если публичный доступ не нужен, и требуйте подтверждение для любых команд `tailscale`.

## Безопасная настройка OpenClaw

### Шаг 1: Установите Tailscale

На вашем VPS или локальном сервере:

```bash
# Установите Tailscale
curl -fsSL https://tailscale.com/install.sh | sh

# Аутентификация (открывает браузер для входа)
sudo tailscale up

# Получите свой IP Tailscale
tailscale ip -4
# Вывод: 100.x.x.x
```

На клиентской машине установите Tailscale с официальной страницы загрузки и войдите в тот же tailnet.

Теперь обе машины находятся в одной частной сети. Вы можете пинговать свой VPS по его IP Tailscale, и трафик будет маршрутизироваться через зашифрованный тоннель.

### Шаг 2: Настройка OpenClaw для использования Tailscale

Самый безопасный текущий паттерн: оставьте шлюз на loopback и предоставьте его вашей tailnet через Tailscale Serve.

В конфиге OpenClaw:

```js
{
  gateway: {
    bind: "loopback",
    tailscale: { mode: "serve" },
  },
}
```

Затем запустите шлюз с Serve:

```bash
openclaw gateway --tailscale serve
```

Документация OpenClaw утверждает, что это оставляет шлюз на `127.0.0.1`, в то время как Tailscale предоставляет HTTPS и маршрутизацию tailnet. Вы открываете его по адресу `https://<magicdns-name>/`, а не по вашему публичному IP VPS.

Если вы предпочитаете прямой bind tailnet вместо Serve, используйте явную аутентификацию шлюза:

```js
{
  gateway: {
    bind: "tailnet",
    auth: {
      mode: "token",
      token: "replace-with-a-long-random-token",
    },
  },
}
```

Затем подключитесь с другого устройства tailnet:

```text
http://<tailscale-ip>:18789/
ws://<tailscale-ip>:18789
```

Если вы запускаете в Docker или другой среде контейнеров, будьте особенно внимательны при публикации портов. Публикация вроде `-p 18789:18789` обычно связывает все интерфейсы хоста. Предпочтительнее loopback плюс Tailscale Serve, или явно свяжите сторону хоста с IP Tailscale после подтверждения, что контейнер всё ещё получает трафик:

```bash
TAILSCALE_IP=$(tailscale ip -4)
docker run ... -p "$TAILSCALE_IP:18789:18789" ...
```

После любых изменений в Docker проверяйте извне с помощью `nmap` и локально с помощью `ss`. Docker может обойти или переупорядочить предположения о брандмауэре хоста, если вы этого не учтёте.

### Шаг 3: Блокировка SSH

Даже с Tailscale вы должны правильно настроить SSH:

```bash
# Оставьте текущий SSH-сеанс открытым, пока это делаете.
# Сначала, с вашей клиентской машины, подтвердите, что вы можете подключиться через Tailscale:
ssh your-user@SERVER_TAILSCALE_IP

# Добавьте настройки упрочнения в отдельный файл вместо перезаписи sshd_config.
sudo tee /etc/ssh/sshd_config.d/99-openclaw-hardening.conf >/dev/null <<'EOF'
PasswordAuthentication no
PermitRootLogin no
KbdInteractiveAuthentication no
EOF

# Проверьте перед перезагрузкой. Не пропускайте этот шаг.
sudo sshd -t
sudo systemctl reload ssh || sudo systemctl reload sshd
```

Это отключает вход с использованием пароля и вход root. Следующий шаг использует UFW, чтобы полностью заблокировать публичный SSH, оставляя при этом доступ через `tailscale0`.

### Шаг 4: Правила брандмауэра

Настройте брандмауэр как второй уровень защиты:

```bash
# Используя UFW (Ubuntu/Debian)
sudo ufw allow in on tailscale0
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw enable
sudo ufw delete allow 22/tcp || true
sudo ufw reload
sudo ufw status verbose
```

Собственный гайд Tailscale по упрочнению Ubuntu использует ту же схему: разрешить `tailscale0`, запретить остальной входящий трафик, затем убедиться, что публичный SSH таймаутит, а SSH к адресу `100.x.y.z` всё ещё работает. Если вы запускаете публичный сайт на том же VPS, оставьте только те правила, которые действительно нужны, например `80/tcp` и `443/tcp`.

## Проверка вашей уязвимости  

### Проверка открытых портов извне  

С устройства, **не** находящегося в вашей Tailscale-сети:  

```bash
# Проверка, не открыты ли общие публичные порты
nmap -p 22,80,443,18789 YOUR_PUBLIC_IP

# Ожидаемый вывод для защищённого экземпляра:
# 22/tcp   filtered ssh
# 18789/tcp filtered unknown
```  

Если порты `22` или `18789` показывают `open` вместо `filtered` или `closed`, это проблема. Если `80` или `443` открыты, убедитесь, что это действительно ваш намеренно открытый сайт или Tailscale Funnel-эндпоинт, а не случайно открытый OpenClaw-шлюз.  

### Проверка локально прослушиваемых портов  

На вашем сервере OpenClaw:

```bash
# Показать все прослушиваемые порты и к чему они привязаны
sudo ss -tulpn | grep LISTEN

# Искать строки вроде этой (хорошо для Serve):
# tcp   LISTEN 0   128   127.0.0.1:18789   *:*
#
# Или такой (приемлемо для прямой привязки в tailnet с аутентификацией):
# tcp   LISTEN 0   128   100.x.y.z:18789   *:*
#
# Не такой (плохо):
# tcp   LISTEN 0   128   0.0.0.0:18789     *:*
```

Если вы видите `0.0.0.0` или `:::` (IPv6-эквивалент), то эта служба доступна всему миру.

### Встроенная проверка безопасности

OpenClaw включает [команду проверки безопасности](https://docs.molt.bot/gateway/security), которая проверяет вашу конфигурацию на соответствие лучшим практикам безопасности:

```bash
openclaw security audit --deep
openclaw security audit --deep --fix
```

Проверка оценивает экспозицию шлюза, режим Tailscale, настройки аутентификации, доступ по каналам, политики инструментов, инвентарь плагинов и разрешения на файлы. Рассматривайте `--fix` как полезного помощника, но не замену для анализа результатов.

## Что это не решает

Tailscale устраняет самую крупную ошибку: публичную экспозицию оператора. Он не решает всё:

**Хранение учетных данных**: OpenClaw сохраняет транскрипты сессий, токены OAuth и ключи API на диске. Убедитесь, что у них есть правильные разрешения на файлы (`chmod 600` для файлов, `chmod 700` для приватных директорий конфигураций) и они не попали в систему контроля версий. Инструмент проверки безопасности встроен для этого.

**Изоляция плагинов**: Плагины запускаются с полными правами пользователя. Устанавливайте плагины только из доверенных источников и проверяйте, какие привилегии они запрашивают. Инструмент проверки составляет инвентарь установленных плагинов.

**Безопасность устройства**: Если злоумышленник скомпрометирует ваш аккаунт Tailscale или украдет устройство в вашем tailnet, он сможет получить доступ к вашему экземпляру OpenClaw. Включите [авторизацию устройств Tailscale](../kb/1099/device-authorization/), чтобы требовать подтверждение для новых устройств.
