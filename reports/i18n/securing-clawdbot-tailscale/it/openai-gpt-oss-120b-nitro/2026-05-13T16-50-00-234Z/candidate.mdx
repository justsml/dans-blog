OpenClaw (precedentemente Clawdbot/Moltbot) ti fornisce un assistente AI personale che opera su WhatsApp, Slack, Discord, iMessage e altri canali. Ma se metti il suo gateway, i controlli dei nodi o l'SSH su Internet pubblico senza un’autenticazione robusta, stai offrendo a sconosciuti un percorso verso l’accesso shell sulla tua macchina.

Questa guida mostra l’impostazione predefinita più sicura: tieni il gateway di OpenClaw legato al loopback, esponilo solo al tuo tailnet tramite Tailscale Serve, rinforza l’SSH e verifica dall’esterno che il gateway non sia pubblico.

L’adozione rapida del progetto ha messo in luce problemi di sicurezza reali: [Scansioni Shodan hanno trovato 2.847 istanze esposte](https://socradar.io/blog/clawdbot-is-it-safe/) nelle prime settimane, e un [issue di audit di sicurezza su GitHub ha segnalato 512 vulnerabilità](https://github.com/moltbot/moltbot/issues/1796) nel codice. Parte di questi risultati proviene da scanner automatici e parte è cambiata dalla rinomina a OpenClaw nel gennaio 2026, quindi considera il numero come un segnale di allarme più che come un conteggio preciso di vulnerabilità attuali. Non è necessario essere esperti di sicurezza – devi solo evitare di pubblicare superfici operative prima del deployment.

## Quello Che Stai Davvero Esporre

A seconda di come lo hai installato e reso accessibile, ci sono tre superfici da controllare:

- **Porta 22**: accesso SSH su un VPS  
- **Porta 18789**: interfaccia di controllo del gateway e API WebSocket  
- **Controllo browser/nodo**: esecuzione remota di nodi e automazione del browser tramite il modello di accoppiamento gateway/nodo  

La documentazione corrente di [OpenClaw remote-access docs](https://docs.molt.bot/gateway/remote) indica che il WebSocket del gateway si lega all’interfaccia di loopback per impostazione predefinita e raccomanda di mantenerlo solo su loopback, a meno che non si scelga intenzionalmente un bind LAN/tailnet/personalizzato. È corretto. Il rischio appare quando si sovrascrive tale impostazione predefinita, si pubblicano porte Docker, si aggiunge un reverse proxy, si attiva Funnel o si lascia SSH aperto a tutto il mondo.  

Il gateway è il punto critico. È la superficie operativa per il tuo assistente, inclusi i percorsi di invocazione degli strumenti. Se è raggiungibile da Internet e l’autenticazione è assente, debole, aggirata o trapelata, un attaccante potrebbe guidare l’agente o invocare strumenti con i permessi dell’utente.  

Il controllo del browser è quasi altrettanto sensibile. La documentazione attuale di OpenClaw consiglia di eseguire il controllo del browser tramite un nodo accoppiato sull’host del browser e di trattare l’accoppiamento del nodo come accesso operativo. Se un gateway può invocare `system.run` su un nodo accoppiato, si tratta di esecuzione di codice remoto su quel nodo, soggetta alla politica dei nodi del gateway e alle approvazioni di esecuzione del nodo stesso.  

SSH è SSH. Se utilizzi l’autenticazione con password abilitata, i tentativi di forza bruta sono inevitabili su un VPS pubblico.

## La Soluzione Tailscale

Per OpenClaw, Tailscale ti consente l’accesso remoto senza pubblicare i servizi operativi:

1. La tua istanza OpenClaw gira su un VPS o su una macchina locale  
2. Il gateway rimane legato al loopback ed è raggiungibile tramite Tailscale Serve, oppure si lega direttamente all’indirizzo IP del tailnet con autenticazione esplicita  
3. Installi Tailscale sia sul server sia sui tuoi dispositivi personali  
4. Accedi a OpenClaw tramite il suo IP Tailscale o il nome MagicDNS  
5. Tutti gli altri su Internet non vedono nulla, a meno che tu non abiliti intenzionalmente Funnel o un altro proxy pubblico  

### Dovresti Far Gestire a OpenClaw Tailscale?

OpenClaw dispone di un’[integrazione Tailscale integrata](https://docs.molt.bot/gateway/tailscale) che può configurare `tailscale serve` o `tailscale funnel` per il gateway.

**La modalità Serve** mantiene tutto all’interno del tuo tailnet. Il gateway resta legato a `127.0.0.1` mentre Tailscale gestisce il routing e HTTPS. Quando `gateway.auth.allowTailscale` è abilitato, OpenClaw può autenticare il traffico Control UI/WebSocket usando le intestazioni di identità di Tailscale e verificare la sorgente con `tailscale whois`. Questa è la modalità corretta per la maggior parte delle distribuzioni personali.

**Modalità Funnel** espone il gateway pubblicamente tramite la funzionalità di endpoint pubblico di Tailscale. La documentazione di Tailscale descrive Funnel come il routing del traffico da Internet verso un servizio locale. OpenClaw rifiuta di avviare Funnel a meno che la modalità di autenticazione del gateway non sia `password`, ma stai comunque scegliendo di pubblicare una superficie operativa.

La [documentazione di sicurezza di OpenClaw](https://docs.molt.bot/gateway/security) è chiara: l’iniezione di prompt e l’accesso agli strumenti sono rischi fondamentali per un assistente personale. Non fornire all’agente un percorso per rendersi pubblico in silenzio. Usa Serve in modo deliberato, evita Funnel a meno che non sia davvero necessario l’accesso pubblico, e richiedi l’approvazione di esecuzione per qualsiasi comando `tailscale`.

## Configurare OpenClaw in modo sicuro

### Passo 1: Installare Tailscale

Sul tuo VPS o server locale:

```bash
# Install Tailscale
curl -fsSL https://tailscale.com/install.sh | sh

# Authenticate (opens a browser to log in)
sudo tailscale up

# Get your Tailscale IP
tailscale ip -4
# Output: 100.x.x.x
```

Sul tuo client, installa Tailscale dalla pagina di download ufficiale e accedi allo stesso tailnet.

Ora entrambe le macchine sono sulla stessa rete privata. Puoi fare ping al tuo VPS usando il suo IP Tailscale, e il traffico verrà instradato attraverso il tunnel cifrato.

### Step 2: Configure OpenClaw to Use Tailscale

Il modello più sicuro attualmente è: tenere il gateway sul loopback e renderlo accessibile al tuo tailnet con Tailscale Serve.

In configurazione di OpenClaw:

```js
{
  gateway: {
    bind: "loopback",
    tailscale: { mode: "serve" },
  },
}
```

Poi avvia il gateway con Serve:

```bash
openclaw gateway --tailscale serve
```

La documentazione di OpenClaw afferma che questo mantiene il gateway su `127.0.0.1` mentre Tailscale fornisce HTTPS e il routing nel tailnet. Lo si espone all’indirizzo `https://<magicdns-name>/`, non all’IP pubblico del VPS.

Se preferisci un bind diretto al tailnet invece di Serve, usa un’autenticazione esplicita del gateway:

```js
{
  gateway: {
    bind: "tailnet",
    auth: {
      mode: "token",
      token: "replace-with-a-long-random-token",
    },
  },
}
```

Poi connettiti da un altro dispositivo del tailnet:

```text
http://<tailscale-ip>:18789/
ws://<tailscale-ip>:18789
```

Se esegui OpenClaw in Docker o in un altro runtime di container, fai particolare attenzione alla pubblicazione delle porte. Un comando di publish come `-p 18789:18789` tipicamente si lega a tutte le interfacce host. Preferisci il loopback più Tailscale Serve, oppure vincola esplicitamente il lato host all’indirizzo IP di Tailscale dopo aver verificato che il container continui a ricevere traffico:

```bash
TAILSCALE_IP=$(tailscale ip -4)
docker run ... -p "$TAILSCALE_IP:18789:18789" ...
```

Dopo qualsiasi modifica a Docker, verifica dall’esterno con `nmap` e localmente con `ss`. Docker può aggirare o riordinare le assunzioni del firewall host se non lo tieni in considerazione.

### Step 3: Lock Down SSH

Anche con Tailscale, è necessario proteggere correttamente SSH:

```bash
# Mantieni aperta la tua sessione SSH corrente mentre esegui queste operazioni.
# Prima, dalla tua macchina client, verifica di poter accedere via SSH tramite Tailscale:
ssh your-user@SERVER_TAILSCALE_IP

# Inserisci l’indurimento in un file drop‑in invece di riscrivere sshd_config.
sudo tee /etc/ssh/sshd_config.d/99-openclaw-hardening.conf >/dev/null <<'EOF'
PasswordAuthentication no
PermitRootLogin no
KbdInteractiveAuthentication no
EOF

# Convalida prima di ricaricare. Non saltare questo passaggio.
sudo sshd -t
sudo systemctl reload ssh || sudo systemctl reload sshd
```

Questo disabilita l’autenticazione basata su password e l’accesso root. Il passo successivo utilizza UFW per bloccare completamente SSH pubblico, consentendo comunque SSH su `tailscale0`.

### Step 4: Regole del firewall

Imposta un firewall come secondo livello:

```bash
# Usando UFW (Ubuntu/Debian)
sudo ufw allow in on tailscale0
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw enable
sudo ufw delete allow 22/tcp || true
sudo ufw reload
sudo ufw status verbose
```

La guida di indurimento di Tailscale per Ubuntu utilizza la stessa struttura: consenti `tailscale0`, nega tutto il traffico in ingresso, quindi verifica che SSH pubblico vada in timeout mentre SSH verso l’indirizzo `100.x.y.z` continui a funzionare. Se sullo stesso VPS ospiti un sito web pubblico, mantieni solo le regole pubbliche realmente necessarie, ad esempio `80/tcp` e `443/tcp`.

## Verifica della tua esposizione

### Controlla le porte aperte dall’esterno

Da una macchina che **NON** fa parte della tua rete Tailscale:

```bash
# Verifica se le porte pubbliche comuni sono esposte
nmap -p 22,80,443,18789 YOUR_PUBLIC_IP

# Output atteso per un'istanza sicura:
# 22/tcp   filtered ssh
# 18789/tcp filtered unknown
```

Se la porta `22` o `18789` risulta `open` invece di `filtered` o `closed`, c’è un problema. Se le porte `80` o `443` sono aperte, assicurati che siano destinate esclusivamente al tuo sito web pubblico intenzionale o al punto finale di Tailscale Funnel, e non al gateway OpenClaw per errore.

### Controlla cosa è in ascolto localmente

Sul tuo server OpenClaw:

```bash
# Show all listening ports and what they're bound to
sudo ss -tulpn | grep LISTEN

# Look for lines like this (good for Serve):
# tcp   LISTEN 0   128   127.0.0.1:18789   *:*
#
# Or this (acceptable for direct tailnet bind with auth):
# tcp   LISTEN 0   128   100.x.y.z:18789   *:*
#
# NOT like this (bad):
# tcp   LISTEN 0   128   0.0.0.0:18789     *:*
```

Se vedi `0.0.0.0` o `:::` (equivalente IPv6), quel servizio è esposto al mondo.

### Audit di Sicurezza Integrato

OpenClaw include un [comando di audit di sicurezza](https://docs.molt.bot/gateway/security) che verifica la tua configurazione rispetto alle migliori pratiche:

```bash
openclaw security audit --deep
openclaw security audit --deep --fix
```

L’audit controlla l’esposizione del gateway, la modalità Tailscale, le impostazioni di autenticazione, l’accesso ai canali, la policy degli strumenti, l’inventario dei plugin e i permessi dei file. Considera `--fix` come un aiuto pratico, non come sostituto della lettura dei risultati.

## Cosa Non Risolve Questo


Tailscale elimina l’errore più grave: l’esposizione pubblica dell’operatore. Non risolve però tutto:

**Archiviazione delle credenziali**: OpenClaw salva le trascrizioni delle sessioni, i token OAuth e le chiavi API su disco. Assicurati che abbiano i permessi corretti (`chmod 600` per i file, `chmod 700` per le directory di configurazione private) e che non siano presenti nel version control. L’audit integrato verifica questo aspetto.

**Sandbox dei plugin**: I plugin vengono eseguiti con i permessi completi dell’utente. Installa solo plugin da fonti di cui ti fidi e controlla quali capacità richiedono. Lo strumento di audit elenca i plugin installati.

**Sicurezza dei dispositivi**: Se qualcuno compromette il tuo account Tailscale o ruba un dispositivo nella tua tailnet, può accedere all’istanza OpenClaw. Attiva [Tailscale device authorization](https://tailscale.com/kb/1099/device-authorization/) per richiedere l’approvazione dei nuovi dispositivi.

## Checklist di distribuzione

Prima di considerare la tua istanza OpenClaw/Moltbot pronta per la produzione:

- [ ] Tailscale installato e autenticato sia sul server sia sul client
- [ ] Gateway mantenuto su loopback con Tailscale Serve, o legato a `tailnet` con autenticazione esplicita
- [ ] SSH configurato per disabilitare l’autenticazione password e il login root
- [ ] Firewall (UFW o iptables/nftables) configurato per consentire `tailscale0` e bloccare l’ingresso pubblico non necessario
- [ ] Scansione nmap esterna mostra tutte le porte `filtered` o `closed`
- [ ] `ss -tulpn` interno mostra il gateway legato a `127.0.0.1`, `::1` o solo all’indirizzo IP di Tailscale
- [ ] I file di credenziali hanno permessi 600 e le directory di configurazione private permessi 700
- [ ] Esegui `openclaw security audit --deep` e risolvi tutti i risultati
- [ ] Se usi la gestione Tailscale di OpenClaw, le approvazioni di esecuzione sono abilitate
- [ ] Backup regolari configurati (dati OpenClaw + configurazioni)
