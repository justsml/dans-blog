OpenClaw（前身为 Clawdbot/Moltbot）为你提供一个跨 WhatsApp、Slack、Discord、iMessage 等渠道工作的个人 AI 助手。但如果把它的网关、节点控制或 SSH 暴露在公共互联网且没有强身份验证，你就在给陌生人提供一条直接访问你机器 shell 的路径。

本指南展示了最安全的默认做法：将 OpenClaw 的网关绑定在回环地址，仅通过 Tailscale Serve 向你的 tailnet 暴露，锁定 SSH，并从外部验证网关并未对外公开。

项目的快速普及暴露了真实的安全风险：在最初几周内，[Shodan 扫描发现了 2,847 个暴露实例](https://socradar.io/blog/clawdbot-is-it-safe/)，而一次 [GitHub 安全审计 issue 报告了 512 条发现](https://github.com/moltbot/moltbot/issues/1796)。其中部分是自动化扫描器的输出，部分自 2026 年 1 月更名为 OpenClaw 后已有所变化，所以把这个数字当作警示信号，而不是精确的当前漏洞计数。你不需要成为安全专家——只要在部署前避免公开操作面即可。

---

## 实际暴露了哪些内容

根据你的安装方式和暴露方式，有三类表面需要检查：

- **端口 22**：VPS 上的 SSH 访问  
- **端口 18789**：网关控制 UI 与 WebSocket API  
- **浏览器/节点控制**：通过网关/节点配对模型进行的远程节点执行和浏览器自动化  

当前的 [OpenClaw 远程访问文档](https://docs.molt.bot/gateway/remote) 说明网关 WebSocket 默认绑定在回环地址，并建议除非你有意选择 LAN、tailnet 或自定义绑定，否则保持回环-only。这是正确的做法。风险出现在你覆盖默认设置、发布 Docker 端口、添加反向代理、开启 Funnel，或让 SSH 对全世界开放时。

网关是最大的问题。它是你助手的操作面，包含工具调用路径。如果它可以被互联网访问且身份验证缺失、弱化、被绕过或泄露，攻击者可能利用你的用户权限驱动代理或调用工具。

浏览器控制的敏感度几乎相当。当前的 OpenClaw 文档建议在浏览器机器上通过配对的节点主机运行浏览器控制，并将节点配对视为操作访问。如果网关能够在配对节点上调用 `system.run`，则相当于对该节点的远程代码执行，受网关的节点策略和节点自身的执行批准约束。

SSH 就是 SSH。如果你开启了密码认证，在公共 VPS 上不可避免会遭遇暴力破解尝试。

## Tailscale 方案

对 OpenClaw 来说，Tailscale 能在不暴露运营服务的前提下提供远程访问：

1. 你的 OpenClaw 实例运行在 VPS 或本地机器上  
2. 网关保持绑定在回环地址，并通过 Tailscale Serve 访问，或直接绑定到 tailnet IP 并使用显式认证  
3. 在服务器和个人设备上都安装 Tailscale  
4. 通过其 Tailscale IP 或 MagicDNS 名称访问 OpenClaw  
5. 互联网上的其他人看不到任何内容，除非你主动启用 Funnel 或其他公共代理  

### 是否让 OpenClaw 管理 Tailscale？

OpenClaw 提供了[内置的 Tailscale 集成](https://docs.molt.bot/gateway/tailscale)，可以为网关配置 `tailscale serve` 或 `tailscale funnel`。

**Serve 模式** 只在你的 tailnet 内部工作。网关仍绑定在 `127.0.0.1`，由 Tailscale 负责路由和 HTTPS。当 `gateway.auth.allowTailscale` 开启时，OpenClaw 能使用 Tailscale 身份头部对 Control UI/WebSocket 流量进行身份验证，并通过 `tailscale whois` 验证来源。这是大多数个人部署的正确选择。

**Funnel 模式** 通过 Tailscale 的公共端点功能将网关公开。Tailscale 官方文档将 Funnel 描述为将来自更广泛互联网的流量路由到本地服务。OpenClaw 在网关认证模式为 `password` 时才会启动 Funnel，但这仍然是对运营面进行公共暴露的选择。

OpenClaw 的[安全文档](https://docs.molt.bot/gateway/security)明确指出，提示注入和工具访问是个人助理的核心风险。不要让代理悄悄把自己暴露到公网。请有意识地使用 Serve，除非确实需要公共访问，否则避免使用 Funnel，并对任何 `tailscale` 命令要求执行批准。

## 安全配置 OpenClaw

### 步骤 1：安装 Tailscale

在你的 VPS 或本地服务器上：

```bash
# 安装 Tailscale
curl -fsSL https://tailscale.com/install.sh | sh

# 认证（会打开浏览器登录）
sudo tailscale up

# 获取你的 Tailscale IP
tailscale ip -4
# 输出示例：100.x.x.x
```

在客户端机器上，从官方下载页面安装 Tailscale 并登录到同一个 tailnet。

此时两台机器已处于同一私有网络。你可以使用 Tailscale IP ping 你的 VPS，流量会走加密隧道。

### 步骤 2：配置 OpenClaw 使用 Tailscale

当前最安全的做法是：让网关仅绑定在回环地址，并通过 Tailscale Serve 将其暴露给 tailnet。

在 OpenClaw 配置中：

```js
{
  gateway: {
    bind: "loopback",
    tailscale: { mode: "serve" },
  },
}
```

随后使用 Serve 启动网关：

```bash
openclaw gateway --tailscale serve
```

OpenClaw 文档说明，这会让网关仅监听 `127.0.0.1`，由 Tailscale 提供 HTTPS 与 tailnet 路由。访问地址为 `https://<magicdns-name>/`，而不是你的公共 VPS IP。

如果更倾向于直接在 tailnet 上绑定而不是使用 Serve，可使用显式网关认证：

```js
{
  gateway: {
    bind: "tailnet",
    auth: {
      mode: "token",
      token: "replace-with-a-long-random-token",
    },
  },
}
```

随后从另一台 tailnet 设备连接：

```text
http://<tailscale-ip>:18789/
ws://<tailscale-ip>:18789
```

在 Docker 或其他容器运行时，务必格外留意端口发布。`-p 18789:18789` 这种发布方式通常会绑定所有宿主机接口。推荐使用回环加 Tailscale Serve，或在确认容器仍能收到流量后，将宿主机侧显式绑定到 Tailscale IP：

```bash
TAILSCALE_IP=$(tailscale ip -4)
docker run ... -p "$TAILSCALE_IP:18789:18789" ...
```

每次修改 Docker 配置后，使用 `nmap` 从外部检查，使用 `ss` 在本机检查。若未考虑容器的影响，Docker 可能会绕过或打乱宿主机防火墙的假设。

### Step 3: Lock Down SSH

即使使用 Tailscale，也必须对 SSH 做好加固：

```bash
# 在执行以下操作时保持当前 SSH 会话打开。
# 首先，在客户端机器上确认可以通过 Tailscale SSH：
ssh your-user@SERVER_TAILSCALE_IP

# 将加固规则写入 drop‑in 文件，而不是直接改动 sshd_config。
sudo tee /etc/ssh/sshd_config.d/99-openclaw-hardening.conf >/dev/null <<'EOF'
PasswordAuthentication no
PermitRootLogin no
KbdInteractiveAuthentication no
EOF

# 重新加载前先验证配置。不要跳过此步骤。
sudo sshd -t
sudo systemctl reload ssh || sudo systemctl reload sshd
```

上述配置关闭密码登录和 root 登录。下一步使用 UFW 完全阻断公共 SSH，同时保留 `tailscale0` 接口上的 SSH 通道。

### Step 4: Firewall Rules

将防火墙作为第二层防护进行配置：

```bash
# 使用 UFW (Ubuntu/Debian)
sudo ufw allow in on tailscale0
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw enable
sudo ufw delete allow 22/tcp || true
sudo ufw reload
sudo ufw status verbose
```

Tailscale 官方的 Ubuntu 加固指南也采用相同的思路：只放行 `tailscale0`，拒绝其它入站流量，然后验证公共 SSH 会超时，而对 `100.x.y.z` 地址的 SSH 仍然可用。如果在同一 VPS 上运行公共网站，只保留真正需要的公共规则，例如 `80/tcp` 和 `443/tcp`。

## 检查你的暴露面

### 从外部检查开放端口

在 **不在你的 Tailscale 网络** 的机器上运行：

```bash
# 检查常见公共端口是否被暴露
nmap -p 22,80,443,18789 YOUR_PUBLIC_IP

# 安全实例的预期输出:
# 22/tcp   filtered ssh
# 18789/tcp filtered unknown
```

如果 `22` 或 `18789` 显示为 `open` 而不是 `filtered` 或 `closed`，说明存在问题。若 `80` 或 `443` 为 open，请确认这仅是你有意公开的站点或 Tailscale Funnel 端点，而不是误将 OpenClaw 网关暴露。

### 检查本机监听情况

在你的 OpenClaw 服务器上：

```bash
# 列出所有监听端口及其绑定地址
sudo ss -tulpn | grep LISTEN

# 应出现类似以下的行（Serve 模式下的理想情况）:
# tcp   LISTEN 0   128   127.0.0.1:18789   *:*
#
# 或者如下（直接 tailnet 绑定并带认证时可接受）:
# tcp   LISTEN 0   128   100.x.y.z:18789   *:*
#
# **不要**出现如下（不安全）:
# tcp   LISTEN 0   128   0.0.0.0:18789     *:*
```

若看到 `0.0.0.0` 或 `:::`（IPv6 等价），说明该服务已对全网开放。

### 内置安全审计


OpenClaw 包含一个[安全审计命令](https://docs.molt.bot/gateway/security)，可将你的配置与安全最佳实践进行对照：

```bash
openclaw security audit --deep
openclaw security audit --deep --fix
```

审计会检查网关暴露情况、Tailscale 模式、认证设置、通道访问、工具策略、插件清单以及文件权限。将 `--fix` 视为便利的助手，而不是阅读审计结果的替代方案。

## 这并不能解决的问题

Tailscale 消除了最大的问题：公开的运营者暴露。但它并不能解决所有风险：

**凭证存储**：OpenClaw 会把会话记录、OAuth 令牌和 API 密钥写入磁盘。确保这些文件的权限设置得当（文件使用 `chmod 600`，私有配置目录使用 `chmod 700`），并且不要将其提交到版本控制。内置审计会检测此类问题。

**插件沙箱**：插件在你的用户权限下运行。只安装可信来源的插件，并审查它们请求的能力。审计工具会列出已安装的插件清单。

**设备安全**：如果有人入侵你的 Tailscale 账户或窃取了 tailnet 上的设备，他们就能访问你的 OpenClaw 实例。启用[Tailscale 设备授权](https://tailscale.com/kb/1099/device-authorization/)以强制对新设备进行批准。

## 部署清单

在将 OpenClaw/Moltbot 实例视为可投入生产之前，请确保：

- [ ] 在服务器和客户端上均已安装并完成 Tailscale 认证
- [ ] 网关仅绑定在回环地址并使用 Tailscale Serve，或在 `tailnet` 上绑定并使用显式认证
- [ ] SSH 已配置为禁用密码登录并禁止 root 登录
- [ ] 防火墙（UFW 或 iptables/nftables）已配置仅允许 `tailscale0`，并阻止所有不必要的公共入口
- [ ] 外部 nmap 扫描显示所有端口均为 `filtered` 或 `closed`
- [ ] 内部 `ss -tulpn` 显示网关仅绑定在 `127.0.0.1`、`::1` 或 Tailscale IP 上
- [ ] 凭证文件权限为 600，私有配置目录权限为 700
- [ ] 运行 `openclaw security audit --deep` 并处理所有发现的问题
- [ ] 若使用 OpenClaw 的 Tailscale 管理功能，已启用执行审批
- [ ] 已配置定期备份（OpenClaw 数据 + 配置）
