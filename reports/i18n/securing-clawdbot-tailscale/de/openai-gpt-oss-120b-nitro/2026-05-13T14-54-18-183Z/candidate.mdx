OpenClaw (früher Clawdbot/Moltbot) liefert einen persönlichen KI‑Assistenten, der über WhatsApp, Slack, Discord, iMessage und weitere Kanäle funktioniert. Wird jedoch das Gateway, die Node‑Steuerung oder SSH ohne starke Authentifizierung öffentlich im Internet bereitgestellt, erhalten Fremde einen direkten Weg zum Shell‑Zugang auf Ihrer Maschine.

Dieser Leitfaden zeigt die sicherste Voreinstellung: Das Gateway von OpenClaw nur an die Loopback‑Schnittstelle binden, es ausschließlich über Ihr Tailnet mit Tailscale Serve zugänglich machen, SSH hart absichern und von außen verifizieren, dass das Gateway nicht öffentlich erreichbar ist.

Die rasche Verbreitung des Projekts hat reale Sicherheitsprobleme ans Licht gebracht: [Shodan‑Scans fanden 2.847 exponierte Instanzen](https://socradar.io/blog/clawdbot-is-it-safe/) in den ersten Wochen, und ein [GitHub‑Security‑Audit‑Ticket meldete 512 Befunde](https://github.com/moltbot/moltbot/issues/1796) im Code. Ein Teil davon stammt von automatisierten Scannern, ein anderer Teil hat sich seit der Umbenennung zu OpenClaw im Januar 2026 geändert. Betrachten Sie die Zahl also eher als Warnsignal denn als exakte aktuelle Schwachstellen‑Count. Sie müssen kein Sicherheitsexperte sein – Sie müssen lediglich vermeiden, Betreiber‑Oberflächen zu veröffentlichen, bevor Sie das System in Betrieb nehmen.

---

## Was Sie tatsächlich exponieren

Je nach Installations‑ und Bereitstellungsart gibt es drei Oberflächen, die Sie prüfen sollten:

- **Port 22**: SSH‑Zugriff auf einem VPS  
- **Port 18789**: Gateway‑Steuer‑UI und WebSocket‑API  
- **Browser/Node‑Steuerung**: Remote‑Node‑Ausführung und Browser‑Automatisierung über das Gateway/Node‑Paarungs‑Modell  

Die aktuellen [OpenClaw remote‑access docs](https://docs.molt.bot/gateway/remote) geben an, dass der Gateway‑WebSocket standardmäßig an die Loopback‑Schnittstelle gebunden ist und empfehlen, ihn nur Loopback‑only zu belassen, sofern Sie nicht bewusst eine LAN‑/Tailnet‑/Custom‑Bindung wählen. Das ist korrekt. Das Risiko entsteht, wenn Sie diese Vorgabe überschreiben, Docker‑Ports veröffentlichen, einen Reverse‑Proxy hinzufügen, Funnel aktivieren oder SSH offen ins Internet stellen.

Das Gateway ist das Hauptproblem. Es ist die Operator‑Oberfläche für Ihren Assistenten, inklusive der Pfade zur Tool‑Aufruf‑Logik. Wenn es aus dem Internet erreichbar ist und Authentifizierung fehlt, schwach, umgangen oder geleakt ist, kann ein Angreifer den Agenten steuern oder Tools mit den Berechtigungen Ihres Benutzers ausführen.

Die Browser‑Steuerung ist fast ebenso kritisch. Die aktuellen OpenClaw‑Dokumente empfehlen, die Browser‑Steuerung über einen gepaarten Node‑Host auf dem Browser‑Rechner laufen zu lassen und die Node‑Paarung wie einen Operator‑Zugang zu behandeln. Wenn ein Gateway `system.run` auf einem gepaarten Node ausführen kann, bedeutet das Remote‑Code‑Execution auf diesem Node, abhängig von der Node‑Policy des Gateways und den eigenen Ausführungs‑Freigaben des Nodes.

SSH ist SSH. Wenn Sie die Passwort‑Authentifizierung aktiviert lassen, sind Brute‑Force‑Versuche auf einem öffentlichen VPS unvermeidlich.

## Die Tailscale‑Lösung

Für OpenClaw liefert Tailscale Fernzugriff, ohne Operator‑Dienste zu veröffentlichen:

1. Ihre OpenClaw‑Instanz läuft auf einem VPS oder einem lokalen Rechner  
2. Das Gateway bleibt an die Loopback‑Schnittstelle gebunden und wird über Tailscale Serve erreicht, oder es bindet direkt an die Tailnet‑IP mit expliziter Authentifizierung  
3. Sie installieren Tailscale sowohl auf dem Server als auch auf Ihren persönlichen Geräten  
4. Sie greifen auf OpenClaw über seine Tailscale‑IP oder den MagicDNS‑Namen zu  
5. Alle anderen im Internet sehen nichts, solange Sie nicht bewusst Funnel oder einen anderen öffentlichen Proxy aktivieren  

### Soll OpenClaw Tailscale selbst verwalten?

OpenClaw verfügt über eine [eingebaute Tailscale‑Integration](https://docs.molt.bot/gateway/tailscale), die `tailscale serve` oder `tailscale funnel` für das Gateway konfigurieren kann.

**Serve‑Modus** hält alles ausschließlich im Tailnet. Das Gateway bleibt an `127.0.0.1` gebunden, während Tailscale das Routing und HTTPS übernimmt. Ist `gateway.auth.allowTailscale` aktiviert, kann OpenClaw den Control‑UI/WebSocket‑Verkehr über Tailscale‑Identitäts‑Header authentifizieren und die Quelle mit `tailscale whois` verifizieren. Das ist der empfohlene Modus für die meisten privaten Deployments.

**Funnel‑Modus** macht das Gateway öffentlich über Tailscales Public‑Endpoint‑Funktion. In den Tailscale‑Dokumenten wird Funnel als Weiterleitung von Internet‑Traffic zu einem lokalen Service beschrieben. OpenClaw startet Funnel nur, wenn der Gateway‑Auth‑Modus auf `password` gesetzt ist – trotzdem wählst du damit eine öffentliche Angriffsfläche für einen Operator.

OpenClaws [security documentation](https://docs.molt.bot/gateway/security) weist eindeutig darauf hin, dass Prompt‑Injection und Werkzeug‑Zugriff die Hauptgefahren für einen persönlichen Assistenten darstellen. Gib dem Agenten keinen Weg, sich stillschweigend öffentlich zu machen. Nutze Serve bewusst, vermeide Funnel, es sei denn, du brauchst wirklich öffentlichen Zugriff, und fordere eine Ausführungs‑Bestätigung für jeden `tailscale`‑Befehl an.

## Sichere Einrichtung von OpenClaw

### Schritt 1: Tailscale installieren

Auf deinem VPS oder lokalen Server:

```bash
# Install Tailscale
curl -fsSL https://tailscale.com/install.sh | sh

# Authenticate (opens a browser to log in)
sudo tailscale up

# Get your Tailscale IP
tailscale ip -4
# Output: 100.x.x.x
```

Auf deinem Client‑Rechner installiere Tailscale von der offiziellen Download‑Seite und melde dich im selben Tailnet an.

Jetzt befinden sich beide Maschinen im gleichen privaten Netzwerk. Du kannst deinen VPS über seine Tailscale‑IP anpingen; der Verkehr wird durch den verschlüsselten Tunnel geleitet.

### Schritt 2: OpenClaw für Tailscale konfigurieren

Das derzeit sicherste Muster lautet: Das Gateway an die Loopback‑Schnittstelle binden und über Tailscale Serve im Tailnet verfügbar machen.

In der OpenClaw‑Konfiguration:

```js
{
  gateway: {
    bind: "loopback",
    tailscale: { mode: "serve" },
  },
}
```

Anschließend das Gateway mit Serve starten:

```bash
openclaw gateway --tailscale serve
```

OpenClaws Dokumentation besagt, dass das Gateway dabei auf `127.0.0.1` bleibt, während Tailscale HTTPS und Tailnet‑Routing bereitstellt. Sie greifen über `https://<magicdns-name>/` darauf zu, nicht über die öffentliche IP Ihres VPS.

Falls Sie stattdessen eine direkte Tailnet‑Bindung statt Serve bevorzugen, verwenden Sie eine explizite Gateway‑Authentifizierung:

```js
{
  gateway: {
    bind: "tailnet",
    auth: {
      mode: "token",
      token: "replace-with-a-long-random-token",
    },
  },
}
```

Dann verbinden Sie sich von einem anderen Tailnet‑Gerät:

```text
http://<tailscale-ip>:18789/
ws://<tailscale-ip>:18789
```

Wenn Sie OpenClaw in Docker oder einer anderen Container‑Runtime betreiben, seien Sie besonders vorsichtig beim Port‑Publishing. Ein Publish wie `-p 18789:18789` bindet in der Regel an alle Host‑Schnittstellen. Bevorzugen Sie Loopback plus Tailscale Serve oder binden Sie die Host‑Seite explizit an die Tailscale‑IP, nachdem Sie bestätigt haben, dass der Container weiterhin Traffic erhält:

```bash
TAILSCALE_IP=$(tailscale ip -4)
docker run ... -p "$TAILSCALE_IP:18789:18789" ...
```

Nach jeder Docker‑Änderung prüfen Sie von außen mit `nmap` und lokal mit `ss`. Docker kann Host‑Firewall‑Annahmen umgehen oder umordnen, wenn Sie das nicht berücksichtigen.

### Schritt 3: SSH absichern

Selbst mit Tailscale müssen Sie SSH ordnungsgemäß absichern:

```bash
# Halten Sie Ihre aktuelle SSH‑Sitzung offen, während Sie das erledigen.
# Zuerst vom Client‑Rechner aus prüfen, ob Sie über Tailscale SSH erreichen können:
ssh your-user@SERVER_TAILSCALE_IP

# Packen Sie die Härtung in eine Drop‑In‑Datei, anstatt sshd_config zu überschreiben.
sudo tee /etc/ssh/sshd_config.d/99-openclaw-hardening.conf >/dev/null <<'EOF'
PasswordAuthentication no
PermitRootLogin no
KbdInteractiveAuthentication no
EOF

# Vor dem Reload prüfen. Nicht überspringen.
sudo sshd -t
sudo systemctl reload ssh || sudo systemctl reload sshd
```

Damit werden passwortbasierte Anmeldungen und der Root‑Login deaktiviert. Der nächste Schritt nutzt UFW, um öffentlichen SSH‑Zugriff komplett zu blockieren, während SSH über `tailscale0` weiterhin erlaubt bleibt.

### Schritt 4: Firewall‑Regeln

Richten Sie eine Firewall als zweite Schutzschicht ein:

```bash
# Mit UFW (Ubuntu/Debian)
sudo ufw allow in on tailscale0
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw enable
sudo ufw delete allow 22/tcp || true
sudo ufw reload
sudo ufw status verbose
```

Tailscales eigene Ubuntu‑Hardening‑Anleitung verwendet exakt dieses Muster: `tailscale0` zulassen, sonstigen eingehenden Traffic verweigern und dann prüfen, dass öffentlicher SSH‑Zugriff zeitlich abläuft, während SSH zur `100.x.y.z`‑Adresse weiterhin funktioniert. Betreiben Sie eine öffentliche Website auf derselben VPS, behalten Sie nur die wirklich benötigten öffentlichen Regeln, etwa `80/tcp` und `443/tcp`.

## Überprüfung Ihrer Angriffsfläche

### Offene Ports von außen prüfen

Von einer Maschine, die **nicht** im Tailscale‑Netzwerk ist:

```bash
# Prüfen, ob gängige öffentliche Ports offen sind
nmap -p 22,80,443,18789 YOUR_PUBLIC_IP

# Erwartete Ausgabe für eine gesicherte Instanz:
# 22/tcp   filtered ssh
# 18789/tcp filtered unknown
```

Zeigt `22` oder `18789` den Status `open` anstelle von `filtered` oder `closed`, besteht ein Problem. Ist `80` oder `443` offen, stellen Sie sicher, dass dies nur Ihre beabsichtigte öffentliche Website oder ein Tailscale‑Funnel‑Endpunkt ist und nicht versehentlich das OpenClaw‑Gateway.

### Lokale Listener prüfen

Auf Ihrem OpenClaw‑Server:

```bash
# Zeigt alle lauschen Ports und deren Bindungen an
sudo ss -tulpn | grep LISTEN

# Suchen Sie nach Zeilen wie diesen (gut für Serve):
# tcp   LISTEN 0   128   127.0.0.1:18789   *:*
#
# Oder diesen (akzeptabel für direkte Tailnet‑Bindung mit Auth):
# tcp   LISTEN 0   128   100.x.y.z:18789   *:*
#
# NICHT wie diesen (schlecht):
# tcp   LISTEN 0   128   0.0.0.0:18789     *:*
```

Wenn Sie `0.0.0.0` oder `:::` (IPv6‑Äquivalent) sehen, ist dieser Dienst für die ganze Welt erreichbar.

### Eingebautes Sicherheits‑Audit

OpenClaw enthält einen [security audit command](https://docs.molt.bot/gateway/security), der Ihre Konfiguration gegen bewährte Sicherheitspraktiken prüft:

```bash
openclaw security audit --deep
openclaw security audit --deep --fix
```

Das Audit prüft Gateway‑Exposition, Tailscale‑Modus, Auth‑Einstellungen, Kanal‑Zugriff, Tool‑Richtlinien, Plugin‑Inventar und Dateiberechtigungen. Betrachten Sie `--fix` als nützliches Hilfsmittel, nicht als Ersatz dafür, die Ergebnisse zu lesen.

## Was das nicht löst


Tailscale eliminiert den größten Fehler: die öffentliche Exposition von Operatoren. Es löst jedoch nicht alles:

**Credential storage**: OpenClaw speichert Sitzungs‑Transkripte, OAuth‑Tokens und API‑Schlüssel auf der Festplatte. Stellen Sie sicher, dass diese korrekte Dateiberechtigungen besitzen (`chmod 600` für Dateien, `chmod 700` für private Konfigurations‑Verzeichnisse) und nicht im Versions‑Control liegen. Das integrierte Audit prüft das.

**Plugin sandboxing**: Plugins laufen mit den vollen Berechtigungen Ihres Benutzers. Installieren Sie nur Plugins aus Quellen, denen Sie vertrauen, und prüfen Sie, welche Fähigkeiten sie anfordern. Das Audit‑Tool inventarisiert installierte Plugins.

**Device security**: Wenn jemand Ihr Tailscale‑Konto kompromittiert oder ein Gerät in Ihrem Tailnet stiehlt, kann er auf Ihre OpenClaw‑Instanz zugreifen. Aktivieren Sie die [Tailscale device authorization](https://tailscale.com/kb/1099/device-authorization/), um für neue Geräte eine Genehmigung zu verlangen.

## Deployment Checklist

Bevor Sie Ihre OpenClaw/Moltbot‑Instanz als produktionsreif einstufen:

- [ ] Tailscale auf Server und Client installiert und authentifiziert
- [ ] Gateway auf Loopback mit Tailscale Serve gehalten oder an das `tailnet` gebunden mit expliziter Authentifizierung
- [ ] SSH so konfiguriert, dass Passwort‑Authentifizierung und Root‑Login deaktiviert sind
- [ ] Firewall (UFW oder iptables/nftables) so eingestellt, dass nur `tailscale0` erlaubt ist und unnötiger öffentlicher Eingangsverkehr blockiert wird
- [ ] Externer nmap‑Scan zeigt alle Ports als `filtered` oder `closed`
- [ ] Internes `ss -tulpn` zeigt das Gateway gebunden an `127.0.0.1`, `::1` oder ausschließlich an die Tailscale‑IP
- [ ] Credential‑Dateien besitzen Berechtigungen 600 und private Konfigurationsverzeichnisse Berechtigungen 700
- [ ] `openclaw security audit --deep` ausführen und alle Befunde beheben
- [ ] Bei Nutzung der OpenClaw‑Tailscale‑Verwaltung sind Ausführungs‑Freigaben aktiviert
- [ ] Regelmäßige Backups konfiguriert (OpenClaw‑Daten + Konfigurationen)
