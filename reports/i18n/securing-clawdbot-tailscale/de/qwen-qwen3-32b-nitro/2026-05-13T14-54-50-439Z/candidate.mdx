OpenClaw (früher Clawdbot/Moltbot) bietet Ihnen einen persönlichen KI-Assistenten, der über WhatsApp, Slack, Discord, iMessage und andere Kanäle arbeitet. Wenn Sie jedoch Gateway, Node-Steuerung oder SSH ohne starke Authentifizierung ins öffentliche Internet stellen, geben Sie Fremden einen Zugang zur Shell auf Ihrem System.  

Dieser Leitfaden zeigt die sicherste Standardkonfiguration: Halten Sie das OpenClaw-Gateway auf der Loopback-Schnittstelle, stellen Sie es nur über Tailscale Serve für Ihr tailnet bereit, sperren Sie SSH ab und überprüfen Sie von außen, ob das Gateway nicht öffentlich zugänglich ist.  

Die schnelle Verbreitung des Projekts hat reale Sicherheitsbedenken aufgezeigt: [Shodan-Scans fanden 2.847 offene Instanzen](https://socradar.io/blog/clawdbot-is-it-safe/) in den ersten Wochen, und ein [GitHub-Sicherheitsaudit meldete 512 Probleme](https://github.com/moltbot/moltbot/issues/1796) im Code. Ein Teil davon stammt von automatisierten Scannern, und einige Punkte haben sich seit dem Januar 2026 geändert, als das Projekt in OpenClaw umbenannt wurde. Behandeln Sie diese Zahlen daher als Warnsignal, nicht als präzise aktuelle Schwachstellenanzahl. Sie müssen kein Sicherheitsexperte sein – vermeiden Sie einfach, Operator-Oberflächen zu veröffentlichen, bevor Sie bereitstellen.  

## Was Sie tatsächlich preiszugeben drohen  

Je nachdem, wie Sie OpenClaw installiert und zugänglich gemacht haben, gibt es drei Oberflächen, die Sie prüfen sollten:

- **Port 22**: SSH-Zugriff auf einem VPS  
- **Port 18789**: Gateway-Steuerungsoberfläche und WebSocket-API  
- **Browser/Node-Steuerung**: Fernausführung auf Nodes und Browserautomatisierung über das Gateway/Node-Paring-Modell  

Die aktuellen [OpenClaw-Remote-Zugriffs-Dokumentation](https://docs.molt.bot/gateway/remote) besagen, dass die Gateway-WebSocket-Schnittstelle standardmäßig an die Loopback-Schnittstelle gebunden ist und empfehlen, dies beizubehalten, es sei denn, man wählt absichtlich eine LAN/tailnet/Anpassung. Das ist gut. Das Risiko tritt auf, wenn man diese Standardkonfiguration überschreibt, Docker-Ports veröffentlicht, einen Reverse-Proxy hinzufügt, Funnel aktiviert oder SSH für die Welt geöffnet lässt.  

Das Gateway ist der größte Kritikpunkt. Es ist die Operator-Oberfläche für Ihren Assistenten, einschließlich der Werkzeugaufruf-Pfade. Wenn es von Internet erreichbar ist und die Authentifizierung fehlt, schwach ist, umgangen wird oder ausgeleakt wird, kann ein Angreifer möglicherweise den Agenten steuern oder Werkzeuge mit den Berechtigungen Ihres Benutzers ausführen.  

Die Browsersteuerung ist fast genauso sensibel. Die aktuellen OpenClaw-Dokumentationen empfehlen, die Browsersteuerung über einen gepaarten Node-Host auf dem Browser-PC auszuführen und Node-Paring wie Operator-Zugriff zu behandeln. Wenn ein Gateway `system.run` auf einem gepaarten Node aufrufen kann, handelt es sich um Fernausführung auf diesem Node, unterliegt der Node-Richtlinie des Gateways und den eigenen Ausführungs-Berechtigungen des Nodes.  

SSH ist SSH. Wenn Sie mit aktivierter Passwortauthentifizierung laufen, sind Brute-Force-Angriffe auf einem öffentlichen VPS unvermeidlich.

## Die Tailscale-Lösung  

Für OpenClaw bietet Tailscale Ihnen Zugriff von außen, ohne Operator-Dienste öffentlich zu machen:  

1. Ihre OpenClaw-Instanz läuft auf einem VPS oder lokalen Rechner  
2. Der Gateway bleibt an loopback gebunden und wird über Tailscale Serve erreicht, oder er bindet direkt an die tailnet-IP mit expliziter Authentifizierung  
3. Sie installieren Tailscale sowohl auf dem Server als auch auf Ihren persönlichen Geräten  
4. Sie greifen auf OpenClaw über dessen Tailscale-IP oder MagicDNS-Name zu  
5. Alle anderen im Internet sehen nichts, es sei denn, Sie aktivieren absichtlich Funnel oder einen anderen öffentlichen Proxy  

### Soll OpenClaw Tailscale verwalten?  

OpenClaw verfügt über [integrierte Tailscale-Integration](https://docs.molt.bot/gateway/tailscale), die `tailscale serve` oder `tailscale funnel` für den Gateway konfigurieren kann.  

**Serve-Modus** hält alles nur innerhalb Ihres tailnets. Der Gateway bleibt an `127.0.0.1` gebunden, während Tailscale Routing und HTTPS übernimmt. Wenn `gateway.auth.allowTailscale` aktiviert ist, kann OpenClaw Control UI/WebSocket-Verkehr mithilfe von Tailscale-Identitäts-Headern authentifizieren und die Quelle mit `tailscale whois` überprüfen. Dies ist der richtige Modus für die meisten privaten Installationen.

**Funnel-Modus** macht den Gateway über das öffentliche Endpunkt-Feature von Tailscale zugänglich. Die eigenen Dokumentationen von Tailscale beschreiben Funnel als Weiterleiten von Traffic vom breiteren Internet zu einem lokalen Dienst. OpenClaw weigert sich, Funnel zu starten, es sei denn, der Gateway-Auth-Modus ist `password`, aber Sie wählen dennoch bewusst die öffentliche Exposition für eine Operator-Oberfläche.  

Die [Sicherheitsdokumentation](https://docs.molt.bot/gateway/security) von OpenClaw ist klar, dass Prompt-Injection und Tool-Zugriff zentrale Risiken für einen persönlichen Assistenten sind. Geben Sie dem Agenten keinen Weg, sich selbst unbemerkt öffentlich zu machen. Verwenden Sie Serve bewusst, vermeiden Sie Funnel, es sei denn, Sie benötigen wirklich öffentlichen Zugriff, und verlangen Sie die Ausführungsbewilligung für alle `tailscale`-Befehle.

## OpenClaw sicher einrichten  

### Schritt 1: Tailscale installieren  

Auf Ihrem VPS oder lokalen Server:

```bash
# Tailscale installieren
curl -fsSL https://tailscale.com/install.sh | sh

# Authentifizieren (öffnet einen Browser zum Anmelden)
sudo tailscale up

# Ihre Tailscale-IP abrufen
tailscale ip -4
# Ausgabe: 100.x.x.x
```

Auf Ihrem Client-Computer installieren Sie Tailscale über die offizielle Download-Seite und melden sich beim gleichen tailnet an.

Jetzt befinden sich beide Geräte im gleichen privaten Netzwerk. Sie können Ihren VPS über seine Tailscale-IP erreichen, und der Datenverkehr wird über den verschlüsselten Tunnel geleitet.

### Schritt 2: OpenClaw für die Verwendung mit Tailscale konfigurieren

Der sicherste aktuelle Ansatz ist: Halten Sie den Gateway auf der Loopback-Schnittstelle und stellen Sie ihn über Tailscale Serve für Ihr tailnet bereit.

In der OpenClaw-Konfiguration:

```js
{
  gateway: {
    bind: "loopback",
    tailscale: { mode: "serve" },
  },
}
```

Starten Sie dann den Gateway mit Serve:

```bash
openclaw gateway --tailscale serve
```

Die OpenClaw-Dokumentation erklärt, dass dies den Gateway auf `127.0.0.1` belässt, während Tailscale HTTPS und tailnet-Routing bereitstellt. Sie öffnen ihn über `https://<magicdns-name>/`, nicht über Ihre öffentliche VPS-IP.

Wenn Sie stattdessen eine direkte tailnet-Bindung ohne Serve bevorzugen, verwenden Sie explizite Gateway-Authentifizierung:

```js
{
  gateway: {
    bind: "tailnet",
    auth: {
      mode: "token",
      token: "replace-with-a-long-random-token",
    },
  },
}
```

Verbinden Sie sich dann von einem anderen tailnet-Gerät:

```text
http://<tailscale-ip>:18789/
ws://<tailscale-ip>:18789
```

Wenn Sie in Docker oder einem anderen Container-Runtime-Umgebung laufen, seien Sie besonders vorsichtig mit Port-Veröffentlichungen. Eine Veröffentlichung wie `-p 18789:18789` bindet normalerweise an alle Host-Schnittstellen. Verwenden Sie stattdessen Loopback plus Tailscale Serve oder binden Sie die Host-Seite explizit an die Tailscale-IP nach Bestätigung, dass der Container weiterhin Datenverkehr empfängt:

```bash
TAILSCALE_IP=$(tailscale ip -4)
docker run ... -p "$TAILSCALE_IP:18789:18789" ...
```

Nach jeder Docker-Änderung prüfen Sie von außen mit `nmap` und lokal mit `ss`. Docker kann Host-Feuerwall-Annahmen umgehen oder neu anordnen, wenn Sie das nicht berücksichtigen.

### Schritt 3: SSH sichern

Selbst mit Tailscale sollten Sie SSH ordnungsgemäß sichern:

```bash
# Halten Sie Ihre aktuelle SSH-Sitzung während dieses Vorgangs geöffnet.
# Zuerst von Ihrem Client-Computer bestätigen, dass Sie über Tailscale SSH können:
ssh your-user@SERVER_TAILSCALE_IP

# Stellen Sie die Sicherheit in einer Drop-in-Datei bereit, anstatt sshd_config zu überschreiben.
sudo tee /etc/ssh/sshd_config.d/99-openclaw-hardening.conf >/dev/null <<'EOF'
PasswordAuthentication no
PermitRootLogin no
KbdInteractiveAuthentication no
EOF

# Validieren Sie vor dem Neuladen. Diesen Schritt nicht überspringen.
sudo sshd -t
sudo systemctl reload ssh || sudo systemctl reload sshd
```

Dies deaktiviert die Passwort-Login-Funktion und den Root-Login. Der nächste Schritt nutzt UFW, um SSH-Verbindungen vollständig zu blockieren, während SSH über `tailscale0` weiterhin erlaubt wird.

### Schritt 4: Firewall-Regeln

Ein weiterer Schutzschicht Firewall-Regeln einrichten:

```bash
# Mit UFW (Ubuntu/Debian)
sudo ufw allow in on tailscale0
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw enable
sudo ufw delete allow 22/tcp || true
sudo ufw reload
sudo ufw status verbose
```

Der eigene Ubuntu-Sicherheitsleitfaden von Tailscale nutzt diese Struktur: Erlauben Sie `tailscale0`, verbieten Sie anderen eingehenden Datenverkehr und überprüfen Sie anschließend, dass öffentliche SSH-Verbindungen ablaufen, während SSH auf die `100.x.y.z`-Adresse weiterhin funktioniert. Wenn Sie eine öffentliche Website auf demselben VPS betreiben, lassen Sie nur die wirklich benötigten öffentlichen Regeln bestehen, z. B. `80/tcp` und `443/tcp`.

## Überprüfung Ihrer Exposition  

### Öffentliche Ports von außen prüfen  

Von einem Gerät, das **nicht** in Ihrem Tailscale-Netzwerk ist:  

```bash  
# Prüfen, ob gängige öffentliche Ports zugänglich sind  
nmap -p 22,80,443,18789 YOUR_PUBLIC_IP  

# Erwartete Ausgabe für eine gesicherte Instanz:  
# 22/tcp   filtered ssh  
# 18789/tcp filtered unknown  
```  

Wenn `22` oder `18789` den Status `open` anzeigt, anstatt `filtered` oder `closed`, besteht ein Problem. Wenn `80` oder `443` geöffnet ist, stellen Sie sicher, dass dies ausschließlich auf Ihren beabsichtigten öffentlichen Website-Endpunkt oder den Tailscale Funnel-Endpunkt zurückgeht und nicht versehentlich auf den OpenClaw-Gateway.  

### Prüfen, was lokal lauscht  

Auf Ihrem OpenClaw-Server:

```bash
# Show all listening ports and what they're bound to
sudo ss -tulpn | grep LISTEN

# Look for lines like this (good for Serve):
# tcp   LISTEN 0   128   127.0.0.1:18789   *:*
#
# Or this (acceptable for direct tailnet bind with auth):
# tcp   LISTEN 0   128   100.x.y.z:18789   *:*
#
# NOT like this (bad):
# tcp   LISTEN 0   128   0.0.0.0:18789     *:*
```

Wenn Sie `0.0.0.0` oder `:::` (IPv6-Äquivalent) sehen, ist dieser Dienst für die Welt zugänglich.

### Integrierte Sicherheitsprüfung

OpenClaw enthält einen [Sicherheitsprüfbefehl](https://docs.molt.bot/gateway/security), der Ihre Konfiguration an Sicherheitsrichtlinien misst:

```bash
openclaw security audit --deep
openclaw security audit --deep --fix
```

Die Prüfung analysiert Gateway-Zugänglichkeit, Tailscale-Modus, Authentifizierungs-Einstellungen, Kanalzugriff, Tool-Richtlinien, Plugin-Verzeichnis und Dateiberechtigungen. Behandeln Sie `--fix` als nützliche Hilfestellung, nicht als Ersatz für das Lesen der Ergebnisse.

## Was dies nicht löst

Tailscale entfernt die größte Fehlkonfiguration: Öffentliche Exposition des Betreibers. Es löst jedoch nicht alles:

**Anmeldeinformationenspeicherung**: OpenClaw speichert Sitzungsprotokolle, OAuth-Token und API-Schlüssel auf der Festplatte. Stellen Sie sicher, dass diese Dateiberechtigungen haben (`chmod 600` für Dateien, `chmod 700` für private Konfigurationsverzeichnisse) und nicht in der Versionskontrolle sind. Das integrierte Audit-Tool prüft dies.

**Plugin-Sandboxierung**: Plugins laufen mit den vollen Rechten des Benutzers. Installieren Sie nur Plugins von vertrauenswürdigen Quellen und prüfen Sie, welche Berechtigungen sie anfordern. Das Audit-Tool erfasst installierte Plugins.

**Gerätesicherheit**: Wenn jemand Ihr Tailscale-Konto kompromittiert oder ein Gerät in Ihrem Tailnet stiehlt, können sie auf Ihre OpenClaw-Instanz zugreifen. Aktivieren Sie [Tailscale-Geräteautorisierung](https://tailscale.com/kb/1099/device-authorization/), um die Genehmigung für neue Geräte zu erzwingen.

## Bereitstellungs-Checkliste

Bevor Sie Ihre OpenClaw/Moltbot-Instanz als produktionsbereit betrachten:

- [ ] Tailscale auf Server und Client installiert und authentifiziert  
- [ ] Gateway auf Loopback beschränkt mit Tailscale Serve oder an `tailnet` gebunden mit expliziter Authentifizierung  
- [ ] SSH so konfiguriert, dass Kennwortauthentifizierung und Root-Login deaktiviert sind  
- [ ] Firewall (UFW oder iptables/nftables) konfiguriert, um `tailscale0` zu erlauben und unnötige öffentliche Eingänge abzulehnen  
- [ ] Externer nmap-Scan zeigt alle Ports `filtered` oder `closed`  
- [ ] Interner `ss -tulpn` zeigt Gateway nur an `127.0.0.1`, `::1` oder der Tailscale-IP gebunden  
- [ ] Anmeldeinformationen-Dateien haben Berechtigungen 600 und private Konfigurationsverzeichnisse 700  
- [ ] Führen Sie `openclaw security audit --deep` aus und beheben Sie alle Befunde  
- [ ] Wenn OpenClaw-Tailscale-Verwaltung verwendet wird, sind Ausführungsberechtigungen aktiviert  
- [ ] Regelmäßige Backups konfiguriert (OpenClaw-Daten + Konfigurationen)
