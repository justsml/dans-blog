Большинство команд использует один инструмент поиска в Postgres. Команды, которые знают все три, доставляют более эффективный поиск с меньшей сложностью — и избегают дорогостоящего отклонения к поисковому сервису, который им ещё не нужен.  

Эйч-эс-гайд охватывает весь набор инструментов, встроенных в Postgres: что делает каждый, когда он подходит, и как их можно комбинировать.  

---

## Три инструмента  

**Полнотекстовый поиск** (`tsvector` / индекс `GIN`) лексический. Он токенизирует текст в лексемы, лемматизирует их и сопоставляет запросы с индексом. "Running" и "runs" сворачиваются в одну лексему. То же самое с "dog" и "dogs". Функция ранжирования (`ts_rank`) награждает документы, где термы запроса встречаются часто или выделяются.  

**Триграммы** (`pg_trgm`) разбивают строки на перекрывающиеся 3-символьные фрагменты и измеряют, сколько фрагментов две строки имеют в общем. "Dan" → `" da"`, `"dan"`, `"an "`. "Micheal" и "Michael" делят большинство своих триграмм, поэтому схожесть высока. Это делает `pg_trgm` отличным решением для размытого сопоставления имен, терпимости к опечаткам и автозаполнения — сферы, где полнотекстовый поиск работает плохо.

**Индексы точного совпадения** (B-tree, hash) обрабатывают первичные ключи, адреса электронной почты, идентификаторы, коды товаров и всё, где ответ бинарный: совпадает или не совпадает. Это не кажется "поиском", но они важны в этом контексте, потому что худшая практика — использовать размытый или семантический поиск для задач с однозначными ответами.

Выбор не зависит от сложности. Это о соответствии инструмента форме запроса.

## Когда полнотекстовый поиск превосходит

**Поиск ключевых слов в прозе.** Блоги, документация, описания продуктов, тикеты поддержки, юридические документы. FTS был разработан для этой формы контента: индексация и ранжирование поиска по естественному языку.

**Пользовательские запросы на основе ключевых слов.** Пользователи вводят поисковый запрос, фильтруют по тегу или просматривают по ключевому слову. FTS справляется с этим встроенными средствами без инфраструктуры встраивания.

**Ранжированные результаты без внешних зависимостей.** Индексы FTS быстры, детерминированы и не требуют вызовов API. Сигнал релевантности формируется на основе частоты терминов с весами, зависящими от позиции поля.

**Логическое фильтрация вместе с поиском.** FTS естественно сочетается с вашей существующей логикой запросов:

```sql
SELECT * FROM posts
WHERE search_vector @@ to_tsquery('english', 'postgres & performance')
  AND category = 'tutorial'
  AND published_at > NOW() - INTERVAL '6 months';
```

### Настройка FTS

```sql
-- Сгенерированный столбец автоматически поддерживает актуальность индекса
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body,  '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Запрос
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`setweight` задаёт приоритет: `A` (заголовок) имеет приоритет над `B` (тело). Это вся модель релевантности для большинства задач поиска в контенте.

### Что FTS обрабатывает плохо

- Ошибки в запросах — "javascipt" не совпадёт с "javascript"  
- Имена людей, адреса, собственные имена, которые не стеммингируются предсказуемо  
- Префиксный/автозаполнительный поиск без специальной настройки  
- Запросы, где пользователь описывает концепцию, а не называет её

## Когда выигрывают триграммы (`pg_trgm`)  

`pg_trgm` охватывает неудобный промежуточный слой, который FTS постоянно обрабатывает неправильно.  

FTS разбивает текст на лексемы и стеммингирует их. Для прозы это верно. Для имен и коротких идентификаторов это часто не так:  

- Имена людей ("Dan Levy" → стемминг зависит от словаря и настройки языка)  
- Имена компаний, адреса, названия продуктов, где важна точная орфография  
- Запросы с опечатками — "Micheal Jordan", "Amaon", "javascipt"  
- Автозаполнение/префиксный поиск  
- Частичное совпадение строк ("son" совпадает с "Johnson", "Anderson")

`pg_trgm` также независим от языка, что важно для имен из различных языковых сред. Полнотекстовый поиск требует настройки словаря для каждого языка.

### Поиск имен с опечатками

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- Находит "Micheal Jordan" при поиске "Michael Jordan"
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % оператор = порог схожести (по умолчанию 0.3)
ORDER BY score DESC
LIMIT 10;
```

Оператор `%` использует `pg_trgm.similarity_threshold` (по умолчанию 0.3, диапазон 0–1). Для поиска имен значение 0.3–0.4 позволяет находить опечатки, сохраняя при этом низкий уровень шума.

### Автозаполнение, префиксный и частичный поиск

```sql
-- Префиксное совпадение для автозаполнения. Индекс GIN на основе триграмм может помочь,
-- но B-дерево с индексом шаблонов может быть лучше для чистых левосторонних префиксов.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- word_similarity для частичных совпадений внутри длинных строк
-- ("Johnson" внутри "Andrew Johnson III")
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name
ORDER BY score DESC
LIMIT 10;
```

Индекс GIN на основе триграмм особенно полезен для запросов `ILIKE '%pattern%'` и совпадений, терпимых к опечаткам — паттернов, которые обычно требуют полного сканирования таблицы без индекса на основе триграмм.

| Сценарий | Использовать |
|---|---|
| Поиск имени человека/компании с опечатками | `pg_trgm` |
| Автозаполнение / префиксный поиск | `pg_trgm` (или FTS с префиксными запросами) |
| Короткие строки, идентификаторы, коды | `pg_trgm` |
| Статьи, документация, тикеты | FTS |
| Лог-сообщения по ключевым словам | FTS |
| Многоязычный поиск имен | `pg_trgm` (независимо от языка) |

## Когда выигрывает точное совпадение SQL

Некоторые "поисковые" задачи на самом деле не являются поиском.

"Найти пользователя с email `dan@example.com`" — это проверка на равенство. "Найти заказ `ORD-12345`" — это поиск по первичному ключу. "Список постов в категории `tutorial`, отсортированных по дате" — это фильтрация. Для таких задач подходят B-tree или хэш-индексы.

Использование FTS или триграмм здесь добавляет сложность без улучшения точности — а для точных идентификаторов близкое совпадение хуже, чем отсутствие совпадения.

```sql
CREATE INDEX users_email_idx ON users (email);

-- Точное соответствие: быстро и однозначно
SELECT id, name FROM users WHERE email = $1;
```

Основной вывод: приближенный поиск для задач с однозначными ответами — это ошибка категории. Он возвращает *что-то*, что может быть уверенно неправильным.

## Комбинация этих инструментов

Эти инструменты легко комбинируются. Вы не выбираете ровно один.

**FTS + pg_trgm для поиска с учетом опечаток в ключевых словах:**

```sql
-- Триграммовая схожесть заголовка ловит опечатки; ts_rank обрабатывает релевантность текста
SELECT id, title,
  ts_rank(search_vector, to_tsquery('simple', $1)) AS fts_rank,
  similarity(title, $1) AS trgm_score
FROM posts
WHERE search_vector @@ to_tsquery('simple', $1)
   OR title % $1
ORDER BY (ts_rank(search_vector, to_tsquery('simple', $1)) + similarity(title, $1)) DESC
LIMIT 10;
```

**FTS + `unaccent` для международного контента:**

```sql
-- Удаляем диакритические знаки, чтобы "José" совпадал с "Jose"
CREATE EXTENSION IF NOT EXISTS unaccent;

CREATE TEXT SEARCH CONFIGURATION public.simple_unaccent (COPY = pg_catalog.simple);

ALTER TEXT SEARCH CONFIGURATION public.simple_unaccent
  ALTER MAPPING FOR hword, hword_part, word
  WITH unaccent, simple;

ALTER TABLE posts ADD COLUMN search_vector tsvector;

CREATE TRIGGER posts_search_vector_refresh
BEFORE INSERT OR UPDATE OF title, body ON posts
FOR EACH ROW EXECUTE FUNCTION
  tsvector_update_trigger(search_vector, 'public.simple_unaccent', title, body);
```

**`unaccent` + `pg_trgm` для поиска имен на международных языках:**

```sql
ALTER TABLE users ADD COLUMN name_search text;

CREATE FUNCTION users_name_search_refresh()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  NEW.name_search := unaccent(coalesce(NEW.name, ''));
  RETURN NEW;
END;
$$;

CREATE TRIGGER users_name_search_refresh
BEFORE INSERT OR UPDATE OF name ON users
FOR EACH ROW EXECUTE FUNCTION users_name_search_refresh();

CREATE INDEX users_name_search_trgm_idx
  ON users USING GIN (name_search gin_trgm_ops);

SELECT id, name
FROM users
WHERE name_search % unaccent($1)
ORDER BY similarity(name_search, unaccent($1)) DESC
LIMIT 10;
```

Примеры триггеров избегают использования `unaccent()` в выражениях сгенерированных столбцов или индексов, где в PostgreSQL важны правила неизменяемости. Если вы оборачиваете `unaccent()` в свою собственную неизменяемую функцию, документируйте, что вы принимаете риск обновления/конфигурации.

## Замечательные расширения

**`pg_trgm`** включён в большинство дистрибутивов PostgreSQL, но требует явного включения. Основа для размытого сравнения строк в PostgreSQL.

**`unaccent`** удаляет диакритические знаки перед индексированием и запросами. Хорошо сочетается как с `pg_trgm`, так и с FTS для контента на языках Европы. Включён в PostgreSQL.

**`pg_bigm`** расширяет подход триграмм до биграмм (срезы из двух символов), что значительно улучшает результаты для языков CJK (Китай, Япония, Корея), где `pg_trgm` работает хуже. Должен быть установлен отдельно; не входит в стандартную поставку.

**`pg_search`** (от [ParadeDB](https://www.paradedb.com/)) заменяет стандартную стек `GIN` / `tsvector` индексом BM25 на основе Tantivy. Это даёт оценку BM25 (часто лучше `ts_rank`), размытое сопоставление в рамках FTS-запросов, фасетный поиск и значительно более быстрое индексирование больших таблиц. Это путь для обновления без изменений, когда стандартный FTS начинает показывать ограничения в ранжировании или производительности.

```sql
-- pg_search: полнотекстовый поиск BM25 с размытым сопоставлением
CREATE INDEX posts_bm25_idx ON posts
  USING bm25 (id, title, body)
  WITH (key_field = 'id', text_fields = '{"title": {}, "body": {}}');

-- Запрос с оценкой BM25 + размытым сопоставлением (находит "javascipt")
SELECT id, title, paradedb.score(id) AS rank
FROM posts
WHERE posts @@@ paradedb.fuzzy_phrase(field => 'title', value => 'postgres performnce')
ORDER BY rank DESC
LIMIT 10;
```

**`pgvector`** добавляет хранение плотных векторов и поиск по схожести. Это правильный инструмент, когда пользователи описывают, что хотят, а не называют это напрямую — семантический поиск, RAG, рекомендации по связанному контенту, мультиязычные запросы. Подробнее см. в [Семантический векторный поиск и гибридные стратегии](../semantic-vector-search-landscape).

| Даты, диапазоны, отсортированные списки | Индекс B-tree |
| Разрешения, категории, фильтры | Обычный WHERE-предикат |
| Вопросы, перефразировки, концепции | pgvector (см. следующую статью) |

Если сомневаетесь: короткие строки со вариациями написания → триграммы. Длинные тексты для запросов по ключевым словам → ПОП. Структурированные идентификаторы → обычные индексы. Концептуальные или естественно-языковые запросы → pgvector.

## Гибридный поиск: два сигнала, один ранг

Когда в строку поиска вводится запрос вроде `"withRetry timeout errors"`, он несёт два вида намерений: точные имена символов, известные пользователю (`withRetry`), и концептуальное описание (`timeout errors`). Никакой одной примитивной операции не хватает, чтобы охватить оба аспекта. Выполнение ПОП и векторного поиска параллельно, а затем объединение их ранжированных списков с помощью Reciprocal Rank Fusion (RRF) — вот решение.

RRF оценивает каждый результат как `1 / (60 + ранг)` в каждом списке и суммирует оценки по спискам. Константа 60 сглаживает преимущество верхних позиций, поэтому результат, занявший второе место в обоих списках, может обогнать результат, выигравший один список и пропустивший другой. Важно, что RRF никогда не усредняет сырые оценки между методами — ранг ПОП и косинусное расстояние — это разные меры и их нельзя комбинировать арифметически.

```sql
-- Гибридный поиск: ПОП + pgvector объединены с RRF
WITH fts AS (
  SELECT id, ts_rank(search_vector, query) AS score,
         ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM docs, to_tsquery('english', 'withRetry & timeout') query
  WHERE search_vector @@ query
  LIMIT 60
),
vec AS (
  SELECT id,
         ROW_NUMBER() OVER (ORDER BY embedding <=> $embedding) AS rank
  FROM docs
  ORDER BY embedding <=> $embedding
  LIMIT 60
)
SELECT COALESCE(fts.id, vec.id) AS id,
       (COALESCE(1.0 / (60 + fts.rank), 0) +
        COALESCE(1.0 / (60 + vec.rank), 0)) AS rrf_score
FROM fts FULL JOIN vec ON fts.id = vec.id
ORDER BY rrf_score DESC
LIMIT 10;
```

Пул из 60 документов на ветку (`LIMIT 60`) — это распространенная отправная точка. Расширьте его, если низкое восстановление; сузьте для повышения скорости.

## Что дальше

Текстовый поиск в Postgres охватывает многое, но у него есть предел. Когда пользователи описывают, что хотят, вместо того чтобы называть это прямо — «что-то, чтобы помочь мне уснуть во время полета», «статьи о отладке уверенности как новичок-инженер» — лексический и триграммный поиск не справляются.

Это территория векторных эмбеддингов, семантического поиска и гибридных архитектур. Рассмотрено в [Семантический векторный поиск и гибридные стратегии](/semantic-vector-search-landscape).
