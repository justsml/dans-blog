ほとんどのチームはPostgreSQLの検索ツールを1つしか使いません。3つのツールをすべて理解したチームは、複雑さを減らしながらより優れた検索を構築でき、まだ必要ない専用検索サービスへのコストのかかる迂回を避けることができます。

本ガイドではPostgreSQLネイティブのオプションを網羅的に解説します。それぞれの仕組み、適応ケース、および組み合わせ方について詳しく説明します。

---

## 3つのツール

**フルテキスト検索**（`tsvector` / `GIN`インデックス）は辞書的処理を行います。テキストを語彙単位（レキシム）に分割し、語幹を抽出してインデックスと照合します。「running」と「runs」は同じレキシムに統一されます。「dog」と「dogs」も同様です。ランク付け関数（`ts_rank`）は、クエリ語が文書内で頻出または目立つ位置に現れるほど高得点を付与します。

**トリグラム**（`pg_trgm`）は文字列を3文字の重複するスライスに分割し、2つの文字列がどの程度共通スライスを共有するかを測定します。「Dan」→ `" da"`、`"dan"`、`"an "`。このトリグラムの類似性により、「Micheal」と「Michael」は高い類似度を示します。この特性により`pg_trgm`は、FTSが苦手な領域（名前の曖昧一致、タイプミスの許容、オートコンプリート）に最適です。

**完全一致インデックス**（B-tree、ハッシュ）は主キー、メールアドレス、ID、SKU、答えが二分法（一致するかしないか）で決まるあらゆるデータを処理します。これらは「検索」とは思えないかもしれませんが、この話題に含めるべきです。なぜなら、正解がある問題に対して曖昧性や意味論的な検索を使うのが最悪のパターンだからです。

選択の基準は洗練度ではありません。クエリの形状にツールをマッチさせることが重要です。

## 全文検索が適している場合

**キーワードでプロゼを検索する。** ブログ投稿、ドキュメント、製品説明、サポートチケット、法的文書。FTSはこのコンテンツ形状（自然言語テキストのインデックス化とランク付け取得）のために設計されています。

**キーワードベースのユーザークエリ。** ユーザーが検索語を入力し、タグでフィルタリングする、またはキーワードでブラウズする。FTSはこれにネイティブに対応し、埋め込みインフラストラクチャを必要としない。

**外部依存なしのランク付き結果。** FTSインデックスは高速で決定的であり、API呼び出しが不要。関連性のシグナルは語頻度がフィールド位置で重み付けされる。

**検索と並行するブール型フィルタリング。** FTSは既存のクエリロジックと自然に組み合わせ可能：

```sql
SELECT * FROM posts
WHERE search_vector @@ to_tsquery('english', 'postgres & performance')
  AND category = 'tutorial'
  AND published_at > NOW() - INTERVAL '6 months';
```

### FTSの設定

```sql
-- 生成された列がインデックスを自動的に最新に保つ
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body,  '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- クエリ
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`setweight`は重要度を割り当てる：`A`（タイトル）は`B`（本文）よりも優先される。これはほとんどのコンテンツ検索ユースケースの関連性モデル全体である。

### FTSが適さないケース

- クエリ内のタイプミス — "javascipt" は "javascript" と一致しない  
- 人名、住所、辞書で予測不能にステミングされる固有名詞  
- 特別な設定なしではプレフィックス検索/オートコンプリートをサポートしない  
- ユーザーが概念を説明するクエリ（名前で直接指定しない場合）

## トリグラムが勝つケース（`pg_trgm`）

`pg_trgm` は、FTSが頻繁に失敗する不器用な中間領域をカバーします。

FTSはテキストをレキシムにトークン化し、ステミングします。これは文章の処理には適切ですが、人名や短い識別子には適していません：

- 人名（"Dan Levy" → 辞書や言語設定によってステミングが異なる）  
- 正確な綴りが重要な会社名、住所、商品タイトル  
- タイポを含むクエリ — "Micheal Jordan", "Amaon", "javascipt"  
- オートコンプリート / プレフィックス検索  
- 部分文字列マッチ（"son" が "Johnson", "Anderson" と一致する）

`pg_trgm` は言語に依存せず、多様な言語背景を持つ名前検索において重要です。FTSは言語ごとに辞書設定が必要です。

### ファジーな名前検索

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- "Michael Jordan" で "Micheal Jordan" を検出
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % 演算子 = 類似度しきい値（デフォルト 0.3）
ORDER BY score DESC
LIMIT 10;
```

`%` 演算子は `pg_trgm.similarity_threshold`（デフォルト 0.3、範囲 0–1）を使用します。名前検索では 0.3–0.4 がタイポを検出しつつノイズを抑えるバランスです。

### オートコンプリート、プレフィックス、部分一致検索

```sql
-- オートコンプリートのプレフィックスマッチ。トリグラム GIN インデックスが有効ですが、
-- 純粋な左固定プレフィックスには B-tree パターンインデックスが適しています。
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- 長い文字列内での部分一致に word_similarity を使用
-- （"Johnson" が "Andrew Johnson III" 内で一致）
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name
ORDER BY score DESC
LIMIT 10;
```

トリグラム GIN インデックスは `ILIKE '%pattern%'` による部分一致クエリや、タイポ対応のマッチングに特に有用です。トリグラムインデックスなしでは通常フルスキャンになるパターンです。

### pg_trgm を FTS に優先するタイミング

| シナリオ | 使用するツール |
|---|---|
| 入力ミスのある人物/企業名検索 | `pg_trgm` |
| オートコンプリート / プレフィックス検索 | `pg_trgm`（またはプレフィックスクエリ付きFTS） |
| 短い文字列、識別子、コード | `pg_trgm` |
| エッセイ形式の記事、ドキュメント、チケット | FTS |
| キーワードのログメッセージ | FTS |
| 多言語対応の名前検索 | `pg_trgm`（言語依存しない） |

## 等値マッチSQLが適切なケース

一部の「検索」問題は検索そのものではない。

「メールアドレス `dan@example.com` を持つユーザーを検索する」は等値チェックである。注文番号 `ORD-12345` の検索は主キー検索である。`tutorial` カテゴリに属する投稿を日付順に並べるクエリはフィルタ付きクエリである。これらはB-treeやハッシュインデックスに属する。

FTSやトリグラムを使用すると、正しさを向上させることなく複雑さを増加させる。識別子の等値検索においては、近似マッチはマッチなしよりも悪影響を及ぼす。

```sql
CREATE INDEX users_email_idx ON users (email);

-- 等値検索: 高速で曖昧性がない
SELECT id, name FROM users WHERE email = $1;
```

重要な教訓: 正解が明確な問題に対して近似検索を適用することはカテゴリエラーである。これは*何か*を返すが、それは確信的に間違っている可能性がある。

## これらのツールの組み合わせ

これらのツールはシームレスに組み合わせ可能である。1つだけを選択する必要はない。

**キーワードのタイポを許容する検索ボックスにFTS + pg_trgmを適用:**

```sql
-- タイトルのトリグラム類似度でタイポを検出し、ts_rankで本文の関連性を評価
SELECT id, title,
  ts_rank(search_vector, to_tsquery('simple', $1)) AS fts_rank,
  similarity(title, $1) AS trgm_score
FROM posts
WHERE search_vector @@ to_tsquery('simple', $1)
   OR title % $1
ORDER BY (ts_rank(search_vector, to_tsquery('simple', $1)) + similarity(title, $1)) DESC
LIMIT 10;
```

**国際的なコンテンツにFTS + `unaccent`を適用:**

```sql
-- ダイアクリティカル記号を除去して'José'と'Jose'が一致するようにする
CREATE EXTENSION IF NOT EXISTS unaccent;

CREATE TEXT SEARCH CONFIGURATION public.simple_unaccent (COPY = pg_catalog.simple);

ALTER TEXT SEARCH CONFIGURATION public.simple_unaccent
  ALTER MAPPING FOR hword, hword_part, word
  WITH unaccent, simple;

ALTER TABLE posts ADD COLUMN search_vector tsvector;

CREATE TRIGGER posts_search_vector_refresh
BEFORE INSERT OR UPDATE OF title, body ON posts
FOR EACH ROW EXECUTE FUNCTION
  tsvector_update_trigger(search_vector, 'public.simple_unaccent', title, body);
```

**`unaccent` + `pg_trgm` を使用した国際的な名前検索:**

```sql
ALTER TABLE users ADD COLUMN name_search text;

CREATE FUNCTION users_name_search_refresh()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  NEW.name_search := unaccent(coalesce(NEW.name, ''));
  RETURN NEW;
END;
$$;

CREATE TRIGGER users_name_search_refresh
BEFORE INSERT OR UPDATE OF name ON users
FOR EACH ROW EXECUTE FUNCTION users_name_search_refresh();

CREATE INDEX users_name_search_trgm_idx
  ON users USING GIN (name_search gin_trgm_ops);

SELECT id, name
FROM users
WHERE name_search % unaccent($1)
ORDER BY similarity(name_search, unaccent($1)) DESC
LIMIT 10;
```

トリガーの例では、PostgreSQLの不変性ルールが影響する生成列やインデックス式内で`unaccent()`を使用しないようにしています。`unaccent()`を独自の不変関数でラップする場合は、アップグレード/設定リスクを受入れていることを明記してください。

## 注目すべき拡張機能

**`pg_trgm`** はほとんどのPostgreSQLディストリビューションにバンドルされていますが、明示的な有効化が必要です。PostgreSQLにおける曖昧な文字列一致の基盤となる拡張機能です。

**`unaccent`** はインデックスおよびクエリの前段階でダイアクリティカル記号を除去します。`pg_trgm`およびFTSと組み合わせてヨーロッパ語系コンテンツに最適です。PostgreSQLにバンドルされています。

**`pg_bigm`** はトリグラムアプローチをビッグラム（2文字のスライス）に拡張し、CJK（中国語、日本語、韓国語）において`pg_trgm`が低性能な場合に結果を大幅に改善します。別途インストールが必要です。バンドルされていません。

**`pg_search`**（[ParadeDB](https://www.paradedb.com/) から）は標準の`GIN`/`tsvector`スタックをTantivyベースのBM25インデックスに置き換えます。これによりBM25スコアリング（多くの場合`ts_rank`より優れた性能）、FTSクエリ内のファジーマッチング、ファセット検索、大規模テーブルでの劇的に高速なインデックス作成が可能になります。標準FTSがランキングやパフォーマンスの限界に達した場合の即座に導入可能なアップグレードパスです。

```sql
-- pg_search: BM25による全文検索とファジーマッチング
CREATE INDEX posts_bm25_idx ON posts
  USING bm25 (id, title, body)
  WITH (key_field = 'id', text_fields = '{"title": {}, "body": {}}');

-- BM25スコアリング + ファジーマッチング（"javascipt"を検出）でクエリ
SELECT id, title, paradedb.score(id) AS rank
FROM posts
WHERE posts @@@ paradedb.fuzzy_phrase(field => 'title', value => 'postgres performnce')
ORDER BY rank DESC
LIMIT 10;
```

**`pgvector`** は密なベクトルの保存と類似性検索を追加します。ユーザーが名前ではなく目的を説明する場合（セマンティック検索、RAG、関連コンテンツの推奨、多言語クエリ）に適したツールです。詳しくは[セマンティックベクトル検索とハイブリッド戦略](/semantic-vector-search-landscape)を参照してください。

## 決定テーブル

| 検索対象 | 推奨 |
|---|---|
| 文章、ドキュメント、チケット | FTS |
| 変換ミスのある人物/企業名 | `pg_trgm` |
| オートコンプリート、プレフィックス検索 | `pg_trgm` |
| 短いコード、識別子 | `pg_trgm` |
| キーワードを含むログメッセージ | FTS |
| 国際的な名前 | `pg_trgm` + `unaccent` |
| 大規模コンテンツ、より良いランキング | `pg_search`（ParadeDB BM25） |
| 主キー、正確なメールアドレス、ID | B-treeインデックス |
| 日付、範囲、ソート済みリスト | B-treeインデックス |
| 権限、カテゴリ、フィルター | 標準のWHERE句 |
| 質問、言い換え、概念 | pgvector（次の記事参照） |

不確実な場合は：綴りの変化がある短い文字列 → トリグラム。キーワードクエリ用の長文 → FTS。構造化された識別子 → 通常のインデックス。概念的または自然言語クエリ → pgvector。

## ハイブリッド検索：2つのシグナル、1つのランク

クエリ「withRetry timeout errors」が検索ボックスに届いた場合、それは2種類の意図を含みます。ユーザーが知っている正確なシンボル名（withRetry）と概念的説明（timeout errors）です。どちらも単一のプリミティブではカバーできません。FTSとベクトル検索を並列で実行し、そのランク付けされたリストを逆数ランクフュージョン（RRF）でマージすることで解決できます。

RRFは各リストで「1 / (60 + ランク)」のスコアを各結果に割り当て、リスト間で合計します。定数60は上位ランクの優位性を抑えるため、どちらのリストでも2位になった結果が一方のリストで1位、もう一方で完全に見逃された結果を上回る可能性があります。重要なのは、RRFはFTSランクとコサイン距離のような異なるスコアを直接平均化しないことです。これらは異なる通貨であり、算術的に結合することはできません。

<figcaption>クエリはFTSとpgvectorに並列で分岐します。それぞれが独自のランク付けリストを生成します。RRFは各リストでのドキュメントの位置に基づいてスコアを計算し、それらを合計します。結果として、両方のシグナルが一致するドキュメントが上位に表示されます。</figcaption>

```sql
-- ハイブリッド検索: FTS + pgvectorをRRFで統合
WITH fts AS (
  SELECT id, ts_rank(search_vector, query) AS score,
         ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM docs, to_tsquery('english', 'withRetry & timeout') query
  WHERE search_vector @@ query
  LIMIT 60
),
vec AS (
  SELECT id,
         ROW_NUMBER() OVER (ORDER BY embedding <=> $embedding) AS rank
  FROM docs
  ORDER BY embedding <=> $embedding
  LIMIT 60
)
SELECT COALESCE(fts.id, vec.id) AS id,
       (COALESCE(1.0 / (60 + fts.rank), 0) +
        COALESCE(1.0 / (60 + vec.rank), 0)) AS rrf_score
FROM fts FULL JOIN vec ON fts.id = vec.id
ORDER BY rrf_score DESC
LIMIT 10;
```

各分岐の60ドキュメント候補プール（`LIMIT 60`）は一般的な初期設定です。回収率が低い場合は拡大し、速度を重視する場合は絞り込みます。
