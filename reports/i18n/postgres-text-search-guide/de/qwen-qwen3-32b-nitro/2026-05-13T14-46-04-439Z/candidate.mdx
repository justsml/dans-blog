Die meisten Teams nutzen nur ein PostgreSQL-Suchtool. Teams, die alle drei kennen, liefern bessere Suchfunktionen mit weniger Komplexität – und vermeiden den kostspieligen Umweg zu einem dedizierten Suchdienst, den sie noch nicht benötigen.  

Dieses Handbuch behandelt die gesamte Palette an PostgreSQL-eigenen Optionen: Was jede leistet, wann sie die richtige Wahl ist und wie man sie kombinieren kann.  

---

## Die drei Tools  

**Volltextsuche** (`tsvector` / `GIN`-Index) ist lexikalisch. Sie tokenisiert Text in Lexeme, stellt sie in ihre Stammform um und vergleicht Suchanfragen mit dem Index. „Running“ und „runs“ werden zum gleichen Lexem reduziert. Ebenso „dog“ und „dogs“. Die Bewertungsfunktion (`ts_rank`) belohnt Dokumente, in denen Suchbegriffe häufig oder prominent vorkommen.  

**Trigramme** (`pg_trgm`) teilen Zeichenketten in überlappende 3-Zeichen-Scheiben und messen, wie viele dieser Scheiben zwei Zeichenketten gemeinsam haben. „Dan“ → „ da“, „dan“, „an “. „Micheal“ und „Michael“ teilen die meisten ihrer Trigramme, weshalb die Ähnlichkeit hoch ist. Das macht `pg_trgm` besonders gut für unscharfe Namenssuche, Fehlertoleranz und Autocomplete – den Bereich, in dem Volltextsuche schlecht abschneidet.

**Exakte-Match-Indizes** (B-Tree, Hash) behandeln Primärschlüssel, E-Mail-Adressen, IDs, SKUs und alles, bei dem die Antwort binär ist: Es passt oder es passt nicht. Diese wirken sich nicht wie „Suche“ an, aber sie gehören in diese Diskussion, weil das schlechteste Muster darin besteht, unscharfe oder semantische Suche für Probleme einzusetzen, bei denen es korrekte Antworten gibt.  

Die Wahl geht nicht um Sophistizierung. Es geht darum, das Werkzeug an die Form der Abfrage anzupassen.

## Wenn Volltextsuche gewinnt  

**Suche nach Schlüsselwörtern in Prosa.** Blogbeiträge, Dokumentation, Produktbeschreibungen, Support-Tickets, Rechtsdokumente. Volltextsuche wurde für diesen Texttyp entwickelt: Indizierung, Ranking und Abruf über natürliche Sprache.

**Suchanfragen basierend auf Schlüsselwörtern.** Benutzer geben eine Suchbegriff ein, filtern nach Tag oder browsen nach Schlüsselwort. FTS behandelt diesen Absichtsgrad nativ, ohne jede Einbettungs-Infrastruktur.  

**Rangierte Ergebnisse ohne externe Abhängigkeiten.** FTS-Indizes sind schnell, deterministisch und benötigen keine API-Aufrufe. Das Relevanzsignal stammt aus der Termdichte, gewichtet nach Feldposition.  

**Boolesche Filterung neben der Suche.** FTS setzt sich natürlicherweise mit Ihrer bestehenden Abfrage-Logik zusammen:  

```sql
SELECT * FROM posts
WHERE search_vector @@ to_tsquery('english', 'postgres & performance')
  AND category = 'tutorial'
  AND published_at > NOW() - INTERVAL '6 months';
```

### FTS einrichten

```sql
-- Generierte Spalte hält den Index automatisch aktuell
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body,  '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Abfrage
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`setweight` weist Wichtigkeit zu: `A` (Titel) übertrifft `B` (Körper). Das ist das gesamte Relevanzmodell für die meisten Inhalts-Such-Use-Cases.  

### Was FTS nicht gut bewältigt

- Schreibfehler in Abfragen — „javascipt“ wird nicht mit „javascript“ übereinstimmen  
- Personennamen, Adressen, Eigennamen, deren Stemming nicht vorhersagbar ist  
- Vorsilbe-/Autocomplete-Suche ohne spezielle Konfiguration  
- Abfragen, bei denen der Benutzer einen Begriff beschreibt, anstatt ihn zu nennen

## Wenn Trigramme gewinnen (`pg_trgm`)  

`pg_trgm` deckt den unbeholfenen Mittelteil ab, den FTS regelmäßig verfehlt.  

FTS tokenisiert Text in Lexeme und stemmt sie. Für Prosa ist das korrekt. Für Namen und kurze Identifikatoren ist das oft nicht der Fall:  

- Personennamen („Dan Levy“ → unterschiedlich gestemmt je nach Wörterbuch und Spracheinstellung)  
- Firmennamen, Adressen, Produkttitel, bei denen die exakte Rechtschreibung entscheidend ist  
- Abfragen mit Schreibfehlern — „Micheal Jordan“, „Amaon“, „javascipt“  
- Autocomplete / Vorsilbe-Suche  
- Teilstring-Matching („son“ passt zu „Johnson“, „Anderson“)

`pg_trgm` ist ebenfalls sprachunabhängig, was bei Namen aus verschiedenen sprachlichen Hintergründen relevant ist. FTS erfordert eine Wörterbuchkonfiguration pro Sprache.  

### Fuzzy-Namen-Suche  

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- Findet "Micheal Jordan" bei der Suche nach "Michael Jordan"
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- %-Operator = Ähnlichkeitsschwellenwert (Standard: 0,3)
ORDER BY score DESC
LIMIT 10;
```  

Der `%`-Operator nutzt `pg_trgm.similarity_threshold` (Standard: 0,3, Bereich: 0–1). Bei Namenssuchen erfasst 0,3–0,4 Tippfehler, behält aber das Rauschen gering.  

### Autocomplete, Präfix- und Enthält-Suche  

```sql
-- Präfix-Matching für Autocomplete. Ein Trigramm-GIN-Index kann helfen,
-- aber ein B-Tree-Musterindex kann für reine links-angefügte Präfixe besser sein.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- word_similarity für partielle Matches innerhalb langer Strings
-- ("Johnson" innerhalb von "Andrew Johnson III")
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name
ORDER BY score DESC
LIMIT 10;
```  

Der Trigramm-GIN-Index ist besonders nützlich für `ILIKE '%pattern%'`-Enthält-Abfragen und fehlertolerantes Matching – Muster, die ohne Trigramm-Index meist Volltextscans wären.  

### Wann zu `pg_trgm` statt FTS greifen

| Szenario | Verwendung |
|---|---|
| Suche nach Personen-/Firmennamen mit Tippfehlern | `pg_trgm` |
| Autocomplete / Präfixsuche | `pg_trgm` (oder FTS mit Präfix-Abfragen) |
| Kurze Zeichenketten, Identifikatoren, Codes | `pg_trgm` |
| Prosa-Artikel, Dokumentation, Tickets | FTS |
| Log-Nachrichten nach Schlüsselwörtern | FTS |
| Multilinguale Namenssuche | `pg_trgm` (sprachunabhängig) |

## Wenn exakte Übereinstimmung in SQL gewinnt

Einige „Suche“-Probleme sind gar keine Suche.  

„Finde den Benutzer mit der E-Mail `dan@example.com`“ ist eine Gleichheitsprüfung. „Finde die Bestellung `ORD-12345`“ ist eine Primärschlüsselsuche. „Liste Beiträge in der Kategorie `tutorial` nach Datum sortiert“ ist eine gefilterte Abfrage. Diese gehören auf B-Tree- oder Hash-Indizes.  

Die Verwendung von FTS oder Trigrammen hier erhöht die Komplexität ohne die Korrektheit zu verbessern – und für exakte Identifikatoren ist eine ungefähre Übereinstimmung schlechter als keine Übereinstimmung.

```sql
CREATE INDEX users_email_idx ON users (email);

-- Exakte Suche: schnell und eindeutig
SELECT id, name FROM users WHERE email = $1;
```

Die allgemeine Lehre: Näherungssuche für Probleme mit korrekten Antworten ist ein Kategoriefehler. Sie liefert *etwas* – das möglicherweise sicher falsch ist.

## Diese Tools kombinieren

Diese Tools lassen sich sauber kombinieren. Sie wählen nicht exakt ein.

**FTS + pg_trgm für eine Suchbox, die Rechtschreibfehler in Schlüsselwörtern toleriert:**

```sql
-- Trigramm-Ähnlichkeit im Titel erfasst Rechtschreibfehler; ts_rank behandelt Relevanz des Inhalts
SELECT id, title,
  ts_rank(search_vector, to_tsquery('simple', $1)) AS fts_rank,
  similarity(title, $1) AS trgm_score
FROM posts
WHERE search_vector @@ to_tsquery('simple', $1)
   OR title % $1
ORDER BY (ts_rank(search_vector, to_tsquery('simple', $1)) + similarity(title, $1)) DESC
LIMIT 10;
```

**FTS + `unaccent` für internationale Inhalte:**

```sql
-- Diakritische Zeichen entfernen, damit "José" mit "Jose" übereinstimmt
CREATE EXTENSION IF NOT EXISTS unaccent;

CREATE TEXT SEARCH CONFIGURATION public.simple_unaccent (COPY = pg_catalog.simple);

ALTER TEXT SEARCH CONFIGURATION public.simple_unaccent
  ALTER MAPPING FOR hword, hword_part, word
  WITH unaccent, simple;

ALTER TABLE posts ADD COLUMN search_vector tsvector;

CREATE TRIGGER posts_search_vector_refresh
BEFORE INSERT OR UPDATE OF title, body ON posts
FOR EACH ROW EXECUTE FUNCTION
  tsvector_update_trigger(search_vector, 'public.simple_unaccent', title, body);
```

**`unaccent` + `pg_trgm` für internationale Namenssuche:**

```sql
ALTER TABLE users ADD COLUMN name_search text;

CREATE FUNCTION users_name_search_refresh()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  NEW.name_search := unaccent(coalesce(NEW.name, ''));
  RETURN NEW;
END;
$$;

CREATE TRIGGER users_name_search_refresh
BEFORE INSERT OR UPDATE OF name ON users
FOR EACH ROW EXECUTE FUNCTION users_name_search_refresh();

CREATE INDEX users_name_search_trgm_idx
  ON users USING GIN (name_search gin_trgm_ops);

SELECT id, name
FROM users
WHERE name_search % unaccent($1)
ORDER BY similarity(name_search, unaccent($1)) DESC
LIMIT 10;
```

Die Triggerbeispiele vermeiden die Verwendung von `unaccent()` in generierten Spalten oder Indizes, wo die Unveränderbarkeitsregeln von PostgreSQL eine Rolle spielen. Wenn Sie `unaccent()` in Ihrer eigenen unveränderlichen Funktion umschließen, dokumentieren Sie, dass Sie Risiken bei Upgrades/Konfigurationen akzeptieren.

## Auffällige Erweiterungen

**`pg_trgm`** ist in den meisten Postgres-Verteilungen enthalten, erfordert aber eine explizite Aktivierung. Die Grundlage für den unscharfen Zeichenkettenabgleich in Postgres.

**`unaccent`** entfernt diakritische Zeichen vor der Indizierung und Abfrage. Passt gut zu `pg_trgm` und FTS für inhaltliche Suchanfragen in europäischen Sprachen. Ist in Postgres enthalten.

**`pg_bigm`** erweitert den Trigram-Ansatz um Bigramme (2-Zeichen-Schnitte), was die Ergebnisse für CJK-Sprachen (Chinesisch, Japanisch, Koreanisch) deutlich verbessert, bei denen `pg_trgm` schlecht abschneidet. Muss separat installiert werden; nicht im Lieferumfang enthalten.  

**`pg_search`** (von [ParadeDB](https://www.paradedb.com/)) ersetzt den Standard-`GIN`/`tsvector`-Stapel durch einen Tantivy-basierten BM25-Index. Dies bietet BM25-Bewertung (oft besser als `ts_rank`), unscharfen Abgleich innerhalb von FTS-Abfragen, Facetten-Suche und deutlich schnellere Indizierung bei großen Tabellen. Es ist der drop-in Upgrade-Pfad, wenn die Standard-FTS-Grenzen bei Bewertung oder Leistung erreicht sind.  

```sql
-- pg_search: BM25 Volltextsuche mit unscharfer Suche
CREATE INDEX posts_bm25_idx ON posts
  USING bm25 (id, title, body)
  WITH (key_field = 'id', text_fields = '{"title": {}, "body": {}}');

-- Abfrage mit BM25-Bewertung + unscharfer Suche (fängt „javascipt“)
SELECT id, title, paradedb.score(id) AS rank
FROM posts
WHERE posts @@@ paradedb.fuzzy_phrase(field => 'title', value => 'postgres performnce')
ORDER BY rank DESC
LIMIT 10;
```

**`pgvector`** fügt dichte Vektor-Speicherung und Ähnlichkeitsuche hinzu. Die richtige Wahl, wenn Benutzer beschreiben, was sie suchen, anstatt es zu benennen – semantische Suche, RAG, Empfehlungen für verwandte Inhalte, mehrsprachige Abfragen. Ausführlich behandelt in [Semantische Vektor-Suche und Hybrid-Strategien](../semantic-vector-search-landscape).

## Entscheidungstabelle  

| Was Sie suchen | Empfohlen |  
|---|---|  
| Prosa-Artikel, Dokumente, Tickets | FTS |  
| Personen-/Unternehmensnamen mit Tippfehlern | `pg_trgm` |  
| Autocomplete, Präfixsuche | `pg_trgm` |  
| Kurze Codes, Identifikatoren | `pg_trgm` |  
| Log-Nachrichten nach Schlüsselwörtern | FTS |  
| Internationale Namen | `pg_trgm` + `unaccent` |  
| Große Inhalte, bessere Bewertung | `pg_search` (ParadeDB BM25) |  
| Primärschlüssel, exakte E-Mails, IDs | B-Tree-Index |  
| Datumsangaben, Bereiche, sortierte Listen | B-Tree-Index |  
| Berechtigungen, Kategorien, Filter | Regulärer `WHERE`-Ausdruck |  
| Fragen, Paraphrasen, Konzepte | `pgvector` (siehe nächsten Artikel) |

Wenn unsicher: kurze Zeichenketten mit Schreibvarianz → Trigramme. Lange Texte für Schlüsselwörter → FTS. Strukturierte Bezeichner → reguläre Indizes. Konzeptionelle oder natürlichsprachliche Abfragen → pgvector.

## Hybride Suche: Zwei Signale, Ein Rang

Wenn eine Abfrage wie „withRetry timeout errors“ ein Suchfeld trifft, enthält sie zwei Arten von Absicht: exakte Symbolnamen, die der Benutzer kennt (`withRetry`), und eine konzeptionelle Beschreibung (`timeout errors`). Kein einzelnes Primitiv deckt beides ab. Das gleichzeitige Ausführen von FTS und Vektor-Suche – gefolgt von der Zusammenführung der sortierten Listen mit Reciprocal Rank Fusion (RRF) – schafft das.

RRF berechnet für jedes Ergebnis `1 / (60 + Rang)` in jeder Liste und summiert die Werte über die Listen hinweg. Die Konstante 60 dämpft den Vorteil der Top-Plätze, sodass ein Ergebnis, das in beiden Listen den zweiten Platz belegt, ein Ergebnis, das eine Liste gewinnt und die andere komplett verfehlt, übertreffen kann. Entscheidend ist, dass RRF niemals Rohwerte über Methoden hinweg durchschnittelt – FTS-Rang und Kosinus-Abstand sind unterschiedliche Maßeinheiten und können nicht arithmetisch kombiniert werden.

```sql
-- Hybrid-Suche: Volltextsuche + pgvector zusammengeführt mit RRF
WITH fts AS (
  SELECT id, ts_rank(search_vector, query) AS score,
         ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM docs, to_tsquery('english', 'withRetry & timeout') query
  WHERE search_vector @@ query
  LIMIT 60
),
vec AS (
  SELECT id,
         ROW_NUMBER() OVER (ORDER BY embedding <=> $embedding) AS rank
  FROM docs
  ORDER BY embedding <=> $embedding
  LIMIT 60
)
SELECT COALESCE(fts.id, vec.id) AS id,
       (COALESCE(1.0 / (60 + fts.rank), 0) +
        COALESCE(1.0 / (60 + vec.rank), 0)) AS rrf_score
FROM fts FULL JOIN vec ON fts.id = vec.id
ORDER BY rrf_score DESC
LIMIT 10;
```

Der Kandidatenpool mit 60 Dokumenten pro Verzweigung (`LIMIT 60`) ist ein gängiger Ausgangspunkt. Erweitern Sie ihn, wenn die Trefferquote niedrig ist; verengen Sie ihn für Geschwindigkeit.

## Was kommt als Nächstes

Die Textsuche in Postgres deckt viel ab, hat aber eine Grenze. Wenn Benutzer beschreiben, was sie wollen, anstatt es zu benennen — „etwas, das mir hilft, im Flugzeug zu schlafen“, „Artikel über Debugging-Vertrauen als neuer Ingenieur“ — scheitern sowohl lexikalische als auch Trigramm-Suchen.

Dies ist das Gebiet von Vektor-Embeddings, semantischer Suche und hybriden Architekturen. Erfahren Sie mehr in [Semantische Vektor-Suche und hybride Strategien](../semantic-vector-search-landscape).
