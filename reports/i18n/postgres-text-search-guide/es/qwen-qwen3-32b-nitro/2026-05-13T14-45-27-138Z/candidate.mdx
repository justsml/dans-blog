La mayoría de los equipos usa una sola herramienta de búsqueda en Postgres. Los equipos que conocen las tres entregan búsquedas más efectivas con menos complejidad y evitan el costoso desvío hacia un servicio de búsqueda dedicado que aún no necesitan.

Esta guía cubre el conjunto completo de opciones nativas de Postgres: qué hace cada una, cuándo es la adecuada y cómo combinarlas.

---

## Las Tres Herramientas

**Búsqueda de texto completo** (`tsvector` / índice `GIN`) es léxica. Tokeniza el texto en lexemas, los lematiza y compara las consultas contra el índice. "Running" y "runs" se reducen al mismo lexema. Lo mismo ocurre con "dog" y "dogs". La función de clasificación (`ts_rank`) premia los documentos donde los términos de la consulta aparecen con frecuencia o en posiciones destacadas.

**Trigramas** (`pg_trgm`) dividen las cadenas en segmentos superpuestos de 3 caracteres y miden cuántos segmentos comparten dos cadenas. "Dan" → `" da"`, `"dan"`, `"an "`. "Micheal" y "Michael" comparten la mayoría de sus trigramas, por lo que su similitud es alta. Esto hace que `pg_trgm` sea excelente para coincidencias borrosas de nombres, tolerancia a errores tipográficos y autocompletado — el espacio donde la Búsqueda de Texto Completo (FTS) se desempeña mal.

**Índices de coincidencia exacta** (B-tree, hash) manejan claves primarias, direcciones de correo electrónico, IDs, códigos SKU y cualquier problema donde la respuesta sea binaria: coincide o no coincide. Estos no parecen "búsqueda", pero pertenecen a esta conversación porque el peor patrón es usar búsqueda borrosa o semántica para problemas que tienen respuestas correctas.

La elección no se trata de sofisticación. Se trata de ajustar la herramienta a la forma de la consulta.

## Cuando gana la Búsqueda de Texto Completo

**Búsqueda de palabras clave en textos largos.** Posts de blog, documentación, descripciones de productos, tickets de soporte, documentos legales. La Búsqueda de Texto Completo (FTS) fue diseñada para este tipo de contenido: indexación y recuperación con clasificación sobre texto en lenguaje natural.

**Consultas basadas en palabras clave.** Los usuarios escriben un término de búsqueda, filtran por etiqueta o navegan por palabra clave. La Búsqueda de Texto Completo (FTS) maneja esta intención de forma nativa sin necesidad de infraestructura de embebidos.

**Resultados clasificados sin dependencias externas.** Los índices de FTS son rápidos, deterministas y no requieren llamadas a API. La señal de relevancia proviene de la frecuencia de términos ponderada por posición del campo.

**Filtrado booleano junto con la búsqueda.** FTS se compone naturalmente con la lógica de consulta existente:

```sql
SELECT * FROM posts
WHERE search_vector @@ to_tsquery('english', 'postgres & performance')
  AND category = 'tutorial'
  AND published_at > NOW() - INTERVAL '6 months';
```

### Configuración de FTS

```sql
-- Columna generada que mantiene el índice actualizado automáticamente
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body,  '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Consulta
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`setweight` asigna importancia: `A` (título) supera a `B` (cuerpo). Este es el modelo completo de relevancia para la mayoría de los casos de uso de búsqueda de contenido.

### Lo que FTS no maneja bien

- Errores tipográficos en las consultas — "javascipt" no coincidirá con "javascript"  
- Nombres de personas, direcciones o sustantivos propios que no se reducen a su raíz de forma predecible  
- Búsqueda por prefijo o autocompletar sin configuración especial  
- Consultas donde el usuario describe un concepto en lugar de nombrarlo directamente

## Cuando ganan los trigramas (`pg_trgm`)  

`pg_trgm` aborda el terreno intermedio incómodo que la Búsqueda de texto completo (FTS) maneja mal consistentemente.  

La Búsqueda de texto completo tokeniza el texto en lexemas y los reduce a su raíz. Esto es correcto para textos narrativos, pero no lo es para nombres y identificadores cortos:  

- Nombres de personas ("Dan Levy" → reducido de forma diferente según el diccionario y la configuración de idioma)  
- Nombres de empresas, direcciones o títulos de productos donde importa la ortografía exacta  
- Consultas con errores tipográficos — "Micheal Jordan", "Amaon", "javascipt"  
- Autocompletar o búsqueda por prefijo  
- Coincidencia parcial de cadenas ("son" coincidirá con "Johnson", "Anderson")

`pg_trgm` también es independiente del idioma, lo cual es relevante para nombres de diversos orígenes lingüísticos. La búsqueda de texto completo (FTS) requiere una configuración de diccionario por idioma.

### Búsqueda de nombres con errores

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- Encuentra "Micheal Jordan" al buscar "Michael Jordan"
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- % = umbral de similitud (predeterminado 0.3)
ORDER BY score DESC
LIMIT 10;
```

El operador `%` utiliza `pg_trgm.similarity_threshold` (predeterminado 0.3, rango 0–1). Para búsqueda de nombres, 0.3–0.4 captura errores tipográficos manteniendo bajo el ruido.

### Autocompletar, prefijo y búsqueda contiene

```sql
-- Coincidencia de prefijo para autocompletar. Un índice GIN de trigramas puede ayudar,
-- pero un índice de patrón B-tree podría ser mejor para prefijos anclados a la izquierda.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- word_similarity para coincidencias parciales en cadenas largas
-- ("Johnson" dentro de "Andrew Johnson III")
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name
ORDER BY score DESC
LIMIT 10;
```

El índice GIN de trigramas es especialmente útil para consultas `ILIKE '%patrón%'` de contiene y coincidencia tolerante a errores tipográficos — patrones que normalmente requieren escaneos de tabla completa sin índice de trigramas.

### Cuándo elegir pg_trgm sobre FTS

| Escenario | Uso |
|---|---|
| Búsqueda de nombres de personas/empresas con errores tipográficos | `pg_trgm` |
| Autocompletar/búsqueda por prefijo | `pg_trgm` (o FTS con consultas de prefijo) |
| Cadenas cortas, identificadores, códigos | `pg_trgm` |
| Artículos, documentación, tickets | FTS |
| Mensajes de registro por palabras clave | FTS |
| Búsqueda multilingüe de nombres | `pg_trgm` (independiente del idioma) |

## Cuando gana el SQL de coincidencia exacta

Algunos "problemas de búsqueda" no son búsquedas en absoluto.

"Encontrar al usuario con correo `dan@example.com`" es una verificación de igualdad. "Encontrar el pedido `ORD-12345`" es una búsqueda de clave primaria. "Listar publicaciones en la categoría `tutorial` ordenadas por fecha" es una consulta filtrada. Estos casos pertenecen a índices B-tree o hash.

Usar FTS o trigramas aquí añade complejidad sin mejorar la corrección — y para identificadores exactos, una coincidencia aproximada es peor que ninguna coincidencia.

```sql
CREATE INDEX users_email_idx ON users (email);

-- Búsqueda exacta: rápida y sin ambigüedad
SELECT id, name FROM users WHERE email = $1;
```

La lección más amplia: usar búsquedas aproximadas para problemas con respuestas correctas es un error categorial. Devuelve *algo* —que podría estar claramente equivocado.

## Combinando estas herramientas

Estas herramientas se integran limpiamente. No elijes exactamente una.

**FTS + pg_trgm para una caja de búsqueda que tolera errores tipográficos en palabras clave:**

```sql
-- La similitud trigramática en el título captura errores tipográficos; ts_rank maneja la relevancia del cuerpo
SELECT id, title,
  ts_rank(search_vector, to_tsquery('simple', $1)) AS fts_rank,
  similarity(title, $1) AS trgm_score
FROM posts
WHERE search_vector @@ to_tsquery('simple', $1)
   OR title % $1
ORDER BY (ts_rank(search_vector, to_tsquery('simple', $1)) + similarity(title, $1)) DESC
LIMIT 10;
```

**FTS + `unaccent` para contenido internacional:**

```sql
-- Eliminar los signos diacríticos para que "José" coincida con "Jose"
CREATE EXTENSION IF NOT EXISTS unaccent;

CREATE TEXT SEARCH CONFIGURATION public.simple_unaccent (COPY = pg_catalog.simple);

ALTER TEXT SEARCH CONFIGURATION public.simple_unaccent
  ALTER MAPPING FOR hword, hword_part, word
  WITH unaccent, simple;

ALTER TABLE posts ADD COLUMN search_vector tsvector;

CREATE TRIGGER posts_search_vector_refresh
BEFORE INSERT OR UPDATE OF title, body ON posts
FOR EACH ROW EXECUTE FUNCTION
  tsvector_update_trigger(search_vector, 'public.simple_unaccent', title, body);
```

**`unaccent` + `pg_trgm` para búsquedas de nombres internacionales:**

```sql
ALTER TABLE users ADD COLUMN name_search text;

CREATE FUNCTION users_name_search_refresh()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  NEW.name_search := unaccent(coalesce(NEW.name, ''));
  RETURN NEW;
END;
$$;

CREATE TRIGGER users_name_search_refresh
BEFORE INSERT OR UPDATE OF name ON users
FOR EACH ROW EXECUTE FUNCTION users_name_search_refresh();

CREATE INDEX users_name_search_trgm_idx
  ON users USING GIN (name_search gin_trgm_ops);

SELECT id, name
FROM users
WHERE name_search % unaccent($1)
ORDER BY similarity(name_search, unaccent($1)) DESC
LIMIT 10;
```

Los ejemplos de gatillo evitan usar `unaccent()` dentro de expresiones de columna generada o índices, donde importan las reglas de inmutabilidad de PostgreSQL. Si envuelve `unaccent()` en su propia función inmutable, documente que acepta el riesgo de actualización/configuración.

## Extensiones destacables

**`pg_trgm`** viene incluido con la mayoría de las distribuciones de Postgres pero requiere habilitación explícita. Fundamento del emparejamiento de cadenas difusas en Postgres.

**`unaccent`** elimina los signos diacríticos antes de indexar y consultar. Combina bien con `pg_trgm` y FTS para contenido en idiomas europeos. Incluido con Postgres.

**`pg_bigm`** extiende el enfoque de trigramas a bigramas (fragmentos de 2 caracteres), lo que mejora significativamente los resultados para idiomas CJK (chino, japonés, coreano), donde `pg_trgm` no funciona bien. Debe instalarse por separado; no está incluido.  

**`pg_search`** (de [ParadeDB](https://www.paradedb.com/)) reemplaza la pila estándar `GIN` / `tsvector` con un índice BM25 basado en Tantivy. Esto ofrece puntuación BM25 (a menudo mejor que `ts_rank`), coincidencia difusa dentro de consultas de FTS, búsqueda facetada y un índice mucho más rápido en tablas grandes. Es una actualización directa cuando la FTS estándar muestra límites en clasificación o rendimiento.  

```sql
-- pg_search: Búsqueda de texto completo BM25 con coincidencia difusa
CREATE INDEX posts_bm25_idx ON posts
  USING bm25 (id, title, body)
  WITH (key_field = 'id', text_fields = '{"title": {}, "body": {}}');

-- Consulta con puntuación BM25 + coincidencia difusa (captura "javascipt")
SELECT id, title, paradedb.score(id) AS rank
FROM posts
WHERE posts @@@ paradedb.fuzzy_phrase(field => 'title', value => 'postgres performnce')
ORDER BY rank DESC
LIMIT 10;
```

**`pgvector`** agrega almacenamiento de vectores densos y búsqueda de similitud. Es la herramienta adecuada cuando los usuarios describen lo que quieren en lugar de nombrarlo: búsqueda semántica, RAG, recomendaciones de contenido relacionado, consultas multilingües. Cubierto en profundidad en [Búsqueda de Vectores Semánticos y Estrategias Híbridas](../semantic-vector-search-landscape).

## Tabla de decisiones  

| Qué estás buscando | Recomendado |
|---|---|
| Artículos, documentos, tickets | FTS |
| Nombres de personas/empresas con errores tipográficos | `pg_trgm` |
| Autocompletar, búsqueda por prefijo | `pg_trgm` |
| Códigos cortos, identificadores | `pg_trgm` |
| Mensajes de registro por palabras clave | FTS |
| Nombres internacionales | `pg_trgm` + `unaccent` |
| Contenido extenso, mejor clasificación | `pg_search` (ParadeDB BM25) |
| Claves primarias, correos exactos, IDs | Índice B-tree |
| Fechas, rangos, listas ordenadas | Índice B-tree |
| Permisos, categorías, filtros | Cláusula WHERE normal |
| Preguntas, paráfrasis, conceptos | pgvector (ver artículo siguiente) |

Cuando no estés seguro: cadenas cortas con variaciones de ortografía → trigramas. Prosa larga para consultas por palabras clave → Búsqueda de texto completo (FTS). Identificadores estructurados → índices normales. Consultas conceptuales o en lenguaje natural → pgvector.

## Búsqueda híbrida: Dos señales, un rango

Cuando una consulta como `"withRetry timeout errors"` llega a un buscador, lleva dos tipos de intención: nombres de símbolos exactos que el usuario conoce (`withRetry`) y una descripción conceptual (`timeout errors`). Ningún primitivo cubre ambos. Ejecutar la búsqueda de texto completo y la búsqueda vectorial en paralelo, y luego fusionar sus listas ordenadas con Fusión de Rango Recíproco (RRF), sí lo hace.

RRF puntuación cada resultado como `1 / (60 + rango)` en cada lista y suma las puntuaciones entre listas. La constante 60 atenúa la ventaja de los primeros rangos, por lo que un resultado que ocupe el segundo lugar en ambas listas puede superar a uno que gane una lista y se pierda la otra por completo. Crucialmente, RRF nunca promedia las puntuaciones brutas entre métodos: el rango de FTS y la distancia coseno son monedas distintas y no se pueden combinar aritméticamente.

```sql
-- Búsqueda híbrida: FTS + pgvector fusionados con RRF
WITH fts AS (
  SELECT id, ts_rank(search_vector, query) AS score,
         ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM docs, to_tsquery('english', 'withRetry & timeout') query
  WHERE search_vector @@ query
  LIMIT 60
),
vec AS (
  SELECT id,
         ROW_NUMBER() OVER (ORDER BY embedding <=> $embedding) AS rank
  FROM docs
  ORDER BY embedding <=> $embedding
  LIMIT 60
)
SELECT COALESCE(fts.id, vec.id) AS id,
       (COALESCE(1.0 / (60 + fts.rank), 0) +
        COALESCE(1.0 / (60 + vec.rank), 0)) AS rrf_score
FROM fts FULL JOIN vec ON fts.id = vec.id
ORDER BY rrf_score DESC
LIMIT 10;
```

La piscina de 60 documentos por rama (`LIMIT 60`) es un punto de partida común. Amplíe esta piscina si la recuperación es baja; reduzca su tamaño para aumentar la velocidad.

## ¿Qué sigue?

La búsqueda de texto en Postgres cubre mucho terreno, pero tiene un techo. Cuando los usuarios describen lo que quieren en lugar de nombrarlo explícitamente — "algo para ayudarme a dormir en un vuelo", "artículos sobre depuración de confianza como ingeniero nuevo" —, tanto la búsqueda léxica como por trigramas fallan.

Ese es el territorio de las incrustaciones vectoriales, la búsqueda semántica y las arquitecturas híbridas. Cubierto en [Búsqueda Vectorial Semántica y Estrategias Híbridas](../semantic-vector-search-landscape).
