La plupart des équipes n’utilisent qu’un seul outil de recherche Postgres. Les équipes qui maîtrisent les trois livrent un meilleur moteur de recherche avec moins de complexité — et évitent le détour coûteux vers un service de recherche dédié qu’elles n’avaient pas encore besoin d’utiliser.

Ce guide couvre l’ensemble complet des options natives de Postgres : ce que chaque option fait, dans quels cas elle convient, et comment les superposer.

---

## Les trois outils

**Recherche plein texte** (`tsvector` / index `GIN`) est lexicale. Elle découpe le texte en lexèmes, les radicialise, puis compare les requêtes à l’index. « Running » et « runs » se réduisent au même lexème. Il en va de même pour « dog » et « dogs ». La fonction de classement (`ts_rank`) favorise les documents où les termes de la requête apparaissent fréquemment ou de façon proéminente.

**Trigrammes** (`pg_trgm`) découpent les chaînes en tranches superposées de trois caractères et mesurent le nombre de tranches partagées entre deux chaînes. « Dan » → `" da"`, `"dan"`, `"an "`. « Micheal » et « Michael » partagent la plupart de leurs trigrammes, donc la similarité est élevée. Cela rend `pg_trgm` excellent pour la correspondance floue de noms, la tolérance aux fautes de frappe et l’autocomplétion — le domaine où la recherche plein texte montre ses limites.

**Les index à correspondance exacte** (B‑tree, hash) gèrent les clés primaires, adresses e‑mail, identifiants, SKU, et tout ce dont la réponse est binaire : ça correspond ou ça ne correspond pas. Cela ne ressemble pas à de la « recherche », mais cela fait partie de la discussion parce que le pire scénario est d’utiliser une recherche floue ou sémantique pour des problèmes qui ont des réponses précises.

Le choix ne dépend pas de la sophistication. Il s’agit d’associer l’outil à la forme de la requête.

## Quand la recherche plein texte l’emporte

**Recherche de mots‑clés dans la prose.** Articles de blog, documentation, descriptions de produits, tickets de support, documents juridiques. Le FTS a été conçu pour ce type de contenu : récupération indexée et classée sur du texte en langage naturel.

**Requêtes utilisateur basées sur des mots‑clés.** Les utilisateurs saisissent un terme de recherche, filtrent par tag ou parcourent par mot‑clé. Le FTS répond à cette intention de façon native, sans aucune infrastructure d’embeddings.

**Résultats classés sans dépendances externes.** Les index FTS sont rapides, déterministes et ne nécessitent aucun appel d’API. Le signal de pertinence provient de la fréquence des termes pondérée par la position du champ.

**Filtrage booléen conjointement à la recherche.** Le FTS se compose naturellement avec votre logique de requête existante :

```sql
SELECT * FROM posts
WHERE search_vector @@ to_tsquery('english', 'postgres & performance')
  AND category = 'tutorial'
  AND published_at > NOW() - INTERVAL '6 months';
```

### Mise en place du FTS

```sql
-- La colonne générée maintient l'index à jour automatiquement
ALTER TABLE posts ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body,  '')), 'B')
  ) STORED;

CREATE INDEX posts_search_idx ON posts USING GIN (search_vector);

-- Requête
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'postgres & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

`setweight` attribue l’importance : `A` (titre) prime sur `B` (corps). C’est l’ensemble du modèle de pertinence pour la plupart des cas d’usage de recherche de contenu.

### Ce que le FTS gère mal

- Fautes de frappe dans les requêtes — « javascipt » ne correspondra pas à « javascript »
- Noms de personnes, adresses, noms propres qui ne se réduisent pas de façon prévisible
- Recherche par préfixe/autocomplétion sans configuration spéciale
- Requêtes où l’utilisateur décrit un concept plutôt que de le nommer

## Quand les trigrammes gagnent (`pg_trgm`)

`pg_trgm` couvre le milieu maladroit que le FTS rate systématiquement.

Le FTS découpe le texte en lexèmes et les ramène à leur racine. Pour la prose, c’est approprié. Pour les noms et les identifiants courts, ce n’est souvent pas le cas :

- Noms de personnes (« Dan Levy » → racine différente selon le dictionnaire et la configuration linguistique)
- Noms d’entreprise, adresses, titres de produits où l’orthographe exacte compte
- Requêtes avec fautes de frappe — « Micheal Jordan », « Amaon », « javascipt »
- Autocomplétion / recherche par préfixe
- Correspondance partielle de chaînes (« son » correspondant à « Johnson », « Anderson »)

`pg_trgm` est également indépendant de la langue, ce qui est important pour les noms issus de contextes linguistiques variés. Le FTS nécessite une configuration de dictionnaire par langue.

### Recherche de noms floue

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX users_name_trgm_idx ON users USING GIN (name gin_trgm_ops);

-- Trouve « Micheal Jordan » lorsqu’on recherche « Michael Jordan »
SELECT id, name, similarity(name, $1) AS score
FROM users
WHERE name % $1          -- opérateur % = seuil de similarité (par défaut 0.3)
ORDER BY score DESC
LIMIT 10;
```

L’opérateur `%` utilise `pg_trgm.similarity_threshold` (par défaut 0.3, intervalle 0–1). Pour la recherche de noms, 0.3–0.4 capture les fautes de frappe tout en maintenant le bruit à un niveau faible.

### Autocomplétion, préfixe et recherche de sous‑chaîne

```sql
-- Correspondance de préfixe pour l’autocomplétion. Un index GIN trigramme peut aider,
-- mais un index B‑tree sur le motif peut être plus efficace pour les préfixes strictement ancrés à gauche.
SELECT name FROM users
WHERE name ILIKE $1 || '%'
ORDER BY name
LIMIT 10;

-- word_similarity pour les correspondances partielles à l’intérieur de chaînes plus longues
-- (« Johnson » dans « Andrew Johnson III »)
SELECT id, name, word_similarity($1, name) AS score
FROM users
WHERE $1 <% name
ORDER BY score DESC
LIMIT 10;
```

L’index GIN trigramme est particulièrement utile pour les requêtes `ILIKE '%pattern%'` de type « contient » et pour la correspondance tolérante aux fautes de frappe — des motifs qui, sans index trigramme, entraînent généralement des scans complets de la table.

### Quand privilégier pg_trgm plutôt que le FTS

| Scénario | Utilisation |
|---|---|
| Recherche de nom de personne/entreprise avec fautes de frappe | `pg_trgm` |
| Autocomplétion / recherche par préfixe | `pg_trgm` (ou FTS avec requêtes préfixées) |
| Chaînes courtes, identifiants, codes | `pg_trgm` |
| Articles de prose, documentation, tickets | FTS |
| Messages de log pour mots‑clés | FTS |
| Recherche multilingue de noms | `pg_trgm` (indépendant de la langue) |

## Quand l’Exact‑Match SQL l’emporte

Certains « problèmes de recherche » ne sont pas de la recherche du tout.

« Trouver l’utilisateur avec l’e‑mail `dan@example.com` » est une vérification d’égalité. « Trouver la commande `ORD-12345` » est une recherche par clé primaire. « Lister les articles de la catégorie `tutorial` triés par date » est une requête filtrée. Ces cas relèvent des index B‑tree ou hash.

Utiliser le FTS ou les trigrammes ici ajoute de la complexité sans améliorer la justesse — et pour des identifiants exacts, une correspondance approximative est pire qu’une absence de correspondance.

```sql
CREATE INDEX users_email_idx ON users (email);

-- Recherche exacte : rapide et sans ambiguïté
SELECT id, name FROM users WHERE email = $1;
```

Leçon générale : appliquer une recherche approximative à un problème qui possède une réponse exacte constitue une erreur de catégorie. On obtient *quelque chose* — qui peut être manifestement erroné.

## Combiner ces outils

Ces outils s’assemblent proprement. Vous ne choisissez pas un seul.

**FTS + pg_trgm pour une boîte de recherche tolérante aux fautes de frappe dans les mots‑clés :**

```sql
-- La similarité trigramme sur le titre capte les fautes ; ts_rank gère la pertinence du corps
SELECT id, title,
  ts_rank(search_vector, to_tsquery('simple', $1)) AS fts_rank,
  similarity(title, $1) AS trgm_score
FROM posts
WHERE search_vector @@ to_tsquery('simple', $1)
   OR title % $1
ORDER BY (ts_rank(search_vector, to_tsquery('simple', $1)) + similarity(title, $1)) DESC
LIMIT 10;
```

**FTS + `unaccent` pour du contenu international :**

```sql
-- Supprimer les marques diacritiques afin que "José" corresponde à "Jose"
CREATE EXTENSION IF NOT EXISTS unaccent;

CREATE TEXT SEARCH CONFIGURATION public.simple_unaccent (COPY = pg_catalog.simple);

ALTER TEXT SEARCH CONFIGURATION public.simple_unaccent
  ALTER MAPPING FOR hword, hword_part, word
  WITH unaccent, simple;

ALTER TABLE posts ADD COLUMN search_vector tsvector;

CREATE TRIGGER posts_search_vector_refresh
BEFORE INSERT OR UPDATE OF title, body ON posts
FOR EACH ROW EXECUTE FUNCTION
  tsvector_update_trigger(search_vector, 'public.simple_unaccent', title, body);
```

**`unaccent` + `pg_trgm` pour la recherche de noms internationaux :**

```sql
ALTER TABLE users ADD COLUMN name_search text;

CREATE FUNCTION users_name_search_refresh()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  NEW.name_search := unaccent(coalesce(NEW.name, ''));
  RETURN NEW;
END;
$$;

CREATE TRIGGER users_name_search_refresh
BEFORE INSERT OR UPDATE OF name ON users
FOR EACH ROW EXECUTE FUNCTION users_name_search_refresh();

CREATE INDEX users_name_search_trgm_idx
  ON users USING GIN (name_search gin_trgm_ops);

SELECT id, name
FROM users
WHERE name_search % unaccent($1)
ORDER BY similarity(name_search, unaccent($1)) DESC
LIMIT 10;
```

Les exemples de déclencheurs évitent d’utiliser `unaccent()` directement dans les colonnes générées ou les expressions d’index, où les règles d’immuabilité de PostgreSQL sont critiques. Si vous encapsulez `unaccent()` dans votre propre fonction immuable, indiquez clairement que vous acceptez le risque lié aux mises à jour ou à la configuration.

## Extensions remarquables

**`pg_trgm`** est fourni avec la plupart des distributions PostgreSQL mais doit être activé explicitement. C’est la base du rapprochement flou de chaînes dans Postgres.

**`unaccent`** supprime les marques diacritiques avant l’indexation et l’interrogation. Il se combine bien avec `pg_trgm` et la recherche plein texte pour le contenu des langues européennes. Fournie avec PostgreSQL.

**`pg_bigm`** étend l’approche trigramme aux bigrammes (tranches de 2 caractères), ce qui améliore nettement les résultats pour les langues CJK (chinois, japonais, coréen) où `pg_trgm` est insuffisant. Doit être installé séparément ; non fourni avec le serveur.

**`pg_search`** (de [ParadeDB](https://www.paradedb.com/)) remplace la pile standard `GIN` / `tsvector` par un index BM25 basé sur Tantivy. Cela vous donne un score BM25 (souvent supérieur à `ts_rank`), un rapprochement flou dans les requêtes FTS, la recherche facettée, et un indexage beaucoup plus rapide sur les tables volumineuses. C’est une voie de mise à niveau directe quand le FTS standard commence à montrer des limites de classement ou de performance.

```sql
-- pg_search : recherche plein texte BM25 avec rapprochement flou
CREATE INDEX posts_bm25_idx ON posts
  USING bm25 (id, title, body)
  WITH (key_field = 'id', text_fields = '{"title": {}, "body": {}}');

-- Requête avec score BM25 + rapprochement flou (capture « javascipt »)
SELECT id, title, paradedb.score(id) AS rank
FROM posts
WHERE posts @@@ paradedb.fuzzy_phrase(field => 'title', value => 'postgres performnce')
ORDER BY rank DESC
LIMIT 10;
```

**`pgvector`** ajoute le stockage de vecteurs denses et la recherche de similarité. C’est l’outil adéquat lorsque les utilisateurs décrivent ce qu’ils recherchent plutôt que de le nommer — recherche sémantique, RAG, recommandations de contenus liés, requêtes multilingues. Détaillé dans [Recherche vectorielle sémantique et stratégies hybrides](../semantic-vector-search-landscape).

## Tableau de décision

| Ce que vous cherchez | Recommandation |
|---|---|
| Articles, documentation, tickets en prose | FTS |
| Noms de personnes/entreprises avec fautes de frappe | `pg_trgm` |
| Autocomplétion, recherche par préfixe | `pg_trgm` |
| Codes courts, identifiants | `pg_trgm` |
| Messages de logs pour mots‑clés | FTS |
| Noms internationaux | `pg_trgm` + `unaccent` |
| Contenu volumineux, meilleur classement | `pg_search` (ParadeDB BM25) |
| Clés primaires, e‑mails exacts, IDs | Index B‑tree |
| Dates, intervalles, listes triées | Index B‑tree |
| Permissions, catégories, filtres | Clause WHERE classique |
| Questions, paraphrases, concepts | pgvector (voir l’article suivant) |

Lorsque le doute persiste : chaînes courtes avec variations orthographiques → trigrammes. Prose longue pour requêtes par mots‑clés → FTS. Identifiants structurés → index classiques. Requêtes conceptuelles ou en langage naturel → pgvector.

## Recherche hybride : deux signaux, un rang

Lorsqu’une requête comme `"withRetry timeout errors"` arrive dans une boîte de recherche, elle porte deux types d’intention : des noms de symboles exacts que l’utilisateur connaît (`withRetry`) et une description conceptuelle (`timeout errors`). Aucun primitive unique ne couvre les deux. Exécuter FTS et recherche vectorielle en parallèle — puis fusionner leurs listes classées avec Reciprocal Rank Fusion — le fait.

RRF attribue à chaque résultat le score `1 / (60 + rank)` dans chaque liste et additionne les scores entre les listes. La constante 60 atténue l’avantage des premiers rangs, de sorte qu’un résultat placé deuxième dans les deux listes peut l’emporter sur un résultat qui gagne une liste mais manque complètement dans l’autre. Crucialement, RRF ne moyenne jamais les scores bruts entre méthodes — le rang FTS et la distance cosinus sont des monnaies différentes et ne peuvent pas être combinés arithmétiquement.

<figcaption>Une requête se ramifie vers FTS et pgvector en parallèle. Chaque moteur produit sa propre liste classée. RRF note chaque document selon sa position dans chaque liste et additionne les scores — le résultat met en avant les documents sur lesquels les deux signaux s’accordent.</figcaption>

```sql
-- Recherche hybride : FTS + pgvector fusionnés avec RRF
WITH fts AS (
  SELECT id, ts_rank(search_vector, query) AS score,
         ROW_NUMBER() OVER (ORDER BY ts_rank(search_vector, query) DESC) AS rank
  FROM docs, to_tsquery('english', 'withRetry & timeout') query
  WHERE search_vector @@ query
  LIMIT 60
),
vec AS (
  SELECT id,
         ROW_NUMBER() OVER (ORDER BY embedding <=> $embedding) AS rank
  FROM docs
  ORDER BY embedding <=> $embedding
  LIMIT 60
)
SELECT COALESCE(fts.id, vec.id) AS id,
       (COALESCE(1.0 / (60 + fts.rank), 0) +
        COALESCE(1.0 / (60 + vec.rank), 0)) AS rrf_score
FROM fts FULL JOIN vec ON fts.id = vec.id
ORDER BY rrf_score DESC
LIMIT 10;
```

Le pool de candidats de 60 documents par branche (`LIMIT 60`) constitue un point de départ raisonnable. L’élargir si le rappel est insuffisant ; le réduire pour gagner en rapidité.

## Et après ?

La recherche textuelle de Postgres couvre un large spectre, mais elle a ses limites. Quand les utilisateurs décrivent ce qu’ils recherchent plutôt que de le nommer — « quelque chose pour m’aider à dormir pendant un vol », « articles sur le débogage de la confiance en tant que nouvel ingénieur » — la recherche lexicale et trigramme échouent toutes deux.

C’est le domaine des embeddings vectoriels, de la recherche sémantique et des architectures hybrides. Voir : [Recherche vectorielle sémantique et stratégies hybrides](../semantic-vector-search-landscape).
