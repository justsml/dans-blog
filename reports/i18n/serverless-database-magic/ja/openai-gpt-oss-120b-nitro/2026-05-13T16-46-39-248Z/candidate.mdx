## Not another Vector DB article

Here is the decision rule I wish I had used earlier:

<p class="inset">データがファイルから再構築可能で、ユーザーが主に読み取りだけを行う場合は、まずオブジェクトストレージデータベースを試してください。ユーザーが常に書き込みを行う場合は、実際のデータベースから始め、S3 にデータベースの振る舞いをさせようとするのはやめましょう。</p>

That is the useful line. Not "serverless is the future." Not "vector databases changed everything." Those sentences have already been printed on enough conference lanyards.

AI did change the shape of a lot of search problems. Suddenly small teams wanted semantic search, hybrid ranking, document chat, multimodal lookup, and analytics over files sitting in object storage. The old answer was "run Postgres with pgvector" or "operate OpenSearch/Elasticsearch" or "buy a managed search service." Those are still good answers when the workload deserves them.

But many workloads do not. They are read-heavy, rebuildable, and tolerant of a short delay between content changing and search catching up. Documentation. Catalog snapshots. Static exports. Internal knowledge bases. Local analytics. Prototype RAG systems. For those, a new class of tools has made the boring architecture unusually powerful: build an index, store it as files, serve it over HTTP.

Snapshot note: エコシステムは急速に変化しています。以下のスター数、機能ラベル、パフォーマンス数値は 2025 年 9 月時点のスナップショットであり、永続的なスコアボードではありません。方向性の参考として捉え、実際に本番移行を検討する際は必ず最新のドキュメントを確認してください。

## 名前は何であれデータベース

これらのサーバーレスかつ CDN 対応のデータストアは、レコード数が約 1,000〜1,000,000 件、または数 GB 程度の中規模ケースで有用です。従来のデータベースインフラは、価値以上に手間がかかることがあります。

- **Pagefind** (2022、約 4.5K ⭐)：純粋な静的アプローチ ― 一度ビルドすれば永遠に検索可能、バックエンド不要
- **Orama** (2023、約 8K ⭐)：ブラウザからサーバーレス関数まで、あらゆる環境で動作するユニバーサルソリューション
- **Chroma** (2022、約 14K ⭐)：AI ネイティブ、RAG アプリケーション向けに特化
- **LanceDB** (2023、約 4K ⭐)：ディスクベースアーキテクチャを持つエンタープライズ向けマルチモーダル機能
- **DuckDB-WASM** (2019、約 23K ⭐)：WebAssembly でブラウザ上で動作するフル SQL 分析データベース

共通のアプローチはシンプルです。耐久性のあるデータをファイルまたはオブジェクトストレージに保存し、ブラウザ、エッジ関数、ワーカー、あるいは軽量サービスからクエリを実行します。これが複雑さを完全に排除するわけではなく、ビルドパイプライン、インデックスの鮮度、キャッシュ無効化、クライアント側の機能へと複雑性が移ります。読み取りが支配的なシナリオでは、これは十分に妥当なトレードオフです。

### チェックボックスの戦い

| Feature | [Pagefind](../pagefind.app) | [Orama](../orama.com) | [Chroma](../www.trychroma.com/) | [LanceDB](../lancedb.com) | [DuckDB-WASM](../duckdb.org/docs/api/wasm) |
|---------|----------|--------|---------|----------|----------|
| **Full-Text Search** | ✅ Advanced stemming | ✅ BM25, 30 languages | ✅ SQLite FTS | ✅ Tantivy | ✅ Full SQL |
| **Vector Search** | ❌ | ✅ Cosine similarity | ✅ HNSW | ✅ IVF_PQ, HNSW, GPU | ⚠️ Extensions |
| **AI/RAG Integrations** | None | ✅ Built-in pipeline | ✅ LangChain, LlamaIndex | ✅ Advanced reranking | ⚠️ Manual setup |
| **Storage** | Static JSON/WASM | Memory + S3 plugins | Server-based* | S3-compatible Lance | WASM + S3/HTTP |
| **Write Support** | Build-time only | Full CRUD | Full CRUD | Full CRUD | Full SQL CRUD |
| **Performance** | Sub-100ms | 0.0001ms - 100ms | Sub-100ms | 3-5ms vector, 50ms FTS | 10ms-1s (complex SQL) |

*2025年9月時点のスナップショット: Chroma はサーバーランタイムが必要で、オブジェクトファイルツールが提供するような直接的な S3 オブジェクトストレージはサポートしていません（[issue #1736](../github.com/chroma-core/chroma/issues/1736)）。

### 実装例

構文の違いが示す本質的な分岐は次の通りです。ビルド時検索、インメモリ検索、ベクトルネイティブストレージ、マルチモーダルテーブル、ブラウザ上の SQL は、すべて AI デモに登場するからといって同一カテゴリの製品ではありません。

#### LanceDB を使ったエンタープライズ向けマルチモーダル

**OpenAI 埋め込みを自動生成する LanceDB テーブル作成コード:**
```typescript
import * as lancedb from "@lancedb/lancedb";
import "@lancedb/lancedb/embedding/openai";
import { LanceSchema, getRegistry } from "@lancedb/lancedb/embedding";
import { Utf8 } from "apache-arrow";

const db = await lancedb.connect("data/multimodal-db");
const func = getRegistry()
  .get("openai")
  ?.create({ model: "text-embedding-ada-002" });

// 自動埋め込み生成を含むスキーマ
const documentsSchema = LanceSchema({
  text: func.sourceField(new Utf8()),
  vector: func.vectorField(),
  category: new Utf8()
});

const table = await db.createEmptyTable("documents", documentsSchema);
await table.add([
  { text: "machine learning concepts", category: "research" },
  { text: "deep learning fundamentals", category: "research" }
]);
```

**LanceDB テーブルのクエリ例:**
```typescript
import * as lancedb from "@lancedb/lancedb";
import "@lancedb/lancedb/embedding/openai";
// URL パスへ「接続」する
const db = await lancedb.connect("data/multimodal-db");
const table = db.getTable("documents");

// SQL とベクトル検索の組み合わせ
const results = await table.search("machine learning concepts")
  .where("category = 'research'")
  .limit(10)
  .toArray();

console.log(results);
```


#### Orama でのユニバーサル検索
```typescript
import { create, insert, search } from '@orama/orama'

const db = create({
  schema: {
    title: 'string',
    content: 'string', 
    embedding: 'vector[1536]'
  }
})

await insert(db, { 
  title: 'Getting Started',
  content: 'Learn the basics',
  embedding: await generateEmbedding('Learn the basics')
})

const results = await search(db, { 
  term: 'basics',
  mode: 'hybrid' // テキスト検索とベクトル検索を併用
})
```

**DuckDB-WASM:**
```typescript
import * as duckdb from "https://cdn.jsdelivr.net/npm/@duckdb/duckdb-wasm@latest/dist/duckdb-browser.mjs";
const bundle = await duckdb.selectBundle(duckdb.getJsDelivrBundles());
const worker = new Worker(bundle.mainWorker);
const db = new duckdb.AsyncDuckDB(new duckdb.ConsoleLogger(), worker);
await db.instantiate(bundle.mainModule, bundle.pthreadWorker);

const conn = await db.connect();
await conn.query(`create table t as select * from (values (1,'hybrid search'),(2,'edge sql')) as v(id,txt);`);
// 任意の全文検索:
await conn.query(`install fts; load fts; select * from t where match_bm25(txt, 'hybrid');`);
```

#### Chroma での AI ネイティブ検索  
```typescript
import { ChromaClient } from "chromadb";

const client = new ChromaClient();
const collection = await client.createCollection({ name: "knowledge-base" });

await collection.add({
  documents: ["AI will transform software development"],
  metadatas: [{ source: "tech-blog", category: "AI" }],
  ids: ["doc1"]
});

const results = await collection.query({
  queryTexts: ["future of programming"],
  where: { category: "AI" },
  nResults: 5

```

## ユースケースガイド

**Pagefind を選ぶべきケース:**
- ドキュメント、ブログ、ナレッジベースの構築
- コンテンツ更新が週に一回以下
- 運用コストゼロで CDN キャッシュが完全に活用できる
- *例: 月に数回更新される 10K+ ページの社内ドキュメント*

**Orama を選ぶべきケース:**
- ダッシュボード、EC、動的アプリケーションの構築
- リアルタイム更新と 100ms 未満の応答が必要
- ブラウザからエッジ関数まで柔軟にデプロイしたい
- *例: 動的商品カタログを持つ SaaS*

**Chroma を選ぶべきケース:**
- RAG アプリケーションや AI ナレッジベースの構築
- LangChain / LlamaIndex との統合が必要
- セマンティック検索がコア機能
- *例: AI カスタマーサポートボット*

**LanceDB を選ぶべきケース:**
- マルチモーダルデータ（画像、音声、動画）を扱う
- エンタープライズ規模のパフォーマンスが必要
- 複雑な分析や再ランク付けが必要
- *例: セマンティック動画検索を提供するメディアプラットフォーム*

**DuckDB-WASM を選ぶべきケース:**
- ブラウザやエッジ関数でフル SQL が必要
- 分析ワークロードや複雑クエリを扱う
- CSV / Parquet を S3 から直接処理したい
- *例: アドホック SQL クエリを実行するビジネスインテリジェンスダッシュボード*

## 判断基準

実際に問われるのは「どのデータベースがベストか？」ではなく、**システムが吸収すべき変化の種類**です。

- **再構築可能なコンテンツ:** Pagefind、Orama スナップショット、Lance ファイル、Parquet 上の DuckDB。痛みが出るまで静的に保つ。
- **頻繁な書き込み:** Postgres、Chroma サーバー、マネージド検索サービス、またはキュー駆動のインデックスパイプライン。調整が必要です。
- **ユーザー固有の結果:** 本格的なバックエンドが必要。オブジェクトストレージは認可モデルではありません。
- **ファイル上の分析:** DuckDB は圧倒的に有用です。SQL に SQL を任せましょう。
- **マルチモーダルまたはベクトル中心の検索:** LanceDB と Chroma は実データでベンチマークすべきで、README のベンチマークではありません。

安価なパスがハッピーです。エッジケースがアーキテクチャを決定します。

## 大局的視点

これらのツールは、実用的な検索に必要な最小インフラを削減します。重要なのはその点です。2020 年頃は「セマンティック検索」といえば多数のサービスと大量の glue コード、そして半分が昼食を考えている会議でベクトルインデックスを説明する必要がありました。2025 年には、少人数のチームがファイル、埋め込み、週末の作業だけで同等のプロダクトをプロトタイプできます。

ただし、すべての検索ボックスが RAG システムになるわけではありません。最初のバージョンは、実証された証拠が出るまでプロダクションインフラを継承する必要がなくなった、ということです。

AWS も S3 に隣接したベクトル検索機能を進めており、これは有用なシグナルです。オブジェクトストレージはもはや古いファイルの納屋ではなく、クエリの表面となりつつあります。

## 実験を始める

1. **まず更新パターンを決める**: ビルド時、時間バッチ、ライブ書き込み、またはユーザー単位の結果。
2. **最小の正直なツールでプロトタイプ**: 静的 HTML なら Pagefind、分析ファイルなら DuckDB、軽量アプリ検索なら Orama、ベクトル中心なら LanceDB または Chroma。
3. **厄介な部分を測定**: インデックス作成時間、鮮度、バンドルサイズ、権限、コールドスタート後の最初のクエリ。
4. **痛みが実感できたら本格化**: ファイルベース版が限界を示したときに、マネージドデータベースの正当性が出てきます。

*実装例は私の [実用的な Pagefind ガイド][1] を参照してください。エッジネイティブデータベースのエコシステムは急速に拡大しています。*

> **免責事項:** 私は何年も Pagefind を使い、2025 年にコントリビュータになりました。小規模プロジェクトで Orama と Chroma を試し、より大規模な AI アプリケーションで LanceDB を検証中です。これらのプロジェクトと金銭的な関係はなく、データベースの進化に対する純粋な関心から執筆しています。
