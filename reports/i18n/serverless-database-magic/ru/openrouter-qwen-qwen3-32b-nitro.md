## Еще одна статья о векторных БД  

Вот правило принятия решений, которым я хотел бы руководствоваться раньше:  

<p class="inset">Если ваши данные можно пересобрать из файлов, а пользователи в основном их читают, начните с базы данных на основе хранилища объектов. Если пользователи активно пишут в нее, начните с настоящей базы данных и перестаньте заставлять S3 притворяться ею.</p>  

Эта линия действительно полезна. Не «серверлес — это будущее». Не «векторные базы данных изменили всё». Эти фразы уже вытатуированы на достаточно конференционных брелках.  

ИИ действительно изменил форму многих задач поиска. Внезапно маленькие команды захотели семантический поиск, гибкую ранжировку, чат с документами, мультимодальный поиск и аналитику по файлам, лежащим в хранилище объектов. Старый ответ был: «запустите Postgres с pgvector» или «воспользуйтесь OpenSearch/Elasticsearch» или «купите управляемую поисковую службу». Эти ответы остаются хорошими, когда нагрузка на них оправдана.  

Но многие нагрузки не требуют этого. Они ориентированы на чтение, допускают пересборку и терпят краткое время задержки между обновлением контента и синхронизацией поиска. Документация. Снапшоты каталогов. Статические экспортные файлы. Внутренние базы знаний. Локальная аналитика. Прототипы RAG. Для этих случаев новое поколение инструментов сделало скучную архитектуру неожиданно мощной: постройте индекс, сохраните его как файлы, подавайте через HTTP.

Снапшот: экосистема быстро развивается. Перечисленные ниже количество звёзд, метки функций и числа производительности — это снапшот сентября 2025 года, а не вечный рейтинг. Рассматривайте их как ориентир, а перед миграцией в production проверите актуальную документацию.

## База данных под любым именем

Эти серверно-независимые и CDN-совместимые хранилища полезны для задач среднего масштаба — примерно 1 000–1 000 000 записей или несколько гигабайт, где традиционная инфраструктура баз данных может быть более сложной, чем полезной:

- **Pagefind** (2022, ~4,5 тыс. ⭐): Чистый статический подход — один раз скомпилировать, искать вечно, без требований к бэкенду  
- **Orama** (2023, ~8 тыс. ⭐): Универсальное решение, работающее от браузеров до серверно-независимых функций  
- **Chroma** (2022, ~14 тыс. ⭐): Природно-ориентированное на ИИ, специально разработано для RAG-приложений  
- **LanceDB** (2023, ~4 тыс. ⭐): Корпоративные мультимодальные возможности с дисковой архитектурой  
- **DuckDB-WASM** (2019, ~23 тыс. ⭐): Полноценная SQL-аналитическая база данных, работающая в браузерах через WebAssembly  

Общее решение простое: храните устойчивые данные в файлах или объектном хранилище, а затем запрашивайте их из браузера, функции на краю, воркера или лёгкой службы. Это не устраняет сложность. Оно перемещает сложность в пайплайны сборки, свежесть индексов, инвалидацию кэша и возможности клиентов. Что является отличной заменой, когда чтения доминируют.

| **Векторный поиск** | ❌ | ✅ Косинусное сходство | ✅ HNSW | ✅ IVF_PQ, HNSW, GPU | ⚠️ Расширения |
| **Интеграции AI/RAG** | Нет | ✅ Встроенный пайплайн | ✅ LangChain, LlamaIndex | ✅ Расширенная переоценка | ⚠️ Ручная настройка |
| **Хранилище** | Статические JSON/WASM | Память + плагины S3 | Серверное* | Lance, совместимый с S3 | WASM + S3/HTTP |
| **Поддержка записи** | Только на этапе сборки | Полный CRUD | Полный CRUD | Полный CRUD | Полный SQL CRUD |
| **Производительность** | <100 мс | 0,0001-100 мс | <100 мс | 3-5 мс для векторов, 50 мс FTS | 10 мс-1 с (сложный SQL) |

#### Предприимчивый многомодальный поиск с LanceDB

**Код для создания таблицы LanceDB с автоматическими векторами OpenAI:**
```typescript
import * as lancedb from "@lancedb/lancedb";
import "@lancedb/lancedb/embedding/openai";
import { LanceSchema, getRegistry } from "@lancedb/lancedb/embedding";
import { Utf8 } from "apache-arrow";

const db = await lancedb.connect("data/multimodal-db");
const func = getRegistry()
  .get("openai")
  ?.create({ model: "text-embedding-ada-002" });

// Схема с автоматической генерацией векторов
const documentsSchema = LanceSchema({
  text: func.sourceField(new Utf8()),
  vector: func.vectorField(),
  category: new Utf8()
});

const table = await db.createEmptyTable("documents", documentsSchema);
await table.add([
  { text: "machine learning concepts", category: "research" },
  { text: "deep learning fundamentals", category: "research" }
]);
```

## Руководство по выбору

**Выбирайте Pagefind, когда:**
- Строите документацию, блоги или базы знаний
- Обновления контента происходят раз в неделю или реже
- Нужно нулевое операционное обслуживание и идеальное кэширование CDN
- *Пример: Корпоративные документы с 10K+ страниц, обновляющихся ежемесячно*

**Выбирайте Orama, когда:**
- Строите дашборды, e-commerce или динамические приложения
- Требуются обновления в реальном времени и производительность менее 100 мс
- Хотите гибкость развертывания от браузера до edge-функций
- *Пример: SaaS с динамическими каталогами товаров*

**Выбирайте Chroma, когда:**
- Строите RAG-приложения или AI-базы знаний
- Нужны интеграции LangChain/LlamaIndex
- Семантический поиск — ключевая функция
- *Пример: AI-бот поддержки клиентов*

**Выбирайте LanceDB, когда:**
- Работаете с многомодальными данными (изображения, аудио, видео)
- Требуется производительность предприятия на масштабных данных
- Необходимы сложные аналитика и ранжирование
- *Пример: Медийная платформа со семантическим поиском видео*

**Выбирайте DuckDB-WASM, когда:**
- Нужны полные SQL-возможности в браузере или edge-функциях
- Работаете с аналитическими нагрузками и сложными запросами
- Хотите обрабатывать CSV/Parquet-файлы напрямую из S3
- *Пример: Дашборд бизнес-аналитики с ad-hoc SQL-запросами*

## Правило принятия решений

Практический вопрос не в том, "какая СУБД лучше?"

Практический вопрос: какой вид изменений должна выдержать система?

- **Перестраиваемый контент:** Pagefind, снимки Orama, файлы Lance, DuckDB через Parquet. Держите статичным, пока не будет болеть.
- **Частые записи:** Postgres, сервер Chroma, управляемая поисковая служба или очередь индексации. Вам нужна координация, а не вайбы.
- **Результаты, специфичные для пользователя:** используйте настоящий бэкенд. Хранилище объектов не является моделью авторизации.
- **Аналитика над файлами:** DuckDB невероятно полезен. Пусть SQL делает SQL-вещи.
- **Многомодальный или векторно-ориентированный поиск:** LanceDB и Chroma стоит тестировать на реальных данных, а не на бенчмарках из README.

Дешевый путь — это счастливый путь. Краевые случаи определяют архитектуру.

## Общий контекст

Эти инструменты снижают минимальную жизнеспособную инфраструктуру для полезного поиска. Это важно. В 2020 году "семантический поиск" часто подразумевал кучу сервисов, много склейки и объяснение векторных индексов на встрече, где половина аудитории хотела обед. В 2025 году небольшая команда может прототипировать ту же идею продукта с файлами, эмбеддингами и выходным днём.

Это не значит, что каждый поисковый блок должен стать RAG-системой. Это значит, что первая версия больше не должна наследовать производственную инфраструктуру до того, как будет доказана её жизнеспособность.

Даже AWS двигается в этом направлении, внедряя векторный поиск, связанный с S3 — полезный сигнал: хранилище объектов больше не просто кладбище старых файлов. Оно становится поверхностью для запросов.

## Начните экспериментировать

1. **Сначала определите паттерн обновления:** сборка, ежечасная партия, живые записи или результаты для пользователя.
2. **Прототипируйте с минимальным честным инструментом:** Pagefind для статичного HTML, DuckDB для аналитических файлов, Orama для лёгкого поиска приложений, LanceDB или Chroma для векторно-интенсивных задач.
3. **Измерьте неприятные части:** время индексации, свежесть, размер пакета, права доступа и первый запрос после холодного старта.
4. **Переводите только при реальном боле:** управляемая СУБД проще обосновать после того, как файловая версия покажет, где именно она "гнёт".

*Ознакомьтесь с моим [практическим руководством по Pagefind][1] для реализации, или изучите растущую экосистему edge-ориентированных баз данных, меняющих данные на масштабе.*

> **Дисклеймер:** Я использовал Pagefind много лет и стал его вкладчиком в 2025. Экспериментировал с Orama и Chroma для небольших проектов и сейчас исследую LanceDB для крупных AI-приложений. Нет финансовых связей с проектами — просто живой интерес к эволюции баз данных.
