## Not another Vector DB article

Here is the decision rule I wish I had used earlier:

<p class="inset">Если ваши данные можно воссоздать из файлов, а пользователи в основном только читают их, сначала попробуйте базу данных на объектном хранилище. Если пользователи постоянно пишут, начните с настоящей СУБД и перестаньте заставлять S3 притворяться ею.</p>

That is the useful line. Not "serverless is the future." Not "vector databases changed everything." Those sentences have already been printed on enough conference lanyards.

AI did change the shape of a lot of search problems. Suddenly small teams wanted semantic search, hybrid ranking, document chat, multimodal lookup, and analytics over files sitting in object storage. The old answer was "run Postgres with pgvector" or "operate OpenSearch/Elasticsearch" or "buy a managed search service." Those are still good answers when the workload deserves them.

But many workloads do not. They are read-heavy, rebuildable, and tolerant of a short delay between content changing and search catching up. Documentation. Catalog snapshots. Static exports. Internal knowledge bases. Local analytics. Prototype RAG systems. For those, a new class of tools has made the boring architecture unusually powerful: build an index, store it as files, serve it over HTTP.

Снимок: экосистема развивается стремительно. Количество звёзд, метки функций и показатели производительности ниже отражают состояние на сентябрь 2025 года, а не вечный рейтинг. Рассматривайте их как ориентир, а перед тем как делать ставку на миграцию в продакшн, проверьте актуальную документацию.

## База данных под любым другим именем

Эти безсерверные хранилища с поддержкой CDN полезны для средних нагрузок — примерно от 1 000 до 1 000 000 записей или нескольких гигабайт, когда традиционная инфраструктура баз данных приносит больше бюрократии, чем пользы:

- **Pagefind** (2022, ~4,5 K ⭐): полностью статический подход — собрать один раз, искать вечно, без бэкенда
- **Orama** (2023, ~8 K ⭐): универсальное решение, работающее в браузерах и безсерверных функциях
- **Chroma** (2022, ~14 K ⭐): AI‑ориентированное, специально построенное для RAG‑приложений
- **LanceDB** (2023, ~4 K ⭐): корпоративные мультимодальные возможности с дисковой архитектурой
- **DuckDB-WASM** (2019, ~23 K ⭐): полнофункциональная аналитическая СУБД SQL, работающая в браузерах через WebAssembly

Общая идея проста: хранить неизменные данные в файлах или объектном хранилище, а затем запрашивать их из браузера, edge‑функции, воркера или лёгкого сервиса. Это не устраняет сложность, а переносит её в конвейеры сборки, актуальность индексов, инвалидацию кэша и возможности клиента. И это вполне приемлемый компромисс, когда чтения доминируют.

#### Enterprise‑grade мультимодальный поиск с LanceDB

**Код для создания таблицы LanceDB с автоматическим встраиванием OpenAI:**
```typescript
import * as lancedb from "@lancedb/lancedb";
import "@lancedb/lancedb/embedding/openai";
import { LanceSchema, getRegistry } from "@lancedb/lancedb/embedding";
import { Utf8 } from "apache-arrow";

const db = await lancedb.connect("data/multimodal-db");
const func = getRegistry()
  .get("openai")
  ?.create({ model: "text-embedding-ada-002" });

// Схема с автоматическим генерированием эмбеддингов
const documentsSchema = LanceSchema({
  text: func.sourceField(new Utf8()),
  vector: func.vectorField(),
  category: new Utf8()
});

const table = await db.createEmptyTable("documents", documentsSchema);
await table.add([
  { text: "machine learning concepts", category: "research" },
  { text: "deep learning fundamentals", category: "research" }
]);
```

**Пример запроса к таблице LanceDB:**
```typescript
import * as lancedb from "@lancedb/lancedb";
import "@lancedb/lancedb/embedding/openai";
// «Подключаемся» к пути URL
const db = await lancedb.connect("data/multimodal-db");
const table = db.getTable("documents");

// Комбинация SQL + векторного поиска
const results = await table.search("machine learning concepts")
  .where("category = 'research'")
  .limit(10)
  .toArray();

console.log(results);
```


#### Универсальный поиск с Orama
```typescript
import { create, insert, search } from '@orama/orama'

const db = create({
  schema: {
    title: 'string',
    content: 'string', 
    embedding: 'vector[1536]'
  }
})

await insert(db, { 
  title: 'Getting Started',
  content: 'Learn the basics',
  embedding: await generateEmbedding('Learn the basics')
})

const results = await search(db, { 
  term: 'basics',
  mode: 'hybrid' // Комбинирует текстовый и векторный поиск
})
```

**DuckDB‑WASM:**
```typescript
import * as duckdb from "https://cdn.jsdelivr.net/npm/@duckdb/duckdb-wasm@latest/dist/duckdb-browser.mjs";
const bundle = await duckdb.selectBundle(duckdb.getJsDelivrBundles());
const worker = new Worker(bundle.mainWorker);
const db = new duckdb.AsyncDuckDB(new duckdb.ConsoleLogger(), worker);
await db.instantiate(bundle.mainModule, bundle.pthreadWorker);

const conn = await db.connect();
await conn.query(`create table t as select * from (values (1,'hybrid search'),(2,'edge sql')) as v(id,txt);`);
// Опциональный полнотекстовый поиск:
await conn.query(`install fts; load fts; select * from t where match_bm25(txt, 'hybrid');`);
```

const client = new ChromaClient();

## Руководство по случаям использования

**Выбирайте Pagefind, когда:**
- Нужно построить документацию, блоги или базы знаний
- Обновления контента раз в неделю или реже
- Требуется нулевой операционный overhead и идеальное кэширование CDN
- *Пример: корпоративные документы с более 10 000 страниц, обновляемые ежемесячно*

**Выбирайте Orama, когда:**
- Создаёте дашборды, e‑commerce или динамические приложения
- Нужны обновления в реальном времени и отклик < 100 мс
- Требуется гибкость развертывания от браузеров до edge‑функций
- *Пример: SaaS с динамическим каталогом товаров*

**Выбирайте Chroma, когда:**
- Строите RAG‑приложения или AI‑базы знаний
- Нужна интеграция с LangChain/LlamaIndex
- Семантический поиск — ключевая функция
- *Пример: чат‑бот поддержки клиентов, работающий на ИИ*

**Выбирайте LanceDB, когда:**
- Работаете с мультимодальными данными (изображения, аудио, видео)
- Требуется корпоративная производительность на огромных объёмах
- Нужна сложная аналитика и переранжирование
- *Пример: медиа‑платформа с семантическим поиском по видео*

**Выбирайте DuckDB‑WASM, когда:**
- Нужны полные возможности SQL в браузере или edge‑функциях
- Обрабатываете аналитические нагрузки и сложные запросы
- Требуется работа с CSV/Parquet напрямую из S3
- *Пример: BI‑дашборд с ad‑hoc SQL‑запросами*

## Правило выбора

Практический вопрос не «какая база данных лучшая?»

Практический вопрос: какой тип изменения должна поглотить система?

- **Контент, который можно перестраивать:** Pagefind, снапшоты Orama, файлы Lance, DuckDB над Parquet. Держите его статическим, пока это не станет проблемой.
- **Частые записи:** Postgres, сервер Chroma, управляемый поисковый сервис или конвейер индексации на очереди. Здесь нужна координация, а не «вайбы».
- **Результаты, зависящие от пользователя:** используйте настоящий бекенд. Объектное хранилище не является моделью авторизации.
- **Аналитика над файлами:** DuckDB удивительно полезен. Позвольте SQL делать то, что умеет SQL.
- **Мультимодальный или векторный поиск:** LanceDB и Chroma стоит тестировать на ваших реальных данных, а не на README‑бенчмарке.

Базовый путь дешевый. Крайние случаи определяют архитектуру.

## Более широкая картина

Эти инструменты снижают минимальный жизнеспособный набор инфраструктуры для полезного поиска. Это имеет значение. В 2020‑м году «семантический поиск» часто означал кучу сервисов, массу клея и кого‑то, объясняющего векторные индексы на встрече, где половина аудитории уже мечтала о обеде. В 2025‑м небольшая команда может прототипировать ту же идею продукта с файлами, эмбеддингами и выходными выходными в выходные.

Это не значит, что каждый поисковый блок должен превращаться в RAG‑систему. Это значит, что первая версия больше не обязана наследовать продакшн‑инфраструктуру до тех пор, пока не появятся продакшн‑данные.

Даже AWS движется в этом направлении с векторным поиском, прилегающим к S3, что служит полезным сигналом: объектное хранилище уже не просто чердак для старых файлов. Оно становится поверхностью запросов.

## Начинаем экспериментировать

1. **Сначала определите паттерн обновления**: сборка, часовой батч, живые записи или результаты per‑user.
2. **Прототипируйте самым небольшим честным инструментом**: Pagefind для статического HTML, DuckDB для аналитических файлов, Orama для лёгкого поиска в приложении, LanceDB или Chroma для векторных задач.
3. **Измерьте «некрасивую» часть**: время индексации, свежесть, размер бандла, права доступа и первый запрос после холодного старта.
4. **Продвигайте только когда боль реальна**: управляемую БД проще оправдать после того, как файловая версия покажет, где именно она гнётся.
