## एक और वेक्टर डेटाबेस लेख नहीं

यहां वह निर्णय नियम है जिसे मैं जल्दी से अपनाना चाहता था:

<p class="inset">अगर आपके डेटा को फ़ाइलों से फिर से बनाया जा सकता है और उपयोगकर्ता इसे अधिकांश रूप से पढ़ रहे हैं, तो पहले एक ऑब्जेक्ट-स्टोरेज डेटाबेस की कोशिश करें। अगर उपयोगकर्ता इसे पूरे दिन लिख रहे हैं, तो एक वास्तविक डेटाबेस से शुरू करें और S3 को एक डेटाबेस बनाने की कोशिश बंद कर दें।</p>

यही उपयोगी रेखा है। "सर्वरलेस भविष्य है" नहीं। "वेक्टर डेटाबेस सब कुछ बदल गया" नहीं। ये वाक्य पर्याप्त कार्यशाला बैजेट पर छप चुके हैं।

एआई ने खोज के कई समस्याओं के आकार को बदल दिया है। अचानक छोटी टीमें अर्थात्मक खोज, हाइब्रिड रैंकिंग, दस्तावेज़ चैट, बहु-माध्यमीय खोज और ऑब्जेक्ट स्टोरेज में फ़ाइलों पर विश्लेषण चाहती हैं। पुराना उत्तर था "पोस्टग्रेस के साथ pgvector चलाएं" या "ओपनसर्च/एलास्टिकसर्च चलाएं" या "एक प्रबंधित खोज सेवा खरीदें"। जब भार के लायक होता है तो ये अभी भी अच्छे उत्तर हैं।

लेकिन कई भार इतने नहीं हैं। वे पढ़ने वाले, फिर से बनाए जाने वाले हैं और डेटा परिवर्तन और खोज के बीच छोटे विलंब के प्रति सहिष्णु हैं। दस्तावेज़। कैटलॉग अंकन। स्थैतिक निर्यात। आंतरिक ज्ञान डेटाबेस। स्थानीय विश्लेषण। प्रोटोटाइप RAG प्रणालियां। उनके लिए एक नई उपकरण श्रेणी ने बोरिंग आर्किटेक्चर को असामान्य रूप से शक्तिशाली बना दिया है: एक इंडेक्स बनाएं, इसे फ़ाइलों के रूप में संग्रहीत करें, इसे HTTP के माध्यम से सर्व करें।

अंकन नोट: परिवेश तेजी से बदल रहा है। नीचे दिए गए स्टार गिनती, विशेषता लेबल और प्रदरम्भन संख्या सितंबर 2025 के अंकन हैं, एक समयरहित स्कोरबोर्ड नहीं। उन्हें दिशा निर्देश के रूप में लें, फिर किसी एक कोशिका पर उत्पादन माइग्रेशन बेट करने से पहले वर्तमान दस्तावेज़ जांच लें।

## एक डेटाबेस कोई भी अन्य नाम से

ये सर्वरलेस और सीडीएन-क्षमता वाले डेटा संग्रह मध्यम आकार के मामलों के लिए उपयोगी हैं, लगभग 1,000 से 1,000,000 रिकॉर्ड या कुछ जीबी, जहां पारंपरिक डेटाबेस बुनियादी ढांचा मूल्य की तुलना में अधिक शिष्टता हो सकती है:

- **पेजफाइंड** (2022, ~4.5K ⭐): शुद्ध स्थैतिक दृष्टिकोण - एक बार कम्पाइल करें, हमेशा खोजें, बैकएंड की आवश्यकता शून्य  
- **ओरामा** (2023, ~8K ⭐): सर्वत्र चलने वाला सार्वभौमिक समाधान - ब्राउज़र से लेकर सर्वरलेस फ़ंक्शन तक  
- **क्रोमा** (2022, ~14K ⭐): एआई-मूल रूप से, RAG एप्लिकेशन के लिए डिज़ाइन किया गया  
- **लैंसडीबी** (2023, ~4K ⭐): डिस्क-आधारित आर्किटेक्चर के साथ उद्यमी बहु-माध्यमीय क्षमता  
- **डकडीबी-वॉसम** (2019, ~23K ⭐): वेबएसेम्बली के माध्यम से ब्राउज़र में चलने वाला पूर्ण SQL विश्लेषण डेटाबेस

सामान्य दृष्टिकोण सरल है: टिकाऊ डेटा को फ़ाइलों या ऑब्जेक्ट स्टोरेज में रखें, फिर ब्राउज़र, एज फ़ंक्शन, वर्कर या हल्के सेवा से इसे प्रश्न करें। यह जटिलता को खत्म नहीं करता। यह बिल्ड पाइपलाइन, इंडेक्स ताजगी, कैश अमान्यता और क्लाइंट क्षमताओं में जटिलता को शिफ्ट करता है। जो पढ़ने के लिए प्रभावी है।

### कार्यान्वयन उदाहरण

सिंटैक्स के अंतर वास्तविक विभाजन का खुलासा करते हैं: बिल्ड-टाइम खोज, मेमोरी-में खोज, वेक्टर-मूल संग्रहण, बहु-माध्यमीय तालिकाएँ और ब्राउज़र SQL एक ही उत्पाद श्रेणी नहीं हैं केवल इसलिए कि वे सभी AI डेमो में दिखाई देते हैं।

#### स्थैतिक साइट खोज पेजफाइंड के साथ
```html
<link href="/pagefind/pagefind-ui.css" rel="stylesheet">
```

#### लैंसडीबी के साथ उद्यमी-ग्रेड बहु-माध्यमीय
**लैंसडीबी तालिका बनाने के लिए कोड जिसमें स्वचालित ओपनएआई एम्बेडिंग है:**
```typescript
import * as lancedb from "@lancedb/lancedb";
import "@lancedb/lancedb/embedding/openai";
import { LanceSchema, getRegistry } from "@lancedb/lancedb/embedding";
import { Utf8 } from "apache-arrow";

const db = await lancedb.connect("data/multimodal-db");
const func = getRegistry()
  .get("openai")
  ?.create({ model: "text-embedding-ada-002" });

// स्वचालित एम्बेडिंग उत्पादन के साथ योजना
const documentsSchema = LanceSchema({
  text: func.sourceField(new Utf8()),
  vector: func.vectorField(),
  category: new Utf8()
});

const table = await db.createEmptyTable("documents", documentsSchema);
await table.add([
  { text: "machine learning concepts", category: "research" },
  { text: "deep learning fundamentals", category: "research" }
]);
```

**लैंसडीबी तालिका के प्रश्न का उदाहरण:**
```typescript
import * as lancedb from "@lancedb/lancedb";
import "@lancedb/lancedb/embedding/openai";
// "कनेक्ट" करें एक URL पथ के साथ
const db = await lancedb.connect("data/multimodal-db");
const table = db.getTable("documents");

// SQL + वेक्टर खोज संयोजन
const results = await table.search("machine learning concepts")
  .where("category = 'research'")
  .limit(10)
  .toArray();

console.log(results);
```

#### ओरामा के साथ सार्वभौमिक खोज
```typescript
import { create, insert, search } from '@orama/orama'

const db = create({
  schema: {
    title: 'string',
    content: 'string', 
    embedding: 'vector[1536]'
  }
})

await insert(db, { 
  title: 'Getting Started',
  content: 'Learn the basics',
  embedding: await generateEmbedding('Learn the basics')
})

const results = await search(db, { 
  term: 'basics',
  mode: 'hybrid' // पाठ + वेक्टर खोज का संयोजन
})
```

**डकडीबी-वॉसम:**
```typescript
import * as duckdb from "https://cdn.jsdelivr.net/npm/@duckdb/duckdb-wasm@latest/dist/duckdb-browser.mjs";
const bundle = await duckdb.selectBundle(duckdb.getJsDelivrBundles());
const worker = new Worker(bundle.mainWorker);
const db = new duckdb.AsyncDuckDB(new duckdb.ConsoleLogger(), worker);
await db.instantiate(bundle.mainModule, bundle.pthreadWorker);

const conn = await db.connect();
await conn.query(`create table t as select * from (values (1,'hybrid search'),(2,'edge sql')) as v(id,txt);`);
// वैकल्पिक पूर्ण-पाठ:
await conn.query(`install fts; load fts; select * from t where match_bm25(txt, 'hybrid');`);
```

## उपयोग केस गाइड

**पेजफाइंड चुनें जब:**
- दस्तावेज़, ब्लॉग, या ज्ञान डेटाबेस बना रहे हों
- सामग्री हफ्ते में या उससे कम अपडेट हो रही हो
- शून्य संचालन अतिरिक्त और परफेक्ट CDN कैशिंग की आवश्यकता हो
- *उदाहरण: 10K+ पृष्ठों के कंपनी दस्तावेज़ जो मासिक अपडेट हो रहे हों*

**ओरामा चुनें जब:**
- डैशबोर्ड, ई-कॉमर्स, या गतिशील एप्लिकेशन बना रहे हों
- रियल-टाइम अपडेट और 100 मिलीसेकंड से कम प्रदरस्ता की आवश्यकता हो
- ब्राउज़र से एज फ़ंक्शन तक तैयारी की लचीलापन की आवश्यकता हो
- *उदाहरण: गतिशील उत्पाद कैटलॉग के साथ SaaS*

**क्रोमा चुनें जब:**
- RAG एप्लिकेशन या AI ज्ञान डेटाबेस बना रहे हों
- LangChain/LlamaIndex एकीकरण की आवश्यकता हो
- सेमेंटिक खोज मुख्य कार्य हो
- *उदाहरण: AI ग्राहक समर्थन बॉट*

**लैंसडीबी चुनें जब:**
- बहु-माध्यमीय डेटा (चित्र, ऑडियो, वीडियो) के साथ काम कर रहे हों
- दूरगामी पैमाने पर उद्यमी प्रदरस्ता की आवश्यकता हो
- जटिल विश्लेषण और फिर रैंकिंग की आवश्यकता हो
- *उदाहरण: सेमेंटिक वीडियो खोज के साथ मीडिया प्लेटफॉर्म*

**डकडीबी-वॉसम चुनें जब:**
- ब्राउज़र या एज फ़ंक्शन में पूर्ण SQL क्षमता की आवश्यकता हो
- विश्लेषणात्मक कार्य भार और जटिल प्रश्नों के साथ काम कर रहे हों
- S3 से सीधे CSV/Parquet फ़ाइलों का उपयोग करना चाहते हों
- *उदाहरण: एड-हॉक SQL प्रश्नों के साथ व्यवसाय बुद्धिमान डैशबोर्ड*

## निर्णय नियम

व्यावहारिक प्रश्न यह नहीं है कि "कौन सा डेटाबेस सर्वोत्तम है?"

व्यावहारिक प्रश्न यह है: प्रणाली को किस प्रकार का परिवर्तन सहना होगा?

- **पुनर्निर्मान योग्य सामग्री:** पेजफाइंड, ओरामा स्नैपशॉट, लैंस फ़ाइलें, पैरक्वेट पर डकडीबी। इसे स्थैतिक रखें जब तक कि यह पीड़ा न हो।
- **अक्सर लिखना:** पोस्टग्रेस, क्रोमा सर्वर, प्रबंधित खोज सेवा, या एक कतार-पिछड़े इंडेक्सिंग पाइपलाइन। आपको समन्वय की आवश्यकता है, न कि वाइब्स।
- **उपयोगकर्ता-विशिष्ट परिणाम:** एक वास्तविक बैकएंड का उपयोग करें। ऑब्जेक्ट स्टोरेज एक अधिकृतन नमूना नहीं है।
- **फ़ाइलों पर विश्लेषण:** डकडीबी अविश्वसनीय रूप से उपयोगी है। SQL को SQL चीजें करने दें।
- **बहु-माध्यमीय या वेक्टर-भारी खोज:** लैंसडीबी और क्रोमा अपन डेटा के खिलाफ परीक्षण करने लायक हैं, एक रीडमी बैचमार्क के खिलाफ नहीं।

सुखद मार्ग सस्ता है। किनारा मामले आर्किटेक्चर का निर्णय लेते हैं।

## बड़ी तस्वीर

ये उपकरण उपयोगी खोज के लिए न्यूनतम व्यवहार्य बुनियादी ढांचे को कम करते हैं। यह महत्वपूर्ण है। 2020 में, "सेमेंटिक खोज" अक्सर एक सेवा के ढेर, बहुत सारे जूड़ कोड़े और किसी के वेक्टर इंडेक्स की व्याख्या करते हुए एक बैठक का अर्थ था जहां कमरे के आधे लोग लंच चाहते थे। 2025 में, एक छोटी टीम फ़ाइलों, एम्बेडिंग्स और एक वीकेंड के साथ एक ही उत्पाद विचार का प्रोटोटाइप बना सकती है।

यह अर्थ नहीं है कि हर खोज बॉक्स एक RAG प्रणाली बन जाए। यह अर्थ है कि पहला संस्करण उत्पादन बुनियादी ढांचा प्राप्त करने के बिना उत्पादन साक्ष्य दिखाने से पहले उत्पादन बुनियादी ढांचा प्राप्त करने के बिना बनाया जा सकता है।

हकीकत में, एएवीएस भी इस दिशा में आगे बढ़ रहा है एस3-संलग्न वेक्टर खोज कार्य के साथ, जो एक उपयोगी संकेत है: ऑब्जेक्ट स्टोरेज अब बस पुरानी फ़ाइलों के लिए एटिक कमरा नहीं है। यह एक प्रश्न सतह बन रहा है।

## प्रयोग करना शुरू करें

1. **पहले अपडेट पैटर्न चुनें**: बिल्ड-टाइम, घंटे में बैच, लाइव लिखना, या उपयोगकर्ता-विशिष्ट परिणाम।
2. **सबसे छोटे सच्चे उपकरण के साथ प्रोटोटाइप करें**: स्थैतिक एचटीएमएल के लिए पेजफाइंड, विश्लेषणात्मक फ़ाइलों के लिए डकडीबी, लाइटवेट एप्लिकेशन खोज के लिए ओरामा, वेक्टर-भारी कार्य के लिए लैंसडीबी या क्रोमा।
3. **गंदा हिस्सा मापें**: इंडेक्सिंग समय, ताजगी, बंडल आकार, अनुमति, और ठंडा स्टार्ट के बाद पहला प्रश्न।
4. **केवल तब प्रमोट करें जब दर्द वास्तविक हो**: एक प्रबंधित डेटाबेस फ़ाइल-आधारित संस्करण के बाद आसानी से बचाया जा सकता है जो ठीक उसी जगह मुड़ता है।

*मेरे [प्रायोगिक पेजफाइंड गाइड][1] को अमलीकरण के लिए देखें, या बड़े पैमाने पर डेटा को बदल रहे एज-मूल डेटाबेसों की बढ़ती अर्थव्यवस्था का अन्वेषण करें।*
