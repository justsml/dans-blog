> API कुंजियों और टोकनों को सुरक्षित करना **अत्यंत महत्वपूर्ण** है!

एक ही गलती से आपका सर्वर और डेटा हैकर्स के हाथों में जा सकता है!

कोई भी विशेष टोकन को छुपाना चाहिए या नहीं, यह तय करना इतना कठिन नहीं होना चाहिए – आधिकारिक दस्तावेज़ीकरण के आधार पर भी!

यह अक्सर उन संबंधित शब्दों के मिश्रण से और बिगड़ जाता है जिनसे आप मिलेंगे: _टोकन_, _कुंजी_, _प्रमाणपत्र_, _सीक्रेट_, _प्राइवेट_, और _पब्लिक_।

इसे `secret` और `non-secret` के बीच के अंतर के रूप में फिर से परिभाषित करते हैं।

* 🔒 [`Secret keys`](#-secret-keys) को हमेशा छिपा कर रखना चाहिए। सामान्यतः इन्हें आपके प्राइवेट सर्वर (या सेवा — जैसे heroku, netlify या travis-ci) से बाहर नहीं जाना चाहिए।
* 🌍 [`Non-secret keys`](#-non-secret-keys) उन स्ट्रिंग्स को दर्शाता है जिन्हें स्वतंत्र रूप से साझा किया जा सकता है और ब्राउज़र अनुरोधों में शामिल किया जा सकता है।

## 🔒 `Secret keys`

** ‼️ Important:** `Secret keys` **MUST** be ignored by Git _AND_ omitted in all browser code. [_How to use dotenv_](../#-how-to-handle-secrets-safely)

<br />

_आप यह कैसे पहचानते हैं कि आप किसी `Secret key` के साथ काम कर रहे हैं?_

<br />

**👍 Rule-of-thumb:** जो सर्वर `CORS errors` लौटाते हैं, उनके पास ब्राउज़र समर्थन नहीं होता। यह स्पष्ट संकेत देता है कि आपको सेवा को **MUST** प्रॉक्सी करना चाहिए, जैसे वह `secret` हो।

**👍 Rule-of-thumb:** महँगी सेवाओं को (लगभग) हमेशा प्रॉक्सी या छिपाया जाना चाहिए।

**👍 Rule-of-thumb:** यदि आप कोई लिखने‑ऑपरेशन करते हैं (**फ़ाइल अपलोड, DB पंक्ति डालना**), तो आप संभवतः `secret keys` के साथ काम कर रहे हैं।

<br />

**_उपयोग‑केस और विशेषताएँ:_** `Secret` keys

- दीर्घकालिक प्राधिकरण (प्रमाणपत्र, एक्सेस टोकन, JSON Web Tokens)
- अल्पकालिक प्राधिकरण (OAuth टोकन, सत्र स्टोर)
- भुगतान‑या महँगी सेवाओं तक पहुँच (प्रमाणीकरण, जियोकोडिंग, फ़ाइल स्टोरेज, आदि)
- सार्वजनिक/निजी जोड़ी का निजी भाग (RECAPTCHA, Stripe, Auth0)
- सेवा प्रमाणपत्र (ई‑मेल/SMTP, LDAP/डायरेक्टरी सर्विसेज)
- डेटा एन्क्रिप्शन और इंटेग्रिटी जाँच

### Checklist: Handling Secrets Safely

#### Quick Overview

Complete the following steps to **eliminate secrets from your code:**

- [ ] हार्ड‑कोडेड कुंजियों को पर्यावरण वेरिएबल्स से बदलें। उदाहरण : `process.env.API_SECRET`
- [ ] एक लाइब्रेरी जैसे [`dotenv`](https://github.com/motdotla/dotenv#dotenv) के साथ `.env` फ़ाइल का उपयोग करें। अपने पहले से हार्ड‑कोडेड सीक्रेट्स को `.env` फ़ाइल में जोड़ें।
- [ ] अपनी `.gitignore` फ़ाइल में एक `.env` लाइन जोड़ें!

> **DON'T** डिप्लॉय किए गए सर्वरों पर `.env` फ़ाइल बनाएं। अपने होस्टिंग सेवा (जैसे [Heroku](https://devcenter.heroku.com/articles/config-vars), Netlify, AWS EC2) द्वारा प्रदान किए गए पर्यावरण वेरिएबल प्रबंधन टूल का उपयोग करें: उदाहरण — **डैशबोर्ड या कमांड लाइन**।

## 🌍 `Non-secret keys`

**👍 Rule-of-thumb:** जब भी कोई कुंजी कोड या इनलाइन में ब्राउज़र को भेजी जानी हो (जैसे `<script src="https://my-api/?apiKey=123-abc-456">` टैग के माध्यम से), **यह निश्चित रूप से `non-secret` है**। एक सामान्य उदाहरण Google Maps है।

<br />

**_Use cases & features:_** `Non-secret` keys

- अल्पकालिक पहुँच (उपयोगकर्ता सत्र ID, JSON Web Tokens)
- एप/डेवलपर द्वारा API पहुँच को सीमित करना (ऑथ, जियोकोडिंग, आदि)
- सार्वजनिक/निजी जोड़ी का सार्वजनिक भाग (RECAPTCHA, Stripe, Auth0)
- एनालिटिक्स ID

#### ✅ Handling Non-secrets:

> **यह सुरक्षित है कि non-secret (public) कुंजियों को हार्ड‑कोड किया जाए!**

लंबी अवधि में इसे आसान बनाने के लिए अपने ऐप के लिए एक साझा `config.js` रखें।

**उदाहरण:**

```js
// config.js
module.exports = {
  googleMapsKey: '123-abc'
};
```

```js
// load-map.js
const config = require('./config.js');
const key = config.googleMapsKey;
const src = `//maps.googleapis.com/maps/api/js?key=${key}`;
// ...
```
