## Когда защищать ваши токены?

> Защита ключей API и токенов **критически важна**!

Одна ошибка может привести к потере контроля над вашим сервером и данными из-за хакеров!

Не должно быть настолько сложно определить, нужно ли скрывать конкретный токен — даже на основе официальной документации!

Всё становится ещё сложнее из-за множества связанных терминов, с которыми вы столкнётесь: _токены_, _ключи_, _учётные данные_, _секреты_, _приватные_ и _публичные_.  

Давайте переформулируем это как `секретные` и `несекретные`.  

* 🔒 [`Секретные ключи`](#-secret-keys) **должны оставаться скрытыми**. В общем случае они **никогда не должны покидать ваш приватный сервер** (или сервис — например, Heroku, Netlify или Travis-CI).  
* 🌍 [`Несекретные ключи`](#-non-secret-keys) — это строки, которые можно свободно делиться и использовать в запросах браузера.

## 🔒 `Секретные ключи`

** ‼️ Важно:** `Секретные ключи` **должны** игнорироваться Git и **никогда не включаться** в клиентский код. [Как использовать dotenv](#-how-to-handle-secrets-safely)

<br />

_Как понять, что вы работаете с `Секретным ключом`?_

<br />

**👍 Правило-памятка:** серверы, возвращающие `CORS ошибки`, не поддерживают браузер. Это ясно указывает, что вы **должны** использовать прокси-сервер, рассматривая его как `секретный`.

**👍 Правило-памятка:** дорогостоящие сервисы (почти) всегда должны быть проксированы или скрыты.

**👍 Правило-памятка:** если вы выполняете операцию записи (**загрузка файла, вставка строки в БД**), вы, возможно, работаете с `секретными ключами`.

<br />

**_Сценарии использования и функции:_** `Секретные` ключи

- Долгосрочная авторизация (учётные данные, токены доступа, JSON Web Tokens)
- Краткосрочная авторизация (OAuth-токены, сессионное хранилище)
- Доступ к оплачиваемым/дорогостоящим сервисам (для аутентификации, геокодирования, хранения файлов и т.д.)
- Приватная часть пары открытый/закрытый ключ (RECAPTCHA, Stripe, Auth0)
- Учётные данные сервиса (Email/SMTP, LDAP/Услуги каталога)
- Шифрование данных и проверка целостности

### Проверочный список: Безопасная работа с секретами

#### Быстрый обзор

Выполните следующие шаги, чтобы **исключить секреты из вашего кода:**

- [ ] Замените жёстко заданные ключи переменными окружения. Например: `process.env.API_SECRET`
- [ ] Используйте библиотеку, например [`dotenv`](https://github.com/motdotla/dotenv#dotenv) вместе с файлом `.env`. Добавьте ранее жёстко заданные секреты в файл `.env`.
- [ ] Добавьте строку `.env` в файл `.gitignore`!

> **НЕ СОЗДАВАЙТЕ** файл `.env` на серверах, где приложение развернуто. Используйте инструменты управления переменными окружения, предоставляемые вашими хостинг-сервисами (например, [Heroku](https://devcenter.heroku.com/articles/config-vars), Netlify, AWS EC2): например, **панель управления или командная строка**.

## 🌍 `Ключи, не являющиеся секретами`

**👍 Правило большого пальца:** если ключ необходимо передавать в браузер в коде или встроенно (например, через тег `<script src="https://my-api/?apiKey=123-abc-456">`), **это точно `не секрет`**. Пример — Google Maps.

<br />

**_Сценарии использования и функции:_** `Не секретные` ключи

- Краткосрочный доступ (идентификаторы сессий пользователей, JSON-токены веб-сессий)
- Ограничение доступа к API по приложению/разработчику (для аутентификации, геокодирования и т.д.)
- Публичная часть пары публичный/приватный ключ (RECAPTCHA, Stripe, Auth0)
- Идентификаторы аналитики

#### ✅ Обработка несекретных ключей:

> **Безопасно жестко кодировать несекретные (публичные) ключи!**

Упростите управление этим в долгосрочной перспективе с помощью общего `config.js` для вашего приложения.

**Пример:**

```js
// config.js
module.exports = {
  googleMapsKey: '123-abc'
};
```

```js
// load-map.js
const config = require('./config.js');
const key = config.googleMapsKey;
const src = `//maps.googleapis.com/maps/api/js?key=${key}`;
// ...
```
