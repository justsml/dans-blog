> Безопасность API-ключей и токенов **критически важна**!

Одна ошибка может привести к потере контроля над вашим сервером и данными хакерами!

Должно быть легче определить, нуждается ли конкретный токен в скрытии — даже на основе официальной документации!

Ситуацию усугубляет смесь связанных терминов, с которыми вы столкнетесь: _токены_, _ключи_, _учетные данные_, _секреты_, _приватные_ и _публичные_.

Давайте переформулируем это как `секретные` и `несекретные`.

* 🔒 [`Секретные ключи`](#-secret-keys) ДОЛЖНЫ храниться в секрете. Обычно они НИКОГДА не должны покидать ваш личный сервер (или сервис, такой как Heroku, Netlify или Travis-CI).
* 🌍 [`Несекретные ключи`](#-non-secret-keys) описывают строки, которые можно свободно делиться и включать в запросы браузера.

## 🔒 `Секретные ключи`

** ‼️ Важно:** `Секретные ключи` **ДОЛЖНЫ** игнорироваться Git _И_ исключаться из всех браузерных скриптов. [Как использовать dotenv](#-how-to-handle-secrets-safely)

<br />

_Как понять, что вы работаете с `Секретным ключом`?_

<br />

**👍 Правило:** серверы, возвращающие `CORS ошибки`, не поддерживают браузер. Это означает, что вы **ДОЛЖНЫ** использовать прокси, как будто это `секрет`.

**👍 Правило:** дорогостоящие сервисы (почти) всегда должны быть проксированы или скрыты.

**👍 Правило:** если вы выполняете операцию записи (**загрузка файла, вставка строки в БД**), вы работаете с `секретными ключами`.

<br />

**Сценарии использования и функции:** `Секретные` ключи

- Долгосрочная авторизация (учетные данные, токены доступа, JSON Web Tokens)
- Краткосрочная авторизация (OAuth-токены, сессионное хранение)
- Доступ к платным/дорогостоящим сервисам (для аутентификации, геокодирования, хранения файлов и т.д.)
- Приватная часть пары открытый/закрытый ключ (RECAPTCHA, Stripe, Auth0)
- Учетные данные сервисов (Email/SMTP, LDAP/Директория)
- Шифрование данных и проверка целостности

### Проверочный список: безопасная работа с секретами

#### Быстрый обзор

Выполните следующие шаги, чтобы **удалить секреты из вашего кода:**

- [ ] Замените жестко заданные ключи переменными окружения. Например: `process.env.API_SECRET`
- [ ] Используйте библиотеку, например [`dotenv`](https://github.com/motdotla/dotenv#dotenv), вместе с файлом `.env`. Добавьте ранее жестко заданные секреты в файл `.env`.
- [ ] Добавьте строку `.env` в файл `.gitignore`!

> **НЕ создавайте** файл `.env` на серверах после деплоя. Используйте инструменты управления переменными окружения, предоставляемые вашим хостинг-провайдером (например, [Heroku](https://devcenter.heroku.com/articles/config-vars), Netlify, AWS EC2): например, **через панель управления или командную строку.**

## 🌍 `Не секретные ключи`

**👍 Правило-подсказка:** когда ключ необходимо отправить в браузер в коде или inline (например, через тег `<script src="https://my-api/?apiKey=123-abc-456">`), **это точно `не секретный` ключ**. Частый пример — Google Maps.

<br />

**_Сценарии использования и особенности:_** `Не секретные` ключи

- Краткосрочный доступ (идентификаторы сессий пользователей, JSON Web Tokens)
- Ограничение доступа к API по приложению/разработчику (для авторизации, геокодирования и т.д.)
- Публичная часть пары публичный/приватный (RECAPTCHA, Stripe, Auth0)
- Идентификаторы аналитики

#### ✅ Работа с несекретными ключами:

> **Безопасно жёстко кодировать несекретные (публичные) ключи!**

Упростите управление на долгосрочной основе с помощью общего `config.js` для вашего приложения.

**Пример:**

```js
// config.js
module.exports = {
  googleMapsKey: '123-abc'
};
```

```js
// load-map.js
const config = require('./config.js');
const key = config.googleMapsKey;
const src = `//maps.googleapis.com/maps/api/js?key=${key}`;
// ...
```
