对话框只是“辅助轮”。

第一代 LLM 应用大多看起来就像是在产品上钉了一个文本框。模型返回散文，前端渲染 Markdown。如果用户需要执行操作，助手会描述一下用户应该去别处点击的那个按钮。

对于演示来说，这没问题。但这绝不是未来的方向。

下一个有价值的阶段是**生成式 UI (Generative UI)**：模型不仅仅是用文字回答，它还负责决定用户当前需要什么样的界面。有时这意味着调用一个工具并渲染一个预制的卡片；有时意味着用新数据填充一个已有的工作流组件；有时则意味着临时组合出一个仪表盘、表单、对比表、图表或交互式小部件。

不幸的是，“生成式 UI”已经变成了那种在早餐前就能演变出五种不同含义的术语。

人们用它来描述：

- 模型从开发者定义的 React 组件中进行选择
- 前端根据 JSON 规范渲染成原生组件
- 从 MCP 工具返回的 iframe 应用
- 支持工具调用的聊天 UI 库
- 在后端和前端之间流式传输状态的智能体协议
- 像 v0、Lovable、Bolt 或 Cursor 这样的设计时代码生成器
- 模型在运行时直接编写 HTML、SVG、Canvas 或 React

这些东西虽然相关，但并不在同一个层级。如果你把它们混为一谈，每一场架构讨论都会变成一锅乱粥。

这是我在开始对比当前技术栈时，希望自己能拥有的一张地图。

![LLM 生成式 UI 领域的层级地图](.././landscape-map.webp)

## 核心误区

最大的错误是将“生成式 UI”视为单一的技术选择。

更好的做法是将问题拆分为四个层级：

1.  **产品外壳 (Product shell)**：用户接触的东西。这可能是聊天窗口、侧边栏 Copilot、仪表盘、工作流构建器、IDE 面板、ChatGPT 应用、移动端屏幕或支持控制台。
2.  **UI 组合模型 (UI composition model)**：允许模型使用的“语法”。它可能是工具调用、JSON、A2UI、json-render、OpenUI Lang、Hashbrown 组件选择，或者是沙箱化的 HTML。
3.  **运行时与传输 (Runtime and transport)**：消息、工具调用、状态增量、用户操作和 UI 产物如何在智能体和前端之间移动。AG-UI、MCP Apps、Apps SDK、A2A、SSE、WebSockets 以及老旧的 HTTP 都属于这一层。
4.  **智能体与工具后端 (Agent and tool backend)**：LangGraph、Google ADK、CrewAI、Mantra、LlamaIndex、Pydantic AI、Agno、OpenAI Agents SDK、自定义函数、数据库、检索，以及所有那些仍然必须保证正确性的乏味的业务逻辑。

一旦拆分了层级，这个生态系统就变得不再那么神秘了。

[AG-UI](https://github.com/ag-ui-protocol/ag-ui) 并不是 [A2UI](https://github.com/google/A2UI) 的真正竞争对手。AG-UI 是一个用于智能体与应用交互的事件协议。A2UI 是智能体可以发送的一种声明式 UI 格式。你可以把 A2UI 跑在 AG-UI 之上，也可以把自定义的工具渲染组件跑在 AG-UI 之上。

[json-render](https://github.com/vercel-labs/json-render) 不是一个聊天产品。它是一个组件目录和渲染器架构：定义模型可以使用的组件，让模型输出有效的 JSON 树，并安全地渲染该树。

[CopilotKit](https://github.com/CopilotKit/CopilotKit) 不仅仅是一个聊天气泡。它是一个用于智能体原生应用的完整前端栈：包含聊天 UI、生成式 UI、共享状态、前端工具以及“人机协同 (human-in-the-loop)”工作流。

[OpenAI Apps SDK](https://developers.openai.com/apps-sdk) 和 [MCP Apps](https://github.com/MCP-UI-Org/mcp-ui) 并不是那种“让我的 React 应用动起来”的工具。它们是宿主集成模型，用于在 ChatGPT 或其他兼容 MCP 的宿主环境中渲染小组件（widgets）。

由于这个领域还很新，名称往往容易混淆。但真正有价值的是其中的层级结构。

## 控制权光谱

生成式 UI 是**开发者控制权**与**智能体自由度**之间的一种权衡。

控制过多，助手感觉就像是一个穿着马甲的命令面板。自由度过高，模型就会带着自信的微笑开始臆造奇怪的布局、模糊的按钮、破碎的视觉层级、不可能的状态以及安全隐患。

诀窍在于：选择能解决用户问题的最小自由度。

![从工具渲染组件到开放式生成 HTML 的光谱](../control-spectrum.webp)

我是这样看待这个光谱的：

**工具到组件的渲染 (Tool-to-component rendering)** 是最安全的默认选择。模型调用 `get_weather`、`search_products`、`compare_plans` 或 `draft_invoice`。应用将该工具的结果映射到你已有的组件：`WeatherCard`、`ProductGrid`、`PlanComparison`、`InvoiceReview`。模型决定*何时* UI 是有用的。开发者仍然掌握布局、样式、无障碍性、加载状态、空状态以及危险操作。

这是 [Vercel AI SDK 生成式 UI 指南](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces)中记录的模式：模型调用工具，工具返回数据，UI 根据结果渲染组件。这也是许多 CopilotKit 和 assistant-ui 实现背后的思维模型。

**声明式组件目录 (Declarative component catalogs)** 给了模型更多空间。模型不再是选择单个组件，而是从允许的部件中组合出一棵树。目录可能包括 `Metric`、`Table`、`Chart`、`FilterBar`、`ApprovalPanel` 和 `Timeline`。模型可以组装仪表盘或工作流步骤，但它不能执行任意代码。这就是 [A2UI](https://github.com/google/A2UI)、[json-render](https://github.com/vercel-labs/json-render)、[Hashbrown](https://github.com/liveloveapp/hashbrown) 和 [OpenUI](https://github.com/thesysdev/openui) 所处的位置。

**Iframe 微应用 (Iframe mini-apps)** 适用于 UI 需要比组件树更丰富，或者由远程工具提供商掌控体验的场景。MCP Apps 和 OpenAI Apps SDK 允许工具返回结构化数据以及一个由宿主在 iframe 中渲染的小组件资源。这对于地图、购物车、预订流程、图表和外部产品界面非常有效。它还在宿主应用和小组件之间建立了一道更硬的边界。

**开放式生成 (Open-ended generation)** 位于光谱的最远端：智能体向沙箱中输出 HTML、SVG、Canvas、WebGL 或其他类似代码的产物。[OpenGenerativeUI](https://github.com/CopilotKit/OpenGenerativeUI) 是目前最好的例子：智能体可以在沙箱化的 iframe 中生成算法可视化、3D 场景、图表和模拟。这非常适合一次性的视觉解释。但对于企业级的审批流，我不会从这里开始。

这里有必要明确一个关键区别：**iframe HTML**（模型在沙箱中编写代码）与 **JSON 目录**（模型输出结构化规范，你的渲染器将其映射到预构建组件）。这两者听起来相关，但风险和复杂度完全不同。Iframe HTML 具有最大的表现力，由 iframe 边界承担安全工作。JSON 目录则不给模型任何可执行的自由——它只能引用你预先定义的组件类型。这个领域的大多数框架都明确属于其中一个阵营。

**超越沙箱**：最近的演示表明第四种模式正在形成——LLM 通过比任何组件目录都更直接地控制视觉输出来驱动类游戏或沉浸式体验。从提示词生成可探索的 3D 世界、运行时由 LLM 指导的 NPC 行为，以及通过 WebGPU 进行的浏览器内模型推理（[WebLLM](https://mlc.ai/web-llm/)）都是早期信号。目前还没有稳定的框架可以用于生产环境。一旦情况发生变化，我将在专门的文章中讨论这个方向。

## 高层级组件 vs 细粒度组件

这是最重要的设计决策。

如果你的目录过于细粒度，模型就必须变成一个前端工程师：

```tsx
Container
Row
Column
Text
Button
Icon
Spacer
Divider
```

这种方案看起来很灵活，但现在模型必须去决定间距、层级、分组、空状态、按钮文案、错误处理以及响应式行为。你不仅让 Prompt 变得臃肿，还让输出结果变得极易崩溃。

如果你的目录层级过高，模型就会被困住：

```tsx
WeatherCard
StockCard
HotelCard
```

这很安全，但它只适用于已知场景。当用户的需求发生变化时，模型无法生成对比矩阵、无法请求缺失的输入，也无法调整信息架构。

最有用的中间地带是**带有约束插槽的领域级组件**：

```tsx
SearchResults
ComparisonTable
MetricGroup
EditablePlan
ApprovalRequest
Timeline
DataCollectionForm
CheckoutReview
```

这些组件编码了产品品味和业务约束。模型负责决定*应该展示什么*，而不是决定每一个 CSS 细节。

例如，一个旅游助手不需要 `div`、`span` 和 `button`。它需要：

- `TripSummary`（行程摘要）
- `FlightOptionList`（航班选项列表）
- `HotelComparison`（酒店对比）
- `TravelerForm`（旅客表单）
- `PolicyNotice`（政策须知）
- `BookingConfirmation`（预订确认）

一个金融助手不需要通用的图表沙盒。它需要：

- `PortfolioSnapshot`（投资组合快照）
- `TransactionTable`（交易表格）
- `RiskBreakdown`（风险分解）
- `ScenarioComparison`（方案对比）
- `ApprovalGate`（审批关口）

组件目录听起来应该像你的产品，而不是像 HTML。

## 不同产品该如何选择

这是我会实际建议团队参考的选择矩阵。

**你正在为现有的 SaaS 应用添加助手。**

从“工具到组件”的渲染开始。根据你对 Agent 状态和前端工具集成的需求程度，选择 [Vercel AI SDK UI](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces)、[assistant-ui](https://github.com/assistant-ui/assistant-ui) 或 [CopilotKit](https://github.com/CopilotKit/CopilotKit)。起步时保持组件目录极小，仅渲染你已经信任的产品组件。

**你正在构建一个需要共享状态的深度应用内 Copilot。**

重点关注 CopilotKit 配合 AG-UI。核心功能不在于“聊天”，而在于共享状态和双向交互：Agent 可以请求输入、渲染 UI、更新状态，并暂停等待审批。

**你拥有需要跨边界发送 UI 的远程 Agent。**

使用 A2UI 或类似 A2UI 的声明式协议。其核心意义在于远程 Agent 可以将 UI 描述为数据，而宿主环境则保留对原生渲染、安全性和样式的控制。如果你还需要实时的 Agent/应用交互，请通过 AG-UI 或你环境标准化的任何传输层来运行。

**你正在 ChatGPT 或兼容 MCP 的宿主环境中构建。**

使用 MCP Apps 和 Apps SDK 路径。OpenAI 目前的文档建议在新项目中使用 MCP Apps 的 `ui/*` 桥接，同时保留 `window.openai` 作为兼容层和可选的扩展接口。此外，参考他们对数据工具和渲染工具的拆分：让模型在决定渲染组件之前，先获取数据并进行推理。

**你想在自己的应用中实现自然语言生成的仪表盘、报告或表单。**

尝试 json-render、Hashbrown 或 OpenUI。关键在于组件库。如果你暴露了 `LineChart`、`DataTable`、`MetricGroup`、`FilterControl` 和 `InsightCallout`，模型就能组装出有用的报表界面，而无需接触任何任意代码。

**你需要教育类、视觉类或高度定制化的 Artifacts。**

使用像 OpenGenerativeUI 这样的开放式沙箱。让模型编写 SVG、Canvas、WebGL 或自包含的 HTML，但要将输出视为不可信的用户内容。对其进行沙箱化处理、限制尺寸、剥夺权限，并使其远离特权应用状态。

**你主要只需要更美观的流式 Markdown 以及少量行内交互。**

不要过度设计。llm-ui 或 assistant-ui 的工具渲染可能就足够了。

## 我会避免的错误

**错误 1：让模型在运行时编写生产环境的 React 代码。**

虽然有例外，但对于产品 UI 来说，这通常是错误的默认选择。运行时代码生成难以保证安全性、难以测试、难以统一主题，且难以保持无障碍性。如果模型可以通过选择受信任的组件来完成任务，那就这样做。

**错误 2：暴露设计原语而非产品原语。**

当你给模型 `Row`、`Column`、`Text` 和 `Button` 时，你是在要求它充当你的设计系统。它会表现得很平庸。请给它更高层级的产品名词。

**错误 3：认为合法的 JSON 就意味着安全的 UI。**

一个 Payload 可以通过 Schema 验证，但仍可能具有误导性或危险性。标签可能写着“查看发票”，而实际操作却是注销账户。将 UI 规范视为行为，而非装饰。对于产生后果的操作，需要进行策略测试、语义检查和人工确认。

**错误 4：将业务逻辑塞进渲染工具。**

渲染工具只负责渲染。数据工具负责获取、计算、变更和验证。OpenAI 的 Apps SDK 文档特意强调这种分离是有原因的：如果每个数据工具都捆绑一个组件，模型在呈现结果之前的推理空间就会被压缩。

**错误 5：追求新奇感而非任务完成度。**

重点不是要把每个回答都做成独一无二的界面，而是要减少摩擦。一个稳定、乏味的审批面板如果能为用户节省四分钟，也远比一个无法产生持续信任的炫酷生成式仪表盘要好。

## 一个实用的架构

如果我今天要启动一个新产品，我会采用分阶段的方法：

1.  **先上线受控的工具 UI。** 将已知工具映射到已知组件。记录每一次工具调用、UI 渲染和用户操作。
2.  **添加领域目录。** 当模式开始重复时，抽象出 `ComparisonTable`（对比表）、`DecisionPanel`（决策面板）、`DataCollectionForm`（数据收集表单）、`Timeline`（时间轴）以及其他特定于产品的组件。
3.  **仅在需要时添加传输标准化。** 如果你同时控制前后端，普通的流式传输（Streaming）就足够了。如果你有多个智能体框架，使用 AG-UI。如果工具跨越产品边界，使用 MCP。如果智能体跨越组织边界，关注 A2A 和 A2UI。
4.  **对外部或复杂的界面使用 iframe 组件。** 地图、购物车、预订流程和第三方小程序都应该放在隔离边界之后。
5.  **将开放式生成保留给“制品”（Artifacts）。** 图表、模拟器、临时说明页和视觉草稿板非常适合这种方式。核心工作流则不然。

最终的架构看起来像这样：

```txt
用户意图
  -> 智能体运行时
  -> 工具/数据调用
  -> 结构化结果
  -> UI 决策
  -> 受信任组件、声明式规范或沙箱化组件
  -> 用户操作
  -> 状态/事件流回传给智能体
```

这个闭环才是真正的产品。聊天框仅仅是其中一种可能的输入设备。

## 评估应包含 UI

LLM 团队正在慢慢学会如何评估 Prompt 和模型输出。生成式 UI 增加了另一个维度：界面本身也可能出错。

至少，应为每个生成的 UI 保存以下资产：

- Prompt 和工具上下文
- 工具调用和工具结果
- 生成的 UI 规范或组件选择
- 渲染的组件名称和 Props
- 用户可见的标签
- 绑定在按钮/表单上的操作
- UI 反馈给模型的可见状态更新
- 用户操作历史

然后编写如下检查项：

- 每一个破坏性操作必须带有确认组件
- 按钮标签必须与操作语义匹配
- 渲染规范只能引用允许的组件
- 用户可见的总计数值必须与工具返回的结果一致
- 表单不得请求任务范围之外的字段
- 组件不得接收只有模型才需要的敏感信息（Secrets）
- 隐藏的元数据不得与可见标签相矛盾

这听起来很繁琐，但这也是生产环境信任的来源。

## 关于项目稳定性的说明

该领域的所有主要协议目前都处于 pre-1.0 阶段。最后验证日期为 2026 年 5 月 8 日；在进行平台选型前，请务必查阅最新文档并做好应对变化的准备。

**Vercel AI SDK RSC** —— 最初作为“生成式 UI”的核心特性推出 —— 已于 2024 年 10 月暂停开发（[讨论帖 #3251](https://github.com/vercel/ai/discussions/3251)），原因是其架构局限性在短期内无法修复。**json-render** (Vercel Labs) 作为替代方案出现：基于目录、框架无关、不与 RSC 耦合。自 2026 年初发布以来，它迅速吸引了 Web 开发者的关注。其原因很可能是 DX（开发者体验）：json-render 可以直接在标准 React 项目中使用，而 A2UI 的跨平台野心带来了更高的配置摩擦。

**A2UI** (Google) 处于 pre-1.0 阶段，次版本号之间存在破坏性变更，路线图沟通也不够透明。它的优势在于真正的跨平台能力（Web、Flutter、SwiftUI），这是 json-render 未触及的领域。对于目前的纯 Web 场景，json-render 的工具链覆盖更佳；对于跨平台或远程 Agent 场景，A2UI 的设计更为合理。这两个规范未来可能会趋同 —— Vercel 已经尝试让 json-render 输出兼容 A2UI 的内容。

**AG-UI** (CopilotKit) 同样处于 pre-1.0。最常见的误解是它的名字：AG-UI 是一个传输协议，而不是 UI 框架。它定义了事件在 Agent 和前端之间*如何*流动；至于渲染什么，依然由你决定。这个概念很稳固且被广泛采用，但规范仍在演进中。

## 我的观点

生成式 UI 不会取代经过精心设计的产品界面。它取代的是那种“聊天记录就是 AI 万能接口”的懒惰假设。

最优秀的系统不会让模型随意发挥。它们会给模型一套精简、锋利且原生于产品的构建块；一个可靠的运行时连接；清晰的安全边界；以及足够的自由度，让界面能够根据任务进行自适应。

未来并不是“模型来写你的前端”。

未来更像是：**你的前端变成了一件 Agent 可以演奏的乐器，但你依然决定这件乐器允许发出什么样的声音。**
