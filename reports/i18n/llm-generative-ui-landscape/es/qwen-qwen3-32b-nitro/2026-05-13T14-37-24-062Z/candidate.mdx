El chat fue la rueda de entrenamiento.  

La primera generación de aplicaciones de LLM se parecía a un cuadro de texto pegado a un producto. El modelo devolvía prosa. La interfaz de usuario representaba markdown. Si el usuario necesitaba tomar una acción, el asistente describía el botón que el usuario debía hacer clic en otro lugar.  

Eso estaba bien para demostraciones. No es a donde esto va.  

El siguiente paso útil es la **interfaz de usuario generativa**: el modelo no responde únicamente con texto; ayuda a decidir qué interfaz necesita el usuario en este momento. A veces significa llamar a una herramienta y representar una tarjeta preconstruida. A veces significa rellenar un componente de flujo de trabajo conocido con datos frescos. A veces significa componer un panel temporal, formulario, tabla de comparación, gráfico o widget interactivo.  

Lamentablemente, "interfaz de usuario generativa" se ha convertido en una de esas frases que significa cinco cosas diferentes antes del desayuno.  

Las personas lo usan para describir:

- un modelo que elige entre componentes React definidos por el desarrollador  
- una especificación JSON que el frontend renderiza en componentes nativos  
- una aplicación iframe devuelta por una herramienta MCP  
- una biblioteca de interfaz de chat que admite llamadas a herramientas  
- un protocolo de agente que transmite el estado entre backend y frontend  
- un generador de código en tiempo de diseño como v0, Lovable, Bolt o Cursor  
- un modelo que literalmente escribe HTML, SVG, Canvas o React en tiempo de ejecución  

Estos están relacionados, pero no son la misma capa. Si los mezclas, cada conversación sobre arquitectura se convierte en sopa.  

Esta es la mapa que hubiera querido tener cuando empecé a comparar la pila actual.  

![Un mapa estratificado del paisaje de interfaz de usuario generativa de LLM](../landscape-map.webp)  

## El malentendido fundamental  

El mayor error es tratar la "interfaz de usuario generativa" como una sola elección tecnológica.

Es mejor separar el problema en cuatro capas:

1. **Carcasa del producto**: lo que los usuarios tocan. Puede ser un chat, copiloto lateral, panel de control, constructor de flujos de trabajo, panel de IDE, aplicación de ChatGPT, pantalla móvil o consola de soporte.
2. **Modelo de composición de la interfaz**: la gramática que el modelo puede usar. Puede ser llamadas a herramientas, JSON, A2UI, json-render, OpenUI Lang, selección de componentes de Hashbrown o HTML aislado.
3. **Entorno de ejecución y transporte**: cómo se mueven los mensajes, llamadas a herramientas, deltas de estado, acciones del usuario y artefactos de interfaz entre el agente y la interfaz frontal. AG-UI, MCP Apps, Apps SDK, A2A, SSE, WebSockets y HTTP tradicional residen aquí.
4. **Backend de agentes y herramientas**: LangGraph, Google ADK, CrewAI, Mastra, LlamaIndex, Pydantic AI, Agno, OpenAI Agents SDK, funciones personalizadas, bases de datos, recuperación y toda la lógica empresarial aburrida que aún debe ser correcta.

Una vez que divides las capas, el ecosistema se vuelve mucho menos misterioso.

[AG-UI](https://github.com/ag-ui-protocol/ag-ui) no es realmente un competidor de [A2UI](https://github.com/google/A2UI). AG-UI es un protocolo de eventos para interacción entre agentes y aplicaciones. A2UI es un formato de interfaz declarativo que el agente puede enviar. Puedes colocar A2UI sobre AG-UI. También puedes colocar componentes personalizados renderizados sobre AG-UI.

[json-render](https://github.com/vercel-labs/json-render) no es un producto de chat. Es una arquitectura de catálogo y renderizado de componentes: define los componentes que el modelo puede usar, haz que el modelo emita un árbol JSON válido y renderiza ese árbol de forma segura.

[CopilotKit](https://github.com/CopilotKit/CopilotKit) no es solo una burbuja de chat. Es una pila frontend para aplicaciones nativas de agentes: interfaz de chat, interfaz generativa, estado compartido, herramientas frontend y flujos con participación humana.

[SDK de Apps de OpenAI](https://developers.openai.com/apps-sdk) y [Apps de MCP](https://github.com/MCP-UI-Org/mcp-ui) no son "herramientas para hacer mi app React dinámica". Son modelos de integración con anfitriones para renderizar widgets dentro de ChatGPT u otros anfitriones compatibles con MCP.

Los nombres son confusos porque el espacio es joven. Las capas son la parte que resulta útil.

## El espectro de control

La interfaz generativa es un equilibrio entre **control del desarrollador** y **libertad del agente**.

Demasiado control y el asistente se siente como una paleta de comandos disfrazada de algo. Demasiada libertad y el modelo comienza a inventar diseños extraños, botones vagos, jerarquía visual rota, estados imposibles y problemas de seguridad con una sonrisa confiada.

La clave es elegir la cantidad mínima de libertad que resuelva el problema del usuario.

![Un espectro desde componentes renderizados por herramientas hasta HTML generado de forma abierta](../control-spectrum.webp)

Pienso en el espectro así:

**El renderizado de herramienta a componente** es el valor por defecto más seguro. El modelo llama a `get_weather`, `search_products`, `compare_plans` o `draft_invoice`. La aplicación mapea ese resultado de la herramienta a un componente que ya posees: `WeatherCard`, `ProductGrid`, `PlanComparison`, `InvoiceReview`. El modelo decide *cuándo* la interfaz de usuario es útil. Los desarrolladores aún controlan el diseño, el estilo, la accesibilidad, los estados de carga, los estados vacíos y las acciones peligrosas.

Este es el patrón documentado en la [guía de interfaces de usuario generativas del SDK de IA de Vercel](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces): el modelo llama a una herramienta, la herramienta devuelve datos y la interfaz de usuario renderiza un componente a partir del resultado. También es el modelo mental detrás de muchas implementaciones de CopilotKit y assistant-ui.

**Catálogos de componentes declarativos** dan al modelo más flexibilidad. En lugar de elegir un componente, el modelo compone un árbol a partir de partes permitidas. Un catálogo podría incluir `Metric`, `Table`, `Chart`, `FilterBar`, `ApprovalPanel` y `Timeline`. El modelo puede ensamblar un panel de control o un paso de flujo de trabajo, pero no puede ejecutar código arbitrario. Esto es donde [A2UI](https://github.com/google/A2UI), [json-render](https://github.com/vercel-labs/json-render), [Hashbrown](https://github.com/liveloveapp/hashbrown) y [OpenUI](https://github.com/thesysdev/openui) se sitúan.

**Miniaplicaciones en iframe** tienen sentido cuando la interfaz de usuario necesita ser más rica que un árbol de componentes, o cuando un proveedor de herramientas remoto controla la experiencia. MCP Apps y OpenAI Apps SDK permiten que una herramienta devuelva datos estructurados más un recurso de widget que el anfitrión renderiza en un iframe. Esto es poderoso para mapas, carritos de compras, flujos de reserva, gráficos y superficies de productos externos. También crea un límite más difícil entre la aplicación anfitriona y el widget.

**Generación abierta** es el extremo opuesto: el agente emite HTML, SVG, Canvas, WebGL u otros artefactos similares a código en un entorno aislado. [OpenGenerativeUI](https://github.com/CopilotKit/OpenGenerativeUI) es el mejor ejemplo actual: el agente puede generar visualizaciones algorítmicas, escenas 3D, diagramas y simulaciones dentro de iframes aislados. Esto es ideal para explicaciones visuales únicas. No es el punto de partida que elegiría para un flujo de aprobación empresarial.

Es útil nombrar la distinción clave aquí: **iframe HTML** (el modelo escribe código en un entorno aislado) vs. **catálogo JSON** (el modelo emite una especificación estructurada y tu renderizador la mapea a componentes preconstruidos). Estos suenan relacionados pero tienen perfiles de riesgo y complejidad muy diferentes. El HTML en iframe es máximamente expresivo; el límite del iframe realiza el trabajo de seguridad. El catálogo JSON no otorga al modelo libertad ejecutable alguna: solo puede referirse a tipos de componentes que definiste previamente. La mayoría de los marcos en este espacio se sitúan claramente en uno de estos dos bandos.

**Más allá del entorno aislado**: demostraciones recientes sugieren una cuarta modalidad en formación: los LLM impulsan experiencias similares a juegos o inmersivas controlando la salida visual de manera más directa que cualquier catálogo de componentes. Proyectos que generan mundos 3D explorables a partir de prompts, comportamiento de NPC dirigido por LLM en tiempo de ejecución e inferencia de modelos en el navegador mediante WebGPU ([WebLLM](https://mlc.ai/web-llm/)) son indicadores iniciales. Aún no existen marcos estables para construir trabajo productivo aquí. Cubriré esta dirección en un artículo dedicado cuando cambie esta situación.

## Componentes de alto nivel vs. componentes granulares

Esta es la decisión de diseño más importante.

Si tu catálogo es demasiado granular, el modelo tiene que convertirse en un ingeniero frontend:

```tsx
Container
Row
Column
Text
Button
Icon
Spacer
Divider
```

Eso parece flexible, pero ahora el modelo debe decidir el espaciado, la jerarquía, la agrupación, los estados vacíos, las etiquetas de botones, el tratamiento de errores y el comportamiento responsivo. También has aumentado el tamaño del prompt y hecho más frágil la salida.

Si tu catálogo es demasiado alto nivel, el modelo queda atrapado:

```tsx
WeatherCard
StockCard
HotelCard
```

Eso es seguro, pero solo funciona para escenarios conocidos. El modelo no puede crear una matriz de comparación, solicitar entradas faltantes o adaptar la arquitectura de información cuando cambia la pregunta del usuario.

El punto intermedio útil es **componentes de nivel de dominio con espacios restringidos**:

```tsx
SearchResults
ComparisonTable
MetricGroup
EditablePlan
ApprovalRequest
Timeline
DataCollectionForm
CheckoutReview
```

Estos componentes codifican el gusto del producto y las restricciones del negocio. El modelo decide *qué mostrar*, pero no cada decisión de CSS.

Por ejemplo, un agente de viajes no necesita `div`, `span` ni `button`. Necesita:

- `ResumenDeViaje`  
- `ListaDeOpcionesDeVuelo`  
- `ComparaciónDeHoteles`  
- `FormularioDeViajero`  
- `AvisoDePolítica`  
- `ConfirmaciónDeReserva`  

Un agente financiero no necesita un entorno de visualización de gráficos genérico. Necesita:  

- `VistaDePortafolio`  
- `TablaDeTransacciones`  
- `DesgloseDeRiesgo`  
- `ComparaciónDeEscenarios`  
- `PuertaDeAprobación`  

El catálogo debe sonar como tu producto, no como HTML.

## Qué usar para qué producto

Aquí está la matriz de recomendación que usaría realmente con un equipo.

**Estás añadiendo un asistente a una aplicación SaaS existente.**

Comienza con el renderizado de herramienta a componente. Usa [Vercel AI SDK UI](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces), [assistant-ui](https://github.com/assistant-ui/assistant-ui) o [CopilotKit](https://github.com/CopilotKit/CopilotKit), dependiendo de cuánto estado de agente y integración de herramientas frontend necesites. Mantén el catálogo pequeño al principio. Renderiza componentes de producto que ya confíes.

Analiza detenidamente CopilotKit junto con AG-UI. La característica importante no es "chat". Es el estado compartido y la interacción bidireccional: el agente puede solicitar entrada, renderizar UI, actualizar el estado y pausar para obtener aprobación.

**Tienes agentes remotos que necesitan enviar UI a través de un límite.**

Usa A2UI o un protocolo declarativo similar a A2UI. El punto clave es que un agente remoto puede describir la UI como datos mientras el anfitrión mantiene el control del renderizado nativo, la seguridad y el estilo. Si también necesitas interacción en vivo entre el agente y la aplicación, ejecútala sobre AG-UI o cualquier transporte que tu entorno estandarice.

**Estás construyendo dentro de ChatGPT o un anfitrión compatible con MCP.**

Usa MCP Apps y la ruta del Apps SDK. Las actuales documentaciones de OpenAI recomiendan el puente MCP Apps `ui/*` para nuevos trabajos, manteniendo `window.openai` como capa de compatibilidad y superficie opcional de extensión. También copia su división entre herramientas de datos y herramientas de renderizado: permite que el modelo obtenga y razone sobre los datos antes de decidir renderizar un widget.

**Quieres dashboards, informes o formularios en lenguaje natural en tu propia aplicación.**

Prueba json-render, Hashbrown o OpenUI. La clave es el catálogo. Si expones `LineChart`, `DataTable`, `MetricGroup`, `FilterControl` y `InsightCallout`, el modelo puede ensamblar superficies de informes útiles sin acercarse a código arbitrario.

**Quieres artefactos educativos, visuales o altamente personalizados.**

Usa un sandbox de enfoque abierto como OpenGenerativeUI. Deja que el modelo escriba SVG, Canvas, WebGL o HTML autocontenido, pero trata la salida como contenido de usuario no confiable. Sáncralo, limita su tamaño, quita permisos y manténlo alejado del estado privilegiado de la aplicación.

**Principalmente necesitas markdown de streaming más atractivo con algunas funcionalidades en línea.**

No sobreconstruyas. Puede que llm-ui o assistant-ui sean suficientes para el renderizado.

## Los errores que evitaría

**Error 1: Permitir que el modelo escriba React de producción en tiempo de ejecución.**  

Hay excepciones, pero para interfaces de producto esto suele ser el enfoque equivocado por defecto. La generación de código en tiempo de ejecución es difícil de asegurar, difícil de probar, difícil de personalizar y difícil de mantener accesible. Si el modelo puede completar la tarea seleccionando componentes de confianza, hazlo.  

**Error 2: Exponer primitivas de diseño en lugar de primitivas de producto.**  

Cuando ofreces al modelo `Row`, `Column`, `Text` y `Button`, lo estás obligando a convertirse en tu sistema de diseño. Se convertirá en uno mediocre. Ofrecele nombres de producto de nivel superior.  

**Error 3: Pensar que JSON válido significa interfaz segura.**  

Una carga útil puede pasar la validación de esquema y aún así ser manipulativa o peligrosa. La etiqueta puede decir "Ver factura" mientras la acción archiva la cuenta. Trata las especificaciones de la interfaz como comportamiento, no como decoración. Necesitan pruebas de políticas, verificaciones semánticas y confirmación humana para acciones con consecuencias.

**Error 4: Colocar lógica de negocio en herramientas de renderizado.**  

Las herramientas de renderizado deben encargarse solo de renderizar. Las herramientas de datos deben obtener, calcular, mutar y validar. Las documentaciones de OpenAI Apps SDK destacan esta separación por una razón: si cada herramienta de datos arrastra un widget consigo, el modelo pierde espacio para razonar antes de presentar.  

**Error 5: Optimizar por novedad en lugar de finalización de tareas.**  

El objetivo no es hacer que cada respuesta sea una interfaz única como un copo de nieve. El objetivo es reducir la fricción. Un panel de aprobación estable y "aburrido" que ahorre al usuario cuatro minutos es mejor que un panel de dashboard generado de forma espectacular que no pueda confiarse ni una segunda vez.  

## Una arquitectura práctica  

Si comenzara hoy un nuevo producto, usaría un enfoque en etapas:

1. **Entrega primero la interfaz de herramientas controladas.** Asigne herramientas conocidas a componentes conocidos. Registre cada llamada a herramientas, renderizado de la interfaz y acción del usuario.  
2. **Agregue un catálogo de dominio.** Una vez que los patrones se repitan, exponga `ComparisonTable`, `DecisionPanel`, `DataCollectionForm`, `Timeline` y otros componentes específicos del producto.  
3. **Agregue estandarización de transporte solo cuando sea necesario.** Si controla tanto el frontend como el backend, el streaming simple puede ser suficiente. Si tiene múltiples marcos de agentes, use AG-UI. Si las herramientas cruzan límites de productos, use MCP. Si los agentes cruzan límites organizacionales, observe A2A y A2UI.  
4. **Use widgets de iframe para superficies externas o complejas.** Mapas, carritos, flujos de reserva y miniaplicaciones de terceros deben estar detrás de un límite.  
5. **Reserve la generación abierta para artefactos.** Diagramas, simulaciones, explicaciones temporales y borradores visuales son ideales. Los flujos de trabajo principales no lo son.  

La arquitectura termina viéndose así:  

```txt
User intent
  -> agent runtime
  -> tool/data calls
  -> structured result
  -> UI decision
  -> trusted component, declarative spec, or sandboxed widget
  -> user action
  -> state/event stream back to the agent
```  

Ese ciclo es el verdadero producto. El cuadro de chat es solo un dispositivo de entrada posible.  

## La evaluación debe incluir la interfaz  

Los equipos de LLM están aprendiendo lentamente a evaluar prompts y salidas del modelo. La interfaz generativa añade otra superficie: la propia interfaz puede estar equivocada.  

Como mínimo, guarde estos artefactos para cada interfaz generada:

- prompt y contexto de la herramienta  
- llamadas a herramientas y resultados de las herramientas  
- especificación de la UI generada o selección de componentes  
- nombre del componente renderizado y sus props  
- etiquetas visibles para el usuario  
- acciones asociadas a botones/formularios  
- actualizaciones de estado visibles para el modelo desde la UI  
- historial de acciones del usuario  

Luego escriba comprobaciones como:  

- cada acción destructiva debe tener un componente de confirmación  
- las etiquetas de botones deben coincidir con la semántica de la acción  
- las especificaciones de renderizado solo pueden referenciar componentes permitidos  
- los totales visibles para el usuario deben coincidir con los totales de los resultados de las herramientas  
- los formularios no pueden solicitar campos fuera del alcance de la tarea  
- los widgets no pueden recibir secretos que solo necesite el modelo  
- los metadatos ocultos no deben contradecir las etiquetas visibles  

Esto suena tedioso. También es de donde proviene la confianza en producción.

## Nota sobre estabilidad de proyectos  

Todos los protocolos principales en este espacio están en versión pre-1.0. Verificado por última vez el 8 de mayo de 2026; planifique cambios y revise las documentaciones actuales antes de comprometerse con una plataforma.  

**Vercel AI SDK RSC** — la característica original "UI Generativa" — tuvo su desarrollo pausado en octubre de 2024 ([Discusión #3251](https://github.com/vercel/ai/discussions/3251)) debido a limitaciones arquitectónicas sin solución inminente. **json-render** (Vercel Labs) emergió como dirección de reemplazo: basado en catálogos, sin dependencia de RSC y compatible con cualquier framework. Aparentemente atrajo rápidamente la atención de desarrolladores web tras su lanzamiento en principios de 2026. La razón probable es la experiencia de desarrollo (DX): json-render funciona inmediatamente en un proyecto React estándar; el alcance multiplataforma de A2UI añade fricción en la configuración.  

**A2UI** (Google) está en pre-1.0 con cambios rotos entre versiones menores y comunicaciones inconsistentes sobre su roadmap. Su ventaja es un alcance real multiplataforma (web, Flutter, SwiftUI) que json-render no aborda. Para casos de uso puramente web, json-render parece tener mejor cobertura de herramientas; para escenarios multiplataforma o agentes remotos, el diseño de A2UI es más adecuado. Es posible que las especificaciones converjan: Vercel ha experimentado con salida compatible con A2UI desde json-render.  

**AG-UI** (CopilotKit) también está en pre-1.0. La confusión más común es el nombre: AG-UI es un protocolo de transporte, no un framework de UI. Define *cómo* fluyen los eventos entre el agente y la interfaz; lo que se renderiza sigue siendo decisión del desarrollador. El concepto es sólido y ampliamente adoptado. La especificación aún está evolucionando.

## Mi perspectiva

La interfaz generativa no reemplazará las interfaces de productos diseñadas con cuidado. Sustituirá la suposición perezosa de que un transcripción de chat es la interfaz universal para la IA.

Los mejores sistemas no permitirán que el modelo improvise todo. Le darán un conjunto pequeño y preciso de componentes nativos del producto; una conexión de tiempo de ejecución confiable; límites claros de seguridad; y suficiente libertad para adaptar la interfaz a la tarea.

El futuro no es "el modelo escribe tu frontend".

El futuro es más cercano a: **tu frontend se convierte en un instrumento que el agente puede tocar, pero tú decides qué sonidos puede emitir.**
