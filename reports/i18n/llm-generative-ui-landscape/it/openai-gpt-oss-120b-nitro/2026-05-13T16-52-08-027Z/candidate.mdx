Chat era le ruote di allenamento.

La prima generazione di app LLM assomigliava per lo più a una casella di testo attaccata a un prodotto. Il modello restituiva prosa. Il frontend rendeva markdown. Se l'utente doveva compiere un'azione, l’assistente descriveva il pulsante che l’utente doveva cliccare altrove.

Per le demo andava bene. Non è la direzione in cui stiamo andando.

Il passo successivo utile è **generative UI**: il modello non si limita a rispondere con testo; aiuta a decidere quale interfaccia l’utente ha bisogno in quel momento. A volte ciò significa chiamare uno strumento e visualizzare una scheda predefinita. A volte significa riempire un componente di workflow noto con dati freschi. A volte significa comporre una dashboard temporanea, un modulo, una tabella comparativa, un grafico o un widget interattivo.

Sfortunatamente, “generative UI” è diventata una di quelle frasi che indicano cinque cose diverse prima della colazione.

Le persone la usano per descrivere:

- un modello che sceglie tra componenti React definiti dallo sviluppatore  
- una specifica JSON che il frontend rende in componenti nativi  
- un’app iframe restituita da uno strumento MCP  
- una libreria UI per chat che supporta chiamate a tool  
- un protocollo agente che trasmette lo stato tra backend e frontend in streaming  
- un generatore di codice a tempo di progettazione come v0, Lovable, Bolt o Cursor  
- un modello che scrive letteralmente HTML, SVG, Canvas o React a runtime  

Sono correlati, ma non appartengono alla stessa layer. Se li confondi, ogni discussione architetturale si trasforma in una zuppa.

Questa è la mappa che avrei voluto avere quando ho iniziato a confrontare lo stack attuale.

![Una mappa a strati del panorama UI generativa LLM](../landscape-map.webp)

## L'Errore Fondamentale

L'errore più grande è trattare “generative UI” come un'unica scelta tecnologica.

È più efficace suddividere il problema in quattro layer:

1.  **Shell del prodotto**: l'elemento con cui gli utenti interagiscono. Può essere una chat, un copilot laterale, un cruscotto, un costruttore di workflow, un pannello IDE, un’app ChatGPT, una schermata mobile o una console di supporto.
2.  **Modello di composizione UI**: la grammatica che il modello è autorizzato a produrre. Può trattarsi di chiamate a tool, JSON, A2UI, json-render, OpenUI Lang, selezione di componenti Hashbrown o HTML sandboxato.
3.  **Runtime e trasporto**: come messaggi, chiamate a tool, delta di stato, azioni dell'utente e artefatti UI si spostano tra l’agente e il frontend. AG-UI, MCP Apps, Apps SDK, A2A, SSE, WebSockets e il classico HTTP rientrano in questa categoria.
4.  **Backend agente e tool**: LangGraph, Google ADK, CrewAI, Mastra, LlamaIndex, Pydantic AI, Agno, OpenAI Agents SDK, funzioni personalizzate, database, retrieval e tutta la logica di business noiosa che deve comunque funzionare correttamente.

Una volta separati i layer, l’ecosistema perde gran parte del suo alone di mistero.

[AG-UI](https://github.com/ag-ui-protocol/ag-ui) non è davvero un concorrente di [A2UI](https://github.com/google/A2UI). AG-UI è un protocollo di eventi per l’interazione agente‑applicazione. A2UI è un formato UI dichiarativo che l’agente può inviare. Puoi mettere A2UI sopra AG-UI. Puoi anche sovrapporre componenti renderizzati da tool personalizzati su AG-UI.

[json-render](https://github.com/vercel-labs/json-render) non è un prodotto di chat. È un catalogo di componenti e un’architettura di rendering: definisci i componenti che il modello può usare, fai emettere al modello un albero JSON valido e rendi quell’albero in modo sicuro.

[CopilotKit](https://github.com/CopilotKit/CopilotKit) non è solo una bolla di chat. È uno stack frontend per app native agente: UI di chat, UI generativa, stato condiviso, tool frontend e flussi human‑in‑the‑loop.

[OpenAI Apps SDK](https://developers.openai.com/apps-sdk) e [MCP Apps](https://github.com/MCP-UI-Org/mcp-ui) non sono strumenti per “rendere dinamica la mia app React”. Sono modelli di integrazione host per il rendering di widget all’interno di ChatGPT o di altri host compatibili con MCP.

I nomi sono fuorvianti perché lo spazio è ancora giovane. Le **layer** sono la parte che rimane utile.

## Il controllo sullo spettro

La UI generativa è un compromesso tra **controllo dello sviluppatore** e **libertà dell’agente**.

Troppo controllo e l’assistente sembra una palette di comandi con un costume. Troppa libertà e il modello inizia a inventare layout stravaganti, pulsanti vaghi, gerarchie visive rotte, stati impossibili e problemi di sicurezza con un sorriso compiaciuto.

Il trucco è scegliere la minima quantità di libertà necessaria a risolvere il problema dell’utente.

![Uno spettro dal rendering di componenti basati su tool alla generazione HTML aperta](../control-spectrum.webp)

Penso allo spettro in questo modo:

**Il rendering tool‑to‑component** è l’opzione predefinita più sicura. Il modello chiama `get_weather`, `search_products`, `compare_plans` o `draft_invoice`. L’app mappa quel risultato di tool a un componente che possiedi già: `WeatherCard`, `ProductGrid`, `PlanComparison`, `InvoiceReview`. Il modello decide *quando* l’interfaccia è utile. Gli sviluppatori mantengono il controllo su layout, styling, accessibilità, stati di caricamento, stati vuoti e azioni pericolose.

Questo è il pattern documentato nella [guida alle UI generative di Vercel AI SDK](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces): il modello chiama un tool, il tool restituisce dati e l’interfaccia rende un componente dal risultato. È anche il modello mentale alla base di molte implementazioni di CopilotKit e assistant‑ui.

**I cataloghi di componenti dichiarativi** concedono al modello più spazio. Invece di scegliere un singolo componente, il modello compone un albero da parti consentite. Un catalogo può includere `Metric`, `Table`, `Chart`, `FilterBar`, `ApprovalPanel` e `Timeline`. Il modello può assemblare una dashboard o un passaggio di workflow, ma non può eseguire codice arbitrario. Qui rientrano [A2UI](https://github.com/google/A2UI), [json‑render](https://github.com/vercel-labs/json-render), [Hashbrown](https://github.com/liveloveapp/hashbrown) e [OpenUI](https://github.com/thesysdev/openui).

**Le mini‑app in iframe** hanno senso quando l’interfaccia deve essere più ricca di un semplice albero di componenti, o quando un provider di tool remoto possiede l’esperienza. MCP Apps e OpenAI Apps SDK consentono a un tool di restituire dati strutturati più una risorsa widget che l’host rende in un iframe. È potente per mappe, carrelli della spesa, flussi di prenotazione, grafici e superfici di prodotto esterne. Crea anche un confine più difficile da attraversare tra l’app host e il widget.

**Open-ended generation** è l’estremo più lontano: l’agente emette HTML, SVG, Canvas, WebGL o altri artefatti simili a codice all’interno di un sandbox. [OpenGenerativeUI](https://github.com/CopilotKit/OpenGenerativeUI) è l’esempio più avanzato attuale: l’agente può generare visualizzazioni algoritmiche, scene 3D, diagrammi e simulazioni dentro iframe isolati. È ottimo per spiegazioni visive occasionali. Non è il punto di partenza consigliato per un flusso di approvazione aziendale.

È utile dare un nome alla distinzione chiave: **iframe HTML** (il modello scrive codice in un sandbox) vs. **catalogo JSON** (il modello emette una specifica strutturata e il tuo renderer la mappa su componenti pre‑costruiti). Sembrano correlati, ma hanno profili di rischio e complessità molto diversi. L’iframe HTML è massimamente espressivo; il confine dell’iframe si occupa del lavoro di sicurezza. Il catalogo JSON non concede al modello libertà eseguibile — può solo fare riferimento a tipi di componente definiti in anticipo. La maggior parte dei framework in questo ambito ricade chiaramente in uno dei due gruppi.

**Oltre il sandbox**: demo molto recenti suggeriscono la formazione di un quarto modo — LLM che guidano esperienze tipo gioco o immersive controllando l’output visivo più direttamente di quanto consenta qualsiasi catalogo di componenti. Progetti che generano mondi 3D esplorabili da prompt, comportamenti NPC diretti da LLM a runtime, e inferenza di modello in‑browser via WebGPU ([WebLLM](https://mlc.ai/web-llm/)) sono segnali iniziali. Non esistono ancora framework stabili per costruire prodotti in produzione su questa base. Tratterò questa direzione in un articolo dedicato non appena cambierà la situazione.

## Componenti ad alto livello vs componenti granulari

Questa è la decisione di progettazione più importante.

Se il tuo catalogo è troppo granulare, il modello deve diventare un ingegnere frontend:

```tsx
Container
Row
Column
Text
Button
Icon
Spacer
Divider
```

Sembra flessibile, ma ora il modello deve decidere spaziatura, gerarchia, raggruppamento, stati vuoti, etichette dei pulsanti, gestione degli errori e comportamento responsivo. Hai anche ingrandito il prompt e reso l'output più facile da spezzare.

Se il tuo catalogo è troppo ad alto livello, il modello è intrappolato:

```tsx
WeatherCard
StockCard
HotelCard
```

È sicuro, ma funziona solo per scenari noti. Il modello non può creare una matrice di confronto, chiedere input mancanti o adattare l'architettura dell'informazione quando la domanda dell'utente cambia.

Il punto medio utile è **componenti a livello di dominio con slot limitati**:

```tsx
SearchResults
ComparisonTable
MetricGroup
EditablePlan
ApprovalRequest
Timeline
DataCollectionForm
CheckoutReview
```

Questi componenti codificano il gusto del prodotto e le restrizioni di business. Il modello decide *cosa mostrare*, ma non ogni decisione CSS.

Ad esempio, un agente di viaggio non ha bisogno di `div`, `span` e `button`. Ha bisogno di:

- `TripSummary`
- `FlightOptionList`
- `HotelComparison`
- `TravelerForm`
- `PolicyNotice`
- `BookingConfirmation`

Un agente finanziario non ha bisogno di un playground di grafici generico. Ha bisogno di:

- `PortfolioSnapshot`
- `TransactionTable`
- `RiskBreakdown`
- `ScenarioComparison`
- `ApprovalGate`

Il catalogo dovrebbe suonare come il tuo prodotto, non come HTML.

## Cosa usare per quale prodotto

Ecco la matrice di raccomandazione che userei realmente con un team.

**Stai aggiungendo un assistente a un’app SaaS esistente.**

Inizia con il rendering tool‑to‑component. Usa [Vercel AI SDK UI](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces), [assistant-ui](https://github.com/assistant-ui/assistant-ui) o [CopilotKit](https://github.com/CopilotKit/CopilotKit), a seconda di quanto stato agente e integrazione di strumenti frontend ti servono. Mantieni il catalogo piccolo all’inizio. Renderizza componenti di prodotto già fidati.

**Stai costruendo un copilot serio in‑app che richiede stato condiviso.**

Guarda attentamente CopilotKit insieme ad AG‑UI. La caratteristica importante non è la “chat”. È lo stato condiviso e l’interazione bidirezionale: l’agente può richiedere input, renderizzare UI, aggiornare lo stato e attendere l’approvazione.

**Hai agenti remoti che devono inviare UI attraverso un confine.**

Usa A2UI o un protocollo dichiarativo simile ad A2UI. L’intero scopo è che un agente remoto possa descrivere la UI come dati mentre l’host mantiene il controllo del rendering nativo, della sicurezza e dello stile. Se ti serve anche un’interazione live agente/app, eseguila tramite AG‑UI o qualunque trasporto il tuo ambiente standardizzi.

**Stai costruendo all’interno di ChatGPT o di un host compatibile MCP.**

Usa MCP Apps e il percorso Apps SDK. La documentazione attuale di OpenAI consiglia il bridge `ui/*` di MCP Apps per i nuovi progetti, mantenendo `window.openai` come livello di compatibilità e superficie di estensione opzionale. Copia anche la loro divisione tra strumenti di dati e strumenti di rendering: lascia che il modello recuperi e ragioni sui dati prima di decidere di renderizzare un widget.

**Vuoi dashboard, report o form in linguaggio naturale nella tua app.**

Prova json-render, Hashbrown o OpenUI. La chiave è il catalogo. Se esponi `LineChart`, `DataTable`, `MetricGroup`, `FilterControl` e `InsightCallout`, il modello può assemblare superfici di reporting utili senza avvicinarsi a codice arbitrario.

**Vuoi artefatti educativi, visuali o altamente personalizzati.**

Usa un sandbox a apertura libera come OpenGenerativeUI. Consenti al modello di generare SVG, Canvas, WebGL o HTML autonomo, ma tratta l’output come contenuto non attendibile dell’utente. Isolalo, limita le dimensioni, rimuovi i permessi e tienilo lontano dallo stato privilegiato dell’app.

**Hai principalmente bisogno di markdown in streaming più gradevole con qualche affordance inline.**

Non eccedere. Gli strumenti di rendering llm-ui o assistant-ui potrebbero bastare.

## Gli errori che eviterei


**Errore 1: Consentire al modello di scrivere React di produzione a runtime.**

Ci sono eccezioni, ma per le interfacce di prodotto è generalmente la scelta sbagliata. La generazione di codice a runtime è difficile da mettere in sicurezza, difficile da testare, difficile da tematizzare e difficile da mantenere accessibile. Se il modello può portare a termine il compito scegliendo tra componenti fidati, fallo così.

**Errore 2: Esporre primitive di design invece di primitive di prodotto.**

Quando fornisci al modello `Row`, `Column`, `Text` e `Button`, gli chiedi di diventare il tuo sistema di design. Ne uscirà uno mediocre. Fornisci sostantivi di prodotto di livello più alto.

**Errore 3: Pensare che JSON valido significhi UI sicura.**

Un payload può superare la validazione dello schema e comunque essere manipolativo o pericoloso. L’etichetta può dire "Visualizza fattura" mentre l’azione archivia l’account. Tratta le specifiche UI come comportamento, non come decorazione. Necessitano di test di policy, controlli semantici e conferma umana per azioni con conseguenze.

**Errore 4: Inserire la logica di business negli strumenti di rendering.**

Gli strumenti di rendering devono solo renderizzare. Gli strumenti di dati devono recuperare, calcolare, modificare e validare. La documentazione dell'OpenAI Apps SDK sottolinea questa separazione per un motivo: se ogni strumento di dati trascina con sé un widget, il modello perde spazio per ragionare prima di presentare.

**Errore 5: Ottimizzare per la novità invece che per il completamento del compito.**

Lo scopo non è trasformare ogni risposta in un’interfaccia scintillante. Lo scopo è ridurre l’attrito. Un pannello di approvazione stabile e noioso che fa risparmiare all'utente quattro minuti è migliore di una dashboard generata e appariscente che non può essere considerata affidabile nemmeno una volta.

## Un'architettura pratica

Se dovessi avviare un nuovo prodotto oggi, adotterei un approccio a fasi:

1.  **Distribuisci prima l’interfaccia controllata dallo strumento.** Mappa gli strumenti noti ai componenti noti. Registra ogni chiamata allo strumento, ogni rendering UI e ogni azione dell’utente.  
2.  **Aggiungi un catalogo di dominio.** Quando i pattern si ripetono, espone `ComparisonTable`, `DecisionPanel`, `DataCollectionForm`, `Timeline` e altri componenti specifici del prodotto.  
3.  **Standardizza il trasporto solo quando necessario.** Se controlli sia il frontend sia il backend, lo streaming semplice può bastare. Se utilizzi più framework per gli agenti, usa AG-UI. Se gli strumenti attraversano i confini del prodotto, usa MCP. Se gli agenti attraversano i confini organizzativi, tieni d’occhio A2A e A2UI.  
4.  **Usa widget iframe per superfici esterne o complesse.** Mappe, carrelli, flussi di prenotazione e mini‑app di terze parti vanno dietro a un confine.  
5.  **Riserva la generazione aperta per gli artefatti.** Diagrammi, simulazioni, spiegazioni temporanee e lavagne visive sono un ottimo caso d’uso. I flussi di lavoro principali no.  

L’architettura finisce per apparire così:

```txt
User intent
  -> agent runtime
  -> tool/data calls
  -> structured result
  -> UI decision
  -> trusted component, declarative spec, or sandboxed widget
  -> user action
  -> state/event stream back to the agent
```

Quel ciclo è il vero prodotto. La casella di chat è solo uno dei possibili dispositivi di input.  

## La valutazione deve includere l’interfaccia

I team LLM stanno lentamente imparando a valutare prompt e output del modello. L’interfaccia generativa aggiunge un’altra superficie: l’interfaccia stessa può essere errata.  

Come minimo, conserva questi artefatti per ogni UI generata:

- contesto del prompt e dello strumento  
- chiamate allo strumento e risultati dello strumento  
- specifica UI generata o selezione del componente  
- nome del componente renderizzato e relative proprietà  
- etichette visibili all’utente  
- azioni associate a pulsanti/moduli  
- aggiornamenti di stato visibili al modello provenienti dall’interfaccia  
- cronologia delle azioni dell’utente  

Quindi definisci controlli del tipo:

- ogni azione distruttiva deve avere un componente di conferma  
- le etichette dei pulsanti devono corrispondere alla semantica dell’azione  
- le specifiche di rendering possono fare riferimento solo a componenti consentiti  
- i totali visibili all’utente devono corrispondere ai totali dei risultati dello strumento  
- i moduli non devono richiedere campi al di fuori dell’ambito del compito  
- i widget non devono ricevere segreti che servono solo al modello  
- i metadati nascosti non devono contraddire le etichette visibili  

Sembra noioso. È anche il punto in cui nasce la fiducia in produzione.

## Una nota sulla stabilità dei progetti  

Ogni protocollo importante in questo spazio è ancora pre‑1.0. Ultima verifica 8 maggio 2026; prevedi cambiamenti e controlla la documentazione corrente prima di puntare su una piattaforma.  

**Vercel AI SDK RSC** — la funzionalità originale “Generative UI” — ha sospeso lo sviluppo nell’ottobre 2024 ([Discussion #3251](https://github.com/vercel/ai/discussions/3251)) a causa di limitazioni architetturali senza una soluzione a breve termine. **json-render** (Vercel Labs) è emerso come direzione sostitutiva: basato su catalogo, indipendente dal framework, senza accoppiamento RSC. Sembra aver guadagnato rapidamente l’attenzione degli sviluppatori web dal lancio all’inizio del 2026. Il motivo più probabile è la DX: json-render funziona subito in un progetto React standard; l’ambito cross‑platform di A2UI aggiunge frizione di configurazione.  

**A2UI** (Google) è pre‑1.0, con cambiamenti breaking tra versioni minori e comunicazioni di roadmap incoerenti. Il suo vantaggio è la reale copertura cross‑platform (web, Flutter, SwiftUI) che json-render non affronta. Per casi d’uso puramente web oggi, json-render sembra offrire una copertura di tooling migliore; per scenari cross‑platform o agenti remoti, il design di A2UI è più adeguato. Una convergenza tra le due specifiche è possibile — Vercel ha sperimentato output compatibile A2UI da json-render.  

**AG-UI** (CopilotKit) è anch'esso pre‑1.0. La confusione più comune riguarda il nome: AG-UI è un protocollo di trasporto, non un framework UI. Definisce *come* gli eventi fluiscono tra agente e frontend; cosa renderizzi rimane una tua decisione. Il concetto è solido e ampiamente adottato. La specifica è ancora in evoluzione.

## My Take

La Generative UI non sostituirà interfacce di prodotto accuratamente progettate. Sostituirà l’assunzione pigra che una trascrizione di chat sia l’interfaccia universale per l’AI.

I migliori sistemi non lasceranno il modello improvvisare tutto. Gli forniranno un piccolo, preciso set di blocchi costruttivi nativi del prodotto; una connessione runtime affidabile; confini di sicurezza chiari; e sufficiente libertà per adattare l’interfaccia al compito.

Il futuro non è “il modello scrive il tuo frontend”.

Il futuro è più vicino a: **il tuo frontend diventa uno strumento che l’agente può suonare, ma sei tu a decidere cosa lo strumento è autorizzato a produrre.**
