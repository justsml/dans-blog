チャットは補助輪だった。

第一世代のLLMアプリは、ほとんどがテキストボックスを製品に貼り付けただけのものだった。モデルは散文を返し、フロントエンドはマークダウンをレンダリングした。ユーザーがアクションを起こす必要がある場合、アシスタントは「他の場所にあるボタンをクリックしてください」と説明するだけだった。

デモとしてはそれで十分だった。しかし、これが進むべき方向ではない。

次の有用なステップは**生成UI**である。モデルは単にテキストで答えるのではなく、ユーザーが今必要としているインターフェースを決定する手助けをする。時にはツールを呼び出して事前に作られたカードをレンダリングすることもある。時には既知のワークフローコンポーネントに新しいデータを流し込むことだ。時には一時的なダッシュボード、フォーム、比較表、チャート、インタラクティブウィジェットを構成することもある。

残念ながら、「生成UI」という言葉は、朝食前に五つの異なる意味を持つようなフレーズになってしまった。

人々はこれを次のように使っている：

- モデルが開発者定義のReactコンポーネントから選択する
- フロントエンドがネイティブコンポーネントにレンダリングするJSON仕様
- MCPツールから返されるiframeアプリ
- ツール呼び出しをサポートするチャットUIライブラリ
- バックエンドとフロントエンド間で状態をストリーミングするエージェントプロトコル
- v0、Lovable、Bolt、Cursorのようなデザイン時コード生成ツール
- モデルが実行時に文字通りHTML、SVG、Canvas、Reactを書き出すこと

これらは関連しているが、同じレイヤーではない。これらを混同すると、あらゆるアーキテクチャの議論がスープ状態になる。

## 中心的な誤解

最大の間違いは、「生成UI」を単一の技術選択として扱うことだ。

問題を4つのレイヤーに分割するのが良い。

1.  **プロダクトシェル**：ユーザーが触れるもの。チャット、サイドバーコパイロット、ダッシュボード、ワークフロービルダー、IDEパネル、ChatGPTアプリ、モバイル画面、サポートコンソールなど。
2.  **UI構成モデル**：モデルが使用を許可される文法。ツール呼び出し、JSON、A2UI、json-render、OpenUI Lang、Hashbrownコンポーネント選択、サンドボックス化されたHTMLなど。
3.  **ランタイムとトランスポート**：メッセージ、ツール呼び出し、状態差分、ユーザーアクション、UI成果物がエージェントとフロントエンド間でどのように移動するか。AG-UI、MCP Apps、Apps SDK、A2A、SSE、WebSocket、そしてプレーンなHTTPがこのあたりに存在する。
4.  **エージェントとツールバックエンド**：LangGraph、Google ADK、CrewAI、Mastra、LlamaIndex、Pydantic AI、Agno、OpenAI Agents SDK、カスタム関数、データベース、検索、そして依然として正しくなければならない退屈なビジネスロジック。

レイヤーを分割すると、エコシステムはずっと神秘性を失う。

[AG-UI](https://github.com/ag-ui-protocol/ag-ui) は [A2UI](https://github.com/google/A2UI) の真の競合ではない。AG-UIはエージェントとアプリケーション間のインタラクションのためのイベントプロトコルである。A2UIはエージェントが送信できる宣言的UIフォーマットである。A2UIをAG-UIの上に載せることもできる。カスタムツールレンダリングコンポーネントをAG-UIの上に載せることもできる。

[json-render](https://github.com/vercel-labs/json-render) はチャット製品ではない。コンポーネントカタログとレンダラーアーキテクチャである：モデルが使用できるコンポーネントを定義し、モデルに有効なJSONツリーを出力させ、そのツリーを安全にレンダリングする。

[CopilotKit](https://github.com/CopilotKit/CopilotKit) は単なるチャットバブルではない。エージェントネイティブアプリのためのフロントエンドスタックである：チャットUI、生成UI、共有状態、フロントエンドツール、ヒューマンインザループフロー。

[OpenAI Apps SDK](https://developers.openai.com/apps-sdk) や [MCP Apps](https://github.com/MCP-UI-Org/mcp-ui) は「Reactアプリを動的にする」ツールではない。ChatGPTや他のMCP互換ホスト内でウィジェットをレンダリングするためのホスト統合モデルである。

名前が紛らわしいのは、この領域がまだ若いからだ。レイヤーこそが、長く使える部分である。

## 制御のスペクトラム

生成UIは**開発者の制御**と**エージェントの自由度**のトレードオフである。

制御が強すぎると、アシスタントは仮面をかぶったコマンドパレットのように感じられる。自由度が高すぎると、モデルは奇妙なレイアウト、曖昧なボタン、壊れた視覚階層、ありえない状態、そして自信満々の小さな笑みを浮かべたセキュリティ問題をでっち上げ始める。

コツは、ユーザーの問題を解決するのに必要な最小限の自由度を選ぶことだ。

![ツールレンダリングコンポーネントから自由生成HTMLまでのスペクトラム](../control-spectrum.webp)

このスペクトラムは次のように考えている。

**ツール→コンポーネントレンダリング**は最も安全なデフォルトだ。モデルは `get_weather`、`search_products`、`compare_plans`、`draft_invoice` を呼び出す。アプリはそのツール結果を、すでに持っているコンポーネント（`WeatherCard`、`ProductGrid`、`PlanComparison`、`InvoiceReview`）にマッピングする。モデルはUIが役立つ*タイミング*を決める。レイアウト、スタイリング、アクセシビリティ、ローディング状態、空状態、危険なアクションは、依然として開発者が所有する。

これは[Vercel AI SDKの生成UIガイド](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces)に記載されているパターンである。モデルがツールを呼び出し、ツールがデータを返し、UIがその結果からコンポーネントをレンダリングする。多くのCopilotKitやassistant-uiの実装の背後にある考え方でもある。

**宣言的コンポーネントカタログ**はモデルにより多くの余地を与える。モデルは1つのコンポーネントを選ぶ代わりに、許可された部品からツリーを構成する。カタログには `Metric`、`Table`、`Chart`、`FilterBar`、`ApprovalPanel`、`Timeline` などが含まれる。モデルはダッシュボードやワークフローステップを組み立てられるが、任意のコードを実行することはできない。ここに[A2UI](https://github.com/google/A2UI)、[json-render](https://github.com/vercel-labs/json-render)、[Hashbrown](https://github.com/liveloveapp/hashbrown)、[OpenUI](https://github.com/thesysdev/openui)が位置する。

**iframeミニアプリ**は、UIがコンポーネントツリーよりもリッチである必要がある場合、またはリモートツールプロバイダーが体験を所有する場合に有効だ。MCP AppsとOpenAI Apps SDKは、ツールが構造化データと、ホストがiframe内でレンダリングするウィジェットリソースを返すことを可能にする。これはマップ、ショッピングカート、予約フロー、チャート、外部プロダクトサーフェスに強力だ。同時に、ホストアプリとウィジェットの間に明確な境界線が生まれる。

**オープンエンドな生成**は極端なケースだ。エージェントがHTML、SVG、Canvas、WebGL、その他コードに近い成果物をサンドボックス内に出力する。[OpenGenerativeUI](https://github.com/CopilotKit/OpenGenerativeUI)が現時点で最も良い例だ。エージェントはサンドボックス化されたiframe内で、アルゴリズム可視化、3Dシーン、図、シミュレーションを生成できる。これは一回限りの視覚的説明には優れている。しかし、エンタープライズの承認フローで最初に選ぶ手段ではない。

ここで重要な区別を明確にしておく。**iframe HTML**（モデルがコードをサンドボックスに書き込む）と**JSONカタログ**（モデルが構造化された仕様を出力し、レンダラが事前定義されたコンポーネントにマッピングする）だ。これらは似ているように聞こえるが、リスクと複雑性のプロファイルは大きく異なる。iframe HTMLは表現力が最大限に高く、iframeの境界がセキュリティを担う。JSONカタログはモデルに実行可能な自由度を与えず、事前に定義したコンポーネントタイプしか参照できない。この領域のフレームワークのほとんどは、どちらかの陣営に明確に分類される。

**サンドボックスを超えて**: ごく最近のデモは、第4のモードが形成されつつあることを示唆している。LLMが、どのコンポーネントカタログよりも直接的に視覚出力を制御することで、ゲーム的または没入型の体験を駆動するものだ。プロンプトから探索可能な3Dワールドを生成するプロジェクト、実行時にLLMが指示するNPCの振る舞い、WebGPUを介したブラウザ内モデル推論（[WebLLM](https://mlc.ai/web-llm/)）は、初期の兆候である。まだプロダクションワークを構築するための安定したフレームワークは存在しない。状況が変わったら、この方向性については専用の記事で取り上げる。

## 高レベルコンポーネント vs 細粒度コンポーネント

これが最も重要な設計判断だ。

カタログが細かすぎると、モデルはフロントエンドエンジニアにならざるを得なくなる。

```tsx
Container
Row
Column
Text
Button
Icon
Spacer
Divider
```

一見柔軟に見えるが、これではモデルがスペーシング、階層、グルーピング、空状態、ボタンラベル、エラー処理、レスポンシブ動作をすべて決定しなければならない。プロンプトは肥大化し、出力は壊れやすくなる。

カタログが高レベルすぎると、モデルは閉じ込められる：

```tsx
WeatherCard
StockCard
HotelCard
```

安全ではあるが、既知のシナリオでしか機能しない。モデルは比較マトリックスを作ったり、不足している入力を要求したり、ユーザーの質問が変わったときに情報アーキテクチャを適応させたりできない。

有用な中間点は、**制約付きスロットを持つドメインレベルのコンポーネント**である：

```tsx
SearchResults
ComparisonTable
MetricGroup
EditablePlan
ApprovalRequest
Timeline
DataCollectionForm
CheckoutReview
```

これらのコンポーネントは、プロダクトのセンスとビジネス上の制約をエンコードする。モデルは *何を表示すべきか* を決定できるが、CSSの細かい決定までは任されない。

例えば、旅行エージェントに `div`、`span`、`button` は必要ない。必要なのは：

- `TripSummary`
- `FlightOptionList`
- `HotelComparison`
- `TravelerForm`
- `PolicyNotice`
- `BookingConfirmation`

金融エージェントに汎用的なチャート作成の遊び場は必要ない。必要なのは：

- `PortfolioSnapshot`
- `TransactionTable`
- `RiskBreakdown`
- `ScenarioComparison`
- `ApprovalGate`

カタログはあなたのプロダクトのように聞こえるべきであり、HTMLのように聞こえるべきではない。

## どのプロダクトに何を使うか

以下は、実際にチームで使うであろう推奨マトリックスです。

**既存のSaaSアプリにアシスタントを追加する場合。**

ツール→コンポーネントレンダリングから始めましょう。必要なエージェント状態とフロントエンドツール統合の度合いに応じて、[Vercel AI SDK UI](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces)、[assistant-ui](https://github.com/assistant-ui/assistant-ui)、または[CopilotKit](https://github.com/CopilotKit/CopilotKit)を使用してください。最初はカタログを小さく保ちます。既に信頼しているプロダクトコンポーネントをレンダリングしてください。

**共有状態が必要な本格的なアプリ内コパイロットを構築する場合。**

CopilotKitとAG-UIの組み合わせを真剣に検討してください。重要な機能は「チャット」ではありません。共有状態と双方向のインタラクションです。エージェントが入力を要求し、UIをレンダリングし、状態を更新し、承認のために一時停止できることです。

**リモートエージェントが境界を越えてUIを送信する必要がある場合。**

A2UIまたはA2UIに類似した宣言的プロトコルを使用してください。要点は、リモートエージェントがUIをデータとして記述できる一方で、ホストがネイティブレンダリング、セキュリティ、スタイルの制御を維持することです。エージェントとアプリのライブインタラクションも必要な場合は、AG-UIまたは環境で標準化されたトランスポート上で実行してください。

**ChatGPTまたはMCP互換ホスト内で構築している場合。**

MCP AppsとApps SDKのパスを使用してください。OpenAIの現在のドキュメントでは、新しい作業にはMCP Appsの`ui/*`ブリッジを推奨し、`window.openai`を互換性レイヤーおよびオプションの拡張面として維持しています。また、データツールとレンダリングツールの分割を真似てください。モデルがウィジェットをレンダリングする前にデータを取得して推論できるようにします。

**自社アプリ内で自然言語によるダッシュボード、レポート、フォームを実現したい場合。**

json-render、Hashbrown、またはOpenUIを試してみてください。鍵はカタログです。`LineChart`、`DataTable`、`MetricGroup`、`FilterControl`、`InsightCallout`を公開すれば、モデルは任意のコードに近づくことなく有用なレポート画面を組み立てられます。

**教育用、ビジュアル、または高度にカスタマイズされた成果物が必要な場合。**

OpenGenerativeUIのようなオープンエンドなサンドボックスを使用してください。モデルにSVG、Canvas、WebGL、または自己完結型のHTMLを書かせますが、出力は信頼できないユーザーコンテンツとして扱います。サンドボックス化し、サイズを制限し、権限を剥奪し、特権的なアプリ状態から遠ざけてください。

**主に、いくつかのインラインアフォーダンスを備えた、より見栄えの良いストリーミングマークダウンが必要な場合。**

過剰に構築しないでください。llm-uiやassistant-uiのツールレンダリングで十分かもしれません。

## 避けるべき過ち

**間違い1: モデルに本番Reactをランタイムで書かせること。**

例外はあるが、プロダクトUIにおいては通常、間違ったデフォルトである。ランタイムでのコード生成は、セキュリティ確保、テスト、テーマ適用、アクセシビリティ維持が難しい。モデルが信頼できるコンポーネントを選択することで仕事を完了できるなら、そうすべきだ。

**間違い2: プロダクトプリミティブではなくデザインプリミティブを公開すること。**

モデルに`Row`、`Column`、`Text`、`Button`を与えると、モデルにデザインシステムになるよう求めていることになる。それは凡庸なものになる。より高レベルのプロダクト名詞を与えよ。

**間違い3: 有効なJSONが安全なUIを意味すると考えること。**

ペイロードはスキーマ検証を通過しても、操作や危険を含む可能性がある。ラベルが「請求書を表示」と表示していても、アクションがアカウントをアーカイブするかもしれない。UI仕様は装飾ではなく振る舞いとして扱うべきだ。ポリシーテスト、セマンティックチェック、そして重要なアクションに対する人間の確認が必要である。

**間違い4: ビジネスロジックをレンダーツールに押し込むこと。**

レンダーツールはレンダリングに徹すべきだ。データツールは取得、計算、変更、検証を行う。OpenAIのApps SDKドキュメントがこの分割をわざわざ指摘しているのには理由がある。すべてのデータツールにウィジェットがくっついてくると、モデルは提示する前に考える余地を失う。

**間違い5: タスク完了ではなく新規性を最適化すること。**

目的はすべての回答を個性的なインターフェースにすることではない。摩擦を減らすことだ。安定していて退屈な承認パネルがユーザーの4分を節約するほうが、2回目には信頼できない派手な生成ダッシュボードよりはるかに優れている。

## 実践的なアーキテクチャ

もし今日新たなプロダクトを始めるとしたら、私は段階的アプローチを取るだろう。

1.  **まずは制御されたツールUIをリリースする。** 既知のツールを既知のコンポーネントにマッピングする。すべてのツール呼び出し、UIレンダリング、ユーザーアクションをログに記録する。
2.  **ドメインカタログを追加する。** パターンが繰り返されるようになったら、`ComparisonTable`、`DecisionPanel`、`DataCollectionForm`、`Timeline`、その他のプロダクト固有のコンポーネントを公開する。
3.  **必要な場合にのみトランスポートの標準化を追加する。** フロントエンドとバックエンドの両方を自分たちで管理しているなら、単純なストリーミングで十分かもしれない。複数のエージェントフレームワークがあるならAG-UIを使う。ツールがプロダクトの境界を越えるならMCPを使う。エージェントが組織の境界を越えるならA2AとA2UIを検討する。
4.  **外部または複雑なサーフェスにはiframeウィジェットを使う。** マップ、カート、予約フロー、サードパーティのミニアプリは境界の内側に配置する。
5.  **オープンエンドな生成はアーティファクトに限定する。** 図、シミュレーション、一時的な説明、ビジュアルスクラッチパッドは最適だが、コアワークフローには向かない。

アーキテクチャは最終的に次のようになる：

```txt
User intent
  -> agent runtime
  -> tool/data calls
  -> structured result
  -> UI decision
  -> trusted component, declarative spec, or sandboxed widget
  -> user action
  -> state/event stream back to the agent
```

このループこそが本当のプロダクトだ。チャットボックスは単なる入力デバイスの一つに過ぎない。

## 評価にはUIを含めるべき

LLMチームは徐々にプロンプトとモデル出力の評価を学んでいる。Generative UIは新たな表面領域を追加する。インターフェース自体が間違っている可能性がある。

最低限、生成されたUIごとに以下のアーティファクトを保存する：

- プロンプトとツールコンテキスト
- ツール呼び出しとツール結果
- 生成されたUI仕様またはコンポーネント選択
- レンダリングされたコンポーネント名とプロパティ
- ユーザーに表示されるラベル
- ボタン/フォームに紐づくアクション
- UIからのモデル可視状態更新
- ユーザーアクション履歴

そして以下のようなチェックを書く：

- すべての破壊的アクションには確認コンポーネントが必要
- ボタンラベルはアクションの意味と一致しなければならない
- レンダリング仕様は許可されたコンポーネントのみ参照可能
- ユーザーに表示される合計はツール結果の合計と一致しなければならない
- フォームはタスク範囲外のフィールドを要求してはならない
- ウィジェットはモデルだけが必要とする機密情報を受け取ってはならない
- 隠しメタデータは表示ラベルと矛盾してはならない

これは面倒に聞こえる。しかし、本番環境の信頼はここから生まれる。

## プロジェクトの安定性に関する注意

この分野の主要プロトコルはすべて 1.0 未満です。最終確認日：2026 年 5 月 8 日。変更を前提に計画し、プラットフォームを決める前に最新のドキュメントを確認してください。

**Vercel AI SDK RSC** — 当初「Generative UI」の目玉機能 — は、2024 年 10 月に開発が一時停止されました（[Discussion #3251](https://github.com/vercel/ai/discussions/3251)）。アーキテクチャ上の制約に近々の修正が見込めなかったためです。**json-render**（Vercel Labs）が代替の方向性として登場しました。カタログベース、フレームワーク非依存、RSC との結合なし。2026 年初頭のリリース以降、Web 開発者の注目を急速に集めているようです。理由はおそらく DX にあります。json-render は標準的な React プロジェクトですぐに動作します。一方、A2UI のクロスプラットフォーム対応はセットアップの摩擦を生みます。

**A2UI**（Google）は 1.0 未満で、マイナーバージョン間で破壊的変更があり、ロードマップのコミュニケーションに一貫性がありません。利点は json-render がカバーしない真のクロスプラットフォーム対応（Web、Flutter、SwiftUI）です。現在の純粋な Web ユースケースでは、json-render の方がツールのカバレッジが優れているように見えます。クロスプラットフォームやリモートエージェントのシナリオでは、A2UI の設計の方が適切です。両仕様の収束は可能です。Vercel は json-render から A2UI 互換の出力を実験しています。

**AG-UI**（CopilotKit）も 1.0 未満です。最もよくある混乱は名前です。AG-UI はトランスポートプロトコルであり、UI フレームワークではありません。エージェントとフロントエンド間でイベントが*どのように*流れるかを定義します。何をレンダリングするかは依然としてあなたの判断です。コンセプトは堅牢で広く採用されています。仕様はまだ進化中です。

## 私の見解

生成UIは、慎重に設計されたプロダクトインターフェースを置き換えることはありません。置き換えるのは、チャットのトランスクリプトがAIの普遍的なインターフェースであるという怠惰な前提です。

優れたシステムは、モデルにすべてを自由にやらせることはしません。代わりに、プロダクト固有の小さく鋭いビルディングブロック、信頼性の高いランタイム接続、明確なセキュリティ境界、そしてタスクに合わせてインターフェースを適応させるための十分な自由を与えます。

未来は「モデルがフロントエンドを書く」ことではありません。未来はむしろこうです：**あなたのフロントエンドはエージェントが演奏できる楽器となるが、その楽器がどんな音を出してよいかは依然としてあなたが決める。**
