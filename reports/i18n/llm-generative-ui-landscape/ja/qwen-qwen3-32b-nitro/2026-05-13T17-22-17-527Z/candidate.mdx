チャットは補助輪に過ぎなかった。  

最初の世代のLLMアプリは、大抵テキストボックスが製品に貼り付けられたような見た目だった。モデルは散文を返し、フロントエンドはマークダウンをレンダリングした。ユーザーがアクションを取る必要があると、アシスタントはユーザーがどこか別の場所でクリックすべきボタンを説明した。  

これはデモには十分だった。しかし、これが最終的な方向ではない。  

次の重要なステップは**生成型UI**だ。モデルは単にテキストで応答するだけでなく、ユーザーが今必要なインターフェースを決定する。場合によってはツールを呼び出して事前に作成されたカードをレンダリングする。場合によっては既知のワークフロー部品に新鮮なデータを詰め込む。場合によっては一時的なダッシュボード、フォーム、比較表、チャート、インタラクティブなウィジェットを構成する。  

残念ながら、「生成型UI」というフレーズは、朝食前には5通りの異なる意味を持つことになる。  

人々はこれを次のように説明するために使う：

- モデルが開発者が定義したReactコンポーネントから選択する  
- フロントエンドがネイティブコンポーネントにレンダリングするJSON仕様  
- MCPツールから返されるiframeアプリ  
- ツール呼び出しをサポートするチャットUIライブラリ  
- バックエンドとフロントエンド間で状態をストリーミングするエージェントプロトコル  
- v0、Lovable、Bolt、Cursorなどの設計時コードジェネレータ  
- ランタイムで実際にHTML、SVG、Canvas、Reactを生成するモデル  

これらは関連しているが、同じレイヤーではない。これらを混同すると、すべてのアーキテクチャ議論が混乱する。

## コアな誤解  

最大の間違いは「生成型UI」を1つの技術選択として扱うことだ。

問題を4つのレイヤーに分けるのが効果的です：

1.  **製品シェル**: ユーザーが直接触れるインターフェース。チャット、サイドバーのコパイロット、ダッシュボード、ワークフロー構築ツール、IDEパネル、ChatGPTアプリ、モバイル画面、サポートコンソールなどが該当します。
2.  **UI構成モデル**: モデルが利用可能な表現ルール。ツール呼び出し、JSON、A2UI、json-render、OpenUI Lang、Hashbrownコンポーネント選択、またはサンドボックスHTMLなどが含まれます。
3.  **ランタイムと通信プロトコル**: エージェントとフロントエンド間でメッセージ、ツール呼び出し、状態変更、ユーザー操作、UIアーティファクトをやり取りする方法。AG-UI、MCP Apps、Apps SDK、A2A、SSE、WebSocket、またはシンプルなHTTPがこのレイヤーに位置付けられます。
4.  **エージェントとツールバックエンド**: LangGraph、Google ADK、CrewAI、Mastra、LlamaIndex、Pydantic AI、Agno、OpenAIエージェントSDK、カスタム関数、データベース、検索ロジック、そして依然として正確でなければならないすべての業務ロジックが含まれます。

レイヤーを分離すると、エコシステムははるかに明確になります。

[AG-UI](https://github.com/ag-ui-protocol/ag-ui)は[A2UI](https://github.com/google/A2UI)の競合ではありません。AG-UIはエージェントとアプリケーション間のイベントプロトコルです。A2UIはエージェントが送信可能な宣言型UIフォーマットです。A2UIをAG-UI上に構築できます。カスタムツールレンダリングコンポーネントもAG-UI上に構築可能です。

[json-render](https://github.com/vercel-labs/json-render)はチャット製品ではありません。これはコンポーネントカタログとレンダラーアーキテクチャです：モデルが利用可能なコンポーネントを定義し、モデルが有効なJSONツリーを出力し、そのツリーを安全にレンダリングします。

[CopilotKit](https://github.com/CopilotKit/CopilotKit)は単なるチャットバブルではありません。これはエージェントネイティブアプリのためのフロントエンドスタックです：チャットUI、ジェネレーティブUI、共有状態、フロントエンドツール、そして人間がループ内に含まれるフローを含みます。

[OpenAI Apps SDK](https://developers.openai.com/apps-sdk) と [MCP Apps](https://github.com/MCP-UI-Org/mcp-ui) は「私のReactアプリを動的にする」ためのツールではありません。これらは、ChatGPTや他のMCP互換ホスト内でウィジェットをレンダリングするためのホスト統合モデルです。

この分野がまだ若いため、名称が混乱しています。有用なのはレイヤーの部分です。

## 制御のスペクトル

ジェネレーティブUIは、**開発者の制御**と**エージェントの自由**のトレードオフです。

制御が過剰になると、アシスタントは仮装したコマンドパレットのように感じられます。自由が過剰になると、モデルは奇妙なレイアウトや曖昧なボタン、破損した視覚的階層、不可能な状態、セキュリティ問題を自信満々の笑みとともに生成し始めます。

ユーザーの問題を解決する最小限の自由を選ぶことがコツです。

![ツールレンダリングコンポーネントからオープンエンド生成HTMLへのスペクトル](../control-spectrum.webp)

私はこのスペクトルをこう考えます：

**ツールからコンポーネントへのレンダリング**は最も安全なデフォルトです。モデルが`get_weather`、`search_products`、`compare_plans`、または`draft_invoice`を呼び出します。アプリはそのツールの結果を、すでに所有しているコンポーネント（`WeatherCard`、`ProductGrid`、`PlanComparison`、`InvoiceReview`）にマップします。モデルがUIが有用なタイミングを決定します。開発者はレイアウト、スタイル、アクセシビリティ、ロード状態、空状態、危険なアクションを依然として所有しています。

これは[Vercel AI SDKのジェネレーティブUIガイド](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces)で記載されたパターンです：モデルがツールを呼び出し、ツールがデータを返し、UIが結果からコンポーネントをレンダリングします。これは多くのCopilotKitやassistant-ui実装の背後にある思考モデルでもあります。

**宣言的コンポーネントカタログ**はモデルにさらに多くの自由を与えます。1つのコンポーネントではなく、モデルが許可されたパーツからツリーを構成します。カタログには`Metric`、`Table`、`Chart`、`FilterBar`、`ApprovalPanel`、`Timeline`が含まれるかもしれません。モデルはダッシュボードやワークフローのステップを構成できますが、任意のコードを実行することはできません。これは[A2UI](https://github.com/google/A2UI)、[json-render](https://github.com/vercel-labs/json-render)、[Hashbrown](https://github.com/liveloveapp/hashbrown)、[OpenUI](https://github.com/thesysdev/openui)が位置づけられる場所です。

**iframeミニアプリ**はUIがコンポーネントツリー以上に豊かである必要がある場合、またはリモートツールプロバイダーが体験を所有している場合に適しています。MCP AppsとOpenAI Apps SDKはツールが構造化データとウィジェットリソースを返し、ホストがiframeでレンダリングできるようにします。これは地図、ショッピングカート、予約フロー、チャート、外部製品の表示に強力です。これはホストアプリとウィジェットの間により厳しい境界を作り出します。

**オープンエンド生成**は、その先端に位置づけられます。エージェントはHTML、SVG、Canvas、WebGL、またはその他のコードに類似したアーティファクトをサンドボックスに生成します。[OpenGenerativeUI](https://github.com/CopilotKit/OpenGenerativeUI)が現在の最良の例です。エージェントはサンドボックスされたiframe内でアルゴリズムの可視化、3Dシーン、図表、シミュレーションを生成できます。これはワンタイムの視覚的説明に最適ですが、企業の承認フローには適していません。

ここで重要な区別を名前付けると便利です：**iframe HTML**（モデルがコードをサンドボックスに書き出す）と**JSONカタログ**（モデルが構造化された仕様を出力し、あなたのレンダラが事前に構築されたコンポーネントにマッピングする）。これらは関連しているように聞こえますが、リスクと複雑さのプロファイルは大きく異なります。iframe HTMLは表現力が最大限に発揮され、iframe境界がセキュリティの仕事を行います。JSONカタログではモデルに実行可能な自由がなく、事前に定義されたコンポーネントタイプのみを参照できます。この分野の多くのフレームワークは明確にどちらかの立場に属しています。

**サンドボックスを超えて**：最近のデモでは第4のモードが形成されている兆しがあります。これは、コンポーネントカタログが許容する以上の直接性で視覚出力を制御し、LLMがゲームのようなまたは没入型の体験を駆動するものです。プロンプトから探索可能な3Dワールドを生成するプロジェクト、実行時にLLMがNPCの行動を制御するもの、WebGPUを介したブラウザ内モデル推論（[WebLLM](https://mlc.ai/web-llm/)）は初期の指標です。ここにはまだ本格的なフレームワークが存在しません。状況が変化した時点で、別の記事でこの方向性を詳しく取り上げます。

## 高水準コンポーネント vs 細粒度コンポーネント

これは最も重要な設計決定です。

カタログが過度に細粒度になると、モデルはフロントエンドエンジニアになる必要があります：

```tsx
Container
Row
Column
Text
Button
Icon
Spacer
Divider
```

これは柔軟性があるように見えますが、今やモデルは間隔、階層、グループ化、空状態、ボタンラベル、エラートレーニング、レスポンシブ動作を決定しなければなりません。また、プロンプトが大きくなり、出力が破損しやすくなります。

カタログが過度に高水準すぎると、モデルは以下のように制限されます：

```tsx
WeatherCard
StockCard
HotelCard
```

これは安全ですが、既知のシナリオでのみ機能します。モデルは比較マトリクスを作成したり、不足している入力を求めたり、ユーザーの質問が変化した際に情報アーキテクチャを調整したりできません。

有用な中間点は**ドメインレベルのコンポーネントと制約付きスロット**です：

```tsx
SearchResults
ComparisonTable
MetricGroup
EditablePlan
ApprovalRequest
Timeline
DataCollectionForm
CheckoutReview
```

これらのコンポーネントは製品の審美とビジネス上の制約をエンコードします。モデルは*表示すべき内容*を決定できますが、すべてのCSSの決定はしません。

たとえば、旅行エージェントは`div`、`span`、`button`は必要ありません。必要なのは：

- `旅行概要`
- `フライトオプション一覧`
- `ホテル比較`
- `旅行者フォーム`
- `ポリシーノーティス`
- `予約確認`

金融エージェントは汎用的なチャートプレイグラウンドは必要ありません。必要なのは：

- `ポートフォリオのスナップショット`
- `取引一覧表`
- `リスク分解`
- `シナリオ比較`
- `承認ゲート`

コンポーネント一覧は製品の専門用語で構成されるべきで、HTMLの要素に聞こえてはなりません。

## どの製品にどの技術を使うべきか

これは実際にチームで使用する推奨マトリクスです。

**既存のSaaSアプリにアシスタントを追加する場合。**

ツールからコンポーネントへのレンダリングから始めましょう。必要に応じて、[Vercel AI SDK UI](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces)、[assistant-ui](https://github.com/assistant-ui/assistant-ui)、または[CopilotKit](https://github.com/CopilotKit/CopilotKit)を使用してください。カタログは最初は最小限に保ち、既に信頼できる製品コンポーネントをレンダリングします。

**共有状態が必要な本格的なインアプリコパイロットを構築する場合。**

CopilotKitとAG-UIを組み合わせて検討してください。重要な機能は「チャット」ではなく、共有状態と双方向の相互作用です。エージェントは入力の要求、UIのレンダリング、状態の更新、承認のための一時停止が可能です。

**境界を越えてUIを送信する必要があるリモートエージェントを扱う場合。**

A2UIまたはA2UIに類似した宣言型プロトコルを使用してください。目的はリモートエージェントがUIをデータとして記述できるようにすることです。ホスト側はネイティブレンダリング、セキュリティ、スタイルの制御を維持します。ライブなエージェント/アプリの相互作用が必要な場合は、AG-UIまたは環境が標準化しているトランスポートを介して実行してください。

**ChatGPTやMCP互換ホスト内で構築している場合。**

MCP AppsとApps SDKのパスを使用してください。OpenAIの現在のドキュメントでは、新規作業にはMCP Appsの`ui/*`ブリッジを推奨しています。`window.openai`は互換性レイヤーおよびオプショナルな拡張表面として維持されます。また、データツールとレンダートールの分離も模倣してください。モデルがウィジェットをレンダリングする前に、データの取得と推論を許可します。

**独自のアプリ内でナチュラルランゲージのダッシュボード、レポート、またはフォームを作成したい場合。**

`json-render`、`Hashbrown`、または`OpenUI`を試してください。重要なのはカタログです。`LineChart`、`DataTable`、`MetricGroup`、`FilterControl`、`InsightCallout`を公開すれば、モデルは任意のコードに近づかずに有用なレポートインターフェースを構築できます。

**教育用、視覚的、または高度にカスタマイズされたアーティファクトが必要な場合**

`OpenGenerativeUI`のようなオープンエンドのサンドボックスを使用してください。モデルにSVG、Canvas、WebGL、または自己完結型HTMLを書かせても構いませんが、出力を信頼できないユーザー内容として扱ってください。サンドボックス化し、サイズを制限し、権限を削除し、特権アプリケーション状態から隔離してください。

**主に、いくつかのインライン機能を備えたより見栄えのするストリーミングマークダウンが必要な場合**

過剰な設計は避けてください。`llm-ui`や`assistant-ui`ツールのレンダリングで十分かもしれません。

## 避けたい間違い
---

**間違い1: モデルにランタイムで本番用Reactコードを書かせること**

例外もありますが、製品UIにおいてこれは通常誤ったデフォルトです。ランタイムでのコード生成はセキュリティが難しい、テストが難しい、テーマ設定が難しい、アクセシビリティを保つのが難しいです。信頼できるコンポーネントから選ばせるだけでも、モデルは仕事の完了が可能です。

**間違い2: デザインプリミティブをプロダクトプリミティブに置き換えないこと**

モデルに`Row`、`Column`、`Text`、`Button`を与えると、それは設計システムとしての役割をモデルに強制しています。結果として中途半端な設計システムになります。より高レベルなプロダクト名詞を与えてください。

**間違い3: 有効なJSONが安全なUIを意味すると思い込むこと**

ペイロードがスキーマ検証を通過しても、依然として操作的または危険である可能性があります。ラベルが「請求書を表示」でも、実際のアクションがアカウントをアーカイブする場合があります。UI仕様は装飾ではなく行動として扱い、重要なアクションにはポリシー検査、意味論的チェック、および人間の確認が必要です。

**間違い4: レンダリングツールにビジネスロジックを置くこと**

レンダリングツールは表示専用でなければなりません。データツールはデータの取得、計算、変更、検証を行うべきです。OpenAIのApps SDKドキュメントがこの分離を強調するのは理由があります。すべてのデータツールがウィジェットを引き摺ってくれば、モデルは提示する前に推論するスペースを失います。

**間違い5: タスク完了ではなく新規性を最適化すること**

目的はすべての回答がスノーフレークインターフェースになることではありません。目的は摩擦を減らすことなのです。ユーザーに4分節約できる安定した、地味な承認パネルは、2度目で信頼できない輝かしい生成されたダッシュボードよりも優れています。

## 実用的なアーキテクチャ

もし今日から新製品を始めるなら、段階的アプローチを採用します：

1. **制御されたツールUIを最初に展開する**。既知のツールを既知のコンポーネントにマッピングする。すべてのツール呼び出し、UIレンダリング、ユーザー操作をログする。  
2. **ドメインカタログを追加する**。パターンが繰り返されるようになったら、`ComparisonTable`、`DecisionPanel`、`DataCollectionForm`、`Timeline`、その他の製品固有コンポーネントを公開する。  
3. **必要に応じてのみトランスポート標準化を追加する**。フロントエンドとバックエンドの両方を所有している場合は、単純なストリーミングで十分かもしれない。複数のエージェントフレームワークがある場合はAG-UIを使用する。ツールが製品境界を越える場合はMCPを使用する。エージェントが組織境界を越える場合は、A2AとA2UIの動向を注視する。  
4. **iframeウィジェットで外部または複雑なUI要素を扱う**。地図、カート、予約フロー、サードパーティのミニアプリは境界の外側に配置されるべきだ。  
5. **アーティファクト専用にオープンエンドの生成を予約する**。図、シミュレーション、一時的な説明、視覚的スクラッチパッドは最適な用途だ。コアワークフローには適さない。  

アーキテクチャは以下のようになる：  

```txt
User intent
  -> agent runtime
  -> tool/data calls
  -> structured result
  -> UI decision
  -> trusted component, declarative spec, or sandboxed widget
  -> user action
  -> state/event stream back to the agent
```  

このループこそが真の製品だ。チャットボックスは単なる入力デバイスの一つに過ぎない。  

## UIを含めた評価  

LLMチームは徐々にプロンプトとモデル出力の評価を学び始めている。生成型UIではもう一つの評価対象領域が追加される：インターフェース自体が誤っている可能性がある。  

最低限、生成されたUIごとに以下のアーティファクトを保存する：

- プロンプトとツールコンテキスト  
- ツール呼び出しとツール結果  
- 生成されたUI仕様またはコンポーネント選択  
- レンダードコンポーネント名とプロパティ  
- ユーザーが見えるラベル  
- ボタン/フォームに付属するアクション  
- UIからモデルに見える状態更新  
- ユーザーアクション履歴  

その後、以下のようなチェックを実施する：  

- 破壊的アクションには確認コンポーネントが必要  
- ボタンラベルはアクションの意味と一致する必要がある  
- レンダースペックは許可されたコンポーネントのみを参照できる  
- ユーザーが見える合計値はツール結果の合計値と一致する必要がある  
- フォームはタスクスコープ外のフィールドをリクエストしてはならない  
- ウィジェットはモデルのみが必要なシークレットを受け取ってはならない  
- 非表示メタデータは表示ラベルと矛盾してはならない  

これは面倒に感じるだろう。しかし、これが本番環境での信頼性を構築する場所だ。

## プロジェクトの安定性に関する注意  

この分野のすべての主要プロトコルは1.0未満です。最終確認日：2026年5月8日。プラットフォーム選定前に現行ドキュメントを確認し、変更を予測してください。  

**Vercel AI SDK RSC** — 元の「ジェネレーティブUI」ヘッドライン機能 — は2024年10月にアーキテクチャ的制約により開発が一時停止しました（[ディスカッション #3251](https://github.com/vercel/ai/discussions/3251)）。**json-render**（Vercel Labs）はその代替としてカタログベース、フレームワーク非依存、RSCとの結合なしの設計となりました。2026年初頭のリリース以降、ウェブ開発者の関心を急速に獲得しています。その理由はDX（開発者経験）です：json-renderは標準Reactプロジェクトで即座に動作しますが、A2UIのクロスプラットフォーム範囲はセットアップの摩擦を追加します。  

**A2UI**（Google）は1.0未満でマイナーバージョン間で破壊的変更があり、ロードマップの明確性に欠けています。その利点はweb、Flutter、SwiftUIを含む真のクロスプラットフォーム対応（json-renderが対応していない）です。現在の純粋webユースケースではjson-renderのツールカバレッジが優れており、クロスプラットフォームまたはリモートエージェントシナリオではA2UIの設計が適しています。2つの仕様の統合は可能で、Vercelはjson-renderからA2UI互換出力を実験しています。  

**AG-UI**（CopilotKit）も1.0未満です。最も一般的な誤解は名前で、AG-UIはUIフレームワークではなくトランスポートプロトコルです。この仕様はエージェントとフロントエンド間のイベントフローを定義しますが、レンダリング内容は開発者の責任です。概念は堅実で広く採用されていますが、仕様はまだ進化中です。

## 私の見解

ジェネレーティブUIは、丁寧に設計された製品インターフェースを代替しない。AIの汎用的なインターフェースとしてチャット転写を仮定する怠惰な姿勢を置き換えるだろう。

最良のシステムはモデルにすべてを自由に生成させるものではない。製品固有の構成要素の限られた鋭いセット、信頼性の高いランタイム接続、明確なセキュリティ境界、インターフェースをタスクに適応させるための十分な自由度を提供する。

未来は「モデルがフロントエンドを書く」ことではない。

未来はもっと近い：**あなたのフロントエンドはエージェントが演奏できる楽器になるが、その楽器が奏でることを許される音はあなたがまだ決める。**
