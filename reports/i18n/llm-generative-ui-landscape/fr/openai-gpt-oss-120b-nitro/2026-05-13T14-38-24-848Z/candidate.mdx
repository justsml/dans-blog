Le chat était la béquille d’apprentissage.

La première génération d’applications LLM ressemblait surtout à une zone de texte agrafée à un produit. Le modèle renvoyait du texte en prose. Le frontend affichait du markdown. Si l’utilisateur devait agir, l’assistant décrivait le bouton que l’utilisateur devait cliquer ailleurs.

C’était acceptable pour des démonstrations. Ce n’est pas la direction que cela prend.

L’étape suivante utile est **l’UI générative** : le modèle ne se contente plus de répondre en texte ; il aide à déterminer quelle interface l’utilisateur a besoin à l’instant T. Parfois cela implique d’appeler un outil et d’afficher une carte pré‑construite. Parfois cela consiste à remplir un composant de flux de travail connu avec des données fraîches. Parfois cela signifie composer un tableau de bord temporaire, un formulaire, un tableau comparatif, un graphique ou un widget interactif.

Malheureusement, « UI générative » est devenu l’une de ces expressions qui désigne cinq choses différentes avant le petit‑déjeuner.

Les gens l’utilisent pour décrire :

- un modèle qui choisit parmi des composants React définis par le développeur  
- une spécification JSON que le frontend rend en composants natifs  
- une application iframe renvoyée par un outil MCP  
- une bibliothèque d’interface de chat qui prend en charge les appels d’outil  
- un protocole d’agent qui diffuse l’état entre le backend et le frontend  
- un générateur de code à la conception comme v0, Lovable, Bolt ou Cursor  
- un modèle qui écrit littéralement du HTML, du SVG, du Canvas ou du React à l’exécution  

Ces éléments sont liés, mais ils n’appartiennent pas au même niveau. Si vous les mélangez, chaque discussion d’architecture se transforme en bouillie.

Voici la carte que j’aurais aimé avoir la première fois que j’ai comparé l’ensemble de la pile actuelle.

## L'incompréhension principale

La plus grande erreur consiste à considérer « UI générative » comme un choix technologique unique.

Il vaut mieux découper le problème en quatre couches :

1.  **Coquille produit** : l’interface que les utilisateurs touchent. Cela peut être un chat, un copilote latéral, un tableau de bord, un constructeur de flux de travail, un panneau IDE, une application ChatGPT, un écran mobile ou une console d’assistance.
2.  **Modèle de composition UI** : la grammaire que le modèle est autorisé à parler. Il peut s’agir d’appels d’outils, de JSON, d’A2UI, de json‑render, d’OpenUI Lang, de la sélection de composants Hashbrown ou de HTML sandboxé.
3.  **Runtime et transport** : la façon dont les messages, les appels d’outils, les deltas d’état, les actions utilisateur et les artefacts UI circulent entre l’agent et le frontend. AG‑UI, MCP Apps, Apps SDK, A2A, SSE, WebSockets et le bon vieux HTTP se situent tous ici.
4.  **Backend agent et outils** : LangGraph, Google ADK, CrewAI, Mastra, LlamaIndex, Pydantic AI, Agno, OpenAI Agents SDK, fonctions personnalisées, bases de données, récupération et toute la logique métier fastidieuse qui doit rester correcte.

Une fois les couches séparées, l’écosystème devient beaucoup moins mystique.

[AG‑UI](https://github.com/ag-ui-protocol/ag-ui) n’est pas vraiment un concurrent de [A2UI](https://github.com/google/A2UI). AG‑UI est un protocole d’événements pour l’interaction agent‑application. A2UI est un format UI déclaratif que l’agent peut envoyer. Vous pouvez placer A2UI au‑dessus d’AG‑UI. Vous pouvez aussi superposer des composants rendus par des outils personnalisés sur AG‑UI.

[json‑render](https://github.com/vercel-labs/json-render) n’est pas un produit de chat. C’est un catalogue de composants et une architecture de rendu : définissez les composants que le modèle peut utiliser, laissez le modèle émettre un arbre JSON valide, puis rendez cet arbre en toute sécurité.

[CopilotKit](https://github.com/CopilotKit/CopilotKit) n’est pas seulement une bulle de chat. C’est une pile frontend pour les applications natives agents : UI de chat, UI générative, état partagé, outils frontend et flux humain‑dans‑la‑boucle.

[OpenAI Apps SDK](https://developers.openai.com/apps-sdk) et [MCP Apps](https://github.com/MCP-UI-Org/mcp-ui) ne sont pas des outils « rendre mon application React dynamique ». Ce sont des modèles d’intégration d’hôte pour le rendu de widgets à l’intérieur de ChatGPT ou d’autres hôtes compatibles MCP.

Les noms sont déroutants parce que le domaine est encore jeune. Les couches, elles, restent utiles.

## Le spectre de contrôle

L’UI générative représente un compromis entre **contrôle du développeur** et **liberté de l’agent**.

Trop de contrôle et l’assistant ressemble à une palette de commandes déguisée. Trop de liberté et le modèle commence à inventer des mises en page étranges, des boutons vagues, une hiérarchie visuelle cassée, des états impossibles et des problèmes de sécurité avec un petit sourire confiant.

L’astuce consiste à choisir la moindre quantité de liberté qui résout le problème de l’utilisateur.

![Un spectre allant des composants rendus par outil à du HTML généré librement](.././control-spectrum.webp)

Je conçois le spectre ainsi :

**Le rendu outil‑vers‑composant** est le réglage par défaut le plus sûr. Le modèle invoque `get_weather`, `search_products`, `compare_plans` ou `draft_invoice`. L’application mappe le résultat de cet outil à un composant que vous possédez déjà : `WeatherCard`, `ProductGrid`, `PlanComparison`, `InvoiceReview`. Le modèle décide *quand* l’UI est utile. Les développeurs conservent la maîtrise de la mise en page, du style, de l’accessibilité, des états de chargement, des états vides et des actions dangereuses.

C’est le pattern décrit dans le [guide d’UI générative du Vercel AI SDK](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces) : le modèle appelle un outil, l’outil renvoie des données, et l’UI rend un composant à partir du résultat. C’est également le modèle mental derrière de nombreuses implémentations CopilotKit et assistant‑ui.

**Les catalogues de composants déclaratifs** offrent plus de marge de manœuvre au modèle. Au lieu de choisir un seul composant, le modèle compose un arbre à partir de parties autorisées. Un catalogue peut contenir `Metric`, `Table`, `Chart`, `FilterBar`, `ApprovalPanel` et `Timeline`. Le modèle peut assembler un tableau de bord ou une étape de workflow, mais il ne peut pas exécuter de code arbitraire. C’est là que se situent [A2UI](https://github.com/google/A2UI), [json‑render](https://github.com/vercel-labs/json-render), [Hashbrown](https://github.com/liveloveapp/hashbrown) et [OpenUI](https://github.com/thesysdev/openui).

**Les mini‑applications iframe** sont pertinentes lorsque l’UI doit être plus riche qu’un simple arbre de composants, ou lorsqu’un fournisseur d’outil distant possède l’expérience. MCP Apps et OpenAI Apps SDK permettent à un outil de renvoyer des données structurées plus une ressource widget que l’hôte rend dans une iframe. C’est puissant pour les cartes, les paniers d’achat, les flux de réservation, les graphiques et les surfaces produit externes. Cela crée également une frontière plus difficile à franchir entre l’application hôte et le widget.

**Open-ended generation** représente l’extrémité la plus libre : l’agent émet du HTML, du SVG, du Canvas, du WebGL ou d’autres artefacts de type code dans un bac à sable. [OpenGenerativeUI](https://github.com/CopilotKit/OpenGenerativeUI) est l’exemple le plus abouti à ce jour : l’agent peut générer des visualisations d’algorithmes, des scènes 3D, des diagrammes et des simulations à l’intérieur d’iframes sandboxées. C’est idéal pour des explications visuelles ponctuelles. Ce n’est pas le point de départ recommandé pour un flux d’approbation en entreprise.

Il est utile de nommer la distinction clé ici : **iframe HTML** (le modèle écrit du code dans un bac à sable) vs. **catalogue JSON** (le modèle émet une spécification structurée et votre moteur de rendu la traduit en composants pré‑construits). Ces deux approches semblent liées mais présentent des profils de risque et de complexité très différents. L’iframe HTML est maximalement expressive ; la frontière de l’iframe assure la majeure partie de la sécurité. Le catalogue JSON ne donne aucune liberté d’exécution au modèle — il ne peut que référencer des types de composants que vous avez définis au préalable. La plupart des frameworks de cet espace se rangent clairement dans l’un ou l’autre camp.

**Au‑delà du bac à sable** : des démonstrations très récentes laissent entrevoir un quatrième mode — les LLM pilotant des expériences de type jeu ou immersives en contrôlant la sortie visuelle plus directement que ne le permet un catalogue de composants. Des projets qui génèrent des mondes 3D explorables à partir d’invites, qui dirigent le comportement de PNJ en temps réel, ou qui exécutent l’inférence de modèle dans le navigateur via WebGPU ([WebLLM](https://mlc.ai/web-llm/)) sont des premiers indicateurs. Aucun framework stable n’est encore disponible pour construire des applications de production dans cette direction. J’aborderai ce sujet dans un article dédié dès que cela évoluera.

## Composants de haut niveau vs composants granulaires

C’est la décision de conception la plus critique.

Si votre catalogue est trop granulaire, le modèle doit se comporter comme un ingénieur frontend :

```tsx
Container
Row
Column
Text
Button
Icon
Spacer
Divider
```

Cela semble flexible, mais maintenant le modèle doit choisir les espacements, la hiérarchie, le groupement, les états vides, les libellés des boutons, le traitement des erreurs et le comportement réactif. Vous avez également alourdi l’invite et rendu la sortie plus facile à fragmenter.

Si votre catalogue est trop abstrait, le modèle est coincé :

```tsx
WeatherCard
StockCard
HotelCard
```

C’est sûr, mais cela ne fonctionne que pour des scénarios connus. Le modèle ne peut pas créer une matrice de comparaison, demander les entrées manquantes, ou adapter l’architecture de l’information lorsque la question de l’utilisateur évolue.

Le compromis utile se situe au **niveau du domaine avec des emplacements contraints** :

```tsx
SearchResults
ComparisonTable
MetricGroup
EditablePlan
ApprovalRequest
Timeline
DataCollectionForm
CheckoutReview
```

Ces composants intègrent les préférences produit et les contraintes métier. Le modèle décide *de ce qui doit être affiché*, sans contrôler chaque décision CSS.

Par exemple, un agent de voyage n’a pas besoin de `div`, `span` et `button`. Il a besoin :

- `TripSummary`
- `FlightOptionList`
- `HotelComparison`
- `TravelerForm`
- `PolicyNotice`
- `BookingConfirmation`

Un agent financier n’a pas besoin d’un bac à sable de graphiques générique. Il a besoin :

- `PortfolioSnapshot`
- `TransactionTable`
- `RiskBreakdown`
- `ScenarioComparison`
- `ApprovalGate`

Le catalogue doit refléter votre produit, pas du HTML.

## What To Use For Which Product

Voici la matrice de recommandations que j’utiliserais réellement avec une équipe.

**Vous ajoutez un assistant à une application SaaS existante.**

Commencez par le rendu outil‑à‑composant. Utilisez [Vercel AI SDK UI](https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces), [assistant-ui](https://github.com/assistant-ui/assistant-ui) ou [CopilotKit](https://github.com/CopilotKit/CopilotKit), selon le degré d’état d’agent et d’intégration d’outils frontaux dont vous avez besoin. Gardez le catalogue très petit au départ. Rendu des composants produit que vous avez déjà validés.

**Vous construisez un copilot sérieux intégré à l’app qui nécessite un état partagé.**

Examinez attentivement CopilotKit avec AG‑UI. La fonctionnalité clé n’est pas le « chat », mais l’état partagé et l’interaction bidirectionnelle : l’agent peut demander une entrée, rendre l’UI, mettre à jour l’état, puis se mettre en pause en attendant une approbation.

**Vous avez des agents distants qui doivent envoyer de l’UI à travers une frontière.**

Utilisez A2UI ou un protocole déclaratif similaire à A2UI. L’idée centrale est qu’un agent distant puisse décrire l’UI sous forme de données tandis que l’hôte conserve le contrôle du rendu natif, de la sécurité et du style. Si vous avez également besoin d’une interaction en temps réel entre l’agent et l’application, faites‑le passer par AG‑UI ou tout autre transport standardisé dans votre environnement.

**Vous développez à l’intérieur de ChatGPT ou d’un hôte compatible MCP.**

Utilisez le chemin MCP Apps et le SDK Apps. La documentation actuelle d’OpenAI recommande le pont `ui/*` de MCP Apps pour les nouveaux projets, tout en conservant `window.openai` comme couche de compatibilité et surface d’extension optionnelle. Copiez également leur séparation entre outils de données et outils de rendu : laissez le modèle récupérer et raisonner sur les données avant qu’il ne décide de rendre un widget.

**Vous voulez des tableaux de bord, rapports ou formulaires en langage naturel dans votre propre application.**

Essayez json-render, Hashbrown ou OpenUI. L’élément clé est le catalogue. Si vous exposez `LineChart`, `DataTable`, `MetricGroup`, `FilterControl` et `InsightCallout`, le modèle peut assembler des surfaces de reporting utiles sans s’approcher du code arbitraire.

**Vous voulez des artefacts éducatifs, visuels ou très sur‑mesure.**

Utilisez un bac à sable ouvert comme OpenGenerativeUI. Laissez le modèle générer du SVG, du Canvas, du WebGL ou du HTML autonome, mais traitez la sortie comme du contenu utilisateur non fiable. Isolez‑le, limitez sa taille, retirez les permissions et gardez‑le éloigné de l’état privilégié de l’application.

**Vous avez surtout besoin d’un markdown en flux plus élégant avec quelques possibilités en ligne.**

Ne surchargez pas. Le rendu d’outils comme llm-ui ou assistant-ui peut suffire.

## Les erreurs que j’éviterais

**Erreur 1 : Laisser le modèle générer du React de production à l’exécution.**  

Il existe des exceptions, mais pour une interface produit c’est généralement le mauvais choix par défaut. La génération de code à la volée est difficile à sécuriser, à tester, à thématiser et à rendre accessible. Si le modèle peut accomplir la tâche en sélectionnant parmi des composants de confiance, faites‑le ainsi.  

**Erreur 2 : Exposer des primitives de conception au lieu de primitives de produit.**  

Lorsque vous fournissez au modèle `Row`, `Column`, `Text` et `Button`, vous lui demandez de devenir votre système de design. Il en résultera un système médiocre. Donnez‑lui des noms de produit de niveau supérieur.  

**Erreur 3 : Penser que du JSON valide équivaut à une UI sûre.**  

Une charge utile peut passer la validation du schéma et rester manipulatrice ou dangereuse. L’étiquette peut indiquer « Voir la facture » alors que l’action archive le compte. Traitez les spécifications UI comme du comportement, pas comme de la simple décoration. Elles nécessitent des tests de politique, des vérifications sémantiques et une confirmation humaine pour les actions à conséquences.

**Erreur 4 : Mettre la logique métier dans les outils de rendu.**

Les outils de rendu doivent uniquement rendre. Les outils de données doivent récupérer, calculer, muter et valider. La documentation de l’OpenAI Apps SDK souligne cette séparation pour une raison : si chaque outil de données entraîne un widget, le modèle perd de l’espace de raisonnement avant de présenter le résultat.

**Erreur 5 : Optimiser pour la nouveauté plutôt que pour l’accomplissement de la tâche.**

Le but n’est pas de transformer chaque réponse en interface « flocon de neige ». Le but est de réduire les frictions. Un panneau d’approbation stable et ennuyeux qui fait gagner quatre minutes à l’utilisateur vaut mieux qu’un tableau de bord généré époustouflant mais dont on ne peut pas faire confiance deux fois.

## Une architecture pratique

Si je devais lancer un nouveau produit aujourd’hui, j’adopterais une approche en plusieurs étapes :

1.  **Livrez d’abord une UI d’outil contrôlée.** Mappez les outils connus sur des composants connus. Consignez chaque appel d’outil, chaque rendu UI et chaque action utilisateur.  
2.  **Ajoutez un catalogue de domaine.** Dès que des motifs se répètent, exposez `ComparisonTable`, `DecisionPanel`, `DataCollectionForm`, `Timeline` et d’autres composants spécifiques au produit.  
3.  **Standardisez le transport uniquement si nécessaire.** Si vous maîtrisez à la fois le front‑end et le back‑end, un simple streaming peut suffire. Si vous avez plusieurs frameworks d’agents, utilisez AG‑UI. Si les outils franchissent les limites du produit, utilisez MCP. Si les agents traversent les frontières organisationnelles, surveillez A2A et A2UI.  
4.  **Utilisez des widgets iframe pour des surfaces étrangères ou complexes.** Cartes, paniers, flux de réservation et mini‑applications tierces doivent être placés derrière une frontière.  
5.  **Réservez la génération ouverte aux artefacts.** Diagrammes, simulations, explications temporaires et tableaux blancs visuels sont un bon cas d’usage. Les flux de travail centraux ne le sont pas.

L’architecture finit par ressembler à ceci :

```txt
User intent
  -> agent runtime
  -> tool/data calls
  -> structured result
  -> UI decision
  -> trusted component, declarative spec, or sandboxed widget
  -> user action
  -> state/event stream back to the agent
```

Cette boucle constitue le vrai produit. La boîte de chat n’est qu’un dispositif d’entrée possible.

## L’évaluation doit inclure l’UI

Les équipes LLM apprennent lentement à évaluer les prompts et les sorties du modèle. L’UI générative ajoute une autre surface d’exposition : l’interface elle‑même peut être erronée.

Au minimum, conservez ces artefacts pour chaque UI générée :

- contexte de l’invite et de l’outil  
- appels d’outil et résultats d’outil  
- spécification UI générée ou sélection de composant  
- nom du composant rendu et ses propriétés  
- libellés visibles par l’utilisateur  
- actions attachées aux boutons / formulaires  
- mises à jour d’état visibles par le modèle provenant de l’UI  
- historique des actions de l’utilisateur  

Ensuite, rédigez des contrôles tels que :

- chaque action destructive doit être accompagnée d’un composant de confirmation  
- les libellés des boutons doivent correspondre à la sémantique de l’action  
- les spécifications de rendu ne peuvent référencer que des composants autorisés  
- les totaux visibles par l’utilisateur doivent correspondre aux totaux des résultats d’outil  
- les formulaires ne doivent pas demander de champs hors du périmètre de la tâche  
- les widgets ne doivent pas recevoir de secrets dont seul le modèle a besoin  
- les métadonnées cachées ne doivent pas contredire les libellés visibles  

Cela paraît fastidieux. C’est pourtant l’endroit d’où provient la confiance en production.

## Une note sur la stabilité des projets

Chaque protocole majeur de cet espace est encore en pré‑1.0. Dernière vérification le 8 mai 2026 ; prévoyez des changements et consultez la documentation actuelle avant de placer un pari sur une plateforme.

**Vercel AI SDK RSC** — la fonctionnalité phare « Generative UI » d’origine — a été mise en pause en octobre 2024 ([Discussion #3251](https://github.com/vercel/ai/discussions/3251)) en raison de limitations architecturales sans correctif à court terme. **json‑render** (Vercel Labs) a émergé comme la direction de remplacement : basé sur un catalogue, agnostique de framework, sans couplage RSC. Il semble avoir rapidement attiré l’attention des développeurs web depuis son lancement début 2026. La raison probable est le DX : json‑render fonctionne immédiatement dans un projet React standard ; la portée multiplateforme d’A2UI ajoute une friction de configuration.

**A2UI** (Google) est pré‑1.0 avec des ruptures entre versions mineures et une communication de feuille de route incohérente. Son atout est une véritable portée multiplateforme (web, Flutter, SwiftUI) que json‑render ne couvre pas. Pour les cas d’usage purement web aujourd’hui, json‑render offre une meilleure couverture d’outils ; pour le multiplateforme ou les scénarios d’agent distant, le design d’A2UI est plus approprié. Une convergence entre les deux spécifications est possible — Vercel a expérimenté une sortie compatible A2UI depuis json‑render.

**AG-UI** (CopilotKit) est également pré‑1.0. La confusion la plus courante porte sur le nom : AG‑UI est un protocole de transport, pas un framework UI. Il définit *comment* les événements circulent entre l’agent et le frontend ; ce que vous rendez reste à votre charge. Le concept est solide et largement adopté. La spécification est encore en évolution.

## Mon point de vue

L’UI générative ne remplacera pas les interfaces produit soigneusement conçues. Elle éliminera l’hypothèse paresseuse selon laquelle un transcript de chat serait l’interface universelle pour l’IA.

Les meilleurs systèmes ne laisseront pas le modèle improviser tout. Ils lui fourniront un petit ensemble pointu de blocs de construction natifs au produit, une connexion d’exécution fiable, des limites de sécurité claires et suffisamment de liberté pour adapter l’interface à la tâche.

L’avenir n’est pas « le modèle écrit votre frontend ».

L’avenir ressemble davantage à : **votre frontend devient un instrument que l’agent peut jouer, mais vous décidez toujours ce que cet instrument est autorisé à produire.**
